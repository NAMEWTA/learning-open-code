# MISSION.md 格式

`MISSION.md` 存放在工作区根目录。它记录了用户学习此主题的 _原因_。每一个教学决策——下一步教什么、呈现哪些资源、设计哪些练习——都应追溯到此文档。

## 模板

```md
# 使命：{主题}

## 为什么
{1-3 句话。用户正在追求的具体的现实世界目标。当他们拥有这项技能时，他们的生活或工作会发生什么变化？避免抽象表述如"理解 X"——追问底层的实际结果。}

## 成功的样子
- {用户将能够做到的一个具体的、可观察的事情}
- {另一个具体的事情}
- {……}

## 约束条件
- {时间、预算、已有承诺、学习偏好，任何限制方法边界的因素}

## 不在范围内
- {用户明确表示现在不想追求的相邻主题——保护最近发展区}
```

## 规则

- **每个工作区一个使命。** 如果用户想学两个不相关的东西，那就是两个工作区。
- **具体优于抽象。**"十月前跑完半程马拉松"优于"变得更健康"。"给我的团队交付一个 Rust CLI"优于"学 Rust"。
- **追问模糊性。** 如果用户无法说清为什么，在写任何内容之前先访谈他们。一个糟糕的使命比没有使命更糟。
- **当现实变化时修订。** 使命会变化。当用户的目标移动时，更新此文件——不要让一个过时的使命引导未来的会话。
- **保持简短。** 如果 `MISSION.md` 超过一屏，它就不再是指南针，而变成了计划书。
