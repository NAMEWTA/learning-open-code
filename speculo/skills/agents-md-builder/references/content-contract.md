# Content Contract

所有 AGENTS.md 都必须写成**模块导读**，不能退化成目录卡片。每个章节都要回答明确问题。

## 必填章节

每个 AGENTS.md 必须包含以下章节。章节名称保留英文，正文使用中文。

### 1. `Scope`

回答：
- 这个文件覆盖哪些目录？
- 哪些更深层 manifest 会接管说明？
- 哪些内容明确不覆盖？

写法：2-4 句话，边界清晰。

### 2. `Module Positioning`

回答：
- 当前目录是什么角色？
- 它在整个项目中承担什么位置？
- 为什么它存在？

写法：标记角色（如 `capability-module`），点出它在项目分层中的坐标。

### 3. `Directory Structure`

回答：
- 直接子目录有哪些？各自承担什么？
- 关键文件有哪些？它们为什么重要？
- 主要实现区在哪？测试区在哪？

写法：只列 5-10 项对排查问题有用的目录和文件，不做全量目录树。

### 4. `Responsibilities`

回答：
- 当前模块负责什么？
- 明确不负责什么？

写法：用列表承载 3-6 条职责。负面影响（不负责什么）同样重要。

### 5. `Core Logic`

回答：
- 模块如何工作？
- 关键装配顺序、处理链路或数据流是什么？
- 它的核心机制是什么？

写法：3-6 句话解释主干逻辑。不要展开每个文件的细节。

### 6. `Key Entry Points`

回答：
- 先看哪些具体文件？
- 从哪个目录开始依次下钻？

写法：**必须点名具体文件、类、目录或测试入口**。禁止只写 `src/`、`api/`、`controller/` 这类空泛词。

好例子：
- `src/main.ts` — 应用启动入口
- `src/router/index.ts` — 路由注册
- `src/api/user.ts` — 用户相关接口封装
- `tests/user.spec.ts` — 用户模块测试

差例子：
- `src/` — 主实现
- `api/` — 接口层

### 7. `Usage And Change Notes`

回答：
- 典型使用入口是什么？
- 修改时优先检查哪些配置、依赖或调用链？
- 最容易误改的边界在哪里？

写法：2-4 条高信息量可执行提醒。禁止写成“修改时注意兼容性”“按需查看相关代码”这类空话。

好例子：
- “新增接口时先确认 controller 所属归档单元，再同步检查 bo/vo 和前端接口契约。”
- “改请求层时先检查 `packages/core`、环境变量和下载链路。”

差例子：
- “修改时注意兼容性。”
- “需要时查看相关代码。”

### 8. `Dependencies And Collaboration`

回答：
- 直接依赖了哪些项目/仓库内部模块？
- 被哪些兄弟模块消费？
- 模块间如何协作？

写法：列表形式。区分“我依赖谁”和“谁依赖我”。

### 9. `Investigation Rule`

回答：
- 当本文件不够时，按什么顺序继续读源码？
- 文档与代码冲突时怎么办？

写法：**必须写成可执行顺序**。每一步点名具体文件或目录。

好例子：
```
1. 先读 pom.xml 了解依赖和模块声明。
2. 再读 src/main/java/.../Application.java 了解启动装配。
3. 然后按 controller → service → domain → mapper 顺序下钻。
4. 仍不够时，读 src/test/ 了解预期行为。
5. 文档与代码冲突时以代码事实为准。
```

差例子：
- “需要时查看源码。”
- “先看 controller、service、domain。”

### 10. `Core Commands`

回答：
- 当前模块最常用的构建/测试/运行命令是什么？
- 哪些命令是非标准的、AGENTS 无法从 manifest 推断的？

写法：只写 AGENTS 猜不到的命令。从 `package.json` scripts 或 `pom.xml` 插件显而易见的命令不用写。

### 11. `Verification`

回答：
- 最小可执行验证是什么？
- 如何确认模块正常工作？

写法：1-2 条具体命令。

### 12. `Routing`

回答：
- 上级 manifest 在哪？
- 子级 manifest 有哪些？

写法：
```
- 上级：[../AGENTS.md](../AGENTS.md)
- 子级：
  - [子模块A](./子模块A/AGENTS.md) — 一句话定位
  - [子模块B](./子模块B/AGENTS.md) — 一句话定位
```

每个子级链接后附一句话定位。`repo-root` 无上级，不写上级行。

### 13. `References`

回答：
- 关键 manifest、配置、源码入口、测试入口在哪？

写法：用相对路径列出 3-8 个关键引用。只列对 AGENTS 有用的，不做全文索引。

## 禁止项

以下内容**一律禁止**出现在 AGENTS.md 中：

| 禁止 | 原因 |
|------|------|
| 逐文件的结构列表 | AGENTS 可通过阅读代码库发现 |
| 标准语言约定 | AGENTS 已知（如“React 是声明式 UI 库”） |
| 通用建议 | “编写干净的代码”“处理错误”等无操作价值 |
| 从 manifest 显而易见的命令 | 如 `npm test`、`mvn clean install` |
| 经常变化的信息 | 版本号、日期、人员名单——用路径引用源文件 |
| 长教程或演练 | 移到 references 中 |
| `Key Entry Points` 中只写目录名 | 必须点名具体文件、类 |
| `Usage And Change Notes` 中的空话 | 必须可执行 |
| `Investigation Rule` 中的空话 | 必须有具体文件和顺序 |
| 复制大段源码或配置全文 | 引用路径即可 |
| 宣传性语言 | “业界领先的”“强大的”“优雅的” |

## 角色额外要求

### `repo-root`

- 解释顶层分层理由，不只列模块名
- 写清“从根级定位问题后，下一步怎么进入子模块”
- `Usage And Change Notes` 要提醒：根级改动容易联动 workspace 配置、顶层 scripts 和索引

### `scripts-docs`

- 明确这是特批文档例外，不是 manifest 模块
- 概括各类子树各自承载什么
- 区分“可直接执行的脚本”“SQL 资产”“配置模板”“运行时数据目录”
- 明确“只到根目录这一层”，不为子目录生成子级文档
- `Usage And Change Notes` 必须写清：哪些可运行、哪些只是资产索引

### `aggregator`

- 解释当前聚合为什么存在
- 给出子模块的分组逻辑，不只是列模块名
- 写清“哪些问题必须进入哪个子模块继续调查”

### `runnable-app`

- 写清启动入口、运行时装配
- 写清内部依赖模块如何接入
- `Usage And Change Notes` 要提醒：运行时改动应先看哪些入口、哪些配置联动最易漏

### `capability-module`

- 写清导出面、核心分层
- 写清内部依赖与上游消费方
- `Usage And Change Notes` 要提醒：新增接口/类型/配置时优先检查的层次

### `contract-module`

- 写清约束了什么、被谁消费
- 没有 `src/` 时显式说明这是约束层
- `Usage And Change Notes` 要提醒：改约束会影响哪些消费方
