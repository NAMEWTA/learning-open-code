# Writing Style

默认使用中文，目标是**平衡型模块导读**：比索引更深，但不写成长篇百科。

## 中文排版

遵循以下排版规则，避免机翻味和目录卡片化。

### 空格

- 中文与英文之间加空格：
  - `使用 Git 管理版本`
  - `接入 Spring Boot`
- 中文与数字之间加空格：
  - `新增 3 个入口`
  - `默认保留 2 条阅读路径`
- 链接前后保留正常中文空格，不要把链接和汉字黏在一起。

### 标点

- 中文语境使用全角标点：
  - `注意：这里只写导航，不复制实现细节。`
- 英文代码、命令、路径保留半角形式：
  - `先读 package.json，再读 src/main.ts。`

### 术语

- 保留业界通用英文术语：`Controller`、`Service`、`Router`、`Store`、`API`、`SDK`、`CLI`
- 有稳定中文表达的通用概念直接写中文：`数据库`、`服务器`、`配置约束`、`工作区`
- 首次出现的重要术语可写中英对照，但不要每次重复翻译
- 不要把常用框架术语硬翻成别扭中文

### 句式

- 优先短句，一句话只说一层意思
- 少用被动句：
  - 差：`该文件被用来描述模块。`
  - 好：`该文件用于描述模块。`
- 少写空话：
  - 差：`如需更多信息请查看源码。`
  - 好：`若要确认运行入口，先读 src/main.ts，再读 router 与 request 链路。`
- 不要把不确定推断写成既定事实

### 结构

- 先写模块位置，再写它如何工作，再写如何继续下钻
- 使用列表承载职责、入口、边界和排查顺序，不堆长段落
- `Usage And Change Notes` 要写成可执行提醒，不要写成泛泛的“修改时注意兼容性”

## 长度与密度

| 角色 | 推荐行数 |
|------|---------|
| `repo-root` | 45-75 行 |
| `scripts-docs` | 40-65 行 |
| `aggregator` | 40-70 行 |
| `runnable-app` | 50-85 行 |
| `capability-module` | 45-80 行 |
| `contract-module` | 35-60 行 |

每个章节尽量 2-5 个高信息量要点。

## 好写法 vs 差写法

### Scope / Module Positioning

- ✅ 先说明模块在项目中的位置，再说明它如何工作
- ✅ 用“入口文件 + 协作目录 + 消费关系”描述排查路径
- ✅ 写清“看完本文件之后，下一步去哪里”

- ❌ “主实现通常位于 `src/`”
- ❌ “当前包暴露共享能力”
- ❌ “如需更多信息请查看源码”

### Key Entry Points

- ✅ 直接点名类、配置、测试与关键目录，例如：
  - `Application.java` — 启动类
  - `web/controller` — REST 控制器目录
  - `application-dev.yml` — 开发环境配置
  - `src/main.ts` — 应用入口
  - `src/router/index.ts` — 路由注册

- ❌ “主实现位于 `src/`”
- ❌ “api/ — 接口层”
- ❌ 只列目录，不解释这些目录为什么重要

### Usage And Change Notes

- ✅ 写成可执行提醒：
  - “新增接口时先确认 controller 所属归档单元，再同步检查 bo/vo 和前端接口契约。”
  - “改请求层时先检查 `packages/core`、环境变量和下载链路。”
  - “改导出面、公共类型或运行时工具时，先列出直接消费它的模块，避免隐式破坏调用方。”

- ❌ “修改时注意兼容性”
- ❌ “按需查看相关代码”
- ❌ “先看 controller、service、domain”

### Investigation Rule

- ✅ 必须写成可执行顺序：
  ```
  1. 先读 pom.xml 了解依赖和模块声明。
  2. 再读启动类或核心入口包。
  3. 再读资源配置与测试。
  4. 仍不够时顺着内部依赖继续下钻。
  5. 文档与代码冲突时以代码事实为准。
  ```

- ✅ 必须明确：
  - 本文件是入口，不是终点
  - 文档与代码冲突时以代码事实为准

- ❌ “需要时查看源码”
- ❌ “如需更多信息请参考代码”

## 语气要求

- 直接、事实化、可执行
- 不写宣传性语言（“业界领先的”“强大的”“优雅的”）
- 不把不确定推断写成既定事实
- 不用感叹号
- 不用“请”“建议”等婉转表达——直接说“先读 X，再读 Y”

## 删除测试

写完每个 AGENTS.md 后，逐行执行删除测试：

> **“删除这一行会导致 AI 代理犯错吗？”**

- 如果不会 → 删除这一行
- 如果会 → 保留

这个测试是 AGENTS.md 质量的核心防线。它强制每一个字都对 AI 代理有操作价值。

### 删除测试示例

| 行 | 删掉会犯错吗？ | 判定 |
|----|-------------|------|
| “该模块负责用户认证与授权” | 会 → AGENTS 可能在错误模块找认证逻辑 | ✅ 保留 |
| “推荐使用 VS Code 进行开发” | 不会 → AGENTS 不需要知道用什么编辑器 | ❌ 删除 |
| “模块使用 TypeScript 编写” | 不会 → AGENTS 打开文件就能看出 | ❌ 删除 |
| “先读 `src/main.ts`，再读 `src/router/index.ts`” | 会 → AGENTS 不知道该从哪开始 | ✅ 保留 |
| “注意代码质量” | 不会 → 这是无操作价值的空话 | ❌ 删除 |
| “改 `src/api/` 时同步检查前端接口契约” | 会 → 这是跨边界风险提醒 | ✅ 保留 |
