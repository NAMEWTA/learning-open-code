# MISSION.md 格式

`MISSION.md` 位于工作区根目录。它捕获用户学习该主题的_原因_。每个教学决策 — 接下来教什么、呈现哪些资源、设计哪些练习 — 都应追溯到此文档。

## 模板

```md
# Mission: {Topic}

## Why
{1-3 句话。用户正在追求的具体的、真实世界中的目标。当拥有这项技能时，他们的生活或工作中会发生什么改变？避免抽象的表述如"理解 X" — 追问底层的成果。}

## Success looks like
- {用户将能够做到的一件具体的、可观察的事情}
- {另一件具体的事情}
- {……}

## Constraints
- {时间、预算、既有承诺、学习偏好，任何限制方法的边界条件}

## Out of scope
- {用户明确不想现在追求的相邻主题 — 保护最近发展区}
```

## 规则

- **每个工作区一个使命。** 如果用户想学习两个不相关的东西，那就是两个工作区。
- **具体优于抽象。** "十月份之前跑完半程马拉松"优于"变得更健康"。"给我的团队交付一个 Rust CLI 工具"优于"学习 Rust"。
- **对模糊性进行追问。** 如果用户无法说清为什么，在写任何东西之前和他们面谈。一个糟糕的使命比没有使命更糟。
- **当现实改变时修订。** 使命会改变。当用户的目标移动时，更新此文件 — 不要让一个过时的使命引导未来的会话。
- **保持简短。** 如果 `MISSION.md` 超过一屏，它已经不再是罗盘，而是变成了计划。
