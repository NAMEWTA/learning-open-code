# 范围外知识库

仓库中的 `.out-of-scope/` 目录存储被拒绝功能请求的持久记录。它有两个目的：

1. **机构记忆** — 为什么一个功能被拒绝，这样理由不会随 issue 关闭而丢失
2. **去重** — 当新 issue 与之前的拒绝匹配时，该 skill 可以呈现之前的决策而非重新讨论

## 目录结构

```
.out-of-scope/
├── dark-mode.md
├── plugin-system.md
└── graphql-api.md
```

每个**概念**一个文件，而非每个 issue 一个文件。请求同一功能的多个 issue 归入同一个文件。

## 文件格式

文件应以轻松、可读的风格编写 — 更像一份简短的设计文档，而非数据库条目。使用段落、代码示例和实例让理由清晰并对初次遇到它的人有用。

```markdown
# Dark Mode

此项目不支持深色模式或面向用户的主题化。

## Why this is out of scope

渲染管线假定在 `ThemeConfig` 中定义单一调色板。
支持多主题将需要：

- 包装整个组件树的主题上下文提供者
- 每个组件的主题感知样式解析
- 用户主题偏好的持久层

这是一个重大的架构变更，不符合项目对内容创作的聚焦。
主题化是下游消费者（嵌入或再分发输出的消费者）的关注点。

```ts
// 当前的 ThemeConfig 接口不是为运行时切换而设计的：
interface ThemeConfig {
  colors: ColorPalette; // 单一调色板，在构建时解析
  fonts: FontStack;
}
```

## Prior requests

- #42 — "添加深色模式支持"
- #87 — "用于无障碍的夜间主题"
- #134 — "深色主题选项"
```

### 文件命名

使用简短的、描述性的 kebab-case 名称来表示概念：`dark-mode.md`、`plugin-system.md`、`graphql-api.md`。名称应足够可辨识，使浏览目录的人无需打开文件就能理解被拒绝的是什么。

### 编写理由

理由应是实质性的 — 不是"我们不想要这个"而是为什么。好的理由引用：

- 项目范围或理念（"本项目聚焦于 X；主题化是下游关注点"）
- 技术约束（"支持这个将需要 Y，这与我们的 Z 架构冲突"）
- 战略决策（"我们选择使用 A 而不是 B，因为……"）

理由应是持久的。避免引用临时情况（"我们目前太忙了"）— 那些不是真正的拒绝，而是推迟。

## 何时检查 `.out-of-scope/`

在分类期间（第 1 步：收集上下文），读取 `.out-of-scope/` 中的所有文件。在评估新 issue 时：

- 检查请求是否匹配现有的范围外概念
- 匹配基于概念相似性，而非关键词 — "night theme" 匹配 `dark-mode.md`
- 如果存在匹配，将其呈现给维护者："这与 `.out-of-scope/dark-mode.md` 相似 — 我们之前因为 [理由] 拒绝了这个。你仍然有同样的感觉吗？"

维护者可能：

- **确认** — 新 issue 被添加到现有文件的"Prior requests"列表中，然后关闭
- **重新考虑** — 范围外文件被删除或更新，issue 进入正常分类流程
- **不同意** — 这些 issue 相关但不同，继续正常分类

## 何时写入 `.out-of-scope/`

仅当**增强**（非 bug）被*拒绝*为 `wontfix` 时。这也适用于增强 PR，就像适用于 issue 一样 — 被拒绝的 PR 被记录在这里，这样相同的请求不会以新代码的形式再次出现。

**不要**在某个功能以 `wontfix` 关闭是因为它**已经实现了**的情况下写入这里。那是已构建的功能，不是被拒绝的功能；记录它将用虚假的拒绝污染去重检查。相反，关闭评论指出功能已经存在的位置。

流程：

1. 维护者判断功能请求超出范围
2. 检查是否已存在匹配的 `.out-of-scope/` 文件
3. 如果是：将新 issue 追加到"Prior requests"列表
4. 如果否：创建一个新文件，包含概念名称、决策、理由和第一个先前的请求
5. 在 issue 上发布一条评论，说明决策并提及 `.out-of-scope/` 文件
6. 用 `wontfix` 标签关闭 issue

## 更新或删除范围外文件

如果维护者改变了关于之前被拒绝概念的想法：

- 删除 `.out-of-scope/` 文件
- 该 skill 不需要重新打开旧的 issue — 它们是历史记录
- 触发重新考虑的新 issue 进入正常分类流程
