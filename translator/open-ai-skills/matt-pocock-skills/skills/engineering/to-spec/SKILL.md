此技能读取当前对话上下文和代码库理解，产出一份 spec（你可能也称之为 PRD）。不要访谈用户 —— 仅综合你已经知道的内容。

Issue tracker 和 triage 标签词汇表应已提供给你 —— 如果没有，运行 `/setup-matt-pocock-skills`。

## 流程

1. 探索仓库以了解代码库的当前状态（如果尚未这样做）。在整个 spec 中使用项目的领域词汇表，并尊重所涉及区域的任何 ADR。

2. 草拟你将用于测试该功能的接缝（seam）。优先使用现有接缝而不是新建。使用尽可能高层的接缝。如果需要新接缝，在尽可能高的层级提出。代码库中的接缝越少越好 —— 理想数量是 1 个。

与用户确认这些接缝是否符合他们的期望。

3. 使用以下模板编写 spec，然后发布到项目 issue tracker。应用 `ready-for-agent` triage 标签 —— 无需额外 triage。

<spec-template>

## 问题陈述

从用户视角描述用户面临的问题。

## 解决方案

从用户视角描述问题的解决方案。

## 用户故事

一个详细的、编号的用户故事列表。每个用户故事格式为：

1. 作为 <角色>，我希望 <功能>，以便 <收益>

<user-story-example>
1. 作为手机银行客户，我希望查看账户余额，以便做出更明智的消费决策
</user-story-example>

用户故事列表应极其详尽，涵盖该功能的所有方面。

## 实现决策

已做出的实现决策列表。可包含：

- 将构建/修改的模块
- 这些模块将被修改的接口
- 开发者的技术澄清
- 架构决策
- Schema 变更
- API 契约
- 具体交互

不要包含具体文件路径或代码片段。它们可能很快过时。

例外：如果原型产生了一个代码片段，它比文字更精确地编码了一个决策（状态机、reducer、schema、类型结构），将其内联在相关决策中，并简要注明来自原型。精简到富含决策的部分 —— 不是可运行的演示，只是关键部分。

## 测试决策

已做出的测试决策列表。包含：

- 什么构成好测试的描述（只测试外部行为，不测试实现细节）
- 哪些模块将被测试
- 测试的先例（即代码库中类似类型的测试）

## 超出范围

描述此 spec 超出范围的内容。

## 补充说明

关于该功能的任何补充说明。

</spec-template>
