## 功能说明

`tdd` 以测试优先的方式构建功能或修复 bug，一次一个行为，通过红-绿循环驱动代码产出。

它**不会**一次性写完所有测试。批量编写测试（"水平切片"）会产生基于*想象中*行为的测试——它们检查的是代码的形态，对真正的变更会变得麻木。`tdd` 则采用垂直切片：一个测试，然后是刚好让它通过的代码，接着下一个测试，每个循环都由上一个循环教给你的东西来指导。测试仅针对公共接口，因此底层实现可以变更而测试无需随之改动。

## 何时使用

输入 `/tdd`，或者当任务匹配时 Agent 会自动调用它——以测试优先的方式构建功能或修复 bug，或者当你说"红-绿-重构"时。

当有具体行为需要构建，并且希望测试能在重构中存活下来时，使用它。如果行为尚未确定，先敲定规范——此时应使用 [to-spec](https://aihero.dev/skills-to-spec)。当工作重点更偏向接口形态而非测试时，使用 [codebase-design](https://aihero.dev/skills-codebase-design)；`tdd` 在规划阶段会调用它来获取深度模块的术语体系。

## 红-绿，一次一个切片

核心理念是**红-绿循环**：写一个失败的测试（红），添加刚好让它通过的代码（绿），然后对下一个行为重复——每个循环都由上一个循环教给你的东西来指导。第一个循环是一个**曳光弹**：一个证明单一路径端到端可工作的测试，然后你再从它向外构建。由于你刚刚写了代码，你清楚地知道哪个行为重要以及如何验证它——你永远不会因为承诺了自己还不了解的测试结构而跑到视野之外。

两条规则保持测试诚实。一个好的测试读起来像一份规范（"用户可以使用有效购物车结账"），并通过公共 API 执行真实的代码路径，因此重命名内部函数不会破坏它。而期望值来自独立的真实来源——一个已知正确的字面量、一个手工演算的示例、规范本身——绝不以代码计算的方式重新计算，那正是**同义反复**测试的成因：仅凭构造方式就能通过，却什么也告诉不了你。

重构只在整套测试变绿之后进行；绝不在红色状态下重构。

## 有效的标志

- 它写一个测试，让它通过，然后才写下一个——而不是一批测试之后跟着一批代码。
- 测试命名的是行为而非内部实现，且在内部重命名时不受影响。
- 期望值来自规范中的字面量，而非与代码相同的推导方式得出的数字。

## 在系统中的位置

`tdd` 是主构建链运行以编写代码的红-绿循环：

```txt
grill-with-docs → to-spec → to-tickets → implement → code-review
```

[implement](https://aihero.dev/skills-implement) 是构建链的构建步骤，它在内部驱动 `tdd` 以测试优先的方式构建每个 ticket，然后交给 [code-review](https://aihero.dev/skills-code-review)——因此 `tdd` 是该步骤内部的引擎，而非独立的步骤。你也可以直接使用它，每当你有一个无需完整规范即可构建的具体行为时。它的另一个邻近技能是 [codebase-design](https://aihero.dev/skills-codebase-design)，它依赖该技能来找到值得测试的深度模块接缝。当你不确定哪个技能或流程适合时，[ask-matt](https://aihero.dev/skills-ask-matt) 会为你导航。
