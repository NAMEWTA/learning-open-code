# 可观测性与仪表化

## 概述

你无法观测的代码就是你无法运维的代码。可观测性是使用代码发出的遥测数据，从外部回答"系统在做什么以及为什么"的能力。仪表化不是上线后的附加项——它与功能一起编写，就像测试一样。如果一个功能在没有遥测数据的情况下上线，第一个用户报告的 bug 就变成了考古学，而不是一次查询。

## 何时使用

- 构建任何将在生产中运行的功能
- 添加新的服务、端点、后台作业或外部集成
- 一次生产事故花费太长时间来诊断（"我们无法判断发生了什么"）
- 设置或审查告警规则
- 审查添加了 I/O、重试、队列或跨服务调用的 PR

**不适用于：**
- 诊断正在发生的故障——使用 `debugging-and-error-recovery` 技能（可观测性是让该技能下次变快的东西）
- 分析和优化已测量的缓慢——使用 `performance-optimization` 技能
- 上线日监控清单和回滚触发器——参见 `shipping-and-launch` 技能；本技能涵盖驱动它们的仪表化

## 流程

### 1. 在仪表化之前定义"正常工作"

没有问题的遥测只是噪音。在添加任何仪表化之前，写下值班工程师会问的关于此功能的 2-4 个问题：

```
功能：结账支付重试
值班工程师会问的问题：
1. 多大比例的支付在首次尝试时成功 vs. 重试后成功？
2. 当支付永久失败时，为什么？（提供商错误？超时？验证？）
3. 支付提供商是否比平时慢？
→ 下面的每个信号必须帮助回答其中一个问题。
```

如果你说不出这些问题，你还没有准备好进行仪表化——你会记录一切却一无所获。

### 2. 为每个问题选择正确的信号

| 信号 | 回答 | 成本特征 | 示例 |
|---|---|---|---|
| **结构化日志** | "在这个具体案例中发生了什么？" | 按事件计；随流量增长 | `payment_failed` 带提供商错误代码 |
| **指标** | "多频繁 / 多快，总体而言？" | 按系列固定；查询成本低 | 提供商调用的 p99 延迟 |
| **链路追踪** | "跨服务的时间花在哪里？" | 按请求计；通常采样 | 一个慢的结账，按跳分解 |

经验法则：指标告诉你**有**问题，链路追踪告诉你**在哪里**，日志告诉你**为什么**。

### 3. 结构化日志

记录事件，而不是散文。每行日志是一个 JSON 对象，带有稳定的事件名称和机器可读的字段：

```typescript
// 差：字符串插值 —— 不可查询，不一致
logger.info(`Payment ${id} failed for user ${userId} after ${n} retries`);

// 好：稳定的事件名称 + 结构化字段
logger.warn({
  event: 'payment_failed',
  paymentId: id,
  provider: 'stripe',
  errorCode: err.code,
  attempt: n,
}, 'payment failed');
```

**日志级别 —— 一致使用：**

| 级别 | 含义 | 值班行动 |
|---|---|---|
| `error` | 不变量被破坏；可能有人需要行动 | 调查 |
| `warn` | 降级但已处理（重试成功、使用了回退） | 观察趋势 |
| `info` | 重要的业务事件（订单已下、作业已完成） | 无 |
| `debug` | 诊断细节 | 生产环境默认关闭 |

**关联 ID 是强制性的。** 在系统边界生成（或接受）一个请求 ID，并将其附加到每一行日志、每个 span 和每个出站调用。没有它，你无法从交错的日志中重建单个请求：

```typescript
// Express：每个请求的子 logger，ID 传播到下游
app.use((req, res, next) => {
  req.id = req.headers['x-request-id'] ?? crypto.randomUUID();
  req.log = logger.child({ requestId: req.id });
  res.setHeader('x-request-id', req.id);
  next();
});
```

**永远不要记录密钥、令牌、密码或完整的 PII。** 这是来自 `security-and-hardening` 技能的硬性规则——遥测管道是经典的数据泄漏路径。使用允许列表字段；不要记录整个请求体。

### 4. 指标

对于请求驱动的服务，在每个端点和每个外部依赖上实施 **RED**：**R**ate（请求/秒）、**E**rrors（失败率）、**D**uration（延迟直方图，而非平均值）。对于资源（队列、池、主机），使用 **USE**：**U**tilization（利用率）、**S**aturation（饱和度）、**E**rrors（错误）。

与链路追踪一样，供应商中立的路径是 OpenTelemetry 指标 API（与步骤 5 相同的 SDK 和上下文）。下面的示例使用 Prometheus 的 `prom-client`——一个常见的后端选择，但不是唯一选择；RED/USE 和基数规则无论如何都是相同的。

```typescript
import { Histogram } from 'prom-client';

const httpDuration = new Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request duration',
  labelNames: ['method', 'route', 'status_class'],  // '2xx'，而非 '200'
  buckets: [0.05, 0.1, 0.25, 0.5, 1, 2.5, 5],
});
```

**基数是失败模式。** 每个唯一的标签组合都是一个单独的时间序列。标签必须来自小的、固定的集合（路由模板、状态类、提供商名称）。永远不要使用用户 ID、原始 URL、错误消息或其他无界值作为标签——这些属于日志和链路追踪。

```
可以作为标签：  route="/api/tasks/:id"   status_class="5xx"   provider="stripe"
绝不作为标签：  user_id、email、request_id、完整 URL、错误消息文本
```

永远不要追踪平均值，永远追踪百分位值：平均值隐藏了那 1% 正在遭受糟糕体验的用户。使用直方图并读取 p50/p95/p99。

### 5. 分布式链路追踪

使用 OpenTelemetry——它是供应商中立的标准，自动仪表化覆盖 HTTP、gRPC 和常见数据库客户端，几乎零代码：

```typescript
// tracing.ts —— 必须在任何其他代码之前导入
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';

const sdk = new NodeSDK({
  serviceName: 'checkout-service',
  instrumentations: [getNodeAutoInstrumentations()],
});
sdk.start();
```

仅在有意义的工作单元周围添加手动 span（例如 `applyDiscounts`、`chargeProvider`），并附加上值班人员将用来过滤的属性。在每一个异步边界上传播上下文——HTTP 头、队列消息元数据——否则链路追踪会在间隙处断裂。默认以低比率进行头部采样；如果你的后端支持尾部采样，保留 100% 的错误。

### 6. 告警

告警针对**用户感受到的症状**，而不是原因：

```
症状（值得通知）：               原因（仪表盘，不通知）：
错误率 > 1% 持续 5 分钟         CPU 在 85%
p99 延迟 > 2 秒                 一个 pod 重启
队列积压 > 10 分钟              磁盘在 70%
```

基于原因的告警在没有问题时也会触发，并错过你没有预测到的故障。基于症状的告警恰好在用户受到影响时触发，无论原因是什么。

你创建的每个告警的规则：

1. **它必须是可操作的。** 如果响应是"忽略它，它会自愈"，删除该告警。
2. **它链接到一个 runbook**——哪怕只有三行：含义、第一个要运行的查询、升级路径。
3. **它有由 SLO 或历史数据证明的阈值和持续时间**，而不是凭猜测。
4. 只使用两种严重级别：**page**（面向用户，立即行动）和 **ticket**（降级，本周内行动）。第三个级别会变成训练人们忽略一切的噪音。

### 7. 验证遥测本身

仪表化是代码；它可能是错的。在宣布工作完成之前，触发这些路径并查看实际输出：

- 在 staging 中强制一个错误 → 通过 `requestId` 在日志中找到它，确认字段是结构化的（不是 `[object Object]`）
- 发送测试流量 → 确认指标系列以预期标签和合理的值出现
- 在链路追踪 UI 中跨服务追踪一个请求 → 没有断裂的 span
- 触发每个新告警一次（暂时降低阈值） → 确认它到达正确的通道并且 runbook 链接有效

## 常见合理化借口

| 合理化借口 | 现实 |
|---|---|
| "等工作后我再添加日志" | "之后"变成"第一次事故之后"，那是发现你处于盲区的最昂贵时刻。边构建边仪表化。 |
| "更多日志 = 更多可观测性" | 非结构化的噪音让事故处理更慢，而不是更快。三个可查询的事件胜过三百行散文。 |
| "console.log 暂时够用" | 非结构化输出无法被过滤、关联或告警。结构化 logger 多花五分钟一次性搞定。 |
| "出问题时我们看仪表盘就行" | 没有定义问题就构建的仪表盘展示了一切，唯独没有答案。从值班问题开始。 |
| "所有重要的都告警，我们以后调优" | 嘈杂的通知训练人们忽略它。调优从未发生；真正的漏报确实发生了。 |
| "用户 ID 作为指标标签让调试更容易" | 它也会让你的指标后端宕机。高基数的查找属于日志和链路追踪。 |
| "链路追踪对我们两个服务来说太过了" | 两个服务已经意味着跨服务延迟问题是日志无法回答的。自动仪表化使成本微不足道。 |

## 红旗信号

- 一个带有重试、队列或外部调用的功能 PR 却没有新的遥测数据
- 通过字符串插值构建的日志行，而非结构化字段
- 没有关联/请求 ID——每行日志都是孤立的
- 指标标签带有用户 ID、原始 URL 或错误消息文本（基数炸弹）
- 延迟以平均值追踪，没有百分位值
- 每天触发并被确认而不采取行动的告警
- 基于原因（CPU、内存）的告警在通知人类，而面向用户的错误率未被监控
- 密钥、令牌或完整请求体出现在日志中
- "在我机器上工作"作为生产功能健康的唯一证据

## 验证

为一个功能添加仪表化后，确认：

- [ ] 此功能的值班问题已写下来，每个信号映射到一个问题
- [ ] 所有日志输出是结构化的（JSON），带有稳定的事件名称，每一行都有关联 ID
- [ ] 任何日志行中没有密钥、令牌或未脱敏的 PII（抽查实际输出）
- [ ] 每个新端点和每个外部依赖都存在 RED 指标，标签集合有界
- [ ] 延迟是直方图；p95/p99 可查询
- [ ] 单个请求可以在链路追踪 UI 中端到端追踪，没有断裂的 span
- [ ] 每个新告警是基于症状的，有 runbook 链接，并至少测试触发过一次
- [ ] 在 staging 中诱导的故障仅通过遥测数据定位，无需阅读源代码
