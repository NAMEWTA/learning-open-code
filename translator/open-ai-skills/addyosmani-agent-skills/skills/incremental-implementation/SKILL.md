# 增量实现

## 概述

以薄纵向切片的方式构建——实现一小部分，测试它，验证它，然后扩展。避免一次性实现整个功能。每个增量都应将系统留在可工作、可测试的状态。这是让大型功能可控的执行纪律。

## 适用场景

- 实现任何多文件变更
- 从任务拆解构建新功能
- 重构现有代码
- 任何你忍不住一次写超过约 100 行代码而不测试的时候

**不适用场景：** 单文件、单函数变更，范围已经足够小。

## 增量循环

```
┌──────────────────────────────────────┐
│                                      │
│   实现 ──→ 测试 ──→ 验证 ──┐         │
│     ▲                      │         │
│     └───── 提交 ◄───────────┘         │
│              │                        │
│              ▼                        │
│          下一个切片                    │
│                                      │
└──────────────────────────────────────┘
```

对于每个切片：

1. **实现** 最小完整的功能片段
2. **测试** —— 运行测试套件（如果没有测试就写一个）
3. **验证** —— 确认该切片按预期工作（测试通过、构建成功、手动检查）
4. **提交** —— 用描述性消息保存进度（原子提交指导参见 `git-workflow-and-versioning`）
5. **进入下一个切片** —— 继续推进，不要重启

## 切片策略

### 纵向切片（推荐）

构建覆盖整个技术栈的一条完整路径：

```
切片 1：创建任务（DB + API + 基础 UI）
    → 测试通过，用户可通过 UI 创建任务

切片 2：列出任务（查询 + API + UI）
    → 测试通过，用户可查看自己的任务

切片 3：编辑任务（更新 + API + UI）
    → 测试通过，用户可修改任务

切片 4：删除任务（删除 + API + UI + 确认）
    → 测试通过，完整的 CRUD 完成
```

每个切片交付可工作的端到端功能。

### 契约优先切片

当后端和前端需要并行开发时：

```
切片 0：定义 API 契约（类型、接口、OpenAPI 规范）
切片 1a：根据契约实现后端 + API 测试
切片 1b：根据匹配契约的 mock 数据实现前端
切片 2：集成并进行端到端测试
```

### 风险优先切片

优先处理风险最高或最不确定的部分：

```
切片 1：证明 WebSocket 连接能工作（最高风险）
切片 2：在已验证的连接上构建实时任务更新
切片 3：添加离线支持与重连
```

如果切片 1 失败，你会在投资切片 2 和切片 3 之前就发现它。

## 实现规则

### 规则 0：简单优先

在编写任何代码之前，先问："可以工作的最简单方案是什么？"

写完代码后，对照以下检查进行审查：
- 能用更少的行数实现吗？
- 这些抽象是否值得它们带来的复杂度？
- 资深工程师看了会说什么——"你为什么不直接……？"
- 我是在为假设的未来需求而构建，还是在为当前任务构建？

```
简单性检查：
✗ 为一次通知构建泛型 EventBus 加中间件管道
✓ 简单的函数调用

✗ 为两个类似组件使用抽象工厂模式
✓ 两个直截了当的组件加共享工具函数

✗ 为三个表单构建配置驱动的表单生成器
✓ 三个表单组件
```

三行相似的代码好过一次过早的抽象。先实现朴素的、明显正确的版本。直到测试证明了正确性再进行优化。

### 规则 0.5：范围纪律

只触碰任务要求的。

不要：
- "清理"与你变更相邻的代码
- 重构未修改文件中的 import
- 删除你不完全理解的注释
- 添加规范中没有的功能，因为"好像有用"
- 对仅阅读的文件进行语法现代化

如果注意到任务范围之外值得改进的地方，记录下来——不要修复它：

```
注意到但未触碰：
- src/utils/format.ts 有未使用的 import（与本任务无关）
- auth 中间件可以有更好的错误消息（独立任务）
→ 需要我为这些问题创建任务吗？
```

### 规则 1：一次只做一件事

每个增量只改变一件逻辑上的事情。不要混在一起：

**错误：** 一个提交包含新增组件、重构现有组件和更新构建配置。

**正确：** 三个独立提交——每个变更一个。

### 规则 2：保持可编译

每个增量之后，项目必须成功构建，现有测试必须通过。不要在切片之间把代码库留在损坏状态。

### 规则 3：未完成功能使用特性开关

如果功能还没准备好给用户使用，但你需合并增量代码：

```typescript
// Feature flag for work-in-progress
const ENABLE_TASK_SHARING = process.env.FEATURE_TASK_SHARING === 'true';

if (ENABLE_TASK_SHARING) {
  // New sharing UI
}
```

这让你可以合并小的增量到主分支，而不暴露未完成的工作。

### 规则 4：安全默认值

新代码应默认采用安全、保守的行为：

```typescript
// Safe: disabled by default, opt-in
export function createTask(data: TaskInput, options?: { notify?: boolean }) {
  const shouldNotify = options?.notify ?? false;
  // ...
}
```

### 规则 5：易于回滚

每个增量应能独立回滚：

- 新增性变更（新文件、新函数）易于回滚
- 对现有代码的修改应该尽量少且聚焦
- 数据库迁移应有对应的回滚迁移
- 避免在一个提交中删除某物又替换它——分开处理

## 与 Agent 协作

当指示 agent 增量实现时：

```
"让我们实现计划中的任务 3。

仅从数据库模式变更和 API 端点开始。
先不要碰 UI —— 放到下一个增量中。

实现完成后，运行 `npm test` 和 `npm run build` 验证
没有东西被破坏。"
```

明确说明每个增量中哪些在范围内、哪些不在范围内。

## 增量检查清单

每个增量之后验证：

- [ ] 变更只做了一件事且完整做完
- [ ] 所有现有测试仍然通过（`npm test`）
- [ ] 构建成功（`npm run build`）
- [ ] 类型检查通过（`npx tsc --noEmit`）
- [ ] Linting 通过（`npm run lint`）
- [ ] 新功能按预期工作
- [ ] 变更已用描述性消息提交

**注意：** 在每次可能影响结果的代码变更后运行各验证命令。一次成功运行后，除非代码有变，不要重复相同命令——在未变更的代码上重新运行不会增加任何信息。

## 常见借口

| 借口 | 事实 |
|---|---|
| "我最后再一起测试" | Bug 会叠加。切片 1 中的 bug 会导致切片 2-5 出错。每个切片都要测试。 |
| "一次性做完更快" | 这样*感觉*更快——直到出了问题时你无法从 500 行变更中找出是哪一行引起的。 |
| "这些变更太小了不值得单独提交" | 小的提交是免费的。大型提交隐藏 bug 并让回滚变得痛苦。 |
| "我后面再加特性开关" | 如果功能未完成，就不应该对用户可见。现在就加开关。 |
| "这次重构足够小，可以一起提交" | 重构与功能混在一起让两者都更难审查和调试。分开处理。 |
| "让我再运行一次构建命令以防万一" | 一次成功运行后，除非代码已经变更，重复相同命令不会增加任何信息。在后续编辑后再运行，而不是为了安心而重复。 |

## 危险信号

- 编写了超过 100 行代码却没有运行测试
- 单个增量中包含多个不相关的变更
- "让我顺便把这个也加上"的范围蔓延
- 为加快速度而跳过测试/验证步骤
- 增量之间构建或测试处于损坏状态
- 大量未提交的变更在积累
- 在第三个用例需要之前就构建抽象
- "顺便"触碰到任务范围之外的文件
- 为一次性操作创建新的工具文件
- 在没有任何代码变更的情况下连续两次运行相同的构建/测试命令

## 验证

完成一个任务的所有增量之后：

- [ ] 每个增量都经过独立测试和提交
- [ ] 完整测试套件通过
- [ ] 构建干净
- [ ] 功能按规范端到端工作
- [ ] 无未提交的变更遗留

## 参见

每个增量的验证是局部检查。在宣布任务完成之前，将项目级的完成定义作为最终门禁来执行——这是每个增量都要达到的基准标准，无论任务是哪个。参见 `references/definition-of-done.md`。
