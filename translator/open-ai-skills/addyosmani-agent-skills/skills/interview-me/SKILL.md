# Interview Me

## 概述

人们要求的东西和他们真正想要的东西是不同的。他们要求"一个仪表盘"，因为那是人们通常会要求的东西，而不是因为仪表盘能解决他们的问题。他们说"让它更快"却没有一个要达成的数字。

发现这个差距的最便宜时机是在任何计划、规格或代码存在之前。一旦你开始构建，切换成本是真实的，用户会将错误的东西合理化成一个"足够好"的东西。不匹配被锁定了。

这个技能在它造成任何代价之前缩小这个差距。其他定义阶段的技能假设你已经大致知道你想要什么：`idea-refine` 从一个想法生成变体，`spec-driven-development` 将需求写下来，`doubt-driven-development` 在你起草计划后对其进行压力测试。interview-me 是在所有这些之前的部分，你一次问一个问题，附上你的最佳猜测，直到你可以预测用户在说出来之前会说什么。

## 何时使用

在以下情况应用此技能：

- 需求至少缺少以下之一：**谁**是用户，**为什么**他们想要它，**成功**是什么样的，绑定的**约束**是什么
- 请求是约定俗成的而非具体的（"帮我构建 X"，"让它更快"），你无法在不猜测的情况下解构这个约定
- 你倾向于从你尚未揭示的假设开始
- 用户没有说明他们正在优化哪个价值，而两个合理的价值之间存在张力（简单 vs. 灵活，成本 vs. 速度）
- 用户明确调用："interview me"、"grill me"、"before we start, are we sure?"、"stress-test my thinking"

**何时不使用：**

- 需求明确且自包含（"重命名这个变量"、"修复这个拼写错误"）
- 用户明确要求速度而非验证
- 纯信息请求（"X 是如何工作的？"、"这段代码做什么？"）
- 机械操作（重命名、格式化、文件移动）
- 你已经有 ≥95% 的置信度；在假设你没有之前重新阅读下面的停止条件

## 加载约束

此技能需要一个活跃的、可响应的用户。**不要在非交互式上下文中调用**，如 CI 流水线、定时运行、`/loop` 或自主循环。如果你在其中一个上下文中并且需求不明确，将其标记为用户的阻塞点，而不是猜测。

## 流程

### 步骤 1：假设，附带置信度数字

在问任何问题之前，用**一句话**写下你当前对用户想要什么的最佳解读，加上一个诚实的置信度数字（0-100%）：

```
HYPOTHESIS: 你想要一种在站会中回答"我们做得怎么样？"的方式，而"仪表盘"是你想到的约定俗成的答案。
CONFIDENCE: ~30% —— 缺少：为谁做的、"指标"在上下文中意味着什么、成功是什么样的
```

数字强迫诚实。如果你写了一个高数字但不能实际预测用户对你接下来要问的三个问题的反应，这个数字就是错的。从你能自圆其说的置信度水平开始。

当置信度低于约 70% 时，在同一行附上一个简短的原因——还有什么尚未解决或缺失。这告诉用户访谈需要揭示什么，并防止数字成为一个模糊的信号。

### 步骤 2：一次问一个问题，每个问题附带一个猜测

格式：

```
Q: <一个聚焦的问题>
GUESS: <你对答案的假设，以及产生它的推理>
```

在问下一个问题之前等待用户反应。

**为什么一次一个，而不是一批：**

- 如果你把假设埋在列表里，用户无法对你的假设做出反应
- 批量鼓励略读和表面回答
- 第三个问题通常取决于第一个问题的答案；一次性全问会锁定错误的框架
- 用户仔细思考的精力是有限的；一次一个问题地花费它

**为什么附带一个猜测：**

- 用户对一个错误猜测的反应比他们从头生成答案更快
- 它让你承诺一个你可以被明显证明是错误的假设，这让你保持诚实
- 它揭示了*你*的假设，这正是访谈旨在暴露的

这里的风险是一个礼貌的用户同意你的猜测以表示配合。通过明显愿意犯错来缓解，偶尔朝你预期用户会反对的方向猜测。

### 步骤 3：倾听"想要 vs. 应该想要"

最危险的答案是用户说出一个深思熟虑的回答*听起来像什么*而不是他们真正想要的。注意：

- 模式匹配最佳实践话语的答案（"我希望它是可扩展的"、"整洁的架构"）但没有具体细节
- 遵从惯例的答案（"大多数应用的做法"、"标准方法"）
- 像"我可能应该……"、"我想我应该……"、"好的工程实践说……"这样的短语
- 流行语作为目标——当"现代的"、"可扩展的"、"健壮的"是答案而不是具体结果时

当你听到这些时，要问的问题是：

> *"如果你不需要为此向任何人解释，你真正想要什么？"*

这一个问题通常比前五个问题做得更多。

### 步骤 4：用用户自己的话重述意图

当你的置信度很高时，写回你现在认为用户想要的。保持紧凑（5-8 行），尽可能使用他们的语言，并结构化以便用户可以逐行确认或纠正：

```
以下是我现在认为你想要的：

- 结果：        <一行>
- 用户：        <一行 —— 谁受益>
- 为什么现在：  <一行 —— 什么改变了>
- 成功：        <一行 —— 我们如何知道它奏效了>
- 约束：        <一行 —— 绑定的限制>
- 超出范围：    <一行 —— 我们明确不做的>

是 / 否 / 细化？
```

包含"超出范围"是不可协商的。一半的不对齐是对于*不*构建什么的沉默分歧。

### 步骤 5：确认——明确的"是"，而非"你觉得好就行"

门槛是一个明确的"是"。以下**不是**是：

- "你觉得好就行。" → 用户在委托，这意味着他们也没有 95% 的置信度。重新问，用两个具体选项构建为一个选择。
- "听起来不错。" → 模糊。问："有什么你想细化的吗？"沉默不是确认。
- "当然，开始吧。" → 通常是一个礼貌的退出，而不是认可。同样的后续跟进。
- 沉默之后"好的开始吧。" → 用户已经放弃了访谈，而不是收敛了。停下来问你是否错过了什么。

如果他们纠正你，将纠正纳入并重新陈述。循环直到你得到一个明确的"是"。

### 95% 置信度停止点

当你能对以下问题回答"是"时，你就完成了：

> *我能预测用户对我接下来要问的三个问题的反应吗？*

如果是，你有了共同理解。停止访谈并产出重述。如果否，你还没完成；问下一个问题。

这是一个可检验的测试，不是一种感觉。它也有一个底线：如果你已经进行了好几轮仍然无法预测，那是关于需求的信息，而不是继续磨下去的理由。停下来告诉用户："我已经问了 X 个问题，我仍然无法预测你的反应。一些基础性的东西缺失了。想要退一步吗？"

## 输出

此技能的输出是一个**已确认的意图声明**：步骤 4 的重述，加上步骤 5 的明确"是"。这就是交付物。规格、计划和任务列表是下游的；它们消费此技能产生的意图。

如果用户希望意图持久化（一个多会话项目、向另一个合作者移交），提议将其保存到 `docs/intent/[topic].md`。仅在用户确认后保存。

## 示例

一个简短的对比。

**没有 `interview-me`：**

```
用户:  帮我构建一个我们指标的仪表盘
Agent: <开始提议图表库和布局>
```

Agent 已经假设了：为谁做的、什么指标、"为我们的团队"意味着什么、成功是什么样的。这些都没有被陈述。每一个假设都是用户后来可以说"那不是我的意思"的地方。

**有 `interview-me`：**

```
用户: 帮我构建一个我们指标的仪表盘

Agent:
HYPOTHESIS: 你想要一种在站会中回答"我们做得怎么样？"的方式，而"仪表盘"是你想到的约定俗成的答案。
CONFIDENCE: ~30% —— 缺少：为谁做的、"指标"是什么、成功是什么样的

Q:     当你说"我们做得怎么样？"，谁在问——你一个人、工程团队在站会上、还是向上汇报（例如，你的经理、高管、董事会）？
GUESS: 工程团队在站会上，因为"我们"通常这样界定范围，站会是问这个问题的地方。如果是为高管做的，指标和框架会变化很大。

用户: 实际上是为我自己。我一直忘了哪些实验在跑以及它们的早期信号是什么。

Agent:
更新解读: 根本不是团队仪表盘。是一个个人实验跟踪器。置信度: ~60% —— 仍然缺少："早期信号"意味着什么、完成是什么样的。

Q:     差距是你不清楚有哪些实验存在，还是你无法在一个地方看到它们的结果？
GUESS: 第二个。你在某个地方有一份实验列表，但它们的结果分散在五个不同的工具中，手工协调这些是你厌倦的事情。

用户: 实际上是第一个。我确实没有列表。它们分散在不同的文档中。

Agent: <继续>
```

两个问题之后，Agent 发现实际需求不是"一个仪表盘"。是"一个列表"。不同的产物，不同的范围，不同的工作量。那个仪表盘本来是错误的。

## 与其他技能的交互

- **`idea-refine`**：下游。如果已确认的意图是"我想要 X 但我不知道如何界定其范围"，移交给 `idea-refine` 以针对现在明确的意图生成变体。
- **`spec-driven-development`**：下游。如果已确认的意图是具体的（"我想要 X 为 Y 用户带有 Z 成功标准"），移交给 `spec-driven-development` 以将其写下来。
- **`planning-and-task-breakdown`**：此技能下游两跳（在规格之后）。
- **`doubt-driven-development`**：时间线的另一端。interview-me 是决策前的意图提取；doubt-driven 是决策后的产物审查。两者都捕获偏离，但在不同的时刻。
- **`source-driven-development`**：正交。interview-me 澄清用户想要什么；SDD 验证框架事实。它们不竞争。

## 常见合理化借口

| 合理化借口 | 现实 |
|---|---|
| "需求够清晰了" | 如果你现在不能用一句话写出用户想要的结果，需求就不清晰。在决定之前执行步骤 1。 |
| "问太多问题浪费他们的时间" | 4-6 个针对性问题浪费的时间很少。构建错误的东西浪费的时间是巨大的，而用户是承担那个成本的人。 |
| "我会在构建过程中搞清楚" | 代码存在后的切换成本是现在的 10 倍。实现过程中的发现是返工。 |
| "他们说了'你觉得好就行'，所以我应该直接决定" | "你觉得好就行"是委托，不是决定。重新问，用两个具体选项构建为一个选择。 |
| "我应该给他们几个选项来选择" | 当用户知道他们想要什么并在权衡之间选择时，选项有效。他们还不知道他们想要什么。列出选项拓宽搜索；提问缩小搜索。 |
| "如果我附上我的猜测，我在引导他们" | 引导是重点。反应比从头生成更快。风险是谄媚，不是引导；通过明显愿意犯错来缓解。 |
| "我们已经聊够了，我懂了" | 测试一下：你能预测他们对接下来三个问题的反应吗？如果不能，你还没懂。 |
| "用户说了是，我们完成了" | 如果这个"是"跟在一个模糊的重述或一个开放式的"听起来不错"之后，这个"是"是空洞的。具体重述并重新确认。 |

## 红旗信号

- 一条消息中包含三个或更多问题：那是批量处理，不是访谈
- 一个没有附带假设的问题：那是在调查，不是在承诺
- 接受"你觉得好就行"作为最终答案
- 在用户明确确认你的重述之前产出一个规格、计划或任务列表
- 问题框架为"最佳实践是什么？"而不是"你真正想要什么？"
- 用户给出了一个表现复杂度的信号答案（"可扩展的"、"整洁的"、"现代的"），而你没有探究这是否是他们真正想要的
- 三轮或更多轮后你的置信度没有明显上升：你在问错误的问题，退一步重新框架
- 置信度数字低于约 70% 而没有附带原因：如果用户不知道缺少什么，他们无法帮助缩小差距
- 在用户确认之前保存意图文档（文档本身暗示了一个用户没有给出的"是"）
- 在重述中跳过"超出范围"这一行（对非目标的沉默分歧是一半的不对齐）

## 验证

应用 interview-me 后：

- [ ] 在第一轮中陈述了一个明确的假设和置信度数字
- [ ] 每个低于约 70% 的置信度数字都附带了一行原因（什么尚未解决或缺失）
- [ ] 问题一次一个，每个附带 agent 的猜测
- [ ] 当用户给出了一个表现复杂度信号或遵从惯例信号的答案时，至少运行了一次"如果你不需要为此解释，你真正想要什么？"的探究
- [ ] 一个具体的重述（结果 / 用户 / 为什么现在 / 成功 / 约束 / 超出范围）被写回给用户
- [ ] 用户以明确的"是"确认了重述（不是"你觉得好就行"，不是"听起来不错"，不是沉默）
- [ ] 在停止点，agent 可以预测对其接下来会问的三个问题的反应
- [ ] 向下游技能（`idea-refine`、`spec-driven-development`）的任何移交都以已确认的意图来框架，而非原始的不明确需求
