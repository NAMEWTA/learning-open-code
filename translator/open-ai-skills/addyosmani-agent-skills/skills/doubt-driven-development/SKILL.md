# 质疑驱动开发

## 概述

自信的答案不等于正确的答案。长时间的会话会积累上下文，悄悄将假设变成"事实"而不被察觉。质疑驱动开发是一门纪律：在任何非平凡输出生效之前，引入一个全新上下文的评审者——偏重于**证伪**，而非通过。

这不是 `/review`。`/review` 是对已完成的产物做出裁决。这是一种进行中的姿态：在纠偏成本仍然很低的时候，非平凡决策就接受交叉审查。

## 适用场景

当决策满足以下至少一项时，即为**非平凡**：

- 引入或修改分支逻辑
- 跨越模块或服务边界
- 断言了类型系统或编译器无法验证的属性（线程安全、幂等性、顺序性、不变量）
- 其正确性依赖于未来阅读者无法看到的上下文
- 其影响范围不可逆（生产部署、数据迁移、公共 API 变更）

在以下情况下应用此技能：

- 即将在不确定性下做出架构决策
- 即将提交非平凡代码
- 即将断言一个非显然的事实（"这是安全的"、"这具备可扩展性"、"这匹配规格"）
- 在你尚未完全理解的代码中工作

**不适用场景：**

- 机械操作（重命名、格式化、文件移动）
- 执行清晰、明确的人类指令
- 阅读或总结已有代码
- 正确性显然的单行更改
- 纯工具操作（运行测试、列出文件）
- 用户明确要求速度优先于验证

如果你每个按键都怀疑，你什么都交付不了。此技能仅适用于上文定义的非平凡决策。

## 加载约束

此技能设计用于**主会话编排器**，其中步骤 3（DOUBT，详见下文）可以生成一个全新上下文的评审者。

- **不要将此技能添加到 Persona 的 `skills:` frontmatter 中。** 遵循步骤 3 的 Persona 会生成另一个 Persona——这是 `references/orchestration-patterns.md`（"personas do not invoke other personas"）明确禁止的编排反模式。
- **如果你发现自己正在子 agent 上下文中应用此技能**（Claude Code 禁止嵌套子 agent 生成）：推荐做法是向用户暴露质疑驱动无法嵌套运行，让主会话处理。仅在最后手段下，存在一个降级的自问回退方案——将 ARTIFACT + CONTRACT 重写为一个全新的自我提示，与之前的推理做硬性心理切割，然后走步骤 1–5。这**不是全新上下文评审**（你带着自己的上下文），因此将结果标记为降级，并尽可能在可联系到用户时优先升级。

## 流程

应用此技能时复制此检查清单：

```
质疑周期：
- [ ] 步骤 1：CLAIM — 撰写主张 + 为何重要
- [ ] 步骤 2：EXTRACT — 隔离产物 + 契约，剥离推理
- [ ] 步骤 3：DOUBT — 以对抗性提示调用全新上下文评审者
- [ ] 步骤 4：RECONCILE — 对照产物文本分类每条发现
- [ ] 步骤 5：STOP — 满足停止条件（平凡发现、3 轮、或用户覆盖）
```

### 步骤 1：CLAIM — 暴露你的主张

用两到三行描述决策：

```
主张："新的缓存层在规格所描述的读密集型工作负载下是线程安全的。"
为何重要：这里的竞态条件会损坏用户数据，且在 QA 中难以检测。
```

如果你无法如此简洁地写出主张，你有的只是一个感觉，而非决策。在审查之前先暴露它。

### 步骤 2：EXTRACT — 最小可评审单元

全新上下文的评审者需要的是**产物**和**契约**，而非推导过程。

- 代码：diff 或函数——而非整个文件
- 决策：3–5 句话的提案加上它必须满足的约束条件
- 断言：主张加上据称支持它的证据（与步骤 1 的 CLAIM 块保持区分，后者是编排者正在审查的假设）

剥离你的推理。如果你交给评审者的是结论，你得到的就是对结论的验证。单元必须足够小，评审者能够一次阅读就掌握全貌——如果是 500 行的 PR，先分解。

### 步骤 3：DOUBT — 调用全新上下文评审者

评审者的提示**必须是对抗性的**。提问方式决定答案。

```
对抗性评审。找出此产物的错误之处。
假设作者过于自信。查找：
- 未声明的假设
- 未处理的边界情况
- 隐藏的耦合或共享状态
- 契约可能被违反的方式
- 这可能违反的现有约定
- 意外输入下的失败模式

不要验证。不要摘要。找出问题，或在彻底检查后
明确声明未发现任何问题。

产物：<粘贴产物>
契约：<粘贴契约>
```

**仅传递 ARTIFACT + CONTRACT。不要传递 CLAIM。** 将你的结论交给评审者会使其偏向同意。评审者必须独立判断产物是否满足契约。

在 Claude Code 中，`agents/` 中基于角色的评审者天然以隔离上下文启动，可在此使用——查看 `agents/` 获取名单及各领域匹配信息。

**上述对抗性提示优先于 Persona 的默认响应形式。** 像 `code-reviewer` 这样的 Persona 被编写为产生包含优点和缺点的平衡裁定；质疑驱动需要仅输出问题。将对抗性提示逐字粘贴到调用中，以覆盖 Persona 的默认行为。如果某个 Persona 的响应形式无法被干净地覆盖，回退到带有对抗性提示的通用子 agent。

#### 跨模型升级

单一模型的评审者与原作者共享盲点——一个更冷、架构不同的模型能发现这些盲点。质疑驱动对于非平凡决策已经是主动选择的，因此在此范围内提供跨模型选择是本技能的价值所在，而非可选的摩擦。

**交互式会话：始终提供。绝不要悄悄跳过。**

**步骤 1：询问用户**

在上述步骤 3 的单一模型评审之后，在进入 RECONCILE 之前，暂停并询问：

> *"单模型评审完成。是否需要跨模型的第二意见？选项：Gemini CLI、Codex CLI、手动外部评审（你粘贴到别处评审）、或跳过。"*

此问题在每个交互式质疑周期中都是强制性的——即使对于感觉低风险的产物也是如此。由用户——而非 agent——决定成本是否值得。agent 的职责是提供这个选择。

**步骤 2：如果用户选择某个 CLI——验证，然后调用**

1. 检查工具是否在 PATH 中（`which gemini`、`which codex`）。
2. 在传递完整提示之前测试其可用性（`gemini --version` 或等效命令）——过期或损坏的二进制文件可能通过 `which` 但在实际输入时失败。
3. 与用户确认确切的调用方式，包括所需的标志、认证和环境变量（例如 API Key）。不同实现有所不同；不要假设。
4. 仅传递 ARTIFACT + CONTRACT + 对抗性提示。不传递会话上下文，不传递 CLAIM。
5. 注意 Shell 转义。如果产物包含引号、`$(...)` 或反引号，建议使用 stdin（`echo … | gemini`）或 heredoc，而非内联的 `-p "…"`。如有疑问，在运行前请用户确认调用命令。
6. 将输出带入步骤 4（RECONCILE）。

**绝不要将产物插值到 Shell 引号参数中。** 代码、Markdown 和评审提示通常包含反引号、`$(...)` 和引号字符，这些字符要么会截断提示，要么会执行嵌入的 Shell 命令。将完整提示写入文件，并通过 stdin 管道传入。

示例形式（根据你安装的工具验证标志——不同实现和版本的语法有所不同）：

```bash
# 先将对抗性提示 + ARTIFACT + CONTRACT 写入一个临时文件。
# 然后通过 stdin 管道传入，使产物中的 shell 元字符保持惰性。

# Codex（只读沙箱防止 CLI 写入你的工作区）：
codex exec --sandbox read-only -C <repo-path> - < /tmp/doubt-prompt.md

# Gemini（'--approval-mode plan' 为只读；'-p ""' 触发非交互模式，
# 提示从 stdin 读取）：
gemini --approval-mode plan -p "" < /tmp/doubt-prompt.md
```

只读沙箱是关键细节：质疑产物本身可能包含指令（有意的或意外的提示注入），跨模型 CLI 否则会将其作为代码执行到你的工作区中。

**步骤 3：如果 CLI 不可用或失败**

明确暴露失败。提供选项：手动运行、尝试其他工具、或跳过。不要悄悄回退到单模型——用户应知情跨模型评审未能执行。

**步骤 4：如果用户选择跳过**

在输出中确认跳过（*"仅使用单模型评审结果继续"*），然后进入 RECONCILE。跳过是可以的；悄悄跳过不行。

**非交互式上下文**（CI、`/loop`、自主循环、定时任务）：

- 跨模型**跳过**，且跳过行为必须在输出中**声明**：*"跨模型已跳过：非交互式上下文。"*
- **绝不在没有明确用户授权的情况下调用外部 CLI**——这是关键的安全属性。

跨模型增加了成本、延迟和工具脆弱性。agent 每个周期都提供选择；用户决定此产物是否值得。

### 步骤 4：RECONCILE — 将发现回馈

评审者的输出是数据，而非裁定。**你仍然是编排者。** 在分类之前，对照每项发现重新阅读产物文本——盲从评审者与忽视它是同一种失败模式。

对每项发现，按此**优先级顺序**分类（首个匹配的类别胜出）：

1. **契约误读**——评审者标记的问题具体是因为你提供的 CONTRACT 不清晰或不完整。先修复契约，下一轮重新分类。
2. **有效 + 可执行**——真实问题，需要对产物进行修改。修改它，重新循环。
3. **有效但为权衡**——问题是真实的，但修复成本超过接受成本。明确记录此权衡，让用户可见。
4. **噪声**——评审者标记的内容在评审者不具备的上下文中实际上是正确的。记录它，继续，并问自己：如果将这些上下文添加到契约中，是否能防止此误报？

一个全新的评审者可能因为缺乏上下文而犯错。不要因为它是"新的"就盲从。

### 步骤 5：STOP — 有界循环，而非递归

以下情况停止：

- 下一轮迭代仅返回平凡或已经考虑过的发现，**或**
- 已完成 3 轮（升级给用户，不要单独死磕第四轮），**或**
- 用户明确说"发版吧"

如果 3 轮后评审者仍然发现实质性问题，产物可能尚未就绪。向用户暴露这一情况——三轮未解决的周期是关于产物的信息，而不是继续循环的理由。

如果"3 轮显然不够"，因为产物太大：那是产物本身太大了——回到步骤 2 进行分解。不要提高循环上限。

## 常见借口

| 借口 | 现实 |
|---|---|
| "我很自信，跳过质疑步骤" | 自信与正确性在新问题上相关性很弱。确信的时刻恰恰是盲点隐藏的地方。 |
| "生成评审者成本太高" | 生产环境中调试一个错误提交成本更高。检查是有界的；bug 不是。 |
| "评审者只会吹毛求疵" | 只有在没有范围限定时才会。将提示约束为"在此契约下导致产物失败的问题"。 |
| "我最后用 `/review` 做质疑就行" | `/review` 是最终关卡。质疑驱动在纠偏成本低廉时尽早捕捉错误方向。到了 PR 阶段就太晚了。 |
| "如果每步都质疑，我永远无法交付" | 此技能适用于非平凡决策，而非每个按键。重新阅读"不适用场景"。 |
| "两个意见总比一个好" | 当第二个意见上下文更少且产生噪声时就不是了。调和，而非盲从。 |
| "评审者不同意，所以我错了" | 评审者缺乏你的上下文——分歧是信息，而非裁定。重新阅读产物，分类，然后决定。 |
| "跨模型总是更好" | 跨模型能捕捉到单模型与自身共享的盲点，但它增加了成本和工具脆弱性。在每次交互式质疑周期中提供此选项——用户决定产物是否值得。agent 的角色是提供选择，而非设卡。 |
| "用户上次说可以，所以我可以继续调用 CLI" | 每次调用都是独立的授权。产物、提示和标志在不同调用之间会变化——每次运行前与用户重新确认确切命令。 |

## 红旗信号

- 为单行重命名或格式化变更生成全新上下文评审者
- 将评审者输出视为权威，而不重新阅读产物文本
- 循环超过 3 轮而未向用户升级
- 用"这好吗？"而非"找出问题"来提示评审者
- 在高风险决策的时间压力下跳过质疑
- 对未更改的产物重新生成全新上下文（你会得到相同的发现；这是在拖延）
- **质疑表演（可检测信号）**：在 2 轮或以上的评审中，评审者发现了实质性发现，但零项发现被分类为可执行。你在验证，而非质疑。停止并升级。
- 仅在提交后才质疑——那是 `/review`，而非质疑驱动开发
- 在与用户确认工具是否存在、已配置且接受确切语法之前，硬编码外部 CLI 调用
- **在交互式质疑周期中悄悄跳过跨模型。** 即便不推荐，也必须可见地提供选项。跳过是可以的；悄悄跳过不行。
- 外部 CLI 出错或缺失时悄悄回退——暴露失败，让用户重新定向
- 从评审者的输入中剥离契约
- 将 CLAIM 传递给评审者（倾向同意）

## 与其他技能的交互

- **`code-review-and-quality` / `/review`**：互补。`/review` 是事后 PR 裁定；质疑驱动是进行中的每决策审查。两者皆用。
- **`source-driven-development`**：SDD 验证*关于框架的事实*，对照官方文档。质疑驱动验证*你对产物的推理*。SDD 检查 API 是否存在；质疑驱动检查你是否在契约下正确使用了它。
- **`test-driven-development`**：TDD 的 RED 步骤是具象化的质疑——一个失败的测试就是一次证伪尝试。当 TDD 适用时，该失败测试*就是*行为主张的质疑步骤。
- **`debugging-and-error-recovery`**：当评审者发现真实失败模式时，切换到调试技能进行定位和修复。
- **仓库编排规则**（`references/orchestration-patterns.md`）：此技能从主会话进行编排。一个 Persona 调用另一个 Persona 是反模式 B——参见上文的加载约束。

## 验证

应用质疑驱动开发后：

- [ ] 每个非平凡决策（按上文定义）在生效前已显式命名为 CLAIM
- [ ] 每个非平凡产物至少经过一次全新上下文评审（TDD 的 RED 步骤产生的失败测试对于行为主张满足此要求，参见与其他技能的交互）
- [ ] 评审者收到的是 ARTIFACT + CONTRACT——而非 CLAIM，亦非你的推理
- [ ] 评审者的提示是对抗性的（"找出问题"），而非验证性的（"它好吗"）
- [ ] 发现已对照产物文本分类（非盲从），采用优先级顺序：契约误读 / 可执行 / 权衡 / 噪声
- [ ] 满足了停止条件（平凡发现、3 轮、或用户覆盖）
- [ ] 在交互模式中，跨模型已**显式提供给用户**（无论产物风险高低），且用户的回应已在输出中确认
- [ ] 在非交互模式中，跨模型已跳过且跳过已声明
- [ ] 任何外部 CLI 调用之前，均已检查 PATH、测试二进制可用性、与用户确认语法、并获得明确运行授权
