# 规划与任务拆解

## 概述

将工作分解为小型、可验证的任务，并带有明确的验收标准。好的任务拆解是 agent 能可靠完成工作与产生一团乱麻之间的区别所在。每个任务都应足够小，使其能在单次聚焦会话中实现、测试和验证。

## 适用场景

- 拥有规格说明，需将其拆解为可实现的单元
- 任务感觉太大或太模糊，无法启动
- 需要在多个 agent 或会话之间并行工作
- 需要向人沟通范围
- 实现顺序不明确

**不适用场景：** 范围明显的单文件变更，或规格说明本身已经包含明确定义的任务。

## 规划流程

### 第 1 步：进入规划模式

在编写任何代码之前，以只读模式操作：

- 阅读规格说明和相关代码库部分
- 识别现有的模式和约定
- 映射组件之间的依赖关系
- 记录风险和未知项

**规划期间不要编写代码。** 输出是保存到 `tasks/plan.md` 的规划文档和保存到 `tasks/todo.md` 的任务列表，而非实现代码。

### 第 2 步：识别依赖图

映射什么依赖什么：

```
数据库模式
    │
    ├── API 模型/类型
    │       │
    │       ├── API 端点
    │       │       │
    │       │       └── 前端 API 客户端
    │       │               │
    │       │               └── UI 组件
    │       │
    │       └── 验证逻辑
    │
    └── 种子数据 / 迁移脚本
```

实现顺序遵循依赖图自底向上：先构建基础。

### 第 3 步：纵向切片

与先构建整个数据库、再构建整个 API、最后构建整个 UI 不同——一次构建一条完整的功能路径：

**错误（横向切片）：**
```
任务 1：构建整个数据库模式
任务 2：构建所有 API 端点
任务 3：构建所有 UI 组件
任务 4：连接所有部分
```

**正确（纵向切片）：**
```
任务 1：用户可以创建账号（注册的模式 + API + UI）
任务 2：用户可以登录（认证模式 + API + 登录 UI）
任务 3：用户可以创建任务（任务模式 + API + 创建 UI）
任务 4：用户可以查看任务列表（查询 + API + 列表视图 UI）
```

每个纵向切片交付可工作的、可测试的功能。

### 第 4 步：编写任务

每个任务遵循以下结构：

```markdown
## 任务 [N]：[简短描述性标题]

**描述：** 用一段话解释此任务要达成什么。

**验收标准：**
- [ ] [具体的、可测试的条件]
- [ ] [具体的、可测试的条件]

**验证：**
- [ ] 测试通过：`npm test -- --grep "feature-name"`
- [ ] 构建成功：`npm run build`
- [ ] 手动检查：[描述需要验证的内容]

**依赖：** [此任务依赖的任务编号，或"无"]

**可能触及的文件：**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**估算范围：** [Small：1-2 个文件 | Medium：3-5 个文件 | Large：5+ 个文件]
```

### 第 5 步：排序与检查点

安排任务顺序，确保：

1. 依赖关系得到满足（先构建基础）
2. 每个任务将系统留在可工作状态
3. 每 2-3 个任务之后设置验证检查点
4. 高风险任务排在最前面（尽早失败）

添加明确的检查点：

```markdown
## 检查点：任务 1-3 完成后
- [ ] 所有测试通过
- [ ] 应用程序构建无错误
- [ ] 核心用户流程端到端工作
- [ ] 继续前与人工审查确认
```

## 任务规模指南

| 规模 | 文件数 | 范围 | 示例 |
|------|-------|-------|---------|
| **XS** | 1 | 单个函数或配置变更 | 添加验证规则 |
| **S** | 1-2 | 一个组件或端点 | 添加新的 API 端点 |
| **M** | 3-5 | 一个功能切片 | 用户注册流程 |
| **L** | 5-8 | 多组件功能 | 带筛选和分页的搜索 |
| **XL** | 8+ | **太大——需进一步拆解** | — |

如果任务为 L 或更大，应拆解为更小的任务。Agent 在 S 和 M 任务上表现最佳。

**何时进一步拆解任务：**
- 需要超过一次聚焦会话才能完成（大约 2 小时以上的 agent 工作）
- 无法用 3 条或更少的要点描述验收标准
- 触及两个或更多独立子系统（如认证和计费）
- 任务标题中包含"和"（暗示这是两个任务）

## 输出文件

- **规划文档：** 将实现计划保存到 `tasks/plan.md`。
- **任务列表：** 将清单风格的任务列表保存到 `tasks/todo.md`。

如果 `tasks/` 目录不存在则创建。这些路径是 `/build` 命令和其他下游工具所期望的约定。

## 规划文档模板

```markdown
# 实现计划：[功能/项目名称]

## 概述
[用一段话总结我们要构建什么]

## 架构决策
- [关键决策 1 及其理由]
- [关键决策 2 及其理由]

## 任务列表

### 第 1 阶段：基础
- [ ] 任务 1：...
- [ ] 任务 2：...

### 检查点：基础阶段
- [ ] 测试通过，构建干净

### 第 2 阶段：核心功能
- [ ] 任务 3：...
- [ ] 任务 4：...

### 检查点：核心功能阶段
- [ ] 端到端流程工作

### 第 3 阶段：打磨
- [ ] 任务 5：...
- [ ] 任务 6：...

### 检查点：完成
- [ ] 所有验收标准满足
- [ ] 准备审查

## 风险与缓解措施
| 风险 | 影响 | 缓解策略 |
|------|--------|------------|
| [风险] | [高/中/低] | [策略] |

## 待定问题
- [需要人工输入的问题]
```

## 并行化机会

当多个 agent 或会话可用时：

- **可安全并行：** 独立的功能切片、已实现功能的测试、文档
- **必须顺序执行：** 数据库迁移、共享状态变更、依赖链
- **需要协调：** 共享 API 契约的功能（先定义契约，再并行化）

## 常见借口

| 借口 | 事实 |
|---|---|
| "我边走边看" | 这正是一团乱麻和返工的来源。10 分钟的规划节省数小时。 |
| "任务挺明显的" | 把它们写下来。明确的任会把隐藏的依赖和被遗忘的边界情况暴露出来。 |
| "规划是额外开销" | 规划本身就是任务。没有计划的实现只是在打字。 |
| "我脑子里都能记住" | 上下文窗口是有限的。书面计划能在会话边界和压缩中存活。 |

## 危险信号

- 没有书面任务列表就开始实现
- 任务描述是"实现 XX 功能"却没有验收标准
- 计划中没有验证步骤
- 所有任务都是 XL 规模
- 任务之间没有检查点
- 未考虑依赖顺序

## 验证

开始实现之前确认：

- [ ] 每个任务都有验收标准
- [ ] 每个任务都有验证步骤
- [ ] 任务依赖已识别并正确排序
- [ ] 没有任何任务触及超过约 5 个文件
- [ ] 各阶段之间有检查点
- [ ] 人工已审查并批准该计划

## 参见

验收标准是针对每个任务的，回答"我们造的东西对吗？"。它们建立在项目级完成定义之上——这是每个任务在算作完成之前必须达到的基准标准。参见 `references/definition-of-done.md`。
