# 弃用与迁移

## 概述

代码是负债，不是资产。每一行代码都有持续的维护成本——需要修复的 bug、需要更新的依赖、需要应用的安全补丁，以及需要培训的新工程师。弃用是移除不再值得保留的代码的纪律，而迁移是将用户安全地从旧系统转移到新系统的过程。

大多数工程组织擅长构建东西。很少有擅长移除东西的。本技能填补这一差距。

## 何时使用

- 用新系统、API 或库替换旧的
- 淘汰不再需要的功能
- 整合重复的实现
- 移除没有人负责但所有人都依赖的死代码
- 规划新系统的生命周期（弃用规划从设计时开始）
- 决定是维护遗留系统还是投资迁移

## 核心原则

### 代码是负债

每一行代码都有持续的成本：它需要测试、文档、安全补丁、依赖更新，以及对任何在其附近工作的人的认知开销。代码的价值是它提供的功能，而不是代码本身。当相同的功能可以用更少的代码、更少的复杂性或更好的抽象来提供时——旧代码应该消失。

### Hyrum 定律使移除变得困难

在有足够多的用户时，每一个可观察的行为都会被依赖——包括 bug、时间特性以及未文档化的副作用。这就是为什么弃用需要主动迁移，而不仅仅是公告。当用户依赖的是替代品没有复制的行为时，他们不能"直接切换"。

### 弃用规划从设计时开始

在构建新东西时，问："我们在 3 年后如何移除这个？"以清晰的接口、特性开关和最小的表面区域设计的系统比到处泄漏实现细节的系统更容易弃用。

## 弃用决策

在弃用任何东西之前，回答这些问题：

```
1. 这个系统仍然提供独特的价值吗？
   → 如果是，维护它。如果否，继续。

2. 有多少用户/消费者依赖它？
   → 量化迁移范围。

3. 是否存在替代品？
   → 如果否，先构建替代品。不要在没有替代方案的情况下弃用。

4. 每个消费者的迁移成本是多少？
   → 如果可以轻松自动化，就自动化。如果是手动且高投入的，与维护成本进行权衡。

5. 不弃用的持续维护成本是多少？
   → 安全风险、工程师时间、复杂性的机会成本。
```

## 建议性弃用 vs. 强制性弃用

| 类型 | 何时使用 | 机制 |
|------|---------|-----------|
| **建议性** | 迁移是可选的，旧系统是稳定的 | 警告、文档、提示。用户按自己的时间表迁移。 |
| **强制性** | 旧系统有安全问题、阻碍进展，或维护成本不可持续 | 硬性截止日期。旧系统将在日期 X 被移除。提供迁移工具。 |

**默认使用建议性。** 仅当维护成本或风险证明强制迁移是合理的时候使用强制性。强制性弃用需要提供迁移工具、文档和支持——你不能只宣布一个截止日期。

## 迁移流程

### 步骤 1：构建替代品

不要在没有可工作的替代方案的情况下弃用。替代品必须：

- 覆盖旧系统的所有关键用例
- 有文档和迁移指南
- 在生产中经过验证（不仅仅是"理论上更好"）

### 步骤 2：公告和文档

```markdown
## 弃用通知：OldService

**状态：** 自 2025-03-01 起已弃用
**替代品：** NewService（见下方迁移指南）
**移除日期：** 建议性 —— 尚无硬性截止日期
**原因：** OldService 需要手动扩缩容且缺乏可观测性。
        NewService 自动处理两者。

### 迁移指南
1. 将 `import { client } from 'old-service'` 替换为 `import { client } from 'new-service'`
2. 更新配置（见下方示例）
3. 运行迁移验证脚本：`npx migrate-check`
```

### 步骤 3：增量迁移

一次迁移一个消费者，而不是一次全部。对每个消费者：

```
1. 识别与被弃用系统的所有接触点
2. 更新为使用替代品
3. 验证行为匹配（测试、集成检查）
4. 移除对旧系统的引用
5. 确认无回归
```

**变更规则：** 如果你拥有被弃用的基础设施，你有责任迁移你的用户——或提供不需要迁移的向后兼容更新。不要宣布弃用然后让用户自己想办法。

### 步骤 4：移除旧系统

仅在所有消费者都已迁移之后：

```
1. 验证零活跃使用（指标、日志、依赖分析）
2. 移除代码
3. 移除相关的测试、文档和配置
4. 移除弃用通知
5. 庆祝 —— 移除代码是一项成就
```

## 迁移模式

### 绞杀者模式（Strangler Pattern）

并行运行新旧系统。逐步将流量从旧系统路由到新系统。当旧系统处理 0% 的流量时，移除它。

```
阶段 1：新系统处理 0%，旧系统处理 100%
阶段 2：新系统处理 10%（金丝雀）
阶段 3：新系统处理 50%
阶段 4：新系统处理 100%，旧系统空闲
阶段 5：移除旧系统
```

### 适配器模式（Adapter Pattern）

创建一个适配器，将调用从旧接口转换到新实现。消费者继续使用旧接口，而你在后台迁移。

```typescript
// 适配器：旧接口，新实现
class LegacyTaskService implements OldTaskAPI {
  constructor(private newService: NewTaskService) {}

  // 旧方法签名，委托给新实现
  getTask(id: number): OldTask {
    const task = this.newService.findById(String(id));
    return this.toOldFormat(task);
  }
}
```

### 特性开关迁移

使用特性开关逐个将消费者从旧系统切换到新系统：

```typescript
function getTaskService(userId: string): TaskService {
  if (featureFlags.isEnabled('new-task-service', { userId })) {
    return new NewTaskService();
  }
  return new LegacyTaskService();
}
```

## 僵尸代码

僵尸代码是没有人负责但每个人都依赖的代码。它没有被积极维护，没有明确的所有者，并积累安全漏洞和兼容性问题。迹象：

- 6 个月以上没有提交但存在活跃消费者
- 没有指定的维护者或团队
- 失败的测试没有人修复
- 存在已知漏洞的依赖没有人更新
- 文档引用了不再存在的系统

**应对：** 要么分配所有者并妥善维护它，要么用具体的迁移计划弃用它。僵尸代码不能停留在边缘地带——它要么获得投资，要么被移除。

## 常见合理化借口

| 合理化借口 | 现实 |
|---|---|
| "它还能用，为什么要移除？" | 能工作但没有人维护的代码积累安全债务和复杂性。维护成本在无声增长。 |
| "以后可能有人需要它" | 如果以后需要，可以重建。保留不用的代码"以防万一"比重建成本更高。 |
| "迁移太贵了" | 将迁移成本与 2-3 年的持续维护成本进行比较。迁移通常在长期更便宜。 |
| "我们完成新系统后再弃用它" | 弃用规划从设计时开始。到新系统完成时，你会有新的优先级。现在就规划。 |
| "用户会自己迁移的" | 他们不会。提供工具、文档和激励——或者你自己做迁移（变更规则）。 |
| "我们可以无限期维护两个系统" | 两个做同样事情的系统意味着双倍的维护、测试、文档和入职成本。 |

## 红旗信号

- 已弃用的系统没有可用的替代品
- 弃用公告没有迁移工具或文档
- 已经建议性弃用多年没有进展的"软"弃用
- 僵尸代码没有所有者且有活跃消费者
- 向已弃用的系统添加新功能（改为投资替代品）
- 没有测量当前使用情况就弃用
- 没有验证零活跃消费者就移除代码

## 验证

完成弃用后：

- [ ] 替代品在生产中得到验证并覆盖所有关键用例
- [ ] 迁移指南存在，包含具体步骤和示例
- [ ] 所有活跃消费者已迁移（通过指标/日志验证）
- [ ] 旧代码、测试、文档和配置已完全移除
- [ ] 代码仓库中不再存在对被弃用系统的引用
- [ ] 弃用通知已移除（它们已完成使命）
