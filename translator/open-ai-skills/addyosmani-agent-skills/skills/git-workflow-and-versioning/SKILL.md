# Git 工作流与版本管理

## 概述

Git 是你的安全网。将提交视为存档点，将分支视为沙箱，将历史视为文档。随着 AI agent 以高速度生成代码，有纪律的版本控制是保持变更可管理、可审查和可回退的机制。

## 适用场景

始终如此。每个代码变更都会经过 git。

## 核心原则

### 基于主干的开发（推荐）

保持 `main` 始终可部署。在 1-3 天内合并回的短期功能分支中工作。长期存在的开发分支是隐性成本 —— 它们会产生分歧、产生合并冲突，并延迟集成。DORA 研究一贯表明，基于主干的开发与高绩效工程团队相关。

```
main ──●──●──●──●──●──●──●──●──●──  （始终可部署）
        ╲      ╱  ╲    ╱
         ●──●─╱    ●──╱    ← 短期功能分支（1-3 天）
```

这是推荐的默认做法。使用 gitflow 或长期分支的团队可以将这些原则（原子提交、小变更、描述性消息）适配到各自的分支模型 —— 提交纪律比特定的分支策略更重要。

- **开发分支是成本。** 分支每多存活一天，就积累一天合并风险。
- **发布分支是可接受的。** 当你需要在 main 向前推进的同时稳定一个发布时。
- **功能标志 > 长期分支。** 优先将未完成的工作部署在标志后面，而不是放在分支上数周。

### 1. 尽早提交，频繁提交

每个成功的增量一个提交。不要累积大量未提交的变更。

```
工作模式：
  实现一个切片 → 测试 → 验证 → 提交 → 下一个切片

不要这样：
  实现所有内容 → 希望它能工作 → 一个巨型提交
```

提交是存档点。如果下一个变更破坏了某些东西，你可以立即回退到上一个已知良好状态。

### 2. 原子提交

每个提交只做一件事：

```
# 好的：每个提交自成一体
git log --oneline
a1b2c3d 添加带验证的任务创建端点
d4e5f6g 添加任务创建表单组件
h7i8j9k 将表单连接到 API 并添加加载状态
m1n2o3p 添加任务创建测试（单元 + 集成）

# 不好的：所有内容混在一起
git log --oneline
x1y2z3a 添加任务功能、修复侧边栏、更新依赖、重构工具函数
```

### 3. 描述性消息

提交消息解释**为什么（why）**，而不仅是**做了什么（what）**：

```
# 好的：解释意图
feat: 为注册端点添加邮箱验证

防止无效邮箱格式进入数据库。
在路由处理器层面使用 Zod schema 验证，
与 auth.ts 中现有的验证模式保持一致。

# 不好的：描述 diff 中已显而易见的内容
update auth.ts
```

**格式：**
```
<type>: <简短描述>

<可选正文，解释为什么，而非做了什么>
```

**类型：**
- `feat` —— 新功能
- `fix` —— Bug 修复
- `refactor` —— 既不修复 bug 也不添加功能的代码变更
- `test` —— 添加或更新测试
- `docs` —— 仅文档变更
- `chore` —— 工具、依赖、配置

### 4. 保持关注点分离

不要将格式化变更与行为变更混在一起。不要将重构与功能混在一起。每种类型的变更应是一个独立的提交 —— 并且理想情况下是一个独立的 PR：

```
# 好的：分离关注点
git commit -m "refactor: 提取验证逻辑到共享工具函数"
git commit -m "feat: 为注册添加手机号验证"

# 不好的：混合关注点
git commit -m "重构验证并添加手机号字段"
```

**将重构与功能工作分开。** 一个重构变更和一个功能变更是两个不同的变更 —— 应分别提交。这使每个变更更容易审查、回退和理解历史。小型清理（如重命名变量）可在审查者同意下包含在功能提交中。

### 5. 控制变更规模

每个提交/PR 的目标是约 100 行。超过约 1000 行的变更应拆分。参见 `code-review-and-quality` 中关于如何分解大型变更的拆分策略。

```
~100 行   → 易于审查，易于回退
~300 行   → 单个逻辑变更可接受的范围
~1000 行  → 应拆分为更小的变更
```

## 分支策略

### 功能分支

```
main（始终可部署）
  │
  ├── feature/task-creation    ← 每个功能一个分支
  ├── feature/user-settings    ← 并行工作
  └── fix/duplicate-tasks      ← Bug 修复
```

- 从 `main`（或团队默认分支）创建分支
- 保持分支生命周期短（1-3 天内合并）—— 长期分支是隐性成本
- 合并后删除分支
- 优先使用功能标志而非长期分支来处理未完成的功能

### 分支命名

```
feature/<简短描述>   → feature/task-creation
fix/<简短描述>       → fix/duplicate-tasks
chore/<简短描述>     → chore/update-deps
refactor/<简短描述>  → refactor/auth-module
```

## 使用 Worktrees 工作

对于并行的 AI agent 工作，使用 git worktrees 同时运行多个分支：

```bash
# 为功能分支创建 worktree
git worktree add ../project-feature-a feature/task-creation
git worktree add ../project-feature-b feature/user-settings

# 每个 worktree 是一个独立的目录，拥有自己的分支
# Agent 可以并行工作，互不干扰
ls ../
  project/              ← main 分支
  project-feature-a/    ← task-creation 分支
  project-feature-b/    ← user-settings 分支

# 完成时合并并清理
git worktree remove ../project-feature-a
```

好处：
- 多个 agent 可以同时处理不同功能
- 无需切换分支（每个目录有自己的分支）
- 如果一个实验失败，删除 worktree —— 不会有任何丢失
- 变更在显式合并前保持隔离

## 存档点模式

```
Agent 开始工作
    │
    ├── 做出一项变更
    │   ├── 测试通过？→ 提交 → 继续
    │   └── 测试失败？→ 回退到上次提交 → 调查
    │
    ├── 做出另一项变更
    │   ├── 测试通过？→ 提交 → 继续
    │   └── 测试失败？→ 回退到上次提交 → 调查
    │
    └── 功能完成 → 所有提交构成一个干净的历史
```

这种模式意味着你永远不会丢失超过一个增量的工作。如果 agent 失控，`git reset --hard HEAD` 让你回到上一个成功状态。

## 变更摘要

在任何修改之后，提供结构化的摘要。这使审查更容易，记录了范围纪律，并暴露了意外变更：

```
CHANGES MADE:
- src/routes/tasks.ts: 为 POST 端点添加验证中间件
- src/lib/validation.ts: 使用 Zod 添加 TaskCreateSchema

THINGS I DIDN'T TOUCH（有意未触碰）:
- src/routes/auth.ts: 存在类似的验证缺陷，但不在本次范围内
- src/middleware/error.ts: 错误格式可改进（独立任务）

POTENTIAL CONCERNS:
- Zod schema 是严格的 —— 拒绝额外字段。请确认是否符合预期。
- 添加了 zod 依赖（72KB gzipped）—— 已在 package.json 中
```

这种模式能及早发现错误假设，并为审查者提供清晰的变更地图。"DIDN'T TOUCH" 部分尤其重要 —— 它表明你遵守了范围纪律，没有进行未经请求的翻新。

## 提交前检查

每次提交之前：

```bash
# 1. 检查即将提交的内容
git diff --staged

# 2. 确保无密钥泄露
git diff --staged | grep -i "password\|secret\|api_key\|token"

# 3. 运行测试
npm test

# 4. 运行代码检查
npm run lint

# 5. 运行类型检查
npx tsc --noEmit
```

使用 git hooks 自动化：

```json
// package.json（使用 lint-staged + husky）
{
  "lint-staged": {
    "*.{ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md}": ["prettier --write"]
  }
}
```

## 处理生成文件

- **提交生成文件** 仅当项目需要时（例如 `package-lock.json`、Prisma migrations）
- **不提交** 构建产物（`dist/`、`.next/`）、环境文件（`.env`）或 IDE 配置（`.vscode/settings.json`，除非作为共享配置）
- **编写 `.gitignore`** 覆盖：`node_modules/`、`dist/`、`.env`、`.env.local`、`*.pem`

## 使用 Git 进行调试

```bash
# 查找哪个提交引入了 bug
git bisect start
git bisect bad HEAD
git bisect good <已知良好的提交>
# Git 会检出中间提交；每次运行测试以逐步缩小范围

# 查看最近的变更
git log --oneline -20
git diff HEAD~5..HEAD -- src/

# 查找最后修改某行的责任人
git blame src/services/task.ts

# 按关键词搜索提交消息
git log --grep="validation" --oneline
```

## 发布与版本管理

提交是**你**追踪变更的方式；**版本号**是你的**消费者**追踪变更的方式。一旦有任何东西依赖你的代码 —— 另一个团队、一个发布的包、一个部署的客户端 —— "main 上的最新版本" 就不再能回答"我在运行什么？升级安全吗？"这个问题。版本号和更新日志就是回答这个问题的契约。

### 语义化版本管理

对于任何有消费者的东西，使用 `MAJOR.MINOR.PATCH` 版本号，让数字承载含义：

```
  MAJOR  破坏性变更 —— 消费者必须修改代码才能升级
  MINOR  新功能，向后兼容 —— 安全升级
  PATCH  Bug 修复，向后兼容 —— 安全升级
```

数字是一种承诺，因此要让代码与之匹配。一个变更为"patch"但改变了消费者依赖的行为，就是披着 patch 外衣的 major（Hyrum 定律 —— 参见 `api-and-interface-design` 技能）。当不确定一个变更是否具有破坏性时，假定它是；一个意外的 major 远比一个被破坏的消费者成本低。

### 为发布打标签，让标签成为唯一可信源

一个发布是历史中不可变的一个点，不是一个移动的分支。打上标签以便始终可以重现：

```bash
git tag -a v1.4.0 -m "Release 1.4.0"
git push origin v1.4.0
```

从标签派生版本号，而不是在分散的文件中手动编辑，这样构建产物、标签和更新日志永远不会不一致。

### 维护一份面向人类的更新日志

更新日志不是 `git log`。它是经过精心的、面向消费者的"发生了什么变化？我需要关心吗？"的回答 —— 按 `Added / Changed / Fixed / Deprecated / Removed / Security` 分组，最新的在上面，每个条目围绕用户影响而非内部机制来描述。

```markdown
## [1.4.0] - 2025-06-12
### Added
- 通过 CSV 批量导入任务
### Fixed
- 修复重复任务截止日期的时区偏移问题
### Deprecated
- `GET /v1/tasks/all` —— 使用分页的 `GET /v1/tasks`（将在 2.0 中移除）
```

条目应在做出变更的同一变更中编写，趁影响还在脑海中 —— 而不是在发布时从提交考古中拼凑。破坏性变更要附带迁移说明和弃用窗口（遵循 `deprecation-and-migration` 技能）；实际发布交付是 `shipping-and-launch` 技能的职责 —— 本节是支撑该流程的版本管理契约。

## 常见借口

| 借口 | 现实 |
|---|---|
| "功能完成了我再提交" | 一个巨型提交无法审查、调试或回退。每个切片都提交。 |
| "消息不重要" | 消息是文档。未来的你（以及未来的 agent）需要理解发生了什么变化以及为什么。 |
| "我以后再 squash 就行" | Squash 破坏了开发叙事。从一开始就提交干净的增量提交。 |
| "分支增加开销" | 短期分支是免费的，并防止冲突工作的碰撞。长期分支才是问题 —— 在 1-3 天内合并。 |
| "我以后再拆分这个变更" | 大型变更更难审查、部署风险更高、更难回退。在提交前拆分，而非之后。 |
| "我不需要 .gitignore" | 直到包含生产密钥的 `.env` 被提交。立即设置。 |
| "只是个小修复，升级 patch 就行" | 检查消费者能观察到什么。他们依赖的行为变更就是一个 major，无论 diff 有多大。 |
| "更新日志就是提交日志" | 提交是给你的；更新日志是给消费者的，按影响精心组织。从原始提交生成的更新日志会淹没真正重要的内容。 |
| "我们发布时再写更新日志" | 到那时影响是靠记忆重建的，一半内容已缺失。条目随变更一起写。 |

## 红旗警示

- 大量未提交的变更堆积
- 提交消息如 "fix"、"update"、"misc"
- 格式化变更与行为变更混在一起
- 项目中没有 `.gitignore`
- 提交了 `node_modules/`、`.env` 或构建产物
- 与 main 严重偏离的长期分支
- 对共享分支进行 force-push
- 破坏性变更在 minor 或 patch 版本号下发布
- 发布没有标签，或版本号手动编辑与标签不一致
- 面向用户的发布没有更新日志条目，或更新日志只是倾倒提交消息

## 验证

每次提交：

- [ ] 提交只做一件逻辑上的事
- [ ] 消息解释为什么（why），遵循类型约定
- [ ] 提交前测试通过
- [ ] diff 中没有密钥
- [ ] 格式化变更与行为变更未混合
- [ ] `.gitignore` 覆盖了标准排除项

每次发布（有消费者的任何内容）：

- [ ] 版本号变更与变更类型匹配：破坏性 → major，新增 → minor，修复 → patch
- [ ] 发布已打标签，版本号从标签派生，而非手动编辑导致不一致
- [ ] 更新日志有针对该版本的、按影响分组的、精心编排的、人类可读的条目
