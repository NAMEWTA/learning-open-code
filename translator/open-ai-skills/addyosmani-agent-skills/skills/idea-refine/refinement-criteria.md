# 精炼与评估标准

在阶段 2（评估与收敛）中使用此准则对想法方向进行压力测试。并非每个标准都适用于每个想法——根据具体上下文判断哪些维度最重要。

## 核心评估维度

### 1. 用户价值

最重要的维度。如果价值不明确，其他一切都不重要。

**止痛药 vs. 维生素：**
- **止痛药：** 解决一个急性的、频繁的问题。用户会主动寻找这个。他们会从当前解决方案切换过来。信号：人们带着情绪描述问题，他们已经构建了变通方案，他们愿意为解决方案付费。
- **维生素：** 锦上添花。让某些东西略微变好。用户不会特意去寻找。信号：人们礼貌地点头，说"挺酷的"，然后不改变行为。

**要问的问题：**
- 你能说出 3 个当前正面临这个问题的具体的人吗？
- 他们今天在用什么替代方案？（真正的竞争对手永远是当前的变通方案。）
- 他们会从当前方式切换过来吗？什么会让他们切换？
- 他们多长时间遇到一次这个问题？（每天出现的问题 > 每月出现的问题）
- 这是一个"拉力"问题（用户在要求这个）还是一个"推力"问题（你认为他们应该想要这个）？

**红旗信号：**
- "每个人都可以用这个"——如果你不能说出一个具体的用户，价值就不明确
- "它像 X 但更好"——边际改进很少驱动采用
- 问题是真实的但很少发生——高强度但低频率很少能支撑一个产品

### 2. 可行性

你真的能构建这个吗？不仅是技术上的，而且是实际上的。

**技术可行性：**
- 核心技术是否存在且可靠运行？
- 最困难的技术问题是什么？是已知难题还是全新的？
- 是否有对第三方、API 或你无法控制的数据源的依赖？
- 所需的最小技术栈是什么？（如果答案是"很多"，那是一个信号。）

**资源可行性：**
- 构建 MVP 的最小团队/工作量是多少？
- 是否需要你没有的专业知识？
- 是否有监管、法律或合规要求？

**价值实现时间：**
- 你能多快将东西放到用户面前？
- 是否有一个版本能在数天/数周内交付价值，而不是数月？
- 关键路径是什么？必须先做什么？

**红旗信号：**
- "我们只需要先解决[非常困难的研究问题]"
- 多个需要同时工作的依赖项
- MVP 仍然需要数月的工作——可能不够精简

### 3. 差异化

什么使这个真正与众不同？不是更好——是**不同**。

**要问的问题：**
- 如果用户向朋友描述这个，他们会怎么说？那个描述是否引人注目？
- 这个东西做的哪一件事是其他东西不做的？（如果你说不出一个，那就是问题。）
- 这种差异化是否持久？竞争对手能在一周内复制它吗？
- 差异是用户真正关心的东西，还是只有构建者觉得有趣的东西？

**差异化类型（从最强到最弱）：**
1. **新能力：** 做到了以前不可能的事情
2. **10x 改进：** 在关键维度上提升如此之大，以至于改变了行为
3. **新受众：** 将现有能力带给以前被排除在外的人
4. **新上下文：** 在现有解决方案失效的情境中工作
5. **更好的 UX：** 相同能力，大幅更简单的体验
6. **更便宜：** 相同的东西，更低的成本（最弱——容易被竞争淘汰）

**红旗信号：**
- 差异化完全关于技术，而非用户体验
- "我们更快/更便宜/更漂亮"没有结构性的原因
- 差异化的功能不是用户最关心的功能

## 假设审计

对每个想法方向，在三个类别中明确列出假设：

### 必须为真（致命项）
如果错了，直接扼杀整个想法的假设。这些需要在构建前验证。

示例："用户会与我们分享他们的数据"——如果他们不分享，整个产品无法工作。

### 应该为真（重要）
显著影响成功但不扼杀想法的假设。如果这些假设错了，你可以调整方法。

示例："用户更喜欢自助服务而不是与人交谈"——如果错了，你需要不同的市场进入策略，但核心产品仍然可以工作。

### 可能为真（锦上添花）
关于次要功能或优化的假设。在核心得到验证之前不要验证这些。

示例："用户会想要与团队成员分享他们的结果"——一个增长功能，不是核心价值主张。

## 决策框架

在方向之间选择时，按此矩阵排序：

|                    | 高可行性 | 低可行性 |
|--------------------|---------|---------|
| **高价值**         | 优先做这个 | 值得冒险 |
| **低价值**         | 仅在不费事时做 | 不要做 |

然后用差异化作为同一象限内选项之间的决胜因素。

## MVP 范围界定原则

在定义选定方向的 MVP 范围时：

1. **做好一项工作。** MVP 应该精准做好一个用户工作。不是三项工作部分完成。
2. **最危险的假设优先。** MVP 的主要目的是测试最可能出错的假设。
3. **时间盒，而非功能列表。** "我们在[时间范围]内可以构建和测试什么？"优于"我们需要什么功能？"
4. **"不做"清单是强制性的。** 明确说出你要砍掉什么以及为什么。这可以防止范围蔓延并迫使诚实的优先级排序。
5. **如果它不令人尴尬，你就等太久了。** 第一个版本应该在构建者看来是不完整的。如果不是，你过度构建了。
