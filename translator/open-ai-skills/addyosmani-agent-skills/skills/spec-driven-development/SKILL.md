# 规格驱动开发

## 概述

在编写任何代码之前，先撰写结构化的规格说明。规格说明是你与人类工程师之间的共享事实来源——它定义了我们正在构建什么、为什么构建，以及如何判断已完成。未经规格说明的编码只是猜测。

## 适用场景

- 启动新项目或新功能
- 需求模糊或不完整
- 变更涉及多个文件或模块
- 即将做出架构决策
- 任务预计需要超过 30 分钟实现

**不适用场景：** 单行修复、拼写更正，或需求明确且自包含的变更。

## 门控工作流

规格驱动开发包含四个阶段。在当前阶段验证通过之前，不得进入下一阶段。

```
SPECIFY ──→ PLAN ──→ TASKS ──→ IMPLEMENT
   │          │        │          │
   ▼          ▼        ▼          ▼
 人类       人类      人类       人类
 评审       评审      评审       评审
```

### 阶段 1：Specify（编写规格）

从高层愿景开始。向人类提问澄清，直到需求具体明确。

**立即暴露假设。** 在编写任何规格内容之前，列出你的假设：

```
我基于以下假设：
1. 这是一个 Web 应用（非原生移动应用）
2. 认证使用基于 Session 的 Cookie（非 JWT）
3. 数据库为 PostgreSQL（基于现有的 Prisma schema）
4. 目标浏览器仅限现代浏览器（不含 IE11）
→ 如有错误请立即纠正，否则我将基于这些假设继续。
```

不要悄无声息地填充模糊需求。规格说明的全部意义就是在代码编写*之前*暴露误解——假设是最危险的误解形式。

**撰写一份覆盖以下六个核心方面的规格文档：**

1. **目标**——我们在构建什么？为什么构建？用户是谁？成功的标准是什么？

2. **命令**——带完整标志的可执行命令，而非仅工具名称。
   ```
   Build: npm run build
   Test: npm test -- --coverage
   Lint: npm run lint --fix
   Dev: npm run dev
   ```

3. **项目结构**——源码所在位置、测试所在位置、文档所在位置。
   ```
   src/           → 应用源代码
   src/components → React 组件
   src/lib        → 共享工具函数
   tests/         → 单元测试和集成测试
   e2e/           → 端到端测试
   docs/          → 文档
   ```

4. **代码风格**——一段真实的代码片段展示你的风格，胜过三段文字描述。包含命名规范、格式化规则以及优秀输出的示例。

5. **测试策略**——使用什么框架、测试存放位置、覆盖率期望、哪些关注点使用哪一层的测试。

6. **边界规则**——三级体系：
   - **总是执行：** 提交前运行测试、遵循命名规范、验证输入
   - **先询问：** 数据库 schema 变更、添加依赖、修改 CI 配置
   - **禁止执行：** 提交密钥、编辑 vendor 目录、未经批准删除失败测试

**规格模板：**

```markdown
# 规格：[项目/功能名称]

## 目标
[我们正在构建什么以及为什么。用户故事或验收标准。]

## 技术栈
[框架、语言、关键依赖及其版本]

## 命令
[构建、测试、检查、开发——完整命令]

## 项目结构
[带说明的目录布局]

## 代码风格
[示例代码片段 + 关键约定]

## 测试策略
[框架、测试位置、覆盖率要求、测试层级]

## 边界规则
- 总是执行：[...]
- 先询问：[...]
- 禁止执行：[...]

## 成功标准
[如何判断已完成——具体、可测试的条件]

## 待解决问题
[任何未解决、需要人类输入的内容]
```

**将指令重新表述为成功标准。** 当收到模糊需求时，将其转化为具体条件：

```
需求："让仪表盘更快"

重新表述的成功标准：
- 仪表盘 LCP 在 4G 连接下 < 2.5 秒
- 初始数据加载在 500 毫秒内完成
- 加载期间无布局偏移（CLS < 0.1）
→ 这些目标是否合适？
```

这让你可以围绕清晰的目标循环、重试和解决问题，而不是猜测"更快"到底是什么意思。

### 阶段 2：Plan（制定计划）

基于验证通过的规格，生成技术实现计划：

1. 识别主要组件及其依赖关系
2. 确定实现顺序（什么必须先构建）
3. 标记风险和缓解策略
4. 识别可并行构建和必须串行的部分
5. 定义各阶段之间的验证检查点

> 请遵循 `planning-and-task-breakdown` 了解这些步骤背后的依赖图映射和垂直切片机制；它是权威来源。以上要点是轻量级摘要；若有分歧，以 `planning-and-task-breakdown` 为准。
>
> **输出约定：** 将计划保存到 `tasks/plan.md`，将任务列表保存到 `tasks/todo.md`，遵循 `/plan` 命令的约定。如果 `tasks/` 不存在则创建。下游命令（`/build` 等）依赖这些路径。

计划必须是可评审的：人类阅读后应能说"是的，这是正确的方法"或"不对，改掉 X"。

### 阶段 3：Tasks（任务分解）

将计划分解为离散的、可执行的任务：

- 每个任务应能在单次集中会话中完成
- 每个任务有明确的验收标准
- 每个任务包含验证步骤（测试、构建、手动检查）
- 任务按依赖关系排序，而非按感知重要性排序
- 单一任务不应修改超过约 5 个文件

> 请遵循 `planning-and-task-breakdown` 了解完整的任务大小估算和依赖排序机制；它是权威来源。以下模板是轻量级内联形式；若有分歧，以 `planning-and-task-breakdown` 为准。

**任务模板：**
```markdown
- [ ] 任务：[描述]
  - 验收：[任务完成时必须满足的条件]
  - 验证：[如何确认——测试命令、构建、手动检查]
  - 文件：[将涉及哪些文件]
```

### 阶段 4：Implement（实现）

逐个执行任务，遵循 `skills/incremental-implementation/SKILL.md`（`incremental-implementation`）和 `skills/test-driven-development/SKILL.md`（`test-driven-development`）。使用 `skills/context-engineering/SKILL.md`（`context-engineering`）在每个步骤加载合适的规格章节和源文件，而不是将整个规格一次性塞给 agent。

## 保持规格的活力

规格是活的文档，不是一次性产物：

- **决策变更时更新**——如果发现数据模型需要修改，先更新规格，再实现。
- **范围变更时更新**——新增或删除的功能应反映在规格中。
- **将规格纳入版本控制**——规格应与代码一起纳入版本控制。
- **在 PR 中引用规格**——链接到每个 PR 所实现的对应规格章节。

## 常见借口

| 借口 | 现实 |
|---|---|
| "这很简单，不需要规格" | 简单任务不需要*长篇*规格，但仍需要验收标准。两行规格也可以。 |
| "我写完代码再补规格" | 那是文档，不是规格。规格的价值在于*写代码之前*强制明确。 |
| "写规格会拖慢进度" | 15 分钟写规格可以避免数小时的返工。15 分钟的瀑布流胜过 15 小时的调试。 |
| "需求反正会变更" | 所以才要让规格成为活的文档。过时的规格仍然好过没有规格。 |
| "用户知道自己想要什么" | 即便是清晰的需求也有隐含假设。规格的作用就是暴露这些假设。 |

## 红旗信号

- 在没有任何书面需求的情况下开始编写代码
- 在明确"完成"意味着什么之前就问"我可以开始构建了吗？"
- 实现了任何规格或任务列表中未提及的功能
- 做出架构决策却未记录
- 因为"要构建什么很明显"而跳过规格

## 验证

在进入实现阶段之前，确认：

- [ ] 规格覆盖了全部六个核心方面
- [ ] 人类已评审并批准了该规格
- [ ] 成功标准具体且可测试
- [ ] 边界规则（总是执行/先询问/禁止执行）已定义
- [ ] 规格已保存到仓库中的文件
