# 发布与上线

## 概述

满怀信心地发布。目标不仅仅是部署 —— 而是安全地部署，同时配备监控、准备好回滚方案，并清楚了解成功的标准。每次上线都应该是可逆的、可观察的和渐进的。

## 适用场景

- 首次将功能部署到生产环境
- 向用户发布重大变更
- 迁移数据或基础设施
- 开放 beta 或早期访问计划
- 任何存在风险的部署（所有部署都有风险）

## 上线前检查清单

### 代码质量

- [ ] 所有测试通过（单元、集成、e2e）
- [ ] 构建成功，无警告
- [ ] Lint 和类型检查通过
- [ ] 代码已审查并批准
- [ ] 没有需要在上线前解决的 TODO 注释
- [ ] 生产代码中没有 `console.log` 调试语句
- [ ] 错误处理覆盖了预期的故障模式

### 安全性

- [ ] 代码或版本控制中无密钥
- [ ] `npm audit` 无 critical 或 high 级别漏洞
- [ ] 所有面向用户的端点都有输入验证
- [ ] 认证和授权检查已就位
- [ ] 安全头已配置（CSP、HSTS 等）
- [ ] 认证端点有速率限制
- [ ] CORS 配置为特定来源（非通配符）

### 性能

- [ ] Core Web Vitals 在"良好"阈值内
- [ ] 关键路径中无 N+1 查询
- [ ] 图片已优化（压缩、响应式尺寸、懒加载）
- [ ] Bundle 体积在预算内
- [ ] 数据库查询有合适的索引
- [ ] 静态资源和重复查询的缓存已配置

### 无障碍性

- [ ] 所有可交互元素支持键盘导航
- [ ] 屏幕阅读器能传达页面内容和结构
- [ ] 颜色对比度满足 WCAG 2.1 AA（文本 4.5:1）
- [ ] 模态框和动态内容的焦点管理正确
- [ ] 错误消息具有描述性并与表单字段关联
- [ ] axe-core 或 Lighthouse 中无无障碍性警告

### 基础设施

- [ ] 生产环境变量已设置
- [ ] 数据库迁移已应用（或准备就绪）
- [ ] DNS 和 SSL 已配置
- [ ] 静态资源的 CDN 已配置
- [ ] 日志和错误报告已配置
- [ ] 健康检查端点存在并能响应

### 文档

- [ ] README 已更新，包含任何新的设置要求
- [ ] API 文档保持最新
- [ ] 架构决策记录（ADR）已编写
- [ ] 更新日志已更新
- [ ] 面向用户的文档已更新（如适用）

## 功能标志策略

将代码放在功能标志后面，以解耦部署与发布：

```typescript
// 功能标志检查
const flags = await getFeatureFlags(userId);

if (flags.taskSharing) {
  // 新功能：任务共享
  return <TaskSharingPanel task={task} />;
}

// 默认：现有行为
return null;
```

**功能标志生命周期：**

```
1. 部署，标志为 OFF     → 代码在生产环境中但处于未激活状态
2. 对团队/beta 启用    → 在生产环境中进行内部测试
3. 逐步上线              → 5% → 25% → 50% → 100% 用户
4. 在每个阶段监控         → 关注错误率、性能、用户反馈
5. 清理                  → 全量上线后移除标志和死代码路径
```

**规则：**
- 每个功能标志有负责人和过期日期
- 全量上线后 2 周内清理标志
- 不要嵌套功能标志（产生指数级组合）
- 在 CI 中测试标志的两种状态（开和关）

## 分阶段上线

### 上线序列

```
1. 部署到 staging
   └── 在 staging 环境中运行完整测试套件
   └── 关键流程的手动冒烟测试

2. 部署到生产环境（功能标志为 OFF）
   └── 验证部署成功（健康检查）
   └── 检查错误监控（无新错误）

3. 对团队启用（内部用户标志为 ON）
   └── 团队在生产环境中使用该功能
   └── 24 小时监控窗口

4. 灰度上线（5% 用户标志为 ON）
   └── 监控错误率、延迟、用户行为
   └── 对比指标：灰度组 vs. 基线组
   └── 24-48 小时监控窗口
   └── 仅当所有阈值通过时才继续（参见下表）

5. 逐步增加（25% -> 50% -> 100%）
   └── 每个步骤同样监控
   └── 随时可以回滚到上一百分比

6. 全量上线（所有用户标志为 ON）
   └── 监控 1 周
   └── 清理功能标志
```

### 上线决策阈值

在每个阶段使用以下阈值决定是推进、暂停还是回滚：

| 指标 | 推进（绿色） | 暂停调查（黄色） | 回滚（红色） |
|--------|-----------------|-------------------------------|-----------------|
| 错误率 | 在基线 10% 以内 | 高于基线 10-100% | 高于基线 2 倍以上 |
| P95 延迟 | 在基线 20% 以内 | 高于基线 20-50% | 高于基线 50% 以上 |
| 客户端 JS 错误 | 无新错误类型 | 新错误在 <0.1% 的会话中出现 | 新错误在 >0.1% 的会话中出现 |
| 业务指标 | 中性或正向 | 下降 <5%（可能为噪音） | 下降 >5% |

### 何时回滚

以下情况立即回滚：
- 错误率超过基线 2 倍以上
- P95 延迟增加超过 50%
- 用户报告问题激增
- 检测到数据完整性问题
- 发现安全漏洞

## 监控与可观测性

### 需要监控的内容

```
应用指标：
├── 错误率（总体和按端点）
├── 响应时间（p50、p95、p99）
├── 请求量
├── 活跃用户数
└── 关键业务指标（转化率、参与度）

基础设施指标：
├── CPU 和内存使用率
├── 数据库连接池使用情况
├── 磁盘空间
├── 网络延迟
└── 队列深度（如适用）

客户端指标：
├── Core Web Vitals（LCP、INP、CLS）
├── JavaScript 错误
├── 来自客户端的 API 错误率
└── 页面加载时间
```

### 错误报告

```typescript
// 设置带报告功能的错误边界
class ErrorBoundary extends React.Component {
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    // 向错误追踪服务报告
    reportError(error, {
      componentStack: info.componentStack,
      userId: getCurrentUser()?.id,
      page: window.location.pathname,
    });
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback onRetry={() => this.setState({ hasError: false })} />;
    }
    return this.props.children;
  }
}

// 服务端错误报告
app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
  reportError(err, {
    method: req.method,
    url: req.url,
    userId: req.user?.id,
  });

  // 不向用户暴露内部细节
  res.status(500).json({
    error: { code: 'INTERNAL_ERROR', message: 'Something went wrong' },
  });
});
```

### 上线后验证

上线后的第一个小时内：

```
1. 检查健康端点返回 200
2. 检查错误监控仪表盘（无新错误类型）
3. 检查延迟仪表盘（无退化）
4. 手动测试关键用户流程
5. 验证日志正在写入且可读
6. 确认回滚机制可用（如可能，进行演练）
```

## 回滚策略

每次部署在执行之前都需要一个回滚计划：

```markdown
## [功能/发布] 回滚计划

### 触发条件
- 错误率 > 基线 2 倍
- P95 延迟 > [X]ms
- 用户报告 [具体问题]

### 回滚步骤
1. 禁用功能标志（如适用）
   或
1. 部署上一个版本：`git revert <commit> && git push`
2. 验证回滚：健康检查、错误监控
3. 沟通：通知团队已回滚

### 数据库考虑
- 迁移 [X] 有一个回滚方案：`npx prisma migrate rollback`
- 新功能插入的数据：[保留 / 清理]

### 回滚时间
- 功能标志：< 1 分钟
- 重新部署上一个版本：< 5 分钟
- 数据库回滚：< 15 分钟
```

## 常见借口

| 借口 | 现实 |
|---|---|
| "在 staging 没问题，生产环境也不会有问题" | 生产环境有不同的数据、流量模式和边缘情况。部署后持续监控。 |
| "这个不需要功能标志" | 每个功能都能从"关闭开关"中受益。即使"简单"的变更也可能出错。 |
| "监控是一种开销" | 没有监控意味着你通过用户投诉而非仪表盘发现问题。 |
| "我们以后再加监控" | 上线前就加上。你看不到的东西就无法调试。 |
| "回滚等于承认失败" | 回滚是负责任的工程实践。发布一个出问题的功能才是失败。 |

## 红旗警示

- 没有回滚计划就部署
- 生产环境没有监控或错误报告
- 大爆炸式发布（一次性全发布，无分阶段）
- 功能标志没有过期时间或负责人
- 上线后第一个小时内无人监控部署情况
- 生产环境配置凭记忆操作而非代码化管理
- "已经是周五下午了，上线吧"
