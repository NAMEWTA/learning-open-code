# 源码驱动开发

## 概述

每个框架特定代码的决策必须由官方文档作为支撑。不要凭记忆实现——先验证、再引用，并让用户看到你的来源。训练数据会过时，API 会被弃用，最佳实践会演进。这项技能确保用户获得的代码值得信赖，因为每个模式都可追溯到他们能够自行核验的权威来源。

## 适用场景

- 用户希望代码遵循给定框架的当前最佳实践
- 构建样板代码、起始代码或将在项目中复制使用的模式
- 用户明确要求文档化、已验证或"正确"的实现
- 实现功能时，框架推荐的方法很重要（表单、路由、数据获取、状态管理、认证）
- 评审或改进使用了框架特定模式的代码
- 任何时候你准备凭记忆编写框架特定代码

**不适用场景：**

- 正确性与特定版本无关（重命名变量、修复拼写、移动文件）
- 纯逻辑，在所有版本中行为一致（循环、条件判断、数据结构）
- 用户明确要求速度优先于验证（"快速搞定就行"）

## 流程

```
DETECT ──→ FETCH ──→ IMPLEMENT ──→ CITE
  │          │           │            │
  ▼          ▼           ▼            ▼
 识别      获取        遵循         展示
 技术栈？   相关        文档中的     你的
            官方文档    模式         来源
```

### 步骤 1：Detect（识别技术栈和版本）

阅读项目的依赖文件以识别确切版本：

```
package.json    → Node/React/Vue/Angular/Svelte
composer.json   → PHP/Symfony/Laravel
requirements.txt / pyproject.toml → Python/Django/Flask
go.mod          → Go
Cargo.toml      → Rust
Gemfile         → Ruby/Rails
```

明确说明你发现的内容：

```
检测到的技术栈：
- React 19.1.0（来自 package.json）
- Vite 6.2.0
- Tailwind CSS 4.0.3
→ 正在获取相关模式的官方文档。
```

如果版本缺失或不明确，**询问用户**。不要猜测——版本决定了哪些模式是正确的。

### 步骤 2：Fetch（获取官方文档）

获取你正在实现的功能所对应的具体文档页面。不是首页，不是全部文档——是相关的页面。

**来源层级（按权威性排序）：**

| 优先级 | 来源 | 示例 |
|----------|--------|---------|
| 1 | 官方文档 | react.dev、docs.djangoproject.com、symfony.com/doc |
| 2 | 官方博客 / 更新日志 | react.dev/blog、nextjs.org/blog |
| 3 | Web 标准参考资料 | MDN、web.dev、html.spec.whatwg.org |
| 4 | 浏览器 / 运行时兼容性 | caniuse.com、node.green |

**不作为权威来源——切勿作为主要来源引用：**

- Stack Overflow 答案
- 博客文章或教程（即便是热门的）
- AI 生成的文档或摘要
- 你自己的训练数据（这正是本技能的核心理念——验证它）

**精确获取你要的内容：**

```
BAD:  获取 React 首页
GOOD: 获取 react.dev/reference/react/useActionState

BAD:  搜索 "django 认证 最佳实践"
GOOD: 获取 docs.djangoproject.com/en/6.0/topics/auth/
```

获取后，提取关键模式并注意任何弃用警告或迁移指引。

当官方来源之间相互矛盾时（例如迁移指南与 API 参考不一致），向用户暴露此差异，与检测到的版本对照验证哪种模式确实有效。

### 步骤 3：Implement（按文档模式实现）

编写与文档展示相匹配的代码：

- 使用文档中的 API 签名，而非凭记忆
- 如果文档展示了做某件事的新方式，使用新方式
- 如果文档弃用了某个模式，不要使用已弃用的版本
- 如果文档未覆盖某项内容，将其标记为未经验证

**当文档与现有项目代码冲突时：**

```
检测到冲突：
现有代码库使用 useState 处理表单加载状态，
但 React 19 文档推荐使用 useActionState 处理此模式。
（来源：react.dev/reference/react/useActionState）

选项：
A) 使用现代模式（useActionState）——与当前文档一致
B) 匹配现有代码（useState）——与代码库一致
→ 你偏好哪种方式？
```

暴露冲突。不要悄悄选择其中一种。

### 步骤 4：Cite（引用你的来源）

每个框架特定模式都要附带引用。用户必须能够验证每个决策。

**在代码注释中：**

```typescript
// React 19 表单处理，使用 useActionState
// 来源：https://react.dev/reference/react/useActionState#usage
const [state, formAction, isPending] = useActionState(submitOrder, initialState);
```

**在对话中：**

```
我使用 useActionState 而非手动 useState 处理表单提交状态。
React 19 用这个 Hook 取代了手动的 isPending/setIsPending 模式。

来源：https://react.dev/blog/2024/12/05/react-19#actions
"useTransition now supports async functions [...] to handle
pending states automatically"
```

**引用规则：**

- 使用完整 URL，不使用短链接
- 尽可能使用含锚点的深层链接（如 `/useActionState#usage` 优于 `/useActionState`）——锚点在文档重构时比顶级页面更持久
- 当支持一个不显然的决策时，引用相关段落
- 当推荐平台特性时，包含浏览器 / 运行时支持数据
- 如果找不到某个模式的文档，明确说明：

```
未经验证：我找不到此模式的官方文档。此内容基于训练数据，
可能已过时。在生产环境使用前请验证。
```

诚实承认无法验证的内容，比虚假自信更有价值。

## 常见借口

| 借口 | 现实 |
|---|---|
| "我对这个 API 很有把握" | 自信不等于证据。训练数据中包含过时的模式，它们看起来正确，但在当前版本上会出错。请验证。 |
| "获取文档浪费 token" | 幻觉出一个 API 浪费更多。用户调试一小时，然后发现函数签名变了。一次文档获取可以避免数小时的返工。 |
| "文档里不会有我需要的东西" | 如果文档没有覆盖，这本身就是有价值的信息——该模式可能不是官方推荐的做法。 |
| "我提一句'可能已过时'就行" | 免责声明没有实际帮助。要么验证并引用，要么明确标记为未经验证。含糊其辞是最差的选择。 |
| "这是简单任务，无需查询" | 使用错误模式的简单任务会成为模板。用户会把你有问题的表单处理代码复制到十个组件中，才发现存在现代做法。 |

## 红旗信号

- 编写框架特定代码前未查阅该版本的文档
- 对某个 API 使用"我相信"或"我觉得"，而非引用来源
- 在不知道模式适用于哪个版本的情况下实现
- 引用 Stack Overflow 或博客文章而非官方文档
- 因为训练数据中包含而使用已弃用的 API
- 实现之前没有阅读 `package.json` / 依赖文件
- 交付代码时对框架特定决策缺少来源引用
- 只需要一个页面却获取了整个文档站点

## 验证

使用源码驱动开发实现后：

- [ ] 框架和库的版本已从依赖文件中识别
- [ ] 框架特定模式已从官方文档中获取
- [ ] 所有来源均为官方文档，非博客文章或训练数据
- [ ] 代码遵循当前版本文档展示的模式
- [ ] 非平凡的决策包含带完整 URL 的来源引用
- [ ] 未使用已弃用的 API（已对照迁移指南检查）
- [ ] 文档与现有代码之间的冲突已向用户暴露
- [ ] 任何无法验证的内容已明确标记为未经验证
