0001 — Web UI 交互全链路地图

学习目标：能画出从股票代码输入到报告展示的完整 6 阶段交互链路，标出每个阶段的线程归属（Streamlit 主线程 / daemon 后台线程），并解释 5 个 UI 状态之间的转换条件。

1. 交互全链路：6 阶段总览

用户一次完整的"输入→等待→查看报告"交互，跨越 6 个阶段、两条线程。理解线程归属是理解整个链路的关键——Streamlit 的请求-响应模型不能阻塞主线程，所以耗时分析必须在 daemon 线程中异步执行。

阶段 1 用户输入[主线程]

sidebar.py：用户输入 ticker（300750 或 宁德时代）→ _resolve_user_input() 调 resolve_ticker() 转 6 位代码 → 点击"开始分析"→ 构造 start_analysis dict 推入 st.session_state

阶段 2 调度启动[主线程]

app.py:189-212：pop start_analysis → 可选清理旧断点 → 创建 ProgressTracker 存入 session_state → 调用 run_analysis_in_thread() 启动 daemon 线程

阶段 3 后台执行[后台线程]

runner.py:_run()：实例化 TradingAgentsGraph → graph.stream() 逐 chunk 迭代 → 每 chunk 调 _detect_completed_stages() 更新 tracker → 检查 stop/pause 信号 → 完成后调 finalize_graph_run() 落盘

阶段 4 进度轮询[主线程]

app.py:232-235：检测 tracker.is_running → render_progress(tracker) 渲染 12 阶段进度条 + 统计指标 → time.sleep(2) → st.rerun() 重新执行脚本以读取最新 tracker 状态

阶段 5 报告展示[主线程]

app.py:238-245：tracker.is_complete 为真 → render_report() 渲染信号卡片（BUY/SELL/HOLD）+ 7 个分析师 expander + 多空辩论 tabs + 风控评估 tabs

阶段 6 报告导出[主线程]

report_viewer.py:91-118：Markdown 导出（零依赖、总是可用）→ PDF 导出（generate_pdf()，需要 CJK 字体，失败时按钮置灰不崩溃）

2. 5 状态机：界面切换逻辑

app.py:216-308 的核心是一个 if-elif 链，以 tracker 和 viewing_history 两个 session_state 变量为判断依据。每个状态渲染不同的 UI 组件：

0 空闲
tracker = None
viewing_history = None
欢迎页 HTML
1 查看历史
viewing_history != None
render_report() 加载旧 JSON
2 分析中
tracker.is_running = True
render_progress() + 2s rerun 轮询
3 完成
tracker.is_complete = True
render_report() + 下载按钮
4 出错
tracker.error != None
错误提示 + 继续按钮

状态 1（查看历史）的优先级最高——它在 if-elif 链的最前面，一旦用户点击侧边栏中的历史记录，无论当前 tracker 是什么状态，都会跳转到历史查看模式。状态 2→3 的切换不需要额外的消息通知：后台线程将 tracker.is_complete = True 后，下次 Streamlit rerun 时自然会进入状态 3。

3. 暂停/恢复/停止的信号路径

这三个控制操作都从侧边栏（sidebar.py）的按钮发起，但信号传递路径各不相同：

操作UI 线程动作后台线程检测点结果
暂停 tracker.pause() → Lock 内设置 is_paused=True → _pause_gate.clear() 关闭闸门 下一轮 wait_if_paused() 时 _pause_gate.wait() 阻塞 后台线程挂起，已经开始的 stream 步骤继续完成
恢复 tracker.resume() → Lock 内设置 is_paused=False → _pause_gate.set() 打开闸门 _pause_gate.wait() 返回，继续下一轮循环 从下一个 stream chunk 继续执行
停止 tracker.request_stop() → Lock 内设置 stop_requested=True 并清空 UI 状态 每轮循环开头检测 stop_requested，调用 _discard_stopped_run() 清理断点+缓存，tracker 回到 stopped 状态

暂停和停止的关键区别：暂停保留已完成阶段以便恢复；停止清空所有状态并删除断点文件，下次开始分析会从头生成。

检索练习

问题 1：用户点击"开始分析"后，在 tracker 对象创建之前，app.py 还做了什么准备工作？

问题 2：为什么状态 2（分析中）需要 time.sleep(2) + st.rerun()？如果去掉这个组合会发生什么？

问题 3：暂停和停止在数据清理上有什么区别？为什么暂停时不去调用 _discard_stopped_run()？

参考答案
1：检查 start_req.fresh 标志——如果为 True，调用 clear_incomplete_task() 删除旧未完成任务记录，然后 clear_checkpoint() 清理 LangGraph 断点缓存。这确保"开始分析"是真正的从头开始。如果是从中断恢复（fresh 为 False），则跳过清理步骤。
2：去掉 sleep 会导致无限循环——每次 rerun 立即再次进入状态 2 并再次 rerun，CPU 100% 且 Streamlit 无法响应其他操作。sleep(2) 给后台线程 2 秒时间推进分析进度，也防止过度渲染。如果去掉 rerun，页面将停留在第一次渲染的静态快照，不会更新进度条。
3：暂停保留 completed_stages 和 stage_reports，并持久化到 incomplete_tasks.json，恢复后可从中断处继续。停止调用 request_stop() 清空所有 UI 状态，后台线程检测后调用 _discard_stopped_run() 同时删除 incomplete_tasks 记录和 LangGraph checkpoint——相当于完全重置。不删断点的话用户下次启动会误以为有可恢复的任务。
