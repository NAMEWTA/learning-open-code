0002 — 输入→分析→展示主成功路径

学习目标：能逐步骤走通从 ticker 解析、后台线程启动、stream chunk 阶段检测、到 finalize 落盘、最终报告渲染的完整主成功路径（happy path），理解每个步骤的输入/输出和线程上下文。

1. 入口：ticker 解析与线程启动

主路径的起点在侧边栏的"开始分析"按钮（sidebar.py:235-253）。用户输入可以是 6 位代码（300750）或中文名（宁德时代），由 _resolve_user_input() 统一转为代码：

Step 1 — ticker 解析

# sidebar.py:241
resolved_code, err = _resolve_user_input(ticker)
# 调用链：resolve_ticker() → _build_name_code_map()
# 通过 mootdx TCP 查询全市场股票名称映射表（有缓存）
# 成功返回 6 位代码，失败返回 (None, "未找到该股票")

解析成功后，将 {"ticker": code, "trade_date": ..., "fresh": True} 推入 st.session_state["start_analysis"]。注意：此时并未直接启动分析——Streamlit 的脚本执行模型要求先完成当前 render 周期，下一次 rerun 时 app.py:189 才会 pop 这个 dict。

Step 2 — app.py 调度

# app.py:189-212
start_req = st.session_state.pop("start_analysis", None)
if start_req:
    if start_req.get("fresh"):          # 仅全新开始才清理
        clear_incomplete_task(...)       # 删 ~/.tradingagents/incomplete_tasks.json
        clear_checkpoint(...)            # 删 LangGraph checkpoint 目录
    tracker = ProgressTracker(ticker=..., trade_date=...)
    st.session_state["tracker"] = tracker
    st.session_state["viewing_history"] = None
    run_analysis_in_thread(ticker, trade_date, config, tracker)

这里 fresh=True 保证"开始分析"按钮始终清理旧状态，而侧边栏"未完成任务"的恢复按钮传 fresh=False，利用已有 checkpoint 从上次中断处继续。

2. 后台执行：stream 遍历与阶段检测

runner.py:_run() 是主路径中唯一跑在 daemon 线程里的函数。它的核心是一个 while 循环遍历 graph.stream() 返回的迭代器：

Step 3 — graph 流式迭代

# runner.py:120-146
stream = graph.graph.stream(init_state, **args)   # LangGraph astream
while True:
    tracker.wait_if_paused()        # 暂停闸门检查
    if tracker.stop_requested:      # 停止信号检查
        _close_and_discard()
        return
    chunk = next(stream)            # 阻塞等待下一个 chunk
    last_chunk = chunk
    _detect_completed_stages(chunk, tracker)  # 检测新完成的阶段
    _infer_active_stage(tracker)    # 推断当前活跃阶段
    record_incomplete_task(...)     # 持久化断点供崩溃恢复
    stats = graph.stats_handler.get_stats()
    tracker.update_stats(llm_calls, tool_calls, tokens_in, tokens_out)

_detect_completed_stages() 是这个循环的核心逻辑——它不是等 graph 跑完才一次性标记，而是每收到一个 chunk 就检查 12 个字段：7 个 analyst report_key、quality_gate、debate judge_decision、trader_plan、risk judge_decision、final_decision。哪个字段首次出现非空内容，就标记对应阶段为 done。

关键设计洞察：12 个阶段不是串行等待的。前 7 个 analyst 在 LangGraph 中可能是并行 fan-out 执行，所以一个 chunk 可能同时包含多个 report_key 的内容。_detect_completed_stages() 用 if content and tracker.stage_status(id) != "done" 保证每个阶段只标记一次。

stream 耗尽（StopIteration）后，进入 finalize 阶段：

Step 4 — finalize 与落盘

# runner.py:152-164
normalize_report_state_mentions(last_chunk, ticker)  # #55: 归一化股票名
signal = graph.finalize_graph_run(ticker, trade_date, last_chunk)
# 内部：investment_plan → trader → 提取 BUY/SELL/HOLD 信号
# 将 last_chunk 写入 ~/.tradingagents/logs/{ticker}/{date}/full_states_log_{date}.json
tracker.mark_complete(last_chunk, signal)  # 设置 is_running=False, is_complete=True
clear_incomplete_task(ticker, trade_date)  # 清理断点记录

3. 前端渲染：从 tracker 到报告

后台线程调用 tracker.mark_complete() 之后，Streamlit 主线程的下一次 st.rerun() 会走到 app.py:238 的完成分支：

Step 5 — 报告渲染

# app.py:238-245
elif tracker and tracker.is_complete:
    render_report(
        tracker.final_state,   # last_chunk 字典（所有 12 个 stage 的输出）
        tracker.ticker,
        tracker.trade_date,
        tracker.signal,        # "BUY" / "SELL" / "HOLD"
        elapsed=tracker.elapsed,
    )

render_report()（report_viewer.py）的渲染顺序：

  1. 信号卡片：渐变色背景 + BUY(绿)/SELL(红)/HOLD(黄) 大字 + ticker 标签 + 耗时
  2. 最终投资建议（如有）：investment_plan 字段完整展示
  3. 7 个分析师报告：每个用 st.expander 折叠，默认关闭
  4. 多空辩论：3 个 tabs（多方/空方/研究经理）
  5. 交易员决策：独立 expander
  6. 风控评估：4 个 tabs（激进/保守/中性/风控决策）
  7. 数据质量摘要：独立 expander

4. 导出：两条路径与降级策略

报告展示后，用户可通过两个下载按钮导出。关键设计原则：Markdown 是可靠路径，PDF 是增强路径。

格式函数依赖失败处理
Markdown generate_markdown() 零外部依赖，纯字符串拼接 总是成功
PDF generate_pdf() fpdf2 + 系统 CJK 字体（文泉驿/Noto/PingFang/微软雅黑） catch 异常后按钮置灰 + tooltip 提示原因，不崩溃页面

PDF 导出的字体探测顺序（pdf_export.py:30-75）：先检查环境变量 TRADINGAGENTS_PDF_FONT，再按内置候选列表依次尝试 Linux → macOS → Windows 路径，接着用 fc-match 命令行兜底，最后遍历 /usr/share/fonts 目录递归搜索。如果全部失败，抛出 PDFExportError 建议安装 fonts-wqy-microhei。

检索练习

问题 1：描述 _detect_completed_stages() 如何判断一个阶段是否已完成——它依赖的输入是什么？用哪个条件保证不会重复标记？

问题 2：为什么 fresh=True 和 fresh=False 在 app.py 中有不同的清理行为？这两种场景分别对应用户的什么操作？

问题 3：PDF 导出失败时为什么不让整个页面崩溃？代码中具体如何处理这个异常？

参考答案
1：输入是 graph.stream() 返回的当前 chunk 字典。对每个 report_key 检查两个条件：(a) chunk.get(report_key, "") 非空——说明 LLM 已经产出了该阶段的报告内容；(b) tracker.stage_status(stage_id) != "done"——该阶段尚未被标记过。两个条件同时满足才调用 tracker.mark_stage_done()，这防止了同一阶段在后续 chunk 中再次出现时重复标记。
2：fresh=True 对应用户点击"开始分析"按钮——全新分析，需要清除旧的 incomplete_tasks 记录和 checkpoint 目录。fresh=False 对应用户点击侧边栏"未完成任务"中的恢复按钮——从中断处继续，保留已有 checkpoint 让 LangGraph 自动从中断节点恢复。不清理 checkpoint 是恢复能成功的关键。
3：report_viewer.py:112-118 用 try-except 包裹 generate_pdf()，捕获所有异常（except Exception as exc），失败时不渲染下载按钮，改为渲染一个 disabled 按钮显示"PDF 不可用"，tooltip 中说明原因并引导用户改用 Markdown 导出。try-except 在整个 render_report 之外，所以 PDF 失败不影响报告展示和 Markdown 下载。
