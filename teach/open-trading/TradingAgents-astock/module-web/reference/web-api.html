2. 通信模型：Thread + Stream + 传播控制

Web 模块采用共享内存 + 轮询的线程通信模型，不依赖 queue.Queue。后台线程写入共享的 ProgressTracker 对象，Streamlit UI 线程每轮 st.rerun() 时读取该对象。

2.1 线程生命周期

用户点击"开始分析"
  │
  ▼
app.py: start_analysis 触发
  │
  ├─ 创建 ProgressTracker(ticker, trade_date)
  ├─ 存入 st.session_state["tracker"]
  │
  ▼
run_analysis_in_thread(ticker, trade_date, config, tracker)
  │
  ├─ tracker.is_running = True
  ├─ tracker.mark_stage_active("market")
  ├─ record_incomplete_task(...)
  │
  ├─ threading.Thread(target=_target, daemon=True).start()
  │     │
  │     ▼
  │   _run() — 在后台线程中执行
  │     │
  │     ├─ 创建 TradingAgentsGraph 实例
  │     ├─ graph.prepare_graph_run() → init_state, args
  │     ├─ graph.graph.stream(init_state) → 迭代器
  │     │
  │     ├─ while True:
  │     │     ├─ tracker.wait_if_paused()    ← Event 阻塞点
  │     │     ├─ if tracker.stop_requested → _close_and_discard() → return
  │     │     ├─ chunk = next(stream)        ← 可能 StopIteration
  │     │     ├─ _detect_completed_stages(chunk, tracker)
  │     │     ├─ _infer_active_stage(tracker)
  │     │     ├─ record_incomplete_task(...) ← 断点续跑
  │     │     └─ tracker.update_stats(...)
  │     │
  │     ├─ normalize_report_state_mentions(last_chunk, ticker)
  │     ├─ graph.finalize_graph_run(ticker, trade_date, last_chunk) → signal
  │     └─ tracker.mark_complete(last_chunk, signal)
  │
  ▼
  返回 threading.Thread 句柄（不 join，不阻塞 UI 线程）

2.2 停止传播路径

停止请求从 UI 线程发起，通过以下路径传播到后台线程：

UI 线程                                   后台线程
────────                                  ────────
用户点击"停止"
  │
  ▼
tracker.request_stop()
  ├─ Lock 保护
  ├─ stop_requested = True
  ├─ 清空 UI 可见状态
  └─ _pause_gate.set() ──────────────►  _pause_gate.wait() 解除阻塞
                                          │
                                          ▼
                                        tracker.stop_requested == True?
                                          │ YES
                                          ▼
                                        _close_and_discard()
                                          ├─ graph.close_graph_run()
                                          └─ _discard_stopped_run()
                                               ├─ clear_incomplete_task()
                                               ├─ clear_checkpoint()
                                               └─ tracker.mark_stopped()

异常路径: _target() 捕获 Exception
  ├─ if tracker.stop_requested → _discard_stopped_run() → return
  └─ else → record_incomplete_task(status="error") + tracker.mark_error()

关键设计原则：

  • 多重检查点：stream 循环的每次迭代前/后、stream next() 后、finalize 前都检查 stop_requested，确保停止信号不丢失。
  • Daemon 线程：线程设为 daemon=True，Streamlit 进程退出时自动终止，不会残留阻塞线程。
  • 异常双路径：正常停止和异常崩溃走不同清理逻辑——停止时丢弃断点（纯清理），异常时保留断点（可续跑）。

2.3 Stage 检测机制

_detect_completed_stages(chunk, tracker) 通过检查流式 chunk 中的字段来判断阶段完成。映射关系：

chunk 字段阶段 ID检测条件
market_report ~ lockup_report（7 个报告键）market ~ lockup字段非空 且 该阶段未完成
data_quality_summaryquality_gate字段非空 且 未完成
investment_debate_state (dict)debate含有 judge_decision 非空 且 未完成
trader_investment_plantrader字段非空 且 未完成
risk_debate_state (dict)risk含有 judge_decision 非空 且 未完成
final_trade_decisionpm字段非空 且 未完成

每检测到一个完成阶段，调用 tracker.mark_stage_done(stage_id, report)，同时执行 normalize_stock_mentions() 和 _strip_think_tags() 清洗文本。

3. 线程安全设计：Event + Lock 暂停/停止机制

核心挑战：ProgressTracker 是一个 @dataclass 可变状态容器，同时被两个线程访问——后台线程写入（mark_*、update_stats），UI 线程读取（stage_status、elapsed）和控制（pause、resume、request_stop）。必须保证线程安全且不引入死锁。

3.1 双原语分工

原语类型用途初始化
_lockthreading.Lock保护所有可变字段的读写原子性field(default_factory=threading.Lock)
_pause_gatethreading.Event后台线程的阻塞/放行闸门field(default_factory=threading.Event) + __post_init__ 中 .set()
关键细节：_pause_gate 使用 init=False 构造（默认 cleared），然后在 __post_init__ 中立即 .set()。这确保初始化完成后门是开的（不阻塞），只有显式调用 pause() 才会关门。

3.2 暂停/恢复时序

┌─ pause() ──────────────────────────────────────────────────┐
│  with self._lock:                                          │
│    guard: not running / complete / error / paused / stop   │
│    → 任一为真则 return False（不允许暂停）                    │
│    self.is_paused = True                                   │
│    self._pause_gate.clear()   ← 关门                        │
│  return True                                               │
└────────────────────────────────────────────────────────────┘

┌─ resume() ─────────────────────────────────────────────────┐
│  with self._lock:                                          │
│    guard: not is_paused or stop_requested                  │
│    → return False                                          │
│    self.is_paused = False                                  │
│    self._pause_gate.set()      ← 开门                        │
│  return True                                               │
└────────────────────────────────────────────────────────────┘

┌─ wait_if_paused() ── （后台线程调用）───────────────────────┐
│  self._pause_gate.wait()     ← 阻塞直到 set() 或 stop 触发   │
│  （无 Lock，因为 Event.wait() 本身线程安全）                 │
└────────────────────────────────────────────────────────────┘

3.3 request_stop() 的激进清空策略

request_stop() 不仅设置标志位，还立即清空所有 UI 可见状态（current_stage、completed_stages、stage_reports、final_state、统计数据），同时 _pause_gate.set() 解除后台线程的阻塞。这确保停止操作有即时视觉反馈，且后台线程能尽快退出阻塞点检测到 stop_requested。

3.4 Lock 保护范围

以下方法均在 with self._lock: 保护下执行：

方法锁内操作
pause()读 is_running/is_complete/error/is_paused/stop_requested → 写 is_paused + clear Event
resume()读 is_paused/stop_requested → 写 is_paused + set Event
request_stop()读状态守卫 → 写 stop_requested + 清空 10+ 字段 + set Event
mark_stopped()重置所有字段至初始值 + set Event
mark_stage_active()读 stop_requested → 写 current_stage
mark_stage_done()读 stop_requested → 写 completed_stages + stage_reports + current_stage
mark_complete()写 final_state/signal + 切换 is_running→is_complete + set Event
mark_error()写 error + 切换 is_running→False + set Event
update_stats()读 stop_requested → 写 4 个统计字段
stage_status()读 completed_stages/current_stage → 返回 "done"/"active"/"pending"

不持锁的操作：wait_if_paused() 只调用 Event.wait()（自带内部锁）；elapsed 属性读取 start_time（float 赋值在 CPython 中是原子操作，但不保证内存可见性——此处容忍微小误差）。

4. 中文字体适配：fpdf2 字体探测全链路

背景：fpdf2 生成 PDF 时需要可嵌入的 TTF/OTF/TTC 字体文件。中文（CJK）字符集远大于 latin-1，必须使用专门的中文字体。常见问题：安装了旧版 pyfpdf（latin-1 编码）导致 UnicodeEncodeError（issue #54）；系统未安装中文字体导致 PDF 导出失败。

4.1 库版本检测

_ensure_fpdf2() 在 PDF 生成前首先检查 fpdf 版本：

fpdf.__version__ 或 fpdf.FPDF_VERSION
  │
  ├─ major >= 2 → 通过（fpdf2 维护分支）
  └─ major < 2  → RuntimeError:
      "检测到旧版 fpdf (pyfpdf)，请执行:
       pip uninstall -y fpdf && pip install 'fpdf2>=2.8.0'"

4.2 字体探测四层优先级

_find_cjk_fonts() → (regular_font_path, bold_font_path)

Layer 1: 环境变量（最高优先级）
  TRADINGAGENTS_PDF_FONT=/path/to/regular.ttf
  TRADINGAGENTS_PDF_BOLD_FONT=/path/to/bold.ttf   ← 可选，缺失则复用 regular

Layer 2: 预置候选路径（11 组，覆盖三大平台）
  Linux:
    /usr/share/fonts/truetype/wqy/wqy-microhei.ttc     ← 首选
    /usr/share/fonts/truetype/wqy/wqy-zenhei.ttc
    /usr/share/fonts/opentype/noto/NotoSansCJK-Regular.ttc
    /usr/share/fonts/noto-cjk/NotoSansCJK-Regular.ttc
    /usr/share/fonts/noto-cjk/NotoSansCJKsc-Regular.otf
    /usr/share/fonts/truetype/noto/NotoSansSC-Regular.ttf
    /usr/share/fonts/truetype/droid/DroidSansFallbackFull.ttf

  macOS:
    /System/Library/Fonts/PingFang.ttc
    /System/Library/Fonts/STHeiti Light.ttc → STHeiti Medium.ttc (Bold)

  Windows:
    C:/Windows/Fonts/msyh.ttc  → msyhbd.ttc (Bold)
    C:/Windows/Fonts/simhei.ttf

Layer 3: 文件系统搜索 (_discover_cjk_fonts)
  搜索根目录:
    /usr/share/fonts
    /usr/local/share/fonts
    ~/.local/share/fonts
    ~/.fonts
    $XDG_DATA_HOME/fonts
  匹配文件模式（按优先级）:
    wqy-microhei.ttc → wqy-zenhei.ttc → NotoSansCJK-Regular.ttc
    → NotoSansCJKsc-Regular.otf → NotoSansSC-Regular.ttf
    → SourceHanSansSC-Regular.otf → DroidSansFallbackFull.ttf

Layer 4: fontconfig 系统查询 (_font_from_fontconfig)
  fc-match -f '%{file}' 'WenQuanYi Micro Hei:style=Regular'
  fc-match 'Noto Sans CJK SC:style=Regular'
  fc-match 'Noto Sans SC:style=Regular'
  fc-match 'Source Han Sans SC:style=Regular'
  （依次尝试，直到找到一个 CJK 字体）

全部失败 → PDFExportError("PDF 导出需要可嵌入的 Unicode 中文字体...")

4.3 TTC 字体集合处理

TTC（TrueType Collection）文件包含多个字体 face。_collection_font_number(path) 根据文件名返回对应的简体中文 face 索引：

TTC 文件Face 索引说明
NotoSansCJK-Regular.ttc2索引 2 为 SC（简体中文）变体
NotoSansCJK-Bold.ttc2同上
NotoSerifCJK-Regular.ttc2衬线 SC 变体
NotoSerifCJK-Bold.ttc2同上
其他（PingFang、msyh 等）0默认取第一个 face

4.4 单文件加粗回退

部分中文字体（wqy-microhei.ttc、wqy-zenhei.ttc、DroidSansFallbackFull.ttf）只有一个文件，不区分 Regular/Bold。对于这些字体，regular 和 bold 使用同一个文件路径，由 fpdf2 的样式引擎模拟加粗。

4.5 降级策略

当 PDF 生成因字体缺失失败时，系统不崩溃，而是：

  • report_viewer.py 中捕获 Exception，将 PDF 下载按钮替换为禁用的 "PDF 不可用" 按钮，并在 tooltip 中显示原因。
  • Markdown 导出（generate_markdown()）始终可用，不依赖任何字体文件——作为安全兜底方案。

名称解析链路：

  1. 输入标准化：去 .SH/.SZ/.BJ 后缀，去 SH/SZ/BJ 前缀，正则验证 6 位代码
  2. 如果含中文 → resolve_ticker() 查询 mootdx 全市场映射
  3. resolve_stock_name()：通过 _build_name_code_map()（LRU 缓存 1024 条）查询代码→名称
  4. 如果本地数据不可用 → 从 final_state 的文本字段按 3 种正则模式提取（"代码(名称)"、"标的：代码 名称"、"名称(代码)"）
  5. 提取的名称经过清洗：去除不可打印字符、截断报告标题关键词（"技术分析报告"等）、去除标点

normalize_stock_mentions() 是幂等的：已格式化过的 "300750 宁德时代" 不会被重复替换。通过检测前后文避免双重归一化。
