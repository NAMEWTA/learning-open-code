2. UI 五状态机

2 分析中tracker.is_runningrender_progress()每 2s st.rerun() 刷新

状态切换通过 st.session_state 驱动：侧边栏按钮写入 start_analysis 字典 → app.py 检测到后创建 tracker + 启动线程 → 下一轮渲染匹配对应状态。

4. ProgressTracker API

方法线程作用
mark_stage_active(stage_id)后台设置当前活跃阶段
mark_stage_done(stage_id, report)后台标记阶段完成，存储报告文本
mark_complete(final_state, signal)后台设置 is_complete=True，存储最终结果
mark_error(err)后台设置 error 字段，停止运行
update_stats(llm, tool, tok_in, tok_out)后台更新统计指标
pause() -> boolUI暂停流水线，clear _pause_gate Event
resume() -> boolUI恢复流水线，set _pause_gate Event
request_stop() -> boolUI请求停止并立即清空 UI 状态
wait_if_paused()后台阻塞等待直到 resume() 或 stop
mark_stopped()后台完全重置至初始状态
stage_status(stage_id) -> strUI返回 "done"/"active"/"pending"
elapsed -> floatUI只读属性：距 start_time 的秒数

关键字段：is_running、is_complete、is_paused、stop_requested、error、current_stage、completed_stages、stage_reports、final_state、signal。

5. 组件三层架构

sidebar.py — 侧边栏组件

函数职责
render_sidebar()侧边栏入口：品牌标题 + 输入区 + 配置区 + 历史列表
_resolve_user_input(raw) -> (code, error)将用户输入解析为 6 位代码（支持中文名）
_render_llm_config()LLM 供应商选择 + 快/慢模型选择 + 代理地址
_render_analysis_controls()暂停/恢复/停止三个按钮的控制逻辑

10 个 LLM 供应商：MiniMax（推荐）、DeepSeek、Qwen、GLM、OpenAI、Anthropic、Google、xAI、OpenRouter、Ollama。模型选项从 MODEL_OPTIONS 动态读取。

progress_panel.py — 进度面板组件

函数职责
render_progress(tracker)渲染全幅进度面板：阶段图标网格 + 进度条 + 4 项统计指标 + 已完成报告折叠预览
_status_badge(status)阶段状态图标：●(done/绿) / ◉(active/橙) / ○(pending/灰)
_format_time(seconds)秒数 → m:ss 格式

分为上下两排：ANALYSTS 7 列 + PIPELINE 5 列。已完成阶段显示为绿色圆点，活跃阶段为橙色带脉冲感圆点，等待阶段为灰色空心圆。下方用 4 个 st.metric 显示 LLM 调用数、工具调用数、输入/输出 token 数。

report_viewer.py — 报告展示组件

函数职责
render_report(final_state, ticker, trade_date, signal, elapsed)全幅报告渲染：信号大卡片 + 投资建议 + 7 个分析师折叠区 + 辩论 tabs + 风控 tabs + 下载按钮
_signal_style(signal)根据 BUY/SELL/HOLD 返回颜色和中文名
_display_report_text(text, ticker, state)去除 think 标签 + 归一化股票名后渲染

下载区提供两种格式：Markdown（总是可用，无字体依赖）和 PDF（需要 CJK 字体，失败时不崩溃而是禁用按钮并提示原因）。

6. 历史记录 API

函数返回值说明
get_history()list[dict]扫描 ~/.tradingagents/logs/ 下所有 full_states_log_*.json，按日期倒序
get_incomplete_history()list[dict]返回未完成任务列表，自动过滤已完成任务，附加 checkpoint_step
record_incomplete_task(ticker, date, status, error, completed_stages)None写入/更新 ~/.tradingagents/incomplete_tasks.json
clear_incomplete_task(ticker, date)None从 incomplete_tasks.json 中删除指定条目
load_analysis(path) -> dictdict加载完整分析 JSON 文件
extract_signal(state) -> str"Buy"/"Sell"/"Hold"/"N/A"从 state 字典中提取交易信号（去 think 标签后匹配关键词）

存储位置：已完成 → ~/.tradingagents/logs/{ticker}/{date}/full_states_log_{date}.json；未完成 → ~/.tradingagents/incomplete_tasks.json。所有写操作使用 tempfile + replace 保证原子性。

7. PDF 导出 API

函数返回值说明
generate_pdf(final_state, ticker, trade_date, signal) -> bytesbytes生成 PDF 字节流（封面 + 全部分区 + 页眉页脚）
generate_markdown(final_state, ticker, trade_date, signal) -> strstr生成 Markdown 文本（无字体依赖的安全导出）
_find_cjk_fonts() -> (Path, Path)tuple按环境变量 → 预置路径 → 文件系统搜索 → fontconfig 的顺序探测中文字体
_collect_sections(final_state, ticker)list[tuple]组装 (标题, 内容) 分区列表，PDF 和 Markdown 共享同一来源
_ensure_fpdf2()None检测 fpdf2 >= 2.0（排除旧版 pyfpdf），不符合则抛 RuntimeError

字体探测优先级：TRADINGAGENTS_PDF_FONT 环境变量 → 预置 11 组候选路径（覆盖 Linux/macOS/Windows）→ _discover_cjk_fonts() 文件系统搜索 → fc-match fontconfig 查询。TTC 字体集合使用 collection_font_number 参数选择简体中文 face（索引 2）。

8. 股票名展示 API

函数返回值说明
stock_display_label(ticker, final_state)str返回 "代码 名称" 格式，无法解析则只返回代码
stock_display_parts(ticker, final_state)(code, name)返回代码和名称的元组
normalize_stock_mentions(text, ticker, state)str将报告中的代码/名称统一为 "代码 名称" 格式，避免重复显示
normalize_report_state_mentions(state, ticker)dict对整个 final_state 中所有报告字段做名称归一化（在落盘前调用）
resolve_stock_name(ticker)str | None通过本地市场数据查询代码对应的 A 股名称

名称解析链路：LB 缓存（lru_cache）→ 正则验证 6 位代码 → _build_name_code_map()（mootdx 全市场映射）。如果本地数据不可用，从 final_state 的文本字段中通过正则提取（支持 "代码(名称)"、"标的：代码 名称" 等 3 种模式）。
