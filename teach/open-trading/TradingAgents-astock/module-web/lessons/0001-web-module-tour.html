0001 — Streamlit Web UI 模块总览

学习目标：能画出 Web 模块的 5 状态 UI 主循环和组件三层架构，理解 ProgressTracker 如何通过 Thread + Event + Lock 实现后台线程与 Streamlit 前端的线程安全通信，并说出暂停/恢复/停止控制流的技术要点。

1. 模块职责与文件结构

web/ 模块是 TradingAgents-Astock 的浏览器端交互入口。它不包含任何分析逻辑，只负责将后台分析流水线的进度和结果以可视化方式呈现。核心职责：接收用户输入（股票代码+日期+LLM 配置），启动后台线程执行 Agent 系统 + LangGraph 工作流，实时展示 12 阶段进度，渲染最终报告，支持 PDF/Markdown 导出和历史回溯。

web/ ├── app.py ← 主页面：5 状态机 + 自定义 CSS ├── runner.py ← 后台线程：Thread + TradingAgentsGraph ├── progress.py ← 进度追踪：ProgressTracker + Event/Lock ├── history.py ← 历史管理：扫描 / 恢复 / 清理 ├── pdf_export.py ← PDF 报告：fpdf2 + 中文字体自动探测 ├── launch.py ← CLI 启动器：streamlit run 封装 ├── stock_display.py ← 股票名展示：代码↔名称解析 └── components/ ← UI 组件三层架构 ├── sidebar.py 侧边栏（输入+配置+历史） ├── progress_panel.py 实时进度面板 └── report_viewer.py 报告展示+导出按钮
依赖关系 上游依赖：tradingagents.graph（工作流引擎）、tradingagents.dataflows（股票名解析）、tradingagents.llm_clients（模型配置选项）。web 模块自身只做 UI 编排，不直接调用 LLM 也不访问数据库。

2. UI 主循环：5 状态机

app.py 的核心是一个基于 st.session_state 的状态机，通过 tracker 对象的状态驱动 5 种界面渲染：

状态触发条件渲染组件
0 空闲tracker 为 None欢迎页：品牌标题 + 操作提示
1 查看历史viewing_history 不为空render_report() 加载旧 JSON
2 分析中tracker.is_runningrender_progress() + 每 2 秒 st.rerun()
3 完成tracker.is_completerender_report() 展示结果 + 下载按钮
4 出错tracker.error 不为空错误提示 + "继续未完成任务"按钮

关键设计：状态 2 中 time.sleep(2) + st.rerun() 形成一个轮询循环——Streamlit 本身是请求-响应模型，不支持服务端推送，所以通过周期性 rerun 来刷新进度显示。这不是最优雅的方案，但在 Streamlit 的单进程架构下是最简单可行的。

3. 后台线程模型与进度追踪

这是整个 Web 模块最核心的设计——如何在 Streamlit 的无状态请求模型中运行一个长时间后台任务并实时汇报进度。

3.1 线程启动流程

runner.py 的 run_analysis_in_thread() 创建一个 daemon 线程执行完整流水线。同时创建一个 ProgressTracker 对象存入 st.session_state，作为 UI 和后台线程的共享内存桥梁：

def run_analysis_in_thread(ticker, trade_date, config, tracker):
    tracker.is_running = True                           # ① 立即标记运行态
    tracker.mark_stage_active("market")                  # ② 激活第一阶段
    record_incomplete_task(ticker, trade_date, ...)      # ③ 持久化断点

    t = threading.Thread(target=_target, daemon=True)   # ④ daemon 线程
    t.start()
    return t

后台线程内部遍历 graph.stream() 的每个 chunk，每收到一个 chunk 就调用 _detect_completed_stages() 检测哪些阶段产出了报告，然后更新 tracker 的完成列表。UI 端的轮询循环读取同一个 tracker 对象来刷新进度条。

3.2 线程安全控制：Event + Lock

ProgressTracker（progress.py）使用两个并发原语：

  • threading.Lock — 保护所有可变字段（completed_stages、is_running 等）的读写原子性
  • threading.Event（_pause_gate）— 实现暂停语义：pause() 调用 _pause_gate.clear() 关闭闸门，后台线程在每轮循环中检查 _pause_gate.wait() 阻塞等待；resume() 调用 _pause_gate.set() 恢复通行

暂停/恢复控制流示例：

# — UI 线程（Streamlit）调用 —
def pause(self):
    with self._lock:
        self.is_paused = True
        self._pause_gate.clear()   # 关闭闸门，阻塞后台线程

# — 后台线程（runner）调用 —
def wait_if_paused(self):
    self._pause_gate.wait()        # 阻塞直到 resume() 调用 set()

停止操作更复杂——request_stop() 设置 stop_requested=True 并清空所有 UI 可见状态。后台线程检测到该标志后，调用 _discard_stopped_run() 清理断点和缓存。

3.3 12 阶段 Pipeline 映射

PIPELINE_STAGES 定义了 12 个阶段的 id、展示名、图标和对应的 state 字段 report_key。前 7 个归类为 ANALYSTS（并行研报生成），后 5 个归类为 PIPELINE（串行决策流程）。_detect_completed_stages() 通过检查 chunk 中各 report_key 是否有内容来判断阶段是否完成。完整的阶段对照见 参考文档。

4. 历史记录与报告导出

历史机制分两层：

  • 已完成分析（get_history()）：扫描 ~/.tradingagents/logs/ 下所有 full_states_log_*.json 文件，按日期倒序展示
  • 未完成任务（get_incomplete_history()）：从 ~/.tradingagents/incomplete_tasks.json 读取持久化的断点信息，交叉比对已完成任务进行去重，支持从中断处继续分析

PDF 导出（pdf_export.py）使用 fpdf2 库，核心复杂度在中文字体探测——按优先级遍历 Linux（文泉驿/Noto）、macOS（PingFang/STHeiti）、Windows（微软雅黑/黑体）等多平台字体路径，并支持通过环境变量 TRADINGAGENTS_PDF_FONT 自定义。Markdown 导出作为无需字体的安全降级方案。

检索练习

问题 1：app.py 的 5 个 UI 状态分别是什么？触发条件各为什么？

问题 2：ProgressTracker 暂停功能使用了哪两个并发原语？为什么不能用 time.sleep() 代替？

问题 3：历史记录中"已完成任务"和"未完成任务"各自存在哪里？侧边栏如何区分展示它们？

参考答案
1：0-空闲（tracker=None）、1-查看历史（viewing_history 不为空）、2-分析中（tracker.is_running）、3-完成（tracker.is_complete）、4-出错（tracker.error）。
2：threading.Lock（保护共享字段原子性）和 threading.Event（_pause_gate 实现阻塞等待/唤醒）。不能用 time.sleep() 因为它只能延时不能响应外部信号恢复，且 sleep 期间无法检测 stop_requested。
3：已完成存在 ~/.tradingagents/logs/ 的 JSON 文件中，通过文件名正则提取 ticker+日期；未完成存在 ~/.tradingagents/incomplete_tasks.json 中。侧边栏分两个区域展示，已完成区域可点击查看报告，未完成区域可点击继续分析。
