1. Analyst 工厂函数（7 个）

所有 Analyst 遵循统一的工厂模式：create_xxx(llm) -> Callable[[dict], dict]。外部注入 LangChain Runnable 实例，返回一个签名为 (state) -> dict 的 LangGraph 节点函数。7 个 Analyst 在 LangGraph 图中并行执行。

1.1 create_market_analyst

文件：analysts/market_analyst.py

create_market_analyst(llm) -> Callable[[dict], dict]
参数类型说明
llmRunnableLangChain ChatModel 实例（quick model），由 graph/setup.py 注入

返回值：返回 market_analyst_node(state) 闭包。该闭包：

  • 输入：state["trade_date"]、state["company_of_interest"]
  • 绑定工具：get_stock_data、get_indicators
  • 输出：{"messages": [AIMessage], "market_report": str} —— 若 LLM 调用工具则 report 为空字符串
边界条件：当 LLM 决定调用工具而非直接输出时，market_report 返回空字符串 ""。下游 Quality Gate 会为此类空报告分配 F 级。
调用示例：
node = create_market_analyst(quick_llm) result = node({"trade_date": "2026-07-08", "company_of_interest": "600519"}) # result["market_report"]: str — 技术分析报告

2. Researcher 工厂函数（2 个）

Bull/Bear Researcher 位于 Quality Gate 之后，两者交替发言进行多空辩论。通过 InvestDebateState 追踪各自历史和当前论点。辩论轮数由 max_debate_rounds 配置（默认 1 轮）。

2.1 create_bull_researcher

文件：researchers/bull_researcher.py

create_bull_researcher(llm) -> Callable[[dict], dict]
参数类型说明
llmRunnablequick model，由 graph/setup.py 注入

返回值：返回 bull_node(state) 闭包。该闭包：

  • 输入：state 中全部 7 份研报 + investment_debate_state（包含辩论历史 + 空头论点）+ data_quality_summary
  • 输出：{"investment_debate_state": {..., "bull_history": str, "current_response": str, "count": int}}
  • 不绑定工具：Bull Researcher 是纯推理 Agent，不调用数据 API
边界条件：当 data_quality_summary 显示某份报告评级为 C/D/F 时，Bull Researcher 应降低对该报告的依赖度（此行为由 LLM prompt 引导，非硬编码）。
调用示例：
node = create_bull_researcher(quick_llm) result = node(state) # state 包含全部 7 份研报 # result["investment_debate_state"]["current_response"] — 多头本轮论点

3. Risk Debator 工厂函数（3 个）

三方风险辩论在 Trader 生成交易方案之后执行。三方同时接收 Trader 提案，按激进/保守/中立立场进行辩论。辩论轮数由 max_risk_discuss_rounds 配置（默认 1 轮）。

3.1 create_aggressive_debator

文件：risk_mgmt/aggressive_debator.py

create_aggressive_debator(llm) -> Callable[[dict], dict]
参数类型说明
llmRunnablequick model

返回值：返回 aggressive_node(state) 闭包。

  • 输入：全部研报 + trader_investment_plan + risk_debate_state（含保守方和中立方论点）
  • 立场：追求高回报，接受高风险，挑战保守方观点
  • 输出：{"risk_debate_state": {..., "current_aggressive_response": str, "aggressive_history": str, "count": int}}

3.3 create_neutral_debator

文件：risk_mgmt/neutral_debator.py

create_neutral_debator(llm) -> Callable[[dict], dict]

立场：平衡风险收益，提供折中方案，挑战激进方和保守方

输出字段：current_neutral_response、neutral_history

边界条件：当 trader_investment_plan 为空时（Trader 调用失败），三方辩论失去了共同参考基准。此时 Prompt 中会提示"交易方案不可用"，各方基于研报直接辩论。

  • aggressive_node(state) -> dict — 读取风险辩论状态、构建激进论点 prompt、调用 LLM
  • conservative_node(state) -> dict — 构建保守论点 prompt、调用 LLM
  • neutral_node(state) -> dict — 构建中立论点 prompt、调用 LLM

4. Manager 工厂函数（2 个）

两个 Manager 分别位于流程的两个关键决策点：Research Manager 在 Bull/Bear 辩论后裁决并生成投资计划；Portfolio Manager 在风险辩论后输出最终交易决策。两者使用 deep model（非 quick model），是全部 17 个 Agent 中仅有的 2 个使用深度模型的角色。

4.1 create_research_manager

文件：managers/research_manager.py

create_research_manager(llm) -> Callable[[dict], dict]
参数类型说明
llmRunnabledeep model，用于结构化输出

返回值：返回 research_manager_node(state) 闭包。

  • 输入：全部研报 + investment_debate_state（含 Bull/Bear 完整辩论历史）
  • 结构化输出 Schema：ResearchPlan（recommendation + rationale + strategic_actions）
  • 降级机制：若结构化输出失败（如旧版 Ollama），invoke_structured_or_freetext() 自动降级为自由文本
  • 输出：{"investment_debate_state": {..., "judge_decision": str}, "investment_plan": str}
边界条件：当辩论历史为空（Bull/Bear 均未产出有效论点）时，Research Manager 仅基于研报做决策。此时 judge_decision 会标注"辩论数据不足"。
调用示例：
node = create_research_manager(deep_llm) result = node(state) # state 包含 debate_state # result["investment_plan"] — markdown 格式的投资计划 # result["investment_debate_state"]["judge_decision"] — 裁决意见

4.2 create_portfolio_manager

文件：managers/portfolio_manager.py

create_portfolio_manager(llm) -> Callable[[dict], dict]

参数：同 Research Manager，使用 deep model。

返回值：返回 portfolio_manager_node(state) 闭包。

  • 输入：investment_plan + trader_investment_plan + risk_debate_state（三方辩论完整历史）+ past_context（历史交易反思，可选）
  • 结构化输出 Schema：PortfolioDecision（rating + executive_summary + investment_thesis + price_target? + time_horizon?）
  • 输出：{"risk_debate_state": {..., "judge_decision": str}, "final_trade_decision": str}
注意：Portfolio Manager 是唯一读取 past_context 的 Agent。该字段由 TradingMemoryLog.get_past_context() 在运行时注入，包含同标的近期决策和跨标的经验反思。
边界条件：若 past_context 为空字符串（首次运行或无历史记录），PM 仅基于当前辩论数据决策。若三方辩论全部失败（所有 risk response 为空），PM prompt 中会明确标注"风险分析数据缺失"。

私有函数汇总

  • research_manager_node(state) -> dict — 构建 instrument context、读取辩论历史、调用结构化输出、更新 debate_state
  • portfolio_manager_node(state) -> dict — 构建 instrument context、读取风险辩论+投资计划+交易方案+历史上下文、调用结构化输出

5. Trader 工厂函数（1 个）

5.1 create_trader

文件：trader/trader.py

create_trader(llm) -> Callable[[dict], dict]
参数类型说明
llmRunnablequick model

返回值：返回一个通过 functools.partial(trader_node, name="Trader") 绑定了固定 name 参数的闭包。

  • 输入：investment_plan（Research Manager 输出）
  • 结构化输出 Schema：TraderProposal（action + reasoning + entry_price? + stop_loss? + position_sizing?）
  • A 股约束：内置 T+1、涨跌停（10%/20%）、最小手数（100 股）、交易时段（9:30-11:30/13:00-15:00）
  • 输出：{"messages": [AIMessage], "trader_investment_plan": str, "sender": "Trader"}
与 LangGraph Send API 的兼容：因为 Send 要求节点接收 state 作为唯一参数，但 Trader 还需要一个固定的 name 标识，因此使用 functools.partial(trader_node, name="Trader") 绑定 name 参数。这是 17 个 Agent 中唯一使用 partial 的角色。
边界条件：
  • 当 investment_plan 为空或格式异常时，Trader 无法生成有效提案，输出 action=Hold 作为安全回退。
  • 结构化输出失败时降级为自由文本，但下游 Risk Debator 需要解析自由文本中的交易信息。
调用示例：
node = create_trader(quick_llm) result = node(state) # state["investment_plan"] 由 Research Manager 填充 # result["trader_investment_plan"] — markdown 格式的交易方案 # result["sender"] — "Trader"

私有函数汇总

  • trader_node(state, name) -> dict — 构建 instrument context、读取投资计划、可选追加 A 股上下文（policy/hot_money/lockup 报告）、调用结构化输出、返回交易方案

6. Quality Gate（质量门控）

6.1 create_quality_gate

文件：quality_gate.py

create_quality_gate(llm) -> Callable[[dict], dict]
参数类型说明
llmRunnablequick model，仅当硬检查失败数 < 4 时触发 LLM 复审

返回值：返回 quality_gate_node(state) 闭包。位于 7 个 Analyst 完成后、消息清除后、Bull Researcher 之前。

两层检查：

  1. 硬检查（代码层） —— 逐份报告检查：
    • 空报告 → F
    • 长度 < MIN_REPORT_LENGTH（200 字符）→ D
    • 包含失败标记（"无法获取"等 FAILURE_MARKERS 关键词）→ D
    • 数据缺失数 >= 3 → C，否则 → B
    • 缺少汇总表格 → B
  2. LLM 复审 —— 仅当硬检查失败数 < 4 时触发，将 7 份报告（每份截断至 3000 字符）发给 LLM 做综合质量评估

输出：{"data_quality_summary": str} —— markdown 格式的质量评估报告

常量

常量值说明
REPORT_FIELDS{"market": "market_report", ...}7 个分析师类型 → state 字段名映射
ANALYST_NAMES{"market": "技术分析师", ...}分析师类型 → 中文显示名映射
MIN_REPORT_LENGTH200最低报告长度阈值（字符）
FAILURE_MARKERS["无法获取", "I cannot retrieve", ...]数据获取失败的标志性文本

私有函数

  • _hard_check_report(analyst_type: str, report: str) -> tuple[str, str] — 对单份报告执行硬检查，返回 (评级, 详情)
  • _build_review_prompt(reports: dict, trade_date: str, ticker: str) -> str — 构建 LLM 复审 prompt，汇编 7 份报告（每份截断至 3000 字符）
  • quality_gate_node(state) -> dict — 读取 state、收集报告、执行硬检查、可选触发 LLM 复审、输出 data_quality_summary

7. 结构化输出 Schema

文件：schemas.py

7.1 评级枚举

PortfolioRating

class PortfolioRating(str, Enum): BUY = "Buy" OVERWEIGHT = "Overweight" HOLD = "Hold" UNDERWEIGHT = "Underweight" SELL = "Sell"

5 级评级枚举，由 Research Manager 和 Portfolio Manager 使用。

TraderAction

class TraderAction(str, Enum): BUY = "Buy" HOLD = "Hold" SELL = "Sell"

3 级交易方向枚举，由 Trader 使用。

7.2 Pydantic 模型

ResearchPlan

class ResearchPlan(BaseModel): recommendation: PortfolioRating # Buy/Overweight/Hold/Underweight/Sell rationale: str # 辩论要点对话式总结 strategic_actions: str # 给 Trader 的具体操作步骤

使用方：Research Manager

TraderProposal

class TraderProposal(BaseModel): action: TraderAction # Buy/Hold/Sell reasoning: str # 操作理由（2-4 句） entry_price: Optional[float] = None # 入场价格目标 stop_loss: Optional[float] = None # 止损价格 position_sizing: Optional[str] = None # 仓位建议，如 "5% of portfolio"

使用方：Trader

注意：entry_price、stop_loss、position_sizing 均为 Optional。LLM 可能不提供精确数值，此时它们为 None。下游消费者需处理缺值情况。

PortfolioDecision

class PortfolioDecision(BaseModel): rating: PortfolioRating # 最终持仓评级 executive_summary: str # 简明行动计划（2-4 句） investment_thesis: str # 基于辩论的详细推理 price_target: Optional[float] = None # 目标价格 time_horizon: Optional[str] = None # 时间范围，如 "3-6 months"

使用方：Portfolio Manager

7.3 Render 函数

每个 Pydantic 模型配有对应的 render 函数，将结构化对象转回 markdown 字符串：

render_research_plan(plan: ResearchPlan) -> str render_trader_proposal(proposal: TraderProposal) -> str render_pm_decision(decision: PortfolioDecision) -> str

各返回包含对应章节标题（Recommendation/Rationale/Strategic Actions 等）的 markdown 文本。

边界条件：当 LLM 供应商不支持 structured output（如旧版 Ollama），invoke_structured_or_freetext() 直接调用 plain_llm.invoke() 获取自由文本，此时 render 函数不会被调用——输出即为 LLM 原始文本。

私有函数汇总

schemas.py 无私有函数，全部为公共导出。

9. Agent 工具函数

文件：utils/agent_utils.py

9.1 get_language_instruction

get_language_instruction() -> str

返回当前配置语言的 prompt 指令。默认英文时返回空字符串 ""（零 token 开销）。仅应用于用户面 Agent（Analyst、Portfolio Manager），内部辩论 Agent 保持英文以保证推理质量。

边界条件：返回空字符串时，调用方直接拼接到 prompt 中不会产生多余的空行或格式问题。

9.2 build_instrument_context

build_instrument_context(ticker: str) -> str
参数类型说明
tickerstr股票代码，如 "600519" 或 "600519.SH"

返回值：一段 prompt 指令字符串，要求 Agent 在所有工具调用、报告和推荐中精确使用该 ticker 并保留交易所后缀。

9.3 create_msg_delete

create_msg_delete() -> Callable[[dict], dict]

返回值：返回 delete_messages(state) 闭包。清除 state 中全部消息，插入 HumanMessage(content="Continue") 占位符（用于满足 Anthropic API 要求）。在管道阶段之间重置对话上下文。

注意：占位符消息是 Anthropic API 兼容性所需——空的 messages 列表会导致 API 错误。其他 LLM 供应商可能不需要此机制，但占位符无害。

私有函数汇总

11. 结构化输出辅助函数

文件：utils/structured.py

11.1 bind_structured

bind_structured( llm: Any, # LangChain ChatModel 实例 schema: type[T], # Pydantic BaseModel 子类 agent_name: str, # Agent 名称（用于日志） ) -> Optional[Any] # 包装后的 LLM 实例，或 None

调用 llm.with_structured_output(schema) 绑定结构化输出。若提供者不支持（如旧版 Ollama），捕获 NotImplementedError / AttributeError，记录 warning 日志，返回 None。

边界条件：返回 None 时，调用方应使用 invoke_structured_or_freetext 的降级机制——直接调用 plain_llm.invoke() 获取自由文本。

11.2 invoke_structured_or_freetext

invoke_structured_or_freetext( structured_llm: Optional[Any], # bind_structured 的返回值 plain_llm: Any, # 原始 LLM 实例 prompt: Any, # prompt（字符串或消息列表） render: Callable[[T], str], # Pydantic → markdown 渲染回调 agent_name: str, # Agent 名称（用于日志） ) -> str # markdown 格式的最终输出

执行逻辑：

  1. 若 structured_llm is not None，调用 structured_llm.invoke(prompt)，通过 render 回调将 Pydantic 实例转为 markdown
  2. 若以上任一步骤抛出异常，记录 warning 并降级为 plain_llm.invoke(prompt)，返回 response.content

使用方：Research Manager、Trader、Portfolio Manager

调用示例：
wrapped = bind_structured(deep_llm, ResearchPlan, "ResearchManager") result = invoke_structured_or_freetext( wrapped, deep_llm, prompt, render_research_plan, "ResearchManager" ) # result — 总是 str（markdown），无论是否走降级路径

12. 评级工具

文件：utils/rating.py

12.1 RATINGS_5_TIER

RATINGS_5_TIER: Tuple[str, ...] = ("Buy", "Overweight", "Hold", "Underweight", "Sell")

5 级评级的权威词汇表（从最看多到最看空）。被 Research Manager、Portfolio Manager、信号处理器和记忆日志统一引用，防止评级术语偏移。

12.2 parse_rating

parse_rating( text: str, # 待解析的文本 default: str = "Hold", # 未匹配时的默认评级 ) -> str # Title-cased 评级字符串

两遍扫描策略：

  1. 正则查找 Rating: X 标签（容忍 markdown 粗体，regex: rating.*?[:-][\s*]*(\w+)）
  2. 回退：扫描全文单词，找到第一个 5-tier 评级词（剥离 markdown 标点后匹配）
  3. 若均无匹配，返回 default（默认 "Hold"）
边界条件：若 LLM 输出的评级词为小写、全大写或混合大小写，该函数通过 .lower() 比对后返回 Title-cased 标准形式。若文本中出现多处评级词（如"Buy vs Hold 的争论"），返回第一个匹配（"Buy"）。
调用示例：
parse_rating("**Rating:** Buy - strong bullish signal") # → "Buy" parse_rating("综合考虑建议 overweight") # → "Overweight" parse_rating("这是一个复杂的市场环境") # → "Hold" (default)

私有变量

  • _RATING_SET — 小写 5-tier 评级集合，用于 O(1) 单词匹配
  • _RATING_LABEL_RE — 编译后的 regex，匹配 "Rating: X" 模式

13.1 TradingMemoryLog

class TradingMemoryLog: """追加式 markdown 交易决策日志。 记录格式：tag行 + DECISION + REFLECTION 支持 pending→resolved 状态生命周期和可配置条目数轮转。"""

构造器

__init__(self, config: dict = None) -> None

从 config dict 读取 memory_log_path（日志文件路径，Path 对象）和 memory_log_max_entries（轮转上限，int）。自动创建父目录。_max_entries=None 时禁用轮转。

公共方法

方法签名说明
store_decision (ticker: str, trade_date: str, final_trade_decision: str) -> None 追加一条 pending 条目到日志末尾。幂等保护：快速原始文本扫描防止同一 (trade_date, ticker, pending) 的重复写入。使用 parse_rating() 解析评级。标签格式：[trade_date | ticker | rating | pending]
load_entries () -> List[dict] 解析日志文件全部条目。每条 dict 含：date, ticker, rating, pending (bool), raw, alpha, holding, decision, reflection。文件不存在返回 []
get_pending_entries () -> List[dict] 仅返回 pending=True 的条目（等待 Phase B 结果更新）
get_past_context (ticker: str, n_same: int = 5, n_cross: int = 3) -> str 返回格式化的历史上下文（用于 Agent prompt 注入）。收集至多 n_same 条同标的已解决条目（完整格式）和至多 n_cross 条跨标的已解决条目（仅反思）。无条目时返回 ""
update_with_outcome (ticker: str, trade_date: str, raw_return: float, alpha_return: float, holding_days: int, reflection: str) -> None 将首个匹配 (trade_date, ticker) 的 pending 条目标记为已解决。通过临时文件 + os.replace() 原子写入防止崩溃数据损坏
batch_update_with_outcomes (updates: List[dict]) -> None 批量应用多个结果更新。每个 dict 需含：ticker, trade_date, raw_return, alpha_return, holding_days, reflection。构建 O(1) 索引 (trade_date, ticker) 一次完成全部替换
边界条件：
  • 并发写入：未使用文件锁，多进程可能竞争。原子写入保证单进程内不会写坏文件，但不保证并发隔离
  • 轮转策略：仅丢弃最旧的已解决条目，pending 条目始终保留。当 _max_entries=None 或实际条目数未超限时跳过轮转
  • 条目分隔符 \n\n<!-- ENTRY_END -->\n\n 不应出现在 LLM 生成文本中，但未做转义检查
  • idempotency guard（store_decision）：仅做原始文本扫描（非解析级），极端情况下可能漏判

私有方法

  • _apply_rotation(self, blocks: List[str]) -> List[str] — 丢弃最旧的已解决条目（超过 _max_entries 时）
  • _parse_entry(self, raw: str) -> Optional[dict] — 解析单条原始文本为 dict，格式异常返回 None
  • _format_full(self, e: dict) -> str — 格式化完整条目（标签 + DECISION + REFLECTION）
  • _format_reflection_only(self, e: dict) -> str — 格式化精简条目（标签 + REFLECTION，decision 截断至 300 字符）
