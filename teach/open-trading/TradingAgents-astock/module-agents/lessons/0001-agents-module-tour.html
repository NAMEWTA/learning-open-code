0001 — Agent 系统模块总览

学习目标：能够画出 agents 模块的 5 层内部架构图，说出每层的 Agent 角色名称及其工厂函数签名，并理解 Agent 之间通过 AgentState 字典传递信息的协作方式。

1. 模块职责与架构位置

tradingagents/agents/ 是整个框架的核心决策引擎。在 L0 项目全景图 中，它位于数据层之上、工作流引擎之下的中间层——负责接收数据、生成研报、组织辩论、输出决策。

模块职责一句话：将 17 个 AI Agent 角色组织成 5 层流水线，通过共享状态字典协作，完成从 7 维度研报生成到 Buy/Hold/Sell 决策的全流程。

关键依赖 上游：dataflows（数据获取）、llm_clients（LLM 调用）。下游：graph（LangGraph 编排）。自身不直接发起 HTTP 请求或调用 LLM SDK——工具的绑定和 LLM 实例的注入均由外部完成。

2. 五层内部架构

agents 模块按职责分为 5 个子包 + 1 个工具层：

tradingagents/agents/ ├── analysts/ ← 第 1 层：研报生成（7 个 Analyst） │ ├── market_analyst.py 市场分析师（技术面） │ ├── social_media_analyst.py 舆情分析师（社交媒体） │ ├── news_analyst.py 新闻分析师（宏观/行业） │ ├── fundamentals_analyst.py 基本面分析师（财报/估值） │ ├── policy_analyst.py 政策分析师（A股特化） │ ├── hot_money_tracker.py 游资追踪师（A股特化） │ └── lockup_watcher.py 解禁监控师（A股特化） ├── researchers/ ← 第 2 层：多空辩论（2 个 Researcher） │ ├── bull_researcher.py 多头研究员 │ └── bear_researcher.py 空头研究员 ├── managers/ ← 第 3 层：决策综合（2 个 Manager） │ ├── research_manager.py 研究经理（辩论裁决 + 投资计划） │ └── portfolio_manager.py 投资组合经理（最终交易决策） ├── risk_mgmt/ ← 第 4 层：风险辩论（3 个 Debator） │ ├── aggressive_debator.py 激进风控 │ ├── conservative_debator.py 保守风控 │ └── neutral_debator.py 中立风控 ├── trader/ ← 第 5 层：交易执行（1 个 Trader） │ └── trader.py 交易员（A 股约束 + 价位/仓位） └── utils/ ← 共享设施 ├── agent_states.py AgentState / 子状态定义 └── agent_utils.py 工具函数导入 + 语言指令

执行顺序：第 1 层 7 个 Analyst 并行生成研报 -> 第 2 层 Bull/Bear 辩论 -> 第 3 层 Research Manager 裁决 -> 第 5 层 Trader 生成交易方案 -> 第 4 层三方风险辩论 -> 第 3 层 Portfolio Manager 最终决策。

3. 统一工厂模式

所有 17 个 Agent 节点遵循统一的工厂函数签名：create_xxx(llm) -> node_function。即外部注入 LLM 实例，返回一个签名为 (state) -> dict 的 LangGraph 节点函数。

以市场分析师为例（源码 market_analyst.py）：

def create_market_analyst(llm):
    def market_analyst_node(state):
        current_date = state["trade_date"]
        tools = [get_stock_data, get_indicators]
        # ... 构建 prompt、绑定工具、调用 LLM ...
        return {
            "messages": [result],
            "market_report": report,
        }
    return market_analyst_node

这个模式的核心优势：

  • 依赖注入：LLM 实例由 graph 层创建并传入，Agent 不关心供应商细节
  • 状态隔离：每个 Agent 只读写自己负责的 state 字段，互不干扰
  • 可测试性：替换 llm 参数即可用 mock 进行单元测试

3 个决策型 Agent（Research Manager、Trader、Portfolio Manager）在此基础上叠加了结构化输出——通过 schemas.py（详见 参考文档）定义的 Pydantic BaseModel 约束 LLM 输出格式。

4. 状态传递与调用示例

Agent 之间的全部信息传递通过一个共享的 AgentState 字典完成（定义在 agent_states.py）。简化后的核心字段如下：

class AgentState(MessagesState):
    company_of_interest: str    # 股票代码
    trade_date: str             # 交易日期
    market_report: str          # 市场分析师报告
    sentiment_report: str       # 舆情报告
    # ... 其余 5 个报告字段 ...
    investment_debate_state: InvestDebateState  # 多空辩论状态
    investment_plan: str        # Research Manager 的投资计划
    trader_investment_plan: str # Trader 的交易方案
    risk_debate_state: RiskDebateState          # 风险辩论状态
    final_trade_decision: str   # Portfolio Manager 最终决策

如果你在 Python 中调用（使用 L0 中介绍的 TradingAgentsGraph）：

from tradingagents.graph.trading_graph import TradingAgentsGraph

graph = TradingAgentsGraph()
result = graph.run(
    ticker="600519",       # 贵州茅台
    date="2026-07-08",
)
# result 包含完整的 AgentState，可提取：
print(result["final_trade_decision"])  # Buy/Hold/Sell
print(result["market_report"])         # 技术分析报告

要点：你不需要直接实例化任何 Agent 工厂函数——TradingAgentsGraph 在内部完成所有 create_xxx 调用、LLM 注入和节点注册。理解工厂模式的意义在于：当你需要自定义 Agent 时，只需实现一个同名签名的 create 函数并注册到图中。

问题 1：agents 模块的 5 层架构分别是哪 5 层？每层包含哪些 Agent 角色？试试不翻看上面内容，凭记忆列出。

问题 2：为什么所有 Agent 都使用 create_xxx(llm) -> node_function 的工厂模式，而不是直接在模块中实例化 LLM？这带来了哪 3 个好处？

问题 3：7 个 Analyst 中有 3 个是 A 股特化的，它们分别是什么？和原版 4 个 Analyst 相比，它们分别填补了 A 股市场的哪些信息盲区？

参考答案
1：analysts（7 个：市场/舆情/新闻/基本面/政策/游资/解禁）、researchers（2 个：Bull/Bear）、risk_mgmt（3 个：激进/保守/中立）、managers（2 个：Research Manager/Portfolio Manager）、trader（1 个）。
2：依赖注入（不关心 LLM 供应商）、状态隔离（只读写自己的 state 字段）、可测试性（替换 llm 参数即可 mock）。
3：政策分析师（产业政策/监管风向）、游资追踪师（龙虎榜/北向资金/概念板块）、解禁监控师（限售解禁/股东减持）。填补了 A 股政策市、游资驱动、解禁冲击三大特有信息盲区。
