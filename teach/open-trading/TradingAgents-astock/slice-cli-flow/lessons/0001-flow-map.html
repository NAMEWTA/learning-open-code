2. 跨文件调用关系

CLI 模块的文件职责按调用层次分为三层。理解这个分层有助于快速定位代码：

层次文件职责被谁调用
调度层 cli/main.py Typer 入口 + 8 步编排 + 流式循环 + 结果保存 用户终端 / 根目录 main.py
交互层 cli/utils.py questionary 交互式提示函数（ticker/日期/分析师/LLM 等） cli/main.py:get_user_selections()
辅助层 cli/config.py
cli/models.py
cli/announcements.py
cli/stats_handler.py		 配置常量、类型枚举、公告拉取、LLM/Tool 统计回调 cli/main.py:run_analysis()

唯一跨模块边界是阶段 5：cli/main.py 调用 tradingagents.graph.TradingAgentsGraph 的 propagate 相关方法。这是 CLI 层与 Graph 层的唯一接触点——所有业务逻辑封装在 Graph 内部，CLI 只负责参数传入和结果渲染。

3. 两条入口路径

项目提供两种启动方式，最终都汇聚到同一个 Graph 实例：

路径 A：脚本模式（根目录 main.py） main.py 直接创建 TradingAgentsGraph(debug=True, config=config)，调用 ta.propagate("NVDA", "2024-05-10")。跳过所有交互步骤，适合 IDE 调试和脚本化批量分析。
路径 B：CLI 模式（cli/main.py） tradingagents analyze 命令通过 Typer 解析，进入 run_analysis()，先收集用户输入再初始化 Graph。包含完整的 8 步交互引导、流式展示和报告保存。

本节后续课程聚焦路径 B（CLI 模式），这是生产环境的主入口。

4. 消息状态机

MessageBuffer 是流式执行阶段的中央状态管理器，维护四类数据：agent_status（Agent 完成进度）、report_sections（报告段落累积）、messages（消息日志 queue）、tool_calls（工具调用记录）。

Agent 状态转换由 update_analyst_statuses() 驱动：检测 chunk 中 report 字段 → 标记对应 Analyst 为 completed → 激活下一个为 in_progress → 全部完成则激活 Bull Researcher。整个转换链为：Analysts → Research → Trader → Risk → Portfolio，无需外部状态机——仅靠 chunk 内容和当前 status 即可推进。
