0002 — 输入→分析→保存主成功路径

学习目标：能逐步说出从用户输入 ticker 到 complete_report.md 写入磁盘的完整数据流转，理解 selections dict → config dict → graph state → final_state 的逐层变换。

1. 阶段一：参数收集 —— 从终端到 Dict

get_user_selections() 是 CLI 的第一个屏障。它编排 8 步交互式输入，全部完成后返回一个扁平 dict。以下为去除了 Rich Panel 装饰后的核心调用链：

get_user_selections() { get_ticker() → "AAPL" # safe_ticker_component 校验 get_analysis_date() → "2025-07-01" # 正则+非未来日期校验 ask_output_language() → "Chinese" # 12 种语言 + 自定义 select_analysts() → [MARKET, SOCIAL] # checkbox 多选 select_research_depth() → 3 # Shallow(1)/Medium(3)/Deep(5) select_llm_provider() → ("openai", url) # 10 种 provider select_shallow_thinker() → "gpt-5.4-mini" # 按 provider 拉取模型列表 select_deep_thinker() → "gpt-5.4" # 同上 # 步骤 8 条件执行（仅 Google/OpenAI/Anthropic） ask_*_config() → thinking_level/effort }

步骤 7 的模型列表依赖于步骤 6 的 provider 选择——select_shallow_thinking_agent(provider) 从 tradingagents.llm_clients.model_catalog 按 provider key 拉取可选模型。这是整个 8 步中唯一的步骤间依赖。

返回的 selections dict 结构如下（所有值均为终端用户输入的直接产物）：

{
    "ticker": "AAPL",
    "analysis_date": "2025-07-01",
    "analysts": [AnalystType.MARKET, AnalystType.SOCIAL],
    "research_depth": 3,
    "llm_provider": "openai",
    "backend_url": "https://api.openai.com/v1",
    "shallow_thinker": "gpt-5.4-mini",
    "deep_thinker": "gpt-5.4",
    "google_thinking_level": None,
    "openai_reasoning_effort": "medium",
    "anthropic_effort": None,
    "output_language": "Chinese",
}

2. 阶段二：图初始化与流式执行

run_analysis() 拿到 selections 后进入核心阶段，分三步：

第一步：从 selections 到 config。 以 DEFAULT_CONFIG 为基座，将用户选择的 research_depth 映射为 max_debate_rounds 和 max_risk_discuss_rounds，将 shallow/deep thinker 映射为 quick_think_llm/deep_think_llm，并合并 provider 专用配置。至此，用户态参数完成到系统态配置的转换。

config = DEFAULT_CONFIG.copy()
config["max_debate_rounds"] = selections["research_depth"]   # 3 → 3 轮辩论
config["quick_think_llm"] = selections["shallow_thinker"]     # 快速模型
config["deep_think_llm"] = selections["deep_thinker"]         # 深度模型
config["backend_url"] = selections["backend_url"]
config["llm_provider"] = selections["llm_provider"]
config["openai_reasoning_effort"] = selections.get("openai_reasoning_effort")
config["output_language"] = selections.get("output_language")

第二步：初始化 Graph 和 UI。 创建 StatsCallbackHandler 用于统计追踪，创建 TradingAgentsGraph 并将 callback 绑定到所有 LLM 节点。同时调用 message_buffer.init_for_analysis() 初始化 Agent 状态表——只有用户选中的 Analyst 才出现在状态表中，其余固定团队（Research/Trading/Risk/Portfolio）始终存在。

第三步：流式循环。 核心是一个 for chunk in graph.graph.stream(...) 循环，包裹在 Rich Live 上下文中（4 fps 刷新）。每个 chunk 的处理流程固定：

for chunk in graph.graph.stream(init_agent_state, **args):
    # 1. 消息分类与去重（按 message.id）
    for message in chunk.get("messages", []):
        msg_type, content = classify_message_type(message)
        if content:
            message_buffer.add_message(msg_type, content)
        # 记录 tool_calls
    # 2. Agent 状态推进（按 report 字段检测）
    update_analyst_statuses(message_buffer, chunk)
    # 3. 处理 Debate State → 更新研究/交易/风控状态
    if chunk.get("investment_debate_state"): ...
    if chunk.get("trader_investment_plan"): ...
    if chunk.get("risk_debate_state"): ...
    # 4. 刷新 Rich 终端布局
    update_display(layout, stats_handler=stats_handler, start_time=start_time)

流式循环退出后，final_state = trace[-1] 持有所有中间报告和最终决策的完整快照。

3. 阶段三：报告保存与展示

流式执行完成后，Rich Live 上下文关闭，进入后处理阶段。两个独立提示（typer.prompt）分别询问是否保存和是否展示：

保存路径（save_report_to_disk）： 用户确认后，final_state 被写入 5 级子目录结构。每个子目录使用 .mkdir(exist_ok=True) 渐进创建，每个报告独立写入 .md 文件。最后汇总为 complete_report.md——包含免责声明 header 和所有报告段落的拼接。保存前会进行路径安全检查——若目标路径在项目目录之外，需用户二次确认。

终端展示（display_complete_report）： 使用 Rich Panel + Markdown 组件，按五大团队顺序渲染：Analyst Reports（蓝框）→ Research（品红框）→ Trading（黄框）→ Risk（红框）→ Portfolio（绿框）。每个 Panel 独立渲染，避免 Rich 因单次输出过大而截断。

4. 数据流转总结

整个主路径中，数据经历了四次关键变换：

用户输入 selections dict config dict │ │ │ │ get_user_selections() │ run_analysis() │ TradingAgentsGraph() └─────────────────────────► └──────────────────────► └──────────────────► │ ▼ complete_report.md ◄─── save_report_to_disk() ◄─── final_state 终端 Rich Panel ◄─── display_complete_report() ▲ ▲ └── 阶段三：持久化与展示 ──┘ └── 阶段二：流式执行 ──┘

注意两个关键边界：selections dict → config dict 是用户语义到系统语义的翻译（"medium depth"变成 max_debate_rounds=3）；config dict → TradingAgentsGraph 是 CLI 层到 Graph 层的控制权交接——此后 CLI 仅被动接收 chunk 并渲染，不再主动干预分析逻辑。
