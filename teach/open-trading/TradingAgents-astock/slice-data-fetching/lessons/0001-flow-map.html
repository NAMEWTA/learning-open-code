0001 — 7 数据源路由地图

学习目标：能说出 VENDOR_METHODS 的 17 个方法如何按 5 大类别分组、3 种 vendor（a_stock / yfinance / alpha_vantage）的覆盖差异，以及 route_to_vendor 的 fallback 链构建规则。

1. 统一入口：从 get_data_inflo 到 route_to_vendor

上层调用方（Agent 的 ToolNode 或 Graph 节点）不直接调用 a_stock.py 中的函数。它们通过 interface.py 的统一入口发起请求。调用链路为：

调用方 → get_data_inflo(method, *args)
       → get_category_for_method(method)     # 查 TOOLS_CATEGORIES
       → get_vendor(category, method)        # 查 config 获取首选 vendor
       → route_to_vendor(method, *args)      # 构建 fallback 链并执行
         → VENDOR_METHODS[method][vendor](*args)

get_data_inflo 本身是一个薄封装（仅做参数透传），真正的路由逻辑在 route_to_vendor 中。这个方法做了三件事：确定方法所属类别、读取配置中的 vendor 偏好、构建 fallback 链。

3. Fallback 降级链机制

route_to_vendor 的降级策略设计得很克制：

# interface.py:202-230 — fallback 链构建与执行
def route_to_vendor(method: str, *args, **kwargs):
    category = get_category_for_method(method)
    vendor_config = get_vendor(category, method)
    primary_vendors = [v.strip() for v in vendor_config.split(',')]

    # 首选在前，其余可用 vendor 追加在后（去重）
    all_available_vendors = list(VENDOR_METHODS[method].keys())
    fallback_vendors = primary_vendors.copy()
    for vendor in all_available_vendors:
        if vendor not in fallback_vendors:
            fallback_vendors.append(vendor)

    for vendor in fallback_vendors:
        try:
            return VENDOR_METHODS[method][vendor](*args, **kwargs)
        except AlphaVantageRateLimitError:
            continue  # 仅在 AV 限流时触发降级
    raise RuntimeError(f"No available vendor for '{method}'")

三个要点：

  1. 仅 AlphaVantageRateLimitError 触发降级 — 网络超时、数据为空、其他异常直接抛出，不静默切换。这是刻意设计：避免在用户不知情时切到不同数据源导致数据口径不一致。
  2. config 支持多 vendor 逗号分隔 — 如 "a_stock,alpha_vantage"，a_stock 失败不会自动切到 AV（因为不是 AV 限流异常），但 config 层面的主备链会在首次路由时按顺序尝试。
  3. signal_data 无降级 — VENDOR_METHODS 中只有一个 a_stock 条目，无法构建 fallback 链，失败直接抛 RuntimeError。
