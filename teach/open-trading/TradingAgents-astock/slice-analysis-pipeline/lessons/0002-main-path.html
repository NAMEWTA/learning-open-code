0002 — 主成功路径：一条 ticker 的完整旅程

目标任务：以 propagate("600519", "2026-07-08") 为起点，追踪全链路中关键方法调用如何修改 AgentState，能解释每个节点返回的 dict 更新了哪些字段。

前一节画出了 7 阶段全景图，本节沿主成功路径（所有分析师正常产报告、质量门控通过、辩论正常结束）逐个节点追踪。

1. 入口三段式：prepare → invoke → finalize

入口 _run_graph()（trading_graph.py:398）做三件事：

# 1. 准备
init_state, args, _ = self.prepare_graph_run(company_name, trade_date)

# 2. 执行
final_state = self.graph.invoke(init_state, **args)

# 3. 落盘
signal = self.finalize_graph_run(company_name, trade_date, final_state)
return final_state, signal

prepare_graph_run() 内部调用 self.propagator.create_initial_state() 构建初始 dict（见 propagation.py:18）。关键字段：

  • messages — LangGraph 内部消息列表，以 ("human", company_name) 开头
  • company_of_interest / trade_date — 股票代码和交易日
  • 报告域 market_report sentiment_report news_report fundamentals_report policy_report hot_money_report lockup_report — 全部初始化为空字符串 ""
  • 辩论域 investment_debate_state（3 字段全空，count=0）+ risk_debate_state（9 字段全空，count=0）
为什么初始化为空字符串？LangGraph StateGraph 使用 additive reducer，messages 会累积。而报告和辩论字段使用 overwrite reducer——每个 Analyst 节点返回的 dict 直接覆盖对应字段。初始化为 "" 保证下游不会因 KeyError 崩溃。

2. 阶段 2：7 Analyst 的 Tool-Call 循环

第一个 Analyst（Market）从 START 边启动。每个 Analyst 的循环逻辑相同（以 Market 为例，setup.py:150-155）：

workflow.add_conditional_edges(
    "Market Analyst",
    self.conditional_logic.should_continue_market,
    ["tools_market", "Msg Clear Market"],
)
workflow.add_edge("tools_market", "Market Analyst")

循环机制：Analyst 节点执行时，LLM 判断需要获取数据 → 返回 tool_call → should_continue_market()（conditional_logic.py:14-20）检测到 last_message.tool_calls 非空 → 路由到 tools_market → ToolNode 执行工具 → 将结果消息追加到 messages → 自动回到 "Market Analyst" → LLM 再决定是继续调工具还是产研报 → 不再有 tool_call → 路由到 "Msg Clear Market" → 删除多余消息 → 进入下一个 Analyst。

Market → Social → News → Fundamentals → Policy → HotMoney → Lockup 依次传递，最后一个 Analyst 的 Msg Clear 直接连到 "Quality Gate"。

3. 阶段 3-7：决策链逐步收敛

质量门控（阶段 3）：quality_gate_node（quality_gate.py:126）读取 7 个报告字段，逐份跑 _hard_check_report()：长度 < 200 字符 → D 级，含失败标记（如"无法获取"）→ D 级，3+ 处 [数据缺失] → C 级。若 D/F 级 < 4 份，追加一次 LLM 复审。最终返回 {"data_quality_summary": summary}。

Bull/Bear 辩论（阶段 4）：setup.py 固定从 "Quality Gate" → "Bull Researcher"。Bull 产多论点 → 条件路由检测 current_response.startswith("Bull") → 进 "Bear Researcher" → Bear 反驳 → 再次条件路由 → 若 count < 2*max_debate_rounds 且当前是 Bear 则回 Bull → 轮次达标后进入 "Research Manager"。

决策收敛（阶段 5→6→7）：

  • Research Manager（deep_thinking_llm）阅读辩论历史 → 输出 ResearchPlan（结构化 Schema，schemas.py:61）→ render 为 markdown 写入 state
  • Trader（quick_thinking_llm）读取计划 → 输出 TraderProposal（schemas.py:109）含 entry_price / stop_loss
  • 风险三方辩论：Aggressive→Conservative→Neutral 循环辩论（最大 3*max_risk_discuss_rounds 轮），到期入 Portfolio Manager
  • Portfolio Manager（deep_thinking_llm）→ PortfolioDecision（schemas.py:171）→ render 为 **Rating**: X 格式的 markdown

4. 终点：从 PM 决策到 5 档评级

LangGraph 执行到 Portfolio Manager → END 后，graph.invoke() 返回最终 state。回到 _run_graph() → finalize_graph_run()（trading_graph.py:369）：

  1. _log_state() 将 7 个报告 + 2 个辩论历史 + 决策写入 JSON 日志文件
  2. memory_log.store_decision() 将决策存入待处理日志（下次同 ticker 运行时做延迟反思）
  3. clear_checkpoint() 清除 SQLite 断点
  4. process_signal() → SignalProcessor.process_signal() → parse_rating()（rating.py:30）从 **Rating**: X 行提取最终评级

最终返回 ("Buy" / "Overweight" / "Hold" / "Underweight" / "Sell")——这就是整个链路从 ticker 到决策的输出。
