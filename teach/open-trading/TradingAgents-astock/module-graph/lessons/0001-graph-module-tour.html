0001 — LangGraph 工作流引擎模块总览

目标任务：能够画出 TradingAgents-Astock 在 LangGraph 中的节点-边拓扑图，说出传播、条件路由、断点续跑与反思的代码实现方式。

1. 模块定位与文件职责

TradingAgents-Astock 的 tradingagents/graph/ 包是整个框架的编排中枢。它不负责分析逻辑本身（那是 agents/ 的职责），也不负责获取数据（那是 dataflows/ 的职责），而是用 LangGraph StateGraph 将所有组件组装为一条可追踪、可恢复的决策流水线。

tradingagents/graph/ ├── __init__.py ← 导出 6 个公开类 ├── trading_graph.py ← 主入口类 TradingAgentsGraph（propagate / 双 LLM / 生命周期） ├── setup.py ← GraphSetup：节点创建 + 边拓扑定义 ├── propagation.py ← Propagator：初始状态构建 + 图运行参数 ├── conditional_logic.py ← ConditionalLogic：7 个工具循环路由 + 2 个辩论轮次路由 ├── reflection.py ← Reflector：延迟反射——用真实收益评估历史决策 ├── signal_processing.py ← SignalProcessor：从 PM 决策文本提取 5 档评级 └── checkpointer.py ← SQLite 断点续跑：分 ticker 存储 + thread_id 路由

调用链：propagate(ticker, date) 是唯一入口。内部委托 _run_graph()，先调 prepare_graph_run() 构建初始状态，再用 graph.invoke() 执行状态机，最后 finalize_graph_run() 落盘 + 清 checkpoint + 提取信号。

双 LLM 分配策略 deep_thinking_llm（深度模型）仅用于 Research Manager 和 Portfolio Manager——两个需要综合全局信息做最终研判的节点。quick_thinking_llm（快速模型）用于其余所有 Agent——7 个 Analyst、Bull/Bear Researcher、Trader、三个风险辩论者。一次完整分析约 30-50 次 LLM 调用，仅 2 次走深度模型，兼顾品质与成本。

2. 七阶段 Pipeline 节点拓扑

setup.py 的 GraphSetup.setup_graph() 将 L0 课程中介绍的 7 阶段 pipeline 映射为 LangGraph 节点-边拓扑。核心特征是：Analyst 串行执行——每个分析师完成研报后清空消息历史，再传给下一个分析师，避免上下文膨胀。

以下代码展示了 setup.py 中 Analyst 节点的核心拓扑逻辑——为每个选中的分析师创建节点、工具循环和消息清理器，然后串联成链：

# setup.py — 节点创建与串联（核心片段）
# 为每个选中分析师创建三个节点：分析师本身 + 工具 + 消息清理
for analyst_type, node in analyst_nodes.items():
    workflow.add_node(f"{analyst_type.capitalize()} Analyst", node)
    workflow.add_node(f"tools_{analyst_type}", tool_nodes[analyst_type])
    workflow.add_node(f"Msg Clear {analyst_type.capitalize()}", delete_nodes[analyst_type])

# 分析师串联：当前 analyst →（条件路由）→ tools 或 msg_clear → 下一个 analyst
for i, analyst_type in enumerate(selected_analysts):
    current = f"{analyst_type.capitalize()} Analyst"
    workflow.add_conditional_edges(current,
        getattr(self.conditional_logic, f"should_continue_{analyst_type}"),
        [f"tools_{analyst_type}", f"Msg Clear {analyst_type.capitalize()}"])
    workflow.add_edge(f"tools_{analyst_type}", current)  # 工具→回分析师
    if i < len(selected_analysts) - 1:
        workflow.add_edge(f"Msg Clear {analyst_type.capitalize()}",
                          f"{selected_analysts[i+1].capitalize()} Analyst")
    else:
        workflow.add_edge(f"Msg Clear {analyst_type.capitalize()}", "Quality Gate")

关键设计决策：为什么 Analyst 不做并行？论文作者（arXiv:2412.20138）认为，分析师串联能逐层累积信息——市场研报先跑，为后续社交/新闻/基本面分析提供技术面锚点；Quality Gate 在 7 份报告齐全后再做质量裁决，而非跳着门控。

3. 状态流转：初始化与条件路由

初始化。 Propagator.create_initial_state() 构建 LangGraph 的初始 AgentState 字典，包含股票名称、交易日期、历史上下文（past_context）、以及两个嵌套的辩论状态 TypedDict：

# propagation.py — create_initial_state 返回值结构
{
    "messages": [("human", company_name)],     # LangGraph 消息流起点
    "company_of_interest": company_name,        # 股票名或代码
    "trade_date": str(trade_date),              # 分析基准日期
    "past_context": past_context,               # 历史交易反思文本（可为空）
    "investment_debate_state": InvestDebateState({  # Bull/Bear 辩论状态
        "bull_history": "", "bear_history": "", "history": "",
        "current_response": "", "judge_decision": "", "count": 0,
    }),
    "risk_debate_state": RiskDebateState({      # 三方风险辩论状态
        "aggressive_history": "", "conservative_history": "",
        "neutral_history": "", "history": "",
        "latest_speaker": "", "judge_decision": "", "count": 0,
    }),
    # 7 个 Analyst 报告占位（初始为空字符串）
    "market_report": "", "fundamentals_report": "", # ... 其余 5 个
}

条件路由。 ConditionalLogic 类提供两类路由函数： （1）工具循环路由——为每个 Analyst 提供 should_continue_{analyst_type}()，检查最后一条消息是否有 tool_calls：有则路由到 tools 节点执行工具调用，执行完回到 Analyst 继续推理；没有则路由到 Msg Clear，结束该 Analyst 的工作。 （2）辩论轮次路由——should_continue_debate() 用计数器控制 Bull/Bear 辩论轮数（默认 1 轮 = 2 次发言），should_continue_risk_analysis() 控制三方风险辩论（默认 1 轮 = 3 次发言），达到上限后路由到 Manager。

# conditional_logic.py — 工具循环路由（以 market 为例）
def should_continue_market(self, state: AgentState):
    messages = state["messages"]
    last_message = messages[-1]
    if last_message.tool_calls:
        return "tools_market"       # 有工具调用→执行工具
    return "Msg Clear Market"       # 无工具调用→分析完成，清消息

# 辩论轮次路由
def should_continue_debate(self, state: AgentState) -> str:
    if state["investment_debate_state"]["count"] >= 2 * self.max_debate_rounds:
        return "Research Manager"   # 辩论轮次已满→Manager 综合
    if state["investment_debate_state"]["current_response"].startswith("Bull"):
        return "Bear Researcher"    # Bull 说完→轮到 Bear
    return "Bull Researcher"        # 否则→Bull 发言

4. 断点续跑与交易反思

断点续跑。 checkpointer.py 基于 LangGraph 的 SqliteSaver 实现。用户开启 checkpoint_enabled=True 后，每个 ticker 对应一个独立的 SQLite 数据库文件（{TICKER}.db），存储在 {data_cache_dir}/checkpoints/ 下。每次 graph 执行完一个节点，SqliteSaver 自动在数据库的 checkpoints 和 writes 表中记录当前状态。

关键流程：prepare_graph_run() 先调用 checkpoint_step() 查询是否有同名 ticker+date 的断点。如果有，返回 initial_state=None——LangGraph 收到 None 时会从断点自动恢复，跳过已完成的节点。分析正常结束后，finalize_graph_run() 调用 clear_checkpoint() 删除该线程的行，避免下次运行时读到过期状态。

交易反思。 Reflector 实现的是延迟反思机制。本次分析的最终决策存入 TradingMemoryLog（pending 状态）。下次同一 ticker 运行时，_resolve_pending_entries() 通过 yfinance 获取持有期真实收益，计算相对于沪深300的 alpha，然后调 reflect_on_final_decision() 让 LLM 自评决策方向是否正确、论据是否成立、下一轮应吸取什么教训。反思文本通过 past_context 字段注入到下一轮图执行的初始状态中，形成持续学习闭环。

点击查看参考答案

1. 三个节点：{Analyst} Analyst（LLM 分析节点）、tools_{analyst}（ToolNode）、Msg Clear {Analyst}（消息清理节点）。边：Analyst →（条件路由）tools 或 Msg Clear；tools → Analyst（回环）；Msg Clear → 下一个 Analyst 或 Quality Gate。

2. 工具循环路由（should_continue_* × 7）——检查 LLM 是否请求工具调用，控制分析师内部的 tool-LLM 循环。辩论轮次路由（should_continue_debate / should_continue_risk_analysis）——用计数器控制辩论轮数，达到上限后路由到 Manager。

3. prepare_graph_run() 调 checkpoint_step() 查询 SQLite 中 ticker+date 的 thread_id 是否有断点。有断点→返回 None 初始状态→LangGraph 自动恢复。正常完成→finalize_graph_run() 调 clear_checkpoint() 删行。

4. 刚做出决策时，持有期收益尚未发生——需要等 5 个交易日后跑 yfinance 获取实际价格才能计算真实收益和 alpha。因此当前运行只存 pending 决策，下次同 ticker 运行时批量回填反思。
