1. TradingAgentsGraph — 主编排器

文件：tradingagents/graph/trading_graph.py
职责：组装所有组件（双 LLM、ToolNode 字典、5 个子组件），对外暴露 propagate() 单入口。

1.1 构造函数

TradingAgentsGraph(
  selected_analysts: list[str] = ["market","social","news","fundamentals","policy","hot_money","lockup"],
  debug: bool = False,
  config: dict[str, Any] | None = None,
  callbacks: list | None = None
)
参数类型默认值说明
selected_analysts list[str] 7 个全选 启用的分析师类型。可选值：market、social、news、fundamentals、policy、hot_money、lockup。顺序决定图中节点的执行顺序。
debug bool False 为 True 时逐 chunk 打印每条消息，适合排查 LLM 输出异常。
config dict | None DEFAULT_CONFIG 配置字典。不传则使用 tradingagents.default_config.DEFAULT_CONFIG。关键字段见下表。
callbacks list | None [] LangChain callback handler 列表。传入后将同时传递给 LLM 构造函数和 get_graph_args()，用于追踪 token 消耗和工具调用。

config 关键字段：

字段类型说明
llm_providerstrLLM 厂商："google" / "openai" / "anthropic"。影响 _get_provider_kwargs() 的参数映射。
deep_think_llmstr深度推理模型 ID。用于 Research Manager 和 Portfolio Manager。
quick_think_llmstr快速推理模型 ID。用于 7 Analyst + 其余 Agent。
backend_urlstrLLM API 端点。默认从环境变量读取。
max_debate_roundsintBull/Bear 辩论最大轮数。实际执行 2x 轮（双方各发言）。
max_risk_discuss_roundsint三方风控辩论最大轮数。实际执行 3x 轮。
checkpoint_enabledbool是否启用 SQLite 断点续跑。
data_cache_dirstr数据缓存目录（checkpoint DB 也存于此）。
results_dirstr分析结果 JSON 输出目录。
边界条件：若 selected_analysts 传入空列表，GraphSetup.setup_graph() 将在编译阶段抛出 ValueError("Trading Agents Graph Setup Error: no analysts selected!")。构造函数本身不做校验。

构造函数初始化流程：

  1. set_config() — 将配置写入全局单例
  2. 创建 data_cache_dir 和 results_dir 目录
  3. _get_provider_kwargs() — 按厂商生成 thinking 控制参数
  4. create_llm_client() ×2 — 构造 deep / quick 两个 LLM 客户端
  5. TradingMemoryLog — 初始化记忆日志（复盘数据库）
  6. _create_tool_nodes() — 创建 7 个类别共 17 个工具函数的 ToolNode
  7. 初始化 5 个子组件：ConditionalLogic → GraphSetup → Propagator → Reflector → SignalProcessor
  8. workflow.compile() — 首次编译（默认不带 checkpointer）

1.2 propagate() — 主入口

tuple[dict, str] propagate(
  company_name: str,
  trade_date: str
)
参数类型说明
company_name str 股票代码（6 位）或中文名称。如 "600519" 或 "贵州茅台"。会经由 safe_ticker_component 做路径安全校验。
trade_date str 分析日期，格式 "YYYY-MM-DD"。作为 thread_id 和日志文件名的一部分。

返回值：

类型说明
tuple[dict, str] (final_state, signal) — final_state 为包含全部 report 字段和决策的 AgentState 字典；signal 为 5 档评级字符串（Buy / Overweight / Hold / Underweight / Sell）。
边界条件：
  • checkpoint 已存在：若 checkpoint_enabled=True 且相同 ticker+date 的 checkpoint 尚未清除，将从中断处恢复执行，不会重跑已完成节点。
  • yfinance 无数据：若分析日期距今过近（股票价格未出），_fetch_returns() 返回 (None,None,None)，pending 条目不会被标记为已解决，下次运行再试。
  • ticker 路径注入：_log_state() 写文件前经过 safe_ticker_component() 校验，防止 ../../etc/passwd 类路径遍历攻击。

1.3 propagate() 状态传播全链路

以下是 propagate() 从入参到返回值的完整调用链路：

1 propagate(company_name, trade_date) → _run_graph()
2 prepare_graph_run()
  • 赋值 self.ticker = company_name
  • 调用 _resolve_pending_entries(company_name) — 处理上次运行的 deferred reflection
  • 若 checkpoint_enabled：用 get_checkpointer() 打开 SQLite DB，workflow.compile(checkpointer=saver) 重新编译；检查是否存在可恢复的 checkpoint
  • 调用 propagator.get_graph_args(callbacks) 构建运行参数
  • 若 checkpoint 存在则返回 (None, args, resume_step) 让 LangGraph 恢复；否则调用 propagator.create_initial_state() 构建全新状态
  • 返回 (init_agent_state, args, checkpoint_step)
3 graph.invoke(init_state, **args) 或 graph.stream(init_state, **args)
  • LangGraph 按照 GraphSetup.setup_graph() 定义的拓扑执行 30 个节点
  • 每个 Analyst 节点：LLM 调用 → tool_calls? → tools → 回到 Analyst → 无 tool_calls → Msg Clear → 下一个 Analyst
  • Bull/Bear 辩论：should_continue_debate 控制轮数，达到上限后进入 Research Manager
  • 三方风控辩论：should_continue_risk_analysis 控制轮数，达到上限后进入 Portfolio Manager
  • debug 模式：逐 chunk 输出 pretty_print()
  • 返回 final_state（包含所有 report 字段 + 最终决策）
4 finalize_graph_run(company_name, trade_date, final_state)
  • 赋值 self.curr_state = final_state
  • _log_state(trade_date, final_state) — 提取 8 个字段序列化为 JSON，写入 results_dir/{ticker}/TradingAgentsStrategy_logs/full_states_log_{date}.json
  • memory_log.store_decision() — 将 final_trade_decision 作为 pending 条目存入 SQLite，供下次运行时复盘
  • 若 checkpoint_enabled：调用 clear_checkpoint() 清理本次 checkpoint，避免脏状态
  • 调用 process_signal(final_trade_decision) 提取 5 档评级
  • 返回 (final_state, signal)
5 close_graph_run()（在 _run_graph() 的 finally 块中执行）
  • _checkpointer_ctx.__exit__() — 关闭 SQLite 连接
  • workflow.compile() — 将 graph 恢复为不带 checkpointer 的版本，避免连接泄漏
异常安全：_run_graph() 使用 try/finally 确保无论 graph 执行是否成功，close_graph_run() 都会执行。若 graph 执行中抛出异常，checkpointer 不会被清除（保留中断现场），下次相同 ticker+date 的 propagate() 将从中断处恢复。

1.4 内部方法速查

方法签名说明
_get_provider_kwargs () -> dict 按 llm_provider 返回厂商特定参数。Google → thinking_level；OpenAI → reasoning_effort；Anthropic → effort。其他厂商返回空字典。
_create_tool_nodes () -> dict[str, ToolNode] 创建 7 个 analyst 类型对应的 ToolNode 字典。共绑定 17 个工具函数。
_fetch_returns (ticker, trade_date, holding_days=5) -> tuple[float|None, float|None, int|None] 用 yfinance 拉取 ticker 和沪深300 基准在 holding_days 内的收益。返回 (raw_return, alpha_return, actual_days)。数据不足（< 2 行）或网络异常时全返回 None。
_resolve_pending_entries (ticker: str) -> None 读取同一 ticker 的 pending 日志条目，计算实际收益，生成反思文本，批量写回。只有价格数据就绪的条目才会被更新。
_log_state (trade_date, final_state) -> None 提取 final_state 中的 8 个关键字段，写入 full_states_log_{date}.json。路径经 safe_ticker_component 安全校验。
process_signal (full_signal: str) -> str 公开方法。从 Portfolio Manager 的完整决策文本中提取 5 档评级，委托给 signal_processor.process_signal()。

2.2 setup_graph()

StateGraph setup_graph(
  selected_analysts: list[str] = ["market","social","news","fundamentals","policy","hot_money","lockup"]
)
参数类型默认值说明
selected_analystslist[str]7 个全选启用的分析师列表。顺序决定图中节点执行顺序。可选值：market、social、news、fundamentals、policy、hot_money、lockup。

返回值：未 compile 的 langgraph.graph.StateGraph 实例。调用方需要自行 .compile() 并可注入 checkpointer。

边界条件：若 selected_analysts 为空列表，立即抛出 ValueError("Trading Agents Graph Setup Error: no analysts selected!")。这是唯一的输入校验点。

图拓扑构建流程：

  1. 依 selected_analysts 顺序创建 3 类节点（Analyst + tools_X + Msg Clear X）
  2. 创建 Quality Gate、Bull/Bear、Research Manager、Trader
  3. 创建 3 Risk Debater + Portfolio Manager
  4. 添加节点到 StateGraph(AgentState)
  5. 边定义：
    • START → first_analyst（固定边）
    • 每个 Analyst → 条件路由 should_continue_{type} → tools_X（回环）或 Msg Clear
    • Msg Clear → 下一个 Analyst（固定边）或 Quality Gate（最后一个 Analyst 时）
    • Quality Gate → Bull（固定边）
    • Bull ↔ Bear（条件路由 should_continue_debate），终止条件 → Research Manager
    • Research Manager → Trader（固定边）
    • Trader → Aggressive（固定边）
    • Aggressive → Conservative → Neutral → Aggressive…（条件路由 should_continue_risk_analysis），终止条件 → Portfolio Manager
    • Portfolio Manager → END（固定边）

各 Analyst 创建的 Agent 函数：

analyst_type创建函数所属模块
marketcreate_market_analyst(llm)tradingagents.agents
socialcreate_social_media_analyst(llm)tradingagents.agents
newscreate_news_analyst(llm)tradingagents.agents
fundamentalscreate_fundamentals_analyst(llm)tradingagents.agents
policycreate_policy_analyst(llm)tradingagents.agents

3. Propagator — 状态初始化与传播

文件：tradingagents/graph/propagation.py
职责：构建 AgentState 初始状态字典，生成 LangGraph invoke/stream 的运行参数。

3.1 构造函数

Propagator(max_recur_limit: int = 100)
参数类型默认值说明
max_recur_limitint100LangGraph recursion_limit。图中每经过一个节点计数 +1，超过此值抛出 RecursionError。全选 7 Analyst + 默认辩论轮数时，总步数约 60-80，100 为安全上限。
边界条件：若设置过小（如 max_recur_limit=10），在选中多个 Analyst 的场景下图遍历 10 个节点后即被 LangGraph 终止并抛异常。建议不低于 50。

4. ConditionalLogic — 条件路由

文件：tradingagents/graph/conditional_logic.py
职责：为 LangGraph 的 add_conditional_edges 提供 9 个路由函数，控制图中 token 的流转方向。

4.1 构造函数

ConditionalLogic(
  max_debate_rounds: int = 1,
  max_risk_discuss_rounds: int = 1
)
参数类型默认值说明
max_debate_roundsint1Bull/Bear 辩论最大轮数。实际执行 2x 此值（因为 count >= 2 * max_debate_rounds 才终止，每次双方各发表一次 count+1）。
max_risk_discuss_roundsint1三方风控辩论最大轮数。实际执行 3x 此值（count >= 3 * max_risk_discuss_rounds 终止）。
边界条件：若 max_debate_rounds=0，由于 count 初始为 0 且条件为 >= 0，第一条消息即直接进入 Research Manager，跳过多空辩论。同理 max_risk_discuss_rounds=0 会直接进入 Portfolio Manager。

4.2 Analyst 路由函数（7 个）

这 7 个函数签名和逻辑完全相同，仅目标节点名称不同。以一个为例说明：

str should_continue_{analyst_type}(state: AgentState)
参数类型说明
stateAgentStateLangGraph 当前状态。从中读取 messages[-1]。

逻辑：检查 state["messages"][-1].tool_calls。若存在 tool_calls → 返回 "tools_{type}"，工具执行完回到同一分析师。否则 → 返回 "Msg Clear {Type}"，清消息历史后进入下一节点。

方法名返回节点 1（有 tool_calls）返回节点 2（无 tool_calls）
should_continue_market"tools_market""Msg Clear Market"
should_continue_social"tools_social""Msg Clear Social"
should_continue_news"tools_news""Msg Clear News"
should_continue_fundamentals"tools_fundamentals""Msg Clear Fundamentals"
should_continue_policy"tools_policy""Msg Clear Policy"
should_continue_hot_money"tools_hot_money""Msg Clear Hot_money"
should_continue_lockup"tools_lockup""Msg Clear Lockup"
边界条件：若 messages 列表为空（不可能在执行中发生，因为第一个节点已注入 human message），messages[-1] 会引发 IndexError。若 last_message 没有 tool_calls 属性（非 AIMessage），Python 抛出 AttributeError。

4.3 should_continue_debate()

str should_continue_debate(state: AgentState)

决策逻辑：

  1. 若 investment_debate_state["count"] >= 2 * max_debate_rounds → 返回 "Research Manager"（辩论结束）
  2. 若 current_response 以 "Bull" 开头 → 返回 "Bear Researcher"（轮到 Bear 发言）
  3. 否则 → 返回 "Bull Researcher"
边界条件：若 current_response 既不以 "Bull" 开头也不以 "Bear" 开头（例如 LLM 输出了非预期前缀），默认进入 Bull Researcher。这种 fallback 行为可以防止辩论死锁。

4.4 should_continue_risk_analysis()

str should_continue_risk_analysis(state: AgentState)

决策逻辑：

  1. 若 risk_debate_state["count"] >= 3 * max_risk_discuss_rounds → 返回 "Portfolio Manager"（风控辩论结束）
  2. 若 latest_speaker 以 "Aggressive" 开头 → 返回 "Conservative Analyst"
  3. 若 latest_speaker 以 "Conservative" 开头 → 返回 "Neutral Analyst"
  4. 否则 → 返回 "Aggressive Analyst"（默认开启新一轮轮转）
边界条件：裁决的次序固定（进攻 → 保守 → 中立 → 进攻…），若 latest_speaker 不匹配任何已知前缀，自动路由到 Aggressive，不会阻塞图执行。

5. SignalProcessor — 信号评级提取

文件：tradingagents/graph/signal_processing.py
职责：从 Portfolio Manager 的完整决策文本中提取 5 档评级字符串。

5.1 构造函数

SignalProcessor(quick_thinking_llm: Any = None)
参数类型默认值说明
quick_thinking_llm Any None 已废弃。保留此参数仅为向后兼容，调用方仍可传入 LLM 实例但不会被使用。评级提取现已改为确定性解析。

5.2 process_signal()

str process_signal(full_signal: str)
参数类型说明
full_signalstrPortfolio Manager 的完整 markdown 决策文本。格式由 render_pm_decision() 渲染，通常以 **Rating**: X 开头。

返回值：5 档评级之一："Buy" | "Overweight" | "Hold" | "Underweight" | "Sell"

实现：委托给 tradingagents.agents.utils.rating.parse_rating()，该函数通过确定性启发式从 markdown 文本中匹配 **Rating**: X 模式，不再发起 LLM 调用。

6. Reflector — 反思机制

文件：tradingagents/graph/reflection.py
职责：在下次运行时对上次决策做复盘反思（Phase B deferred reflection），生成 2-4 句简洁英文反思文本，注入后续分析的 past_context。

6.1 构造函数

Reflector(quick_thinking_llm: Any)
参数类型说明
quick_thinking_llmAny快速推理 LLM 实例。用于生成反思文本——反思只需 2-4 句精简总结，不需要深度推理。

构造函数内部通过 _get_log_reflection_prompt() 生成系统 prompt，要求输出覆盖三个部分：方向判断正确性、投资论据实效性、具体改进建议。

6.2 reflect_on_final_decision()

str reflect_on_final_decision(
  final_decision: str,
  raw_return: float,
  alpha_return: float
)
参数类型说明
final_decisionstr上次 Portfolio Manager 输出的完整决策文本。
raw_returnfloat上次分析后 holding_days 内的绝对收益率。格式化为 +.1%。
alpha_returnfloat相对沪深300 的超额收益（alpha）。格式化为 +.1%。

返回值：2-4 句英文反思文本，纯 prose，无 markdown、无列表、无标题。

LLM 调用方式：

messages = [
    ("system", log_reflection_prompt),
    ("human", f"Raw return: {raw_return:+.1%}\nAlpha vs CSI 300: {alpha_return:+.1%}\n\nFinal Decision:\n{final_decision}")
]
return self.quick_thinking_llm.invoke(messages).content
边界条件：
  • raw_return 或 alpha_return 取值为 float('nan') 或 float('inf') 时，格式化字符串 :+.1% 会输出 nan% 或 inf%，LLM 可能生成扭曲的反思。
  • prompt 明确要求 2-4 句英文纯 prose，但 LLM 不保证一定遵守——反思文本仍可能包含 markdown 或超过长度。

7. Checkpointer — SQLite 断点续跑

文件：tradingagents/graph/checkpointer.py
职责：为 LangGraph 提供 per-ticker 的 SQLite checkpoint 存储，支持中断恢复。每个 ticker 一个独立 .db 文件，避免并发争用。

7.1 thread_id()

str thread_id(ticker: str, date: str)
参数类型说明
tickerstr股票代码。内部转为大写。
datestr分析日期。

返回值：{ticker}:{date} 的 SHA256 哈希前 16 个十六进制字符。

实现：hashlib.sha256(f"{ticker.upper()}:{date}".encode()).hexdigest()[:16]

边界条件：同一 ticker+date 组合始终产生相同 thread_id（确定性），不同 ticker+date 碰撞概率极低（16 hex chars = 64 bits）。此确定性设计确保手动重启后能从上次断点恢复。

7.2 get_checkpointer()

Generator[SqliteSaver, None, None] get_checkpointer(
  data_dir: str | Path,
  ticker: str
)
参数类型说明
data_dirstr | Path数据根目录。DB 文件创建于 {data_dir}/checkpoints/{TICKER}.db。
tickerstr股票代码。经 safe_ticker_component 校验后转为大写。

返回值：@contextmanager 装饰的生成器，yield SqliteSaver 实例。调用方必须使用 with get_checkpointer(...) as saver: 或在 __enter__/__exit__ 中手动管理。

内部流程：

  1. sqlite3.connect(db, check_same_thread=False) — 允许多线程访问
  2. SqliteSaver(conn) — 创建 LangGraph checkpoint saver
  3. saver.setup() — 创建 writes 和 checkpoints 表
  4. yield saver
  5. finally 块中 conn.close()
边界条件：若 ticker 包含 / 或 .. 等路径遍历字符，safe_ticker_component() 将拒绝并抛异常，防止写入 ../../etc/some.db。

7.3 has_checkpoint()

bool has_checkpoint(data_dir, ticker, date)

等价于 checkpoint_step(data_dir, ticker, date) is not None。用于存在性快速判断。

7.4 checkpoint_step()

int | None checkpoint_step(data_dir, ticker, date)

查询指定 ticker+date 的最新 checkpoint 步骤号。

内部流程：

  1. 计算 DB 路径，若 .db 文件不存在 → 返回 None
  2. 用 thread_id(ticker, date) 生成 thread_id
  3. 打开 get_checkpointer() 上下文，调用 saver.get_tuple(config)
  4. 若 cp 为 None → 返回 None（无 checkpoint）
  5. 否则返回 cp.metadata.get("step") — LangGraph 在每次 checkpoint 时写入的步骤号
边界条件：
  • DB 文件存在但表中无该 thread_id 的行 → get_tuple() 返回 None
  • DB 文件损坏 → sqlite3.connect 可能抛 sqlite3.DatabaseError，调用方需自行捕获
  • 同一 ticker 的不同 date 互不干扰（不同 thread_id）

7.5 clear_checkpoint()

None clear_checkpoint(data_dir, ticker, date)

删除指定 ticker+date 在 writes 和 checkpoints 两表中的所有行。

边界条件：
  • DB 文件不存在 → 静默返回（不抛异常）
  • 表不存在（sqlite3.OperationalError）→ 被 except 捕获，静默跳过

7.6 clear_all_checkpoints()

int clear_all_checkpoints(data_dir: str | Path)

删除 {data_dir}/checkpoints/ 下所有 .db 文件。返回删除的文件数。若目录不存在则返回 0。

注意：此函数删除的是整个文件而非单行，一旦执行所有 ticker 的断点全部清空。建议仅在批量重置场景使用，日常使用 clear_checkpoint() 精确清除单次运行的断点。
