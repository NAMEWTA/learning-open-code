0001 — Provider 路由全貌地图

学习目标：能画出 11 种 provider 经工厂路由到 4 个 Client 类的完整映射关系，解释 _OPENAI_COMPATIBLE 元组的 8 合 1 复用设计，并说出双 LLM（quick_think / deep_think）在配置层、注册表层和工厂层的三段体现方式。

1. 配置层入口：两条 LLM 链路的起点

整条路由链路的起点在 default_config.py 的三个配置键：

DEFAULT_CONFIG = {
    "llm_provider": "openai",        # 默认供应商（CLI 可覆盖）
    "deep_think_llm": "gpt-5.4",     # 深度推理模型
    "quick_think_llm": "gpt-5.4-mini", # 快速响应模型
    "backend_url": None,             # 自定义代理网关
}

这三个键定义了两条并行的 LLM 链路——quick_think 链路服务 14 个快速 Agent（7 Analyst + 2 Researcher + 3 Risk Debator + 1 Trader + 1 Quality Gate），deep_think 链路仅服务 2 个核心决策 Agent（Research Manager + Portfolio Manager）。每条约 30-50 次调用中，只有 2 次走深度模型，这是成本控制的核心策略。

CLI 交互时会覆盖 llm_provider，但 quick/deep 的模型选择来自 model_catalog.py 的 MODEL_OPTIONS 注册表——它为 9 个供应商（不含 OpenRouter 和 Azure）各自维护 quick 和 deep 两套候选列表。这构成了配置层 + 注册表层的双层模型选择机制。

2. 工厂路由决策树：11 进 4 出

factory.py 的 create_llm_client() 是整条链路的唯一入口。graph 层的 setup.py 调用它 16 次（14 个 quick + 2 个 deep），每次传入 provider 字符串和模型名，获得一个已配置的 Client 实例。路由逻辑分 4 路：

create_llm_client(provider, model, base_url, **kwargs)
  │
  ├─ provider in _OPENAI_COMPATIBLE? ──→ OpenAIClient      (8 种)
  │   ("openai","xai","deepseek","qwen","glm",
  │    "ollama","openrouter","minimax")
  │
  ├─ provider == "anthropic"? ──→ AnthropicClient           (1 种)
  │
  ├─ provider == "google"? ──→ GoogleClient                 (1 种)
  │
  ├─ provider == "azure"? ──→ AzureOpenAIClient             (1 种)
  │
  └─ 都不匹配 ──→ ValueError("Unsupported LLM provider")

三个关键设计决策：

  • 8 合 1 复用——OpenAI、xAI、DeepSeek、Qwen、GLM、Ollama、OpenRouter、MiniMax 全部兼容 OpenAI Chat Completions 协议，统一走 OpenAIClient。差异点（base_url、API Key 环境变量、DeepSeek thinking-mode）在 get_llm() 内部通过 provider 参数区分
  • 懒加载 import——每个分支内部才 import 具体 Client 类，避免模块导入时就拉入重型 SDK（如 langchain_anthropic），也不会因缺少某家 API Key 而导致整个模块导入失败
  • provider 参数透传——OpenAIClient 构造函数接收 provider= 参数，存储为实例属性，供 get_llm() 内部查表选择 base_url 和 API Key 环境变量

3. base_client 抽象接口契约

所有 4 个 Client 类必须实现 BaseLLMClient(ABC) 的两个抽象方法，构成工厂返回值的统一契约：

class BaseLLMClient(ABC):
    def __init__(self, model, base_url=None, **kwargs): ...

    @abstractmethod
    def get_llm(self) -> Any:
        """返回已配置的 LangChain ChatModel 实例"""
        pass

    @abstractmethod
    def validate_model(self) -> bool:
        """校验模型名是否在已知列表中"""
        pass

    def warn_if_unknown_model(self) -> None:
        """模板方法：校验失败发 RuntimeWarning，不阻断"""

同文件中的 normalize_content(response) 是模块级工具函数——多家供应商返回的 response.content 是 [{"type":"reasoning",...}, {"type":"text","text":"..."}] 列表而非字符串。此函数提取所有 text 块拼接为纯字符串，确保下游 Agent 收到统一格式。

通过这个接口契约，graph 层调用 client.get_llm() 即可获得可用的 LangChain ChatModel，无需知道底层是 ChatOpenAI、ChatAnthropic 还是 ChatGoogleGenerativeAI。

4. 双 LLM 在整条链路中的三段体现

将前三节串起来，双 LLM 策略在链路上有三个明确的体现点：

阶段位置体现
配置层 default_config.py quick_think_llm 和 deep_think_llm 两个键，定义默认模型名
注册表层 model_catalog.py MODEL_OPTIONS 为每个 provider 维护 quick 和 deep 两套候选列表
调用层 graph/setup.py 对 14 个 quick Agent 传 quick 模型创建 Client；对 2 个 deep Agent 传 deep 模型创建 Client

客户端层本身不做 quick/deep 决策——它只负责按传入参数配置 LLM 实例。选择权在上游的 graph 编排层。这种关注点分离使得：切换 provider 只需改配置键；调整某个角色的模型只需改 setup 中的传参；新增模型只需改 MODEL_OPTIONS 注册表。三层各自独立演进。

检索练习

问题 1：不翻看上文，凭记忆写出 11 种 provider 到 4 个 Client 类的完整映射关系。

问题 2：为什么 _OPENAI_COMPATIBLE 元组能包含 8 种不同供应商？它们共享的协议基础是什么？

问题 3：双 LLM 策略在链路中有三段体现——配置层、注册表层、调用层。如果要把 Anthropic 的 quick 模型从 Sonnet 4.6 换成 Haiku 4.5，需要改哪几处？

参考答案
1：OpenAI / xAI / DeepSeek / Qwen / GLM / Ollama / OpenRouter / MiniMax（8 种）→ OpenAIClient；Anthropic → AnthropicClient；Google → GoogleClient；Azure → AzureOpenAIClient。共 4 个具体 Client 类。
2：它们都兼容 OpenAI Chat Completions 协议（/v1/chat/completions 端点），仅在 base_url 和认证方式上有差异。LangChain 的 ChatOpenAI 封装了这个协议，8 种 provider 只需替换 base_url 和 API Key 即可复用同一套代码。
3：只需改一处——model_catalog.py 中 anthropic.quick 列表的默认选项。调用层（setup.py）和配置层（default_config）不需要改动，因为 Anthropic 的 provider 路由和 quick_think_llm 键名不变。
