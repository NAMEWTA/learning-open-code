2. main.py — 项目根入口

位于项目根目录，用作脚本/IDE 调试入口。直接实例化图并运行一次分析，不带 CLI 交互。

入口脚本（无封装函数）

from tradingagents.graph.trading_graph import TradingAgentsGraph from tradingagents.default_config import DEFAULT_CONFIG # 1. 复制默认配置并覆写 config = DEFAULT_CONFIG.copy() config["deep_think_llm"] = "gpt-5.4-mini" config["quick_think_llm"] = "gpt-5.4-mini" config["max_debate_rounds"] = 1 config["data_vendors"] = { ... } # 2. 实例化图 ta = TradingAgentsGraph(debug=True, config=config) # 3. 正向传播 _, decision = ta.propagate("NVDA", "2024-05-10") # 4. （可选）反思与记忆 # ta.reflect_and_remember(1000)
步骤调用说明
1. 配置 DEFAULT_CONFIG.copy() 复制默认配置，覆写 LLM 模型、辩论轮次、数据厂商
2. 建图 TradingAgentsGraph(debug=True, config=config) 用自定义 config 初始化多 Agent 图
3. 分析 ta.propagate(ticker, date) 正向传播，返回 (final_state, decision)
4. 反思 ta.reflect_and_remember(returns) 基于持仓收益反思错误，存入长期记忆（注释掉的示例）
入口关系：此文件绕过 CLI 交互，直接调用 graph 层。CLI 模式的完整流程在 cli/main.py 的 run_analysis() 中。

方法列表

init_for_analysis(selected_analysts: list[str])

根据用户选择的分析师动态构建 agent_status、report_sections，重置所有运行时状态。

关键逻辑：遍历 REPORT_SECTIONS，仅注册与选中 analyst 相关的报告段（analyst_key=None 的段始终包含）。这样未被选中的分析师不会出现在进度面板中。

return None

追加带时间戳的消息到 messages 队列。在 run_analysis 中被装饰器增强，额外写入日志文件。

追加工具调用记录到 tool_calls 队列。在 run_analysis 中被装饰器增强，额外写入日志文件。

return None

update_agent_status(agent: str, status: str)

更新指定 Agent 的运行状态（pending/in_progress/completed/error），同步更新 current_agent。

return None

update_report_section(section_name: str, content: str)

更新指定报告段的文本内容，触发 current_report 和 final_report 的重新拼接。在 run_analysis 中被装饰器增强，额外写入 .md 文件到磁盘。

return None

get_completed_reports_count() → int

统计已完成的报告段数量。判断"完成"的双重条件：报告段有内容 AND 负责最终定稿的 Agent 状态为 completed。防止辩论轮次的中间更新被误计为完成。

3.4 run_analysis() — 核心交互循环（重点展开）

run_analysis(checkpoint: bool = False)

def run_analysis(checkpoint: bool = False) -> None:
参数类型默认说明
checkpointboolFalse是否启用 LangGraph 断点续跑（节点级持久化）

执行流程详解

阶段 A — 用户交互获取选择

  1. 调用 get_user_selections() → 展示欢迎界面 + 公告 + 8 步引导

阶段 B — 组装配置

  1. 从 DEFAULT_CONFIG.copy() 创建运行时配置
  2. 将用户选择写入 config：max_debate_rounds、quick_think_llm、deep_think_llm、backend_url、llm_provider、provider 特有配置、output_language、checkpoint_enabled

阶段 C — 初始化基础设施

  1. 创建 StatsCallbackHandler() — 统计 LLM/Tool/Token
  2. 规范化分析师顺序：用户选集合 ∩ ANALYST_ORDER 固定顺序
  3. 创建 TradingAgentsGraph(analysts, config, debug=True, callbacks=[stats_handler])
  4. 调用 message_buffer.init_for_analysis(analysts)
  5. 创建结果目录：results_dir / ticker / date / reports
  6. 用装饰器增强 message_buffer 的三个方法（写入日志/md 文件）

阶段 D — Rich Live 流式循环

  1. 创建 Rich Layout（header/main[progress+messages]/analysis/footer）
  2. 创建 Live(layout, refresh_per_second=4) 上下文
  3. 创建初始状态：graph.propagator.create_initial_state(ticker, date)
  4. 获取 graph args（含 callbacks）：graph.propagator.get_graph_args(callbacks=[stats_handler])
  5. 核心流：for chunk in graph.graph.stream(init_agent_state, **args)
    • 每条 message → classify_message_type() → HumanMessage/AIMessage/ToolMessage
    • 消息去重（通过 message.id 的 set）
    • tool_calls → 提取 name + args → message_buffer.add_tool_call
    • 每个 chunk 驱动 update_analyst_statuses() 状态机推进
    • 检测 investment_debate_state → Research Team 状态转换
    • 检测 trader_investment_plan → Trader 完成 → Aggressive Analyst 激活
    • 检测 risk_debate_state → Risk + Portfolio Manager 状态转换
    • 每次迭代刷新 update_display(layout)

阶段 E — 后处理

  1. 退出 Live 上下文，所有 Agent 标记为 completed
  2. 提示保存报告 → save_report_to_disk(final_state, ticker, save_path)
  3. 提示展示报告 → display_complete_report(final_state)
装饰器增强说明：在 run_analysis() 内部，message_buffer 的三个方法被动态替换为带文件写入能力的装饰器版本：
  • add_message → 追加写入 message_tool.log
  • add_tool_call → 追加写入 message_tool.log
  • update_report_section → 每个报告段独立写入 {section_name}.md
装饰器通过 @wraps(func) 保留原始签名，在原始调用之后追加文件写入逻辑。

3.5 save_report_to_disk() — 磁盘保存（重点展开）

save_report_to_disk(final_state, ticker, save_path)

def save_report_to_disk( final_state: dict, ticker: str, save_path: Path, ) -> Path:
参数类型说明
final_statedictLangGraph 流式执行的最终状态，包含所有报告段内容
tickerstr股票代码，用于报告标题
save_pathPath报告输出目录（用户指定或默认）

return Path — 返回 complete_report.md 的完整路径

目录结构（5 级子目录 + 合并报告）

{save_path}/ ├── complete_report.md ← 完整合并报告（Markdown 格式） ├── 1_analysts/ │ ├── market.md ← Market Analyst 报告 │ ├── sentiment.md ← Social Analyst 报告 │ ├── news.md ← News Analyst 报告 │ └── fundamentals.md ← Fundamentals Analyst 报告 ├── 2_research/ │ ├── bull.md ← Bull Researcher 论点 │ ├── bear.md ← Bear Researcher 论点 │ └── manager.md ← Research Manager 裁决 ├── 3_trading/ │ └── trader.md ← Trader 交易计划 ├── 4_risk/ │ ├── aggressive.md ← Aggressive Analyst 风险分析 │ ├── conservative.md ← Conservative Analyst 风险分析 │ └── neutral.md ← Neutral Analyst 风险分析 └── 5_portfolio/ └── decision.md ← Portfolio Manager 最终决策

输出格式说明

单个报告文件（Markdown）：

  • 每个 agent 的输出以纯 Markdown 文本写入对应 .md 文件
  • 编码 UTF-8，直接调用 Path.write_text(content, encoding="utf-8")
  • 仅在 final_state 中存在对应 key 时才创建该文件/目录

complete_report.md（合并报告）：

  • 标题：# Trading Analysis Report: {ticker}
  • 生成时间戳 + 免责声明（中文）
  • 五大章节按罗马数字编号：I. Analyst Team / II. Research Team / III. Trading Team / IV. Risk Management / V. Portfolio Manager
  • 每个章节内部以 ### Agent名称 分隔各 agent 内容
注意：此函数仅输出 Markdown 格式。项目不生成 JSON 格式报告。课程中提到的"Markdown + JSON 双格式"为描述不准确——实际为 Markdown 单格式，通过 5 级子目录 + 合并文件实现结构化存储。
