0001 — CLI 命令行界面模块总览

学习目标：能说出 CLI 模块的 8 步交互式引导流程、Typer 命令注册方式、LLM provider 选择逻辑、save_report_to_disk 的 5 级目录结构，以及 CLI 与 trading_graph 的调用边界。

1. 模块定位与职责

cli/ 是整个 TradingAgents-Astock 的用户入口层。它位于架构最顶层——在 Graph 层和 Agent 层之上，负责将用户的命令行输入转化为配置参数，驱动 trading_graph 执行多 Agent 分析流程。模块本身不包含任何业务逻辑，仅做参数收集、流程编排和结果展示。

┌──────────────────────────────────────┐ │ cli/ ◀── 本节重点 │ ├──────────────────────────────────────┤ │ trading_graph (Graph 工作流) │ ├──────────────────────────────────────┤ │ agents / dataflows / llm_clients │ └──────────────────────────────────────┘
cli/ ├── __init__.py ← 包初始化 ├── main.py ← 🔴 Typer 入口 + 8 步交互 + 流式展示 ├── utils.py ← 🔴 交互式选择逻辑（分析师/深度/LLM/思考） ├── config.py ← 🟡 CLI 配置常量 ├── models.py ← 🟡 AnalystType 枚举定义 ├── announcements.py ← 🟡 公告拉取与展示 ├── stats_handler.py ← 🟡 LangChain 回调统计（LLM/Tool 计数） └── static/ └── welcome.txt ← ASCII Art 欢迎界面

入口路径：根目录 main.py 直接创建 TradingAgentsGraph（用于 IDE/脚本调试），Typer CLI 则通过 cli/main.py 的 analyze 命令启动完整交互流。CLI 依赖 Rich（终端美化）、Questionary（交互式问答）、Typer（命令框架）三个核心库。

2. 8 步交互式引导流程

get_user_selections() 是 CLI 的核心函数，按顺序引导用户完成 8 步输入。每个步骤通过 questionary 库提供交互式选择（select / checkbox / text），步骤间无依赖——所有参数收集完毕后才组装 config。

Step 1: Ticker Symbol → 输入股票代码（含安全校验 safe_ticker_component） Step 2: Analysis Date → YYYY-MM-DD 格式，不允许未来日期 Step 3: Output Language → 12 种语言选择（支持自定义输入） Step 4: Analysts Team → 多选：Market / Social / News / Fundamentals Step 5: Research Depth → Shallow(1) / Medium(3) / Deep(5) — 控制辩论轮次 Step 6: LLM Provider → 10 种 provider：OpenAI/Google/Anthropic/xAI/ DeepSeek/Qwen/GLM/OpenRouter/Azure/Ollama Step 7: Thinking Agents → 分 shallow/deep 两个 LLM 引擎独立选择 Step 8: Provider Config → 根据 provider 显示不同选项： Google→thinking_level / OpenAI→reasoning_effort / Anthropic→effort_level

LLM Provider 选择逻辑：select_llm_provider()（在 cli/utils.py）维护了一个 10 元素的 PROVIDERS 列表，每项包含显示名、provider_key 和默认 backend_url。选定后，步骤 7 从 tradingagents.llm_clients.model_catalog 的注册表中按 provider 拉取可选模型列表。最终所有参数（ticker、date、analysts、depth、llm_provider、shallow_thinker、deep_thinker、thinking 配置、output_language）汇入一个 dict 返回。

3. 流式分析与实时展示

run_analysis() 拿到用户选择后，按以下顺序执行：

  1. 组装 config — 将用户选择的 depth、llm 模型、provider、thinking 配置、output_language 合并到 DEFAULT_CONFIG
  2. 初始化 StatsCallbackHandler — 跟踪 LLM 调用次数、Tool 调用次数、token 消耗
  3. 初始化 TradingAgentsGraph — 传入 selected_analysts、config、debug=True、callbacks
  4. 初始化 MessageBuffer — 管理 agent_status、report_sections、消息日志和工具调用记录
  5. 创建 Rich Layout — 四区域：header（欢迎）、progress（Agent 状态表）、messages（消息+工具日志）、footer（统计信息）
  6. 流式执行 graph.stream() — 逐 chunk 处理：分类消息类型 → 更新 agent 状态 → 更新报告段落 → 刷新 Live 布局

流式循环中的状态转换由 update_analyst_statuses() 驱动：检测 chunk 中的 report 字段 → 标记对应 Analyst 为 completed → 激活下一个 Analyst 为 in_progress → 所有 Analyst 完成后激活 Bull Researcher → 依次推进至 Portfolio Manager。整个过程通过 Rich Live 组件（4 fps 刷新）实时渲染终端。

4. 报告保存与展示

流式分析完成后，save_report_to_disk(final_state, ticker, save_path) 将最终状态写入 5 级子目录结构：

{save_path}/ ├── 1_analysts/ │ ├── market.md ← Market Analyst 报告 │ ├── sentiment.md ← Social Analyst 报告 │ ├── news.md ← News Analyst 报告 │ └── fundamentals.md ← Fundamentals Analyst 报告 ├── 2_research/ │ ├── bull.md ← Bull Researcher 辩论记录 │ ├── bear.md ← Bear Researcher 辩论记录 │ └── manager.md ← Research Manager 裁决 ├── 3_trading/ │ └── trader.md ← Trader 投资计划 ├── 4_risk/ │ ├── aggressive.md ← 激进风控分析 │ ├── conservative.md ← 保守风控分析 │ └── neutral.md ← 中性风控分析 ├── 5_portfolio/ │ └── decision.md ← Portfolio Manager 最终决策 └── complete_report.md ← 汇总版完整报告（含免责声明 header）

保存后会依次提示：是否在终端展示完整报告（display_complete_report()，用 Rich Panel 分段渲染五大团队输出）。报告保存路径有安全校验——若用户输入的路径在项目目录之外，需二次确认后才执行写入。
