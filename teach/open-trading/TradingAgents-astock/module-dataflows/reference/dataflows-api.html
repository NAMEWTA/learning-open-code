1. interface.py — 路由层

该模块是 dataflows 的统一入口：上层调用方不直接 import vendor 模块，而是通过 route_to_vendor(method, *args, **kwargs) 将调用派发到配置指定的 vendor 实现。路由支持 tool 级 > category 级 两级覆盖 + 自动 fallback 链。

route_to_vendor

def route_to_vendor(method: str, *args, **kwargs) -> str
参数类型说明
methodstr方法名，如 "get_stock_data"、"get_fundamentals"
*args, **kwargs—透传给 vendor 实现函数的参数
返回：str — vendor 实现函数的返回值（通常是带 header 的 CSV 或格式化文本）。
边界条件：method 不在 VENDOR_METHODS 中 → 抛出 ValueError；所有 vendor 均失败 → 抛出 RuntimeError("No available vendor for ...")。
异常类型：
  • ValueError — method 不在 VENDOR_METHODS 注册表中
  • AlphaVantageRateLimitError — Alpha Vantage 限流，仅此异常触发 vendor 降级（其他异常直接抛出）
  • RuntimeError — fallback 链上所有 vendor 均失败

路由流程：

  1. get_category_for_method(method) → 确定类别（如 "fundamental_data"）
  2. get_vendor(category, method) → 查 tool_vendors（精确匹配优先）→ 查 data_vendors（类别匹配）→ 返回 vendor 字符串（如 "a_stock,yfinance"）
  3. 分割逗号得到首选列表，拼接其余可用 vendor 构建完整 fallback 链
  4. 遍历调用，仅 AlphaVantageRateLimitError 触发降级继续下一个 vendor

get_vendor

def get_vendor(category: str, method: str = None) -> str

返回配置中该类别/方法的 vendor 字符串。优先级：tool_vendors[method] > data_vendors[category] > "default"。

get_category_for_method

def get_category_for_method(method: str) -> str

反向查 TOOLS_CATEGORIES，返回 method 所属类别名。method 不在任何类别中时抛出 ValueError。

常量

TOOLS_CATEGORIES — 5 个类别分组：

类别键包含方法
core_stock_apisget_stock_data
technical_indicatorsget_indicators
fundamental_dataget_fundamentals, get_balance_sheet, get_cashflow, get_income_statement
news_dataget_news, get_global_news, get_insider_transactions
signal_dataget_profit_forecast, get_hot_stocks, get_northbound_flow, get_concept_blocks, get_fund_flow, get_dragon_tiger_board, get_lockup_expiry, get_industry_comparison

VENDOR_LIST — ["a_stock", "yfinance", "alpha_vantage"]

VENDOR_METHODS — dict[str, dict[str, callable]]，17 个方法到 3 个 vendor 实现的映射。signal_data 类别的 8 个方法仅有 a_stock 实现。

2. a_stock.py — A 股数据 vendor

零第三方数据依赖（无 akshare），所有数据源为直接 HTTP API 或 mootdx TCP。以下按 VENDOR_METHODS 注册顺序排列。

2.1 get_stock_data

def get_stock_data( symbol: Annotated[str, "A-stock code (e.g. 688017, SH688017)"], start_date: Annotated[str, "Start date in yyyy-mm-dd format"], end_date: Annotated[str, "End date in yyyy-mm-dd format"], ) -> str
参数类型说明
symbolstr支持纯 6 位代码、SH/SZ/BJ 前缀、.SH/.SZ/.BJ 后缀
start_datestr起始日期，YYYY-MM-DD 格式
end_datestr结束日期，YYYY-MM-DD 格式
返回：str — 带 header 的 CSV（Date, Open, High, Low, Close, Volume）。header 包含代码、日期范围、记录数、数据源标注。
边界条件：mootdx 和新浪均无数据 → 返回错误消息字符串（不抛异常）；日期范围内无交易日 → 返回 "No data found" 消息。
异常：无显式 raise（内部捕获所有异常转为字符串返回）。

数据源降级链：mootdx TCP (bars, category=4) → 新浪 HTTP (money.finance.sina.com.cn) → 返回错误字符串。

价格列（Open/High/Low/Close）自动 round 到 2 位小数。如果 mootdx 数据滞后于 end_date，自动用新浪日线补充缺失日期。

2.3 get_fundamentals

def get_fundamentals( ticker: Annotated[str, "A-stock code"], curr_date: Annotated[str, "current date"] = None, ) -> str

3. a_stock.py — 内部私有函数

以下为 a_stock.py 中的模块级私有函数，按功能域分组简述。

3.1 _em_get — 东财防封限流机制（重点展开）

def _em_get(url, params=None, headers=None, timeout=15, **kwargs) -> requests.Response
设计背景：东财系 HTTP 接口有严格风控——每秒 >5 次 / 单 IP 并发 ≥10 / 1 分钟 ≥200 次 / 5 分钟 ≥300 次 → 临时封 IP。多 Agent 投研跑批量分析时高频请求东财是被封的头号元凶。v0.2.11 起所有 eastmoney.com 请求统一走此入口。

限流机制（三层防护）：

层级机制实现
1. 串行限流 模块级时间戳 _em_last_call，记录上次东财请求时刻。本次调用计算距上次的间隔，若 < EM_MIN_INTERVAL（默认 1.0s），sleep 补齐差值 time.time() - _em_last_call[0]
2. 随机抖动 在补齐间隔基础上叠加 random.uniform(0.1, 0.5) 秒随机延迟，避免多实例的固定节奏被风控识别 time.sleep(wait + random.uniform(0.1, 0.5))
3. Session 复用 模块级 requests.Session 实例 _EM_SESSION，启用 HTTP Keep-Alive，复用 TCP 连接减少握手开销 _EM_SESSION.get(...)

可调参数：

变量默认值覆盖方式说明
EM_MIN_INTERVAL1.0s环境变量 EM_MIN_INTERVAL批量多 Agent 场景建议 1.5~2.0s
timeout15s调用时传参各端点可独立设置
headers默认 UA调用时传参（覆盖 session 默认 header）用于保留各端点自己的 Referer/Origin

覆盖的调用点（7 个）：push2 实时行情、push2 个股信息、push2 资金流分钟线、push2 行业板块列表、push2his 资金流日线、datacenter-web 龙虎榜/解禁、search-api-web 个股新闻、np-weblist 全球资讯。

边界条件：_em_last_call 用单元素 list [0.0] 而非 float 是为了在 finally 块中可靠地更新模块级变量（Python 闭包中对不可变类型的赋值会创建局部变量）；timestamp 更新放在 finally 中确保即使请求异常也记录时间，防止异常后立即重试触发风控。
非东财源不受限流：mootdx (TCP)、腾讯 (qt.gtimg.cn)、新浪 (sina.com.cn)、同花顺 (10jqka.com.cn / hexin.cn)、财联社 (cls.cn)、百度 (pae.baidu.com) — 这 6 个源实测不封 IP 或风控极弱，直连不走 _em_get。新增东财端点时务必走 _em_get 而非裸 requests.get。

3.2 resolve_ticker — 中文 ticker 解析链路（重点展开）

def resolve_ticker(user_input: str) -> str

完整链路：

用户/LLM 输入（"宝光股份" 或 "600379" 或 "SH600379"） │ ├─ 检测是否含中文字符 (Unicode 范围 U+4E00–U+9FFF) │ ├─ 不含中文 → _normalize_ticker() │ ├─ 去 .SH/.SZ/.BJ 后缀 │ ├─ 去 SH/SZ/BJ 前缀 │ └─ safe_ticker_component() 安全校验 │ ├─ 路径注入检测 (_TICKER_PATH_RE) │ ├─ 长度 ≤ 32 │ └─ 不能全是 "." │ └─ 含中文 → 去空格 → _build_name_code_map() ├─ 模块级缓存 (_name_to_code / _code_to_name) │ └─ 已缓存则直接返回 └─ 未缓存 → _get_mootdx_client() ├─ TCP 探测 10 个内置通达信服务器 (port 7709) │ └─ 使用 socket.create_connection 2s 超时探测 ├─ fallback 1: Quotes.factory(bestip=True) 测速 ├─ fallback 2: Quotes.factory() 裸工厂（老用户 config 已有 IP） └─ fallback 3: RuntimeError（TCP 7709 全部不可达） └─ 成功后: client.stocks(market=0/1) 遍历沪深两市 └─ 过滤 code 匹配 ^[036]\d{5}$ 的条目 └─ 构建 n2c / c2n dict → 缓存到模块全局变量 └─ 精确匹配 → 返回 code └─ 模糊匹配（substring in name） ├─ 唯一匹配 → 返回 code ├─ 多个匹配 → ValueError 列出最多 5 个候选 └─ 无匹配 → ValueError
异常：
  • ValueError("输入不能为空") — 空白输入
  • ValueError("ticker must be a non-empty string") — safe_ticker_component 校验失败
  • ValueError("ticker exceeds 32 chars") — 超长代码
  • ValueError("ticker contains characters not allowed...") — 含路径分隔符等危险字符
  • ValueError("无法通过 mootdx 解析股票名称...") — 通达信不可达，包装原始异常
  • ValueError("'{s}' 匹配到多只股票: ...") — 模糊匹配有歧义
  • ValueError("找不到股票 '{s}'") — 无匹配
  • RuntimeError("mootdx 通达信服务器均不可达...") — 所有 fallback 均失败

与 safe_ticker_component 的协作：utils.py 的 safe_ticker_component 是更上层安全门——当它检测到中文字符时，自动调用 resolve_ticker 完成转码后再做路径安全校验。这形成了"中文检测 → 名称解析 → 路径校验"的三段防线。

3.5 OHLCV 缓存与补充链路

_load_ohlcv_astock(symbol, curr_date) │ ├─ 检查 CSV 缓存 (data_cache_dir/{code}-astock-daily.csv) │ ├─ 命中 + 当日修改 → 读 CSV → _supplement_stale_ohlcv_with_sina() │ │ ├─ _needs_sina_supplement(df, curr_date) │ │ │ ├─ last_date < curr_date → 需要补充 │ │ │ └─ last_date >= curr_date → 跳过 │ │ ├─ 需要补充 → _sina_kline_fallback() → _merge_ohlcv() │ │ └─ supplemented=True → 回写 CSV │ │ └─ 按 curr_date 截断返回 │ │ │ └─ 未命中或过期 → mootdx 拉取 │ ├─ client.bars(symbol=code, category=4, offset=800) │ ├─ 列重命名 + normalize │ ├─ 失败 → 新浪 fallback (_sina_kline_fallback) │ ├─ _supplement_stale_ohlcv_with_sina() 补充 │ └─ 写 CSV → 按 curr_date 截断返回

缓存策略：单文件按代码缓存（例：~/.tradingagents/cache/600379-astock-daily.csv），当日已拉取则复用，隔日重新拉取。缓存文件无过期机制——仅当日 mtime 检查。

4. utils.py

safe_ticker_component

def safe_ticker_component(value: str, *, max_len: int = 32) -> str
参数类型默认值说明
valuestr—待校验的 ticker 字符串
max_lenint32最大允许长度
返回：str — 校验通过的原值或中文字符自动解析后的 6 位代码。

安全校验规则（按执行顺序）：

  1. 非空字符串检查 → ValueError("ticker must be a non-empty string")
  2. 中文检测（正则 [一-鿿]）→ 有中文则调用 resolve_ticker(value) 自动转码
  3. 长度检查（> max_len = 32）→ ValueError("ticker exceeds 32 chars")
  4. 路径安全正则 ^[A-Za-z0-9._\-\^]+$ → 不匹配则 ValueError("ticker contains characters not allowed in a filesystem path")
  5. 全点检查（如 "..."）→ ValueError("ticker cannot consist solely of dots")
为什么需要这个函数：ticker 值会被直接插入文件路径（CSV 缓存文件名、数据文件路径）。如果不对用户/LLM 输入做校验，恶意或异常的 ticker（如 ../../etc/passwd）可能导致路径穿越，读写任意文件。中文股票名是最常见的"异常"输入——LLM 在 tool call 时经常返回中文名而非代码，此函数自动兜底转码。

与 resolve_ticker 的关系：safe_ticker_component 是安全边界守卫，resolve_ticker 是业务逻辑。前者检测到中文后 import + 调用后者，然后将解析结果再做路径安全校验。这是两段式防护：先解决业务问题（中文→代码），再确保安全（路径注入防御）。

Yahoo Finance 是面向美股/全球市场的免费 vendor，基于 yfinance 库。共 2 个文件：

关键设计点：

  • 所有 yfinance API 调用通过 yf_retry 包装，自动处理 HTTP 429 限流（指数退避 2/4/8s，最多 3 次重试）

  • get_stock_stats_indicators_window 优先使用 bulk 模式，失败时自动降级到逐日 get_stockstats_indicator
  • yfinance 库对美股符号默认加上 .NS/.BO 等后缀——调用方传入符号时需注意
