4. 双 LLM 策略完整链路

双 LLM 策略贯穿三个层次，从配置声明到实际使用形成一条完整的链路：

层级文件作用示例值
1. 配置声明default_config.py定义 quick/deep 使用的默认模型 IDquick_think_llm="gpt-5.4-mini"
2. 模型选项model_catalog.py为每个 provider 维护 quick/deep 候选列表，CLI 交互选择MODEL_OPTIONS["openai"]["quick"]
3. Client 创建graph/setup.py为不同 Agent 创建 quick 或 deep 的 LLM 实例2 个 Manager 用 deep，其余 15 个用 quick

Agent 分配：

  • quick_think_llm：7 Analyst + Bull + Bear + Quality Gate + 3 Risk Debator + Trader = 14 个节点
  • deep_think_llm：Research Manager + Portfolio Manager = 2 个节点

一次完整的股票分析大约产生 30-50 次 LLM 调用，其中仅 2 次使用深度模型。这种策略将成本控制在合理范围内，同时保证关键决策节点有更强的推理能力。

5. 内容归一化机制

normalize_content(response) 定义在 base_client.py，被所有 4 个 Client 类的子类 ChatModel 的 invoke() 覆写调用。

问题来源：

  • OpenAI Responses API (use_responses_api=True) 返回 [{"type": "reasoning", "reasoning": "..."}, {"type": "text", "text": "最终回答"}]
  • Google Gemini 3 同样返回 list of content blocks
  • 下游 Agent 的 prompt 模板和辩论逻辑期望 response.content 是字符串

处理逻辑：

  1. 如果 content 已是字符串，直接返回
  2. 如果是列表，遍历所有项：
    • type == "text" 的 dict 取出 "text" 值
    • 裸字符串直接保留
    • 其他类型（reasoning / metadata）丢弃
  3. 用换行符拼接所有文本块，写回 response.content

实现位置：每个 Client 类通过定义自己的 NormalizedChatXxx 子类来集成归一化，例如 NormalizedChatOpenAI(ChatOpenAI)、NormalizedChatAnthropic(ChatAnthropic)，均在 invoke() 中调用 normalize_content()。
