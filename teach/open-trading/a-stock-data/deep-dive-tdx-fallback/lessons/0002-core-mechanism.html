三级 fallback：TCP 探测 → bestip 测速 → 裸 factory

L4 深度剖析 mootdx fallback 第 2/2 节 估计时长：15 分钟

1. 三级 fallback 的总体设计

tdx_client() 的三级兜底策略按"可靠性从高到低"排列：

第一级（首选）：遍历 _TDX_SERVERS 列表，TCP 探测每台服务器是否可达，用第一台可达的。
第二级（fallback 1）：调用 Quotes.factory(bestip=True)，让 mootdx 自己测速选最优 IP。
第三级（fallback 2）：调用 Quotes.factory() 裸创建，依赖老用户 config.json 中已有的 IP。
兜底：以上全部失败 → RuntimeError，明确报错而非死等。

顺序不能颠倒——每一级都是对前一级失效场景的"降级兼容"。以下是完整的源码结构：

def tdx_client(market='std'):
    for ip, port in _TDX_SERVERS:
        if _probe(ip, port):
            return Quotes.factory(market=market, server=(ip, port))  # 第一级
    try:
        return Quotes.factory(market=market, bestip=True)            # 第二级
    except Exception:
        pass
    try:
        return Quotes.factory(market=market)                         # 第三级
    except Exception as e:
        raise RuntimeError("所有 mootdx 服务器均不可达...")          # 兜底

2. 第一级：TCP 探测 + 内置服务器列表

_TDX_SERVERS 是 a-stock-data 维护的 10 个实测可用通达信服务器地址（2026-06 验证）：

_TDX_SERVERS = [
    ('119.97.185.59', 7709), ('124.70.133.119', 7709), ('116.205.183.150', 7709),
    ('123.60.73.44', 7709),  ('116.205.163.254', 7709), ('121.36.225.169', 7709),
    ('123.60.70.228', 7709), ('124.71.9.153', 7709),    ('110.41.147.114', 7709),
    ('124.71.187.122', 7709),
]

_probe() 用 Python 标准库 socket 做 TCP 握手探测：

def _probe(ip, port, timeout=2.0):
    try:
        with socket.create_connection((ip, port), timeout=timeout):
            return True
    except Exception:
        return False
为什么 timeout=2.0s？ 10 台服务器在 2s 超时下，最坏情况 20s 完成全部探测。实际大部分服务器在 50-100ms 内即可握手成功，第一台可达的就直接返回。2s 是可靠性与速度的平衡——太短（0.5s）可能因网络抖动误判不可达，太长（5s）则海外用户等待过久。

为什么需手动维护服务器列表？通达信不公开其服务器基础设施，IP 可能变更、退役或被 DDoS 防护封锁。10 个 IP 提供了冗余——单个 IP 故障不会影响连接，多个 IP 同时出现在不同网段也分散了运营商级别的风险。

3. 第二、三级：bestip 测速与裸 factory 兜底

当第一级全部探测失败时，进入第二级：回退到 mootdx 内置的 bestip=True 测速机制。bestip 是 mootdx 内部维护的通达信服务器列表 + 测速逻辑——它比 _TDX_SERVERS 覆盖范围更广，但依赖 config.json 和内部数据文件，这正是 BESTIP 空串 bug 的来源。第二级的作用是覆盖那些"服务器 IP 列表过期但 mootdx 内部列表更全"的场景。

如果 bestip 也失败（例如 config.json 损坏），进入第三级：裸 Quotes.factory(market=market)——它依赖老用户 config.json 中已有的可用 IP。两级兜底确保不同环境都有路径可达。

如果连裸 factory 都失败了（典型场景：海外 IP，所有国内通达信服务器不通），tdx_client() 抛出明确的 RuntimeError：

raise RuntimeError(
    "所有 mootdx 服务器均不可达。海外网络通常全部超时（TCP 7709），"
    "请走国内代理或更新 _TDX_SERVERS 列表。原始错误：%s" % e
)
设计原则：快速失败优于静默死等。裸 Quotes.factory() 在海外环境下可能因为等待 TCP 超时而卡住数分钟。tdx_client() 通过第一级的 2s 超时探测快速识别"全部不可达"，然后在 ~20s 内给出明确报错和解决方案建议。

4. 性能特征与已知局限

场景耗时结果
国内，至少一台可达~50-200ms第一级直接返回
国内，_TDX_SERVERS 全部不可达~20s（10 台 × 2s）+ bestip 耗时第二/三级兜底
海外，全部不通~20s + 少量兜底尝试RuntimeError 快速失败
已知局限 — 海外服务器需国内 IP：mootdx 走通达信 TCP 7709，海外环境通常全部超时。tdx_client() 只能快速失败并给出报错——它无法解决网络层的物理不可达。海外用户需要走国内代理或 VPN。这也是为什么 a-stock-data 把 mootdx 列为"首选但不排他"——行情数据还可以走腾讯财经（HTTP，海外可用）。

检索练习

问题：你在国内服务器上运行 tdx_client()，发现 _TDX_SERVERS 中前 2 台超时、第 3 台可达。以下哪项描述是正确的？

  1. tdx_client() 会返回第 3 台服务器创建的客户端，耗时约 4s
  2. tdx_client() 会先探测全部 10 台，选延迟最低的那台
  3. 前 2 台超时后会跳过第一级，直接进入 bestip 测速
  4. tdx_client() 会在找到第一台可达服务器后立即返回
答案：D。关键代码 if _probe(ip, port): return Quotes.factory(...) —— 找到第一台可达的就立即返回，不会继续探测剩余服务器（A 错）。这与"探测全部选最优"（B 错）是两种不同策略——这里选的是"最快可用"而非"最优延迟"。前 2 台超时是正常的 fallback 流程，不会跳过第一级（C 错）。第一级中每台服务器超时 2s，所以前 2 台超时 + 第 3 台成功 ≈ 4s 总延迟。
