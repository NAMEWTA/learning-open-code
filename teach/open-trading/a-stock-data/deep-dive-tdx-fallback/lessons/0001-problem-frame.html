mootdx 崩溃：BESTIP.HQ 空串的根因分析

L4 深度剖析 mootdx fallback 第 1/2 节 估计时长：12 分钟

1. 崩溃现场：一个让新手困惑的报错

假设你刚装好 mootdx 0.11.x，在干净的 Python 环境中运行：

from mootdx.quotes import Quotes
client = Quotes.factory(market='std')
# ValueError: not enough values to unpack (expected 2, got 0)

这个报错对新手来说非常不友好——没有任何关键信息提示是网络或配置问题。更迷惑的是：同样的代码在老用户的机器上跑得好好的。这就是 BESTIP 空串 bug 的典型表现。

2. 根因：config.json 的 BESTIP.HQ 是空字符串

mootdx 在首次运行时会在 ~/.mootdx/config.json 中写入默认配置。关键文件内容类似：

{
  "BESTIP": {
    "HQ": "",       // ← 初始值是空字符串 ""
    "ZQ": ""
  }
}

mootdx 内部取这个值时用的是 dict.get("HQ", default)。但 Python 的 dict.get() 只有在 键不存在时才返回 default——键存在但值为空字符串 "" 时，返回的就是 ""。mootdx 拿到空字符串后尝试拆包（例如 ip, port = bestip.split(":")），拆出 0 个值，触发 ValueError。

核心认知：这不是 mootdx 的"严重 bug"——它是首次安装时的默认配置文件未填充真实 IP 导致的。老用户之所以不触发，是因为他们的 config.json 曾在某次成功连接后被写入过真实 IP。这是一个环境差异 bug——在开发者的机器上永远测不出来。

3. 为什么不能锁版本解决

一个直觉方案是：锁定 mootdx 版本到 0.10.12（该版本可能用过不同的 BESTIP 处理逻辑）。但这个方案有一个致命问题：

mootdx 0.10.12 在干净的 Python 3.9 环境下 import 直接崩溃。根因是 mootdx 依赖的 numpy/pandas 版本与 0.10.12 编译时使用的版本存在二进制不兼容。不同 Python 版本、不同操作系统，崩溃的概率不同——这意味着锁版本方案在不同机器上的行为不可预测。

对比两种方案：

方案优点致命缺陷
锁 mootdx==0.10.12简单，一行 pip install部分 Python 版本 import 就崩，跨环境不可靠
tdx_client() 动态探测对 0.10/0.11 都适用，跨环境稳定多 20 行代码（可接受的代价）

a-stock-data 选择了后者——不依赖 mootdx 的内部 BESTIP 机制，而是自己管理一组已知可用的通达信服务器 IP，通过 TCP 探测选择可用节点。

4. 不仅是 BESTIP 问题：海外网络超时

除了 BESTIP 空串，mootdx 还有一个对海外用户致命的场景：TCP 7709 端口在海外网络通常全部超时。通达信服务器部署在国内，海外 VPS 直连大概率不通。tdx_client() 的设计在这一点上也优于裸 factory：它会在探测阶段快速识别"全部不可达"并抛出明确的 RuntimeError，而非让用户等到 TCP 超时（默认可能数分钟）。

检索练习

问题：以下哪个场景不是 BESTIP 空串 bug 的触发条件？

  1. 全新安装 mootdx 0.11.x，从未运行过，config.json 中 BESTIP.HQ 为 ""
  2. 已使用 mootdx 半年，某天 config.json 文件被手动清空
  3. 在海外 VPS 上运行 tdx_client()，所有 TCP 探测超时
  4. 卸载重装 mootdx 后直接 Quotes.factory(market='std')
答案：C。海外 TCP 超时不是 BESTIP 空串 bug——它是网络层面的问题。tdx_client() 探测到全部超时后抛出的是 RuntimeError（"所有 mootdx 服务器均不可达"），而非 ValueError。A、B、D 都会使 config.json 中缺少可用 IP，触发 BESTIP 空串 → ValueError 路径。关键区别：BESTIP bug 是配置层问题，海外超时是网络层问题。
