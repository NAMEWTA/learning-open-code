新标的调研 — 7步流水线与数据流

L2 垂直切片 第 1 节 / 共 2 节 估计时长：12 分钟

1. 调研流水线概览

拿到一个陌生代码（如 688017），你需要快速回答：机构怎么看？估值合理吗？属于哪些概念？资金在进还是出？主力上榜了吗？近期有无解禁？两融风险大不大？这7个问题对应 a-stock-data 中 跨4层架构的7个端点，按依赖关系组成一条串行流水线。

步序问题函数所属L1模块数据源
1机构覆盖？ths_eps_forecast()module-research同花顺（低频安全）
2估值贵不贵？tencent_quote()module-market-data腾讯（不封IP）
3属于哪些概念？eastmoney_concept_blocks()module-signals东财 slist
4资金流入流出？eastmoney_fund_flow_minute()module-signals东财 push2
5龙虎榜上榜？dragon_tiger_board()module-signals东财 datacenter
6解禁风险？lockup_expiry()module-signals东财 datacenter
7两融情况？margin_trading()module-capital-flow东财 datacenter
数据源优先级：步骤1走同花顺（低频无风险），步骤2走腾讯（不封IP），步骤3~7全部走东财。东财端点必须经 em_get() 串行限流——6个步骤的理论最低耗时约6秒（每步 ≥1s 间隔）。

以下展示7步的串行依赖与并行优化机会。步骤2（估值）与步骤3（概念板块）之间无数据依赖，可并行拉取。

实际代码中步骤2和3可并发执行（腾讯和东财 slist 是不同数据源、无共享状态），步骤4~6虽然都在信号层，但共享东财会话和限流锁，必须串行。步骤7走 datacenter-web 同一 base URL 但独立端点。

3. 跨层数据流

7步调研的数据如何流转？每个步骤的输出字段是下一步的输入依据——但这是人工判断层面的流转，不是代码级的数据管道。

输入: stock_code = "688017" │ ├─① ths_eps_forecast(code) │ → year, avg_eps, inst_count（机构数） │ → 输出：每年预测EPS + 覆盖机构数 │ ├─② tencent_quote([code]) │ → pe_ttm, pb, mcap_yi（市值亿） │ → 输出：当前PE/PB/市值，结合①算PEG │ ├─③ eastmoney_concept_blocks(code) │ → concept_tags: [BK0001,...] │ → 输出：归属板块列表 + 各板块涨跌幅 │ ├─④ eastmoney_fund_flow_minute(code) │ → main_net（主力净流入，万元） │ → 输出：当日累积分时资金流向 │ ├─⑤ dragon_tiger_board(code, date) │ → records[]（上榜记录 + 买卖席位） │ → 输出：上榜次数 + 机构/游资动向 │ ├─⑥ lockup_expiry(code, date) │ → upcoming[]（未来解禁批次） │ → 输出：解禁日期 + 解禁股数 + 占总股本比 │ └─⑦ margin_trading(code) → rzye（融资余额）, rqye（融券余额） → 输出：两融余额趋势 + 多空比

检索练习

不看屏幕，尝试回答（每题30秒）：

  1. 7步流水线中，哪两步可以并行执行？为什么？
  2. 6个东财端点共享什么限流机制？理论最短耗时是多少？
  3. 步骤4~6同属信号层但为什么必须串行？
  4. ths_eps_forecast 为什么不在东财限流范围内？
参考答案：
1. 步骤2（tencent_quote）和步骤3（eastmoney_concept_blocks）可并行——腾讯和东财是不同数据源、无共享状态、无数据依赖。
2. 共享 em_get() 串行限流（最小1s间隔 + 随机抖动），6步理论最低耗时约6秒。
3. 虽然同属信号层，但走东财共享 HTTP 会话和限流锁，并发会导致 IP 封禁。
4. ths_eps_forecast 走同花顺 basic.10jqka.com.cn，非东财域名，不在 em_get 限流体系内。
