1. 期权链分析流水线概览

期权交易的核心决策永远围绕一个问题：当前这个月份、这个行权价的合约，是否值得交易？回答这个问题需要拉通 5 步数据流水线——每一步都是上一步的自然延续，共同构成从标的选取到风险判断的完整分析链路。

步序操作核心函数输出数据源
1选标的用户输入 underlying"510050" / "510300" / "588000" / "510500"—
2获取合约清单sina_option_codes(){月份: [合约代码,...]}新浪 StockOptionService
3选月份list(codes)[0]近月/次近月 YYMM—（源自步骤2输出）
4T型报价sina_option_tquote()买卖五档、持仓量、行权价、涨跌幅新浪 hq.sinajs.cn (CON_OP_)
5希腊字母+IVsina_option_greeks()Delta/Gamma/Theta/Vega/IV/理论价新浪 hq.sinajs.cn (CON_SO_)
核心设计原则：步骤 2~3 是准备阶段（确定分析的目标合约集合），步骤 4~5 是分析阶段（获取每个合约的报价与风险指标）。步骤 4 和 5 对每个合约独立调用，可以逐个合约处理，也可以批量拉取后对比分析。

2. Mermaid 流程时序图

以下展示 5 步的串行依赖关系。步骤 2 获取月份列表和合约清单，步骤 3 从中选取目标月份，步骤 4~5 对目标月份内的所有合约逐一遍历报价和希腊字母。

sequenceDiagram participant U as 用户选择标的 participant C as 合约清单(Step2) participant M as 月份筛选(Step3) participant T as T型报价(Step4) participant G as 希腊字母(Step5) U->>C: sina_option_codes("510050", call=True) Note over C: ①GET StockOptionService.getStockName Note over C: ②逐月GET OP_UP_510050{YYMM} C-->>M: {"06":[code...], "07":[...], "09":[...], "12":[...]} M->>M: 选近月 = list(codes)[0] Note over M: 近月=06, 共 N 个合约 M-->>T: codes["06"] = [c1,c2,...,cN] loop 近月每个合约 T->>T: sina_option_tquote(ci) Note over T: GET CON_OP_{ci} → 17字段 T-->>G: tquote dict G->>G: sina_option_greeks(ci) Note over G: GET CON_SO_{ci} → 13字段 G-->>G: greeks dict end Note over G: 合并 → 完整期权链 [{**tq, **gk},...]

注意步骤 4 和 5 虽然对每个合约是串行的（先报价再希腊字母），但不同合约之间没有数据依赖。如果网络延迟是瓶颈，可以用 ThreadPoolExecutor 并行拉取多个合约的 tquote + greeks，只需注意不超过新浪的并发上限（建议 ≤5 并发）。

新浪源并发限制：新浪 hq.sinajs.cn 对高频请求会返回空数据甚至暂时拒绝连接。批量拉取期权链时，建议每个请求间隔 ≥200ms，或先串行拉取近月核心合约（平值附近 ±5 档），再逐步扩展。

3. 逐层数据流转

5 步流水线中，数据如何在各步骤间流转？关键是理解每步的输出如何成为下一步的输入：

输入: underlying = "510050", direction = call 步骤1: 选标的 └─ underlying → 映射为新浪分类名 "50ETF" 步骤2: sina_option_codes("510050", call=True) ├─ GET StockOptionService.getStockName?cate=50ETF │ └─ 返回 months = ["2026-06", "2026-07", "2026-09", "2026-12"] ├─ months 转 YYMM: ["06","07","09","12"] └─ 逐月 GET OP_UP_510050{YYMM} └─ 返回 {"06":["10007923",...], "07":[...], ...} 第一个 key "06" = 近月（最活跃） 步骤3: 选月份 near = list(codes)[0] → "06" target_codes = codes["06"] → ["10007923","10007924",...,"10007946"] atm_idx = len(target_codes)//2 → 中间合约 ≈ 平值附近 步骤4: T型报价 (对每个 code) sina_option_tquote("10007923") └─ GET hq.sinajs.cn/list=CON_OP_10007923 返回: {bid, bid_vol, last, ask, ask_vol, open_interest, strike, pct, name, volume, amount, prev_close, ...} 步骤5: 希腊字母 (对每个 code) sina_option_greeks("10007923") └─ GET hq.sinajs.cn/list=CON_SO_10007923 返回: {delta, gamma, theta, vega, iv, strike, last, theory, name, ...} 最终输出: chain = [{**tquote, **greeks}, ...] 按 strike 升序排列 → 完整的近月期权链

关键点：步骤 2 中 sina_option_codes() 返回的字典 key 已经是 YYMM 格式，第一个 key 即近月——这是交易所的标准排序，不需要额外排序。近月合约流动性最好、持仓量最大，是期权分析的首选目标。

检索练习

不看屏幕，尝试回答（每题 30 秒）：

  1. sina_option_codes() 返回的字典第一个 key 是什么？为什么它是近月？
  2. 步骤4和步骤5调用的新浪端点前缀分别是什么？为什么需要不同前缀？
  3. 如果目标月份有 24 个合约，步骤4~5需要发多少次 HTTP 请求？
  4. 为什么不建议对全部 24 个合约开 24 线程并发请求新浪？
参考答案：
1. 第一个 key 是 YYMM 格式（如 "06"），它是上市交易所按到期时间排序后的最近月份，流动性最大、持仓最集中。
2. T型报价用 CON_OP_ 前缀，希腊字母用 CON_SO_ 前缀——新浪用不同前缀区分不同的数据通道，底层返回的字段结构完全不同。
3. 24 个合约 × 2 次请求（tquote + greeks）= 48 次 HTTP 请求。每个合约需要先拉报价再拉希腊字母。
4. 新浪 hq.sinajs.cn 对高并发敏感——24 线程同时请求可能触发临时封禁。建议 ≤5 并发 + 200ms 间隔，或先串行拉核心合约。
