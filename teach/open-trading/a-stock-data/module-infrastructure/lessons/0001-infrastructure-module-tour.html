数据源优先级 — 基础设施（上）

L1 基础设施 第 1 节 / 共 2 节 估计时长：14 分钟

1. 一个问题：所有数据走东财不行吗？

初看 a-stock-data 的 40 个端点，东财占了约 14 个——龙虎榜、解禁、两融、资金流、研报、新闻，看起来覆盖面最广。直觉反应是"统一用东财，代码最简单"。但这个直觉在实战中会直接导致 IP 被封。

a-stock-data 的设计者因此制定了数据源优先级体系——这是整个项目最先读取的章节（SKILL.md:104-150），也是所有 10 层数据层选源时必须遵守的规则。它的核心思想只有一句话：能用不封 IP 的数据源，就别用东财。

设计哲学落地："数据源优先级分明"是 L0 总览中 5 大设计哲学的第 2 条。本节将其展开为可执行的选源决策表。

2. 13 个数据源的封 IP 风险分级

a-stock-data 将 13 个数据源分为 4 个优先级层级，分级依据是「封 IP 风险」而非数据覆盖率：

优先级数据源协议封 IP覆盖数据
1（首选） mootdx（通达信） TCP 7709 不封 K线、五档盘口、逐笔成交、财务快照、F10
2（首选） 腾讯财经 HTTP 不封 实时价、PE/PB/市值/换手率、指数/ETF
3 新浪 / 巨潮 / 同花顺 HTTP 低 财报三表、公告全文、互动易、一致预期
3 百度股市通 / iwencai HTTP 极低 K线(MA均线)、NL语义搜索
4（仅独有） 东财 eastmoney HTTP 会封 IP 龙虎榜、解禁、两融、大宗、资金流、研报等

关键洞察：TCP 协议不封 IP。通达信走的是二进制 TCP 7709 端口，不存在 HTTP 层反爬机制。腾讯财经虽是 HTTP，但实测高频调用也不会触发封禁——它面向的是普通股民页面访问，天然高并发。这两个数据源覆盖了行情、盘口、估值、指数等最常用场景，应当作为所有代码的首选通道。

常见错误：用东财 pull2 接口拉实时行情。每只股票一次请求，100 只就是 100 次/分钟——超过风控阈值，5 分钟内必封。

3. 东财独有数据 —— 不得不走东财的场景

以下 6 类数据只有东财提供，通达信/腾讯/新浪都没有。此时必须走东财，但只要遵守防封铁律，仍然安全：

数据类别端点示例所属层级
龙虎榜个股席位 + 全市场日榜信号层 L3
资金流向分钟级主力/散户/超大单信号层 L3
融资融券 + 大宗交易两融余额、大宗成交明细资金/筹码层 L4
股东户数 + 分红送转筹码集中度、历史分红资金/筹码层 L4
限售解禁日历历史 + 未来待解禁信号层 L3
研报 + 个股新闻reportapi / search研报层 L2 + 新闻层 L5

风控阈值实测数据（社区 2026-05）： 每秒 >5 次请求 / 单 IP 并发 ≥10 / 1 分钟 ≥200 次 / 5 分钟 ≥300 次触发封禁。被封表现：连续 403 / 429 / 连接超时。

防封铁律：① 串行不并发 ② 每次间隔 ≥1s + 随机抖动（QPS ≤2）③ 复用 HTTP 会话（Keep-Alive）④ 正常 UA + Referer ⑤ 批量场景每只股票之间 sleep。

4. 基础设施 —— 所有 10 层数据层的底座

em_get()、tdx_client()、get_prefix() 这三个核心函数构成了 a-stock-data 的基础设施层。它们不直接提供数据，但被所有 10 个数据层调用——没有它们，每一层都要自己处理限流、连接容错、代码格式校验。

SKILL.md 模块依赖全景 │ ├── 基础设施层（本文主题） │ ├── em_get() ← 东财统一入口（被 L2-L10 调用） │ ├── tdx_client() ← mootdx 连接（被 L1、L6 调用） │ ├── get_prefix() ← 代码前缀映射（被所有层调用） │ └── eastmoney_datacenter()← 数据中心查询（被 L3、L4 调用） │ ├── L1 行情层 ───────────── 依赖 tdx_client + get_prefix ├── L2 研报层 ───────────── 依赖 em_get ├── L3 信号层 ───────────── 依赖 em_get + eastmoney_datacenter + get_prefix ├── L4 资金/筹码层 ──────── 依赖 em_get + eastmoney_datacenter ├── L5 新闻层 ───────────── 依赖 em_get ├── L6 基础数据层 ───────── 依赖 tdx_client + em_get + get_prefix ├── L7 公告层 ───────────── 依赖 get_prefix（巨潮） ├── L8 打板层 ───────────── 依赖 em_get ├── L9 ETF期权层 ────────── 依赖 get_prefix └── L10 舆情互动层 ──────── 依赖 em_get + get_prefix

下节预告：第 2 节将深入 em_get() 的限流代码、tdx_client() 的三级 fallback 机制，以及 get_prefix() 的归一化规则。这些是基础设施层"怎么做"的核心代码。

检索练习

不看屏幕，尝试回答（每题 30 秒）：

  1. 为什么 mootdx 和腾讯财经被封 IP 的风险最低？技术原因是什么？
  2. 列出 3 个只有东财才提供的数据类别。
  3. 如果你需要批量获取 100 只股票的 PE/PB 数据，应该选哪个数据源？为什么？
  4. 东财的风控阈值中，5 分钟内超过多少次请求会触发封禁？
参考答案：
1. mootdx 走 TCP 7709 二进制协议，不存在 HTTP 层反爬机制；腾讯财经面向股民页面访问，天然高并发不封 IP。
2. 龙虎榜（席位+全市场日榜）、资金流向（分钟级主力/散户）、融资融券+大宗交易（两融余额+大宗明细）——均由东财 datacenter 独家提供。
3. 用腾讯财经（tencent_quote），一次请求即可拿到 PE/PB/市值/换手率。走东财不仅慢（限流 1s/次），还会无谓累积风控计数。
4. ≥300 次 / 5 分钟。
