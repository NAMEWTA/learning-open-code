核心机制：串行锁 + 随机抖动 + 会话复用 + 连接重试

L4 深度剖析 东财防封限流 第 2/2 节 估计时长：15 分钟

1. 设计方案对比：为什么是串行限流

在设计 em_get() 之初，有三种常见的限流方案可选：

方案思路为什么不用
令牌桶 恒定速率放行请求，允许短时突发 实现复杂（需要后台线程续令牌）；突发恰是东财敏感的——批量场景下容易触发风控
并发池 + 信号量 N 个并发槽位同时请求 并发 ≥10 是东财的高风险行为；并发请求的请求间间隔不可控，容易踩中"1 分钟 200 次"阈值
串行限流（选中） 全局时间戳 + sleep + 抖动 代码不到 20 行、零外部依赖、行为完全可预测、对风控最友好
设计哲学：面对不确定的风控规则，最简单的方案往往最可靠。串行限流不会给你"最优吞吐量"，但能给你"最强稳定性"——而这正是量化脚本长期无人值守运行的核心需求。

2. 逐步拆解 em_get() 的四层防护

em_get() 的限流机制由四层组成，每层解决一个问题。以下逐一拆解（完整源码见 a-stock-data SKILL.md:290-354）：

第一层：串行锁 — 全局时间戳 + time.sleep

机制：_em_last_call 是模块级变量（列表包装以支持闭包内修改），记录上次请求结束的时间戳。每次 em_get() 调用前计算 wait = EM_MIN_INTERVAL - (now - _em_last_call)，若 wait > 0 则 sleep。请求结束后在 finally 块中更新时间戳。 
EM_MIN_INTERVAL = 1.0          # 两次东财请求最小间隔(秒)
_em_last_call = [0.0]          # 模块级上次请求时间戳

def em_get(url, ...):
    wait = EM_MIN_INTERVAL - (time.time() - _em_last_call[0])
    if wait > 0:
        time.sleep(wait + random.uniform(0.1, 0.5))
    try:
        return EM_SESSION.get(url, ...)
    finally:
        _em_last_call[0] = time.time()

为什么用列表包装而非 global？在模块作用域中用列表 [0.0] 包装，可以避免在函数内部写 global _em_last_call，代码更简洁且不易因未来重构引入作用域 bug。

为什么 EM_MIN_INTERVAL 默认 1.0s？东财风控对"每秒 > 5 次"是高危，设 1s 间隔意味着 QPS ≤ 1，留足安全余量。批量筛选场景建议调大到 1.5~2s，牺牲一点点速度换稳定。

第二层：随机抖动 — random.uniform(0.1, 0.5)

机制：在固定间隔之上叠加 100~500ms 的随机延迟。这打破了请求时间戳的规律性——东财风控可能检测"精确每 1.0s 一个请求"这种机械模式。
为什么不是更大范围（如 0.1~2.0s）？抖动过大意味着方差高——偶尔两个请求间隔 0.3s（1.0s 等待中抖动碰巧很小），偶尔间隔 2.5s。0.1~0.5s 的窄范围保证：实际间隔始终在 1.1~1.5s 之间，既破坏规律性又不引入不可预测的超长等待。

第三层：会话复用 — requests.Session + Keep-Alive

机制：EM_SESSION = requests.Session() 在所有 em_get() 调用间共享同一个 TCP 连接池。相比每次 requests.get() 新建连接，Session 复用省去了三次握手 + TLS 协商的开销，且"复用连接"更像正常浏览器的行为模式。

第三方请求不会走 EM_SESSION——只有通过 em_get() 发出的东财请求才复用会话。这是一个隐式的"正常用户行为模拟"：同一个浏览器标签页在短时间内访问同一网站，天然复用 Keep-Alive 连接。

第四层：连接重试 — HTTPAdapter + Retry（指数退避）

from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

_em_adapter = HTTPAdapter(max_retries=Retry(
    total=3, connect=3, backoff_factor=0.6,
    status_forcelist=[429, 500, 502, 503, 504],
    allowed_methods=["GET"]))
EM_SESSION.mount("https://", _em_adapter)
EM_SESSION.mount("http://", _em_adapter)

关键设计细节：

  • 429 重试、403 不重试。 429（Too Many Requests）是临时限流，间隔一会可恢复；403（Forbidden）是风控信号，重试只会加重问题。
  • 5xx 重试。 500/502/503/504 是服务器端临时故障，指数退避重试合理。
  • backoff_factor=0.6。 第一次重试等 0.6s，第二次等 1.2s，第三次等 2.4s。3 次重试最多增加约 4.2s 延迟。
  • 老版本兼容。 try/except 包裹 mount——老版本 urllib3 可能缺少 allowed_methods 参数，降级为无重试但不影响主流程。

3. 性能特征与已知局限

场景吞吐量建议
默认（EM_MIN_INTERVAL=1.0s）~60 req/min日常查询足够
批量筛选（调大到 2.0s）~30 req/min100 只股票约 3.3 分钟
已知局限 — 大陆住宅 IP 间歇风控：即便严格按 em_get() 限流，部分大陆住宅宽带 IP 仍可能遇到 HTTP 000。这不是代码 bug——是运营商层面的连接级封锁。遇到时只能：隔几分钟重试、换网络环境、或调大 EM_MIN_INTERVAL 到 2~3s。

检索练习

问题：你在运行一个需要拉 200 只股票东财数据的脚本，为了加快速度，你把 EM_MIN_INTERVAL 从 1.0 改成了 0.2。运行到第 50 只时开始返回 403。以下哪项是正确的应对措施？

  1. 把 EM_MIN_INTERVAL 改回 1.0 或更大，重启脚本从头跑
  2. 在 403 之后立即重试 10 次直到成功
  3. 改用并发多线程，每线程 0.2s 间隔，分散到不同 IP
  4. 去掉 random.uniform 抖动，减少不必要的延迟
答案：A。0.2s 间隔 = QPS 5，恰好踩中东财"每秒 >5 次 = 高危"的阈值。403 是风控信号，立即重试只会加重问题（B 错，且 Retry 配置已排除 403 不重试）。并发多线程会提高并发连接数（C 错，触发"并发 ≥10 = 高危"）。去掉抖动让请求更机械规整（D 错，更容易被风控检测）。正确做法：恢复安全间隔、等待封禁期过去后重试。
