东财风控：为什么需要限流

L4 深度剖析 东财防封限流 第 1/2 节 估计时长：12 分钟

1. 东财不是"公共 API"

东方财富的数据接口——龙虎榜、研报、资金流、新闻——本质上不是公开 API，而是其网站前端的内部数据通道。东财没有发布过任何 API 文档或使用条款，所有接口都是社区通过浏览器抓包逆向出来的。

这意味着东财可以随时单方面限流或封禁请求来源，而且没有任何申诉渠道。你的脚本对东财服务器来说，跟普通用户的浏览器一模一样——区别只在于请求频率。

核心认知：东财风控不是"bug"——它是东财正常运维策略的一部分。限流代码不是"可有可无的优化"，而是保证脚本能长期运行的必要条件。

2. 风控阈值：什么时候会被封

以下阈值来自社区实测（2026-05），非官方公布但交叉验证可靠：

行为触发阈值风险等级
每秒请求数> 5 次/秒高
单 IP 并发连接≥ 10高
1 分钟请求总数≥ 200 次中高
5 分钟请求总数≥ 300 次触发封禁
User-Agent空 UA / 无浏览器特征中

被封的典型表现：连续请求后返回 403 / 429 / 连接超时 / 返回空数据。临时封禁通常持续几分钟到几小时。

3. 两个真实的触发现象

a-stock-data SKILL.md 中记录了两个风控触发的具体表现，来感受一下：

现象一 — HTTP 000（大陆住宅 IP 间歇封锁）：push2/push2his 系列对部分大陆住宅宽带 IP 有连接级风控。表现为偶发 HTTP 000（连接被拒/超时）或返回空。这不是代码问题——同一代码在其他网络/时段实测正常。遇到时只能隔几分钟重试或换网络。
现象二 — 个股新闻接口间歇返回空：search-api-web 接口对部分 IP 只返回 passportWeb（股民资料）而无 cmsArticleWebOld（文章列表）。同样是东财对该 IP 的间歇风控，非代码 bug。

4. 批量筛选：被封的头号元凶

AI 批量循环是最常见的封禁场景。假设你要筛选 100 只股票，对每只拉龙虎榜 + 资金流——如果不做任何限流：

for code in stock_list:   # 100 只股票
    dragon_tiger_board(code)   # 2 次 API 调用
    stock_fund_flow_120d(code) # 1 次 API 调用
# 总计 300 次请求，5 分钟内打完 → 直接触发 5 分钟 ≥300 次封禁阈值

这就是为什么 a-stock-data 把"批量场景每只股票之间 sleep"列为防封铁律第一条——AI 跑批量循环是被封的头号元凶。

关键领悟：风控不是针对"恶意爬虫"的，而是针对"不像正常人类浏览行为"的流量模式。正常人不会在 5 秒内打开 100 个股票详情页——所以你的脚本也不应该。

检索练习

问题：以下哪种情况最可能触发东财封 IP（而非网络问题）？

  1. 偶尔一次返回 ConnectionError，重试后恢复正常
  2. 跑 100 只股票的批量筛选脚本，前 30 只正常，之后全部返回 403
  3. 服务器在北京，但代码运行在香港 VPS，返回超时
  4. 使用 curl 不带 User-Agent 请求一次，返回空数据
答案：B。前 30 只正常、之后全部 403 是典型的"达到风控阈值后被临时封禁"的模式。A 更可能是偶发网络波动（重试恢复说明不是封禁）。C 可能只是海外网络不通。D 空 UA 可能只是触发降级返回而非封禁。关键判断依据：正常→突然全部异常→不再恢复，是风控信号；偶发→重试恢复，是网络问题。
