公告层模块导览

L1 模块导览 公告层 估计时长：13 分钟

1. 公告层定位：沪深北全量公告的双源架构

公告层是 10 层数据架构中覆盖最全、时效性最强的文本数据层。它负责获取上市公司向交易所提交的正式文件——年报、季报、重大资产重组报告、减持公告、董事会决议等。a-stock-data 采用双源互补策略：

公告层双源分工 │ ├── 巨潮 cninfo (HTTP POST) → 沪深北全量公告全文检索 + 翻页 + PDF详情链接 └── mootdx F10 (TCP 7709) → 最新公告文本摘要，不封 IP，轻量快速

核心原则：巨潮 cninfo 是证监会指定的信披平台，覆盖沪深北三市全部历史公告，是主力数据源。mootdx F10 走 TCP 不封 IP，提供最近公告文本摘要，作为轻量补充。

选源口诀：查全文翻历史找公告走巨潮 cninfo，快速扫一眼最新摘要走 mootdx F10。

2. 巨潮 cninfo：公告全文检索的核心引擎

巨潮公告检索由三个函数协作。其中 _cninfo_orgid() 是本节的精华——背后有一个 V3.2.2 修复故事。

_cninfo_orgid() — 最关键的一个坑（V3.2.2 修复）

巨潮 API 中每只股票用 orgId 而非股票代码来标识。早期版本的 a-stock-data 使用硬编码规则：6 开头拼 gssh0{code}、0/3 开头拼 gssz0{code}。这个规则只对部分老股票成立。

真实案例：平安银行 000001 的 orgId 是 gssz0000001，工商银行 601398 的 orgId 是 jjxt0000019，绿的谐波 688017 的 orgId 是 9900041602。硬编码 gssh0601398 查不到工商银行任何公告，返回 totalAnnouncement=0。

修复方案：V3.2.2 引入 _cninfo_orgid()，首次调用时从巨潮官方映射表 szse_stock.json（含 6198 只股票）动态加载 orgId 到模块级缓存。查不到再回退硬编码规则作为 fallback。这意味着：

股票代码硬编码结果（旧）动态查表结果（新）旧版能否查到公告？
600519（茅台）gssh0600519gssh0600519（碰巧一致）能
601318（平安）gssh06013189900002221不能（查不到）
601398（工行）gssh0601398jjxt0000019不能（查不到）
688017（绿的谐波）gssh06880179900041602不能（查不到）

核心教训：交易所 ID 体系不是简单编码——不同时期/板块的股票 orgId 各异。宁可多一次 HTTP 请求，也别信硬编码。

cninfo_announcements() — 公告检索与翻页

主函数通过 POST 请求巨潮的 hisAnnouncement/query 端点，携带 orgId、pageSize、pageNum 等参数。返回的每条公告包含：

字段含义来源/转换
title公告标题announcementTitle
type公告类型announcementTypeName（如"年报""季报""临时公告"）
date发布日期由 _cninfo_ts_to_date() 将毫秒时间戳转为 YYYY-MM-DD
url详情页链接拼接 disclosure/detail?annoId=...
# 典型用法：获取公告列表 + 翻页
anns = cninfo_announcements("601318", page_size=30)  # 平安最近30条公告
for a in anns:
    print(f"{a['date']} | {a['type']} | {a['title']}")

3. mootdx F10 公告摘要：不封 IP 的轻量补充

mootdx 的 F10 模块通过通达信 TCP 协议提供 9 大类公司资料文本，其中 name='最新提示' 返回最近几条公告摘要（含分红预案、股东大会通知等）。优点：不封 IP、无需 HTTP 请求、速度极快。缺点：只有摘要没有全文、无法翻页、无历史检索能力。 

from mootdx.quotes import Quotes
# 使用统一的 tdx_client() helper（规避 0.11.x BESTIP bug）
client = tdx_client()
text = client.F10(symbol='688017', name='最新提示')
# 返回纯文本：包含最近公告摘要、分红方案、股东大会决议等
print(text[:500])  # 打印前500字

F10 还支持 9 类资料 name 参数，仅 '最新提示' 属公告层。走 TCP 不经 HTTP，无需 em_get() 限流。

4. 公告层协作：与新闻层组成完整舆情覆盖

公告是上市公司的正式信息披露，新闻是媒体的解读与传播。两者组合使用时：

数据层内容性质时效典型用途
公告层（巨潮+F10）正式文件，有法律效力盘后批量发布基本面变化确认、财务数据核查、减持预警
新闻层（东财个股+全球资讯）媒体解读，有主观判断实时推送市场情绪感知、题材发酵跟踪、事件传播链分析

实战模式：先用巨潮查某股票的"减持公告"确认股东减持计划（正式文件），再用新闻层搜索相关解读看看市场怎么反应（情绪面）。公告提供"发生了什么"，新闻提供"大家怎么看"——两者缺一不可。

相关模块：新闻层（module-news）——东财个股新闻 + 7x24 快讯，与公告层互补。
