舆情互动层模块导览

L1 模块导览 舆情层 V3.3.0 新增 估计时长：14 分钟

1. 舆情层定位：AI 问答的独家信源

舆情互动层是 10 层数据架构中 V3.3.0 新增的三层之一，承载投资者互动问答 + 市场热度两大舆情维度。与行情（价格）、研报（机构观点）、新闻（媒体信息）不同，舆情层提供的是来自市场参与者的真实交互数据——这是 AI 投研 Agent 回答"公司如何回应某传闻""当下哪些票最热"等问题的独家信源，别处拿不到。

舆情互动层三源分工 │ ├── 巨潮互动易 (cninfo_irm) → 投资者提问 + 公司官方回复（独家信源） ├── 同花顺热榜 (ths_hot_list) → 热度排名 + 概念标签 + 排名变化 └── 东财 (em_hot_rank / em_hot_concept) ├── 人气榜 → 排名 + 排名变化 + 名称价格 └── 个股概念命中 → 该票被哪些概念覆盖 + 热度值
核心价值：互动易是唯一能回答"公司怎么回应某传闻/利好"的数据源——财报、研报、新闻都无法提供这种公司官方口径。同花顺+东财双热度则构成了"当下市场在炒什么"的交叉验证体系。

2. cninfo_irm()：互动易问答

巨潮互动易是深沪两市官方的投资者互动平台，cninfo_irm() 通过两步请求获取指定股票的提问与回复列表。

两步请求链路

步骤端点作用关键参数
第 1 步 queryKeyboardInfo 用 6 位代码查 orgId keyWord=code (body)
第 2 步 company/question 用 orgId 拉取问答列表 orgId/stockCode/pageSize/pageNum (query string)

关键坑：第二步参数必须放 query string（不是 body），否则 HTTP 400。另外 orgId 取自第一步返回的 secid，即便前缀是 gshk（港股），靠 stockcode 过滤照样拿 A 股问答。

返回字段与翻页

每条记录包含：code（股票代码）、company（公司简称）、question（投资者提问原文）、answer（公司回复，未回复则为 None）、answerer（回复方，如"公司董事会"）、ask_time（提问时间，由毫秒时间戳转换）。 

# ── 示例：看比亚迪(002594)的公司回应 ──
for q in cninfo_irm("002594", page_size=30):
    if q["answer"]:
        print(f"Q: {q['question'][:30]}")
        print(f"A[{q['answerer']}]: {q['answer'][:50]}")

翻页控制：page_size（每页条数，默认 30）、page_num（页码，从 1 开始）。回复率因公司而异——立讯精密(002475)回复率高，京东方(000725)几乎不回复，这是数据源本身的特征，不是 bug。

3. ths_hot_list() 与 em_hot_rank()：双源热度对比

同花顺热榜和东财人气榜从不同角度反映市场关注度。同花顺偏重概念归因（每只票标注了哪些概念标签），东财偏重排名变化（直接告诉你排名升了还是降了）。两个源交叉验证，可以排除单一平台的热度噪音。

同花顺热榜 ths_hot_list()

参数取值字段说明
period"hour" / "day"rank当前排名
heat人气值（数值越大越热）
rank_chg排名变化（正=上升，负=下降）
concepts概念标签列表（如 ["AI","芯片"]）
tag人气标签（如"人气飙升"）

东财人气榜 em_hot_rank()

参数取值字段说明
top默认 50rank当前排名
rank_chg排名变化（正=上升）
name/price/pct名称+价格+涨跌幅（经 ulist.np 补全）

关键坑：东财人气榜 getAllCurrentList 只返回带前缀代码（SZ/SH），名称和价格要再走 push2 ulist.np 接口补全。前缀转换规则：SZ → 0.、SH → 1.。另外 ulist.np 的 diff 字段偶尔是 dict（按序号为键），已做 list(values()) 归一化。 

# ── 双源热度交叉验证 ──
ths = {s["code"]: s for s in ths_hot_list()[:30]}
em  = {s["code"]: s for s in em_hot_rank(30)}
both = set(ths) & set(em)         # 双源共振的票
ths_only = set(ths) - set(em)     # 仅同花顺热
print(f"共振: {len(both)}, 仅同花顺: {len(ths_only)}, 仅东财: {len(set(em)-set(ths))}")

4. em_hot_concept()：个股概念命中

em_hot_concept() 回答了一个关键问题：这只票当下被市场归到哪些概念板块在炒？与 eastmoney_concept_blocks()（拿板块列表）不同，这个端点从一个股票出发，返回它所命中的热门概念及对应热度值。两者互补——一个返回全量静态归属（行业+概念+地域），一个返回热门概念命中（带热度值）。

端点方向返回
eastmoney_concept_blocks()个股 → 所属板块板块名 + BK码 + 涨跌幅 + 龙头股（行业/概念/地域混合）
em_hot_concept()个股 → 命中概念列表概念名 + conceptId + 命中热度(hitCount)
# ── 示例：查人气第一的票被哪些概念覆盖 ──
hot = em_hot_rank(10)
top_code = hot[0]["code"]
print(f"{hot[0]['name']} 概念命中:", em_hot_concept(top_code)[:3])

参数只需 6 位 code，内部自动补前缀（6 开头 → SH，其他 → SZ）。返回按 hit（命中热度）降序排列，越靠前的概念越核心。

检索练习

不查文档，尝试回答（每题 30 秒）：

  1. cninfo_irm() 为什么需要两步请求？orgId 从哪里获取？
  2. cninfo_irm() 第二步参数放在 query string 还是 body？放错了会怎样？
  3. em_hot_rank() 返回的代码带 SZ/SH 前缀，怎么补全名称和价格？前缀转换规则是什么？
  4. em_hot_concept() 与 eastmoney_concept_blocks() 的互补关系是什么？
  5. ths_hot_list() 的 concepts 字段和 em_hot_concept() 的 concept 字段有什么不同？
参考答案：
1. 第一步用 6 位代码查 orgId（来自 queryKeyboardInfo 的 secid），第二步用 orgId 拉问答列表。orgId 是巨潮内部的机构标识，代码不能直接用于第二步。
2. 放在 query string。放 body 会导致 HTTP 400。
3. 走 push2 ulist.np 接口补全。前缀转换：SZ → 0.、SH → 1.。ulist.np 的 diff 偶尔是 dict 需做 list(values()) 归一化。
4. em_hot_concept() 从个股出发查热门概念命中（hitCount热度值），eastmoney_concept_blocks() 同样从个股出发、返回全量板块归属（行业+概念+地域+BK码+龙头股）。前者反映市场热度、后者反映静态分类，两者组合可得板块-概念完整画像。
5. ths_hot_list() 的 concepts 是同花顺标注的概念标签（伴随热榜条目返回），em_hot_concept() 的 concept 是东财热门概念命中（带热度值 hitCount）。数据源不同，标签体系不同，交叉验证可排除单源噪音。
