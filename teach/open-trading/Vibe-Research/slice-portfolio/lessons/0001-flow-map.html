2. 三条主线：录入、盈亏计算、清仓

整个持仓链路围绕 一份本地 JSON 文件 展开，所有修改都经过 threading.Lock() 保护：

主线入口核心逻辑持久化
录入持仓 POST /api/portfolio/holding add_holding()：同代码按加权平均成本合并（加仓），新代码直接追加 _save() 先写 .tmp 再 os.replace() 原子改名，防止并发读到半截 JSON
实时盈亏 GET /api/portfolio 或 POST /api/portfolio/refresh get_portfolio()：批量调 astock.tencent_quote()，逐笔算市值和浮动盈亏，汇总到 totals 只读不写；updated 字段记录北京时间戳
已清仓记录 POST /api/portfolio/close close_position()：算已实现盈亏 = (清仓价 - 成本) / 成本，存入 closed 数组 追加写入；realized_pnl 由 get_portfolio() 动态汇总

前端 Portfolio.tsx 通过 api.ts 中统一的 request() 函数调用这些端点——所有请求经 Vite 代理转发到 FastAPI :8900。每 30 分钟 setInterval 自动刷新；手动点"刷新"按钮则调 POST /api/portfolio/refresh 立即重拉行情。

3. 异常路径与降级

全链路中有三类关键异常，每一类都有明确的错误处理：

异常触发条件检测层降级行为
代码非法 输入非 6 位数字 前端 /^\d{6}$/ 正则 + 后端 isdigit() 双重校验 前端直接 setErr 提示"请输入 6 位股票代码"，不发起请求
数量为零/负 shares <= 0 前端 parseFloat(shares) 检查 + 后端 h.shares <= 0 前端 setErr "数量须大于 0"；后端返回 HTTP 400
行情源失败 腾讯行情接口不可达或超时 get_portfolio() 中 try/except 兜底 quotes = {}——价格字段为 0.0，表格仍渲染但现价/浮盈列可能全为零，不抛 500
后端宕机 FastAPI 未启动或端口不对 api.ts request() 的 fetch catch 抛 ApiError "连接不到后端，请先启动 backend"，前端展示红色错误横幅

4. 关键设计决策

为什么用加权平均成本而不是 FIFO？Vibe-Research 是个人投研工具而非交易系统，不需要精确的税务批次追踪。加权平均成本在代码上只需一行公式，对个人用户来说足够透明——看到的是一个"综合成本价"，加仓后成本自动摊平。

为什么成本和清仓价允许为负数？融券做空时卖出价是正的，但持仓方向相反导致成本为负；某些返息场景也会产生负成本。不限制符号是尊重用户计算逻辑的选择——"你输入的，我们原样算"。

检索练习：如果用户在已持有 100 股成本 10 元的股票上，再录入 200 股成本 13 元，最终的合并成本是多少？get_portfolio() 返回的 totals 中 cost 是合并成本还是总成本？

参考答案：合并成本 = (100/10 + 200/13) / 300 = (1000 + 2600) / 300 = 12.0 元。totals 中的 cost 是所有持仓的总成本（合并成本 × 总股数 = 12.0 \* 300 = 3600），不是合并成本本身。这点容易混淆——每笔 holding.cost 是合并均价，totals.cost 是总持仓成本。
