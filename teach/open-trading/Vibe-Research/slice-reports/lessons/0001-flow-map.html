研报上传与管理全链路 — 从拖拽到归档的完整数据流

学习目标：15 分钟内能画出研报从浏览器拖拽上传到后端本地落盘、行业归档、下载删除的全链路时序图，理解每一层的职责与故障兜底机制。

1. 入口：拖拽文件到 base64 编码

用户在 MyReports 页面将文件拖入虚线框区域，onDrop 事件获取 FileList，调用 upload(files) 进入逐文件上传循环。每个文件通过 fileToB64() 做异步 base64 转换：

const fileToB64 = (file: File): Promise => new Promise((resolve, reject) => { const reader = new FileReader(); reader.onload = () => resolve(String(reader.result)); reader.onerror = () => reject(reader.error); reader.readAsDataURL(file); // → "data:application/pdf;base64,JVBERi..." });

readAsDataURL 会生成带 MIME 前缀的 data URI 字符串，后端 myreports.py 会自动剥掉 data:... 前缀再解码。这样前端无需额外依赖，后端兼容纯 base64 和 data URI 两种格式。

关键观察：下载链路不走 <a download> 直接指向文件 URL，因为后端可能要求 Authorization: Bearer 鉴权头，而浏览器原生 <a> 标签无法携带自定义请求头。因此前端用 fetch → blob → URL.createObjectURL → 程序化点击 完成带鉴权的文件下载。

4. 异常路径：三种典型失败场景

场景 A — 文件过大：用户拖入一个 50MB PDF。save_report 解码后 len(blob) > MAX_BYTES 为真 → 抛出 ReportError("文件过大...") → FastAPI 捕获 → HTTP 400 → 前端 catch 设置 err 状态 → 页面显示红色错误条"上传失败"。

场景 B — 格式不支持：用户拖入 .exe 文件。扩展名不在 ALLOWED_EXT 白名单中 → ReportError("不支持的文件类型") → HTTP 400 → 前端同样显示错误。设计意图：白名单排除了可执行文件和网页文件，避免下载回放时的安全风险。

场景 C — 并发上传覆盖索引：用户同时拖入 5 个文件。upload 循环逐文件 POST，后端 _LOCK（threading.Lock）串行化索引的读-改-写操作。每个请求在写入 index.json 前获取锁，防止并发追加导致数据丢失。索引文件通过先写 .tmp 再 os.replace 原子改名，即使进程被 kill 也不会留下半截损坏的 JSON。

暂停回忆：如果后端 VR_API_KEY 环境变量已设置，但前端 localStorage 中没有存储 access key，用户点击"下载"按钮会发生什么？前端 downloadReport 为什么不直接用 <a href="/api/myreports/file/{rid}" download>？

参考答案：1) fetch 请求因缺少 Authorization 头而收到 HTTP 401 → resp.ok 为 false → 抛出 ApiError("下载失败 HTTP 401", 401) → 前端显示红色错误条。2) 浏览器 <a> 标签发起的 GET 请求是浏览器原生行为，无法注入自定义 HTTP 请求头（如 Authorization: Bearer xxx），因此必须走 fetch → blob → createObjectURL → 程序化 <a> click 的链路来携带鉴权信息。
