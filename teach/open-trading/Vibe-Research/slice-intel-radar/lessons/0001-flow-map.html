资讯雷达主链路：页面加载 → 缓存读取 → 12赛道渲染

学习目标：15 分钟内能画出资讯雷达从页面加载到数据展示的完整时序图，理解 React 状态管理、API 调用封装、后端缓存读取三级降级策略。

1. 入口：Intel 页面组件树

用户点击导航栏"资讯雷达"，React Router 渲染 Intel 组件。该组件使用 Tab 切换模式，默认选中"Investment News"标签页，对应 InvestmentNewsPanel 子组件——它是本切片的真正主角。

frontend/src/pages/Intel.tsx ├── export function Intel() ← 页面入口，Tab 切换容器 │ └── InvestmentNewsPanel ← 资讯雷达核心面板（本课焦点） │ ├── useState<RadarData|null> ← 雷达数据状态 │ ├── useState<string|null> ← 错误信息状态 │ ├── useState("ai") ← 当前选中赛道（默认 AI） │ └── useState<Digest> ← AI 要点提炼状态

InvestmentNewsPanel 在 useEffect 中调用 api.radar() 发起初始数据请求，这是整条链路的起点。

2. 第一层：前端 API 调用

api.radar() 定义在 frontend/src/lib/api.ts 中。它是一个薄封装——通过共享的 request<T>() 泛型函数发出 GET 请求，自动附加鉴权头和处理错误：

Intel.tsx: useEffect → api.radar() → request<RadarData>("/radar","GET") → fetch("/api/radar", {headers}) │ ▼ 失败 throw ApiError("连接不到后端…")

关键类型定义（来自 api.ts:167-176）：

类型字段说明
RadarDatagenerated_at, recent_days, industries[], stats顶层返回结构
Industrykey, name, accent, total, items[]单个赛道（如 AI/大模型）
RadarItemtitle, url, time, source, summary?, zh?单条资讯

3. 后端：三级降级策略与缓存结构

前端请求到达后端 GET /api/radar（app.py:233-239），路由函数只做一件事：调用 newsradar.get_radar(force=False)。它实现了一个三级降级策略——要么读缓存、要么返回骨架，从不主动触发抓取：

get_radar(force=False) │ ├─ 1. load_cache() ← 读 .cache/radar.json │ └─ 命中 → 返回缓存数据（99% 的请求走这条路） │ └─ 2. skeleton() ← 缓存文件不存在 / JSON 解析失败 └─ 返回赛道骨架（空 items）：12 个 industry 对象，items=[]

缓存文件结构（backend/.cache/radar.json）：

radar.json ├── generated_at: "2026-07-08 14:30" ← 缓存生成时间（北京时间） ├── recent_days: 7 ← 只保留最近 7 天资讯 ├── industries: [ ← 12 个赛道数组 │ ├── { key:"ai", name:"AI/大模型", accent:"#ff5a1f", │ │ total:16, items: [...] } ← 该赛道 16 个源，资讯按时间倒序 │ ├── { key:"semi", ... } │ └── ...（共 12 个） └── stats: { industries:12, total_sources:108, failed_sources: 3 } ← 抓取统计（3 个源失败）

设计要点：这种"读和写分离"的设计避免了每次页面加载都触发 108 个 HTTP 请求。skeleton() 降级路径直接从 news_sources.json 构造同样的结构，但 items=[]、generated_at=null。前端通过检查 !!data?.generated_at 判断是否有真实数据。

4. 前端渲染：状态驱动的条件 UI

数据回到前端后，InvestmentNewsPanel 根据 hasData 标志渲染三种状态：

状态条件渲染内容
空状态!hasData && !err引导文案："还没有抓取资讯，点刷新拉取"
错误状态err !== null红色错误卡片，显示 err 消息
数据状态!!data?.generated_at12 赛道 Pill 按钮 + 资讯列表 + AI 要点提炼区

赛道切换通过 active 状态变量实现，点击 Pill 按钮即设置 setActive(ind.key)，触发对应赛道的资讯列表和 AI 提炼区重新渲染——全程纯前端操作，无需再请求后端。

检索练习：如果用户浏览器请求 /api/radar 时 .cache/radar.json 文件不存在，后端返回什么？前端展示什么？

参考答案：后端 load_cache() 捕获 FileNotFoundError 返回 None，get_radar() 转而调用 skeleton()，返回 12 个空赛道的 JSON。前端收到后 data.generated_at 为 null，hasData 为 false，渲染引导文案："还没有抓取资讯，点上方刷新拉取（约 20-40 秒）"。这是优雅降级的典型案例——无缓存时不报错，而是引导用户主动触发抓取。
