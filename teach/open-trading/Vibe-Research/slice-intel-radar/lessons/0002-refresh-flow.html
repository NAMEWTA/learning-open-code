强制刷新链路：108 RSS 并发抓取 → 合规过滤 → 原子缓存

学习目标：15 分钟内能解释 fetch_radar() 的并发模型、单源抓取逻辑、合规过滤机制和原子缓存写入策略。

1. 触发路径：前端 → POST 端点 → fetch_radar()

用户在 上一节课 的页面中点击"刷新"按钮，触发与主链路不同的 POST 路径：

Intel.tsx: refresh() → api.radarRefresh() → POST /api/radar/refresh → newsradar.fetch_radar() │ ▼ 耗时 20-40s app.py:242-248 返回 RadarData

与 GET /api/radar 的关键区别：get_radar(force=False) 只读缓存，而 fetch_radar() 每次都重新抓取全部 108 个 RSS 源，完成后覆盖缓存并返回新鲜数据。

2. fetch_radar() 四阶段流水线

fetch_radar()（newsradar.py:101-143）按以下四阶段执行：

阶段 1: 加载配置 news_sources.json → {industries, sources, redline_keywords, fetch:{recent_days,per_source}} 阶段 2: 构建任务列表 sources[] ──按 hint 分组──▶ byhint: {"ai":[16 个源], "semi":[9 个源], …} 遍历 industries[] → 匹配 byhint → 生成 (industry_index, source_dict) 任务对 阶段 3: 40 线程并发抓取 ThreadPoolExecutor(max_workers=40).map(lambda t: (t[0], _fetch_source(t[1], …)), tasks) → 每个源超时 14s → XML 解析 → 日期过滤 → 合规过滤 → 返回 items[] 阶段 4: 聚合排序 + 原子落盘 按 industry_index 归并 items → 各赛道按 ts 倒序 → JSON 写 tmp → os.replace(tmp, file)

3. 单源抓取：_fetch_source() 内部逻辑

每个 RSS 源的抓取由 _fetch_source()（newsradar.py:56-98）完成。这是链路中最底层的原子操作：

步骤代码位置说明
HTTP GETL59-63urllib.request.urlopen(req, timeout=14)，设 Accept 头声明 RSS/Atom
XML 解析L64ET.fromstring(raw)，标准库 xml.etree.ElementTree
节点遍历L67-97查找所有 item/entry 节点，提取 title/link/pubDate/description
合规过滤L84-86标题+摘要拼接后小写，匹配 redline_keywords（赌/预测市场/加密/色情）
日期过滤L87-92只保留 cutoff（7天前）之后的资讯，转北京时间格式化
截断L68-69每个源最多取 per_source=6 条
异常降级L97-98任何异常（超时/XML格式错/网络不可达）→ 返回 None

关键设计：_fetch_source() 的异常处理是"静默吞掉"。单个 RSS 源失败不影响其他源，最终 fetch_radar() 在 stats.failed_sources 中记录失败数。这是一种"尽力而为"（best-effort）的聚合模式——宁可少几条资讯，也不能因为一个源挂了就让整个页面报错。

4. 并发模型与原子写入

ThreadPoolExecutor(max_workers=40) 是这段代码最有意思的设计决策。108 个 RSS 源分散在全球各地，单线程串行抓取需要 108 x 2s ~ 216 秒。40 个线程并发下，理论耗时约 ceil(108/40) x 2s ~ 6 秒（不含网络抖动）。实际约 20-40 秒，因为部分 RSS 源响应慢（超时 14 秒），且 XML 解析消耗 CPU。40 是经验值——太少则并发度不够，太多可能触发 rate limiting。

缓存写入使用了经典的"写临时文件 + 原子重命名"模式：

1. json.dump(data, ".cache/radar.json.tmp") ← 先写临时文件 2. os.replace(tmp, ".cache/radar.json") ← 原子 rename（Unix 上保证原子性）

如果两个请求同时触发刷新，可能出现两次并发写。临时文件 + 原子 rename 保证了：读缓存的一方要么读到完整的旧缓存，要么读到完整的新缓存——永远不会读到写了一半的损坏 JSON。

判断练习：以下哪些情况会导致 fetch_radar() 返回的某个赛道 items 为空？
A. 该赛道所有 RSS 源服务器都不可达
B. 该赛道所有资讯都包含"bitcoin"关键词
C. 该赛道所有资讯都是 8 天前发布的（recent_days=7）
D. 该赛道在 news_sources.json 中定义了 0 个源

参考答案：A、B、C 都可能。A → 所有 _fetch_source 返回 None，items 为空。B → 合规过滤（redline_keywords 含 "bitcoin"）全部拦截。C → 日期过滤（cutoff=7天前）全部排除。D → industries 定义了该赛道但 byhint 中没有匹配的源，total=0，items 自然为空。四种情况都不会报错，体现了尽力而为的聚合哲学。
