每日复盘全链路地图 — 8 路数据从浏览器到数据源

学习目标：15 分钟内能画出每日复盘页面 8 路并加载的完整调用图，定位每一层的职责与故障兜底策略。

1. 入口点：浏览器 URL 到组件挂载

用户访问 /daily-review（默认首页），React Router 匹配到 DailyReview 组件。组件挂载时 useEffect 立即触发两条路径：

useEffect(() => { loadIndices(); // 路径A：并行 5 路大盘级数据 refreshWatch(loadWatch()); // 路径B：自选股行情（依赖 localStorage） }, []);

loadIndices() 一次性发起 5 个独立的 HTTP GET，互不等待。这是页面性能的关键——如果串行请求，8 个 200ms 的 RTT 将累计 1.6 秒；并行则在最慢请求的延迟内全部返回。

关键观察：Vite 代理层对前端完全透明——fetch("/api/indices") 被 Vite dev server 转发到 localhost:8900。生产部署时换 Nginx 反代，代码零改动。FastAPI 层的 market.py 有全站共享的 5 分钟 TTL 缓存（_CACHE 字典），多次打开同页面不会重复抓取外部数据源。

4. 异常路径：非交易时段数据空白

场景：周末或收盘后打开每日复盘页面，东财涨停池返回空、akshare 乐股数据无更新。

链路表现：

market._emotion() 回溯 8 天仍找不到涨停池 → 返回 {} → get_short_term_emotion() 空字典不缓存（valid=bool 判否）→ FastAPI 返回 {"data": {}} → 前端 emotion?.zt_count === undefined 为真 → 触发 pending(emoDone) 显示"暂无数据：可能是非交易时段或数据源暂时不可用"。

设计意图：不报错、不崩溃、不显示过期数据。用户看到的是解释性占位文本，而非红色报错堆栈。

暂停回忆：如果 api.indices() 成功但 api.emotion() 返回空对象，页面上的短线情绪卡片会显示什么？emoDone 状态是在哪个时刻被设为 true 的？

参考答案：会显示 pending(true) 即"暂无数据：可能是非交易时段或数据源暂时不可用"。emoDone 在 api.emotion().then(...).catch(...).finally(() => setEmoDone(true)) 的 finally 块中设为 true——无论请求成功还是失败，标记"已结束"以区分加载中状态。
