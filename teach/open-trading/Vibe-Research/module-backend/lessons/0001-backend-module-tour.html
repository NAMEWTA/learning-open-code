Vibe-Research 后端 FastAPI 模块总览

学习目标：15 分钟内能说出后端 10 个模块的职责边界、30+ API 端点的分组逻辑，以及请求从 HTTP 入口到数据源的完整流向。

1. 模块职责与架构位置

Vibe-Research 后端是一个以 FastAPI 驱动的 A 股/美港股数据聚合中枢，同时承担 AI 对话引擎和本地持久化职责。它监听 :8900 端口，前端 Vite 开发服务器通过代理将 /api/* 请求全部转发过来。一句话概括：接收前端请求 → 路由到对应数据模块 → 多数据源聚合 → 缓存后返回 JSON。

在 L0 项目地图中，后端位于架构的中间层：

浏览器 :5899 ──Vite代理 /api──▶ FastAPI :8900 ──HTTP/TCP──▶ 腾讯/东财/同花顺/mootdx/akshare

后端不直接面向用户，它只暴露 REST API。前端是它唯一的"用户"；MCP Server 则是面向 Claude Code agent 的另一条通道。

2. 内部分层结构

后端内部按"路由层 → 数据模块层 → 外部数据源"三层划分：

backend/ ├── app.py ← 路由层：FastAPI 应用定义 + 30+ 端点 + 中间件链 │ + 全站共享缓存（行情5分钟/公告15分钟/估值30分钟） ├── astock.py ← 数据模块：A股行情/估值/研报/财务/资金面 ├── gstock.py ← 数据模块：美股/港股行情 + 关键财务 ├── market.py ← 数据模块：市场情绪 + 板块资金流 + 全球指数 ├── newsradar.py ← 数据模块：12赛道108 RSS 源聚合 ├── portfolio.py ← 数据模块：持仓增删改查 + 实时盈亏 ├── chat.py ← 数据模块：AI 对话 (function-calling 流式) ├── cli_runtime.py ← 数据模块：订阅CLI 检测与 spawn ├── mcp_server.py ← 独立进程：MCP JSON-RPC over stdio └── myreports.py ← 数据模块：研报上传/下载/行业归档

请求流向：每个 /api/* 请求先过中间件链（CORS 放行 → 可选 API Key 鉴权），再由 app.py 中对应的路由函数处理。路由函数调用下方的数据模块获取原始数据，模块内部通过 urllib/requests/mootdx 直连外部数据源，最后将结果包装为 {"data": ...} 返回。

4. 关键依赖与启动

后端依赖分两层——核心层秒装可用，全栈数据源按需安装：

层依赖用途缺失时行为
核心fastapi, uvicorn, requestsWeb 框架 + HTTP 客户端服务无法启动
全栈akshare, mootdx, pandas新闻/公告/K线/财务对应端点返回 501 + pip install 提示

启动命令：

uvicorn app:app --host 127.0.0.1 --port 8900

启动后 portfolio.py 会立即注册一个后台线程，每 30 分钟自动刷新一次持仓盈亏。

暂停回忆：Vibe-Research 后端的三个缓存 TTL 值（5分钟/15分钟/30分钟）分别对应什么类型的数据？为什么这样设计？

参考答案：5 分钟用于行情/市场情绪等实时数据（盘中变化快，但 5 分钟内的抖动对投研无意义）；15 分钟用于公告/资金流等日级数据（变化慢，但盘中可能有新公告）；30 分钟用于估值分位/财务/股东户数等季级静态数据（几乎不变，减少对东财限流接口的请求）。这样分级既保证了数据的时效性，又最大限度地降低了对上游数据源的请求压力和被封 IP 的风险。

主要来源：backend/app.py（全部端点注册与中间件链）、backend/requirements.txt（依赖分层声明）、backend/astock.py（惰性导入模式）。完整 API 端点清单和模块详解见 backend-overview 参考图谱。不清楚某个端点或模块时，可以让 agent 顺着对应源码做 L2 垂直切片深挖。
