AI 对话三条通道全景

学习目标：15 分钟内能画出三条 AI 对话通道的端到端数据流向，说出各自的入口、核心模块和适用场景。

1. 三条通道，一个入口

用户在 Settings 页面配置好 AI 后，所有"问 AI"功能（持仓页问 AI、个股页问 AI、每日复盘等）走的是同一条 HTTP 路径——POST /api/chat。差别在于请求体中 llm.provider 字段的值，它决定了后端走哪条通道：

前端发起请求 ─── POST /api/chat ─── 后端 app.py 第 102 行 │ provider=deepseek/… provider=cli-claude/… (外部 agent) │ │ │ 通道 1: API 通道 2: CLI 通道 3: MCP chat.run_chat_stream chat.run_chat_cli_stream mcp_server.py │ │ │ OpenAI 兼容端点 本机 CLI 子进程 JSON-RPC over stdio (function-calling 循环) (纯文本应答, 免key) (暴露数据工具给agent)

关键判断点——app.py:102：is_cli = req.llm.provider.startswith("cli-")。一行代码完成了通道分发。

3. 通道 1：API 接入（function-calling 循环）

最通用的通道。用户填 API key → 前端发 POST /api/chat → 后端调 OpenAI 兼容端点。AI 可以自己决定何时调用数据工具（查行情/估值/研报/新闻），拿到客观数据后再作答。这是唯一一条能让 AI"自主查数据"的通道。

核心流程：

前端 llm.ts:chatStream() │ 发送 {messages, context, llm} ▼ 后端 chat.py:run_chat_stream() │ 构建 messages（SYSTEM_PROMPT + 用户历史） │ for rnd in 1..6: │ │ _call_llm_stream() → SSE 流 │ │ _iter_sse_deltas() → 逐 delta 解析 │ │ 若有 tool_calls → _exec_tool() → 结果注入 messages │ │ 若无 → yield done，结束 ▼ 前端 llm.ts: 解析 NDJSON 行 │ delta → 逐字渲染 │ tool → 显示"正在查 {工具名}" │ done → 取回 trace + rounds ▼ 用户看到流式回答

上限 6 轮（MAX_ROUNDS），防止模型陷入无限调工具的循环。达到上限后不传 tools、直接收尾一次回答。

4. 通道 2：CLI 订阅接入（纯文本应答）

用户在 Settings 选"订阅接入"（如 Claude Code / Qwen / DeepSeek / Codex），后端 spawn 对应的本机 CLI 子进程。数据工具不能由 AI 自行调用——因为 CLI 不像 API 那条有 function-calling 机制。因此前端必须把页面已有数据塞进 context 参数，CLI 只负责基于 context 做文本分析。

适用场景：每日复盘总结、今日要点提炼、个股页已有行情数据的分析——这些场景中数据已经在前端页面上，不需要 AI 再去查。

详细 CLI 适配机制见 第 3 节 CLI 适配与 MCP 服务。

5. 通道 3：MCP 服务（工具暴露）

与前两条不同，MCP 通道没有前端参与。它是独立进程，通过 stdin/stdout 与 Claude Code 等 agent 通信（JSON-RPC 2.0 协议）。agent 发起 tools/list 获取可用工具清单，然后用 tools/call 调用 chat._exec_tool 获取数据。

这条通道的定位是：让 agent 在分析投研问题时，能直接拿到 A 股/美港股的客观数据。agent 用自己的订阅额度做分析，Vibe-Research 只充当"数据工具箱"。
