API 接入通道：function-calling 循环

学习目标：15 分钟内能讲清 API 通道从前端填 key 到流式回答返回的每一步，理解 function-calling 循环的 6 轮上限设计。

1. 前端配置：怎么选模型

用户在 Settings 页面选择"API 接入"模式后，看到的是一个下拉选择 + 三个输入框。下拉框来自 ai-models.ts 的 apiModels 数组（订阅版 subscriptionModels 被过滤掉），共 12 个选项：

apiModels [ ├── DeepSeek V4 Flash/Pro ← provider: deepseek ├── SiliconFlow · DeepSeek V3 ← provider: silicon ├── OpenAI GPT-4o ← provider: openai ├── MiniMax M2 ← provider: minimax ├── 豆包 Pro ← provider: openai-compatible ├── OpenRouter · GPT-4o ← provider: openrouter ├── Groq · Llama 3.3 70B ← provider: groq ├── Together · Llama 3.3 70B ← provider: together ├── MiMo V2.5 Pro ← provider: mimo └── 其它 OpenAI 兼容 ← provider: openai-compatible ]

每个选项绑定了 provider（如 deepseek），选中后 pickApiModel() 自动填入 PROVIDER_BASE 中的默认 baseURL。用户只需再填 API key 和 model 名。所有配置通过 saveLlm() 写入 localStorage 的 vr-llm 键，不上传服务器。

2. 前端发起对话：chatStream

llm.ts:59 的 chatStream() 是前端的唯一对话入口。流程：

chatStream(messages, context, handlers, signal) │ 1. loadLlm() → 从 localStorage 读配置 │ (provider 非 cli-* 需 baseURL+apiKey+model 三者齐全) │ 2. fetch POST /api/chat │ body: { messages, context, llm: {provider, baseURL, apiKey, model} } │ 3. 检查 resp.ok — 401=鉴权错误, 400=配置缺失 │ 4. resp.body.getReader() → 逐字节读 │ 5. 按 \n 切行 → JSON.parse → 分发事件: │ type: "delta" → 追加 content, 回调 onDelta(text) │ type: "tool" → 回调 onTool(tool, args) │ type: "done" → 取 trace + rounds │ type: "error" → 记录 errMsg │ 6. 返回 {content, trace, rounds} │ 7. 支持 AbortController.signal 中止

防乱码技巧：buf.split("\n") 保留最后一段不完整行，等下一块数据到了再拼——这样不会在 UTF-8 多字节字符中间切断。

3. 后端核心：run_chat_stream 的 function-calling 循环

chat.py:320 是 API 通道的核心。它的本质是：反复调 LLM，每次拿到 tool_calls 就执行、注入结果，直到 LLM 不再要工具为止。

run_chat_stream(cfg, user_messages, context) │ ├─ 构建 messages: [system(SYSTEM_PROMPT), ...user_messages] │ ├─ for rnd = 1..6: ← 最多 6 轮 │ │ │ ├─ _call_llm_stream(cfg, messages, use_tools=True) │ │ │ POST {baseURL}/v1/chat/completions │ │ │ payload: {model, messages, tools=TOOLS, tool_choice="auto", stream:true} │ │ └─ 返回 resp (SSE 流) │ │ │ ├─ _iter_sse_deltas(resp) │ │ │ 逐字节缓冲 → 按 \n 切行 → 只处理 "data:" 行 │ │ │ data == "[DONE]" → 终止迭代 │ │ │ 解析 JSON → yield choices[0].delta │ │ └─ 逐 delta 推: content → yield {type:delta, text} │ │ tool_calls → 按 index 归位累积 │ │ │ ├─ 若 tool_acc 为空 → yield {type:done}, 结束 ✓ │ │ │ └─ 若有工具调用: │ │ messages.append(assistant 消息含 tool_calls) │ │ for each tool: │ │ │ 解析 arguments JSON │ │ │ yield {type:tool, tool, args} │ │ │ result = _exec_tool(name, args) ← 实际查数据 │ │ │ messages.append(tool 消息含 result) │ │ └─ trace.append({tool, args}) │ └─ 超 6 轮: 最后一次不传 tools, 取文本收尾 → done

关键设计决策：

  • 为什么是 6 轮？ MAX_ROUNDS = 6：一轮发问、最多 4 轮调工具（查行情+估值+研报+新闻）、一轮收尾回答。实际使用中极少超过 3 轮。
  • 为什么每次 rebuild messages？ 每轮调 API 时带完整的 messages 历史——这是 OpenAI 兼容协议的标准要求，模型不记前一轮的上下文。
  • tool_calls 按 index 归位：SSE 流中可能交错发来多个 tool_calls 的片段，必须按 index（或 id）归入正确的调用槽，否则 arguments 会串在一起。

4. 五个数据工具（TOOLS）

这 5 个工具是 AI 的"数据手臂"——AI 通过它们拿到客观数据。工具定义在 chat.py:61-124：

工具名数据源参数返回
query_quote腾讯行情 APIcodes: 6位代码数组现价/涨跌/PE/PB/市值/换手
query_valuation东财估值 + 腾讯行情code: 单个6位代码完整估值：前向 PE/PEG/PE 消化年数
query_reports东财研报code: 单个6位代码最近 15 篇研报标题/机构/评级
query_news同花顺新闻code: 单个6位代码最近 15 条新闻标题/时间/来源
query_global_stockpush2 美股/港股symbol: 美股代码或港股代码行情 + 关键财务指标

每个工具的结果通过 _TOOL_RESULT_CAP = 6000 截断——防止一次估值查询把整个 JSON 塞进下一轮 context，耗尽 token。

5. SYSTEM_PROMPT：投研五维框架

chat.py:31 的 ANALYSIS_FRAMEWORK 是整条通道的"灵魂"。它规定了 AI 分析个股时的组织方式：估值 → 资金面 → 财报质量 → 行业景气 → 事件催化与风险。这个框架不做成 UI 选项——直接焊进 SYSTEM_PROMPT，用户不用选，问就是这套标准。

合规约束同样焊在 prompt 里：只做信息整理与多视角分析，不荐股、不预测涨跌、不给买卖时机。

检索练习：用户对一只股票同时问了三个问题："现价多少？PE 过高吗？帮我分析一下这个股票。"AI 在 function-calling 循环中最少需要调几次工具？哪几次？

参考答案：最少 2 次。第 1 次：调 query_quote 拿现价 + PE。第 1 轮结果返回后，AI 发现还需要更全面的估值和财务信息，第 2 次同时调 query_valuation（估值分位）和 query_reports（机构观点）。第 2 轮结果拿到后，AI 就有足够数据按五维框架输出完整分析，不再需要调工具。实际可能多达 4 次（再加 query_news 查舆情），但 2 次是最少需要。
