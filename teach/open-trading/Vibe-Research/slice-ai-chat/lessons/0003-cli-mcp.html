CLI 适配与 MCP 服务

学习目标：15 分钟内能说清四种 CLI 的三种投递方式差异、cli_runtime.py 的超时控制设计，以及 MCP server 如何零依赖复用 chat.TOOLS。

1. CLI 通道总览：四种引擎，三种投递方式

cli_runtime.py:25 的 _CLI_DEFS 字典是 CLI 适配的"注册表"。每个 CLI 有三个属性：可执行文件候选名、投递方式、参数构建函数：

CLI可执行文件投递方式关键参数
Claudeclaude / openclaudesystem-file-p --system-prompt-file + 禁用所有工具
Qwenqwenstdin--yolo（跳过确认）
DeepSeekdeepseek / codewhaleargexec --auto（提示词作为位置参数）
Codexcodexstdinexec --skip-git-repo-check -（- 从 stdin 读）

三种投递方式的本质差异——不同 CLI 对"系统提示词"和"用户提示词"的接收方式不同：

方式 1: system-file（Claude 专属） ├─ 系统提示词 → 写临时文件 (Path(tmpdir)/system.txt) ├─ CLI 参数 → -p --system-prompt-file <文件路径> └─ 用户提示词 → 走 stdin (proc.stdin.write(user_prompt)) 方式 2: stdin（Qwen / Codex） ├─ system + user 合并为一段文本 └─ 全部走 stdin (proc.stdin.write(combined)) 方式 3: arg（DeepSeek 专属） ├─ system + user 合并为一段文本 ├─ 作为最后一个位置参数传给 CLI └─ 限制：≤ 110KB（_MAX_ARG_BYTES），超长抛异常

Claude 之所以能单独传 system prompt，是因为 Claude Code CLI 原生支持 --system-prompt-file 参数，这是 Claude 的独家能力。其他 CLI 没有这个接口，只能合并传入。

2. 流式管道：读线程 + 带超时的队列

run_cli_stream()（140 行）是流式版的核心。它不能简单地 for line in proc.stdout——因为这是无限期阻塞读，CLI 卡住时永远走不到 proc.wait(timeout=...)。解决方案：

run_cli_stream(kind, system_prompt, user_prompt) │ ├─ 1. detect_cli(kind) → 查 PATH + 额外目录 │ 遍历 _EXTRA_PATH_DIRS（Homebrew/npm/bun/deno/yarn 全局 bin） │ 判定: os.access(p, os.X_OK) │ ├─ 2. 按投递方式构建 args + stdin_payload │ ├─ 3. subprocess.Popen(..., text=True, bufsize=1) │ stdin → 写入提示词后立即 close() │ stdout → 管道，stderr → DEVNULL │ ├─ 4. 读线程 + queue.Queue │ daemon 线程: for line in proc.stdout → q.put(line) → q.put(None) │ 主线程: q.get(timeout=min(remaining, 1.0)) │ 超时 → RuntimeError("生成超时") │ None → EOF → break │ ├─ 5. proc.wait(timeout=10) → 取退出码 │ rc != 0 → RuntimeError │ └─ 6. finally: 若进程未退出 → kill() 清理临时目录 shutil.rmtree()

设计要点：_CLI_TIMEOUT_S = 300（5 分钟）是兜底超时。读线程 + 队列的组合确保即使子进程挂起，主线程也不会被永久阻塞——每 1 秒检查一次剩余时间。

3. CLI 通道的限制：为什么不能 function-calling

chat.py:251 的 run_chat_cli() 非常简短——拼好 prompt，调 CLI，返回文本。没有循环、没有工具调用。原因是：

  • CLI 不像 API 那样有标准的 tool_calls 响应格式，无法自动触发工具执行。
  • CLI 的一次调用是"黑盒"——你只能读 stdout，无法在中间拦截"它想调工具"的信号。
  • 即使 CLI 输出 JSON 格式的工具调用请求，没有统一的协议标准，每个 CLI 的输出格式不同。

因此 CLI 通道的数据必须已在 context 里。前端在发起请求时，把当前页面的数据（如持仓列表、个股行情、今日复盘摘要）序列化后塞进 context 参数，CLI 只做文本分析。对比 API 通道：context 参数同样可以传，但 API 通道还能让 AI 自己调工具——这是两条通道最本质的区别。

4. MCP Server：将工具暴露给 agent

mcp_server.py 只有 107 行，是三条通道中最精简的。它的设计哲学是零依赖复用——不引入任何第三方库（纯标准库），直接复用 chat.TOOLS 和 chat._exec_tool：

MCP 生命周期（5 步 handshake）: │ ├─ 1. initialize │ 客户端 → {"method":"initialize", "params":{...}} │ 服务端 → {protocolVersion, capabilities:{tools:{}}, serverInfo} │ ├─ 2. initialized（通知，无 id，不响应） │ 客户端 → {"method":"notifications/initialized"} │ ├─ 3. tools/list │ 客户端 → {"method":"tools/list"} │ 服务端 → {tools: [MCP_TOOLS]} ← 从 chat.TOOLS 转换 │ 转换: OpenAI {type:"function", function:{name, description, parameters}} │ → MCP {name, description, inputSchema} │ ├─ 4. tools/call（反复调用） │ 客户端 → {"method":"tools/call", params:{name, arguments}} │ 服务端 → chat._exec_tool(name, args) │ 返回 → {content:[{type:"text", text: JSON结果}], isError: ...} │ └─ 5. ping（心跳） 客户端 → {"method":"ping"} 服务端 → {}

通信方式：JSON-RPC 2.0 over stdio——每行一个 JSON 对象，sys.stdin 读入、sys.stdout 写出。这是 MCP 协议的标准传输层，无需 TCP 端口、无需 HTTP 服务器。

挂载方式：在 Claude Code 中执行 claude mcp add vibe-research -- /path/to/python /path/to/mcp_server.py，之后 agent 就能在对话中直接调用 A 股数据工具。

5. 三通道选型决策树

我要用 AI 分析投研数据…… │ ├─ 场景：自己用，想填个 key 就用 │ → API 接入（通道 1） │ 理由：最通用，支持 function-calling，AI 可以自己查数据 │ ├─ 场景：本机已装 Claude Code / 其他 CLI，不想再买 API key │ → CLI 订阅接入（通道 2） │ 条件：数据已在页面上（复盘/要点/持仓分析），不需要 AI 再查 │ 注意：不能 function-calling，数据靠前端 context │ └─ 场景：用 Claude Code 做深度投研，想让它直接查数据 → MCP 服务（通道 3） 条件：已安装 Claude Code，配置了 mcp add 注意：这是给 agent 用的，不是给网页聊天用的

检索练习：cli_runtime.py 的 run_cli_stream() 中，为什么不用简单的 for line in proc.stdout 读输出，而是要用 daemon 线程 + queue.Queue 的组合？

参考答案：for line in proc.stdout 是无限期阻塞读——只要子进程不关闭 stdout（比如挂起、死循环输出），主线程就会永远卡在这里。即使外层有 timeout 逻辑，也走不到 proc.wait(timeout=...)，超时形同虚设。线程 + 队列的方案让读操作独立于主线程，主线程用 q.get(timeout=1.0) 轮询，每 1 秒检查一次剩余时间，确保 300 秒兜底超时真正生效。
