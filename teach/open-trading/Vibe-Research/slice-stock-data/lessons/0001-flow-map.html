0001 个股数据查询 — 全链路地图

学习目标：15 分钟内能说出 6 位代码输入后经历的三层请求编排（竞态守卫 / 主数据块 / fire-and-forget），以及请求如何从浏览器走到各个后端端点。

1. 入口分叉：A 股还是美股？

用户在输入框敲入代码，点击"查询"或按 Enter 后，StockData.tsx 的 run() 函数被调用。第一步不是发请求，而是判断代码类型：

/^\d{6}$/ 匹配 6 位纯数字 → A 股路径（20+ 端点并行） 非 6 位纯数字 → 美股/港股路径（单端点 /api/global/stock）

这个分支在 StockData.tsx:114 行。A 股路径是整个页面最复杂的部分——单次查询拉起 20+ 个 API。理解它的请求编排是本节的核心。

2. A 股路径：三层请求编排

A 股查询并非一次性发起所有请求。它被设计为三层，每一层有不同的错误处理策略：

层端点策略失败时
竞态守卫runIdRef（非端点）每次查询分配新 ID，回填时只接受最新 ID 的结果旧结果自动丢弃
主数据块valuation / reports / percentile / financials / announcements / newsPromise.all() 并发 + 各自 .catch 降级单端点失败不影响其余；news 独立 try/catch
fire-and-forgetmargin / blockTrade / holders / dividend / fundFlow / dragonTiger / lockup / blocks / hotConcepts / investorQa10 个独立 .then().catch()，不 await，不阻塞主数据静默失败，对应卡片不渲染

关键代码结构（简化）：

// 竞态守卫 const rid = ++runIdRef.current; // 每次查询递增 // 第 3 层 — 先行发射，不阻塞 api.margin(c).then(ok(setMargin)).catch(() => {}); api.blockTrade(c).then(ok(setBlockT)).catch(() => {}); // ... 共 10 个 fire-and-forget // 第 2 层 — 主数据并发 const [v, r, p, f, a] = await Promise.all([ api.valuation(c), api.reports(c).catch(() => []), api.percentile(c).catch(() => null), api.financials(c).catch(() => null), api.announcements(c).catch(() => []), ]); if (rid !== runIdRef.current) return; // 竞态检查 // news 独立降级 try { setNews(await api.news(c)); } catch (e) { setDepNote(e.message); }

为什么这样设计？主数据块（行情/估值/财务/研报/公告）是用户第一眼看到的内容，必须尽快展示。资金面/龙虎榜/解禁等是"进一步探索"的数据——先展示核心卡片，再逐步回填外围卡片，体验更好。竞态守卫确保用户快速切换代码时，旧请求的回填不会覆盖新请求的结果。

3. 从浏览器到后端的物理路径

前端 api.ts 中的 request() 函数将所有调用统一发到 /api/*：

浏览器 :5899 │ fetch("/api/valuation?code=600519") ▼ Vite 开发服务器（代理 /api → localhost:8900） │ ▼ FastAPI app.py（uvicorn :8900） │ 中间件链：CORS 放行 → 可选 Bearer 鉴权 │ 路由匹配：@app.get("/api/valuation") → valuation(code) ▼ astock.py → 腾讯/东财/同花顺/mootdx/akshare

注意：Vite 代理只在开发模式生效。生产部署时通常用 Nginx 反向代理替代。无论哪种方式，前端代码只看到 /api/*，不知道后端跑在哪里——这是前后端分离的标准实践。

4. 全链路时序全景

下面这张图覆盖从输入框到 DOM 更新的完整时间线：

用户输入 "600519" → Enter │ ├─[0ms] run() 调用 │ ├─ 清空所有 state │ ├─ rid = ++runIdRef.current │ └─ setLoading(true) │ ├─[~1ms] 10 个 fire-and-forget 发射（不阻塞） │ margin / blockTrade / holders / dividend │ fundFlow / dragonTiger / lockup / blocks / hotConcepts / investorQa │ ├─[~2ms] Promise.all 发射 5 个主端点 │ valuation / reports / percentile / financials / announcements │ │ ─── 网络往返（50-2000ms，取决于数据源响应速度）─── │ ├─[~200ms] Promise.all 陆续 resolve │ ├─ 估值、财务先到 → setVal / setFin / setPctl │ ├─ 研报、公告稍后 → setReports / setAnns │ └─ rid 检查通过 → setLoading(false) │ ├─[~300ms] news 独立 try/catch 完成 → setNews 或 setDepNote │ └─[~500-2000ms] 资金面陆续回填 margin / blockTrade / holders → 对应卡片渐进出现

暂停回忆：StockData 页面的请求编排为什么不选择 Promise.allSettled 而是用各自 .catch()？这两种方式在失败处理上有什么本质区别？

参考答案：Promise.allSettled 会等待所有请求完成（无论成功失败）后才返回，这意味着一个慢端点会拖慢整个主数据块的渲染。.catch(() => null) 配合 Promise.all 则让该端点立即以 null 结束，不影响其他端点的数据展示。另外，news 端点放在独立 try/catch 而非 Promise.all 里，是因为它的失败需要区分 501（依赖缺失，提示安装）和普通异常（静默降级）——这种细粒度控制在 .catch() 里更容易实现。
