0002 个股数据 — 估值+财务路径

学习目标：15 分钟内能画出 /api/valuation → /api/valuation/percentile → /api/financials 的请求链，说出各端点的数据源、缓存 TTL 和前端二次加工逻辑。

1. 主数据块的五个端点

在 0001 全链路地图 中我们了解到，主数据块通过 Promise.all 并发发起 5 个端点（news 独立降级不计在内）。这 5 个端点构成了个股页面的核心信息密度：

端点后端函数数据源缓存依赖
/api/valuationastock.full_valuation(code)腾讯行情 + 东财一致预期无缓存标准库 + requests
/api/valuation/percentileastock.valuation_percentile(code)同花顺 5 年历史序列30 分钟akshare（501 降级）
/api/financialsastock.financials(code)同花顺财务摘要30 分钟akshare（501 降级）
/api/reportsastock.eastmoney_reports(code)东财研报列表无缓存requests
/api/announcementsastock.announcements(code)东财公告15 分钟requests

缓存设计逻辑：估值分位和财务数据基于同花顺的历史序列和最新报告期，变化频率是季度/年级别，因此给 30 分钟长缓存。公告可能在盘中发布，给 15 分钟缓存。而估值和研报数据涉及实时行情和 PDF 链接，变化快，不做缓存。这体现了分级缓存的核心理念——越静态的数据缓存越久。

2. 估值端点：full_valuation 的数据聚合

/api/valuation 是整个页面中信息密度最高的端点。它返回的数据不仅包含实时行情，还聚合了机构一致预期：

astock.full_valuation(code) │ ├─ 1. 调用 tencent_quote([code]) → 获取实时行情 │ 现价 / PE(TTM) / PB / 总市值 / 涨停价 / 跌停价 │ ├─ 2. 调用 eastmoney_forecast(code) → 获取一致预期 │ 26E EPS / 27E EPS / 机构覆盖家数 │ └─ 3. 客户端计算 → 前向 PE / PEG / 消化年数 PE_26e = price / eps_26e PEG = PE_26e / CAGR（CAGR 为负时 PEG 无效） digest_years = PE_ttm / CAGR（估值需要多少年盈利增长来消化）

注意前向 PE 和 PEG 是在 astock.py 内部计算的，不是直接从数据源获取的。这意味着如果一致预期数据缺失（很多中小票没有分析师覆盖），这些字段就是 null，前端用 "—" 显示。理解这一点很重要：估值的精度取决于一致预期数据的可得性，不同市值的股票信息密度天然不对等。

3. 估值分位带：理杏仁式可视化

估值分位端点的缓存键是 (code)——它是一个全局字典 _PCT_CACHE，直接以代码为键。这与资金面端点的 (endpoint, code) 复合键不同（见 0003 资金面），因为分位数据是单端点的。

_PCT_CACHE[code] = (timestamp, data) ← 30 分钟 TTL data = { period: "近5年", metrics: { pe_ttm: { current, percentile, min, max, p20, p50, p80, n }, pb: { current, percentile, min, max, p20, p50, p80, n } } }

前端 ValBand 组件（StockData.tsx:53-78）将这个数据渲染为一条三色带：绿=低估区（分位 <20%）、灰=合理区（20-80%）、红=高估区（>80%）。这条带子只显示位置，不划买卖线——源码注释反复强调"不构成买卖建议"，这是合规要求。

4. 财报速览：终端二次加工

数据到了前端不是直接展示，EarningsSnapshot 组件（EarningsSnapshot.tsx）对原始数据做了机械分档信号标签：

val/fin/pctl 三个对象 → 纯前端计算 → 信号标签（机械分档，不含买卖倾向）： "营收正增长" ← revenue_yoy >= 0 "营收高增长" ← revenue_yoy >= 30 "利润增速快于营收" ← net_profit_yoy >= revenue_yoy "高 ROE 15%" ← roe >= 15 "PE 低分位 25%" ← pe_ttm.percentile < 30 "PEG 0.8" ← 直接取自 val.peg 前向预期行： "一致预期 26E EPS 3.2 · 前向 PE 18.5 · 估值消化 3 年 · 15 家机构覆盖"

这个组件的设计理念是"结论先行"——把财报、估值、一致预期三个维度的数据压缩成一句话级别的判断。但它严格保持客观：只做机械分档（高/中/低、正/负），不做任何预测或推荐。

暂停回忆：估值分位端点的 30 分钟缓存为什么用代码作为单键而非 (endpoint, code) 复合键？如果后续新增一个也在 _PCT_CACHE 中存数据的端点，会发生什么问题？

参考答案：因为 _PCT_CACHE 是一个专用字典，只服务 /api/valuation/percentile 这一个端点，所以代码本身就是唯一键。但如果将来有人把其他端点的数据也写进这个字典且恰好用了相同的代码键，就会发生缓存污染——估值分位数据被其他端点覆盖，导致展示错误。这就是为什么资金面端点用的是 _DC_CACHE[(endpoint, code)] 复合键——它是 10 个端点共享的通用缓存池，必须用复合键隔离。设计缓存键时，要考虑缓存的共享范围。
