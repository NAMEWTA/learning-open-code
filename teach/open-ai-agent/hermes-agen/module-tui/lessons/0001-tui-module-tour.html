一、双进程架构

Hermes TUI 不是简单的 readline 循环，而是一个由 Node.js/React 前端和 Python Gateway 后端组成的双进程系统。TypeScript 拥有屏幕，Python 拥有会话、工具和模型调用。它们通过 JSON-RPC over stdio 通信——这决定了一切设计决策。
React 19 + Ink 6 Node.js 子进程 JSON-RPC 2.0 nanostores 双进程解耦 11 Python + 大量 TS

启动时，entry.tsx 创建 GatewayClient 实例，spawn 子进程 python -m tui_gateway.entry。两个进程通过 stdin/stdout 交换 JSON-RPC 帧——请求（请求方→响应方）、响应（request/response 对）和事件（Gateway 主动推送）。stderr 被捕获到内存日志环中，不直接写入终端。

用户键盘输入 │ ▼ ┌──────────────────────┐ stdin ── JSON-RPC 请求 ──→ ┌──────────────────────┐ │ ui-tui/ (Node.js) │ │ tui_gateway/ (Python) │ │ React + Ink 渲染 │ ←── stdout ── 响应 + 事件 ── │ 会话、工具、模型调用 │ │ entry.tsx → App │ stderr → 内存日志环 │ entry.py → server.py │ └──────────────────────┘ └──────────────────────┘ │ │ ▼ ▼ Ink render() 输出 ANSI 终端 hermes_cli / agent/

两种运行模式：（1）spawn 模式（默认）——Node 启动 Python 子进程；（2）attach 模式（HERMES_TUI_GATEWAY_URL）——连接到已在运行的 Gateway WebSocket。侧车镜像（sidecar）支持将事件同时转发到 Dashboard 侧边栏。

二、前端组件树与数据流

前端代码分三个层次：入口层（entry.tsx）负责 TTY 检测和 Gateway 启动；组合层（useMainApp Hook）连接所有子 Hooks 并暴露统一接口；渲染层（App + AppLayout）将状态映射为 Ink 组件。

// entry.tsx -- 核心启动序列（简化）
const gw = new GatewayClient()
gw.start()  // spawn python -m tui_gateway.entry

const { App } = await import('./app.js')
ink.render(<App gw={gw} />, { exitOnCtrlC: false })

// app.tsx -- 顶层组件
function App({ gw }: { gw: GatewayClient }) {
  const { appActions, appComposer, appProgress,
          appStatus, appTranscript, gateway } = useMainApp(gw)
  return (
    <GatewayProvider value={gateway}>
      <AppLayout actions={appActions} composer={appComposer}
        progress={appProgress} status={appStatus}
        transcript={appTranscript} />
    </GatewayProvider>
  )
}

状态管理使用 nanostores（而非 Redux/Zustand），关键 Store 包括：$uiState（busy/sid/mouseTracking/theme）、$overlayState（approval/clarify/sudo 弹窗）、turnStore（流式文本、工具状态、推理文本）。Gateway 事件通过 createGatewayEventHandler → turnController 管道驱动状态变更，React 自动重渲染。

components/ 目录包含 20+ 个 Ink 组件，其中 appLayout.tsx 组合 Transcript 区（<Static> 虚拟列表）、Status Bar、Input Row、Completions 列表和 Overlay 层（弹窗）。

三、三大交互系统

3.1 斜杠命令：两级 Dispatch

输入以 / 开头的字符串时，createSlashHandler 先在本地注册表（slash/registry.ts，7 个命令文件）中查找，命中则本地执行；未命中则通过 slash.exec RPC 发往 Python Gateway，后者再做一次 dispatch 并将结果返回。

// createSlashHandler.ts -- 两级 dispatch 核心逻辑（简化）
const found = findSlashCommand(parsed.name)  // 本地查找
if (found) {
  found.run(parsed.arg, runCtx, cmd)          // 本地执行
  return true
}
// 未命中 → RPC 发往 Gateway
gw.request('slash.exec', { command: cmd.slice(1), session_id: sid })
  .then(r => { /* 处理响应 */ })
  .catch(() => {
    // 斜杠执行失败 → 回退 dispatch
    gw.request('command.dispatch', { name: parsed.name, arg: parsed.arg, session_id: sid })
  })

3.2 多行编辑与队列

useComposerState 管理输入状态：Shift+Enter / Alt+Enter 插入换行；\ + Enter 追加到多行缓冲区（终端无修饰键支持的备选方案）；Ctrl+G 启动 $EDITOR 编辑当前草稿。Agent 忙时输入的文本自动入队（useQueue），响应完成后自动出队发送。Up/Down 键优先编辑队列消息，队列为空时走输入历史。

3.3 会话历史与补全

输入历史持久化到 ~/.hermes/.hermes_history，useInputHistory Hook 提供 Up/Down 导航。补全系统按前缀路由：/ 开头走 complete.slash RPC；路径类 token（./、../、~/、@）走 complete.path RPC。请求 60ms 防抖，Tab 键应用当前补全项。

四、Gateway 端：RPC 路由与事件发布

Python 端的 entry.py 是 stdio 事件循环——逐行读 stdin JSON，调用 server.dispatch(req) 处理，将结果写回 stdout。支持的 RPC 方法涵盖会话管理（session.create/resume/close）、消息发送（message.send）、配置读写、斜杠执行、补全等 30+ 种方法。

事件发布是 Gateway 的主动推送通道。40+ 种事件类型驱动前端渲染：message.delta（流式文本增量）、tool.start/complete（工具执行状态）、thinking.delta（推理过程）、approval.request（权限确认弹窗）。事件通过 transport.write() 写入 stdout JSON 帧，前端 GatewayClient.dispatch() 解析后进入 createGatewayEventHandler 管道。

关键设计决策：（1）JSON-RPC 帧是换行分隔的，避免了 HTTP 的头部开销，适合高频流式输出；（2）Transport 抽象层（StdioTransport / TeeTransport）使同一套 handler 可同时服务 stdio 和 WebSocket；（3）孤儿进程看门狗（slash_worker.py）检测父进程死亡，防止残留进程泄露。

尝试在不翻看上面内容的情况下回答：

  1. Hermes TUI 的两个进程各自负责什么？它们之间通过什么协议通信？
  2. 一条斜杠命令（如 /model）从输入到执行经历了哪些步骤？说明两级 dispatch 的 fallback 路径。
  3. TUI 在 Agent 忙时如何处理用户输入？Up/Down 键的导航优先级是什么？
点击查看参考答案

1. Node.js/React 前端（ui-tui/）负责终端渲染和用户交互；Python Gateway（tui_gateway/）负责会话管理、工具执行和模型调用。通过 JSON-RPC 2.0 over stdio 通信：stdin 发送请求，stdout 返回响应+事件，stderr 被捕获到内存日志。

2. ① 输入 /model，composer 检测到 / 前缀，请求 complete.slash；② 用户按 Enter 提交，createSlashHandler 先在本地注册表（slash/registry.ts）查找；③ 命中则本地执行（如 /model 属于 session.ts 命令），未命中则通过 slash.exec RPC 发往 Gateway，Gateway dispatch 失败则回退到 command.dispatch。

3. Agent 忙时，用户输入的文本自动进入队列（而非直接发送），响应完成后自动出队。Up/Down 键优先编辑队列中的消息；队列为空时才走输入历史导航。如果用户编辑了队列消息并重新提交，该队列项被替换并移到队首。
