Agent 对话全链路 L2 垂直切片 -- 从 CLI 输入到 LLM 响应的一条完整请求路径

学习目标：追踪一条用户消息穿越六个阶段到达 LLM 并返回的完整调用链 | 预估时长：15 分钟 | 前置课程：L1 Agent 核心引擎、 L1 Agent 入口层 | 关联 L1：L1 工具层

一、全链路六阶段总览

一条用户消息在 Hermes Agent 中不经过任何"魔法"——它严格穿越六个阶段，每个阶段有明确的入口函数和数据变换。理解这条链路，就掌握了 Agent 的"血液循环系统"。
6 个阶段 调用链深度 ~10 层 核心文件 ~13K 行 L2 垂直切片

下图给出了从终端键入一条消息到最终回复的完整阶段划分：

用户输入 ──▶ 阶段1: CLI入口 (cli.py)
                Fire解析 → AIAgent构建(60+参数) → _run_agent()
                │
                ▼
            阶段2: 前置准备 (conversation_loop.py L518)
                build_turn_context():
                  _restore_or_build_system_prompt()  恢复/构建提示词
                  _sanitize_surrogates()            消息清洗
                │
                ▼
            阶段3: 对话主循环
                while api_call_count < max_iterations:
                  → 中断检测 → 预算检查 → _perform_api_call()
                  → api_mode路由(anthropic/codex/bedrock)
                  → tool_calls分发 → tool_executor执行
                  → 结果回填messages → 继续循环或context_compressor压缩
                终止条件: 纯文本回复/预算耗尽/用户中断/错误

                build_api_kwargs() → interruptible_api_call()

                  · 流式/非流式双路径(stream_callback决定)
                │
                ▼
            阶段5: 响应构建 (chat_completion_helpers.py L879)
                build_assistant_message():
                  提取reasoning → 剥离<think>标签
                  → 清洗surrogates → 脱敏 → 标准化dict
                │
                ▼
            阶段6: 后处理
                context_compressor: token超限时压缩历史+生成摘要
                memory_manager: 同步MemoryProvider插件
                curator: 检查是否触发技能自动学习

2. API 调用：build_api_kwargs() 的多 Provider 路由

# agent/chat_completion_helpers.py L606
def build_api_kwargs(agent, api_messages: list) -> dict:
    """根据 api_mode 路由到不同 Provider 的请求构建器。"""

    if agent.api_mode == "anthropic_messages":
        _transport = agent._get_transport()
        anthropic_msgs = agent._prepare_anthropic_messages_for_api(api_messages)
        return _transport.build_kwargs(model=agent.model, messages=anthropic_msgs, ...)

    if agent.api_mode == "bedrock_converse":
        _bt = agent._get_transport()
        return _bt.build_kwargs(model=agent.model, messages=api_messages, ...)

    if agent.api_mode == "codex_responses":
        _ct = agent._get_transport()
        _msgs = agent._prepare_messages_for_non_vision_model(api_messages)
        # xAI schema 清洗: strip_pattern_and_format + strip_slash_enum
        return _ct.build_kwargs(model=agent.model, input=_msgs, ...)

3. 响应组装：build_assistant_message() 的内容清洗

# agent/chat_completion_helpers.py L879
def build_assistant_message(agent, assistant_message, finish_reason: str) -> dict:
    """从 API 响应构建标准化的 assistant 消息 dict。"""

    reasoning_text = agent._extract_reasoning(assistant_message)
    # 兜底：从 <think> 标签提取 reasoning（部分模型嵌入在 content 中）
    if not reasoning_text:
        think_blocks = re.findall(r'<think>(.*?)</think>', content, re.DOTALL)

    _san_content = _sanitize_surrogates(assistant_message.content)
    _san_content = agent._strip_think_blocks(_san_content)  # 剥离冗余标签
    _san_content = redact_sensitive_text(_san_content)       # 脱敏处理

    return {"role": "assistant", "content": _san_content,
            "reasoning": reasoning_text, "finish_reason": finish_reason}

注意三个关键时序点：(1) build_turn_context() 仅在回合开始时执行一次；(2) 每次 API 调用都经过 build_api_kwargs() 重新组装请求参数（因为 messages 列表在工具结果回填后已变化）；(3) 工具结果回填到 messages 后，下一次循环迭代自动携带上下文。

四、异常路径：Provider 故障链式切换

当 LLM Provider 返回错误（429 限流、5xx 服务端错误、401 认证过期），对话循环不会直接失败——它会触发 try_activate_fallback() 沿 fallback 链逐个切换 Provider，并在切换后同步模型身份到系统提示词。

故障切换流程

_perform_api_call()
    │
    ▼
  API 返回错误（HTTP 429 / 503 / 401）
    │
    ▼
  classify_api_error() → FailoverReason.rate_limit / auth / server_error
    │
    ▼
  try_activate_fallback(reason)           ← chat_completion_helpers.py L1172
    │
    ├── _fallback_index 递增
    ├── 跳过 unavailable_fallback_keys（已标记不可用的条目）
    ├── 跳过与当前 provider+model 相同的条目（去重）
    ├── 检查本地可用性 (network / OAuth / credentials)
    │
    ▼
  切换成功?
    ├── Yes → rewrite_prompt_model_identity() 改写系统提示词中的 Model/Provider 行
    │         → _sync_failover_system_message() 同步到当前 api_messages
    │         → 返回 True，重试循环使用新 Provider 重新发起 API 调用
    │
    └── No (链耗尽) → 设置 _FALLBACK_EXHAUSTED_COOLDOWN_S (5s) 冷却
                      → 返回 False → 循环终止，turn 标记为 failed

关键设计点：切换 Provider 时必须同步系统提示词中的 Model: / Provider: 身份声明行——否则用户会在同一对话中看到"我是 GPT-5"随即变成"我是 Claude"，产生困惑。这是 _sync_failover_system_message() (conversation_loop.py L492) 的职责。

故障切换中的另一条异常路径：内容策略拦截（content_policy_blocked）——当模型 HTTP 200 返回了 refusal 而非正常回复，或其他 Provider 返回了内容审核拒绝，_content_policy_blocked_result() (conversation_loop.py L467) 直接构建失败结果，不触发重试——因为相同 prompt 会被再次拒绝，无意义。

检索练习

  1. 暂停回忆：一条用户消息进入系统后，依次经过哪六个阶段？不翻看上文，在心中默念一遍。
  2. 判断题：每次 API 调用时，build_turn_context() 都会重新执行。
    点击查看答案
    错误。build_turn_context() 仅在 run_conversation() 开头执行一次（阶段 2），负责回合前置准备。后续每次 API 重试调用的是 build_api_kwargs()（阶段 4），它根据最新的 messages 列表重新组装请求参数。
  3. 判断题：Provider 故障切换时，try_activate_fallback() 总会立即切换到下一个 Provider，直到链耗尽。
    点击查看答案
    不完全正确。在切换前有三层过滤：(1) 跳过已标记为 unavailable 的条目；(2) 跳过与当前 provider+model+base_url 相同的条目（防止同 Provider 循环重试）；(3) 检查本地可用性（网络、OAuth、凭证）。只有当这三层过滤都通过，才会真正切换。
  4. 向 Agent 追问：如果你在调试时发现 api_call_count 始终为 1 但返回了最终文本，这说明什么？是否存在工具调用？
