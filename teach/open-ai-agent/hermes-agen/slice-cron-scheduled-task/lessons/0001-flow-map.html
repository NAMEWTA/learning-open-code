二、调度心跳与双线程池并行模型

tick() 每 60 秒由 InProcessCronScheduler 的守护线程触发。它不直接执行任务——而是将到期作业火后即忘（fire-and-forget）地提交到两个持久化线程池，立即返回，绝不阻塞心跳循环。

双线程池分区策略

作业按 workdir 字段分为两组，分别进入不同的线程池：

due_jobs │ ├─ workdir 有值 → sequential_pool (max_workers=1) │ 原因：workdir 作业通过 os.environ["TERMINAL_CWD"] │ 修改进程级全局状态，必须串行执行 │ └─ workdir 为空 → parallel_pool (max_workers 可配) 原因：无 workdir 的作业不修改全局状态，可安全并发 隔离手段：_terminal_cwd_lock (写优先读写锁)

关键代码——tick() 的分区调度核心（cron/scheduler.py:3489-3572）：

# 1. 按 workdir 分区
sequential_jobs = [j for j in due_jobs if (j.get("workdir") or "").strip()]
parallel_jobs   = [j for j in due_jobs if not (j.get("workdir") or "").strip()]

# 2. 顺序池：env 突变作业串行排队（不阻塞 ticker）
if sequential_jobs:
    seq_pool = _get_sequential_pool()  # 持久化，max_workers=1
    for job in sequential_jobs:
        fut = _submit_with_guard(job, seq_pool)

# 3. 并行池：无 workdir 作业并发执行
if parallel_jobs:
    pool = _get_parallel_pool(_max_workers)  # 持久化，可配并发度
    for job in parallel_jobs:
        fut = _submit_with_guard(job, pool)

# 4. 异步模式：fire-and-forget，不等待结果
#    sync 模式（测试/手动 tick）：concurrent.futures.as_completed() 等待全部

设计的核心洞察：两个线程池都是持久化的（模块级单例），跨 tick 复用。一个耗时 15 分钟的 fixer 作业不会阻塞后续 60 秒心跳中其他作业的调度——它运行在自己的线程里，而 ticker 早已返回等待下一秒。

此外，_submit_with_guard 通过 _running_job_ids 集合防止同一作业被重复提交——如果前一次 tick 的作业仍在运行，新 tick 会跳过它。

三、守护防线：双层阻断机制

两条防线在不同时间点阻断不同威胁。理解它们的差异是定位"作业被拒绝"问题的关键。

第 1 道防线：创建时 lifecycle_guard

在 create_job() 中调用 check_gateway_lifecycle(prompt, script)，用四分支正则匹配 shell 级 gateway 自杀命令（cron/lifecycle_guard.py:48-66）：

_GATEWAY_LIFECYCLE_PATTERN = re.compile(r"(?i)"
    # 分支 A: hermes gateway restart|stop
    r"(?:hermes\s+gateway\s+(?:restart|stop))"
    # 分支 B: launchctl kickstart|unload|... hermes[.-]?gateway
    r"|(?:launchctl\s+(?:kickstart|unload|load|stop|restart)"
    r"\b[^\n]*\bhermes[.\-]?gateway)"
    # 分支 C: systemctl restart|stop|start ... hermes[.-]?gateway
    r"|(?:systemctl\s+(?:-\S+\s+)*(?:restart|stop|start)"
    r"\b[^\n]*\bhermes[.\-]?gateway)"
    # 分支 D: pkill / kill 目标为 hermes gateway 进程
    r"|(?:p?kill\b[^\n]*\bhermes\b[^\n]*\bgateway)"
    r"|(?:p?kill\b[^\n]*\bgateway\b[^\n]*\bhermes)"
)
异常路径示例：如果 cron prompt 为 "每天晚上重启 gateway 以清理内存"，create_job() 会抛出 GatewayLifecycleBlocked，作业不会被创建。这防止了 #30719 描述的 SIGTERM-respawn 死循环——cron 触发 gateway restart → gateway 死亡 → supervisor 复活 gateway → 恢复会话 → 再次触发 restart。

第 2 道防线：运行时 prompt injection 扫描

在 _build_job_prompt() 拼装完 prompt + skill + script 输出后，调用 _scan_assembled_cron_prompt()。这填补了 #3968 漏洞：创建时只扫描用户 prompt，但 skill 内容在运行时才从磁盘加载，恶意 skill 可携带注入载荷抵达非交互式（自动审批工具调用）的 cron agent。

两套模式自适应选择：无 skill/数据注入时使用严格模式（阻断裸命令）；有注入内容时使用宽松模式（仅阻断明确的注入指令，不可见 Unicode 清洗而非阻断）。

投递四级回退链

执行完毕后的 _deliver_result() 并非简单发送——它有四级回退（cron/scheduler.py:1308-1876）：

# 级别 1：live adapter（网关运行时，支持 E2EE）
if runtime_adapter and loop.is_running():
    router._deliver_to_platform(target, text, metadata)
    # 失败 → 进入回退

# 级别 2：standalone asyncio.run()
asyncio.run(_send_to_platform(...))
# RuntimeError（已有运行中的 loop）→ 进入回退

# 级别 3：新线程中 asyncio.run()
pool.submit(asyncio.run, _send_to_platform(...))
# 解释器关闭中 → 进入回退

# 级别 4：优雅跳过
if _interpreter_shutting_down():
    logger.warning("delivery skipped — interpreter shutting down")

这个回退链确保：网关正常时走加密通道（E2EE 可行）；网关未运行时走 HTTP API；极端情况下优雅降级而非崩溃。
