插件系统模块总览 L1 模块课程 -- 插件来源、形态分类、注册机制与生命周期

学习目标：理解 Hermes 插件系统的四源五态架构，掌握 PluginManager 加载链路和 PluginContext 注册模式 | 预估时长：15 分钟 | 前置知识：L0 项目全局地图 | 关联参考：插件系统速查手册

一、定位：Agent 的能力扩展中枢

plugins/ 是 Hermes Agent 的功能扩展层——所有非核心能力（浏览器自动化、记忆增强、图像生成、看板管理、平台适配等）都以插件形式存在。核心 Agent 框架只保留对话循环、工具注册表、Gateway 等基础能力，其余全部交由插件注入。
177 个 .py 文件 8.1 MB 源码 18 个子目录 4 种来源 5 种形态 20+ Hook

在 L0 总览中我们提到 Hermes 采用 微内核 + 插件 架构。这里的"微内核"即 agent/ + tools/ + gateway/ 的核心框架，而 plugins/ 目录承载了所有可选的功能模块。插件通过统一的 register(ctx) 入口和 PluginContext 外观向核心框架注册能力——这一模式确保核心不依赖任何具体插件，插件之间也互不耦合。

二、四种来源与五种形态

四种插件来源（后加载者覆盖先加载者）：

1. bundled  —  <repo>/plugins/<name>/     随 hermes-agent 发布，零配置可用
2. user     —  ~/.hermes/plugins/<name>/   用户自行安装，优先级高于 bundled
3. project  —  ./.hermes/plugins/<name>/   项目级插件（需 HERMES_ENABLE_PROJECT_PLUGINS=1）
4. pip      —  importlib.metadata entry_points     pip 安装的 Python 包插件

五种插件形态（决定加载策略）：

形态加载策略典型目录
standalone显式 opt-in（plugins.enabled）disk-cleanup/、observability/
backendbundled 自动加载，其余需 opt-inimage_gen/openai/、browser/browserbase/
exclusive由专属发现系统加载，PluginManager 跳过memory/honcho/、memory/openviking/
platformbundled 延迟加载（首次使用时才 import）platforms/telegram/、platforms/feishu/
model-provider由 providers/__init__.py 发现，PluginManager 跳过model-providers/openai/（若存在）

关键差异：standalone 是纯 opt-in——必须在 plugins.enabled 中显式列出才加载；backend 的 bundled 变体自动加载，确保图像生成、浏览器等内置能力开箱即用；exclusive 和 model-provider 完全脱离 PluginManager 的通用加载流程，由各自的专用发现系统管理。

三、注册机制：register(ctx) + PluginContext

每个插件目录必须包含 plugin.yaml 清单和 __init__.py（含 register(ctx) 函数）。PluginManager 加载插件后，实例化一个 PluginContext 对象作为 ctx 传给 register()。

3.1 典型的 backend 插件注册（以 browser_use 为例）

# plugins/browser/browser_use/__init__.py
from plugins.browser.browser_use.provider import BrowserUseBrowserProvider

def register(ctx) -> None:
    """向核心框架注册 Browser Use 云浏览器后端。"""
    ctx.register_browser_provider(BrowserUseBrowserProvider())

3.2 典型的 standalone 插件注册（以 disk-cleanup 为例）

# plugins/disk-cleanup/__init__.py（简化表示）
def register(ctx) -> None:
    """注册 post_tool_call + on_session_end 钩子追踪临时文件。"""
    ctx.register_hook("post_tool_call", _on_write_tool_result)
    ctx.register_hook("on_session_end", _cleanup_ephemeral_files)
    # 同时注册一个 slash 命令供手动管理
    ctx.register_command("disk-cleanup", _disk_cleanup_command,
                         description="手动清理会话临时文件")

PluginContext 提供了 12+ 种注册方法，覆盖了 Hermes 几乎所有的扩展点：工具（register_tool）、Hook（register_hook）、Provider 后端（register_browser_provider / register_image_gen_provider / register_web_search_provider 等）、CLI 命令（register_cli_command）、Slash 命令（register_command）、平台适配器（register_platform）、Skill（register_skill）等。详见插件系统速查手册的完整清单。

四、生命周期 Hook 系统

Hermes 定义了 20+ 个生命周期 Hook，PluginManager 使用统一的 invoke_hook(name, **kwargs) 接口触发。每个 Hook 回调独立执行，异常被捕获隔离——一个插件的崩溃不会影响 Agent 核心或其他插件。

# PluginManager.invoke_hook() 的隔离执行策略
def invoke_hook(self, hook_name: str, **kwargs):
    callbacks = self._hooks.get(hook_name, [])
    results = []
    for cb in callbacks:
        try:
            ret = cb(**kwargs)
            if ret is not None:
                results.append(ret)
        except Exception as exc:
            logger.warning("Hook '%s' callback raised: %s", hook_name, exc)
    return results

关键 Hook 一览：

Hook触发时机返回值用途
pre_tool_call工具调用前返回 {"action":"block","message":"..."} 可阻止执行
post_tool_call工具调用后观察者模式，返回值被忽略
pre_llm_callLLM 调用前可注入上下文文本到用户消息
transform_llm_outputLLM 输出后返回字符串替换响应文本
pre_verifyAgent 编辑代码后准备停止返回 {"action":"continue","message":"..."} 可继续执行
on_session_start会话开始观察者模式
on_session_end会话结束观察者模式，常用于资源清理
kanban_task_completed看板任务完成观察者模式

点击查看参考答案

1. bundled（随 repo 发布）→ user（~/.hermes/plugins/）→ project（./.hermes/plugins/）→ pip（entry_point）。后者覆盖前者同名插件。注意 project 插件需要 HERMES_ENABLE_PROJECT_PLUGINS=1 环境变量才能启用。

2. standalone 必须由用户在 plugins.enabled 中显式 opt-in；backend 的 bundled 变体自动加载（零配置可用），但 user/project/entry-point 来源的 backend 仍需 opt-in。这是为了使内置的图像生成、浏览器等后端能力开箱即用。

3. 应注册 post_tool_call Hook，在回调中检查工具名是否为 write_file，然后对写入的文件执行格式化。使用 ctx.register_hook("post_tool_call", your_callback) 注册。可参考 disk-cleanup 插件的实现模式。
