一、在整体架构中的位置

acp_adapter/ 是 Hermes Agent 与编辑器之间的协议桥梁。它通过 stdio 上的 JSON-RPC 传输，将内部 AIAgent 暴露为符合 ACP 标准的 Agent，使 Zed、Codex、Claude Code 等编辑器能直接调用 Hermes Agent 进行对话、执行工具和管理会话。
11 个 .py 文件 ~240 KB acp Python SDK stdio JSON-RPC

ACP 适配器在 Hermes Agent 分层架构中位于网关层与应用层之间：

  编辑器客户端 (Zed / Codex / Claude Code)
      │ ACP stdio JSON-RPC (stdout)
      ▼
  ┌───────────────────────────────────────────────────┐
  │              ★ acp_adapter/                       │
  │  entry.py ──► server.py (HermesACPAgent)          │
  │                  │                                │
  │       ┌──────────┼──────────┐                     │
  │       ▼          ▼          ▼                     │
  │  session.py  events.py  tools.py                  │
  │  (会话持久化)  (事件桥接)   (工具映射)               │
  │                                                  │
  │  permissions.py  edit_approval.py  auth.py        │
  │  (危险命令审批)    (编辑前置审批)     (认证方法)      │
  └───────────────────────────────────────────────────┘
      │ AIAgent.run_conversation() (ThreadPoolExecutor)
      ▼
  agent/ + tools/ + providers/  (Hermes 核心引擎)

核心设计原则：stdout 专用于 ACP JSON-RPC 帧，stderr 用于日志输出。entry.py 启动时将全部日志路由到 stderr，并通过 _BenignProbeMethodFilter 抑制健康检查探针产生的噪音回溯。

二、入口与服务器启动

entry.py 的 main() 函数是适配器入口，支持四种运行模式：--version、--check（验证 ACP 依赖）、--setup（交互式配置提供商）、以及默认的 ACP 服务器模式。以下为启动链路的核心代码：

# entry.py — 启动 ACP 服务器（精简版）
def main(argv=None):
    _setup_logging()          # 路由全部日志到 stderr
    _load_env()               # 加载 ~/.hermes/.env

    project_root = str(Path(__file__).resolve().parent.parent)
    if project_root not in sys.path:
        sys.path.insert(0, project_root)

    import acp
    from .server import HermesACPAgent

    agent = HermesACPAgent()
    asyncio.run(acp.run_agent(agent, use_unstable_protocol=True))
    # acp.run_agent 创建 stdio transport，监听 stdin 上的 JSON-RPC 请求

HermesACPAgent 继承 acp.Agent，实现了 8 个关键方法：initialize（握手协商能力集）、authenticate（凭证验证）、new_session/load_session/resume_session（会话生命周期）、prompt（核心交互）、cancel（中断运行）、fork_session（会话分叉）、list_sessions（会话列表）。

三、会话管理与事件桥接

SessionManager（session.py）管理 ACP 会话与 AIAgent 实例的一对一映射。每次 create_session 创建一个带 UUID 的 SessionState，内部持有 AIAgent 实例、对话历史、取消事件和运行时锁。会话通过 SessionDB（~/.hermes/state.db）持久化，进程重启后 get_session 自动从数据库恢复。

AIAgent 在 ThreadPoolExecutor(max_workers=4) 中同步运行，而 ACP 客户端通过事件循环通信。events.py 提供了四个回调工厂函数，通过 asyncio.run_coroutine_threadsafe 实现线程安全的通知转发：

回调工厂AIAgent 触发ACP 通知
make_tool_progress_cb工具开始执行ToolCallStart
make_step_cb工具执行完成ToolCallProgress（含 plan 更新）
make_thinking_cb模型推理输出AgentThoughtChunk
make_message_cb回复文本流AgentMessageChunk

每个回调内部使用 _send_update 辅助函数，通过 safe_schedule_threadsafe 将协程调度到主事件循环，确保工作线程不阻塞 ACP 传输。

四、工具映射与权限审批

tools.py 维护了 TOOL_KIND_MAP 字典，将 Hermes 工具名映射到 ACP 的六种 ToolKind：

# tools.py — Hermes 工具到 ACP ToolKind 的映射（核心条目）
TOOL_KIND_MAP = {
    "read_file": "read",      "write_file": "edit",
    "patch": "edit",          "search_files": "search",
    "terminal": "execute",    "process": "execute",
    "web_search": "fetch",    "web_extract": "fetch",
    "todo": "other",          "_thinking": "think",
    # ... 共 30+ 个映射
}

permissions.py 将 Hermes 的危险命令审批（once / session / always / deny）桥接到 ACP 的 PermissionOption 体系，通过 request_permission 协程在编辑器中弹出审批对话框。edit_approval.py 更进一步，通过 ContextVar 将编辑审批请求器绑定到当前 ACP 会话上下文，支持三种编辑审批策略：ask（每次询问）、workspace_session（自动批准工作区编辑）、session（全部自动批准，敏感路径除外）。

检索练习

尝试在脑中回答以下问题，然后展开查看参考答案：

1. acp_adapter 为什么要把日志路由到 stderr 而不是 stdout？
ACP 协议使用 stdio transport，stdout 专用于 JSON-RPC 协议帧。如果日志输出到 stdout，会破坏协议帧的解析，导致客户端无法正确读取响应。stderr 不受此限制，可安全用于日志。
2. SessionManager 如何支持进程重启后恢复会话？
SessionManager 通过 SessionDB（~/.hermes/state.db）持久化每个会话的模型配置、对话历史和工作目录。当 get_session() 在内存中找不到会话时，自动调用 _restore() 从数据库加载历史记录并重建 AIAgent 实例。
3. 为什么事件桥接需要 asyncio.run_coroutine_threadsafe？
AIAgent 在 ThreadPoolExecutor 工作线程中同步运行，而 ACP 客户端连接运行在主线程的事件循环上。直接用工作线程调用协程会导致事件循环线程安全问题。run_coroutine_threadsafe 将协程安全地调度到正确的事件循环中执行。
