技能生命周期全链路 L2 垂直切片 — 从任务完成到技能创建到自改进

学习目标：掌握技能从诞生到归档的 6 阶段全链路 | 预估时长：15 分钟 | 前置课程：L1 技能系统模块巡礼、 L1 Agent 核心引擎

核心问题：一个对话中 agent 自动创建了技能，7 天后策展器如何决定它的命运？本课从 skill_provenance.py 的 79 行起点出发，纵穿 curator 状态机、usage 遥测、归档保护，打通技能生命周期全链路。

一、触发入口：谁在写技能？

技能生命周期始于一次写入操作，但写入者的身份决定了技能后续是否被 curator 管理。这一分界由 tools/skill_provenance.py 实现——整个链路中最短（79 行）却最关键的文件。

# tools/skill_provenance.py (核心片段)
_write_origin: contextvars.ContextVar[str] = contextvars.ContextVar(
    "skill_write_origin",
    default="foreground",      # 默认：用户直接操作
)

BACKGROUND_REVIEW = "background_review"

def is_background_review() -> bool:
    return get_current_write_origin() == BACKGROUND_REVIEW

# 在 curator fork 中设置（agent/curator.py:1905）：
review_agent._memory_write_origin = "background_review"

当 skill_manage(action='create') 被调用时，tool handler 检查 is_background_review()：

写入来源ContextVar 值技能标记curator 行为
用户对话 / CLI / gatewayforeground用户技能永不触碰

设计意图：用户明确要求创建技能 = 用户财产，curator 无权处置。只有 agent 自改进过程中自动生成的技能才进入策展流程。这是整个系统的安全边界。

二、策展器激活：三重门控

curator 不会在每次对话后立即运行。入口 maybe_run_curator()（curator.py:1958）由 session-start hook 在 agent 空闲时调用，经过三重门控：

# agent/curator.py — should_run_now() 门控逻辑（简化）
def should_run_now(state, config) -> bool:
    if not config.curator_enabled:       # 门控 1：总开关
        return False
    if state.paused:                     # 门控 2：用户暂停
        return False
    if idle_seconds < min_idle_hours * 3600:  # 门控 3：空闲不足
        return False
    if hours_since_last_run < interval_hours: # 门控 4：间隔未到
        return False
    return True

# 默认值：interval_hours=168 (7天), min_idle_hours=2

首次安装后，curator 不会立即运行——它会将 last_run_at 种子化为当前时间，延期一个完整 interval，防止 hermes update 后的第一次后台 tick 就自动 curate。

三、确定性状态机：五层保护

curator 运行分两阶段：Stage 1 是纯确定性状态机（无需 LLM），Stage 2 是 LLM consolidate（默认关闭）。Stage 1 的核心是 apply_automatic_transitions()，它遍历所有 curator-managed 技能，按以下规则流转：

# agent/curator.py — apply_automatic_transitions() 核心循环（简化）
for skill in curator_managed_skills:
    # 保护层 1：pinned 技能永不自动转换
    if skill.pinned: continue
    # 保护层 2：cron job 引用的技能视为 "in use"
    if is_cron_referenced(skill.name): continue
    # 保护层 3：PROTECTED_BUILTIN_SKILLS（如 "plan"）
    if skill.name in PROTECTED_BUILTIN_SKILLS: continue

    days_inactive = (now - skill.last_activity).days
    if days_inactive >= 90:               # 归档阈值
        archive_skill(skill.name)         # → 移到 .archive/
    elif days_inactive >= 30:             # 陈旧阈值
        set_state(skill.name, STALE)      # → 标记陈旧
    elif skill.state == STALE:
        set_state(skill.name, ACTIVE)     # → 重新激活

状态机保护机制共五层：

保护层触发条件行为
1. Pinnedskill.pinned == True跳过所有自动转换
2. Cron 引用cron job 依赖此技能视为活跃使用中
3. PROTECTED_BUILTIN_SKILLS技能名为 "plan"永不归档、永不合并
4. is_agent_created 检查非 agent 创建的技能根本不进入遍历列表
5. Pre-mutation Snapshotcurator 运行前完整备份技能库
异常路径示例：假设 curator 尝试归档 plan 技能——它首先被第 3 层拦截（PROTECTED_BUILTIN_SKILLS 集合包含 "plan"），即使用户未 pin、即使 prune_builtins=True，也不可归档。否则 /plan slash command 将变成 "Unknown command"，这是一个 load-bearing 保护。

  1. 填空题：ContextVar _write_origin 的默认值是 ______，curator fork 中设置为 ______。
  2. 判断题：用户通过 CLI 对话让 agent 创建的技能，7 天后会被 curator 归档。
  3. 场景题：一个 agent 自建技能 35 天未使用，curator 运行后它处于什么状态？如果又过了 60 天呢？
点击查看参考答案

1. "foreground"；"background_review"

2. 错误。用户前台创建的技能不会被 mark_agent_created()，curator 的遍历列表只包含 agent-created 技能，用户技能不受影响。

3. 35 天 → STALE（超过 30 天阈值）；再 60 天（累计 95 天）→ ARCHIVED（超过 90 天阈值，移至 .archive/）。
