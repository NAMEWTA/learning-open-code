一、入口点：消息从哪里来

每条消息的旅程始于一个外部平台用户在 Telegram、Discord 等聊天工具中发送一条文本。平台 API 将这条消息推送到 Hermes Agent 的网关层，由适配器接收并标准化为 MessageEvent。
24 平台 MessageEvent SessionSource

入口的核心数据结构是两个 dataclass：

数据结构位置职责
MessageEvent base.py:1716 标准化入站消息：text, message_type, media_urls, reply_to_message_id
SessionSource session.py:147 消息来源上下文：platform, chat_id, thread_id, user_id, scope_id

两种接收模式：

  1. 直连模式 — 适配器直接连接平台 API（如 Telegram Bot API 通过长轮询拉取更新），内置于 base.py 各子类中
  2. 中继模式 — 适配器通过 WebSocket 连接 Team Gateway Connector，由 connector 统一对接多平台，见 relay/ws_transport.py

二、五层穿线：消息走过的每一站

消息在网关内部依次经过五层。下方的时序图按调用顺序列出了每一层的关键方法，从上到下即为消息流动的时间线。
外部用户（Telegram/Discord） │ ▼ ① 平台适配层 BasePlatformAdapter.connect() │ 轮询/WS 接收 → 构建 MessageEvent │ set_message_handler() 注册回调 ▼ ② 中继路由层 RelayAdapter.connect() → RelayTransport.connect() │ WS handshake → descriptor 能力协商 │ _on_inbound() 将 inbound 帧转交 handler ▼ ③ Agent 处理 message_handler(event) → run_agent() │ conversation_loop 消费消息 → LLM 生成响应 ▼ ④ 投递路由层 DeliveryRouter.deliver(content, targets) │ 解析投递目标 → 选择适配器 → _deliver_to_platform() ▼ ⑤ 平台发送层 adapter.send(chat_id, content, metadata) 平台 API 调用 → 消息送达用户

关键代码片段：消息从接收到投递的交界面。 以下展示了 deliver() 中如何将 Agent 响应发回平台：

# gateway/delivery.py — DeliveryRouter.deliver() 核心循环
async def deliver(self, content, targets, job_id=None, ...):
    results = {}
    for target in targets:
        # 跳过已知死目标（被踢出群、用户注销）
        if target.chat_id and self.dead_targets.is_dead(
            target.platform.value, target.chat_id
        ):
            results[target.to_string()] = {
                "success": False, "skipped": "dead_target"
            }
            continue
        if target.platform == Platform.LOCAL:
            result = self._deliver_local(content, job_id, ...)
        else:
            result = await self._deliver_to_platform(
                target, content, metadata
            )

上例中 _deliver_to_platform() 会先检查超大内容截断（MAX_PLATFORM_OUTPUT = 4000），过滤空响应防循环（_is_silence_narration()），再调用 adapter.send() 完成真正的平台投递。

# gateway/delivery.py — _deliver_to_platform() 关键防护
# 超大内容：先保存完整文件，再截断加脚注
if len(content) > MAX_PLATFORM_OUTPUT:
    saved_path = self._save_full_output(content, job_id)
    footer = f"\n\n... [truncated, full output saved to {saved_path}]"
    content = content[:4000 - len(footer)] + footer

# 防循环：丢弃空/静默响应（*(silent)*, 🔇, "."）
if self._filter_silence_narration_enabled() and _is_silence_narration(content):
    return {"success": True, "filtered": "silence_narration"}

# 真正发送
result = await adapter.send(target.chat_id, content, metadata=send_metadata)

三、异常路径：投递失败的四级回退

当消息无法投递到指定目标时（群组被删除、bot 被踢出、用户注销），DeliveryRouter 不会让消息丢失，而是依次尝试四级回退策略，确保至少有一份输出留存。

DeliveryTarget 解析的四级目标格式：

# gateway/delivery.py — DeliveryTarget.parse() 四级目标解析
# 格式 1："telegram:123456"     → 指定平台+聊天ID（显式目标）
# 格式 2："telegram"            → 仅平台名，使用 Home Channel
# 格式 3："origin"              → 回到消息来源会话
# 格式 4："local"               → 仅保存本地文件，不推送到任何平台

@classmethod
def parse(cls, target: str, origin=None) -> "DeliveryTarget":
    if target_lower == "origin":
        return cls(platform=origin.platform, chat_id=origin.chat_id,
                   thread_id=origin.thread_id, is_origin=True)
    if target_lower == "local":
        return cls(platform=Platform.LOCAL)
    if ":" in target_stripped:
        platform_str, chat_id, thread_id = ...  # 解析 platform:chat:thread
        return cls(platform=Platform(platform_str), chat_id=chat_id,
                   thread_id=thread_id, is_explicit=True)
    # 仅平台名 → 使用 Home Channel
    return cls(platform=Platform(target_lower))

投递失败处理链条：

  1. 死目标跳过 — DeadTargetRegistry.is_dead() 记录已知不可达目标（群解散、bot 被踢），直接标记 skipped 而不浪费 API 调用
  2. 发送异常捕获 — _deliver_to_platform 抛出的异常被外层 try/except 捕获，并调用 _classify_dead_from_error_text() 判断是否应将此目标标记为死目标
  3. Telegram DM Topic 恢复 — 当检测到 thread not found 错误时，自动调用 ensure_dm_topic(force_create=True) 重建已删除的私聊主题，然后重试发送
  4. 本地兜底 — 当所有平台投递失败时，可通过将 target 设为 "local" 确保内容写入 ~/.hermes/cron/output/ 磁盘文件
