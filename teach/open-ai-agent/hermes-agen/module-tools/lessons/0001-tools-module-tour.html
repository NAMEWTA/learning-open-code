一、一句话定位

tools/ 是 Hermes Agent 的"手和眼"——92 个 Python 文件、4.4MB 代码量，涵盖终端命令执行、浏览器自动化、子代理委托、命令审批、技能管理等完整工具生命周期。Agent 核心通过 tool_executor.py 调用本层，本层通过 providers/ 回写模型结果。
92 个 .py 文件 自注册机制 六种终端后端 三层安全闸门 子代理委托

二、分层架构与自注册机制

工具层采用自注册 + 自动发现架构。每个工具模块在模块级别调用 registry.register() 声明自身，discover_builtin_tools() 通过 AST 扫描自动导入所有工具模块。以下是核心骨架：

tools/ ├── __init__.py ← 包命名空间，最小化导入 ├── registry.py ← 工具注册中心（核心枢纽） ├── blueprints.py ← 蓝图→cron 桥接 │ ├── terminal_tool.py ← 终端命令执行（132KB，最大子模块） ├── browser_tool.py ← 浏览器自动化（201KB） ├── browser_cdp_tool.py ← Chrome DevTools 协议 ├── browser_camofox.py ← Camofox 浏览器集成 │ ├── delegate_tool.py ← 子代理委托（148KB） ├── async_delegation.py ← 异步后台委托 │ ├── approval.py ← 命令审批系统（130KB，安全闸门） ├── write_approval.py ← 写文件审批 ├── slash_confirm.py ← 斜杠命令确认 │ ├── file_tools.py ← 文件操作（读写编辑搜索，98KB） ├── mcp_tool.py ← MCP 协议工具（225KB，最大单文件） ├── send_message_tool.py ← 跨平台消息发送 ├── skills_hub.py ← 技能中心（154KB） └── ...（共 92 个 .py 文件）

注册机制

每个工具通过 registry.register() 声明以下元数据：

# 示例：terminal_tool.py 中的自注册调用
from tools import registry

registry.register(
    name="terminal_tool",           # 工具名，模型看到的 function name
    toolset="terminal",             # 所属工具集
    schema={...},                   # OpenAI function-calling JSON Schema
    handler=terminal_tool,          # 实际执行函数
    check_fn=check_terminal_requirements,  # 可用性检测回调
    requires_env=[],                # 所需环境变量
    is_async=False,                 # 是否异步工具
    description="Execute a shell command",
    emoji=">_",
)

discover_builtin_tools() 通过 AST 扫描每个 .py 文件，检测顶层 registry.register(...) 调用后自动导入。Check_fn 带有 30 秒 TTL 缓存和瞬态故障容忍机制——单次探测超时不会导致工具静默丢失。

三、四大核心子系统

3.1 终端工具 — terminal_tool.py

支持 六种执行后端：local（直接宿主机执行）、Docker（容器隔离）、Modal（云端沙箱）、SSH（远程主机）、Singularity（HPC 容器）、Daytona（开发环境）。通过统一的 terminal_tool(command, background=False) 接口调用，后端选择对 Agent 透明。支持后台任务、中断信号传播、输出截断和进程注册表追踪。

3.2 浏览器工具 — browser_tool.py

基于 agent-browser CLI（Node.js 进程），支持三种模式：Browser Use 云（Nous 订阅用户默认）、Browserbase 云（直接凭据）、本地 Chromium（零成本）。通过可访问性树（ariaSnapshot）为无视觉 LLM 提供文本化页面表示，使用 @e1、@e2 等引用选择器定位元素。会话按 task_id 隔离，自动清理。

3.3 委托工具 — delegate_tool.py

生成独立 AIAgent 子实例，每个子代理拥有：全新对话上下文（不继承父历史）、独立 task_id（独立终端会话和文件缓存）、受限工具集（DELEGATE_BLOCKED_TOOLS 禁止递归委托、用户交互、记忆写入等）。支持单任务和批量并行模式，父代理仅看到调用和摘要结果。异步委托（async_delegation.py）通过 daemon 线程池后台运行，完成后通过 process_registry.completion_queue 回传结果。

3.4 审批系统 — approval.py

三层安全闸门：硬线阻断（无条件拒绝 rm -rf /、shutdown 等不可逆操作，即使 YOLO 模式也不放行）、危险模式检测（正则匹配 50+ 危险命令模式，覆盖敏感文件写入、系统配置修改、凭据泄露等）、交互式/智能审批（CLI 下弹出确认提示，gateway 下通过异步队列审批，可选 LLM 辅助判断低风险命令自动放行）。支持永久允许列表持久化到 config.yaml。

关键依赖：工具层依赖 agent/tool_executor（调用入口）、providers/（模型回写）、hermes_cli/config（配置读取）、gateway/session_context（审批会话上下文）。工具层本身不依赖 UI 层，可独立测试。

  2. 委托工具（delegate_tool）禁止子代理使用哪些工具？为什么？
  3. 审批系统的三层安全闸门分别是什么？YOLO 模式下哪一层仍然生效？
点击查看参考答案

1. 自注册机制，每个工具模块在模块级别调用 registry.register()。discover_builtin_tools() 通过 AST 扫描 tools/ 目录下所有 .py 文件，检测顶层 registry.register(...) 调用后自动导入。

2. DELEGATE_BLOCKED_TOOLS 禁止：delegate_task（防递归）、clarify（无用户交互）、memory（不写共享 MEMORY.md）、send_message（无跨平台副作用）、execute_code（应逐步推理而非写脚本）、cronjob（不以父代理名义调度）。防止子代理越权和上下文污染。

3. 硬线阻断（无条件拒绝不可逆操作，YOLO 也生效）→ 危险模式检测（正则匹配 50+ 模式）→ 交互式/智能审批（CLI 弹出确认 / LLM 辅助自动审批）。
