一、模块概览：四个核心文件 + 三个辅助层

cron/ 是 Hermes Agent 的定时任务调度引擎——它不依赖操作系统 cron daemon，而是通过 gateway 守护进程每 60 秒 tick 一次，检查 jobs.json 中的到期任务，在隔离的 agent 会话中执行，并将结果投递到用户指定的聊天平台。
10 个 .py 文件 (344 KB) croniter 条件导入 19+ 投递平台 per-profile 隔离

模块文件按职责分为三层——调度核心（scheduler + jobs）、安全防护（lifecycle_guard）、用户体验（blueprint + suggestion）：

  cron/
  ├── __init__.py              ← 公开 API 导出（create/update/remove/tick 等）
  ├── scheduler.py          ★  ← tick() 调度循环 + run_job() agent 生成 + 投递
  ├── jobs.py               ★  ← jobs.json 存储 CRUD + 文件锁 + 到期判断
  ├── scheduler_provider.py ★  ← CronScheduler ABC + InProcessCronScheduler
  │
  ├── lifecycle_guard.py    ◆  ← 网关生命周期命令阻断（防自毁循环 #30719）
  │
  ├── blueprint_catalog.py  ▵  ← 参数化自动化蓝图（表单 → slash cmd → cron job）
  ├── suggestions.py        ▵  ← 建议式注册（consent-first，dedup_key 防重复）
  ├── suggestion_catalog.py ▵  ← 精选入门自动化目录（每日简报、邮件监控等）
  │
  └── scripts/
      └── classify_items.py     ← 紧急程度分类器脚本

关键设计决策：cron 是 per-profile 隔离的（issue #4707）。每个 profile 拥有独立的 ~/.hermes/profiles/<name>/cron/jobs.json，profile-scoped gateway 只运行本 profile 的定时任务——不同 profile 的 jobs、skills、credentials 不会互相泄漏。

二、调度循环：tick → get_due_jobs → run_job

Gateway 守护进程以 InProcessCronScheduler.start() 启动 60 秒定时器，核心在 tick() 函数中完成三个步骤：

  gateway daemon (每 60s)
         │
         ▼
  tick()                                    ← cron/scheduler.py L3399
    ├── 获取文件锁 (.tick.lock)              ← 跨进程互斥，防重复 tick
    ├── get_due_jobs()                       ← cron/jobs.py: 读取 jobs.json
    ├── advance_next_run()                   ← 先推进 next_run_at（at-most-once）
    └── 分发到线程池
         ├── 并行池 (parallel pool)           ← 普通 job 并发执行
         └── 顺序池 (sequential pool, ×1)     ← workdir job 串行执行
              │
              ▼
         run_one_job()                       ← cron/scheduler.py L3252
              ├── run_job()                   ← 生成 AIAgent → 执行 prompt → 收集输出
              ├── save_job_output()            ← 写入 ~/.hermes/cron/output/
              ├── 判断 [SILENT]                ← 静默标记则跳过投递
              └── _deliver_result()            ← 投递到目标平台

调度器使用 双线程池设计：普通任务在并行池中并发执行（通过 HERMES_CRON_MAX_PARALLEL 控制并发度），设置了 workdir 的任务则在单线程顺序池中串行执行——因为 workdir 任务会修改进程级 os.environ["TERMINAL_CWD"]，必须通过 _ReadWriteLock 保证不与任何其他任务重叠。

三、croniter 集成与自然语言定时

Hermes 通过可选依赖 croniter 库支持两种定时表达方式：

# 方式一：标准 cron 表达式（5 字段）
"schedule": "0 8 * * *"           # 每天 8:00
"schedule": "0 18 * * 0"          # 每周日 18:00
"schedule": "0 9 * * 1-5"         # 工作日 9:00

# 方式二：自然语言间隔描述（croniter 扩展语法）
"schedule": "every 30m"           # 每 30 分钟
"schedule": "every 2h"            # 每小时 第 0、30 分钟
"schedule": "every 6 hours"       # 每 6 小时
"schedule": "every day at 8am"    # 每天早 8 点

在 jobs.py 顶部通过 try: from croniter import croniter 做条件导入，若 croniter 未安装则回退到只支持标准 cron 表达式。Blueprint 系统将用户填入的时间/星期/间隔参数自动拼装为 cron 表达式，终端用户无需手写 cron 语法。

到期判断核心逻辑位于 get_due_jobs()（jobs.py）：遍历 jobs.json 中所有 active 状态的 job，用 croniter 计算每个 job 的 next_run_at，与当前时间比对，筛选出到期任务返回给 tick 调度。

四、多平台投递与安全防护

deliver 字段决定 cron 执行结果发送到哪里，支持五种取值模式：

deliver 值行为示例
local仅保存本地 output 目录，不投递数据采集类任务
origin回投到创建 job 时的原始聊天用户在 Telegram 中创建的日报
telegram投递到该平台配置的 home channel通过 TELEGRAM_HOME_CHANNEL 环境变量
telegram:-1001显式指定平台:chat_id精确投递到指定群组
telegram,discord逗号分隔的多平台广播同时投递到 Telegram + Discord

平台列表包含 19+ 个 (Telegram、Discord、Slack、WhatsApp、Signal、Matrix、Mattermost、钉钉、飞书、企业微信、QQ Bot、短信、邮件 等)。插件平台通过 PlatformEntry.cron_deliver_env_var 注册后，自动加入可用投递目标。

安全防护有两个层次：

  • lifecycle_guard.py——在 create_job 时扫描 prompt + script 内容，正则匹配 hermes gateway restart|stop、launchctl、systemctl、pkill 四种命令分支，阻止创建会导致 gateway 自毁循环的定时任务（#30719）。
  • assembled-prompt scanning——在 run_job 时扫描 runtime 组装后的完整 prompt（含 skill 内容），堵住创建时只扫描用户输入、skill 内容可能夹带 injection payload 的缺口（#3968）。

检索练习

  1. 如果用户在 Telegram 中创建了一个 deliver=origin 的定时任务，但 job 创建后又通过 API 修改了 origin 字段为 null，任务执行时结果会投递到哪里？
  2. 为什么 workdir 任务必须在单线程顺序池中执行，而不能扔进并行池？
  3. [SILENT] 标记有几种可识别的变体形式？
点击查看参考答案

1. 回退到各平台 home channel。_resolve_single_delivery_target() 在 origin 缺失时会遍历 _iter_home_target_platforms()，依次尝试各平台已配置的 home channel（通过 _get_home_target_chat_id() 从环境变量获取），找到第一个命中即投递。找不到时直接丢弃。

2. workdir 任务修改进程级 os.environ["TERMINAL_CWD"]。该环境变量控制所有 agent 工具的工作目录。如果两个 workdir 任务并发执行，一个任务会拾取另一个任务的 workdir，导致命令在错误目录执行。单线程顺序池 + _ReadWriteLock（写者优先）保证互斥。

3. 四种变体：（a）整个响应等于一个 silence token；（b）token 出现在首行或末行独立成行；（c）[SILENT] 作为同行的前缀（如 "[SILENT] No changes detected"）；（d）_CRON_SILENCE_TOKENS = frozenset({"[SILENT]", "SILENT", "NO_REPLY", "NO REPLY"})，大小写不敏感。
