5. 核心设计原则

5.1 自进化学习闭环

Curator（agent/curator.py）在复杂任务完成后自动评估是否创建新技能。技能存储在 ~/.hermes/skills/，在后续对话中可用。技能在被调用时收集反馈并自我改进。跨会话记忆通过 Honcho 辩证用户建模持久化。符合 agentskills.io 开放标准。

5.2 窄腰架构 + Footprint Ladder

每个模型工具随每次 API 调用发送——新增核心工具即为永久成本。能力扩展遵循足迹阶梯：

  1. 扩展现有代码 — 零新增表面
  2. CLI 命令 + 技能 — 零模型工具足迹（默认选择）
  3. 服务门控工具（check_fn） — 仅当前置条件满足时出现
  4. 插件 — 第三方/小众能力，运行时发现
  5. MCP 服务器 — 结构化 I/O 但非核心必需
  6. 新核心工具 — 最后手段，需充分论证

5.3 跨平台统一核心

同一套 agent/ 核心代码服务于所有入口：CLI、TUI、Web Dashboard、Electron Desktop、Telegram、Discord、Slack、WhatsApp、Signal 等 20+ 消息平台。平台接入通过 gateway/platforms/ 下的适配器实现，核心无需感知平台差异。

5.4 子代理并行

tools/delegate_tool.py 支持派生隔离子代理执行并行工作流，每个子代理拥有独立上下文窗口。结果通过结构化协议回传。agent/auxiliary_client.py (315 KB) 管理子代理的生命周期。

5.5 精确依赖锁定

所有核心依赖固定到精确版本（如 openai==2.24.0），不使用范围声明。2026 年 Mini Shai-Hulud 蠕虫事件后强制执行——若某 mistralai 版本被 PyPI 隔离，精确锁定可确保已安装用户不受影响。

5.6 Prompt 缓存神圣不可侵犯

长对话每轮复用缓存前缀。任何中对话中途修改上下文、切换工具集、重建系统提示的行为都会使缓存失效并大幅增加用户成本——这是被严格禁止的。唯一的例外是上下文压缩。
