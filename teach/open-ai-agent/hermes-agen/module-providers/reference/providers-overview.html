2. 注册中心 API

函数签名说明
register_provider(profile: ProviderProfile) -> None注册 profile 到 _REGISTRY[name] 和 _ALIASES[alias]=name。后注册覆盖先注册。
get_provider_profile(name: str) -> ProviderProfile | None按名称或别名查找。首次调用触发 _discover_providers()。返回 None 时回退到通用模式。
list_providers() -> list[ProviderProfile]返回所有已注册 profile（按 id() 去重）。首次调用触发发现。

发现顺序： bundled plugins → user plugins ($HERMES_HOME) → legacy 单文件。后加载者覆盖同名 profile。

3. 钩子方法速查

钩子签名调用时机典型覆盖场景
get_hostname() -> strmodel_metadata.py URL→provider 映射从 base_url 解析 hostname
prepare_messages(messages) -> list传输层构建 kwargs 前，codex 清洗后Qwen 将消息规范化为 list-of-parts；注入 cache_control
build_extra_body(**ctx) -> dict传输层组装 extra_body 时OpenRouter provider prefs；Gemini thinking_config
build_api_kwargs_extras(**ctx) -> (dict, dict)传输层组装 kwargs 时（在 extra_body 之前）Kimi 将 reasoning_effort 放在顶层；Qwen 拆分 enable_thinking/thinking_budget
get_max_tokens(model) -> int|None传输层计算 max_tokens 时多后端 relay 按模型返回不同 token 上限
fetch_models(*, api_key, base_url, timeout) -> list|NoneCLI model lister / doctor 探活Bedrock（无 REST 目录）；Anthropic（OAuth 目录）；OpenRouter（公开目录）

4. Provider 插件目录结构

plugins/model-providers/<name>/
├── __init__.py          ← 模块级调用 register_provider(profile)
├── plugin.yaml          ← 清单：name、kind: model-provider、version、description
└── ...                  ← 可选：自定义 ProviderProfile 子类等辅助模块

__init__.py 被导入时即完成自注册。bundled 插件通过稳定的模块路径 plugins.model_providers.<name> 导入，user 插件通过 importlib.util.spec_from_file_location 以唯一模块名 _hermes_user_provider_<name> 导入，避免多 HERMES_HOME 场景下的别名冲突。

消费者模块用法调用的 API
