# 使命：Codex 扩展系统

## 为什么
学习者需要在 Codex 源码中判断一个扩展能力到底属于 Skills、Plugins、MCP 还是 Hooks，并能顺着源码定位“能力如何被发现、怎样进入上下文或工具表、什么时候被调用”。这能帮助后续阅读 Codex 的可扩展架构、排查扩展未生效问题，或设计新的扩展接入点。

## 成功的样子
- 能用一句话区分 Skills、Plugins、MCP、Hooks 四类扩展边界。
- 能从清单文件或配置入口追到运行时注入点。
- 能画出最小“发现扩展 → 注入上下文/连接工具 → 调用或触发”序列。
- 能按源码阅读顺序继续做 L2 深挖。

## 约束条件
- L1 主题只生成 15 分钟短课和配套参考，不做源码修改。
- 只写入本主题目录，避免影响其他 worker 的进度。
- 教学材料以本地源码快照为事实来源，优先引用具体文件路径。

## 不在范围内
- 不讲完整插件市场安装流程的每个异常分支。
- 不讲 MCP 协议规范全文。
- 不讲每个内置 extension crate 的业务细节。
