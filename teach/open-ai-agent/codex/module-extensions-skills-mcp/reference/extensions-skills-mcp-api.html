层级：L3 微观 API 参考。覆盖 5 个扩展子系统 —— hooks 拦截点、plugin 清单、connectors 协议、file-search 引擎、context-fragments 类型。 配套导览：0001 扩展系统导览； L1 速查：扩展系统参考。

1. Hooks 拦截点 API

源码根：codex-rs/hooks/src/。hooks crate 定义 10 种生命周期事件、命令发现与执行引擎、以及 JSON/Toml schema。

1.1 事件清单（10 种）

事件名 常量 作用域 支持 Matcher 说明
PreToolUse HookEventName::PreToolUse Turn 是（工具名 regex） 工具调用前拦截，可 block / 改写输入 / 追加上下文
PostToolUse HookEventName::PostToolUse Turn 是（工具名 regex） 工具调用后处理，可改写 MCP 工具输出 / 追加上下文
PermissionRequest HookEventName::PermissionRequest Turn 是（工具名 regex） 权限审批阶段决策：allow / deny
PreCompact HookEventName::PreCompact Turn 是（trigger: manual / auto） 上下文压缩前，可返回 additionalInstructions
PostCompact HookEventName::PostCompact Turn 是（trigger: manual / auto） 上下文压缩后通知，无业务决策
SessionStart HookEventName::SessionStart Thread 是（source: startup / resume / clear / compact） 会话启动时注入 initial context
UserPromptSubmit HookEventName::UserPromptSubmit Turn 否（忽略 matcher） 用户提交提示词后，可 block（含 reason）或追加上下文
SubagentStart HookEventName::SubagentStart Thread 是（agent_type regex） 子 agent 启动时注入上下文
SubagentStop HookEventName::SubagentStop Turn 是（agent_type regex） 子 agent 停止时可 block（含 reason）
Stop HookEventName::Stop Turn 否（忽略 matcher） Agent 停止时可 block（含 reason）或追加 prompt fragment

HooksConfig 汇合了配置层栈（含 JSON/Toml hooks）、插件 hook 源和 shell 参数。 传给 Hooks::new(config) 后构造 ClaudeHooksEngine 并运行全量发现。

Outcome 类型速查

1.3 发现与分发流水线

discover_handlers()
  ├── append_managed_requirement_handlers()     ← 托管需求中的 hooks
  ├── 遍历 ConfigLayerStack 各层
  │     ├── load_hooks_json()                   ← <layer>/hooks/hooks.json
  │     └── load_toml_hooks_from_layer()        ← config.toml 中的 [hooks]
  └── append_plugin_hook_sources()               ← 插件 hook 源（含 PLUGIN_ROOT 等环境变量）

每条 handler → ConfiguredHandler { event_name, matcher, command, timeout_sec, ... }
  → 受 HookStateToml 控制 enabled / trust_status
  → trust_status: Managed | Trusted | Modified | Untrusted

dispatch 时:
  select_handlers() → 按 event_name + matcher 过滤
  execute_handlers() → FuturesUnordered 并发执行（stdin 注入 JSON）
  output_parser → 解析 JSON 输出 → 组装 Outcome

5. Context-Fragments 类型 API

源码根：codex-rs/context-fragments/src/。定义上下文片段的注入协议 —— 带标记的消息片段如何被构造、渲染和后续识别。

5.1 ContextualUserFragment（核心 trait）

pub trait ContextualUserFragment { fn role(&self) -> &'static str; // "user" 或 "developer" fn markers(&self) -> (&'static str, &'static str); // 包裹 body 的前后标记 fn body(&self) -> String; // 片段正文 fn type_markers() -> (&'static str, &'static str) // 类型级标记（用于后续匹配） where Self: Sized; fn matches_text(text: &str) -> bool // 判断一段文本是否属于此类型 where Self: Sized; // 派生方法 fn render(&self) -> String; // markers.0 + body + markers.1 fn into(self) -> ResponseItem; // 转为 ResponseItem::Message fn into_boxed_response_item(self: Box<Self>) -> ResponseItem; fn into_response_input_item(self) -> ResponseInputItem; }

5.2 FragmentRegistration（类型注册）

pub trait FragmentRegistration: Sync { fn matches_text(&self, text: &str) -> bool; } pub struct FragmentRegistrationProxy<T> { /* PhantomData */ } impl<T: ContextualUserFragment> FragmentRegistration for FragmentRegistrationProxy<T> { ... }

FragmentRegistrationProxy 是一个零大小类型，将任意 ContextualUserFragment 实现注册为可匹配的片段注册项。用于上下文过滤代码中识别已注入的片段，无需构造具体载荷。

5.3 AdditionalContextUserFragment（用户角色附加上下文）

pub struct AdditionalContextUserFragment { key: String, // 上下文键名 value: String, // 上下文值 } // role → "user" // markers → ("<external_", ">") ... 实际匹配用前缀/后缀检测 // body → "{key}>{truncated_value}</external_{key}>" // 其中 value 被 truncate_middle_with_token_budget(MAX=1000) 截断

5.4 AdditionalContextDeveloperFragment（开发者角色附加上下文）

pub struct AdditionalContextDeveloperFragment { key: String, value: String, } // role → "developer" // markers → ("", "") —— 无标记，不参与 matches_text // body → "<{key}>{truncated_value}</{key}>" // 其中 value 被 truncate_middle_with_token_budget(MAX=1000) 截断

5.5 片段标记匹配规则

matches_marked_text(start, end, text) 使用大小写不敏感的 trim 匹配： text 去掉前导空白后以 start_marker 开头，去掉尾部空白后以 end_marker 结尾。 无标记的片段（markers 均为空串）永远不匹配。

排查扩展相关问题时，按以下链路定位：

  1. Plugins 加载：检查 PluginLoadOutcome 的 error 字段、hook_load_warnings。
  2. Hooks 触发：先用 preview_*() 确认 handler 是否被选中；再检查 trust_status（Managed / Trusted / Modified / Untrusted）。未信任的非 managed hook 不会执行。
  3. MCP tools 可见：确认 effective_mcp_servers() 有目标 server；用 McpConnectionManager 的 startup status 排查连接失败。
  4. Connector 不可用：检查 is_accessible、is_enabled、app_tool_policy 是否禁用。
  5. File-Search 无结果：确认 respect_gitignore 设置；检查 exclude 模式；total_match_count vs matches.len() 判断是否截断。
  6. Context 片段不显示：确认 fragment markers 非空；检查 truncate_middle_with_token_budget(1000) 是否截断了关键内容。
