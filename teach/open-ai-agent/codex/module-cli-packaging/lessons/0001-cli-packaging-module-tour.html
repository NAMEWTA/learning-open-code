CLI 与 npm 包装层：从 codex 命令到 Rust 分发

学习目标：15 分钟内画出 `codex` 命令从 npm 包装层进入 Rust CLI，并被分流到运行模块的路径。

一句话定位：`codex-cli/` 是发布和启动包装层，`codex-rs/cli` 是 Rust 多工具分发层；真正的业务能力继续下沉到 `tui`、`exec`、`mcp`、`plugin`、`app-server` 等 crate。 主要来源：codex-cli/package.json、codex-cli/bin/codex.js、codex-rs/cli/src/main.rs。

1. 两段入口，不要混成一层

README 把 Codex CLI 定义成本地运行的 coding agent；用户可以通过安装脚本、npm、Homebrew 或 GitHub Release 获得它。npm 路径的第一站是 `@openai/codex` 包。这个包本身不实现 agent，只在 `package.json` 里声明 `bin`，把用户敲的 `codex` 映射到 `bin/codex.js`。 

"bin": {
  "codex": "bin/codex.js"
},
"files": [
  "bin/codex.js"
]

因此排查“npm install 后为什么能运行”时，先读包装层；排查“`codex exec` 做了什么”时，马上切到 Rust CLI。这个边界能防止你在 JavaScript 启动脚本里寻找不存在的业务逻辑。

2. npm 包装层只做平台选择和进程转发

`codex.js` 根据 Node 的 `process.platform` 与 `process.arch` 计算 Rust target triple，再映射到平台 optional package，例如 macOS arm64 对应 `@openai/codex-darwin-arm64`。脚本优先从该平台包的 `vendor/` 目录找原生二进制；找不到时退回当前包的 `vendor/`。最后用 `spawn(binaryPath, process.argv.slice(2), { stdio: "inherit" })` 启动原生 `codex`，并转发 `SIGINT`、`SIGTERM`、`SIGHUP`，让 Ctrl-C 等信号落到子进程。 

flowchart LR
  A[用户运行 codex exec "..."] --> B[@openai/codex bin]
  B --> C[codex-cli/bin/codex.js]
  C --> D[选择 target triple]
  D --> E[查找平台 package/vendor 二进制]
  E --> F[spawn 原生 codex]
  F --> G[codex-rs/cli/src/main.rs]

3. Rust CLI 是薄分发层

`codex-rs/cli/src/main.rs` 用 clap 定义 `MultitoolCli`。它先接收全局配置覆盖、feature toggles、remote 选项和交互 TUI 参数，再读取 `subcommand: Option<Subcommand>`。没有子命令时进入 `run_interactive_tui`；有子命令时进入 `cli_main` 的 `match subcommand` 分支。 

match subcommand {
    None => run_interactive_tui(...).await?,
    Some(Subcommand::Exec(mut exec_cli)) => {
        codex_exec::run_main(exec_cli, arg0_paths.clone()).await?;
    }
    Some(Subcommand::Mcp(mut mcp_cli)) => {
        mcp_cli.run(loader_overrides).await?;
    }
    Some(Subcommand::Plugin(plugin_cli)) => { /* plugin_cmd */ }
    Some(Subcommand::AppServer(app_server_cli)) => { /* app-server */ }
    Some(Subcommand::RemoteControl(remote_control_cli)) => { /* daemon */ }
}

这一层的关键依赖也印证了“分发层”职责：`codex-tui` 承接交互界面，`codex-exec` 承接非交互执行和 review，`codex-mcp-server`、`codex-mcp`、`codex-core-plugins`、`codex-app-server`、`codex-sandboxing` 等分别承接外部工具、扩展、富客户端和平台安全边界。

4. 用一个命令练习分流判断

简单调用示例：`codex exec "summarize this repo"`。npm 安装场景下，它先进入 `codex.js`，脚本把 `exec "summarize this repo"` 原样转发给原生二进制。Rust 侧 clap 解析到 `Subcommand::Exec`，合并根级配置与交互共享参数后调用 `codex_exec::run_main`。相反，单独运行 `codex` 不会进 exec，而是进入 TUI；运行 `codex mcp list` 会进入 `mcp_cmd.rs`；运行 `codex app-server` 会启动 app-server transport。

暂停回忆：下面三个命令分别应该从哪个源码区域继续下钻？`codex`、`codex plugin list`、`codex remote-control pair`。

参考答案：`codex` 进入 `codex-rs/tui`；`codex plugin list` 先看 `codex-rs/cli/src/plugin_cmd.rs`，再下钻 `codex-core-plugins`；`codex remote-control pair` 先看 `codex-rs/cli/src/remote_control_cmd.rs`，再关联 app-server daemon。仍不确定时，向 agent 追问某个分支的完整调用链。
