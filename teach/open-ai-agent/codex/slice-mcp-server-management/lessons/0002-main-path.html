主路径：mcp add 从 CLI 到连接建立的逐层追踪

学习目标：逐层追踪一次 stdio MCP 服务器的添加和启动过程，理解每层的输入输出和错误处理。

前置：链路地图与时序图

1. CLI 层：参数解析与 Transport 构造

当用户执行 codex mcp add my-tool -- node server.js --port 3000，clap 首先将参数解析为 AddArgs。关键点：command 字段用 trailing_var_arg = true 收集 -- 之后的所有参数：

// codex-rs/cli/src/mcp_cmd.rs（简化）
pub struct AddArgs {
    pub name: String,                    // "my-tool"
    pub transport_args: AddMcpTransportArgs,
}

pub struct AddMcpStdioArgs {
    #[arg(trailing_var_arg = true, num_args = 0..)]
    pub command: Vec<String>,            // ["node", "server.js", "--port", "3000"]
    #[arg(long, value_parser = parse_env_pair)]
    pub env: Vec<(String, String)>,     // --env 指定的环境变量
}

run_add() 中，command 向量的第一个元素被取出作为 command_bin（即 node），剩余部分作为 args（即 ["server.js", "--port", "3000"]）。然后构造 McpServerTransportConfig::Stdio。

2. 配置层：TOML 序列化与原子写入

构造好的 McpServerConfig 通过 ConfigEditsBuilder 写入 ~/.codex/config.toml。这个 builder 使用 toml_edit 而非简单的 toml::to_string，目的是保留用户已有的格式和注释：

// codex-rs/config/src/mcp_edit.rs（简化）
fn replace_mcp_servers(doc: &mut DocumentMut,
                       servers: &BTreeMap<String, McpServerConfig>) {
    let root = doc.as_table_mut();
    if servers.is_empty() {
        root.remove("mcp_servers");     // 空时删除整个段
        return;
    }
    let mut table = TomlTable::new();
    table.set_implicit(true);
    for (name, config) in servers {
        table.insert(name, serialize_mcp_server(config));
    }
    root.insert("mcp_servers", TomlItem::Table(table));
}

序列化时有两个重要的校验点：

  1. 禁止 inline bearer_token：ensure_no_inline_bearer_tokens() 在读取配置时检查，如果发现 bearer_token 字段（而非 bearer_token_env_var），直接返回错误。内联 token 是安全反模式。
  2. Transport 互斥：在 RawMcpServerConfig::try_into() 中，如果 command 存在则禁用所有 HTTP 字段（url、bearer_token_env_var、http_headers 等），反之亦然。这由 throw_if_set() 辅助函数强制执行。

3. 客户端层：Transport 创建与 Initialize 握手

当 Codex 会话启动，McpConnectionManager::new() 为每个 enabled server 创建一个 AsyncManagedClient。其内部的 ManagedClientStartup::start() 是核心入口，分两步：

步骤 1 — 创建 transport：make_rmcp_client() 根据 transport 类型创建不同的 RmcpClient：

make_rmcp_client() 分支： ├── Stdio │ → RmcpClient::new_stdio_client(program, args, env, launcher) │ → StdioServerLauncher 启动子进程 │ → PendingTransport::Stdio { transport } │ ├── StreamableHttp（无 OAuth） │ → RmcpClient::new_streamable_http_client(server_name, url, ...) │ → StreamableHttpClientTransport 包装 HTTP client │ → PendingTransport::StreamableHttp { transport } │ └── StreamableHttp（有 OAuth） → 同上，但用 AuthClient 包装 → 加载存储的 OAuth token → 尝试 bearer 认证 → PendingTransport::StreamableHttpWithOAuth { transport, oauth_persistor }

步骤 2 — Initialize 握手：创建 transport 后，RmcpClient::initialize() 向 MCP server 发送 initialize 请求，携带 client capabilities（elicitation 类型、sandbox state meta 等）。成功后调用 tools/list 获取服务器提供的工具列表。

// codex-rs/codex-mcp/src/rmcp_client.rs — ManagedClientStartup::start() 核心流程
// 1. 创建 RmcpClient（底层 transport）
let client = Arc::new(make_rmcp_client(&server_name, server, ...).await?);

// 2. 启动并等待完成（initialize + tools/list + 缓存）
let managed = start_server_task(server_name, client, StartServerTaskParams {
    startup_timeout: config.startup_timeout_sec
        .or(Some(DEFAULT_STARTUP_TIMEOUT)),   // 默认 30s
    tool_timeout: config.tool_timeout_sec
        .unwrap_or(DEFAULT_TOOL_TIMEOUT),     // 默认 300s
    tool_filter,
    ...
}).await?;

4. 异常路径与重试机制

连接可能因为多种原因失败，每条路径有不同的处理方式：

异常场景检测位置处理方式
Server name 非法 validate_server_name() 在 CLI 层 立即返回错误，提示只允许字母数字和 - _
内联 bearer_token ensure_no_inline_bearer_tokens() 在配置读取时 返回 InvalidData 错误，指引使用 bearer_token_env_var
启动超时（默认 30s） ManagedClientStartup::start() 中的 timeout 返回 StartupOutcomeError::Failed，提示调整 startup_timeout_sec
OAuth 认证过期 / 未登录 is_authentication_required_error() 匹配错误文本 返回 "Auth required"，提示运行 codex mcp login
Codex Apps 启动失败 reconnect_failed_startup() 独立后台重试循环（指数退避 1s→30s），不影响其他 server

暂停回忆：不看上文，回答三个问题：

  1. trailing_var_arg = true 在 CLI 参数解析中起什么作用？command_bin 和 args 是如何分离的？
  2. 配置层为什么使用 toml_edit::DocumentMut 而非简单的 toml::to_string？
  3. 连接失败时，普通 MCP server 和 Codex Apps MCP server 的重试策略有什么不同？

参考答案：1. trailing_var_arg = true 让 clap 收集 -- 之后的所有参数为一个 Vec，第一个元素是程序名（command_bin），后续元素是参数（args）。2. toml_edit 保留用户已有的 TOML 格式、注释和键顺序，避免每次写入时格式化破坏用户手动编辑的内容。3. 普通 MCP server 失败后只报告错误，不自动重试；Codex Apps MCP server 通过 CodexAppsStartupReconnect 在后台独立重试（指数退避 1s→30s），重试成功后会更新 client 状态。
