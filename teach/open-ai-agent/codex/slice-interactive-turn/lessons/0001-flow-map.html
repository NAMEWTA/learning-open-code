链路地图：一次交互 Turn 的完整时序

学习目标：15 分钟内画出从 TUI 用户输入到模型流式响应渲染的完整 mermaid 时序图，识记链路上每一层的职责边界。
前置：L1 TUI 模块导览、L1 Core Runtime 模块导览

一句话定位：一次交互 turn 是 11 个步骤组成的纵向链路——用户输入逐层穿透 TUI ChatWidget → App 事件循环 → App-server Session → CodexThread → ModelClient → 远端 API，响应再沿同一链路逆流回到 TUI 屏幕。本课提供这张完整地图。

1. 先看清 11 层

抛开编译单元细节，一次 turn 的逻辑层级只有 11 步。这 11 步分布在 CLI、TUI、App-server、Core、Network 五个进程/线程边界之间。每一层的输入输出都有明确的数据类型做合同。

用户 Enter 键 │ ▼ [1] cli/src/main.rs ── run_interactive_tui() → codex_tui::run_main() │ CLI 参数解析 + TERM 检查 ▼ [2] tui/src/lib.rs ── run_main()：加载 config → 启动 app-server → 创建 Tui + App │ App::run() 进入 select! 事件循环 ▼ [3] tui/src/app.rs ── handle_tui_event(TuiEvent::Key) → ChatWidget 处理键盘事件 │ ▼ [4] tui/src/chatwidget.rs ── 用户按 Enter → InputResult::Submitted │ → submit_op(AppCommand::UserInput { items, .. }) │ ▼ [5] tui/app_server_session.rs ── AppServerSession 发送 user input 至 app-server │ 内嵌或远程 app-server 接收并注入 session 队列 ▼ [6] core/src/codex_thread.rs ── CodexThread::submit_user_input_with_client_user_message_id() │ Op 进入 session.input_queue ▼ [7] core/src/session/turn.rs ── run_turn() 启动：pre-compact → step_context → sampling loop │ ▼ [8] core/src/client.rs ── ModelClientSession::stream()：WebSocket/HTTP 发送 Responses API 请求 │ ▼ [9] OpenAI API (Network) ── SSE 事件流返回（reasoning → content delta → finished） │ ▼ [10] core/src/stream_events_utils.rs ── handle_non_tool_response_item() 处理每个 SSE 事件 │ 每个事件转为 EventMsg，经 app-server 推送 TUI │ ▼ [11] tui/src/app.rs ── handle_active_thread_event(EventMsg) tui/src/chatwidget.rs → ChatWidget 更新 active_cell（markdown stream 渲染） → 提交 frame → ratatui 绘制到终端屏幕

3. 边界识别与异常路径

记住这张地图，你就能回答三类排障问题：

  • TUI 无响应？ 检查步骤 [2] 中 TERM 检查是否失败、[3] 中 select! 是否还在运行、[4] 中 ThreadInputState 是否卡在 Submitting。
  • 模型不回答？ 检查步骤 [8] 中 WebSocket 是否连接成功、[9] 中 API 是否返回错误码、[7] 中 cancellation_token 是否被触发。
  • 流式渲染卡顿？ 检查步骤 [11] 中 AdaptiveChunkingPolicy 是否过于保守、commit_tick 间隔是否过大。

一条异常路径：API 返回错误

当步骤 [9] 返回 429（Rate Limit）或 5xx 错误时，client.rs 中的 retry/fallback 逻辑会介入。简化流程如下：

API-->>MC: HTTP 429 / 5xx MC->>MC: handle_retryable_response_stream_error() MC->>MC: 判断是否可重试 (retry budget, HTTP fallback) alt 可重试 MC->>API: 重试请求 (可能走 HTTP fallback) else 不可重试 MC-->>RT: Err(CodexErr::...) RT-->>CW: EventMsg::Error CW->>CW: 显示红色错误信息 + 恢复 composer end

这段逻辑在 core/src/responses_retry.rs 中实现，lesson 0002 会详细展开。速查表见 参考文档。
