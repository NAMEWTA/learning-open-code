1. analytics — 分析事件捕获与上报

codex-analytics crate 负责将 Codex 运行时产生的事件（session 初始化、turn 生命周期、工具调用、Guardian 审查、compaction、插件操作等）归约、批处理并通过 HTTP 发送到后端分析端点。核心入口是 AnalyticsEventsClient，内部使用 AnalyticsReducer 将原始事实（AnalyticsFact）归约为可序列化事件（TrackEventRequest），再通过 tokio mpsc 队列异步发送。

1.5 内部事件队列与批处理

客户端通过 AnalyticsEventsQueue（tokio mpsc channel，容量 256）将事实异步送入后台任务。后台任务逐条调用 AnalyticsReducer::ingest() 归约为 Vec<TrackEventRequest>，然后通过 HTTP POST 发送到 {base_url}/codex/analytics-events/events。

  • 去重：track_app_used 和 track_plugin_used 维护 HashSet（容量 4096），同 turn 内相同 (turn_id, connector_id) 或 (turn_id, plugin_id) 只发一次。
  • 隔离发送：AcceptedLineFingerprints 事件通过 should_send_in_isolated_request() 标记为独立请求，不与其它事件合并。
  • 认证筛选：仅当 auth.uses_codex_backend() 为 true 时才发送。
  • 超时：HTTP 请求超时 10 秒。
  • 调试捕获：debug 构建下，如果设置了 ANALYTICS_EVENTS_CAPTURE_FILE 环境变量，网络传输被禁用，事件写入本地文件。

3. rollout-trace — 特性使用追踪与遥测

codex-rollout-trace crate 拥有 trace schema 的所有权。热路径 Codex 代码依赖其轻量级 writer API；语义回放和查看器投影位于 codex-core 之外。crate 的核心抽象分为三层：writer（热路径追加写入）、raw event（序列化格式）、reducer（确定性回放生成 RolloutTrace 图）。

3.1 TraceWriter — 热路径写入器

方法签名说明
create() pub fn create(bundle_dir: impl AsRef<Path>, trace_id: String, rollout_id: String, root_thread_id: AgentThreadId) -> Result<Self> 创建 trace bundle 目录，写入 manifest.json。bundle_dir 下创建 payloads/ 子目录和 trace.jsonl 事件日志。内部维护 BufWriter 和自增 seq
write_json_payload() pub fn write_json_payload(&self, kind: RawPayloadKind, value: &impl Serialize) -> Result<RawPayloadRef> 将任意可序列化值写入 payloads/{ordinal}.json 并返回引用。payload 文件在引用它的事件之前创建，保证回放中断时不会出现悬空引用
append() pub fn append(&self, payload: RawTraceEventPayload) -> Result<RawTraceEvent> 追加一行 raw event 到 trace.jsonl（无额外上下文）
append_with_context() pub fn append_with_context(&self, context: RawTraceEventContext, payload: RawTraceEventPayload) -> Result<RawTraceEvent> 追加带 thread_id/codex_turn_id 上下文的 raw event。写入后立即 flush

线程安全：TraceWriter 内部使用 Mutex<TraceWriterInner>，所有公开方法都是 &self（不可变借用），适合多线程共享。PoisonError 时保留事件日志。

环境变量：CODEX_ROLLOUT_TRACE_ROOT_ENV 控制本地 trace bundle 录制的根目录。

3.7 Reducer 架构

TraceReducer 是有状态的回放引擎。它维护以下中间状态：

  • thread_conversation_snapshots：每个线程的模型可见对话快照，用于对话归约器的请求/响应对比
  • pending_compaction_replacement_item_ids：压缩安装的替换历史，在下一次 sampling 请求时对比
  • code_cell_ids_by_runtime：运行时代码单元格 handle 到 reducer ID 的映射
  • pending_code_cell_starts：已开始但模型可见源项尚未归约的代码单元格
  • pending_code_cell_lifecycle_events：等待 start 就绪的生命周期事件队列
  • pending_agent_interaction_edges：等待接收方 transcript item 归约的 agent 交互边

4. feedback — 用户反馈收集

codex-feedback crate 提供日志环形缓冲区、tracing 层集成和 Sentry 上传能力。核心入口是 CodexFeedback，它通过 tracing subscriber 的 layer 机制捕获全保真度日志到内存环形缓冲区，并提供 FeedbackSnapshot 供上传。

4.6 环形缓冲区

内部使用 RingBuffer（基于 VecDeque<u8>），默认容量 4 MiB。写入策略：

  • 单次写入超过容量：清空缓冲区，仅保留尾部 max 字节
  • 超出容量：从头部逐字节驱逐
  • snapshot_bytes()：完整拷贝当前缓冲区内容

4.7 标签元数据层

FeedbackMetadataLayer 实现 tracing_subscriber::Layer，从 target="feedback_tags" 的 event 中提取所有字段（i64/u64/bool/f64/str/debug）并存入 BTreeMap<String, String>。标签上限 64 个，已存在的 key 不覆盖。

4.8 上传标签合并策略

upload_tags() 构建最终标签 BTreeMap 的规则：

  1. 预留字段（thread_id、classification、cli_version、session_source、reason）始终使用计算值
  2. 调用方传入的 client_tags 中非预留字段优先（insert if vacant）
  3. 快照中的 tags（来自 metadata layer）作为后备填充
