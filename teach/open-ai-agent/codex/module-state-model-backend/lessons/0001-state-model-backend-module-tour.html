状态、模型提供商与后端通信：15 分钟模块导览

学习目标：能定位认证令牌从平台密钥链到 HTTP 请求头的完整路径，并能区分 state / thread-store / rollout / model-provider / login 五个核心 crate 的职责边界。

架构位置：先看 L0 项目地图；完整清单见 状态与后端通信参考。

1. 这 10 个 crate 如何分四层

本模块不是"一个功能"，而是 Codex 连接外部世界的四层基础设施：状态持久化层（state、thread-store、rollout、rollout-trace）负责把 session 数据写入磁盘和 SQLite；认证密钥层（login、keyring-store）负责 OAuth 登录和令牌安全存储；模型 API 层（model-provider、model-provider-info、backend-client）负责 provider 抽象和 HTTP 通信；配置遥测层（cloud-config）负责云端配置拉取和缓存。

2. 认证令牌的最短路径

理解这四层如何协作，最直观的方式是追踪一次认证令牌的完整生命周期。从用户执行 codex login 到 API 请求携带 Bearer token，经过以下关键节点：

codex login → LoginServer (启动本地 HTTP 服务器接收回调) → DeviceCode (OAuth 设备码流程: request → 浏览器授权 → poll token) → AuthManager (令牌缓存、刷新调度、过期检测) → KeyringStore (macOS Keychain / Windows Credential Manager / Linux secret-service) → ModelProvider::auth() (从 AuthManager 拉取当前有效令牌) → ResolvedProviderAuth (按 scope 解析认证方式) → BearerAuthProvider → HTTP Authorization header

关键边界：login crate 拥有认证流程和令牌生命周期，keyring-store 只负责安全存储的 load/save/delete 三个操作，model-provider 在每次 API 调用时通过 auth() 获取当前有效令牌。三者通过 trait 解耦，测试时可以注入 MockKeyringStore。

3. 状态持久化的读写分离

codex-state 管理 4 个 SQLite 数据库（logs_2、goals_1、memories_1、state_5），通过 StateRuntime 提供统一入口。但 session 的原始数据并不直接写入 SQLite——而是先由 RolloutRecorder 追加到 JSONL rollout 文件，再由 codex-core 中的 backfill 机制提取 metadata 写入 SQLite。

因此持久化存在两条路径：

写入路径（实时）: Turn 事件 → RolloutRecorder::append → JSONL rollout 文件 → 压缩 worker 读取路径（索引）: RolloutRecorder → backfill → codex-state SQLite → ThreadStore::read_thread ThreadStore::list_threads → RolloutConfig → rollout 文件系统扫描 + 游标分页

thread-store 的 ThreadStore trait 定义了 20 个抽象方法（含 3 个带默认实现的可选方法），LocalThreadStore 将其映射到本地文件系统操作。读操作（list/search/read）走 rollout 文件扫描，写操作（create/resume/append）通过 RolloutRecorder 和 LiveThread 管理，flush/shutdown 确保数据落盘。

4. 暂停回忆

不看上文，回答三个问题：

  1. codex-state 管理哪 4 个 SQLite 数据库？各自的文件名是什么？
  2. 认证令牌从 KeyringStore 取出后，经过哪个 crate 的哪个方法到达 HTTP 请求头？
  3. session 数据是先写 SQLite 还是先写 JSONL rollout 文件？backfill 的作用是什么？

参考答案：1. logs_2.sqlite（日志）、goals_1.sqlite（目标）、memories_1.sqlite（记忆）、state_5.sqlite（状态元数据）。2. 经过 model-provider 的 ModelProvider::auth() → resolve_provider_auth() → BearerAuthProvider，最终成为 HTTP Authorization header。3. 先写 JSONL rollout 文件（实时），backfill 机制随后提取 metadata 写入 SQLite 供索引查询。

主要阅读来源：codex-rs/login/src/device_code_auth.rs、codex-rs/model-provider/src/provider.rs 和 codex-rs/rollout/src/recorder.rs。后续可以追问 agent 带你从 codex login 的 CLI 入口一路追到 keyring 写入，或从 ThreadStore::create_thread 追到 rollout 文件落盘。
