反向扫描算法的逐步拆解

学习目标：逐行理解 reconstruct_history_from_rollout() 的枚举反向扫描、三条件提前终止、分段累积和正向 replay 四阶段流程。

前置：问题框架——JSONL 与 SQLite 双轨制

参考：反向重建快速参考

1. 算法总览：为什么是反向扫描？

Resume 只关心最新的状态。正向扫描意味着读完整个文件才能找到最后一个 TurnContext，而反向扫描一旦发现"足够新"的信息就可以提前终止。核心循环只有一行：

for (index, item) in rollout_items.iter().enumerate().rev() {

enumerate() 提供行号索引，用于后续切分 rollout_suffix（Compaction 之后的那一段）。rev() 反转迭代方向，从最新行往最旧行扫描。整个算法可以拆为四个阶段：

阶段1（反向扫描）: 从 JSONL 末尾向头部迭代每个 RolloutItem │ ├─ 遇到 Compaction? → 记录 replacement_history + window 信息 ├─ 遇到 TurnStarted? → 触发 finalize_active_segment（分段累积完成） ├─ 遇到 TurnContext? → 记录 previous_turn_settings + reference_context_item ├─ 遇到 ThreadRolledBack? → 增加 pending_rollback_turns 计数 └─ 遇到 ResponseItem / InterAgentCommunication? → 标记 user_turn 边界 │ 阶段2（提前终止）: 三条件同时满足 → break │ 阶段3（正向 replay rollout_suffix）: 用 ContextManager 重建完整 history │ 阶段4（正向 replay world_state）: 按时间顺序应用 snapshot + patches

反向扫描的核心洞察是：越新的 Compaction 携带越完整的 replacement_history——一旦在反向扫描中找到某个 Compaction 的 replacement_history，它之前（更旧的）所有 ResponseItem 都可以安全丢弃。

2. 阶段 1：反向扫描与分段累积

反向扫描并非每行独立处理——它维护一个 ActiveReplaySegment，将相邻的 rollout items 按 turn 边界分组。一个 segment 从 TurnStarted 到下一个 TurnStarted（反向即从 TurnComplete/TurnAborted 遇到 TurnStarted 为止）。

每个 ActiveReplaySegment 累积以下信息：

struct ActiveReplaySegment {
    turn_id: Option<String>,           // 当前 turn 的 ID
    counts_as_user_turn: bool,        // 是否包含用户消息
    previous_turn_settings: Option<..>, // TurnContext 中的模型配置
    reference_context_item: TRefCtx,  // TurnContextItem 或 Cleared
    world_state_replay: Vec<&RolloutItem>, // 世界状态快照与补丁
    base_replacement_history: Option<&[ResponseItem]>, // Compaction 携带的历史
    window: Option<ReconstructedWindow>, // 上下文窗口信息
}

当反向扫描遇到 TurnStarted 且 turn_id 匹配时，调用 finalize_active_segment() 将当前 segment 的信息写入全局状态变量，然后清空 segment 准备处理下一个（更旧的）turn。

关键分支：RolloutItem 匹配逻辑

RolloutItem 变体反向扫描行为
Compacted 设置 window 信息；若有 replacement_history，记录为 base 并切分 rollout_suffix；将 reference_context_item 标记为 Cleared
TurnStarted turn_id 匹配时调用 finalize_active_segment()，结束当前 segment
TurnContext 记录 previous_turn_settings 和 reference_context_item
EventMsg::ThreadRolledBack 增加 pending_rollback_turns，后续 finalized segment 若 counts_as_user_turn 则跳过
ResponseItem 若 is_user_turn_boundary() 则标记 counts_as_user_turn = true

子阶段 2：三条件提前终止

这是反向扫描的性能关键。每处理一个 item 后检查三个条件：

if base_replacement_history.is_some()
    && previous_turn_settings.is_some()
    && !matches!(reference_context_item, TurnReferenceContextItem::NeverSet)
{
    break;
}

三个条件的语义：

  1. base_replacement_history.is_some()——已找到最新的 Compaction 携带的完整历史基线。更旧的 items 不会再影响 history。
  2. previous_turn_settings.is_some()——已找到最近一次 TurnContext 中的模型配置（model、comp_hash、realtime_active）。
  3. reference_context_item != NeverSet——已确定 reference context baseline（可能是 TurnContextItem 也可能是 Cleared）。NeverSet 表示反向扫描至今尚未遇到任何 TurnContext 或 Compaction。

性能意义：在典型场景（最近几个 turn 内有 Compaction 且有 TurnContext），扫描在几十行内即终止。最坏情况（从头到尾没有 Compaction 也没有 TurnContext），扫描整个文件——O(n) 线性扫描。

3. 阶段 3-4：正向 replay 重建 history 与 world_state

反向扫描完成后，得到了两个关键中间结果：

  • rollout_suffix——从最新 Compaction 的 replacement_history 之后（不含）到文件末尾的 rollout items 切片
  • world_state_replay——按反向顺序收集的 Compaction 和 WorldState items（将被反转后正向应用）

阶段 3（history 重建）：以 base_replacement_history 为起点，正向遍历 rollout_suffix，用 ContextManager 逐条 record_items()。遇到新的 Compaction 时调用 history.replace() 替换整个历史。这确保重建的 history 与写入时完全一致。

legacy compaction 的额外路径

如果 Compaction 的 replacement_history 为 None（legacy 格式），进入备选路径：从当前 history 中收集 user messages，用 build_compacted_history() 重新拼接紧凑版本。同时清除 reference_context_item——因为 legacy 格式无法保证 TurnContext 插入位置正确。

阶段 4（world_state 重建）：将 world_state_replay 反转（变为正向时间顺序），然后按序处理：遇到 full=true 的 WorldState 时覆盖基线；遇到 full=false 的 WorldState 时应用 merge patch；遇到 Compaction 时清空基线（因为 compact 意味着历史压缩，旧的世界状态已不可恢复）。

最终，RolloutReconstruction 将所有结果打包返回给调用方：

RolloutReconstruction {
    history,                    // 完整对话历史
    previous_turn_settings,     // 最近一次 Turn 的配置
    reference_context_item,     // TurnContext 基线
    world_state_baseline,       // 世界状态快照
    window_number,              // 上下文窗口编号
    first_window_id,            // 首个窗口 UUID
    previous_window_id,         // 前一窗口 UUID
    window_id,                  // 当前窗口 UUID
}

不看上文，完成以下任务：

  1. 画出反向扫描遇到 Compaction（有 replacement_history）→ TurnContext → TurnStarted 这三条 rollout line 时，三个全局状态变量各如何变化？
  2. 如果 JSONL 文件没有 Compaction 行也没有 TurnContext 行，三条件提前终止会怎样？
  3. legacy compaction（replacement_history = None）与新版 compaction 在重建路径上有何不同？

参考答案：1. Compaction 设置 base_replacement_history + 切分 rollout_suffix + 标记 reference_context 为 Cleared；TurnContext 设置 previous_turn_settings（model/comp_hash/realtime） + 若 reference_context 尚未设置则写入 TurnContextItem；TurnStarted 触发 finalize_active_segment 将 segment 中的所有信息写入全局变量。2. 三条件永远不会同时满足，扫描将走到文件开头（index 0），等同于全量正向扫描。3. 新版直接从 replacement_history 替换整个 history；legacy 需要从当前 history 中重新收集 user_messages 然后调用 build_compacted_history() 重新拼接，同时清除 reference_context_item。
