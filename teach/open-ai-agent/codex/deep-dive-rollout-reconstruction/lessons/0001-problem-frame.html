问题框架：JSONL 与 SQLite 的持久化分工

学习目标：理解 Codex 为什么将 JSONL 作为热路径、SQLite 作为冷 backfill，能对比两种方案在 Session 重建场景下的设计权衡。

前置：L2 会话持久化链路地图

参考：反向重建快速参考

1. 问题的本质：Session 重建需要什么？

进程重启后恢复一个 Session，需要回答三个问题：

  1. 最近一次 Turn 的配置（模型、comp hash、realtime 开关）——用于向 API 发续接请求
  2. 完整的对话历史（ResponseItem 列表）——用于构建 prompt context
  3. World State 基线（沙箱文件快照）——确保环境一致性

这三个答案全部编码在 rollout JSONL 文件中——一份按时间顺序追加的、人类可读的 JSON 事件日志。重建算法要做的，是从这份 append-only 日志中高效提取上述信息。

关键约束：JSONL 文件在写入时只管追加，不维护索引。写入端（RolloutRecorder）和读取端（Session resume）解耦，读取端不能假设文件结构——只能通过逐行扫描来发现数据。

2. JSONL 方案：append-only 的"源"

Codex 选择 JSONL 作为主持久化介质，有以下设计理由：

特性对 Session 持久化的意义
追加写（append-only） 写入无锁竞争；崩溃时最多丢最后一行，不会损坏已有数据
人类可读 用户可用 jq / fx 直接检查会话内容，调试友好
行级原子性 每行是一个完整的 JSON 对象，写入即为持久化单元
zstd 压缩 旧 rollout 自动压缩为 .jsonl.zst，节省磁盘空间
文件系统天然排序 文件名嵌入时间戳和 UUID，按名称排序即可获得时间顺序

写出端是 RolloutRecorder（codex-rs/rollout/src/recorder.rs），运行在独立的 Tokio 后台任务中。items 通过 mpsc channel 异步投递，由 RolloutWriterState 负责缓冲、写入重试、错误恢复。

3. 双轨制的代价与缓解

Codex 的选择是双轨制——JSONL 作为 source-of-truth（热路径），SQLite 作为元数据索引（冷 backfill）。关键权衡是：宁可让 resume 做一次 O(n) 行扫描，也不在每次 append() 时付出 SQLite 写入代价（WAL 写入放大、单写者锁、migration 负担）。

双轨制引入了"元数据滞后"问题——用户刚结束对话时，SQLite 中的 thread 元数据可能尚未 backfill。Codex 通过以下机制缓解：

  • Backfill 租约（try_claim_backfill）：同一时刻只有一个进程持有 backfill 锁，避免重复扫描
  • 文件系统优先（list_threads_with_db_fallback）：thread 列表先扫文件系统、再与 SQLite 交叉校验
  • Read-repair：读取时发现 SQLite 缺失行，从 JSONL 头中提取元数据补写

这种设计使热路径（rollout 写入、resume 重建）完全独立于 SQLite 可用性——即使 SQLite 损坏或未初始化，Session 的创建和恢复仍能正常工作。

4. 暂停回忆

不看上文，回答三个问题：

  1. Codex 的 Session 重建需要从 rollout 中提取哪三种信息？
  2. JSONL append-only 相比 SQLite 在热路径上的两个关键优势是什么？
  3. "双轨制"中的 SQLite 后置 backfill 解决了什么问题？又引入了什么新问题？

参考答案：1. 最近 Turn 配置（模型、comp hash）、完整对话历史（ResponseItem 列表）、World State 基线。2. 无写锁竞争（append-only 天然无冲突）+ 崩溃安全性（最多丢最后一行，不损坏已有数据）；此外还保留了人类可读性。3. backfill 解决了 SQLite 查询速度（O(log n) 索引过滤线程列表）的优势，但引入了元数据滞后——刚结束的对话在 SQLite 中可能尚未体现，需要文件系统优先 + read-repair 交叉验证。
