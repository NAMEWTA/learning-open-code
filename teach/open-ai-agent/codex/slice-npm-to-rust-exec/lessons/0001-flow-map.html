链路地图：从 npm bin 到 Rust exec 的完整时序

学习目标：15 分钟内画出 `codex exec "hello"` 从终端敲下到 Rust exec 进程启动的完整时序图。

一句话定位：本切片沿 `package.json` → `codex.js` → 平台二进制 → `cli/main.rs` → `codex_exec::run_main` 逐层追踪，每层只做一件事，不越界。 前置知识：L1 CLI 包装层模块导览。

1. 时序总览

下面这张 mermaid 时序图覆盖了从终端敲下命令到 Rust exec 进程启动的全部 6 层。建议先扫一遍全图建立心理模型，再逐层下钻。 

sequenceDiagram
    participant U as 用户终端
    participant NPM as npm bin (package.json)
    participant JS as codex.js (Node.js)
    participant FS as 文件系统 (vendor/)
    participant CLI as codex-rs/cli (Rust)
    participant EXEC as codex-rs/exec (Rust)

    U->>NPM: codex exec "hello"
    NPM->>JS: 解析 bin 映射 → 执行 bin/codex.js
    JS->>JS: platform + arch → target triple
    JS->>FS: require.resolve(platformPackage/package.json)
    alt 二进制存在
        FS-->>JS: vendor/{triple}/bin/codex
        JS->>CLI: spawn(binaryPath, ["exec","hello"])
    else 二进制缺失
        JS-->>U: Error: Missing optional dependency
    end
    CLI->>CLI: arg0_dispatch_or_else → cli_main
    CLI->>CLI: MultitoolCli::parse() → Subcommand::Exec
    CLI->>EXEC: codex_exec::run_main(exec_cli, arg0_paths)
    EXEC->>EXEC: 构建 Config → 启动 agent 循环

2. 第 1-2 层：npm bin 到 Node.js 入口

用户在终端敲下 codex exec "hello"。操作系统根据 $PATH 找到 npm 全局安装的 codex 符号链接，它指向 @openai/codex 包的 bin/codex.js。这一切由 package.json 中的 "bin": { "codex": "bin/codex.js" } 声明。此时 Node.js 进程启动，但尚未触及任何 Rust 代码。 

{
  "name": "@openai/codex",
  "bin": { "codex": "bin/codex.js" },
  "type": "module",
  "engines": { "node": ">=16" }
}

3. 第 3-4 层：平台检测、二进制发现与异常降级

codex.js 的核心逻辑分三步：算 triple（process.platform + process.arch 映射为 Rust target triple，如 aarch64-apple-darwin）、找二进制（优先从对应平台 npm 包的 vendor/ 目录找，找不到回退到当前包的 vendor/）、启动子进程（spawn(binaryPath, argv.slice(2), { stdio: "inherit" })）。其中 spawn 而非 spawnSync 确保了 Node 能响应 Ctrl-C 等信号。 

const targetTriple = (() => {
  switch (platform) {
    case "darwin":
      return arch === "arm64"
        ? "aarch64-apple-darwin"
        : "x86_64-apple-darwin";
    // linux, win32 ...
  }
})();

const binaryPath = findCodexExecutable();
// findCodexExecutable → require.resolve(platformPkg)
// → vendor/{triple}/bin/codex

const child = spawn(binaryPath, process.argv.slice(2), {
  stdio: "inherit",
  env,
});

异常路径：当平台 npm optional package 未安装时，findCodexExecutable() 先后尝试 require.resolve 和本地 vendor/ 回退，均失败后抛出 Missing optional dependency @openai/codex-{platform}。Reinstall Codex: npm install -g @openai/codex@latest。此错误在 Node.js 层就已终止，不会到达 Rust 二进制。排查思路：先确认 platform/arch 是否在 PLATFORM_PACKAGE_BY_TARGET 映射表中；如果在，检查对应 platform package 是否作为 optional dependency 安装。

4. 第 5-6 层：Rust CLI 分发与 exec 子进程

原生 codex 二进制由 codex-rs/cli/src/main.rs 编译。入口 fn main() 首先调用 arg0_dispatch_or_else 处理多名称分发（同一个二进制可能以 codex-linux-sandbox 等名称调用），随后进入 cli_main。clap 解析到 Subcommand::Exec 后，将 exec_cli 与根级配置覆盖合并，调用 codex_exec::run_main(exec_cli, arg0_paths)。至此，从 npm bin 到 exec 的启动链路完成，控制权交给 exec 模块的 agent 循环。 

// codex-rs/cli/src/main.rs (精简)
fn main() -> anyhow::Result<()> {
    arg0_dispatch_or_else(move |arg0_paths| async move {
        cli_main(arg0_paths, remote_control_disabled).await
    })
}

// cli_main 中 Exec 分支
Some(Subcommand::Exec(mut exec_cli)) => {
    exec_cli.shared.inherit_exec_root_options(&interactive.shared);
    prepend_config_flags(&mut exec_cli.config_overrides,
        root_config_overrides.clone());
    codex_exec::run_main(exec_cli, arg0_paths.clone()).await?;
}

检索练习：如果用户在 Linux arm64 上运行 codex exec "test" 时报 "Unsupported platform"，请说出排查方向。如果报 "Missing optional dependency"，排查方向又是什么？两种错误的根本区别是什么？

参考答案：
"Unsupported platform" → 检查 codex.js 中 platform/arch 的 switch-case 是否覆盖该组合；可能是新架构未加入映射表，需修改 PLATFORM_PACKAGE_BY_TARGET。
"Missing optional dependency" → platform/arch 已识别但对应 npm 包未安装；检查 npm ls @openai/codex-linux-arm64 是否存在。
根本区别：前者是"不认识这个平台"，后者是"认识但找不到二进制"。前者发生在 targetTriple 计算阶段，后者发生在 findCodexExecutable 阶段。
还有疑问时，向 agent 追问具体错误栈，让它带你定位到 codex.js 的精确行。
