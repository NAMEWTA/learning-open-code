npm 到 Rust exec 调用链速查表

用途：快速定位 `codex exec` 从 npm bin 到 Rust exec 进程的每一层入口、关键函数和异常处理。基于当前本地源码快照。

层级概览

层 入口文件 关键函数 输入 输出
1. npm bin codex-cli/package.json "bin": { "codex": "bin/codex.js" } 终端命令 codex exec "hello" Node.js 进程执行 bin/codex.js
2. Node.js 入口 codex-cli/bin/codex.js 顶层脚本（ESM） process.platform、process.arch、process.argv target triple + binaryPath
3. 平台检测 codex-cli/bin/codex.js switch (platform/arch) "darwin" + "arm64" "aarch64-apple-darwin"
4. 二进制发现 codex-cli/bin/codex.js findCodexExecutable() target triple vendor/{triple}/bin/codex 的绝对路径
5. spawn 子进程 codex-cli/bin/codex.js spawn(binaryPath, args, opts) binaryPath + ["exec", "hello"] 原生 codex 进程（stdio 继承）
6. Rust CLI 入口 codex-rs/cli/src/main.rs main() → arg0_dispatch_or_else → cli_main OS 进程参数 MultitoolCli 已解析
7. 子命令分发 codex-rs/cli/src/main.rs match subcommand { Subcommand::Exec(..) => .. } Subcommand::Exec(exec_cli) codex_exec::run_main(exec_cli, arg0_paths)
8. exec 模块启动 codex-rs/exec/src/main.rs codex_exec::run_main(inner, arg0_paths) Cli { command, config_overrides, .. } agent 循环（本切片终点）

异常路径一览

异常 触发层 错误信息 排查方向
平台不支持 codex.js switch 语句 Unsupported platform: {platform} ({arch}) 检查 platform/arch 组合是否在 6 种映射中；新增架构需修改 PLATFORM_PACKAGE_BY_TARGET
二进制缺失 findCodexExecutable() Missing optional dependency {pkg}. Reinstall Codex: ... 检查平台 npm 包是否安装；npm ls {platformPkg}；重新 npm install -g
spawn ENOENT child.on("error") 系统级错误（文件不存在/无执行权限） 检查 binaryPath 是否可执行；ls -l 确认权限位
信号中断 信号转发处理器 SIGINT/SIGTERM/SIGHUP → 子进程终止 正常行为；检查子进程是否正确响应终止信号

codex-rs 独立 exec 入口说明

codex-rs/exec/src/main.rs 是 codex-exec 二进制（而非 codex）的入口。它同样使用 arg0_dispatch_or_else，当以 codex-linux-sandbox 名称调用时会切换到沙箱模式。在正常路径中，它解析 TopCli（内嵌 Cli），合并根级配置覆盖后调用同样的 codex_exec::run_main。这意味着 codex exec（通过 cli main.rs）和 codex-exec（独立二进制）共享同一套 exec 运行时。
