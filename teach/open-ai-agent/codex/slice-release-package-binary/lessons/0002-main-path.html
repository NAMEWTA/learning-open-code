主路径：GitHub Release 与 npm 发布

学习目标：15 分钟内理解 release job 如何收集多平台 artifacts、生成 npm tarballs、创建 GitHub Release，以及 publish-npm 如何通过 OIDC 发布到 npm registry。

一句话定位：当所有平台的构建和签名完成后，release job 是汇聚点。它从分散的 workflow artifacts 中收集 6 个 target 的产物，调用 stage_npm_packages.py 生成 npm tarballs，创建 GitHub Release，并根据版本号决定是否触发 npm 发布。 前置：0001 发布链路地图。

1. release job — 汇聚与打包

release job 的 needs 声明了 5 个依赖：tag-check、build、finalize-macos、build-windows、argument-comment-lint-release-assets。它使用 if: always() && ...result == 'success' 确保即使某些依赖被跳过也能正确判断。

关键步骤依次为：

  1. 下载 artifacts：用 actions/download-artifact 配合 pattern 参数，按 target triple 模式匹配下载所有平台的产物到 dist/ 目录。
  2. 生成 SHA256SUMS：扫描所有 codex-package-*.tar.gz 文件计算 SHA256，写入 dist/codex-package_SHA256SUMS。这是用户校验二进制完整性的依据。
  3. 确定 npm 发布策略：npm_publish_settings step 根据版本号决定：纯数字版本（如 0.1.0）→ 发布到 latest tag；-alpha.N 后缀 → 发布到 alpha tag 且仅平台包打平台 tag；其他格式 → 不发布 npm。
  4. 执行 stage_npm_packages.py：安装 pnpm 依赖后，调用 ./scripts/stage_npm_packages.py 为 codex、codex-responses-api-proxy、codex-sdk 三个包生成 .tgz tarballs。
  5. 创建 GitHub Release：使用 softprops/action-gh-release，上传 dist/** 全部文件，根据版本号是否含 - 决定 prerelease 标记。
# .github/workflows/rust-release.yml — release job 关键步骤
- name: Stage npm packages
  env:
    GH_TOKEN: ${{ github.token }}
    RELEASE_VERSION: ${{ steps.release_name.outputs.name }}
  run: |
    ./scripts/stage_npm_packages.py \
      --release-version "$RELEASE_VERSION" \
      --workflow-url "$workflow_url" \
      --artifacts-dir "${GITHUB_WORKSPACE}/dist" \
      --package codex \
      --package codex-responses-api-proxy \
      --package codex-sdk

2. stage_npm_packages.py — npm 包准备引擎

这个脚本是 npm 发布的核心协调器。它从 release workflow 的 artifact 中下载每个 target 的 codex-package-{target}.tar.gz（Codex 自有分发格式），解压到 vendor/{target}/ 目录，然后调用 codex-cli/scripts/build_npm_package.py 为每个包生成标准 npm tarball。

脚本的关键设计：（1）扩展映射——codex 扩展为 1 个主包 + 6 个平台包；（2）并发下载——用 ThreadPoolExecutor 并行下载和解压各 target 的 package archive；（3）平台包命名——tarball_name_for_package() 根据 CODEX_PLATFORM_PACKAGES 映射生成 codex-npm-{platform}-{version}.tgz 格式的文件名。 

# scripts/stage_npm_packages.py — 核心常量
BINARY_TARGETS = (
    "x86_64-unknown-linux-musl",
    "aarch64-unknown-linux-musl",
    "x86_64-apple-darwin",
    "aarch64-apple-darwin",
    "x86_64-pc-windows-msvc",
    "aarch64-pc-windows-msvc",
)
# 从 codex-cli/scripts/build_npm_package.py 加载
CODEX_PLATFORM_PACKAGES  # {"codex-linux-x64": ..., ...}
PACKAGE_NATIVE_COMPONENTS # {"codex": ["codex-package"], ...}

3. publish-npm — OIDC 无密钥发布

publish-npm job 使用 npm 的 OIDC（OpenID Connect）可信发布机制，无需配置 NODE_AUTH_TOKEN。其权限声明 id-token: write 允许 GitHub Actions 向 npm registry 证明身份。

发布顺序很重要：先发布 6 个平台包，最后发布主包。这是因为主包 @openai/codex 的 optionalDependencies 引用了所有平台包——npm registry 不允许依赖尚不存在的包版本。脚本通过 CODEX_PLATFORM_PACKAGES 列表确保平台包先于主包发布。若某包已存在（并发重试场景），脚本捕获 "previously published" 错误并跳过。 

# rust-release.yml — npm 发布顺序
platform_tarballs=(
  "dist/npm/codex-npm-linux-x64-${VERSION}.tgz"
  "dist/npm/codex-npm-linux-arm64-${VERSION}.tgz"
  "dist/npm/codex-npm-darwin-x64-${VERSION}.tgz"
  "dist/npm/codex-npm-darwin-arm64-${VERSION}.tgz"
  "dist/npm/codex-npm-win32-x64-${VERSION}.tgz"
  "dist/npm/codex-npm-win32-arm64-${VERSION}.tgz"
)
# 发布顺序：平台包 → 其他包 → root 包 → SDK 包
tarballs=("${platform_tarballs[@]}" "${other_tarballs[@]}" "${root_tarball}" "${sdk_tarball}")

4. 辅助发布渠道：DotSlash、WinGet、Homebrew

除 GitHub Release 和 npm 外，还有三个辅助发布渠道：

  • publish-dotslash：为 .github/dotslash-config.json 中定义的每个平台二进制创建 DotSlash 可执行条目。DotSlash 是 Meta 开源的工具，允许单个文件包含多平台二进制 URL 引用，用户下载后自动选择匹配平台的版本。
  • winget：仅在稳定版本（不含 - 的版本号）触发，使用 vedantmgoyal9/winget-releaser 自动向 openai-oss-forks/winget-pkgs 提交 PR，由 WinGet 社区审核合并。
  • deploy-dev-website：稳定版本触发 Vercel deploy hook，更新 developers.openai.com 上的 config schema 文档。

场景判断：如果发布 rust-v0.2.0-beta.1 后，用户报告 npm install -g @openai/codex@latest 仍安装 0.1.0，但 @openai/codex@beta 已是 0.2.0-beta.1。这是正常行为还是 bug？为什么？

参考答案：正常行为。npm_publish_settings step 判断版本号 0.2.0-beta.1 不匹配纯数字正则 ^[0-9]+\\.[0-9]+\\.[0-9]+$，也不匹配 alpha 正则 ^[0-9]+\\.[0-9]+\\.[0-9]+-alpha\\.[0-9]+$，因此 should_publish_npm=false，整个 publish-npm job 被跳过。GitHub Release 会标记为 prerelease。如果用户期望 npm 发布 beta 版本，需要将版本号改为 0.2.0-alpha.N 格式或扩展 npm_publish_settings 的判断逻辑。
还有疑问？向 agent 追问 npm_publish_settings step 的完整正则匹配逻辑。
