发布链路地图：从 git tag 到多平台二进制包

学习目标：15 分钟内画出从 git push origin rust-v0.1.0 到 GitHub Release + npm publish 的完整时序图，说出 10 个关键 job 的职责边界。

一句话定位：本切片追踪一条 tag 推送如何触发 GitHub Actions，经 prepare、build（4 平台 x 2 bundle）、sign（macOS AKV + Linux cosign）、package、release、npm publish 共 10 个阶段，最终到达用户手中的多平台二进制包。 前置知识：L1 构建与发布模块导览 | L1 CLI 包装层模块导览。

2. 入口点：tag 推送与版本校验

整个发布链路由 git tag 推送触发。Workflow rust-release 监听 rust-v*.*.* 格式的 tag。首个 job tag-check 做三项校验：（1）确认 GITHUB_REF_TYPE 为 tag；（2）用正则 ^rust-v[0-9]+\\.[0-9]+\\.[0-9]+(-(alpha|beta)(\\.[0-9]+)?)?$ 验证 tag 格式；（3）提取 tag 中的版本号与 codex-rs/Cargo.toml 的 version 字段对比，不一致则失败。只有 tag-check 通过后，后续 build 和 release job 才会执行。 

# .github/workflows/rust-release.yml — tag-check job 核心逻辑
tag_ver="${GITHUB_REF_NAME#rust-v}"
cargo_ver="$(grep -m1 '^version' codex-rs/Cargo.toml \
              | sed -E 's/version *= *"([^"]+)".*/\1/')"
[[ "${tag_ver}" == "${cargo_ver}" ]] \
  || { echo "❌  Tag ${tag_ver} ≠ Cargo.toml ${cargo_ver}"; exit 1; }

异常路径：若 tag 格式不匹配，CI 直接失败，不会进入 build。常见错误：tag 缺少 rust-v 前缀、版本号多了一个 v 前缀、或 Cargo.toml 版本未同步更新。排查方向：运行 git tag -l 'rust-v*' 检查已有 tag 格式，对比 grep version codex-rs/Cargo.toml 输出。

3. 构建矩阵：4 平台 x 2 bundle 的并行构建

tag-check 通过后，build job 以 8 路并行 matrix 执行：macOS arm64/x86_64 各分 primary 和 app-server bundle；Linux x64/arm64 musl 同样各分 primary 和 app-server。每个 cell 产出：二进制文件、符号归档、codex-package-{target}.tar.gz（Codex 自有的分发格式）、Python runtime wheel。

macOS 与 Linux 的关键差异：macOS 构建后只做符号剥离与压缩，暂不签名——签名由后续 sign-macos-binaries job 在 ubuntu runner 上通过 AKV PKCS11 远程完成。Linux 构建后直接调用 cosign 签名，生成 .sigstore 文件。Linux musl 构建还需额外步骤：安装 Zig（交叉编译工具链）、构建 bwrap 沙箱二进制并计算 SHA256 摘要供 codex 嵌入校验。

4. 签名与公证链路

macOS 签名分四步：（1）build 产出 unsigned 二进制上传为 artifact；（2）sign-macos-binaries 在 ubuntu runner 上通过 Azure Key Vault PKCS11 远程签名 + Apple notarization；（3）package-macos 在 macos runner 上 codesign --verify 验证签名，构建 DMG；（4）sign-macos-dmg 对 DMG 签名 + 公证 + staple。finalize-macos 最后挂载 DMG 再次验证所有二进制签名。整个 macOS 签名链涉及 5 个连续 job，任何一步失败都会阻断发布。

Linux 签名简单得多：在 build job 内直接调用 .github/actions/linux-code-sign 对每个二进制执行 cosign 签名。Windows 签名使用 Azure Trusted Signing，通过 .github/actions/windows-code-sign 完成。

检索练习：macOS 签名链路为什么需要 5 个串行 job 而不能合并？如果 sign-macos-binaries 失败报 "AKV authentication error"，排查方向是什么？

参考答案：
5 个 job 分离的原因：（1）签名需要 codesigning GitHub Environment 保护 secrets，与构建环境隔离；（2）AKV PKCS11 工具链仅在 Linux 上可用，签名本身在 ubuntu runner 完成；（3）DMG 构建需要 macOS runner（hdiutil），签名后还需在 macOS 上验证（codesign 命令只在 macOS 上可用）；（4）DMG 签名是独立步骤，因为 DMG 文件在 macOS 上构建后才存在。
AKV 认证失败排查：检查 codesigning environment 的 secrets 是否过期（Azure client secret 有效期通常 6-24 个月）；检查 key vault 的 access policy 是否包含当前 service principal 的 sign 权限。
