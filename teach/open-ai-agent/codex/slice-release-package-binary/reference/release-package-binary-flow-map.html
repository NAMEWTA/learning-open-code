发布链路速查表

从 git tag 推送到多平台用户下载的完整链路速查。用于发布失败时快速定位、或为类似项目搭建 CI/CD 时作为参考清单。

1. Job 执行顺序与职责

顺序 Job 名称 Runner 输入 输出 / 职责
0 rust-release-prepare ubuntu-latest schedule / workflow_dispatch 更新 models.json，有变更时自动提 PR。独立于发布链路，定时运行
1 tag-check ubuntu-latest git tag rust-v*.*.* 校验 tag 格式与 Cargo.toml version 一致。阻塞所有后续 job
2 build (matrix) macos-15-xlarge / linux-x64-xl / linux-arm64 源码 checkout 各平台二进制（.zst + .tar.gz）、符号归档、Python runtime wheel。macOS 产出 unsigned artifacts
3 sign-macos-binaries ubuntu-latest unsigned macOS 二进制 AKV PKCS11 远程签名 + Apple 公证，产出 signed binaries
4 package-macos macos-15-xlarge signed macOS 二进制 codesign 验证、构建 DMG、打包 codex-package archive
5 sign-macos-dmg ubuntu-latest unsigned macOS DMG DMG 签名 + 公证 + staple，产出 signed DMG
6 finalize-macos macos-15-xlarge packaged artifacts + signed DMG 挂载 DMG、验证所有二进制签名与 entitlements，最终产出就绪的 macOS artifacts
2p build-windows windows-x64 / windows-arm64 源码 checkout Windows 二进制（.exe + .zip + .zst）、Azure Trusted Signing 签名、Python runtime wheel
7 release ubuntu-latest 所有平台 artifacts 汇聚 artifacts → SHA256SUMS → npm tarballs → GitHub Release。输出 should_publish_npm 和 npm_tag
8 publish-npm ubuntu-latest GitHub Release 中的 npm tarballs OIDC 认证 → 按序发布平台包 + 主包 + SDK 包到 npm registry
8p publish-dotslash ubuntu-latest GitHub Release assets 为每个平台二进制生成 DotSlash 可执行条目
8p winget ubuntu-latest GitHub Release assets 自动向 winget-pkgs 提交 PR（仅稳定版本）

5. 常见发布故障排查

症状可能原因排查入口
tag push 后 CI 未触发 tag 名称不匹配 rust-v*.*.* 检查 rust-release.yml 的 on.push.tags
tag-check 失败 "Tag ≠ Cargo.toml" Cargo.toml version 未同步更新 grep version codex-rs/Cargo.toml 对比 tag
macOS build 超时 (90min) rusty_v8 / webrtc 编译耗时过长 检查 setup-rusty-v8 action 的缓存命中
sign-macos-binaries 失败 AKV secrets 过期或权限不足 检查 codesigning environment secrets
npm publish HTTP 409 版本已存在或并发发布冲突 脚本自动跳过 "previously published" 错误
npm 安装后找不到二进制 optionalDependencies 未安装 npm ls @openai/codex-{platform}
GitHub Release 缺少某平台资产 对应 build matrix cell 失败 检查 Actions 日志中对应 target + bundle 的 build job
