工具定义与执行抽象：15 分钟模块导览

学习目标：能从一次模型工具调用追到 Codex 的 handler 与 runtime，并说清每一层负责什么。

架构位置：先看 L0 项目地图；完整清单见 工具执行参考。

1. 模块职责

这个模块不是“某一个工具”，而是一组把模型能力接到宿主环境的抽象。`codex-tools` 定义模型可见的工具规格、schema、输出协议和 `ToolExecutor` trait；`codex-core/src/tools` 把这些规格注册进会话，接收 Responses API 返回的 tool call，再交给本地 handler 或外部扩展执行。

最关键的边界是：共享 crate 不知道 session、审批、沙箱；core 知道这些上下文，并在执行前后加上 hook、telemetry、生命周期事件、权限检查和输出转换。

codex-rs/ ├── tools/ ← 共享工具协议 crate │ ├── src/tool_spec.rs ← Responses API 工具形态 │ ├── src/tool_executor.rs ← handler 共同契约 │ └── src/tool_definition.rs ← 动态/插件工具元数据 └── core/src/tools/ ← 会话内注册、路由、执行 ├── spec_plan.rs ← 决定本轮有哪些工具 ├── router.rs ← 模型输出转 ToolCall ├── registry.rs ← 执行包裹层 ├── handlers/ ← 工具入口 └── runtimes/ ← shell / patch 等执行后端

2. 对外接口看三件事

读一个工具时先找三件事：`tool_name()` 决定调用名，`spec()` 决定模型看到的参数，`handle()` 决定真实执行。它们被统一收进 `ToolExecutor<ToolInvocation>`。

pub trait ToolExecutor<Invocation> {
    fn tool_name(&self) -> ToolName;
    fn spec(&self) -> ToolSpec;
    fn exposure(&self) -> ToolExposure { ToolExposure::Direct }
    fn handle(&self, invocation: Invocation) -> ToolExecutorFuture<'_>;
}

`ToolExposure` 决定工具是否直接暴露、延迟搜索、只对模型暴露或仅注册分发。`spec_plan.rs` 再按 feature、模型能力、环境数量合成最终 `model_visible_specs`。

3. 最小执行序列

以 `shell_command` 为例，模型调用不会直接跑进程，而是经过一条稳定管线：

ResponseItem::FunctionCall → ToolRouter::build_tool_call → ToolInvocation → ToolRegistry::dispatch_any_with_terminal_outcome → PreToolUse hook → ShellCommandHandler::handle_call → run_exec_like → ToolOrchestrator + ShellRuntime → sandbox / approval / execute_env → FunctionToolOutput → PostToolUse hook → ResponseInputItem::FunctionCallOutput

`apply_patch` 走同一 registry 包裹层，但 payload 是 custom/freeform，handler 会先解析和验证 patch，再用 `ApplyPatchRuntime` 对选中的环境文件系统执行。shell 命令里出现 apply_patch 时，`run_exec_like` 还会先尝试拦截成内部 patch 流程。

4. 暂停回忆

不看上文，回答三个问题：

  1. 哪个文件决定本轮会有哪些工具进入 registry？
  2. `ToolExecutor` 的哪两个方法分别对应“模型可见形态”和“真实执行”？
  3. shell 工具执行前，为什么还要经过 registry 而不是直接调用 runtime？

参考答案：1. `core/src/tools/spec_plan.rs`。2. `spec()` 对应模型可见形态，`handle()` 对应真实执行。3. registry 负责统一处理工具查找、payload 类型校验、Pre/Post hook、telemetry、生命周期事件、取消语义和输出转换；runtime 只处理具体执行。

主要阅读来源：`codex-rs/tools/src/tool_executor.rs` 和 `codex-rs/core/src/tools/registry.rs`。如果后续追问，可以让 agent 带你从某一个具体工具的 `spec()` 开始单步读到 runtime。
