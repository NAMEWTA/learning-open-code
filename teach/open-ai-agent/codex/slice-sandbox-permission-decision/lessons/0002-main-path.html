主路径详解：execpolicy 决策 → 沙箱创建

学习目标：追踪一条正常执行路径中 execpolicy 三态决策、permissions 编译和沙箱选择的完整代码流。

前置：第 1 节 — 链路地图

1. execpolicy 三态决策

Codex 的执行策略引擎定义三种决策结果，优先级从低到高：

codex-rs/execpolicy/src/decision.rs ├── pub enum Decision { │ ├── Allow ← 命令可直接执行，无需审批 │ ├── Prompt ← 请求用户显式批准；approval_policy="never" 时直接拒绝 │ └── Forbidden ← 命令被无条件阻塞 }

策略匹配流程：Policy::check() 先按命令首 token 精确匹配规则（match_exact_rules），若未命中则尝试 match_host_executable_rules 解析绝对路径后按 basename 匹配；若仍未命中，调用 heuristics_fallback 返回启发式决策。最终结果由 Evaluation::from_matches() 取所有匹配规则中优先级最高的 Decision。

// execpolicy/src/policy.rs — check() 核心逻辑
pub fn check<F>(&self, cmd: &[String], heuristics_fallback: &F) -> Evaluation
where F: Fn(&[String]) -> Decision,
{
    let matched_rules = self.matches_for_command_with_options(
        cmd,
        Some(heuristics_fallback),
        &MatchOptions::default(),
    );
    Evaluation::from_matches(matched_rules)
}

2. 权限 profile 编译

引擎通过 compile_permission_profile_selection() 将配置转为运行时策略。首先判断是否为内置 profile：

// core/src/config/permissions.rs — 内置 profile 匹配
pub(crate) fn builtin_permission_profile(
    profile_name: &str,
    workspace_write: Option<&SandboxWorkspaceWrite>,
) -> Option<PermissionProfile> {
    match profile_name {
        BUILT_IN_READ_ONLY_PROFILE => Some(PermissionProfile::read_only()),
        BUILT_IN_WORKSPACE_PROFILE => Some(match workspace_write {
            Some(config) => PermissionProfile::workspace_write_with(/*...*/),
            None => PermissionProfile::workspace_write(),
        }),
        BUILT_IN_DANGER_FULL_ACCESS_PROFILE => Some(PermissionProfile::Disabled),
        _ => None,
    }
}

若为用户自定义 profile，走 resolve_permission_profile() 解析 TOML 的 extends 继承链，循环检测无误后执行 merge_permission_profiles()（父先子后，子覆盖父），再由 compile_filesystem_permission() 逐条编译成 FileSystemSandboxEntry。

3. 沙箱类型选择决策树

SandboxManager::select_initial() 是沙箱类型选择的核心入口。决策路径如下：

// sandboxing/src/manager.rs — select_initial()
pub fn select_initial(
    &self,
    file_system_policy: &FileSystemSandboxPolicy,
    network_policy: NetworkSandboxPolicy,
    pref: SandboxablePreference,
    windows_sandbox_level: WindowsSandboxLevel,
    has_managed_network_requirements: bool,
) -> SandboxType {
    if self.should_sandbox(/*...*/) {
        get_platform_sandbox(windows_sandbox_level != WindowsSandboxLevel::Disabled)
            .unwrap_or(SandboxType::None)
    } else {
        SandboxType::None
    }
}

should_sandbox() 的三条分支：

  • Forbid — 强制不用沙箱，返回 false
  • Require — 强制用沙箱，返回 true
  • Auto — 委托给 should_require_platform_sandbox()：当有 managed network 要求时必用；当 FileSystemSandboxKind::ExternalSandbox 时不需；当 Restricted 且无全盘写权限时需用

get_platform_sandbox() 按编译目标平台返回对应类型：macOS 返回 MacosSeatbelt，Linux 返回 LinuxSeccomp，Windows 在启用沙箱时返回 WindowsRestrictedToken（否则 None）。

4. 异常路径：权限拒绝与沙箱降级

4.1 权限拒绝路径

当 execpolicy 返回 Decision::Forbidden 时，ToolOrchestrator 在阶段 [1] 后直接终止链路。工具的 approval 回调收到 Forbidden 后构造拒绝响应，不进入权限编译和沙箱创建。代码路径：Policy::check() → Evaluation.decision == Forbidden → 工具调用被阻塞。

4.2 沙箱降级路径

当权限 profile 为 danger-full-access（PermissionProfile::Disabled — 文件系统非受限 + 网络启用）时，should_require_platform_sandbox() 对 FileSystemSandboxKind::Unrestricted 返回 false。此时 select_initial() 返回 SandboxType::None，transform() 中对应的 match 分支仅做简单的 argv 字符串转换，不加任何沙箱包装器。

判断题

以下说法是否正确？"当 SandboxablePreference::Auto 且 NetworkSandboxPolicy::Enabled 时，should_require_platform_sandbox() 一定返回 true。"

参考答案

错误。 当 file_system_policy.kind == FileSystemSandboxKind::Unrestricted 且网络启用时，should_require_platform_sandbox() 返回 false（因为 Unrestricted 变体在 match 中排在 Restricted 之前，直接返回 false）。只有 Restricted + 非全盘写权限 时才会因网络启用而返回 true。
