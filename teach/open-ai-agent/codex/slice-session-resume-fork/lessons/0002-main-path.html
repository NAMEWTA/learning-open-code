冷 Resume 主路径：从 JSONL 到 Session 再生

学习目标：追踪 ThreadStore::resume_thread() -> load_history() -> reconstruct_history_from_rollout() 的完整调用链，理解反向扫描算法。

前置：链路全景地图

1. ThreadStore — 入口契约

一切持久化操作经过 ThreadStore trait。它不是实现，而是一组存储中立的接口契约。理解这 3 个方法就掌握了入口：

// thread-store/src/store.rs — ThreadStore trait（摘录） pub trait ThreadStore: Any + Send + Sync { /// 创建新的 live thread，分配 ThreadId 和 rollout 路径 fn create_thread(&self, params: CreateThreadParams) -> ThreadStoreFuture<'_, ()>; /// 重新打开已有 thread，准备追加新 item fn resume_thread(&self, params: ResumeThreadParams) -> ThreadStoreFuture<'_, ()>; /// 加载持久化历史 → 返回 StoredThreadHistory（含 Vec<RolloutItem>） fn load_history( &self, params: LoadThreadHistoryParams ) -> ThreadStoreFuture<'_, StoredThreadHistory>; /// 将 thread 标记为 archived，移动文件到 archived_sessions/ fn archive_thread(&self, params: ArchiveThreadParams) -> ThreadStoreFuture<'_, ()>; }

create_thread 和 resume_thread 的区别：前者创建全新的 thread（分配新的 ThreadId 和 rollout 路径），后者重新打开已存在的 thread（保持相同的 ThreadId 和 rollout 文件），允许追加新的对话内容。

2. reconstruct_history_from_rollout — 反向扫描算法

这是冷 resume 的核心——从一个扁平的 JSONL 行数组中重建出结构化的 Session 状态。算法采用从新到旧反向扫描，因为 rollout 末尾是最新状态，只需要找到最近的 relevant metadata 即可。

算法的输出是 RolloutReconstruction，包含 5 个关键字段：

字段类型含义
historyVec<ResponseItem>重建后的完整对话历史
previous_turn_settingsOption<PreviousTurnSettings>上一轮使用的 model / comp_hash
reference_context_itemOption<TurnContextItem>最近的 TurnContext 基线（用于 resume 水合）
world_state_baselineOption<WorldStateSnapshot>恢复的世界状态快照
window_numberu64上下文窗口编号
// core/src/session/rollout_reconstruction.rs — 反向扫描主循环（摘录） for (index, item) in rollout_items.iter().enumerate().rev() { match item { RolloutItem::Compacted(compacted) => { // 检查是否有 replacement_history → 截断 rollout_suffix if let Some(replacement_history) = &compacted.replacement_history { active_segment.base_replacement_history = Some(replacement_history); rollout_suffix = &rollout_items[index + 1..]; // ← 截断点 } // 记录 window_number、first_window_id 等 } RolloutItem::TurnContext(ctx) => { // 记录上一轮的 model / comp_hash / realtime_active active_segment.previous_turn_settings = Some(PreviousTurnSettings { ... }); // 记录最近的 TurnContextItem 作为 reference_context_item } // ... Rollback、UserMessage、WorldState 等其他 item 类型的处理 } // 提前终止：当 replacement_history + previous_turn_settings // + reference_context_item 三者都已确定时，停止扫描 if base_replacement_history.is_some() && previous_turn_settings.is_some() && !matches!(reference_context_item, TurnReferenceContextItem::NeverSet) { break; } }

关键性能优化：一旦找到最新的 replacement_history（压缩后的完整历史基线）、previous_turn_settings 和 reference_context_item 三者，扫描立即终止（break），不需要解析全部 JSONL。长会话的 resume 只扫描最近的少量行。

3. SQLite Backfill — 从 JSONL 到结构化查询

Resume 重建的是 Session 的运行时状态。而 SQLite backfill 是独立的后台任务，将 rollout 中的结构化信息（如 ThreadMetadata、agent_jobs）提取到 SQLite，供 list/search 等查询接口使用。

Backfill 的核心函数是 apply_rollout_item()：

// state/src/extract.rs — 单条 rollout item 的 SQLite 提取 pub fn apply_rollout_item( conn: &mut SqliteConnection, thread_id: ThreadId, item: &RolloutItem, anchor: Anchor, ) -> Result<ExtractionOutcome> { match item { RolloutItem::SessionMeta(meta) => { // INSERT OR REPLACE INTO threads (...) // 写入 model、cwd、permission_profile 等 } RolloutItem::Compacted(compacted) => { // UPDATE threads SET window_number = ... } // ... 其他 item 类型的提取 } }

Backfill 和 resume 是两条并行路径：resume 用 reconstruct_history_from_rollout() 重建运行时 Session，backfill 用 apply_rollout_item() 更新 SQLite 查询索引。两者都从同一个 JSONL 文件读取，但服务于不同目的。

4. 暂停回忆

不看上文，回答三个问题：

  1. ThreadStore::create_thread() 和 resume_thread() 的核心区别是什么？
  2. reconstruct_history_from_rollout() 为什么采用反向扫描？什么条件下会提前终止？
  3. SQLite backfill 和 Session resume 都读 JSONL，它们的关系是什么？

参考答案：1. create 分配新的 ThreadId + rollout 路径，resume 重新打开已有的 ThreadId 和 rollout 文件以追加新内容。2. 因为最新的 rollout 行携带了重建所需的最新元数据，反向扫描可以最快找到这些信息。当 replacement_history + previous_turn_settings + reference_context_item 三者都确定时提前终止。3. 它们是并行但独立的路径：resume 重建运行时 Session 状态（内存中的 history），backfill 提取结构化元数据到 SQLite（供查询接口使用）。两者无直接依赖关系。
