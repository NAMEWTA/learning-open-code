异常路径：Resume 失败、Fork 冲突、Archive 回滚

学习目标：理解三条主要异常路径的触发条件、代码位置和恢复策略，能在日志中识别对应错误。

前置：主路径详解

1. Resume 时 Session 不存在

用户执行 codex resume <thread_id> 或 app-server 收到 thread/resume 请求时，可能出现以下失败场景：

失败场景触发条件代码位置返回错误
Thread 不在内存中且 rollout 文件缺失 thread_id 对应 .jsonl 被手动删除或从未创建 thread_processor.rs::thread_resume_inner() JSONRPCError: "failed to load thread"
Thread 正在关闭中 pending_thread_unloads 集合中包含该 thread_id thread_processor.rs:2644-2660 JSONRPCError: "thread is closing; retry after closed"
Rollout JSON 解析失败 .jsonl 文件部分损坏或格式不正确 rollout_reconstruction.rs 主循环 serde_json::Error → 内部错误
// app-server thread_processor.rs — 冷 resume 的三条路径（摘录） let resume_result = if let Some(history) = history { // 路径 A：客户端直接提供了 history array self.resume_thread_from_history(history.as_slice()).await } else if let Some(mut stored_thread) = stored_thread_from_running_probe { // 路径 B：thread 已在 store 中有记录 self.stored_thread_to_initial_history(&mut stored_thread).await } else { // 路径 C：从 rollout 文件加载（最常规的冷 resume） self.resume_thread_from_rollout(&thread_id, path.as_ref()).await }; let (thread_history, resume_source_thread) = match resume_result { Ok(value) => value, Err(error) => { self.outgoing.send_error(request_id, error).await; return Ok(()); // ← 优雅终止，不崩整个 app-server } };

恢复策略：路径 C 失败时不会降级尝试其他路径，而是直接返回错误给客户端。客户端可以重新发送请求（因为错误不改变磁盘状态）。如果 rollout 文件确实损坏且无备份，该 session 无法恢复——但 SQLite 中的 ThreadMetadata 可能仍然存在，list 接口还能看到这个 thread。

2. Fork 冲突

Fork 操作从源 thread 的历史中截断创建新 thread。可能出现的冲突场景：

冲突场景触发条件处理方式
源 thread 未提供 history source_thread.history 为 None 返回 internal_error("did not include persisted history")
last_turn_id 指定的 turn 不存在 truncate_rollout_after_turn_id() 找不到匹配 返回 core_thread_write_error
源 thread 已被 archive 源 thread 的 archived_at 不为 NULL fork 仍然成功（从 archived 源 fork 是允许的）
// app-server thread_processor.rs — fork 核心逻辑（摘录） let history_items = if let Some(last_turn_id) = last_turn_id.as_deref() { Arc::new( truncate_rollout_after_turn_id(&history_items, last_turn_id) .map_err(|err| core_thread_write_error("truncate thread for fork", err))? ) } else { Arc::new(history_items) // ← 不截断，fork 完整历史 }; // 持久化 fork 写独立 rollout；临时 fork 仅内存持有 let mut thread = if session_configured.rollout_path.is_some() { let stored_thread = self .read_stored_thread_for_new_fork(thread_id, include_turns).await?; self.stored_thread_to_api_thread(stored_thread, ...) } else { // ephemeral fork：不写文件，从源 history 构建 UI 展示 let mut thread = build_thread_from_snapshot(...); thread.forked_from_id = Some(source_thread_id.to_string()); // ← 记录来源 thread };

关键设计：ephemeral fork 不写 rollout 文件。它的 ThreadId 仅存在于 app-server 内存中，重启后消失。如果 fork 创建失败（如 ThreadManager 内部错误），整个操作回滚——不会留下半残的新 thread。

3. Archive 失败与子树回滚

Archive 操作不仅归档目标 thread，还会归档其通过 fork 产生的所有后代（spawned descendants）。这是一个多步骤操作，可能出现部分成功：

// app-server thread_processor.rs — archive 子树处理（摘录） async fn thread_archive_response(&self, params) -> Result { let thread_ids = self.state_db_spawn_subtree_thread_ids(thread_id).await?; // thread_ids: [parent, descendant_1, descendant_2, ...] // Step 1：收集所有未归档的 thread for each thread in [parent, ...descendants] { if thread.archived_at.is_none() { archive_thread_ids.push(thread_id); } } // Step 2：逐条归档。父 thread 失败 → 整体返回错误 self.thread_store.archive_thread(parent).await?; // Step 3：后代归档。单个失败只 warn，不阻断整体 for descendant in descendants.rev() { match self.thread_store.archive_thread(descendant).await { Ok(()) => archived_thread_ids.push(...), Err(err) => warn!("failed to archive descendant: {err}"), // ← 仅警告 } } }

失败语义：

  • 父 thread 归档失败 → 整体操作返回错误，后代不归档，RPC 返回 error response。
  • 后代归档失败 → 父 thread 已归档（不回滚），失败的后代保持未归档状态。app-server 仅打印 warn 日志。这是有意设计：archive 的原子性边界是单个 thread，而非整个子树。

归档的具体行为在 ThreadStore 实现中：将 rollout 文件从 sessions/ 移动到 archived_sessions/，并在 SQLite 中设置 archived_at 时间戳。unarchive 是反向操作——移回 sessions/ 并清除 archived_at。

4. 暂停回忆

判断以下说法是否正确：

  1. 冷 resume 失败时，app-server 会崩溃重启。
  2. Fork 一个已 archive 的 thread 会失败。
  3. Archive 父 thread 成功后，如果某个后代归档失败，整个操作会回滚。
  4. Ephemeral fork 的 ThreadId 在 app-server 重启后不再有效。

参考答案：1. 错误——app-server 仅返回 error response 给客户端，不会崩溃。2. 错误——从 archived 源 fork 是允许的，fork 操作不检查 archived_at。3. 错误——父 thread 已归档的不回滚，后代失败仅 warn 日志。4. 正确——ephemeral fork 不写 rollout 文件，只存在于 app-server 内存中。
