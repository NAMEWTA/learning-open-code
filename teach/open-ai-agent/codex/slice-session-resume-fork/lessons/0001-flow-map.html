1. 链路全景：从创建到归档的 5 个阶段

一个 Codex 会话从诞生到退休，经历 5 个离散的持久化阶段。每个阶段涉及不同的 crate 和持久化介质：

用户启动 (CLI / VSCode / Atlas / ChatGPT) │ ▼ [1] Session 发现与创建 扫描 sessions/ 目录下已存在的 rollout → 列出可选 session 或创建新 ThreadStore::create_thread() → 分配 ThreadId、创建 rollout 路径 所属：cli/src/main.rs, rollout/src/list.rs, thread-store/src/store.rs │ ▼ [2] Rollout JSONL 实时写入 RolloutRecorder::append() → 每条 ResponseItem、EventMsg 追加为一行 JSON 文件路径：{codex_home}/sessions/{date}/{thread_id}.jsonl 所属：rollout/src/recorder.rs │ ▼ [3] SQLite Backfill 镜像 StateRuntime 后台任务扫描 rollout → extract rollout_item → 写入 state_5.sqlite 从 JSONL 中提取：ThreadMetadata、Window 编号、agent_jobs 等 所属：state/src/runtime.rs (backfill), state/src/extract.rs │ ▼ [4] Resume 恢复 & Fork 分支 Resume: reconstruct_history_from_rollout() 反向扫描 JSONL → 重建 Session::history Fork: 基于源 thread 的 rollout_items 截断 → 创建新 ThreadId → 独立 rollout 所属：core/src/session/rollout_reconstruction.rs, app-server thread_processor.rs │ ▼ [5] Archive 归档 ThreadStore::archive_thread() → 文件移动到 archived_sessions/ 子树归档：连同 spawned descendants 一起标记为 archived 所属：app-server thread_processor.rs, rollout/src/list.rs (find_archived_thread_path)

2. 四层持久化介质的职责边界

Codex 并非只用一种存储。不同生命周期阶段使用不同的持久化介质，各有侧重：

持久化介质Crate / 文件职责特点
JSONL Rollout rollout/src/recorder.rs 逐行追加 response item 和 event，是会话的完整事件日志 追加写、人类可读、zstd 压缩、用于 replay 重建
SQLite state_5.sqlite state/src/runtime.rs 从 rollout 提取结构化元数据（ThreadMetadata、job、goal） 快速查询、支持 backfill 增量更新、有 migration
SQLite goals_1.sqlite state/src/runtime.rs 存储 thread goal 追踪数据 独立数据库文件，避免锁竞争
SQLite memories_1.sqlite state/src/runtime.rs 存储跨会话的持久化记忆 独立数据库文件，支持清空操作

4. 暂停回忆

不看上文，回答三个问题：

  1. Codex 使用了哪 4 种持久化介质？各自存放什么数据？
  2. 冷 resume 和热 resume 的区别是什么？代码中如何判断走哪条路径？
  3. Fork 时如果指定了 last_turn_id，历史数据如何处理？

参考答案：1. JSONL Rollout（完整事件日志）、state_5.sqlite（结构化元数据）、goals_1.sqlite（goal 追踪）、memories_1.sqlite（持久化记忆）。2. 冷 resume 是 Thread 不在内存中，需要从 JSONL 文件 load_history() 然后 reconstruct_history_from_rollout() 重建整个 Session；热 resume 是 Thread 已加载，直接附加 listener 即可。代码通过 ThreadManager::get_thread() 尝试获取已有 thread 来判断。3. 通过 truncate_rollout_after_turn_id() 截断源 thread 的 rollout_items，只保留指定 turn 及之前的历史，然后基于截断后的 items 创建新 fork。
