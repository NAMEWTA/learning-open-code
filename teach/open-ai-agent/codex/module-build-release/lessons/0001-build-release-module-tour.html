Bazel、CI 与发布链路：从源码到多平台二进制

学习目标：15 分钟内画出 Codex 从 Bazel/Cargo/pnpm 三套构建系统，经 CI 多平台验证，到 npm/Homebrew 发布的最小链路。

一句话定位：Codex 用 Bazel 做统一构建与远程执行、Cargo 做本地 Rust 开发、pnpm 管理 JS 包发布；每个工具在发布链路中承担不同阶段，不要混用。 主要来源：MODULE.bazel、.bazelrc、.github/workflows/rust-release.yml。

1. 三套构建系统，各有其责

Codex 不是"一个构建系统打天下"。日常 Rust 开发用 just 调用 Cargo（just test、just clippy），前端/CLI 包装用 pnpm workspace，但 CI 和正式发布统一走 Bazel。这三者的边界是：

  • Bazel：CI 统一构建入口、远程缓存/执行（BuildBuddy）、跨平台交叉编译、发布二进制构建。
  • Cargo：本地开发迭代，justfile 中的 test/clippy/fix/bench 等命令。
  • pnpm：管理 codex-cli/ 和 sdk/typescript/ 的 JS 依赖，prettier 格式化，以及 npm 发布打包。

2. Bazel 模块化构建与平台矩阵

MODULE.bazel 是 Bazel 构建的入口。它声明了 LLVM 工具链、rules_rs (Rust)、apple_support 等核心依赖，并对多个上游打了补丁（patches/ 目录）以适配 Codex 的特殊需求。Rust 工具链注册了 nightly（用于 proc-macro 和 argument-comment-lint）和 stable（1.95.0）两套。 

nightly_rust.toolchain(
    dev_components = True,
    edition = "2024",
    versions = ["nightly/2025-09-18"],
)
# Windows 双 ABI：MSVC 用于 exec 工具，gnullvm 用于 lint 目标
nightly_rust.repository_set(
    name = "rust_windows_x86_64",
    exec_compatible_with = ["@platforms//os:windows", "windows_msvc"],
    exec_triple = "x86_64-pc-windows-msvc",
    target_compatible_with = ["@platforms//os:windows", "windows_msvc"],
    target_triple = "x86_64-pc-windows-msvc",
)
nightly_rust.repository_set(
    name = "rust_windows_x86_64",
    target_compatible_with = ["@platforms//os:windows", "windows_gnullvm"],
    target_triple = "x86_64-pc-windows-gnullvm",
)

.bazelrc 按平台拆分 CI 配置：Linux 全远程构建/测试（ci-linux），macOS 远程构建 + 本地测试（ci-macos），Windows 通过 Linux RBE 交叉编译 + 本地测试（ci-windows-cross）。rbe.bzl 定义了基于 Docker 容器的远程执行平台，按 x86_64/arm64 选择不同镜像。

3. CI 多平台验证流水线

发布流程由 rust-release.yml 驱动，在推送 rust-v*.*.* tag 时触发。首先 tag-check 验证 tag 版本与 codex-rs/Cargo.toml 中 workspace 版本一致。然后 build job 用矩阵策略并行编译 10 个 target：

  • macOS arm64/x86_64，各分 primary 和 app-server 两 bundle
  • Linux musl arm64/x86_64，含 bwrap 沙箱二进制
  • Windows MSVC arm64/x86_64（由独立 rust-release-windows.yml 处理）

每个 build 完成符号归档与 strip，生成 dSYM（macOS）和压缩包，上传为 workflow artifact。CI 还通过 justfile 中的 build-for-release 命令暴露了 Bazel 侧的统一构建入口：bazel build //codex-rs/cli:release_binaries。

4. 从 tag 到二进制：发布链路练习

发布准备由 rust-release-prepare.yml 和两个 Python 脚本协作完成：build_codex_package.py 组装规范包目录并可选地创建压缩包；stage_npm_packages.py 从 GitHub Release 拉取各平台的 workflow artifact，为 @openai/codex 及 optional platform packages 准备 npm 发布内容。

完整链路可总结为：git tag push 触发 CI 矩阵构建，各平台二进制上传为 artifact，stage_npm_packages.py 拉取所有平台二进制并打包为 npm optional packages，最后 @openai/codex 的 bin/codex.js 按平台选择正确的原生二进制。

暂停回忆：下面三个场景分别应使用哪个构建工具（Bazel / Cargo / pnpm）？

  1. 本地修改了一行 Rust 代码，想快速跑对应 crate 的单元测试
  2. 检查 codex-cli/ 的 JS 代码格式
  3. 为 Linux arm64 musl 目标生成发布用二进制
参考答案：1 用 just test -p <crate>（底层是 Cargo nextest）；2 用 pnpm run format；3 由 CI rust-release.yml 在 aarch64-unknown-linux-musl target 上构建，或本地通过 Bazel 交叉编译。仍不确定时，向 agent 追问任意一个构建工具的完整命令行示例。
