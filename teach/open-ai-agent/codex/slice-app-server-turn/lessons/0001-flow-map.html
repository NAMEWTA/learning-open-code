链路地图：从 JSON-RPC 请求到 CodexThread 执行

本课只训练一个判断：当 SDK 发出 thread/start + turn/start 时，请求穿过哪七层，每层的入口函数是什么。学完后你能画出完整时序图。

学习目标：复述 7 层调用链的入口函数 前置：L1 app-server 导览 前置：L1 core-runtime 导览 配套参考：链路速查表

1. 链路全景：七层架构

一次 thread/start 或 turn/start 请求依次穿过以下七层。记住这个递进顺序是本节的核心目标：

[SDK/富客户端] │ JSON-RPC over stdio / websocket / unix socket ▼ ① transport 层 — transport.rs │ JSONRPCMessage 解码 → ConnectionState ▼ ② message_processor 层 — message_processor.rs │ 反序列化 ClientRequest → initialize 门禁 → 方法路由 ▼ ③ 协议层 — protocol/common.rs, v2/thread.rs, v2/turn.rs, rpc.rs │ 类型定义：ThreadStartParams（v2/thread.rs）、TurnStartParams（v2/turn.rs）、ClientRequest（common.rs）、JSONRPCRequest（rpc.rs） ▼ ④ processor 层 — thread_processor.rs / turn_processor.rs │ thread_start() / turn_start() — 业务逻辑编排 ▼ ⑤ thread_lifecycle 层 — thread_lifecycle.rs │ ensure_conversation_listener() — 事件监听器注册 ▼ ⑥ CodexThread 层 — codex_thread.rs │ submit() / submit_user_input() — 提交 Op 到 core session ▼ ⑦ core 执行引擎 │ 模型调用、工具执行、流式事件产出 ▼ [通知流回 SDK] — thread/started, turn/started, item/*, turn/completed

前五层都在 app-server 内，后两层在 core crate。理解这个分层后，你应该能回答"一个 JSON-RPC 方法在哪个文件被匹配到"。L1 app-server 导览 的第 3 节提供了更简化的四层视角，本节则展开了 processor 和 lifecycle 的细分。

4. 一条异常路径：thread 不存在

当时序图中虚线所示，当客户端用一个不存在的 thread_id 调用 turn/start 时：

  1. turn_start_inner() 调用 self.load_thread(¶ms.thread_id)（turn_processor.rs:450）
  2. load_thread()（turn_processor.rs:300）内部调用 self.thread_manager.get_thread(thread_id)
  3. get_thread() 返回 Err，load_thread() 将其映射为 invalid_request("thread not found: {thread_id}")
  4. 错误一路返回到 handle_initialized_client_request()（message_processor.rs:1517），通过 outgoing.send_error() 发回 JSON-RPC Error

错误码为 -32600（Invalid Request），这是 JSON-RPC 标准错误码。如果你在调试线上问题时看到这个消息，排查入口就是 thread_manager.get_thread() —— thread 可能已被卸载、归档或从未创建。

检索练习

不要回看上文，回答以下三个问题：

  1. thread/start 和 turn/start 在 message_processor.rs 中分别匹配为哪个 ClientRequest 变体？
  2. turn/start 的输入长度校验发生在哪个函数中？
  3. thread/start 的响应是同步返回还是异步通知？

参考答案： ① ClientRequest::ThreadStart（message_processor.rs:1094）和 ClientRequest::TurnStart（message_processor.rs:1324）； ② validate_v2_input_limit()（turn_processor.rs:458），限制为 MAX_USER_INPUT_TEXT_CHARS； ③ thread/start 是异步的——thread_start_inner() 将实际工作 spawn 为 background task（thread_processor.rs:1004），立即返回 Ok(None)（无同步响应体），thread 创建成功后通过 thread/started 通知告知客户端。

如果你对任何阶段不确定，追问 agent：“请带我读 thread_start_task 的完整实现”或“请对比 thread/start 和 turn/start 的同步/异步差异”。
