1. thread/start 的五阶段

thread_start()（thread_processor.rs:422）是入口，但它只是薄层代理，直接调用 thread_start_inner() 并返回 Ok(None)——注意这里的 None 意味着没有同步响应体。真正的 thread 创建是异步任务。

thread_start_inner() 的 5 个阶段： ① 参数解构与校验（:911-942） 解构 ThreadStartParams（model、cwd、sandbox、permissions 等） 校验 sandbox 和 permissions 不能同时指定 ② 配置覆盖构建（:946-960） build_thread_config_overrides() 将请求参数转为 ConfigOverrides 设置 ephemeral 标志 ③ 组装 ListenerTaskContext（:961-971） 聚集 thread_manager、thread_state_manager、outgoing、 pending_thread_unloads、skills_watcher 等共享状态 ④ 异步任务 spawn（:977-1005） 将 thread_start_task 通过 background_tasks.spawn() 提交 thread_start_task 内部：加载配置 → new_thread() → ensure_conversation_listener() 成功后通过 outgoing 发送 thread/started 通知 ⑤ 立即返回（:1006） thread_start_inner 返回 Ok(())，message_processor 收到 Ok(None) 不发送同步响应 —— 客户端必须等待 thread/started 通知

关键设计决策：thread/start 是fire-and-notify模式。方法返回只表示"请求已受理"，真正的 thread 创建和通知是异步的。这与 turn/start 不同——后者同步返回一个 Turn 对象。这种差异的原因是 thread 创建涉及配置加载、thread store 写入和监听器注册，这些 I/O 操作不宜阻塞 JSON-RPC 连接。

2. turn/start 的六阶段

turn_start()（turn_processor.rs:155）先调用 validate_user_input_image_urls() 校验图片 URL，然后进入 turn_start_inner()。

turn_start_inner() 的 6 个阶段： ① 加载 thread（:450-455） load_thread(¶ms.thread_id) → thread_manager.get_thread() 失败 → "thread not found: {thread_id}"（JSON-RPC error -32600） ② 输入校验（:456-465） ensure_direct_input_allowed() — 多 agent v2 子代理不允许直接输入 validate_v2_input_limit() — 输入文本超过 MAX_USER_INPUT_TEXT_CHARS 则拒绝 ③ 设置客户端信息（:466-482） set_app_server_client_info() — 记录 client name/version set_openai_form_elicitation_support() — 标记是否支持 OpenAI 表单诱导 ④ 构建 thread settings 覆盖（:484-519） resolve_turn_environment_selections() — 解析环境选择 build_thread_settings_overrides() — 组装 approval、sandbox、 model、collaboration_mode、personality 等覆盖参数 ⑤ 提交用户输入（:522-540） 构造 Op::UserInput { items, thread_settings, ... } 调用 thread.submit_user_input_with_client_user_message_id() → codex_thread.rs:261 → codex.session.submit_user_input() 返回 turn_id ⑥ 构造同步响应（:557-568） 构造 Turn { id: turn_id, status: InProgress, items: [], ... } 返回 TurnStartResponse { turn } → 通过 JSON-RPC Response 发回客户端

与 thread/start 的 fire-and-notify 不同，turn/start 的⑥返回了同步响应：客户端立即得到 turn_id 和 InProgress 状态，后续的 item 增量事件和 turn/completed 通知通过 notification 通道异步到达。这使得客户端可以立即渲染 "thinking..." 状态，而不需要等待通知。
