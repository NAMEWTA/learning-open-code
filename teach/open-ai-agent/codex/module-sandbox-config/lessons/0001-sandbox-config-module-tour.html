Sandbox 配置到执行的 15 分钟导览

学习目标：看到一段配置时，能说出它如何影响命令执行边界。回到全局架构：L0 项目地图，或查看 L0 参考总览。

1. 先抓住职责边界

Codex 的 sandbox 不是一个单点模块，而是一条管线：配置层读入用户意图，权限层编译成运行态 profile，sandboxing 层决定是否需要平台 sandbox，最后由 macOS、Linux 或 Windows 后端包裹命令。最小骨架如下：

codex-rs/ ├── config/src/ ← TOML schema 与 profile 声明 │ ├── config_toml.rs ← ConfigToml：sandbox/profile 配置入口 │ └── permissions_toml.rs ← [permissions] profile 与继承 ├── core/src/config/ ← 运行态 Config 与权限编译 │ ├── mod.rs ← 合并配置层、requirements、trust │ └── permissions.rs ← 编译 FileSystem/Network policy └── sandboxing/src/ ← 执行前 sandbox 分派 └── manager.rs ← transform 成 SandboxExecRequest

关键源码：codex-rs/config/src/config_toml.rs、codex-rs/core/src/config/mod.rs、codex-rs/sandboxing/src/manager.rs。

2. 最小链路：配置变成 profile

用户可见入口主要有四类：`sandbox_mode` 是旧语法，取值来自 `SandboxMode`：`read-only`、`workspace-write`、`danger-full-access`；`default_permissions` 选择内置或命名 profile；`[permissions]` 定义自定义 profile；`approval_policy` 决定何时请求批准。源码会避免同时使用互斥覆盖项，例如 `sandbox_mode` 不能和 `default_permissions` 覆盖同时出现。

default_permissions = "work"
approval_policy = "on-request"

[permissions.work]
extends = ":workspace"

[permissions.work.network]
enabled = false

这段配置先由 ConfigToml 反序列化，再由 resolve_effective_permission_selection 和 compile_permission_profile_selection 选择并编译。内置 profile 的默认选择还会看项目 trust：可信或不可信项目通常倾向 `:workspace`，否则回到 `:read-only`。

3. profile 变成 sandbox 执行

编译后的核心对象是 `PermissionProfile`，它可以拆成 `FileSystemSandboxPolicy` 与 `NetworkSandboxPolicy`。`Permissions::effective_permission_profile()` 会把运行时 workspace roots 物化进去；`SandboxManager::select_initial()` 决定是否需要平台 sandbox；`SandboxManager::transform()` 再把命令变成可执行请求。

TOML / CLI override → Config::load_config_with_layer_stack → compile_permission_profile_selection → PermissionProfile → SandboxManager::select_initial → SandboxManager::transform → macOS seatbelt / Linux sandbox helper / Windows restricted token

平台差异被压在最后一段：Linux 会构造 `codex-linux-sandbox` 参数并默认走 bubblewrap；macOS 使用 `/usr/bin/sandbox-exec` 生成 Seatbelt policy；Windows 在 restricted token 与 elevated backend 间选择。

4. 暂停回忆

不看上文，回答三个问题：

  1. `default_permissions = "work"` 最终会先进入哪个 TOML 结构？
  2. `[permissions.work.network] enabled = false` 最终影响哪个运行态 policy？
  3. 决定是否包裹平台 sandbox 的统一入口在哪个文件？

参考答案：1. 进入 ConfigToml.default_permissions，并引用 PermissionsToml 中的命名 profile。2. 影响 NetworkSandboxPolicy，`false` 编译为 restricted。3. 入口在 codex-rs/sandboxing/src/manager.rs，重点看 SandboxManager::select_initial 与 SandboxManager::transform。

主要来源：Sandbox 配置参考总览。继续学习时，可以让 agent 带你逐行读 `core/src/config/permissions.rs` 的 profile 编译逻辑。
