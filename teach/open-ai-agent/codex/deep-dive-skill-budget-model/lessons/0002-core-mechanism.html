核心机制：三级降级与描述预算分配

学习目标：能手绘三级降级决策树，写出 2% 预算公式，解释逐字符轮转分配算法的工作原理。

前置：问题背景——为什么需要 Skill 上下文预算模型？

1. 三级降级决策树

render_skill_lines_from_lines（render.rs 第 319 行）是核心决策函数。它按预算压力分三级渲染：

输入：skills 列表 + 预算上限 │ ├─ 计算 full_cost（完整描述 + 路径的总成本） │ ├─ [一级] full_cost ≤ 预算？ │ YES → 全部以【完整描述】渲染 → 返回，无截断 │ NO ↓ ├─ 计算 minimum_cost（仅名称 + 路径，无描述） │ ├─ [二级] minimum_cost ≤ 预算？ │ YES → 所有 skills 保留，但描述被【按预算截断】 │ NO ↓ └─ [三级] minimum_cost > 预算 → 仅保留部分 skills（仅名称 + 路径），其余【完全丢弃】 按 scope 优先级排序，超出预算的 skill 不出现在列表中

对应源码：

// render.rs 第 319-373 行（简化）
fn render_skill_lines_from_lines(skill_lines, total_count, budget) {
    // 一级检查：完整成本是否在预算内？
    let full_cost = skill_lines.iter().fold(0, |used, line| {
        used + line.full_cost(budget)
    });
    if full_cost <= budget.limit() {
        return (全部完整渲染, 零截断);
    }

    // 二级检查：最低成本（无描述）是否在预算内？
    let minimum_cost = skill_lines.iter().fold(0, |used, line| {
        used + line.minimum_cost(budget)
    });
    if minimum_cost <= budget.limit() {
        let desc_budget = budget - minimum_cost;
        return (全部保留 + 描述按预算截断, 截断统计);
    }

    // 三级：连最低成本都超预算，按优先级逐个取舍
    render_minimum_skill_lines_until_budget(budget, skill_lines, total_count)
}

2. 描述预算分配：逐字符轮转算法

当进入二级降级时，Codex 不是简单地把每个 skill 的描述截断到相同长度，而是使用逐字符轮转分配算法——短描述自然退出竞争后，其未使用的预算自动转移给长描述。

算法流程（render_lines_with_description_budget，render.rs 第 599 行）：

1. 为每个 skill 计算 extra_costs[i][k] = "添加第 k 个字符后，总渲染成本增加多少" 2. char_allocations[i] = 0（所有 skill 初始零字符描述） 3. 循环： 对每个 skill i： 如果 char_allocations[i] >= 该 skill 的描述总长度 → 跳过（已完整） 否则： delta = extra_costs[i][next_chars] - extra_costs[i][current_chars] 如果 delta ≤ remaining_budget： char_allocations[i] += 1 remaining_budget -= delta changed = true 直到没有任何 skill 能被分配更多字符（remaining_budget 不够或全部完整）

这个算法保证：

  • 公平性：每个 skill 轮流获得一个字符的预算，不会出现前几个 skill 独占预算的情况
  • 效率：短描述 skill 完整渲染后自动退出循环，其释放的预算归长描述 skill 所用
  • 精确性：以字符为粒度分配，而非固定字节块，最大化预算利用率

测试用例验证了这一点（render.rs 第 1163 行）：

// 短描述 "x"（1 字符）+ 长描述 "abcdefghi"（9 字符）
// 预算 = 最低成本 + 11 字符
// 预期：短描述完整渲染 "x"，长描述得到 8/9 字符 → "abcdefgh"
let short = make_skill_with_description("short-skill", Repo, "x");
let long  = make_skill_with_description("long-skill",  Repo, "abcdefghi");
// ... 结果: long="abcdefgh", short="x"
assert_eq!(rendered.skill_lines, vec![
    expected_skill_line(&long, "abcdefgh"),
    expected_skill_line(&short, "x"),
]);

3. 别名压缩：路径共享优化

当多个 skills 共享同一个插件根目录时，绝对路径中有大量重复前缀。Codex 的别名压缩机制将公共根目录替换为短别名（r0、r1…），显著减少元数据开销。

模式示例
绝对路径/Users/xl/.codex/plugins/cache/openai-curated/github/hash123/skills/gh-fix-ci/SKILL.md
别名路径r0/github/hash123/skills/gh-fix-ci/SKILL.md

别名根表作为额外的元数据注入：

### Skill roots
- `r0` = `/Users/xl/.codex/plugins/cache/openai-curated`

别名压缩不是无条件启用的。Codex 只在别名方案确实更好时才替换绝对路径（aliased_render_is_better，render.rs 第 870 行）：

fn aliased_render_is_better(aliased, absolute, budget) -> bool {
    // 1. 优先比较：包含的 skills 数量（多者胜）
    if aliased.included_count != absolute.included_count {
        return aliased.included_count > absolute.included_count;
    }
    // 2. 次比较：截断字符数（少者胜）
    if aliased.truncated_chars != absolute.truncated_chars {
        return aliased.truncated_chars < absolute.truncated_chars;
    }
    // 3. 最终比较：总成本（低者胜）
    available_skills_cost(budget, aliased) < available_skills_cost(budget, absolute)
}
关键细节：当单个插件版本只有一个 skill 时，别名根会进一步上溯到 marketplace 根（而非停留在 skill root），以最大化共享。当同一插件版本有多个 skills 时，别名根停留在 skill root，避免路径歧义。见 alias_root_for_skill_root（render.rs 第 769 行）。

4. 设计权衡与已知局限

权衡点Codex 的选择代价
预算粒度 Token-based（有窗口信息时）或 Character-based（回退） Token 估算使用 4 bytes/token 近似值，实际 tokenizer 分片可能导致偏差
优先级模型 固定四层：System > Admin > Repo > User 同一 scope 内不做进一步排序（仅按名称字母序），无法表达"GitHub plugin 比 Slack plugin 更重要"
描述截断 逐字符截断，添加 "..." 后缀 截断可能发生在词中间，破坏语义完整性；不尝试做语义感知的句子级截断
别名压缩 基于文件系统路径前缀匹配 不适用于非文件系统 skill（如 orchestrator resource），也不压缩跨 marketplace 的公共前缀
警告阈值 平均截断 > 100 字符或 skill 被丢弃时触发警告 警告是静态文本，不含可操作建议（如"禁用 X plugin 可节省 Y tokens"）

  1. 三级降级分别对应什么渲染输出？给出每级的一个具体示例。
  2. 逐字符轮转分配算法和"均分截断"有什么区别？为什么前者更好？
  3. 什么条件下 Codex 会放弃别名压缩、退回绝对路径模式？

参考答案：

  1. 一级：完整描述 + 路径，如 - github:gh-fix-ci: Automatically fix failing CI checks... (file: /Users/xl/...)；二级：截断描述 + 路径，如 - github:gh-fix-ci: Automatically fix... (file: /Users/xl/...)；三级：仅名称 + 路径，如 - github:gh-fix-ci: (file: /Users/xl/...)，部分 skill 完全不出现在列表中。
  2. 均分截断给每个 skill 固定字符数，短描述 skill 用不满的预算被浪费。逐字符轮转让短描述完整渲染后自动退出竞争，释放的预算转移给长描述。
  3. 当别名根表本身的开销（### Skill roots 段落）超过了路径压缩节省的成本，或者别名方案包含的 skills 数量比绝对路径少时，退回绝对路径。
