问题背景：为什么需要 Skill 上下文预算模型？

学习目标：说出 Skill 元数据注入面临的两个核心矛盾（上下文有限 vs 技能数量、描述完整 vs token 消耗），并理解 Codex 选择预算模型而非硬性数量上限的原因。

1. 上下文窗口是稀缺资源

LLM 的上下文窗口（context window）不是无限的。以 GPT-5 为例，上下文窗口约 200,000 tokens。Codex 的每一次 turn 都需要将大量信息塞进这个窗口：

  • 系统提示词 — Codex 的行为规范、工具使用说明、安全策略（数万 tokens）
  • 会话历史 — 用户与 Codex 的对话记录、工具调用结果
  • 项目上下文 — 当前工作目录、文件列表、git 状态
  • Skills 元数据 — 可用 skills 的名称、描述、路径
  • 用户输入 — 当前 turn 的指令

Skills 元数据只是这五个"租户"之一。如果不对其加以限制，在极端情况下它可能挤占模型理解任务所需的空间。

关键洞察：Skills 元数据的价值边际递减。前 10 个 skills 的描述帮助模型做出正确的 skill 选择；第 50 个 skill 的完整描述可能几乎不被阅读。预算模型的核心目标是在信息完整性和上下文效率之间找到平衡点。

2. Skills 可能很多、很大

Codex 的 skills 来源多样：

来源说明Scope
System skillsCodex 内置的系统级技能，如 skill-creatorSystem
Admin skills管理员在组织层面配置的技能Admin
Plugin skills从插件市场安装的技能包（如 GitHub、Slack 等）Repo/User
Repo skills项目仓库中的 .claude/skills/ 目录Repo
User skills用户个人 ~/.codex/skills/ 目录User

一个活跃用户可能同时拥有 50+ 个 skills。每个 skill 的元数据包含：

// SkillMetadata 结构（model.rs）
pub struct SkillMetadata {
    pub name: String,              // skill 名称，如 "github:gh-fix-ci"
    pub description: String,       // 完整描述，可能数百字
    pub path_to_skills_md: AbsolutePathBuf,  // 文件系统绝对路径
    pub scope: SkillScope,         // System / Admin / Repo / User
    pub plugin_id: Option<String>,
    // ... 还有其他字段
}

如果 50 个 skills 全部以完整描述注入，仅文件绝对路径就可能占去大量 tokens——想象一下 /Users/xl/.codex/plugins/cache/openai-curated/github/hash123/skills/gh-fix-ci/SKILL.md 这样的路径重复 50 次。

3. Codex 的选择：预算模型而非硬性上限

面对"skills 太多"的问题，最直观的方案是设一个硬性数量上限——比如只展示前 20 个 skills。但 Codex 没有走这条路，原因有三：

  1. 优先级不是线性的。 某个 scope 的 skills 天然比另一些更重要——System skills 比 User skills 更应该被模型看到。
  2. 描述长度差异巨大。 一个 skill 的描述可能只有 10 个字，另一个可能有 500 字。按数量截断意味着"浪费"短描述 skills 的预算。
  3. 同源路径重复。 多个 skills 共享同一个插件根目录时，绝对路径中有大量公共前缀。可以压缩。

Codex 的方案是基于 token 预算的动态渲染：

  • 从上下文窗口总容量中划出 2% 给 skills 元数据
  • 按 scope 优先级排序（System > Admin > Repo > User）
  • 在预算内尽可能多地展示信息，超出预算时逐级降级

源码入口：

// render.rs 第 18-23 行
const DEFAULT_SKILL_METADATA_CHAR_BUDGET: usize = 8_000;
const SKILL_METADATA_CONTEXT_WINDOW_PERCENT: usize = 2;
const MAX_DEFAULT_CONTEXT_SKILL_DESCRIPTION_CHARS: usize = 1_024;

// 第 138-153 行：预算计算
pub fn default_skill_metadata_budget(context_window: Option<i64>) -> SkillMetadataBudget {
    context_window
        .and_then(|window| usize::try_from(window).ok())
        .filter(|window| *window > 0)
        .map(|window| {
            SkillMetadataBudget::Tokens(
                window.saturating_mul(2).saturating_div(100).max(1)
            )
        })
        .unwrap_or(SkillMetadataBudget::Characters(8_000))
}

4. 暂停回忆

检索练习：不看上文，回答以下问题：

  1. Codex 为 skills 元数据分配了上下文窗口的百分之几？为什么不是 5% 或 10%？
  2. 如果无法获取模型上下文窗口大小，Codex 回退到什么预算值？
  3. 为什么 Codex 选择预算模型而非简单的数量上限？列举两个原因。

参考答案：

  1. 2%。具体比例的选择没有公开的设计文档说明，但从代码看 2% 是"给技能元数据留一席之地但不过度挤占主任务空间"的经验值。对 200k token 窗口，2% = 4000 tokens，大约能容纳 50-100 个 skills 的基础元数据。
  2. 回退到基于字符的预算：8000 个字符（DEFAULT_SKILL_METADATA_CHAR_BUDGET）。
  3. （1）优先级非线性：不同 scope 重要性不同；（2）描述长度差异大：按数量截断浪费短描述预算。
