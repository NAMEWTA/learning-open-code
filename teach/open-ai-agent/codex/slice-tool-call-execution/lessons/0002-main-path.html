主路径：bash 工具从 handler 到 exec-server 执行详解

学习目标：逐层追踪一次 bash 命令从 ShellCommandHandler 到 exec-server JSON-RPC 调用并拿到 ExecToolCallOutput 的完整过程。

前置：链路地图与时序图

1. 入参：ToolInvocation 携带了什么

当 dispatch_any_with_terminal_outcome 调用 handle_any_tool 时，传入的 ToolInvocation 已经包含了工具执行所需的所有上下文：

ToolInvocation { session: Arc<Session>, ← 会话级服务（extensions、state_db、hooks） turn: Arc<TurnContext>, ← 当前 turn 的配置、权限、cwd、网络状态 step_context: Arc<StepContext>, ← 步骤上下文（模型信息、token 预算） cancellation_token: CancellationToken, ← 取消信号 tracker: SharedTurnDiffTracker, ← 文件变更追踪器 call_id: String, ← 模型分配的调用 ID tool_name: ToolName, ← "shell_command" 或 "bash" source: ToolCallSource, ← Direct 或 CodeMode payload: ToolPayload, ← Function { arguments: "{...}" } }

ToolPayload::Function { arguments } 中的 JSON 字符串会被反序列化为 ShellCommandToolCallParams，其中 command 字段就是模型要执行的 shell 命令文本。

2. ShellCommandHandler：解析参数并委托 run_exec_like

ShellCommandHandler 的 handle() 方法做的事很直接：解析参数、准备执行上下文、调用 run_exec_like。关键代码路径如下：

// codex-rs/core/src/tools/handlers/shell.rs（简化）
async fn run_exec_like(args: RunExecLikeArgs)
    -> Result<FunctionToolOutput, FunctionCallError>
{
    // 1. 获取文件系统和进程执行器
    let fs = turn_environment.environment.get_filesystem();
    let process = turn_environment.environment.get_process();

    // 2. 构造 ExecParams（命令文本、cwd、环境变量、超时等）
    let exec_params = ExecParams { ... };

    // 3. 构造 ShellRuntime 并委托给 ToolOrchestrator
    let mut runtime = ShellRuntime::new(shell_runtime_backend, ...);
    let result = orchestrator.run(&mut runtime, &req, &tool_ctx, ...).await?;

    // 4. 格式化输出为模型可读文本
    let output_text = format_exec_output_for_model(
        &result.output.exec_output, truncation_policy);
    Ok(FunctionToolOutput::from_text(output_text, Some(ok)))
}

3. ToolOrchestrator::run：审批与沙箱选择

这是安全策略的集中决策点，分为两个阶段：

阶段 A — 审批决策：exec_approval_requirement() 根据工具的审批策略、文件系统沙箱策略和用户配置，返回三种结果之一：

结果含义后续行为
Skip无需审批，直接执行跳过用户交互，但 strict_auto_review 下仍走 guardian
NeedsApproval需要用户确认弹出审批对话框或走 guardian 自动审核
Forbidden策略禁止执行立即返回 ToolError::Rejected

阶段 B — 沙箱选择：sandbox_override_for_first_attempt() + select_initial() 决定首次尝试用哪种沙箱（None / LocalSeccomp / BubbleWrap / WorkspaceWrite），然后构造 SandboxAttempt 传给 run_attempt()。

4. run_attempt → exec-server：JSON-RPC 远程执行

run_attempt() 调用 tool.run(req, attempt, tool_ctx)，对于 shell 工具即 ShellRuntime::run()。底层通过 exec-server 的 JSON-RPC 协议与沙箱进程通信：

ShellRuntime::run() → process.exec(ExecOptions { command, cwd, env, timeout }) → ExecServerClient::exec(ExecParams) → JSON-RPC request: { "method": "exec", "params": { "command": "...", ... } } → 沙箱进程 fork + exec → 流式通知: ExecOutputDelta { stdout/stderr chunks } → 终止通知: ExecExited { exit_code } → ExecToolCallOutput { exit_code, aggregated_output, duration }

exec-server 通过 ExecOutputDelta 通知流式传输 stdout/stderr，最终通过 ExecExited 通知返回 exit code。这些事件被聚合到 ExecToolCallOutput 结构体中。

返回后，format_exec_output_for_model() 将原始输出格式化为模型易读的文本：Exit code: 0\nWall time: 1.2 seconds\nOutput:\n...。超过 token 预算的输出会被截断。

主要阅读来源：codex-rs/core/src/tools/handlers/shell.rs、codex-rs/core/src/tools/orchestrator.rs、codex-rs/exec-server/src/client.rs。继续阅读 异常路径 了解权限拒绝、超时和命令失败的处理。
