异常路径：权限拒绝、超时与命令失败

学习目标：能定位三种异常场景的代码位置，说出每种异常如何转化为模型可见的错误消息。

前置：主路径详解

1. 异常类型总览与权限拒绝

工具调用链路上有三类典型的异常，分别发生在不同阶段：

异常类型发生阶段触发条件返回给模型的形式
权限拒绝 Orchestrator 审批 execpolicy 返回 Forbidden 或用户点击拒绝 FunctionCallError::RespondToModel("rejected: ...")
沙箱拒绝 run_attempt 执行 沙箱策略拒绝访问文件/网络 ToolError::Codex(SandboxErr::Denied) → 降级重试或返回错误
命令超时 exec-server 执行 命令执行超过 timeout 限制 输出前缀 "command timed out after Nms"
命令失败 exec-server 执行 命令返回非 0 exit code 正常返回，exit_code != 0 在输出中体现

权限拒绝发生在 Orchestrator 的审批阶段。核心决策由 execpolicy/src/policy.rs 中的 Policy 结构体和 Decision 枚举驱动：

// codex-rs/execpolicy/src/decision.rs
pub enum Decision {
    Allow,     // 无需审批，直接放行
    Prompt,    // 需要用户确认（approval_policy="never" 时直接拒绝）
    Forbidden, // 无条件禁止
}

// codex-rs/execpolicy/src/policy.rs
impl Policy {
    pub fn evaluate(&self, command: &[String], ...) -> Decision {
        // 1. 按可执行程序名查找规则
        // 2. 按网络规则匹配目标 hosts
        // 3. 返回最严格的 Decision
    }
}

在 Orchestrator 中，exec_approval_requirement() 将 Decision 映射为 ExecApprovalRequirement：

// 映射关系
Decision::Allow     → ExecApprovalRequirement::Skip
Decision::Prompt    → ExecApprovalRequirement::NeedsApproval { reason }
Decision::Forbidden → ExecApprovalRequirement::Forbidden { reason }

当返回 Forbidden 时，Orchestrator 立即返回 ToolError::Rejected(reason)，不进入沙箱选择，不启动进程。错误消息直接作为模型可见的响应文本返回。

当返回 NeedsApproval 时，系统弹出审批对话框。如果用户拒绝，同样返回 ToolError::Rejected。guardian 模式下则由 guardian 自动裁决。

2. 沙箱拒绝与降级重试

沙箱拒绝发生在 run_attempt() 执行阶段。例如，当 bwrap 沙箱因为文件系统权限不足而拒绝命令时，exec-server 返回 SandboxErr::Denied。

Orchestrator 的处理策略在 orchestrator.rs 的 run() 方法中：

// codex-rs/core/src/tools/orchestrator.rs（简化逻辑）
Err(ToolError::Codex(CodexErr::Sandbox(SandboxErr::Denied { .. }))) => {
    // 检查是否可以降级重试：
    // 如果用户配置了 allow_unsandboxed_commands，
    // 则尝试以更宽松的沙箱（或 no-sandbox）重新执行
    if unsandboxed_execution_allowed(...) {
        // 构造 escalated sandbox attempt，重新 run_attempt()
        let escalated = SandboxAttempt { sandbox: SandboxType::None, ... };
        Self::run_attempt(tool, req, tool_ctx, &escalated, ...).await
    } else {
        // 无法降级，返回拒绝错误
        Err(ToolError::Rejected(...))
    }
}

这意味着用户如果配置了 allow_unsandboxed_commands，沙箱拒绝后系统会自动降级为无沙箱模式重试——无需重新审批（审批结果会被缓存）。

3. 命令超时与非零退出

超时处理在 format_exec_output_for_model() 的输出格式化阶段体现，而非执行阶段。exec-server 的超时机制由 ExecParams 中的 timeout 字段控制，超时后进程被 kill，ExecToolCallOutput.timed_out 被设为 true：

// codex-rs/core/src/tools/mod.rs
fn build_content_with_timeout(exec_output: &ExecToolCallOutput) -> String {
    if exec_output.timed_out {
        format!(
            "command timed out after {} milliseconds\n{}",
            exec_output.duration.as_millis(),
            exec_output.aggregated_output.text
        )
    } else {
        exec_output.aggregated_output.text.clone()
    }
}

超时命令的部分输出仍然会保留，模型可以看到命令在超时前输出了什么。

命令失败（非零退出）不是错误——它是合法的执行结果。exit code、wall time 和 output 会一起通过 format_exec_output_for_model() 组合成文本返回给模型。模型根据输出内容决定下一步（修正命令、报告错误给用户等）。

主要阅读来源：codex-rs/execpolicy/src/policy.rs、codex-rs/execpolicy/src/decision.rs、codex-rs/core/src/tools/orchestrator.rs、codex-rs/core/src/tools/mod.rs。

4. 检索练习

判断以下场景分别对应哪种异常路径：

  1. 用户执行 rm -rf /，execpolicy 返回 Forbidden
  2. bwrap 沙箱中执行 curl api.github.com，沙箱策略拒绝网络访问
  3. 执行 sleep 100，超过 120 秒 timeout 限制

参考答案：1. 权限拒绝——发生在 Orchestrator 审批阶段，execpolicy 的 Decision::Forbidden 映射为 ExecApprovalRequirement::Forbidden，立即返回 ToolError::Rejected。2. 沙箱拒绝——发生在 run_attempt 执行阶段，SandboxErr::Denied 触发降级重试逻辑；如果 allow_unsandboxed_commands 未配置则返回错误。3. 命令超时——exec-server 在超时后 kill 进程，ExecToolCallOutput.timed_out=true，输出前缀自动加上 "command timed out after Nms"。
