工具调用执行链路地图

学习目标：画出从模型 tool_use 到结果回写的完整 7 步时序图，说出每一层的职责边界。

前置：L1 工具执行模块导览 | L1 Core Runtime 模块导览

1. 链路全景：7 步闭环

一次完整的工具调用，从模型决定使用工具到结果回写供模型继续生成，经过 7 个阶段。每个阶段由不同的模块负责，边界清晰：

模型输出 tool_use content block │ ▼ [1] ToolRouter::build_tool_call() 将 ResponseItem::FunctionCall 解析为 ToolCall { tool_name, call_id, payload } 所属：codex-rs/core/src/tools/router.rs │ ▼ [2] ToolRegistry::dispatch_any_with_terminal_outcome() 查找工具 → PreToolUse hook → 调用 handle_any_tool → PostToolUse hook 所属：codex-rs/core/src/tools/registry.rs │ ▼ [3] ShellCommandHandler::handle() / apply_patch::handle() 解析工具参数，构造 ExecParams 或 ApplyPatchAction 所属：codex-rs/core/src/tools/handlers/shell.rs, apply_patch.rs │ ▼ [4] ToolOrchestrator::run() 审批决策(Allow/Prompt/Forbidden) → 沙箱选择 → 网络审批 所属：codex-rs/core/src/tools/orchestrator.rs │ ▼ [5] Exec-server 沙箱执行 JSON-RPC exec 调用 → 进程启动 → stdout/stderr 流式收集 所属：codex-rs/exec-server/src/client.rs │ ▼ [6] 结果格式化与截断 format_exec_output_for_model() → 加上 exit code、wall time、output 所属：codex-rs/core/src/tools/mod.rs │ ▼ [7] ToolOutput::to_response_item() 序列化为 ResponseInputItem::FunctionCallOutput → 回写 turn context 所属：codex-rs/tools/src/tool_output.rs

2. 三层调度器职责边界

从模型输出到真实执行，经过三层调度。理解它们的分工是读懂这条链路的关键：

调度器文件职责不负责
ToolRouter router.rs 解析模型输出 → 构造 ToolCall；管理 model_visible_specs；判断并行支持 不执行工具，不处理审批
ToolRegistry registry.rs 查找 handler → Pre/Post hook → telemetry → 生命周期通知 → 输出封装 不关心工具内部执行逻辑，不做沙箱决策
ToolOrchestrator orchestrator.rs 审批策略 → 沙箱选择 → 执行尝试 → 拒绝时沙箱降级重试 不直接启动进程，不解析模型输出

4. 暂停回忆

不看上文，回答三个问题：

  1. ToolRouter、ToolRegistry、ToolOrchestrator 三者各自负责什么？
  2. 审批决策发生在时序图的哪个位置？由哪个模块执行？
  3. 工具执行结果最终以什么类型回写到 turn context？

参考答案：1. Router 负责解析模型输出→ToolCall；Registry 负责查找 handler + hook + telemetry + 生命周期；Orchestrator 负责审批 + 沙箱选择 + 执行尝试 + 降级重试。2. 审批在 Orchestrator::run() 的步骤 1，由 ExecApprovalRequirement 驱动，可能弹出用户审批对话框或被 guardian 自动处理。3. 以 ResponseInputItem::FunctionCallOutput 回写，由 ToolOutput::to_response_item() 序列化。
