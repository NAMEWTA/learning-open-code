主路径：从 startThread 到事件消费

学习目标：15 分钟内逐层追踪 TypeScript SDK 的 Codex.startThread() 和 Thread.run() 完整执行路径，理解每层的职责、输入输出和关键决策点。

一句话定位：从 new Codex() 创建 CodexExec，到 thread.run() 消费流式事件，共 4 层。每层只做一件事：构造子进程管理器、组装 CLI 参数、启动 Rust 进程、解析 JSONL 事件流。 前置知识：0001 链路地图。

1. 第 1 层：Codex 构造与 CodexExec 初始化

用户代码 new Codex(options) 时，构造函数只做一件事：创建 CodexExec 实例。CodexExec 负责定位平台原生二进制并管理子进程生命周期。如果用户未通过 codexPathOverride 指定二进制路径，CodexExec 通过 findCodexPath() 自动检测当前平台 target triple，定位 npm 安装的 vendor/ 目录下的原生二进制。

源码：sdk/typescript/src/codex.ts (第 11-18 行)、sdk/typescript/src/exec.ts 第 63-84 行

2. 第 2 层：startThread 与 Thread 延迟绑定

codex.startThread(options) 只是 new Thread(this.exec, this.options, options) 的工厂方法。关键设计：Thread 构造函数不发送任何网络请求，不启动子进程。thread_id 最初为 null，在首次 run() 收到 thread.started 事件后才赋值。这意味着 Thread 对象是轻量的，可以安全地提前创建，真正的通信延迟到 run 时。 

// codex.ts — startThread 和 resumeThread
startThread(options: ThreadOptions = {}): Thread {
  return new Thread(this.exec, this.options, options);
}

resumeThread(id: string, options: ThreadOptions = {}): Thread {
  return new Thread(this.exec, this.options, options, id);
}

// thread.ts — 构造函数，id 默认为 null
constructor(
  exec: CodexExec,
  options: CodexOptions,
  threadOptions: ThreadOptions,
  id: string | null = null,
) {
  this._exec = exec;
  this._options = options;
  this._id = id;       // 新 thread 时为 null
  this._threadOptions = threadOptions;
}

3. 第 3 层：Thread.run() 事件消费循环

thread.run(input) 是同步聚合器，内部调用 runStreamedInternal() 获取异步生成器，然后遍历事件流完成收集。核心循环处理 5 种事件类型： 

// thread.ts — run() 的事件消费循环（精简）
async run(input: Input, turnOptions: TurnOptions = {}): Promise<Turn> {
  const generator = this.runStreamedInternal(input, turnOptions);
  const items: ThreadItem[] = [];
  let finalResponse = "";
  let usage: Usage | null = null;
  let turnFailure: ThreadError | null = null;

  for await (const event of generator) {
    if (event.type === "item.completed") {
      if (event.item.type === "agent_message") {
        finalResponse = event.item.text;   // 捕获最终回复
      }
      items.push(event.item);
    } else if (event.type === "turn.completed") {
      usage = event.usage;                 // 记录 token 用量
    } else if (event.type === "turn.failed") {
      turnFailure = event.error;           // 记录失败原因
      break;
    }
  }
  if (turnFailure) {
    throw new Error(turnFailure.message);
  }
  return { items, finalResponse, usage };
}

注意 run() 和 runStreamed() 的区别：run() 阻塞等待所有事件后返回聚合结果 Turn；runStreamed() 返回 StreamedTurn（包装了异步生成器），用户可自行逐事件消费。两者的底层都是同一个 runStreamedInternal() 生成器。

源码：sdk/typescript/src/thread.ts 第 70-138 行

4. 第 4 层：CodexExec.run() 子进程 JSONL 通信

runStreamedInternal() 将用户参数（model、sandbox、workingDirectory 等）转换为 CodexExecArgs，交给 CodexExec.run()。此方法的核心流程：

  1. 组装 CLI 参数：基础命令为 codex exec --experimental-json，将 config、model、sandbox 等映射为 --config key=value、--model、--sandbox 等 CLI flag
  2. 构造环境变量：注入 CODEX_INTERNAL_ORIGINATOR_OVERRIDE=codex_sdk_ts 标记调用来源，可选设置 CODEX_API_KEY
  3. spawn 子进程：spawn(executablePath, commandArgs, {env, signal})，执行 codex exec --experimental-json ...
  4. 写入输入并关闭 stdin：通过 child.stdin.write(input) 发送用户提示词，随即 .end() 通知 Rust 侧输入完毕
  5. 逐行读取 stdout：用 readline.createInterface 按行解析 JSONL，每行 yield 给上层
  6. 等待退出：子进程退出后检查 exit code，非 0 时合并 stderr 抛出异常
// exec.ts — CodexExec.run() 核心片段
async *run(args: CodexExecArgs): AsyncGenerator<string> {
  const commandArgs = ["exec", "--experimental-json"];
  if (args.model) commandArgs.push("--model", args.model);
  if (args.sandboxMode) commandArgs.push("--sandbox", args.sandboxMode);
  // ... 更多参数映射 ...

  const child = spawn(this.executablePath, commandArgs, {
    env, signal: args.signal,
  });

  child.stdin.write(args.input);
  child.stdin.end();

  const rl = readline.createInterface({ input: child.stdout });
  for await (const line of rl) {
    yield line as string;          // 每行一个 JSONL 事件
  }

  const { code } = await exitPromise;
  if (code !== 0) {
    throw new Error(`Codex Exec exited with code ${code}: ${stderr}`);
  }
}

检索练习：如果 thread.run("hello") 抛出 Codex Exec exited with code 1: error: invalid model 'gpt-99'，请说出这个错误经过了几层传递？每一层做了什么？

参考答案：
1. Rust 层：codex exec 进程检测到无效 model 参数，以 exit code 1 退出，stderr 写入错误信息
2. CodexExec.run()：exitPromise 返回 {code: 1}，合并 stderr 后抛出 Error("Codex Exec exited with code 1: ...")
3. runStreamedInternal()：异步生成器向上传播异常
4. Thread.run()：for await 循环被异常中断，异常透传到用户代码
排查方向：检查 model 参数拼写；运行 codex models 确认可用模型列表。
