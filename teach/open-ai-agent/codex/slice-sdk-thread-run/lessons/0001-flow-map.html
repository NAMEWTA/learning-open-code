链路地图：SDK 调用到事件消费的完整时序

学习目标：15 分钟内画出从 codex.startThread() 到用户消费 turn.completed 事件的完整时序图，理解 TypeScript 和 Python 两条 SDK 路径的共同模式。

一句话定位：本切片追踪 Codex SDK 用户调用 startThread() + run() 到消费 Codex 事件的完整客户端链路。TypeScript SDK 走 spawn + JSONL 子进程路径，Python SDK 走 subprocess + JSON-RPC app-server 路径，两者共享相同的事件语义。 前置知识：L1 SDK 模块导览、L1 app-server 协议概览。

2. 两条 SDK 路径对比

Codex 提供两种 SDK，底层通信方式不同但上层 API 语义一致：

维度TypeScript SDKPython SDK
通信协议子进程 stdin/stdout JSONL子进程 stdio JSON-RPC
命令行codex exec --experimental-jsoncodex app-server --listen stdio://
事件流AsyncGenerator<string> 逐行 yield线程路由队列按 turn_id 分发
thread 创建new Thread(exec, ...) 本地对象JSON-RPC thread/start 请求
thread ID首个 thread.started 事件中获取thread/start 响应中直接返回
子进程管理CodexExec 类，每次 run 创建新进程CodexClient 类，长连接复用

核心差异：TypeScript SDK 每次 run() 启动一个短生命周期的 codex exec 子进程，进程退出即表示 turn 结束。Python SDK 启动一个长生命周期的 codex app-server 进程，通过 JSON-RPC 请求/通知在同一连接上管理多个 thread 和 turn。

3. 关键事件类型与消费时机

无论走哪条路径，SDK 最终暴露给用户的事件语义是一致的。以下 5 种事件覆盖了正常链路的核心节点：

事件含义消费时机
thread.started新 thread 已创建记录 thread_id 用于后续 resume
turn.started一次 turn 开始执行可用于 UI loading 状态
item.completed一个 item（消息/命令/文件变更）完成增量更新对话历史
turn.completedturn 正常结束，含 token 用量获取 usage 统计、结束 loading
turn.failedturn 执行失败错误处理与重试逻辑

4. 异常路径概览

三条关键异常路径，将在 0002 课程中展开分析：

  • 子进程启动失败：spawn() 返回 error 事件（ENOENT / 权限不足）→ CodexExec.run() 中 spawnError 被捕获并抛出
  • 进程异常退出：子进程 exit code 非 0 → CodexExec.run() 合并 stderr 抛出 Codex Exec exited with code N
  • turn 执行失败：收到 turn.failed 事件 → Thread.run() 抛出 Error(event.error.message)

判断题：以下说法正确的是？

  1. TypeScript SDK 的 Thread 类在构造函数中就会向 app-server 发送 thread/start 请求。
  2. Python SDK 的 CodexClient 通过长连接复用同一个子进程处理多个 thread。
  3. thread.started 事件在 TypeScript SDK 中由第一个 JSONL 行携带，在 Python SDK 中由 thread/start 响应携带。
参考答案：
1. 错误 — TypeScript SDK 的 new Thread(exec, ...) 只是创建本地对象，不发送任何请求。thread_id 在首次 run() 收到 thread.started 事件后才填充。
2. 正确 — Python SDK 启动 codex app-server 作为长生命周期子进程，通过 JSON-RPC 在同一连接上管理多个 thread。
3. 正确 — 两种 SDK 获取 thread_id 的时机不同，但语义等价。
