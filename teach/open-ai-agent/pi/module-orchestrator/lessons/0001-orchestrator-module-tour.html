0001 — pi-orchestrator 模块总览

层级：L1 模块总览 预计时长：12 分钟 学习目标：1 个
学习目标

理解 pi-orchestrator 作为多实例编排服务的架构设计：Unix Socket IPC 协议、子进程 JSONL RPC 通信模型、6 种请求类型和 5 种实例状态机。能画出从 CLI 命令到子进程的完整调用链路。

1. 在架构中的位置

pi-orchestrator 包名 @earendil-works/pi-orchestrator，是 Pi monorepo 的最上层包，标注为 "experimental"。它只有一个内部依赖：

pi-orchestrator 通过 子进程 + JSONL RPC 的方式控制多个 pi-coding-agent 实例。每个实例是一个独立的 child_process.spawn，通过 stdin/stdout 的 JSONL 协议与 orchestrator 通信。这种设计让 orchestrator 可以同时管理多个编码 Agent 会话，并通过 Unix Socket IPC 对外暴露管理接口。

2. 内部分层架构

pi-orchestrator 内部按职责分为六层（由上至下）：

packages/orchestrator/src/ ├── cli.ts ← CLI入口 7个子命令 (serve/list/spawn/...) ├── index.ts ← 入口 全部子模块 re-export ├── config.ts ← 配置 版本/路径/Socket 路径 │ ├── ipc/ ← IPC层 Unix Socket 通信 │ ├── protocol.ts ← 6种请求/响应类型 + 编解码 │ ├── server.ts ← Socket 服务器 (支持 rpc_stream 持久连接) │ └── client.ts ← Socket 客户端 (connect → send → read → close) │ ├── handler.ts ← 路由层 请求 → supervisor 方法调度 │ ├── supervisor.ts ← 核心层 实例生命周期管理 ├── rpc-process.ts ← 核心层 子进程 JSONL RPC 通信 ├── storage.ts ← 持久层 实例/Machine 文件 CRUD │ ├── radius.ts ← 远程层 Radius 云服务集成 ├── serve.ts ← 启动 服务启动 + 信号处理 │ └── types.ts ← 数据 核心类型定义

六层职责速查

层级核心文件核心职责
CLI入口 cli.ts 命令行参数解析，分发到 7 个子命令：serve（启动服务）、list/spawn/status/stop（实例管理）、rpc/rpc-stream（远程控制）
IPC层 ipc/protocol.ts / server.ts / client.ts 定义 6 种请求/响应类型，基于 Node.js net 模块实现 Unix Socket 通信。rpc_stream 模式支持持久连接和双向流
路由层 handler.ts 将 IPC 请求按 type 分发给 supervisor 对应方法，组装响应数据。同时提供 openRpcStream() 的桥接
核心层 supervisor.ts / rpc-process.ts OrchestratorSupervisor 管理在线实例集合：spawn → 绑定RPC进程 → 注册Radius → 上线。支持意外退出恢复和优雅关闭
持久层 storage.ts JSON 文件持久化：实例列表（instances.json）和机器记录（machine.json）的增删改查
远程层 radius.ts 向 Radius 云服务注册机器和 Pi 实例，维持心跳，404 后的自动重注册（含指数退避 + 抖动重试）

3. IPC 协议与请求类型

orchestrator 使用换行分隔的 JSON（JSONL）作为 IPC 消息格式，在 Unix Socket 上双向传输。共 6 种请求类型，由 protocol.ts 定义：

请求类型说明CLI 命令
spawn 在新目录下创建一个 pi-coding-agent 子进程实例，自动注册 Radius orchestrator spawn --cwd ./project
list 列出所有已创建的实例（含已停止的）及其状态 orchestrator list
status 查询指定实例的当前状态和元信息 orchestrator status <id>
stop 停止指定实例：取消 Radius 注册、清理子进程、移除记录 orchestrator stop <id>
rpc 向指定实例的 RPC 子进程发送一条命令，等待单次响应后断开 orchestrator rpc <id> '{"type":"prompt",...}'
rpc_stream 建立持久双向流连接：Socket 持续转发客户端命令到子进程，并回传响应和事件 orchestrator rpc-stream <id>

关键差异：rpc 是请求-响应模式（一次调用），rpc_stream 是双向流模式（连接升级后持续通信）。rpc_stream 的连接升级由 server.ts 处理：先完成握手返回 rpc_ready，之后 Socket 进入流模式，所有后续数据被直接路由到 RPC 子进程。

4. 实例生命周期与状态机

每个 managed instance 有 5 种状态（定义于 types.ts）：

stateDiagram-v2
    [*] --> starting: spawn
    starting --> online: RPC进程就绪 + Radius注册成功
    starting --> error: 子进程启动失败
    online --> stopping: stop
    stopping --> stopped: 资源清理完成
    online --> error: 子进程意外退出
    error --> stopped: 资源清理

Spawn 流程（supervisor.spawnInstance()）：

  1. 生成随机 UUID 作为实例 ID，状态设为 starting
  2. 通过 createRpcProcessInstance({cwd}) 启动 pi-coding-agent 子进程
  3. 绑定子进程的 stdout 事件监听器和 exit 处理器
  4. 发送 get_state 命令同步 sessionId/sessionFile
  5. 调用 radiusPresence.registerPi() 向云服务注册
  6. 状态切换为 online

意外退出恢复：子进程的 exit 事件触发 handleUnexpectedRpcExit()，将实例标记为 error 并清理 Radius 注册。

重启恢复：recoverAfterRestart() 方法将所有 online/starting 状态的持久化实例标记为 stopped，并断开所有旧的 Radius 连接。

不看上文，回答以下问题：

  1. pi-orchestrator 内部分为哪六层？每层的核心文件是什么？
  2. IPC 协议支持的 6 种请求类型是什么？rpc 和 rpc_stream 的关键区别是什么？
  3. 一个实例从 spawn 到 online 经历了哪几个步骤？
点击查看参考答案
  1. CLI入口层（cli.ts）→ IPC层（ipc/protocol.ts/server.ts/client.ts）→ 路由层（handler.ts）→ 核心层（supervisor.ts/rpc-process.ts）→ 持久层（storage.ts）→ 远程层（radius.ts）。启动由 serve.ts 编排。
  2. 6 种请求类型：spawn、list、status、stop、rpc、rpc_stream。rpc 是请求-响应模式，一次调用后断开；rpc_stream 建立持久双向流连接，连接升级后持续转发命令和事件。
  3. 生成 UUID → 状态设为 starting → spawn 子进程 → 绑定事件/exit 监听 → get_state 同步元数据 → registerPi 注册 Radius → 状态切换为 online。
