0001 — Pi Agent Harness 项目总览

层级：L0 项目总览 预计时长：12 分钟 学习目标：1 个
学习目标

理解 Pi Agent Harness monorepo 的整体架构，能说出 5 个核心包的名称、职责和依赖关系，并了解项目的技术栈选型理由。

1. 项目定位

Pi Agent Harness（简称 Pi）是一个自扩展的交互式编码 Agent，由 Earendil Works（Mario Zechner）开发并以 MIT 协议开源。 它运行在终端中，通过调用 LLM（大语言模型）来辅助开发者完成代码阅读、编辑、调试等任务。

与 GitHub Copilot 等 IDE 插件不同，Pi 是一个独立的 CLI 工具——你在终端中输入自然语言指令，它在你的工作目录中执行操作。 它不对文件系统、进程、网络施加内置沙箱限制，而是运行在用户自身的权限边界内。如需隔离，可通过 Docker 或 Gondolin 微 VM 进行容器化部署。

Pi 采用 monorepo 架构，将 LLM API 调用、Agent 运行时、终端 UI、编码工具链、多 Agent 编排拆分为 5 个独立 npm 包，每个包可单独发布和使用。

3. 总体架构

Pi 的 5 个核心包按依赖层次自底向上组织。箭头方向表示"依赖于"：

依赖关系解读（从底层到顶层）：

  1. pi-tui 是底层终端渲染库，仅依赖 marked（Markdown 解析）和 get-east-asian-width（东亚字符宽度），不依赖任何 pi 包。
  2. pi-ai 封装了对 OpenAI、Anthropic、Google、Bedrock、Mistral 等多个 LLM Provider 的统一调用接口，也不依赖其他 pi 包。
  3. pi-agent-core 在 pi-ai 之上构建 Agent 运行时，提供传输抽象、状态管理和 attachment 支持。依赖 pi-ai。
  4. pi-coding-agent 是面向用户的 CLI 入口，集成 read/write/bash/edit 工具、会话管理和终端交互。同时依赖 pi-ai、pi-agent-core、pi-tui。
  5. pi-orchestrator 是实验性的多 Agent 编排器，依赖 pi-coding-agent，目前仍在早期阶段。

4. 顶层目录结构

pi/ ├── packages/ ← 5 个核心包 │ ├── tui/ ← 终端 UI 差异渲染库 │ ├── ai/ ← 多 Provider 统一 LLM API │ ├── agent/ ← Agent 运行时 │ ├── coding-agent/ ← 交互式编码 Agent CLI │ └── orchestrator/ ← 实验性编排器 ├── scripts/ ← 构建/检查/发布脚本（20+ 文件） ├── .github/workflows/ ← CI/CD 工作流（9 个） ├── .husky/ ← Git hooks（pre-commit） ├── .pi/ ← Pi 自身配置目录 ├── package.json ← monorepo 根配置 ├── tsconfig.json ← TypeScript 配置（路径别名） ├── tsconfig.base.json ← 公共 TS 编译选项 ├── biome.json ← Lint/Format 配置 ├── AGENTS.md ← AI Agent 协作规范 ├── CONTRIBUTING.md ← 贡献指南 ├── SECURITY.md ← 安全策略 ├── .npmrc ← npm 配置（精确版本 + min-release-age） └── pi-test.sh ← 源码启动脚本

关键文件说明：

  • package.json — 定义 workspaces（packages/*）、统一脚本（build/check/test）和 devDependencies。
  • tsconfig.json — 通过 paths 将 @earendil-works/pi-* 映射到对应包的 src/index.ts，开发时无需先构建依赖包。
  • biome.json — 仅检查 packages/*/src/** 和 packages/*/test/**，排除 node_modules、models.generated.ts 和测试固件。
  • AGENTS.md — AI 编码 Agent（包括 Pi 自身）的行为规范，含代码质量、Git 操作、依赖安全等规则。

检索练习

不看上文，回答以下问题：

  1. Pi 的 5 个核心包分别叫什么？它们的依赖关系是怎样的（谁依赖谁）？
  2. pi-tui 和 pi-ai 为什么可以独立使用（不依赖其他 pi 包）？
  3. Pi 的依赖安全策略中，save-exact=true 和 min-release-age=2 分别起什么作用？
点击查看参考答案
  1. pi-tui → pi-ai → pi-agent-core → pi-coding-agent → pi-orchestrator。pi-tui 和 pi-ai 是基础层，不依赖其他 pi 包；pi-agent-core 依赖 pi-ai；pi-coding-agent 依赖 pi-agent-core、pi-ai、pi-tui；pi-orchestrator 依赖 pi-coding-agent。
  2. pi-tui 只处理终端渲染（Markdown 解析 + 东亚字符宽度），pi-ai 只封装 LLM API 调用，两者都是独立的功能层。但通常用户通过 pi-coding-agent 间接使用它们。
  3. save-exact=true 锁定直接外部依赖为精确版本，防止 semver 范围导致非预期升级。min-release-age=2 禁止 npm 安装发布不足 2 天的包，防止供应链攻击中同日发布的恶意版本被解析。
