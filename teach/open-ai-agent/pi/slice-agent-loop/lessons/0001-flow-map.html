0001 — Agent 对话循环全链路：从 prompt() 到工具调用的完整闭环

层级：L2 垂直切片 预计时长：15 分钟 学习目标：1 个
学习目标

追踪从 agent.prompt("帮我读文件") 到 LLM 响应、工具调用执行、结果回传 LLM 的完整 7 步数据流。能画出事件序列图，并定位工具调用失败时异常如何沿事件管道回传。

1. 入口：两条路径汇入同一循环

对话循环有两个入口，都在 Agent 类中：

prompt(text) — 新对话轮次。规范化输入 → 检查 activeRun → 调用 runAgentLoop()。
continue() — 恢复已有对话。检查末条消息角色（不能是 assistant） → 消费队列 → 调用 runAgentLoopContinue()。

两条路径最终汇聚到 agent-loop.ts 中的相同函数族。核心入口代码：

// agent.ts — prompt() 的输入规范化和循环启动
async prompt(input, images) {
  if (this.activeRun) {
    throw new Error("Agent is already processing...");
  }
  const messages = this.normalizePromptInput(input, images);
  await this.runPromptMessages(messages);
}

private async runPromptMessages(messages, options = {}) {
  await this.runWithLifecycle(async (signal) => {
    await runAgentLoop(messages,
      this.createContextSnapshot(),  // 冻结 systemPrompt + messages + tools
      this.createLoopConfig(options), // 组装 model + 钩子 + 队列函数
      (event) => this.processEvents(event), // 状态归约 + 监听器
      signal,
      this.streamFn); // 默认 streamSimple
  });
}

2. 双层循环的分工与轮询时机

runLoop() 是控制流的核心。外层循环处理 follow-up 消息——Agent 即将停止时才注入；内层循环处理 tool calls 和 steering 消息——在当前 assistant 回复完成后立即注入。

队列轮询时机消费模式注入 API
steering每个 turn 完成后（turn_end 之后）one-at-a-time / allagent.steer()
followUp内层循环退出后（无 tool call 且无 steering）one-at-a-time / allagent.followUp()

结构骨架（关键分支已标注）：

// agent-loop.ts — runLoop 双层 while 结构精简版
async function runLoop(ctx, newMsgs, config, signal, emit, streamFn) {
  let firstTurn = true;
  let pending = (await config.getSteeringMessages?.()) || [];

  while (true) {                       // 外层：followUp 循环
    let hasMoreToolCalls = true;
    while (hasMoreToolCalls || pending.length > 0) {
      if (!firstTurn) emit({ type: "turn_start" });
      else firstTurn = false;

      if (pending.length > 0) { /* emit + push 每条 */ }

      const msg = await streamAssistantResponse(ctx, config, signal, emit, streamFn);
      if (msg.stopReason === "error" || msg.stopReason === "aborted") return;

      const toolCalls = msg.content.filter(c => c.type === "toolCall");
      if (toolCalls.length > 0) {
        const batch = await executeToolCalls(ctx, msg, config, signal, emit);
        hasMoreToolCalls = !batch.terminate;
      } else hasMoreToolCalls = false;

      emit({ type: "turn_end", message: msg, toolResults: batch?.messages ?? [] });
      pending = (await config.getSteeringMessages?.()) || [];
    }
    const followUps = (await config.getFollowUpMessages?.()) || [];
    if (followUps.length > 0) { pending = followUps; continue; }
    break;
  }
  emit({ type: "agent_end", messages: newMsgs });
}

3. LLM 调用层：消息转换与流事件分发

streamAssistantResponse() 是 Agent 消息空间与 LLM 消息空间的唯一桥接点。三个关键步骤：

  1. transformContext（可选）— 在 AgentMessage[] 层面做上下文裁剪或注入
  2. convertToLlm（必需）— 将 AgentMessage[] 转为 Message[]，过滤掉 bashExecution/custom/compactionSummary 等自定义消息类型
  3. streamFn — 调用 LLM API，遍历返回的事件流，将每个 text_delta/toolcall_start 等事件转为 message_update 发出

默认的 convertToLlm 只保留 role 为 user/assistant/toolResult 的消息：

// agent.ts — 默认的 LLM 消息转换器
function defaultConvertToLlm(messages: AgentMessage[]): Message[] {
  return messages.filter(
    (message) => message.role === "user"
             || message.role === "assistant"
             || message.role === "toolResult",
  );
}

转换完成后，streamFn（默认 streamSimple）向 LLM 发送请求。返回的 AssistantMessageEventStream 被逐事件消费，每个 start/text_delta/toolcall_delta/done 事件触发对应的 emit() 调用，驱动 UI 层更新。

4. 工具调用 Pipeline 与异常路径

当 LLM 返回的 AssistantMessage 包含 toolCall 内容块时，进入工具执行 pipeline。每个工具调用经过四阶段：

prepare → 参数校验（TypeBox schema）+ beforeToolCall 钩子（可 block）
execute → 调用 tool.execute()，支持 partial update 回调
finalize → afterToolCall 钩子（可覆盖 content/details/isError/terminate）
emit → 发放 tool_execution_end + toolResult message 事件

异常路径——工具调用可能在三处失败，每一处都产生 isError=true 的 ToolResultMessage 回传给 LLM，由 LLM 决定如何修正：

// agent-loop.ts — prepareToolCall 中的三条异常路径
async function prepareToolCall(ctx, msg, toolCall, config, signal) {
  const tool = ctx.tools?.find(t => t.name === toolCall.name);
  if (!tool) {
    return { kind: "immediate",      // 异常 1: 工具不存在
      result: createErrorToolResult(`Tool ${toolCall.name} not found`),
      isError: true };
  }
  try {
    const validatedArgs = validateToolArguments(tool, preparedToolCall);
    if (config.beforeToolCall) {
      const beforeResult = await config.beforeToolCall(/*...*/, signal);
      if (beforeResult?.block) {
        return { kind: "immediate",   // 异常 2: beforeToolCall 拦截
          result: createErrorToolResult(beforeResult.reason
                    || "Tool execution was blocked"),
          isError: true };
      }
    }
    return { kind: "prepared", toolCall, tool, args: validatedArgs };
  } catch (error) {
    return { kind: "immediate",      // 异常 3: 参数校验失败
      result: createErrorToolResult(
        error instanceof Error ? error.message : String(error)),
      isError: true };
  }
}

所有 kind === "immediate" 的异常结果不执行 tool.execute()，直接产生错误 ToolResultMessage。第四种异常是 tool.execute() 本身 throw——在 executePreparedToolCall 的 catch 分支中捕获，同样产生 isError=true 的结果。这意味着工具执行永远不会让 Agent 崩溃——错误总是被转为结构化的 ToolResultMessage，继续参与对话循环。

  1. prompt() 和 continue() 最终分别调用 agent-loop.ts 中的哪个函数？
  2. 外层循环和内层循环各由什么条件触发继续执行？
  3. LLM 调用前，AgentMessage[] 如何变成 Message[]？哪些消息类型会被过滤掉？
  4. 工具调用的四种异常路径分别发生在哪个阶段？它们有一个共同的安全保证是什么？
点击查看参考答案
  1. prompt() 调用 runAgentLoop()，continue() 调用 runAgentLoopContinue()。两者最终都进入同一个 runLoop() 函数。
  2. 外层循环由 followUp 队列非空 触发继续；内层循环由 hasMoreToolCalls 为 true 或 pendingMessages（steering）非空 触发继续。
  3. 通过 convertToLlm() 函数转换。默认实现过滤掉非 user/assistant/toolResult 角色的消息（bashExecution、custom、branchSummary、compactionSummary 等被排除）。
  4. 四种异常：(1) 工具未找到（prepare 阶段）；(2) beforeToolCall 拦截（prepare 阶段）；(3) 参数校验失败（prepare 阶段）；(4) execute() 抛出异常（execute 阶段）。共同保证：异常永远不会让 Agent 崩溃——总是转为 isError=true 的 ToolResultMessage 回传 LLM。
