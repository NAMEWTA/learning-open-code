0001 — 会话持久化全链路地图

层级：L2 垂直切片 预计时长：14 分钟 学习目标：1 个
学习目标

理解一条 Agent 消息从 agent.prompt() 到 JSONL 磁盘文件的完整数据流——穿越 Session 树、存储引擎、仓库索引、以及上下文压缩这四层，并能说出每层的职责和关键数据结构。

1. 四层架构与入口点

会话持久化链路从上到下分为四层。入口是 Agent.prompt() 触发消息生成后，由 harness 调用 session.appendMessage()：

1
Session 树（session.ts）
将 AgentMessage 包装为 SessionTreeEntry，挂到 parentId 链表上。提供 appendMessage / appendCompaction / moveTo 等操作。
2
存储引擎（jsonl-storage.ts / memory-storage.ts）
实现 SessionStorage 接口。JSONL 引擎逐行追加到文件，内存引擎存入数组。LeafEntry 追踪当前位置。
3
仓库层（jsonl-repo.ts / memory-repo.ts）
管理会话生命周期：create/open/list/delete/fork。cwd 编码为目录名，时间戳+sessionId 组成文件名。
4
上下文压缩（compaction.ts）
当 token 超阈值时，摘要旧消息并注入 compaction entry，保持上下文窗口可控。

入口点：Agent 主循环在每轮对话完成后调用 session.appendMessage()（第 1 层），将产生的 AgentMessage 写入会话树。随后在下一轮开始前检查是否需要压缩（第 4 层）。

2. JSONL 存储引擎

JSONL 文件的第一行永远是 SessionHeader（type 为 "session"，version 固定为 3），后续每行是一个 SessionTreeEntry。LeafEntry 是特殊的"指针"条目——它不包含业务数据，只记录当前会话树走到了哪个节点。

JsonlSessionStorage 核心方法

方法触发时机磁盘操作
static create()新建会话writeFile：写入 SessionHeader 行
static open()打开已有会话readTextFile：解析全部行到内存
appendEntry()每次消息/事件写入appendFile：追加一行 JSON
setLeafId()分支切换 (moveTo)appendFile：追加 LeafEntry 行
getPathToRoot()构建上下文 / Fork纯内存：沿 parentId 回溯

关键设计：内存缓存。open() 时将整个 JSONL 文件解析到内存的 byId Map 和 entries 数组中。后续所有读操作（getEntry、getPathToRoot、findEntries）都是 O(1) 内存查询，不读磁盘。写操作（appendEntry）同步更新内存缓存并追加磁盘。

异常路径：JSONL 解析错误。如果 JSONL 文件损坏（非 JSON 行、缺少 type 字段、version 不匹配），parseHeaderLine() 和 parseEntryLine() 会抛出 SessionError("invalid_session") 或 SessionError("invalid_entry")。仓库层的 list() 方法在枚举会话目录时会捕获 invalid_session 错误并跳过损坏文件，保证其他会话不受影响。

3. 仓库层：会话生命周期管理

JsonlSessionRepo 管理磁盘上的会话目录结构：

~/.pi/sessions/ └── --Users-wta-projects-myapp--/ ← cwd 编码为目录名 ├── 2026-07-09T10-30-00-000Z_abc123.jsonl ├── 2026-07-09T11-15-00-000Z_def456.jsonl └── 2026-07-09T14-00-00-000Z_ghi789.jsonl

cwd 编码规则：encodeCwd() 将路径中的 /、\、: 替换为 -，前后加 --。例如 /Users/wta/myapp 变为 --Users-wta-myapp--。这保证了不同项目目录的会话隔离。

Fork 操作是仓库层最复杂的操作。它不复制 JSONL 文件，而是创建新文件并写入从根到 fork 点的所有条目。核心逻辑在 repo-utils.ts 的 getEntriesToFork()：

// 简化逻辑：getEntriesToFork 的三种路径
// 1. 不指定 entryId → 返回全部 entries（完整拷贝）
// 2. position="before" → 找到 entryId 对应的 user message，取其 parentId
//    然后沿 parentId 回溯到根
// 3. position="at" → 以 entryId 自身为终点回溯到根

双后端设计：InMemorySessionRepo 与 JsonlSessionRepo 实现相同的 SessionRepo 接口，差异仅在于存储介质（Map vs 文件系统）。两者共享 repo-utils.ts 中的 createSessionId、toSession、getEntriesToFork 等工具函数，确保行为一致性。

4. 上下文压缩链路

当上下文 token 数超过 contextWindow - reserveTokens（默认 reserveTokens=16384）时触发压缩。压缩流程分三步：

4.1 判断是否需要压缩

// compaction.ts — shouldCompact
export function shouldCompact(
  contextTokens: number,
  contextWindow: number,
  settings: CompactionSettings,
): boolean {
  if (!settings.enabled) return false;
  return contextTokens > contextWindow - settings.reserveTokens;
}

Token 估算优先使用 provider 返回的 usage 字段（精确值），兜底使用字符数 / 4 的启发式估算（estimateTokens()）。

4.2 选择切点

findCutPoint() 从消息列表尾部向前累积 token，当累积量达到 keepRecentTokens（默认 20000）时，向前搜索最近的合法切点。合法切点包括 user message、assistant message、branch_summary 等，但排除 toolResult（不能在工具结果中间切断）。

如果切点落在一个轮次中间（非 user message 开头），isSplitTurn=true，此时轮次前半部分需要单独的 turn prefix 摘要。

4.3 生成摘要并写入

generateSummary() 将待摘要消息序列化为文本（通过 serializeConversation()），发送给 LLM 生成结构化摘要。摘要格式包含 Goal、Progress、Key Decisions、Next Steps 等字段。如果有前次压缩，使用 UPDATE_SUMMARIZATION_PROMPT 增量更新而非重新生成。

最终 compact() 将摘要文本、文件操作列表（read/modified files）格式化为 CompactionResult，由 harness 调用 session.appendCompaction() 写入会话树。
