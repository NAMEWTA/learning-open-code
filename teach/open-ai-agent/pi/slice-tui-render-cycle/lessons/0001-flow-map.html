图中关键决策点：整个 doRender() 方法实际上是一个多路选择器——首帧、尺寸变化、无变化、纯删除、溢出视口、正常差异渲染。理解每个分支的触发条件，就是理解差异渲染的核心。

2. 入口：渲染请求的节流调度

外部调用 requestRender() 时，并不会立即执行渲染。实际链路经过两层异步调度：

requestRender(force = false): void {
  if (force) {
    // 强制模式：重置所有状态，确保触发全量重绘
    this.previousLines = [];
    this.previousWidth = -1;  // -1 确保 widthChanged 为 true
    this.previousHeight = -1;
    this.renderRequested = true;
    process.nextTick(() => {
      if (this.stopped || !this.renderRequested) return;
      this.renderRequested = false;
      this.doRender();
    });
    return;
  }
  // 普通模式：标记请求，交给 scheduleRender 统一调度
  if (this.renderRequested) return;  // 已有请求排队，跳过
  this.renderRequested = true;
  process.nextTick(() => this.scheduleRender());
}

scheduleRender 实现了 16ms 的节流窗口（MIN_RENDER_INTERVAL_MS），计算距上次渲染的时间差并延迟补足：

private scheduleRender(): void {
  const elapsed = performance.now() - this.lastRenderAt;
  const delay = Math.max(0, TUI.MIN_RENDER_INTERVAL_MS - elapsed);
  this.renderTimer = setTimeout(() => {
    this.renderRequested = false;
    this.lastRenderAt = performance.now();
    this.doRender();
    if (this.renderRequested) this.scheduleRender();
  }, delay);
}

为什么两次异步？ process.nextTick 将渲染推迟到微任务末尾，合并同一宏任务内的多次调用；setTimeout 16ms 进一步合并短时间内的多次请求（如流式输出的高频更新），防止 CPU 过载。

3. 渲染管线：组件树到归一化行数组

doRender() 被调用后的第一件事是执行渲染管线，将组件树转换为可对比的行数组。分三个阶段：

阶段 A：组件树渲染

this.render(width) 调用的是 Container.render(width)（TUI 继承自 Container），递归遍历所有子组件：

// Container.render(width)
render(width: number): string[] {
  const lines: string[] = [];
  for (const child of this.children) {
    const childLines = child.render(width);
    for (const line of childLines) lines.push(line);
  }
  return lines;
}

每个叶子组件（Text、Editor、Markdown 等）返回自己的行数组，Container 将其简单拼接。

阶段 B：Overlay 合成

compositeOverlays() 将弹窗、下拉菜单等 Overlay 的内容叠加到基础内容上。按 focusOrder 排序（后创建的在上层），通过 compositeLineAt() 将 overlay 的每一行精确拼接到目标列的指定位置。

阶段 C：光标提取 + 行重置

聚焦的 Editor 在光标位置插入 \x1b_pi:c\x07（CURSOR_MARKER——零宽度 APC 序列）。extractCursorPosition() 扫描可见视口找到这个标记，记录坐标后从行内容中移除。随后 applyLineResets() 在每行末尾追加 \x1b[0m\x1b]8;;\x07（SEGMENT_RESET），防止行间样式污染。

4. 差异计算与 ANSI 序列生成

归一化后的 newLines 与 previousLines 进行逐行比对：

let firstChanged = -1, lastChanged = -1;
const maxLines = Math.max(newLines.length, this.previousLines.length);
for (let i = 0; i < maxLines; i++) {
  const oldLine = i < this.previousLines.length ? this.previousLines[i] : "";
  const newLine = i < newLines.length ? newLines[i] : "";
  if (oldLine !== newLine) {
    if (firstChanged === -1) firstChanged = i;
    lastChanged = i;
  }
}

基于 firstChanged 和 lastChanged，doRender() 进入多路决策：

条件渲染模式原理
previousLines.length === 0首帧不清屏假设终端已干净，直接输出避免闪烁
widthChanged全量重绘+清屏宽度变化→所有行换行位置改变，比对无意义
heightChanged（非 Termux）全量重绘+清屏高度变化导致视口偏移
firstChanged === -1跳过渲染无变化，仅更新光标
firstChanged >= newLines.length只清除多余行所有变化在已删除的行
firstChanged < prevViewportTop降级全量重绘变化区域在视口上方不可见
其它（正常情况）差异渲染光标移动→逐行\x1b[2K→输出新行→清除多余

差异渲染的 ANSI 缓冲区构造步骤：

  1. 包裹 \x1b[?2026h ... \x1b[?2026l 同步输出协议
  2. 通过 \x1b[nA / \x1b[nB 移动光标到 firstChanged
  3. 每行：先 \x1b[2K 清行，再输出新内容
  4. 若旧内容行更多，向后 \x1b[2K 清除多余行
  5. 若 visibleWidth(line) > width → 写入崩溃日志并抛异常（安全阀）

行宽溢出保护：如果某行的 visibleWidth(line) > width，doRender() 立即停止 TUI、写入崩溃日志 ~/.pi/agent/pi-crash.log、抛出 Error。前面所有处理（组件 truncateToWidth、compositeLineAt sliceByColumn）都是为了不走到这一步。

点击查看参考答案
  1. process.nextTick 合并同一宏任务内的多次调用；setTimeout 16ms 合并短时间内的多次请求（如流式输出高频更新），防止 CPU 过载。
