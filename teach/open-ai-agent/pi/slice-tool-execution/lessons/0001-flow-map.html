0001 — 工具执行全链路

层级：L2 垂直切片 预计时长：15 分钟 学习目标：1 个
学习目标

追踪从 _buildRuntime() 创建工具定义，到 LLM 返回 tool_use，再到具体工具 execute() 的 完整 6 阶段数据流。能画出 mermaid 时序图，定位四类异常路径的捕获点，并理解每个工具的 Operations 可插拔接口如何支持远程代理。

1. 阶段零：工具注册与激活

工具调用发生之前，必须先完成注册。入口在 AgentSession._buildRuntime()：

Step 1 — 调用 createAllToolDefinitions(cwd, options) 生成 7 个内置工具的 ToolDefinition 对象，存入 _baseToolDefinitions Map。
Step 2 — 创建 ExtensionRunner，加载扩展中的自定义工具。
Step 3 — 调用 _refreshToolRegistry() 合并三层工具（base + custom + extension），通过 allowlist/denylist 过滤，用 wrapRegisteredTools 统一包装为 AgentTool[]，最终写入 agent.state.tools。

关于 ToolName 类型：

// tools/index.ts — 7 个标准工具名
export type ToolName = "read" | "bash" | "edit" | "write" | "grep" | "find" | "ls";

// 三个分组策略
createCodingToolDefinitions(cwd)  → [read, bash, edit, write]  // 默认激活
createReadOnlyToolDefinitions(cwd) → [read, grep, find, ls]    // 只读模式
createAllToolDefinitions(cwd)     → Record<ToolName, ToolDef>  // 全部可用

3. 沿途每一层：ToolDefinition → AgentTool → execute()

3.1 包装层：ToolDefinition → AgentTool

每个工具先定义为 ToolDefinition（面向扩展系统的富接口），再通过 wrapToolDefinition() 转为 AgentTool（面向 Agent 运行时的精简接口）：

// tool-definition-wrapper.ts — 适配层
export function wrapToolDefinition(
  definition: ToolDefinition,
  ctxFactory?: () => ExtensionContext,
): AgentTool {
  return {
    name: definition.name,
    description: definition.description,
    parameters: definition.parameters,
    executionMode: definition.executionMode,
    execute: (id, params, signal, onUpdate) =>
      definition.execute(id, params, signal, onUpdate, ctxFactory?.()),
  };
}

关键设计：ctxFactory 延迟创建 ExtensionContext，只在工具真正执行时才注入，避免无谓的上下文创建。

3.2 钩子层：beforeToolCall / afterToolCall

在 AgentSession._installAgentToolHooks() 中安装，位于 prepareToolCall 和 executePreparedToolCall 之间：

// agent-session.ts — 工具钩子安装
this.agent.beforeToolCall = async ({ toolCall, args }) => {
  const runner = this._extensionRunner;
  if (!runner.hasHandlers("tool_call")) return undefined;
  return await runner.emitToolCall({
    type: "tool_call",
    toolName: toolCall.name,
    toolCallId: toolCall.id,
    input: args,
  });
  // 返回值中 {block: true, reason: "..."} 可拦截执行
};

this.agent.afterToolCall = async ({ toolCall, args, result, isError }) => {
  // 可覆盖 result.content / result.details / isError
  const hookResult = await runner.emitToolResult({...});
  return hookResult ? {
    content: hookResult.content,
    details: hookResult.details,
    isError: hookResult.isError ?? isError,
  } : undefined;
};

3.3 工具层：四种核心工具的 execute() 模式

每个工具共享相同的结构骨架——Operations 可插拔接口 + execute() 实现：

工具核心操作Operations 接口关键安全机制
read 文件读取（文本/图片） readFile + access + detectImageMimeType offset/limit 分页、truncateHead 字节+行数双重截断、文件权限检查
bash 命令执行 exec(command, cwd, {onData, signal, timeout, env}) spawn detached 进程树、timeout 超时自动 kill、AbortSignal 优雅中断
write 文件创建/覆盖 writeFile + mkdir file-mutation-queue 串行化、自动创建父目录、abort 后不释放队列锁
edit 精确字符串替换 readFile + writeFile + access(R_OK|W_OK) 每个 edit.oldText 必须唯一且不重叠、BOM 剥离与行尾规范化、diff/patch 生成用于展示

Operations 接口是 Pi 工具系统的核心扩展点。默认使用本地文件系统和本地 shell，但通过传入自定义 Operations，你可以让同一套工具代码操作远程文件系统或 SSH 会话，而无需修改工具的核心逻辑。

4. 异常路径全景

工具执行有四个异常捕获点，每个都产生 isError=true 的 ToolResultMessage 回传 LLM：

异常类型发生阶段捕获位置LLM 看到的内容
工具未找到 prepareToolCall ctx.tools.find() 返回 undefined "Tool xxx not found"
参数校验失败 prepareToolCall validateToolArguments() throw TypeBox 校验错误信息
beforeToolCall 拦截 prepareToolCall beforeResult?.block === true beforeResult.reason \|\| "Tool execution was blocked"
execute() 自身异常 execution executePreparedToolCall 的 catch 工具内部 error.message（如 "Permission denied"、"File not found"）
安全保证：四种异常路径共享同一个设计原则——异常永远不让 Agent 崩溃。所有错误都被转为结构化的 ToolResultMessage {isError: true, content: [...]}，插入消息历史，下一轮 LLM 调用时可以看到错误内容并尝试修正。

以 bash 工具的超时和 abort 为例，看 error.message 约定：

// bash.ts — 结构化错误消息约定
if (err.message === "aborted") {
  throw new Error(appendStatus(text, "Command aborted"));
}
if (err.message.startsWith("timeout:")) {
  const timeoutSecs = err.message.split(":")[1];
  throw new Error(appendStatus(text,
    `Command timed out after ${timeoutSecs} seconds`));
}
// 普通错误原样抛出
throw err;

注意：bash 工具先完成输出收集（finishOutput()）再抛出错误。这样即使命令超时，LLM 仍能看到命令已经产生的部分输出。

检索练习

闭上眼回想，不上翻，回答以下问题：

  1. 从 _buildRuntime() 到 Agent 拿到可用工具列表，经历了哪三个步骤？
  2. wrapToolDefinition() 的 ctxFactory 参数为什么设计为延迟调用（函数而非直接传值）？
  3. Operations 接口的意义是什么？举一个 read 工具中 Operations 的使用场景。
  4. bash 工具的超时和 abort 处理为什么先收集完输出再 throw？这给 LLM 带来了什么好处？
点击查看参考答案
  1. 三步：(1) createAllToolDefinitions(cwd) 生成 7 个内置 ToolDefinition；(2) 创建 ExtensionRunner 加载扩展工具；(3) _refreshToolRegistry() 合并三层工具 → 过滤 → 包装为 AgentTool → 写入 agent.state.tools。
  2. 延迟调用避免了在工具注册阶段（而非执行阶段）创建 ExtensionContext。扩展可能未完全初始化，或 Context 依赖运行时状态（如当前工作目录），延迟到 execute 时调用能拿到最新状态。
  3. Operations 是可插拔操作接口，允许替换底层实现而不改工具核心逻辑。例如 read 工具的默认 defaultReadOperations.readFile 使用本地 fs.readFile，但传入自定义 operations.readFile 可以实现从远程服务器读取文件。
  4. 先收集完输出再 throw，确保即使命令失败（超时/abort），部分输出 仍能写入 ToolResultMessage 的 text content。LLM 看到错误的同时也能看到命令的部分输出，有助于 LLM 做出更准确的修正决策。
