1. ProviderStreams 契约模式

pi-ai 的 API 集成层遵循统一的 ProviderStreams 契约模式 -- 每个 API 适配器模块和每个供应商实现都遵循相同的接口约定。这一模式是整个 packages/ai/src/ 的基石，理解它即可理解剩余 90+ 个文件的结构。

1.1 核心契约：StreamFunction

ProviderStreams 接口定义

export interface ProviderStreams {
  stream(model: Model<Api>, context: Context, options?: StreamOptions): AssistantMessageEventStream;
  streamSimple(model: Model<Api>, context: Context, options?: SimpleStreamOptions): AssistantMessageEventStream;
}

三个核心约束

约束说明
1. 立即返回流 必须同步返回 AssistantMessageEventStream 对象，不得阻塞等待网络连接。实际的 API 调用在工作流的 async IIFE 中执行。
2. 不抛出异常 请求、模型或运行时故障必须编码在返回的流中（通过 stopReason: "error" 事件），而不是抛出到调用方。
3. 双函数设计 stream 暴露供应商完整选项（如 Google 的 thinking 配置）。streamSimple 接收统一选项（仅 reasoning 级别），内部转换为供应商特定选项后委托给 stream。

1.2 数据结构契约

每个 stream 函数必须创建并维护一个 AssistantMessage 对象，其结构如下：

{
  role: "assistant",
  content: Array<TextContent | ThinkingContent | ToolCall>,
  api: Api,           // 例如 "google-generative-ai"
  provider: string,     // 例如 "google"
  model: string,        // 例如 "gemini-2.5-flash"
  usage: Usage,          // { input, output, cacheRead, cacheWrite, totalTokens, cost }
  stopReason: StopReason, // "stop" | "length" | "toolUse" | "error" | "aborted"
  timestamp: number,
  errorMessage?: string,
  responseId?: string,
}

1.3 流事件协议

AssistantMessageEventStream 按以下生命周期推送事件：

事件类型阶段payload 字段
start开始partial -- 初始 AssistantMessage
text_start文本块开始contentIndex, partial
text_delta文本增量contentIndex, delta, partial
text_end文本块结束contentIndex, content, partial
thinking_start思考块开始contentIndex, partial
thinking_delta思考增量contentIndex, delta, partial
thinking_end思考块结束contentIndex, content, partial
toolcall_start工具调用开始contentIndex, partial
toolcall_delta工具调用增量contentIndex, delta, partial
toolcall_end工具调用结束contentIndex, toolCall, partial
done成功终止reason, message
error错误终止reason, error

1.4 供应商工厂模式

每个供应商文件导出一个工厂函数，遵循此模式：

export function xxxProvider(): Provider<"api-name"> {
  return createProvider({
    id: "provider-id",
    name: "显示名称",
    baseUrl: "https://api.example.com",
    auth: { apiKey: envApiKeyAuth("描述", ["ENV_VAR"]) },
    models: Object.values(MODELS),
    api: openAICompletionsApi(),  // 绑定 API 适配器
  });
}
记忆要点：每个 api/*.ts 导出 stream + streamSimple；每个 providers/*.ts 导出工厂函数调用 createProvider 绑定认证、模型列表和 API 适配器。其余 88 个 provider 文件和 21 个 api 文件遵循完全相同的模式。

  • thinkingDisplay（第 1016 行）：GovCloud Bedrock 不支持 thinking.display，isGovCloudBedrockTarget() 检测后自动省略。
  • 错误格式化（第 330 行）：formatBedrockError() 将 SDK 异常名映射为人类可读前缀（如 InternalServerException → "Internal server error"），对 data retention mode 错误附加文档链接。

2.3 mistral-conversations.ts -- Mistral API

packages/ai/src/api/mistral-conversations.ts (665 行)

API 适配器 OpenAI 兼容变体

Mistral 是 OpenAI 兼容模式的变体代表 -- 大部分消息格式兼容，但在 thinking 表示、tool call ID 规范和 reasoning 模式上有自己的特色。

公共接口清单

导出签名/类型说明

  • 用量解析（第 274 行）：getMistralCachedPromptTokens() 兼容 Mistral SDK 的多种缓存 token 字段名（camelCase 和 snake_case）。
  • 图片降级：对于不支持图片的模型，自动替换为占位文本。

公共接口清单

导出函数签名说明

2.6 transform-messages.ts -- 跨供应商消息格式转换

packages/ai/src/api/transform-messages.ts (221 行)

API 适配器 基础设施

处理跨供应商消息的通用转换逻辑，是 pi-ai 支持跨模型对话的核心。负责三件事：图片降级、tool call ID 规范化、孤立 tool call 合成空结果。

公共接口清单

导出函数签名说明
transformMessages<TApi> (messages: Message[], model: Model<TApi>, normalizeToolCallId?: (id, model, source) => string) => Message[] 两遍处理：第一遍处理图片降级、thinking 块转换、tool call ID 规范化；第二遍为孤立的 tool call 插入合成 toolResult

转换规则

输入条件转换行为
不支持图片的模型 + 用户图片 图片替换为 "(image omitted: model does not support images)"，连续图片合并为一个占位
不支持图片的模型 + 工具结果图片 图片替换为 "(tool image omitted: ...)"
跨模型的 thinking 块 若无签名则转为 text 块；若为 redacted thinking 则丢弃
OpenAI Responses API 的 450+ 字符 tool call ID 通过 normalizeToolCallId 回调规范化为 64 字符内
无 toolResult 跟随的孤立 toolCall 自动插入 { role: "toolResult", isError: true, content: "No result provided" }
stopReason 为 error/aborted 的 assistant 消息 从消息列表中移除，不参与重放
跨模型的 thoughtSignature 自动删除（非同一模型无法使用）

4.1 event-stream.ts -- SSE 事件流基础架构

packages/ai/src/utils/event-stream.ts (89 行)

工具函数

所有 API 适配器的流式响应都基于这两个类。它实现了支持背压的异步迭代器，是 pi-ai 流处理的核心数据通道。

公共接口清单

导出签名说明
EventStream<T, R> class implements AsyncIterable<T> 泛型事件流基类，支持 push/end 模式、异步迭代和结果解析
AssistantMessageEventStream class extends EventStream<AssistantMessageEvent, AssistantMessage> 专门用于 AI 响应的流类，终端事件为 done/error
createAssistantMessageEventStream () => AssistantMessageEventStream 工厂函数，用于扩展场景

EventStream 核心行为

方法行为
push(event)若流未结束，将事件推入内部队列。若 isComplete(event) 为 true，标记 done 并 resolve 最终结果。若有待消费方则直接交付，否则入队等待。
end(result?)标记流结束，resolve 最终结果（可选），通知所有等待的消费者流已结束。
[Symbol.asyncIterator]()异步迭代器：先消费队列中的缓存事件，队列空时等待新事件被 push，done 后终止迭代。
result()返回 Promise，在流完成（done/error）时 resolve 为最终结果。
设计要点：这是典型的 push-pull 混合模式。生产者（API 适配器）通过 push 异步写入事件，消费者通过 for-await-of 异步拉取。内部队列和 waiting 数组保证生产者不会因无消费者而阻塞，消费者也不会因无数据而忙等。

6. 真实调用代码片段

6.1 使用 OpenAI 适配器进行流式调用

// 创建 Models 实例并注册供应商
const models = createModels();
const provider = openaiProvider();
models.registerProvider(provider);

// 获取模型并准备上下文
const model = models.getModel("openai", "gpt-4.1");
const context: Context = {
  systemPrompt: "你是一个有帮助的助手。",
  messages: [
    { role: "user", content: "你好！" }
  ],
  tools: [
    { name: "search", description: "搜索网络", parameters: { ... } }
  ]
};

// 调用 stream 并消费事件
const eventStream = provider.stream(model, context, {
  temperature: 0.7,
  maxTokens: 4096,
});

for await (const event of eventStream) {
  switch (event.type) {
    case "start":
      console.log("流式响应开始");
      break;
    case "text_delta":
      process.stdout.write(event.delta);
      break;
    case "toolcall_start":
      console.log(`工具调用: ${event.partial.content.at(-1)?.name}`);
      break;
    case "done":
      console.log(`完成: ${event.reason}`);
      break;
    case "error":
      console.error(`错误: ${event.error.errorMessage}`);
      break;
  }
}

// 获取最终结果
const result = await eventStream.result();
console.log(`用量: ${result.usage.totalTokens} tokens`);
