本文档是 pi-ai 模块总览课程的配套参考资料，提供完整的 exports 子路径清单、Provider 索引、API 协议对比等速查信息。建议先完成 0001-ai-module-tour.html 课程，再回到此处查阅细节。

1. Exports 子路径完整清单

导入路径编译产物主要内容典型用途
@earendil-works/pi-ai dist/index.js 核心类型（Model / Provider / Context）、认证工具（envApiKeyAuth / CredentialStore）、Provider 工厂函数类型、事件流、重试、TypeBox 辅助类型 大多数场景的默认入口
@earendil-works/pi-ai/compat dist/compat.js 旧版全局 API 兼容层，提供 getModels() / getProvider() 等全局注册函数 迁移旧代码或需要全局 Provider 注册的场景
@earendil-works/pi-ai/providers/openai dist/providers/openai.js openaiProvider() 工厂函数 + OpenAI 模型列表 按需加载单个 Provider，减少打包体积

@earendil-works/pi-ai/bedrock-provider dist/bedrock-provider.js AWS Bedrock 专用的独立入口，包含 resolveBedrockAuth() 等 需要直接配置 Bedrock 认证的场景

设计要点：./providers/* 和 ./api/* 使用通配符导出，允许按 Provider 或 API 协议精确定位，避免全量加载。主入口 . 被标记为 sideEffects: false（除了 compat、images 显式声明），对 tree-shaking 友好。

延迟加载机制：每个 API 实现文件都有对应的 *.lazy.ts 包装器（如 openai-responses.lazy.ts），通过 lazyApi() 动态导入，确保只有在实际调用时才加载具体的 SDK 模块。这避免了同时引入 5 个大型 SDK 导致的冷启动性能问题。

4. 认证体系概览

pi-ai 支持两种认证方式，通过 ProviderAuth 接口统一管理：

认证类型接口核心方法适用 Provider
API Key ApiKeyAuth resolve() — 从环境变量或凭据存储解析 API Key
login() — 交互式设置（可选）		 所有 Provider（至少提供 API Key 方式）
OAuth 2.0 OAuthAuth login() — 启动 OAuth 2.0 授权流程
refresh() — Token 过期时自动刷新
toAuth() — 从凭据衍生请求认证头		 Anthropic / GitHub Copilot 等支持 OAuth 的 Provider

凭据存储：CredentialStore 接口提供序列化的读写操作，modify() 方法是唯一的写路径，通过互斥锁保证并发安全。OAuth Token 刷新也在 modify() 内部完成，防止并发请求重复刷新。
