0001 — pi-ai 模块总览：多 Provider 统一 LLM API 层

层级：L1 模块总览 预计时长：12 分钟 学习目标：1 个
学习目标

理解 pi-ai 模块的整体架构与对外接口，能说出它在 Pi monorepo 中的定位、内部分层结构、exports 子路径清单以及关键外部依赖。

1. 模块职责与定位

pi-ai（npm 包名 @earendil-works/pi-ai）是 Pi 的多 Provider 统一 LLM API 层。 它的核心职责只有一句话：屏蔽不同 LLM 服务商的接口差异，向上层提供一致的调用方式。

在 Pi 的 5 包架构中（详见 L0 项目总览 第 3 章），pi-ai 位于最底层，与 pi-tui 并列。它不依赖任何其他 pi 包，只依赖各 Provider 的外部 SDK。pi-agent-core 和 pi-coding-agent 通过 pi-ai 访问 LLM，因此 pi-ai 的设计质量直接影响整个 Agent 系统的模型兼容性。

2. 对外接口清单

pi-ai 通过 package.json 的 exports 字段暴露 6 个子路径入口，支持按需导入：

子路径类型声明用途
../dist/index.d.ts主入口，导出核心类型、认证工具、Provider 工厂
./compat./dist/compat.d.ts旧版全局 API 兼容层
./providers/*./dist/providers/*.d.ts按 Provider 独立导入（如 ./providers/openai）
./api/*./dist/api/*.d.ts按 API 协议独立导入（如 ./api/openai-responses）
./oauth./dist/oauth.d.tsOAuth 认证流程入口
./bedrock-provider./dist/bedrock-provider.d.tsAWS Bedrock 专用的独立入口

此外，pi-ai 提供 CLI 命令 pi-ai（通过 bin 字段注册），支持 login（OAuth 登录）和 list（列出可用 Provider）两个子命令： 

npx @earendil-works/pi-ai login anthropic   # OAuth 登录到 Anthropic
npx @earendil-works/pi-ai list               # 列出所有支持 OAuth 的 Provider

3. 内部分层结构

pi-ai 的 src/ 目录按职责分为 4 层，自底向上构成完整的 LLM 调用链：

packages/ai/src/ ├── api/ ← 9 种 LLM API 协议实现 │ ├── openai-completions.ts ← OpenAI Chat Completions │ ├── openai-responses.ts ← OpenAI Responses API │ ├── anthropic-messages.ts ← Anthropic Messages │ ├── google-generative-ai.ts ← Google Gemini │ ├── bedrock-converse-stream.ts ← AWS Bedrock Converse │ ├── mistral-conversations.ts ← Mistral Chat │ ├── azure-openai-responses.ts ← Azure OpenAI │ ├── openai-codex-responses.ts ← OpenAI Codex │ ├── google-vertex.ts ← Google Vertex AI │ └── lazy.ts ← 延迟加载包装器（核心工具） ├── providers/ ← 35 个 Provider 工厂 + 模型元数据 │ ├── openai.ts ← openaiProvider() 工厂 │ ├── anthropic.ts ← anthropicProvider() 工厂 │ ├── all.ts ← builtinProviders() 聚合 │ └── *.models.ts ← 各 Provider 模型列表 ├── auth/ ← 认证抽象层 │ ├── types.ts ← ProviderAuth / CredentialStore 接口 │ ├── helpers.ts ← envApiKeyAuth 等辅助函数 │ └── resolve.ts ← 认证解析逻辑 ├── utils/ ← 工具函数 │ ├── event-stream.ts ← AssistantMessageEventStream │ ├── retry.ts ← 重试策略 │ └── oauth/ ← OAuth 2.0 完整实现 ├── index.ts ← 主入口，重导出所有公共 API ├── types.ts ← 核心类型定义（Api / Provider / Model） ├── models.ts ← Provider / Models 接口 + createModels └── models.generated.ts ← 自动生成的模型元数据注册表

数据流方向：上层调用 → Models 集合查找模型对应的 Provider → Provider 根据模型的 api 字段路由到对应的 API 实现 → API 实现调用底层 SDK → 通过 AssistantMessageEventStream 逐事件返回结果。

4. 关键依赖与调用示例

外部 SDK 依赖（全部锁定精确版本）：

SDK版本对应 Provider
openai6.26.0OpenAI / Azure / DeepSeek / Groq 等兼容 OpenAI 协议的 Provider
@anthropic-ai/sdk0.91.1Anthropic Claude 系列
@google/genai1.52.0Google Gemini
@aws-sdk/client-bedrock-runtime3.1048.0AWS Bedrock
@mistralai/mistralai2.2.6Mistral

内部依赖：无。pi-ai 不依赖任何 @earendil-works/pi-* 包。

一个最简调用示例——创建 Provider 并用 Models 集合管理：

import { builtinModels } from "@earendil-works/pi-ai";

// 创建包含所有内置 Provider 的 Models 集合
const models = builtinModels();

// 通过模型 ID 查找模型元数据（provider + api 自动确定）
const model = models.getModel("openai", "gpt-5-nano");
if (!model) throw new Error("Model not found");

// 调用 streamSimple 发送消息并获取流式响应
const stream = models.streamSimple(model, {
  messages: [{ role: "user", content: "Hello!" }]
});

for await (const event of stream) {
  if (event.type === "content") console.log(event.text);
}

这个示例展示了 pi-ai 的核心设计理念：使用者只需指定 Provider ID 和模型 ID，无需关心底层是哪个 SDK、哪种 API 协议。认证、流解析、模型发现全部由 pi-ai 内部处理。

点击查看参考答案
  1. 主入口 .（核心 API）、./compat（旧版兼容）、./providers/*（按 Provider 导入）、./api/*（按协议导入）、./oauth（OAuth 认证）、./bedrock-provider（Bedrock 专用入口）。
  2. 4 层结构：src/api/（9 种 LLM API 协议实现）→ src/providers/（35 个 Provider 工厂 + 模型列表）→ src/auth/（认证抽象，含 API Key 和 OAuth）→ src/utils/（事件流、重试、OAuth 等工具）。数据流：调用 → Models 查找 Provider → Provider 路由到 API 实现 → API 调用底层 SDK → EventStream 返回结果。
  3. 外部 SDK：openai、@anthropic-ai/sdk、@google/genai、@aws-sdk/client-bedrock-runtime、@mistralai/mistralai。pi-ai 不依赖任何 pi 内部包。
