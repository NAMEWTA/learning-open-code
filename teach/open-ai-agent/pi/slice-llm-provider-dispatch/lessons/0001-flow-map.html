0001 — LLM 请求全链路调度流图

层级：L2 垂直切片 预计时长：14 分钟 学习目标：1 个
学习目标

能画出从 builtinModels() 创建 Models 集合到 streamSimple() 返回流式响应的完整调用链路时序图，说出每一层的入口函数、关键数据结构和认证解析的三种来源。

1. 链路全景：6 层 9 步

一条 LLM 请求从初始化到收到首个 token 需要经过 6 个逻辑层：

  1. L0 集合初始化 — builtinModels() 创建 Models 并注册 35 个 Provider
  2. L1 模型查找 — getModel(provider, id) 从 Provider 的模型列表中匹配
  3. L2 认证解析 — resolveProviderAuth() 依次查询 override、credential store、环境变量
  4. L3 Provider 调度 — provider.streamSimple() 根据 model.api 分发到正确的 SDK
  5. L4 懒加载 — lazyApi() 动态 import() SDK 模块，lazyStream() 同步返回流
  6. L5 流式传输 — SDK 发起 HTTP 请求，逐 chunk 推入 AssistantMessageEventStream

关键洞察：认证解析是全链路中唯一可能异步失败的环节。Provider 调度和懒加载虽然也是异步的，但失败后会通过 lazyStream 转为流错误事件，不会抛异常。

图中关键决策点：

  • 认证优先级：request override > stored credential > env var。不会在存储凭证过期后静默降级到环境变量。
  • OAuth 特殊路径：如果凭证类型是 OAuth 且已过期，会在 credentialStore.modify() 内部加锁刷新 token，刷新失败抛出 ModelsError("oauth")。
  • Provider 未配置：返回 undefined 而非抛异常，上层可在 status UI 中显示"未配置"而非红色错误。

3. 三个关键节点

节点 1：streamSimple 的调度入口

ModelsImpl.streamSimple() 是整个链路的调度枢纽——找 Provider、解析认证、委托执行，三步走：

// packages/ai/src/models.ts — ModelsImpl.streamSimple
streamSimple(model, context, options) {
  return lazyStream(model, async () => {
    const provider = this.requireProvider(model);       // ① 找 Provider
    const { requestModel, requestOptions } =
      await this.applyAuth(model, options);              // ② 异步认证
    return provider.streamSimple(
      requestModel, context, requestOptions);            // ③ 委托执行
  });
}

注意外层 lazyStream() 保证了函数同步返回一个 EventStream，而认证和 SDK 调用在后台异步执行。这就是 Pi 对上层 Agent 运行时"无等待"的关键设计。

节点 2：认证解析三路分支

resolveProviderAuth() 按优先级查询三种认证来源，每种失败的处理方式不同：

// packages/ai/src/auth/resolve.ts — resolveProviderAuth（简化）
export async function resolveProviderAuth(provider, model, credentials, ctx, overrides?) {
  // ① 最高优先级：请求级 override
  if (overrides?.apiKey !== undefined && provider.auth.apiKey) {
    return resolveApiKey(ctx, provider.auth.apiKey, model,
      { type: "api_key", key: overrides.apiKey });
  }
  // ② 次优先级：持久化存储
  const stored = await readCredential(credentials, provider.id);
  if (stored) {
    if (stored.type === "oauth") return resolveStoredOAuth(...);
    if (stored.type === "api_key") return resolveApiKey(...);
    return undefined;
  }
  // ③ 兜底：环境变量 / AWS profile / ADC 文件
  return provider.auth.apiKey
    ? resolveApiKey(ctx, provider.auth.apiKey, model, undefined)
    : undefined;
}

节点 3：lazyStream 同步返回的秘密

lazyStream 创建外层 EventStream 后立即返回，setup 函数的 Promise 在后台执行：

// packages/ai/src/api/lazy.ts — lazyStream
export function lazyStream(model, setup) {
  const outer = new AssistantMessageEventStream();   // 同步创建
  setup()
    .then(inner => forwardStream(outer, inner))       // 成功：转发事件
    .catch(error => {
      outer.push({ type: "error", ... });              // 失败：推送错误事件
      outer.end(errorMessage);
    });
  return outer;  // 立即返回，不等 setup 完成
}

4. 异常路径：认证失败全追踪

认证失败是链路中最常见的异常。以下追踪一次 API Key 未配置且环境变量为空 的完整错误传播路径：

  1. 触发点：resolveApiKey() 中 apiKey.resolve({model, ctx, credential: undefined}) 被调用
  2. envApiKeyAuth.resolve() 检查 credential.key → 无；遍历 envVars → ctx.env("OPENAI_API_KEY") → undefined
  3. 返回 undefined — 注意这里不抛异常，因为 Provider 未配置是合法状态
  4. applyAuth() 收到 auth === undefined，不做任何合并，原样返回 model 和 options
  5. 请求到达 SDK，SDK 在发起 HTTP 请求时发现无 apiKey → SDK 自身抛出认证错误
  6. lazyStream 的 .catch() 捕获错误 → 构造 {type:"error", stopReason:"error"} 事件推入外层流
  7. 上层 Agent 收到 error 事件 → 在 TUI 中显示"认证失败"或"请配置 API Key"

为什么不在 resolveApiKey 中直接抛异常？

设计意图：返回 undefined 表示"Provider 未配置"（用户还没设置 Key），这是一个预期状态，status UI 应显示灰色而不是红色。而 ModelsError("auth") 表示"认证过程本身出错"（如 credential store 读写失败），这才是需要报红的异常。两种情况的严重程度不同。

不看上文，回答以下问题：

  1. 从 builtinModels() 到 streamSimple() 一共经过哪 6 个逻辑层？每层的入口函数是什么？
  2. resolveProviderAuth 的三种认证来源的优先级顺序是什么？OAuth token 过期后如何刷新？
  3. lazyStream() 为什么要同步返回 EventStream？如果 setup 失败，错误如何传递给调用方？
  4. Provider 未配置（无 API Key）和认证过程出错（如 store 读写失败）在错误处理上有什么不同？
点击查看参考答案
  1. L0 集合初始化 (builtinModels/setProvider) → L1 模型查找 (getModel) → L2 认证解析 (resolveProviderAuth) → L3 Provider 调度 (provider.streamSimple) → L4 懒加载 (lazyApi/lazyStream) → L5 流式传输 (SDK.streamSimple)。
  2. 优先级：request override apiKey > credential store > env var。OAuth 过期后，credentialStore.modify() 内部加锁，重新检查过期状态，调用 oauth.refresh() 刷新，成功后持久化新 token。
  3. 同步返回让上层 Agent 不需要 await 就能拿到流对象，可以立即开始迭代。setup 失败时，.catch() 构造 error 事件推入外层流，调用方通过迭代流自然会收到错误。
  4. 未配置返回 undefined（预期状态，UI 灰色）；认证过程出错抛出 ModelsError（异常状态，UI 红色）。前者在 status 检查时处理，后者在流中作为 error 事件传播。
