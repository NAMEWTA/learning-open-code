0001 — 构建脚本与基础设施模块总览

模块：module-scripts-infra 层级：L1 预计时间：12 分钟

学习目标：理解 Pi monorepo 中 scripts/、.github/workflows/ 和 .husky/ 三者如何构成分层质量门禁体系，并能按职责对 20+ 个脚本进行分类。

1. 模块在架构中的位置

在 Pi 的 monorepo 中，构建脚本与基础设施位于仓库根目录，是所有 5 个核心包（ai、agent、tui、coding-agent、orchestrator）的「外部骨骼」。它们不参与运行时逻辑，但决定了代码从开发者本地到 npm registry 的全过程质量。

分层结构：

scripts/ ← 自定义 Node.js/shell 脚本（20+ 个） ├── check-*.mjs ← 代码质量检查 ├── generate-*.mjs ← 锁文件/发布产物生成 ├── release.mjs ← 版本发布编排 ├── publish.mjs ← npm 发布 ├── build-binaries.sh ← 跨平台二进制构建 └── ... ← 统计/分析工具 .github/workflows/ ← CI/CD 流水线（9 个 workflow） ├── ci.yml ← 主 CI：build → check → test ├── build-binaries.yml ← 二进制构建 + npm 发布 ├── pr-gate.yml ← 外部贡献者 PR 门禁 └── ... .husky/ └── pre-commit ← 本地 Git 提交前检查

三者形成 三层防线：.husky/pre-commit 在本地拦截问题 → ci.yml 在 PR 时验证 → build-binaries.yml 在发布时确保产物正确。

2. 脚本职责分类

将 20+ 个脚本按职责分为 4 类，便于记忆和查找：

类别代表脚本触发方式
质量门禁 check-pinned-deps.mjs、check-ts-relative-imports.mjs、check-lockfile-commit.mjs、check-browser-smoke.mjs npm run check / pre-commit
锁文件生成 generate-coding-agent-shrinkwrap.mjs、generate-coding-agent-install-lock.mjs npm run shrinkwrap:coding-agent / CI
版本与发布 release.mjs、local-release.mjs、publish.mjs、sync-versions.js 手动触发 / CI tag push
二进制构建 build-binaries.sh 手动触发 / build-binaries.yml

此外还有 分析工具（edit-tool-stats.mjs、session-context-stats.mjs、cost.ts 等），用于开发期性能分析和 token 成本追踪，不属于质量门禁流水线。

3. 核心流水线：npm run check

npm run check 是开发者最常接触的命令。从 package.json 的 scripts 段可以看到，它串联了 6 个步骤：

{
  "check": "biome check --write --error-on-warnings . &&
            npm run check:pinned-deps &&
            npm run check:ts-imports &&
            npm run check:shrinkwrap &&
            npm run check:install-lock:coding-agent &&
            tsgo --noEmit &&
            npm run check:browser-smoke"
}

步骤解析：

  1. Biome 格式化 + Lint — 自动修复格式问题，有 warning 则失败。
  2. 依赖精确版本检查 (check:pinned-deps) — 所有外部 npm 依赖必须是精确版本号（不含 ^ ~），防止 floating dependency 引入不可重现构建。
  3. TS 相对导入检查 (check:ts-imports) — .ts 源文件中不允许出现相对路径的 .js 导入，确保编译后的模块解析一致性。
  4. Shrinkwrap 一致性 (check:shrinkwrap) — 验证 npm-shrinkwrap.json 与当前 package-lock.json 同步，防止发布时混入非预期的依赖版本。
  5. Install Lock 一致性 (check:install-lock:coding-agent) — 验证安装器使用的 lockfile 与源码同步。
  6. tsgo 类型检查 — 使用 TypeScript 7 原生编译器做全量类型检查，不产生 emit。
  7. 浏览器冒烟测试 (check:browser-smoke) — 验证 pi-ai 包在浏览器环境可正常加载（ESM 兼容性）。

这 7 步中任何一步失败，pre-commit hook 都会拒绝提交。它的设计哲学是 "pre-commit 即 mini-CI"——在本地尽可能早地发现所有可自动检测的问题。

4. 发布流水线：从本地到 npm

完整发布流程由 release.mjs 编排（9 步），核心代码结构如下：

// release.mjs 主流程（简化）
// 1. 检查工作区是否干净
const status = run("git status --porcelain");
if (status.trim()) { process.exit(1); }

// 2. 递增版本号（lockstep 统一升级）
const version = bumpOrSetVersion("patch"); // 或 minor/major

// 3. 更新所有 CHANGELOG.md：[Unreleased] → [version] - date
updateChangelogsForRelease(version);

// 4. 重新生成发布产物（AI models、shrinkwrap、install-lock）
run("npm --prefix packages/ai run generate-models");
run("npm run shrinkwrap:coding-agent");
run("npm run install-lock:coding-agent");

// 5. 运行 npm run check（再次验证）
run("npm run check");

// 6. git commit + tag（含所有变更）
run(`git commit -m "Release v${version}"`);
run(`git tag v${version}`);

// 7-8. 添加下一周期的 [Unreleased] 占位并提交
// 9. git push → 触发 CI 中的 build-binaries.yml

关键防护点：

  • lockstep 版本 — 5 个包必须同版本号发布，sync-versions.js 自动同步包间依赖为 ^x.y.z。
  • 产物再生 — 每次发布重新生成 AI model types、shrinkwrap 和 install-lock，确保发布包包含最新的依赖图。
  • 二次 check — 发布前再次运行 npm run check，因为版本变更可能引起新的 lint/type 问题。
  • tag 触发 CI — push tag 后 build-binaries.yml 自动构建 6 平台二进制、上传 draft Release、publish npm。

检索练习

问题：Pi 的 npm run check 流水线中，check:pinned-deps 和 check:shrinkwrap 分别解决什么不同的质量风险？为什么两者同时存在？

点击查看参考答案

check:pinned-deps 防止的是 开发期 风险：package.json 中声明 ^1.2.3 会在不同时间 resolve 到不同版本，导致不可重现构建。它要求所有外部依赖必须精确到 patch 版本。

check:shrinkwrap 防止的是 发布期 风险：即使 package.json 锁定了版本，如果 package-lock.json 更新了但忘记重新生成 npm-shrinkwrap.json，用户安装时会拿到过时的依赖图。两者同时存在是因为一个管"声明了什么"，一个管"实际锁定了什么"。
