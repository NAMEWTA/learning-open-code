0001 — Agent Hook 生命周期全链路地图

层级：L2 垂直切片 预计时长：14 分钟 学习目标：1 个
学习目标

理解一个 Hook 事件从 pi.on() 注册到 handler 执行完毕的完整链路——穿越 types.ts（契约）、loader.ts（注册）、runner.ts（执行）三层，区分 6 种专用 emit 方法的执行策略差异。

1. 三层架构与入口点

Hook 系统分布在三个文件中，形成声明→注册→执行三层架构。链路有两个入口——写扩展时的 pi.on("event", handler)（注册入口），运行时 Agent 循环调用 runner.emit(event)（触发入口）。

定义 31 种 Event 接口、Result 接口、ExtensionHandler 泛型、ExtensionAPI.on() 的 31 个重载签名。

ExtensionEvent 联合类型包含 31 种事件，按生命周期阶段分为 6 大类：

分类代表性事件emit 方式策略
Resource project_trust, resources_discover 专用函数 首个决策 / 收集合并
Session session_start/shutdown, session_before_* ×4, session_compact/tree/info_changed emit() 通用 before_* 支持 cancel 短路
Agent before_agent_start, context, before_provider_request, agent_start/end, turn_start/end, message_start/update/end 混合：通用 + 4 种专用 链式修改 / 合并结果
Tool tool_call/result, tool_execution_start/update/end, user_bash 3 种专用 emit block 短路 / 链式修改
Model model_select, thinking_level_select emit() 通用 通知型（无返回值）
Input input emitInput() transform 链 / handled 短路
关键区分：emit() 通用方法处理大部分事件，但 6 种核心事件拥有专用 emit 方法——其返回值和短路逻辑无法用统一签名表达。

3. 六种专用 emit 方法策略

通用 emit() 核心逻辑：遍历扩展 → 匹配 handler → 逐个调用 → 捕获异常。session_before_* 事件中任一 handler 返回 {cancel: true} 立即短路。

emitContext(messages) — 链式修改。handler 返回替换后的 messages 数组，后续 handler 收到累积修改结果。核心用 structuredClone 保护原始数组，支持 AbortSignal 中断。
emitToolCall(event) — block 短路。handler 返回 {block: true} 则工具调用被阻止。event.input 为 mutable，handler 可直接修改参数。
emitToolResult(event) — 链式修改。handler 可部分覆写 content / details / isError，后续 handler 看到累积结果。
emitMessageEnd(event) — 链式替换 + role 校验。handler 替换整个 message 但必须保持原 role。若 role 不一致，记录 ExtensionError 并丢弃该结果。
emitBeforeAgentStart(...) — 收集 + 链式。多个 handler 的 message 结果收集为数组，systemPrompt 链式传递。ctx.getSystemPrompt() 被覆写为返回当前链式结果。
emitInput(text, ...) — transform / handled 短路。返回 {action:"transform", text} 替换文本，{action:"handled"} 短路跳过内置处理。

session_before_* 短路逻辑

// runner.ts — emit() 核心循环
for (const handler of handlers) {
  const handlerResult = await handler(event, ctx);
  if (this.isSessionBeforeEvent(event) && handlerResult) {
    result = handlerResult;
    if (result.cancel) {
      return result; // 立即停止，跳过后续 handler
    }
  }
}

4. 异常路径与防御机制

4.1 Handler 抛出异常

所有 emit 方法用 try/catch 包裹 handler 调用。异常被包装为 ExtensionError{extensionPath, event, error, stack}，通过 emitError() 通知 errorListeners。单个 handler 崩溃不影响后续 handler 执行。

// runner.ts — 异常处理模式
try {
  const handlerResult = await handler(event, ctx);
} catch (err) {
  this.emitError({
    extensionPath: ext.path, event: event.type,
    error: err instanceof Error ? err.message : String(err),
    stack: err instanceof Error ? err.stack : undefined,
  });
}

4.2 运行时未初始化 (Throwing Stub)

loader.ts 创建的 ExtensionRuntime 中所有 action 方法（sendMessage、setModel 等）初始化为 throwing stub。在 factory(api) 执行期间若调用它们，抛出 "Extension runtime not initialized"。bindCore() 注入真实实现后替换所有 stub。

4.3 Stale 扩展实例

reload / newSession / fork / switchSession 后，旧 ExtensionRunner 调用 invalidate() 设置 staleMessage。此后对旧 ctx 的任何访问触发 assertActive() 抛出异常，防止在已销毁上下文中执行操作。
