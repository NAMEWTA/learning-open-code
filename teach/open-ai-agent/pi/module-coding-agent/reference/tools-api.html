2. Bash 工具 (bash.ts) 最复杂工具

源码: packages/coding-agent/src/core/tools/bash.ts (470 行)

Bash 工具是整个工具层中逻辑最复杂的工具。它支持本地 Shell 命令执行、子进程管理、超时控制、AbortSignal 取消、流式输出累加和截断。

2.1 输入 Schema

TypeBox Schema
const bashSchema = Type.Object({
  command: Type.String({ description: "Bash command to execute" }),
  timeout: Type.Optional(Type.Number({ description: "Timeout in seconds (optional, no default timeout)" })),
});
参数类型必需约束说明
command string 是 非空字符串 要执行的 Bash 命令。会被 commandPrefix 包裹（如果配置了）。
timeout number 否 正整数，最大值 2,147,483 秒 超时秒数。为 undefined 时不设超时；非正数或非有限数时抛异常。

2.2 核心接口

BashOperations（可插拔执行后端）

interface BashOperations {
  exec(
    command: string,

exitCode 为 null 表示进程被杀死（未正常退出）。

2.4 边界条件

边界条件与处理策略

  • 工作目录不存在：createLocalBashOperations.exec() 在 spawn 前通过 fsAccess(cwd, F_OK) 检查，不存在时抛出 Error("Working directory does not exist: ...")。
  • 空命令 / 空输出：空命令直接传给 Shell（由 Shell 处理）；无输出时返回 "(no output)"。
  • 命令超时：通过 setTimeout 触发 killProcessTree(child.pid)，然后抛出 Error("Command timed out after {timeout} seconds")。
  • AbortSignal 取消：在 signal.addEventListener("abort", ...) 中调用 killProcessTree，然后抛出 Error("Command aborted")。
  • 非零退出码：不抛异常，而是通过 throw new Error(appendStatus(text, "Command exited with code N")) 向上传递，附带已产生的输出。
  • 输出截断：默认保留最后 2000 行或 50KB。若截断，完整输出写入 $TMPDIR/pi-bash-{random}.log 并在结果中提示路径。
  • Windows 平台：子进程以 detached: process.platform !== "win32" 方式启动，Windows 下不 detach 以避免句柄泄漏。
  • stdin 传输模式：Shell 配置中 commandTransport === "stdin" 时，命令通过 stdin pipe 传入而非作为 CLI 参数。
  • 超时参数验证：最大值 2_147_483_647 ms（约 24.8 天），非有限数或非正数抛出 Error("Invalid timeout: ...")。

2.5 异常类型

可能抛出的异常

  • Error("Invalid timeout: must be a finite number of seconds") — timeout 为非正数或 NaN/Infinity
  • Error("Invalid timeout: maximum is 2147483 seconds") — timeout 超出上限
  • Error("Working directory does not exist: ...") — cwd 路径不存在
  • Error("aborted") — AbortSignal 触发
  • Error("Command aborted") — 用户取消（附带已产生输出）
  • Error("timeout:{seconds}") / Error("Command timed out after {seconds} seconds") — 超时
  • Error("...Command exited with code {N}") — 非零退出码（附带已产生输出）

3. Edit 工具 (edit.ts) 核心工具

源码: packages/coding-agent/src/core/tools/edit.ts (437 行)

Edit 工具实现基于精确文本替换的文件编辑功能。支持单次调用包含多个不重叠的编辑块（edits[]），所有编辑块均基于原始文件内容进行匹配（非增量应用）。

3.1 输入 Schema

TypeBox Schema

  newText: Type.String({ description: "Replacement text for this targeted edit." }),
});

const editSchema = Type.Object({
  path: Type.String({ description: "Path to the file to edit (relative or absolute)" }),
  edits: Type.Array(replaceEditSchema, { description: "One or more targeted replacements..." }),
});
参数类型必需约束说明
path string 是 文件必须存在且可读写 要编辑的文件路径（相对或绝对）
edits Edit[] 是 至少 1 个元素 编辑块数组，每个包含 oldText 和 newText
edits[].oldText string 是 非空，文件中必须唯一匹配 要替换的原始文本

3.4 边界条件

边界条件与处理策略

  • edits 数组为空：validateEditInput 抛出 Error("Edit tool input is invalid. edits must contain at least one replacement.")。
  • 旧版参数兼容：prepareEditArguments 会检测顶层 oldText/newText 字段（旧版调用方式），自动合并到 edits[] 数组末尾。
  • JSON 字符串 edits：某些模型（Opus 4.6、GLM-5.1）将 edits 作为 JSON 字符串发送，prepareEditArguments 会自动解析。
  • 文件不存在：ops.access() 失败时抛出 Error("Could not edit file: {path}. Error code: ...")。
  • oldText 为空：applyEditsToNormalizedContent 抛出 Error("oldText must not be empty in {path}.")。
  • oldText 未找到：若精确匹配和模糊匹配均失败，抛出 Error("Could not find the exact text in {path}.")。
  • oldText 重复匹配：若 oldText 在文件中出现多次，抛出 Error("Found {N} occurrences of the text in {path}.")。
  • 编辑块重叠：若两个 edit 的匹配区域重叠，抛出 Error("edits[{i}] and edits[{j}] overlap in {path}.")。
  • 无实际变更：若 baseContent === newContent，抛出 Error("No changes made to {path}.")。
  • AbortSignal：在每次 await 后检查 signal?.aborted，不通过事件监听器 reject（以保持 mutation queue 锁定）。
  • 文件变更队列：通过 withFileMutationQueue 序列化对同一文件的操作，防止并发写入冲突。
  • BOM 处理：读取时剥离 UTF-8 BOM（﻿），写入时重新添加。
  • 换行符保留：检测原始文件的 CRLF/LF，在输出时还原。
  • 模糊匹配：精确匹配失败后，尝试去除行尾空格、Unicode 智能引号/破折号/空格归一化后再匹配。

3.5 异常类型

可能抛出的异常

  • Error("Edit tool input is invalid. edits must contain at least one replacement.")
  • Error("Could not edit file: {path}. {errorMessage}.")
  • Error("oldText must not be empty in {path}.")
  • Error("Could not find the exact text in {path}.")
  • Error("Could not find edits[{i}] in {path}.")
  • Error("Found {N} occurrences of the text in {path}.")
  • Error("Found {N} occurrences of edits[{i}] in {path}.")
  • Error("edits[{i}] and edits[{j}] overlap in {path}.")
  • Error("No changes made to {path}.")
  • Error("Operation aborted")
  • Error("Cannot preserve unchanged lines because the base content has a different line count.")
  • Error("Replacement range is outside the base content.")

4.4 边界条件

边界条件与处理策略

  • offset 超出文件行数：抛出 Error("Offset {N} is beyond end of file ({M} lines total)")。
  • 首行超过字节限制：返回提示文本，建议用 bash: sed -n 'Lp' {path} | head -c {max} 回退方案。

6. Grep 工具 (grep.ts)

源码: packages/coding-agent/src/core/tools/grep.ts (385 行)

Grep 工具基于 ripgrep (rg) 实现内容搜索，支持正则/字面量搜索、glob 过滤、上下文行等。自动下载 rg 二进制（如果系统未安装）。

6.1 输入 Schema

TypeBox Schema
const grepSchema = Type.Object({
  pattern: Type.String({ description: "Search pattern (regex or literal string)" }),
  path: Type.Optional(Type.String({ description: "Directory or file to search (default: current directory)" })),
  glob: Type.Optional(Type.String({ description: "Filter files by glob pattern, e.g. '*.ts' or '**/*.spec.ts'" })),
  ignoreCase: Type.Optional(Type.Boolean({ description: "Case-insensitive search (default: false)" })),
  literal: Type.Optional(Type.Boolean({ description: "Treat pattern as literal string instead of regex (default: false)" })),
  context: Type.Optional(Type.Number({ description: "Number of lines to show before and after each match (default: 0)" })),
  limit: Type.Optional(Type.Number({ description: "Maximum number of matches to return (default: 100)" })),
});
参数类型必需默认约束说明
patternstring是-非空正则表达式或字面量搜索字符串
pathstring否"."必须存在搜索目录或文件路径
globstring否--文件 glob 过滤器，如 *.ts
ignoreCaseboolean否false-大小写不敏感搜索
literalboolean否false-字面量模式（非正则）
contextnumber否0必须为正匹配行前后的上下文行数
limitnumber否100最小为 1最大匹配数

6.2 返回值详情

GrepToolDetails
interface GrepToolDetails {
  truncation?: TruncationResult;    // 截断信息
  matchLimitReached?: number;       // 达到匹配限制时的实际 limit 值
  linesTruncated?: boolean;         // 是否有行被截断（超过 GREP_MAX_LINE_LENGTH=500 字符）
}

6.3 边界条件

边界条件与处理策略

  • rg 不可用：自动通过 ensureTool("rg", true) 下载。下载失败则抛出 Error("ripgrep (rg) is not available and could not be downloaded")。
  • 路径不存在：isDirectory 检查失败时抛出 Error("Path not found: {path}")。
  • 无匹配：返回 content: [{ type: "text", text: "No matches found" }]，details 为 undefined。
  • 匹配数达到 limit：设置 matchLimitReached，输出末尾提示 "{N} matches limit reached. Use limit={2N} for more, or refine pattern"。
  • 长行截断：超过 500 字符的行截断并在末尾加 ... [truncated]，设置 linesTruncated: true。
  • 输出字节截断：默认 50KB。截断时提示 "{size} limit reached"。
  • AbortSignal：通过 settle 闭包确保只 resolve/reject 一次。abort 时 kill rg 子进程。
  • rg 异常退出码：exit code 不为 0 或 1 时，以 stderr 内容抛出异常。
  • 上下文行：当 context > 0 时，读取匹配文件获取上下文，非匹配行以 {path}-{line}- 格式输出。

9.3 边界条件
