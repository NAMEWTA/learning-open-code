0001 — 扩展系统加载与执行全链路

层级：L2 垂直切片 预计时长：14 分钟 学习目标：1 个 父模块：L1-module-coding-agent
学习目标

理解 Pi 扩展系统从声明到运行时注入的完整链路 —— 掌握扩展发现、加载、类型校验、能力注册、事件触发的六个阶段，能画出全链路时序图并识别关键异常路径。

1. 扩展能力速览

Pi 扩展是 TypeScript 模块，通过默认导出 (pi: ExtensionAPI) => void 的工厂函数来注册能力。 一个扩展可以同时拥有以下五种能力中的任意组合：

能力API 方法运行时存储位置
Hook 事件监听pi.on(event, handler)extension.handlers（Map）
LLM 可调用工具pi.registerTool(def)extension.tools（Map）
斜杠指令pi.registerCommand(name, opts)extension.commands（Map）
自动补全提供者pi.ui.addAutocompleteProvider(factory)TUI 组件内部
UI 组件（页脚/编辑器/Widget）pi.ui.setFooter/setEditor/setWidgetTUI 渲染树

其中前三种（Hook/Tool/Command）是核心，它们的注册数据存储在 Extension 对象的 Map 中，供运行时遍历。后两种直接注入 TUI 渲染管线。

    Note over CLI,Agent: 阶段4-6: 运行时事件

    Agent->>Runner: emitBeforeProviderRequest
关键设计：两阶段初始化 —— loader 创建 ExtensionRuntime 时，action 方法（sendMessage、setModel 等）都是 throwing stub，因为此时会话、模型注册表等核心组件尚未就绪。只有 runner.bindCore() 调用后，stub 才被替换为真实实现。这确保了扩展加载阶段（factory 执行时）不会意外调用未就绪的 API。

3. 发现规则详解

discoverExtensionsInDir(dir) 按以下优先级扫描指定目录：

  1. 直接 .ts / .js 文件 → 作为扩展入口加载
  2. 子目录中的 index.ts / index.js → 加载该入口文件
  3. 子目录中的 package.json（含 pi.extensions 字段） → 加载字段声明的入口文件，同时其 node_modules 对外部依赖可见（如 sandbox、with-deps 示例）

不递归：仅扫描一层目录。复杂扩展包必须通过 package.json 的 pi.extensions 字段声明入口。

with-deps/ ← 目录名不限制 ├── package.json ← pi.extensions: ["./index.ts"] ├── node_modules/ ← 扩展的独立依赖 └── index.ts ← 默认导出的工厂函数

示例中 with-deps/package.json 的 pi 字段：

{
  "pi": {
    "extensions": ["./index.ts"]
  },
  "dependencies": {
    "ms": "2.1.3"
  }
}

在 Bun binary 模式下使用 virtualModules（预打包的 pi 包），Node.js 开发模式下使用 alias（解析到 monorepo 本地路径），确保扩展能 import @earendil-works/pi-coding-agent 等包。

4. 运行时事件分发

ExtensionRunner 提供两类事件分发方法：

方法处理的事件特殊行为
emit(event)通用事件（session_start、turn_start 等 15+ 种）session_before_* 事件支持 cancel 短路
emitToolCall(event)tool_callhandler 可返回 { block: true } 阻止执行
emitToolResult(event)tool_resulthandler 可逐字段修改 content/details/isError
emitContext(messages)context链式替换：每个 handler 接收前一个的输出；支持 AbortSignal
emitBeforeProviderRequest(payload)before_provider_request链式替换请求 payload
emitBeforeAgentStart(...)before_agent_start收集 messages 数组 + 链式替换 systemPrompt
emitMessageEnd(event)message_end链式替换消息体；验证 role 不变

通用 emit() 的核心逻辑：

async emit(event) {
  const ctx = this.createContext(); // 延迟求值的上下文
  for (const ext of this.extensions) {
    for (const handler of ext.handlers.get(event.type) ?? []) {
      try {
        const result = await handler(event, ctx);
        if (this.isSessionBeforeEvent(event) && result?.cancel) {
          return result; // 短路：停止后续 handler
        }
      } catch (err) {
        this.emitError({ ... }); // 通知监听器，不影响其他 handler
      }
    }
  }
}

异常路径：扩展加载失败

两种失败场景：

  1. 模块加载失败（jiti.import 抛异常、factory 不是函数）→ 错误写入 LoadExtensionsResult.errors，该扩展被跳过，不影响其他扩展
  2. handler 执行异常（运行时抛错）→ emitError() 通知监听器（通常打印 warning 到控制台），不中断其他 handler，也不中断 Agent 主流程

此外，session 替换或 reload 后，旧的 pi/ctx 引用通过 assertActive() 检查变为 stale，再次调用会抛出明确错误信息，防止使用过期上下文。

检索练习

不看上文，尝试回答：

  1. 扩展发现的三个优先级是什么？package.json 中的 pi.extensions 字段起什么作用？
  2. 为什么 createExtensionRuntime() 先创建 throwing stub，再由 bindCore() 替换？
  3. emitContext() 和通用 emit() 的核心区别是什么？
点击查看参考答案
  1. 三个优先级：项目本地 .pi/extensions/ → 全局 ~/.pi/extensions/ → 显式配置路径。pi.extensions 字段声明扩展包的入口文件列表，同时使 node_modules 对外部依赖可见。
  2. 扩展加载阶段（factory 执行时），会话、模型注册表等核心组件尚未就绪。stub 确保在此期间调用 action API 会立即抛出明确错误，避免静默失败。bindCore 后才注入真实实现。
  3. emitContext() 是链式替换模式 —— 每个 handler 接收前一个 handler 修改后的 messages 数组；通用 emit() 中每个 handler 接收相同的原始 event。此外 emitContext 支持 AbortSignal 中断。
