0001 - CLI 入口与模式选择全链路

模块：pi-coding-agent 层级：L2 垂直切片 预计时间：12 分钟 依赖：L1-module-coding-agent

学习目标：追踪从终端输入 pi 到 Agent 会话启动的完整代码路径，理解 9 阶段启动管线的每一层职责，以及 interactive / print / RPC 三种模式的分发决策逻辑。

1. 入口文件骨架

用户执行 pi 命令时，Node.js 根据 package.json 的 "bin" 字段定位到两个入口之一：

packages/coding-agent/src/ ├── cli.ts ← 主入口（#!/usr/bin/env node） ├── rpc-entry.ts ← RPC 入口（#!/usr/bin/env node） ├── main.ts ← 共享主流程 main(args) └── cli/ ├── args.ts ← parseArgs + printHelp ├── startup-ui.ts ← shouldRunFirstTimeSetup + showStartupSelector └── session-picker.ts ← selectSession (--resume 的 TUI 选择器)

cli.ts（11 行）只做三件事：设置进程标题、抑制 Node 内部 emitWarning 防止污染 TUI、配置 HTTP dispatcher、然后调用 main(process.argv.slice(2))。rpc-entry.ts 与它几乎相同，唯一区别是在 argv 前插入 --mode rpc，供 orchestrator 通过子进程启动。

3. 关键环节详解

3 参数解析

parseArgs() 使用一个 for 循环遍历 process.argv.slice(2)，用 if-else 链匹配 50+ 个标志。解析结果存入 Args 接口：

  • 运行模式：--mode rpc|json|text、-p / --print
  • 模型选择：--provider、--model、--models、--thinking、--api-key
  • 会话控制：--continue、--resume、--session、--session-id、--fork、--no-session
  • 工具控制：-t / --tools、-xt / --exclude-tools、-nt / --no-tools
  • 扩展与资源：-e / --extension、--skill、--theme、--system-prompt

无法识别的 --flag 不会被丢弃，而是进入 unknownFlags: Map，留待扩展系统自省。

5 模式决策

resolveAppMode() 是 4 行函数，逻辑清晰：

function resolveAppMode(parsed, stdinIsTTY, stdoutIsTTY): AppMode {
  if (parsed.mode === "rpc")   return "rpc";
  if (parsed.mode === "json")  return "json";
  if (parsed.print || !stdinIsTTY || !stdoutIsTTY) return "print";
  return "interactive";
}

优先级：rpc > json > print > interactive。管道输入（echo "..." | pi）会自动将 interactive 降级为 print。

7 会话恢复四种策略

标志行为失败处理
--continue打开当前 cwd 下最近一次会话无历史时创建新会话
--resume弹出 TUI 选择器浏览所有会话用户取消则退出
--session <id>按 ID 前缀匹配：先本地 → 后全局找不到则报错退出
--fork <id>复制已有会话到新会话（保留历史）源不存在则报错退出

8 运行时工厂闭包

createRuntime 是 main() 内部定义的异步工厂函数。它被 createAgentSessionRuntime() 包装后，在首次启动和会话重载（session reload）时各执行一次。工厂内部完成：

  1. 根据 parsed 构建 ResourceLoaderOptions（扩展路径、skills、themes、context files）
  2. 调用 createAgentSessionServices() 创建 SettingsManager / ModelRegistry / ResourceLoader
  3. 加载扩展、收集启动诊断（settings 错误、扩展加载失败）
  4. 解析 --models 为 ScopedModel[]，构建 CreateAgentSessionOptions
  5. 处理 --api-key 的运行时注入
  6. 调用 createAgentSessionFromServices() 生成 Agent 实例

异常路径：三种典型启动失败

  • 无可用模型：print/json 模式下 session.model === undefined → 输出 "No models available" 提示并 exit(1)
  • 扩展加载失败：诊断中包含 "Failed to load extension" → 打印错误 + "Start without extensions using pi -ne" 提示后 exit(1)
  • 会话 cwd 缺失：全局会话没有关联工作目录 → interactive 模式弹出选择器让用户选择 cwd，非 interactive 模式直接报错 MissingSessionCwdError

4. 练习与下一步

检索练习

问题 1： 执行 echo "列出文件" | pi 时，resolveAppMode() 返回什么？为什么？

问题 2： --session abc123 和 --session-id abc123 在本地已存在该会话时的行为有何不同？（提示：阅读 createSessionManager()）

问题 3： createRuntime 工厂闭包为什么会在会话重载时被再次调用？什么场景触发重载？

参考答案

A1： 返回 "print"。因为 process.stdin.isTTY 为 false（管道输入），触发 !stdinIsTTY 分支。即使原本是 interactive，管道输入也会降级为 print。

A2： --session abc123 支持前缀模糊匹配（startsWith），且找不到时会搜索全局项目；--session-id abc123 要求精确匹配，找不到且没有其他会话标志冲突时，会创建新会话（ID 由 SessionManager.create 的 id 选项控制）。

A3： createAgentSessionRuntime() 的设计支持 session reload——当 Agent 运行过程中需要重新初始化（如切换模型、重载配置），会再次调用工厂创建新的 AgentSession。这避免了 main() 中硬编码一次性初始化逻辑。
