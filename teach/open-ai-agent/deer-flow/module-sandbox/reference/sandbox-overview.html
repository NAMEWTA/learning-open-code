Sandbox 与文件系统参考

本页是 L1 速查材料：模块边界、对外接口、内部分层、provider 差异、配置项和测试锚点。微观 API 签名与边界见 L3 sandbox-api.html；短课入口：0001-sandbox-module-tour.html；整体架构锚点见 L0 项目总览。

模块职责与架构位置

维度结论源码依据
一句话职责Sandbox 模块给 agent 提供受控执行环境和线程级文件系统视图：选择 provider、分配 sandbox、映射虚拟路径、执行 bash/file tools，并把 outputs 目录接入 artifact 读取链路。backend/packages/harness/deerflow/sandbox/sandbox.py、backend/packages/harness/deerflow/sandbox/tools.py
整体位置位于 lead agent middleware 链和 Gateway 文件路由之间。Lead agent 通过 tools 使用它；Gateway uploads/artifacts 通过同一套线程目录读写它。backend/packages/harness/deerflow/sandbox/middleware.py、backend/app/gateway/routers/artifacts.py
核心边界agent 看到 `/mnt/user-data/...`、`/mnt/skills/...`、`/mnt/acp-workspace/...` 或配置 mount；host 绝对路径只应在 provider、Gateway 和 path resolver 内部出现。backend/packages/harness/deerflow/config/paths.py、backend/packages/harness/deerflow/sandbox/local/local_sandbox.py
非职责不决定模型策略，不实现前端 artifact UI，不替代 Gateway 认证授权，不保证 LocalSandboxProvider 的 host bash 是强隔离边界。backend/packages/harness/deerflow/sandbox/security.py、backend/docs/CONFIGURATION.md

虚拟路径契约

虚拟路径host 语义读写规则关联入口
/mnt/user-data/workspace线程工作区，agent 放置临时脚本、项目文件、运行日志。读写允许；Local bash 会把相对路径锚定到 workspace。Paths.sandbox_work_dir()、ThreadDataMiddleware
/mnt/user-data/uploads用户通过 Gateway 上传的文件。读允许；写也允许，但语义上不应把最终产物写在这里。uploads.py、upload_virtual_path()
/mnt/user-data/outputsagent 生成的 artifact 文件。读写允许；Gateway artifacts router 从这里读回文件。Paths.sandbox_outputs_dir()、artifacts.py
/mnt/skills/public全局 public skills 目录。只读；disabled skill 会被工具层拒绝访问。LocalSandboxProvider._setup_path_mappings()、tools.py
/mnt/skills/custom当前用户的 custom skills 目录。只读；按 user_id 隔离。LocalSandboxProvider._build_thread_path_mappings()
/mnt/acp-workspace线程级 ACP agent workspace。在工具校验中作为只读路径族；provider 可用 mount 或映射暴露。Paths.acp_workspace_dir()、AIO thread mounts
自定义 mountsandbox.mounts[].host_path 到 container_path 的映射。按 read_only 约束；不能覆盖保留前缀。SandboxConfig.mounts、Local/AIO provider

Provider 差异速查

Provider隔离与执行文件同步模型生命周期与容量主要风险/注意点
LocalSandboxProvider在 Gateway 进程可见的本地文件系统上工作；host bash 默认禁用。直接用 PathMapping 把虚拟路径解析到线程目录；uses_thread_data_mounts=True。同一 `(user_id, thread_id)` 复用一个 LocalSandbox；LRU 默认最多缓存 256 个线程。不是强 shell isolation；只适合可信本地工作流。Docker Compose 下 mount 的 host_path 是 Gateway 容器视角。
AioSandboxProvider 本地容器通过 Docker 或 Apple Container 启动 AIO sandbox 容器。把 workspace/uploads/outputs bind mount 到容器内 `/mnt/user-data/...`；本地 backend 上 uses_thread_data_mounts=True。线程锁 + 跨进程文件锁；确定性 sandbox id；warm pool；idle timeout；replicas 软上限。Docker-outside-of-Docker 模式要理解 Docker socket 风险；per-command secret env 需要支持 /v1/bash/exec 的镜像。
AioSandboxProvider remote/provisioner通过 provisioner 或 remote backend 连接独立 sandbox。remote backend 可能没有共享 host mount，上传文件需要显式 sync。provider 仍负责发现、创建、ready polling、release/warm pool。调试重点从本地目录变成 provisioner、Pod/Service 和 remote backend health。
E2BSandboxProvider使用 E2B cloud micro-VM 和 e2b_code_interpreter SDK。无 host bind mount；启动时 one-shot 上传 configured mounts；`/mnt/user-data` 被映射到 remote home。通过 E2B metadata 发现同一 `(user_id, thread_id)` 的 VM；warm pool；server-side idle timeout。outputs 需要通过 remote file API 下载或同步；VM 可能被控制面回收，provider 会在 ping/reuse 时重建。

Tool 层安全规则

工具/函数安全动作依据
validate_local_tool_path()只允许 `/mnt/user-data/*`、只读 skills、只读 ACP workspace 或配置 mount；拒绝 `..` traversal。backend/packages/harness/deerflow/sandbox/tools.py
validate_local_bash_command_paths()本地 bash 模式下扫描 host 绝对路径、`file://`、危险 `cd`、`..` 等，要求使用虚拟路径。backend/packages/harness/deerflow/sandbox/tools.py
SandboxAuditMiddleware高风险命令 block，如 `rm -rf /`、pipe to shell、`/etc/shadow`、动态链接器劫持；中风险 warn，如 `pip install`、`chmod 777`。backend/packages/harness/deerflow/agents/middlewares/sandbox_audit_middleware.py
is_host_bash_allowed()LocalSandboxProvider 下只有显式 allow_host_bash: true 才允许 bash。backend/packages/harness/deerflow/sandbox/security.py
build_sandbox_env()继承 host env 前剔除 key/secret/token/password/DSN 等疑似凭据，再叠加 request-scoped secrets。backend/packages/harness/deerflow/sandbox/env_policy.py
mask_local_paths_in_output()把工具输出中的 host 路径遮回 `/mnt/user-data`、`/mnt/skills`、`/mnt/acp-workspace`。backend/packages/harness/deerflow/sandbox/tools.py
download_file()只允许 `/mnt/user-data` 前缀，拒绝 traversal，并限制最大 100 MB。backend/packages/harness/deerflow/sandbox/local/local_sandbox.py、AIO/E2B adapters
write_file_tool()单次非 append 写入默认 80 KB 上限；写入时使用 per-sandbox file operation lock。backend/packages/harness/deerflow/sandbox/tools.py

Uploads 与 Artifact 链路

阶段关键行为源码
上传Gateway 校验文件名、数量、大小和总大小，写入当前用户/线程的 uploads 目录，返回 virtual_path 和 artifact_url。backend/app/gateway/routers/uploads.py、backend/packages/harness/deerflow/uploads/manager.py

调用示例

下面是一次“写 artifact”的压缩调用形状，展示 L1 需要掌握的边界，不是完整可运行测试。

# 1. ThreadDataMiddleware 已经把三个 host path 写入 runtime.state["thread_data"]。
runtime.state["thread_data"] = {
    "workspace_path": ".../threads/t1/user-data/workspace",
    "uploads_path": ".../threads/t1/user-data/uploads",
    "outputs_path": ".../threads/t1/user-data/outputs",
}

# 2. 首次文件工具调用触发 sandbox lazy init。
sandbox = ensure_sandbox_initialized(runtime)

# 3. agent 写到虚拟 outputs 路径。
write_file_tool.invoke({
    "runtime": runtime,
    "description": "write report",
    "path": "/mnt/user-data/outputs/report.md",
    "content": "# Report\n..."
})

# 4. Gateway 读取 artifact。
# GET /api/threads/t1/artifacts/mnt/user-data/outputs/report.md

推荐阅读顺序

  1. 先读 backend/packages/harness/deerflow/config/paths.py，建立 `/mnt/user-data` 到 host 目录的映射。
  2. 再读 thread_data_middleware.py 与 sandbox/middleware.py，看路径和 sandbox id 如何进入 state。
  3. 接着读 sandbox/tools.py 的 ensure_sandbox_initialized、validate_local_tool_path、bash_tool、write_file_tool。
  4. 随后读 local_sandbox_provider.py 与 local_sandbox.py，确认公共 API 的虚拟路径契约。
  5. 最后按部署模式选择读 aio_sandbox_provider.py 或 e2b_sandbox_provider.py。

调试经验：如果问题是“agent 看不到上传文件”，先查 uploads router 和 provider 的 uses_thread_data_mounts；如果问题是“写出的文件前端打不开”，先查是否写在 `/mnt/user-data/outputs` 以及 artifact URL 是否传了 `mnt/user-data/...`；如果问题是“bash 被拒绝”，先区分 Local host bash 开关、tool path validation 和 SandboxAuditMiddleware 三层。
