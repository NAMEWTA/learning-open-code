15 分钟理解 lead agent 如何被装配成可运行 agent

学习目标：读完后，你能沿着一次 run 的入口说明 deer-flow 如何把模型、工具、prompt、middleware 和 ThreadState 装配成真正可执行的 LangGraph agent。

1. lead agent 站在 Gateway 后面

L0 已经说明 Gateway 是默认运行外壳。lead agent 是 Gateway 背后的 runtime 核心：backend/langgraph.json 把 graph 名 lead_agent 注册到 deerflow.agents:make_lead_agent，同时声明 auth 和 checkpointer。前端默认也以 assistantId: "lead_agent" 发起线程流。

所以读 runtime 时不要从 UI 组件开始猜模型行为，而应先问：这次 run 的 RunnableConfig 里有哪些 context？它们会被 make_lead_agent 解析成什么模型、工具和中间件？

2. 装配主线只有四步

HTTP / SDK run ↓ resolve_agent_factory(...) → make_lead_agent(...) ↓ _make_lead_agent: 解析 config、agent_name、model、skills ↓ get_available_tools → skill policy → assemble_deferred_tools ↓ create_agent(model, tools, middleware, system_prompt, state_schema=ThreadState)

核心源码在 backend/packages/harness/deerflow/agents/lead_agent/agent.py。入口 make_lead_agent(config) 先合并 configurable 与 context，再进入 _make_lead_agent。后者按“请求模型 → agent config → 全局默认”的顺序解析模型；按 agent_name 读取 custom agent 配置；再加载 skills，用 allowed-tools policy 过滤工具，并把 MCP 工具交给 tool_search 延迟暴露。

最终它调用 LangChain 的 create_agent，传入 apply_prompt_template(...) 生成的 system prompt、build_middlewares(...) 生成的中间件链，以及 ThreadState 作为 state schema。

3. 一次真实调用长这样

agent_factory = resolve_agent_factory(body.assistant_id)
config = build_run_config(thread_id, body.config, body.metadata,
                          assistant_id=body.assistant_id)

agent = agent_factory(config=initial_runnable_config)

async for chunk in agent.astream(
    graph_input,
    config=initial_runnable_config,
    stream_mode=lg_modes,
):
    await bridge.publish(run_id, sse_event, serialize(chunk))

这段链路横跨 backend/app/gateway/services.py 与 backend/packages/harness/deerflow/runtime/runs/worker.py。注意 custom agent 没有独立 graph：Gateway 会把非默认 assistant_id 规范化为 agent_name，仍然路由到同一个 lead_agent graph，由 make_lead_agent 在内部加载对应 SOUL、模型、skill 白名单和工具组。

中间件是这条链最容易漏看的部分。基础层负责输入清洗、线程目录、上传、sandbox、工具错误和读写安全；lead-only 层再追加动态日期与 memory、skill 激活、摘要、title、vision、deferred tool 过滤、subagent 限制、loop/token budget 和 clarification。

4. 暂停回忆

参考答案：RunnableConfig、解析后的 chat model、经过 skill policy 与 deferred tool 处理后的 tools、apply_prompt_template 生成的 system prompt、build_middlewares 生成的 middleware 链，以及作为数据边界的 ThreadState。如果你漏掉 ThreadState，说明还只看到“模型加工具”，没有看到 LangGraph graph 的状态边界。
