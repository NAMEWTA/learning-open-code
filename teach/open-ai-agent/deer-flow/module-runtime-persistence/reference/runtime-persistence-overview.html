Runtime 与 persistence 模块参考

本页是 L1 Runtime 与 persistence 的速查表。短课入口：0001-runtime-persistence-module-tour.html；L3 公共 API 字典：runtime-persistence-api.html。

模块职责与整体位置

在 L0 项目总览 的架构图中，本模块位于 Gateway 与 LangGraph agent 之间：它把 HTTP run 请求变成可恢复、可追踪、可回放的运行记录。

职责说明源码锚点
run 生命周期创建、并发策略、状态推进、取消、shutdown drain、重启后 orphaned active run recovery。backend/packages/harness/deerflow/runtime/runs/manager.py
后台执行驱动 LangGraph agent streaming，注入 runtime context，绑定 checkpointer/store，处理 goal continuation、rollback、workspace changes 和最终清理。backend/packages/harness/deerflow/runtime/runs/worker.py
实时输出把 worker 产生的 metadata、values、updates、messages、custom、error、end 事件转成可订阅的桥接流。backend/packages/harness/deerflow/runtime/stream_bridge/base.py
可回看事件把 LangChain callback、tool result、subagent step、workspace change 标准化为按 thread/run 查询的事件。backend/packages/harness/deerflow/runtime/journal.py、backend/packages/harness/deerflow/runtime/events/store/base.py
持久化 schema维护 DeerFlow 自己的 ORM tables、repository、engine lifecycle 和 Alembic bootstrap；LangGraph checkpointer 表不归 Base.metadata 管。backend/packages/harness/deerflow/persistence/
配置边界database 统一驱动 checkpointer 与应用 ORM；run_events 决定事件存储；stream_bridge 决定 SSE bridge。backend/packages/harness/deerflow/config/database_config.py、backend/packages/harness/deerflow/config/run_events_config.py、backend/packages/harness/deerflow/config/stream_bridge_config.py

Run 生命周期边界

阶段runtime 入口写入/读取边界失败时看哪里
Gateway startuplanggraph_runtime()创建 `StreamBridge`、ORM engine、checkpointer、LangGraph store、repository、event store、`RunManager`。backend/app/gateway/deps.py、backend/packages/harness/deerflow/persistence/engine.py
创建 runstart_run() → RunManager.create_or_reject()`runs` 插入 pending；thread_meta 更新为 running；并发策略 reject/interrupt/rollback 生效。backend/app/gateway/services.py、backend/packages/harness/deerflow/runtime/runs/manager.py
开始执行run_agent()`RunStore.update_status(running)`；`StreamBridge.publish(metadata)`；pre-run checkpoint/workspace snapshot。backend/packages/harness/deerflow/runtime/runs/worker.py
流式输出agent.astream()`StreamBridge.publish(values/updates/messages/custom)`；custom subagent events 批量写 `RunEventStore.put_batch()`。backend/packages/harness/deerflow/runtime/stream_bridge/、backend/packages/harness/deerflow/runtime/runs/worker.py
事件记录RunJournalcallback buffer 写 message/trace/middleware/output events；运行中节流更新 token progress。backend/packages/harness/deerflow/runtime/journal.py、backend/packages/harness/deerflow/runtime/events/store/db.py
checkpoint 状态LangGraph checkpointer保存 thread channel values，如 messages、title、goal；cancel rollback 可恢复到 pre-run checkpoint。backend/packages/harness/deerflow/runtime/checkpointer/async_provider.py、backend/packages/harness/deerflow/runtime/goal.py
结束 runrun_agent() finallyflush journal、写 `RunStore.update_run_completion()`、同步 title/thread status、`publish_end()`、延迟 cleanup。backend/packages/harness/deerflow/runtime/runs/worker.py
重启恢复reconcile_orphaned_inflight_runs()SQLite startup 将没有本地 task 的 pending/running run 标记为 error，并给 retained stream 发 end。backend/packages/harness/deerflow/runtime/runs/manager.py、backend/app/gateway/deps.py

配置与后端选择

配置可选值控制对象注意点
database.backendmemory、sqlite、postgresLangGraph checkpointer + LangGraph store + DeerFlow ORM engine。SQLite 模式共享 {sqlite_dir}/deerflow.db 并启用 WAL；Postgres 使用同一 URL 但连接池生命周期独立。
checkpointermemory、sqlite、postgreslegacy checkpointer 配置。存在时优先于 unified database，用于向后兼容。
run_events.backendmemory、db、jsonlRunEventStore。db 需要 ORM session factory；jsonl 适合单节点轻量持久化，不适合多进程 seq 保证。
stream_bridge.typememory、redisSSE bridge。memory 只适合同进程；redis 使用 Redis Streams 支持多 worker replay。

常见故障定位

现象优先边界原因判断
刷新后消息还在，但 join run stream 失败。RunEventStore vs StreamBridge历史消息和实时 replay 是两套存储面；completed run 的 stream 可能已 cleanup。
run 列表显示 pending/running 很久。RunManager + RunStore检查本进程是否仍有 task；重启场景看 orphaned recovery 是否把 persisted inflight 标为 error。
用户看到答案，但 token 用量为 0。RunJournal + RunRepository.update_run_completion()事件流和完成汇总分开；LLM callback token 可能未进入 journal 或最终 flush 失败。
cancel rollback 后状态不符合预期。checkpointer pre-run snapshotrollback 依赖 worker 开始前捕获的 checkpoint；若捕获失败只能标记错误路径。
subtask card reload 后 step 丢失。_SubagentEventBuffer + RunEventStore.put_batch()SSE 实时显示不等于持久化；custom subagent events 要被识别并批量 flush。

源码阅读顺序

  1. 先读 backend/app/gateway/deps.py 的 langgraph_runtime()，确认所有 runtime/persistence singleton 从哪里来。
  2. 再读 backend/app/gateway/services.py 的 start_run()，看 HTTP 请求如何创建 run 并启动 worker。
  3. 接着读 backend/packages/harness/deerflow/runtime/runs/worker.py 的 run_agent()，只跟主线，不先钻 helper。
  4. 然后读 backend/packages/harness/deerflow/runtime/journal.py 与 runtime/events/store/base.py，理解可回看事件。
  5. 最后读 backend/packages/harness/deerflow/persistence/engine.py、bootstrap.py 和各 model，建立数据库边界。
