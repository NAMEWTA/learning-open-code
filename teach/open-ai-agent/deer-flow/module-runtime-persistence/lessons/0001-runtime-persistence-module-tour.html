Runtime 与 persistence：一次 run 会落到哪些边界

目标：15 分钟内建立一张判断图。以后看到“run 没结束、消息没出现、重启后状态异常、token 不准”时，先知道该去 runtime 还是 persistence。

1. 先抓住四个存储面

DeerFlow 的长任务不是单个“调用模型”的函数，而是四个存储面协作。RunManager 管生命周期，StreamBridge 管实时 SSE，RunEventStore 管可回看的消息和 trace，LangGraph checkpointer 管线程状态恢复。SQLAlchemy persistence 不是替代 runtime，而是给 runtime 提供 durable backing。

RunStoreruns 行：状态、模型、错误、token 汇总、首尾消息摘要。
RunEventStorerun_events 行或 JSONL：message、trace、subagent step。
CheckpointerLangGraph 线程状态：messages、goal、title、可回滚 checkpoint。
StreamBridge短期实时缓冲：SSE replay、heartbeat、end marker。

主参考页在 runtime-persistence-overview.html（模块地图）；接口签名与参数约束见 L3 runtime-persistence-api.html。

2. 一次 run 的主线

Gateway /runs/stream │ ├── start_run() ← 校验请求、创建 RunRecord │ └── RunManager.create_or_reject() ← 写 pending 到 RunStore │ ├── asyncio.create_task(run_agent()) ← 后台执行 LangGraph agent │ ├── worker.run_agent() │ ├── set_status(running) ← RunStore 状态推进 │ ├── bridge.publish(metadata/values) ← SSE 实时输出 │ ├── RunJournal callbacks ← message/trace/token 写入事件缓冲 │ ├── checkpointer ← 保存线程状态，可 rollback/resume │ └── set_status(success/error/...) ← 最终状态和完成数据 │ └── sse_consumer() ← 订阅 StreamBridge，发给前端

真实装配点在 backend/app/gateway/deps.py：lifespan 先创建 make_stream_bridge()，再初始化 SQLAlchemy engine，然后创建 make_checkpointer()、make_store()、RunRepository、make_run_event_store()，最后把 RunManager(store=run_store) 放进 app.state。

3. 调用示例：HTTP 入口怎样接入 worker

backend/app/gateway/services.py 的 start_run() 是最短调用链。下面是压缩后的形状，保留了关键边界：

record = await run_mgr.create_or_reject(
    thread_id,
    body.assistant_id,
    on_disconnect=disconnect,
    metadata=body.metadata or {},
    kwargs={"input": body.input, "config": redact_config_secrets(body.config)},
    multitask_strategy=body.multitask_strategy,
    model_name=model_name,
    user_id=owner_user_id,
)

task = asyncio.create_task(
    run_agent(
        bridge, run_mgr, record,
        ctx=run_ctx,
        agent_factory=agent_factory,
        graph_input=graph_input,
        config=config,
        stream_modes=stream_modes,
    )
)
record.task = task

读这段时要分清两件事：record 是 run 生命周期的可见句柄；run_ctx 是基础设施包，里面带着 checkpointer、LangGraph store、event store、thread store 和 app config。worker 不直接 new 数据库对象，它只消费这个 context。

4. 检索练习

暂停 60 秒，先不要看答案：如果前端刷新后能看到旧消息，但不能继续 join 某个已完成 run 的 SSE，你会优先查哪两个边界？如果用户说“重启后 run 一直显示 running”，你又会查哪一个恢复逻辑？

参考答案：旧消息来自 RunEventStore.list_messages()，而 join SSE 依赖 StreamBridge 是否还保留该 run 的 stream；两者不是一个存储面。重启后 active run 卡住，要查 RunManager.reconcile_orphaned_inflight_runs() 以及 runs 表中的 pending/running 行。
