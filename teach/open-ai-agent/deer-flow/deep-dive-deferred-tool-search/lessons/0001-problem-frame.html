15 分钟理解 MCP 延迟工具发现的设计权衡

学习目标：当 tool_search.enabled 打开、MCP 工具数量暴涨时，你能解释 DeerFlow 为何只把工具名放进 prompt、如何用 tool_search promote schema，以及 fail-closed 与 catalog hash 各自防什么。 查询语法与 reducer 细节见 deferred-tool-search-notes 参考页。

L4 深度剖析 配置：tool_search.enabled 关联：module-tools-mcp

1. 问题：MCP 工具爆炸

每个 MCP 工具携带完整 OpenAI function schema（参数类型、描述、嵌套字段）。十几个 server × 数十工具，会在每一轮 bind_tools 时重复占用 prompt token，且多数工具当轮根本不会被调用。 DeerFlow 的应对是延迟发现：仅对带 deerflow_mcp 元数据的工具 defer；内置工具与已通过 policy 的非 MCP 工具仍直接绑定。

config.yaml → tool_search.enabled ↓ get_available_tools() + skill policy 过滤 ↓ assemble_deferred_tools() ← fail-closed 闸门 ├── 关闭：MCP schema 直接 bind └── 开启：prompt 只列 <available-deferred-tools> 名称 + 追加 tool_search 工具

2. 设计权衡：全量 schema vs 两阶段发现

全量绑定优点是可发现性最高——模型一眼看见所有参数；代价是 token 线性膨胀、前缀缓存友好度差。 延迟发现把成本移到「真正要用时」：模型先见名字，调用 tool_search 才拿到 schema 并写入 state["promoted"]； DeferredToolFilterMiddleware 在下一轮 wrap_model_call 才把这些 schema 放进 bind 列表。 ToolNode 仍持有全部工具以便执行——只是 LLM 在 promote 前「看不见、调不通」未暴露的 MCP 工具。 

sequenceDiagram
    participant M as LLM
    participant MW as DeferredToolFilterMiddleware
    participant TS as tool_search
    participant ST as ThreadState.promoted

    M->>MW: turn 1 bind（隐藏 MCP schema）
    M->>TS: tool_search("select:foo")
    TS->>ST: Command update catalog_hash + names
    M->>MW: turn 2 bind（foo 已 promote）
    MW->>M: 仅暴露 foo + 仍隐藏的其他 deferred

3. fail-closed：宁可报错也不静默泄漏

最坏情况不是「搜不到工具」，而是 MCP 本应 defer 却因装配 bug 把完整 schema 直接 bind 给模型。 assemble_deferred_tools 在 enabled 且 policy 过滤后仍有 MCP 工具、却得不到 deferred set 时抛 RuntimeError，拒绝继续装配。 这与「policy 已剔除全部 MCP」不同——后者合法地返回空 setup，不触发 fail-closed。 未 promote 就硬调 deferred 工具时，middleware 返回 error ToolMessage，提示先 tool_search。 

if enabled and not deferred_setup.deferred_names
   and any(is_mcp_tool(t) for t in filtered_tools):
    raise RuntimeError("... refusing to bind MCP schemas (fail-closed)")

4. catalog hash：让 promotion 跟得上 catalog 漂移

deferred 集合在 agent build 时固化为不可变 DeferredToolCatalog；其 hash 来自排序后的 name+schema 摘要（SHA-256 前 16 位）。 每次 tool_search 成功都把 {catalog_hash, names} 写入图状态； merge_promoted 在 hash 变化时整包替换旧 names，避免 checkpoint 里残留的 bare name 在工具改名/换 schema 后误暴露错误工具。 Middleware 只认「当前 build 的 hash === state 里的 hash」的 promotion。

判断题：thread checkpoint 里已有 promoted.names=["old_mcp"]，但本轮 agent 重建后 catalog hash 因 MCP schema 更新而改变。下一轮模型能否直接调用 old_mcp？

A. 能，names 仍在 state 里   B. 不能，hash 不匹配视为未 promote   C. 能，但仅限 ToolNode 执行   D. 取决于 tool_search 是否关闭

参考答案：B。_promoted() 要求 catalog_hash 一致；hash 漂移后旧 promotion 失效，模型须重新 tool_search。C 错在 middleware 会在 wrap_tool_call 拦截未 promote 的调用。

主要来源：tool_search.py、deferred_tool_filter_middleware.py、thread_state.py::merge_promoted。 想继续下钻 query 语法（select: / +prefix）或 subagent 隔离，向 agent 追问即可。
