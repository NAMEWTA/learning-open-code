15 分钟划清配置热加载边界

学习目标：改完 config.yaml 后，你能判断「下一条消息即生效」还是「必须重启 Gateway」。 完整注册表与字段对照见 热加载边界参考页。

L2 垂直切片 关联：config-deploy · gateway

1. 两条路径：请求时刷新 vs 启动快照

DeerFlow 故意把配置分成两层。请求路径：Gateway 依赖注入里的 deps.get_config() 每次调用 get_app_config()，按文件内容签名决定是否从磁盘重载——因此 models、memory、summarization、tools 等「每轮 agent 构图时读取」的字段，保存后下一条消息即可生效。 启动路径：gateway/app.py 的 lifespan() 把 startup_config = get_app_config() 存为局部变量，只用于一次性引导： configure_logging、langgraph_runtime、IM channels、scheduler。 刻意不把 AppConfig 挂在 app.state，避免 worker 与 HTTP 层读到不同年代的配置（issue #3107）。

2. 机器可读边界：reload_boundary.py

config/reload_boundary.py 的 STARTUP_ONLY_FIELDS 是单一事实来源。 每个需重启的顶层段（database、checkpointer、run_events、stream_bridge、sandbox、log_level、logging、channels、channel_connections、scheduler）都写明「哪段启动代码捕获了快照」。 对应 AppConfig 字段的 Pydantic Field(description=...) 以 startup-only: 开头，IDE hover 与注册表由 test_reload_boundary.py 双向锁死。 不在表里的顶层段默认可热加载——例如改 memory.debounce_seconds 或 subagents.timeout_seconds 无需重启。 

get_app_config()  # mtime + SHA256 签名
  ├── 文件未变 → 返回缓存 AppConfig
  └── 文件已变 → AppConfig.from_file() 重载

lifespan(startup_config)  # 仅启动一次
  ├── langgraph_runtime → engine / checkpointer / store
  ├── start_channel_service → IM 客户端
  └── ScheduledTaskService → 轮询参数

3. 时序：编辑 config 到 agent 读到新值

sequenceDiagram
    participant Op as 运维编辑 config.yaml
    participant Disk as 配置文件
    participant GW as Gateway 请求
    participant CFG as get_app_config()
    participant RT as langgraph_runtime 引擎
    participant AG as make_lead_agent

    Op->>Disk: 保存 models[].max_tokens
    GW->>CFG: get_config() 下一请求
    CFG->>Disk: 签名变化，重载
    CFG-->>AG: 新 AppConfig（热加载字段）
    Note over RT: database/sandbox 等仍绑 startup 快照

    Op->>Disk: 改 sandbox.use
    GW->>CFG: 重载 AppConfig
    CFG-->>AG: 新 sandbox 配置对象
  Note over RT: _default_sandbox_provider 仍是旧 provider → 需重启

4. 边界案例：改了却「像没生效」

典型误报：把 sandbox.use 从 local 改成 aio，或切换 database.backend 为 postgres，保存后立刻发消息—— get_app_config() 确实返回新对象，但 get_sandbox_provider() / SQLAlchemy engine 在启动时已创建，注册表要求重启。 另一类：logging.enhance.enabled 控制 TraceMiddleware 是否在构造时捕获；运行时改 YAML 不会让响应头 X-Trace-Id 与日志格式同步，也需重启。 排障口诀：先查字段是否在 STARTUP_ONLY_FIELDS；不在则确认是否走了 get_config() 而非误读 startup_config 局部变量。

检索练习：判断对错（不看上文）：A. AppConfig 缓存在 app.state 上供请求复用。B. 改 models[0].name 后下一条 run 可读新配置。C. 改 channels.feishu.app_id 保存即生效，无需重启 Gateway。
参考答案：A 错，请求走 get_app_config() 模块级缓存，启动快照仅是 lifespan 局部变量。B 对，models 不在 STARTUP_ONLY_FIELDS。C 错，channels 在注册表中，IM 客户端启动时绑定。
