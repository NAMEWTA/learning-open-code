15 分钟串起一次聊天流式执行

学习目标：用户在聊天框点发送后，你能从前端页面一直指到 lead agent，并说出 SSE 在哪一层产生、UI 在哪一层消费。 完整接口表与异常分支见 流式执行参考页。

L2 垂直切片 关联：workspace · gateway · runtime · lead-agent

1. 入口：工作区页面只接线，不跑 agent

分叉说明：消息「重新生成」走 regenerate/prepare 与 regenerateMessage，不经过本节 sendMessage 主路径；本课只讲首次提交与流式回复。

路由 frontend/src/app/workspace/chats/[thread_id]/page.tsx 是用户可见入口。 它把输入框的 onSubmit 接到 useThreadStream().sendMessage，用 thread.isLoading 驱动「流式中 / 可停止」状态，本身不调用 HTTP。 新会话时 onStart 用 history.replaceState 换成真实 thread_id，避免 Next.js 路由重挂载丢状态（见页面注释）。 

InputBox.onSubmit
  -> handleSubmit -> sendMessage(threadId, message)
  -> thread.submit({ messages: [...] }, { streamSubgraphs, streamResumable, context })

真正的流式逻辑在 core 数据层 的 frontend/src/core/threads/hooks.ts。

2. 前端 → Gateway：SDK 与 SSE

frontend/src/core/api/api-client.ts 的 getAPIClient() 创建 LangGraph SDK， 基址为 /api/langgraph（nginx 改写到 Gateway /api/*）。 useStream 在首次提交时 POST /api/threads/{thread_id}/runs/stream，读 Content-Location 得到 run_id，再消费 text/event-stream。 SDK 包装层处理 CSRF、终端 run 跳过重连、以及 store-only run 的 409。 

useStream (hooks.ts)
  -> getAPIClient().runs.stream
  -> POST /api/threads/{id}/runs/stream
  -> SSE: metadata | values | messages-tuple | custom | end

3. Gateway → Agent：创建 run 与后台 worker

backend/app/gateway/routers/thread_runs.py 的 stream_run 调用 app.gateway.services.start_run，再返回 StreamingResponse(sse_consumer(...))。 start_run 经 RunManager.create_or_reject 登记 run，asyncio.create_task(run_agent(...)) 启动后台任务。 backend/packages/harness/deerflow/runtime/runs/worker.py 里 run_agent 先 bridge.publish(metadata)，再 agent_factory 构图并 agent.astream(...)，把 chunk 写入 StreamBridge。 默认工厂是 backend/packages/harness/deerflow/agents/lead_agent/agent.py 的 make_lead_agent。 

sequenceDiagram
    participant UI as ChatPage
    participant Hook as useThreadStream
    participant SDK as api-client
    participant RT as thread_runs.stream_run
    participant SVC as start_run
    participant W as worker.run_agent
    participant A as make_lead_agent

    UI->>Hook: sendMessage / submit
    Hook->>SDK: runs.stream(threadId, lead_agent, input)
    SDK->>RT: POST .../runs/stream
    RT->>SVC: start_run(body, thread_id)
    SVC->>W: create_task(run_agent)
    RT-->>SDK: SSE (sse_consumer)
    W->>A: agent_factory(config)
    W->>W: agent.astream → bridge.publish
    SDK-->>Hook: metadata / values / messages
    Hook-->>UI: thread.messages 更新

4. 异常路径：线程已有活跃 run

默认 multitask_strategy="reject"：同线程已有 pending/running run 时， RunManager.create_or_reject 抛 ConflictError，Gateway 转 HTTP 409。 前端表现为提交失败，不会静默覆盖旧流。 另一条常见边界：SSE 客户端断开且 on_disconnect=cancel 时， sse_consumer 的 finally 会 run_mgr.cancel，run 状态变为 interrupted。 更多 409 文案（不可取消、无法 join 非本 worker 流）见 参考页错误表。

检索练习

判断对错（不看上文）：

A. page.tsx 直接调用 fetch 打开 SSE。

B. stream_run 在返回 SSE 之前已经 create_task(run_agent)。

C. make_lead_agent 在 Gateway 进程内、与 FastAPI 同进程执行。

参考答案：A 错，页面只调 sendMessage，SSE 由 LangGraph SDK + useStream 处理。B 对，start_run 在 StreamingResponse 之前启动后台任务。C 对，embedded runtime 无独立 LangGraph Server 进程。
