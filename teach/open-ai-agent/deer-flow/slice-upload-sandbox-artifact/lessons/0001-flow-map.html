15 分钟串起上传 → sandbox → artifact

学习目标：用户在聊天框附加文件后，你能从前端上传请求一直指到 sandbox 里的 /mnt/user-data/uploads/...，并说出 UI 预览同一份文件时走的 artifacts URL。 完整接口表与 chmod / 同步分支见 链路参考页。

L2 垂直切片 关联：frontend-core · workspace · gateway · sandbox

1. 前端：先上传，再 submit run

用户选文件后点发送， frontend/src/core/threads/hooks.ts 的 sendMessage 不会把原始 File 直接塞进 LangGraph input，而是先调 frontend/src/core/uploads/api.ts 的 uploadFiles(threadId, files)，即 POST /api/threads/{thread_id}/uploads（multipart files）。 返回的 virtual_path 写入乐观消息的 additional_kwargs.files，再随用户消息一起 thread.submit。 

InputBox 选文件 → sendMessage
  → uploadFiles (FormData)
  → POST .../uploads
  → 响应 files[].virtual_path + artifact_url
  → submit({ messages, additional_kwargs.files })

2. Gateway：落盘、chmod、按需同步 sandbox

backend/app/gateway/routers/uploads.py 的 upload_files 委托 backend/packages/harness/deerflow/uploads/manager.py 做路径校验与分块写入：先写到 .upload-*.part 暂存文件，校验大小后 os.replace 提交。文件落在 users/{user_id}/threads/{thread_id}/user-data/uploads/。 随后对 written paths 加 group/other 读权限，避免 Docker sandbox 内非 root 进程读不到。

若 sandbox provider 没有 bind-mount 线程目录（uses_thread_data_mounts=false），Gateway 会 acquire_async sandbox 并对每个文件调用 sandbox.update_file(virtual_path, bytes)；本地 sandbox 则主要靠挂载，跳过远程同步。 响应里 upload_artifact_url 生成预览地址。 

upload_files
  → ensure_uploads_dir + staged write
  → chmod 可读
  → [可选] sandbox.update_file(/mnt/user-data/uploads/…)
  → UploadResponse.files[].artifact_url

3. Artifact：虚拟路径回到浏览器

artifact_url 形如 /api/threads/{id}/artifacts/mnt/user-data/uploads/{filename}。 backend/app/gateway/routers/artifacts.py 的 get_artifact 用 resolve_thread_virtual_path 把 mnt/user-data/uploads/... 映射回 host 文件，再按 MIME 内联或下载返回。 前端 frontend/src/core/artifacts/utils.ts 的 urlOfArtifact 拼同样前缀； frontend/src/core/artifacts/loader.ts 拉取文本供 workspace 侧栏预览。 HTML/SVG 等 active content 一律强制 attachment，防 XSS。 

sequenceDiagram
    participant UI as ChatPage
    participant Up as uploadFiles
    participant RT as uploads.upload_files
    participant M as uploads/manager
    participant SB as Sandbox
    participant AR as artifacts.get_artifact

    UI->>Up: POST multipart files
    Up->>RT: /api/threads/{id}/uploads
    RT->>M: staged write + enrich listing
    RT->>SB: update_file (非 mount 模式)
    RT-->>Up: virtual_path + artifact_url
    Up-->>UI: submit 带 files 元数据
    UI->>AR: GET artifact_url 预览
    AR-->>UI: 文件内容

4. 异常路径：超限与不安全文件名

默认限制见 uploads.max_files / max_file_size / max_total_size（10 文件、50 MiB、100 MiB）； 超出返回 413，Gateway 会清理本请求已写入的路径。 含路径遍历或非常规文件名的条目进入 skipped_files 而非静默覆盖。 删除走 DELETE .../uploads/{filename}，同样经 manager.delete_file_safe 防穿越。 更多状态码见 参考页错误表。

检索练习

判断对错（不看上文）：

A. sendMessage 把 File 对象直接 POST 到 /runs/stream。

B. Agent 在 sandbox 里读上传文件应使用 /mnt/user-data/uploads/foo.pdf。

C. 所有部署模式下 Gateway 都会对每个上传调用 sandbox.update_file。

参考答案：A 错，文件先走独立 uploads 路由。B 对，upload_virtual_path 即此前缀。C 错，bind-mount 模式（uses_thread_data_mounts=true）跳过远程同步。
