15 分钟理解 DeerFlow 前端 workspace

本课只教一个目标：看到 /workspace 相关代码时，能判断“这是路由壳、页面体验、组件状态，还是接后端的 core 层”。

目标读者：React/Next.js 基础 主线：页面组织到 Gateway 参考：完整清单

1. 先把 workspace 看成四层

DeerFlow 的 workspace 是用户实际工作的区域，不是 landing page；它对应 L0 架构图里的 Next.js frontend/workspace 层。第一层是 Next.js App Router： frontend/src/app/workspace/layout.tsx 先在服务端读取用户状态，决定进入工作台、跳登录/setup，或显示 Gateway 离线兜底。 第二层是 frontend/src/app/workspace/workspace-content.tsx，它挂上 React Query、侧栏、离线横幅、命令面板和 toast。

frontend/src/app/workspace/ ├── layout.tsx ← 认证与 Gateway 离线分流 ├── workspace-content.tsx ← 工作台全局壳 ├── page.tsx ← /workspace 重定向 ├── chats/ ← 聊天列表与线程页 ├── agents/ ← Agent 管理与 agent 专属聊天页 └── scheduled-tasks/ ← 定时任务

第三层是 frontend/src/components/workspace/：侧栏、消息列表、输入框、artifact、sidecar、workspace changes 都在这里。第四层是 core：它把 UI 事件转成 LangGraph SDK 或 Gateway REST 请求。

2. 聊天页是主线样本

线程页入口是 frontend/src/app/workspace/chats/[thread_id]/page.tsx。它不直接写底层请求，而是把体验拼起来： useThreadChat() 处理 URL 里的 thread_id 和新线程临时 ID；useThreadStream() 提供 thread、发送、停止、历史加载；MessageList 显示消息；InputBox 负责提交。 

handleSubmit(message)
  -> sendMessage(threadId, message)
  -> thread.submit({ messages }, { threadId, streamResumable: true })
  -> Gateway /api/langgraph
  -> MessageList 重新渲染 thread.messages

这里的关键判断是：聊天页面只组合“用户体验”，真正的连接点在 frontend/src/core/threads/hooks.ts 和 frontend/src/core/api/api-client.ts。 api-client.ts 创建 LangGraph Client，给状态变更请求补 CSRF，并处理 stream、cancel、joinStream 的兼容逻辑。

3. 其他页面复用同一套边界

Agents 页面从 frontend/src/app/workspace/agents/layout.tsx 先查 /api/features，未开启就显示禁用页；创建 Agent 时复用聊天流发 bootstrap 消息。Agent 专属聊天页位于 frontend/src/app/workspace/agents/[agent_name]/chats/[thread_id]/page.tsx，它复用普通聊天流，但把 URL 中的 agent_name 注入上下文。scheduled tasks 页面在 frontend/src/app/workspace/scheduled-tasks/page.tsx，实际 CRUD API 位于 frontend/src/core/scheduled-tasks/api.ts。注意：frontend/src/core/tasks/ 管的是聊天中的 subtask 事件和卡片状态，不是 scheduled tasks CRUD。

Artifact 和 sidecar 也遵守同一原则：frontend/src/components/workspace/chats/chat-box.tsx 决定右侧面板展示哪个体验；artifact URL、sidecar thread 创建、workspace changes diff 这些后端连接分别留在 core 或专门 API 文件中。读代码时先问“这个文件在四层里的哪一层”，通常就能避免迷路。

练习：30 秒定位

假设你要排查“AI 回复下方的文件变更 badge 不出现”。先别搜全仓库，按本课四层判断：你会先看哪个组件文件？再看哪个 core 文件？

参考答案：先看 frontend/src/components/workspace/changes/workspace-change-badge.tsx， 因为 badge 是消息 UI 的子组件；再看 frontend/src/core/workspace-changes/hooks.ts 和 frontend/src/core/workspace-changes/api.ts，确认 runId、enabled 条件、请求路径和返回的 available/summary。 如果还不清楚，继续追 frontend/src/components/workspace/messages/message-list-item.tsx 里传给 badge 的 threadId 与 runId。

主要来源：本地源码与 Next.js App Router 官方文档。若你想继续，可以让 agent 追问一个 L2 方向：聊天流、scheduled tasks、sidecar、artifacts 或 workspace changes。
