15 分钟摸清 subagent 委派链路

本课只教一个目标：当 lead agent 调用 task 工具把子任务委派出去时，你能从 workspace 里的 Subtask 卡片一路追到后台线程、SSE 事件与结构化状态契约，并说出数据在哪一层落地。

L2 垂直切片 入口：task 工具调用 参考：完整链路图

1. 触发点：lead agent 选中 task 工具

用户在聊天里发出复杂请求后，lead agent（subagent_enabled=true 时）可绑定 task_tool.py 定义的 task 工具。模型一次调用携带 description、prompt、subagent_type（如 general-purpose 或 bash）。前端 message-list.tsx 扫描 AIMessage 的 tool_calls，为每个 task 预注册一条 Subtask，状态先标为 in_progress。 

用户消息 → lead agent 推理
  → tool_call: task(description, prompt, subagent_type)
  → message-list 创建 Subtask(id=tool_call_id)
  → SubtaskCard 渲染「执行中」

2. 后端主路径：配置 → 执行器 → 轮询 → SSE

task_tool 先解析 subagent 配置（未知类型直接返回 status=failed 的 ToolMessage，不启动后台任务）。通过后创建 SubagentExecutor，用 tool_call_id 作为 task_id 调用 execute_async 丢进线程池。工具本体不阻塞事件循环，而是每 5 秒轮询 get_background_task_result，并通过 get_stream_writer() 推送 task_started / task_running / 终态事件。终态时 _task_result_command 写入带 subagent_status 元数据的 ToolMessage。 

task_tool
  → get_subagent_config + SubagentExecutor.execute_async
  → 后台线程 agent.astream（executor.py）
  → 轮询 SubagentResult.status
  → writer: task_running（新步骤消息，含 AI 与 ToolMessage）
  → 终态: task_completed / task_failed / …
  → ToolMessage(additional_kwargs.subagent_status)

同一 run 的 worker.py 里 _SubagentEventBuffer 把 custom 事件批量写成 subagent.start / subagent.step / subagent.end，供刷新后按 task_id 回填步骤。详见 参考 · 持久化。

3. 前端收尾：SSE 步骤 + 结构化结果解析

hooks.ts 的 onCustomEvent 监听 task_running，把每条子 agent 消息追加到 Subtask.steps（不覆盖）。 当对应 ToolMessage 到达，parseSubtaskResult（ subtask-result.ts）只读 additional_kwargs.subagent_status，映射为卡片三态 completed / failed / in_progress。用户展开历史卡片时， SubtaskCard 可调用 fetchSubtaskSteps 从 events API 补全步骤。

练习：未知 subagent 类型算哪一层失败？

Network 里看到 task 的 ToolMessage 正文含 Unknown subagent type 'researcher'，但 Subtask 卡片立刻变红。这是前端 bug 还是预期行为？

预期行为。task_tool 在配置校验阶段就返回 _task_result_command(status="failed")，从未启动后台线程；前端收到 ToolMessage 后按 subagent_status=failed 标记终态。应检查 config.yaml 的 subagents.custom_agents 或内置类型名，而不是查 executor 线程池。
