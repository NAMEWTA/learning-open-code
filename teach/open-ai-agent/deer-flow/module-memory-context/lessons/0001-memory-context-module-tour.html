15 分钟理解 Memory、Summarization 与 Durable Context

学习目标：读完后，你能判断一段对话信息会写入跨线程 memory.json、被压成 summary_text，还是通过 durable context 继续出现在模型请求里。前置：L0 项目地图、lead agent middleware 链。

1. 三条链路，三种时间尺度

deer-flow 用三套机制管理「上下文」，不要混为一谈：

  • Memory（长期、跨线程）：异步写入 users/{user_id}/memory.json，下一轮通过 DynamicContextMiddleware 以 <memory> 注入。入口是 agents/memory/ 与 MemoryMiddleware。
  • Summarization（单线程、压缩）：当 token 或消息数触阈值，DeerFlowSummarizationMiddleware 删掉旧 messages，把 prose 摘要存进 ThreadState.summary_text。
  • Durable context（单线程、防丢）：DurableContextMiddleware 在 summarization 之前把 delegation ledger、skill 引用和 summary_text 投影成隐藏的 <durable_context_data>，避免 tool 细节被压没。
用户消息 → messages（checkpoint） ↓ before_model DurableContext 捕获 delegations / skills，并注入 summary + ledger ↓ before_model（若 summarization.enabled） Summarization 触发 → 旧 messages 移除 → summary_text 更新 ↓ memory_flush_hook（压缩前） 即将被删的消息先入 memory 队列 ↓ after_agent MemoryMiddleware 把本轮 user+AI 最终回复入队 → debounce → LLM 更新 memory.json ↓ 下一轮 before_model DynamicContext 读取 memory.json → <memory> 注入

2. middleware 在 lead 链上的位置

在 agents/lead_agent/agent.py 的 build_middlewares 里，顺序很关键：DurableContextMiddleware 排在 SummarizationMiddleware 之前，这样 summarization 压消息前，delegation 与 skill 已写入 checkpoint；MemoryMiddleware 在 TitleMiddleware 之后、靠链尾 after_agent 入队，不阻塞当前 turn 的模型调用。

summarization 与 memory 还有第二条汇合点：_create_summarization_middleware 在 memory.enabled 时把 memory_flush_hook 挂到 before_summarization，确保即将被摘要掉的历史不会漏进长期记忆队列。

3. 两个入队入口，共用 debounce 队列

长期记忆只关心「用户说了什么 + agent 最终回了什么」，tool call 中间态会被 filter_messages_for_memory 滤掉。正常 turn 结束时 MemoryMiddleware.after_agent 入队；长对话被压缩时 memory_flush_hook 用 add_nowait 立即 flush。两者都写入 agents/memory/queue.py，默认 debounce 30 秒、按 (thread_id, user_id, agent_name) 去重，并在入队时捕获 user_id（Timer 线程拿不到 ContextVar）。

# memory_middleware.py — after_agent 入队（节选）
filtered_messages = filter_messages_for_memory(messages)
queue.add(
    thread_id=thread_id,
    messages=filtered_messages,
    agent_name=self._agent_name,
    user_id=user_id,
    correction_detected=detect_correction(filtered_messages),
)

后台线程调用 MemoryUpdater，用独立 LLM 抽取 facts 与 context 段落，原子写入 memory.json。这与 summarization 用的摘要模型、与主对话模型可以完全不同——分别由 memory.model_name 和 summarization.model_name 控制。

4. 暂停回忆

判断题：某线程里 subagent 刚完成一次 task 委托，随后 summarization 把 tool messages 从 checkpoint 删掉。问：lead agent 下一轮还能知道这次委托的结果吗？长期 memory 会记住 tool 输出吗？
参考答案：委托结果可以继续可见——DurableContextMiddleware 在压缩前把 terminal delegation 写入 ThreadState.delegations，并在每次 model call 注入 ledger；summary_text 也会经 durable 块投影。长期 memory通常不会直接记住 tool 原文——入队前已过滤，只剩 user 输入与最终 AI 文本；若对话被 summarization 截断，memory_flush_hook 会把被删段落先入队再压缩。
