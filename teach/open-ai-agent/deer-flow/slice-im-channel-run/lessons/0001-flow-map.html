15 分钟串起 IM 入站如何触发 run

学习目标：用户在飞书或 Slack 发一条消息后，你能从平台 adapter 一直指到 Gateway 上的一次 agent run，并说出 run 在哪一层被创建。 完整分层表、三种 run 模式与异常分支见 入站执行参考页。

L2 垂直切片 关联：channels · gateway · chat-streaming-run

1. 入口：平台 adapter 只入队，不跑 agent

各 IM 平台在 backend/app/channels/*.py 里收事件或长轮询。 以飞书为例，adapter 把用户文本包装成 InboundMessage，调用 message_bus.py 的 publish_inbound 入队。 此时还没有 thread、也没有 run——adapter 与 agent 完全解耦。 

FeishuChannel（示意）
  -> InboundMessage(channel_name, chat_id, user_id, text, …)
  -> bus.publish_inbound(msg)   # asyncio.Queue.put

service.py 在 Gateway 启动时创建 MessageBus 与 ChannelManager， 并启动各平台 worker。运维侧 GET /api/channels/ 只查状态，不参与消息流（见 gateway/routers/channels.py）。

2. 调度循环：出队、去重、并发处理

manager.py 的 _dispatch_loop 阻塞读取总线： msg = await self.bus.get_inbound()。 先用 _is_duplicate_inbound 吸收平台重投；再 asyncio.create_task(self._handle_message(msg))， 由信号量 _max_concurrency 限制并行度。 普通聊天在 _handle_message 里还会先做绑定身份校验（require_bound_identity）。 

_dispatch_loop
  -> get_inbound()
  -> dedupe?
  -> create_task(_handle_message)
       -> bound identity OK?
       -> _handle_chat 或 _handle_command

3. 触发 run：thread 映射 + SDK 调 Gateway

_handle_chat 是聊天入站的主路径： ① _get_or_create_thread 经 langgraph_sdk 调 client.threads.create（或复用 store 里的映射）； ② _resolve_run_params 合并 session、owner_user_id、channel 策略； ③ 按平台能力选 run API——Feishu/Telegram 走 runs.stream，Slack/Discord 走 runs.wait，GitHub 走 fire-and-forget 的 runs.create。 与浏览器聊天相同，真正执行 lead agent 的是 Gateway 内嵌 runtime，不是 channel 进程里的 Python 图。 

sequenceDiagram
    participant IM as Feishu/Slack adapter
    participant Bus as MessageBus
    participant M as ChannelManager
    participant SDK as langgraph_sdk
    participant GW as Gateway /api

    IM->>Bus: publish_inbound(InboundMessage)
    Bus->>M: get_inbound()
    M->>M: _handle_chat
    M->>SDK: threads.create / get
    SDK->>GW: POST /api/threads
    M->>SDK: runs.wait 或 runs.stream
    SDK->>GW: POST .../runs/wait 或 /stream
    GW-->>M: 最终状态或 SSE 块
    M->>Bus: publish_outbound(OutboundMessage)
    Bus-->>IM: channel 回调发回平台

4. 异常路径：未绑定与线程忙

启用 channel_connections.require_bound_identity 时，缺少 connection_id + owner_user_id 的入站聊天在进信号量前就被 _reject_unbound_channel_message 拒绝，并直接 publish_outbound 提示用户去设置页绑定。 另一条常见边界：multitask_strategy="reject" 下同 thread 已有活跃 run 时，SDK 抛 ConflictError， manager 回复 THREAD_BUSY_MESSAGE 而不重试。处理失败时还会 _release_inbound_dedupe_key，避免重投被 TTL 去重吞掉。 更多错误码与 provider 差异见 参考页错误表。

检索练习

判断对错（不看上文）：

A. feishu.py 里直接 import make_lead_agent 并跑图。

B. ChannelManager 与浏览器前端共用 Gateway 的 thread/run HTTP API。

C. POST /api/channels/feishu/restart 是用户发消息时必经的 HTTP 入口。

参考答案：A 错，adapter 只 publish_inbound，agent 由 manager 经 SDK 在 Gateway 触发。B 对，同一 langgraph_url（默认 http://localhost:8001/api）。C 错，那是管理员重启 worker 的运维接口，与入站消息流无关。
