问题框架：Memory 双入队为何共存

学习目标：解释 DeerFlow 为何对同一 MemoryUpdateQueue 使用 debounce 入队与零延迟入队两条路径，以及 summarization 场景下不能等防抖的原因。 API 对照与测试锚点见 memory-dual-queue-notes 参考页； L2 总览见 memory 注入切片。

L4 深度剖析 源码：queue.py 关联：module-memory-context

1. 问题：记忆更新既要省 LLM，又不能丢历史

长期 memory 由后台 LLM 从对话里抽取事实，每调一次都花钱、也占速率配额。 若每轮 agent 结束立刻写盘，短对话里连续多轮会产生大量重复抽取。 另一方面，SummarizationMiddleware 会把旧消息压成摘要并从 messages 里移除—— 若这些轮次还没进 memory 队列，事实就永远丢失。 DeerFlow 的解法是同一条队列、两种调度：常态走 debounce 合并，摘要前走即时 flush。

每轮 agent 结束 → MemoryMiddleware.after_agent → queue.add() ← debounce_seconds（默认 30s） 上下文逼近上限、即将压缩 → SummarizationMiddleware._fire_hooks → memory_flush_hook → queue.add_nowait() ← _schedule_timer(0)

2. debounce 路径：批量、降本

memory_middleware.py 在 after_agent 过滤出「用户句 + 最终 AI 回复」（去掉 tool call 中间态）， 并在仍存活的请求上下文里捕获 user_id 与 deerflow_trace_id—— 因为后续 threading.Timer 线程不会继承 ContextVar。 然后调用 queue.add()：入队后重置定时器为 config.debounce_seconds。 同一 (thread_id, user_id, agent_name) 在窗口内多次入队会被 _enqueue_locked 合并为最新一份消息快照。 

queue.add(...)          # after_agent
  → _enqueue_locked(...)
  → _reset_timer()      # delay = debounce_seconds

3. 即时路径：摘要前的「最后抢救」

summarization_hook.py 注册为 SummarizationMiddleware 的 before_summarization hook。 在消息被压摘要之前，对 event.messages_to_summarize 做同样的 memory 过滤； 若至少有一对 human/ai，则 queue.add_nowait()，内部把定时器设为 0，尽快触发 _process_queue。 设计意图明确：压缩一旦完成，原始逐句对话不再可供 MemoryUpdater 读取，不能再等 30 秒防抖。 

sequenceDiagram
    participant MW as MemoryMiddleware
    participant Q as MemoryUpdateQueue
    participant SM as SummarizationMiddleware
    participant H as memory_flush_hook

    MW->>Q: add() debounce 30s
    Note over Q: 多轮合并同一 thread key
    SM->>H: before_summarization
    H->>Q: add_nowait() timer=0
    Q->>Q: _process_queue → MemoryUpdater

4. 边界：何时重叠、何时跳过

两条路径共享 _enqueue_locked，不是两个队列。 memory.enabled=false、缺少 thread_id、或过滤后没有 human+ai 组合时，两侧都直接 return。 若 _processing=True 时又来 add_nowait，_process_queue 会再 _schedule_timer(0)，保留「即时 flush」语义而不丢队。 同一轮既 after_agent 又即将 summarization 时，后到的入队会覆盖同 key 的 messages，correction/reinforcement 标志做 OR 合并—— 通常 summarization 批次更完整，这正是期望行为。 memory_flush_hook 用 resolve_runtime_user_id 而非 middleware 那套 trace 捕获；详见参考页对照表。

判断题：用户长对话触发 summarization，但 memory.enabled=true 且 summarization 已开启。被压缩的旧消息若未经过 memory_flush_hook，仅靠 after_agent 的 30s debounce，是否仍能保证写入 memory？

A. 能，after_agent 每轮都已入队   B. 不能，Timer 未触发前消息已被摘要丢弃   C. 能，dedupe 会保留最早一份   D. 取决于 DynamicContext 注入

参考答案：B。debounce 只延迟处理，不阻止 summarization 从 state 删除原文；没有 hook 的即时 flush，队列里可能仍是旧快照或 Timer 尚未跑完。A 忽略了压缩与 Timer 的竞态；C 错在 dedupe 保留的是最新快照而非最早。
