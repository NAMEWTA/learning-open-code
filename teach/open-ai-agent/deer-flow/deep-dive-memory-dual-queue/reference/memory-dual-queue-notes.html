注册与 middleware 顺序

lead_agent/agent.py 在构建 SummarizationMiddleware 时传入 before_summarization=[memory_flush_hook]（summarization 开启时）。 MemoryMiddleware（第 19 位）在 SummarizationMiddleware（第 15 位）之后执行 after_agent，因此正常轮次先完成 agent 图、再入 debounce 队列； summarization 在同一轮或后续轮触发时，hook 在压缩前额外 flush。

_processing 与重入

场景行为
Timer 触发 _process_queue持锁设 _processing=True，拷贝队列后清空
处理中又来 add_nowait新上下文入队；若仍在 processing，结束时再 schedule_timer(0)
多 context 批量处理每个 context 间隔 0.5s，减轻 rate limit
flush() / flush_nowait()测试与关停用；生产路径主要靠 add / add_nowait

排障口诀

  • 长对话后 memory 缺事实：确认 summarization 开启且 hook 已注册；查 hook 是否因无 human/ai 跳过。
  • 短对话重复抽取：正常——debounce 窗口内多次 add 会合并；检查 debounce_seconds 是否过短。
  • Timer 线程无 trace：middleware 路径已捕获 deerflow_trace_id；hook 路径未捕获，日志关联可能断裂。
