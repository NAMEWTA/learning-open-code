15 分钟读懂 scheduled task 创建到执行

学习目标：读完后，你能把「用户在 /workspace/scheduled-tasks 点创建」到「lead agent 真正跑起来」串成一条链，并知道该去哪个文件继续下钻。完整接口表与状态机见 参考页。

1. 前端：表单收集什么

页面在 frontend/src/app/workspace/scheduled-tasks/page.tsx。用户填写标题、prompt、调度（cron 或 once）、时区，并选择 context_mode：每次新建线程，或复用已有 thread_id。提交时 useCreateScheduledTask 调用 frontend/src/core/scheduled-tasks/api.ts 的 POST /api/scheduled-tasks。列表与 run 历史通过 TanStack Query 每 15 秒刷新。

2. Gateway：校验并算 next_run_at

backend/app/gateway/routers/scheduled_tasks.py 的 create_scheduled_task 做三件事：鉴权与 threads:write 权限；校验 context_mode、复用线程是否存在；调用 deerflow/scheduler/schedules.py 的 next_run_at 把 cron/once 转成 UTC 时间点。通过后写入 persistence/scheduled_tasks/，任务初始状态为 enabled，并带上 next_run_at。

3. 调度器：认领到期任务并 launch run

Gateway 启动时（config.scheduler.enabled 为真）会构造 app/scheduler/service.py 的 ScheduledTaskService 并启动轮询。每个周期 run_once 通过 claim_due_tasks 用 lease 锁定 next_run_at <= now 且状态为 enabled 的行，再调用 dispatch_task：先写 scheduled_task_runs 行，再经 launch_scheduled_thread_run 走与普通聊天相同的 start_run 路径，并在 metadata 里打上 scheduled_task_id。背景 run 会注入 context.non_interactive=true，避免 agent 卡在澄清提问。

4. 完成回写：run 结束更新任务

run 进入终态后，handle_run_completion 根据 run 的 success/failed/interrupted 更新对应 task_run，并对 once 任务把父任务标为 completed / failed / cancelled；cron 任务则保持 enabled 并已在 launch 时写好下一次 next_run_at。用户也可在详情面板点「立即触发」，走同一条 dispatch_task(..., trigger="manual")，但路由层会对并发中的手动触发返回 409。

检索练习：下面哪一步不会发生在「仅创建 cron 任务、尚未到点」时？ A. 写入 scheduled_tasks 表 B. 计算并保存 next_run_at C. claim_due_tasks 认领任务 D. ScheduledTaskService.start() 启动轮询循环
参考答案：C。创建阶段只持久化任务与下次执行时间；认领与 dispatch_task 要等 next_run_at 到期且调度器轮询到才会发生。D 在 Gateway 启动且 scheduler 开启时就会发生，与是否已有到期任务无关。

主要来源：scheduled_tasks.py、app/scheduler/service.py、launch_scheduled_thread_run。想沿某一段继续展开（例如 lease 或 overlap skip），向 agent 追问即可。
