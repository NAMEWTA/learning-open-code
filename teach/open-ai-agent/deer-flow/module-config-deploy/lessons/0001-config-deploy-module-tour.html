15 分钟理解配置与部署边界

学习目标：读完后，你能说清 deer-flow 的两份运行时配置各管什么、根 Makefile 如何把前后端与 nginx 拉起来，以及本地 make dev 与 Docker make up 的差别。

1. 配置层：两份文件，两种职责

deer-flow 把主应用配置和扩展配置拆开。主配置在仓库根目录的 config.example.yaml → 复制为 gitignored 的 config.yaml，覆盖 models、sandbox、database、channels、scheduler 等大块能力。扩展配置在 extensions_config.example.json → extensions_config.json，只管 MCP servers 与 skills 开关。首次生成走 make config（调用 scripts/configure.py，若已存在则中止），schema 升级走 make config-upgrade。路径可用 DEER_FLOW_CONFIG_PATH / DEER_FLOW_EXTENSIONS_CONFIG_PATH 覆盖；运行时数据默认落在 .deer-flow/，可用 DEER_FLOW_HOME 改位置。

deer-flow/ ├── config.example.yaml ← 主配置模板（config_version: 18） ├── config.yaml ← 本地真实配置（gitignored） ├── extensions_config.example.json ├── extensions_config.json ← MCP + skills（gitignored） ├── .env.example → .env ← 密钥与 compose 变量 └── backend/pyproject.toml ← Python 3.12+，uv workspace frontend/package.json ← pnpm，Node 22+

2. 编排层：Makefile 是整栈遥控器

根目录 Makefile 不实现业务，只把 scripts/ 里的 shell/Python 脚本接成用户命令。新手推荐 make setup（交互向导）；日常开发用 make check 验工具链后 make dev（热重载）。生产式本地跑用 make start；后台模式加 -daemon。Docker 开发是 make docker-start，Docker 生产是 make up / make down。排障先跑 make doctor。完整命令表见 参考页。

make dev / make start ↓ scripts/serve.sh（--dev 或 --prod） Gateway :8001 + Frontend :3000 + Nginx :2026 ↑ 浏览器统一入口 http://localhost:2026 make up ↓ scripts/deploy.sh docker/docker-compose.yaml nginx + frontend + gateway + redis (+ provisioner 可选)

3. 部署形态：四条常见路径

本地 dev：make dev，Gateway 带 reload，前端 Turbopack，nginx 用本地配置。适合改代码。本地 prod：make start，无热重载，更接近发布行为。Docker dev：make docker-start，compose 文件是 docker/docker-compose-dev.yaml，容器内热重载。Docker prod：make up，docker/docker-compose.yaml 构建镜像，gateway 挂载 config.yaml 与 extensions_config.json，并依赖 Redis 做 stream bridge。无论哪条路径，对外入口都是 nginx :2026，与 L0 总览 一致。

make config          # 首次从 example 生成 config.yaml / .env
make doctor          # 检查配置、依赖与 LLM 连通性
make dev             # 本地开发整栈
make up              # Docker 生产整栈（localhost:2026）

4. 你该如何继续读源码

第一遍只读 Makefile 的 help 目标，建立命令地图。第二遍跟 scripts/serve.sh 看进程如何拉起、端口如何占用与释放。第三遍打开 docker/docker-compose.yaml 顶部注释，对照 gateway 的环境变量与卷挂载。函数级签名、热加载边界与脚本 CLI 见 L3 微观 API 参考；字段语义下钻到 backend/docs/CONFIGURATION.md，不要在本课硬背 config.example.yaml 全文。

检索练习：判断三句话是否正确：A. extensions_config.json 里配置 LLM 模型列表。B. make dev 与 make up 都会让用户从 localhost:2026 访问。C. make config 在已有 config.yaml 时会覆盖原文件。
参考答案：A 错，模型在 config.yaml 的 models 段；extensions_config.json 管 MCP 与 skills。B 对，两者都经 nginx 统一入口 2026。C 错，configure.py 检测到已有配置会中止，避免误覆盖；升级应使用 make config-upgrade。

主要来源：AGENTS.md、Makefile、docker/docker-compose.yaml。若某条部署命令或环境变量不清楚，向 agent 追问「沿着 config-deploy 的 Makefile 或 compose 继续展开」。
