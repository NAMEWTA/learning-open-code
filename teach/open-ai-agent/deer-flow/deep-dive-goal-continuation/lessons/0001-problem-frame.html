goal 续跑：问题背景与设计备选

学习目标：读完后，你能说明「为什么要在 worker 里做隐藏续跑」，并对比 deer-flow 与几种常见备选架构的取舍。下一节：核心机制拆解。

1. 要解决的具体问题

用户用 /goal 或 PUT /goal 设定长期目标后，一次可见 agent 回合往往不足以完成。若只靠用户反复发「继续」，体验差且容易漏掉中间状态。Claude Code 的 goal loop 思路是：可见回合结束后，由系统静默判断是否还需工作，必要时在同一 run_id 内再跑隐藏回合。

难点不在「再调一次模型」，而在三件事同时成立：① 不能把未落盘的半成品当证据；② 用户并发发消息或清空 goal 时，evaluator 的滞后结论不能覆盖新状态；③ 续跑必须有硬顶，防止 token 与工具调用空转。这三条约束直接塑造了 worker.py + goal.py 的分工。

2. 三种备选架构

方案做法优点deer-flow 未采用的原因
前端轮询续跑UI 检测 goal 未满足后再次 POST /runs实现简单多一次 HTTP 往返；竞态与 run 生命周期难与 checkpointer 对齐
外部任务队列Cron/队列在 run 结束后投递新 job与 agent 解耦需额外调度器与 run 归属；IM/TUI 也要走同一语义
worker 内 while 环（现行）run_agent 流式结束后调用 _prepare_goal_continuation_input同一 run、同一 evaluator 模型缓存；与 SSE 流一致worker 变复杂；单次 run 可能变长

类型化 GoalBlocker（见 goal_state.py）是另一项设计选择：只有 goal_not_met_yet 可续跑，其余 blocker 只写 last_evaluation 并 stand down，避免用自由文本 reason 驱动分支。

3. 模块边界为何这样切

goal_state.py 只放 TypedDict，让 harness 与 Gateway 共享契约而不引 FastAPI。goal.py 放纯函数与 checkpoint 读写，便于单测。worker.py 只管编排：何时调 evaluator、何时三次重读、何时注入 hide_from_ui 消息。evaluator 本身是无 thinking 的小模型（create_goal_evaluator_model），与 lead agent 分离，降低「自己判自己完成」的偏差。

若你已读过 L2 流程课，本节回答的是「为什么链路长这样」；机制逐步拆解见 0002，速查表见 参考笔记。

判断题：把续跑做成「前端在 run success 后再 POST 一次 /runs」与现行 worker 内环相比，主要损失的是哪一项？ A. 类型化 blocker B. 与 checkpointer 同一 run 内的原子续跑语义 C. evaluator 使用非 thinking 模型
参考答案：B。前端二次 POST 会新建 run，难以复用同一 run 内的 abort 信号、evaluator 模型缓存，且与「隐藏续跑不新增 run_id」的产品语义不一致。A、C 在两种架构下都可以保留。
