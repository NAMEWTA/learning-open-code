goal 续跑：算法拆解与性能边界

学习目标：读完后，你能按顺序说出 _prepare_goal_continuation_input 的决策步骤，并解释性能开销与已知局限。上一节：问题背景。

1. 决策树（单次续跑判定）

read_thread_goal → 非 active 则结束 ↓ _has_durable_goal_turn_receipt? (无 pending_writes + 末条可见 AI) ↓ 否 → blocker=run_failed, stand down evaluate_goal_completion (LLM, O(对话长度)) ↓ compute_no_progress_count (progress_key = blocker + AI 文本 SHA256) ↓ 重读 goal 实例 + 可见对话签名 → 竞态则 stand down ↓ satisfied? → write_thread_goal(None) ↓ _stand_down_reason / should_continue_goal? → 只写 last_evaluation ↓ continuation_count++ → 第三次重读 → make_goal_continuation_message

_goal_instance_matches 用 objective + created_at + status 识别「同一条 goal」，防止 evaluator 期间用户换了新目标却被旧结论写回。

2. 性能特征

每次 while 迭代至少 2～3 次 checkpointer 读（评估前、评估后、排队续跑前），写 goal 时还有一次带 expected_checkpoint_id 的乐观写。对话证据截断在 30 条 / 12000 字符（goal.py 常量），evaluator 单次 LLM 调用 dominates 延迟。

最坏情况：可见回合 1 次 + 8 次隐藏续跑 ×（1 次 lead agent 流 + 1 次 evaluator）≈ 9 倍可见回合成本，且受 max_no_progress_continuations=2 提前熔断。同一 run 内 evaluator 模型通过 _get_goal_evaluator_model 缓存，避免重复构造。

3. 已知局限与改进方向

  • 证据仅可见对话：沙箱里改了文件但 assistant 没说出来 → missing_evidence，续跑停住。改进需扩展证据源（工具结果摘要、workspace diff），但会增大 evaluator 输入与误判面。
  • LLM evaluator 非确定：同一线程可能因措辞抖动得到不同 blocker；progress_key 故意忽略 reason 文本，只盯 blocker + 可见 AI 签名防空转。
  • 单 worker 串行续跑：长 goal 占用 run 槽位；外部队列方案可缓解，但牺牲当前的原子语义。
  • 无 checkpointer 时完全禁用：checkpointer is None 直接 return，memory 后端无法续跑。
def compute_goal_progress_key(evaluation, *, evidence_signature=""):
    return json.dumps({
        "satisfied": evaluation["satisfied"],
        "blocker": evaluation["blocker"],
        "evidence_signature": evidence_signature,
    }, sort_keys=True)

evidence_signature 来自 latest_visible_assistant_signature（最新可见 AI 文本的 SHA256），故意不用 evaluator 的 reason 文本。

检索练习：连续两次续跑后可见 assistant 文本完全相同，no_progress_count 会是多少（默认配置）？续跑还会继续吗？
参考答案：第二次续跑后 no_progress_count 达到 2，等于 max_no_progress_continuations，should_continue_goal 返回 false，stand_down_reason=no_progress_detected，不再注入隐藏消息。
