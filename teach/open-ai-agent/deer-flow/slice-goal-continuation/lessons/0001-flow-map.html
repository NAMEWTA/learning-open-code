15 分钟读懂 goal 自动续跑主链路

学习目标：读完后，你能把「一次可见 run 结束」到「worker 是否再跑隐藏续跑回合」串成一条链。blocker 表、并发守卫与熔断细节见 参考页。

1. 三类状态各管什么

续跑不是前端轮询，而是 runtime/runs/worker.py 在 run_agent 里包住 LangGraph 流式执行。GoalState 与 GoalEvaluation 的类型在 agents/goal_state.py；读写 checkpoint、调 evaluator、判断 should_continue_goal 在 runtime/goal.py。活跃 goal 存在 checkpointer 的 channel_values.goal，对用户可见对话在 messages。

2. worker 里的 while 续跑环

可见回合先走一次 _stream_once(graph_input)。只要 run 未 abort、无 LLM fallback，就进入 while，调用 _prepare_goal_continuation_input。若返回隐藏输入，再 _stream_once(continuation_input)，用户界面看不到这条 hide_from_ui 的 HumanMessage。

run_agent() │ ├── _stream_once(用户可见输入) ← 正常 SSE 回合 │ └── while 未 abort 且无 LLM 错误: _prepare_goal_continuation_input() ├── read_thread_goal() ├── _has_durable_goal_turn_receipt() ← 末条可见消息是 AI 且无 pending_writes ├── evaluate_goal_completion() ← 非 thinking 小模型 ├── should_continue_goal() ← 仅 goal_not_met_yet 可续 └── make_goal_continuation_message() ← 隐藏 HumanMessage └── _stream_once(续跑输入) ← 同一 run_id 内继续

3. 两道熔断与 stand down

续跑不是无限循环。continuation_count 硬上限 8；no_progress_count 在可见 assistant 证据签名未变时累加，默认连续 2 次无新输出就停。evaluator 若判 satisfied，worker 会清空 goal；其他 blocker（如 needs_user_input）只写 last_evaluation 并 stand down，不注入续跑消息。

4. 并发：为何 evaluator 前后要重读

评估 LLM 调用期间，用户可能发新消息或 /goal clear。_prepare_goal_continuation_input 在评估前后比对 goal 实例与可见对话签名；checkpoint 或对话变了就记录 stand_down_reason 并放弃续跑。写 goal 时用 goal_thread_lock 与 expected_checkpoint_id，避免陈旧 evaluator 结果覆盖新 goal。

检索练习：evaluator 返回 blocker=needs_user_input 且 satisfied=false 时，下面哪项一定会发生？ A. continuation_count 加 1 B. 注入 <goal_continuation> 隐藏消息 C. 写回 last_evaluation D. 清空 channel_values.goal
参考答案：C。该 blocker 不在 CONTINUABLE_GOAL_BLOCKERS（只有 goal_not_met_yet），故不会续跑（B、A 不发生），也不会因未满足而清空 goal（D 仅在 satisfied 时发生）。评估结果仍会持久化供观测。

主要来源：worker.py 第 7 步流式循环与 _prepare_goal_continuation_input，goal.py 的 should_continue_goal。想沿某段继续展开（例如 durable receipt 或 progress_key），向 agent 追问即可。
