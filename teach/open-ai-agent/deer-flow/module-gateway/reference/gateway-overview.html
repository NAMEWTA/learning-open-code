Gateway API 外壳参考

本页承载 L1 Gateway 模块的速查材料：边界、内部结构、router 清单、关键依赖、认证规则和简单调用。L3 微观 API 见 gateway-api.html；短课入口：0001-gateway-module-tour.html；整体锚点见 L0 项目总览。

模块边界

维度结论源码依据
职责创建 FastAPI app，启动 embedded LangGraph runtime，注册认证/CSRF/CORS/trace 中间件，挂载 HTTP router，暴露 health check。backend/app/gateway/app.py
位置位于 nginx 和 deerflow harness 之间，对应 L0 总体架构图中的 Gateway 层。nginx 把 /api/langgraph/* 改写到 Gateway 原生 /api/*；Gateway 再调用 RunManager、StreamBridge、checkpointer、store 和 lead agent。docker/nginx/nginx.conf、backend/README.md
非职责不实现 lead agent 推理策略，不实现工具执行细节，不直接渲染前端，不替代 persistence schema。backend/packages/harness/deerflow/、frontend/src/
默认运行方式Docker 与脚本默认用 uvicorn app.gateway.app:app 启动 Gateway embedded runtime；langgraph.json 主要保留给 LangGraph tooling/Studio 兼容。docker/docker-compose.yaml、backend/langgraph.json

内部分层结构

backend/app/gateway/ ├── app.py ← FastAPI app factory、lifespan、middleware、router 注册 ├── config.py ← GatewayConfig，读取 GATEWAY_HOST、GATEWAY_PORT、GATEWAY_ENABLE_DOCS ├── deps.py ← request 依赖、app.state 单例访问、RunContext 构造、auth helper ├── auth_middleware.py ← fail-closed 认证、公开路径、internal token、user contextvar ├── csrf_middleware.py ← Double Submit Cookie、auth origin 检查、CORS origin 来源 ├── langgraph_auth.py ← LangGraph Server/Studio 兼容认证与 user_id metadata filter ├── trace_middleware.py ← HTTP request trace id 绑定与响应 header ├── routers/ ← 20 个 HTTP router 模块 │ ├── thread_runs.py ← thread-scoped run 生命周期和 SSE │ ├── runs.py ← stateless run stream/wait │ ├── threads.py ← thread metadata、state、goal、history │ └── auth.py ← 登录、注册、初始化、OAuth └── auth/ ← 认证子系统 ├── jwt.py ← access token 创建与验证 ├── local_provider.py ← 本地邮箱密码认证 └── oidc.py ← OIDC 登录流程

启动阶段

顺序动作说明
1加载 AppConfig 与 GatewayConfigAppConfig 用于 deerflow runtime；GatewayConfig 只控制 host、port、docs。
2配置日志与安全提示configure_logging() 使用启动快照；warn_if_auth_disabled_enabled() 提醒 auth-disabled 风险。
3预热 token counting 与清理上传暂存避免首个 memory request 卡住；清理 stale upload staging files。
4进入 langgraph_runtime()构造 stream bridge、persistence engine、checkpointer、store、run store、thread store、scheduled task repo。
5迁移 orphan thread、启动 IM channel 与 scheduler认证升级路径与后台任务都挂在 Gateway 生命周期里。
6关闭阶段 drain in-flight runs先让 `RunManager.shutdown()` 有界收尾，再关闭 persistence engine。

认证与中间件速查

层关键规则源码
公开路径`/health`、docs、OpenAPI、OAuth callback、GitHub webhook、登录/注册/初始化等少数路径公开。backend/app/gateway/auth_middleware.py
session 认证非公开路径要求 `access_token` cookie。JWT 解码后还会查用户表，并校验 `token_version`。backend/app/gateway/deps.py、backend/app/gateway/auth/jwt.py
internal callerIM channel 等可信内部调用可使用 `X-DeerFlow-Internal-Token`，并可带 owner override。backend/app/gateway/internal_auth.py
CSRFPOST、PUT、DELETE、PATCH 默认需要 cookie/header 双提交；auth 首次 POST 有例外，但会做 Origin 限制并设置 cookie。backend/app/gateway/csrf_middleware.py
CORS默认 same-origin；split-origin 需要 `GATEWAY_CORS_ORIGINS`，该 allowlist 也被 CSRF origin 检查复用。backend/app/gateway/csrf_middleware.py、backend/app/gateway/app.py
LangGraph auth`langgraph.json` 的 `auth.path` 指向 `langgraph_auth.py:auth`，复用 JWT 与 CSRF，并在 metadata 写入 `user_id`。backend/app/gateway/langgraph_auth.py、backend/langgraph.json

关键依赖

依赖Gateway 如何使用
deerflow.config.app_config.get_app_config启动拿快照；request 时通过 `get_config()` 重新读取可热加载字段。
RunManager管理 run 创建、查询、取消、shutdown drain 和 SQLite orphan recovery。
StreamBridge把 run 事件送到 SSE；Redis bridge 可支持跨 worker SSE delivery 与 replay。
checkpointer 与 storeLangGraph checkpoint 和 Store，由 `make_checkpointer()`、`make_store()` 在 lifespan 中创建。
RunEventStore持久化或读取 run events，与启动时的 `run_events_config` 绑定。
ThreadMetaStore线程元数据、状态和 owner 过滤的基础存储。
ScheduledTaskService在 Gateway 生命周期里启动轮询，并复用 `launch_scheduled_thread_run()` 创建 run。
app.channels.serviceIM channel 服务随 Gateway 启停，内部调用通过 internal auth 进入同一 API 边界。

调用速查

健康检查

curl -s http://localhost:8001/health
# {"status":"healthy","service":"deer-flow-gateway"}

本地登录后访问业务 API

curl -i -c cookies.txt \
  -X POST http://localhost:8001/api/v1/auth/login/local \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data "username=admin@example.com&password=your-password"

curl -b cookies.txt http://localhost:8001/api/models

对 POST、PUT、PATCH、DELETE 业务请求，还需要从 `csrf_token` cookie 读取值并放入 `X-CSRF-Token` header。登录、注册、初始化这几个 auth POST 是首次会话入口，中间件会在响应中设置 CSRF cookie。
