问题框架：注册表、签名与误报

学习目标：读懂「配置已重载」与「行为已更新」为何可能脱节，并能用注册表与签名逻辑排障。 完整字段表见 边界深潜参考； L2 总览见 热加载切片。

L4 深潜 前置：slice-config-hot-reload

1. 契约：两层配置，一个误报源

issue #3144 把 DeerFlow 配置分成两层。请求层：app_config.py 的 get_app_config() 在每次 Gateway 依赖注入时检查文件签名，必要时重载 AppConfig 对象。 启动层：gateway/app.py 的 lifespan() 把 startup_config = get_app_config() 存为局部变量，只喂给 configure_logging、langgraph_runtime、IM channels、scheduler—— 刻意不写入 app.state（issue #3107）。 误报根因：日志里出现「signature changed, reloading AppConfig」，只证明内存里的配置对象换了， 不证明启动时捕获的单例（engine、sandbox provider、TraceMiddleware）会跟着变。

2. STARTUP_ONLY_FIELDS 为何这样设计

reload_boundary.py 用字典把顶层段映射到「哪段启动代码捕获了快照」。 值不是「需重启」四字，而是可操作的 reason——例如 sandbox 写明 get_sandbox_provider() 缓存了 _default_sandbox_provider。 AppConfig 内字段通过 format_field_description() 生成 startup-only: … 前缀的 Field(description=…)，与注册表双向由 test_reload_boundary.py 锁死。channels 不在 Pydantic schema 里，却必须在注册表单独列出—— 运维改 Feishu 凭证时 IDE 无法 hover，注册表是唯一 canonical 位置。 粒度是段而非叶子键：改 database.sqlite_dir 仍属 database 段，需重启。

3. 签名检测：何时认为「文件变了」

_get_config_signature() 返回 (mtime, size, sha256_hex) 三元组。 get_app_config() 在路径变化或整段签名不等时触发 AppConfig.from_file()。 单独比较 mtime 是为了打日志；真正决策用完整签名，以应对对象存储/NFS 上 mtime 陈旧但内容已换的场景 （test_get_app_config_reloads_when_content_digest_changes_without_metadata）。 热加载字段因此能在「保存 config.yaml → 下一条 HTTP 请求」链路上生效；启动字段即使签名变了，也仍绑定 lifespan 里的旧单例。 

should_reload = (
    _app_config is None
    or _app_config_path != resolved_path
    or _app_config_signature != current_signature
)

4. 三类典型误报与排障口诀

① 配置对象新、provider 旧：改 sandbox.use 后 get_app_config() 返回新 SandboxConfig，但 get_sandbox_provider() 仍是启动时缓存的类实例 → 重启 Gateway。 ② 日志级别/trace 头不变：改 log_level 或 logging.enhance.enabled； apply_logging_level / configure_logging 与 TraceMiddleware 构造只在启动跑 → 重启。 ③ 非边界问题：改 memory.debounce_seconds 已热加载，但事实写入要等 debounce + LLM—— 勿与「需重启」混淆（见 memory 切片）。 口诀：先查段是否在 STARTUP_ONLY_FIELDS；在则重启；不在则查是否误读 startup_config 或异步延迟。

判断练习（先回忆再点开答案）：运维改 config.yaml 后日志出现 Config file content signature changed, reloading AppConfig，但 sandbox 仍走 local provider。最可能原因？
参考答案：sandbox 在 STARTUP_ONLY_FIELDS 中；签名重载只更新 AppConfig 对象，get_sandbox_provider() 单例在 lifespan 启动时已捕获，必须重启 Gateway。
