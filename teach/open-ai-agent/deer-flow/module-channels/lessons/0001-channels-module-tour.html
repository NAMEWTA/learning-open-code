15 分钟理解 IM channels 与 GitHub webhook 边界

学习目标：读完后，你能说明 deer-flow 如何把外部 IM 与 GitHub 事件接到同一条 channel 总线，并区分三类 Gateway 入口各自管什么。

1. 模块在架构里的位置

Channel 层位于 L0 总体架构图 的 Gateway 外侧：浏览器聊天走前端 → Gateway；IM 与 GitHub 则走 backend/app/channels/。Channel 不直接 import harness 图，而是通过 langgraph_sdk 调用 Gateway 的 /api/*（与前端同源），因此 thread、run、goal 都落在同一套 runtime 上。

Feishu / Slack / Telegram / … (long-poll workers) GitHub App webhook ↓ Gateway routers (验签 / 绑定 API / 运维重启) ↓ MessageBus.publish_inbound(InboundMessage) ↓ ChannelManager → langgraph_sdk → Gateway /api ↓ lead_agent graph (deerflow harness)

2. 两条入站路径

路径 A — 传统 IM：各平台 adapter（如 feishu.py）在 service.py 注册后长轮询或收事件，把用户文本包装成 InboundMessage 丢进总线。若启用了 channel_connections，入站前还会把平台账号解析成 DeerFlow 用户绑定。

路径 B — GitHub webhook：github_webhooks.py 校验 X-Hub-Signature-256 后调用 fanout_event，按 custom agent 的 github: 绑定生成多条入站消息。Webhook handler 本身不跑 agent，只负责验签与 fan-out，避免踩 GitHub 10 秒超时。

3. 总线 + 调度器只做三件事

MessageBus 用 asyncio 队列解耦「平台线程」与「agent 调度」：publish_inbound 入队，ChannelManager._dispatch_loop 出队。调度器随后：① 校验绑定身份或去重；② 经 SDK 创建/复用 thread 并调用 runs.wait、runs.stream 或 GitHub 专用的 runs.create（fire-and-forget）；③ 把回复通过 publish_outbound 送回对应 channel。完整接口表与 provider 列表见 channels 参考页。

backend/app/channels/ ├── message_bus.py ← InboundMessage / OutboundMessage / 队列 ├── manager.py ← 消费入站、调 Gateway、发布出站 ├── service.py ← 启停各平台 Channel + Manager 单例 ├── base.py ← Channel 抽象类 └── feishu.py … ← 平台适配器 backend/app/gateway/ ├── routers/channels.py ← 运维：状态 / 重启 ├── routers/channel_connections.py ← 用户：绑定 / runtime-config ├── routers/github_webhooks.py ← GitHub 入站 └── github/dispatcher.py ← fan-out 到 MessageBus

4. 你该如何继续读源码

第一遍读 message_bus.py 的数据结构，确认入站/出站字段（connection_id、owner_user_id、topic_id）。第二遍读 ChannelManager._handle_chat 看 thread 映射与三种 run 模式。第三遍选一个入口下钻：绑定流程看 channel_connections.py，GitHub 看 fanout_event。更细的端到端切片留给 L2 slice-im-channel-run。

检索练习：判断正误：A. GitHub webhook 在路由里直接 runs.wait 等 agent 结束。B. IM channel 与浏览器 共用 Gateway 的 thread API。C. POST /api/channels/telegram/connect 面向终端用户生成绑定码。
参考答案：A 错，webhook 只验签并 fanout_event 到总线，agent 由 ChannelManager 异步触发；GitHub 甚至用 fire-and-forget 的 runs.create。B 对，ChannelManager 通过 langgraph_sdk 调同一 Gateway。C 对，该路由在 channel_connections.py，返回 /start 或 /connect 指令。

主要来源：backend/app/channels/service.py、backend/app/channels/manager.py、backend/app/gateway/routers/github_webhooks.py 与 backend/AGENTS.md IM Channels 章节。不清楚某个 provider 或 webhook 事件时，向 agent 追问「沿 channel 绑定或 GitHub fan-out 继续展开」。
