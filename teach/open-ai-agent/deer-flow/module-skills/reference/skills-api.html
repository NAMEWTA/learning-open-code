Registry — 配置与发现

SkillsConfig

class SkillsConfig(BaseModel): use: str; path: str | None; container_path: str; deferred_discovery: bool

backend/packages/harness/deerflow/config/skills_config.py — AppConfig.skills 子段；决定 storage 实现类、host 路径、容器挂载路径与 deferred discovery 开关。

字段 / 方法含义与约束
use反射路径，默认 deerflow.skills.storage.local_skill_storage:LocalSkillStorage；由 resolve_class(..., SkillStorage) 实例化。
path显式 host skills 根目录；为 None 时走 get_skills_path() 解析链。
container_path沙箱内虚拟路径前缀，默认 /mnt/skills；写入 prompt 与 slash 激活路径。
deferred_discoveryTrue 时 prompt 只渲染 <skill_index> 并注册 describe_skill；默认 False（完整 metadata 注入）。
get_skills_path() → Path解析顺序：显式 path → DEER_FLOW_SKILLS_PATH → project_root()/skills → legacy monorepo repo_root/skills；均不存在时仍返回 project 默认路径（不抛错）。
get_skill_container_path(name, category) → str返回 {container_path}/{category}/{name} 形式的目录路径。

SkillStorage.load_skills

def load_skills(self, *, enabled_only: bool = False) -> list[Skill]

backend/packages/harness/deerflow/skills/storage/skill_storage.py — 模板方法：遍历 _iter_skill_files() → parse_skill_file → 按 name 去重（后者覆盖前者）→ 合并 ExtensionsConfig 启用状态 → 可选过滤 → 按名字排序。

边界行为
frontmatter 缺 name/descriptionparse_skill_file 返回 None，该文件被跳过。

SkillActivationMiddleware

class SkillActivationMiddleware(AgentMiddleware)

def __init__(self, *, available_skills: set[str] | None = None, app_config: AppConfig | None = None, user_id: str | None = None)

backend/packages/harness/deerflow/agents/middlewares/skill_activation_middleware.py — 在 wrap_model_call / awrap_model_call 中运行；读取经 get_original_user_content_text 还原的原始用户文本（绕过 InputSanitization 改写）。

阶段行为
目标消息自后向前找最近真实 HumanMessage（排除 summary、hide_from_ui）
去重同一 target 已有 slash_skill_activation 隐藏消息则跳过
解析失败非 slash 输入 → 无操作；已安装但 disabled / 不在 agent 白名单 / 读盘失败 → 返回 AIMessage 错误说明
成功在 target 前插入隐藏 HumanMessage（hide_from_ui、slash_skill_activation），内含完整 XML 转义后的 SKILL.md
审计__run_journal 记录 middleware:skill_activation（name、path、content_hash，无正文）
secrets同次调用内 _resolve_secret_bindings：slash 路径写入 __slash_skill_secret_source，与 context.secrets 求交后注入 __active_skill_secrets

resolve_slash_skill 只做纯解析；middleware 额外负责读盘、注入消息与 secrets。两者对 enabled / whitelist 的检查语义一致。

filter_tools_by_skill_allowed_tools

def filter_tools_by_skill_allowed_tools(tools, skills, *, always_allowed_tool_names=frozenset()) -> list[ToolT]

场景行为

装配顺序速记

load_skills(enabled_only=True)
  → filter by custom agent skills whitelist
  → apply_prompt_template (full metadata or skill_index + describe_skill)
  → get_available_tools(...)
  → filter_tools_by_skill_allowed_tools(..., always_allowed=read_file + infra)
  → SkillActivationMiddleware(available_skills=whitelist names)

slash 与 allowed-tools 共用同一套「enabled + agent 白名单」后的 skill 集合；slash 在 middleware 层再次校验，不能绕过白名单或 disabled 状态。

需要 Gateway HTTP 路由、custom skill CRUD 或 durable skill_context 细节时，回到 skills-overview.html 的接口表与测试索引。
