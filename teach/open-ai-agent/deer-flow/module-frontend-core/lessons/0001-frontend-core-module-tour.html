15 分钟理解 DeerFlow 前端 core 数据层

本课只教一个目标：看到 frontend/src/core/ 里的代码时，能判断它属于哪条出站通道、与 workspace 页面层 的分工是什么。

目标读者：React Query 基础 主线：SDK 与 REST 双通道 参考：完整清单

1. core 与 workspace 的边界

在 L0 架构图里，Next.js frontend 分成两层： workspace 负责路由壳与可见组件；frontend/src/core/ 负责业务数据、API 适配与 React Query 缓存。 workspace 组件调用 core hooks，core 再连 Gateway。

frontend/src/ ├── app/workspace/ ← 页面路由（workspace 模块） ├── components/workspace/ ← 聊天、侧栏等 UI（workspace 模块） └── core/ ← 本课：数据层与 API client ├── api/ ← LangGraph SDK + REST fetch wrapper ├── auth/ ← 会话守卫与 AuthProvider ├── threads/ ← 线程流式主链路 ├── skills/ ← skills 配置 REST ├── mcp/ ← MCP 配置 REST └── uploads/ ← 线程文件上传 REST

读代码时先问：这是拼 UI，还是取数/提交？后者几乎总在 core。

2. 两条出站通道

DeerFlow 前端有两条固定出站路径。通道 A： frontend/src/core/api/api-client.ts 的 getAPIClient() 创建 LangGraph SDK Client，走 /api/langgraph，负责 threads、runs、stream、joinStream。 通道 B： frontend/src/core/api/fetcher.ts 包装 fetch，走 Gateway REST（/api/skills、/api/mcp/config、 /api/threads/{id}/uploads 等），统一带 cookie 与 CSRF。 

// 通道 A：聊天流式提交
getAPIClient() -> client.runs.stream(threadId, assistantId, payload)

// 通道 B：配置类 REST
fetch(`${getBackendBaseURL()}/api/skills`)  // skills/api.ts
fetch(`${getBackendBaseURL()}/api/mcp/config`)  // mcp/api.ts
fetch(`${getBackendBaseURL()}/api/threads/${threadId}/uploads`, { method: "POST" })  // uploads/api.ts

判断口诀：涉及 thread/run 流式生命周期 → SDK；其余 CRUD/配置 → fetcher.ts。

3. 认证外壳与线程主链路

认证分服务端与客户端两层。 frontend/src/core/auth/server.ts 的 getServerSideUser() 在 layout 阶段用 cookie 向 Gateway 求证，决定跳转登录或 setup。 frontend/src/core/auth/AuthProvider.tsx 只保存展示用 user，不持有 JWT。 聊天主链路在 frontend/src/core/threads/hooks.ts： chats/[thread_id]/page.tsx 持有 useThreadStream； workspace 的 InputBox 通过 onSubmit 回调触发 sendMessage，不直接碰 SDK。 

page.tsx 持有 useThreadStream
  -> InputBox.onSubmit
  -> sendMessage
  -> uploadFiles (uploads/api.ts，通道 B)
  -> getAPIClient().runs.stream (通道 A)
  -> thread.messages 更新 -> MessageList 重渲染

4. skills / mcp / uploads 的同构模式

这三个域都遵循 api.ts（纯 HTTP）+ hooks.ts（React Query）分工。 skills/hooks.ts 的 useSkills 读列表、useEnableSkill 写开关； mcp/hooks.ts 的 useMCPConfig / useEnableMCPServer 同理； uploads/hooks.ts 用 useUploadLimits、useUploadFiles 管线程附件。 设置页、工具页、输入框附件按钮都消费这些 hooks，而不是在组件里写裸 fetch。

练习：30 秒通道判断

你要排查「用户在设置页切换 MCP server 开关后 UI 不刷新」。按本课结构，你会先看哪个 hooks 文件？底层请求走通道 A 还是 B？

参考答案：先看 frontend/src/core/mcp/hooks.ts 的 useEnableMCPServer，确认 mutation 成功后是否 invalidateQueries({ queryKey: ["mcpConfig"] })； 再看 frontend/src/core/mcp/api.ts 的 updateMCPConfig。底层走通道 B（REST PUT /api/mcp/config），不是 LangGraph SDK。
