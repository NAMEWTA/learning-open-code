15 分钟读懂 memory 注入与异步更新

学习目标：读完后，你能把「本轮 agent 跑完」到「下一轮模型看到 <memory>」串成一条链。配置项、存储路径与 token 预算见 参考页。

1. 两条 middleware 各管一半

memory 不是同步写进当轮 prompt 的。MemoryMiddleware（链上约第 19 位）在 after_agent 把对话入队；DynamicContextMiddleware（约第 12 位）在下一轮 before_agent 读取已落盘的 memory.json 并注入。因此本轮聊完 ≠ 本轮立刻记住，通常要等防抖窗口（默认 30s）加 LLM 抽取完成后，下一次用户发言才会带上新 memory。

2. 入队：过滤、捕获身份、防抖合并

MemoryMiddleware.after_agent 先检查 memory.enabled，再用 filter_messages_for_memory 只保留用户输入与最终 AI 回复（跳过 tool call 中间态）。至少一对 human+ai 才入队。入队时显式捕获 user_id 与 deerflow_trace_id——因为后续 threading.Timer 线程拿不到 ContextVar。get_memory_queue().add() 按 (thread_id, user_id, agent_name) 去重：同线程在防抖期内多次回合只保留最新一批消息。

3. 异步更新：Timer → updater → 原子写盘

agents/memory/queue.py 的 MemoryUpdateQueue 在 debounce_seconds 后调用 MemoryUpdater.update_memory。updater 用独立 sync LLM 路径抽取 user 摘要与 facts，经 storage.save 以临时文件 + rename 写入 users/{user_id}/memory.json（或 per-agent 路径）。整个过程不阻塞当轮 SSE 流式返回。

4. 注入：首条 HumanMessage 的 ID-swap

下一轮 DynamicContextMiddleware._inject 通过 _get_memory_context 加载 memory，格式化为 <memory>...</memory>。首条真实用户消息会被拆成三条：带日期的 SystemMessage（框架数据）、隐藏的 memory HumanMessage（用户数据，防 OWASP LLM01）、以及 {id}__user 的原始用户文本。日期与 memory 分离后，system prompt 前缀可缓存复用。若 tiktoken 冷启动超时（5s），本回合会跳过注入而非挂死请求。

回合 N（用户发消息） │ ├── DynamicContextMiddleware.before_agent ← 读上一轮已写好的 memory.json ├── … lead agent 执行 … └── MemoryMiddleware.after_agent filter → queue.add(user_id, messages) → Timer(debounce) debounce 到期（后台线程） └── MemoryUpdater → LLM 抽取 → atomic save memory.json 回合 N+1 └── DynamicContextMiddleware 注入更新后的 <memory>
检索练习：用户在回合 N 纠正了 agent 的错误，希望尽快写入 memory。下面哪项一定为真？ A. 回合 N 的模型调用已包含新 facts B. MemoryMiddleware 在 before_agent 运行 C. 回合 N+1 的注入可能仍用旧 memory，若防抖与 LLM 尚未完成 D. user_id 在 Timer 线程里由 ContextVar 自动恢复
参考答案：C。更新是异步的；注入只读已落盘文件。A 错——当轮不注入新抽取结果。B 错——MemoryMiddleware 在 after_agent。D 错——user_id 必须在入队时显式写入 ConversationContext。
