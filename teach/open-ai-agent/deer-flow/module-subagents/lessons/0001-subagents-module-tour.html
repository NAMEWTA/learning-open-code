Subagent 系统：task tool 怎样生成可追踪 step

目标：15 分钟内看懂一条主链路。lead agent 发出一次 task tool call 后，deer-flow 怎样启动 subagent、直播进度、持久化 step，并让前端 subtask card 能刷新后回看。

1. 边界与入口

在 L0 总览里，subagent delegation 是长任务的核心能力；runtime/persistence 已说明 subagent step 会进入 run event store；frontend workspace 已说明 subtask card 是用户可见入口。本模块位于 lead agent 和 runtime 之间：它不决定“该不该做这个任务”，而是把已经发生的 task 调用变成隔离执行、可追踪、可恢复展示的子任务。

最短入口是 backend/packages/harness/deerflow/tools/builtins/task_tool.py。模型给出的调用形状大致如下：

{
  "name": "task",
  "id": "call_123",
  "args": {
    "description": "调查 run events",
    "prompt": "检查 subagent step 如何持久化",
    "subagent_type": "general-purpose"
  }
}

tool_call_id 会被复用为 task_id。这是一条很重要的线索：后端事件、ToolMessage 元数据、delegation ledger、前端卡片都靠同一个 id 对齐。

2. 委托变成后台任务

task_tool 先查 get_subagent_config()，未知类型会直接返回 failed ToolMessage；bash 还要通过 host bash 安全开关。之后它从 runtime 继承 sandbox、thread、user、run、trace、tool groups、skills allowlist，并调用 get_available_tools(subagent_enabled=False)。这个参数防止 subagent 再拿到 task 工具形成递归委托。

真正执行由 SubagentExecutor.execute_async(prompt, task_id=tool_call_id) 启动：executor 在后台线程池登记 SubagentResult，在隔离事件循环中跑 LangGraph agent，并用 capture_new_step_messages() 抓取 step。单次模型响应内并行 task 调用数由 SubagentLimitMiddleware 限制（默认最多 3，范围 2–4）；超出会被截断。详见 reference 入口表。

3. step 如何可追踪

task_tool 不等整条 run 结束才报告。它每 5 秒轮询后台结果，把生命周期写成自定义 stream chunk：

task_started   -> subagent.start  -> 卡片出现
task_running   -> subagent.step   -> 追加 AI/tool step
task_completed -> subagent.end    -> 卡片完成并显示结果
task_failed    -> subagent.end    -> 卡片失败并显示错误

映射逻辑在 backend/packages/harness/deerflow/subagents/step_events.py。build_subagent_step() 把消息压成 {kind, text, tool_calls/tool_name, message_index}，并限制大文本长度。runtime worker 的 _SubagentEventBuffer 再批量写入 RunEventStore。前端直播时消费 task_running；刷新后，subtask-card.tsx 展开卡片会调用 fetchSubtaskSteps(threadId, runId, taskId)，按 event_types=subagent.step 和 task_id 回填。

主参考页在 subagents-overview.html。长状态契约、executor 责任表、事件表和测试矩阵都放在那里。

4. 检索练习

暂停 60 秒，先从记忆回答：如果用户刷新页面后 subtask card 能看到卡片标题，但展开后没有 step，你会优先查哪三个边界？如果 lead agent 重复委托同一件事，应该先看哪份 durable 上下文？

参考答案：先查 step_events.py 是否生成 subagent.step，再查 worker.py 的 `_SubagentEventBuffer` 是否写入 run event store，最后查 frontend/src/core/tasks/api.ts 是否用正确 threadId/runId/taskId 回填。重复委托先看 delegation_ledger.py 渲染的 “Work already delegated”。

主要来源建议先读 backend/packages/harness/deerflow/tools/builtins/task_tool.py，再顺着 SubagentExecutor 和 step_events.py 追一次。下一步可追问：“帮我只讲 max_turns_reached 这条异常路径”。
