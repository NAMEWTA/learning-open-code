runs 执行模块边界导览

学习目标：15 分钟内能判断一次 `subagent` 调用进入 foreground、background 还是 shared 支撑层。

1. 一句话职责

runs 目录是 `pi-subagents` 的执行内核：它把父 Pi 会话发来的结构化调用，分流成前台单次/并行/链式运行，或后台 detached async run，并把结果、进度、状态、日志和恢复入口统一整理给父会话。L0 已经说明 extension 入口负责注册工具，agent discovery 负责找角色；本课只看执行层边界，不进入每个 helper 的细节。

2. 三层分工

src/runs/
├── foreground/        ← 当前 turn 内等待结果
│   ├── subagent-executor.ts
│   ├── execution.ts
│   └── chain-execution.ts
├── background/        ← detached runner 与状态观测
│   ├── async-execution.ts
│   ├── run-status.ts
│   └── wait.ts
└── shared/            ← 两边复用的运行工具

前台层处理“现在就把结果带回当前对话”。open-ai-agent/pi-subagents/src/runs/foreground/subagent-executor.ts 是门卫：它先识别 management action，再检查递归深度、spawn 限额、输入模式、model scope、预算、session/artifact 路径，然后进入 single、parallel、chain 或 async。open-ai-agent/pi-subagents/src/runs/foreground/execution.ts 才是真正的单 child Pi spawn；它解析 JSONL、累计 usage、更新 progress，并生成 SingleResult。后台层处理“先启动，稍后观察”：open-ai-agent/pi-subagents/src/runs/background/async-execution.ts 生成 runner config，用 `jiti` 启动 detached `subagent-runner.ts`，并发出 async started event。shared 层不拥有流程，它提供 model fallback、workflow graph、worktree、output、budget、control 等通用积木。

3. 一个调用怎么分流

{
  "agent": "reviewer",
  "task": "Review src/runs/foreground/execution.ts",
  "async": false,
  "maxRuntimeMs": 300000
}

这个例子没有 action，没有 tasks，也没有 chain，所以 executor 视为 single foreground run。若改成 tasks: [...]，它会进入 top-level parallel；若改成 chain: [...]，它会进入 chain；若 async: true 或配置强制 top-level async，则前台不会调用 runSync，而是转给后台启动器，返回 run id、async dir 和“不要 sleep-poll，必要时用 wait”的提示。

判断口诀：先看 action，有 action 就是管理/观测路径；再看 async，为真则尝试后台；最后用 chain、tasks、agent 三选一决定 chain、parallel、single。真正的长清单见 runs execution 参考页。

4. 测试佐证与下一步

这个边界不是注释猜出来的。test/integration/single-execution.test.ts 验证 single alias、未知 action、重复调度防护、spawn 限额和前台输出；test/integration/parallel-execution.test.ts 验证并发、顺序保持、输出文件和 timeout；test/integration/chain-execution.test.ts 验证 `{previous}`、命名输出、parallel step、dynamic fanout 与 workflow graph；test/integration/async-execution.test.ts 验证 async runner、status 文件、后台事件、timeout、control state 和 live output。

检索练习：如果一次调用传入 { "chain": [...], "async": true }，但用户想看即时结果，你应该优先检查哪两个位置？

参考答案：先查 executor 是否把 effectiveAsync 算成 true，再查后台 open-ai-agent/pi-subagents/src/runs/background/async-execution.ts 是否成功创建 async dir 并发出 started event。即时结果不会来自前台 runSync。

主要来源：open-ai-agent/pi-subagents/src/runs/foreground/subagent-executor.ts、open-ai-agent/pi-subagents/src/runs/foreground/execution.ts、open-ai-agent/pi-subagents/src/runs/background/async-execution.ts。下一节 L2 建议从“single foreground run 垂直切片”开始；不清楚某个分流条件时，可以继续让 agent 顺着该源码做逐段讲解。
