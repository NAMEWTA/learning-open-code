runs execution 辅助 API 参考（L3）

本页补全 runs 模块中尚未在 L1 图谱展开的辅助源文件：精确到公共导出、调用链与测试锚点。模块地图见 runs-execution-overview.html；短课入口 0001-runs-execution-module-tour.html。

L3 只收录公共导出与边界行为；实现细节留在源码。路径前缀均为 open-ai-agent/pi-subagents/src/runs/。

background 辅助层

chain-root-attachment.ts

background/chain-root-attachment.ts — 处理 chain step 中附着已有 async root（importAsyncRoot）的等待与结果归一化。当父 chain 不重新 spawn 子进程，而是引用先前已启动的后台 run 时，本模块轮询其 status.json 与 result JSON，在终态或超时后产出与前台 SingleResult 兼容的 ImportedAsyncRootResult。

waitForImportedAsyncRoot(root, options?) → Promise<ImportedAsyncRootResult>
resolveAsyncRootResultPath(resultsDir, runId) → string

主要类型：ImportedAsyncRoot（runId、asyncDir、resultPath、index）、ImportedAsyncRootResult（agent、output、success、exitCode、acceptance、structuredOutput 等可选字段）。调用方：subagent-runner.ts 在 chain step 执行路径等待附着 root；subagent-executor.ts 用 resolveAsyncRootResultPath 解析 result 文件路径；async-execution.ts 引用 ImportedAsyncRoot 类型。测试 test/unit/chain-root-attachment.test.ts

completion-batcher.ts

background/completion-batcher.ts — 后台任务成功完成通知的智能批处理。在短窗口内将多个 sibling job 的 completion 合并为一条分组消息；从首条入队起有硬上限 maxWaitMs 防止无限等待；已发出分组后，迟到的 sibling 进入更短的 straggler 窗口。失败与 needs-attention 信号不经过 batcher，由调用方 flush() 后立即发出。

createCompletionBatcher<T>(options) → CompletionBatcher<T>
resolveCompletionBatchConfig(global?, override?) → ResolvedCompletionBatchConfig

主要导出：DEFAULT_COMPLETION_BATCH_CONFIG、CompletionBatchConfig（re-export）、CompletionBatcher 接口（push / flush / dispose）。调用方：notify.ts 为每个 session/cwd 分组创建 batcher；extension 通过 config.completionBatch 传入覆盖配置。测试 test/unit/completion-batcher.test.ts

completion-dedupe.ts

background/completion-dedupe.ts — completion 事件的去重键生成与 TTL 记忆。避免同一 async 完成被 result-watcher 与 notify 双重投递，或在 extension 热重载后重复触发 UI 消息。

buildCompletionKey(data, fallback) → string
markSeenWithTtl(seen, key, now, ttlMs) → boolean
getGlobalSeenMap(storeKey) → Map<string, number>

键策略：优先用 id；否则组合 sessionId、agent、timestamp、taskIndex、success 等元数据。调用方：notify.ts、result-watcher.ts。测试 test/unit/completion-dedupe.test.ts

notify.ts

background/notify.ts — 向 Pi 父会话发送后台 subagent 完成通知。监听 SUBAGENT_ASYNC_COMPLETE_EVENT，仅处理与 state.currentSessionId 匹配的 result；成功 completion 经 batcher 分组，失败/暂停立即 sendMessage（customType: "subagent-notify"，triggerTurn: true）。

registerSubagentNotify(pi, state, options?) → void（default export）
formatSingleCompletion(details) → string
formatGroupedCompletion(details[]) → string
buildCompletionDetails(result) → SubagentNotifyDetails

调用方：extension/index.ts 在 extension 初始化时注册，传入 batchConfig: config.completionBatch。热重载时通过 global store 清理旧 unsubscribe 与 batcher。测试 test/unit/notify.test.ts

parallel-groups.ts

background/parallel-groups.ts — async chain 中扁平 step 索引与逻辑 chain step 索引的映射。parallel group 在 status 里记录 { start, count, stepIndex }，本模块校验边界后排序，并提供 flatToLogicalStepIndex 供 status/fleet/TUI 正确标注「第几步」。

normalizeParallelGroups(groups, stepCount, chainStepCount) → AsyncParallelGroupStatus[]
flatToLogicalStepIndex(flatIndex, chainStepCount, groups) → number

调用方：async-job-tracker.ts、async-status.ts、run-status.ts、stale-run-reconciler.ts；tui/render.ts 用 flatToLogicalStepIndex 渲染 widget。测试 间接覆盖于 test/integration/chain-execution.test.ts、test/integration/async-execution.test.ts

shared 辅助层

acceptance.ts

shared/acceptance.ts — subagent 运行的验收契约（Acceptance Contract）全链路：从 agent/task 推断 level（attested → checked → verified → reviewed）、校验 frontmatter 输入、生成 prompt 段落、解析子 agent 输出的结构化 acceptance report、执行 verify 命令与可选 reviewer 子运行，最终写入 AcceptanceLedger 并可能将 run 标记为失败。

导出职责
normalizeAcceptanceInput / validateAcceptanceInput将 false、字符串 level 或对象规范为 AcceptanceConfig；返回校验错误列表。
resolveEffectiveAcceptance合并显式配置与按 agent/task/mode 推断的 level、criteria、evidence、review。
formatAcceptancePrompt注入子 agent system/task 的 Acceptance Contract 文本块。
parseAcceptanceReport / stripAcceptanceReport从输出中解析/剥离 JSON acceptance report。
aggregateAcceptanceReportchain/parallel group 多 step 验收结果聚合。
evaluateAcceptance异步执行 verify 命令、可选 reviewer spawn，返回 AcceptanceLedger。
acceptanceFailureMessage将 ledger 转为对用户可见的失败说明。

调用方：foreground/execution.ts、foreground/chain-execution.ts、background/subagent-runner.ts、background/async-execution.ts；subagent-executor.ts 与 agents/chain-serializer.ts、slash/slash-commands.ts 做输入校验。测试 test/unit/acceptance.test.ts

completion-guard.ts

shared/completion-guard.ts — 实现类任务却零变更的完成守卫。根据 agent 名、task 文案、可用 tools 推断是否「应发生变更」（expectsImplementationMutation），再扫描 assistant toolCall 是否出现 edit/write/变更型 bash（hasMutationToolCall）。若预期变更但未尝试，evaluateCompletionMutationGuard 返回 triggered: true，运行层可将 exit 视为失败或触发 model fallback。

expectsImplementationMutation(agent, task) → boolean
hasMutationToolCall(messages) → boolean
evaluateCompletionMutationGuard(input) → { expectedMutation, attemptedMutation, triggered }

调用方：foreground/execution.ts、background/subagent-runner.ts。测试 test/unit/completion-guard.test.ts；test/integration/async-execution.test.ts（async-completion-guard-optout 场景）

long-running-guard.ts

shared/long-running-guard.ts — 长运行主动通知与变更型 tool 失败升级的纯函数库。根据 ResolvedControlConfig 与运行指标判断何时触发 active_long_running；识别 mutating bash/edit/write；记录连续 mutating 失败并在达到阈值时建议 needs_attention。

isMutatingBashCommand(command) → boolean
isMutatingTool(toolName, args) → boolean
nextLongRunningTrigger(config, metrics) → triggerReason | undefined
createMutatingFailureState / recordMutatingFailure / shouldEscalateMutatingFailures

调用方：foreground/execution.ts、background/subagent-runner.ts；completion-guard.ts 复用 isMutatingBashCommand。测试 test/unit/subagent-control.test.ts（nextLongRunningTrigger）；test/integration/single-execution.test.ts（controlConfig 通知阈值）

mcp-direct-tool-allowlist.ts

shared/mcp-direct-tool-allowlist.ts — 将 agent 配置中的 mcpDirectTools 解析为可直接暴露给子 Pi 的 builtin tool 名列表。合并项目/全局 mcp.json、可选 import（Cursor/Claude/Codex 等），读取 mcp-cache.json 元数据，按 server 的 directTools 与 allowlist 过滤。

resolveMcpDirectToolNames(mcpDirectTools?, cwd?) → string[]
computeMcpServerHash(definition) → string

调用方：shared/pi-args.ts 在构造子进程 argv 时把解析结果并入 builtinTools。测试 test/unit/pi-args.test.ts（computeMcpServerHash）

nested-events.ts

shared/nested-events.ts — 嵌套 subagent 运行的事件总线与注册表。子 run 通过 env（nestedRouteEnv）向 nested-subagent-events/ 写入 JSONL 事件；父 run 投影为 NestedRegistry / NestedRunSummary 树，供 status、fleet、foreground live control 展示；支持嵌套 interrupt/resume 的 control-request/result 文件协议。

核心导出（节选）：NESTED_EVENTS_DIR；createNestedRoute、resolveNestedRouteFromEnv、writeNestedEvent、projectNestedEvents、findNestedRunById、readNestedControlRequests、writeNestedControlResult；updateAsyncJobNestedProjection、updateForegroundNestedProjection、nestedSummaryFromAsyncStatus、attachRootChildrenToSteps。

调用方：subagent-executor.ts、async-job-tracker.ts、async-status.ts、run-status.ts、fleet-view.ts、result-watcher.ts、run-id-resolver.ts、extension/fanout-child.ts。测试 test/unit/nested-events.test.ts、test/unit/nested-control.test.ts、test/integration/result-watcher.test.ts

nested-path.ts

shared/nested-path.ts — 嵌套路径 env 的编解码与消毒。NestedPathEntry 为 { runId, stepIndex?, agent? } 数组（最多 4 段），防止路径穿越与超长 ID；供 pi-args.ts 与子 run 继承父路径。

sanitizeNestedPath(value) → NestedPathEntry[]
parseNestedPathEnv(value?) / encodeNestedPathEnv(value) → string
isSafeNestedPathId(value) → boolean

调用方：nested-events.ts、shared/pi-args.ts。测试 间接覆盖于 test/unit/nested-events.test.ts、test/unit/pi-args.test.ts

nested-render.ts

shared/nested-render.ts — 将 NestedRunSummary 树格式化为status/fleet 文本行。递归统计 running/paused/failed/complete/queued（countNestedRuns、formatNestedAggregate），按深度与行数上限缩进展示，可选附加 subagent({ action: "status", id }) 命令提示。

formatNestedRunStatusLines(children, options?) → string[]
formatNestedAggregate(children) → string | undefined

调用方：run-status.ts、async-status.ts、fleet-view.ts、foreground/subagent-executor.ts；tui/render.ts 使用 formatNestedAggregate。测试 间接覆盖于 integration status/fleet 场景与 test/integration/result-watcher.test.ts

run-history.ts

shared/run-history.ts — 按 agent 名记录本地运行历史（~/.pi/.../run-history.jsonl）。每次 foreground/chain 完成后 append 一条 RunEntry；超过 1200 行时截断保留最近 1000 行。失败不影响主执行流。

recordRun(agent, task, exitCode, durationMs) → void
loadRunsForAgent(agent) → RunEntry[]

调用方：foreground/subagent-executor.ts、foreground/chain-execution.ts（parallel/chain step 完成后）。测试 test/unit/pi-coding-agent-dir.test.ts

single-output.ts

shared/single-output.ts — 单 agent 运行的输出文件契约：解析 output frontmatter（含 false 禁用）、向 task/system prompt 注入权威路径说明、运行前后快照 mtime、将 child 文本写入或读取文件、在 file-only 模式下只返回引用消息。

normalizeSingleOutputOverride / resolveSingleOutputPath / injectSingleOutputInstruction / injectOutputPathSystemPrompt
captureSingleOutputSnapshot / resolveSingleOutput / finalizeSingleOutput
formatSavedOutputReference / validateFileOnlyOutputMode

调用方：foreground/execution.ts、foreground/subagent-executor.ts、foreground/chain-execution.ts、background/async-execution.ts、background/subagent-runner.ts。测试 test/unit/single-output.test.ts；test/integration/single-execution.test.ts（single-output-override-system-prompt、failed-single-output）

structured-output.ts

shared/structured-output.ts — 带 outputSchema 的 step 要求子 agent 以 structured_output tool 结束。在临时目录写入 schema JSON，通过 env（PI_SUBAGENT_STRUCTURED_OUTPUT_SCHEMA、PI_SUBAGENT_STRUCTURED_OUTPUT_CAPTURE）传给子进程，运行后用 TypeBox 校验 output.json。

createStructuredOutputRuntime(schema, baseDir?) → StructuredOutputRuntime
validateStructuredOutputValue(schema, value) → valid | invalid
readStructuredOutput(runtime) → { value? | error? }
cleanupStructuredOutputRuntime(runtime) → void

调用方：foreground/execution.ts、foreground/chain-execution.ts、background/async-execution.ts、background/subagent-runner.ts；shared/dynamic-fanout.ts、subagent-prompt-runtime.ts、slash/slash-commands.ts。测试 test/unit/subagent-prompt-runtime.test.ts；test/integration/async-execution.test.ts（structured output 场景）

跨文件协作速查

用户可见行为涉及辅助模块
多个后台任务几乎同时完成，父会话只收到一条分组通知completion-batcher.ts + notify.ts + completion-dedupe.ts
chain 引用已有 async run 作为一步chain-root-attachment.ts + parallel-groups.ts
worker 声称完成但未 edit/writecompletion-guard.ts + long-running-guard.ts
status 显示嵌套子 run 树与 +N nested runsnested-events.ts + nested-render.ts + nested-path.ts
验收失败导致 exitCode≠0 或附加 acceptance 字段acceptance.ts
output 写入指定文件而非 inline 返回single-output.ts
outputSchema 强制 JSON 终态structured-output.ts
MCP 工具直接暴露给子 agentmcp-direct-tool-allowlist.ts → pi-args.ts
