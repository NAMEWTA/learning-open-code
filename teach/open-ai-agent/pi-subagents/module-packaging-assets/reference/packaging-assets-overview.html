包装与资产交付速查

本页是 L1 短课的参考资料，适合回查 npm 字段、安装路径、内置资产清单和 CI 发布矩阵。短课只讲判断路径；本页保存完整索引。

1. 模块定位

问题结论主要证据
一句话职责仓库根部把 extension 实现、内置 agents、orchestrator skill、prompt 模板和安装器打包成可发布的 npm Pi 包。open-ai-agent/pi-subagents/package.json
架构位置位于 src/ 执行层之上，是交付边界而非运行时编排层；Pi 通过 pi 块发现能力。open-ai-agent/pi-subagents/README.md
与 L0 的衔接L0 目录地图列出 agents/、prompts/、skills/；本模块解释它们如何进入安装包。L0 项目整体架构与学习地图
用户安装 ├─ pi install npm:pi-subagents → Pi 读 package.json 的 pi 块 └─ npx pi-subagents → install.mjs 克隆到 ~/.pi/agent/extensions/subagent ↓ Pi 运行时加载 ├─ extension: src/extension/index.ts ├─ skills: skills/pi-subagents/SKILL.md ├─ prompts: prompts/*.md └─ agents: agents/*.md（builtin 发现）

2. package.json 关键字段

字段值 / 行为说明
namepi-subagentsnpm 包名；pi install npm:pi-subagents 对应此名。
version当前 0.34.0与 CHANGELOG 和 npm registry 对齐；发布由 release workflow 触发。
typemoduleESM 包；install.mjs 和 TS 入口均按 module 解析。
filessrc/**/*.ts、*.mjs、agents/、skills/**/*、prompts/**/*、文档npm 发布白名单；test/、.github/ 等不进入包。
pi.extensions["./src/extension/index.ts"]Pi extension 入口；注册 subagent/wait 工具与 slash/RPC。
pi.skills["./skills"]技能目录；当前含 pi-subagents orchestrator skill。
pi.prompts["./prompts"]prompt 模板目录；供 slash / prompt-workflow 发现。
binpi-subagents → install.mjsnpx pi-subagents 的 CLI 入口。
peerDependencies@earendil-works/pi-*（optional）声明 Pi 运行时兼容；devDependencies 锁定测试用版本。
dependenciespi-tui、jiti、typeboxextension 运行时直接依赖，随 npm 包安装。
scripts.test:allunit → integration → e2e 串行release workflow 发布前执行。

3. install.mjs 行为矩阵

命令行为目标路径
npx pi-subagents若目录不存在则 git clone；若已是 git repo 则 git pull；否则报错要求先 --remove。~/.pi/agent/extensions/subagent
npx pi-subagents --remove递归删除安装目录。同上
npx pi-subagents --help打印用法与安装目录。—
pi install npm:pi-subagents由 Pi 包管理器处理；不经过 install.mjs。Pi 管理的 extension 目录

4. 内置资产清单

skills/

路径受众职责摘要
skills/pi-subagents/SKILL.md父 orchestrator only委派时机、async 默认、review-loop / parallel-review 等编排模式、builtin agent 表、discovery 规则、intercom 约束。明确禁止普通子 agent 注入此 skill。

prompts/

文件description工作流形状
parallel-review.mdParallel subagents review多 fresh-context reviewer 并行 + 父会话合成。
review-loop.mdReview/fix loop until cleanworker → reviewer → fix-worker 循环，默认最多 3 轮。
parallel-research.md—researcher + scout 并行，外部证据与本地代码上下文。
parallel-context-build.md—chain 内 parallel context-builder，产出 handoff 文件。
parallel-handoff-plan.md—外部研究 + 本地 context + 合成 handoff plan。
gather-context-and-clarify.md—scout/research 后由父 agent 用 interview 澄清。
parallel-cleanup.md—deslop + verbosity 双 reviewer 清理审查。

agents/

8 个内置角色（scout、planner、worker、reviewer、context-builder、researcher、delegate、oracle）随 files 发布，由 agent discovery 以最低优先级加载。详见 agent-system-overview。

5. CI 与发布

Workflow触发环境步骤
test.ymlpush/PR → mainUbuntu + Windows，Node 24npm ci --ignore-scripts → npm test → test:integration → test:e2e
release.ymlworkflow_dispatch（手动）Ubuntu，Node 24npm ci → npm run test:all → npm publish --provenance

release.yml 声明 id-token: write 以支持 npm provenance；concurrency 按 ref 分组且 cancel-in-progress: false，避免并发发布互相取消。
