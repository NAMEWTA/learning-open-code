包装与资产交付模块导览

学习目标：15 分钟内能说清 pi-subagents 如何把 extension、skills、prompts 和内置 agents 一起装到 Pi 里。

1. 一句话职责

仓库根部的包装层不负责 subagent 怎么跑，而负责把可运行能力打成可安装包。依据 open-ai-agent/pi-subagents/package.json，npm 包同时交付 TypeScript extension 入口、8 个内置 agent、父 orchestrator skill、7 个 prompt 工作流模板，以及 README/CHANGELOG。L0 的目录地图里这些目录各自独立；本课要回答的是：它们如何被 pi 块和 files 字段绑进同一个发布单元。

pi-subagents（npm 包） ├── package.json ← pi.extensions / pi.skills / pi.prompts ├── install.mjs ← npx 安装器（bin 入口） ├── src/extension/ ← extension 实现（随 files 发布） ├── agents/ ← 8 个内置角色 ├── skills/pi-subagents/ ← 父 orchestrator skill └── prompts/ ← 7 个工作流 prompt 模板

2. package.json 的三根支柱

第一根是发布白名单：files 只包含 src/**/*.ts、agents/、skills/**/*、prompts/**/* 和文档，避免把测试夹具或开发配置打进 npm 包。第二根是Pi 发现声明：pi.extensions 指向 ./src/extension/index.ts，pi.skills 指向 ./skills，pi.prompts 指向 ./prompts——Pi 安装 npm 包时按这三条路径加载能力。第三根是可选 CLI 安装器：bin.pi-subagents 映射到 install.mjs，供 npx pi-subagents 使用。

"pi": {
  "extensions": ["./src/extension/index.ts"],
  "skills": ["./skills"],
  "prompts": ["./prompts"]
},
"bin": { "pi-subagents": "install.mjs" },
"files": [
  "src/**/*.ts", "agents/", "skills/**/*", "prompts/**/*", "README.md"
]

peerDependencies 声明对 @earendil-works/pi-coding-agent 等 Pi 运行时的依赖，但标记为 optional——extension 在 Pi 进程内运行，npm 包本身只打包自己的 TS 与资产。

3. 两条安装路径与资产角色

路径 A（推荐）：README 写的 pi install npm:pi-subagents。Pi 读取已安装 npm 包里的 pi 块，注册 extension、skills 和 prompts。路径 B（备选）：npx pi-subagents 执行 install.mjs，把 GitHub 仓库克隆到 ~/.pi/agent/extensions/subagent，已存在则 git pull 更新。两条路径最终都让 Pi 看到同一套 extension + 资产，但 B 绕过 npm、直接维护 git 工作副本。

资产分工上，skills/pi-subagents/SKILL.md 只给父 orchestrator：规定何时委派、async 默认、review-loop 等编排纪律，并明确子 agent 不应注入此 skill。prompts/*.md 则是可复用的工作流食谱（如 parallel-review.md、review-loop.md），frontmatter 只有 description，正文告诉父 agent 如何调用 subagent(...) 完成固定编排。slash 命令层可以引用这些 prompt，父 agent 也可直接按 skill 里的「Prompt Techniques」套用同样模式。

4. 发布闸门与回忆练习

CI 把「能发布」和「能合并」分开守门。.github/workflows/test.yml 在 push/PR 到 main 时，于 Ubuntu 与 Windows 双矩阵跑 npm test、test:integration、test:e2e。.github/workflows/release.yml 仅手动触发：先 npm run test:all，再 npm publish --provenance 发到 npm registry。也就是说，发布前测试覆盖面比日常 PR 更严（test:all 串行跑三层）。

暂停回忆：用户说「装了 pi-subagents 但看不到 parallel-review 工作流」，你会先查 extension 注册、skill 注入，还是 prompt 交付？

参考答案：先查 prompt 交付链——pi.prompts 是否被 Pi 加载、prompts/parallel-review.md 是否在安装目录里。extension 未注册会导致整个包不可用；skill 只影响父 agent 编排建议，不决定 prompt 文件是否存在。完整资产清单与 CI 步骤见 packaging-assets-overview 参考页。
