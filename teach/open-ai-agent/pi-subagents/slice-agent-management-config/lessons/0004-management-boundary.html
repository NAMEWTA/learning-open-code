管理写路径与边界

学习目标：预判 disable / enable / reset / eject 在跨 scope 与只读来源下的结果。

1. 入口与 config

subagent-executor 识别管理类 action 后调用 handleManagementAction，并把 loadConfig() 读到的 extension 配置（路径：~/.pi/agent/extensions/subagent/config.json）传给 list 的 proactive skill 建议。这与 project settings 是不同层：前者管 extension 行为，后者管 agent 覆盖。

2. 只读与可写边界

builtin 与 package agent 不可直接 update/delete；改行为用 settings override，完整接管用 eject 复制到 user/project。create/update 要求 JSON config；字符串 config 解析失败返回 config must be valid JSON（见 agent-management.test.ts）。

3. disable 失败路径

handleDisable 用 pickEffectiveAgent 找最高优先级同名 agent，向指定 scope 的 settings 写入 disabled: true。若写入后 agent 仍 enabled，返回 isError 并提示更高优先级 override 可能在 project scope——应改 agentScope 或先处理冲突的 custom 文件。

export function handleManagementAction(action, params, ctx) {
  switch (action) {
    case "list": return handleList(params, ctx);
    case "disable": return handleDisable(params, ctx);
    case "reset": return handleReset(params, ctx);
    // …
    default: return result(`Unknown action: ${action}`, true);
  }
}

4. 检索练习

判断题：对纯 user 自定义 agent（无 bundled 默认）调用 reset，系统会删除该文件吗？

参考答案：不会。reset 只恢复有 bundled/package 默认的角色；纯 custom agent 应使用 delete。eject 在目标 scope 已有同名 agent 或 chain 时会拒绝，避免静默覆盖。
