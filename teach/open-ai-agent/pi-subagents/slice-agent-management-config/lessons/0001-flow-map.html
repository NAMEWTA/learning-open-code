Agent 管理、覆盖与配置解析全链路

学习目标：15 分钟内画出从 subagent 工具到最终 AgentConfig 的两条主路径。

1. 切片边界

本 L2 切片回答一个问题：某个 agent 名称最终解析成什么运行时配置，以及管理动作如何读写它。L1 模块总览已介绍 agents 模块职责；这里从调用方视角串起入口、解析、覆盖与合并。涉及源码：subagent-executor.ts、agent-management.ts、agents.ts、frontmatter.ts、agent-selection.ts、extension/config.ts。

交叉链接：L1 Agent 发现与配置模块 · 本切片速查参考

2. 两条主路径

路径 A — 执行发现（run / chain 选 agent） subagent tool { action: "run", agent: "reviewer" } → discoverAgents(cwd, scope) → 读 settings + 扫描目录 + parseFrontmatter → apply defaultModel / agentOverrides → mergeAgentsForScope → filter disabled → AgentConfig[] 路径 B — 管理面（list / get / create / disable …） subagent tool { action: "list" } → handleManagementAction → discoverAgentsAll（含 disabled） → 读/写 user|project agent 文件或 settings.json → loadConfig() 提供 extension 级选项（如 proactive skills）

执行路径隐藏 disabled agent；管理路径保留它们，以便 enable/reset。完整时序与测试索引见参考页 §1–2。

3. 优先级一句话

同名 agent：builtin < package < user < project（mergeAgentsForScope）。settings 另起一层：project settings 优先于 user；subagents.defaultModel 只填缺省 model；agentOverrides 对 builtin 直接覆盖，对 custom agent 只填 frontmatter 未声明的字段。下一节短课分别展开frontmatter 解析与settings 覆盖。

4. 检索练习

判断题：用户在 project .pi/settings.json 写入 agentOverrides.reviewer.disabled: true，但未删除 project 下同名 reviewer.md。执行 discoverAgents(cwd, "both") 时，reviewer 是否一定不可见？

参考答案：不一定。若 project 自定义 reviewer.md 的 frontmatter 未声明 disabled，override 会把 disabled: true 填上，执行路径会过滤掉它；但若 frontmatter 显式写了 disabled: false，custom override 不会覆盖该字段，agent 仍可见。管理面 disable 若被更高优先级 override 挡住，会返回 isError 提示切换 agentScope。详见0004 管理写路径与边界。

主要来源：Agent 管理配置全链路参考。不清楚某一步时，可让 agent 按「执行发现」或「管理动作」任一方向展开。
