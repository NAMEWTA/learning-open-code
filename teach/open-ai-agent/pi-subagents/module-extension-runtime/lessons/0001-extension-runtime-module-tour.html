扩展入口与 Pi runtime 注册模块

学习目标：15 分钟内能判断 src/extension 哪些代码负责注册到 Pi，哪些只是给注册层提供契约、描述或桥接。

1. 一句话职责

open-ai-agent/pi-subagents/src/extension 是 Pi runtime 适配层：它把已有的 subagent 执行能力注册成 Pi 可见的工具、消息渲染器、slash/RPC 桥、后台追踪器和 session 生命周期处理器，但不把 single、parallel、chain 的执行算法写在自己这里。L0 已经说明父会话如何组织子会话；本课只看入口层边界。

Pi 加载 extension：package.json 的 pi.extensions 配置值是 ./src/extension/index.ts；在本工作区中对应源码是 open-ai-agent/pi-subagents/src/extension/index.ts。
入口建立 runtime 状态：加载 config，创建 watcher、async tracker、scheduled manager、supervisor channel。
注册对外能力：subagent、wait、message renderer、slash bridge、prompt template bridge、RPC bridge。
执行下沉：真正运行任务时交给 createSubagentExecutor。

2. 读 index.ts 的主线

先抓住早退和注册两件事。普通 child 进程检测到 PI_SUBAGENT_CHILD 后直接返回，避免子会话再注册父级工具；fanout child 另由 fanout-child.ts 暴露受限工具。父入口随后清理旧 runtime、确保结果目录可访问、恢复 active jobs，并把同一个 executor 接给 tool、slash、prompt template 和 RPC。

export default function registerSubagentExtension(pi: ExtensionAPI): void {
  if (process.env[SUBAGENT_CHILD_ENV] === "1") return;

  const config = loadConfig();
  const state: SubagentState = { asyncJobs: new Map(), foregroundRuns: new Map(), ... };
  const supervisorChannel = createNativeSupervisorChannel(pi, state);
  const watcher = createResultWatcher(pi, state, RESULTS_DIR, 10 * 60 * 1000);
  watcher.startResultWatcher();
  watcher.primeExistingResults();

  const executor = createSubagentExecutor({ pi, state, config, discoverAgents, ... });
  pi.registerTool(tool);      // subagent
  pi.registerTool(waitTool);  // wait
}

这个摘录省略了细节，但保留了阅读顺序：先确认运行环境，再建共享状态，再注册入口。看到 executeSubagentCollapsed 时要注意，它只是先折叠 UI，再调用 executor；它不是新的执行模式。

3. 契约和桥接

schemas.ts 给 Pi provider 暴露参数形状。它的关键不是字段多，而是 keepTopLevelParameterDescriptions：保留顶层参数说明，删除嵌套 description，避免 provider payload 膨胀或出现不被接受的 schema 形状。

export const SubagentParams = keepTopLevelParameterDescriptions(SubagentParamsSchema);
export const WaitParams = keepTopLevelParameterDescriptions(WaitParamsSchema);

rpc.ts 是 in-process event bus 桥。外部扩展发 subagents:rpc:v1:request，它回到 subagents:rpc:v1:reply:<requestId>。最重要的约束是：RPC spawn 只能 detached async，不能打开 clarify UI，也不能混入管理动作。

events.emit(SUBAGENT_RPC_REQUEST_EVENT, {
  version: 1,
  requestId: "spawn-1",
  method: "spawn",
  params: { agent: "worker", task: "Do work" }
});
// rpc.ts 会补 async: true, clarify: false，再交给 executor。

4. 行为佐证与回忆练习

这一层有直接测试保护：open-ai-agent/pi-subagents/test/unit/index-child-registration.test.ts 验证 child 模式早退、fanout child 不双注册、wait tool 可被配置禁用；open-ai-agent/pi-subagents/test/unit/rpc.test.ts 验证 RPC ping、status、spawn、interrupt、stop；open-ai-agent/pi-subagents/test/unit/schemas.test.ts 验证 schema 描述裁剪；open-ai-agent/pi-subagents/test/unit/doctor.test.ts 验证诊断报告在失败环境下也继续输出。

检索练习：如果你想确认 slash、RPC、tool call 是否最终复用同一个执行器，应先看哪个文件？答案：先看 open-ai-agent/pi-subagents/src/extension/index.ts，找到 createSubagentExecutor 和各桥接函数传入的 execute 回调；再去 executor 追执行分流。
