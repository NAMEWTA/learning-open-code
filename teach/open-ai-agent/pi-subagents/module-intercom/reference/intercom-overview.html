通道生命周期

1. Bridge 激活

open-ai-agent/pi-subagents/src/intercom/intercom-bridge.ts 将 unknown mode 归一到 always，只接受 off、always、fork-only。bridge inactive 的原因只有三类：mode 为 off、fork-only 但当前不是 fork、没有 orchestrator target。active 后，agent 工具列表会被追加 intercom 和 contact_supervisor，系统提示会加上 Intercom orchestration channel: marker，重复调用保持幂等。

2. 子进程 metadata 注入

open-ai-agent/pi-subagents/src/runs/shared/pi-args.ts 在存在 orchestrator target、父 session id、run id 和 child agent name 时创建 channel 目录，并写入 SUBAGENT_SUPERVISOR_CHANNEL_DIR_ENV 等环境变量。open-ai-agent/pi-subagents/src/intercom/native-supervisor-channel.ts 的子端 client 只在这些 metadata 完整时注册。

3. 子端发请求

子端 registerNativeSupervisorClient 注册 contact_supervisor，可选注册 fallback intercom。请求被写成 subagent.supervisor.request JSON，包含 request id、reason、message、run id、agent、child index、父 session id。阻塞类请求会等待 subagent.supervisor.reply，超时或取消时清理 request 文件；progress update 只返回 queued。

4. 父端轮询与回复

open-ai-agent/pi-subagents/src/extension/index.ts 在 session_start 时启动 supervisor channel。父端轮询 request 文件，先按当前 session id 过滤，再排除 resolved、expired、inactive、missing 请求。pending request 会通过 pi.sendMessage 展示，父端工具支持 status、pending/list、reply。如果已有外部 intercom，native 模块保留 subagent_supervisor 作为不冲突的回复工具。

5. 完成结果回传

open-ai-agent/pi-subagents/src/intercom/result-intercom.ts 负责把 child results 归类为 completed、failed、paused、detached，生成 grouped message，并等待 subagent:result-intercom-delivery 确认。foreground 调用方在送达后返回 compact receipt，并移除 heavy output fields；async watcher 则从结果文件里读 intercomTarget，发送 async payload 和 revive 指引。

边界与失败模式

现象 先看哪里 判断点
子 agent 没有 contact_supervisor open-ai-agent/pi-subagents/src/intercom/intercom-bridge.ts bridge mode 是否 off；fork-only 是否运行在 fresh context；orchestrator target 是否为空；agent 工具是否显式限制但仍应追加 native 工具。
子端调用 contact_supervisor 直接失败 open-ai-agent/pi-subagents/src/runs/shared/pi-args.ts 与 open-ai-agent/pi-subagents/src/intercom/native-supervisor-channel.ts 环境变量是否包含 channelDir、runId、agent、childIndex、orchestratorSessionId。
父端没有看到 pending request open-ai-agent/pi-subagents/src/intercom/native-supervisor-channel.ts request 的 orchestratorSessionId 是否匹配当前 session；请求是否已 resolved、expired、inactive；channel 是否 stale empty 被清理。
结果没有 grouped receipt open-ai-agent/pi-subagents/src/intercom/result-intercom.ts 与 foreground/async 调用方 bridge 是否 active；delivery acknowledgement 是否收到；child 是否 detached；async result 是否有 intercomTarget。
receipt 有但普通 details 缺少完整输出 open-ai-agent/pi-subagents/src/intercom/result-intercom.ts 这是预期：完整输出在 grouped payload；receipt details 会移除 heavy output fields。

测试证据

测试文件 覆盖行为

open-ai-agent/pi-subagents/test/unit/result-intercom.test.ts 验证 grouped payload 状态计数、async revive 指引、缺失 session 时不可 revive、嵌套结果脱敏、compact receipt 列出 artifacts/session/intercom target、details 移除 heavy output。
open-ai-agent/pi-subagents/test/integration/intercom-result-delivery.test.ts 验证 foreground single/parallel/chain 只发一个 grouped event；bridge inactive 或 delivery 未确认时回退 legacy foreground output。
