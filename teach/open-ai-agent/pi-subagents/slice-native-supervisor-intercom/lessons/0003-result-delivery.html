L2-native-supervisor-intercom 短课 0003

完成结果 delivery 与 receipt

学习目标：能解释 foreground run 完成后 grouped payload 如何构建、如何等待 delivery acknowledgement，以及为何普通 tool result 会被压缩成 compact receipt。

1. 出口：executor 在返回前尝试 intercom

single、parallel、chain 三条前台路径在拿到 details.results 后，若子结果未 detached 且未 interrupted，会调用 maybeBuildForegroundIntercomReceipt。它先经 emitForegroundResultIntercom 把每个 child 转成 SubagentResultIntercomChild（含 status、summary、artifact、session、intercomTarget），再 buildSubagentResultIntercomPayload 生成 grouped message。

2. delivery acknowledgement：500ms 闸门

deliverSubagentResultIntercomEvent 向事件总线发出 subagent:result-intercom，并订阅 subagent:result-intercom-delivery，默认 500ms 内必须收到 delivered: true 才算成功。成功则返回 payload；失败则 maybeBuildForegroundIntercomReceipt 得到 null，executor 回退展示完整 legacy foreground output。

bridge.active && orchestratorTarget
  → buildPayload(to: orchestratorTarget)
  → emit + wait delivery ack (500ms)
      delivered → formatSubagentResultReceipt + stripDetailsOutputs
      not delivered → null → legacy details 原样返回

3. receipt 与 heavy output 剥离

delivery 成功后，父会话看到的 tool result 文本来自 formatSubagentResultReceipt：列出 run id、children 状态计数、artifacts、sessions、intercom targets，并注明 Full grouped output was sent over intercom。同时 stripDetailsOutputsForIntercomReceipt 从每个 SingleResult 移除 messages、finalOutput、truncation——完整内容在 grouped event 的 message 字段里，不在普通 details 里。

parallel 与 chain 模式只发一个 grouped event（见 integration test），避免父会话收到重复通知。

检索练习

bridge 为 always 且 orchestrator target 存在，但父会话仍看到完整 child output 而非 receipt。最可能的原因是？

  1. 子 agent 使用了 progress_update。
  2. 500ms 内没有 listener 确认 delivery，或 child 被标记 detached。
  3. 父 session 没有启动 supervisor channel poller。

参考答案：2。emitForegroundResultIntercom 在 delivery 失败或 detached children 为空时返回 null；poller 只影响运行中请求，不影响完成 delivery（见 intercom-result-delivery.test.ts）。

主要来源

result-intercom.ts 的 buildSubagentResultIntercomPayload、deliverSubagentResultIntercomEvent、formatSubagentResultReceipt；subagent-executor.ts 的 emitForegroundResultIntercom。
