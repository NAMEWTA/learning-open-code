L2-native-supervisor-intercom 短课 0001

native supervisor 与 intercom 协调全链路

学习目标：15 分钟内能口述一次前台 subagent run 从 bridge 注入到 result delivery 的五段链路，并指出每段对应源码文件。

1. 协调发生在三个时刻

不要把 intercom 想成单一机制。一次前台 run 里，协调分三次发生：启动前 bridge 决定子 agent 是否拥有 contact_supervisor；运行中 native supervisor channel 用文件把阻塞请求递给父会话；完成后 result intercom 把 grouped 子结果经事件总线送回父会话，同时把普通 tool result 压缩成 receipt。

Bridge
intercom-bridge.ts + subagent-executor.ts
Supervisor Channel
native-supervisor-channel.ts + pi-args.ts
Result Intercom
result-intercom.ts + executor 出口

2. 五段分工：谁写盘、谁轮询、谁压缩

前台 subagent run + intercom
├── 1. resolveIntercomBridge          ← subagent-executor 启动前
│      applyIntercomBridgeToAgent     ← 注入工具与系统提示
├── 2. buildPiArgs                    ← pi-args 写 channelDir 等 env
│      registerNativeSupervisorClient ← 子进程注册 contact_supervisor
├── 3. createNativeSupervisorChannel  ← extension/index session_start
│      poll request → pi.sendMessage  ← 父端展示 pending
│      subagent_supervisor reply      ← 写 reply 文件
├── 4. runSync 完成                   ← execution 产出 SingleResult
└── 5. emitForegroundResultIntercom   ← grouped event + delivery ack
       maybeBuildForegroundIntercomReceipt ← 压缩 details

记忆口诀：注入 → 环境 → 轮询回复 → 执行 → 送达压缩。async 路径在第 4 步之后由 result-watcher 接手，本切片聚焦 foreground。

检索练习

一次 parallel foreground run 完成，父会话 tool result 只有简短 receipt，看不到 child 的完整 summary。下面哪种判断最符合源码设计？

  1. 这是 bug，完整输出应仍在 tool result 的 details 里。
  2. bridge inactive，因此从未尝试 grouped delivery。
  3. delivery 成功，完整输出在 grouped intercom event 里，details 被刻意剥离。

参考答案：3（在 bridge active 且 delivery 确认时）。stripDetailsOutputsForIntercomReceipt 会移除 heavy output fields；若 bridge inactive 或 500ms 内未收到 acknowledgement，则回退 legacy foreground output（见 intercom-result-delivery.test.ts）。

主要来源

建议先读 subagent-executor.ts 中 resolveIntercomBridge 与 maybeBuildForegroundIntercomReceipt，再对照 native-supervisor-channel.ts 的 request/reply 协议。
