L2-native-supervisor-intercom 短课 0002

运行中 supervisor 请求与回复

学习目标：给定一次 contact_supervisor({ reason: "need_decision" })，你能从子端 tool call 指到父端 reply 文件，并说出 session 过滤与超时清理规则。

1. 子端前提：metadata 必须完整

pi-args.ts 在 orchestrator target、父 session id、run id、child agent name 齐全时创建 supervisor-channels/<run>-<agent>-<index>/，并写入 PI_SUBAGENT_SUPERVISOR_CHANNEL_DIR 等环境变量。子进程里 registerNativeSupervisorClient 读取这些 env；任一缺失则不会注册 contact_supervisor，调用时报 Native supervisor channel is not available。

2. 发请求：三种 reason 的等待语义

sendSupervisorRequest 把请求写成 requests/<id>.json，类型为 subagent.supervisor.request。need_decision 与 interview_request 设置 expectsReply: true，子端轮询 replies/<id>.json 直至超时（默认 10 分钟，可用 PI_INTERCOM_ASK_TIMEOUT_MS 覆盖）。progress_update 不等待，写完 request 即返回 queued。

reason              expectsReply    子端行为
need_decision       true            阻塞等待 reply 文件
interview_request   true            同上 + 尝试 JSON 解析 structuredReply
progress_update     false           立即返回，父端展示后删 request

3. 父端轮询与回复

extension/index.ts 在 session 启动时调用 supervisorChannel.start()，每 500ms 扫描所有 channel 的 request 文件。只有 orchestratorSessionId 匹配当前父 session 的请求会进入 pending map，并通过 pi.sendMessage 展示。父会话用 subagent_supervisor({ action: "reply", replyTo, message }) 写 reply 文件并删除 request；子端 waitForReply 读到后返回给 tool result。

生命周期状态 resolved、expired、inactive（子 run 已结束）会触发 request 文件清理，避免父端看到陈旧 pending。若已有外部 intercom 工具，native 模块仍注册 subagent_supervisor 作为不冲突的 reply path。

检索练习

子 agent 发出 need_decision 后父会话始终看不到 pending。下列排查顺序哪项最合理？

  1. 先查 result-intercom delivery 是否超时。
  2. 先查 bridge mode 是否为 off，再查 request 的 orchestratorSessionId 是否匹配当前父 session。
  3. 先查 TUI 主题色配置。

参考答案：2。看不到 pending 属于运行中通道问题：bridge off 时子端可能根本没有 contact_supervisor；若工具存在但父端无展示，多半是 session id 不匹配或请求已 expired/inactive（见 native-supervisor-channel.test.ts）。

主要来源

native-supervisor-channel.ts 的 sendSupervisorRequest、createNativeSupervisorChannel.poll、writeReply。
