项目整体架构与学习地图

学习目标：15 分钟内能说清 `pi-subagents` 的定位、运行边界和后续阅读路线。

1. 一句话定位

pi-subagents 是一个 TypeScript/Node 编写的 Pi extension：它给父 Pi 会话注册 subagent 与 wait 工具，让父会话可以把任务委派给内置或自定义的子 Pi 会话，并用前台流式结果、后台状态文件、slash 命令、RPC 和 intercom 把结果带回。依据 open-ai-agent/pi-subagents/README.md 与 open-ai-agent/pi-subagents/package.json，它的主场景不是自动后台审查，而是按需委派：review、scout、plan、worker、parallel audits、saved workflows 和 async jobs。

2. 运行地图

parent Pi session │ ├─ registers subagent / wait tools │ ↓ ├─ subagent tool call / slash / RPC spawn │ ↓ ├─ agent discovery 读取 builtin、package、user、project agents │ ↓ ├─ foreground executor 分流 single、parallel、chain、management、async │ ├─ foreground: child Pi 输出回到当前对话 │ └─ async: runner 写 status json、events jsonl、output log │ ↓ ├─ async tracker / run status / wait 轮询与恢复后台状态 │ └─ intercom / native supervisor channel 处理子进程向父会话要决定

这张图的关键是“父会话拥有编排权”。open-ai-agent/pi-subagents/src/extension/index.ts 只负责把能力挂进 Pi 运行时：创建 executor，连接 async tracker，注册 slash/RPC/native supervisor，并在 session 生命周期里恢复状态。真正决定一次调用是单 agent、并行、链式、管理动作，还是转成后台运行的是 foreground executor。

3. 目录地图

open-ai-agent/pi-subagents/ ├── agents/ ← 8 个内置角色定义 ├── prompts/ ← 可复用编排提示模板 ├── skills/ ← 面向父 orchestrator 的使用说明 │ └── pi-subagents/ │ └── SKILL.md ← subagent 编排规范 ├── src/ ← TypeScript extension 实现 │ ├── extension/ ← Pi 注册入口、schema、RPC、doctor │ ├── agents/ ← agent/chain 发现、覆盖和管理 │ ├── runs/ ← 前台执行、后台运行、共享运行工具 │ ├── slash/ ← slash 命令与 prompt workflow 桥 │ ├── intercom/ ← native supervisor 与结果协调 │ └── shared/ ← 类型、路径、状态、格式化工具 ├── test/ ← unit、integration、e2e 行为佐证 ├── README.md ← 用户模型和运行时行为说明 └── package.json ← npm 与 Pi extension 元数据

后续学习时先跟入口，再跟运行链路：open-ai-agent/pi-subagents/src/extension/index.ts 说明“能力如何注册”，open-ai-agent/pi-subagents/src/agents/agents.ts 说明“agent 从哪里来”，open-ai-agent/pi-subagents/src/runs/foreground/subagent-executor.ts 说明“调用如何分流”。后台状态、slash、RPC、intercom 都是围绕这个主链路补可观测性和控制面。

4. 设计判断与回忆练习

核心设计判断有四条：父会话保持最终编排权；读/写职责默认单写者；后台运行必须落盘为机器可读生命周期文件；子会话默认受递归深度和工具可见性约束。L0 暂时豁免 banner.png、package-lock.json、CHANGELOG.md、发布/CI配置和 open-ai-agent/pi-subagents/test/support/，因为它们支撑品牌、复现、发布或测试夹具，不决定主架构路径。

暂停回忆：如果你看到一个后台 subagent 已启动但没有最终结果，应该优先去哪个概念入口找状态？

参考答案：先看 async lifecycle：status.json、events.jsonl、async tracker、subagent({ action: "status" }) 或 wait。不要从终端输出猜状态。

主要来源：README.md 的“What happens”“Where running subagents show up”“Runtime files”，以及 00-overview 参考图谱。不清楚某个箭头时，可以继续让 agent 顺着对应源码文件做 L1/L2 深挖。
