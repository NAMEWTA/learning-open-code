读懂 slash command 如何从模板进入执行链路

本课只回答一个问题：用户输入 /prompt-workflow parallel-review some-file 后，系统如何把一个 Markdown prompt 变成真正的 subagent 执行请求。先抓住这条主线，其他命令清单放到参考页。

1发现模板：读取 package/user/project 三类 prompt 目录。
2注册命令：把模板命令接到 slash 命令系统。
3执行桥接：转换为 SubagentParamsLike，经事件桥进入 executor。

1. 先把 Markdown 变成 workflow

入口在 open-ai-agent/pi-subagents/src/slash/prompt-workflows.ts。模块先扫描内置 package prompts、用户 prompts、项目 prompts。因为发现结果放进同一个 Map，后发现的同名 workflow 会覆盖前面的同名项；测试里明确验证了项目 prompt 会覆盖用户 prompt。

function promptDirs(cwd: string): string[] {
  return [
    packagePromptsDir(),
    path.join(getAgentDir(), "prompts"),
    path.join(getProjectConfigDir(cwd), "prompts"),
  ];
}

function loadPromptWorkflow(filePath: string): PromptWorkflow | undefined {
  const content = fs.readFileSync(filePath, "utf-8");
  const { frontmatter, body } = parseFrontmatter(content);
  const name = path.basename(filePath, ".md");
  if (!name || RESERVED_COMMAND_NAMES.has(name)) return undefined;
  return {
    name,
    description: stringField(frontmatter, "description") ?? firstNonEmptyLine(body),
    body,
    agent: parseAgent(frontmatter),
  };
}

没有写 subagent 的模板默认走 delegate。模板正文里的 $@、$ARGUMENTS、$1 会在运行时被 slash 命令参数替换，所以 Markdown 文件本身就是可执行 workflow 的“任务蓝图”。

2. 再注册成 slash 命令

普通 slash 命令集中在 open-ai-agent/pi-subagents/src/slash/slash-commands.ts。关键不是 `/prompt-workflow` 自己执行，而是 `registerSlashCommands` 把 `runSlashSubagent` 注入给 prompt workflow 注册器。这样模板命令和 `/run`、`/chain`、`/parallel` 最终共用同一个执行出口。

registerPromptWorkflowCommands({
  pi,
  run: (params, ctx) => runSlashSubagent(pi, ctx, params),
});

pi.registerCommand("prompt-workflow", {
  handler: async (rawArgs, ctx) => {
    const words = shellWords(rawArgs);
    const workflow = findWorkflow(discoverPromptWorkflows(ctx.cwd), words.shift() ?? "");
    const runtime = parseRuntimeOptions(words);
    await run(workflowParams(workflow, runtime.args, runtime), ctx);
  },
});

这一步完成了最重要的翻译：模板 frontmatter 和运行时 flags 被合成为 agent、task、model、skill、context、worktree 等结构化字段。

3. 最后进入 slash 执行桥

runSlashSubagent 先生成可渲染的初始结果，再发出事件请求。真正接住请求的是 open-ai-agent/pi-subagents/src/slash/slash-bridge.ts；扩展入口 open-ai-agent/pi-subagents/src/extension/index.ts 在启动时注册这个 bridge，并把它连到 executor。

const requestId = randomUUID();
const initialDetails = buildSlashInitialResult(requestId, params);
pi.sendMessage({
  customType: SLASH_RESULT_TYPE,
  content: initialText,
  display: true,
  details: initialDetails,
});

const response = await requestSlashRun(pi, ctx, requestId, params);
const finalDetails = finalizeSlashResult(response);
pi.sendMessage({
  customType: SLASH_RESULT_TYPE,
  content: buildSlashExportText(response),
  display: !ctx.hasUI,
  details: finalDetails,
});

这里的 UI 消息不是装饰。open-ai-agent/pi-subagents/src/slash/slash-live-state.ts 会先保存 running placeholder，再把 bridge 回传的 progress 写入 live snapshot，最后用 final snapshot 替换。于是用户看到的是同一张 slash result 卡片从“运行中”变成“完成”。

4. 检索练习

暂停回忆：不看上文，把这四件事按发生顺序排序：A. executor 执行；B. Markdown prompt 被解析；C. `registerPromptWorkflowCommands` 注入 `runSlashSubagent`；D. `buildSlashInitialResult` 创建可渲染占位结果。

查看答案与反馈

答案是 B → C → D → A。B 发生在 workflow 发现阶段；C 发生在 slash 命令注册阶段；用户真正运行命令后，D 先让 UI 有 live card；随后 slash bridge 才把请求交给 executor。若你把 A 放在 D 前面，说明你跳过了 live-state 这层可观测性。
