profiles 模块导览

学习目标：15 分钟内能判断 profile 如何给模型选择、fallback 与能力边界提供事实来源。

1. profile 不是运行器

open-ai-agent/pi-subagents/src/profiles/profiles.ts 的职责是把“供应商有哪些模型、这些模型是否可用、适合哪类子 agent”固化成本地 JSON。它不直接执行任务，也不决定失败后怎么重试；它生成的是事实来源：provider catalog、quota profile、quality profile。profile 文件最终写入 subagents.agentOverrides，让内置角色获得明确的 model。

provider registry + live probe
  ↓
provider catalog: observed + derived + warnings
  ↓
quota / quality profiles
  ↓
settings.subagents.agentOverrides
  ↓
agent discovery → runtime model candidates

2. 从 registry 到三档模型

刷新 catalog 时，模块读取 ctx.modelRegistry.getAvailable() 中同一 provider 的模型，并用 pi -p --model ... --no-tools 做轻量探测。探测结果会归为 ok、auth、unavailable、timeout、error 或 skipped。随后分类器混合官方 metadata 与名称启发式：成本、上下文、输出 token、reasoning、flash/mini/pro/opus 等词都会影响 cost、quality、latency 与 profileRank。没有官方 metadata 时，它会保留 warning，说明分类退回名称启发式。

3. 它怎样影响 fallback 与边界

applySubagentProfile 只替换 settings 里的 subagents 字段；真正消费它的是 open-ai-agent/pi-subagents/src/agents/agents.ts，那里把 agentOverrides 应用到 builtin agent。执行阶段再由 open-ai-agent/pi-subagents/src/runs/shared/model-fallback.ts 把 primary model 与 fallbackModels 合成候选列表，并在 retryable 模型失败时继续尝试。边界检查来自 open-ai-agent/pi-subagents/src/runs/shared/model-scope.ts：显式越界是 error，继承配置越界是 warn。完整字段与证据见 profiles 参考页。

4. 检索练习

判断题：如果一个 generated quality profile 把 reviewer 指向 provider/strong，但运行时 provider/strong 返回 “model unavailable”，profile 模块会自己切到便宜模型吗？

参考答案：不会。profile 只负责生成和检查事实；运行时是否继续尝试，要看该 agent 是否配置了 fallbackModels，以及 open-ai-agent/pi-subagents/src/runs/foreground/execution.ts 是否把失败识别为 retryable，再按 buildModelCandidates 的顺序尝试下一个模型。

主要来源：open-ai-agent/pi-subagents/src/profiles/profiles.ts、open-ai-agent/pi-subagents/src/runs/shared/model-fallback.ts、open-ai-agent/pi-subagents/src/runs/shared/model-scope.ts。不清楚某个字段时，继续让 agent 按参考页中的 L2 候选展开。
