profiles 模块参考图谱

本页承载 L1 的长清单：profile 字段、解析规则、模型能力/供应商差异、调用方与测试证据。短课只保留“profile 是模型事实来源”这一判断技能。

1. 模块边界

文件在本主题中的用途关键结论
open-ai-agent/pi-subagents/src/profiles/profiles.ts核心实现管理 profile 文件、刷新 provider catalog、探测模型、分类分层、生成 quota/quality profile、应用和检查 profile。
open-ai-agent/pi-subagents/src/agents/agents.tssettings 消费侧读取 subagents.agentOverrides、defaultModel、modelScope，把 profile 结果合成 AgentConfig。
open-ai-agent/pi-subagents/src/runs/shared/model-fallback.ts运行时候选与 fallback规范化模型 id，继承父会话模型，合成 primary + fallback 候选列表，识别 retryable 模型失败。
open-ai-agent/pi-subagents/src/runs/shared/model-scope.ts能力边界用 allow pattern 约束模型范围；explicit 越界为 error，inherited 越界为 warn。
open-ai-agent/pi-subagents/src/agents/agent-management.ts管理面models 动作展示有效模型与来源；create/update 提示模型或 fallback 是否存在于 registry。
open-ai-agent/pi-subagents/src/extension/schemas.ts对外 schema任务级参数只暴露 model override；profile 的落点是 settings，而不是 tool schema 中的独立字段。

2. 文件位置与 JSON 形状

getSubagentProfilesRootDir 把所有 profile 数据放在 agent 目录下的 profiles/pi-subagents。普通 profile 文件直接位于该目录；provider catalog 放在其 providers 子目录。profile 名和 provider 名都经过安全文件名校验，拒绝空值、斜杠、反斜杠、.、.. 与路径穿越。

{
  "subagents": {
    "agentOverrides": {
      "scout": { "model": "provider/cheap-model" },
      "reviewer": { "model": "provider/strong-model" }
    }
  }
}
结构字段规则
SubagentProfileFilesubagents.agentOverrides必须是对象；每个 override 必须是对象；model 若存在必须是字符串。
ProviderModelCatalogFileprovider、refreshedAt、maxAgeDays、sources、models记录 provider 目录快照、缓存时效、数据来源和每个模型的 observed/derived 信息。
ProviderModelCatalogModel.observedavailableInRegistry、name、reasoning、thinkingLevels、contextWindow、maxTokens、cost、probe来自 runtime registry、模型信息工具和 live probe，是“看到的事实”。
ProviderModelCatalogModel.derivedprofileRank、costTier、qualityTier、latencyTier、recommendedRoleTier、recommendedAgents、classificationSources由分类器派生，是“如何使用这个模型”的建议。

阶段源码证据规则

4. 模型分层与供应商差异

分类器不硬编码某个供应商品牌的固定等级，而是把同一个 provider 内的模型放入相对比较：成本归一化决定 cheap/medium/expensive，contextWindow/maxTokens/reasoning/名称档位共同影响质量，flash/instant/turbo 等影响 latency。这样可以处理不同供应商字段不齐的情况，但也意味着没有 metadata 时只能靠名称启发式，结果必须带 warning。

派生字段含义调用影响
costTiercheap、medium、expensive中等质量且便宜的模型会被推荐给 cheap role tier；高成本模型更可能进入强模型池。
qualityTierweak、medium、strongstrong 直接映射到 worker、reviewer、oracle 这类强角色建议。
latencyTierfast、medium、slowfast 名称提示会降低 quality score，并施加 profileRank penalty，避免高速变体误占质量最高档。
recommendedRoleTiercheap、medium、strong决定 recommendedAgents：cheap 对应 scout/delegate，medium 对应 planner/context-builder/researcher，strong 对应 worker/reviewer/oracle。
classificationSourcesofficial-metadata、heuristic-name用于统计 heuristic fallback，提示用户哪些选择需要人工复核。

可用模型过滤会排除 unavailable、auth、timeout、error。随后还会过滤被支配模型：如果另一个模型成本不高、rank 不低、reasoning/context/maxTokens 不差，并且至少一项更好，当前模型会被移出 profile 选择池。

7. profile check 与 thinking suffix

checkSubagentProfile 面向“已经保存的 profile 是否还能用”。它会读取 profile，收集所有非空 model override，用 registry 查找模型。若用户写的是短 id 或带 thinking suffix 的 id，它会先用模型信息工具拆出已知 suffix，再把 probe 模型恢复为 canonical provider/id:suffix。同一个 probe id 会缓存，避免重复探测。

9. 常见误判

误判正确判断
profile 负责运行时 fallback。profile 只生成/应用事实。fallback 顺序来自 open-ai-agent/pi-subagents/src/runs/shared/model-fallback.ts 和 agent 配置的 fallbackModels。
live probe 成功就说明模型质量强。probe 只判断可用性。质量来自 metadata 与名称启发式的分类结果。
schema 里应该有 profile 参数。schema 暴露的是调用级 model override；profile 是 settings 侧批量覆盖机制。
modelScope 会阻止所有越界模型。explicit 越界抛错；inherited 或 fallback 越界按源码规则 warn 并保留模型。
没有官方 metadata 时不能生成 profile。可以生成，但 classificationSources 只含 heuristic-name，并带 warning。
