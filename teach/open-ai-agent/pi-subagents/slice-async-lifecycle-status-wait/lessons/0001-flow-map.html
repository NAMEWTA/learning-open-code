L2-async-lifecycle 短课 0001

async lifecycle 全链路流程图

学习目标：15 分钟内能口述一次 async: true 调用从启动到完成通知的五段链路，并指出每段对应源码文件。

1. 入口：前台只负责“点火”

当 subagent-executor 判定 effectiveAsync 为真时，它不会调用前台 runSync，而是转给 executeAsyncSingle 或 executeAsyncChain。启动器做三件事：创建 asyncDir、把 runner 配置写到临时 JSON、用 jiti detached spawn subagent-runner.ts，然后立刻向父 Pi 发出 SUBAGENT_ASYNC_STARTED_EVENT。

{
  "agent": "reviewer",
  "task": "Review src/runs/background/wait.ts",
  "async": true
}

返回文本里会强调：不要用 sleep 或轮询空等；若本 turn 必须等到结果，应调用 wait()；若只是巡检，用 subagent({ action: "status", id })。这段提示来自 formatAsyncStartedMessage。

2. 五段分工：谁写盘、谁记内存、谁阻塞

src/runs/background/
├── async-execution.ts      ← 启动 detached runner，发 started event
├── subagent-runner.ts      ← 运行中写 status.json / events.jsonl / result
├── async-job-tracker.ts    ← 父进程 job map + 轮询刷新 + 控制事件转发
├── run-status.ts           ← status action：一次性快照 / fleet / transcript
├── wait.ts                 ← wait tool：阻塞当前 turn 直到终态或 needs_attention
└── result-watcher.ts       ← 监听 RESULTS_DIR，发 complete event
磁盘真相
status.json 记录运行态；events.jsonl 记录控制事件；RESULTS_DIR/<id>.json 是完成信号。
内存投影
async-job-tracker 把 started event 写入 state.asyncJobs，定时 reconcile 磁盘状态并刷新 TUI widget。
父 agent 观测
status = 看一眼；wait = 本 turn 卡住直到有结果；交互式会话还可依赖下一 turn 的 completion notification。

3. 调用时序（主路径）

Parent turn                async-execution        subagent-runner       job-tracker / watcher
    |                            |                      |                        |
    |-- subagent async:true ---->|                      |                        |
    |                            |-- spawn detached --->|                        |
    |                            |-- started event -------------------------------->| handleStarted
    |<- runId + asyncDir --------|                      |                        |
    |                            |                      |-- write status.json -->|
    |                            |                      |-- write events.jsonl ->|
    |-- status (optional) ------ run-status.ts -------- read disk --------------->|
    |-- wait (optional) -------- wait.ts -------------- poll + event wake ------>|
    |                            |                      |-- write result.json -->| result-watcher
    |<- complete notification --- extension index <------ complete event ----------|

关键接线在 open-ai-agent/pi-subagents/src/extension/index.ts：createAsyncJobTracker 订阅 started/complete；createResultWatcher 监视 result 目录；wait tool 注册为独立阻塞入口。

更完整的状态表、产物清单与异常路径见 async lifecycle 参考页；runs 三层边界见 runs execution L1 参考。

检索练习

你在非交互式 pi -p 单 turn 里启动了 3 个 async: true 子任务，必须全部完成才能结束。下面哪种做法符合源码设计？

  1. 每 5 秒调用一次 subagent({ action: "status" }) 直到都 complete。
  2. 调用 wait({ all: true }) 阻塞到每个 run 进入终态。
  3. 结束 turn，等待 Pi 在下一 turn 自动唤醒你。

参考答案：2。wait.ts 顶部注释写明非交互式单 turn 没有“下一 turn”可接收 completion notification；formatAsyncStartedMessage 也禁止 sleep-poll。选项 1 是反模式；选项 3 在 pi -p 下会丢结果。
