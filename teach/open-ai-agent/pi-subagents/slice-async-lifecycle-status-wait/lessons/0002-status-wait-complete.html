L2-async-lifecycle 短课 0002

status、wait 与完成通知

学习目标：能判断该用 status、wait 还是等 completion notification，并说出 needs_attention 时为什么 wait 会提前返回。

1. 三种观测方式

方式是否阻塞 turn典型场景核心文件
subagent({ action: "status", id })否巡检进度、看 transcript、排查 stale runrun-status.ts
wait({ id?, all?, timeoutMs? })是skill 必须跑完、pi -p 单 turn、fleet 滚动替换wait.ts
completion notification否（下一 turn）交互式会话，父 agent 还有独立工作可做result-watcher.ts + extension 事件

run-status.ts 的 inspectSubagentStatus 会先 reconcileAsyncRun，再读 status.json 或回退到 result 文件；无 id 时列出当前 session 的 active runs。wait.ts 则循环调用 listAsyncRuns，并订阅 complete/control 事件提前唤醒。

2. wait 的两种完成模式

// 默认：第一个 run 离开 active 集就返回（fleet 滚动替换）
wait({ })

// 全部终态才返回
wait({ all: true })

// 指定单个 run（忽略 all）
wait({ id: "abc123" })

无论哪种模式，只要 run 的 activityState === "needs_attention"，wait 都会立即结束并提示你去 status 后 nudge/resume/interrupt。这是因为 skill 或单 turn 场景下没有“下一 turn”来接收 control notice。

async-job-tracker 在轮询时也会读 events.jsonl，把 subagent.control 行转成 SUBAGENT_CONTROL_EVENT，让 UI 与 wait 同步感知 attention 信号。

3. 完成通知的最后一跳

runner 结束时把摘要写入 RESULTS_DIR/<runId>.json。result-watcher.ts 用 fs.watch（失败则降级轮询）捕获新文件，校验 sessionId 后：

  1. 去重（completion-dedupe）避免重复处理；
  2. 可选走 intercom grouped delivery；
  3. 发出 SUBAGENT_ASYNC_COMPLETE_EVENT，由 async-job-tracker.handleComplete 把 job 标为 complete/failed 并安排清理。

若 result 文件已存在但 status.json 仍显示 running，stale-run-reconciler（被 status 与 tracker 共用）会尝试修正——这是 status 与 wait 轮询兜底的共同依赖。

判断练习

子 agent 进入 needs_attention，你在交互式 Pi 里本来打算继续写别的代码，稍后再看结果。此时最稳妥的做法是？

  1. 调用 wait({ all: true, timeoutMs: 3600000 }) 一直等到 complete。
  2. 调用 subagent({ action: "status", id }) 看清阻塞点，再 resume/interrupt/steer。
  3. 忽略 attention，依赖 completion notification 自动处理决策。

参考答案：2。attention 表示子 agent 需要父会话介入；wait 会为此提前返回 precisely 是为了让你行动。选项 1 在 attention 场景下会立刻结束而非傻等；选项 3 可能永远等不到自动决策。
