Slash 命令到 executor 桥接链路图

学习目标：15 分钟内能按顺序说出用户输入 /run reviewer "task" 后，从命令解析到 executor.execute 分流点的八个跳点。

L2 垂直切片 前置：slash-workflows 模块 速查：参考图谱

1. 两条入口，一个 executor

扩展启动时（extension/index.ts）先注册两个桥，再注册 slash 命令。二者最终都调用 executeSubagentCollapsed → executor.execute：

registerSlashSubagentBridge        ← subagent:slash:* 事件
registerPromptTemplateDelegationBridge ← prompt-template:subagent:*
createSubagentExecutor → executor
registerSlashCommands(pi, state)   ← /run、/chain、/parallel …

本课主路径是 slash 命令；Pi prompt 模板委派走第二条桥，参数形态不同，但 executor 分流逻辑相同。

2. 八跳全链路（以 /run 为例）

  1. 命令解析 — slash-commands.ts 的 /run handler 解析 agent token、inline config、--bg/--fork，组装 SubagentParamsLike（clarify: false、agentScope: "both"）。
  2. 发起 live card — runSlashSubagent 生成 requestId，buildSlashInitialResult 创建 placeholder，pi.sendMessage 发出 subagent-slash-result 初始消息并折叠工具面板。
  3. 事件请求 — requestSlashRun 订阅 started/update/response，向事件总线 emit subagent:slash:request（携带 params 与当前 ctx）。
  4. Bridge 接收 — slash-bridge.ts 优先用 request 里的 ctx，否则回退 state.lastUiContext；无 ctx 则立即 error response。建立 AbortController 后同步 emit started。
  5. 调用 executor — bridge 的 execute 回调即 executeSubagentCollapsed，进入 subagent-executor.ts 的 execute。
  6. 门卫与分流 — 有 action 走管理路径（doctor/status 等）；否则检查嵌套深度、normalize params，再按 chain / tasks / agent 与 effectiveAsync 分到 runChainPath、runParallelPath、runSinglePath 或 runAsyncPath。
  7. 进度回流 — executor onUpdate → bridge emit subagent:slash:update → applySlashUpdate 写 live-state；有 UI 时更新 status 行与 Esc 取消。
  8. 收尾展示 — bridge emit response → finalizeSlashResult → 最终 sendMessage（UI 下隐藏正文、保留 details snapshot）。

记忆口诀：解析 → live card → request 事件 → bridge → executor → 分流 → update 回流 → finalize。第 3 步若 bridge 未同步 emit started，requestSlashRun 会立刻报「No slash subagent bridge responded」。

3. 三个关键文件分工

src/
├── slash/
│   ├── slash-commands.ts       ← 命令注册、params 组装、requestSlashRun
│   └── slash-bridge.ts         ← 事件桥：request → execute → response
├── extension/index.ts          ← 注册 bridge 与 executor 接缝
└── runs/foreground/
    └── subagent-executor.ts    ← execute 分流与 run*Path

slash-commands.ts 不 spawn 子进程；slash-bridge.ts 不解析 /run 参数；subagent-executor.ts 不知道 slash 语法。边界清楚后，可先判断卡在「命令没发出 request」还是「bridge 没接到」还是「executor 内分流」。

4. 测试与练习

test/integration/slash-commands.test.ts 用 mock 事件总线模拟 bridge：验证 /run 可无 task、/parallel 转发 inline output config、9 任务不被命令层硬拦截，以及 UI 下初始消息可见、最终消息隐藏。完整断言表见 参考页测试索引。

检索练习：用户输入 /run reviewer "fix bug" --bg，八跳链路在第几步离开前台 runSinglePath？

参考答案：第 6 步。命令层把 --bg 映射为 params.async = true；executor 算出 effectiveAsync === true 后走 runAsyncPath，不会进入 runSinglePath。

主要来源：slash-commands.ts（runSlashSubagent、requestSlashRun）、slash-bridge.ts、extension/index.ts（bridge 注册）、subagent-executor.ts（execute 分流）。spawn 与 JSONL 细节见 单次前台 run。有不清楚的地方，可以继续向 agent 追问。
