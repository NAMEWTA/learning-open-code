第 1 课 · 鸟瞰：一个类如何接入分布式任务调度 入门地图

🎯 本节目标（对应使命 Success #1）：学完你能用一句话说清 ruoyi-common-job 与 SnailJob 的关系，并说出这个模块仅有的一个类做什么事。这是后面两节课所有细节挂靠的骨架——先有地图，再走小路。

我们先不抠任何一行实现，只回答一个问题：打开 ruoyi-common-job 这个模块，你会看到什么？

一、整个模块就这么大

分布式任务调度听起来很重——调度中心、执行节点、任务分片、失败重试……但 RuoYi 的 ruoyi-common-job 拢共只有 1 个 Java 文件 + 1 个 pom：

ruoyi-common-job/ ├── pom.xml ← 依赖 2 个 SnailJob starter + ruoyi-common-core └── src/main/java/org/dromara/common/job/config/ └── SnailJobConfig.java ← 【全部逻辑】装配 + 开关 + 日志上报

真正调度任务、分发指令、持久化记录的脏活累活，全在第三方框架 SnailJob（Aizuda 社区出品）里。ruoyi-common-job 自己一行「调度逻辑」的代码都没有。

💡 一句话定位：
ruoyi-common-job 不是一个任务调度框架，而是一层「客户端接入适配（adapter）」——它把开源框架 SnailJob 的客户端「收编」进 RuoYi 的技术栈，让 SnailJob 用上 RuoYi 自己的自动装配习惯、自己的配置开关、自己的远程日志上报。

二、一个类，四件事

SnailJobConfig 只有 43 行代码，核心就做四件事——分别对应类上的四个注解和类内的一个方法：

机制代码位置一句话职责解决什么问题对应课
@AutoConfiguration 第 21 行 让 Spring Boot 自动发现并加载本配置类 业务模块只要依赖本模块，啥都不用配就能接入 SnailJob 第 2 课
@ConditionalOnProperty 第 22 行 通过 yml 里的 snail-job.enabled 开关控制是否生效 开发环境不想连 SnailJob 服务端？随手关掉，不报错 第 2 课
@EnableSnailJob 第 24 行 触发 SnailJob 客户端的全部自动装配 连接服务端、注册任务执行器、接收调度指令——一个注解搞定 第 2 课
onStarting() + @EventListener 第 32-39 行 SnailJob 客户端启动时挂载远程日志 Appender 任务执行日志自动回传服务端控制台，不用登录每台机器看 log 第 2 课

外加一个 @EnableScheduling（第 23 行），允许 Spring 原生的 @Scheduled 也能用——但这不是本模块的核心价值。

三、它怎么被用起来？（一条最短链路）

模块本身只是配置，真正干活的是业务模块 ruoyi-job 里那些 @JobExecutor 标注的任务类。最典型的链路长这样：

开发阶段 (ruoyi-job 模块) 运行时 运维阶段 ─────────────────── ───────── ──────── ① 编写任务类 @Component @JobExecutor(name = "testJobExecutor") ② SnailJob Server ④ 服务端控制台 public class TestAnnoJobExecutor { 调度中心运行中 查看任务执行 public ExecuteResult ↓ 记录和日志 jobExecute(JobArgs args) { ③ 按 cron 触发 ──► │ doSomething(...); 分派任务到 │ return ExecuteResult.success(); 客户端节点 │ } │ │ ruoyi-common-job │ SnailJobConfig 已自动启用 │ └─ 远程日志 Appender 已就绪 } │ ◄─ 回传日志 (SnailJobLog.REMOTE) ◄─ 返回结果 (ExecuteResult)

看这张图，SnailJobConfig 的位置一目了然：它蹲在「应用启动」的节点上，确保当调度指令到达时，客户端已就绪、日志通道已打通。

四、SnailJob 支持哪些任务类型？（一览 ruoyi-job 示例）

在 ruoyi-modules/ruoyi-job/src/main/java/org/dromara/job/snailjob/ 下有 7 个示例类，覆盖了 SnailJob 的主要任务模式：

示例类任务类型特点
TestAnnoJobExecutor 普通任务 最简单的模式：一个方法，执行完返回成功/失败。适合单一逻辑的定时任务。
TestBroadcastJob 广播任务 任务被分发到所有在线客户端节点同时执行。适合刷新本地缓存、全量数据同步等场景。
TestStaticShardingJob 静态分片 服务端指定分片参数（如 ID 范围），各节点处理属于自己的那一段。适合大数据量批量处理。
TestMapJobAnnotation Map 任务 用 @MapExecutor 将一批数据拆成多个子任务，分发到集群节点并行执行。不关注合并结果。
TestMapReduceAnnotation1 MapReduce 任务 在 Map 基础上增加了 @ReduceExecutor——子任务完成后，汇总所有分片结果做一次聚合。
AlipayBillTask DAG 工作流 通过 jobArgs.getWfContext() 在任务间传递上下文，实现有向无环图（DAG）编排。本示例与 WechatBillTask、SummaryBillTask 串联成账单处理工作流。

记住：不管你写哪种任务，入口都是 SnailJobConfig 上的 @EnableSnailJob。这个注解一开，以上所有任务类型都能用。

🤔 想一想：模块里只有一个配置类，没有任何 Controller 或 Service，为什么业务方在 pom 里加一行依赖就能写 SnailJob 任务了？ 因为 ruoyi-common-job 的 pom 依赖了 snail-job-client-starter 和 snail-job-client-job-core。Spring Boot 的自动装配机制会加载这些 starter 里所有的 @Configuration 类，包括 SnailJob 客户端的核心 Bean（连接器、任务扫描器、调度接收器等）。SnailJobConfig 只是补了一刀：加了开关控制和远程日志。真正让任务跑起来的组件全在 SnailJob starter 里——第 2 课专门讲 @EnableSnailJob 到底启用了什么。

Q1. ruoyi-common-job 模块和 SnailJob 是什么关系？

A. ruoyi-common-job 自己实现了任务调度的全部逻辑，SnailJob 没用上 B. ruoyi-common-job 是一层客户端接入适配，调度能力全靠 SnailJob，自己只补自动装配 + 开关 + 远程日志三件事 C. 两者是竞争关系，二选一 D. SnailJob 依赖 ruoyi-common-job 才能运行
看答案B。模块只有 1 个类，没有一行调度逻辑；真正调度全靠 SnailJob。模块的价值在于「把 SnailJob 客户端接入 RuoYi 技术栈」。

Q2. SnailJobConfig 上的 @ConditionalOnProperty 有什么用？

A. 检查数据库是否连接成功 B. 只有 yml 里 snail-job.enabled=true 时，整个配置类才生效，相当于一键开关 C. 检查任务是否有执行权限 D. 条件判断当前环境是 dev 还是 prod
看答案B。通过 yml 配置 snail-job.enabled: true/false 控制 SnailJob 客户端是否启动。开发时不想连调度中心？设为 false 即可。

Q3. 下面哪个类使用 @JobExecutor(name = "testJobExecutor") 标记了一个最简单的普通任务？

A. TestBroadcastJob B. TestMapReduceAnnotation1 C. TestAnnoJobExecutor D. AlipayBillTask
看答案C。TestAnnoJobExecutor 是最简模式：一个 @JobExecutor + 一个 jobExecute 方法，返回 ExecuteResult.success()。第 3 课以此为例讲解如何写任务。
