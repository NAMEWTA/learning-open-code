第 3 课 · 实战：任务创建与调度全链路 动手实战

🎯 本节目标（对应使命 Success #5、#6）：学完你能从 0 到 1 写出一个 SnailJob 定时任务，理解「编写代码 → 启动客户端 → 服务端创建任务 → 调度执行 → 日志回传」的完整链路，并能对照示例代码讲清 SnailJob 支持的 6 种任务类型各自的特点。

前两课我们读了配置，本课我们真正写一个任务并理解它怎么被调度。

一、从最简单的任务开始

打开 TestAnnoJobExecutor.java，这是仓库里最简的 SnailJob 任务：

// ruoyi-modules/ruoyi-job/.../snailjob/TestAnnoJobExecutor.java
@Component
@JobExecutor(name = "testJobExecutor")          // ← 唯一关键注解
public class TestAnnoJobExecutor {

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        SnailJobLog.LOCAL.info("testJobExecutor. jobArgs:{}",
            JsonUtil.toJsonString(jobArgs));     // 本地日志
        SnailJobLog.REMOTE.info("testJobExecutor. jobArgs:{}",
            JsonUtil.toJsonString(jobArgs));     // 本地 + 远程
        return ExecuteResult.success("测试成功"); // 返回结果
    }
}

拆解这个任务的结构：

元素代码说明
Spring Bean @Component 必须注册为 Spring Bean，SnailJob 才能找到它
任务声明 @JobExecutor(name = "testJobExecutor") name 是任务的唯一标识——服务端创建任务时填的就是这个名字
执行方法 public ExecuteResult jobExecute(JobArgs jobArgs) 约定的方法签名：方法名随意但参数和返回值类型必须匹配
参数 JobArgs jobArgs SnailJob 传入的任务上下文，含任务参数、工作流上下文等
返回值 ExecuteResult.success("测试成功") success 报告成功，failure 报告失败——服务端据此记录执行状态
日志 SnailJobLog.REMOTE.info(...) 回传到 SnailJob Server，在控制台「调度日志」中可见
💡 约定优于配置：SnailJob 的 @JobExecutor 不强制方法名，但通常约定叫 jobExecute。参数类型 JobArgs 和返回类型 ExecuteResult 是固定的——SnailJob 框架通过反射调用，类型不匹配会报错。

二、完整链路：从写代码到看到结果

写一个任务只是开始。真正让它跑起来需要经过 7 个环节：

1写任务代码
ruoyi-job 模块的 snailjob 包
如上例：@Component + @JobExecutor(name = "xxx") + jobExecute(JobArgs) 方法 + 返回 ExecuteResult。
2配置 yml
application-dev.yml 或 application-prod.yml
snail-job:
  enabled: true              # 必须为 true
  group: "ruoyi_group"       # 必须在 Server 后台有对应组
  token: "SJ_cKqBTP..."     # Server 验证令牌
  server:
    host: 127.0.0.1          # Server 地址
    port: 17888              # Server 端口
  namespace: dev             # 命名空间
确保 enabled: true，否则 SnailJobConfig 不会加载。
3启动 SnailJob Server
ruoyi-extend/ruoyi-snailjob-server
运行 SnailJobServerApplication（端口 8800，context-path=/snail-job）。这是调度中心——没有它，客户端启动但收不到调度指令。
4启动业务应用（客户端）
ruoyi-admin
应用启动时 SnailJobConfig 生效 → @EnableSnailJob 触发客户端初始化 → 客户端连接 Server（127.0.0.1:17888）→ 注册心跳 → @JobExecutor 任务被注册到执行器列表。
5在 Server 控制台创建任务
浏览器访问 http://localhost:8800/snail-job
进入「任务管理」→「创建任务」→ 填写：
  • 任务名称：任意描述文字
  • 执行器名称：testJobExecutor（必须与 @JobExecutor 的 name 完全一致！）
  • cron 表达式：如 0/30 * * * * ?（每 30 秒执行一次）
  • 任务参数：可选，会被传入 JobArgs.getJobParams()
  • 路由策略：轮询/广播/故障转移等
  • 阻塞策略：串行/丢弃/覆盖
6等待调度触发
到了 cron 表达式指定的时间点，Server 选择一个在线客户端节点，通过 Netty 通道下发「执行 testJobExecutor」指令。客户端的 SnailJob 执行器接收到指令，从 Spring 容器中找到 name="testJobExecutor" 的 Bean，反射调用 jobExecute(JobArgs)。
7查看执行结果
Server 控制台 → 调度日志
  • 执行状态：成功 / 失败
  • 执行耗时：任务运行了多久
  • 远程日志：SnailJobLog.REMOTE.info() 和通过 SnailLogbackAppender 上报的所有日志内容
💡 调试提示：如果任务创建后一直不执行，按顺序排查：① snail-job.enabled 是否为 true？② Server 是否在运行？③ Server 和客户端的 token 是否一致？④ 客户端所属 group 与任务所属组是否一致？⑤ @JobExecutor 的 name 与 Server 上填的「执行器名称」是否完全一致（包括大小写）？

三、六种任务类型实战对比

仓库的 ruoyi-job/snailjob/ 包下有 7 个示例，覆盖了 6 种模式。以下是每种模式的特点和适用场景：

3.1 普通任务（TestAnnoJobExecutor）

@JobExecutor(name = "testJobExecutor")
public class TestAnnoJobExecutor {
    public ExecuteResult jobExecute(JobArgs jobArgs) {
        // 做一件事，返回结果
        return ExecuteResult.success("测试成功");
    }
}

特点：任务被投递到一个节点上执行一次。最简单的模式。
适用场景：生成日报、清理临时文件、同步数据——任何一个不需要分片/广播的定时任务。

3.2 广播任务（TestBroadcastJob）

@JobExecutor(name = "testBroadcastJob")
public class TestBroadcastJob {
    @Value("${snail-job.port}")
    private int clientPort;          // 每个节点端口不同，证明广播到了所有节点

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        SnailJobLog.REMOTE.info("客户端端口:{}", clientPort);
        // 所有在线节点同时执行这段代码
        return ExecuteResult.success();
    }
}

特点：同一任务被广播到所有在线客户端节点，各节点并行执行。
适用场景：刷新所有节点本地缓存、向所有节点下发最新配置、全量健康检查。

3.3 静态分片（TestStaticShardingJob）

@JobExecutor(name = "testStaticShardingJob")
public class TestStaticShardingJob {
    public ExecuteResult jobExecute(JobArgs jobArgs) {
        String jobParams = Convert.toStr(jobArgs.getJobParams());
        // jobParams 由服务端传入：如 "1,5000"，表示处理 ID 1~5000
        String[] split = jobParams.split(",");
        Long fromId = Long.parseLong(split[0]);
        Long toId = Long.parseLong(split[1]);
        // 处理这个 ID 范围的数据
        return ExecuteResult.success("执行分片任务完成");
    }
}

特点：服务端在创建任务时指定分片参数（如 ID 范围），不同节点拿到不同的参数，各自处理自己的分片。
适用场景：百万级数据的批量加密、大表数据迁移、按 ID 段逐批处理。

3.4 Map 任务（TestMapJobAnnotation）

@JobExecutor(name = "testMapJobAnnotation")
public class TestMapJobAnnotation {

    @MapExecutor                                       // 根任务：拆数据
    public ExecuteResult doJobMapExecute(MapArgs mapArgs, MapHandler mapHandler) {
        List sourceList = ...;                // 1000 条数据
        List<List<Integer>> partition = ...;           // 拆成 10 份，每份 100 条
        return mapHandler.doMap(partition, "doCalc");  // 分发给各节点执行
    }

    @MapExecutor(taskName = "doCalc")                  // 子任务：计算
    public ExecuteResult doCalc(MapArgs mapArgs) {
        List<Integer> sourceList = (List<Integer>) mapArgs.getMapResult();
        int total = sourceList.stream().mapToInt(i -> i).sum();
        return ExecuteResult.success(total);
    }
}

特点：由代码逻辑动态分片（不是服务端静态分片）。根任务用 mapHandler.doMap() 把数据拆成多份，子任务由 SnailJob 分发给集群各节点并行执行。
适用场景：数据量不确定（需要运行时拆分）、各子任务独立、不关心汇总结果。

3.5 MapReduce 任务（TestMapReduceAnnotation1）

@JobExecutor(name = "testMapReduceAnnotation1")
public class TestMapReduceAnnotation1 {

    @MapExecutor                              // 根任务：拆分
    public ExecuteResult rootMapExecute(MapArgs mapArgs, MapHandler mapHandler) { ... }

    @MapExecutor(taskName = "doCalc")         // 子任务：并行计算
    public ExecuteResult doCalc(MapArgs mapArgs) { ... }

    @ReduceExecutor                           // 汇总任务：聚合结果
    public ExecuteResult reduceExecute(ReduceArgs reduceArgs) {
        // reduceArgs.getMapResult() 是所有子任务的返回值集合
        int reduceTotal = reduceArgs.getMapResult().stream()
            .mapToInt(i -> Integer.parseInt((String) i)).sum();
        return ExecuteResult.success(reduceTotal);
    }
}

特点：Map 任务的升级版——所有子任务完成后，增加一个 @ReduceExecutor 汇总阶段，将分散的计算结果聚合。
适用场景：分布式统计（各节点算各自的数据量，汇总得到总量）、大规模数据聚合、排行榜计算。

3.6 DAG 工作流（AlipayBillTask / WechatBillTask / SummaryBillTask）

@JobExecutor(name = "alipayBillTask")
public class AlipayBillTask {
    public ExecuteResult jobExecute(JobArgs jobArgs) {
        // 从上游任务获取上下文
        String settlementDate = (String) jobArgs.getWfContext().get("settlementDate");
        // 做自己的事...
        BillDTO billDTO = new BillDTO(...);
        // 把结果写入上下文，传给下游任务
        jobArgs.appendContext("alipay", JsonUtils.toJsonString(billDTO));
        return ExecuteResult.success(billDTO);
    }
}

特点：通过 jobArgs.getWfContext() / appendContext() 在任务间传递数据，实现 DAG（有向无环图）编排。在 Server 控制台通过拖拽方式编排任务执行顺序。
适用场景：复杂的批处理流水线——如对账流程（支付宝账单 → 微信账单 → 汇总报表），各步骤有明确的先后依赖。

四、任务类型选型速查

需求推荐类型关键注解
单一操作定时执行普通任务@JobExecutor
所有节点同时执行同一操作广播任务@JobExecutor + 服务端选「广播」路由
按固定范围拆分（服务端指定参数）静态分片@JobExecutor + 解析 jobArgs.getJobParams()
按数据量动态拆分Map 任务@JobExecutor + @MapExecutor
动态拆分 + 汇总结果MapReduce@JobExecutor + @MapExecutor + @ReduceExecutor
多步骤有先后依赖DAG 工作流@JobExecutor + jobArgs.getWfContext() + 服务端编排

五、练习：仿写一个自己的任务

练题目：写一个「每日统计数据量」的普通任务
在 ruoyi-job/snailjob/ 下新建 DbCountTask.java，要求：
  1. 用 @Component + @JobExecutor(name = "dbCountTask") 声明
  2. 打印当前日期和一条统计日志（用 SnailJobLog.REMOTE）
  3. 返回 ExecuteResult.success("统计完成")
  4. 在 Server 控制台创建 cron 为 0 0 2 * * ?（每天凌晨 2 点）的任务
  5. 用 SnailJobLog.REMOTE 输出一条关键日志，证明日志回传

@Component

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        String today = LocalDate.now().toString();
        SnailJobLog.REMOTE.info("开始统计 {} 的数据量", today);

}

六、检验一下（提取练习）

不要回看上文，凭记忆作答。

Q1. @JobExecutor(name = "xxx") 中的 name 和 SnailJob 服务端控制台创建任务时填的「执行器名称」是什么关系？

A. 没关系，随便填 B. 必须完全一致——服务端按 name 找到对应的 Bean 来执行 C. name 是类名，执行器名称是方法名 D. 服务端自动发现，不需要手动填
看答案B。@JobExecutor 的 name 是任务执行器的唯一标识，服务端创建任务时填的「执行器名称」必须与此完全一致，否则调度指令到达客户端后找不到对应的 Bean。

Q2. 以下哪种场景最适合使用广播任务？

A. 每天凌晨生成报表 B. 刷新所有节点的本地缓存 C. 按用户 ID 分片处理数据 D. 多步骤工作流编排
看答案B。广播任务的特点是「所有在线节点同时执行」，适合刷新缓存、下发配置等需要在每台机器上都做一遍的操作。

Q3. MapReduce 比 Map 多了什么？

A. 多了 cron 表达式 B. 多了一个 @ReduceExecutor 汇总阶段，所有子任务完成后自动调用它聚合结果 C. 多了服务端手动触发 D. 多了日志功能
看答案B。@ReduceExecutor 在所有 Map 子任务完成后执行，通过 reduceArgs.getMapResult() 获取全部子任务的返回值列表，实现聚合。

Q4. DAG 工作流中，任务之间如何传递数据？

A. 通过数据库 B. 通过 jobArgs.getWfContext() 读取上游数据，通过 jobArgs.appendContext() 写入给下游 C. 通过 HTTP 接口 D. 通过全局变量
看答案B。JobArgs 内置工作流上下文（wfContext），上游任务通过 appendContext(key, value) 写入，下游通过 getWfContext().get(key) 读取，实现 DAG 数据传递。
