第 2 课 · 任务类型与实战示例 动手实战

本节目标：学完你能写出一个 SnailJob 定时任务并理解它的完整生命周期——从 @JobExecutor 声明、服务端创建任务、客户端接收调度、jobExecute() 执行、到 ExecuteResult 和 SnailJobLog.REMOTE 回传。同时能讲清 SnailJob 支持的 6 种任务类型（普通、广播、静态分片、Map、MapReduce、DAG 工作流）各自的特点和适用场景。

上一课我们搞懂了 SnailJobConfig 如何把 SnailJob 客户端接入 RuoYi。本课讲另一面：业务方怎么写一个 SnailJob 任务，以及有哪些任务模式可用。全部示例代码来自仓库 ruoyi-modules/ruoyi-job/src/main/java/org/dromara/job/snailjob/。

一、一个任务的生命周期：7 步走通全链路

在开始写代码之前，先理解一个 SnailJob 任务从诞生到执行完毕的完整链路：

【写代码】 【启动】 【调度】 【观察】 ruoyi-job 模块 应用启动 + Server 运行 Server 按 cron 触发 Server 控制台 ① @Component ④ SnailJobConfig 加载 ⑥ Server 选择在线客户端 ⑦ 查看调度日志 @JobExecutor(name=...) @EnableSnailJob 生效 通过 Netty 下发指令 → 执行状态 public ExecuteResult ↓ 「执行 testJobExecutor」 → 执行耗时 jobExecute(JobArgs) { ③ SnailJob Server 运行中 ↓ → 远程日志 ... 等待客户端注册心跳 ⑤ 客户端执行器收到指令 return success; ↓ → name 匹配找到 Bean } ② 客户端连接 Server → 反射调用 jobExecute() 注册心跳 + 上报执行器列表 → 日志经 SnailLogbackAppender 进入「等待调度」状态 → 上报到 Server → 返回 ExecuteResult

这条链路中的核心概念有四个，后面展开：@JobExecutor（声明）、JobArgs（输入）、ExecuteResult（输出）、SnailJobLog（日志）。

二、核心概念详解

2.1 @JobExecutor —— 任务的唯一标识

// 来自 com.aizuda.snailjob.client.job.core.annotation.JobExecutor
@Component
@JobExecutor(name = "testJobExecutor")   // name 是全局唯一标识
public class TestAnnoJobExecutor {
    public ExecuteResult jobExecute(JobArgs jobArgs) { ... }
}

这个注解有三个要点：

要点说明常见错误
name 必须唯一 name 是任务执行器在 SnailJob 系统中的唯一标识。服务端创建任务时填的「执行器名称」必须与此完全一致（区分大小写）。 服务端填了 testjobexecutor，代码里写的是 testJobExecutor——大小写不匹配，调度指令到达后找不到执行器。
必须是 Spring Bean 只有被 Spring 管理的 Bean（@Component / @Service 等）才能被 SnailJob 扫描到并注册。 忘了加 @Component，SnailJob 启动时发现不了这个类。
方法签名约定 方法名不需要固定叫 jobExecute，但 参数类型必须是 JobArgs，返回值必须是 ExecuteResult。SnailJob 通过反射调用，类型不匹配会抛异常。 写成 public void execute(String args)——SnailJob 反射调用时找不到匹配方法。
还有另一种写法：继承 AbstractJobExecutor——仓库里的 TestClassJobExecutor 用的就是这种方式，覆写 doJobExecute(JobArgs) 方法。效果一样，但注解方式更灵活（不强制继承）。

2.2 JobArgs —— 任务执行的上下文数据

JobArgs 是 SnailJob 框架在调度触发时自动组装并传入 jobExecute() 的参数对象。它携带了三类信息：

方法返回类型含义谁来设置使用场景
getJobParams() Object 任务参数——在服务端控制台创建任务时填入的 JSON/字符串参数 运维在 SnailJob Server 控制台填写 静态分片（传入 ID 范围 "1,5000"）、动态配置（传入处理阈值）
getWfContext() Map<String, Object> 工作流上下文——DAG 工作流中上游任务传给下游的数据 上游任务通过 jobArgs.appendContext(key, value) 写入 对账流水线：支付宝账单任务 → 微信账单任务 → 汇总任务
getShardingArgs() Object 分片参数——SnailJob 自动分配的分片信息（当前分片索引、总分片数等） SnailJob 框架自动注入 分片任务中判断「我负责处理第几片数据」

示例：在 TestStaticShardingJob 中，服务端传入 jobParams = "1,5000"，代码这样解析：

public ExecuteResult jobExecute(JobArgs jobArgs) {
    String jobParams = Convert.toStr(jobArgs.getJobParams()); // → "1,5000"
    String[] split = jobParams.split(",");
    Long fromId = Long.parseLong(split[0]);                   // → 1
    Long toId = Long.parseLong(split[1]);                     // → 5000
    // 处理 ID 范围 [fromId, toId] 的数据
}

2.3 ExecuteResult —— 向服务端报告结果

ExecuteResult 是任务执行结果的封装，必须返回它。SnailJob 客户端会把返回值上报给 Server，Server 据此记录任务的执行状态。

// 报告成功——可附带结果数据（任意对象，会被序列化上报）
ExecuteResult.success("分片处理完成");
ExecuteResult.success(someDataObject);

// 报告失败——传入失败原因，Server 控制台可见
ExecuteResult.failure("数据库连接超时");
ExecuteResult.failure("任务执行失败");
返回值Server 控制台看到的后续行为
ExecuteResult.success("xxx") 执行状态：成功 / 执行结果：xxx 记录成功，等待下一次 cron 触发
ExecuteResult.failure("xxx") 执行状态：失败 / 失败原因：xxx 记录失败。如果任务配置了重试策略，可能会自动重试
抛出未捕获的异常 执行状态：失败 / 错误信息：异常堆栈 同上——SnailJob 会捕获异常并记录
注意：TestBroadcastJob 里故意 throw new RuntimeException(...) 来演示失败场景。在生产代码中，你应该用 ExecuteResult.failure() 主动报告失败，而不是靠抛异常——抛异常会丢失部分日志上下文。

2.4 SnailJobLog —— 双通道日志

上一课讲过 SnailLogbackAppender 挂载后所有日志都会上报，但 SnailJobLog 提供了更精细的控制：

通道输出目的地适用场景代码示例
SnailJobLog.LOCAL 仅本地日志文件（不上报 Server） 调试日志、敏感信息、临时输出 SnailJobLog.LOCAL.info("本地调试信息:{}", data);
SnailJobLog.REMOTE 本地文件 + SnailJob Server 控制台 任务执行的关键节点日志——开始处理、处理进度、处理完成、异常信息。运维需要在控制台看到的内容。 SnailJobLog.REMOTE.info("端口:{},partitionTotal:{}", port, total);
设计洞察 —— 为什么不只用 @Slf4j 的 log.info()？
因为你把 SnailLogbackAppender 挂到了根 Logger 上，用 @Slf4j 的 log.info() 确实也能上报。但 SnailJobLog.REMOTE 明确表达了意图：「这条日志是给运维在 Server 控制台看的」。代码可读性上，看到 REMOTE 就知道这条日志会出现在调度记录里，看到 LOCAL 就知道只是本机调试用。

三、六种任务类型详解

仓库的 ruoyi-modules/ruoyi-job/src/main/java/org/dromara/job/snailjob/ 下有 7 个示例类（外加一个 TestClassJobExecutor 是继承方式写任务），覆盖了 6 种 SnailJob 任务模式。以下是每种模式的完整剖析：

3.1 普通任务 —— 最简模式

示例文件：TestAnnoJobExecutor.java

@Component
@JobExecutor(name = "testJobExecutor")
public class TestAnnoJobExecutor {

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        SnailJobLog.LOCAL.info("testJobExecutor. jobArgs:{}",
            JsonUtil.toJsonString(jobArgs));           // 只写本地
        SnailJobLog.REMOTE.info("testJobExecutor. jobArgs:{}",
            JsonUtil.toJsonString(jobArgs));            // 本地 + 上报
        return ExecuteResult.success("测试成功");
    }
}

特点：

  • 任务被投递到一个在线客户端节点执行一次
  • 不需要任何额外注解，一个 @JobExecutor + 一个 jobExecute 方法即可
  • 路由策略（轮询、随机、故障转移等）在服务端控制台配置

适用场景：生成日报、清理临时文件、同步缓存数据——任何「定时做一件事」的需求。

3.2 广播任务 —— 所有节点同时执行

示例文件：TestBroadcastJob.java

@Slf4j
@Component
@JobExecutor(name = "testBroadcastJob")
public class TestBroadcastJob {

    @Value("${snail-job.port}")
    private int clientPort;        // 每个节点端口不同，证明广播到了所有节点

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        int randomInt = RandomUtil.randomInt(100);
        log.info("随机数: {}", randomInt);
        SnailJobLog.REMOTE.info("随机数:{},客户端端口:{}",
            randomInt, clientPort);
        if (randomInt < 50) {
            throw new RuntimeException("随机数小于50，任务执行失败");
        }
        return ExecuteResult.success("随机数大于50，任务执行成功");
    }
}

特点：

  • 同一任务被广播到所有在线客户端节点，各节点并行执行
  • 代码写法与普通任务一模一样——区别只在于服务端创建任务时选择「广播」路由策略
  • 各节点执行结果独立上报，互不影响

适用场景：刷新所有节点本地缓存、向所有节点下发最新配置、集群全量健康检查、同时清理所有节点的临时文件。

3.3 静态分片 —— 服务端指定分片参数

示例文件：TestStaticShardingJob.java

@Component
@JobExecutor(name = "testStaticShardingJob")
public class TestStaticShardingJob {

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        String jobParams = Convert.toStr(jobArgs.getJobParams());
        // 服务端创建任务时填入参数 "1,5000"
        SnailJobLog.LOCAL.info("开始执行分片任务,参数:{}", jobParams);
        String[] split = jobParams.split(",");
        Long fromId = Long.parseLong(split[0]);
        Long toId = Long.parseLong(split[1]);

        // 对这个 ID 范围的数据做批量处理
        try {
            SnailJobLog.REMOTE.info("开始对id范围:{}进行加密处理",
                fromId + "-" + toId);
            Thread.sleep(3000);                          // 模拟耗时操作
            SnailJobLog.REMOTE.info("对id范围:{}进行加密处理完成",
                fromId + "-" + toId);
        } catch (InterruptedException e) {
            return ExecuteResult.failure("任务执行失败");
        }
        return ExecuteResult.success("执行分片任务完成");
    }
}

特点：

  • 分片参数由运维在服务端控制台手动填写（如 "1,5000" 表示处理 ID 1~5000）
  • 代码通过 jobArgs.getJobParams() 拿到参数后自行解析
  • 适合分片逻辑固定、范围可预知的场景

适用场景：百万级数据批量加密、大表数据迁移（按月份/ID段分批）、定时对账（按账户类型分片）。

3.4 Map 任务 —— 代码动态分片，不汇总

示例文件：TestMapJobAnnotation.java

@SuppressWarnings({"unchecked", "rawtypes"})
@Component
@JobExecutor(name = "testMapJobAnnotation")
public class TestMapJobAnnotation {

    // 根任务：把数据拆成多个分片，分发给集群节点
    @MapExecutor
    public ExecuteResult doJobMapExecute(MapArgs mapArgs,
                                          MapHandler mapHandler) {
        int partitionSize = 50;
        List<Integer> sourceList =
            IntStream.rangeClosed(1, 200).boxed().toList();  // 200 条数据
        List<List<Integer>> partition =
            StreamUtils.groupByKey(sourceList,
                i -> (i - 1) / partitionSize)               // 拆成 4 份
                .values().stream().toList();
        SnailJobLog.REMOTE.info("端口:{}完成分配任务",
            SpringUtil.getProperty("server.port"));
        return mapHandler.doMap(partition, "doCalc");       // 分发子任务
    }

    // 子任务：每个分片独立执行
    @MapExecutor(taskName = "doCalc")
    public ExecuteResult doCalc(MapArgs mapArgs) {
        List<Integer> sourceList =
            (List<Integer>) mapArgs.getMapResult();         // 拿到自己那批数据
        int partitionTotal =
            sourceList.stream().mapToInt(i -> i).sum();     // 累加计算
        ThreadUtil.sleep(3, TimeUnit.SECONDS);
        SnailJobLog.REMOTE.info("端口:{},partitionTotal:{}",
            SpringUtil.getProperty("server.port"), partitionTotal);
        return ExecuteResult.success(partitionTotal);
    }
}

特点：

  • 分片由代码逻辑动态决定（不是服务端静态配置）——数据量不确定时也能自动拆分
  • @MapExecutor（无 taskName）是根任务，负责拆分数据；@MapExecutor(taskName = "xxx") 是子任务，处理一个分片
  • mapHandler.doMap(data, taskName) 将数据分发给集群节点并行执行
  • 不关心子任务的结果汇总——各算各的

适用场景：批量推送（按用户列表拆分，各节点推各自的用户）、数据清洗（动态按行数拆分文件）、并行导出（分页查询后各节点写各自的 Excel）。

3.5 MapReduce 任务 —— 分片后汇总结果

示例文件：TestMapReduceAnnotation1.java

@SuppressWarnings({"unchecked", "rawtypes"})
@Component
@JobExecutor(name = "testMapReduceAnnotation1")
public class TestMapReduceAnnotation1 {

    @MapExecutor                                     // 根任务：拆分
    public ExecuteResult rootMapExecute(MapArgs mapArgs,
                                         MapHandler mapHandler) {
        int partitionSize = 50;
        List<Integer> sourceList =
            IntStream.rangeClosed(1, 200).boxed().toList();
        List<List<Integer>> partition =
            StreamUtils.groupByKey(sourceList,
                i -> (i - 1) / partitionSize)
                .values().stream().toList();
        SnailJobLog.REMOTE.info("端口:{}完成分配任务",
            SpringUtil.getProperty("server.port"));
        return mapHandler.doMap(partition, "doCalc");
    }

    @MapExecutor(taskName = "doCalc")                // 子任务：并行计算
    public ExecuteResult doCalc(MapArgs mapArgs) {
        List<Integer> sourceList =
            (List<Integer>) mapArgs.getMapResult();
        int partitionTotal =
            sourceList.stream().mapToInt(i -> i).sum();
        ThreadUtil.sleep(3, TimeUnit.SECONDS);
        SnailJobLog.REMOTE.info("端口:{},partitionTotal:{}",
            SpringUtil.getProperty("server.port"), partitionTotal);
        return ExecuteResult.success(partitionTotal);
    }

    @ReduceExecutor                                  // 汇总：聚合所有子任务结果
    public ExecuteResult reduceExecute(ReduceArgs reduceArgs) {
        // reduceArgs.getMapResult() 是所有子任务返回值的集合
        int reduceTotal = reduceArgs.getMapResult()
            .stream()
            .mapToInt(i -> Integer.parseInt((String) i))
            .sum();
        SnailJobLog.REMOTE.info("端口:{},reduceTotal:{}",
            SpringUtil.getProperty("server.port"), reduceTotal);
        return ExecuteResult.success(reduceTotal);
    }
}

与 Map 任务的关键区别：增加了一个 @ReduceExecutor 方法。所有 @MapExecutor 子任务完成后，SnailJob 自动调用 @ReduceExecutor 方法，并通过 reduceArgs.getMapResult() 传入所有子任务的返回值集合，实现聚合。

适用场景：分布式统计（各节点统计各自分片的数据量，汇总得总量）、大规模数据聚合求 TopN、并行计算后合并结果。

3.6 DAG 工作流 —— 多任务编排

示例文件：AlipayBillTask.java、WechatBillTask.java、SummaryBillTask.java

// AlipayBillTask.java —— 第一个节点：下载支付宝账单
@JobExecutor(name = "alipayBillTask")
public class AlipayBillTask {
    public ExecuteResult jobExecute(JobArgs jobArgs) {
        // 从 wfContext 读取上游传入的对账日期（工作流启动参数）
        String settlementDate =
            (String) jobArgs.getWfContext().get("settlementDate");
        // 下载并处理支付宝账单...
        BillDTO billDTO = new BillDTO(...);
        // 把结果写入上下文，传给下一个节点
        jobArgs.appendContext("alipay", JsonUtils.toJsonString(billDTO));
        return ExecuteResult.success(billDTO);
    }
}

// SummaryBillTask.java —— 最后一个节点：汇总账单
@JobExecutor(name = "summaryBillTask")
public class SummaryBillTask {
    public ExecuteResult jobExecute(JobArgs jobArgs) {
        // 读取上游两个节点的结果
        Map<String, Object> context = jobArgs.getWfContext();
        BillDTO alipay = JsonUtils.parseObject(
            (String) context.get("alipay"), BillDTO.class);
        BillDTO wechat = JsonUtils.parseObject(
            (String) context.get("wechat"), BillDTO.class);
        // 汇总生成报表...
        return ExecuteResult.success("汇总完成");
    }
}

特点：

  • 通过 jobArgs.getWfContext() 读取上游数据，jobArgs.appendContext(key, value) 写入给下游
  • 任务之间的执行顺序（DAG 拓扑）在 SnailJob Server 控制台通过拖拽方式编排
  • 每个节点本身就是一个普通的 @JobExecutor——编排在服务端做，代码不感知拓扑结构

适用场景：对账流水线（下载账单 → 比对差异 → 生成报告）、ETL 数据管道（抽取 → 清洗 → 转换 → 加载）、多步骤审批流程（提交 → 审核 → 归档）。

四、任务类型速查表

需求关键词任务类型代码特征服务端配置示例类
定时执行一件事 普通任务 @JobExecutor + jobExecute() 路由策略：轮询/随机 TestAnnoJobExecutor
所有节点都执行 广播任务 同普通任务（代码无区别） 路由策略：广播 TestBroadcastJob
按固定范围拆分 静态分片 解析 jobArgs.getJobParams() 填写任务参数（如 "1,5000"） TestStaticShardingJob

五、完整调度链路：从写代码到看日志

结合上一课的配置知识和本课的任务写法，整合成一套操作流程：

1写任务代码
ruoyi-modules/ruoyi-job/.../snailjob/
新建类 → 加 @Component + @JobExecutor(name = "myTask") → 写 jobExecute(JobArgs) 方法 → 用 SnailJobLog.REMOTE 打关键日志 → 返回 ExecuteResult。
2配置 yml
application-dev.yml 或 application-prod.yml
snail-job.enabled: true（关键！否则一切白写）→ 配好 group、token、server.host/port、namespace。
3启动 SnailJob Server
ruoyi-extend/ruoyi-snailjob-server
运行 SnailJobServerApplication。如果没有 Server 在运行，客户端的 @EnableSnailJob 会连不上，但应用本身不会因此启动失败（SnailJob 客户端有重连机制）。
4启动业务应用
ruoyi-admin
观察启动日志：应有「SnailJob 客户端启动成功」「注册执行器: myTask」等日志。如果出现「连接 Server 失败」或「token 验证失败」，回第 2 步检查配置。
5Server 控制台创建任务
浏览器访问 SnailJob Server 控制台
「任务管理」→「创建任务」→ 核心填写项：执行器名称 = myTask（必须与 @JobExecutor 的 name 完全一致）、cron 表达式、任务参数（可选）、路由策略。
6观察调度执行
Server 控制台 → 调度日志
cron 到点后，Server 下发指令 → 客户端执行 → 返回 ExecuteResult → 日志经 SnailLogbackAppender 回传 → 控制台可见完整的执行状态和远程日志。

六、调试指南：任务不执行怎么办？

按以下顺序逐一排查，这是实际开发中最常见的 6 个坑：

排查项检查方法典型错误
① snail-job.enabled 看 yml 中是否为 true dev 环境默认 false，忘记在生产 yml 覆盖
② SnailJob Server 是否运行 访问 Server 控制台，看能否打开 Server 没启动，客户端连不上
③ token 是否一致 客户端 yml 的 token 与 Server 数据库 sj_group_config 表的 token 是否相同 token 不匹配 → Server 拒绝连接 → 启动日志有「token 验证失败」
④ group 是否存在 检查 Server 后台「组管理」中是否有 yml 中配置的 group 名 客户端连上了但属于不存在的组 → 收不到任务
⑤ @JobExecutor name 是否匹配 代码中的 name 与 Server 上任务配置的「执行器名称」完全一致（含大小写） 代码是 myTask，服务端填了 mytask
⑥ 任务是否已启用 Server 控制台查看任务状态是否为「启用」 创建了任务但状态是「停用」——不会触发调度

七、检验一下（提取练习）

Q1. @JobExecutor(name = "xxx") 的 name 与 SnailJob Server 控制台创建任务时的「执行器名称」是什么关系？

A. 没有关系，随便填 B. 必须完全一致（含大小写）——服务端按此名称找到对应 Bean 来执行 C. name 是类名，执行器名称是方法名 D. 服务端自动发现，不需要手动填
看答案B。@JobExecutor 的 name 是执行器的唯一标识，服务端创建任务时填的「执行器名称」必须与此完全一致，否则调度指令到达后找不到对应的 Bean，任务无法执行。

Q2. 以下哪种场景不适合使用广播任务？

A. 刷新所有节点的本地缓存 B. 向所有节点下发最新配置 C. 每天凌晨生成一份日报（只需要一个节点执行） D. 集群全量健康检查
看答案C。生成日报只需要执行一次——如果做成广播任务，每个节点都会生成一份日报，造成重复。这种情况用普通任务 + 轮询路由即可。

Q3. 静态分片和 Map 任务的核心区别是什么？

A. 没有区别，只是叫法不同 B. 静态分片参数由运维在控制台填写，适合范围固定的场景；Map 任务由代码动态拆分数据，适合数据量不确定的场景。此外 Map 的拆分逻辑和子任务逻辑写在同一个类里。 C. 静态分片只能处理数字，Map 只能处理字符串 D. 静态分片需要 @MapExecutor，Map 不需要
看答案B。静态分片：分片参数由人在控制台填写（如 "1,5000"），通过 jobArgs.getJobParams() 获取。Map 任务：分片逻辑写在代码里，通过 mapHandler.doMap() 动态拆分并分发。前者适合预知范围，后者适合数据量运行时才知道。

Q4. MapReduce 比 Map 多了什么？

A. 多了 cron 表达式 B. 多了一个 @ReduceExecutor 方法——所有 Map 子任务完成后自动调用，通过 reduceArgs.getMapResult() 获取所有子任务的返回值列表，实现聚合 C. 多了自动重试 D. 多了日志功能
看答案B。@ReduceExecutor 在所有 Map 子任务执行完毕后自动触发，reduceArgs.getMapResult() 包含所有子任务返回值的 List（Map 阶段每个子任务 return 的值）。适合「先拆分并行处理，再汇总得出最终结果」的场景。

Q5. SnailJobLog.LOCAL 和 SnailJobLog.REMOTE 的区别是什么？

A. 没区别，只是名字不一样 B. LOCAL 只写到本地日志文件，REMOTE 写到本地 + 回传到 SnailJob Server 控制台 C. LOCAL 写控制台，REMOTE 写文件 D. LOCAL 是同步的，REMOTE 是异步的
看答案B。LOCAL 仅本地文件，适合调试信息和敏感内容；REMOTE 本地 + Server 控制台，适合需要运维在 Web UI 看到的关键日志。由于 SnailLogbackAppender 挂到了根 Logger，使用 @Slf4j 的 log.info() 也会上报——但用 REMOTE 能明确表达日志意图。

Q6. DAG 工作流中，任务之间如何传递数据？

A. 通过数据库共享表 B. 通过 jobArgs.getWfContext() 读取上游写入的数据，通过 jobArgs.appendContext(key, value) 写入给下游 C. 通过 HTTP 接口回调 D. 通过 Redis 缓存
看答案B。SnailJob 的工作流上下文（wfContext）是框架内置的 DAG 传参机制。上游任务 appendContext(key, value)，下游任务 getWfContext().get(key)。任务的执行顺序（DAG 拓扑）在服务端控制台拖拽编排。
