第 5 课 · Lambda 查询构建器：链式 API 设计 查询入口

本节目标：学完你能说出三种查询构建入口的区别和适用场景，理解 LambdaQueryBuilder 的链式 API 设计模式，掌握 LambdaQueryCondition 接口的便捷方法机制，以及 LambdaJoinQueryBuilder 为何必须使用表别名。

一、总览：三种查询入口

RuoYi-Vue-Plus 提供三种查询构建入口，覆盖从单表查询到多表联查、从独立查询到 Mapper 链式的全部场景：

入口静态工厂底层包装器适用场景
LambdaQueryBuilder QueryBuilder.lambda(Class<T>) AggregateLambdaQueryWrapper<T> 单表查询：条件筛选 + 聚合统计 + 子查询
LambdaJoinQueryBuilder QueryBuilder.lambdaJoin(Class<T>)
QueryBuilder.lambdaJoin(String, Class<T>)		 MPJLambdaWrapper<T> 多表联表查询：LEFT JOIN + 跨表字段映射
LambdaCrudChainWrapper mapper.lambda() LambdaCrudChainWrapper<T, V> Mapper 级链式操作：查询 + 增删改全链路
设计要点：QueryBuilder 是静态工厂类（final class + 私有构造），只暴露 lambda() 和 lambdaJoin() 两个对外方法。它不直接暴露底层包装器，而是通过 Builder 封装了一层，提供了比原生 MyBatis-Plus 更语义化的 API。

二、LambdaQueryBuilder 深度拆解

2.1 包装器模式

LambdaQueryBuilder<T> 内部持有一个 LambdaQueryWrapper<T> wrapper 字段，但实际运行时类型是 AggregateLambdaQueryWrapper<T>（见第五节详述）。它不继承任何 MyBatis-Plus 类，而是通过组合 + 委派的方式将底层包装器封装起来：

QueryBuilder.lambda(SysUser.class) → new AggregateLambdaQueryWrapper<>(SysUser.class) → 继承 LambdaQueryWrapper 的能力（eq/ne/gt/like/...） → 扩展 aggregateSqlSelect 字段（聚合 SQL 追加） → new LambdaQueryBuilder<>(wrapper) → 持有 wrapper，对外暴露 100+ 链式方法 → 终结方法：build() 返回 LambdaQueryWrapper<T>

核心设计原则："不修改 MyBatis-Plus 源码，通过包装增强"。所有条件方法最终委派给 wrapper.eq()、wrapper.like() 等原生方法，但 LambdaQueryBuilder 额外提供了便捷方法（eqIfPresent）和聚合方法（selectSum）。

2.2 方法分类总览

LambdaQueryBuilder 的 100+ 方法可按功能分为六大类：

分类典型方法说明
字段选择 select(SFunction...) 指定查询列，调用前会自动 resetAggregateSelect() 清空之前的聚合模式
聚合选择 selectSum/Max/Min/Avg/Count/CountAll/CountDistinct(column, alias) 每个方法提供 3 种重载：无别名（取 fieldName）、String 别名、SFunction 别名。内部委托 selectAggregate() → AggregateSelectUtils.aggregateSelect()
子查询 selectSub / eqSub / inSub / notInSub / existsSub / notExistsSub 接收 (Class<S>, Consumer<SubQuery<S>>, alias)，内部构建 SubQuery → build() → 拼入 SQL
标准条件 eq/ne/gt/ge/lt/le/like/notLike/likeLeft/likeRight/between/notBetween/in/notIn/isNull/isNotNull 每种方法有两套重载：无 condition 参数（必加）和有 condition 参数（条件添加）
高级条件 allEq/and/or/nested/not/groupBy/having/orderBy/orderByAsc/orderByDesc 支持嵌套逻辑、分组聚合过滤、多条件排序
扩展能力 exists/notExists/eqSql/inSql/notInSql/apply/func/comment/last 拼接原始 SQL、添加注释、尾部追加等高级玩法

2.3 链式 API 的设计技巧

LambdaQueryBuilder 采用了 Java 中经典的 Fluent API 模式，每个条件方法都返回 this（类型为 LambdaQueryBuilder<T>），使得调用可以无限链下去：

// 典型链式调用：条件筛选 + 排序 + 分页
QueryBuilder.lambda(SysUser.class)
    .eq(SysUser::getStatus, "0")                    // 条件：状态正常
    .likeIfText(SysUser::getUserName, keyword)      // 便捷：keyword 有值才加模糊
    .betweenIfPresent(SysUser::getCreateTime, begin, end)  // 便捷：起止均非空才加
    .orderByAsc(SysUser::getCreateTime)             // 排序：按创建时间升序
    .build();                                       // 终结：返回 LambdaQueryWrapper
tip：注意区分"条件方法"与"终结方法"。eq/like/between 等方法返回 Builder 本身，是条件方法；而 build() 返回底层 LambdaQueryWrapper，是终结方法。这与 MyBatis-Plus 原生的 QueryWrapper 不同——后者本身就是一个 Wrapper，可以直接传给 Mapper。

2.4 聚合方法的参数重载策略

每个聚合选择方法（如 selectSum）都提供 3 种重载：

// 重载 1：无 alias，自动从 column 解析 fieldName .selectSum(SysUser::getAge) → aliasName(SysUser::getAge) → "age" // 重载 2：String alias .selectSum(SysUser::getAge, "totalAge") → checkAlias("totalAge") → "totalAge" // 重载 3：SFunction alias（Lambda 类型安全） .selectSum(SysUser::getAge, UserStatVo::getTotalAge) → aliasName(UserStatVo::getTotalAge) → "totalAge"

最终都汇聚到 selectAggregate(function, column, alias) 私有方法，由 AggregateSelectUtils.aggregateSelect() 生成 SUM(age) AS totalAge。

三、LambdaQueryCondition 接口：便捷方法的秘密

3.1 接口设计

LambdaQueryCondition<T, Children> 是一个泛型接口，定义了两个参数：

  • T — 实体类型（如 SysUser）
  • Children — 链式返回类型（这是实现 Fluent API 的关键）

LambdaQueryBuilder 实现此接口时指定 Children 为自身：LambdaQueryBuilder<T> implements LambdaQueryCondition<T, LambdaQueryBuilder<T>>。这样接口中所有 default 方法的返回值类型都能正确推导为 LambdaQueryBuilder<T>，链式调用不会断。

同样，LambdaCrudChainWrapper<T, V> 也实现了 LambdaQueryCondition<T, LambdaCrudChainWrapper<T, V>>，复用同一套便捷方法。

3.2 *IfPresent 系列方法

这些 default 方法的本质是条件包装：将"是否添加"的判断逻辑从调用方转移到接口内部。例如：

// 接口定义 default Children eqIfPresent(SFunction<T, ?> column, Object value) { return eq(value != null, column, value); // value 为 null 时不添加条件 } // 使用对比 // 传统方式（调用方自己判空）： if (dto.getStatus() != null) { builder.eq(SysUser::getStatus, dto.getStatus()); } // 便捷方式（接口内判空）： builder.eqIfPresent(SysUser::getStatus, dto.getStatus());
方法后缀判断逻辑个数
*IfPresentvalue != nulleq/ne/gt/ge/lt/le → 6 个
*IfTextStringUtils.isNotBlank(value)eq/ne/like/notLike/likeLeft/likeRight → 6 个
*IfPresent（区间）begin != null && end != nullbetween/notBetween → 2 个
*IfNotEmptyvalues != null && !values.isEmpty() 或 .length > 0in(Collection)/in(Object...)/notIn(Collection)/notIn(Object...) → 4 个
betweenParams从 Map<String, Object> 中按 key 取值判空between → 1 个

3.3 findInSet 的委托机制

findInSet 方法用于处理 MySQL 的 FIND_IN_SET 函数场景（如逗号分隔的多选值匹配）。它不直接写 SQL，而是委托给 DataBaseHelper.findInSet(value, column)：

// LambdaQueryCondition 中的 default 方法 default Children findInSet(boolean condition, Object value, String column) { return apply(condition, DataBaseHelper.findInSet(value, column)); } // DataBaseHelper.findInSet 根据当前数据库类型生成对应 SQL： // MySQL: FIND_IN_SET(#{value}, column) // Oracle: ...(不同实现) // PG: ...(不同实现)

这种接口默认方法 + 外部工具类委托的模式，让便捷方法可以跨多种 Builder 共享，同时保持接口轻量（实现类只需关注核心的 eq(condition, column, value) 等基础方法）。

设计优势：LambdaQueryBuilder 和 LambdaCrudChainWrapper 都实现了 LambdaQueryCondition，两者自动获得 20+ 个便捷方法（eqIfPresent/neqIfText/likeLeftIfText/...），无需各自重复编写。这是 Java 8 interface default method 在 DSL 构建器中的经典应用。

四、LambdaJoinQueryBuilder：联表查询

4.1 为什么需要 MPJ

MyBatis-Plus 原生只支持单表的条件构造，当需要 LEFT JOIN 时只能退回到 XML 或注解 SQL。MyBatis-Plus-Join (MPJ) 是一个第三方扩展，在 MPJLambdaWrapper>T< 上提供了 Lambda 风格的 JOIN 语法。RuoYi-Vue-Plus 通过 LambdaJoinQueryBuilder 将它封装了起来。

依赖链路： QueryBuilder.lambdaJoin(SysUser.class) → JoinWrappers.lambda(SysUser.class) ← MPJ 库提供 → new MPJLambdaWrapper<>(SysUser.class) ← MPJ 库提供 → new LambdaJoinQueryBuilder<>(MPJLambdaWrapper) ← RuoYi 封装

4.2 表别名：为什么必须传

与单表的 LambdaQueryBuilder 不同，LambdaJoinQueryBuilder 的所有条件方法都强制要求传入表别名。这不是设计偏好，而是多表场景下的硬需求：

// 多表联查时，两表可能有同名字段（如 user_id, create_time）
QueryBuilder.lambdaJoin("u", SysUser.class)     // 主表别名"u"
    .leftJoin(SysDept.class, "d",                 // 关联表别名"d"
        SysDept::getDeptId,                       // d.dept_id
        SysUser::getDeptId)                       // u.dept_id
    .eq("u", SysUser::getStatus, "0")            // where u.status = '0'
    .likeIfText("d", SysDept::getDeptName, kw)   // and d.dept_name like '%kw%'
    .list(SysUserVo.class);

条件方法签名规律（以 eq 为例）：

// 单表（LambdaQueryBuilder）：列推导自动带表名 eq(SFunction<T, ?> column, Object value) → wrapper.eq(column, value) // 联表（LambdaJoinQueryBuilder）：必须手动指定别名 eq(String alias, SFunction<S, ?> column, Object value) → wrapper.eq(alias, column, value)

4.3 selectAs：跨表字段映射

联表查询的结果通常需要映射到一个自定义 VO 上。MPJ 提供了 selectAs 方法，将不同表的字段映射到同一个返回对象的属性上：

// 联表查询 + 字段映射到 VO
QueryBuilder.lambdaJoin("u", SysUser.class)
    .leftJoin(SysDept.class, "d", SysDept::getDeptId, SysUser::getDeptId)
    .selectAs("u", SysUser::getUserId, UserDeptVo::getUserId)     // u.user_id → userId
    .selectAs("u", SysUser::getUserName, UserDeptVo::getUserName)  // u.user_name → userName
    .selectAs("d", SysDept::getDeptName, UserDeptVo::getDeptName)  // d.dept_name → deptName
    .list(UserDeptVo.class);

selectAs 有三种重载：

重载说明
selectAs(column, alias)不指定表别名，MPJ 自动推导
selectAs(tableAlias, column, alias)指定表别名 + 返回值映射
selectAs(sqlColumn, alias)原始 SQL 片段映射（适用于函数/表达式）

4.4 终结方法：list / page / count

与 LambdaQueryBuilder 仅提供 build() 不同，LambdaJoinQueryBuilder 内置了终结查询方法：

// 直接返回结果 .list(Class<R> resultClass) → wrapper.list(resultClass) .page(P page, Class<R>) → wrapper.page(page, resultClass) .count() → wrapper.count() // 返回底层包装器（用于需要进一步处理的场景） .build() → 返回 MPJLambdaWrapper<T>
区别：LambdaQueryBuilder 只返回 LambdaQueryWrapper，需要调用方自己传给 Mapper；LambdaJoinQueryBuilder 内置了查询执行，因为 MPJ 的 list(page/count) 方法需要知道返回类型，这是 MPJ 框架的设计特性。

五、代码对比：三种写法

5.1 场景：查询某部门下状态正常的用户

// 传统 MyBatis XML 写法
<select id="selectUsersByDept" resultType="SysUser">
    SELECT * FROM sys_user
    WHERE dept_id = #{deptId}
    AND status = '0'
    <if test="userName != null and userName != ''">
        AND user_name LIKE CONCAT('%', #{userName}, '%')
    </if>
    ORDER BY create_time DESC
</select>
// LambdaQueryBuilder 写法（单表场景）
QueryBuilder.lambda(SysUser.class)
    .eq(SysUser::getDeptId, deptId)
    .eq(SysUser::getStatus, "0")
    .likeIfText(SysUser::getUserName, userName)
    .orderByDesc(SysUser::getCreateTime)
    .build();

// 优势：类型安全（字段名不会写错）、无 XML、链式可读

5.2 场景：联表查询用户及其部门名称

// 传统 MyBatis XML 写法
<select id="selectUserDeptList" resultType="UserDeptVo">
    SELECT u.user_id, u.user_name, d.dept_name
    FROM sys_user u
    LEFT JOIN sys_dept d ON d.dept_id = u.dept_id
    WHERE u.status = '0'
</select>
// LambdaJoinQueryBuilder 写法
QueryBuilder.lambdaJoin("u", SysUser.class)
    .leftJoin(SysDept.class, "d", SysDept::getDeptId, SysUser::getDeptId)
    .selectAs("u", SysUser::getUserId, UserDeptVo::getUserId)
    .selectAs("u", SysUser::getUserName, UserDeptVo::getUserName)
    .selectAs("d", SysDept::getDeptName, UserDeptVo::getDeptName)
    .eq("u", SysUser::getStatus, "0")
    .list(UserDeptVo.class);

// 优势：无 XML、编译期检查字段名、JOIN 条件 Lambda 化
结论：链式 API 不是语法糖，而是将"运行时字符串错误"转变为"编译期类型错误"。当 SysUser::getUserName 被重命名时，IDE 会自动报错；而 XML 中的 user_name 只能在运行时才发现问题。

六、知识点检查

Q1: LambdaJoinQueryBuilder 为什么所有条件方法都要求传入表别名？

点击查看答案

B。联表场景中，多张表常有同名字段（如 create_time、status），MPJ 需要表别名来确定条件作用于哪张表。LambdaQueryBuilder 的单表场景不需要此参数，因为它可以通过实体 Class 自动推导表名。

Q2: LambdaQueryCondition 接口中 Children 泛型参数的作用是什么？

点击查看答案

B。这是 Java Fluent API 设计中的 Self-Type 惯用法。如果接口直接返回 Children 类型，实现类 LambdaQueryBuilder 指定 Children 为自身，链式调用时返回值类型就是 LambdaQueryBuilder 而非接口类型，后续可以继续调用实现类特有的方法。

Q3: 以下关于 build() 方法的描述，哪个是正确的？

点击查看答案

B。LambdaQueryBuilder.build() 返回的是 LambdaQueryWrapper<T>，调用方需要传给 Mapper 方法如 mapper.selectList(builder.build()) 来执行。LambdaJoinQueryBuilder.build() 返回的是 MPJLambdaWrapper<T>，但它也提供了 list()、page()、count() 等直接执行的方法。
