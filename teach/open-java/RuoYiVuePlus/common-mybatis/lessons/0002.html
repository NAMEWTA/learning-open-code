第 2 课 · BaseEntity 与通用 Mapper 支柱一

本节目标：学完你能说出 BaseEntity 的 5 个字段及其填充时机，画出 InjectionMetaObjectHandler 的 insertFill/updateFill 流程图，理解 BaseMapperPlus 的双泛型设计原理，并用 mapper.lambda() 链式写出一个带条件的查询。

一、BaseEntity：所有实体的共同祖先

在 RuoYi-Vue-Plus 中，所有业务实体（SysUser、SysDept、SysRole 等）都直接或间接继承 BaseEntity。它定义了 5 个通用审计字段：

字段类型填充时机注解含义
createDept Long 仅插入时 @TableField(fill = FieldFill.INSERT) 创建部门 ID
createBy Long 仅插入时 @TableField(fill = FieldFill.INSERT) 创建者用户 ID
createTime LocalDateTime 仅插入时 @TableField(fill = FieldFill.INSERT) 创建时间
updateBy Long 插入 + 更新时 @TableField(fill = FieldFill.INSERT_UPDATE) 最后更新者用户 ID
updateTime LocalDateTime 插入 + 更新时 @TableField(fill = FieldFill.INSERT_UPDATE) 最后更新时间

源码如下：

@Data
public class BaseEntity implements Serializable {
    @Serial
    private static final long serialVersionUID = 1L;

    /** 创建部门 */
    @TableField(fill = FieldFill.INSERT)
    private Long createDept;

    /** 创建者 */
    @TableField(fill = FieldFill.INSERT)
    private Long createBy;

    /** 创建时间 */
    @TableField(fill = FieldFill.INSERT)
    private LocalDateTime createTime;

    /** 更新者 */
    @TableField(fill = FieldFill.INSERT_UPDATE)
    private Long updateBy;

    /** 更新时间 */
    @TableField(fill = FieldFill.INSERT_UPDATE)
    private LocalDateTime updateTime;
}
关键区分：FieldFill.INSERT 表示仅在插入时自动填充，更新时不动你。比如 createTime 在插入后永不再变。FieldFill.INSERT_UPDATE 表示插入和更新时都会自动填充，比如 updateTime 每次操作都刷新为当前时间。

二、InjectionMetaObjectHandler：自动填充的幕后英雄

@TableField(fill = ...) 只是标记——真正执行填充的是 InjectionMetaObjectHandler。它实现 MyBatis-Plus 的 MetaObjectHandler 接口，重写两个方法：

2.1 insertFill 流程

┌─ MyBatis-Plus 执行 insert 前 │ ├─ 1. metaObject.getOriginalObject() instanceof BaseEntity ? │ │ │ ├─ 是 → 强转为 baseEntity │ │ ├─ 2. 取当前时间 current │ │ ├─ 3. baseEntity.setCreateTime(current) │ │ ├─ 4. baseEntity.setUpdateTime(current) ← 插入时更新时间=创建时间 │ │ │ │ │ ├─ 5. baseEntity.getCreateBy() == null ? │ │ │ ├─ 是 → 尝试 LoginHelper.getLoginUser() │ │ │ │ ├─ 有用户 → setCreateBy(userId), setUpdateBy(userId) │ │ │ │ │ setCreateDept(deptId) │ │ │ │ └─ 无用户 → 全部设为 -1（代表系统操作/无人） │ │ │ └─ 否 → 跳过（保留已有值） │ │ │ │ │ └─ 完成 │ │ │ └─ 否 → strictInsertFill(metaObject, "createTime", ...) │ strictInsertFill(metaObject, "updateTime", ...) │ └─ 异常 → throw new ServiceException("自动注入异常 => ...")
设计要点：为什么用 -1L 作为默认用户 ID？因为定时任务、系统初始化等场景没有登录用户，用 null 会产生 NPE，用 -1 可以明确区分「系统操作」和「真实用户操作」。同时在数据库中有意义的默认值，不会被误认为真实用户 ID。

2.2 updateFill 流程

┌─ MyBatis-Plus 执行 update 前 │ ├─ 1. metaObject.getOriginalObject() instanceof BaseEntity ? │ │ │ ├─ 是 → 强转为 baseEntity │ │ ├─ 2. baseEntity.setUpdateTime(LocalDateTime.now()) ← 总是刷新 │ │ ├─ 3. LoginHelper.getLoginUser() │ │ │ ├─ 有用户 → setUpdateBy(userId) │ │ │ └─ 无用户 → setUpdateBy(-1L) │ │ └─ 完成 │ │ │ └─ 否 → strictUpdateFill(metaObject, "updateTime", ...) │ └─ 异常 → throw new ServiceException("自动注入异常 => ...")
insertFill vs updateFill 的关键区别：insertFill 填充 5 个字段（createDept、createBy、createTime、updateBy、updateTime），仅当 createBy 为空时才获取登录用户；updateFill 只填充 2 个字段（updateBy、updateTime），总是获取最新用户，因为每次更新都应记录最后的操作人。

2.3 注册到 MyBatis-Plus

在 MybatisPlusConfig.java 中注册：

@Bean
public MetaObjectHandler metaObjectHandler() {
    return new InjectionMetaObjectHandler();
}

只要 Spring 容器中存在一个 MetaObjectHandler Bean，MyBatis-Plus 就会在每次 insert/update 时自动调用它。

三、BaseMapperPlus<T, V>：双泛型的魔力

3.1 为什么需要第二个泛型

MyBatis-Plus 原生的 BaseMapper<T> 只有一个泛型（表实体）。这意味着每次查询完 Entity 后，你需要在 Service 层手动写：

// 传统写法：每次都要手动转换
SysUser user = userMapper.selectById(1L);
SysUserVo vo = MapstructUtils.convert(user, SysUserVo.class);  // 重复代码

BaseMapperPlus<T, V> 引入第二个泛型 V（VO 类型），让 Mapper 自己知道「查完实体后该转成什么 VO」：

// BaseMapperPlus 写法：一次调用搞定
public interface SysUserMapper extends BaseMapperPlus<SysUser, SysUserVo> {
}

// 使用时
SysUserVo vo = userMapper.selectVoById(1L);  // 自动 Entity → VO 转换

3.2 泛型类型缓存机制

为了让 Mapper 知道自己绑定的 T 和 V 具体是什么类，BaseMapperPlus 使用了 ClassValue 做缓存：

ClassValue<Class<?>[]> TYPE_ARGUMENT_CACHE = new ClassValue<>() {
    @Override
    protected Class<?>[] computeValue(Class<?> type) {
        return GenericTypeUtils.resolveTypeArguments(type, BaseMapperPlus.class);
    }
};

default Class<V> currentVoClass() {
    return (Class<V>) currentMapperTypes()[1];  // 第二个泛型 V
}

default Class<T> currentModelClass() {
    return (Class<T>) currentMapperTypes()[0];  // 第一个泛型 T
}

ClassValue 是 JDK 7 引入的 Class-keyed 缓存，每个 Mapper 子类第一次调用 currentVoClass() 时解析泛型，之后永久缓存。

3.3 Entity → VO 自动转换的核心方法

这些方法都遵循同一模式：查 Entity → 判空 → MapstructUtils.convert 转 VO：

方法输入输出转换点
selectVoById(id) 主键 ID 单个 V selectById → MapstructUtils.convert
selectVoByIds(idList) ID 集合 List<V> selectByIds → MapstructUtils.convert
selectVoByMap(map) 条件 Map List<V> selectByMap → MapstructUtils.convert
selectVoOne(wrapper) QueryWrapper 单个 V selectOne → MapstructUtils.convert
selectVoList(wrapper) QueryWrapper List<V> selectList → MapstructUtils.convert
selectVoPage(page, wrapper) 分页 + 条件 分页 V selectList(page, ...) → MapstructUtils.convert 后设 records

3.4 批量操作方法

BaseMapperPlus 还封装了 MyBatis-Plus 的 Db 工具类，提供便捷的批处理：

// 批量插入
default boolean insertBatch(Collection<T> entityList) {
    return Db.saveBatch(entityList);
}

// 批量按 ID 更新
default boolean updateBatchById(Collection<T> entityList) {
    return Db.updateBatchById(entityList);
}

// 批量插入或更新
default boolean insertOrUpdateBatch(Collection<T> entityList) {
    return Db.saveOrUpdateBatch(entityList);
}

// 支持指定批次大小
default boolean insertBatch(Collection<T> entityList, int batchSize) {
    return Db.saveBatch(entityList, batchSize);
}
注意：Db.saveBatch 底层是循环单条 insert（非真正的批量 SQL）。若需高性能批量 insert，应使用 MyBatis-Plus 的 InsertBatchSomeColumn SQL 注入器或原生 JDBC batch。

四、LambdaCrudChainWrapper：Mapper 上的链式 CRUD

4.1 创建方式

通过 mapper.lambda() 获取：

// BaseMapperPlus 中的工厂方法
default LambdaCrudChainWrapper<T, V> lambda() {
    return new LambdaCrudChainWrapper<>(this);
}

LambdaCrudChainWrapper 同时实现了 Query、Update、LambdaQueryCondition 三个接口，这意味着同一个链式对象既能查询也能更新。

4.2 查询终端方法

方法返回值说明
list()List<T>查询实体列表
list(page)List<T>分页查询实体列表
voList()List<V>查询 VO 列表（自动转换）
one()T查询单个实体
one(throwEx)T查询单个实体，可控制多条时是否抛异常
oneOpt()Optional<T>查询单个实体 Optional
voOne()V查询单个 VO
voOne(throwEx)V查询单个 VO，可控制多条时是否抛异常
count()Long统计数量
exists()boolean判断是否存在
page(page)分页 T分页查询实体
voPage(page)分页 V分页查询 VO
objs()List<Object>查询单列

4.3 更新终端方法

4.4 常用条件方法（继承自 AbstractLambdaWrapper）

所有标准 MyBatis-Plus 条件方法都可用于链式调用：eq、ne、gt、ge、lt、le、like、likeLeft、likeRight、between、in、notIn、isNull、isNotNull、orderByAsc、orderByDesc、groupBy、having 等。

另外 LambdaCrudChainWrapper 还增加了：

方法说明
setIfPresent(column, value)value 不为 null 时才设置更新字段
setIfText(column, value)value 字符串不为空时才设置

findInSetIfPresent(value, column)value 不为 null 时 FIND_IN_SET
setIncrBy(column, val)字段自增
setDecrBy(column, val)字段自减

五、实战代码示例

示例 1：定义一个继承 BaseEntity 的业务实体

@Data
@EqualsAndHashCode(callSuper = true)
@TableName("sys_user")
public class SysUser extends BaseEntity {
    private Long userId;
    private String userName;
    private String nickName;
    private String email;
    private String phoneNumber;
    private String sex;
    private String status;
    // 无需再定义 createBy/createTime/updateBy/updateTime/createDept
    // 全部继承自 BaseEntity
}

示例 2：定义带 VO 的双泛型 Mapper

public interface SysUserMapper extends BaseMapperPlus<SysUser, SysUserVo> {
    // 继承了所有 BaseMapperPlus 的默认方法
    // 可以继续添加自定义 XML 映射方法
    @Select("SELECT * FROM sys_user WHERE status = #{status}")
    List<SysUser> selectByStatus(String status);
}

示例 3：使用链式查询

// 查询状态为正常、用户名包含"admin"的用户 VO 列表
List<SysUserVo> voList = userMapper.lambda()
    .eq(SysUser::getStatus, "0")
    .like(SysUser::getUserName, "admin")
    .orderByDesc(SysUser::getCreateTime)
    .voList();

// 查询单个用户（多条时报错）
SysUserVo vo = userMapper.lambda()
    .eq(SysUser::getUserId, 1L)
    .voOne();  // 默认 throwEx=true

// 条件更新
boolean updated = userMapper.lambda()
    .eq(SysUser::getUserId, 1L)
    .set(SysUser::getStatus, "1")
    .update();

// setIfPresent：仅当值不为 null 时才设置
boolean partial = userMapper.lambda()
    .eq(SysUser::getUserId, 1L)
    .setIfPresent(SysUser::getEmail, email)       // email 可能为 null
    .setIfPresent(SysUser::getPhoneNumber, phone)  // phone 可能为 null
    .update();

示例 4：批量操作

// 批量插入用户列表
List<SysUser> users = ...;  // 从 Excel 导入的用户列表
boolean success = userMapper.insertBatch(users);

// 指定批次大小，每 500 条执行一次
userMapper.insertBatch(users, 500);

// 批量更新
userMapper.updateBatchById(users);

// 批量插入或更新（有 ID 则更新，无 ID 则插入）
userMapper.insertOrUpdateBatch(users);

示例 5：使用 selectVoById 自动转换

// 根据 ID 查询，自动转为 VO
SysUserVo vo = userMapper.selectVoById(1L);

// 根据 ID 集合批量查询
List<SysUserVo> voList = userMapper.selectVoByIds(Arrays.asList(1L, 2L, 3L));

// 指定目标 VO 类型（不同于 Mapper 声明的默认 V）
UserDetailVo detail = userMapper.selectVoById(1L, UserDetailVo.class);

六、测验

Q1：BaseEntity 中标记为 FieldFill.INSERT_UPDATE 的字段有哪些？

A. createDept 和 createBy B. createTime 和 createBy C. updateBy 和 updateTime D. createTime 和 updateTime
查看答案

答案：C。updateBy 和 updateTime 标记为 FieldFill.INSERT_UPDATE，意味着插入和更新时都会自动填充。createDept、createBy、createTime 标记为 FieldFill.INSERT，仅在插入时填充。

Q2：InjectionMetaObjectHandler.insertFill 中，当获取不到登录用户时，createBy 被设置为什么值？

A. null B. 0L C. -1L D. 1L
查看答案

答案：C。源码中定义 DEFAULT_USER_ID = -1L，代表无用户/系统操作。好处是：有意义的默认值，可区分「系统操作」和「真实用户」，不会和真实用户 ID 冲突。

Q3：BaseMapperPlus 中的 TYPE_ARGUMENT_CACHE 是什么类型？它的作用是什么？

A. HashMap — 缓存 SQL 执行结果 B. ClassValue — 缓存泛型类型解析结果，避免重复反射 C. ConcurrentHashMap — 缓存 VO 转换结果 D. ThreadLocal — 缓存当前线程的 Mapper 实例
查看答案

答案：B。ClassValue<Class<?>[]> 是 JDK 7 的 Class-keyed 缓存，每个 Mapper 子类第一次调用 currentVoClass() 时通过 GenericTypeUtils.resolveTypeArguments 解析泛型，结果永久缓存。比 ConcurrentHashMap<Class<?>, ...> 更高效（不需要显式 key）。

Q4：以下哪种写法可以安全地执行「只更新用户邮箱，如果邮箱不为 null」的操作？

A. mapper.updateById(user) B. mapper.lambda().eq(SysUser::getUserId, 1L).set(SysUser::getEmail, email).update() C. mapper.lambda().eq(SysUser::getUserId, 1L).setIfPresent(SysUser::getEmail, email).update() D. mapper.lambda().set(SysUser::getEmail, email).update()
查看答案

答案：C。setIfPresent 会在 value 为 null 时跳过该字段的设置，避免把数据库中的邮箱误更新为 null。B 会把邮箱更新为 null（如果 email 是 null），A 则可能把整个实体的其他字段也覆盖。
