03 - 查询构建器与多数据源

🎯 本节目标：掌握 QueryBuilder 的两个入口（lambda/lambdaJoin）及其能力差异，理解 PageQuery 的分页参数解析规则，能解释 DataBaseHelper 如何实现多数据库方言兼容，并能讲清 SqlLogInterceptor 的参数还原机制。

1. 全景鸟瞰

如果说数据权限是模块的"安全层"，查询构建器就是模块的"开发体验层"——它让 Java 代码写 SQL 查询像搭积木一样自然。多数据源和 SQL 日志则是"运维支撑层"——让系统能同时连接多个数据库，并完整记录每条 SQL。

2. 核心设计

2.1 两个构建器入口

QueryBuilder 提供两个静态入口，分别创建不同的查询构建器：

入口返回类型底层实现适用场景
QueryBuilder.lambda(SysUser.class)LambdaQueryBuilderAggregateLambdaQueryWrapper单表条件查询、聚合统计
QueryBuilder.lambdaJoin(SysUser.class)LambdaJoinQueryBuilderMPJLambdaWrapper多表 JOIN 查询、别名映射

2.2 LambdaQueryBuilder：单表查询瑞士军刀

包装 AggregateLambdaQueryWrapper<T>，提供完整的方法链：

标准条件：eq、ne、gt、ge、lt、le、like、between、in、notIn、orderBy、groupBy、having

聚合函数：selectSum、selectMax、selectMin、selectAvg、selectCount、selectSub

子查询支持：eqSub、inSub、existsSub

智能判空：eqIfPresent（值非空才加条件）、likeIfText（文本非空才加 LIKE）

数据库方言：findInSet → 自动翻译为 MySQL find_in_set / Oracle instr / PostgreSQL strpos / SQL Server charindex

// 典型用法
List<SysUserVo> users = QueryBuilder.lambda(SysUser.class)
    .eq(SysUser::getStatus, "0")
    .likeIfText(SysUser::getUserName, keyword)
    .between(SysUser::getCreateTime, startTime, endTime)
    .orderByDesc(SysUser::getCreateTime)
    .voList(SysUserVo.class);

2.3 LambdaJoinQueryBuilder：多表 JOIN 查询

基于 MyBatis-Plus-Join（MPJ），在 LambdaQueryBuilder 基础上增加联表能力：

  • leftJoin(UserDept.class, on -> on.eq(UserDept::getUserId, SysUser::getUserId))
  • selectAs(Dept::getDeptName, SysUserVo::getDeptName) —— 字段映射到别名对象
  • 条件方法需显式传表别名：eq("u", SysUser::getUserId, value)
注意：联表查询时，条件方法需要带表别名（如 "u"），因为多个表可能有同名字段。这是 MPJ 的设计要求，不是本模块的额外约束。

2.4 PageQuery：前端分页参数的标准化

PageQuery.build() 将前端传入的 pageNum、pageSize、orderByColumn、isAsc 转换为 MyBatis-Plus 的 Page<T> 对象：

  • 驼峰转下划线：前端传 createTime，自动转为 create_time
  • 多字段排序：orderByColumn="createTime,userName" + isAsc="asc,desc" 各自指定升降序
  • 兼容命名：支持 ascending/descending 前端常用命名
  • 安全校验：校验排序参数合法性，防止 SQL 注入
  • 默认值：pageSize = Integer.MAX_VALUE（不传分页参数时查全部）

2.5 DataBaseHelper：多数据源与 SQL 方言

两个核心能力：

1. 数据库类型识别：

getDataBaseType()
  → DynamicRoutingDataSource.determineDataSource() // 获取当前数据源
  → DatabaseMetaData.getDatabaseProductName()       // 识别产品名
  → 结果缓存到 ConcurrentHashMap                     // 避免重复查询

2. FIND_IN_SET 方言适配：

数据库SQL 函数
MySQLfind_in_set(var1, var2)
Oracleinstr(var2, var1)
PostgreSQLstrpos(var2, var1)
SQL Servercharindex(var1, var2)

2.6 SqlLogInterceptor：完整 SQL 日志

实现 MyBatis 原生 Interceptor，拦截 StatementHandler.query/update/batch：

  1. 拦截 SQL 执行 → 获取 BoundSql
  2. 解析 ? 占位符 → 根据 ParameterMapping 还原实际参数值
  3. 处理特殊类型：Date → 格式化、LocalDateTime → 格式化、Enum → name()
  4. 拼出可执行完整 SQL → 输出日志（含耗时、Mapper ID、错误信息）
  5. 超长 SQL 按 8000 字符分片输出
  6. 控制台模式用 ReentrantLock 防多线程穿插
为什么用 ReentrantLock？控制台输出不是线程安全的——如果两个线程同时输出 SQL，日志会穿插在一起难以阅读。加锁后保证每次只有一个线程输出完整 SQL，但会轻微影响并发性能。日志框架（如 Logback）模式不需要加锁，因为日志框架本身是线程安全的。

3. 实战练习

🧠 提取练习（点击展开）

Q1: 什么时候用 LambdaQueryBuilder，什么时候用 LambdaJoinQueryBuilder？

参考: 单表查询用 LambdaQueryBuilder——它更轻量，不需要引入 MPJ 的联表开销。需要 JOIN 其他表时用 LambdaJoinQueryBuilder。一个判断标准：如果 SELECT 和 WHERE 中引用的字段全部来自一个表，就用 lambda()；如果涉及多个表的字段，就用 lambdaJoin()。

Q2: DataBaseHelper.getDataBaseType() 为什么要缓存结果？

参考: 因为 DatabaseMetaData.getDatabaseProductName() 需要查询数据库元数据，涉及网络 IO。如果每次 findInSet 调用都查一次，会严重影响性能。缓存到 ConcurrentHashMap 后只有首次调用触发查询，后续 O(1) 命中。同时缓存是数据源级别的，切换数据源时会自动获取新的类型。

Q3: SqlLogInterceptor 在处理参数还原时，Date 类型为什么要特殊处理？

参考: MyBatis 的 ParameterMapping 只知道参数类型和 JDBC Type，不知道具体的格式化方式。Date 类型如果直接用 toString() 会依赖默认时区，与数据库实际存储可能不一致。显式格式化为 yyyy-MM-dd HH:mm:ss 保证日志中的 SQL 可以直接复制到数据库客户端执行，结果一致。

4. 调用链路

Controller: pageQuery.build()
  → pageNum=1, pageSize=10, orderByColumn="createTime", isAsc="desc"
  → 驼峰转下划线: createTime → create_time
  → 创建 Page<SysUser>(1, 10, "create_time", false)

Service: QueryBuilder.lambda(SysUser.class)
  → .eq(SysUser::getStatus, "0")
  → .likeIfText(SysUser::getUserName, "admin")
  → .orderByDesc(SysUser::getCreateTime)
  → .voPage(page, SysUserVo.class)

内部执行:
  → AggregateLambdaQueryWrapper 构建 WHERE status = '0' AND user_name LIKE '%admin%'
  → MyBatis-Plus Page 插件自动追加 LIMIT 10 OFFSET 0
  → SqlLogInterceptor 拦截输出完整 SQL + 耗时
  → 查询结果自动 MapstructUtils.convert → SysUserVo
💡 设计洞察：查询构建器的设计体现了「渐进增强」原则。LambdaQueryBuilder 满足 80% 的单表查询需求，LambdaJoinQueryBuilder 只在需要 JOIN 时才引入。两者共享相同的方法命名约定（eq/ne/like/between...），开发者从单表切换到联表时无需学习新 API。PageQuery 的 build() 方法则解决了前后端排序参数的标准转换问题——前端可以用驼峰命名，后端自动转下划线，消除了最常见的「排序不生效」bug 来源。
