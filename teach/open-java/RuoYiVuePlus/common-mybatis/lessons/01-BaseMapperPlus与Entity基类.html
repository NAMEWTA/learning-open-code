01 - BaseMapperPlus 与 Entity 基类

🎯 本节目标：理解 BaseMapperPlus 的双泛型设计与 VO 自动转换机制，掌握 BaseEntity 的 5 个审计字段如何通过 InjectionMetaObjectHandler 实现自动填充，并能说出 MybatisPlusConfig 注册的 8 个核心 Bean。

1. 全景鸟瞰

ruoyi-common-mybatis 是全项目最核心的持久层模块（34 个 Java 文件），将 MyBatis-Plus 从"能用"拔高到"企业级生产就绪"。本课聚焦模块的两大基石：

  • BaseMapperPlus<T, V>：双泛型 Mapper，提供 Entity 到 VO 的零模板转换
  • BaseEntity：5 个审计字段 + 自动填充 = 无需手动 set 创建人/时间

2. 核心设计

2.1 BaseMapperPlus：为什么需要第二个泛型？

MyBatis-Plus 原生的 BaseMapper<T> 只有一个泛型（Entity 类型）。如果查询需要返回 VO，你需要手动写 BeanUtil.copyProperties 或 MapStruct 转换。BaseMapperPlus 通过 第二个泛型 <V> 解决了这个问题：

// MyBatis-Plus 原生写法（需要手动转换）
SysUser user = userMapper.selectById(1L);
SysUserVo vo = MapstructUtils.convert(user, SysUserVo.class);

// BaseMapperPlus 写法（一条语句搞定）
SysUserVo vo = userMapper.selectVoById(1L);

底层原理：selectVoById 内部先调用 selectById 获取 Entity，然后用 MapstructUtils.convert(entity, voClass) 自动转换。泛型 V 的 Class 对象通过反射从继承声明中提取。

除了 VO 转换，BaseMapperPlus 还提供了 selectVoList、selectVoPage、insertBatch、updateBatchById 等便捷方法。入口文件：core/mapper/BaseMapperPlus.java。

2.2 BaseEntity：审计字段的自动填充

所有业务实体的基类，定义了 5 个公共字段：

字段填充策略含义
createDeptFieldFill.INSERT创建部门
createByFieldFill.INSERT创建者 ID
createTimeFieldFill.INSERT创建时间
updateByFieldFill.INSERT_UPDATE更新者 ID
updateTimeFieldFill.INSERT_UPDATE更新时间

填充链路：

业务代码 save(entity)
  → MyBatis-Plus 检测 @TableField(fill = ...)
    → MetaObjectHandler.insertFill()
      → InjectionMetaObjectHandler.insertFill()
        → LoginHelper.getLoginUser() 获取当前用户
        → entity.setCreateBy(userId) / entity.setCreateTime(new Date())

获取不到用户时填默认值 -1——代表无用户/系统自动操作，避免 NPE。

2.3 MybatisPlusConfig：模块的装配中枢

配置类 config/MybatisPlusConfig.java 注册了 8 个核心 Bean，构成了模块的完整能力：

Bean作用
MybatisPlusInterceptor内含数据权限 + 分页 + 乐观锁三个内部拦截器
DataPermissionPointcutAdvisor数据权限切面，桥接注解与拦截器
SqlLogInterceptorSQL 日志（条件装配，通过配置开关）
MetaObjectHandler自动填充处理器（InjectionMetaObjectHandler）
IdentifierGenerator雪花 ID 生成器（用网卡 MAC 防集群重复）
MybatisExceptionHandler全局异常兜底（主键冲突 409、MyBatis 系统异常 500）
PostInitTableInfoHandler逻辑删除开关（通过配置全局关闭逻辑删除）
@MapperScan扫描 Mapper 包（从配置 mybatis-plus.mapperPackage 读取）
关键点：雪花 ID 的 workerId 来自 NetUtil.getLocalhost()（网卡 MAC 地址），保证集群各节点的雪花 ID 不冲突。这是生产环境防止主键碰撞的重要措施。

3. 实战练习

🧠 提取练习（点击展开）

Q1: 为什么 BaseMapperPlus 需要两个泛型参数，而不是只用一个？

参考: 第一个泛型 T 是 Entity（数据库表映射），第二个 V 是 VO（返回给前端的对象）。双泛型使得 selectVoById 等方法在编译期就知道目标 VO 类型，避免运行时 Class 转换错误，同时省去每个业务 Mapper 中手写 convert 的重复代码。

Q2: InjectionMetaObjectHandler.updateFill() 为什么只填 updateBy 和 updateTime，不填 createBy 和 createTime？

参考: 因为 createBy/createTime 的填充策略是 FieldFill.INSERT，仅在插入时触发。updateFill 对应 FieldFill.UPDATE 和 FieldFill.INSERT_UPDATE（后者在插入时已由 insertFill 处理），只需更新 updateBy 和 updateTime 两个字段。

Q3: MybatisExceptionHandler 单独处理 DuplicateKeyException 的设计意义是什么？

参考: 主键冲突是业务层面可预期的错误（如重复提交），不应暴 500。返回 409（Conflict）HTTP 状态码 + 中文提示，让前端可以对用户展示友好提示，而不是"系统错误"这种无意义的信息。

4. 调用链路

一次 insert 操作的完整路径：

Controller: service.save(entity)
  → ServiceImpl: baseMapper.insert(entity)
    → MybatisPlusInterceptor 链:
      ① 数据权限拦截器（INSERT 不触发，跳过）
      ② 分页拦截器（INSERT 不触发，跳过）
      ③ 乐观锁拦截器（INSERT 不触发，跳过）
    → MetaObjectHandler.insertFill(entity)
      → entity.createTime = now()
      → entity.createBy = LoginHelper.getLoginUser()
    → IdentifierGenerator.nextId(entity)
      → 雪花 ID = 时间戳 + 机器 ID + 序列号
    → SQL: INSERT INTO ... (id, create_time, create_by, ...) VALUES (?, ?, ?, ...)
    → SqlLogInterceptor 输出完整 SQL 到日志
💡 设计洞察：BaseEntity 的设计思想来自 JPA 的 @MappedSuperclass，但实现上用了 MyBatis-Plus 的 MetaObjectHandler。这种「借思想、用实现」的做法避免了对 JPA 的依赖，同时保留了审计字段自动填充的便利。5 个字段看似简单，但配合 INSERT/INSERT_UPDATE 两种填充策略和 LoginHelper 的动态获取，形成了一套完整的「谁在什么时候创建/修改了这条数据」的追踪体系。
