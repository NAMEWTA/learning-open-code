0008 — 实战整合：场景剖析与扩展指南 ruoyi-common-mybatis

学习目标：将前 7 节课的知识应用于实际业务场景 —— 数据权限调试、自定义扩展、多数据源实战、常见故障排查以及最佳实践总结。

场景1：数据权限三种角色的完整链路追踪

以 "超级管理员看全部，部门经理看本部门及下级，普通员工只看自己" 为场景，追踪三种角色类型的数据权限 SQL 生成过程。

前置条件

enums/DataScopeType.java

角色dataScope 值DataScopeTypeSpEL 模板
超级管理员"1"ALL空字符串（不追加条件）
部门经理"4"DEPT_AND_CHILD#{#deptName} IN ( #{@sdss.getDeptAndChild( #user.deptId )} )
普通员工"5"SELF#{#userName} = #{#user.userId}

完整执行链路

DataPermissionPointcutAdvisor → DataPermissionAdvice → PlusDataPermissionInterceptor → PlusDataPermissionHandler

【阶段1：AOP拦截 — DataPermissionAdvice】 DataPermissionPointcut 匹配到 @DataPermission 注解的方法 → DataPermissionAdvice.invoke() 被触发 → 获取 @DataPermission 注解（方法级优先，否则类级） → DataPermissionHelper.setPermission(dataPermission) 存入 ThreadLocal → 执行 invocation.proceed() → finally: DataPermissionHelper.removePermission() 清理 【阶段2：MyBatis 拦截 — PlusDataPermissionInterceptor】 Executor 执行 SQL 前触发 beforeQuery() → InterceptorIgnoreHelper.willIgnoreDataPermission() 检查是否被忽略 → dataPermissionHandler.invalid() 检查 ThreadLocal 中有无 @DataPermission → parserSingle(sql, msId) 解析并改写 SQL 【阶段3：SQL 改写 — PlusDataPermissionHandler.getSqlSegment()】 a. currentUser() 获取当前登录用户 b. LoginHelper.isSuperAdmin() → 超级管理员直接返回原 where（不追加权限条件） c. 非超级管理员则调用 buildDataFilter() 【阶段4：数据过滤 SQL 构建 — buildDataFilter()】 1. 确定连接词：SELECT 用 " OR "，UPDATE/DELETE 用 " AND " 2. 构建 SpEL 上下文（NullSafeStandardEvaluationContext，null 返回 "-1"） 3. currentAccess() 解析当前请求的 @SaCheckPermission/@SaCheckRole 4. scopeRoles() 筛选当前用户角色（优先用 dataScopeRoleMap 匹配） 5. 遍历角色，根据 dataScope 值匹配 DataScopeType： 超级管理员 (ALL, code="1"): → buildDataFilter 直接 return ""（不加任何条件） → 最终 SQL: SELECT * FROM sys_user WHERE ... (原条件不变) 部门经理 (DEPT_AND_CHILD, code="4"): → SpEL 解析 "#{#deptName} IN ( #{@sdss.getDeptAndChild( #user.deptId )} )" → deptName = dataColumn.value() 映射的字段名，如 "dept_id" → @sdss 是 Spring Bean，调用 getDeptAndChild(deptId) 返回部门ID列表 → 最终 SQL: SELECT * FROM sys_user WHERE ... AND (dept_id IN (101, 102, 103)) 普通员工 (SELF, code="5"): → SpEL 解析 "#{#userName} = #{#user.userId}" → userName = dataColumn.value() 映射的字段名，如 "user_id" → user.userId = 当前登录用户ID → 最终 SQL: SELECT * FROM sys_user WHERE ... AND (user_id = 1001) 6. 多角色时用 joinStr 连接各角色的条件
关键设计：LoginHelper.isSuperAdmin() 检查在 getSqlSegment() 中首先执行。超管直接返回原 where 条件，跳过了所有 SpEL 解析和 SQL 拼接。这避免了不必要的性能开销，也是最顶层的数据权限豁免机制。
SELECT vs UPDATE/DELETE 的连接词差异：DataScopeType 的 sqlTemplate 使用 OR 连接不同角色的数据范围（因为用户可能有多个角色，任一角色有权限即可），而 UPDATE/DELETE 使用 AND（所有角色的条件必须同时满足）。这个区别在 buildDataFilter() 中通过 isSelect 参数控制。

场景2：自定义数据权限扩展

假设业务需要一个新的数据范围类型 —— "仅查看本部门和指定关联部门"。

扩展步骤

步骤操作说明
1 在 DataScopeType 枚举中添加新值 
DEPT_AND_LINKED("7",
  " #{#deptName} IN ( #{@sdss.getDeptAndLinked( #user.deptId )} )",
  " 1 = 0 ")
code 为 "7"，不与现有值冲突。elseSql 为 "1 = 0" 表示当 SpEL 变量缺失时阻断查询。
2 在 ISysDataScopeService 接口中添加方法 
// 返回当前部门及关联部门ID集合
Set<Long> getDeptAndLinked(Long deptId);
3 在 SysDataScopeServiceImpl 中实现 查询 dept_id 关联表，返回部门ID集合。SpEL 表达式 #{@sdss.getDeptAndLinked(...)} 会通过 Spring 容器调用该 Bean 方法。
4 在角色管理中配置 dataScope=7 前端角色配置页面选择新的数据范围类型，数据库 sys_role 表 data_scope 字段设为 "7"。
5 在 Mapper 方法上使用 @DataPermission 注解 确保 DataColumn 的 key/value 映射与 SpEL 模板中的 #deptName 变量对应： 
@DataPermission({
    @DataColumn(key = "deptName", value = "dept_id")
})
扩展原理：buildDataFilter() 通过 DataScopeType.findCode(role.getDataScope()) 动态匹配枚举。新增枚举值后，框架自动识别 code="7"，取出对应的 sqlTemplate 交由 SpEL 解析器处理。无需修改框架代码，只需扩展枚举和对应的 Service 方法。
SpEL 上下文变量说明：
#deptName / #userName — 来自 @DataColumn 的 key 数组，值为对应的 value 数组（字段名）
#user — 当前登录用户对象（LoginUser），通过 LoginHelper.getLoginUser() 或上下文变量获取
#roleId — 当前角色的 roleId
@sdss — Spring Bean 引用（ISysDataScopeService），通过 BeanFactoryResolver 解析

场景3：添加新的数据源

数据源配置方式

ruoyi-admin/src/main/resources/application-dev.yml

dynamic-datasource 使用 spring.datasource.dynamic 前缀配置多数据源。在 YAML 中添加新数据源：

spring:
  datasource:
    dynamic:
      primary: master          # 默认数据源
      strict: true             # 严格模式：未匹配数据源时抛异常
      datasource:
        master:                # 主库
          driverClassName: com.mysql.cj.jdbc.Driver
          url: jdbc:mysql://localhost:3306/ry-vue?...
          username: root
          password: root
        oracle_db:             # 新增 Oracle 数据源
          driverClassName: oracle.jdbc.OracleDriver
          url: jdbc:oracle:thin:@//192.168.1.100:1521/ORCL
          username: ry_vue
          password: ry_vue
      hikari:                  # HikariCP 连接池全局配置
        maxPoolSize: 20
        minIdle: 10

使用数据源

在 Service 或 Mapper 方法上使用 @DS 注解切换数据源：

@DS("oracle_db")
public List<UserVO> selectOracleUserList() {
    return baseMapper.selectVoList(...);
}

DataBaseHelper 自动适配

首次访问 oracle_db 数据源时：

1. DS.determineDataSource() → 根据 @DS("oracle_db") 切换到 Oracle 数据源 2. getDataBaseType() → 获取 Oracle Connection 3. DatabaseMetaData.getDatabaseProductName() → "Oracle" 4. DataBaseType.find("Oracle") → DataBaseType.ORACLE 5. DB_TYPE_CACHE.put("oracle_db", ORACLE) 6. findInSet() 调用时 → getDataBaseType() 返回 ORACLE → 生成 instr() SQL

零代码改动：只要配置了数据源，DataBaseHelper 会自动检测数据库类型，findInSet() 等跨数据库方法会自动生成对应方言 SQL。

场景4：常见故障排查

故障1：数据权限不生效

可能原因排查方法解决方案
忘记添加 @DataPermission 注解 检查 Mapper 方法或类上是否有 @DataPermission 在需要数据权限的方法上添加注解和 @DataColumn 配置
当前用户是超级管理员 检查 LoginHelper.isSuperAdmin() 返回值 超管故意绕过数据权限，这是预期行为。如需测试，使用非超管账号
InterceptorIgnoreHelper 屏蔽了拦截器 检查是否调用了 InterceptorIgnoreHelper.handle() 检查代码中是否有 DataPermissionIgnoreContext.enable() 调用未配对 disable
角色 dataScope 值未配置或为 "1"(ALL) 检查数据库 sys_role 表 data_scope 字段 将角色的 dataScope 设为对应的数值（4/5 等）

故障2：SQL 日志显示了 ? 占位符但无参数值

可能原因排查方法解决方案
mybatis-plus.sql-log.enabled 未开启 检查 YAML 配置文件 添加配置：mybatis-plus.sql-log.enabled=true
看到的日志是 MyBatis 原生日志，不是 SqlLogInterceptor 输出 SqlLogInterceptor 输出包含 "Consume Time" 和 "Mapper ID" 确认 common-mybatis.yml 中 logImpl 为 NoLoggingImpl，避免日志重复

故障3：数据库唯一键冲突

现象原因排查方法
HTTP 409 "数据库中已存在该记录" DuplicateKeyException 被 MybatisExceptionHandler 捕获 查看日志中具体的请求地址和异常消息，确认是哪个唯一键冲突

故障4：SpEL 变量为 null 导致查询返回 "1 = 0" 无结果

可能原因解决方案
未调用 DataPermissionHelper.setVariable() 设置必要的上下文变量 在调用 Mapper 前确保 SpEL 所需的变量（如 user）已设置。默认由 currentUser() 自动设置
@DataColumn 的 key/value 与 sqlTemplate 中的变量名不匹配 检查 @DataColumn 的 key 名称是否与 DataScopeType 中 #deptName / #userName 一致

故障5："未找到数据源" 错误

现象原因解决方案
HTTP 500 "未找到数据源，请联系管理员确认" MyBatisSystemException 根因为 CannotFindDataSourceException 检查 YAML 中数据源配置是否存在、@DS 注解值是否与配置的 key 一致

场景5：最佳实践总结

三种查询构建器选择指南

场景推荐构建器入口方法
单表 CRUD（含聚合、子查询） LambdaQueryBuilder QueryBuilder.lambda(Entity.class)
多表 JOIN 查询 LambdaJoinQueryBuilder QueryBuilder.lambdaJoin(Entity.class)
Mapper 层链式 CRUD（直接执行） LambdaCrudChainWrapper baseMapper.lambda()

@DataPermission 放置位置决策

放置位置适用场景优先级
方法级别 同一 Mapper 中不同方法需要不同的数据权限配置 高于类级别
类级别 整个 Mapper 的所有方法使用相同的数据权限规则 方法无注解时生效

查找逻辑在 DataPermissionAdvice.getDataPermissionAnnotation() 中：先查方法注解，若为 null 则查类注解（包括 JDK 动态代理的接口注解）。

Mapper 代码组织建议

  • Mapper 接口继承 BaseMapperPlus<T, V>，享受泛型 VO 支持和链式 CRUD
  • 使用 LambdaQueryBuilder / LambdaJoinQueryBuilder 替代传统 XML 编写动态 SQL，类型安全且可读性强
  • 需要数据权限的方法必须标注 @DataPermission，不需要的不标注（默认无权限过滤）
  • 在异步任务、定时任务等非 HTTP 请求场景中注意 DataPermissionHelper 的上下文可能为空（无 HttpServletRequest）
  • 敏感操作（删除、导出）使用 @DataPermission 同时配合 @SaCheckPermission 做双重控制
本节核心要点回顾：
1. 数据权限链路：AOP 注解捕获 → MyBatis 拦截器 SQL 改写 → DataScopeType 枚举匹配 → SpEL 模板解析
2. 扩展 DataScopeType 只需修改枚举 + 添加 Service 方法，无需改动框架代码
3. 多数据源配置后 DataBaseHelper 自动检测类型，findInSet 自动适配方言
4. 常见故障：忘记 @DataPermission、超管豁免、InterceptorIgnoreHelper 屏蔽、变量 null 导致 1=0

自测题

Q1: 在数据权限链路中，超级管理员的数据过滤是在哪个方法中被跳过的？

A. DataPermissionAdvice.invoke() B. PlusDataPermissionInterceptor.beforeQuery() C. PlusDataPermissionHandler.getSqlSegment() D. buildDataFilter()
查看答案答案：C。 getSqlSegment() 方法首先调用 LoginHelper.isSuperAdmin()，为 true 时直接返回原 where 条件。

Q2: 当 DataScopeType 的 sqlTemplate 中引用的 SpEL 变量（如 #deptName）未在 @DataColumn 中声明时，会导致什么？

A. 抛出 ServiceException B. 数据权限静默失效 C. 生成 "1 = 0" 导致查询无结果 D. 整个请求返回 500 错误
查看答案答案：B。 当 sqlTemplate 中的变量 key 不在 @DataColumn 声明的 key 集合中时，isSuccess 保持 false，进入 elseSql 处理。但若 elseSql 也为空，则 conditions 为空，最终返回空字符串（无权限过滤）。对于 CUSTOM 类型的模板，可能走到 elseSql="1=0"。

Q3: 添加新数据源后，DataBaseHelper.findInSet() 是否需要修改代码才能适配新数据源？

A. 需要，必须在 switch 中添加新的 case B. 不需要，自动使用 default 分支（MySQL 语法） C. 不需要，动态数据源框架会自动转换 SQL 方言 D. 取决于新数据源的数据库类型，如果是已知类型（如 Oracle）则自动适配，全新数据库类型需要添加 case
查看答案答案：D。 DataBaseType 仅识别 4 种数据库（find 默认返回 MY_SQL），findInSet 的 switch 覆盖了这 4 种。若新数据源是已知的 4 种之一，自动适配；若是全新数据库类型，需在枚举中添加新值和对应的 switch case。

Q4: 某用户查询列表时总是返回空，但数据库确实有数据，最可能的原因是什么？

A. 数据库连接池耗尽 B. @DataPermission 配置了 DataScopeType 但 SpEL 变量缺失导致生成 "1 = 0" C. MybatisExceptionHandler 捕获了异常返回了空列表 D. SqlLogInterceptor 阻止了查询执行
查看答案答案：B。 当 SpEL 解析失败（变量为 null），NullSafeStandardEvaluationContext 返回默认值 "-1"，加上 elseSql="1=0"，最终 WHERE 条件为 "1=0"，查询无结果。
