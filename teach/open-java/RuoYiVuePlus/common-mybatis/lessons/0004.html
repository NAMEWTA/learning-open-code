0004 - 数据权限体系（下）：DataScopeType、上下文与多租户 核心机制

学习目标：掌握数据权限的后半部分——理解 6 种 DataScopeType 如何通过 SpEL 模板生成 SQL WHERE 条件，上下文变量如何跨层流转，以及端到端完整调用链。

一、DataScopeType 深度解析

org.dromara.common.mybatis.enums.DataScopeType

共 6 种数据范围类型，每种定义了 SpEL 模板和 fallback SQL：

代码名称codesqlTemplate（匹配到 key 时）elseSql（key 未匹配时）
1全部数据权限ALL ""（空字符串） ""
2自定义数据权限CUSTOM #{#deptName} IN ( #{@sdss.getRoleCustom(#roleId)} ) 1 = 0
3本部门数据权限DEPT #{#deptName} = #{#user.deptId} 1 = 0
4本部门及以下DEPT_AND_CHILD #{#deptName} IN ( #{@sdss.getDeptAndChild(#user.deptId)} ) 1 = 0
5仅本人数据SELF #{#userName} = #{#user.userId} 1 = 0
6本部门及以下（或本人）DEPT_AND_CHILD_OR_SELF #{#deptName} IN ( #{@sdss.getDeptAndChild(#user.deptId)} ) OR #{#userName} = #{#user.userId} 1 = 0
elseSql 的作用：当 @DataColumn 中找不到 DataScopeType 模板引用的 key（如 deptName）时，使用 elseSql 作为回退。1 = 0 意味着"该角色在此查询中不生效"，即没有匹配的数据列就返回空集。非 ALL 类型的 elseSql 均为 1 = 0，这是安全优先的设计——找不到匹配列时宁可返回空数据，也不泄漏数据。

1.1 SpEL 模板"双槽位"设计

每个 DataScopeType 模板中同时出现 #{#deptName} 和 #{#userName} 两种占位符。在运行时，handler 会遍历 @DataPermission 中所有的 @DataColumn，用实际列名替换对应的 key：

// 模板： #{#deptName} IN ( #{@sdss.getDeptAndChild(#user.deptId)} )
// 替换后：dept_id IN ( #{@sdss.getDeptAndChild(#user.deptId)} )
// 再执行 SpEL 解析 → dept_id IN (101, 102, 103)

如果一个 @DataColumn key 在模板中不存在（如模板是 #{#deptName} = ... 但只定义了 userName 列），则执行 elseSql。

二、SpEL 解析引擎在 Handler 中的应用

org.dromara.common.mybatis.handler.PlusDataPermissionHandler

2.1 核心解析组件

组件类作用
表达式解析器SpelExpressionParserSpring SpEL 标准解析器，解析 #{...} 模板
模板上下文TemplateParserContext定义前缀 #{ 和后缀 }，使解析器能识别模板边界
Bean 解析器BeanFactoryResolver(@sdss)@sdss 前缀映射到 Spring 容器中的 Bean（如 SysDataScopeServiceImpl），支持在 SQL 模板中调用服务方法

2.2 buildDataFilter() 执行流程

buildDataFilter(ExpressionWhere where, boolean isSelect) {
    1. 获取当前登录用户 LoginUser
    2. 如果是超级管理员 (LoginHelper.isSuperAdmin())
       → 返回空字符串，不附加任何权限条件

    3. 获取用户角色列表 → scopeRoles() 过滤角色
       ↓ (按 @SaCheckPermission / @SaCheckRole 约束)

    4. 遍历每个角色：
       a. 获取角色的 DataScopeType（如 DEPT_AND_CHILD）
       b. 获取对应的 sqlTemplate 字符串
       c. 遍历 @DataPermission 中的所有 @DataColumn
          - 若模板包含该 column.key → 替换为真实列名
          - 使用 SpelExpressionParser 解析模板中的 #{...}
          - NullSafeStandardEvaluationContext 安全替换变量
          - 收集解析后的 SQL 片段
          - 若模板不包含该 key → 使用 elseSql

    5. 将所有角色的 SQL 片段 OR 连接 → 最外层 AND 到原始 WHERE
}
NullSafeStandardEvaluationContext 的设计价值：在分页查询等场景中，@DataPermission 可能定义了多个 @DataColumn（如 deptName 和 userName），但某些角色（如 SELF 类型）只用到了 userName。当模板尝试访问 #deptName 且 handler 按 userName 列遍历时，该变量在上下文中为 null。普通 SpEL 会抛异常，而 NullSafe 返回 -1，优雅地跳过不匹配的列。

三、DataPermissionHelper 上下文桥梁

org.dromara.common.mybatis.helper.DataPermissionHelper

DataPermissionHelper 是 Advice（切面层）和 Handler（拦截器层）之间的桥梁，通过两种存储机制传递上下文：

存储机制存储内容写入方读取方
ThreadLocal<DataPermission> 当前方法的 @DataPermission 注解对象（含 DataColumn[] 和 joinStr） DataPermissionAdvice.invoke() PlusDataPermissionHandler.getSqlSegment()
SaStorage（Sa-Token 请求上下文） 方法参数 Map（parameter name → value），以及 DataPermissionAccess 对象 DataPermissionAdvice.invoke() PlusDataPermissionHandler SpEL 变量查找
为什么用 SaStorage 而非另一个 ThreadLocal：Sa-Token 的 SaStorage 天然与请求生命周期绑定，请求结束自动清理，无需手动管理。且 SpEL 解析时需要访问 #user、#roleId 等请求级变量，SaStorage 提供了统一的作用域管理。

3.1 变量传递链路

// Advice 端（切面层）
DataPermissionHelper.setPermission(dataPermission, args);
  ├─ THREAD_LOCAL.set(dataPermission);          // ThreadLocal 存注解
  └─ SaStorage: {
       "user"      → LoginUser 对象,           // 当前用户
       "roleId"    → 当前遍历的角色 ID,          // Handler 中动态设置
       "deptName"  → 当前 @DataColumn.key,      // Handler 中动态设置
       "userName"  → 当前 @DataColumn.key,      // Handler 中动态设置
       "params"    → Mapper 方法参数 Map         // 原始方法参数
     }

// Handler 端（拦截器层）
DataPermission dataPermission = DataPermissionHelper.getPermission();
// → 从 ThreadLocal 取出注解，获取 DataColumn[] 和 joinStr

四、DataPermissionAccess — 权限约束模型

org.dromara.common.mybatis.core.domain.DataPermissionAccess

4.1 结构定义

DataPermissionAccess 是一个 record 类型：

public record DataPermissionAccess(
    Set<String> perms,       // 权限标识集合，如 {"system:user:list", "system:user:query"}
    Set<String> roleKeys     // 角色标识集合，如 {"admin", "common"}
) {
    public boolean constrained() {
        return (perms != null && !perms.isEmpty())
            || (roleKeys != null && !roleKeys.isEmpty());
    }
}

4.2 resolveAccess() — 从 HandlerMethod 提取约束

PlusDataPermissionHandler.resolveAccess() 通过读取当前请求的 HandlerMethod：

  1. 查找方法或类上的 @SaCheckPermission 注解 → 提取 value() 作为 perms
  2. 查找方法或类上的 @SaCheckRole 注解 → 提取 value() 作为 roleKeys
  3. 封装为 DataPermissionAccess 对象，存入 SaStorage
为什么需要 resolveAccess：不是所有 Mapper 方法都需要数据权限。只有被 Controller 中 @SaCheckPermission 或 @SaCheckRole 保护的方法才需要。resolveAccess() 在请求入口捕获这些约束，scopeRoles() 在 Handler 中根据约束过滤角色，确保"不合法的角色不参与权限条件生成"。

4.3 scopeRoles() — 角色过滤

用户可能拥有多个角色（如 admin、common、dept_leader），但当前请求只需要特定角色对应的数据范围。scopeRoles() 的过滤逻辑：

  1. 如果 DataPermissionAccess 有 roleKeys → 只保留匹配的角色
  2. 如果 DataPermissionAccess 有 perms → 通过 dataScopeRoleMap（权限 → 角色 ID 映射）反向查找关联角色
  3. 如果两者都为空 → 保留所有角色（不受限场景）

五、端到端完整链路追踪

5.1 以 SysUserMapper.selectList() 为例

[1] Controller 层 SysUserController.list() @SaCheckPermission("system:user:list") ← 声明权限约束 ↓ resolveAccess() 提取 perms={"system:user:list"} → SaStorage [2] Mapper 调用 SysUserMapper.selectList(queryWrapper) ↓ @DataPermission({@DataColumn(key="deptName", value="dept_id")}) ↓ 接口方法上有注解 [3] AOP 切面拦截 DataPermissionPointcut.matches() → true DataPermissionAdvice.invoke(): ├─ DataPermissionHelper.setPermission(注解, args) → ThreadLocal ├─ invocation.proceed() │ ↓ │ [4] MyBatis 拦截器触发 │ PlusDataPermissionInterceptor.beforeQuery(): │ ├─ InterceptorIgnoreHelper → 未忽略 │ ├─ DataPermissionHandler.invalid() → 有效 │ └─ JSqlParser 解析 SQL → Select 对象 │ │ [5] Handler 生成权限条件 │ PlusDataPermissionHandler.getSqlSegment(): │ ├─ LoginHelper.isSuperAdmin() → false │ ├─ buildDataFilter(): │ │ ├─ 获取用户角色 → scopeRoles() │ │ │ ├─ DataPermissionAccess.perms={"system:user:list"} │ │ │ └─ dataScopeRoleMap → roleId=2 (common角色) │ │ │ │ │ ├─ 角色 common → DataScopeType=DEPT (code="3") │ │ │ 模板: #{#deptName} = #{#user.deptId} │ │ │ DataColumn[{key="deptName", value="dept_id"}] │ │ │ → 替换 key → "dept_id = #{#user.deptId}" │ │ │ → SpEL 解析 #{#user.deptId} → 103 │ │ │ → SQL 片段: "dept_id = 103" │ │ │ │ │ └─ 返回: (dept_id = 103) │ │ │ └─ 原始 WHERE + AND + (dept_id = 103) [6] 最终 SQL SELECT * FROM sys_user WHERE del_flag = '0' AND (dept_id = 103) [7] finally 清理 DataPermissionHelper.removePermission() → ThreadLocal.remove()

六、超级管理员绕过与多租户

6.1 超级管理员豁免

在 PlusDataPermissionHandler.getSqlSegment() 中，第一步就是检查：

if (LoginHelper.isSuperAdmin()) {
    return where.getSqlSegment();  // 原样返回，不注入任何权限条件
}

超级管理员（roleKey = "admin" 且分配了所有权限的角色）绕过所有数据权限过滤，可以看到全部数据。这是通过 Sa-Token 的 LoginHelper 静态方法判断。

6.2 多租户说明

数据权限（Data Permission）和多租户（Multi-Tenant）是两个独立但可协作的机制：

  • 数据权限：基于用户角色/部门的行级过滤，通过 SpEL 动态生成 WHERE 条件
  • 多租户：基于租户 ID 的行级隔离，通常通过 MyBatis Plus 的租户拦截器在每条 SQL 上自动追加 tenant_id = ?

两者在 MyBatis 拦截器链中按顺序执行，互不干扰。数据权限条件追回在租户条件之后，最终 SQL 同时包含两种约束。

小节测验

Q1: 当 DataScopeType 为 DEPT_AND_CHILD 时，若 @DataPermission 中没有匹配 deptName 这个 key 的 @DataColumn，最终 SQL 会注入什么条件？

查看答案

C。elseSql 为 1 = 0，安全优先——找不到匹配的数据列时返回空结果，防止数据泄漏。

Q2: NullSafeStandardEvaluationContext 在 SpEL 变量为 null 时返回什么值？

查看答案

C。返回 -1 使得 SQL 条件变为 dept_id = -1 或 dept_id IN (-1)，实际查询不到任何数据，安全且不中断流程。

Q3: scopeRoles() 根据什么信息来过滤用户角色？

查看答案

B。DataPermissionAccess 从 @SaCheckPermission 和 @SaCheckRole 注解提取权限和角色标识，scopeRoles() 据此过滤——只保留当前请求关联的角色，避免无关角色的数据范围污染 SQL。

核心要点回顾：
1. 6 种 DataScopeType 覆盖了从"全部数据"到"仅本人"的完整权限粒度
2. SpEL 模板通过 #{#deptName} 占位符 + @sdss Bean 引用实现动态 SQL 生成
3. ThreadLocal + SaStorage 双通道实现 Advice → Handler 的上下文传递
4. resolveAccess() 从 Controller 注解提取权限约束，scopeRoles() 按约束过滤角色
5. 超级管理员完全绕过数据权限；多租户与数据权限是独立的两层过滤
