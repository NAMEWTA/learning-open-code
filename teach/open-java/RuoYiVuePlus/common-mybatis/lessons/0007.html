0007 — 多数据源与 SQL 日志 ruoyi-common-mybatis

学习目标：掌握 ruoyi-common-mybatis 模块如何集成 baomidou dynamic-datasource 实现多数据源支持，以及如何通过 SqlLogInterceptor 提供生产级 SQL 日志输出。

1. Dynamic-Datasource 集成原理

RuoYi-Vue-Plus 使用 baomidou dynamic-datasource 作为多数据源框架。该框架的核心是 DynamicRoutingDataSource，它是一个支持运行时切换数据源的路由数据源。

DataBaseHelper 如何获取数据源

helper/DataBaseHelper.java

DataBaseHelper 是一个纯静态工具类（私有构造器），在类加载时通过 SpringUtils.getBean(DynamicRoutingDataSource.class) 获取全局唯一的 DynamicRoutingDataSource 实例：

private static final DynamicRoutingDataSource DS =
    SpringUtils.getBean(DynamicRoutingDataSource.class);
DataSource 解析链路： DS.determineDataSource() └── DynamicRoutingDataSource 从当前线程上下文获取 dsKey └── DynamicDataSourceContextHolder.peek() 拿到线程绑定的数据源名称 └── 根据 dsKey 从 DS.dataSourceMap 中取出对应的 DataSource
关键设计：DynamicDataSourceContextHolder.peek() 使用 ThreadLocal 存储当前线程的数据源 key。通过 @DS("slave") 注解或在代码中调用 DynamicDataSourceContextHolder.push("slave") 即可切换数据源。若 peek() 返回 null，则 fallback 为 "primary"（即 master 数据源）。
注意：DynamicRoutingDataSource.determineDataSource() 方法不仅返回当前数据源，还会触发数据源的懒初始化（如果该数据源尚未加载）。这意味着首次访问某个从库时会执行连接池初始化。

2. DataBaseType 数据库类型检测

getDataBaseType() 工作流程

helper/DataBaseHelper.java & enums/DataBaseType.java

该方法通过 JDBC 元数据获取数据库产品名称，并与 DataBaseType 枚举匹配，结果缓存于 ConcurrentHashMap：

getDataBaseType() 执行流程： 1. DS.determineDataSource() → 获取当前 DataSource 2. DynamicDataSourceContextHolder.peek() → 获取 dsKey（null 则用 "primary"） 3. DB_TYPE_CACHE.computeIfAbsent(key, ...) → 检查缓存 ├── 命中缓存 → 直接返回 DataBaseType └── 未命中 → 走以下流程： a. dataSource.getConnection() → 获取 JDBC Connection b. conn.getMetaData() → 获取 DatabaseMetaData c. metaData.getDatabaseProductName() → 获取产品名 MySQL → "MySQL" Oracle → "Oracle" PostgreSQL → "PostgreSQL" SQL Server → "Microsoft SQL Server" d. DataBaseType.find(databaseProductName) → 匹配枚举 e. 结果放入 DB_TYPE_CACHE，key = dsName

DataBaseType 枚举

enums/DataBaseType.java

枚举值数据库产品名便捷判断方法
MY_SQL"MySQL"isMySql()
ORACLE"Oracle"isOracle()
POSTGRE_SQL"PostgreSQL"isPostgreSql()
SQL_SERVER"Microsoft SQL Server"isSqlServer()

find(databaseProductName) 方法遍历枚举值匹配产品名，未匹配或参数为空时 默认返回 MY_SQL。这保证了向后兼容性。

缓存设计：使用 ConcurrentHashMap 以数据源名称为 key 缓存检测结果。这意味着每个数据源的数据库类型只在首次访问时检测一次（通过 computeIfAbsent 的原子操作保证线程安全），后续直接从缓存读取，避免重复的 JDBC 连接开销。

还有一个重载方法 getDataBaseType(String dsName)，用于按指定数据源名称检测数据库类型，同样支持缓存。

3. findInSet 跨数据库兼容

MySQL 的 FIND_IN_SET(str, strlist) 函数在其他数据库中不存在，但业务中经常需要查询"某字段是否在逗号分隔的字符串列表中"。DataBaseHelper.findInSet() 通过 switch 语句根据数据库类型生成对应的 SQL 片段。

各数据库 SQL 对照

helper/DataBaseHelper.java - findInSet(var1, var2)

数据库生成的 SQL 片段原理
MySQL（默认） find_in_set('value', column) <> 0 原生函数
Oracle instr(','||column||',' , ',value,') <> 0 用 instr 在拼接的字符串中查找
PostgreSQL (select strpos(','||column||',' , ',value,')) <> 0 用 strpos 在拼接的字符串中查找
SQL Server charindex(',value,' , ','+column+',') <> 0 用 charindex 在拼接的字符串中查找
为什么要在值前后加逗号？假设字段值为 '12,123'，如果直接 instr('12,123', '12') 会误匹配。而 instr(',12,123,', ',12,') 只在值前后都加逗号后精确匹配，避免了值部分重合的误判。
查看 findInSet 源码实现 
public static String findInSet(Object var1, String var2) {
    String var = Convert.toStr(var1);
    SqlUtil.filterKeyword(var);
    SqlUtil.filterKeyword(var2);
    return switch (getDataBaseType()) {
        case ORACLE -> "instr(','||%s||',' , ',%s,') <> 0".formatted(var2, var);
        case POSTGRE_SQL -> "(select strpos(','||%s||',' , ',%s,')) <> 0".formatted(var2, var);
        case SQL_SERVER -> "charindex(',%s,' , ','+%s+',') <> 0".formatted(var, var2);
        default -> "find_in_set('%s' , %s) <> 0".formatted(var, var2);
    };
}

4. SqlLogInterceptor 深度解析

拦截器注册

interceptor/SqlLogInterceptor.java

SqlLogInterceptor 实现了 MyBatis 的 Interceptor 接口，通过 @Intercepts 注解声明拦截点：

@Intercepts({
    @Signature(type = StatementHandler.class, method = "query",
               args = {Statement.class, ResultHandler.class}),
    @Signature(type = StatementHandler.class, method = "update",
               args = {Statement.class}),
    @Signature(type = StatementHandler.class, method = "batch",
               args = {Statement.class}),
    @Signature(type = StatementHandler.class, method = "queryCursor",
               args = {Statement.class})
})

拦截了 StatementHandler 的 query、update、batch、queryCursor 四个方法，覆盖了所有 SQL 执行路径。

intercept() 核心流程

intercept(Invocation invocation): 1. StatementHandler → PluginUtils.realTarget(invocation.getTarget()) 2. BoundSql boundSql ← statementHandler.getBoundSql() 3. MappedStatement ms ← mappedStatement() 4. long startTime ← System.currentTimeMillis() 5. try: Object result = invocation.proceed() ← 执行真实SQL printSql(ms, boundSql, elapsed, null) ← 打印成功日志 return result catch (Throwable e): printSql(ms, boundSql, elapsed, e) ← 打印异常日志（包含错误信息） throw e ← 重新抛出异常

关键特性：无论 SQL 执行成功还是失败，都会打印日志。异常时额外输出错误类型和错误消息。

buildFullSql() — 参数化 SQL 还原

这是最核心的方法，将带 ? 占位符的预编译 SQL 还原为完整的可执行 SQL：

步骤1 — buildParameterValues()：从 BoundSql 中提取参数值并格式化。对于每个 ParameterMapping：

  • OUT 模式的参数跳过
  • 先从 boundSql.getAdditionalParameter(propertyName) 查找（动态SQL 的额外参数如 foreach 集合）
  • 再从 parameterObject 本身或 MetaObject 中获取
  • 最后调用 formatParameter() 格式化

formatParameter() 类型处理：

Java 类型SQL 输出格式
nullnull
Number / Boolean原始值（如 123、true）
java.util.Date'yyyy-MM-dd HH:mm:ss'（通过 Hutool DateUtil）
LocalDateTime'yyyy-MM-dd HH:mm:ss'
LocalDate'yyyy-MM-dd'
LocalTime'HH:mm:ss'
TemporalAccessor'toString()'
Enum'enum.name()'
其他'toString()'（单引号内的单引号转义为 ''）

步骤2 — replacePlaceholders()：字符级遍历 SQL 字符串，将位于引号外的 ? 替换为参数值，引号内的 ? 不替换：

// 核心逻辑：逐字符扫描，跟踪单引号和双引号状态
for (int i = 0; i < sql.length(); i++) {
    char current = sql.charAt(i);
    if (current == '\'' && !inDoubleQuote) inSingleQuote = !inSingleQuote;
    else if (current == '"' && !inSingleQuote) inDoubleQuote = !inDoubleQuote;
    if (current == '?' && !inSingleQuote && !inDoubleQuote
        && parameterIndex < parameters.size()) {
        builder.append(parameters.get(parameterIndex++));
    } else {
        builder.append(current);
    }
}

printChunk() — 超长 SQL 分片

当日志输出模式为 "log" 且 SQL 超过 8000 字符时，使用 UUID 作为 logId 将 SQL 分片输出：

// 分片格式: sqlLogId={uuid} part=1/3 {sql_chunk_1}
//          sqlLogId={uuid} part=2/3 {sql_chunk_2}
//          sqlLogId={uuid} part=3/3 {sql_chunk_3}

通过相同 UUID 可将分片日志聚合还原为完整 SQL。

控制台输出与线程安全

当 output = "console" 时，SQL 输出到 System.err。使用 ReentrantLock 保证多条 SQL 同时输出时不会交错：

private static final ReentrantLock CONSOLE_LOCK = new ReentrantLock();

private void printConsole(String message) {
    CONSOLE_LOCK.lock();
    try {
        System.err.println(message);
        System.err.println();  // 空行分隔
    } finally {
        CONSOLE_LOCK.unlock();
    }
}

5. SqlLogProperties 配置

配置类

config/properties/SqlLogProperties.java

@Data
@ConfigurationProperties(prefix = "mybatis-plus.sql-log")
public class SqlLogProperties {
    private Boolean enabled = false;   // 默认关闭
    private String output = "console"; // 默认控制台输出
}

条件装配

config/MybatisPlusConfig.java

在 MybatisPlusConfig 中，SqlLogInterceptor 的 Bean 注册使用了 @ConditionalOnProperty：

@Bean
@ConditionalOnProperty(prefix = "mybatis-plus.sql-log", name = "enabled",
                       havingValue = "true")
public SqlLogInterceptor sqlLogInterceptor(SqlLogProperties sqlLogProperties) {
    return new SqlLogInterceptor(sqlLogProperties);
}

只有在配置中设置 mybatis-plus.sql-log.enabled=true 时，拦截器才会被创建。

使用建议：开发环境推荐 output=console（直接查看），生产环境推荐 output=log（写入日志文件，带时间戳和 Mapper ID）。注意：SQL 日志会包含完整参数值，生产环境谨慎开启，避免敏感数据泄露。

6. PlusPostInitTableInfoHandler — 全局逻辑删除开关

功能说明

handler/PlusPostInitTableInfoHandler.java

实现了 MyBatis-Plus 的 PostInitTableInfoHandler 接口，在每张表的 TableInfo 初始化完成后回调：

public class PlusPostInitTableInfoHandler implements PostInitTableInfoHandler {
    @Override
    public void postTableInfo(TableInfo tableInfo, Configuration configuration) {
        String flag = SpringUtils.getProperty("mybatis-plus.enableLogicDelete", "true");
        if (!Convert.toBool(flag)) {
            ReflectUtils.setFieldValue(tableInfo, "withLogicDelete", false);
        }
    }
}

通过读取配置 mybatis-plus.enableLogicDelete（默认 true），若设为 false 则使用反射将每张表的 withLogicDelete 字段设为 false，实现全局关闭逻辑删除。

设计意图：MyBatis-Plus 的逻辑删除是表级别的配置（通过 @TableLogic 注解）。这个 Handler 提供了一个全局开关，无需逐个移除注解即可在需要时（如数据迁移、特殊报表查询）统一关闭逻辑删除功能。

自测题

Q1: DataBaseHelper 中，当 DynamicDataSourceContextHolder.peek() 返回 null 时，fallback 的数据源 key 是什么？

A. "master" B. "default" C. "primary" D. "main"
查看答案答案：C。 源码中 final String key = dsKey != null ? dsKey : "primary";

Q2: SqlLogInterceptor 的 replacePlaceholders() 方法中，什么情况下 ? 不会被替换为参数值？

A. 参数列表为空时 B. ? 在单引号或双引号内部时 C. 参数类型为 OUT 时 D. SQL 长度超过 8000 时
查看答案答案：B。 源码使用 inSingleQuote 和 inDoubleQuote 标志跟踪引号状态，引号内的 ? 是字符串字面量，不替换。

Q3: 以下关于 findInSet 的描述，哪个是正确的？

A. 所有数据库都使用相同的 SQL 表达式 B. Oracle 版本使用 regexp_like 实现 C. PostgreSQL 版本使用 strpos 函数，在字段值前后加逗号防止部分匹配 D. SQL Server 版本直接调用 MySQL 的 find_in_set
查看答案答案：C。 PostgreSQL 版本使用 strpos(','||column||',' , ',value,')，逗号包裹确保精确匹配。

Q4: MybatisExceptionHandler 对 DuplicateKeyException 返回的 HTTP 状态码是什么？

A. 400 B. 404 C. 409 D. 500
查看答案答案：C。 409 Conflict，表示资源冲突（唯一键重复）。
