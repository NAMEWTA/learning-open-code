02 - 数据权限三层架构

🎯 本节目标：完整追踪数据权限链路——从 @DataPermission 注解到 SQL WHERE 条件注入，理解切面层、拦截器层、处理器层三层协作机制，并能解释 6 种 DataScopeType 的 SQL 模板差异。

1. 全景鸟瞰

数据权限是整个 ruoyi-common-mybatis 模块中最复杂的一块。它的目标很简单——不同角色看到不同的数据范围。但实现上涉及 8 个类、6 种范围类型、SpEL 模板解析、JSqlParser SQL 注入。一张图看懂三层架构：

2. 核心设计

2.1 切面层：从注解到 ThreadLocal

@DataPermission 注解本身不执行任何 SQL 拼接——它只是标记。真正的工作在切面层完成：

  • DataPermissionPointcut：匹配所有带 @DataPermission 的 Mapper 方法
  • DataPermissionAdvice：在方法执行前，将注解对象存入 DataPermissionHelper 的 ThreadLocal

为什么用 ThreadLocal 传递？因为 MyBatis 拦截器运行在同一个线程上，ThreadLocal 是最轻量的跨层通信方式，无需修改方法签名。

2.2 拦截器层：SQL 执行前的拦截

PlusDataPermissionInterceptor 继承 BaseMultiTableInnerInterceptor，在 SQL 执行前拦截：

  • beforeQuery：处理 SELECT 语句，用 parserSingle 解析
  • beforePrepare：处理 UPDATE/DELETE 语句，用 parserMulti 解析

拦截器先检查 InterceptorIgnoreHelper（是否主动关闭数据权限），再检查 ThreadLocal 中是否有有效注解，然后委托给处理器生成 WHERE 条件。

2.3 处理器层：SpEL + JSqlParser = SQL 注入

PlusDataPermissionHandler.getSqlSegment() 是核心方法，做三件事：

  1. 获取当前用户角色 → 查角色的 dataScope 字段（1-6）
  2. 匹配 DataScopeType → 取对应的 sqlTemplate（SpEL 模板）
  3. SpEL 解析 → 用 DataPermissionHelper 中的变量替换 #{#deptName}、#{@sdss.xxx}
  4. JSqlParser 注入 → 将生成的 SQL 片段（如 AND dept_id IN (1, 2, 3)）注入原始 SQL 的 WHERE 子句

2.4 六种 DataScopeType 对比

Code类型SQL 模板含义谁能看到什么？

2.5 DataPermissionHelper：三层通信的桥梁

三个核心能力：

  • ThreadLocal<DataPermission>：在切面→拦截器之间传递注解对象
  • SaStorage：在切面→处理器之间传递 SpEL 变量和权限访问控制
  • ignore(Runnable/Supplier)：在子查询中关闭数据权限，防止 SpEL 模板内的 SQL 又触发数据权限造成死循环

DataPermissionIgnoreContext 用 ThreadLocal<Deque<Boolean>> 实现状态栈——支持嵌套忽略：方法 A 忽略中调用方法 B 也忽略，B 退出后恢复 A 的状态。

3. 实战练习

🧠 提取练习（点击展开）

Q1: 为什么数据权限需要三层架构，而不是在拦截器里直接读取注解？

参考: 注解在方法上，但 MyBatis 拦截器拦截的是 SQL 执行，二者不在同一层。切面层负责「把注解信息从方法上提取出来」，拦截器层负责「在 SQL 执行前读取信息并注入条件」，处理器层负责「把注解信息翻译成 SQL 片段」。分层使每层职责单一：切面只管信息提取、拦截器只管时机控制、处理器只管 SQL 生成。

Q2: DataPermissionIgnoreContext 为什么用 Deque（双端队列）而不是简单的 boolean？

参考: 因为需要支持嵌套忽略。场景：方法 A 设置了"忽略数据权限"→调用 B →B 也需要"忽略数据权限"。如果只有 boolean，B 退出时会把 A 的忽略状态也关掉。用 Deque 栈：A 压入 true → B 压入 true → B 弹出恢复 → A 弹出恢复，每层状态独立。

Q3: 如果一个 Mapper 方法上没写 @DataPermission，数据权限还会生效吗？

参考: 不会。PlusDataPermissionInterceptor 在 beforeQuery/beforePrepare 中先检查 DataPermissionHelper 的 ThreadLocal 是否有注解对象——没有就直接跳过，不注入任何 WHERE 条件。这就是为什么数据权限是「显式标注才生效」的安全设计。

4. 调用链路

@DataSource
@DataSource(value = DataSourceType.SLAVE)
List<SysUser> selectUserList(SysUser user);
  → DataPermissionPointcut 匹配到此方法
    → DataPermissionAdvice: ThreadLocal.set(注解对象)
      → MyBatis 执行 SELECT
        → PlusDataPermissionInterceptor.beforeQuery()
          → DataPermissionHelper.getPermission() != null ✓
          → PlusDataPermissionHandler.getSqlSegment()
            → LoginHelper.getLoginUser() → 角色 dataScope = 4 (DEPT_AND_CHILD)
            → DataScopeType.DEPT_AND_CHILD.sqlTemplate
            → SpEL 解析: #{#deptName} → dept_id
            → 查部门树: dept_id IN (1, 2, 3)
          → JSqlParser 注入: WHERE ... AND dept_id IN (1, 2, 3)
        → 执行最终 SQL
    → DataPermissionAdvice: ThreadLocal.remove()
💡 设计洞察：数据权限的设计遵循「关注点分离」——业务 Mapper 只需要标注 @DataPermission，SQL 改写完全透明。最巧妙的是 ignore() 方法：当 SpEL 模板中调用 @sdss 的方法内部又查数据库时，如果不关闭数据权限，就会形成「查角色→需要数据权限→查角色→需要数据权限...」的死循环。用一个 ThreadLocal 栈开关+后续拦截器检查的机制，优雅地解决了这个递归问题。
