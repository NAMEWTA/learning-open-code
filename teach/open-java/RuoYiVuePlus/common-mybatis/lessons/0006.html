第 6 课 · 分页与聚合查询 数据统计

本节目标：学完你能说出 PageQuery.build() 的完整处理流程（含多字段排序和 camelCase 转换），理解 AggregateLambdaQueryWrapper 如何在不修改 MyBatis-Plus 源码的前提下支持聚合查询，掌握 SubQuery 的两种参数模式（MyBatis 占位符 vs {0} 占位符），以及 SqlAggregateFunction 和 AggregateSelectUtils 在整个聚合查询体系中的角色。

一、PageQuery：统一分页模型

1.1 字段设计

字段类型默认值说明
pageSizeIntegerInteger.MAX_VALUE分页大小，默认查全部（与常规 10/20 的默认值不同）
pageNumInteger1当前页码，从 1 开始
orderByColumnStringnull排序列名，支持逗号分隔多字段
isAscStringnull排序方向，"asc"/"desc"，支持逗号分隔对应多字段
注意：pageSize 的默认值是 Integer.MAX_VALUE 而非 10。这意味着如果前端不传分页参数，后端默认查询全部数据。这与其他框架（如 MyBatis-Plus 默认的 10）不同，是一个"防御性"设计——宁可查出全部也不要因为默认分页丢掉数据。

1.2 build() 流程

PageQuery.build() 将自身的分页参数转换为 MyBatis-Plus 的 Page<T> 对象：

public <T> Page<T> build() { ① 获取 pageNum，null→默认1；pageNum ≤ 0 → 修正为1 ② 获取 pageSize，null→默认 Integer.MAX_VALUE ③ new Page<T>(pageNum, pageSize) ④ 调用 buildOrderItem() 构建排序列表 ⑤ 如果排序列表非空，page.addOrder(orderItems) ⑥ 返回 page }

1.3 多字段排序机制

buildOrderItem() 的设计是 PageQuery 中最巧妙的部分，它支持三种排序模式：

// 模式 1：单字段排序 {isAsc: "asc", orderByColumn: "createTime"} → ORDER BY create_time ASC // 模式 2：多字段同方向排序 {isAsc: "asc", orderByColumn: "createTime,userName"} → ORDER BY create_time ASC, user_name ASC // 模式 3：多字段独立方向排序 {isAsc: "desc,asc", orderByColumn: "createTime,userName"} → ORDER BY create_time DESC, user_name ASC

核心处理逻辑：

// 1. camelCase → snake_case 转换（通过 StringUtils.toUnderScoreCase）
String orderBy = "createTime" → "create_time"

// 2. 兼容前端框架的排序值（ascending/descending → asc/desc）
String orderDirection = StringUtils.replaceEach(isAsc,
    new String[]{"ascending", "descending"},
    new String[]{"asc", "desc"});

// 3. 逐字段匹配：isAscArr.length==1 时所有字段共用同方向
//    isAscArr.length==orderByArr.length 时各字段独立方向
//    其他情况 → 抛 ServiceException("排序参数有误")
兼容性设计：不同的前端表格组件（Element Plus、Ant Design、VxeTable 等）传递的排序方向值不同（"ascending"/"asc"）。StringUtils.replaceEach 统一兼容这两种写法，避免了需要在前后端都做转换的麻烦。

1.4 额外的 getFirstNum() 方法

PageQuery 还提供了一个辅助方法 getFirstNum()（标记 @JsonIgnore 不序列化），用于计算 LIMIT 的 offset：

getFirstNum() = (pageNum - 1) * pageSize // 例如 pageNum=3, pageSize=20 → firstNum=40 // 对应 SQL: LIMIT 40, 20

这在需要手动拼接分页 SQL 的场景中非常有用（虽然大多数情况下直接用 build() 返回的 Page 对象就够了）。

二、AggregateLambdaQueryWrapper：聚合查询的桥梁

2.1 为什么需要它

MyBatis-Plus 原生的 LambdaQueryWrapper 只能做简单的列选择（select(col1, col2)），当需要 SELECT SUM(age) AS totalAge, COUNT(*) AS userCount FROM ... 这种聚合查询时，原生的 getSqlSelect() 无法追加聚合函数表达式。RuoYi-Vue-Plus 通过子类重写 getSqlSelect() 解决了这个问题：

LambdaQueryWrapper<T> └── AggregateLambdaQueryWrapper<T> (package-private) ├── aggregateSqlSelect: String ← 追加的聚合字段 SQL ├── appendSelectSql(columnSql) ← 追加到 aggregateSqlSelect ├── getSqlSelect() [@Override] ← 优先返回 aggregateSqlSelect ├── resetAggregateSelect() ← 清空 aggregateSqlSelect（切换回普通模式） ├── columnName(SFunction) ← Lambda → 数据库列名 └── formatSubqueryParam(value) ← 子查询参数格式化

2.2 getSqlSelect() 的重写逻辑

@Override
public String getSqlSelect() {
    if (aggregateSqlSelect != null) {
        return aggregateSqlSelect;      // 聚合模式：返回自定义 SQL
    }
    return super.getSqlSelect();         // 普通模式：走父类默认逻辑
}

关键点：aggregateSqlSelect 不为 null 时直接返回它，完全绕过父类的列拼接逻辑。这意味着调用方可以构造任意复杂的 SELECT 子句——包括聚合函数、子查询、CASE WHEN 等。

2.3 select() 调用时的自动重置

当用户调用 LambdaQueryBuilder.select(columns) 时，内部先执行 aggregateWrapper().resetAggregateSelect()，将 aggregateSqlSelect 置为 null。这样从"聚合查询模式"切换回"普通查询模式"，不会出现聚合 SQL 残留的 bug。

设计智慧：AggregateLambdaQueryWrapper 是 package-private（class 前没有 public），外部代码无法直接实例化。它只通过 QueryBuilder.lambda() 内部创建并包装在 LambdaQueryBuilder 中。这种"对外隐藏、对内扩展"的模式既增强了功能，又没有暴露内部实现细节。

三、SqlAggregateFunction：5 个函数的优雅表达

3.1 枚举设计

枚举值SQL 名称format("col") 输出典型用途
SUMSUMSUM(col)求和：订单总额、用户积分
MAXMAXMAX(col)最大值：最新时间、最高分
MINMINMIN(col)最小值：最早时间、最低价
AVGAVGAVG(col)平均值：平均分、平均时长
COUNTCOUNTCOUNT(col)计数：数量统计、COUNT(*)

3.2 在整个体系中的角色

SqlAggregateFunction 是被使用最广泛的枚举——它同时被以下组件引用：

SqlAggregateFunction（枚举定义 5 个聚合函数） │ ├── LambdaQueryBuilder 的 selectSum/Max/Min/Avg/Count/CountAll/CountDistinct │ └── 内部调用 selectAggregate(function, column, alias) │ └── → AggregateSelectUtils.aggregateSelect(function, column, alias) │ └── → function.format(column) + " AS " + checkAlias(alias) │ ├── LambdaJoinQueryBuilder 的 selectSum/Max/Min/Avg/Count/CountAll │ └── 内部调用 wrapper.selectFunc(DefaultFuncEnum.SUM, ...) │ 或 AggregateSelectUtils.aggregateSelect() │ ├── LambdaCrudChainWrapper 的 selectSum/Max/Min/Avg/Count/CountAll/CountDistinct │ └── 内部调用 selectAggregate(function, column, alias) │ └── SubQuery 的 selectSum/Max/Min/Avg/Count/selectCountAll └── 内部调用 function.format(columnName(column))

一个简单的 format(String expression) 方法（返回 name + "(" + expression + ")"）统一了全模块的聚合函数 SQL 生成逻辑，避免了各处硬编码 "SUM(" + col + ")" 的分散写法。

四、SubQuery：独立子查询构造器

4.1 两种参数模式

SubQuery 作为独立构造器，生成的 SQL 需要嵌入到外层查询中。根据嵌入的场景不同，它支持两种参数格式化模式：

模式创建方式参数占位符使用场景
MyBatis 占位符模式 SubQuery.of(entityClass, formatter) #{ew.paramNameValuePairs.MPJvaGEN1} LambdaQueryBuilder 的子查询：通过 aggregateWrapper()::formatSubqueryParam 作为 formatter
占位参数模式 SubQuery.ofPlaceholders(entityClass) {0}, {1}, {2}... LambdaJoinQueryBuilder 的子查询：通过 apply(sql, params) 传参
// MyBatis 占位符模式（LambdaQueryBuilder 内部） private <S> String buildSubQuery(Class<S> entityClass, Consumer<SubQuery<S>> consumer) { SubQuery<S> subQuery = SubQuery.of(entityClass, aggregateWrapper()::formatSubqueryParam); consumer.accept(subQuery); return subQuery.build(); } // → 生成: SELECT user_id FROM sys_user_role WHERE role_id = #{ew.paramNameValuePairs.MPGENVAL1} // 占位参数模式（LambdaJoinQueryBuilder 内部） private <Q> SubQuery<Q> buildPlaceholderSubQuery(Class<Q> entityClass, Consumer<SubQuery<Q>> consumer) { SubQuery<Q> subQuery = SubQuery.ofPlaceholders(entityClass); // ← placeholderParamMode=true consumer.accept(subQuery); return subQuery; } // 后续通过 wrapper.apply(sql, subQuery.params()) 传入参数 // → 生成: SELECT user_id FROM sys_user_role WHERE role_id = {0}
为什么有两种模式？ LambdaQueryBuilder 的子查询通过 wrapper.eqSql(column, subquerySql) 嵌入，MyBatis 会自动处理 #{...} 占位符。但 LambdaJoinQueryBuilder（MPJ）的子查询是通过 wrapper.apply(sql, params) 嵌入的，需要使用 {0} 占位符并由 apply 方法传入参数数组。

4.2 eqColumn：跨表关联

子查询最强大的功能是 eqColumn，它实现了关联子查询——在子查询中引用外层查询的字段：

// 场景：查询每个用户拥有的角色数量（作为 select 字段）
QueryBuilder.lambda(SysUser.class)
    .select(SysUser::getUserId, SysUser::getUserName)
    .selectSub(SysUserRole.class, sub -> sub
        .selectCountAll()
        .eqColumn(SysUserRole::getUserId, SysUser::getUserId),  // ← 跨表关联
        UserStatVo::getRoleCount)
    .build();

// eqColumn 的两种重载：
// 1. 单表场景（子查询引用外层主表的字段）
eqColumn(SFunction column, SFunction otherColumn)
  → 生成: ur.user_id = sys_user.user_id   （自动加表名前缀）

// 2. 联表场景（子查询引用指定别名的表）
eqColumn(SFunction column, String tableAlias, SFunction otherColumn)
  → 生成: ur.user_id = u.user_id          （使用指定别名）

4.3 disableLogicDelete

SubQuery 默认会追加逻辑删除条件（如 del_flag = 0）。调用 disableLogicDelete() 可以关闭：

// 默认行为：append逻辑删除条件
sub.eq(SysUserRole::getUserId, userId)
  → WHERE (del_flag = '0' AND user_id = ?)

// 禁用后：
sub.disableLogicDelete()
   .eq(SysUserRole::getUserId, userId)
  → WHERE user_id = ?
设计考量：自动追加逻辑删除是安全的默认行为，避免子查询查出已删除的数据。但在某些场景下（如统计历史数据）需要禁用它。这是"默认安全 + 可选择性关闭"的经典设计。

五、AggregateSelectUtils：聚合 SQL 工具集

5.1 四个核心方法

方法签名输出示例
appendSelectappendSelect(current, fragment)"col1" + ", SUM(col2) AS total"（逗号拼接）
aggregateSelectaggregateSelect(function, column, alias)"SUM(age) AS totalAge"
subquerySelectsubquerySelect(sql, alias)"(SELECT COUNT(*) FROM ...) AS roleCount"
aliasNamealiasName(SFunction)从 Lambda Getter 解析属性名（如 "totalAge"）

5.2 checkAlias：SQL 注入防护

checkAlias(String alias) 通过正则表达式验证别名是否为合法 SQL 标识符：

private static final Pattern ALIAS_PATTERN = Pattern.compile("[A-Za-z_][A-Za-z0-9_]*"); public static String checkAlias(String alias) { Assert.isTrue(StringUtils.isNotBlank(alias) && ALIAS_PATTERN.matcher(alias).matches(), "查询别名只能包含字母、数字、下划线且不能以数字开头: %s", alias); return alias; }

使用场景：所有聚合方法的 alias 参数、联表查询的 tableAlias 参数，在拼入 SQL 前都会经过 checkAlias 校验。如果别名包含特殊字符（如 "; DROP TABLE"），会直接抛异常。

防御深度：虽然别名通常来自代码中的 Lambda 引用（SFunction）而非用户输入，但 selectAs(String tableAlias, ...) 和 selectSum(tableAlias, ...) 等方法的 alias 参数是 String 类型，可能来自前端入参。checkAlias 提供了额外一层安全保障。

5.3 在 LambdaQueryBuilder 中的调用链

用户调用: builder.selectSum(SysUser::getAge, "totalAge") → LambdaQueryBuilder.selectSum(column, alias) → selectAggregate(function, aggregateWrapper().columnName(column), alias) → aggregateWrapper().appendSelectSql( AggregateSelectUtils.aggregateSelect(function, "age", "totalAge") → SqlAggregateFunction.SUM.format("age") + " AS " + checkAlias("totalAge") → "SUM(age) AS totalAge" ) → aggregateSqlSelect = "SUM(age) AS totalAge" 最终 MyBatis 生成: SELECT SUM(age) AS totalAge FROM sys_user WHERE ...

六、知识点检查

Q1: PageQuery 的 pageSize 默认值是 Integer.MAX_VALUE 而非 10，这样设计的主要原因是什么？

点击查看答案

A 和 B 都对。主要设计意图是 B——防御性地查出全部数据。同时，当 pageSize = MAX_VALUE 时，MyBatis-Plus 的 PaginationInterceptor 会识别为"不需要分页"，不生成 LIMIT 子句，从而避免了不必要的 SQL 开销。

Q2: SubQuery 的两种参数模式（MyBatis 占位符 vs {0} 占位符）分别用于什么场景？

点击查看答案

B。两种模式对应不同的嵌入方式。LambdaQueryBuilder 通过 wrapper.eqSql(column, subquerySql) 嵌入子查询，MyBatis 的 #{...} 占位符会被 SqlSession 自动处理。而 LambdaJoinQueryBuilder（MPJ）通过 wrapper.apply(sql, params...) 嵌入，参数需要单独传递，所以用 {0} 索引占位符，然后通过 subQuery.params() 获取参数数组传给 apply。

Q3: AggregateLambdaQueryWrapper 的 getSqlSelect() 重写逻辑中，什么情况下返回父类的 getSqlSelect()？

点击查看答案

A。当 aggregateSqlSelect 为 null（即尚未调用任何聚合方法，或调用了 select() 触发了 resetAggregateSelect()）时，返回父类的默认 getSqlSelect()。这意味着它支持"普通查询"和"聚合查询"两种模式的无缝切换。

本课小结：PageQuery 通过 build() 统一了分页 + 多字段排序的入口，AggregateLambdaQueryWrapper 通过子类重写 getSqlSelect() 实现了聚合查询而不修改 MP 源码，SqlAggregateFunction 枚举统一了全模块的聚合函数生成，SubQuery 通过两种参数模式适配了不同 Builder 的嵌入需求，AggregateSelectUtils 的 checkAlias 提供了 SQL 注入防护——这五个组件共同构成了 ruoyi-common-mybatis 的"分页与聚合"能力矩阵。
