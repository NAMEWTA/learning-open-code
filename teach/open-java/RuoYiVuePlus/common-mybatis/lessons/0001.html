第 1 课 · MyBatis 模块全景：四大支柱鸟瞰 入门地图

本节目标：学完你能画出 ruoyi-common-mybatis 的包结构树，说清四大支柱各自解决什么痛点，并理解模块在项目依赖链中的位置——它是所有业务模块与数据库之间的唯一通道。

一、模块定位

ruoyi-common-mybatis 是 RuoYi-Vue-Plus 的核心持久层模块，共 34 个 Java 文件，分布于 9 个包 中。它在依赖链中的位置如下：

ruoyi-common-core（基础工具/异常/响应体） └── ruoyi-common-mybatis（本模块）← 所有业务模块的持久层基础 ├── ruoyi-common-satoken（安全认证，数据权限需要获取登录用户） └── ruoyi-system-api（系统远程接口，获取角色/部门信息） └── ruoyi-modules/ruoyi-system（系统管理业务模块） └── ruoyi-modules/ruoyi-xxx（其他业务模块）

任何需要读写数据库的业务模块（system、job、workflow 等），都直接依赖本模块。它封装了所有与 MyBatis-Plus 打交道的通用逻辑，业务开发者只需关注自己的 Mapper 和 Entity。

三、四大支柱

这 34 个文件不是一盘散沙。它们按功能聚合成四条清晰的责任线：

支柱一：BaseMapperPlus — Entity 与 VO 的自动转换桥梁

核心类：BaseMapperPlus<T, V>、BaseEntity、InjectionMetaObjectHandler

一句话：所有业务 Mapper 继承 BaseMapperPlus，获得「查 Entity → 自动转 VO」的能力，无需在每个 Service 里手动写 MapStruct 转换代码。

关键机制：双泛型设计（T=表实体，V=视图对象），通过 ClassValue 缓存泛型类型，MapstructUtils.convert 自动转换。

详见第 2 课 "BaseEntity 与通用 Mapper" 深度讲解

支柱二：数据权限系统 — 声明式行级过滤

核心类：@DataPermission、PlusDataPermissionInterceptor、PlusDataPermissionHandler

一句话：在 Mapper 方法上加一个注解，SQL 执行时自动拼接 WHERE dept_id IN (...) 或 WHERE create_by = ?，无需在 XML 或 Service 中手写权限 SQL。

三层架构：切面层（AOP 拦截 Mapper 调用）→ 拦截器层（MyBatis 执行前注入 SQL）→ 处理器层（SpEL 解析 + JSqlParser 拼 WHERE）

详见后续课程 "数据权限系统" 深度讲解

支柱三：增强查询构建器 — Lambda + JOIN + 子查询 + 聚合

核心类：LambdaQueryBuilder、LambdaJoinQueryBuilder、LambdaCrudChainWrapper、SubQuery

一句话：用纯 Java Lambda 写出类型安全的复杂 SQL——联表、子查询、聚合统计、FIND_IN_SET，告别 XML 和字符串拼接。

两种入口：QueryBuilder.lambda(SysUser.class) 创建独立查询构建器；mapper.lambda() 创建 Mapper 绑定的链式包装器。

详见后续课程 "查询构建器" 深度讲解

支柱四：多数据源 + SQL 日志 — 运维可观测性

核心类：DataBaseHelper、SqlLogInterceptor（依赖 dynamic-datasource 外部库）

一句话：通过 @DS 注解动态切换数据源，DataBaseHelper 自动识别数据库类型并生成方言兼容 SQL；SqlLogInterceptor 将带 ? 的预编译 SQL 还原为完整可执行 SQL 输出到日志。

关键能力：

  • DataBaseHelper.getDataBaseType() — 通过 DatabaseMetaData 自动识别 MySQL/Oracle/PostgreSQL/SQL Server
  • DataBaseHelper.findInSet(var1, var2) — 根据数据库类型自动切换 find_in_set / instr / strpos / charindex
  • SqlLogInterceptor — 解析 ParameterMapping 还原参数值，超长 SQL 按 8000 字符分片，支持控制台/日志文件两种输出模式

四、痛点-能力映射表

痛点传统做法本模块方案对应支柱
每个 Service 都要手动 Entity → VO 转换 每个方法写 BeanUtils.copyProperties 或 MapStruct BaseMapperPlus.selectVoById() 自动转换 支柱一
每个查询都要手写数据权限 SQL XML 中拼接 AND dept_id IN (${deptIds})，SQL 注入风险 @DataPermission 注解声明，拦截器自动注入 支柱二
复杂查询写 XML 难维护，字段名硬编码 XML 中 ${ew.customSqlSegment} 混合使用 LambdaQueryBuilder / LambdaCrudChainWrapper 纯 Java 类型安全 支柱三
多数据源切换、不同数据库方言处理 手动判断数据库类型，写多套 SQL @DS + DataBaseHelper 自动识别和方言生成 支柱四
SQL 调试只能看带 ? 的预编译语句 手写日志、手动替换参数 SqlLogInterceptor 自动还原完整 SQL + 耗时 支柱四
每个实体类都要重复写审计字段 每个 Entity 手写 createBy/createTime/updateBy/updateTime BaseEntity 统一继承，InjectionMetaObjectHandler 自动填充 支柱一

五、配置文件与 Bean 装配

MybatisPlusConfig 是本模块的唯一配置入口。它注册了 8 个 Bean，组装了 MyBatis-Plus 的拦截器链：

@MapperScan("${mybatis-plus.mapperPackage}") ← 自动扫描 Mapper 接口 @EnableTransactionManagement ← 开启声明式事务 MybatisPlusInterceptor（核心拦截器链）: ├── PlusDataPermissionInterceptor ← 数据权限过滤 ├── PaginationInnerInterceptor ← 分页（自动识别数据库类型） └── OptimisticLockerInnerInterceptor ← 乐观锁 独立 Bean: ├── DataPermissionPointcutAdvisor ← 数据权限切面（条件装配） ├── SqlLogInterceptor ← SQL 日志（mybatis-plus.sql-log.enabled=true 时装配） ├── InjectionMetaObjectHandler ← 自动填充处理器 ├── DefaultIdentifierGenerator ← 雪花 ID 生成器（网卡 MAC 防集群冲突） ├── MybatisExceptionHandler ← MyBatis 异常兜底（@RestControllerAdvice） └── PlusPostInitTableInfoHandler ← 表信息初始化后处理（逻辑删除全局开关）

六、测验

Q1：ruoyi-common-mybatis 模块共有多少个包（含子包）？

A. 5 个 B. 9 个 C. 14 个 D. 34 个
查看答案

答案：B。9 个包：annotation、aspect、config、core、enums、handler、helper、interceptor、utils。D 选项 34 是该模块的 Java 文件总数。

Q2：数据权限系统的三层协作是指哪三层？

A. Controller → Service → Mapper B. 切面层 → 拦截器层 → 处理器层 C. 注解层 → 配置层 → 执行层 D. Mapper → XML → 数据库
查看答案

答案：B。切面层（DataPermissionAdvice 将注解存入 ThreadLocal）→ 拦截器层（PlusDataPermissionInterceptor 在 SQL 执行前检查注解）→ 处理器层（PlusDataPermissionHandler 解析 SpEL + JSqlParser 拼接 WHERE）。

Q3：以下哪个类是 MyBatis-Plus 原生就有的，不是本项目自定义的？

A. BaseMapperPlus<T, V> B. LambdaCrudChainWrapper<T, V> C. BaseMapper<T> D. PlusDataPermissionInterceptor
查看答案

答案：C。BaseMapper<T> 是 MyBatis-Plus 原生接口（仅单泛型）。A、B、D 均为本项目自定义扩展。

Q4：SqlLogInterceptor 输出的完整 SQL 相比 BoundSql 的原始 SQL，核心差异是什么？

A. 增加了执行计划分析 B. 将 ? 占位符替换为实际参数值 C. 自动添加了 SQL 注释 D. 对 SQL 进行了语法优化
查看答案

答案：B。SqlLogInterceptor 通过解析 ParameterMapping 和 MetaObject 获取每个 ? 对应的实际 Java 参数值（处理 Date/LocalDateTime/Enum 等类型），替换后得到可直接拷贝到数据库客户端执行的完整 SQL。
