0003 - 数据权限体系（上）：注解、切面、拦截器 核心机制

学习目标：掌握数据权限三层架构的前半部分——从 @DataPermission 注解声明，到 AOP 切面上下文注入，再到 MyBatis 拦截器 SQL 改写，理解每一层如何协作。

一、全链路概览

数据权限体系由 三层协作 完成。以下是完整链路：

用户代码：SysUserMapper.selectList() │ ▼ [第一层：注解 + 切面] @DataPermission ──→ DataPermissionPointcut.matches() │ │ 匹配成功 ▼ ▼ DataPermissionAdvice.invoke() ├─ DataPermissionHelper.setPermission(注解) ├─ DataPermissionHelper.setVariable(方法参数) ← ThreadLocal + SaStorage ├─ invocation.proceed() │ │ │ ▼ [第二层：MyBatis 拦截器] │ PlusDataPermissionInterceptor.beforeQuery() / beforePrepare() │ ├─ InterceptorIgnoreHelper 检查是否忽略 │ ├─ DataPermissionHandler.invalid() 检查权限是否有效 │ └─ JSqlParser 解析原始 SQL │ │ │ ▼ [第三层：Handler 条件生成] │ PlusDataPermissionHandler.getSqlSegment() │ ├─ LoginHelper.isSuperAdmin() → 超级管理员跳过 │ ├─ buildDataFilter() → 遍历角色 DataScopeType │ ├─ SpEL 模板解析 → SQL WHERE 片段 │ └─ AND 拼接返回 │ └─ finally: DataPermissionHelper.removePermission() ← 清理上下文

三层各自职责：注解层声明"谁需要数据权限"；切面层建立上下文（当前注解、方法参数）；拦截器层根据上下文改写 SQL。

二、@DataPermission 与 @DataColumn

2.1 @DataPermission — 方法/类的标记

org.dromara.common.mybatis.annotation.DataPermission

属性类型说明
value()DataColumn[]数据列映射数组，定义 SQL 中哪些列参与权限过滤
joinStr()String连接符，默认 ""；SELECT 语句使用 OR 连接，UPDATE/DELETE 使用 AND 连接
@Target{METHOD, TYPE}支持方法级和类级注解
设计要点：joinStr 为空的默认行为——SELECT 查询中多个角色权限用 OR 拼接（任一部门可见即可），而 UPDATE/DELETE 操作用 AND 拼接（必须同时满足所有条件），保证写操作的安全性。

2.2 @DataColumn — key-value 列映射

org.dromara.common.mybatis.annotation.DataColumn

属性类型默认值说明
key()String"deptName"SpEL 模板中的占位键，对应 DataScopeType 模板里的 #{#deptName}
value()String"dept_id"真实数据库列名，最终注入 SQL 的列名

通过 @DataColumn 将 SpEL 占位符（#{#deptName}）与真实数据库列（dept_id）解耦。同一张表在不同场景下可用不同的列名，只需修改注解即可。

使用示例：@DataPermission(value = {@DataColumn(key = "deptName", value = "dept_id"), @DataColumn(key = "userName", value = "user_id")}) — 同时声明部门列和用户列，支持 DEPT_AND_CHILD 和 SELF 两种数据范围混合使用。

三、切面匹配与上下文建立

3.1 DataPermissionPointcut — 静态方法匹配

org.dromara.common.mybatis.aspect.DataPermissionPointcut

继承 StaticMethodMatcherPointcut，matches(Method, Class) 逻辑：

  1. 优先检查方法本身是否有 @DataPermission 注解
  2. 若无，检查声明类（targetClass）是否有注解
  3. 若类也没有，且 targetClass 是 JDK 动态代理（接口以 $Proxy 开头），则遍历接口方法查找对应方法上的注解
为什么处理 JDK 代理：MyBatis Mapper 接口默认使用 JDK 动态代理，注解在接口方法上，但代理对象的方法上无法直接读取到注解。此处通过接口反射兼容了这种场景。

3.2 DataPermissionAdvice — 环绕通知上下文注入

org.dromara.common.mybatis.aspect.DataPermissionAdvice

实现 MethodInterceptor，invoke() 的核心流程：

public Object invoke(MethodInvocation invocation) throws Throwable {
    DataPermission dataPermission = findAnnotation(invocation);
    // 1. 获取当前请求的 DataPermissionAccess 对象
    DataPermissionAccess access = DataPermissionHelper.getAccess();
    // 2. 无权限约束则跳过（无 @SaCheckPermission 或 @SaCheckRole）
    if (!access.constrained()) {
        return invocation.proceed();
    }
    // 3. 设置注解到 ThreadLocal，同时注入方法参数到 SaStorage
    DataPermissionHelper.setPermission(dataPermission, invocation.getArguments());
    try {
        return invocation.proceed();  // 执行目标方法 → 触发拦截器
    } finally {
        DataPermissionHelper.removePermission();  // 清理 ThreadLocal
    }
}
为什么用 ThreadLocal：Advice 和 Interceptor 运行在同一线程的不同调用栈层级。ThreadLocal 是最轻量的跨层级上下文传递方案，无需修改方法签名或添加额外参数。配合 finally 块保证清理，防止内存泄漏和上下文污染。

四、MyBatis 拦截器 — SQL 改写入口

4.1 PlusDataPermissionInterceptor

org.dromara.common.mybatis.interceptor.PlusDataPermissionInterceptor

继承 BaseMultiTableInnerInterceptor，是 MyBatis Plus 分页插件的内部拦截器。核心方法：

方法触发时机核心逻辑
beforeQuery() SELECT 查询前 ① 调用 InterceptorIgnoreHelper 检查是否标记为忽略
② 调用 dataPermissionHandler.invalid() 检查是否有有效权限上下文
③ 通过 JSqlParser 解析 SQL 为 Statement 对象
beforePrepare() UPDATE/DELETE 前 ① 同样检查忽略和有效性
② 使用 parserMulti() 解析（支持多语句）
processSelect() 改写 SELECT 遍历 FROM 后的表，为每张表调用 handler.getSqlSegment() 获取权限 WHERE 片段，注入 SQL
processUpdate() 改写 UPDATE 获取权限 WHERE 片段，追加到原有 WHERE 条件后（AND 连接）
processDelete() 改写 DELETE 同上，为 DELETE 语句注入数据权限条件
SELECT 与 UPDATE/DELETE 差异：SELECT 使用 OR 拼接多个角色的权限条件（用户有任一角色即可查看对应数据），UPDATE/DELETE 使用 AND 拼接（确保写操作的数据范围是各角色权限的交集，更严格）。

五、防无限递归：DataPermissionHelper.ignore()

当 SpEL 模板中调用 @sdss.getDeptAndChild(#user.deptId) 时，该方法内部会执行 SQL 查询（如查询部门树）。如果这些内部 SQL 再次触发数据权限拦截器，就会形成无限递归。

5.1 ignore 机制

org.dromara.common.mybatis.helper.DataPermissionHelper

ignore(Runnable) 和 ignore(Supplier<T>) 两个重载方法：

public static void ignore(Runnable runnable) {
    // 通过 DataPermissionIgnoreContext 入栈当前状态
    DataPermissionIgnoreContext.ignore(runnable);
}
// DataPermissionIgnoreContext 内部：
// 1. enable(): 将当前 InterceptorIgnoreHelper 状态入栈，设置忽略=true
// 2. 执行 runnable
// 3. disable(): 出栈恢复之前的状态

在 SysDataScopeServiceImpl.getDeptAndChild() 等 SpEL 调用的方法内部，均使用 DataPermissionHelper.ignore(() -> { ... }) 包裹查询逻辑，确保这些"辅助查询"不触发数据权限过滤。

设计精髓：使用双端队列（Deque）实现栈式管理，支持嵌套调用。外层业务查询开启数据权限 → 内层辅助查询入栈忽略 → 辅助查询完成出栈恢复 → 外层继续使用数据权限。完美解决"自己调用自己"的递归困境。

小节测验

Q1: @DataPermission 的 joinStr 默认值为空字符串时，SELECT 语句中各角色权限条件用什么逻辑连接？

查看答案

B。SELECT 场景下，用户拥有多个角色时，任一角色能看的数据都应可见，所以用 OR 拼接。UPDATE/DELETE 则使用 AND。

Q2: DataPermissionAdvice.invoke() 中为什么要在 finally 块中调用 removePermission()？

查看答案

B。ThreadLocal 在线程池复用时如果不清理，会导致下次请求读到上次遗留的权限上下文，造成数据泄漏。finally 块保证即使方法抛出异常也能清理。

Q3: DataPermissionHelper.ignore() 使用什么数据结构来支持嵌套的忽略/恢复？

查看答案

C。DataPermissionIgnoreContext 使用 ThreadLocal<Deque<Boolean>>，每次 enable() 入栈当前状态，disable() 出栈恢复。这支持了嵌套调用场景。

核心要点回顾：
1. @DataPermission 是声明式注解，@DataColumn 做占位符到真实列名的映射
2. 切面负责建立 ThreadLocal 上下文，拦截器负责 SQL 改写，两者通过 ThreadLocal 跨层通信
3. SELECT 用 OR 拼接角色条件，UPDATE/DELETE 用 AND
4. ignore() 使用栈机制防止 SpEL 辅助查询触发递归数据权限过滤
