常用模式速查

// 模式1：等值查询 + VO自动转换 userMapper.lambda() .eq(SysUser::getStatus, "0") .voList(); // 模式2：模糊搜索 + 分页 Page<SysUser> page = new Page<>(1, 10); userMapper.lambda() .like(SysUser::getUserName, "admin") .page(page); // 模式3：条件更新（setIfPresent 防 null） userMapper.lambda() .eq(SysUser::getUserId, userId) .setIfPresent(SysUser::getEmail, email) .setIfPresent(SysUser::getPhoneNumber, phone) .update(); // 模式4：自增/自减 userMapper.lambda() .eq(SysUser::getUserId, userId) .setIncrBy(SysUser::getLoginCount, 1) .update(); // 模式5：判断存在 boolean exists = userMapper.lambda() .eq(SysUser::getUserName, "admin") .exists(); // 模式6：查询单个 VO（多条时抛异常） SysUserVo vo = userMapper.lambda() .eq(SysUser::getUserId, 1L) .voOne(); // 模式7：统计数量 Long count = userMapper.lambda() .eq(SysUser::getStatus, "0") .count(); // 模式8：多条件 + 排序 userMapper.lambda() .eq(SysUser::getStatus, "0") .ge(SysUser::getCreateTime, startTime) .le(SysUser::getCreateTime, endTime) .orderByDesc(SysUser::getCreateTime) .voList();

反模式：应避免的写法

反模式问题正确做法
不用 BaseEntity，每个实体手写审计字段 重复代码，容易遗漏字段或填错填充时机 继承 BaseEntity，让 InjectionMetaObjectHandler 自动处理
调用 selectById() 然后手动 BeanUtils.copyProperties 每个查询都要写转换代码，VO 类型硬编码在 Service 中 使用 selectVoById() 或 voList()，转换逻辑统一在 Mapper
用 mapper.updateById(user) 做部分字段更新 如果 user 对象中某个字段为 null，会把数据库对应字段也更新为 null 用 mapper.lambda().eq(...).setIfPresent(...).update() 精确更新
不判断 selectVoOne() 返回 null，直接 .getXxx() 查不到记录时产生 NPE 判断 null 或使用 oneOpt()
把 BaseMapperPlus 当 BaseMapper 声明 丢失第二个泛型，所有 selectVo* 方法无法自动推导 VO 类型 始终用 BaseMapperPlus<Entity, Vo> 声明
在循环中调 insertBatch insertBatch 内部已经循环，外层再循环造成 N+1 次网络 IO 收集到 List 后一次 insertBatch 调用
在 BaseEntity 的子类中又声明 createBy/createTime 等字段 子类字段覆盖父类，但 InjectionMetaObjectHandler 只检查 instanceof BaseEntity，可能填充到错误的字段上 只继承不覆盖
记忆口诀：「查实体、转 VO，一条 selectVo 全搞定；填字段、靠注入，-1 代表无用户。」
术语参考：BaseEntity、InjectionMetaObjectHandler、BaseMapperPlus、LambdaCrudChainWrapper 等术语定义参见 GLOSSARY.md。
