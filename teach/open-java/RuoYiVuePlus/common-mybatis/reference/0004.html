DataPermissionHelper 方法速查

方法类型存储说明
setPermission(DataPermission, args)写入ThreadLocal缓存当前方法注解 + 注入方法参数到 SaStorage
getPermission()读取ThreadLocalHandler 中获取 @DataPermission 注解
removePermission()清理ThreadLocalfinally 块中移除，防止泄漏
setVariable(String, Object)写入SaStorage存储 SpEL 变量（如 user, roleId）
getVariable(String)读取SaStorage获取 SpEL 变量
setAccess(DataPermissionAccess)写入SaStorage缓存权限约束对象
getAccess()读取SaStorage获取权限约束对象
ignore(Runnable)控制Deque 栈临时关闭数据权限（用于 SpEL 辅助查询）
ignore(Supplier<T>)控制Deque 栈同上，带返回值

DataPermissionAccess 结构

字段类型说明
permsSet<String>权限标识集合，来自 @SaCheckPermission
roleKeysSet<String>角色标识集合，来自 @SaCheckRole
constrained()boolean是否有约束：perms 或 roleKeys 任一非空

端到端调用链

Controller: @SaCheckPermission("system:user:list") │ ├─ resolveAccess() → DataPermissionAccess(perms={"system:user:list"}) │ ▼ Mapper: SysUserMapper.selectList() @DataPermission({@DataColumn(key="deptName", value="dept_id")}) │ ▼ Pointcut.matches() → true │ ▼ Advice.invoke(): setPermission(注解, args) → ThreadLocal + SaStorage │ ▼ Interceptor.beforeQuery(): ├─ 检查 ignore → false ├─ 检查 invalid → 有效 └─ JSqlParser 解析 SQL │ ▼ Handler.getSqlSegment(): ├─ isSuperAdmin? → false └─ buildDataFilter(): ├─ 获取用户角色 [{admin→ALL}, {common→DEPT}] ├─ scopeRoles(): 过滤 → [{common→DEPT}] ├─ 遍历角色 common: │ DataScopeType=DEPT → 模板: #{#deptName}=#{#user.deptId} │ DataColumn[{deptName→dept_id}] → 替换 → dept_id=#{#user.deptId} │ SpEL解析 #{#user.deptId} → 103 │ → SQL片段: dept_id = 103 └─ 返回: (dept_id = 103) │ ▼ 最终 SQL: SELECT * FROM sys_user WHERE del_flag='0' AND (dept_id = 103) │ ▼ finally: removePermission() → ThreadLocal.remove()

常见陷阱速查

问题原因解决
无限递归 SpEL 模板中 @sdss.getDeptAndChild() 内部执行 SQL，再次触发数据权限拦截器 在 sdss 方法内部使用 DataPermissionHelper.ignore(() -> {...}) 包裹查询
SpEL 变量为 null 抛异常 定义了多个 @DataColumn 但某些角色的模板只用到了部分 key，未匹配的 key 为 null NullSafeStandardEvaluationContext 返回 -1 防止异常
注解不生效 注解在实现类上，但 Mapper 是 JDK 动态代理，调用的是接口方法 注解必须放在 Mapper 接口方法上，或通过 Pointcut 的接口遍历逻辑匹配
UPDATE/DELETE 权限过宽 使用默认 joinStr (空串)，SELECT 走 OR 而 UPDATE/DELETE 应走 AND 多数场景默认行为已正确（空串在非 SELECT 时默认 AND），确认 joinStr 语义
ThreadLocal 泄漏 线程池复用导致下次请求读到上次的权限上下文 Advice.invoke() 的 finally 块调用 removePermission()
