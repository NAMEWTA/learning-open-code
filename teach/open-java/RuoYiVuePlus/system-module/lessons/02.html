2. BaseEntity — 审计字段的统一抽象

所有核心实体都继承 BaseEntity，MyBatis-Plus 的自动填充机制会在插入/更新时自动设置这些字段：

// 所有实体共有的审计字段（通过继承获得）
public class BaseEntity implements Serializable {
    private Long createBy;      // 创建者
    private LocalDateTime createTime;  // 创建时间
    private Long updateBy;      // 更新者
    private LocalDateTime updateTime;  // 更新时间
}
设计意图：避免在每个实体中重复定义审计字段。配合 MyBatis-Plus 的 MetaObjectHandler 自动从登录上下文获取当前用户 ID 填充 createBy/updateBy。

3. 四大核心实体详解

3.1 SysUser — 用户

字段类型说明
user_idLong主键，@TableId
dept_idLong所属部门
user_nameString登录账号
nick_nameString显示昵称
passwordStringBCrypt 加密，NOT_EMPTY 策略（不更新时不设值）
statusString0=正常 1=停用
del_flagString@TableLogic 逻辑删除
login_ip/login_dateString/DateTime最后登录信息
核心方法：isSuperAdmin() — 判断是否为超级管理员（userId=1）

3.2 SysRole — 角色

字段类型说明
role_idLong主键
role_nameString角色名称（如"超级管理员"）
role_keyString权限字符（如"super_admin"），用于 Sa-Token 角色校验
data_scopeString数据范围：1=全部 2=自定义 3=本部门 4=本部门及以下 5=仅本人 6=部门及以下或本人
menu_check_strictlyBoolean菜单树是否父子联动
dept_check_strictlyBoolean部门树是否父子联动
role_sortInteger排序字段
关键字段 data_scope：这是数据权限的核心！角色通过此字段决定用户能看到哪些部门的数据。具体实现在第4课详讲。

3.3 SysMenu — 菜单/权限

字段类型说明
menu_idLong主键
parent_idLong父菜单ID（0=根节点）
menu_typeStringM=目录 C=菜单 F=按钮（三种类型对应不同前端渲染方式）
permsString权限标识，如 system:user:list
pathString路由地址，如 system/user
componentString前端组件路径，如 system/user/index
is_frameString是否外链（Y/N）
is_cacheString是否缓存（Y/N）
visibleString是否显示（0=显示 1=隐藏）

SysMenu 的三个核心计算方法（前端路由生成的关键！）：

  • getRouteName() — 生成路由 name：path 首字母大写 + menuId
  • getRouterPath() — 计算实际路由路径（处理内链、外链、顶级目录等场景）
  • getComponentInfo() — 确定前端组件（Layout / InnerLink / ParentView / 自定义组件）

3.4 SysDept — 部门

字段类型说明
dept_idLong主键
parent_idLong父部门ID
ancestorsString祖级列表，如 "0,100,200" — 无限级树的关键
dept_nameString部门名称
leaderLong负责人 user_id
dept_categoryString部门类别编码
ancestors 字段的作用：通过 FIND_IN_SET(dept_id, ancestors) 可以一次性查出某部门的所有子部门，避免递归查询。这是"预排序遍历树"的变体实现。

4. 关联表：表达多对多关系

关联表关联关系用途
sys_user_roleuser_id ↔ role_id用户可拥有多个角色
sys_user_postuser_id ↔ post_id用户可担任多个岗位
sys_role_menurole_id ↔ menu_id角色拥有多个菜单权限
sys_role_deptrole_id ↔ dept_id角色的数据权限部门范围
