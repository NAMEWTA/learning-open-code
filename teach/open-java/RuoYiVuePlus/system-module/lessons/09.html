🔧 第9课：Service 层最佳实践

胜利目标：理解 Service 层的通用编码模式：事务管理、关联数据维护、远程调用接口、事件发布机制。

1. 事务管理模式

// 模式1：单表操作 — 不需要 @Transactional
public int updateUserStatus(Long userId, String status) {
    return userMapper.lambda()
        .set(SysUser::getStatus, status)
        .eq(SysUser::getUserId, userId)
        .updateCount();
}

// 模式2：多表操作 — 必须 @Transactional
@Override
@Transactional(rollbackFor = Exception.class)  // 🔑 rollbackFor = Exception.class
public int insertUser(SysUserBo user) {
    SysUser sysUser = MapstructUtils.convert(user, SysUser.class);
    int rows = userMapper.insert(sysUser);    // ① 插入用户
    user.setUserId(sysUser.getUserId());
    insertUserPost(user, false);              // ② 插入岗位关联
    insertUserRole(user, false);              // ③ 插入角色关联
    return rows;  // ①②③ 任意一步失败都回滚
}

// 模式3：删除关联 + 删除主表
@Override
@Transactional(rollbackFor = Exception.class)
public int deleteUserById(Long userId) {
    userRoleMapper.lambda().eq(SysUserRole::getUserId, userId).delete();  // ① 删角色关联
    userPostMapper.lambda().eq(SysUserPost::getUserId, userId).delete();  // ② 删岗位关联
    int flag = userMapper.deleteById(userId);  // ③ 删用户
    if (flag < 1) throw new ServiceException("删除用户失败!");
    return flag;
}
为什么 rollbackFor = Exception.class？Spring 默认只对 RuntimeException 回滚。ServiceException 是 RuntimeException，但为了防御未来可能抛出的 checked exception，显式声明回滚所有异常是最佳实践。

2. 关联数据维护：先清后建模式

这是 ruoyi 处理 M:N 关联的标准模式：

// 用户角色关联的标准流程
private void insertUserRole(Long userId, Long[] roleIds, boolean clear) {
    if (ArrayUtil.isEmpty(roleIds)) return;

    // ① 权限校验：非超管不能分配超管角色
    if (!LoginHelper.isSuperAdmin(userId)) {
        roleList.remove(SystemConstants.SUPER_ADMIN_ROLE_ID);
    }

    // ② 数据权限校验：确保有权限操作这些角色
    if (roleMapper.selectRoleCount(roleList) != roleList.size()) {
        throw new ServiceException("没有权限访问角色的数据");
    }

    // ③ 先清空旧关联（如果 clear=true）
    if (clear) {
        userRoleMapper.lambda().eq(SysUserRole::getUserId, userId).delete();
    }

    // ④ 批量插入新关联
    List<SysUserRole> list = StreamUtils.toList(roleList, roleId -> {
        SysUserRole ur = new SysUserRole();
        ur.setUserId(userId);
        ur.setRoleId(roleId);
        return ur;
    });
    userRoleMapper.insertBatch(list);
}
"先清后建"的优点：比"对比差异、增量更新"简单得多，避免了复杂的 diff 逻辑。在有事务保护的情况下，性能开销可接受（通常关联数据量不大）。

3. 事件驱动 — 权限变更的副作用处理

// 当角色权限被修改时，需要踢出所有拥有该角色的在线用户
// 实现方式：发布 Spring Event → 监听器处理

// ① Controller 层发布事件
@PutMapping("/permission")
public R<Void> editPermission(@RequestBody SysRoleBo role) {
    if (roleService.updateRolePermission(role) > 0) {
        // 🔑 发布事件：通知清理该角色的在线用户
        SpringUtils.context().publishEvent(
            OnlineUserCleanEvent.byRole(role.getRoleId())
        );
        return R.ok();
    }
    return R.fail("修改失败");
}

// ② RoleService 也提供直接清理能力（被监听器调用）
public void cleanOnlineUserByRole(Long roleId) {
    // 遍历所有在线 Token，登出拥有该角色的用户
    List<String> keys = StpUtil.searchTokenValue("", 0, -1, false);
    keys.parallelStream().forEach(key -> {
        LoginUser loginUser = LoginHelper.getLoginUser(token);
        if (loginUser.getRoles().stream().anyMatch(r -> r.getRoleId().equals(roleId))) {
            StpUtil.logoutByTokenValue(token);
        }
    });
}

4. 远程调用接口 — 跨微服务能力暴露

ruoyi-system 实现了 ruoyi-api 模块定义的接口，供其他微服务通过 Feign 调用：

// SysUserServiceImpl 同时实现了 ISysUserService 和 UserService
@Service
public class SysUserServiceImpl implements ISysUserService, UserService {

    // ISysUserService — 本地调用的完整接口
    //   selectPageUserList(), insertUser(), updateUser(), deleteUserById()...

    // UserService（来自 ruoyi-api）— 远程调用的精简接口
    //   selectById()       → 供其他服务查询用户基本信息
    //   selectListByIds()  → 供其他服务批量查询用户
    //   selectUsersByRoleIds()   → 供工作流等模块按角色查用户
    //   selectUsersByDeptIds()   → 供其他模块按部门查用户
    //   selectUsersByPostIds()   → 供其他模块按岗位查用户

    @Override
    public UserDTO selectById(Long userId) {
        // 只返回用户名、昵称、部门ID等基本信息
        // 不返回密码、手机号等敏感信息
        SysUserVo vo = userMapper.lambda()
            .select(SysUser::getUserId, SysUser::getDeptId, SysUser::getUserName,
                    SysUser::getNickName, SysUser::getStatus)
            .eq(SysUser::getStatus, SystemConstants.NORMAL)
            .eq(SysUser::getUserId, userId)
            .voOne();
        return BeanUtil.copyProperties(vo, UserDTO.class);
    }
}
接口隔离原则的体现：本地调用的 ISysUserService 提供完整 CRUD，远程调用的 UserService 只暴露安全的读方法（脱敏后），不暴露写操作和敏感字段。

5. QueryBuilder — 流式查询条件构建

// ruoyi 封装的 QueryBuilder，在 LambdaQueryWrapper 上增加了语义化方法
private Wrapper<SysUser> buildQueryWrapper(SysUserBo user) {
    return QueryBuilder.lambda(SysUser.class)
        .eq(SysUser::getDelFlag, SystemConstants.NORMAL)       // 固定条件
        .eqIfPresent(SysUser::getUserId, user.getUserId())     // 可选条件
        .likeIfText(SysUser::getUserName, user.getUserName())  // 文本模糊匹配
        .eqIfText(SysUser::getStatus, user.getStatus())        // 字符串精确匹配
        .inIfNotEmpty(SysUser::getUserId, userIds)             // 集合 IN
        .betweenParams(SysUser::getCreateTime, params,         // 时间区间
                       "beginTime", "endTime")
        .and(condition, w -> { ... })                          // 子条件组
        .orderByAsc(SysUser::getUserId)                        // 排序
        .build();
}
// IfPresent/IfText/IfNotEmpty 的后缀表示：参数为空时跳过该条件
