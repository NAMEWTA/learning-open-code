🔐 第3课：RBAC 权限模型 — 完整链路

胜利目标：能口述从用户登录到每次请求权限校验的完整链路，理解角色权限和菜单权限的区别。

1. RBAC 模型在 ruoyi-system 中的映射


   User（用户）──── M:N ──── Role（角色）──── M:N ──── Menu（菜单/权限）
      │                      │
      │                      ├── role_key ──→ Sa-Token @SaCheckRole
      │                      └── data_scope → DataScope 插件
      │
      └── 最终拥有的权限 = 所有角色的菜单权限的并集
RBAC 核心公式：一个用户的最终权限 = Σ(其所有角色的 menu 权限) ∪ (超级管理员特殊处理)

2. 权限的两层含义

ruoyi-system 中"权限"分为两个独立维度：

维度存储位置校验方式示例
角色权限sys_role.role_key@SaCheckRole("super_admin")是否可以访问某功能模块
菜单权限sys_menu.perms@SaCheckPermission("system:user:list")是否可以执行某具体操作

3. 登录时 — 权限加载

当用户登录成功后，系统会将权限信息加载到 LoginUser 对象中并存入 Redis：

// 登录时调用 SysPermissionServiceImpl
LoginUser loginUser = new LoginUser();

// 1. 加载角色权限（role_key 集合）
Set<String> rolePermission = permissionService.getRolePermission(userId);
// → 超管: {"super_admin"}
// → 普通用户: {"common", "xxx"}

// 2. 加载菜单权限（perms 集合）
Set<String> menuPermission = permissionService.getMenuPermission(userId);
// → 超管: {"*:*:*"}
// → 普通用户: {"system:user:list", "system:user:add", ...}

loginUser.setRolePermission(rolePermission);
loginUser.setMenuPermission(menuPermission);

// 存入 Redis (Sa-Token Session)
StpUtil.getTokenSession().set("loginUser", loginUser);

关键源码：SysPermissionServiceImpl

@Service
public class SysPermissionServiceImpl implements ISysPermissionService {
    // 获取角色权限：超管返回 super_admin，否则查 sys_role 表
    public Set<String> getRolePermission(Long userId) {
        if (LoginHelper.isSuperAdmin(userId)) {
            return Set.of("super_admin");
        }
        return roleService.selectRolePermissionByUserId(userId);
    }

    // 获取菜单权限：超管返回 *:*:*，否则查 sys_menu 表
    public Set<String> getMenuPermission(Long userId) {
        if (LoginHelper.isSuperAdmin(userId)) {
            return Set.of("*:*:*");  // 通配符，匹配所有权限
        }
        return menuService.selectMenuPermsByUserId(userId);
    }
}
超级管理员的特殊处理：
  • 角色权限 → 直接返回 super_admin
  • 菜单权限 → 直接返回 *:*:*（万能通配符）
  • 数据权限 → 跳过所有数据过滤
  • 这样避免了为超管分配海量菜单的麻烦，也提升了校验性能

4. 每次请求 — 权限校验

Sa-Token 拦截请求，执行注解校验：

// Controller 层 — 两种校验方式

// 方式1：按角色校验
@SaCheckRole(value = {"super_admin"}, mode = SaMode.OR)
@GetMapping("/list")
public R<...> list() { ... }

// 方式2：按权限字符串校验
@SaCheckPermission("system:user:list")
@GetMapping("/list")
public R<...> list() { ... }

// 方式3：组合校验（OR 模式）
@SaCheckPermission(
    value = {"system:user:add", "system:user:edit"},
    mode = SaMode.OR
)
@PostMapping
public R<...> add() { ... }

5. 权限查询的 SQL 路径

以普通用户为例，查询其菜单权限的完整 SQL 链路：


SysPermissionServiceImpl.getMenuPermission(userId)
  │
  ▼
SysMenuServiceImpl.selectMenuPermsByUserId(userId)
  │
  ▼
SysMenuMapper.selectMenuPermsByUserId(userId)
  │  └── XML SQL:
  │      SELECT DISTINCT m.perms
  │      FROM sys_menu m
  │      LEFT JOIN sys_role_menu rm ON m.menu_id = rm.menu_id
  │      LEFT JOIN sys_user_role ur ON rm.role_id = ur.role_id
  │      WHERE ur.user_id = #{userId} AND m.status = '0'
  │
  ▼
返回: {"system:user:list", "system:user:add", "system:role:list", ...}

6. 权限字符串的命名规范

ruoyi-system 遵循 模块:实体:操作 的命名规范：

模块权限标识示例操作
用户管理system:user:list查询
用户管理system:user:add新增
用户管理system:user:edit修改
用户管理system:user:remove删除
用户管理system:user:export导出
用户管理system:user:import导入
角色管理system:role:list查询
菜单管理system:menu:add新增
为什么菜单管理只允许超管操作？观察 SysMenuController 的代码：@SaCheckRole(SystemConstants.SUPER_ADMIN_ROLE_KEY) 出现在新增/修改/删除方法上。菜单是权限的基础设施，只允许超级管理员修改，防止权限体系被破坏。
