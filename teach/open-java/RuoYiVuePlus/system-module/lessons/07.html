2. 缓存注解使用模式

2.1 @Cacheable — 读时缓存

// 模式1：简单缓存 — 有就返回，没有就查库并缓存
@Cacheable(cacheNames = CacheNames.SYS_USER_NAME, key = "#userId")
public String selectUserNameById(Long userId) {
    return userMapper.lambda().select(SysUser::getUserName)...;
}

// 模式2：条件缓存 — 只在 key 不为 null 时缓存
@Cacheable(cacheNames = CacheNames.SYS_ROLE_CUSTOM, key = "#roleId",
           condition = "#roleId != null")
public String getRoleCustom(Long roleId) {
    // 防止 roleId=null 导致缓存空值污染
}

2.2 @CacheEvict — 写时失效

// 单条失效 — 按 key 精准删除
@CacheEvict(cacheNames = CacheNames.SYS_NICKNAME, key = "#user.userId")
public int updateUserProfile(SysUserBo user) { ... }

// 全量清空 — 当无法按 key 精准删除时（如部门移动影响所有子部门）
@CacheEvict(cacheNames = CacheNames.SYS_DEPT_AND_CHILD, allEntries = true)
public int insertDept(SysDeptBo bo) { ... }

// 🔑 组合失效 — @Caching 同时清理多个缓存
@Caching(evict = {
    @CacheEvict(cacheNames = CacheNames.SYS_DEPT, key = "#bo.deptId"),
    @CacheEvict(cacheNames = CacheNames.SYS_DEPT_AND_CHILD, allEntries = true)
})
public int updateDept(SysDeptBo bo) { ... }

3. 同类方法调用的 AopProxy 技巧

Spring 的 @Cacheable 是基于 AOP 代理的，同类内部调用不经过代理，缓存注解失效。ruoyi 的解决方案：

// ❌ 问题代码 — 同类内部调用不触发缓存
public String selectNicknameByIds(String userIds) {
    for (Long id : StringUtils.splitTo(userIds, Convert::toLong)) {
        // this.selectNicknameById(id);  ← 直接调用，不经过代理，缓存不生效！
    }
}

// ✅ ruoyi 解决方案 — 通过 AopProxy 走代理
public String selectNicknameByIds(String userIds) {
    for (Long id : StringUtils.splitTo(userIds, Convert::toLong)) {
        // 通过 SpringUtils.getAopProxy(this) 获取当前对象的代理
        String nickname = SpringUtils.getAopProxy(this).selectNicknameById(id);
        // ...
    }
}
// SpringUtils.getAopProxy(this) 返回的是代理对象
// 代理对象上的 @Cacheable 才会生效
AopProxy 的本质：从一个 Bean 内部调用自身的另一个方法时，需要显式获取代理对象来调用，才能让 AOP 切面（包括 @Cacheable、@Transactional）生效。ruoyi 封装的 SpringUtils.getAopProxy(this) 就是做这件事。

4. 缓存的失效时机策略

操作失效策略原因
修改个人信息精准失效：按 userId只影响单个用户的昵称
新增部门全量失效：SYS_DEPT_AND_CHILD新部门可能改变任意部门的子部门列表
移动部门精准 + 全量：SYS_DEPT + SYS_DEPT_AND_CHILD自身信息变了 + 子部门树变了
修改角色权限精准失效 + 踢出在线用户权限变更需要立即生效（通过事件）
删除角色（批量）全量失效：allEntries = true批量删除无法逐个精准失效

5. 缓存 Key 设计原则

// ruoyi 的缓存 Key 命名模式
// 1. 对象缓存 — 用对象 ID 作为 key
//    SYS_DEPT::123     → 部门123的详情
//    SYS_USER_NAME::456 → 用户456的登录名

// 2. 列表缓存 — 用关键参数作为 key
//    SYS_ROLE_CUSTOM::789 → 角色789的自定义数据范围
//    SYS_DEPT_AND_CHILD::100 → 部门100及其所有子部门

// 3. 功能缓存
//    PWD_ERR_CNT_KEY::admin → admin 用户的密码错误次数
