🎯 第4课：数据权限（Data Scope）— 从配置到 SQL 注入

胜利目标：能画出数据权限的完整链路，说出 5 种 data_scope 的含义和 SQL 注入方式。

1. 什么是数据权限？

功能权限控制"能不能访问某页面/按钮"，而数据权限控制"能看到哪些数据行"。

例如：两个用户都有 system:user:list 权限，但 A 是总部管理员（能看到所有人），B 是部门经理（只能看到自己部门的人）。这就是数据权限的体现。

2. 六种数据范围

值含义SQL 注入逻辑
1全部数据权限不注入任何条件，能看到所有数据
2自定义数据权限注入 dept_id IN (角色配置的部门列表)
3本部门数据权限注入 dept_id = 当前用户的部门ID
4本部门及以下数据权限注入 dept_id IN (当前部门 + 所有子部门ID)
5仅本人数据权限注入 user_id = 当前用户ID
6部门及以下或本人数据权限注入 dept_id IN (...) OR user_id = 当前用户ID

3. 数据权限的完整链路


【请求到达】
POST /system/user/list
Authorization: Bearer xxx-token
    │
    ▼
【Controller】SysUserController.list()
    @SaCheckPermission("system:user:list")  ← 功能权限检查
    │
    ▼
【Service】SysUserServiceImpl.selectPageUserList()
    │
    ▼
【Mapper】SysUserMapper.selectPageUserList()
    │  ← MyBatis-Plus 拦截器介入！
    ▼
【DataScope 插件】注入 SQL 条件
    │
    │  1. 从 LoginHelper 获取当前用户的所有角色
    │  2. 遍历角色，根据每个角色的 data_scope 计算条件
    │  3. 按 OR 拼接多个角色的条件
    │
    ▼
【最终 SQL】
SELECT * FROM sys_user
WHERE del_flag = '0'
  AND (
    -- 角色1: data_scope=4（本部门及以下）
    dept_id IN (100, 101, 102)
    OR
    -- 角色2: data_scope=5（仅本人）
    user_id = 123
  )

4. 核心源码分析

4.1 SysDataScopeService — 数据范围计算

@Service("sdss")  // 注意 Bean 名称缩写
public class SysDataScopeServiceImpl implements ISysDataScopeService {

    // 获取角色自定义权限：返回角色关联的部门ID列表
    @Cacheable(cacheNames = "sys_role_custom", key = "#roleId")
    public String getRoleCustom(Long roleId) {
        // 从 sys_role_dept 表查询角色关联的部门
        List<SysRoleDept> list = roleDeptMapper.lambda()
            .select(SysRoleDept::getDeptId)
            .eq(SysRoleDept::getRoleId, roleId)
            .list();
        return StreamUtils.join(list, rd -> Convert.toStr(rd.getDeptId()));
    }

    // 获取部门及以下权限：返回当前部门 + 所有子部门ID
    @Cacheable(cacheNames = "sys_dept_and_child", key = "#deptId")
    public String getDeptAndChild(Long deptId) {
        List<Long> deptIds = deptMapper.selectDeptAndChildById(deptId);
        return StreamUtils.join(deptIds, Convert::toStr);
    }
}

4.2 DataScope 拦截器（在 common-mybatis 中）

DataScope 插件是 MyBatis-Plus 的拦截器，拦截所有标注了 @DataScope 注解的查询，自动注入 WHERE 条件。它根据当前登录用户拥有的角色的 data_scope 值动态拼接 SQL：

// 伪代码：DataScope 拦截器的核心逻辑
for (Role role : currentUser.getRoles()) {
    switch (role.getDataScope()) {
        case "1": // 全部 → 不注入条件
            return;
        case "2": // 自定义 → dept_id IN (...)
            String customDeptIds = dataScopeService.getRoleCustom(role.getRoleId());
            conditions.add("dept_id IN (" + customDeptIds + ")");
            break;
        case "3": // 本部门 → dept_id = currentUser.deptId
            conditions.add("dept_id = " + currentUser.getDeptId());
            break;
        case "4": // 本部门及以下 → dept_id IN (deptId + 子部门)
            String allDeptIds = dataScopeService.getDeptAndChild(currentUser.getDeptId());
            conditions.add("dept_id IN (" + allDeptIds + ")");
            break;
        case "5": // 仅本人 → user_id = currentUser.userId
            conditions.add("user_id = " + currentUser.getUserId());
            break;
    }
}
// 多个角色的条件用 OR 拼接
sql += " AND (" + String.join(" OR ", conditions) + ")";
关键设计细节：
  • 多个角色之间用 OR 连接——只要满足任一角色的数据范围即可
  • SysDataScopeServiceImpl 标注了 @Service("sdss") 避免名称过长
  • @Cacheable 缓存数据范围计算结果，避免每次查询都重新计算部门树
  • 注解中包含 condition = "#roleId != null" 防止缓存空值

5. 数据权限在 Controller 层的检查

除了 SQL 层面的注入，Controller 层也有显式的数据权限校验：

// SysUserController.edit()
@PutMapping
public R<Void> edit(@RequestBody SysUserBo user) {
    // 显式检查当前用户是否有权限操作目标用户的数据
    userService.checkUserDataScope(user.getUserId());
    // 显式检查当前用户是否有权限操作目标部门的数据
    deptService.checkDeptDataScope(user.getDeptId());
    // ...
}

// SysDeptServiceImpl.checkDeptDataScope()
public void checkDeptDataScope(Long deptId) {
    if (LoginHelper.isSuperAdmin()) return;  // 超管跳过
    if (deptMapper.countDeptById(deptId) == 0) {
        throw new ServiceException("没有权限访问部门数据！");
    }
}
两重保险：SQL 层面通过拦截器注入 WHERE 条件（读操作）+ Controller 层面通过显式校验（写操作），双重保证数据安全。
