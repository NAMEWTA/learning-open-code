🌲 第6课：部门树与祖级路径 — 无限级树形结构的工程实践

胜利目标：理解 ancestors 字段的设计原理，能解释部门移动时如何更新所有子孙节点的祖先路径。

1. 两种树形存储方案对比

方案存储查子节点查祖先移动节点
邻接表只存 parent_id递归查询（慢）递归查询（慢）只改一行（快）
路径枚举（ruoyi 方案）parent_id + ancestorsFIND_IN_SET（快）FIND_IN_SET 反查（快）需更新所有子孙（慢）
嵌套集left + right范围查询（快）范围查询（快）需重算大部分（很慢）
ruoyi 选择路径枚举的原因：部门的读操作远多于移动操作。用冗余的 ancestors 字段换取查询性能，是典型的"以空间换时间"策略。

2. ancestors 字段的设计


假设部门树结构：
  总公司 (dept_id=100, parent_id=0, ancestors="0")
    ├── 技术部 (dept_id=200, parent_id=100, ancestors="0,100")
    │   ├── 前端组 (dept_id=300, parent_id=200, ancestors="0,100,200")
    │   └── 后端组 (dept_id=301, parent_id=200, ancestors="0,100,200")
    └── 市场部 (dept_id=400, parent_id=100, ancestors="0,100")

ancestors = "0,100,200" 的含义：
  → 该部门的祖先依次是：根(0) → 总公司(100) → 技术部(200)
  → 最后一位是直接父级
  → 新增时：ancestors = 父部门的 ancestors + "," + 父部门的 dept_id

3. 核心操作实现

3.1 新增部门 — 自动计算 ancestors

// SysDeptServiceImpl.insertDept()
public int insertDept(SysDeptBo bo) {
    SysDept info = deptMapper.selectById(bo.getParentId());
    if (ObjectUtil.isNull(info)) {
        throw new ServiceException("父部门不存在");
    }
    if (!SystemConstants.NORMAL.equals(info.getStatus())) {
        throw new ServiceException("部门停用，不允许新增");  // 父部门停用则禁止新增
    }
    SysDept dept = MapstructUtils.convert(bo, SysDept.class);
    // 🔑 核心：拼接祖先路径
    dept.setAncestors(info.getAncestors() + "," + dept.getParentId());
    return deptMapper.insert(dept);
}

3.2 查询子部门 — 一条 SQL 搞定

// 查询 dept_id=200 的所有子部门（包括孙子、曾孙...）
// SQL: SELECT * FROM sys_dept WHERE FIND_IN_SET(200, ancestors)
deptMapper.lambda()
    .findInSet(deptId, SysDept::getAncestors)
    .list();

// 查询正常状态的子部门数量
public long selectNormalChildrenDeptById(Long deptId) {
    return deptMapper.lambda()
        .eq(SysDept::getStatus, SystemConstants.NORMAL)
        .findInSet(deptId, SysDept::getAncestors)
        .count();
}

3.3 移动部门 — 最复杂的操作

// SysDeptServiceImpl.updateDept() — 部门移动逻辑
public int updateDept(SysDeptBo bo) {
    SysDept dept = MapstructUtils.convert(bo, SysDept.class);
    SysDept oldDept = deptMapper.selectById(dept.getDeptId());

    if (!oldDept.getParentId().equals(dept.getParentId())) {
        // 🔑 父部门变了！需要更新自身及所有子孙的 ancestors

        // 1. 校验新父部门权限
        this.checkDeptDataScope(dept.getParentId());

        // 2. 获取新父部门的 ancestors
        SysDept newParent = deptMapper.selectById(dept.getParentId());
        String newAncestors = newParent.getAncestors() + "," + newParent.getDeptId();
        String oldAncestors = oldDept.getAncestors();

        // 3. 更新当前部门的 ancestors
        dept.setAncestors(newAncestors);

        // 4. 🔑 批量更新所有子孙节点的 ancestors
        updateDeptChildren(dept.getDeptId(), newAncestors, oldAncestors);
    }
    // ...
}

// updateDeptChildren — 替换所有子孙的祖先路径
private void updateDeptChildren(Long deptId, String newAncestors, String oldAncestors) {
    // 查出所有 ancestors 中包含当前部门ID的记录（即所有子孙）
    List<SysDept> children = deptMapper.lambda()
        .findInSet(deptId, SysDept::getAncestors)
        .list();

    // 将每个子孙的 ancestors 中的 oldAncestors 替换为 newAncestors
    for (SysDept child : children) {
        child.setAncestors(
            StringUtils.replaceOnce(child.getAncestors(), oldAncestors, newAncestors)
        );
    }
    deptMapper.updateBatchById(list);
}
移动部门的性能注意：如果部门层级很深、子节点很多，updateDeptChildren() 需要更新大量行的 ancestors 字段。但 ruoyi 的假设是部门层级通常不深（3-5层），子节点量级可控（几百个），因此这种方案是合理的。

4. 启用/停用的级联控制

// 启用部门时 — 自动启用所有上级部门
if (SystemConstants.NORMAL.equals(dept.getStatus())
    && StringUtils.isNotEmpty(dept.getAncestors())) {
    updateParentDeptStatusNormal(dept);  // 遍历 ancestors，逐一启用上级
}

// 停用部门时 — Controller 层有严格校验
if (SystemConstants.DISABLE.equals(dept.getStatus())) {
    if (deptService.selectNormalChildrenDeptById(deptId) > 0) {
        return R.fail("该部门包含未停用的子部门!");  // 必须先停用子部门
    }
    if (deptService.checkDeptExistUser(deptId)) {
        return R.fail("该部门下存在已分配用户，不能禁用!");  // 必须先移除用户
    }
}
// → 停用是严格的"自下而上"：必须先清空用户→停用子部门→停用父部门
// → 启用是宽松的"自下而上"：启用一个部门时自动启用所有上级

5. 树结构构建 — TreeBuildUtils

// 构建部门下拉树（支持多根节点）
TreeBuildUtils.buildMultiRoot(depts,
    SysDeptVo::getDeptId,       // ID 获取器
    SysDeptVo::getParentId,     // 父ID 获取器
    (node, treeNode) -> treeNode
        .setId(node.getDeptId())
        .setParentId(node.getParentId())
        .setName(node.getDeptName())
        .setWeight(node.getOrderNum())
        .putExtra("disabled", DISABLE.equals(node.getStatus()))
);

// 构建菜单树（单根节点，根节点ID=0）
TreeBuildUtils.build(menus, Constants.TOP_PARENT_ID,

        menu.setChildren(nodeTreeMaps.getOrDefault(menu.getMenuId(), emptyList()));
    });
