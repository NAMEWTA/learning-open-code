标准启动 vs 委托启动

// 标准 Spring Boot 启动（ruoyi-admin 采用）
@SpringBootApplication
public class RuoYiApplication {
    public static void main(String[] args) {
        SpringApplication.run(RuoYiApplication.class, args);
    }
}

// 委托启动（snailai-server 采用）
@SpringBootApplication
public class SnailAiServerApplication {
    public static void main(String[] args) {
        SnailAiSpringbootApplication.main(args);  // 委托给 starter
    }
}

@SpringBootApplication 在此的真实作用

不是"启动"，而是"标记"。
@SpringBootApplication = @SpringBootConfiguration + @EnableAutoConfiguration + @ComponentScan。
当 SnailAiSpringbootApplication.main() 内部调用 SpringApplication.run() 时，以 SnailAiServerApplication 为起点进行组件扫描——保证 server 模块自己的 Bean（SecurityConfig、ActuatorAuthFilter）能被 Spring 容器发现。

委托的三层动机

动机效果
Server 退化为"部署载体"只提供 main() + yml 配置 + 安全过滤器 + 日志配置。不关心 Spring Boot 启动细节
Starter 控制启动过程自动配置顺序、组件扫描策略、条件装配逻辑、gRPC 服务启动时序全部由 starter 管理
统一启动行为所有 Snail AI 实例共享同一套启动逻辑，升级 starter 时无需修改 server 代码

包边界 = 职责边界

类包路径所属方职责
SnailAiServerApplicationorg.dromara.snailaiRuoYi（本仓库）声明"部署时需要什么"
SnailAiSpringbootApplicationcom.aizuda.snail.ai.starteraizuda（外部 JAR）定义"运行时如何启动"

Server 自己启动 vs 委托启动 对比

维度自己启动委托启动
启动代码SpringApplication.run(SnailAiServerApplication.class, args)SnailAiSpringbootApplication.main(args)
Server 角色"平台构建者"——需理解全部启动机制"部署载体"——只需 main 作为 JVM 入口
需手动配置组件扫描路径、自动配置导入、容器设置...无需——starter 内部已配置好
升级影响starter 升级可能需修改 server 代码完全透明

关键代码位置

文件路径
启动类src/main/java/org/dromara/snailai/SnailAiServerApplication.java
实际启动器（starter 内部）com.aizuda.snail.ai.starter.SnailAiSpringbootApplication（JAR 内，本仓库不可见）
常见误解：「委托启动」不等于「放弃控制」。Server 仍通过 @SpringBootApplication 控制组件扫描起点，确保自己的 Bean 被发现。委托的是启动过程的执行权，不是配置的声明权。
