一、两层安全设计

1. SecurityConfig —— 配置层（40 行）

@Configuration
public class SecurityConfig {

    @Value("${spring.boot.admin.client.username}")
    private String username;
    @Value("${spring.boot.admin.client.password}")
    private String password;

    @Bean
    public FilterRegistrationBean<ActuatorAuthFilter> actuatorFilterRegistrationBean() {
        FilterRegistrationBean<ActuatorAuthFilter> registrationBean = new FilterRegistrationBean<>();
        registrationBean.setFilter(new ActuatorAuthFilter(username, password));
        registrationBean.addUrlPatterns("/actuator", "/actuator/*");
        return registrationBean;
    }
}
设计点实现意图
凭据复用@Value("${spring.boot.admin.client.username}")一套凭据两处使用：Admin 注册 + Actuator 保护
最小拦截范围addUrlPatterns("/actuator", "/actuator/*")仅拦截运维端点，不影响管理后台静态页面
构造器注入new ActuatorAuthFilter(username, password)过滤器无状态，不需要 Spring 容器依赖

2. ActuatorAuthFilter —— 过滤器层（92 行）

public void doFilter(ServletRequest servletRequest, ServletResponse servletResponse,
                     FilterChain filterChain) throws IOException, ServletException {
    HttpServletRequest request = (HttpServletRequest) servletRequest;
    HttpServletResponse response = (HttpServletResponse) servletResponse;

    // 第 1 层：Authorization 头存在且以 "Basic " 开头
    String authHeader = request.getHeader("Authorization");
    if (authHeader == null || !authHeader.startsWith("Basic ")) {
        response.setHeader("WWW-Authenticate", "Basic realm=\"realm\"");
        response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Unauthorized");
        return;
    }

    // 第 2 层：Base64 解码 + 格式校验 "username:password"
    String base64Credentials = authHeader.substring("Basic ".length());
    byte[] credDecoded = Base64.getDecoder().decode(base64Credentials);
    String credentials = new String(credDecoded, StandardCharsets.UTF_8);
    String[] split = credentials.split(":");
    if (split.length != 2) {
        response.setHeader("WWW-Authenticate", "Basic realm=\"realm\"");
        response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Unauthorized");
        return;
    }

    // 第 3 层：凭据比对
    if (!username.equals(split[0]) || !password.equals(split[1])) {
        response.setHeader("WWW-Authenticate", "Basic realm=\"realm\"");
        response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Unauthorized");
        return;
    }

    filterChain.doFilter(request, response);  // 放行
}

三层校验对应三种攻击

校验层防御的攻击失败响应
Authorization 头检查无认证信息的直接访问401 + WWW-Authenticate
Base64 解码 + 格式检查Authorization 头格式错误401
凭据比对错误的用户名/密码401

为什么不用 Sa-Token / Spring Security？

独立部署 = 独立认证。snailai-server 不依赖 ruoyi-common-satoken，没有登录态和会话上下文。它被设计为可被任何系统通过 gRPC 调用的通用 AI 平台。对仅需保护 Actuator 端点的场景，92 行的 Servlet Filter 比引入 Spring Security 更轻量、更合适。

二、监控集成：Spring Boot Admin 客户端

spring.boot.admin.client:
  enabled: true
  url: http://localhost:9090/admin          # → ruoyi-monitor-admin
  instance:
    service-host-type: IP                   # 容器环境用 IP 而非主机名
    metadata:
      username: ${spring.boot.admin.client.username}
      userpassword: ${spring.boot.admin.client.password}
  username: @monitor.username@              # Maven 构建时替换
  password: @monitor.password@
凭据闭环：metadata 中的 username/userpassword 就是 Actuator 的认证凭据。Admin Server 用它们访问 snailai-server 的 Actuator 端点获取监控数据。一套凭据，两端闭环使用。

           -XX:+HeapDumpOnOutOfMemoryError -XX:+UseZGC ${JAVA_OPTS} \
           -jar app.jar

JVM 参数速查

参数含义为何重要
-XX:+UseZGC启用 ZGC 垃圾回收器亚毫秒级暂停——AI 对话对延迟敏感，避免 GC 抖动
-XX:+HeapDumpOnOutOfMemoryErrorOOM 时自动生成堆转储事后分析内存泄漏
-Djava.security.egd=file:/dev/./urandom非阻塞随机数生成器加速 SSL/TLS 握手和 AES 加密操作
${JAVA_OPTS}环境变量注入点运行时通过 -e JAVA_OPTS="-Xmx2g" 追加参数
bellsoft/liberica-openjdk-rocky:21.0.11-cdsBellSoft JDK 21 + CDSCDS 预存档类元数据，加速容器冷启动

GC 策略对比

GC暂停时间适用场景
G1（默认）通常 < 200ms通用服务端应用
ZGC< 1ms低延迟——AI 推理、实时对话
Shenandoah< 1ms低延迟（Red Hat 生态）

四、日志策略：四级 Appender 管道

logback-plus.xml:
├── console (ConsoleAppender)
│   └── 实时输出到标准输出（容器日志）—— 按级别分色
├── file_console (RollingFileAppender, 同步)
│   ├── 文件：console.log，≥INFO 全量
│   ├── 保留 1 天
│   └── 用途：Actuator logfile 端点数据源 / 短期排障
├── async_info (AsyncAppender → file_info)
│   ├── 文件：info.log，仅 INFO 级别
│   ├── 保留 60 天
│   ├── queueSize=1024, discardingThreshold=100
│   └── 用途：业务审计、对话记录追溯
└── async_error (AsyncAppender → file_error)
    ├── 文件：error.log，仅 ERROR 级别
    ├── 保留 60 天
    ├── queueSize=1024, discardingThreshold=100
    └── 用途：告警接入、故障排查
为什么 INFO 和 ERROR 用异步写入？同步日志每次 log.info() 触发磁盘 IO——高并发 AI 对话场景中可能成为性能瓶颈。异步 Appender 将日志写入内存队列，由独立线程批量刷盘。队列满到 discardingThreshold: 100 时丢弃低级别日志。

权衡：file_console 是同步写入，确保 Actuator logfile 端点总能看到最新日志（实时性）。async_info/error 使用异步写入，追求吞吐量（性能）。实时性 vs 吞吐量的经典取舍。
