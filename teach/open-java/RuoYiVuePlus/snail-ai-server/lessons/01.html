第 1 课 · Snail AI Server 全景：装配式 AI 平台的独立运行时 全景地图

🎯 本节目标（对应使命 Success #1、#5）：学完你能一句话讲清 ruoyi-snailai-server 的本质，能画出它在 RuoYi-Vue-Plus 整体架构中的位置，理解「装配式平台」与「客户端适配器」的本质区别。

在 RuoYi-Vue-Plus 的 ruoyi-extend 目录下，并列着三个独立部署的服务：ruoyi-monitor-admin（监控中心）、ruoyi-snailjob-server（任务调度）、ruoyi-snailai-server（AI 平台）。它们有一个共同特征：都是独立进程，通过自己的 main() 启动，拥有独立的端口和配置。

但 ruoyi-snailai-server 有一个让人困惑的特征：它只有 3 个 Java 文件，却声称承载了一个完整的 AI 智能体平台。这种反差正是我们学习的起点——它的核心能力不是「写出来的」，而是「装配出来的」。

一、物理文件结构：3 个 Java 文件 + 3 个配置 + Dockerfile

ruoyi-extend/ruoyi-snailai-server/ ├── pom.xml ← 仅 2 个依赖 ├── Dockerfile ← ZGC + 双端口暴露 └── src/main/ ├── java/ │ ├── org/dromara/snailai/ │ │ └── SnailAiServerApplication.java ← 14 行，启动委托 │ └── com/aizuda/snail/ai/starter/filter/ │ ├── SecurityConfig.java ← 40 行，Actuator 安全 │ └── ActuatorAuthFilter.java ← 92 行，HTTP Basic Auth └── resources/ ├── application.yml ← 70 行，核心配置资产 ├── application-dev.yml ← 28 行，开发环境 ├── application-prod.yml ← 28 行，生产环境 ├── logback-plus.xml ← 87 行，日志策略 └── banner.txt ← 启动横幅

关键数字：3 个 Java 文件（总约 146 行）、4 个配置文件（总约 213 行）。配置文件行数远超 Java 代码行数——这就是「装配式平台」的特征：代码定义骨架，配置注入血肉。

二、一句话本质：它是 Snail AI 平台的「部署载体」

ruoyi-snailai-server 是 Snail AI 平台的独立运行时——只写启动入口和安全配置，AI 能力（模型调用、RAG 知识库、Agent 编排、gRPC 通信、记忆管理）由 snail-ai-starter 通过 Spring Boot 依赖注入与配置声明装配合成。

对比两个易混淆的概念：

模块定位代码量AI 能力来源部署方式
ruoyi-ai（业务模块）Snail AI 的客户端适配器1 个 Java 文件Snail AI SDK → gRPC 调用 Server嵌入 ruoyi-admin 进程中
ruoyi-snailai-server（扩展模块）Snail AI 的平台本体运行时3 个 Java 文件snail-ai-starter 直接内嵌独立进程，端口 8900
💡 理解这个区别的最好方式：
ruoyi-ai 是「让你在 RuoYi 后台里用 AI」——它是一个适配器，帮你把 RuoYi 用户翻译给 Snail AI。
ruoyi-snailai-server 是「让 AI 平台跑起来」——它是平台本身，负责真正调度模型、管理知识库、执行 Agent。
两者通过 gRPC（端口 18888） 通信，就像客户端调用服务器。

三、在 RuoYi-Vue-Plus 整体架构中的位置

🖥️ 前端层
React: @ant-design/x SSE 流式 · Vue: iframe 嵌入
↓ HTTP REST
⚙️ ruoyi-admin（含 ruoyi-ai 适配器）
SnailAiController: 用户身份桥接 → 注册到 Snail AI Server
↓ gRPC（端口 18888）
🖧 ruoyi-snailai-server（本模块）
独立进程 · HTTP 8900 · snail-ai-starter 装配 AI 能力
↓ JDBC / HTTP
🗄️ MySQL + 文件存储（LOCAL / MinIO）
短期记忆持久化 · 资源文件存储 · Skill 脚本存储

每一层的通信协议与职责边界：

1前端 → ruoyi-admin：HTTP REST。前端调 /snail-ai/user/register 注册用户，然后通过 Snail AI SDK 提供的端点进行 AI 对话（SSE 流式）。
协议：HTTP + SSE
2ruoyi-admin → snailai-server：gRPC。ruoyi-ai 适配器通过 Snail AI SDK 封装的 gRPC 客户端，调用 snailai-server 的 AI 能力。
协议：gRPC，端口 18888
3snailai-server → 基础设施：JDBC + 文件 IO。短期记忆的 db 模式走 MySQL；资源文件走本地磁盘或 MinIO 对象存储。
协议：JDBC / HTTP（MinIO）
⚠️ 注意：snailai-server 的 HTTP 端口 8900 不直接服务前端！
端口 8900 提供的是 Snail AI 管理后台的静态页面（classpath:admin/）和 Actuator 运维端点。前端用户的 AI 对话请求走的是 ruoyi-admin（HTTP）→ snailai-server（gRPC 18888）这条链路，而不是直接访问 8900。

四、pom.xml 的极简依赖哲学

<dependencies>
    <!-- Snail AI 服务端 -->
    <dependency>
        <groupId>com.aizuda</groupId>
        <artifactId>snail-ai-starter</artifactId>
    </dependency>

    <!-- Spring Boot Admin 客户端 -->
    <dependency>
        <groupId>de.codecentric</groupId>
        <artifactId>spring-boot-admin-starter-client</artifactId>
    </dependency>
</dependencies>

仅 2 个直接依赖。 没有 ruoyi-common-core、没有 ruoyi-common-security、没有 ruoyi-common-web。这意味着：

设计决策含义
不依赖任何 ruoyi-common 模块snailai-server 是一个完全独立的应用，不共享 RuoYi 的认证体系（Sa-Token）、Web 基础设施、异常处理框架
依赖 snail-ai-starter所有 AI 能力（模型、RAG、Agent、gRPC、记忆）由 starter 的自动配置引入，server 模块不需要知道内部细节
依赖 spring-boot-admin-starter-client作为监控客户端注册到 ruoyi-monitor-admin，接受集中管控——这是 extend 模块的标配
💡 设计洞察：为什么 server 模块不依赖 ruoyi-common？
这是一个关键的架构决策。snailai-server 是 Snail AI 平台的运行时，它应该独立于 RuoYi 的业务体系。它有自己的认证体系（ActuatorAuthFilter），有自己的数据库连接（dev/prod yml 中独立配置），有自己的日志策略（logback-plus.xml）。这种独立性让它可以被任何系统（不仅是 RuoYi）通过 gRPC 调用。

五、检验一下（提取练习）

Q1. ruoyi-snailai-server 的本质是什么？

A. 一个自研的 AI 算法引擎 B. Snail AI 平台的独立运行时——通过 snail-ai-starter 依赖注入和配置声明，将 AI 能力装配合成为一个可独立部署的 Spring Boot 应用 C. 一个 RuoYi 的业务模块，提供 AI 相关的 REST API D. 一个前端管理后台的静态文件服务器
看答案B。snailai-server 只写启动入口和安全配置，AI 能力全部由 starter 装配。它是「部署载体」而非「能力实现」。

Q2. snailai-server 和 ruoyi-ai 之间通过什么协议通信？

A. HTTP REST B. WebSocket C. gRPC（端口 18888） D. 数据库共享
看答案C。ruoyi-ai 适配器通过 Snail AI SDK 的 gRPC 客户端，调用 snailai-server 的 18888 端口。HTTP 8900 端口只提供管理后台静态页面和 Actuator 端点。

Q3. 为什么 snailai-server 的 pom.xml 不依赖任何 ruoyi-common 模块？

A. 作者忘记加了 B. 它是一个完全独立的应用——有自己的认证、数据库连接、日志策略，不应绑定 RuoYi 的业务体系 C. ruoyi-common 的依赖由 snail-ai-starter 传递引入 D. 它不需要数据库和日志
看答案B。snailai-server 被设计为独立于 RuoYi 的 AI 平台运行时——任何系统都可以通过 gRPC 调用它。这种独立性让它不依赖 RuoYi 的 Sa-Token 认证、Web 基础设施等。
