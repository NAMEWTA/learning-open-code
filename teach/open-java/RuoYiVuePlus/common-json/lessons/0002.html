第 2 课 · 序列化处理器 handler 区

🎯 本节目标（对应使命 Success #2）：学完你能解释 BigNumberSerializer 如何通过安全整数判断保护 Long 精度传给 JS 前端，以及两个自定义反序列化器如何借助 Hutool 实现多日期格式自动识别。

第 1 课我们看了全景。这节课聚焦 handler/ 包下的三个处理器——它们是 JacksonConfig.registerJavaTimeModule() 里注册的「零件」。

一、BigNumberSerializer：一个让前后端 "不再吵架" 的精度保护器

这是整个模块里最「小而美」的类。先看完整代码（共 54 行）：

@JacksonStdImpl
public class BigNumberSerializer extends NumberSerializer {

    public static final BigNumberSerializer INSTANCE =
        new BigNumberSerializer(Number.class);

    private static final long MAX_SAFE_INTEGER = 9007199254740991L;   // JS Number.MAX_SAFE_INTEGER
    private static final long MIN_SAFE_INTEGER = -9007199254740991L;  // JS Number.MIN_SAFE_INTEGER

    public BigNumberSerializer(Class<? extends Number> rawType) {
        super(rawType);
    }

    @Override
    public void serialize(Number value, JsonGenerator gen,
                          SerializationContext provider) {
        if (value.longValue() >= MIN_SAFE_INTEGER
            && value.longValue() <= MAX_SAFE_INTEGER) {
            super.serialize(value, gen, provider);   // 安全范围：正常输出数字
        } else {
            gen.writeString(value.toString());        // 超出范围：输出字符串
        }
    }
}
它解决什么问题？
JavaScript 的 Number 类型是 IEEE 754 双精度浮点数，整数安全范围只有 [-2^53+1, 2^53-1]，即 ±9007199254740991。Java 的 Long 最大是 9223372036854775807（19 位），远超 JS 的安全上限。雪花算法生成的 ID（如 1234567890123456789）传到前端会末尾精度丢失（变成 1234567890123456700），导致「前端拿到的 ID 和后端对不上」。

BigNumberSerializer 的做法非常优雅——不是一刀切把所有 Long 都转字符串，而是在安全范围内保持数字类型（方便前端做数值运算），超出范围的才转字符串。这是一条兼顾性能与兼容的折中线。

在 JacksonConfig 中注册的三个目标类型

module.addSerializer(Long.class, BigNumberSerializer.INSTANCE);
module.addSerializer(Long.TYPE, BigNumberSerializer.INSTANCE);   // long 基本类型
module.addSerializer(BigInteger.class, BigNumberSerializer.INSTANCE);
module.addSerializer(BigDecimal.class, ToStringSerializer.instance);
💡 注意 BigDecimal 用的是 Jackson 自带的 ToStringSerializer.instance，而不是自定义的 BigNumberSerializer。因为 BigDecimal 是小数，不存在「整数精度丢失」问题，直接用 toString() 保证精度即可。而 Long.TYPE 是 long.class（基本类型），需要额外注册——Jackson 对基本类型和包装类型是分开处理的。

@JacksonStdImpl 注解

这是 Jackson 的内部标记注解，表示这个序列化器是「标准实现」，Jackson 内部会做优化（如写入缓存跳过某些检查）。在 Jackson 3.x 中位置从 com.fasterxml.jackson.databind.annotation 移到了 tools.jackson.databind.annotation，功能不变。

二、CustomDateDeserializer：不管你传什么日期格式，我都能解析

再看第二个处理器，分析它的巧妙设计：

public class CustomDateDeserializer extends ValueDeserializer<Date> {

    @Override
    public Date deserialize(JsonParser p, DeserializationContext ctxt) {
        String text = p.getString();
        if (text == null || text.isBlank()) {
            return null;
        }
        DateTime parse = DateUtil.parse(text.trim());
        return parse.toJdkDate();
    }
}
巧在哪里？
它没有硬编码日期格式（如 new SimpleDateFormat("yyyy-MM-dd HH:mm:ss").parse(text)），而是委托给 Hutool 的 DateUtil.parse()。Hutool 的 parse 内部维护了一个格式优先级列表：先尝试常见格式（yyyy-MM-dd HH:mm:ss、yyyy/MM/dd HH:mm:ss、yyyyMMddHHmmss 等），再尝试时间戳（纯数字），最后用正则匹配 ISO 格式。这意味着前端传 "2026-07-05"、"2026/07/05 14:30:00"、1751706600000 都能正确解析。
💡 空值处理：if (text == null || text.isBlank()) return null;——注意用的是 isBlank()（Java 11+），连纯空格字符串也会返回 null。这不是「日期解析失败返回 null」，而是「没有值就返回 null」，把「是否允许为空」的控制权交给调用方（通常由 @NotNull / @NotBlank 注解控制）。

三、CustomLocalDateTimeDeserializer：和 Date 版几乎一样，差别只在一行

public class CustomLocalDateTimeDeserializer
        extends ValueDeserializer<LocalDateTime> {

    @Override
    public LocalDateTime deserialize(JsonParser p,
                                      DeserializationContext ctxt) {
        String text = p.getString();
        if (text == null || text.isBlank()) {
            return null;
        }
        DateTime parse = DateUtil.parse(text.trim());
        return parse.toLocalDateTime();  // ← 和 Date 版唯一的区别
    }
}

看到没？和 CustomDateDeserializer 一模一样——同样是空值检查 → 去空格 → Hutool parse → 转换。唯一的区别是最后一行：一个调 toJdkDate() 返回 java.util.Date，一个调 toLocalDateTime() 返回 java.time.LocalDateTime。

为什么需要两个？因为 Jackson 对 Date 和 LocalDateTime 是分开注册反序列化器的——你不能用一个 ValueDeserializer<Date> 去处理 LocalDateTime 字段。但两者确实共享同一套「多格式自动识别」的能力——于是把这个能力封装在 Hutool 的 DateUtil.parse() 里，两个反序列化器各做一层薄薄的适配。这是「薄适配器」模式的典型运用。

四、别忘了：序列化端也配了 LocalDateTime

在 JacksonConfig.registerJavaTimeModule() 中还有一行序列化器的注册：

DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
module.addSerializer(LocalDateTime.class, new LocalDateTimeSerializer(formatter));

这保证输出时 LocalDateTime 固定格式化为 yyyy-MM-dd HH:mm:ss（而不是 Jackson 默认的数组格式 [2026,7,5,14,30,0]）。所以这个模块对时间类型的处理是「输入端宽容（多格式自动识别）、输出端严格（固定格式）」的经典策略。

五、检验一下（提取练习）

Q1. BigNumberSerializer 如何判断一个 Long 该输出为数字还是字符串？

A. 所有 Long 都转字符串，因为 JS 的 Number 永远不安全 B. 所有 Long 都不转，精度丢失是前端的问题 C. 在 JS 安全整数范围 [MIN_SAFE_INTEGER, MAX_SAFE_INTEGER] 内输出数字，超出范围输出字符串 D. 只有雪花 ID 才转字符串，普通 Long 不转
看答案C。通过比较 value.longValue() 与 MAX_SAFE_INTEGER / MIN_SAFE_INTEGER 判断是否在 JS 安全范围内。范围内的保持数字类型（便于前端运算），超出范围的转字符串（保护精度）。

Q2. 为什么 BigDecimal 用 ToStringSerializer 而不是自定义的 BigNumberSerializer？

A. Jackson 不支持 BigDecimal 的自定义序列化器 B. BigDecimal 是小数，不存在整数精度丢失问题，直接用 toString() 保证精度 C. BigNumberSerializer 处理不了 BigDecimal 的 scale 属性 D. 作者忘记了，应该统一用 BigNumberSerializer
看答案B。BigDecimal 涉及小数精度而非整数溢出，用 ToStringSerializer 直接输出字符串是最安全的方式（避免前端浮点数精度问题）。BigNumberSerializer 继承 NumberSerializer，关注的是整数安全范围。

Q3. CustomDateDeserializer 如何做到支持多种日期格式？

A. 用正则表达式穷举了 20 种常见日期格式逐一尝试 B. 委托给 Hutool 的 DateUtil.parse()，它内部有按优先级排列的格式列表自动匹配 C. 只支持 yyyy-MM-dd HH:mm:ss 一种格式 D. 先在注解上读 @DateTimeFormat，按注解指定的格式解析
看答案B。它把多格式解析的能力外包给了 Hutool——DateUtil.parse() 内部按优先级尝试常见格式、时间戳、ISO 格式等。反序列化器本身只做薄适配。

Q4. 为什么需要同时注册 Long.class 和 Long.TYPE 两个序列化器？

A. 代码冗余，只注册 Long.class 就够了 B. Long.class 是包装类型、Long.TYPE 是基本类型 long.class，Jackson 对它们是分开处理的 C. Long.TYPE 是内部标识，不影响实际序列化 D. 为了兼容 Java 8 和 Java 17 的差异
看答案B。Jackson 的序列化器注册是按类型精确匹配的——Long.class 对应 Long 包装类型，Long.TYPE（即 long.class）对应基本类型。只注册一个，另一个不会生效。
