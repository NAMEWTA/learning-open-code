第 4 课 · JSON 增强框架（下）引擎实现

🎯 本节目标（对应使命 Success #4）：学完你能讲清 JsonValueEnhancer 的完整处理链路——对象树递归遍历、属性元数据缓存 (ConcurrentHashMap)、循环引用保护 (IdentityHashMap)、简单值短路、处理后二次增强，以及消息转换器排除逻辑。

第 3 课我们讲了「接口」——三阶段的设计模式。这节课讲「引擎」——JsonValueEnhancer 怎么把这些接口驱动起来。它是 enhance 包里最长的一个类（约 370 行），但结构极其规整。

一、先看入口：enhance() 方法的主流程

public Object enhance(Object body) {
    // ① 快速短路：null、已处理过的 JsonNode、没有处理器
    if (body == null || body instanceof JsonNode || processors.isEmpty()) {
        return body;
    }
    // ② 创建上下文 + collect 扫描
    JsonEnhancementContext context = new JsonEnhancementContext(jsonMapper);
    collectValue(body, context, new IdentityHashMap<>());

    // ③ 短路：没有任何字段需要处理
    if (!context.isProcessingRequired()) {
        return body;
    }
    // ④ prepare 批量预处理
    processors.forEach(processor -> processor.prepare(context));

    // ⑤ render 渲染
    return renderValue(body, context, new IdentityHashMap<>());
}
请求响应对象 body │ ├─ null / JsonNode / 无处理器？ → 原样返回（零开销） │ ├─ [collectValue] 递归扫描对象树 │ └─ 每个字段调 collectField → processor.supports() & collect() │ └─ supports() 返回 true → markProcessingRequired() │ ├─ processingRequired == false？ → 原样返回（零开销） │ ├─ [prepare] 一次调用所有 processor.prepare() │ └─ [renderValue] 递归渲染增强后的 JSON 树 └─ 每个字段调 processor.process() 获取替换值

二、递归遍历：collectValue() 的类型分发

collectValue() 是整个引擎最核心的递归方法。它使用 Java 17 的 switch 模式匹配做类型分发：

private void collectValue(Object value, JsonEnhancementContext context,
                          IdentityHashMap<Object, Boolean> visited) {
    switch (value) {
        case null -> { return; }                              // null 直接跳过
        case Map<?, ?> map -> {                              // Map → 遍历 values
            map.values().forEach(child -> collectValue(child, context, visited));
            return;
        }
        case Iterable<?> iterable -> {                       // 集合 → 遍历元素
            iterable.forEach(child -> collectValue(child, context, visited));
            return;
        }
        default -> {}                                         // 继续向下
    }
    if (value.getClass().isArray()) {                         // 数组 → 遍历元素
        int length = Array.getLength(value);
        for (int i = 0; i < length; i++)
            collectValue(Array.get(value, i), context, visited);
        return;
    }
    if (isSimpleValue(value.getClass())                       // 简单值 → 跳过
        || visited.put(value, Boolean.TRUE) != null) {        // 已访问 → 跳过（循环引用保护）
        return;
    }
    try {
        for (PropertyMetadata metadata : getProperties(value.getClass())) {
            Object propertyValue = metadata.getValue(value);
            JsonFieldContext fieldContext = new JsonFieldContext(
                value, metadata.propertyName(), metadata.member(), propertyValue);
            collectField(fieldContext, context);               // ← 核心：通知所有 Processor
            collectValue(propertyValue, context, visited);     // ← 递归深入
        }
    } finally {
        visited.remove(value);                                 // 退栈时清理，允许不同路径重新访问
    }
}
为什么用 IdentityHashMap 而不是 HashSet？
IdentityHashMap 使用引用相等 (==) 而非 equals() 来判断 key 是否重复。如果两个不同的对象恰好 equals() 返回 true（如两个内容相同的 Map），用 HashSet 会误判为已访问而跳过。而循环引用保护真正关心的是「同一个内存对象是否已被处理」，所以必须用引用相等。另外，IdentityHashMap.put() 返回的是旧值——如果返回非 null 说明之前已 put 过，即循环引用。

visited.remove(value) 在 finally 中的微妙之处

注意 visited 的 remove 放在 finally 块里——保证递归返回后一定清理。而且用的是「入栈 put / 出栈 remove」模式（而非「全局记录所有访问过的对象」），这意味着同一个对象可以在树的不同分支路径中分别被访问——只要不是在同一条路径上自我引用（真正的循环引用），就不会被阻止。

三、属性元数据缓存：getProperties() + ConcurrentHashMap

每个类型的属性元数据只解析一次，之后从缓存取：

private final Map<Class<?>, List<PropertyMetadata>> propertyCache
    = new ConcurrentHashMap<>();

private List<PropertyMetadata> getProperties(Class<?> type) {
    return propertyCache.computeIfAbsent(type, this::resolveProperties);
}

resolveProperties() 使用 Jackson 的 ClassIntrospector 内省类型，找出所有可序列化属性（getter + field，优先 getter）。这个操作涉及反射和注解解析，比较昂贵——但每个 Class 只在第一次遇到时解析一次，之后直接从 ConcurrentHashMap 取。

💡 为什么用 ConcurrentHashMap 而不是普通的 HashMap？虽然 enhance() 整体在单次请求中是单线程执行的，但 propertyCache 是所有请求共享的。多个请求并发首次遇到同一个 Class 时可能同时触发 computeIfAbsent——ConcurrentHashMap 保证线程安全。

resolveProperties 的细节

private List<PropertyMetadata> resolveProperties(Class<?> type) {
    if (isSimpleValue(type) || type.isArray() ||
        Map.class.isAssignableFrom(type) ||
        Iterable.class.isAssignableFrom(type)) {
        return Collections.emptyList();  // 简单类型和容器类型跳过
    }
    JavaType javaType = jsonMapper.constructType(type);
    SerializationConfig config = jsonMapper.serializationConfig();
    ClassIntrospector classIntrospector =
        config.classIntrospectorInstance().forOperation(config);
    AnnotatedClass annotatedClass =
        classIntrospector.introspectClassAnnotations(javaType);
    List<BeanPropertyDefinition> definitions =
        classIntrospector.introspectForSerialization(javaType, annotatedClass)
                       .findProperties();
    // 收集 getter 和 field，优先 getter
    List<PropertyMetadata> properties = new ArrayList<>();
    for (BeanPropertyDefinition definition : definitions) {
        AnnotatedMember member = definition.getAccessor();   // getter
        if (member == null) member = definition.getField();  // 回退 field
        if (member == null) continue;
        member.fixAccess(true);                              // 强制可访问（包括 private）
        properties.add(new PropertyMetadata(definition.getName(), member));
    }
    return Collections.unmodifiableList(properties);
}

使用 Jackson 的 ClassIntrospector 而非纯 Java 反射的原因：它尊重 Jackson 的所有注解规则（@JsonProperty、@JsonIgnore、@JsonUnwrapped 等）——保证增强框架扫描到的属性列表和最终序列化输出的属性列表完全一致。

四、渲染阶段：renderValue() 的类型分发 + process 处理链

renderValue() 的结构与 collectValue() 几乎镜像：同样的类型分发（null/Map/Iterable/Array/简单值/POJO），但这次是「构建 JSON 节点」而非「收集信息」。关键看 renderPojo()：

private ObjectNode renderPojo(Object value, JsonEnhancementContext context,
                               IdentityHashMap<Object, Boolean> visited) {
    ObjectNode objectNode = jsonMapper.createObjectNode();
    for (PropertyMetadata metadata : getProperties(value.getClass())) {
        Object originalValue = metadata.getValue(value);
        JsonFieldContext fieldContext = new JsonFieldContext(
            value, metadata.propertyName(), metadata.member(), originalValue);
        Object processedValue = originalValue;
        boolean changed = false;

        // ★ 处理链：依次调用每个 Processor 的 process()
        for (JsonFieldProcessor processor : processors) {
            if (!processor.supports(fieldContext)) continue;
            Object nextValue = processor.process(fieldContext, processedValue, context);
            changed = changed || !Objects.equals(processedValue, nextValue);
            processedValue = nextValue;
        }

        JsonNode childNode = changed
            ? enhanceTranslatedValue(processedValue, context, visited)  // 有变化→二次增强
            : renderValue(processedValue, context, visited);            // 无变化→常规渲染
        objectNode.set(metadata.propertyName(), childNode);
    }
    return objectNode;
}
注意：processingRequired 标记和 supports() 返回值的不同作用。
• processingRequired 是请求级别标记：整个响应只要有一个字段被至少一个 Processor 关注，就要走完整流程。
• supports() 是字段-Processor 级别过滤：每个字段、每个 Processor 单独判断。即使 processingRequired 是 true，一个 Processor 也只为它 supports 的字段执行 process()。

五、二次增强：enhanceTranslatedValue() 的递归处理

这是增强框架里最微妙的一步。当 Processor.process() 返回了一个非简单值的全新对象（比如翻译后返回了一个包含嵌套字段的 VO），这个新对象也可能需要增强：

private JsonNode enhanceTranslatedValue(Object value,
        JsonEnhancementContext context,
        IdentityHashMap<Object, Boolean> visited) {
    if (value == null || value instanceof JsonNode || isSimpleValue(value.getClass())) {
        return renderValue(value, context, visited);   // 简单值：直接渲染
    }
    return enhanceTree(value);                          // 复杂对象：递归增强
}

private JsonNode enhanceTree(Object value) {
    JsonEnhancementContext context = new JsonEnhancementContext(jsonMapper);
    collectValue(value, context, new IdentityHashMap<>());
    if (!context.isProcessingRequired()) {
        return jsonMapper.valueToTree(value);
    }
    processors.forEach(processor -> processor.prepare(context));
    return renderValue(value, context, new IdentityHashMap<>());
}
为什么需要二次增强？假设 Processor A 把 deptId=3 替换成了 DeptVO{name="研发部", leaderId=5}。这个 leaderId=5 可能也需要被另一个 Processor B 处理（如翻译成 leader 姓名）。二次增强让「替换后的对象」重新走一遍 collect-prepare-process 流程——递归确保嵌套替换也不会遗漏。

六、消息转换器排除：supports() 方法

JsonValueEnhancer 还有一个 supports(Class<?> converterType) 方法，被外部的 ResponseBodyAdvice 调用：

public boolean supports(Class<?> converterType) {
    return !processors.isEmpty()
        && !ByteArrayHttpMessageConverter.class.isAssignableFrom(converterType)
        && !StringHttpMessageConverter.class.isAssignableFrom(converterType)
        && !ResourceHttpMessageConverter.class.isAssignableFrom(converterType);
}

三种被排除的转换器：ByteArray（文件下载，二进制数据不是 JSON 对象树）、String（已经是字符串了，二次序列化会加引号）、Resource（静态资源，不是业务数据）。这个排除是正确性保障而非性能优化——如果不排除，增强框架试图遍历 byte[] 或 String 会出错。

八、检验一下（提取练习）

Q1. collectValue() 中为什么用 IdentityHashMap 而非 HashSet 来记录已访问对象？

A. IdentityHashMap 性能更好，遍历速度是 HashSet 的两倍 B. 循环引用保护关心的是「同一个内存对象」，需要用引用相等（==）而非 equals() C. HashSet 不支持 null key，而域对象可能为 null D. 个人偏好，用 HashSet 也一样
看答案B。循环引用保护需要判断「同一个对象是否在访问链上」，只能用引用相等（==）。如果用 equals()，两个内容相同的不同对象（如两个相等的 Map）会被误判为已访问。

Q2. propertyCache 为什么用 ConcurrentHashMap 而非 HashMap？

A. 增强框架对性能要求极高，ConcurrentHashMap 更快 B. propertyCache 是所有请求共享的，多线程并发可能同时 computeIfAbsent 同一个 Class C. JSON 序列化本身就是多线程的 D. 可以为性能优化用，其实 HashMap 也行
看答案B。虽然单次请求内是单线程，但多个请求并发时可能同时首次遇到同一个 Class，触发 computeIfAbsent 的并发执行。ConcurrentHashMap 保证这个操作线程安全。

Q3. enhanceTranslatedValue 的作用是什么？

A. 为翻译后的文本添加引号 B. 对 Processor.process() 返回的复杂新对象重走一遍 collect-prepare-process 流程 C. 校验翻译结果是否合法 D. 将翻译结果缓存到 Redis
看答案B。Processor 替换后返回的可能是一个包含嵌套字段的新 VO，这个新对象的字段也可能需要增强——二次增强递归确保嵌套替换不遗漏。

Q4. supports() 方法为什么排除 ByteArrayHttpMessageConverter？

A. 性能考虑，文件下载不需要增强 B. 文件下载返回的是 byte[] 而非对象树，增强框架无法遍历——排除它是正确性保障 C. ByteArray 类型的序列化比 String 慢 D. 作者偏好，其实不排除也不会出错
看答案B。ByteArray 是二进制数据而非对象树，增强框架试图遍历 byte[] 会出错。排除它是正确性保障，而非性能优化。
