第 3 课 · JSON 增强框架（上）设计模式

🎯 本节目标（对应使命 Success #3）：学完你能讲清 JSON 增强框架的三阶段设计模式（collect → prepare → process），以及 JsonEnhancementContext 作为数据总线、JsonFieldContext 作为字段快照、JsonFieldProcessor 作为扩展点的各自职责与接口设计。

前两课我们看了配置与处理器。这节课进入本模块最精巧的设计部分——JSON 增强框架（enhance 包下的 4 个类）。它的核心思想是：在 HTTP 响应写出 JSON 之前，对对象树的每个字段做一次「加工流水线」——翻译、脱敏、类型转换、补充附加数据……全都可以通过实现 JsonFieldProcessor 来扩展。

一、问题场景：为什么要"增强"？

假设你有一个 API 返回用户列表，其中 deptId 是数字 3。前端希望直接显示「研发部」而不想再调一个接口查部门名。传统做法：在 Controller 里手动查部门 Map 塞进去，或者给 VO 加 @JsonIgnore 的原始字段 + getter 里查库——前者臃肿，后者可能触发 N+1 查询。

JSON 增强框架的思路：写一个 DeptNameProcessor 实现 JsonFieldProcessor，让框架在序列化时自动把 deptId 替换成部门名。你既不需要改 Controller，也不需要改 VO——完全是非侵入式的。

二、三个核心类的关系

JsonFieldProcessor（接口 · 扩展点） │ │ 实现 ──→ DeptNameProcessor、SensitiveDataProcessor、...（你的业务实现） │ ├── collect(fieldContext, context) 收集阶段：记录需要处理的字段 ├── prepare(context) 预处理阶段：批量查询，消除 N+1 └── process(fieldContext, value, context) 处理阶段：返回替换后的值 JsonEnhancementContext（数据总线） ← 三个阶段共享的"黑板" ├── attributes: LinkedHashMap<String, Object> 存储任意数据 ├── processingRequired: boolean 是否需要增强标记 └── jsonMapper: JsonMapper 回引 Mapper 以便二次序列化 JsonFieldContext（字段快照） ← 描述"正在处理哪个字段" └── record(owner, propertyName, member, value) └── getAnnotation(Class) 从 member 读取注解

三、JsonFieldContext — 用 Java Record 做字段快照

public record JsonFieldContext(
    Object owner,            // 字段所属的对象实例（如 User 对象）
    String propertyName,     // 字段名称（如 "deptId"）
    AnnotatedMember member,  // Jackson 的字段元数据（getter 或 field）
    Object value             // 字段当前值（如 3）
) {
    public <A extends Annotation> A getAnnotation(Class<A> annotationType) {
        return member == null ? null : member.getAnnotation(annotationType);
    }
}
为什么是 Record？Java 14+ 的 Record 天然不可变、自带 equals/hashCode/toString、构造函数简洁。JsonFieldContext 是「某一刻的字段快照」，创建后不应被修改——Record 的不可变性正好匹配这个语义。而且它只有一个自定义方法 getAnnotation()，其余全是自动生成，代码量极少。

getAnnotation() 是关键设计：Processor 通过它判断"这个字段我管不管"。比如一个 @MaskSensitive 注解打在字段上，脱敏 Processor 就通过 fieldContext.getAnnotation(MaskSensitive.class) != null 来判断是否需要处理。

四、JsonFieldProcessor — 三阶段的接口契约

这是整个增强框架的灵魂接口。Javadoc 写得非常清晰，我用中文提炼一下三阶段的职责：

public interface JsonFieldProcessor {

    // ① 判定：这个字段归我管吗？
    default boolean supports(JsonFieldContext fieldContext) { return true; }

    // ② 收集阶段（collect）：每个字段调一次
    //    扫出需要处理的 key，存进 context 供 prepare 阶段用
    default void collect(JsonFieldContext fieldContext,
                         JsonEnhancementContext context) {}

    // ③ 预处理阶段（prepare）：所有字段 collect 完后，只调一次
    //    执行批量 IO（如一次性查所有部门名），结果写入 context
    default void prepare(JsonEnhancementContext context) {}

    // ④ 处理阶段（process）：渲染 JSON 树时，每个字段调一次
    //    从 context 取出 prepare 的结果，返回替换后的值
    default Object process(JsonFieldContext fieldContext,
                           Object value,
                           JsonEnhancementContext context) {
        return value;  // 默认：原样返回，不做修改
    }
}
为什么需要分成三个阶段，而不是在 process 里直接查数据库？
如果你在 process 里对每个字段都调一次 deptMapper.selectById(deptId)，当响应里包含 100 个 User 对象时，就会产生 100 次数据库查询——这就是经典的 N+1 问题。

三阶段分离的本质是：collect 收集所有 ID → prepare 一次性批量查询 → process 直接从内存 Map 取值。100 条数据只需要 1 次数据库查询。

supports() 的设计

默认返回 true（兼容无注解驱动的处理器）。你的 Processor 应该重写它来快速排除不关心的字段：

// 示例：只处理打了 @Translate 注解的字段
@Override
public boolean supports(JsonFieldContext fieldContext) {
    return fieldContext.getAnnotation(Translate.class) != null;
}

如果 supports() 返回 false，后续的 collect/process 都不会被调用——这是一个性能优化点，避免无关字段进处理链。

collect() 的典型实现模式

// 示例：翻译处理器的 collect 阶段——收集需要翻译的 key
@Override
public void collect(JsonFieldContext fieldContext, JsonEnhancementContext context) {
    Translate annotation = fieldContext.getAnnotation(Translate.class);
    String dictType = annotation.type();                    // 如 "sys_dept"
    String key = String.valueOf(fieldContext.value());      // 如 "3"

    // 把 key 存进 "按 dictType 分组" 的集合
    Set<String> keys = context.getOrCreateAttribute(
        "translate:" + dictType,
        LinkedHashSet::new);
    keys.add(key);
}

prepare() 的典型实现模式

// 示例：翻译处理器的 prepare 阶段——批量查询
@Override
public void prepare(JsonEnhancementContext context) {
    for (String attrKey : context.attributes.keySet()) {
        if (!attrKey.startsWith("translate:")) continue;
        String dictType = attrKey.substring("translate:".length());
        Set<String> keys = context.getAttribute(attrKey);
        // 一次性批量查询
        Map<String, String> nameMap = dictService.batchGetNames(dictType, keys);
        // 结果写回 context，供 process 阶段使用
        context.setAttribute("result:" + dictType, nameMap);
    }
}

process() 的典型实现模式

// 示例：翻译处理器的 process 阶段——从 context 查结果替换值
@Override
public Object process(JsonFieldContext fieldContext, Object value,
                       JsonEnhancementContext context) {
    Translate annotation = fieldContext.getAnnotation(Translate.class);
    String dictType = annotation.type();
    Map<String, String> nameMap = context.getAttribute("result:" + dictType);
    if (nameMap == null) return value;  // 没查到，保留原值
    String key = String.valueOf(fieldContext.value());
    return nameMap.getOrDefault(key, String.valueOf(value));
}
attribute key 命名约定：接口 Javadoc 建议「以实现类全限定名作为 attribute key 前缀以避免冲突」。比如 com.example.DeptNameProcessor 的 key 用 "com.example.DeptNameProcessor:collect:keys" 而非裸的 "keys"。多个 Processor 同时运行时，上下文共享同一个 attributes Map——没有前缀隔离会互相覆盖。

五、JsonEnhancementContext — 三个阶段的数据总线

@Getter
public class JsonEnhancementContext {
    private final JsonMapper jsonMapper;
    private final Map<String, Object> attributes = new LinkedHashMap<>();
    private boolean processingRequired;      // ← 关键标记

    public <T> T getAttribute(String key) { ... }
    public <T> T getOrCreateAttribute(String key, Supplier<T> supplier) { ... }
    public void setAttribute(String key, Object value) { ... }
    public void markProcessingRequired() { this.processingRequired = true; }
}

几个值得注意的设计细节：

设计点具体做法为什么
属性表用 LinkedHashMapnew LinkedHashMap<>()保证遍历顺序与插入顺序一致——process 阶段依赖它按属性 key 遍历
getOrCreateAttributecomputeIfAbsent惰性创建 + 线程安全（虽然 context 是请求级别，单线程操作）
processingRequired 标记初始 false，collect 阶段有匹配字段时设 true短路优化：如果整个响应对象里没有任何字段需要增强，直接跳过后续的 prepare + render，原样返回
泛型 getAttribute(T) attributes.get(key)调用方不需要强转（但类型安全由调用方自己保证）
contains / remove 方法完整的 CRUD允许 Processor 清理自己产生的临时数据
processingRequired 的短路逻辑是整个增强框架的性能防线。假设一个 API 返回纯数字列表 [1, 2, 3]，没有任何字段被 Processor 关注。没有这个标记，框架仍然会调用 prepare() 再走一遍 render()——虽然没有副作用，但浪费 CPU。有了这个标记，collect 完后发现 processingRequired=false，直接 return body，零开销。

六、检验一下（提取练习）

Q1. JSON 增强框架分成 collect → prepare → process 三个阶段的核心目的是什么？

A. 让代码更规范，分为三段式结构是 Java 最佳实践 B. 把「收集 key」和「批量查询」分开，消除 N+1 查询问题 C. Jackson 的序列化流程要求分成这三个阶段 D. 提高并发性能，三个阶段可以并行执行
看答案B。核心目的是避免在 process 阶段逐个字段查数据库。collect 收集所有 key → prepare 批量查询一次 → process 从内存取值。这是解决「序列化时的 N+1 问题」的经典设计。

Q2. JsonFieldContext 为什么使用 Java Record？

A. Record 比普通类快 10 倍 B. 它是「字段快照」，创建后不应修改——Record 的不可变性天然匹配这个语义 C. 因为 Jackson 只支持 Record D. 作者的个人偏好，用普通类也一样
看答案B。JsonFieldContext 是某一时刻的字段快照，创建后不应被修改。Record 的不可变性、自动生成 equals/hashCode/toString、简洁语法完全匹配这个语义。

Q3. processingRequired 标记的作用是什么？

A. 记录当前正在处理哪个字段，便于调试 B. 标识整个增强过程是否完成 C. 短路优化——如果 collect 后发现没有任何字段需要增强，跳过后续的 prepare + render，原样返回对象 D. 记录是否有字段处理失败
看答案C。collect 阶段遍历对象树时，只要有一个 Processor 的 supports() 返回 true，就会调用 markProcessingRequired()。collect 完后如果仍是 false，说明这个响应不需要任何增强，直接原样返回。

Q4. 在 collect 阶段，一个 Processor 应该用 context 的什么属性来避免和其他 Processor 冲突？

A. 直接用字段名作为 key，不会有冲突 B. 用全限定类名作为 attribute key 前缀 C. 用 ThreadLocal 存储，绕过 context D. 冲突没关系，Map 会自动去重
看答案B。接口 Javadoc 明确建议用实现类全限定名作为前缀。因为多个 Processor 共享同一个 attributes Map，没有前缀隔离会导致 key 互相覆盖。
