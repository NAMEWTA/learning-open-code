第 1 课 · JSON 模块全景：三驾马车 入门地图

🎯 本节目标（对应使命 Success #1）：学完你能用一张图讲清模块的 三大入口（JacksonConfig 序列化配置、JsonEnhancementConfig 增强入口、JsonUtils 工具门面）各管什么、谁依赖谁，并说出每部分的子组件。这是后面每节细节课挂靠的骨架。

很多人对 RuoYi 的 JSON 模块印象模糊：好像有个 JsonUtils 能转换对象，好像有个 BigNumberSerializer 处理 Long 精度……但说不清它整体是什么。这节课把整个 ruoyi-common-json 摊开，一次看全。全程基于本仓库真实代码。

一、先记住一句话：它是「Jackson 的业务定制层」

整个模块只做一件事：基于 Jackson (tools.jackson 3.x)，把序列化/反序列化/字段增强/JSON校验封装成 RuoYi 业务代码能直接拿来用的能力。看 pom.xml 的依赖就懂它的原材料：

ruoyi-common-core        ← 基础工具（SpringUtils、StringUtils 等）
jackson-databind (tools)  ← 核心：Jackson 3.x re-shade（com.fasterxml → tools.jackson）
jackson-datatype-jsr310   ← Java 8 时间类型支持（LocalDateTime 等）
hutool-core               ← 日期解析（DateUtil.parse 自动识别格式）
jakarta.validation-api    ← Bean Validation（@JsonPattern 注解依赖）
设计动机 · 为什么是 tools.jackson 而不是 com.fasterxml.jackson？
RuoYi-Vue-Plus 通过 Maven Shade 插件把 Jackson 3.x 重打包到 tools.jackson 包名下。这样做是为了隔离版本冲突：项目可能同时依赖了某个还在用 Jackson 2.x 的老库，如果不做 shade，两个版本在 classpath 上打架。重打包后二者各用各的包名，井水不犯河水。功能上完全等价于标准 Jackson。

二、全景图：5 大区域

org.dromara.common.json │ ├── config/ 【配置区】启动时装配，定义全局规则 │ ├── JacksonConfig ← 序列化/反序列化处理器注册中心 │ └── JsonEnhancementConfig ← 字段增强器装配入口 │ ├── handler/ 【处理器区】自定义序列化/反序列化逻辑 │ ├── BigNumberSerializer ← Long/BigInteger 精度保护（超 JS 安全范围转 string） │ ├── CustomDateDeserializer ← Date 反序列化（Hutool 多格式自动识别） │ └── CustomLocalDateTimeDeserializer ← LocalDateTime 反序列化 │ ├── enhance/ 【增强框架区】响应出站前对字段做翻译/脱敏等加工 │ ├── JsonEnhancementContext ← 单次增强的数据总线（属性表 + 状态标记） │ ├── JsonFieldContext ← 字段元数据 Record（owner + name + member + value） │ ├── JsonFieldProcessor ← 处理器接口（collect → prepare → process） │ └── JsonValueEnhancer ← 增强引擎（递归扫描 + 批量预处理 + 渲染） │ ├── validate/ 【校验区】Bean Validation 注解方式校验 JSON │ ├── JsonType ← 枚举：OBJECT / ARRAY / ANY │ ├── JsonPattern ← @Constraint 注解 │ └── JsonPatternValidator ← 校验器实现 │ └── utils/ 【工具区】业务最常调用的 JSON 工具 └── JsonUtils ← 序列化/反序列化/格式校验 10 个公开方法

这 5 块的关系可以这样理解：config 打地基 → handler 装处理器 → enhance 做出站加工 → validate 加校验 → utils 给业务开门。日常写业务你 90% 时间只碰 JsonUtils 和 @JsonPattern 注解；但要读懂「为什么能用」，得往下看到 config 和 enhance。

三、逐区域走一遍

① config · JacksonConfig — 序列化与反序列化的注册中心

config/JacksonConfig.java

它通过两个 @Bean 完成全局配置：

  • registerJavaTimeModule()：创建 SimpleModule，注册了 7 个处理器——3 个序列化器（BigNumberSerializer 管 Long/BigInteger、ToStringSerializer 管 BigDecimal、LocalDateTimeSerializer 管 LocalDateTime 格式化）和 2 个反序列化器（CustomDateDeserializer、CustomLocalDateTimeDeserializer）。
  • jsonInitCustomizer()：返回 JsonMapperBuilderCustomizer，设置默认时区为 TimeZone.getDefault()（系统时区）。
注意 @AutoConfiguration(before = JacksonAutoConfiguration.class)：这个 before 很关键——Spring Boot 的 JacksonAutoConfiguration 会自己创建 JsonMapper。如果 JacksonConfig 排在它后面，咱们注册的 Module 就不会生效。排在前面，等于「我先把我的一堆处理器加进去，你再创建 JsonMapper」。

② config · JsonEnhancementConfig — 字段增强的装配入口

config/JsonEnhancementConfig.java

这个类极其简洁：只有一个 Bean 方法接收 JsonMapper + List<JsonFieldProcessor>（Spring 自动注入所有 Processor 实现），创建 JsonValueEnhancer。它本身不做事，而是把各个 Processor 收集起来交给增强引擎。这就是插件式设计的装配点——你写一个新 Processor，只要打上 @Component，Spring 会自动注入到这个 List 里。

③ utils · JsonUtils — 业务最常用的工具门面

utils/JsonUtils.java

10 个公开方法，覆盖了业务中绝大多数 JSON 操作场景：

方法用途典型场景
toJsonString(Object)对象 → JSON 字符串记录日志、返回响应
parseObject(String, Class)JSON 字符串 → 对象解析配置、反序列化请求体
parseObject(byte[], Class)字节数组 → 对象解析 Redis 中的缓存数据
parseObject(String, TypeReference)复杂泛型反序列化List<SysUser> 这种带泛型的类型
parseMap(String)JSON → Dict（Hutool Map）解析配置 Map
parseArrayMap(String)JSON → List<Dict>解析配置列表
parseArray(String, Class)JSON 数组 → List<T>批量数据反序列化
toJsonStringExcludeFields(Object, String...)序列化并递归移除指定字段脱敏输出（如去掉 password）
isJson / isJsonObject / isJsonArrayJSON 格式校验参数校验、@JsonPattern 底层
为什么 JsonUtils 里的 JsonMapper 是 static final？因为它是 @NoArgsConstructor(access = PRIVATE) 的静态工具类，不能有实例字段注入。它用 SpringUtils.getBean(JsonMapper.class) 在类加载时主动从 Spring 容器拿。代价是：必须在 Spring 容器就绪后才能调用。

四、三驾马车的关系图

┌──────────────────────┐ │ JsonUtils（门面） │ │ 业务代码最常调用 │ └──────────┬───────────┘ │ 委托给全局 JsonMapper ▼ ┌─────────────────────────────────────────────────┐ │ JacksonConfig │ │ ┌───────────────────────────────────────────┐ │ │ │ SimpleModule: BigNumberSerializer, │ │ │ │ CustomDateDeserializer, │ │ │ │ CustomLocalDateTimeDeserializer, │ │ │ │ LocalDateTimeSerializer, ToStringSer │ │ │ └───────────────────────────────────────────┘ │ │ ┌───────────────────────────────────────────┐ │ │ │ JsonMapperBuilderCustomizer: │ │ │ │ defaultTimeZone, 日志输出 │ │ │ └───────────────────────────────────────────┘ │ └─────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────┐ │ JsonEnhancementConfig │ │ 装配 JsonValueEnhancer(JsonMapper, │ │ List<JsonFieldProcessor>) │ │ → 驱动增强链（collect → prepare → process） │ └─────────────────────────────────────────────────┘

三驾马车的关系：JacksonConfig 决定了「对象怎么序列化成 JSON、JSON 怎么反序列化成对象」（规则层）；JsonEnhancementConfig 决定了「响应出站前，JSON 可以做哪些额外加工——翻译、脱敏等」（加工层）；JsonUtils 是这两层的统一门面，业务代码不需要直接碰 Jackson 底层 API。

五、模块依赖关系

ruoyi-common-json 处于 RuoYi 公共模块的中层位置：

依赖方向模块关系

Q1. ruoyi-common-json 的基石库是什么？

A. Gson，Google 的 JSON 库 B. tools.jackson（Jackson 3.x re-shade），RuoYi 为避免版本冲突重打包的 Jackson C. fastjson，阿里的 JSON 库 D. Hutool JSON，国产工具库自带的 JSON 处理
看答案B。全模块使用 tools.jackson 包名的 Jackson 3.x，通过 Maven Shade 重打包以避免与其他依赖的 Jackson 版本冲突。

Q2. 模块的三驾马车是指哪三个类？

A. JsonUtils、JsonMapper、JsonNode B. JacksonConfig、JsonEnhancementConfig、JsonValueEnhancer C. JacksonConfig、JsonEnhancementConfig、JsonUtils D. BigNumberSerializer、CustomDateDeserializer、JsonPatternValidator
看答案C。JacksonConfig 管序列化规则、JsonEnhancementConfig 管字段增强装配、JsonUtils 管业务工具入口。JsonValueEnhancer 是增强引擎，属于增强框架内部的执行器而非入口。

Q3. JacksonConfig 上的 @AutoConfiguration(before = JacksonAutoConfiguration.class) 为什么重要？

A. 性能优化，排在前面加载更快 B. 保证自定义的 SimpleModule 在 Spring Boot 创建 JsonMapper 之前注册进去 C. 避免循环依赖 D. 可以省略，纯粹是代码风格偏好
看答案B。Spring Boot 的 JacksonAutoConfiguration 会自己创建 JsonMapper。如果排在它后面，自定义 Module 就不会生效。before 确保「先注册处理器，再创建 Mapper」。

Q4. 业务代码中想把一个对象转成 JSON 字符串，应该调用什么方法？

A. new ObjectMapper().writeValueAsString(obj) B. JsonUtils.toJsonString(obj) C. JSON.toJSONString(obj) D. new Gson().toJson(obj)
看答案B。JsonUtils 是模块提供的统一门面，底层委托给全局 JsonMapper。直接用 ObjectMapper 或其它库会绕过本模块的定制配置（如 BigNumberSerializer）。
