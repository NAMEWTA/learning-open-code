第 1 课 · Jackson 配置与序列化器

本节目标：学完能讲清 ruoyi-common-json 模块的「三驾马车」架构——JacksonConfig（序列化配置中心）、JsonEnhancementConfig（字段增强入口）、JsonUtils（业务工具门面）。理解 tools.jackson 的 re-shade 背景、JsonMapper 的不可变构造模式，以及 SimpleModule 中注册的 5 个序列化/反序列化器各自的设计动机与核心逻辑。

一、模块总览：它是 Jackson 3.x 的业务定制层

ruoyi-common-json 模块的定位非常清晰：它不是一个新的 JSON 框架，而是在 Jackson 3.x 的基础上，为 RuoYi 业务场景封装了序列化规则、字段增强、格式校验三大能力。整个模块只有 12 个 Java 文件，但覆盖了从基础序列化到响应后处理再到参数校验的完整链路。

从 pom.xml 看它的技术栈：

<!-- 核心模块 -->
ruoyi-common-core              → SpringUtils、StringUtils 等基础工具
<!-- Jackson 序列化 -->
spring-boot-starter-jackson    → Spring Boot 自动装配 Jackson（引入 tools.jackson）
<!-- 隐式依赖 -->
Hutool (DateUtil)              → 日期多格式解析
Jakarta Validation             → @JsonPattern 校验注解

模块的目录结构刚好按职责划分为 5 个包：

org/dromara/common/json/
├── config/        JacksonConfig（规则注册中心） + JsonEnhancementConfig（增强装配入口）
├── handler/       BigNumberSerializer、CustomDateDeserializer、CustomLocalDateTimeDeserializer
├── enhance/       增强框架四件套：Context、FieldContext、Processor、ValueEnhancer
├── validate/      @JsonPattern + JsonPatternValidator + JsonType
└── utils/         JsonUtils（业务统一门面，10 个公开方法）

二、为什么是 tools.jackson？——Maven Shade 隔离版本冲突

RuoYi-Vue-Plus 的 Jackson 不是标准包名 com.fasterxml.jackson，而是 tools.jackson。这不是笔误，而是一项关键的工程决策。

Maven Shade 插件的工作原理：在构建时将 Jackson 3.x 的所有 .class 文件从 com/fasterxml/jackson/ 路径重定位到 tools/jackson/ 路径，同时将所有 import 语句中的包名一并替换。最终产出的 JAR 中不包含任何 com.fasterxml 引用。

这样做的动机是版本冲突隔离。在一个大型 Spring Boot 项目中，不同的依赖库可能引入不同版本的 Jackson：有些老库还在用 Jackson 2.x（com.fasterxml.jackson），而 Spring Boot 3.x 默认已经迁移到 Jackson 3.x（也是 com.fasterxml.jackson）。如果不在包名层面做隔离，classpath 上会出现两个同名但版本不同的类，JVM 只会加载其中一个——另一个直接失效，导致运行时 ClassNotFoundException 或 MethodNotFoundException。

经过 Shade 重打包后：

  • RuoYi 自身的代码全部使用 tools.jackson 包名 —— 指向 Jackson 3.x 的能力
  • 第三方老库仍然使用 com.fasterxml.jackson 包名 —— 走它们自己的 Jackson 版本
  • 两个版本在 JVM 层面是完全不同的类，井水不犯河水
设计洞察：这种方案比「全局统一 Jackson 版本 + exclude 冲突传递依赖」更稳健。因为 exclude 只能解决依赖树上的冲突，但对「某些库直接把 Jackson 类 shade 进自己的 JAR」这种情况无能为力。而 re-shade 是在自己的地盘上彻底改名，从根本上避免了冲突。

三、三驾马车全景

模块有三个入口类，分别对应业务代码与 JSON 能力交互的三种场景：

入口类职责业务场景所在包
JacksonConfig序列化规则注册中心——定义全局的序列化/反序列化行为定义「Java 对象怎么变成 JSON」「JSON 怎么变回 Java 对象」config/
JsonEnhancementConfig字段增强装配入口——收集所有 JsonFieldProcessor 交给引擎定义「响应出站前，JSON 字段还能做什么额外加工」config/
JsonUtils业务工具门面——10 个静态方法覆盖常见 JSON 操作业务代码里「把对象转成 JSON 字符串」「把 JSON 字符串转成对象」utils/

三者关系：JacksonConfig 打地基（规则层）→ JsonEnhancementConfig 建加工链（增强层）→ JsonUtils 开门迎客（门面层）。业务代码 90% 的时间只碰 JsonUtils，但 JsonUtils 底层委托给 JacksonConfig 配置出来的全局 JsonMapper。

四、JsonMapper：Jackson 3.x 的不可变入口

在 Jackson 2.x 中，入口类是 ObjectMapper；在 Jackson 3.x 中，它改名为 JsonMapper。改名不只是换个标识符，而是反映了一个设计哲学的变化：

不可变（immutable）+ 建造者模式（builder pattern）：JsonMapper 实例一旦创建就不可修改。所有配置通过 JsonMapper.builder() 链式调用设置，最后 .build() 产出最终实例。这避免了 ObjectMapper 可变的 setter 方法带来的线程安全隐患。

在 RuoYi 中，全局 JsonMapper 由 Spring Boot 的 JacksonAutoConfiguration 自动创建。JacksonConfig 通过 @AutoConfiguration(before = JacksonAutoConfiguration.class) 确保自定义的 SimpleModule 在 JsonMapper 创建之前注册：

@AutoConfiguration(before = JacksonAutoConfiguration.class)
public class JacksonConfig {

    @Bean
    public SimpleModule registerJavaTimeModule() {
        SimpleModule module = new SimpleModule();
        // ... 注册 5 个序列化/反序列化处理器
        return module;
    }

    @Bean
    public JsonMapperBuilderCustomizer jsonInitCustomizer() {
        return builder -> {
            builder.defaultTimeZone(TimeZone.getDefault());
        };
    }
}

before = JacksonAutoConfiguration.class 是这段代码的关键。Spring Boot 的自动装配机制按顺序执行：先跑 JacksonConfig（注册 Module + 设置时区），再跑 JacksonAutoConfiguration（用已注册的 Module 构建 JsonMapper）。如果顺序反过来，Module 没有被注册进去，所有自定义序列化行为都不会生效。

五、SimpleModule 中的 5 个处理器

registerJavaTimeModule() 方法创建了一个 SimpleModule 并向其中注册了 5 个自定义处理器。SimpleModule 是 Jackson 的插件式扩展机制——你可以在不修改 Jackson 核心代码的前提下，插进去自己的序列化/反序列化逻辑。

5.1 BigNumberSerializer —— Long 精度保护

文件：org/dromara/common/json/handler/BigNumberSerializer.java

这是模块中最具业务价值的自定义序列化器。它解决的是一个经典的跨语言精度问题：

JavaScript 的 Number 类型基于 IEEE 754 双精度浮点数，安全整数范围是 [-9007199254740991, 9007199254740991]（即 Number.MIN_SAFE_INTEGER 到 Number.MAX_SAFE_INTEGER）。Java 的 Long 类型最大值是 9223372036854775807，远超 JS 的安全范围。当后端返回一个 19 位的 Long 给前端时，JS 会丢失最后几位的精度。

BigNumberSerializer 的设计完美应对这个问题：

public class BigNumberSerializer extends NumberSerializer {

    public static final BigNumberSerializer INSTANCE = new BigNumberSerializer(Number.class);
    private static final long MAX_SAFE_INTEGER = 9007199254740991L;
    private static final long MIN_SAFE_INTEGER = -9007199254740991L;

    @Override
    public void serialize(Number value, JsonGenerator gen, SerializationContext provider) {
        if (value.longValue() >= MIN_SAFE_INTEGER && value.longValue() <= MAX_SAFE_INTEGER) {
            super.serialize(value, gen, provider);  // 在安全范围内：JSON number
        } else {
            gen.writeString(value.toString());       // 超出安全范围：JSON string
        }
    }
}

核心逻辑很简单但精准：用 value.longValue() 获取数值后与两个安全边界比较。在范围内的正常写成 JSON 数字（如 12345）；超出范围的写成 JSON 字符串（如 "9223372036854775807"）。前端拿到字符串后可以用 BigInt 处理，不会丢失精度。

JacksonConfig 中注册了三类类型：

module.addSerializer(Long.class, BigNumberSerializer.INSTANCE);      // Long 包装类
module.addSerializer(Long.TYPE, BigNumberSerializer.INSTANCE);       // long 基本类型
module.addSerializer(BigInteger.class, BigNumberSerializer.INSTANCE); // BigInteger

注意 Long.TYPE 和 Long.class 在 Jackson 中是分开注册的——前者对应 long 基本类型字段，后者对应 Long 包装类型字段。两者都注册才能覆盖所有场景。

设计洞察：为什么不用「大于 Integer.MAX_VALUE 就转 String」的简单判断？因为业务中很多 Long 值（如数据库自增主键）虽然没有超过 Long 的理论范围，但远超 Integer 范围，却又在 JS 安全整数之内。用 Integer.MAX_VALUE 做阈值会误杀大量合法场景。用 JS 的安全整数边界才是精准的对症下药。

5.2 ToStringSerializer —— BigDecimal 直接转字符串

BigDecimal 的序列化没有使用自定义实现，而是直接复用 Jackson 内置的 ToStringSerializer：

module.addSerializer(BigDecimal.class, ToStringSerializer.instance);

这意味着任何 BigDecimal 值都会被序列化为 JSON 字符串而非数字。例如 new BigDecimal("123.45") 输出 "123.45" 而不是 123.45。

为什么不用自定义 Serializer？因为 BigDecimal 的核心问题不是数值范围（它没有 JS 精度问题——前端可以用 string 接收），而是数值精度。Java 中 new BigDecimal("0.1") 精确等于 0.1，但如果序列化成 JSON number 0.1，前端再转回时可能变成 0.10000000000000000555（浮点误差）。直接输出字符串彻底避免了这个问题。

5.3 CustomDateDeserializer —— 多格式日期自动识别

文件：org/dromara/common/json/handler/CustomDateDeserializer.java

后端接收前端传来的日期时，格式可能是各式各样的：2024-01-15、2024/01/15 14:30:00、1705312200000（毫秒时间戳）、1705312200（秒时间戳）、ISO 8601 格式……如果每换一个前端就得改后端解析逻辑，成本太高。

CustomDateDeserializer 把这件事完全委托给 Hutool 的 DateUtil.parse()：

public class CustomDateDeserializer extends ValueDeserializer<Date> {

    @Override
    public Date deserialize(JsonParser p, DeserializationContext ctxt) {
        String text = p.getString();
        if (text == null || text.isBlank()) {
            return null;                     // 空字符串 → null
        }
        DateTime parse = DateUtil.parse(text.trim());
        return parse.toJdkDate();            // Hutool 自动识别格式
    }
}

DateUtil.parse() 内部有一套格式试错机制：按优先级依次尝试常见的日期格式模板匹配，同时判断是否为纯数字（时间戳）。这比写一长串 @JsonFormat 注解或自定义多格式解析器简洁得多。

空字符串返回 null 是一个细节但重要的设计：前端可能传空字符串而非 null，Java 的 Date 字段不允许空字符串赋值，转成 null 避免了反序列化异常。

5.4 CustomLocalDateTimeDeserializer —— 同逻辑，不同返回类型

文件：org/dromara/common/json/handler/CustomLocalDateTimeDeserializer.java

与 CustomDateDeserializer 几乎完全相同，唯一的区别在最后一行：

// CustomDateDeserializer:
return parse.toJdkDate();

// CustomLocalDateTimeDeserializer:
return parse.toLocalDateTime();

两者都使用 Hutool 的 DateUtil.parse() 做格式识别，区别仅在于最终返回的 Java 类型。这是 DRY（Don't Repeat Yourself）原则下的合理设计——多格式解析的核心逻辑复用 Hutool，不需要自己再写一套。

5.5 LocalDateTimeSerializer —— 统一输出格式

序列化方向的 LocalDateTime 处理更简单——不需要智能识别，只需要统一输出格式：

DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
module.addSerializer(LocalDateTime.class, new LocalDateTimeSerializer(formatter));

所有 LocalDateTime 字段序列化时统一输出为 "2024-01-15 14:30:00" 格式。这是给前端一个确定性的契约：你不用猜测日期格式，返回的一定是这个样子。

处理器方向目标类型核心逻辑
BigNumberSerializer序列化Long / long / BigInteger超 JS 安全范围→string，范围内→number
ToStringSerializer序列化BigDecimal全部输出为 string，避免浮点精度丢失
LocalDateTimeSerializer序列化LocalDateTime统一格式化为 yyyy-MM-dd HH:mm:ss
CustomDateDeserializer反序列化DateHutool DateUtil.parse 多格式自动识别
CustomLocalDateTimeDeserializer反序列化LocalDateTime同上，转 LocalDateTime

六、JsonEnhancementConfig —— 字段增强的装配入口

文件：org/dromara/common/json/config/JsonEnhancementConfig.java

这个类极其简洁——只有一个 Bean 方法，但它是整个增强框架的插件装配点：

@AutoConfiguration
public class JsonEnhancementConfig {

    @Bean
    public JsonValueEnhancer jsonValueEnhancer(JsonMapper jsonMapper, List<JsonFieldProcessor> processors) {
        return new JsonValueEnhancer(jsonMapper, processors);
    }
}

Spring 的自动收集机制是关键：方法参数 List<JsonFieldProcessor> processors 不需要 @Autowired 注解。Spring 容器会自动扫描所有实现了 JsonFieldProcessor 接口的 Bean，收集成一个 List 注入进来。

这意味着：任何人都可以写一个新的 Processor 实现（比如自定义的脱敏处理器、翻译处理器），只要加上 @Component 注解，Spring 就会自动将其注入到 JsonValueEnhancer 的处理器列表中，无需修改任何配置类。这是典型的开闭原则（对扩展开放、对修改关闭）实践。

七、JsonUtils 门面 —— 业务代码的唯一入口

文件：org/dromara/common/json/utils/JsonUtils.java

JsonUtils 是一个静态工具类，通过 SpringUtils.getBean(JsonMapper.class) 获取全局 JsonMapper 单例：

@NoArgsConstructor(access = AccessLevel.PRIVATE)
public class JsonUtils {
    private static final JsonMapper JSON_MAPPER = SpringUtils.getBean(JsonMapper.class);
    // 10 个公开方法...
}

10 个方法覆盖了业务中最常见的 JSON 操作场景：

方法入参出参典型场景
toJsonStringObjectString对象转 JSON 字符串，记日志、返回响应
parseObjectString, ClassTJSON 字符串转对象
parseObjectbyte[], ClassT从 Redis 取出的字节数组转对象
parseObjectString, TypeReferenceT复杂泛型反序列化，如 List<SysUser>
parseMapStringDictJSON 对象转 Hutool Dict
parseArrayMapStringList<Dict>JSON 数组转 Dict 列表
parseArrayString, ClassList<T>JSON 数组转指定类型列表
toJsonStringExcludeFieldsObject, String...String序列化并移除敏感字段（如 password）
removeFieldsJsonNode, String...JsonNode递归删除 JSON 树中的指定字段
isJson/isJsonObject/isJsonArrayStringbooleanJSON 格式校验

注意 3 个重要的设计细节：

  • 空值处理策略不一致：parseObject(String, Class) 用 StringUtils.isEmpty 判空（空字符串也返回 null），但 parseObject(String, TypeReference) 用 StringUtils.isBlank 判空（空白字符串也返回 null）。这不是 bug——后者处理的通常是复杂泛型场景，对输入格式要求更严格。
  • parseArray 特殊处理：空字符串返回 new ArrayList<>() 而非 null（其它方法返回 null）。这是为了避免调用方做空值判断——业务代码通常期望「空输入 → 空列表」。
  • static 初始化的限制：因为 JsonMapper 通过 SpringUtils.getBean 获取，JsonUtils 必须在 Spring 容器就绪之后才能调用。在 static {} 块或 Spring 容器启动前调用会抛异常。
设计洞察：为什么 parseMap 返回 Dict 而不是 Map？Dict 是 Hutool 提供的扩展 Map，继承了 HashMap 并添加了类型安全的 getter（如 getStr()、getInt()、getBool()）。业务代码中解析配置 JSON 时，用 dict.getStr("key") 比 (String) map.get("key") 更安全也更简洁。这是 RuoYi 体系中广泛使用 Dict 的习惯延续。

检验环节

Q1. Jackson 3.x 的入口类叫什么？它与 Jackson 2.x 的 ObjectMapper 有什么核心区别？

看答案

Jackson 3.x 的入口类是 JsonMapper（替代了 2.x 的 ObjectMapper）。核心区别是 JsonMapper 是不可变（immutable）的，所有配置通过 JsonMapper.builder() 建造者模式设置，避免了 ObjectMapper 可变 setter 带来的线程安全问题。

Q2. BigNumberSerializer 对值为 12345L 和 9223372036854775807L 的 Long，分别序列化出什么？

看答案

12345L 在 [-9007199254740991, 9007199254740991] 范围内，输出 JSON number 12345。9223372036854775807L 超出范围，输出 JSON string "9223372036854775807"。

Q3. JacksonConfig 上 @AutoConfiguration(before = JacksonAutoConfiguration.class) 如果删掉会怎样？

看答案

Spring Boot 的 JacksonAutoConfiguration 会先创建 JsonMapper，然后才执行 JacksonConfig 注册 SimpleModule。结果是注册的 Module 不会生效——BigNumberSerializer、CustomDateDeserializer 等全部不会应用到 JsonMapper 上。所有序列化走 Jackson 默认行为。
