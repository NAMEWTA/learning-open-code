一、@JsonPattern — 一个标准的 Bean Validation 注解

@Documented
@Target({ElementType.METHOD, ElementType.FIELD})
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = JsonPatternValidator.class)   // ← 指定校验器
public @interface JsonPattern {

    JsonType type() default JsonType.ANY;               // 期望的 JSON 类型

    String message() default "不是有效的 JSON 格式";      // 校验失败消息

    Class<?>[] groups() default {};                     // Bean Validation 分组

    Class<? extends Payload>[] payload() default {};     // 负载信息
}

整个注解除 type() 属性外，其余全是 Jakarta Bean Validation 的标准模板。任何一个 @Constraint 注解都有三个必选字段（message、groups、payload）——前者的值最终进入 ConstraintViolation.getMessage()，后者用于分组校验和携带元信息。

type 的默认值为 ANY：表示「只要是合法的 JSON 对象或数组就行」。如果你的字段只应存储 JSON 对象（如 {"name":"张三"}），就显式指定 type = JsonType.OBJECT；如果只应存储数组（如 [1,2,3]），就指定 type = JsonType.ARRAY。

二、JsonType 枚举 — 驱动校验策略的三个值

@Getter
@AllArgsConstructor
public enum JsonType {
    OBJECT,   // JSON 对象，如 {"a":1}
    ARRAY,    // JSON 数组，如 [1,2,3]
    ANY       // 任意 JSON 类型，对象或数组都允许
}

枚举非常简洁，但它是校验策略的关键驱动因素——JsonPatternValidator 根据它的值走不同的校验分支。

三、JsonPatternValidator — 把注解、枚举、JsonUtils 串联起来

public class JsonPatternValidator
        implements ConstraintValidator<JsonPattern, String> {

    private JsonType jsonType;

    @Override
    public void initialize(JsonPattern annotation) {
        this.jsonType = annotation.type();     // ① 从注解读取类型约束
    }

    @Override
    public boolean isValid(String value, ConstraintValidatorContext context) {
        if (StringUtils.isBlank(value)) {
            return true;                       // ② 空值放行，交给 @NotBlank 控制
        }
        return switch (jsonType) {             // ③ 根据枚举分支选择校验方法
            case ANY    -> JsonUtils.isJson(value);
            case OBJECT -> JsonUtils.isJsonObject(value);
            case ARRAY  -> JsonUtils.isJsonArray(value);
        };
    }
}

完整链路：@JsonPattern 注解决定 jsonType → initialize() 读取 → isValid() 根据 jsonType 走不同分支 → 各分支委托给 JsonUtils 的对应校验方法。

💡 注意 isValid 对空值的处理：StringUtils.isBlank(value) 直接返回 true（校验通过）。这不是「空字符串是合法 JSON」，而是把「是否为空的判断权」交还给 @NotBlank / @NotNull。如果你需要「必填且是合法 JSON」，应该同时加 @NotBlank 和 @JsonPattern。

四、JsonUtils 中的 JSON 格式校验方法

校验器底层依赖的三个方法：

// ① 统用校验：是合法 JSON 就行（对象或数组）
public static boolean isJson(String str) {
    return readTreeQuietly(str) != null;
}

// ② 精确校验：必须是 JSON 对象 {}
public static boolean isJsonObject(String str) {
    JsonNode node = readTreeQuietly(str);
    return node != null && node.isObject();
}

// ③ 精确校验：必须是 JSON 数组 []
public static boolean isJsonArray(String str) {
    if (StringUtils.isBlank(str)) return false;
    JsonNode node = readTreeQuietly(str);
    return node != null && node.isArray();
}

// 底层共用：安静解析，失败返回 null
private static JsonNode readTreeQuietly(String str) {
    if (StringUtils.isBlank(str)) return null;
    try {
        return JSON_MAPPER.readTree(str);
    } catch (Exception e) {
        return null;
    }
}
为什么 readTreeQuietly 吞掉异常？因为校验方法的语义是「判断是否合法」而不是「报错」。调用方只关心 true/false。如果用 try-catch 在外面包，校验逻辑会变啰嗦。readTreeQuietly 把异常转换成 null，调用方只需判断 null。

五、实战：如何在业务中使用 @JsonPattern

// 示例 1：字段要求是合法的 JSON（不限制类型）
@JsonPattern(message = "扩展字段必须是合法的 JSON 格式")
@NotBlank(message = "扩展字段不能为空")
private String extraData;   // 用户可以存 {} 或 []

// 示例 2：字段必须是 JSON 对象
@JsonPattern(type = JsonType.OBJECT, message = "配置信息必须是 JSON 对象格式")
private String configData;  // 只允许 {"key":"value"}

// 示例 3：字段必须是 JSON 数组
@JsonPattern(type = JsonType.ARRAY, message = "标签列表必须是 JSON 数组格式")
private String tags;        // 只允许 ["tag1","tag2"]
陷阱提醒：@JsonPattern 只校验格式合法性，不校验 JSON 内容的业务语义。比如字段要求 {"name":"张三","age":25}，但用户传了 {"foo":"bar"}——这在格式上完全合法，校验会通过。如果需要对 JSON 内容做 schema 校验，需要用 JSON Schema 或自定义 Validator。

我们从 5 个区域、13 个 Java 文件中提取一个「设计故事」：

📦 ruoyi-common-json = Jackson 的业务定制层 【第一层 · 基础规则】 config/ JacksonConfig ├─ 注册 SimpleModule（7 个序列化/反序列化处理器） └─ 设置 JsonMapper 默认时区 JsonEnhancementConfig └─ 装配 JsonValueEnhancer(JsonMapper, List<JsonFieldProcessor>) 【第二层 · 数据格式】 handler/ BigNumberSerializer ← 保护 Long 精度传给 JS 前端 CustomDateDeserializer ← Date 多格式自动识别 CustomLocalDateTimeDeserializer ← LocalDateTime 多格式自动识别 【第三层 · 出站加工】 enhance/ JsonEnhancementContext ← 数据总线：共享属性表 + 状态标记 JsonFieldContext ← 字段快照 Record：owner + name + member + value JsonFieldProcessor ← 扩展点接口：collect → prepare → process JsonValueEnhancer ← 引擎：递归扫描 + 批量预处理 + 渲染 【第四层 · 入站校验】 validate/ @JsonPattern ← Bean Validation 注解 JsonType ← 枚举：OBJECT / ARRAY / ANY JsonPatternValidator ← 校验实现：注解 → 枚举 → JsonUtils 【第五层 · 统一门面】 utils/ JsonUtils ← 10 个公开方法：序列化/反序列化/校验

九、最佳实践清单

  1. 永远用 JsonUtils 而非直接 new ObjectMapper——否则 BigNumberSerializer 等定制处理器不会生效，Long 精度丢失问题会重现。
  2. @JsonPattern 要配合 @NotBlank 使用——前者对空值放行，空值校验交给后者。
  3. 写 JsonFieldProcessor 时用全限定类名做 attribute key 前缀——避免多个 Processor 在 context 里互相覆盖。
  4. 复杂泛型反序列化用 TypeReference 而非 Class——parseObject(json, List.class) 会丢失泛型信息，反序列化出来是 List<LinkedHashMap> 而非 List<User>。
  5. toJsonStringExcludeFields 是递归删除——它不仅从顶层对象删字段，嵌套对象中的同名字段也会被删。注意这一点以避免误删。
  6. 增强框架的 process() 中不要做 IO——把 IO 放在 prepare() 里，process 只从 context 取缓存结果。违反这一点会导致 N+1 问题。

十、检验一下（提取练习）

Q1. @JsonPattern 校验注解的完整链路是？

A. @JsonPattern → JsonPatternValidator.initialize() 读 type → isValid() 根据 JsonType 枚举走不同分支 → 各分支委托 JsonUtils 的方法 B. @JsonPattern → Jackson 在序列化时自动校验 C. @JsonPattern → AOP 切面拦截 → JsonUtils 校验 D. @JsonPattern → Spring 启动时预检查
看答案A。这是标准的 Bean Validation 流程：注解上 @Constraint 指定校验器 → 初始化时读取配置 → 校验时根据枚举分支调用 JsonUtils。

Q2. 为什么 JsonPatternValidator.isValid() 对空值返回 true？

A. 空字符串本身就是合法 JSON B. 把「是否为空的判断权」交给 @NotBlank/@NotNull，校验器只负责「如果非空，格式是否合法」 C. Bean Validation 规范要求所有校验器都必须放行空值 D. 作者疏忽，应该对空值返回 false
看答案B。这是 Bean Validation 的最佳实践——每个 ConstraintValidator 只负责自己的校验维度。空值校验交给专门的 @NotBlank/@NotNull，避免职责混乱。

Q3. toJsonStringExcludeFields(obj, "password") 会删除哪些字段？

A. 只删除顶层对象的 password 字段 B. 递归删除整个对象树中所有名为 password 的字段（包括嵌套对象中的） C. 把 password 的值置为 null，字段名保留 D. 抛出异常，因为 password 可能是必传字段
看答案B。removeFields 是递归的——它对顶层是 ObjectNode 时删除字段，然后对每个子节点递归调用自身。嵌套对象中的同名字段也会被删除。

Q4. ruoyi-common-json 在 RuoYi 的模块依赖中处于什么位置？

A. 最顶层业务模块，被所有模块依赖 B. 公共层中层：向上依赖 common-core，向下被 redis/satoken/sms 等模块依赖 C. 独立模块，不依赖任何其他模块 D. 直接依赖所有业务模块
看答案B。它只向上依赖 common-core（基础工具），但被 redis、satoken、sms 等多个上层模块依赖——是典型的基础能力公共模块。
