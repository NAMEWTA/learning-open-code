第 3 课 · JsonUtils 工具类与校验

本节目标：学完能熟练选用 JsonUtils 的 10 个公开方法处理常见业务场景（序列化、反序列化、字段移除、格式校验），理解每个方法空值处理的差异与设计理由。能讲清 @JsonPattern 注解 + JsonPatternValidator + JsonType 枚举的完整校验链路，以及 isJson/isJsonObject/isJsonArray 三者的调用关系。

一、JsonUtils：业务代码唯一该调用的 JSON 入口

文件：org/dromara/common/json/utils/JsonUtils.java

JsonUtils 是模块中对业务代码暴露的唯一门面类。它通过 @NoArgsConstructor(access = PRIVATE) 禁止实例化，所有方法都是静态方法。

全局 JsonMapper 的获取方式值得注意：

@NoArgsConstructor(access = AccessLevel.PRIVATE)
public class JsonUtils {

    private static final JsonMapper JSON_MAPPER = SpringUtils.getBean(JsonMapper.class);

    public static JsonMapper getJsonMapper() {
        return JSON_MAPPER;
    }
    // ...
}

为什么用 SpringUtils.getBean 而不是 @Autowired？静态字段不支持 Spring 注入。JsonUtils 是纯静态工具类，没有实例，@Autowired 和 @PostConstruct 都用不了。折中方案是在类加载时主动从 Spring 容器拉取——代价是 JsonUtils 必须在 Spring 容器初始化完成后才能使用（类加载时机 < Container 就绪时机的时序约束）。

同时也暴露了 getJsonMapper() 方法——当 10 个封装方法不够用时，业务代码可以直接获取 JsonMapper 做更底层的操作。

二、10 个公开方法深度解析

2.1 toJsonString —— 序列化

public static String toJsonString(Object object) {
    if (ObjectUtil.isNull(object)) {
        return null;
    }
    return JSON_MAPPER.writeValueAsString(object);
}

最简单的序列化方法：对象 → JSON 字符串。空对象返回 null 字符串。底层 writeValueAsString() 会触发 JacksonConfig 中注册的所有序列化器（BigNumberSerializer、LocalDateTimeSerializer 等），以及 JSON 增强框架的增强链（如果对象是响应体的话）。

注意：null 对象返回的是 Java 的 null 字符串（String 类型），不是 JSON 的 "null" 字面量，也不是 JSON 的 null 空值。调用方需要判断返回值是否为 null。

2.2 parseObject —— 三个重载

反序列化是 JsonUtils 中最常用的操作类型，提供了三个重载：

重载 1：String + Class

public static <T> T parseObject(String text, Class<T> clazz) {
    if (StringUtils.isEmpty(text)) {
        return null;
    }
    return JSON_MAPPER.readValue(text, clazz);
}

isEmpty 判空：空字符串 "" 返回 null。适用于简单类型反序列化——如 parseObject(jsonStr, SysUser.class)。

重载 2：byte[] + Class

public static <T> T parseObject(byte[] bytes, Class<T> clazz) {
    if (ArrayUtil.isEmpty(bytes)) {
        return null;
    }
    return JSON_MAPPER.readValue(bytes, clazz);
}

专为 Redis 缓存场景设计：从 Redis 取出的数据是 byte[] 类型，不需要先转 String 再解析，直接一步到位。

重载 3：String + TypeReference

public static <T> T parseObject(String text, TypeReference<T> typeReference) {
    if (StringUtils.isBlank(text)) {
        return null;
    }
    return JSON_MAPPER.readValue(text, typeReference);
}

isBlank 判空：比 isEmpty 更严格——" "（纯空格）也返回 null。这是因为 TypeReference 通常用于复杂泛型场景，对输入格式要求更高。例如：

// 反序列化成 List<SysUser>
List<SysUser> users = JsonUtils.parseObject(jsonStr, new TypeReference<List<SysUser>>() {});

TypeReference 是 Jackson 处理泛型的标准方式。由于 Java 的泛型擦除，List<SysUser>.class 在运行时就是 List.class，丢失了 SysUser 这个类型信息。TypeReference 通过匿名子类保留类型信息，Jackson 读取时能正确解析内部元素类型。

2.3 parseMap —— 转 Dict

public static Dict parseMap(String text) {
    if (StringUtils.isBlank(text)) {
        return null;
    }
    return JSON_MAPPER.readValue(text, JSON_MAPPER.getTypeFactory().constructType(Dict.class));
}

Dict 是 Hutool 提供的扩展 HashMap，在 RuoYi 体系中被广泛使用。它与普通 Map 的区别在于提供了类型安全的 getter：

Dict dict = JsonUtils.parseMap("{\"name\":\"张三\",\"age\":25}");
String name = dict.getStr("name");   // 自动转型，比 (String) dict.get("name") 安全
Integer age = dict.getInt("age");     // null 安全，返回 null 而非 NPE

在解析配置 JSON、第三方 API 响应等场景中，Dict 比 Map 更好用。

2.4 parseArrayMap —— 转 Dict 列表

public static List<Dict> parseArrayMap(String text) {
    if (StringUtils.isBlank(text)) {
        return null;
    }
    return JSON_MAPPER.readValue(text,
        JSON_MAPPER.getTypeFactory().constructCollectionType(List.class, Dict.class));
}

将 JSON 数组 [{"a":1}, {"b":2}] 转成 List<Dict>。适用于解析配置数组。

constructCollectionType 的作用：它构建的是一个带泛型信息的 JavaType 对象——List<Dict> 而不仅仅是 List。Jackson 据此知道列表元素的类型是 Dict，从而正确反序列化。

2.5 parseArray —— 转指定类型列表

public static <T> List<T> parseArray(String text, Class<T> clazz) {
    if (StringUtils.isEmpty(text)) {
        return new ArrayList<>();   // 注意：空字符串返回空列表，而非 null
    }
    return JSON_MAPPER.readValue(text,
        JSON_MAPPER.getTypeFactory().constructCollectionType(List.class, clazz));
}

与 parseArrayMap 的一个关键区别：空输入返回空列表而非 null。这是因为业务代码通常期望「空数组输入 → 空列表输出」以简化调用方的判空逻辑：

// 不需要担心 NPE
List<SysUser> users = JsonUtils.parseArray(jsonStr, SysUser.class);
for (SysUser user : users) {  // 空字符串进来，users 是空 List，循环安全
    // ...
}

这个设计也适用于从 Redis 或配置中取数组的场景——配置缺失时返回空列表比返回 null 更友好。

2.6 toJsonStringExcludeFields —— 序列化并移除字段

public static String toJsonStringExcludeFields(Object object, String... fieldNames) {
    if (ObjectUtil.isNull(object)) {
        return null;
    }
    JsonNode node = JSON_MAPPER.valueToTree(object);  // 先序列化成 JsonNode 树
    removeFields(node, fieldNames);                    // 递归移除指定字段
    return toJsonString(node);                         // 再转回 JSON 字符串
}

这个方法的设计思路是「先序列化、再删除、再输出」——不依赖 @JsonIgnore 等注解来隐藏字段，而是在运行时动态决定移除哪些字段。典型场景：

// 同一个 User 对象，管理员看到全部信息，普通用户看不到 password
String adminView = JsonUtils.toJsonString(user);
String userView = JsonUtils.toJsonStringExcludeFields(user, "password", "salt");

工作流程：valueToTree 将 Java 对象转换为 Jackson 的 JsonNode 树结构 → removeFields 递归遍历树删除指定字段 → toJsonString 将修改后的树序列化为 JSON 字符串。

2.7 removeFields —— 递归字段删除

public static JsonNode removeFields(JsonNode node, String... fieldNames) {
    if (node == null || ArrayUtil.isEmpty(fieldNames)) {
        return node;
    }
    if (node.isObject()) {
        ObjectNode objectNode = (ObjectNode) node;
        for (String fieldName : fieldNames) {
            if (StringUtils.isNotBlank(fieldName)) {
                objectNode.remove(fieldName);
            }
        }
    }
    for (JsonNode child : node) {
        removeFields(child, fieldNames);  // 递归到子节点
    }
    return node;
}

递归的关键在于：不仅删除当前层级的对象中的字段，还会递归到当前节点的所有子节点中继续删除。例如：

{
  "user": {"name":"张三", "password":"123"},
  "creator": {"name":"李四", "password":"456"}
}
→ removeFields(node, "password")
→ { "user": {"name":"张三"}, "creator": {"name":"李四"} }

这在批量返回场景中非常实用——一次调用就能递归清除嵌套对象中的敏感字段，不需要在每个嵌套对象上都调用一次。

2.8 格式校验三件套：isJson / isJsonObject / isJsonArray

这三个方法共享同一个核心——readTreeQuietly() 私有方法：

private static JsonNode readTreeQuietly(String str) {
    if (StringUtils.isBlank(str)) {
        return null;
    }
    try {
        return JSON_MAPPER.readTree(str);
    } catch (Exception e) {
        return null;  // 解析失败不抛异常，静默返回 null
    }
}

readTree 是 Jackson 提供的方法，尝试将 JSON 字符串解析为 JsonNode 树结构。如果解析成功，说明字符串是合法的 JSON 构造；如果抛异常，说明不是 JSON。

三个公开方法在这个基础上加了类型判断：

// 是合法的 JSON（对象或数组都可以）
public static boolean isJson(String str) {
    return readTreeQuietly(str) != null;
}

// 是 JSON 对象（{} 形式）
public static boolean isJsonObject(String str) {
    JsonNode node = readTreeQuietly(str);
    return node != null && node.isObject();
}

// 是 JSON 数组（[] 形式）
public static boolean isJsonArray(String str) {
    if (StringUtils.isBlank(str)) return false;  // 注意：这里有冗余的 blank 检查
    JsonNode node = readTreeQuietly(str);
    return node != null && node.isArray();
}

isJsonArray 中有一个冗余的 StringUtils.isBlank 检查——因为 readTreeQuietly 内部已经做了同样的检查。这可能是防御性编程的痕迹。在逻辑上无害，但从代码整洁度看可以移除。

三、@JsonPattern 校验注解体系

这个体系由 3 个文件组成，形成了标准的 Jakarta Bean Validation 扩展模式：自定义注解 → 自定义校验器 → 枚举驱动策略。

3.1 JsonType 枚举 —— 校验策略定义

文件：org/dromara/common/json/validate/JsonType.java

public enum JsonType {
    OBJECT,  // 要求输入必须是 JSON 对象，如 {"a":1}
    ARRAY,   // 要求输入必须是 JSON 数组，如 [1,2,3]
    ANY      // 对象或数组都允许（默认值）
}

三个值的语义完全对应 Jackson 的 JsonNode.isObject() 和 JsonNode.isArray() 判断。

3.2 @JsonPattern 注解 —— 声明式校验

文件：org/dromara/common/json/validate/JsonPattern.java

@Documented
@Target({ElementType.METHOD, ElementType.FIELD})
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = JsonPatternValidator.class)
public @interface JsonPattern {

    JsonType type() default JsonType.ANY;  // 期望的 JSON 类型，默认不限

    String message() default "不是有效的 JSON 格式";  // 校验失败消息

    Class<?>[] groups() default {};        // Bean Validation 分组

    Class<? extends Payload>[] payload() default {}; // 负载信息
}

这是一个标准的 Jakarta Bean Validation 注解定义：

  • @Constraint(validatedBy = ...) 将校验逻辑委托给 JsonPatternValidator
  • @Target 限定在 METHOD 和 FIELD 上（不能用在类上）
  • groups() 和 payload() 是 Bean Validation 规范要求的固定字段

使用示例：

// 只允许 JSON 对象
@JsonPattern(type = JsonType.OBJECT, message = "配置内容必须是 JSON 对象格式")
private String configContent;

// 只允许 JSON 数组
@JsonPattern(type = JsonType.ARRAY)
private String tagList;

// 任意 JSON 格式都可以（默认值）
@JsonPattern
private String jsonData;

3.3 JsonPatternValidator —— 校验逻辑实现

文件：org/dromara/common/json/validate/JsonPatternValidator.java

public class JsonPatternValidator implements ConstraintValidator<JsonPattern, String> {

    private JsonType jsonType;

    @Override
    public void initialize(JsonPattern annotation) {
        this.jsonType = annotation.type();  // 从注解中读取期望的 JSON 类型
    }

    @Override
    public boolean isValid(String value, ConstraintValidatorContext context) {
        if (StringUtils.isBlank(value)) {
            return true;  // 空值放行，交给 @NotBlank 或 @NotNull 控制
        }
        return switch (jsonType) {
            case ANY    -> JsonUtils.isJson(value);
            case OBJECT -> JsonUtils.isJsonObject(value);
            case ARRAY  -> JsonUtils.isJsonArray(value);
        };
    }
}

设计要点：

  1. 空值放行策略：isValid() 中先用 StringUtils.isBlank(value) 判断空值，空值直接返回 true。这是 Bean Validation 校验器的常见模式——校验器只关心「有值时的格式是否正确」，而「是否允许为空」交给 @NotBlank、@NotNull、@NotEmpty 这些专职注解控制。两者结合使用，职责清晰：
// 字段必填且必须是 JSON 对象
@NotBlank(message = "配置不能为空")
@JsonPattern(type = JsonType.OBJECT, message = "配置必须是 JSON 对象格式")
private String configContent;
  1. switch 分支策略：根据 jsonType 枚举值分发到三个不同的校验方法。注意 ANY 用的是 isJson()（不限制类型），而 OBJECT 和 ARRAY 用了更精确的判断。三者可以做到精准校验。

四、校验链路完整数据流

以「用户提交了一个 JSON 配置字段」为例：

@JsonPattern(type = JsonType.OBJECT, message = "配置必须是 JSON 对象格式")
private String config;  // 用户输入: {"timeout":30}
1. Bean Validation 框架检测到 @JsonPattern 注解 ↓ 2. 查找 validatedBy = JsonPatternValidator.class ↓ 3. 调用 JsonPatternValidator.initialize(annotation) → jsonType = JsonType.OBJECT ↓ 4. 调用 JsonPatternValidator.isValid(value, context) → value = "{"timeout":30}" → StringUtils.isBlank(value) = false（继续） → switch(OBJECT) → JsonUtils.isJsonObject(value) ↓ 5. JsonUtils.isJsonObject("{"timeout":30}") → readTreeQuietly("{"timeout":30}") → JSON_MAPPER.readTree("{"timeout":30}") → ObjectNode（解析成功） → node.isObject() = true → 返回 true ✓

如果用户输入了不合法的 JSON：

config = "not json"
→ readTreeQuietly("not json")
→ JSON_MAPPER.readTree("not json") → 抛 JsonParseException
→ catch → return null
→ isJsonObject = false → 校验失败 ✗
→ 返回 "配置必须是 JSON 对象格式" 错误消息

设计洞察：判空标准不一致的深层原因。isEmpty（StringUtils）和 isBlank 的区别在于：isEmpty 认为 " "（纯空格）非空，isBlank 认为它为空。为什么 parseObject(String, Class) 用 isEmpty 而 parseObject(String, TypeReference) 用 isBlank？因为前者处理的通常是简单对象（如 SysUser），纯空格输入解析失败后 Jackson 自己会报错；后者处理复杂泛型，纯空格输入可能导致诡异的类型推断错误，提前拦截更安全。但说到底，这是历史和习惯的结果，并非有意的精密设计。

检验环节

Q1. JsonUtils.parseArray("[]" , SysUser.class) 返回什么？parseArray("" , SysUser.class) 又返回什么？

看答案

parseArray("[]", SysUser.class) 返回一个空的 ArrayList<SysUser>（size=0）。parseArray("", SysUser.class) 同样返回空的 ArrayList——因为空字符串触发了 isEmpty 检查，返回 new ArrayList<>() 而非 null。这是 parseArray 独有的行为。

Q2. @JsonPattern 注解的 type 属性默认值是什么？如果不设置 type，用户可以传 JSON 数组吗？

看答案

默认值是 JsonType.ANY。不设置 type 时，对象和数组都允许——因为校验器会调用 JsonUtils.isJson(value) 而非 isJsonObject 或 isJsonArray。

Q3. 为什么 JsonPatternValidator 中空值直接返回 true？

看答案

这是 Bean Validation 的职责分离原则：@JsonPattern 只负责「有值时的格式校验」，而「是否允许为空」由 @NotBlank、@NotNull 等专职注解控制。如果 @JsonPattern 把空值也拦截了，就必须同时设置 message 来区分「值为空」和「格式错误」两种失败情况，增加了复杂度。

Q4. toJsonStringExcludeFields(user, "password") 能删除嵌套在 user.children 中的 password 字段吗？

看答案

能。removeFields 方法会递归遍历 JsonNode 树的所有子节点，不仅是当前层级。所以如果 user 对象里嵌套了 child 对象，child 的 password 也会被删除。
