第 2 课 · JSON 增强框架三阶段

本节目标：学完能完整复述 JSON 增强框架的三阶段生命周期（collect 收集 → prepare 预处理 → process 渲染），以及四个核心组件（JsonEnhancementContext 数据总线、JsonFieldContext 字段上下文、JsonFieldProcessor 扩展接口、JsonValueEnhancer 执行引擎）各自的职责与协作方式。理解为什么三阶段分离是消除 N+1 查询的关键设计。

一、这个框架解决什么问题？

假设一个典型的 REST API 场景：后端查询了 SysUser 列表返回给前端。但用户表里存的是 dept_id（部门 ID），前端需要显示的是部门名称。你可以在 SQL 里 JOIN 查询部门表，也可以查出用户后循环补上部门名称。

第二种做法就是响应增强——在序列化成 JSON 之前，把某些字段的值替换成业务需要的最终形式。常见场景包括：

  • 字段翻译：把 user_id 替换成用户名、把 dept_id 替换成部门名
  • 字段脱敏：把手机号 13812345678 替换成 138****5678
  • 字典转换：把 sex=0 替换成 sexName="男"
  • 字典拼接：把多个字段组装成一个新字段

但这里有一个性能陷阱：如果业务代码在循环里逐个字段查数据库，100 个用户就会产生 100 次 SQL 查询——典型的 N+1 问题。

JSON 增强框架的核心设计目标就是：让扩展开发者在做字段增强时，自然地采用「先收集、再批量查询、最后渲染」的模式，从架构上消除 N+1 查询。

二、三阶段生命周期全景

框架将一次 HTTP 响应的增强过程拆成三个互不重叠的阶段：

HTTP 响应对象 │ ▼ ┌─ collect（收集阶段）─── 递归扫描对象树 ─────────────────────┐ │ 每个字段被每个 Processor 检查一次 │ │ Processor.collect() 把需要处理的 key 存入 Context │ │ 例：发现 userId=1001, userId=1002, userId=1003 │ │ 存入 context.setAttribute("userIds", [1001,1002,1003]) │ └─────────────────────────────────────────────────────────────┘ │ ▼ ┌─ prepare（预处理阶段）── 所有 Processor 各执行一次 ──────────┐ │ Processor.prepare() 基于 collect 收集的数据执行批量 IO │ │ 例：select * from sys_user where id in (1001,1002,1003) │ │ 结果存入 context.setAttribute("userMap", {1001: "张三"})│ │ 一次 SQL 查出全部，消除 N+1 │ └─────────────────────────────────────────────────────────────┘ │ ▼ ┌─ process（处理阶段）── 递归渲染 JSON 树 ─────────────────────┐ │ 每个字段被检查，从 Context 取 prepare 写入的结果 │ │ Processor.process() 返回替换后的值 │ │ 例：userId=1001 → 去 userMap 查 → 返回 "张三" │ └─────────────────────────────────────────────────────────────┘ │ ▼ 增强后的 JSON 响应

这三个阶段的分离是框架的灵魂。如果 collect 和 process 合并（一边扫描一边处理），就回到了 N+1 模式。把它们拆开，中间插入一个批量的 prepare，才真正做到了性能优化。

三、四个核心组件逐一看

3.1 JsonEnhancementContext —— 单次响应的数据总线

文件：org/dromara/common/json/enhance/JsonEnhancementContext.java

类比：如果把一次响应增强比作一个 HTTP 请求的处理过程，JsonEnhancementContext 就是这个请求级别的「黑板」——collect 阶段往上面写需要处理的 key，prepare 阶段读取 key 做批量查询并回写结果，process 阶段读取结果做字段替换。

它持有三个核心成员：

public class JsonEnhancementContext {
    private final JsonMapper jsonMapper;                        // 用于后续的 JSON 操作
    private final Map<String, Object> attributes = new LinkedHashMap<>(); // 共享数据表
    private boolean processingRequired;                         // 是否需要处理的标记
}

attributes 为什么用 LinkedHashMap？保持插入顺序——虽然三阶段之间不依赖 Map 的顺序，但在同一阶段的多次 setAttribute 调用中保持有序对调试友好。

processingRequired 标记是一个聪明的性能优化。如果遍历完整个对象树发现没有任何字段需要增强（没有 Processor 的 supports() 返回 true），就直接跳过 prepare 和 process 阶段，原样返回对象。避免了对无增强需求的响应做无意义的 JSON 序列化/反序列化。

Context 提供了 6 个属性操作方法和 1 个标记方法：

方法用途使用阶段
setAttribute(key, value)写入共享数据collect、prepare
getAttribute(key)读取共享数据prepare、process
getOrCreateAttribute(key, supplier)懒初始化读取（不存在时创建）prepare
containsAttribute(key)判断 key 是否存在prepare
removeAttribute(key)移除属性prepare
markProcessingRequired()标记需要处理collect
设计洞察：为什么 attribute key 建议用 Processor 全限定类名做前缀？因为多个 Processor 共享同一个 Context。如果两个 Processor 都用 "userMap" 做 key，后写的会覆盖先写的。用 "com.example.UserTranslateProcessor.userMap" 做 key 可以天然隔离命名空间。

3.2 JsonFieldContext —— 字段的元数据快照

文件：org/dromara/common/json/enhance/JsonFieldContext.java

这是一个 Java Record，封装了 Processor 在判断「这个字段归不归我管」和「怎么处理它」时所需要的全部信息：

public record JsonFieldContext(
    Object owner,           // 字段所属的对象实例（如 SysUser 对象本身）
    String propertyName,    // 属性名（如 "deptId"）
    AnnotatedMember member, // Jackson 的元数据包装（可从中读取注解、类型等信息）
    Object value            // 当前值（如 103）
) {
    public <A extends Annotation> A getAnnotation(Class<A> annotationType) {
        return member == null ? null : member.getAnnotation(annotationType);
    }
}

每个字段的信息来源：

  • owner：通过反射从对象实例获取，Processor 可以调用 owner 上的方法
  • propertyName：来自 Jackson 的 BeanPropertyDefinition.getName()，会处理 @JsonProperty 等注解的重命名
  • member：Jackson 的 AnnotatedMember——它封装了 getter 方法或 filed 本身，能读取 Java 注解
  • value：通过 member.getValue(owner) 反射获取的当前值

getAnnotation() 方法是 Processor 判定逻辑的核心入口。例如一个翻译处理器判断某个字段是否需要翻译，就是检查字段上有没有 @Translate 注解：

// Processor 的 supports() 实现
public boolean supports(JsonFieldContext fieldContext) {
    return fieldContext.getAnnotation(Translate.class) != null;
}

3.3 JsonFieldProcessor —— 扩展点接口

文件：org/dromara/common/json/enhance/JsonFieldProcessor.java

这是框架最核心的抽象——所有自定义增强逻辑都通过实现这个接口来接入。它定义了 4 个方法，全部是 default 实现：

public interface JsonFieldProcessor {

    // 判断：这个字段归不归我管？
    default boolean supports(JsonFieldContext fieldContext) {
        return true;  // 默认全管，实现者通常按注解过滤
    }

    // 收集：记录需要增强的 key
    default void collect(JsonFieldContext fieldContext, JsonEnhancementContext context) {
    }

    // 预处理：批量 IO，结果写入 context
    default void prepare(JsonEnhancementContext context) {
    }

    // 处理：返回替换后的值（返回原 value 表示不修改）
    default Object process(JsonFieldContext fieldContext, Object value, JsonEnhancementContext context) {
        return value;
    }
}

全部 default 实现的意义：实现者只需覆盖自己关心的阶段。比如一个只做「把枚举值转成中文名」的 Processor，可能只需要覆盖 process()（有值就直接转换，不需要 collect 和 prepare）。而一个「把部门 ID 翻译成部门名称」的 Processor 需要覆盖全部三个阶段。

每个阶段的调用时机和调用次数：

阶段调用时机调用次数说明
supports()每个字段每次扫描时N（字段数） 次collect 和 process 阶段都会调用
collect()对象树递归扫描时，每个符合 supports 的字段M（需处理的字段数）次将 key 存入 context
prepare()所有 collect 完成后1 次批量 IO，是整个框架的性能核心
process()渲染 JSON 树时，每个符合 supports 的字段M 次从 context 取结果，O(1) 替换

3.4 JsonValueEnhancer —— 增强引擎

文件：org/dromara/common/json/enhance/JsonValueEnhancer.java

这是驱动整个增强流程的执行引擎，也是字段最长的类（约 380 行）。核心方法 enhance(Object body) 是一个典型的模板方法模式：

public Object enhance(Object body) {
    if (body == null || body instanceof JsonNode || processors.isEmpty()) {
        return body;  // 短路：空值、已是 JSON 节点、无处理器 → 直接返回
    }

    // Phase 1: 创建上下文 + 收集
    JsonEnhancementContext context = new JsonEnhancementContext(jsonMapper);
    collectValue(body, context, new IdentityHashMap<>());

    // Phase 1.5: 检查是否需要处理（无字段需要增强则短路）
    if (!context.isProcessingRequired()) {
        return body;
    }

    // Phase 2: 批量预处理
    processors.forEach(processor -> processor.prepare(context));

    // Phase 3: 渲染增强后的 JSON 树
    return renderValue(body, context, new IdentityHashMap<>());
}

递归扫描：collectValue

collectValue 方法递归遍历整个对象树，同时使用 IdentityHashMap 做循环引用保护：

private void collectValue(Object value, JsonEnhancementContext context, IdentityHashMap<Object, Boolean> visited) {
    // 1. null → 返回
    // 2. Map → 遍历 values
    // 3. Iterable（List/Set）→ 遍历元素
    // 4. 数组 → 遍历元素
    // 5. 简单值（String/Number/Boolean/Date/Temporal/Enum/UUID/Class）→ 短路
    // 6. 已访问（visited.contains）→ 短路（循环引用保护）
    // 7. 标记为已访问 → 遍历属性 → 每个字段调用 Processor.collect()
    // 8. 移除访问标记（退出时清理）
}

值得注意的细节：

  • 简单值短路：isSimpleValue() 方法检查类型是否为基本类型、String、Number、Boolean、Character、Date、Temporal、Enum、UUID、Class。这些类型不会有子字段需要增强，直接跳过。
  • IdentityHashMap vs HashMap：用 == 而非 .equals() 判断对象是否相同。防止两个内容相同但不同的对象被误判为循环引用。
  • 进入时标记、退出时清理：在 try-finally 中操作 visited map，确保退出递归栈帧时恢复状态。

属性元数据缓存：getProperties

引擎维护了一个 ConcurrentHashMap<Class<?>, List<PropertyMetadata>> 缓存——同一个类型的属性元数据只解析一次：

private List<PropertyMetadata> getProperties(Class<?> type) {
    return propertyCache.computeIfAbsent(type, this::resolveProperties);
}

resolveProperties() 通过 Jackson 的 ClassIntrospector 获取类型的可序列化属性列表（field 或 getter），并调用 member.fixAccess(true) 强制打开访问权限。这一步生成的 PropertyMetadata 是一个内部 Record，持有属性名和 AnnotatedMember 引用。

为什么不用 Java 反射而用 Jackson 的 ClassIntrospector？因为 Jackson 的 BeanPropertyDefinition 已经处理了 @JsonProperty 重命名、@JsonIgnore 忽略、getter 命名约定等规则。直接复用它比重新实现一套要稳定得多。

渲染增强：renderValue + renderPojo

渲染阶段和收集阶段走的是同一套递归遍历逻辑，区别在于——除了遍历结构相同之外，渲染阶段会对每个字段调用 processor.process() 来获取替换值：

private ObjectNode renderPojo(Object value, JsonEnhancementContext context, IdentityHashMap<Object, Boolean> visited) {
    ObjectNode objectNode = jsonMapper.createObjectNode();
    for (PropertyMetadata metadata : getProperties(value.getClass())) {
        Object originalValue = metadata.getValue(value);
        JsonFieldContext fieldContext = new JsonFieldContext(value, metadata.propertyName(), metadata.member(), originalValue);
        Object processedValue = originalValue;
        boolean changed = false;
        for (JsonFieldProcessor processor : processors) {
            if (!processor.supports(fieldContext)) continue;
            Object nextValue = processor.process(fieldContext, processedValue, context);
            changed = changed || !Objects.equals(processedValue, nextValue);
            processedValue = nextValue;
        }
        JsonNode childNode = changed
            ? enhanceTranslatedValue(processedValue, context, visited)
            : renderValue(processedValue, context, visited);
        objectNode.set(metadata.propertyName(), childNode);
    }
    return objectNode;
}

二次增强（enhanceTranslatedValue）：如果 Processor 返回的值是一个复杂对象（非简单值），引擎会对这个新值重新执行一遍完整的增强流程（collect → prepare → process）。这允许一个增强的返回结果是另一个增强的输入。

例如：用户对象原本有 deptId=103 → 翻译成 DeptUser 对象（含 deptName="研发部" 和 leaderId=1001）→ 二次增强再把 leaderId 翻译成 "张三"。两次增强串联完成。

设计洞察：changed 标记 + 二次增强的组合设计解决了增强值本身需要继续增强的场景，但不会对未变化的值做额外开销。这是一个精密的性能优化——用 Objects.equals 做引用级比较，省去了不必要的 JSON 树重建。

转换器过滤：supports(converterType)

不是所有的 HTTP 响应都需要增强。引擎通过 supports() 方法排除了三类消息转换器：

public boolean supports(Class<?> converterType) {
    return !processors.isEmpty()
        && !ByteArrayHttpMessageConverter.class.isAssignableFrom(converterType)   // 文件下载
        && !StringHttpMessageConverter.class.isAssignableFrom(converterType)        // 纯文本
        && !ResourceHttpMessageConverter.class.isAssignableFrom(converterType);     // 静态资源
}

文件下载返回字节流、纯文本返回 String、静态资源返回 Resource——它们都不需要通过 JSON 增强框架处理。强行处理反而会破坏响应体格式。

四、Processor 列表的排序机制

在构造器中，引擎对 Processor 列表做了排序：

public JsonValueEnhancer(JsonMapper jsonMapper, List<JsonFieldProcessor> processors) {
    this.jsonMapper = jsonMapper;
    List<JsonFieldProcessor> sortedProcessors = new ArrayList<>(processors);
    AnnotationAwareOrderComparator.sort(sortedProcessors);
    this.processors = Collections.unmodifiableList(sortedProcessors);
}

AnnotationAwareOrderComparator 是 Spring 提供的排序器，会读取 @Order 注解或 Ordered 接口来决定执行顺序。排序后转为不可修改列表，确保处理链的执行顺序在运行时是确定的。

这对二次增强场景尤为重要：如果 Processor A 先执行，把 deptId 翻译成 DeptUser 对象；Processor B 再执行，对 DeptUser 中的 leaderId 做翻译。顺序错了就会失败。

五、完整数据流实例

假设有一个返回 List<SysUser> 的接口，用户对象如下：

SysUser { userId: 1, userName: "张三", deptId: 103, phone: "13812345678" }

配置了两个 Processor：DeptTranslateProcessor（把 deptId 翻译成部门名）、PhoneDesensitizeProcessor（把手机号脱敏）。

Collect 阶段：

  1. 递归扫描 SysUser 对象，遍历到 deptId 字段
  2. DeptTranslateProcessor.supports() 返回 true（字段上有 @Translate 注解），collect() 把 deptId=103 记录到 context 的待查询列表
  3. 遍历到 phone 字段，PhoneDesensitizeProcessor.supports() 返回 true（字段上有 @Desensitize 注解）——但脱敏不需要 collect，直接跳过
  4. context.setAttribute("deptIds", [103])
  5. context.markProcessingRequired() → true

Prepare 阶段：

  1. DeptTranslateProcessor.prepare()：select * from sys_dept where dept_id in (103)，结果 {"103":"研发部"} 写入 context.setAttribute("deptMap", ...)
  2. PhoneDesensitizeProcessor.prepare()：不需要准备，空方法直接返回

Process 阶段：

  1. 遍历到 deptId，DeptTranslateProcessor.process() 从 context 取 deptMap，找到 "研发部"，返回新值；
  2. 遍历到 phone，PhoneDesensitizeProcessor.process() 直接做字符串替换，返回 "138****5678"
  3. 最终输出 JSON：{"userId":1, "userName":"张三", "deptId":"研发部", "phone":"138****5678"}

检验环节

Q1. 为什么增强框架要拆成 collect / prepare / process 三个阶段，而不是直接在扫描时调用 process？

看答案

如果在扫描时直接调用 process，每遇到一个需要翻译的字段都会触发一次数据库查询（N+1 问题）。拆成三个阶段后，collect 只收集 key，prepare 一次性批量查询，process 只需 O(1) 从 context 取结果。三阶段的核心价值就是为批量预处理留出空间。

Q2. JsonEnhancementContext 的 attributes 用 LinkedHashMap 而非 HashMap 的原因是什么？

看答案

保持插入顺序——虽然三阶段之间的数据传递不依赖 Map 顺序（通过 key 精确查找），但在调式和日志输出时有序 Map 可读性更好。功能上 HashMap 也能工作。

Q3. JsonValueEnhancer 为什么排除了 ByteArrayHttpMessageConverter 等三种转换器？

看答案

这三种转换器处理的响应类型不是 JSON：ByteArray 处理文件下载（字节流）、String 处理纯文本、Resource 处理静态资源。对它们做 JSON 增强没有意义，反而可能破坏响应内容。

Q4. collectValue 方法中用 IdentityHashMap 而非 HashMap 做 visited 追踪，为什么？

看答案

IdentityHashMap 用 ==（引用比较）而非 .equals()（值比较）。如果两个不同对象恰好 equals 为 true（比如两个内容相同的 User 对象），HashMap 会误判为循环引用并跳过第二个。IdentityHashMap 避免了这个误判。
