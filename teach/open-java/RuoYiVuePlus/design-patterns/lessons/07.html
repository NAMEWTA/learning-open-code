第 7 课 · 模板方法模式 — 骨架统一、细节各异 行为型模式

🎯 本节目标：理解模板方法模式的核心思想——在父类中定义算法骨架，将可变步骤延迟到子类实现。并能识别 RuoYi-Vue-Plus 中的模板方法案例。

模板方法模式是行为型模式中最经典的一个。核心思想只有一句话：父类写流程，子类填空。父类定义一个 final 或不希望被重写的模板方法，其中调用若干个抽象方法（hook），子类只需实现这些 hook 即可。

一、AbstractOssClientImpl — 最完整的模板方法范例

打开 ruoyi-common/ruoyi-common-oss/src/main/java/org/dromara/common/oss/client/AbstractOssClientImpl.java：

public abstract class AbstractOssClientImpl implements OssClient {

    private final AtomicBoolean initialized = new AtomicBoolean(false);
    protected final String clientId;
    protected OssClientConfig config;
    protected S3AsyncClient s3AsyncClient;
    protected S3TransferManager s3TransferManager;
    protected S3Presigner s3Presigner;
    protected ExecutorService asyncExecutor;

    // ── 构造器：模板入口 ──
    public AbstractOssClientImpl(String clientId, OssClientConfig config) {
        Assert.notNull(config, () -> S3StorageException.form("..."));
        this.clientId = StringUtils.isBlank(clientId)
            ? IdUtil.fastSimpleUUID() : clientId;
        this.config = config;
        this.initialize();   // ← 构造时就调用模板方法
    }

    // ── 模板方法（final 语义：不允许子类重写流程）──
    @Override
    public void initialize() {
        if (isInitialized()) {          // 1. 防重复初始化
            return;
        }
        try {
            doInitialize();             // 2. 调用抽象方法——子类填空
            initialized.compareAndSet(false, true);  // 3. 标记已初始化
        } catch (Exception e) {
            throw toStorageException(e);
        }
    }

    // ── 抽象 hook：子类必须实现 ──
    abstract void doInitialize();

    // ── 所有上传/下载/删除操作都实现在此，子类无需关心 ──
    // bucketUpload(), bucketDownload(), bucketDelete() ...
}
1 模板方法 = initialize()：它定义了初始化三步走：检查状态 → 调用 hook → 标记完成。子类永远不需要重写这个方法——流程是固定的。 AbstractOssClientImpl.java:137-149
2 抽象 hook = doInitialize()：不同云厂商（阿里云 OSS、AWS S3、MinIO）的初始化逻辑完全不同——endpoint、认证方式、region 都不同。但父类不关心这些差异，只声明 abstract void doInitialize()，让子类各自实现。 AbstractOssClientImpl.java:154
3 构造器中调用模板方法：this.initialize() 放在构造器最后一行——对象创建完毕，立即执行初始化。这是一种常见的模板方法使用姿势：让构造器成为模板的触发器。 AbstractOssClientImpl.java:99
4 防重复初始化：用 AtomicBoolean initialized 做 CAS 检查。即使模板方法被多次调用，实际初始化只执行一次。这个防御性设计体现了「骨架中的保护逻辑不应由子类负责」的原则。 AbstractOssClientImpl.java:139-141

子类只需关注初始化

假设你要对接阿里云 OSS，你只需：

public class AliyunOssClientImpl extends AbstractOssClientImpl {

    public AliyunOssClientImpl(String clientId, OssClientConfig config) {
        super(clientId, config);  // 父类构造器自动调用 initialize() → doInitialize()
    }

    @Override
    void doInitialize() {
        // 只需配置阿里云专属的 endpoint、凭证
        this.s3AsyncClient = S3AsyncClient.builder()
            .endpointOverride(URI.create(config.getEndpoint()))
            .credentialsProvider(...)
            .region(Region.of(config.getRegion()))
            .build();
        this.s3TransferManager = S3TransferManager.builder()
            .s3Client(this.s3AsyncClient).build();
        this.s3Presigner = S3Presigner.builder()
            .s3Client(this.s3AsyncClient).build();
    }
    // 上传、下载、删除？完全不用写——父类已经实现好了！
}
💡 模板方法的最大价值：子类代码量从 1100+ 行缩减到 ~30 行。所有通用的桶操作、异常处理、路径构建都在父类中，子类只写「差异部分」。这就是 DRY 原则的最高境界。

二、AbstractEncryptor — 更轻量的模板

打开 ruoyi-common/ruoyi-common-encrypt/src/main/java/org/dromara/common/encrypt/core/encryptor/AbstractEncryptor.java：

public abstract class AbstractEncryptor implements IEncryptor {

    public AbstractEncryptor(EncryptContext context) {
        // 用户配置校验与配置注入
        // 子类构造时自动完成通用校验，无需各自重复
    }

    // 子类实现具体的加密/解密算法
    // public abstract String encrypt(String plainText);
    // public abstract String decrypt(String cipherText);
}

这里模板方法的体现更隐式：构造器就是模板。无论子类是 SM2、SM4 还是 AES 加密器，构造时都会执行父类中的配置校验。子类无需关心校验逻辑，只需实现 encrypt() 和 decrypt()。

三、模板方法 vs 策略模式 — 别搞混了

模板方法（Template Method）： AbstractOssClientImpl ├── initialize() ← 骨架固定，final 语义 │ ├── 检查状态 │ ├── doInitialize() ← 子类填空（hook） │ └── 标记完成 └── 所有 bucket 操作 ← 父类实现，子类继承复用 策略模式（Strategy）： IAuthStrategy (接口) ├── PasswordAuthStrategy ← 整个算法替换 ├── SmsAuthStrategy ← 整个算法替换 └── SocialAuthStrategy ← 整个算法替换 核心区别： 模板方法：骨架在父类，子类只填空白 → 复用骨架 策略模式：接口定义契约，每个实现是完整算法 → 替换算法
⚠️ 判断标准：如果你的多个子类有 80% 相同的代码流程，只有 20% 不同——用模板方法。如果每个实现是完全不同的算法，只是接口相同——用策略模式。

四、前端等价物：useSearchReset 组合式函数

打开 plus-ui-vue/src/hooks/form/useSearchReset.ts，这是 Vue 3 中的「模板方法」思想：

// 搜索重置的组合式函数 — 提供模板化的重置流程
export function useSearchReset(options?: {
  afterReset?: () => void;   // ← hook：重置后的自定义逻辑
}) {
  const queryParams = ref({...});

  // 模板方法：重置查询的固定流程
  function resetQuery() {
    queryParams.value = { ...initQueryParams };  // 1. 重置参数
    handleQuery();                                // 2. 重新查询
    options?.afterReset?.();                      // 3. hook：允许自定义后续操作
  }

  return { queryParams, resetQuery, handleQuery };
}

这里的 resetQuery 就是模板方法——它定义了重置的标准流程，通过 afterReset 回调让调用方填入自己的逻辑。和 Java 中的抽象方法异曲同工。

3. 模板方法模式中，模板方法应该用 final 修饰吗？

📖 点击查看答案

1. B — AtomicBoolean 的 CAS 操作是无锁的，且 initialize() 的设计允许重复调用（幂等），不需要互斥。如果用了 synchronized，每次调用都要获取锁，即使已经初始化完毕。
2. B — 云存储场景中上传/下载/删除的流程完全相同（构建请求 → 执行 → 处理结果），只有初始化不同，是模板方法的理想场景。A 和 C 中每种实现是独立算法，适合策略模式。
3. B — Effective Java 的建议是：模板方法应该用 final 修饰，防止子类破坏骨架。但如果框架设计中确实有合理的扩展点需要覆盖整个流程，可以不 final。本项目 initialize() 没有用 final 但注释清楚表达了「不建议重写」。
