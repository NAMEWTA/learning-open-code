第 6 课 · 策略模式 — 多变行为的优雅封装 行为型模式

🎯 本节目标：理解策略模式在 RuoYi-Vue-Plus 中的三种实现形态，能对比 Interface+Spring Beans、Enum+Lambdas、函数参数三种方式的使用场景，并掌握「何时用策略，何时用 if-else」的决策框架。

策略模式（Strategy）的精髓：定义一系列算法，把它们一个个封装起来，并使它们可以相互替换。策略模式让算法的变化独立于使用算法的客户端。

一、形态一：接口 + Spring Beans — IAuthStrategy

打开 ruoyi-admin/.../service/IAuthStrategy.java。这是本项目中策略模式最典型的应用——5 种登录授权类型，每种都是一个独立的策略 Bean：

// IAuthStrategy.java 第 19-81 行（核心结构）
public interface IAuthStrategy {
    String BASE_NAME = "AuthStrategy";

    // ── 工厂方法（静态方法，通过 grantType 派发到具体策略）──
    static LoginVo login(String body, SysClientVo client, String grantType) {
        String beanName = grantType + BASE_NAME;  // 如 "password" + "AuthStrategy"
        if (!SpringUtils.containsBean(beanName)) {
            throw new ServiceException("授权类型不正确!");
        }
        IAuthStrategy instance = SpringUtils.getBean(beanName);
        return instance.login(body, client);
    }

    // ── 策略接口（每个具体策略实现此方法）──
    LoginVo login(String body, SysClientVo client);
}
grantType = "password" ──> SpringUtils.getBean("passwordAuthStrategy") grantType = "sms" ──> SpringUtils.getBean("smsAuthStrategy") grantType = "email" ──> SpringUtils.getBean("emailAuthStrategy") grantType = "social" ──> SpringUtils.getBean("socialAuthStrategy") grantType = "wechat" ──> SpringUtils.getBean("wechatAuthStrategy") 每种策略都是一个 @Service 实现 IAuthStrategy，以 "grantType + AuthStrategy" 命名
1 策略接口：login(body, client) 是抽象策略。每个具体策略（密码登录、短信登录、邮箱登录……）实现相同的签名，但各自的登录逻辑完全不同。 IAuthStrategy.java:79
2 策略工厂：static login(body, client, grantType) 是策略工厂方法。通过 grantType + "AuthStrategy" 拼接 Bean 名称，从 Spring 容器中动态查找对应的策略 Bean。这比写 if-else 分支优雅得多——新增一种登录方式只需新建一个 Bean，不需要修改任何已有代码。 IAuthStrategy.java:31-39
3 公共工具方法：buildLoginParameter(client) 是所有策略共享的 Sa-Token 登录参数构建方法。策略接口不仅可以包含抽象方法，还可以包含公共逻辑——减少策略实现类的重复代码。 IAuthStrategy.java:47-70
⚠️ 为什么用 Spring Bean 而非手动 new？ 因为每种策略可能需要注入不同的依赖（如 PasswordAuthStrategy 需要 SysUserService，SmsAuthStrategy 需要 SmsService）。通过 @Service + @Autowired 让 Spring 管理依赖，策略工厂只需按名字查找即可。如果手动 new，策略内部的依赖注入会失效。

二、形态二：枚举 + Lambda — SensitiveStrategy

打开 ruoyi-common-sensitive/.../core/SensitiveStrategy.java（第 17-121 行）。这是一个非常巧妙的策略模式实现：

// SensitiveStrategy.java 第 17-118 行（核心结构）
@AllArgsConstructor
public enum SensitiveStrategy {
    ID_CARD(s -> DesensitizedUtil.idCardNum(s, 3, 4)),       // 身份证
    PHONE(DesensitizedUtil::mobilePhone),                      // 手机号
    EMAIL(DesensitizedUtil::email),                            // 邮箱
    BANK_CARD(DesensitizedUtil::bankCard),                     // 银行卡
    CHINESE_NAME(DesensitizedUtil::chineseName),               // 中文名
    PASSWORD(DesensitizedUtil::password),                      // 密码
    STRING_MASK(s -> DesensitizedUtils.mask(s, 4, 4, 4)),     // 通用脱敏
    MASK_HIGH_SECURITY(s -> DesensitizedUtils.maskHighSecurity(s, 2, 2)), // 高安全
    CLEAR(s -> DesensitizedUtil.clear()),                      // 清空为""
    CLEAR_TO_NULL(s -> DesensitizedUtil.clearToNull());        // 清空为null
    // ... 共 17 种策略

    private final Function<String, String> desensitizer;

    public Function<String, String> desensitizer() {
        return desensitizer;
    }
}

调用方式极简：

// 在需要脱敏的地方
String masked = SensitiveStrategy.PHONE.desensitizer().apply("13812345678");
// 结果: "138****5678"

// 通过注解使用（框架自动调用）
@Sensitive(strategy = SensitiveStrategy.ID_CARD)
private String idCard;
💡 枚举策略 vs 接口策略：如果策略数量固定、策略内部逻辑简单（无需外部依赖）、枚举常量的名称自带「语义化」，用枚举+Lambdas 比接口+Bean 更轻量。但枚举策略无法注入 Spring Bean，逻辑复杂时不适合。

三、形态三：函数参数 — 策略作为方法参数传入

这是策略模式最轻量的形态——把策略作为一个函数参数传入。本项目中有两个典型例子：

例 1 — TreeBuildUtils 的 classifier 参数：

// TreeBuildUtils.java 第 46 行
// action 和 classifier 都是策略函数——调用者决定如何分组、如何关联子节点
TreeBuildUtils.build(menus, parentId,
    SysMenu::getParentId,              // 策略: 用 parentId 分组
    (menu, nodeTreeMaps) -> {         // 策略: 设置子节点
        menu.setChildren(nodeTreeMaps.get(menu.getMenuId()));
    });

例 2 — 前端 Excel 导出 useTableSelection：

// 前端 useTableSelection.ts — 调用者传入 getRowId 作为行标识策略
const { selectedRows, toggleRowSelection } = useTableSelection(data, {
    getRowId: (row) => row.userId   // 策略函数: 如何标识一行
});
// 不同场景可以传入不同的 getRowId:
// getRowId: (row) => row.orderNo   (订单模块)
// getRowId: (row) => row.productSku (商品模块)

四、三种实现形态对比

维度接口 + Spring Beans枚举 + Lambdas函数参数
策略数量动态增长（新增无需修改代码）编译期固定（新增需改枚举）调用点即时定义
策略复杂度高（可注入依赖）低-中（纯函数，无依赖）任意（可捕获上下文）
开闭原则✅ 完全符合（新增策略 = 新增类）⚠️ 需修改枚举（增加常量）✅ 符合（调用点自行定义）
发现性分散在多个 Bean 文件中集中在一个枚举中，一目了然分布在各调用点，不易发现
本项目案例IAuthStrategySensitiveStrategyTreeBuildUtils classifier/action

五、前端中的策略模式

5.1 加密策略：RSA + AES 混合加密

前端 plus-ui-vue/src/utils/request.ts 中的请求加密使用了策略组合：

// request.ts — 请求加密策略
async function encryptRequest(config: InternalAxiosRequestConfig) {
    // 策略: 随机生成 AES 密钥
    const aesKey = generateAesKey();
    // 策略: 用 AES 加密请求体
    const encryptedBody = aesEncrypt(JSON.stringify(config.data), aesKey);
    // 策略: 用 RSA 公钥加密 AES 密钥
    const encryptedKey = rsaEncrypt(aesKey, publicKey);
    // 将加密后的 AES 密钥放在请求头，密文放在 body
    config.headers['encrypt-key'] = encryptedKey;
    config.data = { encryptBody: encryptedBody };
}

六、单元格合并策略 — CellMergeStrategy

打开 ruoyi-common-excel/.../core/CellMergeStrategy.java（第 22 行）。这是策略模式在 EasyExcel 中的体现——Excel 导出时，将合并策略作为一个参数注入导出流程：

// CellMergeStrategy.java 第 68-79 行
@Override
protected void merge(Sheet sheet, Cell cell, Head head, Integer relativeRowIndex) {
    // 策略逻辑: 单元格如果在合并区域内但非首行 → 清空内容
    for (CellRangeAddress cellAddresses : cellList) {
        if (cellAddresses.isInRange(cell) && rowIndex != cellAddresses.getFirstRow()) {
            cell.setBlank();
        }
    }
}

使用时，EasyExcel 的 registerWriteHandler(new CellMergeStrategy(list, true)) 将合并策略注册到导出流程中。EasyExcel 框架在写入每一行时回调策略的 merge 方法——这是典型的策略模式 + 回调模式组合。

七、策略模式 vs if-else：决策框架

⚠️ 不是所有 if-else 都要改成策略模式。 过度设计策略模式会让代码更复杂。以下是决策框架：
条件用策略模式保留 if-else
分支数量3 个以上，且持续增长2-3 个，基本不变
分支复杂度每个分支有独立业务逻辑（10+ 行）每个分支 1-3 行
是否需要依赖注入是否
是否需要独立测试是（每个策略独立单元测试）否（整体测试即可）
开闭原则新增分支不修改已有代码新增分支需改动
示例登录认证（5 种方式）、数据脱敏（17 种规则）state == 1 ? "启用" : "禁用"
💡 项目中的反面教材：假设你把 IAuthStrategy 的 5 种登录方式写成一个包含 5 个 if-else 分支的方法，当需要新增第 6 种登录方式（如 Apple 登录）时，你必须修改已有的 login() 方法——有破坏已有代码的风险。而用策略模式，你只需新建 AppleAuthStrategy 类并注册为 Bean，零侵入。

2. SensitiveStrategy 使用枚举 + Lambda 实现策略模式，其优点不包括？

3. 以下哪种情况最适合保留 if-else 而非使用策略模式？

📖 点击查看答案

1. B — SpringUtils.getBean(beanName) 通过 grantType 拼接的 Bean 名称从 Spring 容器获取策略实例。这就要求每个策略实现类必须用 @Service("passwordAuthStrategy") 等形式注册为同名 Bean。
2. B — 枚举常量是在类加载时初始化的静态实例，枚举的构造函数在静态初始化阶段执行，Spring 容器此时还未就绪，因此 Lambda 中不能注入 Spring Bean。如果需要依赖注入，应该使用接口 + Bean 模式。
3. B — 3 种固定映射、每行 1 个表达式，可以写成 Map.of(0, "待处理", 1, "处理中", 2, "已完成").get(status) 或简单的 if-else。策略模式在这种情况下属于过度设计。
