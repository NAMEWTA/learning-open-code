第 2 课 · 适配器与装饰器 — 包装的艺术 结构型模式

🎯 本节目标：能区分适配器和装饰器的不同意图——适配器转换接口，装饰器增强功能。能看懂 CaffeineCacheDecorator 如何同时扮演这两个角色。

适配器和装饰器都通过「包装对象」来工作。核心区别：适配器的动机是兼容（让两个不兼容的接口能协作），装饰器的动机是增强（在不修改原对象的前提下增加新功能）。

一、适配器模式：让 Caffeine 适配 Spring Cache

打开 ruoyi-common-redis/.../manager/CaffeineCacheDecorator.java：

public class CaffeineCacheDecorator implements Cache {
    // 原生 Caffeine 缓存（Guava 风格 API）
    private final com.github.benmanes.caffeine.cache.Cache<Object, Object> caffeine;
    // 被包装的 Spring Cache（通常是 RedissonCache）
    private final Cache cache;

    public CaffeineCacheDecorator(String name, Cache cache) {
        this.cache = cache;
        this.caffeine = Caffeine.newBuilder()
            .maximumSize(10000)
            .expireAfterWrite(Duration.ofMinutes(10))
            .build();
    }

    // ── 适配：将 Spring Cache.get(key) 委托给 Caffeine → 被包装 Cache ──
    @Override
    public ValueWrapper get(Object key) {
        String uniqueKey = name + ":" + key;
        // 先查 L1 缓存（Caffeine 本地），未命中则查 L2（Redis）
        return caffeine.get(uniqueKey, k -> cache.get(key));
    }

    @Override
    public void put(Object key, Object value) {
        caffeine.invalidate(name + ":" + key);  // 使 L1 失效
        cache.put(key, value);                   // 写入 L2
    }

    @Override
    public void evict(Object key) {
        caffeine.invalidate(name + ":" + key);  // 清除两级
        cache.evict(key);
    }
}
1 适配器角色：CaffeineCacheDecorator implements Cache（Spring 的 Cache 接口）。Caffeine 原始 API 是 get(key, Function) 风格，与 Spring Cache 的 ValueWrapper get(Object) 不兼容。这个类把 Caffeine API 转换为 Spring Cache API。 CaffeineCacheDecorator.java:13
2 装饰器角色：它在不修改底层 Cache cache（Redis L2）的前提下，在之前加了一层 Caffeine L1 缓存。对于调用方来说，接口没变，但行为增强了——多了本地缓存加速。 CaffeineCacheDecorator.java:69-71
3 L1+L2 读取流程：caffeine.get(uniqueKey, k -> cache.get(key)) 先查 Caffeine 本地缓存，若未命中则通过 lambda 查 Redis，结果自动写入 Caffeine。这是经典的多级缓存读模式。 CaffeineCacheDecorator.java:70
💡 为什么这个类同时是适配器和装饰器？

这是结构型模式中常见的重叠现象：

  • 适配器：Caffeine 的 Cache<K,V> 接口和 Spring 的 Cache 接口不是同一个类型体系。这个类将 Caffeine「翻译」成了 Spring 能理解的形式。
  • 装饰器：它包装了一个已有的 Cache 对象（Redis），在不改变其接口的前提下增添了本地缓存层。

在实践中，这种双重角色很常见——当需要包装的第三方库接口不兼容时，包装器天然就是适配器。如果它还在原功能基础上添加了新行为，那它就是装饰器。

二、Servlet Wrapper 装饰器链

项目中有 4 个 Servlet Request/Response Wrapper，形成一条装饰器链。它们都继承自 HttpServletRequestWrapper / HttpServletResponseWrapper——这是 Servlet 规范提供的装饰器基类：

2.1 XssHttpServletRequestWrapper — XSS 过滤装饰

public class XssHttpServletRequestWrapper extends HttpServletRequestWrapper {
    @Override
    public String getParameter(String name) {
        return cleanXSS(super.getParameter(name));  // 在原始行为上增加过滤
    }
    @Override
    public String[] getParameterValues(String name) {
        String[] values = super.getParameterValues(name);
        if (values != null) {
            return Arrays.stream(values).map(this::cleanXSS).toArray(String[]::new);
        }
        return null;
    }
    private String cleanXSS(String value) {
        // 去除 <script>、onerror= 等危险标签和属性
        return value.replaceAll("<script>", "").replaceAll("on\\w+=", "");
    }
}

位置：ruoyi-common-web/.../filter/XssHttpServletRequestWrapper.java

2.2 RepeatedlyRequestWrapper — 请求体可重复读装饰

public class RepeatedlyRequestWrapper extends HttpServletRequestWrapper {
    private final byte[] body;  // 缓存请求体

    public RepeatedlyRequestWrapper(HttpServletRequest request) throws IOException {
        super(request);
        body = IoUtil.readBytes(request.getInputStream(), false);
    }

    @Override
    public ServletInputStream getInputStream() {
        return new ByteArrayServletInputStream(body);  // 每次返回新的流
    }
    // 这样请求体可以被多次读取——日志记录读一次，业务逻辑读一次
}

位置：ruoyi-common-web/.../filter/RepeatableFilter.java

2.3 DecryptRequestBodyWrapper / EncryptResponseBodyWrapper — 加解密装饰

// 请求体解密装饰器
public class DecryptRequestBodyWrapper extends HttpServletRequestWrapper {
    @Override
    public ServletInputStream getInputStream() throws IOException {
        // 读取加密的 body → RSA 解密 AES key → AES 解密内容 → 返回明文流
        byte[] encryptedBody = IoUtil.readBytes(super.getInputStream());
        byte[] decryptedBody = encryptorManager.decrypt(encryptedBody, encryptContext);
        return new ByteArrayServletInputStream(decryptedBody);
    }
}

// 响应体加密装饰器
public class EncryptResponseBodyWrapper extends HttpServletResponseWrapper {
    @Override
    public ServletOutputStream getOutputStream() throws IOException {
        // 业务写出的明文 → AES 加密 → RSA 加密 AES key → 写入加密后的数据
        return new EncryptServletOutputStream(super.getOutputStream(), encryptContext);
    }
}

位置：ruoyi-common-encrypt/.../filter/

三层的 Filter 执行链

HTTP Request │ ▼ CryptoFilter ────────────────────────────────────────────── │ 检查 @ApiEncrypt 注解，决定是否包装 ▼ RepeatableFilter ───────────────────────────────────────── │ 包装为 RepeatedlyRequestWrapper（可多次读 body） ▼ XssFilter ──────────────────────────────────────────────── │ 包装为 XssHttpServletRequestWrapper（过滤危险标签） ▼ PlusWebInvokeTimeInterceptor (Spring Interceptor) │ 记录请求参数和耗时 ▼ Controller.handleRequest() — 此处拿到的已是层层装饰后的 request
⚠️ 装饰器的顺序很重要！ CryptoFilter 必须在 RepeatableFilter 之前——加密过滤器需要读取原始 body 进行解密，解密后才交给 RepeatableFilter 缓存明文流。如果顺序反了，RepeatableFilter 会缓存密文，后续业务逻辑拿到的是密文而非明文。

四、前端中的装饰器思想

前端 Vue 3 项目中，Axios 拦截器本质上是装饰器模式。打开 plus-ui-vue/src/utils/request.ts：

// 请求拦截器：装饰 config，增加 token、加密等
service.interceptors.request.use(config => {
    config.headers['Authorization'] = 'Bearer ' + getToken();
    if (VITE_APP_ENCRYPT && config.headers.isEncrypt) {
        // 加密请求体——在不修改 axios 源码的前提下增强功能
        const aesKey = generateAesKey();
        config.headers['encrypt-key'] = encrypt(aesKey);
        config.data = aesEncrypt(JSON.stringify(config.data), aesKey);
    }
    return config;
});

// 响应拦截器：装饰 response，增加解密、统一错误处理
service.interceptors.response.use(response => {
    if (response.headers['encrypt-key']) {
        // 解密响应体
        response.data = aesDecrypt(response.data, ...);
    }
    const { code, msg } = response.data;
    if (code === 401) showReloginDialog();  // 统一处理未授权
    return response.data;
});
💡 思想共通性：前端的拦截器和后端的 Servlet Wrapper 本质是一样的——在原始对象外面包一层，在不修改原始对象的前提下添加行为。只是前端用函数式拦截器（中间件），后端用面向对象包装器（Wrapper 类）。
