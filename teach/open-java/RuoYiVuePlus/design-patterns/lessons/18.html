第 18 课 · 包结构与命名规范

面对 30+ 个 Maven 模块和上百个类文件，RuoYi-Vue-Plus 靠什么保持井然有序？本课拆解它的包结构设计哲学与命名约定，让你读代码时一眼就能找到目标文件。

本课目标：阅读本课后，你能独立解释 RuoYi-Vue-Plus 的后端包结构、前端目录约定、以及为什么"I接口/Impl实现"模式在这个项目里是合理的——并能在 10 分钟内写出符合项目规范的 Controller、Service 和 VO 骨架。

1. 顶层命名空间

所有 Java 源码全部位于 org.dromara 包下。dromara 是 RuoYi-Vue-Plus 所属的开源社区组织名称，统一顶层包名避免了与第三方库的包冲突。

模块与包的映射关系

Maven 模块Java 包职责
ruoyi-common-coreorg.dromara.common.core全局常量、通用工具、基础异常、R/PageResult
ruoyi-common-weborg.dromara.common.webWeb 层通用配置、过滤器、全局异常处理
ruoyi-common-redisorg.dromara.common.redisRedis 缓存、Redisson 分布式锁、限流
ruoyi-common-encryptorg.dromara.common.encryptAPI 加解密、字段级脱敏、国密支持
ruoyi-common-mybatisorg.dromara.common.mybatisMyBatis-Plus 增强、BaseMapperPlus
ruoyi-common-satokenorg.dromara.common.satokenSa-Token 权限框架适配
ruoyi-systemorg.dromara.system系统管理：用户、角色、菜单、部门
ruoyi-workfloworg.dromara.workflow工作流引擎集成
ruoyi-adminorg.dromara.web前端控制器入口层
ruoyi-apiorg.dromara.system.api 等模块间远程调用接口契约
关键设计：ruoyi-admin 模块的包名是 org.dromara.web 而非 org.dromara.admin。因为这个模块承载 Web 入口（Controller 层的"胶水代码"），命名为 web 更直观。

2. 业务模块内部标准结构

以 ruoyi-system 为例，每个业务模块内部遵循一致的分层结构：

org.dromara.system ├── controller/ ← 控制器层 │ ├── system/ SysNoticeController、SysMessageController ... │ └── monitor/ CacheController、SysOperlogController ... ├── service/ ← 服务接口层 │ ├── ISysUserService.java │ ├── ISysRoleService.java │ └── impl/ ← 服务实现层 │ ├── SysUserServiceImpl.java │ └── SysRoleServiceImpl.java ├── mapper/ ← 数据访问层 │ ├── SysUserMapper.java │ └── SysRoleMapper.java ├── domain/ ← 领域对象层 │ ├── SysUser.java 实体 Entity │ ├── SysRole.java │ ├── vo/ 视图对象 VO │ │ ├── SysUserExportVo.java │ │ └── SysOperLogVo.java │ ├── bo/ 业务对象 BO │ └── dto/ 数据传输对象 DTO └── listener/ ← 事件监听器

5. 为什么要 IService / ServiceImpl？

很多现代项目已经抛弃了接口-实现分离，直接用 @Service 标注实现类。RuoYi-Vue-Plus 为什么坚持 I{Entity}Service + {Entity}ServiceImpl？

理由说明
模块间 RPC 代理ruoyi-api 模块暴露的 Feign 接口需要定义契约——Service 接口天然就是契约
多实现切换OSS、SMS、Social 等模块存在多厂商实现，接口隔离不同实现
AOP 代理要求Spring AOP 默认使用 JDK 动态代理，需要接口才能代理（否则走 CGLIB）
团队习惯从 RuoYi 经典版继承的编码传统，降低老用户的迁移成本
注意：如果某个 Service 只有一个实现且永远不会被远程调用，可以不创建接口。项目中的 common 模块内部工具类 Service 就没有走 IService/Impl 模式。不要为了"统一"而创建无意义的接口。

6. 反模式警示

反模式表现正确做法
上帝工具类 一个 Utils.java 5000+ 行，什么都往里塞 按职责拆分：DateUtils、StringUtils、FileUtils
项目已有：StreamUtils、MessageUtils、ValidatorUtils
无接口的 Impl 只有 *ServiceImpl.java，没有接口 如果不需要多态，直接用类名如 UserService；
需要接口时，接口必须有明确职责边界
Controller 写业务逻辑 Controller 方法里直接查数据库、拼 JSON Controller 只做参数校验和路由转发；
业务逻辑一律在 Service 层
包名与模块名不一致 模块叫 user-center，包名叫 com.xxx.uc 项目统一用 org.dromara.{模块}，一目了然

自测题

问题 1：以下哪个是 RuoYi-Vue-Plus 中 Service 接口的正确命名？

A. SysUserServiceInterface B. ISysUserService C. sysUserService D. UserService
查看答案

答案：B。项目约定：大 I 前缀 + 驼峰实体名 + Service 后缀。SysUserServiceInterface 冗长，UserService 丢失了模块前缀 Sys。

问题 2：前端 API 模块中，index.ts 和 types.ts 的职责分别是什么？

A. index.ts 放类型定义，types.ts 放 API 函数 B. 两个文件都放 API 函数 C. index.ts 放 API 函数，types.ts 放 TypeScript 接口定义 D. 没有固定规则
查看答案

答案：C。以 api/system/user/ 为例：index.ts 导出 listUser、addUser 等 API 函数；types.ts 定义 UserVO、UserForm、UserQuery 等接口类型。职责分离，互不污染。

问题 3：ruoyi-admin 模块的 Java 包名为什么是 org.dromara.web 而不是 org.dromara.admin？

查看答案

ruoyi-admin 模块承担的角色是聚合各业务模块的 Controller，充当 Web 入口层的"胶水代码"。命名为 web 更能体现其作为 Web 层的定位，而 admin 容易让人误解为只负责后台管理功能。
