第 13 课 · 多级缓存架构 Multi-level Cache

Caffeine L1 + Redis L2 + 事务感知

本课目标：掌握 RuoYi-Vue-Plus 的三层缓存装饰器链 —— CaffeineCacheDecorator（L1 本地）、RedissonCache（L2 分布式）和 TransactionAwareCacheDecorator（事务感知），理解读写路径和缓存命名 DSL。

1. 为什么需要多级缓存

单靠 Redis 已经不够快了吗？是的，在微秒级响应场景下，网络往返（RTT）是不可忽视的开销。
缓存层级技术典型延迟相对速度容量
L1 本地缓存Caffeine（JVM 堆内）~50 ns1x（基准）MB 级
L2 分布式缓存Redis（Redisson）~0.5 ms10,000x 慢于 L1GB 级
L3 数据库MySQL~10 ms20,000x 慢于 L2TB 级

单次 Redis 调用比 Caffeine 慢 10,000 倍。如果 90% 的缓存请求能被 L1 命中，系统吞吐量可以提升一个数量级。

2. 缓存架构全景图

@Cacheable / @CachePut / @CacheEvict (Spring Cache 抽象层) | v PlusSpringCacheManager.getCache("sys_user_name#30d") | |-- 解析 DSL: "sys_user_name" + TTL=30d + maxIdle=0 + maxSize=0 + local=1 | v [Outer] TransactionAwareCacheDecorator —— 事务提交后执行 put/evict | v [Middle] CaffeineCacheDecorator —— L1: JVM 堆内，30s TTL, max 1000 条 | .get() → caffeine.get(key, k -> cache.get(key)) [L1 miss 时穿透到 L2] | .put() → invalidate L1, 再 write L2 | .evict() → evict L2, 成功则 invalidate L1 v [Inner] RedissonCache —— L2: Redis 分布式缓存 | v Redis (持久化、多节点共享)

3. 核心组件 1：CaffeineCacheDecorator（L1 本地缓存）

1 ruoyi-common/ruoyi-common-redis/src/main/java/org/dromara/common/redis/manager/CaffeineCacheDecorator.java

这是一个 Spring Cache 接口的装饰器实现，在 Redisson L2 缓存前面加了一层本地 Caffeine 缓存：

public class CaffeineCacheDecorator implements Cache {

    private final Cache cache;  // 被装饰的 L2 缓存（RedissonCache）

    private final com.github.benmanes.caffeine.cache.Cache<Object, Object> caffeine;

    /** 构造唯一键：cacheName:key */
    private String getUniqueKey(Object key) {
        return name + ":" + key;
    }

    // ===== L1 → L2 的读穿透 =====
    @Override
    public ValueWrapper get(Object key) {
        // caffeine.get(key, loaderFunction) 是原子的"取或加载"
        Object o = caffeine.get(getUniqueKey(key), k -> cache.get(key));
        // L1 命中 → 直接返回（~50ns）
        // L1 未命中 → 从 L2 加载 → 存入 L1 → 返回
        return (ValueWrapper) o;
    }

    // ===== 写入时先失效 L1，再写 L2 =====
    @Override
    public void put(Object key, Object value) {
        caffeine.invalidate(getUniqueKey(key));  // Step 1: 失效本地
        cache.put(key, value);                    // Step 2: 写入 Redis
    }

    // ===== 逐出时先删 L2，成功再失效 L1 =====
    @Override
    public boolean evictIfPresent(Object key) {
        boolean result = cache.evictIfPresent(key);  // Step 1: 删除 Redis
        if (result) {
            caffeine.invalidate(getUniqueKey(key));  // Step 2: 失效本地
        }
        return result;
    }

    // ===== 清空本地缓存（供手动刷新） =====
    @Override
    public void clearLocalCache() {
        caffeine.asMap().keySet().removeIf(key ->
            key.toString().startsWith(name + ":"));
    }
}
关键设计：caffeine.get(key, loaderFunction) 是线程安全的读穿透实现。多个线程同时 miss L1 时，Caffeine 保证只有一个线程去加载 L2，其他线程等待结果。这避免了缓存击穿（多个请求同时穿透到 Redis/DB）。

4. 核心组件 2：PlusSpringCacheManager（缓存管理器 + DSL 解析）

2 ruoyi-common/ruoyi-common-redis/src/main/java/org/dromara/common/redis/manager/PlusSpringCacheManager.java

这是自定义的 RedissonSpringCacheManager 分支，最独特的特性是缓存名称 DSL 解析：

public class PlusSpringCacheManager extends RedissonSpringCacheManager {

    private boolean transactionAware = true;  // 默认开启事务感知

    @Override
    public Cache getCache(String name) {
        // name = "sys_user_name#30d#0#0#1"
        String[] array = name.split("#");
        String cacheName = array[0];           // "sys_user_name"
        String ttl = array.length > 1 ? array[1] : null;       // "30d"
        String maxIdle = array.length > 2 ? array[2] : null;   // "0"
        String maxSize = array.length > 3 ? array[3] : null;   // "0"
        String local = array.length > 4 ? array[4] : "1";      // "1"

        // 动态创建缓存实例
        Cache cache = createMapCache(cacheName, name, config, isLocal(local));
        // ...
    }

    // 装饰器链构建
    private Cache createMapCache(String cacheName, String name,
                                  CacheConfig config, int local) {
        RMapCache<Object, Object> map = RedisUtils.getClient().getMapCache(name);
        Cache cache = new RedissonCache(map, config, allowNullValues);
        // Step 1: 包装 L1 缓存
        if (local == 1 && caffeine != null) {
            cache = new CaffeineCacheDecorator(cacheName, cache, caffeine);
        }
        // Step 2: 包装事务感知
        if (transactionAware) {
            cache = new TransactionAwareCacheDecorator(cache);
        }
        return cache;
    }
}

4.1 DSL 语法

缓存名称中可以嵌入 5 个参数，用 # 分隔：

格式示例含义
name#ttl#maxIdle#maxSize#local demo:cache#60s#10m#20#1 每个字段都可单独配置
参数说明默认值示例
ttl过期时间（支持 s/m/h/d）0（不过期）60s、30d、2h
maxIdleTime最大空闲时间（LRU 清理）0（不检测）10m
maxSize最大条目数（LRU 溢出）0（无限）1000
local是否启用 L1 本地缓存1（开启）1 开启、0 关闭

实际使用示例：

  • test#60s — TTL=60s，其余默认
  • test#0#60s — TTL=0（不过期），MaxIdle=60s
  • test#1h#0#500 — TTL=1h，MaxSize=500
  • test#1h#0#500#0 — TTL=1h，MaxSize=500，禁用 L1
设计优势：每个缓存区域可以独立配置过期策略和容量限制，不需要写任何 Java 配置代码。业务开发者只需在 @Cacheable(cacheNames = "xxx#30d") 中声明参数即可，PlusSpringCacheManager 自动解析并创建对应配置的缓存实例。

5. 核心组件 3：TransactionAwareCacheDecorator（事务感知）

这是 Spring Framework 自带的类（org.springframework.cache.transaction.TransactionAwareCacheDecorator），项目直接使用。它的核心行为：

无事务感知（原始行为）： @Transactional public void updateUser(UserBo bo) { // 1. 更新数据库 baseMapper.update(bo); // 2. 立即更新缓存 cache.put("user:1", bo); // ← 缓存已更新 // 3. 如果事务回滚 → 数据库恢复了，但缓存中的数据是脏的！ } 有事务感知（TransactionAwareCacheDecorator）： @Transactional public void updateUser(UserBo bo) { // 1. 更新数据库 baseMapper.update(bo); // 2. 缓存 put 操作被推迟 cache.put("user:1", bo); // ← 暂存在事务上下文中，不执行 // 3. 事务提交 → 触发注册的回调 → 执行真正的 cache.put() // OR: 事务回滚 → 丢弃暂存的操作 → 缓存保持不变 }
正确性优先：TransactionAwareCacheDecorator 解决了"数据库和缓存不一致"的经典问题。代价是：在事务提交之前，其他线程看到的缓存可能是旧数据。这是一个正确性 vs 实时性的权衡 —— RuoYi-Vue-Plus 选择了正确性。

6. 完整的读写路径

6.1 读路径（@Cacheable 注解）

请求到达 Service 方法 | v TransactionAwareCacheDecorator.get(key) | （事务感知对读操作无影响，直接透传） v CaffeineCacheDecorator.get(key) | ├── L1 命中 ──→ 返回缓存值（~50ns） | └── L1 未命中 | v RedissonCache.get(key) ← 从 Redis 加载 | ├── L2 命中 ──→ 回填 L1 + 返回（~0.5ms） | └── L2 未命中 | v 执行方法体（数据库查询）──→ 回填 L2 + L1 + 返回（~10ms）

6.2 写路径（@CachePut / @CacheEvict 注解）

@CachePut → Service 方法执行完毕 | v TransactionAwareCacheDecorator.put(key, value) | 注册到事务同步回调，等待事务提交 | （事务未提交 = 操作被暂存） v 【事务提交后】 | v CaffeineCacheDecorator.put(key, value) | ├── Step 1: caffeine.invalidate(key) // 先失效 L1 | └── Step 2: cache.put(key, value) // 再更新 L2（Redis） @CacheEvict 同理： TransactionAwareCacheDecorator.evictIfPresent(key) | （等待事务提交） v CaffeineCacheDecorator.evictIfPresent(key) ├── Step 1: cache.evictIfPresent(key) // 先删除 L2 └── Step 2: caffeine.invalidate(key) // 再失效 L1
为什么写入要先失效 L1 再写 L2，而不是先写 L2 再失效 L1？如果先写 L2 再失效 L1，在两步之间可能有其他线程读到旧的 L1 数据。先失效 L1 确保了后续的读操作会穿透到 L2 获取最新数据。虽然 L2 在极短时间内可能也是旧数据，但这个窗口比"旧 L1 + 新 L2"组合导致的长期不一致要安全得多。

8. Caffeine 基础配置

4 ruoyi-common/ruoyi-common-redis/src/main/java/org/dromara/common/redis/config/CacheConfig.java 
@AutoConfiguration
@EnableCaching  // 启用 Spring Cache 抽象
public class CacheConfig {

    @Bean
    public Cache<Object, Object> caffeine() {
        return Caffeine.newBuilder()
            .expireAfterWrite(30, TimeUnit.SECONDS)  // L1 的 TTL: 30 秒
            .initialCapacity(100)
            .maximumSize(1000)                        // L1 最多 1000 条
            .build();
    }

    @Bean
    public CacheManager cacheManager() {
        return new PlusSpringCacheManager();  // Caffeine Bean 通过 Spring 注入
    }
}
注意 L1 和 L2 的 TTL 是独立的：Caffeine 的 expireAfterWrite(30s) 是 L1 的过期时间，与 CacheNames 中定义的 #30d（L2 Redis 的 TTL）是完全独立的两个配置。L1 的短 TTL（30 秒）意味着本地缓存会频繁刷新，确保数据最终一致性，而 L2 的长 TTL（30 天）减少了数据库查询。

9. 常用缓存操作模式总结

模式注解典型场景示例
读穿透@Cacheable高频查询、低频变更用户名、部门名、字典数据
写穿透@CachePut插入/更新后同步缓存系统配置、字典类型
逐出@CacheEvict删除/禁用后清理缓存客户端状态变更、部门删除
组合逐出@Caching一个操作影响多个缓存部门更新 → 逐出 SYS_DEPT + SYS_DEPT_AND_CHILD
编程式操作CacheUtils.evict()循环中逐出、复杂条件批量更新子部门时逐个逐出

自测题

Q1：为什么 CaffeineCacheDecorator 的 put() 方法先失效 L1 再写 L2，而 get() 方法先查 L1 再查 L2？这两个顺序是巧合还是精心设计的？

查看答案

是精心设计的。
读路径（get）：先 L1 后 L2 是自然的读穿透模式 —— 越快的层越先查，命中率最高。
写路径（put）：先失效 L1 再写 L2 的原因是避免长期不一致。如果先写 L2 再失效 L1，在两步之间其他线程可能从 L1 读到旧数据。先失效 L1 后，其他线程的读请求会穿透到 L2 获取最新值，虽然 L2 在极短时间内可能也是旧数据（还没写入），但这个窗口远小于 L1 持有多分钟甚至多天的旧数据。

Q2：TransactionAwareCacheDecorator 将缓存操作推迟到事务提交后执行。如果一个事务耗时 5 秒，这 5 秒内其他请求读到的缓存是旧数据还是新数据？这会引发问题吗？

查看答案

会读到旧数据。因为在事务提交前，新的缓存值还未写入 Redis 或 Caffeine，其他请求读到的仍然是修改前的数据。
这确实是一个权衡：性能和最终一致性 vs 强一致性。RuoYi-Vue-Plus 选择了正确性优先（保证缓存和数据库的一致性），代价是事务期间的缓存可能是旧数据。对于大多数管理后台场景（用户管理、角色管理、字典配置），这种短暂的不一致是可以接受的。如果需要强一致性，可以考虑使用分布式锁或直接读数据库。

Q3：CacheNames 中 SYS_USER_NAME = "sys_user_name#30d" 指定了 Redis TTL=30 天，但 Caffeine L1 的 TTL 是 30 秒。为什么 L1 的 TTL 短这么多？

查看答案

三个原因：
1. 内存容量：L1 是 JVM 堆内存，最大 1000 条，长 TTL 会导致高价值数据被踢出（LRU 逐出）。30 秒的短 TTL 确保热点数据常驻，冷数据自然淘汰。
2. 数据一致性：如果 L1 也有 30 天 TTL，那么某个用户改完名后，其他节点的 L1 可能还缓存着 30 天前的旧名字（尽管通过 Redisson 发布订阅可以失效，但不是 100% 可靠）。30 秒的短 TTL 提供了一个"安全网"——即使失效通知丢失，30 秒后本地缓存自动过期。
3. 性能平衡：30 秒的 L1 TTL 已经能覆盖绝大多数重复查询（同一用户在同一请求链路中可能被查多次），更长的 TTL 边际收益递减。
