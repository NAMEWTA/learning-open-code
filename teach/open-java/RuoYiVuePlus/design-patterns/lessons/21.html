2. 模式密度热力图

哪些模块的模式浓度最高？

模块模式密度包含的模式
ruoyi-common-redis 最高 工厂 (Redisson 连接) + 装饰器 (缓存包装) + 代理 (AOP 限流) + 单例 (Bean) + 模板方法 (缓存基类)
ruoyi-common-encrypt 最高 模板方法 + 工厂 (加解密器) + 装饰器 (字段包装) + 责任链 (过滤链) + 策略 (AlgorithmType)
ruoyi-common-oss 高 工厂 (OssFactory) + 策略 (厂商切换) + 模板方法 (AbstractOssClientImpl) + 桥接 (操作分离)
ruoyi-admin 中 策略 (多入口适配) + 观察者 (事件监听) + 外观 (Controller 聚合) + 单例
ruoyi-system 中 组合 (树形结构) + 观察者 (事件发布) + 备忘录 (操作日志)
为什么 redis 和 encrypt 模块密度最高？这两个模块本质上是对"外部能力"（Redis/加密算法）的封装层。封装层天然需要工厂创建对象 + 策略切换实现 + 装饰器增强 + 代理控制访问——这四种模式恰好是"基础设施模块"的标准配方。

3. 反模式识别

反模式在项目中的表现是否有意为之？
Service Locator SpringUtils.getBean() 静态方法获取 Bean 有意且节制。项目只在少数无法走依赖注入的场景使用（如静态工厂方法、非 Spring 管理的 POJO 需要获取 Bean），并且局限在 ruoyi-common-core 内部。普通业务代码绝不应该使用。
枚举承载行为 SensitiveStrategy 枚举中每个值包含脱敏算法的具体实现 可辩论。优点：代码紧密、一目了然。缺点：违反开闭原则（新增策略需修改枚举类）。对于有限且稳定的脱敏类型（手机号、身份证、银行卡等），这是一个务实的取舍。
Utils 膨胀 某些 Utils 类行数较多 需警惕。项目已经将工具类按职责拆分（StreamUtils、MessageUtils、ValidatorUtils 等），但个别工具类仍在增长。新增工具方法时优先考虑是否应独立为一个类。

4. 设计原则速查

将项目中的实践映射到 SOLID 五原则：

原则项目中如何体现
开闭原则 (OCP)Strategy 模式保证对扩展开放、对修改封闭。新增 OSS 厂商或加密算法只需新增一个类，无需修改已有代码。
单一职责 (SRP)30+ common 模块各司其职。ruoyi-common-redis 只管缓存、ruoyi-common-mail 只管邮件——绝不会出现一个"万能 common"模块。
依赖倒置 (DIP)ruoyi-api 模块定义了 Feign 接口契约，各模块依赖接口而非具体实现。Service 的 IService/ServiceImpl 分层也是 DIP 的体现。
接口隔离 (ISP)UserService 对外只暴露真正需要的方法。feign 接口与本地 Service 接口可以不同——外部调用者只看到它们需要的。
里氏替换 (LSP)AbstractOssClientImpl 定义了完整的抽象方法签名，任何子类（Aliyun/Minio/Qiniu）都可透明替换，不改变程序正确性。

5. 决策框架：何时用哪个模式？

面对新需求时，按以下流程图快速决策：

你的需求是什么？ │ ├── 需要运行时切换算法/实现 │ → Strategy（策略模式） │ 示例：切换 OSS 厂商、切换加密算法、切换短信通道 │ ├── 需要统一多个子系统的接口 │ → Facade（外观模式） │ 示例：Controller 聚合多个 Service 对外提供 REST API │ ├── 需要在不修改原代码前提下增强功能 │ → Decorator（装饰器模式） │ 示例：给缓存加过期检查、给数据库字段加解密 │ ├── 需要控制对象访问 │ → Proxy（代理模式） │ 示例：AOP 实现限流、权限校验、日志记录 │ ├── 需要多个对象协同处理请求 │ → Chain of Responsibility（责任链） │ 示例：过滤器链（加解密 → XSS → 鉴权 → 日志） │ ├── 需要创建一系列相关对象 │ → Factory / Abstract Factory（工厂/抽象工厂） │ 示例：根据配置的不同厂商创建对应的客户端实例 │ ├── 需要定义算法骨架，子步骤可变 │ → Template Method（模板方法） │ 示例：文件上传的通用流程（校验→上传→回调），具体上传方式由子类实现 │ ├── 需要一对多的状态联动通知 │ → Observer（观察者模式） │ 示例：用户注册后发通知、记日志、初始化默认数据 │ └── 需要处理树形/递归结构 → Composite（组合模式） 示例：部门树、菜单树、行政区划树

自测题

问题 1：以下哪个模块的模式密度最高？

A. ruoyi-system B. ruoyi-common-redis C. ruoyi-admin D. ruoyi-gen
查看答案

答案：B。ruoyi-common-redis 和 ruoyi-common-encrypt 是两个模式密度最高的模块，各自包含工厂、装饰器、代理、单例、模板方法等 5 种模式。这是因为"基础设施封装"天然需要这几种模式的组合。

问题 2：项目中 SpringUtils.getBean() 被谨慎使用而非完全禁止，原因是什么？

查看答案

有些场景确实无法走依赖注入——例如静态工厂方法中需要获取一个 Bean，或者非 Spring 管理的 POJO 对象需要调用 Spring 服务。在这些极少数场景下，SpringUtils 是务实的"逃生舱"。但普通业务代码必须使用构造器注入（@RequiredArgsConstructor），绝不能把 SpringUtils 当成偷懒的工具。

问题 3：如果你需要在已有的文件上传功能上增加"上传前自动压缩"和"上传后自动生成缩略图"，应该用什么设计模式？

A. 工厂模式（Factory） B. 装饰器模式（Decorator） C. 观察者模式（Observer） D. 享元模式（Flyweight）
查看答案

答案：B。装饰器模式的核心场景就是"不修改原有类的前提下，动态地增加额外行为"。你可以创建一个 CompressUploadDecorator 和 ThumbnailUploadDecorator，层层包裹原有的上传逻辑，符合开闭原则。观察者模式（C）也能实现"上传后"的响应，但更适合一对多的通知场景（如上传后同时通知 3 个系统），而非修改核心流程。
