第 5 课 · 代理模式 — AOP 切面体系 结构型模式

🎯 本节目标：理解 Spring AOP 如何通过代理模式实现横切关注点，能解释 @Around / @Before / @AfterReturning 的差异，并能区分 AOP 代理、Filter 链和 Interceptor 链的适用场景。

代理模式（Proxy）的精髓：为另一个对象提供一个替身或占位符，以控制对该对象的访问。在 Spring Boot 中，代理模式通过 AOP（面向切面编程）实现——在不修改目标方法代码的前提下，在方法前后织入额外逻辑。

二、LogAspect：环绕通知的经典实现

打开 ruoyi-common-log/.../aspect/LogAspect.java 第 65-77 行：

// LogAspect.java 第 65-77 行
@Around(value = "@annotation(controllerLog)")
public Object doAround(ProceedingJoinPoint joinPoint, Log controllerLog) throws Throwable {
    StopWatch stopWatch = new StopWatch();
    stopWatch.start();
    try {
        // ── 执行目标方法（控制权交给目标）──
        Object jsonResult = joinPoint.proceed();
        // ── 成功后记录日志 ──
        handleLog(joinPoint, controllerLog, null, jsonResult, stopWatch);
        return jsonResult;
    } catch (Exception e) {
        // ── 异常时记录日志 ──
        handleLog(joinPoint, controllerLog, e, null, stopWatch);
        throw e;  // 异常仍然向上抛出，不影响业务
    }
}
1 @Around 的特点：拥有 ProceedingJoinPoint，可以决定是否调用目标方法、修改参数和篡改返回值。它是能力最强的通知类型。 LogAspect.java:65-66
2 StopWatch 计时：第 67 行和 127 行分别在方法执行前后调用 stopWatch.start() 和 stopWatch.stop()，计算耗时——这是典型的「代理增强」行为（统计性能）。 LogAspect.java:67,127
3 异步事件入库：第 130 行通过 SpringUtils.context().publishEvent(operLog) 发布事件，日志的实际写入由事件监听器异步处理，避免阻塞业务线程。 LogAspect.java:130

三、RateLimiterAspect：前置通知 + SpEL 表达式

打开 ruoyi-common-redis/.../aspectj/RateLimiterAspect.java 第 59-86 行：

// RateLimiterAspect.java 第 59-76 行
@Before("@annotation(rateLimiter)")
public void doBefore(JoinPoint point, RateLimiter rateLimiter) {
    // ① 组装限流 key（支持 SpEL 表达式动态计算）
    String combineKey = getCombineKey(rateLimiter, point);
    // ② 调用 Redisson 的 RateLimiter
    long number = RedisUtils.rateLimiter(combineKey, rateType, count, time, timeout);
    // ③ 令牌不足时抛出异常阻断请求
    if (number == -1) {
        throw new ServiceException(message);
    }
}

第 95-124 行的 getCombineKey() 方法是关键——它支持 SpEL 表达式，让限流 key 可以基于请求参数动态计算：

// RateLimiterAspect.java 第 95-113 行（简化）
private String getCombineKey(RateLimiter rateLimiter, JoinPoint point) {
    String key = rateLimiter.key();
    if (StringUtils.containsAny(key, "#")) {  // 包含 SpEL 表达式
        // 解析方法参数，创建 SpEL 上下文
        MethodBasedEvaluationContext context =
            new MethodBasedEvaluationContext(null, targetMethod, args, pnd);
        // 执行 SpEL 表达式得到运行时 key
        key = expression.getValue(context, String.class);
    }
    // 拼接前缀: rate_limit:{URI}:{IP/CLUSTER}:{key}
    return GlobalConstants.RATE_LIMIT_KEY + requestURI + ":" + ipOrCluster + ":" + key;
}
💡 @Around vs @Before：RateLimiter 用 @Before 而非 @Around，因为它只需要在方法执行前判断是否放行——放行则继续，不放行则抛异常。不需要获取返回值、不需要修改参数、不需要计时。选择最简单的通知类型符合最小权限原则。

四、RepeatSubmitAspect：三阶段通知 + ThreadLocal

打开 ruoyi-common-redis/.../aspectj/RepeatSubmitAspect.java。这个切面使用了三种通知类型，并通过 ThreadLocal 在三个阶段之间传递状态：

// RepeatSubmitAspect.java 第 41 行
private static final ThreadLocal<String> KEY_CACHE = new ThreadLocal<>();

// 阶段一：@Before — 请求进入，检查是否重复（第 49-78 行）
@Before("@annotation(repeatSubmit)")
public void doBefore(JoinPoint point, RepeatSubmit repeatSubmit) {
    String cacheRepeatKey = REPEAT_SUBMIT_KEY + url + SecureUtil.md5(submitKey + ":" + params);
    if (RedisUtils.setObjectIfAbsent(cacheRepeatKey, "", Duration.ofMillis(interval))) {
        KEY_CACHE.set(cacheRepeatKey);  // 存入 ThreadLocal
    } else {
        throw new ServiceException("请勿重复提交");
    }
}

// 阶段二：@AfterReturning — 成功时保留 Redis key（第 85-98 行）
@AfterReturning(pointcut = "@annotation(repeatSubmit)", returning = "jsonResult")
public void doAfterReturning(JoinPoint joinPoint, RepeatSubmit repeatSubmit, Object jsonResult) {
    if (jsonResult instanceof R<?> r && r.getCode() == HttpStatus.SUCCESS) {
        return;  // 成功不删除，保证在有效时间内无法重复提交
    }
    deleteRepeatKey();  // 失败则删除，允许重试
}

// 阶段三：@AfterThrowing — 异常时删除 Redis key（第 106-113 行）
@AfterThrowing(value = "@annotation(repeatSubmit)", throwing = "e")
public void doAfterThrowing(JoinPoint joinPoint, RepeatSubmit repeatSubmit, Exception e) {
    deleteRepeatKey();  // 异常时释放锁，允许重试
}
┌──────────────┐ ┌──────────────────────┐ ┌──────────────────────┐ │ @Before │ │ @AfterReturning │ │ @AfterThrowing │ │ 写入 Redis │───>│ 成功→保留 key │ │ 删除 Redis key │ │ key 存入 TL │ │ 失败→删除 key │ │ 允许重新提交 │ └──────────────┘ └──────────────────────┘ └──────────────────────┘ │ │ │ └──────────────────────┴──────────────────────────┘ KEY_CACHE (ThreadLocal) 传递 key
⚠️ 为什么用 ThreadLocal 而非方法参数？ 三个通知方法在不同的时间点执行，无法直接传递参数——@Before 在 KEY_CACHE 中写入 key，@AfterReturning 或 @AfterThrowing 需要读取同一个 key 来决定是否删除 Redis 记录。ThreadLocal 确保了同一请求线程内数据隔离且可见。

六、AOP 代理 vs Filter 链 vs Interceptor 链

这三种机制都能在请求处理流程中插入横切逻辑，但粒度不同：

维度Servlet FilterSpring InterceptorSpring AOP 代理
拦截粒度URL 级别（HttpServletRequest）Controller 方法前后任意 Spring Bean 方法
能否获取方法参数不能（只能拿到原始 request）能（HandlerMethod）能（JoinPoint.getArgs()）
能否获取返回值不能能（postHandle）能（@AfterReturning / @Around）
能否获取注解不能可以拿到方法注解annotation 绑定，最直接
适用场景CORS、XSS 过滤、请求体缓存登录校验、权限校验、日志操作日志、限流、防重、数据权限、加解密
💡 选择策略：如果逻辑只和 HTTP 请求相关（如 XSS 过滤）→ 用 Filter；如果逻辑是 Controller 级别的通用检查（如登录状态）→ 用 Interceptor；如果逻辑和具体业务方法绑定（如加 @Log 注解）、需要灵活切入任意 Bean 方法 → 用 AOP。

七、Spring AOP 底层：JDK 动态代理 vs CGLIB

Spring Boot 3.x 默认使用 CGLIB 代理（基于字节码生成子类），而非 JDK 动态代理（基于接口）。原因：

// Spring Boot 3.x 配置（默认）
spring.aop.proxy-target-class=true  // 默认 CGLIB

// JDK 动态代理：要求目标类实现接口
UserService proxy = (UserService) Proxy.newProxyInstance(
    classLoader, new Class[]{UserService.class}, invocationHandler);

// CGLIB：通过继承生成子类，不要求实现接口
SysUserServiceImpl$$SpringCGLIB$$0 extends SysUserServiceImpl {
    @Override
    public UserDTO selectUserById(Long id) {
        // 先执行切面逻辑...
        return super.selectUserById(id);
    }
}
⚠️ CGLIB 的限制：不能代理 final 类（无法继承）和 final / private 方法（无法重写）。这就是为什么 AOP 切面必须作用于 public 方法，且 Spring 管理的 Bean 不能是 final 类。

3. 关于 AOP 代理、Interceptor 和 Filter，以下说法正确的是？

📖 点击查看答案

1. B — 记录耗时需要在方法执行前后都插入逻辑，这是 @Around 的专长（有 ProceedingJoinPoint，可以显式调用 proceed()）。@Before 只能在方法执行前执行，无法获取返回值和耗时。
2. B — doBefore 写入 key，doAfterReturning 和 doAfterThrowing 读取同一个 key 来决定是否删除 Redis 中的防重记录。三个方法在不同时刻执行，ThreadLocal 是线程安全的桥梁。
3. B — Filter 只拿到原始 HttpServletRequest，无法获取方法参数和返回值（A 错）。Interceptor 只拦截 Controller 方法，AOP 代理可以切入任何 Spring Bean 的方法（C 错）。
