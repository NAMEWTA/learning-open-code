第 16 课 · 前端动态路由工厂 — 后端菜单到 Vue Router 的编译过程 工厂 + 策略 + 组合

🎯 本节目标：理解 filterAsyncRouter() 如何将后端返回的菜单 JSON 编译为 Vue Router 路由表，掌握 Vite import.meta.glob 构建静态视图查找表的原理，能解释三种路由集合（sidebarRoutes、rewriteRoutes、defaultRoutes）各自的服务对象。

RuoYi-Vue-Plus 的后端菜单完全由数据库驱动——管理员在后台配置菜单树，前端在用户登录后动态拉取并编译为 Vue Router 路由。整个过程使用了工厂模式（route 对象生产）、策略模式（组件匹配策略）和组合模式（树形递归处理），三者协同工作。

二、Vite Glob Import：编译时静态分析

打开 plus-ui-vue/src/store/modules/permission.ts 第 14-21 行：

// 匹配 views 里面所有的 .vue 文件，预建查找表避免每次 O(n) 扫描
const modules = import.meta.glob('./../../views/**/*.vue');
const viewModuleMap = new Map<string, () => Promise<any>>();
for (const path in modules) {
  const viewsIndex = path.indexOf('/views/');
  if (viewsIndex === -1) continue;
  const dir = path.substring(viewsIndex + 7, path.lastIndexOf('.vue'));
  viewModuleMap.set(dir, modules[path] as () => Promise<any>);
}
1 编译时扫描：import.meta.glob 是 Vite 的编译时特性，在 vite build 阶段扫描 src/views/ 下所有 .vue 文件，生成一个路径→懒加载函数的静态映射。不是运行时 fs.readdir，而是构建工具在打包时替你完成的静态分析。 permission.ts:14
2 路径归一化：glob 返回的路径是相对于当前文件的——例如 ./../../views/system/user/index.vue。代码裁剪掉 /views/ 前缀，得到 system/user/index 作为 Map 的 key。这个 key 必须与后端菜单表中的 component 字段完全一致。 permission.ts:18-20
3 预建查找表：使用 Map<string, loader> 而非每次 O(n) 扫描 modules 对象。Map 的 get() 是 O(1) 哈希查找，在路由量大的场景下有可感知的性能差异。 permission.ts:15
💡 为什么不用动态 import()？ 动态 import(variable) 在 Vite 中会导致打包器无法做静态分析——它不知道你要导入哪些文件，因此无法做 tree-shaking 和代码分割。而 import.meta.glob 在编译时确定了所有可能的导入路径，Vite 可以为每个视图文件生成独立的 chunk，实现按路由懒加载。

三、filterAsyncRouter：递归编译引擎

这是整个动态路由系统的核心，第 83-109 行：

const filterAsyncRouter = (
  asyncRouterMap: RouteRecordRaw[],
  lastRouter?: RouteRecordRaw,
  type = false
): RouteRecordRaw[] => {
  return asyncRouterMap.filter(route => {
    // 步骤 A：重写模式下展平 ParentView 子路由
    if (type && route.children) {
      route.children = filterChildren(route.children, undefined);
    }

    // 步骤 B：组件类型分发（策略模式）
    if (route.component?.toString() === 'Layout') {
      route.component = Layout;                        // 布局壳
    } else if (route.component?.toString() === 'ParentView') {
      route.component = ParentView;                    // 路由视图包装器
    } else if (route.component?.toString() === 'InnerLink') {
      route.component = InnerLink;                     // iframe 内嵌
    } else {
      route.component = loadView(route.component, route.name as string);
    }                                                     // 普通视图

    // 步骤 C：递归处理子路由
    if (route.children != null && route.children && route.children.length) {
      route.children = filterAsyncRouter(route.children, route, type);
    } else {
      delete route.children;
      delete route.redirect;
    }
    return true;
  });
};

3.1 组件类型分发——策略模式

后端 component 字段值前端映射的组件用途
"Layout"@/layout/index.vue主布局壳（侧边栏 + 顶栏 + 内容区）
"ParentView"@/components/ParentView/index.vuerouter-view 包装器，用于纯目录菜单
"InnerLink"@/layout/components/InnerLink/index.vueiframe 内嵌外部页面
其他字符串（如 "system/user/index"）loadView() 动态导入普通业务页面

三个特殊组件是编译时静态导入（文件顶部 import Layout from '@/layout/index.vue'），会打进主 bundle。普通业务视图是懒加载，只在访问对应路由时才加载 chunk。

3.2 loadView：查找表 + HOC 包装

export const loadView = (view: any, name: string) => {
  const loader = viewModuleMap.get(view);    // O(1) 哈希查找
  if (loader) {
    return createCustomNameComponent(loader, { name });  // HOC 赋名
  }
  return undefined;
};

createCustomNameComponent 是一个高阶组件（HOC）工厂，它的作用是给懒加载的组件显式分配 name——这对 <keep-alive> 缓存至关重要。

// plus-ui-vue/src/utils/createCustomNameComponent.tsx
export function createCustomNameComponent(loader, options = {}) {
  const { name } = options;
  let component = null;
  let wrappedComponent = null;

  return async () => {
    if (!component) {
      const { default: loadedComponent } = await loader();
      component = loadedComponent;
    }
    if (!wrappedComponent) {
      wrappedComponent = defineComponent({
        name,                 // ★ 显式分配组件名，供 keep-alive 的 include 使用
        render() {
          return h(component);
        }
      });
    }
    return Promise.resolve(wrappedComponent);
  };
}
⚠️ 为什么要这层包装？ Vite 使用 import.meta.glob 懒加载的组件，其 name 属性由 SFC 的 <script> 块定义。但动态路由中组件的 name 可能未定义或与路由 name 不一致。而 <keep-alive :include="cachedViews"> 依赖组件 name 做缓存匹配。如果组件没有正确的 name，keep-alive 就无法正常工作。createCustomNameComponent 用一层薄的 HOC 解决了这个问题。

四、三种路由集合：一个 JSON，三种用途

generateRoutes() 方法内部对同一份后端菜单 JSON 做了三次 structuredClone，编译出三套路由：

const generateRoutes = async (): Promise<RouteRecordRaw[]> => {
  const res = await getRouters();
  const data = Array.isArray(res.data) ? res.data : [];
  const sdata = structuredClone(data);    // 副本 1 → 侧边栏
  const rdata = structuredClone(data);    // 副本 2 → 路由注册
  const defaultData = structuredClone(data); // 副本 3 → 顶部栏

  const sidebarRoutes = filterAsyncRouter(sdata);          // 原始树结构
  const rewriteRoutes = filterAsyncRouter(rdata, undefined, true); // 展平 ParentView
  const defaultRoutes = filterAsyncRouter(defaultData);    // 原始树结构

  // ...
  setRoutes(rewriteRoutes);                      // 用于 addRoute() 注册
  setSidebarRouters(constantRoutes.concat(sidebarRoutes));  // 用于侧边栏渲染
  setDefaultRoutes(sidebarRoutes);               // 原始路由（可被顶部栏读取）
  setTopbarRoutes(defaultRoutes);                // 用于顶部栏渲染

  duplicateRouteChecker(asyncRoutes, sidebarRoutes);       // 重复校验
  return rewriteRoutes;
};
后端 JSON (同一份) ──structuredClone──> 三份独立副本 副本 1 (sdata) ──filterAsyncRouter(type=false)──> sidebarRoutes → 侧边栏菜单树 │ 副本 2 (rdata) ──filterAsyncRouter(type=true)───> rewriteRoutes → router.addRoute() │ 副本 3 (default) ──filterAsyncRouter(type=false)──> defaultRoutes → 顶部栏菜单树

filterChildren：ParentView 展平

当 type = true（重写模式）时，filterAsyncRouter 会调用 filterChildren 来展平 ParentView 节点：

const filterChildren = (childrenMap, lastRouter?) => {
  let children = [];
  childrenMap.forEach(el => {
    el.path = lastRouter ? lastRouter.path + '/' + el.path : el.path;
    if (el.children && el.children.length
        && el.component?.toString() === 'ParentView') {
      // ParentView 节点被跳过，其子节点上提一级
      children = children.concat(filterChildren(el.children, el));
    } else {
      children.push(el);
    }
  });
  return children;
};

这是因为 ParentView 只是一个 <router-view> 包装器，在路由注册时不需要作为独立路由注册。展平后，其子路由直接注册到父路由下，避免多一层不必要的路由嵌套。

💡 为什么需要 structuredClone？ 三套路由在编译过程中会被修改（如 route.component 从字符串变为函数，route.children 被删除或重组）。如果共用一个对象引用，修改会相互污染。使用 structuredClone()（ES2021 标准）而非 JSON.parse(JSON.stringify()) 的好处是：(1) 支持更多类型（Date、RegExp、Map、Set）；(2) 性能更好；(3) 语义更清晰。

五、duplicateRouteChecker：防御性校验

function duplicateRouteChecker(localRoutes, routes) {
  function flatRoutes(routes) {
    const res = [];
    routes.forEach(route => {
      if (route.children) {
        res.push(...flatRoutes(route.children));
      } else {
        res.push(route);
      }
    });
    return res;
  }

  const allRoutes = flatRoutes([...localRoutes, ...routes]);
  const nameList = [];
  allRoutes.forEach(route => {
    const name = route.name?.toString() ?? '';
    if (nameList.includes(name)) {
      console.error(`路由名称: [${name}] 重复, 会造成 404`);
      ElNotification({ title: '路由名称重复', message: `路由名称: [${name}] 重复`, type: 'error' });
      return;
    }
    nameList.push(name);
  });
}

在 Vue Router 中，同一名称的路由后注册的会覆盖先注册的，导致旧路由 404。这个检查器将本地路由和后端路由合并展平，检测重复的 name 字段。一旦发现重复，通过 ElNotification 在界面右上角弹出红色错误通知，开发者在开发阶段就能立即感知。

这套动态路由系统的关键在于前后端关于 component 字段的契约：

七、自测

1. import.meta.glob 是运行时扫描还是编译时扫描？

2. createCustomNameComponent 包装懒加载组件的主要目的是什么？

3. 为什么 generateRoutes() 需要三次 structuredClone？

📖 点击查看答案

1. B — import.meta.glob 是 Vite 的编译时特性。它在构建阶段扫描指定目录，生成静态的路径→动态导入函数映射，打包进 bundle。浏览器运行时不会也不应该访问文件系统。
2. B — Vue 的 keep-alive 通过组件 name 做缓存匹配。Vite glob import 的懒加载组件可能没有定义 name，或 name 与路由 name 不一致，导致缓存失效。createCustomNameComponent 用 HOC 显式分配 name。
3. B — filterAsyncRouter 会修改路由对象（component 从字符串变为函数、children 被删除/重组）。rewriteRoutes 还要经过 filterChildren 展平。三份副本各自独立修改，互不干扰。
