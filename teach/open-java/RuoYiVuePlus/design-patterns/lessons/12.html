第 12 课 · 依赖倒置 Dependency Inversion

API 合约 — 模块解耦的核心机制

本课目标：理解依赖倒置原则（DIP）在 RuoYi-Vue-Plus 中的落地实现 —— 如何通过 ruoyi-api 接口层打破模块间的直接耦合，实现独立编译、独立开发和按需注入。

1. 问题的起点：两个模块如何通信

经典场景：工作流引擎需要查询"某个审批人的姓名"。审批人的数据在 ruoyi-system，流程引擎在 ruoyi-workflow。如何让它们通信？

如果采用直接依赖的方案：

❌ 错误的直接依赖： ruoyi-workflow ──Maven 依赖──→ ruoyi-system FlwTaskServiceImpl 直接 @Autowired SysUserServiceImpl

这种方案带来三个严重问题：

  1. 循环依赖：如果 ruoyi-system 也需要调用 ruoyi-workflow（例如创建用户时触发工作流），就会形成循环
  2. 编译耦合：修改 ruoyi-system 的任意代码，ruoyi-workflow 必须重新编译，即使改动与 workflow 无关
  3. 无法独立部署：两个模块绑定在一起，无法按需选择部署范围
依赖倒置原则（DIP）的核心思想：高层模块不应该依赖低层模块，二者都应该依赖抽象。抽象不应该依赖细节，细节应该依赖抽象。

2. RuoYi-Vue-Plus 的解决方案：ruoyi-api 合约层

架构师在 ruoyi-api 模块中定义了接口合约。两个业务模块都依赖这个合约，而不是互相依赖：

✅ 正确的依赖倒置： ruoyi-api（抽象合约层 — 只依赖 ruoyi-common-core） / ↑ ↑ \ / | | \ implements| |injects / | | \ v | | v ruoyi-system ruoyi-workflow (SysUserServiceImpl (FlwTaskServiceImpl implements UserService) 注入 UserService 接口)

关键事实：ruoyi-system 和 ruoyi-workflow 的 pom.xml 中都没有对方的坐标。它们只依赖 ruoyi-api。

3. 合约定义：UserService 接口

1 ruoyi-api/src/main/java/org/dromara/system/api/UserService.java 
public interface UserService {
    /** 根据用户ID查询用户名 */
    String selectUserNameById(Long userId);

    /** 根据用户ID查询用户昵称 */
    String selectNicknameById(Long userId);

    /** 根据用户ID批量查询用户昵称 */
    String selectNicknameByIds(String userIds);

    /** 根据用户ID查询手机号 */
    String selectPhonenumberById(Long userId);

    /** 根据用户ID查询邮箱 */
    String selectEmailById(Long userId);

    /** 根据用户ID查询用户完整信息（返回 DTO） */
    UserDTO selectById(Long userId);

    /** 批量查询用户列表 */
    List<UserDTO> selectListByIds(Collection<Long> userIds);

    /** 根据角色ID查询用户列表 */
    List<Long> selectUserIdsByRoleIds(Collection<Long> roleIds);

    /** 根据角色ID查询用户详情（含昵称） */
    List<UserDTO> selectUsersByRoleIds(Collection<Long> roleIds);

    /** 根据部门ID查询用户列表 */
    List<UserDTO> selectUsersByDeptIds(Collection<Long> deptIds);

    /** 根据岗位ID查询用户列表 */
    List<UserDTO> selectUsersByPostIds(Collection<Long> postIds);

    /** 批量查询用户昵称 Map */
    Map<Long, String> selectUserNicksByIds(Collection<Long> userIds);
}
设计约束：合约接口只暴露只读查询方法，不包含增删改。增删改操作属于模块内部事务，不应跨模块暴露。合约是"信息共享"通道，不是"数据写入"通道。写入操作应通过领域事件或消息队列进行异步通信。

3.1 合约中的 DTO

合约接口返回的是轻量级 DTO，不是内部实体或 VO：

2 ruoyi-api/src/main/java/org/dromara/system/api/domain/UserDTO.java 
@Data
@NoArgsConstructor
public class UserDTO implements Serializable {
    private Long userId;
    private Long deptId;
    private String userName;
    private String nickName;
    private String userType;
    private String email;
    private String phonenumber;
    private String sex;
    private String status;
    private Date createTime;
}
为什么用 DTO 而不是直接返回 SysUser？如果合约返回 SysUser 实体对象，消费者（ruoyi-workflow）就会间接依赖 SysUser 的所有字段和方法，包括 @TableField、@EncryptField 等 MyBatis-Plus 注解。DTO 是纯粹的数据载体，只包含消费者真正需要的字段，屏蔽了内部实现细节。

4. 实现方：SysUserServiceImpl 双接口实现

3 ruoyi-modules/ruoyi-system/src/main/java/org/dromara/system/service/impl/SysUserServiceImpl.java 
@Service
public class SysUserServiceImpl implements ISysUserService, UserService {

    // ===== 内部服务接口方法（ISysUserService）=====
    @Override
    public PageResult<SysUserVo> selectPageUserList(SysUserBo user, PageQuery pageQuery) {
        // 分页查询、权限过滤、部门数据权限...
    }

    // ===== 跨模块合约方法（UserService）=====
    @Override
    @Cacheable(cacheNames = CacheNames.SYS_USER_NAME, key = "#userId")
    public String selectUserNameById(Long userId) {
        SysUserVo user = baseMapper.selectVoById(userId);
        return user != null ? user.getUserName() : null;
    }

    @Override
    public UserDTO selectById(Long userId) {
        SysUserVo vo = baseMapper.selectVoById(userId);
        return BeanUtil.copyProperties(vo, UserDTO.class);
        // 将内部 VO 转换为合约 DTO
    }
}

同一个类实现两个接口：对内部暴露完整的增删改查（ISysUserService），对外部只暴露只读查询（UserService）。Spring 在运行时通过 UserService 类型注入时，只会暴露合约方法。

6. 领域事件 — 另一种模块通信方式

除了接口合约，ruoyi-api 还定义了领域事件，用于异步解耦的场景：

5 ruoyi-api/.../workflow/api/event/ProcessEvent.java 
@Data
public class ProcessEvent implements Serializable {
    private String flowCode;       // 流程编码
    private Long instanceId;        // 实例 ID
    private String businessId;      // 业务 ID
    private Integer nodeType;       // 节点类型: 0=开始, 1=中间, 2=结束
    private String nodeCode;        // 节点编码
    private String nodeName;        // 节点名称
    private String status;          // 状态
    private Map<String, Object> params;
    private Boolean submit;         // 是否发起人提交
}

任何模块都可以通过 Spring 的 @EventListener 监听这些事件：

// ruoyi-workflow/.../TestLeaveServiceImpl.java
@EventListener(condition = "#processEvent.flowCode.startsWith('leave')")
public void processHandler(ProcessEvent processEvent) {
    // 当请假流程完成时，自动更新考勤状态
}

@EventListener(condition = "#processDeleteEvent.flowCode.startsWith('leave')")
public void processDeleteHandler(ProcessDeleteEvent processDeleteEvent) {
    // 当请假流程被删除时，恢复原始状态
}
接口合约 vs 领域事件：接口合约用于同步查询（workflow 需要立即知道用户名）；领域事件用于异步通知（流程完成后通知其他模块更新状态）。两者互补，共同构成了模块间通信的完整方案。

7. 编译时 vs 运行时

依赖倒置的关键在于编译时依赖和运行时依赖的方向相反：

编译时依赖方向： ruoyi-workflow ──→ ruoyi-api ←── ruoyi-system （都依赖抽象） （都依赖抽象） 运行时依赖方向（Spring DI）： ruoyi-workflow ──→ UserService 接口 ←── SysUserServiceImpl （注入接口） （Spring 注入实现类） 真正的实现方（运行时）： SysUserServiceImpl (在 ruoyi-system.jar 中) ↑ | Spring 容器自动装配 | FlwTaskServiceImpl (在 ruoyi-workflow.jar 中) private final UserService userService; // 编译时只知道接口！
阶段ruoyi-workflow 知道的不知道的

编译时ruoyi-api.jar 中的 UserService.classruoyi-system.jar
运行时UserService 类型的 Bean具体是哪个实现类（由 Spring 决定）

8. 收益与代价

8.1 三个核心收益

  1. 独立编译：修改 SysUserServiceImpl 后，ruoyi-workflow 不需要重新编译
  2. 无循环依赖：ruoyi-system 也可以依赖 ruoyi-api 中的其他接口
  3. 独立团队开发：A 团队开发 system 模块，B 团队开发 workflow 模块，只要合约不变，就可以独立迭代

8.2 必须接受的代价

DTO ↔ VO 转换开销：SysUserVo（内部 VO）和 UserDTO（合约 DTO）之间的字段映射需要额外的转换代码（如 BeanUtil.copyProperties）。当字段数量多时，这是可感知的性能开销。但如果直接暴露 VO，就会引入耦合。这是架构层次的有意识权衡。

9. ruoyi-api 的全貌

ruoyi-api 包含两个子包，共计 20 个 Java 文件：

子系统接口DTO / Model事件
system.api UserService, RoleService, DeptService, PostService, ConfigService, MessageService, OssService, TaskAssigneeService UserDTO, RoleDTO, DeptDTO, PostDTO, OssDTO, PushPayloadDTO, TaskAssigneeDTO, UserOnlineDTO + LoginUser, RegisterBody 等 9 个 Model —

统计：ruoyi-api 只依赖 ruoyi-common-core，不依赖任何 common 能力模块。20 个 Java 文件中 8 个是接口、5 个是 DTO、3 个是事件、9 个是 Login 相关 Model。没有一行 @Service 或 @Component 注解 —— 纯粹的定义层。

自测题

Q1：如果 ruoyi-workflow 需要获取用户的邮箱地址，但 UserService 接口没有提供 selectEmailById() 方法，应该怎么处理？

查看答案

应该在 UserService 接口中新增方法，然后在 SysUserServiceImpl 中实现它。不能直接从 ruoyi-workflow 中注入 ISysUserService，因为这会打破模块隔离，引入对 ruoyi-system 的直接依赖。合约的扩展成本很低：改接口、加实现、发布新的 ruoyi-api.jar。

Q2：编译时 ruoyi-workflow 只知道 UserService 接口。Spring 在运行时如何找到正确的实现类（SysUserServiceImpl）？

查看答案

通过 Spring 的依赖注入（DI）机制：SysUserServiceImpl 标注了 @Service 并 implements UserService，因此 Spring 容器会将它注册为一个 UserService 类型的 Bean。当 ruoyi-workflow 中声明 private final UserService userService 并使用 @RequiredArgsConstructor 时，Spring 自动将容器中唯一实现了 UserService 的 Bean（即 SysUserServiceImpl）注入进去。

Q3：领域事件（ProcessEvent）和接口合约（UserService）解决了不同的问题。什么场景下应该用事件，什么场景下应该用接口合约？

查看答案

接口合约用于同步请求-响应场景：工作流引擎需要立即知道审批人的姓名才能继续执行。这是不能延迟的。
领域事件用于异步通知场景：流程审批完成后，需要更新考勤状态。考勤更新不需要实时，可以稍后处理；而且考勤模块作为事件消费者被解耦，流程模块不需要知道考勤的存在。
判断标准：如果调用方必须等待结果才能继续 → 接口合约；如果只是通知"发生了某事"而不需要返回值 → 领域事件。
