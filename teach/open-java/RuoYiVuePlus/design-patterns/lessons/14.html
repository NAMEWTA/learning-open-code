第 14 课 · 前端组合式函数与状态管理 — Composable + Pinia Setup Stores 架构模式

🎯 本节目标：掌握 RuoYi-Vue-Plus 前端的 9 个 Composable 与 7 个 Pinia Setup Store 的组织方式，理解 Pinia store 等价于 Singleton + Observer + Repository 三合一模式，能用 useFormDialog 编写组合式弹窗逻辑。

二、useTableSelection — 泛型行选择

打开 plus-ui-vue/src/hooks/table/useTableSelection.ts：

export function useTableSelection<T, ID extends string | number = string | number>(
  getRowId: (row: T) => ID
) {
  const ids = ref<ID[]>([]);
  const selectedRows = ref<T[]>([]);
  const single = ref(true);   // 禁用按钮：未选或选多时
  const multiple = ref(true); // 禁用按钮：未选时

  const handleSelectionChange = (selection: T[]) => {
    selectedRows.value = selection;
    ids.value = selection.map(getRowId).filter(isSelectableId) as ID[];
    single.value = selection.length !== 1;
    multiple.value = selection.length === 0;
  };

  const clearSelection = () => {
    selectedRows.value = [];
    ids.value = [];
    single.value = true;
    multiple.value = true;
  };

  return { ids, selectedRows, single, multiple,
           handleSelectionChange, clearSelection };
}
1 泛型约束：<T, ID extends string | number> 使得这个 composable 适用于任何表格数据。ID 类型限定为 string 或 number，避免 object 类型作 key。 useTableSelection.ts:5
2 策略注入：getRowId 回调由调用方提供，实现了策略模式——"如何从行数据中提取 ID"这个策略由外部定义。不同表格可以有不同的 ID 字段（userId、id、code），composable 本身不关心。 useTableSelection.ts:5
3 派生状态而非计算：single 和 multiple 直接用 ref 存布尔值并在 handleSelectionChange 中同步更新，而非用 computed。这样做的好处是状态在 ElTable 的 selection-change 事件触发时一次性更新，Giảm 不必要的响应式追踪。 useTableSelection.ts:4-5

三、useLoading — 高阶异步包装器（策略模式 + 模板方法）

打开 plus-ui-vue/src/hooks/async/useLoading.ts：

export function useLoading(initialValue = false) {
  const loading = ref(initialValue);

  const setLoading = (value: boolean) => {
    loading.value = value;
  };

  const withLoading = async <T>(task: () => Promise<T>) => {
    loading.value = true;
    try {
      return await task();
    } finally {
      loading.value = false;   // ← 无论如何都会关闭 loading
    }
  };

  return { loading, setLoading, withLoading };
}
💡 模式识别：withLoading 是模板方法模式的函数式等价物。模板方法在 OOP 中是"定义算法骨架，子类实现具体步骤"；这里骨架是 set true → execute task → set false，具体步骤（task）以回调注入。同时它也是一个高阶函数（接收函数、返回 Promise），与 Python 的 @contextmanager 装饰器、Rust 的 with_scope 理念一致。

在组件中使用：

const { loading, withLoading } = useLoading();

const fetchData = async () => {
  const result = await withLoading(() => api.getUsers(params));
  data.value = result;
};
// loading.value 在 api 调用期间自动为 true，调用结束后自动为 false

四、useFormDialog — 组合模式（Composable Composition = Decorator Pattern）

打开 plus-ui-vue/src/hooks/dialog/useFormDialog.ts：

export function useFormDialog<T extends Record<string, any>>(
  options: UseFormDialogOptions<T>
) {
  const { form, formRef, initialFormData, initialTitle } = options;
  // ★ 组合 useDialogState——装饰器模式的内核
  const { dialog, openDialog, closeDialog, setTitle } = useDialogState(initialTitle);

  const resetForm = () => {
    form.value = { ...initialFormData };   // 浅拷贝重置
    formRef?.value?.resetFields();          // Element Plus 表单组件状态重置
    formRef?.value?.clearValidate?.();      // 清除校验提示
  };

  const openFormDialog = (title?: string) => {
    resetForm();     // ★ 在打开弹窗前自动重置表单
    openDialog(title);
  };

  return {
    dialog,
    resetForm,
    openDialog: openFormDialog,   // 覆盖基 composable 的 openDialog
    showDialog,                   // 不重置的打开方式
    closeDialog: closeFormDialog,
    setTitle
  };
}
┌──────────────────────────────────────┐ │ useDialogState │ ← 基础 composable │ dialog (visible + title) │ │ openDialog / closeDialog / setTitle │ └────────────┬──────────────────────────┘ │ 组合 (内部调用) ▼ ┌──────────────────────────────────────┐ │ useFormDialog │ ← 增强 composable │ dialog (继承自 useDialogState) │ │ openDialog → 先 resetForm 再 open │ ★ 装饰：在原有逻辑前插入表单重置 │ showDialog → 不重置的备用入口 │ │ resetForm / closeDialog │ └──────────────────────────────────────┘

这就是 Vue 3 生态中装饰器模式的实现方式——不用类继承，不用 @decorator 语法，而是通过一个 composable 内部调用另一个 composable，在返回前增强其行为。

5.1 为什么 Pinia Store 是「三合一」？

// 以 useUserStore 为例
export const useUserStore = defineStore('user', () => {
  // ── Singleton（单例）──
  // Pinia 内部保证同一 ID 的 store 全局唯一，多次调用 useUserStore() 返回同一实例
  const token = ref(getToken());

  // ── Observer（观察者）──
  // ref() 创建的响应式数据被 Vue 的 reactivity system 代理
  // 任何组件读取 token.value，自动建立依赖追踪
  // token.value 变化时，所有依赖它的组件自动重渲染
  const roles = ref<Array<string>>([]);
  const permissions = ref<Array<string>>([]);

  // ── Repository（仓库）──
  // store 的方法封装了数据访问逻辑，组件不直接调 API
  const login = async (userInfo: LoginData) => {
    const [err, res] = await to(loginApi(userInfo));
    if (res) {
      setToken(data.access_token);
      token.value = data.access_token;  // 自动触发 Observer 通知
    }
  };

  return { token, roles, permissions, login, getInfo, logout };
});

5.2 useDictStore — Map 响应式字典缓存

export const useDictStore = defineStore('dict', () => {
  const dict = ref<Map<string, DictDataOption[]>>(new Map());

  const getDict = (_key: string): DictDataOption[] | null => {
    if (!_key) return null;
    return dict.value.get(_key) || null;   // 被动缓存：有则返回，无则返回 null
  };

  const setDict = (_key: string, _value: DictDataOption[]) => {
    if (!_key) return false;
    dict.value.set(_key, _value);
    return true;
  };

  const removeDict = (_key: string): boolean => {
    if (!_key) return false;
    return dict.value.delete(_key);
  };

  const cleanDict = (): void => {
    dict.value.clear();   // 页面卸载时清理
  };

  return { dict, getDict, setDict, removeDict, cleanDict };
});

注意 dict 是一个 ref<Map<>>，不是 reactive。这是因为 Map 作为 ES6 集合类型，在 Vue 3.3+ 中 ref 对其有深层响应式支持，但 reactive 在早期版本对 Map/Set 的支持不够稳定。

六、Setup Store vs Options Store

对比维度Setup Store（本项目的选择）Options Store
语法风格函数式，返回对象配置对象，state/getters/actions
组合能力可在函数内直接使用其他 composable无法在定义体内组合
类型推导完整推导，无需额外类型标注需要泛型参数辅助
this 语义无 this，闭包捕获this 指向 store 实例
学习曲线需掌握 Composition API类 Options API，熟悉 Vue 2 则易上手
💡 为什么不混用？ 项目统一采用 Setup Store，确保了代码风格一致。如果混用两种风格，团队维护成本会线性上升——你需要同时理解两套心智模型。这是编码规范中的「约定优于配置」原则。
