第 19 课 · 统一响应与异常体系

一个 API 项目最基础的工程决策是什么？答案是"所有接口返回什么格式"。RuoYi-Vue-Plus 用 R<T> + PageResult<T> + BaseException + GlobalExceptionHandler 四件套构建了完整的"统一响应-异常"闭环，前端不需要猜测每个接口的返回结构。

本课目标：阅读本课后，你能解释 R、PageResult、BaseException、GlobalExceptionHandler 四者如何协同工作，知道如何新增一个自定义异常类型，并理解为什么项目废弃了 AjaxResult 和 TableDataInfo。

1. R<T> — 统一响应包装器

定义文件：
RuoYi-Vue-Plus/ruoyi-common/ruoyi-common-core/src/main/java/org/dromara/common/core/domain/R.java

核心结构仅三个字段：

@Data
@NoArgsConstructor
public class R<T> implements Serializable {
    private int code;      // 状态码
    private String msg;    // 提示信息
    private T data;        // 业务数据
}

静态工厂方法

方法codedata用途
R.ok()200null操作成功，无返回数据
R.ok(T data)200具体数据操作成功，返回业务数据
R.ok(String msg, T data)200具体数据成功 + 自定义提示 + 数据
R.fail()500null操作失败
R.fail(String msg)500null失败 + 自定义错误信息
R.fail(int code, String msg)自定义null失败 + 自定义状态码 + 信息
R.warn(String msg)601null警告提示（非阻塞性）
状态码语义：SUCCESS = 200 成功、ERROR = 500 失败、WARN = 601 警告（需要用户确认但非错误）。前端根据 code 决定展示 toast、弹窗还是静默通过。状态码常量定义在 ruoyi-common-core/.../constant/HttpStatus.java。

2. PageResult<T> — 分页数据对象

定义文件：
RuoYi-Vue-Plus/ruoyi-common/ruoyi-common-core/src/main/java/org/dromara/common/core/domain/PageResult.java

@Data
@NoArgsConstructor
public class PageResult<T> implements Serializable {
    private long total;             // 总记录数
    private Collection<T> rows;     // 列表数据

    // 静态工厂
    public static <T> PageResult<T> build(Collection<T> list, long total) { ... }
    public static <T> PageResult<T> build(Collection<T> list) { ... }
    public static <T> PageResult<T> build() { ... }
}
重要变更：RuoYi 经典版使用了 AjaxResult 和 TableDataInfo 两个类分别处理普通响应和分页响应。RuoYi-Vue-Plus 统一为 R 和 PageResult，并用 PageResult.build() 的 emptyIfNull() 方法保证了 rows 永不为 null——前端可以直接 .forEach()，无需判空。

3. 异常体系

BaseException — 国际化感知的异常基类

定义文件：
RuoYi-Vue-Plus/ruoyi-common/ruoyi-common-core/src/main/java/org/dromara/common/core/exception/base/BaseException.java

使用模块名+错误码+参数组合的国际化模式：

// 异常消息由 module + code + args 组合解析出 i18n 文案
public class BaseException extends RuntimeException {
    private final String module;      // 模块名，如 "system"
    private final String code;        // 错误码
    private final Object[] args;      // 消息参数
    private final String defaultMessage; // 默认消息
}

ServiceException — 最终的业务异常

定义文件：
RuoYi-Vue-Plus/ruoyi-common/ruoyi-common-core/src/main/java/org/dromara/common/core/exception/ServiceException.java

final class 设计，不可被继承，简化了异常继承树：

public final class ServiceException extends RuntimeException {
    private final Integer code;     // 状态码
    private final String message;   // 错误消息
}

领域异常

异常类使用场景
UserException用户不存在、密码错误、账号锁定等
CaptchaException验证码错误或过期
FileException文件上传大小超限、格式不支持
SseExceptionSSE 推送异常（需要特殊处理，不能走普通 JSON 响应）

4. GlobalExceptionHandler — 全局异常拦截器

定义文件：
RuoYi-Vue-Plus/ruoyi-common/ruoyi-common-web/src/main/java/org/dromara/common/web/handler/GlobalExceptionHandler.java

注册为 @RestControllerAdvice，统一拦截 15+ 种异常类型：

异常类型返回码说明
HttpRequestMethodNotSupportedException405请求方法不支持（GET 调了 POST 接口）
ServiceException业务码业务异常，透传 code + message
SseException-SSE 异常走独立通道，不返回 JSON
BindException参数错误GET 请求参数绑定失败
MethodArgumentNotValidException参数错误POST 请求参数校验失败 (@Valid)
AccessDeniedException403无权限访问
HttpMessageNotReadableException请求错误请求体 JSON 格式错误
MultipartException文件错误文件上传超出大小限制

另外，各模块还注册了自己的异常处理器：

  • MybatisExceptionHandler — 捕获数据库唯一键冲突、SQL 异常
  • RedisExceptionHandler — 捕获 Redis 连接异常
  • SmsExceptionHandler — 捕获短信发送失败
  • SaTokenExceptionHandler — 捕获认证/鉴权异常

5. 前端对等实现

前端使用 AxiosPromise<T> 作为统一的异步返回包装器：

// api/types.ts - 全局类型
export interface PageResult<T> {
  total: number;
  rows: T[];
}

// api/system/user/types.ts - 业务类型
export interface UserVO {
  userId: number;
  userName: string;
  // ...
}
export interface UserQuery extends PageQuery {
  userName?: string;
  status?: string;
}
export interface UserForm {
  userName: string;
  // ...
}

// api/system/user/index.ts - API 函数
export const listUser = (query: UserQuery): AxiosPromise<PageResult<UserVO>> => {
  return request({ url: '/system/user/list', method: 'get', params: query });
};
前后端呼应：前端的 PageResult<UserVO> 与后端的 PageResult<SysUser> 结构完全对齐（total + rows）。前端请求拦截器自动解析 R.code 判断成功/失败/警告，业务代码只需关心 R.data 的内容。

自测题

问题 1：以下哪个是正确的后端控制器返回值写法？

A. return AjaxResult.success(userList); B. return R.ok(userList); C. return new ResponseEntity<>(userList, HttpStatus.OK); D. return userList;
查看答案

答案：B。RuoYi-Vue-Plus 使用 R<T> 而非经典版的 AjaxResult。选项 A 是旧版写法，C 是 Spring 原生方式（不符合统一响应规范），D 缺少状态码和消息字段。

问题 2：为什么 ServiceException 被声明为 final class？

查看答案

防止异常继承树过度膨胀。业务异常的表达靠 code + message 的组合，而不是品类繁多的异常子类。final 强制开发者使用组合而非继承来区分异常类型，这符合"一个异常类 + 枚举错误码"的简洁设计原则。

问题 3：GlobalExceptionHandler 使用 @RestControllerAdvice 而非 @ControllerAdvice，有什么区别？

A. 没有区别，只是命名风格不同 B. @RestControllerAdvice 同时包含 @ControllerAdvice 和 @ResponseBody C. @RestControllerAdvice 只能拦截 REST 请求
查看答案

答案：B。@RestControllerAdvice = @ControllerAdvice + @ResponseBody，确保异常处理方法的返回值（如 R.fail()）自动序列化为 JSON，而不是被视图解析器处理。对于纯 API 项目这是必须的。
