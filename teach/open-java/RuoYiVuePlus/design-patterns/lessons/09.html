第 9 课 · 责任链模式 — Servlet Filter + Interceptor + MyBatis Plugin 三层链 行为型模式

🎯 本节目标：理解 RuoYi-Vue-Plus 中三层责任链的架构——一个 HTTP 请求从进入服务器到 SQL 执行，经过了三道独立的拦截链。掌握每层链的职责边界和执行顺序。

责任链模式的核心：将多个处理器串成一条链，请求沿链传递，每个处理器可以选择处理、传递或终止。RuoYi-Vue-Plus 没有手动构建链表，而是利用了三个框架内置的责任链机制：Servlet Filter Chain、Spring Interceptor Chain、MyBatis Plugin Chain。

一个 HTTP 请求的完整旅程： [Layer 1: Servlet Filter Chain] XssFilter → RepeatableFilter → CryptoFilter │ │ │ │ │ └── 解密请求体 / 加密响应体 │ └── 包装可重复读取的 Request（JSON 场景） └── 包装 XSS 清洗的 Request ↓ DispatcherServlet 分发 [Layer 2: Spring Interceptor Chain] PlusWebInvokeTimeInterceptor → ...其他拦截器 │ └── preHandle 记日志+启动计时 / afterCompletion 输出耗时 ↓ Controller 执行 [Layer 3: MyBatis Plugin Chain] PlusDataPermissionInterceptor → SqlLogInterceptor → MybatisEncryptInterceptor → MybatisDecryptInterceptor │ └── 每个插件在 SQL 执行前后插入处理逻辑

一、Layer 1：Servlet Filter Chain（最外层）

Servlet Filter 工作在 DispatcherServlet 之前，是最早接触请求的组件。可以修改请求/响应对象，也可以直接拦截返回（终止链）。

1.1 XssFilter — 安全第一关

打开 ruoyi-common/ruoyi-common-web/src/main/java/org/dromara/common/web/filter/XssFilter.java：

public class XssFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest req = (HttpServletRequest) request;

        // 命中排除规则？直接放行
        if (handleExcludeURL(req, resp)) {
            chain.doFilter(request, response);  // ← 不处理，传给下一个 Filter
            return;
        }

        // 包装请求，清洗 XSS 脚本
        XssHttpServletRequestWrapper xssRequest =
            new XssHttpServletRequestWrapper((HttpServletRequest) request);
        chain.doFilter(xssRequest, response);  // ← 传递包装后的请求
    }
}
1 包装模式 + 责任链：XssFilter 不修改原始 Request，而是创建一个 XssHttpServletRequestWrapper 包装它，然后把包装对象传给下一个 Filter。后续环节拿到的都是已清洗的安全请求。 XssFilter.java:52-62
2 智能跳过：GET 和 DELETE 请求不经过 XSS 过滤（这些方法没有请求体），配置中的排除 URL 也直接放行。不是所有请求都需要走完整条链。 XssFilter.java:71-79

1.2 RepeatableFilter — 让请求体可以多次读取

打开 ruoyi-common/ruoyi-common-web/src/main/java/org/dromara/common/web/filter/RepeatableFilter.java：

public class RepeatableFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        ServletRequest requestWrapper = null;
        // 仅对 JSON 请求做包装
        if (request instanceof HttpServletRequest
            && StringUtils.startsWithIgnoreCase(
                request.getContentType(), MediaType.APPLICATION_JSON_VALUE)) {
            requestWrapper = new RepeatedlyRequestWrapper(
                (HttpServletRequest) request, response);
        }
        if (null == requestWrapper) {
            chain.doFilter(request, response);   // 非 JSON → 直接放行
        } else {
            chain.doFilter(requestWrapper, response);  // JSON → 包装后传递
        }
    }
}
3 解决核心痛点：HttpServletRequest 的 InputStream 默认只能读一次。后面的日志拦截器、加解密过滤器、Controller 都要读请求体 → 必须让流可重复读取。RepeatableFilter 将请求体缓存到内存，允许后续多次读取。 RepeatableFilter.java:38-49

1.3 CryptoFilter — 加解密关卡

打开 ruoyi-common/ruoyi-common-encrypt/src/main/java/org/dromara/common/encrypt/filter/CryptoFilter.java：

public class CryptoFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        // 获取 Controller 方法上的 @ApiEncrypt 注解
        ApiEncrypt apiEncrypt = this.getApiEncryptAnnotation(servletRequest);
        boolean responseFlag = apiEncrypt != null && apiEncrypt.response();

        // 请求解密：POST/PUT + 加密标头 → 解密请求体
        if (isPostOrPut(servletRequest)) {
            String headerValue = servletRequest.getHeader(properties.getHeaderFlag());
            if (StringUtils.isNotBlank(headerValue)) {
                requestWrapper = new DecryptRequestBodyWrapper(
                    servletRequest, properties.getPrivateKey(), properties.getHeaderFlag());
            } else if (apiEncrypt != null) {
                // 有加密注解但没有加密标头 → 拒绝访问
                handlerExceptionResolver.resolveException(..., new ServiceException("没有访问权限"));
                return;  // ← 终止链！不再向后传递
            }
        }

        // 响应加密：标记了 response=true → 包装响应
        if (responseFlag) {
            responseBodyWrapper = new EncryptResponseBodyWrapper(servletResponse);
            responseWrapper = responseBodyWrapper;
        }

        chain.doFilter(requestWrapper, responseWrapper);

        // 在响应写回前，对响应体加密
        if (responseFlag) {
            String encryptContent = responseBodyWrapper.getEncryptContent(...);
            servletResponse.getWriter().write(encryptContent);
        }
    }
}
4 可以终止链：当方法标注了 @ApiEncrypt 但请求缺少加密标头时，CryptoFilter 直接返回 403，不调用 chain.doFilter()。这是责任链模式的关键能力——任一环节可以终止请求。 CryptoFilter.java:78-83
5 请求前 + 响应后双阶段处理：chain.doFilter() 之前的代码处理请求（解密），之后的代码处理响应（加密）。这是一个围绕 chain.doFilter() 的 sandwich 模式。 CryptoFilter.java:93-103
⚠️ Filter 顺序至关重要：CryptoFilter 必须在 RepeatableFilter 之前。因为 CryptoFilter 解密请求体后，RepeatableFilter 才能缓存解密后的请求体供后续环节读取。如果顺序反了，RepeatableFilter 缓存的是加密密文——后续拦截器和 Controller 读到的都是乱码。

二、Layer 2：Spring Interceptor Chain（中间层）

拦截器工作在 DispatcherServlet 之后、Controller 之前。可以拿到目标 Handler 信息，但不能修改请求/响应对象。

PlusWebInvokeTimeInterceptor — 耗时统计

打开 ruoyi-common/ruoyi-common-web/src/main/java/org/dromara/common/web/interceptor/PlusWebInvokeTimeInterceptor.java：

@Slf4j
public class PlusWebInvokeTimeInterceptor implements HandlerInterceptor {

    private final static ThreadLocal<StopWatch> KEY_CACHE = new ThreadLocal<>();

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) throws Exception {
        // 打印请求 URL 和方法
        log.info("[PLUS]开始请求 => URL[{}],参数类型[json],参数:[{}]",
            url, limit(jsonParam));

        StopWatch stopWatch = new StopWatch();
        KEY_CACHE.set(stopWatch);  // 存入 ThreadLocal（线程隔离）
        stopWatch.start();

        return true;  // ← 返回 true 表示继续传递链
    }

    @Override
    public void afterCompletion(HttpServletRequest request,
                                HttpServletResponse response,
                                Object handler, Exception ex) {
        StopWatch stopWatch = KEY_CACHE.get();
        if (stopWatch != null) {
            stopWatch.stop();
            log.info("[PLUS]结束请求 => URL[{}],耗时:[{}]毫秒",
                url, stopWatch.getDuration().toMillis());
            KEY_CACHE.remove();  // 清理 ThreadLocal，防止内存泄漏
        }
    }
}
6 ThreadLocal 存状态：拦截器在 preHandle 中将 StopWatch 存入 ThreadLocal，在 afterCompletion 中取出。因为同一个请求的 preHandle 和 afterCompletion 在同一个线程中执行。 PlusWebInvokeTimeInterceptor.java:31
7 preHandle 返回 true/false 决定链走向：返回 true → 继续执行后续拦截器和 Controller；返回 false → 终止链，请求到此结束。本项目的时间拦截器始终返回 true（不应因日志统计阻止请求）。 PlusWebInvokeTimeInterceptor.java:76

三、Layer 3：MyBatis Plugin Chain（最内层）

MyBatis Plugin 通过动态代理拦截 SQL 执行过程。项目中的执行顺序：

PlusDataPermissionInterceptor → SQL 改写（注入数据权限条件） ↓ SqlLogInterceptor → SQL 日志输出 ↓ MybatisEncryptInterceptor → 入参加密（写入前） ↓ MybatisDecryptInterceptor → 出参解密（读取后）
拦截器位置拦截点职责
PlusDataPermissionInterceptorruoyi-common-mybatisExecutor.query/update根据用户角色动态拼接数据权限 SQL
SqlLogInterceptorruoyi-common-mybatisStatementHandler记录完整 SQL 和执行耗时到日志
MybatisEncryptInterceptorruoyi-common-encryptParameterHandler对加密字段的入参自动加密
MybatisDecryptInterceptorruoyi-common-encryptResultSetHandler对加密字段的出参自动解密
💡 三层链的分工清晰：Filter 管请求/响应的包装和安全（HTTP 层面）；Interceptor 管横切逻辑（日志、权限校验，Spring 层面）；MyBatis Plugin 管数据存取（持久层）。同是责任链模式，但它们工作在完全不同的抽象层级。

五、前端等价物：Axios 拦截器链

打开 plus-ui-vue/src/utils/request.ts，这是前端版的责任链：

service.interceptors.request.use((config) => {
    // ① 注入 Authorization token
    config.headers['Authorization'] = 'Bearer ' + getToken();
    // ② GET 请求参数拼接到 URL
    if (config.method === 'get' && config.params) { ... }

service.interceptors.response.use((res) => {
    // ① 响应解密

8 Axios 拦截器天生就是责任链：多个请求拦截器按注册顺序执行，每个可以修改 config 后传给下一个；多个响应拦截器按注册顺序的逆序执行。this is exactly the chain-of-responsibility pattern. request.ts:110-257

六、自测

1. 为什么 CryptoFilter 必须在 RepeatableFilter 之前？

2. Filter 和 Interceptor 的核心区别是什么？

3. PlusWebInvokeTimeInterceptor 为什么用 ThreadLocal 存 StopWatch？

📖 点击查看答案

1. B — 顺序决定了缓存的是加密数据还是明文。CryptoFilter 先解密，RepeatableFilter 后缓存 → 后续拦截器和 Controller 读到的都是明文。如果顺序反了，RepeatableFilter 缓存密文，后续环节读到的就是乱码。
2. B — Filter 是 Servlet 容器层面的，可以包装/替换 Request 和 Response；Interceptor 是 Spring 层面的，通过 preHandle/postHandle/afterCompletion 三个回调介入，但不能修改请求响应对象本身。
3. A — 拦截器是单例的（Spring Bean 默认为 singleton），所有请求共享同一个拦截器实例。如果 StopWatch 是实例字段，多个并发请求会互相覆盖。ThreadLocal 为每个请求线程提供独立的存储空间。
