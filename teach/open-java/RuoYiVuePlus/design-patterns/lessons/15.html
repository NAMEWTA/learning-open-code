第 15 课 · 前端拦截器与路由守卫 — Axios 中间件链 + Router Guards 责任链模式

🎯 本节目标：理解 Axios 拦截器链如何实现请求/响应的双向中间件管道，掌握路由守卫的三级决策树（token 检查 → 用户信息 → 动态路由），能解释 isRelogin 门面守卫如何防止重登录弹窗重复弹出。

在 HTTP 客户端和前端路由两个层面，RuoYi-Vue-Plus 都使用了责任链模式（Chain of Responsibility）的变体。Axios 的拦截器本质是请求/响应的双向管道；Vue Router 的导航守卫是异步决策树。本节逐一拆解。

一、Axios 拦截器链全景图

请求方向 (request interceptor) → 服务端 → 响应方向 (response interceptor) ┌──────────────────────────────────────────────────────────────────┐ │ 请求拦截器链（按执行顺序） │ │ │ │ ① 注入 Content-Language 头 (getLanguage()) │ │ ② 注入 Authorization: Bearer {token} │ │ ③ GET 请求参数映射到 URL (tansParams) │ │ ④ 重复提交检查 (sessionStorage + 500ms 间隔比对) │ │ ⑤ AES 加密 (条件: VITE_APP_ENCRYPT=true 且 isEncrypt=true) │ │ ⑥ FormData 去 Content-Type 头 │ │ │ ├───────────────────────────────────────────────────────────────────┤ │ 响应拦截器链（按执行顺序） │ │ │ │ ① AES 解密 (条件: VITE_APP_ENCRYPT=true 且有 encrypt-key 头) │ │ ② 二进制数据直通 (blob/arraybuffer → 不解包直接返回) │ │ ③ 状态码路由: │ │ 401 → isRelogin 守卫 → 重登录弹窗 (只弹一次) │ │ 500 → ElMessage.error │ │ 601 → ElMessage.warning │ │ 其他错误码 → ElNotification.error │ │ 200 → 正常 resolve │ │ ④ 网络错误处理器 (Network Error / timeout / status code) │ └──────────────────────────────────────────────────────────────────┘

二、请求拦截器：6 层管道

打开 plus-ui-vue/src/utils/request.ts，请求拦截器注册在第 110-176 行：

2.1 语言与认证头注入

service.interceptors.request.use((config: any) => {
  // 层 1：国际化语言标识
  config.headers['Content-Language'] = getLanguage();

  // 层 2：Token 注入（如果存在且未被显式禁用）
  const isToken = config.headers?.isToken === false;
  if (getToken() && !isToken) {
    config.headers['Authorization'] = 'Bearer ' + getToken();
  }
  // ...
});
💡 设计精妙之处：isToken === false 的检查提供了反向控制——默认所有请求都带 token（安全优先），但允许个别请求通过 header 标记跳过。这是「默认安全」（Secure by Default）原则的体现，类似 Java 安全框架中的 @PermitAll 注解思路。

2.2 GET 参数序列化

// 层 3：GET 请求参数展开到 URL
if (config.method === 'get' && config.params) {
  let url = config.url + '?' + tansParams(config.params);
  url = url.slice(0, -1);
  config.params = {};
  config.url = url;
}

tansParams 函数来自 @/utils/ruoyi，将对象参数展开为 key=val&key=val 格式，并直接拼接到 URL 上。这样做而非依赖 axios 的 params 序列化，原因是可以自定义数组/对象的序列化规则。

2.3 重复提交防护

// 层 4：基于 sessionStorage + 时间戳的重复提交检测
const isRepeatSubmit = config.headers?.repeatSubmit === false;
if (!isRepeatSubmit && (config.method === 'post' || config.method === 'put')) {
  const requestObj = {
    url: config.url,
    data: typeof config.data === 'object' ? JSON.stringify(config.data) : config.data,
    time: new Date().getTime()
  };
  const sessionObj = cache.session.getJSON('sessionObj');
  if (sessionObj) {
    const s_url = sessionObj.url;
    const s_data = sessionObj.data;
    const s_time = sessionObj.time;
    const interval = 500; // 500ms 内同一 URL + 同一 body → 拒绝
    if (s_data === requestObj.data && requestObj.time - s_time < interval
        && s_url === requestObj.url) {
      return Promise.reject(new Error('数据正在处理，请勿重复提交'));
    }
  }
  cache.session.setJSON('sessionObj', requestObj);
}
1 三次比对：URL + JSON 序列化后的 data + 500ms 时间窗口。三个条件同时满足才算重复。只比 URL 不够（同一 URL 不同参数），只比时间不够（正常连续操作），三要素交集确保了精度。 request.ts:132-153
2 可跳过机制：通过 repeatSubmit: false header 标记可绕过此检查。这为批量导入、文件上传等合理高频场景留了后门。 request.ts:117
3 sessionStorage 而非 localStorage：sessionStorage 在标签页关闭时自动清除，不会留下过期数据污染。这是有意的生命周期选择。 request.ts:139

2.4 条件 AES 加密

// 层 5：条件加密（仅在全局开关 + 单请求开关同时为 true 时启用）
if (import.meta.env.VITE_APP_ENCRYPT === 'true') {
  if (isEncrypt && (config.method === 'post' || config.method === 'put')) {
    const aesKey = generateAesKey();                          // 32字节随机 key
    config.headers[encryptHeader] = encrypt(encryptBase64(aesKey)); // RSA 加密 key
    config.data = typeof config.data === 'object'
      ? encryptWithAes(JSON.stringify(config.data), aesKey)   // AES 加密 body
      : encryptWithAes(config.data, aesKey);
  }
}

加密策略将在第 17 课详细展开，这里先理解它在请求管道中的位置——在所有公共头注入之后、在请求发出之前。

2.5 FormData 边界处理

// 层 6：文件上传时不设 Content-Type，让浏览器自动带 boundary
if (config.data instanceof FormData) {
  delete config.headers['Content-Type'];
}

这是 HTML5 FormData 上传的标准做法。如果手动设置了 Content-Type: multipart/form-data，浏览器不会自动追加 boundary=... 参数，导致服务端无法解析。

三、响应拦截器：状态码路由表

打开 plus-ui-vue/src/utils/request.ts，响应拦截器在第 179-257 行：

service.interceptors.response.use(
  (res: any) => {
    // 第一步：AES 解密（如果加密开关开启且有 encrypt-key 头）
    if (import.meta.env.VITE_APP_ENCRYPT === 'true') {
      const keyStr = res.headers[encryptHeader];
      if (keyStr != null && keyStr != '') {
        const base64Str = decrypt(keyStr);                    // RSA 解密
        const aesKey = decryptBase64(base64Str.toString());   // base64 解码
        res.data = JSON.parse(decryptWithAes(res.data, aesKey)); // AES 解密
      }
    }

    const code = res.data.code || HttpStatus.SUCCESS;

    // 第三步：状态码路由
    if (code === 401) {
      // ── isRelogin 门面守卫 ──
      if (!isRelogin.show) {
        isRelogin.show = true;
        ElMessageBox.confirm(
          '登录状态已过期，您可以继续留在该页面，或者重新登录',
          '系统提示',
          { confirmButtonText: '重新登录', cancelButtonText: '取消', type: 'warning' }
        ).then(() => {
          isRelogin.show = false;
          useUserStore().logout().then(() => {
            router.replace({ path: '/login', query: {
              redirect: encodeURIComponent(router.currentRoute.value.fullPath || '/')
            }});
          });
        }).catch(() => { isRelogin.show = false; });
      }
      return Promise.reject('无效的会话，或者会话已过期，请重新登录。');
    } else if (code === HttpStatus.SERVER_ERROR) {  // 500
      ElMessage({ message: msg, type: 'error' });
      return Promise.reject(createHandledError(msg));
    } else if (code === HttpStatus.WARN) {           // 601
      ElMessage({ message: msg, type: 'warning' });
      return Promise.reject(createHandledError(msg));
    } else if (code !== HttpStatus.SUCCESS) {        // 其他非 200
      ElNotification.error({ title: msg });
      return Promise.reject(createHandledError(msg));
    } else {
      return Promise.resolve(res.data);              // 200 → 正常
    }
  },
  async (error: any) => {                             // 网络错误
    const message = (await extractErrorMessage(error)) || errorCode['default'];
    ElMessage({ message, type: 'error', duration: 5 * 1000 });
    error.isHandled = true;
    return Promise.reject(error);
  }
);
⚠️ isRelogin 门面守卫的缘由：如果页面上同时发起 3 个 API 请求且全部返回 401，没有 isRelogin.show 守卫的话，屏幕上会弹出 3 个"登录已过期"的确认框。这是一个典型的并发去重问题，isRelogin.show 是一个线程安全的门面布尔标志（JS 单线程保证了安全性）。

createHandledError：防止重复弹窗的标记模式

function createHandledError(message: string) {
  const error = new Error(message) as Error & { isHandled?: boolean };
  error.isHandled = true;    // ★ 标记为 "已处理"，业务层可跳过
  return error;
}

// 业务层使用：
const [err] = await to(someApi());
if (err) {
  if (!isHandledRequestError(err)) {  // 拦截器已弹窗 → 跳过
    ElMessage.error(err.message);     // 拦截器未弹窗 → 补充弹窗
  }
}

这解决了双重弹窗问题：拦截器中已弹出错误提示，如果业务层的错误处理逻辑再弹一次，用户就会看到两条相同的错误信息。isHandled 标志在拦截器层和业务层之间建立了信息通道。

四、路由守卫：三级决策树

打开 plus-ui-vue/src/permission.ts，守卫注册在第 22-77 行：

router.beforeEach(to, from) │ ├─ ① 有 token 吗？ │ ├─ NO → 在白名单中？ YES → 放行 | NO → 跳转 /login?redirect=... │ └─ YES → │ ├─ 目标路径是 /login？ → 重定向到 / │ ├─ 在白名单中？ → 放行 │ └─ 其他路径 → │ ├─ ② 已加载用户角色？ (roles.length > 0) │ │ ├─ NO → fetch getInfo() → generateRoutes() → addRoute → hack 重定向 │ │ └─ YES → 放行 │ router.afterEach → NProgress.done()

4.1 白名单模式

const whiteList = ['/login', '/register', '/social-callback', '/register*', '/register/*'];

const isWhiteList = (path: string) => {
  return whiteList.some(pattern => isPathMatch(pattern, path));
};

isPathMatch 来自 @/utils/validate，它支持通配符匹配（如 /register/* 匹配 /register/step1），比简单的数组 includes 检查更灵活。

4.2 Hack 重定向：确保 addRoute 完成

// hack 方法 确保 addRoutes 已完成
return {
  path: to.path,
  replace: true,   // ★ 替换当前历史记录，不回退到中间态
  params: to.params,
  query: to.query,
  hash: to.hash,
  name: to.name as string
};

这是 Vue Router 动态路由的标准 hack——首次进入时触发 generateRoutes() 后，不能直接 next()（因为新路由尚未注册），而是用目标路径的完整信息重新发起导航。第二次进入时 roles.length > 0 为 true，直接放行。

4.3 三级错误处理

整个守卫中的错误处理有三层：

  1. Axios 拦截器层：HTTP 错误码 → 弹窗 + createHandledError 标记
  2. await-to-js 层：const [err] = await to(fn) 捕获异步错误，避免 try/catch 嵌套
  3. 守卫路由层：isHandledRequestError(err) 判断是否需要补充弹窗
const [err] = await tos(useUserStore().getInfo());
if (err) {
  await useUserStore().logout();
  if (!isHandledRequestError(err)) {   // 拦截器已弹窗 → 跳过
    ElMessage.error(err instanceof Error ? err.message : String(err));
  }
  return { path: '/' };
}

五、权限指令：v-hasPermi 和 v-hasRoles

打开 plus-ui-vue/src/directive/permission/index.ts：

export const hasPermi: Directive = {
  mounted(el: HTMLElement, binding: DirectiveBinding) {
    const { permissions } = useUserStore();
    const { value } = binding;
    if (value && value instanceof Array && value.length > 0) {
      const hasPermission = permissions.some((permi: string) => {
        return permi === '*:*:*' || value.includes(permi);  // 超级管理员通配
      });
      if (!hasPermission) {
        el.parentNode && el.parentNode.removeChild(el);     // ★ DOM 级删除
      }
    }
  }
};
💡 为什么不隐藏而删除？ v-if 只是不渲染，但 v-hasPermi 直接 removeChild。这有几个好处：(1) DOM 中不存在该元素，前端调试时也看不到；(2) 比 v-if 更彻底——即使有人通过浏览器 DevTools 强制设置 display: block，元素还在，但 removeChild 真正销毁了元素。不过需要注意的是，mounted 阶段执行删除，在 SSR 场景下可能有问题。
