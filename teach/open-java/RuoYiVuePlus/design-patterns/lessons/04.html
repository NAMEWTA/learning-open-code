第 4 课 · 组合模式 — 树结构的统一处理 结构型模式

🎯 本节目标：理解 TreeBuildUtils 如何用组合模式统一处理菜单树、部门树、流程分类树，能说出 buildMultiRoot 和 getLeafNodes 的用途，并解释为何整个系统只维护一套树构建逻辑。

组合模式（Composite）的精髓：将对象组织成树形结构，让客户端对单个对象和组合对象一视同仁。在 RuoYi-Vue-Plus 中，组合模式体现为 TreeBuildUtils —— 一套方法处理所有树结构场景，而非每个模块各自写一套递归拼接。

二、动态规划 + 回溯：核心算法

第 46-52 行的 build() 方法是整个工具的核心。它使用了两个关键操作：

// TreeBuildUtils.java 第 46-52 行
public static <K, T> List<T> build(List<T> items, K parentId,
        Function<T, K> classifier, BiConsumer<T, Map<K, List<T>>> action) {
    // ① 动态规划：按 parentId 分组，构建邻接表
    Map<K, List<T>> nodeTreeMaps = items.stream()
        .collect(Collectors.groupingBy(classifier));
    // ② 回溯：遍历每个 item，通过 action 建立父子关系
    items.forEach(item -> action.accept(item, nodeTreeMaps));
    return nodeTreeMaps.get(parentId);
}
1 动态规划（分组）：Collectors.groupingBy(classifier) 将平铺列表按 parentId 分组，形成 O(1) 查找的邻接表。这和 DP 的「记忆化」思想一致——先计算好所有分组关系，避免 O(n^2) 遍历。 TreeBuildUtils.java:48
2 回溯（关联）：遍历每个 item 执行 action，在邻接表中查找其子节点并递归关联。这一步类似回溯——在每个节点上做决策（把孩子挂到当前节点下），子树构建完成后返回。 TreeBuildUtils.java:50
💡 为什么不用递归遍历？ 如果对每个节点都遍历整个列表找子节点，时间复杂度是 O(n^2)。用 groupingBy 先分组，时间复杂度降为 O(n)。这就是为什么注释里写「动态规划」——本质上是在做状态压缩（将父子关系预计算为 HashMap）。

3.2 部门树 — SysDeptServiceImpl

ruoyi-system/.../impl/SysDeptServiceImpl.java 第 129 行，使用 buildMultiRoot：

// SysDeptServiceImpl.java 第 125-135 行
public List<Tree<Long>> buildDeptTreeSelect(List<SysDeptVo> depts) {
    if (CollUtil.isEmpty(depts)) { return CollUtil.newArrayList(); }
    return TreeBuildUtils.buildMultiRoot(depts,
        SysDeptVo::getDeptId,     // getId: 获取节点ID
        SysDeptVo::getParentId,   // getParentId: 获取父节点ID
        (dept, tree) -> {         // parser: 节点转换
            tree.setId(dept.getDeptId());
            tree.setParentId(dept.getParentId());
            tree.setName(dept.getDeptName());
        });
}
⚠️ 为什么部门树用 buildMultiRoot？ 菜单树有明确的根节点（TOP_PARENT_ID = 0），所以用 build(list, parentId, ...)。但部门数据可能有多棵独立的树（例如不同分公司各自有部门体系），没有共同的根，所以用 buildMultiRoot 自动检测所有顶级节点并分别构建。

3.4 叶子节点提取

第 120 行的 getLeafNodes() 通过递归流遍历，提取所有没有子节点的节点：

// TreeBuildUtils.java 第 120-127 行
public static <K> List<Tree<K>> getLeafNodes(List<Tree<K>> nodes) {
    return nodes.stream()
        .flatMap(TreeBuildUtils::extractLeafNodes)
        .collect(Collectors.toList());
}

// 第 150-158 行: 递归提取
private static <K> Stream<Tree<K>> extractLeafNodes(Tree<K> node) {
    if (!node.hasChild()) {
        return Stream.of(node);  // 无子节点 = 叶子
    } else {
        return node.getChildren().stream()
            .flatMap(TreeBuildUtils::extractLeafNodes);
    }
}

四、前端中的组合模式：递归 SidebarItem

前端 Vue 3 中使用递归组件渲染菜单树。打开 plus-ui-vue/src/layout/components/Sidebar/SidebarItem.vue：

<!-- SidebarItem.vue -- 递归组件实现组合模式 -->
<template>
  <div v-if="item.children && item.children.length > 0">
    <el-sub-menu :index="resolvePath(item.path)">
      <template #title>{{ item.meta?.title }}</template>
      <!-- 递归：SidebarItem 调用自身 -->
      <sidebar-item
        v-for="child in item.children"
        :key="child.path"
        :item="child"
        :base-path="resolvePath(child.path)"
      />
    </el-sub-menu>
  </div>
  <!-- 叶子节点：没有 children，渲染普通菜单项 -->
  <template v-else>
    <el-menu-item :index="resolvePath(item.path)">
      {{ item.meta?.title }}
    </el-menu-item>
  </template>
</template>

这和 Java 端的 extractLeafNodes 呼应对称：Java 端负责构建树（平铺数据变树结构），前端负责渲染树（树结构变 DOM）。两端都依赖组合模式的核心——统一对待叶子和组合节点。

五、组合模式在本项目中的设计价值

1 一处定义，多处复用：菜单、部门、分类、字典、地区……所有需要树形展示的地方都走 TreeBuildUtils。如果新增一个需要树结构的模块，只需提供一个 NodeParser 或 action lambda，不需要重写递归逻辑。
2 类型安全 + 灵活的分组键：classifier 和 action 都是函数式接口，调用方通过方法引用传入分组键和组装逻辑，不依赖固定的接口继承。
3 通用数据结构：使用 Hutool 的 Tree<K> 作为统一数据结构，前端收到的是标准化的 {id, parentId, name, children}，而非每个模块各自定义不同的 VO 结构。
💡 思考题：如果不用组合模式，每个模块各自写一套递归树构建，会有什么后果？（1）菜单递归写一遍，部门递归再写一遍，分类递归又写一遍——大量重复代码；（2）当树构建逻辑需要优化（如改为迭代方式或加入缓存），需要改动 N 处；（3）叶子节点提取、路径拼接等附加功能需要在每个模块各自实现。这就是组合模式的核心价值——统一性。

1. B — 先用 Collectors.groupingBy 构建 HashMap 邻接表（O(n)），再遍历关联子节点（O(n)）。时间复杂度从朴素递归的 O(n^2) 降到 O(n)。
2. B — buildMultiRoot（第 99 行）通过计算哪些 parentId 没有对应的 id（即父节点不在当前列表中），自动识别所有顶级节点，然后分别为每个顶级节点构建子树。
3. B — 递归组件是组合模式的经典前端实现：el-sub-menu 是组合节点（有 children），el-menu-item 是叶子节点，模板对两者的处理统一（都通过 sidebar-item 渲染），符合组合模式「统一对待」的核心思想。
