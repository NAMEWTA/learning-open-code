第 11 课 · 分层架构与模块化 Layered Architecture

五层依赖体系 — 告别"大泥球"

本课目标：掌握 RuoYi-Vue-Plus 的 5 层分层架构、23 个通用模块的组织原则，以及为什么不设 ruoyi-framework 中心模块 —— 理解"框架碎片化"的架构思想。

1. 为什么架构分层？

"Big Ball of Mud"（大泥球）是软件工程中最臭名昭著的反模式 —— 所有代码互相依赖，改动一处牵动全身，无法独立编译、测试和部署。

RuoYi-Vue-Plus 通过严格的五层依赖体系避免了这一问题。核心原则：

依赖铁律：上层可以依赖下层，下层绝对不能依赖上层。依赖方向始终向下。任何打破这一规则的行为都会在 Maven 编译阶段被阻止。

2. 五层全景图

从底层基础设施到顶层入口，RuoYi-Vue-Plus 的结构如下：

┌─────────────────────────────────────────────────────────┐ │ Layer 5: 入口层 (ruoyi-admin) │ │ 依赖所有模块，是唯一的 Spring Boot 可执行 Jar │ ├─────────────────────────────────────────────────────────┤ │ Layer 4: API 合约层 (ruoyi-api) │ │ 只依赖 ruoyi-common-core，定义接口和 DTO │ ├─────────────────────────────────────────────────────────┤ │ Layer 3: 业务模块层 (ruoyi-modules) │ │ system、workflow、job、gen、demo、ai │ ├─────────────────────────────────────────────────────────┤ │ Layer 2: 横向组装层 (ruoyi-common-security) │ │ 薄封装层，编排 satoken 等安全能力 │ ├─────────────────────────────────────────────────────────┤ │ Layer 1: 基础设施能力层 (ruoyi-common-*) │ │ satoken、mybatis、web、log、encrypt、oss、sms... │ ├─────────────────────────────────────────────────────────┤ │ Layer 0: 基础设施基础层 │ │ ruoyi-common-core, ruoyi-common-json, ruoyi-common-redis│ └─────────────────────────────────────────────────────────┘

3. Layer 0 — 基础设施基础层：只能被依赖，不依赖任何业务

这一层是整个项目的基石，包含最原子的工具和抽象：

模块核心内容定位
ruoyi-common-core R 响应体、PageResult、异常体系、枚举、工具类、全局常量 项目的最底层，被所有模块依赖
ruoyi-common-json Jackson 序列化配置 只依赖 core，提供 JSON 序列化能力
ruoyi-common-redis Redisson + Lock4j + Caffeine 多级缓存 依赖 core + json，提供分布式缓存能力
1 ruoyi-common/ruoyi-common-core/pom.xml 
<!-- ruoyi-common-core 是最底层，不依赖任何其他 common 模块 -->
<dependencies>
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-context-support</artifactId>
    </dependency>
    <dependency>
        <groupId>cn.hutool</groupId>
        <artifactId>hutool-core</artifactId>
    </dependency>
    <!-- 无任何 org.dromara 内部依赖 -->
</dependencies>
2 ruoyi-common/ruoyi-common-redis/pom.xml 
<!-- ruoyi-common-redis 依赖基础层 -->
<dependencies>
    <dependency>
        <groupId>org.dromara</groupId>
        <artifactId>ruoyi-common-core</artifactId>
    </dependency>
    <dependency>
        <groupId>org.dromara</groupId>
        <artifactId>ruoyi-common-json</artifactId>
    </dependency>
</dependencies>

4. Layer 1 — 基础设施能力层：23 个通用模块，7 大功能组

这一层是框架能力的核心，每个模块都是按领域隔离的独立 jar：

4.2 关键依赖链

这些模块之间也存在严格的内部依赖方向：

ruoyi-common-core ← 无依赖（叶子节点） ↑ ruoyi-common-json ← core ↑ ruoyi-common-redis ← core + json ↑ ruoyi-common-satoken ← core + redis + ruoyi-api ↑ ↑ ruoyi-common-security ← satoken ruoyi-common-mybatis ← core + satoken + ruoyi-api
关键观察：ruoyi-common-satoken 依赖了 ruoyi-api（Layer 4）！这看似违反"下层不依赖上层"的铁律，但实际上 ruoyi-api 本身只依赖 ruoyi-common-core（Layer 0），不依赖 Layer 1。因此 ruoyi-common-satoken（Layer 1）依赖 ruoyi-api（Layer 4）不形成循环。Layer 4 在语义上高于 Layer 1，但在编译依赖图上属于同一层甚至更低。

5. Layer 2 — 横向组装层：一个特殊的薄封装

3 ruoyi-common/ruoyi-common-security/pom.xml 
<!-- ruoyi-common-security 只有一个核心依赖 -->
<dependencies>
    <dependency>
        <groupId>org.dromara</groupId>
        <artifactId>ruoyi-common-satoken</artifactId>
    </dependency>
</dependencies>

ruoyi-common-security 的职责不是实现，而是编排。它包装 ruoyi-common-satoken 的认证能力，为业务模块提供更简洁的安全 API。这是横切关注点（cross-cutting concern）的标准处理方式。

6. Layer 3 — 业务模块层：按领域垂直切分

六个业务模块各自独立，只能通过 ruoyi-api 互相通信：

模块核心职责依赖的 common 模块数
ruoyi-system用户、角色、菜单、部门、岗位、字典、参数、OSS、客户端管理14
ruoyi-workflow流程定义、实例管理、任务审批（Warm-Flow）若干
ruoyi-job定时任务调度（SnailJob）2（最轻量）
ruoyi-gen代码生成器（Profile 控制，生产环境不打包）若干
ruoyi-demo框架能力演示示例14+
ruoyi-aiAI 业务功能若干
设计亮点：ruoyi-job 只依赖 2 个 common 模块（json + job），是所有业务模块中最轻量的。这意味着你可以单独部署一个只包含 ruoyi-job 的节点，不需要加载 system、workflow 等重型模块。这就是模块化架构带来的按需部署能力。

7. Layer 4 — API 合约层：模块间的唯一桥梁

4 ruoyi-api/pom.xml 
<!-- ruoyi-api 只依赖 core，是真正的"合约层" -->
<dependencies>
    <dependency>
        <groupId>org.dromara</groupId>
        <artifactId>ruoyi-common-core</artifactId>
    </dependency>
</dependencies>

ruoyi-api 只包含接口定义、DTO 和事件类。它不包含任何业务逻辑实现，因此可以安全地被所有模块依赖而不会引入实现耦合。

8. Layer 5 — 入口层：唯一可执行的 Jar

5 ruoyi-admin/pom.xml 
<!-- ruoyi-admin 是唯一的 Spring Boot 可执行 Jar -->
<dependencies>
    <!-- 基础设施 -->
    <dependency><groupId>org.dromara</groupId><artifactId>ruoyi-common-doc</artifactId></dependency>
    <dependency><groupId>org.dromara</groupId><artifactId>ruoyi-common-social</artifactId></dependency>
    <!-- API 合约 -->
    <dependency><groupId>org.dromara</groupId><artifactId>ruoyi-api</artifactId></dependency>
    <!-- 业务模块 -->
    <dependency><groupId>org.dromara</groupId><artifactId>ruoyi-system</artifactId></dependency>
    <dependency><groupId>org.dromara</groupId><artifactId>ruoyi-job</artifactId></dependency>
    <!-- ... -->
</dependencies>
注意：ruoyi-gen 模块通过 Maven Profile 控制 —— 只在 gen Profile 激活时才打包。这意味着生产环境的可执行 Jar 不包含代码生成器，减少攻击面。

9. 为什么不设 ruoyi-framework 模块

原版 RuoYi 有一个巨大的 ruoyi-framework 模块，将所有框架级代码塞在一起。RuoYi-Vue-Plus 的架构团队做了一个关键决策：将 framework 拆解为 23 个粒度更细的 common 模块。

架构哲学："一个 Jar 一件事" vs "一个大 Jar 所有事"。拆解的收益：
1. 业务模块只引入需要的 common 模块（job 模块不需要 mybatis 和 encrypt）
2. 每个 common 模块有清晰的 API 边界，便于版本升级和替换
3. 按需编译、按需测试
4. 新增能力只需新增一个 common 模块，不影响现有模块
对比维度原版 ruoyi-framework（大 Jar）RuoYi-Vue-Plus（碎片化）
模块粒度1 个巨型模块，什么都往里塞23 个领域隔离的小模块
耦合度高：任意两个类都可能互相引用低：通过 pom.xml 显式声明依赖
可替换性困难：改一处影响全局容易：替换整个模块只需新的 jar
按需部署不支持：必须部署整个 jar支持：job 模块可独立部署
新人上手困难：不知道从哪里开始看清晰：先看 core，再看需要的能力

10. 五层依赖链总结

ruoyi-admin (入口) ├── 依赖 ruoyi-common-doc, ruoyi-common-social, ruoyi-common-mail, ruoyi-common-mcp ├── 依赖 ruoyi-api (合约) ├── 依赖 ruoyi-system (实现 system/api/* 接口) │ ├── 依赖 ruoyi-common-core, -doc, -mybatis, -translation, -oss, │ │ -log, -excel, -sms, -security, -web, -sensitive, -encrypt, -push │ └── 依赖 ruoyi-api ├── 依赖 ruoyi-job (只依赖 common-json + common-job) ├── 依赖 ruoyi-workflow (实现 workflow/api/* 接口) ├── 依赖 ruoyi-ai ├── 依赖 ruoyi-demo (演示 14+ common 模块) └── 依赖 ruoyi-gen (可选，Maven Profile 控制) ruoyi-extend (独立部署，不打包到 admin jar) ├── ruoyi-monitor-admin (Spring Boot Admin 服务端) ├── ruoyi-snailai-server (Snail AI Agent 服务端) └── ruoyi-snailjob-server (SnailJob 调度服务端)

自测题

Q1：如果 ruoyi-common-mybatis 需要调用 Redis，它应该依赖哪个模块？为什么不能直接依赖 ruoyi-admin？

查看答案

它应该依赖 ruoyi-common-redis（Layer 0），这是架构允许的（Layer 1 依赖 Layer 0）。绝对不能依赖 ruoyi-admin（Layer 5），因为这会形成循环依赖 —— ruoyi-admin 已经依赖了 ruoyi-common-mybatis → ruoyi-system。Maven 编译会直接报错 "Circular dependency"。

Q2：ruoyi-job 只依赖 2 个 common 模块。如果需求变更，ruoyi-job 需要访问数据库（MyBatis），已有的架构能否支持？需要改哪里？

查看答案

完全支持。只需要在 ruoyi-modules/ruoyi-job/pom.xml 中添加 ruoyi-common-mybatis 的依赖即可。这就是模块化架构的优势 —— 按需引入能力，不引入不需要的依赖。不需要改动 ruoyi-system 或 ruoyi-admin。

Q3：为什么 ruoyi-common-satoken 可以依赖 ruoyi-api，而不会造成"下层依赖上层"的架构违规？

查看答案

因为 ruoyi-api 的 pom.xml 只依赖 ruoyi-common-core，而 ruoyi-common-satoken 的 pom.xml 也依赖 ruoyi-common-core。所以 ruoyi-api 的编译依赖链非常短，不会向上引入 Layer 3 的业务模块。在依赖图上，ruoyi-api → core ← ruoyi-common-satoken 形成的是 V 字形依赖，没有循环。Layer 编号表示语义层级，不总是等于编译依赖层级。
