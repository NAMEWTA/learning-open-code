第 1 课 · 创建型模式三剑客 — 工厂、构建者、单例 基础模式

🎯 本节目标：能用项目中的 3 个真实案例解释工厂模式、构建者模式、单例模式，并能在新模块中复用这些模式。

创建型模式解决「如何创建对象」的问题。RuoYi-Vue-Plus 中这三个模式的使用频率极高，而且是相互配合的——工厂返回单例、构建者构建复杂对象、单例持有工厂创建的实例。

一、工厂模式：OssFactory — 带缓存的智能工厂

打开 ruoyi-common/ruoyi-common-oss/src/main/java/org/dromara/common/oss/factory/OssFactory.java：

public class OssFactory {

    private static final Map<String, OssClient> CLIENT_CACHE = new ConcurrentHashMap<>();
    private static final Map<String, ReentrantLock> LOCK_MAP = new ConcurrentHashMap<>();

    // ── 获取默认实例（从 Redis 读取默认配置 key）──
    public static OssClient instance() {
        String configKey = RedisUtils.getCacheObject(CacheNames.SYS_OSS_CONFIG);
        if (configKey == null) {
            throw new ServiceException("未配置默认OSS!");
        }
        return instance(configKey);
    }

    // ── 按配置 key 获取/创建实例（带双重检查）──
    public static OssClient instance(String configKey) {
        ReentrantLock lock = LOCK_MAP.computeIfAbsent(configKey, k -> new ReentrantLock());
        lock.lock();
        try {
            OssClient client = CLIENT_CACHE.get(configKey);
            if (client == null) {
                // 从数据库加载 OSS 配置，构建 S3 客户端
                client = createOssClient(configKey);
                CLIENT_CACHE.put(configKey, client);
            } else if (configChanged(configKey, client)) {
                // 配置变更了？关闭旧客户端，重建
                client.close();
                client = createOssClient(configKey);
                CLIENT_CACHE.put(configKey, client);
            }
            return client;
        } finally {
            lock.unlock();
        }
    }
}
1 静态工厂方法：OssFactory.instance() 和 OssFactory.instance(configKey) 是典型的静态工厂。外部无需知道 OssClient 如何构建、用什么配置。 OssFactory.java:34,46
2 缓存 + 懒加载：ConcurrentHashMap 缓存已创建的客户端。第一次调用时创建（懒加载），后续直接返回缓存。这是工厂模式与单例思想的结合。 OssFactory.java:28
3 配置变更感知：不是简单地「缓存了就永远不重建」。如果 Redis 中的配置 hash 变了，工厂会关闭旧客户端并创建新的。这个设计细节避免了配置变更后需要重启应用的痛点。 OssFactory.java:48-55
4 每个配置 key 独立的锁：用 ReentrantLock 而非 synchronized，且不同 configKey 使用不同的锁对象。这样 configKey-A 的创建不会阻塞 configKey-B。 OssFactory.java:29

💡 Service Locator 还是 Factory？

IAuthStrategy.login() 中用了 SpringUtils.getBean(beanName)，这是 Service Locator 模式。它与 Factory 的区别在于：Factory 自己负责创建对象；Service Locator 向容器索要已存在的对象。在 Spring 项目中，两者经常混合使用——Spring 容器是最终的工厂，而 OssFactory 这类静态工厂负责管理 Spring 容器不管理的对象。

二、构建者模式：Options 与 QueryBuilder 的流畅 API

2.1 OSS Options：Lombok 链式构建者

打开 ruoyi-common-oss/.../model/Options.java：

@Data
@Accessors(chain = true)  // ← setter 返回 this，支持链式调用
public class Options {
    private Map<String, String> metadata;
    private List<TransferListener> transferListeners;

    // ── 静态工厂入口 ──
    public static Options builder() {
        return new Options();
    }

    // ── 链式添加方法 ──
    public Options addMetadataItem(String key, String value) {
        if (this.metadata == null) this.metadata = new HashMap<>();
        this.metadata.put(key, value);
        return this;  // ← 返回 this 支持链式
    }

    public Options addTransferListener(TransferListener listener) {
        if (this.transferListeners == null) this.transferListeners = new ArrayList<>();
        this.transferListeners.add(listener);
        return this;
    }
}

调用演示：

Options options = Options.builder()
    .addMetadataItem("filename", "report.pdf")
    .addTransferListener(new LoggingTransferListener())
    .addMetadataItem("author", "admin");
// setter 链式 + 语义化添加方法 → 清晰且灵活

2.2 QueryBuilder：类型安全的查询构建者

打开 ruoyi-common-mybatis/.../query/QueryBuilder.java：

public class QueryBuilder {
    private QueryBuilder() {}  // 私有构造器，只能通过静态方法创建

    public static <T> LambdaQueryBuilder<T> lambda(Class<T> entityClass) {
        return new LambdaQueryBuilder<>(entityClass);
    }

    public static <T> LambdaJoinQueryBuilder<T> lambdaJoin(Class<T> entityClass) {
        return new LambdaJoinQueryBuilder<>(entityClass);
    }
}

实际使用中，LambdaQueryBuilder 继续链式构建：

// 在业务代码中
List<SysUserVO> users = userMapper.lambda()
    .eq(SysUser::getStatus, "0")        // WHERE status = '0'
    .like(SysUser::getUserName, "admin") // AND user_name LIKE '%admin%'
    .orderByAsc(SysUser::getCreateTime)  // ORDER BY create_time ASC
    .selectVoList();                     // 执行查询，自动映射为 VO
💡 构建者 vs 普通 setter：构建者模式的核心优势不是「看起来更好看」，而是：1) 可以在 build() 时做完整性校验；2) 支持不可变对象（所有字段 final，只有构建者能设置）；3) 将复杂构建逻辑与对象本身分离。本项目中的 Options 采用 Lombok @Accessors(chain=true) 是一种轻量级替代——没有 build() 校验步骤，但代码量减少 90%。

三、单例模式：Spring 容器 + 模块级单例

3.1 Spring Bean 单例（默认行为）

Spring 容器管理的 Bean 默认是 singleton。项目中所有 @Service、@Component、@Configuration 类都是单例。例如：

@Service("passwordAuthStrategy")
public class PasswordAuthStrategy implements IAuthStrategy {
    // Spring 容器保证全局只有一个 PasswordAuthStrategy 实例
    // 所有请求共享同一个实例 → 因此必须是无状态的
}
⚠️ 单例 Bean 必须是线程安全的！ 所有认证策略中都没有实例级可变状态字段。状态都放在方法参数、局部变量、或通过 final 注入的依赖中。如果你在单例 Bean 中加了可变字段，多个请求会互相踩踏。
