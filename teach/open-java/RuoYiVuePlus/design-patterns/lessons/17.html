第 17 课 · 前端加密策略 — RSA+AES 混合加密的完整设计 安全模式

🎯 本节目标：掌握 RuoYi-Vue-Plus 前端 RSA+AES 混合加密的设计原理，能独立绘制请求加密/响应解密的完整数据流图，理解双层开关（全局 VITE_APP_ENCRYPT + 单请求 isEncrypt）的精妙之处，并能对比后端 ApiEncrypt Filter 的双通道设计。

RuoYi-Vue-Plus 对登录、注册、密码重置等敏感 API 实施了RSA + AES 混合加密。本节从前端视角完整拆解密钥生成、请求加密、响应解密的全流程，并与后端 ApiEncrypt Filter 体系对照理解。

一、为什么是混合加密？

加密方式优点缺点适用场景
纯 RSA非对称，公钥可公开分发慢（比 AES 慢 100-1000 倍），有明文长度限制（取决于密钥长度，2048 位密钥最多加密 245 字节）密钥交换
纯 AES快（硬件加速），无长度限制对称密钥需要双方共享，密钥本身如何安全传递？数据加密
RSA + AES 混合RSA 加密 AES 密钥（小数据），AES 加密 payload（大数据）需要管理 RSA 密钥对 + 每次请求生成 AES 密钥最优方案
💡 一句话总结：RSA 负责在客户端和服务端之间安全地传递临时 AES 密钥，AES 负责用这个临时密钥快速加密实际的请求/响应数据。两者各司其职，互补短板。

前端加密体系由三个文件构成：

文件职责核心导出

三、密钥生成：密码学安全的随机数

打开 plus-ui-vue/src/utils/crypto.ts：

const generateRandomString = (): string => {
  const array = new Uint8Array(32);           // 32 字节 = 256 位
  crypto.getRandomValues(array);              // ★ 密码学安全的随机数
  return Array.from(array, b => b.toString(16).padStart(2, '0'))
    .join('')
    .slice(0, 32);                            // 取前 32 个十六进制字符
};

export const generateAesKey = (): CryptoJS.lib.WordArray => {
  return CryptoJS.enc.Utf8.parse(generateRandomString());
};
1 crypto.getRandomValues()：浏览器原生 API，从操作系统熵源获取随机数。比 Math.random() 安全得多——后者是伪随机，种子可预测。 crypto.ts:11
2 32 字节 → 32 个十六进制字符：每个字节转为 2 位十六进制（0x00 → "00"，0xFF → "ff"），然后截取前 32 个字符作为最终的 256 位 AES 密钥。 crypto.ts:12-14
3 WordArray：CryptoJS 的内部数据格式。CryptoJS.enc.Utf8.parse() 将字符串转为 CryptoJS 可处理的 WordArray 对象。后续的 AES 加密操作要求输入是 WordArray 类型。 crypto.ts:22

四、RSA 工具：密钥来自环境变量

打开 plus-ui-vue/src/utils/jsencrypt.ts：

import { JSEncrypt } from 'jsencrypt';

const publicKey = import.meta.env.VITE_APP_RSA_PUBLIC_KEY;
const privateKey = import.meta.env.VITE_APP_RSA_PRIVATE_KEY;

// 加密（用公钥）
export const encrypt = (txt: string) => {
  const encryptor = new JSEncrypt();
  encryptor.setPublicKey(publicKey);
  return encryptor.encrypt(txt);
};

// 解密（用私钥）
export const decrypt = (txt: string) => {
  const encryptor = new JSEncrypt();
  encryptor.setPrivateKey(privateKey);
  return encryptor.decrypt(txt);
};
⚠️ 安全提示：源码注释中明确写了"前端不建议存放私钥 不建议解密数据 因为都是透明的意义不大"。RSA 私钥出现在前端代码中仅用于响应解密（解密服务端用 RSA 公钥加密的临时 AES 密钥）。在生产环境中，前端代码是公开的，私钥会暴露。RuoYi-Vue-Plus 这么做是在「便利性」和「绝对安全」之间的工程权衡——它提供了传输层加密，比明文传输安全得多，但不是端到端加密。如果需要更强的安全性，应考虑使用 HTTPS 双向认证 + 硬件安全模块（HSM）。

五、请求加密完整流程

当 VITE_APP_ENCRYPT = 'true' 且请求头 isEncrypt = 'true' 时，请求拦截器触发加密：

┌────────────────── 请求加密流程 ──────────────────┐ │ │ │ Step 1 客户端生成 32 字节随机 AES 密钥 │ │ generateAesKey() │ │ ──────────────── │ │ Step 2 RSA 公钥加密 AES 密钥 → 放入请求头 │ │ encrypt(encryptBase64(aesKey)) │ │ → headers['encrypt-key'] = encryptedKey │ │ ──────────────── │ │ Step 3 AES-256-ECB + PKCS7 加密请求体 │ │ encryptWithAes(JSON.stringify(data), aesKey)│ │ → config.data = ciphertext │ │ ──────────────── │ │ Step 4 发送请求到服务端 │ │ POST /api/login │ │ Headers: encrypt-key: [RSA加密的AES密钥] │ │ Body: [AES加密的密文] │ │ │ └────────────────────────────────────────────────────┘

对应代码（plus-ui-vue/src/utils/request.ts 第 155-166 行）：

if (import.meta.env.VITE_APP_ENCRYPT === 'true') {
  if (isEncrypt && (config.method === 'post' || config.method === 'put')) {
    const aesKey = generateAesKey();                              // Step 1
    config.headers[encryptHeader] = encrypt(encryptBase64(aesKey)); // Step 2
    config.data = typeof config.data === 'object'
      ? encryptWithAes(JSON.stringify(config.data), aesKey)        // Step 3
      : encryptWithAes(config.data, aesKey);
  }
}

加密参数说明

参数选择原因
算法AES-256256 位密钥提供足够的安全强度（128 位也够用，但 256 是行业标准）
模式ECB简单、无需 IV，适合短文本加密（JSON 请求体通常较短）
填充PKCS7Java 服务端（Cipher.getInstance）默认使用 PKCS5Padding，与 PKCS7 兼容
⚠️ ECB 模式的注意事项：ECB 模式下，相同的明文块加密后产生相同的密文块，存在模式分析攻击风险。对于用户密码、银行卡号等短字段，影响有限；但对于大段重复文本（如图片 base64），不建议使用 ECB。本项目仅在登录/注册等短 JSON 场景使用加密，选择 ECB 简化了 IV（初始化向量）的传递复杂度——不需要额外传递 16 字节随机 IV。

六、响应解密完整流程

响应拦截器中解密（plus-ui-vue/src/utils/request.ts 第 180-196 行）：

┌────────────────── 响应解密流程 ──────────────────┐ │ │ │ Step 1 从响应头读取 encrypt-key │ │ res.headers['encrypt-key'] │ │ ──────────────── │ │ Step 2 RSA 私钥解密 → 得到 base64 编码的 AES 密钥 │ │ decrypt(keyStr) → base64Str │ │ ──────────────── │ │ Step 3 base64 解码 → 得到原始 AES 密钥（WordArray） │ │ decryptBase64(base64Str.toString()) │ │ → aesKey │ │ ──────────────── │ │ Step 4 AES 解密响应体 → JSON 字符串 │ │ decryptWithAes(data, aesKey) │ │ ──────────────── │ │ Step 5 JSON.parse → 业务数据对象 │ │ res.data = JSON.parse(decryptData) │ │ │ └────────────────────────────────────────────────────┘
service.interceptors.response.use((res: any) => {
  if (import.meta.env.VITE_APP_ENCRYPT === 'true') {
    const keyStr = res.headers[encryptHeader];      // Step 1
    if (keyStr != null && keyStr != '') {
      const base64Str = decrypt(keyStr);            // Step 2：RSA 私钥解密
      const aesKey = decryptBase64(                  // Step 3：base64→WordArray
        base64Str.toString()
      );
      const decryptData = decryptWithAes(res.data, aesKey); // Step 4：AES 解密
      res.data = JSON.parse(decryptData);            // Step 5：JSON 解析
    }
  }
  // ... 状态码路由
});
💡 为什么服务端也在响应中返回 encrypt-key？ 因为请求中的 AES 密钥是一次性的——每个请求生成一个新的 32 字节随机密钥。服务端解密请求后，需要将同一个 AES 密钥通过响应头的 encrypt-key 回传给客户端，客户端才能解密响应。如果服务端也生成一个新的 AES 密钥，客户端就不知道用什么密钥解密了。这是一个完整的对称密钥闭环：客户端生成 → 服务端使用并回传 → 客户端使用同一个密钥解密。

七、双层开关设计：精准控制加密粒度

加密不是全量开启的——只有登录、注册、密码重置等敏感 API 才加密。控制机制是两层：

// 第一层：全局开关（.env 文件）
VITE_APP_ENCRYPT = 'true'           // 控制整个应用的加密功能是否可用

// 第二层：单请求开关（API 调用时设置 header）
headers: { isEncrypt: 'true' }      // 控制当前请求是否加密

// 拦截器中的判断逻辑
if (import.meta.env.VITE_APP_ENCRYPT === 'true') {  // 全局开关
  if (isEncrypt && (config.method === 'post' || config.method === 'put')) {
    // 执行加密                                                // 单请求开关
  }
}

这种设计允许：

  • 开发环境：设置 VITE_APP_ENCRYPT = 'false'，所有请求明文传输，方便调试
  • 生产环境，普通 API：不设 isEncrypt 头，即使全局开关为 true 也不加密
  • 生产环境，敏感 API：同时满足两个条件，执行加密

八、前后端加密体系对照

后端也有一套完整的加密体系，前端的设计是对其的镜像实现：

层面后端实现前端实现协同关系
RSA 加解密SecurityUtils + RSA BouncyCastlejsencrypt.ts (JSEncrypt 库)公钥一致，互相解密
AES 加解密AES BouncyCastlecrypto.ts (CryptoJS 库)算法/模式/填充必须完全一致
请求加密拦截ApiEncryptFilter (Servlet Filter)request.ts 请求拦截器前端加密 → Filter 解密
响应加密拦截ApiEncryptResponseBodyWrapperrequest.ts 响应拦截器Filter 加密 → 前端解密
数据库字段加密MybatisEncryptInterceptor无（后端独有）仅后端

后端的 ApiEncryptFilter 是一个 Servlet Filter（包装了请求/响应的输入输出流），它读取 encrypt-key 头 → RSA 私钥解密获取 AES 密钥 → 读取请求体 → AES 解密 → 将解密后的请求体交给后续的 Controller。响应路径正好相反：Controller 返回数据 → Filter 用同一个 AES 密钥加密响应体 → 将加密后的 AES 密钥通过 encrypt-key 响应头返回。

💡 双通道设计：RuoYi-Vue-Plus 在加密方面有「两个独立的通道」——API 传输加密（ApiEncryptFilter，前后端配合）和数据库存储加密（MybatisEncryptInterceptor，后端独立）。前者保护数据在网络传输过程中不被窃听；后者保护敏感数据（如手机号、身份证号）存入数据库后即使数据库泄露也无法直接读取。两个通道互不干扰，可以独立开启或关闭。

十、自测

1. 为什么使用 RSA + AES 混合加密而非纯 RSA？

2. 响应解密时，服务端回传的 encrypt-key 头中的 AES 密钥来自哪里？

3. 以下哪项不是 VITE_APP_ENCRYPT 和 isEncrypt 双层开关的合理使用场景？

📖 点击查看答案

1. A — RSA 非对称加密速度慢（比 AES 慢 100-1000 倍），且有明文长度限制（2048 位密钥最多加密 245 字节）。混合加密让 RSA 只加密 32 字节的 AES 密钥，AES 加密实际数据，取长补短。
2. B — AES 密钥由客户端在请求时生成，服务端通过 encrypt-key 请求头获取后用于解密请求体；响应时服务端使用同一个 AES 密钥加密响应体，并将这个密钥通过 encrypt-key 响应头回传给客户端用于解密。这是一个完整的对称密钥闭环。
3. C — 双层开关的精确意图就是选择性加密。加密有性能开销（每次请求生成密钥、RSA 运算），不是所有 API 都需要加密。只有登录、注册等敏感操作才开启 isEncrypt。
