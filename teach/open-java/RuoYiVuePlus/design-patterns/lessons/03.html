第 3 课 · 外观模式 — API 合约层与门面封装 结构型模式

🎯 本节目标：理解 ruoyi-api 合约层如何通过外观模式实现模块解耦，能解释为什么业务模块之间不直接互相依赖、以及 EncryptorManager 如何降低加密子系统的使用复杂度。

外观模式（Facade）的精髓：为子系统中的一组接口提供一个统一的高层接口，让子系统更容易使用。在 RuoYi-Vue-Plus 中，外观模式体现在两个层面：模块间调用（ruoyi-api 合约层）和复杂子系统封装（EncryptorManager、PlusSpringCacheManager）。

一、ruoyi-api 合约层：跨模块的外观模式

1.1 问题：没有外观模式的世界

假设 ruoyi-system（系统管理）和 ruoyi-workflow（工作流）是两个独立的 Maven 模块。工作流需要查询用户信息：

// ❌ 错误做法：工作流模块直接依赖系统模块
// ruoyi-workflow/pom.xml
<dependency>
    <groupId>org.dromara</groupId>
    <artifactId>ruoyi-system</artifactId>  ← 循环依赖风险！
</dependency>
// 然后直接注入 System 模块的 Service
@Autowired
private SysUserServiceImpl userService;  ← 耦合了实现细节
⚠️ 问题：如果 ruoyi-system 也依赖 ruoyi-workflow（比如系统配置需要触发流程），就形成了循环依赖。Maven 会拒绝构建。

1.2 解决方案：API 合约层

┌──────────────────────────────────────────────────────────┐ │ ruoyi-api (合约层) │ │ 只依赖 ruoyi-common-core │ │ │ │ org.dromara.system.api.UserService ← 接口（合约） │ │ org.dromara.system.api.domain.UserDTO ← 传输对象 │ │ org.dromara.workflow.api.WorkflowService │ │ org.dromara.workflow.api.event.ProcessEvent │ └──────┬──────────────────────────────┬──────────────────────┘ │ implements │ implements ▼ ▼ ┌──────────────────┐ ┌──────────────────────┐ │ ruoyi-system │ │ ruoyi-workflow │ │ SysUserServiceImpl│ │ WorkflowServiceImpl │ │ implements │ │ implements │ │ UserService │ │ WorkflowService │ └──────────────────┘ └──────────────────────┘

打开 ruoyi-api/.../system/api/UserService.java：

public interface UserService {
    UserDTO selectUserById(Long userId);
    List<UserDTO> selectUserByIds(List<Long> userIds);
    List<UserDTO> selectUsersByRoleId(Long roleId);
    List<UserDTO> selectUsersByDeptId(Long deptId);
    // ... 只暴露外部需要的查询方法，不暴露内部实现细节
}
1 接口最小化：API 合约只暴露其他模块真正需要的方法，而非 SysUserService 的全部 50+ 个方法。这体现了外观模式的「选择性暴露」原则。 UserService.java
2 DTO 隔离：API 模块定义自己的 UserDTO，与 ruoyi-system 内部的 SysUserVO 分离。内部 VO 可以自由变化（加字段、改名），只要实现类在 selectUserById() 里做好转换，外部模块不受影响。 UserService.java + UserDTO
3 依赖方向：所有模块都依赖 ruoyi-api（箭头向下），没有模块直接依赖 ruoyi-system。ruoyi-api 只依赖 ruoyi-common-core（极轻量），避免了传递引入重量级依赖。

1.3 运行时注入

// ruoyi-workflow 中使用
@Service
public class FlwTaskServiceImpl {
    @Autowired
    private UserService userService;  // ← 注入的是 UserService 接口

    public void assignTask(String taskId, Long userId) {
        UserDTO user = userService.selectUserById(userId);
        // Spring 在运行时自动注入 ruoyi-system 提供的 SysUserServiceImpl
    }
}

二、EncryptorManager：加密子系统的外观

打开 ruoyi-common-encrypt/.../core/EncryptorManager.java。加密子系统内部很复杂：5 种算法（AES、RSA、SM2、SM4、Base64）、2 种编码方式、MyBatis 拦截器加解密、API Filter 加解密、密钥管理……但外部使用者只需要两行：

// 外部使用者的视角——极简
encryptorManager.encrypt("13812345678", encryptContext);  // 加密
encryptorManager.decrypt("encrypted_data", encryptContext);  // 解密

这就是外观模式的核心价值——将复杂子系统的 N 个接口收敛为 1-2 个高层接口。

@Component
public class EncryptorManager {
    // 内部复杂度：算法注册表、加密器缓存、上下文工厂
    private final Map<EncryptorCacheKey, IEncryptor> encryptorMap = new ConcurrentHashMap<>();

    // ── 对外暴露的高层接口 ──
    public String encrypt(String value, EncryptContext context) {
        IEncryptor encryptor = registAndGetEncryptor(context);
        return encryptor.encrypt(value, context.getEncodeType());
    }

    public String decrypt(String value, EncryptContext context) {
        IEncryptor encryptor = registAndGetEncryptor(context);
        return encryptor.decrypt(value, context.getEncodeType());
    }

    // ── 内部复杂逻辑：查找或创建加密器 ──
    private IEncryptor registAndGetEncryptor(EncryptContext context) {
        EncryptorCacheKey cacheKey = new EncryptorCacheKey(context);
        return encryptorMap.computeIfAbsent(cacheKey, k -> {
            // 1. 通过 AlgorithmType 枚举确定加密器类
            Class<? extends IEncryptor> clazz = context.getAlgorithm().getClazz();
            // 2. 反射创建实例
            return ReflectUtil.newInstance(clazz, context);
        });
    }
}

三、前端中的外观模式

前端 plus-ui-vue/src/plugins/modal.ts 是 Element Plus 消息系统的外观：

// 将 ElMessage、ElMessageBox、ElNotification、ElLoading 四套 API
// 统一封装为 $modal 的 4 个方法
export default {
    msgSuccess(message: string) {
        ElMessage.success(message);       // 隐藏了 ElMessage 的复杂性
    },
    msgError(message: string) {
        ElMessage.error(message);
    },
    alert(message: string) {
        return ElMessageBox.alert(message, '系统提示');
    },
    confirm(content: string) {
        return ElMessageBox.confirm(content, '警告', { type: 'warning' });
    },
    loading(message: string) {
        loadingInstance = ElLoading.service({ fullscreen: true, text: message });
    },
}
// 使用：$modal.msgSuccess('操作成功') 而非记住 ElMessage、ElMessageBox、ElNotification 的不同 API

四、外观模式的判断标准

特征本项目案例是否符合
隐藏子系统的内部复杂性EncryptorManager 隐藏了算法选择、反射创建、缓存管理✅
提供统一的高层接口UserService 接口统一了系统模块对外暴露的方法✅
子系统仍然可以直接使用ruoyi-system 内部仍然可以直接用 SysUserServiceImpl 的全部方法✅
不强制使用外部模块不强制使用 UserService，可以选择直接依赖（不推荐但技术上可行）✅
💡 外观 vs 适配器 vs 装饰器：外观的目的是简化（把 N 个接口变成 1 个），适配器的目的是兼容（把 A 接口变成 B 接口），装饰器的目的是增强（在不改变接口的前提下加功能）。在 ruoyi-api 中，UserService 是外观（简化了外部对系统模块的调用），同时也是一个间接的适配器（把 SysUserService 的 50 个方法适配为 10 个方法）。
