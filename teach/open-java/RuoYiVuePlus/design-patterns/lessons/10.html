第 10 课 · 命令模式 Command Pattern

SnailJob 任务与工作流指令

本课目标：理解命令模式如何在 RuoYi-Vue-Plus 中将"方法调用"封装为可调度、可分发、可重试的独立任务对象，实现调度器与业务逻辑的彻底解耦。

1. 什么是命令模式

命令模式的本质：将"请求"封装为对象，从而使你可以用不同的请求对客户进行参数化、对请求排队或记录请求日志，以及支持可撤销的操作。

在 RuoYi-Vue-Plus 中，命令模式体现在两个层面：

  • SnailJob 任务执行器：每个定时任务是一个独立命令对象
  • Warm-Flow 工作流指令：完成、启动、驳回等流程操作是命令对象
  • 前端 $modal 插件：封装 UI 操作为指令
关键洞察：命令模式的核心收益不在于"多态替换"，而在于解耦调用者（调度器）与接收者（业务逻辑）。新增任务类型时，调度器代码零改动。

2. SnailJob 中的命令模式

SnailJob 是一个分布式任务调度框架。在 RuoYi-Vue-Plus 中，它的执行器层就是命令模式的完整实现。

2.1 类继承式执行器 — 最简命令

源码位置：

1 ruoyi-modules/ruoyi-job/src/main/java/org/dromara/job/snailjob/TestClassJobExecutor.java 
@Component
public class TestClassJobExecutor extends AbstractJobExecutor {

    @Override
    protected ExecuteResult doJobExecute(JobArgs jobArgs) {
        return ExecuteResult.success("TestJobExecutor测试成功");
    }
}

这就是一个最纯粹的命令对象：

  • AbstractJobExecutor = 命令接口（定义 doJobExecute() 抽象方法）
  • TestClassJobExecutor = 具体命令（封装一段可执行的业务逻辑）
  • SnailJob 调度引擎 = 调用者（只知道调用 doJobExecute()，不知道具体做什么）

2.2 注解式执行器 — 更灵活的命令注册

除了继承方式，SnailJob 还支持注解方式：

2 ruoyi-modules/ruoyi-job/src/main/java/org/dromara/job/snailjob/TestAnnoJobExecutor.java 
@Component
@JobExecutor(name = "testJobExecutor")
public class TestAnnoJobExecutor {

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        SnailJobLog.LOCAL.info("测试注解方式执行任务");
        SnailJobLog.REMOTE.info("测试控制台上报信息");
        return ExecuteResult.success();
    }
}

两种方式的对比：

方式命令注册方法签名约束适用场景
继承 AbstractJobExecutorSpring Bean 自动发现必须 override doJobExecute(JobArgs)简单任务、框架强制性约束
@JobExecutor 注解注解 name 属性注册方法名和签名更灵活分片、广播、MapReduce 等复杂任务

2.3 分片与广播命令

3 ruoyi-modules/ruoyi-job/.../snailjob/TestStaticShardingJob.java 
@Component
@JobExecutor(name = "testStaticShardingJob")
public class TestStaticShardingJob {

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        // 从调度参数解析分片范围：fromId,toId
        String[] jobParams = jobArgs.getJobParams().split(",");
        long fromId = Long.parseLong(jobParams[0]);
        long toId = Long.parseLong(jobParams[1]);
        // 模拟对 [fromId, toId] 范围的数据库记录加密处理
        Thread.sleep(3000);
        return ExecuteResult.success();
    }
}
4 ruoyi-modules/ruoyi-job/.../snailjob/TestBroadcastJob.java 
@Component
@JobExecutor(name = "testBroadcastJob")
public class TestBroadcastJob {
    @Value("${snail-job.port}")
    private int port;

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        // 每个节点各自执行，随机模拟成功/失败
        int randomInt = new Random().nextInt(100);
        if (randomInt < 50) {
            throw new RuntimeException("模拟广播任务失败, port=" + port);
        }
        return ExecuteResult.success("节点 " + port + " 执行成功");
    }
}

无论任务类型是普通、分片还是广播，SnailJob 调度引擎都只调用一个统一的方法。每种任务就是一个可互换的命令对象。

2.4 DAG 工作流命令链

SnailJob 还支持将多个任务编排为 DAG（有向无环图）工作流。每个节点是一个独立的命令对象，通过工作流上下文传递数据：

// WechatBillTask — 第1个命令：生成微信账单
@JobExecutor(name = "wechatBillTask")
public class WechatBillTask {
    public ExecuteResult jobExecute(JobArgs jobArgs) {
        BillDTO bill = new BillDTO(1L, "wechat", "2024-01-01", new BigDecimal("100.00"));
        jobArgs.appendContext("wechat", bill); // 写入工作流上下文
        return ExecuteResult.success();
    }
}

// SummaryBillTask — 第3个命令：汇总账单（读取前两个命令的输出）
@JobExecutor(name = "summaryBillTask")
public class SummaryBillTask {
    public ExecuteResult jobExecute(JobArgs jobArgs) {
        BillDTO wechatBill = jobArgs.getWfContext("wechat", BillDTO.class);
        BillDTO alipayBill = jobArgs.getWfContext("alipay", BillDTO.class);
        BigDecimal total = wechatBill.billAmount().add(alipayBill.billAmount());
        return ExecuteResult.success("总金额: " + total);
    }
}
设计精髓：DAG 工作流将每个步骤封装为命令，命令之间通过上下文传递数据 —— 命令模式的队列化 + 管道模式的组合。每个命令只需要关心自己的输入和输出，不需要知道前序命令是谁、后序命令是谁。

3. Warm-Flow 工作流中的命令模式

在 Warm-Flow 工作流引擎中，每个流程操作也是一个命令对象：

5 ruoyi-modules/ruoyi-workflow/.../service/impl/FlwTaskServiceImpl.java

FlwTaskServiceImpl 将工作流任务操作封装为独立方法：

  • completeTask() — 完成任务（命令：审批通过）
  • rejectTask() — 驳回任务（命令：退回重审）
  • delegateTask() — 委派任务（命令：转交他人）
  • transferTask() — 转办任务（命令：移交权限）
  • addSignTask() — 加签任务（命令：新增审批人）

这些操作在 TaskOperationEnum 枚举中有明确定义：

6 ruoyi-modules/ruoyi-workflow/.../common/enums/TaskOperationEnum.java 
public enum TaskOperationEnum {
    COMPLETE("complete", "审批通过"),
    REJECT("reject", "审批驳回"),
    DELEGATE("delegate", "审批委派"),
    TRANSFER("transfer", "审批转办"),
    ADDSIGN("addSign", "审批加签"),
    // ...
}

工作流引擎根据操作类型分发到不同的命令处理逻辑 —— 这是命令模式 + 策略模式的经典组合。

4. 前端 $modal 插件 — UI 命令封装

前端同样应用了命令模式的思想。每个 UI 交互被封装为 $modal 插件的一个方法调用：

7 plus-ui-vue/src/plugins/modal.ts 
// 加载中遮罩 — 把 UI 操作封装为可复用的命令
loading(content: string) {
    loadingInstance = ElLoading.service({
        lock: true,
        text: content,
        background: "rgba(0, 0, 0, 0.7)",
    });
},
closeLoading() {
    loadingInstance?.close();
}

在组件中使用时，调用者不需要知道底层是 ElLoading.service：

// plus-ui-vue/src/components/FileUpload/index.vue
modal.loading("正在上传文件...");
// ... 上传逻辑 ...
modal.closeLoading();
跨层统一：后端 SnailJob 将定时任务封装为命令，前端 $modal 将 UI 操作封装为命令。两者都实现了调用者与实现细节的解耦。这就是命令模式在前后端的一致性表达。

5. 命令模式的三要素总结

SnailJob 调度引擎 (Invoker) | | 调用 doJobExecute(jobArgs) v AbstractJobExecutor / @JobExecutor (Command 接口) | | 实现 v TestClassJobExecutor / TestBroadcastJob / WechatBillTask (ConcreteCommand) | | 操作 v 业务数据库 / 外部服务 / Redis 缓存 (Receiver)
角色项目中的对应职责
Command 接口AbstractJobExecutor / @JobExecutor定义统一的 execute() 契约
ConcreteCommandTestClassJobExecutor 等封装具体的业务逻辑
InvokerSnailJob 调度引擎触发命令执行，管理调度策略
Receiver数据库、Redis、外部 API实际执行操作的业务对象

6. 为什么不用策略模式

读者可能会问：既然每个任务类型有不同的执行逻辑，为什么不是策略模式？

核心区别：策略模式关注的是算法族的可互换性（同一问题的不同解法）；命令模式关注的是请求的封装（将操作调用打包为对象）。在 SnailJob 中，每个任务不是"同一问题"的不同解法，而是完全不同的业务操作（发账单 vs 加密数据 vs 发邮件）。更重要的是，命令模式天然支持队列、日志、撤销、重试这些调度特性。

自测题

Q1：在 SnailJob 中，调度引擎调用任务执行器时，它知道任务的具体业务逻辑是什么吗？为什么？

查看答案

不知道。调度引擎只知道调用 doJobExecute(JobArgs) 方法（或注解标记的方法），返回 ExecuteResult。具体的业务逻辑（加密数据、生成账单、发送通知）完全封装在具体的 Executor 类中。调度器与业务逻辑之间通过命令接口解耦。

Q2：DAG 工作流中，WechatBillTask 和 SummaryBillTask 之间是如何传递数据的？这体现了命令模式的什么扩展能力？

查看答案

通过 jobArgs.appendContext("wechat", bill) 写入和 jobArgs.getWfContext("wechat", BillDTO.class) 读取工作流上下文。这体现了命令模式的可组合性 —— 每个命令是独立单元，但可以通过共享上下文（管道）串联成复杂的工作流，每个命令仍然只关心自己的输入和输出。

Q3：前端 $modal.loading() 和 SnailJob 的 TestClassJobExecutor 都体现了命令模式。它们的共同设计目标是什么？

查看答案

共同设计目标是将"请求"与"执行者"解耦。前端调用 modal.loading("上传中") 时不需要知道底层是 ElLoading.service 还是其他实现；后端 SnailJob 调度器调用 doJobExecute() 时不需要知道任务要做什么。调用者只依赖抽象命令接口，不依赖具体实现。
