策略 1: PasswordAuthStrategy — 关键代码

密码登录核心逻辑
// 1. 图片验证码校验（一次性，取出即删）
String verifyKey = CAPTCHA_CODE_KEY + uuid;
String captcha = RedisUtils.getCacheObject(verifyKey);
RedisUtils.deleteObject(verifyKey);  // 阅后即焚，防重放
if (captcha == null) throw new CaptchaExpireException();

// 2. 用户加载
SysUserVo user = userMapper.lambda()
    .eq(SysUser::getUserName, username)
    .voOne();  // MyBatis-Plus 直接返回 VO
if (null) throw UserException("user.not.exists");
if (DISABLE) throw UserException("user.blocked");

// 3. 密码校验（Supplier 延迟执行）
loginService.checkLogin(LoginType.PASSWORD, username,
    () -> !BCrypt.checkpw(password, user.getPassword()));

策略 2: SmsAuthStrategy — 关键差异

与密码登录的核心不同点
SmsLoginBody loginBody = JsonUtils.parseObject(body, SmsLoginBody.class);
SysUserVo user = loadUserByPhoneNumber(phoneNumber);
loginService.checkLogin(LoginType.SMS, user.getUserName(),
    () -> !validateSmsCode(phoneNumber, smsCode));
// 短信验证码：比对后不删除（允许重试），靠 TTL 自动过期

策略 3: EmailAuthStrategy — 关键差异

与短信登录结构相同，字段不同
EmailLoginBody loginBody = JsonUtils.parseObject(body, EmailLoginBody.class);
SysUserVo user = loadUserByEmail(email);  // mapper.eq(SysUser::getEmail, email)
loginService.checkLogin(LoginType.EMAIL, user.getUserName(),
    () -> !validateEmailCode(email, emailCode));

策略 4: SocialAuthStrategy — OAuth 流程

最重的策略：JustAuth + 绑定表查询
// 1. JustAuth 回调认证
AuthResponse<AuthUser> response = SocialUtils.loginAuth(
    loginBody.getSource(),      // "gitee" / "github" / ...
    loginBody.getSocialCode(),  // OAuth code
    loginBody.getSocialState(), socialProperties);

// 2. 查询绑定关系
List<SysSocialVo> list = sysSocialService.selectByAuthId(
    authUserData.getSource() + authUserData.getUuid());

// 3. 用绑定的 userId 加载用户
SysUserVo user = loadUser(list.getFirst().getUserId());

JustAuth 支持 30+ 平台，只需配置文件中的 appId/appSecret 即可切换，无需平台特定代码。

策略 5: XcxAuthStrategy — 微信小程序

特殊之处： 1. 不经过 OAuth 重定向，用 wx.login() 的 code 换取 openid 2. 用 JustAuth 的 AuthWechatMiniProgramRequest 直接调用微信接口 3. LoginUser 类型是 XcxLoginUser（多了 openid 字段） 4. LoginVo 返回 openid 给前端 5. loadUserByOpenid() 目前是桩代码（需自行实现）

SysLoginService.checkLogin() — 失败计数机制

public void checkLogin(LoginType loginType, String username, Supplier<Boolean> supplier) { String errorKey = PWD_ERR_CNT_KEY + username; int errorNumber = RedisUtils.getCacheObject(errorKey) or 0; if (errorNumber >= maxRetryCount) → 抛出"超过限制"异常 ← 已锁定 if (supplier.get()) { ← 本次认证失败（Supplier 延迟执行） errorNumber++; RedisUtils.setCacheObject(errorKey, errorNumber, Duration.ofMinutes(lockTime)); if (errorNumber >= maxRetryCount) → 抛出"锁定"异常 else → 抛出"错误次数：N"异常 } RedisUtils.deleteObject(errorKey); ← 登录成功，清空计数 }
⚠️ 为什么用 Supplier 而不是 boolean？延迟计算：如果已达锁定上限，BCrypt.checkpw (昂贵操作) 根本不会执行，节省 CPU。
