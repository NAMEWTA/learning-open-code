第 4 课 · 事件监听 — 登录后的副作用处理 架构解耦

🎯 本节目标：学完你能解释 UserActionListener（Sa-Token 钩子）和 UserLoginSuccessListener（Spring 事件监听器）的分工关系，理解为什么登录成功后不直接在 AuthController 里写日志，以及 Java 17 的 record 如何替代传统 Event 类。

前 3 课我们追踪了 怎么登录。这一课关注：登录之后发生了什么？ 写入在线用户缓存、记录登录日志、解析 IP 归属地、更新最近登录时间——这些"副作用"如果写在 AuthController 里，控制器会膨胀到不可维护。RuoYi 用了一个精巧的双层事件架构来解耦。

一、双层事件的架构全景

LoginHelper.login(loginUser, model) ← 策略中调用，Sa-Token 内部 │ ▼ Sa-Token 框架内部：完成 token 签发 │ ├── 调用已注册的 SaTokenListener 列表 │ │ │ ▼ │ UserActionListener.doLogin() ← 【第一层：Sa-Token 生命周期钩子】 │ │ │ └── 发布 Spring 事件： │ SpringUtils.context() │ .publishEvent(new UserLoginSuccessEvent(loginId, tokenValue, loginParameter)) │ │ │ ▼ │ UserLoginSuccessListener ← 【第二层：Spring 事件监听器】 │ .handleLoginSuccess() │ │ │ ├── 解析 User-Agent → 浏览器 + 操作系统 │ ├── 获取客户端 IP │ ├── 调用 AddressUtils.getRealAddressByIP(ip) → IP 归属地 │ ├── 构建 UserOnlineDTO → 写入 Redis 在线缓存 │ ├── loginService.recordLoginInfo() → 发布 LoginInfoEvent │ └── loginService.updateLastLoginInfo() → 更新数据库登录时间 │ ├── 触发 doLogout() ← 退出登录时 ├── 触发 doKickout() ← 被管理员踢下线时 └── 触发 doReplaced() ← 被新登录顶下线时

二、第一层：UserActionListener — Sa-Token 的生命周期钩子

Sa-Token 框架定义了一个 SaTokenListener 接口，包含 13 个生命周期回调方法。RuoYi 的 UserActionListener 实现了其中与登录/退出相关的 4 个：

@Component
public class UserActionListener implements SaTokenListener {

    // ── 登录成功 ──
    @Override
    public void doLogin(String loginType, Object loginId,
                        String tokenValue, SaLoginParameter loginParameter) {
        // 将 Sa-Token 的事件"翻译"为 Spring 事件
        SpringUtils.context()
            .publishEvent(new UserLoginSuccessEvent(loginId, tokenValue, loginParameter));
    }

    // ── 退出登录 ──
    @Override
    public void doLogout(String loginType, Object loginId, String tokenValue) {
        RedisUtils.deleteObject(CacheNames.ONLINE_TOKEN_KEY + tokenValue);
    }

    // ── 被踢下线 ──
    @Override
    public void doKickout(String loginType, Object loginId, String tokenValue) {
        RedisUtils.deleteObject(CacheNames.ONLINE_TOKEN_KEY + tokenValue);
    }

    // ── 被顶下线 ──
    @Override
    public void doReplaced(String loginType, Object loginId, String tokenValue) {
        RedisUtils.deleteObject(CacheNames.ONLINE_TOKEN_KEY + tokenValue);
    }
}
💡 为什么需要一个"翻译层"？ Sa-Token 的 SaTokenListener 是框架级别的钩子，它的职责是"通知你发生了什么"。而 Spring 的 @EventListener 是业务级别的，允许多个监听器分工协作。UserActionListener 扮演了"桥梁"角色：接收到 Sa-Token 的登录通知，然后发布一个 Spring 事件，让业务代码可以独立响应。

退出/踢出/顶下线的统一处理

三种退出场景做的事完全一样：删除 Redis 中的在线用户缓存。区别在于触发来源：

回调方法触发场景配置关联
doLogout()用户主动点击"退出登录"AuthController.logout()
doKickout()管理员在后台将用户踢下线StpUtil.kickout(loginId)
doReplaced()同一账号在新设备登录，旧会话被顶掉is-concurrent: false

三、Java 17 record：UserLoginSuccessEvent

这是一个教科书级别的 Java 17 record 用法：

public record UserLoginSuccessEvent(
    Object loginId,
    String tokenValue,
    SaLoginParameter loginParameter
) {}

就一行。对比传统的 POJO 写法：

// 传统写法（需要 40+ 行）
public class UserLoginSuccessEvent {
    private final Object loginId;
    private final String tokenValue;
    private final SaLoginParameter loginParameter;

    public UserLoginSuccessEvent(Object loginId, String tokenValue,
                                  SaLoginParameter loginParameter) {
        this.loginId = loginId;
        this.tokenValue = tokenValue;
        this.loginParameter = loginParameter;
    }
    // getters, equals(), hashCode(), toString() ... 30+ 行
}
💡 record 自动生成：规范构造器、equals()、hashCode()、toString()、以及命名良好的访问器方法（loginId() 而非 getLoginId()）。事件对象是"不可变数据载体"的完美场景。

四、第二层：UserLoginSuccessListener — 业务副作用处理

这是实际干活的类。它监听 UserLoginSuccessEvent，执行 4 项业务操作：

@Component
public class UserLoginSuccessListener {

    private final SysLoginService loginService;

    @EventListener
    public void handleLoginSuccess(UserLoginSuccessEvent event) {
        SaLoginParameter loginParameter = event.loginParameter();

        // 1. 解析浏览器和操作系统
        UserAgent userAgent = UserAgentUtil.parse(
            ServletUtils.getRequest().getHeader("User-Agent"));

        // 2. 获取客户端 IP
        String ip = ServletUtils.getClientIP();
        String username = (String) loginParameter.getExtra(LoginHelper.USER_NAME_KEY);
        String tokenValue = event.tokenValue();

        // 3. 组装在线用户信息 + 解析 IP 归属地
        UserOnlineDTO dto = new UserOnlineDTO();
        dto.setIpaddr(ip);
        dto.setLoginLocation(AddressUtils.getRealAddressByIP(ip));  // ip2region
        dto.setBrowser(userAgent.getBrowser().getName());
        dto.setOs(userAgent.getOs().getName());
        dto.setLoginTime(System.currentTimeMillis());
        dto.setTokenId(tokenValue);
        dto.setUserName(username);
        dto.setClientKey((String) loginParameter.getExtra(LoginHelper.CLIENT_KEY));
        dto.setDeviceType(loginParameter.getDeviceType());
        dto.setDeptName((String) loginParameter.getExtra(LoginHelper.DEPT_NAME_KEY));

        // 4. 写入 Redis 在线用户缓存（token 为永久 -1 时不设 TTL）
        if (loginParameter.getTimeout() == -1) {
            RedisUtils.setCacheObject(CacheNames.ONLINE_TOKEN_KEY + tokenValue, dto);
        } else {
            RedisUtils.setCacheObject(CacheNames.ONLINE_TOKEN_KEY + tokenValue,
                dto, Duration.ofSeconds(loginParameter.getTimeout()));
        }

        // 5. 发布登录日志事件
        loginService.recordLoginInfo(username, Constants.LOGIN_SUCCESS,
            MessageUtils.message("user.login.success"));

        // 6. 更新用户表最近登录IP和时间
        loginService.updateLastLoginInfo(
            (Long) loginParameter.getExtra(LoginHelper.USER_KEY), ip);
    }
}

逐项拆解

1 IP 归属地解析：AddressUtils.getRealAddressByIP(ip) 使用 ip2region_v4.xdb 离线数据库，无需网络请求。文件在 ruoyi-admin/src/main/resources/ip2region_v4.xdb。解析结果如 "中国-广东省-深圳市"。
2 User-Agent 解析：Hutool 的 UserAgentUtil.parse() 解析请求头中的 User-Agent 字符串，提取浏览器名称（如 "Chrome"、"Firefox"）和操作系统（如 "Windows 10"、"macOS"）。UserLoginSuccessListener.java:44
3 在线用户缓存：Redis key 为 "online_token:" + tokenValue，value 为 UserOnlineDTO 序列化。TTL 与 token 有效期相同。如果 token 永不过期（timeout == -1），则不设 TTL。这个缓存是后台"在线用户"页面的数据源。UserLoginSuccessListener.java:60-64
4 登录日志事件：recordLoginInfo() 内部发布一个 LoginInfoEvent，由 ruoyi-common-log 模块的异步监听器消费，写入数据库登录日志表。注意这是 又一层事件解耦——日志写入不在当前线程执行。SysLoginService.java:134-141
5 更新登录信息：直接更新 sys_user 表的 login_ip 和 login_date 字段。使用 DataPermissionHelper.ignore() 跳过数据权限过滤——因为更新自己的登录信息不应受数据权限限制。SysLoginService.java:184-191

五、完整的事件链路

策略实例.login() │ └── LoginHelper.login(loginUser, model) │ ├── Sa-Token: 签发 token，写入 Redis 会话 │ ├── UserActionListener.doLogin() │ └── SpringUtils.context() │ .publishEvent(new UserLoginSuccessEvent(...)) │ │ │ └── UserLoginSuccessListener.handleLoginSuccess() │ ├── 在线用户缓存 → Redis │ └── recordLoginInfo() │ └── SpringUtils.context() │ .publishEvent(new LoginInfoEvent(...)) │ │ │ └── LogListener (ruoyi-common-log) │ └── 异步写入数据库登录日志 │ └── 策略返回 LoginVo → AuthController → R.ok(loginVo) 响应客户端

六、为什么这样设计：三个关注点分离

层级组件职责耦合对象
框架层UserActionListener监听 Sa-Token 生命周期，翻译为 Spring 事件；退出时清理 Redis 缓存Sa-Token API
业务层UserLoginSuccessListener登录成功后的业务数据组装：在线信息、IP 归属地、浏览器解析Redis、ip2region、Hutool UA 解析
日志层LoginInfoEvent + LogListener异步写入登录日志到数据库数据库、线程池

好处很明确：

  • AuthController 只需 10 行——解析请求、分发策略、返回结果，不包含任何副作用代码。
  • 新增副作用不影响现有代码——比如想加"登录后发送钉钉通知"，只需新增一个 @EventListener 方法。
  • 测试隔离——可以独立测试策略逻辑（mock 掉事件监听器），也可以独立测试监听器（手动发布事件）。

七、退出登录的完整流程

// AuthController
@PostMapping("/logout")
public R<Void> logout() {
    loginService.logout();  // → 发布 LoginInfoEvent 记录登出日志
    return R.ok("退出成功");
}

// SysLoginService
public void logout() {
    try {
        LoginUser loginUser = LoginHelper.getLoginUser();
        if (ObjectUtil.isNull(loginUser)) return;
        recordLoginInfo(loginUser.getUsername(), Constants.LOGOUT, "退出成功");
    } catch (NotLoginException ignored) {
    } finally {
        try {
            StpUtil.logout();  // → 触发 UserActionListener.doLogout() → 删 Redis 在线缓存
        } catch (NotLoginException ignored) {
        }
    }
}
⚠️ 注意 finally 块：StpUtil.logout() 放在 finally 中，确保即使记录日志时抛异常，token 也会被清除。两层 try-catch 都吞掉了 NotLoginException，因为 logout 本身语义是"如果没有登录态也正常返回"。
