第 3 课 · 五种认证策略深度对比 核心实现

🎯 本节目标：学完你能对比 5 种认证策略的共性流程与差异点，能独立调试每种策略的认证失败原因（验证码过期、用户不存在、账号停用、密码错误、第三方未绑定等）。

上一课我们看到了策略工厂如何分发请求。这一课逐行读完 5 个策略实现，找出它们的 共同骨架 和 独特之处。

一、共同的执行骨架

无论哪种认证方式，每个策略的 login() 方法都遵循同一模式：

1. 解析 body → 具体请求对象 (PasswordLoginBody / SmsLoginBody / ...) 2. 校验 body 字段合法性 (ValidatorUtils.validate) 3. 加载用户信息 (不同策略按不同字段查询) 4. 校验用户状态 (是否存在? 是否停用?) 5. 调用 loginService.checkLogin() 执行业务校验 (验证码/密码/第三方绑定) 6. 调用 loginService.buildLoginUser() 构建 LoginUser 7. 设置 clientKey 和 deviceType 8. IAuthStrategy.buildLoginParameter(client) 构建 SaLoginParameter 9. LoginHelper.login(loginUser, model) → Sa-Token 签发 token 10. 构建 LoginVo 返回

步骤 6-10 在所有策略中完全相同。差异在步骤 1-5。

二、策略 1：PasswordAuthStrategy — 密码登录

最传统的认证方式，也是最复杂的——因为集成了图片验证码和 BCrypt 密码校验。

@Override
public LoginVo login(String body, SysClientVo client) {
    // 1. 解析请求体
    PasswordLoginBody loginBody = JsonUtils.parseObject(body, PasswordLoginBody.class);
    ValidatorUtils.validate(loginBody);
    String username = loginBody.getUsername();
    String password = loginBody.getPassword();
    String code = loginBody.getCode();
    String uuid = loginBody.getUuid();

    // 2. 可选的图片验证码校验
    if (captchaProperties.getEnable()) {
        validateCaptcha(username, code, uuid);  // 详见下文
    }

    // 3. 按用户名加载 + 状态校验
    SysUserVo user = loadUserByUsername(username);

    // 4. 密码校验 + 失败计数/锁定
    //    注意：这里用了 Supplier 延迟执行，只有前面的校验通过才真正执行 BCrypt.checkpw
    loginService.checkLogin(LoginType.PASSWORD, username,
        () -> !BCrypt.checkpw(password, user.getPassword()));

    // 5-10. 通用步骤
    LoginUser loginUser = loginService.buildLoginUser(user);
    loginUser.setClientKey(client.getClientKey());
    loginUser.setDeviceType(client.getDeviceType());
    SaLoginParameter model = IAuthStrategy.buildLoginParameter(client);
    LoginHelper.login(loginUser, model);
    // ... 构建 LoginVo
}

图片验证码校验细节

private void validateCaptcha(String username, String code, String uuid) {
    String verifyKey = GlobalConstants.CAPTCHA_CODE_KEY + uuid;  // "captcha_codes:" + uuid
    String captcha = RedisUtils.getCacheObject(verifyKey);
    RedisUtils.deleteObject(verifyKey);  // 一次性使用，取出即删

    if (captcha == null) {
        loginService.recordLoginInfo(username, Constants.LOGIN_FAIL, "验证码已失效");
        throw new CaptchaExpireException();
    }
    if (!StringUtils.equalsIgnoreCase(code, captcha)) {
        loginService.recordLoginInfo(username, Constants.LOGIN_FAIL, "验证码错误");
        throw new CaptchaException();
    }
}
💡 一次性验证码：RedisUtils.deleteObject(verifyKey) 在读取验证码后立即删除。这意味着：验证码只能使用一次，用完即失效。防止验证码重放攻击。

用户加载与状态校验

private SysUserVo loadUserByUsername(String username) {
    SysUserVo user = userMapper.lambda()
        .eq(SysUser::getUserName, username)
        .voOne();  // MyBatis-Plus 直接返回 VO，跳过 Entity 转换
    if (ObjectUtil.isNull(user)) {
        throw new UserException("user.not.exists", username);  // "登录用户：{} 不存在"
    } else if (SystemConstants.DISABLE.equals(user.getStatus())) {
        throw new UserException("user.blocked", username);     // "登录用户：{} 已被停用"
    }
    return user;
}

三、策略 2：SmsAuthStrategy — 短信登录

和密码登录非常相似，但：用手机号查用户、用短信验证码替代密码、没有图片验证码层。

@Override
public LoginVo login(String body, SysClientVo client) {
    SmsLoginBody loginBody = JsonUtils.parseObject(body, SmsLoginBody.class);
    ValidatorUtils.validate(loginBody);
    String phoneNumber = loginBody.getPhoneNumber();
    String smsCode = loginBody.getSmsCode();

    // 按手机号加载用户
    SysUserVo user = loadUserByPhoneNumber(phoneNumber);

    // 校验短信验证码
    loginService.checkLogin(LoginType.SMS, user.getUserName(),
        () -> !validateSmsCode(phoneNumber, smsCode));
    // ... 后续完全相同
}

短信验证码校验：从 Redis 取 captcha_codes:{phoneNumber}，比对后 不删除（允许重试），过期时间由 Constants.CAPTCHA_EXPIRATION 决定。

四、策略 3：EmailAuthStrategy — 邮箱登录

与短信登录几乎一模一样，区别仅在于查询字段（email 替代 phoneNumber）和请求体类型（EmailLoginBody）：

EmailLoginBody loginBody = JsonUtils.parseObject(body, EmailLoginBody.class);
String email = loginBody.getEmail();
String emailCode = loginBody.getEmailCode();
SysUserVo user = loadUserByEmail(email);  // mapper.eq(SysUser::getEmail, email)
loginService.checkLogin(LoginType.EMAIL, user.getUserName(),
    () -> !validateEmailCode(email, emailCode));
⚠️ 注意：短信和邮箱验证码的 Redis key 前缀是同一个 GlobalConstants.CAPTCHA_CODE_KEY（值为 "captcha_codes:"）。但因为后缀是手机号或邮箱地址，实际不会冲突。图片验证码用 UUID 作为后缀，更不会冲突。

五、策略 4：SocialAuthStrategy — 第三方登录

这是最"重"的策略，涉及 JustAuth 的 OAuth 流程和社交账号绑定校验：

@Override
public LoginVo login(String body, SysClientVo client) {
    SocialLoginBody loginBody = JsonUtils.parseObject(body, SocialLoginBody.class);
    ValidatorUtils.validate(loginBody);

    // 1. 调用 JustAuth 完成 OAuth 回调认证
    AuthResponse<AuthUser> response = SocialUtils.loginAuth(
        loginBody.getSource(),      // 如 "gitee", "github", "wechat_open"
        loginBody.getSocialCode(),   // OAuth 回调返回的 code
        loginBody.getSocialState(),  // OAuth state 参数
        socialProperties);
    if (!response.ok()) {
        throw new ServiceException(response.getMsg());
    }
    AuthUser authUserData = response.getData();

    // 2. 查询该第三方账号是否已绑定系统用户
    List<SysSocialVo> list = sysSocialService.selectByAuthId(
        authUserData.getSource() + authUserData.getUuid());
    if (CollUtil.isEmpty(list)) {
        throw new ServiceException("你还没有绑定第三方账号，绑定后才可以登录！");
    }

    // 3. 用绑定的 userId 加载用户
    SysUserVo user = loadUser(list.getFirst().getUserId());
    // ... 后续通用步骤
}

关键差异：用户信息来源不是请求参数，而是 JustAuth 从第三方平台获取的 OAuth 用户数据。必须先通过 sys_social 表找到绑定的系统用户 ID，再加载用户信息。

💡 JustAuth 支持 30+ 平台：Gitee、GitHub、微信开放平台、QQ、微博、钉钉、百度、Coding、OSChina、支付宝、淘宝、Google、Facebook、抖音、企业微信、微软... 只需在 socialProperties 中配置对应平台的 appId 和 appSecret，不需要写任何平台特定代码。

六、策略 5：XcxAuthStrategy — 微信小程序登录

最特殊的一种：不经过完整的 OAuth 重定向流程，而是用微信小程序的 wx.login() 返回的临时 code 换取 openid。另外，它构建的 LoginUser 类型也不同。

@Override
public LoginVo login(String body, SysClientVo client) {
    XcxLoginBody loginBody = JsonUtils.parseObject(body, XcxLoginBody.class);
    ValidatorUtils.validate(loginBody);
    String xcxCode = loginBody.getXcxCode();  // wx.login() 返回的临时凭证
    String appid = loginBody.getAppid();      // 支持多小程序

    // 1. 用 code + appid + secret 调用微信接口换取 openid
    AuthRequest authRequest = new AuthWechatMiniProgramRequest(AuthConfig.builder()
        .clientId(appid)
        .clientSecret("自行填写密钥")  // TODO: 从数据库或配置读取
        .ignoreCheckRedirectUri(true)
        .ignoreCheckState(true)
        .build());
    AuthCallback authCallback = new AuthCallback();
    authCallback.setCode(xcxCode);
    AuthResponse<AuthUser> resp = authRequest.login(authCallback);

    String openid, unionId;
    if (resp.ok()) {
        AuthToken token = resp.getData().getToken();
        openid = token.getOpenId();
        unionId = token.getUnionId();  // 绑定开放平台后才有
    } else {
        throw new ServiceException(resp.getMsg());
    }

    // 2. 用 openid 查找本地绑定用户
    SysUserVo user = loadUserByOpenid(openid);

    // 3. 构建 XcxLoginUser（而非 LoginUser）
    XcxLoginUser loginUser = new XcxLoginUser();
    loginUser.setUserId(user.getUserId());
    loginUser.setUsername(user.getUserName());
    loginUser.setOpenid(openid);  // 小程序特有字段

    // ... LoginHelper.login(loginUser, model)

    // 4. LoginVo 也返回 openid
    loginVo.setOpenid(openid);
    return loginVo;
}
⚠️ 注意 XcxAuthStrategy 的 loadUserByOpenid() 方法是桩代码：它直接 new SysUserVo() 返回空对象，注释写着 // todo 自行实现 userService.selectUserByOpenid(openid)。这意味着小程序登录的完整实现需要开发者根据自己的业务补充用户绑定逻辑。

八、SysLoginService.checkLogin()：失败计数与锁定

密码、短信、邮箱三种策略都调用了这个方法。第三方和小程序没有调用（因为它们的认证依赖外部平台，不需要本地失败计数）。

public void checkLogin(LoginType loginType, String username, Supplier<Boolean> supplier) {
    String errorKey = CacheNames.PWD_ERR_CNT_KEY + username;
    int errorNumber = ObjectUtil.defaultIfNull(RedisUtils.getCacheObject(errorKey), 0);

    // 已达上限 → 锁定
    if (errorNumber >= maxRetryCount) {
        recordLoginInfo(username, LOGIN_FAIL, "密码错误次数超过限制({}), 请{}分钟后重试");
        throw new UserException("密码错误次数超过限制");
    }

    // supplier.get() 返回 true 表示本次认证失败
    if (supplier.get()) {
        errorNumber++;
        RedisUtils.setCacheObject(errorKey, errorNumber, Duration.ofMinutes(lockTime));
        if (errorNumber >= maxRetryCount) {
            throw new UserException("超过限制，锁定");  // 刚好达到上限
        } else {
            throw new UserException("错误次数：{}", errorNumber);  // 还有余量
        }
    }

    // 登录成功 → 清空错误计数
    RedisUtils.deleteObject(errorKey);
}
💡 为什么用 Supplier 而不是直接传 boolean？ Supplier<Boolean> 实现了延迟计算。如果错误次数已达上限，直接抛异常，supplier.get() 永远不会执行（不会调用 BCrypt.checkpw 或 Redis 查询验证码），节省了不必要的计算。

九、自测

1. 在密码登录中，验证码校验通过后，验证码会被立即从 Redis 删除。这个设计是为了？

2. 第三方登录（SocialAuthStrategy）中，用户身份是如何确定的？

3. 为什么 SocialAuthStrategy 和 XcxAuthStrategy 没有调用 loginService.checkLogin()？

📖 点击查看答案

1. B — 验证码"阅后即焚"，同一个 code 不能使用两次。
2. B — 第三方账号通过 sys_social 表与系统用户关联，不能直接拿到系统用户的 username。
3. B — 密码错误锁定是针对本系统凭证的防护。第三方/OAuth 认证的合法性由外部平台保证，本地只需检查绑定关系。
