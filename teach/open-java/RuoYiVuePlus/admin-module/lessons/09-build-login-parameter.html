第 9 课 · buildLoginParameter 参数构建

🎯 本节目标：彻底理解 SaLoginParameter buildLoginParameter() 的设计意图——它能解决什么问题、被哪些代码使用、什么时候需要自定义扩展，以及如何在你的项目中复用这种"从配置构建参数"的模式。

一、核心问题：没有 buildLoginParameter 会怎样？

假设没有这个方法，每个策略都要自己构建 SaLoginParameter：

// 在每个策略中都要重复这个代码
SaLoginParameter model = new SaLoginParameter();
model.setDeviceType(client.getDeviceType());
model.setTimeout(client.getTimeout());
model.setActiveTimeout(client.getActiveTimeout());
model.setExtra(LoginHelper.CLIENT_KEY, client.getClientId());
model.setExtra(LoginHelper.CLIENT_ACCESS_PATH_KEY, client.getAccessPath());
model.setExtra(LoginHelper.CLIENT_IP_WHITELIST_KEY, client.getIpWhitelist());
// ... 5 个策略 × 7 行 = 35 行重复代码

buildLoginParameter 的本质是提取重复的构建逻辑，所有策略共享同一个参数组装规则。

二、源码逐行解读

IAuthStrategy.java · 第 47-70 行

/**
 * 按客户端配置构建统一登录参数（无扩展）。
 */
static SaLoginParameter buildLoginParameter(SysClientVo client) {
    return buildLoginParameter(client, null);
}

/**
 * 按客户端配置构建统一登录参数，并预留自定义扩展入口。
 *
 * @param client     客户端配置（sys_client 表的 VO）
 * @param customizer 自定义扩展逻辑（Consumer 函数式接口，可传入 Lambda）
 * @return Sa-Token 登录参数
 */
static SaLoginParameter buildLoginParameter(SysClientVo client,
                                              Consumer<SaLoginParameter> customizer) {
    SaLoginParameter model = new SaLoginParameter();

    // ① 设备类型：pc、app、xcx...
    model.setDeviceType(client.getDeviceType());

    // ② token 过期时间（秒），-1 表示永不过期
    model.setTimeout(client.getTimeout());

    // ③ 无操作自动过期时间（秒）
    model.setActiveTimeout(client.getActiveTimeout());

    // ④~⑥ 扩展数据（存入 Sa-Token 的 extra 字典）
    model.setExtra(LoginHelper.CLIENT_KEY, client.getClientId());
    model.setExtra(LoginHelper.CLIENT_ACCESS_PATH_KEY, client.getAccessPath());
    model.setExtra(LoginHelper.CLIENT_IP_WHITELIST_KEY, client.getIpWhitelist());

    // ⑦ 如果调用方传入了自定义扩展逻辑，执行它
    if (ObjectUtil.isNotNull(customizer)) {
        customizer.accept(model);  // Consumer 模式：允许调用方追加自定义配置
    }

    return model;
}

三、SaLoginParameter 的配置字段对照

buildLoginParameter 设置的字段来源作用谁在使用
deviceType sys_client.device_type Sa-Token 多端隔离 Sa-Token 框架、UserLoginSuccessListener、LogAspect
timeout sys_client.timeout token 有效期 Sa-Token 框架（到期自动清除 token）
activeTimeout sys_client.active_timeout 无操作过期时间 Sa-Token 框架（长时间不操作自动登出）
extra[CLIENT_KEY] sys_client.client_id 客户端标识 UserLoginSuccessListener（写入在线缓存）、Token 鉴权时校验客户端
extra[ACCESS_PATH] sys_client.access_path 允许访问的路径 SaPermissionImpl（权限校验时过滤路径）
extra[IP_WHITELIST] sys_client.ip_whitelist IP 白名单 全局过滤器/IP 校验拦截器
关键理解：SaLoginParameter 是 Sa-Token 框架提供的"登录配置参数对象"。它不是一个"一次性的请求参数"——它被 Sa-Token 持久化到 token 的 session 中。这意味着：

① 登录时构建的参数 → ② 存入 Sa-Token session → ③ 后续任何请求都可以通过 StpUtil.getExtra(key) 读取 → ④ LoginHelper.getUserId()、LoginHelper.getUsername() 本质上就是在读这些 extra 数据

四、数据读取端：LoginHelper.login() 的对称操作

public static void login(LoginUser loginUser, SaLoginParameter model) {
    model = ObjectUtil.defaultIfNull(model, new SaLoginParameter());
    fillRequestContext(loginUser, model);
    StpUtil.login(loginUser.getLoginId(),
        model.setExtra(USER_KEY, loginUser.getUserId())         // ← 写入端
            .setExtra(USER_NAME_KEY, loginUser.getUsername())
            .setExtra(DEPT_KEY, loginUser.getDeptId())
            .setExtra(DEPT_NAME_KEY, loginUser.getDeptName())
            .setExtra(DEPT_CATEGORY_KEY, loginUser.getDeptCategory())
    );
    StpUtil.getTokenSession().set(LOGIN_USER_KEY, loginUser);
}

// 读取端
public static Long getUserId() {
    return Convert.toLong(getExtra(USER_KEY));  // ← 读取端：从 extra 中取回
}
private static Object getExtra(String key) {
    return StpUtil.getExtra(key);               // ← Sa-Token API
}

五、Consumer 自定义扩展机制

buildLoginParameter 的第二个重载接受一个 Consumer<SaLoginParameter>，这允许特殊策略在标准配置之外追加自己的参数。例如：

// 场景：XcxAuthStrategy 需要把小程序 openid 也写入登录参数
SaLoginParameter model = IAuthStrategy.buildLoginParameter(client, p -> {
    p.setExtra("openid", openid);             // 自定义扩展
    p.setExtra("sessionKey", sessionKey);     // 微信 session_key
});

// 不使用扩展的常见调用（5 个策略中有 4 个用这个）：
SaLoginParameter model = IAuthStrategy.buildLoginParameter(client);

这种双参数重载的设计（无 Consumer vs 有 Consumer）是建造者模式（Builder Pattern）的轻量级替代，适用于参数构建相对简单的场景。

六、何时获取 buildLoginParameter 中的数据？

数据并非只在登录时使用。由于 Sa-Token 将 extra 数据持久化到 token session，任何持有 token 的请求都可以读取：

时机获取方式使用场景
登录时直接使用 SaLoginParameter 对象LoginHelper.login()、UserLoginSuccessListener
任意请求LoginHelper.getUserId() 等静态方法Controller、Service 中获取当前登录用户
权限校验StpUtil.getExtra(LoginHelper.CLIENT_KEY)SaPermissionImpl 校验接口权限
过滤器/拦截器StpUtil.getExtra(LoginHelper.CLIENT_IP_WHITELIST_KEY)IP 白名单校验
在线用户管理loginParameter.getExtra(LoginHelper.CLIENT_KEY)从 Redis 读取在线用户 DTO 并展示
