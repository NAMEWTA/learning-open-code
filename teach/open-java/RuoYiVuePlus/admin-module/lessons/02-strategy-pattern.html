第 2 课 · 认证策略工厂 — IAuthStrategy 与路由分发 核心架构

🎯 本节目标：学完你能完整追踪一次登录请求从 AuthController 到具体策略实现类的分发链路，能独立新增第六种认证方式而无需改动任何已有代码。

上一课我们看到了启动入口和配置全景。现在深入核心：一个 /auth/login 请求进来后，系统怎么决定用密码、短信、邮箱、第三方还是小程序来认证？ 答案是：策略模式 + Spring Bean 命名约定。

一、请求入口：AuthController.login()

先看控制器如何接收和处理登录请求：

@SaIgnore                              // 登录接口不需要鉴权
@PostMapping("/login")
@ApiEncrypt                            // 请求体加密传输
public R<LoginVo> login(@RequestBody String body) {
    // 1. 解析 JSON → LoginBody
    LoginBody loginBody = JsonUtils.parseObject(body, LoginBody.class);
    ValidatorUtils.validate(loginBody);

    // 2. 提取两个关键字段
    String clientId = loginBody.getClientId();   // 客户端标识
    String grantType = loginBody.getGrantType();  // 授权类型

    // 3. 校验客户端合法性与状态
    SysClientVo client = clientService.queryByClientId(clientId);
    if (ObjectUtil.isNull(client) || !StringUtils.contains(client.getGrantType(), grantType)) {
        return R.fail("授权类型不正确!");
    }
    if (!SystemConstants.NORMAL.equals(client.getStatus())) {
        return R.fail("授权类型已被停用!");
    }

    // 4. 策略分发 — 关键一步
    LoginVo loginVo = IAuthStrategy.login(body, client, grantType);

    // 5. 延时 5 秒发送欢迎消息
    scheduledExecutorService.schedule(() -> {
        messageService.publishMessage(...);
    }, 5, TimeUnit.SECONDS);

    return R.ok(loginVo);
}
⚠️ 注意 @SaIgnore：整个 AuthController 类上标注了 @SaIgnore，意思是这个 Controller 的所有接口都不需要登录就能访问。这是必需的——用户还没登录呢，不能要求登录态。其他 Controller（如 system 模块的接口）则默认需要鉴权。

二、策略接口：IAuthStrategy 的设计

这是整个认证架构的灵魂。只有 81 行代码，但设计精妙：

public interface IAuthStrategy {

    String BASE_NAME = "AuthStrategy";  // Bean 名称后缀

    // ── 静态工厂方法：核心路由逻辑 ──
    static LoginVo login(String body, SysClientVo client, String grantType) {
        String beanName = grantType + BASE_NAME;  // 例: "passwordAuthStrategy"
        if (!SpringUtils.containsBean(beanName)) {
            throw new ServiceException("授权类型不正确!");
        }
        IAuthStrategy instance = SpringUtils.getBean(beanName);
        return instance.login(body, client);
    }

    // ── 构建 Sa-Token 登录参数 ──
    static SaLoginParameter buildLoginParameter(SysClientVo client) { ... }
    static SaLoginParameter buildLoginParameter(SysClientVo client, Consumer<SaLoginParameter> customizer) { ... }

    // ── 子类必须实现的登录方法 ──
    LoginVo login(String body, SysClientVo client);
}
1 Bean 名称约定：grantType + "AuthStrategy"。例如前端传 grantType: "password"，则寻找名为 passwordAuthStrategy 的 Bean。每个策略实现类用 @Service("passwordAuthStrategy") 注册。IAuthStrategy.java:33
2 安全校验：SpringUtils.containsBean(beanName) 先检查 Bean 是否存在，不存在则抛异常。防止传入任意 grantType 值。注意这里 不是 applicationContext.getBean() 直接拿——先校验再取，更安全。IAuthStrategy.java:34-36
3 实例方法委托：拿到策略实例后，调用其 login(body, client) 实例方法。每个策略自己负责解析 body 为具体类型（如 PasswordLoginBody、SmsLoginBody）、执行校验、构建 LoginUser、签发 token。IAuthStrategy.java:37-38
💡 Java 接口中可以有静态方法吗？

Java 8 引入了接口静态方法。这里 IAuthStrategy.login() 是一个静态工厂方法，不属于任何实例。它的作用是"知道去哪找策略实现"，而不是自己完成登录。这比传统的 AuthStrategyFactory 类更内聚——工厂逻辑和策略契约放在同一个接口里。

五、共享方法：buildLoginParameter

所有策略在签发 token 前都调用同一个方法构建登录参数：

static SaLoginParameter buildLoginParameter(SysClientVo client, Consumer<SaLoginParameter> customizer) {
    SaLoginParameter model = new SaLoginParameter();
    model.setDeviceType(client.getDeviceType());                 // PC / App / 小程序
    model.setTimeout(client.getTimeout());                       // token 总有效期
    model.setActiveTimeout(client.getActiveTimeout());           // 无操作超时
    model.setExtra(LoginHelper.CLIENT_KEY, client.getClientId());// 客户端标识
    model.setExtra(LoginHelper.CLIENT_ACCESS_PATH_KEY, client.getAccessPath());
    model.setExtra(LoginHelper.CLIENT_IP_WHITELIST_KEY, client.getIpWhitelist());
    if (ObjectUtil.isNotNull(customizer)) {
        customizer.accept(model);  // 策略可以追加自定义参数
    }
    return model;
}
💡 为什么用 Consumer 回调而不是子类重写？ 这个方法在接口中定义为静态方法，不能被子类重写。通过 Consumer<SaLoginParameter> 参数，每个策略可以在调用时传入 lambda 来追加自己的逻辑（比如小程序策略追加 openid）。这是函数式编程在策略模式中的优雅应用。

七、客户端校验的两道防线

在策略分发之前，AuthController.login() 做了两层前置校验：

校验代码失败返回
clientId 是否存在 + grantType 是否在客户端的授权列表内ObjectUtil.isNull(client) || !StringUtils.contains(client.getGrantType(), grantType)"授权类型不正确!"
客户端状态是否正常!SystemConstants.NORMAL.equals(client.getStatus())"授权类型已被停用!"

这意味着：即使实现类存在，如果该客户端没有被授权使用某种登录方式，请求也会被拦截。这是一个运行时特征——不需要重启应用，只需修改数据库 sys_client 表。
