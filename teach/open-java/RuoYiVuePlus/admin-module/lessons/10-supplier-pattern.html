第 10 课 · Supplier<Boolean> 延迟执行

🎯 本节目标：彻底掌握 Supplier<Boolean> 在 SysLoginService.checkLogin() 中的作用——为什么传的是 Lambda 而不是 boolean 值？理解延迟执行（Lazy Evaluation）和模板方法回调模式，并能识别此模式在你日常编码中的应用场景。

一、核心问题：为什么传 Supplier 而不是 boolean？

先看调用代码：

PasswordAuthStrategy.java · 第 70 行

loginService.checkLogin(LoginType.PASSWORD, username,
    () -> !BCrypt.checkpw(password, user.getPassword()));  // ← 传入 Lambda，不是直接传 boolean

你可能会问：为什么不直接先执行 BCrypt.checkpw()，然后把 boolean 传进去？

❌ 如果直接传 boolean： 
// 先执行密码校验
boolean failed = !BCrypt.checkpw(
    password, user.getPassword());

// 然后调用 checkLogin
loginService.checkLogin(
    LoginType.PASSWORD, username, failed);

// 问题：即使用户已被锁定，
// BCrypt.checkpw() 仍然执行了！
// BCrypt 是 CPU 密集操作，
// 白白浪费计算资源。
✅ 使用 Supplier 延迟执行： 
loginService.checkLogin(
    LoginType.PASSWORD, username,
    () -> !BCrypt.checkpw(password,
        user.getPassword()));

// checkLogin 内部先检查锁定状态：
public void checkLogin(..., Supplier<Boolean> s) {
    // ① 先检查是否已被锁定
    if (errorNumber >= maxRetryCount) {
        throw new UserException(...);
        // ← 这里就抛异常了，
        //    BCrypt.checkpw() 根本没有执行！
    }

    // ② 没被锁定时，才执行密码校验
    if (s.get()) { ... }
}

核心原因：密码验证（BCrypt）是一个 CPU 密集操作。如果用户已经被锁定（10 次错误后锁 10 分钟），浪费时间再做密码比对毫无意义。用 Supplier 将密码校验"包裹"成一个延迟执行的代码块，让 checkLogin 方法自己决定是否需要执行它。

二、Supplier 是什么？

Supplier<T> 是 Java 8 引入的函数式接口：

@FunctionalInterface
public interface Supplier<T> {
    T get();  // 无参数，返回一个值
}

它本质上是一个"值的工厂"——你给我一个 Supplier，我可以在任何时候调用 get() 来获取值。关键特性：get() 被调用时，计算才发生。

对比三种方式： ① 直接传值： checkLogin(type, username, true) → 调用者已经算出了 true，checkLogin 只是读取一个已知值 ② 传 Supplier（Lambda）： checkLogin(type, username, () -> expensiveOperation()) → 调用者传的是一个"计算配方"，checkLogin 决定何时/是否执行 → expensiveOperation() 可能完全不被执行！ ③ 传方法引用： checkLogin(type, username, this::validatePassword) → 和 Lambda 一样，只是写法不同

三、完整执行流程追踪

SysLoginService.java · 第 200-229 行

public void checkLogin(LoginType loginType, String username, Supplier<Boolean> supplier) {
    String errorKey = CacheNames.PWD_ERR_CNT_KEY + username;
    String loginFail = Constants.LOGIN_FAIL;

    // 步骤①：从 Redis 获取错误次数
    int errorNumber = ObjectUtil.defaultIfNull(RedisUtils.getCacheObject(errorKey), 0);

    // 步骤②：是否被锁定？
    if (errorNumber >= maxRetryCount) {
        // ← 锁定状态下，supplier.get() 永远不会被调用！
        recordLoginInfo(username, loginFail, ...);
        throw new UserException(loginType.getRetryLimitExceed(), maxRetryCount, lockTime);
    }

    // 步骤③：锁定检查通过，才执行密码/验证码校验
    if (supplier.get()) {  // ← 这里才真正调用 Lambda
        // 步骤④：校验失败，错误次数+1
        errorNumber++;
        RedisUtils.setCacheObject(errorKey, errorNumber, Duration.ofMinutes(lockTime));
        if (errorNumber >= maxRetryCount) {
            // 达到最大重试 → 锁定
            recordLoginInfo(username, loginFail, ...);
            throw new UserException(loginType.getRetryLimitExceed(), ...);
        } else {
            // 未达最大重试 → 提示剩余次数
            recordLoginInfo(username, loginFail, ...);
            throw new UserException(loginType.getRetryLimitCount(), errorNumber);
        }
    }

    // 步骤⑤：校验成功（supplier.get() 返回 false），清空错误计数
    RedisUtils.deleteObject(errorKey);
}
时间线： 时刻 T1: PasswordAuthStrategy 调用 checkLogin(PASSWORD, "admin", () -> !BCrypt.checkpw(pwd, hash)) 时刻 T2: checkLogin 方法开始执行 ├─ T2.1: 从 Redis 取错误计数 = 10（已达上限） ├─ T2.2: 检测到被锁定 → 抛出 UserException └─ T2.3: BCrypt.checkpw() 从未被调用！✅ --- 如果是另一个场景（未锁定）--- 时刻 T1: PasswordAuthStrategy 调用 checkLogin(PASSWORD, "admin", () -> !BCrypt.checkpw(pwd, hash)) 时刻 T2: checkLogin 方法开始执行 ├─ T2.1: 从 Redis 取错误计数 = 3（未达上限） ├─ T2.2: 未锁定，继续 ├─ T2.3: supplier.get() → 这才调用 BCrypt.checkpw() ← 延迟！ └─ T2.4: 根据结果处理

四、三个策略传入的不同 Supplier

策略Supplier 内容返回 true 的含义
PasswordAuthStrategy () -> !BCrypt.checkpw(password, hash) 密码不匹配 → 校验失败
SmsAuthStrategy () -> !validateSmsCode(phone, code) 短信验证码不匹配 → 校验失败
EmailAuthStrategy () -> !validateEmailCode(email, code) 邮箱验证码不匹配 → 校验失败
⚠️ 注意约定：supplier.get() 返回 true 表示认证失败。这是因为代码中用 if (supplier.get()) 来判断"是否需要记录错误"。这是一种常见的布尔语义反转模式——将 "是否成功" 反转为 "是否失败"，使得 if 分支自然对应错误处理逻辑。

五、这就是「模板方法 + 回调」模式

checkLogin 方法是模板方法模式的现代实践：

模板方法（Template Method）的伪代码： abstract class LoginChecker { // 模板方法：固定算法骨架 public final void checkLogin(username, ...) { if (isLocked(username)) { // ← 通用逻辑 throw LockedException; } if (doAuthenticate(username)) { // ← 可变步骤（子类实现） recordError(username); // ← 通用逻辑 throw AuthException; } clearErrors(username); // ← 通用逻辑 } abstract boolean doAuthenticate(...); // ← 子类覆盖此方法 } // RuoYi 使用 Supplier 回调替代了抽象方法： public void checkLogin(String username, Supplier<Boolean> supplier) { if (isLocked(username)) { ... } // 通用逻辑 if (supplier.get()) { ... } // ← 回调替代抽象方法 clearErrors(username); // 通用逻辑 }

为什么用 Supplier 而不是抽象类？

  1. 更灵活：不需要为每种认证方式创建子类
  2. 更简洁：一行 Lambda 替代一个实现类
  3. 与现有策略模式兼容：策略已经用 IAuthStrategy 接口实现，再套一层抽象类会导致继承层次过深

六、Supplier 在你的项目中的应用场景

当你的方法结构是"先检查 → 再执行 → 后清理"时，考虑用 Supplier 延迟执行"再执行"那一步：

// 场景1：缓存查询
// 先查缓存，缓存没有才执行昂贵的数据库查询
public <T> T getFromCacheOrCompute(String key, Supplier<T> dbQuery) {
    T cached = redis.get(key);
    if (cached != null) {
        return cached;  // ← 缓存命中，dbQuery.get() 不会执行
    }
    T result = dbQuery.get();  // ← 缓存未命中，才查数据库
    redis.set(key, result, Duration.ofMinutes(10));
    return result;
}

// 场景2：重试逻辑
public <T> T retryOnFailure(int maxRetries, Supplier<T> operation) {
    for (int i = 0; i < maxRetries; i++) {
        try {
            return operation.get();  // ← 每次重试才执行
        } catch (Exception e) {
            if (i == maxRetries - 1) throw e;
        }
    }
    return null;
}

// 场景3：权限校验 + 业务执行
public void executeIfAuthorized(String permission, Supplier<Boolean> action) {
    if (!hasPermission(permission)) {
        throw new UnauthorizedException();
    }
    action.get();  // ← 有权限时才执行业务逻辑
}
