第 12 课 · 会话存储架构 深度解析

🎯 本节目标：彻底搞懂四个深度问题——① SaLoginParameter 的完整参数表与内在逻辑，以及它和 LoginUser 为什么"看起来重复"却不冗余；② user_type 的设计初衷和同一用户的多身份问题；③ Convert.toLong() 的类型转换机制；④ StpUtil.getTokenSession() 的底层结构、数据存取全链路，以及如何在后续请求中取回数据。

一、SaLoginParameter 全景：不只是"参数对象"

1.1 全部 18 个字段分类

SaLoginParameter 是 Sa-Token 框架的登录配置对象，不是 RuoYi 定义的。它在 StpUtil.login(loginId, parameter) 时传入，控制本次登录的全部行为。按职责分为四类：

1.2 RuoYi 用了哪些字段？

在 IAuthStrategy.buildLoginParameter() 中只设置了其中 5 个：

static SaLoginParameter buildLoginParameter(SysClientVo client, Consumer<SaLoginParameter> customizer) {
    SaLoginParameter model = new SaLoginParameter();
    model.setDeviceType(client.getDeviceType());       // ← 只设了这 5 个
    model.setTimeout(client.getTimeout());             //    其余 13 个字段使用
    model.setActiveTimeout(client.getActiveTimeout()); //    Sa-Token 全局默认值
    model.setExtra(LoginHelper.CLIENT_KEY, client.getClientId());
    model.setExtra(LoginHelper.CLIENT_ACCESS_PATH_KEY, client.getAccessPath());
    model.setExtra(LoginHelper.CLIENT_IP_WHITELIST_KEY, client.getIpWhitelist());
    return model;
}
// 然后在 LoginHelper.login() 中继续追加：
model.setExtra(USER_KEY, loginUser.getUserId())       // 再追加 5 个 extra
     .setExtra(USER_NAME_KEY, loginUser.getUsername())
     .setExtra(DEPT_KEY, loginUser.getDeptId())
     .setExtra(DEPT_NAME_KEY, loginUser.getDeptName())
     .setExtra(DEPT_CATEGORY_KEY, loginUser.getDeptCategory())
💡 没有设置的字段（isConcurrent、isShare、maxLoginCount 等） 全部走 Sa-Token 全局配置文件 sa-token.timeout、sa-token.is-concurrent 等全局默认值。这是"全局配置 + 客户端级覆盖"的两级配置策略。

二、LoginUser 与 SaLoginParameter：不会重复吗？

你问"不会重复吗？"——答案是：部分字段确实存在于两个对象中，但用途不同，不是冗余。

2.2 为什么 deviceType 在两处都出现？

这是最典型的"看似重复"——deviceType 同时存在于 LoginUser 和 SaLoginParameter：

// deviceType 的三份副本：

// ① LoginUser 中的（登录态上下文，用于后续请求读取）
loginUser.setDeviceType(client.getDeviceType());

// ② SaLoginParameter 中的（传给 Sa-Token 框架，控制多端隔离）
model.setDeviceType(client.getDeviceType());

// ③ LoginHelper.fillRequestContext() 兜底逻辑
//    如果 loginUser.deviceType 为空，从 model 回填
if (StringUtils.isBlank(loginUser.getDeviceType())
    && StringUtils.isNotBlank(model.getDeviceType())) {
    loginUser.setDeviceType(model.getDeviceType());
}

为什么需要两份？

  1. SaLoginParameter.deviceType 是给 Sa-Token 框架用的——框架用它来决定是否允许同端多登、是否顶替旧会话。这是框架级别的行为控制。
  2. LoginUser.deviceType 是给业务代码用的——日志切面记录操作来自哪个设备、在线用户列表展示设备图标。这是业务级别的数据展示。

两者本质上是从同一个源（sys_client.device_type）复制的同一个值，但作用的系统层级不同。这就像你把同一个地址写在信封上（给邮局用）和备忘录上（给自己看）——不是冗余，是不同的使用场景。

2.3 真正"重复"的数据——extra vs TokenSession

同一个 userId=1 被存了两次： 位置一：Token 的 extra 数据（JWT payload 中） StpUtil.login(loginId, model.setExtra("userId", 1)) → 后续通过 StpUtil.getExtra("userId") 读取 → LoginHelper.getUserId() 实际就是读这里 位置二：TokenSession 的 dataMap StpUtil.getTokenSession().set("loginUser", loginUser) → loginUser 对象中包含 userId=1 → 后续通过 LoginHelper.getLoginUser().getUserId() 读取

这确实是冗余，但是有意为之的设计选择：

读 extra读 TokenSession
API：StpUtil.getExtra("userId")API：session.get("loginUser").getUserId()
性能：极快（JWT token 自带，无需查 Redis）性能：需要查 Redis（TokenSession 是 Redis key）
适用：高频读取的简单字段（userId、username）适用：低频读取的复杂对象（角色、权限、部门）
限制：数据量有限（token 会变大）限制：无（存 Redis，数据量灵活）
核心设计思想：LoginHelper 将高频读取的字段（userId、username、deptId）存入 token 的 extra 数据（JWT 模式），实现无 Redis 查询的快速获取；而完整 LoginUser（包含角色列表、权限集合、部门信息等大对象）存入 TokenSession（Redis），按需获取。getUserId() 每秒可能调用数百次，每次都查 Redis 是不必要的开销。

三、user_type 的设计初衷

3.1 UserType 枚举定义

UserType.java · ruoyi-common-core/.../enums/UserType.java

public enum UserType {
    SYS_USER("sys_user"),    // 后台管理系统用户
    APP_USER("app_user");    // 移动客户端用户
}

3.2 loginId 的构成规则

LoginUser.java · 第 141-148 行

public String getLoginId() {
    if (userType == null) throw new IllegalArgumentException("用户类型不能为空");
    if (userId == null) throw new IllegalArgumentException("用户ID不能为空");
    return userType + ":" + userId;  // 例如："sys_user:1"、"app_user:100"
}

3.3 设计意图

user_type 不是用户的一个"标签"，而是 Sa-Token 多账户体系 的隔离键。它的设计目标是：

同一张 sys_user 表中： user_id=1 user_type="sys_user" → 后台管理员 user_id=1 user_type="app_user" → 同一个人的移动端身份 这两个 loginId 在 Sa-Token 中完全独立： "sys_user:1" → Token-A，角色=超级管理员，权限=全部 "app_user:1" → Token-B，角色=普通用户，权限=受限 它们是两个不同的"登录实体"，各有独立的 token、session、权限

这种设计解决的现实问题：

  1. 同一张用户表，不同登录入口：后台管理（Web）和移动端 APP 共用 sys_user 表，但权限体系完全不同
  2. 登录态隔离：管理员在后台的操作不应影响他在 APP 上的登录状态
  3. 权限分级：同样是 userId=1，作为 sys_user 有全部权限，作为 app_user 只有查询权限

3.4 同一个用户可以拥有多个 userType 吗？

在这个系统中，一个物理用户只有一种 userType。 user_type 是 sys_user 表中的一个字段，每个用户记录只有固定的一种类型。从 LoginUser.getLoginId() 的实现看，它直接读取 this.userType，不会返回多个类型。

💡 技术上说：你可以在 sys_user 表中插入两条记录（userId=1, userType=sys_user 和 userId=1, userType=app_user），这样同一个"人"就有了两个身份。但 RuoYi 的设计是每个用户记录只有一种 userType——后台用户和 APP 用户通常是不同的物理账号。如果你想实现"一个用户多条 type 记录"，需要自行扩展 LoginUser.getLoginId() 的逻辑。

3.5 LoginHelper 类上的官方注释

/**
 * user_type 为 用户类型 同一个用户表 可以有多种用户类型 例如 pc,app
 * device 为 设备类型 同一个用户类型 可以有 多种设备类型 例如 web,ios
 * 可以组成 用户类型与设备类型多对多的 权限灵活控制
 *
 * 多用户体系 针对 多种用户类型 但权限控制不一致
 * 可以组成 多用户类型表与多设备类型 分别控制权限
 */

这是一个预留的扩展点。当前实现只有两种 UserType，但架构设计上支持未来扩展更多类型（如商户、供应商、合作伙伴等），每种类型有独立的权限体系。

四、Convert.toLong() — Hutool 的类型安全转换

LoginHelper.java · 第 149 行

public static Long getUserId() {
    return Convert.toLong(getExtra(USER_KEY));
    //     ↑             ↑
    //     │             └─ getExtra() 返回 Object（实际上是 Integer 或 Long）
    //     └─ Convert.toLong() 安全转换为 Long
}

Convert.toLong() 是 Hutool 工具库的核心方法，位于 cn.hutool.core.convert.Convert。它接受任意类型的参数，尽最大努力转换为 Long：

输入类型输入值Convert.toLong() 结果
Long1L1L（原样返回）
Integer11L
String"123"123L
String"abc"抛出异常
nullnullnull
Double1.91L（截断，非四舍五入）
Booleantrue1L
为什么需要它？StpUtil.getExtra("userId") 返回 Object——你存入时是 Long，但取回时类型信息丢失了。而且不同场景可能存入 Integer 或 Long，手动 (Long) getExtra(key) 可能抛出 ClassCastException。Convert.toLong() 能安全处理各种数字类型和 null，是防御性编程的体现。

五、StpUtil.getTokenSession().set() 深度拆解

5.1 这段代码到底做了什么？

StpUtil.getTokenSession().set(LOGIN_USER_KEY, loginUser);
// 拆成三步理解：

// 步骤1：StpUtil.getTokenSession()
//   获取【当前请求 token 对应的 TokenSession】
//   TokenSession 是 SaSession 实例，内部有一个 dataMap（ConcurrentHashMap）

// 步骤2：.set(LOGIN_USER_KEY, loginUser)
//   等价于 session.getDataMap().put("loginUser", loginUser)
//   然后调用 session.update() 将变更持久化到 Redis

// 步骤3：Key = "loginUser"，Value = 完整的 LoginUser 对象（含角色、权限）

5.2 getTokenSession() 返回的到底是什么？

SaSession 是 Sa-Token 框架的核心数据结构。它有三种类型：

SaSession 三种类型对比： ┌─────────────────────────────────────────────────────────────┐ │ Account-Session（账号会话） │ │ Key: accountSession:{loginId} 例如 accountSession:sys_user:1│ │ 生命周期：与账号绑定，同一账号所有设备共享 │ │ 用途：存储跨设备的全局数据 │ └─────────────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────────────┐ │ Token-Session（令牌会话）← getTokenSession() 返回这个 │ │ Key: tokenSession:{tokenValue} 例如 tokenSession:abc123... │ │ 生命周期：与 token 绑定，每个设备独立 │ │ 用途：存储当前设备的登录态数据 ← RuoYi 存 LoginUser 在这 │ └─────────────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────────────┐ │ Custom-Session（自定义会话） │ │ Key: customSession:{customKey} │ │ 用途：任意自定义维度的数据存储 │ └─────────────────────────────────────────────────────────────┘

为什么 RuoYi 选 Token-Session 而非 Account-Session？

  • 如果用 Account-Session，同一账号在 PC、手机、小程序上的 LoginUser 会互相覆盖
  • Token-Session 与 token 一一绑定，每个设备有独立的 LoginUser，互不干扰
  • 当你踢出一个设备时，只删除对应的 Token-Session，不影响其他设备

5.3 SaSession 的内部结构

SaSession {
    id:           "tokenSession:abc123def456..."  // 唯一标识
    type:         "Token"                          // 类型
    loginType:    "login"                          // 登录类型
    loginId:      "sys_user:1"                     // 登录ID
    token:        "abc123def456..."                // token 值
    createTime:   1750000000000                    // 13位时间戳
    dataMap:      ConcurrentHashMap {               // ← 核心：数据存储
        "loginUser" → LoginUser对象,               // RuoYi 存入的
        // 可以存更多 key-value...
    }
    terminalList: [SaTerminalInfo, ...]             // 终端信息列表
}

5.4 后续如何取回 LoginUser？

登录时存入了，后续任何请求都可以通过以下方式取回：

// 方式 1：LoginHelper.getLoginUser() — 最常用（推荐）
LoginUser user = LoginHelper.getLoginUser();
// ↓ 内部实现：
//   StpUtil.getTokenSession()          → 拿到当前请求的 TokenSession
//   session.get("loginUser")           → 从 dataMap 中取出 LoginUser
//   返回 (LoginUser) session.get("loginUser")

// 方式 2：通过指定 token 获取（管理员查看在线用户时使用）
LoginUser user = LoginHelper.getLoginUser("tokenValueHere");
// ↓ 内部实现：
//   StpUtil.getTokenSessionByToken("tokenValueHere")
//   → session.get("loginUser")

// 方式 3：直接读取部分字段（高频场景，绕过 TokenSession）
Long userId = LoginHelper.getUserId();        // StpUtil.getExtra("userId")
String username = LoginHelper.getUsername();   // StpUtil.getExtra("userName")

5.5 全链路数据存取图

【写入（登录时）】 LoginHelper.login(loginUser, model) │ ├─► StpUtil.login(loginId, model.setExtra("userId", 1) │ .setExtra("userName", "admin") ...) │ │ │ └─► Sa-Token 内部： │ ├─ 生成 token 字符串 │ ├─ extra 数据写入 token payload（JWT模式） │ └─ 触发 SaTokenListener.doLogin() │ └─► StpUtil.getTokenSession() ← 获取/创建 TokenSession .set("loginUser", loginUser) ← 存入 dataMap │ └─► Redis: SET tokenSession:abc123... → 序列化的 SaSession 【读取（后续请求）】 请求携带 token: abc123... 到达服务器 │ ├─► LoginHelper.getUserId() │ └─► StpUtil.getExtra("userId") → 从 token 中解析（无 Redis 查询！） │ ├─► LoginHelper.getUsername() │ └─► StpUtil.getExtra("userName") → 从 token 中解析 │ └─► LoginHelper.getLoginUser() └─► StpUtil.getTokenSession() → Redis: GET tokenSession:abc123... └─► session.get("loginUser") → 从 dataMap 取出完整的 LoginUser └─► 返回 LoginUser 对象（含角色、权限等完整数据）

正确答案：1、2、4
