第 8 课 · Java record 事件对象 Java 17

🎯 本节目标：理解 Java 17 record 关键字的语法、语义和设计意图，为什么 RuoYi 用 public record UserLoginSuccessEvent(...) 替代传统的 POJO 事件类，以及 record 在事件/DT O 场景下的最佳实践。

A. 快问快答：record 是什么？

record 是 Java 17 正式发布的一种特殊类。它用一行代码替代传统 JavaBean 的几十行样板代码，自动生成：构造函数、equals()、hashCode()、toString() 和命名的访问器方法。record 是不可变的（immutable）——所有字段都是 final，创建后不能修改。

二、传统 Event 类 vs record — 代码对比

❌ 传统写法（20+ 行） 
public class UserLoginSuccessEvent {
    private Object loginId;
    private String tokenValue;
    private SaLoginParameter loginParameter;

    public UserLoginSuccessEvent(
        Object loginId, String tokenValue,
        SaLoginParameter loginParameter) {
        this.loginId = loginId;
        this.tokenValue = tokenValue;
        this.loginParameter = loginParameter;
    }

    public Object getLoginId() {
        return loginId;
    }
    public String getTokenValue() {
        return tokenValue;
    }
    public SaLoginParameter getLoginParameter() {
        return loginParameter;
    }
    // 还需要 equals, hashCode, toString...
}
✅ record 写法（1 行） 
public record UserLoginSuccessEvent(
    Object loginId,
    String tokenValue,
    SaLoginParameter loginParameter
) {
}

record 自动生成的所有方法：

// 1. 规范构造函数（canonical constructor）
new UserLoginSuccessEvent(loginId, tokenValue, loginParameter)

// 2. 访问器方法（命名就是字段名，不带 get 前缀！）
event.loginId()        // ← 不是 event.getLoginId()！
event.tokenValue()
event.loginParameter()

// 3. equals() — 基于所有字段的值比较
event1.equals(event2)

// 4. hashCode() — 基于所有字段的哈希
event.hashCode()

// 5. toString() — 自动格式化
// 输出: UserLoginSuccessEvent[loginId=sys_user:1, tokenValue=..., loginParameter=...]
⚠️ 注意命名约定：record 的访问器方法名不带 get 前缀！是 event.loginId()，不是 event.getLoginId()。这是初学者最容易犯的错误。Spring 框架能够自动适配 record 的访问器命名。

三、在 UserLoginSuccessListener 中的使用

UserLoginSuccessListener.java · 第 42-68 行

@EventListener
public void handleLoginSuccess(UserLoginSuccessEvent event) {
    // 通过 record 访问器读取数据（注意：没有 get 前缀！）
    SaLoginParameter loginParameter = event.loginParameter();    // ← record 访问器
    String tokenValue = event.tokenValue();                       // ← record 访问器

    // 从 loginParameter 中读取额外数据
    String username = (String) loginParameter.getExtra(LoginHelper.USER_NAME_KEY);

    // 构建在线用户 DTO...
    dto.setTokenId(tokenValue);
    dto.setUserName(username);
    // ...

    // 使用 event 中的数据
    loginService.updateLastLoginInfo(
        (Long) loginParameter.getExtra(LoginHelper.USER_KEY), ip);

    log.info("user doLogin, userId:{}, token:***{}",
        event.loginId(), StringUtils.right(tokenValue, 8));       // ← record 访问器
}

四、为什么用 record 做事件对象？

record 特性为什么适合事件对象
不可变事件是"已发生的事实"，发布后不应被修改。record 强制不可变性。
自动 equals/hashCode事件可能需要去重、缓存、比较，自动生成的方法保证正确性。
简洁事件通常只有数据没有行为，一行声明即可，消除样板代码。
语义清晰看到 record 关键字就知道这是"数据载体"，不是"业务对象"。
透明toString() 自动输出所有字段，debug 时非常方便。

五、record 使用场景 vs 不使用场景

✅ 适合用 record❌ 不适合用 record
Spring 事件对象（Event）JPA/Hibernate Entity
DTO / VO（数据传输对象）需要继承的类（record 不能继承类）
配置属性载体需要字段可变的 Bean
方法返回多个值（替代 Pair/Tuple）需要懒加载或代理的类
Map 的复合 Key需要自定义 setter 逻辑的类
不可变的数据快照Spring MVC 表单绑定对象（需要无参构造）
💡 RuoYi 中的实践：项目中所有 6 个 record 全部用于事件对象——UserLoginSuccessEvent、OnlineUserCleanEvent、OssConfigChangeEvent、WorkflowCopyEvent、WorkflowTaskMessageEvent、WorkflowResultMessageEvent。这是一个强烈的信号：Spring 事件 = record 的最佳场景。

六、record 不可变性的深层意义

为什么事件要不可变？考虑这个场景：

// 如果事件可变会出什么问题？
UserLoginSuccessEvent event = new UserLoginSuccessEvent(id, token, param);

// Listener A 处理事件...
userLoginSuccessListener.handleLoginSuccess(event);
// 如果 Listener A 不小心修改了 event 的某个字段...

// Listener B 处理同一个事件...
anotherListener.handleLoginSuccess(event);
// Listener B 读到的数据已经被 A 污染了！Bug！

在 Spring 的同步事件模型中，多个 @EventListener 按顺序处理同一个事件对象。如果事件可变，后面的 Listener 可能读到被前面 Listener 修改的数据——这是一个非常隐蔽的 bug。

record 从语言层面杜绝了这种可能。 所有字段都是 final，编译器和 JVM 保证不可变性。

七、record 的高级用法

7.1 紧凑构造函数（Compact Constructor）

可以在 record 体中定义参数校验或规范化逻辑：

// RuoYi 中的实际例子 — OnlineUserCleanEvent
public record OnlineUserCleanEvent(Long roleId, Collection<Long> userIds) {
    // 紧凑构造函数：省略参数列表，直接写校验逻辑
    public OnlineUserCleanEvent {
        // 防御性拷贝：防止外部修改传入的集合
        userIds = List.copyOf(userIds);
    }

    // 静态工厂方法（语义化构造）
    public static OnlineUserCleanEvent byRole(Long roleId) {
        return new OnlineUserCleanEvent(roleId, List.of());
    }

    public static OnlineUserCleanEvent byUsers(Collection<Long> userIds) {
        return new OnlineUserCleanEvent(null, userIds);
    }
}

7.2 添加额外方法

record 可以有自己的方法：

public record UserLoginSuccessEvent(Object loginId, String tokenValue,
                                     SaLoginParameter loginParameter) {

    // 可以添加自定义方法
    public boolean isFromMobile() {
        return "app".equals(loginParameter.getDeviceType())
            || "xcx".equals(loginParameter.getDeviceType());
    }
}

7.3 实现接口

record 可以实现接口：

public record UserLoginSuccessEvent(...) implements Serializable {
    // record 可以自动序列化
}

八、record 不能做的事

  • ❌ 不能继承类（record 隐式继承 java.lang.Record）
  • ❌ 不能有非静态字段（所有字段必须在 record 头部声明）
  • ❌ 不能有 setter（字段都是 final）
  • ❌ 不能有无参构造（除非在头部给出所有字段的默认值）
