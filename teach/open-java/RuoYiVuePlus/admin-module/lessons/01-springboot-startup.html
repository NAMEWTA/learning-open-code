二、启动入口：DromaraApplication

这是整个系统的起点，只有 28 行：

@SpringBootApplication
public class DromaraApplication {
    public static void main(String[] args) {
        SpringApplication application = new SpringApplication(DromaraApplication.class);
        application.setApplicationStartup(new BufferingApplicationStartup(2048));
        application.run(args);
        System.out.println("(♥◠‿◠)ﾉﾞ  RuoYi-Vue-Plus启动成功   ლ(´ڡ`ლ)ﾞ");
    }
}

拆解每一个细节

1 @SpringBootApplication — 一个注解包含三个能力：@SpringBootConfiguration（配置类）、@EnableAutoConfiguration（触发自动配置）、@ComponentScan（扫描当前包及子包的 Bean）。位置：DromaraApplication.java:13
2 new SpringApplication(DromaraApplication.class) — 不直接用 SpringApplication.run() 静态方法，而是先构造对象。为什么？因为需要在 run() 之前设置一些定制项。位置：DromaraApplication.java:22
3 BufferingApplicationStartup(2048) — Spring Boot 3.x 引入的应用启动性能追踪。缓冲区容量 2048 个事件，使你可以通过 Actuator 的 /actuator/startup 端点查看启动过程中每个 Bean 的初始化耗时。位置：DromaraApplication.java:23
4 application.run(args) — 触发 Spring Boot 的完整启动流程：环境准备 → ApplicationContext 创建 → 自动配置 → Bean 初始化 → 内嵌 Jetty 启动 → 应用就绪。位置：DromaraApplication.java:24
💡 为什么用 Jetty 而不是默认的 Tomcat？ 查看 application.yml 第 8-16 行，RuoYi 使用 Jetty 作为内嵌容器，并配置了线程池（最小 8，最大 256）。Jetty 在长连接场景下内存占用更小，更适合微服务网关类应用。

第二个入口：DromaraServletInitializer

public class DromaraServletInitializer extends SpringBootServletInitializer {
    @Override
    protected SpringApplicationBuilder configure(SpringApplicationBuilder application) {
        return application.sources(DromaraApplication.class);
    }
}

这个类的存在是为了支持 war 包部署到外部 Servlet 容器（如 Tomcat、Jetty 独立服务器）。如果只用内嵌容器（jar 包部署），这个类不会被用到。它继承 SpringBootServletInitializer 并指定了启动源为 DromaraApplication.class。

三、配置中心：application.yml 认证相关配置

整个 application.yml 有 370+ 行，这里只聚焦与认证直接相关的 4 个配置段：

3.1 验证码配置

captcha:
  enable: true          # 是否启用验证码校验（生产环境建议 true）
  type: math            # 验证码类型：math（数学计算）/ char（字符）
  numberLength: 1       # 数学验证码位数
  charLength: 4         # 字符验证码长度

这个配置由 CaptchaProperties 类绑定。在 PasswordAuthStrategy.validateCaptcha() 中，当 captchaEnabled = true 时才执行验证码校验。如果你的内部工具不需要验证码，把 enable 改成 false 即可全局跳过。

3.2 密码策略配置

user:
  password:
    maxRetryCount: 5    # 密码最大错误次数
    lockTime: 10        # 锁定时间（分钟）

由 SysLoginService 的 @Value 注入：

@Value("${user.password.maxRetryCount}")
private Integer maxRetryCount;

@Value("${user.password.lockTime}")
private Integer lockTime;

错误计数存储在 Redis 中，key 为 PWD_ERR_CNT_KEY + username，TTL 等于 lockTime 分钟。这意味着 锁定时间过后，错误计数自动清零，无需手动解锁。

3.3 Sa-Token 会话配置

sa-token:
  token-name: Authorization                    # token 在请求头中的名称
  is-concurrent: true                          # 是否允许同一账号并发登录
  is-share: false                              # 并发登录时是否共用一个 token
  jwt-secret-key: abcdefghijklmnopqrstuvwxyz   # JWT 签名密钥
配置项含义实际影响
is-concurrent: true允许同一账号多地同时登录两个浏览器用同一账号登录，互不踢出
is-concurrent: false新登录挤掉旧登录触发 UserActionListener.doReplaced()
is-share: false每次登录生成独立 token配合 is-concurrent: true，每个客户端有独立会话

3.4 客户端隔离：SysClientVo 的超时配置

上述 sa-token 是全局默认值。除此之外，每个客户端（sys_client 表）还有独立超时配置，在 IAuthStrategy.buildLoginParameter() 中生效：

SaLoginParameter model = new SaLoginParameter();
model.setDeviceType(client.getDeviceType());       // 设备类型
model.setTimeout(client.getTimeout());             // token 有效期（秒）
model.setActiveTimeout(client.getActiveTimeout()); // 无操作过期时间（秒）
model.setExtra(LoginHelper.CLIENT_KEY, client.getClientId());

这意味着 PC 端、App 端、小程序端可以有完全不同的会话超时策略，都从数据库的 sys_client 表读取，不需要改代码。
