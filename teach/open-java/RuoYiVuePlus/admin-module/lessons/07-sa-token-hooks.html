第 7 课 · Sa-Token 钩子机制 生命周期

🎯 本节目标：彻底搞懂三个容易混淆的问题——① 调用 LoginHelper.login() 时到底发生了什么？② UserActionListener.doLogin() 是如何被触发的？③ doLogout、doKickout、doReplaced 三者的触发条件有什么区别？

一、核心误区澄清

⚠️ 最常见的误解："调用 LoginHelper.login() 就是调用 UserActionListener.doLogin()"——大错特错！它们是间接触发关系，中间隔了 Sa-Token 框架的监听器机制。

正确的调用链是：

【你的代码】 【Sa-Token 框架内部】 【你的监听器】 LoginHelper.login(loginUser, model) │ ├─ fillRequestContext(loginUser, model) │ └─ StpUtil.login(loginId, model) ──────► Sa-Token 内部: │ ├─ 1. 创建/更新 SaSession ├─ 2. 生成 token 字符串 ├─ 3. 将 token 写入 Cookie/Header ├─ 4. 执行所有 SaTokenListener │ │ │ ├─ UserActionListener.doLogin() │ │ └─ publishEvent(UserLoginSuccessEvent) │ │ │ └─ 其他已注册的 SaTokenListener... │ └─ 5. 触发 Sa-Token 内部事件

LoginHelper 和 UserActionListener 之间没有任何直接的调用关系。 它们通过 Sa-Token 框架的监听器机制"解耦连接"。

二、SaTokenListener 是什么？

SaTokenListener 是 Sa-Token 框架提供的生命周期钩子接口。框架在关键事件发生时（登录、退出、被踢、被封禁、Session 创建/销毁等），会遍历所有已注册的实现类并逐个调用。

UserActionListener.java · ruoyi-admin/.../listener/UserActionListener.java

@Component  // ← 关键：Spring Bean，被 Sa-Token 自动发现
@Slf4j
public class UserActionListener implements SaTokenListener {

    //  登录时触发
    @Override
    public void doLogin(String loginType, Object loginId,
                        String tokenValue, SaLoginParameter loginParameter) {
        // 发布 Spring 事件 → 触发 UserLoginSuccessListener
        SpringUtils.context().publishEvent(
            new UserLoginSuccessEvent(loginId, tokenValue, loginParameter));
    }

    //  主动退出时触发
    @Override
    public void doLogout(String loginType, Object loginId, String tokenValue) {
        RedisUtils.deleteObject(CacheNames.ONLINE_TOKEN_KEY + tokenValue);
    }

    //  被管理员踢下线时触发
    @Override
    public void doKickout(String loginType, Object loginId, String tokenValue) {
        RedisUtils.deleteObject(CacheNames.ONLINE_TOKEN_KEY + tokenValue);
    }

    //  被新登录顶下线时触发
    @Override
    public void doReplaced(String loginType, Object loginId, String tokenValue) {
        RedisUtils.deleteObject(CacheNames.ONLINE_TOKEN_KEY + tokenValue);
    }
}
💡 为什么 doLogin() 里只发布事件而不直接处理？
这是一种门面模式 + 关注点分离。UserActionListener 的职责是"桥接"——将 Sa-Token 的生命周期事件翻译为 Spring 事件，让其他 Spring Bean 通过 @EventListener 各司其职。如果直接在 doLogin() 里写 100 行业务逻辑，这个类会变得不可测试、不可复用。

三、四个核心钩子的触发条件对比

钩子方法触发条件谁调用了 Sa-Token触发时机
doLogin() 调用 StpUtil.login(loginId, parameter) LoginHelper.login() → StpUtil.login() 登录成功后
doLogout() 调用 StpUtil.logout() SysLoginService.logout() → StpUtil.logout()
用户点击"退出登录"按钮		 退出成功后
doKickout() 调用 StpUtil.kickout(loginId) 管理员在"在线用户"页面点击"强退"
调用 StpUtil.kickout(tokenValue)		 被踢成功后
doReplaced() 同 loginId + 同 deviceType 再次登录 Sa-Token 框架自动检测到冲突
自动顶替旧 session		 旧 session 被顶替时

3.1 doLogin — 登录钩子

1 触发链路： PasswordAuthStrategy/SocialAuthStrategy/... → LoginHelper.login() → StpUtil.login() → SaTokenListener.doLogin()

这是五个认证策略共用的最终出口。无论用户用密码、短信、邮箱、小程序还是第三方登录，最后都走到 LoginHelper.login()，然后 Sa-Token 触发 doLogin。

3.2 doLogout — 主动退出钩子

2 触发链路： 用户点击退出 → AuthController.logout() → SysLoginService.logout() → StpUtil.logout() → SaTokenListener.doLogout() 
// SysLoginService.logout()
public void logout() {
    try {
        LoginUser loginUser = LoginHelper.getLoginUser();
        if (ObjectUtil.isNull(loginUser)) return;
        recordLoginInfo(loginUser.getUsername(), Constants.LOGOUT, ...);
    } catch (NotLoginException ignored) {
    } finally {
        StpUtil.logout();  // ← 触发 doLogout()
    }
}

3.3 doKickout — 被踢下线钩子

3 触发链路： 管理员操作 → 在线用户管理接口 → StpUtil.kickout(tokenValue) → SaTokenListener.doKickout()

doLogout 和 doKickout 在 RuoYi 中的处理逻辑是一样的（都是清理 Redis 在线缓存），但语义不同：

  • doLogout：用户自己主动退出
  • doKickout：被管理员强制踢出
  • doReplaced：被自己在新设备上的登录顶掉

3.4 doReplaced — 被顶下线钩子

4 触发链路： 同一用户 + 同一设备类型再次调用 StpUtil.login() → Sa-Token 自动顶替 → SaTokenListener.doReplaced()

这是最"隐蔽"的触发方式——你不需要显式调用任何踢人方法。Sa-Token 框架在 StpUtil.login() 内部检查：这个 loginId + deviceType 是否已有活跃 session？如果有，自动顶替旧的。

场景：张三在 Chrome 上登录了后台（deviceType=pc） 张三在 Edge 上也登录了后台（deviceType=pc） 第一次 login → 创建 token-A，doLogin() 第二次 login → Sa-Token 检测到 loginId=sys_user:1, deviceType=pc 已存在 → 自动顶替 token-A，创建 token-B → 触发 doReplaced()（token-A 的监听器收到通知） → 触发 doLogin()（token-B 的监听器收到通知）

四、嵌套关系完整追踪

你问的"嵌套了一层 UserLoginSuccessListener"——现在应该清楚了：

第一层：Sa-Token 框架层钩子 SaTokenListener.doLogin() → UserActionListener 职责：将 Sa-Token 事件"翻译"为 Spring 事件 │ publishEvent(new UserLoginSuccessEvent(...)) ▼ 第二层：Spring 事件层监听器 @EventListener → UserLoginSuccessListener.handleLoginSuccess() 职责：执行具体的业务副作用（在线缓存、登录日志、IP解析） │ recordLoginInfo() → publishEvent(LoginInfoEvent) ▼ 第三层：异步持久化层 @EventListener + @Async → SysLoginInfoServiceImpl 职责：将登录日志写入数据库 sys_login_info 表

五、为什么这样设计？

设计决策理由
用 SaTokenListener 而不是直接调用解耦——LoginHelper 不需要知道"登录成功后要做什么"，Sa-Token 框架管理所有钩子
doLogin 里只发 Spring 事件单一职责——UserActionListener 只管"桥接"，不管"业务"
业务由 @EventListener 处理可扩展——将来加新功能只需加新的 Listener，不动现有代码（开闭原则）
日志写入用 @Async不阻塞主流程——记录登录日志不应拖慢用户登录响应
