第 11 课 · 设计模式全景总结 架构内功

🎯 本节目标：系统梳理 admin-module 认证链路中使用的全部设计模式——每种模式在哪里出现、解决什么问题、如何识别、以及在你自己项目中的适用场景。这是 11 节课的"总复习"。

admin-module 的认证链路展示了现代 Java 企业级开发中设计模式的最佳实践——不是教科书式的生搬硬套，而是每种模式精确地解决一个问题。

一、模式全景地图

二、逐一详解

1. 策略模式（Strategy Pattern） GoF 行为型

IAuthStrategy 接口 + 5 个实现类

解决的问题：多种认证方式（密码、短信、邮箱、小程序、第三方）有不同的验证逻辑，但控制器不应关心这些差异。

// 策略接口
public interface IAuthStrategy {
    LoginVo login(String body, SysClientVo client);
}

// 5 个具体策略
@Service("passwordAuthStrategy")  // PasswordAuthStrategy
@Service("smsAuthStrategy")       // SmsAuthStrategy
@Service("emailAuthStrategy")     // EmailAuthStrategy
@Service("xcxAuthStrategy")       // XcxAuthStrategy
@Service("socialAuthStrategy")    // SocialAuthStrategy

// 新增策略：只需加一个 @Service("ldapAuthStrategy") 实现类即可
// 无需修改 AuthController 或任何已有策略代码（开闭原则）

识别特征：一个接口，多个实现类，运行时根据参数选择使用哪个实现。

什么时候用：当你有一组可互换的算法/行为，且需要在运行时选择时。例如：支付方式（微信/支付宝/银行卡）、导出格式（PDF/Excel/CSV）、通知渠道（短信/邮件/推送）。

2. 简单工厂模式（Simple Factory） GoF 创建型

IAuthStrategy 接口内的静态 login() 方法

// 工厂方法：根据 grantType 拼接 bean 名，从 Spring 容器获取对应策略
static LoginVo login(String body, SysClientVo client, String grantType) {
    String beanName = grantType + BASE_NAME;          // "password" → "passwordAuthStrategy"
    if (!SpringUtils.containsBean(beanName)) {
        throw new ServiceException("授权类型不正确!");
    }
    IAuthStrategy instance = SpringUtils.getBean(beanName);
    return instance.login(body, client);
}

识别特征：一个静态方法，根据参数创建/获取不同的对象，返回统一的接口类型。

RuoYi 的创新：没有单独的 Factory 类，工厂逻辑直接放在接口的静态方法中——省去一个类，保持高内聚。

3. 模板方法模式（Template Method） GoF 行为型

① SysLoginService.checkLogin() + Supplier 回调 ② 5 个策略的 login() 方法遵循相同的步骤序列

变体 A：Supplier 回调式模板方法

public void checkLogin(LoginType loginType, String username, Supplier<Boolean> supplier) {
    // 固定骨架：
    int errorNumber = getErrorCount(username);         // ① 查锁状态
    if (errorNumber >= maxRetryCount) { throw ...; }   // ② 锁定检查
    if (supplier.get()) {                              // ③ 可变步骤（回调）
        errorNumber++;                                  // ④ 错误处理
        if (errorNumber >= maxRetryCount) { throw ...; }
        else { throw ...; }
    }
    clearErrors(username);                              // ⑤ 成功后清理
}

变体 B：约定式模板方法

所有 5 个策略的 login() 方法遵循相同步骤序列，但没有抽象基类强制执行——靠的是代码约定：

// 每个策略的 login() 方法结构：
LoginVo login(String body, SysClientVo client) {
    XxxLoginBody loginBody = JsonUtils.parseObject(body, ...);  // ① 解析
    ValidatorUtils.validate(loginBody);                          // ② 校验
    SysUserVo user = loadUser(...);                              // ③ 加载用户
    loginService.checkLogin(..., () -> ...);                     // ④ 认证
    LoginUser loginUser = loginService.buildLoginUser(user);    // ⑤ 构建
    loginUser.setDeviceType(client.getDeviceType());             // ⑥ 设参数
    SaLoginParameter model = IAuthStrategy.buildLoginParameter();// ⑦ 构建参数
    LoginHelper.login(loginUser, model);                         // ⑧ 签发token
    return buildLoginVo();                                       // ⑨ 返回
}

什么时候用：当多个操作遵循相同的步骤顺序，只是中间某几步的实现不同时。数据库迁移、数据导入校验、审批流程等都是模板方法的典型场景。

4. 观察者模式（Observer Pattern） GoF 行为型

UserActionListener（SaTokenListener）+ UserLoginSuccessListener（@EventListener）+ SysLoginInfoServiceImpl（@Async @EventListener）

// 三层观察者链：

// 第 1 层：Sa-Token 事件 → Spring 事件
UserActionListener implements SaTokenListener {
    doLogin() → publishEvent(UserLoginSuccessEvent)
    doLogout() → 直接清理 Redis
    doKickout() → 直接清理 Redis
    doReplaced() → 直接清理 Redis
}

// 第 2 层：Spring 事件 → 业务处理
UserLoginSuccessListener {
    @EventListener
    handleLoginSuccess(UserLoginSuccessEvent) → Redis 缓存 + 数据库更新
}

// 第 3 层：Spring 事件 → 异步持久化
SysLoginInfoServiceImpl {
    @Async
    @EventListener
    handleLoginInfoEvent(LoginInfoEvent) → INSERT INTO sys_login_info
}

识别特征：一个操作发生（登录成功），多个不相关的处理逻辑被触发（记录在线状态、写登录日志、更新最后登录时间），且这些逻辑之间互不感知。

什么时候用：当你需要在某个事件发生后执行多个副作用操作，且不希望事件源（登录代码）知道有哪些副作用时。典型场景：用户注册后发邮件+发短信+初始化账户+送优惠券。

5. 外观模式 / 门面模式（Facade Pattern） GoF 结构型

LoginHelper 类

// LoginHelper 封装了与 Sa-Token 框架的交互细节
public class LoginHelper {
    public static void login(LoginUser loginUser, SaLoginParameter model) {
        fillRequestContext(...);                     // 填请求上下文
        StpUtil.login(loginId, model.setExtra(...)); // 调 Sa-Token API
        StpUtil.getTokenSession().set(LOGIN_USER_KEY, loginUser); // 存 session
    }

    // 对外暴露简洁的 API
    public static Long getUserId() { ... }
    public static String getUsername() { ... }
    public static boolean isLogin() { return StpUtil.isLogin(); }
}

识别特征：一个工具类统一封装了对某个复杂子系统（Sa-Token）的操作，对外提供简洁的静态方法。如果未来框架换成 Shiro 或 Spring Security，只需修改 LoginHelper。

什么时候用：当你依赖的外部库/子系统 API 复杂且可能变更时，用一个门面类隔离依赖。这是防止第三方库"侵蚀"你业务代码的重要手段。

6. 建造者模式（Builder Pattern）的变体 GoF 创建型

① SaLoginParameter 的流式 setter ② buildLoginParameter + Consumer 扩展

// 变体 A：流式 setter（不是经典 Builder，但效果类似）
model.setTimeout(client.getTimeout())
     .setActiveTimeout(client.getActiveTimeout())
     .setExtra(CLIENT_KEY, client.getClientId())  // setExtra 返回 this
     .setExtra(USER_KEY, loginUser.getUserId());

// 变体 B：buildLoginParameter + Consumer 扩展
SaLoginParameter model = IAuthStrategy.buildLoginParameter(client, p -> {
    p.setExtra("openid", openid);  // 自定义追加
});

7. 不可变值对象模式（Immutable Value Object） 函数式

Java 17 record：UserLoginSuccessEvent 及所有事件类

public record UserLoginSuccessEvent(
    Object loginId, String tokenValue, SaLoginParameter loginParameter
) {}

解决的问题：事件对象（Event）是"已发生的事实"，不应被修改。record 从语言层面保证不可变性，消除"事件在监听器间传递时被污染"的隐患。

9. AOP 切面（Aspect） Spring

LogAspect（操作日志）、@ApiEncrypt（接口加密）、@Lock4j（分布式锁）、@SaIgnore（跳过认证）

这些横切关注点没有侵入认证业务代码，而是通过注解 + AOP 切面织入：

@ApiEncrypt              // ← 请求/响应自动加解密
@PostMapping("/login")
public R<LoginVo> login(@RequestBody String body) { ... }

@Lock4j                  // ← 分布式锁，防止并发绑定
public void socialRegister(AuthUser authUserData) { ... }

四、实际应用决策指南

你遇到的问题推荐使用的模式RuoYi 中的例子
一个接口需要支持多种实现，且未来可能增加策略模式 + 简单工厂5 种 IAuthStrategy
多个实现类的处理步骤一致，只有中间几步不同模板方法（抽象类或 Supplier 回调）checkLogin()
一个操作完成后需要触发多个不相关的后续处理观察者模式（Spring Event）登录后的三层事件链
需要隔离第三方库的 API 变更风险外观模式LoginHelper
需要在方法执行前后加统一逻辑（日志、事务、权限）AOP 切面@ApiEncrypt, @Lock4j
构建复杂对象，有很多可选参数建造者模式SaLoginParameter 的流式 setter
需要传递不可变的数据对象Java recordUserLoginSuccessEvent
某个计算结果可能不需要（条件执行）Supplier 延迟执行checkLogin 中的密码校验

五、本节总结

admin-module 的认证模块展示了 设计模式的工业化应用——每个模式都精确地解决了一个具体问题，模式之间协作流畅，没有过度设计：

  1. 策略模式 解耦了认证方式的选择与实现
  2. 模板方法 统一了认证步骤和重试逻辑
  3. 观察者模式 解耦了登录事件与副作用处理
  4. 外观模式 隔离了 Sa-Token 框架的变更风险
  5. Supplier 延迟执行 避免了不必要的 CPU 密集操作
  6. Java record 保证了事件对象的不可变性
  7. AOP 切面 将横切关注点（加密、锁、日志）从业务代码中剥离

如果你想在自己的项目中参考这种设计，核心原则是：先识别变化点（什么会变？），然后用相应的模式封装变化（怎么隔离变化？）。不要为了用模式而用模式——RuoYi 中的每一个模式都是"不得已而为之"。
