第 5 课 · deviceType 设备类型 多端隔离

🎯 本节目标：彻底理解 deviceType 字段的设计意图——它不是简单标记，而是实现 同一用户 + 同一系统 → 多种设备 → 各自独立会话 的核心机制。学完你能解释为什么 PC 浏览器和手机 APP 登录同一个账号后不会互相踢下线。

在前 4 课中，我们反复看到 loginUser.setDeviceType(client.getDeviceType()) 这一行代码。你可能以为它只是一个"描述性字段"——记录用户用什么设备登录。实际上，它承担了 Sa-Token 多设备登录隔离 的关键角色。

一、为什么需要 deviceType？

想象以下场景：

场景 A：你在公司电脑（Chrome 浏览器）登录了管理后台，正在编辑一篇文章。
场景 B：同时，你在手机上打开同一个系统，审批了一条请假申请。

问题：如果 PC 和手机的登录 session 不隔离，手机登录可能会把 PC 的 session 顶掉（doReplaced），你就丢掉了编辑中的文章。

RuoYi 通过 deviceType 将不同设备的登录会话分开管理：

场景clientIddeviceTypetoken会话
PC 浏览器eurekapctoken-aaa独立，超时 1800s
手机 APPeurekaapptoken-bbb独立，超时 86400s
小程序ruoyi-xcxxcxtoken-ccc独立，超时 -1（永不过期）

同一个 clientId 下，不同 deviceType 各自拥有一份独立的 token 和 session，互不干扰。

三、关键代码位置逐一分析

3.1 数据源头：sys_client 表

deviceType 不是前端传来的，也不是写死的，而是管理员在后台为每个客户端单独配置的：

SysClient.java · ruoyi-modules/ruoyi-system/.../domain/SysClient.java

/** 设备类型 */
private String deviceType;

在客户端管理的后台页面上，管理员可以创建多个客户端配置，比如：

  • PC 管理后台：deviceType = pc
  • 移动端 H5：deviceType = app
  • 微信小程序：deviceType = xcx
  • 第三方开放平台：deviceType = social

3.2 第一传递点：各策略设置 LoginUser

所有 AuthStrategy 实现类 · login() 方法

// 每个策略都做这件事——把客户端配置的设备类型写入登录用户对象
loginUser.setClientKey(client.getClientKey());
loginUser.setDeviceType(client.getDeviceType());  // ← 这里

四、deviceType 如何实现多端隔离？

关键在 Sa-Token 的 SaLoginParameter。当你调用：

StpUtil.login(loginUser.getLoginId(), model);  // model 中已设置 deviceType

Sa-Token 内部使用 loginId + deviceType 作为复合键来管理会话。规则如下：

场景行为触发监听器
同一设备类型再次登录顶替旧 token（旧会话被踢）doReplaced()
不同设备类型登录共存——各自独立 session不触发（不冲突）
管理员踢下线指定 token 被清除doKickout()
主动退出当前 token 被清除doLogout()
⚠️ 关键理解：deviceType 是传给 Sa-Token 框架的参数，框架内部用它来决定是否允许同账号多设备并存。这不是 RuoYi 自己实现的逻辑，而是 Sa-Token 的 SaLoginParameter 原生支持的「同端互斥、多端共存」机制。

五、LoginHelper 类上的注释说明

LoginHelper.java · 第 23-31 行

/**
 * user_type 为 用户类型 同一个用户表 可以有多种用户类型 例如 pc,app
 * deivce 为 设备类型 同一个用户类型 可以有 多种设备类型 例如 web,ios
 * 可以组成 用户类型与设备类型多对多的 权限灵活控制
 *
 * 多用户体系 针对 多种用户类型 但权限控制不一致
 * 可以组成 多用户类型表与多设备类型 分别控制权限
 */

原作者的注释揭示了两层设计：

userType（用户类型） × deviceType（设备类型） │ │ ├── sys_user (系统用户) ├── pc（PC浏览器） └── app_user (APP用户) ├── app（移动端APP） ├── xcx（微信小程序） ├── ios（iOS APP） └── android（Android APP） 组合示例： sys_user:pc → 管理员在电脑上的登录 sys_user:app → 管理员在手机APP上的登录 app_user:android → 普通用户在安卓APP上的登录 app_user:xcx → 普通用户在微信小程序上的登录

每个组合都可以有独立的超时策略（如 APP 超时 30 天，PC 超时 30 分钟），实现精细化的会话管理。

六、实际业务价值

价值说明
不互相踢下线PC 和手机同时登录一个账号，各自独立
差异化超时PC 端敏感度高 30 分钟超时，APP 端 30 天免登录
精确踢人管理员可以只踢掉手机端的会话，保留 PC 端
审计追溯每条操作日志记录设备类型，出问题时知道是哪个端操作的
权限控制可针对不同设备类型授予不同权限（手机端不能执行敏感操作）

七、与你自己的项目

如果你在开发自己的后台系统，考虑是否需要多端登录：

  1. 只有一个 Web 端：不需要 deviceType，直接用 Sa-Token 默认行为即可
  2. Web + APP 双端：引入 deviceType 概念，在登录时从客户端配置读取并传入 SaLoginParameter
  3. 需要更细粒度：可以扩展为 deviceType + deviceId 组合，精确到设备级别
💡 设计启示：deviceType 的设计体现了一个重要原则——把会话策略的决策权交给服务端配置，而不是前端代码。客户端是什么设备、该有多长超时时间、允不允许多端共存，这些都是管理员在后台配置的，前端无法篡改。
