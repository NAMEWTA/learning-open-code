第 3 课 · 模型与 DTO：LoginUser 继承体系 + DTO 独立于 Entity 数据契约

🎯 本节目标（对应使命 Success #6~#8）：学完你能讲清 LoginUser 为什么放 api 模块、XcxLoginUser 怎么扩展它、getLoginId() 的设计意图，以及 DTO 独立于 Entity 的契约价值。还能解释 StartProcessReturnDTO 为什么用 Java record。

上一课我们看了「行为契约」（接口），这一课看「数据契约」——那些被多个模块共享的模型对象和 DTO。它们为什么不在 Entity 旁边放着？为什么有的用 @Data class、有的用 record？

一、LoginUser：为什么它是一个「共享模型」？

LoginUser（151 行）是整个鉴权体系的核心上下文对象。它在 ruoyi-api 里定义，但在至少 4 个模块里流转：

LoginUser 的跨模块流转 ══════════════════════════════════════════════ ruoyi-system ruoyi-admin buildLoginUser() ──────────► AuthStrategy 构建 LoginUser 对象 放入 Sa-Token Session │ │ │ ruoyi-api (LoginUser) │ │ 作为公共数据类型 │ │ │ ▼ ▼ ruoyi-common-satoken ruoyi-common-log 从 Session 读取 LoginUser 记录操作日志时读 LoginUser 做权限校验 获取 userId/username/deptId

如果 LoginUser 放在 ruoyi-system 里，那么 common-satoken 和 common-log 就必须依赖 ruoyi-system——但这两个 common 模块是框架层，不应该反向依赖业务模块。放在 ruoyi-api 里解决了这个方向问题：框架层只需依赖 api，而不需要知道 system 的存在。

🔑getLoginId() 方法的深意
LoginUser.java:141-149
public String getLoginId() {
    if (userType == null) throw new IllegalArgumentException("用户类型不能为空");
    if (userId == null) throw new IllegalArgumentException("用户ID不能为空");
    return userType + ":" + userId;
}
这个方法不是简单的 getter，而是 构造了一个复合标识："sys_user:1"（系统用户） vs "app_user:1"（App 用户）。Sa-Token 用这个字符串作为 loginId，从而在同一个 Redis/Session 里区分不同类型的用户，避免「系统用户 ID=1」和「App 用户 ID=1」被认为同一个人。这就是 LoginUser 必须知道 userType 和 userId 但不直接存 loginId 的原因——loginId 是计算出来的，不是存储的。

二、XcxLoginUser：继承 + 最小扩展

LoginUser（151行） common-core 的 LoginBody │ │ ├── userId, deptId, username, │ │ menuPermission, roles, posts ... │ │ + getLoginId() │ │ │ ▼ ▼ XcxLoginUser（27行） PasswordLoginBody（33行） │ 只多加一个字段：openid │ extends LoginBody │ + 继承所有 LoginUser 的能力 │ + username, password │ │ │ SmsLoginBody（29行） │ extends LoginBody │ + phoneNumber, smsCode │ │ EmailLoginBody（31行） │ extends LoginBody │ + email, emailCode │ │ SocialLoginBody（35行） │ extends LoginBody │ + source, socialCode, socialState │ │ XcxLoginBody（28行） │ extends LoginBody │ + xcxCode │ │ RegisterBody（38行） │ extends LoginBody │ + username, password, userType

设计模式：

1LoginBody 是基类（在 common-core）
含 clientId + grantType——所有登录请求都需要的字段。它在 common-core 因为它是最基础的登录抽象，不依赖任何业务模型。
26 种 XxxLoginBody 在 ruoyi-api
每种登录方式的特有字段（密码登录有 username/password，短信有 phoneNumber/smsCode）。它们在 api 因为需要被 admin 模块的策略类和前端 DTO 两边引用。
3LoginUser → XcxLoginUser 的继承
小程序登录后需要缓存 openid（微信返回的用户标识），后续调用微信 API 时需要它。但 LoginUser 不能加 openid 字段——不是所有登录都需要这个字段（密码登录、短信登录不需要）。继承解决：不动父类，子类只加一个字段。

三、DTO 为什么独立于 Entity？

以 UserDTO vs SysUser 为例：

字段SysUser（Entity，在 ruoyi-system）UserDTO（DTO，在 ruoyi-api）
userId✅✅
deptId✅✅
userName✅✅
nickName✅✅
userType✅✅
email✅✅
phoneNumber✅✅
gender✅✅
status✅✅
createTime✅✅
password✅❌
delFlag✅（逻辑删除）❌
loginIp / loginDate✅❌
remark✅❌
@TableName / @TableId✅（MyBatis-Plus 注解）❌（纯 POJO）

DTO 是 选择性暴露 的——不传 password（安全）、不传 delFlag（无意义）、不传 ORM 注解（消费者不需要 MyBatis-Plus）。这是 接口隔离原则在数据层的应用：消费者只看到它需要的数据，实现方可以自由修改 Entity 结构，只要 DTO 契约不变。

💡 反过来说：如果外部模块直接引用 SysUser Entity，那么修改 sys_user 表结构时，所有引用方都可能编译失败。DTO 的独立存在把这种「数据库变更的影响范围」锁在了 system 模块内部。

四、Java record：新契约的现代选择

ruoyi-api 里有两个文件用了 Java 14 的 record：

// StartProcessReturnDTO.java — 14 行
public record StartProcessReturnDTO(
    Long processInstanceId,
    Long taskId
) {}

// FlowCopyDTO.java — 14 行
public record FlowCopyDTO(
    Long userId,
    String nickName
) {}

为什么这两个用 record 而不是 @Data class？

  1. 语义匹配：它们是不可变的返回值/值对象——启动流程后返回的 ID 不应该被修改，抄送人信息创建后不应该被篡改。record 天生不可变，@Data 天生可变。
  2. 简洁：14 行 vs ~40 行（如果用 @Data + Serializable + serialVersionUID）。对于明确的数据载体，减少代码就是减少出错面。
  3. 一致性信号：项目在新增的 DTO 上使用 record，在遗留的 DTO 上保留 @Data——这是渐进式的现代化。读者一眼就知道：record = 不可变返回值，@Data = 可能需要序列化/设置在 Session 里的可变对象。

注意 StartProcessReturnDTO 和 FlowCopyDTO 都 没有 implements Serializable——record 自动可序列化（当所有字段都可序列化时）。这是 record 相比 @Data class 的另一个优势：不用手写 serialVersionUID。

五、检验一下

Q1. LoginUser 为什么放在 ruoyi-api 而不是 ruoyi-system？

A. 因为 system 模块还没有创建 B. 因为 common-satoken、common-log 等框架模块需要它，但它们不应该反向依赖 ruoyi-system C. 因为 api 包名更短 D. 因为 Maven 中心仓库要求公共模型必须在 api 模块
看答案B。LoginUser 被 common-satoken（从 Session 读）、common-log（记录日志）等框架模块消费。如果放在 system 里，这些 common 模块就得依赖 system——违反了「框架层不应依赖业务层」的原则。放在 api 里，common 模块只需依赖 api。

Q2. XcxLoginUser 继承 LoginUser 后只加了一个字段 openid。为什么不直接把 openid 加到 LoginUser 里？

A. LoginUser 字段数量有限制 B. openid 只在微信小程序登录时才有意义，密码登录/短信登录不需要它。加在父类会让所有登录类型都携带一个无意义的 null 字段 C. openid 字段类型不兼容
看答案B。这是单一职责 + 开放封闭原则。不是所有登录都产生 openid，让父类干净、子类按需扩展，是最小惊讶的设计。

Q3. 为什么 StartProcessReturnDTO 用 record 而 UserDTO 用 @Data class？

A. record 是新语法，作者想试试 B. StartProcessReturnDTO 是不可变的返回值（启动流程后 ID 不应被改），UserDTO 是可能被框架序列化/反序列化、需要 setter 的可变对象 C. UserDTO 字段太多，record 不支持超过 10 个字段
看答案B。record 的字段是 final 的，适合不可变的返回值。UserDTO 通过 @Data 提供 getter/setter，因为它在序列化/反序列化时需要 setter，在 Thymeleaf/模板里需要 getter。两者的语义选择都是正确的。
📖 推荐一手资料：打开 LoginUser.java 第 141 行，看 getLoginId() 方法——它不是简单的字段 getter，而是用 userType + ":" + userId 拼出一个复合登录标识。再去 Sa-Token 登录认证文档 看 loginId 的概念，你会理解这个设计为什么是必须的。
