第 4 课 · 工作流事件解耦：流程内核与业务逻辑的完全分离 事件契约

🎯 本节目标（对应使命 Success #7）：学完你能讲清 ProcessEvent / ProcessTaskEvent / ProcessDeleteEvent 三个事件的发布与消费全链路，理解为什么事件类要放在 api 模块以及 Spring @EventListener 如何实现「业务代码不引入流程引擎依赖」。

前三课我们看了查询契约（Service 接口）和数据契约（DTO/模型）。这一课看 ruoyi-api 里独一档的设计——行为契约：事件对象。它是 api 模块中唯一不涉及数据库查询的部分，却可能是解耦做得最彻底的部分。

一、问题场景：请假审批怎么更新业务表？

假设你有一个请假业务（TestLeaveServiceImpl）：用户提交请假申请 → 领导审批通过 → 请假记录的状态要变成「已通过」。

如果直接在流程引擎的回调里写 leaveMapper.updateById()，那流程引擎就必须知道请假业务的存在——这不合理，流程引擎是通用组件。反过来，如果在请假业务里轮询流程状态，那是低效的轮询模式。

解决方案：事件驱动。流程引擎只负责「发起事件 = 发生了什么」，业务模块只负责「监听事件 = 我关心什么」。中间通过 ruoyi-api 里的事件类作为数据契约。

三、完整链路：从引擎回调到业务更新

事件链路全景（以「领导审批通过请假」为例） ① 流程引擎回调 WorkflowGlobalListener（通用监听器，在 workflow 模块） │ 检测到流程节点变化 │ ▼ ② 事件组装与发布 FlowProcessEventHandler.processHandler(...)（在 workflow 模块） │ ProcessEvent event = new ProcessEvent(); │ event.setFlowCode("leave1"); │ event.setBusinessId("10"); ← 请假记录 ID │ event.setStatus("completed"); │ event.setParams(params); ← 携带审批意见、附件等 │ SpringUtils.context().publishEvent(event); │ ▼ Spring ApplicationEvent 机制 │ ▼ ③ 业务模块监听（完全解耦！） TestLeaveServiceImpl.processHandler(ProcessEvent event)（在 workflow 模块示例里） │ @EventListener(condition = "#processEvent.flowCode.startsWith('leave')") │ public void processHandler(ProcessEvent processEvent) { │ TestLeave leave = leaveMapper.selectById(processEvent.getBusinessId()); │ leave.setStatus(processEvent.getStatus()); │ // 可取出审批意见、附件等扩展数据 │ Map<String, Object> params = processEvent.getParams(); │ leaveMapper.updateById(leave); │ } │ ▼ ④ 业务数据更新完成 请假的 status 从 "waiting" → "completed" 流程引擎对此一无所知——它只发了事件。
🔑关键设计：事件类在 api 模块的好处
TestLeaveServiceImpl 的 @EventListener 方法签名用的是 ProcessEvent（来自 ruoyi-api），不是流程引擎的任何内部类（如 WarmFlow 的 Instance、Task）。这意味着：

✅ 业务模块不依赖流程引擎的具体实现（WarmFlow、Flowable、Camunda...任意替换）
✅ 事件字段是业务语义的（flowCode、businessId、status），不是引擎语义的
✅ 单元测试时可以轻松 mock ProcessEvent，不需要启动完整流程引擎

四、@EventListener 的 condition：精准订阅

Spring 的 @EventListener + SpEL condition 让业务模块只订阅自己关心的流程：

// 只监听「以 leave 开头的流程编码」的事件
@EventListener(condition = "#processEvent.flowCode.startsWith('leave')")
public void processHandler(ProcessEvent processEvent) { ... }

// 也可以精确到单个流程
@EventListener(condition = "#processEvent.flowCode == 'leave1'")
public void processHandler(ProcessEvent processEvent) { ... }

// 任务事件同理
@EventListener(condition = "#processTaskEvent.flowCode.startsWith('leave')")
public void processTaskHandler(ProcessTaskEvent processTaskEvent) { ... }

// 删除事件
@EventListener(condition = "#processDeleteEvent.flowCode.startsWith('leave')")
public void processDeleteHandler(ProcessDeleteEvent processDeleteEvent) { ... }

这个 condition 是 SpEL 表达式，Spring 在运行时会先解析条件再决定是否调用监听方法。如果整个系统有 100 个流程定义但请假业务只关心 leave*，那么其他 90+ 个流程的事件不会触发请假业务的监听器——性能无损。

💡 事件 vs 接口的区别：Service 接口（第 2 课）是 同步请求-响应——调用方主动要数据，响应必须即时返回。事件是 异步通知——发布方不关心「谁在听」，监听方按自己的节奏处理。两者的共性是：契约都在 ruoyi-api 里，实现方和消费方互相不知道对方存在。

五、WorkflowService 与事件的协作

除了事件，workflow.api 还有一个 WorkflowService 接口（99 行，10 个方法），这是整个 ruoyi-api 唯一的 workflow 服务契约。它提供：

  • 启动流程：startWorkFlow(StartProcessDTO) → StartProcessReturnDTO
  • 办理任务：completeTask(CompleteTaskDTO) → boolean
  • 查询状态：getBusinessStatus(businessId) → String
  • 流程变量：setVariable / instanceVariable / getInstanceIdByBusinessId
  • 删除实例：deleteInstance(businessIds) → boolean

这个接口由 WorkflowServiceImpl 实现（带 @ConditionalOnEnable 条件装配，如果没启用工作流则整个 Bean 不加载）。它的存在补全了 workflow 域的契约：主动操作走 Service 接口，被动响应走 Event 监听。

六、检验一下

Q1. 流程事件（ProcessEvent）为什么放在 ruoyi-api 而不是 workflow 模块里？

A. api 模块编译更快 B. 让业务模块（如请假）可以监听事件，而不需要依赖 workflow 引擎模块 C. Event 类必须放在 api 包，这是 Java 规范 D. 放在 workflow 里会导致循环依赖
看答案B。如果 ProcessEvent 在 workflow 模块里，业务模块（TestLeaveServiceImpl）要监听它就必须依赖 workflow 模块——这样业务代码就耦合了流程引擎。放在 api 里，业务模块只需依赖 api 就能订阅事件。

Q2. @EventListener 的 condition 表达式起了什么作用？

A. 装饰作用，实际不过滤 B. 让业务模块只处理自己关心的流程事件（如 flowCode.startsWith('leave')），避免无关事件触发业务逻辑 C. 加速 Spring 启动 D. 替代 if-else 判断
看答案B。condition 是 SpEL 表达式，运行时 Spring 先求值再决定是否调用方法。如果系统有 100 个流程，condition 确保只有匹配的流程事件才会进入业务处理方法。

Q3. 业务模块要使用工作流，需要引入哪些依赖？

A. ruoyi-workflow + WarmFlow 引擎 B. 只需要 ruoyi-api（包含 WorkflowService 接口和事件类），不需要引入工作流引擎模块 C. Spring Cloud Stream D. Apache Kafka
看答案B。业务模块通过 WorkflowService 接口调 API、通过 @EventListener 监听事件，二者都在 ruoyi-api 中。运行时 Spring 注入真实的实现类，编译时业务模块只看到 api。
📖 推荐一手资料：打开 FlowProcessEventHandler.java 的三个 publish 方法（processHandler / processTaskHandler / processDeleteHandler），看事件如何从 WarmFlow 的 Instance/Task 对象转换成 ruoyi-api 里的通用事件对象。然后打开 TestLeaveServiceImpl.java 的三个 @EventListener 方法，看业务代码如何完全不依赖 WarmFlow 却能响应流程变化。这对文件是最佳的解耦范本。
