第 1 课 · ruoyi-api 模块定位：依赖倒置的契约层 全景地图

🎯 本节目标（对应使命 Success #1~#2）：学完你能一句话讲清 ruoyi-api 的角色，能画出它的包结构树，能解释「为什么它存在」而不是每个模块直接互相调用。

在 RuoYi-Vue-Plus 的模块树里，ruoyi-api 只有 1890 行代码，是除 pom.xml 外最小的模块。但它被 11 个其他模块 依赖——没有它，整个系统会陷入循环依赖地狱。这一课我们先建立全景认知，把它的「骨架」看清。

一、先看它长什么样

关键数字：34 个文件，9 个接口，20 个模型/DTO/事件类，0 行业务逻辑代码。这本身就是信号——它是纯契约，不含实现。

二、核心命题：为什么需要这个模块？

假设没有 ruoyi-api。现在 ruoyi-workflow 模块需要查用户名，它有两个选择：

A直接注入 SysUserMapper：ruoyi-workflow 依赖 ruoyi-system。但 ruoyi-system 的某些功能（比如消息推送）又需要依赖 ruoyi-workflow——循环依赖，Maven 直接拒绝编译。
B通过 ruoyi-api 间接调用：ruoyi-workflow 只依赖 ruoyi-api（接口），不依赖 ruoyi-system（实现）。运行时 Spring 把 SysUserServiceImpl 注入给所有声明了 UserService 的消费者。编译时无循环，运行时动态绑定。

这就是 依赖倒置原则（Dependency Inversion Principle） 的实战应用：

高层模块（workflow / common-translation / admin）不应依赖低层模块（system），两者都应依赖抽象（ruoyi-api 里的接口）。

三、依赖关系全景：谁依赖谁？

模块依赖 ruoyi-api？角色
ruoyi-common-core❌ 不依赖基础层——api 依赖它，但它不知道 api 存在
ruoyi-api—契约层——只依赖 common-core
ruoyi-system✅ 依赖实现方——提供接口实现
ruoyi-workflow✅ 依赖消费方 + 实现方——既消费 system.api 也实现 workflow.api
common-translation✅ 依赖框架层消费方——翻译功能需要查用户名/部门名
common-satoken✅ 依赖框架层消费方——Sa-Token 集成需要 LoginUser
common-log✅ 依赖日志记录需要查用户信息
common-push✅ 依赖消息推送需要查用户/部门
common-mybatis✅ 依赖数据权限需要查角色/部门
ruoyi-admin✅ 依赖Web 层——消费所有 api 接口
ruoyi-demo✅ 依赖示例模块
ruoyi-ai✅ 依赖AI 模块
💡 关键洞察：common-core 不依赖 api，但 common-translation 依赖 api。
这反映了 RuoYi 的分层策略：common-core 是最底层工具（StringUtils、SpringUtils、基础注解），不能依赖任何业务抽象；common-translation 是偏业务的框架层（需要知道「用户名」「部门名」这些业务概念），因此可以依赖 api。

四、模块内部的包设计：为什么是两个顶层包？

注意 ruoyi-api 的 Java 包是 两个独立顶层包：org.dromara.system.api 和 org.dromara.workflow.api，而不是 org.dromara.api.system 和 org.dromara.api.workflow。

这个设计的意思是：每个业务域的 API 是自治的，包的命名空间属于业务模块而不是 api 模块。如果未来把 workflow 拆成独立微服务，只需要把 org.dromara.workflow.api 整个包拿走，不跟 system 的包产生命名冲突。

1system.api（8 接口 + 模型 + DTO）
org.dromara.system.api
面向「系统管理」域的查询契约：用户、部门、角色、岗位、配置、消息、OSS、任务办理人。这是被消费最多的域——几乎每个模块都要查用户/部门名称。
2workflow.api（1 接口 + DTO + 事件）
org.dromara.workflow.api
面向「工作流引擎」域的操作契约 + 事件契约。特殊之处在于它不仅有服务接口，还有 3 个 Spring 事件类——这是 ruoyi-api 里唯一的「行为契约」而非「数据查询契约」。

五、检验一下（提取练习）

Q1. ruoyi-api 只依赖哪一个模块？

A. ruoyi-system B. ruoyi-common-core C. ruoyi-workflow D. Spring Boot Starter
看答案B。打开 ruoyi-api/pom.xml，只有一个 dependency：ruoyi-common-core。它是整个模块唯一的编译时依赖。

Q2. 为什么 ruoyi-common-core 不依赖 ruoyi-api，而 ruoyi-common-translation 依赖它？

A. core 模块代码更老，没来得及加依赖 B. core 是最底层工具层，不应知道业务概念；translation 需要查用户名/部门名，属于偏业务框架层 C. core 通过反射调用 api，不需要编译时依赖 D. translation 的作者和 core 的作者不同
看答案B。common-core 提供的是 StringUtils、SpringUtils、基础注解等纯工具，不知道「用户」「部门」这些业务概念的存在。common-translation 的 UserNameTranslationImpl 需要调用 UserService.selectUserNameById()，所以必须依赖 api。

Q3. ruoyi-api 里的 Java 包为什么是 org.dromara.system.api 而不是 org.dromara.api.system？

A. 作者的个人偏好 B. 每个业务域的 API 包属于该业务域自己的命名空间，方便未来拆分成独立服务 C. Maven 的强制要求 D. 为了绕过 Java 模块系统的限制
看答案B。包的命名空间属于业务模块（system / workflow），不属于 api 模块。如果未来把 system 拆成独立微服务，org.dromara.system.api 整个包可以直接独立为一个 jar，不会跟 workflow 的包冲突。
📖 推荐一手资料：打开 ruoyi-api/pom.xml，亲眼看看它唯一的 dependency；再打开 ruoyi-common/ruoyi-common-core/pom.xml 确认它不依赖 ruoyi-api。这个对比比任何文章都有说服力。
