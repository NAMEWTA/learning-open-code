第 2 课 · Service 接口详解：8 个 SPI 的定义、实现与消费 契约层核心

🎯 本节目标（对应使命 Success #3~#5）：学完你能讲清每个 Service 接口「谁定义、谁实现、谁消费」，特别能解释 UserService 的三方协作链路和 ConfigService 的 default 方法模式。

上一课我们看了全景。现在聚焦 system.api 包的 8 个服务接口——它们是 ruoyi-api 被消费最多的部分，也是依赖倒置最生动的体现。

二、明星案例：UserService 的三方协作

这是整个 ruoyi-api 被消费最广的接口（112 行，10 个方法，被至少 7 个模块消费），让我们以此为样本，理解完整的「定义→实现→消费」链路。

❶定义方：ruoyi-api
ruoyi-api/.../system/api/UserService.java
一个纯 Java 接口，10 个方法签名。不依赖任何 ORM、不依赖任何数据库。最复杂的方法像这样：
List<UserDTO> selectUsersByDeptIds(Collection<Long> deptIds);
注意返回的是 UserDTO（api 自己定义的），不是 SysUser（system 模块的 Entity）。
❷实现方：ruoyi-system
ruoyi-system/.../service/impl/SysUserServiceImpl.java:48
public class SysUserServiceImpl implements ISysUserService, UserService {
每个 api 方法在这里有一个真实实现，通常就是调用自己的 Mapper 然后转换 Entity → DTO：
selectUserNameById(Long userId) → baseMapper.selectById(userId).getUserName()
selectById(Long userId) → 查出来的 SysUser 转成 UserDTO 返回
❸消费方 A：ruoyi-workflow（业务模块）
ruoyi-workflow/.../impl/FlwTaskAssigneeServiceImpl.java:47
private final UserService userService;
工作流模块需要查用户名、按角色查用户，但它不知道 SysUserServiceImpl 的存在。Spring 在运行时把 SysUserServiceImpl 这个 Bean 注入给 UserService 类型的字段。
❹消费方 B：common-translation（框架层）
common-translation/.../impl/UserNameTranslationImpl.java:25
private final UserService userService;
翻译框架需要把「用户 ID」翻译成「用户名」显示在页面上。它通过 @TranslationType 注解注册翻译类型，底层调 userService.selectUserNameById()。这是一个框架层依赖业务抽象的典型例子——翻译功能是通用的，但「用户名怎么查」是业务特定的。
❺消费方 C：ruoyi-admin（Web 层）
ruoyi-admin 各种 Controller 和 Service
登录策略（PasswordAuthStrategy 等）通过 UserService 查询用户信息来构建 LoginUser 对象。
💡 一句话记住：UserService 在 api 里是一个「只读用户查询契约」——任何需要知道「用户叫什么」的模块都注入它，但谁都不能通过它修改用户数据。增删改走 ISysUserService（只在 system 模块内部可见）。

三、ConfigService：接口 default 方法的妙用

ConfigService 是一个巧妙的设计案例。它只有 2 个抽象方法 + 6 个 default 方法：

// 抽象方法——实现类必须提供
String getConfigValue(String configKey);      // 从数据库读原始字符串
Dict getConfigMap(String configKey);           // 读 Map 类型配置

// default 方法——接口直接提供，复用 getConfigValue()
default Boolean getConfigBool(String configKey) {
    return Convert.toBool(getConfigValue(configKey));   // "true" → true
}
default Integer getConfigInt(String configKey) {
    return Convert.toInt(getConfigValue(configKey));    // "42" → 42
}
// ... 还有 getConfigLong, getConfigDecimal, getConfigObject, getConfigArray

设计动机：所有配置在数据库里存的是字符串，但消费者要的是 Boolean、Integer、BigDecimal、甚至反序列化后的对象。如果让每个消费者自己做类型转换，会遍地 Convert.toInt(configService.getConfigValue("xxx"))。把转换逻辑收进接口的 default 方法里，消费者只需 configService.getConfigInt("xxx")——一次定义，处处复用，零重复代码。

这也解释了为什么这些 default 方法依赖 cn.hutool.core.convert.Convert——它是 ruoyi-common-core 传递过来的依赖（hutool），不需要额外引入。

🤔 想一想：为什么选择 default 方法而不是抽象类？ 因为 Java 单继承。实现类 SysConfigServiceImpl 已经继承了 ServiceImpl<SysConfigMapper, SysConfig>（MyBatis-Plus 的通用 Service 基类），不能再继承一个抽象类。接口的 default 方法不占用继承名额，这是 Java 8+ 带给 SPI 设计的最大便利。

四、其余接口的设计模式

🔹RoleService / PostService：极端精简
各只有 1 个方法：Map<Long, String> selectXxxNamesByIds(Collection<Long> ids)。这是「只暴露被需要的」——外部模块只需要把 ID 翻译成名称，不需要知道角色的 dataScope、岗位的 postCategory。接口越窄，实现越自由，耦合越低。
🔹MessageService：重载方法族
7 个方法，覆盖 send（单用户/广播）× message（文本/自定义体）+ publish（批量）。注意它与 PushPayloadDTO 的协作——消息体是一个完整 DTO，不是裸字符串。这使得前端能拿到 messageId、type、source、path 等结构化信息来做跳转和展示。
🔹OssService：ID 串 ↔ 文件信息
selectUrlByIds(String ossIds) 接的是逗号分隔的 ID 串，返回逗号分隔的 URL 串。这是为了与数据库存储格式对齐（一个字段存多个文件 ID），也是「契约匹配消费者使用方式」的例子。
🔹TaskAssigneeService：工作流设计器对接
ruoyi-api/.../system/api/TaskAssigneeService.java
4 个方法都是 selectXxxByTaskAssigneeList(TaskAssigneeBody)，返回 TaskAssigneeDTO（含分页总数 + 办理人列表）。它是唯一专门为工作流模块设计的 system.api 接口——工作流设计器需要在画流程图时选「谁来审批」，这个接口提供候选人列表。

五、检验一下

Q1. SysUserServiceImpl 为什么同时 implements ISysUserService 和 UserService？

A. Java 语法限制，多 implements 比单 extends 快 B. 接口隔离：ISysUserService 供内部增删改查，UserService 只供外部查询 C. 作者不小心写多了 D. MyBatis-Plus 要求必须实现两个接口
看答案B。ISysUserService 继承 MyBatis-Plus 的 IService<SysUser>，有完整 CRUD。UserService 只有纯查询方法。外部模块注入 UserService 就只能查不能改——通过接口类型约束权限。

Q2. ConfigService 为什么用 default 方法做类型转换而不是抽象类？

A. default 方法性能更好 B. 实现类已经继承了 MyBatis-Plus 的 ServiceImpl，Java 单继承不允许再加抽象类 C. default 方法可以被消费者覆盖 D. 抽象类不能有方法体
看答案B。SysConfigServiceImpl extends ServiceImpl<...>，Java 不支持多继承。用接口 + default 方法既提供了通用转换逻辑，又不占用继承名额。

Q3. RoleService 和 PostService 为什么各自只有 1 个方法？

A. 功能还没做完 B. 接口最小化原则：外部模块只需要把 ID 翻译成名称，不需要多余的查询能力 C. 这两个表数据太少，不需要更多查询
看答案B。外部模块（如 workflow 的办理人选择、common-translation 的名称翻译）只需要「给你一组角色 ID，你给我对应的角色名 Map」。其他字段（dataScope、roleKey）是 system 内部逻辑用的，不暴露给外部。接口越窄，未来变更实现时影响面越小。
📖 推荐一手资料：打开 SysUserServiceImpl.java 第 48 行，看 implements ISysUserService, UserService 声明；然后打开 FlwTaskAssigneeServiceImpl.java 第 47 行，看 private final UserService userService 注入。这两个文件之间的「接口类型约束」就是依赖倒置的活样本。
