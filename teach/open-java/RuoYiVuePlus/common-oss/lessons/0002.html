第 2 课 · 客户端抽象：从接口到工厂的完整继承链 核心类 ①

🎯 本节目标（对应使命 Success #2）：学完你能画出 OssClient → AbstractOssClientImpl → DefaultOssClientImpl 继承链，说清 OssFactory 的双重检查锁 + 配置热更新机制。

OSS 模块的核心是一套精心设计的三层继承体系，加上一个管理客户端生命周期的工厂。看似类很多，但只要抓住「接口定义契约 → 抽象类收敛共性 → 实现类只覆写初始化 → 工厂缓存所有实例」这条线，就能一通百通。

一、OssClient 接口 —— 契约的设计哲学

为什么要用接口？

亚马逊 S3、阿里云 OSS、腾讯云 COS、华为云 OBS —— 每个厂商都有自己的一套 SDK，但暴露给上层业务的操作高度一致：上传、下载、删除、生成预签名 URL。因此，RuoYi 抽象出一个 OssClient 接口，所有厂商的实现类都必须满足同一套 API 契约：

OssClient (接口) └── AbstractOssClientImpl (抽象类，70+ 方法，统一实现) └── DefaultOssClientImpl (唯一覆写 doInitialize()) └── [未来扩展] AliyunOssClientImpl、TencentOssClientImpl …

两层 API 设计：bucketXxx() vs xxx()

接口把每个操作分成两组：

  • bucketXxx(bucket, key, ...) —— 指定 bucket，适用于多 bucket 场景
  • xxx(key, ...) —— 使用默认 bucket（从配置读取），适用于 90% 的单 bucket 业务
OssClient 接口方法分组 ┌─ bucketUpload() ── 6 种输入类型 × 2（带/不带 Options）= 12 个重载 ├─ upload() ── 同上 12 个重载，调用 defaultBucket() ├─ bucketDownload()── 6 种输出目标（File, OutputStream 等） ├─ download() ── 同上，调用 defaultBucket() ├─ bucketDelete() / delete() ├─ bucketPresignGetUrl() / presignGetUrl() └─ bucketPresignPutUrl() / presignPutUrl()

上传的 6 种输入类型

这是接口最精彩的设计——不管你手里是什么形态的数据，总有一个重载接住你：

输入类型使用场景
PathNIO 路径，本地文件已落盘
File传统 IO，Java 最常用的文件引用
RandomAccessFile大文件断点续传场景
ReadableByteChannelNIO Channel，零拷贝场景
InputStream内存流、网络流，最通用但需指定长度
byte[]小数据（缩略图、JSON），已在内存中

clientId() 默认方法

接口声明了一个 default 方法 clientId()，用 IdUtil.fastSimpleUUID() 生成一个不带横线的 UUID，作为客户端实例的唯一标识：

// OssClient.java 接口中的默认方法
default String clientId() {
    return IdUtil.fastSimpleUUID();
}

这意味着任何 OssClient 实例天生带有一个全局唯一 ID，工厂用它做缓存的 key。抽象实现类会把这个值塞进构造函数并覆盖掉默认行为——后面会看到。

二、AbstractOssClientImpl —— 模板方法模式的精髓

这个抽象类是整个模块的绝对核心。70+ 个公开方法，覆盖了接口中声明的所有操作，但最终全部收敛到两个私有方法。这就是模板方法模式的教科书级应用。

四个核心字段

// AbstractOssClientImpl.java 核心状态
protected S3AsyncClient s3AsyncClient;          // AWS SDK 异步客户端
protected S3TransferManager s3TransferManager;  // 传输管理器（上传/下载）
protected S3Presigner s3Presigner;              // 预签名 URL 生成器
protected ExecutorService asyncExecutor;        // 异步线程池

private final AtomicBoolean initialized = new AtomicBoolean(false);

这 4 个字段构成了客户端的全部运行时依赖。注意 AtomicBoolean——它保证初始化的线程安全性，且只允许一次初始化（直到 close() 重置它）。

所有方法收敛到两个私有方法

12 个 bucketUpload(Path/File/RandomAccessFile/ReadableByteChannel/InputStream/byte[]) │ ├─ 每种输入类型先转换为 AsyncRequestBody │ ├─ Path → AsyncRequestBody.fromFile(path) │ ├─ File → RandomAccessFile → ReadableByteChannel → AsyncRequestBody │ ├─ InputStream → 写临时文件 → 上传 → finally 删除临时文件 ← ⚠️ 关键技巧 │ └─ byte[] → AsyncRequestBody.fromBytes(data) │ └─ 全部流入 ──► private bucketUpload(bucket, key, AsyncRequestBody body, Options) │ └─► doCustomUpload() ──► s3TransferManager.upload()
1InputStream 上传的"先落盘再上传"技巧
AbstractOssClientImpl.java · bucketUpload(InputStream)
S3 传输管理器需要知道文件的 contentLength 才能做分片上传。但 InputStream 本身不提供长度信息（除非调用方传入）。解决方案：先把 InputStream 写到 临时文件，然后从临时文件构建 AsyncRequestBody，上传完成后在 finally 块中删除临时文件。这是用磁盘空间换确定性的经典权衡。
2File → AsyncRequestBody 的转换链
File 不是直接转成 body 的，而是走了一组精细的转换：File → new RandomAccessFile(file, "r") → randomAccessFile.getChannel() → AsyncRequestBody.fromReadableByteChannel(channel, size)。每一步都有目的：RandomAccessFile 支持随机定位（大文件分片需要），ReadableByteChannel 是 NIO 的零拷贝通道。

路径构建：buildPathKey

// buildPathKey(businessPrefix, fileName) 的核心逻辑
prefix = mergePrefix(defaultPrefix, businessPrefix);   // 合并前缀
datePath = yyyy/MM/dd;                                  // 按日期分目录
uuid = IdUtil.fastSimpleUUID();                         // 全局唯一文件名
suffix = FileUtil.getSuffix(fileName);                  // 保留原始扩展名
// 结果: prefix/2026/07/05/a1b2c3d4...jpg

mergePrefix() 和 normalizePrefix() 保证路径永远是 URL 安全的：去空格、去首尾斜杠、空值兜底。这两个方法很短（各约 10 行），但因为每个上传都调用，所以出错影响面极大。

异常统一转换：toStorageException

S3 SDK 的异步操作都包在 CompletableFuture 里，抛出的异常外面会套 CompletionException 或 ExecutionException。业务代码不想关心这些包装层。所以抽象类提供了一个 toStorageException()：

// 解包 CompletionException / ExecutionException → 统一 S3StorageException
private RuntimeException toStorageException(Throwable e) {
    Throwable cause = unwrapAsyncException(e);   // 剥掉异步包装
    return new S3StorageException(cause);         // 换成统一异常
}

这样上层调用方只需要捕获 S3StorageException，不用管底层是 CompletionException 还是 ExecutionException。

close()：有序关闭四件套 + 重置状态位

// close() 方法（伪代码）
public void close() {
    closeQuietly(s3TransferManager);   // 1. 先停传输（防止新任务提交）
    closeQuietly(s3AsyncClient);       // 2. 再停客户端（释放连接池）
    closeQuietly(s3Presigner);         // 3. 停预签名器
    shutdownAndAwait(asyncExecutor);   // 4. 最后停线程池（等任务跑完）
    initialized.compareAndSet(true, false);  // 重置初始化标记
}

关闭顺序不能乱：先断传输、再断连接、然后停辅件、最后收线程。AtomicBoolean 被重置后，同一个实例可以再次初始化——这是 OssFactory 热更新机制的基础。

三、DefaultOssClientImpl —— 唯一需要覆写的 doInitialize()

抽象类把 70+ 个方法都实现完了，子类只有一个任务：告诉框架「怎么连上 S3」。这个唯一需要覆写的方法是 doInitialize()。

1校验凭证：检查 accessKey 和 secretKey 是否为空。缺任何一个直接抛异常——没有凭证，后面的连接构建毫无意义。
2解析端点与域名：endpointUrl 用于 API 调用（如 https://s3.amazonaws.com），domainUrl 用于访问文件（如 https://my-bucket.s3.amazonaws.com）。两者可能不同——endpoint 是 API 网关，domain 是 CDN 或自定义域名。同时解析 region，默认 Region.US_EAST_1。
3构建 S3AsyncClient：这是整个初始化的核心，配置密集：
// DefaultOssClientImpl.doInitialize() —— S3AsyncClient 构建
S3AsyncClient.builder()
    .credentialsProvider(
        StaticCredentialsProvider.create(
            AwsBasicCredentials.create(accessKey, secretKey)))
    .endpointOverride(URI.create(endpointUrl))
    .region(region)
    .forcePathStyle(usePathStyleAccess)         // 路径风格 vs 虚拟主机风格
    .requestChecksumCalculation(
        ChecksumValidatingPublisher.WHEN_REQUIRED)    // 请求校验和
    .responseChecksumValidation(
        ChecksumValidatingPublisher.WHEN_REQUIRED)    // 响应校验和
    .httpClient(
        NettyNioAsyncHttpClient.builder()
            .connectionTimeout(Duration.ofSeconds(60))            // 连接超时 60s
            .connectionAcquisitionTimeout(Duration.ofSeconds(30)) // 获取连接超时 30s
            .maxConcurrency(100)                                   // 最大并发 100
            .maxPendingConnectionAcquires(1000)                    // 等待队列 1000
            .build())
    .build();
⚠️ 校验和策略：ChecksumValidatingPublisher.WHEN_REQUIRED 表示「仅当服务端要求时才计算和验证校验和」。这比 WHEN_SUPPORTED 更省 CPU，大多数对象存储服务（S3、阿里云 OSS）都支持服务端自动校验，不需要客户端重复计算。
4构建 S3TransferManager 和 S3Presigner：TransferManager 基于 S3AsyncClient 构建，负责分片上传/下载的编排；Presigner 用 domainUrl（不是 endpointUrl）作为端点，生成的预签名 URL 才是对外可访问的地址。
5创建 asyncExecutor：虚拟线程 vs 线程池
// 两种异步执行器策略
if (asyncExecutorConfig.enabledVirtualThread()) {
    this.asyncExecutor = Executors.newVirtualThreadPerTaskExecutor();
} else {
    this.asyncExecutor = Executors.newScheduledThreadPool(
        asyncExecutorConfig.corePoolSize());
}

Java 21+ 的虚拟线程是首选：每个异步任务一个虚拟线程，不消耗 OS 线程，可以同时跑成千上万个。但对于 Java 17 或不想用虚拟线程的场景，退回到传统的 ScheduledThreadPool。这种设计让同一个实现类适配不同 JDK 版本。

四、OssFactory —— 门面、缓存、热更新三合一

前面三层继承链解决了「怎么发请求」，但业务代码不需要（也不应该）知道这些细节。OssFactory 提供了一个极简的入口：OssFactory.instance()。但它做的事远不止「new 一个对象」。

完整调用流程

OssFactory.instance() │ ├─ 1. RedisUtils.getCacheObject(OssConstant.DEFAULT_CONFIG_KEY) │ └─ 从 Redis 读出当前生效的配置 key（如 "aliyun-oss"） │ └─ 2. instance("aliyun-oss") │ ├─ 3. CacheUtils.get("sys_oss_config", "aliyun-oss") │ └─ 从缓存（Redis/Caffeine）读配置的 JSON 字符串 │ ├─ 4. JsonUtils.parseObject(json, OssProperties.class) │ └─ JSON → OssProperties POJO │ ├─ 5. OssClientConfig.formProperties(properties) │ └─ OssProperties → OssClientConfig（字段映射 + 策略解析） │ ├─ 6. getClientLock(configKey) // 每个 configKey 一把 ReentrantLock │ ├─ 7. lock.lock() │ │ │ ├─ client = CLIENT_CACHE.get(configKey) │ │ │ ├─ if client != null && client.verifyConfig(newConfig) ← 配置没变 │ │ └─ return client; // 直接返回缓存的实例 │ │ │ ├─ if client != null && !verifyConfig ← 配置变了 │ │ └─ closeClient(configKey, client); // 优雅关闭旧实例 │ │ │ └─ new DefaultOssClientImpl(configKey, newConfig) │ └─ CLIENT_CACHE.put(configKey, newClient); │ └─ return newClient; │ └─ 8. lock.unlock() (finally)

双重检查锁 — 不仅是性能优化

经典的 DCL（Double-Checked Locking）模式：

// OssFactory.instance(configKey) 核心逻辑（简化）
ReentrantLock lock = getClientLock(configKey);
lock.lock();
try {
    OssClient client = CLIENT_CACHE.get(configKey);
    if (client != null && client.verifyConfig(config)) {
        return client;                           // ← 第二次检查：缓存命中
    }
    if (client != null) {
        closeClient(configKey, client);          // 配置变了 → 关旧的
    }
    OssClient newClient = new DefaultOssClientImpl(configKey, config);
    CLIENT_CACHE.put(configKey, newClient);      // ← 第一次检查已在 lock 外
    return newClient;
} finally {
    lock.unlock();
}
🔑为什么必须上锁？ 如果没有锁，两个并发请求同时发现缓存里没有 client，就会创建两个 S3AsyncClient 实例，各占一份连接池（Netty 线程 + 内存），造成资源泄漏。锁保证同一个 configKey 的初始化一定是串行的。
🔑为什么是 per-key 锁而不是一把全局锁？ CLIENT_LOCKS 是 ConcurrentHashMap<String, ReentrantLock>，每个 configKey 有自己的锁。这样「阿里云 OSS 的初始化」不会阻塞「腾讯云 COS 的初始化」——多个厂商的客户端可以并行创建。

配置热更新机制

这是 OssFactory 最体现工程智慧的地方。假设管理员在后台改了 OSS 的 accessKey：

  1. 修改保存后，新的配置 JSON 写入数据库并刷入缓存（CacheUtils 更新）
  2. 下一个请求调用 OssFactory.instance() 时，从缓存读到的已经是新配置
  3. OssClientConfig.formProperties() 生成新的 config 对象
  4. client.verifyConfig(newConfig) 返回 false（accessKey 变了）
  5. closeClient() 优雅关闭旧实例（释放连接池等资源）
  6. 创建新的 DefaultOssClientImpl，用新凭证建立连接
  7. 新实例覆盖 CLIENT_CACHE，整个过程无需重启服务
// closeClient —— 安全关闭 + 日志告警
private static void closeClient(String configKey, OssClient client) {
    try {
        client.close();
    } catch (Exception e) {
        log.warn("关闭OSS客户端[{}]失败", configKey, e);
    }
}

remove() 方法也走同样的 closeClient 逻辑，用于主动销毁某个客户端实例（比如删除了某个厂商的配置）。

💡 为什么用这个模式？ RuoYi 项目里 OSS 配置存在数据库 sys_oss_config 表中，管理员通过后台 UI 修改后，缓存自动更新。OssFactory 的 verifyConfig + close + recreate 三步走，让「改配置就生效」成为可能——不用发版、不用重启、不用手动清理连接池。

Q1. AbstractOssClientImpl 中，所有公开的 bucketUpload() 方法最终汇聚到哪个私有方法？

A. 汇聚到 doCustomUpload() B. 汇聚到 private bucketUpload(bucket, key, AsyncRequestBody body, Options options)，然后再调 doCustomUpload() C. 每个重载独立调用 s3TransferManager.upload() D. 汇聚到 s3AsyncClient.putObject()
看答案B。所有 12 个 bucketUpload 重载都将输入转为 AsyncRequestBody，然后统一涌入私有的 bucketUpload(bucket, key, AsyncRequestBody, Options)，再委托给 doCustomUpload() → s3TransferManager.upload()。这是模板方法模式的精髓——变的部分（输入类型转换）分散在各重载，不变的部分（构建请求、发起传输、处理结果）集中在两个私有方法里。

Q2. DefaultOssClientImpl 的 doInitialize() 中，asyncExecutor 有两种创建策略：虚拟线程和线程池。选择逻辑是什么？

A. 随机选择 B. 根据 asyncExecutorConfig.enabledVirtualThread() 决定：true 用虚拟线程，false 用 ScheduledThreadPool C. 根据 JDK 版本自动判断 D. 根据 CPU 核心数决定
看答案B。由配置项 asyncExecutorConfig.enabledVirtualThread() 控制。虚拟线程是 Java 21+ 的特性，每个任务一个虚拟线程，不占 OS 线程。关闭虚拟线程时退回到 ScheduledThreadPool，核心线程数也由配置决定。这是「能力可选、降级可用」的设计。

Q3. OssFactory 使用 per-key ReentrantLock（每个 configKey 一把锁）而不是一把全局锁，主要原因是什么？

A. 代码更简单 B. 让不同厂商的客户端可以并行初始化，不会互相阻塞 C. ReentrantLock 比 synchronized 快 D. 全局锁会导致死锁
看答案B。CLIENT_LOCKS 是一个 ConcurrentHashMap<String, ReentrantLock>，通过 computeIfAbsent 为每个 configKey 创建独立的锁。这样「阿里云 OSS 的初始化」不会阻塞「腾讯云 COS 的初始化」——不同 key 的操作完全并行。如果是一把全局锁，多厂商场景下所有初始化都要排队，影响启动性能。
