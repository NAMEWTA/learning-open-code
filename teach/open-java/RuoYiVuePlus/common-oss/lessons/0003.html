第 3 课 · 配置体系：从数据库 JSON 到强类型配置的转换链 config

🎯 本节目标（对应使命 Success #3）：学完你能追踪 OssProperties → OssClientConfig 的完整转换链，说清 resolvePathStyleAccess 的自动推断逻辑，理解 AccessPolicy 枚举如何映射到 AWS S3 ACL。

前两节课我们讲了 OssClient 接口和 AbstractOssClientImpl/DefaultOssClientImpl 的模板方法继承链。但有一个核心问题悬而未决：OssClientImpl 里的 s3AsyncClient 和 OssClientConfig 到底从哪来的？本课讲那根「配置链」——从数据库里一行 JSON，到最终变成一个不可变的 OssClientConfig 的全部转换过程。

一、配置全景 —— 数据流向图

数据库表 sys_oss_config 的一行记录 | | （包含字段：access_key, secret_key, endpoint, bucket_name, region, is_https, access_policy ...） v CacheUtils 存入 Redis（JSON 字符串） KEY: "global:sys_oss:default_config" | | （每次 OssFactory.instance() 调用时） v OssFactory.instance() | | ① CacheUtils.get(DEFAULT_CONFIG_KEY) 取出 Redis 中的 JSON 字符串 v JsonUtils.parseObject(jsonStr, OssProperties.class) | → 得到 OssProperties POJO（9 个 @Data 字段） v OssClientConfig.formProperties(ossProperties) | → formPropertiesBuilder(properties) 做字段映射 + 类型转换 v OssClientConfig（11 个 final 字段，@Builder 构建，不可变） | → 传入 DefaultOssClientImpl 构造函数 v DefaultOssClientImpl 持有 config，用于构建 s3AsyncClient 和 URL 拼接

数据库中实际存储的 JSON 结构（sys_oss_config 表的配置字段反序列化前）：

{
  "endpoint": "https://oss-cn-hangzhou.aliyuncs.com",
  "domainUrl": "",
  "prefix": "ruoyi",
  "accessKey": "LTAI5t...",
  "secretKey": "xxx...",
  "bucketName": "my-bucket",
  "region": "cn-hangzhou",
  "isHttps": "Y",
  "accessPolicy": "2"
}

接下来我们逐个拆解这条链上的每一个类。

二、OssProperties —— 外部配置的 POJO

1定位 ruoyi-common-oss：properties/OssProperties.java
@Data
public class OssProperties {

    private String endpoint;        // 访问站点，如 https://oss-cn-hangzhou.aliyuncs.com
    private String domainUrl;       // 自定义域名（可选，优先于 endpoint）
    private String prefix;          // 对象 key 前缀，如 "ruoyi"
    private String accessKey;       // 云服务商 AccessKey
    private String secretKey;       // 云服务商 SecretKey
    private String bucketName;      // 存储桶名称
    private String region;          // 存储区域，如 "cn-hangzhou"
    private String isHttps;         // 是否 HTTPS（"Y" / "N"）—— 注意是字符串不是 boolean
    private String accessPolicy;    // 桶权限类型（"0"=private, "1"=public-read-write, "2"=public-read）
}

为什么需要这个独立的 POJO？理由有三：

  1. 解耦数据库 schema 和内部类型：数据库里存的是字符串（"Y"/"N"、"0"/"1"/"2"），内部 OssClientConfig 需要的是 boolean、Region、AccessPolicy 枚举等强类型。中间放一个纯字符串 POJO 做缓冲，转换逻辑在 OssClientConfig 里统一处理。
  2. 允许前端直接编辑 JSON：@Data 标记（Lombok 自动生成 getter/setter/toString/equals/hashCode），配合 Jackson 序列化，前端可以拿到一个扁平的 JSON 表单直接编辑，不需要理解内部类型。
  3. 单一数据源：无论配置来自数据库、配置文件还是环境变量，最终都收敛为 OssProperties，再经由 OssClientConfig.formProperties() 统一转换。
💡 设计意图：OssProperties 是「外部世界的代表」——和 JSON 一一对应，弱类型、可序列化。而 OssClientConfig 是「内部世界的代表」——强类型、不可变。两者之间的桥就是 formProperties()。

三、OssClientConfig —— 内部不可变配置

1定位 ruoyi-common-oss：config/OssClientConfig.java

3.1 类注解 —— 不可变 + 相等性比较

@RequiredArgsConstructor   // 为所有 final 字段生成构造器
@Builder                    // 生成 Builder 模式
@EqualsAndHashCode          // 用于配置变更检测：比较两个 config 是否相同
public class OssClientConfig implements
    Config<OssClientConfig, OssClientConfig.OssClientConfigBuilder>,
    Serializable { ... }

@EqualsAndHashCode 是关键——OssFactory 用它来检测配置是否变更。如果新旧 config 的 equals() 返回 true，就复用旧客户端；返回 false 才关闭旧的、创建新的。所有字段都是 final，保证配置一旦创建就不能被意外修改。

3.3 转换桥 —— formPropertiesBuilder() 里的三个关键逻辑

formProperties(OssProperties properties) 调用 formPropertiesBuilder(properties).build()。后者是一个纯映射方法，但有三处值得深究的类型转换：

AresolvePathStyleAccess(OssProperties) —— 自动推断路径风格
OssClientConfig.java:247-250 
private static boolean resolvePathStyleAccess(OssProperties properties) {
    // 旧配置没有显式路径风格字段，只能继续按内置云厂商 endpoint 做兼容推断。
    return !StringUtils.containsAny(properties.getEndpoint(), OssConstant.CLOUD_SERVICE);
}
其中 OssConstant.CLOUD_SERVICE = {"aliyun", "qcloud", "qiniu", "obs"}。
逻辑：如果 endpoint 里包含以上四个关键词任意一个，说明是主流云厂商的 endpoint，它们使用 site-style（虚拟主机风格），即 bucket.endpoint；否则认为是自建 S3 兼容服务（如 MinIO），使用 path-style，即 endpoint/bucket。
设计动机：这是向后兼容的妥协。旧版配置没有 pathStyle 字段，只能靠 endpoint 字符串内容猜测。如果你接入一个新厂商，它的 endpoint 不包含这四个关键词且用 site-style，需要修改此方法或增加配置字段。
BresolveAccessControlPolicy(String) —— 解析 ACL 策略
OssClientConfig.java:258-267 
private static AccessControlPolicyConfig resolveAccessControlPolicy(String accessPolicyString) {
    // 绝大多数云厂商不允许操作 ACL，默认禁用；当前业务只用访问策略判断是否生成预签名 URL。
    if (StringUtils.isBlank(accessPolicyString)) {
        return AccessControlPolicyConfig.DEFAULT;   // enabled=false
    }
    return AccessControlPolicyConfig.builder()
        .enabled(true)
        .accessPolicy(AccessPolicy.formType(accessPolicyString))
        .build();
}
空白字符串 → 返回 DEFAULT（禁用）；非空 → 用 AccessPolicy.formType() 把字符串解析为枚举，构建一个 enabled=true 的配置。注意这里的注释：绝大多数云厂商不允许操作 ACL，所以默认是禁用。当前业务只用它来判断「是否生成预签名 URL」。
CparseRegion(String) —— Region 字符串转对象
OssClientConfig.java:234-239 
private static Region parseRegion(String regionString) {
    if (StringUtils.isBlank(regionString)) {
        return Region.US_EAST_1;          // 空值兜底
    }
    return Region.of(regionString);       // AWS SDK 标准解析
}
空字符串或 null → 回退到 Region.US_EAST_1；否则用 AWS SDK 的 Region.of() 解析。注意这个 fallback 意味着如果 region 没填，实际请求会发到美东——对国内用户来说是隐藏的坑。

3.4 URL 构建方法

方法职责核心逻辑
getEndpointUrl() 获取协议 + endpoint 的完整 URL BucketUrlUtil.rebuildUrlHeader(useHttps, getEndpoint()) —— 根据 useHttps 决定加 http:// 还是 https://
getDomainUrl() 获取访问基础 URL（优先自定义域名） domain 非空 → rebuildUrlHeader(useHttps, domain)；否则 fallback 到 getEndpointUrl()
getBucketUrl() 获取桶完整 URL（无参版） 从 bucket() 取桶名，调 getBucketUrl(bucket)
getBucketUrl(String bucket) 获取指定桶的完整 URL 根据 usePathStyleAccess 决定：path-style → host/bucket；site-style → bucket.host
getAccessBaseUrl() 获取基础 URL（私有） domain 非空 → domain；否则 → endpoint（不带协议头，给 URL 拼接用）

3.5 copy() 和 toBuilder() —— Config 接口实现

@Override
public OssClientConfig copy() {
    return toBuilder().build();           // 防御性拷贝：返回一个完全相同的新对象
}

@Override
public OssClientConfigBuilder toBuilder() {
    return builder()
        .endpoint(endpoint)
        .domain(domain)
        .useHttps(useHttps)
        .usePathStyleAccess(usePathStyleAccess)
        .accessKey(accessKey)
        .secretKey(secretKey)
        .bucket(bucket)
        .region(region)
        .prefix(prefix)
        .accessControlPolicyConfig(accessControlPolicyConfig().copy())   // 递归拷贝
        .asyncExecutorConfig(asyncExecutorConfig().copy());              // 递归拷贝
}

copy() 用在 DefaultOssClientImpl 构造时保存一份快照，防止外部修改原对象影响已运行的客户端。toBuilder() 用于需要基于当前配置做局部修改的场景（虽然当前代码暂未使用此场景，但接口定义了契约）。

四、Config<T, B> 接口 —— 配置对象的统一契约

1定位 ruoyi-common-oss：config/Config.java
public interface Config<T, B> {
    T copy();        // 深拷贝（防御性编程）
    B toBuilder();   // 转为 Builder（支持局部修改重构）
}

这个接口被三个配置类实现：

  • OssClientConfig —— 客户端主配置
  • AccessControlPolicyConfig —— ACL 访问策略配置
  • OssAsyncExecutorConfig —— 异步执行器配置

统一契约的意义：

  • 防御性拷贝（defensive copy）：OssFactory 在构造客户端时调用 config.copy()，确保客户端持有的是配置的快照，而非引用。后续即使外部改了原对象，客户端不受影响。
  • Builder 重构（toBuilder）：允许从现有配置出发，改一两个字段就生成新配置，而不是从头 new。
  • 配置不可变性（config immutability）：配合 Builder 模式 + final 字段，配置对象一旦构建完毕就无法修改，消除了并发环境下的「配置在读取过程中被改了一半」的风险。

五、AccessPolicy 枚举 —— 三种访问级别

1定位 ruoyi-common-oss：enums/AccessPolicy.java
@Getter
@AllArgsConstructor
public enum AccessPolicy {

    PRIVATE(0, BucketCannedACL.PRIVATE, ObjectCannedACL.PRIVATE),
    PUBLIC_READ_WRITE(1, BucketCannedACL.PUBLIC_READ_WRITE, ObjectCannedACL.PUBLIC_READ_WRITE),
    PUBLIC_READ(2, BucketCannedACL.PUBLIC_READ, ObjectCannedACL.PUBLIC_READ);
    // ...
}
枚举值type桶 ACL对象 ACL含义
PRIVATE 0 BucketCannedACL.PRIVATE ObjectCannedACL.PRIVATE 私有读写：只有凭证持有者可访问
PUBLIC_READ_WRITE 1 BucketCannedACL.PUBLIC_READ_WRITE ObjectCannedACL.PUBLIC_READ_WRITE 公共读写：任何人可读写（高风险）
PUBLIC_READ 2 BucketCannedACL.PUBLIC_READ ObjectCannedACL.PUBLIC_READ 公共读私有写：适合静态资源（如图片、CSS）

每个枚举直接持有 AWS SDK 的 BucketCannedACL 和 ObjectCannedACL 常量，使用时不需要二次映射。这样当 DefaultOssClientImpl 需要设 ACL 时，直接 policy.getBucketCannedACL() 即可传入 SDK。

formType(String) 解析方法：

public static AccessPolicy formType(String type) {
    return Arrays.stream(values())
        .filter(policy -> policy.getType().toString().equals(type))
        .findFirst()
        .orElseThrow(() -> S3StorageException.form("'type' not found By " + type));
}

把字符串 "0"、"1"、"2" 匹配到对应枚举。注意比较方式是 getType().toString().equals(type)——这意味着如果配置里写的是 "1" 而不是 "01" 或空白，才能匹配。找不到就抛 S3StorageException，不是返回 null 或默认值。

六、AccessControlPolicyConfig —— 策略配置 record

1定位 ruoyi-common-oss：config/AccessControlPolicyConfig.java
@Builder
public record AccessControlPolicyConfig(
    boolean enabled,
    AccessPolicy accessPolicy
) implements Config<AccessControlPolicyConfig, ...>, Serializable {

    public static final AccessControlPolicyConfig DEFAULT = AccessControlPolicyConfig.builder()
        .enabled(false)
        .accessPolicy(AccessPolicy.PUBLIC_READ_WRITE)
        .build();
}

它是一个 Java record，天然不可变 + equals/hashCode 自动生成。两个字段：

  • enabled：是否启用 ACL 操作。默认 false，因为「绝大多数云厂商不允许操作 ACL」。
  • accessPolicy：访问策略。默认 PUBLIC_READ_WRITE。注意 accessPolicy() 方法标记了 @NonNull，如果用户 builder 时没设值，会 fallback 到 PUBLIC_READ_WRITE。

七、OssConstant —— 系统常量

1定位 ruoyi-common-oss：constant/OssConstant.java
public interface OssConstant {

    // Redis 中默认配置的 key
    String DEFAULT_CONFIG_KEY = GlobalConstants.GLOBAL_REDIS_KEY + "sys_oss:default_config";

    // 内置云服务商标识（用于 resolvePathStyleAccess 推断）
    String[] CLOUD_SERVICE = new String[]{"aliyun", "qcloud", "qiniu", "obs"};

    // 系统预设的配置记录 ID（不可删除的种子数据）
    List<Long> SYSTEM_DATA_IDS = Arrays.asList(
        1761900000000000001L, 1761900000000000002L,
        1761900000000000003L, 1761900000000000004L
    );
}
  • DEFAULT_CONFIG_KEY 拼上了 GLOBAL_REDIS_KEY 前缀，保证不同项目的 Redis 实例隔离。
  • CLOUD_SERVICE 是 resolvePathStyleAccess 判断的核心依据——如果在 endpoint 里匹配到这四个关键词之一，就认为用 site-style。
  • SYSTEM_DATA_IDS 是系统预置的四条配置记录 ID，业务层可能会用来判断「这条配置能不能删」。

Q1. OssProperties 里的 isHttps 是 String 类型（"Y"/"N"），而 OssClientConfig.useHttps 是 boolean。这个转换发生在哪里？

A. 在 OssProperties 的 getter 里 B. 在 OssClientConfig.formPropertiesBuilder() 的 .useHttps(SystemConstants.YES.equals(properties.getIsHttps())) C. 在 OssFactory 里 D. 不需要转换，直接传
看答案B。formPropertiesBuilder() 里用 SystemConstants.YES.equals(properties.getIsHttps()) 将字符串 "Y" 转为 boolean true，其他值一律 false。这是转换桥的核心职责之一。

Q2. resolvePathStyleAccess() 的判断逻辑是：如果 endpoint 包含 "aliyun"/"qcloud"/"qiniu"/"obs" 中任意一个，返回 false。这表示什么？

A. 这些厂商不支持 path-style B. 这些厂商使用 site-style（虚拟主机风格），不需要 path-style；非这些厂商（如 MinIO）使用 path-style C. 这些厂商必须用 path-style D. 这是随机决定的
看答案B。这是向后兼容逻辑：旧版配置没有 pathStyle 字段，只能通过 endpoint 字符串猜测。主流云厂商的 endpoint 包含品牌关键词，默认用 site-style（bucket.endpoint）；自建 S3 兼容服务（MinIO 等）不含这些关键词，默认用 path-style（endpoint/bucket）。

Q3. 如果配置里 accessPolicy 字段是空字符串，最终 OssClientConfig 的 accessControlPolicyConfig 会是什么？

A. null B. 抛异常 C. AccessControlPolicyConfig.DEFAULT —— enabled=false, accessPolicy=PUBLIC_READ_WRITE D. 自动推断为 PRIVATE
看答案C。resolveAccessControlPolicy() 对空白字符串返回 AccessControlPolicyConfig.DEFAULT，其 enabled 为 false。这是因为「绝大多数云厂商不允许操作 ACL」，默认禁用是最安全的策略。
