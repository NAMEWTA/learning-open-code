03 - 配置热更新与 URL 构建

🎯 本节目标：理解 OssFactory 中配置热更新的完整流程，掌握 BucketUrlUtil 的两种 URL 风格（path-style vs site-style）及其自动推断逻辑，能解释预签名 URL 的用途和 presignGetUrl/presignPutUrl 的区别。

1. 全景鸟瞰

前两课讲了客户端创建和异步传输。本课聚焦模块中两个"胶水"能力：配置热更新让运维人员在管理后台改 OSS 配置后无需重启服务；URL 构建让上传后的文件能被正确访问，同时支持多种云厂商的域名风格。

2. 核心设计

2.1 配置热更新的完整流程

回顾 OssFactory 的配置变更检测逻辑：

OssFactory.instance(configKey):
  clientConfig = OssClientConfig.formProperties(ossProperties)
  cachedClient = configMap.get(configKey)

  if cachedClient != null:
    oldConfig = cachedClient.getConfig()
    if oldConfig.equals(clientConfig):        // ← equals 比较
      return cachedClient                       // 配置未变，复用
    else:
      cachedClient.close()                      // ← 安全关闭旧客户端
      return createNewClient(configKey, clientConfig)  // 创建新的

热更新的触发路径：

  1. 管理员在后台修改 OSS 配置 → 保存到 sys_oss_config 表
  2. 缓存更新：CacheUtils.put(configKey, newJson)（写入 Redis）
  3. 下一次业务调用 OssFactory.instance() 时：
  CacheUtils.get(configKey) → 新 JSON
  → OssProperties 解析出新的 endpoint/accessKey/secretKey
  → OssClientConfig.formProperties() → 新配置对象
  → equals(旧配置) → false (因为 endpoint 变了)
  → 旧客户端.close() → 释放 Netty 连接池
  → 创建新 DefaultOssClientImpl → 用新 endpoint 连接
关键细节：配置只在 OssFactory.instance() 被调用时才检查更新。这是 Lazy 检测而非主动推送——如果配置改完后再无业务调用 OSS，旧客户端会一直存活但无人使用。实际生产中，配置改完后很快就会有人上传/下载文件，所以这种 Lazy 检测足够及时且实现简单。

2.2 BucketUrlUtil：两种 URL 风格

S3 协议支持两种 Bucket 寻址方式：

风格URL 格式示例
Path-Style{protocol}://{endpoint}/{bucket}/{key}https://oss-cn-hangzhou.aliyuncs.com/my-bucket/avatar.jpg
Site-Style (Virtual-Hosted){protocol}://{bucket}.{endpoint}/{key}https://my-bucket.oss-cn-hangzhou.aliyuncs.com/avatar.jpg

自动推断逻辑：resolvePathStyleAccess() 检查 endpoint 是否包含已知云厂商关键词：

endpoint.contains("aliyun") → Path-Style
endpoint.contains("qcloud") → Path-Style
endpoint.contains("qiniu")  → Path-Style
endpoint.contains("obs")    → Path-Style
其他                         → Site-Style (默认)

为什么阿里云/腾讯云/七牛云/华为云要强制用 Path-Style？因为这些厂商的 DNS 解析策略不同——如果贸然用 Site-Style（子域名方式），可能因为 DNS 未配置而报 404。

2.3 URL 构建工具方法

方法作用示例
rebuildUrlHeader(isHttps, base)重设协议头（http↔https）https://oss.example.com

2.4 预签名 URL（Presigned URL）

预签名 URL 是 OSS 模块中最实用的能力之一：

方法用途使用场景
presignGetUrl(key, duration)生成带时效的下载链接生成临时分享链接、私有文件访问
presignPutUrl(key, duration)生成带时效的上传链接前端直传 OSS（跳过应用服务器）
// 前端直传场景
String presignedUrl = OssFactory.instance().presignPutUrl("avatars/user123.jpg", Duration.ofMinutes(10));
// → https://my-bucket.oss-cn-hangzhou.aliyuncs.com/avatars/user123.jpg
//   ?X-Amz-Algorithm=AWS4-HMAC-SHA256
//   &X-Amz-Credential=...
//   &X-Amz-Signature=...
//   &X-Amz-Expires=600

// 前端直接用这个 URL 做 PUT 请求上传文件，无需经过后端
fetch(presignedUrl, { method: 'PUT', body: file });
安全考虑：预签名 URL 包含签名和时间戳，前端无法伪造。过期时间（如上例的 10 分钟）要设置合理——太短用户可能来不及上传，太长则有泄露风险。生产环境建议 PUT URL 设置 5-10 分钟，GET URL 按业务需要（如临时分享链接设为 1 小时）。

2.5 S3StorageException：统一异常体系

为什么不用 AWS SDK 的原生异常？

  • AWS SDK 抛出的 S3Exception 包含大量 SDK 内部细节（requestId、extendedRequestId 等），不适合直接暴露给业务层
  • 所有异步回调中的异常（包括 ExecutionException、CancellationException、网络异常等）统一转换为 S3StorageException
  • 提供 4 个静态工厂：form(Throwable)、form(String)、form(String, Throwable)、form(Throwable, String)

3. 实战练习

🧠 提取练习（点击展开）

Q1: 为什么阿里云 OSS 需要 Path-Style 而不是 Site-Style？如果不小心配错了会怎样？

参考: 阿里云 OSS 的 Site-Style（子域名方式）需要额外的 DNS CNAME 配置——需要在阿里云控制台绑定自定义域名并完成备案。如果 endpoint 直接填 oss-cn-hangzhou.aliyuncs.com 但用了 Site-Style，URL 会变成 my-bucket.oss-cn-hangzhou.aliyuncs.com——这个 DNS 记录可能不存在，导致连接超时。resolvePathStyleAccess 的自动推断就是为了避免这种问题。

Q2: 配置热更新是 Lazy 检测（调用时才检查），为什么不做成配置中心推送的实时更新？

参考: 实时推送需要监听配置变更事件（如 Nacos Config listener），增加了复杂度和依赖。Lazy 检测的优势：实现简单、不引入额外依赖、对 Redis 缓存天然友好（缓存本身就是最新的）。代价是配置修改后到下一次 OSS 调用之间有一段"无效期"——但在 OSS 场景中，这段间隔通常不到 1 秒，完全可接受。

Q3: 为什么要区分 presignGetUrl 和 presignPutUrl？它们生成的签名有什么不同？

参考: AWS 签名算法将 HTTP Method 作为签名输入的一部分。GET 和 PUT 的签名不同——GET 签名不能用于 PUT 请求（服务端会校验方法不匹配返回 403）。所以必须分别生成。另一个区别是用途：GET URL 给别人下载（服务端签名客户端只读），PUT URL 给前端上传（服务端签名客户端写入，需要限制文件大小和过期时间）。

4. 调用链路

// 场景：管理员在后台把 OSS endpoint 从 MinIO 切换到阿里云

1. 后台保存配置 → DB: sys_oss_config 更新
2. CacheUtils 更新 Redis 缓存
3. 用户上传头像：
   OssFactory.instance()                               // 第 1 次 after 更新
     → CacheUtils.get("sys_oss:default_config")         // 取到新 JSON
     → OssProperties.endpoint = "oss-cn-hangzhou.aliyuncs.com"
     → OssClientConfig(endpoint, accessKey, secretKey, ...)
     → configMap.get("default_config")
       → cachedClient != null
       → oldConfig.equals(newConfig) → false (endpoint 变了!)
       → cachedClient.close()                           // 关闭 MinIO 的 Netty 连接
       → new DefaultOssClientImpl("default_config", newConfig)
         → doInitialize()
           → resolvePathStyleAccess("oss-cn-hangzhou.aliyuncs.com")
             → contains("aliyun") → true = Path-Style
           → S3AsyncClient.builder()
               .endpointOverride(URI.create("https://oss-cn-hangzhou.aliyuncs.com"))
               .forcePathStyle(true)
               .build()
       → configMap.put("default_config", newClient)
     → client.upload(file)                              // 使用新客户端上传到阿里云 OSS
💡 设计洞察：配置热更新的核心不是"怎么检测变更"，而是"怎么安全地切换客户端"。OssFactory 在检测到配置变更后，先调用旧客户端.close()释放资源，再创建新客户端。这个顺序是精心设计的——如果先创建再关闭，短时间内有两个活跃的客户端连接到不同的 endpoint，可能导致内存翻倍和连接泄漏。先关后开保证了资源的串行使用，虽然有一个短暂的"无客户端"窗口（微秒级），但 OssFactory 方法本身是同步的——业务线程会等待 close()+new 完成才返回新客户端，对调用方完全透明。
