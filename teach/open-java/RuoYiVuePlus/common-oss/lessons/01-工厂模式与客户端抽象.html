01 - 工厂模式与客户端抽象

🎯 本节目标：理解 OssClient 接口 → AbstractOssClientImpl → DefaultOssClientImpl 的模板方法继承链，能追踪 OssFactory.instance() 的完整配置加载与客户端创建逻辑，掌握配置热更新的实现机制。

1. 全景鸟瞰

ruoyi-common-oss 包含 18 个 Java 文件，基于 AWS SDK S3 v2 构建了一套 S3 兼容的对象存储抽象。为什么用 S3 而不是各厂商专有 SDK？因为 S3 协议已成为云对象存储的事实标准——阿里云 OSS、腾讯云 COS、七牛云 Kodo、华为云 OBS、MinIO 都提供了 S3 兼容 API。

模块的 7 个子包分工：

common-oss/
├── client/       → 接口 + 抽象实现 + 默认实现（模板方法模式）
├── config/       → 内部强类型配置对象
├── factory/      → 客户端工厂与缓存管理
├── io/           → 响应式流下载订阅器
├── model/        → 异步结果包装 record
├── properties/   → 外部配置 POJO（来自数据库 JSON）
└── util/         → Bucket URL 构建工具

2. 核心设计

2.1 模板方法继承链

OssClient (接口, 7 个子包)
  ↑ 定义契约: upload / download / delete / presignGetUrl / presignPutUrl
  │          + 每个方法提供带 bucket 和不带 bucket 两套重载
  │
AbstractOssClientImpl (抽象基类, 70+ 方法实现)
  ↑ 模板方法: 所有上传重载汇聚到 bucketUpload()
  │          所有下载重载汇聚到 doCustomDownload()
  │          子类只需实现 doInitialize() 一个方法
  │
DefaultOssClientImpl (唯一子类)
  ↑ 创建三大 AWS SDK 组件:
    S3AsyncClient      → 底层异步 HTTP (Netty NIO)
    S3TransferManager  → 分段上传/下载/重试
    S3Presigner        → 预签名 URL 生成
为什么用模板方法模式？上传/下载有 6 种入参（File / Path / InputStream / byte[] / MultipartFile / Resource），每种又有带/不带 bucket 两个重载。AbstractOssClientImpl 在基类中一次性实现了全部 70+ 个方法，子类只需创建具体的 SDK 客户端。切换云厂商（如从阿里云换成 MinIO）只需改配置，代码零修改。

2.2 OssFactory：工厂 + 缓存 + 热更新

OssFactory.instance() 的完整逻辑：

OssFactory.instance()
  → 从 Redis 读取默认配置 key
    → OssFactory.instance(configKey)
      → CacheUtils 取 JSON 字符串
        → JSON 反序列化为 OssProperties
          → OssClientConfig.formProperties() 转为强类型配置
            → 双重检查锁:
              ① configMap.get(key) 检查是否有缓存客户端
              ② 如果有 → verifyConfig(旧配置, 新配置)
                 ├─ 相同 → 复用
                 └─ 不同 → 关闭旧客户端(.close()) → 创建新实例
              ③ 如果没有 → 上锁 → 再次检查 → 创建新实例
                  → configMap.put(key, newClient)

线程安全设计：

  • ConcurrentHashMap 缓存客户端实例（多 key 并发安全）
  • 每个 configKey 独立 ReentrantLock（不同 key 不互相阻塞）
  • 双重检查锁（DCL）减少加锁开销

2.3 配置体系：三层转换

层级类型来源特点
1. 持久层sys_oss_config 表 JSON数据库页面可编辑
2. 传输层OssProperties POJO从 Redis CacheUtils 读取字段名与 JSON 对应
3. 内部层OssClientConfig 不可变对象formProperties() 转换强类型 + URL 计算方法

OssClientConfig 实现了 equals/hashCode——配置变更检测（verifyConfig）正是通过比较新旧两个 OssClientConfig 对象是否相等来判断的。

3. 实战练习

🧠 提取练习（点击展开）

Q1: 为什么 OssFactory 需要每个 configKey 一个独立的 ReentrantLock，而不是全局一把锁？

参考: 全局锁会导致不同 configKey 的客户端创建互相阻塞。比如 configKey A 正在创建 MinIO 客户端（可能涉及网络连接测试，耗时较长），configKey B 需要创建阿里云 OSS 客户端——如果共享锁，B 会无意义地等待 A。独立锁保证了不同配置的并发创建互不影响。

Q2: 配置热更新时，旧客户端为什么需要先 close() 再创建新的？

参考: AWS SDK 的 S3AsyncClient 内部管理了 Netty 连接池和线程资源。如果不关闭旧客户端就创建新的，旧连接会变成资源泄漏（连接未释放、线程未回收）。先 close() 确保旧客户端优雅关闭——释放连接池、关闭 EventLoopGroup——再创建新客户端连接新的 endpoint。

Q3: 模板方法模式在这里解决了什么问题？如果用策略模式会怎样？

参考: 模板方法解决了「算法骨架相同、部分步骤不同」的问题。上传/下载的流程（校验参数→构建请求→执行→处理结果）是固定的，只有 SDK 客户端创建方式因厂商而异。策略模式要求每个策略实现完整接口，导致 70+ 个方法在每个策略类中重复——违反 DRY 原则。模板方法在基类中实现一次，子类只覆写差异部分。

4. 调用链路

// 业务代码一行调用
OssClient client = OssFactory.instance();

内部路径:
1. RedisUtils.get(OSS_DEFAULT_CONFIG_KEY)
   → key = "sys_oss:default_config"
2. CacheUtils.get(key) → JSON 字符串
3. JSON → OssProperties { endpoint, accessKey, secretKey, bucketName, ... }
4. OssProperties → OssClientConfig (不可变对象)
5. configMap.get(configKey)
   ├─ 已缓存 && equals(旧配置) → 复用
   └─ 未缓存 || 配置已变
      → lock(configKey).lock()
      → new DefaultOssClientImpl(configKey, clientConfig)
        → doInitialize()
          → S3AsyncClient.builder()...build()
          → S3TransferManager.builder().s3Client(asyncClient)...build()
          → S3Presigner.builder().s3Client(asyncClient)...build()
      → configMap.put(configKey, newClient)
💡 设计洞察：OssFactory 的设计是「工厂 + 缓存 + 热更新」三种模式的融合。它不是简单的 new 一个对象返回——它管理着客户端的完整生命周期（创建、复用、关闭）。配置变更检测通过 OssClientConfig 的 equals/hashCode 实现——这意味着只要 endpoint/accessKey/secretKey/bucket 任一字段变了，系统就自动重建客户端，对业务代码完全透明。这是面向接口编程 + 不可变配置 + 双重检查锁的工程典范。
