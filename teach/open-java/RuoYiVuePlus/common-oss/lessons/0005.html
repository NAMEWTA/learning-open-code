第 5 课 · 实战串联：从一行 upload() 到 OSS 返回 URL 的完整旅程 实战

🎯 本节目标：学完你能从 OssFactory.instance().upload(file) 一行代码追踪到底层 S3 API 调用，理解 BucketUrlUtil 的两种 URL 风格，知道切换阿里云 OSS → 腾讯云 COS → MinIO 只需要改数据库配置。

前几课我们拆解了各个零件：工厂、配置、异常、IO。这一课把它们串起来——从 Controller 里一行 upload() 出发，追踪数据在 17 个类之间如何流动，直到 OSS 返回一个可访问的 URL。

一、完整上传链路 — 一行代码的背后

假设 Controller 写了这样一行：

PutObjectResult result = OssFactory.instance().upload(file);

下面逐步追踪这条调用链。每一步标注了发生位置。

阶段 A：获取客户端实例

1 OssFactory.java OssFactory.instance() — 从 Redis 读取默认配置 key → 调用 instance(defaultConfigKey)
2 OssFactory.java instance(configKey) — 从 CacheUtils.get(CacheNames.SYS_OSS_CONFIG, configKey) 读取 JSON → 反序列化为 OssProperties → 调用 OssClientConfig.formProperties(properties) 转换为配置对象
3 OssFactory.java 获取 per-configKey 的 ReentrantLock，双重检查缓存：如果已有客户端且 client.verifyConfig(config) 通过，直接返回；如果配置变更了，先 client.close() 再重建
4 OssFactory.java new DefaultOssClientImpl(configKey, config) → 调用父类构造函数 → AbstractOssClientImpl.initialize() → doInitialize() 创建 S3AsyncClient、S3TransferManager、S3Presigner — 存入缓存返回

阶段 B：路由到抽象层

5 OssClient.java / AbstractOssClientImpl.java upload(File file) → bucketUpload(defaultBucket(), buildPathKey(fileName), file) — 自动选择默认 bucket（无参版本）
6 AbstractOssClientImpl.java buildPathKey(fileName) — 拼接：prefix + yyyy/MM/dd + UUID + 文件后缀，得到类似 "ruoyi/2026/07/05/a1b2c3d4e5f6.jpg" 的 key

阶段 C：类型转换链

7 AbstractOssClientImpl.java bucketUpload(bucket, key, File) → File 转为 RandomAccessFile → 读取 ReadableByteChannel（contentLength 来自 file.length()）
8 AbstractOssClientImpl.java bucketUpload(bucket, key, ReadableByteChannel, contentLength, Options) — 如果是 SeekableByteChannel 用 byteChannel.size() 获取精确长度；否则包装为 InputStream（通过 Channels.newInputStream）→ 创建临时文件 → 拷贝流 → 从临时文件 Path 创建 AsyncRequestBody

阶段 D：核心上传

9 AbstractOssClientImpl.java 私有方法 bucketUpload(bucket, key, AsyncRequestBody, Options) — 所有上传路径的汇聚点。解析 contentLength、contentType、md5Digest、metadata、transferListeners，调用 doCustomUpload(body, putObjectBuilder, handleAsyncAction)
10 AbstractOssClientImpl.java doCustomUpload(AsyncRequestBody body, Consumer<PutObjectRequest.Builder> putBuilderConsumer, BiFunction<CompletedUpload, Throwable, HandleAsyncResult<PutObjectResult>> handleAsyncAction) — 构建 UploadRequest → s3TransferManager.upload(uploadRequestBuilder) → 返回 CompletableFuture → .handleAsync(handleAsyncAction) → .join()
11 AbstractOssClientImpl.java 成功回调：从 CompletedUpload 提取信息 → PutObjectResult.form(bucketUrl + "/" + key, key, eTag, size)。得到的 URL 形如 https://bucket.oss-cn-beijing.aliyuncs.com/ruoyi/2026/07/05/a1b2c3d4e5f6.jpg
12 AbstractOssClientImpl.java finally 块：对于 InputStream 路径创建的临时文件，在此处删除
Controller │ OssFactory.instance() │ └─ Redis → OssProperties → OssClientConfig → DefaultOssClientImpl (cached) │ │ .upload(file) ▼ AbstractOssClientImpl │ upload(key, file) → bucketUpload(bucket, buildPathKey(name), file) │ buildPathKey → prefix + yyyy/MM/dd + uuid + .jpg │ bucketUpload → File→RandomAccessFile→ReadableByteChannel │ → InputStream→temp file→AsyncRequestBody │ private bucketUpload → doCustomUpload(body, builder, handler) ▼ S3TransferManager.upload(UploadRequest) │ → CompletableFuture → .handleAsync(handler) → .join() ▼ PutObjectResult { url, key, eTag, size } │ url = bucketUrl + "/" + key │ e.g. https://bucket.oss-cn-beijing.aliyuncs.com/ruoyi/2026/07/05/abc.jpg

一句话总结上传链路：工厂从 Redis 取配置创建客户端 → 客户端把 Java 对象逐层转为 AWS SDK 的 AsyncRequestBody → S3TransferManager 异步上传 → CompletableFuture 同步等待 → 拼接 bucket URL 返回。

二、完整下载链路

下载比上传简单——输入是 key，输出是文件或流。

1 download(key) → bucketDownload(defaultBucket(), key, path) — 自动选取默认 bucket
2 bucketDownload(bucket, key, Path) → OutputStreamDownloadSubscriber.create(path) — 将输出目标包装为 Reactive Streams Subscriber
3 doCustomDownload(getReqBuilder, AsyncResponseTransformer.toPublisher()) → 构建 DownloadRequest → s3TransferManager.download(downloadRequest) → 返回 Publisher<ByteBuffer>
4 Publisher<ByteBuffer> → .subscribe(outputStreamDownloadSubscriber) — ByteBuffer 逐块写入目标文件
5 从 GetObjectResponse 的 metadata 构建 GetObjectResult：key、eTag、lastModified（UTC LocalDateTime）、size、contentType、contentDisposition、contentRange、contentEncoding、contentLanguage、metadata（Map）
download(key) │ bucketDownload(bucket, key, path) ▼ OutputStreamDownloadSubscriber.create(path) ←─ Reactive Streams Subscriber │ │ doCustomDownload(builder, transformer) ▼ S3TransferManager.download(DownloadRequest) │ → Publisher<ByteBuffer> │ → .subscribe(subscriber) ▼ GetObjectResult { key, eTag, lastModified, size, contentType, ... }

三、BucketUrlUtil — URL 的两种面孔

同一个 bucket，有两种 URL 表示方式。这个工具类决定了最终返回给前端的 URL 长什么样。

Path-Style（路径风格）

bucket 名出现在 URL 路径中：

格式: {protocol}://{base}/{bucketName} 示例: https://s3.example.com/my-bucket

适用于：自建 MinIO、非标准 S3 兼容服务。当你部署 MinIO 在 http://192.168.1.100:9000 时，bucket URL 就是 http://192.168.1.100:9000/my-bucket。

Site-Style（虚拟主机风格，Virtual-Hosted）

bucket 名出现在子域名中：

格式: {protocol}://{bucketName}.{base} 示例: https://my-bucket.s3.example.com

适用于：阿里云 OSS、腾讯云 COS、七牛云。阿里云的 endpoint 如 oss-cn-beijing.aliyuncs.com，最终 URL 为 https://bucket.oss-cn-beijing.aliyuncs.com。

四个核心方法

方法作用示例
rebuildUrlHeader(boolean isHttps, String base)剥离已有协议头，重新拼接 http:// 或 https://rebuildUrlHeader(true, "s3.example.com") → "https://s3.example.com"
removeHttpProtocolHeader(String url)去掉 http:// 和 https:// 前缀（大小写不敏感）removeHttpProtocolHeader("https://s3.example.com") → "s3.example.com"
getPathStyleBucketUrl(boolean isHttps, String base, String bucketName)拼接路径风格 URLgetPathStyleBucketUrl(true, "s3.example.com", "images") → "https://s3.example.com/images"
getSiteStyleBucketUrl(boolean isHttps, String base, String bucketName)拼接站点风格 URLgetSiteStyleBucketUrl(true, "oss-cn-beijing.aliyuncs.com", "images") → "https://images.oss-cn-beijing.aliyuncs.com"

在哪里决定用哪种风格

关键在 OssClientConfig.getBucketUrl(String bucket) 方法中：

public String getBucketUrl(String bucket) {
    if (usePathStyleAccess) {
        return BucketUrlUtil.getPathStyleBucketUrl(useHttps, getEndpointUrl(), bucket);
    }
    return BucketUrlUtil.getSiteStyleBucketUrl(useHttps, getEndpointUrl(), bucket);
}

usePathStyleAccess 在 OssClientConfig.formPropertiesBuilder() 中通过 resolvePathStyleAccess(properties) 自动判断：遍历 OssConstant.CLOUD_SERVICE 数组（含 "aliyun"、"qcloud"、"qiniu"、"obs"），如果 endpoint 包含其中任一关键词 → 使用 site-style（false）；否则 → path-style（true）。

四、多厂商切换实战

场景 1：阿里云 OSS → 腾讯云 COS

只需要在数据库配置表里修改 4 个字段：

字段阿里云 OSS→ 腾讯云 COS
endpointoss-cn-beijing.aliyuncs.comcos.ap-beijing.myqcloud.com
accessKeyLTAI5t...AKID...
secretKey（阿里云密钥）（腾讯云密钥）
bucketNamemy-bucketmy-bucket-1234567890

代码一行不变。因为 SDK 对话的对象是 S3 协议，不是厂商 API。

场景 2：公有云 → MinIO 自建

将 endpoint 改为自建 MinIO 地址，如 http://192.168.1.100:9000。由于 endpoint 不含 CLOUD_SERVICE 中的任何关键词，resolvePathStyleAccess 自动设为 path-style，无需手动配置。

场景 3：双厂商并存

在 sys_oss_config 表中创建两条记录，configKey 分别为 "oss_aliyun" 和 "oss_minio"。使用时：

OssClient aliyunClient = OssFactory.instance("oss_aliyun");
OssClient minioClient  = OssFactory.instance("oss_minio");
aliyunClient.upload(file);   // 上阿里云
minioClient.upload(file);    // 上 MinIO

每个 configKey 在工厂中有独立的锁和缓存，互不干扰。

💡 S3 兼容性的魔法：因为 SDK 底层全程使用 S3 协议（REST API），任何兼容 S3 协议的服务都能即插即用。阿里云 OSS、腾讯云 COS、七牛云 Kodo、华为云 OBS、MinIO、Ceph RGW — 只要实现了 S3 API，就无需修改任何 Java 代码。

五、异常处理体系

S3StorageException — 统一异常包装

S3StorageException 继承 RuntimeException，提供 4 个构造函数和 4 个静态工厂方法 form()：

签名用途
S3StorageException(String message)纯消息异常，如 "bucket not configured"
S3StorageException(String message, Throwable cause)包装底层异常，保留原因链
S3StorageException(Throwable cause)仅包装异常
S3StorageException(String message, Throwable cause, boolean enableSuppression, boolean writableStackTrace)完整控制
form() 静态工厂与构造函数的参数一一对应，语义等价的别名。

异常抛出点

位置条件消息
OssFactory.instance()Redis 中读不到配置"oss configKey [xxx] not found"
DefaultOssClientImpl.doInitialize()accessKey 或 secretKey 为空"accessKey or secretKey is empty"
AbstractOssClientImplbucket 未配置"bucket not configured"
AbstractOssClientImpl上传返回 null"upload response is empty"
AbstractOssClientImplS3 SDK 抛出任何异常经 toStorageException() 转换

toStorageException() — 异步异常解包

核心逻辑：

  1. unwrapAsyncException(Throwable) — 循环 while cause 是 CompletionException 或 ExecutionException，逐层剥开拿到真正的根因
  2. 如果剥开后已经是 S3StorageException，直接返回
  3. 否则 new S3StorageException(message, unwrapped) 包装

业务层捕获方式：

try {
    PutObjectResult result = OssFactory.instance().upload(file);
    return R.ok(result.getUrl());
} catch (S3StorageException e) {
    return R.fail(e.getMessage());
}

六、预签名 URL — 临时访问凭证

预签名 URL 让前端可以直接与 OSS 交互，不经过后端服务器中转数据。

方法作用
presignGetUrl(String key, Duration expiredTime)生成限时 GET URL，浏览器直接下载文件
presignPutUrl(String key, Duration expiredTime, Map<String, String> metadata)生成限时 PUT URL，浏览器直接上传文件到 OSS

实现原理：

s3Presigner.presignGetObject(req ->
    req.bucket(bucket)
       .key(key)
       .signatureDuration(expiredTime)
);

典型应用场景：前端请求后端获取预签名 PUT URL → 前端直接用该 URL 上传文件到 OSS → 上传完成后通知后端记录文件信息。后端不处理文件数据流，只负责签发凭证和记录元数据。

两个 Java record，纯数据载体，简洁无冗余。

课后自测

1. 在完整上传链路中，所有输入类型（File、Path、InputStream、byte[] 等）最终汇聚到哪个私有方法？

A. doCustomUpload() B. private bucketUpload(bucket, key, AsyncRequestBody, Options) C. s3TransferManager.upload() D. AbstractOssClientImpl.upload()
查看答案

B。所有上传方法（6 种输入类型 x 2 种 bucket 选择 = 12 个方法）最终都收敛到 private bucketUpload(bucket, key, AsyncRequestBody body, Options options)，再由它调用 doCustomUpload()。

2. 将一个正在使用阿里云 OSS 的项目切换到 MinIO，resolvePathStyleAccess() 返回什么？为什么？

A. false，因为 MinIO 是云服务 B. true，因为 MinIO 的 endpoint 不含 CLOUD_SERVICE 数组中的关键词，无匹配则默认 path-style C. false，需要在数据库配置中手动设置 D. null，由 SDK 自动选择
查看答案

B。resolvePathStyleAccess() 遍历 CLOUD_SERVICE = {"aliyun", "qcloud", "qiniu", "obs"}，检查 endpoint 是否包含其中任一。MinIO 的 endpoint 如 "http://192.168.1.100:9000" 不含任何匹配，返回默认值 true（path-style）。

3. toStorageException() 中 unwrapAsyncException() 的作用是什么？

A. 将所有异常转为 S3StorageException B. 跳过 CompletionException 和 ExecutionException 包装层，取出真正的根因 C. 将异常信息写入日志 D. 将多个异步异常合并为一个
查看答案

B。CompletableFuture 在发生异常时会将真实异常包装在 CompletionException（或 ExecutionException）中。通过 while 循环持续调用 cause.getCause() 直到不再是这两种类型，取出真正的根因（如网络超时、认证失败等），然后再包装为 S3StorageException 返回给业务层。
