第 1 课 · 鸟瞰：18 个类如何协作构建 S3 兼容存储层 入门地图

🎯 本节目标：学完你能画出 ruoyi-common-oss 10 个子包的分工图，并说出 OssClient → AbstractOssClientImpl → DefaultOssClientImpl 的继承链中每一层干什么。这是后面所有细节挂靠的骨架——先有地图，再走小路。

我们先不抠任何一行实现，只回答一个问题：打开 ruoyi-common-oss 这个模块，你会看到什么？

一、整个模块有多大

如果说 ruoyi-common-sms 是一个「三件套」的精简适配垫片，那 OSS 模块就是一座「轻量级工厂」——拢共 18 个 Java 文件，分布在 10 个子包中：

ruoyi-common-oss/src/main/java/org/dromara/common/oss/ ├── client/ ← 【契约 + 骨架 + 实现】(3 个) │ ├── OssClient.java ← 接口：60+ 方法签名，定义全部 OSS 操作 │ ├── AbstractOssClientImpl.java ← 抽象类：实现 70+ 方法，子类只覆写 doInitialize() │ └── DefaultOssClientImpl.java ← 实现类：doInitialize() 中组装 S3 三剑客 ├── config/ ← 【配置对象】(4 个) │ ├── Config.java ← 泛型接口：Config │ ├── OssClientConfig.java ← 核心配置：endpoint/bucket/region/ACL/线程池 │ ├── AccessControlPolicyConfig.java ← ACL 策略配置（record） │ └── OssAsyncExecutorConfig.java ← 异步线程池配置（record） ├── constant/ │ └── OssConstant.java ← 【常量】Redis Key 前缀、默认配置 Key ├── enums/ │ └── AccessPolicy.java ← 【策略枚举】PRIVATE / PUBLIC_READ_WRITE / PUBLIC_READ ├── exception/ │ └── S3StorageException.java ← 【异常】统一 OSS 操作异常 ├── factory/ │ └── OssFactory.java ← 【工厂】单例缓存 / 配置校验 / 客户端生命周期管理 ├── io/ │ └── OutputStreamDownloadSubscriber.java ← 【适配器】Consumer<ByteBuffer> → WritableByteChannel ├── model/ ← 【数据传输对象】(4 个) │ ├── Options.java ← 上传选项：contentType / metadata / listeners │ ├── PutObjectResult.java ← 上传结果：url / key / eTag / size │ ├── GetObjectResult.java ← 下载结果：contentLength / contentType │ └── HandleAsyncResult.java ← 异步结果包装 ├── properties/ │ └── OssProperties.java ← 【配置载体】从 Redis JSON 反序列化 └── util/ └── BucketUrlUtil.java ← 【工具】拼接 Bucket URL（路径式 / 域名式）

与 ruoyi-common-sms 的 3 个类不同，OSS 模块自己承担了大量上传/下载/预签名逻辑，直接把 AWS S3 SDK 封装成了一套统一 API。第三方库只有 software.amazon.awssdk:s3（及其 transfer-manager 扩展），没有引入任何 Dromara 社区的上层框架。

💡 一句话定位：
ruoyi-common-oss 是一层「S3 兼容存储的轻量封装工厂」——它把 AWS S3 SDK 的 S3AsyncClient / S3TransferManager / S3Presigner 三剑客组装成一条 OssClient 接口，让业务代码可以透明地切换阿里云 OSS / 腾讯云 COS / 七牛云 / MinIO 等任何 S3 兼容存储，一套 API，换配置不换代码。

二、核心设计模式

18 个类不是乱堆的——它们遵循一套精心设计的分层模式。记住这张表，本课就成功了一半：

设计模式体现位置解决什么问题
模板方法模式 AbstractOssClientImpl 写了 70+ 方法，DefaultOssClientImpl 只覆写一个 doInitialize() 上传/下载/删除/预签名等通用逻辑一次写完，子类只需实现「如何初始化 SDK 客户端」这一个变化点
工厂模式 OssFactory 管理 CLIENT_CACHE（ConcurrentHashMap）+ CLIENT_LOCKS（ReentrantLock） 按 configKey 缓存客户端实例，配置不变则复用，配置变更则销毁重建，避免频繁创建 S3 连接
策略模式 AccessPolicy 枚举封装 PRIVATE / PUBLIC_READ_WRITE / PUBLIC_READ 三种 ACL 不同权限策略对应不同的 BucketCannedACL 和 ObjectCannedACL，调用方只传一个 type 字符串即可
Builder 模式 OssClientConfig（@Builder）、Options（@Accessors(chain=true)）、AccessControlPolicyConfig（@Builder）都用链式构建 配置对象字段多（11 个），Builder 让构建代码清晰可读，避免冗长构造函数
适配器模式 OutputStreamDownloadSubscriber 实现 Consumer<ByteBuffer>，内部持有 WritableByteChannel S3 TransferManager 的异步下载回调需要 Consumer 接口，但业务侧期望写入 OutputStream/FileChannel，适配器桥接两边
🤔 想一想：为什么 AbstractOssClientImpl 写 70+ 方法，子类只要覆写 doInitialize()？为什么不在接口里直接写 default 方法？ 因为需要共享状态——AbstractOssClientImpl 持有 s3AsyncClient / s3TransferManager / s3Presigner / asyncExecutor 这些 protected 字段，所有上传/下载方法都依赖它们。Java 接口的 default 方法无法访问实例字段。抽象类作为「骨架」提供字段 + 方法实现，子类只需填空 doInitialize() 这一个初始化钩子——这就是模板方法模式的核心价值。

三、核心调用链全景

理解了前两节的结构，现在看一条「一次 OSS 上传」从头到尾经过哪些对象。这条链路是你读懂整个模块的导航线：

业务代码 公共层 (ruoyi-common-oss) 第三方 (AWS SDK) ──────── ──────────────────────── ──────────────── SysOssController.upload(file) │ ▼ OssFactory.instance() ───────────────────────────────────────────────────── OssConstant.DEFAULT_CONFIG_KEY │ 1. 读 Redis: CacheUtils.get(SYS_OSS_CONFIG, configKey) ↓ │ 2. JSON → OssProperties → OssClientConfig.formProperties() Redis 里的配置 JSON │ 3. 拿 ReentrantLock（按 configKey 粒度） │ 4. 若缓存命中且 verifyConfig 通过 → 复用 │ 5. 否则 new DefaultOssClientImpl(configKey, config) ▼ ossClient.upload(key, file) │ OssClient 接口 → AbstractOssClientImpl 骨架方法 ▼ AbstractOssClientImpl.bucketUpload(bucket, key, AsyncRequestBody body, Options) │ 1. 从 body/options 提取 contentLength / contentType / MD5 / metadata / listeners │ 2. 构建 PutObjectRequest ▼ AbstractOssClientImpl.doCustomUpload(body, requestBuilder, listeners) │ ▼ s3TransferManager.upload(UploadRequest) ──────────────────────────────────► AWS S3 SDK │ S3TransferManager（封装 S3AsyncClient） │ │ 返回 CompletableFuture<CompletedUpload> │ ▼ ▼ PutObjectResult(url, key, eTag, size) 阿里云 OSS / 腾讯云 COS │ / 七牛云 Kodo / MinIO ▼ / 任何 S3 兼容存储 返回给 Controller → 存入 sys_oss 表

这条链路上，OssFactory 管「客户端从哪来」，OssClientConfig 管「连到哪个厂商」，AbstractOssClientImpl 管「怎么上传/下载」，DefaultOssClientImpl 管「初始化哪三个 SDK 对象」。四层分工明确，各司其职。

四、两条关键通路：上传与下载

上传和下载是 OSS 模块最核心的两条数据流。它们在 AbstractOssClientImpl 内部走的是两条不同的内部管道：

通路一：上传 (File → OSS)

用户文件 │ ▼ File / Path / InputStream / byte[] / RandomAccessFile / ReadableByteChannel │ （6 种输入类型，6 个 bucketUpload() 重载） │ 统一转换为 ──► ▼ AsyncRequestBody.fromFile(path) 或 fromChannel(channel, contentLength, executor) │ （AWS SDK 的异步请求体） ▼ AbstractOssClientImpl.bucketUpload(bucket, key, body, options) │ 提取 contentLength / contentType / MD5 / metadata / TransferListeners │ 构建 PutObjectRequest.Builder ▼ AbstractOssClientImpl.doCustomUpload(body, requestBuilder, listeners) │ 调用 ──► ▼ s3TransferManager.upload(UploadRequest) │ 返回 CompletableFuture<CompletedUpload> │ .join() 等待完成 ▼ CompletedUpload → 提取 eTag │ ▼ PutObjectResult(url, key, eTag, size)

关键点：所有 6 种输入类型最终都收敛到私有方法 bucketUpload(bucket, key, AsyncRequestBody body, Options options)——这是上传的「唯一漏斗」。

通路二：下载 (OSS → 本地)

GetObjectRequest.Builder（bucket + key） │ ▼ AbstractOssClientImpl.doCustomDownload(requestBuilder, transformer, listeners) │ ▼ s3TransferManager.download(DownloadRequest) │ 返回 CompletableFuture<CompletedDownload> │ ▼ AsyncResponseTransformer<GetObjectResponse, T> │ （三种形式：） │ ├── toFile(Path) → 直接写本地文件 │ ├── toFile(File) → 直接写本地文件 │ └── OutputStreamDownloadSubscriber → WritableByteChannel / OutputStream │ ▼ GetObjectResult(contentLength, contentType)

关键点：OutputStreamDownloadSubscriber 作为 Consumer<ByteBuffer>，接收 S3 SDK 推送的每个数据块，通过 WritableByteChannel.write() 写入目标——当目标 OutputStream 是 FileOutputStream 时，会直接提取 FileChannel 以获得更好的写入性能。

五、与其他 common 模块对比

把 OSS 模块和之前学过的 SMS 模块放在一起看，能找到 RuoYi common 模块的统一设计基因：

维度ruoyi-common-smsruoyi-common-oss
类文件数 3 个 Java 类 + 1 个配置文件 18 个 Java 类
第三方依赖 sms4j（上层封装框架） AWS S3 SDK（底层 SDK，直接封装）
装配方式 SmsAutoConfiguration（Spring @Bean） OssFactory.instance()（静态工厂，Redis 驱动）
缓存策略 PlusSmsDao → RedisUtils（覆盖 sms4j 默认缓存） OssFactory.CLIENT_CACHE → ConcurrentHashMap（客户端实例缓存）
异常处理 SmsExceptionHandler（@RestControllerAdvice 统一兜底） S3StorageException（RuntimeException，调用方自行捕获）
配置来源 application.yml（sms4j 原生配置格式） Redis JSON → OssProperties → OssClientConfig（数据库驱动配置）
设计模式 适配器（适配 sms4j 到 RuoYi 技术栈） 模板方法 + 工厂 + 策略 + Builder + 适配器
💡 共同骨架：虽然规模差 6 倍，但两个模块的核心套路完全一致——依赖一个成熟库（sms4j / AWS SDK），补齐「配置装配 + 客户端缓存 + 异常兜底」三件套。SMS 把这叫「适配垫片」，OSS 把这叫「工厂封装」——本质上都是在降低业务层的接入门槛。

六、为什么值得学

读懂 ruoyi-common-oss，你就拿到了读懂 RuoYi 几乎所有 common 模块的密码：

  • ruoyi-common-mail：同样是工厂模式封装第三方库（JavaMail），同样是配置驱动、客户端缓存；
  • ruoyi-common-redis：同样是抽象层 + 多实现（单机 Redis / 集群 Redis），同样是模板方法 + 工厂；
  • ruoyi-common-sms：骨架完全一致，只是规模更小、依赖更上层；
  • ruoyi-common-* 的通用范式：①依赖成熟库 ②工厂/装配管理客户端生命周期 ③配置从 Redis/yml 读取 ④统一异常封装 ⑤对业务层暴露一个「一句话能调」的 API。

OSS 模块是这套范式里规模最大、模式最全的一个——学透它，剩下的都是降维打击。

七、检验一下（提取练习）

不要回看上文，凭记忆作答——回忆比重读更能形成长期记忆。

Q1. OssClient → AbstractOssClientImpl → DefaultOssClientImpl 这条继承链体现了什么设计模式？

A. 单例模式——保证全局只有一个 OssClient 实例 B. 模板方法模式——父类写 70+ 方法的骨架逻辑，子类只覆写 doInitialize() 这一个钩子 C. 观察者模式——S3 SDK 事件回调给 OssClient D. 装饰器模式——DefaultOssClientImpl 给 AbstractOssClientImpl 增加新功能
看答案B。AbstractOssClientImpl 实现了所有上传/下载/删除/预签名的通用逻辑，子类 DefaultOssClientImpl 只需要覆写 doInitialize() 来组装 S3AsyncClient / S3TransferManager / S3Presigner。这是模板方法模式的经典应用——骨架在父类，变化点留给子类填空。

Q2. OssFactory.instance("config1") 被调用两次，配置没有变化，会创建几个 OssClient 实例？

A. 2 个——每次调用都创建新实例 B. 1 个——第二次命中 CLIENT_CACHE 缓存，verifyConfig 通过则直接复用 C. 0 个——OssFactory 只是返回 null D. 取决于 Redis 里有没有配置
看答案B。OssFactory 内部维护了一个 ConcurrentHashMap 做客户端缓存。第二次调用时，先从缓存取出已有实例，调用 client.verifyConfig(config) 对比新旧配置（基于 OssClientConfig 的 @EqualsAndHashCode），相同则直接返回缓存实例，不重新创建。

Q3. 上传文件时，File / Path / InputStream / byte[] 等 6 种输入类型，最终都收敛到哪个方法？

A. 各自调用不同的 doCustomUpload() 重载，没有收敛点 B. 都收敛到私有方法 bucketUpload(bucket, key, AsyncRequestBody body, Options options) C. 都直接调用 s3TransferManager.upload() D. 都先转成 File 再处理
看答案B。所有公开的 bucketUpload() 重载先将各自输入类型转换为 AsyncRequestBody（Path → fromPath、InputStream → fromChannel 等），再调用私有方法 bucketUpload(bucket, key, body, options)——这是上传的「唯一漏斗」，所有上传逻辑只写一遍。
