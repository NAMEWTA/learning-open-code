02 - 异步上传下载体系

🎯 本节目标：追踪 upload(File) 的完整异步执行路径，理解 CompletableFuture + handleAsync 的异步编排模式，掌握 OutputStreamDownloadSubscriber 的响应式流下载机制，以及 OssAsyncExecutorConfig 的虚拟线程 vs 线程池选择。

1. 全景鸟瞰

传统的文件上传下载是同步阻塞的——线程等待网络 IO 完成才能继续。在大文件场景下，这会导致线程池快速耗尽。OSS 模块全面采用异步非阻塞模式：基于 CompletableFuture 编排、S3TransferManager 分段传输、OutputStreamDownloadSubscriber 响应式流消费。

核心数据流：

上传: File → RandomAccessFile → ReadableByteChannel
      → AsyncRequestBody → S3TransferManager.upload()
      → CompletableFuture → PutObjectResult

下载: S3TransferManager.download()
      → CompletableFuture → OutputStreamDownloadSubscriber
      → Consumer<ByteBuffer> → WritableByteChannel → File/OutputStream

2. 核心设计

2.1 upload(File) 的完整代码路径

以 upload(File) 为例，追踪模板方法如何将所有上传重载汇聚到同一个私有方法：

OssClient.upload(File file)
  → upload(bucket, file)               // 自动填入默认 bucket
    → upload(bucket, file.getName(), file)  // 用文件名作为 key
      → bucketUpload(bucket, key, AsyncRequestBody, Options)
        // ↑ 所有上传重载最终汇聚于此（私有方法）
        → 构建 S3 PutObjectRequest
        → 构建 UploadRequest (含 ACL、分段阈值等)
        → S3TransferManager.upload(uploadRequest)
          → 返回 CompletableFuture<CompletedUpload>
          → handleAsync(future, executor)
            → 成功 → PutObjectResult(response)
            → 失败 → S3StorageException.form(cause)

为什么有 6 种入参但只有一个汇聚点？

  • File → FileChannel.open(path).map(...) → ByteBuffer
  • InputStream → AsyncRequestBody.fromInputStream(is, size, executor)
  • byte[] → AsyncRequestBody.fromBytes(bytes)
  • Path → 同 File 逻辑
  • MultipartFile → 先转 byte[] 或 InputStream
  • Resource → InputStream 包装

所有入参最终都转换为 AsyncRequestBody——AWS SDK 的统一请求体抽象。汇聚到一个私有方法（bucketUpload）后，上传逻辑（构建请求、执行、异常处理、结果包装）只写一次。

2.2 CompletableFuture + handleAsync 异步编排

模块中大量使用 handleAsync 模式：

future.handleAsync((result, throwable) -> {
    if (throwable != null) {
        throw S3StorageException.form(throwable);
    }
    return convertResult(result);
}, executor);

handleAsync 的优势：

  • 同时处理成功和失败（一个回调覆盖两种结果）
  • 指定 executor 执行回调（不占用 SDK 的 IO 线程）
  • 异常统一转换为 S3StorageException（模块自身的异常体系）

2.3 OssAsyncExecutorConfig：虚拟线程 or 线程池

DefaultOssClientImpl.doInitialize() 根据配置决定异步执行器：

配置执行器适用场景
enabledVirtualThread=trueExecutors.newVirtualThreadPerTaskExecutor()高并发小文件（如头像上传）
enabledVirtualThread=falseExecutors.newScheduledThreadPool(corePoolSize)大文件传输（需要限制并发数）
为什么大文件场景需要限流？虚拟线程数量无上限，但网络带宽和磁盘 IO 是有限的。如果同时上传 1000 个大文件，虚拟线程会全部创建但同时争抢带宽，导致每个都慢。固定线程池可以控制并发数（如 corePoolSize=CPU 核数），保证每个线程获得足够的网络资源。

2.4 OutputStreamDownloadSubscriber：响应式流下载

实现 Consumer<ByteBuffer> 和 AutoCloseable，是异步下载的数据消费者：

accept(ByteBuffer byteBuffer):
  while (byteBuffer.hasRemaining()) {
      channel.write(byteBuffer);  // 写入目标 Channel
  }
  // FileOutputStream 优化：直接用 FileChannel (bypass Channels.newChannel)

FileOutputStream 优化：当 target 是 FileOutputStream 时，直接使用 FileOutputStream.getChannel() 而不是 Channels.newChannel() 包装，减少一层间接调用。这是性能敏感的 IO 代码中常见的微优化。

2.5 HandleAsyncResult：成功/失败的统一包装

Java record 类型，两个字段：result: T 和 error: Throwable：

HandleAsyncResult<PutObjectResult> result = client.doCustomUpload(request);
if (result.isSuccess()) {
    return result.getResult().eTag();
} else {
    throw S3StorageException.form(result.getError());
}

为什么不用 CompletableFuture 直接返回？因为 doCustomUpload 需要同时提供同步和异步两种使用方式。HandleAsyncResult 让调用方可以选择同步检查结果，也可以继续用 CompletableFuture 链式编排。

3. 实战练习

🧠 提取练习（点击展开）

Q1: 为什么所有上传方法最终汇聚到 bucketUpload 一个私有方法，而不是每个 public 方法各自实现？

参考: 避免逻辑重复和性能差异。请求构建、ACL 设置、异常处理、结果包装这些逻辑对所有入参类型完全一致。汇聚到一个方法后，修改一次即可影响所有 12 个上传重载。如果各自实现，容易产生行为不一致的 bug（如只给 File 加了 ACL，忘记给 byte[] 加）。

Q2: handleAsync 和 thenApply/thenAccept 在异常处理上有什么区别？

参考: thenApply 只处理成功，异常会沿 CompletableFuture 链传播直到被 exceptionally 捕获。handleAsync 同时接收 result 和 throwable 两个参数，一家回调完成成功和失败的处理。在 OSS 模块中，handleAsync 更适合——因为无论成功还是失败都要转换为统一的 PutObjectResult 或 S3StorageException。

Q3: OutputStreamDownloadSubscriber 为什么要单独优化 FileOutputStream？

参考: FileOutputStream.getChannel() 返回与文件直接关联的 FileChannel，可以调用 position()、truncate() 等原生操作。而 Channels.newChannel(OutputStream) 返回的是适配器模式包装的 WritableByteChannel，无法使用 FileChannel 的额外能力。在大文件下载场景中，直接使用 FileChannel 性能更好且功能更完整。

4. 调用链路

// 一行上传
OssFactory.instance().upload(new File("/tmp/report.pdf"));

完整异步链路:
1. upload(File) → upload(defaultBucket, file)
2. File → RandomAccessFile → FileChannel.map() → ByteBuffer
3. AsyncRequestBody.fromByteBuffer(byteBuffer)
4. S3TransferManager.upload(UploadRequest)
   ├─ 小文件(<分段阈值) → putObject (单次 PUT)
   └─ 大文件(≥分段阈值) → createMultipartUpload
                           → uploadPart × N (并行)
                           → completeMultipartUpload
5. CompletableFuture<CompletedUpload>
6. .handleAsync((response, error) → {
     if error → S3StorageException
     return PutObjectResult(eTag, versionId, ...)
   }, asyncExecutor)
7. 返回 PutObjectResult (含文件 URL、ETag 等信息)
💡 设计洞察：模块使用 AWS SDK 的 S3TransferManager 而不是直接操作 S3AsyncClient，是因为 TransferManager 封装了分段上传的复杂逻辑——自动判断文件大小、超过阈值时智能分段、并行上传段、失败重试、断点续传。开发者只需调用 upload()，不需要关心文件是 1KB 还是 1GB。这是典型的「让专业的人做专业的事」——AWS SDK 负责传输细节，OSS 模块负责项目化适配。
