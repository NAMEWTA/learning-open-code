第 4 课 · 异步体系：虚拟线程、响应式流与结果包装 异步

本节目标（对应使命 Success #6）：学完你能解释 CompletableFuture + handleAsync 的异步上传模式，说清 OutputStreamDownloadSubscriber 如何作为 Consumer<ByteBuffer> 消费字节流，理解 HandleAsyncResult 的 Either 式设计。

前面三节课我们讲了多客户端工厂、七子包架构和 S3 核心 API 目录。但 RuoYi 的 OSS 模块不止是「调 AWS SDK」——它在异步处理上做了精心设计，形成了一套完整的异步体系：配置层（Executor）→ 执行层（handleAsync）→ 结果层（HandleAsyncResult）→ 消费层（Subscriber）。本节课把这条链路完整拆开。

一、为什么需要异步？

OSS 上传/下载本质是网络 I/O，属于I/O 密集型操作。如果用同步阻塞方式，每个上传都会阻塞一个线程，100 并发上传就需要 100 个线程等待，浪费资源。

AWS SDK S3 v2 的选择是全面异步——核心 API 都返回 CompletableFuture：

// SDK 层面已经是异步的
S3AsyncClient        // 异步客户端，基于 Netty NIO
S3TransferManager    // 传输管理器，内部用 CompletableFuture 编排
  .upload(...)
  .completionFuture()  // 返回 CompletableFuture<CompletedUpload>

RuoYi 在 SDK 的异步之上再加了一层——自定义 asyncExecutor 控制回调线程：SDK 的 I/O 线程是 Netty 的 event loop，不应该在上面跑业务回调（如构建 PutObjectResult）。RuoYi 用一个独立的 Executor 承接 handleAsync 的回调，解耦 I/O 线程与业务线程。

核心思想：SDK 负责「怎么传数据」（Netty I/O），RuoYi 负责「传完数据后干什么」（回调处理）。两层异步各司其职。

二、OssAsyncExecutorConfig — 执行器的两种模式

这是一个用 @Builder 标注的 Java record，定义如何创建异步调度线程池：

// ruoyi-common-oss: config/OssAsyncExecutorConfig.java
@Builder
public record OssAsyncExecutorConfig(
    boolean enabledVirtualThread,  // 是否启用虚拟线程
    int corePoolSize               // 核心线程数（虚拟线程模式下失效）
) implements Config<...>, Serializable {

    public static final int DEFAULT_CORE_POOL_SIZE =
        Runtime.getRuntime().availableProcessors();

    public static final OssAsyncExecutorConfig DEFAULT = OssAsyncExecutorConfig.builder()
        .enabledVirtualThread(true)
        .corePoolSize(DEFAULT_CORE_POOL_SIZE)
        .build();
}

两种执行模式

模式条件创建代码特点
模式 A：虚拟线程 enabledVirtualThread = true Executors.newVirtualThreadPerTaskExecutor() Java 21+ 虚拟线程，轻量级（几 KB 栈内存），无界虚拟并发，每个任务一个虚拟线程，适合 I/O 密集型
模式 B：线程池 enabledVirtualThread = false Executors.newScheduledThreadPool(corePoolSize) 传统有界线程池，corePoolSize 默认等于 CPU 核数，适合计算密集型或低版本 Java

在哪里被使用

DefaultOssClientImpl.doInitialize() 在初始化 S3 客户端之后，读取配置并创建 asyncExecutor：

// DefaultOssClientImpl.doInitialize() 末尾
OssAsyncExecutorConfig asyncExecutorConfig = config.asyncExecutorConfig();
if (asyncExecutorConfig.enabledVirtualThread()) {
    this.asyncExecutor = Executors.newVirtualThreadPerTaskExecutor();
} else {
    this.asyncExecutor = Executors.newScheduledThreadPool(asyncExecutorConfig.corePoolSize());
}

这个 asyncExecutor 随后被传入所有 handleAsync 调用，确保上传/下载完成后的回调不占用 Netty I/O 线程。

默认值设计：DEFAULT 配置 enabledVirtualThread=true。这意味着如果不显式配置，RuoYi 默认使用虚拟线程——拥抱 Java 21 新特性，用轻量并发替代传统线程池。

三、doCustomUpload — 异步上传的核心

doCustomUpload 是整个上传链路的「最后一公里」——从 AsyncRequestBody 到 PutObjectResult。它在 AbstractOssClientImpl 中有 4 个重载，以不同粒度控制上传行为：

// 重载 1：完整版 — 自定义 handler + transferListeners（返回泛型 T）
public <T> T doCustomUpload(
    AsyncRequestBody body,
    Consumer<PutObjectRequest.Builder> putObjectRequestBuilderConsumer,
    Collection<TransferListener> transferListeners,
    BiFunction<CompletedUpload, Throwable, T> handleAsyncAction
)

// 重载 2：无 transferListeners 版 — 委托给重载 1
public <T> T doCustomUpload(
    AsyncRequestBody body,
    Consumer<PutObjectRequest.Builder> putObjectRequestBuilderConsumer,
    BiFunction<CompletedUpload, Throwable, T> handleAsyncAction
)

// 重载 3：返回 HandleAsyncResult — 带 transferListeners
public HandleAsyncResult<PutObjectResponse> doCustomUpload(
    AsyncRequestBody body,
    Consumer<PutObjectRequest.Builder> putObjectRequestBuilderConsumer,
    Collection<TransferListener> transferListeners
)

// 重载 4：返回 HandleAsyncResult — 无 transferListeners
public HandleAsyncResult<PutObjectResponse> doCustomUpload(
    AsyncRequestBody body,
    Consumer<PutObjectRequest.Builder> putObjectRequestBuilderConsumer
)

核心执行流程（重载 1 的代码）

public <T> T doCustomUpload(AsyncRequestBody body, ...) {
    try {
        return s3TransferManager.upload(uploadRequestBuilder -> {
                uploadRequestBuilder.requestBody(body)
                    .putObjectRequest(putObjectRequestBuilderConsumer)
                    .transferListeners(transferListeners);
            })
            .completionFuture()                // → CompletableFuture<CompletedUpload>
            .handleAsync(handleAsyncAction)     // → CompletableFuture<T> (用 asyncExecutor)
            .join();                            // 阻塞等待，返回 T
    } catch (Exception e) {
        throw toStorageException(e);
    }
}
上传数据流： AsyncRequestBody → S3TransferManager.upload(builder) → UploadRequest (body + PutObjectRequest + transferListeners) → completionFuture() ← CompletableFuture<CompletedUpload> → handleAsync(BiFunction, asyncExecutor) ├─ 成功: (completedUpload, null) → 构建 PutObjectResult └─ 失败: (null, throwable) → toStorageException(throwable) → .join() 阻塞当前线程直到完成

为什么用 handleAsync 而不是 thenApply？

handleAsync(BiFunction<T, Throwable, R>, Executor) 的关键优势：一个回调同时处理成功和失败。无论 completionFuture 正常完成还是异常完成，BiFunction 都会被调用。而 thenApply 只在成功时触发，异常需要单独 exceptionally 处理。

1handleAsync 的参数语义
java.util.concurrent.CompletableFuture
第一个参数 completedUpload：上传成功时为 CompletedUpload 对象，失败时为 null。
第二个参数 throwable：上传失败时为异常对象，成功时为 null。
两者必有一个为非 null，不可能同时非 null。
2.join() 的作用
join() 阻塞当前线程直到 CompletableFuture 完成。它和 get() 的区别是抛出 CompletionException（unchecked）而不是 ExecutionException（checked）。外层 toStorageException 会解包这些异步异常的包装层，取出根因。

四、HandleAsyncResult<T> — Either 模式的结果包装

重载 3 和重载 4 不抛异常，而是返回 HandleAsyncResult<T>——让调用方自行决定如何处理失败，而非强制 try-catch。

// ruoyi-common-oss: model/HandleAsyncResult.java
public record HandleAsyncResult<T>(
    T result,        // 成功时的结果（失败时为 null）
    Throwable error  // 失败时的异常（成功时为 null）
) {
    public Optional<T> getResult()           // 获取结果（Optional 包装）
    public Optional<Throwable> getError()     // 获取异常（Optional 包装）
    public boolean isSuccess()                // error.isEmpty()
    public boolean isFailure()                // error.isPresent()

    // 静态工厂方法
    public static <T> HandleAsyncResult<T> of(T result, Throwable error)
    public static <T> HandleAsyncResult<T> success(T result)
    public static <T> HandleAsyncResult<T> failure(Throwable error)
}

典型使用场景

在 bucketUpload（私有方法）中，调用方获取 HandleAsyncResult 后自行判断：

// AbstractOssClientImpl.bucketUpload() 片段
HandleAsyncResult<PutObjectResponse> result = doCustomUpload(body, builder -> {
    builder.bucket(bucket).key(key)
        .contentMD5(md5Digest).contentType(contentType)
        .contentLength(contentLength).metadata(metadata);
}, transferListeners);

if (result.isFailure()) {
    throw toStorageException(result.error());  // 调用方决定抛异常
}
Optional<PutObjectResponse> opt = result.getResult();
if (opt.isEmpty()) {
    throw S3StorageException.form("response is empty.");
}
PutObjectResponse response = opt.get();

与其他方案的对比

方案优点缺点
CompletableFuture 直接抛异常 标准 API，链式调用 调用方必须 try-catch，异常即终止链
Optional<T> 提醒调用方处理空值 无法携带异常信息，只能区分「有值/无值」
函数式 Either (如 Vavr) 完备的 Left(Error)/Right(Value) 语义 需引入第三方库
HandleAsyncResult<T> 零依赖、语义清晰、可选抛异常 仅适用于异步场景，非通用 Either
设计意图：HandleAsyncResult 不是通用 Either，而是专为「异步上传结果」设计的轻量包装。它不需要 map/flatMap 等函数式组合子——调用方通常就是 isFailure() ? throw : getResult() 两种分支。

五、OutputStreamDownloadSubscriber — 响应式流消费者

下载侧的异步设计走的是另一条路——响应式流（Reactive Streams）。S3TransferManager 下载返回一个 Publisher<ByteBuffer>，每收到一块数据就推送给订阅者。RuoYi 实现了 Consumer<ByteBuffer> + AutoCloseable 来消费这个流。

// ruoyi-common-oss: io/OutputStreamDownloadSubscriber.java
public class OutputStreamDownloadSubscriber
    implements Consumer<ByteBuffer>, AutoCloseable {

    private final WritableByteChannel channel;
    private final boolean allowAutoClose;

    // Consumer<ByteBuffer> 的核心方法：每收到一个 ByteBuffer 就被调用一次
    @Override
    public void accept(ByteBuffer byteBuffer) {
        try {
            while (byteBuffer.hasRemaining()) {
                channel.write(byteBuffer);  // 写入 WritableByteChannel
            }
        } catch (Exception e) {
            throw S3StorageException.form(e);
        }
    }
}

FileOutputStream 的优化路径

构造函数中有一个巧妙的判断：

private OutputStreamDownloadSubscriber(OutputStream out, boolean allowAutoClose) {
    this.allowAutoClose = allowAutoClose;
    if (out instanceof FileOutputStream outputStream) {
        // 文件输出流 → 直接拿 FileChannel（零拷贝潜力）
        channel = outputStream.getChannel();
    } else {
        // 普通输出流 → Channels.newChannel(out) 包装
        channel = Channels.newChannel(out);
    }
}

如果是 FileOutputStream，直接用 getChannel() 获取底层 FileChannel——避免了 Channels.newChannel() 的额外包装层。FileChannel 本身支持更高效的 I/O 操作。

allowAutoClose 标志

默认 false——订阅器不负责关闭底层流。调用方（如 bucketDownload(bucket, key, Path)）用 try-with-resources 管理 OutputStream 的生命周期，订阅器只负责写入。

四个静态工厂方法

方法入参allowAutoClose 默认
create(OutputStream out)输出流false
create(OutputStream out, boolean allowAutoClose)输出流 + 关闭标志由参数决定
create(WritableByteChannel channel)可写通道false
create(WritableByteChannel channel, boolean allowAutoClose)可写通道 + 关闭标志由参数决定

六、doCustomDownload — 异步下载的两条路径

下载侧提供了两种策略，分别对应不同的使用场景：

路径 1：响应式流（Publisher + Subscriber）

// AbstractOssClientImpl.bucketDownload(bucket, key, OutputStreamDownloadSubscriber)
ResponsePublisher<GetObjectResponse> publisher = doCustomDownload(
    builder -> builder.bucket(bucket).key(key),
    AsyncResponseTransformer.toPublisher(),  // 转为 Publisher
    null
);
GetObjectResult getObjectResult = buildGetObjectResult(key, publisher.response());
publisher.subscribe(downloadSubscriber).join();  // 订阅 + 等待完成
return getObjectResult;
响应式流下载数据流： S3TransferManager.download(DownloadRequest) → .completionFuture().join().result() → ResponsePublisher<GetObjectResponse> → publisher.subscribe(OutputStreamDownloadSubscriber) → 每个 ByteBuffer 到达 → accept(byteBuffer) → channel.write() → 所有数据消费完毕 → .join() 返回 → GetObjectResult

路径 2：阻塞式 InputStream

// AbstractOssClientImpl.bucketDownload(bucket, key, BiFunction)
try (ResponseInputStream<GetObjectResponse> responseInputStream = doCustomDownload(
        builder -> builder.bucket(bucket).key(key),
        AsyncResponseTransformer.toBlockingInputStream(),  // 阻塞流
        null
)) {
    GetObjectResponse response = responseInputStream.response();
    GetObjectResult getObjectResult = buildGetObjectResult(key, response);
    return downloadTransformer.apply(getObjectResult, responseInputStream);
}

路径 2 用 toBlockingInputStream() 把异步流「伪装」成传统的 InputStream，方便调用方用熟悉的阻塞 I/O 方式处理（如复制到 OutputStream、解析文件内容等）。

策略适用场景优点缺点
响应式流 (Publisher + Subscriber)大文件下载、流式处理、可控背压内存友好、可暂停/恢复代码稍复杂
阻塞式 InputStream小文件、需在内存中解析内容代码简单、调试友好阻塞线程、大数据量可能 OOM

七、Options — 上传参数的流畅构建器

Options 不是核心异步组件，但它是 doCustomUpload 输入参数的重要来源——负责把「文件元信息」和「传输监听器」收集起来，最终注入 PutObjectRequest。

// ruoyi-common-oss: model/Options.java
@Data
@Accessors(chain = true)              // 支持链式调用
@NoArgsConstructor(access = AccessLevel.PRIVATE)  // 构造函数私有
public class Options {

    private Long length;                              // 文件长度
    private String md5Digest;                         // MD5 摘要
    private String contentType;                       // 内容类型
    private Map<String, String> metadata;             // 自定义元数据
    private Collection<TransferListener> transferListeners;  // 传输监听器

    public Options addMetadataItem(String key, String value) {
        if (this.metadata == null) { this.metadata = new HashMap<>(); }
        this.metadata.put(key, value);
        return this;  // 返回 this，支持链式
    }

    public Options addTransferListener(TransferListener transferListener) {
        if (this.transferListeners == null) { this.transferListeners = new ArrayList<>(); }
        this.transferListeners.add(transferListener);
        return this;
    }

    public static Options builder() { return create(); }
    public static Options create() { return new Options(); }
}

Options 在上传链路中的角色

在私有方法 bucketUpload(String bucket, String key, AsyncRequestBody body, Options options) 中，Options 的字段被逐一提取并填充到 PutObjectRequest.Builder：

// 内容长度：优先用 body 的长度，其次用 options 设置的
Long contentLength = body.contentLength().orElse(options.getLength());

// 内容类型：优先用 options 设置的，其次用 body 自带的
String contentType = StringUtils.isBlank(options.getContentType())
    ? body.contentType() : options.getContentType();

// MD5、metadata、transferListeners 直接从 options 取
String md5Digest = options.getMd5Digest();
Map<String, String> metadata = options.getMetadata();
Collection<TransferListener> transferListeners = options.getTransferListeners();
1addMetadataItem / addTransferListener 的懒初始化
两个 add 方法在第一次调用时才 new HashMap/ArrayList——避免为不需要元数据/监听器的上传创建空集合。这是典型的「按需分配」模式。
2@Accessors(chain = true) 的流畅 API
Options.create().setLength(1024L).setContentType("image/png").addMetadataItem("author", "ruoyi")——Lombok 的 @Accessors(chain=true) 让 setter 返回 this，支持链式调用。

八、异步体系全景图

上传链路 Options (元数据 + 监听器) → bucketUpload(bucket, key, body, options) → doCustomUpload(body, builderConsumer, transferListeners) → S3TransferManager.upload(...) → .completionFuture() ← CompletableFuture<CompletedUpload> → .handleAsync((result, err) → { if (err != null) return HandleAsyncResult.failure(err) return HandleAsyncResult.success(result.response()) }, asyncExecutor) ← 用虚拟线程/线程池回调 → .join() ← 阻塞等结果 → HandleAsyncResult<PutObjectResponse> → isFailure() ? throw : build PutObjectResult 下载链路（响应式流） bucketDownload(bucket, key, OutputStreamDownloadSubscriber) → doCustomDownload(builderConsumer, toPublisher(), null) → S3TransferManager.download(downloadRequest) → .completionFuture().join().result() → ResponsePublisher<GetObjectResponse> → publisher.subscribe(OutputStreamDownloadSubscriber) → accept(ByteBuffer) → while(hasRemaining) channel.write() → .join() → GetObjectResult

Q1. handleAsync 相比 thenApply + exceptionally 的核心优势是什么？

A. 性能更好 B. 一个回调同时处理成功和失败，且可以指定 Executor 避免占用 Netty I/O 线程 C. 支持链式调用 D. 不需要 .join() 阻塞
看答案B。handleAsync(BiFunction, Executor) 让成功和失败走同一个回调，且第二个参数指定执行回调的线程池，解耦 SDK I/O 线程与业务处理线程。

Q2. OutputStreamDownloadSubscriber 实现了哪两个接口？为什么 allowAutoClose 默认是 false？

A. 实现了 Runnable 和 Closeable；默认 false 是为了性能 B. 实现了 Consumer<ByteBuffer> 和 AutoCloseable；默认 false 表示调用方自己管理 OutputStream 生命周期，订阅器只负责写入 C. 没实现接口，是独立类；默认 false 因为不需要关闭 D. 实现了 Subscriber 和 Closeable；默认 false 避免死锁
看答案B。它实现 Consumer<ByteBuffer> 来消费响应式流推送的每个字节块，实现 AutoCloseable 支持 try-with-resources。allowAutoClose=false 是职责分离——调用方（如 bucketDownload(bucket, key, Path)）通过 try-with-resources 管理流的生命周期。

Q3. HandleAsyncResult<T> 的 isSuccess() 方法是怎么判断成功与否的？

A. 检查 result 是否为 null B. 检查 error 的 Optional 是否为空——isEmpty() 即成功 C. 检查 result 和 error 是否都为 null D. 抛异常判断
看答案B。isSuccess() 的实现是 return getError().isEmpty()——只要 error 是空的 Optional 就认为成功。这和 Either 模式中「Right 为成功」的语义一致。
