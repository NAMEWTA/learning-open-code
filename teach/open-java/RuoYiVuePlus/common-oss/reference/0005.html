toStorageException 解包流程

原始异常 (CompletionException / ExecutionException / ...) → unwrapAsyncException(e) → while (cause instanceof CompletionException || ExecutionException) → cause = cause.getCause() ← 逐层剥开 → 剥开后检查： ├─ 已是 S3StorageException → 直接返回 └─ 其他异常 → new S3StorageException(cause)

记忆锚点

  • 上传 12 步：Redis → JSON → OssProperties → OssClientConfig → DCL → doInitialize → buildPathKey → 类型转换 → 私有汇聚 → doCustomUpload → S3TransferManager → PutObjectResult。
  • 两种 URL 风格：path-style = host/bucket（MinIO），site-style = bucket.host（阿里云/腾讯/七牛/华为）。
  • 切换厂商 = 改 4 个字段：endpoint、accessKey、secretKey、bucketName。Java 代码不变，因为 S3 协议是通用标准。
  • toStorageException 解包：while 循环剥 CompletionException/ExecutionException，取根因，再包装为 S3StorageException。
  • 预签名 URL 用 domainUrl：S3Presigner 用 domainUrl 作为端点，保证生成的 URL 对用户可访问（endpointUrl 可能是内网地址）。
  • 异常统一：所有 OSS 操作异常最终都是 S3StorageException（RuntimeException），业务层只需 catch 这一个类型。
  • 双厂商并存：在 sys_oss_config 表建两条记录，configKey 分别为 "oss_aliyun" 和 "oss_minio"，OssFactory.instance(configKey) 按需获取。

→ 相关术语见 GLOSSARY.md：S3StorageException、BucketUrlUtil、Path-Style、Site-Style、预签名 URL、PutObjectResult、GetObjectResult、toStorageException、解包异常
