一句话定位

异步体系 = 配置层（OssAsyncExecutorConfig）→ 执行层（doCustomUpload + handleAsync）→ 结果层（HandleAsyncResult）→ 消费层（OutputStreamDownloadSubscriber），四层协作实现非阻塞 OSS 上传下载。

上传异步流水线

Options (length, md5Digest, contentType, metadata, transferListeners) │ └─► bucketUpload(bucket, key, body, options) │ └─► doCustomUpload(body, builderConsumer, transferListeners) │ └─► S3TransferManager.upload(uploadRequestBuilder) │ .requestBody(body) │ .putObjectRequest(builderConsumer) │ .transferListeners(transferListeners) │ └─► .completionFuture() ← CompletableFuture<CompletedUpload> │ └─► .handleAsync( (completedUpload, throwable) → { if (err != null) return HandleAsyncResult.failure(err) return HandleAsyncResult.success(response) }, asyncExecutor ← 虚拟线程或线程池 ) │ └─► .join() ← 阻塞等待 │ ├─ isFailure() → throw toStorageException └─ isSuccess() → build PutObjectResult

下载两条路径

路径下载方法核心 API使用场景
响应式流 bucketDownload(bucket, key, OutputStreamDownloadSubscriber) AsyncResponseTransformer.toPublisher() → publisher.subscribe(subscriber) 大文件、流式处理、File/Path 输出
阻塞式流 bucketDownload(bucket, key, BiFunction) AsyncResponseTransformer.toBlockingInputStream() → try-with-resources 小文件、需内存解析、自定义转换

记忆锚点

  • handleAsync 而不是 thenApply：一个回调处理成功+失败，且用独立 Executor 避免占用 Netty I/O 线程。
  • 虚拟线程默认开启：DEFAULT.enabledVirtualThread = true，拥抱 Java 21，I/O 密集场景用轻量虚拟线程。
  • HandleAsyncResult 是轻量 Either：零依赖，不用引入 Vavr，只做成功/失败两路分发。
  • OutputStreamDownloadSubscriber 只管写不管关：allowAutoClose=false 默认，调用方用 try-with-resources 管理 OutputStream 生命周期。
  • FileOutputStream 优化路径：如果是 FileOutputStream 直接拿 FileChannel，跳过 Channels.newChannel() 包装层。
  • Options 链式构建：@Accessors(chain=true) + 懒初始化 Map/List，流畅收集上传参数。
  • 下载两种策略：响应式流（Publisher + Subscriber）适合大文件/流式，阻塞 InputStream 适合小文件/内存解析。
