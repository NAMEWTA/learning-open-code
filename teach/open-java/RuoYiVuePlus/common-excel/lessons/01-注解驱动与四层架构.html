注解驱动与四层架构

🎯 本节目标：学完你能用一句话说清 ruoyi-common-excel 与 Fesod Sheet 的关系，画出模块的「注解层 - 转换器层 - 处理器层 - 工具层」四层架构，并逐一解释 6 个自定义注解各自挂在哪一层、最终由哪个处理器/转换器消费。这是整个模块的地图——先有全景，再走小路。

一、先看全景：Fesod Sheet 与本模块的关系

在 RuoYi-Vue-Plus 的后台管理系统中，几乎没有哪个业务模块不需要导入导出。导出用户列表、导入部门数据、导出操作日志——这些需求高度重复。如果把每个 Controller 里都写一遍 POI 的 Workbook、Sheet、Cell 操作代码，会是灾难性的。

RuoYi-Vue-Plus 的解决方式是：

Fesod Sheet（fesod 是 Apache 的 Java 命名规范变形）是对 Apache POI 的高层封装，提供注解驱动的 Excel 读写、类型转换器扩展点（Converter）、写处理器链（WriteHandler）、事件监听（AnalysisEventListener）等基础设施。

ruoyi-common-excel 不是框架本身，而是对 Fesod Sheet 的项目级定制——通过实现 Fesod Sheet 的 Converter / WriteHandler / ReadListener 接口，注入项目字典服务、接入 Jakarta Validation、挂载自定义写处理器，把 Fesod Sheet 的通用能力"项目化"。

一句话：Fesod Sheet 说"我提供插槽"，本模块说"我在插槽里装上 RuoYi 的东西"。

这意味着本模块的 21 个 Java 文件中，几乎没有直接 import org.apache.poi 的——所有 POI 操作都通过 Fesod Sheet 间接完成。唯一例外是涉及合并区域（CellRangeAddress）、下拉校验（DataValidation）、名称管理器（Name）等 Fesod Sheet 未完全封装的部分，才会直接操作 POI API。

二、四层架构：21 个文件的归属

打开模块源码，你会看到 4 个包。这不是随意组织的——它对应了从"声明意图"到"执行动作"到"对外暴露"的完整分层：

层级包路径文件数核心职责代表文件
注解层 annotation/ 6 声明式标注字段行为——"这个字段要合并""那个字段走字典转换" CellMerge.java, ExcelDictFormat.java
转换器层 convert/ 3 类型双向转换——Java 对象字段 ↔ Excel 单元格字符串 ExcelDictConvert.java, ExcelEnumConvert.java, ExcelBigNumberConvert.java
处理器层 core/ + handler/ 9+1 生命周期钩子——合并单元格、生成下拉、标红必填、监听导入事件 ExcelDownHandler.java, CellMergeStrategy.java, DataWriteHandler.java, DefaultExcelListener.java
工具层 utils/ 2 对外 DSL 入口——业务代码唯一打交道的 API ExcelBuilder.java, ExcelWriterWrapper.java

  ├─ 注册 ExcelDownHandler                         ← 【处理器层】下拉生成

💡 设计洞察：注解层只负责"声明元数据"，自己不执行任何逻辑。转换器层和处理器层各自读取自己关心的注解，独立消费。这种"声明与执行分离"的设计，使得新增一个注解不需要修改已有处理器，新增一个处理器不需要修改注解——完全符合开闭原则。

三、注解层详解：6 个注解，3 种消费模式

6 个自定义注解是模块的"元数据接口"。每个注解定义在 annotation/ 包下，标注在实体类的字段上，声明该字段在 Excel 读写时的特殊行为。关键是理解：谁声明它、谁读取它、谁执行它。

3.1 @CellMerge —— 列值合并

文件：annotation/CellMerge.java

当一个字段在连续行中值相同时，导出 Excel 时自动合并这些行的单元格。最典型的场景是"部门名称"列——10 个员工在同一部门，部门名只显示一次，占据合并后的 10 行高度。

// 注解定义（精简）
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface CellMerge {
    int index() default -1;        // 列索引，-1 时自动推断
    String[] mergeBy() default {};  // 合并依赖字段——依赖字段也相同时才合并
}
属性含义示例场景
index 指定合并列索引，-1 时取字段声明顺序 第 3 列合并：@CellMerge(index = 2)
mergeBy 声明依赖字段，依赖字段值也相同时才合并 合并"部门名称"时必须同"部门编号"：@CellMerge(mergeBy = "deptId")

mergeBy 是最容易忽略但最重要的属性。假设两个部门都叫"技术部"但编号不同——如果只看部门名称就合并，会错误地把两个不同部门的行合并到一起。声明 mergeBy = "deptId" 后，只有部门编号也相同的相邻行才会被合并。

谁声明：业务开发在导出 VO 的字段上标注
谁解析：CellMergeHandler.java（静态工厂，扫描注解生成 List<CellRangeAddress>）
谁执行：CellMergeStrategy.java（继承 AbstractMergeStrategy，在 afterSheetCreate 中写入合并区域）

3.2 @ExcelDictFormat —— 字典转换

文件：annotation/ExcelDictFormat.java

这是最常用的注解——将数据库中存储的编码（如 "0"、"1"）与 Excel 中显示的标签（如"男"、"女"）互相转换。支持两种数据来源：

// 注解定义（精简）
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface ExcelDictFormat {
    String dictType() default "";             // 字典类型，走项目 DictService 查库
    String readConverterExp() default "";     // 内联表达式，如 "0=男,1=女,2=未知"
    String separator() default ",";           // 分隔符
}
数据来源属性适用场景示例
字典服务（查库） dictType 数据量大、频繁变更、跨模块复用 @ExcelDictFormat(dictType = "sys_user_sex")
内联表达式（写死） readConverterExp 数据量小、固定不变、无需查库 @ExcelDictFormat(readConverterExp = "0=启用,1=停用")

两种方式可以同时指定。优先使用哪个？如果 dictType 不为空，走字典服务；否则解析 readConverterExp。这个优先级判断逻辑在 ExcelDictConvert.java 中。

谁声明：业务开发在实体字段上标注
谁解析（转换）：ExcelDictConvert.java（导出：编码→标签；导入：标签→编码）
谁解析（下拉）：ExcelDownHandler.java（导出模板时根据字典选项生成下拉列表）
谁执行：Fesod Sheet 在读写单元格时自动调用 Converter

3.3 @ExcelEnumFormat —— 枚举转换

文件：annotation/ExcelEnumFormat.java

与字典转换类似，但数据源是 Java 枚举类。适合"选项固定、不会频繁变更"的场景。

// 注解定义（精简）
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface ExcelEnumFormat {
    Class> enumClass();  // 枚举类
    String codeField() default "code";      // 枚举中表示"编码"的属性名
    String textField() default "text";      // 枚举中表示"文本"的属性名
}

// 使用示例
public enum UserStatus {
    OK("0", "正常"),
    DISABLE("1", "停用"),
    DELETED("2", "删除");
    private String code;
    private String text;
    // ...
}

// 实体字段标注
@ExcelEnumFormat(enumClass = UserStatus.class)
private String status;

导出时，status = "0" 会被转换为 Excel 中的 "正常"。导入时，用户填的 "正常" 会被反向映射回 "0"。

关键设计：双 Map 缓存。因为枚举的 code 和 text 通过反射读取，每次都反射会很慢。ExcelEnumConvert 内部使用 ConcurrentHashMap 维护两个缓存：

  • ENUM_MAP_CACHE：enumClass → Map<code, text>，导出时用 code 查 text
  • ENUM_REVERSE_MAP_CACHE：enumClass → Map<text, code>，导入时用 text 查 code
谁声明：业务开发在实体字段上标注
谁解析（转换）：ExcelEnumConvert.java（双 Map 缓存，避免反射开销）
谁解析（下拉）：ExcelDownHandler.java（导出模板时从枚举中提取选项生成下拉）
谁执行：Fesod Sheet 在读写单元格时自动调用 Converter

3.4 @ExcelDynamicOptions —— 动态下拉

文件：annotation/ExcelDynamicOptions.java

前两种注解的数据来源是"静态的"——字典表或枚举类中的值在编码阶段就已经确定。但如果下拉选项需要从数据库中实时查询呢？比如"所属部门"的下拉，每次导出模板时都应该查询最新的部门列表。这就是 @ExcelDynamicOptions 的用武之地。

// 注解定义
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface ExcelDynamicOptions {
    Class providerClass();
}

// ExcelOptionsProvider 接口
public interface ExcelOptionsProvider {
    Set<String> getOptions();
}

// 使用示例
@Component
public class DeptOptionsProvider implements ExcelOptionsProvider {
    @Autowired
    private ISysDeptService deptService;

    @Override
    public Set<String> getOptions() {
        return deptService.list().stream()
            .map(SysDept::getDeptName)
            .collect(Collectors.toSet());
    }
}

// 实体字段标注
@ExcelDynamicOptions(providerClass = DeptOptionsProvider.class)
private String deptName;

执行流程：ExcelDownHandler 发现 @ExcelDynamicOptions 后，通过 Spring 容器获取 providerClass 的实例，调用 getOptions() 拿到运行时选项集合，然后生成下拉。

谁声明：业务开发标注字段 + 实现 ExcelOptionsProvider 接口并注册为 Spring Bean
谁解析：ExcelDownHandler.java（通过 Spring 容器获取 provider 实例）
谁执行：ExcelDownHandler 在 afterSheetCreate 中生成下拉

3.5 @ExcelRequired —— 必填标注

文件：annotation/ExcelRequired.java

纯粹的视觉标注——将表头字体标红（默认 IndexedColors.RED），提示用户该列必填。不参与任何校验逻辑。

// 注解定义
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface ExcelRequired {
}

// 使用示例
@ExcelRequired
private String userName;  // 导出模板时"用户名"表头标红
谁声明：业务开发在必填字段上标注
谁解析 + 谁执行：DataWriteHandler.java（在 afterCellDispose 中检测表头行 + 注解，设置字体颜色）

3.6 @ExcelNotation —— 列头批注

文件：annotation/ExcelNotation.java

在导出模板时，给表头单元格添加 Excel 批注（Comment），内容是注解的 value 字符串。典型用途是告诉用户"这一列应该填什么格式"。

// 注解定义
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface ExcelNotation {
    String value();  // 批注内容
}

// 使用示例
@ExcelNotation("请输入手机号，格式如 13800138000")
private String phonenumber;

限制：仅适用于单表头场景（多级表头不做批注处理）。同样由 DataWriteHandler 消费。

谁声明：业务开发在字段上标注
谁解析 + 谁执行：DataWriteHandler.java（在 afterCellDispose 中检测表头行 + @ExcelNotation，创建 POI Comment 并设置到单元格）

💡 设计洞察：为什么有些注解有 Converter 消费者，有些只有 Handler 消费者？

注解分为两大类：数据转换类（@ExcelDictFormat、@ExcelEnumFormat）影响的是"单元格里的值是什么"，需要介入 Fesod Sheet 的类型转换流程，所以通过 Converter 实现；展现格式类（@CellMerge、@ExcelRequired、@ExcelNotation）影响的是"单元格长什么样"，需要介入单元格写入生命周期，所以通过 WriteHandler 实现。

@ExcelDynamicOptions 是个特例——它既不转换值，也不改变展现，而是为模板添加数据验证约束，所以只被 ExcelDownHandler 消费。@ExcelDictFormat 和 @ExcelEnumFormat 之所以同时被 Converter 和 ExcelDownHandler 消费，是因为它们身兼二职：既要在读写时做值转换（Converter），又要在模板导出时生成下拉选项（Handler）。

五、回到全景：四层架构的价值

现在你看到了完整的四层架构和 6 个注解的消费者关系。这个架构的核心价值在于：

  1. 注解层让业务开发用声明式方式描述需求——"这个字段要合并""那个字段走字典"，不需要写任何 POI 代码。
  2. 转换器层封装了"Java 值 ↔ Excel 文本"的映射逻辑，每种映射策略独立实现、互不干扰。
  3. 处理器层封装了 Excel 生命周期钩子中的"副作用"——合并、下拉、标红、批注。每个处理器只关心自己的注解，独立运作。
  4. 工具层把所有复杂性封装在一个链式 DSL 中——ExcelBuilder.of(data, clazz).toResponse(response) 一行代码完成全部导出。

后续两课将分别追踪导出链路和导入链路的完整执行流程，你会看到这四层如何在真实调用链中协作。
