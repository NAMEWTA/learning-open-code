导出链路详解

🎯 本节目标：学完你能完整追踪导出链路——从 ExcelBuilder.of(data, clazz).toResponse(response) 到浏览器下载 Excel 文件，说出每一环经过了哪些处理器和转换器。能解释 Converter 体系中 BigNumberConvert、DictConvert、EnumConvert 各自的介入时机与策略。能讲清 CellMergeHandler/CellMergeStrategy 的分工和 DataWriteHandler 的双重角色。

一、先看全景：导出链路的完整调用链

导出链路的入口只有一行代码：

ExcelBuilder.of(dataList, UserExportVO.class)
    .sheetName("用户数据")
    .merge()          // 可选：开启列值合并
    .toResponse(response);

这一行代码背后，经历了五个阶段。让我们先看全景，再逐阶段深入：

阶段①  Builder 构造
  ExcelBuilder.of(data, clazz)
    │
  ExcelBuilder 构造函数：
    ├─ 保存 data（列表数据）
    ├─ 保存 headType（实体类的 Class）
    ├─ 设置 sheetName / sheetNo
    ├─ 设置 includeFields / excludeFields
    └─ 设置 merge（是否开启列合并）
    │
阶段②  创建 WriterBuilder（核心注册）
  .toResponse(response) 内部调用 createWriterBuilder()
    │
  createWriterBuilder()：
    ├─ 【自动注册 Converter】
    │   ├─ new ExcelBigNumberConvert()    — 大数字精度保护
    │   ├─ new ExcelDictConvert()         — 字典双向转换
    │   └─ new ExcelEnumConvert()         — 枚举双向转换
    │
    ├─ 【自动注册 WriteHandler（每格）】
    │   └─ new DataWriteHandler()         — @ExcelRequired + @ExcelNotation
    │
    ├─ 【自动注册 WriteHandler（每 Sheet）】
    │   └─ new ExcelDownHandler()         — 下拉数据校验
    │
    └─ 【条件注册 — 仅 merge 模式】
        └─ new CellMergeStrategy()        — 列值合并
    │
阶段③  WriteSheet（写入数据）
  builder.doWrite(data)
    │
  Fesod Sheet 内部循环每一行：
    ├─ 对每个字段调用 Converter.convertToExcelData()
    ├─ 写入单元格值
    └─ 触发 CellWriteHandler.afterCellDispose()
    │
阶段④  生命周期回调
  ├─ afterSheetCreate()
  │   ├─ CellMergeStrategy → 写入合并区域（提前占位）
  │   └─ ExcelDownHandler → 生成下拉校验
  │
  ├─ afterCellDispose()
  │   └─ DataWriteHandler → 标红表头 / 添加批注
  │
  └─ merge()（CellMergeStrategy）
      └─ 清理合并区域内非首行的重复值
    │
阶段⑤  写出到 Response
  response.setContentType("application/vnd.openxmlformats-officedocument.spreadsheetml.sheet")
  response.setHeader("Content-Disposition", "attachment; filename=xxx.xlsx")
  workbook.write(response.getOutputStream())

下面我们逐阶段拆解每个参与者的内部逻辑。

二、阶段② 深入：createWriterBuilder 做了什么

位置：utils/ExcelBuilder.java 的 createWriterBuilder() 私有方法。

这是整个导出链路中最关键的工厂方法。它把 ruoyi-common-excel 模块的所有能力（转换器、处理器）一次性注册到 Fesod Sheet 的 WriterBuilder 中。之后 Fesod Sheet 在写入每个单元格时，会自动调用这些注册的组件。

2.1 三个 Converter 的自动注册

Fesod Sheet 的 Converter 机制是：在写入单元格前，框架会遍历已注册的 Converter 列表，调用 supportJavaTypeKey() 判断该 Converter 是否支持当前字段类型，支持则调用 convertToExcelData() 完成转换。

这三个 Converter 的注册是无条件自动的——只要调用 of() 或 writer()，它们就会被挂载：

// ExcelBuilder.createWriterBuilder() 内部（伪代码）
ExcelWriterBuilder builder = FesodSheet.write(outputStream, headType)
    .registerConverter(new ExcelBigNumberConvert())   // ①
    .registerConverter(new ExcelDictConvert())         // ②
    .registerConverter(new ExcelEnumConvert());        // ③

① ExcelBigNumberConvert —— 大数字精度保护

文件：convert/ExcelBigNumberConvert.java

这是三个 Converter 中最特殊的一个——它不依赖任何自定义注解，而是解决一个通用问题：Excel 对超过 15 位的数字会自动转为科学计数法，导致精度丢失。订单号、身份证号等长数字字段尤其危险。

// ExcelBigNumberConvert 的核心策略
supportJavaTypeKey():
    return Long.class.equals(javaTypeKey)
        || BigDecimal.class.equals(javaTypeKey)
        || BigInteger.class.equals(javaTypeKey);

convertToExcelData(Long value):
    // 将 Long 值作为字符串写入 Excel，前面加制表符强制文本格式
    return "\t" + value.toString();  // 1234567890123456 → "\t1234567890123456"

convertToJavaData(CellData cellData):
    // 读取时以字符串形式获取，避免科学计数法
    return Long.valueOf(cellData.getStringValue());
为什么前面加 \t？这是 Excel 的一个约定——如果单元格内容以制表符开头，Excel 会将该单元格视为文本格式，从而避免数字精度丢失。如果你在 Excel 中打开文件，看到的 \t 是不可见字符，不影响显示效果。

② ExcelDictConvert —— 字典双向转换

文件：convert/ExcelDictConvert.java

这是最复杂的 Converter。它需要处理两个方向、两种数据源：

// 导出方向：编码 → 标签
convertToExcelData(Object value, ExcelContentProperty property):
    Field field = property.getField();
    ExcelDictFormat annotation = field.getAnnotation(ExcelDictFormat.class);
    if (annotation == null) return value;  // 无注解，原样返回

    if (StringUtils.isNotBlank(annotation.dictType())) {
        // 方式 A：走字典服务查库
        String label = DictUtils.getDictLabel(annotation.dictType(), value.toString());
        return label;
    } else {
        // 方式 B：解析内联表达式
        // readConverterExp = "0=男,1=女,2=未知"
        return convertByExp(annotation.readConverterExp(), value.toString(), annotation.separator());
    }

// 导入方向：标签 → 编码（反向查找）
convertToJavaData(CellData cellData, ExcelContentProperty property, ...):
    // 同样逻辑，但方向相反——从标签反查编码
💡 设计洞察：dictType 与 readConverterExp 的优先级

两者可以同时指定，但 dictType 优先。这个优先级是合理的：字典服务是项目级基础设施，能保证全局一致性（如果一个字典类型被 5 个模块使用，改字典表就全改了）。readConverterExp 是"兜底方案"——当你不方便建字典类型（比如临时导出一个报表）、或者选项极少且固定（如性别只有男/女）时使用内联表达式更轻量。

导入时的反向转换是最大的价值点。用户填"男"，系统自动存"0"——如果不做反向转换，业务代码需要自己写 if-else 把"男"转回"0"，那 Excel 导入的"自动化"就大打折扣了。

③ ExcelEnumConvert —— 枚举双向转换

文件：convert/ExcelEnumConvert.java

与 DictConvert 类似，但数据源是 Java 枚举类。核心设计是双 Map 缓存：

// ExcelEnumConvert 内部缓存结构
private static final Map> ENUM_MAP_CACHE = new ConcurrentHashMap<>();
// key: "com.xxx.UserStatus"  →  value: {"0": "正常", "1": "停用"}

private static final Map> ENUM_REVERSE_MAP_CACHE = new ConcurrentHashMap<>();
// key: "com.xxx.UserStatus"  →  value: {"正常": "0", "停用": "1"}

// 懒加载构建缓存
private Map<String, String> getCacheMap(Class<? extends Enum<?>> enumClass,
                                          ExcelEnumFormat annotation, boolean reverse) {
    String cacheKey = enumClass.getName();
    Map<String, Map<String, String>> cache = reverse ? ENUM_REVERSE_MAP_CACHE : ENUM_MAP_CACHE;

    return cache.computeIfAbsent(cacheKey, k -> {
        // 通过反射读取枚举的 codeField 和 textField 属性
        // 构建 code → text 或 text → code 的映射
        Enum<?>[] constants = enumClass.getEnumConstants();
        Map<String, String> map = new HashMap<>();
        for (Enum<?> constant : constants) {
            String code = getFieldValue(constant, annotation.codeField());
            String text = getFieldValue(constant, annotation.textField());
            map.put(reverse ? text : code, reverse ? code : text);
        }
        return map;
    });
}

computeIfAbsent 保证了线程安全且只初始化一次——每个枚举类在 JVM 生命周期内只会被反射解析一次，之后都走缓存命中。

💡 设计洞察：为什么用 ConcurrentHashMap 而不是 synchronized？

ConcurrentHashMap.computeIfAbsent 对同一个 key 的并发计算是原子且阻塞的——第一个线程进入 compute 逻辑，后续线程等待结果。这比 synchronized 锁整个方法更精细：不同枚举类的初始化可以并行，互不影响。

另外，导入时的校验逻辑值得关注：如果用户在 Excel 中填了一个不在枚举定义范围内的文本（比如把"正常"写成了"正常 "多一个空格），ENUM_REVERSE_MAP_CACHE.get(text) 返回 null，此时 Convert 会 直接抛出异常，由 DefaultExcelListener.onException() 捕获并记录为错误行。

三、阶段③+④ 深入：三个 WriteHandler 的执行顺序

导出链路中有三个 WriteHandler 参与工作：CellMergeStrategy、ExcelDownHandler、DataWriteHandler。它们各自实现不同的生命周期接口，在 Sheet 和 Cell 的不同阶段被回调。

3.1 CellMergeHandler + CellMergeStrategy —— 分析执行分离

文件：core/CellMergeHandler.java（静态工厂分析器）+ core/CellMergeStrategy.java（合并执行器）

这两个类的分工非常清晰：

类角色做什么何时做
CellMergeHandler 分析器（静态工厂） 扫描实体 @CellMerge 注解，按数据行逐一比对相邻行字段值，生成 List<CellRangeAddress> 在 CellMergeStrategy 构造函数中被调用
CellMergeStrategy 执行器（继承 AbstractMergeStrategy） 在 Sheet 创建后写入合并区域；在合并时清理非首行重复值 afterSheetCreate() + merge() 
// CellMergeHandler 核心逻辑（伪代码）
public static <T> List<CellRangeAddress> buildMergeList(List<T> data, Class<T> clazz, int headRowNum) {
    // 1. 扫描所有带 @CellMerge 的字段
    List<Field> mergeFields = getMergeFields(clazz);

    // 2. 逐行比对相邻两行的字段值
    List<CellRangeAddress> mergeList = new ArrayList<>();
    for (int rowIdx = headRowNum + 1; rowIdx < headRowNum + 1 + data.size(); rowIdx++) {
        T current = data.get(rowIdx - headRowNum - 1);
        T previous = rowIdx > headRowNum + 1 ? data.get(rowIdx - headRowNum - 2) : null;

        for (Field field : mergeFields) {
            Object currentVal = field.get(current);
            Object previousVal = previous == null ? null : field.get(previous);

            // 检查 mergeBy 依赖字段是否也相同
            if (shouldMerge(field, currentVal, previousVal, current, previous)) {
                // 扩展或新建合并区域
                mergeList.add(new CellRangeAddress(startRow, endRow, col, col));
            }
        }
    }
    return mergeList;
}
// CellMergeStrategy 核心逻辑（伪代码）
public class CellMergeStrategy extends AbstractMergeStrategy implements SheetWriteHandler {
    private List<CellRangeAddress> mergeList;

    public CellMergeStrategy(List<T> data, Class<T> headType, int headRowNum) {
        this.mergeList = CellMergeHandler.buildMergeList(data, headType, headRowNum);
    }

    @Override
    public void afterSheetCreate(WriteSheetHolder holder, ...) {
        // 提前写入合并区域——防止后续数据写入覆盖合并配置
        Sheet sheet = holder.getSheet();
        for (CellRangeAddress region : mergeList) {
            sheet.addMergedRegion(region);
        }
    }

    @Override
    protected void merge(Sheet sheet, Cell cell, Head head, Integer relativeRowIndex) {
        // 清理合并区域内非首行的重复值——设 cell.setBlank()
        // 确保被合并的单元格不显示重复文本
    }
}
💡 设计洞察：为什么分析器（Handler）和执行器（Strategy）要分离？

因为合并区域的计算依赖于数据内容（相邻行的值是否相同），而数据在构造 Strategy 时已经确定。把分析逻辑抽成静态工厂 CellMergeHandler，使得分析结果（List<CellRangeAddress>）可以独立测试、独立复用。

Strategy 只负责两件事：在 POI 的 Sheet 对象上执行 addMergedRegion()，以及清理合并区域内的重复值。它不关心"为什么这些区域需要合并"，只关心"怎么合并"。单一职责的极致体现。

3.2 DataWriteHandler —— 一个类，两个接口

文件：handler/DataWriteHandler.java

DataWriteHandler 同时实现了 SheetWriteHandler 和 CellWriteHandler，是模块中唯一一个横跨两个生命周期接口的处理器。它负责两件事：标红必填表头（@ExcelRequired）和 添加列头批注（@ExcelNotation）。

// DataWriteHandler 核心逻辑（伪代码）
public class DataWriteHandler implements SheetWriteHandler, CellWriteHandler {

    @Override
    public void afterCellDispose(CellWriteHandler context) {
        // 仅在表头行生效
        if (!context.getHead()) return;

        Cell cell = context.getCell();
        ExcelContentProperty property = context.getExcelContentProperty();
        Field field = property.getField();

        // ① 处理 @ExcelRequired —— 表头字体标红
        if (field.isAnnotationPresent(ExcelRequired.class)) {
            Workbook workbook = cell.getSheet().getWorkbook();
            CellStyle style = workbook.createCellStyle();
            Font font = workbook.createFont();
            font.setColor(IndexedColors.RED.getIndex());
            style.setFont(font);
            cell.setCellStyle(style);
        }

        // ② 处理 @ExcelNotation —— 添加 POI 批注
        if (field.isAnnotationPresent(ExcelNotation.class)) {
            ExcelNotation notation = field.getAnnotation(ExcelNotation.class);
            Drawing<?> drawing = cell.getSheet().createDrawingPatriarch();
            Comment comment = drawing.createCellComment(
                new XSSFClientAnchor(0, 0, 0, 0,
                    cell.getColumnIndex(), 0,
                    cell.getColumnIndex() + 2, 3)
            );
            comment.setString(new XSSFRichTextString(notation.value()));
            cell.setCellComment(comment);
        }
    }

    // SheetWriteHandler 接口方法——本类不需要，空实现
    @Override
    public void afterSheetCreate(WriteSheetHolder holder) {}
}
为什么同时实现两个接口？CellWriteHandler 的 afterCellDispose() 在每个单元格写入后触发——这正是设置字体颜色和添加批注的最佳时机（单元格已经存在，可以设置样式）。而 SheetWriteHandler 在本类中不需要（空实现），但接口要求必须实现。这个空实现的存在说明把它抽成两个独立的 Handler 可能更清晰，但作者选择了合并以减少类数量。

关键注意：context.getHead() 的判断确保只有表头行被处理——数据行不会触发标红或批注。而 @ExcelNotation 只适用于单表头场景，多级表头不处理（代码中有对应判断）。

四、阶段⑤：toResponse 写出 HTTP 响应

位置：utils/ExcelBuilder.java 的 toResponse() 方法。

当所有数据写入完毕后，toResponse(HttpServletResponse response) 设置两个关键的 HTTP 响应头：

response.setContentType("application/vnd.openxmlformats-officedocument.spreadsheetml.sheet");
response.setCharacterEncoding("utf-8");
String fileName = URLEncoder.encode(sheetName + ".xlsx", StandardCharsets.UTF_8);
response.setHeader("Content-Disposition", "attachment;filename=" + fileName);
workbook.write(response.getOutputStream());
workbook.finish();  // 通过 ExcelWriterWrapper 封装，防止业务代码提前调用

这里有个隐藏设计——ExcelWriterWrapper（utils/ExcelWriterWrapper.java，Java Record）。它包装了 Fesod Sheet 的 ExcelWriter，只暴露 write() 和 fill() 方法，隐藏了 finish()。原因：如果在数据尚未完全写入时就调用 finish()，会导致输出流提前关闭，后续数据写入失败。通过 Record 的安全包装，把 finish() 的调用时机完全交给 ExcelBuilder 内部管理。
