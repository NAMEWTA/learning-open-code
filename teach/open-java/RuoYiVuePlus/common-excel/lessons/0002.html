第 2 课 · 核心注解：谁声明、谁解析、在哪个环节生效 注解层

🎯 本节目标（对应使命 Success #2）：学完你能逐一解释 6 个自定义注解各自挂在哪一层、最终由哪个处理器/转换器消费，并清晰区分「哪些注解作用于导出」「哪些注解同时作用于导入和导出」。

上一课我们看到模块分为四层。本课聚焦第一层——注解层：6 个注解，每个都是一条元数据声明，最终被不同的下游消费者读取和执行。

一、总览：6 个注解 x 3 个消费位置

注解标注位置谁消费它作用方向
@CellMerge 实体字段 CellMergeHandler 解析 → CellMergeStrategy 执行 仅导出
@ExcelDictFormat 实体字段 ExcelDictConvert（值转换） + ExcelDownHandler（下拉选项） 导入 + 导出
@ExcelEnumFormat 实体字段 ExcelEnumConvert（值转换） + ExcelDownHandler（下拉选项） 导入 + 导出
@ExcelDynamicOptions 实体字段 ExcelDownHandler（下拉选项） 仅导出
@ExcelRequired 实体字段 DataWriteHandler（表头字体标红） 仅导出
@ExcelNotation 实体字段 DataWriteHandler（添加批注） 仅导出

核心规律：字典和枚举注解同时被 Converter（值转换）和 WriteHandler（下拉）消费——这是模块设计中最精妙的地方，同一个注解服务于两个完全不同的目的。

二、@CellMerge：列值重复合并

位置：annotation/CellMerge.java

@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
@Inherited
public @interface CellMerge {
    int index() default -1;         // 列索引，-1时自动推断
    String[] mergeBy() default {};  // 依赖的其他字段名
}
1index —— 指定合并哪一列
默认 -1，CellMergeHandler.getFieldColumnIndexMap() 会按优先级推断：先取注解的 index；若为 -1 则取该字段上 @ExcelProperty.index()；再为 -1 则用反射遍历到的字段顺序。
2mergeBy —— 合并的前提条件
annotation/CellMerge.java:27
这是 @CellMerge 最精华的设计。当两个相邻行的当前字段值相同时，不一定合并——还要检查 mergeBy 中指定的依赖字段是否也相同。例如：合并"部门名称"时，必须同时检查"部门编号"也相同，防止同名但不同编号的部门被错误合并。
// 使用示例：合并「部门名称」列，依赖「部门编号」 @ExcelProperty("部门编号") private Long deptId; @CellMerge(index = 2, mergeBy = {"deptId"}) @ExcelProperty("部门名称") private String deptName; // CellMergeHandler.isMerge() 的逻辑： // 遍历 mergeBy 中的每个字段名 // 比对当前行和上一行的对应字段值 // → 全相等才允许合并，任一不等则中断合并区间

三、@ExcelDictFormat：字典转换的两条路径

位置：annotation/ExcelDictFormat.java

public @interface ExcelDictFormat {
    String dictType() default "";           // 字典类型，如 sys_user_gender
    String readConverterExp() default "";   // 转换表达式，如 0=男,1=女,2=未知
    String separator() default ",";         // 分隔符
}
💡 两条路径，互斥使用：dictType 和 readConverterExp 不能同时有值。有 dictType 时走字典服务查数据库；有 readConverterExp 时直接解析内联表达式。如果两个都为空，转换器返回空字符串。

路径 A：dictType → 字典服务查库

// ExcelDictConvert.convertToExcelData() 中： String type = anno.dictType(); // "sys_user_gender" String value = "0"; // 数据库存的编码 // ↓ label = dictService.getDictLabel("sys_user_gender", "0", ","); // → 返回 "男" // 导入时（convertToJavaData）： String label = "男"; // Excel 中填的值 // ↓ value = dictService.getDictValue("sys_user_gender", "男", ","); // → 返回 "0"

路径 B：readConverterExp → 表达式解析

// 注解：@ExcelDictFormat(readConverterExp = "0=男,1=女,2=未知") // ExcelDictConvert.parseConverterExp() 将其解析为 Map: // {"0": "男", "1": "女", "2": "未知"} // 导出：convertByExp("0", "0=男,1=女,2=未知", ",") // → 在 Map 中查 key="0" → 返回 "男" // 导入：reverseByExp("男", "0=男,1=女,2=未知", ",") // → 在 Map 中查 value="男" → 返回 "0"
3separator —— 多值字段的分隔符
对于多选字段（如"角色"字段值可能是 "1,2,3"），导出时会对每个值分别查字典标签后拼回。导入时反向拆分。

四、@ExcelEnumFormat：枚举的声明式转换

位置：annotation/ExcelEnumFormat.java

public @interface ExcelEnumFormat {
    Class> enumClass();  // 枚举类
    String codeField() default "code";     // 编码属性名
    String textField() default "text";     // 文本属性名
}

与 @ExcelDictFormat 的区别：字典的值来自数据库（动态），枚举的值在代码中定义（静态）。ExcelEnumConvert 使用 双重 ConcurrentHashMap 缓存避免重复反射：

// ExcelEnumConvert 的缓存结构： ENUM_MAP_CACHE: ConcurrentHashMap> ENUM_REVERSE_MAP_CACHE: ConcurrentHashMap> // beforeConvert() — 首次访问某字段时初始化： Field field → 读 @ExcelEnumFormat 枚举常量[] → 用 ReflectUtils.invokeGetter 读 codeField 和 textField → 构建 Map 并缓存 // 导出：convertToExcelData(code值) → ENUM_MAP_CACHE.get(field).get(code) → text // 导入：convertToJavaData(text值) → ENUM_REVERSE_MAP_CACHE.get(field).get(text) → code // 如果 text 值不在反向Map中 → 抛出 IllegalArgumentException("枚举值不匹配")
⚠️ 导入时的重要校验：ExcelEnumConvert 在构建反向 Map 时会检查 text 值是否重复——如果有两个枚举常量的 text 相同，抛异常阻止。导入时如果用户填的文本不在反向 Map 中，报错信息会列出所有允许值，非常友好。

五、@ExcelDynamicOptions：运行时动态下拉

位置：annotation/ExcelDynamicOptions.java

public @interface ExcelDynamicOptions {
    Class providerClass();
}

// ExcelOptionsProvider 接口只有一个方法：
public interface ExcelOptionsProvider {
    Set getOptions();
}

这是模块的策略模式扩展点。当字典和枚举都不适合时（如下拉选项需要从数据库动态查询并拼接业务逻辑），实现 ExcelOptionsProvider 接口，通过 @ExcelDynamicOptions(providerClass = XxxProvider.class) 指定即可。Spring 会自动从容器获取实现类实例。

六、@ExcelRequired 和 @ExcelNotation：模板标注

这两个注解仅影响导出模板的视觉效果，不参与数据转换或校验：

// @ExcelRequired — 表头字体标红，提示「这列必填」
public @interface ExcelRequired {
    IndexedColors fontColor() default IndexedColors.RED;
}

// @ExcelNotation — 表头添加批注，鼠标悬停可见
public @interface ExcelNotation {
    String value() default "";
}

两者都由 DataWriteHandler 消费。在 afterCellDispose() 中：

  • 扫描所有带 @ExcelRequired 的字段 → 取 @ExcelProperty.value()[0] 作表头名 → 设置该表头单元格字体颜色
  • 扫描所有带 @ExcelNotation 的字段 → 在对应表头单元格创建 POI Comment 批注

注意注释中强调的约束：这两个注解仅用于单表头，不支持多层级表头。多级表头场景下 @ExcelProperty.value() 数组长度 > 1，取 [0] 可能不准确。

七、检验一下（提取练习）

不要回看上文，凭记忆作答。

Q1. 以下哪个注解同时被 Converter 和 WriteHandler 消费？

A. @CellMerge B. @ExcelRequired C. @ExcelDictFormat D. @ExcelNotation
看答案C。@ExcelDictFormat 被 ExcelDictConvert（值转换）和 ExcelDownHandler（生成下拉选项）同时消费。@ExcelEnumFormat 同理。其他注解只有一个消费者。

Q2. @CellMerge 的 mergeBy 属性是做什么的？

A. 指定哪些列参与合并 B. 声明合并需要依赖的其他字段——只有当这些字段的值在相邻行也相同时，才允许合并当前列 C. 指定合并后的单元格样式 D. 指定合并区域的颜色
看答案B。这是防止"同名不同实体"被错误合并的关键设计。例如「部门名称」列合并时，必须同时检查「部门编号」也相等，否则两个不同部门恰好同名会被错误合并成一个。

Q3. @ExcelEnumFormat 的 codeField 和 textField 默认值是什么？它们的作用是？

A. 默认 "id" 和 "name"，用于标识枚举的唯一键 B. 默认 "code" 和 "text"，用于反射获取枚举中"编码"和"显示文本"的属性值 C. 默认 "key" 和 "label"，无实际作用 D. 默认 "value" 和 "desc"，只用于下拉选项
看答案B。codeField="code" 和 textField="text"。在 ExcelEnumConvert.beforeConvert() 中通过 ReflectUtils.invokeGetter(enumConstant, anno.codeField()) 反射读取这些属性的值来构建转换映射表。
