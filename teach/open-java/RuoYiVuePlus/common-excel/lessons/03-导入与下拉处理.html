导入与下拉处理

🎯 本节目标：学完你能完整追踪导入链路——ExcelBuilder.read(inputStream, clazz).doRead() → DefaultExcelListener（逐行 validate + 收集）→ ExcelResult.getAnalysis() 返回回执。能讲清 ExcelDownHandler 如何处理 4 种下拉数据源，以及"选项 > 20 个或总字符超 255 时切换为隐藏 Sheet 方案"的原因和实现。

一、先看全景：导入链路的三段式结构

导入链路比导出链路更简洁——核心只有 1 个接口 + 1 个默认实现 + 1 个结果接口 + 1 个默认结果，共 4 个文件。但简洁不意味着简单，它的设计精炼度很高。

导入的入口同样是一行代码：

// 同步导入
ExcelResult<UserImportVO> result = ExcelBuilder
    .read(file.getInputStream(), UserImportVO.class)
    .validate(true)    // 开启 Jakarta Validation
    .failFast(false)   // 遇到错误不中断，继续处理后续行
    .doRead();

// 获取分析回执
String analysis = result.getAnalysis();
// → "成功 98 条，错误 2 条：第 3 行（用户名不能为空），第 15 行（手机号格式不正确）"

让我们先看全景调用链，再逐环节深入：

阶段①  ReadBuilder 构造与配置
  ExcelBuilder.read(inputStream, clazz)
    .validate(true)       // 开启 Jakarta Validation
    .failFast(false)      // 容错模式：错误不中断
    .sheetNo(0)           // 读取第一个 Sheet
    .headRowNumber(1)     // 表头在第 1 行
    .ignoreEmptyRow(true) // 跳过空行
    │

阶段②  注册 DefaultExcelListener 并执行读取
  .doRead()
    │
  Fesod Sheet ReadBuilder
    ├─ 注册 DefaultExcelListener 作为 AnalysisEventListener
    ├─ 逐行解析 Excel
    └─ 每行触发 listener.invoke(data, context)
    │

阶段③  DefaultExcelListener 逐行处理
  invoke(data, context):
    ├─ ① 如果 validate=true → ValidatorUtils.validate(data)
    │     └─ 失败抛 ConstraintViolationException（Jakarta Validation）
    ├─ ② 收集成功行 → successList.add(data)
    └─ onException(exception, context):
        ├─ 记录行号、错误信息
        ├─ 构造 errorMsg
        └─ 如果 failFast=true → 重新抛出异常（中断读取）
           如果 failFast=false → 吞掉异常（继续处理后续行）
    │

阶段④  获取结果
  result = listener.getExcelResult()  →  DefaultExcelResult
    ├─ getList()       → 成功数据列表
    ├─ getErrorList()  → 错误信息列表
    └─ getAnalysis()   → 人性化回执字符串

下面我们逐层深入。

二、ExcelListener 接口 —— 仅一个扩展方法

文件：core/ExcelListener.java

这个接口极其精简——它继承 Fesod Sheet 的 ReadListener<T>，只新增了一个方法：

public interface ExcelListener<T> extends ReadListener<T> {
    ExcelResult<T> getExcelResult();
}

ReadListener<T> 本身提供了三个回调生命周期（由 Fesod Sheet 定义）：

回调方法触发时机典型用途
invoke(T data, AnalysisContext context)每解析完一行数据校验数据、收集到列表
doAfterAllAnalysed(AnalysisContext context)全部行解析完成后写日志、汇总统计
onException(Exception e, AnalysisContext context)解析或校验过程中抛出异常记录错误行、决定是否中断

ExcelListener 接口只做了一件事：声明"我支持获取结果"。这使得业务代码可以拿到导入结果做后续处理，而不仅仅是"成功与否"的布尔值。

三、DefaultExcelListener —— 容错与校验的核心

文件：core/DefaultExcelListener.java（约 152 行）

这是整个导入链路中最值得细读的类。它实现了三个回调方法，编码了两条关键的异常处理策略。

3.1 invoke() —— 逐行校验与收集

// DefaultExcelListener.invoke() 核心逻辑（伪代码）
@Override
public void invoke(T data, AnalysisContext context) {
    // ① Jakarta Validation 校验（可选）
    if (validate) {
        try {
            ValidatorUtils.validate(data);
        } catch (ConstraintViolationException e) {
            // 校验失败：构造异常信息并重新抛出
            // onException 会捕获它
            String msg = e.getConstraintViolations().stream()
                .map(v -> v.getMessage())
                .collect(Collectors.joining(", "));
            throw new ExcelAnalysisException(context.readRowHolder().getRowIndex() + 1, msg);
        }
    }

    // ② 收集成功行
    successList.add(data);
}

这里有一个设计细节：ValidatorUtils.validate() 来自 ruoyi-common-core。它内部调用 Jakarta Bean Validation 的标准 API（Validator.validate()），校验实体类上的 @NotNull、@NotBlank、@Size 等标准注解。校验失败时，ConstraintViolationException 被包装为 ExcelAnalysisException，携带行号信息。

ExcelAnalysisException 是本模块自定义的运行时异常（定义在 core/ 包内），继承 RuntimeException。它携带两个关键字段：rowIndex（错误行号）和 message（错误描述）。onException 中通过 instanceof 判断异常类型来提取行号。

3.2 onException() —— failFast 策略的分叉点

// DefaultExcelListener.onException() 核心逻辑（伪代码）
@Override
public void onException(Exception exception, AnalysisContext context) {
    // ① 提取行号和错误信息
    int rowIndex = 0;
    String errorMsg = exception.getMessage();

    if (exception instanceof ExcelAnalysisException) {
        rowIndex = ((ExcelAnalysisException) exception).getRowIndex();
    } else if (exception instanceof ExcelDataConvertException) {
        // Fesod Sheet 的类型转换异常——无法将单元格文本转为目标类型
        ExcelDataConvertException e = (ExcelDataConvertException) exception;
        rowIndex = e.getRowIndex() + 1;  // Fesod Sheet 的行号从 0 开始
        errorMsg = "第 " + e.getColumnIndex() + " 列数据格式错误";
    }

    // ② 记录错误
    errorList.add(new ErrorInfo(rowIndex, errorMsg));

    // ③ failFast 分叉
    if (failFast) {
        // 快速失败模式——重新抛出，中断整个读取
        throw new RuntimeException("导入第 " + rowIndex + " 行失败：" + errorMsg);
    }
    // 否则吞掉异常，继续处理后续行（容错模式）
}

两种异常类型的处理策略不同：

异常类型来源行号提取方式
ExcelAnalysisException 本模块—— Jakarta Validation 校验失败 exception.getRowIndex() 直接获取
ExcelDataConvertException Fesod Sheet——类型转换失败（如"abc"转 Integer 失败） e.getRowIndex() + 1（Fesod 从 0 计数，业务需要从 1 计数）

异常信息最终存储在 List<ErrorInfo> 中，每个 ErrorInfo 是一个内部 Record，包含 rowIndex 和 errorMsg。

3.3 doAfterAllAnalysed() —— 汇总日志

@Override
public void doAfterAllAnalysed(AnalysisContext context) {
    log.info("Excel 导入完成。成功 {} 条，失败 {} 条。", successList.size(), errorList.size());
}

仅做日志记录。实际的结果汇总由 DefaultExcelResult.getAnalysis() 完成。

四、ExcelResult / DefaultExcelResult —— 结果回执

文件：core/ExcelResult.java（接口）+ core/DefaultExcelResult.java（实现）

导入完成后的调用模式：

ExcelResult<UserImportVO> result = listener.getExcelResult();
List<UserImportVO> successData = result.getList();      // 成功数据
List<String> errors = result.getErrorList();              // 错误列表
String analysis = result.getAnalysis();                  // 人性化回执

// getAnalysis() 输出示例：
// "恭喜您，全部数据导入成功！共 100 条。"
// "成功 98 条，错误 2 条：第 3 行（用户名不能为空），第 15 行（手机号格式不正确）"

DefaultExcelResult 的 getAnalysis() 方法根据错误列表是否为空，生成两种不同格式的回执字符串：

// DefaultExcelResult.getAnalysis() 核心逻辑（伪代码）
@Override
public String getAnalysis() {
    if (CollectionUtils.isEmpty(errorList)) {
        return "恭喜您，全部数据导入成功！共 " + list.size() + " 条。";
    }

    StringBuilder sb = new StringBuilder();
    sb.append("成功 ").append(list.size()).append(" 条");
    sb.append("，错误 ").append(errorList.size()).append(" 条");
    sb.append("：");
    for (String error : errorList) {
        sb.append(error).append("，");
    }
    sb.deleteCharAt(sb.length() - 1);  // 去掉末尾逗号
    sb.append("。");
    return sb.toString();
}
💡 设计洞察：为什么要有 ExcelResult 接口？

如果你只需要知道"导入是否全部成功"，一个 List<T> 就够了。但真实的业务场景中，你需要：①拿到成功数据列表（做后续批量处理）；②知道哪些行失败了、为什么失败（展示给用户）；③决定"部分成功算不算成功"（比如需求是"至少导入一条就行"还是"必须全部成功"）。ExcelResult 接口把这三个需求封装在一个返回值中，同时 getAnalysis() 为前端提供了一个开箱即用的人性化提示字符串——Controller 不需要自己写 if-else 拼句子。

五、ExcelDownHandler —— 下拉处理全景

文件：core/ExcelDownHandler.java

这是模块中最复杂的 WriteHandler，也是最能体现"企业级 Excel 导出"需求的类。它负责在导出模板时为每列生成 Excel 数据验证下拉——就是用户在 Excel 中点击单元格时弹出的小箭头选择框。

5.1 四种下拉数据源

ExcelDownHandler 在 afterSheetCreate() 中遍历实体类的所有字段，检查每个字段上的注解，从四种来源收集下拉选项：

#数据源对应注解/参数获取方式适用场景
① 字典服务（查库） @ExcelDictFormat(dictType = "sys_user_sex") 调用 DictUtils.getDictData() 查字典表 数据量大、多模块共用、频繁变更
② 内联表达式 @ExcelDictFormat(readConverterExp = "0=男,1=女") 解析表达式字符串 split 出标签列表 选项少且固定、不走库
③ 枚举类 @ExcelEnumFormat(enumClass = UserStatus.class) 反射枚举的 textField 属性 选项由枚举定义、编译期固定
④ 动态提供者 @ExcelDynamicOptions(providerClass = DeptOptionsProvider.class) 从 Spring 容器获取 provider，调用 getOptions() 运行时查询数据库、选项动态变化

此外，ExcelBuilder 还支持通过 .options(DropDownOptions) 链式方法直接传入外部 DropDownOptions 对象，这是第五种来源（外部传入），但不走注解扫描，而是作为独立参数处理。

// ExcelDownHandler 收集所有字段的下拉选项（伪代码）
@Override
public void afterSheetCreate(WriteSheetHolder holder, ...) {
    Sheet sheet = holder.getSheet();
    List<Field> fields = getAllFields(headType);

    for (int i = 0; i < fields.size(); i++) {
        Field field = fields.get(i);
        List<String> options = new ArrayList<>();

        // ① @ExcelDictFormat.dictType → 字典服务
        ExcelDictFormat dictFormat = field.getAnnotation(ExcelDictFormat.class);
        if (dictFormat != null && StringUtils.isNotBlank(dictFormat.dictType())) {
            List<SysDictData> dictList = DictUtils.getDictData(dictFormat.dictType());
            options = dictList.stream().map(SysDictData::getDictLabel).collect(toList());
        }
        // ② @ExcelDictFormat.readConverterExp → 解析表达式
        else if (dictFormat != null && StringUtils.isNotBlank(dictFormat.readConverterExp())) {
            options = parseExpression(dictFormat.readConverterExp());
        }
        // ③ @ExcelEnumFormat → 枚举反射
        else {
            ExcelEnumFormat enumFormat = field.getAnnotation(ExcelEnumFormat.class);
            if (enumFormat != null) {
                options = getEnumTexts(enumFormat.enumClass(), enumFormat.textField());
            }
        }
        // ④ @ExcelDynamicOptions → Spring provider
        else {
            ExcelDynamicOptions dynamicOpts = field.getAnnotation(ExcelDynamicOptions.class);
            if (dynamicOpts != null) {
                ExcelOptionsProvider provider = SpringUtils.getBean(dynamicOpts.providerClass());
                options = new ArrayList<>(provider.getOptions());
            }
        }

        // 为第 i 列生成下拉
        if (!options.isEmpty()) {
            addValidation(sheet, i, options);
        }
    }
}

5.2 ">20 个选项或 >255 字符"的分叉策略

这是 ExcelDownHandler 中最精妙的设计决策。POI 的显式列表约束（createExplicitListConstraint）有严重的局限：

POI 显式列表约束（DVConstraint.createExplicitListConstraint）的公式参数是一个逗号分隔的字符串，在 Excel 中这个字符串总长度不能超过 255 个字符。如果选项很多（如部门有 50 个），拼接出来的字符串很容易超过 255 字符，导致下拉失效或 Excel 打开报错。

即使不超 255 字符，选项太多（>20 个）时，Excel 的下拉滚动体验也很差——用户需要在小小的下拉框里翻很久才能找到目标选项。

因此，ExcelDownHandler 采用了双轨策略：

// ExcelDownHandler 根据选项数量和总字符数切换策略（伪代码）
private void addValidation(Sheet sheet, int colIndex, List<String> options) {
    String formulaString = String.join(",", options);

    if (options.size() <= 20 && formulaString.length() <= 255) {
        // 策略 A：显式列表约束（简单场景）
        // 直接在公式里写死选项："男,女,未知"
        DataValidationHelper helper = sheet.getDataValidationHelper();
        DataValidationConstraint constraint = helper.createExplicitListConstraint(
            options.toArray(new String[0])
        );
        CellRangeAddressList regions = new CellRangeAddressList(
            1, 65535, colIndex, colIndex  // 从第 2 行到最大行
        );
        DataValidation validation = helper.createValidation(constraint, regions);
        sheet.addValidationData(validation);
    } else {
        // 策略 B：隐藏 Sheet + 名称管理器（复杂场景）
        // 1. 创建一个隐藏的 Sheet，把选项列表写到该 Sheet 的 A 列
        // 2. 给这个区域定义一个名称（如 "dict_sys_user_sex"）
        // 3. 公式约束使用 =INDIRECT("dict_sys_user_sex")
        createFormulaListConstraint(sheet, colIndex, options, colIndex);
    }
}

策略 B 的细节：隐藏 Sheet + 名称管理器

// createFormulaListConstraint 的内部实现（伪代码）
private void createFormulaListConstraint(Sheet sheet, int colIndex, List<String> options, int dictIndex) {
    Workbook workbook = sheet.getWorkbook();

    // 1. 创建隐藏 Sheet
    String hiddenSheetName = "dictSheet" + dictIndex;
    Sheet hiddenSheet = workbook.createSheet(hiddenSheetName);
    workbook.setSheetHidden(workbook.getSheetIndex(hiddenSheet), true);

    // 2. 把选项写入隐藏 Sheet 的 A 列
    for (int i = 0; i < options.size(); i++) {
        Row row = hiddenSheet.createRow(i);
        row.createCell(0).setCellValue(options.get(i));
    }

    // 3. 给数据区域定义名称
    String nameName = "dict_" + dictIndex;
    Name name = workbook.createName();
    name.setNameName(nameName);
    name.setRefersToFormula(hiddenSheetName + "!$A$1:$A$" + options.size());

    // 4. 在下拉列使用 INDIRECT 公式引用名称
    DataValidationHelper helper = sheet.getDataValidationHelper();
    DataValidationConstraint constraint = helper.createFormulaListConstraint(
        "INDIRECT(\"" + nameName + "\")"
    );
    CellRangeAddressList regions = new CellRangeAddressList(1, 65535, colIndex, colIndex);
    DataValidation validation = helper.createValidation(constraint, regions);
    sheet.addValidationData(validation);
}
💡 设计洞察：为什么阈值选 20 和 255？

这两个阈值来自 Excel 的硬性限制和实际体验：
① 255 字符是 Excel 数据验证公式参数的最大长度——这是硬限制，超过会导致校验失效。
② 20 个选项是经验阈值——虽然 30 个选项的字符串可能不超 255 字符（如果每个选项都很短），但 30 个选项的下拉框用户体验极差。20 是一个在"性能开销"（隐藏 Sheet 有额外 IO）和"用户体验"之间的平衡点。

隐藏 Sheet 方案的代价是：多写一个 Sheet、文件体积稍大、打开时需加载额外数据。但它没有选项数量上限——理论上可以放几千个选项。

六、DropDownOptions —— 级联下拉

文件：core/DropDownOptions.java

前面讲了单列下拉，但真实的业务场景经常需要级联下拉——选择"省份"后，"城市"的下拉只显示该省的城市。这就是 DropDownOptions 解决的问题。

// DropDownOptions 的核心结构
public class DropDownOptions {
    private int index;                              // 一级下拉所在列索引
    private int nextIndex;                          // 二级下拉所在列索引
    private List<String> options;                    // 一级选项列表
    private Map<String, List<String>> nextOptions;  // 一级值 → 二级选项列表

    // 从业务实体列表自动构建级联结构
    public static <T> DropDownOptions buildLinkedOptions(
            List<T> data,
            int index, int nextIndex,
            Function<T, String> parentExtractor,    // 提取一级值（如省份）
            Function<T, String> childExtractor       // 提取二级值（如城市）
    ) {
        DropDownOptions ddo = new DropDownOptions();
        ddo.index = index;
        ddo.nextIndex = nextIndex;

        // 收集去重后的一级选项
        Set<String> parentSet = data.stream().map(parentExtractor).collect(Collectors.toSet());
        ddo.options = new ArrayList<>(parentSet);

        // 构建一级→二级映射
        ddo.nextOptions = new HashMap<>();
        for (String parent : parentSet) {
            List<String> children = data.stream()
                .filter(item -> parent.equals(parentExtractor.apply(item)))
                .map(childExtractor)
                .distinct()
                .collect(Collectors.toList());
            ddo.nextOptions.put(parent, children);
        }
        return ddo;
    }
}

使用方式（在 Controller 中）：

// 构建省份-城市级联下拉
List<CityData> cityDataList = cityService.list();
DropDownOptions linkedOptions = DropDownOptions.buildLinkedOptions(
    cityDataList,
    0, 1,                        // 一级在第 0 列，二级在第 1 列
    CityData::getProvince,       // 提取省份
    CityData::getCity            // 提取城市
);

ExcelBuilder.writer(UserImportVO.class)
    .options(linkedOptions)      // 传入外部 DropDownOptions
    .toResponse(response);

ExcelDownHandler 接收到 DropDownOptions 后，为一级和二级分别创建隐藏 Sheet 和对应的名称管理器，二级下拉使用 =INDIRECT(一级选择值) 公式实现联动——这与策略 B 的隐藏 Sheet 方案一脉相承，只是二级的名称映射更复杂（需要为每个一级值创建独立的名称）。

级联下拉的 Excel INDIRECT 原理：假设省份选择了"广东"，城市下拉的公式是 =INDIRECT("city_广东")，指向名为 city_广东 的隐藏 Sheet 区域。当用户改变省份选择时，INDIRECT 函数重新计算，城市下拉自动切换到对应省份的城市列表。这不是 Java 代码在运行——这是 Excel 的公式引擎在工作。Java 代码只负责"预先创建所有名称和对应的隐藏 Sheet 数据"。
