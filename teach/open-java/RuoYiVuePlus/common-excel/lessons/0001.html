第 1 课 · Excel 模块全景：注解、转换、处理三层架构 入门地图

🎯 本节目标（对应使命 Success #1）：学完你能用一句话说清 ruoyi-common-excel 与 Fesod Sheet 的关系，画出模块的「注解层 - 转换器层 - 处理器层 - 工具层」四层架构，并说出 21 个文件各自归属哪一层。这是后面四节课所有细节挂靠的骨架——先有地图，再走小路。

我们先不抠任何一行实现，只回答一个问题：打开 ruoyi-common-excel 这个模块，你会看到什么？

一、整个模块的体量：21 个文件，四个包

与短信模块「3 个类搞定」不同，Excel 模块更厚重——21 个 Java 文件，分布在 4 个包中：

ruoyi-common-excel/src/main/java/org/dromara/common/excel/ ├── annotation/ ← 【注解层】6个自定义注解 │ ├── CellMerge.java — 列值合并 │ ├── ExcelDictFormat.java — 字典转换 │ ├── ExcelEnumFormat.java — 枚举转换 │ ├── ExcelDynamicOptions.java — 动态下拉 │ ├── ExcelRequired.java — 必填标注 │ └── ExcelNotation.java — 批注 ├── convert/ ← 【转换器层】3个Fesod Sheet Converter │ ├── ExcelBigNumberConvert.java — 大数字精度保护 │ ├── ExcelDictConvert.java — 字典双向转换 │ └── ExcelEnumConvert.java — 枚举双向转换 ├── core/ ← 【处理器层】7个核心类 + 1个接口 │ ├── DefaultExcelListener.java — 导入事件监听 │ ├── ExcelListener.java — 监听接口 │ ├── ExcelResult.java — 结果接口 │ ├── DefaultExcelResult.java — 默认结果实现 │ ├── CellMergeStrategy.java — 合并执行器 │ ├── CellMergeHandler.java — 合并分析器 │ ├── ExcelDownHandler.java — 下拉处理器 │ ├── DropDownOptions.java — 下拉选项 POJO │ └── ExcelOptionsProvider.java — 动态数据提供接口 ├── handler/ ← 【辅助处理器】1个 │ └── DataWriteHandler.java — 批注与必填样式 └── utils/ ← 【工具/API层】2个 ├── ExcelBuilder.java — 唯一对外API（导出/导入/模板） └── ExcelWriterWrapper.java — ExcelWriter安全包装
💡 一句话定位：
ruoyi-common-excel 不是一个从零造的 Excel 框架，而是基于 Fesod Sheet（Apache POI 的高层封装）的一层「项目化适配」。它通过实现 Fesod Sheet 的 Converter、WriteHandler、ReadListener 等扩展点，把 RuoYi 自己的字典服务、枚举体系、校验框架注入到 Excel 读写流程中。

二、四层架构：谁声明、谁解析、谁执行

理解本模块最关键的心智模型是「声明 - 解析 - 执行」三层分离：

层谁声明谁解析谁执行
注解层 开发者写在实体字段上
如 @ExcelDictFormat(dictType="sys_user_gender")		 转换器和处理器通过反射读取
如 AnnotationUtil.getAnnotation(field, ExcelDictFormat.class)		 （注解本身不执行，只提供元数据）
转换器层 Fesod Sheet 的 Converter<T> 接口 Fesod Sheet 在读写每个单元格时，按 supportJavaTypeKey() 匹配到对应 Converter Converter 的 convertToJavaData() / convertToExcelData()
处理器层 Fesod Sheet 的 WriteHandler / ReadListener 接口 通过 registerWriteHandler() / registerReadListener() 挂载 Fesod Sheet 在生命周期钩子（Sheet创建后、单元格写入后、读到一行数据时）回调
工具层 ExcelBuilder 链式 DSL （它是发起者，不是被回调者） 组装 WriterBuilder/ReaderBuilder，注册 Converter 和 WriteHandler，发起读写
🤔 想一想：为什么转换器和处理器是分开的两层？不能都写成 WriteHandler 吗？ 因为它们的回调时机和粒度不同。 Converter 是单元格级别的——Fesod Sheet 每写一个格子都会问「这个 Java 值怎么变成 Excel 单元格数据？」，由 supportJavaTypeKey() 按类型匹配。WriteHandler 是生命周期级别的——「Sheet 创建完了」「这行写完了」，适合做跨单元格的操作（合并区域、下拉校验、表头样式）。两者职责清晰：Converter 管「值转换」，WriteHandler 管「结构/样式」。

三、导出链路：一条完整调用链

导出是模块最复杂的能力。从业务代码一行 ExcelBuilder.of(...).toResponse(response) 到浏览器下载 Excel 文件，中间经历了什么？

业务代码 ExcelBuilder Fesod Sheet + 处理器 ──────── ──────────── ────────────────── ExcelBuilder.of(list, SysUser.class) ──► 保存数据 + 头类型 .sheetName("用户数据") .merge() ──► merge = true .toResponse(response) ──► ① resetResponse(设置响应头) ② createWriterBuilder(): 自动注册 ExcelBigNumberConvert ← Converter 自动注册 DataWriteHandler ← 批注+必填 自动注册 ExcelDownHandler ← 下拉 自动注册 列宽策略 ③ createSheetBuilder() ④ 如果 merge=true: 自动注册 CellMergeStrategy ← 合并 ⑤ builder.doWrite(data) │ ┌────────────────────────────┘ ▼ Fesod Sheet 内部写出流程: 每写一个单元格 → 匹配 Converter.convertToExcelData() ← 字典/枚举/大数转换 Sheet创建后 → CellMergeStrategy.afterSheetCreate() ← 写入合并区域 → ExcelDownHandler.afterSheetCreate() ← 生成下拉校验 写单元格后 → CellMergeStrategy.merge() ← 清空非首行重复值 → DataWriteHandler.afterCellDispose() ← 表头标红+批注 数据写完 → response.flush() → 浏览器下载

注意：ExcelBuilder 自动注册了 ExcelBigNumberConvert、DataWriteHandler、ExcelDownHandler 三个处理器——业务代码不需要手动指定。合并策略只在调用 .merge() 时才注册。

四、导入链路：从 MultipartFile 到 ExcelResult

业务代码 ExcelBuilder.ReadBuilder 处理器 ──────── ────────────────────── ────── ExcelBuilder.read(inputStream, SysUser.class) .validate(true) .failFast(false) ──► 保存配置 .doRead() ──► ① 创建 DefaultExcelListener(validate, failFast) ② createReaderBuilder(listener) ③ builder.sheet().doRead() │ ┌─────────────────────────┘ ▼ Fesod Sheet 逐行解析: 读到表头 → DefaultExcelListener.invokeHeadMap() — 记录 headMap（出错时定位列名） 读到一行 → DefaultExcelListener.invoke(data): if validate → ValidatorUtils.validate(data) — Jakarta Validation 成功 → excelResult.getList().add(data) 失败 → onException(ConstraintViolationException) 解析异常 → DefaultExcelListener.onException(): 转换异常 → "第X行-第Y列-表头Z: 解析异常" 校验异常 → "第X行数据校验异常: 错误信息列表" 收集进 errorList if failFast → throw ExcelAnalysisException（终止读取） 全读完 → DefaultExcelListener.doAfterAllAnalysed() — 打日志 返回 → readListener.getExcelResult() — DefaultExcelResult .getList() → 成功数据 .getErrorList() → 错误信息列表 .getAnalysis() → "共X条，成功导入Y条，错误Z条"
💡 导入设计亮点：failFast 参数控制异常行为——true 时遇到第一个错误就抛异常终止（适合严格校验场景），false 时收集全部错误后返回汇总回执（适合「尽量多导入、事后再看哪些失败了」的场景）。错误信息带精确的行号+列号+表头名，非常友好。

六、检验一下（提取练习）

不要回看上文，凭记忆作答。

Q1. ruoyi-common-excel 和 Fesod Sheet 是什么关系？

A. ruoyi-common-excel 自己实现了 Excel 读写的全部逻辑，没用到 Fesod Sheet B. ruoyi-common-excel 是 Fesod Sheet 的一层项目化适配，通过实现其 Converter/WriteHandler/ReadListener 扩展点注入 RuoYi 自身能力 C. Fesod Sheet 依赖 ruoyi-common-excel 才能运行 D. 两者是竞争关系
看答案B。模块的 3 个 Converter 实现 Fesod Sheet 的 Converter<T>，多个 WriteHandler 实现其 SheetWriteHandler/CellWriteHandler，DefaultExcelListener 实现其 AnalysisEventListener。所有 Excel 读写的底层能力都来自 Fesod Sheet。

Q2. 模块的四层架构从上到下依次是？

A. 工具层 → 处理器层 → 转换器层 → 注解层 B. 注解层 → 转换器层 → 处理器层 → 工具层 C. 转换器层 → 注解层 → 处理器层 → 工具层 D. 处理器层 → 转换器层 → 注解层 → 工具层
看答案B。注解层声明元数据 → 转换器层消费注解做类型转换 → 处理器层在生命周期钩子中做结构/样式处理 → 工具层（ExcelBuilder）是业务代码的入口。

Q3. ExcelBuilder.createWriterBuilder() 自动注册了哪三个处理器？

A. CellMergeStrategy, DefaultExcelListener, ExcelWriterWrapper B. ExcelBigNumberConvert, DataWriteHandler, ExcelDownHandler C. ExcelDictConvert, ExcelEnumConvert, CellMergeHandler D. DropDownOptions, ExcelOptionsProvider, DataWriteHandler
看答案B。在 ExcelBuilder.java 的 createWriterBuilder() 方法中，始终注册 ExcelBigNumberConvert（防止长数字精度丢失）、DataWriteHandler(headType)（处理 @ExcelRequired 和 @ExcelNotation）、ExcelDownHandler(options)（处理下拉选项）。CellMergeStrategy 仅在调用 .merge() 时额外注册。
📖 推荐一手资料：本模块的 utils/ExcelBuilder.java——它是模块的唯一对外入口，从它的 createWriterBuilder() 和 ReadBuilder.createReaderBuilder() 两个方法可以看清所有默认注册的处理器。
