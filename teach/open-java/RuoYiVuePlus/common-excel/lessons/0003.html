第 3 课 · 导入处理器链：从 InputStream 到 ExcelResult 处理器层 ①

🎯 本节目标（对应使命 Success #6）：学完你能完整追踪导入链路——ExcelBuilder.read(is, clazz).doRead() → DefaultExcelListener 逐行处理 → ExcelResult.getAnalysis() 返回回执。能解释 failFast 参数如何影响异常行为，以及两种异常类型的不同处理策略。

导入链路比导出链路简单——只有 1 个接口 + 1 个默认实现 + 1 个结果接口 + 1 个默认结果，共 4 个文件。但设计很精炼，值得细读。

二、ExcelListener —— 一个极简的接口

位置：core/ExcelListener.java

public interface ExcelListener<T> extends ReadListener<T> {
    ExcelResult<T> getExcelResult();
}

它继承 Fesod Sheet 的 ReadListener<T>，只加了一个方法 getExcelResult()。Fesod Sheet 的 ReadListener 已经定义了全部分析事件回调：invoke()、doAfterAllAnalysed()、onException()、invokeHeadMap()。RuoYi 只补充了「读取完成后我需要获取结果」的契约。

💡 设计意图：如果你对默认的异常处理或校验策略不满意，可以实现 ExcelListener 接口（或直接继承 DefaultExcelListener），通过 ReadBuilder.listener(customListener) 注入。模块的扩展点非常干净。

三、DefaultExcelListener —— 逐行解析的核心

位置：core/DefaultExcelListener.java（152 行）

3.1 构造器：两个开关控制行为

// 核心字段
private Boolean isValidate = Boolean.TRUE;  // 是否执行JSR303校验
private Boolean failFast = Boolean.TRUE;    // 异常时是否立即终止
private Map<Integer, String> headMap;       // 表头数据（用于错误信息定位列名）
private final ExcelResult<T> excelResult = new DefaultExcelResult<>();

// 三个构造器
public DefaultExcelListener()                          // validate=true, failFast=true
public DefaultExcelListener(boolean isValidate)        // 自定义校验开关
public DefaultExcelListener(boolean isValidate, boolean failFast) // 全自定义
1isValidate —— 是否需要 Jakarta Validation 校验
默认 true。如果导入模板已经通过 Excel 下拉限制了输入值，可以考虑关闭校验提升性能。关闭方式：ExcelBuilder.read(is, clazz).validate(false).doRead()。
2failFast —— 快速失败 vs 全量收集
默认 true（快速失败）。设为 false 时，即使某行出错也不终止读取，而是把所有错误收集起来，最后一次性返回。适合「先导入了再说，回头人工处理错误行」的柔性场景。

3.2 invoke() —— 每读一行回调一次

// core/DefaultExcelListener.java:125
@Override
public void invoke(T data, AnalysisContext context) {
    if (isValidate) {
        ValidatorUtils.validate(data);   // ① 执行 Jakarta Validation
    }
    excelResult.getList().add(data);     // ② 校验通过则加入成功列表
}

逻辑非常直白：先校验，再收集。如果 ValidatorUtils.validate(data) 抛出 ConstraintViolationException，会被下面的 onException() 兜住。

3.3 onException() —— 异常分类处理（最重要的方法）

// core/DefaultExcelListener.java:76-104
@Override
public void onException(Exception exception, AnalysisContext context) throws Exception {
    String errMsg = null;
    if (exception instanceof ExcelDataConvertException e) {
        // 类型一：单元格转换异常 — 能定位到具体行号+列号
        errMsg = StrUtil.format("第{}行-第{}列-表头{}: 解析异常",
            e.getRowIndex() + 1, e.getColumnIndex() + 1,
            headMap == null ? "" : headMap.get(e.getColumnIndex()));
    }
    if (exception instanceof ConstraintViolationException e) {
        // 类型二：JSR303 校验异常 — 只能定位到行号
        String violationsMsg = StreamUtils.join(
            e.getConstraintViolations(), ConstraintViolation::getMessage, ", ");
        errMsg = StrUtil.format("第{}行数据校验异常: {}",
            context.readRowHolder().getRowIndex() + 1, violationsMsg);
    }
    if (errMsg == null) {
        // 类型三：兜底 — 其他未知异常
        errMsg = StrUtil.format("第{}行数据异常: {}",
            context.readRowHolder().getRowIndex() + 1, exception.getMessage());
    }
    excelResult.getErrorList().add(errMsg);  // 收集错误信息
    if (failFast) {
        throw new ExcelAnalysisException(errMsg);  // 快速失败：立即终止
    }
}
💡 两种异常的区别：
ExcelDataConvertException：Fesod Sheet 在把单元格字符串转成 Java 类型时抛出的。发生在某一行数据还没进 invoke() 之前。好处是能拿到精确的 rowIndex + columnIndex，错误信息里带表头名。
ConstraintViolationException：在 invoke() 里调 ValidatorUtils.validate() 抛出的。此时数据已成功转换为 Java 对象，但字段值不满足 @NotNull / @Size 等校验规则。只能拿到行号。
如果两个都不是，走兜底分支——打印完整堆栈（log.warn），收集简化的错误信息。

3.4 invokeHeadMap() —— 记录表头映射

// core/DefaultExcelListener.java:113
@Override
public void invokeHeadMap(Map<Integer, String> headMap, AnalysisContext context) {
    this.headMap = headMap;  // 保存列索引 → 表头名的映射
}

这个方法在 onException() 中用到——当转换异常发生时，通过列索引查出「是哪个表头列出了问题」，拼进错误信息："第3行-第2列-表头性别: 解析异常"。

四、ExcelResult / DefaultExcelResult —— 结果契约

位置：core/ExcelResult.java（27 行）+ core/DefaultExcelResult.java（98 行）

public interface ExcelResult<T> {
    List<T> getList();        // 成功解析的数据
    List<String> getErrorList(); // 错误信息列表
    String getAnalysis();      // 导入回执（人性化摘要）
}

DefaultExcelResult 的精髓在 getAnalysis()：

// core/DefaultExcelResult.java:84-96
public String getAnalysis() {
    int successCount = list.size();
    int errorCount = errorList.size();
    if (successCount == 0) {
        return "读取失败，未解析到数据";
    } else {
        if (errorCount == 0) {
            return StrUtil.format("恭喜您，全部读取成功！共{}条", successCount);
        } else {
            return StrUtil.format("共{}条，成功导入{}条，错误{}条",
                successCount + errorCount, successCount, errorCount);
        }
    }
}

三种情况的分支逻辑：全失败、全成功、部分成功。业务方拿到 ExcelResult 后，可以用 getAnalysis() 直接丢给前端做 toast 提示，也可以用 getList() + getErrorList() 做更精细的处理。

Q1. DefaultExcelListener 处理了哪两种具体的异常类型？它们在错误信息上的差异是什么？

A. IOException 和 NullPointerException，前者有文件名，后者没有 B. ExcelDataConvertException（能定位到行号+列号+表头名）和 ConstraintViolationException（只能定位到行号） C. SQLException 和 ValidationException D. 不区分类型，统一处理
看答案B。ExcelDataConvertException 携带 rowIndex + columnIndex，配合 headMap 可以定位到具体表头列。而 ConstraintViolationException 发生在数据已转换完成后的校验阶段，只能通过 context.readRowHolder() 拿到行号。

Q2. 将 failFast 设为 false 后，onException() 的行为有什么变化？

A. 不再收集错误信息 B. 收集错误信息但不抛出 ExcelAnalysisException，继续读取后续行 C. 跳过校验直接收集数据 D. 行为不变
看答案B。代码中只将 excelResult.getErrorList().add(errMsg) 放在 if (failFast) 之前，因此无论 failFast 是什么值都会收集错误。区别在于 failFast=true 时紧接着 throw new ExcelAnalysisException(errMsg) 终止读取。

Q3. ExcelResult.getAnalysis() 在「5 条数据全部成功导入」时会返回什么？

A. "读取失败，未解析到数据" B. "恭喜您，全部读取成功！共 5 条" C. "共 5 条，成功导入 5 条，错误 0 条" D. "导入完成"
看答案B。getAnalysis() 对 errorCount == 0 的情况单独写了「恭喜您，全部读取成功！」的文案。只有存在错误时才返回「共X条，成功导入Y条，错误Z条」格式。
