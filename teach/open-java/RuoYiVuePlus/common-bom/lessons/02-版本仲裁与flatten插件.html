第 2 课 · 版本仲裁与 flatten 插件 CI Friendly

本节目标：学完你能够解释 ${revision} 变量的 CI Friendly 机制、flatten-maven-plugin 的 resolveCiFriendliesOnly 工作模式、Maven 依赖仲裁的"胜者选择"规则，以及 BOM 如何绕过仲裁不确定性来锁定版本。

上一课我们讲到 ruoyi-common-bom 管理 23 个模块的版本，所有版本号都写成了 ${revision}。但这个变量最终必须变成真实的版本号（如 6.0.0-BETA）才能发布到 Maven 仓库。这节课就来解开这个"变脸"过程。

一、${revision} — Maven 3.2+ 的 CI Friendly 变量

传统 Maven 项目的版本号直接在 <version> 标签中硬编码：

<!-- 传统方式: 版本号硬编码 -->
<version>6.0.0-BETA</version>

<!-- CI Friendly 方式: 用变量占位 -->
<version>${revision}</version>
<properties>
    <revision>6.0.0-BETA</revision>
</properties>

Maven 3.2+ 引入了三个 CI Friendly 变量：

变量含义典型用途
${revision}项目的主版本号日常开发：6.0.0-BETA、1.0.0
${sha1}Git commit SHA1快照版本：1.0.0-abc1234
${changelist}版本后缀标识区分环境：-SNAPSHOT、-RC1、-M1

使用这种机制的最大好处是可以在命令行动态修改版本，而不需要修改任何 POM 文件：

# CI/CD 流水线中动态注入版本号
mvn clean deploy -Drevision=6.0.0-RELEASE

# 日常开发用快照版本
mvn clean install -Drevision=6.0.0-SNAPSHOT

# 打 tag 发布时用正式版本
mvn clean deploy -Drevision=6.0.0

在 RuoYi-Vue-Plus 中，${revision} 的定义位于根 POM 的 <properties> 段：

<!-- RuoYi-Vue-Plus/pom.xml -->
<properties>
    <revision>6.0.0-BETA</revision>   <!-- 全项目统一版本号 -->
    <java.version>21</java.version>
    <spring-boot.version>4.1.0</spring-boot.version>
    <!-- ... 其他属性 ... -->
</properties>

<groupId>org.dromara</groupId>
<artifactId>ruoyi-vue-plus</artifactId>
<version>${revision}</version>         <!-- 这里使用变量 -->
注意：BOM 的 pom.xml（ruoyi-common-bom/pom.xml）也在自己的 <properties> 中定义了 <revision>6.0.0-BETA</revision>。因为 BOM 是一个独立工件（没有 parent），不能依赖父 POM 的属性继承，必须在本地声明。

二、问题来了：变量不能直接发布

如果你直接执行 mvn deploy，发布到 Maven 仓库的 POM 文件会仍然包含 ${revision} 变量：

<!-- 发布后的 POM（有问题的版本） -->
<groupId>org.dromara</groupId>
<artifactId>ruoyi-common-bom</artifactId>
<version>${revision}</version>     <!-- 变量未被替换！ -->
<!-- ...
<dependency>
    <version>${revision}</version>  <!-- 下游消费者无法解析这个变量 -->
</dependency>
-->

下游消费者（引入 ruoyi-common-bom 的项目）没有任何地方定义了 ${revision}，Maven 解析时会直接报错。这就是为什么需要 flatten-maven-plugin。

三、flatten-maven-plugin — 变量的"替身术"

flatten-maven-plugin（版本 1.7.3）是 Maven 官方出品的插件，它在构建过程中生成一个"扁平化"的 POM——把 ${revision} 等变量替换为真实值。

RuoYi-Vue-Plus 的根 POM 中配置了该插件：

<!-- RuoYi-Vue-Plus/pom.xml 的 pluginManagement 段 -->
<plugin>
    <groupId>org.codehaus.mojo</groupId>
    <artifactId>flatten-maven-plugin</artifactId>
    <version>${flatten-maven-plugin.version}</version>  <!-- 1.7.3 -->
    <configuration>
        <updatePomFile>true</updatePomFile>            <!-- 写回原 POM -->
        <flattenMode>resolveCiFriendliesOnly</flattenMode>
    </configuration>
    <executions>
        <execution>
            <id>flatten</id>
            <phase>process-resources</phase>          <!-- 在 compile 之前执行 -->
            <goals>
                <goal>flatten</goal>
            </goals>
        </execution>
        <execution>
            <id>flatten.clean</id>
            <phase>clean</phase>
            <goals>
                <goal>clean</goal>
            </goals>
        </execution>
    </executions>
</plugin>

核心配置解析：

配置项值含义
flattenModeresolveCiFriendliesOnly最保守模式：只替换 ${revision}/${sha1}/${changelist}，保留其它所有内容
updatePomFiletrue用扁平化后的 POM 覆盖原始 POM（install/deploy 时生效）
phaseprocess-resources在 compile 之前执行，确保编译时版本号已正确

flatten-maven-plugin 提供四种 flattenMode：

模式行为使用场景
resolveCiFriendliesOnly只替换 CI 变量（revision/sha1/changelist）RuoYi 的选择：改动最小，风险最低
oss简化到 Maven Central 要求的最小 POM发布到 Maven Central
ossrh类似 oss，但保留更多信息发布到 Sonatype OSSRH
defaults移除所有构建相关配置消费者只需要依赖信息
clean最激进：只保留 GAV + dependencies极简消费者 POM
设计洞察：为什么 RuoYi 选择 resolveCiFriendliesOnly 而不是 oss？因为 resolveCiFriendliesOnly 只动变量，不动 POM 结构，对构建过程的影响最小。如果用 oss 模式，发布后的 POM 会丢失 parent 声明、properties、plugin 配置等信息——这对当前多模块项目的内部引用会造成破坏。选择最保守的模式，是"先保证正确，再考虑精简"的工程实践。

四、完整工作流：从变量到发布

下面是一次完整构建的生命周期：

【Step 1: 声明阶段】 pom.xml 源码中: <version>${revision}</version> <properties><revision>6.0.0-BETA</revision></properties> │ mvn clean install / deploy ▼ 【Step 2: process-resources 阶段】 flatten-maven-plugin 启动 ├── 扫描 POM 中的 ${revision}、${sha1}、${changelist} ├── 从 properties 或 -D 参数中读取真实值 ├── 生成 .flattened-pom.xml（扁平化副本） └── 内容变为: <version>6.0.0-BETA</version> <dependency> <version>6.0.0-BETA</version> </dependency> │ updatePomFile=true ▼ 【Step 3: install/deploy 阶段】 ├── 原始 pom.xml 被 .flattened-pom.xml 覆盖（仅在本次构建周期） ├── 扁平化后的 POM 随工件一起安装/发布到仓库 └── 下游消费者看到的 POM 中不再有任何 ${} 变量 【Step 4: clean 阶段（可选）】 flatten.clean 执行: └── 清理 .flattened-pom.xml 临时文件

关键规则：原始 pom.xml（磁盘上的文件）不会被永久修改。扁平化后的 POM 只在本次 Maven 构建周期的内存和 install/deploy 输出中生效。

五、Maven 依赖仲裁（Dependency Mediation）

有了统一的版本管理机制，还有一个问题要解决：当同一依赖的多个版本出现在依赖树中时，Maven 如何选择？这就是依赖仲裁（Dependency Mediation）。

Maven 的依赖仲裁遵循两条规则：

规则一（距离优先）：离根节点路径最短的版本胜出。
规则二（先声明优先）：当路径深度相同时，在 POM 中先声明的版本胜出。

用一个例子说清楚：

【场景】项目 A 依赖 B 和 C，B 和 C 各自传递依赖了不同版本的 X。 A (root) ├── B (depth=1) │ └── X:1.0 (depth=2) └── C (depth=1) └── D (depth=2) └── X:2.0 (depth=3) 【仲裁结果】X:1.0 胜出！ 理由: X:1.0 的深度=2, X:2.0 的深度=3, 距离优先 → 选 1.0 【另一个场景】两条路径深度相同: A (root) ├── B (depth=1) │ └── X:1.0 (depth=2) └── C (depth=1) └── X:2.0 (depth=2) 【仲裁结果】X:1.0 胜出！（假设 B 在 POM 中先于 C 声明） 理由: 深度相同 → 先声明优先 → 谁在 <dependencies> 中先出现谁赢
危险提示：依赖仲裁的"先声明优先"规则是不可靠的——你调整一下依赖顺序，版本选择就可能改变。这种不确定性正是需要 BOM 来消除的。

六、BOM 如何绕过仲裁的不确定性

依赖仲裁的不确定性来自于传递依赖的版本不可控。BOM 的解决方案是在根节点（最近祖先）就锁定版本：

<!-- 根 POM 导入 BOM，在 dependencyManagement 中锁定版本 -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.dromara</groupId>
            <artifactId>ruoyi-common-bom</artifactId>
            <version>${revision}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

BOM 的工作原理是抢占距离最近的"声明位"：

【无 BOM 的情况 — 依赖仲裁发挥】 项目 A ├── dependencyManagement: (空) ├── 依赖 B (传递 X:1.0, depth=2) └── 依赖 C (传递 X:2.0, depth=2) → Maven 仲裁: 先声明者胜出 → 不确定! 【有 BOM 的情况 — 版本被锁定】 项目 A ├── dependencyManagement: X:3.0 (depth=0) ← BOM 锁定的版本 ├── 依赖 B (传递 X:1.0, depth=2) └── 依赖 C (传递 X:2.0, depth=2) → Maven 仲裁: dependencyManagement 的深度=0, 最近祖先胜出 → X:3.0!

这就是 dependencyManagement 的"override"效应——不管传递依赖带来什么版本，dependencyManagement 中声明的版本一定赢，因为它位于依赖树的最近祖先位置。

七、完整链路：从根 POM 到子模块的版本解析

把前两课的内容串成一条完整的依赖解析链路：

【全链路版本解析】 1. 根 POM (RuoYi-Vue-Plus/pom.xml) └── dependencyManagement └── import ruoyi-common-bom (scope=import, type=pom) └── 导入 23 个 common 模块的版本声明 2. BOM (ruoyi-common-bom/pom.xml) └── dependencyManagement ├── ruoyi-common-core:${revision} ← 变量占位 ├── ruoyi-common-redis:${revision} └── ... (21 more) 3. flatten-maven-plugin (process-resources 阶段) └── ${revision} → 6.0.0-BETA └── 生成 .flattened-pom.xml 4. 子模块 (ruoyi-system/pom.xml) └── dependencies └── ruoyi-common-core └── 继承链查找: parent → root pom → BOM → 版本=6.0.0-BETA 5. Maven 依赖仲裁 └── dependencyManagement 声明在 depth=0 └── 所有传递依赖无法覆盖 → 版本被锁定

这五种机制环环相扣：${revision} 提供灵活性，flatten 保证发布正确性，dependencyManagement 统一版本声明，scope=import 实现 BOM 导入，依赖仲裁规则 确保锁定版本不会被覆盖。

八、检验一下（提取练习）

Q1. flatten-maven-plugin 的 resolveCiFriendliesOnly 模式会替换哪些变量？

A. 所有 ${} 形式的变量 B. 仅 ${revision}、${sha1}、${changelist} C. 仅 ${project.version} D. ${revision} 和 ${java.version}
看答案B。resolveCiFriendliesOnly 是最保守的模式，只替换 Maven 定义的三个 CI Friendly 变量：${revision}、${sha1}、${changelist}。其它属性变量（如 ${spring-boot.version}）保持不变。

Q2. 如果不用 flatten 插件，直接发布含 ${revision} 的 POM 会怎样？

A. 正常使用，Maven 会自动查找 revision 属性 B. 下游消费者因找不到 ${revision} 定义而报错 C. Maven 自动用 1.0.0 作为默认版本 D. 只在 IDE 中有警告，不影响构建
看答案B。下游消费者的 POM 中没有定义 <revision> 属性，Maven 解析 ${revision} 时找不到值会直接报错。这就是 flatten 插件必须存在的原因——把变量替换为真实值。

Q3. flatten-maven-plugin 在哪个 Maven 生命周期阶段执行？

A. validate B. process-resources C. compile D. package
看答案B. process-resources。这个阶段在 compile 之前，确保编译时类路径中的版本号已经是真实值。clean 时有另一个 execution（flatten.clean）在 clean 阶段执行。

Q4. Maven 依赖仲裁中，如果两条依赖路径深度相同，哪个版本胜出？

A. 版本号更高的胜出 B. 版本号更低的胜出 C. 在 POM 中先声明的依赖所引入的版本胜出 D. 随机选择
看答案C。Maven 的仲裁规则是：路径深度相同时，"先声明优先"（first-declared-wins）。这也是为什么需要 BOM——因为依赖声明顺序容易被无意改动，BOM 在 depth=0 的 dependencyManagement 锁定版本可以完全绕过仲裁。

Q5. BOM 通过在什么位置锁定版本，从而绕过依赖仲裁的不确定性？

A. 在每个子模块的 dependencies 中 B. 在根 POM（或 parent POM）的 dependencyManagement 中（depth=0） C. 在 Maven settings.xml 中 D. 通过 maven-enforcer-plugin 强制检查
看答案B。dependencyManagement 中的版本声明位于依赖树的最顶层（depth=0），根据"距离优先"规则，任何传递依赖（depth >= 1）的版本都无法覆盖它。这就是 BOM 锁定版本的底层原理。
📖 推荐一手资料：(1) 阅读根 POM 中 flatten-maven-plugin 的完整配置——RuoYi-Vue-Plus/pom.xml 的 pluginManagement 段；(2) 观察 BOM 在本地仓库中的扁平化结果——执行 mvn clean install 后查看 ~/.m2/repository/org/dromara/ruoyi-common-bom/6.0.0-BETA/ruoyi-common-bom-6.0.0-BETA.pom，确认所有 ${revision} 已被替换为 6.0.0-BETA。
