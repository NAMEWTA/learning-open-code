第 1 课 · BOM 机制与 Maven 依赖管理 版本治理

本节目标：学完你能够解释什么是 BOM（Bill of Materials）、dependencyManagement 与 dependencies 的本质区别、scope=import 的工作原理，以及 RuoYi-Vue-Plus 中 ruoyi-common-bom 如何治理 23 个 common 子模块的版本。

在 RuoYi-Vue-Plus 项目中，打开 ruoyi-system/pom.xml，你会看到一段非常"奇怪"的依赖声明——没有版本号：

<!-- ruoyi-system/pom.xml 中声明 common-core 依赖 -->
<dependency>
    <groupId>org.dromara</groupId>
    <artifactId>ruoyi-common-core</artifactId>
    <!-- 没有 <version> 标签！ -->
</dependency>

那么版本从哪里来？答案就在 ruoyi-common-bom 中。这堂课我们从零开始理解 Maven BOM 机制。

一、什么是 BOM（Bill of Materials）

BOM 的全称是 Bill of Materials（物料清单），在 Maven 生态中特指一种特殊的 POM 文件，它不产出 JAR，唯一职责是集中管理一组依赖的版本号。

一个 BOM POM 有三个标志性特征：

特征说明
packaging=pom不打包为 JAR，这是 POM 类型项目的强制要求
只有 dependencyManagement不写 dependencies 块，不引入实际依赖
被其他 POM 以 scope=import 导入这是 BOM 唯一生效的方式（下面详讲）

打开真实的 ruoyi-common-bom/pom.xml，验证这三个特征：

<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0" ...>
    <modelVersion>4.0.0</modelVersion>

    <groupId>org.dromara</groupId>
    <artifactId>ruoyi-common-bom</artifactId>
    <version>${revision}</version>
    <packaging>pom</packaging>                <!-- 特征①: pom 类型 -->

    <dependencyManagement>                    <!-- 特征②: 只有 dependencyManagement -->
        <dependencies>
            <dependency>
                <groupId>org.dromara</groupId>
                <artifactId>ruoyi-common-core</artifactId>
                <version>${revision}</version>
            </dependency>
            <!-- ... 22 more common modules ... -->
        </dependencies>
    </dependencyManagement>

    <!-- 注意：没有任何 <modules> 标签，也没有 <dependencies> 标签 -->
</project>
关键发现：这个 BOM POM 中没有 <modules> 段，也没有 <dependencies> 段。它不是一个"父 POM"，而是一个版本目录（version catalog）——只为 23 个 ruoyi-common-* 模块声明版本号。

二、dependencyManagement vs dependencies — 本质区别

这是 Maven 初学者最容易混淆的概念。两者的区别可以浓缩为一句话：

dependencyManagement 声明版本但不引入 classpath；dependencies 引入 classpath 但版本可省略（由上层补充）。
特性dependencyManagementdependencies
会下载 JAR 吗？不会会
会加入 classpath 吗？不会会
子模块继承吗？是（声明优先于继承）是（直接继承所有依赖）
主要用途锁定版本、统一治理引入实际依赖
写在哪个 POM 中？父 POM 或 BOM任何需要该依赖的模块

用一个具体的例子对比：

<!-- 方式A: 在 dependencies 中声明（会实际引入） -->
<dependencies>
    <dependency>
        <groupId>org.dromara</groupId>
        <artifactId>ruoyi-common-core</artifactId>
        <version>6.0.0-BETA</version>
    </dependency>
</dependencies>
<!-- 效果: ruoyi-common-core.jar 进入 classpath -->

<!-- 方式B: 在 dependencyManagement 中声明（仅锁定版本） -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.dromara</groupId>
            <artifactId>ruoyi-common-core</artifactId>
            <version>6.0.0-BETA</version>
        </dependency>
    </dependencies>
</dependencyManagement>
<!-- 效果: 仅登记版本，classpath 不受影响 -->

实际使用时的标准模式是 "上管下用"：

<!-- 父 POM 或 BOM: 管版本 -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.dromara</groupId>
            <artifactId>ruoyi-common-core</artifactId>
            <version>6.0.0-BETA</version>
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- 子模块: 用依赖（不用写版本） -->
<dependencies>
    <dependency>
        <groupId>org.dromara</groupId>
        <artifactId>ruoyi-common-core</artifactId>
        <!-- 版本从父 POM 的 dependencyManagement 继承 -->
    </dependency>
</dependencies>

三、scope=import — BOM 生效的唯一方式

scope=import 是 Maven 2.0.9 引入的特殊作用域，只适用于 dependencyManagement 内部且 type=pom 的依赖。它的语义是：

将目标 POM 的 dependencyManagement 完整合并到当前 POM 的 dependencyManagement 中，就像把目标 POM 的版本声明全部"复制粘贴"过来一样。

在 RuoYi-Vue-Plus 的根 POM 中，你会看到这个关键配置：

<!-- RuoYi-Vue-Plus/pom.xml 根 POM 的 dependencyManagement -->
<dependencyManagement>
    <dependencies>
        <!-- Spring Boot BOM: 管理所有 Spring 生态的版本 -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>${spring-boot.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>

        <!-- RuoYi Common BOM: 管理所有 common 子模块的版本 -->
        <dependency>
            <groupId>org.dromara</groupId>
            <artifactId>ruoyi-common-bom</artifactId>
            <version>${revision}</version>
            <type>pom</type>
            <scope>import</scope>       <!-- 关键: 导入 BOM 的版本清单 -->
        </dependency>

        <!-- hutool BOM、sa-token BOM 等同样如此 -->
    </dependencies>
</dependencyManagement>

必须同时满足三个条件，scope=import 才能生效：

条件说明
<type>pom</type>被导入的工件必须是 POM 类型（因为 JAR 没有 dependencyManagement）
<scope>import</scope>必须显式声明 import 作用域
写在 dependencyManagement 内不能在 dependencies 段中使用 scope=import
常见错误：新手经常忘记写 <type>pom</type>，Maven 默认 type=jar，会导致找不到对应的 JAR 文件而报错。

四、packaging=pom 详解

在 Maven 中，packaging 决定项目的输出类型：

packaging 值输出典型用途
jar（默认）.jar普通 Java 库或应用
war.warWeb 应用
pom.pom（无 JAR）父 POM、聚合器 POM、BOM

一个 packaging=pom 的项目不会触发 maven-jar-plugin 等打包插件，只会产出 .pom 文件。这适用于三种角色：

  1. 父 POM — 被 parent 继承，提供公共配置
  2. 聚合器 POM — 通过 modules 控制多模块构建顺序
  3. BOM — 通过 dependencyManagement 提供版本目录

一个 POM 可以同时承担多种角色。例如 RuoYi-Vue-Plus 的根 POM 同时是：父 POM（定义公共配置）+ 聚合器 POM（定义 modules）+ BOM 导入者（import 多个 BOM）。

五、聚合器 POM vs BOM — 两种不同的"中心"

这是最容易混淆的对比。虽然两者都使用 packaging=pom，但它们的职责截然不同：

维度聚合器 POM（如 ruoyi-common/pom.xml）BOM（如 ruoyi-common-bom/pom.xml）
核心标签<modules><dependencyManagement>
解决的问题"先构建哪个？"（构建顺序）"用哪个版本？"（版本一致性）
影响范围同仓库内的子模块任何导入该 BOM 的项目（含下游消费者）
是否有 parent有（继承根 POM）无（独立工件）
可否独立发布通常不单独发布是，发布到 Maven 仓库供外部引用

来看 RuoYi-Vue-Plus 中的实际对比：

<!-- 聚合器: ruoyi-common/pom.xml — 控制构建顺序 -->
<project>
    <parent>
        <artifactId>ruoyi-vue-plus</artifactId>
        <groupId>org.dromara</groupId>
        <version>${revision}</version>
    </parent>
    <artifactId>ruoyi-common</artifactId>
    <packaging>pom</packaging>

    <modules>                          <!-- 聚合器的核心 -->
        <module>ruoyi-common-bom</module>
        <module>ruoyi-common-core</module>
        <module>ruoyi-common-redis</module>
        <module>ruoyi-common-sms</module>
        <!-- ... 21 more ... -->
    </modules>
</project>

<!-- BOM: ruoyi-common-bom/pom.xml — 管理版本 -->
<project>
    <!-- 注意: 没有 <parent> 标签, 独立 POM -->
    <groupId>org.dromara</groupId>
    <artifactId>ruoyi-common-bom</artifactId>
    <version>${revision}</version>
    <packaging>pom</packaging>

    <dependencyManagement>             <!-- BOM 的核心 -->
        <dependencies>
            <dependency>...ruoyi-common-core...</dependency>
            <!-- ... 22 more version declarations ... -->
        </dependencies>
    </dependencyManagement>
</project>
设计洞察：BOM 没有 <parent> 标签——它是一个独立工件，可以单独发布到 Maven 中央仓库。groupId=org.dromara，artifactId=ruoyi-common-bom。这样，即使是外部项目（不属于当前多模块构建），也可以通过 scope=import 直接导入 ruoyi-common-bom，获得所有 common 模块的统一版本。这就是 BOM 作为"独立版本目录"的设计价值——它不依赖任何父 POM 继承链。

六、BOM 管理 23 个 common 子模块全景

ruoyi-common-bom/pom.xml 在 dependencyManagement 中声明了 23 个 ruoyi-common-* 子模块：

ruoyi-common-bom 管理的依赖清单（dependencyManagement 内） ┌── 核心层 ────────────────────────── │ ruoyi-common-core ← 所有模块的基石 (R/异常/工具) │ ruoyi-common-web ← Web 层增强 │ ruoyi-common-doc ← 接口文档（SpringDoc） │ ├── 数据层 ────────────────────────── │ ruoyi-common-mybatis ← 数据库 & MyBatis-Plus │ ruoyi-common-elasticsearch ← ES 搜索引擎 │ ruoyi-common-redis ← Redis 缓存 & 分布式锁 │ ├── 安全与认证 ────────────────────── │ ruoyi-common-security ← 安全框架集成 │ ruoyi-common-satoken ← Sa-Token 权限认证 │ ruoyi-common-encrypt ← 数据库字段加解密 │ ruoyi-common-sensitive ← 数据脱敏 │ ├── 业务能力 ──────────────────────── │ ruoyi-common-excel ← Excel 导入导出 │ ruoyi-common-oss ← 对象存储 (OSS) │ ruoyi-common-sms ← 短信服务 │ ruoyi-common-mail ← 邮件服务 │ ruoyi-common-job ← 定时任务调度 │ ruoyi-common-log ← 日志记录 │ ruoyi-common-social ← 第三方社交登录 │ ruoyi-common-translation ← 数据翻译 (字典/枚举) │ ruoyi-common-push ← 消息推送 │ ├── 基础设施 ──────────────────────── │ ruoyi-common-json ← JSON 序列化 │ ruoyi-common-mqtt ← MQTT 物联网协议 │ ruoyi-common-ai ← AI 集成 │ ruoyi-common-mcp ← MCP 协议支持

GAV 坐标：所有这些依赖共享相同的 groupId=org.dromara 和 version=${revision}，仅 artifactId 不同。这是多模块项目的标准约定——同源同版。

七、为什么 BOM 不写 modules？

你可能会问：BOM 声明了 23 个模块的版本，为什么不在 BOM 里写 <modules> 呢？

答案是：构建顺序由聚合器 POM 控制，BOM 只负责版本声明。 在 RuoYi-Vue-Plus 中，构建顺序的职责属于这三个聚合器：

RuoYi-Vue-Plus/pom.xml（根聚合器）
  └── ruoyi-common/pom.xml（common 聚合器）
        ├── ruoyi-common-bom        ← BOM 本身先构建（发布 .pom）
        ├── ruoyi-common-core       ← 然后构建 core（被其它模块依赖）
        ├── ruoyi-common-redis
        ├── ruoyi-common-sms
        └── ... (22 more)

BOM 的 dependencyManagement 只是一个版本声明表，它不关心这些模块实际何时构建、由谁构建。这种关注点分离让 BOM 可以独立发布，被外部项目引用时不需要引入任何构建逻辑。

设计洞察：分离 BOM 与聚合器满足了两种不同的消费者——内部构建者（需要 modules 控制顺序）和外部使用者（只需要 dependencyManagement 锁定版本）。BOM 作为独立工件发布后，外部项目只需一行 scope=import 就能获得所有 common 模块的版本治理，完全不依赖 RuoYi 的内部构建结构。

八、检验一下（提取练习）

Q1. dependencyManagement 中声明的依赖会不会出现在编译 classpath 中？

A. 会，只要是 dependencyManagement 里的都会引入 B. 不会，它只声明版本，不实际引入依赖 C. 取决于 scope 属性的值 D. 如果 scope=compile 就会引入
看答案B. 不会。 dependencyManagement 只是版本声明表，实际依赖由 dependencies 段引入。子模块必须在自己的 dependencies 中显式声明需要的依赖（可以省略 version），才会实际加入 classpath。

Q2. 以下哪个是 scope=import 的正确用法？

A. 在 <dependencies> 中导入 BOM B. 在 <dependencyManagement> 中，且必须配合 <type>pom</type> C. 在任何 dependency 标签中都可以使用 D. 只能用于 Spring Boot 的 BOM
看答案B。scope=import 必须同时满足三个条件：(1) 写在 dependencyManagement 内；(2) type=pom；(3) 目标工件的 packaging=pom。缺一不可。

Q3. 聚合器 POM 和 BOM 的最核心区别是什么？

A. 没有区别，都是 packaging=pom B. 聚合器用 <modules> 控制构建顺序，BOM 用 <dependencyManagement> 管理版本 C. BOM 必须有 parent，聚合器不能有 parent D. 聚合器不能独立发布，BOM 可以
看答案B。两者都使用 packaging=pom，但核心标签不同：聚合器靠 <modules> 控制"先构建谁"，BOM 靠 <dependencyManagement> 控制"用哪个版本"。实际项目中 BOM 通常独立（无 parent），可单独发布。但"能否独立发布"只是结果，不是本质区别。

Q4. 子模块 ruoyi-system 声明依赖 ruoyi-common-core 时不写 <version>，版本从哪里来？

A. Maven 自动使用最新版本 B. 从 ruoyi-system 的 parent 继承链 → 根 POM → scope=import → ruoyi-common-bom C. 从 Maven 中央仓库查最新稳定版 D. 构建时会报错，必须写版本号
看答案B。完整链路：ruoyi-system 继承 → ruoyi-modules → 根 POM（import ruoyi-common-bom）→ BOM 中声明了 ruoyi-common-core 的 version=6.0.0-BETA → 版本最终由 BOM 提供。
📖 推荐一手资料：对比阅读三个文件——(1) RuoYi-Vue-Plus/ruoyi-common/ruoyi-common-bom/pom.xml（纯 BOM，只有 dependencyManagement，无 parent 无 modules）；(2) RuoYi-Vue-Plus/ruoyi-common/pom.xml（聚合器，只有 modules）；(3) RuoYi-Vue-Plus/pom.xml 的 dependencyManagement 段（通过 scope=import 导入 BOM）。三个文件对照读，BOM 机制一目了然。
