一句话定位

BOM (Bill of Materials) 是 Maven 多模块项目的统一版本管理中心——通过 dependencyManagement 集中声明版本，通过 scope=import 传播给所有子模块，实现「一处声明，全局生效」。

BOM 定义 vs 传统方式

传统方式（无 BOM）BOM 方式
版本定义22 个子 POM 各写各的 versionBOM 的 properties 一处定义 ${revision}
版本升级逐个改 22 个 POM改 BOM 中一个数字
版本一致性依赖人工检查，易分裂自动统一，仲裁机制保证
可复用性只能被继承该 POM 的子模块用任何项目都可以 scope=import

三大核心机制

机制作用关键参数/位置
dependencyManagement声明版本，不引入 JARBOM 和父 POM 的 <dependencyManagement> 区块
${revision} + flatten用变量写版本，打包时替换flattenMode=resolveCiFriendliesOnly，在 process-resources 阶段执行
scope=import把 BOM 的版本声明合并到当前 POM父 POM 的 <dependency> 中 <scope>import</scope> + <type>pom</type>

dependencyManagement vs dependencies

dependencyManagementdependencies
是否引入 JAR❌ 否，仅声明版本✅ 是，加入 classpath
子模块需再声明✅ 是，在 dependencies 中引用❌ 否，自动传递
典型用途统一版本治理声明直接依赖

flatten-maven-plugin 工作流

开发时: <version>${revision}</version> → process-resources 阶段: flatten 插件替换变量 → .flattened-pom.xml: <version>6.0.0-BETA</version> → mvn install/deploy: 发布含真实版本号的 POM → 下游消费者: 看到的是 6.0.0-BETA（无变量）✓

记忆锚点

  • BOM 是「采购标准手册」，不是「库存」——声明了 22 个依赖的版本，但一个都不实际引入。
  • dependencyManagement = 版本声明（不给 classpath），dependencies = 实际引入（给 classpath）。
  • ${revision} 是开发时写法，flatten-maven-plugin 保证发布时变量消失。
  • scope=import = 把别人 BOM 的版本声明「复制粘贴」到我的 POM。
