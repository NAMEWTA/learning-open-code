第 1 课 · ruoyi-ai 模块全景：可拔插 AI 的三层架构 全景地图

🎯 本节目标（对应使命 Success #1~#2）：学完你能一句话讲清 ruoyi-ai 模块的本质，能画出它的三层物理架构，理解它为什么是「适配器」而非「AI 实现」。

ruoyi-ai 是整个 RuoYi-Vue-Plus 中最「空」的业务模块——只有 2 个文件、控制器仅 73 行 Java 代码。但它承载的能力却是最「重」的：一个完整的 AI 智能体平台。这种反差本身就是设计信号：模块的职责不是实现 AI，而是集成 AI。这一课我们先建立全景认知，看清它的骨架。

一、物理文件结构：为什么这么少？

ruoyi-modules/ruoyi-ai/ ├── pom.xml ← 依赖声明（5 个内部模块） └── src/main/java/org/dromara/ai/ └── controller/ └── SnailAiController.java ← 73 行，唯一 Java 文件 ruoyi-common/ruoyi-common-ai/ ├── pom.xml ← 依赖 snail-ai SDK └── src/main/java/org/dromara/common/ai/ ├── config/ │ └── SnailAiConfig.java ← 16 行，自动配置 └── handler/ └── SnailAiChatExceptionHandler.java ← 125 行，异常处理 plus-ui-react/src/ ├── api/ai/agent/ │ ├── index.ts ← 290 行，API 调用 + SSE 流式 │ └── types.ts ← TypeScript 类型定义 ├── pages/ai/chat/ │ └── index.tsx ← 530 行，完整 AI Chat 页面 └── assets/styles/modules/ └── ai-chat.less ← 220 行，Chat 页面样式 plus-ui-vue/src/ ├── api/ai/agent/ ← 同 React 结构 ├── views/ai/chat/ │ └── index.vue ← 82 行，iframe 嵌入 └── views/monitor/snailai/ └── index.vue ← 21 行，管理后台 iframe

关键数字：后端仅 3 个 Java 文件（含 common-ai），总代码量约 210 行。但前端 React 端有 4 个文件约 1040 行，Vue 端 2 个页面约 100 行。后端做的是「接线」，前端做的是「体验」——这是 SDK 集成模式的典型特征。

二、一句话本质：它不是 AI 实现，是 AI 适配器

ruoyi-ai 不实现任何 AI 能力。它是 Snail AI SDK 的 Spring Boot 适配器——通过自动配置、条件装配和双异常处理体系，将外部 AI 智能体平台「嵌入」RuoYi-Vue-Plus。

对比其他 ruoyi-modules 模块：

模块核心职责代码量是否自实现核心能力
ruoyi-system系统管理（用户/角色/菜单）大量✅ 自己实现
ruoyi-workflow工作流引擎集成大量❌ 集成 WarmFlow
ruoyi-aiAI 智能体集成极少❌ 集成 Snail AI SDK
ruoyi-demo示例代码少—

注意 ruoyi-workflow 和 ruoyi-ai 的相似性：两者都是外部引擎的集成适配层。但 workflow 需要定义流程模型、办理人逻辑、事件监听等大量适配代码，而 AI 模块因为有 Snail AI SDK 的完整封装，适配代码极少。这就是 SDK 厚度决定适配层厚度 的规律。

三、三层物理架构

🖥️ 前端层
React: @ant-design/x 原生组件 · SSE 流式
Vue: iframe 嵌入 · 免登 URL
↓ HTTP / SSE / iframe
⚙️ 业务模块层 · ruoyi-modules/ruoyi-ai
SnailAiController: 用户身份桥接 REST 端点
↓ 依赖注入
🔧 通用配置层 · ruoyi-common/ruoyi-common-ai
SnailAiConfig: @EnableSnailAiAgent + @EnableSnailAiOpenApi
SnailAiChatExceptionHandler: AI Chat 独立异常处理
↓ SDK 调用
📦 Snail AI SDK
com.aizuda.snail.ai.* — Snail AI Starter 自动配置
↓ gRPC
🖧 Snail AI Server
独立的 AI 智能体平台（AI Agent 调度、模型管理、知识库）

每一层的职责边界（从下往上）：

1Snail AI Server：独立的 AI 平台进程，提供 gRPC 接口。管理智能体、调度模型、处理对话。ruoyi-ai 模块完全不知道它的内部实现。
不属于本仓库——通过 gRPC 连接
2Snail AI SDK（Starter）：Java 客户端 SDK。提供 @EnableSnailAiAgent、@EnableSnailAiOpenApi 注解和 OpenApiUserClient 等 API。封装了 gRPC 通信细节。
com.aizuda.snail.ai.* — Maven 依赖引入
3ruoyi-common-ai（通用配置层）：将 SDK 适配到 Spring Boot 自动配置体系，提供 AI Chat 专属异常处理。这是「Spring Boot 集成」的关键层。
org.dromara.common.ai.config / handler
4ruoyi-modules/ruoyi-ai（业务模块层）：暴露 REST 端点给前端，处理 RuoYi 用户到 Snail AI 用户的身份映射。这是「RuoYi 业务集成」的关键层。
org.dromara.ai.controller
5前端层：React 端通过 @ant-design/x 组件库 + SSE 流式实现了原生对话体验；Vue 端通过 iframe 嵌入 Snail AI 的独立前端页面。
plus-ui-react / plus-ui-vue
💡 关键洞察：为什么分成 ruoyi-ai（业务模块）和 ruoyi-common-ai（通用配置）两层？
ruoyi-common-ai 是框架层配置——任何模块（system、workflow 等）都可以依赖它来获得 AI 能力；ruoyi-modules/ruoyi-ai 是业务层端点——它依赖 ruoyi-common-satoken 和 ruoyi-common-web 来接入 RuoYi 的认证体系和 Web 基础设施。这种分层让「AI 能力」和「AI 业务端点」解耦。

四、核心设计哲学：可拔插（Pluggable）

ruoyi-ai 模块最重要的设计特征是可拔插。注意 application-dev.yml 中的配置：

snail-ai:
  enabled: false        # ← 默认关闭！

当 snail-ai.enabled=false 时：

组件条件装配注解disabled 时的行为
SnailAiConfig@ConditionalOnProperty(prefix="snail-ai", name="enabled", havingValue="true")不加载 → @EnableSnailAiAgent 和 @EnableSnailAiOpenApi 均无效 → 整个 SDK 不初始化
SnailAiController@ConditionalOnProperty(prefix="snail-ai.open-api", name="enabled", havingValue="true")不注册 → /snail-ai/user/register 路由不存在 → 前端调用 404
⚠️ 注意两层条件装配的差异：
SnailAiConfig 的条件是 snail-ai.enabled=true——控制整个 AI 能力是否激活。
SnailAiController 的条件是 snail-ai.open-api.enabled=true——在 AI 已激活的前提下，进一步控制用户注册端点是否暴露。
这意味着你可以开启 AI 对话能力但关闭用户注册端点（例如在内网环境中通过其他方式管理用户）。

五、与 ruoyi-api 的关系：AI 模块也消费 API 契约

在 ruoyi-api 课程第 1 课的依赖表中，我们看到 ruoyi-ai 依赖 ruoyi-api。它消费了哪些 API？

→间接消费 LoginUser：SnailAiController 通过 LoginHelper.getLoginUser() 获取当前用户，而 LoginUser 定义在 ruoyi-system-api 中。AI 模块需要知道「当前用户是谁」才能注册到 Snail AI。

这种消费是极轻量的——AI 模块只用了 api 模块中的 LoginUser 模型，不调用任何 Service 接口。这符合它的定位：它是 AI 平台的入口，不是业务数据的消费者。

六、检验一下（提取练习）

Q1. ruoyi-ai 模块有几个 Java 文件？

A. 1 个（SnailAiController.java） B. 2 个（pom.xml + SnailAiController.java） C. 3 个（含 ruoyi-common-ai 的 SnailAiConfig 和 SnailAiChatExceptionHandler） D. 5 个以上
看答案C。注意问题的措辞——ruoyi-ai 模块本身只有 1 个 Java 文件（SnailAiController），但加上 ruoyi-common-ai 的 2 个文件（SnailAiConfig、SnailAiChatExceptionHandler），整个 AI 能力的后端代码是 3 个 Java 文件。

Q2. ruoyi-ai 的本质是什么？

A. 一个自研的 AI 对话引擎 B. Snail AI SDK 的 Spring Boot 适配器——通过自动配置、条件装配将外部 AI 平台嵌入 RuoYi C. 一个 LLM 模型的 Java 封装 D. 一个基于 WebSocket 的聊天服务
看答案B。ruoyi-ai 的代码只有用户注册桥接 + 配置激活 + 异常处理，没有任何 AI 算法或模型调用代码。所有 AI 能力来自 Snail AI SDK 和 Snail AI Server。

Q3. 为什么 ruoyi-ai 的代码这么少？

A. 模块还未开发完成 B. Snail AI SDK 封装了所有复杂逻辑，ruoyi-ai 只需要做 Spring Boot 适配和身份桥接 C. AI 功能由前端直接实现，后端不需要代码 D. 作者偷懒
看答案B。这是 SDK 集成模式的典型特征——SDK 越厚，适配层越薄。ruoyi-ai 的职责是「接线」：把 Snail AI SDK 接入 Spring Boot 自动配置体系、把 RuoYi 用户映射到 Snail AI 用户、把 RuoYi 异常体系适配到 AI Chat 需要的响应格式。

Q4. SnailAiConfig 和 SnailAiController 的 @ConditionalOnProperty 有什么区别？

A. 没有区别，两者条件相同 B. Config 控制整个 AI 能力开关（snail-ai.enabled）；Controller 在 AI 已激活的前提下进一步控制用户注册端点（snail-ai.open-api.enabled） C. Config 使用的是 Java 17 语法，Controller 使用的是 Java 8 D. Config 是生产环境专用，Controller 是开发环境专用
看答案B。这是两层条件装配的精细控制：SnailAiConfig 是总开关，关闭后整个 AI SDK 不初始化；SnailAiController 是子功能开关，允许在 AI 已激活的情况下关闭用户注册端点（例如使用其他用户管理方式）。
📖 推荐一手资料：打开 RuoYi-Vue-Plus/ruoyi-modules/ruoyi-ai/pom.xml 和 RuoYi-Vue-Plus/ruoyi-common/ruoyi-common-ai/pom.xml，对比两者的依赖差异——前者依赖 5 个 ruoyi 内部模块但没有任何 snail-ai 依赖，后者依赖 snail-ai SDK。这个对比是理解「分层」的最佳入口。
