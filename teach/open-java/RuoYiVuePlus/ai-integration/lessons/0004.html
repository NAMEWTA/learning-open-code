第 4 课 · 双前端集成路线对比 React vs Vue

🎯 本节目标（对应使命 Success #7~#8）：学完你能解释 React 端和 Vue 端为何走向两条完全不同的技术路线，能讲清 React 端 SSE 流式对话的完整组件架构，能解释 Vue 端 iframe 嵌入模式中 trustedCredential 免登机制的安全模型。

这是 ruoyi-ai 模块中最令人困惑的设计决策：同一个后端，为什么 React 端和 Vue 端的前端实现完全不同——一个是 530 行原生组件 + SSE 流式，另一个是 82 行 iframe 嵌入？这一课我们在对比中理解两种路线的取舍。

2.3 关键设计：normalize 系列函数

React 端有 4 个 normalizeXxx 函数（normalizeAgentList、normalizeConversationList、normalizeMessageList、normalizeStreamChunk）。它们存在的共同原因是：

Snail AI SDK 的返回格式可能因版本不同而略有差异——有时返回 {data: [...]}（RuoYi 格式封装），有时返回 {rows: [...]}（分页格式），有时直接返回数组。normalize 函数通过 getList() 统一处理这些变体，确保了前端对后端返回格式的鲁棒性。 
function getList<T>(payload: unknown): T[] {
  const container = payload?.data ?? payload;
  if (Array.isArray(container)) return container;
  const record = container as { rows?: T[]; list?: T[]; records?: T[] };
  return record?.rows ?? record?.list ?? record?.records ?? [];
}
💡 设计洞察：防御性数据提取
getList() 是一个「乐观提取器」——它不抛出异常，而是按优先级尝试多种数据路径（data → rows → list → records），都不匹配就返回空数组。这保证了即使后端返回格式变化，前端也不会崩溃，只是显示空列表。这是 SDK 集成场景下的常见策略——你无法控制 SDK 的返回格式演进。

三、Vue 端：82 行的 iframe 嵌入模式

3.1 实现全景

Vue 端的 AI Chat 仅 82 行（含样式），核心逻辑不到 30 行：

async function loadChat() {
  const token = getToken();                          // 1. 获取 RuoYi 登录 token
  const { data: user } = await registerCurrentSnailUser(); // 2. 注册/获取 Snail AI 用户
  chatUrl.value = buildChatUrl(user.openId, token);  // 3. 拼接嵌入 URL
}

function buildChatUrl(openId: string, trustedCredential: string) {
  return `${baseUrl}/snail-chat/?openId=${openId}&trustedCredential=${trustedCredential}`;
}

// 模板：<iframe :src="chatUrl" />

3.2 为什么 Vue 端选择 iframe？

这不是技术能力问题，而是资源分配决策：

1Snail AI 自带完整前端：Snail AI Server 提供了独立的内建 Chat UI（通过 application.yml 中放行 /snail-chat/** 路由可见）。这个 UI 由 Snail AI 团队维护，包含了智能体管理、知识库、工具调用等完整功能。
2iframe 嵌入是零维护成本的集成：不需要像 React 端那样实现智能体列表管理、对话历史管理、SSE 解析等复杂逻辑。Snail AI 前端更新时，Vue 端自动获得新功能——无需修改任何代码。
3Vue 端还多了一个管理后台入口：views/monitor/snailai/index.vue（21 行）通过 VITE_APP_SNAILAI_ADMIN 环境变量嵌入 Snail AI 的管理后台，这是 React 端没有的。

3.3 trustedCredential 免登机制

Vue 端的 URL 中有两个关键参数：

/snail-chat/?openId=xxx&trustedCredential=yyy
参数来源作用
openIdregisterCurrentSnailUser() 返回的 user.openId标识 Snail AI 中的用户身份
trustedCredentialRuoYi 的登录 token（Sa-Token）Snail AI 内嵌页面用此 token 向 RuoYi 后端发起 API 请求时携带认证
⚠️ 安全警示：trustedCredential = RuoYi 的完整登录 token。这意味着任何拥有此 URL 的人都能以该用户身份调用 RuoYi API。iframe 嵌入模式下需要确保：(1) 页面使用 HTTPS；(2) iframe 的 allow 属性限制了敏感操作（当前代码中仅开放了 clipboard-read; clipboard-write）；(3) Snail AI Server 端验证了 token 的有效性和来源。

3.4 嵌入模式服务端配置

回看 application-dev.yml 中的配置：

chat.ui.embed:
  enabled: true           # 启用嵌入模式
  show-header: false      # 隐藏 Snail AI 的顶部栏（RuoYi 已有自己的导航）
  show-sidebar-user: false # 隐藏侧边栏用户信息（RuoYi 已显示用户）
  show-agent-market: false # 隐藏智能体市场（不需要）
  compact-input: true     # 使用紧凑输入框（嵌入在 RuoYi 页面内空间有限）
  lock-agent: false       # 不锁定智能体（用户可以切换）

这些配置是专门为 iframe 嵌入场景设计的——目标是让 Snail AI 的 UI 尽可能像 RuoYi 的原生功能，而不是一个独立应用。

四、两种路线的取舍总结

选择 React 原生组件路线，当...选择 Vue iframe 嵌入路线，当...
需要对 UI 有完全控制权（与 RuoYi 风格一致）希望零维护成本集成
愿意承担前端开发维护成本接受 Snail AI 内建 UI 的设计风格
需要自定义 AI 交互体验（如预设问题、自定义气泡）不需要自定义交互体验
团队有 React / @ant-design/x 技术栈团队使用 Vue，不想引入 React 依赖
需要精细控制认证和授权流程信任 Snail AI 内建的安全机制
💡 最佳实践建议：两种路线并非互斥。在真实项目中，你可以像 Vue 端那样用 iframe 快速上线，后续如果需要更精细的 UI 控制，再像 React 端那样实现原生组件。关键是理解 registerCurrentSnailUser() 提供的 openId 和 trustedCredential 是两个路线的共同基础——无论哪种方式，身份桥接是必需的。

五、检验一下（提取练习）

Q1. React 端的 normalizeAgentList() 等 normalize 函数存在的根本原因是什么？

A. 代码作者喜欢函数式编程 B. Snail AI SDK 的返回格式可能因版本不同而变化（data/rows/list/records），normalize 函数确保前端对格式变化具有鲁棒性 C. TypeScript 类型安全检查要求 D. 为了提高代码复用性
看答案B。getList() 函数按优先级尝试多种数据路径，确保无论 SDK 返回哪种分页/列表格式，前端都能正确提取数据。这是在 SDK 集成场景下的防御性编程实践。

Q2. Vue 端的 trustedCredential 参数是什么？

A. 一个随机生成的加密字符串 B. RuoYi 的登录 token（Sa-Token），用于 Snail AI 内嵌页面调用 RuoYi API 时的认证 C. Snail AI 的 API Key D. 用户的密码哈希
看答案B。trustedCredential = getToken() 返回的 RuoYi 登录 token。它被放在 URL 中传递给 iframe 内的 Snail AI 页面，使得内嵌页面可以代表当前用户调用 RuoYi 后端 API。

Q3. 为什么 embed 配置中要设置 show-header: false 和 show-sidebar-user: false？

A. 为了节省服务器带宽 B. 因为 RuoYi 已经有自己的导航栏和用户信息显示，Snail AI 的内建 UI 元素会造成重复和视觉冲突 C. 这些功能在开发环境中不可用 D. 为了遵守 GDPR 隐私法规
看答案B。iframe 嵌入的目标是让 Snail AI 看起来像 RuoYi 的原生功能。隐藏 Snail AI 自己的导航和用户信息，可以避免与 RuoYi 的导航重复，提供一致的视觉体验。
