第 3 课 · 异常处理双轨制与 SSE 流式对话 异常 & 流式

🎯 本节目标（对应使命 Success #6~#7）：学完你能解释为什么 AI Chat 接口需要独立的异常处理器，能讲清 Result<Void> 与 R<*> 双响应格式的设计动机，能追踪 React 端 SSE 流式对话的完整数据流。

上一课我们看到了自动配置和用户身份桥接。这一课聚焦两个更深的设计问题：(1) 为什么 AI Chat 需要独立的异常处理体系？(2) 前端 SSE 流式对话的数据流是如何工作的？

一、双响应格式问题：Result vs R

先看一个有趣的事实。在 RuoYi-Vue-Plus 中，所有标准 REST 接口的统一响应格式是：

// RuoYi 统一响应格式
public class R<T> {
    private int code;    // 状态码
    private String msg;  // 消息
    private T data;      // 数据
}

但 SnailAiChatExceptionHandler 返回的是 Snail AI SDK 的响应格式：

// Snail AI SDK 响应格式
public class Result<T> {
    private int status;   // 状态码（注意不是 code）
    private String message; // 消息（注意不是 msg）
    private T data;        // 数据
}

这两个格式不兼容。为什么不能统一？

⚠️ 核心矛盾：/api/snail/chat/** 路径的请求来自 Snail AI 自己的前端 SDK，不是 RuoYi 的前端。Snail AI 前端 SDK 期望的是 {status, message, data}，如果收到 {code, msg, data} 会解析失败。同时，/snail-ai/** 路径（即 SnailAiController 的端点）的请求来自 RuoYi 前端，需要 R<*> 格式。

这就产生了 ruoyi-ai 模块中的异常处理双轨制：

请求路径消费者期望响应格式异常处理器
/api/snail/chat/**Snail AI 前端 SDK（由 SDK 内置的 SnailAiChatGatewayController 处理）Result{status, message, data}SnailAiChatExceptionHandler
/snail-ai/**RuoYi 前端（React/Vue）R{code, msg, data}RuoYi 全局异常处理器

二、SnailAiChatExceptionHandler 深度解析

1限定作用范围
@RestControllerAdvice(assignableTypes = SnailAiChatGatewayController.class)
这不是一个全局异常处理器——它只处理 SnailAiChatGatewayController（Snail AI SDK 提供的控制器）抛出的异常。其他控制器的异常继续由 RuoYi 的全局异常处理器处理。
关键：assignableTypes 限定范围
2最高优先级
@Order(Ordered.HIGHEST_PRECEDENCE)
确保在 Spring 的异常处理器链中，这个处理器先于 RuoYi 的全局处理器执行。如果优先级不够高，RuoYi 的处理器可能先拦截异常并返回 R 格式，破坏了前端 SDK 的预期。
3异常分类处理
按异常类型分层处理，共 8 个 @ExceptionHandler 方法：
异常类型HTTP 层含义返回消息策略
SnailAiAuthenticationException认证失败（token 过期/无效）status=5001，提示重新登录
BaseSnailAiExceptionSnail AI 通用业务异常返回异常消息或默认提示
ModelCallExceptionAI 模型调用失败返回异常消息或默认提示
BindException 等请求参数校验失败提取校验错误消息
ConstraintViolationException方法级参数校验失败拼接所有约束冲突消息
HttpMessageNotReadableException请求体 JSON 解析失败固定提示"格式错误"
IllegalArgumentException参数不合法返回异常消息或默认提示
IllegalStateException会话状态异常建议刷新后重试
Exception（兜底）未预期的异常通用提示，打 error 日志
💡 关键洞察：defaultMessage() 的防御性设计
注意 defaultMessage(e, fallback) 方法：return hasText(e.getMessage()) ? e.getMessage() : fallback;
这是因为异常消息可能为 null（如 NullPointerException 没有 message）。如果直接返回 null，前端 SDK 会收到 {"status":0,"message":null,"data":null}——用户体验很差。fallback 参数提供了友好的中文默认提示。

三、与 RuoYi 全局异常处理器的对比

四、SSE 流式对话：React 端的数据流

React 端的 fetchAgentChat() 函数（plus-ui-react/src/api/ai/agent/index.ts）实现了一个完整的 SSE（Server-Sent Events）客户端。让我们追踪数据流的每一步：

1发起请求
fetch(`${baseURL}/snail-ai/agent/${agentId}/chat/stream`, {
  method: 'POST',
  headers: {
    ...globalHeaders(),        // RuoYi 认证头（包含 token）
    'Content-Language': language,
    Accept: 'text/event-stream', // ← 告诉服务器返回 SSE 流
    'Content-Type': 'application/json;charset=utf-8'
  },
  body: JSON.stringify(data),
  signal: options.signal       // 支持 AbortController 取消
})
2建立 ReadableStream
const reader = response.body?.getReader();
浏览器以流式方式读取服务器响应的字节块——不会等待完整响应。每个 reader.read() 返回一个 {done, value} 块。
3SSE 事件解析
// SSE 协议的原始格式
event: thinking\ndata: {"content":"让我想想..."}\n\n
event: message\ndata: {"content":"你好！"}\n\n
event: done\ndata: \n\n
event: error\ndata: 错误消息\n\n
解析逻辑：
• buffer.split(/\r?\n\r?\n/) —— 按双换行分割事件块（SSE 协议的事件分隔符）
• 对每个事件块，解析 event: 和 data: 行
• event: thinking → onThinking(chunk)（思考过程）
• event: message → onMessage(chunk)（对话内容）
• event: done → onDone()（流结束）
• event: error → onError(error)（流异常）
4JSON 解析与规范化和
function normalizeStreamChunk(chunk: string) {
  // 尝试单对象解析：{"content":"你好"}
  // 若失败，按 }\\n{ 分割多对象数组
  // 合并所有 content 字段
}
这是对 Snail AI 返回格式的兼容处理——SDK 可能返回单个 JSON 对象，也可能返回多个 JSON 对象的拼接。
5React 状态增量更新
onMessage(chunk) {
  setMessages(items => {
    const next = [...items];
    const lastIndex = next.length - 1;
    next[lastIndex] = {
      ...next[lastIndex],
      content: next[lastIndex].content + normalizeStreamChunk(chunk)
    };
    return next;
  });
}
每次收到新的 chunk，不是替换整个消息，而是追加到当前 AI 消息的 content 末尾。这就是「逐字输出」效果的实现原理。
💡 关键洞察：为什么不用 WebSocket？
SSE 比 WebSocket 更适合 AI 对话场景：(1) AI 对话是单向流（服务器→客户端），不需要双向通信；(2) SSE 使用标准 HTTP，天然支持 HTTP/2 多路复用、代理穿透、认证头；(3) SSE 内置自动重连机制；(4) 实现更简单——不需要升级协议。

五、流完成后的数据刷新

注意 fetchAgentChat 的 onDone 回调中还有一步关键操作：

async onDone() {
  finishSending();
  await refreshConversationData(currentAgent.id, targetConversationId);
}

这调用了 queryClient.invalidateQueries() 刷新对话列表和消息列表。为什么流已经完成了还要重新拉数据？

因为 SSE 流只传递增量内容（chunks），而服务器端可能对消息做了后处理（如保存到数据库、关联元数据、生成消息 ID）。流结束后重新拉取一次完整数据，确保前端状态与服务器端一致。

六、同步模式作为降级方案

除了 SSE 流式，前端还支持同步模式（fetchAgentChatSync）：

// 同步模式：一次请求-响应
const res = await fetchAgentChatSync(agentId, { conversationId, content });
// 从响应中提取 AI 回复
const reply = extractSyncReply(res.data);

同步模式的选择由服务器端配置决定（fetchChatMode() 返回 mode: 'stream' | 'sync'）。当网络环境不支持 SSE（如某些代理服务器会缓冲响应）时，可切换到同步模式。

七、检验一下（提取练习）

Q1. 为什么 SnailAiChatExceptionHandler 返回 Result{status, message, data} 而不是 R{code, msg, data}？

A. 作者更喜欢 Result 的命名 B. 因为 /api/snail/chat/** 的消费者是 Snail AI 前端 SDK，它期望的是 Result 格式；返回 R 格式会解析失败 C. Result 是 Java 17 的 record，性能更好 D. RuoYi 的 R 类不支持 Void 泛型
看答案B。这是兼容性驱动的设计。/api/snail/chat/** 由 Snail AI SDK 的 SnailAiChatGatewayController 处理，它的消费者是 Snail AI 自己的前端 SDK。这个 SDK 不认识 RuoYi 的 R 格式。

Q2. @Order(Ordered.HIGHEST_PRECEDENCE) 在 SnailAiChatExceptionHandler 上的作用是什么？

A. 让这个处理器最先被加载 B. 确保在异常处理器链中，这个处理器先于 RuoYi 的全局处理器执行，避免异常被错误地转换为 R 格式 C. 提高异常处理的性能 D. 让日志输出更优先
看答案B。Spring 支持多个 @RestControllerAdvice，它们按 @Order 优先级排序。如果不设置最高优先级，RuoYi 的全局异常处理器可能先匹配异常并返回 R 格式，Snail AI 前端 SDK 就会收到错误格式的响应。

Q3. SSE 流式对话中，前端如何实现「逐字输出」效果？

A. 服务器每次发送一个字符，前端每次渲染一个字符 B. 服务器发送文本块（chunks），前端通过 React setState 函数式更新将每个 chunk 追加到消息末尾 C. 前端使用 CSS animation 模拟打字效果 D. 前端使用 Web Worker 在后台处理
看答案B。onMessage 回调中的 setMessages(items => { ... next[lastIndex].content + normalizeStreamChunk(chunk) }) 是关键——每次收到新的 chunk，通过函数式 setState 追加到当前 AI 消息的 content 末尾，React 自动触发重新渲染。
