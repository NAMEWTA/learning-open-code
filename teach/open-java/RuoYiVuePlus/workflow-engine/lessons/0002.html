第 2 课 · Warm-Flow 引擎集成：开关、配置与核心服务 基础架构

🎯 本节目标（对应使命 Success #2）：学完你能讲清 ruoyi-workflow 通过什么机制集成 Warm-Flow：warm-flow.enabled 开关、@ConditionalOnEnable 注解、POM 依赖栈、四个核心引擎 Service 与映射出来的四个扩展 Mapper，以及两套状态枚举（BusinessStatusEnum / TaskStatusEnum）的角色。

一、POM 依赖栈

RuoYi-Vue-Plus/ruoyi-modules/ruoyi-workflow/pom.xml

项目在根 POM 统一管理 Warm-Flow 版本（warm-flow.version=1.8.8），workflow 模块引入两个核心依赖：

依赖作用
warm-flow-mybatis-plus-sb3-starter引擎核心包，自带 MyBatis-Plus 集成，提供 TaskService/InsService/DefService/NodeService/HisTaskService 及四张表 Mapper
warm-flow-plugin-ui-sb-web流程设计器前端插件（warm-flow-ui），提供 HandlerSelectService 选人接口。在 Spring Boot 背景下自动注册设计器页面路径

workflow 模块还依赖了 ruoyi 家族的 push、mail、sms、mybatis、web、log、excel、translation、security 等公共模块，说明 workflow 不是孤岛——它要发推送通知、用短信、用翻译注解、受 Sa-Token 安全保护。

二、开关机制：@ConditionalOnEnable

ruoyi-workflow/common/ConditionalOnEnable.java:26-29

@Target({ElementType.TYPE, ElementType.METHOD})
@ConditionalOnProperty(value = "warm-flow.enabled", havingValue = "true")
public @interface ConditionalOnEnable {
}

这是一个自定义 Spring 条件注解：只在配置文件里 warm-flow.enabled=true 时，标了它的类才被加载。也就是说——工作流模块可以一个开关整体关掉。

项目中几乎所有组件都带着它：WarmFlowConfig、各 Controller、各 ServiceImpl、各 Listener、FlowProcessEventHandler、SpelRuleComponent。统一控制，不留死代码。

三、WarmFlowConfig — 空的配置入口

ruoyi-workflow/config/WarmFlowConfig.java:11-16

@ConditionalOnEnable
@Configuration
public class WarmFlowConfig {
}

这是一个纯占位的配置类。引擎本身通过 warm-flow-mybatis-plus-sb3-starter 的自动配置完成 Bean 注册（扫描 engine 的 mapper、service 等）。这个类存在的意义是：当 warm-flow.enabled=true 时触发此 scan；当 false 时连这个空壳都不加载。

四、引擎核心 Service — 本项目如何调用引擎

Warm-Flow starter 自动向 Spring 容器注册了这些核心 Service，ruoyi-workflow 通过构造函数注入使用。以 FlwTaskServiceImpl 为例：

// FlwTaskServiceImpl.java:82-87
private final TaskService taskService;      // 任务操作：skip、start、addSignature、depute、transfer...
private final InsService insService;         // 实例操作：start、updateById、getById...
private final DefService defService;         // 定义操作：getPublishByFlowCode、getFlowCombineNoDef...
private final HisTaskService hisTaskService;  // 历史任务：setSkipHisTask、save、getByInsId...
private final NodeService nodeService;        // 节点操作：getNextNodeList、getByNodeCodes...

另有一个重要的工具入口 FlowEngine，它是一个静态门面，底层也是取 Spring Bean：

`FlowEngine.defService()`     → DefService Bean
`FlowEngine.userService()`    → 引擎的 UserService（管理 task 关联的办理人列表）
`FlowEngine.insService()`     → InsService Bean
💡 设计观察：ruoyi-workflow 不直接操作 Warm-Flow 的 Mapper（如 FlowTaskMapper、FlowInstanceMapper），而是通过引擎的 Service 层操作。这保证了：① 业务逻辑不由引擎表直接 SQL 控制；② 引擎升级时 Mapper 变化不影响业务代码；③ 引擎的状态机转换（如 skip 后自动写 his_task）在 Service 层内部完成。

五、扩展 Mapper — ruoyi-workflow 自己扩展的查询

当需要按复杂业务条件查询（如「我收到的抄送，带分类筛选、按流程名称查」），引擎自带 Mapper 不够用。ruoyi-workflow 自己搞了几个扩展 Mapper：

Mapper表扩展了什么查询
FlwTaskMapperflow_taskgetListRunTask（待办列表，join 分类、实例、定义）、getTaskCopyByPage（我的抄送）
FlwHisTaskMapperflow_his_taskgetListFinishTask（已办列表，join 分类）
FlwCategoryMapperflw_category流程分类树查询、子分类收集
FlwInstanceBizExtMapperflw_instance_biz_extsaveOrUpdateByInstanceId（实例扩展信息的 UPSERT）

六、两套状态枚举

注意：工作流模块有两套状态，分别回答不同问题：

枚举文件回答的问题取值
BusinessStatusEnumruoyi-common-core/.../BusinessStatusEnum.java这个单据/流程整体处于什么阶段？draft / waiting / finish / cancel / back / invalid / termination
TaskStatusEnumruoyi-workflow/.../TaskStatusEnum.java这个具体的审批动作是什么？pass / back / depute / transfer / sign / sign_off / copy / cancel / invalid / termination

BusinessStatusEnum 写在 flow_instance.flow_status，代表这张单据现在在哪里（等待审批、已完成、已退回）。TaskStatusEnum 写在 flow_his_task.task_status，代表这次审批人做了什么（通过、驳回、转办）。

BusinessStatusEnum 还有一系列状态校验方法（checkStartStatus / checkBackStatus / checkCancelStatus / checkInvalidStatus），在发起/驳回/撤销/作废前做状态机门禁——防止并发状态下重复操作。

七、常量与路径

ruoyi-workflow/common/constant/FlowConstant.java

所有业务流程变量的 key 名都定义在这里：INITIATOR="initiator"、BUSINESS_ID="businessId"、AUTO_PASS="autoPass"、FLOW_COPY_LIST="flowCopyList"、SUBMIT="submit" 等。前端跳转页面路径也在这里：PATH_TASK_WAITING="/task/taskWaiting"（用于消息通知点击后跳转到待办页）。

💡 一句话总结：Warm-Flow 通过 starter 自动配置注册引擎 Bean → ruoyi-workflow 通过 @ConditionalOnEnable 统一控制加载 → 业务通过 engine Service (TaskService/InsService/DefService) 操作引擎 → 自己扩展 Mapper 做复杂查询 → BusinessStatusEnum/TaskStatusEnum 两套状态分别管实例状态和动作记录。

八、检验一下

Q1. 如果我想在生产环境临时关掉工作流功能，最快的方式是什么？

A. 删除 workflow 模块的 jar 包 B. 在配置文件中设置 warm-flow.enabled=false C. 删掉所有 @ConditionalOnEnable 注解 D. 卸载 MySQL 中的 flow_ 前缀表
看答案B。@ConditionalOnEnable 注解绑定了 warm-flow.enabled 属性，设为 false 后整个模块的 Controller、Service、Listener 都不会加载。

Q2. 下面哪个不是 Warm-Flow 引擎自带的 Service？

A. TaskService B. InsService C. FlwCategoryMapper D. DefService
看答案C。FlwCategoryMapper 是 ruoyi-workflow 自己扩展的 Mapper，用于流程分类查询。引擎自带的 Mapper 是 FlowTaskMapper、FlowInstanceMapper 等。

Q3. BusinessStatusEnum 和 TaskStatusEnum 的关系是？

A. 同一个东西的两个名字 B. BusinessStatusEnum 管「单据整体状态」，TaskStatusEnum 管「具体审批动作」 C. TaskStatusEnum 是 BusinessStatusEnum 的子集
看答案B。实例状态回答「这单子审批到哪了」，任务状态回答「审批人做了什么」。
