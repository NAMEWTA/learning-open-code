一、completeTask — 审批通过的核心

FlwTaskServiceImpl.java:207-258

public boolean completeTask(CompleteTaskBo completeTaskBo) {
    Long taskId = completeTaskBo.getTaskId();
    // 从入参取 消息类型、通知内容、抄送人列表
    List<String> messageType = completeTaskBo.getMessageType();
    String notice = completeTaskBo.getNotice();
    List<FlowCopyBo> flowCopyList = completeTaskBo.getFlowCopyList();

    Map<String, Object> variables = completeTaskBo.getVariables();
    variables.put(FLOW_COPY_LIST, flowCopyList);   // 抄送人
    variables.put(MESSAGE_TYPE, messageType);       // 消息类型
    variables.put(MESSAGE_NOTICE, notice);           // 通知内容

    // 1. 查 task 与 instance，校验存在性
    FlowTask flowTask = flowTaskMapper.selectById(taskId);
    Instance ins = insService.getById(flowTask.getInstanceId());

    // 2. 如果是 草稿/已撤销/已退回 → 标记 SUBMIT=true（触发申请人提交事件）
    if (BusinessStatusEnum.isDraftOrCancelOrBack(ins.getFlowStatus())) {
        variables.put(SUBMIT, true);
    }

    // 3. 弹窗选人 → 追加带状态前缀的处理人变量
    Map<String, Object> assigneeMap = setPopAssigneeMap(completeTaskBo.getAssigneeMap(), insVariableMap);
    variables.putAll(assigneeMap);

    // 4. 构建 FlowParams
    FlowParams flowParams = FlowParams.build()
        .handler(completeTaskBo.getHandler())
        .variable(variables)
        .ignore(...)          // ignore/ignoreDepute/ignoreCooperate
        .skipType(SkipType.PASS.getKey())              // 审批通过
        .message(completeTaskBo.getMessage())           // 审批意见
        .flowStatus(BusinessStatusEnum.WAITING.getStatus())  // 实例→待审核
        .hisStatus(TaskStatusEnum.PASS.getStatus())     // 历史→通过
        .hisTaskExt(completeTaskBo.getFileId());        // 附件

    // 5. 执行流转
    Boolean autoPass = Convert.toBool(insVariableMap.getOrDefault(AUTO_PASS, false));
    skipTask(taskId, flowParams, flowTask.getInstanceId(), autoPass);
    return true;
}

核心：skipTask 递归自动审批 (:269-296)

private void skipTask(Long taskId, FlowParams flowParams, Long instanceId, Boolean autoPass) {
    taskService.skip(taskId, flowParams);              // 引擎流转当前节点
    List<FlowTask> flowTaskList = selectByInstId(instanceId);

    for (FlowTask task : flowTaskList) {
        if (!task.getId().equals(taskId) && autoPass) {
            // 查办理人：如果下一个节点也是当前登录用户 → 自动审批
            List<User> users = StreamUtils.filter(userList, e ->
                task.getId().equals(e.getAssociated())
                && LoginHelper.getUserIdStr().equals(e.getProcessedBy()));
            if (CollUtil.isEmpty(users)) continue;

            flowParams.message("流程引擎自动审批！")
                .variable(Map.of(SUBMIT, false, FLOW_COPY_LIST, Collections.emptyList(), ...));
            skipTask(task.getId(), flowParams, instanceId, true);  // ← 递归！
        }
    }
}
💡 autoPass 的时机：autoPass 是流程定义扩展属性，在发起时从 definition.getExt() 解析（第3课已讲）。如果开了，同一个人兼任多个审批节点时，第一个审批通过后引擎自动审批后续同人的节点——减少无意义操作。

弹窗选人的变量名约定

前端审批弹窗里让用户选的「下一步审批人」，后端通过 setPopAssigneeMap (:305-330) 把选中的人追加到变量里，key 为 "pass:nodeCode" 和 "back:nodeCode" 两种前缀。这个变量在 WorkflowGlobalListener.assignment() 里被消费，把指定办理人写入 flowTask.permissionList。pass 前缀的变量只消费一次就删除；back 前缀的保留给后续可能反复驳回。

二、驳回 backProcess

FlwTaskServiceImpl.java:490-525

与 completeTask 类似但关键差异：

  • SkipType.REJECT（不是 PASS）
  • flowStatus：如果驳回到申请节点 → BACK；否则保持 WAITING
  • 判驳回的目标节点通过 flwCommonService.applyNodeCode(definitionId) 找到流程定义里的申请节点编码，与 bo.nodeCode 比对决定状态

可驳回节点列表由 getBackTaskNode() (:534-591) 通过 BFS 遍历前置节点得出，优先用节点配置的 anyNodeSkip（固定驳回目标），否则从历史任务中筛出中间节点。

四、四种运行时任务操作：taskOperation 统一分发

FlwTaskServiceImpl.taskOperation():864-950

前端 URL 是 POST /workflow/task/taskOperation/{taskOperation}，path variable 取值自 TaskOperationEnum：

操作code引擎 API结果
委派delegateTasktaskService.depute()把任务委派给另一个人审批（原审批人不再参与）
转办transferTasktaskService.transfer()把任务转给另一个人（原审批人退出当前任务）
加签addSignaturetaskService.addSignature()在当前节点增加审批人（必须是会签/票签节点，nodeRatio>0）
减签reductionSignaturetaskService.reductionSignature()在当前节点删除审批人（会签/票签节点）

统一分发逻辑：switch 匹配 TaskOperationEnum → 校验参数（委派/转办需单人选、加签/减签需多人选）→ 校验节点类型（加减签必须是会签票签）→ 调引擎 API → 发消息通知。

💡 委派 vs 转办：委派后任务仍保留在原审批人名下（委托给别人代理审批），完成后记录在 his_task 里标记 DEPUTE。转办后任务彻底交给新人，原审批人不再参与。这是审批责任归属的区别。

六、检验一下

Q1. 驳回到申请节点时，流程实例的状态会变成什么？

A. WAITING B. BACK C. FINISH
看答案B。代码 520 行：flowStatus(applyNodeCode.equals(bo.getNodeCode()) ? TaskStatusEnum.BACK.getStatus() : TaskStatusEnum.WAITING.getStatus())。

Q2. completeTask 时前端传了 nextAssignee（下一步办理人），这个值是怎么到达引擎的？

A. 存在请求头里 B. 通过 variables 里的 "pass:nodeCode" 前缀变量，在 WorkflowGlobalListener.assignment() 里写入 permissionList C. 直接写在 task 表里
看答案B。setPopAssigneeMap 把选中人追加到 variables，assignment 回调从 variables 取并写入 task.permissionList。pass 前缀的用一次就删；back 前缀保留。

Q3. 为什么加签/减签必须要求会签票签节点？

A. 审批评审节点没有「多人审批」概念，加减签无意义 B. 引擎 bug C. 性能考虑
看答案A。会签和票签节点是 Warm-Flow 里专门处理多人审批的节点类型，nodeRatio>0 表示是合议节点。普通审批节点只有一个人，不存在加减签。
