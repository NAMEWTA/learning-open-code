二、API 封装层：以 task/index.ts 为例

plus-ui-vue/src/api/workflow/task/index.ts

Vue 版把每个后端接口封装成一个独立函数，返回 AxiosPromise：

export const startWorkFlow = (data: object): any => request({ url: '/workflow/task/startWorkFlow', method: 'post', data });
export const completeTask = (data: object) => request({ url: '/workflow/task/completeTask', method: 'post', data });
export const backProcess = (data: any): any => request({ url: '/workflow/task/backProcess', method: 'post', data });
export const getTask = (taskId: string) => request({ url: '/workflow/task/getTask/' + taskId, method: 'get' });
export const taskOperation = (data: TaskOperationBo, operation: string) =>
  request({ url: `/workflow/task/taskOperation/${operation}`, method: 'post', data });
export const getNextNodeList = (data: any): any => request({ url: '/workflow/task/getNextNodeList', method: 'post', data });
export const terminationTask = (data: any) => request({ url: `/workflow/task/terminationTask`, method: 'post', data });
// ...还有 pageByTaskWait、pageByTaskFinish、pageByTaskCopy、getBackTaskNode、currentTaskAllUser、updateAssignee、urgeTask

React 版 api/workflow/task/index.ts 结构和函数名完全一样，差异仅在于 request 的具体实现（Vue 用 axios 拦截器、React 用 UmiJS request）。

三、审批组件 submitVerify — 核心交互

plus-ui-vue/src/components/Process/submitVerify.vue (约 500 行)

审批弹窗是整个前端工作流对接里最复杂的组件，核心流程：

打开弹窗 (openDialog, :311-340)

const openDialog = async (id?: string) => {
    taskId.value = id;
    dialog.visible = true;
    // 1. getTask(taskId) → 获取任务详情 + 按钮权限 + 抄送列表 + 变量列表
    const response = await getTask(taskId.value);
    task.value = response.data;
    // 2. 遍历 buttonList 设置按钮显示/隐藏
    task.value.buttonList?.forEach(e => { buttonObj.value[e.code] = e.show; });
    // 3. 获取已设抄送人列表
    selectCopyUserList.value = task.value.copyList;
    // 4. getNextNodeList() → 获取下一审批节点(用于弹窗选人)
    const nextData = await getNextNodeList({ taskId: taskId.value, variables: props.taskVariables });
    nestNodeList.value = nextData.data;
};

提交审批 (handleCompleteTask, :345-390)

const handleCompleteTask = async () => {
    form.value.taskId = taskId.value;
    form.value.variables = props.taskVariables;   // 业务变量（如请假天数）
    // 校验弹窗选人
    if (buttonObj.value.pop && nestNodeList.value?.length > 0) {
        // 必须为每个下一节点都选了审批人
    }
    // 组装抄送人
    form.value.flowCopyList = selectCopyUserList.value.map(e => ({ userId: e.userId, nickName: e.nickName }));
    // 确认后调 completeTask
    await completeTask(form.value);
}

按钮权限控制 (buttonObj, :252-260)

弹窗底部的操作按钮（委托/转办/加签/减签/终止/退回）不是硬编码的——它们由 getTask() 返回的 buttonList 控制显隐。buttonList 来自节点扩展 JSON 里的 ButtonPermissionEnum（第5课已讲）。前端只负责：

// :323-325
task.value.buttonList?.forEach(e => { buttonObj.value[e.code] = e.show; });

然后模板里 v-if="buttonObj.trust"、v-if="buttonObj.transfer" 等条件渲染。

React 版差异

plus-ui-react/src/components/workflow/SubmitVerify.tsx 逻辑完全同构，差异在技术栈：用 Ant Design 的 Modal/Form/Checkbox 替换 Element Plus 的 el-dialog/el-form/el-checkbox；用 React useState/useCallback 替换 Vue ref/reactive。

四、流程设计器：iframe 嵌入

plus-ui-vue/src/views/workflow/processDefinition/design.vue (约 60 行)

const baseUrl = import.meta.env.VITE_APP_BASE_API;
const open = async (definitionId, disabled) => {
    const url = baseUrl + `/warm-flow-ui/index.html?id=${definitionId}&onlyDesignShow=true`;
    iframeUrl.value = url + '&Authorization=Bearer ' + getToken()
        + '&clientid=' + import.meta.env.VITE_APP_CLIENT_ID;
};

<iframe :src="iframeUrl" class="warm-flow-designer-page__iframe"></iframe>

设计要点：

  • 设计器是 Warm-Flow 的 web 插件（warm-flow-plugin-ui-sb-web），后端自动注册了 /warm-flow-ui/index.html 路径。
  • 传入 id=definitionId（编辑已有定义）或空（新建定义）。onlyDesignShow=true 告诉设计器只显示设计界面。
  • Authorization=Bearer + token 把当前登录态传给 iframe（设计器 API 也在同域，需要鉴权）。
  • clientid 传给后端做客户端校验。
  • iframe 内部通过 postMessage({ method: 'close' }) 通知宿主关闭（design.vue 监听 message 事件）。

五、请假业务页 — 如何接入

plus-ui-vue/src/views/workflow/leave/leaveEdit.vue:246-268

const handleStartWorkFlow = async (data: LeaveForm) => {
    submitFormData.value.flowCode = flowCode.value;        // "leave1"
    submitFormData.value.businessId = data.id;              // 请假记录 ID
    taskVariables.value = {
        leaveDays: data.leaveDays,                          // 业务变量：影响条件网关
        userList: ['1', '3', '4']                           // 业务变量：示例数据
    };
    bizExt.value = {
        businessTitle: '请假申请',                           // 在审批列表里显示标题
        businessCode: data.applyCode                        // 业务编号
    };
    submitFormData.value.variables = taskVariables.value;
    submitFormData.value.bizExt = bizExt.value;

    const resp = await startWorkFlow(submitFormData.value);
    // 发起成功后打开审批弹窗办理首任务
    submitVerifyRef.value.openDialog(resp.data.taskId);
};

这就是业务接入工作流的全部前端代码：① 决定 flowCode；② 把业务数据放进 variables；③ 把展示信息放进 bizExt；④ 调 startWorkFlow；⑤ 弹审批窗。其余都是通用组件。

六、Vue vs React：技术栈差异

维度Vue 版React 版
UI 框架Element PlusAnt Design (antd)
状态管理ref / reactive（组件内部）useState / useCallback
HTTP 请求axios + utils/request.ts 拦截器UmiJS request + 拦截器
用户选择components/UserSelect/index.vuecomponents/UserSelect/UserSelect.tsx
审批弹窗submitVerify.vue (SFC, el-dialog)SubmitVerify.tsx (React FC, Modal)
设计器design.vue (iframe + postMessage)Design.tsx (同构，ifram + postMessage)
路由Vue RouterUmiJS 约定式路由

核心结论：两套前端在工作流模块上是同构的 API 封装 + 同构的组件逻辑，差异只有 UI 库和语法糖。

💡 一句话总结前端对接：API 层封装所有 workflow 接口 → 审批组件 submitVerify 是通用弹窗（开时查详情+下节点列表，关时调 completeTask/taskOperation） → 设计器是 iframe 嵌入 warm-flow-ui → 业务页面只需调 startWorkFlow+弹审批窗。

七、检验一下

Q1. 前端审批弹窗里的「委托」「转办」按钮有时显示有时不显示，决定因素是什么？

A. 当前用户的角色 B. 流程节点 ext 里的 ButtonPermissionEnum 配置 C. 前端代码注释掉的
看答案B。buttonList 来自节点的 ext JSON → NodeExtVo.buttonPermissions → getTask() 返回 → 前端遍历渲染。

Q2. 流程设计器的 iframe URL 为什么需要传 Authorization 和 clientid？

A. 设计器是独立应用，需要认证 token 才能访问同域的后端 API B. 只是为了好看 C. 不需要，可以去掉
看答案A。设计器页面的 Ajax 请求会和主应用一样走后端接口，需要 Authorization 做 Sa-Token 鉴权 + clientid 做客户端校验。

Q3. 我的业务（如合同审批）要接入工作流，前端最少要改哪几个地方？

A. 重写 submitVerify 组件 B. 只需 ① 在自己的编辑页调 startWorkFlow 传入 flowCode+businessId+variables+bizExt ② 复用 submitVerify 弹窗 C. 改 workflow 后端代码
看答案B。所有审批交互都在通用组件里，业务只需组装参数发起流程。
