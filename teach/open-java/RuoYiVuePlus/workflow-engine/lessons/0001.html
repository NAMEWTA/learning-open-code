第 1 课 · 鸟瞰：一张请假审批单的全栈链路 入门地图

🎯 本节目标（对应使命 Success #1）：学完你能用一段话 + 一张图，讲清一张请假申请从「前端提交」到「后端发起流程 → 审批办结 → 自动回写业务状态」的完整调用链，并说出每一环的关键文件。这是整个 ruoyi-workflow 模块的骨架——先有地图，再走小路。

我们不抠任何一行细节，只追一条线：用户在请假页填好请假信息，点「提交」，发生了什么？ 全程基于仓库真实代码。

二、拆成 6 个关键动作

1前端保存请假数据 + 发起流程
plus-ui-vue/src/views/workflow/leave/leaveEdit.vue:246-268
用户填好请假表，点提交。前端先保存 test_leave 数据到数据库，再组装 StartProcessBo：businessId（请假记录 ID）、flowCode（"leave1"）、variables（业务变量如 leaveDays）、bizExt（标题/编号）。调 POST /workflow/task/startWorkFlow 把流程发起来。
2后端启动流程实例
ruoyi-workflow/.../impl/FlwTaskServiceImpl.java:109-171
startWorkFlow() 是发起流程的核心方法。先校验 businessId 并去重——如果已有流程实例且状态允许，直接复用。然后往流程变量里塞 initiator（当前用户 ID）、initiatorDeptId、businessId。通过 FlowEngine.defService().getPublishByFlowCode() 查到已发布的流程定义，组装 FlowParams，调 insService.start() 让 Warm-Flow 引擎创建实例 + 首任务。最后保存 FlowInstanceBizExt 扩展信息，返回 StartProcessReturnDTO（含 instanceId + taskId）。
3审批弹窗：收集办理参数
plus-ui-vue/src/components/Process/submitVerify.vue:311-340
前端拿到 taskId 后打开审批弹窗。弹窗调 getTask() 查任务详情（按钮权限、抄送列表、变量列表），调 getNextNodeList() 获取下一审批节点（用于弹窗选人）。用户填写：消息类型、抄送人、审批意见、下一步审批人。点提交时调 completeTask(form.value)。
4后端办理任务 — taskService.skip() 流转
ruoyi-workflow/.../impl/FlwTaskServiceImpl.java:207-258
completeTask() 把前端的审批参数（消息类型、抄送人、审批意见、弹窗选人）组装进 FlowParams，设置 SkipType.PASS（审批通过），调 taskService.skip(taskId, flowParams) 让引擎执行节点流转。如果流程定义开了 autoPass 且下一个节点仍归属当前用户，还会递归 skipTask() 自动审批。
5引擎回调 → 发 Spring 事件
ruoyi-workflow/.../listener/WorkflowGlobalListener.java:178-258
引擎流转完节点后，触发 GlobalListener.finish()。在回调里判断流程是否已结束（flwTaskService.isTaskEnd()），若是则状态置为 FINISH；判断是否驳回、是否退回等。然后通过 FlowProcessEventHandler.processHandler()（`handler/FlowProcessEventHandler.java:35-49`）构造 ProcessEvent，用 SpringUtils.context().publishEvent() 发出去。
6业务模块接收事件 → 回写状态
ruoyi-workflow/.../impl/TestLeaveServiceImpl.java:211-237
@EventListener(condition = "#processEvent.flowCode.startsWith('leave')") 监听 ProcessEvent。方法里根据 processEvent.getBusinessId() 查出 test_leave 记录，把 processEvent.getStatus() 写回去——草稿/待审核/已完成/已退回。整个链路闭环。
💡 一句话记住这条链：
「前端发起 → startWorkFlow 建实例 → completeTask 流转节点 → 引擎回调 finish → 发 Spring 事件 → 业务 @EventListener 回写状态」。
这条主干，无论请假、报销、合同审批都一样。变的只是变量（variables）、流程定义（flowCode）、业务回写时具体写哪个表。

三、这条链上涉及的四张引擎表

表内容关键字段
flow_definition流程定义（设计器画出来的流程）flow_code、flow_name、version、is_publish
flow_instance流程实例（每发起一次就一条）business_id、flow_status、definition_id
flow_task待办任务（当前正卡在哪个节点）instance_id、node_code、node_name
flow_his_task历史任务（已办完/抄送的记录）task_id、node_code、task_status

注意：这些是 Warm-Flow 引擎的内置表，ruoyi-workflow 自己扩展了 flw_category（流程分类）、flw_instance_biz_ext（实例业务扩展）、flw_spel（SpEL 表达式库）、test_leave（请假示例业务表）等。

Q1. 发起流程的核心方法是哪个？它在哪个文件里？

A. WorkflowServiceImpl.startWorkFlow() — WorkflowServiceImpl.java B. FlwTaskServiceImpl.startWorkFlow() — FlwTaskServiceImpl.java:109 C. AuthController.login() — AuthController.java D. TaskService.start() — Warm-Flow 引擎自带
看答案B。封装层在 FlwTaskServiceImpl.startWorkFlow():109，它内部调 insService.start()。A 是 API 实现，最终也是转发给 B。D 是引擎能力但不是本项目的入口。

Q2. 业务模块（如请假）是怎么收到「流程已完成」这个通知的？

A. 轮询 /workflow/instance/getInfo 接口 B. 引擎直接调业务模块的方法 C. WorkflowGlobalListener.finish() → FlowProcessEventHandler 发 ProcessEvent → 业务 @EventListener 监听 D. 数据库触发器
看答案C。这是整个事件驱动解耦的核心设计：引擎回调 发 Spring 事件，业务模块通过 @EventListener 接收并更新自己的业务表状态。

Q3. 发起流程后，用户首次看到的审批弹窗里的「按钮权限（提交/退回/委托/转办等）」是从哪里来的？

A. 前端硬编码 B. getTask() 查到的 task.buttonList，来自节点扩展配置（NodeExtEnum / ButtonPermissionEnum） C. 角色权限表 D. 菜单路由配置
看答案B。selectById():712-738 里从流程节点的 ext 字段解析 NodeExtVo，其中 buttonPermissions 决定了弹窗显示哪些按钮（default:信/退回复选框、提交按钮等），前端 submitVerify.vue:322-325 遍历展示。
