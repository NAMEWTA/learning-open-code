第 5 课 · 事件驱动解耦 + 办理人解析 + 节点扩展 核心设计

🎯 本节目标（对应使命 Success #5、#6）：学完你能讲清三个关键设计——① 引擎回调→Spring 事件→业务模块 @EventListener 的完整事件驱动链；② 办理人是按 USER/ROLE/DEPT/POST/SPEL 五种类型通过 storageId 前缀解析的；③ 节点扩展（按钮权限/抄送配置/SpEL 动态办理人）如何挂在流程节点上。

一、事件驱动解耦：引擎 → 业务

整条链路

引擎流转完节点 │ ▼ GlobalListener.finish() ← Warm-Flow 引擎接口（四个回调之一） │ WorkflowGlobalListener.java:178 ▼ determineFlowStatus() ← 判断流程是否已结束 (:310) │ 如果 isTaskEnd → 置 FINISH ▼ flowProcessEventHandler. processHandler(flowCode, instance, status, params, submit) │ FlowProcessEventHandler.java:35 ▼ new ProcessEvent{flowCode,instanceId,businessId,...} SpringUtils.context().publishEvent(processEvent) ← :48 │ ▼ 业务模块 @EventListener 接收 │ TestLeaveServiceImpl.java:211 │ @EventListener(condition = │ "#processEvent.flowCode.startsWith('leave')") ▼ testLeave.setStatus(processEvent.getStatus()) ← 回写业务表

四个引擎回调时机

WorkflowGlobalListener.java:60-258 实现了 GlobalListener 接口，四个回调：

回调行号做了什么
create():66空方法，预留扩展
start():76-101解析节点 ext 中的抄送配置和默认变量，写入变量 Map。后续 finish 里用这些变量来创建抄送记录
assignment():109-132任务分派时动态调整办理人：处理弹窗选人（pass:nodeCode / back:nodeCode 变量）、将申请人设为申请节点的办理人
finish():178-258最重要：判断流程状态 → 发 ProcessEvent（总体状态变化）、发 ProcessTaskEvent（新任务创建）、发消息通知（抄送/待办提醒/结果通知），最后清理临时变量

三种 Spring 事件

ruoyi-api/.../api/event/ 定义了三个事件 DTO（跨模块契约）：

事件发的时机包含什么业务怎么区分的
ProcessEvent流程状态变化（完成/退回/撤销/作废/终止）flowCode、businessId、status、nodeCode、params@EventListener condition 按 flowCode 过滤
ProcessTaskEvent每个新任务创建时flowCode、businessId、taskId、nodeCode同上
ProcessDeleteEvent删除流程实例时flowCode、businessId同上

关键设计点：事件定义在 ruoyi-api 里，不是 ruoyi-workflow。这意味着业务模块只依赖 api 包就能监听流程事件，完全不需要依赖 workflow 实现模块。这是真正的解耦。

二、办理人解析：五种类型 + storageId 前缀

FlwTaskAssigneeServiceImpl.java 实现了 HandlerSelectService（Warm-Flow 设计器的选人接口）和 IFlwTaskAssigneeService。核心是 storageId 的编码规则：

类型storageId 格式示例解析为
USER纯数字 或 "user:" 前缀被省略"123"直接查 sys_user 表
ROLE"role:" + 角色ID"role:456"按角色 ID 查 sys_user_role 关联
DEPT"dept:" + 部门ID"dept:789"按部门 ID 查 sys_user_dept 关联
POST"post:" + 岗位ID"post:101"按岗位 ID 查 sys_user_post 关联
SPEL$ 或 # 开头"#{@spelRuleComponent.selectDeptLeaderById...}"SpEL 表达式，运行时动态执行

解析逻辑在 parseStorageId() (:307-326)：先看是否 SpEL 表达式（$ 或 # 开头），再看是否含 ":"，有则按前缀匹配枚举，没有则默认为 USER。

fetchUsersByStorageIds() (:204-218) 聚合所有类型的 ID，分别调对应的 system-api Service 查出用户列表，最后合并、去重。这就是设计器选人后落到 task.permissionList 的全过程。

三、SpEL 规则组件

rule/SpelRuleComponent.java:23-41 — 目前只有一个方法 selectDeptLeaderById(Long initiatorDeptId)：根据发起人部门 ID 查出部门负责人。这个 Bean 被注册到 Spring 容器后，流程设计器的 SpEL 表达式就可以写 #{@spelRuleComponent.selectDeptLeaderById(initiatorDeptId)} 来动态获取办理人。

domain/FlowSpel.java / FlwSpelServiceImpl.java — SpEL 表达式本身是预定义的模板，存在 flw_spel 表里。FlwSpelController 提供 CRUD，FlwSpelServiceImpl 把 SpEL 当一种「办理人类型」通过 selectSpelByTaskAssigneeList() 供设计器选人列表。

四、节点扩展（NodeExtVo）

流程设计器里每个节点可以配一个 JSON ext 字段。引擎流转到节点时，WorkflowGlobalListener.start() 调 flwNodeExtService.parseNodeExt(ext, variables) 解析成 NodeExtVo，包含三块配置：

扩展项对应枚举作用
按钮权限ButtonPermissionEnumpop/trust/transfer/copy/back/addSign/subSign/termination/file — 控制前端审批弹窗显示哪些按钮。默认：抄送/退回/终止/附件打开，其余关闭
抄送配置CopySettingEnum设置默认抄送人（按用户/角色/部门等）
节点变量—设置该节点运行时注入的默认变量

前端看到这些按钮权限的来源：FlwTaskServiceImpl.selectById():712-738 里从 NodeExtVo 的 buttonPermissions 映射出 flowTaskVo.buttonList，前端 submitVerify.vue:322-325 遍历展示。

五、消息通知：三支路

finish() 里根据变量发不同的 Spring 事件，对应三个专用监听器：

变量条件事件用途
FLOW_COPY_LIST 存在WorkflowCopyEvent创建抄送记录 + 发抄送消息
MESSAGE_TYPE 存在 + 非退回申请人WorkflowTaskMessageEvent给下一节点办理人发待办消息
已完成/已退回WorkflowResultMessageEvent给发起人发结果通知
💡 设计总结：整个事件驱动设计有三层解耦——① 引擎通过 GlobalListener 回调不直接接触业务代码；② FlowProcessEventHandler 把引擎数据转成跨模块 DTO 再发 Spring 事件；③ 事件定义在 ruoyi-api 里，业务模块只依赖 api。每一层都是单向依赖，业务模块完全不知道引擎的存在。

六、检验一下

Q1. 如果我要让「财务审批」这个业务收到流程事件，下面哪个是正确的做法？

A. 在 WorkflowGlobalListener 里加 if (flowCode == "finance") B. 在自己的业务 Service 里写 @EventListener(condition = "#processEvent.flowCode == 'finance'") C. 改引擎源码
看答案B。这正是 TestLeave 的做法（:211）。事件是全局广播，通过 SpEL condition 按 flowCode 过滤。A 会污染引擎层，不可扩展。

Q2. storageId "role:5" 会解析出什么办理人？

A. 用户 ID 为 5 的人 B. 角色 ID 为 5 的所有成员 C. 部门 ID 为 5 的所有人
看答案B。前缀 "role:" 匹配 TaskAssigneeEnum.ROLE，调 userService.selectUsersByRoleIds([5]) 查出角色下所有用户。

Q3. 流程设计器里关了某节点的「退回」按钮，这个配置是怎么到达前端审批弹窗的？

A. 写在前端菜单权限里 B. 存在 flow_node.ext JSON 里 → parseNodeExt → buttonList → getTask() 返回 → submitVerify 遍历显示 C. 存在 Redis 里
看答案B。设计器存 ext → NodeExtVo.buttonPermissions → selectById 读 node ext 映射为 buttonList → 前端根据 buttonList 判断显示哪些按钮。
