第 3 课 · 发起流程：startWorkFlow 全解析 核心方法

🎯 本节目标（对应使命 Success #3）：学完你能逐行解释 FlwTaskServiceImpl.startWorkFlow() 的每一步做了什么，以及 WorkflowService API 如何让任意业务模块只用一行代码发起流程。

一、入参：StartProcessBo

ruoyi-workflow/domain/bo/StartProcessBo.java:22-68

字段类型含义
businessIdString (notBlank)业务数据的唯一 ID（如 test_leave.id）。流程实例通过此字段与业务单据一对一绑定。
flowCodeString (notBlank)流程定义的编码（设计器里填的，如 "leave1"、"expense"）。用来查 flow_definition 表。
handlerString (optional)覆盖首节点的办理人。不填时引擎按节点配置的权限规则解析。
variablesMap流程变量——引擎流转时会用到的数据。业务传给引擎的变量（如 leaveDays、userList），也影响条件网关判断和办理人表达式。
bizExtFlowInstanceBizExt实例业务扩展信息（businessTitle、businessCode），存在扩展表 flw_instance_biz_ext 中，用于前端展示。
💡 为什么 businessId 不是 Long 而是 String？ 因为不同业务模块的主键类型可能不同（Long / UUID / 复合键），引擎需要通用性。实际上代码里用 Convert.toStr() 统一转 String。

二、核心方法：逐段精读

FlwTaskServiceImpl.java:109-171

第一段：防重复发起（:110-143）

// 1. 校验 businessId 不为空
String businessId = startProcessBo.getBusinessId();
if (StringUtils.isBlank(businessId)) throw new ServiceException("启动工作流时必须包含业务ID");

// 2. 装配系统变量
Map<String, Object> variables = startProcessBo.getVariables();
variables.put(INITIATOR, LoginHelper.getUserIdStr());       // 发起人
variables.put(INITIATOR_DEPT_ID, LoginHelper.getDeptId());  // 发起人部门
variables.put(BUSINESS_ID, businessId);                     // 业务ID

// 3. 查是否已有同 businessId 的流程实例
FlowInstance flowInstance = flowInstanceMapper.selectOne(
    QueryBuilder.lambda(FlowInstance.class)
        .eq(FlowInstance::getBusinessId, businessId).build());

if (ObjectUtil.isNotNull(flowInstance)) {
    // 已存在 → 状态校验（不能已结束/已作废/已终止再发起）
    BusinessStatusEnum.checkStartStatus(flowInstance.getFlowStatus());
    // 取出已有任务 → 合并变量 → 更新实例
    List<Task> taskList = taskService.list(new FlowTask().setInstanceId(flowInstance.getId()));
    taskService.mergeVariable(flowInstance, variables);
    insService.updateById(flowInstance);
    return new StartProcessReturnDTO(...);
}

这段的意义是：同一业务 ID 不允许重复发起新实例。如果之前发起过且还在审批中（draft/waiting），直接复用已有实例、合并新变量、返回已有任务。BusinessStatusEnum.checkStartStatus() 在已结束/已作废/已终止时会抛异常。

🤔 想一想：为什么不是「已结束允许重发」？ 设计选择：一个业务单据对应一条流程实例。如果业务需要「重新提交」，应该先把旧实例作废/删除，再发起新的，而不是一个 businessId 挂多个 instance。这简化了「按业务查流程状态」的逻辑。

第二段：发起新流程（:146-169）

// 4. 查流程定义（必须已发布）
Definition definition = FlowEngine.defService()
    .getPublishByFlowCode(startProcessBo.getFlowCode());
if (ObjectUtil.isNull(definition)) throw new ServiceException("流程未发布");

// 5. 解析定义扩展属性 — autoPass
Dict dict = JsonUtils.parseMap(definition.getExt());
boolean autoPass = !ObjectUtil.isNull(dict) && dict.getBool(AUTO_PASS);
variables.put(AUTO_PASS, autoPass);

// 6. 生成业务编号（若未提供则用时间戳）
variables.put(BUSINESS_CODE, this.generateBusinessCode(bizExt));

// 7. 构建 FlowParams 并启动
FlowParams flowParams = FlowParams.build()
    .handler(startProcessBo.getHandler())
    .flowCode(startProcessBo.getFlowCode())
    .variable(startProcessBo.getVariables())
    .flowStatus(BusinessStatusEnum.DRAFT.getStatus());  // 初始状态：草稿
Instance instance = insService.start(businessId, flowParams);

// 8. 保存实例业务扩展信息
this.buildFlowInstanceBizExt(instance, bizExt);

// 9. 获取首任务 — 必须只有一个
List<Task> taskList = taskService.list(...);
if (taskList.size() != 1) throw new ServiceException("...");
return new StartProcessReturnDTO(instance.getId(), taskList.getFirst().getId());

关键设计要点：

  • 发起时实例状态为 DRAFT（草稿）——不是 WAITING。真正的「待审核」状态在首次审批提交后才写入。
  • autoPass 是定义扩展属性（在流程设计器里配的 JSON），如果为 true，后续办理时 completeTask 会跳过同用户的下级节点自动审批。
  • generateBusinessCode 目前用时间戳，注释里写了 TODO: 按照自己业务规则生成编号——实际项目要覆盖。
  • 发起后必须恰好一个任务——Warm-Flow 要求流程起点后第一个节点是申请人节点，如果不只一个就报错。

三、两条调用路径：Controller vs API

发起流程有两条入口，对应两种使用场景：

场景入口文件
前端用户手动发起FlwTaskController.startWorkFlow()FlwTaskController.java:49 — 带 @SaCheckPermission，受鉴权保护
后端系统/定时任务自动发起WorkflowService.startWorkFlow(StartProcessDTO)WorkflowServiceImpl.java:117 — API 层，不检查当前登录用户，适合定时任务无登录态场景

更进一步的模式是 startCompleteTask()（WorkflowServiceImpl.java:159-174）：先发起、再立即办理首任务——适合后端自动化场景（比如定时任务收到第三方回调后自动提交流程）。TestLeave 正是用的这个：

// TestLeaveServiceImpl.java:170
boolean flag1 = workflowService.startCompleteTask(startProcess);

发起后马上自动办理申请人节点，把状态从 DRAFT 推成 WAITING。

四、检验一下

Q1. 同一 businessId 第二次调用 startWorkFlow 会怎样？

A. 创建一个新的流程实例 B. 复用已有实例、合并新变量，状态不可复用时抛异常 C. 把旧实例删除再新建
看答案B。代码 126-143 行先查已有 flowInstance，存在则走 checkStartStatus 校验，通过后合并变量、更新实例、返回已有 taskId。

Q2. 发起时流程实例的初始状态是什么？

A. waiting B. draft C. finish
看答案B。代码 158 行 .flowStatus(BusinessStatusEnum.DRAFT.getStatus())。在首次办理后才会变成 waiting。

Q3. WorkflowService.startCompleteTask() 与两次单独调用 startWorkFlow + completeTask 的区别？

A. 没有区别，是同一个功能 B. startCompleteTask 是事务性合并——先发起再自动办理首任务，适合后端自动化 C. startCompleteTask 不需要 businessId
看答案B。它在同一个事务里完成「发起 + 首任务办理」，并且自动设置 ignore=true 忽略权限，把消息类型设为只发站内信。
