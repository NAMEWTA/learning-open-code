01 | OAuth 2.0 与 26 平台支持：JustAuth 框架收编与配置体系

本节目标：理解 RuoYi-Vue-Plus 如何基于 JustAuth 收编 27 个第三方登录平台，掌握 OAuth 2.0 授权码模式的四阶段流程，理解 JustAuth 核心抽象与整个模块的配置体系。

1. 全景鸟瞰：13 个文件撑起 27 个平台

ruoyi-common-social 模块总计只有 13 个 Java 文件，却支撑了 27 个第三方登录平台（JustAuth 内置 24 个 + RuoYi 自定义 3 个）。这种极简的文件数量得益于两层设计：

  • JustAuth 框架层：提供通用的 OAuth 协议抽象，每个平台只需实现 2 个方法即可完成接入。
  • RuoYi 适配层：通过配置属性、Redis 状态缓存、统一调度器将 JustAuth 无缝融入 Spring Boot 体系。
文件角色关键职责
SocialUtils.java统一调度中心switch 语句匹配平台 → 实例化 AuthRequest；构建 AuthCallback 执行登录
SocialProperties.java配置顶层容器@ConfigurationProperties(prefix="justauth")，持有 Map<String, SocialLoginConfigProperties>
SocialLoginConfigProperties.java单平台凭证模型clientId / clientSecret / redirectUri + 扩展字段（serverUrl、agentId、scopes 等）
AuthRedisStateCache.java集群安全状态缓存实现 AuthStateCache，将 state 存入 Redis 替代 JVM 本地缓存
SocialAutoConfiguration.java自动装配入口注册 AuthRedisStateCache Bean，绑定 SocialProperties
AuthGiteaSource.java自定义平台 - 端点定义定义 Gitea 私有部署的 authorize / accessToken / userInfo 三个 URL
AuthGiteaRequest.java自定义平台 - 请求实现继承 AuthDefaultRequest，实现 getAccessToken + getUserInfo
AuthMaxKeySource.java自定义平台 - 端点定义MaxKey IAM 系统的 OAuth 端点
AuthMaxKeyRequest.java自定义平台 - 请求实现MaxKey token 与用户信息获取逻辑
AuthTopIamSource.java自定义平台 - 端点定义TopIAM 系统的 OAuth 端点
AuthTopIamRequest.java自定义平台 - 请求实现TopIAM token 与用户信息获取逻辑
关键洞察：SocialUtils 是整个模块的"交通警察"——所有平台的登录请求都经过它的 switch 分支路由到对应的 AuthRequest 实例。新增一个平台意味着新增一个 case 分支，而 JustAuth 内置的 24 个平台已经预先定义好了这些分支。

2. OAuth 2.0 授权码模式速览

JustAuth 实现的是 OAuth 2.0 的授权码模式（Authorization Code Grant），这是第三方登录最安全、最常用的授权方式。整个流程分为四个阶段：

阶段一：构建授权 URL

服务端调用 authRequest.authorize(state) 生成第三方平台的授权页面 URL。这个 URL 包含 client_id、redirect_uri、state（防 CSRF 随机串）等参数。用户浏览器被重定向到该 URL。

阶段二：用户授权 → 回调

用户在第三方平台完成登录授权后，平台将浏览器重定向回 redirect_uri，并在 URL 参数中携带 code（授权码）和 state（回调原样返回）。RuoYi 的 AuthController 接收这个回调。

阶段三：code 换 token

服务端用 code + client_id + client_secret 向第三方的 token 端点发起服务端到服务端的 POST 请求，换取 access_token。这一步是关键安全点——client_secret 永远不出现在浏览器端。对应 getAccessToken()。

阶段四：token 换用户信息

服务端用 access_token 请求第三方的用户信息端点，获取用户的 openId、昵称、头像等数据。对应 getUserInfo()。最终封装为 AuthUser 对象返回给业务层。

用户浏览器                 RuoYi 服务端                第三方平台 (如 GitHub)
   |                          |                              |
   |--点击"GitHub登录"-------->|                              |
   |                          |--authorize(state)----------->|
   |                          |<--授权页URL-------------------|
   |<--302重定向到授权页------|                              |
   |                                                         |
   |--用户在GitHub授权--------------------------------------->|
   |<--302重定向回callback?code=xxx&state=yyy----------------|
   |                          |                              |
   |--GET /auth/callback----->|                              |
   |                          |--getAccessToken(code)------->|
   |                          |<--access_token---------------|
   |                          |--getUserInfo(token)--------->|
   |                          |<--用户信息--------------------|
   |                          |                              |
   |<--登录成功/注册-----------|                              |
为什么是授权码模式而不是隐式模式？授权码模式中，access_token 只在服务端之间传输，浏览器从未接触过它。而隐式模式将 token 直接暴露在 URL fragment 中，安全性较差。JustAuth 选择授权码模式是正确且安全的设计决策。

3. JustAuth 核心抽象体系

JustAuth 框架通过一组定义清晰的接口和抽象类，将 OAuth 2.0 的通用流程模板化，平台差异被压缩到最小范围：

AuthRequest 接口

定义了两个核心方法：

  • authorize(String state) — 生成授权页面 URL，返回字符串。
  • login(AuthCallback callback) — 执行完整登录流程（code→token→用户信息），返回 AuthUser。

AuthDefaultRequest 抽象类

这是 JustAuth 框架的灵魂。它实现了 authorize() 和 login() 的标准流程，子类只需实现两个抽象方法：

  • getAccessToken(AuthCallback) — 用 code 换取 AuthToken。
  • getUserInfo(AuthToken) — 用 token 换取 AuthUser。

这意味着接入一个新平台，在大多数情况下只需要写这两个方法的 HTTP 调用逻辑。

AuthSource 接口

定义平台的三个端点 URL：

  • authorize() — 授权端点 URL。
  • accessToken() — token 端点 URL。
  • userInfo() — 用户信息端点 URL。

每个平台都有一个对应的 AuthSource 枚举值，JustAuth 内置了 24 个枚举常量。

AuthCallback

回调参数容器，字段：code（授权码）、state（防 CSRF）、auth_code（支付宝等平台专用）。从回调 URL 的 query 参数反序列化而来。

AuthToken

用户令牌容器，核心字段：accessToken、refreshToken、expireIn、openId、unionId 等。不同平台返回的字段组合不同。

AuthUser

统一用户信息模型，字段：uuid（平台内唯一标识）、username、nickname、avatar、email、rawUserInfo（平台原始 JSON）。这是最终交给业务层的数据结构。

AuthStateCache 接口

定义 state 参数的缓存操作：cache(String key, String value)、cache(String key, String value, long timeout)、get(String key)、containsKey(String key)。state 用于防止 CSRF 攻击——发起授权时随机生成并缓存，回调时比对。

4. 配置体系：一行 YAML 接入一个平台

SocialProperties — 顶层容器

使用 @ConfigurationProperties(prefix = "justauth") 绑定，结构极简：

@Data
@ConfigurationProperties(prefix = "justauth")
public class SocialProperties {
    private Map<String, SocialLoginConfigProperties> type;
}

Map 的 key 是平台标识符（如 gitee、github、gitea），value 是该平台的 OAuth 凭证配置。

SocialLoginConfigProperties — 单平台凭证

字段类型说明适用范围
clientIdStringOAuth 应用的 Client ID所有平台
clientSecretStringOAuth 应用的 Client Secret所有平台
redirectUriString授权回调地址所有平台
serverUrlString私有部署服务器地址Gitea / MaxKey / TopIAM
agentIdString企业微信应用 AgentId企业微信
alipayPublicKeyString支付宝公钥支付宝
tenantIdString微软 AAD 租户 ID微软 Azure AD
scopesString自定义授权范围可选，覆盖默认 scope

对应的 YAML 配置示例：

justauth:
  type:
    gitee:
      client-id: your-gitee-client-id
      client-secret: your-gitee-client-secret
      redirect-uri: http://localhost:8080/auth/callback
    gitea:
      client-id: your-gitea-client-id
      client-secret: your-gitea-client-secret
      redirect-uri: http://localhost:8080/auth/callback
      server-url: https://git.your-company.com   # 私有 Gitea 地址
    wechat-enterprise:
      client-id: your-corp-id                     # 注意：企业微信用 corpId
      client-secret: your-corp-secret
      redirect-uri: http://localhost:8080/auth/callback
      agent-id: "1000001"
设计亮点：Map 结构的配置方式使得平台增删极为灵活。添加一个新平台只需在 YAML 中增加一组配置项，无需修改任何 Java 配置类。这正是"开闭原则"在配置层的体现。

5. AuthRedisStateCache：从 JVM 到 Redis 的升级

JustAuth 默认提供的 state 缓存实现是基于 ConcurrentHashMap 的 JVM 本地缓存。这在单机部署时工作正常，但在集群环境下存在严重问题：

  • 问题：用户发起授权时 state 缓存在节点 A，但回调可能被负载均衡分配到节点 B。节点 B 找不到 state，判定为 CSRF 攻击，登录失败。
  • 解决：将所有 state 缓存操作委托给 Redis，实现集群内所有节点共享。

AuthRedisStateCache 实现细节

public class AuthRedisStateCache implements AuthStateCache {

    private static final String SOCIAL_AUTH_CODE_KEY = "social_auth_code:";
    private static final long TIMEOUT = 180;  // 3 分钟

    @Override
    public void cache(String key, String value) {
        RedisUtils.setCacheObject(SOCIAL_AUTH_CODE_KEY + key, value, TIMEOUT, TimeUnit.SECONDS);
    }

    @Override
    public void cache(String key, String value, long timeout) {
        RedisUtils.setCacheObject(SOCIAL_AUTH_CODE_KEY + key, value, timeout, TimeUnit.MILLISECONDS);
    }

    @Override
    public String get(String key) {
        return RedisUtils.getCacheObject(SOCIAL_AUTH_CODE_KEY + key);
    }

    @Override
    public boolean containsKey(String key) {
        return RedisUtils.hasKey(SOCIAL_AUTH_CODE_KEY + key);
    }
}

设计要点：

  • Key 前缀隔离：使用 social_auth_code: 作为前缀，在 Redis 中形成命名空间，避免与其他业务缓存键冲突。
  • 3 分钟 TTL：state 只在授权流程期间有效。3 分钟足够用户完成第三方平台授权，过期自动清理也避免了 Redis 内存泄漏。

SocialAutoConfiguration — 唯一的自动装配

@AutoConfiguration
@EnableConfigurationProperties(SocialProperties.class)
public class SocialAutoConfiguration {

    @Bean
    public AuthRedisStateCache authRedisStateCache() {
        return new AuthRedisStateCache();
    }
}

极度精简的配置类：启用 SocialProperties 配置绑定 + 注册一个 AuthRedisStateCache Bean。通过 AutoConfiguration.imports 文件被 Spring Boot 自动发现。

为什么不需要注册 AuthRequest Bean？因为 SocialUtils 是静态工具类，在每次请求时根据平台标识动态创建 AuthRequest 实例。这种"每次新建"的模式避免了 Bean 的复杂状态管理，每个平台的配置只存在于 SocialProperties 中。
