02 | JustAuth 扩展与自定义提供者：3 件套范式与企业平台适配

本节目标：掌握 RuoYi 自定义 OAuth 提供者的"3 件套"开发范式，理解企业微信和钉钉与标准 OAuth 2.0 的差异处理，通晓从用户点击到登录成功的完整链路。

1. 自定义提供者 3 件套范式

当需要接入的第三方平台不在 JustAuth 内置的 24 个平台列表中时，RuoYi 提供了一套标准化的"3 件套"扩展范式。这套范式让每个自定义平台仅需创建 3 个文件 + 添加 1 行 switch case 即可完成接入：

步骤文件名继承/实现必须提供什么
步骤 1AuthXxxSource.java实现 AuthSource 接口3 个端点 URL：authorize()、accessToken()、userInfo()
步骤 2AuthXxxRequest.java继承 AuthDefaultRequest2 个方法：getAccessToken()、getUserInfo()
步骤 3SocialUtils.java switch添加 1 个 case 分支new AuthXxxRequest(config, cache) 的构造调用
范式精髓：AuthDefaultRequest 已经实现了 authorize() 和 login() 的完整通用流程。你只需要告诉它两个平台特定的细节——"token 端点在哪里、怎么请求"以及"用户信息端点在哪里、怎么解析"。这是典型的模板方法模式。

RuoYi 使用这套范式接入了 3 个自定义平台：Gitea（自托管 Git 服务）、MaxKey（国产 IAM 系统）、TopIAM（国产 IAM 系统）。三者的代码结构完全一致，差别仅在 API 端点的 URL 和返回 JSON 的字段映射。

2. Gitea 实例拆解

Gitea 是一个自托管的 Git 服务（类似 GitHub / GitLab）。许多企业内部部署了私有 Gitea 实例，其 OAuth API 端点与标准 JustAuth 内置的 Gitee 端点不同，因此需要自定义提供者。

步骤 1：AuthGiteaSource — 定义端点

public enum AuthGiteaSource implements AuthSource {

    GITEA {
        @Override
        public String authorize() {
            return "{serverUrl}/login/oauth/authorize";
        }

        @Override
        public String accessToken() {
            return "{serverUrl}/login/oauth/access_token";
        }

        @Override
        public String userInfo() {
            return "{serverUrl}/api/v1/user";
        }
    };

    // 动态替换 {serverUrl} 占位符
    public String authorize(String serverUrl) {
        return authorize().replace("{serverUrl}", serverUrl);
    }
    // ... accessToken(String)、userInfo(String) 同理
}

关键设计：端点 URL 中使用 {serverUrl} 占位符，通过重载方法接收 SocialLoginConfigProperties 中的 serverUrl 配置值进行动态替换。这使得同一套代码可以适配不同企业各自的 Gitea 私有实例地址。

步骤 2：AuthGiteaRequest — 实现请求

public class AuthGiteaRequest extends AuthDefaultRequest {

    private final String serverUrl;

    public AuthGiteaRequest(AuthConfig config, AuthStateCache cache) {
        super(config, AuthGiteaSource.GITEA, cache);
        // 从 config 中提取 serverUrl（存入 SocialLoginConfigProperties 的扩展字段）
        this.serverUrl = config.getServerUrl();
    }

    @Override
    protected AuthToken getAccessToken(AuthCallback authCallback) {
        // 向 Gitea token 端点发送 POST 请求
        // 请求体：client_id、client_secret、code、grant_type、redirect_uri
        // 解析返回的 JSON 获取 access_token
    }

    @Override
    protected AuthUser getUserInfo(AuthToken authToken) {
        // 向 Gitea userInfo 端点发送 GET 请求（Header: Authorization: token xxx）
        // 解析返回的 JSON 映射到 AuthUser 的 uuid、username、nickname、avatar、email
    }
}

只实现两个方法。token 请求通常是一个带着 application/x-www-form-urlencoded 表单体的 POST，用户信息请求通常是一个带着 Authorization Header 的 GET。AuthDefaultRequest 框架已经帮你处理了 HTTP 客户端、异常处理、state 校验等通用逻辑。

步骤 3：SocialUtils 中的 switch 分支

// SocialUtils.getAuthRequest() 中
case GITEA:
    return new AuthGiteaRequest(config, authRedisStateCache);

仅一行。这就是"3 件套"的完整成本：2 个类（约 80-120 行）+ 1 行 switch。

3. MaxKey 与 TopIAM：私有 IAM 系统的 OAuth 适配

MaxKey 和 TopIAM 都是国产的企业级统一身份认证（IAM）系统。企业内部往往用它们统一管理员工账号，并与各类业务系统对接。通过 RuoYi 的自定义提供者，可以将 MaxKey / TopIAM 作为社交登录的"平台"之一，实现单点登录。

MaxKey 端点定义

public enum AuthMaxKeySource implements AuthSource {
    MAXKEY {
        @Override
        public String authorize() {
            return "{serverUrl}/auth/oauth/v20/authorize";
        }

        @Override
        public String accessToken() {
            return "{serverUrl}/auth/oauth/v20/token";
        }

        @Override
        public String userInfo() {
            return "{serverUrl}/api/oauth/v20/userinfo";
        }
    };
}

TopIAM 端点定义

public enum AuthTopIamSource implements AuthSource {
    TOPIAM {
        @Override
        public String authorize() {
            return "{serverUrl}/authentication/oauth2/authorize";
        }

        @Override
        public String accessToken() {
            return "{serverUrl}/authentication/oauth2/token";
        }

        @Override
        public String userInfo() {
            return "{serverUrl}/authentication/oauth2/userinfo";
        }
    };
}

两者的实现模式与 Gitea 完全一致——唯一的区别是端点 URL 路径不同。它们同样使用 {serverUrl} 占位符，同样从 SocialLoginConfigProperties 的 serverUrl 字段获取私有部署地址。

特性MaxKeyTopIAM
授权端点/auth/oauth/v20/authorize/authentication/oauth2/authorize
Token 端点/auth/oauth/v20/token/authentication/oauth2/token
用户信息端点/api/oauth/v20/userinfo/authentication/oauth2/userinfo
版本OAuth 2.0OAuth 2.0
为什么这些系统需要自定义提供者？标准 JustAuth 支持的 24 个平台都是公共 SaaS 服务（GitHub、Google、微信等）。企业内部自建的 IAM 系统 API 端点与这些公共平台不同，JustAuth 无法内置覆盖。自定义提供者机制解决了"标准框架无法覆盖非标准端点"的问题。

4. 企业微信与钉钉的特殊处理

并非所有平台都遵循标准的 OAuth 2.0 授权码模式。企业微信和钉钉是两个典型的"非标"案例。虽然 JustAuth 已经内置了它们的支持，但理解其特殊之处有助于在下节课阅读 JustAuth 源码时抓住重点。

企业微信 (WeCom) 的特殊性

差异点标准 OAuth 2.0企业微信
客户端标识client_id + client_secretcorpId + corpSecret（企业维度而非应用维度）
Token 请求体格式form-urlencodedJSON 格式请求体
扩展凭证无agentId（企业应用 ID）
敏感信息无user_ticket 字段，需要特殊处理

JustAuth 中对应的实现是 AbstractAuthWeChatEnterpriseRequest，它绕过了 AuthDefaultRequest 的标准 flow，自定义了 token 请求和用户信息获取的逻辑。这解释了为什么 SocialLoginConfigProperties 需要 agentId 这样的扩展字段。

钉钉 (DingTalk) 的特殊性

钉钉使用的 AuthDingTalkV2Request 同样偏离标准 OAuth 流程：

  • 使用 JSON 请求体而非标准的 form-urlencoded。
  • 钉钉的 access_token 获取需要先获取企业内部应用的 access_token，再通过临时授权码换取用户信息。
  • 这是钉钉开放平台 V2 版 API 的特殊要求，与标准 OAuth 2.0 存在显著差异。
关键启示：标准框架不能包打天下。企业级平台往往有自己的 API 约定。JustAuth 通过让每个平台拥有自己的 Request 子类来处理这种差异——标准平台复用 AuthDefaultRequest，特殊平台重写 login() 中的关键步骤或直接实现 AuthRequest 接口。RuoYi 的 SocialLoginConfigProperties 中预留的扩展字段（agentId、alipayPublicKey、tenantId 等）就是为了适配这些非标参数。

5. 完整登录链路：从点击按钮到业务处理

把前两节课学到的所有组件串联起来，展示一个完整的第三方登录请求在 RuoYi 中的生命周期：

┌─────────────────────────────────────────────────────────────────────┐
│ 步骤 1：前端发起                                                      │
│ 用户点击 "Gitee 登录" 按钮                                           │
│ 前端 GET /auth?source=gitee                                         │
└───────────────────────────────┬─────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 步骤 2：AuthController 接收请求                                       │
│ ① 调用 SocialUtils.getAuthRequest("gitee", socialProperties)        │
│    → switch("gitee") → new AuthGiteeRequest(config, cache)           │
│ ② 调用 authRequest.authorize(state)                                 │
│    → 生成 Gitee 授权页 URL                                            │
│    → state 存入 AuthRedisStateCache → Redis                          │
│ ③ 返回 302 重定向到 Gitee 授权页                                      │
└───────────────────────────────┬─────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 步骤 3：用户授权 → 回调                                                │
│ 用户在 Gitee 完成授权                                                │
│ Gitee 302 重定向回 redirect_uri?code=abc&state=xyz                  │
└───────────────────────────────┬─────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 步骤 4：SocialUtils.loginAuth() 执行登录                               │
│ ① 构建 AuthCallback(code="abc", state="xyz")                        │
│ ② 调用 authRequest.login(callback)                                  │
│    → AuthDefaultRequest.login() 内部:                                │
│      a. 校验 state：AuthRedisStateCache.get("social_auth_code:xyz") │
│      b. getAccessToken(callback) → POST Gitea token 端点              │
│      c. getUserInfo(authToken) → GET Gitea userInfo 端点             │
│    → 返回 AuthUser{uuid, username, nickname, avatar, email}         │
│ ③ 返回 AuthUser 给 Controller                                        │
└───────────────────────────────┬─────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 步骤 5：业务层处理                                                     │
│ Controller 调用 SysLoginService.socialLogin(AuthUser)               │
│ → 根据 uuid 查询是否已存在绑定用户                                     │
│ → 新用户：创建账号 + 绑定社交账号                                      │
│ → 老用户：直接登录，签发 token                                        │
│ → 返回登录成功                                                        │
└─────────────────────────────────────────────────────────────────────┘

整个链路中，RuoYi 开发者需要关注的只有两件事：

  1. 配置 YAML：填写 client-id、client-secret、redirect-uri。
  2. （如果是新平台）按 3 件套范式扩展：创建 AuthXxxSource、AuthXxxRequest、添加 switch case。

其余的工作——state 校验、token 交换、用户信息获取、HTTP 调用、异常处理——全部由 JustAuth 框架和 AuthRedisStateCache 自动完成。
