第 3 课 · Sa-Token 怎么签发 token 认证内核

🎯 本节目标（对应使命 Success #3）：讲清 Sa-Token 如何签发 token、LoginUser 存在哪里、JWT 简单模式与 Redis 各自承担什么角色。这是第 1 课全景图第 6 步的放大。

策略校验完身份，调用 LoginHelper.login(loginUser, model)。这一步看似一行，背后是整个认证内核。我们拆开看。

一、LoginHelper.login() 做了三件事

// LoginHelper.java:54
public static void login(LoginUser loginUser, SaLoginParameter model) {
    model = ObjectUtil.defaultIfNull(model, new SaLoginParameter());
    fillRequestContext(loginUser, model);          // ① 补充 IP/浏览器/OS 等终端信息
    StpUtil.login(loginUser.getLoginId(),           // ② Sa-Token 签发 token
        model.setExtra(USER_KEY, loginUser.getUserId())
             .setExtra(USER_NAME_KEY, loginUser.getUsername())
             .setExtra(DEPT_KEY, loginUser.getDeptId())
             .setExtra(DEPT_NAME_KEY, loginUser.getDeptName())
             .setExtra(DEPT_CATEGORY_KEY, loginUser.getDeptCategory())
    );
    StpUtil.getTokenSession().set(LOGIN_USER_KEY, loginUser);  // ③ 把 LoginUser 存进 Token-Session
}

① loginId 是什么？为什么是「userType:userId」？

// LoginUser.java:141
public String getLoginId() {
    return userType + ":" + userId;   // 例如 "sys_user:1"
}

Sa-Token 用 loginId 唯一标识「谁登录了」。RuoYi 把它设计成 userType:userId，是为了支持多用户体系：同一张表可有 pc/app 多种用户类型，甚至不同用户表共存，靠前缀区分，权限互不串。

② 扩展信息（extra）塞进了 token

userId、userName、deptId、clientid 等被塞进 token 的扩展字段。好处：之后想拿当前用户 id，不必反序列化整个 LoginUser，直接 StpUtil.getExtra("userId") 即可，轻量。

③ 完整的 LoginUser 存进 Token-Session

角色、岗位、权限集合这些重对象，存进 Sa-Token 的 Token-Session（最终落 Redis）。取的时候：

// LoginHelper.getLoginUser()
return StpUtil.getTokenSession().get(LOGIN_USER_KEY);

二、关键分工：JWT 简单模式 vs Redis

这是本课最容易混淆、也最值得搞清的点。看配置：

// SaTokenConfig.java:30
@Bean
public StpLogic getStpLogicJwt() {
    return new StpLogicJwtForSimple();   // JWT「简单模式」
}
@Bean
public SaTokenDao saTokenDao() {
    return new PlusSaTokenDao();          // 自定义 Redis 持久层
}
JWT（token 本身）Redis（PlusSaTokenDao）
存什么loginId + 扩展信息，签名防篡改Token-Session（完整 LoginUser、权限等）+ 登录态记录
角色「身份凭证」，无状态可自解析「会话数据 + 可控状态」
为何需要它token 自带身份，减少一次查库能主动踢人下线、改权限即时生效、查在线用户
✅ 「简单模式」的含义：token 是 JWT（自带 loginId 和 extra，可离线验签），但会话数据仍走 Redis。这样既享受 JWT「身份自携带」的便利，又保留服务端「能踢人、能改权限、能查在线」的控制力——纯无状态 JWT 做不到主动失效，这是工程上的折中。

PlusSaTokenDao 的小心思：多级缓存

它不是裸调 Redis，而是 Caffeine（本地，5 秒过期）+ Redis 两级：

// PlusSaTokenDao.java
private static final Cache<String, Object> CAFFEINE = Caffeine.newBuilder()
    .expireAfterWrite(5, TimeUnit.SECONDS)   // 本地缓存只活 5 秒
    .maximumSize(1000).build();
private <T> T getCacheValue(String key) {
    return (T) CAFFEINE.get(key, RedisUtils::getCacheObject); // 本地没有才查 Redis
}

高并发下，每个请求都要读登录态，本地缓存挡掉绝大多数 Redis 往返；5 秒过期保证踢人/改权限的延迟可接受。写操作会 invalidate(key) 让本地缓存失效。

三、token 怎么回到前端、又怎么带回来

签发后： StpUtil.getTokenValue() → LoginVo.accessToken → 前端 setToken() 后续请求： 前端 Authorization: Bearer <token> (request.ts 拦截器加) + clientid: <clientId> 请求头 ↓ 后端 SaInterceptor 读取并校验（第 4 课）

配置里 token-prefix: "Bearer"、is-read-header: true、is-read-cookie: false（关 cookie 杜绝 CSRF）。

🤔 想一想：既然 JWT 自带身份可离线验签，为什么登录还要往 Redis 写一份？纯 JWT 不香吗？ 纯 JWT 的致命短板是无法主动失效：token 签发后在过期前一直有效，管理员「踢人下线」「禁用账号即时生效」「改了角色立刻收权」都做不到，只能等过期。RuoYi 的「JWT 简单模式 + Redis 会话」就是为了拿回这份控制力——异常处理里那些 BE_REPLACED（被顶号）、KICK_OUT（被踢）状态，全靠 Redis 侧的会话记录才能实现。

四、检验一下（提取练习）

Q1. 完整的 LoginUser（含角色、权限集合）存在哪里？

A. 全部编码进 JWT token 字符串里 B. Sa-Token 的 Token-Session 中（经 PlusSaTokenDao 落 Redis） C. 前端 localStorage D. 数据库 sys_user 表
看答案B。token 里只放轻量的 loginId + extra；完整 LoginUser 经 StpUtil.getTokenSession().set() 存入 Token-Session，由 PlusSaTokenDao 持久化到 Redis（带 Caffeine 本地缓存）。

Q2. 管理员把某用户「强制下线」能立即生效，靠的是？

A. JWT 自带的过期时间 B. 前端轮询 C. Redis 侧的会话记录可被服务端主动清除/标记，验签时发现状态异常即拒绝 D. 修改用户密码
看答案C。这正是「JWT 简单模式 + Redis」组合的价值——纯无状态 JWT 在过期前无法主动失效，Redis 会话让服务端重获控制权。

Q3. loginId 设计成 userType:userId（如 sys_user:1）主要为了？

A. 好看 B. 支持多用户体系，不同用户类型按前缀区分、权限隔离 C. 加密需要 D. Redis key 长度限制
看答案B。见 LoginHelper 类注释：user_type 可让同表多类型用户、甚至多用户表共存，组成「用户类型 × 设备类型」的灵活权限控制。
