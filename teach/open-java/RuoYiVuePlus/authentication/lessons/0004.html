第 4 课 · 统一拦截器 + 权限加载 + 异常 鉴权关卡

🎯 本节目标（对应使命 Success #4）：讲清统一拦截器（SecurityConfig）如何做登录校验 + 客户端一致性校验 + 访问路径/IP 白名单，StpInterface 如何按需加载权限，以及未登录/无权限时异常如何被统一处理。前三课讲「怎么拿到 token」，这一课讲「带着 token 来了，怎么放行或拦下」。

一、所有请求的总闸：SaInterceptor

注册在 SecurityConfig.addInterceptors()，拦 /**，排除白名单：

// SecurityConfig.java:81
registry.addInterceptor(new SaInterceptor(handler -> {
    SaRouter.match(allUrlHandler.getUrls())   // 匹配所有已注册接口
        .check(() -> {
            StpUtil.checkLogin();              // ① 必须已登录，否则抛 NotLoginException
            // ② 客户端一致性校验
            String headerCid = request.getHeader(LoginHelper.CLIENT_KEY);
            String clientId  = StpUtil.getExtra(LoginHelper.CLIENT_KEY).toString();
            if (!StringUtils.equalsAny(clientId, headerCid, paramCid)) {
                throw NotLoginException.newInstance(..., "-100", "客户端ID与Token不匹配", ...);
            }
            validateClientAccessRules(request); // ③ 访问路径 + IP 白名单
        });
})).addPathPatterns("/**")
  .excludePathPatterns(securityProperties.getExcludes());  // 白名单：/auth/login 等

三道关卡，逐个拆：

① 登录校验 StpUtil.checkLogin()

读 Authorization: Bearer ... 里的 token，验签 + 查会话。无效就抛 NotLoginException（交给异常处理器）。

② 客户端一致性校验（RuoYi 的加固）

token 里记录了签发时的 clientid（第 3 课的 extra）。这里要求请求头/参数里的 clientid 必须和 token 里的一致。

✅ 为什么要这道校验：防止一个端签发的 token 被挪到另一个端冒用。token 和客户端绑定，「PC 后台的 token」不能拿去当「App 的 token」用。这是 RuoYi 在 Sa-Token 基础上加的安全约束。

③ 访问路径 + IP 白名单（按客户端配置）

// SecurityConfig.java:147 validateClientAccessRules()
String accessPath = getTokenExtra(CLIENT_ACCESS_PATH_KEY);   // 来自 sys_client
if (路径不匹配 accessPath) throw new NotPermissionException("当前客户端未授权访问该接口路径");
String ipWhitelist = getTokenExtra(CLIENT_IP_WHITELIST_KEY);
if (clientIp 不在 ipWhitelist) throw new NotPermissionException("当前客户端IP不在白名单内");

这就接上了第 2 课的 sys_client.accessPath / ipWhitelist 字段——配置在登录时塞进 token extra，请求时在这里生效。

💡 白名单从哪来：securityProperties.getExcludes() 配置了不需要登录的路径（如 /auth/login、/auth/code、静态资源）。另外 Controller 上的 @SaIgnore 注解（如 AuthController）也表示「整个类放行」。

二、权限怎么来：StpInterface（按需加载）

登录校验只管「是不是登录了」。具体接口要 @SaCheckPermission("system:user:list") 这种权限码时，Sa-Token 回调 StpInterface 拿权限列表。RuoYi 的实现是 SaPermissionImpl：

// SaPermissionImpl.java:32
public List<String> getPermissionList(Object loginId, String loginType) {
    return resolvePermissionList(loginId,
        LoginUser::getMenuPermission,              // 优先：从当前 LoginUser 直接取
        PermissionService::getMenuPermission);     // 兜底：远程/服务查
}

逻辑很聪明：如果要查的就是当前登录用户，直接从内存里的 LoginUser 取权限集合（登录时 buildLoginUser 已装好）；只有跨用户查询才走 PermissionService 兜底。避免每次鉴权都查库。

注解校验什么数据来源
@SaCheckLogin是否登录token 会话
@SaCheckPermission菜单权限码getPermissionList → menuPermission
@SaCheckRole角色标识getRoleList → rolePermission

三、拦下之后：统一异常处理

SaTokenExceptionHandler（@RestControllerAdvice）把认证异常翻译成统一响应：

// 无权限/无角色 → 403
@ExceptionHandler({NotPermissionException.class, NotRoleException.class})
→ R.fail(403, "没有访问权限，请联系管理员授权");

// 未登录 → 401，按类型给不同提示
@ExceptionHandler(NotLoginException.class)
switch (e.getType()) {
    case TOKEN_TIMEOUT  -> "登录已过期，请重新登录";
    case BE_REPLACED    -> "当前账号已在其他设备登录，您已被强制下线";
    case KICK_OUT       -> "账号已被管理员强制下线";
    case TOKEN_FREEZE   -> "账号已被冻结，请联系管理员处理";
    default             -> "登录状态异常，请重新登录";
}  → R.fail(401, msg);
✅ 注意这里 BE_REPLACED（被顶号）、KICK_OUT（被踢）这些状态，正是第 3 课「JWT + Redis 会话」才能实现的主动失效——纯 JWT 给不出这些细分原因。前端拿到 401 就触发「重新登录」弹窗（第 7、8 课）。
🤔 想一想：拦截器里「客户端一致性校验」抛的是 NotLoginException（401）而不是 NotPermissionException（403），为什么？ 因为客户端 ID 与 token 不匹配，本质上说明「这个 token 在当前上下文里不可信 / 不是合法登录态」，属于认证问题而非授权问题。返回 401 让前端走「重新登录」流程才合理；403 会被理解成「登录有效但权限不够」，语义错误。区分 401（认证失败）和 403（授权失败）是 HTTP 语义的基本功。

四、检验一下（提取练习）

Q1. 一个带合法 token 的请求，但权限码不够，会得到什么？经过哪个处理器？

A. 401，SaTokenExceptionHandler 处理 NotLoginException B. 403，SaTokenExceptionHandler 处理 NotPermissionException C. 500 服务器错误 D. 直接放行
看答案B。登录有效但缺权限码 → NotPermissionException → 403「没有访问权限」。401 是「没登录/登录失效」，403 是「登录了但权限不够」。

Q2. 鉴权时拿当前用户的权限列表，正常情况下会查数据库吗？

A. 每次都查库 B. 通常不查——直接从内存中当前 LoginUser 的 menuPermission 取，登录时已装好 C. 查 MySQL 的权限表 D. 查 ES
看答案B。SaPermissionImpl 优先从当前 LoginUser 取权限集合（登录时 buildLoginUser 已组装并随会话缓存），只有跨用户查询才走 PermissionService 兜底。

Q3. sys_client 里配的 accessPath、ipWhitelist 在哪一步真正生效？

A. 登录时 B. 每次受保护请求经过 SaInterceptor 的 validateClientAccessRules() 时 C. 前端校验 D. 数据库约束
看答案B。这些配置登录时被塞进 token extra，之后每个请求在拦截器里读出来校验路径与 IP，不匹配抛 NotPermissionException。
