🎯 本节目标：读透 user.ts 的 Pinia Setup Store 写法——defineStore 的三段式结构（state → actions → return）、await-to-js 错误处理模式、登录后 token/roles/permissions 的更新链路。

一、Setup Store 语法：Pinia 的 Composition API 写法

// store/modules/user.ts:11-12
export const useUserStore = defineStore('user', () => {
  // state: 用 ref() 定义响应式状态
  const token = ref(getToken());
  const roles = ref<Array<string>>([]);
  const permissions = ref<Array<string>>([]);

  // actions: 用普通 async function 定义
  const login = async (userInfo: LoginData): Promise<void> => { ... };
  const getInfo = async (): Promise<void> => { ... };
  const logout = async (): Promise<void> => { ... };

  // return: 暴露给组件使用
  return { token, roles, permissions, login, getInfo, logout };
});

这是 Pinia 的 Setup Store 语法（对标 Vue 3 <script setup>），与传统的 Options Store 的区别：

特性Options StoreSetup Store（本项目）
状态定义state: () => ({...})const x = ref(...)
动作定义actions: { fn() {...} }const fn = () => {...}
计算属性getters: { x() {...} }const x = computed(...)
this 访问需要 this不需要 this，直接访问变量
TypeScript需要复杂的类型标注自动推断，类型安全
💡 为什么选 Setup Store？ 不需要 this，代码更接近 Vue 3 组合式 API；TypeScript 类型推断开箱即用；可以在 store 内使用 watch、inject 等组合式 API。RuoYi-Vue 全项目都使用这种写法。

二、await-to-js：Go 风格的错误处理

// store/modules/user.ts:26, 41
import { to } from 'await-to-js';

const [err, res] = await to(loginApi(userInfo));  // 登录
const [err, res] = await to(getUserInfo());        // 获取用户信息

await-to-js 是一个微型库，将 Promise 的结果包装成 [error, result] 元组。核心实现就一行：

// await-to-js 核心逻辑
export function to<T>(promise: Promise<T>): Promise<[Error | null, T | undefined]> {
  return promise
    .then(data => [null, data] as const)
    .catch(err => [err, undefined] as const);
}

这模仿了 Go 语言的多返回值错误处理模式。与 try-catch 的对比：

// 传统 try-catch（嵌套层级深）
try {
  const res = await loginApi(userInfo);
  // 处理 res
} catch (err) {
  // 处理 err
}

// await-to-js（扁平化）
const [err, res] = await to(loginApi(userInfo));
if (res) { /* 处理成功 */ }
return Promise.reject(err);  // 失败透传
⚠️ 注意：await-to-js 不是必须的，但在需要区分业务错误和网络错误的场景下非常有用。这里的用法是：接口返回了但业务逻辑不对（比如没有 access_token）→ res 存在但需要手动 reject；网络挂了 → err 存在。

三、login → getInfo 调用链：store 如何与后端对接

用户点击登录后，前端 store 的调用顺序如下：

① userStore.login(form) └─ loginApi({ username, password, clientId, grantType }) └─ POST /auth/login （isToken:false, isEncrypt:true） → 后端返回 { access_token } └─ setToken(access_token) // 存 localStorage └─ token.value = access_token // 更新响应式状态 ② permission.ts 路由守卫检测到 roles.length === 0 └─ userStore.getInfo() └─ GET /system/user/getInfo （自动带 token） → 后端返回 { user, roles, permissions } └─ roles.value = ['admin', ...] // 角色编码 └─ permissions.value = ['*:*:*'] // 权限编码 ③ permission.ts 拿到 roles → permissionStore.generateRoutes() └─ GET /system/menu/getRouters // 获取菜单路由 └─ filterAsyncRouter() // 后端路由 → Vue Router └─ router.addRoute() // 动态注册路由

核心洞察：token 和 roles 是分两步获取的。登录只拿 token，用户信息和角色权限在 getInfo 里拿。这是因为 Sa-Token 的 token 签发和权限加载是解耦的——token 是「入场券」，权限是「座位号」。
