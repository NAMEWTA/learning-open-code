第 5 课 · 五种登录 + 注册 + 三类验证码 方式全览

🎯 本节目标（对应使命 Success #5）：说清五种登录方式（password / sms / email / social / xcx）+ 注册 + 三类验证码（图形/短信/邮箱）各自的校验要点与差异。有了前 4 课的骨架，这一课是把每个策略「填进模板」。

记住第 1 课的主干：五种方式只有「身份校验」这一步不同，后面 buildLoginUser → LoginHelper.login 完全一样。所以我们只对比「各策略怎么验身份」。

一、五种策略横向对比

策略入参 Body身份校验核心用户从哪查
passwordusername, password, code, uuid图形验证码 + BCrypt.checkpw 比对密码按 userName 查
smsphoneNumber, smsCode比对 Redis 里的短信验证码按 phoneNumber 查
emailemail, emailCode比对 Redis 里的邮箱验证码按 email 查

password — 最完整的范例

// PasswordAuthStrategy.java:56
if (captchaEnabled) validateCaptcha(username, code, uuid);   // 图形验证码（开关控制）
SysUserVo user = loadUserByUsername(username);
loginService.checkLogin(LoginType.PASSWORD, username,
    () -> !BCrypt.checkpw(password, user.getPassword()));     // 密码比对，失败计数

checkLogin（在 SysLoginService）封装了失败次数限制：连续错 N 次锁定 M 分钟，计数存 Redis，成功即清零。五种带「凭证校验」的方式都复用它。

sms / email — 验证码即凭证

// SmsAuthStrategy.java:81 —— email 与此结构完全对称
String code = RedisUtils.getCacheObject(CAPTCHA_CODE_KEY + phoneNumber);
if (StringUtils.isBlank(code)) throw new CaptchaExpireException();  // 过期
return code.equals(smsCode);                                        // 匹配

没有密码概念，「能收到并填对验证码」就是身份证明。验证码由 /resource/sms/code、/resource/email/code 接口提前发送并存 Redis。

social — 唯一「先绑定才能登」

// SocialAuthStrategy.java:59
AuthResponse<AuthUser> response = SocialUtils.loginAuth(source, socialCode, socialState, ...);
List<SysSocialVo> list = sysSocialService.selectByAuthId(source + uuid);
if (CollUtil.isEmpty(list)) throw new ServiceException("你还没有绑定第三方账号，绑定后才可以登录！");
SysUserVo user = loadUser(list.getFirst().getUserId());  // 通过绑定关系找到本地用户
✅ 设计要点：第三方账号本身不是系统用户，必须先绑定到一个本地 sys_user（存在 sys_social 表）。登录时用 source+uuid 反查绑定的 userId。绑定流程是另一条线：已登录用户 → /auth/binding/{source} 拿授权 URL → 第三方回调 → /auth/social/callback 调 socialRegister() 落库（见 AuthController）。

xcx — 官方留的扩展模板

XcxAuthStrategy 演示了微信小程序 code → openid 的换取，但 loadUserByOpenid 是 TODO 桩（密钥、建用户逻辑需自行补全）。它的价值是示范「第 3 课的 LoginUser 可以被继承扩展」——这里用了 XcxLoginUser 多带一个 openid 字段。

二、注册流程（SysRegisterService）

// SysRegisterService.java:43
if (captchaEnabled) validateCaptcha(...);                     // 同密码登录的图形验证码
sysUser.setPassword(BCrypt.hashpw(password));                 // BCrypt 加盐哈希存储
if (用户名已存在) throw new UserException("user.register.save.error");
userService.registerUser(sysUser);

注册入口 AuthController.register() 还有一道开关：configService.selectRegisterEnabled()，系统没开注册直接拒绝。注意密码存储用 BCrypt.hashpw，校验用 BCrypt.checkpw——单向哈希，库里永远没有明文。

三、三类验证码对比

图形验证码短信验证码邮箱验证码
接口/auth/code/resource/sms/code/resource/email/code
生成hutool 验证码（random/math）4 位数字 + sms4j 发送4 位数字 + 邮件发送
存储Redis: CAPTCHA_CODE_KEY + uuidRedis: key + phoneNumberRedis: key + email
限流IP 维度 60s/10 次手机号 60s/1 次邮箱 60s/1 次
开关captcha.enable—mail.enabled

都用 @RateLimiter 注解防刷。图形验证码若是 math 类型，还会用 SpEL 算出表达式结果存库（见 CaptchaController.getCodeImpl）。

💡 统一约定：所有验证码都存在 Redis 的 GlobalConstants.CAPTCHA_CODE_KEY 前缀下，带过期时间（Constants.CAPTCHA_EXPIRATION 分钟）。校验失败分两种异常：CaptchaExpireException（过期/不存在）和 CaptchaException（值不匹配），便于前端区分提示。
🤔 想一想：sms 和 email 策略里，为什么校验验证码也包在 loginService.checkLogin(...) 里，而不是直接比对？ 为了复用失败次数限制。checkLogin 不只是「跑一次校验」，它在校验前后维护了 Redis 里的失败计数：超过阈值就锁定该账号一段时间。把验证码比对作为 supplier 传进去，短信/邮箱登录就自动获得了和密码登录一致的「防爆破」能力。这是「把横切逻辑收敛到一个方法」的复用设计。

四、检验一下（提取练习）

Q1. 五种登录策略中，哪一种必须「先完成绑定」才能登录？

A. password B. sms C. social（第三方账号需先绑定到本地 sys_user） D. xcx
看答案C。social 策略用 source+uuid 查 sys_social 绑定关系，查不到就抛「你还没有绑定第三方账号」。绑定通过 /auth/social/callback → socialRegister() 完成。

Q2. 用户密码在数据库里以什么形式存储？登录时如何校验？

A. 明文存储，字符串相等比较 B. BCrypt 哈希存储（hashpw），登录用 checkpw 比对，库中无明文 C. MD5 D. AES 加密，登录时解密比较
看答案B。注册 BCrypt.hashpw，登录 BCrypt.checkpw，单向哈希加盐，无法反解。AES 是用于「接口传输加密」（第 6 课），别和密码存储混淆。

Q3. 短信验证码发送接口的限流策略是？

A. 不限流 B. 按手机号 60 秒最多 1 次（@RateLimiter key=#phoneNumber） C. 全局每秒 1 次 D. 按 IP 每分钟 100 次
看答案B。@RateLimiter(key="#phoneNumber", time=60, count=1)。图形验证码才是 IP 维度 60s/10 次。
