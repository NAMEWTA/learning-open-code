第 16 课 · StpInterface 权限回调：SaPermissionImpl 详解 plus-backend

🎯 本节目标：理解 Sa-Token 的 StpInterface 回调机制——什么时候被调用、谁来调、为什么这样设计。重点分析 SaPermissionImpl 的双源权限加载策略（本地缓存优先 → 降级到数据库查询）和 loginId 格式解析。

一、StpInterface：一个接口，两个方法，Sa-Token 自动回调

// Sa-Token 框架定义（非本项目代码）
public interface StpInterface {
    List<String> getPermissionList(Object loginId, String loginType);
    List<String> getRoleList(Object loginId, String loginType);
}

// SaTokenConfig.java:38-41 — 注册实现
@Bean
public StpInterface stpInterface() {
    return new SaPermissionImpl();   // Spring Bean 注入
}

这个接口实现了策略模式/回调模式：Sa-Token 框架定义了"我需要权限列表"的协议，但不关心你怎么获取。你的实现可以查数据库、调微服务、读缓存——只要返回 List<String> 即可。

调用时机： 每当代码中调用 StpUtil.checkPermission("system:user:list") 或使用 @SaCheckPermission 注解时，Sa-Token 内部会回调 stpInterface.getPermissionList(loginId, loginType) 获取当前用户的完整权限列表，然后做 contains() 判断。

⚠️ 性能注意：StpInterface 的 getPermissionList 在每次权限校验时都会调用（除非配置了缓存）。RuoYi 的优化策略是：登录时把权限预加载到 LoginUser 对象 → Session 缓存 → getPermissionList 优先从 LoginUser 的字段读取，不查数据库。

二、SaPermissionImpl 的双源策略

// SaPermissionImpl.java:56-72
private List<String> resolvePermissionList(
    Object loginId,
    Function<LoginUser, Collection<String>> localExtractor,   // ①
    BiFunction<PermissionService, Long, Collection<String>> remoteExtractor // ②
) {
    LoginUser loginUser = LoginHelper.getLoginUser();  // ③ 尝试从当前 Token-Session 获取
    if (ObjectUtil.isNull(loginUser) || !loginUser.getLoginId().equals(loginId)) {
        // ④ Session 中没有或 loginId 不匹配 → 降级查数据库
        PermissionService permissionService = SpringUtils.getBean(PermissionService.class);
        return new ArrayList<>(remoteExtractor.apply(permissionService, resolveUserId(loginId)));
    }
    // ⑤ Session 中有且匹配 → 直接空返回（权限已在 LoginUser 里）
    Collection<String> permissionList = localExtractor.apply(loginUser);
    if (CollUtil.isNotEmpty(permissionList)) {
        return new ArrayList<>(permissionList);
    }
    return new ArrayList<>();
}

这是一个两级回退（fallback）策略：

需要权限列表 │ ├─ ① 取当前 Token-Session 中的 LoginUser │ ├─ 有 → loginId 匹配？ │ │ ├─ 匹配 → 从 LoginUser.getMenuPermission() / getRolePermission() 取 → 返回 │ │ └─ 不匹配 → 走远程查询（②） │ └─ 没有 → 走远程查询（②） │ └─ ② 远程查询 └─ SpringUtils.getBean(PermissionService.class) └─ permissionService.getMenuPermission(userId) / getRolePermission(userId)
💡 为什么需要 loginId 匹配检查？ 因为 Sa-Token 支持"以他人身份查询权限"的场景（如管理员查看某用户的权限），此时传入的 loginId 可能不是当前请求用户的。如果直接返回当前 LoginUser 的权限就错了。这个检查保证了跨用户查询时的正确性。

三、loginId 格式：userId 的提取逻辑

// SaPermissionImpl.java:93-100
private Long resolveUserId(Object loginId) {
    String loginIdStr = loginId.toString();      // 例如："sys_user:1"
    int separatorIndex = loginIdStr.indexOf(':');
    if (separatorIndex < 0 || separatorIndex == loginIdStr.length() - 1) {
        throw new ServiceException("登录ID格式错误");
    }
    return Long.parseLong(loginIdStr.substring(separatorIndex + 1));  // 提取 "1" → 1L
}

RuoYi 的 loginId 格式为 用户类型:用户ID，例如 sys_user:1。这个设计服务于多用户体系——sys_user 是后台用户，还可能有 app_user（App 用户）。resolveUserId 通过冒号分割提取出纯数字 userId，用于数据库查询。

四、设计启示：SPI 扩展点模式

这套设计的精华在于扩展点（Extension Point）：

  1. Sa-Token 定义 StpInterface 作为扩展契约
  2. 你的项目通过 Spring Bean 注入自己的实现
  3. 框架在需要时自动发现并回调你的实现
  4. 实现可以被替换——比如切换到 OAuth2 或 LDAP，只需改一个 Bean，整个系统的鉴权调用方零改动

这与 Java SPI、Spring Boot AutoConfiguration 设计哲学一致：定义接口 → 允许替换 → 约定优于配置。
