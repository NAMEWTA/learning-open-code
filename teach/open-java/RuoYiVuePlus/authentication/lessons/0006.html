第 6 课 · 接口加解密在登录链路中的位置 传输安全

🎯 本节目标（对应使命 Success #7）：说清接口加解密（@ApiEncrypt + AES/RSA）在登录链路中的位置与作用。这是第 1 课全景图里被一笔带过的「第 2.5 步」，也是前端 isEncrypt 标记的归宿。

登录接口的请求头里有 isEncrypt: true。意味着：账号密码不是明文上网的。这一课讲清这套「混合加密」怎么运作，以及它和密码 BCrypt、token 是三件完全不同的事。

一、先厘清三种「加密」，别再混淆

名字解决什么算法在哪
密码存储库被脱也拿不到明文密码BCrypt（单向哈希）注册/登录校验（第 5 课）
token身份凭证、可验签JWT 签名签发/校验（第 3 课）
接口加解密请求/响应体在传输中不被窃听AES + RSA 混合本课，CryptoFilter

本课只讲第三种。它和前两种正交——一次登录请求会同时用到全部三种。

二、为什么是「AES + RSA 混合」而不是单用一种？

  • AES（对称）：加解密快，适合加密大的请求体；但密钥怎么安全送给对方是难题。
  • RSA（非对称）：公钥加密、私钥解密，适合传密钥；但慢，不适合加密大数据。
✅ 混合方案取两者之长：用一次性 AES 密钥加密请求体（快），再用 RSA 公钥加密这个 AES 密钥（安全传钥）。这正是 HTTPS / TLS 的经典思路。RuoYi 把它做在应用层，即便没上 HTTPS 或想多一层防护时也能用。

三、前端做了什么（request.ts + crypto.ts）

// plus-ui-vue/src/utils/request.ts:155
if (isEncrypt && (method==='post' || method==='put')) {
    const aesKey = generateAesKey();                       // ① 随机生成一次性 AES 密钥
    config.headers['encrypt-key'] = encrypt(encryptBase64(aesKey)); // ② RSA 公钥加密 AES 密钥，放头
    config.data = encryptWithAes(JSON.stringify(config.data), aesKey); // ③ AES 加密请求体
}

注意 encrypt() 是 RSA（jsencrypt），encryptWithAes() 是 AES（crypto-js，ECB/Pkcs7）。AES 密钥每次请求新生成，用完即弃。

四、后端做了什么（CryptoFilter）

关键：解密发生在 Controller 之前的 Filter 层，所以 Controller 拿到的已是明文。

// CryptoFilter.java:64
String headerValue = request.getHeader(properties.getHeaderFlag()); // "encrypt-key"
if (StringUtils.isNotBlank(headerValue)) {
    // 有加密头 → 包一层解密 Wrapper（用 RSA 私钥解出 AES 密钥，再 AES 解请求体）
    requestWrapper = new DecryptRequestBodyWrapper(servletRequest, privateKey, headerFlag);
} else {
    // 没有加密头，但接口标了 @ApiEncrypt（强制加密）→ 拒绝！
    if (ObjectUtil.isNotNull(apiEncrypt)) {
        handlerExceptionResolver.resolveException(..., new ServiceException("没有访问权限...", 403));
        return;
    }
}
💡 @ApiEncrypt 是「强制」加密注解：标了它的接口（/auth/login、/auth/register），如果请求没带加密头就直接拒绝。防止有人绕过前端、明文打登录接口。response()=true 时还会把响应体也加密回去（用 EncryptResponseBodyWrapper）。

完整往返

前端 后端 ──── ──── 明文 {username, password} │ ① 生成 AES 密钥 k │ ② RSA 公钥加密 k → encrypt-key 头 │ ③ AES(k) 加密 body ▼ 密文 body + encrypt-key 头 ───────► CryptoFilter │ 读 encrypt-key 头 │ RSA 私钥解出 k │ AES(k) 解 body ▼ Controller 拿到明文 {username, password} （继续走第 3~6 步登录）
🤔 想一想：CryptoFilter 为什么必须做在 Filter 层，而不是用 Spring 的 @RequestBody 反序列化后再解密？ 因为请求体是一次性流，@RequestBody 一旦读取并反序列化，流就消费完了，后面再想解密原始密文已经来不及。Filter 在最外层用 DecryptRequestBodyWrapper 包装原始请求，重写 getInputStream() 返回解密后的内容，这样下游 Spring MVC 拿到的「就像」一开始就是明文，无感接入。这也是为什么登录 Controller 能用 @RequestBody String body 直接拿到明文 JSON。

五、检验一下（提取练习）

Q1. 接口加解密里，AES 密钥本身是怎么安全传到后端的？

A. 明文放在请求头 B. 用 RSA 公钥加密后放在 encrypt-key 头，后端用 RSA 私钥解出 C. 写死在前后端代码里 D. 通过 token 传递
看答案B。这就是混合加密：AES 加密数据（快），RSA 加密 AES 密钥（安全传钥）。AES 密钥每次请求随机生成。

Q2. 一个标了 @ApiEncrypt 的接口，收到一个没有加密头的明文请求，会怎样？

A. 正常处理 B. 被 CryptoFilter 拒绝，返回 403「没有访问权限」 C. 自动加密后处理 D. 返回 401
看答案B。@ApiEncrypt 是强制加密：无加密头则 Filter 直接用 handlerExceptionResolver 抛 403，防止绕过前端明文打接口。

Q3. 接口传输加密（AES/RSA）和密码存储加密（BCrypt）的关系是？

A. 是同一回事 B. 完全独立的两件事：前者保护「传输中」，后者保护「数据库里」，一次登录两者都用 C. 互斥，开了一个就不用另一个 D. BCrypt 可以解密 AES
看答案B。正交关系。请求体 AES 加密 → 传到后端解密 → 拿到明文密码 → 用 BCrypt.checkpw 和库里的哈希比对。两层各管一段。
