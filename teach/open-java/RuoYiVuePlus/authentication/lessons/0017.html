第 17 课 · useStorage 语法 + 前后端白名单对比 plus-ui-vue plus-backend

🎯 本节目标：① 理解 useStorage<null | string>(TokenKey, null) 的类型语法和底层原理；② 对比前端的 whiteList 路由守卫白名单、后端的 @SaIgnore 注解白名单、以及 security.excludes 配置白名单——理解它们的职责边界和互补关系。

一、useStorage 类型语法详解

// utils/auth.ts:1-3
const TokenKey = 'Admin-Token';
const tokenStorage = useStorage<null | string>(TokenKey, null);
//                   ^^^^^^^^^ 泛型参数   ^key      ^初始值

export const getToken = () => tokenStorage.value;       // 读取
export const setToken = (t: string) => (tokenStorage.value = t); // 写入
export const removeToken = () => (tokenStorage.value = null);   // 清除

useStorage 是 VueUse 库提供的组合式函数，返回一个 Ref。逐层拆解：

部分含义等价理解
useStorageVueUse 函数创建一个与 localStorage/sessionStorage 双向绑定的响应式引用
<null | string>TypeScript 泛型这个 ref 的值可以是 null 或 string
TokenKeystorage keylocalStorage 的 key 名 = 'Admin-Token'
null初始值当 localStorage 中没有此项时，ref.value 返回 null

核心特性：写入 tokenStorage.value = 'abc' 会自动同步到 localStorage；其他标签页修改 localStorage 也会自动同步到当前页面的 ref（通过 storage 事件监听）。这就是"响应式持久化"——你只需要像普通 ref 一样读写，不用手动调 setItem/getItem。

💡 为什么用 null | string 而不是 string？ 因为没有登录时 token 不存在，null 正确表达了「空」的语义。removeToken() 设置为 null 而非 ''（空字符串），避免 Auth header 变成 Bearer （没有 token 值但有个空 Bearer）。

二、前端白名单：路由守卫的第一道门

// permission.ts:16-20
const whiteList = ['/login', '/register', '/social-callback', '/register*', '/register/*'];

const isWhiteList = (path: string) => {
  return whiteList.some(pattern => isPathMatch(pattern, path));
};

isPathMatch （utils/validate.ts:7-16）支持通配符：* 匹配单层路径段，** 匹配任意层级。例如 /register* 能匹配 /register、/register/step1 等。

白名单的作用：在路由守卫（router.beforeEach）中，未登录用户只能访问白名单页面；已登录用户访问白名单页面不会被拦截。

router.beforeEach(to, from) │ ├─ 有 token？ │ ├─ 是 → to.path === '/login'？ → 已登录去登录页 → 重定向到 / │ │ → isWhiteList(to.path)？ → 在白名单直接放行 │ │ → 否则 → 检查 roles → 拉取用户信息 → 生成动态路由 │ └─ 否 → isWhiteList(to.path)？ │ ├─ 是 → 放行（登录、注册页） │ └─ 否 → 重定向到 /login?redirect=原页面

三、后端三层白名单：@SaIgnore、security.excludes、AllUrlHandler

层级位置作用粒度
@SaIgnoreAuthController 类上整个 Controller 所有方法跳过 Sa-Token 鉴权类/方法级
security.excludesapplication.yml静态资源路径（*.html, *.css, *.js）跳过拦截器路径模式
AllUrlHandlerSecurityConfig 中 SaRouter.match(allUrlHandler.getUrls())收集所有 URL → Sa-Token 对它们全部进行鉴权全局 URL

关键理解：

  • @SaIgnore 是 Sa-Token 注解，放在 AuthController 类上意味着 /auth/login、/auth/register、/auth/logout 等都不走 SecurityConfig 的 SaInterceptor。登录接口当然不能要求已登录——这是鸡生蛋问题。
  • security.excludes 是 Spring Boot 配置，排除的是静态资源（HTML/CSS/JS/favicon），这些不需要鉴权。它作用于 Interceptor 注册层面。
  • AllUrlHandler 在启动时通过 RequestMappingHandlerMapping 收集了系统中所有注册的 URL，然后 SaRouter.match(urls).check(...) 对它们进行鉴权。但 @SaIgnore 注解的方法会被 Sa-Token 内部跳过。

四、为什么前端和后端都需要白名单？

这是一个常见的困惑。答案：前端白名单和后端白名单解决的是不同层次的问题。

维度前端白名单（Vue Router）后端白名单（@SaIgnore / excludes）
目的用户体验：未登录用户看到的是登录页而非空白安全性：不经过认证的请求不能访问受保护资源
谁决定的前端路由配置后端安全配置
可绕过吗可以——用户可以直接改 URL 或 curl不能——后端拦截器在请求处理前执行
不配置的后果未登录用户看到空白或报错页面任何人都能访问敏感接口
⚠️ 安全原则：前端白名单只是UX 优化，不是安全措施。后端白名单才是真正的安全边界。前端路由守卫可以被绕过（curl、Postman、浏览器 DevTools 改 JS），但后端拦截器是绕不过的。

练习：对比分析

❓ /auth/login 同时在「前端 whiteList + 后端 @SaIgnore」中。如果后天加了 /auth/wechat-login 但前端忘了加白名单，用户能正常使用吗？

点击查看答案

不能。用户点击「微信登录」按钮时，前端路由守卫检测到没有 token 且 /auth/wechat-login 不在白名单，会直接重定向到 /login，用户根本到不了微信登录页。后端虽然有 @SaIgnore 允许请求通过，但前端先拦了。正确做法：前端和后端的白名单需要保持同步——任何后端开放接口，前端对应的路由也必须加入白名单。
