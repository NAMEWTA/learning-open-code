第 14 课 · Sa-Token Session 存储模型与 LoginHelper plus-backend

🎯 本节目标：理解 StpUtil.getTokenSession().get(LOGIN_USER_KEY) 中的两层存储模型——Token ↔ Token-Session 的映射关系，以及为什么 LoginUser 存在 Session 而非直接存在 Token 里。

一、Sa-Token 的三层存储模型

Sa-Token 的存储不是简单的 "token → user"，而是三层嵌套：

Token (字符串) │ └── Token-Session ── 专属于此 token 的会话空间 │ ├── "loginUser" → LoginUser 对象 ← 你的代码在这里读写 ├── "userId" → 用户ID快照 ← LoginHelper 辅助字段 └── （其他自定义数据） Account-Session ── 专属于此 loginId 的会话空间（跨 token 共享，一个用户多个设备间共享） │ └── （用户级别的配置、缓存等）
层次key生命周期典型数据
Token 本体token 字符串token 创建→过期token 值、loginId、过期时间
Token-Sessiontoken → session 映射跟 token 同生共死LoginUser、userId、clientId 扩展字段
Account-SessionloginId → session 映射独立于 token跨设备共享配置

二、login() 方法：两步写入

// LoginHelper.java:54-64
public static void login(LoginUser loginUser, SaLoginParameter model) {
    fillRequestContext(loginUser, model);     // ① 补充 IP/浏览器/设备信息

    StpUtil.login(loginUser.getLoginId(),     // ② Sa-Token 登录（生成 token）
        model.setExtra(USER_KEY, loginUser.getUserId())     // 写入 token 扩展字段
             .setExtra(CLIENT_KEY, clientId)                // 客户端ID
             .setExtra(CLIENT_ACCESS_PATH_KEY, accessPath)  // 访问路径白名单
    );

    StpUtil.getTokenSession()                 // ③ 获取刚创建的 Token-Session
        .set(LOGIN_USER_KEY, loginUser);      // ④ 存入 LoginUser 完整对象
}

两步操作对应两个存储位置：setExtra() 写到 token 里（轻量字段），getTokenSession().set() 写到 token 专属的 Session 里（重量对象）。

💡 为什么 LoginUser 不直接存 token 里？ token 存储的是 key-value 格式的扩展字段，适合放轻量数据（userId、clientId 等）。LoginUser 是包含角色、权限等的大对象，放 Session 里更合适。而且 Session 可以设置独立过期策略。

三、getLoginUser()：取值链路

// LoginHelper.java:102-108
public static <T extends LoginUser> T getLoginUser() {
    try {
        return getLoginUser(StpUtil.getTokenSession());  // ① 先拿当前 Token-Session
    } catch (NotLoginException e) {
        return null;                                      // ② 未登录返回 null
    }
}

// LoginHelper.java:136-141
private static <T extends LoginUser> T getLoginUser(SaSession session) {
    return (T) session.get(LOGIN_USER_KEY);               // ③ 从 Session 里取 loginUser
}

StpUtil.getTokenSession() 做了三件事：

  1. 从请求头/参数中提取 token： 根据配置的 token-name: Authorization，从 Bearer xxx 中提取 token 字符串
  2. 校验 token 有效性： 检查 token 是否存在、是否过期、是否被冻结/踢出。如果无效，直接抛出 NotLoginException（这就是 SecurityConfig 拦截器能感知到未登录的原因）
  3. 返回对应的 SaSession 对象： 通过 SaTokenDao（本项目用 PlusSaTokenDao）从 Redis 中查出 Token-Session
⚠️ 关键理解：getTokenSession() 第 2 步的校验逻辑本质上等价于先调用了一次 StpUtil.checkLogin()。所以 SecurityConfig 的拦截器里调用 StpUtil.checkLogin() 是用最低成本只校验不取数据，业务代码里调用 LoginHelper.getLoginUser() 是校验+取数据一步到位。

四、设计启示：三层分离的好处

这套设计解决了三个问题：

  1. Token 小而快： JWT 模式下 token 就是 JWT，不能太大。userId 塞进 token，LoginUser 留 Session
  2. 多端独立： 同一用户在 PC 端和手机端各有一个 token → 各有一个 Token-Session → forceLogout(token) 只踢一个设备
  3. 扩展弹性： 想加字段？追加 setExtra() 或 session.set()，不破坏现有结构

练习：暂停回忆

❓ StpUtil.getTokenSession() 内部如果发现 token 已过期，是返回 null 还是抛异常？

点击查看答案

抛 NotLoginException。这就是 Sa-Token 的设计哲学：异常即状态。调用方要么 try-catch 处理，要么让全局异常处理器统一处理。SaTokenExceptionHandler 就是用 @ExceptionHandler(NotLoginException.class) 统一返回 401 错误码。
