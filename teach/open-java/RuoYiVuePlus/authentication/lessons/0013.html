第 13 课 · @ApiEncrypt 注解与 CryptoFilter 全链路 plus-backend

🎯 本节目标：理解 @ApiEncrypt 注解如何通过 Servlet Filter（非 Spring AOP）实现请求解密 + 响应加密。重点：为什么用 Filter 而非 Interceptor 或 AOP？因为 Filter 能拿到 HttpServletRequest 原始流，而流只能读一次。

一、先看注解：一个标记，两个语义

// ApiEncrypt.java
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface ApiEncrypt {
    boolean response() default false;  // 是否加密响应
}

放在 Controller 方法上，比如 AuthController.login()：

// AuthController.java:75-76
@ApiEncrypt                       // 请求解密（默认行为）
@PostMapping("/login")
public R<LoginVo> login(@RequestBody String body) { ... }

这个注解有两个语义：

  1. 标记接口需要请求解密（无论 response 值为多少，注解存在即触发）
  2. 标记接口需要响应加密（仅当 response = true 时）
⚠️ 注意：参数类型是 @RequestBody String body 而非对象。因为 Filter 解密后的结果直接是 JSON 字符串，需要 Controller 内部手动 JsonUtils.parseObject(body, LoginBody.class) 转换。

二、为什么用 Filter？流只能读一次

Servlet 的请求流（InputStream）只能读取一次。如果用 Interceptor 或 AOP，此时流已经被 Spring MVC 的参数解析器读走了，再也拿不到原始密文。Filter 是在参数解析之前执行的：

请求到达 │ ├─ Filter ← CryptoFilter 在这里！解密请求体，包装成新的 RequestWrapper │ ├─ DispatcherServlet │ └─ Interceptor (SaInterceptor 鉴权) │ └─ HandlerAdapter 参数解析 → 读到的是已解密的明文 │ └─ AOP (@ApiEncrypt 对 Controller 方法无效，因为流已消费) │ └─ Controller 方法执行 │ └─ Filter 响应返回路径 ← CryptoFilter 在这里加密响应体
💡 设计启示：当需要操作原始请求流/响应流时，Filter 是唯一选择。拦截器和 AOP 只能操作参数解析后的对象。这就是为什么 Spring Security、Sa-Token 的认证过滤器也是 Filter——它们都需要在参数解析前工作。

三、CryptoFilter.doFilter：四阶段流水线

// CryptoFilter.java:58-104
public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {
    // 阶段 1：查找 @ApiEncrypt 注解
    ApiEncrypt apiEncrypt = this.getApiEncryptAnnotation(servletRequest);

    // 阶段 2：请求解密（POST/PUT + 有 encrypt-key 请求头）
    if (isPostOrPut && StringUtils.isNotBlank(headerValue)) {
        requestWrapper = new DecryptRequestBodyWrapper(req, privateKey, headerFlag);
    } else if (apiEncrypt != null) {
        // 有注解但没加密头 → 403 拒绝
        throw new ServiceException("没有访问权限");
    }

    // 阶段 3：包装响应（如果注解要求 response=true）
    if (apiEncrypt != null && apiEncrypt.response()) {
        responseWrapper = new EncryptResponseBodyWrapper(res);
    }

    // 阶段 4：放行 → 执行后续 Filter → Controller → 回到这里
    chain.doFilter(requestWrapper, responseWrapper);

    // 阶段 4 续：响应加密写出
    if (responseFlag) {
        String encrypted = responseWrapper.getEncryptContent(res, publicKey, headerFlag);
        res.getWriter().write(encrypted);
    }
}

核心设计：装饰器模式——DecryptRequestBodyWrapper 和 EncryptResponseBodyWrapper 都继承了 HttpServletRequestWrapper/HttpServletResponseWrapper，重写 getInputStream()/getOutputStream()，对下游（Controller、Spring MVC）完全透明。

四、getApiEncryptAnnotation：运行时反射

// CryptoFilter.java:112-129
private ApiEncrypt getApiEncryptAnnotation(HttpServletRequest req) {
    HandlerExecutionChain chain = requestMappingHandlerMapping.getHandler(req);
    if (chain != null) {
        Object handler = chain.getHandler();
        if (handler instanceof HandlerMethod hm) {
            return hm.getMethodAnnotation(ApiEncrypt.class);  // 反射拿注解
        }
    }
    return null;
}

这里用到 Spring MVC 的 RequestMappingHandlerMapping，本质上等价于「给定请求 URL，找到对应的 Controller 方法，反射获取其 @ApiEncrypt 注解」。这个技巧在自定义 Filter/Interceptor 中非常通用。

练习：对比思考

❓ 如果把 @ApiEncrypt 放在类上而非方法上，CryptoFilter 会识别到吗？为什么？

点击查看答案

不会。CryptoFilter 只调用 handlerMethod.getMethodAnnotation(ApiEncrypt.class)，不检查类级注解。如果需要，应该同时调用 handlerMethod.getBeanType().getAnnotation(ApiEncrypt.class)。这是当前实现的一个边界。
