第 2 课 · 客户端体系 + grantType + 策略模式 设计动机

🎯 本节目标（对应使命 Success #2）：讲清「客户端（sys_client）+ grantType + 策略模式」这套设计解决了什么问题，以及新增一种登录方式时，系统的哪几处会参与。这是 第 1 课全景图里第 3、4 步的放大。

上一课我们看到登录请求带了两个字段：clientId 和 grantType。这一课我们回答：为什么要有这两个东西？少了它们会怎样？

一、问题：一个系统要面对多少种「登录」？

想象 RuoYi 要支撑的真实场景：

  • PC 管理后台 → 账号密码 + 图形验证码登录
  • 手机 App → 短信验证码登录
  • 微信小程序 → openid 静默登录
  • 对接企业 SSO / Gitee / GitHub → 第三方社交登录

这些登录方式入参不同、校验逻辑不同、token 有效期诉求也不同（App 的 token 可能要存 30 天，后台只存几小时）。如果全塞进一个 if/else` 大方法，会迅速腐烂成谁也不敢动的 800 行怪物。RuoYi 用两个正交的维度把它拆开。

二、维度一：grantType + 策略模式（「怎么校验」）

每一种登录方式 = 一个实现 IAuthStrategy 接口的策略 Bean。关键在 Bean 的命名约定：

// 五个策略类的注解，注意 Bean 名字
@Service("password" + IAuthStrategy.BASE_NAME)  // → "passwordAuthStrategy"
@Service("sms" + IAuthStrategy.BASE_NAME)       // → "smsAuthStrategy"
@Service("email" + IAuthStrategy.BASE_NAME)     // → "emailAuthStrategy"
@Service("social" + IAuthStrategy.BASE_NAME)    // → "socialAuthStrategy"
@Service("xcx" + IAuthStrategy.BASE_NAME)       // → "xcxAuthStrategy"

分发时不靠 switch，而是用 grantType 直接拼出 Bean 名去容器里取：

// IAuthStrategy.java:31
static LoginVo login(String body, SysClientVo client, String grantType) {
    String beanName = grantType + BASE_NAME;        // "sms" + "AuthStrategy"
    if (!SpringUtils.containsBean(beanName)) {
        throw new ServiceException("授权类型不正确!");
    }
    IAuthStrategy instance = SpringUtils.getBean(beanName);
    return instance.login(body, client);
}
✅ 这样设计的好处：新增一种登录方式，只需写一个新策略类、起对名字，AuthController 和分发逻辑一行都不用改——这就是「对扩展开放，对修改关闭」的开闭原则落地。差异被关进各策略，主干永远干净。

三、维度二：sys_client（「谁来登录、token 怎么配」）

sys_client 是一张配置表，每行是一个接入端。看它的视图对象 SysClientVo 字段：

字段含义在认证里的作用
clientId客户端唯一 id登录请求带它，后端据此查配置
grantType该端允许的授权类型（逗号分隔）限制「这个端只能用这些登录方式」
deviceType设备类型（pc/app…）Sa-Token 多设备登录的区分维度
timeout / activeTimeouttoken 固定/活跃超时每个端 token 有效期可不同
accessPath允许访问的路径拦截器据此限制接口范围（第 4 课）
ipWhitelistIP 白名单拦截器据此限制来源 IP（第 4 课）
status状态（0 正常 1 停用）停用的客户端直接拒绝登录

AuthController 在分发前先做两道客户端校验：

// AuthController.java:83
SysClientVo client = clientService.queryByClientId(clientId);
// ① 查不到 client，或 client 不允许这个 grantType → 拒绝
if (ObjectUtil.isNull(client) || !StringUtils.contains(client.getGrantType(), grantType)) {
    return R.fail(MessageUtils.message("auth.grant.type.error"));
// ② 客户端被停用 → 拒绝
} else if (!SystemConstants.NORMAL.equals(client.getStatus())) {
    return R.fail(MessageUtils.message("auth.grant.type.blocked"));
}
LoginVo loginVo = IAuthStrategy.login(body, client, grantType);  // 通过才分发
💡 两个维度正交：grantType 决定怎么校验身份，sys_client 决定谁能用这种方式、token 怎么配。同一个 grantType（如 sms）可被多个客户端共用，同一个客户端也能开放多个 grantType。组合自由。

四、串起来：新增「钉钉扫码登录」要动哪几处？

这是检验你是否真懂这套设计的试金石。答案是三处：

  1. 写一个 DingTalkAuthStrategy implements IAuthStrategy，注解 @Service("dingtalk" + BASE_NAME)，在 login() 里实现钉钉的身份校验 + LoginHelper.login()。
  2. 在 sys_client 表里，给需要的客户端的 grantType 字段追加 dingtalk。
  3. 前端登录时传 grantType: 'dingtalk' 和对应入参。

AuthController、IAuthStrategy、LoginHelper、拦截器——全都不用改。 这就是这套设计的回报。

🤔 想一想：分发用「拼 Bean 名」而不是 switch(grantType)，除了开闭原则，还有什么隐藏好处？ 解耦 + 可插拔。各策略是独立的 Spring Bean，能各自注入自己需要的依赖（密码策略要 CaptchaProperties，社交策略要 SocialProperties 和 ISysSocialService）。用 switch 的话所有依赖都得堆在一个类里。而且策略还能放在不同模块/jar 里，靠 Bean 名约定「即插即用」。

五、检验一下（提取练习）

Q1. 同一个 grantType:'sms' 的登录请求，带不同的 clientId，会发生什么？

A. 报错，一个 grantType 只能绑一个客户端 B. 走同一个 smsAuthStrategy 校验，但 token 超时/设备类型按各自客户端配置生成 C. 完全一样，clientId 无意义 D. 第二个客户端会把第一个踢下线
看答案B。策略由 grantType 决定（都走 smsAuthStrategy），但 buildLoginParameter(client) 会按各自 SysClientVo 的 timeout/deviceType 生成不同的登录参数。这正是两个维度正交的体现。

Q2. 如果前端传了 grantType:'wechat' 但没有对应的策略 Bean，会在哪一步、怎么失败？

A. 前端 axios 拦截器拦下 B. AuthController 客户端校验阶段（若 client.grantType 不含 wechat）或 IAuthStrategy 分发时 containsBean 为 false 抛「授权类型不正确」 C. 数据库报错 D. 静默成功，返回空 token
看答案B。两道防线：先在 Controller 校验 client.getGrantType() 是否包含该类型；即便包含，分发时 SpringUtils.containsBean("wechatAuthStrategy") 为 false 也会抛 ServiceException。
