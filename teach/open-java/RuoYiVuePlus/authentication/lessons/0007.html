第 7 课 · 前端 Vue 鉴权对接 plus-ui-vue

🎯 本节目标（对应使命 Success #6 的 Vue 部分）：讲清 Vue 前端（Pinia + axios + Vue Router 守卫）如何存 token、加请求头、处理 401、做路由与按钮级权限。后端讲完了，这一课回到第 1 课全景图的左半边。

前端鉴权可拆成四个职责，各有一个文件主管。记住这张责任分工表，代码就不难找：

职责主管文件关键动作
存 tokenutils/auth.tsget/set/removeToken
登录态 + 用户信息store/modules/user.tslogin / getInfo / logout
请求加工 + 401utils/request.ts加头/加密、响应码处理
路由准入 + 动态路由permission.ts守卫、生成路由

一、存 token：极简的 auth.ts

// utils/auth.ts
const TokenKey = 'Admin-Token';
const tokenStorage = useStorage<null | string>(TokenKey, null);  // VueUse，响应式持久化
export const getToken = () => tokenStorage.value;
export const setToken = (t: string) => (tokenStorage.value = t);
export const removeToken = () => (tokenStorage.value = null);

就存后端返回的 access_token 字符串，落在浏览器存储里。前端不解析 token 内容——它对 JWT 内部一无所知，只负责「带着它请求」。

二、登录态：userStore（Pinia）

// store/modules/user.ts
const login = async (userInfo) => {
    const res = await loginApi(userInfo);
    setToken(res.data.access_token);   // 存 token
    token.value = res.data.access_token;
};
const getInfo = async () => {          // 登录后单独拉
    const res = await getUserInfo();
    roles.value = res.data.roles;       // 角色编码 → 路由权限
    permissions.value = res.data.permissions;  // 权限编码 → 按钮权限
    name.value = res.data.user.userName;
};
const logout = async () => {
    await logoutApi(); token.value=''; roles.value=[]; permissions.value=[]; removeToken();
};
✅ 印证第 1 课 Q3：login 只拿到 token，roles/permissions 是 getInfo 单独拉的。这是关注点分离——登录只管「换凭证」，用户画像与权限是另一次请求。

三、请求加工：request.ts 拦截器（呼应后端）

请求拦截器——给后端拦截器喂料

// 默认头：clientid（第 4 课客户端一致性校验要用它！）
axios.defaults.headers['clientid'] = import.meta.env.VITE_APP_CLIENT_ID;
// 每个请求带 token（登录接口标 isToken:false 则不带）
if (getToken() && !isToken) config.headers['Authorization'] = 'Bearer ' + getToken();
// isEncrypt 的接口做 AES+RSA 加密（第 6 课）
// repeatSubmit:false 之外的 post/put 做 500ms 防重提交

这里的 clientid 头不是可有可无——第 4 课后端拦截器会校验它和 token 里的 clientid 一致，否则 401。

响应拦截器——401 的「温柔」处理

// request.ts:205
if (code === 401) {
    if (!isRelogin.show) {
        isRelogin.show = true;
        ElMessageBox.confirm("登录状态已过期，您可以继续留在该页面，或者重新登录", ...)
            .then(() => { useUserStore().logout().then(() => router.replace('/login?redirect=...')); });
    }
    return Promise.reject('无效的会话...');
}
💡 两个细节：① isRelogin.show 是幂等锁——并发请求同时 401 时，只弹一次重登框，不会弹一堆。② 401 不强制跳转，而是询问，让用户可以「留在页面」避免丢失未保存数据。这个 401 正对应第 4 课后端 SaTokenExceptionHandler 返回的各种登录态异常。

四、路由准入：permission.ts 全局守卫

router.beforeEach： 有 token？ ├─ 否 → 白名单(/login 等)放行；否则跳 /login?redirect=... └─ 是 → ├─ 去 /login → 重定向到首页 ├─ roles 为空（刚登录/刷新）→ getInfo() 拉信息 │ 成功 → generateRoutes() 按权限生成动态路由 → addRoute → 放行 │ 失败 → logout() → 回首页 └─ roles 已有 → 直接放行

这是 RuoYi 动态路由的核心：菜单/路由不是写死的，而是登录后按当前用户的角色权限动态生成并 addRoute。所以刷新页面（store 清空）会重新 getInfo + 生成路由。

按钮级权限

路由级靠 roles 控制能不能进页面；按钮级（能不能看到「删除」按钮）靠 permissions 集合 + v-hasPermi 指令实现，数据同样来自 getInfo。后端则用 @SaCheckPermission 做最终校验——前端隐藏只是体验，后端校验才是安全。

🤔 想一想：为什么前端控制了按钮显隐，后端还要再用 @SaCheckPermission 校验一遍？不是重复吗？ 不重复，是纵深防御。前端的 v-hasPermi 只是「不显示按钮」，但攻击者完全可以绕过前端、直接构造 HTTP 请求打接口。前端权限控制是为了用户体验（不给看不该用的功能），后端 @SaCheckPermission 才是真正的安全边界。任何「只靠前端控制权限」的系统都是裸奔。这条铁律对 Vue / React 都成立。

五、社交登录的前端特殊一环

社交登录前端走 /auth/binding/{source} 拿第三方授权 URL → 跳转 → 第三方回调到前端 /social-callback 页（在白名单里）→ 该页拿 code/state 调后端 callback（grantType=social）完成登录。所以白名单里专门有 /social-callback。

六、检验一下（提取练习）

Q1. 刷新页面后，为什么会重新发一次 getInfo 请求？

A. Bug B. Pinia store 是内存态，刷新后 roles 清空；守卫发现 roles 为空就重新 getInfo 并生成动态路由 C. 为了刷新 token D. 后端要求
看答案B。token 在持久化存储里还在，但 store 的 roles/permissions 是内存态被清空，守卫据此重新拉取并 addRoute 重建动态路由。

Q2. 请求头里的 clientid 如果和 token 里记录的不一致，会发生什么？

A. 没影响 B. 后端拦截器客户端一致性校验失败，返回 401（呼应第 4 课） C. 前端报错 D. 自动修正
看答案B。前端 axios 默认带 clientid 头，后端 SaInterceptor 校验它与 token extra 里的 clientid 一致，不一致抛 NotLoginException（401）。前后端在此对接。

Q3. 前端 v-hasPermi 隐藏了删除按钮，能否替代后端权限校验？

A. 能，前端控制就够了 B. 不能，前端仅为体验，可被绕过；后端 @SaCheckPermission 才是安全边界 C. 看情况 D. 能，因为有 token
看答案B。纵深防御原则。前端权限控制可被绕过（直接打接口），后端校验才是真安全。
