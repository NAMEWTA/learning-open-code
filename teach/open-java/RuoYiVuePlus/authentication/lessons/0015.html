第 15 课 · 强制下线：NotLoginException 状态码与验签链路 plus-backend

🎯 本节目标：理解「强制下线」从触发到前端感知的完整链路——管理员点击下线 → Sa-Token 标记 token 状态 → 用户下次请求时 SaTokenDao 检测到异常状态 → NotLoginException 携带类型码 → SaTokenExceptionHandler 转换为人性化文案。

一、三种"下线"的区分

用户看到的都是"你被下线了"，但技术上有三种不同的触发路径：

场景NotLoginException type触发动作前端看到
Token 过期TOKEN_TIMEOUTTLL 到期自动清除"登录已过期"
顶号下线（BE_REPLACED）BE_REPLACED同一账号新登录时，旧 token 被标记"已在其他设备登录"
管理员强制踢出（KICK_OUT）KICK_OUT管理后台调用 StpUtil.kickout(loginId)"已被管理员强制下线"
Token 冻结TOKEN_FREEZE安全策略触发"登录已过期"

二、Sa-Token 的 Token 状态标记机制

Sa-Token 在 Redis 中存储的 token 数据结构（简化版）：

Key: "Authorization:login:token:{tokenValue}" Value: { "loginId": "sys_user:1", "loginType": "login", "tokenValue": "xxxxx", "isActive": true, ← 关键字段 "createTime": 1234567890, "timeout": 2592000 }

当管理员调用 StpUtil.kickout(loginId) 时：

  1. 找到此 loginId 关联的所有 token（一个用户可能有多个设备登录）
  2. 逐个执行 StpUtil.logoutByTokenValue(tokenValue) — 从 Redis 中删除 token 记录
  3. 或者标记 isActive = false（取决于配置）

下次该 token 的请求到达 SecurityConfig 拦截器 → StpUtil.checkLogin() 从 Redis 中查找 token → 发现不存在或 isActive = false → 抛出 NotLoginException，其 getType() 返回 KICK_OUT。

💡 配置驱动：application.yml:93 配置 is-concurrent: true（允许并发登录）意味着不会自动 BE_REPLACED。如果想启用自动顶号，设为 false 并配置 is-share: false。RuoYi 当前的设置是：允许多端登录，但只要管理员手动踢出就会标记 KICK_OUT。

三、SaTokenExceptionHandler：状态码 → 用户提示

// SaTokenExceptionHandler.java:44-56
@ExceptionHandler(NotLoginException.class)
public R<Void> handleNotLoginException(NotLoginException e, HttpServletRequest request) {
    String msg = switch (e.getType()) {
        case NotLoginException.TOKEN_TIMEOUT,
             NotLoginException.TOKEN_FREEZE -> "登录已过期，请重新登录";
        case NotLoginException.BE_REPLACED   -> "当前账号已在其他设备登录，您已被强制下线";
        case NotLoginException.KICK_OUT      -> "账号已被管理员强制下线";
        default                              -> "登录状态异常，请重新登录";
    };
    return R.fail(HttpStatus.HTTP_UNAUTHORIZED, msg);
}

这使用了 Java 17 的新 switch 表达式语法。不同类型码对应不同的用户提示：

  • TOKEN_TIMEOUT 和 TOKEN_FREEZE 合并处理——都提示"登录已过期"
  • BE_REPLACED（顶号）和 KICK_OUT（管理员踢）分开了——用户需要知道是谁把自己下线的
  • default 兜底——未知异常类型不暴露技术细节

前端收到 401 + msg 后，request.ts 的响应拦截器弹出 ElMessageBox.confirm："登录状态已过期，您可以继续留在该页面，或者重新登录"。

四、设计启示：异常驱动的状态机

Sa-Token 采用了异常驱动的状态验证模式，而非"先检查返回值"。这有三个好处：

  1. 全局异常处理器统一处理，Controller 代码零侵入
  2. 类型码携带语义（BE_REPLACED / KICK_OUT / TIMEOUT），前端可以根据不同码做不同 UI
  3. Filter 层/Interceptor 层/Service 层任何地方都能感知到状态异常，不需要层层传递返回值

练习：场景分析

❓ 用户 A 在 Chrome 登录了系统。管理员在后台点击「强退」按钮。用户 A 正在操作页面，此时点击了一个菜单——他会看到什么？整个链路有几个关键步骤？

点击查看答案

链路：① 管理员 → POST /monitor/online/强退 → StpUtil.kickout(userA_loginId) → Redis 中 userA 的 token 被标记/删除；② 用户 A 点击菜单 → 前端发请求 → SecurityConfig 拦截器 → StpUtil.checkLogin() → 从 Redis 查不到 token → 抛 NotLoginException(KICK_OUT)；③ SaTokenExceptionHandler 捕获 → 返回 R.fail(401, "账号已被管理员强制下线")；④ 前端 request.ts 响应拦截器检测到 401 → 弹出"登录已过期"提示并引导重新登录。
