第 9 课 · request.ts 请求拦截器全解析 plus-ui-vue

🎯 本节目标：逐行读懂 request.ts 的请求拦截器——axios 实例创建、自定义 header 管理、防重复提交、前端 AES 加密，理解为什么每段代码这样设计。

一、为什么需要 axios 实例？

// request.ts:97-107
axios.defaults.headers['Content-Type'] = 'application/json;charset=utf-8';
axios.defaults.headers['clientid'] = import.meta.env.VITE_APP_CLIENT_ID;

const service = axios.create({
  baseURL: import.meta.env.VITE_APP_BASE_API,
  timeout: 50000,
  transitional: { clarifyTimeoutError: true }
});

这 10 行代码做了一个关键决策：创建独立 axios 实例，而非修改全局 axios。

第 97-98 行是全局默认配置：所有通过 axios 发出的请求都会带上 Content-Type 和 clientid。注意 clientid 是 RuoYi 多客户端体系的核心——后端 SecurityConfig 会校验请求头中的 clientid 是否与 token 签发时一致。

第 100 行 axios.create 创建独立实例 service。这个实例有自己的拦截器，不会污染其他可能使用 axios 的地方。与第 97 行的区别：第 97 行是直接改 axios.defaults，影响全局；第 100 行是实例级配置。

💡 设计思路：clientid 设在全局默认值是「兜底策略」——即使某个请求忘了单独配置，也会带上这个字段。而后端会做二次校验，保证安全性不降级。

二、三大自定义 Header：isToken / isEncrypt / repeatSubmit

// request.ts:115-119 (请求拦截器内部)
const isToken = config.headers?.isToken === false;
const isRepeatSubmit = config.headers?.repeatSubmit === false;
const isEncrypt = config.headers?.isEncrypt === 'true';

if (getToken() && !isToken) {
  config.headers['Authorization'] = 'Bearer ' + getToken();
}

这三个自定义 header 是前端层面的控制开关，它们不会被发送到后端——axios 拦截器读取后决定行为，然后不会传给服务器。

Header含义典型使用场景
isToken: false不发 token登录接口（还没 token 呢）
repeatSubmit: false允许重复提交需要短时间内多次调用的接口
isEncrypt: 'true'请求体需 AES 加密登录接口（密码传输）
⚠️ 注意等号的值：isToken === false 才是「不发 token」——如果 isToken 是 true 或 undefined，都会发 token。这个默认值为安全优先：不显式关闭就发。

三、防重复提交：「同一请求 500ms 内不重复」

// request.ts:132-153
if (!isRepeatSubmit && (config.method === 'post' || config.method === 'put')) {
  const requestObj = {
    url: config.url,
    data: typeof config.data === 'object' ? JSON.stringify(config.data) : config.data,
    time: new Date().getTime()
  };
  const sessionObj = cache.session.getJSON('sessionObj');
  if (sessionObj === undefined || sessionObj === null || sessionObj === '') {
    cache.session.setJSON('sessionObj', requestObj);
  } else {
    if (s_data === requestObj.data && requestObj.time - s_time < 500 && s_url === requestObj.url) {
      return Promise.reject(new Error('数据正在处理，请勿重复提交'));
    } else {
      cache.session.setJSON('sessionObj', requestObj);
    }
  }
}

实现原理很简单：

  1. 把当前请求的 url + data + timestamp 存到 sessionStorage
  2. 下次请求时，比较 url 和 data 是否完全相同、时间差是否 < 500ms
  3. 三个条件全满足 → 认为是重复提交 → Promise.reject
  4. 否则更新缓存为新请求

适用场景：防止用户手抖连续点击「保存」按钮导致数据重复写入。

局限：只防同一个浏览器标签页内的 500ms 级重复，不防多标签页、不防恶意伪造。

四、前端 AES 加密：请求体加密流程

// request.ts:155-166
if (import.meta.env.VITE_APP_ENCRYPT === 'true') {
  if (isEncrypt && (config.method === 'post' || config.method === 'put')) {
    const aesKey = generateAesKey();
    config.headers[encryptHeader] = encrypt(encryptBase64(aesKey));
    config.data = typeof config.data === 'object'
      ? encryptWithAes(JSON.stringify(config.data), aesKey)
      : encryptWithAes(config.data, aesKey);
  }
}

加密流水线分三步：

  1. generateAesKey() 生成一个随机 AES 密钥（每次请求不同）
  2. 用 RSA 公钥 加密这个 AES 密钥 → 放入请求头 encrypt-key
  3. 用 AES 密钥加密整个请求体 JSON
💡 为什么 AES+RSA 混合？ AES 对称加密快，适合加密大请求体；RSA 非对称加密慢但不需共享密钥，适合加密 AES 密钥。每次请求生成新 AES 密钥，即使某次密钥泄露也不影响其他请求。

在后端，CryptoFilter 会读取 encrypt-key 请求头，用 RSA 私钥解密出 AES 密钥，再用 AES 密钥解密请求体。

❓ 登录接口调用 login() 时，传入的 config.headers 如何控制 token 行为？

点击查看答案

登录接口在 api/login.ts 中设置 isToken: false。请求拦截器读到 isToken === false 时，跳过 Authorization 头的添加——因为登录时还没有 token，带了也是无效的。同时设置 isEncrypt: true（触发 AES 加密密码）和 repeatSubmit: false（防止快速重复提交登录）。
