一、响应 AES 解密：后于请求加密的对称操作

// request.ts:181-196
if (import.meta.env.VITE_APP_ENCRYPT === 'true') {
  const keyStr = res.headers[encryptHeader];    // ① 取响应头中的 RSA 加密的 AES 密钥
  if (keyStr != null && keyStr != '') {
    const data = res.data;
    const base64Str = decrypt(keyStr);          // ② RSA 私钥解密 → base64
    const aesKey = decryptBase64(base64Str.toString()); // ③ base64 解码 → AES 密钥
    const decryptData = decryptWithAes(data, aesKey);  // ④ AES 解密 → JSON 字符串
    res.data = JSON.parse(decryptData);         // ⑤ JSON 字符串 → JS 对象
  }
}

这 5 步正好与第 9 课的请求加密对称：后端 CryptoFilter 在响应加密模式下：生成随机 AES 密钥 → 加密响应体 → RSA 加密 AES 密钥放入响应头 → 前端此处逆向操作。

⚠️ 关键判断：keyStr != null && keyStr != ''。如果响应头里没有 encrypt-key，说明后端这个接口不需要加密响应——直接透传，不做解密。这意味着加密是接口级别的可选行为，由后端 @ApiEncrypt(response = true) 决定。

二、HTTP 状态码的分层处理策略

// request.ts:198-249
const code = res.data.code || HttpStatus.SUCCESS;  // 默认成功

if (code === 401) {
  // ... 弹出重新登录对话框
} else if (code === HttpStatus.SERVER_ERROR) {
  ElMessage({ message: msg, type: 'error' });
  return Promise.reject(createHandledError(msg));
} else if (code === HttpStatus.WARN) {
  ElMessage({ message: msg, type: 'warning' });
  return Promise.reject(createHandledError(msg));
} else if (code !== HttpStatus.SUCCESS) {
  ElNotification.error({ title: msg });           // 通知
  return Promise.reject(createHandledError(msg));
} else {
  return Promise.resolve(res.data);               // 正常返回
}

这段代码体现了错误严重性分级的设计思路：

code组件用户体验设计意图
401ElMessageBox.confirm模态弹窗阻塞操作认证失败是阻断性的，用户必须处理
500ElMessage.error顶部短暂提示服务器错误用户无法处理，提醒即可
601 (WARN)ElMessage.warning顶部警告提示业务警告，用户需要注意但不阻塞
其他错误码ElNotification.error右上角通知未知错误，通知后自动消失

三、401 防重复弹窗：isRelogin 全局标记

// request.ts:21, 206-236
export const isRelogin = { show: false };  // ① 模块级单例标记

if (code === 401) {
  if (!isRelogin.show) {                   // ② 检查是否已有弹窗
    isRelogin.show = true;                 // ③ 加锁
    ElMessageBox.confirm('登录状态已过期...')
      .then(() => {
        isRelogin.show = false;            // ④ 用户点击"重新登录"，解锁
        useUserStore().logout().then(() => { router.replace({ path: "/login" }); });
      })
      .catch(() => {
        isRelogin.show = false;            // ⑤ 用户点击"取消"，解锁
      });
  }
  return Promise.reject('无效的会话...');
}

为什么需要 isRelogin？防止并发请求同时触发多个 401 弹窗。场景：页面加载后同时发 5 个 API 请求，token 恰好过期——没有这个标记会弹出 5 个「重新登录」对话框，用户体验极差。

💡 设计模式：这是简化的互斥锁（Mutex）模式。用 { show: false } 对象包装（而非 let show = false），是因为需要跨模块共享引用——permission.ts 的路由守卫也会读写 isRelogin.show。

四、通用下载方法：blob 响应处理

// request.ts:259-294
export function download(url: string, params: any, fileName: string) {
  downloadLoadingInstance = ElLoading.service({ text: '正在下载数据，请稍候' });
  return service.post(url, params, {
    transformRequest: [(params: any) => { return tansParams(params); }],
    headers: { "Content-Type": "application/x-www-form-urlencoded" },
    responseType: "blob",
  }).then(async (resp: any) => {
    const isLogin = blobValidate(resp);       // ① 校验 blob 是否是正常文件
    if (isLogin) {
      saveBlob(new Blob([resp]), fileName);   // ② 触发浏览器下载
    } else {
      const rspObj = JSON.parse(await blob.text()); // ③ blob 是错误 JSON
      ElMessage.error(rspObj.msg);
    }
    downloadLoadingInstance?.close();
  });
}

关键设计点：

  1. responseType: "blob" — 后端返回二进制文件流
  2. blobValidate(resp) — 判断返回的 blob 是真的文件还是包装在 blob 里的错误 JSON
  3. transformRequest — 覆盖默认 JSON 序列化，改成表单格式，前端文件导出通常用 POST + 表单参数
  4. saveBlob — 创建临时 <a> 标签，设置 download 属性，触发浏览器下载，40 秒后释放 URL

❓ isRelogin 为什么不定义在 useUserStore 里用 Pinia 管理，而是用模块级对象？

点击查看答案

两个原因：① isRelogin 是纯 UI 状态（弹窗是否显示），不属于用户业务数据，放 store 里会污染业务状态；② Pinia store 需要通过 useUserStore() 获取实例，在 axios 拦截器（模块顶层）中使用会导致循环依赖。模块级常量无依赖、可被任何文件 import，是最简单可靠的方案。
