第 1 课 · 鸟瞰：一次密码登录的全栈链路 入门地图

🎯 本节目标（对应使命 Success #1）：学完你能用一段话 + 一张图，讲清「一次密码登录」从前端点击到后端签发 token、再回到前端的完整调用链，并说出每一环的关键文件。这是后面所有细节挂靠的骨架——先有地图，再走小路。

我们不抠任何一行细节，只追一条线：用户在登录页输入账号密码，点「登录」，发生了什么？ 全程基于本仓库真实代码（RuoYi-Vue-Plus 后端 + plus-ui-vue 前端）。

一、先看全景图

前端 (plus-ui-vue) 后端 (RuoYi-Vue-Plus) ───────────────── ────────────────────── login.vue 点击登录 │ ▼ userStore.login() │ 组装 { username, password, code, uuid, │ clientId, grantType:'password' } ▼ api/login.ts → axios(request.ts) │ 请求头打上 clientid；isEncrypt 时 AES 加密 │ │ POST /auth/login ┌─────────────────────────────┐ └──────────────────────────────►│ AuthController.login() │ │ 1. 解析 LoginBody │ │ 2. 按 clientId 查 sys_client│ │ 3. 校验 client 含该 grantType│ │ IAuthStrategy.login(...) │ │ │ │ │ ▼ 按 grantType 分发 │ │ passwordAuthStrategy.login() │ │ · 校验图形验证码 │ │ · BCrypt 校验密码 │ │ · buildLoginUser() │ │ · LoginHelper.login() │ │ → StpUtil.login() 签发 │ │ · 返回 LoginVo(accessToken)│ └─────────────┬───────────────┘ ┌───────────────────────────────────────────┘ ▼ 拿到 access_token │ setToken() 存入本地 ▼ 路由守卫 permission.ts │ 有 token → getInfo() 拉用户信息+权限 │ → 生成动态路由 → 进入首页

二、把它拆成 7 个关键动作

1前端组装登录请求
plus-ui-vue/src/api/login.ts
登录函数会把表单数据和两个关键字段合并：clientId（PC 端固定值，来自环境变量 VITE_APP_CLIENT_ID）和 grantType: 'password'。这两个字段是后端选择「认证方式」的钥匙。
2axios 拦截器加工请求
plus-ui-vue/src/utils/request.ts
请求发出前，拦截器统一打上 clientid 请求头；登录接口标了 isEncrypt，于是请求体会被 AES 加密（这块第 6 课细讲）。登录接口标了 isToken:false，因为此刻还没有 token。
2.5接口加解密入口 CryptoFilter
后端 @ApiEncrypt 注解 + CryptoFilter 过滤器先把加密请求体解开，再交给 Controller。登录/注册接口都带这个注解。
3AuthController 接客 + 客户端校验
ruoyi-admin/.../web/controller/AuthController.java:77
login() 解析出 clientId 与 grantType，用 clientService.queryByClientId() 查出 sys_client 配置，校验「这个客户端是否允许这种 grantType」「客户端是否正常状态」。校验通过才往下走。
4策略分发
ruoyi-admin/.../web/service/IAuthStrategy.java:31
核心一行：IAuthStrategy.login(body, client, grantType)。它按 grantType + "AuthStrategy" 拼出 Spring Bean 名字（如 passwordAuthStrategy），从容器里取出对应策略执行。这就是「策略模式 + Spring Bean 名约定」——第 2 课主角。
5密码策略校验身份
ruoyi-admin/.../web/service/impl/PasswordAuthStrategy.java:56
按开关校验图形验证码 → 按用户名查用户 → 用 BCrypt.checkpw() 比对密码（带失败次数限制，超限锁定）。校验通过后，buildLoginUser() 把用户的部门、角色、岗位、权限装进一个 LoginUser 对象。
6Sa-Token 签发 token
ruoyi-common-satoken/.../utils/LoginHelper.java:54
LoginHelper.login() 调用 StpUtil.login() 完成登录，把 LoginUser 存进 Token-Session，并往 token 里塞入 userId、clientid 等扩展信息。token 值通过 StpUtil.getTokenValue() 取出，包进 LoginVo 返回。
7前端落地 token + 拉信息
plus-ui-vue/src/store/modules/user.ts + permission.ts
前端 userStore.login() 拿到 access_token 后 setToken() 存起来。之后路由守卫 permission.ts 发现有 token，就调 getInfo() 拉取用户信息与权限码，生成动态路由，放行进入系统。
💡 一句话记住这条链：
「客户端 + 授权类型」选策略 → 策略校验身份 → Sa-Token 签发 token → 前端存 token 换信息」。
这条主干在 5 种登录方式里都一样，变的只是第 5 步「策略怎么校验身份」。

三、为什么值得先记这张图？

RuoYi 的登录代码乍看分散在十几个文件里，但它的骨架极其规整：一个统一入口（AuthController）+ 一套策略分发（IAuthStrategy）+ 一个签发助手（LoginHelper）。只要抓住「入口 → 分发 → 策略 → 签发」这四个锚点，后面无论是短信登录、社交登录还是权限拦截，都只是往这张图上挂细节。

🤔 想一想：登录接口为什么用 @RequestBody String body 接收，而不是直接接成 LoginBody 对象？ 因为登录请求体是加密的，且不同 grantType 的字段不同（密码登录有 username/password，短信登录有 phoneNumber/smsCode）。先用原始字符串接收，由各策略自己 JsonUtils.parseObject() 解析成自己需要的 Body 类型，最灵活。这正是策略模式「把差异留给各策略」的体现。

四、检验一下（提取练习）

不要回看上文，凭记忆作答——回忆比重读更能形成长期记忆。

Q1. 后端是靠什么决定「这次登录该用哪种校验逻辑」的？

A. 请求的 URL 路径不同 B. 请求体里的 grantType 字段，拼成 Bean 名后从 Spring 容器取对应策略 C. 用户名前缀 D. HTTP 方法（GET/POST）
看答案B。IAuthStrategy.login() 用 grantType + "AuthStrategy" 拼 Bean 名（如 smsAuthStrategy），SpringUtils.getBean() 取出执行。URL 始终是 /auth/login。

Q2. token 是在哪一步、由谁真正签发的？

A. 前端 axios 拦截器生成 B. AuthController 直接 new 一个字符串 C. 策略里调 LoginHelper.login() → StpUtil.login() 由 Sa-Token 签发 D. 数据库触发器生成
看答案C。身份校验通过后，策略调用 LoginHelper.login()，内部由 Sa-Token 的 StpUtil.login() 完成签发，前端只是「接收并存储」token。

Q3. 登录成功后，前端为什么还要再发一次 getInfo() 请求？

A. 登录响应里没有权限信息，需要凭 token 再拉用户详情与权限码用于路由/按钮鉴权 B. 为了刷新 token C. 为了记录登录日志 D. 没有必要，是冗余设计
看答案A。LoginVo 只含 token，不含权限。路由守卫 permission.ts 发现有 token 后调 getInfo() 拉取 roles/permissions，据此生成动态路由、控制按钮显隐。
