第 8 课 · React 鉴权对接 + 全栈对比收官 收官 · plus-ui-react

🎯 本节目标（对应使命 Success #6 的 React 部分 + 全局收束）：讲清 React 前端（Zustand + axios + UmiJS 布局守卫）的同一套对接，逐项对比 Vue 与 React 的异同，并把八节课收成一张完整的全栈大图。这是终点站。

核心洞察先抛出：React 和 Vue 面对的是同一个后端、同一套鉴权契约（Bearer token + clientid 头 + 401 重登 + 权限码）。所以它们的「职责分工」完全一样，只是「用什么工具实现」不同。

一、React 版四职责（和第 7 课同构）

职责Vue (plus-ui-vue)React (plus-ui-react)
存 tokenutils/auth.ts（VueUse useStorage）utils/auth.ts（localStorage 直存）
登录态/用户信息Pinia: user.tsZustand: userStore.ts + permissionStore.ts
请求加工 + 401utils/request.tsapi/request.ts（逻辑几乎照搬）
路由准入permission.ts 全局守卫BasicLayout.tsx 的 useEffect 守卫

存 token：更直白

// plus-ui-react/src/utils/auth.ts
const TokenKey = 'Admin-Token';
export const getToken = () => localStorage.getItem(TokenKey);
export const setToken = (t) => localStorage.setItem(TokenKey, t);
export const removeToken = () => localStorage.removeItem(TokenKey);

同样的 Admin-Token key，同样只存字符串。

请求拦截器：契约完全一致

// api/request.ts —— 和 Vue 版对照看
config.headers.set('Authorization', `Bearer ${getToken()}`);  // 同
headers: { clientid: appEnv.clientId }                         // 同（第 4 课要用）
// isEncrypt → AES+RSA 加密（第 6 课，同）
// repeatSubmit → 500ms 防重（同）
if (code === 401) { showReloginConfirm(); }                    // 同：Modal 询问重登
✅ 几乎是逐行对应。差异只在 API：Vue 用 config.headers['x']=…（axios 旧式），React 用 config.headers.set('x',…)（axios 新式 AxiosHeaders）；Vue 弹 ElMessageBox，React 弹 antd Modal。鉴权逻辑零差异。

路由准入：守卫位置不同，思路相同

// BasicLayout.tsx:69
useEffect(() => {
    if (!getToken()) {                                  // 无 token → 跳登录
        history.replace(`/login?redirect=${encodeURIComponent(...)}`);
        return;
    }
    reloadMenus()                                       // 有 token → 拉菜单+用户信息
        .catch(() => { removeToken(); history.replace('/login'); });
}, [location.pathname, ...]);

Vue 用 Router 的全局前置守卫 beforeEach；React（UmiJS）没有等价物，改用布局组件的 useEffect 在进入受保护区域时检查 token + 拉数据。reloadMenus（在 permissionStore）一次性 Promise.all([getInfo, getRouters]) 拿用户信息和后端路由。

💡 权限拆分差异：Vue 把 user 信息和 permission（路由）放两个 Pinia store；React 也是两个 Zustand store（userStore 存 userInfo，permissionStore 存菜单/路由）。React 的按钮级权限用纯函数 hasPermi(userInfo, ['x:y:z']) / hasRole(...)（utils/permission.ts），逻辑和 Vue 的 v-hasPermi 一致：含 *:*:* 或 admin 直接放行。

[前端 Vue/React] [后端 RuoYi] 登录页 → store.login AuthController(@ApiEncrypt) └ api: +clientId +grantType └ ① CryptoFilter 解密(第6课) └ request: AES加密(第6) +clientid头 └ ② 查 sys_client 校验(第2课) │ POST /auth/login └ ③ IAuthStrategy 分发(第2课) └──────────────────────────────► └ xxxAuthStrategy 验身份(第5课) └ BCrypt/验证码/JustAuth └ ④ buildLoginUser(第3课) └ ⑤ LoginHelper→StpUtil.login JWT签发 + LoginUser存Redis(第3课) setToken ◄──── LoginVo.accessToken ────────┘ └ getInfo 拉 roles/permissions └ 守卫/useEffect 生成动态路由(第7,8课) 后续每个请求：Bearer token + clientid头 └──► SaInterceptor: checkLogin + 客户端一致 + 路径/IP(第4课) @SaCheckPermission ──► StpInterface 取权限(第4课) 异常 ──► SaTokenExceptionHandler ──► 401/403 ──► 前端重登(第7,8课)
🤔 想一想：如果要给这套系统再加一个 Flutter App 客户端，前端要重做哪些、后端要改哪些？ 前端（Flutter）：四个职责得用 Dart 重新实现一遍——存 token、HTTP 拦截器（加 Bearer + clientid 头 + 可选加密）、401 处理、本地权限判断。但契约不变，照着 Vue/React 抄逻辑即可。
后端：理想情况零改动——只需在 sys_client 表加一行新客户端配置（设好 grantType、deviceType、token 超时）。这正是前 8 课反复强调的设计回报：客户端体系 + 策略模式让「接新端」变成配置问题而非改代码问题。这就是你现在应该能自信讲出来的整体设计价值。

三、检验一下（综合提取）

Q1. Vue 用 Router 全局守卫做路由准入，React（UmiJS）用什么？

A. 也是全局守卫 B. BasicLayout 组件里的 useEffect，在进入受保护区域时检查 token 并 reloadMenus C. 后端控制 D. 不做准入
看答案B。UmiJS 无 Vue Router 式全局前置守卫，改用布局组件 useEffect。思路相同：无 token 跳登录，有 token 拉数据生成菜单。

Q2. Vue 和 React 前端在「与后端的鉴权契约」上有何差异？

A. 差异很大，token 格式都不同 B. 完全相同：都用 Bearer token + clientid 头 + 401 重登 + 权限码 + AES/RSA 加密 C. React 不需要 clientid D. Vue 不加密
看答案B。契约由后端定义，两个前端只是用不同框架实现同一份契约。差异仅在状态管理库、UI 库、axios API 版本等「实现层」。

Q3. 给系统接入一个全新客户端（如 Flutter App），后端通常需要改代码吗？

A. 要大改登录逻辑 B. 理想情况零改动，只在 sys_client 加一行配置（若沿用已有 grantType） C. 要新增一个 Controller D. 要改数据库结构
看答案B。客户端体系（第2课）+ 策略模式让「接新端」成为配置问题。仅当需要全新登录方式时才加策略类（第2课「新增钉钉登录动三处」）。
