02 | 发码校验完整链路：从验证码发送到短信登录

学习目标：跟踪一条验证码从"点击发送"到"登录成功"的完整生命周期，理解 yml 配置、Redis key 一致性、@RateLimiter 防刷和多厂商共存这四个核心设计。

1. yml 配置解析：sms.blends 的运作机制

在 application-dev.yml（或对应的环境配置）中，短信模块的配置结构如下：

sms:
  blends:
    config1:
      supplier: alibaba
      access-key-id: your-access-key
      access-key-secret: your-secret
      signature: 若依管理系统
      template-id: SMS_123456789
      restricted: 10        # 每分钟最大发送数
      minute-max: 10        # 每小时最大发送数
      account-max: 30       # 每天最大发送数
    config2:
      supplier: tencent
      access-key-id: your-tencent-key
      access-key-secret: your-tencent-secret
      signature: 若依管理
      template-id: 1234567

1.1 configId 自定义命名

你注意到 config1 和 config2 不是 sms4j 框架规定的固定名称，而是项目自定义的 configId。sms4j 在启动时会遍历 sms.blends 下的所有子节点，将每个节点名作为一个配置标识注册到 SmsFactory 中。

这意味着你可以按业务场景命名：

sms:
  blends:
    captcha-config:       # 验证码专用
      supplier: alibaba
      ...
    marketing-config:     # 营销短信专用
      supplier: tencent
      ...

命名没有限制，见名知意即可。后续调用时通过 configId 选择使用哪个厂商：

SmsBlend blend = SmsFactory.getSmsBlend("captcha-config");
SmsResponse response = blend.sendMessage(phone, msg);

1.2 拦截参数的作用

三个关键参数控制发送频率：

参数含义示例值
restricted每分钟最大发送数10
minute-max每小时最大发送数10
account-max每天最大发送数30

sms4j 框架利用 SmsDao 缓存（即 PlusSmsDao → Redis）来追踪每个手机号的发送频率。如果超过限制，框架会直接抛出 SmsBlendException，不会真正调用厂商 API。

关键认知：这就是为什么第 1 课中的 PlusSmsDao（Redis 缓存接管）如此重要。如果缓存是 ConcurrentHashMap 而非 Redis，这些频率限制在集群环境下形同虚设 —— 攻击者只需把请求分散到不同节点就能绕过。

1.3 如何新增一个厂商配置

新增厂商只需在 yml 的 sms.blends 下添加新节点，并在 pom.xml 中引入对应厂商的 sms4j 适配器依赖。sms4j 的 SPI 机制会自动发现可用厂商。

2. 发码链路：从 Controller 到 Redis

验证码发送的完整调用链如下：

用户请求 POST /captcha/smsCode?phoneNumber=13800138000
    │
    ▼
CaptchaController.smsCode()
    │  @RateLimiter(key = "#phoneNumber", time = 60, count = 1)
    │  同一手机号 60 秒内最多调用 1 次
    │
    ▼
SmsFactory.getSmsBlend("config1")
    │  从 yml 中读取 config1 的配置，获取对应的 SmsBlend 实例
    │  如果 config1 配置的是阿里云，拿到的是 AlibabaSmsBlendImpl
    │
    ▼
smsBlend.sendMessage(phoneNumber, code)
    │  sms4j 内部：
    │  1. 检查 SmsDao 中的频率计数（Redis global:xxx）
    │  2. 调用厂商 SDK（阿里云/腾讯云 API）
    │  3. 将验证码存入 SmsDao（Redis）
    │  4. 返回 SmsResponse{success: true, configId: "config1", data: "..."}
    │
    ▼
Controller 收到 SmsResponse，检查 success
    │  成功 → 返回 R.ok()
    │  失败 → 返回 R.fail("发送失败")

2.1 @RateLimiter 防刷机制

@RateLimiter(key = "#phoneNumber", time = 60, count = 1) 这个注解的语义是：以手机号为维度，60 秒内最多允许 1 次请求。这是 RuoYi 框架层提供的 AOP 限流，独立于 sms4j 内部的频率拦截，形成了双层防护：

防护层提供方粒度机制
第一层：@RateLimiter RuoYi 框架 接口级 60 秒内同一手机号最多调用 1 次接口
第二层：sms4j 内部拦截 sms4j 框架 厂商级 每分钟/每小时/每天限制，基于 SmsDao 记录的具体发送次数

两层防护的定位不同：第一层防止接口滥用（连短信都不发），第二层防止短信被盗刷（控制实际发送量）。

2.2 验证码存入 Redis

短信发送成功后，sms4j 会将验证码存入 SmsDao（实际是 Redis），存储 key 的格式为：

global:captcha_codes:13800138000

其中：

  • global: 来自 GlobalConstants.GLOBAL_REDIS_KEY
  • captcha_codes: 来自 GlobalConstants.CAPTCHA_CODE_KEY
  • 13800138000 是手机号

过期时间由 yml 中的 cache-expire-time 控制，默认 5 分钟。

生产提醒：验证码 5 分钟过期是行业惯例，但不要设置太短（用户可能收短信有延迟）也不要太长（安全性下降）。3-5 分钟是合理的平衡点。

3. 校验链路：从 Redis 到登录态

用户输入验证码点击登录后，触发 SmsAuthStrategy.validateSmsCode()：

SmsAuthStrategy.validateSmsCode(phone, userInputCode)
    │
    ▼
从 Redis 读取验证码
    key = "global:captcha_codes:" + phone
    storedCode = RedisUtils.getCacheObject(key)
    │
    ▼
比对验证码
    if (userInputCode.equals(storedCode))
    │
    ▼ (匹配成功)
查询用户信息
    SysUser user = userService.selectUserByPhone(phone)
    │
    ▼
签发登录态
    LoginHelper.login(user)
        生成 token → 存入 Redis → 写入 SecurityContext
    │
    ▼
返回登录成功

3.1 为什么校验和发码能读到同一个 key？

这是整个链路中最关键的设计 —— Redis key 的一致性。

发码时 sms4j 把验证码存入 global:captcha_codes:{phone}，校验时 SmsAuthStrategy 也用同样的 key 格式去读。两者使用的是同一个常量 GlobalConstants.CAPTCHA_CODE_KEY，保证了 key 的绝对一致。

如果不一致会发生什么？校验时读不到验证码，用户永远登录失败。这是一个"静默的 bug"—— 不会报错，只会让用户反复操作然后放弃。

设计原则：跨模块共享的 Redis key 格式应该定义为常量，而不是在两个地方各写一遍字符串。CAPTCHA_CODE_KEY 被 CaptchaController（写）和 SmsAuthStrategy（读）共同引用，保证了"写"和"读"永远使用相同的 key 前缀。

3.2 GLOBAL_REDIS_KEY 的隔离作用

GLOBAL_REDIS_KEY = "global:" 作为全局前缀，将短信相关的缓存与以下数据隔离：

  • 租户数据：多租户场景下，租户数据 key 通常以 tenantId 为前缀，不与 "global:" 冲突
  • 业务缓存：其他模块的缓存 key 有自己的前缀规范
  • 框架缓存：Sa-Token、Spring Cache 等框架级的缓存也有各自的前缀

这种前缀约定的另一个好处是运维方便：在 Redis CLI 中执行 keys global:* 就能看到所有全局缓存，执行 ttl global:captcha_codes:13800138000 可以检查验证码是否还在有效期内。

3.3 LoginHelper.login() 做了什么

校验通过后，LoginHelper.login(user) 完成以下操作：

  1. 调用 Sa-Token 的 StpUtil.login(userId) 生成会话
  2. 将用户信息序列化存入 Redis（Sa-Token 的 token → user 映射）
  3. 将登录态写入 SecurityContextHolder，后续请求可以直接获取当前用户
  4. 返回 token 给前端

至此，短信验证码登录的完整链路闭合。

4. 多厂商共存机制

config1 和 config2 并存的核心价值是：同一套代码同时对接多个 SMS 厂商。具体实现依赖于 sms4j 的工厂模式：

// 验证码短信走阿里云
SmsBlend captchaBlend = SmsFactory.getSmsBlend("config1");
captchaBlend.sendMessage(phone, captchaCode);

// 营销短信走腾讯云
SmsBlend marketingBlend = SmsFactory.getSmsBlend("config2");
marketingBlend.sendMessage(phone, marketingContent);

SmsFactory 内部维护了一个 Map<String, SmsBlend>，启动时根据 yml 配置初始化所有厂商实例。getSmsBlend(configId) 是 O(1) 的 Map 查询操作。

4.1 多厂商共存的使用场景

场景策略原因
验证码 + 通知 走阿里云 到达率高，验证码场景对可靠性要求最高
营销短信 走腾讯云 单价更低，大批量发送时节省成本
容灾备份 config1 失败降级到 config2 阿里云故障时自动切换到腾讯云，保证可用性
地域合规 国内走 config1，海外走 config2 不同厂商在不同地区的牌照和覆盖范围不同

容灾降级的实现思路：

public SmsResponse sendWithFallback(String phone, String msg) {
    try {
        return SmsFactory.getSmsBlend("config1").sendMessage(phone, msg);
    } catch (SmsBlendException e) {
        log.warn("主厂商(config1)发送失败，降级到备用厂商(config2)");
        return SmsFactory.getSmsBlend("config2").sendMessage(phone, msg);
    }
}
工程建议：不要在生产环境把两个不同短信厂商的配置混用同一个 configId。每个 configId 应对应一个特定的厂商+场景组合，这样出问题时能快速定位是哪个厂商、哪个配置出了问题。

5. 总结：一条验证码的一生

把发码和校验两条链路拼接起来，一条验证码的完整生命周期如下：

  1. 生成：Controller 收到请求 → @RateLimiter 防火 → SmsFactory 路由到 config1 → 阿里云短信网关发送 → 验证码写入 Redis (global:captcha_codes:{phone}, 5 分钟 TTL)
  2. 存储：在 Redis 中以 "global:captcha_codes:{phone}" 为 key，等待用户输入
  3. 校验：SmsAuthStrategy 用相同 key 从 Redis 读取 → 与用户输入比对
  4. 登录：匹配成功 → 查用户表 → LoginHelper.login() → 签发 token
  5. 销毁：Redis 5 分钟 TTL 自动过期，验证码消失

整个链路涉及 3 个模块（Controller、SMS 适配、Auth Strategy）、2 个框架（sms4j、Sa-Token）、1 个基础设施（Redis），但代码层面非常干净 —— 每个环节都只做自己的事，通过 Redis key 常量和 SmsDao 接口解耦。

📝 自测题（点击展开）

Q1：如果发码时 sms4j 把验证码存到了 Redis key global:captcha_codes:13800138000，但 SmsAuthStrategy 校验时去读 captcha:13800138000（少了 "global:" 前缀），会发生什么？

参考答案：校验时在 Redis 中找不到该 key，返回 null，永远比对失败。用户输入正确的验证码也无法登录。这是一个静默 bug —— 不报错、不抛异常，但功能完全不可用。

Q2：@RateLimiter(key = "#phoneNumber", time = 60, count = 1) 和 sms4j yml 中配置的 restricted/minute-max 有什么区别？为什么需要两层防护？

参考答案：@RateLimiter 是接口级防护，限制同一个手机号在 60 秒内最多调用接口 1 次，防止频繁请求打到短信 API。sms4j 的 restricted/minute-max 是更深层的防护，基于实际发送记录计数，防止绕过接口限流直接调用发送逻辑。两层防护的定位不同：第一层减少无效请求，第二层兜底防刷。

Q3：config1 和 config2 可以配置同一个供应商但不同模板吗？这样做有什么价值？

参考答案：可以。例如 config1 使用"验证码模板"发送验证码，config2 使用"通知模板"发送系统通知。这样做的好处是：不同场景使用不同的模板（不同的签名、内容格式），但底层是同一家厂商，配置管理更清晰。
