第 3 课 · 广播任务：让所有实例一起跑 模式 3

🎯 本节目标（对应使命 Success #3）：学完你能讲清广播任务的执行模型——一个任务触发后，所有在线客户端实例各自执行一次，并理解它和普通任务的本质区别以及适用场景。

普通任务：3 个实例在线 → 随机挑 1 个执行。广播任务：3 个实例在线 → 3 个都执行。这就是广播的核心语义。

一、TestBroadcastJob — 每个实例都跑一次

源码拆解

ruoyi-job/src/main/java/org/dromara/job/snailjob/TestBroadcastJob.java 
@Slf4j
@Component
@JobExecutor(name = "testBroadcastJob")
public class TestBroadcastJob {

    @Value("${snail-job.port}")
    private int clientPort;                         // 注意：每个实例的端口不同！

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        int randomInt = RandomUtil.randomInt(100);
        log.info("随机数: {}", randomInt);
        SnailJobLog.REMOTE.info("随机数: {},客户端端口:{}", randomInt, clientPort);
        if (randomInt < 50) {
            throw new RuntimeException("随机数小于50，收集日志任务执行失败");
        }
        return ExecuteResult.success("随机数大于50，收集日志任务执行成功");
    }
}

关键观察：

  • @Value("${snail-job.port}") — 注入当前客户端的端口号。回顾第 1 课的配置：port: 2${server.port}，如果应用跑在 8080，SnailJob 客户端端口就是 28080。多实例部署时每个实例端口不同，这是区分「谁在执行」的关键。
  • 随机成功/失败 — randomInt < 50 时抛异常，模拟真实场景中部分实例执行失败的情况
  • Lombok @Slf4j — 和 SnailJobLog.REMOTE 配合使用，前者本地、后者远程

二、广播任务的执行模型

SnailJob 服务端 │ │ 触发广播任务 testBroadcastJob │ ┌────────────┼────────────┐ ▼ ▼ ▼ ┌────────┐ ┌────────┐ ┌────────┐ │ 实例 A │ │ 实例 B │ │ 实例 C │ │ 端口 │ │ 端口 │ │ 端口 │ │ 28080 │ │ 28081 │ │ 28082 │ │ │ │ │ │ │ │ 随机数 │ │ 随机数 │ │ 随机数 │ │ =72 ✓ │ │ =23 ✗ │ │ =88 ✓ │ └────────┘ └────────┘ └────────┘ 结果：A 成功、B 失败、C 成功 → 服务端会收到 3 条执行记录（各自独立）
和普通任务的核心区别：普通任务（如 0002 课的注解式任务）由 SnailJob 服务端从在线实例中选一个执行——这是「负载均衡」策略。广播任务则是所有在线实例都收到指令并执行——这是「全量下发」策略。路由策略在 SnailJob 后台创建任务时选择。
⚠️ 广播任务的幂等性要求：广播任务会在所有实例上执行，如果你的业务逻辑有副作用（如「给所有用户发短信」），广播会导致每个实例都发一遍——重复发送！广播适用的是「各实例各自需要做的事」：清自己的本地缓存、刷新自己的配置、收集自己的 JVM 指标，而不是「全局只做一次的事」。

三、适用场景（什么时候用广播）

场景为什么适合广播为什么不适合普通任务
全实例缓存清理 每个实例 JVM 内有各自的 Caffeine 本地缓存，必须各自清 只清一个实例的缓存，其他实例还在用旧数据
配置热刷新 每个实例从配置中心拉最新配置到内存 只有一个实例刷新了配置无用
JVM 指标采集 每个实例的堆内存、线程数不同，各自上报 只采一个实例看不到集群全貌
批量数据同步 只适合分片模式——广播各实例再按分片各自处理一段 数据同步应配合静态分片或 Map 任务

四、广播 + 分片 = 集群并行批处理

广播任务本身是「每个实例执行相同逻辑」。但结合下一课的静态分片或 Map 任务，可以实现广播分发 + 分片处理的组合模式：服务端广播 → 每个实例收到分片参数 → 各处理一段数据。这是分布式批处理的经典模式。

为什么广播任务要注入端口号？这不是 demo 随意写的。在生产中，当你看到 SnailJob 后台日志里「端口:28080 执行成功」「端口:28081 执行失败」，你能立刻定位到哪个实例出了问题。端口号是广播任务排查的「实例标识」。

五、检验一下（提取练习）

Q1. 一个应用部署了 5 个实例，触发广播任务后，会执行几次？

A. 1 次——服务端随机挑一个实例执行 B. 5 次——每个在线实例各执行一次 C. 取决于谁先抢到锁 D. 0 次——广播任务只是通知，不执行业务逻辑
看答案B。广播任务的核心语义就是「全量下发」：所有在线客户端实例都收到指令并执行。5 个实例 = 5 次独立执行 = 5 条执行记录。

Q2. 以下哪个场景不适合用广播任务？

A. 清理每个实例的本地 Caffeine 缓存 B. 每个实例上报自己的 JVM 堆内存指标 C. 给所有注册用户发一封邮件（全局只需发一次） D. 刷新每个实例从配置中心加载的内存配置
看答案C。发邮件是全局副作用——只需一个实例执行即可。用广播会导致 5 个实例各发一次，用户收到 5 封重复邮件。这种「全局做一次」的场景应该用普通任务（路由策略选「单机执行」）。

Q3. 广播任务中如果某个实例执行失败了，其他实例会怎样？

A. 全部回滚，因为广播是事务性的 B. 其他实例继续执行不受影响，各自向服务端上报各自的结果 C. 服务端通知所有实例停止 D. 失败实例自动重启
看答案B。广播任务中各实例独立执行、独立上报。一个实例失败不影响其他实例——这从 TestBroadcastJob 中每个实例自己生成随机数独立判断成功/失败就能看出来。服务端会收到 N 条独立的执行记录。

Q4. @Value("${snail-job.port}") 在广播任务中的实际价值是什么？

A. 只是 demo 演示 Spring 注入，没有实际意义 B. 作为实例标识——排查问题时能知道具体是哪个实例执行成功或失败了 C. 用来连接 SnailJob 服务端 D. 用来和其他实例通信
看答案B。多实例部署时，每个实例的 SnailJob 客户端端口不同（2${server.port}）。在远程日志中带上端口号，运维能立刻定位到出问题的机器。这是可观测性的基础实践。
