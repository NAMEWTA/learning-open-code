第 7 课 · DAG 工作流：用 wfContext 串联多个任务节点 模式 6b

🎯 本节目标（对应使命 Success #7）：学完你能讲清 DAG 工作流的原理——通过 JobArgs.wfContext 在工作流节点间传递数据，appendContext() 写入、getWfContext() 读取，以及 WechatBillTask → AlipayBillTask → SummaryBillTask 三个节点如何协作完成「账单汇总」。

前面 6 种模式都是「单个任务」的调度。但现实业务常常是多步骤流水线：先拉微信账单 → 再拉支付宝账单 → 最后汇总。SnailJob 的 DAG 工作流就是为这种场景设计的——你可以定义多个任务节点、编排它们的执行顺序、通过一个共享的上下文 Map 在节点间传数据。

一、DAG 工作流的拓扑

wechatBillTask
拉微信账单 → summaryBillTask
汇总账单
alipayBillTask
拉支付宝账单 → （两个上游并行 → 汇入汇总节点）

微信账单和支付宝账单并行执行，两者都完成后触发汇总任务。这是一个典型的 DAG（有向无环图）结构：节点 = 任务，边 = 执行依赖。

二、三个任务节点逐一看

① WechatBillTask — 拉微信账单，写入上下文

ruoyi-job/src/main/java/org/dromara/job/snailjob/WechatBillTask.java 
@Component
@JobExecutor(name = "wechatBillTask")
public class WechatBillTask {

    public ExecuteResult jobExecute(JobArgs jobArgs) throws InterruptedException {
        // 从上下文中读取清算日期（如果有的话）
        String settlementDate = (String) jobArgs.getWfContext().get("settlementDate");
        if (StringUtils.equals(settlementDate, "sysdate")) {
            settlementDate = DateUtils.now();
        }
        // 模拟构造微信账单数据
        BillDTO billDTO = new BillDTO(123456789L, "wechat",
            settlementDate, new BigDecimal("1234.56"));
        // ★ 关键：把 billDTO 写入上下文，传给下游节点
        jobArgs.appendContext("wechat", JsonUtils.toJsonString(billDTO));
        SnailJobLog.REMOTE.info("上下文: {}", jobArgs.getWfContext());
        return ExecuteResult.success(billDTO);
    }
}

核心操作：jobArgs.appendContext("wechat", jsonString)——把序列化后的账单对象写入工作流上下文，key 为 "wechat"。

② AlipayBillTask — 拉支付宝账单，同样写入上下文

ruoyi-job/src/main/java/org/dromara/job/snailjob/AlipayBillTask.java 
@Component
@JobExecutor(name = "alipayBillTask")
public class AlipayBillTask {

    public ExecuteResult jobExecute(JobArgs jobArgs) throws InterruptedException {
        String settlementDate = (String) jobArgs.getWfContext().get("settlementDate");
        if (StringUtils.equals(settlementDate, "sysdate")) {
            settlementDate = DateUtils.now();
        }
        BillDTO billDTO = new BillDTO(23456789L, "alipay",
            settlementDate, new BigDecimal("2345.67"));
        // ★ 写入上下文，key 为 "alipay"
        jobArgs.appendContext("alipay", JsonUtils.toJsonString(billDTO));
        SnailJobLog.REMOTE.info("上下文: {}", jobArgs.getWfContext());
        return ExecuteResult.success(billDTO);
    }
}

③ SummaryBillTask — 从上下文读取两个账单，汇总

ruoyi-job/src/main/java/org/dromara/job/snailjob/SummaryBillTask.java 
@Component
@JobExecutor(name = "summaryBillTask")
public class SummaryBillTask {

    public ExecuteResult jobExecute(JobArgs jobArgs) throws InterruptedException {
        // 从上下文读取微信账单
        BigDecimal wechatAmount = BigDecimal.valueOf(0);
        String wechat = (String) jobArgs.getWfContext("wechat");
        if (StringUtils.isNotBlank(wechat)) {
            BillDTO wechatBillDTO = JsonUtils.parseObject(wechat, BillDTO.class);
            wechatAmount = wechatBillDTO.billAmount();
        }
        // 从上下文读取支付宝账单
        BigDecimal alipayAmount = BigDecimal.valueOf(0);
        String alipay = (String) jobArgs.getWfContext("alipay");
        if (StringUtils.isNotBlank(alipay)) {
            BillDTO alipayBillDTO = JsonUtils.parseObject(alipay, BillDTO.class);
            alipayAmount = alipayBillDTO.billAmount();
        }
        // ★ 汇总
        BigDecimal totalAmount = wechatAmount.add(alipayAmount);
        SnailJobLog.REMOTE.info("总金额: {}", totalAmount);
        return ExecuteResult.success(totalAmount);
    }
}

核心操作：jobArgs.getWfContext("wechat") 和 getWfContext("alipay")——从前置节点写入的上下文中取出账单 JSON，反序列化，累加金额。

三、wfContext — 工作流的「共享黑板」

wfContext（Workflow Context）的本质： 它就是一个 Map<String, Object>，在 DAG 工作流的各个节点之间共享。 写入端： jobArgs.appendContext("key", value) → 追加一个键值对 读取端： jobArgs.getWfContext() → 获取整个上下文的 Map 引用 jobArgs.getWfContext("key") → 按 key 取值 jobArgs.getWfContext().get("key") → 等价的另一种写法 传递数据的最佳实践： → 复杂对象序列化为 JSON 字符串再写入（如 JsonUtils.toJsonString(billDTO)） → 下游节点用 JsonUtils.parseObject(json, BillDTO.class) 反序列化 → 基本类型（String、int）可以直接写入
为什么上下文数据要序列化为 JSON 而不是直接传对象？虽然代码中有 Map<String, Object>，但不同节点可能在不同的 JVM 实例上执行。直接传 Java 对象依赖序列化机制，JSON 是最安全、可读性最强的跨节点传递方式。另外，这也方便在 SnailJob 后台的日志中查看上下文内容——JSON 可读。

四、workflow data vs jobParams — 两个容易混淆的数据入口

维度jobArgs.getWfContext()jobArgs.getJobParams()
数据来源前置节点写入（动态的）服务端任务配置（静态的）
生命周期一个工作流实例内，跨节点共享本次任务执行时下发
写入者前置任务节点（appendContext）服务端管理员（后台配置）
典型用途传递中间计算结果（如账单对象）传递执行参数（如日期、ID 范围）
在 DAG 中的角色核心——没有上下文就没有 DAG辅助——提供初始参数

settlementDate 的双重获取逻辑

注意 WechatBillTask 和 AlipayBillTask 中都有一段相同的日期处理逻辑：

String settlementDate = (String) jobArgs.getWfContext().get("settlementDate");
if (StringUtils.equals(settlementDate, "sysdate")) {
    settlementDate = DateUtils.now();
}

这展示了 DAG 工作流的常见模式：初始参数由 DAG 工作流在服务端配置时注入 wfContext（如 {"settlementDate": "sysdate"}），任务节点读取并使用。"sysdate" 是一个特殊标记，表示「用当前日期」——这是代码层的灵活约定。

五、DAG 工作流 vs 单体 MapReduce 任务

维度MapReduce 任务DAG 工作流
拓扑固定：Map → Reduce灵活：任意 DAG（多源汇聚、多路分发）
数据流子 Map 各自返回 → Reduce 聚合wfContext 在任意节点间共享
并行度Map 子任务并行，Reduce 串行无依赖节点可并行（如微信、支付宝同时执行）
适用场景同一类数据的并行计算+聚合不同类型任务的编排（异构工作流）
配置位置SnailJob 后台选路由策略SnailJob 后台画 DAG 拓扑图

本模块的三个 BillTask 之所以能组合成工作流，是因为它们在 SnailJob 后台被画成了一个 DAG 图（在后台「工作流管理」中可视化编排），而不是代码中写了什么串联逻辑。代码只需要关注「自己该从上下文读什么、该往上下文写什么」。

六、BillDTO — 工作流的数据契约

一个 Java Record 作为数据传输对象

ruoyi-job/src/main/java/org/dromara/job/entity/BillDTO.java 
public record BillDTO(
    Long billId,
    String billChannel,   // 渠道：wechat / alipay
    String billDate,      // 账单日期
    BigDecimal billAmount // 账单金额
) {}

BillDTO 是这个 DAG 工作流 demo 的数据契约。每个节点都遵守它——微信节点生成 wechat 渠道的 BillDTO，支付宝节点生成 alipay 渠道的，汇总节点读取两者的 billAmount 相加。所有节点共享同一个 DTO 结构，这就是工作流节点间的「接口约定」。

七、检验一下（提取练习）

Q1. DAG 工作流中，两个上游节点（微信、支付宝）是什么关系？

A. 串行——微信跑完再跑支付宝 B. 并行——两个节点可以同时执行，因为它们之间没有依赖 C. 互斥——只有一个会执行 D. 广播——两个节点在各自实例上各执行一次
看答案B。DAG 中，没有箭头连接的节点可以并行执行。微信和支付宝都只依赖 DAG 启动事件，互相没有依赖，所以 SnailJob 会让它们同时跑。只有当两者都完成后，汇总节点才会被触发。

Q2. jobArgs.appendContext("wechat", json) 做了什么？

A. 把数据写入当前节点的本地变量，仅供本方法后续使用 B. 把键值对追加到工作流的共享上下文 Map 中，下游节点可以读取 C. 把数据发送到 SnailJob 服务端存储，下次任务执行时可以获取 D. 把数据写入数据库的上下文表
看答案B。wfContext 是一个在 DAG 工作流实例内跨节点共享的 Map。上游节点通过 appendContext 写入，下游节点通过 getWfContext 读取。工作流结束后上下文销毁——它不是持久化存储。

Q3. 如果微信账单任务执行失败，会发生什么？

A. 支付宝和汇总任务继续正常执行 B. 汇总任务跳过微信数据，只用支付宝数据汇总 C. 整个 DAG 工作流失败，汇总任务不会被触发（因为依赖的微信节点失败了） D. 微信任务自动重试 3 次后跳过
看答案C。DAG 的依赖语义是严格的——如果微信节点失败，汇总节点（依赖微信和支付宝两个上游）不会被执行。这和 MapReduce 的「所有子任务成功才 Reduce」逻辑一致。由此也能看出容错设计的重要性——关键路径上的节点需要合理的重试策略。

Q4. jobArgs.getWfContext() 和 jobArgs.getJobParams() 的使用场景怎么区分？

A. 两者完全相同，可以互换 B. wfContext 用于节点间动态传数据，jobParams 用于服务端静态配置参数 C. wfContext 只在 DAG 工作流中能读写，jobParams 只在普通任务中能读 D. jobParams 只能存 String，wfContext 只能存 Object
看答案B。wfContext 是 DAG 节点间传递中间结果的「动态通道」（运行时可写），jobParams 是服务端下发初始参数的「静态通道」（运行时只读）。类比：jobParams = 函数入参，wfContext = 全局变量。
