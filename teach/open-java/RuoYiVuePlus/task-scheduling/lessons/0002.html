第 2 课 · 基础任务：注解式与继承式任务执行器 模式 1+2

🎯 本节目标（对应使命 Success #3）：学完你能区分 @JobExecutor 注解式和 extends AbstractJobExecutor 继承式两种任务编写方式，说清它们的注册机制差异、JobArgs 参数对象的结构、以及 ExecuteResult 的返回约定。

在 SnailJob 里写一个任务极其简单，但你有两种写法可选。这节课我们通过两个 demo 类看清它们的差异，并理解背后的注册机制。

一、注解式：@JobExecutor — 最常用

TestAnnoJobExecutor — 贴个注解就完事

ruoyi-job/src/main/java/org/dromara/job/snailjob/TestAnnoJobExecutor.java 
@Component
@JobExecutor(name = "testJobExecutor")
public class TestAnnoJobExecutor {

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        SnailJobLog.LOCAL.info("testJobExecutor. jobArgs:{}", JsonUtil.toJsonString(jobArgs));
        SnailJobLog.REMOTE.info("testJobExecutor. jobArgs:{}", JsonUtil.toJsonString(jobArgs));
        return ExecuteResult.success("测试成功");
    }
}

总共 3 个关键要素：

  • @Component — 交给 Spring 容器管理（SnailJob 从容器中获取执行器实例）
  • @JobExecutor(name = "testJobExecutor") — 声明「我是一个任务执行器，服务端用这个名字调度我」
  • jobExecute(JobArgs jobArgs) — 约定的方法签名。SnailJob 客户端通过反射调用这个方法，JobArgs 携带服务端下发的参数
为什么方法名必须是 jobExecute？这是 SnailJob 客户端框架的约定。当你在类上贴 @JobExecutor，框架会扫描类中名为 jobExecute 且参数为 JobArgs 的方法，作为任务入口。不是 Spring 的依赖注入，是框架级的反射查找。

二、继承式：extends AbstractJobExecutor — 有基类可扩展

TestClassJobExecutor — 继承基类，覆盖 doJobExecute

ruoyi-job/src/main/java/org/dromara/job/snailjob/TestClassJobExecutor.java 
@Component
public class TestClassJobExecutor extends AbstractJobExecutor {

    @Override
    protected ExecuteResult doJobExecute(JobArgs jobArgs) {
        return ExecuteResult.success("TestJobExecutor测试成功");
    }
}

和注解式的差异：

  • 不需要 @JobExecutor 注解——但注意，继承式必须在 SnailJob 服务端后台创建「类任务」类型的作业，并指定全限定类名
  • 方法名是 doJobExecute（不是 jobExecute），且是 protected 的
  • 基类 AbstractJobExecutor 提供了模板方法骨架，子类只填核心逻辑

三、对比：什么时候用哪种？

维度注解式 @JobExecutor继承式 extends AbstractJobExecutor
注册方式注解的 name 属性即是任务名，客户端启动时自动向服务端注册需要在 SnailJob 后台手动创建任务，填「全限定类名」
Spring 管理必须 @Component（或其它 Spring Bean 注解）同样需要 @Component
方法签名public ExecuteResult jobExecute(JobArgs)（反射查找）protected ExecuteResult doJobExecute(JobArgs)（继承覆盖）
可扩展性轻量，适合简单任务可在基类统一处理异常、日志、前置/后置钩子
适用场景90% 的场景——简单直观、名字即注册有统一基类诉求（如所有金融任务要同一套审计日志）
核心差异：注册机制的自动化程度
注解式任务在客户端启动时由 @EnableSnailJob 自动扫描并向服务端注册，你不需要在后台手动创建——改代码就生效。继承式则需要在后台手动指定类名，灵活但不够自动化。所以官方推荐注解式，本模块 8 个 demo 中 7 个都是注解式。

四、JobArgs — 服务端给执行器的「信使」

JobArgs 携带的信息： ├── getJobParams() → 服务端配置的「任务参数」（JSON 字符串，静态分片那课重点讲） ├── getWfContext() → DAG 工作流上下文（Map<String, Object>，DAG 那课重点讲） ├── getWfContext(key) → 按 key 取上下文中的一个值 ├── appendContext(k, v) → 向上文追加数据 ├── getTaskBatchId() → 本次执行批次 ID └── getJobId() → 任务定义 ID

现阶段先记住 getJobParams() 和 getWfContext() 两个核心方法。前者是「服务端静态配置的参数」，后者是「工作流节点间动态传递的上下文」。

五、ExecuteResult — 告诉服务端「我成了还是败了」

三个工厂方法： ExecuteResult.success() → 执行成功，参数可带返回对象（会被服务端记录） ExecuteResult.success(obj) → 执行成功 + 携带结果数据 ExecuteResult.failure() → 执行失败 ExecuteResult.failure("原因") → 执行失败 + 失败原因描述

注意：success() 不等于「任务不出错」——如果你在 jobExecute 里抛了未捕获的异常，SnailJob 客户端会自动捕获并标记本次执行为失败（并可能触发重试，取决于服务端配置）。返回 ExecuteResult.failure() 是主动宣告失败。

六、LOCAL vs REMOTE 日志 — 日志的两条路

为什么同一个 demo 里写了两行日志？

SnailJobLog.LOCAL.info("testJobExecutor. jobArgs:{}", JsonUtil.toJsonString(jobArgs));
SnailJobLog.REMOTE.info("testJobExecutor. jobArgs:{}", JsonUtil.toJsonString(jobArgs));
  • LOCAL — 写本地日志文件（和普通 log.info() 一样，落盘在服务器的 log 目录）
  • REMOTE — 写远程日志（通过 SnailLogbackAppender 上报到 SnailJob 服务端，在后台的「调度日志」面板可见）
一个常见陷阱：如果你只写 LOCAL，在 SnailJob 管理后台的日志面板里什么也看不到——运维排查问题时只能登机器看文件。REMOTE 解决了「任务失败了，到底报了什么错？」的远程诊断需求。通常情况下两种都写：LOCAL 给本地排查，REMOTE 给远程监控。

七、检验一下（提取练习）

Q1. @JobExecutor(name = "testJobExecutor") 中的 name 有什么作用？

A. 只是日志里的标识，没有实际功能 B. 作为任务名向 SnailJob 服务端注册，服务端通过这个名字来调度任务 C. 作为 Spring Bean 的名称 D. 作为类名缩写，方便代码跳转
看答案B。客户端启动时扫描 @JobExecutor，把 name → 类映射注册到服务端。管理员在后台创建任务时选这个名字，服务端就知道该调哪个类。

Q2. 继承式任务（extends AbstractJobExecutor）相比于注解式，主要优势是什么？

A. 执行速度更快，因为少了注解反射 B. 不需要 Spring 管理，可以 new 出来直接用 C. 可以在基类统一处理异常、日志、前置/后置钩子，适合有统一诉求的场景 D. 不需要在服务端后台创建任务
看答案C。继承式的价值在于基类模板方法——你可以把异常转换、审计日志、执行耗时统计等横切关注点写在 AbstractJobExecutor 的子类基类里。但它需要在服务端手动创建任务（填类名），不如注解式自动化。

Q3. 如果 jobExecute 方法里抛了 RuntimeException，SnailJob 会怎样？

A. 该实例崩溃，需要重启 B. 客户端框架捕获异常，标记本次执行为失败，根据服务端配置决定是否重试 C. 异常被吞掉，静默失败 D. 自动转为 ExecuteResult.failure() 返回
看答案B。SnailJob 客户端框架会捕获 jobExecute 抛出的未处理异常，标记本次执行为失败并上报服务端。是否重试、重试几次由服务端的任务配置决定。注意这和 return ExecuteResult.failure() 不同——前者是框架兜底，后者是主动宣告。

Q4. SnailJobLog.REMOTE.info(...) 的底层机制是什么？

A. 直接发 HTTP 请求到 SnailJob 服务端 B. 通过 SnailLogbackAppender（Logback 追加器）异步上报到服务端 C. 写本地文件，SnailJob 服务端定时来拉取 D. 通过 WebSocket 实时推送
看答案B。回顾第 1 课讲的 SnailJobConfig.onStarting()——它在客户端启动时创建 SnailLogbackAppender 挂到 Root Logger 上，所有走 Logback 的日志都会经它异步上报到服务端。
