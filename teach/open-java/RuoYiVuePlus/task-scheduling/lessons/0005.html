第 5 课 · Map 任务：动态分片，只分不汇 模式 5

🎯 本节目标（对应使命 Success #5）：学完你能讲清 Map 任务的完整链路——根 Map 方法拆分数据 → MapHandler.doMap() 将子任务分发给集群 → 各子 Map 方法并行执行 → 结果各自返回（不汇总），并理解它为什么比静态分片更灵活。

上一课静态分片的问题是：分片数写死在服务端。如果要处理 1~200 的累加，静态分片得手动配 "1,50" "51,100" "101,150" "151,200"。Map 任务则让代码自己决定怎么拆——数据变了，改代码就行。

一、TestMapJobAnnotation — 动态拆分 + 并行执行

完整源码（两个 @MapExecutor 方法）

ruoyi-job/src/main/java/org/dromara/job/snailjob/TestMapJobAnnotation.java 
@Component
@JobExecutor(name = "testMapJobAnnotation")
public class TestMapJobAnnotation {

    // ========== ① 根 Map 任务：拆分数据 ==========
    @MapExecutor
    public ExecuteResult doJobMapExecute(MapArgs mapArgs, MapHandler mapHandler) {
        // 生成 1~200 并每 50 个一组拆成 4 片
        int partitionSize = 50;
        List<Integer> sourceList = IntStream.rangeClosed(1, 200).boxed().toList();
        List<List<Integer>> partition = StreamUtils.groupByKey(sourceList,
            i -> (i - 1) / partitionSize)
            .values().stream().toList();

        SnailJobLog.REMOTE.info("端口:{}完成分配任务", SpringUtil.getProperty("server.port"));
        // 关键：把 4 片分发给集群中的子任务 "doCalc"
        return mapHandler.doMap(partition, "doCalc");
    }

    // ========== ② 子 Map 任务：处理一片数据 ==========
    @MapExecutor(taskName = "doCalc")
    public ExecuteResult doCalc(MapArgs mapArgs) {
        List<Integer> sourceList = (List<Integer>) mapArgs.getMapResult();
        // 累加这一片的所有数字
        int partitionTotal = sourceList.stream().mapToInt(i -> i).sum();
        ThreadUtil.sleep(3, TimeUnit.SECONDS);  // 模拟耗时
        SnailJobLog.REMOTE.info("端口:{},partitionTotal:{}",
            SpringUtil.getProperty("server.port"), partitionTotal);
        return ExecuteResult.success(partitionTotal);
    }
}

二、Map 任务的执行模型

① 服务端触发 testMapJobAnnotation │ ▼ ② 根 Map: doJobMapExecute() 执行 ├── 生成 1~200 的列表 ├── 按每 50 个拆分 → 4 片: [1~50] [51~100] [101~150] [151~200] └── mapHandler.doMap(partition, "doCalc") │ │ SnailJob 将这 4 片子任务放入队列 │ 由集群中的执行器认领 │ ├──→ doCalc([1~50]) 执行器 A → sum=1275 ├──→ doCalc([51~100]) 执行器 B → sum=3775 ├──→ doCalc([101~150]) 执行器 A → sum=6275 └──→ doCalc([151~200]) 执行器 C → sum=8775 ③ 每个子任务各自返回 ExecuteResult.success(partitionTotal) → 4 条独立的执行记录，各自有各自的 sum → 没有汇总阶段！（这是 Map vs MapReduce 的关键区别）

三、Map 任务的三个关键角色

角色注解方法签名职责
根 Map @MapExecutor（不带 taskName） ExecuteResult root(MapArgs, MapHandler) 准备数据 → 拆分 → 调用 mapHandler.doMap() 分发
子 Map @MapExecutor(taskName = "xxx") ExecuteResult sub(MapArgs) 接收一片数据 → 处理 → 返回结果
MapHandler 由框架注入 doMap(List<?> list, String taskName) 将拆分后的数据列表分发给集群中的子 Map 任务
@MapExecutor 不带 taskName = 根任务，带 taskName = 子任务。根任务只有一个（被 @JobExecutor(name) 指向），子任务可以多个。根任务通过 mapHandler.doMap(data, "子任务名") 将数据发给对应的子任务。如果数据量大（比如 10 万条拆成 1000 片），子任务会被多个执行器实例并行消费。

四、Map 任务 vs 静态分片（核心对比）

维度静态分片Map 任务（动态分片）
分片规则定义者服务端（后台配置）客户端（代码中的拆分逻辑）
分片数灵活度固定，改要动后台动态，数据变分片自动变
适合场景分片策略稳定（如每天按日期范围跑）数据量不可预测（如某个表行数不定）
子任务分发服务端直接调度mapHandler.doMap() 客户端触发分发
结果处理各自独立返回各自独立返回（Map 不汇总）

五、MapHandler.doMap() 的机制

mapHandler.doMap(partition, "doCalc") 做了什么？ 1. 接收参数：一个 List（4 片数据）+ 目标子任务名 2. 将 List 中的每个元素（一片数据）封装成一个 Map 子任务 3. 将这些子任务提交到 SnailJob 的任务队列 4. 集群中的执行器实例认领并执行（负载均衡） 5. 根任务的返回值 = doMap 的调度结果（不是子任务的汇总） 注意：doMap 只是触发分发，子任务是异步执行的。根任务返回 success 不代表子任务已经跑完——它们可能在队列中排队等待。
⚠️ Map 任务的「只分不汇」意味着什么？如果你需要知道「所有分片的累加总和」，仅靠 Map 任务是拿不到的——因为每个子任务的结果是独立的。你需要的是下一课的 MapReduce 任务，它有一个 @ReduceExecutor 来收集所有子任务结果并做汇总。Map = 分片处理 + 各自独立结果，MapReduce = 分片处理 + 汇总成一个最终结果。

六、检验一下（提取练习）

Q1. mapHandler.doMap(partition, "doCalc") 中第二个参数 "doCalc" 是什么？

A. 根任务的方法名 B. 子 Map 任务的名字——对应 @MapExecutor(taskName = "doCalc") C. 日志中的标签，没有实际路由功能 D. SnailJob 服务端的任务名
看答案B。根任务通过 taskName 指定要调用哪个子 Map 执行器。你可以在同一个类中定义多个子 Map（如 "doCalc"、"doEncrypt"、"doExport"），根任务选择性地分发给不同的子任务。

Q2. Map 任务中，如果 4 片子任务中有一片执行失败了，会发生什么？

A. 所有 4 片自动回滚 B. 失败的那片根据服务端配置重试，其余 3 片的结果不受影响 C. 根任务也标记为失败 D. 整个集群宕机
看答案B。Map 子任务是独立调度单元。每个子任务有自己的执行记录和重试策略（在服务端配置）。成功的那 3 片已经返回了各自的结果，失败的那片会按重试策略处理。

Q3. 根任务返回 ExecuteResult.success(...) 时，子任务一定都执行完了吗？

A. 一定执行完了，因为是同步调用 B. 不一定——mapHandler.doMap() 只是把子任务提交到队列，它们可能还在排队或执行中 C. 取决于子任务的数量 D. 子任务一定会失败，因为根任务先返回了
看答案B。doMap 是异步分发——它把子任务丢进 SnailJob 队列就返回了。根任务的返回值只表示「分发成功」，不代表子任务已执行完毕。这个异步特性是 Map 任务实现并行处理的基础。

Q4. @MapExecutor 不带 taskName 的方法有几个限制？

A. 没有限制，可以定义多个 B. 一个类里只能有一个不带 taskName 的 @MapExecutor——它是根任务 C. 必须和 @JobExecutor 分开在不同的类中 D. 必须返回 void
看答案B。不带 taskName 的 @MapExecutor 是根 Map 任务，一个 @JobExecutor 类中只能有一个。它必须有 MapArgs + MapHandler 两个参数。带 taskName 的可以有多个——它们是不同的子任务。
