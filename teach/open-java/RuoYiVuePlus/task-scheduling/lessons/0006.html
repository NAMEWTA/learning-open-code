第 6 课 · MapReduce 任务：分片 + 汇总 模式 6a

🎯 本节目标（对应使命 Success #6）：学完你能讲清 MapReduce 任务的完整链路——它在 Map 任务基础上增加了 @ReduceExecutor 汇总阶段，Reduce 方法如何通过 reduceArgs.getMapResult() 收集所有 Map 子任务的结果并做最终聚合。

上一课的 Map 任务各子任务结果独立返回。但很多场景下你需要一个最终结果——比如「1~200 的总和是多少」。MapReduce 就是 Map 任务 + 一个 Reduce 汇总阶段。

一、TestMapReduceAnnotation1 — Map 分片 + Reduce 汇总

完整源码（三个方法：根 Map + 子 Map + Reduce）

ruoyi-job/src/main/java/org/dromara/job/snailjob/TestMapReduceAnnotation1.java 
@Component
@JobExecutor(name = "testMapReduceAnnotation1")
public class TestMapReduceAnnotation1 {

    // ========== ① 根 Map：拆分数据（和上一课完全一样）==========
    @MapExecutor
    public ExecuteResult rootMapExecute(MapArgs mapArgs, MapHandler mapHandler) {
        int partitionSize = 50;
        List<Integer> sourceList = IntStream.rangeClosed(1, 200).boxed().toList();
        List<List<Integer>> partition = StreamUtils.groupByKey(sourceList,
            i -> (i - 1) / partitionSize).values().stream().toList();
        SnailJobLog.REMOTE.info("端口:{}完成分配任务", SpringUtil.getProperty("server.port"));
        return mapHandler.doMap(partition, "doCalc");
    }

    // ========== ② 子 Map：处理一片数据（和上一课完全一样）==========
    @MapExecutor(taskName = "doCalc")
    public ExecuteResult doCalc(MapArgs mapArgs) {
        List<Integer> sourceList = (List<Integer>) mapArgs.getMapResult();
        int partitionTotal = sourceList.stream().mapToInt(i -> i).sum();
        ThreadUtil.sleep(3, TimeUnit.SECONDS);
        SnailJobLog.REMOTE.info("端口:{},partitionTotal:{}",
            SpringUtil.getProperty("server.port"), partitionTotal);
        return ExecuteResult.success(partitionTotal);
    }

    // ========== ③ Reduce：汇总所有子任务结果 ★ 这是新增的！==========
    @ReduceExecutor
    public ExecuteResult reduceExecute(ReduceArgs reduceArgs) {
        // reduceArgs.getMapResult() → 所有子 Map 返回值的集合
        int reduceTotal = reduceArgs.getMapResult().stream()
            .mapToInt(i -> Integer.parseInt((String) i))
            .sum();
        SnailJobLog.REMOTE.info("端口:{},reduceTotal:{}",
            SpringUtil.getProperty("server.port"), reduceTotal);
        return ExecuteResult.success(reduceTotal);  // 返回总和 20100
    }
}

二、MapReduce 的完整执行模型

① 服务端触发 testMapReduceAnnotation1 │ ▼ ② 根 Map: rootMapExecute() └→ mapHandler.doMap(4片数据, "doCalc") │ ├──→ doCalc([1~50]) → sum=1275 ──┐ ├──→ doCalc([51~100]) → sum=3775 ──┤ ├──→ doCalc([101~150]) → sum=6275 ──┤ 所有子任务执行完毕 └──→ doCalc([151~200]) → sum=8775 ──┘ │ ▼ ③ Reduce: reduceExecute() reduceArgs.getMapResult() = [1275, 3775, 6275, 8775] → 汇总计算: 1275+3775+6275+8775 = 20100 → return ExecuteResult.success(20100) 最终结果：20100（1+2+...+200 的总和）
MapReduce 和 Map 的唯一区别就是多了 Reduce！根 Map 和子 Map 的代码完全不变。你只需要添加一个 @ReduceExecutor 方法，它会在所有子 Map 任务执行完毕后自动被调用，通过 reduceArgs.getMapResult() 拿到所有子任务返回值的集合，做最终聚合。

三、ReduceArgs — 汇总阶段的「数据收集器」

ReduceArgs 的核心方法： ├── getMapResult() → 所有子 Map 返回值的 List（类型取决于子 map 返回了什么） ├── getTaskBatchId() → 本次执行批次 ID └── getJobId() → 任务定义 ID 注意：getMapResult() 返回的每个元素是对应子 Map 任务 ExecuteResult.success(obj) 中的 obj。 在本例中，每个子 Map 返回 ExecuteResult.success(partitionTotal)，partitionTotal 是 int。 所以 getMapResult() 返回的是包含 4 个 int 的集合。

类型转换的坑

注意代码中的类型转换：

int reduceTotal = reduceArgs.getMapResult().stream()
    .mapToInt(i -> Integer.parseInt((String) i))  // ← 为什么要 toString？
    .sum();

因为 SnailJob 框架在传输过程中可能将子任务返回值统一序列化为 String。所以 Reduce 中需要 Integer.parseInt((String) i) 来还原。这是一个常见的坑——Reduce 中拿到的类型不一定和子 Map 返回的类型一致，取决于框架的序列化机制。

四、Map vs MapReduce 场景选型指南

场景选 Map选 MapReduce原因
统计所有用户订单总金额✓需要最终汇总为一个总数
给每个用户发个性化通知✓每个用户的结果独立有意义，不需要汇总
批量导出报表文件✓每个分片生成一个文件，不需要合并
全量用户标签重新计算✓每个用户的标签独立写入，各自成功即可
计算所有商品的平均评分✓需要汇总所有分片的 (总分, 总数) 再算平均
数据对账（A表 vs B表）✓各分片返回差异数，Reduce 汇总成差异报告

五、MapReduce 的扩展：多 Reduce 阶段

SnailJob 支持在一个 Job 中定义多组 Map-Reduce 链。但本模块的 demo 只展示了最基础的一层 Map + 一层 Reduce。更复杂的拓扑（如 Map → Reduce → Map → Reduce）需要通过 DAG 工作流来实现——这是下一课的内容。

六、检验一下（提取练习）

Q1. @ReduceExecutor 方法什么时候被调用？

A. 和根 Map 同时执行 B. 在所有子 Map 任务执行完毕后自动调用 C. 每个子 Map 执行完就调一次 D. 需要手动在代码中调用
看答案B。Reduce 阶段的触发条件是所有子 Map 任务执行完毕。SnailJob 框架自动等待，然后聚合结果调用 Reduce 方法。这是一个隐式的屏障（barrier）。

Q2. 如果 4 个子 Map 中有一个执行失败了，Reduce 方法还会被调用吗？

A. 会，Reduce 会跳过失败的那个子任务的结果 B. 不会，Reduce 只在所有子 Map 都成功后才执行 C. 会，但 reduceArgs.getMapResult() 中失败的那一条值为 null D. 取决于服务端配置的失败策略
看答案B。MapReduce 的屏障语义是「所有子 Map 成功」→ Reduce。如果有子任务失败，Reduce 不会执行，整批任务标记为失败（具体行为可能受服务端重试策略影响，但默认如此）。

Q3. 子 Map 返回 ExecuteResult.success(1275)，在 Reduce 中 getMapResult() 拿到的元素类型是什么？

A. 一定是 int 类型 B. 可能是 String（框架序列化机制可能转换类型），需要手动转换 C. 是 ExecuteResult 对象 D. 是 MapArgs 对象
看答案B。注意代码中用了 Integer.parseInt((String) i) 来做类型转换——这说明框架在传输过程中把 int 转成了 String。这是 MapReduce 开发中最常见的坑：不要在 Reduce 中假设子 Map 返回值的类型，先打日志看实际类型。

Q4. 和上一课的 Map 任务相比，MapReduce 多出了什么？

A. 多了一个根 Map 方法 B. 多了一个 @ReduceExecutor 方法 + 子任务全部完成后自动调用 Reduce C. 多了一个 MapHandler D. 多了一个 @JobExecutor 注解
看答案B。根 Map 和子 Map 的代码完全不变（你可以对比两个 demo 类）。唯一的增量就是 @ReduceExecutor 汇总方法。从 Map 到 MapReduce，加一个方法就够了。
