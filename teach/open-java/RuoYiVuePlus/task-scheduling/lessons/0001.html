🎯 本节目标（对应使命 Success #1）：学完你能说清 ruoyi-job 模块的整体定位、它和 ruoyi-common-job 的「客户端 / 自动装配」分工、以及 SnailJob 的 6 种任务模式各是什么。这是后续每节课的挂载骨架——先有全景地图，再逐条走小路。

很多人打开 ruoyi-job 目录，看到一堆 TestXxxJob 类，第一反应是「这些 demo 在演示什么？」。这节课我们把整个模块摊开看——它不只是「一个跑定时任务的地方」，它是 SnailJob 分布式任务调度框架在 RuoYi 里的落地层。

一、先记住一句话：它是「SnailJob 的业务 Demo + 自动装配层」

和 ruoyi-common-redis 那种「封装一个框架给业务用」不同，ruoyi-job 的角色更纯粹：

ruoyi-common/ruoyi-common-job/ ← 【自动装配层】引入 SnailJob，配好客户端，开箱即用 └── SnailJobConfig ← @EnableSnailJob + 远程日志挂载 ruoyi-modules/ruoyi-job/ ← 【业务 Demo 层】展示 6 种任务模式怎么写 ├── entity/BillDTO.java ← DAG 工作流 demo 用的数据对象 └── snailjob/ ├── TestAnnoJobExecutor ← 模式1：注解式任务 ├── TestClassJobExecutor ← 模式2：继承式任务 ├── TestBroadcastJob ← 模式3：广播任务 ├── TestStaticShardingJob ← 模式4：静态分片 ├── TestMapJobAnnotation ← 模式5：Map 任务（分片不汇总） ├── TestMapReduceAnnotation1 ← 模式6：MapReduce 任务（分片+汇总） ├── WechatBillTask ← 模式6扩展：DAG工作流-微信账单 ├── AlipayBillTask ← 模式6扩展：DAG工作流-支付宝账单 └── SummaryBillTask ← 模式6扩展：DAG工作流-汇总账单

设计动机 · 为什么是 SnailJob 而不是 XXL-JOB？
SnailJob 是爱组搭（aizuda）开源的企业级分布式任务调度与重试平台。相比 XXL-JOB，SnailJob 提供了一些差异化能力：MapReduce 分片计算模型、DAG 工作流编排、广播任务、以及天然的重试模块集成。RuoYi-Vue-Plus 选它，看中的是「调度 + 重试 + 工作流」三合一的开箱能力。课程中的 8 个 demo 类就是这些能力的完整展示。

二、SnailJob 客户端是怎么「通电」的？

任何一个 RuoYi 模块要跑 SnailJob 任务，只需两步：

① 引入 ruoyi-common-job 依赖（自动装配启动）

ruoyi-common/ruoyi-common-job/src/main/java/org/dromara/common/job/config/SnailJobConfig.java 
@AutoConfiguration
@ConditionalOnProperty(prefix = "snail-job", name = "enabled", havingValue = "true")
@EnableScheduling
@EnableSnailJob
public class SnailJobConfig {

    @EventListener(SnailClientStartingEvent.class)
    public void onStarting(SnailClientStartingEvent event) {
        // 在 SnailJob 客户端启动时，挂载远程日志 appender
        LoggerContext lc = (LoggerContext) LoggerFactory.getILoggerFactory();
        SnailLogbackAppender<ILoggingEvent> ca = new SnailLogbackAppender<>();
        ca.setName("snail_log_appender");
        ca.start();
        Logger rootLogger = lc.getLogger(Logger.ROOT_LOGGER_NAME);
        rootLogger.addAppender(ca);
    }
}

四个关键注解，缺一不可：

  • @AutoConfiguration — Spring Boot 3.x 自动装配入口（通过 AutoConfiguration.imports 文件注册）
  • @ConditionalOnProperty — 用 snail-job.enabled=true 做总开关，关了就什么都不加载
  • @EnableScheduling — 开启 Spring 的定时任务线程池（SnailJob 依赖它调度）
  • @EnableSnailJob — 激活 SnailJob 客户端，扫描所有 @JobExecutor 并注册到服务端
SnailLogbackAppender 是做什么的？普通 log.info() 只写本地日志文件。SnailLogbackAppender 是一个 Logback 追加器，它把日志同时上报到 SnailJob 服务端。这就是为什么你在代码里看到 SnailJobLog.REMOTE.info(...) 能直接写到 SnailJob 后台的日志面板——它背后是 Logback Appender 机制，不是魔法。

三、6 种任务模式速览

模式Demo 类一句话适用场景
1. 注解式任务 TestAnnoJobExecutor 贴 @JobExecutor(name="xxx") 就注册 简单定时任务：报表生成、数据同步
2. 继承式任务 TestClassJobExecutor 继承 AbstractJobExecutor 重写 doJobExecute 需要基类统一处理（如统一异常、统一日志格式）
3. 广播任务 TestBroadcastJob 所有在线实例各执行一次 全实例缓存清理、配置刷新、批量通知
4. 静态分片 TestStaticShardingJob 服务端下发分片参数（如 ID 范围），各实例各管一段 按 ID/日期范围批量处理数据
5. Map 任务 TestMapJobAnnotation 根任务动态拆分 → 子任务并行执行 → 不汇总 大量数据并行处理，只关心每个分片的结果
6. MapReduce + DAG TestMapReduceAnnotation1 + 三个 BillTask Map 拆分并行 + Reduce 汇总；DAG 用上下文串联节点 并行计算后需聚合、多步骤工作流编排
为什么 Map 和 MapReduce 是分开的两种模式？Map 任务「只分不汇」——适用于每个分片的结果独立有意义（如给每个用户发通知，发送结果各自记录即可）。MapReduce 任务「分完汇总」——适用于分片结果是中间产物（如各分片算累加值，最终要一个总和）。DAG 工作流则是 MapReduce 的上层编排：你可以把多个 MapReduce 任务串成一个有向无环图，通过 wfContext 传递上下文。

四、从「客户端注册」到「服务端调度」的完整链路

┌─────────────────────────────────────────────────┐ │ SnailJob 服务端 (独立部署，端口 17888) │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ 命名空间 │ │ 分组管理 │ │ 任务管理 │ │ │ │ namespace │ │ group │ │ job CRUD │ │ │ └──────────┘ └──────────┘ └──────────┘ │ └──────────────┬──────────────────────────────────┘ │ HTTP/gRPC │ 客户端上报 + 心跳 + 日志 ┌──────────────▼──────────────────────────────────┐ │ RuoYi 应用 (SnailJob 客户端) │ │ │ │ ① SnailJobConfig.@EnableSnailJob │ │ └→ 启动时扫描所有 @JobExecutor │ │ └→ 向服务端注册任务 (name → 类映射) │ │ │ │ ② SnailClientStartingEvent │ │ └→ 挂载 SnailLogbackAppender (远程日志) │ │ │ │ ③ 等待调度 │ │ └→ 服务端按时触发 → 客户端执行 → 上报结果 + 日志 │ └─────────────────────────────────────────────────┘

关键理解：任务代码写在 RuoYi 应用里，但调度规则（cron、参数、分片数等）在 SnailJob 服务端管理后台配置。这种「代码在客户端、配置在服务端」的分离设计，让你可以不改代码就调整调度策略。

Q1. ruoyi-job 和 ruoyi-common-job 的分工是什么？

A. ruoyi-job 做自动装配，ruoyi-common-job 写业务 demo B. ruoyi-common-job 做自动装配（SnailJobConfig），ruoyi-job 写业务 demo（8 个示例任务） C. 两者完全相同，只是为了避免循环依赖拆成了两个模块 D. ruoyi-common-job 是 SnailJob 服务端，ruoyi-job 是客户端
看答案B。common-job 是自动装配层（引入 SnailJob 依赖 + @EnableSnailJob + 远程日志挂载），ruoyi-job 是业务 Demo 层（展示 6 种任务模式怎么写）。

Q2. snail-job.enabled=false 之后会发生什么？

A. SnailJob 客户端仍然启动，但不执行任何任务 B. 整个 SnailJobConfig 不会被加载（@ConditionalOnProperty 拦截），不连服务端、不注册任务 C. 只有远程日志被关闭，任务调度正常进行 D. 编译报错，因为 enabled 是必填项
看答案B。@ConditionalOnProperty(prefix="snail-job", name="enabled", havingValue="true") 会让 SnailJobConfig 整段不加载，就像没引入 SnailJob 一样。这是开发环境省资源的关键设计。

Q3. 以下哪个不是 ruoyi-job 模块中展示的任务模式？

A. 广播任务（所有实例同时执行） B. MapReduce 任务（分片 + 汇总） C. 流水线任务（Pipeline，多个任务串联执行） D. DAG 工作流任务（用 wfContext 在节点间传递上下文）
看答案C。本模块展示了 6 种模式：注解式、继承式、广播、静态分片、Map、MapReduce+DAG。Pipeline 流水线不是本模块的模式（但 DAG 工作流可以实现类似效果）。

Q4. SnailLogbackAppender 是做什么的？

A. 替代 Logback，提供更快的日志框架 B. 一个 Logback 追加器，将日志同时上报到 SnailJob 服务端，让你在后台看到任务执行日志 C. 压缩日志文件以节省磁盘空间 D. 把日志从中文翻译成英文
看答案B。它是一个 Logback Appender，挂在 Root Logger 上，运行时把日志实时推送到 SnailJob 服务端。这就是 SnailJobLog.REMOTE.info() 能远程可见的底层机制。
