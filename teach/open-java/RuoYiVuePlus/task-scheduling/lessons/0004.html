第 4 课 · 静态分片：按参数范围切割任务 模式 4

🎯 本节目标（对应使命 Success #4）：学完你能讲清静态分片的原理——服务端通过 jobArgs.getJobParams() 下发分片参数（如 ID 范围），每个执行器各取一段数据独立处理，并理解它和下一课「动态分片（Map）」的本质区别。

想象你要对 100 万用户做数据加密。单机跑通宵，分 10 台机器各跑 10 万条就快多了。静态分片就是干这个的：服务端定义分片规则 → 下发分片参数给执行器 → 各执行器按参数处理自己那一段。

一、TestStaticShardingJob — 服务端告诉我范围，我干我的

源码拆解

ruoyi-job/src/main/java/org/dromara/job/snailjob/TestStaticShardingJob.java 
@Component
@JobExecutor(name = "testStaticShardingJob")
public class TestStaticShardingJob {

    public ExecuteResult jobExecute(JobArgs jobArgs) {
        // ① 从服务端获取分片参数
        String jobParams = Convert.toStr(jobArgs.getJobParams());
        SnailJobLog.LOCAL.info("开始执行分片任务,参数:{}", jobParams);

        // ② 解析分片范围（格式: "fromId,toId"）
        String[] split = jobParams.split(",");
        Long fromId = Long.parseLong(split[0]);
        Long toId = Long.parseLong(split[1]);

        // ③ 模拟数据库操作：对 fromId ~ toId 范围进行加密
        try {
            SnailJobLog.REMOTE.info("开始对id范围:{}进行加密处理", fromId + "-" + toId);
            Thread.sleep(3000);  // 模拟耗时处理
            SnailJobLog.REMOTE.info("对id范围:{}进行加密处理完成", fromId + "-" + toId);
        } catch (InterruptedException e) {
            return ExecuteResult.failure("任务执行失败");
        }
        return ExecuteResult.success("执行分片任务完成");
    }
}

「静态分片」的「静态」体现在：分片规则在服务端创建任务时就固定了——比如在 SnailJob 后台配置 3 个分片，参数分别为 "1,10000"、"10001,20000"、"20001,30000"。调度时服务端把这 3 组分片参数分别发给 3 个执行器实例（或分多次发给同一个实例）。

二、静态分片的执行模型

SnailJob 服务端（任务已配置 3 个分片） │ │ 分片 0: jobParams = "1,33333" │ 分片 1: jobParams = "33334,66666" │ 分片 2: jobParams = "66667,100000" │ │ 调度时分配： │ ├──→ 实例 A (端口 28080) 拿到 "1,33333" → 加密 ID 1~33333 ├──→ 实例 B (端口 28081) 拿到 "33334,66666" → 加密 ID 33334~66666 └──→ 实例 C (端口 28082) 拿到 "66667,100000" → 加密 ID 66667~100000 关键点：分片参数在服务端定义，执行器只管「接参数→干活→返回」 → 如果要改分片数，在后台改，不需要改代码
为什么叫「静态」？因为分片的数量和每片的参数在服务端任务配置里是预先写死的。比如你写了 3 片就是 3 片。与之对应的是下节课的 Map 任务「动态分片」——分片数量和内容由执行器代码在运行时动态决定。静态 = 服务端定规则，动态 = 客户端定规则。

三、分片参数的设计模式

TestStaticShardingJob 用了最简单的格式——逗号分隔的起止 ID。但在实际项目中，jobParams 可以是任意字符串，常见的包括：

常见的 jobParams 格式： ├── "1,10000" ← 起止范围（本课用） ├── '{"startDate":"2026-01","endDate":"2026-06"}' ← JSON 日期范围 ├── "shard:0,total:3" ← 分片索引 + 总数（执行器自己算范围） ├── "TABLE_0,TABLE_1" ← 分表名（按表分片） └── "province:guangdong" ← 业务维度（按地区分片）

分片设计的第一原则：每片的数据量大致相等，避免「某些片跑完了、某个片还在跑」的倾斜。

四、静态分片 vs 广播任务（容易混淆）

维度广播任务静态分片
各实例执行的内容相同（都跑同一段逻辑）不同（各跑不同数据范围）
参数来源无特殊参数（或全局统一参数）服务端为每个分片单独配置参数
典型场景清缓存、刷配置、采指标批量加密、数据迁移、大批量推送
数据重复处理会（每个实例都处理全量）不会（各处理一片，理论无重叠）
⚠️ 分片重叠陷阱：如果静态分片参数写成了 "1,10000" 和 "5000,15000"——ID 5000~10000 被两个分片重复处理！分片参数必须不重不漏。这是最容易出错的地方。

五、静态分片的局限

  • 分片数固定：要改分片数就得去 SnailJob 后台改配置。数据量暴增 10 倍 → 手动改 10 个分片参数？麻烦。
  • 不能动态调整：如果某片数据特别多（如 ID 1~10000 有 9000 条数据，10001~20000 只有 100 条），会严重倾斜。
  • 分片失败处理：如果执行分片 1 的实例挂了，服务端需要重试机制——这不是代码能解决的，依赖服务端配置。

这些局限正是下一课 Map 任务（动态分片）要解决的问题。

六、检验一下（提取练习）

Q1. 静态分片的「分片参数」是在哪里定义的？

A. 在执行器的 Java 代码中硬编码 B. 在 SnailJob 服务端后台创建任务时配置的 jobParams C. 在执行器的 yml 配置文件中 D. 由执行器在运行时自己生成
看答案B。静态分片的分片参数在 SnailJob 服务端后台配置（创建任务 → 选择分片路由 → 填写每个分片的 jobParams）。执行器通过 jobArgs.getJobParams() 获取自己该处理哪一段。这是「静态」的含义来源。

Q2. 如果需要处理 100 万条数据，配置了 10 个静态分片，但只有 3 个实例在线，会发生什么？

A. 只有 3 个分片被执行，其余 7 个丢失 B. 10 个分片任务排队，3 个实例逐个认领执行——有些实例会执行多个分片 C. 报错，因为分片数和实例数不匹配 D. 服务端自动合并成 3 个分片
看答案B。分片数和实例数不需要相等。服务端会把 10 个分片任务放入队列，3 个在线实例各自认领——快的实例多领几个。分片是逻辑概念，实例是物理概念，两者解耦。

Q3. jobArgs.getJobParams() 返回的是什么类型？

A. 直接返回 String，不需要转换 B. 返回 Object，需要 Convert.toStr() 转成 String C. 返回 Map<String, Object>，按 key 取 D. 返回 byte[]，需要反序列化
看答案B。注意代码中用的是 Convert.toStr(jobArgs.getJobParams())（Hutool 的 Convert 工具），说明原始返回类型是 Object。这是因为服务端可能下发不同类型的数据。

Q4. 静态分片和动态分片（Map 任务）最本质的区别是什么？

A. 静态分片用 XML 配置，动态分片用注解 B. 静态分片的分片规则在服务端预先定义，动态分片的分片规则由执行器代码在运行时决定 C. 静态分片只能处理数字，动态分片可以处理任意类型 D. 没有本质区别，只是两种不同的 API 写法
看答案B。静态分片 = 服务端定规则（改分片要改后台配置），动态分片 = 客户端定规则（改分片改代码即可）。这是理解后续 Map/MapReduce 任务的前置知识。
