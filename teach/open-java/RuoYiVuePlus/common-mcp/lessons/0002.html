第 2 课 · McpAutoConfiguration 与 McpClientTemplate 源码拆解 核心机制

🎯 本节目标（对应使命 Success #3~#4）：学完你能解释 McpAutoConfiguration 的条件装配逻辑，并逐一说出 McpClientTemplate 的 6 个核心方法各自做什么、为什么这样设计。

一、McpAutoConfiguration：什么是「条件装配」？

先看源码（McpAutoConfiguration.java，完整 29 行）：

@AutoConfiguration                                          // 1. 声明为自动配置类
public class McpAutoConfiguration {

    @Bean
    @ConditionalOnBean(McpSyncClient.class)                 // 2. 前提：必须有 McpSyncClient Bean
    @ConditionalOnMissingBean                                // 3. 如果没有用户自定义的同名 Bean
    public McpClientTemplate mcpClientTemplate(
                List<McpSyncClient> mcpSyncClients) {       // 4. 注入所有 McpSyncClient
        return new McpClientTemplate(mcpSyncClients);
    }
}

这 10 行代码里藏着三个关键设计决策：

1.1 决策一：@AutoConfiguration 而非 @Configuration

Spring Boot 3.x 建议使用 @AutoConfiguration 替代 @Configuration 用于自动配置场景。区别在于：

  • @AutoConfiguration 的加载顺序晚于用户自定义的 @Configuration，避免自动配置「抢跑」。
  • 它需要通过 AutoConfiguration.imports 文件显式注册（第 1 课已看到），而不是靠 @ComponentScan 发现。

1.2 决策二：@ConditionalOnBean(McpSyncClient.class) — 条件装配的核心

这是整个模块最精妙的设计。McpSyncClient Bean 是谁创建的？不是本模块创建的——它是 Spring AI MCP Client Starter 根据 application.yml 中的配置自动创建的。

工作流程：

  1. 用户（或运维）在 application.yml 中配置了 MCP Server 连接信息 → Spring AI MCP Starter 创建对应的 McpSyncClient Bean。
  2. @ConditionalOnBean(McpSyncClient.class) 检测到 McpSyncClient Bean 存在 → 条件满足 → 创建 McpClientTemplate。
  3. 如果没有配置任何 MCP 连接 → 没有 McpSyncClient Bean → 条件不满足 → McpClientTemplate 不创建。
💡 设计动机：ruoyi-common-mcp 被设计为所有 ruoyi-common-* 模块的兄弟，会被父 POM 全局引入。但并不是每个 RuoYi 项目都需要 MCP 能力——条件装配保证了「依赖了但不用，不报错也不浪费资源」。

1.3 决策三：@ConditionalOnMissingBean — 允许业务方覆盖

如果业务模块自己定义了一个 McpClientTemplate Bean（例如添加日志、限流、自定义扩展），这里的 @ConditionalOnMissingBean 会检测到已存在同名 Bean，从而跳过自动创建。这是 Spring Boot 自动配置的经典模式——「自动配置兜底，用户自定义优先」。

1.4 参数注入：List<McpSyncClient> — 收集所有已连接的 Client

Spring 自动将容器中所有 McpSyncClient 类型的 Bean 收集成一个 List 注入。如果用户配置了 3 个 MCP Server 连接，这里就能拿到 3 个 McpSyncClient。这就是后面「支持多 Server」的基础。

二、McpClientTemplate：6 个方法的封装逻辑

看源码（McpClientTemplate.java）：

@RequiredArgsConstructor                                      // Lombok：生成含全部 final 字段的构造器
public class McpClientTemplate {

    private final List<McpSyncClient> mcpSyncClients;         // 启动时注入所有 MCP Client

    // 方法 1：查询所有 Server 的工具列表       → 广播模式
    // 方法 2：广播调用所有 Server 的同名工具    → 广播模式
    // 方法 3：精确调用指定 Server 的工具       → 单点模式
    // 方法 4：广播读取所有 Server 的同名资源    → 广播模式
    // 方法 5：精确读取指定 Server 的资源       → 单点模式
    // 方法 6：按名称查找某个 Client           → 内部辅助
}

核心设计模式：每种操作都有两个版本——广播版（所有 Server）和单点版（指定 Server）。

2.1 方法一：listTools() — 列出所有工具

public Map<String, List<String>> listTools() {
    Map<String, List<String>> result = new LinkedHashMap<>();  // 保持插入顺序
    for (McpSyncClient client : mcpSyncClients) {
        List<String> tools = client.listTools()                 // ← Spring AI MCP 的方法
            .tools()                                           // 拿到 List<Tool>
            .stream()
            .map(McpSchema.Tool::name)                         // 只取工具名
            .toList();
        result.put(getServerName(client), tools);              // key=Server 名称, value=工具列表
    }
    return result;
}

调用链：McpClientTemplate.listTools() → 遍历每个 McpSyncClient → client.listTools()（Spring AI MCP SDK 方法）→ 提取工具名列表 → 按 Server 名称分组返回。

LinkedHashMap 的选择：保证遍历顺序与 mcpSyncClients 列表一致，方便调试。

2.2 方法二（广播版）：callTool(toolName, arguments)

public Map<String, McpToolCallResult> callTool(
            String toolName, Map<String, Object> arguments) {
    Map<String, McpToolCallResult> result = new LinkedHashMap<>();
    for (McpSyncClient client : mcpSyncClients) {
        McpSchema.CallToolResult callResult = client.callTool(  // ← MCP 协议的工具调用
            new McpSchema.CallToolRequest(toolName, arguments)  // 构建请求：工具名 + 参数 Map
        );
        result.put(getServerName(client),                       // key=Server 名称
            McpToolCallResult.of(getServerName(client), callResult));  // 包装成项目结果对象
    }
    return result;
}

为什么需要封装这一步？ 如果不封装，业务代码需要：1) 手动遍历 mcpSyncClients；2) 手动 new CallToolRequest(...)；3) 手动处理返回的 CallToolResult 原始对象（它的 content 是 SDK 类型，不方便直接序列化返回前端）。封装后业务代码只需一行：mcpClientTemplate.callTool("weather", Map.of("city", "北京"))。

2.3 方法三（单点版）：callTool(serverName, toolName, arguments)

public Optional<McpToolCallResult> callTool(
            String serverName, String toolName,
            Map<String, Object> arguments) {
    return findClient(serverName)                               // 先按名称找到对应 Client
        .map(client -> McpToolCallResult.of(serverName,
            client.callTool(
                new McpSchema.CallToolRequest(toolName, arguments))));
}

返回 Optional 而非直接抛异常：如果 Server 名称写错了（找不到对应的 Client），返回 Optional.empty()，由调用方决定如何处理——这是 Java 函数式编程的惯用做法。

2.4 方法四+五：readResource — 资源读取（结构完全对称）

广播版：readResource(String uri) → 向所有 Server 请求同名资源 → 返回 Map<String, McpResourceReadResult>

单点版：readResource(String serverName, String uri) → 向指定 Server 请求资源 → 返回 Optional<McpResourceReadResult>

与 callTool 的设计完全对称：遍历逻辑相同、分组方式相同、Result 包装方式相同。这降低了学习成本——懂了 callTool 就懂了 readResource。

2.5 辅助方法：findClient 与 getServerName

// 按名称查找 Client（供 callTool(serverName, ...) 和 readResource(serverName, ...) 使用）
public Optional<McpSyncClient> findClient(String serverName) {
    return mcpSyncClients.stream()
        .filter(client -> serverName.equals(getServerName(client)))
        .findFirst();
}

// 从 McpSyncClient 中解析 Server 名称
private String getServerName(McpSyncClient client) {
    McpSchema.Implementation serverInfo = client.getServerInfo();
    if (serverInfo == null || serverInfo.name() == null) {
        return "unknown";                                      // 防御性编程：拿不到名称也不崩
    }
    return serverInfo.name();
}

每个 McpSyncClient 连接建立后，MCP 协议的 initialize 握手阶段会交换双方的能力信息，其中包含 serverInfo（Server 名称 + 版本号）。getServerName 把 Server 名称提取出来作为 Map 的 key。

💡 设计亮点：为什么是 "unknown" 而非 null？ 防御性编程。MCP 协议的 serverInfo.name 是可选字段——不能假定每个 Server 都填了名称。如果为 null，直接用 "unknown" 占位，避免后续 Map.put(null, ...) 引发 NPE。

四、检验一下

Q1. @ConditionalOnBean(McpSyncClient.class) 的「条件」具体是什么？谁负责创建 McpSyncClient？

A. 条件是有任何 Bean，由 McpAutoConfiguration 创建 McpSyncClient B. 条件是存在 McpSyncClient Bean，由 Spring AI MCP Starter 根据 yml 配置自动创建 C. 条件是配置文件中有 mcp 前缀，由 application.yml 直接创建 D. 条件是引入了 spring-ai-starter-mcp-client，由 pom.xml 创建
看答案B。McpSyncClient 由 Spring AI MCP Client Starter 根据 yml 配置自动创建。当用户配置了 MCP Server 连接 → Starter 创建对应的 McpSyncClient Bean → 条件满足 → 本模块创建 McpClientTemplate。

Q2. callTool(toolName, args) （广播版）向所有 Server 发送同一个工具调用，返回的 Map 的 key 是什么？

A. 工具名称（toolName） B. Server 名称（通过 client.getServerInfo().name() 获取） C. 参数 args 的 hashCode D. 固定字符串 "default"
看答案B。每个 McpSyncClient 在连接 MCP Server 后会拿到 serverInfo，Map 的 key 就是 getServerName(client) 也就是 serverInfo.name()。

Q3. findClient("database-server") 找不到时返回什么？

A. null B. 抛出 IllegalArgumentException C. Optional.empty() D. 返回第一个 Client 作为兜底
看答案C。Optional.empty()。这是 Java 函数式编程的惯用做法——让调用方自己决定找不到时如何处理（抛异常 / 给默认值 / 跳过）。
