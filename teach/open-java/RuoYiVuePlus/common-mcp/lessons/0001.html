第 1 课 · MCP 协议简介与模块全景 入门地图

🎯 本节目标（对应使命 Success #1~#2）：学完你能用一句话说清 MCP 协议是什么，画出 Client-Server 通信模型图，并说出 ruoyi-common-mcp 模块仅有的四个类各自干什么。这是后面两节课所有细节挂靠的骨架——先理解协议，再看 Java 封装。

我们先不抠任何一行实现代码，只回答一个核心问题：MCP 协议是什么？RuoYi 又是怎么把它接入 Java 项目的？

一、MCP 协议：AI 应用的「USB-C 接口」

MCP（Model Context Protocol） 是由 Anthropic 提出并已演化为行业标准的开放协议。你可以把它理解为 AI 应用与外部世界的「USB-C 接口」——只要双方都遵循 MCP，AI 应用（Host）就能与各种外部服务（Server）通信，获取工具能力和数据资源。

MCP 核心模型：Client-Server 架构 
┌─────────────────────┐         MCP 协议（JSON-RPC）         ┌──────────────────────┐
│                     │ ◄──────────────────────────────────►  │                      │
│  MCP Client (Host)  │    Tools: 调用函数（有参数/有返回）     │  MCP Server (服务)    │
│  例如：RuoYi 后端     │    Resources: 读取数据（URI 访问）      │  例如：数据库 Server   │
│    + Spring AI MCP   │    Prompts: 获取提示模板               │  例如：文件系统 Server  │
│                     │                                       │                      │
└─────────────────────┘                                       └──────────────────────┘

三大原语（Primitives）：

原语语义类比本模块支持
Tools（工具）Client 调用 Server 暴露的函数，传参并获取返回类似 RPC / API 调用✅ callTool()
Resources（资源）Client 读取 Server 暴露的数据（文件内容、数据库记录等）类似 REST GET / 文件读取✅ readResource()
Prompts（提示模板）Server 提供可复用的提示/指令模板给 LLM类似预置 Prompt 库❌ 本模块未封装

MCP 的底层走 JSON-RPC 2.0 消息，传输层支持 SSE（Server-Sent Events）、stdio（标准输入输出）、Streamable HTTP 等多种方式。这些细节全被 Spring AI MCP 封装了，本模块不需要关心。

💡 为什么叫「USB-C 接口」？ 在 MCP 之前，每个 AI 应用要对接外部工具都要写定制集成代码（就像每种设备需要不同的接口线）。MCP 定义了统一标准后，AI 应用只需实现一次 MCP Client，就能连接任何 MCP Server——就像 USB-C 统一了充电和数据传输标准。

二、Spring AI MCP：Java 世界的 MCP 桥接

Spring AI 是 Spring 生态面向 AI 应用开发的项目，其中的 spring-ai-starter-mcp-client 和 spring-ai-starter-mcp-server-webmvc 是 MCP 协议的 Java 实现（本项目使用 2.0.0 版本）。

Spring AI MCP 为你做了这些事：

  • 连接管理：根据 application.yml 中 spring.ai.mcp.client.connections 的配置，自动创建并初始化 McpSyncClient 实例。
  • 协议封装：将 MCP 的 JSON-RPC 消息、Lifecycle（initialize/initialized 握手）、传输层细节全部封装为 Java 方法调用。
  • Spring 集成：将 McpSyncClient 注册为 Spring Bean，支持依赖注入。
  • Schema 定义：提供 McpSchema.Tool、McpSchema.CallToolRequest、McpSchema.ReadResourceResult 等类型安全的 Java Record。
💡 一句话定位：
ruoyi-common-mcp 不是 MCP 的 Java 实现——那由 Spring AI MCP 完成。
它是一个「项目级便利封装」：把 Spring AI 提供的 McpSyncClient 用起来更顺手，统一返回格式、支持按名称筛选、批量广播。

打开 ruoyi-common-mcp，里面只有这些文件：

记住这张表，本课就成功了一半：

四、POM 依赖：两个 Starter 分别干什么

看 pom.xml（ruoyi-common-mcp/pom.xml）：

依赖作用是否必须
ruoyi-common-core 依赖 RuoYi 核心模块（工具类、常量等） 是（项目内依赖）
spring-ai-starter-mcp-server-webmvc MCP Server 端支持：让你的应用也可以作为 MCP Server 对外暴露工具 本模块作为 Client 使用时，非必须（但统一引入以便后续扩展）
spring-ai-starter-mcp-client MCP Client 端支持：让你的应用作为 Client 连接外部 MCP Server 是——本模块的核心能力来源
💡 有意思的设计：本模块同时引入了 Client 和 Server 两个 Starter。这意味着依赖了 ruoyi-common-mcp 的业务模块，既可以作为 Client 调用别人，也可以作为 Server 被别人调用。这是 RuoYi「集大成」的典型设计思路。

五、自动配置清单：入口只有一个类名

文件路径：src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports

org.dromara.common.mcp.config.McpAutoConfiguration

就一行，告诉 Spring Boot「启动时加载这个自动配置类」。Spring Boot 会扫描所有 jar 包中这个固定路径的文件，读取类名列表并逐类加载——这就是「依赖即生效」的机制。第 2 课会打开 McpAutoConfiguration 看它怎么用 @ConditionalOnBean 做条件装配。

Q1. MCP 协议的三大原语是什么？本模块支持哪两个？

A. REST / RPC / GraphQL — 本模块全部支持 B. Tools / Resources / Prompts — 本模块支持 Tools 和 Resources C. GET / POST / PUT — 本模块支持 GET 和 POST D. Client / Server / Proxy — 本模块支持 Client 和 Server
看答案B。MCP 的三大原语是 Tools（工具调用）、Resources（资源读取）、Prompts（提示模板）。本模块用 callTool() 支持 Tools，用 readResource() 支持 Resources。Prompts 暂未封装。

Q2. ruoyi-common-mcp 和 Spring AI MCP 是什么关系？

A. ruoyi-common-mcp 自己实现了 MCP 协议的所有传输层逻辑 B. ruoyi-common-mcp 是一层便利封装，协议实现和连接管理由 Spring AI MCP 完成 C. 两者互不依赖，各自独立 D. Spring AI MCP 依赖 ruoyi-common-mcp 才能使用
看答案B。Spring AI MCP（2.0.0）负责协议的完整 Java 实现（JSON-RPC、传输层、Lifecycle），本模块只是对 McpSyncClient 做了一层「用起来更顺手」的封装：统一返回格式、封装遍历逻辑、提供按 Server 名称筛选。

Q3. 如果一个业务模块依赖了 ruoyi-common-mcp，但没有在 yml 里配置任何 MCP Server 连接，会发生什么？

A. 启动报错，因为找不到 MCP Server B. 正常启动，但 McpClientTemplate 不会创建（因为 @ConditionalOnBean 不满足） C. 创建一个空的 McpClientTemplate，里面没有 Client D. Spring Boot 会自动连接一个默认的 MCP Server
看答案B。McpAutoConfiguration 用 @ConditionalOnBean(McpSyncClient.class) 做条件装配：没有配置 MCP 连接 → Spring AI 不会创建 McpSyncClient Bean → 条件不满足 → McpClientTemplate 不创建。这就是「零配置不报错」的设计。
