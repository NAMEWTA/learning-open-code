第 3 课 · 实战 MCP 工具调用与资源读取 综合实战

🎯 本节目标（对应使命 Success #5~#8）：学完你能正确使用 McpClientTemplate 调用 MCP Server 的工具和资源，区分并解析 McpToolCallResult 和 McpResourceReadResult，并能通过完整场景理解「配置→连接→调用→解析」的全链路。

一、两个 Result 模型：结构对比

先理解返回模型的字段含义，才能正确解析结果。

1.1 McpToolCallResult — 工具调用结果

public record McpToolCallResult(                   // Java Record：不可变 + 自动生成 getter/equals/hashCode
    String serverName,                              // 来源 Server 名称（如 "weather-server"）
    boolean error,                                  // 调用是否出错（来自 CallToolResult.isError()）
    List<McpSchema.Content> content,                // 文本内容列表（每段文本是一个 Content 对象）
    Object structuredContent                        // 结构化数据（如 JSON 对象，不是文本）
) {
    public static McpToolCallResult of(
            String serverName,
            McpSchema.CallToolResult result) {      // ← 静态工厂：SDK 类型 → 项目类型
        return new McpToolCallResult(
            serverName,
            Boolean.TRUE.equals(result.isError()),  // ← 注意：防御性处理 isError 可能为 null
            result.content(),                        // 从 SDK 的 CallToolResult 提取
            result.structuredContent()
        );
    }
}
字段来源（SDK 的 CallToolResult）用途
serverName由 McpClientTemplate 调用时传入标识这个结果是哪个 Server 返回的（多 Server 场景关键）
errorresult.isError()让调用方先判断是否出错再解析内容。注意用 Boolean.TRUE.equals() 防御 null 值
contentresult.content()List 类型，包含 McpSchema.Content——MCP 协议的文本内容片段。每个 Content 对象有 type 和 text 字段
structuredContentresult.structuredContent()类型为 Object，实际可能是 Map/List 等结构化 JSON 数据。与纯文本 content 不同，它是带结构的（可以直接转为 JSON）

1.2 McpResourceReadResult — 资源读取结果

public record McpResourceReadResult(
    String serverName,                               // 来源 Server 名称
    List<McpSchema.ResourceContents> contents        // 资源内容列表（文本或二进制）
) {
    public static McpResourceReadResult of(
            String serverName,
            McpSchema.ReadResourceResult result) {
        return new McpResourceReadResult(
            serverName,
            result.contents()                        // SDK 的 ReadResourceResult 的 contents 字段
        );
    }
}
字段来源（SDK 的 ReadResourceResult）用途
serverName由 McpClientTemplate 调用时传入标识来源 Server
contentsresult.contents()List 类型，包含 McpSchema.ResourceContents——可能是 TextResourceContents（含 uri + mimeType + text）或 BlobResourceContents（含 uri + mimeType + blob 二进制）

1.3 两个 Result 的差异速查

McpToolCallResultMcpResourceReadResult
字段数量4 个2 个（更简单）
content 类型List<McpSchema.Content>List<McpSchema.ResourceContents>
有无 error 标记✅ 有 boolean error❌ 无（读取失败通常抛异常）
有无结构化数据✅ 有 Object structuredContent❌ 无
工厂方法参数McpSchema.CallToolResultMcpSchema.ReadResourceResult
💡 content vs structuredContent：这是 MCP 协议的一个精细设计。content 是文本片段列表（通常是对话型内容），适合展示/流式输出；structuredContent 是单个结构化对象（JSON Map/List），适合程序化解析。两者不互斥——一个工具调用可能同时返回两者。

二、实战场景一：调用「天气预报」工具

场景：RuoYi 后端调用外部 MCP 天气 Server 查询天气

假设运维已在 application.yml 中配置了一个名为 weather-server 的 MCP Server 连接，它暴露了一个 get_forecast 工具，接收 city 参数，返回该城市的天气预报数据。

步骤 1：查看可用工具

// 在你的 Service 中注入 McpClientTemplate
@Autowired
private McpClientTemplate mcpClientTemplate;

// 列出所有 Server 的工具
Map<String, List<String>> tools = mcpClientTemplate.listTools();
// 输出：{weather-server=[get_forecast, get_alerts, get_radar]}
System.out.println(tools);

步骤 2：调用工具（单点模式）

// 调用指定 Server 的 get_forecast 工具，传入城市参数
Optional<McpToolCallResult> resultOpt = mcpClientTemplate.callTool(
    "weather-server",                                        // Server 名称
    "get_forecast",                                          // 工具名称
    Map.of("city", "北京", "days", 3)                        // 参数 Map
);

if (resultOpt.isEmpty()) {
    throw new RuntimeException("找不到 weather-server");     // Server 名字写错了
}

McpToolCallResult result = resultOpt.get();

步骤 3：解析结果

// 先判断是否出错
if (result.error()) {
    // 从 content 中提取错误信息
    String errorText = result.content().stream()
        .filter(c -> "text".equals(c.type()))                // 筛选文本类型的 Content
        .map(McpSchema.Content::text)
        .collect(Collectors.joining("\n"));
    throw new RuntimeException("天气查询失败: " + errorText);
}

// 方式 A：读取文本内容（适合展示给前端）
String textContent = result.content().stream()
    .filter(c -> "text".equals(c.type()))
    .map(c -> c.text())
    .collect(Collectors.joining("\n"));
// 返回："北京 2026-07-05 晴 25°C~32°C ..."

// 方式 B：读取结构化数据（适合程序化处理）
if (result.structuredContent() instanceof Map structured) {
    String city = (String) structured.get("city");           // "北京"
    List<Map> forecasts = (List<Map>) structured.get("forecasts");
    // 可以进一步处理后存入数据库或返回前端
}
💡 实战技巧：当调用多个 Server 的同名工具时，用广播版 callTool("get_forecast", args) 一次拿到所有 Server 的结果，然后根据 result.serverName() 筛选你需要的。

三、实战场景二：读取「数据库表结构」资源

场景：RuoYi 后端通过 MCP 数据库 Server 动态读取表结构

假设有一个名为 db-server 的 MCP Server，它将数据库 Schema 暴露为 MCP Resources（URI 格式如 schema://sys_user），客户端可以通过 readResource 读取指定表的 DDL 信息。

步骤 1：读取资源

// 读取指定 Server 上的表结构资源
Optional<McpResourceReadResult> resultOpt = mcpClientTemplate.readResource(
    "db-server",                                             // Server 名称
    "schema://sys_user"                                      // 资源 URI
);

if (resultOpt.isEmpty()) {
    throw new RuntimeException("找不到 db-server");
}

McpResourceReadResult result = resultOpt.get();

步骤 2：解析资源内容

// 遍历内容列表（可能有多个 ResourceContents）
for (McpSchema.ResourceContents rc : result.contents()) {
    // 判断是文本还是二进制
    if (rc instanceof McpSchema.TextResourceContents textRc) {
        // 文本资源：获取 URI、MIME 类型、文本内容
        System.out.println("资源 URI: " + textRc.uri());      // "schema://sys_user"
        System.out.println("MIME 类型: " + textRc.mimeType());// "application/sql"
        System.out.println("内容:\n" + textRc.text());        // CREATE TABLE sys_user ...
    } else if (rc instanceof McpSchema.BlobResourceContents blobRc) {
        // 二进制资源：获取 URI、MIME 类型、Base64 编码的 Blob
        System.out.println("二进制数据 (MIME: " + blobRc.mimeType() + ")");
        byte[] decoded = Base64.getDecoder().decode(blobRc.blob());
        // 进一步处理二进制数据 ...
    }
}

四、实战场景三：广播模式 — 同时查询多个数据源

场景：系统同时连接了两个 MCP Server（db-server 和 cache-server），都暴露了 health 工具

你想一次性检查所有 Server 的健康状态。

// 广播调用：向所有 Server 的 health 工具发送请求
Map<String, McpToolCallResult> allHealth = mcpClientTemplate.callTool(
    "health",                                                // 工具名称（所有 Server 都暴露）
    Map.of()                                                 // 无参数
);

// 遍历每个 Server 的返回
allHealth.forEach((serverName, result) -> {
    if (result.error()) {
        System.out.println("❌ " + serverName + " 健康检查失败");
    } else {
        String status = result.content().stream()
            .filter(c -> "text".equals(c.type()))
            .map(c -> c.text())
            .findFirst().orElse("unknown");
        System.out.println("✅ " + serverName + ": " + status);
    }
});

五、完整调用链路图

业务代码 McpClientTemplate Spring AI MCP MCP Server ──────── ──────────────── ────────────── ───────── mcpClientTemplate .callTool("weather-server", → findClient("weather-server") "get_forecast", │ Map.of("city","北京")) ▼ Optional<McpSyncClient> │ ▼ .map(client -> ) client.callTool( → new CallToolRequest → JSON-RPC 消息 new CallToolRequest( ("get_forecast", → (SSE/stdio) "get_forecast", {city:"北京"}) {city:"北京"})) │ │ │ │ ◄──────────────── CallToolResult ◄───────────┘ │ (isError=false, │ content=[...], │ structuredContent={...}) ▼ McpToolCallResult.of( "weather-server", result) │ 解析 result.content() ◄──────────────┘ 或 result.structuredContent()

关键节点标注：

  1. findClient：按 Server 名称查找 Client（名称来自 MCP 初始化握手中的 serverInfo.name）。
  2. CallToolRequest：封装工具名 + 参数 Map，是 MCP 协议的 Java 类型表示。
  3. JSON-RPC → SSE/stdio：Spring AI MCP 将 Java 调用转换为 MCP 协议的 JSON-RPC 消息，通过配置的传输层发送。
  4. McpToolCallResult.of：将 SDK 的原始 CallToolResult 包装为项目的结果 Record，附加 serverName。

六、检验一下

Q1. McpToolCallResult.error() 为什么用 Boolean.TRUE.equals(result.isError()) 而不是 result.isError()？

A. 为了性能优化 B. 因为 MCP 协议的 isError 字段可能为 null，直接调用 isError() 可能 NPE C. 这是代码风格偏好 D. 为了兼容旧版本 MCP 协议
看答案B。CallToolResult.isError() 返回的是 Boolean（包装类型，可 null），不是 boolean。用 Boolean.TRUE.equals() 可以安全处理 null 值：null 时返回 false 而非 NPE。这是一种常见的 Java 防御性编程模式。

Q2. McpResourceReadResult 的 contents 字段类型是 List<ResourceContents>。实际运行时，列表中的元素可能是什么类型？

A. 永远是 String B. TextResourceContents（文本）或 BlobResourceContents（二进制） C. 永远是 Map<String, Object> D. 永远是 byte[]
看答案B。ResourceContents 是 MCP 协议的接口，两个实现类：TextResourceContents（含 text 字段，用于文本资源）和 BlobResourceContents（含 blob 字段，用于二进制资源）。解析时需要用 instanceof 判断。

Q3. 你想调用 'search' 工具，但只知道它存在，不确定是哪个 Server 提供的。你会用什么方法？

A. callTool("unknown-server", "search", args) B. callTool("search", args) — 广播版，让所有 Server 都试试 C. 先用 listTools() 查看所有 Server 的工具列表，找到后再精确调用 D. 检查 application.yml 找出哪个 Server 有 search 工具
看答案C是推荐做法（B 也可以但浪费）。先用 listTools() 拿到 {serverA=[search, ...], serverB=[...]} 的 Map，找到提供 search 的 Server，然后用单点版 callTool("serverA", "search", args) 精确调用。
