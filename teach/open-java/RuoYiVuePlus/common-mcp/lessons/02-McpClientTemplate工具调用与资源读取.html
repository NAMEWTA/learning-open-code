第2课 · McpClientTemplate 工具调用与资源读取

🎯 本节目标：学完你能逐行读懂 McpClientTemplate 的 7 个方法、解释为什么业务代码不直接操作 List<McpSyncClient>、区分 McpToolCallResult 和 McpResourceReadResult 的结构差异、说出 Tool 与 Resource 两种 MCP 原语在 Java 调用层面的区别，并能在实际业务中正确调用 callTool() 和 readResource()。

上一课我们建立了 MCP 协议的全局认知，理解了 Spring AI MCP 和自动配置的机制。这一课打开 McpClientTemplate——这个模块的真正「肉」——逐方法拆解它如何把底层的 McpSyncClient 列表转化为业务代码可以舒适使用的 API。

1. McpClientTemplate 概览：为什么需要它

回顾一下：Spring AI MCP 在 Bootstrap 阶段为每个配置的 MCP Server 连接创建一个 McpSyncClient Bean。如果你的项目连接了 3 个 MCP Server，Spring 容器中就有 3 个 McpSyncClient 类型的 Bean。

如果业务代码直接使用这些 Bean，会出现以下痛点：

  • 遍历嘈杂：每次调用工具都要写 for (McpSyncClient client : clients) { client.callTool(...); }，代码冗余。
  • 路由困难：想调用"weather-server 上的 get_forecast 工具"这个特定目标时，需要手写按 serverName 筛选的逻辑。
  • 返回格式不一致：底层 McpSchema.CallToolResult 和 McpSchema.ReadResourceResult 不包含 serverName 字段——当你有多个 Server 返回结果时，无法区分哪个结果来自哪个 Server。
  • 类型转换繁琐：每次都要把 McpSchema.CallToolResult 的 content 字段、structuredContent 字段转成业务需要的格式。

McpClientTemplate 的出现就是解决这四个痛点。它用不到 132 行代码，提供了 7 个简洁方法：

方法签名返回类型模式说明
listTools() Map<String, List<String>> 广播 列出所有 Server 的所有工具
callTool(toolName, arguments) Map<String, McpToolCallResult> 广播 在所有 Server 上调用同名工具
callTool(serverName, toolName, arguments) Optional<McpToolCallResult> 指定 在指定 Server 上调用工具
readResource(uri) Map<String, McpResourceReadResult> 广播 从所有 Server 读取同名资源
readResource(serverName, uri) Optional<McpResourceReadResult> 指定 从指定 Server 读取资源
findClient(serverName) Optional<McpSyncClient> 查找 按名称查找特定 Client
getClients() List<McpSyncClient> 透传 获取全部 Client 列表（逃生舱）

你会发现一个清晰的模式：每个核心操作都有两个重载——「广播版」和「指定 Server 版」。广播版遍历所有 Client 并返回 Map，key 是 serverName；指定版通过 findClient 定位单个 Client 并返回 Optional。

2. listTools()：发现所有可用工具

方法源码：

public Map<String, List<String>> listTools() {
    Map<String, List<String>> result = new LinkedHashMap<>();
    for (McpSyncClient client : mcpSyncClients) {
        List<String> tools = client.listTools().tools().stream()
            .map(McpSchema.Tool::name)
            .toList();
        result.put(getServerName(client), tools);
    }
    return result;
}

执行流程：

  1. 遍历 mcpSyncClients（所有已连接的 MCP Server）。
  2. 对每个 Client，调用 client.listTools() 获取该 Server 的全部工具列表。
  3. 从 McpSchema.ListToolsResult 中提取 .tools()，再映射为工具名称的字符串列表。
  4. 以 serverName 为 key、工具名列表为 value，存入 LinkedHashMap（保持连接顺序）。

这个方法通常用于启动时或调试时查看「当前有哪些 MCP Server 可用、每个 Server 暴露了哪些工具」。实际例子：

// 启动时打印所有可用工具
Map<String, List<String>> allTools = template.listTools();
// 输出可能是：
// {weather-server=[get_forecast, get_alerts],
//  filesystem-server=[read_file, list_directory, search_files]}

3. callTool()：两种调用模式

这是本模块最核心的方法——一切 MCP 工具调用都通过它。两个重载覆盖了「广播调用」和「指定 Server 调用」两种场景。

3.1 指定 Server 调用（精确路由）

public Optional<McpToolCallResult> callTool(
        String serverName, String toolName, Map<String, Object> arguments) {
    return findClient(serverName)
        .map(client -> McpToolCallResult.of(serverName,
            client.callTool(new McpSchema.CallToolRequest(toolName, arguments))));
}

执行链路：

  1. findClient(serverName) → 从 mcpSyncClients 列表中按 serverName 查找目标 Client，返回 Optional<McpSyncClient>。找不到则返回 Optional.empty()。
  2. 如果找到 Client → client.callTool(new McpSchema.CallToolRequest(toolName, arguments)) → 调用底层 MCP Client 的工具调用方法。
  3. McpToolCallResult.of(serverName, callResult) → 将底层 CallToolResult 包装为带 serverName 标记的项目结果对象。
  4. 如果找不到 Client → 返回 Optional.empty()。

关键类：CallToolRequest

McpSchema.CallToolRequest 是 Spring AI MCP 定义的工具调用请求对象，封装了两个字段：

  • name()：工具名称（String），对应 Server 上注册的工具名。
  • arguments()：工具参数（Map<String, Object>），以键值对形式传递。

构造示例：

new McpSchema.CallToolRequest("get_forecast", Map.of("city", "Beijing", "days", 7));

3.2 广播调用（所有 Server）

public Map<String, McpToolCallResult> callTool(
        String toolName, Map<String, Object> arguments) {
    Map<String, McpToolCallResult> result = new LinkedHashMap<>();
    for (McpSyncClient client : mcpSyncClients) {
        McpSchema.CallToolResult callResult =
            client.callTool(new McpSchema.CallToolRequest(toolName, arguments));
        result.put(getServerName(client),
            McpToolCallResult.of(getServerName(client), callResult));
    }
    return result;
}

广播版本遍历所有 Client，在每个 Server 上调用同一个工具名称和参数，返回一个 Map，key 为 serverName。这种模式适用于：你有多台数据库 MCP Server，想同时查询所有数据库的某个表状态；或者有多个文件系统 Server，想同时搜索一个文件。

注意：广播版不处理个别 Server 调用失败的情况——某个 Server 抛异常会直接向上传播。如需容错，业务代码应在调用外层 try-catch，或自己逐个调用指定版。

3.3 完整调用示例

// 注入模板
@Autowired
private McpClientTemplate mcpClientTemplate;

// 场景一：调用 weather-server 的天气查询工具
Map<String, Object> args = Map.of("city", "Beijing", "days", 3);
Optional<McpToolCallResult> result = mcpClientTemplate.callTool(
    "weather-server", "get_forecast", args);

result.ifPresent(r -> {
    System.out.println("来源 Server：" + r.serverName());      // "weather-server"
    System.out.println("是否出错：" + r.error());
    System.out.println("文本内容：" + r.content());           // List<Content>
    System.out.println("结构化数据：" + r.structuredContent()); // Object
});

// 场景二：在所有 Server 上调用 search 工具
Map<String, McpToolCallResult> allResults = mcpClientTemplate.callTool(
    "search", Map.of("keyword", "MCP协议"));
// allResults.keySet() = ["weather-server", "filesystem-server", "database-server"]

4. McpToolCallResult Record：工具调用结果的标准化包装

完整源码（core/McpToolCallResult.java）：

public record McpToolCallResult(
    String serverName,
    boolean error,
    List<McpSchema.Content> content,
    Object structuredContent
) {
    public static McpToolCallResult of(String serverName, McpSchema.CallToolResult result) {
        return new McpToolCallResult(
            serverName,
            Boolean.TRUE.equals(result.isError()),
            result.content(),
            result.structuredContent()
        );
    }
}

字段解析：

字段类型来源含义
serverName String 由 Template 从 client.getServerInfo().name() 获取 核心附加值：标识结果来自哪个 MCP Server。底层 CallToolResult 没有这个字段
error boolean 对应 CallToolResult.isError() 工具执行是否出错。注意 Boolean.TRUE.equals() 的安全判空
content List<McpSchema.Content> 对应 CallToolResult.content() 工具返回的文本内容列表。每个 Content 是一个文本片段（类似 ChatMessage）
structuredContent Object 对应 CallToolResult.structuredContent() 工具返回的结构化数据（JSON Object），可以为 null

为什么要有 content 和 structuredContent 两个字段？

这是 MCP 协议的设计：Tool 的返回可以同时包含「人类可读的文本说明」(content) 和「程序可处理的结构化数据」(structuredContent)。例如天气查询工具的返回：

  • content 可能是："北京未来三天天气：晴天，22-30°C..."（给 LLM 或人看）
  • structuredContent 可能是：{"city":"Beijing","forecasts":[{"date":"...","temp":22},...]}（给程序解析）

与 CallToolResult 的对应关系：

Spring AI MCP (底层)                         ruoyi-common-mcp (本模块)
─────────────────────                        ─────────────────────────
McpSchema.CallToolResult                     McpToolCallResult
  ├── .isError()          ──映射──►  .error()
  ├── .content()          ──映射──►  .content()
  ├── .structuredContent()──映射──►  .structuredContent()
  └── (无 serverName)      ──新增──►  .serverName()  ← 这是本模块的附加值

静态工厂方法 of() 的设计：使用静态工厂而非直接调用构造函数，提供了一层转换隔离——如果将来 CallToolResult 的 API 变化，只需修改这个工厂方法，所有调用方不受影响。同时它负责了 Boolean.TRUE.equals(result.isError()) 的安全判空逻辑。

5. readResource()：从 MCP Server 读取数据

与 callTool 对称，readResource 也有广播版和指定 Server 版。

5.1 指定 Server 读取

public Optional<McpResourceReadResult> readResource(String serverName, String uri) {
    return findClient(serverName)
        .map(client -> McpResourceReadResult.of(serverName,
            client.readResource(new McpSchema.ReadResourceRequest(uri))));
}

与 callTool 的结构几乎一样——先 findClient，再调用底层方法，再包装结果。区别仅在于底层调的是 client.readResource() 而非 client.callTool()。

关键类：ReadResourceRequest

McpSchema.ReadResourceRequest 封装了资源 URI：

  • uri()：资源地址（String），由 MCP Server 定义。例如 "weather://cities/beijing"、"file:///documents/report.txt"。

注意：这个 URI 是 MCP 协议级的标识符，不一定对应文件系统路径。它是由 MCP Server 自定义的命名空间——Server 决定 "weather://cities/beijing" 映射到什么数据。

5.2 广播读取

public Map<String, McpResourceReadResult> readResource(String uri) {
    Map<String, McpResourceReadResult> result = new LinkedHashMap<>();
    for (McpSyncClient client : mcpSyncClients) {
        McpSchema.ReadResourceResult readResult =
            client.readResource(new McpSchema.ReadResourceRequest(uri));
        result.put(getServerName(client),
            McpResourceReadResult.of(getServerName(client), readResult));
    }
    return result;
}

在所有 Server 上读取同一个 URI 的资源。

5.3 完整调用示例

// 读取 filesystem-server 上 .gitignore 文件内容
Optional<McpResourceReadResult> resourceResult = mcpClientTemplate.readResource(
    "filesystem-server", "file:///.gitignore");

resourceResult.ifPresent(r -> {
    System.out.println("来源 Server：" + r.serverName());   // "filesystem-server"
    for (McpSchema.ResourceContents c : r.contents()) {
        if (c instanceof McpSchema.TextResourceContents text) {
            System.out.println("文本内容：" + text.text());
        }
    }
});

6. McpResourceReadResult Record：资源读取结果的标准化包装

完整源码（core/McpResourceReadResult.java）：

public record McpResourceReadResult(
    String serverName,
    List<McpSchema.ResourceContents> contents
) {
    public static McpResourceReadResult of(
            String serverName, McpSchema.ReadResourceResult result) {
        return new McpResourceReadResult(serverName, result.contents());
    }
}

字段解析：

字段类型来源含义
serverName String 由 Template 从 client.getServerInfo().name() 获取 标识资源内容来自哪个 MCP Server
contents List<McpSchema.ResourceContents> 对应 ReadResourceResult.contents() 资源内容列表。每个元素是 ResourceContents 的子类型

ResourceContents 的两种子类型：

类型字段适用场景
TextResourceContents uri()、mimeType()、text() 文本资源：文件内容、JSON 响应、配置文本
BlobResourceContents uri()、mimeType()、blob() 二进制资源：图片、音频、PDF 等

解析时需要 instanceof 判断：

for (McpSchema.ResourceContents rc : result.contents()) {
    if (rc instanceof McpSchema.TextResourceContents text) {
        System.out.println(text.text());       // 文本内容
    } else if (rc instanceof McpSchema.BlobResourceContents blob) {
        byte[] data = blob.blob();             // 二进制数据
    }
}

与 ReadResourceResult 的对应关系：

Spring AI MCP (底层)                          ruoyi-common-mcp (本模块)
─────────────────────                         ─────────────────────────
McpSchema.ReadResourceResult                  McpResourceReadResult
  ├── .contents()         ──映射──►  .contents()
  └── (无 serverName)      ──新增──►  .serverName()  ← 本模块的附加值

7. findClient()：按名称定位的底层方法

public Optional<McpSyncClient> findClient(String serverName) {
    return mcpSyncClients.stream()
        .filter(client -> serverName.equals(getServerName(client)))
        .findFirst();
}

这是 callTool(serverName, ...) 和 readResource(serverName, ...) 的底层依赖。它遍历 Client 列表，通过 client.getServerInfo().name() 匹配 serverName，返回第一个匹配的 Client。

serverName 的来源链路：

yml 配置的 key            →  serverInfo.name()  →  findClient 匹配  →  结果 Record.serverName
"weather-server"          →  "weather-server"   →  按此名称过滤    →  "weather-server"

注意 findFirst() ——如果有多个 Client 返回了相同的 serverName（理论上不应出现，但以实际为准），只取第一个。这是防御性编码。

8. getServerName() 私有方法：名称解析

private String getServerName(McpSyncClient client) {
    McpSchema.Implementation serverInfo = client.getServerInfo();
    if (serverInfo == null || serverInfo.name() == null) {
        return "unknown";
    }
    return serverInfo.name();
}

从 Client 的 ServerInfo 中提取名称。如果信息缺失（理论上不应发生，但防御性处理），返回 "unknown"。这确保了整个模板不会因为 Server 元信息缺失而 NPE。

9. Tool 与 Resource 的最终辨析

现在我们可以从「Java 调用方式」的角度彻底区分这两种 MCP 原语：

维度Tool（工具）Resource（资源）
请求对象 CallToolRequest(name, arguments) ReadResourceRequest(uri)
请求参数 工具名称 + 业务参数 Map 资源 URI 字符串
结果对象 CallToolResult → McpToolCallResult ReadResourceResult → McpResourceReadResult
结果内容 文本内容 (content) + 结构化数据 (structuredContent) + 错误标记 (isError) 资源内容列表 (contents)，每个是 Text 或 Blob
Java 类比 调用一个方法：server.getForecast("Beijing", 7) 读取一个文件：Files.readString(Path.of(uri))
语义 「请执行这个操作」 「请提供这个数据」
有副作用？ 可能有（发邮件、写数据库） 一般无（纯读取）
💡 设计洞察：为什么 Tool 和 Resource 是两个独立原语？
MCP 协议的设计者刻意把「执行操作」（Tool）和「读取数据」（Resource）分开，而不是像 REST 那样用 HTTP method（GET/POST/PUT/DELETE）区分。原因：给 LLM Agent 更清晰的指令。对 AI 来说，「调用一个函数并传入参数」和「读取一个 URI 的内容」是两种截然不同的认知模式——Tool 需要理解参数语义、构造参数、验证返回；Resource 只需要给出一个定位符。分开定义让 Agent 更容易选择正确的交互方式。

Q1. McpClientTemplate 提供了哪三种调用模式？（提示：广播、指定、查找）

看答案

广播模式（如 callTool(toolName, arguments)）：遍历所有 Client，每个都调；
指定模式（如 callTool(serverName, toolName, arguments)）：只调特定的一个 Client；
查找模式（findClient(serverName)）：按名称找到 Client 对象本身。

Q2. McpToolCallResult 比底层的 CallToolResult 多了什么字段？为什么要加？

看答案

多了 serverName 字段。底层 CallToolResult 不包含来源 Server 信息——当有多个 Server 返回结果时，无法区分哪个结果来自哪个 Server。serverName 提供了多 Server 场景下的结果溯源能力。

Q3. 说出 CallToolRequest 和 ReadResourceRequest 在参数结构上的根本区别。

看答案

CallToolRequest 需要：toolName（调用哪个工具）+ arguments（Map 参数）。类似调用一个有名字和参数的方法。
ReadResourceRequest 只需要：uri（资源地址字符串）。类似从文件路径读取内容，不需要额外的业务参数。

Q4. McpResourceReadResult.contents() 返回的是什么类型？里面的元素可能是哪两种子类型？

看答案

返回 List<McpSchema.ResourceContents>。每个元素可能是：
TextResourceContents（文本资源，通过 .text() 获取内容）
或 BlobResourceContents（二进制资源，通过 .blob() 获取字节数组）。
解析时需要 instanceof 判断。

Q5. 为什么业务代码应该用 McpClientTemplate 而不是直接注入 List<McpSyncClient>？说出至少两个理由。

看答案

理由一：统一遍历——不用每次手写 for 循环；理由二：按名称路由——通过 serverName 精确定位目标 Server，无需手写 filter；理由三：结果包装——自动附加 serverName 溯源信息；理由四：类型转换——将底层的 CallToolResult/ReadResourceResult 转为项目专用的 Record，统一返回格式。

Q6. 请写出调用 weather-server 的 get_current_weather 工具、传入城市参数 {"city": "Shanghai"} 的完整 Java 代码。

看答案 
Map<String, Object> args = Map.of("city", "Shanghai");
Optional<McpToolCallResult> result =
    mcpClientTemplate.callTool("weather-server", "get_current_weather", args);

result.ifPresent(r -> {
    if (!r.error()) {
        // 处理文本内容
        List<McpSchema.Content> texts = r.content();
        // 处理结构化数据
        Object data = r.structuredContent();
    }
});
