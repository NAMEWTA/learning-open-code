第1课 · MCP协议与Spring AI集成

🎯 本节目标：学完你能用一句话说清 MCP 协议是什么，画出 MCP 的 Client-Server 通信架构图，理解 Spring AI MCP 在 Java 生态中的角色，并说出 ruoyi-common-mcp 模块仅有的四个 Java 类各自解决什么问题。核心关注点：协议概念与 Java 代码之间如何桥接。

1. 问题背景：AI应用为什么需要一个「万能接口」？

在 MCP 出现之前，如果一个 AI 应用（比如大语言模型 Agent）想要查天气、读文件、调数据库、发送邮件——它需要为每一种外部服务写一套定制的集成代码。天气预报 API 的认证方式是一种、文件系统的读取接口是另一种、数据库的协议又是另一种。每增加一个外部能力，开发团队就要读懂一套新 SDK，写一套新适配层。

这跟 2010 年代移动设备的充电接口混乱非常像：Micro-USB、Lightning、30-pin 接口各自为政。直到 USB-C 统一了充电和数据传输，局面才改变。

MCP（Model Context Protocol）就是 AI 世界的「USB-C 接口」。它定义了一套开放、标准的通信协议，让所有 AI 应用（作为 Client）用同一种语言与任何外部服务（作为 Server）交互。服务提供方只需要实现 MCP Server 接口，不必为每个 AI 平台写适配；AI 应用只需要实现 MCP Client 接口，不必为每种外部能力学一套新 SDK。

2. MCP协议：Anthropic 提案到行业标准

MCP 由 Anthropic 于 2024 年底提出并开源，最初是为 Claude 等大模型设计的上下文通信协议。由于协议设计简洁、原语清晰、传输层灵活，迅速被业界接纳，演化为 AI 行业标准。

MCP 的 Client-Server 通信架构：

┌──────────────────────┐                              ┌──────────────────────┐
│                      │     MCP 协议 (JSON-RPC)        │                      │
│   MCP Client (Host)  │ ◄──────────────────────────► │   MCP Server (服务)   │
│                      │                              │                      │
│   例如：RuoYi 后端     │   ● Tools: 调用函数           │   例如：天气查询 Server │
│   例如：Claude Desktop │   ● Resources: 读取数据       │   例如：文件系统 Server │
│   例如：VS Code 插件   │   ● Prompts: 获取提示模板      │   例如：数据库 Server  │
│                      │                              │   例如：邮件 Server    │
└──────────────────────┘                              └──────────────────────┘

通信模型要点：

  • 一对一连接：一个 McpSyncClient 实例代表与一个 MCP Server 的一条连接。
  • JSON-RPC 2.0 消息：底层走 JSON-RPC 协议，请求-响应模式，每个消息都是结构化 JSON。
  • 传输层无关：底层传输可以用 SSE（Server-Sent Events）、stdio（标准输入输出）、Streamable HTTP。Java 开发者不需要管传输细节——Spring AI MCP 已封装好。
  • Lifecycle 管理：连接有初始化（initialize）、已初始化（initialized）、关闭等生命周期阶段，由框架自动维护。

3. 三大原语（Primitives）：Tools、Resources、Prompts

MCP 协议定义了三种核心交互原语（primitive）。原语是协议层的概念，不是 HTTP API 接口，它们抽象地定义了 Client 和 Server 之间「能做什么」：

原语语义Java 中类比本模块支持
Tools（工具） Client 调用 Server 暴露的函数，传入参数，获取计算结果。语义类似远程过程调用。 类似 RPC / @PostMapping + 请求体 ✅ callTool()
Resources（资源） Client 读取 Server 暴露的数据——文件内容、数据库记录、配置项等。通过 URI 标识要读什么。 类似 REST GET / 文件读取 ✅ readResource()
Prompts（提示模板） Server 提供可复用的提示/指令模板，供 LLM 使用。模板可带参数。 类似预置 Prompt 库 ❌ 本模块未封装

Tools 与 Resources 的关键区别：

Tool 像是 RPC 调用——你告诉 Server "帮我算一下天气"，传入城市名和日期作为参数，Server 返回计算结果。有输入参数，有输出结果。本质是「让 Server 执行一个操作」。

Resource 像是 REST GET——你告诉 Server "把 /documents/report.txt 的内容给我"，通过 URI 指定要读什么，Server 返回该资源的内容。无业务参数（只有 URI），返回数据。本质是「让 Server 提供一段数据」。

💡 设计洞察：为什么叫「原语」而不是「API」？
MCP 的设计刻意使用 primitive 一词，强调这些交互模式是协议级别的抽象，与具体的传输协议（HTTP/SSE/stdio）解耦。一个 callTool 操作在 SSE 传输下是 JSON-RPC 请求，在 stdio 传输下也是 JSON-RPC 请求——上层 Java 代码调用方式不变。这就是协议层抽象的威力。

4. Spring AI MCP：Java 世界的 MCP 桥接

MCP 是协议规范，规范本身不包含 Java 代码。要在 Spring Boot 项目里使用 MCP，需要一套 Java 实现来桥接协议规范与 Spring 生态。这就是 Spring AI MCP 的职责。

Spring AI MCP（本项目使用 2.0.0 版本）是 Spring 生态官方提供的 MCP 协议 Java 实现。它提供了两个独立 Starter：

StarterGAV 坐标功能角色
MCP Client Starter org.springframework.ai:spring-ai-starter-mcp-client 让你的应用作为 MCP Client，连接外部 MCP Server 消费者 — 调用别人的工具和资源
MCP Server WebMVC Starter org.springframework.ai:spring-ai-starter-mcp-server-webmvc 让你的应用作为 MCP Server，对外暴露工具和资源 提供者 — 让别人调用你的工具和资源

Spring AI MCP 为你做了什么（这些都是框架层面的，本模块不重复实现）：

  1. Spring Boot 自动配置：读取 spring.ai.mcp.client.connections 下的 yml 配置，自动创建并初始化 McpSyncClient Bean。你只需要写配置，不需要写任何连接代码。
  2. 传输层封装：SSE 长连接管理、JSON-RPC 消息编解码、心跳维持——全被封装在 Spring Boot Starter 的自动配置里。Java 开发者只看到方法调用。
  3. Lifecycle 管理：initialize 握手、connection 状态跟踪、graceful shutdown——由框架自动处理。
  4. 类型安全的 Schema：McpSchema.Tool、McpSchema.CallToolRequest、McpSchema.CallToolResult、McpSchema.ReadResourceRequest、McpSchema.ReadResourceResult 等，全部是类型安全的 Java Record/Class。
💡 关键定位：
ruoyi-common-mcp 不实现 MCP 协议。协议实现（JSON-RPC 消息、传输层、连接生命周期）由 Spring AI MCP 完成。
ruoyi-common-mcp 只是对 Spring AI 提供的 McpSyncClient 做了一层「项目级便利封装」：统一返回格式、封装遍历逻辑、支持按 Server 名称筛选、广播调用。

5. ruoyi-common-mcp 模块全景：4 个类

这是本模块的全部 Java 文件（不含 pom.xml 和 AutoConfiguration.imports）：

ruoyi-common-mcp/src/main/java/org/dromara/common/mcp/
├── config/
│   └── McpAutoConfiguration.java      ← 【装配】自动配置
└── core/
    ├── McpClientTemplate.java         ← 【封装】7 个便捷方法
    ├── McpToolCallResult.java         ← 【模型】工具调用结果 Record
    └── McpResourceReadResult.java     ← 【模型】资源读取结果 Record

四个类的职责一目了然：

类数量解决什么问题关键设计
McpAutoConfiguration 30 行 决定「什么时候创建 McpClientTemplate Bean」 条件装配 + 依赖注入 List<McpSyncClient>

这是典型的「薄封装」设计——不增加概念，不改变底层 API，只是在 Spring AI MCP 的基础上加了一层「用起来更顺手」的包装。四个文件加起来不到 230 行代码。

6. McpAutoConfiguration：条件装配的精妙设计

打开 config/McpAutoConfiguration.java（完整源码）：

package org.dromara.common.mcp.config;

import io.modelcontextprotocol.client.McpSyncClient;
import org.dromara.common.mcp.core.McpClientTemplate;
import org.springframework.boot.autoconfigure.AutoConfiguration;
import org.springframework.boot.autoconfigure.condition.ConditionalOnBean;
import org.springframework.boot.autoconfigure.condition.ConditionalOnMissingBean;
import org.springframework.context.annotation.Bean;
import java.util.List;

@AutoConfiguration
public class McpAutoConfiguration {

    @Bean
    @ConditionalOnBean(McpSyncClient.class)
    @ConditionalOnMissingBean
    public McpClientTemplate mcpClientTemplate(List<McpSyncClient> mcpSyncClients) {
        return new McpClientTemplate(mcpSyncClients);
    }
}

逐行拆解这个自动配置的每个注解：

6.1 @AutoConfiguration

这是 Spring Boot 的自动配置注解（非 @Configuration）。与普通 @Configuration 的区别在于：Spring Boot 的自动配置加载机制会在用户自定义 Bean 注册之前执行自动配置，确保用户代码可以覆盖自动配置的行为。

配合 src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件（内容只有一行 org.dromara.common.mcp.config.McpAutoConfiguration），Spring Boot 启动时自动发现并加载这个配置类。不需要任何 @EnableXxx 注解——依赖即生效。

6.2 @ConditionalOnBean(McpSyncClient.class)

这是关键的条件守卫。它的含义是：「只有当 Spring 容器中已经存在至少一个 McpSyncClient 类型的 Bean 时，才创建下面的 McpClientTemplate Bean。」

谁创建 McpSyncClient Bean？是 Spring AI MCP Bootstrap——它读取 spring.ai.mcp.client.connections 下的 yml 配置，为每个配置的连接创建一个 McpSyncClient 实例并注册为 Spring Bean。

这意味着：

  • 如果项目中没有配置任何 MCP Server 连接 → Spring AI 不会创建 McpSyncClient Bean → 条件不满足 → McpClientTemplate 不创建 → 启动正常，不报错。
  • 如果项目中配置了 MCP Server 连接 → Spring AI 创建 McpSyncClient Bean(s) → 条件满足 → McpClientTemplate 自动创建 → 依赖即用。

这就是「零配置不报错」的优雅设计——依赖了 ruoyi-common-mcp 但还没决定连哪个 MCP Server？没关系，系统正常运行，等你在 yml 里加上配置后再自动激活。

6.3 @ConditionalOnMissingBean

含义：「如果容器中还没有同类型的 Bean，才创建我这个。」这是业务覆盖机制：如果业务模块定义了自己的 McpClientTemplate Bean（比如添加了额外的日志、缓存、权限检查），自动配置就会退让，使用业务自定义版本。

6.4 参数注入：List<McpSyncClient>

Spring 容器会自动收集所有 McpSyncClient 类型的 Bean，注入为一个 List。这意味着：如果你在 yml 里配置了 3 个 MCP Server 连接，这个方法会收到一个包含 3 个 McpSyncClient 实例的 List。然后全部传给 McpClientTemplate 的构造函数。

💡 设计洞察：
这三个注解的组合创造了一个完美的「懒激活」模式：
@AutoConfiguration → 声明为自动配置
@ConditionalOnBean(McpSyncClient.class) → 有连接才激活
@ConditionalOnMissingBean → 允许业务覆盖
这就是 Spring Boot 自动配置的最佳实践——用最少的代码实现最大的灵活性。

7. McpSyncClient：连接一个 MCP Server 的句柄

McpSyncClient 是 Spring AI MCP 提供的同步客户端接口（全路径：io.modelcontextprotocol.client.McpSyncClient）。一个 McpSyncClient 实例代表与一个 MCP Server 的已初始化连接。

核心方法（本模块用到的）：

方法返回类型说明
listTools() McpSchema.ListToolsResult 获取该 Server 暴露的所有工具列表。结果中 .tools() 返回 List<McpSchema.Tool>
callTool(CallToolRequest) McpSchema.CallToolResult 调用该 Server 上的一个工具。请求包含 toolName 和 arguments
readResource(ReadResourceRequest) McpSchema.ReadResourceResult 读取该 Server 上的一个资源。请求包含资源 URI
getServerInfo() McpSchema.Implementation 获取 Server 的元信息（名称、版本）。本模块用于解析 serverName

重要认知：McpSyncClient 不是本模块创建的。它由 Spring AI MCP 的 Bootstrap 阶段创建——在 yml 中配置好 MCP Server 连接信息后，Spring AI 自动创建对应的 McpSyncClient Bean。本模块只是收集并封装这些现成的 Bean。

8. POM 依赖：为何同时引入 Client 和 Server？

打开 pom.xml，核心依赖只有三个：

依赖作用是否必须
ruoyi-common-core RuoYi 核心模块（工具类、常量、异常等基础设施） 是 — 项目内公共依赖
spring-ai-starter-mcp-server-webmvc MCP Server 端支持：让当前应用也可以对外暴露 MCP 工具和资源 本模块仅作 Client 使用时非必须，但统一引入以便扩展
spring-ai-starter-mcp-client MCP Client 端支持：让当前应用可以连接外部 MCP Server 是 — 本模块核心能力来源

同时引入两个 Starter 意味着：依赖了 ruoyi-common-mcp 的业务模块，既可以作为 Client 连接外部 MCP Server（读天气、查数据库），也可以作为 Server 对外暴露自己的工具（让别人通过 MCP 调用你的业务能力）。这是 RuoYi「集大成」的典型设计。

9. YML 配置：如何声明 MCP Server 连接

虽然本模块不直接读取 yml 配置（那是 Spring AI MCP Bootstrap 的职责），但理解配置结构对于排查连接问题至关重要。

所有 MCP Client 连接都在 spring.ai.mcp.client.connections 下声明，每个连接是一个 key-value 配置块：

spring:
  ai:
    mcp:
      client:
        connections:
          weather-server:              # 连接名称（将作为 serverName）
            type: SYNC                 # 同步客户端
            transport: http            # 传输方式：sse / http / stdio
            url: http://localhost:8081/mcp  # MCP Server 地址
          filesystem-server:           # 第二个连接
            type: SYNC
            transport: http
            url: http://localhost:8082/mcp

Spring AI MCP Bootstrap 读取这段配置后：

  1. 为 weather-server 创建一个 McpSyncClient Bean
  2. 为 filesystem-server 创建另一个 McpSyncClient Bean
  3. 两个 Bean 都被注入到 McpAutoConfiguration.mcpClientTemplate(List<McpSyncClient>) 的 List 参数中
  4. ConditionalOnBean 条件满足，McpClientTemplate Bean 被创建

没有这段配置 → 没有 McpSyncClient Bean → McpClientTemplate 不创建 → 项目正常运行（不报错）。

💡 设计洞察：
配置键名 weather-server 就是后续代码中的 serverName。这个名字不在代码里硬编码，而是从 yml 配置流入 → 通过 McpSyncClient.getServerInfo().name() 读取 → 最终出现在 McpToolCallResult.serverName() 和 McpResourceReadResult.serverName() 中。整个链路没有魔法，全部是配置驱动的。

10. 检验一下（提取练习）

不要回看上文，凭记忆作答。

Q1. MCP 协议的三大原语是什么？本模块支持哪两个？

看答案

三大原语：Tools（工具调用）、Resources（资源读取）、Prompts（提示模板）。本模块用 callTool() 支持 Tools，用 readResource() 支持 Resources。Prompts 暂未封装。

Q2. ruoyi-common-mcp 和 Spring AI MCP 是什么关系？

看答案

Spring AI MCP 是 MCP 协议的完整 Java 实现（JSON-RPC、传输层、Lifecycle、Bootstrap）。ruoyi-common-mcp 是对 McpSyncClient 的项目级便利封装——统一返回格式、封装遍历逻辑、支持按 Server 名称筛选。本模块不实现协议层。

Q3. 如果一个业务模块依赖了 ruoyi-common-mcp，但没有在 yml 中配置任何 spring.ai.mcp.client.connections，会发生什么？

看答案

正常启动，不报错。因为 McpAutoConfiguration 使用 @ConditionalOnBean(McpSyncClient.class) 做条件装配——没有配置连接 → Spring AI 不会创建 McpSyncClient Bean → 条件不满足 → McpClientTemplate 不创建。这就是「零配置不报错」的设计。

Q4. @ConditionalOnMissingBean 在 McpAutoConfiguration 中起什么作用？

看答案

它允许业务模块覆盖默认的 McpClientTemplate Bean。如果业务代码自己定义了一个 McpClientTemplate（比如加了日志、缓存、权限检查），自动配置会自动退让，使用业务自定义版本。

Q5. 一个 McpSyncClient 实例代表什么？

看答案

代表与一个 MCP Server 的已初始化连接。如果你连接了 3 个 MCP Server（weather-server、filesystem-server、database-server），Spring AI 会创建 3 个 McpSyncClient Bean，每个对应一个连接。
