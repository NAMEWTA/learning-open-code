06 Handler 大脑下 — buildDataFilter 详解

本节目标：你能逐行解释 buildDataFilter 的每一个步骤，包括角色遍历、SpEL 渲染、NullSafe 兜底、ignore 防递归，理解为什么每个设计选择存在。

buildDataFilter 全貌

PlusDataPermissionHandler.java:107-179。这个方法按六个阶段执行：

阶段1: 确定 joinStr → 阶段2: 构建 SpEL 上下文 → 阶段3: 筛选参与角色
→ 阶段4: 注册列名变量 → 阶段5: 逐角色遍历渲染模板 → 阶段6: 组装最终 SQL

阶段1：确定拼接符 joinStr

// Handler:109-112
String joinStr = isSelect ? " OR " : " AND ";       // 默认：SELECT→OR, UPDATE/DELETE→AND
if (StringUtils.isNotBlank(dataPermission.joinStr())) {
    joinStr = " " + dataPermission.joinStr() + " ";  // 注解显式指定则覆盖
}

假设用户有两个角色（部门经理 dataScope=4 + 普通用户 dataScope=5），SELECT 场景下最终条件为：

dept_id IN (103,104,105) OR create_by = 1

即"能看到部门及以下的数据，或者自己创建的"。OR 逻辑意味着权限是累加的。

阶段2：构建 SpEL 上下文

// Handler:113-117
Object defaultValue = "-1";                           // ① null 的兜底值
NullSafeStandardEvaluationContext context =           // ② 自定义上下文（变量为null返回"-1"）
    new NullSafeStandardEvaluationContext(defaultValue);
context.addPropertyAccessor(new NullSafePropertyAccessor(...));
context.setBeanResolver(beanResolver);                // ③ 让 #{@sdss.xxx} 可以调Spring Bean
DataPermissionHelper.getContext().forEach(context::setVariable); // ④ 注入扩展变量

四个关键设计：

  • defaultValue = "-1"：当 SpEL 变量取不到（用户没部门、没角色）时，返回 "-1"。这样 IN (-1) 或 = -1 不会导致数据库语法错误，也不会匹配任何真实数据。
  • NullSafeStandardEvaluationContext（Handler:335-358）：覆写了 lookupVariable()，变量为 null 时返回 "-1" 而非 null。
  • NullSafePropertyAccessor（Handler:365-447）：装饰原 PropertyAccessor，对象属性读取为 null 时返回 "-1"。
  • beanResolver：让 @sdss 这样的 bean 引用能被 SpEL 解析。
为什么用 "-1"？如果 IN () 空参数或 = null，数据库会直接报错。用 "-1" 让它生成 IN (-1)，不报错但也查不到数据——安全失败而非崩溃。

阶段3：筛选参与数据权限的角色

// Handler:119-126
DataPermissionAccess access = currentAccess();        // 当前接口的权限约束
List<RoleDTO> scopeRoles = scopeRoles(user, access);   // 筛选角色
if (CollUtil.isEmpty(scopeRoles)) {
    if (access.constrained()) {
        return " 1 = 0 ";    // 接口有约束但用户无匹配角色 → 查不到数据
    }
    return StringUtils.EMPTY; // 无角色约束 → 不加过滤
}

scopeRoles()（Handler:219-250）的逻辑：

  • 接口没有权限注解约束 → 使用用户的全部角色
  • 接口有 @SaCheckPermission("system:user:list") → 从 user.dataScopeRoleMap 中查找真正拥有该权限码的角色子集

这项"接口约束"是新版加入的精细控制：用户可能有多个角色，但当前接口只允许其中某个角色的数据权限生效。防止角色 A（全部数据）的宽松范围被带进角色 B 才能进的接口。

阶段4：注册列名变量

// Handler:128-138
for (DataColumn dataColumn : dataPermission.value()) {
    // key 和 value 数组长度必须一致
    if (dataColumn.key().length != dataColumn.value().length) {
        throw new ServiceException("角色数据范围异常 => key与value长度不匹配");
    }
    for (int i = 0; i < dataColumn.key().length; i++) {
        context.setVariable(dataColumn.key()[i], dataColumn.value()[i]);
        // 即: #deptName = "dept_id", #userName = "create_by"
    }
    keys.addAll(Arrays.stream(dataColumn.key()).map(key -> "#" + key).toList());
}

把注解的 key→value 映射注入 SpEL 上下文。此刻 #deptName 在 SpEL 里的值就是字符串 "dept_id"。

阶段5：逐角色遍历，渲染模板

这是最核心的循环，Handler:140-173：

for (RoleDTO role : scopeRoles) {
    context.setVariable("roleId", role.getRoleId());         // 注入 #roleId
    DataScopeType type = DataScopeType.findCode(role.getDataScope()); // 查枚举
    if (type == DataScopeType.ALL) {
        return StringUtils.EMPTY;  // 任一角色是ALL → 整个不过滤！
    }
    boolean isSuccess = false;
    for (DataColumn dataColumn : dataPermission.value()) {    // 遍历每个 DataColumn
        // 模板不含这个 key → 跳过这个 DataColumn
        if (!StringUtils.containsAny(type.getSqlTemplate(), keys.toArray(String[]::new))) continue;
        if (!StringUtils.containsAny(type.getSqlTemplate(), dataColumn.key())) continue;
        // ★ 关键：ignore 包裹，防止 SpEL 里 @sdss 再查 DB 时触发数据权限死循环
        String sql = DataPermissionHelper.ignore(() ->
            parser.parseExpression(type.getSqlTemplate(), parserContext)
                  .getValue(context, String.class));
        conditions.add(joinStr + sql);
        isSuccess = true;
    }
    // 模板没匹配上任何列 → 用 elseSql 兜底（通常是 " 1 = 0 "）
    if (!isSuccess && StringUtils.isNotBlank(type.getElseSql())) {
        conditions.add(joinStr + type.getElseSql());
    }
}

为什么需要 DataPermissionHelper.ignore？

看 CUSTOM 的模板：#{#deptName} IN ( #{@sdss.getRoleCustom( #roleId )} )

@sdss.getRoleCustom(roleId) → roleDeptMapper.lambda().select(...) → 这个 Mapper 方法可能也标了 @DataPermission！如果不 ignore，就进入 无限递归：拼 SQL 触发 SQL 解析 → 再进 buildDataFilter → 再拼 SQL → …

DataPermissionHelper.ignore(Supplier)（Helper:145-152）调 DataPermissionIgnoreContext.enable()，用反射操作 MyBatis-Plus 的 IGNORE_STRATEGY_LOCAL ThreadLocal，把 dataPermission=true，从而让 InterceptorIgnoreHelper.willIgnoreDataPermission() 返回 true，后续 SQL 不再触发数据权限。

阶段6：组装最终 SQL

// Handler:175-178
if (CollUtil.isNotEmpty(conditions)) {
    String sql = StreamUtils.join(conditions, Function.identity(), "");
    return sql.substring(joinStr.length());   // 去掉开头的多余 joinStr
}

conditions 里存的格式是 " OR dept_id IN (103,104,105)"、" OR create_by = 1"。join 后变成 " OR dept_id IN (103,104,105) OR create_by = 1"，再 substring 去掉开头的 " OR "，得到干净的过滤片段。

完整流程总结

buildDataFilter(注解, 用户, isSelect)
 │
 ├─ joinStr = isSelect ? "OR" : "AND"
 ├─ 构建 SpEL 上下文（NullSafe + beanResolver + 自定义变量）
 ├─ scopeRoles: 接口权限约束 → 筛选角色
 │   └─ 无角色 + 有约束 → "1=0"
 │   └─ 无角色 + 无约束 → 空
 │
 ├─ 注册 key→value 列名映射
 │
 ├─ 逐角色循环:
 │   ├─ ALL? → 直接返回空！（短路）
 │   ├─ 找 DataScopeType 枚举
 │   ├─ 模板含此注解的 key? → ignore 包裹 → SpEL 渲染 → 加入 conditions
 │   └─ 模板不含? → elseSql 兜底
 │
 └─ 组装: conditions.join → 去开头 joinStr → 返回
