04 运行时入口 — 装配、切面与拦截器

本节目标：你能画出"注解 → AOP → ThreadLocal → 拦截器"的数据流图，理解两个 Spring Bean 如何协作完成注解传递。

双通道协作模型

数据权限在运行时靠两个独立的 Spring Bean 协作，缺一不可：

1. AOP 切面（DataPermissionPointcutAdvisor）
   职责：在 Mapper 方法被调用前，把 @DataPermission 注解塞进 ThreadLocal
   触发时机：Spring AOP 代理拦截到 Mapper 方法调用

2. MyBatis 拦截器（PlusDataPermissionInterceptor）
   职责：在 SQL 发给 JDBC 前，从 ThreadLocal 取注解，改写 SQL
   触发时机：MyBatis Executor → StatementHandler → beforeQuery / beforePrepare

两者都在 MybatisPlusConfig.java 中装配：

// MybatisPlusConfig.java:46-55 — 拦截器链装配
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor() {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
    interceptor.addInnerInterceptor(dataPermissionInterceptor());  // ① 数据权限（第一个！）
    interceptor.addInnerInterceptor(paginationInnerInterceptor()); // ② 分页
    interceptor.addInnerInterceptor(optimisticLockerInnerInterceptor()); // ③ 乐观锁
    return interceptor;
}

// MybatisPlusConfig.java:67-71 — AOP 切面装配
@Bean
@Role(BeanDefinition.ROLE_INFRASTRUCTURE)
public DataPermissionPointcutAdvisor dataPermissionPointcutAdvisor() {
    return new DataPermissionPointcutAdvisor();
}

数据权限必须是第一个 InnerInterceptor——如果分页插件先拦截，SQL 已经被改写成分页查询了，数据权限的 WHERE 拼装就可能拼到错误的位置。

通道一：AOP 切面 — 注解入 ThreadLocal

切面由三个类组成：Advisor 组装 Advice+Pointcut → Pointcut 匹配 → Advice 执行。

Pointcut：谁该被拦截？

DataPermissionPointcut.java:24-34 判断逻辑：

public boolean matches(Method method, Class<?> targetClass) {
    // 1. 方法上有 @DataPermission 注解 → 命中
    if (method.isAnnotationPresent(DataPermission.class)) {
        return true;
    }
    // 2. 目标类（处理 JDK 代理）→ 接口上有 @DataPermission → 命中
    Class<?> targetClassRef = resolveTargetClass(targetClass);
    return targetClassRef.isAnnotationPresent(DataPermission.class);
}

关键点：MyBatis 的 Mapper 是 JDK 动态代理对象。Spring AOP 拿到的 targetClass 是 $ProxyXX，isAnnotationPresent 查不到注解。所以 resolveTargetClass()（第 42-55 行）通过 targetClass.getInterfaces() 遍历代理实现的接口，在真实 Mapper 接口上找注解。

Advice：把注解塞进 ThreadLocal

DataPermissionAdvice.java:26-37：

public Object invoke(MethodInvocation invocation) throws Throwable {
    Object target = invocation.getThis();
    Method method = invocation.getMethod();
    // ① 把注解放进 ThreadLocal
    DataPermissionHelper.setPermission(getDataPermissionAnnotation(target, method));
    try {
        // ② 执行真正的 Mapper 方法（这时 MyBatis 拦截器才会触发）
        return invocation.proceed();
    } finally {
        // ③ 不管成功失败都清理 ThreadLocal
        DataPermissionHelper.removePermission();
    }
}

getDataPermissionAnnotation()（第 47-60 行）的查找优先级：方法注解 > 类注解。如果是 JDK 代理（Proxy.isProxyClass），走 getProxyClassDataPermission() 到代理的接口上找。

通道二：MyBatis 拦截器 — SQL 改写

PlusDataPermissionInterceptor.java:36-173 继承 MyBatis-Plus 的 BaseMultiTableInnerInterceptor。

SELECT 拦截：beforeQuery

// Interceptor:55-67
public void beforeQuery(Executor executor, MappedStatement ms, ...) {
    // ① MyBatis-Plus 的忽略标记 → 跳过
    if (InterceptorIgnoreHelper.willIgnoreDataPermission(ms.getId())) return;
    // ② ThreadLocal 里没注解 → 跳过
    if (dataPermissionHandler.invalid()) return;
    // ③ 解析 SQL，内部回调 processSelect → setWhere
    PluginUtils.MPBoundSql mpBs = PluginUtils.mpBoundSql(boundSql);
    mpBs.sql(parserSingle(mpBs.sql(), ms.getId()));
}

UPDATE/DELETE 拦截：beforePrepare

// Interceptor:77-95
public void beforePrepare(StatementHandler sh, ...) {
    SqlCommandType sct = ms.getSqlCommandType();
    if (sct == SqlCommandType.UPDATE || sct == SqlCommandType.DELETE) {
        if (InterceptorIgnoreHelper.willIgnoreDataPermission(ms.getId())) return;
        if (dataPermissionHandler.invalid()) return;
        mpBs.sql(parserMulti(mpBs.sql(), ms.getId())); // 回调 processUpdate/processDelete
    }
}

processSelect / processUpdate / processDelete：回调 Handler

// Interceptor:106-113
protected void processSelect(Select select, int index, String sql, Object obj) {
    if (select instanceof PlainSelect) {
        this.setWhere((PlainSelect) select, (String) obj);
    }
}

// Interceptor:153-157
protected void setWhere(PlainSelect plainSelect, String mappedStatementId) {
    Expression sqlSegment = dataPermissionHandler.getSqlSegment(plainSelect.getWhere(), true); // isSelect=true
    if (null != sqlSegment) plainSelect.setWhere(sqlSegment);
}

// Interceptor:124-128 (update)
protected void processUpdate(Update update, int index, String sql, Object obj) {
    Expression sqlSegment = dataPermissionHandler.getSqlSegment(update.getWhere(), false); // isSelect=false
    if (null != sqlSegment) update.setWhere(sqlSegment);
}

ThreadLocal 中的数据：DataPermissionHelper

DataPermissionHelper.java 有两个存储空间：

存储作用生命周期
PERMISSION_CACHE (ThreadLocal)当前线程正在执行的 Mapper 的 @DataPermission 注解对象AOP Advice.setPermission → removePermission
getContext() (Sa-Token SaStorage)SpEL 变量 Map，存放 user、access 等整个 HTTP 请求生命周期

invalid()（Handler:328-330）就是检查 PERMISSION_CACHE.get() == null——没注解就跳过整个改写流程。

关键认知：数据权限不生效的第一个排查点永远是"ThreadLocal 里有没有注解"。如果 Mapper 方法没标 @DataPermission，AOP 切面不会拦截，ThreadLocal 就是 null，拦截器直接跳过。这解释了为什么框架自带的通用 Mapper 方法（如 selectById）不自动过滤——它们没标注解。
