01 全局心智模型 — 一条 SQL 的"变身"

本节目标：在你脑中种下一张"数据权限全链路地图"。学完后，你能不看代码说出从 Mapper 方法被调用到最终 SQL 多出 WHERE 片段，中间经过了哪几个关键步骤。

先看效果：同一条 SQL，不同角色看到不同数据

假设有这样一个用户列表查询：

SELECT * FROM sys_user WHERE status = '0'

当超级管理员执行时，SQL 不变——看到所有启用用户。但当一个普通部门经理（角色配置为「本部门及以下」）执行时，最终发给数据库的 SQL 变成了：

BEFORE SELECT * FROM sys_user WHERE status = '0'

AFTER  SELECT * FROM sys_user WHERE status = '0' AND ( dept_id IN ( 103, 104, 105 ) )

数据权限做的就是这件事：在 SQL 执行前，自动往 WHERE 里塞一段"你是谁，你只能看什么"的条件。

全链路六大环节

这是 RuoYi-Vue-Plus 数据权限体系从入口到结果的全景地图：

#环节发生了什么核心代码
① 注解声明 开发者在 Mapper 方法上加 @DataPermission + @DataColumn，声明"这条 SQL 的哪些列要受数据权限控制" SysUserMapper:37-39 — key=deptName → value=dept_id
② AOP 切面入栈 Mapper 方法被调用时，Spring AOP 切面把注解塞进 ThreadLocal DataPermissionAdvice:26-37 — setPermission → proceed → removePermission
③ 角色数据配置 登录时从 DB 读出当前用户所属角色及其 dataScope（1-6），存进 LoginUser SysLoginService:149 — buildLoginUser 组装 roles + dataScopeRoleMap
④ MyBatis 拦截器 MyBatis 执行 SQL 前，拦截器检查 ThreadLocal 有无注解，有则进入改写流程 PlusDataPermissionInterceptor:55-67 — beforeQuery 拦截 SELECT
⑤ Handler 拼 WHERE Handler 从 ThreadLocal 取出当前注解和当前用户角色 → 遍历角色 → 拿每个角色的 dataScope → 找到对应 SpEL 模板 → 渲染成 SQL 片段 → 拼回原 WHERE PlusDataPermissionHandler:72-97 — getSqlSegment
⑥ 最终 SQL 执行 改写后的 SQL 发给数据库，包含权限条件的行返回，不该看的行被过滤 无特定代码——这就是改写后的 SQL

这幅图的核心设计理念

理解这六个环节，你需要抓住三个关键设计决策：

1. 注解和拦截器是"配置"层，Handler 是"引擎"层。
注解 (@DataColumn) 告诉 Handler "用 dept_id 这列去做过滤"。Handler 不管 Mapper 的业务逻辑，它只做一件事：往 WHERE 里拼片段。

2. DataScopeType 是真正的"规则引擎"。
六种数据范围不是写死的 if-else，而是六条 SpEL 模板（见 DataScopeType:36-56）。模板里用 #{#deptName} 引用列名、用 #{@sdss.xxx} 调用方法。新增一种规则只要加一条枚举 + 写一个对应的 SpEL 模板。

3. "ALL" 是短路器。
用户角色中只要有一个是「全部数据权限」(code=1)，整个数据过滤就被跳过。这是通过 buildDataFilter 里的 return StringUtils.EMPTY 实现的（Handler:148-149）。

一个具体例子走一遍

场景：部门经理张三（deptId=103，角色 dataScope=4「本部门及以下」）查看用户列表。

步骤① Mapper: SysUserMapper.selectPageUserList() 被调用
       注解: @DataPermission({
           @DataColumn(key="deptName", value="dept_id"),
           @DataColumn(key="userName", value="create_by")
       })

步骤② Advice: DataPermissionHelper.setPermission(注解) → proceed()

步骤④ Interceptor: beforeQuery → ThreadLocal 里 != null → 进入改写

步骤⑤ Handler:
       currentUser() → 张三 deptId=103, roles=[经理(dataScope=4)]
       buildDataFilter():
         dataScope=4 → DataScopeType.DEPT_AND_CHILD
         模板: #{#deptName} IN ( #{@sdss.getDeptAndChild( #user.deptId )} )
         渲染: dept_id IN ( 103, 104, 105 )    ← sdss查了子部门
         注: key="deptName" → value="dept_id" 映射了列名
       getSqlSegment: 原WHERE AND ( dept_id IN (103,104,105) )

步骤⑥ 最终SQL: SELECT ... FROM sys_user WHERE status='0'
                                AND ( dept_id IN (103,104,105) )

现在你知道什么

  • 数据权限体系有六个环节：注解 → AOP 切面 → 角色配置 → MyBatis 拦截器 → Handler 拼 WHERE → 最终 SQL
  • 核心决策：注解是配置、DataScopeType 是规则引擎、"ALL" 是短路器
  • 一条简单的用户查询，最终会被塞进 AND ( dept_id IN (...) )
