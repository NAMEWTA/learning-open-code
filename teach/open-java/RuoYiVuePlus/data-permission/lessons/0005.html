05 Handler 大脑 — getSqlSegment 与 WHERE 拼接

本节目标：你能口述 getSqlSegment 的执行流程：超管检查 → 构建过滤条件 → 包括号 → 与原 WHERE 做 AND 拼接。

getSqlSegment：入口方法全貌

这是每次 SQL 改写的入口。来自 PlusDataPermissionHandler.java:72-97：

public Expression getSqlSegment(Expression where, boolean isSelect) {
    try {
        LoginUser currentUser = currentUser();           // ① 获取当前用户（有缓存）
        if (LoginHelper.isSuperAdmin()) {                // ② 超管短路
            return where;
        }
        String dataFilterSql = buildDataFilter(           // ③ 构建过滤条件
            getDataPermission(), currentUser, isSelect);
        if (StringUtils.isBlank(dataFilterSql)) {        // ④ 无过滤条件
            return where;
        }
        Expression expression = CCJSqlParserUtil          // ⑤ 解析过滤 SQL
            .parseExpression(dataFilterSql);
        // ⑥ 包一层括号，防止与原条件优先级冲突
        ParenthesedExpressionList<Expression> parenthesis =
            new ParenthesedExpressionList<>(expression);
        if (ObjectUtil.isNotNull(where)) {
            // ⑦ 原条件 AND (数据权限)
            return new AndExpression(where, parenthesis);
        } else {
            // ⑧ 如果原来没有 WHERE，直接用数据权限括号
            return parenthesis;
        }
    } catch (JSQLParserException e) {
        throw new ServiceException("数据权限解析异常 => " + e.getMessage());
    } finally {
        DataPermissionHelper.removePermission();          // ⑨ 清理 ThreadLocal
    }
}

逐步拆解

① currentUser() — 获取用户（含缓存）

Handler:187-195：先查 DataPermissionHelper.getVariable("user") 缓存，没有才调 LoginHelper.getLoginUser() 并写回缓存。同一请求内多次 SQL 命中同一用户对象，避免反复查 Sa-Token。

② 超管短路 — LoginHelper.isSuperAdmin()

LoginHelper.java:236-238：

public static boolean isSuperAdmin() {
    return isSuperAdmin(getUserId());          // → 比对 SystemConstants.SUPER_ADMIN_USER_ID
}

超管用户 ID 是常量（通常为 1761300000000000001）。如果是超管，原样返回 where，不添加任何过滤条件。这是整个体系最高优先级的短路。

注意：即使超管有多个角色，也不会进入 buildDataFilter。也就是说，超管 不会 受到任何数据权限限制。

③ buildDataFilter() — 构建过滤条件

这是下一节课的完整内容。这里只需要知道：它返回一个 SQL 片段的字符串，例如 "dept_id IN (103,104,105)"。

如果返回空字符串（用户无角色、或所有角色都是 ALL），则 不拼接任何条件。

⑥⑦⑧ WHERE 拼接策略

这是容易被忽略的细节，但决定了最终 SQL 的正确性：

原 SQL 有 WHERE：
原 WHERE AND ( 数据权限 )

例如：WHERE status = '0' AND ( dept_id IN (103,104,105) )
原 SQL 无 WHERE：
直接 ( 数据权限 )

例如：WHERE ( dept_id IN (103,104,105) )

ParenthesedExpressionList（来自 JSqlParser）确保数据权限的条件整体被包在括号里。这防止数据权限的条件与原有 WHERE 条件之间出现运算符优先级问题——比如原条件里如果有 OR，不加括号可能改变语义。

⑨ finally 清理

DataPermissionHelper.removePermission() 放在 finally 块。这意味着：不管 buildDataFilter 是否异常，ThreadLocal 都会清理。否则这个线程复用时（Tomcat 线程池），上次残留的注解会污染下次请求。

isSelect 参数的影响

isSelect 只影响 buildDataFilter 中 joinStr 的默认值：

  • isSelect=true（SELECT）：默认 OR。用户多个角色，任一角色的条件满足即可。
  • isSelect=false（UPDATE/DELETE）：默认 AND。用户多个角色，必须满足所有角色的条件。

整个 getSqlSegment 的决策树

getSqlSegment(where, isSelect)
 ├─ isSuperAdmin? → 原样返回（短路！）
 ├─ buildDataFilter 返回空? → 原样返回
 ├─ 原 where 非空? → where AND ( dataFilter )
 └─ 原 where 为空? → ( dataFilter )
 always finally → removePermission()
