一、SQL 拼接实例：dept_id IN (103,104,105) OR create_by = 1 是怎么来的？

场景设定

假设当前登录用户 张三，userId=1，所属部门 deptId=103。张三有两个角色：

角色dataScope对应的 DataScopeTypeSpEL 模板
部门经理 (roleId=10) 4 DEPT_AND_CHILD #{#deptName} IN ( #{@sdss.getDeptAndChild( #user.deptId )} )
普通用户 (roleId=20) 5 SELF #{#userName} = #{#user.userId}

Mapper 方法的注解假设为：

@DataPermission({
    @DataColumn(key = "deptName", value = "dept_id"),
    @DataColumn(key = "userName", value = "create_by")
})

当前为 SELECT 操作。

逐步骤追踪

步骤1 — 确定 joinStr（Handler:109-112）：

String joinStr = isSelect ? " OR " : " AND ";  // isSelect=true → joinStr = " OR "

步骤2 — 注册列名变量（Handler:128-138）：

context.setVariable("deptName", "dept_id");   // #deptName = "dept_id"
context.setVariable("userName", "create_by");  // #userName = "create_by"

步骤3 — 注入用户变量：

context.setVariable("user", loginUser);  // #user = 张三的 LoginUser 对象
// loginUser.userId = 1, loginUser.deptId = 103

步骤4 — 遍历角色，渲染模板：

角色1：部门经理 (dataScope=4 → DEPT_AND_CHILD)

// 模板： #{#deptName} IN ( #{@sdss.getDeptAndChild( #user.deptId )} )
//
// SpEL 求值过程：
// ① #deptName      → context.lookupVariable("deptName")  → 返回 "dept_id"
// ② #user.deptId   → context.lookupVariable("user")
//                   → NullSafePropertyAccessor.read(context, loginUser, "deptId")
//                   → 返回 103
// ③ @sdss          → beanResolver.resolve(context, "sdss")
//                   → beanFactory.getBean("sdss")
//                   → 返回 SysDataScopeServiceImpl 实例
// ④ .getDeptAndChild(103)
//                   → deptMapper.selectDeptAndChildById(103)
//                   → 查询部门树，返回 [103, 104, 105]
//                   → StreamUtils.join → "103,104,105"
//
// 最终渲染结果：
// "dept_id IN ( 103,104,105 )"

加入 conditions 集合：

conditions.add(" OR " + "dept_id IN ( 103,104,105 )");
// → conditions = {" OR dept_id IN ( 103,104,105 )"}

角色2：普通用户 (dataScope=5 → SELF)

// 模板： #{#userName} = #{#user.userId}
//
// SpEL 求值过程：
// ① #userName      → context.lookupVariable("userName") → 返回 "create_by"
// ② #user.userId   → NullSafePropertyAccessor.read(context, loginUser, "userId")
//                   → 返回 1
//
// 最终渲染结果：
// "create_by = 1"

加入 conditions 集合：

conditions.add(" OR " + "create_by = 1");
// → conditions = {" OR dept_id IN ( 103,104,105 )", " OR create_by = 1"}

步骤5 — 组装最终 SQL（Handler:175-178）：

// StreamUtils.join 将集合拼接为一个字符串
String sql = StreamUtils.join(conditions, Function.identity(), "");
// sql = " OR dept_id IN ( 103,104,105 ) OR create_by = 1"

// substring 去掉开头的 " OR "（5个字符）
return sql.substring(joinStr.length());
// 返回: "dept_id IN ( 103,104,105 ) OR create_by = 1"
为什么用 Set 而非 List 存 conditions？防止同一个角色+同一个 DataColumn 组合被重复添加。如果两个角色用了完全相同的模板和数据，Set 自动去重。
为什么是 OR 不是 AND？SELECT 场景下，只要任一角色的数据权限条件满足，该行数据就可见。这是"权限累加"模型——多角色看到的数据是各角色权限的并集。如果改为 AND，意味着某行数据必须同时属于部门 103 且由 userId=1 创建——几乎不可能有数据满足。

二、逐行详解 Handler:113-117 + BeanResolver 深度剖析

每一行到底在做什么

// Handler.java:113
Object defaultValue = "-1";

定义一个安全兜底值。当 SpEL 变量或属性为 null 时，返回 "-1" 而不是 null。为什么是 "-1"？因为：

  • dept_id IN (-1)：语法正确，不报错，且不匹配任何真实数据（部门ID从1开始）
  • dept_id IN (null)：SQL 语法错误
  • dept_id IN ()：SQL 语法错误（空 IN 列表）
// Handler.java:114
NullSafeStandardEvaluationContext context =
    new NullSafeStandardEvaluationContext(defaultValue);

创建自定义 SpEL 上下文。看它的实现（Handler:336-358）：

private static class NullSafeStandardEvaluationContext extends StandardEvaluationContext {
    private final Object defaultValue;

    @Override
    public Object lookupVariable(String name) {
        Object obj = super.lookupVariable(name);
        if (obj == null) {
            return defaultValue;  // 变量为 null → 返回 "-1"
        }
        return obj;
    }
}

作用：当 SpEL 表达式引用 #someVar 但这个变量在上下文中不存在或值为 null 时，返回 "-1" 而非抛异常或返回 null。

// Handler.java:115
context.addPropertyAccessor(
    new NullSafePropertyAccessor(
        context.getPropertyAccessors().getFirst(), defaultValue));

注册自定义属性访问器。看实现（Handler:365-447）：

private static class NullSafePropertyAccessor implements PropertyAccessor {
    private final PropertyAccessor delegate;  // 原始的 PropertyAccessor
    private final Object defaultValue;        // 兜底值 "-1"

    @Override
    public TypedValue read(EvaluationContext context, Object target, String name)
            throws AccessException {
        TypedValue value = delegate.read(context, target, name);
        if (value.getValue() == null) {
            return new TypedValue(defaultValue);  // 属性为 null → 返回 "-1"
        }
        return value;
    }
    // canRead/canWrite/write 全部委托给 delegate
}

装饰器模式（Decorator Pattern）：NullSafePropertyAccessor 包装了 SpEL 默认的 ReflectivePropertyAccessor。当 SpEL 访问对象属性（如 #user.deptId）时，先由原始访问器读取属性值，如果读到 null，返回 "-1"。

两层 NullSafe 防护对比：

层次覆盖场景示例
NullSafeStandardEvaluationContext.lookupVariable() 变量本身不存在或为 null #unknownVar → "-1"
NullSafePropertyAccessor.read() 变量存在，但其属性为 null #user.deptId 但 user.getDeptId()==null → "-1" 
// Handler.java:116
context.setBeanResolver(beanResolver);

注入 Bean 解析器。这里的 beanResolver 在 Handler:63 初始化：

private final BeanResolver beanResolver =
    new BeanFactoryResolver(SpringUtils.getBeanFactory());

BeanResolver 是什么？

BeanResolver 是 Spring SpEL 的一个接口：

// org.springframework.expression.BeanResolver
public interface BeanResolver {
    BeanInstance resolve(EvaluationContext context, String beanName);
}

BeanFactoryResolver 是 Spring 提供的唯一实现：

// org.springframework.context.expression.BeanFactoryResolver
public class BeanFactoryResolver implements BeanResolver {
    private final BeanFactory beanFactory;

    public BeanFactoryResolver(BeanFactory beanFactory) {
        this.beanFactory = beanFactory;
    }

    @Override
    public BeanInstance resolve(EvaluationContext context, String beanName) {
        return new BeanInstance(beanFactory.getBean(beanName));
    }
}

解析流程：当 SpEL 解析器遇到 @beanName 语法时：

  1. SpEL 识别 @ 前缀 → 这是一个 Bean 引用
  2. 调用 context.getBeanResolver().resolve(context, "beanName")
  3. BeanFactoryResolver 调用 beanFactory.getBean("beanName")
  4. 从 Spring 容器获取该名称的 Bean 实例
  5. 返回 BeanInstance 包装对象
  6. SpEL 继续解析后面的方法调用 .methodName(args)
// Handler.java:117
DataPermissionHelper.getContext().forEach(context::setVariable);

将 DataPermissionHelper 中通过 setVariable() 存入的自定义扩展变量注入 SpEL 上下文。这些变量来自 DataPermissionHelper 内部的 ThreadLocal 存储（本质是 Sa-Token 的 Storage）。这样业务代码可以通过 DataPermissionHelper.setVariable("myKey", myValue) 向数据权限模板注入自定义变量。

完整的 SpEL 求值链路（以 @sdss 为例）

表达式: #{@sdss.getDeptAndChild(#user.deptId)}

1. TemplateParserContext 提取 #{ 和 } 之间的内容
   → "@sdss.getDeptAndChild(#user.deptId)"

2. SpelExpressionParser 解析为 AST（抽象语法树）
   → MethodReference(BeanReference("sdss"), "getDeptAndChild", [VariableReference("user.deptId")])

3. Expression.getValue(context) 递归求值:
   3a. BeanReference("sdss")
       → context.getBeanResolver().resolve(context, "sdss")
       → BeanFactoryResolver.resolve("sdss")
       → beanFactory.getBean("sdss")
       → 返回 SysDataScopeServiceImpl@1234

   3b. VariableReference("user.deptId")
       → context.lookupVariable("user")
       → NullSafePropertyAccessor.read(context, loginUser, "deptId")
       → 返回 103

   3c. MethodReference.getDeptAndChild(103)
       → 反射调用 SysDataScopeServiceImpl.getDeptAndChild(103)
       → deptMapper.selectDeptAndChildById(103)
       → 返回 "103,104,105"

4. 字符串拼接："" + "103,104,105" → "103,104,105"
   返回给 TemplateParserContext 做最终字符串拼接

三、SpEL 深度解析 — 是什么、怎么理解、什么原理

SpEL 是什么？

Spring Expression Language (SpEL) 是 Spring 框架内置的表达式语言，类似于 JSP 的 EL、JSF 的表达式语言，但功能更强大。它可以在运行时查询和操作对象图。

核心组件

组件接口/类职责
表达式解析器 ExpressionParser → SpelExpressionParser 将字符串解析为可执行的 Expression 对象
表达式 Expression → SpelExpression 已编译的表达式，可反复求值
求值上下文 EvaluationContext → StandardEvaluationContext 提供变量、函数、类型转换器、Bean 解析器
模板上下文 ParserContext → TemplateParserContext 定义模板边界符号（默认 #{ 和 }）
属性访问器 PropertyAccessor → ReflectivePropertyAccessor 读写 Java 对象的属性（通过 getter/setter 或直接字段访问）
Bean 解析器 BeanResolver → BeanFactoryResolver 将 @beanName 解析为 Spring 容器中的 Bean

SpEL 语法全貌

// 1. 字面量
"hello"           // 字符串
123               // 数字
true              // 布尔

// 2. 变量引用（#变量名）
#user             // 从 EvaluationContext 中查找变量 "user"
#user.deptId      // 访问属性（通过 PropertyAccessor）
#user.getName()   // 调用方法

// 3. Bean 引用（@bean名）
@sdss             // 通过 BeanResolver 从 Spring 容器获取 Bean
@sdss.getRoleCustom(#roleId)  // 调用 Bean 的方法

// 4. 运算符
#a + #b           // 算术
#a > #b           // 比较
#a and #b         // 逻辑
#a ?: 'default'   // Elvis 运算符（null 时取默认值）

// 5. 集合
{1, 2, 3}         // 列表
{a: 1, b: 2}      // Map

// 6. 模板字符串（TemplateParserContext）
"hello #{#name}"  // #{...} 内是表达式，外是字面量

SpEL 解析原理（源码级）

以数据权限的模板 " #{#deptName} IN ( #{@sdss.getDeptAndChild( #user.deptId )} )" 为例：

// 步骤1: 词法分析（Lexer/Tokenizer）
// 输入字符串被切分为 token 序列
Tokens: " ", "#{", "#deptName", "}", " IN ( ", "#{", "@sdss", ".", "getDeptAndChild",
        "(", "#user", ".", "deptId", ")", " ", "}", " )"

// 步骤2: 模板解析（TemplateParserContext）
// TemplateParserContext 识别 #{ ... } 边界
// 将 #{ } 内部的内容提取为表达式，外部的为字面量
[literal: " "]
[expression: "#deptName"]
[literal: " IN ( "]
[expression: "@sdss.getDeptAndChild( #user.deptId )"]
[literal: " )"]

// 步骤3: 表达式解析（SpelExpressionParser）
// 每个表达式字符串被解析为 AST
Expression 1: VariableReference("deptName")
Expression 2: MethodReference(
    BeanReference("sdss"),
    "getDeptAndChild",
    [PropertyOrFieldReference(VariableReference("user"), "deptId")]
)

// 步骤4: 求值（Expression.getValue(context)）
// 遍历 AST 节点，递归求值
// - VariableReference    → context.lookupVariable()
// - BeanReference        → context.getBeanResolver().resolve()
// - PropertyOrFieldRef   → context.getPropertyAccessors().read()
// - MethodReference      → 反射 invoke()

// 步骤5: 结果拼接
// 将所有表达式求值结果插回字面量位置
" " + "dept_id" + " IN ( " + "103,104,105" + " )"
→ " dept_id IN ( 103,104,105 )"

Handler 中的解析器初始化

// Handler:53
private final ExpressionParser parser = new SpelExpressionParser();

// Handler:57-58
// 定义模板格式 —— #{ 开头、} 结尾
private final ParserContext parserContext = new TemplateParserContext();

// 使用:
parser.parseExpression(type.getSqlTemplate(), parserContext)
      .getValue(context, String.class);
//     ↑                    ↑
//     编译为 Expression     在指定上下文中求值，结果转为 String

为什么用 TemplateParserContext？模板中既有字面量（IN (、=）又有表达式（#deptName、@sdss.xxx）。不用模板上下文的话，整个字符串都会被当作一个表达式来解析，字面量会导致解析失败。

有无 TemplateParserContext 的区别：
有："hello #{#name}" → 字面量 "hello " + SpEL 求值 #name → "hello world"
无："hello #{#name}" → 整个作为 SpEL 表达式解析 → 语法错误（"hello " 不是合法 SpEL）

四、@sdss 到底是什么？

本质：Spring Bean 的名称引用

@sdss 中的 sdss 是 Spring 容器中一个 Bean 的名称。对应的 Bean 定义：

// SysDataScopeServiceImpl.java:28
@Service("sdss")                          // ← Bean 名称就是 "sdss"
public class SysDataScopeServiceImpl implements ISysDataScopeService {
    ...
}

解析链路：

SpEL 模板: "#{@sdss.getRoleCustom(#roleId)}"

1. SpEL 解析器看到 @sdss
   → 这是一个 Bean 引用，beanName = "sdss"

2. 调用 BeanResolver.resolve(context, "sdss")
   → BeanFactoryResolver.resolve("sdss")
   → beanFactory.getBean("sdss")

3. Spring 容器查找名为 "sdss" 的 Bean
   → 找到 @Service("sdss") 标注的 SysDataScopeServiceImpl

4. 返回 Bean 实例 → SpEL 继续调用 .getRoleCustom(roleId)
   → 反射: Method.invoke(beanInstance, roleId)
   → 执行业务逻辑

sdss 提供的两个方法

方法1：getRoleCustom(Long roleId)（SysDataScopeServiceImpl:41-52）

@Cacheable(cacheNames = CacheNames.SYS_ROLE_CUSTOM, key = "#roleId")
public String getRoleCustom(Long roleId) {
    if (ObjectUtil.isNull(roleId)) {
        return "-1";
    }
    List<SysRoleDept> list = roleDeptMapper.lambda()
        .select(SysRoleDept::getDeptId)
        .eq(SysRoleDept::getRoleId, roleId)
        .list();
    if (CollUtil.isNotEmpty(list)) {
        return StreamUtils.join(list, rd -> Convert.toStr(rd.getDeptId()));
    }
    return "-1";
}

查询 sys_role_dept 表，获取角色自定义授权的部门 ID 列表。结果如 "101,102,103"。有 Redis 缓存。未配置或 roleId 为 null 返回 "-1"。

方法2：getDeptAndChild(Long deptId)（SysDataScopeServiceImpl:60-71）

@Cacheable(cacheNames = CacheNames.SYS_DEPT_AND_CHILD, key = "#deptId")
public String getDeptAndChild(Long deptId) {
    if (ObjectUtil.isNull(deptId)) {
        return "-1";
    }
    List<Long> deptIds = deptMapper.selectDeptAndChildById(deptId);
    return CollUtil.isNotEmpty(deptIds) ?
        StreamUtils.join(deptIds, Convert::toStr) : "-1";
}

查询部门树，返回当前部门及其所有子部门的 ID 串。如 deptId=103，子部门有 104、105，返回 "103,104,105"。有 Redis 缓存。同样 null-safe 返回 "-1"。

为什么方法名是 @sdss 而不是 @sysDataScopeService？

这是 Spring 的 Bean 命名：@Service("sdss") 显式指定了 Bean 名称为 "sdss"。这样做的好处：

  • SpEL 模板中的引用简短：@sdss vs @sysDataScopeServiceImpl
  • 解耦：模板不依赖具体的类名，只依赖 Bean 名称
  • 可替换：换一个实现了相同接口的 Bean 并注册为 "sdss"，模板无需改动

五、computeIfAbsent 语法详解

语法

dataScopeRoleMap.computeIfAbsent(perm, k -> new ArrayList<>()).add(role.getRoleId());

拆解

这行代码分三部分理解：

部分代码含义
Map 方法 dataScopeRoleMap.computeIfAbsent(perm, ...) 如果 key=perm 不存在，用 lambda 计算一个 value 放入 Map；总是返回该 key 关联的 value（原有或新创建的）
Lambda 工厂 k -> new ArrayList<>() 当 key 不存在时，创建一个空的 ArrayList 作为初始 value
链式调用 .add(role.getRoleId()) 在返回的 List 上添加当前角色 ID

等价传统写法

// computeIfAbsent 一行代码
dataScopeRoleMap.computeIfAbsent(perm, k -> new ArrayList<>()).add(role.getRoleId());

// 等价传统写法（5行）
List<Long> roleIdList = dataScopeRoleMap.get(perm);
if (roleIdList == null) {
    roleIdList = new ArrayList<>();
    dataScopeRoleMap.put(perm, roleIdList);
}
roleIdList.add(role.getRoleId());

完整业务上下文

这段代码出自 SysPermissionServiceImpl.getDataScopeRoleMap()（PermissionServiceImpl:67-77）：

public Map<String, List<Long>> getDataScopeRoleMap(List<RoleDTO> roles) {
    List<Long> roleIds = StreamUtils.toList(roles, RoleDTO::getRoleId);
    // DB 查询：返回 Map<roleId, Set<权限码>>
    // 例: {101: ["system:user:list", "system:user:query"], 102: ["system:user:list"]}
    Map<Long, Set<String>> permsRoleIds =
        menuService.selectMenuPermsByRoleIds(roleIds);

    Map<String, List<Long>> rolePermsMap = new LinkedHashMap<>();
    // 反转 Map：从 "角色→权限列表" 变成 "权限→角色列表"
    permsRoleIds.forEach((roleId, perms) ->
        perms.forEach(perm ->
            rolePermsMap.computeIfAbsent(perm, key -> new ArrayList<>())
                        .add(roleId)
        )
    );
    return rolePermsMap;
}

数据转换示意

// 输入 (从DB查出):
permsRoleIds = {
    101 (roleId): ["system:user:list", "system:user:query"],
    102 (roleId): ["system:user:list"]
}

// 输出 (computeIfAbsent 构建):
dataScopeRoleMap = {
    "system:user:list":  [101, 102],   // 两个角色都有此权限
    "system:user:query": [101]         // 只有角色101有此权限
}

业务意义：当用户访问 /system/user/list 接口（权限码 system:user:list），Handler 的 scopeRoles() 方法用这个 Map 反查出只有角色 101 和 102 参与数据权限计算，而不是用户的全部角色。这就是第6课讲的"接口约束"机制。

computeIfAbsent vs putIfAbsent：
putIfAbsent(key, value)：key 不存在时放入 value，返回旧值（可能 null）
computeIfAbsent(key, function)：key 不存在时才执行 function 生成 value，总是返回该 key 关联的 value（永不为 null）。适合链式调用。

★ 步骤B 详解：buildLoginUser — LoginUser 的构建

这是最核心的一步（SysLoginService.java:126-147）：

public LoginUser buildLoginUser(SysUserVo user) {
    LoginUser loginUser = new LoginUser();
    Long userId = user.getUserId();

    // === 第一组：基础身份信息（同步设置）===
    loginUser.setUserId(userId);              // 用户ID
    loginUser.setDeptId(user.getDeptId());    // 部门ID
    loginUser.setUsername(user.getUserName());// 用户名
    loginUser.setNickname(user.getNickName());// 昵称
    loginUser.setUserType(user.getUserType());// 用户类型 (sys_user / app_user)

    // 部门名称和类别（通过 deptService 查询）
    if (ObjectUtil.isNotNull(user.getDeptId())) {
        Opt<SysDeptVo> deptOpt = Opt.of(user.getDeptId())
            .map(deptService::selectDeptById);
        loginUser.setDeptName(deptOpt.map(SysDeptVo::getDeptName).orElse(""));
        loginUser.setDeptCategory(deptOpt.map(SysDeptVo::getDeptCategory).orElse(""));
    }

    // === 第二组：权限与角色（4个虚拟线程并行加载）===
    ThreadUtils.virtualInvokeAll(
        () -> {
            // 任务1：获取菜单权限码集合
            loginUser.setMenuPermission(
                permissionService.getMenuPermission(userId));
        },
        () -> {
            // 任务2：获取角色标识集合
            loginUser.setRolePermission(
                permissionService.getRolePermission(userId));
        },
        () -> {
            // 任务3：获取角色列表 + 数据权限角色映射
            List<SysRoleVo> roles = roleService.selectRolesByUserId(userId);
            List<RoleDTO> roleDtos = BeanUtil.copyToList(roles, RoleDTO.class);
            loginUser.setRoles(roleDtos);
            loginUser.setDataScopeRoleMap(
                permissionService.getDataScopeRoleMap(roleDtos));
        },
        () -> {
            // 任务4：获取岗位列表
            List<SysPostVo> posts = postService.selectPostsByUserId(userId);
            loginUser.setPosts(BeanUtil.copyToList(posts, PostDTO.class));
        }
    );
    return loginUser;
}

数据权限视角：哪些字段参与 SQL 过滤？

当后续请求触发数据权限过滤时，Handler 从 LoginUser 中使用以下字段：

// Handler.currentUser() → 从 LoginUser 提取
#user.userId        → loginUser.getUserId()        → 用于 SELF 模板
#user.deptId        → loginUser.getDeptId()        → 用于 DEPT / DEPT_AND_CHILD 模板

// Handler.buildDataFilter() → 遍历角色
#roleId             → role.getRoleId()              → 用于 CUSTOM 模板
role.getDataScope() → DataScopeType.findCode(...)   → 选择模板

// Handler.scopeRoles() → 接口约束筛选
user.getRoles()              → 用户的全部角色
user.getDataScopeRoleMap()   → 权限码→角色ID 的反查映射
access.perms()               → 当前接口所需的权限码
关键理解：LoginUser 的 dataScopeRoleMap 不直接参与 SQL 生成，而是用于 scopeRoles() 方法从用户的所有角色中筛选出当前接口真正需要的角色子集。这一步避免了"用户在接口A拥有角色B(全部数据)的权限"的越权问题。

   登录流程                         数据权限运行时
   ────────                         ──────────────
   LoginUser                        PlusDataPermissionHandler
