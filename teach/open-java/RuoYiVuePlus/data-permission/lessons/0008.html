08 Vue 前端 — 数据权限配置全流程

本节目标：你能够从前端操作到后端接口，完整追踪数据权限配置的数据流：点"分配权限"按钮 → 选 dataScope → 勾部门树 → 提交 deptIds → PUT 接口。

整体结构

Vue 版的数据权限配置没有独立弹窗，而是和菜单权限合并到同一个「分配权限」弹窗（role/index.vue），用 el-tabs 分「菜单权限」和「数据权限」两个 Tab。所有逻辑代码都在一个文件（~1040 行）。

关键：提交接口是统一的 PUT /system/role/permission（updateRolePermission），菜单权限和数据权限一次提交。

六步操作流程

步骤1：点击"分配权限"按钮

行内操作列有「分配权限」按钮（index.vue:138-146），点击触发 handleDataScope(row)：

const handleDataScope = async (row) => {
  permissionTab.value = 'menu';              // 默认切到菜单Tab
  const response = await getRole(row.roleId); // ① 获取角色详情(含dataScope)
  Object.assign(form.value, response.data);
  const menuRes = await getRoleMenuTreeselect(row.roleId); // ② 菜单树+已选
  const res = await getRoleDeptTreeSelect(row.roleId);     // ③ 部门树+已选
  openDataScope.value = true;
  await nextTick(() => {
    // ④ 菜单树回填勾选
    initPermissionState(menuRes.checkedKeys);
    // ⑤ 部门树回填勾选
    deptRef.value?.setCheckedKeys(res.checkedKeys);
  });
};

注意 ③ getRoleDeptTreeSelect 调的接口是 GET /system/role/deptTree/{roleId}（deptTreeSelect），不同于部门管理用的 /system/dept/list。这个接口一次返回 { depts: 部门树, checkedKeys: 已勾选部门ID }。

为什么用 nextTick？ 弹窗打开后 DOM 需要渲染，部门树 (el-tree) 需要在挂载后才能调用 setCheckedKeys。

步骤2：选择 dataScope

下拉定义（index.vue:402-409）：

const dataScopeOptions = ref([

下拉绑定了 @change="dataScopeSelectChange"（index.vue:994-998）：

const dataScopeSelectChange = (value) => {
  if (value !== '2') {
    deptRef.value?.setCheckedKeys([]);  // 非"自定义"时清空已勾选的部门
  }
};

步骤3：勾选部门树（仅 dataScope=2 时）

部门树区域由 v-show="form.dataScope === '2'" 控制显隐（index.vue:296-317）：

<el-form-item v-show="form.dataScope === '2'" label="数据权限">
  <el-checkbox v-model="deptExpand">展开/折叠</el-checkbox>
  <el-checkbox v-model="deptNodeAll">全选/全不选</el-checkbox>
  <el-checkbox v-model="form.deptCheckStrictly">父子联动</el-checkbox>
  <el-tree
    ref="deptRef"
    :data="deptOptions"
    show-checkbox
    default-expand-all
    node-key="id"
    :check-strictly="!form.deptCheckStrictly"   <!-- 取反：联动时非严格模式 -->
    :props="{ label: 'label', children: 'children' }"
  />
</el-form-item>

check-strictly 的逻辑容易晕：el-tree 的 check-strictly 含义是「父子不联动」。而 UI 的 Checkbox 叫「父子联动」。所以绑定是 :check-strictly="!form.deptCheckStrictly"——联动时严格模式关，不联动时严格模式开。

步骤4：点击"确定"提交

index.vue:1017-1028：

const submitDataScope = async () => {
  if (form.value.roleId) {
    syncFormMenuPermissionIds();               // ① 收菜单勾选 → form.menuIds
    form.value.deptIds = getDeptAllCheckedKeys(); // ② 收部门勾选 → form.deptIds
    await updateRolePermission(form.value);      // ③ 一次提交全部
    modal.msgSuccess('修改成功');
    openDataScope.value = false;
    getList();                                   // ④ 刷新列表
  }
};

getDeptAllCheckedKeys 的细节（index.vue:875-884）：

const getDeptAllCheckedKeys = () => {
  const checkedKeys = deptRef.value?.getCheckedKeys();       // 全选节点
  const halfCheckedKeys = deptRef.value?.getHalfCheckedKeys(); // 半选节点
  if (halfCheckedKeys) {
    checkedKeys?.unshift(...halfCheckedKeys);  // 半选节点也带上！
  }
  return checkedKeys;
};

为什么半选节点也要带？ el-tree 父子联动时，子节点全选了父节点才显示为全选；如果只选了部分子节点，父节点是半选状态。但后端保存时需要知道「父节点被部分选中」，否则下次回显无法还原半选态。这个半选收集是 React 版需要手写 collectAncestorKeys 的原因（第9课对比）。

修改后，用户下次登录时 buildLoginUser 重新查 sys_role 的 dataScope，运行时 Handler 读新值。注意：已登录用户的角色不会立即刷新，需重新登录。
