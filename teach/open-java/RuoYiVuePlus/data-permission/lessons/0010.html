10 扩展与排错 — 新增规则、ignore 场景、三类故障排查

本节目标：你能新增一种自定义数据范围规则（改 3 个文件）、能在 3 种场景正确使用 ignore、能按排查清单快速定位"权限不生效/越权/SQL 报错"。

扩展一：新增一个数据范围规则

假设你要新增一种"仅本部门及上级部门"（DataScopeType code=7）。需要改 3 个地方：

1. 加枚举值

DataScopeType.java：

// 新增枚举值
DEPT_AND_PARENT("7",
    " #{#deptName} IN ( #{@sdss.getDeptAndParent( #user.deptId )} ) ",
    " 1 = 0 ");

2. 加 sdss 方法

SysDataScopeServiceImpl.java（或 ISysDataScopeService）：

// 新增方法，返回当前部门及上级部门ID串
@Cacheable(cacheNames = CacheNames.SYS_DEPT_AND_PARENT, key = "#deptId")
public String getDeptAndParent(Long deptId) {
    // 通过 ancestors 字段递归查找上级部门
    // 实现类似于 getDeptAndChild，但方向相反
}

3. 前端加选项

Vue：dataScopeOptions 加 { value: '7', label: '本部门及上级数据权限' }

React：dataScopeOptions 加 { value: '7', label: '本部门及上级数据权限' }

不改的文件：Handler、Interceptor、AOP 切面、注解都不用改。因为 Handler 是通过循环 DataScopeType.findCode(role.getDataScope()) 动态查找模板，新增枚举值自动生效。

扩展二：ignore 的三种使用场景

DataPermissionHelper.ignore() 的核心作用是临时关闭数据权限。三种场景：

场景1：SpEL 内防止死循环（框架内置）

PlusDataPermissionHandler:162：

String sql = DataPermissionHelper.ignore(() ->
    parser.parseExpression(type.getSqlTemplate(), parserContext)
          .getValue(context, String.class));

这是最关键的用法——渲染模板时 @sdss.getRoleCustom() 会查 DB，如果不 ignore，查 DB 又会触发数据权限。死循环。

场景2：业务代码中临时查全部数据

// 管理员操作：查看所有用户
DataPermissionHelper.ignore(() -> {
    List<SysUser> allUsers = userMapper.selectList();
    // 不受当前用户数据权限限制
});

注意用 DataPermissionHelper.ignore(Runnable) 或 DataPermissionHelper.ignore(Supplier)，都有 finally 自动恢复。

场景3：报表/统计类查询

// 统计报表需要全量数据
DataPermissionHelper.ignore(() -> {
    return reportMapper.selectMonthlyStats();
});
ignore 的使用原则：
  1. 最小作用域：ignore 包裹的代码越少越好，只包需要跳过权限的那次查询
  2. 一定要带 finally：Supplier/Runnable 版已内置，自己手写要确保恢复
  3. 不要滥用：如果到处都在 ignore，说明设计有问题——可能应该用角色配置来调整

三类典型故障排查清单

类型一：权限"不生效"（应该过滤但没过滤）

  1. Mapper 方法没加注解？ → 检查 Mapper 接口方法上有没有 @DataPermission。AOP 切点只匹配有注解的方法。
  2. 当前用户是超级管理员？ → LoginHelper.isSuperAdmin() 返回 true 则直接短路。检查用户 userId 是否等于 SUPER_ADMIN_USER_ID。
  3. 角色 dataScope=1（全部数据）？ → buildDataFilter 中遇到 ALL 直接返回空。检查 sys_role.data_scope。
  4. MyBatis-Plus 忽略标记？ → 检查 XML Mapper namespace 或代码中是否调了 InterceptorIgnoreHelper 跳过数据权限。
  5. 方法走了缓存？ → 如果方法加了 @DataPermission 但实际调用走了其他缓存路径（如一级缓存、Redis 缓存），可能绕过了 MyBatis 拦截器。

类型二：权限"越权"（能看不该看的数据）

  1. dataScope 配置错误？ → 检查角色 dataScope 字段。预期是本部门(3)但实际配了全部(1)。
  2. 多角色权限叠加？ → 用户有角色A(dataScope=1 全部)和角色B(dataScope=5 仅本人)。由于 OR 拼接，角色A 的 ALL 直接短路——用户看到全部数据。这是设计的正确行为：权限取并集。
  3. 接口约束未生效？ → 检查 Controller 有没有 @SaCheckPermission。如果没有，scopeRoles 会用全部角色参与计算，可能导致宽松角色越权。
  4. deptIds 未清理？ → 前端改了 dataScope 从 2（自定义）到 3（本部门），但 deptIds 没清空。Vue 版依赖 dataScopeSelectChange，React 版在 submit 时强制 dataScope !== '2' ? [] : nextDeptIds。

类型三：SQL 执行报错

  1. 多表 join 列名冲突？ → DataColumn.value 没加表别名，如写成 "dept_id" 而非 "d.dept_id"。
  2. key/value 数组长度不匹配？ → Handler:131 直接抛异常 "key与value长度不匹配"。检查注解的 key 和 value 数组。
  3. SpEL 表达式解析失败？ → 模板中有语法错误或引用了不存在的 bean。检查 DataScopeType.sqlTemplate。
  4. sdss 方法返回 null？ → 应该返回 "-1" 而非 null。NullSafe 框架会兜 default "-1"，但如果方法直接抛 NPE 就不同了。
  5. 部门树为空时 generate IN() 空参数？ → sdss 返回 "-1" 保证 IN (-1) 语法正确。如果忘了 null 检查，可能生成 IN () 导致 SQL 语法错误。
