02 注解层 — @DataPermission 与 @DataColumn 怎么写

本节目标：你能够在自己的 Mapper 方法上正确写出 @DataPermission + @DataColumn，覆盖单表查询、多表 join 别名、update/delete 三种场景。

两个注解的关系

@DataColumn 是子注解，@DataPermission 是容器。关系像这样：

@DataPermission(              ← 容器，默认 select 用 OR 连接多角色条件
    joinStr = "",             ← 可选，指定 OR 或 AND
    value = {
        @DataColumn(           ← 子注解，声明"占位符 → 列名"映射
            key = "deptName",  ← SpEL 模板里的变量名（#deptName）
            value = "dept_id"  ← 真实数据库列名
        ),
        @DataColumn(
            key = "userName",
            value = "create_by"
        )
    }
)

注解源码速览

@DataColumn（源码 DataColumn.java:17-31）定义了两个 String 数组属性：

String[] key() default "deptName";     // SpEL 里的变量名，可多个
String[] value() default "dept_id";   // 对应的列名，与 key 等长

@DataPermission（源码 DataPermission.java:14-28）：

DataColumn[] value();                 // 一组 DataColumn
String joinStr() default "";         // 多角色条件拼接符，默认 select=OR, update/delete=AND

两个注解都标注 @Target(ElementType.METHOD)（@DataPermission 额外支持 TYPE），所以可以放在方法或接口上。

场景一：单表查询（最常见的用法）

看用户列表查询，SysUserMapper.java:37-42：

@DataPermission({
    @DataColumn(key = "deptName", value = "dept_id"),
    @DataColumn(key = "userName", value = "create_by")
})
default Page<SysUserVo> selectPageUserList(Page<SysUser> page, Wrapper<SysUser> queryWrapper) {
    return this.selectVoPage(page, queryWrapper);
}

这里用了双列模板（dept_id + create_by）。为什么两个？因为 DataScopeType.DEPT_AND_CHILD 的模板是：

#{#deptName} IN ( #{@sdss.getDeptAndChild( #user.deptId )} )

而 DataScopeType.SELF 的模板是：

#{#userName} = #{#user.userId}

——如果你只声明 key=deptName，那仅本人的角色就不会产生任何过滤条件（因为模板里不含 #deptName）。只有两个 key 都声明，两个角色类型才都生效。

铁律：声明哪些列取决于你希望覆盖哪些 DataScopeType。至少要有 deptName 和 userName 两个 key，才能让部门过滤和本人过滤同时生效。

场景二：多表 join 用别名

当 Mapper 涉及多表 join 时，列名必须带表别名前缀，否则会列名冲突。看 SysUserMapper.java:66-69：

@DataPermission({
    @DataColumn(key = "deptName", value = "d.dept_id"),   ← 注意表别名 d
    @DataColumn(key = "userName", value = "u.create_by")   ← 注意表别名 u
})
default List<SysUserExportVo> selectUserExportList(SysUserBo user, List<Long> deptIds) {
    MPJLambdaWrapper<SysUser> wrapper = QueryBuilder.lambdaJoin("u", SysUser.class)
        .leftJoin(SysDept.class, "d", SysDept::getDeptId, SysUser::getDeptId)
        // ...
    return this.selectJoinList(SysUserExportVo.class, wrapper);
}

这里 MPJ（MyBatis-Plus-Join）给用户表起了别名 u，部门表起了别名 d。所以 value 必须写 "d.dept_id" 而非 "dept_id"。

常见错误：join 查询忘记改别名。SQL 会变成 WHERE ... AND (dept_id IN (...)) 但 join 后实际列名叫 d.dept_id，dept_id 不明确导致 SQL 报错。

场景三：update / delete 也要过滤

数据权限对 update 和 delete 同样生效。看 SysUserMapper.java:146-163：

@Override
@DataPermission({
    @DataColumn(key = "deptName", value = "dept_id"),
    @DataColumn(key = "userName", value = "create_by")
})
int update(@Param(Constants.ENTITY) SysUser user, @Param(Constants.WRAPPER) Wrapper<SysUser> updateWrapper);

@Override
@DataPermission({
    @DataColumn(key = "deptName", value = "dept_id"),
    @DataColumn(key = "userName", value = "create_by")
})
int updateById(@Param(Constants.ENTITY) SysUser user);

update/delete 与 select 的关键区别是 joinStr 默认值不同：select 默认 OR（任一角色匹配即通过），update/delete 默认 AND（所有角色条件都满足才允许）。这更严格，防止误修改不属于自己范围的数据。

joinStr 指定：什么时候用 AND？

在 demo 模块里可以看到 TestDemoMapper.java:77 用 joinStr="AND" 覆盖默认值：

@DataPermission(
    joinStr = "AND",
    value = { @DataColumn(key = "deptName", value = "dept_id") }
)
default List<TestDemoVo> selectByIds(Collection<Long> ids) {
    return this.selectVoBatchIds(ids);
}

什么时候用 AND？当你要求用户必须同时满足多个角色的所有数据约束。但实际业务中极少使用，默认的 OR 更符合直觉：「任意角色拥有某数据的查看权限，用户就能看到」。

实践检查清单

加数据权限注解前，问自己三个问题：

  1. 这张表的哪列代表"所属部门"？ → 填入 key="deptName" 对应的 value（通常 dept_id）
  2. 这张表的哪列代表"创建人/所属人"？ → 填入 key="userName" 对应的 value（通常 create_by 或 user_id）
  3. 有 join 吗？ → 有则给列名加表别名前缀
