07 角色配置与 sdss 服务 — 数据权限的"原材料"从哪来

本节目标：你能说清数据权限从角色配置到运行时数据的完整数据流：角色 dataScope 字段 → 登录态组装 → sdss 方法 → SpEL 模板渲染。

整体数据流

角色配置（前端） → sys_role.dataScope + sys_role_dept（数据库）
    → 登录时组装 LoginUser.roles / dataScopeRoleMap（登录态）
        → 运行时 Handler 读取 #user / #roleId（SpEL 变量）
            → @sdss 方法查 DB/缓存生成 id 串（SpEL Bean 调用）
                → 渲染进 SQL 模板（最终 SQL 片段）

第一层：角色数据范围落库

sys_role 表 — dataScope 字段

SysRole.java:47 的 dataScope 字段（String 类型，存 "1"~"6"）是角色级别的核心标记。一个角色一个 dataScope。

sys_role_dept 表 — 自定义部门关联

只有 dataScope=2（自定义）时需要。实体 SysRoleDept（角色ID + 部门ID），由 SysRoleServiceImpl.updateRolePermission()（第 337 行起）维护：

// 先删旧记录，再批量插入
roleDeptMapper.delete(roleDeptLambda.eq(SysRoleDept::getRoleId, roleId));
// ...
roleDeptMapper.insertBatch(roleDepts);  // 按 bo.getDeptIds() 逐条插入

提交时带 @CacheEvict(cacheNames = CacheNames.SYS_ROLE_CUSTOM) 清理缓存。

第二层：登录态组装

SysLoginService.buildLoginUser()（SysLoginService.java:149）：

// 1. 查出用户角色列表
List<RoleDTO> roles = roleService.selectRolesByUserId(userId);
// 2. 放入 LoginUser
loginUser.setRoles(roles);
// 3. 构建权限码→角色ID的映射（供接口约束使用）
loginUser.setDataScopeRoleMap(permissionService.getDataScopeRoleMap(roles));
// 4. 设置用户部门ID
loginUser.setDeptId(sysUser.getDeptId());

dataScopeRoleMap 的构建（SysPermissionServiceImpl.java:72）：

// 遍历角色 → 查每个角色的菜单权限 → 构建 权限码 → [角色ID列表]
Map<String, List<Long>> dataScopeRoleMap = new HashMap<>();
for (RoleDTO role : roles) {
    List<String> perms = menuService.selectMenuPermsByRoleId(role.getRoleId());
    perms.forEach(perm -> dataScopeRoleMap.computeIfAbsent(perm, k -> new ArrayList<>())
                                          .add(role.getRoleId()));
}

结果样例如：{"system:user:list": [100, 101], "system:user:edit": [100]}

当用户访问 /system/user/list 接口（注解 @SaCheckPermission("system:user:list")），resolveAccess()（Handler:257-281）读到 perms=["system:user:list"]，scopeRoles() 从 dataScopeRoleMap 反查出角色 100 和 101——只有这两个角色参与数据权限计算。角色 102（即使也是用户角色，但无 system:user:list 权限）被排除。

第三层：sdss 服务 — 运行时生成 ID 串

SysDataScopeServiceImpl.java（@Service("sdss")）提供两个核心方法：

getRoleCustom(roleId) — 自定义数据用

// SysDataScopeServiceImpl:42-54
@Cacheable(cacheNames = CacheNames.SYS_ROLE_CUSTOM, key = "#roleId")
public String getRoleCustom(Long roleId) {
    if (ObjectUtil.isNull(roleId)) return "-1";
    List<SysRoleDept> list = roleDeptMapper.lambda()
        .select(SysRoleDept::getDeptId)
        .eq(SysRoleDept::getRoleId, roleId)
        .list();
    return CollUtil.isNotEmpty(list)
        ? StreamUtils.join(list, rd -> Convert.toStr(rd.getDeptId()))
        : "-1";
}

查 sys_role_dept 表，返回逗号分隔的部门 ID 串，如 "101,102,103"。无配置返回 "-1"。

getDeptAndChild(deptId) — 部门及以下用

// SysDataScopeServiceImpl:64-69
@Cacheable(cacheNames = CacheNames.SYS_DEPT_AND_CHILD, key = "#deptId")
public String getDeptAndChild(Long deptId) {
    if (ObjectUtil.isNull(deptId)) return "-1";
    List<Long> deptIds = deptMapper.selectDeptAndChildById(deptId);
    return CollUtil.isNotEmpty(deptIds)
        ? StreamUtils.join(deptIds, Convert::toStr)
        : "-1";
}

deptMapper.selectDeptAndChildById 用 findInSet 递归查出所有子部门 ID，加上本身。

⚠️ 循环调用陷阱：这个类注释明确警告"此 Service 内不允许调用标注数据权限注解的方法"（SysDataScopeServiceImpl:20-23）。原因是 SpEL 渲染时 DataPermissionHelper.ignore 已启用忽略，但类内直接调用（不走代理）Ignore 不生效，会触发循环解析。解决方案是 deptMapper 内部只用不标注解的方法。
