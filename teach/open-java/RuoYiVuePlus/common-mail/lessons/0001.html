第 1 课 · 鸟瞰：三层架构如何封装 Jakarta Mail 入门地图

🎯 本节目标（对应使命 Success #1）：学完你能用一句话说清 ruoyi-common-mail 的三层架构：MailProperties → MailConfig → MailBuilder，并理解它与 ruoyi-common-sms 的架构同构与差异。这是后面两节课所有细节挂靠的骨架——先有地图，再走小路。

我们先不抠 MailBuilder 的 400 行源码，只回答一个问题：打开 ruoyi-common-mail 这个模块，你会看到什么？

一、整个模块就这么大

和短信模块一样精简：3 个 Java 类 + 1 个配置清单 + 1 个 pom：

ruoyi-common-mail/ ├── pom.xml ← 依赖 3 样：ruoyi-common-core + jakarta.mail-api + angus-mail └── src/main/ ├── java/org/dromara/common/mail/ │ ├── config/ │ │ ├── MailConfig.java ← 【装配】把 MailProperties 转成 MailAccount Bean │ │ └── properties/ │ │ └── MailProperties.java ← 【映射】yml 的 mail.* 配置 → POJO │ └── core/ │ └── MailBuilder.java ← 【发送】流式构建器，模块唯一对外 API └── resources/META-INF/spring/ └── ...AutoConfiguration.imports ← 【发现】登记 MailConfig 为自动配置类

真正发邮件的脏活累活，全在第三方库 Hutool（JakartaMail + MailAccount）里，Hutool 底下才是 Jakarta Mail（jakarta.mail-api + Eclipse Angus 实现）。ruoyi-common-mail 自己一行「SMTP 协议交互」的代码都没有。

💡 一句话定位：
ruoyi-common-mail 不是一个邮件发送库，而是一层「流畅封装（fluent wrapper）」——它把 Hutool 的 Jakarta Mail 能力用 Builder 模式包起来，通过 Spring Boot 自动配置无感知接入，让业务代码能用 .to().subject().text().send() 一行链式调用发出邮件。

二、三层架构，三层职责

记住这张表，本课就成功了一半。后面两课各讲透一个层面：

层次类一句话职责对应课
数据层 MailProperties 把 yml 中 mail.host / mail.port 等 10 个属性绑定为 POJO，并提供 toMailAccount() 方法将其转为 Hutool 的 MailAccount 第 2 课
装配层 MailConfig 激活 MailProperties 绑定，根据 mail.enabled 开关条件创建 MailAccount Bean，通过 AutoConfiguration.imports 实现依赖即生效 第 2 课
API 层 MailBuilder 提供流式 Builder API 组合邮件参数（收件人/抄送/主题/正文/图片/附件），最终调用 send() 发送，返回 message-id 第 3 课

三、一条最短链路：从 yml 到邮件发出

模块本身不暴露 Controller。它是「公共能力层」，被上层业务依赖。最典型的使用现场是发送邮箱验证码（第 3 课细讲），链路长这样：

配置层 (yml) 装配层 (Spring) API 层 底层 ─────────── ─────────── ─────── ────── application-dev.yml: mail: enabled: true host: smtp.163.com port: 465 from: xxx@163.com user: xxx@163.com pass: xxxxxx │ │ │ ... ▼ │ │ ① Spring Boot 启动 │ │ 扫 .imports → 发现 MailConfig │ │ │ │ │ ② @EnableConfigurationProperties │ │ → MailProperties 绑定 yml │ │ │ │ │ ③ mail.enabled=true │ │ → mailAccount() 被调用 │ │ → mailProperties.toMailAccount() │ │ → 容器中注册 MailAccount Bean │ │ │ │ │ 业务调用 (CaptchaController) │ ▼ │ │ ④ MailBuilder.of() │ │ .to(email) │ │ .subject("验证码") │ │ .text(code) │ │ .send() ◄────────────────────┤ │ │ │ │ ⑤ validate() │ │ 校验收件人/标题/正文 │ │ │ │ │ ⑥ resolveMailAccount() │ │ SpringUtils.getBean │ │ (MailAccount.class) │ │ │ │ │ ⑦ JakartaMail.create() │ │ .send() ─────────────────► Hutool → Jakarta Mail → SMTP 服务器

看这张图，三层各司其职：MailProperties 在启动时把 yml 变成对象；MailConfig 在启动时把对象变成 Bean；MailBuilder 在运行时把 Bean + 业务参数组装成邮件发出。

四、与 ruoyi-common-sms 的架构对比

两个模块都遵循 RuoYi 的「common 模块标准范式」，但设计选择有微妙差别。理解这些差别等于理解两种不同的封装策略：

对比维度ruoyi-common-smsruoyi-common-mail
底层依赖 sms4j 3.3.5（聚合框架） Hutool JakartaMail + Jakarta Mail API
类数量/结构 3 个：config + dao + handler 3 个：config + properties + builder
有没有 Properties 类 ❌ 没有（配置由 sms4j starter 内部解析） ✅ 有 MailProperties（显式声明 10 个字段 + toMailAccount 转换）
有没有 Builder 模式 ❌ 没有（直接调 sms4j 的 SmsBlend.sendMessage()） ✅ MailBuilder（400 行，私有构造器 + 静态工厂 + 链式 API）
异常处理 ✅ SmsExceptionHandler（全局 @RestControllerAdvice 兜底） ❌ 没有（运行时抛 IllegalArgumentException 或 Jakarta Mail 原生异常，由业务方或全局异常处理器兜底）
启用控制 通过 @Primary 覆盖默认 Bean 通过 @ConditionalOnProperty(mail.enabled) 条件创建
缓存/存储 ✅ PlusSmsDao（Redis 缓存拦截计数） ❌ 没有（邮件发送本身就是一次性 IO 操作，无需缓存计数）
💡 关键洞察：短信模块是「适配垫片」——把 sms4j 的接口用 RuoYi 自己的 Redis、异常处理重新包裹；邮件模块是「流畅封装」——对 Hutool 的邮件 API 做 Builder 模式包装，方便业务方链式调用。两者的底层都有成熟库兜底，RuoYi 只做「最后一公里」的体验优化。

五、Jakarta Mail vs Spring Mail：为什么不用 spring-boot-starter-mail？

Spring Boot 官方有 spring-boot-starter-mail，提供 JavaMailSender + MimeMessageHelper。但 RuoYi 选择了另一条路：

Spring Boot MailRuoYi 路线 (Hutool JakartaMail)
配置方式spring.mail.*mail.*（自定义前缀，解耦 Spring 官方约定）
API 风格JavaMailSender.send(MimeMessage)，需要手动构造 MimeMessageMailBuilder.of().to().subject().text().send()，链式一行搞定
附件/图片需 MimeMessageHelper.addAttachment()MailBuilder.files(File...).image(cid, stream)
Session 管理自动管理通过 useGlobalSession 灵活控制
依赖需要 spring-boot-starter-mail只需 jakarta.mail-api + angus-mail（更轻量）

选择 Hutool 路线的核心原因：Hutool 本身就是 RuoYi 的底层工具基础（全项目到处是 StrUtil / CollUtil / ObjectUtil），用它的邮件工具不需要引入额外依赖。而且 MailBuilder 的 Builder 模式比 Spring 的 MimeMessageHelper 更符合 RuoYi 代码风格中的「链式调用」偏好。

六、模块的 pom 依赖：轻得出奇

<!-- pom.xml 的三个依赖 -->
<dependency>
    <groupId>org.dromara</groupId>
    <artifactId>ruoyi-common-core</artifactId>  <!-- SpringUtils、StringUtils -->
</dependency>
<dependency>
    <groupId>jakarta.mail</groupId>
    <artifactId>jakarta.mail-api</artifactId>  <!-- Jakarta Mail 接口 -->
</dependency>
<dependency>
    <groupId>org.eclipse.angus</groupId>
    <artifactId>jakarta.mail</artifactId>       <!-- Eclipse Angus 实现 -->
</dependency>

注意：没有直接依赖 Hutool 的邮件模块。Hutool 的 JakartaMail 和 MailAccount 来自 cn.hutool:hutool-all，而这个依赖已经被 ruoyi-common-core 传递引入（RuoYi 全项目统一使用 Hutool 6.x 的 all 包）。

🤔 想一想：模块里明明没有任何 @Import 或 @ComponentScan，Spring 怎么知道要加载 MailConfig？和 sms 模块的机制一样吗？ 完全一样。靠 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件。这个文件只有一行 org.dromara.common.mail.config.MailConfig。Spring Boot 启动时扫所有 jar 的这个固定路径，把列出的类当作自动配置类加载。短信模块的 SmsAutoConfiguration 也是同一机制。这是 RuoYi 所有 common 模块「依赖即生效」的统一入口——第 2 课专门讲它。

七、检验一下（提取练习）

不要回看上文，凭记忆作答——回忆比重读更能形成长期记忆。

Q1. ruoyi-common-mail 的三层架构从下到上分别是什么？

A. MailBuilder → MailConfig → MailProperties B. MailProperties（数据层）→ MailConfig（装配层）→ MailBuilder（API 层） C. Controller → Service → DAO D. pom.xml → .imports → Java 类
看答案B。MailProperties 从 yml 取数据 → MailConfig 把数据变成 Spring Bean → MailBuilder 在运行时使用 Bean 发送邮件。三层各管一层抽象。

Q2. ruoyi-common-mail 和 ruoyi-common-sms 在结构上最大的不同是什么？

A. mail 模块没有异常处理器、没有 Redis 缓存组件、但有独立的 Properties 类和 Builder 模式封装 B. 完全一样，只是换了类名 C. mail 模块多了一个 Controller D. mail 模块没有自动配置
看答案A。短信模块有 SmsExceptionHandler（全局异常兜底）和 PlusSmsDao（Redis 缓存），因为短信需要防刷拦截和统一错误响应。邮件是一次性 IO 操作，不需要这些——它用 @ConditionalOnProperty 控制启停，用 Builder 模式提升调用体验。

Q3. 为什么不直接用 Spring Boot 的 spring-boot-starter-mail？

A. Spring Boot Mail 不好用 B. Hutool 是 RuoYi 的工具基础，无需额外引入 spring-boot-starter-mail；且 MailBuilder 的 Builder 模式比 MimeMessageHelper 更符合 RuoYi 的代码风格 C. Jakarta Mail 已经过时了 D. 作者不知道有这个 starter
看答案B。RuoYi 全项目使用 Hutool 作为基础工具库（StrUtil/CollUtil/ObjectUtil 随处可见），Hutool 自带 JakartaMail 封装。用 Hutool 路线无需额外依赖，且 MailBuilder 的链式 API 与 RuoYi 整体代码风格一致。
