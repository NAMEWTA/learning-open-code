一、MailBuilder 的设计骨架

先读类声明和核心字段：

public final class MailBuilder {                    // final → 不可继承

    private MailAccount mailAccount;                // 可选的账户覆盖
    private boolean useGlobalSession = true;        // 全局会话标志

    // ── 邮件参数 ──
    private final List<String> tos = new ArrayList<>();      // 收件人
    private final List<String> ccs = new ArrayList<>();      // 抄送
    private final List<String> bccs = new ArrayList<>();     // 密送
    private final Map<String, InputStream> images = new LinkedHashMap<>(); // 内嵌图片
    private File[] files = new File[0];                       // 附件
    private String from;                                      // 覆盖发件人
    private String user;                                      // 覆盖用户名
    private String pass;                                      // 覆盖密码
    private String subject;                                   // 标题
    private String content;                                   // 正文
    private boolean html;                                     // 是否HTML

    private MailBuilder() {}                         // 私有构造器！
}
1为什么是 private 构造器？
这是 Builder 模式的标准做法——不允许直接 new MailBuilder()，必须通过静态工厂 MailBuilder.of() 创建。好处：①入口统一，未来可以加校验/初始化逻辑；②语义化— MailBuilder.of() 比 new MailBuilder() 更清晰地表达「开始构建邮件」。
2两个静态工厂重载
// 默认：使用全局 MailAccount Bean（从 Spring 容器获取）
public static MailBuilder of() {
    return new MailBuilder();          // mailAccount = null
}

// 指定账户：传入自定义 MailAccount
public static MailBuilder of(MailAccount mailAccount) {
    return new MailBuilder().account(mailAccount);  // mailAccount 被设置
}
两个入口对应两种场景：常规发送用 of()（走 yml 配置的全局账户）；多账户/动态账户用 of(customAccount)（比如发件人是用户自己的邮箱）。

二、链式 API 全景图

MailBuilder 提供了约 18 个链式方法，分三类：

💡 为什么覆盖字段会让 useGlobalSession = false？
因为覆盖了 from/user/pass 后，需要克隆一份 MailAccount 来承载这些临时值（见下文 resolveMailAccount 分析）。克隆出来的账户和原始全局账户是两个不同的对象，不应共用全局 Session——否则可能污染其他使用全局账户的邮件发送。

三、send() — 魔术发生的 23 行

send() 方法（第 317-340 行）是 Builder 的终止操作——调用它，前面链式设置的所有参数才真正组合并发出：

public String send() {
    validate();                              // ① 校验必填项
    MailAccount account = resolveMailAccount(); // ② 解析账户
    JakartaMail mail = JakartaMail.create(account)
        .setUseGlobalSession(useGlobalSession); // ③ 创建邮件对象
    mail.setTos(tos.toArray(new String[0]));    // ④ 填充收件人
    if (CollUtil.isNotEmpty(ccs)) {
        mail.setCcs(ccs.toArray(new String[0])); //     抄送人
    }
    if (CollUtil.isNotEmpty(bccs)) {
        mail.setBccs(bccs.toArray(new String[0])); //     密送人
    }
    mail.setTitle(subject);                      //     标题
    mail.setContent(content);                    //     正文
    mail.setHtml(html);                          //     是否HTML
    mail.setFiles(files);                        //     附件
    try {
        if (MapUtil.isNotEmpty(images)) {
            images.forEach(mail::addImage);      // ⑤ 添加内联图片
        }
        return mail.send();                      // ⑥ 真正发送！返回 message-id
    } finally {
        images.values().forEach(IoUtil::close);  // ⑦ 无论成功失败，关闭图片流
    }
}

四、resolveMailAccount() — 最体现设计功力的一笔

这是整个 MailBuilder 最核心的私有方法（第 362-375 行）：

private MailAccount resolveMailAccount() {
    MailAccount account = mailAccount;              // ① 先看是否传入了自定义账户
    if (account == null) {
        account = SpringUtils.getBean(MailAccount.class); // ② 没有？从容器拿全局 Bean
    }
    if (StringUtils.isAllBlank(from, user, pass)) { // ③ 没有任何覆盖字段？
        return account;                              //    直接返回原对象（全局或自定义）
    }
    // ④ 有覆盖字段 → 克隆 + 覆盖
    MailAccount copy = ObjectUtil.clone(account);    //    深克隆（序列化方式）
    copy.setFrom(StringUtils.blankToDefault(from, copy.getFrom()));
    copy.setUser(StringUtils.blankToDefault(user, copy.getUser()));
    copy.setPass(StringUtils.blankToDefault(pass, copy.getPass()));
    return copy;
}
1为什么克隆而不直接改 account？
MailBuilder.java:370 — ObjectUtil.clone(account)
如果直接 account.setFrom("override@test.com")，改的是容器里的全局单例 Bean。其他正在使用这个 Bean 的线程也会受到影响——发出去的邮件 From 全变了。克隆一份副本，在副本上设覆盖值，不影响原 Bean。这是不可变共享对象的经典做法。
2StringUtils.blankToDefault(from, copy.getFrom()) 是什么意思？
MailBuilder.java:371
语义：「如果 from 不为空，用 from；如果为空，用 copy.getFrom()（原账户的值）」。也就是说，覆盖是选填的——你可以只覆盖 from 不改 user，也可以只覆盖 pass 而 from 和 user 沿用配置。三个覆盖字段独立生效，互不影响。
3怎么知道 MailAccount 是可克隆的？
Hutool 的 MailAccount 实现了 Serializable 接口。Hutool 的 ObjectUtil.clone() 使用 JDK 序列化/反序列化做深拷贝（类似 ByteArrayOutputStream + ObjectOutputStream 的经典写法），所以能完整复制 MailAccount 的所有字段。

五、validate() — 前置校验

private void validate() {                               // 行号 345-355
    if (CollUtil.isEmpty(tos)) {
        throw new IllegalArgumentException("邮件收件人不能为空");
    }
    if (StrUtil.isBlank(subject)) {
        throw new IllegalArgumentException("邮件标题不能为空");
    }
    if (content == null) {                              // 注意：content 可以是空字符串 ""
        throw new IllegalArgumentException("邮件正文不能为空");
    }
}

注意 content == null 和 content.isEmpty() 的区别。这里只检查 null，允许发送空内容的邮件（虽然业务上不太合理，但技术上 SMTP 允许）。如果要更严格，可以改用 StrUtil.isBlank(content)。

💡 为什么不用 Bean Validation (@NotNull 注解)？
MailBuilder 不是 Spring Bean，是每次通过 new 创建的普通对象，不在 Spring 的管理范围内，所以 @NotNull / @Validated 不生效。用手写的 validate() 更直接，而且抛出的异常消息是中文的——对用户更友好。

六、try-finally 保图片流不泄漏

try {
    if (MapUtil.isNotEmpty(images)) {
        images.forEach(mail::addImage);     // 添加图片到邮件
    }
    return mail.send();                     // 发送
} finally {
    images.values().forEach(IoUtil::close); // ← 无论成败，关闭所有图片 InputStream
}

这是资源管理的关键防御。使用者传进来的 InputStream 可能是文件流、网络流、内存流——不管是什么，发完邮件（或发送失败）之后，这个流都得关。如果不关，文件句柄会泄漏，累积多了就会 Too many open files。finally 确保即使 mail.send() 抛异常（比如 SMTP 连接超时），图片流也会被关闭。

7.2 邮箱验证码 — CaptchaController（ruoyi-admin）

// ruoyi-admin/.../CaptchaController.java:112
MailBuilder.of()
    .to(email)
    .subject("登录验证码")
    .text("您本次验证码为：" + code + "，有效性为" + Constants.CAPTCHA_EXPIRATION + "分钟，请尽快填写。")
    .send();
RedisUtils.setCacheObject(key, code, Duration.ofMinutes(Constants.CAPTCHA_EXPIRATION));

注意这段代码的结构：先发邮件，再存 Redis。而不是先存 Redis 再发邮件。为什么？因为如果先存 Redis 再发邮件，而邮件发送失败，Redis 里就有了一个「永远用不上」的验证码——用户没收到，却占着 key。先发邮件的话，万一发送失败抛异常，Redis 写操作不会执行，数据一致。

Q3. 解析 resolveMailAccount() 的三种分支路径。

参考答案 分支1：mailAccount != null && from/user/pass 全空 → 直接返回调用者传入的 account（如 MailBuilder.of(customAccount) 的场景）。

分支2：mailAccount == null && from/user/pass 全空 → 从 Spring 容器获取全局 MailAccount Bean 返回（如 MailBuilder.of() 的场景）。

分支3：from/user/pass 任一非空 → 克隆账户 + 覆盖非空字段后返回（如调了 .from("boss@company.com") 覆盖发件人的场景）。

Q4. 为什么 image() 方法的图片 Map 用 LinkedHashMap 而不是 HashMap？

看答案LinkedHashMap 保持插入顺序。HTML 邮件中多个内嵌图片按 cid 引用的顺序添加更可预测，虽然 Jakarta Mail 协议本身不关心顺序，但从代码可读性和调试角度，有序的内部状态是个好习惯。
