第1课 · MailBuilder与条件装配：三层架构的设计密码

本节目标：学完你能说清 ruoyi-common-mail 的三层架构如何各司其职——MailProperties 映射 yml 配置、MailConfig 条件装配 Bean、MailBuilder 流式发送邮件。能解释 @ConditionalOnProperty 为何是邮件模块的"总开关"，以及 Builder 模式中的 useGlobalSession 标志在何时为何变化。能对比本模块与 ruoyi-common-sms 的架构同构与设计差异。

打开 ruoyi-common-mail 模块源码，你会看到极致的精简——整个模块只有 5 个文件。但对这 5 个文件的设计理解，决定了你能否在生产环境中正确、安全地使用它。本课从鸟瞰开始，逐个击破三层架构的设计密码。

一、模块全景：5 个文件，三层职责

模块目录结构一目了然：

ruoyi-common-mail/
├── pom.xml                           ← 依赖：ruoyi-common-core + jakarta.mail-api + angus-mail
└── src/main/
    ├── java/org/dromara/common/mail/
    │   ├── config/
    │   │   ├── MailConfig.java                ← 装配层：把 MailProperties 转成 MailAccount Bean
    │   │   └── properties/
    │   │       └── MailProperties.java        ← 数据层：绑定 yml 的 mail.* 配置
    │   └── core/
    │       └── MailBuilder.java               ← API 层：流式构建器，对外唯一入口
    └── resources/META-INF/spring/
        └── org.springframework.boot.autoconfigure.AutoConfiguration.imports  ← 发现机制

3 个 Java 类 + 1 个配置清单 + 1 个 pom。这个规模与 ruoyi-common-sms 的 3 个 Java 类 + 1 个 imports + 1 个 pom 完全一致——两者共享同一套"RuoYi common 模块标准范式"。但文件内部的职责划分和设计选择各有千秋，后文会逐项对比。

三层架构的数据流方向是单向的：

yml 配置文件
    │
    ▼  @ConfigurationProperties 绑定
MailProperties  (数据层 — 10 个字段的 POJO)
    │
    ▼  toMailAccount() 转换
MailConfig      (装配层 — 条件开关 + Bean 注册)
    │
    ▼  @Bean 注册到 Spring 容器
MailAccount Bean (Hutool 的邮件账户对象)
    │
    ▼  SpringUtils.getBean() 获取
MailBuilder     (API 层 — 流式构建 + 发送)
设计洞察：模块本身没有一行 SMTP 协议代码。真正的网络通信、MIME 编码、SSL 握手全部在 Hutool（JakartaMail + MailAccount）和底层的 Jakarta Mail（jakarta.mail-api + Eclipse Angus 实现）中完成。ruoyi-common-mail 的本质是一层"流畅封装（fluent wrapper）"——它让业务代码能用 .to().subject().text().send() 一行链式调用发出邮件，而不需要直接操作 Hutool 的 API。

二、数据层：MailProperties — 10 个 yml 属性一把抓

config/properties/MailProperties.java 只有 97 行，核心是两样东西：10 个 Lombok 字段 + 一个 toMailAccount() 转换方法。

2.1 字段绑定机制

@Data
@ConfigurationProperties(prefix = "mail")
public class MailProperties {
    private Boolean enabled;          // 总开关
    private String host;              // SMTP 服务器地址
    private Integer port;             // SMTP 端口
    private Boolean auth;             // 是否需要认证
    private String user;              // 登录用户名
    private String pass;              // 密码/授权码
    private String from;              // 发件人（RFC-822 标准）
    private Boolean starttlsEnable;   // STARTTLS 加密
    private Boolean sslEnable;        // SSL 加密
    private Long timeout;             // SMTP 超时（毫秒）
    private Long connectionTimeout;   // Socket 连接超时（毫秒）
    // ... toMailAccount() 方法
}

@ConfigurationProperties(prefix = "mail") 注解告诉 Spring Boot：扫描 yml/properties 文件中所有以 mail. 开头的键，通过 setter 方法（Lombok @Data 生成）注入到对应字段。配套的 yml 配置长这样：

# application-dev.yml
mail:
  enabled: false           # 默认关闭！
  host: smtp.163.com
  port: 465                # SSL 端口
  auth: true
  from: xxx@163.com
  user: xxx@163.com
  pass: xxxxxxxxxx         # 注意：SMTP 授权码，非登录密码
  starttlsEnable: true
  sslEnable: true
  # timeout 和 connectionTimeout 省略 → null → 不超时

2.2 toMailAccount()：从 POJO 到 Hutool 对象的关键转换

为什么需要这个方法？因为发送邮件的是 Hutool 的 MailAccount 对象，不是 MailProperties。这个方法把 10 个分散的字段逐个 set 进 MailAccount：

public MailAccount toMailAccount() {
    MailAccount account = new MailAccount();
    account.setHost(host);
    account.setPort(port);
    account.setAuth(auth);
    account.setFrom(from);
    account.setUser(user);
    account.setPass(pass);
    account.setSocketFactoryPort(port);   // ← 关键细节
    account.setStarttlsEnable(starttlsEnable);
    account.setSslEnable(sslEnable);
    account.setTimeout(timeout);
    account.setConnectionTimeout(connectionTimeout);
    return account;
}
注意 setSocketFactoryPort(port)：SSL 连接时 SMTP 通信和 SSL 握手在同一端口完成，所以 Socket 工厂端口必须等于 SMTP 端口（如 465）。Hutool 的 MailAccount 区分了 port（SMTP 端口）和 socketFactoryPort（SSL Socket 端口），RuoYi 将其统一设为一个值。

hostStringsmtp.163.comSMTP 服务器域名
portInteger465 / 587465=SSL，587=STARTTLS
authBooleantrue几乎所有邮箱都需要 SMTP 认证
userStringxxx@163.com登录用户名（通常等于发件人）

三、装配层：MailConfig — 一个注解决定模块生死

config/MailConfig.java 是三个类中最短的——全文仅 31 行。但麻雀虽小五脏俱全，四个注解各怀绝技：

@AutoConfiguration
@EnableConfigurationProperties(MailProperties.class)
public class MailConfig {

    @Bean
    @ConditionalOnProperty(value = "mail.enabled", havingValue = "true")
    public MailAccount mailAccount(MailProperties mailProperties) {
        return mailProperties.toMailAccount();
    }
}

3.1 @AutoConfiguration — 如何被发现？

这是 Spring Boot 3.x 的自动配置标准入口。不同于旧版的 spring.factories 文件，Spring Boot 3.x 通过 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件发现自动配置类。这个文件只有一行：

org.dromara.common.mail.config.MailConfig

任何模块只要在 pom.xml 中依赖了 ruoyi-common-mail，Spring Boot 启动时就会扫描这个 jar 中的 imports 文件，加载 MailConfig。这就是 RuoYi 所有 common 模块的"依赖即生效"统一入口——不需要在业务代码中写 @Import 或 @ComponentScan。

与 sms 模块的细微差异：SmsAutoConfiguration 用了 @AutoConfiguration(after = DataRedisAutoConfiguration.class) 来保证 Redis 先就绪（因为 PlusSmsDao 依赖 Redis）。MailConfig 没有 after 约束——它只依赖 MailProperties（纯 POJO 绑定），不需要任何外部中间件先启动。

3.2 @EnableConfigurationProperties — 激活配置绑定

这个注解做两件事：① 将 MailProperties 注册为 Spring Bean（尽管它自己没有 @Component 注解）；② 触发 @ConfigurationProperties(prefix = "mail") 的 yml 属性绑定。没有它，MailProperties 的 10 个字段全是 null。

3.3 @ConditionalOnProperty — 邮件模块的"总开关"

这是本模块与 sms 模块最显著的设计差异，也是最值得深入理解的一笔：

@ConditionalOnProperty(value = "mail.enabled", havingValue = "true")

翻译成人话：只有当 yml 中 mail.enabled 的值等于字符串 "true" 时，mailAccount() 方法才会被执行，MailAccount Bean 才会被创建。

这意味着两种截然不同的运行时状态：

mail.enabled = true  → @ConditionalOnProperty 满足 → 创建 MailAccount Bean
                     → MailBuilder.of().send() 正常发送邮件

mail.enabled = false → @ConditionalOnProperty 不满足 → 容器中没有 MailAccount Bean
                     → MailBuilder.of() 仍可调用（它不依赖 Spring）
                     → 但 .send() 时 resolveMailAccount() 调用 SpringUtils.getBean(MailAccount.class)
                     → Spring 找不到 Bean → 抛出 NoSuchBeanDefinitionException
设计洞察：为什么 sms 模块不需要条件开关？因为 sms4j 是一个"要么用要么不引"的框架——引入 sms4j-spring-boot-starter 依赖本身就意味着你决定要用短信了，所以 SmsAutoConfiguration 无条件装配（通过 @Primary 覆盖默认实现）。但 Jakarta Mail 的依赖是 mail 模块自带的，任何依赖 ruoyi-common-mail 的业务模块都会间接引入 Jakarta Mail jar——但不一定配置了 SMTP。所以需要 @ConditionalOnProperty 给开发者一个显式的"我已准备好 SMTP 配置，请启用邮件功能"的声明。默认值 false 是一种"安全默认"（safe by default）——不配置 SMTP 就不创建 Bean，避免运行时因为 host=null 而炸掉。

3.4 启动时序完整走一遍

应用启动
  │
  ├─ ① Spring Boot 扫描所有 jar 中的 AutoConfiguration.imports
  │     → 发现 org.dromara.common.mail.config.MailConfig
  │
  ├─ ② 解析 @AutoConfiguration（无 after 约束，直接排队）
  │
  ├─ ③ 执行 @EnableConfigurationProperties(MailProperties.class)
  │     → 注册 MailProperties 为 Bean
  │     → @ConfigurationProperties(prefix="mail") 绑定 yml 属性
  │     → 10 个字段被填充
  │
  ├─ ④ 评估 @ConditionalOnProperty(value="mail.enabled", havingValue="true")
  │     ├─ yml 中 mail.enabled = true  → 继续执行步骤⑤
  │     └─ yml 中 mail.enabled = false → 跳过 mailAccount()，不注册 MailAccount Bean
  │
  └─ ⑤ mailProperties.toMailAccount() → 逐字段 set → 注册 MailAccount Bean → 启动完成

四、API 层：MailBuilder — Builder 模式的教科书级实现

core/MailBuilder.java 共 409 行，是三个类中最大、也是最精妙的一个。它不仅是模块对外的唯一 API，更是一份 Builder 模式 + Fluent API 的教科书级示例。

4.1 设计骨架：private 构造器 + static 工厂

public final class MailBuilder {                    // final → 禁止继承

    // ── 账户相关 ──
    private MailAccount mailAccount;                 // 可选的自定义账户
    private boolean useGlobalSession = true;         // 全局会话标志（默认 true）

    // ── 邮件参数 ──
    private final List<String> tos = new ArrayList<>();
    private final List<String> ccs = new ArrayList<>();
    private final List<String> bccs = new ArrayList<>();
    private final Map<String, InputStream> images = new LinkedHashMap<>();
    private File[] files = new File[0];
    private String from;                             // 可选覆盖
    private String user;                             // 可选覆盖
    private String pass;                             // 可选覆盖
    private String subject;
    private String content;
    private boolean html;

    private MailBuilder() {}                         // 私有构造器！外部不能 new

    public static MailBuilder of() {                 // 静态工厂方法
        return new MailBuilder();
    }

    public static MailBuilder of(MailAccount mailAccount) {
        return new MailBuilder().account(mailAccount);
    }
}

为什么用 private 构造器？这是 Builder 模式的核心约束——强制所有创建路径都必须通过静态工厂 of()。好处：① 入口统一，未来可以在工厂方法中加初始化逻辑；② 语义化——MailBuilder.of() 比 new MailBuilder() 更清晰地表达"开始构建邮件"的意图；③ 两个 of() 重载对应两种使用场景（默认账户 vs 自定义账户），调用者不需要知道内部实现。

观察 to()/cc()/bcc() 三个方法族的设计：每个都有 String 和 Collection 两个重载，内部先调 splitAddress()（先用分号替换逗号、再用 StrUtil.splitTrim 切分、去空白），再调 normalizeAddresses()（过滤空值、trim）。这种"输入宽松、内部严格"的策略让调用者可以用任意格式传邮箱地址，而内部始终保证数据干净。

4.3 useGlobalSession：一个容易被忽视但至关重要的标志

这个布尔字段（默认 true）控制 Jakarta Mail 的全局 Session 复用行为。理解它的关键在于 Jakarta Mail 的架构：

Jakarta Mail 在 JVM 级别维护一个全局的 javax.mail.Session 对象。当 useGlobalSession = true 时，Hutool 的 JakartaMail.create(account) 会复用这个全局 Session——所有使用同一账户的邮件发送共享同一个连接池和认证信息，效率更高。当 useGlobalSession = false 时，每次 create 都会创建一个新的独立 Session——完全隔离，但每次都要重新建立连接和认证。

那么 useGlobalSession 何时从 true 变成 false？答案在四个账户覆盖方法中：

public MailBuilder account(MailAccount mailAccount) {
    this.mailAccount = Objects.requireNonNull(mailAccount);
    this.useGlobalSession = false;              // ← 自定义账户 → 不共用全局 Session
    return this;
}

public MailBuilder from(String from) {
    this.from = from;
    this.useGlobalSession = false;              // ← 覆盖发件人 → 不共用全局 Session
    return this;
}
// user() 和 pass() 同理

为什么要这样设计？因为当你覆盖了 from/user/pass 之后，最终发送时 resolveMailAccount() 会克隆一份 MailAccount 并在克隆体上设新值（见第 2 课详解）。克隆出来的账户对象和原始全局 Bean 是两个不同的实例，如果还共用全局 Session，可能导致线程 A 用克隆账户发邮件时，Session 里缓存的是原始账户的认证信息——认证失败。所以一旦有任何覆盖行为，必须切到独立 Session，隔离不同账户的认证上下文。

设计洞察：这个设计体现了"默认安全、主动降级"的原则。最常见的使用场景（80%+）是不覆盖任何账户信息，直接用 yml 配置的全局账户发邮件——这时 useGlobalSession = true，享受连接复用带来的性能优势。少数需要临时切换账户的场景，通过覆盖方法自动关闭全局 Session，保证隔离性。调用者不需要手动管理这个标志——框架帮你做了正确的选择。

五、与 ruoyi-common-sms 的全维度对比

两个模块在文件数量和组织方式上高度同构（都是 3 个 Java 类 + AutoConfiguration.imports + pom），但设计选择各有侧重。理解这些差异，等于理解两种不同的"公共模块封装策略"：

对比维度ruoyi-common-mailruoyi-common-sms
底层依赖 Hutool JakartaMail + Jakarta Mail API（jakarta.mail-api + Eclipse Angus 实现） sms4j 3.3.5（sms4j-spring-boot-starter 聚合框架）
模块依赖 只需 ruoyi-common-core（Hutool 传递引入） 需要 ruoyi-common-redis（PlusSmsDao 用 Redis 做拦截计数）

Bean 注册策略 @ConditionalOnProperty(mail.enabled) — 条件装配，默认关闭 @Primary — 无条件装配，通过优先权覆盖默认实现
自动配置排序 无 after 约束（不依赖 Redis） after = DataRedisAutoConfiguration（需要 Redis 先就绪）

API 风格 ✅ Builder 模式 — MailBuilder.of().to().subject().text().send() — 链式 API，400+ 行私有实现 ❌ 无 Builder — 直接调用 sms4j 的 SmsBlend.sendMessage()，业务方自己拼参数

画龙点睛的总结：短信模块是"适配垫片"（把 sms4j 的接口用 RuoYi 自己的 Redis、异常处理器重新包裹）；邮件模块是"流畅封装"（对 Hutool 的邮件 API 做 Builder 模式包装，提升调用体验）。两者底层的脏活累活都由成熟三方库兜底，RuoYi 只做"最后一公里"的开发体验优化——这正是 common 模块的价值所在。

六、检验一下（提取练习）

Q1. 说出 ruoyi-common-mail 的三层架构及每层的核心类。

数据层：MailProperties — 通过 @ConfigurationProperties(prefix = "mail") 绑定 yml 的 10 个属性，提供 toMailAccount() 将 POJO 转为 Hutool MailAccount。

装配层：MailConfig — 通过 @AutoConfiguration + AutoConfiguration.imports 被 Spring 发现，在 mail.enabled=true 时创建 MailAccount Bean。

API 层：MailBuilder — 私有构造器 + 静态工厂 + 链式 API，组合邮件参数并调用 Hutool 发送。

Q2. mail.enabled=false 时，MailBuilder.of().to("a@b.com").subject("Hi").text("Hello").send() 会发生什么？

前三步（of、to、subject、text）都正常执行——MailBuilder 是普通 Java 对象，不依赖 Spring 容器。但 send() 内部调用 resolveMailAccount() 时，SpringUtils.getBean(MailAccount.class) 在容器中找不到 MailAccount 类型的 Bean（因为 @ConditionalOnProperty 条件不满足，MailConfig 没创建它），抛出 NoSuchBeanDefinitionException。

Q3. useGlobalSession 何时从 true 变成 false？为什么？

当调用四个账户覆盖方法中的任意一个时——account()、from()、user()、pass()——useGlobalSession 被设为 false。因为这些方法会触发 resolveMailAccount() 克隆 MailAccount 对象，克隆后的对象不应与原全局 Bean 共用同一个 Session，否则认证信息可能冲突。

Q4. 对比 ruoyi-common-mail 和 ruoyi-common-sms，最核心的三个差异是什么？

① API 风格：mail 有 Builder 模式（MailBuilder 链式 API），sms 无（直接调 sms4j 接口）。

② 启用控制：mail 用 @ConditionalOnProperty 条件开关（默认关闭），sms 用 @Primary 无条件装配。

③ 辅助组件：sms 有异常处理器（SmsExceptionHandler）和 Redis 缓存（PlusSmsDao），mail 都没有——邮件是一次性 IO 操作，不需要这些。
