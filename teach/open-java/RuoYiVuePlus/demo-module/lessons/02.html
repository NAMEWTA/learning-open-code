第 2 课 · 缓存与并发控制：用声明式注解保护你的接口 性能 + 安全

🎯 本节目标（对应使命 Success #2）：学完你能独立为任意接口添加 Spring Cache 缓存、@Lock4j 分布式锁、@RateLimiter 限流和 @RepeatSubmit 防重，并说清每种注解的适用场景和关键参数。

第 1 课我们搭建了 CRUD 骨架，但真实生产环境还需要解决三个问题：查询太慢（频繁查数据库）、并发冲突（多人同时操作同一条数据）、流量冲击（恶意刷接口）。ruoyi-demo 用四个 Controller 演示了这些问题的标准解法，而且全部是一个注解搞定。

一、Spring Cache 缓存三件套：10 倍查询加速

源码 controller/RedisCacheController.java 演示了 Spring Cache 最核心的三个注解：

// ===== ① @Cacheable —— 查询时自动缓存 =====
@Cacheable(cacheNames = "demo:cache#60s#10m#20#1",
           key = "#key",
           condition = "#key != null")
@GetMapping("/test1")
public R<String> test1(String key, String value) {
    return R.data(value);  // 第二次调用直接从缓存返回，不执行方法体
}

// ===== ② @CachePut —— 新增/更新时刷新缓存 =====
@CachePut(cacheNames = CacheNames.DEMO_CACHE,
          key = "#key",
          condition = "#key != null")
@GetMapping("/test2")
public R<String> test2(String key, String value) {
    return R.data(value);  // 方法体始终执行，但返回值会写入缓存
}

// ===== ③ @CacheEvict —— 删除时清除缓存 =====
@CacheEvict(cacheNames = CacheNames.DEMO_CACHE,
            key = "#key",
            condition = "#key != null")
@GetMapping("/test3")
public R<String> test3(String key, String value) {
    return R.data(value);  // 方法体执行后清除指定缓存
}

cacheNames 命名规则（框架扩展）

"demo:cache#60s#10m#20#1" 这种 # 分隔的命名格式是 RuoYi-Vue-Plus 对 Spring Cache 的增强，含义如下：

段值含义
名称demo:cache缓存名称（必填）
ttl60s过期时间 60 秒
maxSize10m最大缓存条目数 10
maxIdleTime20最大空闲时间 20 秒
maxSizePolicy1淘汰策略（1=条目数限制）

也可以使用预定义的常量 CacheNames.DEMO_CACHE，其值同样是 "demo:cache#60s#10m#20#1"。

重要警告：缓存注解严禁与数据权限等筛选功能一起使用，否则会造成缓存击穿和数据不一致。因为缓存基于 key 存储，但数据权限可能在每次查询时过滤不同的数据子集。

手动缓存操作：RedisUtils

// RedisCacheController.test6() —— 手动控制缓存生命周期
@GetMapping("/test6")
public R<Boolean> test6(String key, String value) {
    RedisUtils.setCacheObject(key, value);           // 写入缓存
    boolean flag = RedisUtils.expire(key, Duration.ofSeconds(10));  // 设置10秒过期
    ThreadUtil.sleep(11 * 1000);                     // 等待11秒
    Object obj = RedisUtils.getCacheObject(key);     // 再次读取（应为null）
    return R.ok(value.equals(obj));                  // 返回 false
}

@Cacheable 声明式

✅ 零侵入，一行注解
✅ Spring AOP 自动管理
❌ 只能在 Spring Bean 方法上使用
❌ 不够灵活

RedisUtils 手动式

✅ 任意位置可用
✅ 精细控制过期时间
✅ 支持所有 Redis 数据结构
❌ 需要手动管理缓存一致性

二、分布式锁：注解式 vs 编程式

源码 controller/RedisLockController.java 演示了 Lock4j 的两种使用方式：

// ===== 方式一：@Lock4j 注解（声明式）=====
@Lock4j(keys = {"#key"})          // 基于方法参数 key 的值加锁
@GetMapping("/testLock4j")
public R<String> testLock4j(String key, String value) {
    System.out.println("start:" + key + ",time:" + LocalTime.now());
    ThreadUtil.sleep(10000);       // 模拟 10 秒业务处理
    System.out.println("end :" + key + ",time:" + LocalTime.now());
    return R.data(value);
}

// ===== 方式二：LockTemplate（编程式）=====
@GetMapping("/testLock4jLockTemplate")
public R<String> testLock4jLockTemplate(String key, String value) {
    // 参数：锁key, 锁超时30秒, 获取锁超时5秒, 执行器类型
    final LockInfo lockInfo = lockTemplate.lock(
        key, 30000L, 5000L, RedissonLockExecutor.class
    );
    if (null == lockInfo) {
        throw new RuntimeException("业务处理中,请稍后再试");
    }
    try {
        ThreadUtil.sleep(8000);    // 业务处理
    } finally {
        lockTemplate.releaseLock(lockInfo);  // 必须手动释放！
    }
    return R.data(value);
}
对比维度@Lock4j 注解LockTemplate 编程
代码量1 行注解~10 行 try-finally
锁释放自动（方法结束即释放）必须手动 releaseLock
获取失败处理抛异常，全局异常处理器接管返回 null，可自定义降级逻辑
适用场景简单互斥，标准 CRUD需要自定义超时、降级、重试等复杂逻辑
底层实现Redisson 分布式锁（看门狗自动续期）
选型口诀：能用注解就用注解，代码最少、不会忘释放锁；需要自定义降级逻辑（如返回"请稍后重试"而非抛异常）时再用 LockTemplate。

三、分布式限流：三种粒度的流量保护

源码 controller/RedisRateLimiterController.java 演示了 @RateLimiter 的三种限流模式：

// ===== ① 全局限流：所有请求共享配额 =====
@RateLimiter(count = 2, time = 10)     // 10秒内最多2次请求
@GetMapping("/test")
public R<String> test(String value) {
    return R.data(value);
}

// ===== ② IP 限流：同一 IP 独享配额 =====
@RateLimiter(count = 2, time = 10, limitType = LimitType.IP)
@GetMapping("/testip")
public R<String> testip(String value) {
    return R.data(value);               // 每个 IP 每10秒2次
}

// ===== ③ 集群实例限流：每个服务实例独享配额 =====
@RateLimiter(count = 2, time = 10, limitType = LimitType.CLUSTER)
@GetMapping("/testcluster")
public R<String> testcluster(String value) {
    return R.data(value);               // 每个实例每10秒2次
}

// ===== ④ 基于参数的 IP 限流（SpEL 表达式）=====
@RateLimiter(count = 2, time = 10, limitType = LimitType.IP, key = "#value")
@GetMapping("/testObj")
public R<String> testObj(String value) {
    return R.data(value);
    // key 支持简单变量 #value 和复杂表达式 #{#value != 1 ? 1 : 0}
}
limitType计数维度适用场景
默认（全局限流）接口全局保护数据库，防止整体过载
LimitType.IP请求 IP防爬虫、防单用户恶意刷接口
LimitType.CLUSTER服务实例多实例部署时各自限流，互不影响
注意：key 属性支持 SpEL 表达式。#变量 获取简单变量值，#{复杂表达式} 计算后取值。例如 #{#value != 1 ? 1 : 0} 会在 value=1 时使用 key=0，否则 key=1。

四、批量操作：insertBatch 万级数据秒插入

源码 controller/TestBatchController.java 演示了高性能批量操作：

// 批量插入：1000 条数据一次提交，秒级完成
@PostMapping("/add")
public R<Void> add() {
    List<TestDemo> list = new ArrayList<>();
    for (int i = 0; i < 1000; i++) {
        TestDemo testDemo = new TestDemo();
        testDemo.setTestKey("批量新增");
        testDemo.setValue("测试新增");
        list.add(testDemo);
    }
    return toAjax(testDemoMapper.insertBatch(list));  // 核心方法
}

// 批量新增或更新：有 ID 则更新，无 ID 则插入
testDemoMapper.insertOrUpdateBatch(list);

// 批量删除：Lambda 条件删除
testDemoMapper.lambda()
    .eq(TestDemo::getOrderNum, -1L)
    .deleteCount();
性能提示：在 JDBC 连接 URL 中添加 rewriteBatchedStatements=true 参数，可将多条 INSERT 合并为一条批量 SQL，性能提升 10-50 倍。MyBatis-Plus 3.5.0+ 原生批处理配合该参数可达到与 insertBatch 相同的速度。

五、防重提交 @RepeatSubmit

这个注解在第 1 课已经见过，这里补充完整参数说明：

@RepeatSubmit(interval = 2, timeUnit = TimeUnit.SECONDS, message = "{repeat.submit.message}")
// interval: 间隔时间
// timeUnit: 时间单位（默认秒）
// message: 超出间隔时的提示信息（支持国际化 {key}）

底层基于 Redis 实现，以"用户 ID + 请求 URL + 参数"为 key 记录最近一次提交时间。

本节小结

注解解决什么问题一句话说明
@Cacheable查询慢第一次查库后缓存结果，后续命中缓存不再查库
@CachePut缓存更新写入数据库后同步刷新缓存
@CacheEvict缓存清理删除数据后同步清除缓存
@Lock4j并发冲突同一 key 同时只能一个线程执行
@RateLimiter流量过载限制单位时间内的请求次数
@RepeatSubmit重复提交同一用户短时间内不能提交相同请求
五个注解，覆盖了从「查询加速」到「写入保护」到「流量控制」的全部场景。在新业务模块中，这五个注解就是你的标准工具箱。
