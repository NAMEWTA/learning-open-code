第 3 课 · 数据安全：脱敏、加密与国际化一条龙 安全防线

🎯 本节目标（对应使命 Success #2~#3）：学完你能为任意实体配置数据脱敏和数据库字段加密，能区分五种脱敏策略和四种加密算法的适用场景，能用国际化让校验错误信息随语言切换。

前两课我们学会了写 CRUD 和加缓存锁，但数据安全问题还没解决：手机号不能明文返给前端（脱敏）、敏感字段不能明文存数据库（加密）、校验错误信息要支持多语言（国际化）。ruoyi-demo 用三个 Controller 演示了这些安全的「注解式」解法——同样是一个注解搞定。

一、数据脱敏 @Sensitive：五种策略 + 条件判断

源码 controller/TestSensitiveController.java 定义了一个内部类演示五种脱敏策略：

@Data
static class TestSensitive {

    @Sensitive(strategy = SensitiveStrategy.ID_CARD)
    private String idCard;     // 210397198608215431 → 210397**********31

    @Sensitive(strategy = SensitiveStrategy.PHONE, roleKey = "common")
    private String phone;      // 17640125371 → 176******71

    @Sensitive(strategy = SensitiveStrategy.ADDRESS, perms = "system:user:query")
    private String address;    // 北京市朝阳区某某四合院1203室 → 北京市******

    @Sensitive(strategy = SensitiveStrategy.EMAIL, roleKey = "common", perms = "system:user:query1")
    private String email;      // 17640125371@163.com → 1**********@163.com

    @Sensitive(strategy = SensitiveStrategy.BANK_CARD, roleKey = "common1", perms = "system:user:query")
    private String bankCard;   // 6226456952351452853 → 6226**********853
}

五种脱敏策略

策略输入示例输出示例脱敏规则
ID_CARD210397198608215431210397**********31保留前6位+后2位
PHONE17640125371176******71保留前3位+后2位

条件脱敏：roleKey 和 perms

@Sensitive 支持两个条件参数，实现「有权限时看明文，无权限时看脱敏」：

参数含义示例
roleKey拥有指定角色时不脱敏roleKey = "common" → 有 common 角色看明文
perms拥有指定权限时不脱敏perms = "system:user:query" → 有此权限看明文

当 roleKey 和 perms 同时存在时是 AND 关系：两者都满足才不脱敏。例如 email 字段需要用户同时拥有 common 角色 AND system:user:query1 权限才看明文。

管理员特殊处理：框架默认管理员不脱敏（显示所有明文）。如需修改，重写 SensitiveService 的实现。这是为了开发调试方便，生产环境建议按需调整。
实现原理：脱敏发生在 Jackson 序列化阶段（返回 JSON 给前端时），数据库里存的始终是明文。脱敏是「看的时候遮住」，不是「存的时候加密」——那是下一节要讲的 @EncryptField。

二、数据库字段加密 @EncryptField：透明加解密

源码 domain/TestDemoEncrypt.java 继承 TestDemo，用 @EncryptField 注解标记需要加密的字段：

@Data
@EqualsAndHashCode(callSuper = true)
@TableName("test_demo")
public class TestDemoEncrypt extends TestDemo {

    // testKey: RSA 非对称加密，写入时用公钥加密，读取时用私钥解密
    @EncryptField(algorithm = AlgorithmType.RSA,
        privateKey = "MIICdQIBADANBgkqhkiG9w...",
        publicKey = "MIGfMA0GCSqGSIb3DQEB...")
    private String testKey;

    // value: AES 对称加密，指定密码
    @EncryptField(algorithm = AlgorithmType.AES,
        password = "10rfylhtccpuyke5")
    private String value;
}

Controller 调用验证（源码 controller/TestEncryptController.java）：

@GetMapping()
public R<Map<String, TestDemoEncrypt>> test(String key, String value) {
    if (!encryptEnable) {     // 检查 mybatis-encryptor.enable 配置
        throw new RuntimeException("加密功能未开启!");
    }
    Map<String, TestDemoEncrypt> map = new HashMap<>(2);
    TestDemoEncrypt demo = new TestDemoEncrypt();
    demo.setTestKey(key);
    demo.setValue(value);
    mapper.insert(demo);                    // 写入：自动加密
    map.put("加密", demo);                   // demo 里存的是加密后的值

    TestDemoEncrypt testDemo = mapper.selectById(demo.getId());
    map.put("解密", testDemo);               // 读取：自动解密，显示明文
    return R.ok(map);
}
Java 明文→@EncryptField 拦截→加密后写入 DB→从 DB 读取密文→@EncryptField 拦截→解密后 Java 明文

整个过程对业务代码完全透明——Service 和 Controller 操作的始终是明文，加密解密由 MyBatis 拦截器自动处理。

四种加密算法

算法类型需要配置适用场景
AES对称加密password通用字段加密，性能好
RSA非对称加密privateKey + publicKey高安全场景，公钥加密私钥解密
SM2国密非对称privateKey + publicKey信创/国产化要求
SM4国密对称password信创/国产化要求

如果不指定 algorithm，则走默认的 YML 配置。也可以完全不写 @EncryptField 的任何参数（@EncryptField 空注解），同样走 YML 默认配置。

启用开关：加密功能需要 mybatis-encryptor.enable=true 才会生效。Controller 中通过 @Value("${mybatis-encryptor.enable}") 读取配置并做前置检查，防止未开启时误操作。

三、国际化 i18n：三种使用方式

源码 controller/TestI18nController.java 演示了国际化的三种使用场景：

// ===== 方式一：代码中获取国际化消息 =====
@GetMapping()
public R<Void> get(String code) {
    return R.ok(MessageUtils.message(code));
    // code = "user.register.success"
    // → zh_CN: "注册成功"  /  en_US: "Registration successful"
}

// ===== 方式二：Validator 校验国际化 =====
@GetMapping("/test1")
public R<Void> test1(
    @NotBlank(message = "{not.null}") String str  // {not.null} 指向 messages.properties
) {
    return R.ok(str);
}

// ===== 方式三：Bean 校验国际化 =====
@GetMapping("/test2")
public R<Void> test2(@Validated TestI18nBo bo) {
    return R.ok(bo);
}

@Data
public static class TestI18nBo {
    @NotBlank(message = "{not.null}")
    private String name;

    @NotNull(message = "{not.null}")
    @Range(min = 0, max = 100, message = "{length.not.valid}")
    private Integer age;
}

三种方式的区别：

  • MessageUtils.message(code)：代码中主动获取国际化文本，适合业务提示、邮件内容等。
  • @NotBlank(message="{not.null}")：校验失败时自动从 messages.properties 取出对应语言的错误提示。
  • Bean 校验：通过对整个对象 @Validated，批量校验所有带国际化注解的字段。

国际化消息的 key 定义在 messages.properties（默认中文），扩展语言只需添加 messages_en_US.properties 等文件，框架根据请求头 Accept-Language 自动切换。

四、安全防线全景图

安全维度注解生效时机保护目标
数据脱敏@Sensitive(strategy=...)Jackson 序列化时（返回前端）API 响应中的敏感字段
数据库加密@EncryptField(algorithm=...)MyBatis 读写拦截器数据库存储的敏感字段
国际化message="{key}"校验失败 / 代码调用时多语言用户的错误提示
权限校验@SaCheckPermission(...)请求进入 Controller 前未授权访问
防重提交@RepeatSubmit(...)请求进入方法前重复提交导致的脏数据
从网络请求到数据库存储，五层安全防线各司其职。特别记住脱敏和加密的区别：脱敏管「看」（前端显示），加密管「存」（数据库落盘），两者互补而非替代。
