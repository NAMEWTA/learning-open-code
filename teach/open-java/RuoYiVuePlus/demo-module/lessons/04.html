第 4 课 · 消息通信全景：Pub/Sub、WebSocket、MQTT 与优先队列 消息通道

🎯 本节目标（对应使命 Success #3）：学完你能对比说出四种消息通信方式的差异与选型原则，能为业务选择合适的通道，并能独立接入 Redis Pub/Sub、WebSocket 推送、MQTT 发布订阅和优先级队列。

前三课我们覆盖了 CRUD、缓存锁、数据安全——都是「请求-响应」模式。但真实系统还需要主动推送：管理员操作后通知在线用户、IoT 设备上报数据、异步任务的结果回调。ruoyi-demo 用四个 Controller 演示了从轻到重四种消息通信方案，每种都有明确的最佳适用场景。

一、Redis Pub/Sub：最轻量的发布订阅

源码 controller/RedisPubSubController.java，整个实现只有 20 行：

// ===== 发布消息 =====
@GetMapping("/pub")
public R<Void> pub(String key, String value) {
    RedisUtils.publish(key, value, consumer -> {
        System.out.println("发布通道 => " + key + ", 发送值 => " + value);
    });
    return R.ok("操作成功");
}

// ===== 订阅消息 =====
@GetMapping("/sub")
public R<Void> sub(String key) {
    RedisUtils.subscribe(key, String.class, msg -> {
        System.out.println("订阅通道 => " + key + ", 接收值 => " + msg);
    });
    return R.ok("操作成功");
}

📡 Redis Pub/Sub 特点

✅ 零依赖（Redis 自带）
✅ 极简 API（一行发布一行订阅）
✅ 毫秒级延迟
❌ 不持久化（离线期间消息丢失）
❌ 不保证送达（fire-and-forget）
❌ 无 Ack 机制

最佳场景：集群内缓存失效通知、配置刷新广播——允许偶尔丢失、但要求极低延迟的通知。

二、WebSocket：点对点 + 全服广播

源码 controller/WebSocketController.java，通过 MessageService 实现双向推送：

private final MessageService messageService;

@GetMapping("/send")
public R<Void> send(Long userId, String message) {
    PushPayloadDTO payload = PushPayloadDTO.of(
        PushTypeEnum.MESSAGE,      // 消息类型
        PushSourceEnum.BACKEND,    // 来源：后端推送
        message,                   // 消息内容
        null                       // 扩展参数
    );
    if (userId == null) {
        messageService.publishAll(payload);               // 广播：所有在线用户
    } else {
        messageService.publishMessage(List.of(userId), payload);  // 点对点：指定用户
    }
    return R.ok("操作成功");
}

🔌 WebSocket 特点

✅ 全双工长连接（服务端可随时推送）
✅ 支持点对点和广播
✅ 低延迟（毫秒级）
❌ 需维护连接状态（心跳、重连）
❌ 仅在线用户（离线无法收到）
❌ 无持久化

最佳场景：站内信、实时通知、在线用户踢出——需要即时送达给在线用户，离线用户可通过其他方式补偿。

PushPayloadDTO.of() 四个参数的含义：

参数类型含义
pushTypePushTypeEnum消息类型（MESSAGE / NOTICE / WARNING 等）
pushSourcePushSourceEnum来源（BACKEND 后端推送 / FRONTEND 前端触发）
messageString消息内容
extendObject扩展参数（可携带业务数据）

三、MQTT：IoT 场景的标准协议

源码 controller/MqttController.java，基于 mica-mqtt 实现注解式 MQTT 客户端：

// ===== 发布消息 =====
@GetMapping("/send")
public boolean send() {
    client.publish("/test/client", "测试测试".getBytes(StandardCharsets.UTF_8));
    return true;
}

// ===== 订阅 QoS 0（最多一次）=====
@MqttClientSubscribe("/test/#")          // # 通配符匹配所有子主题
public void subQos0(String topic, byte[] payload) {
    log.info("topic:{} payload:{}", topic, new String(payload, StandardCharsets.UTF_8));
}

// ===== 订阅 QoS 1（至少一次）=====
@MqttClientSubscribe(value = "/qos1/#", qos = MqttQoS.QOS1)
public void subQos1(String topic, byte[] payload) { ... }

// ===== 占位符主题：${productKey}/${deviceName} =====
@MqttClientSubscribe("/sys/${productKey}/${deviceName}/thing/sub/register")
public void thingSubRegister(String topic, byte[] payload) {
    // ${} 占位符会被替换为 +（单层通配符），匹配任意 productKey 和 deviceName
    log.info("topic:{} payload:{}", topic, new String(payload, StandardCharsets.UTF_8));
}

// ===== JSON 反序列化：自动将 payload 转为业务对象 =====
@MqttClientSubscribe(
    value = "/test/json",
    deserialize = MqttJsonDeserializer.class   // 自定义反序列化器
)
public void testJson(String topic, MqttPublishMessage message, TestDemo data) {
    // 参数映射规则（按类型自动识别）：
    //   String → topic（主题）
    //   MqttPublishMessage → 原始消息（含 MQTT5 props）
    //   byte[] / ByteBuffer → payload 原始字节
    //   其他类型 → JSON 反序列化后的业务对象
    log.info("topic:{} json data:{}", topic, data);
}

🌐 MQTT 特点

✅ QoS 三级保证（0/1/2）
✅ 主题通配符（# 多层 / + 单层）
✅ 自动反序列化（JSON → Java 对象）
✅ 离线消息（Broker 持久化）
❌ 需外部 Broker（EMQX / Mosquitto）
❌ 学习成本较高
❌ 延迟高于 Redis Pub/Sub

最佳场景：IoT 设备通信、车联网、智能家居——需要可靠送达、支持通配符主题、设备离线后能补收消息。

参数映射规则（mica-mqtt 2.4.5+）：@MqttClientSubscribe 方法的参数按类型自动映射——String 第一个参数映射 topic、MqttPublishMessage 映射原始消息（可获取 MQTT5 props）、byte[] 或 ByteBuffer 映射 payload 原始字节、其他类型走 JSON 反序列化。参数个数可以是 2 或 3 个。

四、优先级队列：轻量级 MQ 替代方案

源码 controller/queue/PriorityQueueController.java，基于 Redis Sorted Set 实现：

// ===== 添加队列数据 =====
@GetMapping("/add")
public R<Void> add(String queueName) {
    QueueUtils.destroyPriorityQueue(queueName);      // 先清空旧数据
    for (int i = 0; i < 10; i++) {
        PriorityDemo data = new PriorityDemo();
        data.setName("data-" + i);
        data.setOrderNum(RandomUtil.randomInt(10));   // 优先级分数
        QueueUtils.addPriorityQueueObject(queueName, data);
    }
    return R.ok("操作成功");
}

// ===== 消费队列数据（按优先级从高到低）=====
@GetMapping("/get")
public R<Void> get(String queueName) {
    PriorityDemo data;
    do {
        data = QueueUtils.getPriorityQueueObject(queueName);
        log.info("通道: {} , 获取数据: {}", queueName, data);
    } while (data != null);
    return R.ok("操作成功");
}

// ===== 删除指定数据 =====
@GetMapping("/remove")
public R<Void> remove(String queueName, String name, Integer orderNum) {
    PriorityDemo data = new PriorityDemo();
    data.setName(name);
    data.setOrderNum(orderNum);
    QueueUtils.removePriorityQueueObject(queueName, data);
    return R.ok("操作成功");
}

📊 优先级队列特点

✅ 零外部依赖（Redis Sorted Set）
✅ 支持优先级排序（orderNum 越小越优先）
✅ 集群安全（同一条消息只被消费一次）
❌ 无 Ack 机制（消费即删除）
❌ 无死信队列
❌ 仅适合轻量级场景

最佳场景：订单按优先级处理、延迟任务调度——数据量不大、需要按权重排序、不想引入 RabbitMQ/RocketMQ 等重量级 MQ。

集群注意事项：优先队列在集群环境下同一条消息只会被一个实例消费。但因为消费即删除（无 Ack），如果消费后业务处理失败，消息就丢失了。源码注释明确提醒：做好事务补偿。重要业务建议使用真正的 MQ。

五、四种通道的选型决策表

通道可靠性延迟外部依赖典型场景
Redis Pub/Sub低（不持久化）极低无（Redis 自带）缓存失效通知、配置刷新
WebSocket中（仅在线用户）低无（框架内置）站内信、实时通知、在线踢出
MQTT高（QoS 1/2）中需要 MQTT BrokerIoT 设备、车联网、智能家居
优先级队列中（无 Ack）低无（Redis Sorted Set）订单优先级处理、延迟调度
选型口诀：丢得起选 Pub/Sub，在线用户选 WebSocket，IoT 设备选 MQTT，要排序不想装 MQ 选优先队列。如果以上都不满足——引入 RabbitMQ 或 RocketMQ。
