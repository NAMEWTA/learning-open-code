第 4 课 · 条件装配与配置 配置

🎯 本节目标（对应使命 Success #2）：学完你能解释 @ConditionalOnMessageTransport + MessageTransportCondition 的条件装配机制：注解如何声明期望传输方式、条件类如何读取 message.transport 配置、如何实现「SSE 配置类和 WebSocket 配置类只有一个会被激活」。

一、条件装配的三层结构

模块有三个配置类，它们不是平级的，而是有层次和依赖关系：

MessageAutoConfiguration ← 第 1 层：公共配置，始终生效 │ (通过 .imports 文件自动发现) │ @ConditionalOnProperty(prefix="message", name="enabled") │ @EnableConfigurationProperties(MessageProperties.class) │ ├── MessageSseConfiguration ← 第 2 层：SSE 通道配置 │ @AutoConfiguration(after = MessageAutoConfiguration.class) │ @ConditionalOnMessageTransport("sse") │ └── MessageWebSocketConfiguration ← 第 2 层：WebSocket 通道配置 @AutoConfiguration(after = MessageAutoConfiguration.class) @ConditionalOnMessageTransport("websocket")

三层设计意图：

  • 第 1 层做两件公共的事：检查总开关 message.enabled、激活配置绑定 @EnableConfigurationProperties——无论选 SSE 还是 WebSocket 都需要。
  • 第 2 层的两个配置类互斥——由 @ConditionalOnMessageTransport 裁决谁生效。两者都声明 after = MessageAutoConfiguration.class，确保公共层先加载。

二、自定义条件注解：@ConditionalOnMessageTransport

代码在 annotation/ConditionalOnMessageTransport.java，只有 13 行：

@Documented
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Conditional(MessageTransportCondition.class)   // ← 关键
public @interface ConditionalOnMessageTransport {
    String value();   // "sse" 或 "websocket"
}

这个注解做的事情任何人都能一眼看穿：

  • 元注解 @Conditional 是 Spring 4.0 提供的条件装配入口。它的参数是一个 Condition 实现类。
  • value() 是注解的使用者传入的期望传输方式字符串。
  • 可以用在类上（TYPE）或方法上（METHOD）——实际使用中贴在配置类上。

实际使用场景：

// SSE 配置类
@ConditionalOnMessageTransport("sse")
public class MessageSseConfiguration { ... }

// WebSocket 配置类
@ConditionalOnMessageTransport("websocket")
public class MessageWebSocketConfiguration { ... }

三、条件判断核心：MessageTransportCondition

代码在 condition/MessageTransportCondition.java。这是整个条件装配的裁决逻辑：

public class MessageTransportCondition implements Condition {

    @Override
    public boolean matches(@NonNull ConditionContext context,
                           AnnotatedTypeMetadata metadata) {
        // ① 从注解上取出 value() 的值（"sse" 或 "websocket"）
        Map<String, Object> attributes =
            metadata.getAnnotationAttributes(
                ConditionalOnMessageTransport.class.getName());
        if (attributes == null) {
            return true;  // 没有注解 → 直接通过
        }

        // ② 从配置中读取 message.enabled（默认 true）
        Boolean enabled = context.getEnvironment()
            .getProperty("message.enabled", Boolean.class, true);

        // ③ 从配置中读取 message.transport（默认 "sse"）
        String transport = context.getEnvironment()
            .getProperty("message.transport",
                MessageTransportEnum.SSE.getCode());

        // ④ 注解期望值 vs 配置实际值
        String expected = (String) attributes.get("value");
        return enabled && expected.equalsIgnoreCase(transport);
    }
}

这个 matches() 方法回答了三个问题：

1 怎么知道注解声明的值？通过 metadata.getAnnotationAttributes() 反射拿到 @ConditionalOnMessageTransport 的所有属性值，再取 "value" 字段。
2 怎么读到 yml 配置？通过 context.getEnvironment().getProperty("message.transport")——这是 Spring 的标准 Environment API，能读到 application.yml / 环境变量 / 命令行参数等所有来源的配置。
3 怎么判断？enabled && expected.equalsIgnoreCase(transport)——两个条件都必须满足：①总开关打开、②注解期望值与配置匹配。注意用了 equalsIgnoreCase，大小写不敏感。
💡 如果 message.enabled = false：matches() 返回 false，两个通道配置类都不会被加载。这意味着 SseEmitterSessionManager 和 WebSocketSessionManager 都不会注册成 Bean。PushHelper.isEnabled() 也返回 false，业务代码中所有推送调用都被跳过——整个推送模块静默关闭。

四、公共配置类：MessageAutoConfiguration

代码在 config/MessageAutoConfiguration.java，只有 3 个注解：

@AutoConfiguration
@ConditionalOnProperty(prefix = "message", name = "enabled",
    havingValue = "true", matchIfMissing = true)
@EnableConfigurationProperties(MessageProperties.class)
public class MessageAutoConfiguration {
}

这个类本身是空的——没有 @Bean 方法。它的作用全靠注解：

  • @AutoConfiguration：声明这是一个 Spring Boot 3.x 的自动配置类。它会被 AutoConfiguration.imports 文件引用。
  • @ConditionalOnProperty(prefix="message", name="enabled", havingValue="true", matchIfMissing=true)：总开关。如果 yml 中写了 message.enabled: false，这个类不加载，后续所有的 MessageSseConfiguration 和 MessageWebSocketConfiguration 也不会加载（因为它们的 after 依赖它）。matchIfMissing=true 表示如果没配这个属性，默认为 true。
  • @EnableConfigurationProperties(MessageProperties.class)：让 @ConfigurationProperties("message") 标注的 MessageProperties 生效，把 yml 中的 message.* 字段绑定到这个类的属性上。

五、配置绑定：MessageProperties

代码在 properties/MessageProperties.java：

@Data
@ConfigurationProperties("message")
public class MessageProperties {
    private Boolean enabled = true;
    private String transport = MessageTransportEnum.SSE.getCode();
    private String path = "/resource/message";
    private String[] allowedOrigins = {"*"};
    private long sseTimeout = 86_400_000L;          // SSE 超时 24h
    private long heartbeatInterval = 60L;            // 心跳间隔 60s
    private int webSocketSendTimeLimit = 10_000;     // 发送超时 10s
    private int webSocketBufferSizeLimit = 64_000;   // 缓冲区 64KB
}

对应的 yml 配置示例：

message:
  enabled: true
  transport: sse
  path: /resource/message
  allowed-origins: "*"
  sse-timeout: 86400000
  heartbeat-interval: 60
  web-socket-send-time-limit: 10000
  web-socket-buffer-size-limit: 64000

注意 @ConfigurationProperties 支持松弛绑定（Relaxed Binding）：Java 的 webSocketSendTimeLimit（驼峰）对应 yml 的 web-socket-send-time-limit（短横线）。

六、自动发现：AutoConfiguration.imports

三个配置类能被 Spring 找到，靠的是标准路径下的清单文件：

META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports

内容：
org.dromara.common.push.config.MessageAutoConfiguration
org.dromara.common.push.config.MessageSseConfiguration
org.dromara.common.push.config.MessageWebSocketConfiguration

Spring Boot 3.x 启动时会扫描所有 jar 的这个文件，把里面列出的类名都当作自动配置类处理。不需要任何 @Import 或 @ComponentScan——依赖即生效。

但注意：清单中列了三个类，不代表三个都会生效。Spring 会逐一处理它们，当处理到 MessageSseConfiguration 时，会先检查 @ConditionalOnMessageTransport("sse") → 如果配置是 websocket → matches() 返回 false → 这个类跳过。同理 MessageWebSocketConfiguration 也会被检查条件。

七、配置切换实战

从 SSE 切换到 WebSocket，只需要改一行 yml：

# 从
message:
  transport: sse

# 改为
message:
  transport: websocket

重启后发生的事：

  1. MessageAutoConfiguration 加载 → MessageProperties.transport = "websocket"
  2. MessageSseConfiguration 被检查 → matches() 读到 transport=websocket，注解期望 sse → 不匹配 → 跳过，不注册 SSE 相关 Bean
  3. MessageWebSocketConfiguration 被检查 → matches() 读到 transport=websocket，注解期望 websocket → 匹配 → 注册所有 WebSocket Bean
  4. 容器中只有一个 PushSessionManager Bean（类型是 WebSocketSessionManager）
  5. PushHelper 拿到的是 WebSocket 管理器

八、检验一下（提取练习）

Q1. message.enabled: false 时，哪些配置类的 Bean 会被注册？

A. MessageSseConfiguration 的 Bean 仍会注册 B. MessageWebSocketConfiguration 的 Bean 仍会注册 C. 两个通道配置类都不会注册，但 MessageAutoConfiguration 本身可能会加载（仅激活 MessageProperties 绑定） D. 所有配置类都不会加载，包括 MessageAutoConfiguration
看答案D。MessageAutoConfiguration 上有 @ConditionalOnProperty(prefix="message", name="enabled", havingValue="true")，enabled=false 时它不加载。而两个子配置类都声明了 after = MessageAutoConfiguration.class，依赖于它，所以也不会加载。整个推送模块静默关闭。

Q2. MessageTransportCondition.matches() 中用来读取 yml 配置的方法是？

A. context.getBeanFactory().getBean() B. context.getEnvironment().getProperty() C. metadata.getAnnotationAttributes() D. System.getProperty()
看答案B。通过 Spring 的 Environment API 读取配置，能覆盖 yml / 环境变量 / 命令行参数等所有来源。C 是读取注解值的方法，不是读 yml 的。

Q3. @ConfigurationProperties("message") 的松弛绑定指的是什么？

A. 配置值和字段名必须完全一致 B. 驼峰式字段名可以匹配短横线式 yml key（如 webSocketSendTimeLimit ↔ web-socket-send-time-limit） C. 可以不写 yml 配置，全部用默认值 D. 配置值可以为 null
看答案B。Spring Boot 的 Relaxed Binding 支持驼峰、短横线、下划线、全大写环境变量等多种格式互相映射。比如 webSocketSendTimeLimit 可以匹配 web-socket-send-time-limit 或 WEB_SOCKET_SEND_TIME_LIMIT。
