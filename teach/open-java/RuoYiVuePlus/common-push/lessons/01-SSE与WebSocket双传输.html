01 - SSE 与 WebSocket 双传输

🎯 本节目标：理解 SSE 和 WebSocket 的协议差异与适用场景，掌握 @ConditionalOnMessageTransport + MessageTransportCondition 的条件装配机制，能讲清两个通道各自的连接建立与生命周期管理流程。

1. 全景鸟瞰

ruoyi-common-push 包含 17 个 Java 文件，同时支持 SSE 和 WebSocket 两种消息推送协议。模块的核心设计哲学：双通道并存、配置二选一、接口统一抽象。

模块分层结构：

common-push/
├── annotation/  → @ConditionalOnMessageTransport（条件注解）
├── condition/   → MessageTransportCondition（条件判断逻辑）
├── config/      → MessageAutoConfiguration + 两个通道配置类
├── constant/    → MessageConstants（Redis Topic、心跳标识等）
├── core/        → PushSessionManager 接口 + 两个通道管理器
├── dto/         → PushDTO / PushPayloadDTO（数据传输对象）
├── helper/      → PushHelper（业务调用入口）
├── handler/     → PlusWebSocketHandler（WebSocket 事件处理）
├── interceptor/ → PlusWebSocketInterceptor（WS 握手拦截器）
├── listener/    → MessageTopicListener（Redis 消息监听）
├── properties/  → MessageProperties（配置属性类）
└── web/         → SseController（SSE 连接端点）

2. 核心设计

2.1 SSE vs WebSocket：协议差异

维度SSEWebSocket
协议HTTP 长连接（text/event-stream）独立 ws:// 协议（RFC 6455）
通信方向单向（服务端→客户端）全双工（双向）
数据格式仅文本文本 + 二进制
自动重连浏览器内置（EventSource API）需手动实现
穿透代理天然支持（基于 HTTP）需 WebSocket Upgrade 支持
Spring 实现SseEmitterWebSocketHandler 体系
怎么选？如果只有服务端推送通知（如审批提醒、系统公告），SSE 更轻量——复用 HTTP 端口、无需额外协议处理。如果需要双向通信（如在线聊天、协作编辑），必须用 WebSocket。默认配置建议 SSE（大部分管理系统的推送需求是单向的）。

2.2 条件装配机制：二选一如何实现？

三个组件的协作：

@ConditionalOnMessageTransport("sse")
  ↓ 标注在 MessageSseConfiguration 上
  ↓
MessageTransportCondition.matches()
  ↓ 读取 message.enabled 和 message.transport 配置
  ↓ 与注解 value() 比对
  ↓
message.transport = "sse"  → 激活 MessageSseConfiguration
message.transport = "websocket" → 激活 MessageWebSocketConfiguration

公共层不受影响：MessageAutoConfiguration 本身不标注条件——它只管总开关（message.enabled）和配置绑定，无论哪个通道生效都需要它。两个特定通道配置类用 @AutoConfigureAfter(MessageAutoConfiguration.class) 保证顺序。

application.yml:
message:
  enabled: true
  transport: sse           # ← 改成 "websocket" 就切换通道
  path: /message
  sseTimeout: 86400        # SSE 超时（秒）
  heartbeatInterval: 60    # 心跳间隔（秒）

2.3 SSE 通道：SseEmitterSessionManager

核心数据结构：Map<userId, Map<token, SseEmitter>>

连接建立流程：

SseController.connect()
  → LoginHelper.getLoginUser() → 获取当前用户
  → SseEmitter emitter = new SseEmitter(sseTimeout)
  → USER_TOKEN_EMITTERS.computeIfAbsent(userId, k → new ConcurrentHashMap<>())
  → 检查同 token 是否存在旧连接 → 有则 emitter.complete() 关闭旧连接
  → put(token, emitter)
  → emitter.onCompletion(() → 清理)
  → emitter.onTimeout(() → 清理)
  → emitter.onError(e → 清理)
  → 发送连接成功事件
  → 返回 emitter 给 Spring MVC 管理

心跳保活机制：@Scheduled(fixedDelay = heartbeatInterval) 定时任务 sseMonitor() 遍历所有连接，发送空注释行（:）作为心跳。Timeout/Error 的 emitter 自动从 Map 中移除。

消息发送：sendMessage(userId, dto) → 遍历该 userId 下的所有 token emitter → emitter.send(SseEmitter.event().data(dto).name("message"))。如果发送失败（连接已断），自动移除该 token。

2.4 WebSocket 通道：WebSocketSessionManager

核心数据结构：Map<userId, Map<token, WebSocketSession>>，接口与 SSE 管理器几乎对称。

连接建立流程：

1. HTTP Upgrade 握手 → PlusWebSocketInterceptor.beforeHandshake()
   → 从请求中提取 token（URL 参数或 Header）
   → 验证登录状态 → 提取 userId
   → 将 LoginUser 和 token 存入 WebSocketSession attributes

2. 连接建立 → PlusWebSocketHandler.afterConnectionEstablished()
   → 从 session attributes 获取 userId 和 token
   → WebSocketSessionManager.connect(userId, token, session)
   → 驱逐同 token 旧连接 → 发送 KICKED 消息

3. 消息处理 → handleTextMessage(session, message)
   → 检查是否为心跳 PING → 回复 PONG
   → 否则为自定义消息 → 业务处理

4. 连接关闭 → afterConnectionClosed(session, status)
   → WebSocketSessionManager.disconnect(session)
WebSocket 的心跳设计：客户端定时发送 PING 文本消息，服务端回复 PONG。MessageConstants.PING = "PING"、MessageConstants.PONG = "PONG"。这不是 WebSocket 协议的 Ping/Pong 帧（协议层的控制帧是二进制的），而是应用层的心跳——可以携带额外信息（如连接状态）。

2.5 PushSessionManager 统一接口

public interface PushSessionManager {
    void subscribeMessage(Consumer<PushDTO> consumer);  // 订阅 Redis 消息
    void sendMessage(Long userId, PushDTO dto);          // 发送给指定用户
    void publishMessage(PushDTO dto);                    // 发布到 Redis
    void publishAll(PushDTO dto);                        // 广播
}

SSE 和 WebSocket 管理器都实现这个接口，PushHelper 通过 SpringUtils.getBean(PushSessionManager.class) 获取当前激活的实例，对业务代码透明。

3. 实战练习

🧠 提取练习（点击展开）

Q1: 为什么 SSE 需要心跳保活，而 HTTP 本身就有 keep-alive？

参考: HTTP keep-alive 是 TCP 层面的连接复用，不解决应用层的问题。SSE 是长时间无数据流动的连接——客户端和代理服务器可能会因为长时间无数据而关闭连接（典型超时 60-120 秒）。心跳定期发送数据，告诉整条链路「我还活着」，防止被中间代理或负载均衡器断开。

Q2: 同 token 的旧连接被驱逐时为什么要发 KICKED 消息？

参考: KICKED 消息让旧连接的前端知道「你被顶替了」，可以展示友好提示（如"您的账号在其他设备登录"），而不是默默地断连让用户困惑。这是一种用户体验设计——技术上是关闭连接，产品上是告知原因。

Q3: 如果 message.transport 配置既不是 "sse" 也不是 "websocket"，会发生什么？

参考: MessageTransportCondition.matches() 比对不通过，两个通道配置类都不会被激活。但由于 PushHelper 通过 SpringUtils.getBean(PushSessionManager.class) 获取管理器，如果没有任何管理器 Bean 注册，容器启动时就会报 NoSuchBeanDefinitionException。这是一种 fail-fast 设计——配错就启动不了，而不是运行时才发现推送功能不工作。

4. 调用链路

应用启动:
1. AutoConfiguration.imports → MessageAutoConfiguration
2. message.enabled=true → 继续
3. 检查 message.transport
   ├─ "sse" → MessageSseConfiguration 激活
   │           → @Bean SseEmitterSessionManager
   │           → @Bean SseController
   └─ "websocket" → MessageWebSocketConfiguration 激活
                     → @Bean WebSocketSessionManager
                     → @Bean PlusWebSocketHandler
                     → @Bean PlusWebSocketInterceptor
4. MessageTopicListener (ApplicationRunner, Ordered=-1)
   → pushSessionManager.subscribeMessage(consumer)
     → RedisUtils.subscribe(MESSAGE_TOPIC, consumer)
💡 设计洞察：条件装配是 Spring Boot 最强大的扩展机制之一。模块用它实现了「改一行配置就能切换推送协议」的效果——不是简单的 if-else 判断，而是整个 Bean 注册的有无。配合 PushSessionManager 接口，两个通道管理器对外暴露完全相同的 API，业务代码（PushHelper）无需关心底层是 SSE 还是 WebSocket。这是开闭原则（对扩展开放、对修改关闭）在 Spring 自动配置中的经典应用。
