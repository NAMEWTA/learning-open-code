第 2 课 · SSE 通道：SseController、SseEmitterSessionManager、消息推送链路 SSE

🎯 本节目标（对应使命 Success #3）：学完你能讲清 SSE 通道的完整链路：SseController.connect() → SseEmitterSessionManager.connect() 建立连接 → 心跳定时器 sseMonitor() 维持连接 → sendMessage() 发送事件 → onCompletion/onTimeout/onError 自动清理。能解释为什么 prepareSseResponse() 要手动设四个响应头。

一、前端怎么接入？（从全栈视角开始）

SSE 通道的前端入口极其简单——浏览器原生 EventSource 一行搞定：

// 前端代码（示意）
const eventSource = new EventSource('/resource/message');
eventSource.onmessage = (event) => {
    const payload = JSON.parse(event.data);
    console.log('收到消息:', payload.message);
};
// 浏览器会自动处理重连，无需手动干预

这就是 SSE 最大的优点：前端零依赖、浏览器原生支持、自动重连。后端要做的是：提供一个长连接的 GET 端点，持续向这个 EventSource 推送事件。

二、后端入口：SseController

代码在 controller/SseController.java。这个 Controller 非常简洁——只有两个公开端点：

端点HTTP 方法路径（默认）作用
connect()GET/resource/message建立 SSE 长连接
close()GET/resource/message/close手动关闭 SSE 连接

路径是可配置的——通过 @GetMapping(value = "${message.path:/resource/message}") 中的占位符：如果 yml 里配了 message.path 就用配的值，否则用默认值 /resource/message。

关键看 connect() 方法的代码结构：

@GetMapping(value = "${message.path:/resource/message}",
            produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter connect(HttpServletResponse response) {
    // ① 设置 SSE 响应头
    prepareSseResponse(response);

    // ② 取当前登录用户信息
    String tokenValue = StpUtil.getTokenValue();
    Long userId = LoginHelper.getUserId();

    // ③ 委托给 SessionManager 建立连接
    return sessionManager.connect(userId, tokenValue);
}

三个步骤，分别对应三个设计要点：

1 响应头设置（prepareSseResponse）：SSE 要求 Content-Type: text/event-stream，而且因为框架层统一拦截器可能已经设置了 application/json，这里必须强制覆盖。X-Accel-Buffering: no 是针对 Nginx 反向代理——告诉 Nginx 不要缓冲这个长连接。
2 鉴权：StpUtil.getTokenValue() 从请求中提取 Sa-Token，LoginHelper.getUserId() 从当前会话中取 userId。这意味着 SSE 连接必须登录后才能建立——未登录请求会被 Sa-Token 拦截器拦在 Controller 之外。
3 委托：Controller 自己不管理连接，直接委托给 SseEmitterSessionManager.connect(userId, token)。这是典型的「瘦 Controller + 胖 Manager」分层。
💡 注意：Controller 的 @GetMapping 上写了 produces = MediaType.TEXT_EVENT_STREAM_VALUE。这告诉 Spring MVC 这个端点返回的是 SSE 流而非普通 JSON——Spring 将使用异步请求处理，不会阻塞 Tomcat 线程。

三、核心：SseEmitterSessionManager.connect()

这是 SSE 通道的灵魂方法。代码在 core/SseEmitterSessionManager.java。我们把它拆成 6 个步骤看：

1 获取或创建用户的 token→emitter 映射表：USER_TOKEN_EMITTERS.computeIfAbsent(userId, k -> new ConcurrentHashMap<>())。注意外层 Map 的泛型是 Map<Long, Map<String, SseEmitter>>，即一个 userId 可以对应多个 emitter（多个 token = 多个设备同时在线）。
2 顶替旧连接：SseEmitter oldEmitter = emitters.remove(token) → 如果存在同 token 的旧 emitter，先发一条 KICKED 消息通知旧连接，再 oldEmitter.complete() 关闭它。这保证了「一个 token 只有一个活跃 SSE 连接」。
3 创建新 SseEmitter：new SseEmitter(messageProperties.getSseTimeout())。默认超时是 86_400_000 毫秒 = 24 小时。这个超时值设得很大，因为前端 EventSource 会自动重连——超时不太可能发生。
4 注册三个生命周期回调：onCompletion / onTimeout / onError——三个回调做的事情完全一样：从 emitters 中移除当前 token。这是在「无论什么原因断开，都清理内存」。
5 发送连接成功事件：emitter.send(SseEmitter.event().comment("connected"))。注意用的是 comment() 而不是 data()——comment 在 SSE 协议中会被浏览器 EventSource 忽略，不会触发 onmessage，只是一个「连接建立」的信号。
6 返回 emitter：Spring MVC 会把返回的 SseEmitter 注册到异步请求上下文中，从此这个 HTTP 连接保持打开，服务端随时可以向 emitter 写事件。

四、心跳保活：sseMonitor()

SSE 长连接可能因为网络问题变得「半死不活」——TCP 连接还在，但数据发不出去了。sseMonitor() 定时巡检所有连接：

// 构造函数中启动定时任务
scheduledExecutorService.scheduleWithFixedDelay(
    this::sseMonitor,
    messageProperties.getHeartbeatInterval(),  // 默认 60 秒
    messageProperties.getHeartbeatInterval(),
    TimeUnit.SECONDS
);

// sseMonitor 核心逻辑
public void sseMonitor() {
    final SseEmitter.SseEventBuilder heartbeat =
        SseEmitter.event().comment("heartbeat");
    List<Long> toRemoveUsers = new ArrayList<>();

    USER_TOKEN_EMITTERS.forEach((userId, emitterMap) -> {
        emitterMap.entrySet().removeIf(entry -> {
            try {
                entry.getValue().send(heartbeat);
                return false;  // 发送成功 → 保留
            } catch (Exception ex) {
                entry.getValue().complete();
                return true;   // 发送失败 → 移除
            }
        });
        if (emitterMap.isEmpty()) {
            toRemoveUsers.add(userId);
        }
    });
    toRemoveUsers.forEach(USER_TOKEN_EMITTERS::remove);
}

设计细节：

  • 先发送 comment 心跳，再看是否抛异常来判断连接是否存活——这比用 isOpen() 之类的状态检查更可靠，因为 TCP 半开连接的状态位可能是 OPEN 但实际数据发不出去。
  • 收集再统一删除：用 toRemoveUsers 列表收集，循环结束后再 forEach remove，避免在 forEach 中直接修改外层 Map 导致 ConcurrentModificationException。
  • 心跳也是 comment：和连接成功事件一样，心跳用 comment("heartbeat")，浏览器不会触发 onmessage，对前端透明。

五、消息发送：sendMessage() 的三个重载

SseEmitterSessionManager 有多个 sendMessage 重载，可以按目标维度分类：

方法目标消息格式实现方式
sendMessage(Long userId, String message)指定用户纯文本字符串遍历该用户的 token→emitter 映射，逐一 emitter.send(event)
sendMessage(Long userId, PushPayloadDTO payload)指定用户JSON（先 JsonUtils.toJsonString）调用上面的 String 版本
sendMessage(Long userId, PushDTO pushDTO)指定用户JSON从 PushDTO 取 payload，调用 String 版本
sendMessage(String message)所有本地用户纯文本收集所有 userId，用 ThreadUtils.virtualInvokeAll 并发发送
sendMessage(PushPayloadDTO payload)所有本地用户JSON先 JSON 序列化，再调用 String 广播版本

核心发送代码：

// 向指定用户的所有 SSE 会话发送
public void sendMessage(Long userId, String message) {
    Map<String, SseEmitter> emitters = USER_TOKEN_EMITTERS.get(userId);
    if (MapUtil.isNotEmpty(emitters)) {
        for (Map.Entry<String, SseEmitter> entry : emitters.entrySet()) {
            try {
                entry.getValue().send(SseEmitter.event()
                    .name("message")
                    .data(message));
            } catch (Exception e) {
                // 发送失败 → 移除这个 emitter
                SseEmitter remove = emitters.remove(entry.getKey());
                if (remove != null) {
                    remove.complete();
                }
            }
        }
    } else {
        USER_TOKEN_EMITTERS.remove(userId);
    }
}

注意 .name("message")——这对应前端 EventSource.addEventListener('message', ...)。如果不设 name，浏览器用默认的 onmessage 处理。

💡 sendMessage vs publishMessage：sendMessage 只发送给本节点的 SSE 连接。如果没有跨节点需求，直接调 sendMessage 即可。需要跨节点时，调 publishMessage（第 5 课细讲），它通过 Redis Pub/Sub 让所有节点都收到。

六、手动断开：disconnect()

public void disconnect(Long userId, String token) {
    if (userId == null || token == null) {
        return;
    }
    Map<String, SseEmitter> emitters = USER_TOKEN_EMITTERS.get(userId);
    if (MapUtil.isNotEmpty(emitters)) {
        try {
            SseEmitter sseEmitter = emitters.get(token);
            sseEmitter.send(SseEmitter.event().comment("disconnected"));
            sseEmitter.complete();
        } catch (Exception ignore) {}
        emitters.remove(token);
    } else {
        USER_TOKEN_EMITTERS.remove(userId);
    }
}

和自动清理不同，手动断开会先发一条 comment("disconnected") 通知前端「是服务端主动关的」，再 complete() 关闭连接。

七、检验一下（提取练习）

Q1. prepareSseResponse() 设置了哪些响应头？（可多选）

A. Content-Type: text/event-stream B. Cache-Control: no-cache C. X-Accel-Buffering: no D. Content-Type: application/json
看答案A、B、C。还设置了 CharacterEncoding=UTF-8。D 是错的——SSE 的 Content-Type 必须是 text/event-stream，这个方法是用来覆盖框架默认的 JSON 响应的。

Q2. 同一用户同一 token 建立新 SSE 连接时，旧连接会收到什么消息？

A. "disconnected" B. "heartbeat" C. "kicked" D. 不会收到任何消息，直接断开
看答案C。sendKickedMessage() 向旧 emitter 发送 name("message").data("kicked")，告诉前端「你被同 token 的新连接顶替了」。

Q3. emitter.send(SseEmitter.event().comment("xxx")) 和 .data("xxx") 的区别是什么？

A. comment 会触发前端 onmessage，data 不会 B. comment 不会被前端 EventSource 触发回调，data 会 C. 没有区别 D. comment 只能发一次，data 可以发多次
看答案B。在 SSE 协议中，comment 行以冒号开头，浏览器 EventSource 会忽略它，不触发任何事件回调。本模块用 comment 发心跳和连接/断开通知，就是利用了这个特性——前端感知不到心跳。
