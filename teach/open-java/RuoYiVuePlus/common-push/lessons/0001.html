一、模块全景：17 个文件，5 个包，1 个接口

模块共有 17 个 Java 文件，组织在 6 个子包里。每个文件的角色如下：

ruoyi-common-push/src/main/java/org/dromara/common/push/ │ ├── annotation/ │ └── ConditionalOnMessageTransport.java ← 【条件注解】声明期望的传输方式 │ ├── condition/ │ └── MessageTransportCondition.java ← 【条件判断】读配置 vs 注解做匹配 │ ├── config/ │ ├── MessageAutoConfiguration.java ← 【公共装配】总开关 + 配置绑定 │ ├── MessageSseConfiguration.java ← 【SSE 装配】注册 SSE 三件套 │ └── MessageWebSocketConfiguration.java ← 【WS 装配】注册 WS 四件套 │ ├── constant/ │ └── MessageConstants.java ← 【常量】topic / ping / pong / kicked │ ├── controller/ │ └── SseController.java ← 【SSE 入口】GET /resource/message │ ├── core/ │ ├── PushSessionManager.java ← 【统一接口】subscribe / send / publish │ ├── SseEmitterSessionManager.java ← 【SSE 实现】Map<userId, Map<token, SseEmitter>> │ └── WebSocketSessionManager.java ← 【WS 实现】Map<userId, Map<token, WebSocketSession>> │ ├── dto/ │ └── PushDTO.java ← 【传输对象】userIds + payload │ ├── enums/ │ └── MessageTransportEnum.java ← 【枚举】SSE / WEBSOCKET │ ├── handler/ │ └── PlusWebSocketHandler.java ← 【WS 处理器】afterConnectionEstablished 等 │ ├── helper/ │ └── PushHelper.java ← 【业务入口】全静态工具类 │ ├── interceptor/ │ └── PlusWebSocketInterceptor.java ← 【WS 握手】提取登录信息到 attributes │ ├── listener/ │ └── MessageTopicListener.java ← 【Redis 监听】集群消息分发 │ └── properties/ └── MessageProperties.java ← 【配置】message.* 绑定

二、核心设计：双通道互斥选一

模块同时支持 SSE 和 WebSocket，但运行时只有一个通道生效。这个选择由 application.yml 中的一行配置决定：

message:
  enabled: true          # 总开关
  transport: sse         # sse 或 websocket

这个互斥机制是这样实现的：

1 MessageSseConfiguration 上贴了 @ConditionalOnMessageTransport("sse") config/MessageSseConfiguration.java
2 MessageWebSocketConfiguration 上贴了 @ConditionalOnMessageTransport("websocket") config/MessageWebSocketConfiguration.java
3 MessageTransportCondition.matches() 读 message.transport 配置，与注解 value() 比对 condition/MessageTransportCondition.java
4 比对命中 → 该配置类的所有 @Bean 注册到 Spring 容器；不命中 → 整个配置类跳过
💡 为什么互斥而不是同时启用？
SSE 走 HTTP 长连接、WebSocket 走独立协议升级。同一用户同时保持两种连接会增加服务端线程和内存消耗，且消息投递逻辑需要区分为谁走哪条通道。互斥设计简化了这个问题：模块只有一个 PushSessionManager Bean，PushHelper 拿到的永远是当前生效的那个。

三、一个接口，两种实现：PushSessionManager

模块的核心抽象是 PushSessionManager 接口：

public interface PushSessionManager { void subscribeMessage(Consumer<PushDTO> consumer); // 订阅 Redis 消息 void sendMessage(Long userId, PushPayloadDTO payload); // 单发 void sendMessage(PushPayloadDTO payload); // 广播 void publishMessage(PushDTO pushDTO); // 发布到 Redis（跨节点） void publishAll(PushPayloadDTO payload); // 广播到 Redis（跨节点） }

SseEmitterSessionManager 和 WebSocketSessionManager 都实现这个接口。这就是 PushHelper 能对 SSE 和 WebSocket 用同一套代码调用的原因。

四、消息流转的完整链路（跨节点）

这是模块最精彩的设计——从业务代码调用到前端收到消息的完整路径（以 SSE 为例）：

【节点 A：消息源】 【Redis】 【节点 B：目标用户在的节点】 ───────────────── ──────── ───────────────────────────── 业务代码 │ ▼ PushHelper.publishMessage(userIds, payload) │ isEnabled() 检查 → 通过 ▼ PushSessionManager.publishMessage(dto) │ (实际是 SseEmitterSessionManager) ▼ RedisUtils.publish("global:message", dto) │ ─────► "global:message" 频道 │ │ │ ▼ │ MessageTopicListener.run() 中的 consumer │ │ (ApplicationRunner, order=-1) │ ▼ │ 判断 userIds： │ ├─ 非空 → forEach(userId → │ │ pushSessionManager.sendMessage(userId, payload)) │ └─ 空 → pushSessionManager.sendMessage(payload) // 广播 │ │ │ ▼ │ SseEmitterSessionManager.sendMessage(userId, message) │ │ 从 USER_TOKEN_EMITTERS 取出 Map<token, SseEmitter> │ ▼ │ emitter.send(SseEmitter.event().name("message").data(json)) │ │ │ ▼ │ 浏览器 EventSource.onmessage 收到

关键洞察：

  • publishMessage（发布到 Redis）和 sendMessage（直接发送）的分工：publish 解决跨节点，send 解决本节点投递。
  • MessageTopicListener 监听的是 global:message 这个 Redis 频道——集群里所有节点的所有推送管理器都监听同一个频道。
  • 收到消息后，每个节点各自检查目标用户是否连接在本节点上——连接了则发送，没连则跳过。这就实现了「消息精确投递到目标用户所在节点」。
💡 如果消息源和目标用户恰好在同一个节点呢？
Redis Pub/Sub 默认不会把消息投递回发布者自己。但 MessageTopicListener 在每个节点（包括发布者自己）都注册了消费者，所以同节点内也会收到。同节点场景下，消息路径是：RedisUtils.publish → Redis 转发 → 本节点 MessageTopicListener 消费 → sendMessage → 用户。

五、SSE vs WebSocket：选型指南

两个通道都能推送消息，但适用场景不同。本模块帮你屏蔽了这个选择——改一行配置即可切换：

维度SSEWebSocket
通信方向服务端 → 客户端（单向）双向（客户端也能主动发消息）
协议层HTTP 长连接（text/event-stream）独立 ws:// 协议（HTTP Upgrade）
断线重连浏览器 EventSource 自动重连需要手动实现重连逻辑
消息格式纯文本文本 + 二进制
前端 APInew EventSource(url)new WebSocket(url)
本模块独有GET 端点，用 Sa-Token 鉴权握手拦截器提取登录信息
💡 一句话选型：如果你的前端只需要「收通知」（公告、审批结果、系统消息），用 SSE 足够且更简单；如果前端还需要「主动发消息给服务端」（聊天、协同编辑），必须 WebSocket。RuoYi 默认用 SSE，配置 message.transport=sse。

六、检验一下（提取练习）

Q1. @ConditionalOnMessageTransport("sse") 贴在哪个配置类上？

A. MessageAutoConfiguration B. MessageWebSocketConfiguration C. MessageSseConfiguration D. SseEmitterSessionManager
看答案C。MessageSseConfiguration 上贴 @ConditionalOnMessageTransport("sse")，MessageWebSocketConfiguration 上贴 @ConditionalOnMessageTransport("websocket")。MessageAutoConfiguration 不含这个注解——它是公共配置，始终生效。

Q2. 模块通过什么机制实现集群跨节点消息投递？

A. 每个节点轮询数据库消息表 B. 通过 Nginx 广播到所有节点 C. Redis Pub/Sub + MessageTopicListener 监听 global:message 频道 D. WebSocket 自带集群支持
看答案C。publishMessage() 通过 RedisUtils.publish("global:message", dto) 发布，MessageTopicListener（项目启动时注册）收到后在本节点 sendMessage() 给目标用户。

Q3. 两个 SessionManager 的内部存储结构 Map<Long, Map<String, Xxx>> 的外层 key 和内层 key 分别是什么？

A. 外层 sessionId，内层 userId B. 外层 userId，内层 token C. 外层 token，内层 sessionId D. 外层 userId，内层 sessionId
看答案B。外层 key 是 userId（Long），内层 key 是 token（String）。这个结构允许一个用户同时多终端登录（不同 token 对应不同连接），并且同 token 的新连接会顶替旧连接。
