🎯 本节目标（对应使命 Success #5、#6）：学完你能完整追踪「业务发送消息 → 前端收到」的跨节点链路，并能在业务代码中正确调用 PushHelper 推送消息。本节是综合课，把前 4 课的知识串成一条完整链路。

一、回顾：模块的两套消息传递机制

前 4 课我们拆开看了每个组件。现在把它们拼回整体。模块提供两种消息发送方式，适用于不同场景：

方法路径适用场景跨节点？
sendMessage()直接发送到本节点连接单节点部署或确定目标用户在本节点否
publishMessage()发布到 Redis Pub/Sub → 所有节点各自发送集群部署，不确定目标用户在哪个节点是
💡 经验法则：在集群环境下总是用 publish 系列方法（publishMessage / publishAll）。因为你不确定目标用户的 SSE/WS 连接落在哪个节点上。单节点部署下 publish 也能正常工作——消息走 Redis 绕一圈回来，同节点的 MessageTopicListener 也能收到。

二、跨节点推送完整链路（逐帧回放）

假设集群有 Node-A 和 Node-B，目标用户连接在 Node-B 上，消息从 Node-A 的业务代码发出：

Node-A Redis Node-B ────── ───── ────── 业务代码 PushHelper.publishMessage(userIds, payload) │ isEnabled() → true ▼ getSessionManager() │ SpringUtils.getBean(PushSessionManager.class) │ → SseEmitterSessionManager (或 WebSocketSessionManager) ▼ sessionManager.publishMessage(dto) │ SseEmitterSessionManager (Node-A) │ ▼ RedisUtils.publish("global:message", dto) │ Redisson RTopic.publish(dto) ──────────► global:message 频道 │ │ │ ▼ │ MessageTopicListener (Node-B) │ │ ApplicationRunner.run() │ │ 启动时注册的 consumer │ │ │ ▼ │ 收到 dto: {userIds:[1001], payload:{...}} │ │ CollUtil.isNotEmpty(userIds) → true │ ▼ │ forEach userId → │ pushSessionManager.sendMessage(1001, payload) │ │ Node-B 的 SseEmitterSessionManager │ ▼ │ USER_TOKEN_EMITTERS.get(1001) │ │ → Map{token1→emitter1, token2→emitter2} │ ▼ │ emitter1.send(event.name("message").data(json)) │ │ │ ┌─────────────────────┘ (日志) │ ▼ │ MessageTopicListener (Node-A) Node-B 的 emitter2.send(...) │ 也收到了这条消息 │ │ userIds 包含 1001 ▼ │ 但 1001 不在 Node-A 上 用户浏览器 EventSource.onmessage │ → USER_TOKEN_EMITTERS.get(1001) → 收到 {type:"message", source:"backend", │ 返回 null message:"您的审批已通过", ...} │ → 跳过

关键洞察：

  1. Redis Pub/Sub 是广播模型——一条消息发布后，所有订阅该频道的节点都会收到。不是「精准路由」。
  2. 每个节点自己判断——收到消息后检查 userIds，遍历检查每个 userId 是否在本节点有连接。有则发，无则跳过。
  3. 发布者自己也会收到——Redisson 的 RTopic.publish 默认不会把消息投递回发布者线程，但不阻止同一个节点的其他 Consumer 收到。在本模块中，Node-A 的 MessageTopicListener 同样会消费这条消息。

三、MessageTopicListener：跨节点的桥梁

代码在 listener/MessageTopicListener.java。它是一名 ApplicationRunner + Ordered：

@Slf4j
@RequiredArgsConstructor
public class MessageTopicListener implements ApplicationRunner, Ordered {

    private final PushSessionManager pushSessionManager;

    @Override
    public void run(ApplicationArguments args) {
        // 注册消息消费者——一个 lambda
        pushSessionManager.subscribeMessage(message -> {
            if (message == null || message.getPayload() == null) {
                return;
            }
            log.info("消息主题订阅收到消息userIds={} message={}",
                message.getUserIds(), message.getPayload().getMessage());

            if (CollUtil.isNotEmpty(message.getUserIds())) {
                // 有目标用户 → 逐一发送
                message.getUserIds().forEach(userId ->
                    pushSessionManager.sendMessage(userId, message.getPayload()));
            } else {
                // 无目标用户（userId 为空）→ 广播
                pushSessionManager.sendMessage(message.getPayload());
            }
        });
        log.info("初始化消息主题订阅监听器成功");
    }

    @Override
    public int getOrder() {
        return -1;  // 最高优先级，确保消息订阅最早初始化
    }
}

关键设计：

  • ApplicationRunner：在 Spring Boot 启动完成后执行 run()。此时所有 Bean 都已就绪，可以安全地注册 Redis 订阅。
  • Ordered(-1)：优先级最高，确保在所有业务 ApplicationRunner 之前完成消息订阅的初始化。如果别的 Runner 中就要发消息，这里的订阅必须已经就绪。
  • 消费者逻辑是 lambda：传入了 pushSessionManager.subscribeMessage(consumer)。这个 consumer 被 RedisUtils.subscribe() 调用，实际运行在 Redisson 的线程池中。

四、subscribeMessage 的底层：RedisUtils.subscribe

PushSessionManager.subscribeMessage() 的实现在两个管理器中完全相同：

// SseEmitterSessionManager 和 WebSocketSessionManager 的 subscribeMessage 实现
@Override
public void subscribeMessage(Consumer<PushDTO> consumer) {
    RedisUtils.subscribe(MessageConstants.MESSAGE_TOPIC, PushDTO.class, consumer);
}

这调用了 RedisUtils（来自 ruoyi-common-redis 模块）的订阅方法，底层是 RTopic.addListener()。PS：Redis Pub/Sub 消息不持久化——如果消息发布时某个节点恰好没在监听（重启中），那段时间的消息就丢了。这正是它适合「实时推送」而非「消息队列」的原因。

五、publishMessage：发布端的实现

两个管理器的 publishMessage() 实现也几乎相同（以 SSE 为例）：

@Override
public void publishMessage(PushDTO pushDTO) {
    if (pushDTO == null || pushDTO.getPayload() == null) {
        return;
    }
    RedisUtils.publish(MessageConstants.MESSAGE_TOPIC, pushDTO, consumer ->
        log.info("发送主题订阅消息topic:{} userIds:{} message:{}",
            MessageConstants.MESSAGE_TOPIC,
            pushDTO.getUserIds(),
            pushDTO.getPayload() == null ? null : pushDTO.getPayload().getMessage()
        ));
}

注意 RedisUtils.publish 的第三个参数是一个 Consumer——这是发布成功后的回调，用于日志记录。它不是消息消费者，消息消费者是通过 subscribeMessage 注册的那个 lambda。

六、PushHelper：业务层的唯一入口

代码在 helper/PushHelper.java。它是一个全静态方法的工具类（私有构造器），业务代码只需要：

// 示例 1：给指定用户发一条文本消息
PushHelper.sendMessage(1001L, "您的审批已通过");

// 示例 2：给指定用户发结构化的 JSON 消息
PushPayloadDTO payload = PushPayloadDTO.of(
    PushTypeEnum.MESSAGE,
    PushSourceEnum.BACKEND,
    "您的审批已通过",
    Map.of("orderId", 12345)
);
PushHelper.sendMessage(1001L, payload);

// 示例 3：发布到集群（推荐集群环境使用）
PushHelper.publishMessage(List.of(1001L, 1002L), payload);

// 示例 4：全局广播
PushHelper.publishAll("系统将于 30 分钟后维护");

// 示例 5：用完整 PushDTO 发布
PushDTO dto = PushDTO.of(List.of(1001L), payload);
PushHelper.publishMessage(dto);

PushHelper 内部的两个关键方法：

// 获取当前生效的会话管理器
private static PushSessionManager getSessionManager() {
    return SpringUtils.getBean(PushSessionManager.class);
}

// 检查功能是否开启
public static boolean isEnabled() {
    return Boolean.TRUE.equals(
        SpringUtils.getProperty("message.enabled", Boolean.class, Boolean.TRUE));
}

每个公开方法开头都先调 isEnabled()——如果推送功能关闭，整条链路就此打住，不会抛异常也不会空指针。这是防御性设计的关键一环。

💡 sendMessage vs publishMessage 如何选择？
sendMessage 只发给本节点的连接——适合单节点部署或你知道用户就在本节点。publishMessage 走 Redis →所有节点各自投递——适合集群部署。集群环境推荐用 publish，性能损失极小（Redis Pub/Sub 是纯内存操作），换来正确性。

七、PushDTO：消息的载体

代码在 dto/PushDTO.java：

@Data
public class PushDTO implements Serializable {
    private List<Long> userIds;       // null = 广播
    private PushPayloadDTO payload;   // 消息体

    public static PushDTO of(List<Long> userIds, PushPayloadDTO payload) {
        // ... 普通推送
    }

    public static PushDTO broadcast(PushPayloadDTO payload) {
        return of(null, payload);     // userIds=null → 广播
    }
}

而 PushPayloadDTO（在 ruoyi-api 模块中）是推送给前端的统一 JSON 结构：

{
  "messageId": null,
  "type": "message",        // message / warning / notification 等
  "source": "backend",      // backend / client
  "message": "您的审批已通过",
  "data": null,             // 扩展数据，可以是任意对象
  "path": null,             // 前端跳转路径
  "timestamp": 1751683200000
}

这个结构是前后端约定好的——前端 EventSource.onmessage 或 WebSocket.onmessage 收到的就是这段 JSON。前端根据 type 决定渲染样式，根据 path 决定点击跳转。

八、综合练习：追踪完整链路

Q1. 在集群环境下，PushHelper.publishMessage(List.of(1001), payload) 调用了 Redis 的什么频道？

A. "message" B. "global:message" C. "push:message" D. "rtopic:message"
看答案B。MessageConstants.MESSAGE_TOPIC = "global:message"。所有节点都监听这个频道，收到后各自检查目标用户是否在本节点。

Q2. MessageTopicListener 的 getOrder() 返回 -1，为什么？

A. 没有特别原因，随便写的 B. 确保消息订阅在所有 ApplicationRunner 中最早初始化，防止其他 Runner 发消息时订阅还没就绪 C. 确保最后初始化 D. -1 表示禁用这个 Listener
看答案B。Spring 的 Ordered 接口值越小优先级越高。如果有业务 ApplicationRunner 在启动时就要发消息，必须先确保 MessageTopicListener 的订阅已经注册好。

Q3. 如果目标用户不在线（没有建立 SSE/WS 连接），publishMessage 会怎样？

A. 抛出异常 B. 消息被 Redis 缓存，用户上线后补发 C. 消息发送到 Redis，各节点检查后发现没有该用户的连接，静默跳过——消息丢失 D. 消息存入数据库等待重试
看答案C。Redis Pub/Sub 不持久化消息。如果目标用户不在线，每个节点的 sendMessage 都找不到连接，消息就此丢失。这是推送系统的设计取舍——如果需要离线消息，应该配合数据库消息表实现。

Q4. 以下哪个方法是 PushHelper 中用来检查推送总开关的？

A. PushHelper.checkEnabled() B. PushHelper.isEnabled() C. PushHelper.validate() D. PushHelper.canPush()
看答案B。PushHelper.isEnabled() 通过 SpringUtils.getProperty("message.enabled") 检查。所有公开方法（sendMessage / publishMessage / publishAll）开头都调它。enabled=false 时所有方法静默返回。
