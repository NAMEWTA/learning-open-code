一、WebSocket 与 SSE 的本质差异

在深入代码前，先建立直觉：WebSocket 通道和 SSE 通道在建立连接的方式上根本不同。

SSEWebSocket
建立连接普通 HTTP GET 请求HTTP Upgrade 握手 → ws:// 协议
入口@GetMapping ControllerWebSocketHandler 接口
鉴权Sa-Token 统一拦截器握手拦截器手动提取
会话对象SseEmitter（Spring 包装）WebSocketSession（Spring 包装）
消息接收不支持（单向）handleTextMessage()
连通性检测心跳发 comment 事件心跳 Ping/Pong 帧
前端入口new EventSource(url)new WebSocket(url)
💡 核心差异：SSE 走的是普通 HTTP 请求链路（经过 Filter → Interceptor → Controller），鉴权天然可用 Sa-Token 拦截器。WebSocket 的 HTTP Upgrade 握手完成后就脱离了 HTTP 层，所以必须在握手阶段（HandshakeInterceptor）就把用户信息提取到 WebSocketSession 的 attributes 里，后续消息处理全靠这些 attributes。

二、握手阶段：PlusWebSocketInterceptor

代码在 interceptor/PlusWebSocketInterceptor.java。这是一个 HandshakeInterceptor 的实现，核心逻辑全在 beforeHandshake()：

public boolean beforeHandshake(ServerHttpRequest request,
                               ServerHttpResponse response,
                               WebSocketHandler wsHandler,
                               Map<String, Object> attributes) {
    // 从当前线程的 Sa-Token 会话中取用户信息
    LoginUser loginUser = LoginHelper.getLoginUser();
    String tokenValue = StpUtil.getTokenValue();

    // 塞进 WebSocketSession 的属性表
    attributes.put(MessageConstants.LOGIN_USER_KEY, loginUser);
    attributes.put(MessageConstants.LOGIN_TOKEN_KEY, tokenValue);

    return true;  // 允许握手
}

这里有一个巧妙的先决条件：WebSocket 握手请求首先经过 HTTP 层的 Sa-Token 拦截器。握手请求本身是一个 HTTP GET（带 Upgrade: websocket 头），所以会经过 `StpUtil` 的统一鉴权。鉴权通过后，`LoginHelper.getLoginUser()` 和 `StpUtil.getTokenValue()` 才能正确取到值。

然后这些值被放入 attributes Map，这个 Map 最终会成为 WebSocketSession.getAttributes()——在后续的 PlusWebSocketHandler 中，所有方法都通过 session.getAttributes().get(LOGIN_USER_KEY) 获取用户信息。

💡 为什么 afterHandshake() 是空的？因为用户信息已经在 beforeHandshake 阶段完全提取完毕了。握手后不需要额外处理。afterHandshake 的存在只是为了满足 HandshakeInterceptor 接口的两个方法都要实现。

三、全生命周期处理器：PlusWebSocketHandler

代码在 handler/PlusWebSocketHandler.java。它继承 AbstractWebSocketHandler，覆盖了 WebSocket 连接的全部生命周期方法。

3.1 连接建立：afterConnectionEstablished()

public void afterConnectionEstablished(WebSocketSession session) throws IOException {
    // ① 从 session attributes 中取出握手时存入的用户信息
    LoginUser loginUser = (LoginUser) session.getAttributes()
        .get(MessageConstants.LOGIN_USER_KEY);
    String token = (String) session.getAttributes()
        .get(MessageConstants.LOGIN_TOKEN_KEY);

    // ② 校验——如果信息不全，直接拒绝连接
    if (ObjectUtil.hasNull(loginUser, token)) {
        session.close(CloseStatus.BAD_DATA);
        return;
    }

    // ③ 注册到 WebSocketSessionManager
    webSocketSessionManager.connect(
        loginUser.getUserId(),
        token,
        new ConcurrentWebSocketSessionDecorator(
            session,
            messageProperties.getWebSocketSendTimeLimit(),    // 10000ms
            messageProperties.getWebSocketBufferSizeLimit()   // 64000 bytes
        )
    );
}

这里的 ConcurrentWebSocketSessionDecorator 值得注意——它是 Spring 提供的并发安全包装器，给原生 WebSocketSession 加上发送超时和缓冲区限制。因为 WebSocket 连接可能被多个线程同时写（定时心跳 + 消息推送），不用这个包装器会有并发问题。

3.2 消息接收：handleTextMessage()

这是 WebSocket 通道独有的能力——SSE 做不到的双向通信：

protected void handleTextMessage(WebSocketSession session, TextMessage message) {
    LoginUser loginUser = (LoginUser) session.getAttributes()
        .get(MessageConstants.LOGIN_USER_KEY);
    if (ObjectUtil.isNull(loginUser)) {
        return;  // 未登录用户的会话不处理
    }

    // 心跳：ping → pong
    if (MessageConstants.PING.equalsIgnoreCase(message.getPayload())) {
        webSocketSessionManager.sendMessage(session, MessageConstants.PONG);
        return;
    }

    // 自定义消息：构建 PushDTO 并发布
    PushDTO dto = PushDTO.of(
        List.of(loginUser.getUserId()),
        PushPayloadDTO.of(
            PushTypeEnum.CUSTOM,
            PushSourceEnum.CLIENT,
            message.getPayload(),
            null
        ));
    webSocketSessionManager.publishMessage(dto);
}

两类消息处理：

  • 心跳：前端发 "ping"，服务端回 "pong"。这是应用层心跳，走 TextMessage，区别于 WebSocket 协议层的 Ping/Pong 帧。
  • 自定义消息：客户端主动发送的任何非 ping 文本，都会被包装成 PushDTO（类型 = CUSTOM，来源 = CLIENT），然后走 publishMessage 发布到 Redis。这意味着客户端发给服务端的消息，也可以跨节点投递给其他用户。

3.3 连接关闭：afterConnectionClosed()

public void afterConnectionClosed(WebSocketSession session, CloseStatus status) {
    LoginUser loginUser = (LoginUser) session.getAttributes()
        .get(MessageConstants.LOGIN_USER_KEY);
    String token = (String) session.getAttributes()
        .get(MessageConstants.LOGIN_TOKEN_KEY);

    if (ObjectUtil.hasNull(loginUser, token)) {
        return;
    }
    webSocketSessionManager.disconnect(loginUser.getUserId(), token);
}

简单直接——从 attributes 中取回用户信息，调用 disconnect() 从 WebSocketSessionManager 中移除。

3.4 异常处理：handleTransportError()

public void handleTransportError(WebSocketSession session, Throwable exception) {
    log.error("[transport error] sessionId: {}, exception:{}",
        session.getId(), exception.getMessage(), exception);
    // 有关联用户→断开该用户的指定会话
    // 无关用户→直接关闭 session
}

四、会话管理：WebSocketSessionManager

代码在 core/WebSocketSessionManager.java。它的结构和 SseEmitterSessionManager 高度对称：

方法作用SSE 对等体
connect(userId, token, session)注册会话，顶替旧同 token 连接connect(userId, token)（返回 emitter）
disconnect(userId, token, status)关闭并移除会话disconnect(userId, token)
sessionMonitor()定时清理已关闭的会话sseMonitor()
sendMessage(userId, payload)向指定用户发送 JSON 消息同名方法
sendMessage(payload)向所有本地用户广播同名方法
publishMessage(pushDTO)发布到 Redis Pub/Sub同名方法
sendPongMessage(session)回复 Pong 心跳无（SSE 用 comment 心跳）

注意几点不同：

  • WebSocket 的 connect() 接收外部传入的 session（在 Handler 里已经包了 ConcurrentWebSocketSessionDecorator），而 SSE 的 connect() 自己在内部 new SseEmitter()。
  • WebSocket 的 sessionMonitor() 通过 session.isOpen() 判断连接状态，而 SSE 的 sseMonitor() 通过发送心跳检测——因为 WebSocket 有明确的状态 API。
  • WebSocket 的 sendMessage 返回 boolean（发送成功/失败），SSE 版本不返回——这是为了实现「发送失败自动移除会话」的设计。

五、WebSocketSessionManager 的 sessionMonitor

public void sessionMonitor() {
    List<Long> toRemoveUsers = new ArrayList<>();
    USER_TOKEN_SESSIONS.forEach((userId, sessionMap) -> {
        if (CollUtil.isEmpty(sessionMap)) {
            toRemoveUsers.add(userId);
            return;
        }
        // 移除所有已关闭的会话
        sessionMap.entrySet().removeIf(entry -> {
            WebSocketSession session = entry.getValue();
            if (session == null || !session.isOpen()) {
                closeSession(session, CloseStatus.NORMAL);
                return true;
            }
            return false;
        });
        if (sessionMap.isEmpty()) {
            toRemoveUsers.add(userId);
        }
    });
    toRemoveUsers.forEach(USER_TOKEN_SESSIONS::remove);
}

和 SSE 的心跳检测不同，WebSocket 的 monitor 只是做了被动清理——检查 isOpen() 状态，而不是主动发心跳。为什么？因为 WebSocket 协议层自带 Ping/Pong 机制（由容器如 Tomcat 维护），应用层不需要额外发心跳来检测连通性。

配合 handleTextMessage("ping") → sendPongMessage 的应用层心跳，WebSocket 通道有两层保活：协议层 Ping/Pong（容器自动） + 应用层 ping/pong（前后端约定）。

六、检验一下（提取练习）

Q1. WebSocket 握手拦截器把用户信息存在哪里，供后续 Handler 使用？

A. HTTP Session B. ThreadLocal C. WebSocketSession.attributes D. Redis 缓存
看答案C。beforeHandshake() 的第三个参数 Map<String, Object> attributes 最终成为 WebSocketSession.getAttributes()。PlusWebSocketHandler 的所有方法都通过它取用户信息。

Q2. ConcurrentWebSocketSessionDecorator 的作用是什么？

A. 美化 WebSocket 消息的格式 B. 给 WebSocketSession 加并发安全 + 发送超时 + 缓冲区限制 C. 压缩 WebSocket 消息 D. 加密 WebSocket 连接
看答案B。它是 Spring 提供的包装器，保证多线程并发写 WebSocketSession 时不会出现数据竞争。同时限制单次发送超时（10s）和缓冲区大小（64KB）。

Q3. PlusWebSocketHandler.handleTextMessage() 收到 "ping" 时，回复的是什么？

A. "heartbeat" B. "pong" C. "connected" D. WebSocket Ping 帧
看答案B。webSocketSessionManager.sendMessage(session, "pong") 发的是文本消息 "pong"，不是 WebSocket 协议层的 Ping/Pong 帧。协议层的 Ping/Pong 由容器自动维护，这是应用层的。
