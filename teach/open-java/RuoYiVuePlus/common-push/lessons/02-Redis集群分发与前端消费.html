02 - Redis 集群分发与前端消费

🎯 本节目标：完整追踪「业务发送消息 → 前端收到」的跨节点链路，理解 Redis Pub/Sub 在集群消息分发中的作用，掌握 PushHelper 的 API 使用方式，并能解释 MessageTopicListener 的启动时机与消费逻辑。

1. 全景鸟瞰

上一课讲了两个通道的本地连接管理。本课聚焦消息如何跨节点传递——当用户 A 连接在服务器 1 上、用户 B 连接在服务器 2 上时，一条消息如何准确送达目标用户。

集群消息流全景：

服务器 1 (用户 A 在线)                   服务器 2 (用户 B 在线)
  │                                          │
  │ PushHelper.publishMessage(userB, msg)     │
  │   → pushSessionManager.publishMessage()   │
  │     → RedisUtils.publish(TOPIC, dto)      │
  │         │                                 │
  │         └──── Redis Pub/Sub ─────────────→│
  │                                           │
  │         ←──── 广播到所有节点 ────────────→│
  │                                           │
  │  MessageTopicListener 收到                 │  MessageTopicListener 收到
  │   → dto.userIds.contains(userA)? No       │   → dto.userIds.contains(userB)? Yes!
  │   → 跳过                                   │   → pushSessionManager.sendMessage(userB, dto)
  │                                           │     → SSE/WebSocket 发送到用户 B 的浏览器

2. 核心设计

2.1 Redis Pub/Sub 跨节点投递

为什么需要 Redis Pub/Sub？因为 HTTP 推送连接是绑定在服务器进程上的——用户 B 的 WebSocket 连接在服务器 2 的 JVM 内存中，服务器 1 无法直接访问。Redis Pub/Sub 作为「消息总线」，让集群中所有节点都能收到通知。

关键常量：MessageConstants.MESSAGE_TOPIC = "global:message"

发布端（publishMessage）：

// SseEmitterSessionManager / WebSocketSessionManager 都实现
publishMessage(PushDTO dto) {
    RedisUtils.publish(MESSAGE_TOPIC, dto);
}

订阅端（MessageTopicListener）：

// 实现 ApplicationRunner, Ordered(-1) → 优先级最高，最先启动
@Component
public class MessageTopicListener implements ApplicationRunner, Ordered {

    @Override
    public void run(ApplicationArguments args) {
        pushSessionManager.subscribeMessage(dto → {
            if (dto.getUserIds() != null && !dto.getUserIds().isEmpty()) {
                // 有目标用户 → 逐一投递
                dto.getUserIds().forEach(userId →
                    pushSessionManager.sendMessage(userId, dto));
            } else {
                // userIds 为空 → 全局广播
                pushSessionManager.publishAll(dto);
            }
        });
    }

    @Override
    public int getOrder() {
        return -1;  // 最先启动，确保消息不丢失
    }
}
Ordered(-1) 的意义：ApplicationRunner 按 order 升序执行。Ordered(-1) 保证 MessageTopicListener 在所有业务 Runner 之前启动。这意味着 Redis 消息订阅在系统还没完全启动时就已经就绪，不会丢失启动过程中的任何推送消息。

2.2 PushHelper：业务层的唯一入口

全静态方法、私有构造函数——典型的工具类模式。业务代码无需关心底层是 SSE 还是 WebSocket：

// 发送给指定用户
PushHelper.sendMessage(userId, "您的审批已通过");

// 发送给多个用户
PushHelper.publishMessage(Arrays.asList(1L, 2L, 3L), payloadDTO);

// 全局广播
PushHelper.publishAll(payloadDTO);

PushHelper 内部逻辑：

public static void sendMessage(Long userId, String message) {
    if (!isEnabled()) return;  // ❶ 兜底检查：推送功能是否开启

    PushSessionManager manager = SpringUtils.getBean(PushSessionManager.class);
    PushDTO dto = PushDTO.of(Collections.singletonList(userId), payload);
    manager.sendMessage(userId, dto);
}

isEnabled() 的重要性：如果在没有引入 push 模块的环境中调用 PushHelper（如单元测试），方法直接 return 不抛异常。这消除了业务代码对 push 模块的强依赖——引入模块推送生效、不引入模块静默跳过。

2.3 PushDTO 与 PushPayloadDTO：消息结构

字段类型含义
userIdsList<Long>目标用户列表（空=广播）
payloadPushPayloadDTO消息体（title + content + type + data）

Payload 结构：

PushPayloadDTO:
  title:   "新任务通知"
  content: "您有一条新的审批任务"
  type:    "TASK_NOTIFY"
  data:    { taskId: 12345, assignee: "张三" }

静态工厂方法：PushDTO.of(userIds, payload) → 定点推送；PushDTO.broadcast(payload) → 全局广播。

2.4 两种发送方式的区别

方法作用范围是否经 Redis适用场景
sendMessage(userId, dto)本节点本地连接否（直接发）用户在本节点在线时
publishMessage(dto)集群所有节点是（Pub/Sub）不确定用户在哪个节点时
publishAll(dto)集群所有节点的所有在线用户是（Pub/Sub）系统公告、全局通知
最佳实践：业务代码应优先使用 publishMessage 而不是 sendMessage。因为集群环境下你不知道目标用户连接在哪个节点——sendMessage 只在本地查找，目标用户可能连接在其他节点上收不到。publishMessage 通过 Redis 广播到所有节点，保证送达。

2.5 前端消费（SSE / WebSocket）

SSE 前端（JavaScript）：

const eventSource = new EventSource('/message/connect');
eventSource.addEventListener('message', (event) => {
    const data = JSON.parse(event.data);
    console.log('收到推送:', data.title, data.content);
});
eventSource.onerror = () => {
    // EventSource 会自动重连
};

WebSocket 前端（JavaScript）：

const ws = new WebSocket('ws://localhost:8080/message?token=' + token);
ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    if (data === 'PONG') return;  // 心跳忽略
    console.log('收到推送:', data.title, data.content);
};
// 手动心跳
setInterval(() => ws.send('PING'), 30000);
ws.onclose = () => { /* 手动重连 */ };

3. 实战练习

🧠 提取练习（点击展开）

Q1: 为什么 publishMessage 会发送到所有节点，目标用户不在线的节点也收到消息？这不浪费吗？

参考: 确实会浪费——每个节点都收到消息、检查 userIds、发现本地无此用户、丢弃。但这是 Redis Pub/Sub 的固有特性：消息广播给所有订阅者，无法按条件路由。在节点数少的场景（3-5 个节点），浪费可忽略。如果节点数很多（50+），可以考虑改用 Redis Stream + Consumer Group 模式——每个用户连接时注册到特定的 Stream Group，消息只投递给用户所在节点。

Q2: PushHelper 为什么用 isEnabled() 做兜底而不是依赖 @ConditionalOnProperty？

参考: @ConditionalOnProperty 控制的是 Bean 是否注册。如果 push 模块未引入或 message.enabled=false，PushSessionManager Bean 不存在。此时业务代码调用 PushHelper.sendMessage() 如果没有 isEnabled() 检查，SpringUtils.getBean() 会抛 NoSuchBeanDefinitionException。isEnabled() 通过检查配置项避免了这个异常，使 push 成为可选依赖而不是强依赖。

Q3: MessageTopicListener 的 Ordered(-1) 如果改成 Ordered(100) 会有什么影响？

参考: 如果某个 Ordered(50) 的业务 Runner 在启动时调用 PushHelper.publishMessage() 发送消息，而此时 MessageTopicListener 的 subscribeMessage 还没注册，消息发布到 Redis Pub/Sub 后没有消费者——消息丢失。Ordered(-1) 保证订阅在一切业务逻辑之前就绪，消息零丢失。

4. 调用链路

// 场景：审批通过后通知申请人

审批 Service:
  PushHelper.publishMessage(
      Arrays.asList(applicantUserId),
      PushPayloadDTO.of("审批通知", "您的请假申请已通过", "APPROVAL_NOTIFY")
  );

内部路径:
1. PushHelper.publishMessage(userIds, payload)
   → isEnabled() → true
   → SpringUtils.getBean(PushSessionManager.class)
     → 返回 SseEmitterSessionManager (假设 transport=sse)

2. SseEmitterSessionManager.publishMessage(PushDTO)
   → RedisUtils.publish("global:message", dto)
   → Redisson RTopic.publish(topic, dto)
   → Redis PUBLISH global:message {...json...}

3. 集群所有节点的 MessageTopicListener 收到消息
   → consumer.accept(dto)

   节点 1 (申请人不在线):
   → dto.userIds = [applicantUserId]
   → 遍历 userIds → sendMessage(applicantUserId, dto)
     → USER_TOKEN_EMITTERS.get(applicantUserId) → empty (不在线)
     → 无操作

   节点 2 (申请人在线):
   → dto.userIds = [applicantUserId]
   → 遍历 userIds → sendMessage(applicantUserId, dto)
     → USER_TOKEN_EMITTERS.get(applicantUserId) → { token1 → emitter1 }
     → emitter1.send(SseEmitter.event().data(dto))
     → 浏览器 EventSource 收到事件 → 弹出通知
💡 设计洞察：模块的集群消息分发设计体现了「发布订阅模式」的精髓——消息生产者（PushHelper）不关心消费者在哪里，消费者（MessageTopicListener）不关心消息从哪来。Redis 充当消息总线，所有节点平等——没有主节点、没有选举、没有单点故障。同时，PushHelper 的 isEnabled() 兜底设计体现了一种务实的防御性编程：让推送成为可选的能力增强模块，而不是强依赖——不引入不会报错、引入了自动生效。这是微服务拆分中「可选模块」的理想形态。
