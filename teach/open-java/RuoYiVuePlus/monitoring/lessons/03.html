第 3 课 · 安全守护：SecurityConfig 过滤链逐行解析 安全

🎯 本节目标（对应使命 Success #3）：学完你能逐行读懂 SecurityConfig.java 的安全过滤链——如何放行静态资源（/assets/**）和登录页、要求其余请求认证、配置表单登录 + HTTP Basic 双认证、以及禁用 CSRF 和 frameOptions 的原因。

如果你的 SBA 管理后台没有安全保护，任何人都能通过浏览器看到所有微服务的健康状态、配置信息、甚至日志文件——这在生产环境是不可接受的。这节课我们逐行拆解 SecurityConfig，理解它是如何用最小的配置代价为 SBA 建立安全防线。

一、全貌：65 行代码 = 一条完整的过滤链

SecurityConfig.java — 完整源码

ruoyi-monitor-admin/src/main/java/org/dromara/monitor/admin/config/SecurityConfig.java 
@EnableWebSecurity
@Configuration
public class SecurityConfig {

    private final String adminContextPath;

    public SecurityConfig(AdminServerProperties adminServerProperties) {
        this.adminContextPath = adminServerProperties.getContextPath();
    }

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity httpSecurity) throws Exception {
        SavedRequestAwareAuthenticationSuccessHandler successHandler =
            new SavedRequestAwareAuthenticationSuccessHandler();
        successHandler.setTargetUrlParameter("redirectTo");
        successHandler.setDefaultTargetUrl(adminContextPath + "/");

        PathPatternRequestMatcher.Builder mvc = PathPatternRequestMatcher.withDefaults();

        return httpSecurity
            .headers((header) ->
                header.frameOptions(HeadersConfigurer.FrameOptionsConfig::disable))
            .authorizeHttpRequests((authorize) ->
                authorize.requestMatchers(
                        mvc.matcher(adminContextPath + "/assets/**"),
                        mvc.matcher(adminContextPath + "/login")
                    ).permitAll()
                    .anyRequest().authenticated())
            .formLogin((formLogin) ->
                formLogin.loginPage(adminContextPath + "/login")
                    .successHandler(successHandler))
            .logout((logout) ->
                logout.logoutUrl(adminContextPath + "/logout"))
            .httpBasic(Customizer.withDefaults())
            .csrf(AbstractHttpConfigurer::disable)
            .build();
    }
}

一眼看过去，filterChain 方法就是个链式配置，每行一个安全策略。我们分 6 处逐行拆解。

二、6 处关键配置逐行读

第0步：获取 context-path

AdminServerProperties 是 SBA 框架提供的配置 Bean，它自动读取 spring.boot.admin.context-path 的值。在第 2 课我们配的是 /admin，所以 adminContextPath = "/admin"。

下面所有涉及路径的规则都基于这个前缀，保证了无论你把 SBA 挂载到 /admin、/monitor 还是 /，安全规则自动适配——不需要硬编码路径。

禁用 frameOptions — 让 SBA 页面能在 iframe 中渲染

.headers((header) ->
    header.frameOptions(HeadersConfigurer.FrameOptionsConfig::disable))

SBA 的管理后台 UI 在某些功能中使用了 iframe（例如日志查看器就是在 iframe 中加载的）。Spring Security 默认会设置 X-Frame-Options: DENY 响应头，禁止页面被嵌入 iframe，导致这些功能白屏。

安全影响：禁用 frameOptions 意味着你的 SBA 页面可以被其他网站用 iframe 嵌入（clickjacking 风险）。但 SBA 是内部运维工具，通常部署在内网，攻击面有限。如果你的 SBA 暴露在公网，考虑用 CSP（Content-Security-Policy）的 frame-ancestors 指令做更精细的控制，而不是一刀切禁用。

放行静态资源 + 登录页，其余需认证

.authorizeHttpRequests((authorize) ->
    authorize.requestMatchers(
            mvc.matcher(adminContextPath + "/assets/**"),
            mvc.matcher(adminContextPath + "/login")
        ).permitAll()
        .anyRequest().authenticated())

这是安全模型的精要——「除了门口那几样，全锁上」：

  • /admin/assets/** — SBA UI 的静态资源（CSS、JS、图片、字体）。不放行则登录页连样式都加载不出来。
  • /admin/login — 登录页本身。不放行则弹出登录框让你登录才能看到登录页——死循环。
  • anyRequest().authenticated() — 所有其他请求（包括 /admin/、/admin/applications、/admin/instances 等 SBA API）需要认证。
为什么用 PathPatternRequestMatcher 而不是 AntPathRequestMatcher？这是 Spring Security 6.x（基于 Spring Boot 3.x）的新默认匹配器。PathPattern 比 Ant 风格性能更好，且天然支持 {variable} 路径变量。RuoYi-Vue-Plus 全系统统一使用这个匹配器。

表单登录 — 输账号密码，登录后跳回原页面

.formLogin((formLogin) ->
    formLogin.loginPage(adminContextPath + "/login")
        .successHandler(successHandler))

loginPage("/admin/login") 指定登录页的地址（SBA 自带的登录页，不是自定义的）。
SavedRequestAwareAuthenticationSuccessHandler 是 Spring Security 提供的智能跳转器：

  • 如果你直接访问 /admin/login 登录 → 跳到 /admin/（defaultTargetUrl）
  • 如果你访问 /admin/applications/xxx 被拦截到登录 → 登录成功后回到这个页面（SavedRequest）
  • redirectTo 参数可以覆盖默认跳转目标

注销 — 清理会话，回到登录页

.logout((logout) ->
    logout.logoutUrl(adminContextPath + "/logout"))

简洁的注销配置。POST 到 /admin/logout 即清除 SecurityContext 和 Session。注意默认 SBA UI 的注销按钮发送的是 POST 请求（CSRF 禁用后才不会报 403——见下一步）。

HTTP Basic 认证 — 双重认证方式

.httpBasic(Customizer.withDefaults())

除了浏览器登录（表单登录），还支持 HTTP Basic 认证。用 curl -u ruoyi:123456 http://localhost:9090/admin/applications 也能访问。什么场景需要这个？

  • SBA Client 注册时 — Client 向 Server 发 POST 注册请求时，用的是 HTTP Basic 认证（账密来自 spring.boot.admin.client.username/password）
  • Server 回调 Client — Server 调用 Client 的 actuator 端点时，同样用 HTTP Basic（凭证来自 metadata）
为什么需要两种认证方式并存？人（运维人员）用表单登录——浏览器弹出登录页，输账号密码。程序（SBA Client）用 HTTP Basic——没有浏览器界面，只能通过 HTTP 头传递凭证。两者服务于不同类型的「用户」，缺一不可。

禁用 CSRF — 让 POST 请求不报 403

.csrf(AbstractHttpConfigurer::disable)

Spring Security 默认开启 CSRF 保护，要求所有 POST/PUT/DELETE 请求携带 CSRF token。但 SBA 管理后台的场景是：

  • SBA 的 UI 是内嵌的 Vue.js SPA，它的 AJAX 请求不携带 CSRF token
  • SBA Client 通过 HTTP Basic 方式调用 Server API，也没有 CSRF token 机制
  • 注销操作是 POST 请求，CSRF 不关闭会直接 403

因此必须禁用 CSRF。这个决策在 SBA 官方文档中明确推荐，不是偷懒。

四、跟 Spring Security 7.x 的差异说明

RuoYi-Vue-Plus 使用的是 Spring Boot 3.x + Spring Security 6.x。代码中使用了 Lambda DSL 风格（authorizeHttpRequests(customizer -> ...)），这是 6.x 标配。如果你看到旧项目用 http.authorizeRequests()（返回 ExpressionInterceptUrlRegistry）的写法，那是 Spring Security 5.x 的风格，在 6.x 已废弃。

关键版本差异对照：

Spring Security 5.x（已废弃）Spring Security 6.x（当前）
http.authorizeRequests()http.authorizeHttpRequests()
.antMatchers("/path/**")mvc.matcher("/path/**")（PathPattern）
.and() 串联Lambda DSL 嵌套，不需要 .and()
默认 X-Frame-Options: DENY同上（需显式 disable）

五、检验一下（提取练习）

Q1. 为什么要禁用 frameOptions？

A. 为了支持 HTTPS，因为 frameOptions 会阻止 TLS 握手 B. SBA 管理后台的某些功能（如日志查看器）使用 iframe 加载内容，不关闭 frameOptions 会导致白屏 C. 为了兼容 IE6 浏览器 D. 因为 Spring Security 6.x 默认没有 frameOptions
看答案B。SBA 的日志查看器等面板在 iframe 中渲染，X-Frame-Options: DENY 会阻止 iframe 加载。禁用它是为了功能可用，但需注意 clickjacking 风险——通常 SBA 部署在内网，风险可控。

Q2. permitAll() 放行的两个路径是什么？为什么是这两个？

A. /admin/** 和 /actuator/** — 因为它们是静态资源 B. /admin/assets/** 和 /admin/login — 前者是前端静态资源（不放行则登录页白屏），后者是登录页本身（不放行则死循环） C. /admin/logout 和 /admin/applications — 因为它们是公开 API D. / 和 /favicon.ico — 根路径和图标
看答案B。放行 /assets/** 保证 CSS/JS/图片能加载，否则登录页只有白底黑字。放行 /login 避免「需要登录才能看到登录页」的死循环。

Q3. 为什么需要 HTTP Basic 和表单登录两种认证方式并存？

A. 为了向后兼容 Spring Security 4.x 的用户 B. 表单登录给人用（浏览器），HTTP Basic 给程序用（SBA Client 注册、Server 回调 actuator） C. HTTP Basic 是主认证，表单登录是备用的 D. 两种认证方式让人可以选择难度
看答案B。运维人员在浏览器中通过表单登录；SBA Client 通过 HTTP 协议通信时只能用 HTTP Basic 传递凭证。两者服务于不同消费方，缺一不可。

Q4. 禁用 CSRF 保护的原因是什么？

A. SBA Server 没有 CSRF 攻击面，它不处理用户数据 B. SBA 管理后台的 Vue.js 前端 AJAX 请求不携带 CSRF token，且 SBA Client 的 HTTP Basic 认证也不支持 CSRF，不关闭会导致 403 C. Spring Security 6.x 已经废弃了 CSRF 功能 D. 项目中使用了 JWT 替代了 CSRF
看答案B。两个原因：(1) SBA 自带的 Vue.js UI 在做 AJAX 请求时不带 CSRF token；(2) Client 通过 HTTP Basic 认证也不走 CSRF。这是 SBA 官方文档明确推荐的配置。
