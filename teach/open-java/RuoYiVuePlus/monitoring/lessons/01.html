第 1 课 · 全景鸟瞰：SBA 监控中心三层架构 入门地图

🎯 本节目标（对应使命 Success #1）：学完你能用一段话讲清 @EnableAdminServer 如何将一个普通 Spring Boot 应用变成监控中心，以及 SBA Server（展示层）+ SBA Client（数据上报）+ Spring Boot Actuator（数据源）三者的协作关系。

打开 ruoyi-monitor-admin 模块，你会发现它只有 3 个 Java 文件 + 1 个 YML 配置——就这样，一个企业级服务监控中心就立起来了。这节课我们把整个模块的骨架摊开看，理解 Spring Boot Admin 的三层协作模型。

一、一句话记住本质：它是「SBA Server + Client 合一的自监控应用」

绝大多数 Spring Boot Admin 教程都会告诉你「一个应用做 Server，其他应用做 Client」。但 RuoYi-Vue-Plus 的做法更巧妙——monitor-admin 自己同时是 Server 和 Client。也就是说，它既提供监控管理后台，也把自己作为一个被监控的实例注册上去。打开 pom.xml 验证一下：

<!-- Spring Boot Admin 服务端 — 提供监控管理后台 UI -->
<dependency>
    <groupId>de.codecentric</groupId>
    <artifactId>spring-boot-admin-starter-server</artifactId>
</dependency>

<!-- Spring Boot Admin 客户端 — 将自己注册到 Server 上 -->
<dependency>
    <groupId>de.codecentric</groupId>
    <artifactId>spring-boot-admin-starter-client</artifactId>
</dependency>
为什么要在 Monitor 应用里同时引入 Server 和 Client？这叫「自注册模式」（Self-Registration）。管理员打开 SBA 后台时，第一眼就能看到 monitor-admin 自身的健康状态——如果连监控中心自己都挂了，那还监控什么？这是一种「兽医给自己量体温」的务实践。

二、三层协作关系：Server → Client → Actuator

SBA 的架构本质是三层分工，每一层只做一件事：

┌──────────────────────────────────────────────────────┐ │ 第 1 层：SBA Server（展示与聚合层） │ │ ┌──────────────────────────────────────────────┐ │ │ │ @EnableAdminServer + SBA UI (Vue.js 构建) │ │ │ │ 职责：接收客户端上报的数据，在 Web 管理后台展示 │ │ │ │ 产物：http://localhost:9090/admin → 服务墙/详情面板 │ │ │ └──────────────────────────────────────────────┘ │ │ ▲ 客户端主动向 Server 上报（HTTP POST + 心跳） │ │ │ │ │ ┌──────┴───────────────────────────────────────┐ │ │ │ 第 2 层：SBA Client（注册与上报层） │ │ │ │ 职责：启动时向 Server 注册自己，定时发送心跳和状态数据 │ │ │ │ 配置：spring.boot.admin.client.url = Server地址 │ │ │ └──────────────────────────────────────────────┘ │ │ ▲ HTTP GET（Client → Actuator 端点） │ │ │ │ │ ┌──────┴───────────────────────────────────────┐ │ │ │ 第 3 层：Spring Boot Actuator（数据源层） │ │ │ │ 职责：暴露 /actuator/health, /metrics, /env 等端点 │ │ │ │ 配置：management.endpoints.web.exposure.include='*' │ │ │ └──────────────────────────────────────────────┘ │ └──────────────────────────────────────────────────────┘

关键理解：SBA Client 不自己产生数据，它只是一个「信使」——从 Actuator 端点采集数据，打包上报给 Server。Server 收到后存在内存（InstanceRepository），并在 UI 上渲染成服务墙、详情面板、日志查看器等视图。

三、看 @EnableAdminServer 的真实面目

MonitorAdminApplication.java — 一行注解变身监控中心

ruoyi-extend/ruoyi-monitor-admin/src/main/java/org/dromara/monitor/admin/MonitorAdminApplication.java 
@EnableAdminServer
@SpringBootApplication
public class MonitorAdminApplication {

    public static void main(String[] args) {
        SpringApplication.run(MonitorAdminApplication.class, args);
        System.out.println("Admin 监控启动成功");
    }
}

@EnableAdminServer 这行注解背后，SBA 自动装配做了什么？

自动装配项作用
AdminServerAutoConfiguration注册 InstanceRepository（内存存储所有注册实例）、启动 HTTP 端点接收客户端上报
AdminServerWebConfiguration挂载 SBA 管理界面（Vue.js SPA），默认路径 /，可配 context-path
NotifierAutoConfiguration扫描所有 Notifier Bean（包括你的 CustomNotifier），在事件发生时自动调用
UI 静态资源内嵌 spring-boot-admin-server-ui jar，提供列表墙、详情面板、日志查看器、Journal 等全套界面
所以「搭一个监控中心」的本质是什么？只需三步：(1) 引入 server starter → 自动装配出 UI + InstanceRepository；(2) 引入 client starter → 自动装配出注册 + 心跳机制；(3) 暴露 Actuator 端点 → 客户端有数据可采。第 2 课会深入到 YML 配置层面逐行讲解。

五、SBA 状态模型速览（6 种状态码）

所有被监控的实例在 SBA 中都有一个状态。状态来源于 Actuator 的 /health 端点响应，SBA 将其抽象为 6 种：

状态码含义触发条件（典型）
UP服务在线，一切正常/health 返回 UP
OFFLINE服务离线（主动/被动移除）Client 主动注销，或心跳超时被 Server 移除
DOWN服务宕机/health 返回 DOWN（如数据库连不上）
RESTRICTED服务受限限流触发，或部分功能被降级
OUT_OF_SERVICE停止服务（计划内维护）手动标记 OUT_OF_SERVICE，如预发布验证
UNKNOWN未知状态刚注册还未上报，或网络中断无法获取状态

第 4 课会讲 CustomNotifier 如何用 switch 表达式将这 6 种状态映射为中文业务描述，并在状态变化时打日志。

六、检验一下（提取练习）

不要回看上文，凭记忆作答。

Q1. @EnableAdminServer 注解做了什么？

A. 启动一个内嵌的 Prometheus 采集器 B. 自动装配 InstanceRepository（实例存储）、SBA UI（管理界面）、Notifier（通知器）体系 C. 在 Linux 上自动安装 Grafana D. 只是标记性注解，不触发任何自动装配
看答案B。它触发 SBA 的一整套自动装配：AdminServerAutoConfiguration（实例仓库 + 上报端点）、AdminServerWebConfiguration（管理 UI）、NotifierAutoConfiguration（通知器扫描）。

Q2. 三层协作关系中，哪一层负责「产生数据」？

A. SBA Server — 它主动探测所有实例 B. SBA Client — 它内置了健康检查引擎 C. Spring Boot Actuator — 它暴露 /health、/metrics 等端点，Client 从它这里采集数据 D. CustomNotifier — 它实时计算健康度
看答案C。Actuator 是数据源，Client 是信使，Server 是展示者。没有 Actuator 暴露端点，Client 采集不到任何数据。

Q3. 为什么 monitor-admin 要同时引入 server 和 client 两个 starter？

A. 为了凑依赖数，达到 Maven 的推荐最小依赖量 B. 自注册模式——让它既提供监控 UI，也把自己注册为被监控实例，开后台就能看到自身状态 C. server-starter 包含了 client-starter，不引会编译报错 D. 为了让 undertow 容器支持双端口
看答案B。Server 提供后台 UI，Client 让 monitor-admin 自己也作为被监控实例出现在服务墙上。这是一种务实的自监控设计。
