第 2 课 · 自注册模式：逐行读懂 application.yml 配置全景

🎯 本节目标（对应使命 Success #2）：学完你能逐行解释 application.yml 中自注册模式的完整配置逻辑——monitor-admin 如何通过 Server + Client 的配置共存实现自监控，以及 service-host-type: IP、metadata 凭证传递、Actuator 端点暴露等关键配置的作用。

第 1 课讲了「Server + Client 合一」的架构思路。这节课我们钻进 application.yml，看每一段配置在做什么。YML 中的 --- 分隔符将配置分为四个逻辑段，我们来逐段拆解。

一、基础段：端口与日志

段 1：服务器基础配置

ruoyi-monitor-admin/src/main/resources/application.yml 
server:
  port: 9090                        # 监控中心独立端口，与业务应用隔离
spring:
  application:
    name: ruoyi-monitor-admin       # 在 SBA 管理后台显示的服务名
  profiles:
    active: @profiles.active@       # Maven 变量替换，打包时注入环境名

logging:
  config: classpath:logback-plus.xml # 引用父模块的统一日志配置

@profiles.active@ 是 Maven resource filtering 的占位符——pom.xml 里定义了 profiles.active 属性，打包时注入实际的 profile 值（如 dev 或 prod）。这和框架主应用 ruoyi-admin 的做法一致。

二、SBA Server 段：监控中心自身的配置

段 2：SBA Server（监控中心服务端）

--- # 监控中心服务端配置
spring:
  security:
    user:
      name: @monitor.username@       # 登录账号（Maven 变量）
      password: @monitor.password@   # 登录密码（Maven 变量）
  boot:
    admin:
      ui:
        title: RuoYi-Vue-Plus服务监控中心 # 浏览器标签页标题
      context-path: /admin           # 关键！后台 UI 挂载到 /admin 路径
  thymeleaf:
    check-template-location: false   # 消除 SBA 内嵌 Thymeleaf 的警告

这段配置的核心是 context-path: /admin。当你配置了这个值：

  • SBA 管理后台 UI 的访问路径变为 http://localhost:9090/admin
  • Client 向 Server 注册的 URL 也需要带上 /admin（见段 4）
  • SecurityConfig 中的放行规则也需要基于 /admin 前缀（第 3 课会读到）
一个常见翻车点：如果你改了 context-path，但忘记同步修改 spring.boot.admin.client.url 中的路径，客户端会注册到错误的地址，导致 SBA 后台看不到任何实例。两个配置是耦合的——改一处必须改另一处。

@monitor.username@ 和 @monitor.password@ 同样是 Maven 资源过滤变量，定义在父 POM 的 <properties> 中。这行配置实际上是在定义 Spring Security 的内存用户——SecurityConfig 的表单登录就用这个账号密码验证。

thymeleaf.check-template-location: false 是一个消除噪音的小技巧。SBA Server 内部依赖了 Thymeleaf（用于渲染部分静态页面），但 monitor-admin 并未使用 Thymeleaf 模板，Spring Boot 会打印一条 WARN 日志。这个配置将它静默。

三、Actuator 段：数据暴露配置

段 3：Actuator 端点暴露

--- # Actuator 监控端点的配置项
management:
  endpoints:
    web:
      exposure:
        include: '*'                # 暴露所有 Actuator 端点
  endpoint:
    health:
      show-details: ALWAYS          # 健康检查显示完整细节
    logfile:
      external-file: ./logs/ruoyi-monitor-admin.log # 日志文件路径

这三个配置决定了 SBA Client「能采到什么数据」：

配置项作用没配会发生什么
include: '*'暴露出所有 actuator 端点：health、metrics、env、loggers、threaddump、heapdump、mappings 等二十多个SBA 管理后台只能看到应用在线/离线，没有详细指标
show-details: ALWAYShealth 端点返回完整健康细节（数据库连接、磁盘空间、Redis 等各组件的状态）SBA 详情面板看不到「哪个健康组件挂了」，只知道 UP/DOWN
logfile.external-file告诉 Actuator 日志文件在哪里，让 SBA 的「日志查看器」能读取SBA 后台的「Logging」面板看不到日志内容
为什么用 '*' 通配符而不是逐条列出？在开发/测试环境，暴露所有端点方便调试。生产环境建议按需暴露，避免泄漏敏感信息（如 heapdump 可能包含内存中的密码明文）。但 SBA 后台本身有 Spring Security 保护（第 3 课），所以这里的 '*' 是安全的——未登录的人连后台都进不去。

四、SBA Client 段：自注册的核心配置

段 4：SBA Client（自注册模式 — 本课精华）

--- # 监控配置
spring.boot.admin.client:
  enabled: true                     # 启用客户端注册（默认 true，显式声明更清晰）
  url: http://localhost:9090/admin  # ★ 指向自己！Server 的 context-path 就是 /admin
  instance:
    service-host-type: IP           # ★ 用 IP 而非主机名注册，避免 DNS 依赖
    metadata:                       # 自定义元数据，传递给 Server 用于认证
      username: ${spring.boot.admin.client.username}
      userpassword: ${spring.boot.admin.client.password}
  username: @monitor.username@      # 客户端注册时使用的账号
  password: @monitor.password@      # 客户端注册时使用的密码

这段配置是整个自注册模式的心脏。逐行拆解：

① url: http://localhost:9090/admin — 指向自己

这个 URL 就是 SBA Client 向 Server 注册的地址。关键洞察：它的 host:port 指向 monitor-admin 自己。所以应用的启动流程是：

1. Spring Boot 启动，加载 all AutoConfiguration 2. SBA Server AutoConfig → 启动内嵌 Tomcat/Jetty，监听 9090，挂载 /admin UI 3. SBA Client AutoConfig → 读取 url=http://localhost:9090/admin 4. Client 向这个 URL POST 注册请求 → Server 收到，写入 InstanceRepository 5. Client 启动心跳线程 → 定时向 Server 报告状态 6. 你打开 http://localhost:9090/admin → 看到 monitor-admin 自己出现在服务墙上

② service-host-type: IP — 为什么用 IP？

SBA Client 注册时，会告诉 Server「我的地址是 XXX」。service-host-type 决定了这个地址的格式：

  • IP（本项目的选择）→ 注册为 http://192.168.1.100:9090/admin
  • HOSTNAME（默认值） → 注册为 http://my-server:9090/admin
  • CANONICAL_HOST_NAME → 注册为 FQDN（如 http://my-server.example.com:9090/admin）
生产环境的大坑：如果你的服务器 hostname 解析有问题（比如 /etc/hosts 没配、DNS 不通），用 HOSTNAME 模式会导致其他微服务无法通过该地址访问此实例的健康端点。用 IP 是生产环境最稳妥的选择。

③ metadata 凭证传递 — 客户端的「身份证」

metadata 是一个 Map<String, String>，随注册请求一起发送给 Server。这里传递了 username 和 userpassword，用途是：当 其他应用（非本 monitor-admin）也作为 Client 接入时，Server 可以用这些凭证去调用这些 Client 的受保护 Actuator 端点。

注意 ${spring.boot.admin.client.username} 这种 Spring 占位符引用——它实际上取的是同一 YML 中下面定义的 username: @monitor.username@ 的值。这是一种 YML 内部的引用链。

注意区分两套认证：(1) spring.security.user.name/password 是 SBA 管理后台的登录凭证——你打开浏览器输入它。(2) spring.boot.admin.client.username/password 是 SBA Client 向 Server 注册时携带的凭证——Server 用它们调用 Client 的 actuator 端点。两者在本项目中用了同一个 Maven 变量值，但语义上它们是两套独立的认证。

六、检验一下（提取练习）

不要回看上文，凭记忆作答。

Q1. spring.boot.admin.client.url 设置为 http://localhost:9090/admin 意味着什么？

A. 浏览器访问该 URL 就能打开 SBA 后台 B. monitor-admin 作为 Client 向自己（作为 Server）注册，实现自监控 C. 这是 Actuator 端点的暴露地址 D. 这是监控告警的 Webhook 回调 URL
看答案B。该 URL 的 host:port 指向 monitor-admin 自己，Client 启动后向此地址发送注册请求，Server 接收并存入 InstanceRepository。这就是自注册模式的关键。

Q2. service-host-type: IP 的作用是什么？

A. 让 SBA 后台只能通过 IP 访问（禁用域名） B. 告诉 SBA Client 用本机 IP 地址（而非 hostname）向 Server 注册自己 C. 强制使用 IPv6 协议 D. 只允许 IP 白名单中的客户端注册
看答案B。它决定 Client 注册时上报的地址格式。IP 模式避免 hostname 解析失败导致的不可达问题，生产环境更稳妥。

Q3. management.endpoints.web.exposure.include: '*' 在生产环境是否安全？为什么？

A. 不安全，应该改为只暴露 health 和 info B. 安全的，因为 SBA 后台有 Spring Security 保护，且 actuator 端点本身不对外暴露（内网通信） C. 不安全，任何人都能直接访问 /actuator/env 看到环境变量 D. 无法回答，取决于网段配置
看答案B。在本项目架构中：(1) SBA 后台有 SecurityConfig 保护，未登录无法访问；(2) actuator 端点通常只在内部网络开放，外部不可达。但安全最佳实践建议按需暴露——生产环境可以结合网络隔离和端点白名单双重加固。

Q4. metadata 中的 username 和 userpassword 是用来做什么的？

A. 作为 SBA 后台登录的备选密码 B. 当 Server 需要调用 Client 的受保护 Actuator 端点时，用这些凭证进行 HTTP Basic 认证 C. 加密 Client 和 Server 之间的通信 D. 只是自定义标签，不影响实际行为
看答案B。metadata 是 SBA 的凭证传递机制——Client 把自己的账密告诉 Server，Server 再拿着这些凭证去调 Client 的 actuator 端点。注意这是 SBA 框架层面的功能，不是 Spring Security 的标准行为。
