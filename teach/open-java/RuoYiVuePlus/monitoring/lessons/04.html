第 4 课 · 事件驱动：CustomNotifier 状态变化通知器 扩展

🎯 本节目标（对应使命 Success #4）：学完你能读懂 CustomNotifier.java 的事件驱动设计——继承 AbstractEventNotifier、通过 doNotify() 拦截 InstanceStatusChangedEvent、使用 Reactor Mono.fromRunnable() 异步处理、以及将 6 种 SBA 状态码映射为中文业务语义的 switch 表达式。

前 3 课我们搭好了监控中心的骨架——Server 接收数据、Client 上报数据、Security 保护后台。但这一切发生时，你作为运维人员如何第一时间知道某个服务挂了？这就轮到 CustomNotifier 上场了。它是 SBA 事件体系中的「哨兵」——当实例状态发生变化，立刻触发通知逻辑。

一、全貌：68 行代码 = 继承 + 过滤 + 映射 + 记录

CustomNotifier.java — 完整源码

ruoyi-monitor-admin/src/main/java/org/dromara/monitor/admin/notifier/CustomNotifier.java 
@Slf4j
@Component
public class CustomNotifier extends AbstractEventNotifier {

    protected CustomNotifier(InstanceRepository repository) {
        super(repository);
    }

    @Override
    @SuppressWarnings("all")
    protected Mono<Void> doNotify(InstanceEvent event, Instance instance) {
        return Mono.fromRunnable(() -> {
            // 实例状态改变事件
            if (event instanceof InstanceStatusChangedEvent) {
                // 获取实例注册名称
                String registName = instance.getRegistration().getName();
                // 获取实例ID
                String instanceId = event.getInstance().getValue();
                // 获取实例状态
                String status = ((InstanceStatusChangedEvent) event)
                    .getStatusInfo().getStatus();
                // 获取服务URL
                String serviceUrl = instance.getRegistration().getServiceUrl();
                String statusName = switch (status) {
                    case STATUS_UP -> "服务上线";
                    case STATUS_OFFLINE -> "服务离线";
                    case STATUS_RESTRICTED -> "服务受限";
                    case STATUS_OUT_OF_SERVICE -> "停止服务状态";
                    case STATUS_DOWN -> "服务下线";
                    case STATUS_UNKNOWN -> "服务未知异常";
                    default -> "未知状态";
                };
                log.info("Instance Status Change: 状态名称【{}】, "
                    + "注册名称【{}】, 实例ID【{}】, 状态【{}】, 服务URL【{}】",
                    statusName, registName, instanceId, status, serviceUrl);
            }
        });
    }
}

二、三层继承链：理解 SBA 的通知器体系

SBA 事件通知器的继承体系： Notifier (接口) └── notify(InstanceEvent) → Mono<Void> │ AbstractEventNotifier (抽象类) ← CustomNotifier 继承的就是它 ├── 持有 InstanceRepository 引用（可查询其他实例） ├── 内置 event type 过滤（可配置只关心某些类型） └── 定义 doNotify(InstanceEvent, Instance) 模板方法 │ CustomNotifier (你的实现) └── 覆盖 doNotify()，只处理 InstanceStatusChangedEvent

理解这个继承链的两个要点：

  • AbstractEventNotifier 提供的是「骨架」——它帮你做了 event 类型过滤、去重检查等通用逻辑，你的 doNotify() 只被调用「当且仅当事件是你关心的类型」。也就是说，抽象的过滤在基类完成，你不需要在子类里写重复的类型判断。
  • InstanceRepository 是 SBA 的内存数据库——它存储了所有注册实例的完整信息（registration、status、endpoints 等）。通过它可以查询「当前有哪些实例在线」「某个实例的详细信息」。CustomNotifier 虽然没直接用到它，但通过 super(repository) 传递给了基类。

三、Reactor 异步模型：为什么是 Mono.fromRunnable()？

doNotify() 的返回值不是 void，而是 Mono<Void>。这是 SBA 通知器体系的强制约定——所有通知器必须基于 Project Reactor 响应式编程模型。

protected Mono<Void> doNotify(InstanceEvent event, Instance instance) {
    return Mono.fromRunnable(() -> {
        // 你的通知逻辑
    });
}

这行代码的意图是：

Mono 操作含义
Mono.fromRunnable()将一段同步代码包装成响应式的 Mono<Void>。Runnable 中的代码会在 订阅时执行，而不是定义时。
Mono.fromCallable()（未使用）包装有返回值的同步代码为 Mono<T>
Mono.just()（未使用）包装一个已知值为 Mono
为什么不简单地写 void 方法？SBA Server 内部的事件总线是响应式的——当多个实例状态同时发生变化，Server 可以用 Reactor 的调度器在非阻塞线程上并发调用所有已注册的 Notifier。如果你的通知器里做了慢操作（如发邮件、调 HTTP 接口），Mono 可以让你用 .subscribeOn(Schedulers.boundedElastic()) 把慢操作挪到弹性线程池，不阻塞事件总线。
一个注意：本课程的实现是在 Runnable 里做 log.info()，不涉及真正的异步。如果以后扩展为真实告警（发邮件/钉钉/企微），你应该在 Mono.fromRunnable() 后链式调用 .subscribeOn() 指定调度器，并返回新的 Mono。否则慢 I/O 会阻塞事件分发线程。

四、事件过滤：instanceof 精确拦截

if (event instanceof InstanceStatusChangedEvent) {
    // 只处理状态变化事件
}

SBA 的事件类型不止一种。除了 InstanceStatusChangedEvent，还有：

事件类型触发场景
InstanceRegisteredEvent新实例注册到 Server
InstanceDeregisteredEvent实例从 Server 注销
InstanceStatusChangedEvent实例状态变化（UP→DOWN 等），本 Notifier 关注这个
InstanceInfoChangedEvent实例元数据（metadata）或 info 端点信息变更
InstanceEndpointsDetectedEvent探测到实例暴露了新的 actuator 端点

基类 AbstractEventNotifier 的 notify() 方法也有一个事件类型过滤（通过构造函数参数配置），但那个过滤粒度较粗。在 doNotify() 里用 instanceof 再做一次精确过滤，是最清晰的写法。

五、switch 表达式：6 种状态码 → 中文业务语义

状态映射表 — 本课最精华的设计决策

String statusName = switch (status) {
    case STATUS_UP              -> "服务上线";
    case STATUS_OFFLINE         -> "服务离线";
    case STATUS_RESTRICTED      -> "服务受限";
    case STATUS_OUT_OF_SERVICE  -> "停止服务状态";
    case STATUS_DOWN            -> "服务下线";
    case STATUS_UNKNOWN         -> "服务未知异常";
    default                     -> "未知状态";
};

这段 switch 的精彩之处：

(1) 用的是 Java 14+ 的 switch 表达式（Expression Switch），不是传统的 switch 语句。箭头语法 -> 自动 break，不穿透；返回值直接赋给变量 statusName。比传统写法简洁一个数量级。

(2) 常量来自静态导入，不写硬编码字符串：

import static de.codecentric.boot.admin.server.domain.values.StatusInfo.*;

把 STATUS_UP、STATUS_DOWN 等常量直接导入命名空间，避免写 StatusInfo.STATUS_UP 这种冗长的前缀。这些常量定义在 SBA 框架的 StatusInfo 类中，值是 "UP"、"DOWN" 等字符串。

(3) 「服务上线」和「服务下线」的区分有业务意义：

  • UP → "服务上线" — 应用的 /health 返回 UP，一切正常
  • OFFLINE → "服务离线" — 实例从注册中心移除（可能是手动注销或心跳超时）
  • DOWN → "服务下线" — 应用还在，但 /health 返回 DOWN（如数据库连不上）

细品这三个中文翻译的差异——「上线」是正常状态，「离线」是非正常退出，「下线」是挂了但进程可能还在。运维人员看到这条日志能立刻判断严重程度。

(4) default 分支是防御性编程的体现。虽然有 6 个已知状态，但如果 SBA 未来版本新增了状态码（如 STATUS_STARTING），这个 default -> "未知状态" 保证了 switch 不会因为缺少 case 而抛异常。日志仍然会打印出来，只是状态名是「未知状态」。

六、事件处理的完整数据流

SBA 内部事件流： 1. Client 心跳上报 → Server 更新 Instance 状态 │ 2. Server 对比新旧状态 → 发现状态变化（如 UP→DOWN） │ 3. Server 发布 InstanceStatusChangedEvent 到事件总线 │ 4. 事件总线遍历所有 Notifier Bean │ 5. AbstractEventNotifier.notify(event) ├→ 检查 event type 过滤 → 匹配，继续 └→ 调用 doNotify(event, instance) │ 6. CustomNotifier.doNotify() ├→ instanceof InstanceStatusChangedEvent ✓ ├→ 提取 registName, instanceId, status, serviceUrl ├→ switch 映射 status → 中文名称 └→ log.info("状态名称【服务上线】, 注册名称【ruoyi-monitor-admin】, ...") │ 7. 日志落盘 → 运维人员在 ELK/Splunk 中搜索到状态变化记录

七、从日志记录到真实告警的扩展思路

本课的 CustomNotifier 只做了 log.info()，但你能扩展什么？
这是通知器体系设计的巧妙之处——doNotify() 模板方法只定义了「什么时候触发」和「给你什么数据」，剩下的「怎么通知」完全由你决定。以下是一些可扩展的方向（属于 Out of Scope，但作为课外思考）：
1. 发钉钉/飞书/企业微信机器人消息 — 在 Runnable 里调 HTTP API
2. 发邮件 — Spring Mail + Thymeleaf 模板
3. 写数据库 — 建一张 monitor_event_log 表，持久化状态变化
4. 触发 Prometheus AlertManager — 调 AlertManager 的 HTTP API
5. 发短信/电话 — 对接云厂商的通知服务

Q1. doNotify() 为什么返回 Mono<Void> 而不是 void？

A. 因为 Spring 要求所有方法返回 Mono B. SBA 通知器体系基于 Project Reactor 响应式编程，用 Mono 支持非阻塞异步处理。多个 Notifier 可并发执行，慢操作不阻塞事件总线 C. 因为 Java 不支持 void 返回类型 D. 为了兼容 Kotlin 的协程
看答案B。SBA 内部事件总线是响应式的，所有通知器返回 Mono 以支持非阻塞调度。如果你的通知逻辑有慢 I/O（发邮件、调 HTTP），可以通过 .subscribeOn(Schedulers.boundedElastic()) 把慢操作挪到弹性线程池，不阻塞事件分发。

Q2. InstanceStatusChangedEvent 和 InstanceRegisteredEvent 有什么区别？

A. 完全相同，只是命名不同 B. RegisteredEvent 是新实例首次注册时触发，StatusChangedEvent 是实例状态发生改变（UP→DOWN 等）时触发 C. RegisteredEvent 只在 Windows 上触发 D. StatusChangedEvent 只在使用 MySQL 时触发
看答案B。RegisteredEvent 在实例首次向 Server 注册时触发一次。StatusChangedEvent 在每次状态发生变化（如 UP→DOWN、DOWN→UP）时触发——这是一个高频事件，是本 Notifier 关注的核心事件。

Q3. switch 的 default 分支返回 "未知状态"，它的防御作用是什么？

A. 没有实际作用，纯粹为了凑够代码行数 B. 如果 SBA 未来版本新增了状态码，不会因为缺少 case 分支而导致 switch 异常，日志仍能打印 C. 为了兼容 Java 8 D. 为了让 IDE 停止显示警告
看答案B。防御性编程的最佳实践——default 分支确保即使出现未知状态码，switch 表达式也有一个合法返回值，不会抛异常。同时日志仍然会被打印，只是状态名为「未知状态」。

Q4. STATUS_UP -> "服务上线" 和 STATUS_DOWN -> "服务下线" 这两行翻译的业务考量是什么？

A. 没有区别，只是为了让日志看起来好看 B. UP 表示实例 /health 返回正常，DOWN 表示 /health 返回不健康（如数据库断开），两个状态反映的是服务的可用性级别，运维人员能据此判断严重程度 C. UP 比 DOWN 多一个字母 D. 它们只是枚举值，日志里都会打印原始状态码
看答案B。「上线/下线」的区分在运维场景中有明确业务含义：(1) DOWN 比 OFFLINE 更严重——DOWN 说明进程可能还在但功能坏了，OFFLINE 是正常的注销；(2) 运维人员看到「服务下线」会直接冲去排查数据库/Redis 连接，看到「服务离线」可能是计划内重启或缩容。中文翻译让非技术人员也能直观判断。
