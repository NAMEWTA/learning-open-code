第 1 课 · SaToken 模块全景 认证架构

本节目标：理解 ruoyi-common-satoken 在平台中的定位、五个 Java 类的职责分工，以及一次登录请求的宏观流程。

在 RuoYi-Vue-Plus 中，用户从输入密码到访问受保护接口，中间经过一套完整的认证鉴权管线。这套管线的骨干就是 ruoyi-common-satoken 模块 — 它不是从零手写的认证框架，而是在 Sa-Token 的基础上做了 JWT 化、Redis 持久化、Caffeine 加速等工程化加固。

一、为什么选 Sa-Token + JWT？

Java 生态里做认证，Spring Security 是事实标准，但它有两个问题：配置繁琐（需要写 Filter Chain、Provider、UserDetailsService 等一整套）和 Session 依赖（默认基于 Tomcat Session，分布式需要额外改造）。

Sa-Token 的设计哲学不同：

对比维度Spring SecuritySa-Token
上手成本高，需理解 Filter Chain 模型低，一行 StpUtil.login() 即可
分布式支持需额外集成 Spring Session内置 Dao 层抽象，换存储即可
权限模型Role-based，扩展需自定义权限码 + 角色 双模型，开箱即用
踢人下线需手写逻辑StpUtil.kickout() 一行代码
JWT 集成需引入 jose4j/nimbus 等三方库官方插件 sa-token-jwt，无缝切换

RuoYi-Vue-Plus 选择 Sa-Token + JWT 的组合，本质是用轻量框架替代重型方案，同时通过 JWT 获得无状态 Token 的自解释能力——Token 本身携带 userId 等信息，不依赖集中式 Session，但又通过 Redis 保留了服务端可控性（可随时踢人、冻结）。

二、模块目录与五个类的分工

打开模块目录，结构极其精简 — 5 个 Java 文件，各司其职：

ruoyi-common-satoken/
├── pom.xml                          # 依赖：sa-token-core + jwt + caffeine
└── src/main/
    ├── java/org/dromara/common/satoken/
    │   ├── config/
    │   │   └── SaTokenConfig.java           # [装配] 注册 4 个 Bean，模块入口
    │   ├── core/
    │   │   ├── dao/
    │   │   │   └── PlusSaTokenDao.java      # [存储] Redis + Caffeine 双层缓存
    │   │   └── service/
    │   │       └── SaPermissionImpl.java    # [权限] 提供用户权限码和角色列表
    │   ├── handler/
    │   │   └── SaTokenExceptionHandler.java # [异常] 全局翻译认证异常为中文提示
    │   └── utils/
    │       └── LoginHelper.java             # [工具] 登录/退出/用户信息获取
    └── resources/
        └── common-satoken.yml               # [配置] Token 前缀、读取方式等
类/文件角色一句话职责
SaTokenConfig装配者注册 StpLogic(JWT)、StpInterface(权限)、SaTokenDao(存储)、异常处理器
PlusSaTokenDao存储层替代默认内存存储，用 Caffeine + Redis 实现分布式 Token 持久化
SaPermissionImpl权限源每次鉴权时被 Sa-Token 回调，返回当前用户的权限码和角色列表
SaTokenExceptionHandler翻译官捕获 NotLoginException 等异常，转为前端可读的中文错误信息
LoginHelper门面工具封装登录流程（写入用户信息到 TokenSession），提供 getLoginUser() 等便捷方法
common-satoken.yml默认配置Bearer 前缀、允许从 Header/参数读取 Token、关闭 Cookie 鉴权

三、认证流程鸟瞰

一次完整的登录到鉴权链路如下：

┌──────────┐ 登录请求 ┌──────────────┐ 写入TokenSession ┌──────────────┐ │ 前端 │ ────────────> │ LoginHelper │ ──────────────────> │ PlusSaTokenDao│ │ POST /login│ loginUser │ .login(user) │ setObject(...) │ Caffeine+Redis│ └──────────┘ └──────────────┘ └──────────────┘ │ │ StpLogicJwtForSimple │ 生成 JWT Token ▼ ┌──────────────┐ │ 返回 Token │ "Bearer eyJhbG..." └──────────────┘ │ ┌─────────────────────────┘ ▼ ┌──────────────┐ 携带Token ┌──────────────────┐ 回调权限列表 ┌──────────────────┐ │ 前端 │ ────────────> │ Sa-Token 拦截器 │ ────────────> │ SaPermissionImpl │ │ Authorization│ 请求接口 │ checkPermission() │ │ getPermissionList│ └──────────────┘ └──────────────────┘ └──────────────────┘ │ │ │ 403/401 │ 通过则放行 ▼ ▼ ┌──────────────┐ ┌──────────────┐ │ ExceptionHandler│ │ Controller │ │ 中文错误提示 │ │ 业务逻辑 │ └──────────────┘ └──────────────┘

关键设计：Token 本身是 JWT，包含基础用户信息（可自解释），但完整的 LoginUser 对象（含部门、权限列表）存储在 Redis 的 TokenSession 中。这兼顾了 JWT 的无状态优势和 Redis 的服务端可控性。

四、与 ruoyi-common-sms 的架构对比

你之前学过的 sms 模块与本模块有相似的"装配 + DAO + 异常处理"结构，但定位不同：

对比维度ruoyi-common-smsruoyi-common-satoken
领域短信发送认证鉴权
核心框架阿里云/腾讯云/云片 SDKSa-Token
DAO 缓存内存 Map 缓存（短信重试间隔）Caffeine + Redis 双层缓存
配置入口SmsAutoConfigurationSaTokenConfig
异常处理SmsExceptionHandler（业务异常）SaTokenExceptionHandler（认证异常 + 业务异常）
工具类SmsUtils（发送/校验）LoginHelper（登录/用户信息）
注意：sms 的 PlusSmsDao 只用内存 Map 做限流缓存（数据非关键、丢了也无妨），而 satoken 的 PlusSaTokenDao 用 Redis 做持久化 + Caffeine 做读加速（数据关键、必须可靠）。两种 DAO 设计体现了不同业务诉求下的存储策略差异。

🤔 想一想

为什么 Sa-Token 需要一个 SaTokenDao 接口来抽象存储？直接用 RedisTemplate 不行吗？

框架解耦：Sa-Token 框架本身不知道你的项目用的是 Redis、MongoDB 还是内存。通过 SaTokenDao 接口，框架核心逻辑与存储实现解耦。RuoYi 只需实现这个接口（PlusSaTokenDao），框架就用 Redis；换成别的项目可能实现一套基于数据库的 Dao。
测试友好：单元测试时可以注入一个基于 Map 的 Dao 实现，不需要启动 Redis。
多存储策略：你可以写一个「Redis 主存储 + Caffeine 读缓存」的 Dao（这正是 PlusSaTokenDao 的做法），框架完全不感知这些优化。

JWT Token 自带了 userId，为什么还要从 Redis 的 TokenSession 里读 LoginUser？

JWT 不可撤销：JWT 一旦签发，在过期前无法主动使其失效。如果管理员踢人，Token 本身不会变，只有 Redis 里的 TokenSession 被清除了。
数据量限制：把所有权限码、部门信息、角色列表都塞进 JWT，Token 会变得非常臃肿（HTTP Header 有大小限制）。Redis 存全量数据，JWT 只存关键标识，是最佳平衡。

提取练习

Q1. 以下哪个类负责将 Sa-Token 的 NotLoginException 翻译为中文提示？
A. SaTokenConfig
B. SaPermissionImpl
C. SaTokenExceptionHandler
D. LoginHelper
看答案 C。SaTokenExceptionHandler 是 @RestControllerAdvice，专门捕获 Sa-Token 异常并返回中文消息。A 是配置类，B 是权限实现，D 是登录工具类。
Q2. RuoYi-Vue-Plus 使用的 JWT 模式对应的 StpLogic 实现类是？
A. StpLogicForDefault
B. StpLogicJwtForMixin
C. StpLogicJwtForSimple
D. StpLogicJwtForStyle
看答案 C。SaTokenConfig 中注入的正是 StpLogicJwtForSimple。Simple 模式由 Sa-Token 完全接管 JWT 生成，Mixin 模式需要用户自己实现部分 JWT 逻辑。A 是默认模式（随机字符串 Token），D 是虚构的选项。
Q3. 以下哪个文件声明了"关闭 Cookie 鉴权，杜绝 CSRF 风险"的配置？
A. SaTokenConfig.java
B. application.yml
C. common-satoken.yml
D. PlusSaTokenDao.java
看答案 C。common-satoken.yml 中配置了 is-read-cookie: false，关闭 Cookie 读取 Token。A 是 Java 配置类（注册 Bean），B 是主配置文件，D 是 DAO 实现。
Q4. PlusSaTokenDao 使用 Caffeine 的主要目的是什么？
A. 替代 Redis，降低运维成本
B. 减少对 Redis 的读取次数，降低 QPS 压力
C. 提供更强的数据一致性保证
D. 存储 Token 的完整生命周期数据
看答案 B。Caffeine 是读缓存加速层，Redis 是主存储。每次鉴权都查 Redis 会带来巨大 QPS 压力，Caffeine 挡在前面吸收热数据读。A 错误（Redis 仍是主存储），C 错误（多级缓存会降低一致性），D 错误（完整数据仍在 Redis）。
