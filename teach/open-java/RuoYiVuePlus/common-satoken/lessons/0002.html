第 2 课 · 配置与 DAO — SaTokenConfig 与 PlusSaTokenDao 深度解析 存储与配置

本节目标：理解 SaTokenConfig 四个 Bean 的注册逻辑，掌握 PlusSaTokenDao 的 Caffeine + Redis 双层缓存读写策略，并与 PlusSmsDao 对比两种 DAO 设计思路。

第 1 课我们建立了模块全景视图。这一节聚焦两个核心文件：SaTokenConfig（装配入口）和 PlusSaTokenDao（存储引擎）。前者决定了"框架跑起来的姿势"，后者决定了"每毫秒认证请求的数据路径"。

一、SaTokenConfig：四个 Bean 的装配逻辑

SaTokenConfig 是一个 Spring Boot @AutoConfiguration 类，通过 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件声明，由 Spring Boot 自动加载。

类上有两个关键注解：

@AutoConfiguration
@PropertySource(factory = YmlPropertySourceFactory.class, value = "classpath:common-satoken.yml")

@PropertySource 配合自定义 YmlPropertySourceFactory 将 common-satoken.yml 加载到 Spring 环境变量，使 Sa-Token 框架能读取到自定义配置。

类内部注册了 4 个 @Bean，每一个都在替代 Sa-Token 的默认行为：

Bean 方法返回类型实例替代默认逻辑的原因
getStpLogicJwt() StpLogic new StpLogicJwtForSimple() 默认 StpLogic 生成随机字符串 Token；JWT 模式下改为签发 JWT Token（简单模式，框架完全接管生成逻辑）
stpInterface() StpInterface new SaPermissionImpl() 框架默认无权限实现；注入自定义实现后，每次鉴权时回调获取当前用户的权限码和角色列表
saTokenDao() SaTokenDao new PlusSaTokenDao() 默认用内存 Map 存储 Token，单机可用但分布式不可共享；替换为 Caffeine + Redis 双层存储
saTokenExceptionHandler() SaTokenExceptionHandler handler 实例 框架默认异常处理返回英文错误信息；替换为中文提示，前端可直接展示

设计模式：Sa-Token 框架通过"接口 + 默认实现 + Bean 覆盖"的方式实现了策略模式。你不需要修改框架源码，只需向容器注册自己的 Bean，Spring 的依赖注入自动将你的实现替换默认实现。这正是 Spring Boot AutoConfiguration 的经典用法。

二、common-satoken.yml 配置详解

这是模块内置的默认配置（可被 Nacos 同名配置覆盖）：

# 内置配置 不允许修改 如需修改请在 nacos 上写相同配置覆盖
# Sa-Token配置
sa-token:
  # 允许动态设置 token 有效期
  dynamic-active-timeout: true
  # 允许从 请求参数 读取 token
  is-read-body: true
  # 允许从 header 读取 token
  is-read-header: true
  # 关闭 cookie 鉴权 从根源杜绝 csrf 漏洞风险
  is-read-cookie: false
  # token前缀
  token-prefix: "Bearer"

逐条说明每一项的含义：

配置项值含义
dynamic-active-timeout true 动态设置 Token 有效期。开启后，每次操作 Token 时都会自动续期，实现"活跃不过期"的效果。比如用户 30 分钟内一直在操作，Token 就不会过期
is-read-body true 允许从请求体中读取 Token。适用于 POST/PUT 请求中 Token 放在 JSON 或表单参数里的场景
is-read-header true 允许从 HTTP Header 读取 Token。标准做法是 Authorization: Bearer eyJhbG...
is-read-cookie false 关闭 Cookie 鉴权。Cookie 会被浏览器自动携带，攻击者可通过 CSRF 伪造请求。关闭后在根源上杜绝 CSRF 漏洞风险
token-prefix "Bearer" Token 前缀。框架在解析时会自动去掉此前缀，前端请求时需加此前缀（如 Bearer eyJhbG...）
安全提醒：is-read-cookie: false 是安全关键配置。Cookie 鉴权模式下，浏览器会自动在每次请求中带上 Cookie，攻击者只需诱使用户点击一个链接即可发起 CSRF 攻击。关闭 Cookie 后，前端必须主动在 Header 中设置 Token，攻击者无法跨域伪造这个 Header。

三、PlusSaTokenDao：双层缓存的读写策略

PlusSaTokenDao 实现了 SaTokenDaoBySessionFollowObject 接口，这是 Sa-Token 框架的 DAO 抽象层。它的核心工程决策是 Caffeine（本地缓存）+ Redis（网络存储）双层架构：

3.1 Caffeine 缓存配置

new CaffeineCache(
    cacheName,
    Caffeine.newBuilder()
        .expireAfterWrite(5, java.util.concurrent.TimeUnit.SECONDS)  // 写入后5秒过期
        .initialCapacity(100)                                         // 初始容量100
        .maximumSize(1000)                                            // 最大容量1000
        .build(),
    false,  // 不允许存null值
    true    // 微秒级时间戳（用于判断缓存新鲜度）
);

三个参数的含义：

  • 5 秒过期：本地缓存的生命周期极短。这意味着最终一致性窗口不超过 5 秒
  • 初始 100 / 最大 1000：控制内存占用，1000 条缓存条目在 JVM 中几乎无感
  • 不允许 null：避免缓存穿透 — null 值不会被放入 Caffeine，每次都回源 Redis

3.2 读取流程（get / getObject）

所有读操作最终调用私有方法 getCacheValue(key)，流程如下：

读请求 (key = "Authorization:login:token:xxx") │ ├─ 1. Caffeine.get(key) ── 命中？──> 直接返回 │ └─ 2. 未命中 → RedisUtils.getCacheObject(key) │ ├─ 3. Redis 有值？──> put into Caffeine → 返回数据 │ └─ 4. Redis 也无值？──> return null

3.3 写入流程（set / setObject）

所有写操作最终调用 writeValue(key, value, timeout)：

写请求 (key + value + TTL) │ ├─ 1. RedisUtils.setCacheObject(key, value, timeout) │ └─ timeout < 0？永久有效 | timeout > 0？定时过期 │ └─ 2. Caffeine.invalidate(key) ← 关键：让本地缓存失效 └─ 下次读时从 Redis 重新加载，保证数据一致性

为什么不直接更新 Caffeine？ 如果写操作同时更新 Caffeine 和 Redis，分布式环境下不同节点的 Caffeine 内容不同，无法保证一致性。让 Caffeine 失效 + 读时回源是分布式缓存的经典模式 — "Cache-Aside" 策略。

3.4 搜索操作（searchData）

searchData(prefix, keyword, start, size, sortType) 用于模糊搜索 Token（例如管理员查看所有在线用户）。它也使用了 Caffeine 做结果缓存，减轻 RedisUtils.keys(pattern) 的性能压力。

3.5 TTL 精度补偿（toTimeoutSeconds）

从 Redis 获取的过期时间通常是毫秒级，转换为秒时存在精度损失。方法中使用了 +1 补偿：

// Redis 返回的毫秒 TTL 转为秒，+1 补偿精度损失
private long toTimeoutSeconds(Long ttl) {
    if (ttl == null || ttl < 0) return ttl;
    return (ttl / 1000) + 1;  // 向上取整补偿
}

这样确保 Token 有效期不会因为取整而被低估，避免用户 Token 提前失效。

四、与 PlusSmsDao 的对比

两个模块都有 DAO 层，但设计思路截然不同，体现了"不同业务需求 → 不同存储策略"的工程决策：

对比维度PlusSmsDao (sms)PlusSaTokenDao (satoken)
存储后端内存 ConcurrentHashMapRedis（主）+ Caffeine（读缓存）
数据性质短信发送间隔的临时限流数据Token / Session 等持久认证数据
读策略直接 Map.get()Caffeine 查 → Redis 查 → null
写策略直接 Map.put()Redis 写入 + Caffeine.invalidate()
过期机制调用方自行控制，无 TTLRedis TTL 精确控制
一致性单机强一致（仅本 JVM 可见）最终一致（最多 5 秒窗口）
分布式支持不支持（每节点独立计数）天然支持（Redis 共享存储）
数据安全性丢失无所谓（进程重启即清空）关键数据（Redis 持久化 + AOF）
核心启示：PlusSmsDao 存的是"发短信的频控记录"，丢了最多让用户等 60 秒重新发一次，代价极低；PlusSaTokenDao 存的是"你是谁、你有多少权限"，丢了意味着用户被踢下线或权限失效，代价极高。两种 DAO 的设计差异正是"业务数据价值"决定"存储策略投入"的体现。

想一想

为什么写操作不更新 Caffeine 而是让它失效？

分布式一致性问题：在多实例部署下，一个写请求只会到达一个节点。如果写操作直接更新该节点的 Caffeine，其他节点的 Caffeine 仍然保留旧数据，造成节点间数据不一致。
失效 + 回源（Cache-Aside）模式：让本地缓存失效，下次任何节点读取时都从 Redis（共享数据源）重新加载。这样保证了"所有节点的数据最终都从同一个源头（Redis）获得"，实现了最终一致性。
成本考量：如果要保证强一致，需要广播失效消息到所有节点（如 Redis Pub/Sub 或 MQ），增加系统复杂度。5 秒的最终一致性窗口对于 Token 有效期管理来说足够实用。

Caffeine 配置 5 秒过期，如果用户在这 5 秒内被管理员踢下线会怎样？

最坏情况：5 秒窗口期：管理员调用 StpUtil.kickout(userId) 后，Redis 中的 Token 立即被删除。但其他节点（或同一节点）的 Caffeine 缓存可能仍有该 Token 的副本。
实际影响：在 Caffeine 过期前的 5 秒内，持有该 Token 的用户仍可能通过鉴权。5 秒后缓存过期，下次鉴权请求回源 Redis 发现 Token 已不存在，返回 null，用户被实际拦截。
工程取舍：这是一个典型的 CP vs AP 取舍 — 选择 AP（可用性 + 性能），牺牲了 CP 中的强一致性。对于绝大多数业务场景，踢人后 5 秒内完全踢下线是可接受的。如果业务要求实时踢人，需要引入 Redis Pub/Sub 广播缓存失效消息。

提取练习

Q1. SaTokenConfig 注册了哪个 Bean 来替代 Sa-Token 默认的随机字符串 Token？
A. SaTokenDao
B. StpInterface
C. StpLogic（StpLogicJwtForSimple）
D. SaTokenExceptionHandler
看答案 C。getStpLogicJwt() 方法注册了 StpLogicJwtForSimple，替代默认的 StpLogic（随机字符串模式），使框架使用 JWT 签发 Token。A 是存储层，B 是权限回调，D 是异常处理。
Q2. PlusSaTokenDao 读取数据时的回源顺序是什么？
A. Redis → Caffeine → null
B. Caffeine → Redis → null
C. Redis → null
D. Caffeine → null
看答案 B。先查本地 Caffeine 缓存，命中则直接返回；未命中则回源 Redis，Redis 有值则更新 Caffeine 后返回；Redis 也无值则返回 null。
Q3. 为什么 common-satoken.yml 中 is-read-cookie 设为 false？
A. Cookie 传输效率低
B. 禁用 Cookie 鉴权，从根源杜绝 CSRF 风险
C. Cookie 无法跨域传递
D. Sa-Token 不支持 Cookie 方式
看答案 B。Cookie 会被浏览器自动携带，攻击者可通过 CSRF 伪造请求。设置 is-read-cookie: false 后 Token 必须通过 Header 或请求体传递，攻击者无法跨域设置，从根源上杜绝了 CSRF 漏洞。
Q4. PlusSaTokenDao.set() 方法写入 Redis 后对 Caffeine 做了什么？
A. put() — 同时更新 Caffeine
B. refresh() — 异步刷新 Caffeine
C. invalidate() — 让本地缓存失效
D. 不操作 Caffeine
看答案 C。写操作调用 writeValue() 方法，写入 Redis 成功后执行 invalidate(key) 让本地缓存失效。下次读时从 Redis 重新加载，保证数据一致性（Cache-Aside 模式）。
