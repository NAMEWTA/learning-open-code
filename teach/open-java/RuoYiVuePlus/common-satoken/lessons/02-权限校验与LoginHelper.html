02 - 权限校验与 LoginHelper

🎯 本节目标：追踪登录从 Controller 到 Token 写入 Redis 的完整调用链，理解 SaPermissionImpl 的权限获取机制，能区分 NotLoginException 的五种子类型，掌握 LoginHelper 在项目中的桥接作用。

1. 全景鸟瞰

上一课讲了 SaTokenConfig 的静态配置和基础设施。本课聚焦运行时——用户登录时发生了什么、每次请求如何鉴权、权限校验失败时异常怎么处理。这是认证模块的「动态面」。

2. 核心设计

2.1 登录流程：从 Controller 到 Redis

完整链路：

1. Controller: SysLoginService.login(username, password)
   → 验证用户名密码 → 查数据库获取 SysUser
   → 构建 LoginUser 对象（userId, userName, deptId, 角色列表, 权限列表）

2. LoginHelper.login(loginUser):
   → StpUtil.login(userId)                              // ① 框架登录
     → StpLogicJwtForSimple.createLoginSession()
       → 生成 JWT Token
       → Token 存入 PlusSaTokenDao → Redis + Caffeine
   → StpUtil.getTokenSession()                          // ② 获取 Token Session
     → 写入用户详情:
       session.set("deptId", loginUser.getDeptId())
       session.set("loginUser", loginUser)              // ← LoginUser 对象
   → 返回 JWT Token 字符串

3. Controller: 返回 Token 给前端
   → 前端存入 localStorage / Cookie
   → 后续请求带 Header: Authorization: Bearer <token>
Token Session vs Account Session：Token Session 按 Token 维度隔离——同一账号多设备登录时，每台设备的 Token Session 独立。Account Session 按账号维度——所有设备共享。LoginHelper 使用 Token Session 存放用户信息，因为每个登录会话需要独立管理（踢人时只踢特定设备）。

2.2 SaPermissionImpl：权限与角色的动态获取

实现 StpInterface 接口，框架在每次权限校验时自动回调：

@Component
public class SaPermissionImpl implements StpInterface {

    @Override
    public List<String> getPermissionList(Object loginId, String loginType) {
        // 从 Token Session 中取出 LoginUser
        LoginUser loginUser = LoginHelper.getLoginUser();
        // 返回权限码列表: ["system:user:list", "system:user:add", ...]
        return loginUser.getMenuPermission();
    }

    @Override
    public List<String> getRoleList(Object loginId, String loginType) {
        LoginUser loginUser = LoginHelper.getLoginUser();
        // 返回角色列表: ["admin", "common"]
        return loginUser.getRolePermission();
    }
}

调用时机：每次请求中遇到 @SaCheckPermission("system:user:list") 时，框架自动调用 getPermissionList() → 检查返回列表中是否包含 "system:user:list" → 包含则放行，否则抛 NotPermissionException。

性能考虑：每次鉴权都调这个方法，如果每次都查数据库会很慢。但 LoginUser 已经在登录时查过一次并存入 Token Session——getPermissionList 直接从内存对象中取，无 DB IO。这就是 Token Session 缓存的价值。

2.3 LoginHelper：项目与框架的桥接器

Sa-Token 自己的 StpUtil 提供了所有认证 API，但 LoginHelper 封装了一层项目适配：

方法底层调用项目适配的内容
login(LoginUser)StpUtil.login() + TokenSession写入 deptId、LoginUser 完整对象到 Token Session
getLoginUser()TokenSession.get("loginUser")从 Session 取出并类型转换为 LoginUser
getUserId()StpUtil.getLoginIdAsLong()转换为 Long 类型（框架默认返回 String）
getDeptId()TokenSession.get("deptId")读取登录时写入的部门 ID
isAdmin()getLoginUser().isAdmin()判断是否为超级管理员（跳过数据权限的关键）
isLogin()StpUtil.isLogin()检查当前请求是否已认证
// 业务代码中的典型用法：
LoginUser user = LoginHelper.getLoginUser();
Long currentUserId = LoginHelper.getUserId();
if (LoginHelper.isAdmin()) {
    // 超级管理员：查全部数据
} else {
    // 普通用户：只查自己的数据（数据权限自动过滤）
}

2.4 NotLoginException 的五种子类型

当用户未登录或登录状态异常时，Sa-Token 抛出 NotLoginException。不同子类型对应不同的异常场景：

子类型触发场景HTTP 状态码前端提示建议
NOT_TOKEN请求未携带 Token401跳转登录页
INVALID_TOKENToken 无效（伪造/格式错误）401跳转登录页
TOKEN_TIMEOUTToken 已过期401提示"登录已过期，请重新登录"
BE_REPLACED被顶下线（另一设备登录了同账号）401提示"您的账号在其他设备登录"
KICK_OUT被管理员踢下线401提示"您已被管理员强制下线"

SaTokenExceptionHandler 捕获这些异常并返回对应的中文提示：

@ExceptionHandler(NotLoginException.class)
public R<Void> handleNotLogin(NotLoginException e) {
    String message;
    switch (e.getType()) {
        case NOT_TOKEN:     message = "未登录"; break;
        case TOKEN_TIMEOUT: message = "登录已过期"; break;
        case BE_REPLACED:   message = "账号在其他设备登录"; break;
        case KICK_OUT:      message = "您已被强制下线"; break;
        default:            message = "未登录或登录已过期";
    }
    return R.fail(HttpStatus.HTTP_UNAUTHORIZED, message);
}

2.5 NotPermissionException：权限不足

触发时机：@SaCheckPermission("system:user:delete") 标注的方法被调用，但 SaPermissionImpl.getPermissionList() 返回的列表中不包含 "system:user:delete"。

异常信息包含缺失的权限码，便于调试：无此权限：system:user:delete

3. 实战练习

🧠 提取练习（点击展开）

Q1: LoginHelper 为什么需要用 getLoginUser() 从 Token Session 取数据，而不是直接在类中存静态引用？

参考: 因为是 Web 多线程环境——每个请求在不同的线程中处理。LoginHelper 的操作都是基于当前请求的 Token → 当前请求的 Session。如果存静态引用，所有请求会共享一个 LoginUser，用户的请求会拿到别人的登录信息——这是严重的生产事故。Sa-Token 的 StpUtil 内部用 ThreadLocal 隔离，LoginHelper 通过 StpUtil 间接获取当前线程的登录信息。

Q2: 为什么 SaPermissionImpl 的 getPermissionList 可以直接从 LoginUser 对象取权限，而不是每次查数据库？

参考: 权限列表在登录时已经查过一次（LoginService.login() 构建 LoginUser 时注入了 menuPermission 和 rolePermission 列表），并存储在 Token Session 中。每次鉴权时从 Token Session 取 LoginUser 对象（读 Caffeine/Redis），无需 DB IO。代价是权限变更有延迟——管理员改了角色权限后，已登录用户需要在 Token 过期重新登录后才能获得新权限。这是「性能」与「实时性」的权衡。

Q3: BE_REPLACED 和 KICK_OUT 都导致 token 失效，但它们有什么区别？

参考: BE_REPLACED 是「被自己顶替」——同一账号在新设备登录，旧设备被自动踢下线（is-concurrent=false 时触发）。KICK_OUT 是「被管理员踢」——管理员在后台调用 StpUtil.kickout(userId) 主动踢人。区别前端提示：BE_REPLACED → "账号在其他设备登录"（属于正常行为），KICK_OUT → "您已被管理员强制下线"（属于人工干预）。

4. 调用链路

// 场景：用户登录并访问需要权限的接口

1. POST /login { username, password }
   → SysLoginService.login()
     → 验证密码 → 查数据库 → 构建 LoginUser
     → LoginHelper.login(loginUser)
       → StpUtil.login(userId) → 生成 JWT
       → TokenSession.set("loginUser", loginUser)
       → TokenSession.set("deptId", deptId)
     → 返回 JWT Token

2. GET /api/system/user/list (Header: Authorization: Bearer <jwt>)
   → Sa-Token 全局过滤器
     → 解析 JWT → 提取 loginId
     → 读取 Token Session → 获取 LoginUser
   → @SaCheckPermission("system:user:list")
     → SaPermissionImpl.getPermissionList(1, "login")
       → LoginHelper.getLoginUser().getMenuPermission()
       → ["system:user:list", "system:user:add", ...]
       → 包含 "system:user:list" ✓ → 放行

3. GET /api/system/user/delete (Header: Authorization: Bearer <jwt>)
   → ...同上...
   → @SaCheckPermission("system:user:delete")
     → getPermissionList() → ["system:user:list", "system:user:add"]
     → 不包含 "system:user:delete" ✗
     → throw NotPermissionException("无此权限：system:user:delete")
     → SaTokenExceptionHandler → 返回 403: "无此权限"
💡 设计洞察：SaTokenConfig 的四个 Bean 中，StpInterface 的 SaPermissionImpl 是「权限校验的策略注入点」——它将「怎么查权限」的决策权交给项目，而不是框架硬编码。这样项目可以根据自身数据模型（RBAC、ABAC、自定义权限表）来提供权限列表，框架只管匹配。LoginHelper 则是对 StpUtil 的项目化封装——它屏蔽了框架细节（如 getLoginIdAsLong 的类型转换），提供了业务友好的 API（如 getDeptId、isAdmin），让业务代码不用直接依赖 Sa-Token API。
