第 5 课 · 实战 — 完整登录认证链路追踪与模式切换 实战闭环

本节目标：独立追踪从 POST /login 到 @SaCheckPermission 的完整调用链，理解 JWT 模式与 Redis 模式的切换原理，能独立完成认证相关问题的排查。

前面四节课我们逐一剖析了 SaTokenConfig、PlusSaTokenDao、SaPermissionImpl、SaTokenExceptionHandler 和 LoginHelper 五个类的内部实现。本节课将它们串联成一条完整的链路，并对比 JWT 模式与 Redis 模式的设计差异 — 这是从"看懂代码"到"能排查问题"的关键跨越。

一、登录全链路追踪

从用户点击"登录"按钮到前端拿到 Token，后端经历了以下调用链：

1
POST /login → SysLoginController.login()
入口：ruoyi-admin 模块的 Controller 层
2
SysLoginService.login() 校验用户名/密码/验证码
业务层：验证码校验 → 用户查询 → 密码匹配 → 状态检查
3
构建 LoginUser 对象
封装 userId、userName、deptId、roles、permissions 等信息
4
LoginHelper.login(loginUser) — sa-token 模块正式介入
位于 ruoyi-common-satoken 模块

4a fillRequestContext(loginUser) — 填充 IP、地理位置、浏览器、操作系统信息到 LoginUser 的扩展字段

4b StpUtil.login(loginUser.getLoginId(), parameter) — 调用 Sa-Token 核心登录

  • Sa-Token 调用 StpLogicJwtForSimple.createLoginSession()
  • StpLogic 通过 sa-token-jwt 插件生成 JWT Token（Header.Payload.Signature）
  • StpLogic 调用 PlusSaTokenDao.setObject() 将 TokenSession 数据写入存储层
  • PlusSaTokenDao 先写 Redis（持久化），再失效 Caffeine 中对应的旧缓存

4c 在 TokenSession 中写入扩展数据：userId、userName、deptId、deptName、deptCategory

4d 调用 LoginHelper.setLoginUser(loginUser) 将完整的 LoginUser 对象写入 TokenSession（含部门列表、角色列表、权限码列表）

5
返回 Token 给前端
格式：String 类型，带 "Bearer " 前缀或原始 JWT 字符串 (eyJhbG...)

注意执行顺序：fillRequestContext 先于 StpUtil.login 执行。因为 fillRequestContext 只是将请求上下文信息写入 LoginUser 对象（纯内存操作），而 StpUtil.login 会触发 JWT 生成和 Redis 写入（涉及网络 I/O）。登录成功后 TokenSession 中才有数据，所以 4c 和 4d 必须在 login() 之后执行。

二、Token 校验与权限鉴权链路

登录成功后，前端每次请求都携带 Token，后端拦截器的处理流程如下：

┌──────────┐ Authorization: Bearer {jwt} ┌─────────────────────┐ │ 前端 │ ────────────────────────────────> │ Sa-Token 拦截器 │ └──────────┘ └─────────────────────┘ │ ① 从 Header 读取 Token（is-read-header: true） │ ② 剥离 "Bearer " 前缀 │ ③ StpLogicJwtForSimple.getTokenValue() 解析 JWT │ ④ 使用配置的 secret-key 验证 JWT 签名 │ ⑤ 从 JWT Payload 中提取 loginId │ ⑥ PlusSaTokenDao.getObject() 读取 TokenSession │ ├─ 先查 Caffeine 本地缓存（快速路径） │ └─ 未命中 → 查 Redis → 回填 Caffeine → 返回 │ │ 当遇到 @SaCheckPermission("system:user:list") 时： │ │ ⑦ Sa-Token 调用 SaPermissionImpl.getPermissionList() │ ⑧ SaPermissionImpl 通过 LoginHelper 从 TokenSession │ 获取 LoginUser 对象 │ ⑨ 返回 LoginUser 中存储的权限码列表 │ ⑩ Sa-Token 检查 "system:user:list" 是否在列表中 │ ├─ 在列表中 → 放行到 Controller │ └─ 不在列表中 → 抛出 NotPermissionException │ → SaTokenExceptionHandler 捕获 │ → 返回 403 + 中文错误提示 │

这条链路中，性能关键路径在步骤 6：Caffeine 命中时，一次鉴权只涉及 CPU 计算（验签）+ 本地缓存读取（纳秒级）。Redis 只在 Caffeine 未命中时被访问，这是 PlusSaTokenDao 双层缓存设计的核心价值。

三、JWT 模式 vs Redis 模式的切换原理

RuoYi-Vue-Plus 当前使用 JWT 模式，但架构上支持一键切换到 Redis 模式（传统 Token 模式）。两种模式的核心差异在于 StpLogic 实现类的选择：

维度JWT 模式（当前）Redis 模式
Token 格式JWT 三段式（eyJhbG...）随机字符串（uuid 风格）
StpLogic 实现StpLogicJwtForSimpleStpLogicForDefault
Token 验证方式本地验签，纯 CPU 计算，不依赖 Redis必须查询 Redis 验证 Token 是否存在
踢人下线机制JWT 本身不过期，靠清除 Redis 中 TokenSession 使后续请求拿不到用户数据直接删除 Redis 中的 Token 记录，请求立即失效
性能高（验签是纯 CPU 操作，无网络 I/O）中（每次请求都需要 Redis 查询）
服务端可控性中（JWT 签发后内容不可修改，只能通过 Redis 侧数据间接控制）高（Redis 中的数据可随时修改、删除、冻结）

如何切换？

切换过程极其简单，只需要两步：

  1. 移除依赖：从 pom.xml 中删除 sa-token-jwt 依赖
  2. 切换 Bean：修改 SaTokenConfig.java，将注入的 Bean 从 StpLogicJwtForSimple 替换为 StpLogicForDefault
// JWT 模式（当前）
@Bean
public StpLogic stpLogic() {
    return new StpLogicJwtForSimple();  // JWT Token
}

// 切换为 Redis 模式
@Bean
public StpLogic stpLogic() {
    return new StpLogicForDefault();    // 随机字符串 Token
}

其他代码完全不变 — PlusSaTokenDao、SaPermissionImpl、SaTokenExceptionHandler、LoginHelper 都不需要任何修改。这正是 Sa-Token 框架 Dao 层抽象和 StpLogic 多态设计的威力：上层业务代码零感知，底层存储和 Token 生成机制随意切换。

为什么 RuoYi 选择 JWT 模式？ 微服务网关场景是关键原因。在网关层（如 Spring Cloud Gateway），网关可以直接验证 JWT 签名并提取用户信息，无需将请求转发到认证服务。如果使用 Redis 模式，网关每次验签都必须回调 Redis，在网络延迟和架构复杂度上都不可接受。JWT 的"自解释"特性天然适合网关鉴权。

🤔 想一想

JWT 模式下，管理员踢人后前端仍持有 JWT Token，为什么用户还是无法访问接口？

JWT Token 本身仍然有效（签名正确、未过期），但 Redis 中的 TokenSession 已被清除。

当用户再次请求时，链路走到「PlusSaTokenDao.getObject() 读取 TokenSession」步骤 — Caffeine 已被踢人操作主动失效，Redis 中也无数据，返回 null。后续代码（如 SaPermissionImpl、LoginHelper）尝试从 TokenSession 获取 LoginUser 时得到 null，最终抛出 NotLoginException，由 SaTokenExceptionHandler 转换为"登录已过期，请重新登录"返回前端。

本质：JWT 模式下的踢人不是"让 Token 失效"，而是"让 Token 对应的服务端数据消失"。Token 是一个钥匙，但锁芯（TokenSession）被拆了，钥匙自然打不开门。

如果要在网关层做认证（不转发到后端），为什么必须选 JWT 模式？

网关层无法访问 Redis。 微服务架构中，网关通常是独立的服务，它不一定有 Redis 的连接权限，也不应该依赖 Redis 来做鉴权（这会引入不必要的耦合和网络延迟）。

JWT 模式的优势在于：网关只需拿到 JWT Token，用本地配置的 secret-key 验签，就能从 Payload 中提取 loginId、验证有效期，整个过程零网络调用。验证通过后，网关将解析出的用户信息通过 Header 传给下游服务，下游服务直接信任即可。

如果使用 Redis 模式，网关每次请求都得连接 Redis 查询 Token 是否存在 — 这在高并发场景下会成为性能瓶颈，且将 Redis 暴露给网关增加了安全风险面。

提取练习

Q1. 登录链路中，fillRequestContext() 和 StpUtil.login() 哪个先执行？
A. StpUtil.login() 先执行，因为需要先生成 Token
B. fillRequestContext() 先执行，因为它是纯内存操作，先准备好用户上下文信息
C. 两者并发执行，互不依赖
D. 顺序无关紧要
看答案 B。fillRequestContext() 填充 IP、地理位置、浏览器、操作系统到 LoginUser 对象（纯内存操作），StpUtil.login() 触发 JWT 生成和 Redis 写入（涉及网络 I/O）。先做轻量操作再触发重量操作是合理的设计。另外 fillRequestContext 填充的信息会随 LoginUser 存入 TokenSession，所以必须在 login() 之前完成。
Q2. JWT 模式下，管理员踢人后会发生什么？
A. JWT Token 被立即标记为失效，前端请求直接被拦截器拒绝
B. JWT Token 仍然有效，但 Redis 中的 TokenSession 被清除，后续请求无法获取用户数据而报错
C. 前端会自动删除本地的 JWT Token
D. 后端会向前端推送踢人通知，前端主动跳转到登录页
看答案 B。JWT 的无状态特性决定了它一旦签发就无法被服务端主动失效。踢人的本质是清除 Redis 中的 TokenSession 和失效 Caffeine 缓存，使后续鉴权流程在获取用户数据时返回 null，从而触发 NotLoginException。
Q3. 以下哪个类负责生成 JWT Token？
A. PlusSaTokenDao
B. SaPermissionImpl
C. StpLogicJwtForSimple
D. LoginHelper
看答案 C。StpLogicJwtForSimple 是 Sa-Token JWT 插件的核心实现类，负责 JWT 的生成（createLoginSession）、解析（getTokenValue）和验签。A 是存储层，B 是权限源，D 是封装工具类（它调用 StpUtil，进而委托给 StpLogicJwtForSimple）。
Q4. 如何从 JWT 模式切换到 Redis 模式？
A. 重写 PlusSaTokenDao，改用不同的存储逻辑
B. 修改 application.yml 中的 sa-token.mode 配置项
C. 移除 sa-token-jwt 依赖，将 SaTokenConfig 中注入的 StpLogic 改为 StpLogicForDefault
D. 替换整个 ruoyi-common-satoken 模块
看答案 C。切换只需两步：1) 移除 pom.xml 中的 sa-token-jwt 依赖；2) 将 SaTokenConfig 中 stpLogic() 方法的返回值从 StpLogicJwtForSimple 改为 StpLogicForDefault。PlusSaTokenDao、SaPermissionImpl、LoginHelper 等全部无需修改，体现了 Sa-Token 框架良好的分层解耦。
