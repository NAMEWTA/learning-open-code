第 3 课 · 权限验证 — SaPermissionImpl 权限码校验全流程 权限鉴权

本节目标：理解 StpInterface 的回调机制，掌握 SaPermissionImpl 的两种权限获取策略（本地提取 vs 远程调用），追踪一个 @SaCheckPermission 注解的完整校验链路。

上一节我们了解了 SaTokenConfig 如何装配模块和 PlusSaTokenDao 如何做双层缓存。这一节深入权限校验的核心 — 当你给 Controller 方法加上 @SaCheckPermission("system:user:list") 后，Sa-Token 到底走了哪些逻辑来判定你是否有这个权限？

一、StpInterface：Sa-Token 的权限回调契约

Sa-Token 框架本身不知道你的用户有哪些权限。它定义了一个接口 StpInterface，每次需要检查权限时，就回调你的实现类来获取权限数据：

public interface StpInterface {

    /**
     * 返回指定 loginId 所拥有的权限码集合
     */
    List<String> getPermissionList(Object loginId, String loginType);

    /**
     * 返回指定 loginId 所拥有的角色标识集合
     */
    List<String> getRoleList(Object loginId, String loginType);
}

三个关键点你必须记住：

要点说明
回调时机不是只在登录时调用一次，而是每次鉴权都回调。Sa-Token 内部有缓存优化（默认缓存 5 秒），但即便是缓存过期后的下一次鉴权，仍然会再次回调。
权限数据源模式这是典型的"框架提问，你回答"模式。Sa-Token 不关心权限从哪里来 — 数据库、配置文件、RPC 调用都可以 — 你只需在接口实现里返回结果。
两个方法getPermissionList 返回权限码（如 system:user:list），getRoleList 返回角色标识（如 admin）。两者独立，互不干扰。

在 SaTokenConfig 中，SaPermissionImpl 正是作为这个接口的实现注册给 Sa-Token 的：

@Bean
public StpInterface stpInterface() {
    return new SaPermissionImpl();
}
思考：为什么 Sa-Token 设计成"每次回调"，而不是"登录时一次加载、后续从 Session 中读取"？因为权限可能变更 — 管理员在后台修改了某用户的权限，如果只在登录时加载，该用户必须重新登录才能生效。每次回调（配合缓存）可以在可控的延迟内反映权限变更。

二、SaPermissionImpl 的两种权限获取策略

打开 SaPermissionImpl.java 的 getPermissionList 方法，你会发现它并不简单 — 内部有两套策略来获取权限码：

策略 1：快速路径 — 从当前 LoginUser 中直接提取

// 策略 1: 优先从当前登录用户的 LoginUser 对象中提取
LoginUser loginUser = LoginHelper.getLoginUser();
if (loginUser != null &&
    StringUtils.equals(loginUser.getUserId().toString(), resolveUserId(loginId))) {
    // 当前登录用户就是被检查的用户，直接从 LoginUser.permissions 中取
    return loginUser.getMenuPermission();
}

这个分支的逻辑非常直接：

  • 通过 LoginHelper.getLoginUser() 拿到当前线程绑定的 LoginUser 对象（它是登录时写入 TokenSession 的）
  • 比较当前登录用户的 userId 和被检查的 loginId 是否一致
  • 如果一致，直接从 loginUser.getMenuPermission() 返回权限码列表 — 零网络开销、零数据库查询

覆盖 99% 的场景：绝大多数情况下，用户访问接口时检查的是自己的权限。这个快速路径避免了每次鉴权都去查数据库或调远程服务。

策略 2：降级路径 — 通过 PermissionService 远程获取

// 策略 2: 降级 — 通过远程 PermissionService 获取权限
try {
    PermissionService permissionService =
        SpringContextHolder.getBean(PermissionService.class);
    return permissionService.getMenuPermission(Long.valueOf(resolveUserId(loginId)));
} catch (NoSuchBeanDefinitionException e) {
    throw new ServiceException("PermissionService bean not found — " +
        "确保 ruoyi-framework 模块已引入", e);
}

这个分支用于跨用户权限检查（例如管理员想检查某个普通用户是否有某权限）：

  • 通过 SpringContextHolder.getBean() 动态获取 PermissionService Bean（注意是动态查找，不是 @Autowired）
  • 调用 getMenuPermission(userId) 远程查询指定用户的权限码
  • 如果 PermissionService 不存在（比如模块未引入），抛出明确的 ServiceException
为什么用 SpringContextHolder 动态查找而不是 @Autowired 注入？
ruoyi-common-satoken 是 common 模块，它不应该对 ruoyi-framework 有编译期依赖（Maven 依赖方向是 framework → common，不能反向）。但运行时 framework 确实提供了 PermissionService 实现，所以用运行时动态查找来解耦。

三、权限校验的完整时序

下面用一个 @SaCheckPermission("system:user:list") 注解追踪一次完整的权限校验链路：

@SaCheckPermission("system:user:list") │ ▼ Sa-Token 拦截器 (StpLogic.checkPermission) │ ▼ StpUtil.checkPermission("system:user:list") │ ▼ 检查缓存: permissionListCache (默认 5s TTL) │ ┌────┴────┐ │ 命中 → 直接返回权限列表 │ 未命中 → 继续 │ └─────────┘ │ ▼ SaPermissionImpl.getPermissionList(loginId, loginType) │ ├──[路径1: 本地LoginUser]──→ 从 LoginUser.permissions 中提取 │ │ │ ┌────┴────┐ │ │ userId 匹配 → 直接返回权限列表 │ │ │ userId 不匹配 → 走路径2 │ │ └─────────────────────────┘ │ └──[路径2: 远程PermissionService]──→ getMenuPermission(userId) │ ┌────┴────┐ │ Bean 存在 → RPC 查询权限 │ │ Bean 不存在 → 抛异常 │ └─────────────────────────┘ │ ▼ 得到权限码列表 ["system:user:list", "system:user:add", ...] │ ▼ contains("system:user:list") ? │ ┌────┴────┐ │ 是 → 放行，进入 Controller │ 否 → 403 NotPermissionException │ └──────────────────────────────────┘

四、resolveUserId 的用户类型解析

SaPermissionImpl 里还有一个关键方法 resolveUserId，它负责从 Sa-Token 的 loginId 中提取数字 userId：

private String resolveUserId(Object loginId) {
    // loginId 格式: "type:userId"，例如 "sys_user:1"
    return loginId.toString().split(":")[1];  // 取冒号后面的 userId 部分
}

这个设计源于 RuoYi-Vue-Plus 的多用户类型架构：

用户类型loginId 示例解析出的 userId
后台管理员sys_user:11
App 端用户app_user:4242

与此相反的操作在 LoginHelper 中 — getUserType() 方法解析 loginId 的类型前缀（冒号前的部分）来获取 UserType 枚举：

public static UserType getUserType() {
    String loginId = LoginHelper.getLoginId();
    // "sys_user:1" → "sys_user" → UserType.SYS_USER
    return UserType.getUserType(loginId.split(":")[0]);
}

设计优势：这种双段式 loginId 格式（类型:ID）让一个 Sa-Token 实例可以同时服务多种用户体系。Sa-Token 框架本身不关心 loginId 的内容是什么，只要它是唯一的即可。类型前缀是 RuoYi 自己加的语义层。

🤔 想一想

为什么 SaPermissionImpl 优先从 LoginUser 提取权限，而不是每次都远程调用？

性能与延迟：远程调用（即使是 Dubbo 或 HTTP RPC）每次都有网络往返延迟（通常 1-10ms），加上序列化/反序列化开销。而对于 99% 的日常请求，被检查的用户就是当前登录用户本人，LoginUser 对象已经存在于线程上下文或 TokenSession 中，直接读取是纯内存操作，纳秒级耗时。

可用性：如果每次都依赖远程 PermissionService，一旦那个服务不可用，整个系统的鉴权都会失败（单点故障）。本地提取策略让最常见场景不依赖外部服务，大幅提高鉴权链路的可靠性。

缓存设计：即使远程调用，Sa-Token 也会缓存权限列表 5 秒。但本地提取连这个缓存查询都省了 — 直接从 LoginUser 对象里读，是比缓存更快的"热数据"。

如果微服务环境下 PermissionService 在另一个服务里，当前实现能直接工作吗？

能工作，但需要正确配置。SaPermissionImpl 通过 SpringContextHolder.getBean(PermissionService.class) 获取 Bean，只要这个 Bean 在 Spring 容器中存在即可。在微服务架构中，PermissionService 通常是一个 Dubbo 或 Feign 的远程调用代理，它作为 Spring Bean 注入到当前服务的容器中。

潜在问题：如果网络抖动导致 RPC 调用超时，鉴权也会超时。建议对远程 PermissionService 调用添加熔断降级（如 Sentinel），在远程服务不可用时使用本地缓存的权限数据或默认拒绝。

另外：SaPermissionImpl 的策略 1（本地提取）本身就是一种"降级设计" — 只要当前用户检查的是自己的权限，就不会触发远程调用，天然规避了微服务间通信的问题。

提取练习

Q1. SaPermissionImpl 实现了哪个接口，使其能被 Sa-Token 框架回调？
A. SaTokenDao
B. StpInterface
C. StpLogic
D. PermissionService
看答案 B。SaPermissionImpl 实现 StpInterface 接口，重写 getPermissionList 和 getRoleList 方法。A 是存储层接口，B 是权限回调接口，C 是认证逻辑接口，D 是业务层权限服务。
Q2. getPermissionList 方法的返回值类型是什么？
A. Set<String>
B. String[]
C. List<String>
D. Collection<Permission>
看答案 C。StpInterface 的 getPermissionList 返回 List<String>，每个字符串是一个权限码，如 "system:user:list"。Sa-Token 通过 contains() 判断用户是否拥有指定权限。
Q3. 对于 loginId = "sys_user:1"，resolveUserId 返回什么？
A. "sys_user"
B. "1"
C. "sys_user:1"
D. 抛出异常
看答案 B。resolveUserId 按 ":" 分割后取索引 [1]，即冒号后面的数字部分 "1"。如果需要解析出用户类型（"sys_user"），应使用 LoginHelper.getUserType()。
Q4. SaPermissionImpl 获取权限的两种策略分别是什么？
A. 从数据库查询 / 从 Redis 缓存读取
B. 从 LoginUser 本地提取 / 调用 PermissionService 远程获取
C. 从 JWT Token 解析 / 从 Session 中读取
D. 从配置文件读取 / 从注解中获取
看答案 B。策略 1 是从当前登录用户的 LoginUser 对象中本地提取权限（快速路径，覆盖 99% 场景），策略 2 是通过 PermissionService 远程获取指定用户的权限（降级路径，用于跨用户权限检查）。
