第 4 课 · 异常处理与登录工具 工具与边界

本节目标：掌握 SaTokenExceptionHandler 对五类 NotLoginException 的区分处理逻辑，熟练使用 LoginHelper 进行登录、用户信息获取、租户切换等日常操作。

在认证系统中，异常处理不是可有可无的配角——它决定了用户看到的是"系统异常，请联系管理员"还是一句能指导下一步行动的提示。同时，开发者在日常业务代码中几乎不会直接调用 StpUtil，而是通过 LoginHelper 这个门面工具完成所有登录态操作。本节把这两块一起讲透。

一、SaTokenExceptionHandler：认证异常的翻译官

SaTokenExceptionHandler 是一个标注了 @RestControllerAdvice 的全局异常处理器。它的职责非常单一：把 Sa-Token 框架抛出的英文异常类型翻译成用户能看懂的中文消息。

类中只有两个处理方法：

1
handleNotPermissionException — 权限不足异常
捕获 NotPermissionException 和 NotRoleException，统一返回 403，消息为"没有访问权限，请联系管理员授权"。
来源：SaTokenExceptionHandler.java
2
handleNotLoginException — 未登录异常（区分五种场景）
捕获 NotLoginException，根据 exception.getType() 做 switch 分支，返回 401，不同场景返回不同消息。
来源：SaTokenExceptionHandler.java

五种 NotLoginException 子类型的对照表：

异常类型触发场景返回给用户的消息用户应做的操作
TOKEN_TIMEOUTToken 超过有效期（如 30 分钟无操作）"登录已过期，请重新登录"跳转登录页，重新输入密码
BE_REPLACED同一账号在另一设备登录，当前 Token 被顶替"当前账号已在其他设备登录，您已被强制下线"确认是否本人操作，重新登录
KICK_OUT管理员在后台将用户踢下线"账号已被管理员强制下线"联系管理员了解原因
TOKEN_FREEZE管理员冻结了该账号"账号已被冻结，请联系管理员处理"联系管理员申请解冻
default其他未知登录异常（如 Token 格式非法、已被删除等）"登录状态异常，请重新登录"重新登录

所有异常处理方法最终都调用 R.fail(HttpStatus.HTTP_UNAUTHORIZED, message) 返回统一响应格式。R 是 RuoYi 平台的响应包装类，HttpStatus.HTTP_UNAUTHORIZED 对应 HTTP 401 状态码。

设计要点：为什么不像大部分项目那样写一个笼统的 ExceptionHandler？因为不同的"未登录"需要用户采取不同的行动。Token 过期只需重新登录；被顶替需要确认安全性；被冻结需要联系管理员。前端根据不同消息可以展示不同的 UI（如弹窗提示 vs 跳转登录页 vs 展示客服联系方式）。

二、LoginHelper：登录操作的门面工具

LoginHelper 是一个工具类，标注了 @NoArgsConstructor(access = AccessLevel.PRIVATE)，表示不对外暴露构造器，所有方法都是静态方法。它是开发者日常编码中使用频率最高的认证工具类。

类中定义的关键常量（对应 TokenSession 中存储数据的 key）：

常量名实际值用途
LOGIN_USER_KEY"loginUser"TokenSession 中存储完整 LoginUser 对象的 key
USER_KEY"userId"TokenSession extra 数据中存储 userId 的 key
USER_NAME_KEY"userName"TokenSession extra 数据中存储 userName 的 key
DEPT_KEY"deptId"TokenSession extra 数据中存储部门 ID 的 key
DEPT_NAME_KEY"deptName"TokenSession extra 数据中存储部门名称的 key
DEPT_CATEGORY_KEY"deptCategory"TokenSession extra 数据中存储部门类别的 key
CLIENT_KEY"clientKey"客户端标识（如 PC、APP、H5）
CLIENT_ACCESS_PATH_KEY"clientAccessPath"客户端访问路径
CLIENT_IP_WHITELIST_KEY"clientIpWhitelist"IP 白名单

三、login() 方法详解

LoginHelper 提供了两个重载的 login() 方法：login(LoginUser loginUser) 和 login(LoginUser loginUser, SaLoginParameter parameter)。后者允许配置额外参数（如是否允许并发登录、Token 过期时间等）。

一次 login() 调用内部做了以下四件事：

1
fillRequestContext(loginUser) — 自动补全登录终端信息
填充当前请求的 IP 地址、登录地点（通过 IP 地理定位）、浏览器类型（User-Agent 解析）、操作系统、设备类型（手机/桌面/平板）。这些信息写入 LoginUser 对象的属性中，用于审计追溯。
LoginHelper.fillRequestContext()
2
StpUtil.login(loginUser.getLoginId(), parameter) — 执行 Sa-Token 登录
调用 Sa-Token 的登录 API，生成 Token 并在服务端记录会话。这一步完成后，用户即处于"已登录"状态。
Sa-Token StpUtil.login()
3
设置 TokenSession extra 数据 — 写入常用字段
调用 StpUtil.getTokenSession().set(key, value) 将 userId、userName、deptId、deptName、deptCategory 写入 TokenSession 的 extra data。这些字段被高频访问，存为 extra 数据可以避免每次都需要反序列化整个 LoginUser 对象。
TokenSession.set()
4
存储完整 LoginUser 对象
调用 StpUtil.getTokenSession().setObject(LOGIN_USER_KEY, loginUser) 将整个 LoginUser 对象（含权限列表、角色列表等）存入 TokenSession。当需要完整用户信息时（如鉴权），再从 TokenSession 中取出。
TokenSession.setObject(LOGIN_USER_KEY, loginUser)
为什么同时存 extra 数据和 LoginUser 对象？extra 数据存的是常用标量字段（userId、userName 等），读取时零反序列化开销；LoginUser 对象存的是完整用户信息（含权限列表、部门树等），只在需要时反序列化。这是典型的热数据分离策略——高频访问的字段直接取，低频访问的完整对象按需加载。

四、用户信息获取方法

LoginHelper 提供了丰富的静态方法来获取当前登录用户的各种信息：

方法签名返回类型说明
getLoginUser()LoginUser从 TokenSession 中通过 LOGIN_USER_KEY 反序列化获取完整 LoginUser 对象
getLoginUser(String token)LoginUser根据指定 Token 获取对应用户的 LoginUser（不限于当前请求用户）
getUserId()Long从 TokenSession extra 数据中通过 USER_KEY 获取 userId
getUserIdStr()String同 getUserId()，但返回字符串类型
getUsername()String从 TokenSession extra 数据中通过 USER_NAME_KEY 获取用户名
getDeptId()Long从 TokenSession extra 数据中通过 DEPT_KEY 获取部门 ID
getDeptName()String从 TokenSession extra 数据中通过 DEPT_NAME_KEY 获取部门名称
getDeptCategory()String从 TokenSession extra 数据中通过 DEPT_CATEGORY_KEY 获取部门类别
getUserType()String获取当前登录的用户类型（如 sys_user、app_user 等多用户体系）
isSuperAdmin()boolean判断当前用户是否为超级管理员。检查 userId 是否在超级管理员列表中（默认 userId=1）
isLogin()boolean判断当前会话是否已登录（调用 StpUtil.isLogin()）

isSuperAdmin() 的实现原理：平台通常会定义一个超级管理员 ID 列表（如配置文件中指定 super-admin-ids: [1]），该方法获取当前 userId 后判断是否在列表中。超级管理员绕过所有权限校验，拥有全部接口的访问权。

五、fillRequestContext：登录终端的自动补全

fillRequestContext(LoginUser loginUser) 是 login() 调用的第一个步骤，它在用户登录时自动捕获并记录当前请求的终端环境信息：

填充字段数据来源示例值
IP 地址HttpServletRequest.getRemoteAddr() 或 X-Forwarded-For 头192.168.1.100
登录地点IP 地理定位服务（如 IP2Region 或在线 API）北京市 联通
浏览器类型User-Agent 请求头解析Chrome 120
操作系统User-Agent 请求头解析Windows 10 / macOS 14
设备类型User-Agent 请求头解析Desktop / Mobile / Tablet

这些信息最终写入 LoginUser 对象并随 TokenSession 持久化到 Redis。它们的核心价值在于审计追溯——管理员可以在后台看到每个登录会话的来源 IP、地点、设备，当出现安全事件时能快速定位异常登录。

登录时自动采集终端信息： HttpServletRequest │ ├──→ getRemoteAddr() ──→ IP: 192.168.1.100 ├──→ IP 地理定位服务 ──→ 地点: 北京市·联通 └──→ User-Agent 解析 ──→ 浏览器: Chrome 120 ──→ 操作系统: macOS 14 ──→ 设备类型: Desktop 写入 LoginUser 对象 → TokenSession → Redis 持久化 → 后台审计可查

🤔 想一想

为什么 NotLoginException 要分五种类型，而不统一返回"未登录"？

不同的"未登录"需要不同的用户行动：Token 过期只需重新登录；被顶替（BE_REPLACED）说明账号在别处登录，用户需要确认是否本人操作；被踢下线（KICK_OUT）和被冻结（TOKEN_FREEZE）是管理员主动操作，用户需要联系管理员。如果统一返回"未登录"，用户反复登录失败却不知道原因，体验极差。
前端可以根据异常类型做差异化处理：Token 过期 → 静默跳转登录页；被顶替 → 弹窗提示"您的账号在其他设备登录"；被冻结 → 展示管理员联系方式。这比笼统的错误处理要专业得多。

LoginHelper.login() 为什么把 userId/deptName 同时存为 TokenSession 的 extra 数据和 LoginUser 对象的属性？

性能优化——热数据分离：extra 数据（userId、userName、deptId 等）是标量字段，读取时不需要反序列化，O(1) 直接取出。LoginUser 对象包含权限列表、角色列表等复杂结构，反序列化成本高。
访问频率不同：获取当前用户 ID 或部门 ID 是极高频率操作（几乎每个接口都会调用），每次都反序列化整个 LoginUser 对象会造成不必要的 CPU 开销。extra 数据就是为这些高频字段设计的快速通道。
两者互补：extra 数据覆盖 90% 的日常查询场景；需要完整用户信息（如鉴权时需要权限列表）时，再从 TokenSession 取出 LoginUser 对象。

提取练习

Q1. 当用户账号在另一设备登录，当前 Token 被顶替时，SaTokenExceptionHandler 返回的提示是什么？异常类型是？
A. "登录已过期，请重新登录" — TOKEN_TIMEOUT
B. "当前账号已在其他设备登录，您已被强制下线" — BE_REPLACED
C. "账号已被管理员强制下线" — KICK_OUT
D. "账号已被冻结，请联系管理员处理" — TOKEN_FREEZE
看答案 B。BE_REPLACED 是 Sa-Token 的顶替登录机制——当同一账号在新设备登录时，旧设备的 Token 会被标记为"被替换"。A 是 Token 过期，C 是管理员踢人，D 是账号冻结。
Q2. fillRequestContext() 在登录时自动填充以下哪些信息？
A. 用户密码的哈希值
B. IP 地址、登录地点、浏览器类型、操作系统、设备类型
C. 用户的权限码列表
D. JWT Token 的过期时间
看答案 B。fillRequestContext 专门负责从 HttpServletRequest 和 User-Agent 中提取终端环境信息，用于审计追溯。权限码在登录之前就已加载到 LoginUser 中，Token 过期时间由 Sa-Token 的 SaLoginParameter 控制。
Q3. LoginUser 完整对象在 TokenSession 中通过哪个常量 key 存储？
A. USER_KEY ("userId")
B. USER_NAME_KEY ("userName")
C. LOGIN_USER_KEY ("loginUser")
D. DEPT_KEY ("deptId")
看答案 C。LOGIN_USER_KEY（值为 "loginUser"）存储完整的 LoginUser 对象。A、B、D 都是存储到 TokenSession extra 数据中的标量字段 key。
Q4. isSuperAdmin() 方法的判断逻辑是什么？
A. 检查用户是否拥有 "admin" 角色
B. 检查 userId 是否在超级管理员 ID 列表中（默认 userId=1）
C. 检查用户名是否为 "admin"
D. 检查用户是否拥有所有权限码
看答案 B。isSuperAdmin() 通过判断当前登录用户的 userId 是否在配置的超级管理员 ID 列表中来决定。默认情况下只有 userId=1（即系统初始管理员账号）是超级管理员。超级管理员绕过所有权限校验，而非通过角色或权限码来判断。
