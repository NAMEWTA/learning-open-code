第 1 课 · 全景鸟瞰：ruoyi-common-redis 到底给了你什么 入门地图

🎯 本节目标（对应使命 Success #1）：学完你能用一张图，讲清这个模块的 5 大区域（config / handler / aspectj / manager / utils）各管什么、谁依赖谁，并说出每块的关键类。这是后面每节细节课挂靠的骨架——先有地图，再走小路。

很多人对 RuoYi 的「缓存模块」印象模糊：好像有个 RedisUtils 能存东西，好像有个 @RateLimiter 能限流……但说不清它整体是什么。这节课我们把整个 ruoyi-common-redis 摊开，一次看全。全程基于本仓库真实代码。

一、先记住一句话：它是「Redisson 的业务封装层」

整个模块只做一件事：把 Redisson 这个强大但偏底层的库，封装成 RuoYi 业务代码能直接拿来用的几样能力。看 pom.xml 的依赖就懂它的「原材料」：

redisson-spring-boot-starter   ← 核心：操作 Redis 的客户端
redisson-spring-cache          ← 把 Redis 接进 Spring Cache 抽象
caffeine                       ← 本地一级缓存（JVM 内存）
lock4j-redisson-...-starter    ← 分布式锁（本模块只接管它的异常）
ruoyi-common-json / sa-token   ← 序列化、token 名读取等协作
设计动机 · 为什么是 Redisson 而不是 RedisTemplate？
Spring 自带的 RedisTemplate 只把 Redis 当「键值存储」。而 Redisson 把 Redis 的能力包装成了 Java 程序员熟悉的对象：分布式的 Map、Set、Queue、AtomicLong、限流器、发号器、锁……RuoYi 选 Redisson，就是为了让「用 Redis」像「用一个 Java 集合」一样自然。这是理解整个模块的总钥匙。

二、全景图：5 大区域

org.dromara.common.redis │ ├── config/ 【配置区】启动时装配，定义"地基规则" │ ├── RedisConfig ← Redisson 序列化/前缀/单机集群 (最重要) │ ├── CacheConfig ← @EnableCaching + Caffeine + 缓存管理器 │ ├── RateLimiterConfig ← 注册限流切面 Bean │ ├── IdempotentConfig ← 注册防重提交切面 Bean │ └── properties/RedissonProperties ← 映射 yml 的 redisson.* 配置 │ ├── handler/ 【处理器区】挂在框架扩展点上的小钩子 │ ├── KeyPrefixHandler ← 给所有 key 自动加/去项目前缀 │ └── RedisExceptionHandler ← 兜底 Lock4j 锁失败异常 │ ├── aspectj/ 【切面区】把注解变成真正的拦截行为 │ ├── RateLimiterAspect ← 处理 @RateLimiter（限流） │ └── RepeatSubmitAspect ← 处理 @RepeatSubmit（防重提交） │ ├── annotation/ 【注解区】给业务方法贴的"标签" │ ├── RateLimiter ← @RateLimiter │ └── RepeatSubmit ← @RepeatSubmit │ ├── enums/ LimitType ← 限流类型：DEFAULT / IP / CLUSTER │ ├── manager/ 【缓存管理区】二级缓存的核心 │ ├── PlusSpringCacheManager ← 自定义 Spring CacheManager │ └── CaffeineCacheDecorator ← 给 Redis 缓存套一层本地缓存 │ └── utils/ 【工具区】业务最常直接调用的入口 ├── RedisUtils ← 通用读写/限流/发布订阅/原子值 ├── CacheUtils ← 操作 Spring Cache 命名空间 ├── QueueUtils ← 分布式队列 └── SequenceUtils ← 分布式发号器（生成单号）

这 5 块的关系可以这样理解：config 打地基 → handler/aspectj 装扩展点 → manager 搭二级缓存 → utils 给业务开门。日常写业务你 90% 时间只碰 utils 和那两个注解；但要读懂「为什么能用」，得往下看到 config。

三、逐区域走一遍（每块抓一个关键点）

① config · 地基：RedisConfig 三处定制

config/RedisConfig.java

它通过 RedissonAutoConfigurationCustomizer 定制 Redisson，记住三处就够：

  • 序列化：CompositeCodec(StringCodec, jsonCodec, jsonCodec) —— key 用纯字符串、value 用 JSON。这样你在 redis-cli 里能看懂 key，value 又能存复杂对象。
  • key 前缀：setNameMapper(new KeyPrefixHandler(...)) —— 所有 key 自动加项目前缀（见 handler 区）。
  • 单机/集群双模式：读 RedissonProperties，哪个配置非空就用哪个（singleServerConfig / clusterServersConfig）。
一个安全彩蛋：代码注释里专门讲了 Jackson 3.x 因安全策略禁用了 LaissezFaireSubTypeValidator，改用 BasicPolymorphicTypeValidator 防反序列化漏洞。当前默认放行所有类型，但官方建议改成包名白名单 allowIfBaseType("org.dromara")。这是反序列化攻击防护的典型现场。

② handler · KeyPrefixHandler：给 key 戴帽子

handler/KeyPrefixHandler.java

实现 Redisson 的 NameMapper 接口，只有两个方法：map() 存时加前缀，unmap() 取时去前缀。前缀为空就什么都不做。

为什么需要它？多个项目/环境共用一个 Redis 实例时，前缀（如 ruoyi:）能避免 key 撞车，也方便按前缀批量清理。它对业务代码完全透明——你写 setCacheObject("foo")，redis 里实际是 ruoyi:foo。

同区的 RedisExceptionHandler 是个 @RestControllerAdvice，专门把 Lock4j 的 LockFailureException 兜成「业务处理中，请稍后再试」的友好响应——抢锁失败不该让用户看到 500。

③ aspectj + annotation · 两个注解，两种保护

aspectj/RateLimiterAspect.java · RepeatSubmitAspect.java

这是模块里最"聪明"的部分。两个注解贴在 Controller 方法上，切面在方法执行前拦截：

@RateLimiter 限流@RepeatSubmit 防重提交
意图控制访问频率（保护系统不被刷爆）防止同一次操作被重复提交（保护数据）
底层Redisson RRateLimiter 令牌桶setObjectIfAbsent（SETNX + 过期）
key 由什么组成URI + (IP/实例) + SpEL 解析的业务 keyURI + token + 请求参数的 MD5
触发后抛 ServiceException(限流提示)抛 ServiceException(重复提示)

两个切面都用 SpEL 表达式动态拼 key（如 @RateLimiter(key="#user.id")），都支持国际化消息（{rate.limiter.message} 这种花括号格式会被 MessageUtils 翻译）。

防重的精妙之处：RepeatSubmitAspect 用 setObjectIfAbsent（不存在才写入）抢占一个 key。请求成功（返回码 SUCCESS）时保留这个 key 直到过期——保证间隔内无法再提交；失败或异常时立即删除——允许用户马上重试。这套"成功留键、失败删键"是幂等设计的经典手法。

④ manager · 二级缓存：Caffeine（本地） + Redis（分布式）

manager/PlusSpringCacheManager.java · CaffeineCacheDecorator.java

CacheConfig 用 @EnableCaching 打开 Spring 缓存注解（@Cacheable 等），并装配自定义的 PlusSpringCacheManager。它在原版 Redisson 缓存管理器上加了两个关键能力：

  • 缓存名带参数：缓存名支持 cacheName#ttl#maxIdle#maxSize#local 格式。比如 @Cacheable(cacheNames="user#30s#100") 就能直接在注解里声明这个缓存 30 秒过期、最多 100 条——不用改配置类。
  • 本地一级缓存：末位 #local=1（默认开）时，用 CaffeineCacheDecorator 把 Redis 缓存包一层 Caffeine。读先走 JVM 本地内存（30 秒过期），命中就不查 Redis。
为什么要两级？纯 Redis 缓存每次都要走网络。对"读多写少、能容忍 30 秒陈旧"的数据（如字典、配置），加一层本地 Caffeine 能把大量读请求挡在 JVM 内，大幅降低 Redis 压力和延迟。写入时 CaffeineCacheDecorator 会先 invalidate 本地副本再写 Redis，避免脏读。

⑤ utils · 业务最常用的四扇门

utils/RedisUtils.java · CacheUtils.java · QueueUtils.java · SequenceUtils.java
工具类定位（什么时候用它）底层对象
RedisUtils最通用。存取对象/List/Set/Map、设过期、限流、发布订阅、原子计数RBucket/RList/RMap/RAtomicLong/RTopic
CacheUtils手动操作 Spring Cache 命名空间（配合 @Cacheable 的那套缓存）Spring CacheManager
QueueUtils轻量分布式队列（普通/优先级队列、订阅消费）。重量级用 MQRBlockingQueue
SequenceUtils生成分布式唯一单号（如 P20260630000001）RIdGenerator
易混点 · RedisUtils vs CacheUtils：RedisUtils 是"裸操作 Redis"，key 是你自己拼的；CacheUtils 操作的是"被 Spring Cache 管理的命名空间缓存"，和 @Cacheable 注解是同一套数据。两者都用 Redis，但抽象层级不同——选错会导致"注解缓存和手动缓存对不上"。

四、为什么值得先记这张图？

RuoYi 的缓存能力乍看散落在十几个类里，但骨架极其规整：一套配置（config）定地基 → 一批扩展点（handler/aspectj）加能力 → 一层缓存管理（manager）做加速 → 一组工具（utils）开门。只要抓住「config 定规则、utils 是入口、注解管保护、manager 管加速」这四个锚点，后面无论深入限流令牌桶、防重幂等还是二级缓存一致性，都只是往这张图上挂细节。

🤔 想一想：为什么 RedisUtils / QueueUtils / SequenceUtils 都把 RedissonClient 存成 private static final，而不用 @Autowired 注入？ 因为这些是静态工具类（构造器私有，全是 static 方法），无法用实例字段注入。它们用 SpringUtils.getBean(RedissonClient.class) 在类加载时主动从 Spring 容器拿到客户端。代价是：这些工具类必须在 Spring 容器就绪后才能用——这也是为什么它们属于"业务运行期"工具，而非"启动配置期"组件。

五、检验一下（提取练习）

Q1. 这个模块的"原材料/基石"库是哪个？

A. Spring 自带的 RedisTemplate，模块只是简单封装了一下 B. Redisson，模块把它封装成业务可直接用的缓存/限流/队列等能力 C. Caffeine，所有缓存都存在 JVM 本地内存里 D. Lock4j，模块核心是分布式锁，其余都是附属功能
看答案B。模块全程用 Redisson 的 RedissonClient（见 RedisUtils 等的 CLIENT 字段），把 Redis 包装成 Java 程序员熟悉的对象。Caffeine 只是二级缓存的本地层，Lock4j 只被异常处理器接管。

Q2. @RateLimiter 和 @RepeatSubmit 的核心区别是什么？

A. 前者用 Redis、后者用 Caffeine，存储介质不同 B. 前者控制访问频率防刷爆系统，后者防止同一操作被重复提交防数据错 C. 前者用在前端、后者用在后端，运行位置不同 D. 没有本质区别，只是两个名字不同的相同功能
看答案B。限流用令牌桶控"频率"；防重用 setIfAbsent 占位控"同一次操作的重复"。两者都基于 Redis、都在切面里抛 ServiceException，但保护意图完全不同。

Q3. 业务里想读取一个"用 @Cacheable 注解缓存的用户对象"，应该用哪个工具？

A. RedisUtils.getCacheObject(key)，因为它最通用 B. CacheUtils.get(cacheNames, key)，因为它操作的是 Spring Cache 命名空间 C. QueueUtils.getQueueObject(key)，从队列里取 D. 直接 new 一个 RedissonClient 自己查
看答案B。@Cacheable 的数据归 Spring CacheManager 管，CacheUtils 才是同一套抽象的入口。用 RedisUtils 裸读可能因 key 拼法、序列化不一致而对不上。

Q4. 缓存名写成 "dict#1h#500" 是什么意思？

A. 三个不同的缓存名，用 # 分隔 B. 缓存名 dict、TTL 1 小时、最大 500 条，参数直接写在缓存名里 C. dict 缓存的第 1 个版本，哈希值 500 D. 非法写法，缓存名不能包含 # 符号
看答案B。PlusSpringCacheManager.getCache() 按 # 拆分 cacheName#ttl#maxIdle#maxSize#local，让你不改配置类就能在注解里声明缓存策略。这是该模块对原版 Redisson 缓存管理器的核心增强。
