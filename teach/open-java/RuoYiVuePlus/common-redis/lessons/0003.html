第 3 课 · 限流：@RateLimiter 链路与令牌桶 切面区

🎯 本节目标（对应使命 Success #3）：学完你能完整讲清一次 @RateLimiter 从注解到 Redis 令牌桶的链路，说清 SpEL 动态 key 怎么拼、DEFAULT / IP / CLUSTER 三种 LimitType 的差异，以及令牌不足时为什么返回 -1 就抛异常。

限流的目的：保护系统不被请求刷爆。RuoYi 用一个注解 + 一个切面 + Redisson 令牌桶实现。我们追一条完整的线。

一、注解：声明限流规则

annotation/RateLimiter.java 
@RateLimiter(
    key = "#user.userId",          // 限流维度（SpEL，可选）
    time = 60,                     // 时间窗口（秒），默认 60
    count = 100,                   // 窗口内允许次数，默认 100
    limitType = LimitType.IP,      // 限流类型，默认 DEFAULT
    timeout = 86400,               // 限流策略存活，默认 1 天
    message = "{rate.limiter.message}"  // 提示，支持国际化
)
public R<Void> submit(@RequestBody User user) { ... }

二、五步链路

@RateLimiter 注解的方法被调用 │ ▼ RateLimiterAspect.doBefore() ← @Before 在方法执行前拦截 │ 1. 取出 time / count / timeout │ 2. getCombineKey() 拼出完整限流 key │ 3. 选 RateType（DEFAULT/IP→OVERALL, CLUSTER→PER_CLIENT） ▼ RedisUtils.rateLimiter(key, rateType, count, time, timeout) │ 4. CLIENT.getRateLimiter(key) 拿 RRateLimiter │ trySetRate(...) 设置速率 │ tryAcquire() 尝试扣一个令牌 ▼ ├─ 成功 → 返回 availablePermits()（剩余令牌数 ≥ 0） └─ 失败 → 返回 -1 │ ▼ number == -1 ? ├─ 是 → 翻译 message → throw ServiceException(限流提示) └─ 否 → 放行，记日志"剩余令牌 => N"

三、key 是怎么拼出来的（getCombineKey）

aspectj/RateLimiterAspect.java:95

这是最值得细看的方法。key 由三段拼成：固定前缀 + URI + (维度) + 业务 key。

1解析 SpEL 业务 key
如果注解的 key 含 #，说明是表达式（如 #user.userId）。切面用 MethodBasedEvaluationContext 把方法参数喂给 SpEL 解析器，求出真实值。还支持 #{...} 模板格式和 @bean 引用（设了 BeanFactoryResolver）。
2拼前缀和 URI
GlobalConstants.RATE_LIMIT_KEY + requestURI + ":"。所以不同接口天然隔离。
3按 LimitType 加维度
IP → 追加 ServletUtils.getClientIP()；CLUSTER → 追加 RedisUtils.getClient().getId()（客户端实例 id）；DEFAULT → 不加。
LimitTypekey 含什么效果RateType
DEFAULTURI + 业务key全局限流：所有人共享一个桶OVERALL
IPURI + IP + 业务key按来源 IP 各自限流OVERALL
CLUSTERURI + 实例id + 业务key按后端实例各自限流PER_CLIENT
DEFAULT vs CLUSTER 的微妙差异：DEFAULT/OVERALL 是「整个集群共用一个全局令牌桶」（无论几台后端，总量受限）。CLUSTER/PER_CLIENT 是「每个后端实例一个桶」（部署 3 台就是 3 倍总量）。前者控总量、后者控单机——别记反。

四、令牌桶：为什么是 Redisson 来做

// RedisUtils.rateLimiter
RRateLimiter rateLimiter = CLIENT.getRateLimiter(key);
rateLimiter.trySetRate(rateType, rate, Duration.ofSeconds(rateInterval), Duration.ofSeconds(timeout));
if (rateLimiter.tryAcquire()) {
    return rateLimiter.availablePermits();   // 剩余令牌
} else {
    return -1L;                              // 没令牌了
}
为什么不自己用 incr 计数？「读计数 → 判断 → 加一」三步在并发下不原子，会超发。Redisson 的 RRateLimiter 用 Lua 脚本在 Redis 端原子完成取令牌，天然支持分布式多实例。trySetRate 是幂等的——只在桶不存在时设速率；timeout 让策略过期后能被重置（清掉旧策略数据）。

五、国际化消息处理

注解默认 message = "{rate.limiter.message}"。切面发现消息被 {} 包裹，就剥掉花括号、用 MessageUtils.message() 按当前语言翻译。所以你看到的是「访问过于频繁，请稍候再试」而非一串 code。

六、检验一下（提取练习）

Q1. limitType = IP 和 limitType = DEFAULT 的区别是？

A. IP 限流更快，DEFAULT 限流更慢 B. IP 在 key 里加来源 IP，每个 IP 各自一个桶；DEFAULT 所有请求共享一个全局桶 C. IP 用 Redis，DEFAULT 用本地内存 D. 没有区别，只是名字不同
看答案B。getCombineKey 在 IP 类型时追加 ServletUtils.getClientIP()，使每个 IP 拥有独立令牌桶；DEFAULT 不加维度，全局共享。

Q2. 令牌桶为什么用 Redisson 的 RRateLimiter 而不是自己 incr 计数？

A. incr 不支持过期时间 B. RRateLimiter 用 Lua 在 Redis 端原子取令牌，并发下不会超发，且天然支持分布式 C. incr 只能存整数，存不下令牌 D. RRateLimiter 不需要连接 Redis，速度更快
看答案B。"读-判断-写"三步非原子会超发；RRateLimiter 把扣令牌做成 Redis 端的原子 Lua 操作，多实例共享一致。

Q3. 切面里 RedisUtils.rateLimiter(...) 返回 -1 意味着什么？

A. Redis 连接失败 B. 令牌已耗尽（tryAcquire 失败），切面将抛出限流 ServiceException C. 限流配置错误，key 为空 D. 还剩 1 个令牌，可以放行
看答案B。tryAcquire() 失败返回 -1L，切面据此抛出翻译后的限流提示；成功则返回 ≥0 的剩余令牌数。
