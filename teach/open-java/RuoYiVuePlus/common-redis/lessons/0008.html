第 8 课 · 异常处理、依赖边界与全局复盘 完结课

🎯 本节目标（对应使命 Success #7 + 全局复盘）：学完你能说清 RedisExceptionHandler 的职责（以及它为什么只处理 Lock4j）、模块与它依赖的 6 个外部库的边界在哪、自动装配是如何把 4 个 Config 一起拉起来的，并能在脑中重建整个模块的完整大图。

一、RedisExceptionHandler：只做一件事

handler/RedisExceptionHandler.java 
@RestControllerAdvice
public class RedisExceptionHandler {

    @ExceptionHandler(LockFailureException.class)
    public R<Void> handleLockFailureException(LockFailureException e, HttpServletRequest request) {
        log.error("获取锁失败了'{}',发生Lock4j异常.", requestURI, e);
        return R.fail(HTTP_UNAVAILABLE, "业务处理中，请稍后再试...");
    }
}
为什么只处理 LockFailureException？Lock4j 是 baomidou 的分布式锁框架，本模块在 pom.xml 里引入了 lock4j-redisson-spring-boot-starter。当两个请求同时抢同一把锁失败时，Lock4j 会抛 LockFailureException——这不是系统 bug，是正常的并发竞争结果。如果让这个异常冒泡到全局异常处理器，可能被当成 500 返回，用户看到的是"系统错误"而非"别人正在处理"。所以 Redis 模块自己把它兜成友好的 "业务处理中，请稍后再试"。

注意：RedisExceptionHandler 不处理 Redis 连接超时、序列化失败等异常——那些归全局异常处理器管。它只兜 Lock4j。这说明它就是 Lock4j 的一个"文明降级"适配器。

二、自动装配：4 个 Config 怎么一起拉起来

META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 
org.dromara.common.redis.config.RedisConfig
org.dromara.common.redis.config.CacheConfig
org.dromara.common.redis.config.IdempotentConfig
org.dromara.common.redis.config.RateLimiterConfig

这 4 个类都标了 @AutoConfiguration，在 Spring Boot 3.x 的自动装配机制下，只要 pom.xml 引入了 ruoyi-common-redis，Spring 就会扫描这个文件，把 4 个配置类全部实例化：

spring.factories (旧) → AutoConfiguration.imports (Spring Boot 3.x) 引入 ruoyi-common-redis → Spring 读 imports 文件 → 装配： RedisConfig → Redisson 序列化/前缀/连接 CacheConfig → @EnableCaching + Caffeine + PlusSpringCacheManager RateLimiterConfig → RateLimiterAspect（被 @RateLimiter 触发） IdempotentConfig → RepeatSubmitAspect（被 @RepeatSubmit 触发）
💡 RateLimiterConfig 和 IdempotentConfig 都加了 @AutoConfiguration(after = RedisConfiguration.class)，确保在 Redis 连接就绪后才装配切面——否则 RedisUtils 里的 CLIENT 拿不到。

三、依赖边界：模块到底依赖了什么

pom.xml

回到 pom.xml，看看这个模块引了哪些外部库，以及它们各管什么边界：

依赖在本模块的角色跨边界说明
ruoyi-common-core拿 Spring 容器工具类、常量、异常、HTTP 工具依赖方向：redis ← core（redis 依赖 core 的工具）
ruoyi-common-jsonJSON 序列化（Jackson3 + Hutool JSON）RedisConfig 用 Jackson3 Codec；RepeatSubmit 用 JsonUtils
redisson-spring-boot-starter★ 核心：操作 Redis 的客户端整个模块的基石，所有工具类的 CLIENT 都来自这个依赖
lock4j-redisson-...分布式锁（本模块只接管它的异常）边界清晰：锁由 Lock4j 管，异常归 RedisExceptionHandler 兜
caffeine本地一级缓存可选：不传 caffeine 构造则二级缓存不启用
sa-token-core读 token 名（RepeatSubmit key 用）仅 SaManager.getConfig().getTokenName() 一处

现在回顾 8 节课的全部内容，把 5 大区域汇总成一张最终地图。这是你以后"回来找东西"的索引。

ruoyi-common-redis（Redisson 业务封装层） │ │ 依赖: redisson / caffeine / lock4j / sa-token / ruoyi-common-core / ruoyi-common-json │ ├── config/ ── 启动装配，定地基规则 │ ├── RedisConfig → 序列化(CompositeCodec)、前缀(KeyPrefixHandler)、单机/集群(RedissonProperties) │ ├── CacheConfig → @EnableCaching + Caffeine 本地缓存 + PlusSpringCacheManager │ ├── RateLimiterConfig → 注册限流切面 @AutoConfiguration(after=RedisConfiguration) │ ├── IdempotentConfig → 注册防重切面 │ └── RedissonProperties → yml: redisson.* 映射（线程/前缀/单机/集群） │ ├── handler/ ── 挂在框架扩展点上的小钩子 │ ├── KeyPrefixHandler → 实现 NameMapper，给所有 key 加/去前缀（对业务透明） │ └── RedisExceptionHandler → 兜 Lock4j 锁失败，降级为"请稍后再试" │ ├── aspectj/ + annotation/ + enums/ ── 注解驱动的拦截行为 │ ├── @RateLimiter → 控频率：RRateLimiter 令牌桶，3 种 LimitType（DEFAULT/IP/CLUSTER） │ └── @RepeatSubmit → 控重复：setIfAbsent 占位 + 成功留键/失败删键 │ ├── manager/ ── 二级缓存核心 │ ├── PlusSpringCacheManager → 自定义 CacheManager，支持 cacheName#参数 语法 │ └── CaffeineCacheDecorator → 给 Redis Cache 装饰一层本地 Caffeine（30s/1000条） │ └── utils/ ── 业务最常直接调用的入口 ├── RedisUtils → 裸键值：对象/集合/Map/原子/发布订阅/keys扫描/限流 ├── CacheUtils → Spring Cache 命名空间（和 @Cacheable 同源） ├── QueueUtils → 轻量分布式队列（普通+优先级） └── SequenceUtils → 分布式发号器（P20260630000001）

用一张图记住这个模块：Redisson 是底层引擎 → config 把它配成"懂 RuoYi 规则的 Redisson"→ handler 加透明前缀和异常降级 → aspectj 提供注解式限流和防重 → manager 搭二级缓存加速 → utils 给业务开门。你日常写的代码里 90% 只碰到 utils 里的四个工具类和那两个注解——但你现在知道，它们背后是这张完整的地图。

五、检验一下（最终提取练习）

Q1. RedisExceptionHandler 为什么只处理 Lock4j 的异常，而不处理 Redis 连接超时？

A. Redis 连接超时不会被抛出，不需要处理 B. 它只负责把 Lock4j 的锁竞争降级成友好提示；Redis 连接异常归全局异常处理器管 C. Lock4j 异常是唯一的异常类型，Redis 不会出异常 D. Redis 连接超时会被 KeyPrefixHandler 自动重试
看答案B。本模块和 Lock4j 有边界：锁由 Lock4j 管、锁失败由本模块兜（降级友好提示）。Redis 自身异常不属于本模块范围。

Q2. Spring Boot 3.x 是通过什么机制把 4 个 Config 自动装配的？

A. spring.factories 文件 B. AutoConfiguration.imports 文件（Spring Boot 3.x 新机制） C. 编译时注解处理器自动扫描包 D. Maven 插件在打包时注入
看答案B。META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 是 Spring Boot 3.x 的自动装配入口文件，替代了旧版的 spring.factories。

Q3. 读完 8 节课，一个新人问你"RuoYi 的缓存模块到底是什么"，用一句话回答？

A. 就是一个 RedisUtils 工具类而已，没别的了 B. 是 Redisson 的业务封装层：配好序列化/前缀/连接 → 提供限流防重注解 → 搭二级缓存 → 给四个工具类开门 C. 核心是 Lock4j 分布式锁模块 D. 核心是 Caffeine 本地缓存
看答案B。8 节课的最终答案——config 定地基、handler 装钩子、aspectj 加保护、manager 做加速、utils 开大门。一句话讲清模块的全貌和设计分层。
