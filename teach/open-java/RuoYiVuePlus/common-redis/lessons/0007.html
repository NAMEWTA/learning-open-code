第 7 课 · CacheUtils / QueueUtils / SequenceUtils utils 区

🎯 本节目标（对应使命 Success #6 中后三个工具部分）：学完你能说清 CacheUtils 和 RedisUtils 的根本区别、QueueUtils 的定位（以及什么时候该用 MQ 而非它）、SequenceUtils 的发号器怎么生成 P20260630000001 这种单号。

一、CacheUtils：操作的是 Spring Cache 命名空间

utils/CacheUtils.java

CacheUtils 很小，只有 get / put / evict / clear 四个方法。但它有个容易踩的坑：它操作的不是裸 Redis key，而是 Spring Cache 命名空间下的数据。

// 用 @Cacheable 存的
@Cacheable(cacheNames = "sys_dict", key = "#dictType")
public List<SysDictData> getDict(String dictType) { ... }

// CacheUtils 读到同一份数据
List<SysDictData> list = CacheUtils.get("sys_dict", dictType);

// RedisUtils 读不到！（key 拼法不同、序列化可能不同）
List<?> list = RedisUtils.getCacheObject("sys_dict:xx"); // NO!
根本区别：RedisUtils 是"裸键值"——key 是你自己拼的，序列化是你控制的；CacheUtils 走 CacheManager 抽象——key 是 Spring Cache 生成的（按 @Cacheable 的 key 表达式），序列化也是 PlusSpringCacheManager 决定的。

简单记：注解存在 CacheUtils 取，裸存就用 RedisUtils。用错就是"对不上"。

实现上，CacheUtils 通过内部类 CacheManagerHolder 延迟持有 CacheManager（懒加载模式），和 RedisUtils 直接拿 RedissonClient 是一个思路。

二、QueueUtils：轻量分布式队列

utils/QueueUtils.java

代码注释写得很直白："轻量级队列，重量级数据量请使用 MQ"。基于 Redisson 的 RBlockingQueue 和 RPriorityBlockingQueue：

// 普通队列
QueueUtils.addQueueObject("task", data);       // 入队
T item = QueueUtils.getQueueObject("task");    // 出队（poll，没数据返回 null）
QueueUtils.removeQueueObject("task", data);     // 删指定元素

// 优先级队列
QueueUtils.addPriorityQueueObject("task", data);
T item = QueueUtils.getPriorityQueueObject("task");

// 订阅消费（阻塞等待）
QueueUtils.subscribeBlockingQueue("task", element -> { ... });

底层 RBlockingQueue 就是一个 Redis List（LPUSH + BRPOP）。特点和限制都很明显：

适合不适合（请用 MQ）
小数据量、临时性任务需要持久化、重试、死信队列
不需要严格可靠性保证海量数据堆积
同项目内简单任务分发跨系统异步通信
优先级队列延迟消息（Redisson 已废弃 delay queue）
💡 代码里有一行被注释掉的 // CLIENT.getDelayedQueue(queue)，标注了 Redisson 已废弃延迟队列。需要延迟消息的场景，老老实实用 RocketMQ/Kafka 的延迟消息能力。

三、SequenceUtils：分布式发号器

utils/SequenceUtils.java

这可能是四个工具类里"最有花活"的一个。本质是 Redisson 的 RIdGenerator，但包装了丰富的 API 层来生成业务单号。

底层：RIdGenerator

RIdGenerator idGenerator = CLIENT.getIdGenerator(key);
idGenerator.tryInit(initValue, stepValue);  // 初始值=1, 步长=1
idGenerator.expire(expireTime);             // 不同时间粒度的过期
long nextId = idGenerator.nextId();         // 拿下一个

RIdGenerator 在 Redis 里用一个 key 存当前值，每次 nextId() 通过 Lua 原子递增。步长机制支持批量预取以减少 Redis 调用。

两种时间格式 + 业务前缀

getDateId("P", true, 6) ↓ "P" + "20260630" + "000001" └─前缀 └─日期 └─当天序号(6位不足补0) getDateTimeId("O", true, 6) ↓ "O" + "20260630153045" + "000001" └─前缀 └─日期时间(到秒) └─该分钟序号(6位不足补0)

两个版本的关键差异在过期策略：

方法时间粒度默认过期典型用途
getDateId天（yyyyMMdd）1 天订单号、工单号
getDateTimeId到秒（yyyyMMddHHmmss）1 分钟流水号、请求编号
为什么 sequence key 要过期？key 名是 前缀 + 时间戳，每过一天/一分钟就是一个新 key。老 key 过期自动清理，否则每天一个新 key 会无限堆积。这就是"按时间分片 + 自动清理"——用 Redis 的过期机制替代定时任务。

补零：minIdCapacityBits

getPaddedDateId("P", true) 默认补到 6 位（最少能生成 999999 个序号）。代码里 DEFAULT_MIN_ID_CAPACITY_BITS = 6，可自定义。

废弃方法

getDateId() 和 getDateTimeId() 的两个无参重载被标记了 @Deprecated，注释说"请使用带 prefix 的版本，确保不同业务的 ID 连续性"——因为如果多个业务共用一个 key，会互相干扰序号。

四、检验一下（提取练习）

Q1. 用 @Cacheable(cacheNames="sys_dict") 存的数据，想在另一个 Service 里手动读取，应该用？

A. RedisUtils.getCacheObject("sys_dict:xxx") B. CacheUtils.get("sys_dict", key) C. QueueUtils.getQueueObject("sys_dict") D. 直接 new 一个 RedisConnection
看答案B。@Cacheable 的数据归 Spring CacheManager 管，CacheUtils 走的同一套抽象，key 拼法和序列化都一致。用 RedisUtils 可能对不上。

Q2. SequenceUtils 为什么给不同时间粒度的 key 设不同过期时间？

A. 为了节省带宽，小 key 传得快 B. 按时间分片的 key（如 P20260630）如果不自动过期，每天产生一个会无限堆积；用 Redis 过期机制自动清理 C. 不同业务需要不同格式的单号 D. 这是 Redisson 的强制要求，没设计原因
看答案B。每天/每分钟产生一个新 key，老 key 过期自动清理，避免 key 堆积。

Q3. QueueUtils 注释里说"重量级数据量请使用 MQ"，主要因为？

A. Redis 队列比 MQ 更贵 B. Redis 队列没有持久化、重试、死信等可靠性保证，数据堆积会消耗 Redis 内存 C. MQ 由 Java 实现，速度更快 D. Redis 不支持阻塞队列
看答案B。Redis List 做队列是内存型、无持久化、无重试/死信/事务消息等企业级能力。适合临时轻量任务，生产级异步该用 MQ。
