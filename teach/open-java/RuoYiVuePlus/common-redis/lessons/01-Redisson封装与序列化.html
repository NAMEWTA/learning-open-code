01 - Redisson 封装与序列化

🎯 本节目标：理解为什么选 Redisson 而不是 RedisTemplate，掌握 CompositeCodec（key String + value JSON）和 KeyPrefixHandler 的 key 前缀处理，能讲清 RedisConfig 中单机/集群双模式的配置差异。

1. 全景鸟瞰

ruoyi-common-redis 是整个项目使用 Redis 的唯一入口。它不直接依赖 Spring 的 RedisTemplate，而是基于 Redisson——一个将 Redis 能力包装成 Java 分布式对象（Map、Set、Queue、AtomicLong、限流器、发号器）的框架。

模块五大区域：

common-redis/
├── config/      → RedisConfig、CacheConfig、RateLimiterConfig、IdempotentConfig
├── handler/     → KeyPrefixHandler、RedisExceptionHandler
├── aspectj/     → RateLimiterAspect、RepeatSubmitAspect
├── cache/       → PlusSpringCacheManager、CaffeineCacheDecorator
└── utils/       → RedisUtils、CacheUtils、QueueUtils、SequenceUtils

2. 核心设计

2.1 为什么选 Redisson 而不是 RedisTemplate？

维度RedisTemplate (Spring Data Redis)Redisson
编程模型opsForValue / opsForHash / opsForList... 操作命令RBucket / RMap / RQueue... Java 分布式对象
序列化需手动配置 Serializer（默认 JdkSerializer）Codec 体系，开箱即用 JSON
高级功能需额外实现或集成内置限流器、锁、发号器、Pub/Sub
分布式锁需用 Lock4j 或其他框架RLock（看门狗自动续期）
连接管理Lettuce/Jedis 连接池Netty 异步连接，单连接多路复用
一句话总结：RedisTemplate 是「操作 Redis 命令的模板」，Redisson 是「把 Redis 包装成 Java 对象」。选择 Redisson 的原因是它内置了丰富的分布式对象，省去了大量底层实现。

2.2 CompositeCodec：Key 和 Value 分开编码

RedisConfig 的核心配置：

@Bean
public RedissonClient redissonClient(Config config) {
    config.setCodec(new CompositeCodec(
        StringCodec.INSTANCE,   // ← Key 用 String（redis-cli 中可读）
        JsonJacksonCodec.INSTANCE // ← Value 用 JSON（支持复杂对象）
    ));
    return Redisson.create(config);
}

为什么要分开？如果 Key 也用 JSON 序列化，redis-cli KEYS * 会看到乱码——无法排查问题。StringCodec 让 Key 保持可读，JSON 让 Value 能存储复杂对象并自动还原类型。

2.3 KeyPrefixHandler：防止 Key 冲突

实现 Redisson 的 NameMapper 接口，对 Key 名做自动转换：

public class KeyPrefixHandler implements NameMapper {
    private final String prefix;  // 来自配置: redis.keyPrefix

    public String map(String name) {
        if (prefix == null || prefix.isEmpty()) return name;
        if (name.startsWith(prefix)) return name;  // 防重复添加
        return prefix + ":" + name;                 // 如 "ruoyi:cache:user:1"
    }

    public String unmap(String name) {
        if (prefix == null || prefix.isEmpty()) return name;
        if (name.startsWith(prefix)) return name.substring(prefix.length() + 1);
        return name;
    }
}

配置：redis.keyPrefix = "ruoyi" → 所有 Key 自动变成 ruoyi:xxx。业务代码仍写 redisUtils.get("user:1") 即可，前缀自动添加，对代码完全透明。

实际价值：如果多项目/多环境共用 Redis（开发环境/测试环境共用同一 Redis 实例），不同环境的 Key 会冲突。通过配置不同前缀（dev: / test:），即使 Key 名相同也不会互相覆盖。

2.4 RedisConfig：单机 vs 集群双模式

redis:
  mode: standalone            # standalone / cluster / sentinel
  host: 127.0.0.1
  port: 6379
  password: 123456
  database: 0
  keyPrefix: ruoyi            # Key 前缀（防冲突）
  timeout: 3000               # 连接超时（毫秒）

两种模式的配置差异：

模式配置对象连接方式适用场景
standaloneSingleServerConfig单一 Redis 地址开发环境、小规模部署
clusterClusterServersConfig多个节点地址（自动发现）生产环境、高可用要求
sentinelSentinelServersConfig哨兵节点列表传统主从 + 哨兵架构

3. 实战练习

🧠 提取练习（点击展开）

Q1: KeyPrefixHandler.map() 中 startsWith 防重复添加的判断为什么重要？

参考: 如果前缀是 "ruoyi"，原始 key 是 "ruoyi:user:1"，不判断 startsWith 会变成 "ruoyi:ruoyi:user:1"。这种 bug 通常在多级调用中出现——方法 A 传了完整 key 进来、方法 B 又 mapper 一次。startsWith 保证无论调用多少次 map()，前缀只加一次。

Q2: 为什么 CompositeCodec 对 Key 和 Value 分别用 StringCodec 和 JsonJacksonCodec？

参考: 两个实际需求：① 运维排查——redis-cli KEYS * 能看到 "ruoyi:user:1" 这样的可读 key，方便定位问题；② 类型还原——从 Redis 读取一个 User 对象时，JSON 保留类型信息（@class），反序列化时能准确还原为 User 对象而不是 LinkedHashMap。如果 Key 也用 JSON，KEYS 命令看到的是 Base64 编码，对排查毫无帮助。

Q3: 单机模式改集群模式需要改代码吗？

参考: 不需要。Redisson 的 RedissonClient 接口完全屏蔽了底层连接方式差异。只需要把 redis.mode 从 standalone 改成 cluster、并配置 nodeAddresses，代码中通过 RedisUtils 操作 Redis 的逻辑零修改。这就是面向接口编程 + 配置驱动的优势。

4. 调用链路

// 一行 setObject
RedisUtils.setObject("user:1", user, Duration.ofMinutes(30));

内部路径:
1. RedisUtils.setObject(key, value, timeout)
2. RedissonClient.getBucket(KEY_PREFIX + key)
   → 触发 KeyPrefixHandler.map("user:1") → "ruoyi:user:1"
3. RBucket.set(value, timeout, TimeUnit)
   → 触发 CompositeCodec 编码:
     Key: "ruoyi:user:1" → StringCodec → Redis String
     Value: User{id=1, name="张三"} → JsonJacksonCodec → {"@class":"...User","id":1,"name":"张三"}
4. Redisson 异步执行 Redis SET ruoyi:user:1 '{"@class":...}' EX 1800
💡 设计洞察：RedisConfig 的三处关键定制（CompositeCodec / KeyPrefixHandler / 单机集群双模式）各自解决了一个生产级问题：Codec 解决「key 可读 + value 能还原类型」、KeyPrefixHandler 解决「多环境 key 隔离」、双模式解决「开发期跑单机、上线切集群」的平滑过渡。三者结合使模块在一个 RedisConfig 类中完成全部基础设施建设——引入依赖即可用、改配置即可换环境。
