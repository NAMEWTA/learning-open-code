第 6 课 · RedisUtils 全能工具实战 utils 区

🎯 本节目标（对应使命 Success #6 中 RedisUtils 部分）：学完你能区分 RedisUtils 里6 组操作（对象 / 集合 / Map / 原子值 / 发布订阅 / keys 扫描）各自用哪个底层对象、各有哪些关键 API，并在遇到「该用哪个」时做出正确选择。

RedisUtils 是整个模块最常用的类——也是代码最长的一个（600+ 行）。把它按底层 Redisson 对象分 6 组来记，比一个一个 API 背高效得多。

一、工具类的使用方式

RedisUtils 构造器私有（@NoArgsConstructor(access = PRIVATE)），全部 static 方法。Redisson 客户端在类加载时从 Spring 容器一次性获取：

private static final RedissonClient CLIENT = SpringUtils.getBean(RedissonClient.class);

优点：业务代码直接用 RedisUtils.setCacheObject(...) 一行搞定，不需要注入任何东西。约束：它必须在 Spring 容器就绪后才能调用，启动阶段不能用。

三、重点 API 逐组放大

① RBucket 对象：TLL 保留与 setIfAbsent

对象操作里有一个容易忽略的细节：setCacheObject(key, value, true) 的第三个参数 isSaveTtl。

// 保留 TTL：set 后过期时间不变（如原来剩 90s，set 后还是 90s）
bucket.setAndKeepTTL(value);                // Redis 6.x 原生支持
// 兼容 Redis 5.x 的回退：
bucket.set(value, Duration.ofMillis(remainTimeToLive));  // 手动重设
什么时候要保留 TTL？验证码场景：用户每 60 秒才能发一次，到期自动失效。set 新验证码时不能重置过期时间（那样就无限期了），必须保持原来剩余的时间窗口。

另外 setObjectIfAbsent / setObjectIfExists 是防重提交的底层 API（见第 4 课）。

④ RMap Hash：批量操作与 RBatch

Map 操作覆盖了安全、删除和多字段读取：

setCacheMap(key, dataMap);                          // 整 Map 写入
setCacheMapValue(key, hKey, value);                 // 写单个 field
getCacheMapValue(key, hKey);                        // 读单个 field
delCacheMapValue(key, hKey);                        // 删单个 field
getMultiCacheMapValue(key, hKeys);                  // 批量读多个 field
delMultiCacheMapValue(key, hKeys);                  // 批量删多个 field — 用 RBatch

delMultiCacheMapValue 是唯一用到 RBatch 的地方：把多个 removeAsync 打包成一次网络往返，大幅减少 Redis 调用次数。

⑤ RAtomicLong：分布式计数器

setAtomicValue("order:count", 0);
incrAtomicValue("order:count");    // 原子 +1，并发安全
decrAtomicValue("order:count");    // 原子 -1
getAtomicValue("order:count");     // 读当前值

Redis 的 INCR 本身是原子的，Redisson 只是给它包了一层 Java API。

⑥ RTopic 发布订阅

publish("channel:alert", msg);                              // 发消息
subscribe("channel:alert", MyClass.class, msg -> {...});     // 订阅，返回 listenerId
unsubscribe("channel:alert", listenerId);                    // 取消订阅

发布订阅让你在不同服务实例间做简单的事件通知（如「字典更新了，各位清缓存」）。注意：Redis Pub/Sub 不持久化消息，订阅者不在线就丢了。

⑥ keys 扫描：用 SCAN 而不是 KEYS

keys() 方法用 CLIENT.getKeys().getKeysStream()，底层是 Redis 的 SCAN 命令（分页迭代），不会像 KEYS 那样阻塞 Redis。默认 chunkSize=1000，支持按 pattern / type 过滤，也可传自定义 KeysScanOptions。

四、检验一下（提取练习）

Q1. setCacheObject(key, value, true) 的第三个参数 true 表示？

A. 强制覆盖已存在的值 B. 保留 key 原有的 TTL，set 新值后过期时间不变 C. 开启事务支持 D. 将序列化方式切换为 JSON
看答案B。isSaveTtl=true 会调用 setAndKeepTTL（Redis 6.x）或手动拿剩余时间重设。验证码等场景需要不重置过期窗口。

Q2. delMultiCacheMapValue 为什么用 RBatch？

A. RBatch 是 Redisson 唯一支持的删除方式 B. 把多个 field 的异步删除打包成一次网络往返，而不是每个 field 发一次请求 C. 只有 RBatch 支持删除 Map 的 field D. 为了在日志里记录批量删除的轨迹
看答案B。逐条 removeAsync 进同一个 batch，最后 batch.execute() 一次性发给 Redis，减少网络开销。

Q3. RedisUtils.keys() 底层用的是 Redis 什么命令？

A. KEYS *（一次性全扫） B. SCAN（分页迭代，不阻塞 Redis） C. TYPE（按类型过滤） D. GET（逐个获取）
看答案B。getKeysStream 底层是 SCAN 命令，默认 chunkSize=1000 分批迭代，不会像 KEYS 那样在生产环境阻塞 Redis。
