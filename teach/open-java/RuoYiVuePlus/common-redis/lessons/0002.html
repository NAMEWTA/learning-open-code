第 2 课 · 地基：RedisConfig 与 Redisson 序列化 配置区

🎯 本节目标（对应使命 Success #2）：学完你能解释「为什么选 Redisson 而不是 RedisTemplate」，并讲清 RedisConfig 里三处定制——组合序列化、key 前缀、单机/集群双模式——各自解决了什么问题，外加 Jackson 3.x 那处反序列化安全设计。

第 1 课我们说整个模块是「Redisson 的业务封装层」。这节课就钻进那块地基：config/RedisConfig.java。它在 Spring 启动时跑一次，决定了之后所有缓存操作的底层规则。

一、入口：一个"定制器"而非直接 new 客户端

注意 RedisConfig 没有自己创建 RedissonClient，而是返回一个 RedissonAutoConfigurationCustomizer：

@AutoConfiguration
@EnableConfigurationProperties(RedissonProperties.class)
public class RedisConfig {
    @Bean
    public RedissonAutoConfigurationCustomizer redissonCustomizer() {
        return config -> { /* 在这里改 config */ };
    }
}
为什么用 Customizer？
Redisson 的 Spring Boot Starter 会自己读 spring.data.redis.* 创建客户端。RuoYi 不想从头接管这个过程，只想在它创建前插一脚改几个参数。Customizer 就是 starter 暴露的「定制回调」——你拿到半成品 config，改完交还给 starter 去 new。这是「扩展而非替换」的典型做法。

二、定制点 1：组合序列化 CompositeCodec

JsonMapper jsonMapper = JsonMapper.builder()
    .addModules(simpleModule)           // LocalDateTime 用 yyyy-MM-dd HH:mm:ss
    .activateDefaultTyping(...)          // 存对象时带上全类名，反序列化能还原类型
    .build();
TypedJsonJackson3Codec jsonCodec = new TypedJsonJackson3Codec(Object.class, jsonMapper);
// key 用 String，value 用 JSON
CompositeCodec codec = new CompositeCodec(StringCodec.INSTANCE, jsonCodec, jsonCodec);
config.setCodec(codec);

CompositeCodec 的三个参数分别是 map-key 编解码、value 编解码、其它。这里的安排是：

部分用什么好处
keyStringCodec在 redis-cli 里直接能读懂 key，不是一串二进制乱码
valueJSON（带类型信息）能存任意复杂对象，反序列化时还能还原成原本的类
💡 activateDefaultTyping(..., NON_FINAL) 是关键：它让 JSON 里额外保存对象的全类名。否则反序列化时 Jackson 不知道该还原成 SysUser 还是 Map。代价是 JSON 体积变大、且引入了反序列化安全问题（见下）。

三、一个安全彩蛋：反序列化漏洞防护

代码注释里专门写了一大段，讲 Jackson 3.x 的变化：

反序列化漏洞：当 JSON 里带类型信息、且数据来源不可信时，攻击者可以构造一个「全类名指向危险类」的 payload，让程序在反序列化时实例化恶意对象（著名的 fastjson/jackson 漏洞家族）。

Jackson 2.x 时代常用 LaissezFaireSubTypeValidator（放行一切）。Jackson 3.x 出于安全禁止外部引用它，本模块改用 BasicPolymorphicTypeValidator： 
// 当前：默认放行所有类型（行为等价 Jackson 2.x 的 LaissezFaire）
.activateDefaultTyping(
    BasicPolymorphicTypeValidator.builder()
        .allowIfSubType((ctxt, clazz) -> true)   // ← 全放行
        .build(),
    DefaultTyping.NON_FINAL)

注释里还给了更安全的建议写法——用包名白名单只放行自己项目的类：

BasicPolymorphicTypeValidator.builder()
    .allowIfBaseType("org.dromara")
    .allowIfSubType("org.dromara")
    .build()
读到这里要带走的判断力：默认配置图省事放行所有类型，在「Redis 数据可能被外部污染」的场景下是有风险的。如果你做安全加固，这就是一个该收紧成白名单的点。

四、定制点 2：key 前缀（接上 KeyPrefixHandler）

config.setNameMapper(new KeyPrefixHandler(redissonProperties.getKeyPrefix()));

第 1 课见过 KeyPrefixHandler：它实现 Redisson 的 NameMapper，map() 存时加前缀、unmap() 取时去前缀。逻辑很轻：

// 构造时：前缀为空则空，否则补一个冒号
this.keyPrefix = StringUtils.isBlank(keyPrefix) ? "" : keyPrefix + ":";
// map：还没加前缀才加，避免重复
if (isNotBlank(keyPrefix) && !name.startsWith(keyPrefix)) return keyPrefix + name;

效果：业务写 setCacheObject("login_tokens:123", ...)，Redis 里实际是 ruoyi:login_tokens:123，而你读回来时看到的还是原始 key。对业务完全透明。

五、定制点 3：单机 / 集群双模式

剩下半段在读 RedissonProperties（映射 yml 的 redisson.*），哪个配置块非空就用哪种模式：

redissonProperties.getSingleServerConfig() != null → config.useSingleServer()... redissonProperties.getClusterServersConfig() != null → config.useClusterServers()...

两个配置块都是 RedissonProperties 的内部静态类，字段类似（连接池大小、超时、空闲超时、订阅连接池等），集群版多了 readMode（读 SLAVE）和 subscriptionMode（订阅走 MASTER）。RedisConfig 底部的大段注释给了完整的集群 yml 示例。

💡 还有两个全局参数：setThreads / setNettyThreads（默认都是 CPU 核数 × 2），以及 setUseScriptCache(true)——因为 Redisson 大量功能基于 Lua 脚本，缓存脚本能减少每次把脚本传给 Redis 的网络开销。

六、检验一下（提取练习）

不要回看上文，凭记忆作答。

Q1. CompositeCodec(StringCodec, jsonCodec, jsonCodec) 里 key 用 String、value 用 JSON 的主要好处是？

A. 加快网络传输速度，String 比 JSON 快 B. key 在 redis-cli 里人能读懂，value 又能存复杂对象并还原类型 C. 节省内存，String 不占空间 D. 兼容老版本 Redis，新版本不支持 JSON
看答案B。key 用 StringCodec 保证可读性（便于排查），value 用带类型的 JSON 保证能存任意对象且反序列化可还原类型。

Q2. 为什么 RedisConfig 返回 RedissonAutoConfigurationCustomizer 而不是直接创建 RedissonClient？

A. 因为 RedissonClient 不能被 Spring 管理 B. 想在 starter 创建客户端前插一脚改参数，扩展而非替换默认装配 C. 因为单机和集群需要两个不同的客户端 D. 这是唯一能配置序列化的方式，没有别的办法
看答案B。Redisson starter 会自己读配置创建客户端，Customizer 是它暴露的定制回调，让你只改几个参数而不接管整个创建过程。

Q3. 当前 BasicPolymorphicTypeValidator 配成"放行所有类型"，在安全角度的隐患与官方建议是？

A. 没有隐患，放行所有类型是最佳实践 B. 数据源不可信时有反序列化漏洞风险，建议改成 org.dromara 包名白名单 C. 会导致序列化变慢，建议关闭类型信息 D. 会让 key 变长，建议缩短前缀
看答案B。带类型信息的反序列化在数据可被污染时存在漏洞，代码注释明确建议用 allowIfBaseType("org.dromara") 白名单收紧。
