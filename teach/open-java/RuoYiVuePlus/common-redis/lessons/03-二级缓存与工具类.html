03 - 二级缓存与工具类

🎯 本节目标：理解 PlusSpringCacheManager 的二级缓存设计（Caffeine 本地 + Redis 分布式），掌握 cacheName#ttl#maxIdle#maxSize#local 参数化配置的含义，能区分 RedisUtils、CacheUtils、QueueUtils、SequenceUtils 四个工具类的使用场景。

1. 全景鸟瞰

前面两课讲了 Redisson 基础封装和注解切面。本课聚焦模块的缓存加速能力（二级缓存）和四个工具类——这是日常开发中最常用的部分。理解它们，你就能在代码中正确选用 Redis 的各种能力。

2. 核心设计

2.1 二级缓存架构

为什么需要两级？单级 Redis 缓存每次读取都要网络往返（~1ms），高频访问时 QPS 瓶颈明显。二级缓存在 JVM 内部加一层 Caffeine（纳秒级），大部分请求命中本地，大幅降低 Redis 压力。

读数据流程:
请求 → Caffeine (L1, 本地内存, ~200ns)
  ├─ 命中 → 直接返回（99% 的情况）
  └─ 未命中 → Redis (L2, 网络, ~1ms)
                ├─ 命中 → 写回 Caffeine → 返回
                └─ 未命中 → 查数据库 → 写 Redis → 写 Caffeine → 返回

写数据流程:
请求 → 先 invalidate Caffeine → 写 Redis → 返回
      （先清本地再写远程，保证一致性）

2.2 PlusSpringCacheManager：自定义 CacheManager

继承自 Redisson 的 RedissonSpringCacheManager，增加了两个关键能力：

1. cacheName 参数化配置：

// 传统 Spring Cache 写法（需要写配置类）
@Cacheable(value = "user", key = "#userId")

// PlusSpringCacheManager 支持的写法（在缓存名中声明策略）
@Cacheable(value = "user#3600#1800#1000#local", key = "#userId")
//                    ↑名称↑ttl↑maxIdle↑maxSize↑local

格式：cacheName#ttl#maxIdle#maxSize#local

参数含义示例
ttl存活时间（秒）3600 → 1 小时后过期
maxIdle最大空闲时间（秒）1800 → 30 分钟无访问则过期
maxSize最大条目数1000 → 本地缓存最多存 1000 条
local是否启用本地缓存有 "local" → 启用 Caffeine
为什么把策略参数放在缓存名里？这是 Spring Cache 注解限制下的务实方案——@Cacheable 注解不支持直接传 TTL/Size 等缓存策略参数。Redisson 官方需要额外配置文件或编程式注册。把参数编码在缓存名中，不改 Spring Cache 注解定义、不写额外配置，一行注解搞定。

2.3 CaffeineCacheDecorator：本地缓存装饰器

实现 Spring Cache 接口，包装 Caffeine 作为一级缓存：

// 读操作：先本地后回源
get(key, Callable<Object> valueLoader):
    return caffeine.get(key, loader)
    // ↑ Caffeine 的 get 方法自动处理:
    //   命中 → 直接返回
    //   未命中 → 调 valueLoader（查 Redis） → 写入本地 → 返回

// 写操作：先清本地再写远程
put(key, value):
    caffeine.invalidate(key)     // ← 清除本地（防脏数据）
    delegate.put(key, value)     // ← 写 Redis

为什么 put 时要先 invalidate？因为集群环境下其他节点可能已经更新了 Redis。不清直接写，本地缓存的旧值会在其他节点的 get(key, loader) 中污染新请求——因为 loader 只查 Redis，但其他节点已经 update 过 Redis 了，不清会导致本地缓存中残留旧值。

2.4 四个工具类定位

工具类底层对象主要用途典型场景
RedisUtilsRBucket / RMap / RAtomicLong / RTopic / RRateLimiter通用 Redis 读写、限流、发布订阅存用户 token、缓存配置、分布式计数
CacheUtilsSpring Cache 体系（通过 PlusSpringCacheManager）操作具名缓存空间字典数据缓存、菜单缓存、租户缓存
QueueUtilsRBlockingQueue（基于 Redis List）分布式阻塞队列轻量任务排队、延时处理
SequenceUtilsRIdGenerator（分布式发号器）生成业务单号订单号、流水号（如 P20260630000001）

2.5 SequenceUtils：分布式发号器

基于 Redisson 的 RIdGenerator，用 Lua 原子递增生成唯一 ID：

// 获取今天的订单号
String orderNo = SequenceUtils.nextId("P");  // → "P20260630000001"

内部原理:
1. key = "P" + "20260630"           // 按时间分片（每天一个新 key）
2. RIdGenerator.incrementAndGet()   // Lua 原子递增
3. 老 key 自动过期（避免堆积百万个日 key）
4. 支持配置初始值、步长、过期时间

为什么按时间分片？如果所有流水号共用一个 key，这个 key 会变得非常热（所有请求竞争同一个 Lua 锁）。按天分片后，今天的并发全部集中在今天的 key 上——但这已经比全局 key 好得多。如果并发极高，还可以按小时或分钟分片。

2.6 事务感知缓存与异常兜底

TransactionAwareCacheDecorator：当 transactionAware=true 时，PlusSpringCacheManager 自动包一层 Spring 的 TransactionAwareCacheDecorator——缓存写入在事务提交后才执行。事务回滚则缓存不写。

RedisExceptionHandler：兜底 Lock4j 分布式锁框架的 LockFailureException——将锁竞争失败从「系统异常 500」降级为「业务处理中，请稍后再试」。锁由 Lock4j 管理，异常由 redis 模块兜。

3. 实战练习

🧠 提取练习（点击展开）

Q1: CacheUtils 和 RedisUtils 都在操作 Redis，什么时候用哪个？

参考: CacheUtils 操作的是 Spring Cache 体系——有命名空间隔离、支持二级缓存、支持 @Cacheable 注解声明。RedisUtils 是原生 Redis 操作——可以精确控制数据结构和过期策略。选择标准：如果你的数据是「读多写少」的缓存（如字典、菜单），用 CacheUtils + @Cacheable 享受二级缓存加速；如果数据需要精确控制（如验证码 60 秒过期、分布式计数器），用 RedisUtils。

Q2: 二级缓存中「先清本地再写 Redis」为什么不反过来（先写 Redis 再清本地）？

参考: 如果先写 Redis → Redis 写入成功 → 清本地——这中间有一个时间窗口，其他节点的请求可能读到 Redis 的新值并缓存到他们的本地（正常），但本节点的本地可能还没清。反过来（先清本地再写 Redis）保证了本地一定读不到旧值。极端情况下本节点读本地 miss → 读 Redis（可能还在写）→ 读到旧值，但这是短暂窗口，且不会长期残留。

Q3: QueueUtils 的 RBlockingQueue 为什么不适合做重量级任务队列？

参考: RBlockingQueue 基于 Redis List（LPUSH + BRPOP），消息不持久化且无 ACK 机制——消费者崩了消息丢失、消费失败无法重试、没有死信队列。重量级任务队列需要 RabbitMQ/Kafka 这样的专业 MQ：消息持久化到磁盘、ACK 确认机制、重试策略、死信队列、延迟消息等。QueueUtils 只适合「丢了也能接受的轻量临时任务」。

4. 调用链路

// 场景：查询用户信息（带二级缓存）
@Cacheable(value = "user", key = "#userId")
public User getUser(Long userId) {

第一次调用 (userId=1):
1. PlusSpringCacheManager.getCache("user")
2. CaffeineCacheDecorator.get(1, loader)
   → caffeine.getIfPresent(1) → null（首次，未缓存）
   → loader.call()
     → RedisUtils.getObject("user:1") → null（首次，Redis 也无）
     → 查数据库 → User{id=1, name="张三"}
     → RedisUtils.setObject("user:1", user, 3600s)
     → 写入 Caffeine
   → 返回 User

第二次调用 (userId=1):
1. CaffeineCacheDecorator.get(1, loader)
   → caffeine.getIfPresent(1) → User{id=1, name="张三"} ✓
   → 直接返回（全程无 Redis/DB 调用，~200ns）

更新用户 (userId=1):
1. @CachePut(value = "user", key = "#user.id")
2. CaffeineCacheDecorator.put(1, user)
   → caffeine.invalidate(1)     // 清除本地旧值
   → RedisUtils.setObject("user:1", user, 3600s)  // 写 Redis
💡 设计洞察：PlusSpringCacheManager 的 cacheName 参数化配置是一种「在不完美的世界中做务实选择」的设计。理想情况下，TTL/Size 等参数应该有独立的配置方式，但 Spring Cache 注解不支持。在缓存名中编码参数虽然看起来 hack，但它解决了实际问题且对代码侵入性最低。CaffeineCacheDecorator 的 get(key, loader) 模式更是巧妙——用 Caffeine 内置的「缓存未命中自动回源」机制，一行代码就实现了「先本地后 Redis」的降级逻辑，无需手写 if-else 判断。
