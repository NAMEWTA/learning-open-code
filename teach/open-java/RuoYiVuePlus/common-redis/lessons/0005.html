第 5 课 · 二级缓存：Caffeine（本地）+ Redis（分布式） manager 区

🎯 本节目标（对应使命 Success #5）：学完你能讲清为什么需要两级缓存、PlusSpringCacheManager 的 cacheName#ttl#maxIdle#maxSize#local 约定怎么来的、CaffeineCacheDecorator 装饰器如何把本地和远端串起来，并判断什么数据适合开二级缓存。

纯 Redis 缓存每次都是网络往返（即使 < 1ms，高并发下也是瓶颈）。对那些"读极其频繁、写很少、能忍几秒陈旧"的数据，加一层 JVM 本地内存缓存能直接砍掉大部分 Redis 调用。

一、三件套：Config → Manager → Decorator

CacheConfig PlusSpringCacheManager CaffeineCacheDecorator ───────────────── ────────────────────────── ────────────────────── @EnableCaching 实现 CacheManager 接口 实现 Cache 接口 开启 @Cacheable/@CacheEvict 接管所有 Spring Cache 操作 装饰一个 Redis Cache 给它包一层 Caffeine @Bean caffeine() 解析 # 语法 → Caffeine.newBuilder() → RMap 或 RMapCache expireAfterWrite(30s) → ± CaffeineCacheDecorator maximumSize(1000) → ± TransactionAwareCacheDecorator

二、CacheConfig：两条 bean 启动二级缓存

config/CacheConfig.java 
@AutoConfiguration
@EnableCaching                                          // ★ 打开 Spring Cache 注解
public class CacheConfig {

    @Bean
    public Cache<Object, Object> caffeine() {           // ★ 本地缓存实例
        return Caffeine.newBuilder()
            .expireAfterWrite(30, TimeUnit.SECONDS)     // 写后 30 秒自动过期
            .initialCapacity(100)
            .maximumSize(1000)                          // 最多 1000 条
            .build();
    }

    @Bean
    public CacheManager cacheManager() {
        return new PlusSpringCacheManager();            // 没传 caffeine? 见下面
    }
}
💡 注意 cacheManager() 创建时没传 caffeine！那么 PlusSpringCacheManager 的 caffeine 字段是怎么填上的？——它会在构造后通过 Spring 注入。实际上看代码，构造函数允许 caffeine 为 null，不为 null 时才启用本地一级缓存。这意味着如果你明确不要两级缓存，传 null 就够了。
暂未启用：当前 CacheConfig.cacheManager() 创建的 PlusSpringCacheManager 不传 caffeine 参数，这意味着默认未启用 Caffeine 本地缓存。真正要用两级缓存，需要把 caffeine() bean 作为参数传入 cacheManager(caffeine)。

三、PlusSpringCacheManager · 缓存名里塞参数

manager/PlusSpringCacheManager.java:148

getCache(String name) 是核心方法。它把缓存名按 # 拆分，支持 5 个位置：

"dict#1h#30m#200#1" │ │ │ │ │ │ │ │ │ └─ positional[4]: local (1=开启本地缓存，0=关闭) │ │ │ └───── positional[3]: maxSize (最大条数) │ │ └────────── positional[2]: maxIdleTime (最长空闲) │ └────────────── positional[1]: ttl (存活时间) └─────────────────── positional[0]: 真正的 cacheName

解析逻辑：StringUtils.delimitedListToStringArray(name, "#") 拆出数组 → array[0] 是真名 → array[1~4] 按位置覆盖配置 → DurationStyle.detectAndParse 把 1h / 30s / 500ms 这种格式转成毫秒。

为什么要这么设计？Spring Cache 的 @Cacheable(cacheNames="xxx") 只能声明缓存名，不能传 TTL。你当然可以新写一个注解，但那太重了。把参数偷偷藏在缓存名里，用 # 分隔，是用最小改动实现最大灵活性的讨巧方案——不改 Spring 接口，不改注解定义，就能在注解里声明缓存策略。

四、CaffeineCacheDecorator · 怎么把两层串起来

manager/CaffeineCacheDecorator.java 
@Override
public ValueWrapper get(Object key) {
    // getuniqueKey = "cacheName:key"
    Object o = caffeine.get(getUniqueKey(key), k -> cache.get(key));
    //          ^^^^^^^^ 先查本地  查不到则      ^^^^^^^^ 回源查 Redis
    return (ValueWrapper) o;
}

@Override
public void put(Object key, Object value) {
    caffeine.invalidate(getUniqueKey(key));  // 先让本地副本失效
    cache.put(key, value);                  // 再写 Redis
}
读取：caffeine.get(key, loader) 是 Caffeine 的原子读——如果 key 存在（本地命中），直接返回；不存在则调 loader（回源 Redis），结果自动写入本地缓存。Caffeine 自己加了 30 秒过期。

写入：先 invalidate 清本地、再写 Redis。如果反过来（先写 Redis 再清本地），中间有一瞬间别的线程可能读到旧本地值；先清本地再写 Redis 更安全。极端并发下仍有短暂不一致窗口，但对非强一致性场景（字典/配置）可接受。

删除/清空：先干 Redis 再干本地。如果只干 Redis 不干本地，后续本地缓存命中会把旧值还回来。

五、两种 Redis 对象的选择

PlusSpringCacheManager 根据是否有 TTL/maxIdle/maxSize 决定用哪种 Redisson 对象：

条件用特点
TTL=0, maxIdle=0, maxSize=0RMap普通 Map，不过期
三者任一非 0RMapCache带 TTL / maxIdle / maxSize 的 Map

两种都会再包一层 TransactionAwareCacheDecorator（当 transactionAware=true 时），保证在 Spring 事务提交后才真正写入缓存。

六、适合 / 不适合两级缓存的数据

适合不适合
字典表（如 sys_dict_data）实时性要求 < 30s 的数据
系统配置项（参数表）频繁写入的数据
读多写少、改动不频繁的列表登录态等与单机无关的会话数据
能容忍 30s 内陈旧的数据集群下需要即时一致性的计数器

七、检验一下（提取练习）

Q1. CaffeineCacheDecorator.get() 的读取流程是？

A. 先查 Redis，再查本地 B. 先用 caffeine.get(key, loader) 查本地，查不到时 loader 回源 Redis 并自动填入本地 C. 本地和 Redis 同时查，取最快的 D. 只查本地，本地没有就直接返回 null
看答案B。caffeine.get(key, loader) 的 loader 就是回源读 Redis；命中本地直接返回，不碰 Redis。结果自动缓存到本地，受 Caffeine 的 30s 过期控制。

Q2. 为什么要先在缓存名里塞参数（dict#1h#200#1）这种设计？

A. 为了炫技，证明框架能力强 B. 因为 Spring Cache 的注解只能传缓存名不能传 TTL；把参数藏在缓存名里是最小改动的妥协方案 C. 因为 Redis 不支持中文 key，用 # 做转义 D. 这是 Redisson 原生的功能，PlusSpringCacheManager 只是透传
看答案B。@Cacheable(cacheNames="xxx") 不等价于"带 TTL 的缓存"。本模块用 # 分隔把 TTL 等参数藏在缓存名里，不改 Spring 接口、不改注解定义，就实现了注解级别的缓存策略声明。

Q3. 写入时为什么 caffeine.invalidate（清本地）要在 cache.put（写 Redis）之前？

A. Caffeine 和 Redis 的写入速度不同，先清本地更快 B. 先清本地再写 Redis，避免写 Redis 后、清本地前这一瞬间别的线程读到旧本地值 C. Redis 要求必须先清本地缓存 D. 其实先写 Redis 再清本地才是对的，代码写反了
看答案B。先清本地再写 Redis：如果反过来，写入 Redis 后清本地前，另一个线程会从本地读到旧值。先清本地保证写入后的下一次读一定回源 Redis。
