第 4 课 · 防重提交：@RepeatSubmit 与幂等 切面区

🎯 本节目标（对应使命 Success #4）：学完你能讲清 @RepeatSubmit 的链路，以及它的核心设计——「setIfAbsent 占位 + 成功留键 / 失败删键」这套幂等手法，并能说清它和上一课 @RateLimiter 在意图上的根本区别。

典型场景：用户手抖双击「提交订单」，或网络慢时反复点。防重提交要保证短时间内的重复请求只有第一次生效。代码注释说它参考了「美团 GTIS 防重系统」。

一、注解很简单

annotation/RepeatSubmit.java 
@RepeatSubmit(
    interval = 5000,                  // 间隔(ms)，小于此视为重复，默认 5 秒
    timeUnit = TimeUnit.MILLISECONDS,
    message = "{repeat.submit.message}"  // 国际化提示
)
public R<Void> addOrder(@RequestBody OrderBo bo) { ... }

二、三个切面方法：前、后、异常

和限流只有 @Before 不同，防重提交切面有三个通知，构成一个完整的「占位—确认—回滚」闭环：

@Before doBefore() 请求进来 → 抢占 key @AfterReturning doAfterReturning() 正常返回 → 看返回码决定留键还是删键 @AfterThrowing doAfterThrowing() 抛异常 → 删键（允许重试）

三、doBefore：怎么抢占、key 怎么生成

aspectj/RepeatSubmitAspect.java:49
1校验间隔
interval 换算成毫秒，小于 1000ms 直接报错（"重复提交间隔不能小于 1 秒"）。
2生成唯一 key
取请求参数拼成字符串 nowParams，取请求头里的 token（SaManager.getConfig().getTokenName() 读 token 名），两者拼起来做 MD5：
submitKey = md5(token + ":" + nowParams)
最终 key = REPEAT_SUBMIT_KEY + url + submitKey。
3原子抢占
RedisUtils.setObjectIfAbsent(cacheRepeatKey, "", Duration.ofMillis(interval))：
  • 抢到（之前没有这个 key）→ 把 key 存进 ThreadLocal，放行。
  • 没抢到（已存在）→ 翻译 message → 抛 ServiceException（重复提交）。
key 为什么要拼 token + 参数 + URL？三者共同定义「同一个人对同一个接口提交同样的数据」。换了用户（token 不同）、换了接口（url 不同）、改了参数（MD5 不同）都算不同操作，不会误伤。用 MD5 是为了把任意长的参数压成定长 key。

四、最精妙处：成功留键、失败删键

aspectj/RepeatSubmitAspect.java:85 
@AfterReturning(..., returning = "jsonResult")
public void doAfterReturning(..., Object jsonResult) {
    if (jsonResult instanceof R<?> r) {
        if (r.getCode() == HttpStatus.SUCCESS) {
            return;            // ★ 成功：不删 key，间隔内无法再提交
        }
        deleteRepeatKey();     // 业务失败：删 key，允许马上重试
    }
}
@AfterThrowing(...)
public void doAfterThrowing(...) {
    deleteRepeatKey();         // 异常：删 key，允许重试
}
这套设计的精髓：
- 成功了就该「锁住」一段时间——用户双击的第二下会撞上还没过期的 key，被拦下。
- 失败/异常了说明这次没成，应该让用户能立刻重试，所以主动删 key，不必等过期。
如果不分情况一律靠过期，用户提交失败后还得干等 5 秒才能再试，体验很差。
💡 KEY_CACHE 是 ThreadLocal<String>：doBefore 存入、后置方法取用并在 finally 里 remove()。用 ThreadLocal 是因为「这次请求抢到的 key」要在同一线程的前置和后置之间传递，又不能串到别的请求。

五、参数拼装的细节：过滤不可序列化对象

argsArrayToString 把参数转 JSON 拼接，但 isFilterObject 会跳过 MultipartFile（文件）、HttpServletRequest/Response、BindingResult 这些——它们无法或不该参与 JSON 序列化，且对「是否重复」没有判定意义。数组、集合、Map 会递归检查。

六、和 @RateLimiter 的根本区别

@RateLimiter@RepeatSubmit
保护什么系统（别被刷爆）数据（别重复落库）
控制维度频率（N 次/时间窗）同一操作的重复（一次只生效一次）
底层令牌桶 RRateLimitersetIfAbsent 占位 + 过期
成功后令牌随时间恢复留键锁定到过期
失败后不影响删键，可立即重试

七、检验一下（提取练习）

Q1. 防重提交在请求成功（返回码 SUCCESS）后为什么不删除 Redis 里的 key？

A. 删除操作太慢，影响性能 B. 故意保留到过期，让间隔内的重复提交（如双击）撞上 key 被拦下 C. 因为成功的 key 不能被删除，是 Redis 的限制 D. 留着做日志统计用
看答案B。成功就该锁定一段时间防重复；只有失败/异常才删 key 以允许用户立即重试。这是"成功留键、失败删键"的核心。

Q2. 防重 key 由 token + 请求参数 MD5 + URL 组成，这样设计是为了？

A. 让 key 尽量短，节省 Redis 内存空间 B. 精确定义"同一个人对同一接口提交同样数据"，换人/换接口/改参数都算不同操作 C. 加密用户信息，防止泄露 D. 兼容不同的 HTTP 方法
看答案B。token 区分人、url 区分接口、参数 MD5 区分内容，三者共同界定"重复"的边界，避免误伤不同请求。

Q3. setObjectIfAbsent 在防重里扮演什么角色？

A. 设置缓存的过期时间 B. 原子地"占位"：不存在才写入并返回 true（抢到），已存在返回 false（判定重复） C. 删除已存在的重复 key D. 读取上一次提交的内容
看答案B。它底层是 SETNX + 过期的原子操作。返回 true 表示首次抢到放行；false 表示 key 已在（重复），抛异常拦截。
