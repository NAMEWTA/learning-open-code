密钥协商协议速查

阶段客户端服务端
请求前生成随机 AES 密钥，RSA 公钥加密后放 header—
请求到达—从 header 取 RSA 密文 → 私钥解密 → 得到 AES 密钥 → 解密 body
响应前—生成新的随机 AES 密钥 → RSA 公钥加密 → 放响应 header → AES 加密 body
响应到达从 header 取 RSA 密文 → 公钥解密 → 得到 AES 密钥 → 解密 body—

请求头标志说明

配置项默认值作用
api-decrypt.header-flagencrypt-key用于传输 RSA 加密后的 AES 密钥的请求头名称

通过 ApiDecryptProperties.getHeaderFlag() 获取，可在 yml 中自定义。

常见陷阱

  • GET 请求不拦截： 只用 POST/PUT，加密数据必须随 body 发送。如果你的加密接口用了 GET，CryptoFilter 不会解密。
  • @ApiEncrypt(response=true) 影响所有响应： 即使请求没有加密头，只要注解标记了 response=true，响应也会被加密返回。注意前后端的同步。
  • CORS + encrypt-key： 如果前端跨域请求，必须确认 Access-Control-Expose-Headers 包含了 encrypt-key，否则前端 JS 无法读取加密密钥。
  • RSA 密钥格式： 必须是 PEM 格式的 BASE64 编码。启动时会校验，密钥格式错误直接启动失败（Fail-Fast）。
  • 加密后 Content-Length 变化： AES 加密后数据长度会变化，响应头的 Content-Length 由 Wrapper 自动更新为密文长度。
