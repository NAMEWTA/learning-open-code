常见陷阱

  • 加密-恢复时序： encrypt() 必须在 proceed() 之前，restore() 必须在 finally 中。顺序颠倒会导致数据库存明文或业务代码见密文。
  • String 参数跳过： MybatisEncryptInterceptor 对 parameterObject instanceof String 直接跳过。如果你的 Mapper 方法是 insert(@Param("phone") String phone) 而没有使用实体对象，参数不会被加密——不管字段有没有 @EncryptField。
  • null 结果检查： MybatisDecryptInterceptor 在 result == null 时直接 return null，不调用 decrypt()。如果忘记这步检查，EncryptedFieldProcessor 的 handle() 遇到 null 也会跳过，但这是防御性编程。
  • 批量操作： EncryptedFieldProcessor 能处理 List/Collection，批量插入时 List 中每个实体的加密字段都会被处理。
  • 循环引用： 通过 IdentityHashMap 追踪已访问对象，防止 A→B→A 的循环引用导致栈溢出。
  • 字段值类型： EncryptedFieldProcessor 通过 Convert.toStr() 将字段值转为字符串再处理。非字符串类型的字段（如 Integer）也能被加密，但加密后变为密文字符串存入数据库，解密时也是字符串——需要确保数据库字段类型是 VARCHAR 而非数值类型。
  • 二次加密风险： 如果忘了恢复原始值，后续的数据库操作会把密文当原文再次加密，导致数据永久损坏。这不是"多解一次就能恢复"的问题——加密函数是幂等的，但解密不是（解一次得到的是上一次的密文，不是原文）。
