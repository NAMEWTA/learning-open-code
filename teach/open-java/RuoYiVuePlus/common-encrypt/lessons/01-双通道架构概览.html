第 1 课 · 双通道架构概览：算法体系与核心组件

本课任务：建立对 ruoyi-common-encrypt 模块的顶层认知——理解双通道架构的设计动机、掌握 IEncryptor 接口到 AlgorithmType 枚举的算法体系全貌、理解 EncryptContext / EncryptContextFactory / EncryptorManager 三组件的分工协作，以及两套独立配置系统的启用方式。

1. 模块定位

ruoyi-common-encrypt 提供透明的数据加解密能力，服务两个完全不同的安全场景：

  • API 通信加密通道：在 Controller 层拦截 HTTP 请求/响应，保护网络传输中的数据。前端加密请求体，后端解密后处理；后端加密响应体，前端解密后使用。
  • DB 字段加密通道：在 MyBatis 层拦截 SQL 入参/出参，保护数据库存储中的数据。写入时自动将敏感字段加密为密文落库，查询时自动解密还原为明文。
关键认知：两条通道虽然共用同一套算法体系，但它们的拦截层次、密钥来源、触发方式、启用开关完全独立。你可以只启用 API 加密而不启用 DB 加密，反之亦然。这种"共用算法 + 独立通道"的设计是本模块最核心的架构思想。

设计洞察：为什么需要两条独立通道？
API 加密和 DB 加密的威胁模型完全不同。API 通道面对的是网络链路攻击者（中间人抓包），需要密钥协商协议（RSA 保证密钥安全传输）。DB 通道面对的是数据库层面攻击者（拖库、运维查表），没有密钥交换需求，密钥直接配置在服务端。两者不可混用——一个在 HTTP 层拦截，一个在 SQL 层拦截，处于完全不同的抽象层次。

3. IEncryptor 接口 —— 加密算法的顶层契约

org.dromara.common.encrypt.encryptor.IEncryptor 定义了三个方法：

方法签名说明
algorithm()AlgorithmType algorithm()返回当前加密器的算法类型枚举值，作为缓存键的一部分
encrypt()String encrypt(String value, EncodeType encodeType)加密明文，指定输出编码格式（BASE64 或 HEX）
decrypt()String decrypt(String value)解密密文。不接收 EncodeType——Hutool 的 decryptStr() 自动识别编码格式

接口只有三个方法，设计极为精简。加密时需指定编码，解密时自动检测——这符合"产生密文的人知道编码方式，解密时不需要额外告知"的原则。

4. AbstractEncryptor 抽象基类

org.dromara.common.encrypt.encryptor.AbstractEncryptor —— 所有五种加密器实现的共同基类。构造函数接收 EncryptContext 进行配置注入。子类在构造时做各自校验：RsaEncryptor 和 Sm2Encryptor 校验公私钥完整性，AesEncryptor 和 Sm4Encryptor 延迟到 EncryptUtils 调用时校验。

AlgorithmType 的 clazz 字段直接指向加密器实现类。EncryptorManager 通过 ReflectUtil.newInstance(clazz, encryptContext) 反射创建加密器实例，添加新算法只需新增枚举值+加密器类，Manager 层无需改动。

5.1 五种加密器简述

  • AesEncryptor：对称加密，委托 EncryptUtils.encryptByAes(data, password) / decryptByAes(data, password)
  • RsaEncryptor：非对称加密，公钥加密+私钥解密。构造时校验公私钥完整性（Fail-Fast）
  • Sm2Encryptor：国密非对称，底层依赖 Bouncy Castle。委托 EncryptUtils.encryptBySm2(data, publicKey)
  • Sm4Encryptor：国密对称，密钥必须恰好 16 字节（不同于 AES 的 16/24/32）
  • Base64Encryptor：编码，非加密。仅用于开发调试占位，生产环境不要用它保护敏感数据

6. EncryptContext —— 加密材料的"清单"

org.dromara.common.encrypt.core.EncryptContext 是一个 DTO，携带算法、密码、公私钥、编码方式五个字段。它在 EncryptContextFactory → EncryptorManager → IEncryptor 之间流转，是加密器创建的"材料清单"。

7. EncryptContextFactory —— "注解优先，默认兜底"

org.dromara.common.encrypt.core.EncryptContextFactory 的核心逻辑：当 @EncryptField 的属性是 DEFAULT 或空字符串时，回退到 yml 全局配置：

create(Field field):
  algorithm = field注解 == DEFAULT ? yml全局algorithm : 注解值
  password  = field注解 == ""      ? yml全局password  : 注解值
  publicKey = field注解 == ""      ? yml全局publicKey : 注解值
  privateKey= field注解 == ""      ? yml全局privateKey: 注解值
  encode    = field注解 == DEFAULT ? yml全局encode    : 注解值
  → 返回完整 EncryptContext

这使得同一表中大部分字段用全局 AES 加密，某个特别敏感的字段可以单独标注 @EncryptField(algorithm=SM4, password="customKey")。

8. EncryptorManager —— 加解密调度中心

org.dromara.common.encrypt.core.EncryptorManager 维护两级缓存：

缓存键值目的
encryptorMapEncryptorCacheKey（算法+编码+密钥的元组）IEncryptor 实例避免每次加密都反射创建新加密器
fieldCacheClass<?>（实体类）Set<Field>（@EncryptField 字段集合）避免每次加密都反射扫描类的所有字段

ENCRYPT_HEADER 防重复加密逻辑：

encrypt(value, ctx):
  if (value.startsWith("ENC_")) return value;  // 已是密文，跳过
  encrypted = encryptor.encrypt(value, ctx.getEncode());
  return "ENC_" + encrypted;

decrypt(value, ctx):
  if (!value.startsWith("ENC_")) return value; // 不是密文，跳过
  stripped = value.substring(4);
  return encryptor.decrypt(stripped);

10. 课后练习

练习 1 · 算法选型（点击展开）

Q: 某业务要求数据库字段加密后仍能等值查询（WHERE = 密文）。以下哪种算法最合适？

A. RSA（每次加密结果不同，随机填充导致无法等值匹配）
B. AES（对称加密，确定性加密时同一明文产生相同密文，支持等值查询）
C. BASE64（无安全性）
D. SM2（同 RSA，非确定性加密）

答案：B。对称加密（使用固定 IV 或 ECB 模式）对同一明文产生相同密文，支持等值查询。RSA/SM2 每次加密结果都不同。但确定性加密会暴露"相同明文=相同密文"的模式，安全性弱于随机化加密。

练习 2 · 算法定位（点击展开）

Q: 关于五种算法定位，正确的是？

A. BASE64 是一种轻量级加密
B. SM4 密钥长度与 AES 相同（16/24/32）
C. RSA 和 SM2 都是非对称加密，RSA 基于大素数分解，SM2 基于椭圆曲线
D. API 通道也支持通过 @EncryptField 切换算法

答案：C。A 错误——BASE64 是编码。B 错误——SM4 必须恰好 16 字节。D 错误——API 通道固定 RSA+AES。

练习 3 · EncryptContextFactory 行为（点击展开）

Q: @EncryptField 的 algorithm=SM4、password="" 时，最终 EncryptContext 的 password 是什么？

A. 空字符串
B. EncryptorProperties.password（yml 全局密码）
C. null
D. 启动报错

答案：B。password=""（默认值）被视为"未设置"回退到 yml。但 algorithm=SM4 是显式指定的不被覆盖。结果是：SM4 算法 + yml 全局密码。

练习 4 · 缓存设计（点击展开）

Q: EncryptorManager 为什么需要 encryptorMap 和 fieldCache 两级缓存？

参考：去掉加密器缓存每次加密都反射 new 加密器+校验密钥，高并发下 GC 和 CPU 开销大。去掉字段缓存每次加密/解密都反射遍历类的所有字段，性能影响显著。两级 ConcurrentHashMap 保证线程安全，启动时预热 fieldCache，运行时按需填充 encryptorMap。
