第 3 课 · DB 字段加解密：拦截器与快照恢复机制

本课任务：完整追踪 DB 字段加密通道的全链路——从 @EncryptField 注解的字段标注、到 MybatisEncryptInterceptor 的入参加密、MybatisDecryptInterceptor 的出参解密、EncryptedFieldProcessor 的递归处理算法、FieldSnapshot 的恢复机制、以及 ENCRYPT_HEADER 前缀防重复加密设计。同时掌握 EncryptUtils 工具类的静态方法体系。

1. @EncryptField 注解 —— 字段级加密的标记

org.dromara.common.encrypt.annotation.EncryptField 标注在实体类 String 字段上。五个可配置属性：

属性类型默认值说明
algorithmAlgorithmTypeDEFAULT加密算法。DEFAULT 表示使用 yml 全局默认
passwordString""AES/SM4 对称加密密钥
publicKeyString""RSA/SM2 非对称加密公钥
privateKeyString""RSA/SM2 非对称加密私钥
encodeEncodeTypeDEFAULT加密结果编码方式（BASE64 或 HEX）

@Inherited 元注解使子类实体自动继承父类的字段加密配置。注解优先于 yml 全局配置——当 algorithm=DEFAULT 或 password="" 时回退到 yml 默认值。

public class User {
    @EncryptField                          // 使用全局默认算法和密钥
    private String phone;

    @EncryptField(algorithm = AlgorithmType.SM4, password = "myCustomSm4Key01")
    private String idCard;                 // 指定 SM4 + 自定义密钥

    @EncryptField(algorithm = AlgorithmType.RSA,
                  publicKey = "MIGfMA0G...", privateKey = "MIICdgIB...")
    private String bankAccount;            // 指定 RSA + 独立密钥对

    private String nickname;               // 不加密
}
注意：@EncryptField 只支持 String 类型字段。EncryptorManager 扫描时过滤 field.getType() == String.class，Integer/Long/Date 等类型即使标注了注解也会被忽略。加密后字段是 Base64 或 Hex 文本，数据库列需为 VARCHAR 且长度充足。

2. EncodeType 编码类型

org.dromara.common.encrypt.enums.EncodeType 控制加密后字节数据的编码方式：

枚举值输出格式适用场景
DEFAULT使用 yml 全局配置（默认 BASE64）大部分字段无需单独指定
BASE64A-Za-z0-9+/=，比 HEX 短约 25%数据库存储空间敏感
HEX0-9A-F，纯数字字母需要人工查看或字符级比对

对 Base64Encryptor 而言 EncodeType 不生效——编码结果本身就是 Base64，不存在二次编码。

3. MybatisEncryptInterceptor —— 入参加密拦截器

org.dromara.common.encrypt.interceptor.MybatisEncryptInterceptor 是 MyBatis Plugin，拦截 ParameterHandler.setParameters(PreparedStatement)：

@Intercepts({@Signature(
    type = ParameterHandler.class,
    method = "setParameters",
    args = {PreparedStatement.class}
)})

拦截逻辑（简化）：

// 1. 获取参数对象
Object parameterObject = boundSql.getParameterObject();
if (parameterObject == null || parameterObject instanceof String)
    return invocation.proceed();  // null 或裸String 跳过

// 2. 加密字段，生成快照
List<FieldSnapshot> snapshots = processor.encrypt(parameterObject);

// 3. 执行 SQL（此时参数值已是密文）
invocation.proceed();

// 4. finally 中恢复原始值（无论 SQL 成功或失败）
finally {
    snapshots.forEach(FieldSnapshot::restore);
}

关键设计：

  • 裸 String 跳过：如果 Mapper 方法是 insert(@Param("phone") String phone) 而非实体对象，参数无法按字段反射处理
  • finally 恢复：即使 SQL 抛异常，也必须恢复原始值——保证业务对象不受污染

4. MybatisDecryptInterceptor —— 出参解密拦截器

org.dromara.common.encrypt.interceptor.MybatisDecryptInterceptor 拦截 ResultSetHandler.handleResultSets(Statement)：

@Intercepts({@Signature(
    type = ResultSetHandler.class,
    method = "handleResultSets",
    args = {Statement.class}
)})
// 1. 先执行查询（获取密文结果）
Object result = invocation.proceed();
if (result == null) return null;

// 2. 解密字段
processor.decrypt(result);

// 3. 返回明文结果（无需恢复——明文就是要给业务的）
return result;

5. EncryptedFieldProcessor —— 递归字段处理引擎

org.dromara.common.encrypt.core.EncryptedFieldProcessor 是 DB 通道最复杂的组件，负责递归遍历对象树并对 @EncryptField 字段执行加解密。

5.1 encrypt(sourceObject) 流程

encrypt(sourceObject):
  visited = new IdentityHashMap<>()
  snapshots = new ArrayList<>()
  handle(sourceObject, visited, (target, field, value) -> {
      // 加密回调
      EncryptContext ctx = contextFactory.create(field);
      String encrypted = encryptorManager.encrypt(value, ctx);
      field.set(target, encrypted);
      snapshots.add(new FieldSnapshot(target, field, value));
  })
  return snapshots;

handle(obj, visited, handler):
  if obj == null || obj instanceof String → return
  if visited.containsKey(obj) → return        // 循环引用检测
  visited.put(obj, null)

  if obj instanceof Map → 遍历 values 递归
  else if obj instanceof Collection → 遍历 items 递归
  else:
      从 fieldCache 获取 @EncryptField 字段
      for each field:
          value = field.get(obj)
          if value != null → handler.handle(obj, field, value)
          if isComplexType(field.getType()) → handle(value, ...) 继续递归

5.2 decrypt(sourceObject) 流程

与 encrypt 对称，但回调不同——decrypt 回调直接 encryptorManager.decrypt(value, ctx) → field.set(target, 明文)，不产生快照。

IdentityHashMap 为什么不用普通 HashMap？IdentityHashMap 使用 == 比较对象身份而非 equals()。两个不同对象即使 equals 相等也应独立处理，只有真正的同一引用（A→B→A 循环）才需跳过以避免栈溢出。

6. FieldSnapshot —— 原始值的"时空胶囊"

FieldSnapshot 记录三个信息：target（所属对象）、field（哪个字段）、originalValue（原始值）。restore() 方法将字段值设回原始值。

阶段操作时机
创建new FieldSnapshot(target, field, value)encrypt() 中加密后保存原始值
SQL 执行field 值是密文ParameterHandler 使用加密后的值
恢复snapshot.restore()finally 块中——无论 SQL 成败都执行
恢复后field 值 = 原始明文业务代码看到的仍是原始值
设计洞察：透明加密的本质
DB 字段加密通道的目标是"业务代码完全无感知"——开发者写 user.setPhone("13800138000")，数据库自动存密文；写 user.getPhone()，自动返回明文。这由三个机制协同实现：(1) MyBatis 拦截器在 SQL 执行前后介入；(2) FieldSnapshot 在 SQL 后恢复原始值；(3) ENCRYPT_HEADER 前缀保证幂等性。三个缺一不可。

7. ENCRYPT_HEADER 前缀 —— 防重复加密标记

定义在 ruoyi-common-core 的 Constants.ENCRYPT_HEADER = "ENC_"。EncryptorManager 的 encrypt/decrypt 方法使用它：

// 加密时：先检查前缀，再加密，再拼接前缀
encrypt(value, ctx):
  if (value.startsWith("ENC_")) return value;  // 已是密文，跳过
  encrypted = encryptor.encrypt(value, ctx.getEncode());
  return "ENC_" + encrypted;

// 解密时：先检查前缀，去除前缀，再解密
decrypt(value, ctx):
  if (!value.startsWith("ENC_")) return value; // 不是密文，跳过
  stripped = value.substring(4);
  return encryptor.decrypt(stripped);

解决的核心场景：

  • 多次 save：同一实体被多次 save，第二次 save 时字段值已是 "ENC_xxx"，自动跳过加密
  • 非加密字段：没有 "ENC_" 前缀的字段自动跳过解密
  • 外部直接 INSERT 的数据：明文数据没有前缀，不会误解密
双重防线的必要性：FieldSnapshot 恢复 + ENCRYPT_HEADER 前缀构成双重保障。Snapshot 恢复确保业务代码操作的是明文；前缀确保即使 Snapshot 恢复失败，底层也不会对已加密值重复加密。如果双重防线都失效，密文会被当作明文再次加密——双重加密的数据无法通过一次解密还原，数据永久损坏。

9. 课后练习

练习 1 · 单选题 · FieldSnapshot 恢复时机（点击展开）

Q: MybatisEncryptInterceptor 中 FieldSnapshot.restore() 在 finally 块中执行。以下哪个场景最能体现这个设计的必要性？

A. Controller 需要读取加密后的值进行二次处理
B. SQL 执行失败时，保证业务代码中的对象不被污染为密文状态
C. 为了满足 MyBatis Plugin 接口的规范要求
D. 提高加密性能，减少反射开销

答案：B。如果 SQL 抛异常，try 块中 proceed() 后的 restore() 不会执行，对象字段保留密文。操作同一对象的后续业务代码可能将密文当明文处理。finally 块保证无论 SQL 成败都恢复。

练习 2 · 单选题 · ENCRYPT_HEADER 行为（点击展开）

Q: 数据库中某字段存储为 "ENC_aGVsbG8="，应用层执行 userService.getById() 查询后，user.getPhone() 的值是什么？

A. "ENC_aGVsbG8="（原始数据库值）
B. "aGVsbG8="（去除前缀后的值）
C. "hello"（解密后的明文）
D. null

答案：C。MybatisDecryptInterceptor → EncryptorManager.decrypt() 检查 "ENC_" 前缀 → 去除 → 加密器解密 → 返回明文。不带前缀的值（如直接 INSERT 的明文）会直接返回不解密。

练习 3 · 判断题 · 递归范围（点击展开）

Q: EncryptedFieldProcessor 的递归处理能处理哪些类型？（多选逻辑）

A. 只处理实体顶层字段
B. 嵌套对象、Collection、Map
C. 用 IdentityHashMap 防循环引用
D. String 类型的 parameterObject 也能按字段加密

答案：B 和 C 正确。handle() 通过 instanceof 分支处理 Map（遍历 values）、Collection（遍历 items）、普通对象（反射字段），对复杂类型继续递归。IdentityHashMap 追踪已访问对象。D 错误——String 入口即被跳过。

练习 4 · 简答题 · Snapshot vs 深拷贝（点击展开）

Q: 如果不用 FieldSnapshot，改用"加密前深拷贝 → 加密原始对象 → SQL 执行 → 深拷贝覆盖回"方案，有什么优缺点？

参考：优点：实现直观。缺点：(1) 性能差——每次加密深拷贝整个实体；(2) 内存浪费，GC 压力大；(3) 语义不正确——深拷贝覆盖会丢失 SQL 执行后 MyBatis 可能回填的自增 ID 等值。Snapshot 只恢复被加密字段，是最小侵入性设计。

学习提示：EncryptedFieldProcessor 的递归遍历是模块最精妙的部分。建议在 IDE 中打开源码，重点关注: Map 处理、Collection 处理、普通对象处理三个分支，以及 IdentityHashMap 的选择理由。
