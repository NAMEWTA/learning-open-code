第 1 课 · 鸟瞰：代码生成器的全流程管线 入门地图

🎯 本节目标（对应使命 Success #1）：学完你能画出 ruoyi-gen 从「点击导入表」到「下载 ZIP」的完整处理流程图，说出每一层的核心文件和职责。这是后面所有细节挂靠的骨架——先有地图，再走小路。

ruoyi-gen 不是简单的字符串替换工具，而是一套元数据驱动的代码生成框架。它通过 AnyLine 元数据引擎读取数据库表结构，结合 FreeMarker 模板引擎，一键生成从后端（Java + MyBatis）到前端（Vue/React）再到数据库（SQL）的完整功能模块代码。本节我们先看清它到底分几层、每层干什么。

一、五层架构全景图

层次职责关键文件
1. API 接入层12 个 REST 端点：表查询、导入、配置、预览、下载、同步、数据源列表GenController.java
2. 业务服务层导入表时调用 AnyLine 读元数据、构建模板渲染上下文、ZIP 打包、同步数据库结构GenTableServiceImpl.java
3. 模板引擎层FreeMarker 引擎初始化、生成上下文（Dict 字典）、按 tplCategory + frontendType + 数据库类型选择模板组合、输出文件名计算TemplateEngineUtils.java
PathNamedTemplate.java
4. 类型推断层数据库类型→Java 类型→HTML 控件类型的自动映射；字段名后缀匹配智能控件（如 status→开关、image→图片上传）GenUtils.java
5. 数据适配层AnyLine 与 baomidou 动态数据源桥接：解决同一 JdbcTemplate 对应多种数据库的 adapter 匹配问题MyBatisDataSourceMonitor.java

二、一条完整的代码生成链路

我们选一条最典型的路径：用户在界面上选中一张数据库表，点击「生成代码」，下载 ZIP 文件。

三、模板产出物一览

一次标准生成产出 13~16 个文件（取决于是否启用导出/状态/排序/树表），分类如下：

①后端 Java 源码（8 个文件）
fm/java/*.ftl
Domain（实体）→ VO（视图对象）→ BO（业务对象）→ Mapper（数据访问）→ Service（接口）→ ServiceImpl（实现）→ Controller（控制器）+ Mapper XML
②前端 TS/Vue/React 源码（4 个文件）
fm/{vue|react}/*.ftl
API 封装（index.ts）→ TypeScript 类型定义（types.ts）→ CRUD 页面（index.vue/tsx）或树表页面（index-tree.vue/tsx）
③数据库菜单 SQL（1 个文件）
fm/sql/{mysql|oracle|postgres|sqlserver}.sql.ftl
按数据库类型自动选择方言，生成菜单权限的 INSERT 语句
💡 一句话记住这条链：
「用户导表 → AnyLine 读元数据 → GenUtils 推断类型 → FreeMarker 渲染模板 → ZIP 打包下载」

四、Controller 的 12 个 API 端点速览

GenController.java 是所有操作的统一入口，路径前缀 /tool/gen，12 个端点分为四组：

分组方法路径干什么
数据浏览GET/list分页查已导入的表
数据浏览GET/db/list分页查数据库中可导入的表
数据浏览GET/getDataNames获取可用数据源名称列表
数据浏览GET/{tableId}查某表的完整配置（表+字段）

五、核心数据结构：gen_table + gen_table_column

两张表承载了代码生成所需的全部元信息。理解它们的字段含义，是读懂整个模块的前提：

Agen_table -- 生成配置主表
GenTable.java
数据源名 → 表名/表注释 → 类名/包名/模块名/业务名/功能名 → 模板分类（crud/tree）→ 前端类型（vue/react）→ options（JSON：导出开关/状态字段/唯一校验/排序/树参数）
Bgen_table_column -- 字段配置明细表
GenTableColumn.java
列名/列类型 → Java 类型/Java 字段名 → 是否主键/自增/必填 → 是否参与增/改/查/列表 → 查询方式（EQ/LIKE/BETWEEN）→ 显示类型（input/select/switch/datetime/editor...）→ 字典类型

六、为什么这个设计值得学？

ruoyi-gen 展示了企业级代码生成器的标准范式：

  • 数据与模板分离：元数据（gen_table + gen_table_column）与渲染模板（FreeMarker .ftl）彻底解耦，加新语言/新框架只需加模板目录
  • 类型推断引擎：不是简单的一对一映射，而是「数据库类型 + 字段长度 + 字段名约定 + 业务语义」四维联合推断
  • 动态数据源适配：AnyLine 与 baomidou 的桥接方案是「多个框架共享一个连接池」的经典案例
🤔 想一想：为什么前端模板的页面文件（index.vue.ftl）用通配符动态发现，而不是像后端模板那样写死路径？ 因为后端模板种类固定（domain/vo/bo/mapper/service/serviceImpl/controller），而前端模板的文件扩展名由页面框架决定。Vue 用 .vue，React 用 .tsx，未来加 Svelte 可能是 .svelte。通配符匹配（classpath*:fm/{type}/index.*.ftl）让模板路径适配层不需要维护框架类型→扩展名的映射表——全由模板文件名自己表达。看 TemplateEngineUtils.getFrontendPageExtension() 的实现就清楚了。

七、检验一下（提取练习）

Q1. ruoyi-gen 的五层架构中，哪一层负责「数据库类型 varchar → Java 类型 String → 前端控件 input」的映射？

A. GenController B. TemplateEngineUtils C. GenUtils.initColumnField() D. MyBatisDataSourceMonitor
看答案C。GenUtils.initColumnField() 在表导入时对每个字段调用，依次执行：getDbType() 提取纯类型→按 COLUMNTYPE_STR/TIME/NUMBER 分组→设置 javaType→按字段名后缀匹配 htmlType。

Q2. 下载代码时，模板是按什么条件选择组合的？

A. 只按 tplCategory B. 按 tplCategory + frontendType + 数据库类型 三者组合 C. 所有模板全部渲染 D. 用户手动勾选
看答案B。TemplateEngineUtils.getTemplateList(tplCategory, dsName, frontendType) 按三个维度选择：模板分类决定用 index 还是 index-tree 页面；前端类型决定用 vue 还是 react 模板目录；数据库类型决定用哪种 SQL 方言模板。

Q3. gen_table 表中 options 字段存的是什么？

A. 纯文本描述 B. JSON 字符串，包含导出开关/状态字段/唯一校验/排序/树参数等 C. 生成的 SQL 脚本 D. 模板文件列表
看答案B。options 是 JSON 格式的扩展配置，由 updateGenTable() 在保存时通过 JsonUtils.toJsonString(params) 序列化，在 buildContext() 和 setTableFromOptions() 中反序列化使用。设计目的是避免为每个可选功能都加一行数据库字段。
