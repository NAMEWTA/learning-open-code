一、FreeMarker 引擎的初始化

ruoyi-gen 使用 Hutool 封装的 FreeMarker 引擎。初始化在 TemplateEngineUtils 的静态代码块中完成，全模块共享同一个引擎实例：

TemplateEngineUtils.java:60-69 
static {
    // 1. 创建模板配置：UTF-8编码，从 classpath 加载模板
    TemplateConfig templateConfig = new TemplateConfig(
        StandardCharsets.UTF_8,
        StringUtils.EMPTY,
        TemplateConfig.ResourceMode.CLASSPATH
    );
    // 2. 指定使用 FreeMarker 引擎（Hutool 支持多种模板引擎）
    templateConfig.setCustomEngine(FreemarkerEngine.class);
    TEMPLATE_ENGINE = TemplateUtil.createEngine(templateConfig);

    // 3. 禁用数字千分位格式化 —— 关键！
    //    否则 ID=1234567890 渲染出来变成 "1,234,567,890"
    if (TEMPLATE_ENGINE instanceof FreemarkerEngine freemarkerEngine) {
        freemarkerEngine.getConfiguration().setNumberFormat("computer");
    }

    // 4. 预加载所有固定路径的模板
    TEMPLATE_MAPPER = PathNamedTemplate.form(
        TEMPLATE_ENGINE, GenConstants.TEMPLATE_PATHS
    );
}

第 3 步值得单独强调：FreeMarker 默认会按 locale 对数字加千分位逗号。Snowflake 雪花 ID 是 Long 类型，不加 setNumberFormat("computer") 会导致生成的代码中 ID 值变成 1,234,567,890——这在 Java/TypeScript 中都是非法字面量。

二、PathNamedTemplate：模板的"身份证"

每个模板需要同时知道两件事：我是谁（模板路径，用于选取和日志）和我能渲染（调用 FreeMarker）。PathNamedTemplate 用装饰器模式将两者绑定：

PathNamedTemplate.java 
public class PathNamedTemplate implements Template {
    @Getter
    private final String pathName;      // "fm/java/controller.java.ftl"
    private final Template delegate;     // FreeMarker 的 Template 对象

    // 批量构建：路径集合 → 路径→模板的 Map
    public static Map<String, PathNamedTemplate> form(
            TemplateEngine engine, Set<String> pathNames) {
        Map<String, PathNamedTemplate> result = new HashMap<>();
        for (String pathName : pathNames) {
            result.put(pathName, form(engine, pathName));
        }
        return result;
    }

    @Override
    public String render(Map<?, ?> bindingMap) {
        return delegate.render(bindingMap);  // 委托给 FreeMarker
    }
}

设计意图：两个模板可能渲染出相同路径的文件（不同前端框架的 index 页面），但它们的模板路径不同（fm/vue/index.vue.ftl vs fm/react/index.tsx.ftl）。pathName 是区分它们的唯一标识。

三、模板选择逻辑：三维组合

getTemplateList(tplCategory, dsName, frontendType) 按三个维度拼出 13~16 个模板的清单：

TemplateEngineUtils.getTemplateList("crud", "master", "vue") │ ├─ 后端模板（固定 8 个，与前端类型无关） │ domain.java.ftl → vo.java.ftl → bo.java.ftl │ → mapper.java.ftl → service.java.ftl │ → serviceImpl.java.ftl → controller.java.ftl │ → mapper.xml.ftl │ ├─ 前端 API + 类型（按 frontendType 动态拼接路径） │ getFrontendTemplatePath("vue", "api.ts.ftl") │ → "fm/vue/api.ts.ftl" │ getFrontendTemplatePath("vue", "types.ts.ftl") │ → "fm/vue/types.ts.ftl" │ ├─ SQL 模板（按数据库类型） │ DataBaseHelper.getDataBaseType("master") │ → isMySQL()? → fm/sql/mysql.sql.ftl │ → isOracle()? → fm/sql/oracle.sql.ftl │ → isPostgreSql()? → fm/sql/postgres.sql.ftl │ → isSqlServer()? → fm/sql/sqlserver.sql.ftl │ └─ 前端页面模板（按 tplCategory） tplCategory == "crud" → getFrontendIndexTemplatePath("vue") tplCategory == "tree" → getFrontendIndexTreeTemplatePath("vue") → 通配符匹配: classpath*:fm/vue/index.*.ftl → 取第一个匹配结果: fm/vue/index.vue.ftl → 扩展名 .vue 由模板文件名决定
💡 关键设计：前端页面模板路径不写死，而是通过 classpath*:fm/{frontendType}/index.*.ftl 通配符动态发现。这样做的好处是：如果将来加 Svelte 前端，只需在 fm/svelte/ 下放 index.svelte.ftl，不需要改一行 Java 代码。

四、buildContext：GenTable → FreeMarker 字典

TemplateEngineUtils.buildContext(GenTable) 是整个模块信息密度最高的方法。它将 GenTable 对象的所有信息展开为扁平的键值字典，供 FreeMarker 模板使用。核心变量分组如下：

分组变量列表来源

设计亮点：setColumnFeatureContext() 预先扫描所有列的 htmlType，生成诸如 needEditor、needImageUpload 等布尔开关。这让模板不需要在渲染时遍历列列表去做条件判断。例如 Controller 模板中 <#if needSwitchField> 只需要一个判断，而不是在渲染时扫描整个 columns 列表。

现在我们把理论变为实践。假设要支持 Svelte 作为第三种前端框架，需要以下步骤：

1创建模板目录和文件
ruoyi-gen/src/main/resources/fm/svelte/

不需要改 GenConstants.java

但 API 和 types 模板是固定文件名 api.ts.ftl 和 types.ts.ftl，也需要放在 fm/svelte/ 下。

3验证路径计算

八、检验一下

Q1. FreeMarker 引擎初始化时为什么要 setNumberFormat("computer")？

A. 让 FreeMarker 使用科学计数法显示数字 B. 禁用数字千分位格式化，防止雪花 ID 被渲染成 "1,234,567,890" C. 提高渲染速度 D. 这个设置没有实际作用
看答案B。FreeMarker 默认对数字使用千分位逗号格式化。RuoYi-Vue-Plus 使用 19 位雪花 ID，不关闭会导致 Java 代码中出现非法数字字面量。

Q2. 前端页面模板路径为什么用 classpath*:fm/{type}/index.*.ftl 通配符，而不是像后端模板那样写死？

A. 后端模板也有很多扩展名，用通配符太乱 B. 前端模板的扩展名由页面框架决定（.vue/.tsx/.svelte），通配符让框架扩展不需要改 Java 代码——模板文件名本身就是扩展名声明 C. 这是历史遗留代码
看答案B。后端模板的扩展名固定为 .java，不需要通配符。前端页面模板的扩展名随框架变化（Vue 用 .vue、React 用 .tsx、Svelte 用 .svelte），通配符匹配让文件名本身携带扩展名信息，getFrontendPageExtension() 从文件名中提取扩展名用于生成目标文件。

Q3. setColumnFeatureContext() 扫描所有列后设置了 needEditor、needSwitchField 等布尔变量。模板中直接判断这些变量，而不在模板里扫描 columns。这有什么用？

A. 纯粹为了代码风格统一 B. 减少模板复杂度——模板只需判断一个布尔值，不需要写 FreeMarker 的循环和过滤逻辑；同时集中 Java 端维护扫描逻辑，便于测试和复用 C. 因为 FreeMarker 不支持扫描列表
看答案B。将列扫描逻辑集中在 Java 端的 setColumnFeatureContext() 中，让 FreeMarker 模板保持简洁。模板只需要 <#if needEditor> 这类简单判断，不需要写 <#if columns?filter(c -> c.htmlType == 'editor')?size gt 0> 这样的复杂表达式。集中管理也让扫描逻辑可测试、可维护。
