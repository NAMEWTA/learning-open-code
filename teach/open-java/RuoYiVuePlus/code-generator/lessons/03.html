第 3 课 · 字段类型自动推断：从数据库列到前端控件 核心引擎

🎯 本节目标（对应使命 Success #4）：学完你能口述字段类型自动推断的完整规则链——数据库列类型 → Java 类型 → 前端控件类型的三层映射，以及按字段名后缀匹配控件的命名约定。理解为什么同一个 varchar 字段，叫 status 生成开关控件，叫 name 生成文本框。

二、第二层映射：提取纯数据库类型

原始列类型可能带长度和精度，如 varchar(255)、decimal(10,2)。第一步是提取"纯类型名"，并获取长度和精度：

columnType = "decimal(10,2)" │ ├─ GenUtils.getDbType("decimal(10,2)") │ → 截取 '(' 之前 → "decimal" │ ├─ GenUtils.getColumnLength("decimal(10,2)") │ → 取括号内逗号之前 → 10 │ └─ GenUtils.getColumnScale("decimal(10,2)") → 取括号内逗号之后 → 2

三、第二层映射：数据库类型 → Java 类型

纯类型名被分入四个大类，每类走不同的 Java 类型推断逻辑：

类 1：字符串类型 → String / Boolean / 文本域

数据库类型（来源：COLUMNTYPE_STR + COLUMNTYPE_TEXT）默认 Java 类型例外：tinyint 长度=1 且字段名是开关类 → Boolean
char, varchar, enum, set, nchar, nvarchar, varchar2, nvarchar2String—
tinytext, text, mediumtext, longtext, blob, clob, ntext, bytea ...String—
额外逻辑：字符串类型下，如果字段名被判定为布尔字段（如 tinyint(1) + 字段名为 status/enabled/is_active），javaType 改为 Boolean，htmlType 改为 switch。

类 2：时间类型 → LocalDateTime

数据库类型（来源：COLUMNTYPE_TIME）Java 类型
datetime, time, date, timestamp, year, interval, smalldatetime, datetime2, datetimeoffset, timestamptzLocalDateTime

类 3：数值类型 → 五级精度阶梯

这是最复杂的推断。在 resolveNumberJavaType() 中，按「类型 + 长度 + 精度 + 字段名」四维联合判断：

数据库类型条件Java 类型
decimal, numeric, money, smallmoneyscale > 0BigDecimal
decimal, numeric, money, smallmoneyscale = 0, length <= 9Integer
decimal, numeric, money, smallmoneyscale = 0, length > 9Long
float, double, real, float4, float8, double precision—Double
bigint, int8, bigserial—Long
smallint, mediumint, int, int2, int4, integer, serial—Integer
number (Oracle)scale > 0BigDecimal
number (Oracle)scale = 0, length <= 9Integer
number (Oracle)scale = 0, length > 9Long
💡 设计要点：isBooleanColumn() 优先于数值类型判断。如果字段类型是 tinyint/number/char 且长度为 1，且字段名匹配开关类后缀（status/flag/enabled/disabled/available/visible 或 is_/has_/enable_/disable_ 前缀），则 javaType 为 Boolean。

四、第三层映射：字段名后缀 → HTML 控件类型

这是整个推断引擎最"智能"的部分。在第二层设置了默认 htmlType 之后，第三层按字段名的语义后缀进行覆盖。匹配顺序即优先级：

列名: user_status (varchar) 列名: avatar_image (varchar) │ │ 第二层默认 → htmlType = "input" 第二层默认 → htmlType = "input" │ │ 第三层覆盖： 第三层覆盖： isSwitchColumn("user_status") isSwitchColumn("avatar_image") → endsWith("status") ✓ → htmlType = "switch" → ✗ endsWith("image") ✓ → htmlType = "imageUpload" 最终输出: javaType=String, htmlType=switch 最终输出: javaType=String, htmlType=imageUpload

完整匹配规则表（按源码顺序，即优先级从高到低）：

匹配规则触发条件（字段名后缀/前缀）覆盖为 htmlType
字符串长文本列长度 >= 500 或属于 TEXT 类型textarea
布尔/开关优先isSwitchColumn() 或 javaType=Booleanswitch
状态字段endsWith("status")radio（非 Boolean）或 switch（Boolean）
类型/性别endsWith("type") 或 endsWith("sex")select
排序字段isSortColumn()：endsWith(sort/order_num/order/rank/seq/sequence)inputNumber
图片字段endsWith(image/avatar/logo/picture)imageUpload
文件字段endsWith(file/attachment)fileUpload
备注描述endsWith(remark/description/desc/note)textarea
内容字段endsWith(content/html/body)editor
关键理解：第三层的匹配是覆盖关系，不是兜底关系。一个字段先由第二层设置默认 htmlType（如 varchar→input），再被第三层按后缀精确覆盖（如 ...status→radio）。正是这个两层机制，让同名约定优于类型推断。

五、额外推断：查询方式 + 参与场景

除了类型映射，initColumnField() 还做了两项自动配置：

查询方式自动推断

①姓名类字段 → 模糊查询
endsWith(columnName, "name") 时，queryType 设为 LIKE（默认是 EQ）。

②日期字段 + 参与查询 → 范围查询
htmlType == "datetime" && isQuery() 时，queryType 设为 BETWEEN（实现日期范围的起止时间段查询）。

默认参与哪些场景？黑名单机制

每个列有四个开关（isInsert / isEdit / isList / isQuery），默认全部开启，但有黑名单排除：

场景排除的字段（COLUMNNAME_NOT_*）
新增（isInsert）create_dept, create_by, create_time, del_flag, update_by, update_time, version + 主键字段
编辑（isEdit）create_dept, create_by, create_time, del_flag, update_by, update_time, version
列表（isList）create_dept, create_by, create_time, del_flag, update_by, update_time, version
查询（isQuery）id + create_dept, create_by, create_time, del_flag, update_by, update_time, remark, version

六、GenTableColumn 的辅助方法：让模板更干净

列对象提供了一系列 is*() 方法，让 FreeMarker 模板里的判断语简洁——模板不需要自己比较字符串 "1"：

方法作用模板中的典型用法
isPk()是否主键<#if column.pk>@TableId(...)
isInsert()是否参与新增<#if column.insert> 控制 BO 字段可见性
isEdit()是否参与编辑同上
isList()是否参与列表<#if column.list> 控制 VO 和前端表格列
isQuery()是否参与查询<#if column.query> 控制 BO 的查询条件
isSuperColumn()是否基类公共字段<#if !table.isSuperColumn(column.javaField)> 跳过 Domain 生成
getTsType()TypeScript 类型types.ts.ftl 中自动输出 string | number | boolean
isDictColumn()是否字典控件列决定是否生成字典翻译代码

七、检验一下

Q1. 一个字段 columnType = tinyint(1)，columnName = is_enabled，最终 javaType 和 htmlType 是什么？

A. javaType=Integer, htmlType=input B. javaType=Long, htmlType=inputNumber C. javaType=Boolean, htmlType=switch —— tinyint(1) 且字段名 is_enabled 匹配 isSwitchColumn D. javaType=String, htmlType=radio
看答案C。isBooleanColumn() 判断 tinyint 长度=1 且 isSwitchColumn("is_enabled") 命中 startsWith("is_") → true。因此 javaType=Boolean，htmlType=switch。

Q2. varchar 类型的 user_name 和 product_price 的 htmlType 和 queryType 分别是什么？

A. user_name: input + EQ; product_price: input + EQ B. user_name: input + LIKE; product_price: input + EQ C. user_name: textarea + LIKE; product_price: inputNumber + EQ
看答案B。user_name 命中 endsWith("name") → queryType=LIKE。product_price 不命中任何第三层后缀规则，保持默认 htmlType=input、queryType=EQ。price 虽然暗示数字，但 varchar 类型走字符串分类，第三层也没有专门匹配 price 后缀的规则。

Q3. 为什么 isInsert 黑名单包含主键字段，而 isEdit 不包含？

A. 代码 bug B. 主键通常自增，新增时不需要传，但编辑时需要标识要修改哪一行 C. 为了节约代码量
看答案B。新增时主键由数据库自增或雪花算法生成，不应由前端传入。但编辑（updateById）时必须用主键标识目标行，所以 edit 场景不排除主键。看代码：!arraysContains(COLUMNNAME_NOT_ADD, columnName) && !column.isPk() —— isPk() 是 isInsert 的额外排除条件。
