第 2 课 · AnyLine 元数据引擎与动态数据源桥接 数据适配

🎯 本节目标（对应使命 Success #5）：学完你能口述 MyBatisDataSourceMonitor 如何解决「同一个 JdbcTemplate 对应多种数据库」的适配问题，理解 KEEP_ADAPTER、feature()、key() 三个关键方法的作用，并掌握 AnyLine 元数据 API 的两种核心用法（列出表、读列信息）。

一、问题背景：两个框架的"领域冲突"

代码生成器需要读取任意数据源中的数据库表结构（表名、列名、列类型、主键、自增等元数据）。项目中使用了两套数据访问框架：

框架职责核心抽象
baomidou dynamic-datasource多数据源切换：@DS("slave") 注解控制当前线程用哪个数据源DynamicRoutingDataSource
AnyLine通用数据操作：提供元数据 API（ServiceProxy.metadata().tables()）自动读取表结构DataRuntime + adapter

问题出在这里：baomidou 的 DynamicRoutingDataSource 是一个路由代理——它包裹了多个真实数据源（MySQL、Oracle、PostgreSQL 等）。切换 @DS 注解只是改了路由目标，但 AnyLine 看到的 JdbcTemplate 始终是同一个对象。

⚠️ 核心矛盾：AnyLine 默认假设一个 JdbcTemplate 对应一种数据库类型，会为它绑定一个固定的 adapter（适配器）。但 DynamicRoutingDataSource 下同一个 JdbcTemplate 可能背后切换成 MySQL → Oracle → PostgreSQL。如果 AnyLine 缓存了 MySQL 的 adapter，切到 Oracle 后仍然用 MySQL adapter 解析元数据，就会出错。

二、MyBatisDataSourceMonitor：桥接器的设计

MyBatisDataSourceMonitor 实现了 AnyLine 的 DataSourceMonitor 接口，用三个方法精确控制 adapter 的匹配与缓存策略。源码位置：

1构造函数：设置执行模式
MyBatisDataSourceMonitor.java:31-36 
public MyBatisDataSourceMonitor() {
    // KEEP_ADAPTER = 2: 自定义模式，由 keepAdapter() 方法决定
    ConfigTable.KEEP_ADAPTER = 2;
    // 禁用元数据缓存（数据源可能切换）
    ConfigTable.METADATA_CACHE_SCOPE = 0;
}

两行配置定死基调：不用 AnyLine 默认的缓存策略，一切由我们自己的 Monitor 接管。

2keepAdapter()：告诉 AnyLine "别缓存 adapter"
MyBatisDataSourceMonitor.java:102-109 
public boolean keepAdapter(DataRuntime runtime, Object datasource) {
    if (datasource instanceof JdbcTemplate jdbc) {
        DataSource ds = jdbc.getDataSource();
        // 关键：DynamicRoutingDataSource 不保持 adapter
        return !(ds instanceof DynamicRoutingDataSource);
    }
    return true;
}

当数据源是 DynamicRoutingDataSource 时返回 false，告诉 AnyLine：不要为这个 JdbcTemplate 缓存 adapter，每次操作都需要重新匹配。这就解决了「同一 JdbcTemplate 对应多种数据库」的问题。

3key()：用当前数据源名当唯一标识
MyBatisDataSourceMonitor.java:83-92 
public String key(DataRuntime runtime, Object datasource) {
    if (datasource instanceof JdbcTemplate jdbc) {
        DataSource ds = jdbc.getDataSource();
        if (ds instanceof DynamicRoutingDataSource) {
            // 用 baomidou 当前上下文中持有的数据源名
            return DynamicDataSourceContextHolder.peek();
        }
    }
    return runtime.getKey();
}

AnyLine 需要给 adapter 一个唯一键。如果直接用 JdbcTemplate 的 hashCode，所有数据源共享同一把钥匙，adapter 缓存就乱套了。所以这里取 baomidou 当前线程的 数据源名称（如 "master"、"slave"）作为 key。

4feature()：构造 adapter 匹配指纹
MyBatisDataSourceMonitor.java:47-74 
public String feature(DataRuntime runtime, Object datasource) {
    String feature = null;
    if (datasource instanceof JdbcTemplate jdbc) {
        DataSource ds = jdbc.getDataSource();
        if (ds instanceof DynamicRoutingDataSource) {
            String key = DynamicDataSourceContextHolder.peek();
            feature = features.get(key);
            if (null == feature) {
                Connection con = null;
                try {
                    con = DataSourceUtils.getConnection(ds);
                    DatabaseMetaData meta = con.getMetaData();
                    // 产品名 + URL 合成唯一特征值
                    feature = meta.getDatabaseProductName()
                        .toLowerCase().replace(" ", "")
                        + "_" + meta.getURL();
                    features.put(key, feature);
                } finally { /* 释放连接 */ }
            }
        }
    }
    return feature;
}

feature 是 AnyLine 用来匹配正确 adapter 的指纹。它通过 JDBC 元数据拿到 产品名（如 "MySQL"、"PostgreSQL"） + 连接 URL 拼接成特征串。AnyLine 根据这个特征串选择对应的方言 adapter。同时用 Map<String, String> features 缓存，避免每次读取元数据都开一次连接。

三、AnyLine 元数据 API 的实际调用

有了 Monitor 保障 adapter 正确匹配，代码生成器就可以安全地调用 AnyLine 的元数据 API。在 GenTableServiceImpl 中有两个最典型的调用场景：

场景 1：列出数据库中所有表

GenTableServiceImpl.java:126-184 — selectPageDbTableList() 
@DS("#genTable.dataName")   // ← baomidou 切换到目标数据源
public PageResult<GenTable> selectPageDbTableList(GenTable genTable, PageQuery pageQuery) {
    // AnyLine 一行读所有表元数据
    LinkedHashMap<String, Table<?>> tablesMap =
        ServiceProxy.metadata().tables();

    // 过滤掉系统表前缀 + 已导入的表
    tablesMap.values().stream()
        .filter(x -> !StringUtils.startWithAnyIgnoreCase(
            x.getName(), "sai_", "sj_", "flow_", "gen_"))
        .filter(x -> !已导入表名列表.contains(x.getName()))
        .map(x -> { /* 转换为 GenTable 对象 */ })
        .toList();
}

关键：@DS("#genTable.dataName") 在方法执行前切数据源，ServiceProxy.metadata().tables() 透传当前 JdbcTemplate 的 connection 给 AnyLine，Monitor 的 feature() 正确识别了数据库类型，因而能正确解析该数据源的元数据。

场景 2：读取指定表的列详细信息

GenTableServiceImpl.java:296-316 — selectDbTableColumnsByName() 
@DS("#dataName")
public List<GenTableColumn> selectDbTableColumnsByName(
        String tableName, String dataName) {
    // AnyLine 一行读指定表
    Table<?> table = ServiceProxy.metadata().table(tableName);

    LinkedHashMap<String, Column> columns = table.getColumns();
    columns.forEach((columnName, column) -> {
        GenTableColumn tc = new GenTableColumn();
        tc.setIsPk(column.isPrimaryKey() ? "1" : "0");
        tc.setColumnName(column.getName());
        tc.setColumnComment(column.getComment());
        tc.setColumnType(column.getOriginType().toLowerCase());
        tc.setSort(column.getPosition());
        tc.setIsRequired(column.isNullable() ? "0" : "1");
        tc.setIsIncrement(column.isAutoIncrement() ? "1" : "0");
    });
}

从 Column 对象能读到六个关键属性：名称、注释、原始类型、是否主键、是否可空、是否自增、字段位置。这就是类型推断的全部原始材料。

💡 一句话总结这套桥接方案：
@DS 注解决定「连哪个库」→ keepAdapter=false 阻止 AnyLine 缓存 adapter → key() 用数据源名区分 → feature() 用产品名+URL 匹配正确 adapter → metadata().tables() 正确读取表结构。
五个环节，环环相扣。

四、导入表的完整流程（带分布式锁）

导入表是唯一需要防并发的操作，因为它同时操作 gen_table 和 gen_table_column 两张表，且依赖 AOP 代理调用（切数据源）。

POST /tool/gen/importTable?tables=sys_user&dataName=master │ ├─ GenController.importTableSave() │ · @Lock4j(keys={"#dataName"}) ← 同一数据源串行导入 │ · @RepeatSubmit() ← 防重复提交 │ ├─ selectDbTableListByNames(tables, dataName) │ · @DS("#dataName") → AnyLine 读表 │ └─ importGenTable(tableList, dataName) · @DSTransactional ← 支持跨库事务 · for each table: GenUtils.initTable(table) ← 初始化表配置 tableMapper.insert(table) ← 保存 gen_table SpringUtils.getAopProxy(this) ← 获取 AOP 代理 .selectDbTableColumnsByName() ← 切数据源读列 GenUtils.initColumnField() ← 推断每个列的类型 columnMapper.insertBatch() ← 保存 gen_table_column

注意 SpringUtils.getAopProxy(this) 的用法。因为 importGenTable 上挂了 @DSTransactional，直接 this.selectDbTableColumnsByName() 会绕过切面，@DS 不生效。所以必须通过 AOP 代理对象 调用，才能切到正确数据源读列信息。

五、检验一下

Q1. MyBatisDataSourceMonitor 中哪个方法直接回答了「同一个 JdbcTemplate 可能对应多种数据库，要不要缓存 adapter」这个问题？

A. feature() B. key() C. keepAdapter() —— 对 DynamicRoutingDataSource 返回 false，告诉 AnyLine 不缓存 adapter D. 构造函数中的 ConfigTable
看答案C。keepAdapter() 是核心开关。对于 DynamicRoutingDataSource 返回 false，AnyLine 每次都会根据 feature() 重新匹配正确的 adapter，从而支持数据库类型切换。

Q2. feature() 方法的返回值包含什么信息？为什么选这些信息？

A. 只含数据源名称 B. 只含 JDBC URL C. 数据库产品名 + JDBC URL，产品名决定方言种类，URL 区分同类型的不同实例 D. JdbcTemplate 的 hashCode
看答案C。产品名（如 MySQL、PostgreSQL）用于匹配 AnyLine 的方言 adapter，URL 用于区分同一类数据库的不同实例（如开发库 vs 测试库）。两者拼接确保唯一性。

Q3. importTableSave 为什么同时用了 @Lock4j 和 @RepeatSubmit？

A. 装饰性注解，没实际作用 B. Lock4j 防并发导入同一数据源下的表，RepeatSubmit 防用户连续双击提交 C. 这是复制粘贴错误
看答案B。Lock4j 是分布式锁（按数据源名锁定），防止多个请求同时导入同一数据源的表导致重复记录；RepeatSubmit 是请求级防重（基于 Redis），防止用户快速双击提交按钮。
