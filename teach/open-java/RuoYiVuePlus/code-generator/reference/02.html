速查 02 · AnyLine 元数据引擎与动态数据源桥接 cheat sheet

核心矛盾：为什么需要桥接器？

baomidou 的 DynamicRoutingDataSource 是路由代理，包裹多个真实数据源。@DS 注解只改路由目标，但 AnyLine 看到的 JdbcTemplate 始终是同一个对象。AnyLine 默认假设 1 个 JdbcTemplate = 1 种数据库类型，会缓存固定 adapter —— 数据源切换后就会用错 adapter。

MyBatisDataSourceMonitor 四个关键方法

方法作用核心代码/逻辑
构造函数 禁用 AnyLine 默认缓存，由 Monitor 接管 ConfigTable.KEEP_ADAPTER = 2; // 自定义模式
ConfigTable.METADATA_CACHE_SCOPE = 0; // 禁用元数据缓存
keepAdapter() 告诉 AnyLine 是否缓存 adapter return !(ds instanceof DynamicRoutingDataSource);
key() 用 baomidou 当前数据源名当唯一键 return DynamicDataSourceContextHolder.peek();
feature() 构造 adapter 匹配指纹 meta.getDatabaseProductName() + "_" + meta.getURL();
// 从 JDBC Connection 元数据获取，缓存到 Map<String,String>

桥接方案完整链路

@DS("#genTable.dataName")           → 切到目标数据源
  │
keepAdapter() → false               → 不缓存 adapter，每次都重新匹配
  │
key() → dataSourceName              → 用数据源名区分不同连接
  │
feature() → "mysql_jdbc:mysql://..." → 产品名决定方言，URL 区分实例
  │
AnyLine 根据 feature 匹配 adapter   → 用正确的方言解析元数据
  │
ServiceProxy.metadata().tables()    → 正确读取表结构
@DS 切库 → keepAdapter=false 禁止缓存 → key() 用数据源名区分 → feature() 用 DB产品名+URL 匹配 adapter → metadata() 正确读表结构。五个环节，环环相扣。

AnyLine 元数据 API 两种典型调用

场景API返回值
列出数据库中所有表 ServiceProxy.metadata().tables() LinkedHashMap<String, Table>，过滤 sai_/sj_/flow_/gen_ 前缀
读指定表的列详情 ServiceProxy.metadata().table(tableName) Table → getColumns() → Column: name, comment, originType, isPrimaryKey, isNullable, isAutoIncrement, position

导入表流程（带防并发）

POST /tool/gen/importTable?tables=sys_user&dataName=master
  │
  ├─ GenController.importTableSave()
  │   · @Lock4j(keys={"#dataName"})        ← 同一数据源串行导入
  │   · @RepeatSubmit()                     ← 防重复提交（基于 Redis）
  │
  ├─ selectDbTableListByNames(tables, dataName)
  │   · @DS("#dataName") → AnyLine 读表
  │
  └─ importGenTable(tableList, dataName)
      · @DSTransactional                    ← 跨库事务
      · for each table:
           GenUtils.initTable(table)         ← 表名→类名，设默认值
           tableMapper.insert(table)         ← 保存 gen_table 记录
           SpringUtils.getAopProxy(this)     ← ⚠️ 必须用 AOP 代理！
             .selectDbTableColumnsByName()   ← @DS 才能生效
           GenUtils.initColumnField()        ← 推断字段类型
           columnMapper.insertBatch()        ← 批量保存列配置

关键陷阱：AOP 代理调用

写法结果
this.selectDbTableColumnsByName()❌ @DS 不生效 → 读错数据源
SpringUtils.getAopProxy(this).selectDbTableColumnsByName()✅ AOP 拦截 → @DS 正确切数据源

原因：@DSTransactional 和 @DS 都是 AOP 切面。直接 this.xxx() 绕过代理，切面不触发。

系统表过滤前缀

导入时自动过滤以下前缀的表：sai_、sj_、flow_、gen_（代码生成器自身的两张表）。
