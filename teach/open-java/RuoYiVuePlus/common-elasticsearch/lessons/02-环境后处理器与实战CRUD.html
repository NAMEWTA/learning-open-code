第 2 课 · 环境后处理器与实战 ES CRUD 启动时序 + 实战

学习目标：学完本课，你能讲清 ActuatorEnvironmentPostProcessor 如何通过 EnvironmentPostProcessor + Ordered.HIGHEST_PRECEDENCE 在 Spring 启动最早阶段同步配置开关，并能追踪从 application.yml 开启 ES 到 EsCrudController 完成 CRUD 的完整链路——甚至可以自己新建一个 ES 索引和查询接口。

上一课讲了 EasyEsConfiguration 的自动装配三件套，这一课讲模块的第二个 Java 类——ActuatorEnvironmentPostProcessor，然后结合 DEMO 代码走通完整的 ES CRUD 链路。

第一部分：ActuatorEnvironmentPostProcessor

一、源码全文（37 行）

package org.dromara.common.elasticsearch.config;

import org.springframework.boot.EnvironmentPostProcessor;
import org.springframework.boot.SpringApplication;
import org.springframework.core.Ordered;
import org.springframework.core.env.ConfigurableEnvironment;

/**
 * 健康检查配置注入
 *
 * @author Lion Li
 */
public class ActuatorEnvironmentPostProcessor
        implements EnvironmentPostProcessor, Ordered {

    @Override
    public void postProcessEnvironment(
            ConfigurableEnvironment environment,
            SpringApplication application) {
        String enable = environment.getProperty(
            "easy-es.enable", "false");
        System.setProperty(
            "management.health.elasticsearch.enabled", enable);
    }

    @Override
    public int getOrder() {
        return Ordered.HIGHEST_PRECEDENCE;
    }
}

代码非常简短，但承载了一个重要的架构模式——配置联动。我们拆开看。

二、它解决什么问题？

Spring Boot Actuator 内置了一个 Elasticsearch 健康检查指示器（ElasticsearchHealthIndicator）。当 management.health.elasticsearch.enabled=true 时，Actuator 的 /actuator/health 端点会尝试连接 ES 集群并报告连接状态。

问题来了：如果你把 easy-es.enable 设成 false 不使用 ES，但 management.health.elasticsearch.enabled 仍然是默认的 true……会发生什么？

健康检查会报 DOWN：Actuator 尝试连接 localhost:9200，但根本没有 ES 服务在运行，健康检查返回 DOWN。如果你的应用跑在 Kubernetes 上，健康探针检测到 DOWN 会触发 Pod 重启——而重启后还是会 DOWN，陷入无限重启循环。

ActuatorEnvironmentPostProcessor 做的事就是在 Spring 启动的最早阶段，把 easy-es.enable 的值同步到 management.health.elasticsearch.enabled：

  • easy-es.enable=true → 健康检查也开启（正常检查 ES 连接状态）
  • easy-es.enable=false → 健康检查也关闭（不检查不报错）

一个开关控制两个组件的启停——这就是配置联动的经典应用场景。

三、为什么必须在「最早阶段」？

Spring Boot 的启动有一个严格的时序：

Spring Boot 启动时序（简化版） ═══════════════════════════════════════════════════════ 第 1 阶段：准备 Environment ├── 读取 application.yml、系统属性、环境变量等 ├── ★ EnvironmentPostProcessor 在此阶段执行 ★ └── ActuatorEnvironmentPostProcessor 在此设置配置 第 2 阶段：准备 ApplicationContext ├── 扫描 .imports 文件 └── 加载 AutoConfiguration 类（EasyEsConfiguration 在此） 第 3 阶段：刷新 ApplicationContext ├── 实例化所有单例 Bean ├── ★ 健康检查组件在此初始化 ★ └── 读取 management.health.elasticsearch.enabled 如果 = false → ElasticsearchHealthIndicator 不创建

关键点在时序上：

  • EnvironmentPostProcessor 在第 1 阶段执行
  • 健康检查组件在第 3 阶段初始化
  • 必须确保「设置」在前，「读取」在后

这就是 getOrder() 返回 Ordered.HIGHEST_PRECEDENCE（值为 Integer.MIN_VALUE）的原因：在所有 EnvironmentPostProcessor 中第一个执行，不给任何其他处理器留下「读到旧值」的时间窗口。

设计洞察：EnvironmentPostProcessor 是 Spring Boot 提供的最早干预点——比 BeanFactoryPostProcessor 更早，比 BeanPostProcessor 早得多。它存在于 ApplicationContext 创建之前，专用于在配置层面做「环境修正」。本模块用它解决了「一个开关 vs 两个组件」的同步问题，但这只是冰山一角——任何需要「用一个配置影响多个框架行为」的场景都可以用这个模式。

四、注册方式：spring.factories

注意：EnvironmentPostProcessor 不是通过 AutoConfiguration.imports 注册的。它用的是传统的 spring.factories：

# src/main/resources/META-INF/spring.factories
org.springframework.boot.EnvironmentPostProcessor=\
  org.dromara.common.elasticsearch.config.ActuatorEnvironmentPostProcessor

为什么不能用 .imports？因为 .imports 是给 @AutoConfiguration 类用的（第 2 阶段加载），而 EnvironmentPostProcessor 不是一个配置类——它是 Spring Boot 的 SPI 扩展点，必须通过 spring.factories 中对应的 key 注册（第 1 阶段加载）。

现在你能分清本模块的两个注册文件了：

文件加载阶段注册的内容用途
AutoConfiguration.imports第 2 阶段EasyEsConfiguration自动配置类发现
spring.factories第 1 阶段ActuatorEnvironmentPostProcessor环境后处理器注册

两者职责不同，时序不同，注册方式不同——虽然它们在同一个模块里，但走的是完全不同的两条 SPI 通道。

五、设计细节解读

System.setProperty vs environment.getPropertySources().addFirst()：有人会问为什么不直接用 Spring Environment API 来设置属性？

答案是：Actuator 的健康检查配置绑定在某些版本中不经过 Spring Environment。Actuator 的 ElasticsearchHealthIndicator 使用 management.health.elasticsearch.enabled 作为系统属性后备。直接设置 JVM 系统属性能确保无论 Actuator 从哪个属性源读取，都能拿到正确的值——这是防御性编程：做最可靠的设置，覆盖所有可能的读取路径。

environment.getProperty("easy-es.enable", "false") 中的 "false" 是默认值——如果 application.yml 中根本没写这个配置项（虽然不太可能），就默认视为 false，安全地关闭健康检查。这个默认值的策略和 @ConditionalOnProperty 的 matchIfMissing=false 形成双重保障：即使配置缺失，也不会误开启 ES 健康检查。

第二部分：完整 CRUD 链路追踪

六、从配置到查询：一条请求的完整旅程

现在把第 1 课和第 2 课的知识串起来，追踪一次真实的 ES 查询请求——从 Spring 启动到请求返回的完整链路：

GET /es/select?title=手册 ═══════════════════════════════════════════════════════ 【启动阶段：Spring Boot 初始化】 第 1 步：spring.factories → 加载 ActuatorEnvironmentPostProcessor → postProcessEnvironment() 执行 → 读 easy-es.enable → 设为 true → System.setProperty("management.health.elasticsearch.enabled", "true") → 此后健康检查组件初始化时会读到这个值 第 2 步：.imports 文件 → 发现 EasyEsConfiguration → @ConditionalOnProperty 检查 easy-es.enable=true ✓ → @EsMapperScan("org.dromara.**.esmapper") → 扫描到 org.dromara.demo.esmapper.DocumentMapper → 检查 extends BaseEsMapper<Document> ✓ → JDK 动态代理生成实现 → 注册为 Spring Bean (documentMapper) 第 3 步：依赖注入 → EsCrudController 构造器 → private final DocumentMapper documentMapper; → Spring 注入代理对象 【运行阶段：请求到达】 第 4 步：HTTP 请求 → GET /es/select?title=手册 → EsCrudController.select("手册") 第 5 步：构造查询条件 → LambdaEsQueryWrapper<Document> wrapper = new LambdaEsQueryWrapper<>(); → wrapper.eq(Document::getTitle, "手册"); 第 6 步：调用 Mapper → documentMapper.selectOne(wrapper); → 动态代理拦截方法调用 第 7 步：Easy-Es 核心引擎处理 → Lambda 表达式 → ES JSON DSL: {"query": {"term": {"title": "手册"}}} → 通过 ES 7.17.28 REST 客户端发送请求 第 8 步：ES 集群响应 → 返回 JSON → 反序列化为 Document 对象 第 9 步：返回给前端 → Document{id="...", title="手册", content="..."}

注意：如果 easy-es.enable=false，这个流程在第 2 步就会中止——EasyEsConfiguration 不会被加载，DocumentMapper 不会被扫描，EsCrudController 上的 @ConditionalOnProperty 也不会让它成为 Bean。同时第 1 步已经把健康检查也关了。整个 ES 相关能力完全消失，零开销。

七、DEMO 代码详解

7.1 Document 实体

位置：org.dromara.demo.domain.Document

@Data
public class Document {
    private String id;
    private String title;
    private String content;
}

最简单的 POJO，三个字段：id（文档唯一标识）、title（标题）、content（内容）。没有 @IndexName 注解时，Easy-Es 默认使用类名小写作为索引名（Document → document）。生产环境建议显式使用 @IndexName("my_index") 指定索引名。

7.2 DocumentMapper 接口

位置：org.dromara.demo.esmapper.DocumentMapper

package org.dromara.demo.esmapper;

import org.dromara.demo.domain.Document;
import org.dromara.easyes.core.kernel.BaseEsMapper;

public interface DocumentMapper
        extends BaseEsMapper<Document> {
}

关键点：包名以 esmapper 结尾，匹配 @EsMapperScan("org.dromara.**.esmapper") 的扫描约定。接口继承 BaseEsMapper<Document> 即获得全部 CRUD 能力，不需要任何实现类——Easy-Es 在启动时用 JDK 动态代理生成实现。

7.3 EsCrudController

位置：org.dromara.demo.controller.EsCrudController

@ConditionalOnProperty(value = "easy-es.enable",
                       havingValue = "true")
@RequiredArgsConstructor
@RestController
@RequestMapping("/es")
public class EsCrudController {

    private final DocumentMapper documentMapper;

    /** 精确查询 */
    @GetMapping("/select")
    public Document select(String title) {
        LambdaEsQueryWrapper<Document> wrapper =
            new LambdaEsQueryWrapper<>();
        wrapper.eq(Document::getTitle, title);
        return documentMapper.selectOne(wrapper);
    }

    /** 模糊搜索 */
    @GetMapping("/search")
    public List<Document> search(String key) {
        LambdaEsQueryWrapper<Document> wrapper =
            new LambdaEsQueryWrapper<>();
        wrapper.like(Document::getTitle, key);
        return documentMapper.selectList(wrapper);
    }

    /** 插入文档 */
    @PostMapping("/insert")
    public Integer insert(@RequestBody Document doc) {
        return documentMapper.insert(doc);
    }

    /** 按 id 更新 */
    @PutMapping("/update")
    public Integer update(@RequestBody Document doc) {
        return documentMapper.updateById(doc);
    }

    /** 按 id 删除 */
    @DeleteMapping("/delete/{id}")
    public Integer delete(@PathVariable String id) {
        return documentMapper.deleteById(id);
    }
}

注意 EsCrudController 上也有 @ConditionalOnProperty——这是双重保险。即使 Easy-Es 开启了，如果不想暴露 ES 接口，也可以独立控制。DEMO 的 5 个 API 覆盖了完整的 CRUD：

APIHTTPEasy-Es 方法查询条件
/es/select?title=xxxGETselectOne(wrapper)wrapper.eq(Document::getTitle, title) 精确匹配
/es/search?key=xxxGETselectList(wrapper)wrapper.like(Document::getTitle, key) 模糊匹配
/es/insertPOSTinsert(document)无需条件
/es/updatePUTupdateById(document)按 id 更新
/es/delete/{id}DELETEdeleteById(id)按 id 删除

八、如何创建你自己的 ES 索引？

掌握了以上模式，为你的业务创建一个 ES 索引只需要三步：

1
定义实体 
// org.dromara.xxx.domain.Product.java
@Data
public class Product {
    private String id;
    private String name;
    private BigDecimal price;
    private String description;
}
2
编写 Mapper（必须放在 esmapper 包下） 
// org.dromara.xxx.esmapper.ProductMapper.java
package org.dromara.xxx.esmapper;  // ← 包名以 esmapper 结尾

import org.dromara.easyes.core.kernel.BaseEsMapper;
import org.dromara.xxx.domain.Product;

public interface ProductMapper
        extends BaseEsMapper<Product> {
}
3
注入使用 
// 在任意 Service 或 Controller 中
@RequiredArgsConstructor
@Service
public class ProductService {

    private final ProductMapper productMapper;

    public List<Product> search(String keyword) {
        LambdaEsQueryWrapper<Product> wrapper =
            new LambdaEsQueryWrapper<>();
        wrapper.like(Product::getName, keyword)
               .or()
               .like(Product::getDescription, keyword);
        return productMapper.selectList(wrapper);
    }
}
最容易踩的坑：Mapper 接口的包必须以 esmapper 结尾。如果放在 org.dromara.xxx.mapper 下，@EsMapperScan("org.dromara.**.esmapper") 扫描不到，注入时会报 NoSuchBeanDefinitionException。没有例外——这是扫描约定的硬性要求。

九、检验一下（提取练习）

Q1. EnvironmentPostProcessor 在 Spring 启动的哪个阶段执行？

A. Bean 实例化之后 B. ApplicationContext 刷新完成之后 C. Environment 准备之后、ApplicationContext 刷新之前（最早阶段） D. 和 AutoConfiguration 类的加载同时执行
看答案C。EnvironmentPostProcessor 在 Environment 准备之后、ApplicationContext 创建之前执行，是 Spring 启动流程中最早的开发者可干预点。它比 BeanFactoryPostProcessor 和 AutoConfiguration 都早。

Q2. 为什么 getOrder() 返回 Ordered.HIGHEST_PRECEDENCE（值为 Integer.MIN_VALUE）？

A. 随便写的，没什么特别原因 B. 确保在所有 EnvironmentPostProcessor 中第一个执行，避免其他处理器在配置同步之前就读到了旧值 C. 为了让它最后执行，覆盖其他处理器的设置 D. 为了和 Actuator 的加载顺序精确匹配
看答案B。Spring 排序规则：值越小越优先。Integer.MIN_VALUE 是最小的可能值，确保这个处理器在所有 Processor 中第一个执行。它必须在其他任何处理器读取 management.health.elasticsearch.enabled 之前完成配置同步。

Q3. ActuatorEnvironmentPostProcessor 同步的是哪两个配置项？

A. easy-es.address → spring.elasticsearch.uris B. easy-es.enable → management.health.elasticsearch.enabled C. easy-es.compatible → elasticsearch.version D. spring.data.elasticsearch.enabled → management.endpoint.health.enabled
看答案B。它读取 easy-es.enable 的值，同步写入 management.health.elasticsearch.enabled，使得 Easy-Es 的开关能联动控制 Actuator 的健康检查行为。

Q4. 关于 EnvironmentPostProcessor 和 AutoConfiguration 的执行顺序，哪个描述正确？

A. AutoConfiguration 先执行，EnvironmentPostProcessor 后执行 B. 两者同时执行，没有先后顺序 C. EnvironmentPostProcessor 先执行（准备环境），AutoConfiguration 后执行（准备容器） D. 取决于 @Order 注解的值
看答案C。EnvironmentPostProcessor 在 Environment 准备阶段执行（Spring 启动最早阶段），AutoConfiguration 类在 ApplicationContext 准备阶段执行（晚得多）。这是 Spring Boot 启动流程的固定时序，与 @Order 无关——它们的执行窗口本身就不重叠。

Q5. 模块同时有 spring.factories 和 AutoConfiguration.imports 两个注册文件，它们各自的职责是什么？

A. 两个文件功能完全一样，这是历史遗留的冗余 B. spring.factories 注册 EnvironmentPostProcessor（第 1 阶段），.imports 注册 AutoConfiguration 类（第 2 阶段），职责不同、时序不同 C. spring.factories 是旧机制，应该迁移到 .imports D. .imports 是给外部模块用的，spring.factories 是模块内部用的
看答案B。两者服务于完全不同的 SPI 通道：spring.factories 中 EnvironmentPostProcessor key 用于环境后处理器注册（最早阶段），AutoConfiguration.imports 用于自动配置类注册（容器准备阶段）。它们不可互相替代。

Q6. （实战题）给 DEMO 增加一个按内容搜索的接口，写出代码骨架。

看参考实现 
/**
 * 按内容搜索
 * @param keyword 内容关键词
 */
@GetMapping("/searchContent")
public List<Document> searchContent(String keyword) {
    LambdaEsQueryWrapper<Document> wrapper =
        new LambdaEsQueryWrapper<>();
    wrapper.like(Document::getContent, keyword);
    return documentMapper.selectList(wrapper);
}

// 如果想同时搜索标题和内容：
@GetMapping("/searchAll")
public List<Document> searchAll(String keyword) {
    LambdaEsQueryWrapper<Document> wrapper =
        new LambdaEsQueryWrapper<>();
    wrapper.and(w -> w
        .like(Document::getTitle, keyword)
        .or()
        .like(Document::getContent, keyword));
    return documentMapper.selectList(wrapper);
}

关键是复用已有的 documentMapper，用 LambdaEsQueryWrapper 构建不同的查询条件即可。不需要改任何配置或注册代码。
