第 1 课 · SpringDoc 定制与自动配置 装配+配置

🎯 学习目标：学完本课，你能逐行解释 SpringDocConfig 上的 4 个注解、5 个 @Bean 方法各自做了什么、SpringDocProperties 如何映射 yml 配置、PlusPaths 标记类如何防止路径重复，以及 Spring Boot 3.x 的 AutoConfiguration.imports 机制是如何让模块"零配置接入"的。

一、模块总览：8 个文件，一个目标

ruoyi-common-doc 模块一共 8 个 Java 类，它不是一个独立的文档生成框架，而是建立在 springdoc-openapi 之上的项目定制增强层。扫描 Controller、生成 OpenAPI JSON 的脏活累活全在 SpringDoc 里，本模块只做三件事：中文 JavaDoc 注入、权限注解说明、上下文路径修正。

二、SpringDoc 与 Swagger：一个关键区别

很多教程还在讲 Swagger，但 RuoYi-Vue-Plus 用的是 SpringDoc。它们的关系是：

Swagger (旧)SpringDoc (新)
规范版本OpenAPI 2.0 (Swagger 2.0)OpenAPI 3.0
Spring 集成springfox (已停止维护)springdoc-openapi (活跃维护)
Spring Boot 3.x不兼容 (javax → jakarta)原生兼容
配置前缀swagger.*springdoc.*
文档端点/v2/api-docs/v3/api-docs
💡 一句话总结：SpringDoc 是 OpenAPI 3.0 规范的 Spring Boot 实现，Swagger 是旧版 2.0 规范。springfox（Swagger 的 Spring 集成库）已停更，Spring Boot 3.x 项目必须用 SpringDoc。

三、SpringDocConfig：注解四件套

类声明上的 4 个注解决定了这个配置类"什么时候加载""在谁前面加载""读什么配置""什么条件下生效"：

// 文件：org.dromara.common.doc.config.SpringDocConfig.java 第 35-39 行
@RequiredArgsConstructor
@AutoConfiguration(before = SpringDocConfiguration.class)
@EnableConfigurationProperties(SpringDocProperties.class)
@ConditionalOnProperty(name = "springdoc.api-docs.enabled", havingValue = "true", matchIfMissing = true)
public class SpringDocConfig {

注解 1：@RequiredArgsConstructor

Lombok 注解。为 final 字段（只有 ServerProperties serverProperties）生成构造器，实现构造器注入。Spring 自动从容器中注入 ServerProperties Bean。

注解 2：@AutoConfiguration(before = SpringDocConfiguration.class) —— 时序声明

这是本模块最关键的注解。意思是："我在 SpringDocConfiguration（SpringDoc 自己的配置类）之前加载"。

为什么必须写在 SpringDoc 之前？ 因为 openApi() 方法上有 @ConditionalOnMissingBean(OpenAPI.class)——如果 SpringDoc 的默认配置先加载并注册了 OpenAPI Bean，本模块的 openApi() 就会因为"已经有了"而被跳过，你的文档 info 就全是默认空值。

💡 设计洞察：@AutoConfiguration(before = ...) 是 Spring Boot 3.x 引入的新注解，专门用于控制自动配置类之间的加载顺序。它比老式的 @AutoConfigureBefore 更语义化——"我是一个自动配置类，我需要在某个标准配置之前加载"。

注解 3：@EnableConfigurationProperties(SpringDocProperties.class)

激活 SpringDocProperties 的属性绑定。Spring Boot 会从 application.yml 中读取 springdoc.* 前缀的配置，映射到 SpringDocProperties 的字段上。

注解 4：@ConditionalOnProperty(name = "springdoc.api-docs.enabled", ...)

条件开关。只有在 yml 中 springdoc.api-docs.enabled=true 时（未配置则默认 true），整个配置类才生效。这是 SpringDoc 的标准开关——设为 false 就能一键关闭文档。

四、5 个 @Bean 逐个拆解

Bean 1: openApi() —— 构建 OpenAPI 根对象

// SpringDocConfig.java 第 49-71 行
@Bean
@ConditionalOnMissingBean(OpenAPI.class)
public OpenAPI openApi(SpringDocProperties properties) {
    OpenAPI openApi = new OpenAPI();
    // 文档基本信息：标题、描述、联系人、许可证、版本
    SpringDocProperties.InfoProperties infoProperties = properties.getInfo();
    Info info = convertInfo(infoProperties);
    openApi.info(info);
    // 扩展文档信息
    openApi.externalDocs(properties.getExternalDocs());
    openApi.tags(properties.getTags());
    openApi.paths(properties.getPaths());
    // 安全方案：遍历 securitySchemes 的 key，创建 SecurityRequirement
    if (properties.getComponents() != null) {
        openApi.components(properties.getComponents());
        Set<String> keySet = properties.getComponents().getSecuritySchemes().keySet();
        List<SecurityRequirement> list = new ArrayList<>();
        SecurityRequirement securityRequirement = new SecurityRequirement();
        keySet.forEach(securityRequirement::addList);
        list.add(securityRequirement);
        openApi.security(list);
    }
    return openApi;
}

这个方法的核心逻辑：把 SpringDocProperties 中 yml 映射来的配置值，逐个搬到 OpenAPI 模型对象上。其中 security 部分是重点——它遍历 components.securitySchemes 的所有 key（如 Bearer Token），创建对应的 SecurityRequirement，这样 Swagger UI 上才会出现"Authorize"按钮。

Bean 2: classTagOperationCustomizer() —— 类级 Tag 增强

// SpringDocConfig.java 第 92-96 行
@Bean
public ClassTagOperationCustomizer classTagOperationCustomizer(
        Optional<JavadocProvider> javadocProvider,
        PropertyResolverUtils propertyResolverUtils) {
    return new ClassTagOperationCustomizer(javadocProvider, propertyResolverUtils);
}

这个定制器同时实现了 GlobalOperationCustomizer 和 GlobalOpenApiCustomizer 两个接口。它的职责是把 Controller 分组名从 SpringDoc 默认的驼峰拆分（"Sys User Controller"）改为类 JavaDoc 第一行（"系统用户管理"）。

Bean 3: javadocOperationCustomizer() —— 方法级 JavaDoc 增强

// SpringDocConfig.java 第 101-105 行
@Bean
public JavadocOperationCustomizer javadocOperationCustomizer(
        Optional<JavadocProvider> javadocProvider,
        List<JavadocResolver> javadocResolvers) {
    return new JavadocOperationCustomizer(javadocProvider, javadocResolvers);
}

关键设计：List<JavadocResolver> 参数——Spring 自动注入容器中所有 JavadocResolver 类型的 Bean。目前只有 SaTokenAnnotationMetadataJavadocResolver 一个，但你可以写自己的 Resolver 实现类并注册 @Bean，就能无侵入地扩展文档描述。

Bean 4: openApiCustomizer() —— 上下文路径修正 + 防重复

// SpringDocConfig.java 第 110-129 行
@Bean
public GlobalOpenApiCustomizer openApiCustomizer() {
    String contextPath = serverProperties.getServlet().getContextPath();
    String finalContextPath;
    if (StringUtils.isBlank(contextPath) || "/".equals(contextPath)) {
        finalContextPath = "";
    } else {
        finalContextPath = contextPath;
    }
    return openApi -> {
        Paths oldPaths = openApi.getPaths();
        if (oldPaths instanceof PlusPaths) {
            return;  // 已处理过，跳过
        }
        PlusPaths newPaths = new PlusPaths();
        oldPaths.forEach((k, v) -> newPaths.addPathItem(finalContextPath + k, v));
        openApi.setPaths(newPaths);
    };
}

如果你的应用设置了 server.servlet.context-path=/api，SpringDoc 生成的路径是 /system/user，但实际请求要走 /api/system/user。这个定制器就是给所有路径加前缀。

Bean 5: saTokenAnnotationJavadocResolver() —— 权限解析器

// SpringDocConfig.java 第 134-137 行
@Bean
public JavadocResolver saTokenAnnotationJavadocResolver() {
    return new SaTokenAnnotationMetadataJavadocResolver();
}

注册 SaToken 权限注解解析器。返回类型是 JavadocResolver 接口，Spring 自动把它注入到 Bean 3 的 List<JavadocResolver> 中。

五、PlusPaths：一个 5 行的空类，解决了路径重复问题

// SpringDocConfig.java 第 144-152 行
static class PlusPaths extends Paths {
    public PlusPaths() {
        super();
    }
}

这个类没有额外字段，没有重写方法。它就是一个标记类（Marker Class）。问题的核心：GlobalOpenApiCustomizer 可能被 SpringDoc 回调多次，每次回调都会给路径再加一层前缀，造成 /api/api/system/user。

第 1 次回调: oldPaths 是普通 Paths → 不是 PlusPaths 实例 → 创建 PlusPaths → 路径加 /api 前缀 → setPaths(newPaths) 第 2 次回调: oldPaths 是上次的 PlusPaths → instanceof 命中 → return（跳过，路径已修正）
💡 设计洞察：用一个空子类作为标记来实现幂等保护，比用外部 boolean 标志更可靠——因为标记信息"附着"在对象本身上（通过类型体系），不会因为 lambda 被重新创建而丢失状态。这是生产级代码中处理"回调可能多次触发"的经典模式。

六、SpringDocProperties：yml 到 Java 的类型映射

// 文件：org.dromara.common.doc.config.properties.SpringDocProperties.java 第 20-22 行
@Data
@ConfigurationProperties(prefix = "springdoc")
public class SpringDocProperties {

@ConfigurationProperties(prefix = "springdoc") 告诉 Spring Boot：把 yml 中 springdoc 前缀下的所有配置，按字段名映射。它直接复用 SpringDoc 的标准配置位置，不需要额外的配置前缀。

它的 5 个字段直接对应 OpenAPI 对象的顶层属性：

字段类型yml 示例映射到
infoInfoPropertiesspringdoc.info.title=XX系统openApi.info
externalDocsExternalDocumentationspringdoc.external-docs.url=...openApi.externalDocs
tagsList<Tag>springdoc.tags[0].name=...openApi.tags
pathsPathsspringdoc.paths.*openApi.paths
componentsComponentsspringdoc.components.security-schemes.*openApi.components

为什么要有 InfoProperties 内部类？

原代码注释（第 60 行）写着："为了 springboot 自动生产配置提示信息，所以这里复制一个类出来"。原生的 io.swagger.v3.oas.models.info.Info 字段上没有 Spring Boot 的元数据注解。自定义 InfoProperties 的每个字段都写了 Javadoc（标题、描述、联系人等），Spring Boot 的配置处理器会扫描这些 Javadoc 生成 spring-configuration-metadata.json，IDE 就能在 yml 中给出智能提示。

// SpringDocProperties.java 第 62-92 行
@Data
public static class InfoProperties {
    /** 标题 */
    private String title = null;
    /** 描述 */
    private String description = null;
    /** 联系人信息 */
    @NestedConfigurationProperty
    private Contact contact = null;
    /** 许可证 */
    @NestedConfigurationProperty
    private License license = null;
    /** 版本 */
    private String version = null;
}

注意 Contact 和 License 直接用了 io.swagger.v3.oas.models.info 包里的类——结构简单且稳定，不需要额外的 IDE 提示。

七、AutoConfiguration.imports：一行的力量

在 resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件中只有一行：

org.dromara.common.doc.config.SpringDocConfig
💡 关键认知：这是 Spring Boot 3.x 的自动配置发现机制，替代了旧版（2.x）的 spring.factories 文件。格式更简单——每行一个自动配置类全限定名。Spring Boot 启动时会扫描所有 jar 中这个固定路径的文件，把里面列出的类名当作自动配置类加载。

对比新旧机制：

Spring Boot 2.xSpring Boot 3.x
文件名META-INF/spring.factoriesMETA-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
格式键值对（key=value1,value2）纯文本，一行一个类名
注解@Configuration@AutoConfiguration
排序注解@AutoConfigureBefore / @AutoConfigureAfter直接在 @AutoConfiguration 中用 before/after 属性

八、检验一下（提取练习）

Q1. @AutoConfiguration(before = SpringDocConfiguration.class) 中的 before 是什么意思？为什么不能写成 after？

A. before 表示"在 SpringDocConfiguration 之前加载"；因为本模块的 openApi() 用了 @ConditionalOnMissingBean，必须先加载来"抢占" OpenAPI Bean，否则 SpringDoc 的默认 Bean 先生效 B. before 是性能优化，写在前面启动更快 C. 写成 after 也没关系，@ConditionalOnMissingBean 不受加载顺序影响 D. before 表示本模块比 SpringDoc 更早编译
看答案A。@ConditionalOnMissingBean 的判断依据是"容器中是否已存在同类型 Bean"。如果 SpringDoc 默认配置先加载，容器里就有了 OpenAPI Bean，本模块的自定义 Bean 就会被跳过，文档 info 变成默认空值。

Q2. PlusPaths 内部类的作用是什么？为什么不能用一个 boolean 变量代替？

A. PlusPaths 给 Paths 添加了新方法，boolean 做不到 B. PlusPaths 是一个标记类（Marker Class），通过 instanceof 判断防止 openApiCustomizer 被多次回调导致路径重复加前缀；boolean 变量在 lambda 外部定义可以工作，但标记类把幂等信息附着在对象上，更可靠且不依赖外部状态 C. PlusPaths 是为了提升路径遍历性能 D. boolean 变量在 multi-threading 下不安全
看答案B。两种方案都能解决问题，但标记类方案更优雅：幂等性保护信息通过类型体系附着在 Paths 对象上，不依赖外部可变状态。每次 openApiCustomizer 回调时拿到的是当前的 Paths 对象，instanceof 判断天然正确。

Q3. Spring Boot 3.x 的自动配置发现机制与 2.x 有什么不同？

A. 完全一样，只是文件名变了 B. 2.x 用 META-INF/spring.factories（键值对格式），3.x 用 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports（一行一个类名）；3.x 的 @AutoConfiguration 注解自带 before/after 排序，不再需要单独的排序注解 C. 3.x 不再需要自动配置，全用手动 @Import D. 2.x 用 XML 配置，3.x 用注解配置
看答案B。3.x 的新机制（spring boot 2.7+ 引入）更简洁：.imports 文件每行一个类名，@AutoConfiguration 自带 before/after 属性控制顺序。不再需要 spring.factories 的键值对格式和 @AutoConfigureBefore/@AutoConfigureAfter 分离注解。

Q4. 5 个 @Bean 中，哪个使用了 @ConditionalOnMissingBean？为什么只有它需要这个条件注解？

A. classTagOperationCustomizer()，因为 ClassTag 是可选功能 B. openApi()，因为 SpringDoc 自己也会注册一个默认的 OpenAPI Bean；用 @ConditionalOnMissingBean 确保"如果用户没自定义则用默认，如果本模块定义了则覆盖默认" C. javadocOperationCustomizer()，因为 JavaDoc 可能不可用 D. saTokenAnnotationJavadocResolver()，因为 SaToken 可能没引入
看答案B。SpringDoc 的 SpringDocConfiguration 里默认注册了一个 OpenAPI Bean。本模块用 @ConditionalOnMissingBean + @AutoConfiguration(before = ...) 组合拳，在默认 Bean 注册之前"抢占"位置，达到覆盖效果。其他 4 个 Bean 是纯新增，不需要条件注解。
📖 推荐一手资料：Spring Boot 自动配置官方文档 — 读 @AutoConfiguration、@ConditionalOnMissingBean、AutoConfiguration.imports 三节。本模块的装配逻辑完全是官方推荐写法。
