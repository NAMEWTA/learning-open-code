第 2 课 · JavaDoc 解析与权限文档生成 定制+扩展

🎯 学习目标：学完本课，你能讲清 JavadocProvider 如何从编译期 JSON 读取 JavaDoc、两个定制器（JavadocOperationCustomizer、ClassTagOperationCustomizer）的完整工作流程、JavadocResolver 扩展点接口如何实现"对扩展开放、对修改关闭"，以及 SaTokenAnnotationMetadataJavadocResolver 如何通过字符串类名 + ClassLoaderUtil 避免硬依赖 SaToken。

一、数据源头：JavadocProvider 如何读到编译期注释

Java 的注释在编译后会被丢弃（不会进入 .class 文件）。那 SpringDoc 是怎么读到 JavaDoc 的呢？答案在 therapi-runtime-javadoc 这个库上。

它的原理很简单：在编译期通过 annotation processor 扫描所有 JavaDoc，序列化为 JSON 嵌入到 jar 包里。运行时通过 JavadocProvider 读取这些 JSON，就能拿到原始的 JavaDoc 文本。

方法输入输出用途
getClassJavadoc(Class)类对象完整类 JavaDoc 文本ClassTagOperationCustomizer 用来取 Controller 的中文描述
getMethodJavadocDescription(Method)方法对象完整方法 JavaDoc 文本JavadocOperationCustomizer 用来设置 API 描述
getFirstSentence(String)完整 JavaDoc 文本第一句话（到第一个句号为止）JavadocOperationCustomizer 用来设置 API 摘要（summary）
💡 关键点：JavadocProvider 是被 Optional 包装的——如果 classpath 上没有 therapi-runtime-javadoc，它就是 Optional.empty()。两个定制器都用 javadocProvider.ifPresent() 安全处理，没有 JavaDoc 时优雅降级而不报错。

二、JavadocOperationCustomizer：方法级描述增强

文件：org.dromara.common.doc.core.customizer.JavadocOperationCustomizer，实现 GlobalOperationCustomizer 接口。SpringDoc 每生成一个 API 操作（Operation），就回调一次它的 customize() 方法。

它持有两个 final 字段（构造器注入）：

  • Optional<JavadocProvider> javadocProvider —— JavaDoc 读取器
  • List<JavadocResolver> javadocResolvers —— 所有扩展解析器，Spring 自动注入

customize() 方法分两阶段

// JavadocOperationCustomizer.java 第 41-69 行
@Override
public Operation customize(Operation operation, HandlerMethod handlerMethod) {
    // ===== 阶段一：取方法 JavaDoc 作为基础描述 =====
    javadocProvider.ifPresent(provider -> {
        String description = provider.getMethodJavadocDescription(handlerMethod.getMethod());
        if (StringUtils.isNotBlank(description)) {
            // summary = 首句，description = 全文
            operation.setSummary(provider.getFirstSentence(description));
            operation.setDescription(description);
        }
    });

    // ===== 阶段二：遍历 Resolver 列表追加扩展描述 =====
    if (CollectionUtils.isEmpty(javadocResolvers)) {
        return operation;
    }
    StringBuilder description = new StringBuilder(
        Optional.ofNullable(operation.getDescription()).orElse(""));
    List<String> resolvedDescriptions = javadocResolvers.stream()
        .sorted()                                  // 按 getOrder() 排序
        .filter(resolver -> resolver.supports(handlerMethod))  // 只调支持的
        .map(resolver -> resolver.resolve(handlerMethod, operation)) // 执行解析
        .filter(StringUtils::isNotBlank)           // 过滤空结果
        .toList();
    if (!resolvedDescriptions.isEmpty()) {
        resolvedDescriptions.forEach(description::append);
        operation.setDescription(description.toString());
    }
    return operation;
}

以 SysUserController.list() 为例，它的 JavaDoc 是"查询系统用户列表。支持按用户名、手机号、状态筛选，分页返回。"：

阶段一：JavadocProvider 可用 getMethodJavadocDescription(method) → "查询系统用户列表。支持按用户名、手机号、状态筛选，分页返回。" getFirstSentence() → "查询系统用户列表。" operation.summary = "查询系统用户列表。" operation.description = "查询系统用户列表。支持按用户名..." 阶段二：javadocResolvers = [SaTokenResolver] .sorted() → SaTokenResolver 排在第一位 .filter(supports) → 方法上有 @SaCheckPermission → true .map(resolve) → 反射读注解 → SaTokenSecurityMetadata → toMarkdownString() → "<br><h3>访问权限</h3><br>**权限校验：**<br>- `system:user:list`<br>" description.append → 最终 description = 基础JavaDoc + 权限说明

三、ClassTagOperationCustomizer：双接口的巧妙配合

文件：org.dromara.common.doc.core.customizer.ClassTagOperationCustomizer，同时实现了两个接口：

接口方法签名触发时机职责
GlobalOperationCustomizercustomize(Operation, HandlerMethod)每个 Controller 方法生成 Operation 时替换单个 API 的 Tag 名称
GlobalOpenApiCustomizercustomise(OpenAPI)所有 Operation 处理完后清理旧的 Tag、补充新的中文 Tag

切入点 1：Operation 级 Tag 替换

SpringDoc 默认把 Controller 类名驼峰拆分作为 Tag。比如 SysUserController → "Sys User Controller"。定制器用类的 JavaDoc 第一行替代它：

customize(operation, handlerMethod) │ ├─ ① 检查 Controller 类上有没有 @Tag / @Tags 注解 │ 有 → 使用注解值（尊重用户显式声明），return │ ├─ ② 没注解 → getClassJavadocTagName(beanType) │ javadocProvider.getClassJavadoc(beanType) 取类 JavaDoc │ IoUtil.readLines → filter notBlank → findFirst → 第一行就是 tag 名 │ 例如："系统用户管理 Controller" → tagName = "系统用户管理 Controller" │ ├─ ③ 计算默认 tag 名 = splitCamelCase("SysUserController") = "Sys User Controller" │ ├─ ④ 判断是否应该替换（operation 没有 tags 或包含默认 tag） │ ├─ ⑤ 执行替换：remove "Sys User Controller" → add "系统用户管理 Controller" │ 记录到 replacedAutoTagNames 和 tags 缓存 │ └─ return operation

切入点 2：全局 Tag 列表清理

所有 Operation 处理完后，customise(OpenAPI) 被回调：

// ClassTagOperationCustomizer.java 第 95-106 行
@Override
public void customise(OpenAPI openApi) {
    // ① 移除已被 JavaDoc tag 替换的默认类名 tag
    if (!CollectionUtils.isEmpty(openApi.getTags()) && !CollectionUtils.isEmpty(replacedAutoTagNames)) {
        openApi.getTags().removeIf(tag -> replacedAutoTagNames.contains(tag.getName()));
    }
    // ② 将类 JavaDoc 描述补充到 OpenAPI 顶层 tags
    tags.values().forEach(tag -> {
        if (openApi.getTags() == null || openApi.getTags().stream()
                .noneMatch(item -> Objects.equals(item.getName(), tag.getName()))) {
            openApi.addTagsItem(tag);
        }
    });
}

两个 ConcurrentHashMap（tags 和 replacedAutoTagNames）保证了在多线程处理 Controller 时的并发安全。

四、JavadocResolver 扩展点：对扩展开放，对修改关闭

这是整个模块架构上最值得学习的设计。先看接口定义：

// 文件：org.dromara.common.doc.core.resolver.JavadocResolver.java 第 14 行
public interface JavadocResolver extends Comparable<JavadocResolver>, Ordered {
    boolean supports(HandlerMethod handlerMethod);
    String resolve(HandlerMethod handlerMethod, Operation operation);
    default int getOrder() { return Ordered.LOWEST_PRECEDENCE; }
    default String getName() { return this.getClass().getSimpleName(); }
    default int compareTo(JavadocResolver o) { return Integer.compare(getOrder(), o.getOrder()); }
}

接口同时继承了 Comparable（可排序）和 Ordered（Spring 标准的优先级接口）。supports() + resolve() 是典型的策略模式（Strategy Pattern）：先问"你能处理这个吗"，再请"那你来处理"。每个 Resolver 只关心自己能处理的情况，互不干扰。

AbstractMetadataJavadocResolver：模板方法模式

文件：org.dromara.common.doc.core.resolver.AbstractMetadataJavadocResolver。通过泛型 <M> 管理元数据对象，提供丰富的注解检测工具方法：

// 关键结构
public abstract class AbstractMetadataJavadocResolver<M> implements JavadocResolver {
    private final Supplier<M> metadataProvider;  // 元数据工厂
    private final int order;

    // 模板方法：先创建元数据，再委托子类
    public String resolve(HandlerMethod hm, Operation op) {
        return resolve(hm, op, metadataProvider.get());
    }

    // 子类必须实现的抽象方法
    public abstract String resolve(HandlerMethod hm, Operation op, M metadata);

    // ===== 工具方法（子类直接复用，不用写反射代码）=====
    public boolean hasClassAnnotation(HandlerMethod hm, Class<?> annClass) {...}
    public boolean hasClassAnnotation(HandlerMethod hm, String annTypeName) {...}
    public boolean hasMethodAnnotation(HandlerMethod hm, Class<?> annClass) {...}
    public boolean hasMethodAnnotation(HandlerMethod hm, String annTypeName) {...}
    public boolean hasAnnotation(HandlerMethod hm, Class<?> annClass) {...}
    public boolean hasAnnotation(HandlerMethod hm, String annTypeName) {...}
    public Map getClassAnnotationValueMap(HandlerMethod hm, ...) {...}
    public Map getMethodAnnotationValueMap(HandlerMethod hm, ...) {...}
}
💡 设计洞察——模板方法模式：抽象类定义了算法的骨架（先创建元数据 → 委托解析），具体的解析逻辑由子类实现。策略模式体现在 supports() + resolve() 组合——不同的 Resolver 处理不同的注解类型，通过 supports() 决定谁参与本次解析。扩展点设计体现在 List<JavadocResolver>——Spring 自动注入所有实现类，JavadocOperationCustomizer 遍历列表依次调用，新增 Resolver 无需修改任何现有代码。

五、SaTokenAnnotationMetadataJavadocResolver：无硬依赖的注解解析

文件：org.dromara.common.doc.core.resolver.SaTokenAnnotationMetadataJavadocResolver。它继承了 AbstractMetadataJavadocResolver<SaTokenSecurityMetadata>。

设计亮点：用字符串类名避免硬依赖

这个模块不直接 import SaToken 的注解类（如 cn.dev33.satoken.annotation.SaCheckPermission），而是通过字符串类名 + ClassLoaderUtil.loadClass() 动态加载：

// SaTokenAnnotationMetadataJavadocResolver.java 第 33-73 行
private static final String BASE_CLASS_NAME = "cn.dev33.satoken.annotation";
private static final String SA_CHECK_ROLE_CLASS_NAME = BASE_CLASS_NAME + ".SaCheckRole";
private static final String SA_CHECK_PERMISSION_CLASS_NAME = BASE_CLASS_NAME + ".SaCheckPermission";
private static final String SA_IGNORE_CLASS_NAME = BASE_CLASS_NAME + ".SaIgnore";
private static final String SA_CHECK_LOGIN_NAME = BASE_CLASS_NAME + ".SaCheckLogin";

private static final Class<? extends Annotation> SA_CHECK_ROLE_CLASS;
private static final Class<? extends Annotation> SA_CHECK_PERMISSION_CLASS;
// ... etc

static {
    SA_CHECK_ROLE_CLASS = (Class<? extends Annotation>) ClassLoaderUtil.loadClass(SA_CHECK_ROLE_CLASS_NAME, false);
    SA_CHECK_PERMISSION_CLASS = (Class<? extends Annotation>) ClassLoaderUtil.loadClass(SA_CHECK_PERMISSION_CLASS_NAME, false);
    // ... etc
}
💡 为什么这样设计？ 如果直接 import cn.dev33.satoken.annotation.SaCheckPermission，那么所有使用 ruoyi-common-doc 的模块都必须引入 SaToken 依赖。通过字符串类名动态加载，即使 classpath 上没有 SaToken，也只是这些静态字段为 null，supports() 返回 false，不会报错。这就是可选依赖（Optional Dependency）的处理模式。

解析流程：两路并行

supports() 方法检查方法或类上是否有 @SaCheckRole、@SaCheckPermission 或 @SaIgnore 注解。

resolve() 方法分两步：

resolve(handlerMethod, operation, metadata) │ ├─ ① 检查 @SaIgnore │ hasAnnotation(handlerMethod, SA_IGNORE_CLASS_NAME) │ → true → metadata.setIgnore(true) → toMarkdownString() │ → "<br><h3>访问权限</h3><br>> **权限策略**：忽略权限检查<br>" │ ├─ ② resolvePermissionCheck(handlerMethod, metadata) │ 分别检查方法和类上的 @SaCheckPermission │ 反射读取注解属性：value、mode、type、orRole │ → metadata.addPermission(values, mode, type, orRoles) │ └─ ③ resolveRoleCheck(handlerMethod, metadata) 分别检查方法和类上的 @SaCheckRole 反射读取注解属性：value、mode、type → metadata.addRole(values, mode, type) 最终 toMarkdownString() 输出 Markdown 格式权限说明

六、SaTokenSecurityMetadata：权限元数据模型

文件：org.dromara.common.doc.core.model.SaTokenSecurityMetadata。它是权限解析的"中间数据容器"，存储：

字段类型对应注解
permissionsList<AuthInfo>@SaCheckPermission 解析结果
rolesList<AuthInfo>@SaCheckRole 解析结果
ignoreboolean@SaIgnore 标记

其中 AuthInfo 内部类存储单条权限/角色的详细信息：values（权限值数组）、mode（AND/OR）、type（类型）、orValues（或角色）。getModeSymbol() 方法把 "AND" 转为 " & "、"OR" 转为 " | "，用于 Markdown 展示。

toMarkdownString() 方法根据 ignore、permissions、roles 三个字段的状态，生成不同格式的 Markdown 输出：

三种输出形态： ① @SaIgnore 标记： "<br><h3>访问权限</h3><br>> **权限策略**：忽略权限检查<br>" ② 需要登录但无具体权限角色： "<br><h3>访问权限</h3><br>> **权限策略**：需要登录<br><br>" ③ 有具体权限/角色： "<br><h3>访问权限</h3><br> **权限校验：**<br><br> - `system:user:list` & `system:user:query`<br> - 或角色：`admin`<br> <br> **角色校验：**<br><br> - `admin`<br>"
💡 设计洞察：Supplier<M> metadataProvider 的设计确保了每次 resolve() 调用都创建全新的元数据对象（SaToken 实现中使用 SaTokenSecurityMetadata::new），避免不同 Controller 方法之间的状态污染。这是函数式编程中"不可变数据"思想的体现。

七、如何新增一个自定义 Resolver？只需三步

假设你想为 @MyRateLimit 自定义注解在文档中生成限流说明：

第 1 步：写一个类继承 AbstractMetadataJavadocResolver<YourMetadata> - supports(): 检查方法上有没有 @MyRateLimit - resolve(): 反射读注解属性 → 拼 Markdown → 返回 第 2 步：注册 @Bean @Bean public JavadocResolver myRateLimitResolver() { return new MyRateLimitJavadocResolver(); } 第 3 步（可选）：设置 order 控制执行顺序 return new MyRateLimitJavadocResolver(order=100);
不需要修改 JavadocOperationCustomizer，不需要修改 SpringDocConfig（除了加 @Bean），不需要改任何现有代码。这就是开闭原则的完整实践。

八、检验一下（提取练习）

Q1. JavadocOperationCustomizer 的两个阶段分别做什么？它们的输入来源有何不同？

A. 阶段一取 JavadocProvider 的方法 JavaDoc 设置 summary 和 description；阶段二遍历 JavadocResolver 列表追加扩展描述（如权限说明） B. 阶段一处理 GET 请求，阶段二处理 POST 请求 C. 阶段一在启动时执行，阶段二在第一次请求时执行 D. 阶段一用反射读取注解，阶段二用 JavaDoc 读取注释
看答案A。阶段一依赖 therapi-runtime-javadoc 编译期生成的 JSON，通过 JavadocProvider 读取；阶段二通过反射读取运行时的注解信息。两个阶段的输入来源完全不同：一个是编译期产物，一个是运行时的 Class 对象。

Q2. ClassTagOperationCustomizer 为什么需要同时实现 GlobalOperationCustomizer 和 GlobalOpenApiCustomizer？

A. 作者想少写一个类，把两个功能塞一起 B. Tag 替换需要两个时机：Operation 生成时替换单个 API 的 tag（GlobalOperationCustomizer）；所有 Operation 处理完后清理全局 tag 列表（GlobalOpenApiCustomizer），否则 Swagger UI 会出现空分组 C. SpringDoc 强制要求同时实现两个接口 D. 两个接口分别处理不同的注解类型
看答案B。在 customize() 中替换单个 API 的 tag 时，旧的 "Sys User Controller" 仍在 OpenAPI 的全局 tags 列表中。需要在 customise() 中用 removeIf 清理，再用 tags 缓存补充新的中文 Tag。不清理的话 Swagger UI 会同时显示旧英文 tag 和新中文 tag。

Q3. SaTokenAnnotationMetadataJavadocResolver 为什么用字符串类名 + ClassLoaderUtil.loadClass()，而不是直接 import SaToken 的注解类？

A. 字符串比 import 更简洁 B. 为了避免硬依赖 SaToken——如果直接 import，所有使用 ruoyi-common-doc 的模块都强制依赖 SaToken；用字符串动态加载，没有 SaToken 时优雅降级（supports 返回 false） C. loadClass 比 import 性能更好 D. 作者不知道 SaToken 的包名
看答案B。这是可选依赖（Optional Dependency）的标准处理模式。静态初始化块中 loadClass 的结果可能为 null，但 supports() 和 resolve() 方法中用 hasAnnotation(handlerMethod, STRING_NAME) 调用父类的字符串重载版本，它内部先做 null 检查再反射。没有 SaToken 时整个 Resolver 静默跳过。

Q4. 描述 JavadocResolver 扩展点中涉及的两种设计模式，并说明它们各自解决了什么问题。

A. 单例模式和工厂模式 B. 策略模式（Strategy）：supports() + resolve() 让不同 Resolver 处理不同类型的注解，互不干扰；模板方法模式（Template Method）：AbstractMetadataJavadocResolver 定义了解析骨架（创建元数据→委托子类），子类只需实现业务逻辑 C. 观察者模式和适配器模式 D. 装饰器模式和代理模式
看答案B。策略模式体现在 JavadocResolver 接口的 supports()+resolve() 组合——每个实现类是独立的策略，通过 supports() 判断是否适用。模板方法模式体现在 AbstractMetadataJavadocResolver——resolve(hm, op) 先调用 metadataProvider.get() 创建元数据，再委托给子类的 resolve(hm, op, metadata)，骨架固定，细节可变。两者叠加实现了"对扩展开放、对修改关闭"的开闭原则。
