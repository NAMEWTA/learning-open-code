第 2 课 · MqttAutoConfiguration 自动配置详解 核心类

🎯 本节目标（对应使命 Success #3 / #4）：学完你能解释 MqttAutoConfiguration 如何通过 @AutoConfiguration + @ConditionalOnProperty + AutoConfiguration.imports 实现零代码接入，以及 MqttClientCustomizer 中用虚拟线程替换默认线程池的意义。

上一课我们知道这个模块有三个类、一个 .imports 文件、一个 pom。这一课我们打开 模块的装配中枢——MqttAutoConfiguration，看看它是如何把 Mica MQTT 无缝接入 RuoYi 的。

一、先看文件全貌

把这个配置类拆开，它总共就做了三件事——注册两个监听器、提供一个定制器：

// ruoyi-common-mqtt/config/MqttAutoConfiguration.java（完整源码）
@AutoConfiguration                                          // ① 声明为自动配置类
@ConditionalOnProperty(value = "mqtt.client.enabled",        // ② 开关控制
                       havingValue = "true")
public class MqttAutoConfiguration {

    @Bean
    public MqttClientConnectListener mqttClientConnectListener(
            MqttClientCreator mqttClientCreator) {          // ③ 注册连接监听器
        return new MqttClientConnectListener(mqttClientCreator);
    }

    @Bean
    public MqttClientGlobalMessageListener mqttClientGlobalMessageListener() {
        return new MqttClientGlobalMessageListener();        // ④ 注册消息监听器
    }

    @Bean
    public MqttClientCustomizer mqttClientCustomizer() {     // ⑤ 注册虚拟线程定制器
        return creator -> {
            int corePoolSize = ThreadUtils.CORE_POOL_SIZE;

            ThreadFactory factory = new VirtualThreadTaskExecutor(
                "tio-worker-virtual").getVirtualThreadFactory();
            SynThreadPoolExecutor tioExecutor = new SynThreadPoolExecutor(
                corePoolSize, corePoolSize, 0L,
                new LinkedBlockingQueue<>(), factory, new TioCallerRunsPolicy());
            tioExecutor.prestartCoreThread();
            creator.tioExecutor(tioExecutor);           // 替换 t-io 工作线程池

            // ... 同样方式替换 groupExecutor 和 mqttExecutor
        };
    }
}

只有 3 个 @Bean 方法 + 2 个注解，没有一行业务逻辑。但每一步都有它的设计原因。

二、逐注解拆解

2.1 @AutoConfiguration — "我是自动配置类，请加载我"

Spring Boot 3.x 引入了新的 @AutoConfiguration 注解替代旧的 @Configuration 用于自动配置场景。但光有这个注解还不够——Spring Boot 怎么知道要加载它？

答案在 jar 包内的这个文件里：

// src/main/resources/META-INF/spring/
// org.springframework.boot.autoconfigure.AutoConfiguration.imports
org.dromara.common.mqtt.config.MqttAutoConfiguration
1固定的文件路径是「约定」
Spring Boot 3.x 约定
Spring Boot 启动时会按固定路径 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 扫描所有 jar，把文件中列出的类名作为自动配置类加载。这就是「依赖即生效」的底层原理——只要 pom.xml 里依赖了本模块，jar 包内的这个 .imports 文件就会被扫描到，配置类即被加载。不需要任何 @Import 或 @ComponentScan。
⚡ 对比旧方式：Spring Boot 2.x 使用 spring.factories 文件（META-INF/spring.factories），格式为 org.springframework.boot.autoconfigure.EnableAutoConfiguration=com.xxx.XxxConfig。3.x 改为了 .imports 文件，一行一个类名，更简洁。

2.2 @ConditionalOnProperty — "配置没开就别加载我"

这是 MQTT 模块的总开关：

@ConditionalOnProperty(value = "mqtt.client.enabled",
                       havingValue = "true")
// 默认值 matchIfMissing = false（非必须指定，因为默认就是 false）

结合 application.yml 中的配置：

mqtt.client:
  enabled: false    # 默认关闭！需要用时才改成 true

这条注解的效果链：

mqtt.client.enabled = false（或未配置） → @ConditionalOnProperty 条件不满足 → MqttAutoConfiguration 整个类不加载 → 3 个 @Bean 方法都不执行 → 两个监听器不注册、虚拟线程定制器不生效 → Mica MQTT Starter 自己的自动配置可能仍会尝试连接 （但因为没有连接参数或 enabled=false，通常也不会真正连接） mqtt.client.enabled = true → 条件满足 → 配置类加载 → 三个 Bean 注册 → MQTT 功能激活
🤔 想一想：如果 yml 里根本没写 mqtt.client.enabled，会导致什么？ @ConditionalOnProperty 默认 matchIfMissing = false，意味着如果 yml 里没有这个属性，条件不满足，配置类不加载。所以 MQTT 功能默认是关闭的，这是一个安全的设计——不会因为误引依赖就自动连接外部 Broker。

三、三个 Bean 方法逐一分析

3.1 mqttClientConnectListener — 注册连接监听器

@Bean
public MqttClientConnectListener mqttClientConnectListener(
        MqttClientCreator mqttClientCreator) {
    return new MqttClientConnectListener(mqttClientCreator);
}

关键点：

  • 参数注入：方法签名中声明了 MqttClientCreator mqttClientCreator，Spring 会自动从容器中找到这个 Bean 注入。这个 Bean 是由 Mica MQTT 自己的 Starter 创建的（通过它的自动配置），不是本模块创建的。
  • 传入构造器：把 mqttClientCreator 传给 MqttClientConnectListener 的构造函数。为什么？因为在 onDisconnect 回调中，可能需要动态修改 clientId/username/password（比如凭证轮换），这需要持有 Creator 的引用。
  • 无参数方法 vs 有参方法：对比下一个 Bean mqttClientGlobalMessageListener() 不需要任何参数——因为它只负责打日志，不修改连接参数。

3.2 mqttClientGlobalMessageListener — 注册消息监听器

@Bean
public MqttClientGlobalMessageListener mqttClientGlobalMessageListener() {
    return new MqttClientGlobalMessageListener();
}

最简单的 Bean 注册——什么都不依赖，直接 new 一个实例。它只做一件事：收到消息后打日志。但它的价值在于提供了一个扩展点：如果业务需要在收到消息后做更复杂的处理（存入数据库、触发工作流、转发到其它系统），只需替换这个 Bean，或在它的 onMessage 方法里添加逻辑。第 3 课会详细讲这一点。

3.3 mqttClientCustomizer — 虚拟线程定制器（核心亮点）

这是本模块最有设计含量的部分。先看它做了什么：

@Bean
public MqttClientCustomizer mqttClientCustomizer() {
    return creator -> {                                     // Lambda 实现 MqttClientCustomizer
        int corePoolSize = ThreadUtils.CORE_POOL_SIZE;       // 当前 CPU 核数

        // 1. 替换 t-io 工作线程（处理 TCP 读写）
        ThreadFactory factory = new VirtualThreadTaskExecutor(
            "tio-worker-virtual").getVirtualThreadFactory();
        SynThreadPoolExecutor tioExecutor = new SynThreadPoolExecutor(
            corePoolSize, corePoolSize, 0L,
            new LinkedBlockingQueue<>(), factory, new TioCallerRunsPolicy());
        creator.tioExecutor(tioExecutor);

        // 2. 替换 t-io 组线程（管理连接组）
        ThreadFactory factory1 = new VirtualThreadTaskExecutor(
            "tio-group-virtual").getVirtualThreadFactory();
        ThreadPoolExecutor groupExecutor = new ThreadPoolExecutor(
            corePoolSize, corePoolSize, 0L,
            TimeUnit.SECONDS, new LinkedBlockingQueue<>(),
            factory1, new TioCallerRunsPolicy());
        creator.groupExecutor(groupExecutor);

        // 3. 替换业务消息处理线程（处理 onMessage 回调）
        ThreadFactory factory2 = new VirtualThreadTaskExecutor(
            "biz-worker-virtual").getVirtualThreadFactory();
        ThreadPoolExecutor mqttExecutor = new ThreadPoolExecutor(
            corePoolSize, corePoolSize, 0L,
            TimeUnit.SECONDS, new LinkedBlockingQueue<>(),
            factory2, new TioCallerRunsPolicy());
        creator.mqttExecutor(mqttExecutor);
    };
}

这个定制器用一个 Lambda 实现了 Mica MQTT 的 MqttClientCustomizer 接口（函数式接口），在 Mica MQTT 创建客户端之前被回调，允许替换三个线程池。

为什么要替换线程池？

⚠️ 传统线程池的困境：操作系统线程（Platform Thread）创建成本高（约 1MB 栈空间 + 内核调度开销），一个 JVM 进程能创建的 OS 线程数有上限（通常几千）。在物联网场景中，如果设备数达到数万甚至数十万，每条 TCP 连接、每条消息处理都占用一个 OS 线程，系统很快就会触达瓶颈。

虚拟线程（Virtual Thread） 是 Java 21（Project Loom）引入的解决方案：

特性传统平台线程虚拟线程
创建开销~1MB 栈 + 内核调度~数百字节 + JVM 管理
数量上限数千数百万
阻塞时OS 线程被占用，无法复用JVM 自动挂起虚拟线程，释放底层平台线程
适用场景CPU 密集型IO 密集型（网络连接/消息处理）

MQTT 通信典型是 IO 密集型——大量时间在等待网络 IO，CPU 实际计算时间很短。虚拟线程正好：大量阻塞操作不消耗真正的 OS 线程资源。

三组线程池的分工

1tioExecutor（t-io 工作线程）
MqttAutoConfiguration.java:63-67
处理底层的 TCP 读写与报文编解码。对应 Mica MQTT 底层 t-io 框架的工作线程。使用 SynThreadPoolExecutor（t-io 定制的同步线程池），这是因为 t-io 内部有自己的任务编排逻辑。
2groupExecutor（t-io 组线程）
MqttAutoConfiguration.java:70-74
管理 MQTT 连接组（connection group），负责连接的创建、心跳维护、关闭等生命周期操作。
3mqttExecutor（业务消息处理线程）
MqttAutoConfiguration.java:76-80
执行业务层的消息处理逻辑——也就是 MqttClientGlobalMessageListener.onMessage() 的回调。如果消息量大、处理逻辑复杂，这个线程池的瓶颈最明显。用虚拟线程后，每条消息处理都可以搭一个轻量虚拟线程。

为什么 corePoolSize 不重要？

代码里有注释：// 这个数不重要 已经使用虚拟线程 就是填一下防止报错。因为 ThreadPoolExecutor 构造函数要求填写 corePoolSize，但虚拟线程的 ThreadFactory 每次 newThread() 都会创建新的虚拟线程（不受 corePoolSize 限制），所以填什么值都只是满足 API 约定。作者用了 ThreadUtils.CORE_POOL_SIZE（通常等于 CPU 核数）作为占位值。

💡 关键认知：MqttClientCustomizer 是 Mica MQTT 预留的扩展点。RuoYi 利用这个扩展点，不是改 Mica MQTT 源码，而是通过 Lambda 注入定制——这是「开闭原则」的体现：对扩展开放（允许定制线程池），对修改关闭（不改 Mica MQTT 代码）。

四、启动时的完整装配流程

应用启动 │ ├─ Spring Boot 扫描所有 jar 的 .imports 文件 │ → 发现 MqttAutoConfiguration │ ├─ 检查 @ConditionalOnProperty("mqtt.client.enabled") │ ├─ false → 跳过，MQTT 功能不激活 │ └─ true → 继续 ↓ │ ├─ Mica MQTT Starter 自己的自动配置先执行 │ → 读取 mqtt.client.* 配置 │ → 创建 MqttClientCreator Bean（包含 ip/port/clientId/...） │ ├─ MqttAutoConfiguration 执行 │ ├─ MqttClientCustomizer Bean 先执行 │ │ → 替换 tioExecutor / groupExecutor / mqttExecutor 为虚拟线程 │ │ │ ├─ MqttClientConnectListener Bean → 持有 MqttClientCreator │ └─ MqttClientGlobalMessageListener Bean │ └─ Mica MQTT 用定制后的 Creator 连接 Broker → onConnected() 回调 → 日志: "MqttConnectedEvent:..." → 进入消息监听循环

五、检验一下（提取练习）

Q1. MqttAutoConfiguration 是什么时候、被谁发现并加载的？

A. 被 @ComponentScan 扫描到 B. 被 @Import 显式引入 C. Spring Boot 启动时扫描所有 jar 内 META-INF/spring/...AutoConfiguration.imports 文件，发现其中列出的类名 D. 在 application.yml 中配置了 scan-path
看答案C。Spring Boot 3.x 使用固定路径的 .imports 文件来发现自动配置类，不需要任何显式的 @Import 或 @ComponentScan。这是「依赖即生效」的实现原理。

Q2. 为什么 MQTT 场景用虚拟线程比传统线程池好？

A. 虚拟线程更快（CPU 密集型计算更快） B. MQTT 通信是 IO 密集型，虚拟线程阻塞时自动释放底层平台线程，可支撑百万级连接而无需消耗等量 OS 线程 C. 虚拟线程更安全 D. 没有区别，只是作者的个人偏好
看答案B。虚拟线程的核心优势在于「阻塞时释放平台线程」——MQTT 场景大量时间在等待网络 IO，虚拟线程在阻塞时不占用 OS 线程资源，使得少量 OS 线程即可驱动海量虚拟线程。

Q3. 如果 yml 里 mqtt.client.enabled 根本没写，MQTT 功能会激活吗？

A. 会，因为没写就是 true B. 不会，因为 @ConditionalOnProperty 默认 matchIfMissing=false C. 会，因为 Mica MQTT 有自己的默认值 D. 不确定，取决于 Mica MQTT 的版本
看答案B。@ConditionalOnProperty 的 matchIfMissing 默认为 false，未配置的属性被视为「条件不满足」。MQTT 默认关闭，需要显式设置 mqtt.client.enabled: true 才会激活。

Q4. MqttClientConnectListener 的构造函数接收 MqttClientCreator，这个 Creator 是谁创建的？

A. ruoyi-common-mqtt 模块自己创建的 B. Mica MQTT 的 Starter 自动配置创建的，ruoyi 只是注入并使用它 C. 手动在 yml 里配置的 D. Spring Boot 内建的
看答案B。MqttClientCreator 由 Mica MQTT 的 mica-mqtt-client-spring-boot-starter 自动配置创建并注册到 Spring 容器。本模块通过方法参数注入拿到它，传给监听器。这体现了模块间的「依赖但不对抗」——增强 Mica MQTT 的行为而不取代它。
