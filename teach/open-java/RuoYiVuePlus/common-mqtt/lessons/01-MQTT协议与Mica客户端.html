01 - MQTT 协议与 Mica 客户端

🎯 本节目标：理解 MQTT 发布订阅模型的核心概念（Broker/Topic/QoS），能说清 ruoyi-common-mqtt 模块与 Mica MQTT 的关系，并能画出 MQTT 网络的拓扑结构图。

1. 全景鸟瞰

MQTT（Message Queuing Telemetry Transport）是 OASIS 标准的轻量级物联网消息协议，专为低带宽、不可靠网络设计。它不叫"消息队列"——核心是发布订阅模型，而非 RabbitMQ/Kafka 那样的队列。

一张图看清 MQTT 的角色关系：

┌──────────┐     publish "sensor/temp"     ┌────────────┐
│ 设备 A   │ ─────────────────────────────▶│            │
│Publisher │                               │   Broker   │
└──────────┘                               │  (EMQX /   │
                                           │  Mosquitto)│
┌──────────┐     subscribe "sensor/#"      │            │
│ 后端服务  │ ◀─────────────────────────────│            │
│Subscriber│                               └────────────┘
└──────────┘

ruoyi-common-mqtt 的角色是什么？它是 MQTT 体系中的 Client——连接到 Broker，既可以发布消息也可以订阅消息。模块本身只有 3 个 Java 文件，所有 MQTT 通信能力来自 Mica MQTT（Dromara 社区出品，基于 t-io 网络框架）。本模块做的事情是"项目化适配"——用 Spring Boot 自动配置把 Mica MQTT 包装成即插即用的 Bean。

2. 核心设计

2.1 MQTT 协议的三个关键概念

概念含义示例
Topic消息路由标签，UTF-8 字符串，用 / 分层sensor/temperature/room1
QoS交付质量保证：0=至多一次、1=至少一次、2=恰好一次温度上报用 QoS 0，告警用 QoS 1
Wildcard订阅通配符：+ 单层、# 多层sensor/+/room1、sensor/#
关键区别：Topic 不需要预先创建，发布消息时自动创建。这和 Kafka 的 Topic 或 RabbitMQ 的 Queue 不同——MQTT 的 Topic 更像一个「动态路由标签」，而非预先声明的存储结构。

2.2 模块三剑客

打开 config/ 目录，核心只有三个文件：

文件职责
MqttAutoConfiguration.java自动配置入口，用 @ConditionalOnProperty(value="mqtt.client.enabled", havingValue="true") 控制开关，注册两个监听器和一个定制器 Bean
MqttClientConnectListener.java实现 IMqttClientConnectListener，处理连接/断连事件，断连时可动态更换 clientId/username/password
MqttClientGlobalMessageListener.java实现 IMqttClientGlobalMessageListener，接收所有已订阅主题的消息，是消息的统一入口

注意还有一个隐形的第四位——MqttClientCustomizer，它是 Mica MQTT 的扩展点，通过 Lambda 表达式在自动配置中以内联 Bean 形式创建，负责用虚拟线程替换默认线程池。下一课详细讲。

2.3 配置驱动：application.yml

mqtt:
  client:
    enabled: true
    ip: 127.0.0.1
    port: 1883
    client-id: "000001"
    username: admin
    password: public
    version: mqtt_5              # 支持 mqtt_3_1 / mqtt_3_1_1 / mqtt_5
    keep-alive-secs: 60
    session-expiry-interval-secs: 300

client-id 是 MQTT 协议层面的唯一标识，与 Mica MQTT 层面的 name（客户端名称）是不同的两个概念。MQTT 5 相比 3.1.1 增加了会话过期时间、主题别名等新特性，本模块配置支持版本切换。

3. 实战练习

🧠 提取练习（点击展开）

Q1: MQTT 发布订阅模型与 RabbitMQ 队列模型的本质区别是什么？

参考: MQTT 是 Topic 广播——发布者和订阅者空间解耦，互不知晓；RabbitMQ 是 Queue 点对点——生产者发到队列、消费者从队列取。MQTT 不保证消息被消费（除非 QoS > 0），队列模型保证至少一次投递。

Q2: sensor/+/temperature 和 sensor/# 分别能匹配哪些 Topic？

参考: + 匹配单层：sensor/room1/temperature、sensor/room2/temperature；# 匹配多层：sensor/room1/temperature、sensor/room1/humidity、sensor/room1/device/status 等所有子孙节点。

4. 调用链路

从应用启动到消息接收的完整路径：

Spring Boot 扫描 AutoConfiguration.imports
  → MqttAutoConfiguration 被激活（因为 mqtt.client.enabled=true）
    → 注册 MqttClientCustomizer（定制 MqttClientCreator）
    → 注册 MqttClientConnectListener（连接监听器）
    → 注册 MqttClientGlobalMessageListener（消息监听器）
    → Mica MQTT 自动创建 MqttClient，连接到 Broker
      → 连接成功 → onConnected() 回调
      → 收到消息 → onMessage(topic, payload) 回调
💡 设计洞察：Mica MQTT 的 Java 版 MQTT 实现与 Mosquitto/EMQX 等 C/C++ Broker 在协议上完全互通——只要有 S3 级别的标准协议兜底，语言和具体实现从来不重要。本模块通过标准 MQTT 5 协议连接到任意兼容 Broker，实现了厂商无关性。
