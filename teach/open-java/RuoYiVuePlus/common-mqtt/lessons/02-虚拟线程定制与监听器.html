02 - 虚拟线程定制与监听器

🎯 本节目标：理解 MqttClientCustomizer 为何要用虚拟线程替换默认线程池，能讲清两个监听器各自的责任边界，并能编写简单的消息发布与订阅代码。

1. 全景鸟瞰

上一课讲了 MQTT 协议和模块结构。本课聚焦模块中仅有的 3 个类如何在运行时协同工作：

MqttAutoConfiguration
├── @Bean MqttClientCustomizer  → 用 Virtual Thread 换掉默认线程池
├── @Bean MqttClientConnectListener → 连接生命周期管理
└── @Bean MqttClientGlobalMessageListener → 全局消息接收

2. 核心设计

2.1 虚拟线程：为什么要换掉默认线程池？

Mica MQTT 默认使用普通 OS 线程池处理消息。在物联网场景中可能有数千台设备同时上报数据，每台设备对应一个 MQTT 连接——OS 线程数是有限的（通常数百到一千），传统线程池会快速耗尽。

Java 21 引入的虚拟线程（Virtual Thread）解决了这个问题：

维度OS 平台线程虚拟线程
内存开销~1MB/线程~数百字节/线程
创建成本毫秒级微秒级
阻塞行为阻塞整个 OS 线程阻塞时自动释放底层平台线程
数量上限受 OS 限制（通常数千）可达百万级

在 MqttAutoConfiguration 中的关键代码等价于：

@Bean
public MqttClientCustomizer mqttClientCustomizer() {
    return creator -> {
        // 用虚拟线程执行器替换 Mica MQTT 默认线程池
        creator.taskExecutor(new VirtualThreadTaskExecutor("mqtt-"));
    };
}
注意：虚拟线程不是协程（如 Go 的 goroutine），它是 JVM 管理的轻量级线程，在 JVM 层面实现了阻塞时自动 yield 的调度行为。

2.2 MqttClientConnectListener：连接生命的守护者

实现 IMqttClientConnectListener 接口，两个回调方法：

回调触发时机能力
onConnected(ChannelContext, boolean isReconnect)首次连接成功 + 断线重连成功记录连接日志；isReconnect 参数区分首次/重连
onDisconnect(ChannelContext, Throwable, String, boolean isRemove)任何原因导致断开记录断连日志 + 动态修改重连参数（clientId/username/password）以支持凭证轮换

凭证轮换场景：如果 Broker 要求定期更换密码，可以在 onDisconnect 中从配置中心或数据库获取新密码，更新到 ChannelContext 的属性中。下次重连时 Mica MQTT 会使用新的凭证——对业务代码完全透明。

2.3 MqttClientGlobalMessageListener：消息的统一入口

实现 IMqttClientGlobalMessageListener，只有一个回调：

onMessage(ChannelContext context, String topic, MqttPublishMessage message, byte[] payload)

这是 全局监听器——一个客户端只有一个，收到的所有 Topic 消息都经过它。业务代码中的职责是：

  1. 将 payload（byte[]）按 UTF-8 解码为字符串
  2. 根据 topic 路由到不同的业务处理逻辑
  3. 根据 message.getQos() 决定是否需要回执

3. 实战练习

🧠 提取练习（点击展开）

Q1: 为什么虛拟线程比传统线程更适合 MQTT 消息处理场景？

参考: MQTT 场景中常出现大量并发连接和 IO 阻塞等待（等待 Broker 响应、等待网络 IO）。虚拟线程在阻塞时自动释放底层平台线程，一个 OS 线程可以承载数千个虚拟线程的调度，避免线程池耗尽。

Q2: onDisconnect 回调中修改 clientId 是一种什么设计模式？有什么实际价值？

参考: 这是一种「凭证轮换」策略——利用断连窗口期注入新凭证，下次重连时自动生效。实际价值是实现了零停机的客户端凭证更新，适用于需要定期更换密码的安全场景。

Q3: 如果需要在代码中发布一条 MQTT 消息，应该怎么做？

参考: 注入 Mica MQTT 的 MqttClient 实例，调用 client.publish(topic, payload.getBytes(StandardCharsets.UTF_8), qos)。注意选择合适的 QoS 级别：心跳/高频数据用 QoS 0，重要通知用 QoS 1。

4. 调用链路

从应用启动到消息收发的完整时序：

1. AutoConfiguration.imports → MqttAutoConfiguration
2. 注册 3 个 Bean 到 Spring 容器
3. Mica MQTT 自动创建 MqttClient（使用 Customizer 定制后的配置）
4. MqttClient 向 Broker 发起 TCP 连接
5. 连接成功 → onConnected(ctx, false)  → 日志："MQTT 连接成功"
6. 业务线程调用 client.subscribe("sensor/#", qos0)
7. Broker 推送消息 → onMessage(ctx, "sensor/temp", msg, payload)
8. 连接断开 → onDisconnect(ctx, cause, reason, isRemove=false)
9. 自动重连 → onConnected(ctx, true) → 日志："MQTT 重连成功"
💡 设计洞察：模块设计遵循「最小化」原则——只有 3 个 Java 文件，却能支撑起完整的设备通信场景。这不是因为功能少，而是因为 Mica MQTT 已经提供了完整的 MQTT 客户端实现，本模块只在框架与项目之间做了最少量的适配：虚拟线程优化 IO、统一日志规范、凭证动态管理。这是微服务「组合优于继承」思想的典型体现。
