第一部分：MQTT 协议——为什么物联网世界选它？

1.1 一句话定义

MQTT（Message Queuing Telemetry Transport） 是一个基于 发布/订阅（Pub/Sub）模型 的轻量级消息协议，运行在 TCP/IP 之上，由 IBM 于 1999 年发明，2013 年成为 OASIS 标准。最新版本为 MQTT 5.0（2019 年发布）。

💡 记住一句话：MQTT = 「发布订阅 + 主题路由 + 三级 QoS + 遗嘱消息」的极简消息协议。它不是 RabbitMQ/Kafka 那样的消息队列中间件——它只负责消息的实时推送转发。

1.2 发布订阅模型拓扑图

理解 MQTT 最关键的就是这张三角关系图：

MQTT Broker（消息代理） ┌───────────────────────┐ │ EMQX / Mosquitto │ │ ┌─────────────────┐ │ │ │ Topic 路由表 │ │ │ │ sensor/temp/room1 │ │ │ │ device/+/status │ │ │ │ ... │ │ │ └─────────────────┘ │ └──┬────────────────┬────┘ publish │ │ subscribe "25.5°C" │ │ "sensor/temp/room1" ┌──────────┘ └──────────┐ │ │ ┌─────┴─────┐ ┌─────┴─────┐ │ Publisher │ │ Subscriber │ │ 温度传感器 │ │ 监控大屏 │ └───────────┘ └───────────┘ 关键约束： - Publisher 和 Subscriber 互不知晓对方存在（空间解耦） - 它们可以不同时在线（时间解耦——靠 Broker 缓存 + 持久会话） - 一个 Topic 可以有多个 Subscriber（一对多广播） - 消息无持久化保证，实时推送（和 Kafka 的持久化日志不同）

1.3 核心概念速览

概念说明类比（邮政系统）
Broker 中央消息代理，接收所有消息并按 Topic 路由给订阅者 邮政总局——收件后按地址分拣投递
Topic UTF-8 字符串，用 / 分层，如 sensor/temperature/room1 收件地址——决定了消息去哪
Publisher 发消息的一方，向 Broker 的某个 Topic 发布消息 寄信人
Subscriber 收消息的一方，向 Broker 声明自己订阅哪些 Topic 收信人
QoS 服务质量：0=至多一次，1=至少一次，2=恰好一次 平信 / 挂号信 / 签收回执
Keep Alive 心跳间隔（秒），客户端定期发 PINGREQ 证明自己还活着 定期报平安——超时不报则 Broker 认为已离线
Persistent Session 持久会话：Broker 记住客户端的订阅和未投递消息，重连后补发 出差回来，传达室把你不在时的信件都给你
Last Will（遗嘱消息） 客户端连接时预设一条消息，一旦异常断连，Broker 代为发布到指定 Topic 「如果我失联了，帮我发一条通知给所有人」

1.4 Topic 通配符

订阅时可以使用两种通配符来批量匹配主题：

# 单层通配符 '+' ：匹配一个层级 sensor/+/temperature → 匹配 sensor/room1/temperature 匹配 sensor/room2/temperature 不匹配 sensor/room1/humidity # 多层通配符 '#' ：匹配任意多个层级（必须在末尾） sensor/# → 匹配 sensor/temperature 匹配 sensor/room1/humidity 匹配 sensor/room1/nested/deep # 项目中常见命名规范： ruoyi/device/{设备ID}/event → 设备事件上报 ruoyi/device/{设备ID}/command → 向设备下发指令 ruoyi/device/{设备ID}/status → 设备状态上报

1.5 MQTT 5 vs 3.1.1

本模块的 yml 配置中默认使用 version: mqtt_5。MQTT 5 相对于 3.1.1 的主要改进：

特性MQTT 3.1.1MQTT 5
Session 过期持久/非持久 二选一session-expiry-interval-secs 精确控制过期时间
错误反馈仅返回失败，无原因原因码（Reason Code）精确说明失败原因
用户属性不支持支持键值对元数据（User Properties）
消息过期不支持支持 message-expiry-interval 过期丢弃
主题别名不支持用数字别名替代长 Topic 字符串，减少带宽

第二部分：ruoyi-common-mqtt 模块全景

2.1 整个模块只有三个类

打开 ruoyi-common-mqtt 模块，你会看到：

ruoyi-common-mqtt/ ├── pom.xml ← 依赖 mica-mqtt-client-starter │ + ruoyi-common-core │ + ruoyi-common-json └── src/main/ ├── java/org/dromara/common/mqtt/ │ ├── config/MqttAutoConfiguration.java ← 【装配】注册所有 Bean │ └── listener/ │ ├── MqttClientConnectListener.java ← 【连接】连接生命周期回调 │ └── MqttClientGlobalMessageListener.java ← 【消息】全局消息接收 └── resources/META-INF/spring/ └── org.springframework.boot.autoconfigure .AutoConfiguration.imports ← 【发现】告诉 Spring Boot 加载配置类

真正处理 MQTT 协议细节（TCP 连接、报文编解码、QoS 握手、心跳维护）的脏活累活，全在第三方框架 Mica MQTT（Dromara 社区出品，基于 t-io 网络框架）里。本模块一行「操作 MQTT 报文」的代码都没有。

💡 一句话定位：
ruoyi-common-mqtt 不是一个 MQTT 框架，而是一层「接入垫片（adapter）」——它把 Mica MQTT 客户端「收编」进 RuoYi 的 Spring Boot 自动配置体系，注入虚拟线程优化性能，并注册标准化的连接/消息监听器。

2.2 三个类，三件事

类一句话职责解决什么问题对应课
MqttAutoConfiguration 装配入口：注册两个监听器 + 虚拟线程定制器 依赖即生效 + 配置开关控制 + 虚拟线程 IO 优化 第 2 课
MqttClientConnectListener 实现 IMqttClientConnectListener，监听连接/断连 统一管理 MQTT 连接生命周期，日志记录 + 断线后可动态更新凭证 第 3 课
MqttClientGlobalMessageListener 实现 IMqttClientGlobalMessageListener，监听所有消息 提供一个全局消息入口点，在此做日志、转发、业务分发 第 3 课

2.3 模块如何被激活？

模块不暴露 Controller，它是底层通信能力。激活链路非常简单：

1. pom.xml 引入 mica-mqtt-client-spring-boot-starter → Mica MQTT 自己的自动配置启动，创建 MqttClientCreator Bean 2. MqttAutoConfiguration 被 AutoConfiguration.imports 发现 → 读取 yml 中 mqtt.client.* 配置 → 创建两个监听器 + MqttClientCustomizer 定制虚拟线程 → 监听器注册到 MqttClientCreator 3. Mica MQTT Client 根据 yml 配置自动连接 Broker → 连接成功 → MqttClientConnectListener.onConnected() → 收到消息 → MqttClientGlobalMessageListener.onMessage() → 连接断开 → MqttClientConnectListener.onDisconnect() 4. 业务代码通过 @Autowired MqttClientCreator 发消息 → mqttClientCreator.publish(topic, payload)

注意第 3 步是 Mica MQTT 自动完成的——只要 yml 里配置了连接参数，它的 Starter 就会在应用启动时自动连上 Broker。本模块的自动配置只是在 Mica MQTT 的基础上注入自定义监听器和线程池，而不是取而代之。

2.4 为什么值得学这个模块？

这个模块代表了 RuoYi 对接 非 Web 协议（非 HTTP） 的标准范式：

  • 依赖成熟开源实现（Mica MQTT），不自己造 TCP 轮子；
  • 用自动配置包装，让业务模块零代码接入 MQTT 通信；
  • 用监听器模式统一管理连接状态和消息处理，方便扩展业务逻辑；
  • 用虚拟线程替代传统线程池，适配高并发 IoT 场景。
🤔 想一想：ymL 里 mqtt.client.enabled: false，模块还加载吗？两个监听器还会创建吗？ MqttAutoConfiguration 上有 @ConditionalOnProperty(value="mqtt.client.enabled", havingValue="true")，默认值为 false。这意味着：
  • 自动配置类本身不会被加载——整个 MqttAutoConfiguration 被跳过；
  • 两个监听器 Bean 不会创建——因为它们是在这个配置类里用 @Bean 定义的；
  • Mica MQTT 自己的 Starter 可能会加载，但因为没有连接参数可能不会真正连接。
所以这个 enabled 是一个总开关：关掉后 MQTT 功能零开销。第 2 课细讲这个机制。

Q1. MQTT 使用什么通信模型？和 RabbitMQ 的 queue 模型有什么本质区别？

A. 点对点模型，和 RabbitMQ 一样 B. 发布订阅模型，消息按 Topic 广播而非按 queue 消费；Publisher 和 Subscriber 互不知晓 C. 请求响应模型，和 HTTP 一样 D. 流处理模型，和 Kafka 一样
看答案B。MQTT 是发布订阅模型，Publisher 发消息到 Broker 的 Topic，Broker 推送给所有订阅该 Topic 的 Subscriber——不存在 "queue 消费后删除" 的概念。RabbitMQ 的 queue 模型是点对点消费，消息被一个消费者取走后即删除。

Q2. ruoyi-common-mqtt 模块和 Mica MQTT 是什么关系？

A. ruoyi-common-mqtt 自己实现了 MQTT 协议，没用任何第三方库 B. ruoyi-common-mqtt 是接入垫片，MQTT 协议处理全靠 Mica MQTT，自己只补装配/监听器/虚拟线程三件事 C. 两者是竞争关系，二选一 D. Mica MQTT 依赖 ruoyi-common-mqtt
看答案B。模块只有 3 个类，没有一行 MQTT 协议实现代码。真正处理 TCP 连接、报文编解码、QoS 握手等工作全在 Mica MQTT。模块的价值在于「把 Mica MQTT 接入 RuoYi 的自动配置体系」。

Q3. 下面哪个 Topic 通配符用法是正确的？

A. sensor/+/temperature 匹配 sensor/room1/humidity B. sensor/# 匹配 sensor/temperature 和 sensor/room1/humidity C. #/sensor 匹配 a/sensor D. sensor/+ 匹配 sensor/room1/temperature
看答案B。sensor/# 匹配所有以 sensor/ 开头的 Topic。A 错在 + 只匹配一个层级，sensor/+/temperature 不匹配三个层级的 sensor/room1/humidity；C 错在 # 必须在末尾；D 错在 + 只匹配 room1 这一个层级，不匹配 room1/temperature。

Q4. ruoyi-common-mqtt 模块为什么不暴露任何 Controller？

A. 它是底层通信能力层，提供 MQTT 连接/订阅/发布能力给上层业务使用，自己不直接接客 B. MQTT 不需要 HTTP 接口 C. Controller 写在 Mica MQTT 里了 D. 模块功能太简单，不值得暴露
看答案A。ruoyi-common-* 是被业务模块依赖的底层能力。MQTT 消息的收发通过 MqttClientCreator Bean 进行，业务层可以通过注入该 Bean 直接发消息；收消息则通过自定义 IMqttClientGlobalMessageListener 实现来处理。
