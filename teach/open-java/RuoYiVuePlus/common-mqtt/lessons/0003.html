第一部分：MqttClientConnectListener —— 连接生命周期的守护者

1.1 完整源码

先看原始代码（注释保留）：

// ruoyi-common-mqtt/listener/MqttClientConnectListener.java
@Slf4j
public class MqttClientConnectListener implements IMqttClientConnectListener {

    private final MqttClientCreator mqttClientCreator;

    public MqttClientConnectListener(MqttClientCreator mqttClientCreator) {
        this.mqttClientCreator = mqttClientCreator;
    }

    @Override
    public void onConnected(ChannelContext context, boolean isReconnect) {
        // 创建连接
        log.info("MqttConnectedEvent:{}", context);
    }

    @Override
    public void onDisconnect(ChannelContext context, Throwable throwable,
                             String remark, boolean isRemove) {
        // 离线时更新重连
        log.info("MqttDisconnectEvent:{}", context, throwable);
        // 在断线时更新 clientId、username、password
//      mqttClientCreator.clientId("newClient" + System.currentTimeMillis())
//          .username("newUserName")
//          .password("newPassword");
    }
}

1.2 接口定义

实现的接口是 IMqttClientConnectListener（Mica MQTT 定义），包含两个回调：

回调方法触发时机参数作用
onConnected 客户端成功连接到 Broker（首次连接和重连均触发） ChannelContext context — 通道上下文（含 IP/端口/Session 信息）
boolean isReconnect — 是否重连 		记录连接状态；是重连时可以恢复业务逻辑
onDisconnect 客户端与 Broker 断开连接（主动断开或异常断连） ChannelContext context — 通道上下文
Throwable throwable — 断开异常（null = 正常断开）
String remark — 断开说明
boolean isRemove — 是否移除连接 		记录断开原因；动态更新重连凭证

1.3 关键设计点

1持有 MqttClientCreator 引用是为了凭证轮换
注意第 2 课提到构造器接收了 MqttClientCreator，但代码里却注释掉了动态修改凭证的逻辑。这是预设的扩展点——如果你的场景需要：
  • 断线后用新的 clientId 重连（避免旧 ID 在 Broker 端有残留 Session）
  • 使用短期 token 做认证，断线后自动获取新 token 填入 username/password
只需取消注释并实现获取新凭证的逻辑即可。框架不做你不需要的复杂度，但留好了口子。
2isReconnect 参数区分首次连接与重连
MqttClientConnectListener.java:34
onConnected 的第二个参数 isReconnect 是关键区分器。
  • false → 首次连接成功，可以做初始化操作（如重订阅 Topic、通知业务模块）
  • true → Broker 重连成功（断线后 Mica MQTT 自动重连），需要恢复状态
当前实现只打日志，但在生产环境中，你应该在 isReconnect == false 时做订阅操作（如果 Persistent Session 不可靠的话）。
3throwable 为 null 时的含义
MqttClientConnectListener.java:48
onDisconnect 的第二个参数 Throwable throwable：
  • 非 null → 异常断连（网络中断、Broker 崩溃、Keep Alive 超时被踢），需要告警
  • null → 正常断开（客户端主动 disconnect），不需要告警
当前代码对所有情况都打日志，生产环境可以用 if (throwable != null) 区分告警级别。

第二部分：MqttClientGlobalMessageListener —— 所有消息的统一入口

2.1 完整源码

// ruoyi-common-mqtt/listener/MqttClientGlobalMessageListener.java
@Slf4j
public class MqttClientGlobalMessageListener
        implements IMqttClientGlobalMessageListener {

    @Override
    public void onMessage(ChannelContext context, String topic,
                          MqttPublishMessage message, byte[] payload) {
        log.info("MqttGlobalMessageEvent => topic: {}, msg: {}",
                 topic, new String(payload, StandardCharsets.UTF_8));
    }
}

2.2 接口与参数

实现的接口是 IMqttClientGlobalMessageListener（Mica MQTT 定义），只有一个回调方法：

参数类型说明
context ChannelContext 通道上下文，含发送端连接信息（客户端 IP 等）
topic String 消息的 Topic 全名（非通配符，是实际匹配到的具体 Topic）
message MqttPublishMessage Mica MQTT 包装的 MQTT PUBLISH 报文对象，含 QoS 等级、是否保留消息（Retain）、消息 ID 等协议层信息
payload byte[] 消息的有效载荷（原始字节），需按业务约定解码（UTF-8 / JSON / Protobuf 等）

2.3 "全局（Global）"的含义

Mica MQTT 支持两种消息监听方式：

监听类型粒度适用场景
全局监听器
IMqttClientGlobalMessageListener		 一个客户端只有一个，所有 Topic 的消息都进来 统一的消息入口：日志审计、消息转发、总控分发
Topic 级监听器
IMqttClientMessageListener		 按 Topic 注册，只处理匹配的消息 具体业务处理：设备数据入库、告警判断、指令执行

本模块选择的是全局监听器，作为所有消息的统一入口点。在这个入口里，你可以：

  1. 打日志（当前实现）——便于排查和审计；
  2. 按 topic 分流——根据 topic 前缀路由到不同的业务处理器；
  3. 消息格式校验——检查 payload 是否符合预期的 JSON Schema 或 Protobuf 定义；
  4. 前置处理——解密、解压缩、格式转换等。

第三部分：实战 —— 编写消息收发代码

理论讲完，来写代码。下面两段可以直接放到 RuoYi 项目里跑（需要先安装并启动一个 MQTT Broker，如 Mosquitto 或 EMQX）。

3.1 场景设定

假设你有一个智能家居系统，设备（如温度传感器）通过 MQTT 上报数据，后端需要：

  • 接收设备上报的温度数据，存入数据库；
  • 当温度超过阈值时，向设备下发告警指令。

3.2 发布消息（向设备下发指令）

// 场景：业务层创建一个 MqttPublisher 工具类，用于向设备下发指令

import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.dromara.mica.mqtt.core.client.MqttClientCreator;
import org.springframework.stereotype.Component;

import java.nio.charset.StandardCharsets;

@Slf4j
@Component
@RequiredArgsConstructor
public class MqttPublisher {

    private final MqttClientCreator mqttClientCreator;  // Mica MQTT 自动注入

    /**
     * 向指定设备下发指令。
     *
     * @param deviceId 设备 ID
     * @param command  指令内容（JSON 字符串）
     */
    public void sendCommand(String deviceId, String command) {
        String topic = "ruoyi/device/" + deviceId + "/command";
        byte[] payload = command.getBytes(StandardCharsets.UTF_8);

        // publish() 返回 CompletableFuture，可异步等待发送结果
        mqttClientCreator.publish(topic, payload)
            .whenComplete((result, throwable) -> {
                if (throwable != null) {
                    log.error("下发指令失败 => device:{}, topic:{}", deviceId, topic, throwable);
                } else {
                    log.info("指令已下发 => device:{}, topic:{}, payload:{}", deviceId, topic, command);
                }
            });
    }

    /**
     * 示例：某个设备温度超标，下发关闭指令。
     */
    public void handleOverTemperature(String deviceId, double temperature) {
        if (temperature > 60.0) {
            String command = """
                {"action":"shutdown","reason":"over_temperature","value":%.1f}
                """.formatted(temperature);
            sendCommand(deviceId, command);
        }
    }
}

3.3 扩展消息监听器（按 Topic 分发业务处理）

// 场景：扩展全局监听器，按 topic 前缀分流到不同的业务处理器

import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import net.dreamlu.mica.net.core.ChannelContext;
import org.dromara.mica.mqtt.codec.message.MqttPublishMessage;
import org.dromara.mica.mqtt.core.client.IMqttClientGlobalMessageListener;
import org.springframework.stereotype.Component;

import java.nio.charset.StandardCharsets;

@Slf4j
@Component
@RequiredArgsConstructor
public class CustomMqttMessageListener implements IMqttClientGlobalMessageListener {

    private final DeviceDataService deviceDataService;  // 你的业务服务
    private final MqttPublisher mqttPublisher;

    @Override
    public void onMessage(ChannelContext context, String topic,
                          MqttPublishMessage message, byte[] payload) {
        String msgBody = new String(payload, StandardCharsets.UTF_8);
        log.info("收到消息 => topic:{}, payload:{}", topic, msgBody);

        // 按 topic 前缀分流
        if (topic.startsWith("ruoyi/device/") && topic.endsWith("/data")) {
            // 设备数据上报：解析 JSON → 存入数据库 → 检查阈值 → 告警
            handleDeviceData(topic, msgBody);
        } else if (topic.startsWith("ruoyi/device/") && topic.endsWith("/status")) {
            // 设备状态上报：更新设备在线状态
            handleDeviceStatus(topic, msgBody);
        } else if (topic.startsWith("ruoyi/device/") && topic.endsWith("/event")) {
            // 设备事件上报：记录事件日志
            handleDeviceEvent(topic, msgBody);
        } else {
            log.warn("未识别的 topic: {}", topic);
        }
    }

    private void handleDeviceData(String topic, String msgBody) {
        // 从 topic 中提取设备 ID
        // ruoyi/device/{deviceId}/data → 取第二个路径段
        String deviceId = topic.split("/")[2];

        // 解析消息体（假设是 JSON: {"temperature": 25.5, "humidity": 60}）
        DeviceData data = parseDeviceData(msgBody);

        // 存入数据库
        deviceDataService.save(deviceId, data);

        // 检查阈值，超标则告警
        if (data.getTemperature() > 60.0) {
            log.warn("设备 {} 温度超标: {:.1f}°C", deviceId, data.getTemperature());
            mqttPublisher.handleOverTemperature(deviceId, data.getTemperature());
        }
    }

    // ... handleDeviceStatus / handleDeviceEvent / parseDeviceData 省略
}

/**
 * 注意：当自定义了这个 Bean 后，MqttAutoConfiguration 中定义的
 * MqttClientGlobalMessageListener 默认 Bean 仍会存在。
 * 如果你只需要自定义的监听器，可以使用以下方式之一：
 *
 * 方案 A: @Primary 标注你自定义的，让它优先注入
 * 方案 B: 排除默认 Bean（@SpringBootApplication(exclude=...)）
 * 方案 C: 使用 @ConditionalOnMissingBean 让默认 Bean 自动退让
 *
 * 实际项目中，通常保留默认 Bean 做全局日志，另起一个自定义 Bean
 * 注册为 IMqttClientMessageListener（Topic 级）做具体业务。
 */

3.4 订阅 Topic

以上代码是接收消息和发送消息，但还缺一步——告诉 Broker 我要收哪些 Topic 的消息。订阅操作可以在应用启动后执行：

// 在某个 @PostConstruct 或 ApplicationRunner 中订阅 Topic

import jakarta.annotation.PostConstruct;
import lombok.RequiredArgsConstructor;
import org.dromara.mica.mqtt.core.client.MqttClientCreator;
import org.springframework.stereotype.Component;

@Component
@RequiredArgsConstructor
public class MqttTopicSubscriber {

    private final MqttClientCreator mqttClientCreator;

    @PostConstruct
    public void subscribeTopics() {
        // 订阅设备数据上报 Topic（通配符匹配所有设备）
        mqttClientCreator.subscribe("ruoyi/device/+/data");

        // 订阅设备状态
        mqttClientCreator.subscribe("ruoyi/device/+/status");

        // 订阅设备事件
        mqttClientCreator.subscribe("ruoyi/device/+/event");

        log.info("MQTT Topic 订阅完成");
    }
}
⚠️ 订阅注意事项：
  1. 订阅应在 onConnected 后执行——如果使用 Persistent Session，重连后 Broker 会恢复之前的订阅；如果不依赖 Persistent Session，应在 onConnected（尤其是 isReconnect==false）中重新订阅。
  2. @PostConstruct 可能在 MQTT 连接建立之前执行——更可靠的做法是在 MqttClientConnectListener.onConnected() 中触发订阅。
  3. 通配符 + 匹配任意单个设备 ID，# 可匹配所有子层级。

第四部分：对比总结 —— 两个监听器的职责边界

维度MqttClientConnectListenerMqttClientGlobalMessageListener
核心关注连接（连上了没？断线了没？怎么重连？）消息（收到了什么？Topic 是什么？内容是什么？）
触发频率低频——每次连接/断连触发一次高频——每条消息触发一次（可能每秒成千上万条）
典型操作日志、告警、更新凭证、触发重订阅日志、消息路由、数据解析、业务处理
扩展方式取消注释 + 实现凭证获取逻辑新建实现类 + 按 topic 分流 + 调用业务服务
性能要求低——偶发操作高——可能成为消息吞吐的瓶颈（所以用了虚拟线程）

五、检验一下（提取练习）

Q1. onConnected 的 isReconnect 参数为 true 意味着什么？这种场景下通常需要做什么？

A. 意味着首次连接成功，需要初始化所有状态 B. 意味着断线后重连成功，可能需要恢复订阅（如果不依赖 Persistent Session） C. 意味着连接即将断开 D. 没有实际意义
看答案B。isReconnect=true 表示这是一次重连（之前断线了）。如果 Persistent Session 不可靠或未启用，应该在这里重新订阅之前的所有 Topic。首次连接时 isReconnect=false。

Q2. MqttClientGlobalMessageListener 的 payload 参数是什么类型？如何正确读取？

A. String 类型，直接 toString() 即可 B. byte[] 类型，需要按业务约定的编码/格式解码（如 new String(payload, StandardCharsets.UTF_8)） C. JSONObject 类型 D. InputStream 类型
看答案B。payload 是 byte[]，因为 MQTT 协议传递的是原始字节（可以是文本、JSON、Protobuf、二进制图片等）。必须按业务约定来解码。当前代码用 UTF-8 解码表明约定为文本消息。

Q3. 为什么 MqttClientConnectListener 要持有 MqttClientCreator？

A. 为了在 onConnected 中再次连接 B. 为了在 onDisconnect 中动态修改 clientId/username/password，实现断线重连时的凭证轮换 C. 为了让 Creator 不被 GC 回收 D. 没有实际用途，只是遗留代码
看答案B。Creator 允许在断线时更新凭证（如轮换 clientId 避免 Session 冲突、刷新短期 token）。虽然当前注释掉了，但这是留给使用者的扩展点。

Q4. 以下哪个 Topic 匹配 ruoyi/device/+/data？

A. ruoyi/device/sensor001/data B. ruoyi/device/sensor001/status C. ruoyi/device/sensor001/nested/data D. ruoyi/gateway/data
看答案A。+ 通配符精确匹配一个层级，所以 ruoyi/device/+/data 匹配 ruoyi/device/任意字符串/data。B 的最后一段是 status 不匹配；C 有四个层级（nested/data 是两层）不匹配；D 前缀不是 device 不匹配。
