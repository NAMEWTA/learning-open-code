01 | 17种脱敏策略：@Sensitive 注解与策略枚举体系

ruoyi-common-sensitive 模块是整个 RuoYi-Vue-Plus 生态中体量最小、设计最精炼的模块之一——它只有 5 个 Java 文件，却支撑了整个系统的脱敏能力。本课聚焦于 @Sensitive 注解与 SensitiveStrategy 枚举，它们是脱敏体系的"心脏"。

1. 全景鸟瞰：5 个文件的协作关系

在深入细节之前，先建立整体认知。这 5 个 Java 文件各司其职：

文件包路径角色
@Sensitive 注解org.dromara.common.sensitive.annotation标记字段需要脱敏，定义脱敏策略和权限条件
SensitiveStrategy 枚举org.dromara.common.sensitive.core17 种脱敏策略，每个枚举值持有一个 Function<String,String> 脱敏器
SensitiveJsonFieldProcessororg.dromara.common.sensitive.handler实现 JsonFieldProcessor，在 JSON 序列化时检测 @Sensitive 并执行脱敏
SensitiveConfigorg.dromara.common.sensitive.config自动装配类，将 SensitiveJsonFieldProcessor 注册为 Spring Bean
SensitiveService 接口org.dromara.common.sensitive.core条件脱敏接口，根据角色/权限判断当前用户是否需要脱敏

这 5 个文件构成了一条清晰的协作链：

@Sensitive 标记字段 → SensitiveJsonFieldProcessor 检测 → SensitiveService 判断是否脱敏
    → SensitiveStrategy 选择脱敏器 → hutool DesensitizedUtil 执行脱敏
关键洞察：整个脱敏模块是一个典型的 声明式处理架构——开发者只需在实体类字段上加 @Sensitive(strategy = SensitiveStrategy.PHONE)，其余全部由框架自动完成。这与 Spring 的 @Transactional 理念一致：声明意图，框架执行。

2. @Sensitive 注解详解

@Sensitive 是脱敏体系的入口注解，标注在 Java Bean 的字段上。它有三个属性：

@Target({ElementType.FIELD})
@Retention(RetentionPolicy.RUNTIME)
public @interface Sensitive {
    SensitiveStrategy strategy();          // 必填：脱敏策略
    String[] roleKey() default {};         // 可选：角色 key 数组
    String[] perms() default {};           // 可选：权限标识数组
}

2.1 strategy() —— 必填属性

strategy() 没有默认值，必须显式指定。这确保了开发者不会"忘记"选择脱敏方式而意外暴露敏感数据。它接收一个 SensitiveStrategy 枚举值，例如：

@Sensitive(strategy = SensitiveStrategy.PHONE)
private String phone;         // 13812345678 → 138****5678

@Sensitive(strategy = SensitiveStrategy.ID_CARD)
private String idCard;        // 110101199001011234 → 110***********1234

@Sensitive(strategy = SensitiveStrategy.BANK_CARD)
private String bankCard;      // 6222021234567890 → 622202********7890

2.2 roleKey() 与 perms() —— 条件脱敏

这两个属性是可选的，不填写时对所有用户生效。一旦填写，SensitiveService.isSensitive(roleKey[], perms[]) 会被调用来判断当前用户是否需要脱敏。

// 场景：仅对普通角色脱敏，admin 角色看到明文
@Sensitive(strategy = SensitiveStrategy.PHONE, roleKey = {"common"})
private String phone;

// 场景：仅对没有 "system:user:query" 权限的用户脱敏
@Sensitive(strategy = SensitiveStrategy.ID_CARD, perms = {"system:user:query"})
private String idCard;
roleKey 和 perms 是 OR 关系：只要用户满足任一 roleKey 或任一 perms，就会进行脱敏。如果两者都不配置，默认对所有用户脱敏。
实践建议：常见的权限设计是 admin 角色看到明文，普通用户看到脱敏数据。此时在 admin 角色对应的 SensitiveService 实现中，对 roleKey 为 admin 的情况返回 false（不脱敏），对 common 返回 true（脱敏）。

3. SensitiveStrategy 枚举详解

SensitiveStrategy 是整个模块最核心的类。它是一个存储在 org.dromara.common.sensitive.core 包下的枚举，每个枚举值都是一个独立的脱敏策略。

3.1 策略模式实现

每个枚举值在构造时接收一个 Function<String, String> 类型的脱敏器：

public enum SensitiveStrategy {
    PHONE(s -> DesensitizedUtil.mobilePhone(s)),
    ID_CARD(s -> DesensitizedUtil.idCardNum(s, 3, 4)),
    // ... 其他 15 个策略

    private final Function<String, String> desensitizer;

    SensitiveStrategy(Function<String, String> desensitizer) {
        this.desensitizer = desensitizer;
    }

    public String desensitize(String value) {
        return desensitizer.apply(value);
    }
}

这是一个干净的策略模式实现。每个枚举值就是一个小型策略对象。当处理器需要执行脱敏时，调用 strategy.desensitize(value) 即可。

4. hutool 与 RuoYi 自定义脱敏的分工

4.1 DesensitizedUtil（hutool）：标准化脱敏

hutool 的 DesensitizedUtil 提供了 11 种固定格式的脱敏方法，覆盖了国内企业最常见的敏感数据类型：

方法适用场景特点

4.2 DesensitizedUtils（RuoYi 自定义）：灵活掩码

DesensitizedUtils 继承自 hutool 的 DesensitizedUtil，扩展了两个关键方法：

public class DesensitizedUtils extends DesensitizedUtil {
    /**
     * 通用掩码脱敏
     * @param value         原始值
     * @param prefixVisible 开头保留字符数
     * @param suffixVisible 结尾保留字符数
     * @param maskLength    掩码长度（超出部分用 * 填充）
     */
    public static String mask(String value, int prefixVisible,
                              int suffixVisible, int maskLength);

    /**
     * 高安全掩码脱敏：中间全部掩码，只保留头尾
     * @param value         原始值
     * @param prefixVisible 开头保留字符数
     * @param suffixVisible 结尾保留字符数
     */
    public static String maskHighSecurity(String value,
                                          int prefixVisible,
                                          int suffixVisible);
}

三个通用策略对 mask() 的使用：

  • STRING_MASK → mask(s, 4, 4, 4)：保留前 4 后 4 字符，中间用 4 个 * 掩码。适合金额、数量等字符串。
  • FIRST_MASK → mask(s, 0, s.length()-1)：只保留最后一个字符，前面全部掩码。适合只展示尾号的场景。
  • MASK_HIGH_SECURITY → maskHighSecurity()：头尾各保留少量字符，中间完全掩码（掩码长度自动计算为中间字符数）。适合高安全场景。
架构思考：为什么 RuoYi 不把所有策略都用 DesensitizedUtils.mask() 实现？因为 hutool 的内置方法经过了大量生产验证，对电话号码、身份证、银行卡等格式有专门的边界处理（比如手机号自动识别 86 前缀）。直接复用这些久经考验的实现比自己写掩码逻辑更可靠。RuoYi 自定义的 DesensitizedUtils 仅用于 hutool 没有覆盖的通用场景，边界清晰、职责分明。

5. 小结

本课覆盖了 ruoyi-common-sensitive 模块的两个核心组件：

  • @Sensitive 注解：声明式标记，3 个属性支撑从简单脱敏到条件脱敏的全场景。
  • SensitiveStrategy 枚举：17 种即开即用的脱敏策略，底层复用了 hutool 的成熟实现和 RuoYi 自研的灵活掩码工具。

📝 课后测验（点击展开）
  1. @Sensitive 注解的 strategy() 属性为什么没有默认值？
    这是强制约束设计。如果提供了默认值，开发者可能忘记显式指定脱敏策略，导致字段以非预期的方式脱敏（或不脱敏）。要求必填确保开发者有意识地选择了脱敏方式，减少敏感数据泄露风险。
  2. roleKey 和 perms 同时配置时，脱敏的判断逻辑是什么？
    是 OR 关系。SensitiveService.isSensitive(roleKey[], perms[]) 的实现中，只要当前用户满足任一 roleKey 或任一 perms，返回 true 即执行脱敏。这意味着"任何一个条件命中就脱敏"，而不是"所有条件都满足才脱敏"。
  3. STRING_MASK 和 MASK_HIGH_SECURITY 的核心区别是什么？
    STRING_MASK 使用固定长度的掩码（4 个 *），保留前后指定字符数，掩码部分始终是 4 个 * 无论中间多长。
    MASK_HIGH_SECURITY 使用动态长度掩码，中间部分完全被 * 等量替换——如果中间有 8 个字符，就生成 8 个 *。这在高安全场景下不会泄露字符串的实际长度信息。
