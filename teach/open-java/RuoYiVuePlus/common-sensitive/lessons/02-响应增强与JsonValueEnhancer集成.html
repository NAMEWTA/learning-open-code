02 | 响应增强与 JsonValueEnhancer 集成：处理器注册与条件脱敏

上一课我们学习了 @Sensitive 注解和 17 种脱敏策略的定义层面。本课深入执行层——注解是如何被检测并处理的？JSON 序列化管道如何拦截字段？条件脱敏又如何与权限体系结合？答案在 SensitiveJsonFieldProcessor、SensitiveConfig 和 SensitiveService 之中。

1. SensitiveConfig：自动装配的入口

在 Spring Boot 的自动装配体系中，SensitiveConfig 是整个脱敏模块被激活的唯一入口。它极其简洁：

@AutoConfiguration
public class SensitiveConfig {

    @Bean
    public SensitiveJsonFieldProcessor sensitiveJsonFieldProcessor() {
        return new SensitiveJsonFieldProcessor(sensitiveService);
    }

    // sensitiveService 通过 @Autowired(required = false) 注入
}

关键设计点：

  • @AutoConfiguration：声明这是一个自动配置类，由 Spring Boot 的 spring.factories 或 org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件发现。不需要任何手动 @Import 或 @ComponentScan。
  • 唯一的 @Bean：整个模块只注册一个 Bean——SensitiveJsonFieldProcessor。这个处理器会被 ruoyi-common-json 模块的 JsonValueEnhancer 自动发现并纳入 JSON 增强管道。
  • SensitiveService 为可选依赖：如果项目中没有提供 SensitiveService 的实现，注入为 null，处理器仍然正常工作——只是对所有用户都执行脱敏。
设计哲学：SensitiveConfig 遵循了 Spring Boot 的"约定优于配置"原则。只要 ruoyi-common-sensitive 在 classpath 中，自动装配就生效；只要在实体类上加 @Sensitive 注解，脱敏就自动执行。对于 90% 的场景，零配置即可使用。

2. SensitiveJsonFieldProcessor：脱敏的执行引擎

SensitiveJsonFieldProcessor 实现了 JsonFieldProcessor 接口，是整个脱敏逻辑的执行器。它被标注为 @Order(100)，确保在 JSON 字段增强管道中的执行顺序靠后（保证其他处理器先完成字段值的转换，脱敏作为最后一步）。

2.1 supports() —— 判断是否需要脱敏

public boolean supports(JsonFieldContext fieldContext) {
    return fieldContext.getAnnotation(Sensitive.class) != null;
}

supports() 的逻辑极其简单：检查字段上是否有 @Sensitive 注解。JsonFieldContext 是一个 Java 14 record，它封装了字段的完整上下文信息：

属性类型说明
ownerObject字段所属的对象实例
propertyNameString字段名（Java 属性名）
memberMember字段的反射成员（Field 或 Method）
valueObject字段的当前值

getAnnotation(Class<A>) 是 JsonFieldContext 上最关键的方法——它从 member 上反射读取注解，避免每个处理器都需要手动编写反射代码。干净的 API 设计。

2.2 process() —— 执行脱敏

public Object process(Object value, JsonFieldContext fieldContext) {
    Sensitive sensitive = fieldContext.getAnnotation(Sensitive.class);
    SensitiveStrategy strategy = sensitive.strategy();

    // 条件检查：判断当前用户是否需要脱敏
    if (sensitiveService != null) {
        boolean shouldDesensitize = sensitiveService.isSensitive(
            sensitive.roleKey(),
            sensitive.perms()
        );
        if (!shouldDesensitize) {
            return value;  // 不需脱敏，原样返回
        }
    }

    // 执行脱敏
    if (value instanceof String) {
        return strategy.desensitize((String) value);
    }
    return value;
}

处理流程分三步：

  1. 获取策略：从注解中拿到 SensitiveStrategy 枚举值。
  2. 条件判断：如果 sensitiveService 不为 null，调用 isSensitive() 判断是否需要脱敏。不需要则直接返回原值。
  3. 执行脱敏：调用 strategy.desensitize(value)，底层委托给 hutool 的 DesensitizedUtil 或 RuoYi 的 DesensitizedUtils。
注意：process() 只处理 String 类型的值。如果字段是 Integer、Long 或其他类型，直接原样返回。这是合理的——脱敏本质上是字符串操作，数值类型的 ID 或编号不需要（也无法）用 * 替换。

3. JsonFieldProcessor 生命周期：理解 JSON 增强管道

SensitiveJsonFieldProcessor 实现的 JsonFieldProcessor 接口来自 ruoyi-common-json 模块，它定义了标准的 JSON 字段处理三阶段生命周期：

阶段方法调用时机用途
1. collectcollect(JsonFieldContext)序列化准备阶段，收集需要处理的字段翻译处理器用此阶段收集所有 @Translation 注解的字段，异步查询翻译数据
2. prepareprepare()所有字段收集完毕后，统一预处理翻译处理器在此阶段批量查询数据库，将翻译数据加载到缓存
3. processprocess(value, context)对每个字段执行最终处理脱敏处理器在此阶段直接对字段值做掩码替换

SensitiveJsonFieldProcessor 为什么只实现 supports() 和 process()？

因为脱敏是无状态、即时操作——它不需要预先收集数据（跳过 collect），也不需要批量查询（跳过 prepare）。传入一个字符串，立即返回脱敏后的结果。这是 JsonFieldProcessor 接口最简单的实现方式，也是该接口设计灵活性的体现：复杂的处理器（如翻译）利用全部三个阶段，简单的处理器（如脱敏）只用最后一步。

架构要点：JsonFieldProcessor 的三阶段设计体现了"收集-准备-执行"的批处理模式。翻译处理器需要先收集所有翻译注解、再批量查数据库（避免 N+1 查询）、最后逐一替换——三个阶段缺一不可。脱敏处理器因为操作足够简单（纯 CPU 计算），只需要最后的执行阶段。这种设计让简单的事简单做、复杂的事结构化做。

4. SensitiveService：条件脱敏的闸门

SensitiveService 是一个单一方法的接口：

public interface SensitiveService {
    /**
     * 判断当前请求是否需要脱敏
     * @param roleKey  注解上配置的角色 key 数组
     * @param perms    注解上配置的权限标识数组
     * @return true 需要脱敏，false 不需要脱敏
     */
    boolean isSensitive(String[] roleKey, String[] perms);
}

4.1 OR 关系详解

当 @Sensitive 同时配置了 roleKey 和 perms 时，判断逻辑是 OR：

// 伪代码示例
public boolean isSensitive(String[] roleKey, String[] perms) {
    // 获取当前用户的角色和权限
    Set<String> userRoles = getCurrentUserRoles();
    Set<String> userPerms = getCurrentUserPerms();

    // roleKey 命中 OR perms 命中 → 脱敏
    for (String rk : roleKey) {
        if (userRoles.contains(rk)) return true;
    }
    for (String p : perms) {
        if (userPerms.contains(p)) return true;
    }
    return false;  // 都不命中则不脱敏
}

4.2 实战场景：管理员看明文，普通用户看脱敏

假设系统有两个角色 admin 和 common。管理员应该看到完整手机号，普通用户只能看到脱敏后的数据。

// 实体类
public class UserVO {
    @Sensitive(strategy = SensitiveStrategy.PHONE, roleKey = {"common"})
    private String phone;
}

// SensitiveService 实现
@Component
public class DefaultSensitiveService implements SensitiveService {

    @Override
    public boolean isSensitive(String[] roleKey, String[] perms) {
        LoginUser loginUser = LoginHelper.getLoginUser();
        if (loginUser == null) return true;  // 未登录，默认脱敏

        // 检查 roleKey
        for (String rk : roleKey) {
            if (loginUser.getRoles().contains(rk)) {
                return true;  // 命中 common → 脱敏
            }
        }
        // admin 不匹配任何 roleKey → 不脱敏，看到明文
        return false;
    }
}

执行效果：

  • admin 用户请求 → isSensitive(["common"], []) → admin 不在 common 角色中 → 返回 false → 手机号原样输出：13812345678
  • common 用户请求 → isSensitive(["common"], []) → common 在角色列表中 → 返回 true → 手机号脱敏输出：138****5678
  • 未登录请求 → loginUser == null → 返回 true → 脱敏，保守策略
安全提示：默认策略应该是"未登录/无法判断时执行脱敏"——宁可多脱敏，不可少脱敏。这与安全领域的"默认拒绝"原则一致。如果你的实现中 loginUser == null 时返回 false，未登录用户可能看到本不该看到的明文数据。

5. 完整调用链路回顾

将两个课的内容串起来，一次完整的脱敏调用如下：

1. Controller 返回 UserVO {phone: "13812345678"}
        │
2. Jackson 序列化 UserVO 对象
        │
3. JsonValueEnhancer 拦截 phone 字段
        │
4. 遍历已注册的 JsonFieldProcessor 列表
        │
5. SensitiveJsonFieldProcessor.supports(fieldContext)
   → fieldContext.getAnnotation(Sensitive.class) != null? ✓
        │
6. SensitiveJsonFieldProcessor.process("13812345678", fieldContext)
   → 读取 @Sensitive(strategy=PHONE, roleKey={"common"})
   → sensitiveService.isSensitive(["common"], []) → true (当前用户是 common 角色)
   → SensitiveStrategy.PHONE.desensitize("13812345678")
        │
7. DesensitizedUtil.mobilePhone("13812345678")
        │
8. 返回 "138****5678"
        │
9. JSON 输出: {"phone": "138****5678", ...}

这条链路从 Controller 返回对象开始，经过 Jackson 序列化、JsonValueEnhancer 字段拦截、处理器链匹配、条件判断、策略选择、最终脱敏执行——9 步走完，开发者只需写一行 @Sensitive(strategy = SensitiveStrategy.PHONE)。

小结

本课覆盖了脱敏模块的执行层三个核心组件：

  • SensitiveConfig：零配置自动装配，一个 Bean 激活整个脱敏能力。
  • SensitiveJsonFieldProcessor：最小化的 JsonFieldProcessor 实现，仅用 supports + process 完成脱敏。
  • SensitiveService：灵活的权限闸门，让不同角色看到不同级别的数据。

结合上一课的注解和策略体系，ruoyi-common-sensitive 的完整架构已经清晰：它用 5 个文件、17 种策略、一个处理器，实现了企业级的数据脱敏——简洁而强大。

📝 课后测验（点击展开）
  1. 为什么 SensitiveJsonFieldProcessor 不需要实现 collect() 和 prepare() 方法？
    因为脱敏是无状态、即时、纯 CPU 计算的操作。它不需要像翻译处理器那样先收集注解字段（collect）、再批量查询数据库（prepare）、最后逐一替换（process）。脱敏只需拿到字符串和策略，立即返回结果。这体现了 JsonFieldProcessor 接口的灵活性——简单处理器只用 process()，复杂处理器用全部三个阶段。
  2. 如果项目中不提供 SensitiveService 的实现，脱敏还会工作吗？
    会。SensitiveService 的注入是 @Autowired(required = false)，当它为空时，SensitiveJsonFieldProcessor.process() 跳过条件检查，对所有用户的所有字段都执行脱敏。这是一种安全的默认行为——"全脱敏"比"不脱敏"更安全。
  3. process() 方法中，如果字段值是 Integer 类型会怎样？
    直接原样返回。process() 中有 if (value instanceof String) 的类型检查，非 String 类型不会被执行脱敏。这是合理的设计——像 userId 这样的数值型 ID 不适合用 * 替换。如果需要脱敏数值，应先在实体类中将字段声明为 String 类型。
