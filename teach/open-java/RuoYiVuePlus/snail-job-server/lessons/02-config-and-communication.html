第2课：配置体系与通信模型

本课目标：逐段拆解 application.yml + application-dev.yml 的核心配置项，理解服务端的双端口通信模型、MyBatis-Plus 持久化策略，以及 Dockerfile 的 JVM 调优方案。学完本课，你能独立看懂并调整 SnailJob 服务端的所有关键配置。

1. application.yml：全局配置骨架

1.1 服务端口与上下文路径

server:
  port: 8800
  servlet:
    context-path: /snail-job

服务端 HTTP 运行在 8800 端口，所有 HTTP 请求前缀为 /snail-job。这意味着 Dashboard 管理界面访问地址为 http://localhost:8800/snail-job/。

注意：8800 是 HTTP 端口（提供 Dashboard 管理界面和 API），不是客户端连接端口。客户端通过 17888 Netty 端口与服务端通信（见下文）。

1.2 静态资源托管

spring:
  web:
    resources:
      static-locations: classpath:admin/

Dashboard 管理后台的静态页面、JS、CSS 由 snail-job-server-starter 内置，存放在 classpath:admin/ 下。Spring Boot 从此路径加载前端资源，无需额外配置 Nginx。

1.3 MyBatis-Plus 持久化

mybatis-plus:
  typeAliasesPackage: com.aizuda.snailjob.template.datasource.persistence.po
  global-config:
    db-config:
      where-strategy: NOT_EMPTY
      logic-delete-value: 1
      logic-not-delete-value: 0
  configuration:
    map-underscore-to-camel-case: true

关键点是 typeAliasesPackage 指向 SnailJob 框架内置的 PO 实体包——不是 RuoYi 项目自己写的类。这些 PO 对应 SnailJob 的调度元数据表：任务定义、重试记录、调度配置、工作流 DAG、节点信息等。服务端通过这些 PO 管理 MySQL 中的调度元数据。

设计意图：SnailJob 框架将持久化层放在 snail-job-template 模块中（服务端和客户端共享），通过 MyBatis-Plus 自动映射。RuoYi 项目只需配置好数据源，SnailJob 启动时会自动创建/更新所需的数据库表。

2. application-dev.yml：环境专属配置

2.1 数据源

spring:
  datasource:
    type: com.zaxxer.hikari.HikariDataSource
    url: jdbc:mysql://localhost:3306/ry-vue?...
    hikari:
      minimum-idle: 10
      maximum-pool-size: 20
      idle-timeout: 600000
      max-lifetime: 900000

使用 HikariCP 连接池，连接到与主应用相同的 ry-vue 数据库。SnailJob 的调度元数据表与 RuoYi 业务表共存于同一数据库，通过表名前缀（如 sj_）区分。

2.2 SnailJob 服务端核心配置（逐项解读）

配置项值含义
server-port 17888 Netty 通信端口。客户端（ruoyi-common-job）通过此端口上报心跳、任务执行结果、远程日志
retry-pull-page-size 100 每批次从数据库拉取的重试任务条数。值越大吞吐越高，但单次查询耗时增加
log-storage 7 任务执行日志保留天数，超期自动清理
merge-log-days 1 合并日志的天数窗口（将多条日志合并为一条保存）
merge-log-num 500 合并日志的最大条数阈值
bucket-total 128 分桶总数。所有任务被哈希分配到 128 个桶，每个桶由一个 Akka Actor 负责调度。影响负载均衡粒度
load-balance-cycle-time 10 (秒) 负载均衡周期。每 10 秒检查一次各桶的负载，在服务端节点间重新分配
retry-max-pull-parallel 2 重试任务拉取的并行度（并发拉取线程数）
summary-day 7 Dashboard 仪表盘数据容错天数
面试常见问题："bucket-total 128 是什么意思？" —— 它是 SnailJob 分布式调度的核心概念。所有任务按一致性哈希分散到 128 个桶中，每个桶对应一个 Akka Actor。服务端集群中的节点通过 Actor 协议协商各自负责哪些桶，从而实现无中心化的负载均衡。桶数越大，负载分配越均匀，但 Actor 调度开销也越大。128 是官方推荐的平衡点。

2.3 Spring Boot Admin 客户端

spring.boot.admin.client:
  enabled: true
  url: http://localhost:9090/admin
  instance:
    metadata:
      username: ${spring.boot.admin.client.username}
      userpassword: ${spring.boot.admin.client.password}
  username: @monitor.username@
  password: @monitor.password@

将本服务端注册到 Spring Boot Admin 监控中心（默认 localhost:9090/admin）。metadata 中的用户名密码用于 Admin Server 回调 Actuator 端点时的 Basic Auth 认证（详见第3课）。

3. 双端口通信模型

SnailJob 服务端同时暴露两个端口，各司其职：

┌─────────────────────────────────┐ │ SnailJob 服务端 │ │ │ 浏览器 / Admin ───│─► :8800 HTTP (Tomcat) │ │ ├── Dashboard 管理界面 │ │ ├── REST API │ │ └── Actuator 健康端点 │ │ │ ruoyi-common-job ─│─► :17888 TCP (Netty) │ (客户端) │ ├── 心跳上报 │ │ ├── 任务执行结果回传 │ │ ├── 远程日志 (LogbackAppender) │ │ └── 任务分派指令接收 │ └─────────────────────────────────┘

Dockerfile 中的双端口暴露

FROM bellsoft/liberica-openjdk-rocky:21.0.11-cds

# 两个端口同时暴露
EXPOSE 8800
EXPOSE 17888

ENTRYPOINT java \
           -Djava.security.egd=file:/dev/./urandom \
           -XX:+HeapDumpOnOutOfMemoryError \
           -XX:+UseZGC \          ← 低延迟 ZGC
           ${JAVA_OPTS} \
           -jar app.jar

Dockerfile 使用 BellSoft Liberica JDK 21 + ZGC（低延迟垃圾收集器），这对调度系统（对延迟敏感）很重要。两个 EXPOSE 指令确保容器化部署时 8800 和 17888 端口均可被外部访问。
