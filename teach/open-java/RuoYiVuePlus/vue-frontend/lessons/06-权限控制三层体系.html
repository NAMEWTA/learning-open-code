06 权限控制三层体系

RuoYi-Vue-Plus 实现了从前端到后端的三层权限防线：路由级（控制页面访问）、按钮级（控制 UI 元素显示）、函数级（在逻辑代码中判断权限）。每层都有超级管理员的绕过逻辑。学完本课后，你将能在项目中正确使用三层权限控制手段，并理解它们的实现原理。

第一层：路由级权限

路由级权限通过两个机制实现：filterDynamicRoutes（本地路由过滤）和后端菜单下发（后端路由过滤）。

// 本地路由过滤 (permission store)
export const filterDynamicRoutes = (routes: RouteRecordRaw[]) => {
  const res: RouteRecordRaw[] = [];
  routes.forEach(route => {
    if (route.permissions) {
      // 按 permissions 字段过滤
      if (auth.hasPermiOr(route.permissions)) {
        res.push(route);
      }
    } else if (route.roles) {
      // 按 roles 字段过滤
      if (auth.hasRoleOr(route.roles)) {
        res.push(route);
      }
    }
  });
  return res;
};

本地路由定义方式：

// router/index.ts 中的 dynamicRoutes
export const dynamicRoutes: RouteRecordRaw[] = [
  {
    path: '/monitor',
    component: Layout,
    permissions: ['monitor:logininfor:list'], // 需要此权限才可访问
    children: [...]
  }
];

后端菜单下发是主要方式——/system/menu/getRouters 返回的菜单树已经按用户角色过滤，前端只需要直接注册。双重过滤确保即使前端路由注册了，后端不会返回该用户无权访问的数据。

第二层：按钮级权限（v-hasPermi / v-hasRoles 指令）

指令通过 DOM 移除 实现无权限按钮的隐藏。核心代码：

// src/directive/permission/index.ts
export const hasPermi: Directive = {
  mounted(el: HTMLElement, binding: DirectiveBinding) {
    const { permissions } = useUserStore();
    const { value } = binding; // 如 ['system:user:add', 'system:user:edit']

    if (value && value instanceof Array && value.length > 0) {
      const hasPermission = permissions.some(permi => {
        return permi === '*:*:*'            // 超级管理员拥有所有权限
            || value.includes(permi);       // 只要满足其中一个权限即可
      });
      if (!hasPermission) {
        el.parentNode && el.parentNode.removeChild(el); // 无权限 → 从 DOM 移除
      }
    }
  }
};

export const hasRoles: Directive = {
  mounted(el: HTMLElement, binding: DirectiveBinding) {
    const { value } = binding; // 如 ['admin', 'test']
    const { roles } = useUserStore();

    if (value && value instanceof Array && value.length > 0) {
      const hasRole = roles.some(role => {
        return role === 'superadmin'
            || role === 'admin'              // admin 角色也有所有权限
            || value.includes(role);         // 满足其中一个角色即可
      });
      if (!hasRole) {
        el.parentNode && el.parentNode.removeChild(el);
      }
    }
  }
};

在模板中使用：

<!-- 权限指令：需要 system:user:add 权限 -->
<el-button v-hasPermi="['system:user:add']">新增</el-button>

<!-- 角色指令：需要 admin 或 common 角色 -->
<el-button v-hasRoles="['admin', 'common']">管理操作</el-button>
DOM 移除 vs display:none： 指令选择 removeChild 而非 el.style.display = 'none'，因为移除的 DOM 节点无法被用户通过浏览器开发者工具恢复显示，安全性更高。但这也意味着指令只在 mounted 时执行一次——如果权限在运行时发生变化（如切换账号），需要重新挂载组件才能生效。

第三层：函数级权限（$auth 全局方法）

auth 插件提供 6 个方法，可在组件逻辑中程序化判断权限：

// src/plugins/auth.ts
export default {
  // 验证用户是否具备某个权限（精确匹配）
  hasPermi(permission: string): boolean {
    const all_permission = '*:*:*';
    const permissions = useUserStore().permissions;
    return permissions.some(v => all_permission === v || v === permission);
  },

  // 验证用户是否含有指定权限，只需包含其中一个
  hasPermiOr(permissions: string[]): boolean {
    return permissions.some(item => authPermission(item));
  },

  // 验证用户是否含有指定权限，必须全部拥有
  hasPermiAnd(permissions: string[]): boolean {
    return permissions.every(item => authPermission(item));
  },

  // 验证用户是否具备某个角色
  hasRole(role: string): boolean {
    const super_admin = 'admin';
    const roles = useUserStore().roles;
    return roles.some(v => super_admin === v || v === role);
  },

  // 验证用户是否含有指定角色，只需包含其中一个
  hasRoleOr(roles: string[]): boolean { ... },

  // 验证用户是否含有指定角色，必须全部拥有
  hasRoleAnd(roles: string[]): boolean { ... }
};

在组件中使用（两种方式）：

// 方式1: Options API 通过全局属性
this.$auth.hasPermi('system:user:edit')

// 方式2: Composition API 直接导入
import auth from '@/plugins/auth';
if (auth.hasPermi('system:user:add')) {
  // 执行新增操作
}

// 批量导出场景
const columns = [
  { label: '操作', visible: auth.hasPermiOr(['system:user:edit', 'system:user:remove']) }
];
与 React 对比： React 中按钮级权限通常通过自定义 Hook（如 usePermission()）返回布尔值，结合条件渲染（{hasPerm('xxx') && <Button>}）实现。Vue 的 v-hasPermi 指令方案将权限判断从 JSX 逻辑中抽离到指令层，模板更干净，但灵活性（如动态权限变化）不如 Hook 方案。

超级管理员绕过逻辑汇总

系统中权限判断的超级管理员逻辑分布在三个地方，必须保持一致：

// 1. v-hasPermi 指令 → permi === '*:*:*'
// 2. v-hasRoles 指令 → role === 'superadmin' || role === 'admin'
// 3. auth 插件：
//    hasPermi → v === '*:*:*'
//    hasRole  → v === 'admin'  (注意：这里没有 superadmin!)
不一致风险： auth.hasRole 只检查 admin，不检查 superadmin；而 v-hasRoles 指令同时检查 admin 和 superadmin。这是历史遗留的设计差异，如果后端为某用户分配了 superadmin 角色而没有 admin 角色，通过 $auth.hasRole('admin') 会返回 false，这是一个已知的不一致点。

权限判断流程图

用户请求访问 /system/user
  │
  ├─ 路由守卫 beforeEach
  │   └─ 检查 token → 拉取 roles/permissions → 生成路由
  │       └─ 如果该路由不在 permission store 的 sidebarRouters 中
  │           → 404 或配置的 fallback 页面
  │
  ├─ 页面组件渲染
  │   ├─ v-hasPermi="['system:user:add']" → 检查 permissions 数组 → 移除或保留 DOM
  │   └─ v-hasRoles="['admin']" → 检查 roles 数组 → 移除或保留 DOM
  │
  └─ 组件逻辑中
      └─ auth.hasPermi('system:user:edit') → 返回布尔值 → 条件分支
最佳实践：
1. 路由级：用后端菜单控制，无需在前端 hardcode 权限字符串
2. 按钮级：优先用 v-hasPermi（以权限字符为准，比角色更细粒度）
3. 函数级：在动态构建配置对象（如表头）时使用 auth.hasPermiOr
4. 切勿在前端权限判断中做安全校验——真正的安全在后端接口层
