04 Axios 封装与接口加密

RuoYi-Vue-Plus 的 Axios 封装不仅是简单的请求库包装——它实现了一套完整的企业级安全传输方案：Token 注入、RSA+AES 混合加密、防重复提交、响应解密、统一错误处理、401 过期弹窗。学完本课后，你将能逐行解释 request.ts 中的请求/响应拦截器链。

Axios 实例创建

// 创建 axios 实例（不是 axios 全局单例）
const service = axios.create({
  baseURL: import.meta.env.VITE_APP_BASE_API, // 开发: /dev-api  生产: /prod-api
  timeout: 50000,                             // 50 秒超时
  transitional: {
    clarifyTimeoutError: true                 // Axios 1.x+ 清晰超时错误
  }
});

所有 API 请求都通过这个实例发出。自定义实例允许配置独立的拦截器链，不与第三方库共享。

请求拦截器：三层安全检查

请求拦截器处理三个关键 header 标志：isToken、repeatSubmit、isEncrypt：

service.interceptors.request.use(
  (config) => {
    // 【第1层】Token 注入
    const isToken = config.headers?.isToken === false;
    if (getToken() && !isToken) {
      config.headers['Authorization'] = 'Bearer ' + getToken();
    }

    // 【第2层】GET 请求参数扁平化映射
    if (config.method === 'get' && config.params) {
      let url = config.url + '?' + tansParams(config.params);
      url = url.slice(0, -1);
      config.params = {};
      config.url = url;
    }

    // 【第3层】防止重复提交（仅 POST/PUT）
    if (!isRepeatSubmit && (config.method === 'post' || config.method === 'put')) {
      const requestObj = {
        url: config.url,
        data: JSON.stringify(config.data),
        time: new Date().getTime()
      };
      const sessionObj = cache.session.getJSON('sessionObj');
      if (sessionObj && s_data === requestObj.data
          && requestObj.time - s_time < 500      // 500ms 间隔
          && s_url === requestObj.url) {
        return Promise.reject(new Error('数据正在处理，请勿重复提交'));
      }
      cache.session.setJSON('sessionObj', requestObj);
    }

    // 【第4层】参数加密（见下一节）
    if (isEncrypt && (config.method === 'post' || config.method === 'put')) {
      // RSA 加密 AES 密钥 + AES 加密请求体
    }

    return config;
  }
);
防重复提交原理： 使用 sessionStorage 存储最近一次请求的 (url, data, time)。若新请求在 500ms 内具有相同的 url 和 data，判定为重复提交而拒绝。当用户猛击"保存"按钮时，第二次点击会被拦截。注意：repeatSubmit: false 是关闭防重复（相当于"本接口允许重复提交"），语义上与名字相反。

RSA + AES 混合加密方案

当 VITE_APP_ENCRYPT=true 且接口声明 isEncrypt: true 时，请求/响应体经过如下处理：

// ====== 加密（crypto.ts）======

// 1. 生成一个随机的 32 位 AES 密钥
export const generateAesKey = (): WordArray => {
  const array = new Uint8Array(32);
  crypto.getRandomValues(array);                  // Web Crypto API 真随机数
  return CryptoJS.enc.Utf8.parse(
    Array.from(array, b => b.toString(16).padStart(2, '0'))
      .join('').slice(0, 32)
  );
};

// 2. AES-ECB + PKCS7 加密请求体
export const encryptWithAes = (message, aesKey): string => {
  const encrypted = CryptoJS.AES.encrypt(message, aesKey, {
    mode: CryptoJS.mode.ECB,
    padding: CryptoJS.pad.Pkcs7
  });
  return encrypted.toString();
};

// 3. RSA 公钥加密 AES 密钥（jsencrypt.ts）
export const encrypt = (txt: string) => {
  const encryptor = new JSEncrypt();
  encryptor.setPublicKey(import.meta.env.VITE_APP_RSA_PUBLIC_KEY);
  return encryptor.encrypt(txt);
};

在请求拦截器中的组装逻辑：

// 请求拦截器中
if (isEncrypt && (config.method === 'post' || config.method === 'put')) {
  const aesKey = generateAesKey();                     // 生成 AES 密钥
  config.headers['encrypt-key'] = encrypt(
    encryptBase64(aesKey)                              // RSA 公钥加密 AES 密钥 → 放入 header
  );
  config.data = encryptWithAes(                        // AES 加密请求体
    JSON.stringify(config.data), aesKey
  );
}

在响应拦截器中的解密逻辑：

// 响应拦截器中
if (VITE_APP_ENCRYPT === 'true') {
  const keyStr = res.headers['encrypt-key'];     // 后端返回的加密 AES 密钥
  if (keyStr) {
    const base64Str = decrypt(keyStr);            // RSA 私钥解密 → Base64 字符串
    const aesKey = decryptBase64(base64Str);      // Base64 解码 → AES 密钥
    const decryptData = decryptWithAes(res.data, aesKey); // AES 解密响应体
    res.data = JSON.parse(decryptData);           // JSON 字符串 → 对象
  }
}
为什么是 RSA + AES 混合？ RSA 加密速度慢但适合加密小数据（如密钥），AES 加密速度快但密钥分发困难。混合方案中：AES 加密大体积的请求体，RSA 只加密 AES 的 32 字节密钥并通过 header 传输。每次请求生成新的随机 AES 密钥，即使截获一次请求的 AES 密钥也无法解密其他请求。

响应拦截器：统一错误码处理

service.interceptors.response.use(
  (res) => {
    const code = res.data.code || HttpStatus.SUCCESS; // 200
    const msg = res.data.msg || errorCode[code] || errorCode['default'];

    // 二进制数据直接返回（文件下载）
    if (res.request.responseType === 'blob' || res.request.responseType === 'arraybuffer') {
      return res.data;
    }

    if (code === 401) {
      // Token 过期 → 弹出重新登录对话框
      if (!isRelogin.show) {
        isRelogin.show = true;                         // 全局锁，防止重复弹窗
        ElMessageBox.confirm('登录状态已过期...', '系统提示', {
          confirmButtonText: '重新登录',
          cancelButtonText: '取消',
          type: 'warning'
        }).then(() => {
          useUserStore().logout().then(() => {
            router.replace({ path: '/login', query: { redirect: ... } });
          });
        }).catch(() => { isRelogin.show = false; });
      }
      return Promise.reject('无效的会话...');
    } else if (code === 500) {
      ElMessage({ message: msg, type: 'error' });
      return Promise.reject(createHandledError(msg));
    } else if (code === HttpStatus.WARN) {
      ElMessage({ message: msg, type: 'warning' });
      return Promise.reject(createHandledError(msg));
    } else if (code !== HttpStatus.SUCCESS) {
      ElNotification.error({ title: msg });
      return Promise.reject(createHandledError(msg));
    }
    return Promise.resolve(res.data);
  },
  async (error) => {
    // 网络错误 / 超时 / 状态码错误统一处理
    const message = (await extractErrorMessage(error)) || errorCode['default'];
    ElMessage({ message, type: 'error', duration: 5000 });
    error.isHandled = true;   // 标记为已处理，防止上层重复弹窗
    return Promise.reject(error);
  }
);
isRelogin 全局锁： 当 token 过期时，可能同时有多个请求返回 401。如果没有全局锁，用户会看到多个"重新登录"弹窗堆叠。isRelogin.show 是一个模块级变量，第一个 401 将其设为 true，后续 401 检测到已显示则跳过弹窗。

createHandledError 与 isHandledRequestError

在路由守卫中，当 getInfo() 请求失败时，如果错误已被响应拦截器处理过（弹了提示），就不应再弹一次：

// request.ts
function createHandledError(message) {
  const error = new Error(message);
  error.isHandled = true;
  return error;
}

export function isHandledRequestError(error) {
  return Boolean(error?.isHandled);
}

// permission.ts 路由守卫中
const [err] = await tos(useUserStore().getInfo());
if (err) {
  if (!isHandledRequestError(err)) {
    ElMessage.error(err.message); // 仅未处理错误才弹窗
  }
  return { path: '/' };
}

拦截器链全景图


请求出发
  │
  ├─ 1. Content-Language 注入
  ├─ 2. Authorization: Bearer <token>
  ├─ 3. GET params → URL query string
  ├─ 4. 防重复提交检查（sessionStorage）
  ├─ 5. RSA加密AES密钥 → encrypt-key header
  │    AES加密请求体
  │    FormData 去除 Content-Type
  ▼
发送请求 ───────────────────────────→ 服务器
                                      │
  ┌───────────────────────────────────┘
  ▼
响应返回
  │
  ├─ 6. RSA解密 encrypt-key → 得到AES密钥
  │    AES解密响应体 → JSON.parse
  ├─ 7. 二进制数据直接返回
  ├─ 8. code===401 → 重新登录弹窗（全局锁）
  ├─ 9. code===500 → ElMessage.error + createHandledError
  ├─ 10. code===WARN → ElMessage.warning + createHandledError
  ├─ 11. code!==SUCCESS → ElNotification.error
  └─ 12. code===SUCCESS → resolve(res.data)
