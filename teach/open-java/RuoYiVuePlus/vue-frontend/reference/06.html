速查 06：权限控制三层体系

三层权限总览

层级机制作用范围实现方式
第1层：路由级菜单过滤控制页面是否可访问后端菜单下发 + filterDynamicRoutes(本地路由)
第2层：按钮级DOM 移除控制按钮/元素是否显示v-hasPermi / v-hasRoles 自定义指令
第3层：函数级布尔判断逻辑代码中的权限分支$auth.hasPermi / hasPermiOr / hasPermiAnd / hasRole / hasRoleOr / hasRoleAnd

超级管理员绕过逻辑对比（已知不一致）

位置Permi 超级管理员Role 超级管理员
v-hasPermi 指令permi === '*:*:*'—
v-hasRoles 指令—role === 'superadmin' || role === 'admin'
auth.hasPermi()v === '*:*:*'—
auth.hasRole()—v === 'admin'（不含 superadmin！）

使用速查

// === 按钮级（模板中） ===
<el-button v-hasPermi="['system:user:add']">新增</el-button>
<el-button v-hasRoles="['admin', 'common']">管理</el-button>

// === 函数级（逻辑中） ===
// Options API
this.$auth.hasPermi('system:user:edit')
// Composition API
import auth from '@/plugins/auth';
auth.hasPermi('system:user:add')         // 精确匹配
auth.hasPermiOr(['a', 'b'])             // 满足其一
auth.hasPermiAnd(['a', 'b'])            // 全部满足
auth.hasRole('admin')                   // 精确匹配
auth.hasRoleOr(['admin', 'common'])     // 满足其一
auth.hasRoleAnd(['admin', 'common'])    // 全部满足

// === 路由级（本地路由定义） ===
{
  path: '/monitor',
  permissions: ['monitor:logininfor:list'],  // 需要此权限
  // 或
  roles: ['admin']                            // 需要此角色
}

权限判断流程

用户访问 /system/user │ ├── beforeEach → roles.length===0? → getInfo() → permissions[]/roles[] │ → generateRoutes() → 后端只返回有权限的菜单 │ → addRoute() → 无权路由不在路由表中 → 404 │ ├── 页面渲染 → v-hasPermi="['system:user:add']" │ → mounted() → permissions.some(p => value.includes(p) || p==='*:*:*') │ → 无权限 → parentNode.removeChild(el) │ └── 组件逻辑 → auth.hasPermi('system:user:edit') → userStore.permissions.some(v => v === permission || v === '*:*:*') → 返回 true/false → if 分支

关键知识点

知识点说明
DOM 移除 removeChild比 display:none 更安全，用户无法通过 DevTools 恢复
时机 mounted 一次性指令只在 mounted 执行，权限变化需重新挂载组件
不一致 superadminv-hasRoles 检查 superadmin，auth.hasRole 不检查，历史遗留
最佳实践 后端校验前端权限仅控制 UI 展示，真正的安全校验在后端接口层
注意：切勿在前端做安全校验。v-hasPermi 移除的 DOM 在 mounted 后再也无法恢复（除非组件重新挂载）。如需要动态权限变化，考虑用 v-if + computed 替代指令方案。
