类型定义位置何时注入示例

路由守卫核心流程

router.beforeEach(to, from) ├── getToken() 存在? │ ├── YES: to.path === '/login'? → redirect '/' │ │ roles.length === 0? │ │ ├── YES → getInfo() → generateRoutes() → addRoute() │ │ │ → return { path: to.path, replace: true } (hack 重匹配) │ │ └── NO → return true (直接放行) │ └── NO: whiteList.includes(to.path)? │ ├── YES → return true │ └── NO → redirect /login?redirect=xxx

generateRoutes 6 步

1. getRouters() → 获取后端菜单树
2. structuredClone(data) ×3 → sdata(侧边栏) / rdata(注册路由) / defaultData(顶栏)
3. filterAsyncRouter(sdata) → 侧边栏菜单
4. filterAsyncRouter(rdata, ..., true) → 完整路由（含组件重写）
5. filterAsyncRouter(defaultData) → 顶栏菜单
6. filterDynamicRoutes(dynamicRoutes) → 本地权限路由 addRoute
7. setRoutes / setSidebarRouters / setTopbarRoutes → 存入 store
8. duplicateRouteChecker → name 重复检测

关键知识点

知识点说明
Hack addRoute 重匹配addRoute() 不会触发当前导航重新匹配，需 return { path: to.path, replace: true }
克隆 structuredClone ×3同一份菜单数据生成三个视图：侧边栏菜单、注册路由、顶栏菜单，需独立深拷贝
Vite import.meta.glob构建时静态分析，等价 webpack 的 require.context
KeepAlive createCustomNameComponent异步组件无 name → 壳层组件注入 name → keep-alive 的 include 匹配
检测 duplicateRouteCheckername 重复导致 keep-alive 缓存混乱 → 404
坑点：本地 dynamicRoutes 和后端菜单的 name 冲突 → 404。用 duplicateRouteChecker 在开发时早发现。
