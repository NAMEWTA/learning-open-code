01 | 三条安全防线：SecurityConfig 的一站式装配

本节目标：理解 SecurityConfig 如何通过三层防御体系构建应用级安全——上下文过滤器保证 Sa-Token 环境可达、路由拦截器实现登录与客户端鉴权、actuator 过滤器保护运维端点。

1. 全景鸟瞰：三条防线与模块分工

SecurityConfig 是 ruoyi-common-security 模块的核心配置类。它扮演"安全装配工"的角色——本身不实现认证逻辑，而是将 Sa-Token 提供的原语组装成面向应用的安全防线。理解这条装配线，首先要分清两个模块的职责：

模块角色提供的关键能力
ruoyi-common-satoken 能力层 / 基础设施 LoginHelper（客户端常量、登录工具方法）、StpUtil（Sa-Token 登录/鉴权 API）、SaTokenConfiguration（Sa-Token 框架的自动装配）
ruoyi-common-security 装配层 / 应用配置 SecurityConfig（组装三条防线）、AllUrlHandler（收集全量 URL）、SecurityProperties（排除路径配置）
一句话关系：ruoyi-common-satoken 提供积木（LoginHelper 常量、StpUtil 工具），ruoyi-common-security 搭积木（把 filter、interceptor、router 组装成针对本应用的防线）。

SecurityConfig 内部配置了三条防线，按请求到达顺序依次是：

防线组件负责什么执行时机
第一道Sa-Token 上下文过滤器（重注册版）确保 SaHolder / StpUtil 在 ASYNC/ERROR 场景下也能获取请求上下文Filter 链（最早）
第二道SaInterceptor 路由拦截器登录校验 + 客户端 ID 一致性校验 + 客户端访问规则校验Spring Interceptor 链
第三道Actuator Basic Auth 过滤器对 /actuator/** 路径施加 HTTP Basic 认证Filter 链（路径匹配）

2. 第一道防线：Sa-Token 上下文过滤器的重注册

2.1 为什么要"重注册"？

Sa-Token 框架在自动装配时已经注册了一个 SaTokenContextFilterForJakartaServlet（上下文过滤器），它负责在每次请求开始时将 HttpServletRequest / HttpServletResponse 绑定到 Sa-Token 的线程上下文中。这样后续代码调用 StpUtil.checkLogin() 时，Sa-Token 才能从上下文中拿到当前请求的 token。

但官方自动注册的过滤器有一个关键缺陷：它默认只在 DispatcherType.REQUEST 下生效。而在以下场景中，Servlet 容器会发起额外的分发：

DispatcherType触发场景为什么需要上下文
ASYNCSSE（Server-Sent Events）推送、WebSocket 握手升级、AsyncContext 异步请求异步线程池中的任务可能调用 StpUtil.getLoginId() 获取当前用户信息，没有上下文会抛 NPE
ERROR请求处理过程中抛出未捕获异常，容器转发到 /error错误页面或全局异常处理器中可能需要读取当前用户信息用于日志记录

2.2 FilterRegistrationBean 的配置细节

SecurityConfig 通过 Spring 的 FilterRegistrationBean API 重新注册上下文过滤器，覆盖了官方默认行为：

@Bean
public FilterRegistrationBean<SaTokenContextFilterForJakartaServlet> saTokenContextFilterRegistration(
    SaTokenContextFilterForJakartaServlet filter) {
    FilterRegistrationBean<...> registration = new FilterRegistrationBean<>();
    registration.setFilter(filter);
    registration.setName("saTokenContextFilterForServlet");
    registration.addUrlPatterns("/*");                           // 拦截所有路径
    registration.setDispatcherTypes(EnumSet.of(
        DispatcherType.REQUEST,
        DispatcherType.ASYNC,    // 关键：覆盖异步分发
        DispatcherType.ERROR     // 关键：覆盖错误分发
    ));
    registration.setAsyncSupported(true);                        // 开启异步支持
    registration.setOrder(Ordered.HIGHEST_PRECEDENCE);           // 最高优先级
    return registration;
}

逐行解读：

  • addUrlPatterns("/*") — 全局拦截，确保所有请求都经过上下文绑定
  • setDispatcherTypes(EnumSet.of(REQUEST, ASYNC, ERROR)) — 这是重注册的核心目的。原本只有 REQUEST，现在加上 ASYNC 和 ERROR，上下文在所有分发类型下都可用
  • setAsyncSupported(true) — 向容器声明本过滤器支持异步处理，否则 ASYNC 分发也不会触发
  • setOrder(HIGHEST_PRECEDENCE) — 将优先级提到最高，保证在所有其他过滤器和拦截器之前执行
设计洞察：这本质上是 Sa-Token 在 Spring Boot 自动装配下的一个兼容方案。Spring Boot 3.x 使用 Jakarta Servlet，SaTokenContextFilterForJakartaServlet 已经是适配版本。但 Sa-Token 的自动装配无法预知应用是否有 SSE/WebSocket 需求，所以默认只处理 REQUEST。SecurityConfig 通过 FilterRegistrationBean 的"覆盖注册"机制，用更全面的配置替换了默认行为。

3. 第二道防线：SaInterceptor 路由拦截器

3.1 addInterceptors 整体结构

SecurityConfig 实现 WebMvcConfigurer，在 addInterceptors() 中注册核心鉴权拦截器：

@Override
public void addInterceptors(InterceptorRegistry registry) {
    registry.addInterceptor(new SaInterceptor(handler -> {
        AllUrlHandler allUrlHandler = SpringUtils.getBean(AllUrlHandler.class);
        SaRouter
            .match(allUrlHandler.getUrls())   // 获取全量 URL 进行匹配
            .check(() -> {
                // ... 鉴权逻辑 ...
            });
    })).addPathPatterns("/**")
       .excludePathPatterns(securityProperties.getExcludes());
}

理解这段代码的骨架：

  • 外层 — addPathPatterns("/**") 声明对所有路径生效，excludePathPatterns() 排除配置中指定的路径（如登录接口、验证码接口等）
  • 内层 — SaRouter.match(urls).check(handler) 是 Sa-Token 的路由匹配 API。match() 检查当前请求是否命中全量 URL 列表中的一个，命中则执行 check() 里的鉴权逻辑

3.2 鉴权链：三道检查

check() 内部顺序执行三道校验，任何一道失败都会抛出异常中断请求：

检查一：登录校验

StpUtil.checkLogin();

这是最基础的检查——当前请求是否携带了有效 token。调用的是 Sa-Token 的标准 API，内部会从请求头/参数/Cookie 中提取 token 并验证有效性。未登录时抛出 NotLoginException。

检查二：客户端 ID 一致性校验

String headerCid = request.getHeader(LoginHelper.CLIENT_KEY);       // 请求头中的 clientid
String paramCid = ServletUtils.getParameter(LoginHelper.CLIENT_KEY); // URL 参数中的 clientid
String clientId = StpUtil.getExtra(LoginHelper.CLIENT_KEY).toString(); // Token 中存储的 clientid

if (!StringUtils.equalsAny(clientId, headerCid, paramCid)) {
    throw NotLoginException.newInstance(StpUtil.getLoginType(),
        "-100", "客户端ID与Token不匹配", StpUtil.getTokenValue());
}

这段代码做了一个巧妙的三重校验：token 签发时的客户端 ID（存储在 token 的 extra 字段）必须与当前请求携带的 clientId 一致。clientId 可以从请求头或 URL 参数中获取——两者只要有一个匹配即可（StringUtils.equalsAny() 语义）。

为什么需要这个检查？防止 token 跨客户端滥用。例如，用户 A 在 Web 端登录获得的 token，不应该能用于移动端 API 调用（即使 token 本身有效），因为签发时的 clientId 和当前请求的 clientId 不匹配。

检查三：客户端访问规则校验

调用私有方法 validateClientAccessRules(request)，内部又包含两个子检查：

  • 路径白名单校验 — 从 token 的 clientAccessPath extra 字段读取该客户端允许访问的路径列表，当前请求路径必须命中列表中的一条
  • IP 白名单校验 — 从 token 的 clientIpWhitelist extra 字段读取该客户端允许的 IP 范围，当前请求来源 IP 必须命中

具体实现细节将在下一节课 "URL 扫描与客户端鉴权" 中深入讲解。

4. 第三道防线：Actuator Basic Auth 过滤器

生产环境的 actuator 端点（如 /actuator/health、/actuator/env）暴露了应用内部状态，必须加保护。SecurityConfig 通过一个独立的 SaServletFilter 来实现：

@Bean
public SaServletFilter getSaServletFilter() {
    String username = SpringUtils.getProperty("spring.boot.admin.client.username");
    String password = SpringUtils.getProperty("spring.boot.admin.client.password");
    return new SaServletFilter()
        .addInclude("/actuator", "/actuator/**")                   // 只拦截 actuator 路径
        .setAuth(obj -> {
            SaHttpBasicUtil.check(username + ":" + password);      // HTTP Basic Auth 校验
        })
        .setError(e -> {
            response.setContentType("application/json");
            return SaResult.error(e.getMessage()).setCode(HttpStatus.UNAUTHORIZED);
        });
}

关键点：

  • 账号来源 — 用户名和密码从 spring.boot.admin.client.username / spring.boot.admin.client.password 配置中读取。这说明这些凭证是给 Spring Boot Admin 客户端用的，actuator 端点的保护天然复用了 SBA 的配置，不需要额外设置
  • SaHttpBasicUtil.check() — Sa-Token 提供的 HTTP Basic 认证工具，会从请求头 Authorization: Basic xxx 中提取并校验凭证
  • addInclude() — 精确限定拦截范围，只对 actuator 相关路径生效，不影响业务接口的 token 鉴权流程
  • setError() — 认证失败时返回 JSON 格式的 401 Unauthorized，而不是默认的 HTML 错误页，符合 RESTful API 风格
设计洞察：第三条防线与前两条完全独立。它不需要 token、不经过 SaInterceptor 链、不依赖客户端配置——只靠一对固定的用户名/密码。这是因为 actuator 端点的访问者通常是运维工具（如 Prometheus、Spring Boot Admin），而非业务用户。把运维认证和业务认证分开，是最小化攻击面的经典做法。

5. 三条防线的协作关系

一个典型的业务请求经过的完整链路如下：

请求到达
  │
  ▼
【第一道防线】SaTokenContextFilterForJakartaServlet
  绑定 HttpServletRequest/Response 到 Sa-Token 上下文
  （DispatcherTypes: REQUEST + ASYNC + ERROR）
  │
  ▼
【路径匹配】Spring 分发到具体 Handler
  │
  ▼
【第二道防线】SaInterceptor（仅对非排除路径生效）
  ① StpUtil.checkLogin()                 → 未登录？拒绝
  ② clientId 一致性校验                   → 不匹配？拒绝
  ③ validateClientAccessRules()          → 路径/IP不在白名单？拒绝
  │
  ▼
Controller 执行业务逻辑
  │
  ▼
响应返回

而 actuator 请求走的是不同的路径——它被第三道防线的 SaServletFilter 拦截（路径匹配 /actuator/**），只做 Basic Auth 校验，不走 SaInterceptor。
