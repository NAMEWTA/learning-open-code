02 | URL 扫描与客户端鉴权：AllUrlHandler 的自动收集与客户端校验

本节目标：理解 AllUrlHandler 如何在 Spring 启动时自动收集全量 URL、为什么 InitializingBean 是唯一正确的时机、收集到的 URL 如何驱动 Sa-Token 的路由匹配，以及客户端鉴权的完整校验链路。

1. 为什么需要"扫描全量 URL"？

回顾上一节课，SecurityConfig 在 SaInterceptor 中调用 SaRouter.match(allUrlHandler.getUrls()).check(...)。这里的 match() 需要知道系统中所有合法的请求路径才能做匹配。一个自然的想法是手动维护一个 URL 列表——但这在项目变大后极易遗漏。

AllUrlHandler 解决的就是这个问题：自动发现 Controller 上所有 @RequestMapping 及其变体（@GetMapping、@PostMapping 等）标注的路径，运行时零配置。

2. InitializingBean：唯一正确的收集时机

2.1 Spring 容器的启动时序

要理解为什么 afterPropertiesSet() 是"唯一正确"的时机，需要先了解 Spring 容器在启动时的几个关键阶段：

阶段发生了什么RequestMappingHandlerMapping 状态
Bean 实例化构造器调用，依赖注入可能尚未完成扫描（取决于 Bean 创建顺序）
@PostConstructJSR-250 生命周期回调不确定——依赖于 Bean 后处理器顺序，RequestMappingHandlerMapping 可能还没完成 afterSingletonsInstantiated()
InitializingBean.afterPropertiesSet()Spring 生命周期回调（在所有属性注入完成后）通常已完成——Spring 保证在调用 InitializingBean 之前完成所有依赖注入和 Aware 回调，且 RequestMappingHandlerMapping 通常在此时已扫描完所有 Controller
SmartInitializingSingleton.afterSingletonsInstantiated()所有单例 Bean 实例化后保证完成——这是最安全的时机，但 AllUrlHandler 没有实现这个接口
设计洞察：AllUrlHandler 选择 InitializingBean 而非 SmartInitializingSingleton，说明它对启动时序有一定信心——它的 Bean 定义靠后创建（没有 @Component 注解，依赖显式注册），此时 RequestMappingHandlerMapping 通常已完成扫描。但如果极端场景下 AllUrlHandler 比 RequestMappingHandlerMapping 更早初始化，收集到的 URL 列表可能为空。这也是为什么代码中用 SpringUtils.getBean("requestMappingHandlerMapping") 而非构造注入——在 afterPropertiesSet 时才去拿，降低了对 Bean 创建顺序的依赖。

2.2 为什么不在构造函数或 @PostConstruct 中收集？

两种方式都有可能导致拿到不完整的 URL 列表：

  • 构造函数中收集：依赖注入还没完成，SpringUtils.getBean() 可能因为循环依赖或其他 Bean 未就绪而失败
  • @PostConstruct 中收集：虽然依赖注入已完成，但 RequestMappingHandlerMapping 的 afterSingletonsInstantiated() 回调可能还没执行，Controller 的映射注册可能尚未完全就绪

afterPropertiesSet() 处于两者之间，在大多数 Spring Boot 场景下是足够安全的。

3. AllUrlHandler 的实现逐行解析

@Data
public class AllUrlHandler implements InitializingBean {

    private static final Pattern PATTERN = Pattern.compile("\\{(.*?)\\}");

    private List<String> urls = new ArrayList<>();

    @Override
    public void afterPropertiesSet() {
        Set<String> set = new HashSet<>();
        RequestMappingHandlerMapping mapping = SpringUtils.getBean(
            "requestMappingHandlerMapping", RequestMappingHandlerMapping.class);
        Map<RequestMappingInfo, HandlerMethod> map = mapping.getHandlerMethods();
        map.keySet().forEach(info -> {
            Objects.requireNonNull(info.getPathPatternsCondition().getPatterns())
                .forEach(url -> set.add(
                    ReUtil.replaceAll(url.getPatternString(), PATTERN, "*")
                ));
        });
        urls.addAll(set);
    }
}

逐行解读：

3.1 获取 RequestMappingHandlerMapping

RequestMappingHandlerMapping mapping = SpringUtils.getBean(
    "requestMappingHandlerMapping", RequestMappingHandlerMapping.class);

RequestMappingHandlerMapping 是 Spring MVC 的核心组件——它在启动时扫描所有 @Controller / @RestController 的方法，解析 @RequestMapping 及其快捷注解，构建请求路径到处理器方法的映射表。通过 SpringUtils.getBean() 按名称和类型获取这个 Bean，拿到的是全局唯一的映射表实例。

3.2 遍历映射表

Map<RequestMappingInfo, HandlerMethod> map = mapping.getHandlerMethods();
map.keySet().forEach(info -> {
    Objects.requireNonNull(info.getPathPatternsCondition().getPatterns())
        .forEach(url -> set.add(ReUtil.replaceAll(url.getPatternString(), PATTERN, "*")));
});

getHandlerMethods() 返回所有注册的请求映射。Key 是 RequestMappingInfo（包含路径、请求方法、参数条件等元数据），Value 是对应的处理器方法。

关键调用链：

  • info.getPathPatternsCondition() — 获取路径模式条件（Spring 5.3+ 使用 PathPattern 代替旧的 AntPathMatcher）
  • .getPatterns() — 返回该映射的所有路径模式（一个方法可以有多个 URL，如 @RequestMapping({"/list", "/search"})）
  • Objects.requireNonNull() — 防御性编程，如果有映射没有路径（理论上极少见），直接抛 NPE 暴露问题

3.3 路径变量替换：{xxx} → *

ReUtil.replaceAll(url.getPatternString(), PATTERN, "*")

这是最关键的一步。原始路径如 /system/user/{userId}，其中 {userId} 是路径变量，在运行时会被替换为实际值（如 /system/user/100）。但 SaRouter.match() 的匹配逻辑需要通配符模式，所以将 {xxx} 统一替换为 *，得到 /system/user/*。

使用 Hutool 的 ReUtil.replaceAll() 结合正则 \{(.*?)\}（Pattern.compile 编译的常量），非贪婪匹配任意花括号包围的内容。

3.4 去重与存储

使用 HashSet 去重（同一个 URL 可能在多个 Controller 方法上出现），最终存入 List<String> urls。注意这里的 Set 只在方法内使用，通过 addAll 传递到成员变量的 List 中——避免成员变量类型暴露内部去重的实现细节。

4. URL 列表如何驱动 SaRouter.match()

回到 SecurityConfig 中的调用：

SaRouter
    .match(allUrlHandler.getUrls())
    .check(() -> { /* 鉴权逻辑 */ });

SaRouter.match() 接收一个路径模式列表，内部判断当前请求路径是否命中列表中的任意一条。命中的话执行 check() 的鉴权逻辑。这意味着：

  • 只有 Controller 上显式声明的路径才会被拦截——没有注册映射的路径返回 404，不会进入鉴权链
  • 路径变量被替换为 * 后，/system/user/100 能正确匹配 /system/user/*
  • 如果有两个 Controller 方法映射了同一个路径（重复映射），HashSet 去重保证不重复校验
为什么不在 addPathPatterns("/**") 之后就结束了？因为 addPathPatterns("/**") 只是声明拦截器对所有路径生效，真正的"是否拦截"逻辑由 SaRouter.match().check() 控制。如果不用 match() 而是直接 check()，那每个请求（包括不存在的路径）都会执行鉴权逻辑，增加不必要的开销。

5. 客户端鉴权的完整流程

当请求通过了登录校验（StpUtil.checkLogin()），接下来进入客户端鉴权阶段。这一阶段分为两个维度：

5.1 客户端 ID 一致性校验

token 签发时会在 extra 字段中存储 clientid（即 LoginHelper.CLIENT_KEY）。后续请求必须在头部或参数中携带相同的 clientId：

String headerCid = request.getHeader(LoginHelper.CLIENT_KEY);      // 优先从 Header 取 "clientid"
String paramCid = ServletUtils.getParameter(LoginHelper.CLIENT_KEY); // 其次从 URL 参数取 "clientid"
String clientId = StpUtil.getExtra(LoginHelper.CLIENT_KEY).toString(); // Token 中的 clientid

if (!StringUtils.equalsAny(clientId, headerCid, paramCid)) {
    throw NotLoginException.newInstance(StpUtil.getLoginType(),
        "-100", "客户端ID与Token不匹配", StpUtil.getTokenValue());
}

这里 StringUtils.equalsAny() 的语义是：clientId 等于 headerCid 或等于 paramCid 中任意一个即为通过。这允许前端灵活选择通过 Header 或 URL 参数传递 clientId。

5.2 客户端访问规则校验

validateClientAccessRules() 方法根据 token 中预置的客户端配置，进行两个维度的访问控制：

校验维度Token Extra Key常量校验逻辑
路径白名单 clientAccessPath LoginHelper.CLIENT_ACCESS_PATH_KEY 从 token 读取该客户端允许访问的路径模式列表（逗号、分号或换行分隔），当前请求路径必须匹配其中至少一条。使用 StringUtils.matches() 支持通配符匹配
IP 白名单 clientIpWhitelist LoginHelper.CLIENT_IP_WHITELIST_KEY 从 token 读取该客户端允许的 IP 范围列表（同上分隔符），当前请求来源 IP 必须命中其中至少一条。使用 NetUtils.isMatchIpRule() 支持 IP 段和 CIDR 格式

关键细节：

  • 分隔符 — CLIENT_RULE_SEPARATOR_REGEX 定义为 "[,;\\r\\n]+"，支持逗号、分号、回车、换行四种分隔符，兼容不同格式的配置输入
  • 空值保护 — 如果 token 中没有对应的 extra 字段（值为 null 或空字符串），validateClientAccessRules() 直接跳过该项校验。这意味着路径/IP 白名单都是可选的——不配置则不限制
  • 异常类型 — 校验失败抛出 NotPermissionException（不是 NotLoginException），语义上区分"你没权限"和"你没登录"
实际业务场景：在 SaaS 多租户系统中，可以为租户 A 的客户端配置只允许访问 /api/tenant-a/** 的路径白名单，并限制 IP 为租户 A 的办公网段。即使 token 本身有效，跨租户访问或从非授权 IP 访问都会被拦截。

6. SecurityProperties.excludes：排除路径配置

回到 SecurityConfig 的 addInterceptors() 末尾：

.excludePathPatterns(securityProperties.getExcludes());

SecurityProperties 的定义极为简洁：

@Data
@ConfigurationProperties(prefix = "security")
public class SecurityProperties {
    /** 排除路径 */
    private String[] excludes;
}

这意味着在 application.yml 中配置：

security:
  excludes:
    - /login
    - /captcha/**
    - /swagger-ui/**
    - /v3/api-docs/**

被 excludePathPatterns() 排除的路径完全不走 SaInterceptor——包括 SaRouter.match().check() 的整个鉴权链。这些路径只会经过第一道防线（上下文过滤器），因为那条过滤器是 /* 全局拦截且不受 InterceptorRegistry 控制。

安全建议：excludes 列表应严格控制。每增加一条排除路径，就是为整个系统开了一个免鉴权通道。登录接口和验证码接口必须排除（否则用户还没登录就被拦截，形成死循环），但其他路径要谨慎评估。
