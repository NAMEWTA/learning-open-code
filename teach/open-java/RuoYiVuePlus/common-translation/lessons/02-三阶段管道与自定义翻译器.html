02 | 三阶段管道与自定义翻译器：collect → prepare → process 全链路

课程目标：掌握 TranslationJsonFieldProcessor 的 collect → prepare → process 三阶段生命周期，理解批量翻译的性能优势，能够从零实现一个自定义翻译器并注册到系统中。

1. collect → prepare → process 三阶段详解

TranslationJsonFieldProcessor 实现了 JsonFieldProcessor 接口（来自 ruoyi-common-json 模块），通过 @Order(0) 注解确保它在所有字段处理器中第一个执行。它遵循严格的三阶段生命周期：

                 collect                            prepare                          process
                    ↓                                  ↓                                ↓
  遍历响应对象树的每一个字段    →    聚合后批量调用翻译器    →    将翻译结果写回字段值

  扫描 @Translation 注解         调用 translationBatch        从 context 取出翻译值
  按 (type, other) 分组          获取 Map<Object, String>     覆盖到字段上
  存入 JsonEnhancementContext     存入 JsonEnhancementContext   字段变成可读名称

1.1 第一阶段：collect（收集）

collect 阶段递归遍历整个响应对象树，找出所有带 @Translation 注解的字段，将它们的原始值按 (type, other) 分组存入上下文。

// collect 阶段核心逻辑（简化）
void collect(JsonEnhancementContext ctx, JsonFieldContext fieldCtx) {
    Translation annotation = fieldCtx.getAnnotation(Translation.class);
    if (annotation == null) return;  // 没有 @Translation，跳过

    // 1. 确定翻译的输入值
    Object rawValue;
    if (StringUtils.isNotBlank(annotation.mapper())) {
        // 有 mapper：反射读取 owner 对象上的源字段
        rawValue = ReflectionUtils.getFieldValue(
            fieldCtx.getOwner(), annotation.mapper()
        );
    } else {
        // 无 mapper：直接取当前字段的值
        rawValue = fieldCtx.getValue();
    }
    if (rawValue == null) return;  // 空值不翻译

    // 2. 构建分组 key
    String groupKey = annotation.type() + ":" +
        (StringUtils.isNotBlank(annotation.other()) ? annotation.other() : "");

    // 3. 将值收集到对应分组中
    ctx.getOrCreateAttribute(groupKey, HashSet::new).add(rawValue);
}

遍历策略：使用反射递归检查响应对象及其嵌套对象和集合元素。即使 VO 内部嵌套了子 VO 或 List，只要子对象的字段上有 @Translation，都会被扫描到并纳入翻译计划。

分组的意义：

假设一个响应对象包含以下 3 个带 @Translation 的字段：

字段 A: @Translation(type="user_id_to_name"), 值=1
字段 B: @Translation(type="user_id_to_name"), 值=3
字段 C: @Translation(type="dept_id_to_name"), 值=5

collect 后的分组：
  "user_id_to_name:" → Set{1, 3}     ← 两个字段分到同一组
  "dept_id_to_name:" → Set{5}        ← 独立一组

同一 (type, other) 下的所有值会作为一个整体传给 translationBatch()，这是批量优化的基础。

1.2 第二阶段：prepare（准备）

prepare 阶段遍历 collect 阶段建立的所有分组，对每组调用对应翻译器的 translationBatch() 方法，将返回的映射结果存入上下文。

// prepare 阶段核心逻辑（简化）
void prepare(JsonEnhancementContext ctx) {
    for (Map.Entry<String, Set<Object>> entry : ctx.getAttributeMap().entrySet()) {
        String groupKey = entry.getKey();        // "user_id_to_name:"
        Set<Object> rawValues = entry.getValue(); // {1, 3}

        // 1. 从 groupKey 中拆分出 type 和 other
        String type = groupKey.split(":")[0];       // "user_id_to_name"
        String other = groupKey.split(":").length > 1
            ? groupKey.split(":")[1] : null;

        // 2. 根据 type 查找对应的 TranslationInterface 实现
        TranslationInterface translator = translatorMap.get(type);

        // 3. 批量翻译
        Map<Object, ?> translatedMap = translator.translationBatch(rawValues, other);
        // 返回: {1→"张三", 3→"李四"}

        // 4. 将翻译结果存入上下文
        ctx.setAttribute(groupKey + "_result", translatedMap);
    }
}

为什么分阶段？如果边收集边翻译（collect 中直接调 translator），则无法实现批量——先收集所有待翻译的值，再一次性批量查询，是批量优化的前提条件。

1.3 第三阶段：process（处理）

process 阶段再次遍历所有带 @Translation 的字段，从上下文中取出 prepare 阶段准备好的翻译结果，覆盖到字段值上。

// process 阶段核心逻辑（简化）
Object process(JsonEnhancementContext ctx, JsonFieldContext fieldCtx) {
    Translation annotation = fieldCtx.getAnnotation(Translation.class);
    if (annotation == null) return fieldCtx.getValue();

    // 1. 确定原始值（与 collect 逻辑一致）
    Object rawValue = getRawValue(fieldCtx, annotation);
    if (rawValue == null) return null;

    // 2. 计算分组 key，获取翻译结果映射
    String groupKey = annotation.type() + ":" +
        (annotation.other() != null ? annotation.other() : "");
    Map<Object, String> resultMap = ctx.getAttribute(groupKey + "_result");

    // 3. 查表替换：原始值 → 翻译后的可读名称
    String translated = resultMap.get(rawValue);
    return translated != null ? translated : rawValue;
}
关键细节：process 阶段替换的是 JSON 序列化输出中的值，而不是 Java 对象的字段值。因此翻译不会污染原始 VO 对象——下次序列化时如果字段值变化，翻译结果也会自动更新。

1.4 JsonEnhancementContext 上下文

JsonEnhancementContext 是三个阶段的共享数据容器，生命周期与单次 JSON 序列化绑定：

方法作用使用阶段
getOrCreateAttribute(key, factory)获取或初始化一个属性collect
setAttribute(key, value)存储翻译结果prepare
getAttribute(key)读取翻译结果process

每个 HTTP 请求的 JSON 序列化都是独立的一次三阶段流程，context 不跨请求共享。

2. 性能优势：批量翻译 vs N+1 退化

2.1 批量优化的原理

假设一个列表接口返回 100 条记录，每条记录有 2 个需要翻译的字段（创建人和部门），未批量优化时的查询次数：

方案查询次数数据库往返典型耗时
N+1（退化，逐条查询）200 次200 次~2000ms
批量优化（覆盖 translationBatch）2 次2 次~30ms

批量优化的核心是将分散的 ID 聚合后一次性查询：

// N+1 退化模式（默认行为，每个 ID 单独查一次）
for (Object id : ids) {
    result.put(id, remoteUserService.selectById(id));  // 100 次 RPC
}

// 批量优化模式（所有内置实现都采用）
Set<Long> allIds = collectLongIds(ids);
List<SysUserVo> users = remoteUserService.selectListByIds(allIds);  // 1 次 RPC
Map<Long, String> result = users.stream()
    .collect(Collectors.toMap(SysUserVo::getUserId, SysUserVo::getUserName));
自定义翻译器必须注意：一定要覆盖 translationBatch() 方法。默认实现的循环调用会在日志中输出 WARNING，同时产生严重的性能问题。这不是可选的优化，而是生产环境的基本要求。

2.2 分组策略对批量的影响

分组 key 是 type + ":" + other。这意味着：

  • 同一个翻译类型且 other 也相同的字段会共用一次批量查询
  • 同一个翻译类型但 other 不同的字段会各自独立查询（对于 DictTypeTranslation 来说这是必要的——不同字典类型有不同数据）
  • 翻译器不需要感知分组逻辑——它只接收一组 key 和一个 other 参数

3. TranslationConfig 自动装配机制

3.1 配置类的完整结构

TranslationConfig 是 @AutoConfiguration 类，只有一个 @Bean 方法：

@AutoConfiguration
public class TranslationConfig {

    @Bean
    @Order(0)  // 确保 TranslationJsonFieldProcessor 在所有字段处理器中第一个执行
    public TranslationJsonFieldProcessor translationJsonFieldProcessor(
            List<TranslationInterface<?>> list) {

        // 构建 type → 实现类的映射
        Map<String, TranslationInterface<?>> translatorMap = new HashMap<>();
        for (TranslationInterface<?> impl : list) {
            TranslationType annotation = impl.getClass()
                .getAnnotation(TranslationType.class);
            if (annotation != null) {
                translatorMap.put(annotation.type(), impl);
            }
        }

        return new TranslationJsonFieldProcessor(translatorMap);
    }
}

3.2 Spring DI 如何收集所有实现

List<TranslationInterface<?>> list

这行参数声明利用了 Spring 的类型注入能力——Spring 会自动查找容器中所有 TranslationInterface 的子类型 Bean 并注入为一个 List。编写自定义翻译器时，不需要修改任何配置文件，只需要：

  1. 让实现类实现 TranslationInterface
  2. 添加 @Component（或 @Service）注册为 Spring Bean
  3. 添加 @TranslationType(type = "...") 声明类型

TranslationConfig 会自动发现新实现并注册到 translatorMap 中。

3.3 @Order(0) 的含义

@Order(0) 确保 TranslationJsonFieldProcessor 在 JSON 序列化的字段处理器链中第一个执行。这是必要的，因为后续处理器可能会对字段值做进一步处理（如加密、脱敏），而翻译需要在原始值上操作。

4. 自定义翻译器实战：订单 ID → 订单摘要

实战场景：业务系统中有一个订单模块，VO 里存储的是订单 ID，但前端需要展示一个包含订单号和金额的摘要字符串（如 "ORD-2024001 ￥1,280.00"）。我们需要创建一个自定义翻译器来实现这个转换。

Step 1：定义常量

在 TransConstant 接口中新增常量（或创建自己的常量接口）：

public interface TransConstant {
    // ... 已有常量 ...

    // 新增：订单 ID → 订单摘要
    String ORDER_ID_TO_SUMMARY = "order_id_to_summary";
}

Step 2：创建翻译器实现类

@Component
@TranslationType(type = TransConstant.ORDER_ID_TO_SUMMARY)
public class OrderTranslation implements TranslationInterface<String> {

    @Autowired
    private RemoteOrderService remoteOrderService;

    @Override
    public String translation(Object key, String other) {
        Long orderId = Long.valueOf(key.toString());
        OrderVo order = remoteOrderService.selectById(orderId);
        if (order == null) return null;
        return formatOrderSummary(order);
    }

    @Override
    public Map<Object, String> translationBatch(Set<Object> keys, String other) {
        // 1. 收集所有订单 ID
        Set<Long> orderIds = collectLongIds(keys);

        // 2. 批量查询（关键：一次 RPC 查全部）
        List<OrderVo> orders = remoteOrderService.selectListByIds(orderIds);

        // 3. 构建 ID → 摘要 的映射
        return orders.stream().collect(Collectors.toMap(
            OrderVo::getOrderId,
            this::formatOrderSummary
        ));
    }

    private String formatOrderSummary(OrderVo order) {
        return String.format("%s ￥%.2f", order.getOrderNo(), order.getAmount());
    }
}

Step 3：在 VO 中使用

public class PaymentVO {

    // 存储数据库字段
    private Long orderId;

    // 前端展示字段
    @Translation(type = "order_id_to_summary", mapper = "orderId")
    private String orderSummary;

    // 也可以直接用当前字段的值翻译
    @Translation(type = "order_id_to_summary")
    private Long anotherOrderId;
}

Step 4：验证

无需修改任何配置类或 XML。重启服务后，Spring 自动发现 OrderTranslation Bean，TranslationConfig 自动将其注册到 translatorMap 中。

调用任何返回 PaymentVO 的接口，orderSummary 字段会自动从 "ORD-2024001 ￥1,280.00"。

零配置扩展：这种设计的好处是，添加新翻译类型不需要改动任何现有代码——不需要修改 TranslationConfig、不需要修改 TranslationJsonFieldProcessor、不需要修改任何 XML/Properties 配置。这就是 Spring DI + 策略模式 + 约定优于配置的威力。

5. mapper 机制深入

5.1 mapper 的反射原理

当 @Translation(mapper = "createById") 时，处理器需要从 owner 对象上读取 createById 字段的值。reflection 工作原理：

// 伪代码：mapper 获取源值
Object owner = fieldCtx.getOwner();                         // NoticeVO 实例
Field sourceField = owner.getClass()
    .getDeclaredField(annotation.mapper());                // "createById" 字段
ReflectionUtils.makeAccessible(sourceField);
Object sourceValue = sourceField.get(owner);               // 读取值 = 1001L

5.2 逗号分隔多值字符串的处理

某些业务场景中，一个字段可能存储多个 ID（逗号分隔），翻译后也需要逗号分隔的名称：

// 输入字段
private String userIds = "1,2,3";

// 翻译后输出
private String userNames = "张三,李四,王五";

这时使用的工具链是：collectLongIds 拆分 ID → translationBatch 批量查询 → joinMappedValues 拼接输出。

// collectLongIds 会将 "1,2,3" 拆分为 {1L, 2L, 3L} 加入待翻译集合
// translationBatch 一次性查询返回 {1L→"张三", 2L→"李四", 3L→"王五"}
// process 阶段将单个字段的 "1,2,3" 拆开逐个查映射，再拼回 "张三,李四,王五"

5.3 mapper 的典型应用模式

模式注解位置mapper 值说明
ID+Name 双字段 name 字段上 对应的 id 字段名 最常见：createById + createByName
自翻译 id 字段上 无（不设置 mapper） 直接翻译当前字段，如 userId 变成 "张三"
跨对象引用 子对象字段上 父对象的字段名 较少见，需确保 owner 可达

6. 综合练习与自测

自测题 1：三阶段流程（点击展开）

Q1：为什么需要将翻译流程分成 collect、prepare、process 三个阶段？不能在一个阶段内完成吗？

Q2：prepare 阶段中，如果某个类型的翻译器在处理时抛出了异常，会发生什么？会影响其他类型的翻译吗？

Q3：JsonEnhancementContext 的声明周期是多久？它是否跨请求共享数据？

A1：不能。如果边收集边翻译，就无法实现批量优化——必须先收集完所有待翻译的值，才能对每组值进行一次批量查询。分离三个阶段是批量优化的前提。

A2：取决于具体实现。通常翻译异常应该被 try-catch 捕获并记录日志，不应该阻断整个 JSON 序列化流程。合理的做法是：某个翻译器失败时，该类型的字段保持原值（ID 而非名称），其他类型的翻译不受影响。

A3：生命周期与单次 JSON 序列化绑定——一次 HTTP 响应序列化 → 创建 context → collect → prepare → process → context 销毁。它不跨请求共享数据，每次序列化都是全新的三阶段流程。

自测题 2：自定义翻译器（点击展开）

Q1：自定义一个翻译器需要改动哪些现有文件？

Q2：如果自定义翻译器没有覆盖 translationBatch()，会发生什么？日志中会看到什么？

Q3：以下代码有什么问题？

@Component
public class MyTranslation implements TranslationInterface<String> {
    @Override
    public String translation(Object key, String other) {
        return "hello";
    }
}

A1：理论上只需要创建一个新类。但最佳实践是同时在 TransConstant 中定义新的类型常量。不需要修改 TranslationConfig、TranslationJsonFieldProcessor 或任何配置类——这些都是自动装配的。

A2：默认实现会循环调用 translation() 方法，每次调用都打印一条 WARNING 级别的日志。如果翻译列表中有 100 个值，就会看到 100 条警告日志 + 产生 N+1 查询性能问题。

A3：缺少 @TranslationType 注解。虽然该类实现了 TranslationInterface，但 TranslationConfig 需要通过 @TranslationType.type() 来注册它。没有这个注解，这个翻译器永远不会被调用——它是 Spring Bean 但不会进入 translatorMap。
