01 | @Translation 注解与五种内置翻译器：注解声明与 SPI 实现

课程目标：理解 @Translation 注解的全部参数语义，掌握 TranslationInterface SPI 的扩展机制，熟记五种内置翻译器的类型常量与对应的远程服务。

1. 全景鸟瞰：从 VO 字段到可读名称的完整链路

在典型的 RuoYi-Vue-Plus 项目中，数据库存储的是 ID 和编码，但接口返回给前端时需要的是人类可读的显示名称。例如：

  • 数据库存 create_by = 1，接口返回 createByName = "张三"
  • 数据库存 oss_id_list = "1,2,3"，接口返回文件的可访问 URL
  • 数据库存 status = "0"，接口返回 statusLabel = "正常"

ruoyi-common-translation 模块的核心职责，就是在 JSON 序列化阶段自动完成这个转换，无需在每个 Controller 方法里手动查询和拼接。

模块规模：共 11 个 Java 文件，按职责分为以下五层：

层级文件职责
注解层@Translation、@TranslationType声明翻译意图与 SPI 类型绑定
SPI 接口TranslationInterface定义翻译行为契约
处理器层TranslationJsonFieldProcessor三阶段管道：collect → prepare → process
配置层TranslationConfig自动装配 Spring Bean 映射
实现层5 个内置 TranslationInterface 实现用户、部门、字典、OSS 的具体翻译逻辑

核心流程图：

VO 字段上 @Translation(type="user_id_to_name")
        ↓
TranslationJsonFieldProcessor.collect()  扫描所有带 @Translation 的字段
        ↓
TranslationJsonFieldProcessor.prepare()  按 (type, other) 分组，调用 translationBatch()
        ↓    ↓    ↓
   UserNameTranslation  →  SysUserService.selectListByIds()
   DeptNameTranslation  →  SysDeptService.selectListByIds()
   DictTypeTranslation  →  SysDictDataService.selectListByType()
        ↓    ↓    ↓
TranslationJsonFieldProcessor.process()  将翻译结果写回字段值
关键洞察：翻译逻辑不是侵入式地嵌在每个 Controller 里，而是作为 JSON 序列化管道的一个后处理器，在序列化时自动执行。这意味着任何返回 VO 的 Controller 方法，只要 VO 上有 @Translation 注解，就自动获得翻译能力。

2. @Translation 注解详解

注解定义在 org.dromara.common.translation.annotation.Translation，有三个参数：

参数类型必填说明
typeString是翻译类型字符串，必须与某个 @TranslationType 的 type 值精确匹配
mapperString否源字段名。设置后，处理器读取 owner 对象 上 mapper 指向的字段值作为翻译输入，翻译结果写入当前字段
otherString否额外参数，传给 TranslationInterface.translation(key, other)。典型用途：字典类型编码

2.1 type 参数（翻译类型标识）

type 是整个翻译机制的"钥匙"。它通过 TranslationConfig 在运行时映射到具体的 TranslationInterface 实现：

@Translation(type="user_id_to_name")
        ↓ 运行时查找
TranslationConfig.translatorMap.get("user_id_to_name")
        ↓ 返回
UserNameTranslation 实例

这种设计使得 注解声明与具体实现对业务代码完全透明——VO 上只写字符串，不需要 import 任何实现类。

2.2 mapper 参数（源字段映射）

这是最精妙的设计之一。考虑一个典型的 VO 场景：

public class NoticeVO {
    private Long createById;    // DB 存储的创建人 ID = 1001
    private String createByName; // 接口需要返回 "张三"

    // 字段 createByName 本身的值可能是 null（数据库不存名称）
    // 真正需要翻译的值在 owner 对象的 createById 字段上
    @Translation(type = "user_id_to_name", mapper = "createById")
    private String createByName;
}

当 mapper = "createById" 时，处理器的行为是：

  1. 通过反射获取 当前字段所属的 owner 对象（即 NoticeVO 实例）
  2. 找到 owner 上名为 createById 的字段，读取其值 1001L
  3. 用 1001L 作为 key 调用 TranslationInterface.translation(1001L, null)
  4. 将返回的 "张三" 写入当前字段 createByName

等价伪代码：

// 有 mapper 时：
Object sourceValue = owner.getCreateById();   // 从同对象另一个字段取
currentField.set(翻译(sourceValue));

// 无 mapper 时（默认行为）：
Object sourceValue = currentField.get();      // 从当前字段自身取值
currentField.set(翻译(sourceValue));

2.3 other 参数（额外参数传递）

当一个翻译器需要区分多种翻译规则时，other 参数提供了额外信息。最典型的应用场景是字典翻译：

// 翻译 "性别" 字典：value "0" → label "男"
@Translation(type = "dict_type_to_label", other = "sys_user_sex")
private String sex;

// 翻译 "状态" 字典：value "0" → label "正常"
@Translation(type = "dict_type_to_label", other = "sys_normal_disable")
private String status;

两个字段使用同一个翻译类型 dict_type_to_label，但通过 other 参数区分要查询哪个字典类型。处理器在分组时会以 (type, other) 作为复合 key：

groupKey = type + ":" + (other != null ? other : "")
// "dict_type_to_label:sys_user_sex" vs "dict_type_to_label:sys_normal_disable"
设计要点：other 参数的存在使得同一个 TranslationInterface 实现可以处理同一翻译类型的多个变体场景，避免了"每种字典各写一个翻译器"的冗余。

2.4 完整 VO 示例

以下是一个同时使用五种内置翻译的完整 VO 示例：

public class ComprehensiveVO {

    // 1. 用户 ID → 用户姓名（字段自身值翻译）
    @Translation(type = "user_id_to_name")
    private Long userId;

    // 2. 用户 ID → 用户昵称（通过 mapper 从其他字段取值）
    private Long createById;
    @Translation(type = "user_id_to_nickname", mapper = "createById")
    private String createByName;

    // 3. 部门 ID → 部门名称
    @Translation(type = "dept_id_to_name")
    private Long deptId;

    // 4. 字典值 → 字典标签（使用 other 指定字典类型）
    @Translation(type = "dict_type_to_label", other = "sys_user_sex")
    private String sex;

    // 5. OSS ID → 可访问 URL
    @Translation(type = "oss_id_to_url")
    private String ossId;

    // getters and setters...
}

当这个 VO 被序列化为 JSON 时，userId 字段的值会从 1001 被替换为 "张三"，deptId 从 200 变为 "技术部"，以此类推。

3. @TranslationType 与 TranslationInterface SPI

3.1 @TranslationType：类型级注解绑定

@TranslationType 是加在 TranslationInterface 实现类上的注解，只有一个参数 type: String：

@Component
@TranslationType(type = "user_id_to_name")
public class UserNameTranslation implements TranslationInterface<String> {
    // ...
}

TranslationConfig 通过 Spring 的 List<TranslationInterface<?>> 注入收集所有实现类 Bean，然后读取每个实现类的 @TranslationType.type() 构建查找表：

Map<String, TranslationInterface<?>> translatorMap = new HashMap<>();
for (TranslationInterface<?> impl : allImplementations) {
    TranslationType annotation = impl.getClass().getAnnotation(TranslationType.class);
    translatorMap.put(annotation.type(), impl);
}

这本质上是 Spring DI + 策略模式，而不是 Java ServiceLoader。每一个带有 @TranslationType 的 Spring Bean 都会被自动发现和注册。

3.2 TranslationInterface 核心契约

public interface TranslationInterface<T> {

    // 单个翻译（所有实现必须覆盖）
    T translation(Object key, String other);

    // 批量翻译（建议覆盖，默认循环调用 translation 并打印警告）
    default Map<Object, T> translationBatch(Set<Object> keys, String other) {
        log.warn("translationBatch is not overridden, falling back to N+1 mode");
        Map<Object, T> result = new HashMap<>();
        for (Object key : keys) {
            result.put(key, translation(key, other));
        }
        return result;
    }

    // === 工具方法（default，无需覆盖）===

    // 将 Set<Object> 转为 Set<Long>，支持逗号分隔的复合值拆分
    default Set<Long> collectLongIds(Set<Object> keys) { ... }

    // 将逗号分隔字符串解析为 List<Long>
    default List<Long> parseLongIds(String ids) { ... }

    // 批量映射 + 逗号拼接输出
    default String joinMappedValues(String ids, Function<String, String> mapper) { ... }
}

3.3 泛型参数 T 的含义

泛型 T 定义翻译结果的类型。所有内置实现都使用 String，但接口设计本身是开放的——如果你需要返回一个复杂对象（如 OrderSummary），只需要让 TranslationInterface<OrderSummary> 的实现返回该类型即可。

3.4 批量翻译的性能退化陷阱

重要警告：如果自定义实现没有覆盖 translationBatch()，默认实现会循环调用 translation()，产生 N+1 问题。
例如：100 个用户 ID = 100 次 SelectById 查询 = 100 次数据库往返。
所有五个内置实现都主动覆盖了 translationBatch() 以避免此问题。

3.5 工具方法详解

collectLongIds：将翻译输入值统一转为 Long 集合。关键能力是处理逗号分隔的多值字符串：

// 输入 Set{"1", "2,3,4", 5L, "6,7"}
// 输出 Set{1L, 2L, 3L, 4L, 5L, 6L, 7L}

parseLongIds：将单个逗号分隔字符串拆分为 Long 列表。与 collectLongIds 的区别是它只处理单个字符串，不处理集合。

joinMappedValues：将逗号分隔的 ID 字符串逐一映射后重新用逗号拼接。典型场景是 OSS ID 列表翻译：

// 输入 ids="1,2,3"，mapper = (id) -> "https://oss.example.com/file/" + id
// 输出 "https://oss.example.com/file/1,https://oss.example.com/file/2,..."

4. 五种内置翻译器详解

常量实现类远程服务输入输出
USER_ID_TO_NAME UserNameTranslation RemoteUserService userId (Long) userName (String)
USER_ID_TO_NICKNAME NicknameTranslation RemoteUserService userId (Long) nickName (String)
DEPT_ID_TO_NAME DeptNameTranslation RemoteDeptService deptId (Long) deptName (String)
DICT_TYPE_TO_LABEL DictTypeTranslation RemoteDictService dictValue (String) dictLabel (String)
OSS_ID_TO_URL OssUrlTranslation RemoteOssService ossId (Long) fileUrl (String)

4.1 UserNameTranslation — 用户 ID 转用户姓名

@Component
@TranslationType(type = TransConstant.USER_ID_TO_NAME)
public class UserNameTranslation implements TranslationInterface<String> {

    @Autowired
    private RemoteUserService remoteUserService;

    @Override
    public String translation(Object key, String other) {
        // 单个查询
    }

    @Override
    public Map<Object, String> translationBatch(Set<Object> keys, String other) {
        Set<Long> userIds = collectLongIds(keys);
        List<SysUserVo> users = remoteUserService.selectListByIds(userIds);
        // 构建 Map<userId, userName> 返回
    }
}

特点：调用远程 Feign 接口 RemoteUserService，一次性传入所有用户 ID 批量查询，避免 N+1。

4.2 NicknameTranslation — 用户 ID 转用户昵称

与 UserNameTranslation 调用同一个远程服务 RemoteUserService，但映射到 nickName 字段。这两个翻译器同时存在是因为业务场景中有时需要姓名、有时需要昵称，分别使用不同的翻译类型。

// VO 上的典型用法
private Long userId;
@Translation(type = "user_id_to_name")
private String userNameDisplay;

private Long approverId;
@Translation(type = "user_id_to_nickname", mapper = "approverId")
private String approverNickname;

4.3 DeptNameTranslation — 部门 ID 转部门名称

调用 RemoteDeptService 批量查询部门信息，映射到部门名称。处理逻辑与用户翻译器完全一致，只是远程服务和映射目标不同。

4.4 DictTypeTranslation — 字典类型转标签（含 other 参数）

这是唯一一个必须依赖 other 参数的翻译器。因为系统中存在多种字典（性别、状态、类型等），翻译器需要知道查询哪个字典类型：

@Component
@TranslationType(type = TransConstant.DICT_TYPE_TO_LABEL)
public class DictTypeTranslation implements TranslationInterface<String> {

    @Autowired
    private RemoteDictDataService dictDataService;

    @Override
    public Map<Object, String> translationBatch(Set<Object> keys, String other) {
        // other 参数 = 字典类型编码，如 "sys_user_sex"
        List<SysDictDataVo> dictList = dictDataService.selectDictDataByType(other);
        // 构建 Map<dictValue, dictLabel> 返回
    }
}

关键点：在 TranslationJsonFieldProcessor.collect() 阶段，所有 other="sys_user_sex" 的字段会被分到同一组，other="sys_normal_disable" 的字段会被分到另一组。每组独立调用一次 translationBatch()，保证字典数据只查询一次。

4.5 OssUrlTranslation — OSS ID 转可访问 URL

将文件存储 ID 翻译为可访问的完整 URL。输入可能是单个 ID 或多个逗号分隔的 ID：

// 单个文件
@Translation(type = "oss_id_to_url")
private Long avatarOssId;

// 多个文件（逗号分隔）
@Translation(type = "oss_id_to_url")
private String attachmentIds;  // "1,2,3" → 翻译结果也是逗号分隔的 URL

OssUrlTranslation 利用 joinMappedValues 工具方法处理逗号分隔的多值场景：

@Override
public String translation(Object key, String other) {
    String ids = (String) key;
    return joinMappedValues(ids, id -> {
        // 按 ID 查询 OSS 记录，拼接完整 URL
    });
}

5. TransConstant 常量的设计价值

TransConstant 是一个接口，定义了 5 个字符串常量：

public interface TransConstant {
    String USER_ID_TO_NAME = "user_id_to_name";
    String USER_ID_TO_NICKNAME = "user_id_to_nickname";
    String DEPT_ID_TO_NAME = "dept_id_to_name";
    String DICT_TYPE_TO_LABEL = "dict_type_to_label";
    String OSS_ID_TO_URL = "oss_id_to_url";
}

为什么不用字符串字面量？

  1. 编译期校验：使用 TransConstant.USER_ID_TO_NAME 代替 "user_id_to_name"，IDE 可以自动补全和重构
  2. 避免拼写错误："user_id_to_nmae"（typo）在运行时才会被发现，而常量名写错会直接编译失败
  3. 统一双端绑定：同一个常量同时用在 @Translation(type = TransConstant.XXX) 和 @TranslationType(type = TransConstant.XXX) 上，确保注解端与实现端永远一致
  4. 可发现性：打开 TransConstant.java 即可一览所有可用翻译类型，不需要在代码库里搜索字符串
最佳实践：自定义翻译器时，新定义的翻译类型常量也应该追加到 TransConstant 中，而非在 VO 或实现类里私自定义字符串。

6. 综合练习与自测

自测题 1：参数匹配（点击展开）

Q1：VO 上写了 @Translation(type = "user_id_to_name", mapper = "ownerId")，注解加在 userName 字段上。处理器在执行翻译时的输入值来自哪里？

Q2：如果不设置 mapper 参数，翻译的输入值取自哪里？

Q3：other 参数在 DictTypeTranslation 中的实际作用是什么？

A1：来自 owner 对象的 ownerId 字段值。处理器通过反射读取当前字段所属对象的 ownerId 字段，作为翻译 key。

A2：取自当前字段自身的值。即注解加在哪个字段上，就用那个字段的值作为翻译 key。

A3：other 传入的是字典类型编码（如 "sys_user_sex"），翻译器据此查询特定字典的数据字典表，构建 value→label 的映射。

自测题 2：架构理解（点击展开）

Q1：TranslationConfig 是如何发现并注册所有翻译器实现的？用的是 Java ServiceLoader 吗？

Q2：五个内置翻译器中，哪个翻译类型使用了 other 参数？为什么其他四个不需要？

Q3：如果新增了一个翻译器并为其定义了新的常量，需要修改 TranslationConfig 吗？

A1：不是 Java ServiceLoader。而是通过 Spring DI：TranslationConfig 的 translationJsonFieldProcessor(List<TranslationInterface<?>> list) 方法接收 Spring 注入的所有 Bean，然后读取每个实现类的 @TranslationType 注解构建 type→impl 映射。

A2：DICT_TYPE_TO_LABEL（DictTypeTranslation）。因为字典翻译需要区分不同字典类型（性别、状态等），同一个翻译器要处理多种字典。其他四个翻译器的翻译规则是唯一的，不需要额外参数区分。

A3：不需要。只要实现类上有 @TranslationType 注解并注册为 Spring Bean，TranslationConfig 会自动发现它。这是自动装配的优势。
