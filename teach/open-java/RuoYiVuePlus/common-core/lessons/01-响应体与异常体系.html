第 1 课 · 响应体与异常体系 R / PageResult / Exception

本节目标：学完你能说清 R<T> 的统一响应设计、PageResult<T> 的分页结构、以及三层异常体系（BaseException 分支 / ServiceException / SseException）各自的使用场景和设计动机。

ruoyi-common-core 模块包含约 65 个 Java 文件，是整个 RuoYi-Vue-Plus 项目所有其它模块的基石。任何一个 Controller 的返回值、任何一个 Service 抛出的异常、任何一个工具类的调用，最终都追溯到 core 模块。今天我们先看其中最核心的两块：统一响应体和分层异常体系。

一、统一响应体 R<T> — 前后端数据契约

R.java 位于 org.dromara.common.core.domain.R，是所有 Controller 返回值的标准信封。前端解析任何一个 API 响应时，都会看到这三个字段：

字段类型说明
codeint响应状态码，默认成功为 HttpStatus.SUCCESS (200)，失败为 HttpStatus.ERROR (500)
msgString提示信息，如 "操作成功"、"用户名已存在"
dataT（泛型）业务数据载荷，可以是单个对象、List、Map 或 null

静态工厂方法

R 的设计模式是私有构造 + 公开静态工厂方法 — 外部不能 new，只能通过以下工厂方法创建实例：

// 成功场景
R.ok()              → code=200, msg="操作成功", data=null
R.ok(data)          → code=200, msg="操作成功", data=具体值
R.ok(msg, data)     → code=200, msg=自定义, data=具体值

// 失败场景
R.fail()            → code=500, msg="操作失败"
R.fail(msg)         → code=500, msg=自定义
R.fail(code, msg)   → code=自定义, msg=自定义

// 警告场景（业务警告，非系统错误）
R.warn(msg)         → code=601 (HttpStatus.WARN)

// 语义别名
R.data(data)        → 等价于 R.ok(data)，让调用更自然

所有工厂方法最终都汇入私有方法 restResult(T data, int code, String msg)，完成三个字段的赋值。

前端解析契约：前端统一检查 response.code === 200 判断成功，取 response.msg 弹 toast 提示，从 response.data 取业务数据。这个契约一旦确定，整个项目的前后端交互就有了统一的沟通语言。
设计洞察 — 为什么是 601 而不是 200 的子码？ 业务警告（如"库存不足，已自动调整为最大可用数量"）并非系统错误，用 500 不合适；但它也不是完全成功，用 200 又会让前端误以为一切正常。独立一个 601 作为 WARN 码，前端可以单独处理（比如橙色提示而非绿色/红色），这是 语义精确性 的体现。

二、分页响应 PageResult<T> — 与 MyBatis-Plus 的解耦

PageResult.java 位于 org.dromara.common.core.domain.PageResult，是分页查询的统一返回结构：

public class PageResult<T> {
    private long total;              // 总记录数（注意是 long，不是 int）
    private Collection<T> rows;      // 当前页数据集合

    // 推荐：用 build() 工厂方法创建
    public static <T> PageResult<T> build() {
        PageResult<T> pageResult = new PageResult<>();
        pageResult.setRows(new ArrayList<>());  // 内置空集合兜底
        return pageResult;
    }
}

关键设计点对比：

设计点PageResult（RuoYi）MyBatis-Plus PageInfo
定位通用分页响应结构，与持久层无关MyBatis-Plus 专用，紧耦合框架
依赖不依赖任何 ORM 框架依赖 MyBatis-Plus IPage
兜底build() 自动初始化空 ArrayList无内置兜底
用途Controller 返回给前端Service 层内部分页查询
为什么不用 MyBatis-Plus 的 PageInfo？ 架构分层原则：Controller 层不应该暴露持久层框架的实现细节。PageResult 是纯 POJO，即使将来换掉 MyBatis-Plus，前端契约不受影响。Service 层拿到 MyBatis-Plus 的 IPage 后，映射为 PageResult 再返回给 Controller。

三、分层异常体系 — 三条并行分支

core 模块的异常设计不是一棵单继承树，而是三条独立分支并行发展，各司其职：

RuntimeException
│
├── 【分支 A】BaseException（国际化异常基类）
│   │   ├── module: String     ← 归属模块标识
│   │   ├── code: String       ← i18n 消息键
│   │   ├── args: Object[]     ← 消息参数
│   │   └── defaultMessage     ← 回退默认消息
│   │
│   ├── FileException (module="file")
│   │   ├── FileNameLengthLimitExceededException
│   │   ├── FileSizeLimitExceededException
│   │   └── ...
│   │
│   └── UserException (module="user")
│       ├── CaptchaException           (code="user.jcaptcha.error")
│       └── CaptchaExpireException      (code="user.jcaptcha.expire")
│
├── 【分支 B】ServiceException（final，通用业务异常）
│   └── 消息模板 + {} 占位符，链式调用
│
└── 【分支 C】SseException（final，SSE 专用异常）
    └── 结构类似 ServiceException，但独立类便于全局异常处理器差异化处理

BaseException — 支持国际化的异常基类

BaseException 位于 org.dromara.common.core.exception.base.BaseException，重写了 getMessage()：

@Override
public String getMessage() {
    return MessageUtils.message(code, args);
}

当抛出 new UserException("user.not.exists", new Object[]{userId}) 时，getMessage() 会调用 MessageUtils.message("user.not.exists", ...) 去 i18n 资源文件中查找对应的中文/英文消息。

BaseException 有四个核心字段：

  • module — 模块标识（如 "file"、"user"），方便全局异常处理器按模块分流
  • code — 消息键，对应 i18n 文件的 key
  • args — 消息占位符参数，如用户 ID、文件名等动态值
  • defaultMessage — 当 i18n 文件中找不到对应 key 时的回退消息

ServiceException — 追求极简使用的业务异常

ServiceException 位于 org.dromara.common.core.exception.ServiceException，是 final 类，独立于 BaseException 继承树。它不依赖 i18n，而是通过 Hutool 的 StrFormatter 做 {} 占位符格式化：

// 一行抛异常，不需要配 i18n 文件
throw new ServiceException("用户{}不存在", userId);

// 支持链式调用
throw new ServiceException("操作失败").setCode(403);

全局异常处理器捕获到 ServiceException 后，统一转为 R.fail(code, message) 返回前端。

设计洞察 — 为什么 ServiceException 不继承 BaseException？ 这是两种不同哲学的选择：BaseException 分支走的是规范化路线 — 需要预先配好 i18n 文件、定义模块归属、归类异常类型。适合文件模块、用户模块这种跨语言、多场景的领域。而 ServiceException 走的是实用主义路线 — 开发者写代码时随手一抛即可，不需要提前规划。两条分支并存，各取所需，互不干扰。

SseException — 为 SSE 长连接单独设类

SseException 位于 org.dromara.common.core.exception.SseException，结构上与 ServiceException 非常相似（同样有 code、message、占位符格式化），但它是一个独立的类。原因：

  1. 全局异常处理器通过 instanceof SseException 做差异化分流 — SSE 场景不能返回 JSON 格式的 R，而要向 EventStream 写入错误事件
  2. 语义清晰 — 代码中看到 SseException 就知道是推送相关异常，方便日志检索和问题排查

四、异常体系的完整流转链路

Controller → Service → throw new ServiceException("库存不足")
                           │
                           ▼
              GlobalExceptionHandler
              @ExceptionHandler(ServiceException.class)
                           │
                           ▼
              return R.fail(e.getCode(), e.getMessage())
                           │
                           ▼
              HTTP Response: {"code":500, "msg":"库存不足", "data":null}

整个链路中，Controller 和 Service 都不需要 try-catch — 全局异常处理器统一兜底，保证所有异常最终都以 R 的 JSON 格式返回给前端。

五、检验一下（提取练习）

Q1. R.warn("库存不足，已自动调整") 返回的 code 值是多少？

A. 200 B. 400 C. 500 D. 601
看答案D. 601。R.warn() 调用 R.restResult(code=HttpStatus.WARN=601)。区别于 R.fail() 的 500 和 R.ok() 的 200。

Q2. BaseException 和 ServiceException 的核心区别是什么？

A. 没有本质区别，只是作者写了两个类 B. BaseException 通过 MessageUtils 做 i18n 解析，ServiceException 通过 StrFormatter 做 {} 占位符替换，两者继承树独立 C. BaseException 是 checked exception，ServiceException 是 runtime exception D. BaseException 只能用于文件模块
看答案B。BaseException 重写 getMessage() 走 i18n 路径，子类 FileException/UserException 继承它；ServiceException 是 final 类，独立于 BaseException 继承树，用 Hutool StrFormatter 做占位符替换。

Q3. 为什么 SseException 要独立于 ServiceException 成为一个单独的类？

A. 因为 SSE 异常需要更多字段 B. 为了让全局异常处理器能够通过 instanceof 做差异化处理，SSE 场景不能返回 JSON 格式的 R C. 因为 SseException 继承的是不同的父类 D. 没有特别原因
看答案B。SSE 长连接场景下，异常不能以 JSON 格式返回 — 需要向 EventStream 发送错误事件。全局异常处理器通过 instanceof 区分 SseException 和 ServiceException，执行不同的处理逻辑。

Q4. PageResult.build() 为什么要在构造时初始化一个空 ArrayList？

A. 为了性能优化 B. 为了防止前端收到 null 导致空指针异常，这是防御性编程 C. MyBatis-Plus 要求必须初始化 D. 这只是代码风格问题
看答案B。build() 中 setRows(new ArrayList<>()) 保证即使查询返回空结果，前端拿到的也是 [] 而非 null，避免前端空指针判断的遗漏。
