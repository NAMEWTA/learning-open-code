第 2 课 · 常量枚举与设计模式 Constants / Enum

本节目标：学完你能解释为什么 core 模块的常量用 interface 而非 class，读懂 GlobalConstants 的 Redis Key 前缀协议和 CacheNames 的缓存策略编码格式，并掌握 BusinessStatusEnum 的内置状态机设计。

core 模块的常量体系有 6 个接口 + 7 个枚举类。它们不是简单的 public static final String 堆砌——每个常量集都承载了特定的架构约定。今天我们从 interface 常量模式这个设计选择讲起。

一、Interface 常量模式 — 为什么不用 class？

core 模块的 6 个常量集合——Constants、GlobalConstants、HttpStatus、CacheNames、RegexConstants、SystemConstants——全部使用 interface 定义，而非 class：

// 典型写法：interface 常量
public interface HttpStatus {
    int SUCCESS = 200;       // 不需要写 public static final
    int BAD_REQUEST = 400;   // 编译器自动隐式添加
    int WARN = 601;
}

// 对比：如果用 class 实现
public class HttpStatus {
    public static final int SUCCESS = 200;   // 必须显式声明
    public static final int BAD_REQUEST = 400;
}

为什么选 interface？

对比维度interface 常量class + private 构造
简洁性字段隐式 public static final，少写 3 个修饰符每个字段都要显式写全
防实例化interface 天生不能 new需要手动写 private 构造()
多实现调用方可以 implements 直接使用常量名调用方需要 import static 或写全限定名
争议Effective Java 不推荐（污染命名空间）更符合「常量工具类」传统
作者的选择：RuoYi-Vue-Plus 选择了 interface 模式。在框架层面，常量被大量类引用，implements HttpStatus 后直接写 SUCCESS 比 HttpStatus.SUCCESS 少敲不少字。这种权衡在业务代码中不推荐，但在框架常量的场景下是务实的选择。

二、GlobalConstants — Redis Key 前缀协议

GlobalConstants 位于 org.dromara.common.core.constant.GlobalConstants，定义框架级别的 Redis Key 前缀：

public interface GlobalConstants {
    String GLOBAL_REDIS_KEY = "global:";                     // 框架根前缀

    // 各类缓存的二级前缀
    String CAPTCHA_CODE_KEY = GLOBAL_REDIS_KEY + "captcha_codes:";   // 验证码
    String RATE_LIMIT_KEY = GLOBAL_REDIS_KEY + "rate_limit:";        // 限流
    String PWD_ERR_CNT_KEY = GLOBAL_REDIS_KEY + "pwd_err_cnt:";      // 密码错误计数
    String SYS_CONFIG_KEY = GLOBAL_REDIS_KEY + "sys_config:";         // 系统配置
    String SYS_DICT_KEY = GLOBAL_REDIS_KEY + "sys_dict:";             // 数据字典
    String REPEAT_SUBMIT_KEY = GLOBAL_REDIS_KEY + "repeat_submit:";  // 防重复提交
}

这个设计的意义在于 Key 命名空间隔离：

Redis Key 命名空间三层结构： global:* ← 框架级缓存（验证码、限流、防重提交） ├── captcha_codes:* ├── rate_limit:* └── repeat_submit:* tenant:* ← 租户级缓存（多租户隔离） business:* ← 业务级缓存（订单、用户等业务数据）

框架级缓存与租户、业务缓存使用不同的一级前缀，确保 FLUSHDB 或 Key 扫描时不会互相干扰。

三、CacheNames — 缓存名 + 策略编码协议

CacheNames 位于 org.dromara.common.core.constant.CacheNames，它的常量值遵循一种自解释的编码格式：

命名格式： name#ttl#maxIdle#maxSize#local

示例：
SYS_CONFIG     = "sys_config"                         // 永不过期
SYS_USER_NAME  = "sys_user_name#30d"                  // TTL = 30 天
SYS_ONLINE     = "sys_online#2m#1m#200#true"          // 2min TTL, 1min 最大空闲, 200 条, 本地缓存
SYS_OSS_CONFIG = "sys_oss_config#30d"                 // OSS 配置，30 天过期
SYS_DICT       = "sys_dict#30d"                       // 数据字典，30 天过期

字段含义：

字段含义格式示例
name缓存名称标识（必填）sys_user_name
ttl过期时间（可选）30d（天）、2h（小时）、5m（分钟）
maxIdle最大空闲时间（可选）1m
maxSize最大缓存条目数（可选）200
local是否启用本地缓存（可选）true / false
设计洞察 — 字符串编码 vs 配置对象：传统的做法是为每个缓存建一个 CacheConfig 对象（name + ttl + maxSize...），需要写大量样板代码。CacheNames 把策略编码直接嵌入字符串，解析时按 # 分割即可。好处是：常量即文档——看一眼 "sys_user_name#30d" 就知道这个缓存 30 天过期，不需要再去找对应的配置类。这是 "convention over configuration" 原则在缓存管理中的体现。

四、HttpStatus — 独立于 Spring 的状态码体系

HttpStatus 位于 org.dromara.common.core.constant.HttpStatus，它不是 org.springframework.http.HttpStatus。

public interface HttpStatus {
    int SUCCESS = 200;             // 成功
    int CREATED = 201;             // 已创建
    int MOVED_PERM = 301;          // 永久移动
    int BAD_REQUEST = 400;         // 参数错误
    int UNAUTHORIZED = 401;        // 未授权
    int FORBIDDEN = 403;           // 禁止访问
    int NOT_FOUND = 404;           // 未找到
    int ERROR = 500;               // 服务器错误
    int WARN = 601;                // 业务警告（自定义）
}

为什么要自建一套而不是直接用 Spring 的？

  • 解耦 Spring Web 依赖：core 模块是最底层，不应强依赖 Spring Web。自建 HttpStatus 只需要 int 常量，不引入额外依赖
  • 扩展自定义码：Spring 的 HttpStatus 枚举没有 601，而业务需要 WARN 码
  • 轻量级：int 常量比枚举更轻，传递给 R 对象时不需要转换

五、BusinessStatusEnum — 内置状态机的流程枚举

BusinessStatusEnum 位于 org.dromara.common.core.enums.BusinessStatusEnum，它不仅仅是 7 个状态值的数据枚举，更像一个迷你状态机引擎：

public enum BusinessStatusEnum {
    CANCEL("cancel", "已撤销"),
    DRAFT("draft", "草稿"),
    WAITING("waiting", "待提交"),
    FINISH("finish", "已完成"),
    INVALID("invalid", "已作废"),
    BACK("back", "已退回"),
    TERMINATION("termination", "已终止");

    // === 状态机校验方法 ===
    public static void checkStartStatus(String status) {
        // CANCEL、DRAFT、INVALID、FINISH、TERMINATION 不能启动
        if (CANCEL.getStatus().equals(status) || DRAFT.getStatus().equals(status)
            || INVALID.getStatus().equals(status) || FINISH.getStatus().equals(status)
            || TERMINATION.getStatus().equals(status)) {
            throw new ServiceException("该状态不能启动流程");
        }
    }

    // 类似的方法还有 checkCancelStatus / checkBackStatus / checkInvalidStatus
}

状态转换矩阵：

当前状态可→启动可→撤销可→退回可→作废
草稿 DRAFT✗ 否✗ 否✗ 否✗ 否
待提交 WAITING✔ 是✔ 是✔ 是✔ 是
已退回 BACK✔ 是✔ 是✔ 是✔ 是
已撤销 CANCEL✔ 是✗ 否✗ 否✗ 否

枚举中还维护了 STATUS_MAP 缓存，将状态字符串映射到枚举实例，提供 O(1) 查找和辅助方法：

// 辅助分组方法
public static List<String> runningStatus() {
    // 返回 [DRAFT, WAITING, BACK, CANCEL] — 流程仍在运行中的状态
}
public static List<String> finishStatus() {
    // 返回 [FINISH, INVALID, TERMINATION] — 流程已终结的状态
}

七、检验一下（提取练习）

Q1. core 模块的常量为什么用 interface 而不是 class？

A. 因为 interface 性能更好 B. 因为 interface 字段隐式 public static final，写法更简洁，且天生不能实例化 C. 因为作者不知道可以用 class D. 因为 interface 支持多继承
看答案B。interface 的字段自动获得 public static final 修饰符，省去样板代码。同时 interface 不能 new，天然防实例化——如果用 class，还需要手动写 private 构造函数。

Q2. CacheNames 中 SYS_USER_NAME = "sys_user_name#30d" 的 #30d 表示什么？

A. 缓存条目的唯一 ID B. 缓存的 TTL（过期时间）为 30 天 C. 最大缓存 30 条数据 D. 这是注释信息，不影响行为
看答案B。CacheNames 的字符串编码格式为 name#ttl#maxIdle#maxSize#local，#30d 位于第二个字段，表示 TTL=30 天。系统解析时会按 # 分割取各字段。

Q3. 关于 BusinessStatusEnum 的状态机，以下哪个说法正确？

A. 任何状态都可以启动流程 B. DRAFT（草稿）状态可以直接撤销 C. checkStartStatus() 在校验时会拒绝 INVALID（已作废）和 FINISH（已完成）状态 D. BusinessStatusEnum 只是普通枚举，没有任何校验逻辑
看答案C。checkStartStatus() 明确拒绝 CANCEL、DRAFT、INVALID、FINISH、TERMINATION 五种状态的启动请求，只有 WAITING、BACK 等运行态才能启动。

Q4. org.dromara.common.core.constant.HttpStatus 和 org.springframework.http.HttpStatus 是什么关系？

A. 前者继承后者 B. 完全独立的两套体系——前者是 int 接口常量，后者是 Spring 的 enum C. 前者是后者的包装类 D. 它们是同一个类
看答案B。core 模块的 HttpStatus 是自建的 interface + int 常量集合，不依赖 Spring Web。这是为了保持 core 模块的轻量性，且可以自由扩展自定义状态码（如 WARN=601）。
