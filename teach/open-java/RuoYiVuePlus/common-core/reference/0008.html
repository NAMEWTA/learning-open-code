扩展指南一：添加工具类

步骤操作检查点
1创建 utils/XxxUtils.java放在 core 模块 utils 包下
2@NoArgsConstructor(access=PRIVATE)私有构造，防实例化
3所有方法 public static纯工具类，不持有状态
4有 Hutool 同名类则 extends继承优先，扩展其次
5加 @Slf4j，异常用 ServiceException统一日志和异常风格
6不 import 业务实体类保持 core 的纯技术性

扩展指南二：添加枚举

模板A (90%): 双字段 code+desc @Getter @AllArgsConstructor enum XxxEnum { A("0","描述1"), B("1","描述2"); } 模板B: 反向查找 public static XxxEnum getByCode(String code) { for (XxxEnum e : values()) if(e.getCode().equals(code)) return e; throw new ServiceException("无效值: {}", code); } 模板C: 状态机 (参考 BusinessStatusEnum) 仅在状态流转需校验时使用，含 checkXxxStatus() + runningStatus() 等方法

扩展指南三：添加异常

90% 场景 → 直接用 ServiceException: throw new ServiceException("消息{}", arg); throw new ServiceException("消息", errorCode); 10% 场景 → extends BaseException (需 i18n + 按类型捕获): class PayException extends BaseException { public PayException(String code, Object... args) { super("pay", code, args); } } 然后在 i18n 文件中配: pay.xxx=支付异常: {0} 不需要给每种错误建异常类！ServiceException + message 已覆盖绝大多数场景。

扩展指南四：添加校验注解

5 步创建 @Phone 注解: Step1: Phone.java — @Constraint(validatedBy={PhoneValidator.class}) 包含 message(), groups(), payload() 三个标准属性 Step2: PhoneValidator.java — implements ConstraintValidator<Phone, String> isValid() 中: if (blank) return true; (空值放行约定!) Step3: DTO 使用: @Phone(message="手机号格式不正确") Step4: Controller: @Validated @RequestBody DTO dto Step5: 手动触发: ValidatorUtils.validate(dto) 如需查 Spring Bean → SpringUtils.getBean(Xxx.class) (桥接模式)

记忆锚点

  • core 是被依赖方 — 永远不依赖业务模块，只定义接口契约（如 DictService）。
  • 工具类三要素 — 私有构造 + 纯静态方法 + @Slf4j。
  • 异常选型 — ServiceException 覆盖 90%，BaseException 覆盖需 i18n 的 10%。
  • 校验注解约定 — 空值放行 + message/groups/payload 三个标准属性。
  • 枚举模板 — @Getter + @AllArgsConstructor + private final 字段，90% 用双字段模式。
