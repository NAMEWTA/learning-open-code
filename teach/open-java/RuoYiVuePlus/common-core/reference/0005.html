速查 05 · 枚举体系设计模式

配套课程：lessons/0005.html · DeviceType、LoginType、UserStatus、BusinessStatusEnum

一句话定位

core 的 7 个枚举展示了三种进阶设计模式：字段化枚举 → 策略枚举 → 状态机枚举。全部使用 @Getter + @AllArgsConstructor。

四种设计模式

模式代表枚举特点复杂度
字段化枚举DeviceType, UserStatus枚举值挂载自定义字段低
反向查找UserType从字符串反查枚举实例低
策略枚举LoginType枚举值携带行为配置 (i18n key)中
状态机枚举BusinessStatusEnum内置状态流转校验+分类方法高

7 个枚举一览

枚举字段模式
DeviceTypedevice (String): pc/app/xcx/social字段化
UserTypeuserType (String) + getUserType(str)反向查找
UserStatuscode(String) + info(String): 0/正常双字段
PushSourceEnumsource (String)字段化
PushTypeEnumtype (String)字段化
LoginTyperetryLimitExceed + retryLimitCount (i18n key)策略
BusinessStatusEnumstatus + desc + 8个方法状态机

BusinessStatusEnum — 状态机核心

7 种状态: CANCEL, DRAFT, WAITING, FINISH, INVALID, BACK, TERMINATION 状态分类: runningStatus() → [DRAFT, WAITING, BACK, CANCEL] (运行中) finishStatus() → [FINISH, INVALID, TERMINATION] (终结态) 状态机校验: checkStartStatus(status) → 已提交/已完成/已作废的不能启动 checkCancelStatus(status) → 已撤销/已完成/已终止的不能撤销 checkBackStatus(status) → 已退回/已完成/已终止的不能驳回 checkInvalidStatus(status) → 已完成/已作废/已终止的不能作废 isDraftOrCancelOrBack(status) → 草稿/已撤销/已退回时可编辑 STATUS_MAP: ConcurrentMap<String, BusinessStatusEnum> 按 status 字段索引

LoginType — 策略枚举

PASSWORD → retryLimitExceed="user.password.retry.limit.exceed" SMS → retryLimitExceed="sms.code.retry.limit.exceed" EMAIL → retryLimitExceed="email.code.retry.limit.exceed" XCX → retryLimitExceed=""（小程序暂不支持重试） 使用: MessageUtils.message(LoginType.SMS.getRetryLimitExceed()) → "短信验证码重试次数超限" 设计洞察: 用数据表替 if-else。新增登录方式只加枚举值，不改业务代码。

记忆锚点

  • 三件套：@Getter + @AllArgsConstructor + private final 字段。
  • 数据库存 code，前端用 info，name() 只在代码逻辑用 — UserStatus 的设计约定。
  • STATUS_MAP — ConcurrentMap 做 O(1) 枚举值查找，线程安全。
  • Rich Enum — BusinessStatusEnum 是教科书级「富枚举」，枚举 = 数据 + 行为 + 校验。
