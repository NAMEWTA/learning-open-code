第 3 课 · LogAspect 切面处理全流程：从拦截到发布事件 核心引擎

本节目标：逐行读懂 LogAspect.java 的 276 行代码，理解从 AOP 拦截到 SpringUtils.context().publishEvent() 的完整链路——包括参数收集、异常处理、长度限制、过滤机制的每一个设计决策。

一、LogAspect 的定位：整个模块的「发动机」

如果把 ruoyi-common-log 模块比作一辆车：

  • @Log 注解 = 方向盘（你告诉它去哪）
  • LogAspect = 发动机（一切动力来源）
  • OperLogEvent = 传送带（把数据运到目的地）
  • SysOperLogServiceImpl.recordOper() = 仓库（最终存储）

LogAspect 坐在这条链路的正中间——它拦截被 @Log 标记的方法，收集运行时上下文（IP、用户、参数、结果、耗时），组装成 OperLogEvent，然后通过 Spring 事件机制「发射」出去。后面的监听器只管接收，完全不知道前面发生了什么。

一句话定位：LogAspect 是一个 Spring AOP 环绕切面，它把「注解声明 + 运行时上下文收集 + 事件发布」三个动作串联在一起，是整个操作日志模块唯一的调度中心。

二、@AutoConfiguration 标记了什么？为什么不需要 @ComponentScan？

在 Spring Boot 2.7 之前，要自动注册一个切面，你需要：

// 旧方式：需要 META-INF/spring.factories
@Configuration
@ComponentScan("org.dromara.common.log.aspect") // ← 需要注解扫描
public class LogAutoConfiguration { }

RuoYi-Vue-Plus 用的是 Spring Boot 3.x 的新机制：

@Slf4j
@Aspect          // ← 告诉 Spring：这是一个切面
@AutoConfiguration  // ← 替代 @Configuration，自动注册到 Spring 容器
public class LogAspect { ... }
对比项旧方式 (@Configuration + @ComponentScan)新方式 (@AutoConfiguration)
注册方式需要 spring.factories 或 @ComponentScanSpring Boot 自动发现并注册
文件数量需要一个单独的配置类切面类自己就是配置入口，少一个文件
排除控制需要 @ConditionalOnProperty 单独声明可以用 spring.autoconfigure.exclude 排除
适用版本Spring Boot 2.xSpring Boot 3.x（使用了 spring-boot-autoconfigure 的新 SPI）
关键点：@AutoConfiguration 让 LogAspect 既是切面又是配置类，一个注解干两件事。Spring Boot 3.x 的自动配置机制会自动读取 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件，不需要 @ComponentScan 扫描。

三、切点表达式 "@annotation(controllerLog)" 如何工作

来看切面的核心入口：

@Around(value = "@annotation(controllerLog)")
public Object doAround(ProceedingJoinPoint joinPoint, Log controllerLog) throws Throwable {

这个切点表达式的含义：

  • @Around：环绕通知，在目标方法执行前后都能插入逻辑
  • @annotation(controllerLog)：匹配任何标有 @Log 注解的方法
  • 括号内的 controllerLog 是参数名，Spring AOP 会把匹配到的 @Log 注解实例自动注入

实际效果：当一个 Controller 写这样的代码时——

@Log(title = "用户管理", businessType = BusinessType.DELETE)
@DeleteMapping("/{userIds}")
public R<Void> remove(@PathVariable Long[] userIds) {
    userService.deleteUserByIds(userIds);
    return R.ok();
}

AOP 代理会自动拦截：①执行前开秒表 → ②执行目标方法 → ③成功后收集上下文 → ④组装事件 → ⑤发布。如果第②步抛异常，走异常分支。

四、doAround 方法的双路径：正常返回 vs 异常

这是整个切面里最精简也最重要的逻辑分叉：

@Around(value = "@annotation(controllerLog)")
public Object doAround(ProceedingJoinPoint joinPoint, Log controllerLog) throws Throwable {
    StopWatch stopWatch = new StopWatch();
    stopWatch.start();
    try {
        Object jsonResult = joinPoint.proceed();             // ← 执行目标方法
        handleLog(joinPoint, controllerLog, null, jsonResult, stopWatch);  // 路径 A
        return jsonResult;                                    // ← 返回原结果
    } catch (Exception e) {
        handleLog(joinPoint, controllerLog, e, null, stopWatch);           // 路径 B
        throw e;                                               // ← 原样抛出异常
    }
}
用户请求 → Controller方法(@Log标记) │ ▼ doAround() 开始 │ StopWatch.start() ← 开始计时 │ joinPoint.proceed() ← 执行真正的业务方法 ╱ ╲ 成功 异常 │ │ handleLog(e=null) handleLog(e=异常) jsonResult=返回值 jsonResult=null │ │ return jsonResult throw e (原样抛出) │ │ 监听器收到事件 监听器收到事件(status=FAIL) 状态=SUCCESS 包含错误消息

两个关键设计：

  1. 不吞异常：catch(Exception e) 后记录日志，然后 throw e 原样抛出——日志记录失败不影响业务流程。
  2. 无论成败都记录：成功时 jsonResult 有值、e 为 null；失败时 jsonResult 为 null、e 有值。两个分支都调用 handleLog，只是参数不同。

六、参数收集逻辑详解：paramsMap vs argsArrayToString

这是整个切面里最复杂的逻辑分支，定在 setRequestValue() 方法中：

private void setRequestValue(JoinPoint joinPoint, OperLogEvent operLog,
                             String[] excludeParamNames) {
    Map<String, String> paramsMap = ServletUtils.getParamMap(ServletUtils.getRequest());
    String requestMethod = operLog.getRequestMethod();
    if (MapUtil.isEmpty(paramsMap)
        && StringUtils.equalsAny(requestMethod, HttpMethod.PUT.name(),
                                 HttpMethod.POST.name(), HttpMethod.DELETE.name())) {
        // 路径 A：paramsMap 为空 + 请求方式是 PUT/POST/DELETE
        String params = argsArrayToString(joinPoint.getArgs(), excludeParamNames);
        operLog.setOperParam(limit(params, MAX_CONTENT_LENGTH));
    } else {
        // 路径 B：paramsMap 不为空，或 GET 请求
        MapUtil.removeAny(paramsMap, SystemConstants.EXCLUDE_PROPERTIES);
        MapUtil.removeAny(paramsMap, excludeParamNames);
        operLog.setOperParam(limit(JsonUtils.toJsonString(paramsMap), MAX_CONTENT_LENGTH));
    }
}

为什么有两条路径？

场景走哪条路原因
GET 请求（参数在 URL 上）路径 B（paramsMap）GET 参数自然在 request.getParameterMap() 中，用 paramsMap 即可
POST 表单提交路径 B（paramsMap）表单数据也在 request.getParameterMap() 中
POST JSON（@RequestBody）路径 A（argsArrayToString）JSON 体不在 request.getParameterMap() 中，只能从方法参数里取
PUT JSON（@RequestBody）路径 A（argsArrayToString）同上
DELETE JSON（@RequestBody）路径 A（argsArrayToString）同上

核心判断逻辑：paramsMap 为空 并且 请求方式是 PUT/POST/DELETE 时，说明请求体是 JSON（没有出现在 URL 参数中），需要从方法参数序列化。否则直接用 paramsMap。

七、参数过滤机制：isFilterObject / isFilterValue

argsArrayToString() 在拼接参数时会调用 isFilterObject() 过滤不合适的对象：

public boolean isFilterObject(final Object o) {
    Class<?> clazz = o.getClass();
    if (clazz.isArray()) {
        if (MultipartFile.class.isAssignableFrom(clazz.getComponentType())) {
            return true;  // ← MultipartFile[] 直接过滤
        }
        // 递归检查数组中的每个元素
        for (int i = 0; i < Array.getLength(o); i++) {
            if (isFilterValue(Array.get(o, i))) return true;
        }
        return false;
    } else if (Collection.class.isAssignableFrom(clazz)) {
        // 遍历集合的每个元素
        for (Object value : (Collection<?>) o) {
            if (isFilterValue(value)) return true;
        }
    } else if (Map.class.isAssignableFrom(clazz)) {
        // 遍历 Map 的每个 value
        for (Object value : ((Map<?,>) o).values()) {
            if (isFilterValue(value)) return true;
        }
    }
    return isFilterValue(o);  // 最终判断单个对象
}

private boolean isFilterValue(Object value) {
    return value instanceof MultipartFile       // 文件上传对象
        || value instanceof HttpServletRequest   // 请求对象（包含大量不需要的信息）
        || value instanceof HttpServletResponse  // 响应对象
        || value instanceof BindingResult;       // 参数校验结果
}

为什么过滤这 4 种类型？

过滤类型原因
MultipartFile文件上传是二进制流，序列化无意义且可能引起 OOM；数组场景还做了特殊处理——如果数组组件类型就是 MultipartFile，整组跳过递归
HttpServletRequest请求对象内部关联大量容器对象（Session、ServletContext、RequestDispatcher），序列化会引发循环引用或输出大量无用数据
HttpServletResponse同上，响应对象也不应被记录
BindingResultSpring MVC 参数校验的结果对象，里面包含校验错误信息，但业务日志关注的是业务参数而非校验元数据
设计启示：这个方法对数组、集合、Map 做了三层递归检查——即使你传了一个 List<HttpServletRequest> 进来，它也能在遍历时发现并过滤。这是一种「防御性编程」，确保无论参数结构多复杂，不该记录的对象一个都不会漏。

八、excludeParamNames 的工作原理

@Log 注解有一个 excludeParamNames 属性：

@Log(title = "用户管理", businessType = BusinessType.INSERT,
     excludeParamNames = {"avatar", "remark"})  // ← 排除头像和备注
@PostMapping
public R<Void> add(@Validated @RequestBody SysUserBo user) { ... }

排除流程分两步：

  1. 系统级排除：SystemConstants.EXCLUDE_PROPERTIES = {"password", "oldPassword", "newPassword", "confirmPassword"}——所有日志默认排除密码字段
  2. 注解级排除：excludeParamNames 属性指定的字段名

在路径 B（paramsMap）中，两者合并后用 MapUtil.removeAny() 删除；在路径 A（argsArrayToString）中，两者合并后传给 JsonUtils.toJsonStringExcludeFields() 作为排除字段列表。

安全提醒：密码类字段（password、oldPassword、newPassword、confirmPassword）在 SystemConstants.EXCLUDE_PROPERTIES 中硬编码，无论你写不写注解都会自动排除。这是一种「默认安全」策略——即使开发者忘记排除密码，框架也不会把密码记录到日志表中。

九、三个长度限制的设计考量

常量值限制对象设计考量
MAX_URL_LENGTH255请求URL数据库 varchar(255) 字段的常见上限；URL 超过 255 字符通常意味着带了大量 query 参数，记录完整 URL 意义不大
MAX_CLIENT_KEY_LENGTH32客户端标识ClientKey 设计为短标识（如 "eip"、"pc"），32 字符足够且防止异常请求头注入超长值
MAX_CONTENT_LENGTH3800请求参数、响应结果、错误消息最大 3800 字符——平衡了「能看到完整请求」和「不撑爆数据库」两个需求。MySQL TEXT 类型可存 65535 字符，3800 是保守上限，考虑万一用了 varchar 存储也能兼容

截断工具方法：

private String limit(String value, int maxLength) {
    return StringUtils.substring(value, 0, maxLength);
}

简单但有效——如果 value 为 null 则返回空字符串，不会抛 NPE。

十、异常安全：外层 try-catch 保护

回到 handleLog 的整体结构，最外层有一个容易被忽略的 try-catch：

protected void handleLog(...) {
    try {
        // ... 7 个步骤的全部逻辑 ...
    } catch (Exception exp) {
        log.error("记录操作日志异常", exp);  // ← 唯一的处理：打日志
    }
}

这意味着什么？如果在第 2 步获取 LoginUser 时 Redis 超时了，在第 6 步序列化 JSON 时循环引用了，在任何一步出问题——最坏的情况就是「日志没记录成功」，业务请求照常返回。

这是一个经典的「非功能需求不应影响功能需求」的设计。操作日志是辅助功能（横切关注点），它的失败不应该导致用户的业务操作失败。用户删了一条数据，数据确实删了，只是日志表里没记录下来——这是可以接受的权衡。

十一、Loader 与运行时上下文获取

LogAspect 依赖三个「数据源」来获取运行时信息，它们各自从不同的地方拿数据：

工具类获取的信息数据来源底层机制
ServletUtils HttpServletRequest、IP、URI、请求参数 RequestContextHolder ThreadLocal 绑定——每个请求线程在进入 DispatcherServlet 时，Spring 把 request 绑定到线程的 ThreadLocal 中
LoginHelper 登录用户（LoginUser）、客户端标识 Sa-Token 的 StpUtil Sa-Token 在用户登录时将 LoginUser 存入 Redis（key=token），通过当前线程的 token 取值。本质上也是 ThreadLocal（token 绑定在请求上下文中）
SpringUtils ApplicationContext（用于 publishEvent） ApplicationContextAware Spring 容器启动时回调 setApplicationContext()，静态持有 ApplicationContext 引用。不依赖线程，全局唯一
为什么这些工具类可以「无参调用」？
三个工具类共享一个前提假设：请求在单线程内处理。Tomcat 线程池中，每个请求独占一个线程，ThreadLocal 变量自然不会串。但要注意：如果你在业务代码中新开了线程，ThreadLocal 就失效了——这也是为什么 @Async 的监听器需要重新获取 request 信息，不能直接用 ThreadLocal 数据。

十二、检验一下（提取练习）

Q1. doAround() 方法中，如果目标方法抛出异常，handleLog 会被调用吗？

A. 不会，异常直接抛出 B. 会，且传入 e=null, jsonResult=null C. 会，且传入 e=异常对象, jsonResult=null D. 只有 BusinessException 才会调用
看答案C。catch 块中调用 handleLog(joinPoint, controllerLog, e, null, stopWatch)——异常作为第三个参数传入，jsonResult 传 null。然后 throw e 原样抛出。

Q2. POST JSON 请求（@RequestBody）的参数为什么走 argsArrayToString 而不是 paramsMap？

A. JSON 格式复杂，paramsMap 无法序列化 B. @RequestBody 的参数不在 request.getParameterMap() 中，paramsMap 为空 C. 这是一个 bug，应该走 paramsMap D. argsArrayToString 效率更高
看答案B。@RequestBody 将 JSON 反序列化为方法参数，这个数据不会出现在 Servlet 的 ParameterMap 中。所以 MapUtil.isEmpty(paramsMap) 为 true，加上请求方式是 POST/PUT/DELETE，触发 argsArrayToString 路径直接从方法参数序列化。

Q3. 如果 handleLog 内部的 7 个步骤中某一步抛出了异常，会发生什么？

A. 异常会中断业务请求，用户看到 500 错误 B. 异常被 catch 后打印 error 日志，业务请求不受影响 C. 异常被 catch 后静默忽略，日志也不打 D. 只有 RuntimeException 会被捕获
看答案B。handleLog 最外层的 catch(Exception exp) 捕获所有异常，只执行 log.error(...) 打印日志，不会向上抛出。业务请求正常返回。这是「日志失败不影响业务」的底线设计。

Q4. excludeParamNames 和 SystemConstants.EXCLUDE_PROPERTIES 是什么关系？

A. 互斥，二选一使用 B. 合并使用：系统级排除 + 注解级排除都会生效 C. excludeParamNames 优先级更高，覆盖系统级 D. 系统级只在 paramsMap 路径生效
看答案B。在两条路径中都是合并使用。在 paramsMap 路径中用 MapUtil.removeAny 分别删除；在 argsArrayToString 路径中用 ArrayUtil.addAll 合并后传给序列化方法。密码字段永远不会出现在操作日志中。
进阶思考：为什么用 ordinal() 存枚举而不是 name()？ 存 ordinal()（整数）比存 name()（字符串）有两个好处：①数据库存储更紧凑，int 占 4 字节，varchar 至少占 10+ 字节；②查询效率更高，整型索引比字符串索引快。代价是枚举顺序不能随意调整——如果有人在中间插入新枚举值，历史数据的 ordinal 就对不上了。所以你会看到 BusinessType 只有 10 个值且注释写了序号（0 其它、1 新增、2 修改……），这是一种约定大于配置的策略。
