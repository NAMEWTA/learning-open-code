第 1 课 · 鸟瞰：@Log 注解 + Spring 事件 = 零侵入操作日志

本课任务： 看懂 ruoyi-common-log 模块的全局架构 —— 它有哪些文件？它们怎么协作？一条 Controller 请求如何最终变成数据库里的 sys_oper_log 记录？读完本课，你将能向同事画出完整链路图。

1. 模块全景：7 个 Java 文件，3 层架构

打开模块源码目录，你会发现它只有 7 个 Java 文件，结构极其精简：

文件层级角色
annotation/Log.java 注解层 声明式标记 —— 加在 Controller 方法上即生效
enums/BusinessType.java 注解层 定义 10 种业务操作类型（OTHER、INSERT、UPDATE…）
enums/BusinessStatus.java 注解层 定义 2 种操作结果（SUCCESS、FAIL）
enums/OperatorType.java 注解层 定义 3 种操作人类别（OTHER、MANAGE、MOBILE）
aspect/LogAspect.java 切面层 AOP 核心引擎 —— 拦截 @Log 方法，提取上下文，发布事件
event/OperLogEvent.java 事件层 操作日志事件载体 —— 20+ 字段，Spring Event 传递
event/LoginInfoEvent.java 事件层 登录日志事件载体 —— 5 字段，手动发布（非 AOP 拦截）
关键观察： 整个模块没有一行 insert into sys_oper_log 的 SQL，也没有一个 Mapper 或 Repository。日志写入数据库的职责完全委托给外部监听器 —— 这是刻意为之的架构设计。

2. 一句话定位

ruoyi-common-log 不是日志框架，而是 "把一条 Controller 方法调用自动变成一条数据库日志" 的声明式中间件。

它不替代 Logback / Log4j（那些是给开发/运维看的排查日志），它专做 业务操作审计：谁在什么时间，用什么设备，调了哪个接口，传了什么参数，返回了什么结果，花了多少毫秒。

3. 核心设计模式：声明式 + AOP + 事件

模块拆成三个正交的设计维度，每个维度解决一类问题：

1 声明式注解 @Log annotation/Log.java
业务开发者只需在 Controller 方法上加一行注解，不需要写任何日志代码。标题、业务类型、操作人类别等元信息全部声明式配置。
2 AOP 拦截 LogAspect aspect/LogAspect.java:65-77
用 @Around("@annotation(controllerLog)") 包裹目标方法。方法执行前启动秒表，执行后从请求上下文提取用户、IP、参数、耗时，组装成 OperLogEvent。
3 Spring 事件解耦 OperLogEvent event/OperLogEvent.java — LogAspect.java:130
切面不直接写数据库，而是 SpringUtils.context().publishEvent(operLog) 发布事件。由外部的 @EventListener 监听器消费，写入 sys_oper_log 表。

4. 为什么用事件而不是直接写库？

如果 LogAspect 在 handleLog() 里直接调 Mapper 插入数据库，架构上会引入三个问题：

问题事件方案如何解决
存储紧耦合 —— 假如将来要换成 Elasticsearch 或 ClickHouse 存日志？ 替换监听器即可，切面代码零改动。监听器是 Spring Bean，随时可替换实现。
监听器无法热插拔 —— 某些场景不需要记录操作日志（如开发环境） 通过 @ConditionalOnProperty 控制监听器是否加载，做到零配置开关。
日志与业务耦合 —— 未获取到用户信息、IP 解析失败等日志异常会污染业务流程 事件发布是异步可选的（可配 @Async），发送失败不影响主业务。切面内部 catch(Exception) 仅打本地 log.error，不向上抛。
架构原则： "切面负责采集，事件负责传递，监听器负责持久化。" 三层职责分明，互不侵入。

5. 一条完整链路：从 Controller 到数据库

下面以一个用户删除数据为例，追踪 @Log 注解从触发到入库的每一步：

[1] Controller 方法上标记 @Log(title="用户管理", businessType=DELETE) │ ▼ [2] LogAspect.doAround() 拦截该方法 (@Around 切面触发) │ 启动 StopWatch 计时 │ 执行 joinPoint.proceed() → 原 Controller 方法执行 │ ▼ [3] handleLog() 组装 OperLogEvent ├─ ServletUtils.getClientIP() → operIp ├─ request.getRequestURI() → operUrl (截断至 255 字符) ├─ LoginHelper.getLoginUser() → userId, deptId, operName, deviceType... ├─ log.title() → title = "用户管理" ├─ log.businessType().ordinal() → businessType = 3 (DELETE) ├─ log.operatorType().ordinal() → operatorType = 1 (MANAGE) ├─ setRequestValue() → operParam (序列化请求参数) ├─ JsonUtils.toJsonString(jsonResult) → jsonResult (序列化返回值) ├─ stopWatch.getDuration().toMillis() → costTime └─ 若异常: e.getMessage() → errorMsg, status = 1 (FAIL) │ ▼ [4] SpringUtils.context().publishEvent(operLog) │ Spring ApplicationContext 广播事件 │ ▼ [5] @EventListener 监听器接收 OperLogEvent │ 调用 Mapper 写入 sys_oper_log 表 │ ▼ [6] 数据库: SELECT * FROM sys_oper_log WHERE title='用户管理';

6. 两个事件的区别

模块里有两个事件类，初学者容易混淆：

OperLogEventLoginInfoEvent
用途 通用操作日志（增删改查、导出等） 登录日志（登录成功/失败）
触发方式 AOP 自动拦截 —— 只要方法上有 @Log 注解，切面自动发布 手动发布 —— 在登录方法里 new LoginInfoEvent() 后手动 publishEvent
字段数量 20+ 字段（用户、部门、IP、URL、参数、返回值、耗时…） 5 字段（username, status, message, request, args）
写入表 sys_oper_log sys_logininfor
设计意图 通用、声明式、零侵入 专用、手动控制、适合登录这种特殊场景
为什么登录不用 @Log？ 登录发生在前端尚未获取 token 的阶段，此时 LoginHelper.getLoginUser() 拿不到用户信息。而且登录成功/失败需要记录不同的状态码和消息 —— 这些信息 AOP 切面很难自动化提取。因此登录日志采用手动 new LoginInfoEvent 的方式，由开发者精确控制每个字段。

7. RuoYi 日志体系的核心定位

总结 ruoyi-common-log 在整个 RuoYi-Vue-Plus 项目中的位置：

  • 声明式 —— 一行 @Log(title="...") 即可启用操作审计，对业务代码零侵入
  • 零侵入 —— Controller 方法不需要注入 Logger、不需要 try-finally 包裹、不需要手动拼接日志字符串
  • 可扩展 —— 通过 Spring Event 机制，你可以新增监听器将日志同时写入 MySQL + Elasticsearch + Kafka，而无需改切面代码
  • 与框架正交 —— 不依赖特定的 ORM（没有 MyBatis 耦合），不依赖特定的 Web 框架（仅依赖 Servlet API）
它是 RuoYi 日志体系的核心引擎。ruoyi-system 模块引入本模块后，只需提供一个 @EventListener 即可完成操作日志的完整生命周期。
延伸思考： 如果你要把操作日志从 MySQL 迁移到 Elasticsearch，需要改哪些代码？答案：0 行切面代码。只需写一个新的 @EventListener，让 Spring 同时把它和原来的监听器都注册上（或替换掉旧监听器），切面和注解完全无感。

课后自测

Q1：LogAspect 在方法执行成功和抛出异常时，分别如何设置 OperLogEvent 的状态？

A. 都用 SUCCESS B. 成功设 SUCCESS（ordinal=0），异常设 FAIL（ordinal=1），并将异常消息写入 errorMsg C. 只记录异常，成功时不记录 D. 依赖外部监听器自行判断
点击查看答案

正确答案：B

参见 LogAspect.java 第 92-117 行：初始化时 operLog.setStatus(BusinessStatus.SUCCESS.ordinal())，catch 块中 operLog.setStatus(BusinessStatus.FAIL.ordinal()) 并 operLog.setErrorMsg(limit(e.getMessage(), MAX_CONTENT_LENGTH))。

Q2：为什么 OperLogEvent 用 ordinal() 存储 businessType 而不是 name()？

A. ordinal() 更快 B. 数据库存 int 比存字符串节省空间且查询效率更高 C. name() 不支持序列化 D. 没有特别原因，只是习惯
点击查看答案

正确答案：B

BusinessType.DELETE.ordinal() 返回 3，存入数据库是一个 tinyint/int 列；如果用 name() 则存入 "DELETE" 字符串。int 类型占用更小、索引更快、且不受枚举重命名影响。详见下节课。

Q3：假设你写了一个新的 Service 方法（非 Controller），想用 @Log 记录操作日志，能直接用吗？

A. 能，@Log 注解可以加在任何方法上 B. 不能，LogAspect 用 @Around 拦截，需要该方法能被 Spring AOP 代理到（即必须是 Spring Bean 的 public 方法） C. 能，但需要改切面表达式 D. 不能，@Log 仅限 Controller 层
点击查看答案

正确答案：B

@Log 注解本身可以用在任何方法上（@Target({PARAMETER, METHOD})），但 LogAspect 使用的是 @Around("@annotation(controllerLog)") —— 这是一个基于注解的切点，不限定包名或类名。因此理论上任何被 Spring 代理到的 Bean 的 public 方法都可以被拦截。但实践中，日志上下文提取逻辑（如 ServletUtils.getRequest()）依赖 HTTP 请求上下文，在非 Controller 层可能获取不到部分信息。
