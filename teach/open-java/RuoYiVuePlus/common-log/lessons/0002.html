第 2 课 · @Log 注解定义与三枚举体系

本课任务： 逐行拆解 @Log 注解的 6 个属性、3 个配套枚举的完整定义，以及它们在实际 Controller 中如何组合使用。读完本课，你将能独立为任何 Controller 方法添加审计日志。

1. @Log 注解完整拆解

先看注解的完整源码（annotation/Log.java，共 48 行）：

@Target({ElementType.PARAMETER, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface Log {

    /** 模块 */
    String title() default "";

    /** 功能 */
    BusinessType businessType() default BusinessType.OTHER;

    /** 操作人类别 */
    OperatorType operatorType() default OperatorType.MANAGE;

    /** 是否保存请求的参数 */
    boolean isSaveRequestData() default true;

    /** 是否保存响应的参数 */
    boolean isSaveResponseData() default true;

    /** 排除指定的请求参数 */
    String[] excludeParamNames() default {};
}

1.1 元注解解读

元注解含义为什么这样设计
@Target({PARAMETER, METHOD}) 可标注在方法和参数上 方法级是常规用法（标记整个接口）。参数级用于特殊场景：同一个 Controller 方法接收多个参数，只想排除某个特定参数的序列化（配合 excludeParamNames 按名称过滤）。Spring AOP 的 @annotation() 切点默认只匹配方法上的注解，参数级注解需要额外处理。
@Retention(RUNTIME) 保留到运行时 必须 RUNTIME —— LogAspect 在运行时通过反射读取注解属性（log.title()、log.businessType()），如果设为 CLASS 或 SOURCE，切面将无法获取注解信息。
@Documented 纳入 Javadoc 生成的 API 文档中会显示该注解，便于阅读者知道哪些方法开启了操作审计。
为什么 @Target 包含 PARAMETER？ 这是 RuoYi 的一种扩展预留。标准用法是加在方法上，但如果你需要过滤特定参数（比如密码字段），可以通过在参数前加 @Log(excludeParamNames={"password"}) 的方式更细粒度地控制。实践中绝大多数场景只用到 METHOD。

2. 三枚举体系

2.1 BusinessType —— 10 种业务操作

源码：enums/BusinessType.java（共 58 行）

public enum BusinessType {
    OTHER,    // 其它
    INSERT,   // 新增    → ordinal = 1
    UPDATE,   // 修改    → ordinal = 2
    DELETE,   // 删除    → ordinal = 3
    GRANT,    // 授权    → ordinal = 4
    EXPORT,   // 导出    → ordinal = 5
    IMPORT,   // 导入    → ordinal = 6
    FORCE,    // 强退    → ordinal = 7
    GENCODE,  // 生成代码 → ordinal = 8
    CLEAN,    // 清空数据 → ordinal = 9
}
枚举值ordinal典型使用场景
OTHER0未分类的通用操作（默认值）
INSERT1新增用户、新增角色、新增菜单
UPDATE2编辑用户信息、修改配置
DELETE3删除用户、批量删除
GRANT4分配角色、授权数据权限
EXPORT5导出 Excel、导出 PDF
IMPORT6导入用户数据、批量导入
FORCE7强制踢出在线用户
GENCODE8代码生成器生成代码
CLEAN9清空操作日志、清空登录日志

2.2 OperatorType —— 3 种操作人类别

源码：enums/OperatorType.java（共 23 行）

public enum OperatorType {
    OTHER,   // 其它      → ordinal = 0
    MANAGE,  // 后台用户   → ordinal = 1
    MOBILE,  // 手机端用户 → ordinal = 2
}

这个枚举区分了操作来源。当一个后台管理员通过 Web 页面删除用户时，operatorType = MANAGE(1)；当手机 App 用户修改个人资料时，operatorType = MOBILE(2)。RuoYi-Vue-Plus 默认的后台 Controller 全部使用默认值 MANAGE。

2.3 BusinessStatus —— 2 种操作结果

源码：enums/BusinessStatus.java（共 18 行）

public enum BusinessStatus {
    SUCCESS,  // 成功 → ordinal = 0
    FAIL,     // 失败 → ordinal = 1
}

这个枚举不需要你在 @Log 注解里指定 —— 它由 LogAspect 自动判断：方法正常返回时设为 SUCCESS，抛出异常时在 catch 块中覆写为 FAIL。

注意： BusinessStatus 不是 @Log 注解的属性，而是作为 OperLogEvent 的 status 字段值使用。它在 LogAspect.java:93 初始化时默认设为 SUCCESS，在 LogAspect.java:115 被异常路径覆写为 FAIL。

3. 为什么用 ordinal() 而不是 name() 存入数据库？

在 LogAspect 中，所有枚举到数据库的转换都使用 .ordinal()：

// LogAspect.java:93
operLog.setStatus(BusinessStatus.SUCCESS.ordinal());       // → 0

// LogAspect.java:148
operLog.setBusinessType(log.businessType().ordinal());     // → 0~9

// LogAspect.java:152
operLog.setOperatorType(log.operatorType().ordinal());     // → 0~2

选择 ordinal() 存入 int 列而非 name() 存入 varchar 列，理由有三：

维度ordinal() → intname() → varchar
存储空间tinyint 占 1 字节"DELETE" 占 6+ 字节
索引效率整型比较极快字符串比较较慢
枚举稳定性⚠️ 不可随意调整枚举声明顺序✅ 重命名枚举值不影响数据
ordinal() 的风险： 如果有一天你在 BusinessType 枚举中间插入一个新值（比如在 INSERT 和 UPDATE 之间加一个 UPSERT），所有后续值的 ordinal 都会 +1，导致历史日志数据错乱。因此，只能在枚举末尾追加新值，绝不能在中间插入。这是一个需要团队规范的约定。

4. 实际使用示例

以下是在 RuoYi Controller 中 @Log 注解的典型用法：

// 示例 1：新增操作 —— 最简形式
@Log(title = "用户管理", businessType = BusinessType.INSERT)
@PostMapping
public R<Void> add(@Validated @RequestBody SysUserBo user) {
    userService.insertUser(user);
    return R.ok();
}

// 示例 2：修改操作 —— 排除密码字段
@Log(title = "用户管理", businessType = BusinessType.UPDATE,
     excludeParamNames = {"password", "oldPassword", "newPassword"})
@PutMapping
public R<Void> edit(@Validated @RequestBody SysUserBo user) {
    userService.updateUser(user);
    return R.ok();
}

// 示例 3：删除操作 —— 不保存返回值（减少日志量）
@Log(title = "用户管理", businessType = BusinessType.DELETE,
     isSaveResponseData = false)
@DeleteMapping("/{userIds}")
public R<Void> remove(@PathVariable Long[] userIds) {
    userService.deleteUserByIds(userIds);
    return R.ok();
}

// 示例 4：导出操作 —— 不保存请求参数（导出参数无审计价值）
@Log(title = "用户管理", businessType = BusinessType.EXPORT,
     isSaveRequestData = false)
@PostMapping("/export")
public void export(@RequestBody UserQuery query, HttpServletResponse response) {
    userService.exportExcel(query, response);
}

// 示例 5：使用默认值 —— 全部为 OTHER + MANAGE
@Log(title = "数据统计")
@GetMapping("/statistics")
public R<Map<String, Object>> statistics() {
    return R.ok(statService.getStatistics());
}

5. excludeParamNames 的妙用：过滤敏感参数

这是一个容易被忽视但非常重要的安全特性。看 LogAspect 中的过滤逻辑：

// LogAspect.java:172-182
private void setRequestValue(JoinPoint joinPoint, OperLogEvent operLog,
                              String[] excludeParamNames) throws Exception {
    Map<String, String> paramsMap = ServletUtils.getParamMap(ServletUtils.getRequest());
    ...
    // 从 paramsMap 中移除排除字段
    MapUtil.removeAny(paramsMap, SystemConstants.EXCLUDE_PROPERTIES);
    MapUtil.removeAny(paramsMap, excludeParamNames);
    operLog.setOperParam(limit(JsonUtils.toJsonString(paramsMap), MAX_CONTENT_LENGTH));
}

两层过滤机制：

  1. 系统级排除 —— SystemConstants.EXCLUDE_PROPERTIES 定义了框架级别的敏感字段（如 password、pwd 等），对所有日志全局生效
  2. 注解级排除 —— excludeParamNames 让业务开发者针对特定接口追加排除项（如 "idCard"、"bankAccount" 等自定义敏感字段）
最佳实践： 任何接收密码、手机号、身份证、银行卡、Token 等敏感信息的接口，务必在 @Log 中配置 excludeParamNames。这不是性能问题，是安全合规问题 —— 操作日志表可能被审计、导出、甚至被 SQL 注入攻击读取，绝不能在其中存放明文敏感数据。

6. isSaveRequestData / isSaveResponseData 的性能考量

两个 boolean 开关看似简单，但合理的配置能显著降低日志表的数据量和序列化开销：

场景建议配置理由
文件上传接口 isSaveRequestData = false 请求体包含二进制流，序列化为字符串无意义且极其庞大。LogAspect 的 isFilterObject() 会过滤 MultipartFile，但关闭更保险
数据导出接口 isSaveRequestData = false 导出时请求参数通常是查询条件（分页、排序），没有审计价值
批量删除接口 isSaveResponseData = false 批量操作返回值通常是 R.ok()，保存无意义
查询类接口 isSaveResponseData = false 查询可能返回大量数据，全量序列化会撑爆 jsonResult 字段（即便有 3800 字符截断也会产生大量无效日志）
开放 API / 高并发接口 两者都关闭 高并发下每个请求都做 JSON 序列化（JsonUtils.toJsonString）是显著的 CPU 开销
性能提醒： LogAspect 中有三级截断保护 —— URL 最大 255 字符（MAX_URL_LENGTH）、ClientKey 最大 32 字符（MAX_CLIENT_KEY_LENGTH）、日志内容最大 3800 字符（MAX_CONTENT_LENGTH）。即使你不关闭保存开关，也不会出现一个超大 JSON 撑爆数据库字段的情况。但序列化本身的 CPU 开销仍然存在，所以高并发场景仍建议按需关闭。

@Log(title = "角色管理", businessType = BusinessType.GRANT)
A. "GRANT" B. 4 C. 5 D. "GRANT".hashCode()

正确答案：B (4)

BusinessType 枚举的声明顺序为 OTHER(0)、INSERT(1)、UPDATE(2)、DELETE(3)、GRANT(4)、EXPORT(5)... 切面代码 log.businessType().ordinal() 取的是 ordinal 值，即 4。

Q2：如果你想在操作日志中排除 idCard 和 bankAccount 两个字段，同时还想排除系统的全局敏感字段，应该怎么写？

A. @Log(title="...", excludeParamNames={"idCard", "bankAccount"}) —— 只写这两个就行 B. 需要把系统的全局敏感字段也列进去 C. 无法同时排除两套字段 D. 需要改系统源码
点击查看答案

正确答案：A

LogAspect 内部做了两层 merge：MapUtil.removeAny(paramsMap, SystemConstants.EXCLUDE_PROPERTIES) 先移除系统全局敏感字段，再 MapUtil.removeAny(paramsMap, excludeParamNames) 移除注解指定的字段。你在 @Log 里只需写自定义的敏感字段，系统的自动带。

Q3：对于数据导出接口，为什么建议同时关闭 isSaveRequestData 和 isSaveResponseData？

A. 导出参数包含查询条件，返回数据量太大 B. 导出请求参数无审计价值，返回值是二进制流无法序列化 C. 两者都是：请求参数无审计价值 + 返回值序列化无意义 + 减少 CPU 开销 D. 导出接口不需要日志
点击查看答案

正确答案：C

导出请求参数（分页、排序、模糊查询条件）对审计价值不大；导出返回值通常是 Excel 二进制流，虽然会被 LogAspect 的 isFilterObject() 过滤掉，但关闭开关更直接；高并发导出时关闭两个开关可以省掉 JSON 序列化的 CPU 开销。但注意：title 和 businessType 仍然会被记录，审计的基本信息不丢失。
