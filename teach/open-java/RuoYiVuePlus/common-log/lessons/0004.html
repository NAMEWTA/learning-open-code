第 4 课 · 实战：自定义日志事件、监听器与扩展 扩展指南

本节目标：学完你能独立完成三项实战：①添加新的 BusinessType 枚举值；②创建自定义事件并在业务代码中发布；③为 OperLogEvent 添加新的消费监听器（如推送到 ELK、钉钉通知）。同时理解事件解耦的核心设计哲学。

一、回顾：OperLogEvent vs LoginInfoEvent 两个现有事件的对比

ruoyi-common-log 模块提供了两个内置事件，它们代表两种不同的日志记录范式：

对比维度OperLogEventLoginInfoEvent
触发方式AOP 切面自动拦截（@Log 注解）业务代码手动构建并发布
发布者LogAspect.handleLog()登录相关 Service（如 SysLoginService）
字段数量20+ 字段（完整操作上下文）5 个字段（精简信息）
是否自动获取用户/IP是（AOP 切面中统一获取）是（监听器中自行解析）
消耗时间记录有（StopWatch 耗时统计）无（登录场景不需要）
请求/响应参数有（可开关）无（登录请求体已包含在 message 中）
设计哲学「声明式」：加注解即可，零侵入「命令式」：在哪里发日志，就手动构建事件
两种日志范式对比： 声明式（OperLogEvent）: Controller (@Log) → AOP拦截 → 自动收集上下文 → publishEvent → 监听器入库 命令式（LoginInfoEvent）: Service.login() → 手动 new LoginInfoEvent() → 手动 set 各字段 → publishEvent → 监听器入库
为什么登录日志不走 AOP？ 登录场景比较特殊：①登录的 Controller 方法本身可能不需要记录（登录成功/失败是 LoginService 的逻辑）；②登录有很多异常分支（密码错误、验证码错误、账号锁定），这些在 Service 层判断最自然；③登录日志需要携带 username 和 status 字符串，这两个信息在 Controller 层面（AOP 切入点）不一定都已经确定。

二、OperLogEvent 的完整字段解析

从源码 OperLogEvent.java 看，这是一个 @Data 注解的 POJO，实现 Serializable（跨服务传输兼容）。共 21 个字段：

分类字段名类型说明谁设置的

costTimeLong消耗时间（毫秒）StopWatch.getDuration().toMillis()

注意一个有趣的细节：operLocation 和 operTime 不是在 LogAspect 中设置的——operLocation 由监听器 SysOperLogServiceImpl.recordOper() 调用 AddressUtils.getRealAddressByIP() 填充，operTime 在入库时设为 LocalDateTime.now()。这体现了「事件只是数据载体，监听器负责补充和持久化」的分工。

为什么 businessTypes 是数组？ businessTypes 是 Integer[] 类型，它不存储到数据库，而是供前端查询使用。比如管理员想同时查看「新增」和「修改」两类操作日志时，前端传 [1,2]，后端在 buildQueryWrapper() 中用 in(SysOperLog::getBusinessType, Arrays.asList(operLog.getBusinessTypes())) 构造 SQL 的 IN 条件。这是一种常见的「事件对象复用为查询条件」的模式。

三、LoginInfoEvent 的设计：为什么比 OperLogEvent 简单？

LoginInfoEvent 只有 5 个字段，结构极其精简：

@Data
public class LoginInfoEvent implements Serializable {
    private String username;           // 用户账号
    private String status;             // 登录状态 "0"成功 "1"失败
    private String message;            // 提示消息（如"登录成功"、"密码错误"）
    private HttpServletRequest request; // 原始请求对象（在监听器中再解析）
    private Object[] args;             // 其他参数（用于日志格式化）
}

它比 OperLogEvent 简单 4 倍，原因是：

  1. 底层信息延迟解析：IP、浏览器、OS、地点等不在事件中设置，而是在监听器 SysLoginInfoServiceImpl.recordLoginInfo() 中通过传入的 request 重新解析。这意味着发布方只需传入 request 对象，不需要关心浏览器解析的具体逻辑。
  2. 状态使用字符串：不同于 OperLogEvent 用 Integer 存状态，LoginInfoEvent 用 String（"0"/"1"），因为登录状态的含义更多样（登录成功、登录失败、退出、注册）。在监听器中再统一映射为 Constants.SUCCESS / Constants.FAIL。
  3. 不记录请求响应内容：登录日志不需要记录请求参数和响应结果（敏感信息），所以省掉了 operParam 和 jsonResult 字段。
设计启发：事件设计不需要一开始就「大而全」。LoginInfoEvent 采用「携带原始材料 + 监听器加工」的模式，发布方只传必要信息，监听器再根据需要做二次解析。这种模式在微服务架构中也很常见——消息体尽量精简，消费者根据需要做富化（enrichment）。

四、实战路径一：如何添加一个新的 BusinessType？

假设你要为「数据导出」功能增加一个细分的「导出为 PDF」业务类型。

1在 BusinessType 枚举中添加新值 ruoyi-common-log/.../enums/BusinessType.java 
public enum BusinessType {
    OTHER,     // 0  其它
    INSERT,    // 1  新增
    UPDATE,    // 2  修改
    DELETE,    // 3  删除
    GRANT,     // 4  授权
    EXPORT,    // 5  导出
    IMPORT,    // 6  导入
    FORCE,     // 7  强退
    GENCODE,   // 8  生成代码
    CLEAN,     // 9  清空数据
    EXPORT_PDF, // 10 导出PDF   ← 新增
}

重要约定：只能追加，不能插入或重排。因为 LogAspect 存的是 ordinal()——如果在中间插入，历史数据的序号就全错了。

2在 Controller 中使用新的枚举值 任意 Controller 
@Log(title = "报表管理", businessType = BusinessType.EXPORT_PDF)
@PostMapping("/exportPdf")
public R<Void> exportPdf(@RequestBody ReportBo bo) {
    reportService.exportToPdf(bo);
    return R.ok();
}

只需要改注解，切面代码不需要任何修改。

3同步前端字典（可选但推荐）

如果前端有业务类型的下拉筛选，需要在字典维护中添加 EXPORT_PDF 对应的字典项（数据字典表 sys_dict_data 中 dict_type = 'sys_oper_type' 的记录）。这样前端查询时 [10] 能正确显示为「导出PDF」。

五、实战路径二：如何创建一个自定义事件（AuditLogEvent）

假设业务需求：需要对「审批通过」操作单独记录，不走通用操作日志表，而是存入独立的 sys_audit_log 表。

1创建事件类 ruoyi-common-log/.../event/AuditLogEvent.java 
package org.dromara.common.log.event;

import lombok.Data;
import java.io.Serial;
import java.io.Serializable;
import java.time.LocalDateTime;

/**
 * 审计日志事件——用于审批流程等敏感操作
 */
@Data
public class AuditLogEvent implements Serializable {
    @Serial
    private static final long serialVersionUID = 1L;

    /** 审批流程名称 */
    private String processName;
    /** 审批节点 */
    private String nodeName;
    /** 审批人 */
    private String auditorName;
    /** 审批结果：PASS / REJECT / RETURN */
    private String result;
    /** 审批意见 */
    private String comment;
    /** 审批时间 */
    private LocalDateTime auditTime;
    /** 关联业务ID */
    private String businessId;
}
2在业务代码中发布事件 审批 Service 中 
@Service
public class ApprovalServiceImpl implements IApprovalService {

    @Override
    public void approve(String businessId, String comment, boolean pass) {
        // 1. 执行业务逻辑：更新审批状态
        approvalMapper.updateStatus(businessId, pass ? "PASS" : "REJECT");

        // 2. 构建并发布审计日志事件
        AuditLogEvent event = new AuditLogEvent();
        event.setProcessName("请假审批");
        event.setNodeName("部门经理审批");
        event.setAuditorName(LoginHelper.getUsername());
        event.setResult(pass ? "PASS" : "REJECT");
        event.setComment(comment);
        event.setAuditTime(LocalDateTime.now());
        event.setBusinessId(businessId);
        SpringUtils.context().publishEvent(event);
    }
}
3创建监听器消费事件 ruoyi-system 或任意模块 
@Component
@Slf4j
public class AuditLogListener {

    @Async                    // ← 异步执行，不阻塞审批业务
    @EventListener
    public void handleAuditLog(AuditLogEvent event) {
        // 存入审计日志表
        SysAuditLog log = new SysAuditLog();
        log.setProcessName(event.getProcessName());
        log.setNodeName(event.getNodeName());
        log.setAuditorName(event.getAuditorName());
        log.setResult(event.getResult());
        log.setComment(event.getComment());
        log.setAuditTime(event.getAuditTime());
        log.setBusinessId(event.getBusinessId());
        auditLogMapper.insert(log);
    }
}

三个步骤完成后，审批操作就会自动写入 sys_audit_log 表，而审批 Service 只需要一行 publishEvent()——完全不依赖审计日志表的存在。

六、实战路径三：为 OperLogEvent 添加新的监听器

这是最常见、最有价值的扩展场景。现有 SysOperLogServiceImpl.recordOper() 只做了「存入 MySQL」一件事。你可以添加新的监听器，让一条操作日志同时写入多个地方：

1添加 ELK 推送监听器 新建任意类，无需修改现有代码 
@Component
@Slf4j
@ConditionalOnProperty(prefix = "log.elk", name = "enabled", havingValue = "true")
public class OperLogElkListener {

    @Async
    @EventListener
    public void sendToElk(OperLogEvent event) {
        // 发送给 Elasticsearch
        // elkClient.index("ruoyi-oper-log", event);
        log.info("操作日志已推送至 ELK: method={}, status={}",
            event.getMethod(), event.getStatus());
    }
}

注意 @ConditionalOnProperty——可以通过 yml 配置 log.elk.enabled=true/false 动态开关，不需要改代码、不需要重启就能通过配置中心切换。

2添加钉钉通知监听器（仅异常时发送） 
@Component
@Slf4j
@ConditionalOnProperty(prefix = "log.dingtalk", name = "enabled", havingValue = "true")
public class OperLogDingTalkListener {

    @Async
    @EventListener
    public void notifyError(OperLogEvent event) {
        // 只通知异常操作
        if (event.getStatus() != null
            && event.getStatus() == BusinessStatus.FAIL.ordinal()) {
            String msg = String.format(
                "【操作异常告警】\n用户: %s\n方法: %s\nURL: %s\n错误: %s",
                event.getOperName(), event.getMethod(),
                event.getOperUrl(), event.getErrorMsg());
            // dingTalkClient.send(msg);
            log.warn("钉钉告警: {}", msg);
        }
    }
}
3添加敏感操作实时预警监听器 
@Component
@Slf4j
public class OperLogSensitiveActionListener {

    private static final Set<String> SENSITIVE_URLS = Set.of(
        "/system/user/delete", "/system/role/delete",
        "/monitor/server/shutdown"
    );

    @Async
    @EventListener
    public void alertSensitiveAction(OperLogEvent event) {
        if (SENSITIVE_URLS.stream().anyMatch(url -> event.getOperUrl().contains(url))) {
            log.warn("【敏感操作预警】用户 {} 执行了 {}，IP: {}",
                event.getOperName(), event.getOperUrl(), event.getOperIp());
            // 可记录到专用安全日志表或发送短信
        }
    }
}

关键点：以上三个监听器（ELK、钉钉、敏感预警）全部独立——互不影响，互不依赖。你可以随时添加或删除任何一个，其他监听器不受影响。

七、事件解耦的核心价值总结

ruoyi-common-log 的事件驱动架构体现了四个核心价值：

价值说明代码体现
发布者不关心谁来听 LogAspect 只负责 publishEvent，完全不知道有哪些监听器。今天只有 MySQL 入库，明天加了 ELK、钉钉，切面代码一行不用改。 SpringUtils.context().publishEvent(operLog);
监听器可独立开关 通过 @ConditionalOnProperty 或 @Profile，每个监听器可以独立控制启用/禁用。开发环境可能不需要 ELK，生产环境可能不需要详细日志打印。 @ConditionalOnProperty(prefix = "log.elk", name = "enabled")
多监听器并行处理 每个监听器都标注 @Async，Spring 在不同线程中执行它们，互不阻塞。MySQL 入库不会拖慢 ELK 推送，ELK 超时不会影响钉钉通知。 @Async + @EventListener
事件即数据契约 OperLogEvent 的 21 个字段就是发布方和消费方的「数据契约」。只要字段不变，双方可以独立演化——监听器内部的实现随便改，发布方无感知。 MapstructUtils.convert(operLogEvent, SysOperLogBo.class)
事件驱动的完整拓扑（当前 + 扩展后）： Controller(@Log) │ ▼ LogAspect.doAround() │ ▼ publishEvent(OperLogEvent) │ ├─→ SysOperLogServiceImpl.recordOper() [MySQL 入库] 已有 ├─→ OperLogElkListener.sendToElk() [ELK 推送] 扩展① ├─→ OperLogDingTalkListener.notifyError() [钉钉告警] 扩展② └─→ OperLogSensitiveActionListener... [敏感操作预警] 扩展③ 所有监听器 @Async 并行执行，互不阻塞

八、常见问题排查清单

问题现象可能原因排查方法
方法加了 @Log 但日志表没记录 ①方法不是 public
②方法在同类内部调用（不走代理）
③被调用的 Bean 不是 Spring 代理		 检查方法修饰符；如果是 this.method() 调用，改为注入自身或提取到另一个 Service
日志表有记录但缺少参数 ①isSaveRequestData = false
②GET 请求参数在 URL 上但 paramsMap 处理路径不同
③参数被 excludeParamNames 排除了		 检查 @Log 注解的 isSaveRequestData 和 excludeParamNames 属性；检查请求方式（GET 参数在 paramsMap，POST JSON 在 argsArrayToString）
响应结果为空 ①isSaveResponseData = false
②Controller 返回 void（jsonResult 为 null）
③方法抛异常了（走异常分支，jsonResult=null）		 检查注解设置和方法返回值类型；异常情况下 jsonResult 就是 null
监听器没被调用 ①@ConditionalOnProperty 条件不满足
②Spring 没有扫描到这个类（包路径不对）
③@Async 线程池没配置好导致执行失败但没日志		 检查 yml 配置中的开关；确认监听器类在 @ComponentScan 范围内；为 @Async 方法添加 try-catch 打印异常
operName 为空 用户未登录（公开接口）或 Sa-Token 的 TokenSession 过期 检查接口是否需要登录；查看 LoginHelper.getLoginUser() 的返回值
日志表数据量爆炸 没有定时清理机制 建议添加定时任务：每月清理 3 个月前的操作日志；或使用 XXL-JOB 定时执行 cleanOperLog()
Spring AOP 代理陷阱：最常见的「日志不生效」原因是同类内部调用。Spring AOP 默认使用 JDK 动态代理或 CGLIB 代理，只有通过代理对象调用的方法才会被拦截。this.xxx() 是直接调用，不走代理，所以 @Log 不会生效。解决方案：①注入自身（@Autowired private XxxService self;）或 ②将方法提取到另一个 Service。

九、检验一下（提取练习）

Q1. 要给系统添加一个新的日志监听器（比如写入 MongoDB），正确做法是？

A. 修改 LogAspect.handleLog()，在 publishEvent 后添加 MongoDB 写入代码 B. 修改 SysOperLogServiceImpl.recordOper()，在其内部添加 MongoDB 写入 C. 新建一个类，加 @EventListener 注解监听 OperLogEvent，不需要修改任何现有代码 D. 在 OperLogEvent 类中添加 MongoDB 字段，让 LogAspect 直接写入
看答案C。这是事件驱动架构的核心优势——新增消费者不需要修改发布方代码。新建一个 @EventListener 类，Spring 会自动注册并分发事件。

Q2. 为什么 OperLogEvent 的 operLocation 和 operTime 不是在 LogAspect 中设置的？

A. LogAspect 中无法获取 IP 地址 B. 这是设计分工：事件是数据载体，监听器负责补充和持久化 C. LogAspect 忘了设置这两个字段 D. 这两个字段不需要设置
看答案B。operLocation 需要调用 AddressUtils.getRealAddressByIP() 做 IP 地址解析（可能涉及远程调用或数据库查询），operTime 是入库时间——这些操作放在监听器中更合理，保持 LogAspect 专注于「收集原始数据」这一单一职责。

Q3. 在 BusinessType 枚举中，为什么只能在末尾追加，不能在中间插入？

A. Java 枚举的限制，中间插入会导致编译错误 B. LogAspect 存的是 ordinal()，如果插入中间，历史数据的序号就对不上了 C. 前端代码不支持中间插入 D. 没有什么限制，可以随意插入
看答案B。LogAspect 使用 log.businessType().ordinal() 把枚举序号存为整型（0、1、2...）。如果在中间插入新值，后续已有枚举的序号会顺移，导致历史数据中存的值与新的语义不匹配。正确做法是只在末尾追加，新值用下一个序号。

Q4. 使用 @ConditionalOnProperty 控制监听器开关有什么好处？

A. 提高代码可读性 B. 可以通过配置文件动态控制，不需要改代码和重新编译 C. 自动提升性能 D. 防止并发问题
看答案B。@ConditionalOnProperty 让监听器的启用/禁用由 yml 配置控制（如 log.elk.enabled=false），开发环境、测试环境、生产环境可以有不同的开关策略，并且可以通过配置中心（如 Nacos）动态调整，不需要重新部署。
最佳实践：生产环境的日志监听器应该注意什么？ ① @Async 自定义线程池：不要使用 Spring 默认的 SimpleAsyncTaskExecutor（来一个任务建一个线程），配置专用线程池并设置队列上限和拒绝策略；② 异常捕获：监听器内部必须 try-catch 所有异常，避免因一个监听器报错导致事件广播中止（Spring 默认的 SimpleApplicationEventMulticaster 串行执行同线程监听器）；③ 消息大小控制：如果推送到 Kafka/RabbitMQ，注意 operParam 和 jsonResult 可能有 3800 字符，加上其他字段，单条消息可能超过 broker 的默认大小限制；④ 幂等性：如果监听器接收的是异步消息（MQ），要设计幂等逻辑，防止重复写入。
参考源码：SysOperLogServiceImpl.java（监听器实现）、SysLoginInfoServiceImpl.java（登录事件监听器实现）、OperLogEvent.java（事件模型）、SysOperLogBo.java（MapStruct 映射——展示了 @AutoMapper 如何将事件映射为数据库实体）。
