第2课 · 事件驱动与异步消费 —— 跨模块协作的艺术

🎯 本节目标：理解 ruoyi-common-log 最核心的设计模式 —— 事件驱动解耦。能够：(1) 说出 OperLogEvent 和 LoginInfoEvent 的字段构成与区别；(2) 画出从 @Log Controller 到 sys_oper_log 表的完整跨模块链路；(3) 解释为什么选择 Spring 事件而非直接写数据库；(4) 区分操作日志和系统日志的边界。

1. 设计核心：发布者与消费者彻底分离

在第1课中，我们追踪了 LogAspect 的完整流程，它的最后一步是：

SpringUtils.context().publishEvent(operLog);

这句话之后发生了什么？LogAspect 不知道，也不关心。它只负责把 OperLogEvent "扔出去"，至于谁来接收、怎么处理、存到哪里，完全由 Spring 的事件机制来决定。这就是事件驱动架构的核心 —— 发布者与消费者之间零耦合。

💡 模块边界感受：common-log 模块的代码中，你不会找到任何 import org.dromara.system.xxx 的引用。它完全不依赖 ruoyi-system 模块。但最终日志数据却准确地写入了 system 模块的 sys_oper_log 表。这种"我不认识你，但你能接到我的消息"的设计，是模块化架构中最高级的解耦方式之一。

整个事件驱动体系包含三个角色：

角色所在模块具体类职责
事件定义common-logOperLogEvent、LoginInfoEvent定义事件的数据结构（POJO）
事件发布common-logLogAspect（操作日志）
SysLoginService / SysRegisterService（登录日志）		在合适的时机构建事件并发布
事件消费ruoyi-systemSysOperLogServiceImpl
SysLoginInfoServiceImpl		通过 @EventListener 接收事件，异步写入数据库

注意事件定义和事件发布都在 common-log 模块，但事件消费在 ruoyi-system 模块。这种"定义归我，消费归你"的模式，是跨模块事件驱动架构的标准实践。

2. OperLogEvent：操作日志的 23 字段载体

event/OperLogEvent.java 是一个纯数据对象（使用 Lombok @Data），实现了 Serializable 接口。它承载了操作日志所需的全部信息，共 23 个字段：

字段名类型来源说明
operIdLong数据库自增日志主键，由数据库自动生成，发布时为空
titleString@Log 注解操作模块名称，如 "用户管理"
businessTypeInteger@Log 注解业务类型（ordinal 值），0-9 对应 10 种类型
businessTypesInteger[]查询参数业务类型数组，用于后台多选筛选，不作为日志写入字段
methodString切面反射方法全限定名，如 com.xxx.controller.UserController.add()
requestMethodStringHttpServletRequestHTTP 请求方式，如 GET / POST / PUT / DELETE
operatorTypeInteger@Log 注解操作人类别（ordinal 值），0=OTHER, 1=MANAGE, 2=MOBILE
operNameStringLoginHelper操作人员登录名
userIdLongLoginHelper操作用户 ID
deptIdLongLoginHelper操作部门 ID
deptNameStringLoginHelper部门名称
clientKeyString请求头 / LoginUser客户端标识（如 PC、APP），优先从请求头获取
deviceTypeStringLoginHelper设备类型
browserStringLoginHelper浏览器类型
osStringLoginHelper操作系统
operUrlStringHttpServletRequest请求 URL，截断到 255 字符
operIpStringServletUtils操作 IP 地址
operLocationString消费者填充操作地点（IP 归属地），由消费者端查询填充
operParamString方法参数请求参数 JSON，截断到 3800 字符
jsonResultString方法返回值响应结果 JSON，截断到 3800 字符
statusInteger异常判定操作状态，0=成功, 1=失败
errorMsgStringException错误消息，截断到 3800 字符
operTimeLocalDateTime消费者填充操作时间，由消费者端在 insert 时设置
costTimeLongStopWatch方法执行耗时（毫秒）

一个关键的观察：operLocation（IP 归属地）和 operTime（操作时间）两个字段在发布事件时是空值。它们由消费者 SysOperLogServiceImpl.recordOper() 在处理事件时填充：

// 消费者端：
operLog.setOperLocation(AddressUtils.getRealAddressByIP(operLog.getOperIp()));
// operTime 在 insertOperlog() 中设置：
operLog.setOperTime(LocalDateTime.now());

这种"发布者只提供原始数据，消费者负责增强"的分工，体现了单一职责原则。发布者（LogAspect）不需要知道 IP 归属地查询服务的存在，消费者（SysOperLogServiceImpl）在写入前做最后的字段补全。

3. LoginInfoEvent：登录日志的精简载体

与 OperLogEvent 的 23 个字段相比，event/LoginInfoEvent.java 极其精简，只有 5 个字段：

字段名类型说明
usernameString登录用户名
statusString登录状态 —— "0" 表示成功，"1" 表示失败（注意是字符串而非整数）
messageString提示消息，如 "登录成功"、"密码错误"
requestHttpServletRequestHTTP 请求对象，用于提取 IP、User-Agent 等信息
argsObject[]其他参数，用于日志格式化输出

注意两个关键差异：

差异 1 —— status 的类型：OperLogEvent 的 status 是 Integer（0/1），而 LoginInfoEvent 的 status 是 String（"0"/"1"）。这是因为登录事件的状态不仅包含 "0" 和 "1"，在消费者端还会与 Constants.LOGIN_SUCCESS、Constants.LOGOUT、Constants.REGISTER 等常量做比对，使用字符串类型更灵活。

差异 2 —— 携带原始 request：LoginInfoEvent 直接将 HttpServletRequest 作为字段携带。这在 OperLogEvent 中是不存在的 —— OperLogEvent 在发布前就把 IP、URL 等从 request 中提取好了。为什么 LoginInfoEvent 不一样？因为登录日志需要从 request 中提取 User-Agent 来解析浏览器和操作系统信息，而这个解析逻辑在消费者端（SysLoginInfoServiceImpl）执行，所以必须把原始 request 传过去。

3.1 LoginInfoEvent 的发布源

OperLogEvent 只有一个发布源（LogAspect），但 LoginInfoEvent 有多个发布源：

发布者所在模块触发场景status 取值
SysLoginServiceruoyi-admin（或 admin 模块）用户登录成功 / 失败"0"（成功）/ "1"（失败）
SysRegisterServiceruoyi-admin（或 admin 模块）用户注册"0"（注册成功视为成功）

发布代码大致如下（以 SysLoginService 为例）：

// 登录成功时
LoginInfoEvent loginInfo = new LoginInfoEvent();
loginInfo.setUsername(username);
loginInfo.setStatus(Constants.LOGIN_SUCCESS);   // 实际上是 "0"
loginInfo.setMessage("登录成功");
loginInfo.setRequest(ServletUtils.getRequest());
SpringUtils.context().publishEvent(loginInfo);

// 登录失败时
loginInfo.setStatus(Constants.LOGIN_FAIL);       // 实际上是 "1"
loginInfo.setMessage("用户不存在/密码错误");

4. 跨模块全链路追踪：从 Controller 到数据库

这是本课最重要的部分 —— 追踪一条操作日志从诞生到落库的完整生命周期。

4.1 操作日志链路（OperLogEvent）

┌──────────────────────────────────────────────────────────────────┐
│ 【阶段 1：声明】                                                  │
│ Controller 方法上标注 @Log(title="用户管理", businessType=INSERT)     │
│ 位置：ruoyi-admin 或 ruoyi-system 模块的 Controller 层              │
├──────────────────────────────────────────────────────────────────┤
│ 【阶段 2：拦截】                                                  │
│ LogAspect.@Around("@annotation(controllerLog)")                  │
│ 位置：ruoyi-common-log / aspect/LogAspect.java                   │
│ doAround() -> joinPoint.proceed() -> handleLog()                  │
├──────────────────────────────────────────────────────────────────┤
│ 【阶段 3：构建事件】                                               │
│ handleLog() 创建 OperLogEvent，填充 20+ 字段：                     │
│   · 从 LoginHelper 获取用户信息（operName, userId, deptId...）    │
│   · 从 ServletUtils 获取 IP、URL                                  │
│   · 从注解获取 title, businessType, operatorType                 │
│   · 从方法参数获取请求/响应数据                                     │
│   · 从 StopWatch 获取耗时                                         │
│   · 异常时设置 status=FAIL, errorMsg=异常消息                    │
├──────────────────────────────────────────────────────────────────┤
│ 【阶段 4：发布事件】                                               │
│ SpringUtils.context().publishEvent(operLog);                     │
│ ↓                                                                │
│ Spring ApplicationEventMulticaster 将事件广播给所有匹配的 Listener │
│ ↓                                                                │
│ 【模块边界：common-log ────────────→ ruoyi-system】               │
│ ↓                                                                │
│ 【阶段 5：异步消费】                                               │
│ SysOperLogServiceImpl.recordOper(OperLogEvent operLogEvent)      │
│ 位置：ruoyi-system/.../service/impl/SysOperLogServiceImpl.java   │
│ 注解：@Async + @EventListener                                     │
│   · MapstructUtils.convert() 将 OperLogEvent 转为 SysOperLogBo   │
│   · AddressUtils.getRealAddressByIP() 查询 IP 归属地             │
│   · insertOperlog() 转为 SysOperLog 实体                          │
│   · 设置 operTime = LocalDateTime.now()                         │
│   · operLogMapper.insert(operLog)                                │
├──────────────────────────────────────────────────────────────────┤
│ 【阶段 6：持久化】                                                 │
│ MyBatis-Plus -> sys_oper_log 表                                  │
│ INSERT INTO sys_oper_log (title, business_type, oper_name, ...)   │
│ VALUES ('用户管理', 1, 'admin', ...)                             │
└──────────────────────────────────────────────────────────────────┘

关键点：步骤 4 到步骤 5 之间，Spring 的事件广播器完成了模块边界的跨越。common-log 模块只知道自己发布了一个 OperLogEvent，完全不知道这个事件将被 ruoyi-system 的哪个方法消费。这种"发布-订阅"模式是模块解耦的经典实践。

4.2 登录日志链路（LoginInfoEvent）

登录日志的链路与操作日志类似，但发布源在 admin 模块：

┌──────────────────────────────────────────────────────────────────┐
│ 【阶段 1：登录/注册】                                             │
│ SysLoginService.login() 或 SysRegisterService.register()          │
│ 位置：ruoyi-admin 或 admin 模块                                    │
├──────────────────────────────────────────────────────────────────┤
│ 【阶段 2：构建并发布 LoginInfoEvent】                              │
│ 填充 username, status("0"/"1"), message, request, args            │
│ SpringUtils.context().publishEvent(loginInfo);                   │
├──────────────────────────────────────────────────────────────────┤
│ 【模块边界：admin ────────────→ ruoyi-system】                    │
│ ↓                                                                │
│ 【阶段 3：异步消费】                                               │
│ SysLoginInfoServiceImpl.recordLoginInfo(LoginInfoEvent event)     │
│ 位置：ruoyi-system/.../service/impl/SysLoginInfoServiceImpl.java │
│ 注解：@Async + @EventListener                                     │
│   · 从 request 提取 IP、User-Agent                                │
│   · UserAgentUtil.parse() 解析浏览器和操作系统                     │
│   · AddressUtils 查询 IP 归属地                                   │
│   · 构建 SysLoginInfoBo，设置所有字段                             │
│   · insertLoginInfo() 写入 sys_login_info 表                     │
├──────────────────────────────────────────────────────────────────┤
│ 【阶段 4：slf4j 日志输出】                                         │
│ 在 recordLoginInfo() 中，还会拼接 IP + 地址 + 用户名 + 状态        │
│ 并通过 log.info(s, args) 输出到 slf4j 日志文件                    │
│ 格式：[192.168.1.1][上海市][admin][0][登录成功]                    │
└──────────────────────────────────────────────────────────────────┘

注意登录日志比操作日志多做了一件事：在写入数据库的同时，也会通过 slf4j 输出到日志文件。这是为了满足运维需求 —— 登录行为需要在系统日志文件中也有记录，方便实时监控和告警。

5. 为什么选择事件驱动？三个层面的理由

如果不用事件驱动，最"直接"的做法是：在 LogAspect.handleLog() 中直接注入 SysOperLogService，调用其 insert 方法写入数据库。但 RuoYi-Vue-Plus 没有这么做，原因有三个层面：

5.1 架构层面：模块独立性

如果直接写数据库：common-log 模块必须依赖 ruoyi-system 模块（或至少依赖其 Service 接口），这意味着 common-log 不再是"通用"模块 —— 它和 system 形成了强耦合。任何想复用 common-log 的项目都必须同时引入 system 模块，或者自己实现 Service 接口。

使用事件驱动：common-log 模块不 import 任何 system 的类。它只发布 Spring 标准事件，任何一个模块都可以通过 @EventListener 来消费。如果你想把日志写到 MongoDB 而不是 MySQL，只需在任意模块中新增一个 Listener，无需改动 common-log 的任何代码。

5.2 性能层面：异步非阻塞

SysOperLogServiceImpl.recordOper() 和 SysLoginInfoServiceImpl.recordLoginInfo() 都标注了 @Async：

@Async
@EventListener
public void recordOper(OperLogEvent operLogEvent) { ... }

@Async 让事件消费在独立的线程池中执行，不阻塞主业务线程。这意味着：即使用户操作日志的写入需要 200ms（包括 IP 归属地查询的网络延迟），Controller 的响应时间完全不受影响。用户不会因为"日志写得太慢"而感觉到页面卡顿。

注意：Spring 事件默认是同步的。如果不加 @Async，publishEvent() 会阻塞直到所有 Listener 执行完毕。加上 @Async 后，Listener 在线程池中异步执行，publishEvent() 立即返回。

5.3 扩展层面：零侵入增加新消费者

假设未来需求变更："当检测到 DELETE 操作时，额外发送一条钉钉通知"。在事件驱动架构下，你只需要在任何模块中新增一个 Listener：

@Async
@EventListener
public void notifyOnDelete(OperLogEvent event) {
    if (BusinessType.DELETE.ordinal() == event.getBusinessType()) {
        dingTalkService.sendAlert("用户 " + event.getOperName()
            + " 删除了 " + event.getTitle());
    }
}

不需要修改 LogAspect，不需要修改 SysOperLogServiceImpl，不需要修改 @Log 注解。新增消费者对现有代码零侵入。这就是发布-订阅模式的核心优势。

6. 操作日志 vs 系统日志：两条平行线

理解 ruoyi-common-log 的设计，还需要区分两种"日志"的不同用途：

维度操作日志（OperLog）系统日志（System Log）
目的业务审计 —— 谁在什么时候做了什么操作开发运维 —— 程序运行状态、异常堆栈、性能指标
受众审计人员、管理员、部门主管开发人员、运维人员
存储数据库（sys_oper_log 表），可查询、筛选、导出文件系统（slf4j 输出到 app.log），可被 ELK 采集
触发用户执行了带 @Log 注解的操作代码中的 log.info() / log.error() 语句
生命周期持久化存储，可定期归档或清空按文件滚动策略归档，超过保留期自动删除
示例"admin 在 2024-01-15 14:30 新增了用户张三""UserController.add 方法耗时 45ms"
"数据库连接池使用率 80%"

在 LogAspect.handleLog 的 catch 块中，当事件发布本身失败时，代码降级为 slf4j 输出：

} catch (Exception exp) {
    // 记录本地异常日志
    log.error("记录操作日志异常", exp);
}

这是"操作日志的失败记录为系统日志"的降级策略 —— 即使数据库写入失败，运维人员仍然能在系统日志文件中看到异常信息。两种日志机制形成互补，确保任何情况下都有迹可循。

7. MapstructUtils 的桥梁角色

在消费者端，有一个容易被忽略的细节：事件对象不能直接插入数据库。OperLogEvent 和 SysOperLog 是两个不同的类，前者是 common-log 模块的事件 POJO，后者是 ruoyi-system 模块的 MyBatis-Plus 实体。中间的转换由 MapstructUtils 完成：

// SysOperLogServiceImpl.recordOper():
SysOperLogBo operLog = MapstructUtils.convert(operLogEvent, SysOperLogBo.class);
// ...补充 operLocation 等字段...
insertOperlog(operLog);

// insertOperlog():
SysOperLog operLog = MapstructUtils.convert(bo, SysOperLog.class);
operLog.setOperTime(LocalDateTime.now());
operLogMapper.insert(operLog);

这里做了两次转换：OperLogEvent -> SysOperLogBo（用于业务处理）-> SysOperLog（用于数据库插入）。MapStruct 在编译期生成转换代码，性能接近手写 getter/setter，但比手写代码更安全（字段名变更时编译期报错）。
