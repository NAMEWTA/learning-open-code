12 · ruoyi-common-satoken：认证与权限核心

核心问题：如何在微服务架构下实现统一的 Token 认证、分布式会话、权限校验和多账号体系？

模块定位：5 个 Java 文件。基于 Sa-Token（国产权限认证框架）深度定制。JWT 简单模式（Token 自包含），Caffeine+Redis 二级缓存，支持多账号体系（sys_user / app_user）。

1. 五大核心组件

组件职责
SaTokenConfig@AutoConfiguration，注册 StpLogic（JWT）、StpInterface（权限源）、SaTokenDao（持久层）
PlusSaTokenDaoCaffeine 本地缓存 + Redis 分布式缓存，二级缓存架构
SaPermissionImpl实现 StpInterface，提供权限列表和角色列表
SaTokenExceptionHandler将 Sa-Token 异常转为统一 R 响应
LoginHelper静态工具方法：getLoginUser()、getUserId()、isSuperAdmin() 等

3. Token 生命周期

# common-satoken.yml
sa-token:
  dynamic-active-timeout: true   # 每次操作自动续期（滑动过期）
  is-read-body: true             # 支持从请求体读取 token
  is-read-header: true           # 支持从 Header 读取 token
  is-read-cookie: false          # 关闭 Cookie 鉴权（杜绝 CSRF）
  token-prefix: "Bearer"         # Token 前缀

JWT 简单模式：Token 本身即包含用户信息（自包含 JWT），无需每次都查询 Redis 做身份校验。Redis 仅存储用户扩展信息（session）和权限数据。

4. PlusSaTokenDao — 二级缓存架构


读取流程:
  Caffeine 本地缓存（命中?） → 返回
                      ↓（未命中）
  Redis 分布式缓存（加载） → 回填 Caffeine → 返回

Caffeine 配置:
  maximumSize: 1000
  expireAfterWrite: 5s     # 写入后 5 秒过期
  initialCapacity: 100

写入流程:
  Redis 写入 → Caffeine.invalidate(key)  # 保证一致性

5. 权限数据双来源

// SaPermissionImpl 解析用户权限：
1. 先尝试从当前登录用户的 TokenSession 获取（零查询）
2. 若不存在，通过 Spring 容器获取 PermissionService Bean（远程服务）
3. 从 loginId 解析 userId，调用 getMenuPermission(userId) 或 getRolePermission(userId)

7. 验证要点

  1. 在浏览器 DevTools → Application → Local Storage 中找到 Admin-Token
  2. 用两个不同角色的账号登录，对比 getInfo 接口返回的 permissions 数组差异
  3. 在前端通过 v-hasPermi 隐藏一个按钮，然后在 Postman 中直接调用该 API，确认后端也做了权限校验（Sa-Token 拦截）
