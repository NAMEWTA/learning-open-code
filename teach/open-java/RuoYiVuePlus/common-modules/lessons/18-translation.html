18 · ruoyi-common-translation：翻译器框架

核心问题：数据库里存的都是 ID（userId=123, deptId=5, status=0），但前端需要展示的是名称（用户名、部门名、状态文本）。如何在 JSON 序列化时自动把 ID 翻译成对应的名称，而无需在 Controller 中手动查询和拼接？

模块定位：11 个 Java 文件。创新的三阶段自动翻译流水线（collect-prepare-process），基于 JsonFieldProcessor SPI（common-json 的响应增强框架）。N+1 查询问题通过批量预处理解决。

1. 核心注解

// @Translation — 标注在 VO 类的字段上
public class SysUserVo {
    private Long userId;

    @Translation(type = "user_id_to_name", mapper = "userId")
    private String userName;  // 自动翻译为用户名

    @Translation(type = "dept_id_to_name", mapper = "deptId")
    private String deptName;  // 自动翻译为部门名

    private Long deptId;
}

3. 三阶段处理流水线（解决 N+1 问题）

假设返回一个用户列表（90 个用户），每个需要翻译 userName 和 deptName

传统做法（N+1 问题）:
  遍历 90 个用户 → 每个用户查一次用户表 → 90 次 SQL

翻译器框架（collect-prepare-process）:
  ① collect: 扫描所有字段，收集需要翻译的 userId 集合 {1,2,3,...,90}
  ② prepare: 批量调用 UserNameTranslationImpl.translationBatch(Set.of(1..90))
            → 一次 SQL: SELECT * FROM sys_user WHERE user_id IN (1..90)
            → 构建 Map: {1→"张三", 2→"李四", ...}
            → 缓存到 JsonEnhancementContext
  ③ process: 对每个字段，从缓存取翻译值

4. 自定义翻译器的扩展点

// 三步实现自定义翻译器

// 1. 实现 TranslationInterface
@Component
@TranslationType(type = "order_status_to_text")
public class OrderStatusTranslationImpl implements TranslationInterface<String> {

    // 2. 单条翻译
    @Override
    public String translation(Object key, String other) {
        return switch((String) key) {
            case "0" -> "待付款";
            case "1" -> "已付款";
            case "2" -> "已发货";
            default -> "未知";
        };
    }

    // 3. 批量翻译（覆盖父类默认的逐条退化行为）
    @Override
    public Map<Object, String> translationBatch(Set<Object> keys, String other) {
        return keys.stream().collect(Collectors.toMap(
            k -> k, k -> translation(k, other)
        ));
    }
}
