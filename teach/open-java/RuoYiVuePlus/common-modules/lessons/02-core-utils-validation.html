第 2 课 · ruoyi-common-core 核心基石：异常体系、工具类与参数校验 基础设施

🎯 本节目标（对应使命 Success #1/#2）：学完你能说出 ruoyi-common-core 的 6 大区域（exception / domain / config / constant / utils / validate）各管什么，理解 R<T> 统一响应的设计、异常体系的继承层级、校验分组的实际用法，并知道 30+ 个工具类中哪些是业务代码最常直接调用的。这是读懂整个 RuoYi 项目的第一块基石——因为其他所有模块都依赖它。

ruoyi-common-core 是整个 RuoYi-Vue-Plus 项目中被依赖次数最多的模块。打开任意一个子模块的 pom.xml，第一行依赖几乎都是它。但它不涉及数据库、不涉及网络、不涉及缓存——它只提供 Java 层面的基础抽象：怎么报错、怎么响应、怎么校验、怎么处理字符串和日期。这节课把它的 62 个 Java 文件拆成 6 块，一次讲透。

一、先记住一句话：它是「所有模块的共同语言」

整个模块做一件事：定义一套所有业务模块共享的编程约定。所有模块返回给前端的数据格式是 R<T>，所有模块抛出的业务异常是 ServiceException，所有模块的字符串处理走 StringUtils，所有模块的参数校验走 ValidatorUtils + 校验分组。看它的 pom.xml，依赖极简——几乎不引入第三方业务库，只靠 Hutool 和 Apache Commons：

hutool-all              ← 国产 Java 工具库（字符串/日期/JSON/HTTP 一把梭）
commons-lang3           ← Apache 字符串/字符集处理
jakarta.validation-api  ← Bean Validation 3.0 注解标准
hibernate-validator     ← Validation 的具体实现（快速失败模式）
mapstruct-plus          ← 编译期对象转换（MapStruct 的增强版）
设计动机 · 为什么要把这些放在一个基础模块里？
如果每个业务模块各自定义 Result、各自封装 StringUtil、各自写异常类——项目稍大就会陷入"同一个项目里三套响应格式"的混乱。ruoyi-common-core 用强约定消除这种混乱：你只能在 R.ok(data) 和 R.fail("原因") 之间选，没有第三条路。这种"少即是多"的约束，是整个项目的秩序来源。

三、逐区域深入（每块抓一个关键设计）

① 异常体系 · 三条继承链，一个兜底

exception/ 目录下 9 个异常类

核心异常体系分三层：

层级类继承自核心能力
基础层BaseExceptionRuntimeException国际化：持有 module（模块名）+ code（错误码）+ args（参数），getMessage() 时自动调 MessageUtils.message(code, args) 查找 i18n 资源文件中的对应消息。查不到则 fallback 到 defaultMessage
业务层ServiceExceptionRuntimeException（独立！）独立继承，不走 BaseException。支持 StrFormatter.format(message, args) 占位符拼接。有 code + detailMessage 字段用于内部调试。支持链式 setMessage() / setDetailMessage()
场景层UserException 等BaseException用户异常、验证码过期、文件超限、SSE 异常——每个预置了默认错误码，直接 new 即可用
⚠️ 关键点：BaseException 和 ServiceException 是两条独立的继承链。ServiceException 不继承 BaseException，它更"轻"——不需要 module/code 体系，直接传字符串就能用，适合"我只是想用中文告诉大家出错了"的场景。而 BaseException 需要 module+code 的国际化体系支撑，适合需要多语言支持的 SaaS 场景。
为什么 ServiceException 的 setter 返回 this？为了链式调用：throw new ServiceException().setMessage("用户不存在").setDetailMessage(userId)。这在异常需要动态补充信息时特别方便，不需要额外构造器重载。这是 Builder 模式在异常类中的轻量应用。

② 统一响应 · R<T> 的三态设计

domain/R.java

整个项目前后端通信的唯一格式：{"code":200, "msg":"操作成功", "data":{...}}。R<T> 提供三种状态：

状态构造方式code典型场景
成功R.ok() / R.ok(data) / R.ok(msg, data)200增删改查正常返回
失败R.fail() / R.fail(msg) / R.fail(code, msg)500 或自定义业务校验不通过、服务异常
警告R.warn(msg) / R.warn(msg, data)601操作部分成功、需要用户确认的中间态

所有工厂方法最终都走一个私有的 restResult(data, code, msg)，确保响应的三个字段（code / msg / data）一定被设置。还提供了两个静态判断方法：R.isSuccess(ret) 和 R.isError(ret)，供调用方（如远程调用、消息队列消费端）判断上游返回是否成功。

为什么不用 Spring 的 ResponseEntity？因为 R<T> 是"业务语义"的响应，而 ResponseEntity 是"HTTP 协议"的响应。在微服务/Feign 远程调用场景下，你需要判断的是"业务是否成功"而非"HTTP 200 还是 500"——R.isSuccess() 屏蔽了 HTTP 层的细节，让你在代码中只关心业务结果。

③ 常量区 · 消灭项目中的魔法数字

constant/ 目录下 6 个常量接口

RuoYi 把所有常量放在接口中（而非类中），字段默认 public static final：

常量接口存放内容使用示例
HttpStatusHTTP 状态码（200/400/401/403/404/500/601）R.fail(HttpStatus.FORBIDDEN, "权限不足")
Constants通用业务常量（YES/NO、成功/失败标记等）if (Constants.SUCCESS.equals(result))
GlobalConstants全局标识（超级管理员角色ID、默认密码、设备类型）if (GlobalConstants.SUPER_ADMIN_ID.equals(userId))
RegexConstants正则表达式（手机号、邮箱、身份证、IP等）pattern.matches(RegexConstants.PHONE_REGEX)
CacheNames缓存名称常量（避免字符串写错）@Cacheable(cacheNames = CacheNames.SYS_CONFIG)
SystemConstants系统级常量（字符集、资源路径等）MediaType.parse(MediaType.APPLICATION_JSON)
为什么用接口而不是类？接口中的字段默认 public static final——省掉了这三个修饰符。但这是一种"老派"做法，现代 Java 更推荐用 final class + private constructor 防止被实现或实例化。RuoYi 团队保留了接口风格是为了与老版 RuoYi 保持一致，降低迁移成本。

④ 工具类矩阵 · 你不需要记住全部 30 个，先记住这 8 个

utils/ 目录下 30+ 个工具类
工具类一句话定位底层依赖
StringUtils字符串判空、切分、格式化、驼峰/下划线互转、url匹配。继承 org.apache.commons.lang3.StringUtils 并用 Hutool 增强commons-lang3 + Hutool
DateUtils日期解析（多格式自动识别）、格式化、加减计算、时间差Hutool DateUtil
SpringUtils获取 Spring Bean、判断 singleton/prototype、检查虚拟线程。继承 SpringUtil 扩展Hutool SpringUtil
ValidatorUtils一行代码触发 Bean Validation：ValidatorUtils.validate(object, AddGroup.class)jakarta.validation
MessageUtils根据 code + args 从 i18n 资源文件获取国际化消息Spring MessageSource
ServletUtils获取当前 Request/Response、写 JSON 响应、取客户端 IPSpring WebUtils
StreamUtils将 Stream/Iterable 转 String（支持分隔符、前缀后缀包裹）Java Stream API
MapstructUtilsBean 拷贝（编译期生成代码，比反射快），支持 List 批量转换MapStruct Plus
细节点 · StringUtils 的继承链：org.dromara.common.core.utils.StringUtils extends org.apache.commons.lang3.StringUtils。这意味着你 import 一个 RuoYi 的 StringUtils，就同时拥有了 Apache Commons 的全部方法（如 isBlank / join / split）+ Hutool 的增强（如 format / toCamelCase）+ RuoYi 自己加的业务方法（如 str2List / splitTo / matches 用 AntPathMatcher）。一个 import 三份能力——这就是"框架级封装"的价值。

⑤ 校验体系 · 分组校验 + 自定义注解

config/ValidatorConfig.java + validate/ 目录

这是模块中最"有设计感"的部分。它把 Bean Validation 从一个简单的"判空/判长度"框架，升级成了分组驱动的业务校验引擎。

5.1 快速失败模式

ValidatorConfig 装配 LocalValidatorFactoryBean，设置 hibernate.validator.fail_fast=true——校验不通过时立即停止，不继续检查后续字段。这避免了用户一次性收到 20 条错误信息不知所措的问题，改为"修一个、再查下一个"的渐进修复体验。同时注入了 MessageSource 使校验消息支持国际化。

5.2 校验分组：同一字段，不同场景不同规则

三个标记接口（AddGroup / EditGroup / QueryGroup）解决了经典难题：

public class SysUser {
    @NotNull(message = "用户ID不能为空", groups = {EditGroup.class})
    private Long userId;   // 新增时不需要ID，编辑时必须

    @NotBlank(message = "用户名不能为空", groups = {AddGroup.class, EditGroup.class})
    private String userName;  // 新增和编辑都需要用户名
}

调用时指定分组：ValidatorUtils.validate(user, AddGroup.class)——只会校验标注了 groups = AddGroup.class 的注解。

5.3 自定义校验注解

注解作用使用示例
@EnumPattern校验值必须在指定枚举的范围内@EnumPattern(type=UserStatus.class, fieldName="code")
@DictPattern校验值在指定字典类型中存在@DictPattern(dictType="sys_user_sex")
@Xss校验字符串不包含 HTML/脚本标签@Xss(message="输入包含非法字符")
@EnumPattern 的精妙之处：它不仅要求值在枚举中存在，还通过 fieldName 指定用什么字段比对（比如 "code" 而非枚举的 name()），这使它能适配"数据库存数字编码、代码用枚举映射"的常见场景。底层 EnumPatternValidator 用反射获取指定字段值再比对，避免了和 @EnumValue 等第三方注解的耦合。
XSS 校验的防护边界：XssValidator 用 Hutool 的 HtmlUtil.RE_HTML_MARK 正则检测 HTML 标签。它不是"过滤所有特殊字符"，而是"拒绝包含 HTML 标签的输入"——这是白名单式防护：默认只放行纯文本。但注意它只校验 @Xss 标注的字段，不会全局过滤，所以 XSS 防护是声明式的、字段级的、有选择的。

⑥ 服务接口 · SPI 契约

service/DictService.java · PermissionService.java

ruoyi-common-core 定义了两个接口但没有实现：DictService（字典查询）和 PermissionService（权限检查）。实现类在 ruoyi-modules-system 中通过 @Service 注册。

为什么要把接口放在 core 里？这是依赖倒置的典型应用。像 common-translation（翻译模块）需要查字典，但它不应该依赖 ruoyi-modules-system（那是业务模块！）。把接口放在被所有人依赖的 core 中，实现放在业务模块中，就解开了"底层依赖高层"的死结。Spring 在运行时自动注入实现——编译期没有循环依赖，运行期功能完整。

四、为什么这张地图值得刻在脑子里？

你每次写 RuoYi 业务代码时，实际上都在走这套流程：

Controller 收到请求 → ValidatorUtils.validate(dto, AddGroup.class) // 校验区 → 业务处理中抛出 ServiceException("库存不足") // 异常区 → 被全局异常处理器捕获，转成 R.fail("库存不足") // 响应区 → 中间用 StringUtils.format / DateUtils.parse 处理数据 // 工具区 → 用 MapstructUtils 做 Bean 转换 // 工具区 → 所有魔法值都引用 constant 中的常量 // 常量区

只要记住 R 响应、ServiceException 异常、ValidatorUtils 校验、StringUtils 字符串 这四个锚点，再加上 AddGroup/EditGroup 分组校验的用法，你就掌握了 RuoYi 项目 80% 的基础设施调用入口。

🤔 想一想：为什么 ValidatorUtils 用 SpringUtils.getBean(Validator.class) 获取校验器，而不直接 new？ 因为 ValidatorConfig 装配的 Validator Bean 经过了定制（快速失败模式 + 国际化消息源）。直接 new 一个 LocalValidatorFactoryBean 会丢失这些配置，导致"快速失败不生效"和"校验消息无法国际化"。通过 Spring 容器获取确保拿到的是被项目配置增强过的实例——这是静态工具类对接 Spring Bean 的标准模式。

Q1. BaseException 和 ServiceException 的核心区别是什么？

A. 没区别，只是两个不同名字的相同功能 B. BaseException 走国际化体系（module+code+MessageUtils），ServiceException 直接传字符串用占位符拼接 C. BaseException 用于 Controller 层，ServiceException 用于 Service 层，使用位置不同 D. ServiceException 继承 BaseException，是父子关系
看答案B。两者是独立的两条继承链，都直接继承 RuntimeException。BaseException.getMessage() 会调用 MessageUtils.message(code, args) 查 i18n 资源文件；ServiceException 用 StrFormatter.format(message, args) 做占位符替换。

Q2. 新增用户时，userId 字段不需要校验，编辑用户时必须校验——怎么实现？

A. 写两个 DTO 类，一个叫 AddUserDTO 一个叫 EditUserDTO B. userId 字段的 @NotNull 注解设置 groups=EditGroup.class，新增时传 AddGroup.class 校验 C. 在 Controller 方法里写 if 手动判断 D. 数据库层面设 NOT NULL 约束让 SQL 自动拦截
看答案B。校验分组正是为此设计：@NotNull(groups = EditGroup.class) Long userId 标注"只在编辑场景需要"，新增时调用 ValidatorUtils.validate(user, AddGroup.class) 会跳过 userId 的校验。

Q3. R.ok(data) 返回的 JSON 中 code 是多少？

A. 0（表示无错误） B. 200（HTTP 成功状态码） C. 1（表示一次成功的操作） D. 根据 data 的内容动态决定
看答案B。ok() 方法内部调用 restResult(data, SUCCESS, "操作成功")，其中 SUCCESS = 200（定义在 HttpStatus 接口中）。warning 状态用 601，fail 默认用 500。

Q4. 为什么 DictService 接口放在 ruoyi-common-core 而不是 ruoyi-modules-system？

A. 放错了，应该放在 system 模块里 B. 因为是跨模块调用的接口，放在被所有人依赖的 core 中实现依赖倒置 C. 因为 core 模块有数据库访问能力，可以直接实现 D. 为了省一个模块，减少项目复杂度
看答案B。这是依赖倒置原则的体现：common-translation 等底层模块需要查字典，但不能依赖业务模块 ruoyi-modules-system。把接口放在 core（所有人都依赖它），实现在 system，Spring 运行时注入——编译期不耦合，运行期功能完整。

Q5. XSS 校验的防护策略是什么？

A. 全局过滤所有请求参数中的特殊字符 B. 声明式字段级防护：只对标注了 @Xss 的字段检测 HTML 标签 C. 在数据库写入前自动转义所有字段 D. 前端负责过滤，后端不管
看答案B。@Xss 是一个 Bean Validation 注解，需要手动标注在需要防护的字段上（如用户昵称、留言内容等），XssValidator 只在 @Validated 触发校验时工作。不是全局的。
