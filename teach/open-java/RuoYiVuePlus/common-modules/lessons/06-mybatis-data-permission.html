06 · ruoyi-common-mybatis（中）：数据权限体系全链路

核心问题：部门经理只能看自己部门的数据，普通员工只能看自己的数据，超级管理员可以看全部——这个「同一张表，不同人看到不同行」的效果如何实现？

三级协作机制：AOP 切面（Spring）→ 注解收集 → MyBatis 拦截器（JSqlParser）→ SpEL 模板解析 → SQL WHERE 注入。最终做到：在 Mapper 上加两个注解，所有查询自动附加数据权限过滤。

1. 核心注解

// 标注在 Mapper 接口的方法或类上
@DataPermission({
    @DataColumn(key = {"deptName"}, value = {"d.dept_id"}),
    @DataColumn(key = {"userName"}, value = {"u.user_id"})
})
public interface SysUserMapper extends BaseMapperPlus<SysUser, SysUserVo> {
    // 此接口的所有 SELECT/UPDATE/DELETE 自动应用数据权限
}

@DataColumn：声明 SpEL 模板中的占位符 key 与数据库列名的映射。
@DataPermission.joinStr：SELECT 默认 OR（任一角色满足即可），UPDATE/DELETE 默认 AND。

2. 数据权限范围枚举（DataScopeType）

类型编码SpEL SQL 模板（核心）
ALL（全部）1空 — 不过滤，返回全部数据
CUSTOM2#{#deptName} IN ( #{@sdss.getRoleCustom(#roleId)} )
DEPT（本部门）3#{#deptName} = #{#user.deptId}
DEPT_AND_CHILD4#{#deptName} IN ( #{@sdss.getDeptAndChild(#user.deptId)} )
SELF（仅本人）5#{#userName} = #{#user.userId}
DEPT_AND_CHILD_OR_SELF6按部门及以下或本人过滤（组合条件）

SpEL 变量：user→当前 LoginUser，sdss→数据权限服务 Bean，deptName/userName→由 @DataColumn 替换，roleId→遍历用户角色时注入。

3. 全链路执行流程

Step 1: AOP 切面拦截（Sping AOP）
    DataPermissionPointcut.matches()
      → 检查方法/类是否有 @DataPermission
    DataPermissionAdvice.invoke()
      → DataPermissionHelper.setPermission(annotation)  // 写入 ThreadLocal
      → invocation.proceed()
      → finally: DataPermissionHelper.removePermission() // 清理

Step 2: MyBatis 拦截器
    PlusDataPermissionInterceptor
      .beforeQuery() 或 .beforePrepare()
      检查: InterceptorIgnoreHelper.willIgnoreDataPermission()?
      检查: dataPermissionHandler.invalid()? (ThreadLocal 有注解吗?)

Step 3: PlusDataPermissionHandler.getSqlSegment（核心）
    1. 获取当前登录用户 → 超级管理员直接返回（无过滤）
    2. 筛选参与数据权限的角色 (通过 DataPermissionAccess)
    3. 遍历角色，使用 SpEL 解析各角色的 dataScope 模板
       - NullSafe: null 变量返回默认值 "-1"
       - 使用 DataPermissionHelper.ignore() 防止模板内 SQL 死循环
    4. SELECT: OR 拼接各角色条件
       UPDATE/DELETE: AND 拼接

Step 4: JSqlParser 修改 SQL AST
    SELECT ... FROM sys_user u WHERE (原条件) AND (数据权限条件)

4. 权限上下文管理

// DataPermissionHelper — 三级存储
ThreadLocal<DataPermission>    → 当前 Mapper 的权限注解
SaStorage（Sa-Token 请求域）    → 通用上下文变量 {user, roleId...}
DataPermissionIgnoreContext     → 忽略状态栈（嵌套支持）

// 临时忽略数据权限（防止 SpEL 模板内 SQL 死循环）
DataPermissionHelper.ignore(() -> {
    // 这里的 SQL 不会被数据权限过滤
    return someMapper.selectById(id);
});

5. DataBaseHelper — 跨数据库方言

// 动态获取当前数据源类型
DataBaseType type = DataBaseHelper.getDataBaseType();
// 支持: MY_SQL, ORACLE, POSTGRE_SQL, SQL_SERVER

// 跨数据库 FIND_IN_SET
String sql = DataBaseHelper.findInSet(value, columnName);
// MySQL: FIND_IN_SET(value, columnName)
// Oracle: INSTR(',' || columnName || ',', ',' || value || ',') > 0
// PostgreSQL: POSITION(',' || value || ',' IN ',' || columnName || ',') > 0

6. 前端对应

前端 store/modules/user.ts 中的 getInfo() 获取用户 roles/permissions，然后 v-hasPermi 指令控制按钮可见性。但数据权限是后端层面的行级过滤——前端即使绕过按钮，后端 SQL 也会自动过滤，确保了安全。

7. 验证要点

  1. 在 SysUserMapper 上找到 @DataPermission 注解，查看它的 @DataColumn 映射
  2. 用两个不同角色（部门经理 vs 普通员工）登录，查看同一用户列表页面，对比返回数据量
  3. 在 PlusDataPermissionHandler.getSqlSegment 方法打断点，观察 SpEL 模板解析过程
