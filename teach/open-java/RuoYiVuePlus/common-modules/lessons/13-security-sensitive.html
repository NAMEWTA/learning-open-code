13 · ruoyi-common-security + sensitive：安全拦截与数据脱敏

核心问题：如何在请求层面控制接口访问权限（IP白名单、客户端校验），同时在响应层面自动脱敏敏感数据（手机号、身份证、银行卡）？

Part A: ruoyi-common-security — 请求级安全拦截

关键发现：这个模块虽然叫"security"，但并不依赖 Spring Security。它是基于 Sa-Token 的拦截器配置层，负责在请求进入 Controller 之前的三层校验。

三层拦截校验

// SecurityConfig 注册的 SaInterceptor
SaRouter.match(allUrlHandler.getUrls()).check(() -> {
    StpUtil.checkLogin();  // 第一层：登录态检查

    // 第二层：clientId 一致性校验
    // Header 或参数中的 clientId 必须与 Token 中存储的一致
    // 防止 Token 在不同客户端之间被盗用

    // 第三层：客户端访问规则
    // 路径白名单校验 → 按客户端配置限制可访问的接口
    // IP白名单校验   → 按客户端配置限制来源 IP
});

Part B: ruoyi-common-sensitive — JSON 序列化脱敏

@Sensitive 注解

@Sensitive(strategy = SensitiveStrategy.PHONE)
private String phone;  // 138****1234

@Sensitive(strategy = SensitiveStrategy.ID_CARD)
private String idCard; // 前3后4可见

@Sensitive(strategy = SensitiveStrategy.EMAIL, roleKey = {"admin"})
private String email;  // admin 角色不脱敏，其他人看到 t***@example.com

角色级脱敏豁免

// SensitiveService 接口
public interface SensitiveService {
    boolean isSensitive(String[] roleKey, String[] perms);
}
// 默认：管理员不过滤（不脱敏），其他人脱敏
// 业务方可自定义实现

脱敏执行流程

Controller 返回 UserVO
    ↓
ResponseEnhancementAdvice.beforeBodyWrite()
    ↓
JsonValueEnhancer.enhance(body)
    ↓
SensitiveJsonFieldProcessor.process(fieldContext, value)
    ├ 检查字段是否有 @Sensitive 注解
    ├ 调用 SensitiveService.isSensitive() 判断是否需要脱敏
    └ 执行 strategy.desensitizer.apply(text)
    ↓
Jackson 序列化为 JSON → 返回前端
