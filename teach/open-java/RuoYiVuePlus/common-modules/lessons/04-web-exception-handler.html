04 · ruoyi-common-web：全局异常处理与响应增强

核心问题：一个 Controller 方法可能抛出 20+ 种不同类型的异常，如何统一处理并返回一致的前端响应格式？

1. GlobalExceptionHandler — 16 个异常处理分支

技术栈：@RestControllerAdvice + @ExceptionHandler。共 19 个 @ExceptionHandler 方法。Spring 的异常匹配遵循「最具体优先」原则。

业务异常分支

异常类型处理逻辑
ServiceException返回 R.fail(code, msg)。如果 e.getCode() != null 则使用指定 code
BaseException国际化异常。调用 getMessage() 走 i18n 翻译，返回 R.fail(msg)
SseException@ResponseStatus(401)。直接返回 JSON 字符串（非 R 包装），适配 SSE 协议

参数校验异常分支（4 种）

异常类型触发场景
BindException@Valid 在 Controller 参数上绑定失败
ConstraintViolationException@Validated 在 Service 层或方法参数上校验失败
MethodArgumentNotValidException@Valid 在 @RequestBody 上校验失败
HandlerMethodValidationExceptionSpring Boot 3.x 新增的 @Validated 方法级校验

这 4 种处理器的共同模式（实际使用项目统一的 StreamUtils.join()）：

// 提取校验错误信息，用 ", " 拼接
// BindException:
String message = StreamUtils.join(e.getAllErrors(),
    DefaultMessageSourceResolvable::getDefaultMessage, ", ");
// ConstraintViolationException:
String message = StreamUtils.join(e.getConstraintViolations(),
    ConstraintViolation::getMessage, ", ");
// MethodArgumentNotValidException:
String message = StreamUtils.join(e.getBindingResult().getAllErrors(),
    DefaultMessageSourceResolvable::getDefaultMessage, ", ");
// HandlerMethodValidationException:
String message = StreamUtils.join(e.getAllErrors(),
    MessageSourceResolvable::getDefaultMessage, ", ");
return R.fail(message);

兜底异常分支

异常类型处理方式
RuntimeException生成 8 位随机 errorId，记录完整堆栈，返回含 errorId 的提示
Exception兜底，同样生成 errorId
ServletException记录堆栈，返回"发生未知异常"
errorId 机制：对于 RuntimeException 和 Exception，返回 "系统异常，错误代码：" + errorId，用户可凭此 ID 反馈给管理员。日志中用 ERROR 级别记录完整堆栈和 errorId。

SSE 特殊处理

AsyncRequestTimeoutException → 完全静默（空方法体）。IOException 且请求路径包含 SSE 路径 → 静默吞掉。这是为了防止 SSE 长连接频繁断开产生噪音日志。

其他分支

异常场景
HttpRequestMethodNotSupportedException请求方法不对（POST 访问 GET 接口）
MissingPathVariableException缺少路径变量
MethodArgumentTypeMismatchException参数类型不匹配（如期望 Long 传入 String）
NoHandlerFoundException404
JsonParseExceptionJSON 格式错误
HttpMessageNotReadableException请求体反序列化失败
ExpressionExceptionSpEL 表达式异常

2. ResponseEnhancementAdvice — 响应体后处理

实现 ResponseBodyAdvice<Object>，在 Jackson 序列化之前对响应对象做增强：

// beforeBodyWrite 核心逻辑
Object enhancedBody = JsonValueEnhancer.enhance(body);
// JsonValueEnhancer 来自 ruoyi-common-json 模块
// 递归遍历响应对象树，执行 JsonFieldProcessor 链：
//   - TranslationJsonFieldProcessor → 字段翻译（字典/用户/部门/OSS）
//   - SensitiveJsonFieldProcessor → 字段脱敏（手机号/身份证等）

对应前端：翻译后的字段直接展示、脱敏后的字段自动隐藏部分内容。

3. 前后端协作全景图

后端抛出异常
    │
    ▼
GlobalExceptionHandler 捕获 → R.fail(code, msg)
    │
    ▼
Jackson 序列化 → {"code":500,"msg":"用户不存在","data":null}
    │
    ▼
前端 request.ts 响应拦截器
    ├── code=200 → Promise.resolve(res.data)
    ├── code=401 → 弹出重新登录确认框
    ├── code=500 → ElMessage.error(msg) + reject
    ├── code=601 → ElMessage.warning(msg) + reject
    └── 其他 → ElNotification.error + reject

4. BaseController — 便捷方法

public class BaseController {
    protected R<Void> toAjax(int rows) {
        return rows > 0 ? R.ok() : R.fail();
    }
    protected R<Void> toAjax(boolean result) {
        return result ? R.ok() : R.fail();
    }
    protected String redirect(String url) {
        return StringUtils.format("redirect:{}", url);
    }
}

注意：目前业务 Controller 大多数直接使用 R.ok() / R.fail() 而非继承 BaseController。
