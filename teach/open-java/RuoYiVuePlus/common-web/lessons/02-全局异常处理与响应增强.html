02 | 全局异常处理与响应增强：GlobalExceptionHandler 16 种异常 + ResponseEnhancementAdvice

课程目标：掌握 GlobalExceptionHandler 的 16 种异常覆盖策略及内外有别（内部全栈日志 + 外部友好提示）的核心纪律，理解 ResponseEnhancementAdvice 的响应增强机制，以及 WaveAndCircleCaptcha 验证码生成、I18nLocaleResolver 国际化等 Web 层配套能力。

1. GlobalExceptionHandler 全景

1.1 角色定位

GlobalExceptionHandler 使用 @RestControllerAdvice 注解，拦截所有 Controller 抛出的异常，是系统的最后一道防线。它的核心纪律是：

内部：log.error() 记录完整堆栈，方便开发人员排查。
外部：R.fail() 返回友好消息，不暴露内部细节给客户端。

这保证了两个目标同时达成：日志有足够信息溯源问题，响应对用户友好不泄露技术细节。

2. 关键异常处理策略详解

2.1 RuntimeException/Exception 的兜底策略 —— 8 位错误码

这是最重要的兜底处理器。对于未预料的 RuntimeException 和 Exception，系统生成一个8 位唯一错误码：

@ExceptionHandler(RuntimeException.class)
public R<Void> handleRuntimeException(RuntimeException e, HttpServletRequest request) {
    // 生成 8 位错误码（基于异常内容 hash 或随机生成）
    String errorCode = generateErrorCode();
    log.error("请求地址'{}', 发生运行时异常. 错误码: {}", request.getRequestURI(), errorCode, e);
    return R.fail("系统异常，请联系管理员. 错误码: " + errorCode);
}

@ExceptionHandler(Exception.class)
public R<Void> handleException(Exception e, HttpServletRequest request) {
    String errorCode = generateErrorCode();
    log.error("请求地址'{}', 发生系统异常. 错误码: {}", request.getRequestURI(), errorCode, e);
    return R.fail("系统异常，请联系管理员. 错误码: " + errorCode);
}

工作流程：

  1. 用户调用接口发生未知异常。
  2. GlobalExceptionHandler 捕获，生成 8 位错误码（例如 A3F7B9C2）。
  3. 日志中输出：完整异常堆栈 + 错误码 + 请求地址。
  4. 用户收到："系统异常，请联系管理员. 错误码: A3F7B9C2"。
  5. 用户反馈错误码 → 运维/开发通过 grep A3F7B9C2 app.log 立即定位异常。
为什么不用异常消息直接返回给用户？异常消息（如 "NullPointerException: Cannot invoke method on null"）会暴露代码细节，存在安全隐患。错误码机制让用户只拿到一个无意义的编码，而内部人员可以精确追溯。这是日志安全的最佳实践。

2.2 AsyncRequestTimeoutException —— SSE 超时的静默处理

当 SSE（Server-Sent Events）连接超时时，Spring MVC 的异步机制会抛出 AsyncRequestTimeoutException。这不是真正的错误——SSE 连接本身就有生命周期，超时是正常行为。

@ExceptionHandler(AsyncRequestTimeoutException.class)
public void handleAsyncRequestTimeoutException(AsyncRequestTimeoutException e) {
    // 空方法体 —— 不做任何处理，不返回任何响应
}

关键点：

  • 返回类型是 void，不是 R<Void>——Spring 不会写入响应体。
  • 方法体为空——不记录日志，不返回错误信息。SSE 超时是预期行为，不是故障。
  • 如果对此异常返回错误信息，前端 SSE 客户端会在每次超时后看到一个错误 JSON，破坏 SSE 的事件流。

2.3 IOException 对 SSE 端点的特殊处理

当 SSE 客户端断开连接（关闭浏览器、刷新页面）时，服务端写入响应会抛出 IOException。对普通接口这是需要处理的异常，但对 SSE 端点这是正常的客户端行为。

@ExceptionHandler(IOException.class)
public void handleIOException(IOException e, HttpServletRequest request) {
    String uri = request.getRequestURI();
    // 仅对 SSE 端点忽略
    if (StringUtils.containsIgnoreCase(uri, "sse")) {
        return;  // 静默忽略
    }
    // 非 SSE 端点的 IOException 仍然记录并返回
    log.error("请求地址'{}', 发生 IO 异常.", uri, e);
    // 返回错误信息
}

这个设计避免了 SSE 服务日志被大量"Broken pipe"错误淹没。

2.4 ServiceException —— 业务异常的错误码透传

ServiceException 是若依框架内业务代码抛出的异常，可能携带自定义错误码和消息：

@ExceptionHandler(ServiceException.class)
public R<Void> handleServiceException(ServiceException e) {
    // 提取错误码，无自定义则使用默认值
    Integer code = e.getCode();
    String message = e.getMessage();
    return R.fail(code, message);
}

与 RuntimeException 不同，ServiceException 的 message 是业务人员编写的、面向用户的提示（如"用户名已存在"），直接返回给客户端是安全的。

3. 响应增强：ResponseEnhancementAdvice

3.1 实现机制

ResponseEnhancementAdvice 实现了 Spring MVC 的 ResponseBodyAdvice<Object> 接口，在 Controller 返回数据写入响应体之前进行拦截和增强：

@ControllerAdvice
public class ResponseEnhancementAdvice implements ResponseBodyAdvice<Object> {

    @Override
    public boolean supports(MethodParameter returnType, Class converterType) {
        // 委托给 JsonValueEnhancer 判断是否需要增强
        return jsonValueEnhancer.supports(converterType);
    }

    @Override
    public Object beforeBodyWrite(Object body, MethodParameter returnType,
                                   MediaType selectedContentType,
                                   Class selectedConverterType,
                                   ServerHttpRequest request,
                                   ServerHttpResponse response) {
        // 检查 Content-Type 是否兼容 JSON
        if (!isJsonCompatible(selectedContentType)) {
            return body;  // 非 JSON 响应不增强
        }

        // 委托 ruoyi-common-json 的 JsonValueEnhancer 执行实际增强
        return jsonValueEnhancer.enhance(body);
    }
}

3.2 双层检查机制

检查点位置目的
supports()判断消息转换器类型确保是 Jackson 等 JSON 转换器才走增强逻辑
beforeBodyWrite()判断 Content-Type确保响应确实是 JSON 格式，避免对 XML/文件下载等误增强

3.3 实际增强逻辑在哪？

注意：ResponseEnhancementAdvice 本身不包含增强逻辑。它只是一个适配器，将 Spring MVC 的 ResponseBodyAdvice 接口适配到 ruoyi-common-json 模块的 JsonValueEnhancer。实际的增强功能（日期格式化、Long 精度处理、敏感字段脱敏等）全部在 JSON 模块中实现。

模块解耦设计：Web 层只负责"拦截时机"（when to intercept），JSON 层负责"如何增强"（how to enhance）。这种分离使得增强逻辑可以独立演进，Web 层不依赖任何具体的 JSON 序列化实现。

4. 验证码生成：WaveAndCircleCaptcha

4.1 类结构

WaveAndCircleCaptcha 继承自 Hutool 的 AbstractCaptcha，通过三步绘制生成抗 OCR 识别的验证码图片：

public class WaveAndCircleCaptcha extends AbstractCaptcha {

    // 多个构造器重载
    public WaveAndCircleCaptcha(int width, int height, int codeCount, int interfereCount) { ... }
    public WaveAndCircleCaptcha(int width, int height, int codeCount, int interfereCount, float fontScale) { ... }

    @Override
    protected Image createImage(String code) {
        // ① drawString —— 绘制验证码文字（带颜色 + 抗锯齿）
        // ② shear —— X/Y 双向正弦波扭曲（抗 OCR 分割）
        // ③ drawInterfere —— 绘制干扰元素（圆点 + 正弦波曲线）
    }
}

4.2 三步绘制详解

步骤方法效果对抗目的
1drawString()设置随机颜色 + Graphics2D 抗锯齿，绘制验证码文字让文字有颜色变化，简单的灰度二值化无法分离文字
2shear()对图像进行 X 轴和 Y 轴的双向正弦波扭曲变形破坏文字的连续笔画特征，OCR 无法通过字符分割识别
3drawInterfere()绘制随机圆点 + 平滑正弦波干扰线增加背景噪声，干扰 OCR 的像素级分析

4.3 构造器重载设计

// 方式一：指定宽高和字符数
new WaveAndCircleCaptcha(200, 100, 5, 10);
// → 200x100 图片，5 个验证码字符，10 个干扰元素

// 方式二：额外指定字体缩放
new WaveAndCircleCaptcha(200, 100, 5, 10, 0.8f);
// → 字体缩小至 80%，增加识别难度

// 方式三：使用默认值
new WaveAndCircleCaptcha(200, 100, 4, 5);
// → 标准验证码尺寸
为什么扭曲（shear）是核心？现代 OCR 通过先分割字符再逐个识别来工作。正弦波扭曲让相邻字符粘连在一起，分割算法会失败。即使是最先进的深度学习 OCR，在字符粘连的场景下准确率也会大幅下降。而人类视觉系统对形状的感知不依赖字符的独立性，所以扭曲后仍然可读。

5. 国际化：I18nLocaleResolver

5.1 前端驱动的 Locale 策略

I18nLocaleResolver 实现了 Spring 的 LocaleResolver 接口，采用前端驱动的区域设置方案：

public class I18nLocaleResolver implements LocaleResolver {

    @Override
    public Locale resolveLocale(HttpServletRequest request) {
        // 从 HTTP 请求头 content-language 读取语言标签
        String language = request.getHeader("content-language");

        if (StringUtils.isBlank(language)) {
            // 未设置时回退到系统默认区域
            return Locale.getDefault();
        }

        // BCP47 格式解析，如 "zh-CN" → Locale.SIMPLIFIED_CHINESE
        return Locale.forLanguageTag(language);
    }

    @Override
    public void setLocale(HttpServletRequest request, HttpServletResponse response, Locale locale) {
        // 空方法 —— 不支持服务端切换语言
        // 语言由前端通过 content-language 请求头控制
    }
}

5.2 关键设计决策

设计点选择原因
区域信息来源content-language 请求头前端每次请求携带，无需服务端保存状态
解析方式Locale.forLanguageTag()支持 BCP47 标准格式（zh-CN, en-US, ja-JP）
setLocale() 实现空方法不维护服务端语言状态，由前端驱动
回退策略Locale.getDefault()无请求头时使用 JVM 默认区域
为什么不支持 setLocale？传统的服务端国际化方案会在 Session 中存储用户的语言偏好（如 Spring 的 SessionLocaleResolver）。但若依是前后端分离架构，前端通过请求头控制语言，后端无状态，更符合 RESTful 设计原则。每次请求独立决定语言，避免了 Session 状态管理。

5.3 I18nConfig 注册顺序

为了让自定义的 I18nLocaleResolver 优先生效，必须确保在 Spring MVC 的默认 AcceptHeaderLocaleResolver 之前注册：

@AutoConfiguration(before = WebMvcAutoConfiguration.class)
public class I18nConfig implements WebMvcConfigurer {

    @Bean
    public LocaleResolver localeResolver() {
        return new I18nLocaleResolver();
    }

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        // 注册 LocaleChangeInterceptor（如果需要）
    }
}

@AutoConfiguration(before = WebMvcAutoConfiguration.class) 是关键——它确保自定义的 LocaleResolver Bean 在 Spring Boot 的 WebMvcAutoConfiguration 注册默认 LocaleResolver 之前完成注册，从而覆盖默认行为。
