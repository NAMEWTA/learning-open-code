01 | Filter 链与请求包装：RepeatableFilter → XssFilter 的顺序设计

课程目标：掌握 ruoyi-common-web 模块中 Filter 链的设计理念，理解 RepeatableFilter 先行缓存请求体、XssFilter 后行清洗的安全架构，以及 PlusWebInvokeTimeInterceptor 如何利用可重复读取的请求体实现完整的请求耗时统计。

1. 全景鸟瞰：ruoyi-common-web 的 5 大能力域

ruoyi-common-web 是若依框架的 Web 层基础设施模块，仅 17 个 Java 类，却支撑了整个系统的 HTTP 请求处理管道。它提供 5 大能力域：

请求统计PlusWebInvokeTimeInterceptor接口耗时监控与日志
异常处理GlobalExceptionHandler16 种异常的全局捕获与友好响应
响应增强ResponseEnhancementAdvice统一响应格式、日期格式化、脱敏

核心设计思想：整个管道遵循"缓存 → 清洗 → 统计 → 处理 → 增强"的线性流程。其中 RepeatableFilter 必须排在最前面——这是整个架构正确性的前提：必须在任何消费请求体的逻辑之前，先将请求体缓存起来。

2. Filter 链的顺序设计：为什么 RepeatableFilter 必须在 XssFilter 之前

2.1 顺序的决定机制

在 Filter 链中，顺序由注册方式决定：

  • RepeatableFilter：通过 FilterConfig 配置类中 FilterRegistrationBean 注册，未显式设置 order，默认按注册顺序。
  • XssFilter：通过 @FilterRegistration(order = HIGHEST_PRECEDENCE + 1) 注解注册，order 值为 Integer.MIN_VALUE + 1，在 Filter 链中优先级极高。

虽然 XssFilter 的 order 值更小（数值越小越先执行），但 RepeatableFilter 在 FilterConfig 中先于 XssFilter 注册到 FilterRegistrationBean，在 Spring Boot 的 Filter 注册机制中，同类型的 Filter 按注册顺序执行。实际运行中，RepeatableFilter 确实在 XssFilter 之前执行。这是靠配置类中的注册顺序保证的，而非依赖 order 数值。

2.2 顺序不可颠倒的根本原因

考虑一个 JSON 请求体：{"name": "<script>alert('xss')</script>", "age": 25}

正确的顺序（RepeatableFilter 在前）：

Step 1: RepeatableFilter 读原始 body → 缓存 byte[]{123, 34, 110, ...} (原始数据)
Step 2: XssFilter 读缓存 → 清洗 → "{\"name\":\"alert('xss')\",\"age\":25}"
Step 3: Interceptor 可再次读缓存 → 用于日志记录（拿到原始数据或清洗后数据均可）

如果顺序颠倒（XssFilter 在前）：

Step 1: XssFilter 读原始 body → 清洗 → 原始数据被消费，无法找回
Step 2: RepeatableFilter 尝试读 → 但流已被消费 → 得到空 body 或不完整数据
Step 3: Interceptor 读取失败 → 日志中 body 为空
       Controller 也可能读取失败 → 业务异常！
设计原则：缓存必须在清洗之前。"先拍照，再修图"——先保存原始副本，再对副本进行加工。一旦清洗操作执行了，原始内容就不可逆地丢失了。对于日志审计和问题排查来说，有时需要看到原始请求内容（即使包含 XSS 攻击代码），这有助于安全分析和攻击溯源。

3. RepeatableFilter 与 RepeatedlyRequestWrapper：请求体的"时光机"

3.1 RepeatableFilter 的过滤逻辑

RepeatableFilter 的 doFilter 方法极其简洁：

public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
    ServletRequest requestWrapper = null;
    if (request instanceof HttpServletRequest httpServletRequest) {
        String contentType = httpServletRequest.getContentType();
        // 仅对 JSON 请求 (application/json) 进行包装
        if (StringUtils.startsWithIgnoreCase(contentType, "application/json")) {
            requestWrapper = new RepeatedlyRequestWrapper(httpServletRequest, response);
        }
    }
    if (requestWrapper == null) {
        requestWrapper = request;
    }
    chain.doFilter(requestWrapper, response);
}

关键设计点：

  • 精准匹配：仅对 Content-Type 为 application/json 的请求进行包装。multipart/form-data 文件上传、application/x-www-form-urlencoded 表单提交不受影响。
  • 静默降级：非 JSON 请求直接放行原 request，零开销。
  • 无侵入：对下游 Filter 和 Controller 完全透明——它们拿到的都是 HttpServletRequest 实例，不知道底层已被包装。

3.2 RepeatedlyRequestWrapper 的缓存机制

这是整个可重复读取能力的核心：

public class RepeatedlyRequestWrapper extends HttpServletRequestWrapper {
    private final byte[] body;              // 请求体缓存

    public RepeatedlyRequestWrapper(HttpServletRequest request, ServletResponse response) {
        super(request);
        // 设置请求和响应字符集为 UTF-8
        request.setCharacterEncoding("UTF-8");
        response.setCharacterEncoding("UTF-8");

        // 一次性读取请求体到 byte[] 缓存
        body = IoUtil.readBytes(request.getInputStream(), false);
    }

    @Override
    public ServletInputStream getInputStream() {
        // 每次调用都从缓存的 byte[] 创建新的 ByteArrayInputStream
        final ByteArrayInputStream bais = new ByteArrayInputStream(body);
        return new ServletInputStream() {
            @Override
            public int read() { return bais.read(); }
            @Override
            public boolean isFinished() { return bais.available() == 0; }
            @Override
            public boolean isReady() { return true; }
            @Override
            public void setReadListener(ReadListener listener) {}
        };
    }
}

工作原理：

  1. 构造时读取：在构造函数中调用 IoUtil.readBytes(request.getInputStream(), false)，一次性将原始请求体的所有字节读入 byte[] body。参数 false 表示不关闭流。
  2. 按需创建：每次调用 getInputStream() 时，从 body 新建一个 ByteArrayInputStream，封装为 ServletInputStream 返回。每次都是全新的流，从头开始读。
  3. UTF-8 兜底：同时设置 request 和 response 的字符编码为 UTF-8，避免中文乱码。
为什么 PlusWebInvokeTimeInterceptor 需要这个能力？Interceptor 的 preHandle 方法中需要记录请求体内容用于日志。如果没有 RepeatedlyRequestWrapper，Controller 读取完 body 后流就关闭了，Interceptor 无法再次读取。有了缓存，Interceptor 和 Controller 各读各的流，互不干扰。

4. XssFilter 三层防护体系

XSS 防护采用了分层的纵深防御架构，每一层各司其职：

4.1 第一层：Filter 层 —— 路由决策

XssFilter 作为 XSS 防护的入口，负责"谁需要被清洗"的决策：

@ConditionalOnProperty(value = "xss.enabled", havingValue = "true")
public class XssFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        HttpServletRequest req = (HttpServletRequest) request;

        // ① GET/DELETE 请求不包含请求体，直接跳过
        if (HttpMethod.GET.name().equals(method) || HttpMethod.DELETE.name().equals(method)) {
            chain.doFilter(request, response);
            return;
        }

        // ② 在排除列表中的 URL 跳过
        String uri = req.getRequestURI();
        if (StringUtils.containsAnyIgnoreCase(uri, xssProperties.getExcludeUrls())) {
            chain.doFilter(request, response);
            return;
        }

        // ③ 其余请求进行 XSS 清洗
        chain.doFilter(new XssHttpServletRequestWrapper(req), response);
    }
}

三个决策点：

  • GET/DELETE 跳过：这两个 HTTP 方法没有请求体，参数在 URL 中（由 Web 容器自行处理），不需要 body 级别的清洗。
  • excludeUrls 跳过：某些接口可能故意接收 HTML 内容（如富文本编辑器），通过配置排除。
  • 开关控制：@ConditionalOnProperty(value = "xss.enabled") 实现零成本禁用——配置 xss.enabled=false 即可关闭整个 Filter，类不会被加载。

4.2 第二层：Wrapper 层 —— 实际清洗

XssHttpServletRequestWrapper 是真正执行清洗的地方，它覆盖了所有获取参数和请求体的方法：

public class XssHttpServletRequestWrapper extends HttpServletRequestWrapper {

    @Override
    public String getParameter(String name) {
        return HtmlUtil.cleanHtmlTag(super.getParameter(name)).trim();
    }

    @Override
    public String[] getParameterValues(String name) {
        String[] values = super.getParameterValues(name);
        if (values != null) {
            for (int i = 0; i < values.length; i++) {
                values[i] = HtmlUtil.cleanHtmlTag(values[i]).trim();
            }
        }
        return values;
    }

    @Override
    public Map<String, String[]> getParameterMap() {
        Map<String, String[]> map = super.getParameterMap();
        map.forEach((k, v) -> {
            for (int i = 0; i < v.length; i++) {
                v[i] = HtmlUtil.cleanHtmlTag(v[i]).trim();
            }
        });
        return map;
    }

    @Override
    public ServletInputStream getInputStream() {
        // 对 JSON 请求体整体清洗
        String body = new String(IoUtil.readBytes(in, false));
        body = HtmlUtil.cleanHtmlTag(body).trim();
        // 返回清洗后的流
    }
}

全覆盖清洗：无论上层代码通过 getParameter()、getParameterValues()、getParameterMap() 还是 getInputStream() 获取数据，都会经过 HtmlUtil.cleanHtmlTag() 清洗。没有遗漏的入口。

清洗逻辑：HtmlUtil.cleanHtmlTag() 使用正则表达式移除 HTML 标签。例如 <script>alert(1)</script> → alert(1)。注意它不是 HTML 转义，而是直接剥离标签——攻击代码被移除，留下纯文本内容。

4.3 第三层：Config 层 —— 配置驱动

XssProperties 提供配置入口：

@ConfigurationProperties(prefix = "xss")
public class XssProperties {
    private boolean enabled;          // 主开关：xss.enabled
    private String excludeUrls;       // 排除路径：xss.excludeUrls
}

application.yml 配置示例：

xss:
  enabled: true
  excludeUrls: /system/notice,/system/richText/preview
配置项作用默认值
xss.enabled总开关，false 时 Filter 不加载true
xss.excludeUrls逗号分隔的 URL 列表，匹配则跳过清洗空
三层协同：Config 层决定"要不要防"（总开关），Filter 层决定"哪些要防"（路由决策），Wrapper 层决定"怎么防"（清洗实现）。三层职责分明，任何一层的改动不影响其他层。例如想替换清洗算法，只需修改 Wrapper 层即可。

5. PlusWebInvokeTimeInterceptor：请求耗时统计

5.1 StopWatch 计时机制

PlusWebInvokeTimeInterceptor 实现了 HandlerInterceptor 接口，通过 Spring MVC 的拦截器机制嵌入请求处理流程：

public class PlusWebInvokeTimeInterceptor implements HandlerInterceptor {

    // ThreadLocal 保证线程安全
    private static final ThreadLocal<StopWatch> STOP_WATCH_THREAD_LOCAL = new ThreadLocal<>();

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                             Object handler) {
        StopWatch stopWatch = new StopWatch();
        STOP_WATCH_THREAD_LOCAL.set(stopWatch);
        stopWatch.start();

        // 记录请求信息用于日志
        String url = request.getRequestURI();
        String params = maskSensitiveFields(request.getParameterMap());
        // 从 RepeatedlyRequestWrapper 再次读 body
        String body = getRequestBody(request);

        log.info("请求开始 - URL: {}, 参数: {}, Body: {}", url, params, body);
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                Object handler, Exception ex) {
        StopWatch stopWatch = STOP_WATCH_THREAD_LOCAL.get();
        stopWatch.stop();

        log.info("请求完成 - 耗时: {}ms, URL: {}",
                 stopWatch.getTotalTimeMillis(), request.getRequestURI());

        STOP_WATCH_THREAD_LOCAL.remove();  // 防止内存泄漏
    }
}

5.2 关键设计细节

设计点实现方式原因
线程安全ThreadLocal<StopWatch>每个请求一个 StopWatch 实例，互不干扰
内存泄漏防护afterCompletion 中 remove()线程池复用场景下，不清除会导致脏数据和内存占用
请求体读取依赖 RepeatedlyRequestWrapperbody 已在 Filter 层缓存，Interceptor 可安全读取
敏感字段脱敏password、phone 等字段写入日志前替换为 ***防止密码、手机号等敏感信息泄露到日志文件
精确计时StopWatch 从 preHandle 到 afterCompletion包含 Controller 执行 + 视图渲染 + 拦截器后处理的全链路耗时
为什么不用 Filter 做耗时统计？Filter 虽然也能计时，但它覆盖的范围太广——所有请求（包括静态资源、404 页面）都会经过 Filter。Interceptor 只拦截进入 Spring MVC 的请求（匹配 HandlerMapping 的请求），统计更精准，噪声更少。此外，Interceptor 可以通过 handler 参数获取具体的 Controller 方法信息，日志更有价值。
