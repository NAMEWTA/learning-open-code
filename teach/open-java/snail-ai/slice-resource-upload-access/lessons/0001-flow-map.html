资源上传和访问流程地图

这一课只做一件事：把 Admin 资源上传到公开访问 URL 的主链路画清楚。学完后，你应该能判断问题卡在入口权限、资源元数据、存储后端、storageKey 校验还是 /files/** 响应。

学习目标：画出上传到访问主链路 预计 12 分钟 主来源：ResourceService.java 资源访问参考图谱

1. Admin 入口只收请求

ResourceController 暴露三条入口：POST /resource/upload、DELETE /resource/{id} 和 GET /resource/page。三者都有 @LoginRequired，按当前注解默认值是管理员角色入口。真正业务在 ResourceAdminService：上传时取当前用户作为 creatorId，如果请求没有传 bizType，默认写 GENERAL；分页按 bizType、originalName、bizId 和时间范围过滤；删除前会做资源存在性和所有者校验。

2. 上传先写文件，再写资源行

Admin multipart file
  ↓
ResourceController.upload
  ↓
ResourceAdminService.upload
  ↓
ResourceService.upload
  ↓
ResourceStorageFactory.getDefault()
  ↓
LocalResourceStorageService 或 MinioResourceStorageService
  ↓
sai_resource(storageKey, storageType, accessUrl, bizType, creatorId)

ResourceService.upload 先让默认存储策略执行 store，拿到 storageKey，再检测 mimeType 并插入 ResourcePO。插入后才调用 getAccessUrl(resourceId, storageKey) 回写 accessUrl。当前 LOCAL 和 MINIO 都生成 /files/{storageKey}，所以 URL 看起来一样，差异在底层文件位置和对象存储配置。

ResourceStorageService storage = storageFactory.getDefault();
String storageKey = storage.store(bizType, originalName, inputStream);
ResourcePO resource = ResourcePO.builder()
        .storageKey(storageKey)
        .storageType(storage.getType())
        .bizType(bizType)
        .creatorId(creatorId)
        .build();
resourceMapper.insert(resource);
resource.setAccessUrl(storage.getAccessUrl(resource.getId(), storageKey));
resourceMapper.updateById(resource);

3. 访问入口按 storageKey 反查

sequenceDiagram
  participant Browser as Browser
  participant Files as FileAccessController
  participant DB as ResourceMapper
  participant Factory as ResourceStorageFactory
  participant Store as LOCAL/MINIO
  Browser->>Files: GET /files/{storageKey}?download=true
  Files->>Files: 解码 URI 并校验 storageKey 正则
  Files->>DB: getByStorageKey(storageKey)
  alt 资源不存在或 storageType=EXTERNAL
    Files-->>Browser: 404
  else LOCAL 或 MINIO
    Files->>Factory: get(resource.storageType)
    Files->>Store: load(storageKey)
    Files-->>Browser: InputStreamResource + Content-Type
  end

/files/** 被认证拦截器排除，因此访问不再看登录态，而是依赖难猜的 storageKey。FileAccessController 会要求路径符合 {bizType}/{yyyy}/{MM}/{dd}/{uuid} 加可选扩展名的形态，再查 sai_resource.storage_key。它只代理 LOCAL 和 MINIO；EXTERNAL 是外部 URL 资源，不应通过 /files/** 读取。

4. 排障先定位边界

上传成功但访问失败时，先查 sai_resource：确认 access_url 是否是 /files/...，storage_type 是否为 LOCAL 或 MINIO，storage_key 是否满足日期路径和 UUID 格式。若是 LOCAL，检查 snail-ai.resource.upload-dir 下文件是否存在；若是 MINIO，检查 endpoint、凭据、bucket 和对象 key；若是删除问题，要记住文件删除失败只记录 warning，资源行仍会被删除。

暂停回忆：不看流程图，判断三件事：accessUrl 是在资源行插入前还是插入后生成？/files/** 会检查登录 token 吗？EXTERNAL 能否由 FileAccessController 代理读取？
参考答案：accessUrl 在插入后生成并回写；/files/** 被拦截器排除，不检查登录 token；EXTERNAL 会被文件访问入口返回 404，应直接使用它自己的外部 accessUrl。

主要推荐来源：先读 ResourceService.java，再对照 资源访问参考图谱 查 API、字段、存储策略差异和调试 checklist。遇到生产环境 404 或权限边界问题时，继续向 agent 追问具体部署配置和日志样本。
