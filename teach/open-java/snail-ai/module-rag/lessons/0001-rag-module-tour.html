1. 职责：给模型补上下文

在 Snail AI 的全局架构里，RAG 是 Server 侧的上下文能力，不是 Chat 模型本身。它负责把用户上传或导入的文档变成可检索的 chunk，并在提问时找回相关片段，交给问答服务或 Agent 工具链注入 Prompt。你可以把它分成两条线：摄入线把“文件”变成“索引”；检索线把“query”变成“参考资料”。真正的模型调用由模型运行层处理，RAG 只决定哪些知识片段值得带给模型。

2. 位置：Admin 入口到 feature-rag

snail-ai-server/ ├── snail-ai-server-admin/ ← 管理端 HTTP 入口 │ └── controller/ │ ├── RagDocumentController.java ← 上传、预览、提交、重处理、分页 │ └── RagSearchController.java ← 知识库、搜索调试、流式问答 └── snail-ai-server-features/ ← Server 侧业务能力 └── snail-ai-feature-rag/ ← RAG 核心实现 ├── pipeline/ ← 文档摄入管线 ├── search/ ← 检索管线与 handler └── strategy/ ← parser、chunker、fusion、rerank

Admin 层只接 HTTP 请求并转给 service。若需要先回到系统全图，可对照 L0 项目地图 再看这里的模块位置。文档上传会先落资源和文档行，状态为 PENDING；后台定时任务再批量扫描 pending 文档，调用 DocumentPipeline.processDocument。搜索调试走 POST /rag/search，流式知识库问答走 POST /rag/qa/stream。运行时 Agent 也能通过 gRPC 回调进入同一套 RagSearchService。

3. 分层：两条管线加四类策略

Admin / gRPC
  ↓
RagDocumentService / RagSearchService
  ↓
DocumentPipeline        RagSearchPipeline
  ↓                     ↓
parser + chunker        Config → Vector → BM25 → Fusion → Rerank → Finalize
  ↓                     ↓
RagDocument/RagChunk    SearchResult + Metrics
  ↓                     ↓
VectorStore + SearchEngine + DB

摄入管线依赖 DocumentParserFactory 选择 PDF、DOCX、Excel、Markdown、HTML 或 TXT parser，再由 DocumentChunkingService 按 default、delimiter、regex、smart 分片。检索管线很薄：它只遍历 Spring 注入的 RagSearchHandler 列表。当前 handler 顺序是配置解析、QueryRewrite 空实现、向量检索、BM25、融合、重排、收尾。注意官方文档提到 Query 改写和 PARSING 状态，但当前源码里 Query 改写已移除，文档处理也直接从 PENDING 抢占到 PROCESSING。

4. 最小调用示例

POST /rag/search
Content-Type: application/json

{
  "ragId": 12,
  "query": "如何配置向量库？",
  "debug": true
}

这个请求由 RagSearchController 转成 RagSearchRequestDTO，再交给 RagSearchService.search。如果 debug 为 true，响应会带上 vector、BM25、fusion、rerank 和 total 耗时。下一步深入时，把摄入细节交给 slice-rag-ingestion，把检索注入细节交给 slice-rag-retrieval；本课只要求你能把文件放回正确层级。

暂停回忆：不看上文，回答三个判断：上传接口会立刻解析文档吗？BM25 关闭时融合阶段还会工作吗？RAG 搜索结果最终由哪个 DTO 返回？
参考答案：不会，上传后主要落 PENDING，定时任务触发处理；BM25 没有结果时融合 handler 直接返回，保留向量结果；搜索响应是 RagSearchResponseDTO，其中结果项使用公共 SearchResult。
