Commons 与 gRPC 基础模块导览

本课只完成一个目标：看懂 commons-core 的共享 DTO/枚举/工具类，如何通过 commons-grpc 的统一信封支撑 Server 与 Agent Client 之间的心跳、回调和聊天分发。

层级：L1 模块总览 预计 12 分钟 前置：L0 项目地图

1. 先分清两层契约

snail-ai-commons 有两个子模块。commons-core 放跨端业务对象，例如 ChatDispatchRequest、ChatStreamResponse、RagSearchRequest、SkillContentResponse、枚举、Result 和 JsonUtil。commons-grpc 不表达业务领域，它只提供统一 gRPC 外壳、常量、分发 SPI 和调用工具。

snail-ai-commons/ ├── snail-ai-commons-core/ ← DTO、枚举、Result、JsonUtil、异常和通用工具 └── snail-ai-commons-grpc/ ← proto 信封、URI 常量、handler SPI、client/server 工具 └── src/ └── main/ ├── proto/ ← 统一 request/result/metadata 消息 └── java/ ← 生成类、GrpcChannelUtil、Dispatcher、ServiceDefinitionBuilder

记住这个边界：业务字段变动先看 commons-core DTO；传输方法、header、URI 或流式方式变动才看 commons-grpc。

2. gRPC 只传信封，业务放 JSON

当前 proto 只有三个 message：请求、响应和 metadata。没有 ChatRequest、HeartbeatRequest 这样的业务 message，也没有生成 service stub。GrpcChannelUtil 会手工构建 unary 或 server-streaming MethodDescriptor，把 uri 写进 metadata，把业务 DTO 序列化成 body 字符串。

open-java/snail-ai/snail-ai-commons/snail-ai-commons-grpc/src/main/proto/snail_ai_grpc_service.proto

message GrpcSnailAiRequest {
  int64 reqId = 1;
  Metadata metadata = 2;
  string body = 3;
}

message Metadata {
  string uri = 3;
  map<string, string> headers = 7;
}

GrpcRequestDispatcher 再按 uri 找 GrpcRequestHandler 或 GrpcStreamingRequestHandler。因此 /beat、/callback/rag/search、/chat/dispatch 是真正的业务路由键；proto 字段本身只是运输容器。

3. Server 与 Client 怎么接上

Client 启动后，ClientHeartbeatScheduler 每 10 秒用 unary /beat 上报 app-id、token、host-id、host-ip、host-port 和并发数据。Server 的 BeatRequestHandler 校验 token 后交给 ClientInstanceManager 注册实例、刷新 DB 过期时间并创建到 Client 的 channel。

用户对话走相反方向：Server 责任链终点 LlmCallHandler 把上下文装成 ChatDispatchRequest，通过 server-streaming /chat/dispatch 发给目标 Client；Client 的 ChatDispatchStreamingHandler 解析 JSON，执行本地模型调用后，把 ChatStreamResponse 序列化进 GrpcSnailAiResult.data 流式返回。

ClientHeartbeatScheduler └─ unary /beat └─ SnailAiServerGrpcServer └─ BeatRequestHandler └─ ClientInstanceManager LlmCallHandler └─ streaming /chat/dispatch └─ ClientGrpcServer └─ ChatDispatchStreamingHandler └─ ChatStreamResponse in result.data

Agent Client 也会通过 RpcClient 回调 Server，例如保存会话记录、读取 Skill、执行 RAG 搜索。这个接口同样走 unary 信封，只是 path 来自 @Mapping 注解。

4. 检索练习

暂停回忆：如果一次聊天流能连上 Client，但 Client 报“解析 dispatch body 失败”，你优先检查 proto、UriConstants.CHAT_DISPATCH，还是 ChatDispatchRequest 与 JsonUtil？为什么？

参考答案与反馈

优先检查 ChatDispatchRequest 与 JsonUtil。连接已建立且命中了 /chat/dispatch，说明 gRPC 信封和 URI 路由大概率可用；失败点更可能是 Server 序列化出的 JSON 与 Client 反序列化目标 DTO 不匹配。若错误是 unknown URI，才优先查 UriConstants 和 handler 注册；若连方法都找不到，才回到 gRPC 服务名、方法名或 proto 生成类。

主要阅读来源：ChatDispatchRequest、ChatStreamResponse、snail_ai_grpc_service.proto、GrpcChannelUtil、BeatRequestHandler 和 ChatDispatchStreamingHandler。接着打开 速查参考，按 DTO、URI、Server feature-common、Agent Client 边界和兼容风险五块复盘。遇到字段版本、URI 扩展或心跳注册问题，可以继续向 agent 追问具体调用栈。
