1. 在分发链路中的位置

Agent 聊天责任链 ContextCollectorHandler (@Order 75) │ ├─ instanceManager.getAliveInstances(appId) → List<ClientInstanceInfo> candidates ├─ app.routeStrategy（空则 LEAST_LOAD） → routeStrategyManager.get(type) └─ routeStrategy.select(candidates, conversationId) → ctx.setTargetClient(...) │ └─ LlmCallHandler 用 targetClient 的 hostIp:grpcPort 发 /chat/dispatch

4. 六种实现：算法、线程安全、场景

策略总览

实现类算法摘要使用 routeKey线程安全典型场景
FirstClientRoute 始终 candidates.get(0) 否 无状态，并发安全 单 Client、主备、开发环境
RandomClientRoute ThreadLocalRandom 均匀抽索引 否 ThreadLocalRandom 线程本地，安全 测试、对均衡要求不高的同构集群
RoundRobinClientRoute AtomicLong 自增后对候选数取模 否 getAndIncrement 原子，安全 同构集群、希望长期均匀分摊
ConsistentHashClientRoute 每节点 100 虚拟节点上环；routeKey 哈希后顺时针找宿主 是（会话亲和） 每次 select 重建环，无共享可变状态 同一会话固定落同一 Client
LruClientRoute 按 routeKey 记住上次 hostId，优先选不同节点；缓存最多 100 个 routeKey 是 synchronized(accessOrder) 同会话在节点间轮换，减轻单点粘连
LeastLoadClientRoute min(activeChats / maxConcurrent) 否 无状态，并发安全 生产默认；异构并发容量

候选列表顺序取决于 getAliveInstances 缓存迭代顺序，不是 DB 主键序。「第一个」= 当前快照中的首个活跃实例。

RandomClientRoute

private int generateRandomIndex(int candidateSize) {
    return ThreadLocalRandom.current().nextInt(candidateSize);
}

RoundRobinClientRoute

private int calculateIndex(int candidateSize) {
    long currentCount = counter.getAndIncrement();
    int index = (int) (currentCount % candidateSize);
    return Math.abs(index);
}

计数器为全局（非按 appId 分桶）。多应用共享同一 JVM 时轮询相位连续，不影响单 app 内均匀性，但不同 app 的请求会共用计数序列。

ConsistentHashClientRoute

参数值说明
VIRTUAL_NODES_COUNT100每个物理节点映射 100 个虚拟点
HASH_SEED31字符串滚动哈希（非 Murmur/CRC）
虚拟键格式{hostId}#{index}区分同 host 多虚拟点
 
private ClientInstanceInfo selectFromRing(TreeMap<Long, ClientInstanceInfo> ring, String routeKey) {
    long keyHash = calculateHash(routeKey);
    SortedMap<Long, ClientInstanceInfo> tailMap = ring.tailMap(keyHash);
    return tailMap.isEmpty()
            ? ring.firstEntry().getValue()
            : tailMap.get(tailMap.firstKey());
}

每次调用都 buildHashRing，节点数变化会立即反映；无跨请求环缓存。官方文档 docs/guide/app/route-strategy.md 描述一致，但实现未做环持久化。

LruClientRoute

synchronized (accessOrder) {
    String lastUsedHostId = accessOrder.get(routeKey);
    for (ClientInstanceInfo c : candidates) {
        if (!c.getHostId().equals(lastUsedHostId)) {
            accessOrder.put(routeKey, c.getHostId());
            return c;
        }
    }
    ClientInstanceInfo first = candidates.get(0);
    accessOrder.put(routeKey, first.getHostId());
    return first;
}

与官方文档差异：源码按 conversationId → 上次 hostId 做「避开上次节点」轮换，并非按全局节点最近分配时间排序的经典 LRU。单节点时行为退化为始终选 candidates.get(0)。

6. 决策速查

需求推荐策略原因
生产通用 / 节点并发上限不同LEAST_LOAD读心跳 activeChats/maxConcurrent，动态避过载
同构集群、长期均匀ROUND_ROBIN原子计数轮询，不依赖心跳精度
会话粘滞（同 conversationId 同节点）CONSISTENT_HASHrouteKey 即 conversationId
同会话刻意轮换 ClientLRU按会话记住上次 host，下次选其他节点
单 Client 或主备FIRST零开销，行为可预测
压测 / 快速验证RANDOM无状态，实现最简单

检索练习：某 Agent 绑定应用 A，A 配置 CONSISTENT_HASH，同一 conversationId 连续发两条消息，第二条是否保证落到同一 Client？若此时新增一个 Client 节点上线，该会话是否一定仍落原节点？
参考答案

同一 conversationId 在节点集合不变时，哈希环映射稳定，通常仍落同一 Client。新增节点会重建环，部分会话（含该会话）可能映射到新节点——与一致性哈希扩缩容特性一致，非 bug。
