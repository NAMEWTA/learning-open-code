Commons 与 gRPC 基础模块速查

这份参考用于快速定位 snail-ai-commons-core 与 snail-ai-commons-grpc 的共享契约。短课负责建立地图，本文负责放长清单、边界和风险。

1. 模块定位

模块职责主要消费者关键源码
snail-ai-commons-core 公共 DTO、枚举、响应模型、异常、JSON 工具、RAG 工具和 OpenAPI DTO。 Server features、Admin/OpenAPI、Agent Client、Models、Persistence。 open-java/snail-ai/snail-ai-commons/snail-ai-commons-core/src/main/java/com/aizuda/snail/ai/common
snail-ai-commons-grpc 统一 gRPC request/result 信封、URI/header 常量、unary/streaming handler SPI、client/server 构建工具。 Server feature-common、Agent common、Agent executor、Server feature-agent/RAG/Skill。 open-java/snail-ai/snail-ai-commons/snail-ai-commons-grpc/src/main/java/com/aizuda/snail/ai/common/grpc

2. 公共 DTO 与枚举分类

主题范围说明：本页不是 commons-core 的完整 API 字典，而是面向 gRPC、聊天分发、OpenAPI 和跨模块共享契约的公共类型索引；未展开异常、日志、WebSearch 等旁支类型。

类别对象用途风险提示
agent DTO ChatDispatchRequest、ChatStreamResponse、ConversationCreateRequest、ConversationRecordRequest、SkillContentRequest、SkillContentResponse、HeartbeatBody 聊天分发、流式回传、会话回调、Skill 内容回调和旧心跳模型。ChatDispatchRequest 还内嵌 agent/model/MCP/Skill/history 描述符。 新增必填字段会影响 Client 反序列化后的执行期校验；字段语义变更要同步 Server 与 Client。
beat DTO HeartbeatBodyRequest、HeartbeatBody 上报 maxConcurrentChats 与 activeChats。当前 Server handler 使用 HeartbeatBodyRequest。 两者字段相同但包不同，后续应避免继续扩散重复 DTO。
mcp DTO / 引用 ChatDispatchRequest.McpServerDescriptor、McpServerRef 分发到 Client 的 MCP server 运行参数，以及公共 MCP 引用模型。 transport、endpoint、authConfig、envVars 会影响 Client 侧工具初始化；只改 Server 配置表不等于 Client 可兼容。
model DTO ChatDispatchRequest.ModelConfig、ConfigExtAttrsDTO、EmbeddingModelConfigDTO、EmbeddingResponseDTO 聊天分发携带模型 provider、adapter、endpoint、apiKey 和扩展配置；公共 model 包承载 embedding / rerank 调用形状。 模型配置常包含密钥和 endpoint；日志、课程截取和跨进程传输都要避免泄露真实凭据。
rag DTO RagSearchRequest、RagSearchResponse、SearchResult Client 本地 RAG 工具通过 Server 检索知识库，返回 chunkId、content、score、documentName 和 metadata。 score 是 double，排序和阈值语义由 RAG 模块决定，Commons 只保存结果形状。
memory DTO ShortTermMemoryRequest、MemoryRetrieveRequest 短期历史加载和长期记忆检索请求模型。当前短期历史 URI 只在 RpcClient 声明；长期检索只有 URI 常量和 DTO 预留。 不要把 DTO 存在误读为 gRPC 闭环已接通；Server 聊天链路当前由 ContextCollectorHandler 直接读取 ShortTermMemoryStore。
OpenAPI DTO OpenApiChatRequest 等 common/openapi/dto 类型 OpenAPI chat、agent、conversation、user、embed token 等接口共享请求/响应模型；聊天入口代表是 OpenApiChatRequest。 OpenAPI DTO 面向 HTTP/SSE 客户端，不等同于 gRPC 信封内的 ChatDispatchRequest。
响应模型 Result<T>、PageResult<T> HTTP/API 层通用成功失败包装，status 默认 1。 gRPC result 另有 GrpcSnailAiResult.status，不要混用。
枚举 CommonStatusEnum、AgentStatusEnum、MemoryStatusEnum、StoreCategoryEnum、RagCallModeEnum、McpTransportTypeEnum、McpAuthTypeEnum、MemoryTypeEnum、DocumentStatusEnum、StoreInstanceTypeEnum 通用启停、Agent 状态、记忆状态、存储分类、RAG 调用方式、MCP transport/auth、记忆类型和 RAG 文档/存储实例状态。 带 @EnumValue 的枚举与数据库值绑定；改值比改描述风险高。
工具 JsonUtil、RagResultReorderUtil JSON 序列化/反序列化和 RAG lost-in-the-middle 重排。 JsonUtil 忽略未知字段、空值不序列化，利于增量字段兼容，但会隐藏拼写错误。

3. gRPC proto、方法和消息

当前源码没有业务 service stub。通信方法由 GrpcServiceDefinitionBuilder 和 GrpcChannelUtil 共同使用固定服务名、方法名和统一消息类型手工构建。

层级名称类型或字段说明
消息GrpcSnailAiRequestreqId、metadata、body所有请求的统一信封。body 是 JSON 字符串。
消息GrpcSnailAiResultreqId、status、message、data所有响应的统一信封。业务响应通常放在 data JSON 中。
消息Metadatauri、headersuri 负责业务路由，headers 负责 app/token/host/version 等元数据。
Unary 方法UnaryRequest/unaryRequestGrpcSnailAiRequest -> GrpcSnailAiResult心跳、ping、Client 回调 Server 都走这个形态。
Server Streaming 方法ServerStreamingRequest/serverStreamingRequestGrpcSnailAiRequest -> stream GrpcSnailAiResult聊天分发走这个形态，Client 连续返回 text/thinking/completion/error。

方向URI方式处理边界

Client 到 Server/callback/conversation/recordUnaryAgent feature 回调 handler 保存助手消息、thinking 和 token。

5. Agent Client 消费边界

边界组件它消费或产生什么不要误解成
本地 gRPC 入口 ClientGrpcServer 注册 unary 和 server-streaming service，接收 Server 的 /ping 与 /chat/dispatch。 它不执行业务，只把请求交给外部传入的 handler。
启动装配 SnailAiAgentAutoConfiguration 启动 Client gRPC server、创建 RpcClient 动态代理、启动心跳任务。 它不是 Server feature-common 的一部分。
心跳 ClientHeartbeatScheduler 构建 app/token/host headers 和 HeartbeatBodyRequest，调用 unary /beat。 心跳不是聊天分发；它只让 Server 发现和更新节点状态。
回调 Server RpcClient + GrpcClientInvokeHandler 按 @Mapping path 把 DTO 转 Map，再通过 unary 请求 Server。 它不是 REST client，也不会使用业务 proto message。
聊天消费 ChatDispatchStreamingHandler 解析 ChatDispatchRequest，调用会话运行时，按 text/thinking/completion/error 写回 GrpcSnailAiResult.data。 它不负责选择 Client，不负责 Server 上下文装配。
Ping PingRequestHandler 返回 timestamp 与 activeChats。 它不等价于心跳注册；心跳方向是 Client 到 Server。

6. 兼容风险清单

变更点影响面当前源码依据处理原则
修改 proto 字段号或消息名 所有 gRPC 请求和响应。 snail_ai_grpc_service.proto 只有统一信封，已生成 Java 类随源码提交；gRPC Maven 生成插件当前被注释。 不能只改 proto；必须同步生成类、client/server marshaller 和双方发布版本。
修改 GrpcConstants 服务名或方法名 方法匹配失败，Server 与 Client 无法互通。 GrpcChannelUtil 和 GrpcServiceDefinitionBuilder 都使用相同常量手工构建 descriptor。 服务名、方法名应作为跨端协议稳定维护。
修改或新增 UriConstants 业务路由找不到 handler，返回 unknown URI。 GrpcRequestDispatcher 只按 handler.supports(uri) 顺序匹配。 新增 URI 时要同时新增发送端 mapping 和接收端 handler。
修改 header 名称 心跳认证、实例注册和回调 header 透传。 HeaderConstants 定义 app-id、token、host-id、host-ip、host-port、version；GrpcHandlerRequest 直接读取这些键。 header 变更必须双端同步，并注意 host-port 非法值会触发 parseInt 异常。
新增 DTO 字段 JSON body 兼容。 JsonUtil 反序列化忽略未知属性，序列化跳过 null 字段。 可选字段通常可渐进发布；必填语义应先让消费端兼容默认值。
删除或重命名 DTO 字段 Server 与 Client 运行期行为偏差。 业务 DTO 没有 schema 校验，反序列化成功也可能得到 null。 先引入新字段并双写/双读，再清理旧字段。
新增 ChatStreamResponse.type Server SSE 转发和持久化逻辑。 当前固定值为 text、thinking、completion、error。 新增类型时同步 Server observer、前端消费和持久化策略。
调整心跳间隔或过期阈值 Client 上下线判定和路由可用性。 Client 每 10 秒心跳；Server 内存过期阈值 40 秒，DB expireDt 偏移 40 秒。 间隔、过期、DB 清理和路由降级要成组调整。
官方文档与源码不一致 学习和排查误判。 docs/architecture/distributed.md 描述了示例业务 proto service，但当前源码使用统一信封和 URI 分发。 以当前源码、pom 和生成类为准；文档只作为背景材料。
