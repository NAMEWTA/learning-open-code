1. 示例工程站在 Server 外面

snail-ai-agent-example 不是 Server OpenAPI 模块；它是一个独立 Spring Boot 示例应用，依赖 snail-ai-openapi-starter，暴露本地 /demo/** 接口，再用 SDK 访问 Server 的 /snail-ai/openapi/v1/**。它还启用 @EnableSnailAiAgent，所以同一个进程也可以作为 Agent Client 接收 Server 的 gRPC 分发。

业务调用方
  -> 示例工程 /demo/agent/1/chat/sync 或 /demo/agent/1/chat/stream
  -> OpenApiChatClient 动态代理
  -> OpenApiHttpInvokeHandler
  -> http://127.0.0.1:8900/snail-ai/openapi/v1/agent/chat
  -> OpenApiChatController
  -> OpenApiChatService
  -> 已有 slice-openapi-chat 后半段：Agent 链、gRPC、Client 模型执行

2. SDK 负责 URL 和认证头

SDK 的 SnailAiOpenApiAutoConfiguration 在 snail-ai.openapi.enabled=true 时创建 OpenApiChatClient 代理。代理里的 OpenApiHttpInvokeHandler 使用 serverHost、webPort、https、prefix 拼出 base URL，再从 SnailAiAgentProperties 读取 appId 和 token 写成 Server 需要的两个 Header。

private HttpRequest.Builder baseRequest(String url) {
    return HttpRequest.newBuilder()
            .uri(URI.create(url))
            .timeout(Duration.ofMillis(openApiProperties.getReadTimeoutMs()))
            .header("Content-Type", "application/json")
            .header("Snail-Ai-App-Id", agentProperties.getAppId())
            .header("Snail-Ai-Token", agentProperties.getToken());
}

因此排查 401/认证类失败时，先不要看 /demo Header；示例工程入口没有读取外部 Header，真正发给 Server 的凭证来自应用配置。

3. chat 分成同步和 SSE 两条出口

同步示例把请求体补上 agentId 后直接调用 chatClient.chatSync(request)，返回 Result<OpenApiChatSyncResponse>。流式示例把 query 参数组装成 OpenApiChatRequest，调用 chatClient.chatStream(request)，再把 SDK 收到的 OpenApiChatStreamEvent 映射成本地 ServerSentEvent。

return chatClient.chatStream(request)
        .map(event -> ServerSentEvent.<String>builder(event.getData())
                .event(event.getType())
                .build());

SDK 侧当前源码没有文档里的 listener(...) Builder 方法；SnailAiOpenApi.chat(agentId).stream() 返回的是 Flux<OpenApiChatStreamEvent>。Server 侧事件类型来自 OpenApiChatStreamEvent：text、thinking、done、error。

conversationId 是一个容易踩坑的字段：示例工程流式入口把它标成可选 query 参数，但当前 Server DTO 上是 @NotBlank。按源码调用时，缺失它可能在进入 chat 服务前被参数校验挡住。
