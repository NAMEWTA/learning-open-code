Agent 责任链回调与边缘 Handler API

本页是 L3 微观 API 字典，覆盖责任链中的边缘 Handler 与 Client 回调 Handler。agent-chain-overview.html 负责模块地图与完整 Handler 顺序表；本文聚焦 WebSearchHandler、ConversationCreateCallbackHandler、ConversationRecordCallbackHandler 三类的入参/出参、触发时机，以及与主链 Handler 的差异。

覆盖范围 3 个 Handler 类 + 2 个回调 DTO + 2 个接口契约（AgentChatHandler / GrpcRequestHandler）
源码 commit 2ae56c0（open-java/snail-ai）

1. 两类 Handler 在架构中的位置

本模块存在两条互不串联的 Handler 管线：

类型 接口 触发方 执行器 典型时机
主责任链 Handler AgentChatHandler Server 内部（REST/SSE 对话入口） AgentChatChainService.proceed 用户发起一次对话请求后，按 @Order 同步顺序执行
回调 Handler GrpcRequestHandler Agent Client 经 gRPC unary 回调 GrpcRequestDispatcher.dispatchUnary Client 通过 RpcClient 动态代理调用 Server 时

关键边界：回调 Handler 不在 AgentChatChainService 的 Handler 列表中；它们由 Spring 注入到 GrpcRequestDispatcher，按 supports(uri) 路由。主链的会话持久化主要由 ConversationHandler（用户消息）和 ChatResultPersistService（助手消息）完成；回调 Handler 提供Client 侧主动回写的备用通道。

2. 主链 vs 回调：持久化路径对照

动作 主链路径（当前默认） 回调路径（基础设施已就绪）
创建会话 + 保存用户消息 ConversationHandler @Order(20)，同步写库 Client 调用 RpcClient.createConversation → ConversationCreateCallbackHandler
保存单条对话记录 ChatResultPersistService.persistAll（流结束后异步） Client 调用 RpcClient.saveRecord → ConversationRecordCallbackHandler
联网搜索工具注入 WebSearchHandler（设计 order=65，当前未注册） 无对应回调；工具在 Client 端由 ToolRuntime 解析

3. WebSearchHandler — 主链边缘 Handler

snail-ai-server/snail-ai-server-features/snail-ai-feature-agent/src/main/java/com/aizuda/snail/ai/feature/agent/chain/WebSearchHandler.java

在链中的位置

属性值
设计 order65（介于 RagHandler 60 与 SkillAgentChatHandler 70 之间）
当前注册状态@Component 与 @Order(65) 均被注释，默认不进入 Spring Handler 列表
接口AgentChatHandler
前置依赖InitContextHandler 已写入 ctx.agent；建议在 LlmCallHandler 之前执行以填充 toolCallbacks

方法 API

方法签名入参出参 / 副作用边界 / 异常
handle void handle(AgentChatContext ctx) ctx.agent.webSearchEnabled、ctx.toolCallbacks（已有列表） 向 ctx.toolCallbacks 追加 WebSearchTool 的 ToolCallback 见下方触发条件表；不设置 terminated，失败时静默跳过

触发条件与短路语义

条件行为
agent.webSearchEnabled != true直接 return，不修改上下文
snail-ai.web-search.tavily.api-key 为空记录 warn 日志后 return
两项均满足实例化 WebSearchTool(apiKey, maxResults) 并注入回调列表

配置项

配置键类型默认用途
snail-ai.web-search.tavily.api-keyString空Tavily API 密钥
snail-ai.web-search.max-resultsint5单次搜索最大结果数

与主链 Handler 的差异

维度主链典型 Handler（如 McpHandler）WebSearchHandler
写入目标ChatDispatchRequest 可序列化字段（如 mcpServerDescriptors）仅写内存 toolCallbacks，当前 LlmCallHandler.buildDispatchRequest 未读取 该字段
失败策略部分 Handler 会 terminated=true 短路永不终止链路，配置缺失时跳过
注册方式@Component + @Order注释掉注解，属于预留扩展点
执行位置Server 同步责任链同上（若启用），但工具实际执行在 Client ToolRuntime

实现注意：当前源码中 handle 先把已有 toolCallbacks 复制到新列表再 addAll 回原列表，会导致列表内容翻倍；启用前需修复。且 ChatDispatchRequest 无 tools 字段，即使注册该 Handler，联网搜索工具也无法经 gRPC 下发到 Client——完整打通需同步扩展 DTO 与 LlmCallHandler。

4. ConversationCreateCallbackHandler — 会话创建回调

snail-ai-server/snail-ai-server-features/snail-ai-feature-agent/src/main/java/com/aizuda/snail/ai/feature/agent/callback/ConversationCreateCallbackHandler.java

在系统中的位置

属性值
接口GrpcRequestHandler
URI/callback/conversation/create（UriConstants.CALLBACK_CONVERSATION_CREATE）
Client 入口RpcClient.createConversation(ConversationCreateRequest)
传输gRPC unary；body 为 DTO 的 JSON；经 GrpcClientInvokeHandler 代理发送
注册@Component，由 GrpcRequestDispatcher 自动收集

方法 API

方法签名入参返回值边界 / 异常
supports boolean supports(String uri) gRPC 请求 URI URI 等于 CALLBACK_CONVERSATION_CREATE 时返回 true —
handle GrpcSnailAiResult handle(GrpcHandlerRequest request) request.body → ConversationCreateRequest 成功：status=1, message="OK"；失败：status=0, message=异常信息 任意异常被 catch，记录 error 日志，不向外抛出

请求 DTO：ConversationCreateRequest

字段类型必填用途
agentIdLong是智能体 ID
userIdLong是用户 ID
conversationIdString是会话唯一标识
userMessageString是用户首条消息内容

处理逻辑与数据库副作用

  1. 按 conversationId 查询会话；不存在则插入 AgentConversationPO，标题取 userMessage 前 16 字符。
  2. 插入 AgentConversationRecordPO：role=USER，status=RUNNING。

与 ConversationHandler 逻辑高度重叠，但回调路径由 Client 主动发起，适用于不经 Server 主链、或 Client 独立管理会话生命周期的场景。当前 executor 源码中尚无对 createConversation 的调用点。

属性值

方法 API

对比 ChatResultPersistService：主链持久化会写入 thinking、token 统计，并追加短期记忆与用量统计；回调 Handler 仅做裸插入，无后续副作用。

6. 回调触发时序

Client 业务代码调用 rpcClient.createConversation(req) 或 saveRecord(req)
GrpcClientInvokeHandler读取 @Mapping.path，DTO 序列化为 JSON body，经 GrpcChannelUtil.sendUnary 发往 Server
GrpcRequestDispatcher遍历 List<GrpcRequestHandler>，首个 supports(uri)==true 的 Handler 处理
Callback Handler解析 body → 写库 → 返回 GrpcSnailAiResult
Client 代理校验status != 1 时抛 CallbackServerErrorException；网络失败按配置重试

7. 与主链 Handler 差异总表

对比维度 主链 Handler（如 ConversationHandler） 本文三个 Handler
编排入口 AgentChatChainService 有序循环 WebSearchHandler：同主链（若注册）；回调类：GrpcRequestDispatcher
上下文对象 AgentChatContext WebSearchHandler 用 AgentChatContext；回调用 GrpcHandlerRequest
返回值 void，通过 mutating context 传递状态 回调返回 GrpcSnailAiResult；WebSearchHandler 无返回值
失败处理 可 terminated=true 或抛异常中断 回调吞异常转 status=0；WebSearchHandler 静默跳过
触发时机 每次 Server 对话请求，同步 回调：Client 主动发起；WebSearch：主链装配阶段（未启用）
与 gRPC 分发关系 在 LlmCallHandler 之前/之中执行 回调在分发之外独立到达；WebSearch 应在分发前填充工具（当前未接通）

Client 侧触发会话创建回调

rpcClient.createConversation(ConversationCreateRequest.builder()
        .agentId(1L)
        .userId(100L)
        .conversationId("conv-uuid")
        .userMessage("你好，请介绍自己")
        .build());

Client 侧触发记录保存回调

rpcClient.saveRecord(ConversationRecordRequest.builder()
        .agentId(1L)
        .userId(100L)
        .conversationId("conv-uuid")
        .role("ASSISTANT")
        .content("我是 Snail AI 助手。")
        .build());

遇到 URI 路由、回调重试或主链/回调双写冲突时，可结合 gRPC 切片速查 与 commons-grpc API 继续向 agent 追问。
