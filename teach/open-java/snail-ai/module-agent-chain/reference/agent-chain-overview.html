Agent 责任链模块速查

Server 端 `snail-ai-feature-agent` 的 Handler、上下文、回调、持久化和 gRPC 分发边界。

模块职责与边界

负责什么 加载 Agent 配置、写入会话记录、解析模型、装配 MCP/Skill/RAG、收集短期历史、选择目标 Client、发起 gRPC 流式分发。
不负责什么 不在 Server 端直接调用大模型，不展开 Client Advisor 的 Token 统计、工具循环或模型执行细节。
主要数据流 REST/SSE 入口 → `AgentChatContext` → 有序 Handler → `ChatDispatchRequest` → gRPC Client → `ChatStreamObserver` → `ChatStreamWriter`。

当前源码 Handler 顺序

官方文档 `docs/architecture/agent-chain.md` 说明了 10 个阶段的概念顺序；当前源码以 Spring `@Order` 为准。`WebSearchHandler` 的 `@Component` 与 `@Order` 被注释，默认不会进入 Spring 注入的 Handler 列表。

内部分层结构

层 类或包 判断边界
HTTP/SSE 入口 `AgentController`、`FluxBridgeChatStreamWriter` 把 Web 请求转为 `AgentChatCommand`，并把 Server 输出写回 Flux。
应用编排 `AgentChatService`、`AgentChatChainService` 创建上下文并调用 Handler 列表；不直接读写模型响应。
链路核心 `chain/*Handler.java`、`AgentChatContext` 以 `@Order` 串联能力阶段，使用 `terminated` 表示失败短路。
能力依赖 Model、MCP、RAG、Skill、Memory、Route 相关服务 Handler 只装配描述符或上下文，具体执行大多交给 Client 或下游服务。
远程分发 `LlmCallHandler`、`ChatDispatchRequest`、`GrpcChannelUtil` Server/Client 分界线；请求离开 Server 端责任链。
流式回写与持久化 `ChatStreamObserver`、`ChatResultPersistService`、callback handlers 模型返回后转写 SSE，并保存助手消息、短期记忆和用量统计。

关键依赖

依赖 用在何处 读源码时看什么
Spring `@Component` / `@Order` / `List<AgentChatHandler>` 注册并排序 Handler 源码 order 与文档 order 是否一致；未注册 Handler 不会进入链路。
MyBatis Mapper `InitContextHandler`、`ConversationHandler`、callback、persist 哪些阶段同步写数据库，哪些阶段异步写。
`DispatchModelConfigAssembler` `ModelResolveHandler` Server 分发前需要完整模型配置，而不是只传模型 ID。
`McpToolService`、`SkillHandler` MCP/Skill Handler Server 传描述符和 skill prompt，具体调用交给 Client。
`RagSearchService` `RagHandler` 强制模式 强制模式在 Server 端直接检索并追加 prompt；智能模式只给 Client 工具使用提示。
`ShortTermMemoryStore` `ContextCollectorHandler`、`ChatResultPersistService` 用户消息先写入历史，助手消息在 completion 后写入。
`ClientInstanceManager`、`ClientRouteStrategyManager` `ContextCollectorHandler` Client 选择失败是 gRPC 分发前的主要短路点。
`GrpcChannelUtil`、`UriConstants.CHAT_DISPATCH` `LlmCallHandler` 责任链终点的网络协议和目标 URI。
