Snail AI Starter 模块导览

这一课只做一件事：学会把 snail-ai-starter 当作 Snail AI 的启动壳和配置边界来读。学完后，你应该能判断一个启动、端口、静态页面或部署问题该留在 starter 排查，还是继续下钻到 Server、Agent 或 Persistence。

学习目标：识别 starter 边界 预计 12 分钟 主来源：snail-ai-starter/pom.xml snail-ai-starter/src/main/java/com/aizuda/snail/ai/starter/SnailAiSpringbootApplication.java snail-ai-starter/src/main/resources/application.yml

1. 模块职责

L0 总览/架构图 已经把 Snail AI 分成 Server、Agent Client、Models、Persistence 和 Starter。这里的关键判断是：starter 不实现核心业务链路，它把可运行应用装起来。它的 pom.xml 依赖 snail-ai-server-admin、snail-ai-server-openapi 和三种关系库存储实现，同时引入 Actuator 与 Spring AI MCP Server WebMVC。也就是说，starter 是“把接口、存储适配、配置和静态后台装进一个 Spring Boot 进程”的模块。

<artifactId>snail-ai-starter</artifactId>
<finalName>snail-ai-server</finalName>
<classifier>exec</classifier>
<mainClass>com.aizuda.snail.ai.starter.SnailAiSpringbootApplication</mainClass>

finalName 是 snail-ai-server，Spring Boot repackage 的 classifier 是 exec，所以可执行包是 snail-ai-server-exec.jar，主类固定为 SnailAiSpringbootApplication。如果你要找某个 REST Controller、RAG Pipeline 或模型适配器，入口不在 starter；如果你要找“为什么服务从 8900 起、为什么静态后台能访问、为什么默认连 MySQL”，starter 才是第一站。

2. 启动入口

主类只有少量代码，但信息密度很高。@SpringBootApplication(scanBasePackages = "com.aizuda.snail.ai") 把扫描范围放到整个 Snail AI 包名前缀，starter 的依赖一旦进入 classpath，Server、Persistence、Model 等模块中的 Bean 就会被纳入容器。@EnableAsync 和 @EnableScheduling 说明运行时允许异步任务和定时任务；main 方法先把 JVM 默认时区设为 Asia/Shanghai，再调用 Spring Boot。

public static void main(String[] args) {
    TimeZone.setDefault(TimeZone.getTimeZone("Asia/Shanghai"));
    SpringApplication.run(SnailAiSpringbootApplication.class, args);
}

启动监听和关闭监听位于 listener/ 包：Spring 上下文刷新后打印 Logo，并遍历所有自定义 Lifecycle.start()；上下文关闭时遍历 Lifecycle.close()。这使 PgVector、Milvus、Elasticsearch 等存储注册或清理逻辑不需要写进主类。完整清单见 starter 参考页。

3. 配置边界

application.yml 是 starter 最重要的运行时契约。HTTP 默认监听 8900，上下文路径是 /snail-ai；Server gRPC 监听 18888，供 Agent Client 注册、心跳和任务分发；默认数据库是 MySQL，配置了 Hikari 连接池。资源文件默认走本地目录，Skill 上传目录是 ./upload/skills，短期记忆默认用内存存储，重启后不会保留。

server:
  port: 8900
  servlet:
    context-path: /snail-ai

spring:
  web:
    resources:
      static-locations: classpath:admin/

snail-ai:
  server:
    grpc-port: 18888

这里还划清了前端边界：classpath:admin/ 指向已构建的管理后台静态资源，admin/index.html 引用带 hash 的 JS/CSS 产物。这些文件决定页面如何被 Spring Boot 暴露，但不承载后端业务逻辑，后续源码学习不逐个解析 admin/assets/**。

4. 启动一次

本地开发时，依赖组件可按部署文档用 Docker Compose 启动；Snail AI Server 本体可以直接从根目录运行：

mvn spring-boot:run -pl snail-ai-starter

启动成功后，浏览器入口是 http://localhost:8900/snail-ai；OpenAPI 和 Admin API 共享这个 HTTP 端口；Agent Client 连接的是 18888。日志由 logback-boot.xml 控制：控制台输出所有 info 级别以上日志，同时把 INFO、WARN、ERROR 分别写入 ./data/log/snail-ai/ 下的滚动文件。

snail-ai-starter/ ├── pom.xml ← 聚合依赖、可执行 JAR 和主类声明 └── src/ └── main/ ├── java/ ← Spring Boot 主类和生命周期监听器 └── resources/ ← 应用配置、日志配置和静态后台资源
检索练习：不看正文回答：如果页面能打开但 Agent Client 连不上，应先看 HTTP 端口还是 gRPC 端口？如果你想改后台静态资源的服务位置，应看主类还是 application.yml？
参考答案：Agent Client 连接 Server gRPC，先看 snail-ai.server.grpc-port 与网络连通性；静态资源位置由 spring.web.resources.static-locations 定义，先看 application.yml。如果问题进入具体 Controller、RAG 或模型调用，再跳出 starter 到对应业务模块。

主要推荐来源：先读 snail-ai-starter/src/main/resources/application.yml，再读 docs/deploy/configuration.md 对照每个配置项。遇到不清楚的启动日志或端口问题，可以继续向 agent 追问“这个配置最终被哪个 Bean 使用”。
