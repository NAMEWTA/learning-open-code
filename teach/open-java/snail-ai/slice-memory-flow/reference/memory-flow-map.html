用户消息保存 → 短期历史加载 → 长期召回插槽 → dispatch → completion 持久化 → memory 更新

  └─ insert user AgentConversationRecord

ContextCollectorHandler(@Order 75)

  └─ ctx.memoryContext 当前没有来源

  └─ ChatDispatchRequest(historyMessages, memoryContext, systemPrompt, userMessage)
      └─ gRPC /chat/dispatch

Client DefaultPromptFactory
  ├─ SystemMessage(systemPrompt)
  ├─ HistoryMessage... 
  └─ UserMessage(userMessage)

ChatStreamObserver completion
  └─ ChatResultPersistService.persistAsync
      ├─ insert assistant AgentConversationRecord
      ├─ shortTermMemoryStore.append("assistant", fullText)
      └─ update usage stat

3. 记忆模式与 store 差异

模式当前状态写入时机读取时机边界
短期记忆：db已实现，DbShortTermMemoryStore 为默认匹配实现。append 是空操作；实际写入发生在 ConversationHandler 和 ChatResultPersistService 插入会话记录时。loadHistory 从 AgentConversationRecordMapper 按 createDt DESC 取窗口，再反转为正序并排除最后一条。多实例共享 DB，重启不丢；性能取决于 DB 查询；evict 会删除会话记录。
短期记忆：memory已实现，由 snail-ai.memory.short-term.store-type: memory 激活。ContextCollectorHandler 追加 user；completion 后 ChatResultPersistService 追加 assistant。优先读 JVM 队列；队列为空时从 DB 回填窗口，再排除最后一条。单实例性能好；多实例队列互不共享；进程重启后靠 DB 回填。
长期记忆文档有目标设计，当前主链未闭环。架构文档描述 completion 后抽取、向量化和保存；当前源码未找到主链触发点。架构文档描述对话前向量召回并注入系统提示；当前源码未找到 MemoryInjectionAdvisor 或同等调用。MemoryRetrieveRequest、ConversationMemoryDTO、枚举和 callback URI 存在，但不能证明已接入对话主路径。

memoryEnabled 的命名容易误导。AgentPO 注释写“长期记忆”，但当前 ContextCollectorHandler 和 ChatResultPersistService 用它控制短期历史写读。

assistant 回复记录ChatResultPersistServiceClient completion chunk 回来后异步写入。下一轮短期历史加载。
assistant 短期队列项ChatResultPersistServiceassistant 记录插入后追加。下一轮 InMemoryShortTermMemoryStore#loadHistory。
长期记忆当前主链未确认架构文档描述 completion 后抽取，但当前源码未找到触发服务。当前主链未确认。

6. 失败、重启与并发边界

场景现象排查含义
appId 缺失、应用停用或没有 alive ClientContextCollectorHandler 发送 [ERROR] 并终止；此时 user 记录已经由 ConversationHandler 插入。会话表可能出现只有 user 记录、没有 assistant 回复的记录。
Client 调用失败ChatStreamObserver#handleError 转发错误文本；没有 completion 就不会触发 persistAsync。assistant 记录和 assistant 短期记忆可能缺失。
异步持久化失败ChatResultPersistService#persistAsync 捕获日志错误，不回滚已经发给用户的流式响应。用户看到回答，但下一轮历史可能缺 assistant。
使用 memory store 后重启JVM 队列丢失，下一次 loadHistory 从 DB 回填。如果 DB 中没有 assistant 记录，重启后无法恢复这部分短期上下文。
多实例部署使用 memory store每个实例各自维护 ConcurrentHashMap 队列。同一会话被路由到不同实例时，短期队列不共享；DB 回填是兜底。
同一会话并发提交sid 用于区分提交和回流 chunk，但短期 store 的 key 是 conversationId。并发 user append 和 completion append 可能影响窗口顺序；排查时同时看 sid 和记录创建时间。
长期记忆无效memoryContext 日志可能为空，Client Prompt 不包含长期记忆。先确认是否存在 setMemoryContext 或长期检索 handler，再谈向量召回质量。

7. 调试 checklist

  • 确认当前 Agent 的 memoryEnabled 是否为 false；当前源码用它控制短期历史写读。
  • 确认 shortTermMemorySize 是否为正数；否则使用默认窗口 20。
  • 查看 snail-ai.memory.short-term.store-type，判断当前注入的是 memory 还是 db。
  • 查 ContextCollectorHandler 日志：historySize、shortTermEnabled、memoryPresent。
  • 查 AgentConversationRecord：是否已有当前 user、上一轮 assistant、状态和 createDt 是否符合预期。
  • 如果前端看到 completion 但下一轮没有 assistant 历史，查 ChatResultPersistService 的 persistAll 日志。
  • 如果怀疑长期记忆，先全局搜索 setMemoryContext、MemoryInjectionAdvisor、MemoryRetrieveRequest 的生产者和消费者。
  • 验证最终 Prompt 时，以 DefaultPromptFactory 为准：当前只消费 systemPrompt、historyMessages、userMessage。
