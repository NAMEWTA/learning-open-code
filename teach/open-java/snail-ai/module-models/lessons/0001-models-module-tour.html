模型适配层模块导览

这一课只做一件事：建立 snail-ai-models 的模块边界。学完后，你应该能判断一次模型配置会被哪条能力线接住，并由哪个 adapter 创建实际可调用对象。

学习目标：识别模型适配边界 预计 12 分钟 先回 L0：项目地图 主来源：open-java/snail-ai/snail-ai-models/pom.xml open-java/snail-ai/snail-ai-models/snail-ai-model-chat/snail-ai-model-chat-core/src/main/java/com/aizuda/snail/ai/model/chat/ChatModelRuntime.java

1. 职责：把配置变成模型对象

L0 已把模型能力分成 Chat、Embedding、Rerank；RAG 模块会依赖 embedding/rerank，但真正“怎么创建模型客户端”不在 RAG 里。snail-ai-models 的职责是接收标准化配置，选择匹配的 provider adapter，再产出 Spring AI 的 ChatModel、EmbeddingModel，或 Snail 自己的 RerankApiClient。它不负责管理模型配置页面、不负责解密 API Key、不负责 Prompt 组装，也不负责完整对话运行时；这些事情分别属于 Server、Agent 或 RAG。

2. 位置：四条 Maven 边界

snail-ai-models/ ├── pom.xml ← 聚合 common、chat、embedding、rerank ├── snail-ai-model-common/ ← adapterKey、能力枚举、descriptor、异常 ├── snail-ai-model-chat/ ← Chat core、OpenAI-compatible provider、starter ├── snail-ai-model-embedding/ ← Embedding core、OpenAI-compatible provider、starter └── snail-ai-model-rerank/ ← Rerank core、Qwen HTTP provider、starter

这个目录是横向能力层：Server 侧 ServerModelFacade 会同时注入三条 runtime；Agent Client 侧当前主要通过 DefaultChatModelFactory 构建 ChatModel。也就是说，models 模块站在“配置”和“具体 SDK/HTTP 客户端”之间，向上给业务层一个稳定入口，向下隔离不同模型服务的构造细节。

3. 分层：common、core、provider、starter

ModelConfig / DispatchConfig
  ↓
SpecFactory 生成 ChatModelSpec / EmbeddingModelSpec / RerankModelSpec
  ↓
Runtime 校验 adapterKey、baseUrl、apiKey、modelKey
  ↓
Adapter.create(spec)
  ↓
ChatModel / EmbeddingModel / RerankApiClient

公共接口清单放在 模型适配层参考。这里先记住四层：common 定义能力、默认 adapter 和错误码；每条 core 定义 Spec、Adapter、Runtime；provider 把 spec 翻译成具体 SDK 或 HTTP 客户端；starter 用 Spring Boot 自动装配把 runtime、内置 adapter 和可展示的 ModelAdapterDescriptor 注册为 Bean。

4. 示例：Chat adapter 如何被选中

以 Server 侧 Chat 为例，调用从 ServerModelFacade 进入，ServerModelSpecFactory 先把 ModelConfigInfoDTO 转成 ChatModelSpec。如果配置没填 adapterKey，ModelAdapterDefaults.resolve 会把 Chat 默认成 openai-compatible。随后 ChatModelRuntime 在 Spring 注入的 adapter 列表里做大小写不敏感匹配，命中 OpenAiCompatibleChatModelAdapter 后才创建 OpenAiChatModel。

ServerModelFacade.buildChatModel(...)
  → specFactory.chatSpec(config, decryptedApiKey)
  → chatModelRuntime.build(spec)
  → adapters.filter(adapterKey == spec.adapterKey)
  → OpenAiCompatibleChatModelAdapter.create(spec)

这一层最关键的判断是：providerKey 会被放进 spec，但当前内置 adapter 的实际选择依据是 adapterKey。所以排查“为什么没有走某个模型实现”时，先看配置表或请求里的 adapterKey，再看 starter 是否注册了同名 Adapter Bean。

暂停回忆：不看上文，判断三个说法：Chat 和 Embedding 默认 adapterKey 相同吗？Rerank 返回的是 Spring AI 的 EmbeddingModel 吗？ModelAdapterDescriptor 负责真正创建模型客户端吗？
参考答案：Chat 和 Embedding 默认都是 openai-compatible；Rerank 返回的是 RerankApiClient；ModelAdapterDescriptor 只给管理端列出 adapter 元数据，真正创建对象的是对应 Adapter.create(spec)。如果你把 providerKey 当成当前内置实现的选择键，需要回到本课第 4 节重看。

主要推荐来源：先读 ChatModelRuntime.java，再对照 OpenAiCompatibleChatModelAdapter.java。后续遇到 RAG 中 embedding/rerank 的问题，可以继续向 agent 追问“这条调用最后进入哪个 runtime”。
