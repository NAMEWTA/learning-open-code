一条 RAG 请求如何选中存储后端

本课只练一个判断：看到一个知识库 ID，如何从 Admin 配置追到真实的 PgVector、Milvus、Elasticsearch 向量索引，或 Elasticsearch 全文索引。

1. 路由地图

StoreInstance 本身不存向量，也不存全文索引。它只是把后端连接参数放进 sai_store_instance.config。知识库创建时，KnowledgeService 把选中的 vectorStoreInstanceId、可选的 searchEngineInstanceId 和 dimensionOfVectorModel 写进 RagPO。摄入或检索发生时，工厂再按 ID 回查配置并创建后端。

sequenceDiagram
  actor Admin as Admin UI
  participant SIS as StoreInstanceService
  participant KS as KnowledgeService
  participant VDC as VectorDimensionConstraintService
  participant DP as DocumentPipeline / VectorSearchHandler
  participant VSF as VectorStoreFactory
  participant SEF as SearchEngineFactory
  participant ES as ElasticsearchSearchEngine

  Admin->>SIS: 保存 StoreInstance(category/type/config)
  SIS->>SIS: 校验分类与类型
  Admin->>KS: 创建知识库(vectorStoreInstanceId/searchEngine*)
  KS->>VDC: 校验 embedding 维度与向量后端上限
  DP->>VSF: create(RagPO) 路由到 PgVector/Milvus/ES Vector
  DP->>SEF: searchEngineEnable=true 时路由搜索实例
  SEF->>ES: insert/search indexName = "rag_" + ragId

容易误判：isDefault 只用于同分类默认标记和列表排序；当前 KnowledgeService 要求显式传入 vectorStoreInstanceId，运行时没有“未选则用默认实例”的 fallback。

2. 三段源码证据

第一段是 Admin 配置入库。StoreInstanceService.create 校验分类和类型，把表单里的 config 序列化为 JSON；如果设为默认，则先清掉同分类其他默认实例。

validateCategoryType(vo.getCategory(), vo.getType());
po.setCategory(vo.getCategory());
po.setType(vo.getType());
po.setConfig(JsonUtil.toJsonString(vo.getConfig() != null ? vo.getConfig() : Map.of()));
po.setIsDefault(Boolean.TRUE.equals(vo.getIsDefault()));
if (Boolean.TRUE.equals(po.getIsDefault())) {
    clearDefaultInCategory(po.getCategory(), null);
}

第二段是向量工厂。VectorStoreFactory.create 从 RagPO 取实例 ID、embedding 模型 ID 和维度，再查 StoreInstancePO，把业务枚举转成 VectorStoreType，最后从 REGISTER 里拿构造器。

StoreInstancePO inst = getStoreInstanceOrThrow(vectorStoreInstanceId);
StoreInstanceTypeEnum typeEnum = getStoreInstanceTypeOrThrow(inst);
ModelConfigInfoDTO modelConfig = getEmbeddingModelConfigOrThrow(embeddingModelId);
VectorStoreType vectorStoreType = VectorStoreType.valueOf(typeEnum.name());
Function<VectorStoreConfigDTO, SnailAiVectorStore> factory = REGISTER.get(vectorStoreType);
EmbeddingModel model = modelRuntimeHandler.buildEmbeddingModel(modelConfig);

第三段是 RAG 调用点。DocumentPipeline 和两个检索 handler 都用 IndexNameBuilder.KNOWLEDGE 生成同一个 indexName：前缀 rag_ 加知识库 ID。PgVector 把它当表名，Milvus 把它当 collection，Elasticsearch 把它当 index。

String indexName = IndexNameBuilder.KNOWLEDGE.build(Map.of("ragId", knowledge.getId()));
snailAiVectorStore.add(VectorAddRequest.builder()
        .indexName(indexName)
        .documents(toEmbed)
        .build());

3. 维度和错误路径

VectorDimensionConstraintService 会取 embedding 模型最大维度，再按后端类型限制：PG_VECTOR 为 2000，MILVUS 与 ELASTICSEARCH 为 4096，有效上限取两者较小值。知识库保存前，validateRequestedDimension 会拒绝空维度、小于 64 或超过任一上限的值。

向量写入失败会让 DocumentPipeline.processDocument 进入失败状态；搜索引擎写入失败只记录 warning，不中断文档成功。检索阶段如果 searchEngineEnable 不是 true，Bm25SearchHandler 直接跳过。

还有两个边界要分清：第一，KnowledgeService.validateKnowledgeStorage 只在 searchEngineInstanceId != null 时校验搜索实例，所以“启用搜索但没传 ID”可能保存成功；摄入时搜索双写异常会被当成 non-fatal warning，检索时 Bm25SearchHandler 会让 SearchEngineFactory 抛出“搜索引擎实例 ID 不能为空”。第二，向量侧 PgVector、Milvus、Elasticsearch Vector 的 client factory 会拦截 enabled=false，但 Elasticsearch Search 的 createClient 当前不会检查这个字段。

4. 检索练习

暂停回忆：一个知识库保存了 vectorStoreInstanceId=7、searchEngineEnable=true、searchEngineInstanceId=9。用户执行 POST /rag/search，你会查哪几处来确认后端？

参考答案：先查 sai_rag 的三个字段；再查 sai_store_instance.id=7 的 type 和 config，确认向量后端；如果启用 BM25，再查 id=9 的搜索实例；最后确认 IndexNameBuilder.KNOWLEDGE 生成的 rag_ 加知识库 ID 是否存在于对应后端。
