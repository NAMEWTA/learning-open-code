Agent 责任链到 gRPC 分发速查

这份参考用于排查一次 Agent 对话如何从 Server handler chain 变成 /chat/dispatch server-streaming 调用，并被 Agent Client streaming handler 消费。

1. 垂直切片总图

HTTP/SSE 入口AgentChatService 创建 AgentChatContext，复制 sid、禁用 MCP/Skill 和输入消息。
Handler ChainAgentChatChainService 顺序调用 Spring 注入的 AgentChatHandler 列表。
上下文汇总Handler 写入模型、MCP、Skill、RAG prompt、历史和目标 Client。
DTO 构造LlmCallHandler 把上下文组装成 ChatDispatchRequest JSON。
gRPC 信封GrpcChannelUtil 写入 GrpcSnailAiRequest(reqId, metadata, body)。
Client 消费ChatDispatchStreamingHandler 解析 body，调用 ChatSessionRuntime，流式返回结果。

4. URI、metadata 与 gRPC 方法

层级当前源码事实排查含义
proto 信封 GrpcSnailAiRequest 有 reqId、metadata、body；Metadata 有 uri 和 headers。 业务字段不在 proto message 中，而在 JSON body 中。
方法名 server-streaming 使用 ServerStreamingRequest/serverStreamingRequest。 多个业务 streaming URI 可以共用同一个 gRPC 方法。
业务 URI 聊天分发使用 UriConstants.CHAT_DISPATCH，值为 /chat/dispatch。 Client 侧 handler 是否命中，取决于 metadata URI。
metadata headers LlmCallHandler 调用 sendServerStreaming(..., Map.of(), ...)，聊天分发 headers 为空。 不要在 /chat/dispatch 上找 appId/token header；Client 在线身份来自此前心跳注册。
reqId GrpcChannelUtil 用进程内 AtomicLong 自增生成。 它只用于 gRPC 请求/响应匹配；不要和业务 requestId 混用。
CallOptions server-streaming 调用使用 CallOptions.DEFAULT。 桥接工具层没有统一 deadline；超时要继续看 channel、网络、上层请求生命周期和 Client runtime。

5. Client 在线、路由和心跳

步骤源码关键字段失败症状
Client 启动 SnailAiAgentAutoConfiguration 创建 ClientGrpcServer 并启动心跳任务。 snail-ai.agent.port、Server host/port、appId、token。 Client 没启动 gRPC server 时，Server 可有旧节点但 streaming 连接失败。
心跳发送 ClientHeartbeatScheduler 每 10 秒 unary 调 /beat。 headers: hostId、hostIp、hostPort、appId、token、version；body: maxConcurrentChats、activeChats。 token 错误会被 Server 拒绝；Server 端不会把该 Client 加入 alive 表。
心跳处理 BeatRequestHandler 校验 token 后调用 ClientInstanceManager.registerOrUpdate。 appId/token、hostId、hostIp、grpcPort。 app 停用或 token 不匹配时返回失败状态。
在线判断 ClientInstanceManager.getAliveInstances 40 秒心跳阈值、channel 非 shutdown、DB 未过期节点降级发现。 ContextCollectorHandler 报“没有可用的客户端实例”。
路由选择 ClientRouteStrategyManager + 默认 LeastLoadClientRoute activeChats / maxConcurrent 最小者优先。 maxConcurrent 小于等于 0 会被视为极高负载。

6. Streaming 生命周期

阶段发生位置事件结束条件
发起 Server LlmCallHandler 序列化 ChatDispatchRequest，调用 GrpcChannelUtil.sendServerStreaming。 方法调用返回后设置 streamDispatchStarted=true；这不代表已经收到第一个 chunk。
入站分发 Client ClientGrpcServer + ClientRequestDispatcher server-streaming service 收到统一请求，从 metadata 取 URI 并委托 GrpcRequestDispatcher。 找不到 streaming handler 时返回 Unknown URI 并 completed。
业务执行 Client ChatDispatchStreamingHandler + ChatSessionRuntime 解析 body，active chat +1，解析工具，执行流式模型调用。 completion 或 error 回调都会清理 ToolResolution 并 active chat -1。
增量回流 Client handler text/thinking 变成 ChatStreamResponse JSON，再放入 GrpcSnailAiResult.data。 发送失败只记录 warn，不会自动补偿。
完成回流 Client handler completion 包含 fullText、fullThinking、promptTokens、completionTokens、cacheTokens、durationMs。 发送 completion 后调用 observer.onCompleted()。
Server 回写 Server ChatStreamObserver 解析 ChatStreamResponse，text/thinking 写前端流；completion 提交异步持久化。 gRPC onCompleted 时调用 streamWriter.complete()。
错误回流 Client handler 或 gRPC observer Client runtime error 变成 type=error；网络/连接异常进入 Server observer 的 onError。 Client error 会 completed；Server observer onError 会写“Client disconnected”并 complete。

7. 错误、超时和不在线边界

现象优先检查依据下一步
前端立刻收到 Agent not found InitContextHandler 找不到 Agent 时直接写错误并终止。 核对 agentId、数据库和入口权限上下文。
前端立刻收到未配置对话模型或模型配置加载失败 ModelResolveHandler 默认 chat 模型不存在或 dispatch model config 装配异常。 核对 Agent chatModelId、默认模型、provider/adapter 配置和密钥解密。
前端收到 appId 或应用停用错误 ContextCollectorHandler.findEnabledApp 分发前必须有启用状态的 app。 核对 Agent appId、App 表 status、routeStrategy。
前端收到没有可用 Client ClientInstanceManager.getAliveInstances 40 秒内无心跳、channel shutdown、DB 无未过期节点都会导致候选为空。 看 Client 心跳日志、Server /beat 返回、token、host/port、DB client_node 过期时间。
已选中 Client，但 streaming 建连失败 LlmCallHandler catch 和 ChatStreamObserver.onError 同步发起异常会向外抛；异步连接或流异常由 observer onError 写前端错误。 核对 Client gRPC 端口、网络、channel 状态、防火墙和 Client server 是否启动。
Client 收到请求但返回 Unknown URI GrpcRequestDispatcher.dispatchStreaming metadata URI 未命中任何 GrpcStreamingRequestHandler.supports。 确认发送的是 UriConstants.CHAT_DISPATCH，Client 自动配置已创建 ChatDispatchStreamingHandler。
请求长时间无 completion ChatSessionRuntime、模型流、工具解析、网络 GrpcChannelUtil.sendServerStreaming 没有设置 deadline。 查看 Client activeChats 是否增长后未下降、模型 SDK 日志、工具调用是否阻塞、上层 HTTP/SSE 是否超时。
收到 completion 但数据库没有助手消息 ChatStreamObserver.handleCompletion 与 ChatResultPersistService completion 时只提交异步持久化命令。 看 persistService 异步任务日志、Agent/user/conversationId、token 字段和短期记忆写入。

8. 调试 checklist

检查点命令或源码位置期望信号
确认 Handler 是否注册并排序 chain/*Handler.java 的 @Component 与 @Order 10、20、30、40、50、60、70、75、80 按序存在；WebSearch 默认不注册。
确认上下文是否在 gRPC 前已终止 ctx.isTerminated() 检查点 若已终止，LlmCallHandler 不会发送 /chat/dispatch。
确认分发 DTO 是否完整 LlmCallHandler.buildDispatchRequest modelConfig、targetClient、systemPrompt、historyMessages、skills、mcpServers 符合预期。
确认业务 URI UriConstants.CHAT_DISPATCH metadata.uri 为 /chat/dispatch，headers 为空是当前聊天分发的正常状态。
确认 Client 在线来源 ClientHeartbeatScheduler 与 BeatRequestHandler 10 秒心跳成功；Server 注册 appId/hostId/hostIp/grpcPort；40 秒内未过期。
确认 Client streaming service 已启动 ClientGrpcServer.start 日志 Client gRPC server 监听配置端口，并注册 unary 与 server-streaming 服务。
确认 Client handler 命中 ChatDispatchStreamingHandler.supports 与日志 日志出现 requestId 和 sid；未命中时检查 URI 或自动配置条件 snail-ai.enabled=true。
确认回流类型 ChatStreamResponse.TYPE_TEXT/THINKING/COMPLETION/ERROR text/thinking 可多次，completion 或 error 结束流。
确认持久化触发点 ChatStreamObserver.handleCompletion 只有 completion 会提交助手消息、thinking 和 token 的异步保存。

阅读顺序建议：先从 ContextCollectorHandler 判断是否真的会走到 gRPC，再看 LlmCallHandler.buildDispatchRequest 的 DTO 完整性，然后检查 GrpcChannelUtil 的 metadata URI，最后到 Client 的 ChatDispatchStreamingHandler 和 ChatSessionRuntime 看流是否被消费并完成。
