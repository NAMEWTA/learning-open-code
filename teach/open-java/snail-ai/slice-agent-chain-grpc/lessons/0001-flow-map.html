责任链状态如何变成 gRPC streaming 调用

本课只完成一个目标：把 Server 端 AgentChatContext 的字段接力，追到 ChatDispatchRequest、gRPC 统一信封，再追到 Client 的 ChatDispatchStreamingHandler。

层级：L2 垂直切片 预计 15 分钟 主线：Server → gRPC → Client

1. 先把状态接力画出来

一次 Agent 对话不是直接调用模型。Server 端先让 Handler Chain 逐段填充 AgentChatContext：初始化用户和 Agent，保存用户消息，解析模型，收集 MCP、RAG、Skill、历史消息和目标 Client。LlmCallHandler 只在最后统一读取这些状态。

10-30user、agent、modelId、modelConfig
40-70mcpServers、systemPrompt、skills
75targetClient、historyMessages

2. DTO 边界在 LlmCallHandler

LlmCallHandler 是 Server 端责任链终点。它先校验 targetClient 和 modelConfig；缺失时直接写 [ERROR] 并结束前端流。校验通过后，它把上下文重新打包：Agent 元数据进 agentConfig，用户输入进 userMessage，模型、MCP、Skill、历史和 systemPrompt 原样成为 Client 可执行的请求负载。

open-java/snail-ai/.../feature/agent/chain/LlmCallHandler.java

ChatDispatchRequest request = ChatDispatchRequest.builder()
        .requestId(UUID.randomUUID().toString())
        .sid(ctx.getSid())
        .agentConfig(agentConfig)
        .conversationId(ctx.getConversationId())
        .userMessage(ctx.getContent())
        .modelConfig(ctx.getModelConfig())
        .mcpServers(ctx.getMcpServerDescriptors())
        .skills(ctx.getSkillDescriptors())
        .historyMessages(ctx.getHistoryMessages())
        .memoryContext(ctx.getMemoryContext())
        .systemPrompt(ctx.getSystemPrompt())
        .serverHttpPort(serverHttpPort)
        .build();

GrpcChannelUtil.sendServerStreaming(
        ctx.getTargetClient().getChannel(),
        UriConstants.CHAT_DISPATCH,
        JsonUtil.toJsonString(request),
        Map.of(),
        new ChatStreamObserver(ctx, chatResultPersistService, shortTermWindow));

这里要分清两层 DTO：ChatDispatchRequest 是 JSON 业务体；gRPC proto 只提供 GrpcSnailAiRequest(reqId, metadata, body)。业务路由不是 proto 方法名，而是 metadata.uri=/chat/dispatch。

3. Client streaming handler 消费它

Client 启动时注册 server-streaming service，ClientRequestDispatcher 从 metadata 取 URI，再交给共用的 GrpcRequestDispatcher。当 URI 命中 /chat/dispatch，ChatDispatchStreamingHandler 解析 body，调用 ChatSessionRuntime，并把 runtime 的四个回调翻译为 GrpcSnailAiResult.data 中的 ChatStreamResponse。

open-java/snail-ai/.../agent/core/grpc/handler/ChatDispatchStreamingHandler.java

ChatDispatchRequest dispatchRequest = JsonUtil.parseObject(body, ChatDispatchRequest.class);
chatSessionRuntime.execute(ChatSessionRequest.builder()
        .dispatchRequest(dispatchRequest)
        .textConsumer(text -> handleTextChunk(reqId, sid, text, observer))
        .thinkingConsumer(thinking -> handleThinkingChunk(reqId, sid, thinking, observer))
        .completionConsumer(done -> handleCompletion(reqId, requestId, sid, done, observer))
        .errorConsumer(error -> handleError(reqId, sid, error, observer))
        .build());

回程也分两层：Client 连续 onNext 业务 JSON；Server 端 ChatStreamObserver 再把 text/thinking 写给前端，把 completion 交给异步持久化，最后在 onCompleted 关闭前端流。

参考答案与反馈

没有走到 gRPC。这个错误由 ContextCollectorHandler 在 order 75 选择 Client 前后产生，LlmCallHandler 看到 ctx.terminated=true 会直接返回。优先查 ContextCollectorHandler 的 appId/app 状态/候选 Client 逻辑，再查 ClientInstanceManager 是否因心跳、token、40 秒过期阈值或 channel 状态导致没有 alive instance。
