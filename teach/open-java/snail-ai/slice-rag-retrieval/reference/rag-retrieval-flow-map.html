RAG 检索注入参考图谱

一句话：Snail AI 的 Agent RAG 有两条注入路径：FORCED 在 Server 端用用户消息预检索所有绑定知识库，并把结果追加到 systemPrompt；SMART 在 Server 端提示可用知识库，在 Client 端注册 rag_search 工具，模型按需回调 Server 执行同一套 RagSearchPipeline。

源码边界：QueryRewriteHandler 当前没有实现；RagStageMetricsDTO.embeddingMs 当前没有被显式设置；RagSearchTool 保存了构造时的 ragIds，但工具方法没有校验传入 ragId 是否属于该列表；FinalizeHandler 直接访问 modelParams.nearbySliceCount，空配置是需要重点检查的风险点。

真实源码片段

1. RagHandler.handle：强制/智能模式分流

先解析 Agent 绑定的知识库，再用 RagCallModeEnum.of 决定注入路径。强制模式立即检索，智能模式只追加可用知识库说明。

RagCallModeEnum callMode = RagCallModeEnum.of(ctx.getAgent().getRagCallMode());

if (callMode == RagCallModeEnum.FORCED) {
    // 强制调用：服务端直接并行检索所有知识库，结果注入 prompt
    String retrievedContent = parallelSearch(ragIds, ctx.getContent());
    ctx.setSystemPrompt(ctx.getSystemPrompt() + buildForcedPrompt(ragList, retrievedContent));
} else {
    // 智能调用：注入可用知识库信息，由 LLM 通过 rag_search tool 决定
    ctx.setSystemPrompt(ctx.getSystemPrompt() + buildSmartPrompt(ragList));
}

2. RagSearchService.search：创建检索上下文

request.query 先进入 originalQuery，pipeline 后续 handler 再把配置、召回结果、融合结果和指标写回同一个 RagSearchContext。

public RagSearchResponseDTO search(RagSearchRequestDTO request, RagConfigDO configDO) {
    RagSearchContext ctx = RagSearchContext.builder()
            .originalQuery(request.getQuery())
            .ragId(request.getRagId())
            .debug(Boolean.TRUE.equals(request.getDebug()))
            .externalConfigDO(configDO)
            .metrics(new RagStageMetricsDTO())
            .build();

    pipeline.execute(ctx);

    return RagSearchResponseDTO.builder()
            .results(ctx.getResults())
            .metrics(ctx.isDebug() ? ctx.getMetrics() : null)
            .build();
}

3. ClientRagToolResolver.resolve：注册 rag_search 工具

Client 端只在非强制模式注册 RagSearchTool。该类的 @Tool(name = "rag_search") 暴露工具名；工具方法参数是 ragId 和 queryQuestion，不是 query。

public List<ToolCallback> resolve(ChatDispatchRequest request) {
    ChatDispatchRequest.AgentConfig agentConfig = request.getAgentConfig();
    if (agentConfig == null
            || !Boolean.TRUE.equals(agentConfig.getRagEnabled())
            || StrUtil.isBlank(agentConfig.getRagIds())) {
        return List.of();
    }

    // 强制调用模式不注册 tool（服务端已预检索注入 prompt）
    if (RagCallModeEnum.FORCED.getMode() == agentConfig.getRagCallMode()) {
        return List.of();
    }

    List<Long> ragIds = Arrays.stream(agentConfig.getRagIds().split(","))
            .map(String::trim)
            .filter(StrUtil::isNotBlank)
            .map(Long::parseLong)
            .toList();

    log.info("RAG tool resolved (smart mode): ragIds={}", ragIds);
    return Arrays.asList(ToolCallbacks.from(
            new RagSearchTool(ragIds, rpcClient)
    ));
}

推荐阅读顺序

  1. 先读 Agent 责任链 与 RagHandler.java，确认 FORCED 与 SMART 的分流。
  2. 再读 RAG 模块、RagSearchService.java 和 search/pipeline/handler，把检索顺序画出来。
  3. 然后读 Agent Client、ClientRagToolResolver.java、RagSearchTool.java 和 RagSearchCallbackHandler.java，追智能调用工具闭环。
  4. 最后对照 持久化模块、模型模块、VectorStoreFactory.java、DefaultRerankService.java、ElasticsearchSearchEngine.java 排查存储和模型后端。
