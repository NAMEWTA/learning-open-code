Admin 智能体流式对话流程图

本课只完成一个目标：把 Admin 端一次智能体流式对话从 HTTP/SSE 入口追到 Agent Client 执行，再追回应答回写和持久化边界。

层级：L2 垂直切片 预计 15 分钟 主来源：AgentController.java LlmCallHandler.java ChatDispatchStreamingHandler.java

1. 请求先变成一条可写的流

Admin 对话入口是 POST /agent/{id}/chat，方法带 @LoginRequired(role = USER)，因此请求先经过 Admin 认证拦截器：读取 Snail-Ai-Auth，校验 JWT 和角色，把用户放进 UserSessionUtils。Controller 返回 Flux<String>，并用内部类 FluxBridgeChatStreamWriter 把责任链的 send/complete 适配成 SSE 输出。

请求体只要求 conversationId 和 content，可选禁用 MCP Server 和 Skill。Controller 不自己调用模型，只创建 AgentChatCommand，把 agentId、会话、内容、禁用项和 stream writer 交给 AgentChatService。

2. Server 责任链补齐分发上下文

AgentChatService 创建 AgentChatContext 后，AgentChatChainService 顺序调用 Spring 注入的 Handler。关键顺序是：10 初始化用户和 Agent，20 创建会话并保存用户消息，30 解析模型配置，40 装配 MCP，50 初始化 system prompt，60 处理 RAG，70 装配 Skill，75 选择目标 Client 并加载短期历史，80 组装 ChatDispatchRequest 并通过 gRPC 发往 Client。

Admin HTTP/SSE
  └─ AgentController.chat
      └─ AgentChatService.chat
          └─ AgentChatChainService.proceed
              ├─ InitContextHandler
              ├─ ConversationHandler
              ├─ ModelResolveHandler
              ├─ McpHandler / SystemPromptHandler / RagHandler / SkillAgentChatHandler
              ├─ ContextCollectorHandler
              └─ LlmCallHandler
                  └─ gRPC server streaming: /chat/dispatch

注意短路规则：循环本身不会停，失败的 Handler 会设置 ctx.terminated=true，后续 Handler 需要自己先检查。找不到 Agent、模型配置、app 或可用 Client 时，会直接向 stream writer 写错误并完成，不会进入 Client 执行，也不会产生 completion 持久化。

3. Client 执行后把 chunk 带回 Server

LlmCallHandler 使用 UriConstants.CHAT_DISPATCH，也就是 /chat/dispatch。Client 端本地 gRPC server 收到请求后，ClientRequestDispatcher 交给 ChatDispatchStreamingHandler。它解析 ChatDispatchRequest，调用 ChatSessionRuntime；运行时增加活跃计数，解析 MCP/RAG/Skill/基础/自定义工具，再让 ClientChatExecutor 构建 Prompt 和 Spring AI ChatClient。

流式模型响应中，StreamChunkForwarderAdvisor 回传文本 chunk，ThinkingCollectorAdvisor 回传 thinking，TokenUsageCollectorAdvisor 在最终 chunk 采集 usage。Client 发回 ChatStreamResponse：text 和 thinking 被 ChatStreamObserver 写回 SSE；completion 触发异步持久化，保存助手消息、thinking、token、短期记忆和每日统计。

4. 检索练习

暂停回忆：如果 Admin 页面已经连上 SSE，但只有用户消息入库，没有助手消息和 usage，你优先查哪三个点？

参考答案与反馈

先查 ContextCollectorHandler 是否选到可用 Client，再查 LlmCallHandler 是否成功发出 /chat/dispatch，最后查 Client 端 ChatDispatchStreamingHandler 是否返回 completion。用户消息在 ConversationHandler 已提前持久化；助手消息和 usage 只有 ChatStreamObserver.handleCompletion 收到 completion 后才会提交给 ChatResultPersistService。

主要推荐来源：先读本课对应的三个源文件，再打开 切片参考核对入口端点、DTO、Handler 顺序、gRPC URI、失败/取消/usage/持久化边界。遇到“到底该改 Server 还是 Client”的问题，可以继续向 agent 追问具体调用栈。
