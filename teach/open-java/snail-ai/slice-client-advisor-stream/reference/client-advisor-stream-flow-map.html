Client Advisor Stream Flow Map

这是 L2-slice-client-advisor-stream 的速查图谱。它以当前源码为准，追踪 Server 发起 /chat/dispatch 后，Client 侧如何把模型流转换为 ChatStreamResponse 并通过 gRPC observer 回写 Server。

入口：ChatDispatchStreamingHandler 状态：ClientStreamExecutionContext 响应：ChatStreamResponse 清理：ToolResolution.close() 计数：ActiveChatCounter

横向模块背景见 ../module-agent-client/reference/agent-client-overview.html。L3 API 参考暂待后续覆盖率扫描生成；本页先保持 L2 切片图谱和源码锚点。

调用时序图

sequenceDiagram participant Server as Server LlmCallHandler participant Handler as Client ChatDispatchStreamingHandler participant Runtime as ChatSessionRuntime participant Executor as ClientChatExecutor participant Advisor as Stream Advisors participant Observer as gRPC StreamObserver Server->>Handler: /chat/dispatch body Handler->>Handler: parse body, read requestId/sid Handler->>Runtime: execute(ChatSessionRequest) Runtime->>Runtime: ActiveChatCounter.increment() Runtime->>Executor: executeStream(...) Executor->>Advisor: ChatClient.stream().chatResponse() Advisor-->>Observer: text / thinking chunks Advisor-->>Executor: complete signal Executor->>Runtime: acceptCompletion(state.toCompletionResult()) Runtime->>Handler: completion consumer Handler->>Observer: onNext(ChatStreamResponse.completion) Handler->>Observer: onCompleted() Runtime->>Runtime: cleanup + ActiveChatCounter.decrement() Observer-->>Server: completion event then stream end
Server
LlmCallHandler 构造 ChatDispatchRequest，通过 GrpcChannelUtil.sendServerStreaming() 调用 UriConstants.CHAT_DISPATCH。
ChatStreamObserver.onNext() 解析 Client 回写的 JSON，根据 type 分发到 text、thinking、completion、error。
Client gRPC
ClientRequestDispatcher.streamingHandler() 把请求交给 GrpcRequestDispatcher.dispatchStreaming()。
ChatDispatchStreamingHandler.supports() 匹配 /chat/dispatch，handle() 先解析 body 并读取 requestId/sid，再注册四个 consumer；解析阶段外层没有 try/catch 和 ChatStreamResponse.error() 包装。
Runtime
ChatSessionRuntime.execute() 先 activeChatCounter.increment()，再调用 toolRuntime.resolve()。
completion/error consumer 外层使用 finally 调用 cleanup()，执行 activeChatCounter.decrement() 和 ToolResolution.close()。
Executor
ClientChatExecutor.executeStream() 创建 ClientStreamExecutionContext，订阅 streamResponses()。
streamResponses() 构造 ChatClient 与 Prompt，把 DISPATCH、STREAM_STATE、CHUNK_CONSUMER、THINKING_CONSUMER 写入 Advisor context。
ChatClient
DefaultChatClientFactory.build() 用 ChatModelFactory 生成模型，追加默认 Advisor、ToolCallingAdvisor、tools 和 tool context。
DefaultPromptFactory.build() 依次放入 system prompt、history messages 和 user message。
Advisor 链
InterceptorChainAdvisor 执行 SnailAiInterceptorChain 的 before/after。
TokenUsageCollectorAdvisor 从 chunk metadata 的 Usage 写入 promptTokens、completionTokens、cacheTokens。
ThinkingCollectorAdvisor 从 metadata 或 additional properties 提取 thinking，追加到 thinkingText 并回调。
StreamChunkForwarderAdvisor 从 AssistantMessage.getText() 取 text，追加到 fullText 并回调。
回写
handleTextChunk() 和 handleThinkingChunk() 发送 status 1 的 GrpcSnailAiResult，发送失败只 log warn，不结束也不中断模型流。
handleCompletion() 在同一个 try 内先 observer.onNext()，再 observer.onCompleted()；如果发送 completion 失败，只 log warn，不会继续进入 onCompleted()。
handleError() 发送 status 0 和 LLM_FAILED，随后调用 observer.onCompleted()；error 响应发送失败时同样只记录 warn。

chatSessionRuntime.execute(ChatSessionRequest.builder()
    .dispatchRequest(dispatchRequest)
    .textConsumer(text -> handleTextChunk(reqId, sid, text, observer))
    .thinkingConsumer(thinking -> handleThinkingChunk(reqId, sid, thinking, observer))
    .completionConsumer(completion -> handleCompletion(reqId, requestId, sid, completion, observer))
    .errorConsumer(error -> handleError(reqId, sid, error, observer))
    .build());

调试 checklist

text chunk

  1. 确认 ClientChatExecutor.streamResponses() 已传入 CHUNK_CONSUMER。
  2. 确认 StreamChunkForwarderAdvisor.extractText() 能从 ChatClientResponse 拿到文本。
  3. 确认 ChatDispatchStreamingHandler.handleTextChunk() 写入 status 1 和 JSON data。
  4. 确认 Server ChatStreamObserver.handleText() 收到同一 sid。

thinking

  1. 确认模型确实返回推理字段，不要只看普通 text。
  2. 优先检查 metadata key：reasoningContent、thinking、reasoning。
  3. 兼容路径检查 chunkChoice、message、_additionalProperties；reasoning_content 只属于这些 additional properties 路径。
  4. 确认 THINKING_CONSUMER 进入 handler 的 handleThinkingChunk()。

TokenUsage

  1. 确认最终 chunk 的 ChatResponse.metadata.usage 非空。
  2. 确认 totalTokens 大于 0，否则源码不会写入 state。
  3. 确认 OpenAI-compatible 流式 usage 已按模型适配器要求开启。
  4. cache token 只在 native usage 暴露 promptTokensDetails.cachedTokens 时可提取。

completion

  1. 确认 Flux 进入 complete 分支，而不是 error 分支。
  2. 确认 state.toCompletionResult() 输出 fullText、fullThinking 和 duration。
  3. 确认 handler 发送 ChatStreamResponse.completion() 后调用 observer.onCompleted()；两者在同一个 try 内，发送失败不会继续 completed。
  4. 确认 Server handleCompletion() 进入异步持久化。

error

  1. 请求 body 解析异常发生在 ChatSessionRuntime 前，不会被 ChatStreamResponse.error() 包装。
  2. 启动前异常会被 ChatSessionRuntime 的 catch 捕获并回调 error。
  3. 订阅中异常会进入 ClientChatExecutor.subscribe() 的 error consumer。
  4. handler 用 LLM_FAILED 和 status 0 写回，再 completed；error 响应发送失败只记录 warn。
  5. Server handleError() 只发送错误文本，不执行 completion 持久化。

active count 和 cleanup

  1. 每次 ChatSessionRuntime.execute() 入口都会 increment。
  2. completion/error 的 finally 都会调用 cleanup。
  3. 启动异常也会 cleanup 已解析出的 ToolResolution。
  4. ActiveChatCounter.get() 通过 PingRequestHandler 的 activeChats 和 ClientHeartbeatScheduler 的心跳 body 暴露。
  5. ToolResolution.close() 当前主要关闭 ClientMcpToolResolver。

最小排障路径

现象 先看 再看 常见结论
前端完全无输出 ChatDispatchStreamingHandler.handle() 是否收到请求并成功解析 body ClientChatExecutor.streamResponses() 是否订阅并进入模型流 请求未路由、body 解析失败、模型创建失败或订阅前异常
有 text，无 completion Flux 是否 complete handleCompletion() 是否 onCompleted() 模型流未结束、订阅异常或 observer 回写失败
completion 有 fullText，usage 为 0 TokenUsageCollectorAdvisor.extractUsage() 模型最终 chunk 是否包含 Usage 模型适配器未返回 streaming usage
thinking 不展示但模型支持 ThinkingCollectorAdvisor.extractThinkingText() metadata key 是否在兼容列表内 字段名不匹配或模型把 thinking 混入 text
active count 持续升高 ChatSessionRuntime.cleanup() PingRequestHandler 或 ClientHeartbeatScheduler 暴露的 activeChats 流没有终态、回调链被阻断，或 observer 终态发送失败
