Client Advisor 流式调用地图

本课只解决一个问题：当 Server 调用 Client 的 /chat/dispatch 后，模型流如何被拆成 text、thinking、completion、usage 和 error，再通过 gRPC observer 回写。

一图定位

1. gRPC 入口ChatDispatchStreamingHandler解析 ChatDispatchRequest，把四个回调交给 runtime。
2. 会话运行时ChatSessionRuntimeactive count 加一，解析工具，包住 completion/error cleanup。
3. 执行器ClientChatExecutor创建 ClientStreamExecutionContext，注入 Advisor 参数并订阅流。
4. ChatClientDefaultChatClientFactory装配模型、默认 Advisor、ToolCallingAdvisor 和 tools。
5. Advisor 链StreamAdvisor收集 usage/thinking，转发 text chunk，累积完整结果。
6. 回写 ServerStreamObserver发送 GrpcSnailAiResult，completion/error 后结束流。

核心分工是：handler 不碰模型，只负责把回调转成 gRPC 响应；runtime 保护会话资源；executor 只发起模型流；Advisor 链才是 text、thinking、usage 的处理位置。

三段源码

ChatDispatchStreamingHandler 在第 45-51 行把业务事件绑定到 observer 回写方法：

chatSessionRuntime.execute(ChatSessionRequest.builder()
    .dispatchRequest(dispatchRequest)
    .textConsumer(text -> handleTextChunk(reqId, sid, text, observer))
    .thinkingConsumer(thinking -> handleThinkingChunk(reqId, sid, thinking, observer))
    .completionConsumer(completion -> handleCompletion(reqId, requestId, sid, completion, observer))
    .errorConsumer(error -> handleError(reqId, sid, error, observer))
    .build());

ChatSessionRuntime 在 completion/error 外层加 finally，因此成功和失败都会 decrement 并关闭 ToolResolution：

.completionConsumer(completion -> {
    try {
        request.acceptCompletion(completion);
    } finally {
        cleanup(currentResolution);
    }
})
.errorConsumer(error -> {
    try {
        request.acceptError(error);
    } finally {
        cleanup(currentResolution);
    }
})

ClientChatExecutor 在第 75-80 行把 STREAM_STATE、CHUNK_CONSUMER、THINKING_CONSUMER 放入 Advisor context。随后 StreamChunkForwarderAdvisor 追加 fullText 并调用 chunk consumer，ThinkingCollectorAdvisor 追加 thinkingText，TokenUsageCollectorAdvisor 写入 token 字段。流完成时 state.toCompletionResult() 生成 completion。

return chatClient.prompt(prompt).advisors(a -> a
    .param(ClientAdvisorKeys.DISPATCH, dispatchRequest)
    .param(ClientAdvisorKeys.STREAM_STATE, state)
    .param(ClientAdvisorKeys.CHUNK_CONSUMER, safeConsumer(request.getMessageDeltaConsumer()))
    .param(ClientAdvisorKeys.THINKING_CONSUMER, safeConsumer(request.getThinkingDeltaConsumer())))
    .stream().chatResponse();

调试切入点

  • 没有 text：看 StreamChunkForwarderAdvisor.extractText() 是否拿到 AssistantMessage.getText()，再看 handleTextChunk() 是否调用 observer.onNext()。
  • 没有 thinking：先看直接 metadata key 是否包含 reasoningContent、thinking、reasoning；reasoning_content 只在 _additionalProperties 等兼容路径中提取。
  • usage 为 0：看最终 chunk 是否带 Usage；源码注释提示 OpenAI-compatible 流式 usage 常依赖 streamUsage(true)。
  • active count 不回落：看流是否进入 completion/error；cleanup 只在这两个回调或启动异常中触发。
  • Server 没收尾：看 handler 的 completion/error 是否调用 observer.onCompleted()，以及 Server ChatStreamObserver 是否收到 completion。

检索练习

暂停回忆：如果用户看到前端有零散 text，但最终对话没有持久化完整内容，你会优先检查哪三个点？

参考答案：先看 ClientStreamExecutionContext.fullText 是否由 StreamChunkForwarderAdvisor 累积；再看 ClientChatExecutor 的 complete 回调是否执行 state.toCompletionResult()；最后看 ChatDispatchStreamingHandler.handleCompletion() 和 Server ChatStreamObserver.handleCompletion() 是否收到并处理 fullText。
