异常边界：哪些错误会回写，哪些只记录日志

本课只练一个判断：同样是失败，哪些会变成 ChatStreamResponse.error()，哪些只会被吞掉并留下 warn 日志。

先分两道门

第一道门在 ChatDispatchStreamingHandler.handle() 的开头：它先解析 body，并立即读取 requestId 与 sid。这段代码外面没有 try/catch，也没有包装成 ChatStreamResponse.error()。所以 body 解析失败或字段对象异常时，请求还没进入 ChatSessionRuntime。

第二道门在 ChatSessionRuntime.execute() 内部。进入 runtime 之后，工具解析、执行器启动异常、订阅中的 error 都会走 request.acceptError()，并在 completion/error 的 finally 或启动异常 catch 中执行 cleanup。

源码边界

try {
    resolution = toolRuntime.resolve(request.getDispatchRequest());
    chatExecutor.executeStream(ClientChatExecutionRequest.builder()
        .completionConsumer(completion -> {
            try {
                request.acceptCompletion(completion);
            } finally {
                cleanup(currentResolution);
            }
        })
        .errorConsumer(error -> {
            try {
                request.acceptError(error);
            } finally {
                cleanup(currentResolution);
            }
        })
        .build());
} catch (Exception e) {
    cleanup(resolution);
    request.acceptError(e);
}

这段说明了 runtime 内的异常会触发 error consumer。相反，handler 开头的 parseDispatchRequest() 不在这个 try 里，所以它不是同一类边界。

四种现象速判

现象源码结论排查入口
body 解析失败未进入 runtime，无 ChatStreamResponse.error() 包装。ChatDispatchStreamingHandler.handle() 第 39-41 行。
模型或工具启动失败runtime catch 后 cleanup，再调用 error consumer。ChatSessionRuntime.execute() catch 分支。
text/thinking chunk 发送失败handleTextChunk()、handleThinkingChunk() 只 log warn，不抛回 Reactor，流会继续。handler 第 58-83 行。
completion 发送失败onNext() 与 onCompleted() 在同一个 try 内；发送失败只 log，不会继续调用 onCompleted()。handleCompletion() 第 86-111 行。

error 回写本身也遵循同类边界：handleError() 内部先 onNext()，再 onCompleted()，如果发送 error 响应失败，同样只记录 warn。

检索练习

判断题：前端没有收到错误消息，就能说明模型调用没有失败吗？

参考答案：不能。请求 body 解析异常发生在 runtime 前，不会被包装成 ChatStreamResponse.error()；completion/error 的 observer 发送失败也可能只留下 warn，且不会完成 gRPC 流。只有进入 runtime 后并成功写回 error，Server 才能收到标准错误事件。
