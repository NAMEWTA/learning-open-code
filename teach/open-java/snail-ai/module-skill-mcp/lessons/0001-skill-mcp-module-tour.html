Skill 与 MCP 能力模块导览

本课只完成一个目标：把 Skill 和 MCP Server 从 Admin 配置、Agent 绑定、Server 责任链描述符，到 Agent Client 工具 resolver 的执行边界串成一条线。

层级：L1 模块总览 预计 12 分钟 前置：Agent 责任链与 Client 执行层

1. 先分清两类能力

Skill 是可复用的指令包，Admin 端通过在线创建或 ZIP 上传写入 sai_skill 和 sai_skill_file；Agent 只保存与 Skill 的关联。MCP Server 是外部工具服务配置，Admin 端写入 sai_mcp_server，再用 sai_agent_mcp_server 绑定到 Agent。二者都不是直接在 Admin 执行，而是在一次对话中被 Server 责任链转成 ChatDispatchRequest 描述符，交给 Agent Client 的工具解析器。

skill-mcp 主线源码 ├── admin/ ← Skill、MCP、Agent 配置入口 ├── feature-agent/chain/ ← McpHandler、SkillAgentChatHandler、LlmCallHandler ├── feature-skill/handle/ ← Skill 查询与 read_skill 回调 ├── commons/dto/agent/ ← ChatDispatchRequest 与 Skill 内容 DTO └── agent/core/resolver/ ← Client 侧 ToolCallback 边界

2. Admin 只负责配置和绑定

SkillController 管理上传、在线创建、文件树、内容保存和下载；McpServerController 管理 MCP Server CRUD 与连接测试。Agent 的创建和更新请求带有 skillIds、mcpServerIds、skillEnabled、mcpEnabled，AgentService 会分别调用两个服务维护关联表。

open-java/snail-ai/snail-ai-server/snail-ai-server-admin/src/main/java/com/aizuda/snail/ai/admin/service/agent/AgentService.java

if (request.getMcpServerIds() != null) {
    mcpServerService.updateAgentMcpServers(id, request.getMcpServerIds());
}

if (request.getSkillIds() != null) {
    skillService.updateAgentSkills(id, request.getSkillIds());
}

对话时还有一个轻量开关：AgentChatRequestVO 可带 disabledMcpServerIds 和 disabledSkillIds。这不会改数据库绑定，只影响当前请求中哪些描述符被过滤。

3. Server 把配置变成分发合同

责任链当前源码顺序里，McpHandler 是 @Order(40)，SkillAgentChatHandler 是 @Order(70)，LlmCallHandler 是 @Order(80)。MCP Handler 查询 Agent 绑定的 MCP Server，过滤本次禁用 ID，然后把连接参数转成 McpServerDescriptor。Skill Handler 查询 Agent 绑定的 Skill，过滤禁用 ID，然后生成 SkillDescriptor。最后 LlmCallHandler 把两组描述符放进同一个 ChatDispatchRequest。

open-java/snail-ai/snail-ai-commons/snail-ai-commons-core/src/main/java/com/aizuda/snail/ai/common/dto/agent/ChatDispatchRequest.java

private List<McpServerDescriptor> mcpServers;
private List<SkillDescriptor> skills;

public static class McpServerDescriptor { /* id, transport, endpoint, command... */ }
public static class SkillDescriptor { /* id, name, description, version, skillPrompt */ }

源码优先：SkillAgentChatHandler 构造 SkillDescriptor 时仍调用 skillPrompt(s.getSkillContent())，DTO 会携带正文；但当前 ClientSkillToolResolver 不直接把这个字段拼成 prompt 正文，只把 Skill 名称、描述和临时目录列表追加到 system prompt，并注册 read_skill。模型需要完整指令时，ReadSkillTool 再通过 RPC 回调 Server 重新拉取内容。

4. Client resolver 才进入工具执行边界

Agent Client 接到 /chat/dispatch 后，ChatSessionRuntime 先调用 ToolRuntime.resolve。这里会解析 MCP 工具、基础工具、RAG 工具、Skill 工具和本地 @Tool Bean，随后 DefaultChatClientFactory 通过 defaultTools 交给 Spring AI 的 ToolCallingAdvisor。

open-java/snail-ai/snail-ai-agent/snail-ai-agent-executor/snail-ai-agent-executor-core/src/main/java/com/aizuda/snail/ai/agent/core/runtime/tool/ToolRuntime.java

tools.addAll(resolveMcpTools(dispatchRequest.getMcpServers(), mcpResolver));
addResolvedTools(tools, "base", () -> baseToolResolver.resolve(dispatchRequest));
addResolvedTools(tools, "RAG", () -> ragToolResolver.resolve(dispatchRequest));
addResolvedTools(tools, "Skill", () -> skillToolResolver.resolve(dispatchRequest));
tools.addAll(Arrays.asList(customToolCallbackProvider.getToolCallbacks()));

MCP resolver 负责按 Streamable HTTP 或 Stdio 建立 McpSyncClient，发现远端工具并转为 ToolCallback；SSE 在当前代码中被跳过。Skill resolver 只注册 read_skill，不消费描述符里的 skillPrompt 作为正文提示词；完整内容由 ReadSkillTool 通过 RpcClient.fetchSkillContent 回调 Server 的 SkillContentCallbackHandler 重新获取，再缓存并异步物化支撑文件。一次会话 completion 或 error 后，ToolResolution.close() 会关闭 MCP 连接。

暂停回忆：用户说“这个 Agent 绑定的 Skill 没有被模型使用”，你会先查 Admin 的 SkillController，Server 的 SkillAgentChatHandler，还是 Client 的 ClientSkillToolResolver？请按顺序说出三个检查点。

参考答案与反馈

先查 Agent 是否 skillEnabled=true 且 skillIds 已写入关联表；再查 SkillAgentChatHandler 是否加载到 Skill、是否被 disabledSkillIds 过滤；最后查 ClientSkillToolResolver 是否追加 Skill 列表并注册 read_skill。如果模型看到了列表但没读内容，问题多半在模型选择工具或 ReadSkillTool 回调 Server 的链路。

主要阅读来源：ToolRuntime、ClientSkillToolResolver、ClientMcpToolResolver。读完后打开 模块地图，用表格核对 Admin、DTO、Handler、resolver 和易错点。遇到禁用 ID、MCP 认证或 Skill 支撑文件物化不清楚的地方，可以继续向 agent 追问具体调用栈。
