模型 L1 Client L1 RAG L1 模型运行时速查

模型运行时垂直切片流程图

本课只完成一个目标：能从模型配置追到三类运行时调用，并判断错误或 usage 会在哪个出口出现。

预计 15 分钟层级：L2 垂直切片主来源：ModelRuntimeHandler 与 ModelResolveHandler

1. 两条主路径

Snail AI 的模型运行不是单一路径。Agent 对话先在 Server 解析 CHAT 配置，再把已解密的 apiKey 下发给 Client；RAG QA、向量库和重排则大多留在 Server 本地，通过 ModelRuntimeHandler 构建 runtime。

Agent Chat 路径
AgentChatContext
  -> ModelResolveHandler
  -> ModelConfigHandler.getDefaultModelByType(CHAT) 或 agent.chatModelId
  -> DispatchModelConfigAssembler
  -> ChatDispatchRequest.ModelConfig(apiEndpoint, apiKey, modelKey, adapterKey)
  -> LlmCallHandler
  -> Client DefaultChatClientFactory
  -> ClientChatModelSpecFactory
  -> ChatModelRuntime
  -> ChatClient.stream()
Server Runtime 路径
RagQAService.qaStream -> doQaStream -> ModelRuntimeHandler.chatStream
VectorStoreFactory / DefaultRerankService
  -> ModelRuntimeHandler
  -> ModelConfigHandler.getConfigInfo(modelConfigId)
  -> ModelConfigHandler.decryptApiKey(encryptedApiKey)
  -> ChatClientBuilder / EmbeddingClientBuilder / RerankClientBuilder
  -> ServerModelFacade
  -> ChatModelRuntime / EmbeddingModelRuntime / RerankModelRuntime
  -> provider adapter 调用

2. 配置先变成 spec

真正决定 provider 实现的不是 providerKey，而是 adapterKey。如果配置没有填，ModelAdapterDefaults.resolve 会让 CHAT 和 EMBEDDING 默认走 openai-compatible，让 RERANKER 默认走 qwen-rerank。进入 runtime 前，ServerModelSpecFactory 会校验 providerKey、apiEndpoint、modelKey 和解密后的 apiKey。

ModelConfigInfoDTO
  providerKey + adapterKey + apiEndpoint + modelKey + encryptedApiKey + configJson
        |
        | decryptApiKey()
        v
ChatModelSpec / EmbeddingModelSpec / RerankModelSpec
  adapterKey + providerKey + baseUrl + apiKey + modelKey + configJson
        |
        v
Runtime.validate()
        |
        v
Adapter.create(spec)

排查第一问：如果报“找不到 adapter”，优先查 adapterKey 和对应 starter 是否注册了同名 adapter bean；如果报“模型配置加载失败”，优先查 Server 侧配置字段和 API Key 解密。

3. 三类模型的出口不同

ChatModelRuntime 最终产出 ChatModel，再被 ChatClient 调用。Client 流式路径中，TokenUsageCollectorAdvisor 从最终 chunk 提取 promptTokens、completionTokens 和 cacheTokens，completion 回到 Server 后由 ChatStreamObserver 持久化。

EmbeddingModelRuntime 产出 EmbeddingModel。ModelRuntimeHandler.embedBatch 调用后把 Spring AI 响应转换为 SnailEmbeddingResponse，当前主要填 vectors、dimensions 和 costTimeMs。RAG 维度探测失败时，EmbeddingModelDimensionService 会使用默认维度 1024。

RerankModelRuntime 产出 RerankApiClient。QwenRerankModelAdapter 通过 HTTP 返回 index 与 score；响应为空、含 error 或无结果时返回空列表，DefaultRerankService 会降级为按原 score 排序。

4. 检索练习

暂停回忆：不看上文，判断三个说法是否正确。

1. Client 侧会再次调用 decryptApiKey。2. providerKey 是当前内置 runtime 选择 adapter 的主键。3. RAG rerank 调用失败后会直接中断搜索。

参考答案与反馈

三个都不正确。Client 消费的是 Server 下发的 apiKey；adapter 选择依据是 adapterKey；RAG rerank 对空结果和异常都会降级为按原 score 排序。主要阅读来源是 ModelResolveHandler.java、DispatchModelConfigAssembler.java、ModelRuntimeHandler.java 和 DefaultRerankService.java。下一步可打开 速查参考，按字段表和 checklist 复盘一次真实配置。

遇到不清楚的错误栈，可以继续向 agent 追问：“这个 modelConfigId 最后进入了哪一个 runtime？”
