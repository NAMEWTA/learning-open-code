1. 先记住主路径

OpenAPI chat 不是一套独立模型调用逻辑。它只是在 Server 入口层把外部用户、会话、内容和禁用能力列表转换为内部 Agent 对话命令，随后复用 Agent 责任链。默认部署路径会叠加 application.yml 的 /snail-ai context path，所以外部流式接口通常是 POST /snail-ai/openapi/v1/agent/chat，同步接口是 POST /snail-ai/openapi/v1/agent/chat/sync。

OpenAPI HTTP
  ├─ OpenApiAuthInterceptor
  │   └─ 校验 Snail-Ai-App-Id / Snail-Ai-Token，写入 OpenApiSessionUtils
  ├─ OpenApiChatController
  │   ├─ chat      → Flux<ServerSentEvent<String>>
  │   └─ chatSync  → Result<OpenApiChatSyncResponse>
  ├─ OpenApiChatService
  │   └─ OpenApiChatRequest → AgentChatCommand
  ├─ AgentChatChainService
  │   └─ Init → Conversation → Model → MCP → Prompt → RAG → Skill → Context → LLM
  ├─ LlmCallHandler
  │   └─ ChatDispatchRequest → gRPC /chat/dispatch
  └─ Agent Client
      └─ ChatDispatchStreamingHandler → ChatSessionRuntime → ClientChatExecutor

2. 转换点决定能追到哪里

鉴权阶段只证明调用方应用有效；真正的用户身份在 OpenApiChatService#resolveRequestUser 中由 appId + openId 解析成平台 UserPO。随后服务层构造 AgentChatCommand：agentId、conversationId、content、disabledMcpServerIds、disabledSkillIds、requestUser、openId、sid 会进入内部链路；sid 缺省时由服务端生成 UUID。

入口路由和字段转换的短源码片段见 Controller 路由片段 与 Service 命令构造片段。

这里有一个排查时很重要的边界：OpenApiChatRequest 目前还有 deepPlanEnabled、webSearchEnabled、timeout 字段，但当前转换没有把它们写入 AgentChatCommand。如果调用方说“我传了 timeout 但同步仍等 300 秒”，源码答案是同步路径固定使用 SYNC_CHAT_TIMEOUT_MS = 300_000L。

3. 响应分成两种出口

流式出口使用 FluxBridgeChatStreamWriter。Client 每个文本 chunk 会先变成 ChatStreamResponse.text(sid, text)，Server 再转为 OpenAPI 的 event: text，data 仍是那段 JSON 字符串；thinking 同理。Client 的 completion chunk 不直接透传给 OpenAPI，ChatStreamObserver 用它做持久化，gRPC 完成后 writer 的 complete() 触发 OpenAPI event: done，其中 data 只包含 conversationId 和累计 fullText。

同步出口使用 CollectingChatStreamWriter，它阻塞等待 complete()，只聚合 text chunk 的 content，最后返回 Result.ok(OpenApiChatSyncResponse)。因此 usage、thinking、cacheTokens 都不在同步响应里；usage 的真正链路在 Client 的 TokenUsageCollectorAdvisor，随后经 completion chunk 进入 ChatStreamObserver#handleCompletion 持久化。
