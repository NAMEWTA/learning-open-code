Snail AI 项目总览与学习地图

这一课只做一件事：建立 Snail AI 的全局地图。学完后，你应该能判断一个功能属于 Server 编排、Agent Client 执行、模型适配、RAG、记忆还是持久化。

学习目标：识别项目分层 预计 12 分钟 主来源：open-java/snail-ai/README.md open-java/snail-ai/pom.xml open-java/snail-ai/docs/architecture/overview.md

1. 一句话定位

Snail AI 是一个基于 Java 21、Spring Boot 4.1.0 和 Spring AI 2.0.0 的企业级 AI 智能体平台。它不是单个聊天 Demo，而是把多模型管理、Agent 编排、RAG 知识库、短期与长期记忆、MCP/Skill 扩展、OpenAPI 和管理后台放进同一套 Server-Agent 架构中。理解它的第一步，是把“请求编排”和“模型执行”分开：Server 负责鉴权、配置、上下文、RAG、记忆和分发；Agent Client 通过 gRPC 接任务，实际调用模型和工具。

2. 技术栈骨架

根 POM 是最可信的版本锚点：Java 21、Spring Boot 4.1.0、Spring AI 2.0.0、MyBatis-Plus 3.5.16、gRPC 1.76.0、Protobuf 3.25.8。存储上，业务库支持 MySQL、PostgreSQL 和达梦脚本；向量后端覆盖 PgVector、Milvus、Elasticsearch；搜索侧有 Elasticsearch；资源文件可走本地目录或 MinIO。部署方式分两层看：依赖服务由 docs/docker/docker-compose.yaml 编排；应用本体可用本地 Maven/Spring Boot 启动、Docker Compose/镜像部署，或打包 JAR 后交给 systemd 等进程管理器运行。文档站使用 VitePress，后台管理界面在 starter 中以已构建静态资源交付，后续源码学习不把这些构建产物当作核心 Java 源码。

3. 目录地图

snail-ai/ ├── README.md ← 项目定位、特性和公开入口 ├── pom.xml ← 5 个顶层 Maven 模块与版本管理 ├── docs/ ← 架构、指南、API、部署和 SQL 文档 ├── snail-ai-commons/ ← 2 个共享基础与 gRPC 子模块 ├── snail-ai-models/ ← Chat、Embedding、Rerank 模型适配 ├── snail-ai-server/ ← Admin、OpenAPI、Features、Persistence ├── snail-ai-agent/ ← Client 执行、OpenAPI SDK、Chat API ├── snail-ai-starter/ ← Spring Boot 启动入口和静态后台 └── snail-ai-agent-example/ ← 独立示例工程，不在根 reactor modules

根 reactor modules 只有 snail-ai-commons、snail-ai-models、snail-ai-server、snail-ai-agent、snail-ai-starter；snail-ai-agent-example/ 是独立示例工程，不随根 POM 一起构建。读源码时优先沿“入口到落地”走，而不是按文件名顺序扫。对话请求通常从 Admin 或 OpenAPI Controller 进入 Server，经过 Agent 责任链补齐上下文，再由 gRPC 分发到 Client 端。RAG、记忆、模型和存储不是孤立模块，它们是在这条链路上被选择性激活的能力。

4. 后续阅读顺序

建议先读 总览参考 中的 L1/L2 候选清单，再选择一个切片深入。第一批最值得讲的是 Server 责任链、RAG 检索、记忆注入、Agent Client Advisor、模型适配和持久化后端。每个后续主题都应回到本课的分层图：先问“入口在哪里”，再问“上下文由谁补齐”，最后问“数据或模型调用落到哪里”。

检索练习：不看上面的目录树，回答三个问题：RAG 检索主要在哪个顶层模块？gRPC 协议在哪个共享模块？Spring Boot 启动入口在哪里？
参考答案：RAG 在 snail-ai-server 的 features 下；gRPC 协议在 snail-ai-commons；启动入口在 snail-ai-starter。如果你答成了 docs 或静态 admin 目录，说明还需要把“说明材料”和“运行时代码”分开。

主要推荐来源：先读 docs/architecture/overview.md，再回到根 pom.xml 对照模块边界。遇到不清楚的模块职责，可以继续向 agent 追问“这个模块在请求链路中承担什么角色”。
