Snail AI 文档与部署模块导览

这一课只做一件事：学会从文档站、部署脚本、SQL 初始化和 CI workflow 还原 Snail AI 的“如何运行”。学完后，你应该能判断一个启动问题该先看文档、Compose、SQL、运行配置还是发布流水线。

L1 架构位置：该模块位于运行支撑层，连接 L0 的 Server-Agent 架构、数据库/向量/对象存储依赖和发布产物。

学习目标：还原运行支撑链路 预计 12 分钟 主来源：docs/.vitepress/config.mts docs/docker/docker-compose.yaml snail-ai-starter/src/main/resources/application.yml

1. 文档站是入口地图

Snail AI 的文档站在 docs/ 下，package.json 用 vitepress dev/build/preview 管理本地预览和构建。.vitepress/config.mts 使用 VitePress 并套上 Mermaid 插件；nav.ts 决定顶部导航，sidebar.ts 把部署页集中到 /deploy/ 分组。读部署前先看这三处，可以快速知道“官方希望你按什么路径学习”。

docs/ ├── package.json ← VitePress 脚本和依赖 ├── .vitepress/ │ ├── config.mts ← 站点主配置、Mermaid、搜索、主题挂载 │ └── config/ │ ├── nav.ts ← 顶部导航 │ └── sidebar.ts ← 部署指南侧边栏 └── deploy/ ← Docker、生产、配置、升级、故障排除

2. 启动顺序从依赖开始

本地学习常见路径是：准备数据库和依赖组件，导入 SQL，再启动 Server。当前 docs/docker/docker-compose.yaml 实际包含 snail-ai、mysql、pgvector、elasticsearch、etcd、minio、milvus。其中 snail-ai 依赖 MySQL 健康检查，暴露 8900 和 18888；MySQL 会把 docs/sql/snail_ai_schema.sql 挂进初始化目录。

MySQL / PostgreSQL 初始化
  -> 导入 docs/sql 下匹配数据库的 schema
  -> 启动向量、搜索、对象存储依赖
  -> 启动 Snail AI Server
  -> 访问 http://localhost:8900/snail-ai
  -> Agent Client 连接 Server gRPC 18888

关键差异：docs/deploy/docker.md 和 guide/quick-start.md 仍说 Compose 不包含 Server，但当前 YAML 已包含 snail-ai 服务。排查时以实际执行的 YAML 和 application.yml 为准。还要注意部署路径漂移：当前 checkout 以 docs/docker/docker-compose.yaml 和 docs/sql/ 为准，源文档中根目录 script/ 的旧说法不要作为当前文件位置依据。

3. 配置决定运行边界

application.yml 给出默认运行契约：HTTP 端口 8900，上下文路径 /snail-ai，gRPC 端口 18888，默认 MySQL 地址是本机 snail_ai 库。Compose 会通过环境变量覆盖数据库地址，把 Server 连到容器内 mysql:3306，并把上传目录和日志目录挂到 volume。SQL 初始化要按数据库匹配：MySQL 用 snail_ai_schema.sql，PostgreSQL 用 snail_ai_schema_pgsql.sql，不要混用。

边界默认值或来源排查意义
HTTP8900 + /snail-ai后台、Admin API、OpenAPI、健康检查入口。
gRPC18888Agent Client 注册、心跳和任务分发入口。
数据库MySQL 默认；PostgreSQL 可切换启动报连接或表缺失时先查这里。
资源LOCAL 或 MinIO上传文件、Skill 和多节点部署的持久化边界。

4. CI 把源码变成产物

两个 workflow 都由 v* tag 推送触发。docker-image.yml 在 JDK 21 上执行 Maven package，随后用 snail-ai-starter/ 作为 Docker build context，推送 latest 和项目版本号镜像。maven-publish.yml 同样先构建，再用 GPG 和 Maven Central 凭据执行 mvn deploy -P release。所以发布链路的关键前提是：根 Maven 构建能产出 snail-ai-starter/target/snail-ai-server-exec.jar，而 Dockerfile 会把它放进镜像内的 app.jar。

检索练习：不看正文回答三个判断：数据库表缺失时先看 workflow 还是 SQL？Agent Client 连不上时先看 HTTP 端口还是 gRPC 端口？Docker 文档和 Compose 不一致时，以哪个文件为执行事实？
参考答案：表缺失先看对应数据库的 SQL 初始化脚本；Agent Client 先看 gRPC 18888；实际执行事实优先看 docs/docker/docker-compose.yaml，再用部署文档解释意图。更多端口、环境变量和风险清单见 文档与部署速查。

主要推荐来源：先读 docs/docker/docker-compose.yaml，再对照 docs/deploy/configuration.md 和 snail-ai-starter/src/main/resources/application.yml。遇到启动失败，可以继续向 agent 追问“这个错误属于依赖组件、SQL 初始化、Server 配置还是发布产物”。
