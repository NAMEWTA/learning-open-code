Parent 易错点：snail-ai-server-features 在目录上位于 snail-ai-server/ 内，但 POM 的 <parent> 直接指向根 snail-ai，不是 snail-ai-server。判断版本与插件继承时看 parent 声明，不要只看文件夹嵌套。

Reactor 位置

根 pom.xml 把 snail-ai-server 列为五大顶层模块之一。执行 mvn package 时，Maven 按 reactor 顺序编译整棵子树；Docker/CI 最终只关心 snail-ai-starter 产出的 snail-ai-server-exec.jar，但 Server 业务 JAR 均来自本树。

snail-ai/ ← 根 reactor（com.aizuda:snail-ai） ├── pom.xml ← dependencyManagement、build 插件、release profile └── snail-ai-server/ ├── pom.xml ← 聚合 pom，列出 4 个子模块 ├── snail-ai-server-persistence/ ├── snail-ai-server-features/ │ ├── pom.xml ← 聚合 pom，列出 7 个 feature 子模块 │ └── snail-ai-feature-common/ │ └── pom.xml ← 可发布 jar，依赖 model + grpc ├── snail-ai-server-admin/ └── snail-ai-server-openapi/

Artifact 坐标

模块groupIdartifactIdversionpackaging本地路径
Snail AI 根 com.aizuda snail-ai ${revision}（当前 1.0.0） pom open-java/snail-ai/pom.xml
Server 聚合 com.aizuda snail-ai-server ${revision} pom snail-ai-server/pom.xml

Feature 公共层 com.aizuda snail-ai-feature-common ${revision} jar .../snail-ai-feature-common/pom.xml

三个目标 POM 均未重写 groupId 或 version，全部继承根 POM 的 com.aizuda:${revision}。子模块在 dependencyManagement 中已登记，互相引用时可省略 <version>。

Parent 继承链

模块parent artifactIdrelativePath继承内容

模块依赖树

snail-ai-server 子模块（聚合）

snail-ai-server/ packaging=pom ├── snail-ai-server-persistence/ 持久化：biz/vector/search 存储 ├── snail-ai-server-features/ 业务 feature 聚合（见下） ├── snail-ai-server-admin/ Admin REST + 静态后台依赖入口 └── snail-ai-server-openapi/ OpenAPI 对外接口

snail-ai-server-features 子模块（聚合）

snail-ai-server-features/ packaging=pom ├── snail-ai-feature-common/ 跨 feature 公共能力（jar） ├── snail-ai-feature-model/ 模型配置与管理 ├── snail-ai-feature-agent/ 智能体业务 ├── snail-ai-feature-rag/ RAG 知识库 ├── snail-ai-feature-skill/ Skill 管理 ├── snail-ai-feature-memory/ 记忆 └── snail-ai-feature-resource/ 资源与存储

Reactor 顺序 <modules> 列表中 feature-common 排在 feature-model 之前，但 common 的 POM 声明依赖 model。Maven 按依赖图而非列表顺序编译，不会因此失败。

汇入可执行 JAR 的路径（部署相关）

snail-ai-feature-common (+ 其他 feature-*)
  -> snail-ai-server-admin / snail-ai-server-openapi
  -> snail-ai-starter（spring-boot-maven-plugin repackage）
  -> target/snail-ai-server-exec.jar
  -> Dockerfile ADD 为 app.jar
  -> Compose opensnail/snail-ai:latest

snail-ai-feature-common 编译依赖

这是三个目标 POM 中唯一声明 <dependencies> 的模块，也是 Server 侧 feature 层的“薄公共层”。

依赖 artifactIdscope版本来源部署含义
snail-ai-feature-model compile（默认） dependencyManagement → ${revision} 拉入模型 starter、MyBatis-Plus、Spring AI 等；最终进入 fat JAR
snail-ai-commons-grpc compile（默认） dependencyManagement → ${revision} gRPC 服务端/客户端基础设施；对应运行时 18888 端口能力

snail-ai-server-admin 同时直接依赖全部 7 个 feature 模块；common 提供跨 feature 共享类型与配置，不是唯一入口，但改 common 会影响 admin/openapi 编译 classpath。

根 POM 构建插件（三模块均继承）

目标 POM 自身未声明 <build>，以下插件均来自根 open-java/snail-ai/pom.xml。

插件版本属性关键配置对本地构建的影响
maven-compiler-plugin 3.15.0 <release>21</release>；注解处理器：lombok、spring-boot-configuration-processor 所有 jar 模块统一 Java 21；与 CI/Docker 的 JDK 21 一致
flatten-maven-plugin 1.7.3 flattenMode=oss；process-resources 阶段 flatten；clean 阶段 clean 发布前把 ${revision} 解析为字面版本，生成可发布的扁平 POM
maven-source-plugin 3.4.0 默认配置；release profile 下 attach-sources 本地 package 不强制附加源码；Central 发布时附加

可执行 JAR spring-boot-maven-plugin 仅配置在 snail-ai-starter/pom.xml：finalName=snail-ai-server、classifier=exec、主类 SnailAiSpringbootApplication。三目标 POM 产出的是库 JAR，不是可直接 java -jar 的产物。

Profile：release

元素id / 命令行为与部署关系
Profile release 仅在显式 -P release 时激活 本地学习/CI 构建镜像通常不需要此 profile
maven-gpg-plugin verify → sign GPG 签名，--pinentry-mode loopback maven-publish.yml 在 tag 推送后执行
maven-javadoc-plugin attach-javadocs doclint=none Maven Central 要求 javadoc 附件
central-publishing-maven-plugin ossrh autoPublish=false mvn deploy -P release 上传 staging
distributionManagement — snapshot / release 仓库指向 Sonatype OSSRH 与 Docker 镜像发布并行、独立

与部署产物的命令对照

场景命令Profile关键产物

速查练习：① snail-ai-server-features 的 parent 是谁？② 哪个模块产出 snail-ai-server-exec.jar？③ 发布到 Maven Central 需要哪个 profile？
答案：① 根 snail-ai（不是 server 聚合 pom）；② snail-ai-starter；③ release。
