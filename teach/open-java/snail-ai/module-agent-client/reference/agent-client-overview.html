Agent Client 执行层速查

这份参考用于快速定位 `snail-ai-agent` 中 Client 执行层的职责、接口、依赖和扩展点。短课负责建立地图，本文负责放长清单。

1. 架构位置与主链路

Agent Client 执行层位于 L0 总体架构中的 Client 节点，接在 Server 端 Agent 责任链之后。Server 端完成上下文装配并通过 gRPC 下发 `ChatDispatchRequest`；Client 端负责实际模型调用、本地工具执行、Advisor/Interceptor 观测与流式回传。

Server Agent Chain └─ LlmCallHandler └─ gRPC /chat/dispatch └─ ClientRequestDispatcher └─ ChatDispatchStreamingHandler └─ ChatSessionRuntime ├─ ToolRuntime └─ ClientChatExecutor ├─ PromptFactory ├─ ChatClientFactory └─ Advisor / Interceptor / Tool / ChatModel

3. 公共接口清单

第 3 节列接入方常用公共接口，第 5 节补充 Tool resolver 公共类，便于后续 L3 API 覆盖。

接口 / 类使用者看它解决什么问题
@EnableSnailAiAgent接入方 Spring Boot 应用导入 Client 自动配置。
SnailAiAgentProperties部署与配置`snail-ai.*` 配置：appId、token、port、server、并发、临时目录。
ClientGrpcServer自动配置启动 Client 本地 gRPC 服务，注册 unary 与 server-streaming service。
ClientRequestDispatchergRPC server把请求的 URI 交给通用 `GrpcRequestDispatcher`。
RpcClientTool 与 callback回调 Server：创建会话、保存记录、短期记忆、获取 Skill 内容、RAG 搜索。
GrpcChannelProvider`RpcClient` 动态代理保存心跳建立的 Server channel 和认证 header。
ChatDispatchStreamingHandlerClient 分发入口把 `/chat/dispatch` 请求转为 `ChatSessionRequest`，并把 text/thinking/completion/error 写回 observer。
ChatSessionRuntimegRPC handler一次 Chat 会话的运行壳：计数、解析工具、执行、清理。
ClientChatExecutor会话运行时构建 `Prompt` 与 `ChatClient`，设置 Advisor context，订阅响应流。
PromptFactory`ClientChatExecutor`把 system prompt、history、user message 变成 Spring AI `Prompt`。
ChatClientFactory`ClientChatExecutor`构建 Spring AI `ChatClient`，装配模型、Advisor 和工具。
ChatModelFactory`DefaultChatClientFactory`把 Server 下发的 `ModelConfig` 转为可调用 `ChatModel`。
ChatClientCustomizer接入方扩展在 `ChatClient.Builder` build 前做横向定制。
SnailAiInterceptor接入方扩展在 LLM 调用前后修改请求或响应，按 order 进入 `InterceptorChainAdvisor`。
CustomToolCallbackProvider工具扩展扫描业务 Spring Bean 中的 `@Tool` 方法，缓存为 `ToolCallback`。

5. Tool resolver 速查

Resolver / Provider何时注册工具产物
BaseToolResolver每次对话都会创建按 agentId/conversationId 隔离的工作目录。`shell`、`http_request`。
ClientRagToolResolverAgent 启用 RAG、有 ragIds，且不是强制调用模式。`rag_search`，通过 `RpcClient.searchRag` 回调 Server。
ClientSkillToolResolver`ChatDispatchRequest.skills` 非空。`read_skill`，并把可用 skill 列表追加到 system prompt。
ClientMcpToolResolver`mcpServers` 非空，支持 Streamable HTTP 与 Stdio；SSE 在当前代码中告警后跳过。MCP Server 暴露的 `ToolCallback`。
CustomToolCallbackProviderSpring 容器启动后扫描非内部包 Bean。业务自定义 `@Tool` 方法。
ClientMemoryToolResolver当前返回空列表，且未注入 `ToolRuntime` 构造器。无独立 memory tool。

6. 调用示例与排查入口

你看到的问题优先入口判断依据
Client 没上线或 Server 找不到实例ClientHeartbeatScheduler、`snail-ai.app-id/token/server/port`心跳负责注册 hostId、hostIp、port、并发数和认证 header。
收到请求但没有输出ChatDispatchStreamingHandler、ChatSessionRuntime先确认是否解析 dispatch 成功、是否进入 `execute`、是否触发 error consumer。
模型配置报错ClientModelConfigValidator、ClientChatModelSpecFactory当前校验要求 `apiEndpoint`、`apiKey`、`modelKey` 非空。
工具没有被模型看到ToolRuntime、各 resolver、CustomToolCallbackProvider工具最终必须作为 `ToolCallback` 进入 `DefaultChatClientFactory.defaultTools`。
thinking 或 token 没记录ThinkingCollectorAdvisor、TokenUsageCollectorAdvisor依赖模型响应 metadata；测试只覆盖 reasoningContent metadata 的 happy path。

最小接入示例来自 `snail-ai-agent-example`：应用类加 `@EnableSnailAiAgent`，配置 `snail-ai.enabled=true`、`app-id`、`token`、Client 端口和 Server gRPC 地址。之后 Server 端 Agent 关联该应用，Server 的 `LlmCallHandler` 才会把聊天请求分发到这个 Client。
