资源管理模块导览

这一课只做一件事：建立“上传文件如何变成业务可复用资源”的主链路。学完后，你应该能判断一个问题属于上传管理、存储策略、元数据持久化、公开访问，还是 RAG/头像等上层业务复用。

学习目标：识别资源模块边界 预计 12 分钟 先回 L0：项目地图 主来源：open-java/snail-ai/snail-ai-server/snail-ai-server-features/snail-ai-feature-resource/src/main/java/com/aizuda/snail/ai/features/resource/ResourceService.java open-java/snail-ai/snail-ai-server/snail-ai-server-admin/src/main/java/com/aizuda/snail/ai/admin/controller/FileAccessController.java

1. 职责：保存文件，也保存“如何找到它”

资源模块不是某个业务页面的上传组件，而是 Server 侧的通用文件底座。它接收文件流，选择默认存储后端，把文件写到本地目录或 MinIO，再把 storageKey、原始文件名、大小、MIME、存储类型、业务类型、业务 ID、创建人写入 sai_resource。上层业务通常只保存 resourceId 或 accessUrl：RAG 文档保存 resourceId 用于后续解析，用户头像校验 bizType=AVATAR 后绑定到用户，OpenAPI 外部头像则直接写一条 EXTERNAL 资源记录。

2. 位置：三个入口，一个核心服务

snail-ai-server/ ├── snail-ai-server-admin/ ← HTTP 入口与管理端授权 │ ├── controller/ │ │ ├── ResourceController.java ← /resource 上传、删除、分页 │ │ └── FileAccessController.java ← /files/** 公开文件访问 │ └── service/resource/ │ └── ResourceAdminService.java ← 当前用户、分页筛选、删除授权 ├── snail-ai-server-features/ │ └── snail-ai-feature-resource/ ← 资源核心能力 │ ├── ResourceService.java ← 上传、加载、删除、查询 │ └── strategy/ ← LOCAL 与 MINIO 存储策略 └── snail-ai-server-persistence/snail-ai-biz-storage/snail-ai-biz-template/ └── src/main/java/com/aizuda/snail/ai/persistence/resource/ ├── po/ResourcePO.java ← 资源表实体 └── mapper/ResourceMapper.java ← 资源表 Mapper

ResourceController 是登录态管理入口，适合“我上传了什么、能否删除”的问题；ResourceService 是业务复用入口，适合“如何把文件保存并返回资源记录”的问题；FileAccessController 是文件访问入口，适合“浏览器如何拿到文件流”的问题。完整清单见 资源管理模块参考。

3. 主链路：上传后两次写资源表

POST /resource/upload
  → ResourceAdminService.upload
  → ResourceService.upload
  → ResourceStorageFactory.getDefault()
  → Local/MinIO.store(...) 返回 storageKey
  → insert ResourcePO
  → storage.getAccessUrl(id, storageKey)
  → update ResourcePO.accessUrl

这里最容易漏掉的是最后一次更新：ResourceService.upload 先插入资源行拿到自增 ID，再调用存储策略生成访问 URL，最后更新 accessUrl。当前 LOCAL 和 MINIO 都返回 /files/{storageKey}。StorageKeyGenerator 生成的 key 形如 document/2026/07/07/uuid.pdf，业务类型会变成路径分区；RAG 预览上传先用 DOCUMENT_PREVIEW，提交成功后再调整为 DOCUMENT。

4. 访问边界：公开 URL 不等于资源管理权限

SnailJobWebMvcConfigurerAdapter 明确把 /files/** 排除在认证拦截器之外；这意味着文件访问链路不看登录态，而是把符合格式的 storageKey 当作访问凭证。FileAccessController 不接受自增资源 ID，只从 URL 中解析 storageKey，再查资源表；它只代理 LOCAL 和 MINIO，不会代理 EXTERNAL。响应侧会设置 MIME、缓存、X-Content-Type-Options: nosniff，并按白名单决定 inline 还是 attachment。管理接口的删除和分页仍由 ResourceAdminService 按管理员或创建人限制。

暂停回忆：不看上文，判断三个说法：访问 /files/document/日期/uuid.pdf 一定需要登录吗？RAG 解析文档时读取的是 accessUrl 还是 resourceId？EXTERNAL 头像会被 FileAccessController 代理吗？
参考答案：/files/** 被认证拦截器排除，不一定需要登录；RAG 管线通过 resourceId 调用 ResourceService.load；EXTERNAL 不走文件代理，它的 accessUrl 就是外部 URL。若你把公开文件访问和资源管理权限混为一谈，需要重看第 4 节。

主要推荐来源：先读 ResourceService.java，再对照 FileAccessController.java。后续如果想继续细化，可以向 agent 追问“RAG 预览上传如何清理临时资源”或“公开 storageKey 是否适合当前部署安全策略”。
