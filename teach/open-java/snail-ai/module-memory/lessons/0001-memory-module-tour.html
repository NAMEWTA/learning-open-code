1. 职责：给下一轮对话补上下文

在 L0 项目地图 里，记忆和 RAG 一样属于 Server 侧上下文能力，不是模型适配层。它的目标是让 Agent 在当前请求里拿到“之前说过什么”和“长期应该记住什么”。当前源码最明确落地的是短期记忆：按会话保存最近 N 条消息，并在下一次分发给 Client 前写入 ChatDispatchRequest.historyMessages。

长期记忆在官方架构文档和公共 DTO/枚举中已经有完整设计，包括记忆类型、状态、抽取事件、召回请求和 memoryContext 字段。但源码检索没有找到文档里提到的 MemoryInjectionAdvisor 实现，变更记录也写过移除。因此本课把长期记忆视为“契约和后续切片入口”，不把它讲成当前已跑通的主链路。

2. 位置：feature-memory 被 Agent 链消费

snail-ai-server/ ├── snail-ai-server-features/ ← Server 侧业务能力聚合 │ ├── snail-ai-feature-memory/ ← 记忆 DTO、指标、短期 store │ └── snail-ai-feature-agent/ ← Agent Handler Chain 消费记忆能力 └── snail-ai-server-persistence/ ← 对话记录表与 Mapper snail-ai-commons/ └── snail-ai-commons-core/ ← 公共记忆 DTO 与枚举契约

snail-ai-feature-memory 的公共类不直接处理 HTTP 入口；它更像一个被 Agent 链和 Admin VO 复用的能力包。模块内部分成三层：dto 表达记忆数据和指标，store 定义短期记忆端口，两个 store 实现负责 JVM 内存或数据库读取。完整类表见 记忆速查。

3. 分层：短期窗口是运行路径

Agent 请求进入
  ↓
McpHandler(@Order 40) / RagHandler(@Order 60) / SkillAgentChatHandler(@Order 70)
  ↓ MCP、RAG、Skill 上下文就绪
ContextCollectorHandler(@Order 75)
  ↓ 接口层 append(user) + loadHistory(...)
ShortTermMemoryStore
  ├─ InMemoryShortTermMemoryStore  ← 单机高性能，cache miss 回落 DB
  └─ DbShortTermMemoryStore        ← 从对话记录表读取，append 为空实现
  ↓
ChatDispatchRequest.historyMessages
  ↓
LlmCallHandler(@Order 80) → Agent Client

ContextCollectorHandler(@Order 75) 位于 SkillAgentChatHandler(@Order 70) 之后、LlmCallHandler(@Order 80) 之前运行。也就是说，RAG、Skill、MCP 等上下文已经由前序 Handler 放入 AgentChatContext，它再在真正分发前收集目标 Client 和短期历史。

关键依赖有三类：配置来自 snail-ai.memory.short-term.store-type；历史数据来自 AgentConversationRecordMapper；消费方来自 Agent 责任链。InMemoryShortTermMemoryStore 用 ConcurrentHashMap 维护滑动窗口，重启后可从 DB 回填；DbShortTermMemoryStore 每次查询对话记录表，更适合多实例，但它的 append 当前为空实现。

4. 调用示例：接口先 append，再读历史

open-java/snail-ai/snail-ai-server/snail-ai-server-features/snail-ai-feature-agent/src/main/java/com/aizuda/snail/ai/feature/agent/chain/ContextCollectorHandler.java

int shortTermWindow = agent.getShortTermMemorySize() != null && agent.getShortTermMemorySize() > 0
        ? agent.getShortTermMemorySize()
        : DEFAULT_SHORT_TERM_WINDOW;
boolean shortTermEnabled = !Boolean.FALSE.equals(agent.getMemoryEnabled());
if (shortTermEnabled) {
    shortTermMemoryStore.append(ctx.getConversationId(), "user", ctx.getContent(), shortTermWindow);
    ctx.setHistoryMessages(loadHistoryMessages(ctx, shortTermWindow));
} else {
    ctx.setHistoryMessages(List.of());
}

这个片段说明的是接口层调用顺序：ContextCollectorHandler 先调用 append(user)，再调用 loadHistory。在 memory 模式下，InMemoryShortTermMemoryStore.append 会把当前用户消息写入 JVM 队列；在 db 模式下，ConversationHandler(@Order 20) 已经更早把当前用户消息写入对话记录表，DbShortTermMemoryStore.append 当前为空实现。

因此，DB store 的“先 append”不是再写一次数据库，而是保持同一个接口调用形态；真正的历史来源是 AgentConversationRecordMapper 读取的对话记录。store 的 loadHistory 会排除最后一条消息，因为当前用户消息已经在本次 Prompt 中，重复注入会让模型看到两遍同一句话。模型完成后，ChatStreamObserver 触发 ChatResultPersistService 持久化 assistant 回答，并在 memory 模式追加到短期窗口，供下一轮使用。

暂停回忆：如果一次对话的 memoryEnabled=false，当前源码会影响哪一个字段：historyMessages、modelConfig 还是 mcpServers？
参考答案：影响 historyMessages，它会被设置为空列表；模型配置和 MCP 描述符由其他 Handler 填充。若你选择了 memoryContext，要注意当前 L1 只确认它是分发请求字段，未确认有长期记忆实现写入它。
