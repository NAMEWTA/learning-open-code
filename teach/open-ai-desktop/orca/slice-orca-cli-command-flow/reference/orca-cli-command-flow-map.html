Orca CLI 命令执行参考

范围：CLI 入口、command spec、handler、RuntimeClient、本地/远程 transport、main runtime RPC 边界与测试护栏。

端到端时序

阶段 主文件 关键事实
进程入口 src/cli/index.ts 处理 `agent-teams-tmux`、`claude-teams` special shim；普通命令进入 parse、help、validate、RuntimeClient、dispatch。
参数解析 src/cli/args.ts 非 flag token 进入 commandPath；boolean flag 不消费值；`--flag=value` 支持值以 `--` 开头；positionals 由 spec 归一化。
命令声明 src/cli/specs/index.ts `COMMAND_SPECS` 聚合 core、project、file、browser、orchestration、computer、environment、linear、vm、emulator 等 spec。
handler 查找 src/cli/dispatch.ts 多个 handler group 合并成 Map，重复 key 立即失败；实际查找 key 是 `commandPath.join(" ")`。
命令适配 src/cli/handlers/terminal.ts 代表性普通命令：解析 terminal/worktree/timeout flag，调用 `terminal.*` runtime RPC，再格式化输出。
本地 runtime client src/cli/runtime/client.ts 无 remote pairing 时读取 metadata 并调用 `sendRequest()`；失败由 RuntimeClientError/RuntimeRpcFailureError 统一表达。
远程 runtime client src/cli/runtime/websocket-transport.ts pairing code 或 saved environment 转成 WebSocket request，错误映射回 CLI RuntimeClientError。
runtime server src/main/runtime/runtime-rpc.ts main 进程发布 metadata，校验 auth/device token，控制 long-poll，并调用 RpcDispatcher。
runtime 方法 src/main/runtime/rpc/methods/index.ts `ALL_RPC_METHODS` 是 server 暴露方法的总清单；各 method 文件才是真正业务入口。

入口和语法校验

src/cli/index.ts 的普通路径先调用 parseArgs() 和 normalizeCommandPositionals()。 如果用户请求 help，直接打印，不创建 RuntimeClient。没有 commandPath 时打印 root help。 只有非 help 的真实命令才进入 try/catch。 

parse argv
  -> help? printHelp and return
  -> empty? root help and return
  -> validateCommandAndFlags
  -> new RuntimeClient
  -> dispatch

`validateCommandAndFlags()` 先找 command spec，不存在就抛 invalid_argument。 然后检查 positional 和 flag 是否重复提供，最后逐个 flag 检查它是否是 global flag、spec allowed flag， 或 browser 命令允许的 --page。这个顺序是用户体验护栏：语法错误不应被 runtime unavailable 掩盖。 具体 allowed flags 不在聚合文件里，而在 src/cli/specs/core.ts、 src/cli/specs/serve.ts、src/cli/specs/browser-basic.ts、 src/cli/specs/browser-advanced.ts、src/cli/specs/orchestration.ts 等具体 spec 文件中声明。

src/cli/args.test.ts 覆盖了 --text=--help、repeatable --label、 unknown command-specific flag 等边界；src/cli/index.test.ts 还覆盖 positional 和 flag 同时传入时的冲突提示。

remote selection 和本地命令

RuntimeClient 构造参数里，undefined 表示允许读取默认环境变量， null 表示明确禁用 remote fallback。src/cli/index.ts 对 environment、serve、agent、vm 传入 null， 这样即使 shell 里有 ORCA_ENVIRONMENT，本地管理命令也不会误连远程 runtime。

src/cli/runtime/client.ts 的 resolveRemotePairing() 保证 pairing code 与 environment selector 互斥。 environment selector 会调用 saved environment store，pairing code 会解析 `orca://pair?...` 或 bare payload。 malformed pairing code 在本地 metadata lookup 前就失败。

remote 命令的另一个边界在 src/cli/selectors.ts。active/current 这种 cwd shortcut 只能用于本机 Orca，因为 paired CLI 的 cwd 属于客户端机器，不属于远程 server。 browser 命令在远程省略 worktree 时让 server-side focus 决定目标；terminal create 在远程必须显式给 --worktree。 worktree current 是更严格的例子：src/cli/specs/core.ts 没有给它声明 --worktree，src/cli/handlers/worktree.ts 也固定解析当前 cwd。 远程场景应改用 worktree show --worktree id:<id> 这类接受显式 selector 的命令。

handler 形状

src/cli/handlers/core.ts 说明 core 命令和普通 RPC 命令的不同： open 调 RuntimeClient.openOrca() 并轮询 reachable； status 调 getCliStatus()，普通模式下 runtime unreachable 会设置 exit code 1； serve 直接调用 serveOrcaApp() 启动 foreground server，并校验 port、pairing flag、recipe-json 约束。 src/cli/help.ts 和 src/cli/runtime/launch.ts 分别是 help 文案与 app/server 启动的支撑文件。

普通命令 handler 通常有四步：读 flag、解析 selector、调用 client.call()、调用 printResult()。 src/cli/handlers/worktree.ts 把 repo、parent、agent、setup、Linear issue 等 flag 组装成 worktree.create params；src/cli/handlers/browser-interact.ts 把 element/value/mouse flag 映射到 browser.click、browser.fill、browser.mouseMove 等 method； src/cli/handlers/browser-nav.ts 给 goto/reload/wait 这种慢操作设置 per-call timeout。

src/cli/handlers/terminal.ts 是理解 selector 的好入口。没传 --terminal 时， 它会通过 terminal.resolveActive 找当前 worktree 的活跃终端。terminal.wait 对 timeout 有单独设置，未满足等待条件时会把 CLI exit code 设为 1。远程 terminal create 不允许隐式 cwd 推断，必须传 server-side worktree selector。

src/cli/handlers/orchestration.ts 有两个特例。check --wait 会向 stderr 输出 JSON heartbeat， 防止父进程因为长时间无输出把 subprocess 背景化；ask --json 输出 bare JSON result， 不走普通 pretty RPC envelope，方便 worker 直接管道到 `jq`。src/cli/handlers/orchestration.test.ts 覆盖 check/ask 的 timeout flag；src/main/runtime/orchestration-cli-subprocess.test.ts 以 subprocess 方式验证 check --wait heartbeat 只写 stderr，stdout 保持干净 JSON。

RuntimeClient 本地路径

src/cli/runtime/metadata.ts 先决定 userData 路径。macOS 默认是 ~/Library/Application Support/orca，Windows 走 APPDATA，Linux 走 XDG config；ORCA_USER_DATA_PATH 可以覆盖。metadata 必须包含兼容 transport 和 authToken。metadata 的 shared shape 和 transport 查找来自 src/shared/runtime-bootstrap.ts。

src/cli/runtime/transport.ts 用 metadata 里的 Unix socket 或 named pipe 建连， 写入一行 JSON request，并解析 newline-delimited response frame。它会拒绝 invalid JSON、schema 不匹配、 response id 不匹配，以及 runtimeId 在 in-flight 时变化的情况。keepalive frame 是 {"_keepalive":true}，只刷新 timeout，不交给 caller。

src/cli/runtime/status.ts 是本地 status 的特殊包装。读不到 transport 时返回 not_running 或 stale_bootstrap； transport 存在但 RPC 不通时，会用 pid 判断 app 是 starting 还是 stale_bootstrap。 这比普通 RPC 的 runtime unavailable 更适合 CLI status。

RuntimeClient 远程路径

src/cli/runtime/client.ts 如果发现 remotePairing，就用 src/cli/runtime/websocket-transport.ts 调 shared src/shared/remote-runtime-client.ts。 除 status.get 外，第一次普通 RPC 之前会先请求 status，结合 src/shared/protocol-version.ts 和 src/shared/protocol-compat.ts 判定 client/server protocol 是否兼容。

远程 status 的语义也不同：它证明 paired runtime reachable，但本机 app.running 固定为 false、pid 为 null。 如果使用 saved environment，成功后会记录 recently used runtimeId，供环境列表排序或状态显示。 旧导入路径由 src/cli/runtime-client.ts 继续导出，错误类型定义在 src/cli/runtime/types.ts。

main runtime RPC 边界

CLI 传来的 method 不是直接进入业务函数。src/main/runtime/runtime-rpc.ts 启动时创建 Unix socket transport，生成 authToken，写入 metadata。WebSocket transport 是可选附加路径， 用 device token、E2EE channel 和 allowlist 区分 mobile/runtime scope。

本地 socket message 先经过 parseAndAuth()：必须有 id、method、authToken，且 token 等于 server 生成值。 long-poll request 由 isLongPollRequest() 分类，只有 terminal.wait 和 orchestration.check 且 wait=true 会占用 long-poll slot 并启动 keepalive。cap 满时返回 runtime_busy，短 RPC 仍可通过。 orchestration.ask 虽然也是阻塞式 RPC，CLI handler 会给它设置 per-call timeout override； 但 main runtime 的 isLongPollRequest() 不把它纳入 long-poll keepalive/cap 分类。

src/main/runtime/rpc/unix-socket-transport.ts 负责 socket idle timeout、1MB message cap、 newline frame、per-request AbortController 和 keepalive timer。WebSocket 路径在 src/main/runtime/rpc/ws-transport.ts 管连接数、pre-auth timeout、ping/pong heartbeat。

通过边界后，src/main/runtime/rpc/dispatcher.ts 从 registry 找 method， 用 method 自带 Zod schema parse params，调用 handler，并把 success/failure 包成 src/shared/runtime-rpc-envelope.ts 兼容的 envelope。src/main/runtime/rpc/core.ts 定义了普通 method、streaming method、request/response 和 registry。 CLI 侧 formatter 与 handler 的 result 类型主要引用 src/shared/runtime-types.ts。

方法表和业务入口

src/main/runtime/rpc/methods/index.ts 是 server 暴露方法的 manifest。 CLI handler 调的 worktree.create、terminal.wait、browser.click、 orchestration.check 都必须在这个总表里间接出现。最小例子是 src/main/runtime/rpc/methods/status.ts：`status.get` 无 params，handler 直接返回 runtime.getStatus()。

这也解释了排障顺序：如果 CLI 能 dispatch 但 runtime 返回 method_not_found， 先查 CLI handler method string 是否和 runtime method manifest 一致；如果返回 invalid_argument， 再查 runtime method schema，而不是只查 CLI allowed flags。

输出和错误

src/cli/format.ts 的 printResult() 在 --json 下输出 runtime success envelope， 普通模式只输出 formatter 文本。reportCliError() 在 JSON 模式下对 local CLI error 构造 failure envelope； 对 RuntimeRpcFailureError 则原样打印 runtime failure。普通模式下，runtime unavailable 会补充 “Run 'orca open' first” 的下一步。

这意味着测试 CLI 时要区分三类错误来源：CLI local validation error、transport/runtime availability error、 runtime method schema 或业务错误。它们都可能显示为 nonzero exit，但 JSON envelope 的 code/data 来源不同。

调试入口

  • unknown flag 却提示 Orca 未运行：先查 src/cli/index.ts 的 validate 是否仍早于 RuntimeClient lookup。
  • 命令在 help 里出现但无法执行：查 src/cli/specs/index.ts 和 src/cli/dispatch.ts 是否同时注册。
  • 远程命令误用客户端路径：查 src/cli/selectors.ts 的 remote cwd guard，以及 handler 是否绕过 selector helper。
  • long-poll 60 秒失败：查 src/cli/runtime/client.ts 的 timeout policy 和 runtime server keepalive/cap。
  • runtime 返回 method_not_found：查 CLI handler method string 与 src/main/runtime/rpc/methods/index.ts。
  • JSON 输出解析失败：查是否遇到 orchestration ask --json 这种 deliberately bare JSON 的特例。
