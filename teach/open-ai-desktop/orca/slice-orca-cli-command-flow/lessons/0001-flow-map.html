Orca CLI Command Flow：一条命令如何变成 runtime RPC

目标：15 分钟内能从 orca terminal send --text hi 追到 runtime method。

先抓四段主路径

Orca CLI 的主路径不是“解析完就执行业务”。它先在 src/cli/index.ts 处理 special shim、 help、命令规格校验和 remote selection，再交给 src/cli/dispatch.ts 找 handler。 handler 大多只把 flag 和 selector 变成 RPC method/params，最后由 src/cli/runtime/client.ts 选择本地 socket 或远程 WebSocket。

shell argv -> src/cli/index.ts parse/help/validate/RuntimeClient -> src/cli/dispatch.ts commandPath.join(" ") -> handler -> src/cli/handlers/*.ts flags/selectors -> client.call(method, params) -> src/cli/runtime/client.ts local metadata socket or remote pairing WebSocket

入口：先拒绝错误命令，再找 runtime

src/cli/args.ts 把非 -- token 收进 commandPath，把 flag 收进 Map； positional 参数会按 COMMAND_SPECS 归一化成 flag。关键顺序在 src/cli/index.ts： validateCommandAndFlags() 先运行，然后才构造 RuntimeClient。 所以 unknown command、unknown flag、positionals 冲突会报 CLI 语法错误，不会误导成 runtime unavailable。 

main(argv)
  -> normalizeCommandPositionals(COMMAND_SPECS, parseArgs(argv))
  -> resolveHelpPath(parsed)
  -> validateCommandAndFlags(COMMAND_SPECS, parsed)
  -> new RuntimeClient(...)
  -> dispatch(parsed.commandPath, ctx)

remote selection 也在入口决定。普通命令能用 --pairing-code、--environment 或环境变量； 但 environment、serve、agent、vm 会传入 null， 明确禁止 RuntimeClient 默认参数重新读取远程环境变量。

handler：薄适配，不藏业务

dispatch.ts 把 core、worktree、terminal、browser、orchestration 等 handler 组合成一个表， 并在重复 key 时直接抛错。比如 terminal send 会先用 selector helper 找 terminal handle， 再调用 terminal.send；browser click 解析 element 和 target 后调用 browser.click。这些文件的价值是看清 CLI flag 到 RPC params 的映射，而不是业务实现。 

orca terminal send --text hi --enter
  -> handler "terminal send"
  -> getTerminalHandle(flags, cwd, client)
  -> client.call("terminal.send", { terminal, text: "hi", enter: true, ... })
  -> printResult(...)

core handler 是例外：open 调 client.openOrca()，status 调 client.getCliStatus()，serve 直接走 serveOrcaApp() 启动 foreground server。

RuntimeClient：本地和远程在这里分叉

本地路径会读 userData 下的 runtime metadata，取 endpoint、authToken 和 runtimeId， 用 src/cli/runtime/transport.ts 发一行 JSON：{ id, authToken, method, params }。 client 只接受匹配 request id 和 runtimeId 的 terminal frame；keepalive frame 会刷新 timeout。

远程路径来自 pairing code 或 saved environment。普通 RPC 发送前，RuntimeClient 会先用 status.get 做 protocol compatibility check；不兼容时抛 incompatible_runtime。 terminal.wait 和 orchestration.check --wait 这类 long-poll 会按内层 timeoutMs 放宽 client-side timeout，避免默认 60 秒先超时。

检索练习

用户执行 orca worktree current --pairing-code remote --json，为什么不能让它改用显式远端 selector 继续完成？

参考路径：先看 src/cli/specs/core.ts，worktree current 的 spec 没有 --worktree 这类 selector flag；再看 src/cli/handlers/worktree.ts， handler 固定调用 resolveCurrentWorktreeSelector(cwd, client)。远程 runtime 的 worktree 在服务器文件系统上，客户端 cwd shortcut 只在本地有意义，所以这条命令在 remote 下应被拒绝。 需要显式远端 selector 时，改用 orca worktree show --worktree id:<id> --pairing-code remote --json 这类接受 selector 的命令。 完整来源见 Orca CLI 命令执行参考。

主要来源：src/cli/index.ts、 src/cli/args.ts、 src/cli/dispatch.ts、 src/cli/runtime/client.ts。
