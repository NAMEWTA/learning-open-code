Relay Reconnect 与 Grace Socket 参考

范围：部署复用、版本握手、socket ownership、grace timer、dispatcher stale request、main mux/session backoff 和测试护栏。

故障分层速查

层级 主要源码 判断问题
部署层 src/main/ssh/ssh-relay-deploy.ts 是否能找到 target-specific endpoint；是否复用旧 daemon；是否需要 fresh detached launch 或 Windows fallback pipe。
bridge ready 层 src/main/ssh/ssh-relay-deploy-helpers.ts、src/relay/relay-handshake.ts ready sentinel 是否出现；handshake mismatch 是否在 sentinel 前变成 exit 42 和 typed terminal error。

部署复用：旧 socket 优先，新 daemon 兜底

deployAndLaunchRelay() 先解析远端平台、版本目录和 node 路径，再进入 launchRelay()。 POSIX 路径用 relaySocketNameForInstanceId() 为 target id 生成 socket name，避免同一远端账号下不同 target 互相接上。 如果 test -S 返回 alive，主进程执行 relay.js --connect --sock-path ...。 如果这次 bridge 失败，部署层会把它当成可恢复 stale socket，尝试删除 socket 后 fresh launch。

POSIX launchRelay() ├─ probe socket ├─ ALIVE -> exec "node relay.js --connect --sock-path ..." ├─ bridge fail -> rm stale socket └─ fresh launch -> nohup node relay.js --detached --grace-time N ... └─ poll READY -> exec "--connect"

Windows 路径使用 named pipe。因为 pipe 不能按 Unix socket 模型 unlink，src/main/ssh/ssh-relay-endpoints.ts 维护 active pipe marker；旧 pipe 连接失败时，ssh-relay-deploy.ts 会切到确定性的 fallback pipe relaunch， 并把 marker 写回，下一次连接优先读这个 active endpoint。socket 名称本身由 src/main/ssh/ssh-relay-instance-id.ts 派生，保证同一远端账号下不同 target 的 relay endpoint 隔离。

版本握手：sentinel 只代表安全接管

src/relay/relay-handshake.ts 在 daemon socket 交给 dispatcher 前读取一个 handshake frame。 bridge 端的 runConnectHandshake() 发送当前 package version；daemon 端的 setupDaemonHandshake() 只接受 orca-relay-handshake 且版本匹配的请求。匹配后返回 ok；不匹配则写 mismatch frame 并关闭 socket。

src/main/ssh/ssh-relay-deploy-helpers.ts 的 waitForSentinel() 在 sentinel 前把 stdout 当作 startup buffer。 如果 bridge 在 sentinel 前以 exit 42 关闭，helper 会用 src/main/ssh/ssh-relay-handshake-mismatch.ts 构造 RelayVersionMismatchError，错误类型定义在 src/main/ssh/ssh-relay-version-mismatch-error.ts。这个错误到达 SshRelaySession 后触发 terminal callback， 再由 src/main/ipc/ssh.ts 发布 error state，跳过普通 relay-lost backoff。

注意边界：已有 socket 的一次 --connect 失败不一定立刻 terminal error。POSIX deploy path 会先把它当 stale socket 恢复机会处理，fresh launch 后仍失败才会冒泡。Windows path 则可能先切 fallback pipe。

Leftover bytes：握手和业务帧可以同包到达

src/relay/protocol.ts 的 FrameDecoder.drain() 是 handshake 交接的关键。 TCP/pipe 可能把 handshake-ok 和第一个 JSON-RPC frame 合在一个 chunk 里。daemon 侧和 bridge 侧都在 handshake 接受后 drain leftover， 再交给后续 dispatcher 或 main mux。src/relay/relay-handshake-roundtrip.test.ts 分别覆盖 daemon side leftover 与 bridge side leftover。 main 侧的普通 frame 常量与 sentinel 常量在 src/main/ssh/relay-protocol.ts，handshake frame 的独立解析由 src/relay/protocol-handshake.test.ts 保护，覆盖编码、解析、mismatch message、未知 handshake type 拒绝， 以及 Handshake message type 与 Regular/KeepAlive 的区分。

Socket ownership：旧 relay 不能删除新 relay 的 socket

src/relay/relay.ts 记录当前进程 bind socket 时读到的 identity，包括 dev、ino、ctimeNs。 ownsCurrentSocketPath() 只在当前路径仍指向同一个 socket identity 时允许 cleanup。 这保护了一个常见竞态：旧 relay 正在 shutdown，而新 relay 已经在同一路径 bind 成功，旧进程不能执行 unlink 把新 socket 删掉。

处理 EADDRINUSE 时，relay 也不是直接删文件。它先记录 blocked identity，再用短 timeout 尝试 connect。 connect 成功说明另一个 live relay 在服务，当前进程应该失败退出；connect 拒绝或路径消失才按 stale socket 处理。

Grace timer：保 PTY，不保空 daemon

场景 行为 源码锚点
用户配置 grace src/shared/ssh-types.ts 定义 default/min/max 和 0 值语义；main deploy/session 路径把非 0 值 clamp 到合法范围。 normalizeRelayGracePeriodSeconds()、launchRelay()
detached startup fresh daemon 没有 stdout client，启动后立即 enter grace；第一个 socket client 接入会 cancel grace。 src/relay/relay.ts
空 detached daemon 如果从未有 socket client 且没有 PTY，即使 grace 为 0，也使用短启动窗口，避免空 daemon 永久滞留。 EMPTY_DETACHED_STARTUP_GRACE_MS
系统睡眠 prepareForHostSleep() 发送 relay.configureGraceTime 为 0，让睡眠时间不吃掉普通 grace。 src/main/ssh/ssh-relay-session.ts

Dispatcher：多 client 不是共享一个流

src/relay/dispatcher.ts 的 primary client 代表 relay 启动时的原始 stdin/stdout。 socket client 通过 attachClient() 加入，每个 client 拥有自己的 decoder、write、seq、ack、generation。 通知会广播给所有未关闭 client；relay-originated request 使用 requestAnyClient()，并优先选择真实 socket client， 避免把 orca.cli 等请求发给已经失效的 synthetic primary。

generation 是 stale work 的判据。请求 handler 开始时捕获 client generation；如果 handler 返回前 client 被 invalidate/detach， response 被丢弃。对应 AbortSignal 也会被触发，长任务可以主动停下。rpc.cancel 通知则按 client/request id 取消指定请求。

Main mux 与 session backoff

src/main/ssh/ssh-channel-multiplexer.ts 把 relay frame 变成 JSON-RPC request/notification。 keepalive 每 5 秒发送；timeout 检查要求“长时间没有收到任何数据”和“最老未 ack 消息超时”同时成立，才 dispose 为 connection_lost。单个 request timeout 会拒绝 promise，并发送 rpc.cancel。

SshRelaySession.watchMuxForRelayLoss() 监听 mux dispose。如果 reason 是 connection_lost， 交给 src/main/ipc/ssh.ts 的 relay-lost backoff。该 backoff 最多 6 次，延迟从 500ms 指数增长到 15s； ready 后还要稳定 5 秒才清零，避免一个刚 ready 又立刻断的 relay 把计数重置。

PTY lease 和 replay 的边界

main store 的 SSH remote PTY lease 是 durable 索引，真正 live PTY 在远端 PtyHandler.ptys map。 reconnect 时，SshRelaySession.reattachKnownPtys() 合并当前 ownership 与 durable lease， 调 SshPtyProvider.attachForReconnect()。relay 侧 pty.attach 返回最近 100KB replay buffer； session 侧用 fingerprint 做 1 秒重复 replay 去重。

如果 attach 返回 not found，说明旧 relay 没保住该 PTY，session 会清 local ownership，把 lease 标为 expired， 并向 renderer 发送 pty:exit。这和 PtyHandler.revive() 不同：revive 只是在同一 relay 进程内根据序列化 PID 重新包装仍存活的 PTY，不能跨 relay 进程退出恢复。

诊断问答

问题 socket alive 但没有 sentinel，先看哪里？ 答案 先看 bridge handshake 和 exit code。sentinel 只在 handshake ok 后出现；mismatch 会在 sentinel 前关闭。

问题 reconnect 成功但旧终端没有 replay，先看哪里？ 答案 看 lease 是否还存在、`pty.attach` 是否 not found、relay 是否 fresh launch 了新 daemon，以及 100KB replay buffer 是否为空。

问题 为什么旧 stdin 要在第一个 socket client 接入后暂停？ 答案 防止 SSH 旧 channel 的残留 data listener 继续向 dispatcher feed bytes，和新 socket client 的 frame 流交错。
