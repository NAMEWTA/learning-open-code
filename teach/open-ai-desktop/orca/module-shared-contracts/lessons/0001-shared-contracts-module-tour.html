src/shared：Orca 多端之间的契约层

在 Orca 里，renderer、main、CLI、relay 和 mobile 不应各自猜测数据形态。src/shared 的职责就是提供统一类型、schema、纯函数和协议版本，让这些运行边界能用同一套语言交流。完整类型簇索引见 共享协议参考。

1. 为什么它是契约层

如果一个字段会跨进程、跨窗口、跨 SSH relay 或跨移动端传输，就不能只藏在某个 React 组件或 main service 里。比如 runtime 状态要被桌面、CLI、mobile 共同理解；SSH target 要被设置页、main 连接逻辑和 relay 会话复用；workspace session 要从磁盘恢复进 renderer。shared 层让这些边界先对齐，再各自实现。

这也解释了为什么 shared 里既有静态类型，也有 Zod schema 和纯函数。静态类型帮助 TypeScript 调用方编译期对齐；schema 保护磁盘、WebSocket、runtime RPC 等运行时输入；纯函数则把 worktree id、协议版本和路径判断这类规则集中在同一个地方，避免各端复制一份微妙不同的逻辑。

2. 核心文件速览

src/shared/ ├── runtime-rpc-envelope.ts ← RPC 成功/失败/keepalive 外壳 ├── runtime-types.ts ← runtime graph、terminal、mobile session ├── ssh-types.ts ← SSH target、relay grace、PTY lease ├── workspace-session-schema.ts ← 持久化 session 的 Zod 校验 ├── worktree-id.ts ← worktreeId 拆分与文件系统路径处理 └── protocol-version.ts ← 兼容版本和 capability

阅读顺序通常是：先看协议版本和 envelope，再看某条功能链路的类型簇，最后回到 main/renderer/mobile 的具体调用点。

3. 一个小例子

export const RuntimeRpcEnvelopeSchema = z.union([Success, Failure, Keepalive])

export function isKeepaliveFrame(frame: unknown): frame is RuntimeRpcKeepaliveFrame {
  return (
    typeof frame === 'object' &&
    frame !== null &&
    '_keepalive' in frame &&
    (frame as { _keepalive: unknown })._keepalive === true
  )
}

这段代码说明 shared 不只是类型声明。它还提供运行时校验和类型收窄：CLI、desktop 或未来客户端接收 runtime frame 时，可以先判定是成功、失败还是 keepalive，而不是用脆弱的字段猜测。

4. 检索练习

判断题：如果你要新增 mobile 读取的 runtime capability，应该只改 mobile 代码，还是也要检查 shared 的协议版本和 capability 清单？

参考答案：必须检查 shared。protocol-version.ts 明确记录 runtime 兼容窗口和 capability 名称；新增可忽略 capability 通常不用 bump 协议，但客户端和服务端都应基于同一清单协商。

课后主来源建议读 src/shared/runtime-rpc-envelope.ts 和 src/shared/protocol-version.ts。如果某个跨端字段不清楚，可以继续向 agent 追问“这个字段在哪些进程间传递、由哪个 schema 保护”。
