Orca 的第一张源码地图

本课只解决一个问题：当你打开 Orca 这个大型仓库时，先把它看成哪些运行环境和边界，而不是立刻陷入几千个文件。更完整的目录、技术栈和后续学习清单见 项目总览参考。

1. 项目定位

Orca 是一个面向并行 agent 开发的桌面 ADE。用户在一个工作区里启动多个 CLI agent，每个 agent 可运行在独立 git worktree 中；桌面端负责窗口、终端、编辑、任务、源码控制和集成，CLI 与远端 relay 负责把同一能力暴露给脚本、SSH 主机和移动端。

一句话记忆：Orca 不是单纯的终端壳，而是把 agent、worktree、terminal、browser、review 和 remote runtime 编排在一起的多进程系统。

2. 五个运行边界

Electron main src/main/index.ts 装配窗口、IPC、runtime、daemon、PTY、agent hooks、更新和系统能力。
Renderer src/renderer/src/App.tsx 承载 React UI、Zustand store、终端面板、侧栏和工作区交互。
Preload src/preload/index.ts 是安全桥，只把审计过的 API 暴露给 renderer。
CLI / Relay / Mobile src/cli/index.ts、src/relay/relay.ts 和 mobile/README.md 把桌面能力延伸到命令行、远端主机和手机。

3. 入口如何串起来

用户操作或 CLI 命令
├── renderer UI / orca CLI
├── preload API 或 RuntimeClient
├── Electron main 的 IPC / runtime / daemon 服务
├── 本地 PTY、Git、browser、agent hooks
└── SSH relay 或 mobile companion 复用同一运行时语义

这张链路图帮助你判断源码位置：UI 事件先看 renderer，安全边界看 preload，系统副作用看 main，远端执行看 relay，脚本入口看 CLI，共享类型和协议看 src/shared。后续 L1 课程会按这些边界展开。

4. 检索练习

暂停 30 秒，凭记忆回答：如果你要追踪“用户点击创建 worktree 后，agent 在远端 SSH 主机启动”的链路，至少要优先查看哪三个运行边界？

参考答案：先看 renderer 的创建入口和 store 状态，再看 Electron main 的 runtime / worktree / SSH 服务，最后看 relay 的远端 PTY、文件和 Git 处理。preload 与 shared 类型通常作为两端契约一起检查。

课后主来源建议先读 README.md 和 L0 参考文档。如果某个边界仍不清楚，可以继续向 agent 追问“这个功能从 UI 到 main/relay 的链路怎么走”，再进入后续 L1 主题。
