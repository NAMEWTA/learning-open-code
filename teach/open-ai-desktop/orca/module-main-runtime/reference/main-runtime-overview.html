主进程 Runtime 与 Daemon 模块总览参考

定位：L1 模块速查。配合 导览短课 使用。

职责边界

层 核心文件 一句话职责 典型调用者
状态层 src/main/runtime/orca-runtime.ts OrcaRuntimeService 持有 renderer graph、terminal handle、PTY 输出、mobile 会话、worktree、automation 和账号快照等活状态。 IPC handler、runtime RPC method、窗口装配、headless server。
控制面 src/main/runtime/runtime-rpc.ts OrcaRuntimeRpcServer 接收 Unix socket / named pipe 和 WebSocket 请求，完成认证、授权、keepalive、dispatch 和 metadata 发布。 src/cli、移动端、web runtime、headless serve。
PTY 宿主 src/main/daemon/daemon-init.ts、src/main/daemon/daemon-entry.ts 在独立 Node 进程中保留终端会话，并在当前协议、legacy 协议、degraded fallback 之间路由。 本地 terminal IPC、worktree teardown、Manage Sessions、应用退出/更新。

对外接口清单

接口 来源 用途
new OrcaRuntimeService(store, stats, deps) orca-runtime.ts 创建主进程 runtime；deps.getLocalProvider 必须延迟解析 daemon provider。
getRuntimeId() / getStartedAt() / getStatus() orca-runtime.ts 供 RPC error、metadata、status client 和调试面板识别当前 runtime。
attachWindow() / syncWindowGraph() orca-runtime.ts 接收 renderer 发布的 tab/leaf/PTY 图，成为 terminal 查询、移动端投屏和 headless 状态的依据。
registerPty() / onPtyData() / subscribeToTerminalData() orca-runtime.ts 维护 PTY 到 terminal handle、移动端输出流和 agent 状态检测的热路径。
new OrcaRuntimeRpcServer(options) runtime-rpc.ts 构造 runtime 控制面；内部创建 RpcDispatcher 并绑定同一个 runtime 实例。
start() / stop() runtime-rpc.ts 启动/停止 socket 和 WebSocket transport，启动成功后写 runtime metadata。
createPairingOffer() runtime-rpc.ts 生成带 endpoint、device token、E2EE public key 和 scope 的 pairing URL；可返回 web client URL。
initDaemonPtyProvider() daemon-init.ts 启动或复用 daemon，创建 current adapter，发现 legacy session，并安装为 local PTY provider。
restartDaemon() daemon-init.ts 执行 7 步重启：先 fanout synthetic exits，再解绑监听器、清理当前 daemon、复用 spawner、替换 provider、重新绑定。
cleanupDaemonForProtocol() daemon-init.ts 按协议版本关闭 daemon、统计被杀 session，必要时 fallback 到 PID 级清理。
MOBILE_PAIRING_USERDATA_FILES src/main/runtime/mobile-pairing-files.ts 把 userData 中的 device registry 文件与 E2EE keypair 文件绑定迁移，避免 token 与密钥分裂。

内部分层与关键依赖

关系 说明 读源码入口
index.ts → runtime 构造 runtime，并把 agent hook、账号、stats、store 注入进去。 src/main/index.ts 创建 runtimeService 附近。
runtime → PTY provider runtime 通过 thunk 获取当前 provider，避免 daemon 初始化后 provider 替换失效。 OrcaRuntimeService.constructor 与 getLocalProvider()。
RPC server → runtime RPC server 不直接实现业务，认证后把请求交给 RpcDispatcher，dispatcher 再调 runtime 或 method 模块。 OrcaRuntimeRpcServer.handleMessage()。
WebSocket → mobile WebSocket channel 经过 E2EE 握手和 device token 验证，mobile scope 还要经过 method allowlist。 handleWebSocketMessage() 与 MOBILE_RPC_METHOD_ALLOWLIST。
daemon-init → ipc/pty 安装 DaemonPtyAdapter / DaemonPtyRouter 后调用 setLocalPtyProvider()，让既有 IPC 路径透明使用 daemon。 initDaemonPtyProvider() 与 replaceDaemonProvider()。

简单调用示例

const server = new OrcaRuntimeRpcServer({
  runtime,
  userDataPath: getCanonicalUserDataPath(),
  enableWebSocket: true
})

await server.start()
const offer = server.createPairingOffer({
  address: '100.64.1.20',
  name: 'CLI test',
  scope: 'runtime'
})

这个示例对应 runtime-rpc.test.ts 的 pairing 测试：server 启动后，metadata 中已有 transport 和 auth token； pairing URL 会携带 endpoint、device token、public key 和 scope。若传入完整 wss://runtime.example.com/orca， 测试还验证 web client URL 会保留 proxy path 前缀。

行为测试佐证

行为 测试文件 锁定的不变量
long-poll keepalive src/main/runtime/runtime-rpc.test.ts orchestration.check 和 terminal.wait 长等待期间会发送 {"_keepalive": true} frame。
long-poll cap src/main/runtime/runtime-rpc.test.ts cap 满时返回 runtime_busy，短 RPC 仍可执行，关闭中的 long-poll 会释放槽位。
pairing endpoint src/main/runtime/runtime-rpc.test.ts IPv6、host:port、完整 wss URL 和 proxy path 都会被格式化进 pairing URL。
daemon restart 顺序 src/main/daemon/daemon-init.test.ts synthetic exits 必须在解绑监听器前发出，provider 替换后再 rebind。
legacy / degraded provider src/main/daemon/daemon-init.test.ts 旧协议 session 保留原 adapter；degraded current daemon 的 session 也会在重启时 fanout exit。
