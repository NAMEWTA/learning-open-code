Worktree 创建链路总览参考

定位：L2 跨模块切片。配合 导览短课 使用。

端到端调用链

阶段 核心文件 行为 不变量
modal 入口 src/renderer/src/components/NewWorkspaceComposerModal.tsx 根据 store 的 active modal 展示新建界面，选择 quick agent，提交到 composer hook。 modal 是薄壳，不直接触碰 main IPC。
请求组装 src/renderer/src/hooks/useComposerState.ts 解析 workspace name、base、linked review、setup trust、sparse checkout、Agent startup 和 telemetry。 所有需要用户参与的决策必须在关闭 modal 前完成。
pending 请求模型 src/renderer/src/lib/pending-worktree-creation.ts 定义可重试的 WorktreeCreationRequest 和 session-only pending entry。 请求保持 plain-serializable，Retry 复用同一份事实来源。
后台创建 src/renderer/src/lib/worktree-creation-flow.ts 生成 creationId，显示 pending surface，调用 store create，成功后激活或后台 seed terminal。 用户离开 pending surface 后，创建完成不能抢回焦点。
终端交接 src/renderer/src/lib/worktree-activation.ts、src/renderer/src/lib/new-workspace.ts 激活新 worktree、创建/复用初始 terminal，并在 backend 未 spawn 时投递 Agent startup plan。 startupTerminal.spawned 为真时，renderer 不再重复启动 Agent。
VM 前置准备 src/renderer/src/lib/ephemeral-vm-worktree-creation.ts 处理 VM recipe provisioning、runtime repo 替换、workspace attach 和失败 cleanup。 这是创建前旁支；最终仍回到同一个 store create 调用。
renderer store 分流 src/renderer/src/store/slices/worktrees.ts 构造 CreateWorktreeArgs，按 active runtime target 走本地 preload IPC 或 runtime RPC。 最多重试 25 个候选名；watcher 先发现 worktree 时必须去重合并。
preload 桥 src/preload/index.ts 暴露 worktrees.create()、prefetchCreateBase() 和 onCreateProgress()。 renderer 只能通过受控 API 跨进程，不直接访问 Electron main。
进度事件消费 src/renderer/src/hooks/useIpcEvents.ts 监听 createWorktree:progress，按 creationId 更新对应 pending entry。 没有 correlation id 的旧事件不能覆盖并发创建的状态。
main IPC 入口 src/main/ipc/worktrees.ts 校验 repo，解析 telemetry/provenance，分流到 folder/local/SSH helper。 失败要释放 automation provenance，并发出 create failed telemetry。
Git 创建 helper src/main/ipc/worktree-remote.ts 本地或 SSH 创建 worktree，写 metadata/lineage，准备 setup/default tabs；本地 helper 还可 backend-spawn startup terminal。 desktop SSH helper 不消费 startup；Git 成功后的 setup/startup 问题不能伪装成 Git 创建失败。
runtime RPC 入口 src/main/runtime/rpc/methods/worktree.ts worktree.create 把 mobile/CLI/remote runtime 请求映射到 runtime service。 RPC 参数要保留 linked review、setup、startup、lineage 和 automation provenance。
runtime managed create src/main/runtime/orca-runtime.ts 为 CLI、mobile 和 runtime host 提供同语义创建路径，并在 managed local/SSH 路径处理 activation/setup/startup。 runtime 已 spawn 的 terminal 不应被 renderer 或 RPC client 再 spawn 一次。

共享契约

文件 契约 读法
src/shared/types.ts CreateWorktreeArgs、CreateWorktreeResult、WorktreeStartupLaunch 先看参数和返回值，而不是先看 UI；这能确定哪些字段真的跨进程传输。
src/shared/worktree-id.ts repoId::worktreePath 解析、filesystem path 归一化、folder workspace suffix 去除。 任何拿到 worktree id 后要访问磁盘的代码，都要分清 UI id 和 filesystem path。

请求字段怎么流动

字段族 产生位置 消费位置 作用
身份与路径 repoId、name、displayName store、main helper、metadata 决定 worktree 路径 seed、可读 display name 和最终 worktree id。
Git 起点 baseBranch、compareBaseRef、branchNameOverride resolveWorktreeCreateBase()、branch/path resolver、metadata 分开“从哪里 checkout”和“Source Control 对比哪里”。
review/linkage linkedIssue、linkedPR、GitLab/Bitbucket/Azure/Gitea 字段 metadata、push target、review conflict 检测 让创建出的 workspace 继承 issue/PR/MR 语境。
setup 与 sparse setupDecision、sparseCheckout hook policy、Git helper、setup runner 决定是否运行 setup，以及是否创建 sparse worktree。
Agent 启动 startup、startupPlan、quickPrompt backend startup、renderer terminal handoff 自包含命令可由 backend 先 spawn；需要 prompt 投递的命令由 renderer 接手。
进度关联 creationId main progress event、renderer pending entry 并发创建时把 fetching/creating 更新回正确 panel。

base ref 与 Git 创建

问题 源码 规则
用户显式选 base src/main/worktree-create-base.ts 直接返回，不探测默认值和持久化值。
repo 有持久化 base src/main/worktree-create-base.ts 如果可用则保留；陈旧不可用时回退检测到的默认 base。
base 缺失 src/main/ipc/worktree-remote.ts 不伪造 origin/main，而是抛出清晰错误，要求用户显式选择 base。
本地 add src/main/git/worktree.ts 新分支走 git worktree add --no-track -b branch path base，并写 branch base 与 push ergonomics。
SSH add src/main/ipc/worktree-remote.ts 通过 relay provider 在远端执行 add，并在 sparse 失败时回滚刚创建的 worktree/branch。

进度和失败语义

事件/状态 来源 renderer 行为
preparing renderer staged preflight 先显示 pending row，异步 preflight 完成后替换 request。
provisioning-vm ephemeral VM recipe 显示 provider provisioning，失败时可清理 VM runtime。
fetching local main helper 或 retry 初始 phase 表示本地创建正在刷新 base ref 或准备进入 Git 创建；SSH/runtime 可能没有该细分事件。
creating local main helper 表示本地创建即将或正在执行 git worktree add；SSH/runtime 可使用 indeterminate 进度。
status: error background flow catch 错误留在同一个 pending surface；用户离开时额外 toast，用户正在看 panel 时不重复提醒。

读源码时的常见误判

误判 更准确的说法
modal 关闭表示 workspace 已经创建完成。 modal 关闭只表示请求已经交给后台 pending flow；Git 可能还在 fetch 或 add。
所有创建都走 main IPC。 local target 走 preload/main IPC；runtime target 走 RPC worktree.create。
desktop SSH helper 也会直接启动 Agent terminal。 desktop SSH helper 只创建远端 worktree 并返回 setup/default tabs；managed SSH runtime 在 runtime 包装层启动 terminal。
setup hook 失败就表示 worktree 创建失败。 Git 创建成功后，setup runner 或 terminal 生成失败通常以 warning/setup payload 形式返回，不能抹掉真实 worktree。
renderer 总是负责启动 Agent。 自包含 startup 可由 backend 先 spawn；需要 draft/follow-up 的启动才留给 renderer handoff。
