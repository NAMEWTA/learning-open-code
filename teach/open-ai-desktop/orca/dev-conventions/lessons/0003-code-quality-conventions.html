Orca 的代码质量三支柱：注释、错误、跨平台

前两课讲了怎么写代码、放哪里、叫什么。本课讲三个影响代码可维护性的深层约定：注释怎么写、错误怎么处理、平台差异怎么隔离。 完整速查见 参考手册代码质量章节。

1. 注释铁律：只写"为什么"，不写"是什么"

Orca 的注释哲学极其精简——一行 // Why: 解释非显而易见的决策原因。绝不复述代码做了什么。 

// ✅ 正确：解释约束原因
// Why: forward-slash UNC roots need win32 ops; POSIX joins collapse //Server to /Server.
const pathOps = isWindowsAbsolutePathLike(root) ? win32Path : posixPath

// Why 5 minutes: GC runs ~10s after startup. Without an age guard,
// GC would delete its freshly-created history directory (TOCTOU race).
const HISTORY_GC_MIN_AGE_MS = 5 * 60 * 1000

// Why: quitAndInstall is a side effect. React StrictMode double-invokes
// render functions — calling it inline would trigger a double-install.
const handleQuitAndInstall = useCallback(() => quitAndInstall(), [])
记忆公式：// Why: <非显而易见的约束/安全/trade-off 原因>。 不超过两行。如果你发现自己写了三行解释，说明这段代码需要拆分而非加注释。

注释的三种合法形态：

形态用途示例
// Why:解释非显而易见的决策// Why: /d disables AutoRun so background spawns can't inherit side effects
/** 一句话 */导出函数的用途描述（JSDoc）/** Resolve the shell kind from a shell binary path. */
// ─── Section ───长文件内部的视觉分隔线// ─── Garbage Collection ───

绝对不写：复述代码的注释（如 // increment counter）、机制叙述（如 // first we check if x is null, then we...）、设计文档引用（// see design doc section 3.2）。

2. 错误处理：每个领域管理自己的 Error 类

Orca 没有集中式的错误基类或 errors/ 目录。每个领域在需要时自行定义 Error 类，遵循统一模板： 

// 模板：所有自定义 Error 类都这样写
export class RemoteRuntimeClientError extends Error {
  readonly code: string                    // ← 必有字段

  constructor(code: string, message: string) {  // ← 参数签名固定
    super(message)
    this.name = 'RemoteRuntimeClientError'       // ← 必须设置 name
    this.code = code
  }
}

项目中至少存在 20+ 个这样的 Error 类：BrowserError、DaemonProtocolError、SessionNotFoundError、CredentialDecryptionError、JiraApiError、StreamProtocolError、RelayVersionMismatchError……

每种 Error 类通常伴随一个类型守卫函数，用于调用方安全判断：

export function isClipboardTextTooLargeError(error: unknown): boolean {
  return error instanceof ClipboardTextTooLargeError
}
export function isPermissionError(error: unknown): boolean {
  return error instanceof Error && 'code' in error && error.code === 'PERMISSION_DENIED'
}
判断法则：需要根据错误类型做不同处理 → 定义 Error 子类 + 守卫函数。不需要区分 → 直接用 throw new Error('描述')。

3. 跨平台：运行时检测 + 依赖注入，禁止文件后缀分裂

Orca 同时支持 macOS、Linux、Windows。但没有 .mac.ts、.win.ts、.linux.ts 文件后缀。所有平台逻辑通过三种方式管理：

策略适用场景示例
领域文件分离整个功能仅在某平台存在win32-utils.ts、linux-lid-sleep-assertion.ts、macos-system-sleep-assertion.ts
运行时 process.platform同功能不同平台有分支if (process.platform === 'win32') { ... }
依赖注入 platform 参数需要测试不同平台分支constructor({ platform = process.platform })，测试可传 'darwin'

路径处理特别规则：涉及跨平台路径的文件需要同时导入 posix 和 win32，通过路径形状检测（而非 process.platform）选择操作：

import * as posixPath from 'node:path/posix'
import * as win32Path from 'node:path/win32'

function getPathOps(p: string): typeof posixPath {
  return isWindowsAbsolutePathLike(p) ? win32Path : posixPath
}

快捷键规则（AGENTS.md 明文规定）：

  • 键盘事件：navigator.userAgent.includes('Mac') 判断 metaKey vs ctrlKey
  • Electron 菜单：统一使用 CmdOrCtrl
  • UI 快捷键标签：Mac 显示 ⌘/⇧，其他平台显示 Ctrl+/Shift+

4. 检索练习

以下代码片段在 Orca 项目中有几处不符合规范？

// Check if the user is on Windows and handle accordingly
// First we get the platform, then we check if it equals 'win32',
// and if so we call the Windows-specific handler
function handlePlatformSpecific(filePath: string) {
  if (process.platform === 'win32') {
    // Call win32 handler
    return handleWin32(filePath)
  } else if (process.platform === 'darwin') {
    return handleDarwin(filePath)
  }
  // For linux, we just use the default path
  return filePath
}
至少 3 处不符合：
① 第一个注释 "Check if the user is on Windows and handle accordingly" 是复述代码——应删除。代码本身已经自解释。
② 第二段注释 "First we get the platform..." 是机制叙述——应删除。如果这里有非显而易见的约束（如为什么要区分处理），才应该用 // Why: 解释。
③ // Call win32 handler 和 // For linux, we just use the default path 是废话注释——应删除。
④ 没有违反跨平台规范本身（运行时用 process.platform 是可接受的），但如果需要测试不同平台分支，应该用依赖注入模式。
⑤ 函数名 handlePlatformSpecific 太模糊——应改为具体描述（如 resolveShellPathForPlatform）。
总结：Orca 的注释哲学是"少即是多"。如果代码已经说清楚了，加注释反而是噪音。
