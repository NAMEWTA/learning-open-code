Orca 的三层规范体系

本课帮你建立 Orca 项目的规范心智模型——代码怎么写、文件放哪里、东西叫什么。读完你就能在 6000+ 源码文件中快速定位、判断代码是否合规范。 完整规则速查见 开发规范参考手册。

1. 工具链：oxlint + oxfmt，没有 ESLint/Prettier

Orca 使用 oxlint 做 lint、oxfmt 做格式化，没有 .eslintrc 也没有 .prettierrc。这套 Rust 工具链比传统方案快 50-100 倍。

核心记忆：lint 规则定义在 .oxlintrc.json，格式规则定义在 .oxfmtrc.json，pre-commit hook 自动执行。
维度规则来源
引号单引号 'hello'.oxfmtrc.json
分号不加分号.oxfmtrc.json
行宽100 字符.oxfmtrc.json
尾逗号不使用 trailing comma.oxfmtrc.json
type vs interface只用 type，禁止 interface.oxlintrc.json
any 类型禁止显式 any（rest args 除外）.oxlintrc.json
类型导入必须 import type.oxlintrc.json
Node 协议必须 node:fs 而非 fs.oxlintrc.json
Switch必须穷举所有分支，禁止 default fallback.oxlintrc.json
行数上限.ts 300 / .tsx 400 / .mjs 600 / test 800.oxlintrc.json

提交前 .husky/pre-commit 会通过 lint-staged 对暂存文件依次执行 oxlint → oxlint-react-doctor → oxfmt。CI 中还有 .d.ts 守卫：src/preload/ 和 src/shared/ 禁止项目自有 .d.ts（仅允许 ambient shim）。

2. 目录规范：按运行环境分层的 src/

Orca 的 src/ 不是按"功能"分，而是按运行环境分。这是理解目录结构最关键的心智模型。 

src/
├── main/          ← Electron 主进程（Node 环境，系统调用、IPC、daemon、PTY）
│   ├── index.ts      入口：app 生命周期、服务装配、窗口创建
│   ├── ipc/          IPC handler 注册 → 连接 preload 暴露的 API
│   ├── runtime/      运行时编排：agent 生命周期、RPC 控制面
│   ├── daemon/       后台 daemon 进程：持久化 PTY 宿主
│   ├── git/          Git 操作封装
│   └── ...（70+ 功能域目录，每个目录 ≤ 10 个文件）
├── renderer/      ← Electron 渲染进程（浏览器环境，React UI）
│   └── src/
│       ├── App.tsx        React 根组件
│       ├── components/    功能域组件（sidebar、terminal、agent…）
│       ├── components/ui/ shadcn 原语（button、dialog、tabs…）
│       ├── store/         Zustand store + slices
│       ├── hooks/         React hooks
│       └── assets/main.css  设计 token（规范唯一来源）
├── preload/       ← contextBridge 安全桥（仅暴露审计过的 API）
├── shared/        ← 纯逻辑与类型（无副作用，main + renderer + CLI 共用）
├── cli/           ← 独立 CLI 工具（Node 环境，RuntimeClient 通信）
├── relay/         ← SSH 远端 agent 执行器（独立 Node 进程）
└── types/         ← 仅 ambient shim 声明（build-constants.d.ts）
判断法则：产生系统副作用的代码进 main，纯 UI 进 renderer，跨端共享的类型/工具进 shared，远端执行进 relay，脚本入口进 cli。

3. 命名规范：拒绝模糊，拥抱具体

Orca 的命名哲学只有一句话——文件名必须准确描述它包含的概念。这条来自 AGENTS.md 的规则被 lint 强制执行。

类别✅ 正确❌ 错误
文件tab-group-state.tstabs-helpers.ts
文件terminal-orphan-cleanup.tsterminal-utils.ts
文件agent-awake-service.tsagent-common.ts
类class AgentAwakeService—
函数detectAgentStatus()check()
常量STALE_AFTER_MStimeout
布尔isQuitting、hasAgentNameflag、state
类型type AgentStatus = 'working' | 'idle'interface IAgentStatus

禁用词黑名单：helpers、utils、common、misc、shared-stuff。这些词不携带任何信息，会变成代码垃圾场。如果你发现自己想用它们，说明文件职责过多，应该拆分。

命名格式约定：

  • 文件：kebab-case（worktree-linear-issue-link.ts）
  • 类 / React 组件 / 类型别名：PascalCase（KeybindingService、SidebarToolbar）
  • 函数 / 变量：camelCase（getAgentLabel、persistedUIReady）
  • 原始常量：SCREAMING_SNAKE_CASE（AGENT_AWAKE_STATUS_STALE_AFTER_MS）
  • React 组件文件：PascalCase 或 kebab-case 均可，组件名为 PascalCase

4. 检索练习

凭记忆判断以下代码片段有几处违反 Orca 规范：

// file: src/main/agent-helpers.ts
import { readFile } from 'fs';
import { AgentStatus } from '../shared/types';

interface AgentInfo {
  name: string;
  status: any;
}

export function check(info: AgentInfo): boolean {
  if (info.status == 'working') {
    return true;
  }
  return false;
}
至少 6 处违规：① 文件名 agent-helpers.ts 用了禁用词且不描述具体概念；② 应写 import { readFile } from 'node:fs'；③ 应写 import type { AgentStatus }；④ 禁止 interface，应写 type AgentInfo = { ... }；⑤ 禁止显式 any，status: any 应改为具体联合类型；⑥ 函数名 check 过于模糊；⑦ 缺少分号——等一下，Orca 实际上不加分号，所以这不算违规。但 == 应改为 ===（unicorn 规则会要求严格相等）。

课后建议通读 开发规范参考手册 获取完整规则清单，并在 IDE 中安装 oxlint 插件获得实时反馈。如有疑问可向 agent 追问具体规则的适用场景。
