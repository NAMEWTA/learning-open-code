Orca 目录结构的五条铁律

上一课讲了 src/ 按运行环境分层的顶层设计。本课深入每层内部，回答一个问题：当你需要在 Orca 中新建文件时，它应该放在哪、和谁做邻居？ 完整目录地图见 目录结构速查手册。

1. 铁律一：扁平领域目录，禁止深层嵌套

Orca 不使用 src/main/services/agent/status/... 这种嵌套模块树。所有领域目录平铺在一层，按业务概念直接命名。 

src/main/
├── agent-hooks/    ← agent hook 安装与生命周期
├── browser/        ← 浏览器标签页管理
├── daemon/         ← 后台 daemon 进程
├── git/            ← Git 操作
├── ipc/            ← IPC handler 注册（180+ 文件，仍扁平）
├── keybindings/    ← 快捷键
├── pty/            ← 伪终端
├── runtime/        ← 运行时编排（唯一含深嵌套的例外）
│   └── rpc/methods/  ← ~40 个 RPC 方法，因数量过大才拆一层
├── ssh/            ← SSH 连接
├── terminal/       ← 终端
└── ...（共 72 个平铺目录 + 113 个散文件）
心智模型：如果你在 main 下建了 foo/bar/baz/ 三层目录，99% 的概率你违反了规范。嵌套仅用于 runtime/rpc/methods 这类"子方法数量过多必须拆分"的场景。

2. 铁律二：测试文件与源文件同居

测试文件 绝不放在单独的 __tests__/ 目录，而是紧挨源文件。整个 src/ 树只有 1 个 __tests__/（在 providers/ 下，属于历史遗留）。 

src/shared/
├── workspace-name.ts
├── workspace-name.test.ts      ← 紧挨源文件
├── agent-detection.ts
├── agent-detection.test.ts     ← 紧挨源文件
src/main/ipc/
├── pty.ts
├── pty.test.ts                 ← 紧挨源文件
src/main/git/
├── status.ts
├── status.test.ts              ← 紧挨源文件

特殊测试文件通过后缀区分的意图：

后缀含义示例
.test.ts单元测试agent-detection.test.ts
.integration.test.ts集成测试ssh-relay-session-agent-hooks.integration.test.ts
.benchmark.test.ts性能基准store-snapshot.benchmark.test.ts
.repro.test.tsBug 复现issue-4631-terminal-hover.repro.test.ts
-test-harness.ts测试辅助工具（非测试本身）agent-hooks/server.test.ts 使用的 test-harness.ts

3. 铁律三：index.ts 只用在这四种场景

Orca 不为每个目录自动建 barrel 文件。index.ts 只出现在以下场景，其余目录直接从具名文件导入：

场景示例原因
进程入口src/main/index.ts、src/cli/index.tsapp 生命周期、服务装配
聚合注册runtime/rpc/methods/index.ts收集 ~25 个 RPC 方法到一个数组
隐藏内部结构ghostty/index.ts、warp-themes/index.ts对外暴露公共 API，隐藏内部 4-5 个实现文件
状态组合renderer/src/store/index.tsZustand create 合并 ~20 个 slice
判断法则：如果你准备新建 index.ts 只为了少写几个 import 路径——别建。直接按文件名导入即可。

4. 铁律四：components/ 分两层的唯一理由

src/renderer/src/components/ 下只有两种目录，按是否承载业务逻辑区分： 

components/
├── ui/                  ← 纯 UI 原语（shadcn 风格）：button、dialog、select…
│   │                     无业务逻辑、无 store 依赖、可跨项目复用
│   └── （35 个文件，全是 forwardRef + cva 变体）
├── terminal/            ← 业务领域组件：含 hooks、store 交互、xterm 绑定
├── settings/            ← 业务领域组件：含设置持久化、账户管理
├── agent/               ← 业务领域组件：含 agent 状态展示
├── sidebar/             ← 业务领域组件：含项目树、拖拽、右键菜单
└── ...（共 45 个平铺业务目录）

如果你在写一个包含 useAppStore 或 ipcRenderer 调用的组件，它必须放在业务领域目录；如果是一个纯受控的按钮/弹窗/选择器，放 ui/。

5. 铁律五：shared/ 是扁平文件池

src/shared/ 是 Orca 最极端的扁平实践——542 个 .ts 文件平铺在一个目录下，除了一个 network/ 子目录外零嵌套。 

src/shared/
├── agent-detection.ts           ← 文件名 = 概念名
├── agent-detection.test.ts
├── agent-status-types.ts
├── browser-url.ts
├── clipboard-text.ts
├── constants.ts
├── keybindings.ts
├── remote-runtime-client.ts
├── runtime-types.ts             (24.5 KB)
├── telemetry-events.ts
├── terminal-custom-themes.ts
├── types.ts                     (138 KB — 最大文件)
├── workspace-name.ts
├── network/                     ← 唯一的嵌套目录（4 个文件）
│   ├── manual-address.ts
│   └── server-share-address.ts
└── ...（共 542 个 .ts 文件）

文件以描述性前缀自然分组（terminal-*、git-*、agent-*、worktree-*），不需要子目录。新增共享类型时，直接在 shared/ 下建一个 <概念名>.ts 文件。

6. 检索练习

判断以下场景分别违反了几条铁律：

  1. 你在 src/main/ 下创建 services/terminal/shell/ 三层目录来组织终端 shell 相关代码。
  2. 你在 src/shared/ 下新建 __tests__/ 目录来放 agent-detection.test.ts。
  3. 你在 src/main/agent-hooks/ 下新建 index.ts，导出该目录所有 5 个文件的函数，方便其他模块一行 import。
A：违反铁律一。 三层嵌套等同于引入不必要的分类层级。应在 src/main/ 下直接建一个 terminal-shell/ 平铺目录。

B：违反铁律二。 测试必须紧挨源文件，放在 src/shared/agent-detection.test.ts。

C：不完全违规但不符合惯例。 agent-hooks 只有 5 个导出的公共函数，每个导入方可以精确 import 所需文件，barrel 只是增加了一个需要维护的中间层。如果 agent-hooks 对外暴露的是一个复杂内部结构需要隐藏，才考虑建 index.ts（铁律三的场景 3）。
