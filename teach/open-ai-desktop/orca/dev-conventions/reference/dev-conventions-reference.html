六、文件命名禁用词

以下词不得出现在文件名、目录名或模块名中：

  • helpers
  • utils
  • common
  • misc
  • shared-stuff

替代策略：以文件所操作的具体概念命名，如 tab-group-state.ts、terminal-orphan-cleanup.ts。

七、TypeScript 项目约定

  • 类型声明：项目自有类型放在 .ts 文件，不用 .d.ts。仅 ambient shim 可用 .d.ts（如 env.d.ts、build-constants.d.ts）。
  • 路径别名：renderer 中 @/ 和 @renderer/ 映射到 src/renderer/src/。
  • 类型检查：使用 tsgo（TypeScript 原生编译器的 Go 移植版），通过 3 个 tc config 并行检查。
  • 导出方式：优先命名导出（named export）。React 组件可使用默认导出。

八、src/ 目录职责速查

目录运行环境职责判断信号
src/main/Electron 主进程 (Node)系统调用、IPC、daemon、PTY、Git、窗口代码中有 app、BrowserWindow、ipcMain

十一、代码注释规范

注释类型格式用途
Why 注释// Why: <原因>解释非显而易见的决策（安全约束、兼容 shim、trade-off）
JSDoc/** 一句话描述 */导出函数的用途（只写功能，不写实现）
分节线// ─── Section Name ───长文件内部视觉分隔

禁止：复述代码的注释、机制叙述注释、设计文档引用注释。不超过两行。

lint disable 必须加 Why：/* eslint-disable max-lines -- Why: <原因> */

十二、错误处理规范

规则说明
自定义 Error 类模板export class XxxError extends Error { readonly code: string; constructor(code: string, message: string) { ...; this.name = 'XxxError'; } }
错误守卫函数export function isXxxError(error: unknown): boolean { ... } — 与 Error 类成对出现
无集中错误基类每个领域自行定义 Error 类，没有 errors/ 目录
通用错误不需要区分的场景用 throw new Error('描述')

十三、跨平台开发规范

策略适用场景示例
领域文件分离整个功能仅在某平台存在win32-utils.ts、macos-system-sleep-assertion.ts
process.platform同功能不同平台有分支if (process.platform === 'win32')
依赖注入 platform需要测试不同平台分支this.platform = options.platform ?? process.platform
路径形状检测跨平台路径操作同时导入 node:path/posix 和 node:path/win32，按路径形状选择

快捷键：键盘事件用 navigator.userAgent.includes('Mac') 判断；Electron 菜单用 CmdOrCtrl；UI 标签 Mac 显示 ⌘，其他平台显示 Ctrl+。

禁止：文件后缀分裂（.mac.ts、.win.ts）、硬编码 e.metaKey、路径中写死 / 或 \。

十四、React 组件规范速查

规则说明
导出方式页面级用 export default function，工具组件用 export function
Props 类型内联解构 { prop }: { prop: Type }，仅复用类型时才提取为命名 type
返回类型必须显式标注 React.JSX.Element 或 React.JSX.Element | null
React.memo仅用于性能敏感的大列表项（WorktreeCard、Sidebar 等）
Hook 顺序useState → useAppStore → useMemo → useCallback → useEffect → 自定义 hooks
子组件不导出的内部组件留在同文件，不单独提取

十五、Zustand 状态管理规范速查

规则说明
Store 创建create<AppState>()((...a) => ({ ...createXxxSlice(...a) })) — 单 store 多 slice
Slice 文件导出 createXxxSlice(...a) 函数，返回含 state + actions 的对象
组件使用useAppStore((s) => s.xxx)，数组/对象用 useShallow
缓存选择器热路径派生数据用 WeakMap 按 store 引用缓存
状态更新仅通过 slice 定义的 action 函数，组件不直接调用 setState

十六、测试规范速查

规则说明
框架Vitest（import { describe, expect, it, vi } from 'vitest'）
描述风格describe('ModuleName', () => { it('does X when Y', () => { ... }) })
测试数据工厂函数 makeXxx()，每次创建新实例，不共享可变 fixture
平台 MockObject.defineProperty(process, 'platform', { value: 'win32' }) + vi.resetModules()
Store 测试createTestStore() 创建完整 store，seedStore(store, state) 填充状态
生命周期极少使用 beforeEach/beforeAll，setup 内联在测试中保持可见

十七、贡献与 PR 规范速查

规则说明
分支命名fix/、feat/、chore/ + kebab-case（如 fix/ctrl-backspace-delete-word）
提交前检查必须通过 pnpm lint、pnpm typecheck、pnpm test、pnpm build
PR 必须包含摘要、截图/录屏、checklist、AI review report（跨平台/SSH/agent 兼容）、security audit
禁止提交 PR 截图到仓库、在 PR 中 bump 版本号、使用 gh-attach
