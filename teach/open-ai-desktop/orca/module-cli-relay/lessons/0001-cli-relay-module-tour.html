CLI 与 Relay：Orca 的脚本入口和远端执行边界

L0 已把 Orca 分成 renderer、preload、main、CLI、relay、mobile 和 shared。本文只看其中一条边界： src/cli 让用户或 agent 用命令调用 Orca；src/relay 让远端 SSH 主机保留 PTY、文件、Git 和 hook 能力。完整索引见 CLI 与 Relay 参考。

1. 两个入口，两种位置

CLI 在本地 Node 进程中启动。src/cli/index.ts 先处理特殊命令，例如 agent-teams-tmux 和 claude-teams，再解析参数、校验命令、创建 RuntimeClient，最后把命令路径交给 dispatch。Relay 则是部署到远端主机的独立 Node 进程，src/relay/relay.ts 用 stdin/stdout 或 Unix socket 与桌面端通信。

2. CLI 的内部层次

src/cli/ ├── index.ts ← 入口：parse/help/RuntimeClient/特殊 shim ├── specs/ ← 命令规格：哪些命令和 flag 合法 ├── dispatch.ts ← commandPath → handler ├── handlers/ ← 实际命令动作 └── runtime/ ← 连接桌面或 headless runtime

读 CLI 时先找命令规格，再找 handler。dispatch.ts 明确把 core、project、repo、worktree、file、terminal、browser、computer、orchestration、linear、vm、emulator 等 handler group 汇总成一张表；重复注册会直接抛错。

3. Relay 的内部层次

Relay 的关键不是“远端执行一个脚本”，而是保持一条可恢复的协议通道。protocol.ts 定义 13 字节 frame header、handshake、keepalive 和 16 MB payload 上限；dispatcher.ts 将 frame 解码成 JSON-RPC 请求、响应或通知，并维护 client id、ack、generation、超时和断开状态。 

orca browser click
→ parseArgs / validateCommandAndFlags
→ RuntimeClient
→ dispatch(commandPath)
→ browser handler 或 runtime RPC

4. 检索练习

判断题：如果 orca browser click 这类命令参数写错，错误应该优先在连接 runtime 之前暴露，还是连接之后由主进程暴露？

参考答案：优先在连接 runtime 之前暴露。src/cli/index.ts 的注释明确说明语法和 flag 错误应先由 CLI 校验，否则用户会看到误导性的“Orca 未运行”类错误。

课后主来源建议读 src/cli/index.ts、src/relay/relay.ts 和本主题参考文档。仍不清楚时，可以继续追问“某个 CLI 命令如何穿过 RuntimeClient 到 main 或 relay”。
