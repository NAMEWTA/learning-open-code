SSH Relay 会话保活参考

范围：renderer 启动恢复、main SSH/relay session、远端 relay daemon、PTY lease 与测试护栏。

terminal session 标记 src/renderer/src/store/slices/terminals.ts 水合时保留 pending reconnect worktree、tab 和 session id；SSH repo 的 remote session id 来自 session 的 remoteSessionIdsByTabId。

连接建立细节

src/main/ipc/ssh.ts 的 doConnect() 先检查已有 ready session。如果 SSH 状态仍 connected、mux 未 disposed、 且没有 relay override 或 backoff，它只刷新 renderer 状态，不拆旧 session。否则先 detach 旧 session， 再创建新的 SshRelaySession。 

doConnect(targetId)
  -> connectionManager.connect(target)
  -> broadcast deploying-relay
  -> session.establish(conn, grace)
  -> broadcast connected after session ready
  -> persist lastRequiredPassphrase

SshRelaySession.establish() 和 reconnect() 共享 provider 注册序列。它们都会先调用 deployAndLaunchRelay()，创建 SshChannelMultiplexer，用 session.resolveHome 进行 round-trip 健康检查，然后注册 PTY、filesystem、Git provider。 健康检查必须早于 provider 注册，因为 --connect bridge 可能已经关闭，不能让 session 误进入 ready。

relay 部署与 socket 复用

src/main/ssh/ssh-relay-deploy.ts 会把 relay 安装到 content-hashed 版本目录，并用 target id 生成 target-specific socket。 POSIX 上先用 test -S probe socket；如果 alive，直接 relay.js --connect。 如果 probe dead 或握手失败，才删除 stale socket 并 fresh launch detached relay。 

socket alive
  -> node relay.js --connect --sock-path relay.sock
socket dead
  -> nohup node relay.js --detached --grace-time N --sock-path relay.sock &
  -> poll socket READY
  -> node relay.js --connect --sock-path relay.sock

Windows 不能像 Unix socket 那样 unlink occupied named pipe，所以部署层维护 active pipe marker； 当旧 pipe 连接失败时，会用确定性的 fallback pipe relaunch relay，并把 marker 写回。相关测试覆盖 fallback pipe。

版本握手和 sentinel

src/relay/relay-handshake.ts 在 dispatcher 接管 socket 之前处理 handshake frame。 --connect bridge 发送当前 relay package version，daemon 返回 ok 或 mismatch。 只有 handshake 通过后，bridge 才向主进程 stdout 写 RELAY_SENTINEL，让 waitForSentinel() 放行 multiplexer。版本不匹配会让 bridge 以 exit 42 退出，并由 src/main/ssh/ssh-relay-deploy-helpers.ts 映射为 RelayVersionMismatchError； 但已有 socket 的 --connect 失败会先被 src/main/ssh/ssh-relay-deploy.ts 当成 stale socket/pipe 恢复机会处理，POSIX 路径会删除 stale socket 后 fresh launch，Windows 路径可能切换 fallback pipe。 只有最终仍把版本不匹配抛到 SshRelaySession 时，才会走 terminal relay error 并跳过普通 relay-lost backoff。

src/relay/protocol.ts 额外提供 FrameDecoder.drain()，保证 handshake ok 与 JSON-RPC frame 同包到达时， leftover bytes 不会丢。roundtrip 测试覆盖 daemon 侧和 bridge 侧 leftover。

relay 内部 client 模型

RelayDispatcher 有一个 primary client，也能 attach 多个 socket client。每个 client 独立维护 decoder、seq、ack 和 generation。通知会广播给所有 client；反向请求如 orca.cli 会通过 requestAnyClient() 优先选择真实 socket client，避免发给 detached relay 留下的 dead stdout。

当初始 stdin 结束或 socket client 关闭时，dispatcher 会 invalidate/detach 对应 client。 这会让 stale client 的请求变成 stale，防止 late pty.spawn 创建无人拥有的远端 shell。

grace 语义

信号 语义
relayGracePeriodSeconds 用户可配置目标级 grace。src/shared/ssh-types.ts 定义 0 表示不自动过期以及 min/max/default 常量；main session/deploy 路径负责把非 0 值 clamp 到 60 秒到 7 天，远端 relay 只接收已归一化的非负秒数。
fresh detached startup relay 没有 stdout client，会立即 start grace；第一个 socket client 接入后 cancel grace。
empty detached startup 如果没有 PTY 且从未接入 client，即使 grace 为 0，也用一个短启动窗口避免空 daemon 永久滞留。
host sleep prepareForHostSleep() 发送 relay.configureGraceTime，把 grace 设置为 0，保护 live PTY。
socket/stdin close 没有 client 时 start grace；grace 到期调用 shutdown，dispose dispatcher/PTY/FS/hook server 并清理 socket。

重连路径

有两条失败路径会回到同一个 session reconnect：

  • SSH transport 掉线：SshConnection 收到 end/close/error 后 schedule reconnect；重新 connected 且 reconnectAttempt = 0 时，src/main/ipc/ssh.ts 调 session.reconnect()。
  • relay bridge 掉线：SshChannelMultiplexer 的 transport close、write EPIPE 或 timeout 会 dispose 为 connection_lost；SshRelaySession.watchMuxForRelayLoss() 触发 relay-lost backoff。

session.reconnect() 会停止 port scanning、移除 port forwards、广播空列表、以 connection_lost teardown provider，然后重新 deploy/bridge/register/reattach。版本 mismatch 是 terminal error；普通失败保持 reconnecting 并让 src/main/ipc/ssh.ts 的 bounded backoff 接管。

PTY lease、reattach 和 replay

main 侧 src/main/ipc/pty.ts 在 SSH PTY spawn 成功后把 relay PTY id 写入 sshRemotePtyLeases。这个 lease 是重连索引，不是 PTY 本体。真正的 PTY 本体在远端 PtyHandler.ptys map 中，受 relay grace window 保护。

reconnect 时，SshRelaySession.reattachKnownPtys() 合并 durable lease 与当前 getPtyIdsForConnection()，通过 SshPtyProvider.attachForReconnect() 调 pty.attach。relay 侧如果 buffer 有内容，会返回最近 100KB replay；session 侧做 1 秒重复 replay 去重， 再发送 pty:replay 给 renderer。如果 attach 返回 not found，lease 标为 expired，local ownership 清掉， renderer 收到 pty:exit 后可以冷启动新 shell。

renderer 启动恢复分两层。src/renderer/src/App.tsx 先让无凭据目标 eager connect；passphrase 目标或 15 秒内未恢复的目标进入 deferred。src/renderer/src/store/slices/terminals.ts 把 session id 存到 deferredSshSessionIdsByTabId。 当 TerminalPane 挂载或用户聚焦时，src/renderer/src/components/terminal-pane/pty-connection.ts 再检查是否需要 prompt，等待用户主动 connect， 然后调用 transport.connect({ sessionId })。若 SSH_SESSION_EXPIRED，会清旧绑定并 fresh spawn。

workspace session 和 PTY 的持久性差异

src/relay/workspace-session-handler.ts 把 workspace session 写入远端用户主目录下的 Orca sessions JSON， 有 revision、atomic rename 和 presence TTL。 这份 JSON 能跨 relay 进程重启保留 UI session。远端 PTY replay buffer 不一样，它在 relay 进程内存中； relay 退出后，PTY 和 buffer 都不存在，只能靠 cold restore 或 fresh spawn。

调试入口

  • UI 一直停在 reconnecting：先查 src/main/ipc/ssh.ts 的 relay override/backoff，再查 SshConnection 的 reconnectAttempt。
  • SSH connected 但远端文件不可用：查 SshRelaySession.establish() 是否通过 health check 和 provider 注册。
  • 终端没有旧输出：查 lease 是否存在、pty.attach 是否返回 replay、renderer 是否走 deferred path。
  • 每次重启都 fresh shell：查远端 socket 是否 alive、handshake 是否 mismatch、grace 是否过期。
  • 多个 Orca 客户端状态不同步：查 RelayDispatcher socket client 数量、workspace revision 和 presence。
