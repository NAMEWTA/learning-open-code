SSH Relay Session Flow：远端终端为什么能活过断线

目标：15 分钟内能从 SSH connect 追到 detached relay、grace socket 和 PTY reattach。

先抓三段主路径

Orca 的 SSH 远端会话不是普通的 ssh command 长连接。主进程先建立 SSH transport， 再部署或连接远端 relay daemon，最后把 filesystem、Git、PTY provider 都挂到同一个 relay mux 上。 读源码时先抓这三个文件：src/main/ipc/ssh.ts 是用户入口和重连策略， src/main/ssh/ssh-relay-session.ts 是每个 target 的 session 状态机， src/relay/relay.ts 是远端 daemon 的 socket、grace 和 handler 装配。

renderer or user action └─ src/main/ipc/ssh.ts ← ssh:connect、state override、relay-lost backoff main session authority └─ src/main/ssh/ssh-relay-session.ts ← deploy、provider、reattach、mux loss watcher remote daemon └─ src/relay/relay.ts ← detached socket、--connect bridge、grace shutdown

连接：先 SSH，再 relay ready

ssh:connect 先通过 SshConnectionManager 得到可用 SSH transport。 但 renderer 不能只看 SSH 已 connected，因为这时远端 provider 可能还没注册。ssh.ts 会显式广播 deploying-relay，再调用 session 的 establish()。 session 内部部署 relay，创建 multiplexer，用 session.resolveHome 做健康检查， 然后注册 PTY、filesystem、Git provider，最后才进入 ready 并广播 connected。 

ssh:connect
  -> connectionManager.connect(target)
  -> status: deploying-relay
  -> session.establish(conn, target.relayGracePeriodSeconds)
  -> mux health check
  -> register providers
  -> reattach known PTYs
  -> status: connected

已经 ready 的 session 再次被窗口激活时不会被拆掉。ssh.ts 会直接广播现有状态， 避免刷新 BrowserWindow 时把活跃 relay、端口转发和远端终端意外断开。

远端：detached daemon 和 --connect bridge

部署层会先 probe 目标 socket。如果 socket 仍 alive，就执行 relay.js --connect --sock-path ...， 把新的 SSH exec channel 桥到旧 relay 进程。如果 socket 不可用，才 fresh launch 一个 detached relay： POSIX 用 nohup ... --detached --grace-time ... &，Windows 用 WMI 重新挂载进程树。 fresh launch 后仍然通过 --connect 连接 daemon socket，主进程始终只和 bridge 的 stdin/stdout 通信。

relay daemon 启动后注册 PTY、FS、Git、workspace、agent hook 等 handler，并监听 Unix socket 或 named pipe。 detached 模式一开始没有直接 stdout client，所以会进入 grace；第一个 socket client 接入后取消 grace。 relay.configureGraceTime 可以更新 PTY grace timer；graceTimeSeconds = 0 表示不自动过期，不是立即关闭。系统睡眠前主进程会把 grace 设成 0，防止睡眠时间吃掉普通窗口。

恢复：SSH 掉线和 relay 掉线走同一个重建口

SSH transport 掉线由 SshConnection 的 disconnect handler 发现，进入 bounded reconnect。 当它重新 connected 且 reconnectAttempt = 0，ssh.ts 会把 UI 覆盖为 reconnecting，调用 session.reconnect() 重建 relay provider。 如果 SSH 没掉但 relay bridge 关了，multiplexer 的 onDispose("connection_lost") 会触发同一个 relay-lost backoff，然后也回到 session.reconnect()。

恢复终端不是重新 spawn。session 会从 durable remote PTY lease 和当前 ownership 里收集已知 relay PTY id， 调 pty.attach，把 replay buffer 发回 renderer，并把 lease 标成 attached。 如果 relay 返回 PTY not found，说明 grace 已过期或进程已死：Orca 会标记 expired，发送 pty:exit， renderer 再冷启动一个新 shell。启动恢复时，passphrase 目标会先变成 deferred，等用户聚焦终端或主动连接时再 attach。

检索练习

用户说“SSH 状态显示已连接，但远端终端没有恢复旧输出”，你会先查哪几层？

参考路径：先看 ssh.ts 是否已经从 deploying-relay/reconnecting 进入 ready connected； 再看 ssh-relay-session.ts 的 reattach 是否命中 lease、是否收到 replay 或 not found； 最后看 relay.ts 的 grace/socket 日志，确认是连上旧 daemon 还是 fresh launch 了新 daemon。 如果是 passphrase 目标，还要检查 renderer 是否走了 deferred SSH reattach。完整来源见 SSH Relay 会话保活参考。
