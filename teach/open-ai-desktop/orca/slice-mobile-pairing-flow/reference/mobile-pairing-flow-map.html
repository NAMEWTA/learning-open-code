移动端配对与会话恢复参考

范围：桌面 QR、main IPC、runtime device registry、E2EE WebSocket、mobile pairing screens、host 持久化、共享 client 和 Resume。

端到端时序

阶段 主文件 关键事实
桌面生成 QR src/renderer/src/components/settings/MobilePane.tsx 调用 window.api.mobile.getPairingQR()，保存 qrDataUrl、pairingUrl、endpoint，并记录 mobile-pairing feature interaction。
选择可达地址 src/main/ipc/mobile.ts 枚举非 internal IPv4，tailnet 地址优先；renderer 也可传选中的 address。

桌面 QR：renderer 只是发起者

桌面设置页的入口在 src/renderer/src/components/settings/MobilePane.tsx。 generateQR() 调 window.api.mobile.getPairingQR()，如果成功就写入 qrDataUrl、pairingUrl、endpoint，并刷新 paired devices。 src/renderer/src/components/settings/MobileNetworkInterfaceSection.tsx 让用户选择写进 QR 的网络地址；src/renderer/src/components/settings/MobilePairingQrSection.tsx 展示二维码，也把同一个 pairingUrl 作为可复制 code。

main 边界在 src/main/ipc/mobile.ts。mobile:getPairingQR 默认选择可达 IPv4， tailnet 地址排在前面；然后调用 rpcServer.createPairingOffer({ scope: "mobile" })。 这个 handler 还负责 mobile:listDevices：只展示 scope === "mobile" 且 lastSeenAt > 0 的设备。换句话说，生成 QR 只创建 pending token， 真正扫码并连上后才出现在 paired devices 列表。

对比 mobile:getRuntimePairingUrl 很重要。runtime/web/desktop remote clients 会请求 scope: "runtime"，并可能拿到 web client URL；mobile QR 固定用 mobile scope， 因为手机端应该被 runtime 的 mobile allowlist 限制。 mobile/README.md 可用于开发环境和网络要求参考，但其中 QR 编码 TLS fingerprint 的旧说法不再作为 payload 权威来源。

runtime offer 和凭据文件

src/main/runtime/runtime-rpc.ts 的 createPairingOffer() 先要求 WebSocket endpoint、DeviceRegistry 和 E2EE public key 都可用。 它会解析 address override，把 0.0.0.0 这类监听地址替换成用户选中的 LAN/tailnet 地址。 然后用 getOrCreatePendingDevice() 或 rotatePendingDevice() 生成 token，调用 src/shared/pairing.ts 的 encodePairingOffer()。 

Shared PairingOffer v2
  endpoint       ws://192.168.x.y:6768 或 tailnet/wss endpoint
  deviceToken    DeviceRegistry 里的 per-device bearer token
  publicKeyB64   runtime E2EE public key
  scope          "mobile" 或 "runtime"

这说的是桌面/shared payload。手机端 mobile/src/transport/types.ts 的 PairingOfferSchema 当前只保留 v、endpoint、 deviceToken 和 publicKeyB64，不暴露也不依赖 scope。 手机侧授权判断来自 token registry、runtime allowlist 和 status.get 返回的 deviceScope。

src/main/runtime/device-registry.ts 把 token 写进 DEVICE_REGISTRY_FILENAME 指向的 userData 文件。 getOrCreatePendingDevice() 会复用同 scope、lastSeenAt === 0 的 pending token， 避免用户反复打开设置页产生一堆永远未扫描的凭据；显式 regenerate 才 rotate pending token。 lastSeenAt 在 E2EE auth 成功后更新。

src/main/runtime/e2ee-keypair.ts 负责 E2EE_KEYPAIR_FILENAME 指向的 userData 文件。 public key 进 QR；secret key 留在桌面 userData。src/main/runtime/mobile-pairing-files.ts 把 devices registry 和 E2EE keypair 声明为同一迁移单元。src/main/runtime/mobile-pairing-userdata-path.test.ts 保护这两个文件写在 canonical userData path，并从旧路径成对迁移。

手机入口：scan、deep link、paste

mobile/app/_layout.tsx 安装 RpcClientProvider，同时监听 Linking.getInitialURL() 和 warm-start URL event。只要 extractPairingCodeFromUrl() 找到 code，就 router.replace("/pair-confirm")。 mobile/app/pair.tsx 是文件路由上的 redirect screen：如果 URL 参数里有 code， 直接进 confirm；否则再读 initial URL。

mobile/app/pair-scan.tsx 处理相机扫码和手动粘贴。扫码结果先进入 decodePairingUrl()，粘贴输入进入 parsePairingCode()。 mobile/src/transport/pairing.ts 为 Metro/Hermes 复制了一份 shared parser： 它用 atob/btoa，接受 query code、hash code、bare base64，并拒绝 orca://pairing 这类 lookalike route。

mobile/app/pair-confirm.tsx 用 mobile/src/transport/pair-confirm-state.ts 把 route param 分成 missing、invalid、ready。ready 只代表 payload 结构正确，还没有证明桌面可达。

临时连接：E2EE/auth/status.get

pair scan 和 pair confirm 的验证逻辑相同：创建临时 RpcClient， 调 connect(offer.endpoint, offer.deviceToken, offer.publicKeyB64, { onLog })， 再等待 client.sendRequest("status.get")。 mobile/src/transport/pairing-connection-attempt.ts 给这个初次配对套 25 秒总超时； 超时或 route unmount 都会 dispose，并关闭临时 client。

mobile/src/transport/rpc-client.ts 的握手顺序是： 打开 WebSocket，生成一次性 mobile keypair，发明文 e2ee_hello， 收到明文 e2ee_ready 后，用共享密钥加密发送 e2ee_auth 和 deviceToken， 收到加密的 e2ee_authenticated 后才进入 connected。 后续 request、stream subscribe、terminal/browser binary frame 都走同一个 shared key。 具体加解密 primitive 在 mobile/src/transport/e2ee.ts，它用 tweetnacl、ExpoCrypto 和 base64 text frame。

桌面对应实现是 src/main/runtime/rpc/e2ee-channel.ts。 channel 在 awaiting_hello 收 hello，派生 shared key 后发 ready； 在 awaiting_auth 解密 auth 并调用 validateToken()。 token 不存在时会返回 encrypted e2ee_error 并关闭 socket。 只有 ready 后，channel 才把解密后的 JSON 交给 runtime message handler。

runtime WebSocket 授权边界

WebSocket transport 在 src/main/runtime/runtime-rpc.ts 启动时创建。 src/main/runtime/rpc/ws-transport.ts 处理连接数、pre-auth timeout、ping/pong heartbeat 和 close cleanup；runtime server 为每个 WebSocket 创建 E2EEChannel， auth 成功后调用 wsTransport.setClientId()，并把 device 的 lastSeenAt 更新为现在。

进入 handleWebSocketMessage() 后，server 不信任 request 里重复携带的 token。 如果 E2EE channel 已绑定 authenticatedDeviceToken，就以 channel identity 为准； request token 不匹配会返回 unauthorized。然后用 DeviceRegistry.validateToken() 找 device entry。mobile scope 还必须通过 MOBILE_RPC_METHOD_ALLOWLIST， 否则返回 forbidden。

status.get 有一处特殊处理：runtime 会把 authenticated device 的 scope 注入 success result。 共享类型在 src/shared/runtime-types.ts，字段是 RuntimeStatus.deviceScope。 这样旧的 saved WebSocket pairing 不需要从 URL payload 重新推断 scope。

保存 host：本地状态和 token 分离

当 status.get 成功，mobile/app/pair-confirm.tsx 和 mobile/app/pair-scan.tsx 都会生成 host-${Date.now()}， 用 getNextHostName() 取默认名称，并调用 saveHost()。 saveHost() 先校验 HostProfileSchema，再把不含 token 的 StoredHostProfile 写入 AsyncStorage，最后写 token。

mobile/src/transport/host-store.ts 的关键边界是 storage split： native 上 deviceToken 写入 Expo SecureStore，配置为 WHEN_UNLOCKED_THIS_DEVICE_ONLY； web fallback 才把 token 放到 AsyncStorage 的 web token key。loadHosts() 读取 metadata 后再从 SecureStore 拼回 token。没有 token 的 orphan metadata 会被跳过； pre-v0.0.3 把 token 放 metadata 的旧记录也被丢弃，要求用户重新配对。

所以错误文案要分层理解。Cannot connect、Authentication failed、 Couldn't connect within 25s 是连接/认证阶段；Pairing succeeded but couldn't save the host 是本地持久化阶段。排障时先看 AsyncStorage/SecureStore 和 schema，而不是重新查 WebSocket 握手。

后续恢复：共享 client，不重新配对

App 根部 mobile/app/_layout.tsx 包了一层 RpcClientProvider。 mobile/src/transport/client-context.tsx 用 Map 保存每个 host 的 RpcClient、 connection state 和 refCount。屏幕通过 useHostClient() 或 useAllHostClients() acquire；如果 client 尚未打开，provider 会异步 loadHosts()，用保存的 endpoint/token/publicKey 重建连接。

provider 还接入 mobile/src/transport/connection-revival-triggers.ts。 当 AppState 变为 active，或网络从 offline 回 online、Wi-Fi/蜂窝切换时，它会对所有 live client 调 notifyForeground()。如果当前 state 是 connected，client 发一次 status.get activity probe，8 秒无有效响应就关闭 socket 进入 reconnect；如果 state 是 reconnecting， 它会清掉旧 backoff，重置 attempt budget，并马上 openConnection。

mobile/src/transport/rpc-client.ts 还会在 reconnect 后 replay 已登记的 stream subscription。 terminal subscribe 会重建 stream id 路由，browser screencast 会等待新的 ready 后再路由 binary frame。 这些都是“会话恢复”，不是“重新配对”。

首页 Resume：从 host 恢复到 worktree

mobile/app/index.tsx 启动时 loadHosts()，然后把 hosts 传给 primeHosts()，避免 provider 再做一轮 Keychain 读取。它用 useAllHostClients() 为所有 host 建共享连接，并对每个 client 请求 worktree.ps 和 accounts snapshot。

Resume 卡片的来源有两层。第一层是本机上次打开的 orca:last-visited-worktree 和 worktree cache；第二层是按 host lastConnected 排序后的每个 host 的 lastActiveWorktree。 mobile/src/worktree/resume-worktree.ts 的 fallback 策略是优先 isActive 的 desktop-focused worktree，再按 lastOutputAt 选最近输出的 worktree；这个策略只发生在单个 host 的 worktree 列表内。 首页只对 state 为 connected 的 host 展示 Resume，避免给用户一个暂时无法加载的 stale route。 进入 host 后，mobile/app/h/[hostId]/index.tsx 复用同一个 host client 读取 worktree.ps 并进入 session route。

撤销和 cleanup

桌面侧 mobile:revokeDevice 调 rpcServer.revokeMobileDevice()。 runtime 会确认 device scope 是 mobile，删除 registry entry，并调用 wsTransport.terminateClientConnections(device.token)。active socket 关闭时， runtime 会按 WebSocket connection id 清理 stream subscriptions、mobile dictation 和 binary handlers； 如果这个 token 没有其他连接，还会调用 runtime.onClientDisconnected(clientId)。

runtime scope 有单独的 revokeRuntimeAccess()，mobile revocation 不会误删 runtime grant。 src/main/runtime/runtime-rpc.test.ts 对 mobile/runtime 两类 revoke 都有覆盖。
