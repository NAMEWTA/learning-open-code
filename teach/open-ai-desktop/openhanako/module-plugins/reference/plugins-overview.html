2. 插件架构通用约定

2.1 manifest.json 结构

每个插件通过 manifest.json 声明元信息、权限和能力：

{
  "id": "plugin-id",              // 唯一标识，未声明时从目录名推导
  "name": "显示名称",              // 管理 UI 展示
  "version": "1.0.0",
  "description": "...",
  "trust": "restricted|full-access",  // 权限级别（默认 restricted）
  "hidden": true,                     // 内置插件通常隐藏
  "minAppVersion": "0.82.0",          // 最低 HanaAgent 版本（可选）
  "activationEvents": ["onStartup"],  // 激活条件
  "capabilities": [                   // 普通能力声明
    "resource.read", "resource.write", "network.fetch"
  ],
  "sensitiveCapabilities": [          // 敏感能力（未来授权流）
    "filesystem.write"
  ],
  "network": {                        // 外部 HTTP 访问边界
    "allowedHosts": ["api.example.com"],
    "methods": ["GET"],
    "defaultTimeoutMs": 8000,
    "maxResponseBytes": 1048576
  },
  "ui": {
    "hostCapabilities": ["external.open", "clipboard.writeText"]
  },
  "contributes": {                    // 贡献点
    "configuration": { ... },         // JSON Schema 配置
    "page": { ... },                  // 全页面视图
    "widget": { ... },                // 侧栏组件
    "settingsTab": { ... }            // 内置设置页（仅 built-in）
  },
  "depends": {
    "capabilities": ["bridge:send"]   // 软依赖声明
  }
}

2.2 目录规范

插件目录结构如下（标注 ⚡ 的能力需要 full-access 权限）：

my-plugin/
├── manifest.json          # 可选，复杂声明才需要
├── index.js               # 可选，有状态生命周期入口 ⚡
├── tools/                 # 工具（Agent 调用）
│   └── *.js
├── skills/                # 知识注入（Markdown）
│   └── my-skill/
│       └── SKILL.md
├── commands/              # 用户命令（斜杠触发）
│   └── *.js
├── agents/                # Agent 模板（JSON）
│   └── *.json
├── routes/                # HTTP 路由 ⚡
│   └── *.js
├── providers/             # Provider 声明 ⚡
│   └── *.js
├── extensions/            # Pi SDK extension 工厂 ⚡
│   └── *.js
├── assets/                # 静态前端资源
└── lib/                   # 插件私有库代码
Tool-only 插件（仅 tools/ 目录）无需 manifest.json 和 index.js，权限默认为 restricted。这是社区插件的最小形态。

2.3 权限模型（restricted vs full-access）

能力restrictedfull-access
tools/*.js — 声明工具供 Agent 调用✅✅
skills/ — Markdown 知识注入✅✅
commands/*.js — 用户命令✅✅
agents/*.json — Agent 模板✅✅
ctx.config — 读写自己的配置✅✅
ctx.dataDir — 私有数据目录✅✅
bus.emit / subscribe / request — 事件总线✅✅
contributes.configuration — JSON Schema 配置✅✅
bus.handle — 注册能力供其他插件调用❌✅
routes/*.js — HTTP 端点❌✅
extensions/*.js — Pi SDK 事件拦截❌✅
providers/*.js — Provider 声明❌✅
ctx.registerTool — 运行时动态注册工具❌✅
onload / onunload — 生命周期钩子❌✅
安全边界：full-access 插件需要用户在设置中开启"允许全权插件"开关才会加载。开关关闭时，full-access 插件完全不会被加载（不会部分加载）。未声明 trust 或声明为其他值的插件，一律按 restricted 处理。

2.4 生命周期与激活事件

有状态插件通过 index.js 导出 class，提供 onload() 和可选的 onunload() 方法：

export default class MyPlugin {
  declare ctx: any;        // PluginManager 注入的上下文
  declare register: any;   // 注册清理函数

  async onload() {
    // ctx.bus       — EventBus（emit/subscribe/request/handle）
    // ctx.config    — 配置读写
    // ctx.dataDir   — 私有数据目录
    // ctx.log       — 带 pluginId 前缀的 logger
    // ctx.pluginId  — 插件 ID
    // ctx.registerTool — 动态工具注册

    // register() 注册的资源在卸载时自动清理（逆序）
    this.register(
      this.ctx.bus.handle("some:event", async (payload) => { ... })
    );
  }

  async onunload() {
    // 只清理框架管不到的资源（WebSocket、定时器等）
    // register() 注册的东西自动清理
  }
}

激活事件控制插件何时执行 onload()：

事件触发时机
onStartup插件加载时立刻执行（默认）
onPageOpen用户打开插件页面 route
onWidgetOpen用户打开插件 widget route
onToolCall任意静态 tool 被调用
onToolCall:name指定静态 tool 被调用
*任意已知触发原因

内置插件的 4 个有状态插件（beautify、image-gen、jimeng-cli、mcp）全部使用 onStartup 激活。

2.5 EventBus 通信机制

插件间通信通过 EventBus 的请求-响应模型。命名规范为 领域:动作（冒号分隔）：

API权限说明
bus.emit(type, payload)所有插件发布事件（fire-and-forget）
bus.subscribe(handler)所有插件订阅事件
bus.request(type, payload)所有插件请求-响应，等待结果
bus.handle(type, handler)full-access注册能力处理函数

SKIP 链：同一事件类型可注册多个 handler，按注册顺序调用。handler 返回 HANA_BUS_SKIP 表示"我不处理，交给下一个"，直到某个 handler 返回实际值。

内置 Session / Agent / 媒体能力（插件可通过 bus 直接调用）：

能力说明
session:create / get / update / send / abort / history / listSession 生命周期管理
agent:list / profile / create / updateAgent 管理
model:sample-text非流式文本采样
media:generate-image / generate / generate-video / transcribe-audio媒体生成
task:register / update / complete / fail / cancel / remove后台任务管理
deferred:register / resolve延迟结果送达

4. MCP 插件协议支持

mcp 插件（在 UI 中称为 "Connectors / 连接器"）是 HanaAgent 的 Model Context Protocol 桥接层，将外部 MCP 服务器的 tools 动态注册为 Agent 可调用的工具。它同时支持 HTTP 和 stdio 两种传输模式，并内置完整的 OAuth 2.0 认证流程。

4.1 传输层：HTTP + stdio 双模式

传输模式实现类协议特点适用场景
stdio McpStdioClient JSON-RPC 2.0 over stdin/stdout；spawn 子进程 本地 MCP 服务器（npx、uvx、python 等）
streamable-http McpStreamableHttpClient MCP 2025 Streamable HTTP 规范；POST + SSE 响应；Session ID 管理 远程 MCP 服务器（新规范）
sse（legacy） McpLegacySseClient SSE GET 长连接 + POST message endpoint；endpoint 事件协商 旧版 SSE MCP 服务器
remote（auto） McpAutoHttpClient 自动探测：先尝试 streamable-http，失败后降级到 SSE 用户只需提供 URL，无需了解协议细节

HTTP 客户端架构

McpAutoHttpClient           ← 自动探测入口
  ├─ McpStreamableHttpClient (优先尝试)
  │   ├─ POST / (JSON-RPC 2.0)
  │   ├─ Accept: application/json, text/event-stream
  │   ├─ MCP-Session-Id header
  │   └─ Session 过期自动重初始化
  └─ McpLegacySseClient (降级)
      ├─ GET /sse (SSE 长连接)
      ├─ endpoint 事件 → message endpoint
      └─ POST message endpoint (JSON-RPC 2.0)

stdio 客户端架构

McpStdioClient
  ├─ spawn(command, args, { cwd, env, shell: false })
  ├─ stdin  → JSON-RPC 请求 (行分隔)
  ├─ stdout → JSON-RPC 响应 (行分隔)
  ├─ stderr → debug 日志
  ├─ 优雅关闭: stdin EOF → SIGTERM (2s) → SIGKILL (3s)
  └─ 注册表支持: NPM_CONFIG_REGISTRY / UV_DEFAULT_INDEX 等
协议版本：MCP 协议版本为 2025-11-25，通过 MCP-Protocol-Version header 协商。clientInfo 统一标识为 name: "hana", title: "Hana", version: "0.1.0"。

4.2 OAuth 2.0 认证与自愈

MCP 插件支持三种认证类型，并有完整的 OAuth 2.0 自愈机制：

认证类型说明
none无认证
bearer静态 Bearer Token（authorizationToken）
oauthOAuth 2.0 Authorization Code + PKCE，支持 DCR (RFC 7591)、Token 刷新、自愈

OAuth 自愈机制（方案 A）：

  1. 预刷新：每次请求前，getValidToken() 检查 access token 是否在 60s 内过期，若是则自动刷新
  2. 并发去重：多个并发请求触发刷新时，共享同一个 in-flight promise（refreshInFlight Map）
  3. 401 自愈：请求收到 401 后，refreshAuthToken() 强制刷新 token，成功后仅重放一次（不循环）
  4. 写回不中断：刷新成功后 token 写回 config，但不重启 connector（live client 通过回调获取新 token）

4.3 自动重连与容错

McpRuntime 实现了完整的连接生命周期管理：

  • 指数退避重连：从 1s 开始，每次翻倍，上限 30s（RECONNECT_INITIAL_DELAY_MS=1000, RECONNECT_MAX_DELAY_MS=30000, GROW_FACTOR=2）
  • 意图门控：重连只发生在 desiredStates.get(id) === "running" 时；用户手动 stop / 全局禁用 / autoReconnect=false 会终止重连
  • Auth-terminal 检测：401/403 或 OAuth invalid_grant 错误触发 needsAuth 状态，停止重连，等待用户重新授权
  • establishing 窗口：连接建立期间，transport 的 onClose 被抑制，避免 start() promise 和 close 事件双重处理
  • stale client 保护：onClose handler 捕获 client 实例引用，只有当前活跃 client 的 close 事件才被处理

4.4 动态工具注册

MCP 插件的工具是完全动态的：

  1. Connector 启动后调用 client.listTools() 获取远程工具列表
  2. 工具元数据（name、description、inputSchema）缓存到 config
  3. 通过 ctx.registerTool() 动态注册为 Agent 可见工具
  4. 工具名格式：mcp_{connectorId}_{toolName}（前缀 mcp_ 由 PluginManager 添加）
  5. 提供内置诊断工具 mcp_connectors_status，Agent 可自行查询连接器状态
  6. 支持 per-Agent 的 connector/tool 启用控制

5.2 jimeng-cli — 即梦生成器扩展

jimeng-cli 是字节跳动即梦（Dreamina）CLI 工具的系统集成。它通过 Provider + Adapter 双入口方式接入图片和视频生成能力：

入口文件职责

加载流程：

  1. onload() 中通过 bus.request("media-gen:register-adapter", { adapter }) 将 jimengImageAdapter 和 jimengVideoAdapter 注册到 image-gen 运行时
  2. 同时 Provider 声明进入 ProviderRegistry，使即梦模型出现在媒体模型选择器中
  3. 卸载时通过 media-gen:unregister-adapter 清理
解耦模式：jimeng-cli 不依赖 image-gen 的任何内部实现细节，仅通过 EventBus 的 media-gen:register-adapter 事件注入能力。这是 插件间扩展的典范——新供应商只需实现 Adapter 接口并通过 bus 注册，无需修改 image-gen 源码。

5.3 media — 媒体生成工具

media 是纯 tool-only 插件（无 index.ts），为 Agent 提供面向用户的媒体生成接口。它通过 EventBus 将请求转发给底层的 Hana Media Manager。

工具清单

工具名功能权限
generate-image提交图片生成任务。支持 prompt、count、ratio、resolution、referenceImages、model/provider 选择、provider optionsexternal_side_effect
generate-video提交视频生成任务。支持 prompt、image(图生视频)、duration、ratio、resolution、modeexternal_side_effect
describe-options查询指定 provider/model 支持的参数选项（用于 Agent 动态发现能力）readOnly

Skill 注入

skills/media-generation-guide/SKILL.md 作为知识注入，Agent 在用户请求媒体生成时会自动加载该指南，其中包含各 provider 的推荐参数、最佳实践和限制说明。

总线转发：media 工具本身不执行任何媒体生成逻辑。所有工具通过 ctx.bus.request("media:generate-image" / "media:generate-video") 将请求转发给由 image-gen 插件注册的 handler。这是一种 Facade 模式：media 提供 Agent 接口层，image-gen 提供执行层。

5.4 office — 办公文档处理

office 是唯一使用 restricted 权限的内置插件，提供办公文档读写能力。它无需 lifecycle hook，仅通过 3 个静态工具暴露功能。
