packages/ 插件 SDK 体系总览

L1 模块 插件 SDK Monorepo 子包

packages/ 目录包含 HanaAgent 插件 SDK 的四个独立子包，构成从协议定义到 UI 组件的完整四层体系。 插件开发者可按需选择所需层级：仅需类型共享引入 plugin-protocol，编写 iframe UI 引入 plugin-sdk + plugin-components， 编写 Node 端逻辑引入 plugin-runtime。

1.2 四包职责对比表

包名 npm 名称 运行环境 核心职责 依赖关系 入口文件
protocol @hana/plugin-protocol Browser + Node 共享协议常量、消息类型定义、资源类型、错误码 无依赖（基础层） src/index.ts
sdk @hana/plugin-sdk Browser/iframe WebView 宿主通信：ready、resize、toast、clipboard、resources、assets、api.fetch 依赖 protocol src/index.ts
runtime @hana/plugin-runtime Node.js Node 端 helpers：defineTool、definePlugin、EventBus、Session/Agent API、ResourceIO、Media、Task 依赖 protocol src/index.ts
components @hana/plugin-components Browser/iframe React React UI 组件：Button、CardShell、SettingRow、HanaThemeProvider、Select、Switch 等 依赖 React（peer） src/index.ts（重导出）
关键设计原则：四个包按运行环境分层，protocol 是唯一跨端共享层。 Browser 端代码不直接访问 Node API，Node 端代码不引入 React 组件。插件 iframe 内的网络请求通过 hana.api.fetch() 代理到插件自身路由， 路由中再通过 ctx.network.fetch() 发出实际 HTTP 请求，确保 API 密钥不泄露到前端 bundle。

2.5 PluginUiMessage 消息结构

插件与宿主之间通过 window.postMessage 通信，消息格式如下：

interface PluginUiMessage {
  protocol: 'hana.plugin.ui';        // 固定协议标识
  version: 1;                        // 协议版本
  id?: string;                       // 请求/响应/错误消息的唯一 ID（event 可选）
  kind: 'event' | 'request' | 'response' | 'error';  // 消息种类
  type: string;                      // 消息类型（如 'hana.ready'、'toast.show'）
  payload?: unknown;                 // 消息载荷
  error?: PluginUiError;             // 错误详情（kind='error' 时必填）
}
消息验证规则：parsePluginUiMessage() 严格校验 protocol、version、kind、type 字段。 kind 为 request、response、error 时 id 必填。 kind 为 error 时 error.code 和 error.message 必填。

2.6 PluginResourceRef 资源引用类型

插件通过统一资源引用模型访问不同类型的资源：

type PluginResourceRef =
  | { kind: 'local-file';     path: string }                         // 本地文件
  | { kind: 'mount';          mountId: string; path: string }        // 挂载点文件
  | { kind: 'session-file';   fileId: string; sessionId?: string; sessionPath?: string }  // Session 文件
  | { kind: 'resource';       resourceId: string }                   // 资源记录
  | { kind: 'url';            url: string };                         // URL 资源（只读）

2.7 UI Slot 类型

插件可申请以下 UI 插槽类型：

type PluginUiSlot = 'page' | 'widget' | 'card' | 'settings';
  • page -- 独立页面插槽
  • widget -- 侧边栏小组件
  • card -- 聊天流中的内嵌卡片（如 chat.surface）
  • settings -- 设置面板插槽

3. plugin-sdk -- WebView/iframe 端 SDK

@hana/plugin-sdk 提供 iframe 内插件与宿主桌面应用之间的双向通信能力。 所有 API 通过 createHanaPluginSdk() 工厂函数创建，同时暴露单例 hana 全局对象方便直接调用。

3.1 HanaPluginSdk 接口全貌

interface HanaPluginSdk {
  // 就绪信号
  ready(payload?: unknown): void;

  // 静态资源 URL 辅助
  assets: { url(path: string): string };

  // 插件自身路由 API 调用
  api: {
    url(path: string): string;
    fetch(path: string, init?: RequestInit): Promise<Response>;
  };

  // UI 控制
  ui: { resize(size: HanaPluginSize): void };

  // 主题系统
  theme: {
    getSnapshot(): HanaPluginThemeSnapshot;
    subscribe(callback: (theme: HanaPluginThemeSnapshot) => void): () => void;
  };

  // 低级宿主请求
  host: { request<T>(type: string, payload?: unknown, options?: HanaPluginRequestOptions): Promise<T> };

  // Toast 通知
  toast: { show(input: HanaToastShowInput, options?: HanaPluginRequestOptions): Promise<HanaToastShowResult> };

  // 外部链接
  external: { open(input: HanaExternalOpenInput, options?: HanaPluginRequestOptions): Promise<HanaExternalOpenResult> };

  // 剪贴板
  clipboard: { writeText(input: HanaClipboardWriteTextInput, options?: HanaPluginRequestOptions): Promise<HanaClipboardWriteTextResult> };

  // 资源操作（宿主中介）
  resources: {
    open(input: PluginResourceOpenInput, options?: HanaPluginRequestOptions): Promise<PluginResourceOpenResult>;
    pick(input?: PluginResourcePickInput, options?: HanaPluginRequestOptions): Promise<PluginResourcePickResult>;
    requestAccess(input: PluginResourceRequestAccessInput, options?: HanaPluginRequestOptions): Promise<PluginResourceRequestAccessResult>;
  };
}

3.2 API 详解

ready hana.ready(payload?)

发送 hana.ready 事件给宿主，表示插件 iframe 已加载完成。可附带自定义 payload。

hana.ready({ version: '1.0.0' });
assets hana.assets.url(path)

生成插件静态资源的安全 URL。自动从 iframe 路由提取 pluginId，路径经严格校验（拒绝路径穿越、绝对 URL、dotfiles）。返回格式：

/api/plugins/{pluginId}/assets/{safePath}

资源由宿主通过 HttpOnly cookie 认证的短生命周期 session 提供。支持 JS、CSS、图片、字体、JSON、WASM、视频（含 byte-range）。

const logo = hana.assets.url('images/logo.svg');
const video = hana.assets.url('videos/demo.mp4');
api.fetch hana.api.fetch(path, init?)

向当前插件的服务端路由发起 HTTP 请求。自动附加 X-Hana-Plugin-Surface-Session 头进行认证。

URL 自动解析为 /api/plugins/{pluginId}/{apiPath}。用于替代插件前端代码中的直接 fetch('/api/...') 调用。

const res = await hana.api.fetch('api/translate', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ text: 'hello' }),
});
const data = await res.json();
安全约束：不允许硬编码 /api/plugins/{pluginId}/... 路径，不允许重用 pluginIframeTicket 做 XHR/fetch 认证。 前端代码不应直接调用第三方 API，应通过自身路由 + ctx.network.fetch() 代理。
ui.resize hana.ui.resize(size)

请求宿主调整 iframe 尺寸。发送 ui.resize capability 事件。

hana.ui.resize({ width: 400, height: 320 });
hana.ui.resize({ height: 600 }); // 仅调整高度
theme hana.theme.getSnapshot() / subscribe()

获取当前主题快照或订阅主题变更。主题信息从 iframe URL 参数 hana-theme 和 hana-css 初始化，运行时通过 hana.theme.changed 事件更新。

// 获取当前主题
const { theme, cssUrl } = hana.theme.getSnapshot();

// 订阅主题变更
const unsubscribe = hana.theme.subscribe((newTheme) => {
  console.log('主题变更:', newTheme);
});
toast.show hana.toast.show(input)

弹出 Toast 通知。支持 success / error / info / warning 四种类型，可自定义持续时间。

await hana.toast.show({ message: '操作成功', type: 'success', duration: 3000 });
external.open hana.external.open(input)

在系统默认浏览器中打开外部链接。需要 manifest 声明 external.open host capability。

await hana.external.open('https://example.com');
await hana.external.open({ url: 'https://example.com' });
clipboard.writeText hana.clipboard.writeText(input)

将文本写入系统剪贴板。需要 manifest 声明 clipboard.writeText host capability。

await hana.clipboard.writeText('复制的内容');
await hana.clipboard.writeText({ text: '复制的内容' });
resources hana.resources.open / pick / requestAccess

宿主中介的资源操作（仅浏览器端可用，不暴露文件系统读写）。

// 打开资源
await hana.resources.open({
  resource: { kind: 'session-file', fileId: 'sf_1' },
  mode: 'preview',
});

// 选择文件/目录
const result = await hana.resources.pick({ mode: 'file', multiple: true });

// 请求资源访问权限
const access = await hana.resources.requestAccess({
  capability: 'resource.read',
  reason: '需要读取用户文件进行分析',
});

3.3 通信机制详解

SDK 内部使用 window.postMessage 与父窗口（宿主）通信，核心流程如下：

1
目标来源解析：从 iframe URL 参数 hana-host-origin 或 document.referrer 推断宿主的 origin，确保消息只发往信任的宿主窗口。
2
消息构造：所有消息遵循 PluginUiMessage 格式，包含 protocol、version、kind、type 字段。request 类消息自动生成 UUID id。
3
request-response 模式：request 消息通过 Promise 封装，监听 message 事件等待匹配 id 的 response/error 消息。默认超时 10 秒（可通过 requestTimeoutMs 配置）。
4
安全检查：每个 message 事件都验证 event.source === parentWindow 和 origin 匹配，防止跨源攻击。

3.4 单例模式

SDK 暴露 hana 全局单例对象，首次调用时自动创建。也可通过 createHanaPluginSdk(options) 创建独立实例：

import { hana, createHanaPluginSdk } from '@hana/plugin-sdk';

// 使用全局单例（推荐）
hana.ready();

// 创建独立实例（高级场景）
const sdk = createHanaPluginSdk({
  targetOrigin: 'https://my-hana-host.example.com',
  requestTimeoutMs: 15_000,
});

4. plugin-runtime -- Node.js 运行时 Helpers

@hana/plugin-runtime 是插件服务端代码的核心工具箱，提供工具定义、生命周期管理、EventBus 通信、Session/Agent API、 资源 I/O、媒体生成、任务调度等完整能力。所有 helpers 基于 HanaPluginContext (ctx) 工作。

4.1 HanaPluginContext -- 插件上下文

每个插件在加载时获得一个完整的运行时上下文，包含以下关键属性：

interface HanaPluginContext {
  pluginId: string;              // 插件 ID
  pluginDir: string;             // 插件目录路径
  dataDir: string;               // 插件数据目录（可写入）
  serverId: string;              // 服务器 ID
  studioId: string;              // Studio ID
  userId: string;                // 用户 ID
  sessionId?: string | null;     // 当前 Session ID
  sessionRef?: HanaSessionRef;   // Session 引用对象
  capabilities?: string[];       // 已授权的 capability 列表
  sensitiveCapabilities?: string[]; // 敏感 capability 列表

  bus: HanaEventBus;             // 事件总线
  network: HanaPluginNetwork;    // 网络请求（替代裸 fetch）
  resources: HanaPluginResources;// 资源 I/O
  config: HanaPluginConfigStore; // 配置存储
  log: HanaPluginLogger;         // 日志器（debug/info/warn/error）

  // 插件注册 helpers
  registerTool?(tool: HanaToolDefinition): () => void;
  registerSessionFile?(input: Record<string, unknown>): HanaSessionFile;
  stageFile?(input: Record<string, unknown>): HanaStagedSessionFile;
}

4.4 ResourceIO -- 服务端资源操作

通过 ctx.resources 操作用户资源（区别于浏览器端的 hana.resources.* 宿主中介接口）：

interface HanaPluginResources {
  stat(ref): Promise<HanaResourceStat>;
  read(ref): Promise<HanaResourceReadResult>;
  list(ref): Promise<HanaResourceListResult>;
  search(ref, options?): Promise<HanaResourceSearchResult>;
  materialize(ref): Promise<HanaResourceMaterializeResult>;
  write(ref, content, options?): Promise<HanaResourceMutationResult>;
  writeExpectedVersion(ref, content, expectedVersion, options?): Promise<HanaResourceWriteExpectedVersionResult>;
  edit(ref, edits[], options?): Promise<HanaResourceMutationResult>;
  mkdir(ref, options?): Promise<HanaResourceMutationResult>;
  delete(ref, options?): Promise<HanaResourceMutationResult>;
  copy(from, to, options?): Promise<HanaResourceMutationResult>;
  rename(from, to, options?): Promise<HanaResourceMoveResult>;
  move(from, to, options?): Promise<HanaResourceMoveResult>;
  trash(ref, trashOptions?, options?): Promise<HanaResourceTrashResult>;
  watch(ref, options?): HanaResourceWatchSubscription;
  subscribe(resources[], options?): HanaResourceWatchSubscription;
}
ResourceIO 原则：ResourceIO 是用户资源的唯一权威接口。local-file、mount、session-file、 resource、url 引用是身份输入，不保证插件能直接获取本地路径。需要具体路径时使用 ctx.resources.materialize(ref)， 并通过 ResourceIO 写回而非直接编辑具体化文件。stageFile() 仅用于插件生成物，应以 SessionFile 媒体形式交付。

5.2.3 主题回退顺序

当 HanaThemeProvider 渲染时，按以下顺序确定最终样式（后面的覆盖前面的）：

┌─────────────────────────────────────────────────────────┐
│ 1. SDK 默认值 (@hana/plugin-components/styles.css)      │  ← 基础兜底
├─────────────────────────────────────────────────────────┤
│ 2. 宿主 CSS 变量 (mode="inherit")                       │  ← 从宿主桌面主题继承
├─────────────────────────────────────────────────────────┤
│ 3. Hana 内置主题 (mode="hana" + theme="warm-paper" 等)  │  ← 10 套预设
├─────────────────────────────────────────────────────────┤
│ 4. 自定义令牌 (mode="custom" + HanaThemeTokens 对象)    │  ← 插件自定，最高优先级
└─────────────────────────────────────────────────────────┘

关键行为：

  • mode="inherit" -- 不设置任何 CSS 变量，完全依赖宿主 CSS 变量和 SDK 默认样式表。适用于希望融入宿主 UI 的插件。
  • mode="hana" -- 使用内置 Hana 主题。可通过 theme 属性指定具体主题 ID，未指定时默认使用 warm-paper。
  • mode="custom" -- 使用 theme 属性传入的 HanaThemeTokens 自定义令牌对象。部分令牌可不填，未填项回退到下层。

6.4 tools/todo-tools.js -- 工具定义

import { defineTool } from '@hana/plugin-runtime';

const TODOS_FILE = 'todos.json';

export const listTodosTool = defineTool({
  name: 'list_todos',
  description: '列出所有待办事项',
  sessionPermission: {
    readOnly: true,
    kind: 'read_only',
  },
  parameters: {
    type: 'object',
    properties: {
      filter: {
        type: 'string',
        enum: ['all', 'active', 'completed'],
        description: '筛选条件',
      },
    },
  },
  async execute(input, ctx) {
    const ref = { kind: 'local-file', path: `${ctx.dataDir}/${TODOS_FILE}` };
    let todos = [];
    try {
      const result = await ctx.resources.read(ref);
      todos = JSON.parse(new TextDecoder().decode(result.content));
    } catch {
      // 文件不存在时返回空列表
    }

    if (input.filter === 'active') todos = todos.filter(t => !t.completed);
    if (input.filter === 'completed') todos = todos.filter(t => t.completed);

    return {
      content: [{ type: 'text', text: JSON.stringify(todos, null, 2) }],
    };
  },
});

export const addTodoTool = defineTool({
  name: 'add_todo',
  description: '添加新的待办事项',
  sessionPermission: {
    kind: 'plugin_output',
    describeSideEffect: () => ({
      kind: 'session_file_output',
      summary: '将新的待办事项写入插件数据目录',
    }),
  },
  parameters: {
    type: 'object',
    properties: {
      title: { type: 'string', description: '待办标题' },
      priority: {
        type: 'string',
        enum: ['low', 'medium', 'high'],
        default: 'medium',
      },
    },
    required: ['title'],
  },
  async execute(input, ctx) {
    const ref = { kind: 'local-file', path: `${ctx.dataDir}/${TODOS_FILE}` };
    let todos = [];
    try {
      const result = await ctx.resources.read(ref);
      todos = JSON.parse(new TextDecoder().decode(result.content));
    } catch { /* 文件不存在 */ }

    todos.push({
      id: crypto.randomUUID(),
      title: input.title,
      priority: input.priority || 'medium',
      completed: false,
      createdAt: new Date().toISOString(),
    });

    await ctx.resources.write(ref, JSON.stringify(todos, null, 2));

    return {
      content: [{ type: 'text', text: `已添加待办: ${input.title}` }],
    };
  },
});

export const completeTodoTool = defineTool({
  name: 'complete_todo',
  description: '将待办事项标记为已完成',
  sessionPermission: {
    kind: 'plugin_output',
    describeSideEffect: () => ({
      kind: 'session_file_output',
      summary: '更新待办事项完成状态',
    }),
  },
  parameters: {
    type: 'object',
    properties: {
      todoId: { type: 'string', description: '待办事项 ID' },
    },
    required: ['todoId'],
  },
  async execute(input, ctx) {
    const ref = { kind: 'local-file', path: `${ctx.dataDir}/${TODOS_FILE}` };
    const result = await ctx.resources.read(ref);
    const todos = JSON.parse(new TextDecoder().decode(result.content));

    const todo = todos.find(t => t.id === input.todoId);
    if (!todo) return { content: [{ type: 'text', text: '未找到该待办事项' }] };

    todo.completed = true;
    todo.completedAt = new Date().toISOString();
    await ctx.resources.write(ref, JSON.stringify(todos, null, 2));

    return {
      content: [{ type: 'text', text: `已完成: ${todo.title}` }],
    };
  },
});

6.5 routes/api.js -- 路由处理

import { getPluginRequestContext } from '@hana/plugin-runtime';

export function registerRoutes(router) {
  // 插件自身 API 路由 -- 前端通过 hana.api.fetch('api/stats') 调用
  router.get('/api/stats', async (c) => {
    const ctx = c.get('pluginCtx');
    const ref = { kind: 'local-file', path: `${ctx.dataDir}/todos.json` };

    try {
      const result = await ctx.resources.read(ref);
      const todos = JSON.parse(new TextDecoder().decode(result.content));
      const active = todos.filter(t => !t.completed).length;
      const completed = todos.filter(t => t.completed).length;
      return c.json({ total: todos.length, active, completed });
    } catch {
      return c.json({ total: 0, active: 0, completed: 0 });
    }
  });

  // 调用外部 API -- 通过 ctx.network.fetch() 代理
  router.get('/api/external-data', async (c) => {
    const ctx = c.get('pluginCtx');
    const res = await ctx.network.fetch('https://api.example.com/data', {
      cacheTtlMs: 60_000,
      timeoutMs: 5000,
    });
    return c.json(await res.json());
  });
}

7. 运行时边界与生产发布

7.1 运行时边界

SDK 包是开发期依赖，不是宿主注入的全局模块。理解以下边界至关重要：

场景正确做法错误做法
iframe 前端使用 SDK 将 @hana/plugin-sdk 和 @hana/plugin-components bundle 到 iframe 资源中 期望宿主自动提供这些模块
Node 端使用 runtime bundle 服务端代码或随插件目录分发 node_modules 依赖 monorepo 工作区符号链接
第三方 API 调用 前端 → hana.api.fetch() → 路由 → ctx.network.fetch() 前端直接 fetch 第三方 API（密钥泄露）
资源路径获取 使用 ctx.resources.materialize(ref) 并通过 ResourceIO 写回 假设 local-file.path 是可直接访问的本地路径
静态资源引用 使用 hana.assets.url('images/logo.svg') 硬编码 /api/plugins/{pluginId}/assets/... 或手动追加 token

7.2 生产发布检查清单

  1. 检查所有服务端入口文件（index.js、tools/*、routes/*、providers/*、extensions/*）的 import 语句
  2. 确保 @hana/plugin-runtime 等包的依赖可在插件安装目录下解析
  3. Bundle 服务端代码，或随插件目录/zip 一同分发已安装的依赖
  4. 在 ${HANA_HOME}/plugins 中进行冒烟测试，确认无 Cannot find package 错误
  5. 验证插件诊断和服务器日志无导入异常
  6. 勿将 workspace 模式的插件目录发布为 release 包
冒烟测试金标准：安装用户将收到的确切文件夹或 zip 包，检查插件诊断和服务器日志中的导入错误。 在仓库中能加载但从 ${HANA_HOME}/plugins 加载失败的插件，说明运行时边界处理不正确。
