1. 模块职责一句话概括

core/ 是 HanaAgent 的引擎编排层——它在 lib/（基础库）之上编排所有核心能力（Agent 生命周期、会话管理、插件系统、Provider 注册、LLM 调用、视觉处理、安全权限），并向上对 server/ 和 desktop/ 暴露统一的 HanaEngine 外观（Facade）。它是应用的"大脑"和"中央调度器"。

具体来说，core/ 模块承担以下职责：

  • 总控调度 -- HanaEngine 类持有所有子 Manager 的引用，对外暴露统一 API（消息发送、配置读写、Agent 切换等）
  • Agent 生命周期 -- 创建、初始化、切换、删除 Agent；管理 Agent 的身份、记忆、工具集和对话循环
  • 会话编排 -- 管理 session 的创建/切换/关闭/列表，孤立执行、标题生成、会话压缩、缓存快照、提示词注入
  • 插件热加载 -- 扫描、加载、卸载、配置插件，注入 PluginContext（日志、网络、资源读写、UI 宿主能力）
  • Provider 注册 -- 管理所有 AI 模型提供商的声明式注册表，处理凭证读取、模型列表 CRUD 和多 Provider 兼容适配
  • LLM 调用统一入口 -- 非流式 LLM 调用的 HTTP 封装，以及标题摘要、翻译等轻量 utility 调用的纯函数工具集
  • 视觉桥接 -- 为不支持直接图片输入的模型提供辅助视觉分析（用 vision model 预处理图片 → 生成文本描述注入上下文）
  • 安全与权限 -- grant 注册表、设备凭证认证、会话权限模式分类（auto/operate/ask/read_only）、执行沙箱边界

2. 在整体架构中的位置

2.1 六层架构回顾

回顾 L0 总览中定义的 HanaAgent 六层架构（自上而下）：

Desktop ──→ Server ──→ Core (本模块) ──→ Hub ──→ Lib ──→ Plugins
  UI      网络/协议    引擎编排              协作   基础库   扩展能力

core/ 处于架构的第三层，是连接"前端/协议层"与"底层能力层"的关键枢纽：

  • 向上：被 server/ 通过 createHanaEngine() 工厂函数实例化，Engine 实例作为 HTTP/WS 请求处理的核心上下文；被 desktop/ 直接引用来提交用户消息
  • 向下：大量依赖 lib/（Pi SDK、工具实现、技能系统、channel 存储、browser 管理等）和 plugins/（图片生成、语音识别）
  • 水平：与 shared/ 层交互读取安全文件系统、模型引用、工具分类等共享工具

3.1 各层说明

第一层：Engine 外观 (Facade)

只有一个类 HanaEngine（2654 行），它是整个 core/ 模块的唯一入口。外部代码只与 HanaEngine 交互，不直接接触内部的 Manager。Engine 的构造函数接收 hanakoHome、productDir 等顶层目录，然后按序初始化所有子 Manager。它申明了 60+ 个私有字段（_agentMgr、_sessionCoord、_pluginManager 等），通过委托模式将所有业务逻辑转发给对应的 Manager。

第二层：领域 Manager

7 个核心领域管理器，每个从 Engine 中提取出来，不持有 engine 引用，通过构造器注入依赖。它们各自负责一个明确的领域：

  • AgentManager（1164 行）-- Agent 的扫描/初始化/创建/切换/删除，管理 agents 目录下的 YAML 配置
  • SessionCoordinator（5220 行）-- 整个 core/ 最大的文件，负责 session 的创建/切换/关闭/列表、isolated 执行、标题生成、activity session 提升，以及复杂的 prompt 拼装（模型选择、工具快照、技能快照、思考级别、缓存前缀、视觉注入等）
  • PluginManager（1712 行）-- 插件扫描、加载（TS/JS source + skills/routes/tools/agents/commands/providers 贡献目录）、卸载、配置、超时保护、UI 宿主能力声明
  • ProviderRegistry（1421 行）-- 声明式 Provider 注册表，管理所有已知 provider 的静态声明，合并插件声明与用户配置，读取凭证，管理模型列表的 CRUD
  • ConfigCoordinator（616 行）-- per-agent 模型选择、共享模型角色（utility/utility_large/vision）、搜索/utility 配置
  • PreferencesManager（942 行）-- 用户级全局配置（bridge、agent 排序、computer use 设置、通知、浏览器偏好、编辑器排版、网络代理等）
  • ModelManager（532 行）-- Pi SDK AuthStorage / ModelRegistry 基础设施，模型发现、同步、凭证解析、thinking level 规范化

第三层：专项服务

在 Manager 之下的一层专项服务，每个聚焦一个技术领域：

  • VisionBridge（892 行）-- 为 text-only 模型提供辅助视觉分析，用 vision model 预处理图片生成文本描述，带 LRU 缓存（256 条目）
  • llm-client / llm-utils（648 + 618 行）-- 非流式 LLM 调用的统一入口，直接 HTTP POST，不走 Pi SDK 的流式路径，用于 utility 短文本生成
  • SessionCompactor（689 行）-- 对长会话进行摘要压缩，支持 cache-preserving 模式（保留前缀缓存）和增量更新
  • UniversalMediaManager（1126 行）-- 统一的图片/视频生成管理，适配注册、任务提交、轮询和结果投递
  • BridgeSessionManager（1572 行）-- 外部平台（Telegram/Discord 等）的 session 管理，消息注入和 bridge 上下文拼装
  • 安全注册表 -- grant-registry（权限授予注册）、device-registry（设备凭证配对认证）、execution-boundary（执行边界控制）

第四层：外部依赖

core/ 大量依赖 lib/ 的基础设施（Pi SDK 的 SessionManager、AuthStorage、ModelRegistry，各种 tool 实现，技能解析器，channel 存储等），以及 plugins/ 中的扩展插件。
