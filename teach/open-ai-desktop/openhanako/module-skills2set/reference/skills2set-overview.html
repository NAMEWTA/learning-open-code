1. 模块概览

skills2set/ 是 HanaAgent 内置的4 套技能定义，以独立的 SKILL.md 文件 + 配套资源的形式存放在仓库中。它们构成了 HanaAgent 的"出厂能力库"：部分是面向终端用户的功能引导（user-guide），部分是面向开发者的创建工具（skill-creator、hana-plugin-creator），部分是面向 AI 自身的推理增强（quiet-musing）。

🔌

hana-plugin-creator

为 HanaAgent 创建插件。支持 beginner / developer 两种用户模式，可生成 direct / guided-react / professional-react 三种模板。

插件开发 Python 脚手架

🛠️

skill-creator

创建、评测、迭代优化 Agent Skill。包含完整评测流水线：测试用例生成 → 并行跑分 → 断言评分 → 基准报告 → 浏览器可视化。

技能开发 评测引擎

🧠

quiet-musing

深度推理协议。5 阶段思考框架（理解 → 拆解 → 多路径 → 执行 → 验证），用于复杂决策、架构设计、系统调试。

3. hana-plugin-creator — Hana 插件脚手架

3.1 目录结构

hana-plugin-creator/

├── agents/openai.yaml             # OpenAI 兼容 Agent 配置模板

│   ├── check_env.mjs              # Node 环境预检（检查 Python 可用性）
│   └── create_hana_plugin.py      # 插件脚手架生成器

    ├── hana-plugin-sdk-0.0.0.tgz        # @hana/plugin-sdk
    ├── hana-plugin-components-0.0.0.tgz  # @hana/plugin-components
    └── hana-plugin-protocol-0.0.0.tgz   # @hana/plugin-protocol

3.2 核心设计：双用户模式

技能在首次使用时通过用户反馈选择模式：

模式适用人群交互风格开场引导
beginner 新手、非技术人员 鼓励式、渐进式、手把手 三个引导问题：想要什么能力？自动还是手动？需要界面/文件/联网吗？
developer 有编程经验的开发者 简洁协作式、先展示能力面 直接询问目标贡献类型和集成边界

3.3 模板体系（3 种模板 x 4 种贡献类型）

模板类型：

  • direct — 无 npm/bundler，纯静态文件，适合新手第一��可运行插件
  • guided-react — React/Vite/SDK 起手模板，附带更友好的 README
  • professional-react — React/Vite/SDK 专业模板，含 package scripts 和类型化 UI 代码

贡献类型（--kind）：

  • tool — 受限插件，仅提供 tools/*.js
  • ui — 全权 WebView/iframe 页面和 Widget
  • full — 工具 + 生命周期/EventBus + WebView/iframe UI
  • provider — 全权 Provider 声明（providers/*.js），用于媒体/聊天能力贡献

3.4 能力地图

Hana 插件可提供的能力覆盖：

  • 工具与动作：Agent 可调用的 tools 和 slash 命令
  • 技能与知识：Skills、Agents、Knowledge 指导模型行为
  • UI 界面：WebView/iframe 页面、Widget、Card，使用 Hana 主题
  • Session 控制：通过 @hana/plugin-runtime 创建/管理会话和 Agent
  • 上下文注入：sendSessionMessage(..., { context }) 注入 RAG/世界设定
  • 非流式模型调用：sampleText() 用于摘要/分类/路由
  • 外部 HTTP 访问：ctx.network.fetch() + manifest 声明的 hosts
  • 用户资源访问：ctx.resources 读写本地文件/SessionFile/mounts
  • 媒体生成：generateImage() / generateMedia() / transcribeAudio()
  • Provider 贡献：图片/视频/语音 Provider（HTTP/OAuth/CLI 等后端）
  • Pi SDK 扩展：extensions/*.js 观察/转换 LLM 请求流水线

3.5 环境预检与脚手架命令

# 预检 Python 环境

# Beginner 示例

  --path examples/plugins --audience beginner --template direct

# Developer 示例

# Provider 插件示例

4. skill-creator — 技能创建与评测引擎

4.1 目录结构

skill-creator/
├── SKILL.md                       # 技能正文（481 行）
├── LICENSE.txt

├── scripts/
│   ├── check_env.mjs              # Node 环境预检
│   ├── __init__.py                # Python 包初始化
│   ├── quick_validate.py          # SKILL.md 快速校验
│   ├── package_skill.py           # 打包为 .skill 文件
│   ├── run_eval.py                # 运行触发评测
│   ├── run_loop.py                # 全自动描述优化循环
│   ├── improve_description.py     # 单次描述优化
│   ├── aggregate_benchmark.py     # 基准结果聚合
│   ├── generate_report.py         # 评测报告生成
│   └── utils.py                   # 工具函数
├── eval-viewer/
│   ├── generate_review.py         # 启动评测结果浏览器（Flask server）
│   └── viewer.html               # 浏览器端 UI（Outputs + Benchmark 双 Tab）
└── assets/
    └── eval_review.html           # 评测集审核 HTML 模板

4.2 核心工作流：迭代式技能开发

主循环：Draft → Test → Evaluate → Improve → Repeat
  1. Capture Intent — 理解用户意图，提取需求
  2. Interview & Research — 深入访谈边界情况、输入输出格式
  3. Write SKILL.md — 填充 name / description / body
  4. Create Test Cases — 写 2-3 个真实测试 prompt
  5. Spawn All Runs — 同回合并行启动 with-skill + baseline 子 Agent
  6. Draft Assertions — 在等待期间设计定量断言
  7. Grade & Aggregate — 评分 → 聚合 → 基准报告
  8. Launch Viewer — 在浏览器中展示评测结果（Outputs + Benchmark 双 Tab）
  9. Read Feedback — 用户在线填写反馈 → 读取 feedback.json
  10. Improve & Repeat — 基于反馈改进技能，进入下一轮迭代

4.3 评测流水线详解

Step 1: 并行跑分

针对每个测试用例，同时启动两个子 Agent：

  • with-skill 执行：加载目标技能完成任务
  • baseline 执行：不加载技能（新建技能时）或使用旧版本（改进已有技能时）

结果按 <workspace>/iteration-N/eval-{name}/ 组织，包含 with_skill/outputs/ 和 without_skill/outputs/（或 old_skill/outputs/）。

Step 2: 断言设计

在等待执行结果期间，设计可客观验证的定量断言。保存在 eval_metadata.json 和 evals/evals.json 中。

Step 3: 评分与基准

# 聚合基准数据
python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>

# 启动评测浏览器
python <skill-creator-path>/eval-viewer/generate_review.py \
  <workspace>/iteration-N \
  --skill-name "my-skill" \
  --benchmark <workspace>/iteration-N/benchmark.json

# 迭代 2+ 时附加对比
python ... --previous-workspace <workspace>/iteration-<N-1>

Step 4: Analyst Pass

在量化数据之外，进行分析师层面的观察：是否有些断言永远通过（无区分力）、高方差评测是否不稳定、时间/token 的 tradeoff 如何。

4.4 描述优化（Description Optimization）

SKILL.md 的 description 字段是 Skill 触发与否的首要机制。skill-creator 提供了一套完整的描述优化流水线：

  1. 生成评测集：20 个查询（8-10 should-trigger + 8-10 should-not-trigger），聚焦边界情况和近失匹配
  2. 用户审核：通过 HTML 模板可视化审核、编辑、导出
  3. 运行优化循环：python -m scripts.run_loop 自动拆分训练/测试集，每轮用 claude -p 多次评估触发率，用 extended thinking 提出改进方案
  4. 应用最优描述：选择在测试集（非训练集）上表现最好的描述，更新 SKILL.md frontmatter

4.5 技能写作指南

skill-creator 同时是一份"元技能写作指南"，包含以下范式：

范式说明
Progressive Disclosure三级加载：metadata (~100词) → SKILL.md body (<500行) → 按需加载 bundled resources
Principle of Lack of Surprise技能内容不得包含恶意代码、漏洞利用；不得误导用户
Imperative Form指令优先使用祈使句
Explain the Why解释为什么这样要求，而非堆砌 ALWAYS/NEVER 大写约束
Keep Lean移除不产生价值的指令，阅读对话记录判断是否过度引导
Look for Repeated Work如果多个测试案例都写同样的辅助脚本，应将其内化为 skill 的 bundled script

5. quiet-musing — 深度推理协议

5.1 设计目标

quiet-musing 是一套面向 AI Agent 的结构化推理框架。它与 MOOD（情感捕捉）互补 —— MOOD 抓取直觉和情绪，quiet-musing 管逻辑推理。

5.2 触发条件

满足以下任一条件即启用：

  • 问题有多个合理方案，需要权衡取舍
  • 需求模糊或隐含，不能直接动手
  • 涉及架构、策略、设计层面的决策
  • 调试需要系统性排查而非一眼看出答案
  • 任务会影响多个模块或有连锁效应
  • 用户明确要求深入分析

不启用：改 typo、回答事实性问题、单步操作。判不断就不启用，简单问题用复杂流程是浪费。

5.3 五阶段推理协议

Phase 1：理解

  1. 用自己的话复述问题
  2. 分清已知和未知
  3. 找到真正的问题（用户问的 vs 用户需要的）
  4. 标记不确定性

Phase 2：拆解

  1. 识别 2~5 个子问题
  2. 理清依赖关系（并行/串行）
  3. 用 todo 工具建立清单（粒度：一口气能做完、做完能验证）

Phase 3：多路径思考

  1. 至少想两条路
  2. 显式写出取舍（好处、代价、风险）
  3. 选路时给理由
  4. 保持可推翻 —— 走错了要有勇气换路

Phase 4：执行

  1. 单线程推进
  2. 动态调整 todo 清单
  3. 每一步都可验证
  4. 遇到阻塞不硬冲，停下来想为什么

Phase 5：验证

  • 回到 Phase 1 的问题复述，实际结果是否匹配？
  • 有没有遗漏的边界情况？
  • 改动是否引入了新问题？
  • 用户真正需要的东西是否被满足了？

5.4 推理姿态

像侦探，不像法官。 侦探跟着线索走，允许自己改变想法。法官一开始就要下结论。在 Phase 3 结束之前，你是侦探。

错误是线索。 发现自己想错了，要显式说出来：「刚才假设 X 成立，但看了代码发现不是，所以换个方向。」

深度匹配复杂度。 改 CSS 颜色不需要"多路径思考"。自适应，别教条。

跟用户同步。 Phase 1 和 Phase 3 是与用户对齐的好时机。

7. SKILL.md 格式约定

4 套技能共享相同的文件格式规范，遵循 AgentSkills 标准：

7.1 标准目录结构

skill-name/
├── SKILL.md           # 必需：技能定义文件
│   ├── YAML frontmatter  # 必需：name + description
│   └── Markdown body     # 技能正文指令
└── Bundled Resources  # 可选
    ├── scripts/       # 可执行代码（确定性/重复性任务）
    ├── references/    # 按需加载的参考文档
    └── assets/        # 输出中使用的文件（模板、图标、字体等）

7.2 YAML Frontmatter 字段

字段必需说明
name是技能标识符，用于系统引用
description是首要触发机制：包含技能功能 + 具体触发上下文。可中英双语，推荐写得稍"pushy"以避免不触发
compatibility否所需工具和依赖说明（如 hana-plugin-creator 注明需要 Node + Python 3.9+）
metadata否额外元数据（如 default-enabled: false 表示默认不启用）

7.3 Progressive Disclosure 加载机制

三级加载策略
Level 1: Metadataname + description，始终在上下文中（约 100 词）
Level 2: SKILL.md Body技能触发时加载（建议 < 500 行，超限时增加层级 + 指针）
Level 3: Bundled Resources按需加载，无容量限制（scripts 可直接执行无需加载到上下文）

7.4 实际应用中的格式变体

技能description 策略body 组织方式
hana-plugin-creator长描述：列举全部能力 + 触发条件按 Section 组织：First Contact → Capability Map → Workflow → SDK Rules → UI Rules → ...
skill-creator中英双语 + MANDATORY TRIGGERS 关键词按流程组织：Creating → Running Evals → Improving → Description Optimization
quiet-musing中英双语 + 正反例触发词 + DO NOT 排除词按 Phase 组织：理解 → 拆解 → 多路径 → 执行 → 验证 → 推理姿态
user-guideMANDATORY TRIGGERS + 高频中文触发词列表按功能模块组织：14 个章节 + 常见问题

8. 技能安装与加载机制

这 4 套技能作为 HanaAgent 的内置技能，随仓库源码分发。它们在 HanaAgent 中的位置和加载方式如下：

8.1 技能来源体系

来源存储位置说明
内置技能仓库 skills2set/软件自带，如本文档所述的 4 套技能
用户技能全局技能目录用户拖入或手动安装的 .skill / .zip / 文件夹
自学技能Agent 自身目录某个 Agent 在工作中自行创建并安装的技能
外部兼容技能.claude/skills/、.codex/skills/、.openclaw/skills/、.agents/skills/自动发现其他 AI 工具的技能文件

8.2 技能触发决策流程

  1. HanaAgent 将全部已启用技能的 name + description 注入 available_skills 列表
  2. AI 模型根据当前对话上下文和用户意图，决定是否需要查阅某个技能
  3. 简单单步操作不会触发技能（即使 description 匹配），因为模型可直接处理
  4. 复杂多步骤、专业化任务触发对应技能
  5. 技能触发后，完整 SKILL.md body 被加载到上下文
  6. 若 body 指向 bundled resources，按需进一步加载

关键：default-enabled: false 的技能（如 hana-plugin-creator）默认不在 available_skills 中，需要用户或 Agent 显式启用后才会参与触发判断。

8.3 按 Agent 隔离的技能开关

全局安装技能后，需要在设置 → 技能 → Agent 技能开关中为每个 Agent 单独启用。新建 Agent 默认继承当前已安装的用户技能快照，自学技能属于写出它的 Agent。这意味着不同 Agent 可以拥有不同的技能矩阵。

9. 跨 Skill 关系与触发边界

9.1 互操作关系

关系涉及 Skill说明
skill-creator 可创建 skill-creator 风格的 Skill skill-creator → 任意 skill-creator 本身定义了"如何写 Skill"的元方法论
