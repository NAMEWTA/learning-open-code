1. 模块职责一句话概括

shared/ 是 HanaAgent 的跨层共享类型定义与工具函数层——它不依赖于 core/、server/、desktop/ 任何一层，仅依赖 Node.js 内置模块（fs、path、os、url），向所有上层模块提供统一的配置 Schema、模型引用、安全密钥保管、错误总线、工作区管理、网络代理、MIME 类型定义等基础能力。它是整个项目的"契约层"和"安全基线"。

具体来说，shared/ 模块承担以下核心职责：

  • 配置契约 -- 声明所有配置字段的 global / agent scope，提供拆分与注入逻辑，一次性迁移 agent config 到全局 preferences
  • 实验特性 -- 定义实验状态（alpha/beta/deprecated/retired）、scope（global/agent/session）、值 schema 校验与规范化
  • 模型标识 -- 定义 (provider, id) 复合键的解析与校验规则，模型能力探测、已知模型词典查询
  • 安全基线 -- API Key 脱敏与 patching、安全文件读写、日志敏感信息自动遮蔽（PII + 凭证 + Token）
  • 错误体系 -- 统一的 AppError 类型（含 severity/category/retryable/i18nKey）、ErrorBus 事件总线（含去重与 breadcrumb）
  • 工作区管理 -- 默认工作区路径、workspace scope 规范化、输出目录解析、历史记录合并与过期清理
  • 网络代理 -- 系统/手动/直连三种代理模式，URL 校验与规范化，本地地址判定
  • UI 偏好 -- 浏览器行为、通知模式、快捷对话、侧边栏状态、访问权限 profile 等前端共享类型

2. 在整体架构中的位置

2.1 在六层架构中的定位

回顾 L0 总览中定义的 HanaAgent 六层架构（自上而下）：

Desktop -- Server -- Core -- Hub -- Lib -- Plugins
  UI      网络     引擎    协作   基础   扩展
          |         |       |      |      |
          +---------+-------+------+------+
                    |
               shared/ (本模块)
               跨层共享类型 + 工具函数

shared/ 不属于六层中的任何一层，而是作为横向切面被所有层共同依赖。它是整个项目中唯一可以安全地被任何模块 import 而不产生循环依赖的目录。

2.2 依赖关系

  • 被依赖方（所有上层）：core/ 通过 safe-fs 读写配置文件和密钥；server/ 通过 network-proxy 配置出站代理；desktop/ 通过 browser-preferences 控制浏览器行为；lib/ 通过 model-ref 查找模型；plugins/ 通过 error-bus 上报错误
  • 自身依赖：仅依赖 Node.js 内置模块（fs、path、os、url、crypto）以及少量 npm 包（js-yaml 用于配置解析）
  • 模块内依赖链：error-bus.ts 依赖 errors.ts + log-redactor.ts；safe-fs.ts 依赖 errors.ts + error-bus.ts；config-scope.ts 依赖 config-schema.ts；migrate-config-scope.ts 依赖 config-schema.ts

4. 核心 Schema / 类型定义概述

4.1 CONFIG_SCHEMA -- 配置字段单一事实来源

config-schema.ts 是整个应用配置系统的契约文件。每个可配置字段在此声明其 scope 和 engine 上的 getter/setter：

// 共 22 个 global scope 字段示例
locale          → scope: 'global', setter: 'setLocale',         defaultValue: ''
sandbox         → scope: 'global', setter: 'setSandbox',        defaultValue: true
thinking_level  → scope: 'global', setter: 'setThinkingLevel',  defaultValue: 'medium'
network_proxy   → scope: 'global', setter: 'setNetworkProxy',   getter: 'getNetworkProxy'

// 嵌套路径示例（最多 2 级）
'capabilities.learn_skills'  → 实际存储路径: preferences.learn_skills
'desk.heartbeat_master'      → 实际存储路径: preferences.heartbeat_master
'channels.enabled'           → 实际存储路径: preferences.channels_enabled
'bridge.permissionMode'      → 实际存储路径: preferences.bridge (嵌套对象)

关键设计决策：未在 CONFIG_SCHEMA 中声明的字段默认为 agent scope，写入各 Agent 各自的 config.yaml。global scope 字段写入共享的 preferences.json，跨 Agent 生效。

4.2 配置 Scope 拆分与注入流程

config-scope.ts 提供两个方向的数据流：

  • 向上拆分：splitByScope(partial) -- 前端发来的 config patch 按 schema 拆分，global 字段提取为 {key, value, setter} 数组调用 engine setter，剩余 agent 字段写回 config.yaml
  • 向下注入：injectGlobalFields(config, engine) -- 从 engine 通过 getter 读取 global 字段值注入到 config 对象，供 Agent 运行时使用

4.3 实验特性体系

experiments-schema.ts 定义了严格的实验特性生命周期：

  • status: alpha → beta → deprecated → retired（单向演进）
  • scope: global / agent / session（不同粒度开关）
  • value schema: boolean（toggle 开关）、enum（select 下拉）、number（数值调节）
  • presentation: toggle / select / segmented / paired_toggles（UI 呈现方式）

4.4 模型复合键 (provider, id)

model-ref.ts 建立了严格的模型引用纪律：

  • parseModelRef(ref) -- 宽松解析器，接受对象、"provider/id" 字符串、裸 "id" 三种历史格式
  • requireModelRef(ref) -- 严格校验器，任一字段缺失则抛错（运行时非 UI 层必须使用）
  • modelRefEquals(a, b) -- 复合键精确比较，不做按 id 降级匹配
  • modelRefKey(ref) -- 序列化为 "provider/id" 字符串，用于持久化和日志

设计原则："运行时不做按 id 猜测 provider 的智能推断"，provider 推断一律放在启动期一次性迁移或 UI 选择层。

4.5 AppError 错误类型体系

errors.ts 定义了 20 种预定义错误码，每种错误关联：

5. 安全机制

5.1 密钥保管（Secret Custody）

secret-custody.ts 实现了三层密钥保护机制：

┌─────────────────────────────────────────────────────────┐
│ 1. 默认密钥字段集合（不区分大小写）                       │
│    api_key, apiKey, token, botToken, appSecret,         │
│    secret, password, accessToken, refreshToken           │
├─────────────────────────────────────────────────────────┤
│ 2. 掩码常量: MASKED_SECRET = "********"                 │
│    • isMaskedSecretValue() → 判断是否为掩码值             │
│    • maskSecretValue()     → 将任意值替换为掩码           │
│    • maskObjectSecrets()   → 递归遍历对象，遮蔽所有密钥字段 │
├─────────────────────────────────────────────────────────┤
│ 3. Secret Patch 合并（resolveSecretPatch）               │
│    前端回传掩码值 "********" 时：                         │
│    → 检测到掩码 → 回填已有密钥（不覆盖）                   │
│    → 非掩码新值 → 使用新值                                │
│    → 无已有值 + 掩码 → 跳过（不写入空掩码）                │
└─────────────────────────────────────────────────────────┘

5.2 日志脱敏（Log Redaction）

log-redactor.ts 是浏览器兼容的 ESM 版本日志脱敏引擎，采用多层正则替换策略，依次处理 12 类敏感信息：

优先级遮蔽目标示例替换为
1已知路径（homeDir 等）/Users/alice/project~/project
2Base64 Data URIdata:image/png;base64,iVBORw...data:image/png;base64,[redacted]
3URL 内嵌凭证https://user:pass@hosthttps://[credentials]@host
4URL 查询参数密钥?token=abc123?token=[redacted]
5Authorization HeaderAuthorization: Bearer eyJ...Authorization: Bearer [redacted]
6Cookie / Set-CookieCookie: session=abc123Cookie=[redacted]
7密钥赋值表达式api_key="sk-abc123"api_key=[redacted]
8已知 API Key 模式sk-... / AKIA... / gsk_... / ghp_...[redacted]
9信用卡号1234-5678-9012-3456[credit_card]
10中国身份证号110101199001011234[id_card]
11邮箱地址user@example.com[email]
12长随机字符串（>40 字符）A1b2C3d4E5f6G7h8I9...[token]

关键设计：脱敏发生在日志输出前，ErrorBus 内部报告时自动调用 redactLogText，确保写入日志文件和 ErrorBus 监听器的内容已脱敏。

5.3 安全文件操作（Safe FS）

safe-fs.ts 提供一组带容错和错误上报的文件操作函数：

  • 容错读取：safeReadFile -- ENOENT 返回 fallback（不报错），EACCES 通过 ErrorBus 上报 FS_PERMISSION
  • 安全解析：safeReadJSON / safeReadYAML -- 解析失败通过 ErrorBus 上报 CONFIG_PARSE，返回 fallback
  • 原子写入：atomicWriteFile -- 先写临时文件再 rename，防止进程崩溃导致半写文件；支持 mode 参数（如 0o600 保护敏感凭证）

5.4 安全解析（Safe Parse）

safe-parse.ts 提供两个纯解析层面的安全包装：

  • safeParseJSON(text, fallback) -- 解析失败上报 CONFIG_PARSE 并截取前 100 字符作为上下文
  • safeParseResponse(res, fallback) -- HTTP Response 安全解析，非 2xx 状态码上报 FETCH_SERVER_ERROR

6. 跨层通信机制 -- ErrorBus 错误总线

6.1 ErrorBus 类设计

error-bus.ts 中的 ErrorBus 类是 shared/ 层唯一的发布-订阅事件总线，也是跨层错误通信的核心机制：

┌──────────────────────────────────────────────────────────┐
│                    ErrorBus 单例                          │
│                                                          │
│  _listeners[]      ← 各层注册的监听器                     │
│  _breadcrumbs[]    ← 最近 50 条面包屑（环形缓冲）          │
│  _recentFingerprints ← 去重 Map   │
│  _dedupeWindowMs   ← 去重窗口 5000ms                      │
│                                                          │
│  addBreadcrumb()   → 记录上下文面包屑                     │
│  report(error)     → 上报错误（自动脱敏 + 去重）           │
│  on(listener)      → 注册监听器                           │
│  off(listener)     → 移除监听器                           │
│                                                          │
│  _autoRoute(err)   → 自动路由:                            │

6.2 三层消费架构

ErrorBus 被项目中多个层级订阅和消费：

消费层用途典型监听器行为
shared/ 自身 内部错误上报 safe-fs.ts、safe-parse.ts 在文件/解析失败时调用 errorBus.report()
core/ 引擎级错误处理 Engine 初始化时注册 listener，将错误转发到日志系统和 Session 诊断面板
server/ HTTP 错误响应 根据 appErr.httpStatus 返回对应状态码，通过 WebSocket 推送到前端
desktop/ UI 错误展示 根据 route 渲染 toast 提示 / statusbar 警告 / boundary 错误页面
lib/ 工具层错误上报 工具执行失败、LLM 调用异常、权限拒绝等通过 ErrorBus 通知 core 层
plugins/ 插件错误上报 插件加载失败、执行异常通过 ErrorBus 通知 plugin-manager

6.3 去重机制

ErrorBus 内置 5 秒去重窗口：同 dedupeKey（默认取 error.code）的错误在 5 秒内只报告一次，防止 LLM 速率限制或网络抖动时刷屏。去重 Map 超过 200 条时触发惰性清理。

8. 关键设计原则

8.1 零上层依赖

shared/ 中的任何文件都不 import core/、server/、desktop/、lib/ 或 plugins/ 中的模块。外部依赖仅限：

  • Node.js 内置模块：fs、path、os、url、crypto
  • 唯一 npm 依赖：js-yaml（仅 migrate-config-scope.ts 和 safe-fs.ts 使用）

这保证了 shared/ 是所有上层模块的安全依赖原点，任何模块 import shared/ 都不会引入意外的传递依赖。

8.2 浏览器兼容分层

部分模块存在 CJS/ESM 双版本：

  • log-redactor.ts（ESM，browser-safe）与 log-redactor.cjs（CJS，Electron main process）
  • hana-runtime-paths.ts（ESM wrapper）与 hana-runtime-paths.cjs（CJS 实现）
  • workspace-output.ts（ESM wrapper）与 workspace-output.cjs（CJS 实现）

ESM 版本负责 re-export，CJS 版本包含实际实现，确保在 Electron main process（CJS）和 renderer（ESM）中均可使用。

8.3 严格类型校验

关键路径上拒绝"宽松降级"：

  • requireModelRef() -- provider 缺失则抛错，不做按 id 猜测 provider 的智能推断
  • normalizeExperimentDefinition() -- 任何字段不符合 schema 则抛错，不做静默修正
  • ProviderModelValidationError -- 将 provider ID 误填为 model ID 时明确拒绝并给出修正建议

8.4 安全优先

安全机制内建在 shared/ 层而非上层：

  • 密钥从不离开 shared/ 层明文传输 -- secret-custody 的 patch 合并确保掩码值不覆盖真实密钥
  • 日志在输出前脱敏 -- ErrorBus.report() 内部调用 log-redactor，上层监听器收到的已是脱敏内容
  • 文件操作原子化 -- atomicWriteFile 防止崩溃导致密钥文件损坏
  • 本地地址免认证 -- isLocalBaseUrl() 判定后允许 loopback 服务跳过 API Key

8.5 单一事实来源

多个关键概念在 shared/ 中只有一份定义：

  • CONFIG_SCHEMA -- 所有配置字段的 scope 唯一定义点
  • tool-categories -- 所有内置工具的 CORE/STANDARD/OPTIONAL/GLOBAL 分类唯一定义点
  • ERROR_DEFS -- 所有错误码的 severity/category/retryable 唯一定义点
  • search-providers -- 所有搜索 Provider 的注册表唯一定义点
