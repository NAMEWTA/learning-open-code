🎯 本节目标：学完你能分析三条核心异常路径（文件监听失效、Cron 错过触发、Agent 超时）的检测-降级-恢复机制，理解 skip/error/timeout 三级分类处理的差异，并完成跨课程的综合测验。

前三课我们沿着正向链路走完了完整流程。本课聚焦异常路径——当系统组件出问题时，HanaAgent 如何在"检测 → 降级 → 恢复"中保持可用性。

一、异常路径 1：文件监听失效

chokidar 文件监听器失效

触发条件：操作系统 inotify/FSEvents 句柄耗尽、被监听目录被删除或权限变更、chokidar 内部错误导致 watcher 静默停止。

核心特征：静默失效——不会有错误日志，因为没有"失败"，只是"不再触发"。

文件监听失效 → 系统响应： ├── [检测层] ActivityStore 无新记录 → 心跳间隔超时 │ └── reconcileOverdueRunning() 可将超时活动标记为 timeout │ ├── [降级层] Cron 调度器仍独立运行 (setInterval 不依赖 chokidar) │ └── 已注册的 cron job 不受影响，按时触发 │ ├── [恢复层] 应用重启时： │ ├── MountAwareFileService 重新初始化 root │ ├── ActivityStore.finalizeRunningAsInterrupted() 清理残留 │ └── chokidar watcher 重新注册 │ └── [预防层] MountAwareFileService 的 root 机制 ├── resolveInsideRoot() 使用 fs.realpathSync 实时验证 └── 不依赖 inotify，每次操作独立校验路径合法性

对用户的影响：文件拖拽后的即时响应可能延迟到下次 cron 轮询（最多 60 秒）或应用重启。已注册的定时任务不受影响。

二、异常路径 2：Cron 错过触发

系统休眠/高负载导致错过触发窗口

触发条件：操作系统休眠/待机、Node.js 事件循环被长时间阻塞、进程被 SIGSTOP 暂停后恢复。

核心设计：调度器使用 now >= nextRunAt 而非 === 判断，不会"跳过"任务。

批量追赶的风险控制：

  • 调度器逐个串行执行 job（for (const job of jobs)），避免 Promise.all 导致资源竞争。
  • 如果某个 job 执行超过 60 秒，下一个轮询因 _checking 锁跳过。
  • 长时间休眠后，nextRunAt 不会一次性跳过多个周期——每次执行只推进一个周期。

三、异常路径 3：Agent 执行超时

Agent LLM 调用卡住超过 20 分钟

触发条件：LLM API 无响应（网络断开/API 限流/服务端卡死）、prompt 导致无限循环、模型生成速度极慢。

超时处理完整链路： Promise.race([executeJob(job), setTimeout(20min, reject)]) │ ├─ [正常完成] resolve │ └─ clearTimeout(timer) + logRun("success") + markRun(true) │ └─ [超时触发] reject(new Error("execution timeout")) │ ├─ abortJob(job.id) → engine 层取消 LLM 调用 / 关闭 session │ ├─ catch 块处理： │ ├─ err.skipped? → logRun("skipped")，不推进 nextRunAt │ └─ 否 → logRun("error") + markRun(false) │ └─ CronStore 退避: min(2^errors * 60s, 1h) │ └─ onJobDone → 通知用户："xxx 任务已超时"

双重保险：即使 Promise.race 没触发（极端情况下 JS timer 被延迟），ActivityStore 的 reconcileOverdueRunning() 在下次应用启动时会检查所有 status === "running" 的条目，将超时的标记为 timeout。

四、三级错误处理对比

级别条件处理nextRunAt

💡 skip 与 error 的核心区别：skip 表示外部条件不满足（如 Agent 未就绪），下次应重试；error 表示真正执行失败，应退避后再试。CronScheduler 通过 err.skipped 属性区分。

五、历史污染修复：CronStore 的 JSONL 设计

CronStore 使用 .jsonl 追加写入而非覆盖式 JSON 来记录执行历史。这是一种防御性设计，防止"历史污染"问题：

// cron-runs/{jobId}.jsonl — 每行一条独立的 JSON，追加写入
{"ts":"2026-07-07T10:00:00Z","status":"success","runId":"r1"}
{"ts":"2026-07-07T10:30:00Z","status":"error","error":"timeout","runId":"r2"}
{"ts":"2026-07-07T11:00:00Z","status":"skipped","runId":"r3"}

为什么不用单个 JSON 文件？如果每次写入都覆盖整个 JSON 文件，一次写入错误可能损坏整个历史记录。追加式 .jsonl 即使某一行写入失败，也只影响当前条目，历史数据完好。而且读取时只需要逐行解析，跳过格式错误的行即可。

与 activities.json 的对比：ActivityStore 使用覆盖式 JSON（必要时使用原子写入），因为 activities.json 只需要当前状态（最近 100 条），不需要完整历史。CronStore 的 .jsonl 保留了完整的时间线，便于追溯"某次失败是什么原因"。

💡 选型洞察：覆盖式 JSON 适合"当前快照"（activities.json），追加式 .jsonl 适合"事件日志"（cron-runs）。同样的数据持久化需求，选型取决于"你需要的是状态还是历史"。

六、综合测验（跨课程）

Q1. 系统休眠 2 小时后恢复，CronScheduler 如何处理错过的任务？

A. 跳过所有错过的任务，只执行未来的任务 B. 使用 now >= nextRunAt 判断，立即执行所有错过窗口的 job，但串行逐个处理 C. 将 nextRunAt 直接跳到当前时间，不执行错过的任务 D. 报告错误并停止调度器
看答案B。调度器不跳过任务。使用 >= 判断而非 ===，所有错过的 job 会立即执行。但串行 + _checking 锁防止堆积。

Q2. 文件监听器静默失效后，哪种任务不受影响？

A. 文件拖拽触发的即时自动化任务 B. 已注册到 CronStore 的定时任务 C. 所有自动化任务都会失效 D. ActivityStore 的心跳检测
看答案B。CronScheduler 的 setInterval 不依赖 chokidar，已注册的 cron job 按时触发。只有依赖文件变更触发的即时任务会延迟。

Q3. 为什么需要 ActivityStore.reconcileOverdueRunning() 作为 Promise.race 超时的补充？

A. 因为 Promise.race 在 Node.js 中不可靠 B. 极端情况下 JS timer 可能被延迟，app 崩溃重启后 ActivityStore 的终检可清理残留的 running 状态 C. 因为需要记录超时的具体原因 D. 因为需要向用户发送额外的通知
看答案B。这是双重保险设计。如果 app 在超时前崩溃，Promise.race 无法触发，但下次启动时 ActivityStore 构造函数会清理残留的运行中条目。

Q4. 以下哪个是 HanaAgent 自动化系统没有采用的设计决策？

A. 调度与执行分离——CronScheduler 不涉及 LLM 调用 B. 确认即写入——Agent 创建建议，用户确认后才写入 CronStore C. 并行执行所有到期 job 以提高吞吐量 D. 版本兼容层——AutomationNormalizer 升级旧版 job 格式
看答案C。系统明确使用串行执行（for...of 而非 Promise.all），避免批量执行导致资源竞争。
