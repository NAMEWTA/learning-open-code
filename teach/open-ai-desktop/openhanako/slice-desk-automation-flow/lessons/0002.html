🎯 本节目标：学完你能画出从「文件变更被感知」到「Agent Session executor 构造完成」的完整 6 步数据流，理解 automation 工具的「建议-确认」双路径设计，以及 AutomationNormalizer 如何实现版本兼容。

二、第 1-2 步：文件感知与活动存储

MountAwareFileService（core/mount-aware-file-service.ts）是文件系统的统一入口，通过 root 机制将不同存储后端抽象为统一资源引用：

→root 隔离：所有文件操作必须在一个已注册的 root（挂载点）内。通过 resolveInsideRoot() 使用 fs.realpathSync 验证目标路径确实在 root 下，防止符号链接逃逸。
→能力检查：每个 root 声明 capabilities（list/read/write），操作前通过 requireCapability() 校验。
→路径穿越防护：normalizeSubdirOrThrow() 拒绝 ..、.、\ 和以 . 开头的路径段。
→原生路径披露控制：_discloseNativeRoot 标志仅在本地 owner principal 时为 true，远端 principal 一律不披露绝对路径。
💡 文件感知层不只是"读文件"，而是一个完整的安全沙箱。自动化任务能访问的目录严格受 root 注册表控制。

三、第 3 步：automation 工具入口

createAutomationTool()（lib/tools/automation-tool.ts）返回 Agent 可调用的工具定义，支持三个 action：list、create、update。

核心函数：

  • contextForTool()：根据是否 bridge session、是否跨 Agent 调用，决定 executionContext 的 cwd 和 workspaceFolders 来源。
  • normalizeSchedule()：将 scheduleType（at/every/cron）和 schedule 标准化。对 every 类型将分钟数转为毫秒。
  • genericAgentRun()：构造 agent_session 类型的 executor。

四、第 4 步：建议草稿 — "确认即写入"模式

这是 HanaAgent 自动化系统的核心安全设计。Agent 通过 automation 工具创建的不是直接生效的 cron job，而是 AutomationSuggestion：

automation tool execute(): │ ├── action === "list" │ └── 直接返回 cronStore.listJobs() │ └── action === "create" | "update" │ ├── autoApprove === true │ └── 直接 commitAutomationDraft() → 写入 CronStore │ └── autoApprove === false (默认) └── AutomationSuggestionStore.create() ├── 桌面端: 弹出交互卡片，用户点击确认 └── 文本平台: 返回 /apply 指引文本

AutomationSuggestionStore 是纯内存存储：

  • create()：生成 suggestionId（automation_时间戳_序号）和 shortCode（4 位随机数字）
  • apply()：按 sessionPath / bridgeSessionKey 查找并执行回调，完成后从 Map 中删除
  • shortCode 冲突处理：最多重试 200 次，全部冲突时降级为 5 位数字

applyConfirmedAutomationDraft() 将用户确认值与原始 draft 合并，仅合并 white-listed 字段（type/schedule/prompt/label/actorAgentId/executionContext/model），防止用户注入非法字段。

五、第 5 步：数据标准化 — 版本兼容层

AutomationNormalizer（lib/desk/automation-normalizer.ts）将旧版 job 格式升级到 v3 schema（trigger + executor 分离）：

旧格式 (type)旧格式 (schedule)新格式 (trigger)
"at"ISO 时间字符串{ kind: "at", schedule }
"every"分钟数字{ kind: "every", intervalMs }
"cron"5 字段表达式{ kind: "cron", expression }

Executor 同样从旧格式推导：direct_action + notify 转为 agent_session executor（附 migratedFrom 标记）；plugin_action 同理；无 executor 时从顶层字段构造默认 executor。

七、检验一下

Q1. automation 工具默认采用什么模式确认任务？

A. 直接写入 CronStore，无需用户确认 B. 创建 AutomationSuggestion，用户必须确认后才写入 CronStore C. 发送邮件给管理员审批 D. 等待 24 小时后自动确认
看答案B。这是核心安全设计——Agent 创建的是"建议"而非"命令"。autoApprove 默认为 false，用户必须确认（桌面端点击卡片 / 文本平台回复 /apply）后才生效。

Q2. MountAwareFileService 如何防止路径穿越攻击？

A. 使用正则表达式过滤输入 B. 通过 normalizeSubdirOrThrow() 拒绝 ..、.、\ 和以 . 开头的路径段 C. 将所有路径转为小写后比较 D. 使用 HTTP Only Cookie 保护
看答案B。normalizeSubdirOrThrow() 严格拒绝危险路径段，resolveInsideRoot() 使用 fs.realpathSync 验证目标路径确实在 root 目录下，防止符号链接逃逸。

Q3. applyConfirmedAutomationDraft() 合并哪些字段？

A. 所有 draft 中的字段无条件合并 B. 仅 white-listed 字段：type/schedule/prompt/label/actorAgentId/executionContext/model C. 仅 schedule 和 prompt 字段 D. 随机选择 3 个字段合并
看答案B。白名单机制防止用户确认时注入非法字段。合并后还会自动同步 executor 中的对应子字段。

Q4. AutomationSuggestionStore 的 shortCode 冲突时如何处理？

A. 直接抛出异常 B. 最多重试 200 次生成唯一码，全部冲突时降级为 5 位数字 C. 使用 UUID 替代 D. 暂停 1 秒后重试
看答案B。_nextShortCode() 循环尝试最多 200 次，确保在正常使用中几乎不会冲突。极端情况下降级为 5 位数字（100000 个可能值）。
