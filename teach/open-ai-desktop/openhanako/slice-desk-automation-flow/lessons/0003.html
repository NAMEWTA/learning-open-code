第 3 课 · 定时调度与执行：CronScheduler + Agent Session + 通知回传 调度执行

🎯 本节目标：学完你能讲清 CronScheduler 的 60 秒轮询机制、三层控制流（轮询/超时竞速/重启终检）、StudioCronService 的旧版迁移策略，以及 NotificationService 的双通道通知与幂等保护。

前两课我们覆盖了存储架构和任务创建流程。本课聚焦任务的定时触发与执行——CronScheduler 如何发现到期任务、如何控制超时、如何与 engine 层协作完成 Agent Session 执行，最后如何将结果通知用户。

一、CronScheduler：调度核心

createCronScheduler()（lib/desk/cron-scheduler.ts）是调度层核心，不涉及任何 LLM 调用。核心机制：

1轮询间隔：固定 60 秒（CHECK_INTERVAL），通过 setInterval 驱动。定时器不调用 unref()，确保空闲时也能可靠触发。
2防重入锁：_checking 布尔标志。如果上一个 checkJobs() 还在执行（例如某个 job 跑了超过 60 秒），下一个轮询周期直接跳过，不会堆积。
3超时竞速：使用 Promise.race([executeJob(job), timeout(timer)])。默认超时 20 分钟（DEFAULT_ACTIVITY_EXECUTION_TIMEOUT_MS）。超时时调用 abortJob(job.id) 取消正在执行的任务。
4优雅关闭：stop() 清除 interval 后 await _checkPromise，确保正在执行的 check 完成后才返回。
// CronScheduler 核心控制流
setInterval(checkJobs, 60_000)
  ├── cronStore.listJobs()
  ├── 过滤: job.enabled && job.nextRunAt <= now
  └── 逐个 executeJob(job)     // 串行，非 Promise.all
        │
        ├── Promise.race([
        │     executeJob(job),     // engine 层执行
        │     timeout(20min)       // 超时 abort
        │   ])
        │
        ├── 正常完成 → logRun("success") + markRun(true)
        ├── 超时     → abortJob(id) + logRun("error")
        └── skipped  → logRun("skipped")，不推进 nextRunAt

二、三层控制流

自动化任务的触发和执行涉及三层调度，每层有各自的超时和重试策略：

第 1 层：CronScheduler 轮询（60 秒间隔） setInterval(checkJobs, 60_000) → 过滤到期 job → 逐个 executeJob 第 2 层：executeJob 超时竞速（默认 20 分钟） Promise.race([executeJob(job), timeout(20min)]) 正常完成 → success | 超时 → abortJob | skipped → 重试 第 3 层：ActivityStore 启动时终检（应用重启时） finalizeRunningAsInterrupted() → 所有 running → interrupted reconcileOverdueRunning() → 超时的 running → timeout

控制流关键特性：

  • 防重入：_checking 标志位确保同一时间只有一个 checkJobs() 在执行。
  • skipped 不退避：当 job 抛出 { skipped: true } 时，不推进 nextRunAt，下一次轮询重试。
  • 真正失败有退避：通过 CronStore 内部的失败计数与指数退避（已在第 1 课详述）。

三、StudioCronService：多 Agent 统一管理

StudioCronService（core/studio-cron-service.ts）是对 CronStore 的 Studio 级包装，核心职责：

→延迟初始化：_getStore() 在首次访问时才创建 CronStore，路径为 {hanakoHome}/studios/{studioId}/desk/。studioId 变更时自动重建 store。
→旧版迁移（_importLegacyJobs）：扫描 agentsDir 下所有目录，读取每个 agent 的 desk/cron-jobs.json。通过 studioCronMigration marker 和 legacyRef { agentId, jobId } 机制避免重复迁移。
→ExecutionContext 校验：normalizeExecutionContext() 要求 cron job 的 executionContext 必须包含 kind（session_workspace / legacy_agent_home）、cwd、workspaceFolders。
// StudioCronService 旧版迁移流程
_importLegacyJobs(store, studioId):
  ├─ 扫描 agentsDir 下所有 agent 目录
  ├─ 对每个 agent 读取 {agentDir}/desk/cron-jobs.json
  ├─ 检查 studioCronMigration marker (version + status)
  │   已标记 "imported" → 跳过
  ├─ 对每个旧 job：
  │   ├─ 通过 legacyRef { agentId, jobId } 查找是否已迁移
  │   ├─ 未迁移 → _toStudioJob() → store.addImportedJob()
  │   └─ _copyLegacyRunHistory() → 复制 .jsonl 执行历史
  └─ 全部完成 → markLegacyCronStoreMigrated()
       写入 studioCronMigration: { version: 1, status: "imported" }

四、NotificationService：双通道通知与幂等保护

NotificationService（lib/notifications/notification-service.ts）支持两种投递通道：

通道方法特性
桌面_emitDesktop()title/body/agentId + desktopFocusPolicy（always/when_unfocused/when_session_unfocused）
Bridge_deliverBridge()仅 audience="owner"，通过 BridgeManager.sendProactive() 发送文本

幂等保护：使用 _idempotency Map，key 为 idempotencyKey，TTL 10 分钟。发现重复 key 时返回 { skipped: true } 而不重复发送。

五、ActivityStore 启动终检：崩溃恢复的第二道防线

除了 CronScheduler 的运行时超时控制，ActivityStore 在应用启动时执行终检，确保崩溃残留的 running 状态被正确清理：

// ActivityStore 构造函数中的双重终检
new ActivityStore(path, dir, { finalizeOrphanedRunning: true })
  │
  ├── finalizeRunningAsInterrupted()
  │     → 所有 status="running" → status="error", error="interrupted"
  │     → 场景：应用崩溃时，正在执行的任务被标记为中断
  │
  └── reconcileOverdueRunning()
        → 运行时间超过 timeoutMs → status="timeout"
        → 场景：即使 Promise.race 没触发，下次启动时也会清理

为什么需要终检？Promise.race 的超时检测依赖 JavaScript 事件循环正常运行。如果应用在超时触发前崩溃（OOM、SIGKILL、系统断电），timer 不会触发。下次启动时 ActivityStore 构造函数执行终检，确保所有残留的 running 状态被转换为 interrupted 或 timeout。

六、检验一下

Q1. CronScheduler 的轮询间隔是多少？

A. 10 秒 B. 30 秒 C. 60 秒 D. 120 秒
看答案C。CHECK_INTERVAL 固定为 60 秒，通过 setInterval 驱动。

Q2. CronScheduler 如何防止并发 check？

A. 使用数据库事务锁 B. 使用 _checking 布尔标志作为防重入锁 C. 使用 Redis 分布式锁 D. 每次 check 前重启进程
看答案B。_checking 标志确保同一时间只有一个 checkJobs() 在执行。如果上一个 check 超过 60 秒，下一个轮询直接跳过。

Q3. StudioCronService 如何避免重复迁移旧版 cron job？

A. 每次迁移前删除所有旧数据 B. 使用 legacyRef { agentId, jobId } 查找 + studioCronMigration marker C. 通过文件修改时间判断 D. 依赖用户手动确认
看答案B。legacyRef 追踪从 Agent 个人 cron 到 Studio cron 的对应关系；studioCronMigration marker（version + status）标记整个 agent 目录是否已完成迁移。

Q4. NotificationService 的幂等保护 TTL 是多长？

A. 1 分钟 B. 5 分钟 C. 10 分钟 D. 60 分钟
看答案C。idempotencyKey 在 _idempotency Map 中的 TTL 为 10 分钟。重复 key 返回 { skipped: true }。
