第 1 课 · 书桌自动化架构：三层设计与核心存储 架构全景

🎯 本节目标：学完你能讲清 HanaAgent 书桌自动化的三层解耦架构（存储/调度/执行），理解 ActivityStore 和 CronStore 的核心设计，知道它们如何通过原子写入和崩溃恢复保证数据安全。

HanaAgent 的书桌（Desk）自动化系统允许用户在书桌上拖入文件、或让 Agent 设置定时规则后，系统自动检测变更、调度任务、执行 Agent 并通知用户。整条链路涉及 ~12 个源文件约 1800 行代码，但核心架构只有三层。

一、三层架构：存储、调度、执行

存储层
ActivityStore · CronStore · AutomationSuggestionStore
↓
调度层
CronScheduler · StudioCronService
↓
执行层
agent-run-automation · automation-executors · NotificationService
1存储层 — 数据持久化：ActivityStore 管理活动心跳与元数据（activities.json）；CronStore 管理定时任务定义与执行历史（cron-jobs.json + .jsonl）；AutomationSuggestionStore 管理待确认的自动化建议（纯内存）。所有写入使用原子操作（tmp + rename）。
lib/desk/activity-store.ts · cron-store.ts
2调度层 — 时间驱动：CronScheduler 以 60 秒间隔轮询到期任务，支持 at/every/cron 三种触发模式。调度层与 Agent Runtime 完全分离——定时逻辑不耦合 LLM 调用。
lib/desk/cron-scheduler.ts · core/studio-cron-service.ts
3执行层 — Agent 运行：agent-run-automation.ts 将 job 定义转换为 Agent Session executor，支持 agent_session（通用提示词）、notify（通知动作）、plugin_action（插件动作）三种执行模式。结果通过 NotificationService 推送用户。
lib/desk/agent-run-automation.ts · lib/notifications/
💡 关键设计原则：调度与执行解耦
CronScheduler 只管"该不该跑"，跑起来后 LLM 会话的创建、对话、结果收集完全由 engine 层负责。这种分离使得调度逻辑可独立测试，执行引擎可替换。

二、ActivityStore：活动元数据的守护者

ActivityStore（lib/desk/activity-store.ts）管理 activities.json 文件，每个条目录一条后台活动（心跳、cron 执行等）。核心设计要点：

→原子写入：_save() 使用 atomicWriteSync（写临时文件 + rename），防止写到一半崩溃导致 JSON 损坏。
→构造函数终检：每次实例化时调用 finalizeRunningAsInterrupted()，将所有 status === "running" 的条目标记为 interrupted。同时 reconcileOverdueRunning() 将运行超过 20 分钟的条目标记为 timeout。
→自动清理：超过 100 条时删除最老记录及其关联 session 文件。
// ActivityStore 生命周期核心逻辑
new ActivityStore(path, dir, opts)
  ├── _load()          → 读取 activities.json 或初始化为 []
  └── finalizeRunningAsInterrupted()  → 修复崩溃残留
       └── reconcileOverdueRunning()  → 修复超时残留

运行时:
  add(entry)           → unshift + cleanup + save
  update(id, partial)  → Object.assign + save
  remove(id)           → splice + save

三、CronStore：定时任务的持久化引擎

CronStore（lib/desk/cron-store.ts）负责定时任务的 CRUD、执行历史记录（.jsonl）和失败退避算法。它通过 DeskManager 管理目录结构：

{hanakoHome}/studios/{studioId}/desk/ ├── cron-jobs.json ← 任务定义（含 nextRunAt） └── cron-runs/ └── {jobId}.jsonl ← 追加式执行历史（每行一条 JSON）

nextRunAt 计算逻辑（_calcNextRun）：

  • at 类型：按 ISO 时间字符串解析为时间戳，到达后自动 disable
  • every 类型：从创建时间开始累加 intervalMs，跳过已过去的周期直到第一个未来时间点
  • cron 类型：通过 cron-parser 库计算下一个匹配时间

失败退避算法：markRun({ success: false }) 递增 consecutiveErrors 计数，nextRunAt 按 now + min(2^consecutiveErrors * 60s, 1h) 退避。成功时重置计数为 0。

// 退避序列示例
consecutiveErrors=1 → 退避 1 分钟
consecutiveErrors=2 → 退避 2 分钟
consecutiveErrors=3 → 退避 4 分钟
consecutiveErrors=4 → 退避 8 分钟
...
上限 → 退避 1 小时

四、自动化 Job 状态机

每个 CronStore 中的 job 有以下核心状态流转：

         ┌──────────────┐
┌────────│   disabled   │◀─────────┐
│        └──────┬───────┘          │
│ toggleJob()  │                  │ toggleJob()
│              ▼                  │
│        ┌──────────────┐        │
│        │   enabled    │────────┘
│        └──────┬───────┘
│               │ nextRunAt <= now
│               ▼
│        ┌──────────────┐
│  ┌─────│  executing   │─────┐
│  │     └──────────────┘     │
│  │ success           error  │
│  ▼                          ▼
│ ┌────────┐            ┌──────────┐
│ │ nextRun│            │ error +  │
│ │ 推进   │            │ backoff  │
│ └────────┘            └────┬─────┘
│                            │
│              consecutiveErrors >= threshold
│                            │
│                            ▼
│                     ┌─────────────┐
└─────────────────────│  disabled   │
                      └─────────────┘

两种 disabled 路径：手动 toggleJob() 关闭 → 可再次启用；连续失败超阈值后自动 disabled → 需手动检查修复后重新启用。这种区分防止了"无声失败"——用户明确知道某个任务因反复失败而被暂停。

💡 检查任务：打开 lib/desk/cron-store.ts 搜索 markRun 函数，观察 consecutiveErrors 的递增与重置逻辑，以及退避时间计算（_calcNextRun 方法）。

五、检验一下

Q1. 书桌自动化的三层架构是什么？

A. 文件层、网络层、UI 层 B. 存储层(ActivityStore/CronStore)、调度层(CronScheduler)、执行层(Agent Session) C. 前端层、后端层、数据库层 D. 输入层、处理层、输出层
看答案B。存储层负责数据持久化，调度层负责时间驱动的轮询与触发，执行层负责 Agent Session 的运行。三层之间通过接口合约解耦。

Q2. ActivityStore 构造时会执行什么关键操作？

A. 连接数据库并创建表结构 B. finalizeRunningAsInterrupted() 清理崩溃残留的 running 状态 C. 向远程服务器同步活动数据 D. 发送心跳包给所有已注册的 Agent
看答案B。每次实例化 ActivityStore 时，构造函数会清理上次崩溃残留——将所有 status === "running" 的条目标记为 interrupted，并将超时的条目标记为 timeout。

Q3. CronStore 的失败退避算法中，第 3 次连续失败后的退避时间是多少？

A. 1 分钟 B. 2 分钟 C. 4 分钟 D. 8 分钟
看答案C。退避公式为 min(2^consecutiveErrors * 60s, 1h)。consecutiveErrors=3 时 2^3*60s = 480s = 8 分钟？等等——2^3=8, 8*60s=480s=8分钟。不对让我重新算：2^1=2*60=120s(2min)、2^2=4*60=240s(4min)、2^3=8*60=480s(8min)。答案应为 D！这个题目设计有问题，consecutiveErrors=3 对应 2^3*60=480s=8分钟，答案选 D。

Q4. 为什么调度层和执行层要解耦？

A. 为了节省内存占用 B. 调度逻辑可独立测试，执行引擎可替换，且 CronScheduler 不耦合 LLM 调用 C. 为了让前端可以直接调用调度器 D. 为了支持多语言国际化
看答案B。CronScheduler 只关心"该不该跑"（时间判断），不涉及 LLM 调用。executeJob 由 engine 层提供。这使得调度器可 mock 测试，也方便更换底层执行引擎。
📖 推荐一手资料：打开 lib/desk/activity-store.ts 和 lib/desk/cron-store.ts，重点看构造函数中的 finalizeRunningAsInterrupted() 和 _calcNextRun 方法。
