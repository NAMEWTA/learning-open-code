1. 模块定位

hub/ 是 HanaAgent 的后台任务调度与事件路由中心。它位于 Core（引擎编排层）和 Lib（核心库）之间，是连接用户消息、系统定时任务、频道群聊、Agent 私信等所有异步事件的"交通枢纽"。

hub/ 不定义任何业务逻辑本身 —— 它只负责"把正确的消息在正确的时间送到正确的 Agent 手中"，并通过统一的事件总线让系统的各个部分解耦通信。

一句话：hub/ 负责 Heartbeat 巡检、Cron 定时任务、频道消息轮转送达（Agent Phone 模型）、Agent 私信、事件订阅/广播，以及将这些后台活动统一路由到 Agent 执行引擎。

7. Scheduler — 双轨调度模型

7.1 双轨对比

Heartbeat（心跳巡检）

  • 粒度：每个 Agent 独立一个实例（Map<agentId, heartbeat>）
  • 数据源：lib/desk/heartbeat.ts → createHeartbeat()
  • 触发：基于 agent.config.desk.heartbeat_interval 定时轮询
  • 巡检内容：书桌文件变更检测 + 笺（Jian）任务执行
  • 总开关：engine.getHeartbeatMaster() && agent.config.desk.heartbeat_enabled
  • 并行性：所有 Agent 各自独立并行，不互斥
  • 切换行为：Agent 切换时通过 pauseForAgentSwitch() 暂停，resumeAfterAgentSwitch() 恢复

Studio Cron（定时任务）

  • 粒度：全局共享一个 CronScheduler 实例
  • 数据源：lib/desk/cron-scheduler.ts → createCronScheduler()
  • 触发：从 engine.getStudioCronStore() 读取任务，按 cron 表达式调度
  • 执行器：仅 agent_session 类型（通过 Agent 执行 prompt）
  • 防并发：同一 jobId 同时只运行一个（per-job 锁 AbortController）；同一 Agent 不同 job 可并行
  • 超时：通过 AbortController 支持超时 abort
  • 切换行为：不随 Agent 切换暂停（Studio 级任务独立运行）

7.2 统一执行路径

Heartbeat 和 Cron 都通过 _executeActivityForAgent() 执行：

// scheduler.ts — 统一执行入口 async _executeActivityForAgent(agentId, prompt, type, label, opts) { // 1. 确保 Agent 运行时存在（后台优先级） await engine.ensureAgentRuntime(agentId, { priority: "background" }); // 2. 隔离执行（非持久化 session） const result = await engine.executeIsolated(prompt, { agentId, persist: activityDir, signal, // AbortSignal 支持超时终止 activityType: type, ...restOpts, }); // 3. 生成活动摘要 → 写入 ActivityStore → EventBus 广播 const summary = await engine.summarizeActivity(sessionPath); engine.getActivityStore(agentId).add(entry); eventBus.emit({ type: "activity_update", activity: entry }, null); }
统一执行模型：Heartbeat 和 Cron 都走 executeIsolated()，不持久化 session，不暴露给用户 UI。Heartbeat 不传 withMemory（由 master 开关控制），确保巡检行为与用户会话记忆策略解耦。

7.3 Cron 执行防并发机制

Cron 任务使用三级控制防止并发写冲突：

  1. per-job 锁：_executingJobs Map 按 jobId 存储 AbortController，同一 jobId 重复触发时直接抛出 skipped 错误
  2. 超时 abort：CronScheduler 的 abortJob 回调通过 AbortController 终止超时任务
  3. finally 清理：无论成功或失败，finally 块删除 per-job 锁，确保下次可重试

8. ChannelRouter — Agent Phone 送达模型

8.1 核心概念

频道（Channel）= 内部群聊，多个 Agent 为成员。消息送达使用 Agent Phone 模型（非直接 LLM 调用）：

  1. ChannelTicker（lib/channels/channel-ticker.ts）轮询检测频道新消息
  2. Ticker 按 Agent 顺序依次回调 _executeCheck()
  3. ChannelRouter 构建 Phone Prompt（身份引导 + 行为引导 + 消息内容 + 投递窗口信息）
  4. 调用 AgentExecutor.runAgentPhoneSession() 将消息送入 Agent 的可复用 Phone Session
  5. Agent 必须使用三个频道专属工具之一完成本轮决策

8.2 频道专属工具

工具名操作副作用说明
channel_reply 发送频道消息 写入频道 .md 文件 + emit channel_new_message 只有此工具的 content 进入群聊；普通生成文本留在 Phone Activity
channel_pass 本轮不发言 无（仅记录 Activity） 可选 reason 和 mood 参数
channel_read_context 读取更早记录 无（只读） 默认读取最近 20 条，最多 50 条；数据源是频道 Truth，非 Phone Session
设计意图：Agent 不能在频道中"自由发言"——普通文本输出只记录在 Phone Activity 中，只有通过 channel_reply 写入的内容才进入频道聊天记录 Truth。这防止了 Agent 的内部思考/工具调用日志污染频道。

8.3 决策修复机制

如果 Agent 在本轮没有调用任何频道决策工具（channel_reply 或 channel_pass），ChannelRouter 会触发最多 1 次修复重试（MAX_CHANNEL_DECISION_REPAIR_ATTEMPTS = 1）：

  • 重试时 prompt 明确说明"这不是重复推送，而是让你补交本轮频道决定"
  • 重试仍失败 → 记录 implicitPass，活动标记为 "error"
  • 修复期间的诊断信息（diagnostics）携带到 Activity 记录中

8.4 频道记忆摘要

_memorySummarize() 在 Ticker 触发时将频道内容压缩为 1-3 条可搜索事实：

  • 使用 Agent 的 utility 模型执行（非主模型）
  • 记忆主开关关闭时跳过
  • 写入 Agent 的 FactStore（SQLite），session_id = channel-{channelName}
  • 已有事实会先删除再写入（全量替换，非追加）
  • 无检索价值时输出 NO_MEMORY，清空该频道的旧记忆

8.5 身份注入与防混淆

_formatChannelIdentityGuidance() 向 Agent 注入：

  • 频道完整成员表（含显示名，"你"标记）
  • 身份声明："你只代表自己，其他成员的发言、人设和记忆不属于你"
  • 频道 memory search 工具默认作用域隔离（conversation-scoped），跨频道检索需 cross_channel: true

9.2 多轮对话流程

5处理结果：[NO_REPLY] → break；正常文本 → 写入双方 dm/ 文件 + emit dm_new_message
6多轮控制：含 <done/> → break；否则交换角色继续（fromId ↔ toId），最多 3 轮

10.3 关键设计决策

  • 内省剥离：执行完成后自动移除 MOOD 块和 XML 内部叙述标签（stripClosedInternalNarrationBlocks()）
  • Session 复用：频道会话复用同一个 session 文件以保持上下文连续性；工具和 system prompt 在复用时可更新
  • 模型覆盖：agentPhoneModelOverride 允许频道/DM 使用独立模型（与 Agent 主模型分离），通过 frontmatter 配置
  • Abort 集成：注册到 engine.registerAgentPhoneAbortHandler()，频道关闭时批量终止所有进行中的 Phone 会话
  • 诊断信息：returnDiagnostics: true 时返回 { text, diagnostics: { toolCallCount, toolCallNames, stopReason, ... } }

11. EventBus — 事件总线机制

11.1 双模式通信

发布/订阅 (Pub/Sub)

  • API：subscribe(callback, filter?) / emit(event, sessionPath?)
  • 过滤：filter.sessionPath (按 session 隔离) + filter.types (Set 快速匹配)
  • 索引优化：无 sessionPath → _globalSubs Set；有 sessionPath → _sessionIndex Map，emit 时 O(1) 查找相关订阅者
  • 典型事件：activity_update、channel_new_message、dm_new_message、cron_job_done、session_created

请求/响应 (Req/Res)

  • API：handle(type, handler) / request(type, payload, options?)
  • 超时：默认 30 秒（BusTimeoutError）
  • SKIP 链：handler 返回 EventBus.SKIP → 继续尝试下一个 handler（链式责任）
  • 错误：BusNoHandlerError (无 handler) / BusTimeoutError (超时)
  • 能力目录：通过 EventBusCapabilityDirectory 注册声明（schema + permission + stability）

11.2 emit 索引机制

EventBus 维护两级索引，避免每次 emit 遍历全部订阅者：

  • 全局广播订阅者 (_globalSubs)：每次 emit 都需要通知的订阅者（无 sessionPath 过滤），使用 Set 存储 subscriber ID
  • 按 sessionPath 索引 (_sessionIndex)：Map<sessionPath, Set<subscriberId>>，emit 时只遍历匹配路径的订阅者
  • 类型过滤：在回调中 O(1) 检查 filter.types Set（而非 Array.includes O(n)）
// event-bus.ts — emit 核心逻辑 emit(event, sessionPath) { const ids = this._globalSubs; // 广播订阅者 const sessionIds = sessionPath ? this._sessionIndex.get(sessionPath) // session 相关订阅者 : null; for (const id of ids) notify(id); // O(广播订阅者数) if (sessionIds) { for (const id of sessionIds) notify(id); // O(session 相关订阅者数) } }

13. GuestHandler — 访客消息包装器

最简子模块（61 行），职责单一：

  • 消息前缀：在文本前添加 [来自 {name}] 或 [From {name}]
  • 上下文注入：群聊时注入 当前对话来自群聊。，非群聊时注入 当前对话来自外部访客。

  • 不暴露隐私：contextTag 只说明来源类型，不暴露主人信息

14. FreshCompactMaintainer — 每日上下文压缩

由 Scheduler 通过 createFreshCompactDailyScheduler() 每天触发一次：

  • 遍历所有 Agent 实例
  • 从 bridgeSessionManager.listDailyFreshCompactTargets() 获取需压缩的 Bridge 会话
  • 检查 isFreshCompactAlreadySatisfied() — 已满足则标记跳过
  • 调用 agent.memoryTicker.flushSessionAndCompile() + freshCompactSession()
  • 每个 job 之间延迟 5 秒（delayBetweenJobsMs = 5000），防止瞬时负载
  • 运行中标记 _running = true，防重入

16. 设计原则总结

#原则体现
1 单向依赖 Hub 持有 Engine 引用；Engine 通过 setHubCallbacks() + setEventBus() 接收注入，不感知 Hub
2 统一消息入口 所有消息（用户、Bridge、定时任务、频道）均经 Hub.send() 优先级路由表分发
3 后台隔离执行 Heartbeat / Cron / Channel 均走 executeIsolated()，不持久化 session，approvalPolicy: deny_on_prompt
4 Agent Phone 模型 频道和 DM 不直接操作 LLM，通过"手机送达"将消息送入 Agent 会话，Agent 使用专用工具回复
5 事件驱动解耦 组件间、Hub-Engine-前端间通过 EventBus 通信，支持 Pub/Sub + Req/Res 双模式
6 能力声明式 EventBus handler 均有声明式 capability（schema + permission + stability），API 契约可发现可验证
7 记忆作用域隔离 频道记忆和 Agent 主记忆分离，channel memory search 默认不跨频道检索（conversation-scoped）
8 防并发与防重入 Cron per-job 锁 (AbortController)、DM processing Map + cool down、Scheduler _running 标记

在 L0 架构中的位置

  • 向上 (Core/Engine)：Hub 持有 Engine 引用，向 Engine 发送 prompt、注册回调、注入事件总线
  • 向下 (Lib)：Hub 使用 lib/desk（Heartbeat、CronScheduler、AutomationExecutors）、lib/channels（Ticker、Store、Mentions）、lib/conversations（AgentPhoneSession、Projection、Runtime）、lib/memory（FactStore）
  • 横向 (Server)：Hub 事件经 EventBus → Engine → WebSocket 广播前端
  • 外部 (Bridge)：GuestHandler + send() 路由处理来自 Telegram/飞书等平台的消息
