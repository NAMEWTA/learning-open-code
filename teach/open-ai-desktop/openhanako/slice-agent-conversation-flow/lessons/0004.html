第 4 课 · LLM 调用与流式处理 LLM & 流式

🎯 本节目标：学完你能画出 4 种 LLM 协议适配的请求构造差异，能解释 ThinkTagParser / MoodParser / CardParser 三重解析器链的流式解析管道模型，能说清流式事件 Ring Buffer 与断线续传机制。

上一课我们到达了 Pi SDK 入口。本课先看 Pi SDK 底层的 LLM HTTP 调用（L7），再看响应如何通过 EventBus 回流到 WebSocket 广播到前端（L8）——这是对话链路响应路径的完整演绎。

一、LLM callText——4 种 API 协议适配

core/llm-client.ts 的 callText() 提供统一的非流式 LLM 调用入口（用于标题生成、摘要等 utility 场景）。主对话的流式调用走 Pi SDK 内部的 completeSimple 管道，但同样经 lib/llm/provider-client.ts 做 URL 构造和认证注入。

// core/llm-client.ts -- callText 协议适配（真实代码片段）
// 3. 按协议构造请求——支持 4 种 API 格式
let endpoint, headers, body;
if (api === "anthropic-messages") {
  endpoint = appendProviderApiPath(base, "/v1/messages");
  headers = { "Content-Type": "application/json", "anthropic-version": "2023-06-01" };
  if (apiKey) headers["x-api-key"] = apiKey;
  body = { model: modelId, max_tokens: ..., system: mergedSystem, messages: ... };
} else if (api === "openai-codex-responses") {
  // Codex OAuth: 需要 account-id header + /codex/responses endpoint
  const accountId = resolveCodexAccountId(modelObj, apiKey);
  endpoint = resolveCodexResponsesUrl(baseUrl);
  headers = { "Content-Type": "application/json", "chatgpt-account-id": accountId };
  body = { model: modelId, instructions: mergedSystem, input: ..., stream: true };
} else if (api === "openai-responses") {
  endpoint = `${base}/responses`;
  body = { model: modelId, instructions: mergedSystem, input: ... };
} else {
  // 默认 OpenAI Completions API
  endpoint = `${base}/chat/completions`;
  body = { model: modelId, messages: [...systemMsg, ...normalizedMessages] };
}
协议认证方式System Prompt 字段消息格式
anthropic-messagesx-api-key headersystem（顶层字段）messages[]（user/assistant）
openai-codex-responsesOAuth + chatgpt-account-idinstructionsinput[]（扁平数组）
openai-responsesBearer tokeninstructionsinput[]
默认 (OpenAI Completions)Bearer tokenmessages[0] role=systemmessages[]（system/user/assistant）
💡 关键差异：System Prompt 在 Anthropic 中是顶层 system 字段，在 OpenAI Responses 中是 instructions 字段，在 OpenAI Completions 中是 messages[0] 伪装为消息。同一段 System Prompt 需要 3 种不同的注入方式——这就是 normalizeProviderPayload 存在的意义。

二、流式响应回流——EventBus to WebSocket to React

Pi SDK 产生的流式事件通过 EventBus 传回 chat.ts 的 hub.subscribe 回调，经过三重解析器链处理后再通过 WebSocket 广播到前端：

// server/routes/chat.ts -- hub.subscribe 事件回调（真实代码片段）
hub.subscribe((event, sessionPath) => {
  const ss = sessionPath ? getState(sessionPath) : null;

  if (event.type === "message_update") {
    const sub = event.assistantMessageEvent?.type;

    if (sub === "text_delta") {
      // ★ 文本增量通过三重解析器链
      const delta = event.assistantMessageEvent.delta || "";
      ss.thinkTagParser.feed(delta, (tEvt) => {
        if (tEvt.type === "text") feedMoodPipeline(tEvt.data);
        else if (tEvt.type === "think_text") emitStreamEvent(..., "thinking_delta", ...);
      });
    }
  } else if (event.type === "tool_execution_start") {
    ss.hasToolCall = true;
    emitStreamEvent(sessionPath, ss, {
      type: "tool_start", id: event.toolCallId, name: event.toolName,
      args: summarizeToolStartArgs(event.toolName, event.args),
    });
  } else if (event.type === "tool_execution_end") {
    emitStreamEvent(sessionPath, ss, {
      type: "tool_end", id: event.toolCallId, name: event.toolName,
      success: !event.isError, details: event.result?.details,
    });
  } else if (event.type === "turn_end") {
    // 空回复检测 + Token 统计 + 清理
    if (!ss.hasOutput && !ss.hasToolCall && !ss.hasThinking
        && !ss.hasError && !ss.isAborted) {
      broadcast({ type: "error", message: t("error.modelNoResponse"), sessionPath });
    }
    finishStreamingState(ss, sessionPath);
  }
});

三、三重解析器链——流式解析管道

文本增量（text_delta）不是直接广播的，而是依次通过三个解析器，各自产生对应的事件类型：

1ThinkTagParser——识别 <think>...</think> 标签，分离模型的思考过程。产生 text（普通文本）和 think_text（思考内容）事件。
core/events.ts
2MoodParser——在 ThinkTagParser 输出的 text 事件中识别情绪表情标记。产生 mood_start / mood_end 事件。
core/events.ts
3CardParser——识别交互式卡片语法（如 <card>...</card>）。产生 card_start / card_delta / card_end 事件。
core/events.ts
// 流式解析管道模型（简化示意）
text_delta 流入
  → ThinkTagParser.feed(delta)
    → type="text" → MoodParser.feed(text)
      → type="mood_start" / "mood_end"
      → CardParser.feed(text)
        → type="card_start" / "card_delta" / "card_end"
    → type="think_text" → 直接产生 thinking_delta 事件
管道设计的精妙之处：每个解析器只关心自己的语法，互相独立。ThinkTagParser 先剥离思考标签，MoodParser 和 CardParser 在「净文本」上工作。这是编译器前端「词法分析 → 语法分析」流水线模式在流式文本处理中的经典应用。

四、流式事件 Ring Buffer 与断线续传

前端切走 session 后，流式事件仍在产生。系统通过 Ring Buffer 暂存事件，前端切回时请求补发：

// server/session-stream-store.ts（概念代码）
export function createSessionStreamState() {
  return {
    streamId: null,
    events: [],           // Ring buffer，最多保留最近 N 个事件
    maxEvents: 500,
    nextSeq: 1,
    isStreaming: false,
  };
}

// 前端切回 session 时请求补发离屏期间的流式事件
if (msg.type === "resume_stream") {
  const ss = getExistingState(currentPath);
  const resumed = resumeSessionStream(ss, {
    streamId: msg.streamId, sinceSeq: msg.sinceSeq
  });
  wsSend(ws, createStreamResumeWsMessage({
    sessionPath: currentPath, streamId: resumed.streamId,
    events: resumed.events,          // 补发的事件列表
    reset: resumed.reset,            // 溢出标记——事件太多需全量刷新
    isStreaming: resumed.isStreaming,
  }));
}
💡 reset 标记的作用：当 Ring Buffer 溢出（事件超过 500 条）时，无法补发全部事件。此时设置 reset: true，前端放弃增量更新，改为全量拉取最新消息状态。这是「最终一致性」在流式 UI 中的工程实现。

五、检验一下（提取练习）

Q1. callText() 支持几种 LLM 协议？

A. 1 种——只有 OpenAI Completions B. 2 种——OpenAI + Anthropic C. 3 种——OpenAI + Anthropic + Google D. 4 种——anthropic-messages、openai-codex-responses、openai-responses、默认 OpenAI Completions
看答案D。4 种协议：anthropic-messages（x-api-key）、openai-codex-responses（OAuth + account-id）、openai-responses（instructions 字段）、默认 OpenAI Completions（messages 数组）。每种协议的 System Prompt 注入方式和消息格式都不同。

Q2. 三重解析器链的执行顺序是什么？

A. CardParser → MoodParser → ThinkTagParser B. ThinkTagParser → MoodParser → CardParser C. 三个解析器并行执行 D. MoodParser → ThinkTagParser → CardParser
看答案B。ThinkTagParser 先剥离 <think> 标签，产生纯文本后交由 MoodParser 识别情绪标记，最后 CardParser 识别交互式卡片。这是顺序管道——前一个解析器的输出是后一个的输入。

Q3. 为什么 callText 不走 Pi SDK 的 completeSimple（强制流式）？

A. Pi SDK 不支持非流式调用 B. utility 短文本（50-200 token）不需要流式传输，直接 HTTP POST 避免 SSE 延迟膨胀 C. callText 不需要 LLM 调用 D. Pi SDK 只支持 OpenAI 协议
看答案B。callText 用于标题生成、摘要等短文本场景（50-200 token），不需要流式传输。直接 HTTP POST 可以避免 DashScope 等 Provider 的 SSE 延迟膨胀问题。

Q4. 断线续传中 reset 标记意味着什么？

A. 重新连接 WebSocket B. Ring Buffer 溢出——前端应放弃增量更新，改为全量拉取最新状态 C. 重置 LLM session D. 清空用户消息历史
看答案B。Ring Buffer 最多保留 500 条事件。溢出时设置 reset=true，前端不再尝试增量补发，而是全量拉取最新消息列表。这是流式 UI 中「最终一致性」的工程实现。
