第 3 课 · Agent 初始化与 System Prompt 拼装 Agent & Prompt

🎯 本节目标：学完你能列出 Agent.init() 的 7 个初始化步骤，能背出 System Prompt 的 10 段来源及其加载顺序，能解释 SessionCoordinator 的 promptSession() 在前半段（请求路径）中做的 6 项准备工作。

消息经过 Hub 路由和防重入检查后，到达 SessionCoordinator。但在此之前，Agent（助手实例）必须已经完成初始化——包括 System Prompt 的拼装。这一课聚焦 Agent 初始化（L5）和 SessionCoordinator 的请求准备（L6 前半段）。

一、Agent 初始化——7 步启动助手实例

core/agent.ts 中的 Agent 类代表一个助手实例。每个 Agent 有独立的数据目录（~/.hanako/agents/{id}/），包含配置、记忆、工具和自定义 System Prompt：

// core/agent.ts -- Agent.init()（真实代码片段，约 500 行初始化核心）
async init(log, sharedModels, resolveModel) {
  if (this._runtimeInitialized) return;

  // 0. 兼容性检查（目录结构、数据库文件、配置文件完整性）
  await runCompatChecks({ agentDir: this.agentDir, factoriesDir: ... });

  // 1. 加载 config.yaml
  this._config = loadConfig(this.configPath);

  // 2. 身份 + 语言
  const isZh = String(this._config.locale || "").startsWith("zh");
  this.userName = this._config.user?.name || (isZh ? "用户" : "User");
  this.agentName = this._config.agent?.name || "Hanako";

  // 3. 记忆系统：FactStore（DuckDB）+ SessionSummaryManager
  this._factStore = new FactStore(this.factsDbPath);
  this._summaryManager = new SessionSummaryManager(this.summariesDir);

  // 4. Memory Ticker（定期触发记忆整理，默认每 5 分钟）
  this._memoryTicker = createMemoryTicker({
    factStore: this._factStore, summaryManager: this._summaryManager,
    getConfig: () => this._config, ...
  });

  // 5. 构建工具集（20+ 工具，按 agent.tools 配置开关）
  this._buildTools(sharedModels, resolveModel);

  // 6. 组装 System Prompt
  this._systemPrompt = this.buildSystemPrompt({ ... });

  // 7. Desk 系统（定时任务 CronStore + DeskManager）
  if (this._cronEnabled) {
    this._cronStore = new CronStore(this.cronsDir);
    this._deskManager = createDeskManager({ ... });
  }
  this._runtimeInitialized = true;
}
0兼容性检查——确保目录结构、数据库文件、配置文件完整，旧版本迁移。
runCompatChecks()
1加载配置——config.yaml 中定义 agent 身份、语言、启用的工具、模型偏好等。
loadConfig(this.configPath)
2身份解析——从配置中提取 userName/agentName/locale，后续注入 System Prompt。
this._config.user / agent
3记忆初始化——FactStore（DuckDB 存事实）和 SessionSummaryManager（JSONL 存摘要）。
FactStore + SessionSummaryManager
4Memory Ticker——后台定时器，每 5 分钟触发记忆整理（事实提取 + 摘要生成）。
createMemoryTicker()
5工具注册——20+ 工具（文件读写、命令执行、浏览器、搜索、Subagent 等），按配置过滤。
this._buildTools()
6System Prompt 拼装——10 段来源合并为最终 System Prompt（下一节详解）。
this.buildSystemPrompt()

二、System Prompt 10 段拼装——对话质量的基石

Agent 的 buildSystemPrompt() 从 10 个来源拼装最终发送给 LLM 的 System Prompt。这是决定对话质量和 Agent 行为的核心：

  1. Identity（身份）——来自 identity.md 或 identity-templates/{yuan}.md，定义 Agent 人格
  2. Yuan（能力元）——来自 yuan/{yuan}.md，定义输出协议和能力边界
  3. Ishiki（意识）——来自 ishiki.md 或 ishiki-templates/{yuan}.md，定义行为准则
  4. 平台提示——操作系统、Shell、时区、当前日期
  5. 工作区指令——CLAUDE.md、AGENTS.md、CODEBUDDY.md 等项目规则文件
  6. 工具列表 + 使用说明——20+ 工具的名称、参数、用法说明
  7. 用户档案——user.md（用户偏好、背景信息）
  8. 记忆注入——FactStore 最近事实 + SessionSummary 历史摘要
  9. 团队花名册——多 Agent 场景下其他 Agent 的描述
  10. Provider 补丁——针对特定模型/Provider 的提示词修正
// core/agent.ts -- personality getter（真实代码片段）
get personality() {
  const isZh = String(this._config.locale || "").startsWith("zh");
  const fill = (text) => text
    .replace(/\{\{userName\}\}/g, this.userName)
    .replace(/\{\{agentName\}\}/g, this.agentName);
  const yuanType = this._config?.agent?.yuan || "hanako";

  const identityMd = readFile(path.join(this.agentDir, "identity.md"))
    || readFile(path.join(this.productDir, "identity-templates", `${yuanType}.md`));
  const yuanMd = this._readYuan();
  const ishikiMd = readFile(path.join(this.agentDir, "ishiki.md"))
    || readFile(path.join(this.productDir, "ishiki-templates", `${yuanType}.md`));

  return fill(identityMd) + "\n\n" + fill(yuanMd || "")
       + "\n\n" + fill(ishikiMd);
}
💡 模板优先级：Agent 目录下的自定义文件优先级高于 productDir 下的模板。identity.md → identity-templates/{yuan}.md 的回退链允许用户覆盖默认人格。模板中的 {{userName}} 和 {{agentName}} 占位符在运行时填充。

三、SessionCoordinator promptSession——LLM 调用前的 6 项准备

core/session-coordinator.ts（5220 行）是对话链路的真正执行者。promptSession() 在提交到 Pi SDK 之前做 6 项准备工作：

// core/session-coordinator.ts -- promptSession 流程（真实逻辑简化）
async promptSession(sessionPath, text, opts) {
  // 1. 确保 session 已加载（延迟加载/缓存命中）
  let entry = this._getSessionEntryByPath(sessionPath);
  if (!entry) {
    await this.ensureSessionLoaded(sessionPath);
    entry = this._getSessionEntryByPath(sessionPath);
  }

  // 2. 视觉预处理——图片 to 文本模型适配（Vision Bridge）
  const visionResult = await prepareVisionInputForTextOnlyModel({
    targetModel: entry.session.model, text, opts, sessionPath,
    getVisionBridge: () => engine?.getVisionBridge?.(),
  });

  // 3. 图片预处理——resize/格式转换/压缩
  const imageResult = await prepareModelImageInputsForPrompt({ text, opts });

  // 4. 模态校验——视频/音频输入是否被当前模型支持
  assertVideoInputSupported(entry.session.model, opts?.videos);
  assertAudioInputSupported(entry.session.model, opts?.audios);

  // 5. 构建 Pi SDK prompt 参数
  const promptOpts = buildPromptMediaOptions(opts);

  // 6. ★ 提交到 Pi SDK——LLM 调用入口
  await entry.session.prompt(text, promptOpts);
}
⚠️ Vision Bridge 的作用：当用户使用的模型不支持视觉输入（如纯文本模型），Vision Bridge 会将图片发送给支持视觉的模型做描述，然后将描述文本注入到消息中替代原始图片。这使纯文本模型也能「看」图片——但代价是额外的 API 调用和 token 消耗。
Pi SDK 的角色：SessionManager（来自 lib/pi-sdk）负责实际的 LLM 对话管道——消息历史 JSONL 持久化、上下文窗口 token 计数、Compaction 摘要生成、流式 SSE 事件解析。SessionCoordinator 是 Pi SDK 的「适配层」，将 Hana 的配置翻译为 Pi SDK 的调用参数。

四、检验一下（提取练习）

Q1. Agent.init() 中，System Prompt 是在哪一步拼装的？

A. 第 1 步——加载 config.yaml 时 B. 第 5 步——构建工具集时 C. 第 6 步——在所有依赖（配置、工具、记忆）就绪后 D. 第 7 步——Desk 系统初始化时
看答案C。System Prompt 拼装是第 6 步，放在配置加载（步骤 1）、工具注册（步骤 5）之后。因为 Prompt 需要引用工具列表和使用说明，所以必须在工具注册完成后才能拼装。

Q2. 以下哪个不是 System Prompt 的 10 段来源之一？

A. Identity（身份定义）——identity.md B. Yuan（能力元）——yuan/{yuan}.md C. 数据库 Schema——db/schema.sql D. 记忆注入——FactStore + SessionSummary
看答案C。数据库 Schema 不是 System Prompt 的来源。10 段来源是：Identity、Yuan、Ishiki、平台提示、工作区指令、工具列表、用户档案、记忆注入、团队花名册、Provider 补丁。

Q3. Vision Bridge 的作用是什么？

A. 加速图片加载 B. 将图片发送给视觉模型生成描述，让纯文本模型也能处理图片输入 C. 压缩图片以减少传输带宽 D. 将图片转为 base64 编码
看答案B。当用户使用的模型不支持视觉输入时，Vision Bridge 调用视觉模型生成图片描述，注入到消息中替代原始图片。这实现了纯文本模型对图片的「间接理解」。

Q4. promptSession() 在调用 Pi SDK 之前做的最后一项检查是什么？

A. 用户权限校验 B. 模态校验——检查视频/音频是否被当前模型支持 C. 网络连接测试 D. 数据库连接检查
看答案B。在构建 promptOpts 之前，assertVideoInputSupported 和 assertAudioInputSupported 检查当前模型是否支持视频/音频输入。不支持时直接抛错，避免请求到达 LLM Provider 后才失败。
