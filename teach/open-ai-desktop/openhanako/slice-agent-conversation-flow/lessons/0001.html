第 1 课 · 对话架构全景：8层穿透与WebSocket入口 全景地图

🎯 本节目标：学完你能画出一条消息从 React 输入框到 LLM 再回到屏幕的 8 层穿透路径，能解释 ChatArea 的 LRU 面板存活策略，能说清 WebSocket 入口的两道防线（权限校验 + 媒体限制）。

当用户在 HanaAgent 桌面应用中输入一条消息并按下回车，消息会经历 8 层穿透，跨越 5 个模块（desktop、server、hub、core、lib），涉及 React 组件、WebSocket 协议、Hub 路由、Pi SDK 会话管道、LLM Provider HTTP 流式调用等多项技术。这一课先建立全景认知，再深入到前端和 WS 入口层。

一、8层穿透速查表

数据流方向（完整一圈）
React 输入框 → WebSocket (type:"prompt") → chat.ts onMessage（wsParse + 权限校验 + 媒体限制）→ hub/index.ts send()（路由表匹配: 桌面 owner）→ desktop-session-submit.ts（防重入 + ensureSessionLoaded）→ engine.ts promptSession（Thin Facade）→ session-coordinator.ts（视觉预处理 + 模型解析 + Pi SDK）→ Pi SDK → LLM Provider HTTP（流式 SSE）→ EventBus（hub 事件回调）→ chat.ts hub.subscribe（ThinkTag/Mood/Card 解析链）→ WebSocket 推送 → React ChatMessageSurface 增量渲染
层模块关键文件核心职责
L1desktopChatArea.tsx消息列表渲染、LRU 面板存活策略
L2serverchat.tsWebSocket 连接管理、消息解析校验、事件广播
L3hubhub/index.ts消息路由中枢：按来源分发
L4coreengine.tsHanaEngine Thin Facade、防重入、Session 加载
L5coreagent.tsAgent 初始化、System Prompt 拼装
L6coresession-coordinator.tsSession 生命周期、promptSession、Pi SDK 调用
L7core/libllm-client.tsLLM HTTP 调用、多 Provider 协议适配
L8serverchat.ts流式事件 to 解析链 to WS 广播 to 前端
💡 关键洞察：这是典型的"请求-流式响应"全链路。前半段（L1-L6）是请求路径——消息逐层下钻到 LLM Provider；后半段（L7-L8）是响应路径——流式事件逐层上传到前端。两段在 Pi SDK 处交汇。

二、入口点：两个协议的握手

HanaAgent 有 两个入口点：主对话走 WebSocket 长连接，后台任务走 HTTP REST。

入口1 WebSocket /ws（主对话入口）

桌面应用启动后建立一条长连接 WebSocket 到 ws://localhost:XXXX/ws，整个会话生命周期内复用。核心消息类型：

// Client → Server
{ type: "prompt", text: "...", sessionPath: "...", images?: [...], uiContext?: {...} }
{ type: "interject", text: "..." }   // 流式进行中插入
{ type: "abort", sessionPath: "..." }
{ type: "resume_stream", sessionPath: "...", streamId: "..." }

// Server → Client
{ type: "text_delta", delta: "..." }          // 流式文本增量
{ type: "thinking_start" / "thinking_delta" }
{ type: "tool_start", id: "...", name: "..." }
{ type: "turn_end" }                          // 本轮结束

入口2 HTTP REST POST /task/:taskId/abort

后台任务（Cron 定时任务、Deferred Result）通过 REST 端点终止，不依赖 WS 连接。

连接生命周期保护：所有 WS 客户端断开后，server 启动可配置的倒计时（默认 5 min，HANA_WS_DISCONNECT_ABORT_GRACE_MS 环境变量）。到期后自动中断所有仍在 streaming 的 session，防止空转浪费 LLM token。

三、第 1 层——React 前端输入与 ChatArea LRU 策略

前端通过 ChatArea 组件渲染聊天面板。多个 session 面板通过 LRU 策略保持存活（最多 5 个），切换 session 时不销毁旧面板，而是通过 visibility: hidden 保持滚动位置。

// desktop/src/react/components/chat/ChatArea.tsx
const MAX_ALIVE = 5;

function PanelHost() {
  const currentPath = useStore(s => s.currentSessionPath);
  const [alive, setAlive] = useState<string[]>([]);

  useEffect(() => {
    if (!currentPath) return;
    setAlive(prev => {
      if (prev.includes(currentPath)) return prev;
      if (prev.length >= MAX_ALIVE) {
        // LRU 淘汰：移除最早的非当前项
        const evictIdx = prev.findIndex(p => p !== currentPath);
        const next = [...prev];
        next.splice(evictIdx, 1);
        next.push(currentPath);
        return next;
      }
      return [...prev, currentPath];
    });
  }, [currentPath]);

  return alive.map(path => (
    <ChatMessageSurface key={path} sessionPath={path}
      active={path === currentPath} />
  ));
}

前端消息类型：除文本外还需渲染 ToolGroupBlock（工具调用卡片）、ThinkingBlock（思考过程）、MoodBlock（情绪）、InteractiveCard（确认卡片）、SubagentCard（子 Agent 执行卡片）等。

要点：前端不直接发起 HTTP 请求，所有用户输入都通过 WebSocket 发送。前端通过 zustand store（chat-slice.ts）管理消息状态。

四、第 2 层——WebSocket 消息路由

server/routes/chat.ts 的 createChatRoute() 创建 Hono 子路由，管理 WS 连接生命周期。核心在 onMessage 回调中做三道处理：

1安全解析：wsParse(event.data) 解析 JSON，失败返回 null 静默丢弃。
server/routes/chat.ts
2权限校验：wsClientCanSendMessage(client, msg) 检查 client 是否有权发送此消息类型，无权限返回 insufficient_scope 错误。
server/ws-scope.ts
3媒体限制：图片 ≤ 10 张、单张 ≤ 20MB，视频/音频 ≤ 3 个。不合规直接拒绝。
server/routes/chat.ts onMessage
// server/routes/chat.ts -- prompt 消息处理（真实代码片段）
if (msg.type === "prompt" && (msg.text || msg.images?.length)) {
  // 媒体限制
  if (msg.images?.length > MAX_IMAGES) {
    wsSend(ws, { type: "error", message: t("error.maxImages") });
    return;
  }
  // Skill invocation tags 注入
  let promptText = msg.text || "";
  if (msg.skills?.length) {
    const skillNote = msg.skills.map(s => `[Use skill: ${s}]`).join('\n');
    promptText = `${skillNote}\n${promptText}`;
  }
  // 防重复发送
  if (engine.isSessionStreaming(promptSessionPath)) {
    wsSend(ws, { type: "error", message: t("error.stillStreaming") });
    return;
  }
  // ★ 进入 Hub 路由
  await hub.send(promptText, {
    sessionPath: promptSessionPath, images: msg.images,
    videos: msg.videos, audios: msg.audios, uiContext: msg.uiContext,
    clientMessageId: msg.clientMessageId,
  });
}
💡 设计要点：（1）媒体限制在 WS 层做第一道防线；（2）Skill 标签在 WS 层注入 [Use skill: xxx] 前缀，不污染 Engine 逻辑；（3）interject 类型允许在 streaming 进行中插入消息。

五、检验一下（提取练习）

Q1. 一条用户消息从输入到 AI 响应回到屏幕，共穿透几层？

A. 5 层 B. 6 层 C. 8 层 D. 10 层
看答案C。8 层：React → WebSocket → chat.ts → Hub → Engine → SessionCoordinator → LLM Client → 响应回流（chat.ts broadcast → WebSocket → React）。其中 L8 和 L2 同属 chat.ts，但方向相反。

Q2. ChatArea 的 MAX_ALIVE=5 策略是什么意思？

A. 最多同时打开 5 个聊天窗口 B. 最多保持 5 个 session 面板在 DOM 中存活，超出时 LRU 淘汰 C. 每个用户最多发 5 条消息 D. LLM 最多同时处理 5 个对话
看答案B。通过 LRU 策略保持最多 5 个 ChatMessageSurface 组件存活，切换 session 时用 visibility:hidden 而非销毁重建，保持滚动位置和消息状态。

Q3. WebSocket 入口在 prompt 消息到达时做了哪三道处理？

A. 反序列化 → 签名验证 → 频率限制 B. wsParse 安全解析 → wsClientCanSendMessage 权限校验 → 媒体数量/大小限制 C. JSON 解析 → Token 计数 → 上下文窗口检查 D. 文本清洗 → 敏感词过滤 → 长度截断
看答案B。三道防线：wsParse 安全解析 JSON（防注入）、wsClientCanSendMessage 权限校验（防越权）、媒体限制（图片 ≤10、视频/音频 ≤3，防滥用）。

Q4. 所有 WS 客户端断开后，server 会做什么？

A. 立即关闭所有 session B. 等待 5 分钟后自动中断所有仍在 streaming 的 session C. 重启 LLM 连接 D. 什么都不做
看答案B。启动可配置的倒计时（默认 5 min，通过 HANA_WS_DISCONNECT_ABORT_GRACE_MS 控制），到期后自动中断所有仍在 streaming 的 session，防止空转浪费 LLM token。
