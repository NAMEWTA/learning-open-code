🎯 本节目标：学完你能独立说出 6 条异常路径的触发条件和处理链，能从时序图中复述完整调用流程，能回答涵盖全 8 层的综合测验题。这是对话链路的最后一课——把前面 4 课串成完整的心理模型。

前面的课程按「请求→响应」主线走完了理想路径。但真实系统中，异常才是常态。本课聚焦 6 条异常路径的触发条件与处理链，然后用综合测验检验你对全链路的掌握。

一、6 条异常路径速查

路径 1：LLM API 超时 / 网络不可达

触发：LLM Provider 在 60 秒内未响应。
处理：Pi SDK 的 HTTP fetch 抛出 AbortError → 发射 error 事件 → chat.ts broadcast error → 前端红色错误提示。超时信号通过 AbortSignal.any([userSignal, timeoutSignal]) 合并用户 abort 和超时。

// core/llm-client.ts -- 超时信号构造
const timeoutSignal = AbortSignal.timeout(timeoutMs);  // 默认 60s
const combinedSignal = signal
  ? AbortSignal.any([signal, timeoutSignal])
  : timeoutSignal;

路径 2：Token 超限自动压缩（Session Compaction）

触发：对话历史 token 数接近模型上下文窗口上限。
处理：Pi SDK 在 prompt 前检查 → 超阈值时调用 LLM 对早期历史做摘要 → 摘要替换原始消息。如果摘要请求也超窗，降级为硬截断（computeHardTruncation），直接丢弃早期消息。

路径 3：工具执行失败

触发：Agent 调用工具时抛出异常（文件不存在、命令失败、网络错误）。
处理：工具执行器捕获异常，返回 { isError: true, error: "..." } → Pi SDK 发射 tool_execution_end → chat.ts 标记 success: false → 前端 ToolGroupBlock 显示红色错误标记。

路径 4：用户主动中断（Abort）

触发：用户点击停止按钮，前端发送 { type: "abort" }。
处理：chat.ts 标记 ss.isAborted = true → hub.abort() → SessionCoordinator → Pi SDK AbortController.abort()。如果 hub.abort 失败（Agent 已停止），降级为直接 finishStreamingState + broadcast。

路径 5：Turn 停滞看门狗（Turn Stall Watchdog）

触发：流式生成过程中超过 20 分钟没有任何事件（LLM Provider 卡死）。
处理：beginStreamingTurnState 启动定时器（默认 20 min）。每次收到流式事件时调用 markTurnStreamActivity 重置。超时后调用 hub.abort(sessionPath, { reason: "turn_stall_timeout" })。

// server/routes/chat.ts -- turn stall watchdog
function scheduleTurnStallWatchdog(sessionPath, ss) {
  const delay = Math.max(0, turnStallAbortMs - (Date.now() - ss.lastStreamActivityAt));
  ss.turnStallTimer = setTimeout(() => {
    if (!isSessionRuntimeStreaming(sessionPath)) return;
    ss.isAborted = true;
    hub.abort(sessionPath, { reason: "turn_stall_timeout" });
  }, delay);
}

路径 6：空回复检测

触发：模型没有输出任何文本、没有工具调用、没有 thinking 内容。
处理：turn_end 时检测 hasOutput、hasToolCall、hasThinking 均为 false 且未 aborted → broadcast error "modelNoResponse"。

// server/routes/chat.ts -- turn_end 空回复检测
if (!ss.hasOutput && !ss.hasToolCall && !ss.hasThinking
    && !ss.hasError && !ss.isAborted) {
  ss.hasError = true;
  broadcast({ type: "error",
    message: t("error.modelNoResponse"), sessionPath });
}

四、综合测验——全 8 层串讲检验

Q1. 以下哪条路径的描述是正确的？

A. 用户消息：React → HTTP POST → Engine → LLM B. 用户消息：WebSocket → chat.ts → Hub.send → Engine → SessionCoordinator → Pi SDK → LLM Provider C. 用户消息：gRPC → Hub → Engine → LLM D. 用户消息：WebSocket → 直接调用 LLM API
看答案B。正确路径经 8 层：React → WebSocket → chat.ts onMessage → Hub.send 路由 → desktop-session-submit 防重入 → Engine Thin Facade → SessionCoordinator 预处理 → Pi SDK 管道 → LLM Provider HTTP。不经过 gRPC，也不是直接 HTTP。

Q2. 当 LLM Provider 在 60 秒内未响应时，超时信号是如何传播的？

A. 60 秒后前端自动重试 B. AbortSignal.timeout 触发 → AbortError → Pi SDK 捕获发射 error → chat.ts broadcast error → 前端红色提示 C. 60 秒后 WebSocket 断开重连 D. 超时不处理，等待 LLM 自然恢复
看答案B。超时信号通过 AbortSignal.any 与用户 abort 信号合并，任一触发即中断 HTTP 请求。异常经 Pi SDK → EventBus → chat.ts → WebSocket 链路传播到前端。

Q3. 综合排序题：以下步骤的先后顺序是？
① System Prompt 拼装 ② 路由表匹配 ③ 防重入检查 ④ 视觉预处理 ⑤ LLM HTTP 调用

A. ②→③→④→①→⑤ B. ①→②→③→④→⑤（Agent 初始化在消息发送前已完成） C. ③→②→①→⑤→④ D. ⑤→④→③→②→①
看答案B。Agent 初始化（含 System Prompt 拼装）在 Engine 启动时已完成，发生在消息发送之前。消息到达后的顺序是：路由匹配(L3) → 防重入(L4) → 视觉预处理(L6) → LLM 调用(L7)。注意 System Prompt 是提前拼装好的，不在每条消息的处理路径中。

Q4. Turn Stall Watchdog 与用户 Abort 有什么区别？

A. 没有区别——都是停止 LLM 生成 B. Watchdog 是 20 分钟无活动自动触发（LLM 卡死）；Abort 是用户点击停止按钮主动触发 C. Watchdog 只在前端生效，Abort 只在后端生效 D. Watchdog 触发后重试，Abort 触发后不重试
看答案B。Turn Stall Watchdog 是后端防御机制——LLM Provider 可能返回了一半响应后卡死，没有任何事件。20 分钟无活动后自动 abort，reason 为 "turn_stall_timeout"。用户 Abort 是前端主动操作，reason 为 "user_abort"。
