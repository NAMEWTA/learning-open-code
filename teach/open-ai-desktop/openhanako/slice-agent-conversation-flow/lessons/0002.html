第 2 课 · Hub 路由与 Engine 调度 路由 & 编排

🎯 本节目标：学完你能背出 Hub.send() 的 4 条路由分支及其匹配条件，能解释防重入互斥锁的工作原理，能画出 HanaEngine Thin Facade 的依赖注入方向。

上一课我们看到消息通过 WebSocket 进入 hub.send()。这一课深入 Hub 路由表和 Engine 编排层——消息如何被分发到不同处理路径，以及系统如何防止同一 session 的并发提交。

一、Hub 构造与 Engine 注入

hub/index.ts 中的 Hub 类是消息调度中枢。它在构造器中组装了全部子系统并建立单向回调注入：

// hub/index.ts -- Hub 构造器（真实代码片段）
export class Hub {
  constructor({ engine }) {
    this._engine = engine;
    this._eventBus = new EventBus();
    this._channelRouter = new ChannelRouter({ hub: this });
    this._guestHandler = new GuestHandler({ hub: this });
    this._scheduler = new Scheduler({ hub: this });
    this._dmRouter = new DmRouter({ hub: this });

    // 注入 Hub 回调到 Engine（单向：Hub → Engine）
    engine.setHubCallbacks({
      hub: this, scheduler: this._scheduler,
      dmRouter: this._dmRouter, channelRouter: this._channelRouter,
      eventBus: this._eventBus,
    });

    // 注入 EventBus 到 Engine
    engine.setEventBus(this._eventBus);

    // 建立 Session 事件处理器
    this._setupSessionHandlers();
  }
}
💡 架构要点：Hub 持有 Engine 引用并注入回调，但 Engine 不持有 Hub 引用——这是单向依赖。EventBus 作为解耦中介，Engine 发射事件，Hub 的 subscribe 回调消费事件，两者通过 EventBus 通信而非直接引用。

二、Hub.send() 路由表——4 条分支

hub.send() 是消息分发入口，采用路由表模式——按顺序匹配，第一条命中即执行：

// hub/index.ts -- async send(text, opts) 路由表（真实代码片段）
async send(text, opts = {}) {
  const o = { sessionKey, role, ephemeral, meta, isGroup, cwd, model,
              persist, permissionMode, from, to, onDelta, images, videos,
              audios, clientMessageId, sessionPath, agentId, uiContext, ... };

  // 图片预处理：base64 解码 → 磁盘持久化 → [attached_image] 标记
  if (o.images?.length && !o.inboundFiles?.length) { /* ... */ }

  // ★ 路由表：按顺序匹配，第一条命中即执行
  const routes = [
    { // ① 桌面端 owner（你正在看的这条路径）
      match: o => !o.sessionKey && !o.ephemeral && o.role === "owner",
      handle: () => o.sessionPath
        ? submitDesktopSessionMessage(this._engine, { ... })
        : this._engine.prompt(text, { ... }),
    },
    { // ② Bridge guest（外部平台访客）
      match: o => o.sessionKey && o.role === "guest",
      handle: () => this._guestHandler.handle(text, o.sessionKey, ...),
    },
    { // ③ Bridge owner（外部平台主人）
      match: o => o.sessionKey && !o.ephemeral,
      handle: () => this._engine.executeExternalMessage(text, ...),
    },
    { // ④ 隔离执行（cron / heartbeat / channel）
      match: o => o.ephemeral,
      handle: () => this._engine.executeIsolated(text, { ... }),
    },
  ];

  for (const route of routes) {
    if (route.match(o)) return route.handle();
  }
  throw new Error("no matching route for message");
}
路由匹配条件目标处理函数典型场景
① 桌面 owner无 sessionKey、非 ephemeral、role=ownersubmitDesktopSessionMessage 或 engine.prompt用户在桌面应用中输入消息
② Bridge guest有 sessionKey、role=guestguestHandler.handleTelegram/飞书用户通过 Bridge 发消息
③ Bridge owner有 sessionKey、非 ephemeralengine.executeExternalMessageBridge 连接的主人发指令
④ 隔离执行ephemeral=trueengine.executeIsolatedCron 定时任务、Heartbeat、Channel 消息
路由优先级原则：桌面 owner 路径优先级最高，其次是 Bridge guest/owner，最后是后台隔离执行。这保证了用户直接输入的消息不会被 Bridge 或后台任务抢占。

三、防重入互斥锁——session_busy

路由匹配到桌面 owner 后，进入 desktop-session-submit.ts。它的核心职责是防重入：

// core/desktop-session-submit.ts（真实代码片段）
const pendingDesktopSessionSubmissions = new Set();

export async function submitDesktopSessionMessage(engine, opts) {
  const { sessionPath, text, images, videos, audios, uiContext,
          clientMessageId, displayMessage, sessionFileRefs } = opts;

  if (!sessionPath) throw new Error("sessionPath is required");

  // ★ 防重入：同一 session 不能并发提交
  const sessionId = engine.getSessionIdForPath?.(sessionPath);
  const submissionKey = sessionId || sessionPath;
  if (pendingDesktopSessionSubmissions.has(submissionKey)) {
    throw new Error("session_busy");  // ← 互斥锁命中
  }
  if (engine.isSessionStreaming(sessionPath)) {
    throw new Error("session_busy");  // ← 正在流式生成中
  }

  pendingDesktopSessionSubmissions.add(submissionKey);
  try {
    // 确保 session 已加载（首次对话需从磁盘恢复或创建新 session）
    await engine.ensureSessionLoaded(sessionPath);

    // 来源标注（Bridge/RC 等非桌面来源）
    const session = engine.getSessionByPath(sessionPath);
    recordMessageOriginEntry(session, sessionPath, displayMessage);

    // ★ 提交到引擎
    await engine.promptSession(sessionPath, text, {
      images, videos, audios, uiContext,
      clientMessageId, inboundFiles: opts.inboundFiles,
      onDelta: opts.onDelta, sessionFileRefs,
    });
  } finally {
    pendingDesktopSessionSubmissions.delete(submissionKey);  // 释放锁
  }
}
⚠️ 防重入的双重检查：
（1）pendingDesktopSessionSubmissions（Set）——同一 session 已有待处理提交时拒绝，防止用户快速连击。
（2）engine.isSessionStreaming(sessionPath)——session 正在流式生成中时拒绝，防止在 AI 回复过程中再发消息。
两重检查覆盖了「提交排队中」和「生成进行中」两种并发场景。

四、HanaEngine Thin Facade——单向依赖注入

HanaEngine 是整个系统的 Thin Facade，持有所有 Manager 引用，对外暴露统一 API：

// core/engine.ts -- Manager 组装（真实代码片段）
export class HanaEngine {
  constructor({ hanakoHome, productDir, agentId, appVersion }) {
    this._prefs  = new PreferencesManager({ userDir, agentsDir, ... });
    this._models = new ModelManager({ hanakoHome });
    this._channels = new ChannelManager({ channelsDir, ... });

    // Agent Manager：持有 Engine 回调以间接访问其他 Manager
    this._agentMgr = new AgentManager({
      getPrefs: () => this._prefs,
      getModels: () => this._models,
      getEngine: () => this,   // 单向引用
    });

    // Session Coordinator：不持有 Engine 引用，所有需求通过 deps 注入
    this._sessionCoord = new SessionCoordinator({
      getAgent: () => this.agent,
      getModels: () => this._models,
      buildTools: (cwd, ct, opts) => this.buildTools(cwd, ct, opts),
      emitEvent: (e, sp) => this._emitEvent(e, sp),
    });
  }

  // 对外委托方法（Thin Facade）
  async promptSession(sessionPath, text, opts) {
    return this._sessionCoord.promptSession(sessionPath, text, opts);
  }
}
💡 依赖注入方向：Engine → Manager 是单向的。SessionCoordinator 不持有 Engine 引用，所有对外需求通过构造器注入的 this._d（依赖对象 deps）间接访问。这避免了循环引用，也简化了单元测试。这是经典的「好莱坞原则」——Don't call us, we'll call you。

五、检验一下（提取练习）

Q1. Hub.send() 路由表中，桌面 owner 路径的匹配条件是什么？

A. o.sessionKey && o.role === "admin" B. !o.sessionKey && !o.ephemeral && o.role === "owner" C. o.ephemeral && o.role === "owner" D. o.sessionKey && !o.ephemeral && o.role === "guest"
看答案B。桌面 owner 路由的条件是：没有 Bridge 的 sessionKey、不是隔离执行（非 ephemeral）、角色为 owner。这是最优先匹配的路由。

Q2. 以下哪个场景会触发 "session_busy" 错误？

A. 用户在一个 session 中快速连续发送两条消息 B. 用户的网络断开 C. LLM Provider 返回错误 D. session 文件被删除
看答案A。快速连击时，第一条消息正在 pendingDesktopSessionSubmissions 中排队，第二条消息的 has(submissionKey) 检查会命中，抛出 session_busy。这是防止并发提交的第一道防线。

Q3. SessionCoordinator 为什么不持有 Engine 引用？

A. 因为 TypeScript 不支持循环引用 B. 为了避免循环引用——所有对外需求通过构造器注入的 deps 间接访问 C. 因为 Engine 还未实现 D. SessionCoordinator 不需要任何外部依赖
看答案B。SessionCoordinator 通过构造器的 deps 参数接收所有外部依赖（getAgent、getModels、buildTools、emitEvent），不直接引用 Engine。这是单向依赖注入模式，避免循环引用并简化单元测试。

Q4. 路由表第④条（隔离执行）匹配 ephermal=true 的消息，典型场景是什么？

A. 用户在桌面应用中发消息 B. Telegram 用户通过 Bot 发消息 C. Cron 定时任务触发、Heartbeat 健康检查、Channel 消息处理 D. WebSocket 心跳 ping/pong
看答案C。ephemeral 消息是后台自动触发的，不需要持久化 session、不产生 UI 交互。典型场景包括 Cron 定时任务、Agent Heartbeat、多 Agent Channel 消息等。
