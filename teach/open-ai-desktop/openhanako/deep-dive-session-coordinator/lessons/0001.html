第 1 课 · SessionCoordinator 架构全景：5220 行功能域分解与双模型 全景地图

🎯 本节目标：学完你能讲清 SessionCoordinator 的 8 大职责域、18 个私有字段的分组逻辑、16 个注入依赖的名称与作用，以及 promptSession 与 executeIsolated 两种模型的核心区别。

core/session-coordinator.ts 是整个 HanaAgent 中最大的单文件——217KB / 5220 行。但它不是上帝对象反模式，而是聚合根 (Aggregate Root)：通过构造器注入 16 个外部能力切片，内部管理 8 个紧密耦合的职责域，所有域共享同一份 session 运行时缓存 _sessions Map。

💡 为什么不拆？拆分会引入 5 个现实问题：(1) 16 个外部依赖需要在子 Manager 间传递；(2) createSession 的 ~700 行中子步骤有严格数据依赖顺序；(3) _sessions Map 是唯一数据源，拆分需要跨 Manager 缓存共享；(4) Path 感知 API (Phase 2) 重构尚未完成；(5) 当前测试覆盖依赖整体行为。

二、16 个构造器注入依赖

构造器（L678-698）通过 _d 对象接收 16 个外部能力：

依赖来源作用
getAgent / getAgentById / getAgentsEngine / AgentManager获取 Agent 实例，调用 buildSystemPrompt()
getModelsProviderRegistry获取模型列表、当前模型、Thinking Level 默认值
getResourceLoaderEngine获取基础 resourceLoader，构建代理层
getSkills / buildToolsPluginManager获取 Skills 列表、构建 session 工具快照
getConfig / getPrefsConfig / Prefs读取配置和用户偏好
emitEventEngine向 IPC 总线发射事件（UI 更新）
agentsDir / getHomeCwdPathsAgent 目录和用户 home 路径
getUsageLedgerUsageLedgerToken 用量账本
onBeforeSessionCreate / onSessionRuntimeDiscardedEngine hooks生命周期钩子

设计洞察：这不是"依赖注入容器"模式，而是依赖倒置。Coordinator 不持有 Engine 实例，只持有 getAgent() 等 injectable 函数——它是外部模块的"切片视图"。这让 Coordinator 可以独立于 Engine 进行测试。

三、两种执行模型的对比

维度promptSessionexecuteIsolated
用途用户主动对话——"发送消息"后台子任务——子 Agent / Workflow / Cron
session复用主 session（agents/{id}/sessions/*.jsonl）创建独立 session 目录
prompt 快照使用 createSession 时冻结的快照每次独立构建 prompt
工具集主 session 的工具快照可指定子集或独立工具
并发同一 session 串行（一次只能一个 turn）多个 isolate 可并行（不同 session 目录）
结果回传直接流式返回给 UI通过 turnContext 回传到主 session
resume不支持（session 是连续的）支持——通过独立 session 目录持久化，中断后可恢复
核心判断：promptSession 是"在线"模式——用户在等回复；executeIsolated 是"离线"模式——后台执行，结果通过 turnContext 注入主对话。两者共享同一套 Pi SDK 管道但 session 隔离策略完全不同。

四、依赖方向：Coordinator 是下游消费者

Engine（顶层编排）
↓ 构造器注入 16 个依赖
SessionCoordinator
↓ 调用 agent.buildSystemPrompt() / buildTools() / getSkills()
Agent PluginManager ProviderRegistry
↓ 创建 Pi SDK AgentSession
Pi SDK → LLMClient → LLM API

Engine 是顶层所有者，持有 Coordinator 实例。Agent 通过 _d.getAgent() 动态获取（不持有实例）。Pi SDK 的 createAgentSession() 接收 Coordinator 构建的 resourceLoader 代理和 session 配置。Coordinator 不直接调用 LLM API——所有 LLM 调用通过 Pi SDK 的 agent.streamFn() 完成。

关键观察：18 个字段中 Map 占了 9 个。Coordinator 本质上是多个 per-session 键值存储的组合——它把本该是独立数据结构的职责揉进了同一个类。这也是其庞大的另一个结构性原因。

六、检验一下

Q1. SessionCoordinator 是上帝对象反模式吗？

A. 是——5220 行单文件违反了单一职责原则 B. 不是——它是聚合根，把 8 个共享同一份 session 缓存的紧密耦合域聚合在一起，拆分会导致跨 Manager 协调开销和缓存一致性问题 C. 是——但因为时间不够才没拆分 D. 不是——它实际上只有 200 行业务逻辑，其他都是注释
看答案B。它是聚合根，不是上帝对象。8 个职责域共享 _sessions Map 和 16 个注入依赖，createSession 的子步骤有严格的数据依赖顺序，拆分会增加跨类传参复杂度。这是在"简单"和"解耦"之间的务实折中。

Q2. 构造器注入的 16 个依赖中，哪个用于获取模型列表？

A. getConfig B. getSkills C. getModels —— 通过 ProviderRegistry 获取 D. getResourceLoader
看答案C。getModels 来自 ProviderRegistry，返回模型列表、当前模型、Thinking Level 默认值等。Coordinator 不直接持有 ProviderRegistry 实例，通过 injectable 函数解耦。

Q3. promptSession 和 executeIsolated 的核心区别是什么？

A. 没有区别，executeIsolated 是 promptSession 的别名 B. promptSession 是 HTTP API，executeIsolated 是 WebSocket 连接 C. promptSession 复用主 session 的冻结快照，串行执行；executeIsolated 创建独立 session 目录，支持并行和 resume D. promptSession 用于测试，executeIsolated 用于生产
看答案C。promptSession 在用户主动对话时使用，复用 createSession 时冻结的 prompt 快照（保护 prefix cache）；executeIsolated 用于后台子任务，每次独立构建 prompt，使用独立 session 目录支持并行执行和中断恢复。

Q4. 18 个私有字段中，Map 类型占了几个？这说明了什么？

A. 2 个——只用于 session 和 meta 的简单缓存 B. 5 个——均匀分布在各个职责域 C. 9 个——Coordinator 本质上是多个 per-session 键值存储的组合，暴露出数据结构未充分封装 D. 18 个——所有字段都是 Map 类型
看答案C。18 个字段中 Map 占 9 个：_sessions、_hibernatedSessionMeta、_titlesCache、_metaCache、_runtimePressureTimers、_prePromptAbortControllers、_turnContextBySession、_headlessOps（Set）、_sessionManifestStore。这些本应是独立数据结构的封装，目前直接以原生 Map 形式存在于 Coordinator 中。
