三、ResourceLoader 代理 —— Prompt 注入的机制

Coordinator 不直接写入 prompt 字符串。它使用 Object.create(baseResourceLoader, props) 创建原型链代理（L1491）：

const resourceLoader = Object.create(baseResourceLoader, {
  getSystemPrompt:      { value: () => systemPromptSnapshot },
  getExtensions:        { value: () => [turnCtx, visionAux, ...base] },
  getAppendSystemPrompt:{ value: () => [...appendSystemPromptSnapshot] },
  getSkills:            { value: () => resolveSessionSkillsForRuntime(...) },
  getAgentsFiles:       { value: () => freezeAgentsFilesResult(...) },
});

这个代理对象被传入 createAgentSession({ resourceLoader })。Pi SDK 在每次 LLM 请求中通过 resourceLoader.getSystemPrompt() 获取 prompt，实现了 per-session prompt 隔离——每个 session 读到的 prompt 是创建时冻结的版本。

四、Prompt 冻结策略 —— 保护 Prefix Cache

设计意图：LLM API 的 prompt caching 是成本优化的关键。如果中间某次请求的 system prompt 发生任何变化，Anthropic/OpenAI 等提供商判定 cache miss，导致全量 token 计费。冻结 prompt snapshot 的目的就是让同一 session 的每次请求都能命中 cache prefix，每次节省数万 token 的处理费用。

冻结覆盖以下 4 项（统一存入 promptSnapshot）：

  • systemPrompt —— 主体 system prompt
  • appendSystemPrompt —— 追加 prompt 片段
  • skillsResult —— 技能列表
  • agentsFilesResult —— Agent 文件快照

以上内容在 restoredPromptSnapshot 存在时直接复用，不重新计算（L1255-1260），确保同一 session 的 prefix cache 永远匹配。

五、扩展管线：视觉预处理与 AbortController

第 5 步 prepareVisionInputForTextOnlyModel 是 promptSession 中最容易被忽略但最脆弱的一环。当目标模型不支持图片输入时，Coordinator 需要将图片"翻译"为文本描述：

// 伪代码：视觉预处理流程
async function prepareVisionInputForTextOnlyModel(opts) {
  if (modelSupportsVision(targetModel)) return; // 直接支持 → 跳过

  // 1. 调用 Vision Bridge（独立的视觉模型 API）
  const descriptions = await visionBridge.describeImages({
    images: opts.images,
    signal: opts.abortSignal,  // ← 可被取消
  });

  // 2. 将文本描述注入到用户消息中
  opts.text = `[用户发送了 ${images.length} 张图片]\n`
    + descriptions.join('\n') + `\n\n用户消息：${opts.text}`;
}
⚠️ 为什么这一步需要 AbortController？视觉描述 API 可能因为图片过大、网络超时、模型限流等原因长时间卡住。没有取消机制，用户只能强制关闭应用。AbortController 允许用户在 promptSession 的第 5 步被阻塞时取消预处理，保留 session 状态不被破坏。

第 11 步的 finally 清理块同样值得关注——它不仅是"打扫战场"，还承担了关键的数据完整性职责：

  • 投射 oversized JSONL 条目：异常大的消息行会导致 JSONL 解析性能下降，需拆分为多行
  • 修剪内联媒体历史：避免图片/视频的 base64 数据在 session 文件中无限累积
  • 调度内存压力检查：通知 Coordinator 评估是否需要 hibernate 当前 session

六、检验一下

Q1. promptSession 的 12 步管线中，哪一步是委托给 Pi SDK 的核心调用？

A. 第 5 步 —— prepareVisionInputForTextOnlyModel B. 第 10 步 —— entry.session.prompt(text, promptOpts)，触发 agent.streamFn() → LLM API C. 第 11 步 —— finally 清理逻辑 D. 第 12 步 —— agent._memoryTicker.notifyTurn
看答案B。第 10 步是核心——将用户文本和组装好的选项传给 Pi SDK 的 AgentSession.prompt()，其内部触发 agent.streamFn() 调用 LLM API，并处理流式 token 回调和 tool_use → tool_result 循环。前 9 步都是预处理，后 2 步是清理。

Q2. 为什么 createSession 要冻结 prompt snapshot？

A. 为了减少内存占用——冻结的对象可以被 GC 更高效回收 B. 为了保护 LLM API 的 prompt prefix cache——同一 session 的 system prompt 不变，确保每次请求命中 cache，节省 token 费用 C. 为了防止插件代码篡改 system prompt D. 为了支持 session 的序列化和反序列化
看答案B。这是成本优化设计。LLM API 的 prompt caching 按 prefix 匹配——system prompt 是对话中最固定的部分。一旦冻结，同一 session 的所有后续请求共享相同的 system prompt prefix，命中 cache 时不计费或大幅折扣。

Q3. ResourceLoader 代理使用什么机制实现 per-session prompt 隔离？

A. 使用 ES6 Proxy 对象拦截所有属性访问 B. 使用 Object.create() 原型链继承 + 属性描述符覆盖 getter C. 使用闭包捕获 sessionId 变量 D. 使用 Symbol 作为私有属性键
看答案B。Coordinator 使用 Object.create(baseResourceLoader, props) 创建原型链代理，通过属性描述符覆盖 getSystemPrompt、getExtensions 等 getter。Pi SDK 在每次 LLM 请求中调用这些 getter，读取到 Coordinator 注入的冻结 prompt 数据。

Q4. finally 清理块中"投射 oversized JSONL 条目"的目的是什么？

A. 压缩 JSONL 文件以节省磁盘空间 B. 将异常大的消息行拆分为多行，避免 JSONL 单行解析性能下降和内存尖峰 C. 删除所有 base64 编码的图片数据 D. 将 JSONL 格式转换为二进制格式以提高读取速度
看答案B。JSONL 是逐行解析的格式——如果某一行过大（例如包含大量 tool_result 数据），解析该行时会消耗大量内存和时间。投射（project）机制将超大行拆分为多行引用，主行只保留元数据指针。
