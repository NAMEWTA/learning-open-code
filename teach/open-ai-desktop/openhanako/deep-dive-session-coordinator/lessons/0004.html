二、Turn 停滞看门狗 —— _prePromptAbortControllers

_prePromptAbortControllers 是一个 Map<string, AbortController>，每个 session 一个。它的核心职责是取消视觉预处理——当图片描述生成卡住时（模型 API 超时或网络故障），用户可以取消当前 turn 而不会丢失整个 session。

// 简化逻辑
class SessionCoordinator {
  _prePromptAbortControllers = new Map();

  async promptSession(path, text, opts) {
    // 1. 为本次 prompt 创建 AbortController
    const ac = new AbortController();
    this._prePromptAbortControllers.set(path, ac);

    try {
      // 2. 视觉预处理可被取消
      await prepareVisionInputForTextOnlyModel({
        signal: ac.signal,  // ← 传入 signal
      });
      // 3. 核心 LLM 调用
      await entry.session.prompt(text, promptOpts);
    } finally {
      // 4. 清理
      this._prePromptAbortControllers.delete(path);
    }
  }

  // 外部可调用此方法取消
  cancelPrePrompt(sessionPath) {
    this._prePromptAbortControllers.get(sessionPath)?.abort();
  }
}
💡 为什么只取消预处理而非整个 LLM 调用？视觉预处理（图片描述生成）是独立的 API 调用，发生在主 LLM 调用之前。如果图片描述 API 超时（例如图片过大），用户不应被迫等待或丢失 session。而主 LLM 调用通过 Pi SDK 的流式管道处理中断。

三、Token 超限压缩 —— 两级降级策略

当 session 的上下文窗口超过模型限制时，Coordinator 采用两级降级：

级别方法策略代价
L1: Compact_compactWithModel用旧模型对历史对话生成摘要，保留最近 effectiveWindow tokens需要一次额外的 LLM 调用（摘要生成）
L2: Hard Truncate_hardTruncate不调用 API，用固定文本"[由于模型切换，早期对话历史已被截断]"作为摘要，直接丢弃最旧的消息丢失对话历史细节，但不消耗 token

四、内存压力保护 —— _runtimePressureTimers

Coordinator 维护 3 个内存阈值和 per-session 检查定时器：

// 内存压力阈值
_memoryPressure = {
  rss: 1.5 * 1024 * 1024 * 1024,      // 1.5 GB RSS
  external: 512 * 1024 * 1024,          // 512 MB external
  payload: 64 * 1024 * 1024,            // 64 MB payload
};

// per-session 定时器 Map
_runtimePressureTimers = new Map();

工作流程：每次 promptSession 的 finally 块中调用 scheduleRuntimePressureCheck() → 定时检查内存使用 → 超过阈值时调用 hibernateSessionRuntime() 将 session 从内存卸载到磁盘 → 需要时再 ensureSessionLoaded() 从磁盘恢复。

⚠️ Hibernate 的影响：休眠的 session 不在 _sessions Map 中，下次访问需要从磁盘 JSONL 恢复。恢复过程会重放 JSONL 中的消息构建 LLM 上下文，对于超大 session 可能需要数秒。这就是为什么 _sessions Map 有 LRU 上限 20——超过的自动休眠。

六、检验一下

Q1. _prePromptAbortControllers 主要取消的是什么操作？

A. 整个 LLM API 调用——包括流式 token 生成 B. 视觉预处理阶段的图片描述生成——这是独立的 API 调用，发生在主 LLM 调用之前 C. session 的 JSONL 文件写入 D. tool_use 循环中的工具执行
看答案B。AbortController 传入 prepareVisionInputForTextOnlyModel 的 signal 参数，用于取消视觉预处理（图片描述生成）。主 LLM 调用的中断由 Pi SDK 的流式管道处理。解析取舍：只在可能长时间卡住的预处理阶段提供取消能力。

Q2. Compact 和 Hard Truncate 的区别是什么？

A. Compact 用 AI 生成摘要，保留最近 tokens；Hard Truncate 用固定文本直接丢弃最旧消息，不消耗 API token B. Compact 是自动的，Hard Truncate 需要用户手动触发 C. 两者效果完全相同，只是实现方式不同 D. Compact 只删除消息，Hard Truncate 只压缩消息
看答案A。Compact 调用 LLM API 生成有意义的摘要，代价是一次额外 API 调用但保留更多上下文信息。Hard Truncate 不调用 API，用固定文本替代被截断的历史，零 token 成本但丢失细节。Compact 失败时自动降级到 Hard Truncate。

Q3. session 被 hibernate 后，下次访问会发生什么？

A. 直接报错——休眠的 session 不可恢复 B. 从磁盘 JSONL 恢复——重放消息构建 LLM 上下文，大 session 可能需要数秒 C. session 数据被永久删除 D. 自动创建一个新的空 session 替代
看答案B。休眠是将 session 从 _sessions Map 中移除以释放内存，但 JSONL 文件和 meta 文件保留在磁盘上。下次访问时 ensureSessionLoaded() 从 JSONL 恢复，包括重放消息和重建 prompt snapshot。LRU 上限 20 意味着超过此数的最旧 session 自动休眠。

Q4. capabilityDrift snapshot 检测到什么时会提示用户？

A. session 的 JSONL 文件损坏 B. 当前可用工具与 session 创建时快照的工具不一致——有新工具可用或旧工具已移除 C. LLM API 返回了错误 D. session 的 token 用量超过预算
看答案B。capabilityDrift 对比 toolSnapshot（创建时冻结的工具列表）与当前 _computeLiveToolSnapshotForEntry（实时的可用工具列表）。如果有差异——例如插件更新了工具定义或用户安装了新插件——会提示用户刷新 session 以使用最新工具集。
