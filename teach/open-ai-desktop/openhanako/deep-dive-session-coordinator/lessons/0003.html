第 3 课 · executeIsolated 隔离执行：子Agent/Workflow/Cron 共用模型 隔离模型

🎯 本节目标：学完你能画出 executeIsolated 的完整执行流程图，讲清独立 session 目录的创建策略，理解 resume 机制如何在中断后恢复运行，以及子 Agent / Workflow / Cron 三者如何共用同一执行模型。

executeIsolated 位于 L4799，是 SessionCoordinator 的"后台引擎"。它与 promptSession 的区别不是技术实现的不同，而是场景语义的不同——用户在等回复 vs 后台静默执行。

一、三种使用场景，一种执行模型

场景触发方式session 行为结果处理
子 Agent (Subagent)主 Agent 的 tool_use 触发创建独立 session，注入子 Agent 的 system prompt结果通过 turnContext 回传主 session
Workflow插件 defineTask() 或手动触发创建独立 session，可能使用不同的 Agent结果写入 task 输出，通过事件通知
Cron / 定时任务scheduleTask() 调度触发每次执行创建新 session（或复用上次的 resume）结果更新 task 状态，支持下次 resume
💡 设计洞察：三种场景共享同一个 executeIsolated 方法。这得益于"隔离执行"的抽象——不管是子 Agent 调用、Workflow 步骤还是 Cron 触发，都需要：(1) 独立 session 目录避免污染主对话；(2) 独立的 prompt 构建不受主 session 快照限制；(3) resume 能力支持长任务的中断恢复。

二、执行流程：7 步从接收到完成

1参数校验与 session 目录创建：为 isolate 创建独立的 session 目录（区别于主 session 的 agents/{id}/sessions/*.jsonl）。如果 resume，读取已有 session 的状态。
independent session directory
2Prompt 构建：独立构建 system prompt——不复用主 session 的冻结快照。根据场景类型注入不同的上下文（子 Agent 的 task description、Workflow 的步骤定义、Cron 的任务描述）。
fresh prompt build
3工具集选择：可指定工具子集或独立工具列表。子 Agent 可能只拥有主 Agent 的部分工具；Workflow 可能使用专属工具。
tool subset selection
4Pi SDK 执行：调用 AgentSession.prompt()，与主 session 相同的 LLM 调用管道。但 session 对象是独立的，不影响主对话的状态。
isolated AgentSession
5中断处理：如果执行被中断（进程退出、超时、用户取消），session 目录中的 JSONL 文件保留已完成的 turns。下次 resume 时从断点继续。
resume from JSONL
6结果序列化：执行完成后，将结果序列化为标准格式（文本 + tool_use 记录 + token 用量）。
result serialization
7结果回传：子 Agent → 通过 turnContext 注入主 session 的下一个 turn。Workflow/Cron → 更新 task 状态，触发事件通知。
result delivery

三、Resume 机制详解

Resume 是 executeIsolated 区别于 promptSession 的核心能力。实现原理：

// 伪代码：resume 逻辑
async function executeIsolated(opts) {
  let sessionEntry;
  if (opts.resumeSessionPath) {
    // 1. 从磁盘恢复已有 session
    sessionEntry = await restoreSessionFromJsonl(
      opts.resumeSessionPath
    );
    // 2. JSONL 中已包含之前所有的 user/assistant/tool_result 消息
    // 3. Pi SDK 会自动跳过已完成的消息，从断点继续
  } else {
    // 创建全新 session
    sessionEntry = await createIsolatedSession(opts);
  }
  // 继续执行...
  const result = await sessionEntry.session.prompt(opts.prompt);
  // 结果 + 中间状态全部在独立 JSONL 中
  return result;
}
⚠️ Resume 的限制：Resume 只能从中断的 turn 边界恢复（JSONL 以 turn 为单位）。如果一个 turn 内部正执行 tool_use 时中断，恢复后该 tool_use 会重新执行（不是继续执行）。这是因为 LLM API 不支持"恢复半个 turn"。工具设计时应考虑幂等性。

四、三种场景的 session 目录策略

场景session 目录位置清理策略
子 Agent主 session 目录下的子目录（如 sessions/{main}/subagents/{sub-id}/）随主 session 关闭时清理
Workflow插件 dataDir 下的独立目录workflow 完成后清理（或保留用于审计）
CronAgent 的任务目录（如 agents/{id}/tasks/{task-id}/）保留最后一次执行的 session 用于下次 resume
💡 为什么不用内存缓存？子 Agent / Workflow / Cron 的执行可能跨进程生命周期（进程重启、崩溃恢复），必须持久化到磁盘。JSONL 格式天然支持追加写——每完成一个 turn 就追加一行，即使进程崩溃也只丢失最后一个 turn。

Resume 的边界条件处理

Resume 不仅是"从文件恢复"，还涉及多个边界条件：

  • Agent 变更：如果子 Agent 的定义在两次执行之间被修改（如更新了 system prompt），resume 后的行为取决于 capabilityDrift 检测——如果 prompt 快照与当前 Agent 不一致，Coordinator 可能拒绝 resume 或提示用户。
  • 工具集变更：如果插件的工具定义在 resume 前后不同，markCapabilitySnapshotsStale() 会标记漂移，提示用户刷新 session。此时 resume 仍可继续，但使用旧工具快照。
  • 模型切换：如果 resume 时使用了与创建时不同的模型，可能触发 compact 流程——因为不同模型的上下文窗口大小不同。
  • 超时处理：长时间未完成的 isolate（如 Cron 任务执行超过 24 小时）会被标记为 abandoned，需要显式重新触发而非自动 resume。

五、检验一下

Q1. executeIsolated 支持哪些使用场景？

A. 仅子 Agent 调用 B. 子 Agent、Workflow、Cron 定时任务——三种场景共用同一执行模型 C. 子 Agent 和 Workflow，但不包括 Cron D. 仅限插件发起的后台任务
看答案B。三种场景共享 executeIsolated：子 Agent 由 tool_use 触发，Workflow 由 defineTask 触发，Cron 由 scheduleTask 触发。它们都需要独立 session 目录、独立 prompt 构建和 resume 能力。

Q2. Resume 机制的限制是什么？

A. 只能 resume 一次，不能多次 resume B. 只能从 turn 边界恢复——如果一个 turn 内部的 tool_use 执行中断，恢复后该 tool_use 会重新执行而非继续 C. 只能 resume 最近 24 小时内的 session D. Resume 后无法使用任何工具
看答案B。JSONL 以 turn 为单位记录，LLM API 不支持"恢复半个 turn"。如果 tool_use 正在执行时进程崩溃，恢复后该 tool_use 会重新执行。工具应设计为幂等操作以应对这种场景。

Q3. 子 Agent 的执行结果如何回传到主 session？

A. 通过 WebSocket 推送到 UI 前端 B. 通过 turnContext 机制——在下一个主 session prompt 时，turnContext 被注入到扩展中 C. 写入共享数据库，主 session 轮询读取 D. 通过 process.send() IPC 消息
看答案B。子 Agent 完成后，结果存入 turnContext。在下一个 promptSession 调用的第 1 步 normalizeSessionTurnContext 和第 7 层的 turnContextExtension 中被注入到 system prompt，作为上下文提供给主 Agent。

Q4. 子 Agent 的 session 目录为什么放在主 session 的子目录下而非独立位置？

A. 为了减少文件系统的 inode 消耗 B. 确保子 Agent session 随主 session 关闭时一起清理，避免孤立的 session 文件累积；同时便于用户理解 session 的层级关系 C. 因为文件系统不支持跨目录的原子操作 D. 这是 Pi SDK 的限制，Coordinator 无法选择其他位置
看答案B。子 Agent 的 session 与主 session 有生命周期绑定——主 session 关闭时子 session 的对话历史通常不再有意义。放在子目录下确保批量清理的原子性（删除主目录即可），且为用户提供了清晰的 session 层级视图。
