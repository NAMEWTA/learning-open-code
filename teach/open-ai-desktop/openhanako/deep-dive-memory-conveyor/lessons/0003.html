第 3 课 · 折叠与装配机制：零LLM拼接与滚动窗口 装配机制

🎯 本节目标：学完你能解释 assembleWeekFromDaily 为什么不再用 LLM（以及旧 v3 方案的三大问题），理解 rollDailyWindow 的"失败保留源文件"保护机制，掌握 compileLongterm 作为通用 fold 入口的复用设计，以及 compileEditableFacts 的增量编译与独立水位线。

在 compileToday 和 compileDaily 完成今天的草稿和昨天的日记后，传送带还需要三个关键步骤来维护 week、longterm 和 facts 这三个长期产物。这一课我们逐一拆解。

一、assembleWeekFromDaily：零 LLM 周段装配

1.1 为什么不再用 LLM 编译 week？

这是 v4 传送带的关键创新点之一：week.md 不再是独立的 LLM 编译产物，而是从 memory/daily/ 目录纯文件装配——取最近 6 个已结束逻辑日的日记条目，按日期正序拼接。零 LLM 调用。

旧版 v3 设计是每天对过去 7 天的摘要做整周编译（compileWeek），有三大问题：

1重复计算：每天重编过去 7 天，产生 7 倍的 LLM 调用。每天的 daily 条目已经蒸馏过一次，week 编译是对"摘要的摘要"再做一次蒸馏——信息在每层都有损失。
v3 方案：compileWeek 每天全量重编
2输出不稳定：每天给出的 week 摘要可能因模型随机性而不同（temperature 0.3 仍有波动），用户会看到"历史在变化"——同一周的内容在不同日子里看起来不一样。
v3 方案：LLM 非确定性输出
3摘要精度递减：对摘要再做摘要（摘要的摘要），信息在每层蒸馏中都有损失。compileDaily 已经产出 60 字的精炼日记，再让 LLM 从中提炼 week 摘要反而可能丢掉关键信息。
v3 方案：信息二次蒸馏损失
💡 v4 方案核心洞察：用 compileDaily 一次性写好每天 60 字的日记，week 只是把它们拼起来。每天的内容是独立产出的，不随天数增加而变化。这不只是降本，更是保证信息稳定性——每天的 daily 条目一旦产生就不再变化。

1.2 算法实现

function assembleWeekFromDaily(dailyDir, weekPath,
                                maxDays=6, maxChars=1200):
    entries = listDailyFiles(dailyDir)
               .slice(-6)                     // 取最近 6 天
    blocks = entries.map(readFile).filter(nonEmpty)

    content = blocks.join("\n\n")

    // 超长截断：从最老的条目开始丢弃
    if content.length > maxChars:
        while blocks.length > 1 and blocks.join("\n\n").length > maxChars:
            blocks.shift()                    // 丢弃最老的
        content = blocks.join("\n\n")
        if content.length > maxChars:
            content = content.slice(0, maxChars)
        log.warn("week 段总长超过上限，已从最老条目开始截断")

    writeFile(weekPath, content)
设计点值理由
窗口大小DAILY_WINDOW_RETENTION_DAYS = 6一周减掉今天 = 6 个已结束日
总长上限WEEK_ASSEMBLY_MAX_CHARS = 12006 条 x ~200 字/条 = 1200 字上限
截断策略从最老条目开始丢弃最近的信息更重要，符合时间衰减原则
LLM 调用0纯文件 I/O 操作，这是 week 段的核心降本设计

二、rollDailyWindow：滚动窗口折叠到长期

2.1 功能定位

这是传送带的"出口"：把滚出 6 日窗口的 daily 条目 fold 进 longterm.md，成功后删除源文件。失败的条目保留在 daily/ 目录，交给下一轮重试，不静默丢弃。

2.2 五步算法流程

1计算窗口边界：cutoffDate = shiftLogicalDate(referenceDate, -retentionDays)。referenceDate 默认为今天，retentionDays 默认为 6。
compile.ts → rollDailyWindow()
2筛选溢出条目：列出 daily/ 目录下所有 YYYY-MM-DD.md 文件，筛选 date < cutoffDate 的条目。
compile.ts → 窗口边界过滤
3拼接待折叠内容：将所有溢出条目的内容拼接为一个文本块（保留 ## {date} 抬头），作为 compileLongterm 的输入。
compile.ts → 内容拼接
4fold 进 longterm：调用 compileLongterm(combined, longtermPath, resolvedModel)。成功则删除源 daily 文件。
compile.ts → compileLongterm()
5失败重试保护：如果 compileLongterm 抛出异常，保留所有 daily 文件，下一轮 _doDaily() 重试。不静默丢弃任何内容。
compile.ts → try/catch 保护

2.3 核心伪代码

function rollDailyWindow(dailyDir, longtermPath, resolvedModel,
                          referenceDate = today, retentionDays = 6):
    cutoffDate = shiftLogicalDate(referenceDate, -retentionDays)
    entries = listDailyFiles(dailyDir)
               .filter(date => date < cutoffDate)

    if entries is empty:
        return { folded: [], failed: [] }

    combined = entries.map((date, path) => {
        body = readFile(path)
        return body ? "## {date}\n\n{body}" : ""
    }).filter(nonEmpty).join("\n\n")

    if combined is empty:
        for each entry: delete(entry.path)
        return { folded: entries.map(e => e.date), failed: [] }

    try:
        await compileLongterm(combined, longtermPath, resolvedModel)
        for each entry: delete(entry.path)    // 成功才删除
        return { folded: entries.map(e => e.date), failed: [] }
    catch err:
        log.error("fold 失败，保留 {n} 份 daily 条目待下轮重试")
        return { folded: [], failed: entries.map(e => e.date) }
⚠️ 失败保留源文件是核心安全机制：try/catch 捕获异常后，失败条目留在 daily/ 目录。下一轮 daily 任务运行时，这些条目仍会被 fold 逻辑处理。如果 compileLongterm 的指纹去重判断"内容与上次相同"会跳过，但 rollDailyWindow 在失败场景下指纹不会更新，确保了重试能真正执行。

三、compileLongterm：通用 fold 入口

3.1 功能定位

compileLongterm 是一个通用 fold 入口，接受任意文本内容（已由调用方读好的文本，不是文件路径），将其合并进 longterm.md。被两个场景复用：

  • rollDailyWindow：把滚出 6 日窗口的 daily 条目 fold 进 longterm
  • migrateLegacyWeekToLongterm：一次性、幂等地把旧版 v3 的 week.md 整段 fold 进 longterm（迁移后更名为 .migrated.bak）

3.2 核心伪代码

function compileLongterm(content, longtermPath, resolvedModel):
    newContent = String(content).trim()
    if newContent is empty: return "skipped"

    fp = md5([newContent])
    if readFile(fpPath) == fp and fileExists(longtermPath):
        return "skipped"              // 指纹去重

    prevLongterm = readFile(longtermPath)

    input = prevLongterm
        ? "## 上一份长期情况\n\n{prevLongterm}\n\n## 新沉淀内容\n\n{newContent}"
        : "## 新沉淀内容\n\n{newContent}"

    result = callLLM(
        systemPrompt = buildCompileLongtermPrompt(locale),
        userContent  = input,
        maxTokens    = 600,
        temperature  = 0.3
    )

    writeFile(longtermPath, normalizeOutput(result))
    writeFile(fpPath, fp)
    return "compiled"
设计点实现理由
指纹去重按输入内容的 MD5 哈希去重同一批 daily 条目可能因断点续跑被重复 fold
取舍而非追加Prompt 要求"不要追加，必须做取舍、抽象、合并"longterm 是唯一长期产物，每次追加会无限膨胀
maxTokens: 600长期记忆允许最长输出需容纳用户身份、人格、长期偏好等较多信息
通用 fold 设计接受内容字符串而非文件路径允许调用方自行决定输入来源，不绑定文件系统

四、compileEditableFacts：重要事实增量编译

4.1 功能定位

compileEditableFacts 从 session 摘要的 "### 重要事实 / Key Facts" 段提取用户画像类信息，编译成 facts.md。这是唯一一个用户可编辑的记忆产物——用户可以手动修正 LLM 提取的事实。

💡 关键设计：增量编译 + 独立水位线。与 compileToday 和 compileDaily 不同，compileEditableFacts 有独立的水位线状态文件 editable-facts-state.json，只处理上次编译之后新增/修订的摘要中的 facts 段。这是唯一的事实产物——曾经的 memory.editable_facts 实验字段已毕业。

4.2 核心伪代码

function compileEditableFacts(summaryManager, outputPath, resolvedModel):
    baseline = ensureEditableFactsBaseline(memoryDir)
    if baseline.changed: return "compiled"

    state = readEditableFactsState(statePath)
    since = max(state.lastCompiledAt, opts.since)

    newSessions = summaries.filter(s => s.updated_at > since)
    if newSessions is empty: return "skipped"

    factParts = []
    for each session in newSessions:
        if not hasFactSectionHeading(session.summary):
            continue
        factText = extractFactSection(session.summary)
        if factText and not isEmptyFactSection(factText):
            factParts.push(factText)

    nextWatermark = max(updated_at of newSessions)
    if factParts is empty:
        updateWatermark(statePath, nextWatermark)
        return "compiled"

    prevFacts = readFile(outputPath)
    newFacts = factParts.join("\n")
    combined = prevFacts
        ? "## 当前可信 Facts\n\n{prevFacts}\n\n## 新增候选 Facts\n\n{newFacts}"
        : "## 新增候选 Facts\n\n{newFacts}"

    result = callLLM(
        systemPrompt = buildCompileEditableFactsPrompt(locale),
        userContent  = combined,
        maxTokens    = 300,
        temperature  = 0.3
    )

    writeFile(outputPath, normalizeOutput(result))
    updateWatermark(statePath, nextWatermark)
    return "compiled"

五、检验一下（提取练习）

Q1. 为什么 v4 不再用 LLM 编译 week 摘要？

A. LLM 太贵了，用不起 B. 旧方案有三大问题：重复计算（每天重编7天）、输出不稳定（模型随机性）、摘要精度递减（摘要的摘要） C. week 摘要不重要，直接删掉就行 D. v4 改用了更简单的算法，但效果更差
看答案B。v3 的 compileWeek 每天对过去 7 天重编，LLM 输出不稳定导致"历史在变化"，且对已蒸馏内容做二次蒸馏造成信息损失。v4 用零 LLM 的文件拼接替代，每天的 daily 条目独立产出、不再变化。

Q2. rollDailyWindow 的"失败保留源文件"机制在什么情况下触发？

A. 当 daily/ 目录为空时 B. 当 compileLongterm 抛出异常时——保留所有 daily 文件，下一轮重试，不静默丢弃 C. 当用户手动删除了 longterm.md 时 D. 当日期切换发生在凌晨时
看答案B。try/catch 包裹 compileLongterm 调用，只有成功后才会 delete 源 daily 文件。如果 LLM 调用失败（网络错误、凭证过期等），所有 daily 条目完好保留，等待下一轮 daily 任务重试。

Q3. compileLongterm 的"通用 fold 入口"设计有什么好处？

A. 代码更短 B. 接受内容字符串而非文件路径，允许 rollDailyWindow 和 migrateLegacyWeekToLongterm 两个场景复用同一套指纹去重 + LLM 合并逻辑 C. 让 longterm.md 文件更大 D. 减少了对文件系统的依赖
看答案B。compileLongterm 不绑定文件路径——调用方决定输入来源（daily 条目拼接、旧 week 迁移等），compileLongterm 只负责指纹去重 + LLM 取舍合并 + 落盘。这是典型的"通用核心 + 场景适配"设计。

Q4. compileEditableFacts 与 compileToday 在水位线机制上有什么不同？

A. 完全相同，没有任何区别 B. compileEditableFacts 有独立的 editable-facts-state.json 水位线文件，只跟踪 facts 段相关摘要的时间戳，与 compileToday 的水位线完全独立 C. compileEditableFacts 没有水位线，每次都全量编译 D. compileEditableFacts 共享 compileToday 的水位线
看答案B。compileEditableFacts 有独立的状态文件 editable-facts-state.json 和自己的 lastCompiledAt 水位线。它只关心摘要中的 "### 重要事实" 段的变化，与 compileToday 关注的 timeline 段完全独立。
