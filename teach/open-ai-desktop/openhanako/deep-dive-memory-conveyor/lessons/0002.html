第 2 课 · 编译流水线深度拆解：水位线状态机与指纹去重 算法核心

🎯 本节目标：学完你能解释 compileToday 的水位线增量扫描机制（7 步算法），理解 compileDaily 的指纹去重原理和三输入源降级策略，以及为什么 compileDaily 必须在 compileToday 之前执行。

传送带的前两级编译器——compileToday 和 compileDaily——是每天运行最频繁的两个组件。它们共享一个关键约束：执行顺序不可颠倒。理解这个约束的原因，是理解整个 daily 任务调度逻辑的钥匙。

一、compileToday：水位线增量编译

1.1 功能定位

compileToday 是传送带的第一级蒸馏器，负责将当天进行中的 session 摘要编译成 today.md 草稿。它是唯一一个高频更新的编译器（每 10 轮对话触发一次）。

💡 关键设计：水位线增量扫描（watermark incremental scan）。每次调用不重新扫描所有摘要，而是通过 today-state.json 记录上次扫过的摘要时间戳，只处理新增或修订过的摘要，然后以 delta 形式注入 LLM 续写草稿。

1.2 七步算法流程

1逻辑日检测与切换：调用 getLogicalDay() 获取当前逻辑日（按 DAY_BOUNDARY_HOUR 划分，默认凌晨 4 点为日界）。如果 today-state.json 中的日期与当前逻辑日不一致，清空 today.md 并重置状态。
compile.ts → getLogicalDay()
2水位线读取：从 today-state.json 读取 lastCompiledSummaryUpdatedAt——上次编译时处理过的最新摘要的 updated_at 时间戳。
compile.ts → readTodayState()
3增量扫描：调用 getCandidateSummariesForCompile(summaryManager, watermark)，只返回 updated_at > watermark 的摘要。如果为 0 条且已有水位线，保留当前草稿退出（暂无新内容）。如果为 0 条且无水位线，清空草稿（防止陈旧内容残留）。
compile.ts → getCandidateSummariesForCompile()
4时间线条目提取与过滤：对增量摘要调用 timelineEventsForLogicalDate()，从 "### 事情经过" 段提取条目，按 YYYY-MM-DD HH:MM 正则提取时间戳，只保留逻辑日匹配当前日的条目。对没有规范 timeline 段的旧摘要，使用 fallbackSummaryAsEvent。
compile.ts → timelineEventsForLogicalDate()
5delta 标注（新增 vs 修订）：对每条事件判断其 session 的 created_at 是否晚于水位线——晚于则标注"新增"，早于但 updated_at 晚于则标注"取代先前相关记述"（旧内容需被替换而非并列）。
compile.ts → formatTimelineEvents()
6LLM 续写：将上一版草稿 + delta 时间线条目拼接为输入，调用 LLM（maxTokens: 450，温度 0.3），由 buildCompileTodayPrompt 提供 System Prompt。
compile.ts → buildCompileTodayPrompt()
7落盘与水位线更新：将 LLM 输出写入 today.md，更新 today-state.json 的 lastCompiledSummaryUpdatedAt 为本次处理过的最新 updated_at。
compile.ts → updateWatermark()

1.3 核心伪代码

function compileToday(summaryManager, outputPath, resolvedModel):
    state = readTodayState("today-state.json")
    today = getLogicalDay().logicalDate

    // 日期切换：重置草稿和状态
    if state exists and state.logicalDate != today:
        truncate(outputPath, "")
        state = null

    watermark = max(state.lastCompiledAt, opts.since)
    sessions = getSummariesNewerThan(watermark)

    if sessions is empty:
        if state is null: truncate(outputPath, "")
        return "compiled"

    nextWatermark = max(updated_at of sessions)
    events = extractTimelineEvents(sessions)
               .filter(event.logicalDate == today)

    if events is empty:
        updateWatermark(statePath, today, nextWatermark)
        return "compiled"

    previousDraft = readFile(outputPath)
    delta = formatTimelineEvents(events, since=watermark,
                                  includeRevisionMarker=true)
    input = buildInput(previousDraft, delta)

    result = callLLM(
        systemPrompt = buildCompileTodayPrompt(locale),
        userContent  = input,
        maxTokens    = 450,
        temperature  = 0.3
    )

    writeFile(outputPath, result)
    updateWatermark(statePath, today, nextWatermark)
    return "compiled"

1.4 设计要点总结

设计点实现理由
水位线增量today-state.json + lastCompiledSummaryUpdatedAt避免每天对 50+ 份摘要重复编译，只在有新摘要时才调 LLM
逻辑日归属DAY_BOUNDARY_HOUR (凌晨 4:00)凌晨对话属于前一天，避免跨夜 session 被切分成两天
修订标注includeRevisionMarker → "(取代先前相关记述)"让 LLM 理解旧条目需要被替换而非并列，避免时间线上的自相矛盾
日期切换保护逻辑日变化时清空 today.md保证 today.md 始终只含今天的内容，不会残留昨天的草稿
空 sessions 兼容无水位线时清空草稿老用户升级或状态丢失时，不能信任旧草稿的时效性

二、compileDaily：日记条目指纹编译

2.1 功能定位

compileDaily 将已经翻篇的昨天蒸馏成 memory/daily/{date}.md 里的一条 2-3 句日记条目。与 compileToday 的关键区别：compileToday 编译"当天进行中"的草稿（高频更新），compileDaily 编译"已经翻篇的那天"（一天只落一次盘，按指纹覆盖不追加）。

⚠️ 执行顺序约束：compileDaily 必须先于 compileToday 执行。原因：compileDaily 读取的是"昨天最终版 today.md 草稿"，这份草稿在日期切换前仍然躺在 today.md 里；compileToday 一旦先跑，日期切换会把 today.md 重置为新一天的空白草稿，昨天的内容就再也读不到了。这个顺序约束在 memory-ticker.ts 中通过 DAILY_STEP_KEYS 数组的固定顺序强制执行。

2.2 六步算法流程

1输入源优先级匹配：先尝试从 session 摘要的 timeline 段提取精确时间线条目。如果 timelineEventsForLogicalDate 返回非空，以 timeline 为输入。如果无 timeline 则回落到 today.md 草稿。两者都没有则回落到按当天摘要整段编译（includeFallback: true）。
compile.ts → compileDaily()
2指纹计算与去重：对输入源的事件 key 列表（sessionId:updatedAt:index:timeLabel:sha1(前12位)）做 MD5 哈希，写入 {date}.md.fingerprint。如果指纹文件已有且与当前指纹一致且产物文件存在，直接跳过（返回 "skipped"）。
compile.ts → 指纹去重逻辑
3草稿不可用时的受控降级：如果 todayDraftPath 指向的文件不存在或为空且 timeline 条目也为空，通过 fallbackSummaryAsEvent 回退到整段摘要编译。记录 warn 日志。
compile.ts → fallbackSummaryAsEvent()
4零占位策略：如果当天确实没有任何内容（无 timeline、无草稿、摘要也无法归到该逻辑日），不产文件。同时清理可能存在的旧指纹文件，避免之后补齐时被过期指纹挡住。
compile.ts → 零占位检查
5LLM 蒸馏：maxTokens: 100，温度 0.3。这是所有编译器中 token 预算最紧的一个——每条日记只有 60 字（中文）或 30 词（英文）的上限。
compile.ts → callLLM()
6落盘：输出格式 ## {date}\n\n{body}\n，写入 daily/{date}.md + 指纹文件。
compile.ts → writeFile()

2.3 核心伪代码

function compileDaily(summaryManager, dailyDir, logicalDate, resolvedModel):
    outputPath = dailyDir + "/{logicalDate}.md"
    fpPath = outputPath + ".fingerprint"

    // 优先使用 timeline 条目，回落到 today 草稿
    draftText = readFile(todayDraftPath)
    summaries = getSummariesForDate(logicalDate)
    timelineEvents = extractTimelineEvents(summaries)
                       .filter(event.logicalDate == logicalDate)

    if timelineEvents is not empty:
        input = formatTimelineEvents(timelineEvents)
        fpKeys = timelineEvents.map(event => event.key)
    else if draftText is not empty:
        input = draftText
        fpKeys = ["draft:" + draftText]
    else:
        fallback = extractFallbackEvents(summaries, logicalDate)
        if fallback is empty:
            delete(fingerprintFile)      // 零占位
            return "skipped"
        log.warn("草稿不可用，回落到摘要编译")
        input = formatTimelineEvents(fallback)
        fpKeys = fallback.map(event => event.key)

    // 指纹去重
    fp = md5(fpKeys.join("\n"))
    if readFile(fpPath) == fp and fileExists(outputPath):
        return "skipped"

    result = callLLM(
        systemPrompt = buildCompileDailyPrompt(locale),
        userContent  = input,
        maxTokens    = 100,          // 最紧的 budget
        temperature  = 0.3
    )

    body = normalizeOutput(result)
    writeFile(outputPath, "## {logicalDate}\n\n{body}\n")
    writeFile(fpPath, fp)
    return "compiled"

2.4 设计要点总结

设计点实现理由
指纹去重事件 key 的 MD5 哈希 + 独立 .fingerprint 文件防止同一天被重复编译（多次切换日期、重启），指纹不与产物文件耦合
三输入源降级timeline → draft → fallback summary确保任何数据状态下都能产出日记：有规范 timeline 最好，没有就用草稿，草稿也没有就用旧摘要
零占位无内容时不产文件 + 清理旧指纹避免 daily/ 目录被空文件污染；清理旧指纹防止后续真实编译被误判为"已处理"
不追加指纹命中直接 skip，不覆盖已存在文件compileDaily 是一天只落一次盘的"终版"编译，不同于 compileToday 的草稿增量模型

三、检验一下（提取练习）

Q1. compileToday 的水位线机制解决了什么问题？

A. 让 LLM 调用更快 B. 避免每天对 50+ 份摘要重复编译——只处理上次编译之后新增或修订的摘要，以 delta 形式续写草稿 C. 让 today.md 文件更小 D. 让日期切换更准确
看答案B。水位线通过 today-state.json 记录 lastCompiledSummaryUpdatedAt，每次只扫描 updated_at > watermark 的新摘要。这是编译链降本的核心机制之一。

Q2. 为什么 compileDaily 必须在 compileToday 之前执行？

A. 因为 compileDaily 更快，先跑可以减少等待时间 B. 因为 compileDaily 需要读取昨天的 today.md 草稿，而 compileToday 一旦先跑会把 today.md 重置为新一天的空白草稿 C. 因为 compileDaily 的 token 预算更紧 D. 两者没有顺序依赖，可以任意执行
看答案B。日期切换时，compileDaily 需要昨天的 today.md 作为输入源之一。如果 compileToday 先执行，日期切换逻辑会把 today.md 清空（新一天从空白开始），compileDaily 就读不到昨天的内容了。

Q3. compileDaily 的指纹去重是如何工作的？

A. 比较 today.md 文件的修改时间 B. 对输入源的事件 key 列表做 MD5 哈希，与 .fingerprint 文件对比，相同则跳过 C. 比较 daily/{date}.md 的文件大小 D. 让 LLM 判断内容是否重复
看答案B。指纹由事件 key 列表（sessionId:updatedAt:index:timeLabel:sha1前12位）的 MD5 哈希生成，存入 {date}.md.fingerprint。如果指纹相同且产物文件已存在，直接返回 "skipped"，完全不调 LLM。

Q4. compileDaily 的三输入源降级顺序是什么？

A. today.md → daily.md → longterm.md B. timeline 条目 → today.md 草稿 → 摘要回落（fallbackSummaryAsEvent） C. session 摘要 → LLM 生成 → 人工审核 D. facts → today → week → longterm
看答案B。优先级：规范 timeline 条目（最佳）→ today.md 草稿（次选）→ 旧摘要整段回落（兜底）。三级降级保证在任何数据状态下都能产出日记，不会静默丢失数据。
