前面三课聚焦于"如何生产记忆内容"。这一课我们关注"如何把记忆注入到每次 LLM 调用中"——从 assemble 拼接、到 token 预算控制、到 Prompt 缓存优化。

一、assemble：四段拼接终产物 memory.md

1.1 功能定位

assemble 是一个纯同步函数（不调 LLM），将四个中间文件按固定模板拼成 memory.md：

facts.md today.md week.md longterm.md ↓ ↓ ↓ ↓ └─────────────┴─────────────┴───────────────┘ ↓ buildCompiledMemoryMarkdown ↓ memory.md ┌──────────────────────────────────────────────┐ │ ## 重要事实 / Key facts │ │ (facts.md 内容 或 "(暂无)") │ │ │ │ ## 今天 / Today │ │ (today.md 内容 或 "(暂无)") │ │ │ │ ## 本周早些时候 / Earlier this week │ │ (week.md 内容 或 "(暂无)") │ │ │ │ ## 长期情况 / Long-term context │ │ (longterm.md 内容 或 "(暂无)") │ └──────────────────────────────────────────────┘

1.2 关键约束

1空栏占位符：四个标题始终保留。空内容栏写 "(暂无)" 或 "(none)"，避免格式漂移——即使某天没有任何记忆产物，模板结构不变。
2标题固定双语：中文环境用 "重要事实/今天/本周早些时候/长期情况"，英文环境对应英文标题。标题不随内容变化。
3零 LLM：终极核心理念——任何一个编译步骤（compileToday/compileDaily/compileLongterm/compileEditableFacts）都是独立对各段内容做蒸馏，assemble 只做拼接。分层编译 + 统一拼接 = 关注点分离。

二、Token 预算控制：2000 token 上限的分配

2.1 硬约束

HanaAgent 记忆系统的硬约束是：memory.md 总长度不超过 2000 token（在 buildSystemPrompt 注入时截断）。这个约束驱动了每一层编译器的设计参数。

2.2 各层 maxTokens 分配表

编译器输出
maxTokens		Prompt
字数上限		产物
文件		目的
compileToday 450 300 字 / 180 词 today.md 今天的粗颗粒事件，3-5 条，每条 1-2 句
compileDaily 100 60 字 / 30 词 daily/{date}.md 每天 2-3 句日记，6 天拼起来约 360 字
compileLongterm 600 400 字 / 240 词 longterm.md 稳定用户画像核心，允许最长
compileEditableFacts 300 200 字 / 120 词 facts.md 可编辑的重要事实，宁概括勿堆叠

2.3 层级分配的深层逻辑

💡 为什么 today 450、daily 100、longterm 600？这不是简单的等比例分配，而是基于信息时效性与稳定性的权衡：
  • today (450 tokens)——今天的内容最直接影响用户当前对话体验，需要保持最高细节度。
  • daily (100 tokens)——已结束的日子只需要"一眼扫过"级别的粒度。
  • longterm (600 tokens)——虽然信息是"旧的"，但它沉淀的是用户最稳定的画像核心（人格、审美、长期关系、关注方向），需要较多文字来描述，因此给最高 budget。
  • facts (300 tokens)——信息密度极高（原子事实），不需要很大的 token 预算。

2.4 推理模型缓冲区

llm-budget.ts 中的 withMemoryReasoningBuffer 函数处理推理模型（如 Claude Opus 4.5 + extended thinking）的特殊情况：

function withMemoryReasoningBuffer(visibleMaxTokens, resolvedModel):
    visible = positiveInteger(visibleMaxTokens)
    if not isReasoningModel(model):
        return visible            // 非推理模型：不调整

    buffer = 1024                  // 默认推理缓冲区
    requested = visible + buffer

    modelLimit = model.maxTokens   // 模型自身最大输出限制
    if not modelLimit: return requested

    return max(visible, min(modelLimit, requested))

如果模型开启了 reasoning，实际请求的 maxTokens 会在 visibleMaxTokens 基础上额外加 1024 作为推理缓冲区。最终不超过模型自身最大输出限制。

2.5 总 token 预算验证

✓峰值估算：facts(200字) + today(300字) + week(6x60=360字) + longterm(400字) + 标题与分隔符(~50字) = 约 1310 字。按中文约 1.5 token/字计算，约 1965 token，刚好在 2000 token 上限内。

实战中大多数天的实际 token 远低于此峰值——空闲日 today.md 只有 2-3 条短事件、week 段通常不到 6 条（有些天没有活动）、longterm 做了取舍合并不会无限增长。

三、Prompt 工程设计

3.1 Prompt 体系总览

记忆系统的 Prompt 主要分布在 lib/memory/prompts/compile.ts（164 行），导出四个函数：

导出函数用途templateVersion
buildCompileTodayPromptcompileToday 的 System Promptcompile-today.v2
buildCompileDailyPromptcompileDaily 的 System Promptcompile-daily.v2
buildCompileLongtermPromptcompileLongterm 的 System Promptcompile-longterm.v1
buildCompileEditableFactsPromptcompileEditableFacts 的 System Promptcompile-editable-facts.v1

3.2 各层 Prompt 的核心差异

Prompt输入输出粒度核心过滤逻辑字数上限
compileToday 上一版草稿 + delta 时间线条目 3-5 条粗颗粒事件，保留时间锚点 保留粗颗粒事件和用户画像，过滤工作细节 300 字
compileDaily 一天的时间线条目或最终版草稿 2-3 句日记 与 compileToday 一样，但更强调"一眼扫过"的概览性 60 字
compileLongterm 旧长期内容 + 新沉淀内容 用户画像核心 必须做取舍、抽象、合并——不允许追加式复述 400 字
compileEditableFacts 当前可信 Facts + 新增候选 Facts 原子事实列表 新增与当前冲突时以新增为准；宁可概括合并 200 字

3.3 共享的"不要记录"清单

四个编译 Prompt 共享几乎相同的过滤语义——每一层都在做同一件事：把过程噪音过滤掉，只保留用户画像信息。不同层级的过滤力度不同。

✗所有 Prompt 都禁止的：执行步骤、文件名、工具、命令；协作偏好、检查顺序、汇报格式；具体子问题、具体方案、具体改法；助手产出的具体内容；来回修改、重试、被打断这类过程波动。
✓所有 Prompt 都允许的：用户的身份、人格特质、审美、兴趣；用户喜欢或讨厌的事物；长期关系和稳定生活背景；用户最近关注的大主题（不是细节）；生活、创作或长期关注方向的变化。

3.4 Prompt 缓存策略

每个 Prompt 都声明了 cacheGroup（如 memory.compile.today），配合 buildUtilityPromptLayout 实现 prompt 缓存：

const layout = buildUtilityPromptLayout({
    cacheGroup: "memory.compile.today",
    templateVersion: "compile-today.v2",
    systemPrompt: promptSpec.systemPrompt,
    userContent: input,
})

同一类型的编译调用（如所有 compileToday 调用）共享缓存的 System Prompt prefix，减少每次调用的 token 消耗和延迟。

四、检验一下（提取练习）

Q1. assemble 函数的核心理念是什么？

A. 用 LLM 对四段内容做最终润色后拼接 B. 分层编译 + 统一拼接——每个编译步骤独立蒸馏各自段落，assemble 只做零 LLM 的模板拼接 C. 把四段内容压缩成一段 D. 按重要性排序后注入 system prompt
看答案B。compileToday/compileDaily/compileLongterm/compileEditableFacts 各自独立编译，assemble 不做任何 LLM 调用，只是按固定模板拼接四段内容。这是"关注点分离"的体现——编译和装配是两个正交的职责。

Q2. 2000 token 预算中，为什么 longterm 分配最多（600 tokens）？

A. 因为 longterm.md 文件最大 B. 因为长期记忆沉淀的是用户最稳定的画像核心（人格、审美、长期关系），需要较多文字描述，虽然信息是"旧的"但不随时间迅速衰减 C. 因为 compileLongterm 的调用频率最高 D. 因为 longterm 的内容最重要，应该无限制
看答案B。today 的 450 服务于"当前细节"，而 longterm 的 600 服务于"稳定画像"。长期记忆需要容纳用户身份、人格、审美、长期兴趣等多维度信息，文字量自然需要更多。这是"时效性 vs 稳定性"的权衡。

Q3. 推理模型缓冲区的作用是什么？

A. 让推理模型输出更长 B. 在 visibleMaxTokens 基础上额外加 1024 作为推理过程的 token 预算，因为推理模型的 thinking 过程会消耗输出 token C. 减少推理模型的延迟 D. 让推理模型更准确
看答案B。推理模型（如 extended thinking）在输出最终答案前会进行内部推理，这个过程也消耗输出 token。如果不加缓冲区，visibleMaxTokens 中会被推理过程占用一部分，实际可见输出会减少。1024 缓冲区确保用户看到的记忆内容不受推理过程挤占。

Q4. 四个编译 Prompt 的共享"不要记录"清单体现了什么设计原则？

A. 代码复用——四个 Prompt 共享同一段文本 B. 过滤语义一致性——每一层都在做同一件事：把过程噪音过滤掉，只保留用户画像信息。不同层级的过滤力度不同，但过滤方向一致 C. 为了减少 token 消耗，所有 Prompt 共享同一个 D. 没有特殊原因，只是巧合
看答案B。四个 Prompt 的"允许/禁止"清单高度一致，因为它们共同服务于同一个目标——把对话中的过程噪音（命令、文件名、具体方案）过滤掉，只保留对用户画像有价值的信息。区别在于各层的过滤力度：today 保留更多细节，longterm 只保留最稳定的核心。
