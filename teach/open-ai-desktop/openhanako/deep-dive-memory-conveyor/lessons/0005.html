💡 成本优化的四个关键点：
  • 指纹去重——compileDaily 和 compileLongterm 的指纹机制使相同输入的重复调用被跳过（成本为零）
  • 水位线增量——compileToday 和 compileEditableFacts 只处理新增摘要，避免每天全量扫描
  • 零 LLM week——assembleWeekFromDaily 纯文件操作，完全无 LLM 成本
  • token 预算紧缩——每层编译器的 maxTokens 都经过精心校准，不浪费任何 token

二、六大已知局限与改进方向

2.1 相关性排序缺失

⚠️ 问题：记忆系统目前只按时间维度组织信息（today → daily → week → longterm），缺少主题相关性排序。例如，当用户讨论"记忆系统"时，today.md 可能讲的是"桌面自动化"，而 longterm.md 中关于"记忆系统"的内容虽然更相关，却排在后面。
改进方向：在 assemble 阶段引入轻量语义重排，把与当前对话最相关的段落放在最前面。

2.2 遗忘曲线不精确

⚠️ 问题：当前的时间衰减是离散的阶梯函数——今天(today) → 昨天(daily) → 6天内(week) → 长期(longterm)。真实的人类记忆遗忘服从连续的艾宾浩斯曲线，而非阶梯跳跃。今天 23:59 的对话属于 today.md，明天 00:01 就变成 60 字日记——信息细节度在分钟级别发生巨大跳跃。
改进方向：引入连续衰减权重——today.md 条目带上"距现在多少小时"的权重字段，week 段按天数衰减排序。

2.3 中文时间提取精度

⚠️ 问题：时间条目标题提取依赖正则匹配 YYYY-MM-DD HH:MM 格式，但 LLM 有时输出"2024年1月15日 下午3点"或"昨天 14:30"，这些格式会被正则完全跳过。中文语境的相对时间表达（"昨天下午"、"上周五"）也无法被正则提取。
改进方向：引入轻量 NLP 时间解析器，在正则失败后做第二级提取；对相对时间结合消息时间戳推算绝对时间。

2.4 compileDaily 输入依赖的脆弱性

⚠️ 问题：compileDaily 的输入链（timeline → today.md → 摘要回落）有两个脆弱点：today.md 可能因 runningSummary 持续失败而为空；compileDaily 必须先于 compileToday 执行的顺序约束仅靠 DAILY_STEP_KEYS 数组维护，没有代码层面的 enforce。
改进方向：在日期切换前保存 today.md 的快照副本，确保"昨天的草稿"在任何执行顺序下都能读取。

2.5 longterm.md 的不可逆转性

⚠️ 问题：一旦 daily 条目被 fold 进 longterm.md 并删除源文件，原始 daily 内容永久丢失。如果 compileLongterm 做了不恰当的抽象（比如把"喜欢爵士乐"概括成"喜欢古典音乐"），没有回溯修正的机制。
改进方向：保留 daily 文件的快照备份（压缩归档而非删除），或在 longterm.md 中保留对源 daily 日期的引用（traceability link）。

2.6 无跨 session 的语义关联

⚠️ 问题：每个 session 的滚动摘要是独立生成的，跨 session 的语义关联完全依赖后续编译链。如果用户周一讨论"想学 Rust"，周三又讨论"在看 Rust 教程"，两个 session 会独立记录，没有跨天关联概览。
改进方向：在 compileToday 之前新增"跨 session 主题聚类"步骤，把多个 session 摘要中相同主题的事件合并成带跨天时间范围的事件。

三、设计精髓总结

HanaAgent v4 记忆传送带模型用一套精心校准的多层编译器，解决了"有限 token 预算内提供最高信号密度"这一核心问题。三个关键工程决策：

  1. week 段零 LLM——不再对已蒸馏内容做二次蒸馏，改为纯文件装配。这不只是降本，更是保证信息稳定性。
  2. 指纹 + 水位线双重去重——指纹防止相同内容被重复编译（compileDaily/compileLongterm），水位线防止每天全量扫描（compileToday/compileEditableFacts）。
  3. 每层可编辑——用户可以对 facts.md、today.md、daily/{date}.md、longterm.md 的每一层手动编辑。用户永远是记忆的最终权威。

四、综合测验（提取练习）

Q1.（综合）以下哪项不属于 HanaAgent v4 传送带模型的编译器？

A. compileToday —— 今天草稿水位线增量编译 B. compileDaily —— 日记条目指纹编译 C. compileWeek —— 每周摘要 LLM 编译 D. compileEditableFacts —— 重要事实增量编译
看答案C。compileWeek 是旧版 v3 的设计，v4 已废弃。v4 用 assembleWeekFromDaily（零 LLM 文件拼接）替代了 compileWeek。五个编译器是：compileToday、compileDaily、compileLongterm、compileEditableFacts、以及 assembleWeekFromDaily（装配器而非编译器）。

Q2.（综合）compileDaily 的"零占位策略"意味着什么？

A. 每天必定产出一个 daily 文件，即使内容为空 B. 当天确实没有任何内容时不产文件，同时清理旧指纹，防止后续真实编译被过期指纹挡住 C. daily 文件占用零字节磁盘空间 D. 用空字符串占位，等待后续补充
看答案B。零占位策略的核心是"没有内容就不产文件"——不会在 daily/ 目录写入空文件。同时清理可能存在的旧 .fingerprint 指纹文件，确保之后有内容需要编译时不会被旧指纹误判为"已处理"而跳过。

Q3.（综合）rollDailyWindow 和 compileLongterm 的关系是什么？

A. rollDailyWindow 是 compileLongterm 的替代品，两者互斥 B. rollDailyWindow 负责筛选溢出条目并拼接内容，compileLongterm 是通用 fold 引擎负责指纹去重和 LLM 合并；rollDailyWindow 调用 compileLongterm C. compileLongterm 是 rollDailyWindow 的别名 D. 两者完全独立，没有调用关系
看答案B。rollDailyWindow 是"调度层"——计算窗口边界、筛选溢出条目、拼接内容、管理文件删除。compileLongterm 是"引擎层"——接受内容字符串、做指纹去重、调 LLM 合并、落盘。rollDailyWindow 调用 compileLongterm 完成实际的 fold 操作。

Q4.（综合）以下哪个场景体现了传送带模型的"断点续跑"设计？

A. 用户关闭应用后重新打开，记忆从零开始重建 B. 如果 compileDaily 执行失败（网络错误），daily 文件不会丢失——compileToday 仍然正常执行，下一轮 daily 任务重试 compileDaily C. 用户每天必须手动触发记忆编译 D. 记忆系统每次启动都重新编译所有历史数据
看答案B。断点续跑体现在两个层面：1）daily 任务中各步骤独立执行，前一步失败不阻塞后续独立步骤（compileFacts 和 deepMemory 与前三步独立）；2）rollDailyWindow 失败时保留源文件，下一轮重试。系统不会因为单次失败而丢失数据。

Q5.（综合）如果让你为传送带模型增加一个"月度总结"功能，最合理的设计是？

A. 新增 compileMonth 编译器，对 longterm.md 再做一次 LLM 蒸馏 B. 参考 assembleWeekFromDaily 的零 LLM 设计——从 daily/ 目录取最近 30 天的日记条目拼接，不做 LLM 编译 C. 直接把 longterm.md 改名为 month.md D. 让用户每天手动写月度总结
看答案B。遵循 v4 的设计哲学——"对已蒸馏内容不做二次蒸馏"。assembleWeekFromDaily 已经证明零 LLM 文件拼接既能降本又能保证信息稳定性。月度总结可以类似地从 daily/ 目录取 30 天条目拼接，配合超长截断策略（从最老条目开始丢弃）。如果确实需要抽象，可以让 compileLongterm 的 fold 过程自然完成月度级别的信息沉淀。

Q6.（综合）以下哪个是传送带模型的正确数据流路径？

A. session 摘要 → compileDaily → today.md → compileToday → daily/{date}.md B. session 摘要 → compileToday → today.md → (日期切换) → compileDaily → daily/{date}.md → (滚出窗口) → compileLongterm → longterm.md C. session 摘要 → compileLongterm → longterm.md → compileToday → today.md D. longterm.md → compileDaily → daily/{date}.md → compileToday → today.md
看答案B。正确路径：session 摘要 → compileToday（当天高频更新 today.md）→ 日期切换 → compileDaily（蒸馏昨天为 diary 条目）→ 滚出 6 日窗口 → compileLongterm（fold 进 longterm）。同时 facts.md 通过 compileEditableFacts 独立于这条主线增量更新。
