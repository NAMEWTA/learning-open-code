第 1 课 · 记忆传送带设计哲学：为什么需要分层记忆 设计哲学

🎯 本节目标：学完你能解释为什么 AI 桌面伴侣不能简单地"记住所有对话"，能画出 HanaAgent v4 的四层记忆架构，能对比五种记忆方案的优劣并说出分层蒸馏传送带方案的核心取舍。

AI 桌面伴侣每天与用户进行数十轮对话。如果将这些对话的原始文本全量塞进每次 LLM 调用的 system prompt 中，会遇到四个核心矛盾。理解这些矛盾，是理解整个记忆系统设计的起点。

一、四个核心矛盾：为什么"全记住"行不通

矛盾一：上下文窗口有限

即使现代模型支持 128K+ token 上下文，记忆的本质是在有限的预算内提供最高信号密度的信息。把全部对话历史灌进去，模型会在噪音中迷失，反而降低回答质量。

矛盾二：成本与延迟不可忽视

每次调用 LLM 都把 50 轮对话全文塞入上下文，token 花费巨大，首 token 延迟也显著增加。对于高频交互的桌面应用不可接受。

矛盾三：信号密度随时间递减

3 天前的详细对话记录对今天的价值远低于最近 1 小时内的。信息时效性天然服从衰减曲线：越近越重要，越远越需要压缩。

矛盾四：用户画像 vs 工作日志

全量保留会把"执行步骤、文件名、工具命令"等过程噪音与"用户人格特质、长期兴趣"等核心信息混在一起，稀释了真正有用的内容。

💡 核心洞察：分级蒸馏不是技术妥协，而是对"信息价值随时间衰减"这一客观规律的工程化建模。不同时效性的信息需要不同粒度的摘要——新近信息保持细节、中期信息压缩成日记条目、远期信息只保留最稳定的用户画像核心。

二、传送带模型：四层记忆架构

HanaAgent v4 记忆系统采用"每日滚动传送带"模型。类比物理世界的传送带——信息从新鲜到沉淀，逐层向下传递，越远越压缩：

今天 昨天 N天前 ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ session 摘要 │ ──→ compileDaily │ ──→ ... → │ 滚出窗口 │ │ compileToday │ │ → daily/{date}.md │ │ fold 进 │ │ → today.md │ │ (2-3句日记) │ │ longterm.md │ └──────────────┘ └──────────────┘ └──────────────┘ ↓ ↓ ↓ [高频更新] [每天一次] [按需折叠] (每10轮触发) (日期切换触发) (窗口外丢弃)

四个中间产物文件的对应关系：

中间产物时效性信息粒度更新频率编译器
today.md 当天进行中 粗颗粒事件（3-5条） 每 10 轮对话 / session 结束 compileToday
daily/{date}.md 已结束的那天 2-3句日记 每天一次（日期切换后） compileDaily
week.md 最近 6 天已结束 每日一条目 纯文件装配，零 LLM assembleWeekFromDaily
longterm.md 长期（>6天前） 用户画像核心 滚出窗口时触发 compileLongterm
facts.md 永久 可编辑的重要事实 每天一次（增量） compileEditableFacts
1第 1 层 — today.md：今天的粗颗粒草稿，每 10 轮对话增量更新。它捕捉"用户今天在做什么"，是最高频更新的层级。
编译器：compileToday，maxTokens=450
2第 2 层 — daily/{date}.md：已经翻篇的昨天，蒸馏成 60 字的日记条目。一天只落一次盘，通过指纹去重避免重复编译。
编译器：compileDaily，maxTokens=100
3第 3 层 — week.md：最近 6 天的日记条目纯文件拼接，零 LLM 调用。这是 v4 的关键降本设计——不再让 LLM 对已蒸馏的内容做"摘要的摘要"。
装配器：assembleWeekFromDaily，零 LLM
4第 4 层 — longterm.md + facts.md：长期记忆存储最稳定的用户画像核心（人格、审美、长期关系）；facts.md 存储用户可编辑的原子事实。
编译器：compileLongterm(600 tokens) + compileEditableFacts(300 tokens)

三、五种方案对比：为什么选传送带？

在设计记忆蒸馏方案时，有五种可选路径，各有明显优缺点：

方案核心做法致命缺陷
A: 全量保留 对话原文存入向量数据库，检索 top-K 注入 信号密度极低，90% 是过程噪音；无法提炼用户画像
B: 滑动窗口 只保留最近 N 条消息原文，更早的直接丢弃 丢失所有长期信息；窗口内的消息只是碎片化片段
C: 纯向量检索 每条消息 embed 化，语义相似度搜索取 top-K 丢失时间结构（不知道用户"最近"在关注什么）；碎片化无压缩
D: 摘要链 每次对话后递归压缩：当前摘要 + 新对话 → 新摘要 错误累积（一次偏差污染后续所有摘要）；无法回溯修正
E: 分级蒸馏传送带 ✅ 5 层独立编译器，每层从原始摘要重新编译，数据逐层向下传递 LLM 调用次数更多（3-5 次/天），但每次 token 量很小，综合成本可控
💡 方案 E 的核心优势：融合了前四种方案的优点，同时避免了它们的致命缺陷—— 比 A 有更高的信号密度（每层编译器做语义提炼）； 比 B 保留了长期用户画像（longterm.md 不受窗口限制）； 比 C 保留了时间结构（today/daily/week 有明确的时效性层级）； 比 D 避免了错误累积（每层从原始 session 摘要重新编译，不依赖上层输出的正确性）。

六维度对比总览：

方案长期画像时间结构信号密度LLM调用错误恢复用户可编辑
A: 全量保留差差极低0好可编辑
B: 滑动窗口无中中0好N/A
C: 纯向量检索中无中0好N/A
D: 摘要链好中高1次/对话差不可编辑
E: 分级蒸馏传送带好好高3-5次/天好每层可编辑

四、检验一下（提取练习）

Q1. 为什么"全量保留"方案对 AI 桌面伴侣是致命的？

A. 向量数据库太贵，成本不可接受 B. 对话历史中 90% 是过程噪音，无法提炼用户画像，且缺少时间衰减机制 C. 128K token 上下文窗口太小，装不下全部对话 D. 检索速度太慢，每次调用需要 5 秒以上
看答案B。全量保留的核心问题不是技术成本，而是信息质量——对话原文中绝大部分是执行步骤、命令、文件名等过程噪音，无法抽象出"用户是一个什么样的人"。此外没有时间衰减，半年前的命令和今天的对话权重相同。

Q2. 摘要链方案（方案 D）的核心缺陷是什么？

A. LLM 调用次数太多，成本过高 B. 错误累积——一次摘要偏差会污染后续所有摘要，且无法回溯修正 C. 摘要太短，信息丢失严重 D. 无法处理中文对话
看答案B。摘要链的递归压缩模式意味着每一次摘要都在前一次基础上生成。如果某次摘要产生了偏差（比如把"用户喜欢爵士乐"错误概括），后续所有摘要都会继承这个错误。v4 方案通过"每层从原始 session 摘要重新编译"来避免这个问题。

Q3. HanaAgent v4 记忆系统的四层架构从近到远是？

A. session → daily → week → month B. today → daily → week → longterm + facts C. cache → summary → archive → delete D. hot → warm → cold → frozen
看答案B。四层架构：today.md（今天草稿）→ daily/{date}.md（日记条目）→ week.md（6天窗口装配）→ longterm.md + facts.md（长期画像+可编辑事实）。

Q4. v4 传送带方案为什么比摘要链方案（方案 D）的 LLM 调用次数更多，却被认为是更好的方案？

A. 因为 LLM 调用越多，效果越好 B. 因为每次调用都是小 token 量，综合成本可控，且避免了错误累积，每层可独立编辑和修正 C. 因为 v4 用了更便宜的模型 D. 因为 v4 不需要 LLM 调用
看答案B。v4 每天 3-5 次编译调用，每次 maxTokens 只有 60-600，综合成本 ~$0.03-0.05/天。关键优势是：每层从原始摘要重新编译（而非依赖上层输出），避免了错误累积；每层产物（today/daily/longterm/facts）都可以由用户手动编辑。
