上一课我们了解了插件系统的整体架构和安装入口。这一课深入插件发现的起点——scan() 如何从文件系统发现插件，以及 _readPluginDescriptor() 如何将文件目录转化为结构化的 PluginDescriptor。

一、scan()：启动时批量发现 + 双层去重

scan() 遍历多个 pluginsDirs（靠前的优先，如先 builtin 后 community），对每个子目录调用 _readPluginDescriptor()。

scan() {
  const results = [];
  const seen = new Set<string>();
  for (let i = 0; i < this._pluginsDirs.length; i++) {
    const dir = this._pluginsDirs[i];
    const source = i === 0 ? "builtin" : "community"; // 第一个目录 = builtin
    if (!fs.existsSync(dir)) continue;
    for (const entry of fs.readdirSync(dir, { withFileTypes: true })) {
      if (!entry.isDirectory() || entry.name.startsWith(".")) continue;

      // 第一层去重：同 source + 同目录名
      const dirKey = `${source}:${entry.name}`;
      if (seen.has(dirKey)) continue;
      seen.add(dirKey);

      const pluginDir = path.join(dir, entry.name);
      const desc = this._readPluginDescriptor(pluginDir, entry.name);
      desc.source = source;
      desc.pluginKey = createPluginKey(source, desc.id);

      // 第二层去重：同 source + 同 id（目录名可能不同于 manifest.id）
      const idKey = `${source}:id:${desc.id}`;
      if (seen.has(idKey)) {
        log.warn(`plugin id "${desc.id}" 冲突，跳过`);
        continue;
      }
      seen.add(idKey);
      results.push(desc);
    }
  }
  this._scanned = results;
  return results;
}
1第一层去重（dirKey）：source:目录名 组合——防止同 source 下出现同名目录的重复扫描。
core/plugin-manager.ts:502-504
2第二层去重（idKey）：source:id:manifest.id 组合——目录名可能不同于 manifest.json 中的 id，防止同一 id 多次注册。
core/plugin-manager.ts:509-514
关键设计：跨 source 的同 id 插件不会被去重——这正是 shadow 机制的基础。当一个插件同时存在于 builtin 和 community 目录时，优先级规则 dev (0) > community (1) > builtin (2) 决定哪个生效，被 shadow 的插件贡献点会被 _isPluginKeyRuntimeActive() 过滤掉。

二、_readPluginDescriptor()：10 步解析过程

1格式检测：detectIncompatiblePluginFormat(pluginDir) — 检测 OpenClaw 格式。如果是则标记 formatIssue 但仍允许读取 manifest 中的 id/name/version。
2manifest.json 读取：fs.readFileSync(manifestPath, "utf-8") + JSON.parse。
3ID/名称/版本：优先从 manifest 取，fallback 到 formatIssue 中的值，最后 fallback 到目录名。
4UI 能力归一化：normalizeUiHostCapabilities() — 只保留 6 个已知能力名（external.open / clipboard.writeText / sessionFile.open / resource.open / resource.pick / resource.requestAccess），未知名被忽略并 warn。
5配置 Schema：normalizePluginConfigSchema() — 解析 manifest.contributes.configuration，归一化为 { type: "object", properties: {...}, required: [...] }。
6目录扫描：遍历 KNOWN_CONTRIBUTION_DIRS = ["tools", "routes", "skills", "agents", "commands", "providers"]，检查目录是否存在。
7生命周期检测：resolvePluginEntry() 搜索 index.ts 或 index.js 入口文件。存在则 hasLifecycle: true。
8trust 字段：只有显式写 "full-access" 才为 full-access，其他值或省略均默认 restricted。
关键规则：trust 是唯一进入高权限的钥匙
9激活事件：normalizeActivationEvents() — 如果 manifest 声明了 activationEvents[] 则取声明值（去重）；如果没声明但有 lifecycle，默认注入 ["onStartup"]。
10能力列表：normalizeCapabilityList() 区分"字段缺失"（返回 null，legacy 模式——视为声明全部）与"显式空数组"（返回 []——不需要任何敏感能力）。
这是 legacy 兼容的关键设计
注意 legacy 兼容陷阱：capabilities 为 null（老插件未声明）与 []（新插件显式空数组）语义完全不同。null 在 bus 权限校验时走 legacy 模式放行；[] 表示不需要任何能力，bus 请求敏感能力时会被拒绝。升级老插件时必须补上 capabilities: [] 声明。

三、loadAll() 的四道前置筛选

loadAll() 在扫描完成后，对每个插件描述符执行四道筛选（沿顺序短路），决定是否进入加载：

async loadAll() {
  const descriptors = this._scanned.length > 0 ? this._scanned : this.scan();
  for (const desc of descriptors) {
    const entry = this._entryFromDescriptor(desc);

    // ① disabled 列表检查（builtin 不受影响）
    if (entry.source === "community" && disabledList.includes(entry.id)) {
      entry.status = "disabled"; continue;
    }
    // ② 格式问题检查（OpenClaw 等）
    if (desc.formatIssue) {
      entry.status = "incompatible"; continue;
    }
    // ③ 社区 full-access 需用户授权
    if (desc.source === "community" && desc.trust === "full-access") {
      if (!preferencesManager?.getAllowFullAccessPlugins()) {
        entry.status = "restricted"; continue;
      }
    }
    // ④ minAppVersion 检查
    if (minVer && !semverGte(this._appVersion, minVer)) {
      entry.status = "incompatible"; continue;
    }
    // 通过全部筛选 → 进入 _loadPluginWithBoundary()
    await this._loadPluginWithBoundary(entry);
  }
}

五、检验一下（提取练习）

Q1. scan() 为什么要做两层去重？

A. 没有特别原因，只是代码冗余 B. 第一层防止同目录名重复扫描；第二层防止同 id 重复注册（目录名可能不同于 manifest 中的 id） C. 第一层是性能优化，第二层是安全检查 D. 两层都是为了防止跨 source 的同 id 冲突
看答案B。第一层用 source:目录名 去重，第二层用 source:id:manifest.id 去重。因为目录名不一定等于 manifest 中的 id（用户可以重命名目录），两层去重覆盖了两种冲突场景。注意跨 source 的同 id 不会被去重——这是 shadow 机制的基础。

Q2. _readPluginDescriptor() 第 10 步中，capabilities 为 null 和 [] 有什么区别？

A. 没有区别，都表示不需要任何能力 B. null 表示老插件未声明字段（legacy 模式，bus 权限校验放行）；[] 表示显式不需要任何敏感能力（bus 请求敏感能力时会被拒绝） C. null 表示需要所有能力，[] 表示不需要任何能力 D. null 是 JSON 解析错误，[] 是正常值
看答案B。这是 normalizeCapabilityList() 的核心设计：null（字段缺失）→ 老插件兼容模式，视为声明了全部能力；[]（显式空数组）→ 新插件明确声明不需要任何敏感能力。升级老插件时必须补上声明。

Q3. 社区插件 trust: "full-access" 但用户未开启全权开关，最终状态是什么？

A. disabled — 完全不可用 B. incompatible — 不兼容 C. restricted — 基础功能（tools/skills/commands/agents/configuration）仍可用，但不能注册 routes/extensions/providers 等 D. failed — 加载失败
看答案C。Restricted 不是错误状态——插件的核心价值（工具、技能、命令、Agent 模板、配置）仍然可用。只有 routes、extensions、providers、page、widget、slash commands 这些高权限贡献点被限制。这确保用户在未授权情况下仍能使用插件的基础能力。

Q4. loadAll() 的四道筛选中，builtin 插件不受哪几道的约束？

A. 只不受 disabled 列表约束 B. 不受 disabled 列表、用户授权开关、minAppVersion 全部三道约束 C. 只不受 minAppVersion 约束 D. 全部受约束，没有例外
看答案B。builtin 插件是系统的可信基，不经过任何筛选——不检查 disabled 列表、不检查用户全权开关、不检查 minAppVersion。它们随 App 打包分发，假设已经过审核且版本匹配。
