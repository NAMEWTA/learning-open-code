本节目标：学完你能画出 PluginBusProxy 的四级权限矩阵，能追踪 executePluginTool() 从 Agent 决策到结果返回的完整链路，理解 dev 循环与生产安装路径的关键差异。

上一课我们了解了 PluginContext 的构建和工具注册。这一课聚焦插件如何与系统通信（EventBus），以及 Agent 如何调用插件工具——从工具列表获取到结果返回的每个环节。

一、PluginBusProxy：四级权限矩阵

createPluginBusProxy() 在全局 EventBus 和插件之间建立权限化代理层，支持 4 种通信模式 + 2 种查询模式：

function createPluginBusProxy(bus, { ownerContext, grantedPermissions, allowHandle }) {
  const fullAccess = allowHandle === true;
  const canReadUsage = fullAccess || hasPermission(grantedPermissions, "usage.read");

  const proxy = {
    // 1. 发布事件 — restricted 插件不能 emit llm_usage
    emit(event, sessionPath) {
      if (!fullAccess && event?.type === "llm_usage") {
        throw forbiddenBusError("llm_usage", "emit", "usage.read");
      }
      return bus.emit(event, sessionPath);
    },

    // 2. 订阅事件 — restricted 静默过滤 llm_usage
    subscribe(callback, filter = {}) {
      const wrapped = (event, sessionPath) => {
        if (!fullAccess && !canReadUsage && event?.type === "llm_usage") return;
        return callback(event, sessionPath);
      };
      return bus.subscribe(wrapped, filter);
    },

    // 3. 请求-响应 — 校验 usage.read permission
    async request(type, payload, options) {
      assertUsagePermission(type, "request");
      return bus.request(type, payload, { ...options, caller: ownerContext });
    },

    // 4. 能力查询 — 无限制
    hasHandler: bus.hasHandler?.bind(bus) || (() => false),
    listCapabilities: bus.listCapabilities?.bind(bus) || (() => []),
    getCapability,
  };

  // 5. 注册处理器 — 仅 full-access
  if (allowHandle && typeof bus.handle === "function") {
    proxy.handle = bus.handle.bind(bus);
  }

  return Object.freeze(proxy);
}
模式Restricted 插件Full-Access 插件说明
emit(event, sessionPath)允许（不能 emit llm_usage）允许全部向全局 EventBus 发布事件
subscribe(cb, filter)允许（llm_usage 被静默过滤）允许全部按 sessionPath + types 订阅
request(type, payload)允许（受 capability 声明限制）允许全部请求-响应模式，默认 30s 超时
handle(type, handler)禁止允许注册请求处理器——最危险的权限
getCapability(type)允许允许查询能力详情（schema/permission/stability）
listCapabilities()允许允许列出所有已注册能力
关键设计：为什么 handle 是最危险的？
emit 是"往外说"、subscribe 是"听别人说"、request 是"问别人"。但 handle 是"替别人做事"——注册一个处理器意味着任何组件都可以通过 bus.request 触发插件的代码。这相当于在系统中开放了一个新的 API 端点，所以只有 full-access 插件才能注册 handler。

二、Agent 调用全链路：从工具列表到结果返回

当 Agent 决定调用某个插件工具时，完整链路如下：

1获取工具列表：Agent 调用 PluginManager.getAllTools()，返回所有活跃插件（未被 shadow）的工具列表。默认过滤被 shadow 插件的工具。
core/plugin-manager.ts:942-948
2选择工具并传参：Agent 选择工具并传入参数 → executePluginTool(tool, { input, toolCallId })。
core/plugin-manager.ts:933-938
3执行包装层展开：工具的执行函数是包装过的（见第 3 课），在这一步展开：
  • activatePlugin(pluginId, { event: "onToolCall:toolName" }) — 懒激活
  • normalizeToolRuntimeContext() — 解析 signal/piCtx 双参数签名
  • normalizeToolSessionRef() — 提取 session 信息
  • { ...ctx, ...runtimeCtx, ...sessionCtx, ...helperCtx } — 合并上下文
  • origExecute(params, mergedCtx) — 调用原始函数
4结果标准化：normalizePluginToolResult(raw, pluginId) 将字符串、对象等转为统一格式 { content: [{ type: "text", text }] }。
core/plugin-manager.ts:948
5Agent 消费结果：Agent 读取标准化后的 content 数组，继续推理或展示给用户。
async executePluginTool(tool, { toolCallId, input = {}, runtimeCtx = {} } = {}) {
  if (!tool || typeof tool.execute !== "function") {
    throw new Error("plugin tool is not executable");
  }
  const callId = toolCallId || `plugin-tool-${Date.now()}`;
  const raw = await tool.execute(callId, input, runtimeCtx);
  return normalizePluginToolResult(raw, tool._pluginId);
}

三、工具可见性过滤：shadow 机制

getAllTools() 默认不返回被 shadow 插件的工具：

getAllTools(options: any = {}) {
  this.reconcileMissingPluginDirectories();
  const includeShadowed = options.includeShadowed === true;
  return this._tools.filter((tool) => (
    includeShadowed || !tool._pluginKey ||
    this._isPluginKeyRuntimeActive(tool._pluginKey) // 被 shadow 的插件工具不可见
  ));
}

优先级规则：dev (0) > community (1) > builtin (2)。同 id 插件，低优先级者被 shadow，其贡献点默认不可见。只有传入 includeShadowed: true 才返回全部（用于诊断场景）。

四、开发循环：PluginDevService 的独立路径

与生产安装路径不同，dev 循环通过独立的 PluginDevService 管理，不写入用户偏好、不污染社区插件目录：

// Dev 循环的 6 个核心操作
installFromSource({ sourcePath })   // 校验 → 复制到 plugins-dev/ → installPlugin(source:"dev")
reloadPlugin(pluginId, { devRunId }) // 从记忆的 source slot 重新复制源码 → 重装
enablePlugin() / disablePlugin()    // persist: false，不写入用户偏好
invokeTool({ pluginId, toolName })  // 直接执行工具，不经过 Agent（用于 smoke test）
runScenario({ pluginId, id })       // 执行 manifest.dev.scenarios 中的声明式测试
uninstallPlugin(pluginId)           // 删除 plugins-dev/ 目录并移除注册

五、路由代理：外部请求如何到达插件

外部 HTTP 请求通过 proxyToPlugin()（server/routes/plugins.ts:66-95）转发到插件子应用。代理时剥离安全凭证（iframe ticket/surface session query params），通过 Hono env 的 pluginRouteRequest 传入请求级身份上下文：

// 插件路由的错误隔离：区分 bus 能力拒绝（403）和内部错误（500）
app.onError((err, c) => {
  if (isPluginBusCapabilityError(err)) {
    return c.json({ error: err.code, detail: err.message,
      capability: err.capability, permission: err.permission,
      pluginId: err.pluginId, declared: err.declared, granted: err.granted }, 403);
  }
  return c.json({ error: "Plugin internal error", plugin: entry.id }, 500);
});

// 请求级中间件：注入 pluginCtx + agentId + pluginRequestContext
app.use("*", async (c, next) => {
  c.set("pluginCtx", ctx);
  c.set("agentId", c.req.header("X-Hana-Agent-Id") || null);
  c.set("pluginRequestContext", createPluginRouteRequestContext({ ... }));
  await next();
});
注意：PluginBusCapabilityError 和内部错误的区分。
能力拒绝返回 403 + 完整的拒绝上下文（code / capability / permission / declared / granted），帮助插件开发者诊断权限问题。内部错误返回 500 且不泄露内部细节，仅包含 plugin id。

六、检验一下（提取练习）

Q1. PluginBusProxy 中，restricted 插件不能使用哪种通信模式？

A. emit — 完全禁止发布事件 B. handle — 注册请求处理器。emit 可以用但不能发 llm_usage；subscribe 可以用但 llm_usage 被静默过滤 C. request — 完全禁止请求-响应 D. listCapabilities — 完全禁止查询能力
看答案B。restricted 插件可以使用 emit（不能发 llm_usage）、subscribe（llm_usage 静默过滤）、request（受 capability 声明限制）、getCapability、listCapabilities。唯独 handle（注册请求处理器）被完全禁止——这是最危险的权限，相当于在系统中开放新的 API 端点。

Q2. executePluginTool() 中，工具执行包装层合并了哪些上下文？

A. 只合并了 pluginId 和 toolName B. runtimeCtx（signal/session/agent）+ sessionCtx（sessionPath/sessionId）+ helperCtx（registerSessionFile/stageFile）+ 原始 ctx C. 只合并了 session 信息 D. 不需要合并上下文，工具直接使用全局状态
看答案B。工具执行时合并了四层上下文：原始 ctx（插件自己的 context）、runtimeCtx（包含 signal、session、agent 引用）、sessionCtx（sessionPath、sessionId）、helperCtx（registerSessionFile、stageFile 等辅助函数）。这确保工具能访问当前会话的完整上下文。

Q3. dev 插件的 full-access 权限与社区插件有什么不同？

A. 完全相同，没有任何区别 B. dev 插件需要显式传 allowFullAccess: true，不受全局社区插件开关影响——开发者选择和应用用户偏好是独立的两个维度 C. dev 插件永远没有 full-access 权限 D. dev 插件自动获得 full-access 权限
看答案B。在 _isFullAccessAllowed() 中，source === "dev" 走独立分支：如果 options.allowFullAccess === true 则返回 true，否则返回 false。这不受全局 getAllowFullAccessPlugins() 偏好影响——开发者在自己的开发环境中可以独立控制权限。

Q4. getAllTools() 默认不返回被 shadow 插件的工具，shadow 优先级是什么？

A. builtin > community > dev B. community > dev > builtin C. dev (0) > community (1) > builtin (2) — 数字越小优先级越高 D. 随机选择
看答案C。优先级由数字表示：dev=0（最高）、community=1、builtin=2（最低）。当同一 id 的插件出现在多个 source 中时，高优先级者生效，低优先级者的贡献点被过滤。只有传入 includeShadowed: true 才返回全部工具（用于诊断）。
