本节目标：学完你能画出 PluginContext 的 7 大组件及其权限约束，能解释工具注册的命名空间前缀设计和懒激活机制，能区分命令的 Slash 路径和 Palette 路径的权限差异。

上一课我们了解了插件如何被发现和判定权限。这一课进入插件获得运行环境的核心阶段——createPluginContext() 构建隔离沙箱，以及贡献点如何被扫描、包装、注册到全局注册表。

一、createPluginContext()：7 大隔离能力组件

这是插件获取运行环境的唯一入口。每个插件获得一个完全隔离的上下文对象，不能直接访问宿主进程的全局状态。

export function createPluginContext({
  pluginId, pluginKey, source, pluginDir, dataDir, bus, accessLevel,
  permissions, capabilities, sensitiveCapabilities, network,
  registerSessionFile, emitResourceChanged, resourceIO, resourceWatch,
  configSchema, logSink, runtimeContext
}) {
  // 1. 配置存储：JSON Schema 驱动的 key-value，三级作用域
  const config = createPluginConfigStore({ dataDir, schema: configSchema });

  // 2. 权限与能力声明归一化
  const grantedPermissions = normalizePermissions(permissions);
  const declaredCapabilities = normalizeCapabilityList(capabilities);

  // 3. 网络能力（受 capabilities 和 manifest.network 限制）
  const pluginNetwork = createPluginNetwork({ pluginId, network,
    capabilities: declaredCapabilities,
    sensitiveCapabilities: declaredSensitiveCapabilities });

  // 4. 资源 IO（文件系统操作，按 capability 鉴权）
  const pluginResources = createPluginResources({ pluginId, resourceIO,
    resourceWatch, capabilities, sensitiveCapabilities, runtimeScope });

  // 5. 结构化日志（带 [plugin:pluginId] 前缀）
  const prefix = `[plugin:${pluginId}]`;
  const log = {
    info: (...args) => { recordLog("info", args); console.log(prefix, ...args); },
    warn: (...args) => { recordLog("warn", args); console.warn(prefix, ...args); },
    error: (...args) => { recordLog("error", args); console.error(prefix, ...args); },
  };

  // 6. EventBus 代理（受权限约束）— 详见第 4 课
  const pluginBus = createPluginBusProxy(bus, { ownerContext, grantedPermissions,
    allowHandle: accessLevel === "full-access" });

  // 7. 只读身份描述（嵌入 bus 事件中用于审计）
  const ownerContext = Object.freeze({ kind: "plugin", pluginId, pluginKey,
    source, pluginDir, dataDir });

  return Object.freeze({ config, bus: pluginBus, network: pluginNetwork,
    resources: pluginResources, log, appEvents, ownerContext, ... });
}
组件权限约束说明
config按 JSON Schema 校验；sensitive 字段自动脱敏key-value 配置，支持 global / per-agent / per-session 三级作用域
busrestricted: 不能 emit llm_usage、不能 handleEventBus 代理，所有操作经过权限检查转发到全局 bus
network需 network.fetch capability；受 allowedDomains/methods/allowLocalhost 限制包装的 fetch，限制域名、方法、响应大小（默认 5MB），默认超时 15 秒
resources按操作类型：resource.read/write/search/materialize/watch文件系统操作（stat/read/write/delete/copy/move/list/search/watch）
log无限制带 [plugin:pluginId] 前缀，同时发送到诊断面板
appEvents无限制向宿主发送应用级事件，自动标注 source: "plugin:pluginId"
ownerContext只读（Object.freeze）插件身份描述，嵌入 bus 事件中用于审计追踪
关键洞察：config 的三级作用域
global（所有 Agent 共享）→ per-agent（特定 Agent 覆盖）→ per-session（单次会话覆盖）。读取时按 session > agent > global 优先级合并。sensitive 字段（如 API key）读取时自动脱敏为 "***"，仅在 config.get("key", { reveal: true }) 时返回明文。

二、工具注册：命名空间前缀 + 执行包装 + 懒激活

_loadTools() 扫描 tools/*.ts 文件，每个文件必须导出 { name, description, execute }。

async _loadTools(entry) {
  const toolsDir = path.join(entry.pluginDir, "tools");
  if (!fs.existsSync(toolsDir)) return;
  const files = fs.readdirSync(toolsDir).filter(isPluginSourceFile);
  for (const file of files) {
    const mod = await freshImport(path.join(toolsDir, file));
    if (!mod.name || !mod.description || typeof mod.execute !== "function") continue;

    const origExecute = mod.execute;
    this._tools.push({
      name: `${entry.id}_${mod.name}`,        // 命名空间前缀：pluginId_toolName
      description: mod.description,
      parameters: mod.parameters ?? {},
      execute: async (_toolCallId, params, signalOrRuntimeCtx, _onUpdate, piCtx) => {
        // ① 懒激活：首次工具调用时触发 activatePlugin
        await this.activatePlugin(entry.id,
          { event: `onToolCall:${mod.name}`, toolName: mod.name });

        // ② 上下文合并：runtime + session + helper
        const { ctx: runtimeCtx } = normalizeToolRuntimeContext(signalOrRuntimeCtx, piCtx);
        const sessionCtx = normalizeToolSessionRef(runtimeCtx, fallbackSessionPath);
        const mergedCtx = { ...entry.ctx, ...runtimeCtx, ...sessionCtx, ...helperCtx };

        // ③ 执行原始工具函数
        const raw = await origExecute(params, mergedCtx);
        return normalizePluginToolResult(raw, entry.id);
      },
      _pluginId: entry.id, _pluginKey: entry.pluginKey,
    });
  }
}
1命名空间前缀：工具名称格式为 {pluginId}_{toolName}，避免跨插件名称冲突。Agent 调用时 getPluginTool() 可同时支持带前缀和不带前缀的名称。
core/plugin-manager.ts:822-867
2懒激活：每次执行工具前自动调用 activatePlugin() 触发激活事件匹配。如果插件 activationEvents 不含 onStartup，首次工具调用时才激活。
激活事件：onToolCall / onBusRequest / onPageOpen / onWidgetOpen
3上下文合并：将 runtimeCtx（signal/session/agent）、sessionCtx（sessionPath/sessionId）、helperCtx（registerSessionFile/stageFile）合并到插件 ctx。
normalizeToolRuntimeContext() + normalizeToolSessionRef()
4结果标准化：normalizePluginToolResult() 将字符串、{content}、或其他格式统一转为 { content: [{ type: "text", text }] }。

三、命令注册：Slash vs Palette 两条路径

命令有两条注册路径，权限要求不同：

路径权限要求注册到说明
Slash（handler）仅 full-accessslashRegistry.registerCommand()默认 permission: "owner"（最严）
Palette（execute）所有插件可用this._commands[]向后兼容的非 slash 命令
// Slash 注册的 full-access 闸门
if (hasHandler) {
  if (entry.accessLevel !== "full-access") {
    log.warn(`"${entry.id}/${file}" declares slash handler but plugin is restricted; skipped.`);
    continue;
  }
  this._slashRegistry.registerCommand({ name: mod.name, handler: mod.handler, ... },
    { source: "plugin", sourceId: entry.pluginKey });
}

四、生命周期激活：_activatePluginEntry() 的 6 步流程

1入口解析：resolvePluginEntry() 按优先级查找 index.ts（首选）→ index.js。无入口则 hasLifecycle = false，跳过激活。
core/plugin-manager.ts:747-801
2动态导入：freshImport() 使用时间戳参数（?t=Date.now()）绕过 Node.js 模块缓存，确保热重载时获取最新代码。
3实例化：取 mod.default 作为插件类，new PluginClass() 创建实例。
4注入上下文与工具：将 entry.ctx 赋值给 instance.ctx；注入 instance.register()（注册清理函数）和 ctx.registerTool()（动态注册工具）。
5调用 onload：带超时保护（默认 15 秒）执行 instance.onload()。负责初始化——连接外部服务、预热缓存等。
6状态更新：成功 → activationState = "activated"；失败 → "failed" 并记录 activationError。注意 status 可能仍为 "loaded"（静态贡献已加载，仅生命周期失败）。
重要：status 和 activationState 是两个独立维度。
status: "loaded" && activationState: "failed" 表示插件静态贡献（tools/skills 等）加载成功，但 onload 失败。这种 dual-status 编码避免了使用额外字段——工具仍然可用，只是插件没有完成初始化。

五、检验一下（提取练习）

Q1. createPluginContext() 创建的 7 大组件中，哪个组件操作前需要校验 capability 声明？

A. config（配置存储）— 所有读写都需要 capability B. network 和 resources — 分别需要 network.fetch 和 resource.read/write 等 capability 声明 C. log（日志）— 写日志需要 logging.write capability D. appEvents — 发送事件需要 event.emit capability
看答案B。network 需要 network.fetch capability，且受 manifest.network 中 allowedDomains/methods/allowLocalhost 的三重限制。resources 按操作类型需要 resource.read/write/search/materialize/watch 等 capability。config、log、appEvents、ownerContext 无限制或只读。

Q2. 工具命名空间前缀 {pluginId}_{toolName} 的设计目的是什么？

A. 为了让工具名更长更好看 B. 避免跨插件名称冲突——两个插件都可以定义名为 "hello" 的工具，运行时通过前缀区分 C. 为了性能优化——前缀让查找更快 D. 为了让用户知道工具来自哪个插件
看答案B。命名空间前缀是插件系统避免名称冲突的核心机制。Agent 调用时可通过 getPluginTool() 同时支持带前缀和不带前缀的名称——如果当前只有一个工具匹配不带前缀的名称，可以直接使用短名。

Q3. Restricted 插件注册命令时，Slash 路径会发生什么？

A. 正常注册，和 Palette 路径一样 B. 跳过并打印 warn 日志——restricted 插件不能注册 slash 命令，只能注册 palette 命令 C. 抛出异常，阻止插件加载 D. 自动降级为 Palette 命令
看答案B。Slash 注册有显式的 accessLevel !== "full-access" 闸门——如果受限插件尝试声明 slash handler，系统会打印 warn 日志并 continue 跳过该文件。但 Palette 路径的 execute 命令不受影响，两种路径在同一次 _loadCommands() 中独立处理。

Q4. _activatePluginEntry() 中，如果 onload() 抛出异常，工具的可用性如何？

A. 所有工具都不可用 B. 工具仍然可用——静态贡献（tools/skills 等）已在 onload 之前注册完成，只有 activationState 变为 failed C. 只有部分工具可用 D. 插件被完全卸载
看答案B。这是 dual-status 编码的核心价值：工具注册发生在生命周期激活之前，所以即使 onload 失败，插件的静态贡献仍然可用。状态变为 status: "loaded" && activationState: "failed"——用户仍然可以调用工具，只是插件的初始化逻辑未完成。
