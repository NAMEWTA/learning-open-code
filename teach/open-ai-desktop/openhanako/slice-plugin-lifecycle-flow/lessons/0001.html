第 1 课 · 插件系统架构总览：6 插件对比 + 权限模型 + 安装入口 架构全景

本节目标：学完你能一句话讲清 HanaAgent 插件系统的本质，能画出三种插件类型的权限边界，理解 installPluginFromPath() 的 6 步安装编排流程。

HanaAgent 的插件系统是一个三层能力扩展框架，通过声明式 manifest.json + 用户授权式门禁，在"能力交付"和"安全保障"之间取得精确平衡。整个系统涉及约 10 个源文件、3000 行代码，但核心架构可以用一张图概括。

一、三种插件类型：能力分层的设计

Builtin 内置

source: "builtin"
始终加载，不可禁用。位于 plugins/ 目录，随 App 打包分发。拥有所有权限。

Full-Access 社区

trust: "full-access"
需用户全局开关放行。可注册路由/Slash/扩展/供应商/UI 页面。

Restricted 受限

trust 省略或非 "full-access"
只能注册工具/技能/命令(无Slash)/Agent模板/配置。默认安全级别。

关键数字：三种类型共享 12 种贡献点，但权限矩阵精确控制每种类型能注册什么（详见第 2 课）。Restricted 不是错误状态——插件的基础功能仍然可用，确保即使 full-access 被拒绝，插件仍能提供核心价值。

二、三条关键路径

1启动加载路径：PluginManager.scan() 扫描插件目录 -> loadAll() 批量加载 -> 每个插件独立的 _loadPlugin() 阶段

core/plugin-dev-service.ts

四、安装入口详解：installPluginFromPath()

这是 REST API POST /api/plugins/install 的处理器，接受 { path, sessionPath, allowDowngrade } 参数。它负责编排整个安装流程，而 PluginManager 仅负责"加载"核心职责。

1注册源文件：registerSessionFileFromRequest() 将安装源挂载到当前 session，前端可展示安装来源。
server/routes/plugins.ts
2暂存处理 (staging)：stagePluginSource() 解压 .zip 到临时目录（.installing-xxxxx），自动检测单层包裹目录，同时检测 OpenClaw 格式并拒绝。
server/routes/plugins.ts:392-429
3描述符读取：readPluginDescriptorForInstall() 从暂存目录读取 manifest.json 并校验格式。
core/plugin-manager.ts
4版本校验：assertExpectedPlugin() 检查 marketplace 安装时 ID/版本匹配；comparePluginVersions() 防止降级安装。
server/routes/plugins.ts
5备份与原子替换：创建备份 → 删除旧版本 → fs.renameSync() 原子移动（同文件系统内 rename 具有原子性）。
server/routes/plugins.ts
6触发热加载：调用 pm.installPlugin(targetDir, {source: "community"})，失败时从备份恢复。
core/plugin-manager.ts
关键洞察：为什么分两阶段（暂存 + 正式安装）？
stagePluginSource() 先验证、再复制到插件目录内的 .installing- 临时目录，确保整个安装过程的文件操作始终在插件目录内。通过后使用 renameSync 原子替换——避免了"半安装"状态。

五、数据流的五阶段形态转换

用户上传 .zip │ ▼ [Raw binary] ──extractZip()──▶ [临时目录 stagingDir] │ ▼ [文件系统目录] ──_readPluginDescriptor()──▶ [PluginDescriptor] │ 字段: id, name, version, trust, contributions[], capabilities, ... │ ▼ [PluginDescriptor] ──_entryFromDescriptor()──▶ [PluginEntry] │ 新增: source, pluginKey, status, activationState, accessLevel, ctx │ ▼ [PluginEntry] ──createPluginContext()──▶ [HanaPluginContext] │ 新增: bus(proxy), network, resources, config, log │ ▼ Agent 运行时可见：tools[], skills[], commands[], routes[], providers[]

六、检验一下（提取练习）

Q1. HanaAgent 插件系统有哪三种插件类型？

A. 本地插件、远程插件、混合插件 B. Builtin 内置、Full-Access 社区、Restricted 受限 C. 同步插件、异步插件、流式插件 D. 系统插件、用户插件、开发者插件
看答案B。三种类型按权限等级划分：Builtin（始终全权）、Full-Access 社区（需用户授权）、Restricted（默认安全，只能注册基础贡献点）。

Q2. installPluginFromPath() 的 6 个步骤中，哪一步保证安装的原子性？

A. registerSessionFileFromRequest() — 记录安装源文件 B. stagePluginSource() — 解压到临时目录 .installing-xxxxx C. fs.renameSync() — 同文件系统内 rename 具有原子性，避免半安装状态 D. pm.installPlugin() — PluginManager 内部队列保证
看答案C。fs.renameSync() 在同文件系统内是原子操作——要么完全成功，要么完全不变。这确保了不会出现"插件目录存在但文件不完整"的中间状态。

Q3. Restricted 插件不能注册哪些贡献点？

A. 工具（tools）和技能（skills） B. 路由（routes）、扩展（extensions）、供应商（providers）、UI页面（page/widget）和 Slash 命令 C. 配置（configuration） D. Agent 模板（agents）
看答案B。Restricted 插件可以注册 tools、skills、commands（palette）、agents、configuration。但不能注册 routes、extensions、providers、page、widget、slash commands。这是安全模型的核心设计。

Q4. 整个安装流程中，PluginManager 的职责范围是什么？

A. 负责整个安装流程的编排，包括解压、校验、备份、替换 B. 只负责"加载"核心职责——HTTP 层的 installPluginFromPath() 负责编排，PluginManager 负责加载 C. 只负责管理插件的前端展示 D. 负责与 Snail AI Server 通信
看答案B。这是职责分离的体现：HTTP 层负责文件操作（解压、校验、备份、原子替换）和请求编排；PluginManager 负责核心加载逻辑（权限判定、上下文注入、贡献点注册、生命周期激活）。
推荐一手资料：打开 server/routes/plugins.ts 第 481-563 行，亲手追踪 installPluginFromPath() 的完整调用链。然后打开 core/plugin-manager.ts 的 scan() 和 loadAll() 方法，对比启动加载与运行时安装的差异。
