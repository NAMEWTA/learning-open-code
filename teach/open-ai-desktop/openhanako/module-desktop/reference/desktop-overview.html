1. 主进程 + 渲染进程架构

HanaAgent 桌面端采用标准 Electron 多进程架构，主进程负责窗口管理、系统集成和 Server 子进程生命周期；渲染进程运行 React 19 应用，通过 HTTP/WebSocket 与 Server 通信，IPC 仅用于窗口控制和系统 API。

关键设计决策：渲染进程不直连 Server IPC，所有业务通信走 HTTP/WebSocket。preload.cjs 通过 contextBridge 暴露 window.hana API，仅包含窗口管理、文件对话框、系统通知等 Electron 特定能力。Server 运行在独立 Node.js 进程中（spawn() 启动，detached: true），桌面端退出时可选择保留 Server 继续运行。

2. Electron 主进程详解

2.1 启动链路

bootstrap.cjs
  │
  ├─ 1. 诊断目录创建 (HANA_HOME/diagnostics/desktop-launch)
  ├─ 2. 全局错误捕获 (uncaughtException / unhandledRejection)
  ├─ 3. 解析 HANA_HOME → resolveHanakoHome()
  ├─ 4. 写入启动标记 (launch-marker.json)
  ├─ 5. Windows 安装完整性检查 (仅 packaged)
  ├─ 6. Office PDF Helper 检测 (--hana-office-html-to-pdf)
  └─ 7. loadDesktopMain() → require('./main.cjs')

main.cjs
  │
  ├─ 0. 预检 preload.bundle.cjs 存在性（缺失则硬崩）
  ├─ 1. 解析登录 Shell PATH（macOS/Linux）
  ├─ 2. 单实例锁 + GPU 启动策略
  ├─ 3. 创建 Splash 窗口（透明无框，380x280）
  ├─ 4. startServer() → spawn hana-server 子进程
  │     ├─ 检测已有 server (server-info.json)
  │     ├─ 验证可复用性 (版本/studioId/网络模式)
  │     ├─ spawn server 子进程
  │     └─ 轮询 server-info.json 等待就绪
  ├─ 5. 创建系统托盘 (Tray)
  ├─ 6. 注册 Quick Chat 全局快捷键
  ├─ 7. 根据 setupComplete 状态决定：
  │     ├─ 未完成 → createOnboardingWindow()
  │     └─ 已完成 → createMainWindow()
  ├─ 8. 监听 app-ready IPC → 关闭 splash，显示主窗口
  └─ 9. 挂载 Server 崩溃监控 (monitorServer)

2.3 Server 生命周期管理

主进程用 spawn() 启动 Server 作为独立子进程（detached: true），关键流程：

  1. 复用检测：读取 server-info.json，检测已有 Server PID 是否存活；存活则通过 /api/health 和 /api/server/identity 验证版本 + studioId 匹配
  2. 启动：打包模式使用 resources/server/hana-server（Windows 加 .exe），开发模式使用 node server/bootstrap.ts
  3. 等待就绪：轮询 server-info.json（间隔 200ms，超时约 16 秒），获取 port 和 token
  4. 崩溃监控：monitorServer() 监听 exit 事件，自动重启一次，二次失败后写 crash.log 并弹窗
  5. 优雅关闭：退出时先发 /api/shutdown（gracefulShutdown 内部 15s force timer），等待进程退出或 5 秒后 SIGKILL

3.1 Quick Chat 浮窗机制

Quick Chat 是桌面端特有的浮动聊天窗口，特性：

  • 两种模式：compact（固定 480x142，不可调大小）和 chat（可调大小，最小 360x118）
  • 全局快捷键：通过 globalShortcut.register 注册，默认可从 preferences.json 读取，支持运行时动态更换
  • 位置记忆：compact 和 chat 模式分别记忆窗口位置和尺寸（quick-chat-window-state.json）
  • macOS 特殊处理：关闭 Quick Chat 时临时禁用主窗口 focusable（300ms），防止焦点回弹抢夺

3.2 Viewer 派生窗口

Viewer 窗口是主面板 tab 的"弹出查看"模式，设计理念：

  • 只读副本：读取启动时传入的本地文件内容，不与主面板通信
  • Live reload：通过 retainViewerLocalFileResourceWatch 监听文件变更并自动刷新
  • 支持的格式：markdown、code（带语法高亮）、CSV
  • 生命周期：主进程 spawn BrowserWindow → IPC viewer-load 送文件元信息 → readFile → 渲染 PreviewEditor(readOnly)

8.3 翻译加载机制

渲染进程：通过 use-i18n.ts hook 加载对应 locale JSON 文件，t(key, vars) 函数查找翻译并替换 {varName} 占位符。locale 选择优先级：preferences.json 的 locale → 环境变量 → 浏览器语言检测 → zh 回退。

主进程：独立的 mt(dotPath, vars, fallback) 函数，从 locale JSON 的 main 命名空间加载。主要用于托盘菜单、系统对话框等 Electron 原生 UI 文案。locale 解析规则：zh-TW/zh-Hant → zh-TW，zh* → zh，ja* → ja，ko* → ko，其余 → en。

9.1 HanaComputerUseHelper（macOS · Swift）

语言：Swift
平台：macOS
包管理：Swift Package Manager

native/HanaComputerUseHelper/ 是基于 CuaDriverCore 的 macOS Computer Use 守护进程。通过 Unix Domain Socket 提供服务，Desktop 主进程通过 spawn 调用此 helper 执行 macOS 辅助功能操作。

├── Package.swift                    # Swift 包定义（依赖 CuaDriverCore）

核心能力：

  • 守护进程模式（serve）：启动 DaemonServer，监听 Unix Domain Socket（默认 ~/Library/Caches/hana-computer-use/hana-computer-use-helper.sock）
  • 工具调用：通过 ToolRegistry 执行工具（如 list_apps、get_window_state），支持 JSON 参数

9.2 HanaWindowsSandboxHelper（Windows · C++）

语言：C++ (Win32 API)
平台：Windows
编译：MSVC

native/HanaWindowsSandboxHelper/main.cpp（约 1500 行）是一个 Windows 原生沙箱程序，使用 Restricted Token + 自定义 Sandbox Desktop 实现进程级隔离，替代旧的 AppContainer 方案。

核心机制：

  1. Writable Root ACL：为每个可写根目录生成唯一的 SID，在目录上添加 ALLOW ACE，在禁止写入的子目录上添加 DENY ACE
  2. Restricted Token：通过 CreateRestrictedToken 创建受限令牌，包含 Everyone SID + 当前登录 SID + 各 Writable Root SID + WRITE_RESTRICTED_CODE_SID，禁用所有特权
  3. Sandbox Desktop：使用 CreateDesktopW 创建独立桌面，仅允许受限令牌访问，防止 Shatter Attack
  4. Kill-on-Close Job：将子进程加入 Job Object（JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE），确保父进程退出时子进程被系统终止
  5. ACL Restore：进程退出时恢复被修改的 ACL

CLI 参数：

参数说明
--cwd <path>工作目录（必填）
--writable-root <path>可写根目录（可多个，至少一个）
--writable-root-optional <path>可选可写根目录
--deny-write <path>禁止写入的子目录
--diagnose-token诊断模式：输出 SID 信息但不执行
--cleanup-hana-write-acl <path>清理遗留的 hana-write ACL
--cleanup-legacy-acl清理旧版 AppContainer ACL
--cleanup-legacy-profile <name>删除旧版 AppContainer Profile
-- <exe> [args...]被沙箱化的目标可执行文件及参数

11.3 关键设计决策

  1. Server 独立进程：Server 不是 Electron 的 fork，而是独立 spawn() 进程（detached: true），桌面端退出时默认保留 Server。这允许多个桌面端实例共享同一个 Server，也允许 Server 在没有 GUI 的环境下独立运行。
  2. IPC 最小化：渲染进程与 Server 的业务通信全部走 HTTP/WebSocket，Electron IPC 仅用于窗口管理、文件对话框、系统通知等 OS 级能力。这使得渲染进程代码与 Electron 解耦，移动端 PWA 可复用相同的业务逻辑。
  3. 多窗口隔离：设置窗口、浏览器查看器、Viewer 窗口各自拥有独立的渲染进程，崩溃不会影响主窗口。通过 IPC settings-changed 实现跨窗口状态同步。
  4. 渐进式增强：mobile-main.tsx 支持 PWA 模式，注册 Service Worker 实现离线缓存和更新通知，使同一套 React 代码可在桌面端和移动端浏览器中运行。
  5. 安全沙箱：Windows 上使用 Restricted Token + Sandbox Desktop 替代旧版 AppContainer，macOS 上使用专用 Helper 进程 + Unix Socket 管理 Computer Use 权限。
