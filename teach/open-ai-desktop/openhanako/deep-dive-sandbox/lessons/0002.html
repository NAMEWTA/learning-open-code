第 2 课 · PathGuard 深度拆解：11 级级联判定链 核心算法

🎯 本节目标：学完你能逐级画出一条路径在 PathGuard 11 级判定链中的全部匹配过程，理解 BLOCKED / READ_ONLY / READ_WRITE / FULL 四种访问级别的判定逻辑，掌握符号链接追踪和存在性检查的安全机制。

源码位置：lib/sandbox/path-guard.ts — getAccessLevel() 方法（107-183 行）和 check() 方法（185-207 行）。PathGuard 是整个沙盒系统的第一道防线，使用纯 JavaScript 实现，跨平台行为完全一致。

一、四种访问级别与操作需求映射

PathGuard 的输出不是简单的"允许/拒绝"，而是四种访问级别：

BLOCKED    — 完全禁止访问（敏感认证文件、浏览器数据）
READ_ONLY  — 只读（Agent 配置、用户目录、系统文件）
READ_WRITE — 可读写但不能删除（Agent 工作目录、日志、上传）
FULL       — 完全访问（workspace 根目录，可读可写可删）

每种操作需要的最低级别通过 OP_REQUIREMENTS 映射表定义：

// path-guard.ts — check() 方法中的操作要求映射
const OP_REQUIREMENTS = {
  read:   AccessLevel.READ_ONLY,   // 至少 READ_ONLY 即可读
  write:  AccessLevel.READ_WRITE,  // 至少 READ_WRITE 才可写
  delete: AccessLevel.FULL,        // 只有 FULL 才可删
};

// check(path, operation) 的逻辑：
//   level = getAccessLevel(path)
//   required = OP_REQUIREMENTS[operation]
//   return level >= required  // 级别越高权限越大
💡 为什么删除操作要求 FULL 而非 READ_WRITE？
删除是不可逆操作。Agent 在 workspace 内可以随意创建和修改文件，但只有 workspace 根目录下才能删除文件（FULL 级别）。这防止了 Agent 意外删除 agentDir/memory/ 中的重要数据——memory 目录是 READ_WRITE，可以写入但不能删除文件。

二、11 级级联判定链（逐级拆解）

判定链按优先级从高到低排列，一旦匹配即返回，不继续后续判定。这是"级联"的核心含义。

级别 1-2：BLOCKED 匹配（最高优先级）

1级别 1 — hanakoHome 根级禁止文件
if (resolved === hanakoHome/auth.json) → BLOCKED
if (resolved === hanakoHome/models.json) → BLOCKED
精确匹配 BLOCKED_FILES 列表。这些是认证凭据和模型密钥，必须在任何规则之前拦截。
policy.ts → BLOCKED_FILES 常量
2级别 2 — hanakoHome 下禁止目录
if (resolved starts with hanakoHome/browser-data/) → BLOCKED
if (resolved starts with hanakoHome/playwright-browsers/) → BLOCKED
前缀匹配 BLOCKED_DIRS 列表。浏览器数据和自动化浏览器缓存包含 cookie、localStorage 等敏感信息。
policy.ts → BLOCKED_DIRS 常量

级别 3-5：READ_ONLY 匹配

3级别 3 — Agent 配置文件（精确匹配）
if (resolved === agentDir/ishiki.md)    → READ_ONLY
if (resolved === agentDir/config.yaml)  → READ_ONLY
if (resolved === agentDir/identity.md)  → READ_ONLY
精确匹配 READ_ONLY_AGENT_FILES。Agent 可以读取这些配置文件来了解自身设置，但不能修改它们。修改必须通过 settings API。
policy.ts → READ_ONLY_AGENT_FILES 常量
4级别 4 — Agent 只读目录（扩展预留）
// 当前为空数组，保留扩展点
// 未来可添加如 agentDir/capabilities/ 等只读目录
通用前缀匹配机制，遍历 READ_ONLY_AGENT_DIRS 数组。当前无实际匹配项。
policy.ts → READ_ONLY_AGENT_DIRS（空数组）
5级别 5 — 全局只读目录
if (resolved starts with hanakoHome/skills/)        → READ_ONLY
if (resolved starts with hanakoHome/session-files/)  → READ_ONLY
if (resolved starts with hanakoHome/user/)           → READ_ONLY
前缀匹配 READ_ONLY_HOME_DIRS。Skills 定义、会话文件、用户配置等属于全局共享资源，Agent 只读。
policy.ts → READ_ONLY_HOME_DIRS 常量

级别 6-7：READ_WRITE 匹配

6级别 6 — Agent 工作目录 + 可写文件
// 6a: 工作目录（前缀匹配）
if (resolved starts with agentDir/memory/)    → READ_WRITE
if (resolved starts with agentDir/sessions/)  → READ_WRITE
if (resolved starts with agentDir/desk/)      → READ_WRITE
if (resolved starts with agentDir/heartbeat/) → READ_WRITE
if (resolved starts with agentDir/book/)      → READ_WRITE

// 6b: 可写文件（精确匹配）
if (resolved === agentDir/pinned.md) → READ_WRITE
Agent 运行时产生的数据（记忆、会话、桌面文件、心跳、书签）可读写但不可删除。pinned.md 是唯一可写的 Agent 配置文件。
policy.ts → READ_WRITE_AGENT_DIRS + READ_WRITE_AGENT_FILES
7级别 7 — 全局可写目录
if (resolved starts with hanakoHome/channels/)   → READ_WRITE
if (resolved starts with hanakoHome/logs/)       → READ_WRITE
if (resolved starts with hanakoHome/uploads/)    → READ_WRITE
if (resolved starts with hanakoHome/.ephemeral/) → READ_WRITE
前缀匹配 READ_WRITE_HOME_DIRS。日志、上传文件、临时文件属于全局可写。.ephemeral 是沙盒临时目录。
policy.ts → READ_WRITE_HOME_DIRS 常量

级别 8-11：兜底与外部路径

8级别 8 — hanakoHome 未匹配兜底
if (resolved starts with hanakoHome) {
  if (allowExternalReads) → READ_ONLY
  else → BLOCKED
}
hanakoHome 内但未被前面规则覆盖的路径。按 read-all 契约默认只读，但如果 allowExternalReads=false 则全部 BLOCKED。
path-guard.ts — getAccessLevel() 末尾逻辑
9级别 9 — Workspace Roots → FULL
if (resolved starts with any workspaceRoot) → FULL
用户项目目录是 Agent 的主要工作区，应拥有完全的读写删权限。workspaceRoots 来自 deriveSandboxPolicy()，包含 workspace + workspaceFolders + authorizedFolders + cwd。
path-guard.ts — workspaceRoots 遍历匹配
10级别 10 — 运行时可写路径 → READ_WRITE
if (resolved starts with any policyWritablePath) → READ_WRITE
运行时缓存、沙盒临时目录等非 workspace 但需要写入的路径，由 runtimeWritablePaths 参数注入。
policy.ts → deriveSandboxPolicy() 构建
11级别 11 — 外部系统路径最终兜底
if (allowExternalReads) → READ_ONLY
else → BLOCKED
对于不在 hanakoHome、不在 workspace 的普通系统路径（如 /usr/bin、/etc/hosts），按 read-all 契约只允许读取。
path-guard.ts — 最终 return 语句

三、符号链接安全：_resolveReal() 先行解析

🛡关键安全机制：所有判定之前先解析真实路径
// path-guard.ts — _resolveReal() 方法（72-95 行）
_resolveReal(inputPath: string): string {
  try {
    return fs.realpathSync(inputPath);  // 跟随符号链接
  } catch {
    // 路径不存在时，递归向上查找最近存在的祖先
    let current = inputPath;
    while (!fs.existsSync(current) && current !== path.dirname(current)) {
      current = path.dirname(current);
    }
    const existing = fs.realpathSync(current);
    const remaining = path.relative(current, inputPath);
    return path.join(existing, remaining);
  }
}

这防止了通过符号链接绕过检查的攻击：/tmp/link → ~/.ssh 会在判定前被解析为真实路径 /home/user/.ssh，然后被 BLOCKED 级别拦截。
path-guard.ts — getAccessLevel() 第一行即调用 _resolveReal()

⚠ 警惕 TOCTOU 窗口：PathGuard 先调用 _resolveReal() 做判定，然后 OS 沙盒才执行命令。在判定通过后、命令执行前的窗口内，攻击者理论上可以替换符号链接。但 OS 沙盒通过 mount 绑定（bwrap）或 ACL（Windows）进一步限制了实际文件访问范围，两层防线互补削弱了这个窗口。

四、Managed Config Guard — 额外防线

managed-config-guard.ts（57 行）在 PathGuard 之外提供额外保护：

// managed-config-guard.ts — 拦截直接编辑 config.yaml
if (path matches agents/*/config.yaml) {
  throw new Error(
    "managed config files must be changed through settings APIs"
  );
}

即使 PathGuard 判定该路径为 READ_WRITE，Managed Config Guard 也会单独拦截，强制 Agent 通过 settings API 修改配置。这防止了 Agent 通过文件编辑工具绕过配置管理直接篡改自身安全设置。

五、检验一下（提取练习）

Q1. 一条路径 /home/user/workspace/project/src/main.ts 在 PathGuard 中会被判定为什么级别？

A. BLOCKED — 这是系统路径 B. FULL — 因为在 workspace root 下（级别 9 匹配） C. READ_ONLY — 级别 11 兜底 D. READ_WRITE — 级别 10 运行时可写路径
看答案B。workspace root 路径在级别 9 就会被匹配为 FULL。这是 Agent 最常见的操作场景——在用户项目目录中工作。

Q2. 为什么 BLOCKED 判定放在最前面（级别 1-2）？

A. 因为这些路径最常见，放在前面可以加速判定 B. 因为安全规则必须最先执行——敏感文件的保护不能被后续规则"覆盖" C. 因为代码可读性更好 D. 因为 BLOCKED 判定最复杂，需要先计算
看答案B。级联判定的"一旦匹配即返回"特性要求最高优先级的规则放在最前面。如果 BLOCKED 放在后面，敏感的 auth.json 可能被前面的 READ_ONLY 规则错误地判定为可读。

Q3. _resolveReal() 方法处理不存在路径的策略是？

A. 直接返回原路径不做解析 B. 抛出错误要求路径必须存在 C. 递归向上查找最近存在的祖先目录做 realpath，再把不存在的段拼回去 D. 假设路径都是安全的直接通过
看答案C。这是处理 mkdir -p /tmp/a/b/c 场景的关键设计——中间目录可能尚不存在，但判定仍需准确。

Q4. Managed Config Guard 阻止的是什么行为？

A. Agent 通过 curl 下载恶意配置文件 B. Agent 通过文件编辑工具直接修改 agents/*/config.yaml 绕过配置管理 C. 用户通过 UI 手动修改配置 D. 其他 Agent 进程修改当前 Agent 的配置
看答案B。这是防止 Agent 自我篡改的最后一道防线——即使 PathGuard 允许写入，Managed Config Guard 也会单独拦截 config.yaml 的直接编辑，强制走 settings API。
