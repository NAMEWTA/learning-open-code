第 3 课 · OS 级沙盒实现对比：三平台的隔离艺术 跨平台

🎯 本节目标：学完你能讲清 Linux bwrap、macOS seatbelt、Windows Restricted Token 三种 OS 沙盒的参数构造流程、核心差异和适用场景，理解为什么不同平台选择了不同的安全原语。

OS 级沙盒是三层防线的第二层——在 PathGuard 通过语义判定后，由 OS 原生机制提供真正的进程级隔离。三个平台使用了三种截然不同的安全原语，各自遵循不同的匹配语义和限制。

一、Linux Bubblewrap：命名空间 + bind mount 的轻量容器

源码位置：lib/sandbox/bwrap.ts — buildBwrapArgs() 函数（116-200 行）

Bubblewrap（bwrap）利用 Linux user namespace + bind mount 实现文件系统隔离。不需要 root 权限，不需要 daemon，启动时间约 5-15ms。核心逻辑分 8 个阶段构建参数：

阶段 1：基础隔离

--ro-bind / /          # 将整个根文件系统只读挂载（read-all 模式）
--dev /dev             # 挂载设备文件系统
--proc /proc           # 挂载进程文件系统（PID 命名空间）
--tmpfs /tmp           # /tmp 用独立的 tmpfs
--unshare-pid          # 创建新的 PID 命名空间
[--unshare-net]        # 可选：创建新的网络命名空间（断网模式）
--new-session          # 创建新的会话
--die-with-parent      # 父进程退出时自动清理

阶段 2：私有运行时环境

--dir /tmp/hana-home       # 私有 HOME
--dir /tmp/hana-cache      # 私有 XDG 缓存
--setenv HOME /tmp/hana-home
--setenv XDG_CACHE_HOME /tmp/hana-cache
--setenv npm_config_cache /tmp/hana-npm-cache     # 私有 npm 缓存
--setenv PIP_CACHE_DIR /tmp/hana-pip-cache        # 私有 pip 缓存

阶段 3-5：路径挂载（mount order matters）

// 阶段 3：可写挂载工作目录
--bind {cwd} {cwd}       # 可写挂载工作目录
--chdir {cwd}            # 设置当前工作目录

// 阶段 4：可写路径逐一覆盖挂载
for each path in policy.writablePaths:
  --bind {path} {path}   # 以可写模式覆盖全局只读

// 阶段 5：只读授权路径
for each path in [policy.readablePaths, externalReadPaths]:
  --ro-bind {path} {path}

阶段 6-8：写保护 + 拒绝 + 遮蔽

// 阶段 6：写保护（在可写范围内再限制）
for each path in policy.protectedPaths:
  --ro-bind {path} {path}    # .git 和 session-files 无法被写入

// 阶段 7：读取拒绝
for each path in policy.denyReadPaths:
  目录 → --tmpfs {path}      # 用空 tmpfs 遮蔽整个目录
  文件 → --ro-bind /dev/null {path}  # 绑定到 /dev/null

// 阶段 8：宿主机缓存遮蔽
if HOME/.cache 存在且未被授予写入权限:
  --tmpfs {path}             # 防止回落到真实用户缓存
💡 核心技巧：mount order matters
bwrap 使用后挂载覆盖先挂载语义。先 --ro-bind / / 将整个系统设为只读，再用 --bind 把特定路径覆盖为可写。这避免了显式白名单模式的复杂度——不需要列举所有需要读的系统路径，只需要标注需要写的少数路径。当 readAll=false 时，跳过 root 绑定，改为显式列出 SYSTEM_READONLY_PATHS（/bin, /usr, /lib, /etc/ssl 等），形成真正的白名单模式。
⚠ 注意：Linux 上 bwrap 不可用时的行为
当 bwrap 未安装时，沙盒创建逻辑（index.ts 262-267 行）不会降级到仅 PathGuard——而是返回一个始终输出错误消息的 bash 工具，提示用户安装 bwrap。这意味着 Linux 上要么有完整 OS 沙盒保护，要么命令完全不可执行。这与 macOS 的行为不同。

二、macOS Seatbelt：SBPL 声明式沙盒

源码位置：lib/sandbox/seatbelt.ts（129 行）

Seatbelt 通过 sandbox-exec -f profile.sb 包裹子进程，使用 SBPL (Sandbox Profile Language) 声明式语言定义沙盒规则。

(version 1)
(deny default)              ← 默认拒绝一切

;; 进程权限
(allow process-exec* process-fork signal)
(allow sysctl-read)
(allow mach*)               ← macOS 特有的 Mach 内核通信
(allow ipc-posix*)

;; 全局可读 → 对应 read-all 契约
(allow file-read*)

;; 可写路径 → 白名单
(allow file-write* (subpath "/path/to/workspace"))
(allow file-write* (subpath "/private/tmp"))

;; 写保护 → deny 覆盖 allow（SBPL last-match-wins）
(deny file-write* (subpath "/path/to/workspace/.git"))

;; 读取拒绝 → 敏感数据遮蔽
(deny file-read* (subpath "/path/to/hanakoHome/auth.json"))

;; 终端 + PTY
(allow file-write* (literal "/dev/null"))
(allow pseudo-tty)

;; 网络控制
(allow network-outbound)    ← 允许出站
; 或
(deny network*)             ← 完全封死
!关键差异 vs. Bubblewrap：
• Seatbelt 没有命名空间隔离——所有路径使用真实文件系统，仅通过 SBPL 规则限制
• SBPL 使用 last-match-wins 语义，deny 规则必须放在 allow 规则之后（与 bwrap 的 mount order 类似但方向相反）
• Mach port 权限必须显式授予，否则 macOS 上基础操作（如获取进程列表）会失败
• 没有 --die-with-parent 等价能力，子进程生命周期管理较弱
lib/sandbox/seatbelt.ts
⚠ macOS 15+ 废弃风险：Apple 已将 sandbox-exec 标记为 deprecated。如果 checkAvailability() 检测到 seatbelt 不可用，HanaAgent 降级到仅 PathGuard（无 OS 层沙盒），安全强度大幅降低。可能的替代方案：App Sandbox（需应用签名、无法动态调整策略）、XPC Service（增加架构复杂度）。

三、Windows Restricted Token：NTFS ACL + 令牌降权

源码位置：lib/sandbox/win32-exec.ts（1645 行）+ desktop/native/HanaWindowsSandboxHelper/main.cpp（1509 行）

Windows 沙盒是三平台中最复杂的实现，通过 5 个阶段完成进程隔离：

阶段 1：SID 生成

// win32-policy.ts → buildWin32SandboxGrants()
writePaths = [cwd]
optionalWritePaths = policy.writablePaths
denyWritePaths = policy.protectedPaths

// main.cpp → sidForWritableRoot()
// 每个可写根路径通过确定性哈希生成 S-1-5-21-* SID
// 哈希输入: "hana-win32-write-root-v3:" + fullPathForKey(root)
// 确定性保证同一路径在不同进程中生成相同 SID

阶段 2：NTFS ACL 应用

// main.cpp → applyWriteAcls()
for each writableRoot:
  ensureAce(root.path, root.sid, GRANT_ACCESS, WRITE_ALLOW_MASK)
  // WRITE_ALLOW_MASK = FILE_GENERIC_READ | FILE_GENERIC_WRITE |
  //                    FILE_GENERIC_EXECUTE | DELETE | FILE_DELETE_CHILD

for each denyWritePath:
  ensureAce(denyPath, matched.sid, DENY_ACCESS, WRITE_DENY_MASK)
  // DENY ACE 的优先级高于 ALLOW ACE，保证 .git 目录写保护

阶段 3-5：Token 创建 + Desktop + 启动

// 阶段 3：CreateRestrictedToken
HANDLE baseToken = OpenProcessToken(...)
CreateRestrictedToken(baseToken,
  DISABLE_MAX_PRIVILEGE |  // 移除所有特权
  LUA_TOKEN |              // 转换为低权限令牌
  WRITE_RESTRICTED,        // 只有列出的 SID 可写
  ..., restrictingSids, &restrictedToken)

// 阶段 4：独立 Desktop + Job Object
CreateDesktopW("hana-win-sandbox-{pid}-{tick}", ...)
CreateJobObjectW(...) + JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE

// 阶段 5：CreateProcessAsUserW 启动
CreateProcessAsUserW(restrictedToken, executable, commandLine,
  CREATE_SUSPENDED | CREATE_NO_WINDOW, ...)
AssignProcessToJobObject(job, process.hProcess)
ResumeThread(process.hThread)
⚠ Windows 网络隔离缺失：macOS 可通过 (deny network*) 阻断网络，Linux 可通过 --unshare-net 创建独立网络命名空间。但 Windows Restricted Token 无法控制网络访问——网络能力绑定在 token 的登录会话中，而非 token 本身。代码中 assertSandboxNetworkSupported() 函数直接抛出错误："Windows restricted-token sandbox does not support network-off mode."

五、检验一下（提取练习）

Q1. bwrap 使用 --ro-bind / / 后再用 --bind 覆盖特定路径，利用了 bwrap 的什么语义？

A. 先挂载的优先级更高 B. 后挂载覆盖先挂载——先设全局只读，再用可写 mount 覆盖特定路径 C. 所有挂载点平等，互相不影响 D. mount 选项与顺序无关，由 bwrap 内部排序
看答案B。这是 bwrap 设计的核心技巧——利用后挂载覆盖前挂载的语义，避免显式白名单的复杂度。

Q2. macOS Seatbelt SBPL 的 deny 规则为何放在 allow 规则之后？

A. 这是 Apple 的编码风格约定 B. SBPL 使用 last-match-wins 语义——deny 放在后面才能覆盖前面的 allow C. SBPL 使用 first-match-wins 语义 D. 顺序无关紧要，SBPL 引擎会自动排序
看答案B。SBPL 的 last-match-wins 语义要求 deny 规则写在 allow 规则之后才能生效。这与 bwrap 的 mount order 语义类似但方向相反。

Q3. Windows Restricted Token 模式下无法断网的根本原因是什么？

A. Windows API 不支持网络控制 B. 网络能力绑定在登录会话中而非 token 本身——Restricted Token 无法修改网络权限 C. 为了性能考虑故意不实现 D. NTFS ACL 不支持网络权限
看答案B。macOS 的 seatbelt 和 Linux 的 network namespace 都可以在网络层面做隔离，但 Windows 的 Restricted Token 机制设计上不涉及网络控制。

Q4. 哪个平台的 OS 沙盒在不可用时选择"拒绝执行"而非"降级到仅 PathGuard"？

A. macOS B. 只有 Linux C. Linux 和 Windows D. 三个平台全部拒绝执行
看答案C。Linux 上 bwrap 不可用时拒绝执行（提示安装 bwrap），Windows 上 helper 不可用时也拒绝执行。只有 macOS 在 seatbelt 不可用时降级到仅 PathGuard。
