1. 模块定位

server/ 是 HanaAgent 的 HTTP + WebSocket API 服务层，作为独立 Node.js 进程运行（通过 child_process.spawn 或 fork() 启动）。它基于 Hono 4 轻量 Web 框架构建，向上承接 Electron 桌面端 / Web 客户端 / 移动端 / CLI 的请求，向下调用 core/ 引擎和 lib/ 工具库。

核心职责：请求鉴权与路由授权、会话生命周期管理、LLM 流式响应广播、插件 iframe 安全沙箱、外部平台（Telegram / 飞书 / QQ 等）桥接、文件上传与工作台 API。
前置知识：L0 已讲解 Server 是 4 大运行时进程之一（Electron Main → Server → Browser → Bridge），通过 server-info.json 与 Electron Main 进程通信端口号。每个客户端通过 Bearer token 或 loopback token 鉴权。

2. 启动流程

Server 进程支持两种启动入口：

  • 独立运行：node server/boot.cjs（CJS 包装）→ server/bootstrap.ts（ESM 动态 import）→ server/index.ts
  • Electron 内嵌：Main 进程通过 child_process.fork("server/index.js") 直接启动，通过 IPC 通道接收端口号

2.1 三层启动桥

1
boot.cjs — CJS 启动包装器
用 CJS 的 dynamic import() 包装 ESM 入口 index.ts。ESM 的 static import 失败时进程直接崩溃无法输出诊断信息；CJS 的 try/catch 可以捕获模块加载阶段错误，将堆栈通过 stderr 传回父进程。
2
bootstrap.ts — ESM 动态导入 + 独立保活线程
使用 import(pathToFileURL(serverEntry)) 动态加载打包后的 bundle/index.js。启动独立的 Worker 线程作为 keepalive —— 每 5 秒通过 fs.writeSync(1, ...) 直接写 stdout fd（绕过主线程消息队列），防止主线程被 native module 加载（如 better-sqlite3）阻塞时，Electron 误判启动失败。
3
index.ts — 主启动入口
完整的 9 阶段启动序列，详见下方。

2.2 index.ts 9 阶段启动序列

1
环境初始化
解析 HANA_HOME 目录（默认 ~/.hanako/），创建 Pi SDK 目录结构，读取 package.json 获取版本号。
2
首次运行播种 (ensureFirstRun)
复制默认 config / agent / skills 到 ~/.hanako/，检测无效 agent 目录，备份损坏的默认助手 config.yaml。
3
初始化调试日志
在 ~/.hanako/logs/ 下创建持久化日志文件，后续所有模块通过 createModuleLogger("模块名") 写入。
4
创建 HanaEngine 并初始化
加载 agent 配置、模型列表、i18n、session 协调器、LLM 客户端。注入出站代理运行时。设置 BrowserManager 依赖。
5
注册 Framework Extensions
在插件 onStartup 之前注册 DeferredResult 和 CompactionGuard 两个核心 extension factory。时序关键：extension 必须在任何 session 创建前注册完毕，否则通过 session:send 创建的 session 会缺失核心 handler。
6
初始化插件系统 + 启动 Hub 调度器
engine.initPlugins() 加载所有已安装插件，hub.initSchedulers() 启动 Scheduler + ChannelRouter 定时任务。
7
创建 Hono 实例 + 注册路由
创建 new Hono()，注册全局中间件（CORS + 传输上下文 + 鉴权），挂载全部 38 个路由模块到 /api 前缀下。
8
绑定端口 + 写入 server-info.json
调用 server.listen(port, host) 开始接受请求。将端口号、token、进程 PID、网络模式等写入 ~/.hanako/server-info.json（权限 0o600），供 Electron Main 进程检测复用或外部工具查询。
9
后置启动：Bridge 平台 + CLI 模式
异步加载 BridgeManager（Telegram / 飞书 / QQ 等外部平台适配器）。如果 stdin 是 TTY 且带有 --cli 参数，附加 CLI 交互界面。

6. HTTP 中间件链

Hono 框架的全局中间件注册在 app.use("*", ...) 中，形成一条 6 层处理管道。每层按序执行，任一层拒绝即短路返回 403 或 404。

6.1 中间件链逐层解析

1
CORS 策略 (cors-policy.ts)
逻辑：检查 Origin 请求头，与允许列表比对。默认允许 localhost / 127.0.0.1 回环地址、Electron file:// 协议、以及 "null" origin。可通过 HANA_CORS_ORIGIN 环境变量收紧到单一来源。
响应头注入：Access-Control-Allow-Origin / Access-Control-Allow-Credentials / Access-Control-Allow-Methods / Access-Control-Allow-Headers。
OPTIONS 预检：直接返回 204，不进入后续鉴权链。
2
传输上下文推断 (transport-context.ts)
逻辑：根据 Host 头 + remoteAddress + networkMode 推断 connectionKind：
local — 回环地址或 localhost，loopback 模式下强制要求；
lan — 局域网模式下的非回环地址；
custom_remote — 自定义远程模式。
若推断失败（如 loopback 模式下收到外部 IP 请求），返回 invalid_transport 403。
3
特殊请求快速通道
在进入主鉴权前，按序检查 4 类特殊请求：
(a) Resource Ticket 请求：GET /api/resources/:id/content?ticket= — 直接放行，由路由内部校验 ticket；
(b) 插件 iframe Ticket 请求：GET /api/plugins/:id/*?pluginIframeTicket= — 校验 ticket 后放行；
(c) 畸形插件资源请求：直接返回 404；
(d) 插件资源 Session 请求：校验 plugin asset session 后放行。
4
公开路由快速通道 (route-security.ts)
逻辑：调用 isPublicHttpRoute() 检查是否为无需鉴权的公开路由。当前公开路由包括：移动/桌面静态资源 (/mobile/*, /desktop/*)、Web 认证端点 (/api/web-auth/*)、MCP OAuth 回调、HTML 预览文档页面。
公开路由直接进入业务 handler，不检查任何凭证。
5
主鉴权 + 路由授权 (request-principal.ts)
三层鉴权策略：
(a) 主鉴权：依次尝试 Bearer token → query ?token= → Web session cookie → WS ticket（仅 /ws 路由）；
(b) 插件 Surface Session 后备：仅当主凭证"缺席"（missing_credential）时尝试，防止无效 bearer 被 surface token 静默掩盖；
(c) 路由授权：调用 authorizeHttpRoute() 根据路由策略（public / authenticated / local_only / scope / studio_owner / plugin_route）校验 principal 是否有权访问该路由。
失败响应：403 + 结构化错误 body（error / reason / requiredScope / connectionKind）。
6
业务路由处理
鉴权通过后，请求进入对应的 Hono sub-app 路由处理器。每个路由内部可进一步调用 createRequestContext(c, engine) 构造请求上下文（含 capability 授权方法），以及 denyWithoutScope() / denySecretMutationWithoutScope() 做细粒度权限校验。

6.2 路由安全策略分级 (route-security.ts)

每个路由在 classifyHttpRoute() 中被分配一个安全策略等级：

策略等级含义示例路由
public 无需任何凭证，直接放行 /mobile/*、/desktop/*、/api/web-auth/login、/preview/html/:id
local_only 仅本机 loopback token 持有者可访问 /api/shutdown、/api/access/*、/api/devices/*、/api/plugins/install
authenticated 任意有效凭证即可（不限作用域） /api/health、/api/server/identity
scope 需持有特定 capability scope /api/chat → chat、/api/config → settings.read/write、/api/sessions → chat、/api/providers/* → providers.manage
studio_owner 仅 Studio owner 可访问 /api/usage/llm (GET)、未匹配具体策略的 /api/*
plugin_route 仅匹配的插件 surface principal 或 owner /api/plugins/:pluginId/（排除宿主自有 ID）

8.3 WS 作用域控制 (ws-scope.ts)

每个 WS 客户端连接被封装为 WsClientRecord，包含：

  • principal：连接时的鉴权身份
  • subscriptions：订阅列表（按 { kind: "session"|"studio", studioId, sessionPath } 订阅）

事件广播时的过滤逻辑：

  • Local owner：接收所有事件（不过滤）
  • Session 事件：必须 chat.read scope + 同 studioId + sessionPath 匹配订阅
  • 全局安全事件：session_created、notification 只需 chat.read scope
  • 缩略图保护：非 local owner 不接收 thumbnail 数据

上行消息的权限校验：

  • 写消息（prompt / abort / steer / interject / slash / compact）：需要 chat.write scope
  • 读消息（resume_stream / context_usage）：需要 chat.read scope

8.4 流式事件存储 (session-stream-store.ts)

每个 session 维护一个 Ring Buffer 存储流式事件：

  • 容量：默认 5000 条事件 / 8 MB 总大小 / 单条事件 256 KB 上限
  • 生命周期：每轮 turn 开始时重置 buffer，turn 结束时清空
  • 超大事件压缩：超出 256 KB 的事件自动压缩（截断大文本字段、限制数组/对象深度）
  • stream_resume：客户端断线重连后，根据 sinceSeq 批量回放遗漏事件

10. 架构要点

10.1 关键设计决策

  1. 独立进程架构：Server 是独立 Node.js 进程，与 Electron Main 进程通过 IPC + server-info.json 文件通信。这种设计使 server 崩溃不会拖垮整个桌面应用，也支持独立部署为 headless 服务器。
  2. Hono 而非 Express：选择 Hono 4 是因为其轻量（~13KB）、TypeScript-first、原生支持 Web Standard Request/Response、内置 WebSocket helper。相比 Express 更适合嵌入式场景。
  3. 两层鉴权分离：全局中间件负责"你是谁"（principal 解析），路由内部负责"你能做什么"（capability 授权）。request-principal.ts 抽取为共享模块，测试和生产走同一条链路防止漂移。
  4. 插件沙箱：插件 iframe 页面的 HTTP 请求通过 3 种独立凭证体系隔离 —— plugin iframe ticket（一次性页面加载凭证）、plugin asset session（静态资源访问）、plugin surface session（插件自有路由 API 调用）。
  5. 流式事件 Ring Buffer：每轮 turn 的事件缓存在内存中（5000 条 / 8MB 上限），支持断线重连后的 stream_resume 续传。超大事件自动压缩，防止单条 tool 结果（如大文件内容）撑爆内存。
  6. 双 WebSocket Server：Chat WS 走 Hono 的 @hono/node-ws；内置浏览器控制 WS 走独立的 ws.WebSocketServer。升级处理器通过 URL pathname 分流，避免两个 handler 对同一 upgrade 请求重复处理导致 socket 损坏。
  7. 启动时序保证：Framework extensions 必须在插件 onStartup 和任何 session 创建前注册，否则 ExtensionRunner 构造时拿不到完整的 factory 集合。
