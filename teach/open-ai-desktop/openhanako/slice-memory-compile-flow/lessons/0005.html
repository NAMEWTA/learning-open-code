第 5 课 · 异常路径全解与综合测验 容错设计

🎯 本节目标：学完你能解释 HanaAgent 记忆系统的 5 条异常路径及其防护机制，画出全链路时序图，并通过综合测验验证对整套编译→存储→检索链路的端到端理解。

前四课我们按"编译→存储→检索"的顺序走完了正常路径。但现实中的代码运行充满异常——LLM 返回半截输出、API 凭证过期、进程崩溃、并发竞态。这一课逐条分析 5 种异常场景及防护机制，最后用综合测验检验全链路掌握程度。

一、异常路径 1：LLM 返回未闭合 think 标签

这是真实发生过并已修复的 bug 场景。LLM 可能输出 <think>用户今天主要讨论了...<EOF>——只有开头没有结尾的半截思考块。

Bug 描述：LLM 输出 "<think>用户今天主要讨论了...<EOF>"——缺少 </think> 闭合标签。stripThinkTagBlocks 会返回空字符串，导致 today.md 被清空。

修复（compiled-memory-state.ts）：normalizeCompiledLLMResult() 在 strip 后检测结果为空的场景——如果原值以 <think> 开头且不含 </think>，则抛出异常 "compileToday returned an unterminated thinking block"。异常传播到 MemoryTicker 的步骤级 catch，today.md 保持旧内容不丢失，health 状态记录 failCount++。

二、异常路径 2：API 凭证过期导致连续编译失败

API key 过期是最常见的编译失败原因。MemoryTicker 实现了三层防护：

层级机制代码位置
错误去重_lastErrorSig：相同根因只在 console 打一次日志，避免每 10 轮都刷屏_logStepError()
恢复检测_markStepRecovered()：连续失败后首次成功时，显式记录恢复事件每步的 catch 块
健康状态每步独立 failCount：一次成功立即清零，方便 UI 层读取_health 对象 + getHealthStatus()
💡 设计原则：错误应该被精确去重、可恢复、可观测。不要淹没在重复的日志中，不要让一次失败永久影响状态，不要让 UI 层不知道发生了什么。

三、异常路径 3：rollDailyWindow fold 失败不丢数据

关键设计：rollDailyWindow 的 fold 失败时不删除源 daily 文件。源文件保留在 daily/ 目录，交给下一轮每日任务重试。

catch (err) {
  log.error("rollDailyWindow: fold 进 longterm 失败，" +
    `保留 ${entries.length} 份 daily 条目待下轮重试`);
  return { folded: [], failed: entries.map(e => e.date) };
}
返回值中的 failed 数组被 MemoryTicker 记录，下一次 _doDaily() 执行时重新尝试 fold。这是不静默丢弃数据的最后防线。

四、异常路径 4：断点续跑 —— 进程崩溃后恢复

daily-state.json 持久化每日流水线中每步的完成状态。如果进程在步骤 3 崩溃重启：

// daily-state.json 示例（进程崩溃时）
{
  "schemaVersion": 4,
  "logicalDate": "2026-07-07",
  "completedSteps": {
    "compileDaily": "2026-07-07T04:05:00.000Z",     // 已完成
    "compileToday": "2026-07-07T04:05:30.000Z",      // 已完成
    "rollDailyWindow": null                            // 未完成
  }
}

// 下次 tick 恢复时：
//   _restoreDailyProgress() 读取 daily-state.json
//   _dailyStepsCompleted = { compileDaily, compileToday }
//   _doDaily() 从 rollDailyWindow 开始执行（跳过已完成的步骤）
//   不会重复调用已完成的 LLM 步骤
💡 schemaVersion 迁移机制：当 daily 步骤语义发生变化时（如 v3→v4 的 compileDaily 输入从"当天全部摘要"改为"昨天最终版草稿"），schemaVersion 提升。旧 schema 的状态文件被判定为不匹配 → 返回 null → 触发一次性重算。重算是幂等的——同一批数据重复编译不产生内容偏移。

五、异常路径 5：并发与竞态保护

保护机制变量作用
每日任务互斥_dailyRunning防止 _checkDailyJob() 被多次触发时重复执行 _doDaily()
Tick 串行化_tickInFlight确保同一时刻只有一个 tick() 在执行
摘要并发保护_summaryInProgress (Set)防止同一 session 的滚动摘要被并发重复调用
优雅关闭_activeJobs (Set)stop() 时等待所有后台异步任务完成后才退出
原子写入atomicWriteSync先写临时文件再 rename，防止写到一半崩溃损坏数据
→五层防护的协作关系：_dailyRunning 和 _tickInFlight 保证宏观调度互斥；_summaryInProgress 保证同一 session 不被重复摘要；_activeJobs 保证优雅关闭不丢任务；atomicWriteSync 保证单次写入的原子性。从调度层到 I/O 层，每层都有对应的并发保护。

七、容错设计原则总结

从 5 条异常路径可以提炼出 4 条核心设计原则：

八、综合测验

Q1. 从用户发送消息到记忆在下次对话中生效果，经历的正确步骤顺序是什么？

A. notifyTurn → compileToday → assemble → buildSystemPrompt → rollingSummary B. notifyTurn → rollingSummary → compileToday → assemble → buildSystemPrompt C. rollingSummary → buildSystemPrompt → compileToday → assemble → notifyTurn D. assemble → compileToday → rollingSummary → notifyTurn → buildSystemPrompt
看答案B。完整链路：notifyTurn 触发 → rollingSummary 生成结构化摘要 → compileToday 更新今日草稿 → assemble 拼接 memory.md → buildSystemPrompt 注入 System Prompt。记忆在下一次新 session 创建时生效。

Q2. 以下哪个场景会导致数据永久丢失？

A. API 凭证过期，compileToday 连续失败 10 次 B. rollDailyWindow 的 compileLongterm 失败后，立即删除源 daily 文件 C. 进程在 _doDaily() 执行 compileDaily 时崩溃 D. today-state.json 文件被误删
看答案B。A 只会导致 today.md 暂时不更新，摘要数据仍在 summaries/ 中。C 会被 daily-state.json 的断点续跑机制恢复（从崩溃的步骤重新开始）。D 会在下次 compileToday 时触发无水位线路径（扫描全部摘要重建）。只有 B——如果 fold 失败后立即删除源文件——会丢失还没沉淀进 longterm.md 的 daily 条目。这正是为什么 rollDailyWindow 采用"先 fold 后 delete"策略。

Q3. MemoryTicker 使用 _dailyRunning、_tickInFlight、_summaryInProgress 三个锁，它们各自保护什么？

A. 三个锁保护同一个资源，冗余设计 B. _dailyRunning 保护 _doDaily 不被重复执行，_tickInFlight 保护 tick() 串行化，_summaryInProgress 保护同一 session 不被并发生成摘要 C. 分别对应三个 LLM 模型的并发限制 D. 分别对应三个不同 session 的并发控制
看答案B。三个锁保护不同粒度：_dailyRunning 是每日任务的互斥锁（每日任务包含多个步骤，整体不能重入）；_tickInFlight 是单次 tick 的互斥锁（notifyTurn 和定时器可能同时触发 tick）；_summaryInProgress 是 per-session 的摘要锁（同一个 session 的摘要不能并发重复生成，但不同 session 可以并行）。

Q4. 综合判断题：以下哪些说法是正确的？（多选思维——选出最准确的一项）

A. memory.md 每次对话开始时从磁盘重新读取，所以后台编译更新会立即影响当前对话 B. compileDaily 的 fingerprint 去重可以防止同一批数据重复编译，而 daily-state.json 的断点续跑可以防止进程崩溃后重复执行已完成的步骤 C. compileToday 和 compileEditableFacts 使用不同的状态文件和水位线机制 D. normalizeCompiledLLMResult 检测到未闭合 think 标签时会自动补上闭合标签
看答案B。A 错——memory.md 在 session 创建时冻结快照到 _systemPrompt，后台更新不影响当前 session。C 错——两者使用同构的水位线模式（today-state.json 和 editable-facts-state.json 结构相同）。D 错——检测到未闭合 think 标签时不是自动修复，而是抛出异常让步骤级 catch 处理，保持旧内容不丢失。
