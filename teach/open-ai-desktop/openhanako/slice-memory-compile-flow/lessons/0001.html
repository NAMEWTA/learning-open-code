第 1 课 · 记忆系统架构：三层存储与三条触发路径 架构概览

🎯 本节目标：学完你能画出 HanaAgent 记忆系统的三层存储架构（短期摘要 / 长期编译产物 / 工作记忆注入），讲清 MemoryTicker 的三条触发路径分别在什么条件下、执行哪些步骤。

HanaAgent 的记忆系统不是"存下来就行"——它是一套分级编译流水线。对话先被压缩为结构化摘要（短期），然后通过多级 LLM 编译蒸馏为分块记忆文件（长期），最终在下次对话时注入 System Prompt（工作记忆）。这一课我们先建立全局架构认知，看清记忆数据的三层形态和编译的触发时机。

一、三层存储架构

记忆系统维护三种不同粒度的数据，对应不同的存储位置和生命周期：

L1 · 工作记忆
memory.md 注入 System Prompt · 当前 session 冻结快照 · 约 2000 tokens
↑ 每次 compile 后 assemble 装配
L2 · 长期编译产物
facts.md / today.md / week.md / longterm.md 四个分文件
↑ compileToday / compileDaily / compileEditableFacts / compileLongterm
L3 · 短期会话摘要
memory/summaries/{sessionId}.json · 结构化 facts + timeline
↑ SessionSummaryManager.rollingSummary() 每 10 轮生成
原始对话消息
用户与 Agent 的每轮对话 · core/engine.ts 驱动
层级数据形态存储位置生命周期生成方式
L1 工作记忆一段 ≤2000 token 的 Markdown 文本内存中的 _systemPrompt当前 session 内不变，跨 session 刷新buildSystemPrompt() 注入
L2 编译产物4 个分文件 + 1 个聚合文件memory/facts.md 等磁盘文件持续更新，日期切换时重算compileToday/Daily/Longterm LLM 编译
L3 会话摘要JSON 文件，含 facts + timeline 两节memory/summaries/随会话积累，删除需主动操作rollingSummary() LLM 生成
💡 关键洞察：为什么需要三层？
原始对话太长（几千到几万 token）→ 摘要压缩到几百 token → 编译再压缩到几十 token → 最终 memory.md 整段 ≤2000 token。每一层都在蒸馏——保留语义，丢弃冗余。如果直接存原始对话注入 System Prompt，每次调用都要消耗大量上下文窗口。

数据流概览：从原始对话到 System Prompt

原始对话消息（每轮 ~200-2000 tokens）
  │  SessionSummaryManager.rollingSummary()
  │  压缩比约 10:1，按轮数预算 40 字/轮
  ▼
会话摘要 JSON（facts + timeline，约 200-400 字）
  │  compileToday() 提取 time 事件 → compileDaily() 蒸馏日记
  │  每级再压缩约 3:1 ~ 5:1
  ▼
编译分文件（facts.md / today.md / daily/{date}.md / longterm.md）
  │  assemble() 纯文本拼接，零 LLM
  ▼
memory.md（四个 ## 段，≤ 2000 tokens）
  │  buildSystemPrompt() 注入 System Prompt 尾部
  ▼
Agent 下次对话中无感使用记忆

注意压缩比的逐级衰减：原始对话 → 摘要约 10:1，摘要 → 编译产物约 5:1，最终注入量仅约 2000 tokens。整个链路的 token 成本大头在 rollingSummary 的 LLM 调用（每次约 400 字输出），后续编译级的 budget 极紧（compileToday 450 tokens、compileDaily 100 tokens）。

二、入口点：MemoryTicker.notifyTurn

整套链路的总入口是 lib/memory/memory-ticker.ts 的 notifyTurn(sessionPath)。每次用户与 Agent 完成一轮对话后，core/engine.ts 的 HanaEngine 会调用它。

// lib/memory/memory-ticker.ts notifyTurn 核心逻辑（简化）
function notifyTurn(sessionPath) {
  if (_stopped) return;
  const sessionKey = _sessionIdentityForPath(sessionPath);
  const count = (_turnCounts.get(sessionKey) || 0) + 1;
  _turnCounts.set(sessionKey, count);

  const memoryOn = _isSessionMemoryOn(sessionPath);

  // 每 10 轮触发一次滚动摘要 + 今日编译
  if (count % TURNS_PER_SUMMARY === 0 && memoryOn) {
    _trackJob(_doRollingSummary(sessionPath, "threshold")
      .then(() => _doCompileTodayAndAssemble())
      .catch(() => {}));
  }

  // 检测日期变化，触发每日任务
  if (memoryOn) _checkDailyJob();
}

三、三条触发路径

触发路径触发条件执行内容代码位置
Turn-based每 TURNS_PER_SUMMARY = 10 轮对话rollingSummary → compileToday → assemblenotifyTurn() 每 10 轮分支
Session 结束notifySessionEnd() 被调用时final rollingSummary → compileToday → assemblenotifySessionEnd()
每日一次逻辑日发生变化（凌晨 4:00 边界）完整 5 步流水线：compileDaily → compileToday → rollDailyWindow → compileEditableFacts → assemble_checkDailyJob() → _doDaily()
1Turn-based 路径：最频繁（每 10 轮），只做增量——生成新摘要 + 更新今日草稿 + 重新装配 memory.md。不触发跨日逻辑。
notifyTurn() → _doRollingSummary() → _doCompileTodayAndAssemble()
2Session 结束路径：用户结束会话时触发。与 Turn-based 路径执行相同逻辑，但目的是确保最后几轮对话也被摘要（不足 10 轮也会触发）。
notifySessionEnd() → _doRollingSummary("final") → _doCompileTodayAndAssemble()
3每日一次路径：最重——在逻辑日切换时（凌晨 4 点边界），将昨天的 today.md 蒸馏为 daily/{date}.md，清空 today.md 开始新一天，折叠旧 daily 条目进 longterm.md，更新 facts.md。
_checkDailyJob() → _doDaily() → 5 步顺序执行

五、检验一下（提取练习）

Q1. HanaAgent 记忆系统的三层存储从上到下是什么？

A. 原始对话 → 会话摘要 → 编译产物 B. 工作记忆（System Prompt 注入）→ 编译产物（四个分文件）→ 会话摘要（summaries/*.json） C. 向量数据库 → 关系数据库 → 文件系统 D. facts.md → today.md → memory.md
看答案B。L1 工作记忆是注入 System Prompt 的 memory.md 片段（约 2000 tokens），L2 编译产物是 facts/today/week/longterm 四个分文件，L3 会话摘要是 summaries/ 目录下的 JSON 文件。A 缺少注入层，C 是错误的技术栈，D 是 L2 内部的分文件而非层级。

Q2. Turn-based 路径每多少轮对话触发一次？

A. 每 5 轮 B. 每 10 轮 C. 每 20 轮 D. 每轮都触发
看答案B。TURNS_PER_SUMMARY = 10，定义在 memory-ticker.ts 中。每 10 轮触发一次 rollingSummary → compileToday → assemble 的增量编译链。

Q3. 每日一次路径和 Turn-based 路径的最大区别是什么？

A. 每日路径不调用 LLM B. 每日路径额外执行 compileDaily（昨日蒸馏为日记）、rollDailyWindow（折叠旧日记进 longterm）、compileEditableFacts（更新事实） C. 每日路径只在 Session 结束时调用 D. 两者完全一样，只是触发条件不同
看答案B。每日路径在 Turn-based 路径的基础上，额外增加了三个步骤：compileDaily（将昨天的 today.md 蒸馏为 daily 日记）、rollDailyWindow（将超出 6 日窗口的日记折叠进 longterm.md）、compileEditableFacts（增量更新 facts.md）。

Q4. notifyTurn 中为什么需要检查 _stopped 和 memoryOn？

A. 纯粹是代码风格，没有实际作用 B. _stopped 防止关闭后继续触发编译（优雅关闭），memoryOn 允许按 session 级别关闭记忆功能（节省 token） C. _stopped 是调试开关，memoryOn 是性能开关 D. 两者都是为了控制 LLM 调用的并发数
看答案B。_stopped 在 stop() 被调用后置为 true，防止关闭过程中新触发编译任务（配合 _activeJobs 等待已有任务完成）。memoryOn 通过 _isSessionMemoryOn() 检测当前 session 是否启用了记忆功能——不是所有 session 都需要记忆（例如临时调试 session）。
