一、assembleWeekFromDaily：零 LLM 的周报装配

compile.ts 第 531-552 行。week.md 不再是独立 LLM 编译产物，改为从 memory/daily/ 目录纯文件拼接最近 6 个已结束逻辑日的日记条目。零 LLM 调用——每天省下一次 600 token 的编译。

export function assembleWeekFromDaily(dailyDir, weekPath, opts) {
  const maxDays = opts.maxDays || 6;   // DAILY_WINDOW_RETENTION_DAYS
  const maxChars = opts.maxChars || 1200; // WEEK_ASSEMBLY_MAX_CHARS

  // 1. 扫描 daily/ 目录，找到最近 6 天的日记文件
  const entries = _listDailyEntries(dailyDir).slice(-maxDays);

  // 2. 读取每个文件内容，过滤空文件
  const blocks = entries.map(read).filter(非空);

  // 3. 拼接
  let content = blocks.join("\n\n");

  // 4. 超长截断：从最老的条目开始丢弃
  if (content.length > maxChars) {
    从 blocks 头部开始 shift，直到总长 ≤ maxChars
    仍超限则从尾部截断 content
    log.warn 记录截断事件（不静默丢弃）
  }

  atomicWrite(weekPath, content);
}
💡 设计意图：每条 daily 编译时 LLM budget 仅 100 tokens，6 条拼起来 ≤1200 字符。与旧版 LLM 编译 week 段的体量相当，但成本降为零。这是"上游精准控制、下游零成本装配"的范例——在 compileDaily 阶段就把每条日记压到足够短，下游直接拼接即可。

二、rollDailyWindow：滚动窗口折叠进长期记忆

compile.ts 第 564-596 行。将超出 6 日窗口的 daily 条目折叠进 longterm.md。核心安全设计：fold 成功后才删源文件。

1计算截止日期：cutoffDate = shiftLogicalDate(referenceDate, -6)。所有 date < cutoffDate 的 daily 条目需要折叠。
6 天窗口与 assembleWeekFromDaily 的 maxDays=6 对齐
2拼装内容：将筛选出的条目按 ## {date}\n\n{body} 格式拼接为一段文本。
保持 Markdown 格式，方便 compileLongterm 识别日期边界
3调用 compileLongterm fold：compileLongterm 是通用 fold 入口——将新旧内容合并后喂给 LLM 压缩（maxTokens=600）。
fingerprint 去重：内容未变就跳过 LLM 调用
4安全删除：fold 成功后才删除源 daily 文件。失败则保留源文件，交给下一轮重试。
这是防止数据静默丢失的关键保障

compileLongterm — 通用 fold 入口

export async function compileLongterm(content, longtermPath, resolvedModel) {
  const fp = computeFingerprint([content]);
  if (已有指纹匹配 && 文件存在) return "skipped";

  const input = 已有 longterm
    ? `## 上一份长期情况\n\n${已有}\n\n## 新沉淀内容\n\n${content}`
    : `## 新沉淀内容\n\n${content}`;

  const result = await _compactLLM(input, compileLongtermPrompt, resolvedModel, 600);
  atomicWrite(longtermPath, result);
  fs.writeFileSync(fpPath, fp);
  return "compiled";
}

三、compileEditableFacts：重要事实增量提取

compile.ts 第 906-968 行。facts.md 是唯一的事实产物路径——不再有"短期事实"和"长期事实"之分。采用与 compileToday 同构的水位线增量编译（editable-facts-state.json）。

// editable-facts-state.json 结构（与 today-state.json 同构）
{
  "lastCompiledSummaryUpdatedAt": "2026-07-07T12:00:00.000Z",
  "updatedAt": "2026-07-07T14:30:00.000Z"
}

// 增量逻辑：
// - 只处理 updated_at > 水位线的摘要中的事实段
// - 只处理含有 "### 重要事实" 标题的摘要（hasFactSectionHeading）
// - 跳过只有空事实段的摘要（isEmptyFactSection）
// - 新旧事实合并后喂给 LLM（maxTokens=300），由 LLM 去重、更新、整理
→ensureEditableFactsBaseline：确保 facts.md 存在并在状态文件中补上首次水位线。如果 facts.md 不存在则创建空文件；如果状态文件中无水位线但有摘要，则用水位线回填（标记所有已有摘要为已处理），避免把已沉淀过的旧摘要重新计入。

四、assemble：最终 memory.md 装配（零 LLM）

compile.ts 第 978-1002 行。将四个分文件同步拼接为 memory.md——整个编译链路的最终输出。零 LLM 调用，纯模板拼接。

export function assemble(factsPath, todayPath, weekPath, longtermPath, memoryMdPath) {
  const facts    = normalizeCompiledSectionBody(read(factsPath));
  const today    = normalizeCompiledSectionBody(read(todayPath));
  const week     = normalizeCompiledSectionBody(read(weekPath));
  const longterm = normalizeCompiledSectionBody(read(longtermPath));

  const memoryMd = buildCompiledMemoryMarkdown({ facts, today, week, longterm });
  atomicWrite(memoryMdPath, memoryMd);
}

export function buildCompiledMemoryMarkdown({ facts, today, week, longterm }) {
  const empty = isZh ? "（暂无）" : "(none)";
  return [
    `## 重要事实\n\n${facts || empty}`,
    `## 今天\n\n${today || empty}`,
    `## 本周早些时候\n\n${week || empty}`,
    `## 长期情况\n\n${longterm || empty}`,
  ].join("\n\n") + "\n";
}

四个标题始终保留，空栏写占位符"（暂无）"——避免格式漂移导致下游解析出错。总长控制在约 2000 token 以内。

💡 编译流水线全貌：rollingSummary（LLM）→ compileToday（LLM）→ compileDaily（LLM）→ assembleWeekFromDaily（零LLM）→ rollDailyWindow + compileLongterm（LLM）→ compileEditableFacts（LLM）→ assemble（零LLM）。5 次 LLM 调用 + 2 次纯文件操作。每条 daily 编译仅 100 tokens 预算，longterm fold 600 tokens——整个每日流水线约 1000+ tokens 的总 LLM 成本。

五、检验一下（提取练习）

Q1. assembleWeekFromDaily 为什么不需要 LLM？

A. 因为 week 段不重要，随便拼一下就行 B. 因为 compileDaily 阶段已经把每条日记压到 100 tokens 以内，6 条拼接 ≤1200 字符直接可用 C. 因为 LLM 不支持周报格式 D. 因为 week 段的内容每周才变一次，不需要频繁更新
看答案B。这是"上游精准控制、下游零成本装配"的设计：compileDaily 的 LLM budget 仅 100 tokens，确保每条日记足够紧凑。下游只需按日期排序拼接即可，省下一次 600 token 的 LLM 调用。

Q2. rollDailyWindow 的 fold 失败时为什么不能删除源 daily 文件？

A. 因为文件系统权限问题 B. 因为删除操作会触发并发冲突 C. 因为 fold 失败意味着内容尚未沉淀进 longterm.md——删除源文件会导致数据永久丢失 D. 因为需要保留文件用于审计日志
看答案C。rollDailyWindow 采用"先 fold 后 delete"策略：只有 compileLongterm 成功返回后才删除源 daily 文件。失败则保留源文件，交给下一轮 _doDaily() 重试。这是防止数据静默丢失的关键设计。

Q3. compileEditableFacts 与 compileToday 在增量机制上有什么共同点？

A. 都使用同一个状态文件 B. 都使用水位线增量编译——通过状态文件跟踪 lastCompiledSummaryUpdatedAt，只处理新增/修订的摘要 C. 都完全不需要状态文件 D. 都只在每日任务中执行
看答案B。两者使用同构的水位线模式：compileToday 使用 today-state.json，compileEditableFacts 使用 editable-facts-state.json。状态文件结构相同（lastCompiledSummaryUpdatedAt + updatedAt），只处理 updated_at > 水位线的摘要。compileEditableFacts 额外检查是否有 "### 重要事实" 标题和是否为空事实段。

Q4. assemble 函数中四个标题为什么始终保留，空栏写"（暂无）"？

A. 为了美观 B. 为了避免格式漂移导致下游解析出错——如果某天没有事实或长期记忆更新，标题缺失会破坏 memory.md 的结构约定 C. 因为 LLM 要求必须有四个段 D. 为了给用户一种"功能齐全"的感觉
看答案B。buildCompiledMemoryMarkdown 始终输出四个 ## 标题，即使某个段为空也写入"（暂无）"。下游 readCompiledMemorySnapshot 解析 memory.md 时依赖标题匹配来映射四个 key——如果标题缺失，解析结果会错位或丢失字段。
