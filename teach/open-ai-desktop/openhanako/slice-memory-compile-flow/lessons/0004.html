三、config-loader：三通道模型解耦

lib/memory/config-loader.ts 职责单一：读写 agent 的 config.yaml，提供缓存和原子写入。关键设计是记忆编译使用独立的模型通道。

通道YAML key用途
主通道api对话模型（chat model）
Embedding 通道embedding_api向量嵌入（可选）
工具模型通道utility_api记忆编译使用此通道——compileToday/Daily/Longterm/Facts 都通过 getResolvedMemoryModel() 获取凭证和模型名
💡 模型解耦的价值：对话可能使用旗舰模型（如 Claude Opus），记忆编译使用更便宜的模型（如 GPT-4o-mini）。每天编译消耗约 1000+ tokens——如果全用旗舰模型，日积月累成本可观。所有 LLM 调用通过 _compactLLM() 统一包装：cacheGroup prompt layout、60 秒超时、temperature=0.3。

四、buildSystemPrompt：检索的终点

记忆的检索方式不是 RAG、不是向量搜索、不是按需查询——而是直接把整个 memory.md（≤2000 token）注入到 System Prompt。这是整个记忆链路最终的目的地。

System Prompt 结构与缓存前缀优化

记忆使用规则（注入在 memory.md 之前）

记忆和用户档案是你内化的背景知识。你和{userName}是认识很久的人，这些事你本来就知道。你对{userName}的了解应该像空气一样，在场但不可见。记忆的存在感应该是零，它的作用应该是满的。

- 只有当{userName}提到了相关内容，记忆才参与进来。
- 永远不要让{userName}感觉到"记忆"这个东西的存在。禁止使用"我记得""你之前说过""根据记忆"这类表述。
- 记忆可能过时，当前对话永远优先。信息冲突时以对话为准。

Session 级别快照隔离

每个 session 在创建时冻结一份 memory.md 的快照（存储在 agent._systemPrompt 中）。后续对话过程中 memory.md 被后台 compile 更新时，不影响已经开始的 session。只在切换 session 或开启新对话时才读到最新版记忆。onCompiled 回调通知 agent 刷新 _systemPrompt，但只在下次新 session 创建时生效。

六、检验一下（提取练习）

Q1. readCompiledMemorySnapshot 的读取策略是什么？

A. 始终解析 memory.md B. 优先读四个分文件（facts/today/week/longterm.md），分文件不存在时才 fallback 解析 memory.md C. 从向量数据库检索 D. 从 summaries/ 目录读取摘要文件
看答案B。分文件是编译的直接产物，memory.md 是 assemble 的聚合输出。优先读分文件更可靠（避免聚合过程中的中间状态），fallback 到 memory.md 解析保证了向后兼容。

Q2. 记忆编译为什么使用独立的 utility_api 通道而非对话的 api 通道？

A. 因为 utility_api 速度更快 B. 为了模型解耦——对话用旗舰模型，记忆编译用便宜的小模型，大幅降低每日编译的 token 成本 C. 因为对话模型不支持编译功能 D. 因为 embedding 通道不可用时的回退方案
看答案B。config-loader 定义了三通道：api（对话）、embedding_api（向量嵌入）、utility_api（工具调用，含记忆编译）。通过 getResolvedMemoryModel() 获取 utility 通道的凭证和模型名，允许用户配置独立的低成本模型处理每天约 1000+ tokens 的编译任务。

Q3. buildSystemPrompt 中缓存前缀的作用是什么？

A. 让前端渲染更快 B. 将跨 session 不变的静态内容（平台标识、工具纪律、安全策略等）放在前面，让 LLM 服务端的 KV cache / prompt cache 命中，减少每次调用的 token 消耗 C. 减少网络传输量 D. 避免 System Prompt 被用户看到
看答案B。System Prompt 的前 10 项是跨 session 不变的静态内容，构成缓存前缀。LLM 服务端（如 Anthropic 的 prompt cache）可以缓存这部分内容，后续调用只需发送变化的后半段（用户档案、记忆内容等），大幅降低 token 消耗。

Q4. 为什么每个 session 要冻结一份 memory.md 快照？

A. 为了节省磁盘空间 B. 为了避免 session 进行中记忆被后台编译更新导致 System Prompt 不一致——session 创建时冻结快照，整个会话期间记忆内容保持不变 C. 为了实现多用户并发 D. 为了支持离线模式
看答案B。agent._systemPrompt 在 session 创建/切换时重建，注入当时的 memory.md 快照。后续后台 compile 更新 memory.md 不影响已有 session——避免对话进行到一半时记忆突然变化导致上下文不一致。下次新 session 创建时才会读到最新记忆。
