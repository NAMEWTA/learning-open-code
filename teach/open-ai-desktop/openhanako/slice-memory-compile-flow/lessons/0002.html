🎯 本节目标：学完你能讲清 rollingSummary 的 6 步生成流程、compileToday 的水位线状态机三种转换、compileDaily 的输入优先级链和 fingerprint 去重机制，以及为什么 compileDaily 必须先于 compileToday 执行。

上一课我们看到了三条触发路径，这一课深入 Turn-based 和每日路径中最核心的三个编译步骤：rollingSummary 生成结构化摘要（数据源头）、compileToday 增量编译今日草稿（调用最频繁）、compileDaily 蒸馏昨日日记（每日一次）。这三个步骤构成了编译流水线的上半段。

一、rollingSummary：结构化摘要的生命周期

lib/memory/session-summary.ts 的 SessionSummaryManager.rollingSummary() 是整个记忆系统的数据源头——所有后续编译都依赖它产出的结构化摘要。

1增量消息获取：对比 existing.messageCount 与实际消息数，只取上次摘要之后的新消息，避免长会话上下文爆炸。
_buildConversationText() 构建带时间戳和角色标注的文本
2预算分配：_rollingSummaryBudget(turnCount) 按轮数线性缩放——每轮 40 字配额，10 轮封顶 400 字。facts 占 30%，timeline 占 70%。
预算精确到字符数而非 token 数，给 LLM 更直观的约束
3LLM 调用：_callRollingLLM() 构建缓存友好的 prompt layout（cacheGroup: "memory.rolling_summary"），temperature=0.3，timeout=60s。
低 temperature 保证摘要格式稳定，cacheGroup 让 LLM 提供商复用 KV cache
4格式校验+修复：validateRollingSummaryFormat() 检查输出是否有 facts + timeline 两节标题。不满足则调用 _callRollingRepairLLM() 自动修复，最多 3 次。
格式契约是下游 compileEditableFacts 能正确提取 fact 段的前提
5PII 脱敏 + 落盘：scrubPII() 检测并脱敏个人身份信息，最终通过 saveSummary() 原子写入到 summaries/{sessionId}.json。
原子写入：先写临时文件再 rename，防止崩溃损坏数据

二、compileToday：水位线状态机

lib/memory/compile.ts 的 compileToday() 是整个管线中调用最频繁的步骤——每 10 轮对话和 session 结束都会触发。它使用 today-state.json 跟踪水位线，只处理新增/修订的摘要，不重复编译已处理内容。

// today-state.json 结构
{
  "schemaVersion": 1,
  "logicalDate": "2026-07-07",
  "lastCompiledSummaryUpdatedAt": "2026-07-07T14:30:00.000Z",
  "updatedAt": "2026-07-07T14:30:01.000Z"
}

三种状态转换：

状态条件行为关键细节
逻辑日切换清空 today.md + 重置水位线，新一天独立积累state.logicalDate !== 当前逻辑日
同一逻辑日、水位线存在只取 updated_at > 水位线的摘要的时间线条目作为 delta，与已有 today.md 合并喂给 LLM 续写增量编译的核心——避免重复处理
同一逻辑日、水位线不存在扫描所有摘要中属于今天的时间线条目，与已有 today.md 合并首次编译 / 状态丢失时的幂等回退
// compileToday 核心逻辑（compile.ts 第 305-373 行 简化）
async function compileToday(summaryManager, outputPath, resolvedModel, opts) {
  const state = readTodayState(statePath);
  const dayChanged = state && state.logicalDate !== 当前逻辑日;
  if (dayChanged) { 清空 today.md; state = null; }

  const sessions = getCandidateSummariesForCompile(summaryManager, 水位线);
  if (sessions.length === 0) return "compiled"; // 无新数据

  const events = timelineEventsForLogicalDate(sessions, logicalDate);
  const delta = formatTimelineEventsForCompile(events, {
    since: watermark, includeRevisionMarker: true,
  });

  // 合并旧草稿 + delta → LLM 续写（maxTokens=450）
  const input = 旧草稿存在
    ? `## 上一版今日草稿\n\n${旧草稿}\n\n## 新增或修订的时间线条目\n\n${delta}`
    : `## 新增或修订的时间线条目\n\n${delta}`;

  const result = await _compactLLM(input, compileTodayPrompt, resolvedModel, 450);
  atomicWrite(outputPath, result);
  writeTodayState(statePath, logicalDate, nextWatermark);
  return "compiled";
}
⚠️ 空 sessions 的两种现场：(1) 已有水位线 → 草稿是合法产物，保留不覆盖；(2) 从无水位线 → 无法证明 today.md 当前内容对应"今天"的真实摘要，清空陈旧内容避免显示错误旧稿。

三、compileDaily：日记蒸馏与 fingerprint 去重

compileDaily() 将"已经翻篇的那天"的 today.md 草稿蒸馏为 memory/daily/{logicalDate}.md。每天只落一次盘，通过 fingerprint 去重避免重复 LLM 调用。

输入优先级链

  1. 优先：时间线条目 — 从 session 摘要的 timeline 段提取该逻辑日的事件
  2. 次级：today.md 草稿 — 摘要中无时间线条目时使用
  3. 回落：session 摘要 — 新旧草稿都不可用时，直接用当天 session 摘要编译
  4. 零占位 — 三者都没有时不产文件，避免 daily/ 目录被空文件污染
// fingerprint 去重机制
const fp = computeFingerprint(timelineEvents.map(e => e.key));
// key = `${sessionId}:${updatedAt}:${index}:${timeLabel}:${sha1(body).slice(0,12)}`

// 检查 {date}.md.fingerprint 文件
if (已有的 fp === 新算出的 fp && 文件已存在) return "skipped";

// LLM budget: maxTokens = 100（极紧，保证 6 条 daily 装配 ≤ 1200 字符）
const result = await _compactLLM(input, compileDailyPrompt, resolvedModel, 100);
atomicWrite(outputPath, `## ${logicalDate}\n\n${result}\n`);
fs.writeFileSync(fpPath, fp);
⚠️ 执行顺序约束（关键！）：compileDaily 必须先于 compileToday 执行。compileDaily 读取"昨天最终版 today.md 草稿"，这份草稿在日期切换前仍躺在 todayMdPath 里。一旦 compileToday 先跑，日期切换会把 today.md 重置为空的新一天草稿，昨天的内容就永远读不到了。MemoryTicker._doDaily() 中严格保证了这个顺序。

四、检验一下（提取练习）

Q1. rollingSummary 的摘要预算是如何分配的？

A. 固定 500 字，不分段 B. 每轮 40 字，10 轮封顶 400 字，facts 占 30%，timeline 占 70% C. 按 token 数预算，不限字数 D. 每次调用动态计算，没有固定规则
看答案B。_rollingSummaryBudget(turnCount) 按轮数线性缩放：每轮 40 字，最多 400 字。其中 facts 段占 30%（约 120 字），timeline 段占 70%（约 280 字）。

Q2. compileToday 的水位线存储在哪个文件中？

A. memory.md B. today-state.json C. daily-state.json D. config.yaml
看答案B。today-state.json 存储 compileToday 的水位线（lastCompiledSummaryUpdatedAt 和 logicalDate）。daily-state.json 是每日流水线的进度状态（第 5 课详讲），config.yaml 是模型配置。

Q3. compileDaily 使用 fingerprint 去重的目的是什么？

A. 防止同一批数据重复调用 LLM 编译 B. 对日记内容做哈希签名以便检索 C. 验证 LLM 输出的完整性 D. 加密存储日记文件
看答案A。fingerprint 通过输入事件 key 的 SHA1 计算，如果与已有指纹匹配且文件存在，直接返回 "skipped"，跳过 LLM 调用。这在每日任务重跑或进程重启恢复时避免浪费 token。

Q4. 为什么 compileDaily 必须先于 compileToday 执行？

A. 纯粹是代码风格约定，顺序无关紧要 B. compileDaily 需要读取"昨天的 today.md"，而 compileToday 在日期切换后会清空 today.md——先跑 compileToday 会导致昨天的内容永久丢失 C. compileDaily 需要 compileToday 的输出作为输入 D. 为了减少 LLM 调用的并发冲突
看答案B。日期切换时 compileToday 会检测到 logicalDate 变化并清空 today.md。如果 compileDaily 还没有来得及读取昨天的草稿，这份数据就永久丢失了。因此 MemoryTicker._doDaily() 严格保证 compileDaily 在 compileToday 之前执行。
