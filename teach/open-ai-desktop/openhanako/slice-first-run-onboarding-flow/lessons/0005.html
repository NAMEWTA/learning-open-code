第 5 课 · 异常路径全景与综合测验 异常全景

🎯 本节目标：学完你能列出引导链路的 13 种异常场景及其处理策略，理解 HanaAgent "分层宽容"的异常处理哲学，并完成跨 4 课的综合测验。

引导链路穿越 Electron 主进程、Node.js Server、React 渲染进程三层，每个层都有特定的异常处理策略。本课汇总全部 13 种异常场景，形成完整的覆盖矩阵，并检验你对整个引导流程的综合理解。

二、分层宽容：HanaAgent 的异常处理哲学

💡 设计模式观察：HanaAgent 的异常处理策略是分层宽容——每层的异常不会向上传播阻断整个系统。播种阶段通过分类处置跳过损坏目录，UI 阶段通过表单校验阻塞非法输入，Server 阶段通过 HTTP 错误码返回结构化错误。这种策略与 Rust 的 Result<T, E> 模式有异曲同工之妙。

三种异常处置模式对比：

  • 硬阻断（Hard Fail）：preload 缺失、HANA_HOME 解析失败——这些是无法降级的致命错误，必须 exit
  • 软阻断（Soft Block）：连接测试失败、用户名未填——UI 层拒绝前进，但不影响系统其他部分
  • 静默降级（Silent Degrade）：locale 保存失败、非默认 agent 损坏——记录日志但继续运行

三、连接测试失败的完整状态机（重点回顾）

ProviderStep 的连接测试是引导链路中唯一存在阻塞式前置条件的步骤

// 关键代码：四个条件缺一不可
const nextDisabled = !(hasProvider && hasUrl && hasKey && connectionTested);

// 修改 Key 时自动重置
const onApiKeyInput = (val) => {
  setApiKey(val.replace(/[^\x20-\x7E]/g, '').trim());
  setConnectionTested(false);  // ← 旧测试结果立即失效
};

// 用户操作循环：
// 输入 Key → 测试 → 失败 → 修改 Key（自动重置）→ 重新测试 → 成功 → 继续

没有"跳过"选项。用户只能：返回 Step 1 修改身份、继续尝试不同 API Key、选择本地 Ollama（无需 Key）。

四、Splash 最少 3 秒与网络不可用的设计验证

如果网络不可用（用户离线或防火墙阻止），引导流程在 Step 2 会被阻断——testConnection 发不出 HTTP 请求或收到网络错误，进入 catch 块。此时 Splash 早已关闭，Onboarding 窗口正常显示，用户可以看到完整的 UI 界面，只是在 ProviderStep 卡住。

这不是 bug——这是设计意图。HanaAgent 作为本地优先的桌面应用，需要有效的 AI Provider 才能工作。网络不可用时，用户可以选择本地 Ollama（如果已安装），或者稍后再试。

五、Server 启动失败的特殊处理

main.cjs 中对 Server 启动失败有两种专门处理：

端口冲突处理

// main.cjs:1057-1065
// 解析 stderr 中的 PORT_IN_USE 错误
if (stderr.includes('PORT_IN_USE')) {
  throw new Error('端口已被占用，请关闭占用程序后重试');
}

不是直接抛出原始错误，而是解析后替换为中文友好消息。

模块解析失败重试

// main.cjs:1068-1082
// 最多重试 1 次（等待自动更新落地竞态）
if (err.code === 'ERR_MODULE_NOT_FOUND' && retries < 1) {
  await new Promise(r => setTimeout(r, 2000));
  return startServer(retries + 1);
}

自动更新过程中，新模块文件可能正在写入但尚未完成。2 秒等待+重试覆盖了这种竞态条件。

六、Splash 最少 3 秒与网络不可用的共存设计

Splash 窗口的 3 秒最小显示时间保证了即使 Server 在 500ms 内启动完成，用户也能看到一个平滑的加载过渡。但如果网络不可用（用户离线、防火墙阻止）：

  • Splash 正常显示 3 秒后关闭，进入 Onboarding 窗口
  • Onboarding 窗口的 LocaleStep、NameStep 都能正常完成（这些步骤不需要网络）
  • 用户在 ProviderStep 卡住——testConnection 发出 HTTP 请求失败，进入 catch 块
  • 用户可切换至本地 Ollama（如已安装），此时 isLocalProvider=true，连接测试访问 localhost

这不是 bug，而是设计意图。HanaAgent 作为本地优先的桌面应用，需要有效的 AI Provider。如果无法连接任何 Provider，后续所有 AI 功能（对话、自动化、记忆）都无法使用。引导流程在 Step 2 阻止用户继续，避免了"配置完发现不能用"的更差体验。

七、综合测验（跨课程）

Q1. ensureFirstRun 对默认 Agent 和非默认 Agent 的损坏处理策略有何不同？

A. 两者都备份损坏文件并重新播种 B. 默认 Agent 损坏→备份+重新播种；非默认→跳过不修复，不阻断启动 C. 两者都跳过不修复 D. 两者都弹出错误对话框要求用户处理
看答案B。这是"分类处置"策略的核心——默认 Agent 必须可用，所以修复它；非默认 Agent 损坏不影响基本功能，跳过即可。

Q2. 老用户（有 api_key 但无 setupComplete）走哪条路径？

A. 直接进入主界面 B. 走完整 7 步引导 C. createOnboardingWindow({ skipToTutorial: "1" })，直接跳到 Step 6 Tutorial D. 弹出对话框询问用户选择
看答案C。hasExistingConfig() 检测到 api_key 存在时，createOnboardingWindow 传入 skipToTutorial 参数，OnboardingApp 初始 step 设为 6，跳过 Locale/Name/Provider/Model/Theme/Workspace 步骤。

Q3. 以下哪个步骤不涉及 Server API 调用？

A. LocaleStep — saveLocale() 写入 config.yaml B. ProviderStep — testConnection() 验证 API Key C. ThemeStep — setTheme() 切换主题并通过 localStorage 持久化 D. WorkspaceStep — saveWorkspace() 写入 desk 配置
看答案C。ThemeStep 是纯前端步骤，通过 localStorage 和 theme-registry 模块管理，不调用任何 Server API。

Q4. 以下关于异常处理的描述，哪个是错误的？

A. locale 保存失败时静默忽略，使用默认 zh-CN 继续 B. API Key 连接测试失败时显示红色错误，提供"稍后配置"跳过按钮 C. 模型列表加载失败时不阻断流程，用户可手动输入模型 ID D. setupComplete 写入失败时抛出异常，通过 toast 通知用户
看答案B。ProviderStep 明确不提供跳过按钮。连接测试失败时"下一步"按钮保持 disabled，用户必须解决问题或切换 Provider（如本地 Ollama）。这是强制完成设计。
