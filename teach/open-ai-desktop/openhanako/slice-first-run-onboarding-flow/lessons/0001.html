第 1 课 · 引导系统架构：三步检测阶梯与数据目录播种 架构全景

🎯 本节目标：学完你能讲清 HanaAgent 首次运行引导的四个核心任务（自动播种/引导式配置/容错启动/状态标记），理解三步检测阶梯如何精准区分全新用户、老版本升级用户和已完成配置用户，以及 ensureFirstRun 的"分类处置"策略。

HanaAgent 是一款本地优先的 AI Agent 桌面应用。用户首次双击应用图标时，~/.hanako/ 目录完全为空——没有 agent 配置、没有 provider 凭证、没有模型选择。系统必须在零预置数据的前提下，完成自动播种、引导配置、容错启动和状态标记四项任务。整个流程跨越 Electron 主进程、Node.js 服务端、React 渲染进程三层。

一、启动入口：bootstrap.cjs 与应用的三层架构

应用从 desktop/bootstrap.cjs 开始，在 Electron 的 app 模块就绪前建立诊断基础设施：

// bootstrap.cjs 核心流程
// 1. 解析 HANA_HOME 环境变量 / 默认路径
const hanakoHome = resolveHanakoHome(process.env.HANA_HOME);
process.env.HANA_HOME = hanakoHome;

// 2. 注册全局错误钩子（uncaughtException / unhandledRejection）
process.on("uncaughtException", (err) => {
  if (err?.code === "EPIPE" || err?.code === "ERR_IPC_CHANNEL_CLOSED") return;
  recordProcessError("uncaughtException", err);
});

// 3. 加载 main.cjs
require(app.isPackaged ? "./main.bundle.cjs" : "./main.cjs");
💡 设计要点：bootstrap.cjs 先于 main.cjs 加载，确保即使 main.cjs 加载失败（如 preload 缺失、模块解析错误），也能写出诊断日志到 ~/.hanako/diagnostics/desktop-launch/，用户不会看到无信息的崩溃。

涉及的三层模块边界：

层级模块/文件职责
Electron 主进程desktop/main.cjs、desktop/bootstrap.cjs启动 server、创建 BrowserWindow、决定显示 Splash/Onboarding/Main
Node.js 服务端core/first-run.ts、server/routes/数据目录播种、Provider 连接测试、模型发现、配置持久化
React 渲染进程desktop/src/react/onboarding/7 步 Wizard UI、API 调用、步骤间状态管理

二、三步检测阶梯：精准区分用户类型

在 app.whenReady() 中，主进程通过三个条件分支决定走哪条路径：

app.whenReady().then(async () => {
  createSplashWindow();
  await startServer();
  ensureFirstRun(hanakoHome, productDir);

  const migratedSetupComplete = await migrateSetupCompleteViaServerIfNeeded();

  if (isSetupComplete() || migratedSetupComplete) {
    // 🟢 已完成引导 → 直接进入主界面
    createMainWindow();
  } else if (hasExistingConfig()) {
    // 🟡 老用户（有 api_key）→ 跳过配置，直接看教程
    createOnboardingWindow({ skipToTutorial: "1" });
  } else {
    // 🔴 全新用户 → 完整 7 步引导
    createOnboardingWindow();
  }
});

三个检测函数的实现细节：

1isSetupComplete()：读取 preferences.json 中的 setupComplete 字段。这是最优先的检测，标志着用户已完成完整的引导流程。
2hasExistingConfig()：检测 config.yaml 中是否已有有效 api_key。用正则 /api_key:\s*["']?[^"'\s]+/ 匹配。有 key 说明老用户已配置过，跳过填写直接看教程。
3migrateSetupCompleteViaServerIfNeeded()：判断 added-models.yaml 是否存在且含真实 api_key，通过 server 标记 setupComplete 来迁移老用户状态。
⚠️ 三步阶梯的精妙之处：检测顺序不可颠倒。setupComplete 是最高优先级的"硬标记"；hasExistingConfig 是"软检测"，只判断 api_key 存在与否；迁移检测是兼容逻辑，处理从旧版本升级的用户。这种分层检测确保了三类用户都能获得正确的体验。

三、数据目录播种：ensureFirstRun 的分类处置策略

ensureFirstRun()（core/first-run.ts）对 agent 目录采用"分类处置"而非 fail-fast：

→分类器：classifyAgentDirectoryForStartup() 仅通过 config.yaml 判断目录有效性——文件存在且可被 YAML 解析即为 valid。config_missing 和 config_unreadable 为 invalid。
→处置规则：默认 Agent（Hanako）损坏 → 备份损坏文件 + 重新播种；非默认 Agent 损坏 → 跳过不修复，记入 FirstRunReport.invalidAgentDirs，不阻断启动。
→播种内容：从产品模板目录复制 config.example.yaml，注入 desk/memory 默认值；按 locale 选择对应语言的 identity.md、ishiki.md 模板；同步 skills 目录。

四、播种后的目录树

~/.hanako/ ├── agents/ │ └── hanako/ ← 默认 Agent │ ├── config.yaml ← 从 lib/config.example.yaml 播种 │ ├── identity.md ← 从 lib/identity-templates/ 播种 │ ├── ishiki.md ← 从 lib/ishiki-templates/ 播种 │ ├── public-ishiki.md ← 对外意识模板 │ ├── pinned.md ← 空文件 │ ├── memory/ ← 记忆存储目录 │ ├── sessions/ ← 会话历史目录 │ ├── avatars/ ← 头像目录 │ └── desk/ ← 工作台配置目录 ├── user/ │ ├── preferences.json ← { primaryAgent: "hanako" } │ └── user-profile.md ← 空文件 ├── skills/ ← 从 skills2set/ 同步 └── diagnostics/ ← 启动诊断日志

五、检验一下

Q1. 三步检测阶梯的正确优先级顺序是什么？

A. hasExistingConfig → isSetupComplete → 迁移检测 B. isSetupComplete → hasExistingConfig → 迁移检测 C. 迁移检测 → isSetupComplete → hasExistingConfig D. 三者无优先级关系，并行检测
看答案B。setupComplete 是最高优先级的"硬标记"；hasExistingConfig 检测 api_key 存在性；迁移检测处理旧版本升级用户。

Q2. 非默认 Agent 目录的 config.yaml 损坏时，ensureFirstRun 如何处理？

A. 备份损坏文件并重新播种该 agent B. 跳过不修复，记入 FirstRunReport，不阻断启动 C. 弹出错误对话框要求用户手动修复 D. 删除整个 agent 目录
看答案B。分类处置策略对非默认 agent 采取"跳过并报告"，只修复默认 Agent。保证即使历史脏数据存在，应用也能启动。

Q3. bootstrap.cjs 为什么在 main.cjs 之前加载？

A. 为了提高启动速度 B. 确保即使 main.cjs 加载失败，也能写出诊断日志，用户不会看到无信息的崩溃 C. 因为 Electron 的加载顺序要求 D. 为了加载第三方插件
看答案B。bootstrap.cjs 注册了全局错误钩子和 HANA_HOME 解析，诊断基础设施早于业务代码建立，确保任何阶段的崩溃都能被记录。

Q4. hasExistingConfig() 用什么方式检测 api_key？

A. 调用 API 接口验证 key 的有效性 B. 用正则表达式 /api_key:\s*["']?[^"'\s]+/ 匹配 config.yaml C. 读取 preferences.json 中的 hasApiKey 字段 D. 检查环境变量中的 API_KEY
看答案B。hasExistingConfig() 通过正则匹配 config.yaml 中是否存在非空的 api_key 值。这仅是存在性检测，不做有效性验证。
