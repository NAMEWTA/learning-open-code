第 3 课 · Provider 配置关卡：连接测试状态机与强制完成设计 配置核心

🎯 本节目标：学完你能讲清 ProviderStep 的三子流程（预设选择/连接测试/配置保存）、连接测试的状态机流转、以及"必须测试通过才能下一步"的强制完成设计意图。同时理解 ModelStep 的模型发现机制和三种模型（chat/utility/utility_large）的用途。

ProviderStep 是整个引导流程中最复杂也是唯一存在阻塞式前置条件的步骤。用户必须成功完成连接测试才能继续——没有"跳过"按钮。这种强制设计背后有深刻的用户体验考量。

一、ProviderStep 的三个子流程

1Provider 预设选择：内置 OpenRouter、Ollama、OpenAI、Anthropic 等提供商预设。每个预设定义了 base_url、api 类型（openai-completions / anthropic-messages）。也支持自定义 Provider 输入。
2连接测试：通过 POST /api/providers/test 验证 API Key 和 URL 的有效性。这是进入下一步的必经关卡。
3配置保存：通过 PUT /api/agents/hanako/config 写入 Provider 凭证和 api 类型。

二、连接测试状态机

这是引导链路中最关键的状态机。ProviderStep 使用四个状态变量精确控制按钮行为：

// ProviderStep 的核心禁用逻辑
const hasKey = !!apiKey || isLocalProvider;       // 有 Key 或是本地 Provider
const hasProvider = !!providerName;                // 已选择 Provider
const hasUrl = !!providerUrl;                      // 已填入 URL
const testBtnDisabled = !(hasProvider && hasUrl && hasKey);
const nextDisabled = !(hasProvider && hasUrl && hasKey && connectionTested);
//                                                        ^^^^^^^^^^^^^^^^
//                              必须 connectionTested === true 才启用"下一步"

连接测试的状态流转：

用户输入 API Key → 点击"测试连接" │ ├─ POST /api/providers/test { base_url, api, api_key } │ │ │ ├─ 200 { ok: true } │ │ → connectionTested = true（绿灯） │ │ → setTestStatus({type:'success', text:'连接成功'}) │ │ → "下一步"按钮变为可用 │ │ │ ├─ 200 { ok: false } │ │ → connectionTested = false（红灯） │ │ → "下一步"按钮保持 disabled │ │ │ └─ 网络错误 → catch 块 │ → connectionTested = false │ └─ 用户修改 API Key → onApiKeyInput() 自动调用 setConnectionTested(false) → 之前的测试结果立即失效

三、修改 Key 时自动重置测试状态

这是一个容易被忽略但非常重要的细节：

// ProviderStep.tsx:109-114
const onApiKeyInput = useCallback((val: string) => {
  const cleaned = val.replace(/[^\x20-\x7E]/g, '').trim(); // 清理非 ASCII
  setApiKey(cleaned);
  setConnectionTested(false);       // ← 任何修改都使之前的测试结果失效
  setTestStatus({ type: '', text: '' });
}, []);
💡 设计意图：如果用户修改了 Key 而不重置测试状态，可能出现"用旧 Key 测试通过，但实际使用的是新 Key"的不一致状态。重置确保 Key 与测试结果始终配对。

四、为什么没有"跳过"按钮？

⚠️ ProviderStep 不提供"跳过"按钮是一种有意设计——如果没有有效的 AI Provider，后续所有功能（对话、自动化、记忆）都无法使用。强制用户完成连接测试可以避免"辛辛苦苦配置完 7 步，进入主界面后发现无法对话"的糟糕体验。

例外：本地 Ollama 用户——系统自动检测 preset.local === true，此时 isLocalProvider 为 true，不需要 API Key 即可测试连接。

五、testConnection 的 API 实现

// onboarding-actions.ts
export async function testConnection({ hanaFetch, providerUrl, providerApi, apiKey }) {
  const res = await hanaFetch('/api/providers/test', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      base_url: providerUrl,
      api: providerApi,        // 'openai-completions' | 'anthropic-messages'
      api_key: apiKey,
    }),
  });
  const data = await res.json();
  if (data.ok) return { ok: true, text: t('onboarding.provider.testSuccess') };
  return { ok: false, text: t('onboarding.provider.testFailed') };
}

六、Step 3：模型发现与选择（ModelStep）

ModelStep 在 Provider 连接成功后自动加载可用模型列表，用户需选择三种模型：

模型类型用途说明
主对话模型 (chat)用户日常对话必选，写入 config.yaml → models.chat
辅助模型 (utility)轻量任务（摘要、分类等）可选，写入 preferences.json → models.utility
大型辅助模型 (utility_large)复杂推理任务可选，写入 preferences.json → models.utility_large
// ModelStep 挂载时自动加载模型列表
useEffect(() => {
  const doLoad = async () => {
    if (preview) {
      setFetchedModels([{id:'model-a'}, {id:'model-b'}]);
      return;
    }
    setModelLoading('正在加载模型列表...');
    const result = await loadModelsAction({
      hanaFetch, providerName, providerUrl, providerApi, apiKey
    });
    if (result.error) {
      setModelLoading(result.error);   // 显示错误但不阻断！用户可手动输入
      return;
    }
    setFetchedModels(result.models);
  };
  doLoad();
}, [providerName]);
💡 模型列表加载失败不阻断：与连接测试不同，模型列表加载失败仅在 UI 中显示错误消息，用户仍可手动输入模型 ID 并继续。这是一种务实的降级策略。

七、检验一下

Q1. "下一步"按钮可用的必要条件是什么？

A. 只要填了 API Key 就可以 B. hasProvider && hasUrl && hasKey && connectionTested 全部为 true C. 只要选择了 Provider 预设就可以 D. 等待 5 秒后自动启用
看答案B。四个条件缺一不可。connectionTested 是最关键的——必须成功完成连接测试，不能跳过。

Q2. 用户修改 API Key 后会发生什么？

A. 自动使用新 Key 重新测试连接 B. connectionTested 自动重置为 false，测试状态被清空 C. 保留旧测试结果，但提示 Key 已变更 D. 自动切换到自定义 Provider 模式
看答案B。onApiKeyInput() 中调用 setConnectionTested(false) 和 setTestStatus({ type: '', text: '' })，确保旧 Key 的测试结果不会与新 Key 混淆。

Q3. 模型列表加载失败时 ModelStep 如何处理？

A. 显示错误并阻止用户继续 B. 在 UI 中显示错误消息，但允许用户手动输入模型 ID 继续 C. 自动重试 3 次后放弃 D. 使用预设的默认模型列表
看答案B。与连接测试不同，模型加载失败不阻断流程。setModelLoading(errorMsg) 显示错误文字，用户可手动输入模型 ID。

Q4. 为什么 ProviderStep 不提供"跳过"按钮？

A. 技术上实现不了 B. 强制用户完成 Provider 配置，避免进入主界面后发现无法对话 C. 因为 UI 空间不够放跳过按钮 D. 因为法律合规要求
看答案B。没有有效 Provider 时后续所有功能无法使用。强制完成连接测试是"第一印象"保护——避免用户经历 7 步引导后才发现无法使用。
