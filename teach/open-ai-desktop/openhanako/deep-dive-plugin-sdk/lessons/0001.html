第 1 课 · 插件SDK四包架构总览：依赖关系与职责边界 全景地图

🎯 本节目标：学完你能画出四包的依赖关系图，讲清为什么必须拆成4个包而非1个，辨识 Facade / Adapter / Dependency Inversion 三种设计模式在代码中的落地位置。

HanaAgent 插件体系由 4 个 npm 包构成：plugin-protocol、plugin-sdk、plugin-runtime、plugin-components。它们拆分的根本原因是一个硬约束：插件 UI 运行在 WebView/iframe 沙箱中，服务端逻辑运行在 Node.js 进程里——两种运行时拥有完全不同的全局对象、安全边界和依赖解析规则。

💡 关键设计原则：plugin-protocol 是唯一被所有四个包共享的模块。它只包含 TypeScript 类型定义、字符串常量和消息解析/校验函数，不引入 react、fs、postMessage 等任何运行时依赖——确保在浏览器和 Node.js 中都可以零成本导入。

三、不分离会怎样？四个致命问题

1浏览器端无法使用：Node 专用 API（fs、path、process、EventBus）在浏览器不可用，打包工具报错或产生巨大 polyfill。
运行时冲突
2服务端代码泄露到 iframe：工具注册、文件系统操作、session 管理等能力暴露在前端，违反安全边界。
安全边界
3打包体积膨胀：iframe 插件 UI 被迫携带不需要的 Node 运行时依赖。
性能浪费
4类型定义污染：开发者在一个 import 中同时看到浏览器 API 和 Node API 的类型提示，IDE 自动补全混乱。
开发体验

模式落地点作用

包名核心文件规模关键导出

规模对比：protocol（341行）定义契约；sdk（541行）轻量封装 iframe 通信；runtime（1537行）承载服务端全部能力——工具注册、会话管理、模型调用、任务调度；components（634行）提供 UI 控件和主题。四包的规模与其职责重量成正比。

六、渐进式复杂度——插件开发的"阶梯"

四包设计支持开发者按需选择复杂度级别：

L1极简插件：只需 tools/*.js + manifest.json，不 import 任何 SDK 包。宿主通过 lifecycle 传入 ctx 对象，直接使用底层 bus 请求。零依赖，绝对可移植。
适合：单工具插件、概念验证
L2标准插件：使用 plugin-runtime 的 definePlugin/defineTool/defineCommand。获得类型安全和高阶 API（createSession、generateImage 等）。
适合：绝大多数插件
L3全功能插件：L2 + routes + providers + extensions + iframe UI（使用 plugin-sdk + plugin-components）。完整的服务端能力 + 自定义 UI。
适合：复杂业务插件、需要自定义界面的场景

这种渐进式设计确保开发者不会被不必要地强制引入复杂依赖——想要快速交付一个工具？L1 就够了。想要完整交互体验？L3 全部支持。

七、架构决策的权衡：postMessage 线协议

为什么选择 postMessage 而非 JS bridge（如 Electron 的 contextBridge）？这是四包设计中最关键的跨平台决策：

方案桌面 ElectronMobile PWA浏览器客户端
postMessage✅ 原生支持✅ 原生支持✅ 原生支持
contextBridge✅ Electron 专用❌ 不可用❌ 不可用

postMessage 的代价是所有数据必须可序列化（不能传函数、Promise、WeakMap 等），但对于插件 UI 通信而言，这个限制恰好强制了协议即边界的设计原则——没有隐式依赖，没有内存共享，所有交互都通过 protocol 中明确定义的消息类型。

八、检验一下

Q1. 为什么必须拆成 4 个包而不是 1 个？

A. 因为 npm 不允许单包超过 1MB B. 因为 WebView 和 Node 是两种运行时，全局对象和依赖解析规则完全不同 C. 因为 4 个包由 4 个不同团队维护 D. 为了通过 monorepo 的 CI 并行构建加速
看答案B。插件 UI 运行在 WebView/iframe，服务端运行在 Node.js——window/document/postMessage 在 Node 中不存在，fs/path/EventBus 在浏览器中不存在。单包会导致运行时崩溃、安全边界模糊、打包体积膨胀。

Q2. plugin-protocol 为什么能做到"零运行时依赖"？

A. 它使用了 WebAssembly 编译到二进制 B. 它只包含 TypeScript 类型定义、字符串常量和纯函数校验逻辑，不 import react/fs 等 C. 它通过动态 import() 按需加载依赖 D. 它把所有依赖都设为 peerDependencies
看答案B。plugin-protocol 只导出 interface、const 字符串、parsePluginUiMessage() 等纯函数。不依赖 react、fs、postMessage 等任何运行时库，可在浏览器和 Node.js 中零成本导入。

Q3. BusProxy 使用了哪种设计模式来保护 EventBus？

A. Singleton 单例模式——全局只有一个 BusProxy 实例 B. Adapter 适配器模式——将 EventBus 适配为带权限控制的代理层 C. Observer 观察者模式——所有插件订阅同一个事件流 D. Factory 工厂模式——每次调用动态创建新的 Bus 实例
看答案B。createPluginBusProxy() 创建代理层，在 emit/subscribe/request 时检查 accessLevel 和 permissions。restricted 插件无法 emit llm_usage 事件，无法注册 bus handler。

Q4. 极简插件（L1）如何获得 ctx 对象而不 import 任何 SDK 包？

A. 通过 WebSocket 连接向宿主请求注入 B. 宿主通过 lifecycle 函数参数传入 ctx，插件直接使用不依赖任何 npm 包导入 C. 需要先 import @hana/plugin-runtime 再调用 getContext() D. ctx 存储在浏览器 localStorage 中，插件从那里读取
看答案B。极简插件不 import SDK 包——宿主加载插件时通过 lifecycle（如 onload(ctx)）或 tool execute(input, ctx) 的参数直接传入 ctx 对象。这种方式零依赖、绝对可移植，但放弃类型安全和高阶 API。
