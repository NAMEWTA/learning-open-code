第 5 课 · 完整插件开发实战与综合测验 终课

🎯 本节目标：学完你能从零开始写出一个完整插件的 manifest.json + iframe UI + 服务端工具，掌握 7 步发布检查清单，并通过综合测验检验全链路理解。

前 4 课我们拆解了四包的原理和机制。这一课把它们串起来——从 manifest.json 声明，到 iframe UI 调用 hana SDK，到服务端工具注册，再到生产分发检查。

一、开发流程：6 步从零到完整插件

1manifest.json 声明：定义 id、capabilities（resource.read/write）、network.allowedHosts、accessLevel（full-access/restricted）、configSchema。这是插件的"宪法"——宿主据此分配权限。
插件根目录 / manifest.json
2服务端入口 index.js：使用 definePlugin() 注册生命周期（onload/onunload），defineTool() 注册工具（含 sessionPermission 声明），defineCommand() 注册命令。
index.js / index.ts
3iframe UI 页面：import hana SDK，调用 hana.ready() 通知宿主加载完成，通过 hana.api.fetch() 调用同插件后端路由，使用 plugin-components 渲染 UI。
ui/index.html + ui/app.tsx
4工具实现：在 execute(input, ctx) 中使用 ctx.resources.read() 操作文件，ctx.network.fetch() 调用外部 API，ctx.stageFile() 注册输出文件。所有操作经过 ResourceIO 审计。
tools/*.js
5本地测试：monorepo 内使用 --sdk-mode workspace，通过 symlink 解析 SDK 依赖。在 Hana 桌面中加载插件，检查诊断信息和日志。
开发环境
6生产分发：选择 bundle 服务端代码 / 附带 node_modules / 不使用 SDK helper 三种方式之一。验证在 ${HANA_HOME}/plugins 目录下可独立加载。
生产环境

二、正确模式 vs 反模式速查

场景正确做法反模式
同插件 API 调用hana.api.fetch('api/translate', {...}) —— 自动附加 Surface Session 头硬编码 pluginId 路径；复用 iframeTicket
外部 API 调用后端路由代理 + ctx.network.fetch() + manifest 声明 allowedHosts浏览器端直接 fetch，API key 暴露在 JS bundle
文件操作ctx.resources.read/write() —— 经过 ResourceIO 审计直接用 fs.writeFileSync()，绕过权限
工具权限声明 sessionPermission：readOnly / plugin_output / external_side_effect不声明权限，Auto 模式无法精确决策
资产文件assets/ 目录 + hana.assets.url() + HttpOnly cookie 保护自建路由服务静态文件，增加复杂度
可见性visibility: 'plugin_private' —— 内部 Agent/Session 不污染主列表使用 public 可见性，内部对象暴露给用户

三、7 步发布检查清单

  1. 检查服务端入口点：index.js/ts、tools/*、routes/*、providers/*、extensions/* 及所有导入链中无未解析的裸包 import
  2. 验证 SDK 解析：从 ${HANA_HOME}/plugins 加载（非 monorepo 环境），确认无 Cannot find package 错误
  3. sha256 校验：marketplace 的 release 分发要求提供 zip 的 sha256，Hana 安装时校验完整性
  4. 版本管理：versions[] 支持多 SemVer 共存，Hana 自动选择最高兼容版本，禁止隐式降级
  5. 回滚机制：旧版本备份到 .bak，安装记录写入 ${HANA_HOME}/plugin-installs.json
  6. 冒烟测试：用安装用户收到的确切文件夹或 zip 进行端到端测试
  7. 权限最小化：manifest 中的 capabilities、network.allowedHosts、accessLevel 只声明必需的最小集合

四、全链路代码骨架示例

// === manifest.json ===
{
  "id": "my-translator",
  "capabilities": ["resource.read"],
  "network": { "allowedHosts": ["api.translate.com"] },
  "accessLevel": "full-access"
}

// === index.js (服务端入口) ===
import { definePlugin, defineTool } from '@hana/plugin-runtime';
export default definePlugin({
  id: 'my-translator',
  onload(ctx) { ctx.log.info('loaded'); },
  tools: [
    defineTool({
      name: 'translate',
      sessionPermission: { kind: 'plugin_output' },
      async execute(input, ctx) {
        const res = await ctx.network.fetch(
          'https://api.translate.com/v2',
          { method: 'POST', body: JSON.stringify(input) }
        );
        return await res.json();
      },
    }),
  ],
});

// === ui/app.tsx (iframe UI) ===
import { hana } from '@hana/plugin-sdk';
import { HanaThemeProvider, Button } from '@hana/plugin-components';
hana.ready();
// 调用同插件后端 translate 路由
const result = await hana.api.fetch('translate', {
  method: 'POST', body: JSON.stringify({ text: 'hello' }),
});

五、综合测验（提取练习）

Q1. 从 manifest.json 到 iframe UI 的完整链路中，哪个环节负责将用户文件操作重定向到 ResourceIO 审计层？

A. manifest.json 中的 capabilities 声明——静态权限检查 B. plugin-protocol 中的 parsePluginUiMessage()——消息格式校验 C. plugin-runtime 中的 createPluginResources() + createPluginContext()——构建带权限的资源代理 D. plugin-components 中的 HanaThemeProvider——CSS 变量注入
看答案C。createPluginContext() 调用 createPluginResources() 创建受 manifest capabilities 限制的资源代理。所有 ctx.resources.read/write() 调用通过此代理，自动进行权限校验和审计日志记录。capabilities 只是声明，ResourceIO 代理才是执行点。

Q2. 生产分发时，workspace 模式插件直接复制到 plugins 目录会怎样？

A. 正常运行——workspace 模式自动降级为 bundled 模式 B. 加载失败——依赖 repo root node_modules 的 symlink，生产环境找不到包 C. Hana 会自动运行 npm install 补全依赖 D. 插件会以 restricted 模式降级运行
看答案B。workspace 模式通过 monorepo symlink 解析 SDK 依赖（指向 repo root 的 node_modules）。将插件目录直接复制到 ${HANA_HOME}/plugins 后，symlink 断裂，Node resolver 无法找到 @hana/plugin-runtime 等包。

Q3. 以下哪个不是 HanaAgent 插件体系的正确设计原则？

A. 插件 UI 运行在 iframe 沙箱中，服务端逻辑运行在 Node.js 中 B. plugin-protocol 是所有包的共享契约，零运行时依赖 C. BusProxy 采用默认允许策略，只拦截已知危险事件 D. 主题系统通过四级回退保证从完全自定义到零配置的渐进式兼容
看答案C。BusProxy 采用"默认拒绝"策略，而非"默认允许"。restricted 插件无法 emit llm_usage 事件，handle 方法仅在 full-access 时暴露。Object.freeze() 确保插件无法动态添加属性绕过限制。

Q4. 一个插件的工具需要在每次执行后把结果写入 session 文件。最安全的做法是？

A. 在 execute() 中直接 fs.writeFileSync(sessionDir + '/result.json', data) B. 使用 ctx.stageFile() + sessionPermission: { kind: 'plugin_output' } C. 通过 postMessage 把数据发回 iframe，由 UI 层写入 localStorage D. 使用 ctx.network.fetch() 把数据上传到第三方云存储
看答案B。正确做法：(1) 在 defineTool 中声明 sessionPermission: { kind: 'plugin_output', describeSideEffect: () => ({ kind: 'session_file_output' }) }；(2) 在 execute 中将数据写入 ctx.dataDir；(3) 调用 ctx.stageFile() 注册为 session 文件。这样 Auto 模式可以精确决策，所有操作经过审计。
