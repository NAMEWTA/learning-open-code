第 4 课 · plugin-components 与主题系统：组件矩阵与四层回退设计 UI 体系

🎯 本节目标：学完你能列出 11 个 React 组件的分类和关键属性，画出主题数据的跨层流向，讲清四级回退中每一级的优先级和使用场景。

@hana/plugin-components 是 iframe 插件的 UI 工具箱——约 634 行，分 3 个文件，提供 11 个 React 组件和 10 套内置主题。设计哲学是零外部依赖（除 React）和完整 ARIA 支持。

💡 设计亮点：Select 组件自行实现下拉面板（不依赖 headless UI），所有样式通过 CSS 类名控制（hana-plugin-* 前缀），组件不内联关键样式——允许宿主 CSS 覆盖。这种"样式外置"策略让同一套组件可以在不同宿主的视觉风格下呈现一致交互。

三、四级回退链（从高到低）

1显式自定义 token（最高优先级）：传入 HanaThemeTokens 对象到 theme prop。完全覆盖所有内置值，适合插件需要独特品牌的场景。
themeStyleFor(customTokens)
2命名 Hana 主题：mode="hana" + theme="warm-paper"（或其他 9 个内置名）。使用内置 palette 中的 16 个 token。
HANA_BUILT_IN_THEMES[themeId]
3宿主 CSS 变量：mode="inherit"。Iframe 页面使用宿主注入的 CSS 变量，如 var(--hana-plugin-bg)。自动跟随宿主主题。
themeStyleFor → return {}（跳过 token 计算）
4SDK 默认值（最低优先级）：styles.css 中的回退值。当没有任何上层配置时生效，保证组件始终有基础样式。
@hana/plugin-components/styles.css

所有主题统一使用 16 个 token：bg、bgCard、accent、accentHover、accentLight、text、textLight、textMuted、border、danger、radiusInput、radiusCard、fontUi、fontSerif、fontMono。通过 TOKEN_TO_CSS_VAR 映射为 CSS 变量如 --hana-plugin-bg。

⚠️ 回退陷阱：mode="inherit" 时 themeStyleFor() 返回空对象 {}——这意味着完全依赖宿主 CSS 变量。如果宿主没有注入这些变量且 styles.css 未加载，组件将没有任何样式。确保 mode="inherit" 的页面正确引用了基础样式表。

四、themeStyleFor() 实现原理

function themeStyleFor(mode, theme) {
  if (mode === 'inherit') return {};          // 第3级：宿主CSS变量

  const tokens = resolveThemeTokens(mode, theme);
  // resolveThemeTokens 逻辑：
  //   字符串 themeId → HANA_BUILT_IN_THEMES[themeId]（第2级）
  //   对象 theme → 直接使用（第1级）
  //   mode='hana' 无 theme → warm-paper（默认）
  //   mode='custom' 无 theme → undefined

  const css = {};
  for (const [key, cssVar] of TOKEN_TO_CSS_VAR) {
    if (tokens?.[key]) css[cssVar] = tokens[key];
  }
  return css;  // 通过 React style prop 注入到容器 div
}

返回的 CSS 样式对象通过 React 的 style prop 注入到容器 div——因为 style prop 的优先级高于外部样式表，这天然实现了第 1/2 级 > 第 4 级的覆盖关系。而 mode="inherit" 返回空对象时不添加任何 style，完全退回到 styles.css 和宿主 CSS 变量。

五、检验一下

Q1. 为什么 Select 组件要自实现下拉而非依赖 headless UI？

A. headless UI 性能太差 B. 零外部依赖策略——最小化 iframe bundle 体积，避免依赖版本冲突 C. headless UI 不支持 ARIA D. Select 只需要 5 行代码，不值得引入库
看答案B。plugin-components 的设计目标是零外部依赖（除 React）。iframe 插件的 bundle 体积直接影响加载性能，每增加一个依赖都增加用户等待时间，且可能与宿主环境中的依赖版本冲突。

Q2. mode="inherit" 在四级回退中处于第几级？

A. 第 1 级——最高优先级 B. 第 2 级——次于自定义 token C. 第 3 级——次于内置主题，由宿主 CSS 变量控制 D. 第 4 级——作为最后的回退值
看答案C。优先级：自定义 token > 命名 Hana 主题 > mode="inherit"（宿主 CSS 变量）> styles.css 默认值。mode="inherit" 时返回空对象，完全依赖宿主注入的 CSS 变量。

Q3. 所有 10 个内置主题共享多少个 CSS token？

A. 8 个——bg、text、accent、border、danger、font、radius、shadow B. 12 个——含 spacing 相关的 token C. 16 个——含 bgCard、accentHover、accentLight、textLight、textMuted、radiusInput、radiusCard、fontUi、fontSerif、fontMono 等扩展 token D. 20 个——每种组件都有独立 token
看答案C。16 个统一 token：bg、bgCard、accent、accentHover、accentLight、text、textLight、textMuted、border、danger、radiusInput、radiusCard、fontUi、fontSerif、fontMono。通过 TOKEN_TO_CSS_VAR 映射为 CSS 自定义属性。

Q4. 如果插件在 mode="hana" theme="warm-paper" 下又传入了自定义 HanaThemeTokens，最终渲染的主题由谁决定？

A. warm-paper——mode="hana" 的优先级最高 B. 自定义 HanaThemeTokens——它处于第 1 级回退，完全覆盖 mode/theme 的内置配置 C. 两者合并——warm-paper 的未覆盖 token 保留 D. styles.css 默认值——所有显式配置都被忽略
看答案B。四级回退中，传入 HanaThemeTokens 对象到 theme prop 是第 1 级（最高优先级），完全覆盖内置主题。themeStyleFor() 检测到 theme 是对象（非字符串）时直接使用，不查询 HANA_BUILT_IN_THEMES。
