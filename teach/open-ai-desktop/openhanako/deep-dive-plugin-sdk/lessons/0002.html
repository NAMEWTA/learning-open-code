第 2 课 · plugin-protocol 协议层：消息格式、错误码与能力标识 协议宪法

🎯 本节目标：学完你能写出 PluginUiMessage 的 4 种 kind 及其语义，列出 7 种错误码和 8+5 种能力标识，讲清 parsePluginUiMessage() 的七步校验流程。

@hana/plugin-protocol 是整个插件体系的"宪法"——约 341 行，定义四种核心契约：线协议常量、消息结构、错误码体系、能力标识。没有运行时依赖，只有 interface、const、纯函数。

一、PluginUiMessage —— 一切通信的载体

interface PluginUiMessage {
  protocol: 'hana.plugin.ui';     // 协议标识
  version: 1;                     // 版本号（兼容检测）
  id?: string;                    // request/response/error 必须有 id
  kind: 'event' | 'request' | 'response' | 'error';
  type: string;                   // 如 'toast.show'、'ui.resize'
  payload?: unknown;              // 携带数据
  error?: PluginUiError;          // error kind 时必须包含
}

这个结构类似简化版的 JSON-RPC，专为插件 UI 通信设计。四种 kind 的语义：

Kind方向需要 id？典型 type
event单向推送，无需响应否hana.ready、ui.resize、hana.theme.changed
request请求-响应模式是（用于响应匹配）toast.show、resource.open
response对 request 的应答是（与请求一致）对应请求的 type
error请求失败是携带 error.code + error.message

二、7 种错误码体系

错误码含义触发场景
BAD_MESSAGE消息格式不合法JSON 解析失败、protocol 字段不匹配
BAD_REQUEST请求参数无效缺少必填字段、type 为空
TIMEOUT请求超时默认 10 秒内未收到响应
NOT_FOUND资源不存在请求的 resource 或 route 未找到
FORBIDDEN权限不足restricted 插件调用受限事件
UNSUPPORTED不支持的能力宿主未实现请求的 capability
INTERNAL内部错误未预期的运行时异常
💡 设计洞察：这 7 个错误码覆盖了从序列化层（BAD_MESSAGE）到权限层（FORBIDDEN）再到能力协商层（UNSUPPORTED）的完整纵深。不是随意定义——每个错误码对应一个具体的防御层。

三、13 种 Capability 能力标识

8 种 UI 宿主能力：toast、resize、clipboard.write、external.open、resource.open、resource.pick、resource.requestAccess、host.request —— 控制 iframe 插件可以向宿主请求哪些 UI 操作。

5 种 Resource 能力：resource.read、resource.write、resource.delete、resource.list、resource.watch —— 在 manifest 中声明，决定插件对文件系统资源的访问范围，由 ResourceIO 层强制执行。

四、七步校验流程 —— parsePluginUiMessage()

1检查 object 类型：非对象或 null → BAD_MESSAGE。防御 JSON.parse 产生非预期类型。
2校验 protocol 字段：必须等于 'hana.plugin.ui'。防止其他来源的消息混入。
3校验 version 字段：当前仅接受 version === 1。为未来协议升级预留版本协商空间。
4校验 kind 合法性：必须是 event、request、response、error 之一。
5校验 type 非空：type 是消息路由的唯一标识，不能为空字符串。
6event 外的消息必须有 id：request/response/error 必须携带 id 用于匹配。
7error kind 必须包含合规 error 对象：error.code 和 error.message 缺一不可。

任何一步不通过都返回 { ok: false, error }，而不是抛出异常——这是函数式错误处理风格，确保调用方始终拿到明确结果。

// 解析结果的类型
type ParseResult =
  | { ok: true; message: PluginUiMessage }
  | { ok: false; error: { code: string; message: string } }

五、实战：消息校验的防御层级

理解 parsePluginUiMessage() 为什么分 7 步而非 2 步（JSON 解析 + 类型检查），看这个攻击场景：

// 恶意 iframe 可能发送：
// 场景 A：protocol 字段伪装
{ protocol: 'other-app.protocol', version: 1, kind: 'event', type: 'malicious' }
// → 第 2 步 protocol 校验拦截

// 场景 B：无 id 的 request（可能重放攻击）
{ protocol: 'hana.plugin.ui', version: 1, kind: 'request', type: 'toast.show' }
// → 第 6 步 id 缺失拦截

// 场景 C：error 但无合规 error 对象
{ protocol: 'hana.plugin.ui', version: 1, kind: 'error', id: 'e1', type: 'fail', error: 'oops' }
// → 第 7 步 error 对象结构校验拦截

每一步都是独立的防御线——攻击者必须同时绕过 7 层校验才能让非法消息进入业务逻辑。这种纵深防御策略是安全关键系统的基础设计原则。

六、检验一下

Q1. 以下哪条消息会被 parsePluginUiMessage() 拒绝？

A. { protocol:'hana.plugin.ui', version:1, kind:'event', type:'hana.ready' } B. { protocol:'hana.plugin.ui', version:1, kind:'request', type:'toast.show' } C. { protocol:'hana.plugin.ui', version:2, kind:'request', id:'r1', type:'toast.show' } D. { protocol:'hana.plugin.ui', version:1, kind:'error', id:'e1', type:'fail', error:{code:'TIMEOUT',message:'...'} }
看答案B 和 C。B 缺少 id（request 必须有 id），C 的 version=2 不被当前版本接受。

Q2. 7 种错误码覆盖了几个防御层级？

A. 2 层——序列化层和应用层 B. 3 层——序列化层、权限层、能力协商层 C. 4 层——序列化层、路由层、权限层、内部错误层 D. 错误码彼此无关，没有分层设计
看答案B。BAD_MESSAGE/BAD_REQUEST 在序列化+验证层，FORBIDDEN 在权限层，UNSUPPORTED 在能力协商层，NOT_FOUND 在路由层，TIMEOUT 在传输层，INTERNAL 在内部错误层——共 3 个明确的防御层级。

Q3. resource 类 capability 有多少种？

A. 3 种——read、write、delete B. 5 种——read、write、delete、list、watch C. 7 种——含 mount/unmount 操作 D. 8 种——与 UI capability 数量相同
看答案B。5 种 resource capability：read、write、delete、list、watch。加上 8 种 UI capability，共 13 种能力标识。

Q4. 为什么 parsePluginUiMessage() 使用 7 步校验而非 2 步（JSON 解析 + 类型检查）？

A. 为了代码可读性——7 个 if 比 2 个 if 更容易理解 B. 纵深防御——每一步是一个独立防线，攻击者必须同时绕过所有 7 层。protocol/version/kind/type/id/error 各有独立的校验逻辑 C. 为了性能优化——7 步校验可以提前退出 D. 因为 TypeScript 编译器要求每个字段单独校验
看答案B。这是纵深防御（defense in depth）策略。如果只有 2 步，恶意消息只需绕过 JSON 解析和类型检查。7 步各自校验不同的攻击面：协议伪装、版本不兼容、请求重放、错误对象注入等。
