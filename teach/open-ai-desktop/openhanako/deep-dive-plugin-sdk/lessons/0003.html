第 3 课 · plugin-sdk 与 plugin-runtime：双端 API 全景与 PluginContext 注入 API 地图

🎯 本节目标：学完你能列出 hana 对象的 9 组 API 及其通信模式，讲清 plugin-runtime 的 7 大类导出，画出 PluginContext 从构建到注入的完整流程。

如果说 protocol 是"宪法"，那么 plugin-sdk（~541 行）和 plugin-runtime（~1537 行）就是"政府机构"——前者是 iframe 端的瑞士军刀，后者是 Node 端的重型武器。两者共享 protocol 类型，但服务于完全不同的运行时。

三、PluginContext 注入机制 —— 从构建到注入

// 1. PluginManager._loadPlugin() 中调用 createPluginContext()
const ctx = createPluginContext({
  pluginId, pluginDir, dataDir, bus,
  accessLevel, permissions, capabilities,
  network, resourceIO, resourceWatch,
  configSchema, logSink, runtimeContext
});

// 2. 注入到插件条目
entry.ctx = ctx;

// 3. _activatePluginEntry 中运行时注入
const instance = new PluginClass();
instance.ctx = entry.ctx;        // ← 插件实例获得 context
instance.register = (fn) => ...  // ← disposable 注册
instance.onload();               // ← 触发生命周期

工具定义的类型安全示例

// plugin-runtime 的 defineTool 提供完整类型推导
const translateTool = defineTool({
  name: 'translate',
  description: '将文本翻译为目标语言',
  parameters: {
    type: 'object',
    properties: {
      text: { type: 'string', description: '待翻译文本' },
      target: { type: 'string', enum: ['en', 'zh', 'ja'] },
    },
    required: ['text', 'target'],
  },
  sessionPermission: {
    kind: 'plugin_output',  // ← Auto 模式据此判断是否自动批准
    describeSideEffect: () => ({
      kind: 'session_file_output',
      summary: '将翻译结果写入 session 文件',
    }),
  },
  async execute(input, ctx) {
    // input 类型自动推导为 { text: string; target: string }
    const result = await ctx.network.fetch(
      `https://api.translate.com?text=${input.text}&to=${input.target}`
    );
    return await result.json();  // 返回值类型自动推导
  },
});
💡 类型安全链路：definePlugin → defineTool → execute(input, ctx) 全链路 TypeScript 类型推导。input 的 properties 自动映射到 execute 参数类型，sessionPermission 的 describeSideEffect 返回值被 Auto 模式的决策引擎消费。这是"类型即文档，类型即安全"的最佳实践。
⚠️ BusProxy 是安全网关：它采用"默认拒绝"策略。即使核心 bus 对象被传递给插件，插件也只能通过代理访问。restricted 插件无法 emit llm_usage 事件，回调中静默丢弃受限事件。代理使用 Object.freeze() 防止插件动态添加属性绕过限制——这是经典的 capability-based security 模式。

四、检验一下

Q1. hana 对象的 hana.api.fetch() 与普通 fetch() 有什么区别？

A. 没有区别，只是语法糖 B. 自动从 iframe URL 提取 pluginId，自动附加 X-Hana-Plugin-Surface-Session 头 C. 使用 WebSocket 替代 HTTP 协议 D. 内置了重试和超时机制，普通 fetch 没有
看答案B。hana.api.fetch() 自动解析 iframe URL 中的 pluginId，拼接 /api/plugins/{pluginId}/ 前缀，并附加 Surface Session 认证头，无需插件手动处理身份信息。

Q2. BusProxy 防止 restricted 插件越权的核心机制是什么？

A. 编译时静态分析插件代码 B. 在 emit/subscribe/request 时检查 accessLevel 和 permissions，静默丢弃受限事件 C. 使用 JWT token 验证每次调用 D. 限制插件的 CPU 和内存使用量
看答案B。createPluginBusProxy() 在每次 emit/subscribe/request 调用时进行权限检查。restricted 插件无法 emit llm_usage 事件，handle 方法仅在 full-access 时暴露。

Q3. 工具执行时 context 会被增强什么？

A. 增加文件系统写入权限 B. 增加 registerSessionFile 和 stageFile，自动绑定当前 sessionId/sessionRef C. 增加数据库连接池 D. 增加 LLM API 的直接调用权限
看答案B。通过 withInvocationSessionHelpers() 增强，使工具 execute(input, ctx) 中的 ctx 包含 registerSessionFile 和 stageFile，自动绑定当前会话。还有 __createInvocationResources 用于工具调用级别的资源范围控制。

Q4. hana.assets.url() 如何推导插件 ID？

A. 从 manifest.json 中读取 id 字段 B. 从当前 iframe 的 URL 路由中解析——iframe 的 src URL 包含 /api/plugins/{pluginId}/ 路径段 C. 通过 postMessage 向宿主查询 D. 从 navigator.userAgent 中提取
看答案B。hana.assets.url() 通过解析 iframe 自身的 src URL 来提取 pluginId（/api/plugins/{pluginId}/ 路径段），然后拼接资源路径。这是"本地计算"——不需要任何跨域通信，比向宿主查询更快更可靠。
