第 2 课 · BridgeManager 与防抖聚合 消息中枢

🎯 本节目标：学完你能解释 _handleMessage() 的完整消息处理流程，理解 Slash 命令拦截的 gate 设计原理，说清 debounce + abort 防抖机制在私聊和群聊中的不同策略。

上一课我们追踪了消息从平台到达 Adapter 并被规范化的过程。本课聚焦 BridgeManager——它是整个 Bridge 子系统的消息总入口，处理所有五个平台的消息，负责角色判定、命令拦截、防抖聚合和附件下载。

一、_handleMessage：消息总入口的 4 步检查

BridgeManager._handleMessage()（位于 lib/bridge/bridge-manager.ts）处理所有五个平台的消息，依次执行以下检查：

async _handleMessage(platform, msg) {
  const { sessionKey, text, senderName, userId, isGroup, chatId,
          attachments, agentId: msgAgentId, messageThreadId } = msg;

  // 0. 解析 agentId：消息自带 > platform entry 绑定 > 丢弃
  const entry = this._platforms.get(this._getPlatformKey(platform, msgAgentId));
  const agentId = msgAgentId || entry?.agentId || null;

  // 1. 判定 owner/guest
  const isOwner = this._isOwner(platform, userId, agentId,
    { isGroup, aliases: identityAliases });
  const bridgeRole = isGroup ? "guest" : isOwner ? "owner" : "guest";

  // 2. Slash 命令拦截（仅 owner）
  if (text.startsWith("/") && isOwner) {
    const dispatchResult = await dispatcher.tryDispatch(text.trim(), { ... });
    if (dispatchResult.handled) return;  // 命令已处理，不进入 LLM
  }

  // 3. RC Pending 拦截
  if (rcState?.isPending(sessionKey) && text && isOwner) {
    const r = await handleRcPendingInput({ engine, sessionKey, text, ... });
    if (r?.handled) return;  // /rc 模式下的选择回复
  }

  // 4. 群聊 vs 私聊：不同缓冲策略
  if (isGroup) {
    pending.batches.push(batch);
    this._triggerPendingFlush(sessionKey);  // 立即入队
  } else {
    pending.lines.push(line);
    this._schedulePendingFlush(sessionKey, isActive ? 1000 : 2000);
  }
}

二、Slash 命令拦截：gate before LLM

斜杠命令（/stop、/compact、/rc 等）在消息进入 LLM 之前被拦截处理，由 slashDispatcher.tryDispatch() 统一分发。避免将管理命令当作普通聊天内容喂给模型。

1触发条件：消息以 / 开头 + 发送者是 owner

2分发机制：slashDispatcher.tryDispatch(text, { platform, sessionKey, ... }) 内部维护命令注册表，匹配后执行对应 handler
lib/bridge/slash-dispatcher.ts
3返回值：{ handled: true/false } — handled=true 表示命令已处理，消息不进入 LLM；handled=false 则继续走正常流程
保证：命令执行失败也不阻塞正常消息流
⚠️ 纪律约束：只有 owner 能触发斜杠命令。
Guest 发 /foo 会直接进入 LLM——原因是所有 bridge 命令都是 owner-only。若 dispatcher 对 guest 返回"静默拒绝"，消息会被完全吞掉，群聊里其他成员甚至 Agent 都看不到这条发言。所以 guest 的命令被视为普通文本发送给 LLM，由模型自行理解。

三、debounce + abort 防抖聚合机制

用户在 IM 中经常快速连发多条消息（如"帮我写周报" / "加上昨天的数据" / "用表格格式"）。如果每条消息都立即触发一次 LLM 调用，不仅浪费 token，还会因并发导致上下文混乱。

场景缓冲策略触发时机原因
私聊（DM）debounce 聚合正在 streaming 则 1s 后，否则 2s 后 flush用户可能连续发多条消息，等待 1-2 秒再批量发送可合并上下文
群聊（Group）立即入 batch queue即时串行处理群聊消息来自不同用户，无需等待聚合；同一 sessionKey 串行保证顺序
 
// 私聊 debounce 调度逻辑
_schedulePendingFlush(sessionKey, delayMs) {
  // 取消之前同一 sessionKey 的定时器
  clearTimeout(this._flushTimers.get(sessionKey));

  // 设置新的延时 flush
  const timer = setTimeout(() => {
    this._flushPending(sessionKey);
  }, delayMs);

  this._flushTimers.set(sessionKey, timer);
}
💡 设计巧思：steer 注入
如果 Agent 正在 streaming 回复期间用户又发了新消息，BridgeManager 不会启动新的 session，而是通过 engine.steerBridgeSession() 将新消息注入到正在进行的 streaming 中。Agent 会在当前回复中自然地包含新消息的内容，而不是另起一个对话。这避免了两个并发 session 争抢上下文的问题。

四、_flushPending：消息冲洗的 7 个步骤

debounce 定时器到期后，_flushPending() 从缓冲中取出所有消息，进行最终处理并发送：

async _flushPending(sessionKey) {
  // 1. 防并发：同一 sessionKey 正在处理中则跳过
  if (this._processing.has(sessionKey)) return;

  // 2. 取出缓冲的所有消息
  const batch = this._takePendingBatch(sessionKey);
  const { lines, attachments, platform, chatId, ... } = batch;

  // 3. 解析附件（图片/语音/视频/文件）
  const { images, textNotes, inboundFiles } =
    await this._resolveAttachments(platform, attachments, agentId);

  // 4. 拼装完整 prompt — 时间标记 + 文本 + 附件说明
  const prompt = textNotes
    ? `${lines.join("\n")}\n${textNotes}`
    : lines.join("\n");
  const merged = `<t>MM-DD HH:mm</t> ${prompt}`;

  // 5. steer 注入（适用于 streaming 中的新消息）
  if (!isGroup && !images.length &&
      this.engine.steerBridgeSession(sessionKey, merged, { role })) {
    return;
  }

  // 6. 启动 LLM 等待回执（"正在输入..."）
  receiptDelivery = await this._startLlmWaitingReceipt({ ... });

  // 7. 通过 Hub 发送消息
  const result = await this._hub.send(merged, {
    sessionKey, agentId, role, meta, onDelta: delivery.onDelta,
    images, inboundFiles, displayMessage, ...
  });
}
→防并发保护：this._processing.has(sessionKey) 确保同一 sessionKey 不会有两个并发的 flush 在进行。这是通过一个 Set 实现的简单互斥锁。

五、消息广播与环形缓冲

每条收到的消息立即通过 _pushMessage() 写入环形缓冲（per-agent 最多 200 条），供 UI "消息日志"面板展示。这个缓冲是独立于 debounce buffer 的——即使消息因 debounce 被合并或丢弃，日志面板仍能看到原始到达记录。

💡 关键设计：双缓冲结构
BridgeManager 维护两套缓冲：（1）debounce buffer（pending.lines / pending.batches）用于聚合后发送给 LLM；（2）环形缓冲（per-agent 200 条）用于 UI 日志展示。两套缓冲独立运作，互不干扰。日志缓冲容量有上限，防止内存泄漏。

六、检验一下（提取练习）

Q1. 为什么只有 owner 能触发 Slash 命令？

A. 因为 guest 的权限不够——dispatcher 返回"静默拒绝"后消息被吞掉，群聊里其他人看不到 B. 因为 guest 没有安装斜杠命令插件 C. 因为数据库中没有 guest 的命令权限记录 D. 因为 WebSocket 协议限制了 guest 的消息类型
看答案A。所有 bridge 命令都是 owner-only，若 dispatcher 对 guest 返回"静默拒绝"，消息会被完全吞掉，群聊里其他成员甚至 Agent 都看不到这条发言。因此 guest 的命令被视为普通文本发送给 LLM。

Q2. 私聊消息的 debounce 延迟是多少？为什么 streaming 中更短？

A. 固定 3 秒，与 streaming 状态无关 B. 正在 streaming 时 1 秒，空闲时 2 秒——streaming 中用户的消息更可能是对当前回复的追问，需要更快响应 C. 固定 500 毫秒，所有场景相同 D. 随机 1-5 秒，避免服务器峰值
看答案B。正在 streaming 时延迟 1 秒（因为新消息很可能与当前对话相关），空闲时延迟 2 秒（等待用户可能的连发）。更短的延迟让 steering 注入更及时。

Q3. _flushPending 中第 5 步的 steer 注入解决了什么问题？

A. 解决了多平台消息格式不兼容的问题 B. 解决了 Agent 正在回复时用户发新消息的并发问题——新消息注入到当前 streaming 而非启动新 session C. 解决了 WebSocket 连接超时的问题 D. 解决了附件下载失败的问题
看答案B。steer 注入将新消息合并到正在进行的 streaming 回复中，避免两个并发 session 争抢上下文。Agent 会在当前回复中自然地响应新消息。

Q4. BridgeManager 的双缓冲结构分别用于什么目的？

A. 一个用于文本消息，一个用于附件消息 B. debounce buffer 用于聚合发送给 LLM；环形缓冲用于 UI 日志展示（per-agent 200 条上限） C. 一个用于钉钉平台，一个用于 Telegram 平台 D. 一个用于 owner 消息，一个用于 guest 消息
看答案B。两套缓冲独立运作：debounce buffer 负责消息聚合和去重后发送给 LLM；环形缓冲（per-agent 200 条）负责 UI "消息日志"面板展示。
