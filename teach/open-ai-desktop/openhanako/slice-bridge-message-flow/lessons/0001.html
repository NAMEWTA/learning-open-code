第 1 课 · 桥接架构总览：Adapter 注册表与消息规范化 架构全景

🎯 本节目标：学完你能画出五平台 Adapter 矩阵，理解 sessionKey 的编码规则，追踪钉钉 WebSocket 消息从到达 → 解析 → 规范化的完整流程。这是理解整个 Bridge 链路的起点。

HanaAgent 的 Bridge 子系统连接了 5 个外部 IM 平台（钉钉 / 飞书 / Telegram / QQ / 微信），将用户消息桥接到 Agent 的 LLM 对话引擎。整个链路共 10 层穿透，涉及 lib/bridge/ 下 26 个源文件。本课聚焦前两层：平台消息如何到达，以及 Adapter 如何将其规范化。

一、架构全景：10 层调用链

从外部平台消息到 Agent 响应回传，每条消息都经过如下调用链：

5 个平台各有独立 Adapter，通过 ADAPTER_REGISTRY 统一注册（含 create / getCredentials / ownerSessionKey）。新增平台只需添加条目 + 实现 Adapter 接口，不改 BridgeManager 核心逻辑。这是经典的开闭原则实践。

三、核心数据结构：sessionKey

sessionKey 是贯穿整个链路的"身份证"，格式为 {platformPrefix}_dm_{userId}@{agentId}，同时编码了平台、用户和 Agent 三者信息。

→解析规则：parseSessionKey()（位于 lib/bridge/session-key.ts）可反向解析出 platform / chatId / agentId
示例：dt_dm_zhangsan123@hanako → platform=dingtalk, chatId=zhangsan123, agentId=hanako
// Adapter 构造 sessionKey 的三种场景
dt_dm_zhangsan123@hanako      // 钉钉用户 zhangsan123 与 Agent hanako 的私聊
fs_dm_ou_abc456@butter        // 飞书用户 ou_abc456 与 Agent butter 的私聊
dt_group_conv789@hanako       // 钉钉群 conv789 → hanako（群聊）

四、钉钉 Stream 连接（代表案例）

1注册 Stream 连接 — POST https://api.dingtalk.com/v1.0/gateway/connections/open 获取 endpoint + ticket

dingtalk-adapter.ts · connect()
3监听消息 — ws.on("message", handleStreamMessage) 处理每条推送

const { endpoint, ticket } = await openStreamRegistration();
ws = new WebSocket(appendTicket(endpoint, ticket));

五、消息规范化：handleStreamMessage 的 5 步处理

收到 WebSocket 消息后，handleStreamMessage() 执行以下处理流程：

1解析信封 — safeJsonParse(raw) 将 Buffer/ArrayBuffer 转为 JSON envelope
2Topic 过滤 — 仅处理 CALLBACK 类型 + /v1.0/im/bot/messages/get topic
3ACK 应答 — acknowledge(envelope) 回传 {"code":200,"message":"OK"}，防止平台重复推送
4机器人过滤 — payload 的 robotCode 与配置不符则丢弃
5消息规范化 — normalizeDingTalkInboundMessage() 转为统一结构

六、normalizeDingTalkInboundMessage 详解

这是关键的"翻译层"，将钉钉原生消息格式转为 bridge 统一消息契约：

function normalizeDingTalkInboundMessage(payload, agentId, profileCache) {
  // 1) 提取身份
  const userId = cleanString(payload.senderStaffId)
    || cleanString(payload.senderId)
    || cleanString(payload.senderUnionId);
  const isGroup = payload.conversationType !== "1";

  // 2) 提取文本内容（兼容 text / audio / richText）
  let text = extractDingTalkText(payload);

  // 3) 构造统一 sessionKey
  const sessionKey = isGroup
    ? `dt_group_${chatId}@${agentId}`
    : `dt_dm_${userId}@${agentId}`;

  // 4) 提取头像与昵称（含 profileCache 去重）
  const profile = resolveDingTalkProfile(payload, profileCache.get(userId));

  // 5) 返回统一消息结构
  return {
    platform: "dingtalk", agentId, chatId, userId,
    sessionKey, text, senderName, displayName,
    avatarUrl, principalId: userId, isGroup,
    _msgId: cleanString(payload.msgId),
  };
}
⚠️ 注意文本提取的三种格式兼容：
• text 型：payload.text.content — 普通文本消息
• audio 型：payload.content.recognition — 钉钉语音识别文本
• richText 型：拼接 payload.content.richText[].text — 富文本消息
如果只处理 text 型，语音消息和富文本消息将丢失内容。

七、检验一下（提取练习）

Q1. sessionKey 的格式编码了哪三部分信息？

A. 消息ID、用户ID、平台类型 B. 平台前缀、用户/群ID、Agent ID C. 时间戳、消息类型、会话状态 D. AppKey、AppSecret、机器人编码
看答案B。sessionKey = {platformPrefix}_dm_{userId}@{agentId}，编码了平台（如 dt=dingtalk）、用户/群ID、以及目标 Agent ID。

Q2. 钉钉 Adapter 收到 WebSocket 消息后，为什么必须发 ACK 应答？

A. 为了告知用户消息已被阅读 B. 为了防止钉钉平台因未收到确认而重复推送同一条消息 C. 为了激活 WebSocket 长连接的心跳机制 D. 为了在钉钉后台记录消息处理日志
看答案B。ACK 应答是钉钉 Stream 模式的协议要求——如果未在规定时间内回传 {"code":200}，钉钉会认为消息投递失败并重复推送，导致同一条消息被处理多次。

Q3. normalizeDingTalkInboundMessage 处理了哪三种消息格式？

A. 文本、图片、视频 B. text 型、audio 型（语音识别）、richText 型（富文本） C. Markdown、HTML、纯文本 D. 单聊、群聊、机器人指令
看答案B。text 型取 payload.text.content，audio 型取 payload.content.recognition（钉钉的语音转文字结果），richText 型拼接多个 richText[].text 段落。

Q4. 为什么五平台 Adapter 各自独立实现，而不是用一个通用的"消息解析器"？

A. 因为各平台的 API 协议、消息格式、连接方式差异太大，通用解析器会变成难以维护的 if-else 地狱 B. 因为代码作者喜欢写重复代码 C. 因为每个 Adapter 需要独立的数据库连接 D. 因为它们运行在不同的进程中
看答案A。五个平台的协议完全不同（Telegram 用 Bot API 轮询、钉钉用 WebSocket Stream、飞书用 SDK 异步回调），消息格式也各不一样。Adapter 模式让每个平台的差异被封装在各自的 Adapter 中，BridgeManager 只依赖统一的消息契约。
