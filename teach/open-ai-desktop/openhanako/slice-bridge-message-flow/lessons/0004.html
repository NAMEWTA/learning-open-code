二、事件驱动的响应捕获：4 种关键事件

session.subscribe() 的回调处理 4 种事件类型，这是流式输出的核心机制：

事件类型触发时机处理逻辑
text_delta每次 LLM 输出 token 块capturedText += delta；调用 onDelta(delta, capturedText) 回传给 BridgeManager 做流式投递
tool_execution_endAgent 调用工具完成提取 details.media（生成的图片/视频）；格式化 automationSuggestion 和 settingsUpdate
message_endLLM 输出完成检查 stopReason === "error" → 记录 providerErrorMessage；用量统计写入 ledger
usage每次 token 消耗更新通过 recordBridgeAssistantUsage() 写入用量账本（source.subsystem = "phone"）
💡 设计亮点：三者正交（#1607）
返回结果为 正文 / 错误 / 截断 三者正交：text（可见正文）、error（provider/transport 错误）、truncated（正文是否因错误不完整）。错误发生时已生成的 partial text 不丢弃，与错误信息分开返回。发送侧根据三个字段独立决策——有 partial text 则发送 + 中断说明，无 text 则发送失败提示。

三、出站清洗：保护 code fence，绝不"吃"正文

LLM 输出含大量内部结构化标签（用于内省、思考、工具调用等），需在发送前清洗。核心纪律：保护 code fence 和行内 backtick 中的内容，绝不误删正文。

// 成对删除（连内容一起删）的标签：
//   <mood>...</mood>       — Agent 内省情绪
//   <pulse>...</pulse>     — Agent 内心状态
//   <reflect>...</reflect> — Agent 反思
//   <think>...</think>     — Agent 思考过程
//   <tool_code>...</tool_code> — 工具调用代码

// 单标签删除（token-only，无配对内容）：
//   </think>               — 裸闭合标签
//   <t>07-07 14:30</t>     — 内部时间戳
//   <|channel-marker|>     — 频道标记

// 保护规则：```...``` 和 `...` 内的标签不删
⚠️ 为什么必须保护 code fence？
用户可能正在让 Agent 写 HTML/XML 代码，其中包含 <think>、</reflect> 等标签。如果不保护 code fence，这些代码中的标签也会被删除，导致用户收到的代码不完整。清洗逻辑必须先识别 code fence 区域，只清洗区域外的标签。

四、流式投递策略矩阵：6 种模式

不同平台的 API 能力差异巨大，因此 Bridge 支持 6 种投递模式：

投递模式适用平台工作原理用户感知
batch钉钉、QQ、微信累积全部输出后一次性发送等待后一次性收到完整回复
blockTelegramBlockChunker 按段落/列表/表格分块发送逐段收到，像打字机效果
draftTelegram每 500ms 发送 draft 草稿预览，最终发正式消息实时看到草稿更新
edit_messageTelegram先发空消息占位 → 持续 editMessageText 更新看到消息逐渐变长
rich_draft飞书类似 draft + rich 消息格式（支持 Markdown）实时看到富文本草稿
cardkit_stream飞书飞书卡片消息流式更新（start → update → finish）卡片内容逐步填充

注意：钉钉、QQ、微信三个平台只能使用 batch 模式——它们的 API 不支持消息编辑或流式更新。Telegram 的 Bot API 最为灵活，支持 3 种流式模式。

五、钉钉发送：UTF-8 字节安全分片

钉钉消息按 UTF-8 字节数（非字符数）分片，限制单条 12000 字节。中文每字 3 字节，约 4000 字即需分片：

async sendReply(chatId, text, options = {}) {
  const scope = dingTalkTargetScope(options);  // dm 或 group
  const chunks = splitTextByUtf8Bytes(text, 12000);  // 按 12KB 分批

  for (let i = 0; i < chunks.length; i++) {
    const msgParam = JSON.stringify({
      title: chunks.length > 1 ? `Hana ${i+1}/${chunks.length}` : "Hana",
      text: chunks[i],
    });
    if (scope === "group") {
      await requestRobotApi("send_group", groupUrl, {
        robotCode, openConversationId: chatId,
        msgKey: "sampleMarkdown", msgParam,
      });
    } else {
      await requestRobotApi("send_dm", dmUrl, {
        robotCode, userIds: [chatId],
        msgKey: "sampleMarkdown", msgParam,
      });
    }
  }
}
→关键细节：splitTextByUtf8Bytes() 不是简单的 text.slice(0, 12000)——它确保不会从多字节字符（如中文、emoji）的中间切割，避免产生乱码。

六、检验一下（提取练习）

Q1. 响应清洗时，为什么要保护 code fence（```...```）内的内容？

A. 因为 code fence 内的代码可能包含类似内部标签的字符串（如 <think>），误删会导致用户收到的代码不完整 B. 因为 code fence 是 HTML 标签，不能删除 C. 因为删除 code fence 会被钉钉 API 拒绝 D. 因为 code fence 内的内容更耗时，不需要清洗
看答案A。用户可能让 Agent 写 HTML/XML/JSX 代码，其中包含 <think>、</mood> 等字符串。如果不保护 code fence，这些代码也会被当作内部标签删除。

Q2. "三者正交"设计中的三者指什么？为什么这样设计？

A. 请求、响应、错误——API 设计的三要素 B. 正文（text）、错误（error）、截断（truncated）——错误发生时已生成的 partial text 不丢弃，与错误信息分开返回 C. 文本、图片、视频——多模态输出的三种格式 D. 同步、异步、流式——三种通信模式
看答案B。错误发生时已生成的 partial text 保留在 text 字段中，与 error 信息分开。发送侧根据三个字段独立决策：有 partial text 则发送 + 中断说明，无 text 则发送失败提示。这保证了用户不会完全收不到任何回复。

Q3. 为什么钉钉只能用 batch 模式，而 Telegram 可以用 edit_message 流式模式？

A. 因为 Telegram 的网络延迟更低 B. 因为钉钉的消息 API 不支持编辑已发送的消息，Telegram 的 Bot API 提供了 editMessageText 接口 C. 因为钉钉的开发文档规定必须用 batch D. 因为 Telegram 使用 HTTP/2，钉钉使用 HTTP/1.1
看答案B。这是平台 API 能力的差异。Telegram Bot API 提供了 editMessageText 方法，允许在发送后修改消息内容，因此可以实现 draft / edit_message 等流式模式。钉钉的机器人消息 API 不支持修改已发送的消息。

Q4. splitTextByUtf8Bytes 为什么不能简单用 text.slice(0, 12000)？

A. 因为 slice 是 JavaScript 的字符级操作，12000 字符不等于 12000 字节（中文每字 3 字节） B. 因为 slice 会改变文本的编码格式 C. 因为 slice 不支持 Unicode 字符 D. 因为 12000 不是 2 的幂
看答案A。JavaScript 的 slice 按字符数切割，但钉钉限制的是 UTF-8 字节数（12000 字节）。中文每字 3 字节，emoji 可能 4 字节，直接按字符数切割会导致超出字节限制或被平台拒绝。且必须在字符边界切割，避免截断多字节字符产生乱码。
