第 5 课 · 异常路径全解与综合测验 终章

🎯 本节目标：学完你能识别 8 种异常点的发生位置、用户可见现象和自愈能力，能画出完整时序图，通过综合测验检验对 10 层链路的全链路理解。这是本系列的最后一课。

前 4 课我们逐层穿透了消息桥接的完整链路。本课首先拆解 5 条核心异常路径——理解系统如何"优雅地失败"，然后展示完整时序图，最后通过综合测验检验你对全链路的掌握程度。

一、完整时序图

先回顾一条正常消息的端到端时序（从钉钉用户发消息到收到回复）：

钉钉用户 → DingTalkAdapter → BridgeManager → Hub → BridgeSessionManager
                                                      ↓
                                                   Pi SDK (Agent)
                                                      ↓
                                                   LLM Provider
                                                      ↓
钉钉用户 ← DingTalkAdapter ← BridgeManager ← BridgeSessionManager

关键节点：
1. normalizeDingTalkInboundMessage() 消息规范化
2. _handleMessage() owner/guest 判定 + Slash 拦截 + debounce
3. _flushPending() 附件下载 + steer 注入尝试 + Hub 发送
4. Hub.send() 路由规则匹配（本文中命中规则 3 → owner 路径）
5. executeExternalMessage() 会话索引查找 + Prompt 构建 + 工具过滤
6. createAgentSession() + session.prompt() Agent 执行
7. session.subscribe() 事件驱动的响应捕获
8. buildExternalMessageResult() 三者正交的结果封装
9. _cleanReplyForPlatform() + stripInternalTags() 清洗
10. sendReply() + splitTextByUtf8Bytes() 平台发送

二、异常路径 1：平台认证失败（钉钉 AppKey/Secret 错误）

1触发：钉钉 AppKey 或 AppSecret 填写错误
所在层：L1 DingTalkAdapter
2链路：autoStart() → startPlatform("dingtalk") → openStreamRegistration() → POST 钉钉 API → 返回 InvalidSignature → validateDingTalkApiResponse 检测到非 0 错误码 → 抛 dingTalkApiError
3自愈：connect() catch → onStatus("error") → scheduleReconnect()，进入 5 秒间隔重连循环直至成功或 stopped
  • 用户可见：bridge 状态面板显示钉钉 = "error: InvalidSignature"
  • 影响范围：仅钉钉平台，不影响其他平台（Telegram、飞书等独立运行）
  • 消息安全：消息本身没有到达，无"未回复"问题

三、异常路径 2：Agent 未配置聊天模型

1触发：Agent config.yaml 中 models.chat 为空或缺少 provider
所在层：L5 BridgeSessionManager
2链路：消息经 Adapter → BridgeManager → Hub → 进入 executeExternalMessage() → _buildOwnerSessionOpts() → agent.config?.models?.chat 解析为 null → 抛错 "bridge agent has no chat model configured"
3降级：最外层 catch → buildExternalMessageResult({ error: err.message }) → 回到 BridgeManager → delivery.fail("回复失败，请稍后重试") → 平台收到友好提示，不暴露技术细节

四、异常路径 3：LLM Transport 超时 / Provider 5xx

1触发：LLM 请求超时、网络中断、Provider 返回 5xx
所在层：L6 Pi SDK / L7 BridgeSessionManager
2链路：session.prompt() 内部抛出 transport 错误 → 三者正交生效：出错瞬间 capturedText 已有内容 → text 保留 partial、error 记录原因、truncated 标记不完整
3降级：有 partial text → 发送已生成文本 + "(回复被中断)"说明；无 text → 发送 "(回复失败)"
💡 三者正交的价值在此体现：如果错误处理是"要么成功返回全部，要么失败返回 null"，超时时用户会看到一个空白回复——但 LLM 可能已经生成了 80% 的内容。三者正交让已生成的 partial text 能被发送，用户至少能看到大部分回复。

五、异常路径 4：流式发送失败降级

1触发：Telegram edit_message 流式发送中途 adapter 抛错
所在层：L9 Adapter
2链路：enqueueSnapshot() 中 adapter.updateStreamReply() 失败 → failed = true — 标记 delivery 已失败，后续 onDelta 跳过 → finish() 检测 failed → 降级为 _sendStreamFallbackReply()（普通 sendReply）
3降级：降级也失败 → 记录日志，不再重试。用户看到的是批量模式发送的完整文本（而非流式更新的过程）

六、异常路径 5：RC 桌面 Session 失效

1触发：桌面端 App 关闭或 session 过期时，bridge 用户仍处于 /rc 模式
所在层：L3 BridgeManager
2链路：bridge 用户发消息 → _handleMessage() 检测 rcState.isPending(sessionKey) → handleRcPendingInput() 尝试转发到桌面 session → 桌面 session 不存在或已关闭 → 返回 handled=false
3降级：自动退出 /rc 模式 → bridge 用户收到 "接管已失效" 提示 → 后续消息恢复正常 bridge 处理流程

七、异常路径速查表

异常点所在层用户可见自愈能力

八、关键设计模式回顾

1Adapter 模式 + 注册表：5 平台独立 Adapter，通过 ADAPTER_REGISTRY 统一注册，新增平台只加条目不改核心
2debounce + abort 防抖聚合：私聊用 1-2s 窗口聚合连发消息，streaming 中通过 steer 注入而非新建 session
3依赖倒置：BridgeSessionManager 不持有 engine 引用，所有依赖通过 deps 对象注入，使单元测试成为可能
4Prompt Snapshot 快照：Bridge session 创建时对 system prompt 做快照，后续 prompt 变更不影响已有 session
5三者正交：错误发生时已生成 partial text 不丢弃，与 error 信息分开返回，发送侧根据三个字段独立决策
6幂等去重（10min TTL）：通知、桥接消息、延迟结果均带幂等键，防止平台重推导致重复处理

九、综合测验（提取练习）

Q1（综合）. 一条钉钉群聊消息从用户发送到 Agent 回复，共经过哪些层？选出正确的顺序。

A. Adapter → Hub → BridgeManager → BridgeSessionManager → LLM B. Adapter → BridgeManager → Hub → BridgeSessionManager → Pi SDK → LLM → BridgeSessionManager → BridgeManager → Adapter C. BridgeManager → Adapter → Hub → LLM → Adapter D. Adapter → BridgeSessionManager → Hub → BridgeManager → LLM
看答案B。完整路径：Adapter（消息解析）→ BridgeManager（防抖/命令拦截）→ Hub（路由分发）→ BridgeSessionManager（会话匹配/Prompt构建）→ Pi SDK（Agent执行）→ LLM → 响应按原路返回。

Q2（异常分析）. LLM 请求超时时，用户可能看到部分回复 + "(回复被中断)"。这个行为是由哪个设计模式保证的？

A. Adapter 模式 B. 依赖倒置 C. 三者正交——正文/错误/截断独立返回，已生成的 partial text 不丢弃 D. 幂等去重
看答案C。buildExternalMessageResult() 返回的 { text, error, truncated } 三者独立——超时时 text 保留已生成内容，error 记录超时原因，truncated 标记不完整。发送侧根据这三个字段决定展示什么。

Q3（跨层对比）. 平台认证失败（异常1）和 Agent 无聊天模型（异常2），哪一方的自愈能力更强？为什么？

A. 异常 1 更强——钉钉 Adapter 会自动重连（5s 间隔），只需修复凭证即可恢复；异常 2 需人工修改 config.yaml B. 异常 2 更强——系统会自动创建默认模型配置 C. 两者自愈能力相同 D. 两者都没有自愈能力
看答案A。平台认证失败有自动重连机制（5s 间隔循环），一旦用户修复凭证，下次重连即成功。Agent 无聊天模型需要人工编辑 config.yaml 并重启，没有自动修复路径。

Q4（综合）. Guest 在群聊中发送 "/stop" 命令，会发生什么？

A. 命令被 dispatcher 拦截并执行，Agent 停止回复 B. "/stop" 被当作普通文本发送给 LLM，Agent 可能回复"好的，已停止"或类似内容 C. 命令被静默忽略，用户收不到任何反馈 D. 系统返回"权限不足"的提示
看答案B。回顾第 2 课的设计：只有 owner 能触发斜杠命令。guest 的 "/stop" 因不是 owner 而跳过命令拦截检查，进入正常消息流，最终作为普通文本发送给 LLM。

Q5（综合）. 以下关于链路设计的说法，哪个是正确的？

A. BridgeManager 直接调用 LLM API，不经过中间层 B. BridgeSessionManager 持有 engine 的直接引用，方便调用 C. debounce 期间新消息通过 steer 注入而非新建 session，避免两个并发 session 争抢上下文 D. 清洗标签时，code fence 内的标签也会被删除以保持输出干净
看答案C。A 错误——BridgeManager 通过 Hub → BridgeSessionManager → Pi SDK 多层间接调用 LLM。B 错误——BridgeSessionManager 使用依赖倒置，通过 deps 对象注入依赖。D 错误——清洗逻辑必须保护 code fence 内的内容。

Q6（综合）. 钉钉 12000 字节分片限制意味着一条约 5000 个汉字的回复，用户会收到几条消息？

A. 1 条（5000 字 × 3 字节 = 15000 字节，刚好超出 12000） B. 2 条（第一条约 4000 字 = 12000 字节，第二条约 1000 字） C. 3 条（因为还需要标题占用字节） D. 无法确定，取决于文本中是否包含英文和 emoji
看答案B。5000 汉字 = 15000 字节（每汉字 3 字节）。12000 字节 / 3 = 4000 汉字/条。因此第一条约 4000 字，第二条约 1000 字。标题 "Hana 1/2" 占用少量字节但可忽略。
