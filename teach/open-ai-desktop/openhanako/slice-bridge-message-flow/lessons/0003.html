第 3 课 · Hub 路由与 Session 匹配 路由 & 会话

🎯 本节目标：学完你能说出 Hub 的 4 条路由规则及匹配顺序，理解 owner vs guest 两种模式的 Prompt 构建差异，知道 4 个被屏蔽的工具及原因。这是整个链路中逻辑密度最高的一层。

上一课我们追踪了消息在 BridgeManager 中的防抖聚合与命令拦截。本课聚焦两个核心组件：Hub（消息路由中枢）和 BridgeSessionManager（会话匹配与 Prompt 构建）。后者是整个链路中逻辑密度最高的一层（~320 行方法体）。

一、Hub.send()：4 条路由规则按序匹配

Hub 是消息路由中枢（位于 hub/index.ts），管理 4 条路由规则，顺序匹配，第一条命中即执行：

// Hub 路由表（顺序匹配，第一条命中即执行）：
const routes = [
  { // 规则 1：桌面端 owner — 无 sessionKey + 非 ephemeral
    match: o => !o.sessionKey && !o.ephemeral && o.role === "owner",
    handle: () => submitDesktopSessionMessage(engine, { ... })
  },
  { // 规则 2：Bridge guest — 有 sessionKey + role = "guest"
    match: o => o.sessionKey && o.role === "guest",
    handle: () => guestHandler.handle(text, sessionKey, ...)
  },
  { // 规则 3：Bridge owner — 有 sessionKey + 非 ephemeral
    match: o => o.sessionKey && !o.ephemeral,
    handle: () => engine.executeExternalMessage(text, sessionKey,
      meta, { guest: false, agentId, onDelta, images, ... })
  },
  { // 规则 4：独立执行 — ephemeral（cron / heartbeat / phone）
    match: o => o.ephemeral,
    handle: () => engine.executeIsolated(text, { ... })
  },
];
规则匹配条件处理器典型场景
1无 sessionKey + role=owner + 非 ephemeral桌面端会话提交用户在桌面 App 中直接对话
2有 sessionKey + role=guestGuestHandler群聊中非 owner 成员发言
3有 sessionKey + 非 ephemeral（隐含 owner）engine.executeExternalMessageBridge 私聊（最常见路径）
4ephemeral=true独立执行（无会话持久化）cron 定时任务 / heartbeat / phone channel
⚠️ 顺序敏感！规则 3 的匹配条件是 sessionKey && !ephemeral，没有检查 role。因为规则 2 已经将 guest 拦截，到达规则 3 的必然是 owner。如果规则顺序颠倒，guest 消息会被错误地路由到 owner 处理器，导致权限越界。

二、BridgeSessionManager：会话索引查找与恢复

对于 bridge 私聊消息（命中规则 3），Hub 调用 engine.executeExternalMessage()，最终委托给 BridgeSessionManager.executeExternalMessage()。这是 core/bridge-session-manager.ts 中约 320 行的方法体。

// 1. 读取 bridge-sessions.json 索引（per-agent）
const index = this.readIndex(agent);
const raw = index[sessionKey];
const entry = this._normalizeIndexEntry(raw);
const existingFile = entry.file || null;  // e.g. "owner/dt_dm_xxx.jsonl"

// 2. 尝试打开已有 session（带修复）
if (existingPath && !roleChanged) {
  // 修复：清理孤儿 toolResult entry
  repairOrphanToolResultEntriesInFile(existingPath);
  // 修复：内联媒体历史
  this._repairInlineMediaHistory(existingPath, "bridge session restore");
  mgr = SessionManager.open(existingPath, sessionDir);
}

// 3. 打不开则创建新 session（自愈）
if (!mgr) {
  mgr = SessionManager.create(homeCwd, sessionDir);
}

core/bridge-session-manager.ts · executeExternalMessage() 第 1 步
2修复打开：打开旧 session 前执行两项修复——清理孤儿 toolResult entry（#1285 修复）、修复内联媒体历史
自愈机制：不因历史数据损坏而阻塞新对话
3自愈创建：如果旧 session 打不开（文件损坏 / 格式变更），自动创建新 session，不阻塞用户对话

三、Owner vs Guest 模式的 Prompt 构建差异

这是 BridgeSessionManager 中最重要的设计决策——owner 和 guest 使用截然不同的 Prompt 和工具集：

方面Owner 模式Guest 模式
System Prompt完整（yuan + ishiki + memory + skills + agents files）+ bridge 上下文行仅 yuan + public-ishiki + contextTag + bridge 上下文行
模型agent.config.models.chatagent.config.models.chat
工具集全部 agent 工具，屏蔽 4 个禁用工具空（无工具执行能力）
thinking level用户配置的思考级别off（不启用思考）
Vision Bridge启用视觉上下文注入启用视觉上下文注入

💡 关键洞察：为什么 guest 没有工具？
Guest 是群聊中的非 owner 成员——可能是同事、朋友或陌生人。给 guest 开放工具执行能力（如文件读写、命令执行、浏览器操控）是严重的安全风险。guest 只能进行纯文本对话，所有工具调用请求由 LLM 自行拒绝（因为没有可用工具）。

四、工具过滤规则

Owner 模式下，4 个工具被显式从 bridge session 移除：

const BRIDGE_OWNER_DENIED_TOOL_NAMES = Object.freeze([
  "computer",        // 电脑操控 — bridge 场景不适用（远程用户无法操作桌面）
  "browser",         // 浏览器管理 — bridge 场景不适用
  "hana_card_guide", // 卡片引导 — 桌面 UI 专属功能
  "show_card",       // 卡片展示 — 桌面 UI 专属功能
]);

这 4 个工具的共性：它们依赖桌面 UI 环境。Bridge 用户通过手机 IM 交互，没有桌面端的卡片渲染引擎和窗口管理系统，因此这些工具在 bridge 场景下没有意义。

五、Bridge 上下文构建

buildBridgeContext()（位于 lib/bridge/bridge-context.ts）将 sessionKey / 平台名 / 用户信息 / 角色拼接为一行注入 system prompt：

// 示例输出（中文环境）
"[Bridge 上下文] 这是来自 钉钉 的消息。你的名字是 Hana。
 发送者名称：张三。请使用简洁、自然的语言回复。"

这一行上下文让 Agent 知道自己正在通过外部 IM 平台与用户对话，从而调整回复风格（更简洁、避免代码块过长、不使用桌面专属功能）。

六、检验一下（提取练习）

Q1. Hub 的 4 条路由规则为什么必须按当前顺序排列？

A. 因为按字母顺序排列代码更整洁 B. 因为规则 2 拦截 guest 后，规则 3 才能安全地不检查 role 字段——顺序颠倒会导致 guest 被错误路由到 owner 处理器 C. 因为规则 1 的处理速度最快，放前面提升性能 D. 因为它们按创建时间排序
看答案B。路由规则顺序敏感：规则 2（guest）必须在规则 3（owner）之前，否则 guest 消息会匹配到不检查 role 的规则 3，被当作 owner 处理——这会导致权限越界。

Q2. BridgeSessionManager 打开旧 session 时的两项修复是什么？

A. 修复数据库连接和网络超时 B. 清理孤儿 toolResult entry 和修复内联媒体历史 C. 修复 JSON 格式化错误和编码问题 D. 修复时间戳偏移和用户权限
看答案B。两项修复：（1）repairOrphanToolResultEntriesInFile() 清理因崩溃/中断产生的孤儿 toolResult 条目；（2）_repairInlineMediaHistory() 修复内联媒体引用的历史记录。

Q3. guest 模式的工具集为什么是空的？

A. 因为 guest 的模型不支持工具调用 B. 因为安全考虑——guest 可能是群聊中的陌生人，不应有文件读写、命令执行等能力 C. 因为 SDK 版本限制 D. 因为 guest 的工具需要额外付费
看答案B。安全是第一位的。guest 是群聊中的非 owner 成员，开放工具执行能力（文件读写、命令执行、浏览器操控）是严重的安全风险。

Q4. 哪 4 个工具在 bridge owner 模式下被屏蔽？共性是什么？

A. read、write、edit、delete — 文件操作 B. computer、browser、hana_card_guide、show_card — 都依赖桌面 UI 环境 C. search、fetch、bash、grep — 数据查询 D. memory、skills、agents、config — 配置管理
看答案B。computer（桌面操控）、browser（浏览器管理）、hana_card_guide（卡片引导）、show_card（卡片展示）——它们的共性是都依赖桌面 UI 环境，bridge 用户通过手机 IM 交互无法使用。
