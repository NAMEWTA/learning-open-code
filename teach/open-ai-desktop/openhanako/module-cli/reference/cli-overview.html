1. 模块定位

cli/ 是 HanaAgent 的终端命令行界面入口模块。它不参与核心业务逻辑，而是作为最外层的「薄壳」，将用户从终端连接到 HanaAgent Server。其职责可概括为三件事：

  1. 启动服务 — hana serve 可在终端前台启动一个 headless HanaAgent Server
  2. 发现连接 — 三层降级策略（显式 URL → 本地 server-info.json → 自动启动）解析 Server 地址
  3. 交互对话 — 提供基于 WebSocket 的实时终端聊天界面，支持 ANSI 色彩与按键绑定
设计原则：CLI 模块是 HanaAgent 最薄的入口层。它不包含任何 AI 逻辑、持久化逻辑或业务规则——一切委托给 Server 的 HTTP/WS API。这种「薄入口」设计使得桌面端（Electron）和 CLI 端可以共享同一 Server，无需重复实现。

一句话：7 个文件、~480 行代码，实现了一个自发现的、可自动拉起 Server 的终端 Agent 对话客户端。

4. entry.ts — CLI 入口编排

属性值
行数78
核心导出main(argv)
依赖args.ts, local-server.ts, client.ts, chat.ts, server-runner.ts, terminal-theme.ts
被引用方package.json 的 "bin" 字段（作为 hana 命令入口）

关键设计

shebang 自执行：文件顶部 #!/usr/bin/env node 使其可直接作为可执行脚本。底部通过比较 process.argv[1] 与 import.meta.url 判断是否为直接入口（而非被 import），实现「既可被 import 也可独立运行」的模块模式。

PROJECT_ROOT 计算：通过 fileURLToPath(import.meta.url) 获取当前文件路径，再 path.resolve(__dirname, "..") 定位到项目根目录。这用于后续 Server 进程的 spawn 路径推导。

命令路由表：入口函数按照 args.command 进行分发，优先级如下：

command处理函数说明
help直接输出 helpText()未知命令也会 fallback 到 help
servespawnServerForeground()不建立连接，直接 spawn 子进程
statusprintStatus()需先建立连接
sessionsprintSessions()需先建立连接
continuestartChat()需先建立连接，传入 target
chatstartChat()需先建立连接，传入 session

自动启动策略：shouldAutoStartServer() 仅在用户未指定 --url 且命令为 chat 或 continue 时返回 true。如果本地 Server 不存在且不可自动启动，则提示用户手动执行 hana serve。

5. args.ts — 命令行参数解析

属性值

5.1 命令枚举

支持 6 种命令，通过 COMMANDS Set 验证：

命令用途示例
serve启动 headless Serverhana serve -- --port 3000
status查看 Server 状态hana status
sessions列出最近会话hana sessions
continue继续最近会话hana continue 3 或 hana continue /path/to/session
chat新建聊天hana chat --plain
help帮助信息hana help（也是默认 fallback）

5.2 选项速查

选项类型默认值说明
--plainbooleanfalse纯文本模式（禁用 raw mode，行导向输入）
--url <url>stringnull显式指定 Server 地址，跳过本地自动发现
--token <token>stringnullBearer token 认证
--session <path>stringnull指定会话路径
--分隔符—之后的所有参数作为 passthrough 传给 Server 进程

5.3 continue 命令的特殊处理

continue 命令接受一个位置参数（不带头缀 -），可以是数字索引（如 3 表示第 3 个会话）或会话路径。解析器在未遇到其他选项时，将第一个非选项参数赋值给 result.target。若后续还有参数，它们落入 passthrough。

5.4 错误处理

  • 未知命令 → 返回 { command: "help", error: "unknown command: ..." }
  • 带值选项缺少值（如 --url 后无参数）→ 抛异常，由 entry.ts 捕获后输出红色错误 + help

6.1 startChat() 架构

这是 CLI 模块的核心函数，实现了完整的终端聊天交互循环。内部结构分为三层：

第一层：初始化阶段（同步 + 异步并行）

  • 并行加载 loadContext(client)（health + agents API）
  • 基于 agentYuan 创建终端主题
  • 解析会话（continue 定位 或 取最新 或 新建）
  • 创建 WebSocket 连接 + Readline 接口

第二层：WebSocket 事件循环（消息类型分发）

消息类型处理方式视觉效果
text_delta首条 delta 时换行，后续直接 process.stdout.write流式逐字输出
thinking_start启动 500ms 间隔帧动画定时器"✿ Hana 正在思考..." 轮播
thinking_end清除定时器 + 擦除当前行思考动画消失
tool_start停止思考动画，输出工具名◇ tool_name（角色色）
tool_end根据 success 字段输出绿色 done / 红色 failed
mood_start/text/end累积 mood delta，结束时渲染倾斜体 + 灰色心情文本
turn_end重置 streaming 状态 + 输出 prompt换行 + 等待输入
error停止思考 + 输出红色错误红色错误信息
app_event检测 agent-switched/updated，刷新主题角色主题热更新

第三层：用户输入处理

  • 普通文本 → 包装为 { type:"prompt", text, sessionPath } 通过 WebSocket 发送
  • 斜杠命令 → handleCommand() 分发到 6 种内置命令（quit/sessions/continue/new/status/help）
  • 流式中 → 输入被忽略（防抖保护）

6.2 按键绑定（仅 TTY + 非 plain 模式）

通过 readline.emitKeypressEvents + stdin.setRawMode(true) 实现：

按键流式状态行为
Escape流式中发送 { type:"abort", sessionPath } 中断生成
Escape空闲无操作（Readline 默认处理）
Ctrl+C流式中中断生成（同 Escape）
Ctrl+C空闲退出程序 (closeAndExit(0))

6.3 思考动画机制

一个基于 setInterval 的简单帧动画系统，在 Agent 处理消息时给用户视觉反馈：

  • 3 帧文本轮播："正在思考" → "正在整理上下文" → "正在看工具轨迹"
  • 每 500ms 切换一帧，追加 "."（1~3 个）增强动感
  • 使用 \r\x1b[K 覆盖当前行（回到行首 + 清除到行尾）
  • 收到任何实质消息（text_delta/tool_start/error）立即停止

7.1 类结构

成员类型说明
baseUrlstringServer 基础 URL（已去除尾部斜杠）

7.2 HTTP 方法速查

方法HTTP端点说明
health()GET/api/health获取 Server 健康状态（agent、model、yuan 等）
identity()GET/api/server/identity获取 Server 身份（version、studioLabel 等）

7.3 request() 通用方法

所有 HTTP 方法共享同一个内部 request(path, opts) 实现：

  • 认证注入：若 token 非空，自动添加 Authorization: Bearer <token> 头
  • JSON 序列化：若 body 为对象且非 Uint8Array，自动设置 Content-Type 并 JSON.stringify
  • 错误处理：非 2xx 响应时，从响应体中提取 detail/reason/error 字段构造 Error 对象，并附加 status 和 data 属性
  • 响应解析：尝试 JSON.parse，失败时返回原始文本

7.4 createWebSocket() 连接工厂

  • URL 转换：将 baseUrl 的 http 替换为 ws，pathname 设为 /ws
  • Token 传递：若 queryTokenAllowed=true（本地连接），token 通过 URL query string 传递；否则通过 Authorization 头传递
  • 协议差异说明：本地连接使用 query token 是因为 ws 库在 Node.js 中设置自定义 header 存在兼容性问题

8.1 连接解析三层降级

resolveConnection() 实现了从高优先级到低优先级的连接源降级：

resolveConnection({ url, token, hanaHome })
  │
  ├─ 第 1 层：显式 URL
  │   if (url) → { ok:true, baseUrl:url, token, source:"explicit", queryTokenAllowed:false }
  │
  └─ 第 2 层：本地 Server 信息文件
      readLocalServerInfo({ hanaHome })
        │
        ├─ ~/.hanako/server-info.json 不存在
        │   → { ok:false, reason:"missing_server_info" }
        │
        ├─ 文件格式无效
        │   → { ok:false, reason:"invalid_server_info" }
        │
        ├─ port/token 缺失
        │   → { ok:false, reason:"incomplete_server_info" }
        │
        ├─ 进程不再存活
        │   → { ok:false, reason:"stale_server_info" }
        │
        └─ 一切正常
            → { ok:true, baseUrl:"http://127.0.0.1:{port}", token, source:"server-info",
                 queryTokenAllowed:true }

8.2 关键函数详解

函数输入输出核心逻辑
resolveCliHanaHome() env string 优先读 HANA_HOME 环境变量，否则使用 ~/.hanako。
支持 ~ 展开和 ~/... 路径
readLocalServerInfo() { hanaHome, checkProcess } { ok, ... } 读取 server-info.json，校验 pid 存活（可选），构造连接信息
isProcessAlive() pid boolean 使用 process.kill(pid, 0) 检测进程是否存在。
信号 0 不发送实际信号，仅检查权限和存在性

9. server-runner.ts — 服务进程启动器

属性值

9.1 两种启动模式

模式函数stdiodetached等待就绪使用场景
前台 spawnServerForeground() "inherit" 否 否（阻塞终端） hana serve 命令
后台 startLocalServerAndWait() "ignore" 是 + unref() 是（轮询 server-info.json） chat/continue 自动启动

9.2 自动模式探测（resolveServerSpawnSpec）

CLI 需要兼容两种运行环境：开发中的源码模式和打包后的发行模式。探测逻辑如下：

resolveServerSpawnSpec({ projectRoot, env, extraArgs })
  │
  ├─ 检查 HANA_ROOT 环境变量
  │   if (env.HANA_ROOT + bootstrap.js 存在) → packagedRoot = HANA_ROOT
  │
  ├─ 检查打包产物
  │   if (projectRoot/bootstrap.js 存在 AND projectRoot/bundle/index.js 存在)
  │     → mode: "packaged"
  │       command: node
  │       args: [bootstrap.js, ...extraArgs]
  │       env: { HANA_ROOT, HANA_SERVER_ENTRY: bundle/index.js }
  │
  └─ fallback: 源码模式
        → mode: "source"
          command: node
          args: [server/index.ts, ...extraArgs]
          env: 原样透传

关键差异：

  • packaged 模式 — 通过 bootstrap.js 启动，需设置 HANA_SERVER_ENTRY 指向打包产物 bundle/index.js
  • source 模式 — 直接运行 server/index.ts（依赖 Node.js 的 TypeScript 支持）

9.3 startLocalServerAndWait() 轮询机制

  • 超时：默认 30 秒（通过 timeoutMs 参数可调）
  • 间隔：每 250ms 检查一次 server-info.json
  • 短路：若已有运行中的 Server（server-info.json 存在且进程存活），立即返回，不重复启动
  • 清理：子进程通过 detached: true + child.unref() 与父进程解绑，确保 CLI 退出后 Server 继续运行

10.1 ANSI 基础色板 (ansi)

通过 Object.freeze 冻结的常量对象，提供 8 种标准 ANSI 转义码：

属性转义码效果
reset\x1b[0m重置所有样式
bold\x1b[1m加粗
dim\x1b[2m暗淡（灰色辅助信息）
italic\x1b[3m斜体（心情文本）
red\x1b[31m错误信息
yellow\x1b[33m警告信息
green\x1b[32m成功状态
gray\x1b[90m额外辅助灰色

10.2 角色主题映射

createTerminalTheme(yuan) 将 Agent 角色标识（"hanako" | "butter" | "ming"）映射为终端视觉主题。通过调用 shared/yuan-visuals.ts 的 getYuanVisual() 获取角色元数据：

YuanSymbolMoodLabelAccent (hex)终端色彩
hanako✿✿ MOOD#537D96蓝灰色调
butter❊❊ PULSE#5BA88C绿色调
ming◈◈ REFLECT#8BA4B4浅蓝灰调

24-bit 真彩色：color(hex) 将 hex 颜色转换为 ANSI 24-bit 真彩色转义码 \x1b[38;2;R;G;Bm，使终端可以显示任意 RGB 颜色（需要终端支持）。

paint() 快捷函数：paint(theme, text) 等价于 theme.accent + text + ansi.reset，用于高亮角色相关文本。

10.3 未定义 yuan 的 fallback

如果 health API 返回的 agentYuan 不在 YUAN_VISUALS 字典中，normalizeYuan() 会 fallback 到 "hanako"（✿ 蓝灰色调），确保终端界面不会因主题缺失而崩溃。
