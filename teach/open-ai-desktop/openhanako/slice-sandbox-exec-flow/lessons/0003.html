第 3 课 · Bubblewrap 容器执行：bwrap参数构造+spawn+流式输出 执行引擎

本节目标（对应使命 Success #3）：学完你能解释三种命令执行路径（Direct/Once/TTY）的选择逻辑，掌握 Bubblewrap 的 bwrap 参数构造策略，理解 CommandOutputCollector 的双缓冲设计和输出截断机制。

前两课覆盖了命令的"检查层"（分类+路径判决+审批）。当命令通过所有检查后，进入真正的执行层——第 6 层（执行器选择）、第 7 层（OS 沙盒）和第 8 层（输出收集）。这三层是纵深防御的"执行臂"。

一、三路径执行模型（Layer 6）

命令通过所有校验后，tool.ts 的 execute 函数根据可用资源和参数选择三种执行路径之一：

// lib/exec-command/tool.ts — 三路径选择
if (value.tty) {
  // 路径 A：交互式 PTY 终端
  return startExecCommandTty({ manager, getAgentId, command, ... });
}
if (commandExec) {
  // 路径 B：Direct 执行（使用 OS 沙盒化的 exec 函数）
  return runExecCommandDirect({ commandExec, command: renderedCommand, workdir, ... });
}
// 路径 C：Once 执行（通过 Pi SDK bash 工具）
return runExecCommandOnce({ bashTool, toolCallId, command: renderedCommand, ... });
路径函数适用场景输出方式
A. TTYstartExecCommandTty()交互式长时间运行（dev server、REPL）通过 PTY session 持续交互
B. DirectrunExecCommandDirect()有 OS 沙盒化 exec 函数时CommandOutputCollector 流式收集
C. OncerunExecCommandOnce()通过 Pi SDK bash 工具执行bashTool.execute 一次性获取结果
ATTY 路径：通过 node-pty 创建伪终端，支持 cols x rows 尺寸控制和 xterm-256color。返回 terminalId（即 processId），后续通过 write_stdin 工具发送输入。Windows 上用 taskkill /T /F 确保进程树完整终止。
lib/exec-command/runner.ts + lib/terminal/node-pty-backend.ts
BDirect 路径（主线）：当沙盒开启且有 commandExec（被 bwrap/seatbelt 包装的 spawn 函数）时走此路径。创建 CommandOutputCollector 进行流式管理，是最常用也最安全的路径。
lib/exec-command/runner.ts — runExecCommandDirect()
COnce 路径（兜底）：当没有 OS 沙盒化 exec 函数时，回退到 Pi SDK 的 bashTool.execute() 一次性获取结果。缺少流式输出和细粒度超时控制。
lib/exec-command/runner.ts — runExecCommandOnce()
路径 B 是主线：当沙盒开启且有 commandExec（被 bwrap/seatbelt/restricted-token 包装的 spawn 函数）时，走 Direct 路径。这是最常用也最安全的路径。路径 C 是兜底——当没有 OS 沙盒时回退到 Pi SDK 的 bash 工具。

CommandOutputCollector：双缓冲设计

路径 B 使用 CommandOutputCollector（lib/exec-command/runner.ts）进行流式输出管理，解决了"长时间运行的命令可能产生大量输出，需要同时支持流式回传和最终截断"的关键问题：

// lib/exec-command/runner.ts — CommandOutputCollector 核心逻辑
class CommandOutputCollector {
  private chunks: Buffer[] = [];
  private chunksBytes = 0;
  private totalBytes = 0;
  private tempFilePath: string | undefined;

  append(data: Buffer) {
    this.totalBytes += chunk.length;
    // 超过 50KB 阈值时自动创建临时文件保存完整输出
    if (this.totalBytes > DEFAULT_MAX_BYTES) this.ensureTempFile();
    if (this.tempFileStream) this.tempFileStream.write(chunk);
    // 内存中保留最近 100KB 用于实时预览（滚动窗口）
    this.chunks.push(chunk);
    this.chunksBytes += chunk.length;
    while (this.chunksBytes > MAX_ROLLING_BYTES && this.chunks.length > 1) {
      const removed = this.chunks.shift();
      this.chunksBytes -= removed?.length || 0;
    }
  }
}

收集器采用"双缓冲"策略：内存中的滚动窗口（最近 100KB）用于流式快照回传给 LLM，超过阈值后自动写入临时文件保留完整输出。最终结果构建时优先返回内存内容，超限则附上临时文件路径。

二、OS 级沙盒执行（Layer 7）

HanaAgent 在三个平台上采用不同的 OS 级沙盒机制：

平台机制关键文件隔离原理
LinuxBubblewrap (bwrap)lib/sandbox/bwrap.tsLinux user namespace + bind mount
macOSSeatbelt (sandbox-exec)lib/sandbox/seatbelt.tsApple Sandbox SBPL 策略文件
WindowsRestricted Tokenlib/sandbox/win32-exec.tsWindows 访问令牌降权

Shell 命令运行器（command-runner.ts）

lib/shell/command-runner.ts 中的 createCommandRunner() 是连接"工具层"与"OS 沙盒层"的关键桥梁。它根据平台解析 Shell profile（可执行文件路径 + 参数模板），将命令包装为可 spawn 的完整调用：

// lib/shell/command-runner.ts — Shell 命令运行器工厂
export function createCommandRunner({ resolveProfile, spawnCommand, platform }) {
  return async function runShellCommand(command, cwd, opts = {}) {
    const profile = resolveProfile({ platform, profile: opts.profile || "default", env });
    return spawnCommand({
      executable: profile.executable,
      args: profile.argsForCommand(String(command ?? "")),
      cwd, env: profile.env || env,
      onData: opts.onData, signal: opts.signal, timeout: opts.timeout,
    });
  };
}

当沙盒开启时，spawnCommand 指向被 OS 沙盒包装后的 spawn 函数，从而在 Shell profile 解析之后、实际进程启动之前插入 OS 级隔离层。

Linux Bubblewrap 参数构造

buildBwrapArgs()（lib/sandbox/bwrap.ts）构造完整的 bwrap 参数列表，典型调用为：bwrap [args...] -- /bin/bash /tmp/hana-script-XXXX.sh

// lib/sandbox/bwrap.ts — 参数构造策略（核心参数）
const args = [
  // 1. 根文件系统：只读绑定 / → /（限制系统文件修改）
  ...(readAll ? ["--ro-bind", "/", "/"] : []),
  // 2. 虚拟文件系统
  "--dev", "/dev",        // 设备节点
  "--proc", "/proc",      // 进程信息
  "--tmpfs", "/tmp",      // 独立临时目录
  // 3. 进程隔离
  "--unshare-pid",        // 新建 PID namespace
  "--new-session",        // 脱离终端控制
  "--die-with-parent",    // 父进程退出时自动清理
];
if (!allowNetwork) args.push("--unshare-net"); // 可选网络隔离

// 4. 工作目录：可写绑定
if (cwd && fs.existsSync(cwd)) {
  addMount(args, "--bind", cwd, cwd, ...);
  args.push("--chdir", cwd);
}
// 5. 受保护路径：在可写范围内覆盖为只读
for (const p of policy.protectedPaths) addMount(args, "--ro-bind", p, p, ...);
// 6. 读取拒绝：文件绑 /dev/null，目录绑 tmpfs
for (const p of policy.denyReadPaths) {
  fs.statSync(p).isDirectory() ? args.push("--tmpfs", p) : args.push("--bind", "/dev/null", p);
}

macOS Seatbelt 沙盒

macOS 使用 sandbox-exec + SBPL (Sandbox Profile Language) 策略文件：sandbox-exec -f /tmp/hana-profile-XXXX.sb /bin/bash /tmp/hana-script-XXXX.sh。策略文件以 (deny default) 开头，默认拒绝一切，仅显式 allow 的路径可访问。

exec-helper.ts：跨平台 spawn 底层

所有平台的 OS 沙盒最终通过 spawnAndStream()（lib/sandbox/exec-helper.ts）执行，处理进程 spawn、流式数据回调、超时机制（killTree 杀进程树）、AbortSignal 中断、退出码处理、以及进程退出后 100ms 的 grace 延迟确保管道排空不丢尾部输出。

沙盒不是全局开关：沙盒通过 getSandboxEnabled 回调在每次工具调用时动态求值。用户可以在运行时切换沙盒偏好而无需重建 session。沙盒关闭时回退到 normalBashTool（直接 spawn，无 OS 隔离）。

三、输出收集与结果回传（Layer 8）

命令执行完成后，最终阶段是将输出格式化并返回给 Agent。

输出截断策略

采用"从末尾向前保留"策略，保留最后 2000 行或 50KB：

// lib/exec-command/runner.ts — 截断策略（核心逻辑）
function truncateTail(content, { maxLines = 2000, maxBytes = 50 * 1024 } = {}) {
  const lines = content.split("\n");
  for (let i = lines.length - 1; i >= 0 && outputLinesArr.length < maxLines; i--) {
    const lineBytes = Buffer.byteLength(line, "utf-8");
    if (outputBytesCount + lineBytes > maxBytes) break; // 超过字节上限
    outputLinesArr.unshift(line);
    outputBytesCount += lineBytes;
  }
  // 截断说明附在返回文本中，含完整输出临时文件路径
  return { content, truncated, truncatedBy, totalLines, outputLines };
}

截断不是简单丢弃数据——系统会创建临时文件保存完整输出，并在返回文本中附上路径说明（如 Full output: /tmp/hana-exec-command-a1b2c3d4.log），LLM 可通过 read 工具读取完整文件。

编码检测

在 Windows 平台上，命令输出可能是 GBK 编码。先尝试 UTF-8 解码，失败后回退到 GBK：

export function decodeCommandOutput(buffer: Buffer, { platform } = {}) {
  if (platform === "win32" && !isValidUtf8(buffer)) {
    try { return { text: new TextDecoder("gbk").decode(buffer), encoding: "gbk" }; } catch {}
  }
  return { text: buffer.toString("utf-8"), encoding: "utf-8" };
}

四、检验一下（提取练习）

Q1. 三路径执行模型中，哪个路径使用了 CommandOutputCollector？

A. TTY 路径（startExecCommandTty） B. Direct 路径（runExecCommandDirect） C. Once 路径（runExecCommandOnce） D. 三个路径都使用
看答案B。只有 Direct 路径创建 new CommandOutputCollector() 进行流式输出收集。TTY 路径通过 PTY session 持续交互，Once 路径通过 bashTool.execute 一次性获取结果。

Q2. Bubblewrap 的 --die-with-parent 参数的作用是什么？

A. 当子进程退出时杀死父进程 B. 当父进程（HanaAgent）退出时自动清理沙盒进程，防止孤儿进程泄漏 C. 当沙盒内的进程崩溃时重启它 D. 当网络断开时终止沙盒
看答案B。--die-with-parent 确保当 HanaAgent 进程退出或被 kill 时，bwrap 沙盒内的所有子进程也会被自动清理，避免孤儿进程残留。

Q3. CommandOutputCollector 为什么使用"双缓冲"设计？

A. 为了支持同时输出到 stdout 和 stderr B. 内存滚动窗口（最近 100KB）用于实时流式回传 LLM，临时文件保存完整输出用于最终结果构建 C. 为了支持 UTF-8 和 GBK 双编码 D. 为了同时支持 Linux 和 Windows
看答案B。双缓冲解决了"长输出需要流式实时反馈"和"最终结果需要完整或截断后返回"的矛盾。内存中的 100KB 滚动窗口保证实时快照的低延迟，临时文件保证不丢失数据。
