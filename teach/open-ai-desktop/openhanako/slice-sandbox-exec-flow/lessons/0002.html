第 2 课 · 路径守卫与命令分类：PathGuard 11级级联+preflight检查 安全核心

本节目标（对应使命 Success #2）：学完你能解释 classifyExecCommand() 的五级风险分类体系，掌握 PathGuard 的 11 级级联判定逻辑和四级访问级别（BLOCKED/READ_ONLY/READ_WRITE/FULL），理解 preflight 模式匹配如何拦截高危命令。

上一课我们了解了 8 层架构的全貌。这一课深入到第 3 层（命令风险分级）和第 4 层（PathGuard 路径判决）——这是整个安全体系中最核心的决策引擎。所有命令在进入 OS 沙盒前，都必须经过这两层的检查。

一、命令风险五级分类（Layer 3）

classifyExecCommand()（lib/exec-command/policy.ts）将命令分为五个风险等级，决定后续的审批策略：

级别含义示例命令典型处理
safe 安全的只读操作，无副作用 git status、cat、ls、Get-ChildItem 直接执行，通常不需要审批
probe 环境探测，检查依赖可用性 node --version、which npm 允许执行；失败标记为依赖缺失
mutation 本地文件变更（创建、修改） touch file.txt、mkdir dir 需 PathGuard 写权限校验 + 可能触发审批
dangerous 破坏性操作，需高度警惕 rm -rf、git push -f、reg delete 强制审批 + PathGuard FULL 权限校验
unknown 未分类命令，默认保守处理 任意不在白名单中的可执行文件 按潜在危险处理，需审批

命令分词器：避免字符串欺骗

风险分析的第一步是将命令字符串拆分为 token。分词器正确处理引号和转义，避免被 echo "rm -rf /" 这类命令欺骗：

// lib/exec-command/policy.ts — 命令分词核心逻辑
function tokenizeCommand(command: string) {
  const tokens: string[] = [];
  let token = "", quote = null, escaped = false;
  for (const char of command) {
    if (escaped) { token += char; escaped = false; continue; }
    if (char === "\\") { escaped = true; continue; }
    if (quote) {
      if (char === quote) quote = null;
      else token += char;
      continue;
    }
    if (char === '"' || char === "'") { quote = char; continue; }
    if (/\s/.test(char)) { if (token) tokens.push(token); token = ""; continue; }
    token += char;
  }
  if (token) tokens.push(token);
  return tokens;
}

Git 命令的特殊处理

Git 命令有独立的分支逻辑 classifyGit()：git status/log/diff 为 safe，git push 为 mutation，git push -f/--force/--all/--mirror 为 dangerous。

局限性：policy.ts 的分类是静态的、基于文件名的。它不能检测 Shell 注入（如 curl example.com | bash 中的管道后命令），这是 PathGuard preflight 和 OS 沙盒的职责。

二、PathGuard 四级访问级别（Layer 4）

PathGuard（lib/sandbox/path-guard.ts）是应用层安全的核心组件，对每个文件路径返回四个访问级别之一：

级别readwritedelete典型场景
BLOCKED否否否auth.json、browser-data、未授权外部路径
READ_ONLY是否否config.yaml、session-files、user/ 目录
READ_WRITE是是否memory/、sessions/、channels/、logs/
FULL是是是workspace roots（用户项目目录）

操作所需的最低权限级别由 OP_REQUIREMENTS 映射定义：

// lib/sandbox/path-guard.ts — 操作-权限映射
const OP_REQUIREMENTS = {
  read:   new Set([READ_ONLY, READ_WRITE, FULL]),
  write:  new Set([READ_WRITE, FULL]),
  delete: new Set([FULL]),
};

三、11 级级联判定流程

getAccessLevel(rawPath) 按优先级从高到低依次匹配 11 个条件，第一个命中即返回。这种级联设计确保"拒绝优先于允许，精确匹配优先于通配"：

级联判定顺序（优先级从高到低，逐级匹配，命中即返回）：
1. BLOCKED 文件      → hanakoHome 根级别黑名单文件
2. BLOCKED 目录      → hanakoHome 根级别黑名单目录
3. READ_ONLY agent文件 → agent 目录下只读文件（ishiki.md、config.yaml 等）
4. READ_ONLY agent目录 → 通用机制（当前为空数组）
5. READ_ONLY 全局目录  → hanakoHome 下的 user/、skills/、session-files/
6. READ_WRITE agent目录 → agent 目录下 7 个可写目录（memory/、sessions/等）
7. READ_WRITE agent文件 → agent 目录下可写文件（pinned.md、channels.md）
8. READ_WRITE 全局目录 → hanakoHome 下 channels/、logs/、uploads/、.ephemeral/
9. hanakoHome 未匹配   → 根据 allowExternalReads 决定 READ_ONLY 或 BLOCKED
10. workspace roots    → 用户项目目录，FULL 权限
11. policyWritablePaths → 运行时缓存、沙盒临时目录等非 workspace 写入点
12. 其他（兜底）        → 根据 allowExternalReads 决定 READ_ONLY 或 BLOCKED

关键机制拆解

1符号链接跟踪：PathGuard 在执行任何检查前，先通过 _resolveReal() 解析路径的真实位置。对于存在的路径使用 fs.realpathSync()；对于不存在的路径，递归向上找最近祖先做 realpath。
lib/sandbox/path-guard.ts — _resolveReal()
2级联判定：getAccessLevel() 从优先级 1（BLOCKED 文件）到优先级 12（兜底），逐级 if 判断，第一个命中即返回。拒绝优先于允许，精确匹配优先于通配。
lib/sandbox/path-guard.ts — getAccessLevel()
为什么需要符号链接跟踪？如果用户创建了一个符号链接 ln -s /etc/passwd ~/project/link，不跟踪符号链接的话 PathGuard 会认为这是在 workspace 内的安全路径。跟踪后会发现真实路径指向 /etc/passwd，属于 BLOCKED 级别。

四、Preflight 模式匹配

对于 bash 命令执行，tool-wrapper.ts 的 wrapBashTool() 在 PathGuard 检查之前先运行 preflight 正则匹配：

// lib/sandbox/tool-wrapper.ts — preflight 模式匹配（部分）
const PREFLIGHT_UNIX = [
  [/\bsudo\s/,    () => t("sandbox.noSudo")],
  [/\bsu\s+\w/,   () => t("sandbox.noSu")],
  [/\bchmod\s/,   () => t("sandbox.noChmod")],
  [/\bchown\s/,   () => t("sandbox.noChown")],
];

const PREFLIGHT_WIN32 = [
  [/\bdel\s+\/s/i,              () => t("sandbox.noDelRecursive")],
  [/\brmdir\s+\/s/i,            () => t("sandbox.noRmdirRecursive")],
  [/\breg\s+(delete|add)\b/i,   () => t("sandbox.noRegEdit")],
  [/\btakeown\b/i,              () => t("sandbox.noTakeown")],
  [/\bicacls\b/i,               () => t("sandbox.noIcacls")],
  [/\bformat\s+[a-z]:/i,        () => t("sandbox.noFormat")],
  [/\bbcdedit\b/i,              () => t("sandbox.noBcdedit")],
];

Preflight 在 macOS/Linux 上作为体验层优化（OS 沙盒兜底），在 Windows 上作为关键安全层。命中后直接拦截，命令不会进入 PathGuard 检查和 OS 沙盒。

五、Managed Config Guard

除了 PathGuard，还有一个更精细的 ManagedConfigWriteGuard（lib/sandbox/managed-config-guard.ts），专门保护 agents/*/config.yaml 文件。即使路径在 PathGuard 中属于可写区域，此 guard 也会拦截对 agent 配置文件的直接写入，强制通过 settings API 修改：

// lib/sandbox/managed-config-guard.ts
export function createManagedConfigWriteGuard({ hanakoHome }) {
  return (absolutePath, operation) => {
    if (operation !== "write" && operation !== "delete") return { allowed: true };
    if (!isManagedAgentConfigPath(absolutePath, { hanakoHome })) return { allowed: true };
    return {
      allowed: false,
      reason: "managed config files must be changed through settings APIs",
    };
  };
}

六、检验一下（提取练习）

Q1. 某命令的第一个可执行文件名为 mkdir，它会被 classifyExecCommand 分到哪个级别？

A. safe B. probe C. mutation D. dangerous
看答案C。mkdir 是本地文件变更操作，属于 mutation 级别。需 PathGuard 写权限校验，可能触发审批。safe 是只读操作（如 ls、cat），dangerous 是破坏性操作（如 rm -rf）。

Q2. PathGuard 的级联判定中，如果路径同时匹配了 READ_WRITE agent 目录（优先级 6）和 workspace roots（优先级 10），最终返回什么级别？

A. READ_WRITE B. FULL C. 合并为 READ_WRITE（取更严格者） D. 抛出冲突异常
看答案A。级联判定是"第一个命中即返回"。优先级 6 的 READ_WRITE agent 目录先命中，直接返回 READ_WRITE，不会到达优先级 10 的 workspace roots（FULL）。级联设计确保精确规则优先于宽泛规则。

Q3. preflight 正则匹配和 PathGuard 检查的关系是什么？

A. 两者并行执行，任一失败即拦截 B. preflight 先执行，命中则直接拦截（不进入 PathGuard）；未命中才进入 PathGuard C. PathGuard 先执行，命中则直接拦截；未命中才进入 preflight D. preflight 只在 Windows 上执行，PathGuard 在所有平台执行
看答案B。在 wrapBashTool() 中，preflight 正则匹配最先运行。如果命中了 sudo/chmod/reg delete 等模式，直接返回 blockedResult，不会进入后续的 PathGuard 检查。preflight 是一个快速的模式匹配层。
