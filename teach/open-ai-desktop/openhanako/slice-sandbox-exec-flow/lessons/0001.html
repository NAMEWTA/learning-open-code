第 1 课 · 沙盒架构总览：安全模型+分层设计+入口点 架构全景

本节目标（对应使命 Success #1）：学完你能画出从 Agent tool_use 到 OS 沙盒执行的完整 8 层数据流，理解双层沙盒模型（PathGuard + OS 隔离）的设计哲学，知道 exec_command 工具的注册方式和参数定义。

HanaAgent 的命令执行安全体系采用 双层沙盒模型，这是整个安全架构的基石。源文件分布在 lib/exec-command/、lib/sandbox/、lib/permission/、lib/shell/ 四个模块中，共涉及 20+ 个关键文件。这一课我们先建立全景认知。

一、双层沙盒模型：纵深防御

1应用层（PathGuard）：纯 JavaScript 实现的四级访问控制（BLOCKED / READ_ONLY / READ_WRITE / FULL），通过 11 级级联条件判定每个路径的操作权限。作为第一道防线，在任何文件 I/O 发生前拦截。
lib/sandbox/path-guard.ts
2OS 层（Bubblewrap / Seatbelt / Windows Restricted Token）：利用操作系统内核机制进行进程级隔离，将命令执行限制在受控文件系统子集中。作为纵深防御的第二道防线。
lib/sandbox/bwrap.ts / seatbelt.ts / win32-exec.ts
关键设计原则：这两层是"与"的关系而非"或"——PathGuard 负责"文件级别的访问控制"，OS 沙盒负责"进程级别的隔离"，两者互补。单层失效不会导致整体失陷。

二、8 层数据流：从 Agent 发起到结果回传

Agent 模型输出 tool_use(exec_command)
→ Tool Registry 路由
→ createExecCommandTools.execute() 参数规范化
→ classifyExecCommand() 命令风险分级（safe/probe/mutation/dangerous/unknown）
→ resolveExecShell() Shell 解析（powershell/cmd/bash/posix）
→ PathGuard.check() 应用层路径访问判决（11 级级联）
→ 权限审批拦截（sessionPermission + safety-policy）
→ runner 选择（runExecCommandDirect / runExecCommandOnce / startExecCommandTty）
→ OS 沙盒 spawn（bwrap / sandbox-exec / restricted-token）
→ CommandOutputCollector 流式收集 → 截断 + 编码检测 → 结果回传 Agent

三、第 1 层 — 工具注册：exec_command 如何被激活

当 LLM 模型输出 tool_use(exec_command, ...) 时，Pi SDK 的 Tool Registry 将其路由到对应实现。exec_command 工具在 lib/sandbox/index.ts 中通过工厂函数装配：

// lib/sandbox/index.ts — 工具装配入口
const createExecToolsForBash = (bashTool, commandExec = null) => createExecCommandTools({
  bashTool,         // Pi SDK bash 工具（已被 OS 沙盒包装）
  commandExec,      // 直接 exec 函数（Windows restricted-token 包装）
  getTerminalSessionManager,
  getAgentId,
  getCwd: () => cwd,
  platform: process.platform,
});

这里 bashTool 已经是经过 wrapBashTool() 包装的工具——它的 execute 方法内部会先调用 PathGuard 进行 preflight 检查，再委托给 OS 沙盒化的 exec 函数。这就是双层沙盒模型在代码层面的具体体现。

工具向 LLM 声明如下参数 schema：

// lib/exec-command/tool.ts — 参数 schema（精简）
parameters: Type.Object({
  cmd: Type.String(),                    // 要执行的命令（Windows 上默认 PowerShell）
  workdir: Type.Optional(Type.String()), // 工作目录，默认 session cwd
  shell: Type.Optional(Type.String()),   // 可选 shell 覆盖：auto|powershell|pwsh|cmd|bash
  tty: Type.Optional(Type.Boolean()),    // 启动交互式 PTY 进程
  timeout: Type.Optional(Type.Number()), // 超时秒数
  max_output_tokens: Type.Optional(Type.Number()),
  yield_time_ms: Type.Optional(Type.Number()),
}),

四、第 2 层 — 参数规范化与 Shell 解析

工具入口的第一步是参数规范化，由 normalizeExecCommandParams()（lib/exec-command/schema.ts）处理：

// 1. 命令文本提取：支持 cmd 和 command 两种参数名
const cmd = typeof params.cmd === "string" ? params.cmd
  : typeof params.command === "string" ? params.command : "";

// 2. 工作目录解析：workdir > cwd > sessionManager.getCwd() > defaultCwd
const workdir = params.workdir || params.cwd
  || ctx?.sessionManager?.getCwd?.() || defaultCwd || process.cwd();

// 3. 数值参数：maxOutputTokens 默认 6000、yieldTimeMs 默认 10000
const maxOutputTokens = optionalPositiveNumber(params.max_output_tokens, 6000);

五、检验一下（提取练习）

Q1. HanaAgent 的双层沙盒模型是哪两层？

A. Docker 容器 + Kubernetes Pod 隔离 B. 应用层 PathGuard（文件级访问控制）+ OS 层 Bubblewrap/Seatbelt（进程级隔离） C. SELinux + AppArmor D. chroot + seccomp
看答案B。PathGuard 是纯 JS 实现的四级访问控制，作为第一道防线；Bubblewrap/Seatbelt/Restricted Token 利用 OS 内核机制做进程级隔离，作为纵深防御。两者是"与"的关系，互补而非重复。

Q2. exec_command 工具的参数中，shell 参数的作用是什么？

A. 指定要执行的 Shell 脚本文件路径 B. 覆盖平台默认 Shell，可选值包括 auto/powershell/pwsh/cmd/bash C. 启用 Shell 注入保护 D. 设置环境变量 SHELL 的值
看答案B。shell 参数允许 LLM 或用户覆盖平台默认 Shell。在 Windows 上默认 PowerShell，可以切换到 cmd 或 Git Bash；在 Linux/macOS 上默认 POSIX shell，可以指定 bash。

Q3. 8 层数据流中，L2（参数规范化）和 L3（命令风险分级）的执行顺序是什么？

A. 先风险分级，再参数规范化 B. 先参数规范化，再风险分级——因为需要先从参数中提取 cmd 字符串才能分析 C. 两者并行执行 D. 风险分级在参数规范化内部执行
看答案B。必须先通过 normalizeExecCommandParams() 从参数对象中提取出 cmd 字符串，然后才能调用 classifyExecCommand(cmd) 进行风险分析。这是顺序依赖关系。
