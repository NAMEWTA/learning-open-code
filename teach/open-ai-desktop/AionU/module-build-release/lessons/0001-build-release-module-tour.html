构建与发布模块：从一个 dist 命令读懂发布边界

学习目标：15 分钟内说清 AionU 如何把源码变成桌面包和 WebCLI 发布资产，并知道失败时从哪里开始排查。

L1 模块总览 预计 12 分钟 主来源：构建脚本 + electron-builder

1. 模块职责

构建与发布模块的职责不是“编译 TypeScript”这么窄，而是把 AionU 的多运行形态变成可分发、可更新、可校验的产物。它位于 L0 构建与部署 之后，向上承接 open-ai-desktop/AionU/package.json 的脚本入口，向下调用 Electron Vite、Electron Builder、aioncore 资源准备、Hub 离线资源、WebCLI 打包和发布资产校验。

读这个模块时先记住两个边界：桌面包由 open-ai-desktop/AionU/scripts/build-with-builder.js 驱动，最终受 open-ai-desktop/AionU/packages/desktop/electron-builder.yml 约束；WebCLI 包由独立脚本生成 tarball，再交给发布资产整理脚本统一校验。

2. 一条主线

以 bun run dist:mac 为例，入口不是直接调用 electron-builder，而是先进入构建协调脚本：

{
  "dist:mac": "node scripts/build-with-builder.js auto --mac",
  "build-mac": "node scripts/build-with-builder.js auto --mac --arm64 --x64",
  "package": "electron-vite build --config packages/desktop/electron.vite.config.ts"
}
第一步：计算源码 hash，必要时执行 electron-vite build，生成 out/main、out/preload、out/renderer。
第二步：打包内置 MCP server，并检查 main 与 renderer 输出是否存在。
第三步：准备 aioncore 和 Hub 离线资源，确保运行时不只依赖前端 bundle。
第四步：调用 electron-builder，由配置文件决定 dmg/zip、nsis、deb、asar、extraResources、afterPack 与 afterSign。

3. 分层地图

module-build-release/ ├── package.json ← 用户与 CI 可调用命令 ├── packages/ │ ├── desktop/ │ │ ├── electron.vite.config.ts ← main/preload/renderer 编译 │ │ └── electron-builder.yml ← 桌面分发物与资源白名单 │ └── shared-scripts/ │ └── src/ ← aioncore 准备与校验 ├── scripts/ ← 构建、发布、安装、冒烟脚本 ├── homebrew/ ← Homebrew cask 模板 └── tests/ └── unit/ ← 构建脚本与资源校验断言

这张地图也说明了依赖方向：构建脚本依赖桌面编译配置和共享资源脚本；afterPack 反过来调用共享校验，确认打进包内的 runtime 可用。桌面运行时怎么启动这些资源，属于 module-main-entry；WebCLI 运行起来后如何服务 WebUI，属于 module-web-runtime。

4. 检索练习

暂停回忆：如果 macOS 构建已经生成 `.app`，但没有生成 `.dmg`，你应该先查哪个脚本里的哪类逻辑？如果打包后缺少 `managed-resources`，又该查哪里？

参考答案与反馈

先查 scripts/build-with-builder.js 的 DMG retry 逻辑，它会在 `.app` 存在但 `.dmg` 缺失时尝试用 electron-builder --prepackaged 重建分发物。若缺少 managed-resources，先看 packages/shared-scripts/src/prepare-aioncore.js 是否生成 bundle，再看 verify-bundled-aioncore-resources.js 和 scripts/afterPack.js 的校验失败信息。
