mobile/ 怎样把 AionU 后端能力带到手机上

学习目标：15 分钟内看懂 AionU 的移动端工程怎样完成连接、路由落点和会话通信，把手机变成远程 Cowork 入口。

L1 模块总览 预计 15 分钟 主来源：app / context / services

1. 先定模块位置：它不是独立后端，而是移动壳

open-ai-desktop/AionU/mobile/package.json 把这个模块定义成一个基于 Expo 的独立移动端工程，入口是 expo-router/entry。它在 AionU 总体架构里的位置，和 L0 里桌面 renderer 的角色很像：自己不实现 Cowork 核心能力，而是把远端 AionU backend 暴露出的鉴权、会话、消息流和文件能力，重组为适合手机的交互入口。

因此读 mobile/ 最重要的判断不是“页面有哪些”，而是“手机怎样拿到后端状态”。对应答案集中在四层：app/ 决定页面入口，context/ 保存连接与业务状态，services/ 负责 REST 和 WebSocket 协议，utils/ 处理消息、工作区和时间线等适配细节。完整模块清单请看 reference；如果要先回到全仓库地图，打开 L0 项目地图短课。

2. 第一条主链：从扫码连接到聊天页

open-ai-desktop/AionU/mobile/app/connect.tsx 是移动端最关键的外部入口。它支持扫码或粘贴 /qr-login 链接，先调用后端的 /api/auth/qr-login 换到 JWT，再把 host / port / token 交给 ConnectionContext。只有等 WebSocket 真正进入 connected，页面才会跳到 /(tabs)/chat。这说明移动端把“拿到 token”和“连上消息流”分成了两个闸门。

const response = await axios.post(`http://${host}:${port}/api/auth/qr-login`, {
  qrToken,
});
const jwt: string = response.data.token;

await connect(host, port, jwt);

await new Promise<void>((resolve, reject) => {
  const unsub = wsService.onStateChange((state) => {
    if (state === 'connected') resolve();
    if (state === 'auth_failed') reject(new Error('auth_failed'));
  });
});

router.replace('/(tabs)/chat');
graph LR U[手机用户] --> C[connect.tsx 扫码或粘贴] C --> A[/api/auth/qr-login/] A --> K[JWT] K --> CC[ConnectionContext.connect] CC --> WS[wsService.connect] WS --> B[bridge request/on] B --> TAB[/(tabs)/chat]

与桌面端不同，这里没有 preload 或 Electron IPC。移动端直接面对远端后端，所以连接页本身就是“移动 Cowork 入口”的门厅。

3. 第二条主链：provider 栈怎样把远端能力变成页面状态

open-ai-desktop/AionU/mobile/app/_layout.tsx 先初始化 i18n，再按 ConnectionProvider → WebSocketProvider → ConversationProvider → WorkspaceProvider → FilesTabProvider 的顺序包住整个路由树。意思很直接：先保证知道连哪台服务器，再保证消息通道可用，然后才谈会话列表、当前工作区和文件标签页。

<ConnectionProvider>
  <WebSocketProvider>
    <ConversationProvider>
      <WorkspaceProvider>
        <FilesTabProvider>
          <Stack screenOptions={{ headerShown: false }}>
            <Stack.Screen name='index' />
            <Stack.Screen name='connect' />
            <Stack.Screen name='(tabs)' />
          </Stack>
        </FilesTabProvider>
      </WorkspaceProvider>
    </ConversationProvider>
  </WebSocketProvider>
</ConnectionProvider>

真正把“后端消息”翻译成“聊天页可渲染状态”的，是 ChatContext.tsx。它通过 bridge.on('chat.response.stream') 订阅流式事件，再借助 transformMessage() 和 composeMessage() 把原始协议消息拼成移动端消息列表。所以移动端的 UI 不直接理解底层协议，它依赖 context 把协议层折叠成可消费的 React 状态。

4. 第三条主链：bridge 把后端协议收口，测试要分层看待

open-ai-desktop/AionU/mobile/src/services/websocket.ts 管连接寿命：携带 token 建立连接、处理 ping/pong、指数退避重连、鉴权过期恢复；open-ai-desktop/AionU/mobile/src/services/bridge.ts 再在它之上提供 request / emit / on 三种更稳定的调用形式。页面和 context 因而不必自己拼 WebSocket 报文，只需要按业务名调用诸如 database.get-user-conversations、chat.send.message、confirmation.confirm 这样的桥接接口。

request<T = unknown>(name: string, data?: unknown, timeoutMs = 30000): Promise<T> {
  const id = generateRequestId();
  const callbackName = `subscribe.callback-${name}${id}`;

  return new Promise<T>((resolve, reject) => {
    const unsub = this.on(callbackName, (responseData) => {
      unsub();
      resolve(responseData as T);
    });

    wsService.send(`subscribe-${name}`, { id, data });
  });
}

这条链路有专门的回归证据，但要分层看：open-ai-desktop/AionU/mobile/__tests__/services/websocket.test.ts 检查连接、心跳、鉴权失败与重连；open-ai-desktop/AionU/mobile/__tests__/utils/messageAdapter.test.ts 检查流式消息怎样合并成 UI 消息；open-ai-desktop/AionU/mobile/__tests__/services/bridge.test.ts 表达了订阅、一次性订阅、请求超时和回调清理这些抽象语义。当前 bridge.ts 已通过 subscribe-${name} 发送请求，而该测试里的直发事件名断言需要后续校准，所以不要把它当成 subscribe 发送协议已经完全锁死的证据。

暂停回忆：如果用户扫码成功后拿到了 JWT，但聊天页仍然没有打开，你会优先检查 connect.tsx 的哪个等待点？

参考答案与反馈

先看它是否等到了 wsService.onStateChange 的 connected。因为页面跳转不是在 REST 换 token 后立刻发生，而是在 WebSocket 会话真正连通后触发。若你只盯着 axios.post('/api/auth/qr-login')，说明你抓到了鉴权，却还没抓到移动端真正依赖的是“长连接就绪”。

推荐主来源：先读 app/connect.tsx，再顺着 ConnectionContext.tsx → websocket.ts → bridge.ts → ChatContext.tsx 走一遍。接着用 reference 路由表 和 测试清单 做交叉验证。
