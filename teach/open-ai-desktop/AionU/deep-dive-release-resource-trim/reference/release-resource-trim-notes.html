2. electron-builder 边界

配置保留内容裁剪内容风险
open-ai-desktop/AionU/packages/desktop/electron-builder.yml 的 filesout/main、out/preload、out/renderer、public、package.json、必要 runtime deps。workspace sibling packages 源码、desktop src、source map、node_modules 文档/测试、部分大包、JAR/JNI、tree-sitter native binaries。漏掉 runtime dep 会运行时报错；误包含 unsigned native/vendor 会影响签名或包体。
extraResourcespublic、app.png、bundled-aioncore、hub。不在此处做裁剪，依赖上游准备目录本身正确。删掉 bundled-aioncore 或 hub 会造成后端启动/离线资源缺失。
asarUnpacknative modules、tree-sitter WASM、内置 MCP server 脚本。不是任意资源都应解包。需要外部进程执行或原生加载的文件若留在 asar 内，可能无法被运行时读取。
afterPack打包后检查 resources 目录，并执行 bundled-aioncore 校验。不负责生成资源。这是发现 electron-builder 复制后缺失的最后一层本地闸门。

3. bundled-aioncore 不变量

不变量来源校验方式
目录名必须是 <electronPlatformName>-<targetArch>。open-ai-desktop/AionU/packages/shared-scripts/src/verify-bundled-aioncore-resources.js用入参组合 runtimeKey，然后在 resources 下查找。
后端 binary 必须存在。Windows 为 aioncore.exe，其他平台为 aioncore。同上requireRelativePath 加 isFile。
来源优先级：完整本地 bundle 先短路；否则依次尝试 Actions artifact、release asset、本地 binary。open-ai-desktop/AionU/packages/shared-scripts/src/prepare-aioncore.js完整 bundle 必须同时有目标 binary 与 managed-resources；只给 binary 时仍会重新生成 managed resources。
manifest.json 必须是有效 JSON，且 platform/arch 匹配目标。同上不匹配时把 <platform:...> 或 <arch:...> 放入 missing。
managed Node runtime 必须存在。同上Windows 查 node.exe；非 Windows 查 bin/node。
codex-acp 与 claude-agent-acp 必须有 manifest、entrypoint、package、lockfile、node_modules。同上枚举版本目录和 runtimeKey 子目录逐项检查。
Windows ACP 平台可执行文件必须存在。同上codex-acp 查 bin/codex-acp.exe，claude-agent-acp 查 claude.exe。

4. release-assets 校验

脚本当前硬校验注意事项
open-ai-desktop/AionU/scripts/prepare-release-assets.sh去重 distributable basename；写 canonical updater metadata；要求 latest.yml、latest-mac.yml、latest-linux.yml、latest-linux-arm64.yml；要求 mac x64/arm64 的 dmg+zip；要求五个平台 WebCLI tarball 和 sha256；要求 install-web.sh。使用 package version 或 MOCK_VERSION；不动态逐项检查 Windows exe 和 Linux deb，只通过 metadata 与后续脚本间接覆盖。
open-ai-desktop/AionU/scripts/verify-release-assets.shmetadata 指向文件必须存在；架构 metadata 必须存在；固定检查 1.0.0 的 Windows、macOS、Linux、WebCLI mock 文件。适合 PR 的 mock release test；不是任意真实版本 release 目录的通用校验器。

6. 新发现缺口

版本假设不一致：当前 open-ai-desktop/AionU/package.json 版本为 2.1.30，但 open-ai-desktop/AionU/scripts/verify-release-assets.sh 固定检查 1.0.0 文件名。它符合 mock release test，但不能直接作为真实版本目录的通用校验。

缺口影响对应源码行 / 脚本段可能的后续补强
verify-release-assets.sh 未读取 package version 或 metadata version。真实发布版本变化时，脚本只能在 mock 资产语境下稳定运行。固定 1.0.0 的 expected file name 段。增加 VERSION 参数或从 metadata/package.json 解析版本。
prepare-release-assets.sh 动态硬校验 mac 和 WebCLI，但 Windows/Linux 桌面包主要靠 metadata/后续 verify 间接覆盖。某些桌面资产缺失可能要到上传或用户下载链路才暴露。canonical metadata 检查与 mac/WebCLI assert 段。按 platform/arch 动态补齐 exe/deb 预期清单。
bundled-aioncore 校验关注存在性与 manifest 匹配。文件存在但不可执行、内容损坏或 checksum 不匹配时，afterPack 可能无法提前发现。verify-bundled-aioncore-resources.js 的 requireRelativePath / manifest 检查。对 binary 增加可执行位/最小运行检查，或在 smoke test 中固定覆盖。
checksum 文件只检查存在。checksum 内容格式或 tarball hash 不匹配可能未在 prepare/verify 阶段被发现。WebCLI sha256 expected file 检查段。在 verify 脚本中执行 sha256 校验。
