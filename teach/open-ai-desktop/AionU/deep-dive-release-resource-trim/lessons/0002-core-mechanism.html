bundled-aioncore：从准备到 afterPack 校验

学习目标：看懂 bundled-aioncore 的最小可用结构，并能解释校验器为什么检查这些文件。

L4 深度剖析 02 预计 13 分钟 主来源：verify-bundled-aioncore-resources.js

1. bundled-aioncore 是运行时契约

AionU 的桌面包不是只有 Electron 前端。后端 runtime 被准备到 open-ai-desktop/AionU/resources/bundled-aioncore/<platform>-<arch>，再由 open-ai-desktop/AionU/packages/desktop/electron-builder.yml 的 extraResources 复制到安装包资源目录。这个目录必须同时满足“目标平台可执行”和“托管工具资源完整”两个条件。

bundled-aioncore/<platform>-<arch>/ ├── aioncore 或 aioncore.exe ← 后端 binary ├── manifest.json ← platform、arch、version、source └── managed-resources/ ← 由 aioncore 准备的托管运行资源 ├── node/ ← Node runtime，Windows 与非 Windows 布局不同 └── acp/ ← codex-acp 与 claude-agent-acp

2. 打包前准备

准备阶段由 open-ai-desktop/AionU/packages/shared-scripts/src/prepare-aioncore.js 完成。完整本地 bundle 如果存在且同时包含 binary 与 managed-resources，会先复制并短路返回；否则才按 AIONUI_BACKEND_RUN_ID 指向的 Actions artifact、GitHub release asset、显式本地 binary 继续解析。只拿到 binary 还不够，脚本会执行后端命令生成 managed resources，再写入 manifest。

execFileSync(binaryPath, [
  '--data-dir',
  dataDir,
  'prepare-managed-resources',
  '--bundle-out',
  bundleOut,
], {
  stdio: 'inherit',
  env: { ...process.env, AIONUI_BUNDLED_MANAGED_RESOURCES: '' },
});

这就是“必须保留 managed-resources”的原因：它不是构建缓存，而是打包时预生成的运行资源。WebCLI 打包脚本也复用同一准备逻辑，然后把对应平台的 bundle 复制进 tarball。

3. 打包后校验

打包完成后，electron-builder 的 afterPack hook 会在实际 app resources 目录里调用 open-ai-desktop/AionU/packages/shared-scripts/src/verify-bundled-aioncore-resources.js。这一步比“源目录存在”更强，因为它检查的是 electron-builder 已经复制后的结果。

function verifyBundledAioncoreResources({ resourcesDir, electronPlatformName, targetArch }) {
  const runtimeKey = `${electronPlatformName}-${targetArch}`;
  const baseDir = path.join(resourcesDir, 'bundled-aioncore', runtimeKey);
  const checked = [];
  const missing = [];

  requireRelativePath(baseDir, runtimeKey, [backendBinaryName(electronPlatformName)], checked, missing);
  verifyBundleManifest(baseDir, runtimeKey, electronPlatformName, targetArch, checked, missing);
  requireRelativeDirectory(baseDir, runtimeKey, ['managed-resources'], checked, missing);
  requireManagedNode(baseDir, runtimeKey, electronPlatformName, checked, missing);
  requireManagedAcpTool(baseDir, runtimeKey, electronPlatformName, 'codex-acp', checked, missing);
  requireManagedAcpTool(baseDir, runtimeKey, electronPlatformName, 'claude-agent-acp', checked, missing);
  return { runtimeKey, checked, missing };
}

4. 检索练习

暂停回忆：如果某个 Windows 包里有 aioncore.exe，但缺少 managed-resources/acp/codex-acp/.../manifest.json，它应该通过 afterPack 吗？为什么？

参考答案与反馈

不应该。校验器不仅检查后端 binary，还会枚举 managed ACP 工具版本，要求每个工具在目标 runtimeKey 下有 manifest.json、manifest 指向的 entrypoint、package.json、package-lock.json 和 node_modules。Windows 对 codex-acp 与 claude-agent-acp 还会额外检查平台可执行文件。
