发布资源校验：脚本、测试与缺口

学习目标：能区分 release-assets 脚本校验、bundled-aioncore 单测和 CI mock release test 各自证明的范围。

L4 深度剖析 03 预计 14 分钟 主来源：prepare / verify / releasePackagingConfig

1. release-assets 校验什么

open-ai-desktop/AionU/scripts/prepare-release-assets.sh 的职责是把 CI 下载回来的多目录 artifacts 扁平化到 release-assets/，并避免同名覆盖。它复制桌面安装包、WebCLI tarball/checksum、版本替换后的 install-web.sh，再把各平台 updater metadata 规范成 canonical 名称和架构专属名称。

这一步的硬校验包括：canonical metadata 必须存在；macOS x64/arm64 的 dmg 与 zip 都必须存在；五个平台的 WebCLI tarball 和 sha256 文件必须存在；install-web.sh 必须存在。它使用当前 package.json 版本，测试里也允许用 MOCK_VERSION 固定版本。

2. verify 脚本校验什么

open-ai-desktop/AionU/scripts/verify-release-assets.sh 是独立校验入口。它先检查 canonical metadata，再从 yml 的 path 或 url 解析引用文件，确认 metadata 没有指向不存在的资产。随后它检查 latest-win-arm64.yml、latest-arm64-mac.yml，以及一组固定命名的桌面包和 WebCLI 包。

assert_metadata_points_to_existing_file "latest.yml" "(win-x64|win32-x64|x64)"
assert_metadata_points_to_existing_file "latest-mac.yml" "(mac-x64|darwin-x64|x64)"
assert_metadata_points_to_existing_file "latest-linux.yml" "(linux|AppImage|deb)"
assert_metadata_points_to_existing_file "latest-linux-arm64.yml" "(arm64|aarch64)"

这里要注意边界：当前 verify 脚本的桌面包和 WebCLI 文件名写死为 1.0.0，所以它非常适合配合 mock release artifacts 做 PR 脚本测试，却不等同于对任意真实版本 release 目录的通用校验器。

3. 测试证据怎么读

配置不变量

open-ai-desktop/AionU/tests/unit/releasePackagingConfig.test.ts 锁住 mac zip 仍启用、Windows zip 不生成、workflow 上传 mac zip、mac prepackaged retry 仍包含 dmg+zip。

缺失资产

同一测试会创建 mock artifacts，删除 arm64 mac zip，再运行 prepare-release-assets.sh，断言脚本失败并输出 “Missing macOS zip artifact”。

bundled-aioncore

对应单测覆盖 Node runtime、manifest mismatch、非 Windows node 布局、ACP manifest 和 entrypoint 缺失；完整文件路径和检查项放在参考页。

4. 检索练习

判断题：如果 release-script-test 通过，就能证明真实 2.1.30 发布包不会缺 Windows exe、Linux deb 或 WebCLI checksum。对还是错？

参考答案与反馈

错。PR 的 release-script-test 使用 mock artifact 脚本生成 1.0.0 mock 文件，再跑 prepare 与 verify，主要证明脚本契约不会明显漂移。真实发布 workflow 会运行 prepare 并上传 release-assets，但 verify 的固定 1.0.0 命名不能直接证明当前真实版本目录完整。真实包还需要结合构建 job artifact、metadata 指向、安装/冒烟测试来判断。
