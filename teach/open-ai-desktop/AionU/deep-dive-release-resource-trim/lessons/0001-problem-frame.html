发布资源裁剪：先定义误删边界

学习目标：15 分钟内建立 AionU 发布包资源裁剪的判断框架，知道哪些文件可以裁、哪些必须被校验保留。

L4 深度剖析 01 预计 12 分钟 主来源：electron-builder.yml

1. 裁剪的真实问题

发布包裁剪不是为了“越小越好”，而是在三个约束之间取平衡：安装包不能带源码、测试、文档和无关 vendor；Electron 运行时又必须能读到主进程、preload、renderer、native modules、Web/PWA 静态资源和后端 runtime；macOS 签名、公证还会被未签名 native/JNI/JAR 文件拖累。AionU 把这个边界主要写在 open-ai-desktop/AionU/packages/desktop/electron-builder.yml，发布脚本再从 CI 产物中整理用户真正下载的文件。

所以读这组配置时不要先背字段名，而要问一个更稳定的问题：这个文件属于“应用代码”“外层运行资源”“必须解包执行的二进制”，还是“发布页资产”？四类资源失败形态不同，校验点也不同。

性能维度也在这里：包体越大，下载和自动更新成本越高；CI 整理越复杂，发布等待和失败重试成本越高；asarUnpack 会影响磁盘布局和运行时文件访问；native rebuild 与预打包资源校验会直接拉长构建时间。裁剪不是只追求体积，而是把包体、启动路径、磁盘访问和构建时间放在同一张表里取舍。

2. 四类资源

应用代码

out/main、out/preload、out/renderer 和少量 runtime 依赖进入 app bundle。它们由 files 白名单控制。

外层资源

public、app.png、bundled-aioncore、hub 放入 Electron resources 目录，供运行时按文件系统路径访问。

解包执行

native modules、tree-sitter WASM、内置 MCP server 脚本需要在 app.asar.unpacked 或外层资源中可执行/可读取。

发布资产

dmg、zip、exe、deb、updater yml、WebCLI tarball、checksum、install script 被整理到 release-assets/。

3. 源码里的边界

files 的策略是显式包含构建输出和必要依赖，再用排除规则砍掉源码、测试、文档、source map、大 vendor、tree-sitter native 二进制、JAR/JNI 等风险文件。关键片段如下：

files:
  - out/main/**/*
  - out/preload/**/*
  - out/renderer/**/*
  - public/**/*
  - package.json
  - node_modules/better-sqlite3/**/*
  - '!packages/web-host/**'
  - '!packages/web-cli/**'
  - '!packages/shared-scripts/**'
  - '!packages/desktop/src/**'
  - '!**/*.map'

真正不能进 asar 的资源走 extraResources：resources/bundled-aioncore 被复制为 bundled-aioncore，resources/hub 被复制为 hub。这解释了为什么“误删 bundled-aioncore”不是普通静态资源缺失，而是后端进程无法启动。

4. 检索练习

判断题：看到一个新目录想放进发布包时，第一反应应该是把它加入 files，因为这样最简单。对还是错？

参考答案与反馈

错。先判断运行时访问方式：如果代码可被 asar 内 require/import 读取，才考虑 files；如果是外部进程、后端 binary、Hub 离线包或需要文件系统路径的资源，应考虑 extraResources 或 asarUnpack。裁剪的目标是让每类资源进入正确位置，而不是统一塞进一个白名单。
