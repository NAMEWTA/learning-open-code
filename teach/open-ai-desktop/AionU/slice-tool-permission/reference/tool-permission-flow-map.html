1. 切片边界与入口点

本切片从 renderer 收到“需要用户确认”的会话消息开始，不追入 aioncore 内部工具执行器。入口有三类：

入口消息进入点语义后续 UI
acp_permissionplatforms/acp/useAcpMessage.tsACP agent 请求用户批准某个 tool call。MessageAcpPermission
permissionplatforms/aionrs/useAionrsMessage.ts 或 pending recovery通用 confirmation 形状，包含 call_id、options、command_type。MessagePermission
tool_group.confirmationDetailsMessageToolGroup普通工具组内联确认，常见于 edit / exec / info / mcp。ConfirmationDetails

acp_tool_call 不是确认入口；它是 tool call 的状态、输入、diff、文本或图片输出展示。

sequenceDiagram
  participant Stream as ACP / aionrs stream
  participant Recovery as pending recovery
  participant Merge as Messages hooks
  participant List as MessageList
  participant Card as Permission card

  participant Backend as backend
  Stream->>Merge: acp_permission / permission

  Card->>Adapter: confirmMessage / confirmation.confirm
  Adapter->>Backend: POST /confirmations/:call_id/confirm

1. ACP / aionrs stream 发出需要确认的消息 2. useAcpMessage 或 useAionrsMessage 把消息转成 TMessage 3. useMergeLiveMessage 在 hooks.ts 中合并进 MessageList 状态 4. MessageList 按 type 分派到 MessageAcpPermission / MessagePermission / MessageToolGroup 5. 用户选择 option；按钮禁用直到有选项且正在响应时防重复 6. UI 调 common adapter：confirmMessage 或 confirmation.confirm 7. common adapter 发 POST /api/conversations/:id/confirmations/:call_id/confirm 8. backend 决定批准、拒绝或继续等待，并通过 WS 发 remove / tool update / turn.completed 9. renderer 移除 pending 卡、展示工具结果或显示错误 / tips

3. 沿途每一层

层级关键文件职责注意点
平台消息流platforms/acp/useAcpMessage.ts、platforms/aionrs/useAionrsMessage.ts消费 stream，决定是否设置 running / streamRunning，并把消息交给 merge。aionrs 会把 confirmation-shaped 的 acp_permission 改标成 permission。
会话页面挂载AcpChat.tsx、AionrsChat.tsx挂载 useMessageLstCache() 与 usePendingConfirmationsRecovery()，再渲染 MessageList。pending recovery 是页面生命周期能力，不是单个卡片能力。
消息状态Messages/hooks.ts按 msg_id、call_id、tool_call_id 建索引，合并流式消息。permission 按 content.call_id 去重；普通 tool_call 缺 call_id 会被丢弃并 warn。
消息分派Messages/MessageList.tsx把消息 type 分派到 text、tips、tool、permission、acp_permission、acp_tool_call 等组件。tool call 会被聚合为 summary；permission 不走 tool summary。
确认卡 UIMessageAcpPermission.tsx、MessagePermission.tsx展示选项、处理本地响应中状态、调用确认接口。前端只传选择结果，不在本地判断 tool 是否真的可执行。
普通工具展示MessageAcpToolCall.tsx、MessageToolGroup.tsx展示工具执行状态、输入、diff、图片、结果；tool group 可内嵌确认细节。不要把 acp_tool_call 当成确认状态源。
adapter contractcommon/adapter/ipcBridge.ts定义 confirmMessage、confirmation.confirm/list/remove、approval.check。confirmMessage 和 confirmation.confirm 指向同一 confirm endpoint，但 payload 形状不同。
preload 安全边界preload/main.ts通过 contextBridge 暴露受控 electronAPI.emit/on、文件路径 helper、反馈和启动状态。确认业务不在 preload 内做授权判断；preload 只是隔离 renderer 与宿主能力。
approval memorycommon/chat/approval/ApprovalStore.ts提供会话级 approval key cache，支持 isApproved、approve、allApproved、approveAll。renderer 未直接 import 这个 store；always allow 通过确认 payload / backend contract 传播。

4. 权限确认 UI 与普通消息渲染

组件输入确认动作不是它负责的事
MessageAcpPermissionIMessageAcpPermission.content.tool_call + ACP options把 option_id 作为 confirm_key 发给 conversation.confirmMessage.invoke()。不展示工具最终结果；缺 tool_call 时返回 null。
MessagePermissionIConfirmation把 { value: selected } 和 always_allow 发给 conversation.confirmation.confirm.invoke()。不决定 selected 是批准还是拒绝，选项语义由 backend 提供。
MessageToolGroup普通工具组 content 数组当 confirmationDetails 存在时发送 ToolConfirmationOutcome，包括 Cancel。不是 ACP 专用确认卡。
MessageAcpToolCallToolCallUpdate没有确认按钮。只展示 pending / in progress、raw input、diff、图片或文本结果。

5. Approval store 与 always allow

BaseApprovalStore 是 common 层的抽象缓存，key 由 action 与可选 identifier 序列化为字符串。它只记 true，clear() 清空整张 Map，allApproved(keys) 要求 keys 非空且全部已批准。

UI 侧能观察到两条入口：通用 MessagePermission 在用户选中 proceed_always 时传 always_allow: true；common adapter 还提供 approval.check，用 action 和可选 command_type 查询是否已批准。源码没有显示 renderer 直接写 BaseApprovalStore，这正是安全边界：UI 发意图，是否缓存由后端 / agent adapter 执行。

6. 异常 / 边界路径

边界源码行为风险 / 结论
缺失 pending confirmationusePendingConfirmationsRecovery() 调 confirmation.list，把缺失项构造成 permission 消息，msg_id 为 confirmation:{id}，并按 call_id 跳过重复项。刷新或错过 live stream 后仍能补回确认卡。
pending 被移除订阅 confirmation.remove，remove 事件携带 id，前端用该值同时尝试匹配 confirmation id 和 call_id。确认完成、拒绝或后端取消后，UI 不应继续显示旧卡。
recovery 请求失败catch 中只打印 [pending-confirmations] failed to recover pending confirmations。没有用户可见 fallback；如果 live stream 也没有重发，用户可能看不到卡。
拒绝 / 取消MessageToolGroup 的 outcome 包含 Cancel；MessagePermission 和 ACP 卡把用户选择原样传回。前端不硬编码“拒绝”语义，拒绝后的状态由 backend 决定并通过 remove / tool status / tips 回传。
普通 tool_call 缺 call_idlogDroppedToolCallWithoutCallId() 会 warn 并丢弃。避免不可合并、不可确认的工具状态污染消息列表。

7. E2E 证据

测试证明什么与本切片的关系
tests/e2e/features/conversations/acp/cron-busy.e2e.ts创建需要权限的 ACP 会话，等待权限卡出现，确认目标会话处于 active turn，再验证 Cron run-now 遇忙返回 409，释放 busy 后可重试成功。证明“权限确认会让会话保持忙态，外部调度不能绕过”。
tests/e2e/features/conversations/acp/send-error-surfacing.e2e.ts拦截发送接口返回 ACP 初始化失败，UI 展示 backend 原始错误文本。证明非确认失败路径不会被普通重试文案吞掉。
tests/e2e/features/conversations/acp/empty-turn-warning.e2e.ts注入 ACP empty turn info tip，断言显示本地化中性提示且后续对话仍可用。证明没有可见回复时走 tips，而不是误造权限卡。

补充单元证据：tests/unit/renderer/pendingConfirmationsRecovery.test.ts 覆盖 pending confirmation 构造、按 call_id 检测和按 id / call_id 移除。
