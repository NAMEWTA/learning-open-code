1. 本课只记一个问题

用户说“WebUI 远程打不开”时，不要先猜端口或前端页面。先问：这是 Electron --webui、桌面设置页恢复，还是独立 aionui-web CLI？正常 backend 可用路径会调用 startWebHost()；WebCLI 找不到 backend 时会降级为 startStaticServer() frontend-only，因此 backend 来源、配置来源、密码提示和失败方式不同。

推荐主来源：WebUI 远程访问参考图谱。它列出源码行号、调用时序、文档漂移和 E2E 证据。

2. 主链路地图

入口一：AionUi --webui [--remote] [--port n] -> desktop/src/index.ts 解析 hasSwitch / getSwitchValue -> initializeProcess() -> startBackendOrExit() -> loadUserWebUIConfig() -> resolveWebUIPort() / resolveRemoteAccess() -> startWebHost({ backend: useExistingBackend }) -> startStaticServer() -> 浏览器访问 localUrl 或 networkUrl 入口二：aionui-web start [--remote] [--port n] -> web-cli/src/index.ts parseArgs() -> resolveStaticDir() / resolveDataDir() / resolveBackendBinary() -> backend 存在：startWebHost({ backend: ownBackend }) -> backend 缺失：startStaticServer(frontend-only) -> ensureAdminPassword() -> 浏览器访问 localUrl 或 networkUrl

第一条路径发生在 Electron 主进程里。`index.ts` 先启动 aioncore，再把 `globalThis.__backendPort` 交给 WebHost，因此 WebUI 不会再启动第二个 backend，避免抢同一份 SQLite。第二条路径是独立 CLI，自行找 backend 二进制和 `static/`，启动后再检查管理员密码。

3. 一条关键边界路径

远程访问不是默认打开的。桌面路径里，`resolveRemoteAccess()` 只有在 `--remote`、`AIONUI_ALLOW_REMOTE`、`AIONUI_REMOTE`、`AIONUI_HOST=0.0.0.0` 或用户配置 `allowRemote=true` 时返回 true。CLI 路径里，`resolveAllowRemote()` 只认 `--remote` 或远程 env。最后 `static-server.ts` 用这个布尔值选择监听地址：本地模式绑定 `127.0.0.1`，远程模式绑定 `0.0.0.0` 并生成 `networkUrl`。

文档漂移要记住：`docs/guides/webui.md` 开头和排障段仍写 `3000`，但实现默认值不是它。当前实现是生产 `25808`，开发 `25809`，多实例开发 `25810`；E2E 也以 `25808` 或 WebUI 面板存在作为证据。

4. 检索练习

先回忆再看答案：如果用户运行 `AionUi --webui --remote --port 8080`，请从记忆写出 5 个连续层次。

参考答案：命令行参数进入 `desktop/src/index.ts`，`getSwitchValue('port')` 让 `resolveWebUIPort()` 得到 `8080`，`--remote` 让 `resolveRemoteAccess()` 得到 true，`startWebHost()` 复用已启动 backend，`startStaticServer()` 绑定 `0.0.0.0` 并返回局域网 URL。

反馈：如果你的答案里出现“启动第二个 backend”或“默认端口 3000”，说明混入了旧文档或独立 CLI 的语义。回到参考页的“配置优先级”和“文档漂移”两节复查。

追问建议：让 agent 带你继续检查 `webuiConfig.ts` 的桌面设置恢复路径，或把 `static-server.ts` 的 TCP splice 分支画成更细的请求路由图。
