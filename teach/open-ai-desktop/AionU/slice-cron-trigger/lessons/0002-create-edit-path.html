Cron 创建与编辑分流

学习目标：只读懂 CreateTaskDialog.handleSubmit() 如何把同一张表单分成 addJob 与 updateJob 两条写入路径。

短课 0002 入口：CreateTaskDialog 判断：新建不是编辑
返回 0001 flow map 参考图谱 Cron 模块

1. 先定位分流点

创建和编辑共用一个 modal，所以调试时不要只看按钮文案。真正分叉的是 isEditMode：编辑会保留已有 job_id，新建会构造 ICreateCronJobParams 并把 conversation_id 填成当前会话或空字符串。

2. 真实分流摘录

if (isEditMode) {
  const metadata: ICronJobUpdateParams['metadata'] = {
    conversation_title: editJob!.metadata.conversation_title,
  };
  if (canEditAgentConfig) {
    metadata.agent_config = agent_config;
  }

  // Edit mode: update existing job
  const updates: ICronJobUpdateParams = {
    name: values.name,
    schedule,
    target: {
      payload: { kind: 'message', text: values.prompt },
      execution_mode: resolvedExecutionMode,
    },
    metadata,
    state: {
      max_retries: editJob!.state.max_retries,
    },
  };

  await ipcBridge.cron.updateJob.invoke({
    job_id: editJob!.id,
    updates,
  });
  Message.success(t('cron.page.updateSuccess'));
}

新建路径则进入 else，构造 ICreateCronJobParams 后调用 addJob。

} else {
  // Create mode
  const params: ICreateCronJobParams = {
    name: values.name,
    schedule,
    prompt: values.prompt,
    conversation_id: _conversation_id ?? '',
    conversation_title,
    created_by: 'user',
    execution_mode: resolvedExecutionMode,
    agent_config,
  };
  await ipcBridge.cron.addJob.invoke(params);
  Message.success(t('cron.page.createSuccess'));
}

核心判断是：编辑路径的目标是“更新已有 job”，不是删除重建；新建路径才会构造 ICreateCronJobParams 并调用 addJob。

3. Adapter 落点

ipcBridge.cron.updateJob 会把 updates.target.payload.text 映射成后端 message 字段；addJob 则直接 POST /api/cron/jobs。因此“AI 修改任务后 job id 保持不变”的 E2E 应该对照 updateJob 分支，而不是 create 分支。

4. 检索练习

暂停回忆：如果编辑任务后详情页显示成新任务，你会先检查 CreateTaskDialog 的哪个条件？

参考答案：先检查 isEditMode 是否为 true，以及传入的 editJob!.id 是否进入 ipcBridge.cron.updateJob.invoke({ job_id, updates })。如果实际走了 addJob，历史和 job id 都会断裂。
