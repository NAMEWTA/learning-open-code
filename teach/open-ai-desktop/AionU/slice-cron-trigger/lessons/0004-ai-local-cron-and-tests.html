1. AI 创建入口在哪里

/scheduled 的“对话创建”入口会跳到 /guid 并预填 prompt。真正创建、更新、列出 Cron 任务的能力由注入 HTTP helper 和后端完成；renderer 的 localCronCommands 不再解析自然语言命令。

/**
 * Strip think tags from the assistant message for display.
 * Cron job creation/update/listing is handled through the injected HTTP helper.
 */
export async function processLocalCronResponse(
  _conversationId: string,
  rawContent: string
): Promise<LocalCronProcessingResult> {
  if (!rawContent.trim()) {
    return { systemResponses: [] };
  }

  const thinkStripped = stripThinkTags(rawContent);
  return {
    displayContent: thinkStripped !== rawContent ? thinkStripped : undefined,
    systemResponses: [],
  };
}

2. 测试证明了什么

cron-crud.e2e.ts 主要证明 AI 对话路径的用户可见行为：创建后任务出现在 Scheduled Tasks，修改后 job id 保持，删除后原始会话仍可访问。tests/unit/cron/* 证明 hook、timezone 修复、状态标签和工具函数边界。

E2E：更接近用户行为，但依赖可用 assistant 和 backend helper。
Unit：更稳定地证明 hook 和工具函数，不证明真实 HTTP 服务。
源码证明：run-now 目前属于这一类，需要后续补测试才能升级为测试证据。

3. 不要夸大的地方

本切片没有覆盖后端 scheduler 内部如何到点触发，也没有证明 run-now 的 UI 按钮在自动化中被点击。看到任务“执行后刷新”的现象时，可以先读 useCronJobsMap 的 last_run_at_ms 逻辑，再决定是否要补一条 run-now unit 或 E2E。

4. 检索练习

暂停回忆：如果读者问“AI 创建任务是 localCronCommands 解析出来的吗？”，你应该怎么回答？

参考答案：不是。当前 localCronCommands 只清理 <think> 展示文本；创建、更新、列表由注入 HTTP helper 处理，E2E 通过 bridge/API 和页面可见结果验证。
