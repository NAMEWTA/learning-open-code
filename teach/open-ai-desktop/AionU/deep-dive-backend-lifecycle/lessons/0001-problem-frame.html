Backend lifecycle 为什么必须单点管理

学习目标：能用源码判断一条启动路径是否会制造第二个 aioncore，从而引发 SQLite 数据目录竞争。

L4 深度剖析 主入口：backend-launcher 配套参考：backend-lifecycle-notes
1. 问题框架 2. 单 owner 机制 3. SQLite 边界 4. 检索练习

1. 问题框架

AionU 的 backend 不是一个随手可起的无状态服务。它持有端口、子进程、agent 子进程清理、健康检查状态，还会接触同一个本地数据目录。真正危险的场景不是“WebUI 能不能打开”，而是 Desktop main、headless WebUI、设置页 WebUI 或恢复路径各自都以为自己可以启动 backend。只要两个 aioncore 指向同一个 --data-dir，SQLite 就可能出现锁等待、迁移交错或用户看到两套状态。

因此本课的判断标准很窄：一条路径要么复用已有 backend port，要么由同一个 BackendLifecycleManager 启动并停止；不要在旁路里再 spawn 一个 backend。完整 WebUI 链路见 L2 WebUI 图谱，本课只深挖这个约束为什么存在。

方案取舍
每个入口 ownBackend启动逻辑简单，但最容易让相同 dataDir 出现双 backend。
单 owner + useExistingBackend入口要传递 port，但 owner、stop、diagnostics 和 recovery 边界集中。
只依赖 SQLite WAL / busy_timeout能缓冲瞬时锁等待，不能治理两个长期 backend 写同一库。
只依赖 Electron 单实例锁能挡普通桌面双开，但挡不住 WebUI、测试和显式多实例路径。

2. 单 owner 机制

open-ai-desktop/AionU/packages/web-host/src/backend-launcher.ts 把 owner 需要记住的状态集中在一个类里：子进程、端口、状态、上次数据目录、上次目录配置和重启窗口。它不是纯函数，因为 backend 生命周期本身就是长期状态。

export class BackendLifecycleManager {
  private childProcess: ChildProcess | null = null;
  private _port = 0;
  private _status: BackendStatus = 'stopped';
  private _lastDbPath = '';
  private _lastLogDir?: string;
  private _lastDirs?: BackendDirConfig;
  private _lastOptions?: BackendStartOptions;
  private restartCount = 0;
  private restartWindowStart = 0;
  private readonly maxRestarts = 3;
  private readonly restartWindowMs = 60_000;

  constructor(
    private readonly appMeta: AppMetadata,
    private readonly resolveBackend: BackendBinaryResolver
  ) {}

启动时，它把 dbPath 写入 --data-dir，等待 aioncore 输出 AIONCORE_LISTENING，再轮询 /health。失败时，错误详情会带上 dataDir、stderr tail 和 backend boundary stage。停止时，它先 SIGTERM，再超时 SIGKILL，并用最后一次 dataDir 清理登记过的 agent 进程。这个 owner 是“知道怎么起，也知道怎么停”的地方。

3. SQLite 边界

Desktop main 的启动顺序先把短期开库窗口收掉，再让 backend 长期持有数据目录。注释直接说明：initializeProcess() 里的 initStorage 可能为 v26 一次性迁移打开 legacy Electron SQLite catalog，必须先关闭，backend 才能碰同一个文件。

// Start aioncore only after initializeProcess(). initStorage may open
// the legacy Electron SQLite catalog for a one-shot v26 migration and must
// close it before the backend touches the same file.
const backendStartup = await startBackendOrExit({
  startBackend: async () => {
    assertStartupArchitectureCompatible({
      arch: process.arch,
      isPackaged: app.isPackaged,
      platform: process.platform,
    });
    const { getDataPath } = await import('./process/utils/utils');
    const { getSystemDir } = await import('./process/utils/initStorage');
    const sysDir = getSystemDir();
    return backendManager.start(
      getDataPath(),
      sysDir.logDir,
      {
        cacheDir: sysDir.cacheDir,
        workDir: sysDir.workDir,
        logDir: sysDir.logDir,
      },

设置页启动 WebUI 时也不重新起 backend。open-ai-desktop/AionU/packages/desktop/src/process/utils/webuiConfig.ts 先读 globalThis.__backendPort，没有端口就抛错；有端口才调用 startWebHost({ backend: useExistingBackend })。

// Reuse the backend already spawned by backendManager.start() in src/index.ts.
// Spawning a second backend here would race the first on the same SQLite file.
const backendPort = (globalThis as typeof globalThis & { __backendPort?: number }).__backendPort;
if (!backendPort) {
  throw new Error('[WebUI] Cannot start: aioncore is not running (globalThis.__backendPort unset)');
}

数据库服务本身还有 busy_timeout 和 WAL，但那是短期缓冲，不是允许双 backend 的理由。详细证据见 backend lifecycle 速查。

密码重置也共享这个约束：isResetPasswordMode 下启动策略不允许 pending health，失败会退出；所以 resetpass 不是独立第二个 backend，而是同一个启动管线上的特殊模式。

4. 检索练习

判断题：用户在设置页打开 WebUI，如果没有 globalThis.__backendPort，正确行为是临时启动一个新 backend，让 WebUI 至少可用。

答案：错。源码选择直接抛错，因为第二个 backend 会与桌面 backend 争同一个 SQLite 文件；正确排查方向是回到 backendManager.start() 为什么没成功暴露端口。

继续追问：如果你想练习排查，请让 agent 带你从一次 database is locked 日志反推是 legacy migration、WebUI 复用失败，还是 backend 自身启动边界。
