1. 设计不变量

可选方案好处为什么不是最终主策略
每个入口都 ownBackend局部代码简单。Desktop、WebUI、resetpass 和 recovery 会争同一 dataDir。
单 owner + useExistingBackend生命周期、诊断、停止和恢复集中。入口必须正确传递/读取现有 port，失败时要早报错。
SQLite WAL / busy_timeout缓解短暂锁等待。不能替代进程级 owner，不应掩盖双 backend。
Electron 单实例锁挡住普通桌面双开。测试、多实例 env、WebUI 和 CLI 路径仍要自己守边界。
不变量判断方式破坏后的典型现象
同一个 dataDir 只能有一个长期 backend owner查是否复用 globalThis.__backendPort 或同一个 BackendLifecycleManagerdatabase is locked、迁移失败、WebUI 与桌面状态不一致
backend 启动晚于 main process storage 初始化查 initializeProcess() 是否已经完成legacy SQLite 迁移与 backend 数据初始化争同一文件
WebUI inside Electron 复用现有 backend查 backend.kind === 'useExistingBackend'设置页或 headless WebUI 额外 spawn backend
失败需要回传边界信息查 BackendStartupError.details 是否含 stage、dataDir、stderrTailUI 只能显示泛化启动失败，无法区分迁移、损坏或安装问题
恢复标志只用于用户确认后的重启查 crash restart 是否复用 --recover-corrupted-database未经用户确认的后续重启继续破坏性恢复

3. SQLite 竞争规避

层次源码证据作用边界
Electron 单实例open-ai-desktop/AionU/packages/desktop/src/index.ts:77-87第二个桌面实例在早期退出，减少重复启动 backend 的机会。E2E 或 AIONUI_MULTI_INSTANCE=1 可跳过，不能只依赖它。
先 storage 后 backendopen-ai-desktop/AionU/packages/desktop/src/index.ts:658-661legacy SQLite 一次性迁移先完成并关闭连接，backend 再接管数据目录。只约束 Desktop main 启动顺序，不说明 aioncore 内部锁策略。
legacy migration 短期开连接open-ai-desktop/AionU/packages/desktop/src/process/services/database/runLegacyDatabaseMigrations.ts:45-91打开 aionui.db、跑 schema/migration/repair，finally 中关闭 driver。这是迁移交接，不是长期业务读写 owner。
driver 是真实 SQLite 连接open-ai-desktop/AionU/packages/desktop/src/process/services/database/drivers/BetterSqlite3Driver.ts:23-42new BetterSqlite3(dbPath) 说明该迁移路径确实打开本地 DB 文件。driver 本身不协调多个 backend 进程。
busy_timeout 与 WALopen-ai-desktop/AionU/packages/desktop/src/process/services/database/schema.ts:15-28瞬时锁等待最多 5 秒，并启用 WAL 改善并发读写表现。它是缓冲，不是允许两个长期 backend 写同一 DB。
WebUI 复用 backendopen-ai-desktop/AionU/packages/desktop/src/process/utils/webuiConfig.ts:228-262设置页 WebUI 必须拿现有 backend port，并让 dataDir 与桌面 IPC 路径一致。拿不到 port 就抛错，不能自行启动第二个 backend。
Headless WebUI 复用 backendopen-ai-desktop/AionU/packages/desktop/src/index.ts:767-804Electron --webui 分支也复用前面已经启动的 backend。与独立 WebCLI 的 ownBackend 场景不同。
Reset password 特殊模式open-ai-desktop/AionU/packages/desktop/src/index.ts:679-708、index.ts:747-758isResetPasswordMode 不允许 pending health，失败直接退出。它共享启动 owner，不是旁路启动第二个 backend。

4. 源码与测试证据

证据证明了什么读法
open-ai-desktop/AionU/packages/web-host/src/backend-launcher.test.ts:275-314aioncore 自己报告监听端口，launcher 不预绑定 backend 端口；启动参数包含同一 --data-dir。先看 mock 的 AIONCORE_LISTENING，再看 spawn args 断言。
open-ai-desktop/AionU/packages/web-host/src/backend-launcher.test.ts:393-457早退 stderr 中的 database.open、database.migration 能进入 BackendStartupError.details。这是 UI 能分类数据库边界失败的前提。
open-ai-desktop/AionU/packages/web-host/src/backend-launcher.test.ts:532-552health timeout 诊断保留 stderrTail，包括 database is locked。用于排查真实用户日志时判断是 backend 启动边界，而非 renderer 问题。
open-ai-desktop/AionU/packages/web-host/src/backend-launcher.test.ts:1069-1094用户确认的 corrupted database recovery 标志不会在 crash restart 中复用。说明恢复是受控动作，不是普通重启策略。
open-ai-desktop/AionU/packages/desktop/src/process/startup/backendStartup.tsstartBackendOrExit() 只包装策略：成功回调端口，失败采集并按模式退出。不要把它看成 lifecycle owner；owner 仍是 manager。

排查口诀：先问“谁是 owner”，再问“dataDir 是否一致”，最后才看 UI。看到 database is locked 时，优先检查是否有第二个 backend、legacy migration 是否未关闭、WebUI 是否误走 ownBackend。

5. 缺口

  • 本页未分析 aioncore Rust 侧实际 SQLite 文件名、migration 顺序和恢复标志消费。
  • 当前 TypeScript 测试覆盖诊断和 owner 行为，但没有启动真实 aioncore 做跨进程 SQLite 锁复现。
  • 如果后续新增独立 CLI 与桌面共用 dataDir 的入口，需要补一条“是否必须检测已有 backend port”的规则。
