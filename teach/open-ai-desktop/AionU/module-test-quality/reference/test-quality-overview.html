3. 分层结构

open-ai-desktop/AionU/tests/ ├── unit/ ← Vitest 主力：函数、hooks、components、adapter ├── integration/ ← Vitest 集成：bootstrap、migration 等链路 ├── e2e/ ← Playwright Electron 用户链路 ├── fixtures/ ← fake ACP CLI、fake extension 等测试替身 └── vitest.*.setup.ts ← node / jsdom 全局测试环境
层职责典型文件何时使用
Unit / node验证纯逻辑、process service、common adapter、DTO mapper、配置迁移。open-ai-desktop/AionU/tests/unit/common-adapter/httpBridge.test.ts依赖可 mock，行为可在 Node 中观察。
Unit / jsdom验证 React 组件、renderer hooks、浏览器 API、Arco DOM 状态。open-ai-desktop/AionU/tests/unit/cron/useCronJobs.dom.test.ts需要 DOM，但不需要真实 Electron 窗口。
Integration验证跨 service / migration / bootstrap 的数据流。open-ai-desktop/AionU/tests/integration/bootstrap/windowsPath.test.ts多个模块协作，但仍可在 Vitest 中完成。
E2E fixtures启动 Electron、复用主窗口、提供 backend bridge、native dialog mock、截图。open-ai-desktop/AionU/tests/e2e/fixtures.ts真实用户链路前置设施。
E2E specs / features验证用户可见流程和 backend 最终状态。open-ai-desktop/AionU/tests/e2e/specs/cron-crud.e2e.tsUI、Electron、backend、路由、侧栏刷新必须一起证明。
CI / packaging验证跨 OS、format/lint/typecheck、build artifact、install smoke。open-ai-desktop/AionU/.github/workflows/pr-checks.yml本地测试不足以覆盖平台或打包差异时。

4. 关键依赖

依赖用途读码提示
Vitest 4单元、DOM、集成、coverage、benchmark。配置使用 projects 区分 node 和 jsdom。
@testing-library/react / @testing-library/jest-domReact hook / component 断言。DOM setup 注入 jest-dom matcher 和浏览器 API mock。
jsdomrenderer DOM 测试环境。文件名带 .dom.test.* 才进入 dom project。
PlaywrightElectron E2E runner、trace、video、HTML report。项目覆盖 tests/e2e/**/*.e2e.ts，worker 固定为 1。
ElectronE2E app 启动目标。fixture 支持 dev 与 packaged 模式；CI 默认 packaged，除非 workflow 设置 E2E_DEV=1。
aioncore backendrenderer HTTP 真实后端。E2E README 要求 PATH 上可找到 aioncore，否则 window.__backendPort 可能为 0。
Bun包管理、脚本运行、Vitest 命令入口。CI 固定使用 setup-bun 和 frozen lockfile。
just本地命令编排。just e2e-test 比裸 bun run test:e2e 多一步构建。
oxlint / oxfmt / tsc / prek代码质量、格式、类型、提交前文件检查。属于质量链，不直接替代测试。
Codecov上传和展示 coverage。当前 status informational，适合看趋势。

cd open-ai-desktop/AionU
bunx vitest run tests/unit/cron/useCronJobs.dom.test.ts

适合修改 Cron hook、localStorage、事件订阅、renderer 状态同步。

示例 B：跑指定 Team E2E

cd open-ai-desktop/AionU
bunx playwright test --config playwright.config.ts \
  tests/e2e/cases/teams/team-create.e2e.ts --reporter=list

适合验证侧边栏、创建 modal、助手选择、导航和 UI 刷新链路。

cd open-ai-desktop/AionU
just e2e-test

会先 bun run package 再跑 Playwright，避免 E2E 读取旧构建产物。

8. 风险与空白

项观察行动建议
覆盖率目标与硬闸门不一致open-ai-desktop/AionU/AGENTS.md 和 testing skill 写目标为 80%；open-ai-desktop/AionU/vitest.config.ts thresholds 为 0；open-ai-desktop/AionU/codecov.yml informational；coverage job warning-only。把 coverage 当趋势和审查信号；若要硬化，需单独改配置和 CI。
E2E 依赖构建产物open-ai-desktop/AionU/tests/e2e/README.md 说明 E2E 读取 out/；裸跑 Playwright 可能测到旧 UI。本地优先用 just e2e-test 或先 bun run package。
E2E 环境前置需要 Electron 可启动，PATH 上有 aioncore；否则 renderer backend port 可能不可用。失败时先查 README 的 build 与 PATH 排障，而不是直接怀疑业务代码。
共享 singleton appPlaywright 配置 workers: 1，fixture 注释明确不在每个 describe 后关闭 app。E2E 要主动清理测试数据，避免跨 spec 污染。
Skills Hub evidence 演进当前源码中 tests/e2e/features/settings/skills/ 有 boards-rendering、core-ui、path-export、url-highlight 等新文件。后续更新 module-assistants-skills 时应同步 reference 的测试证据表。
未读取真实 CI run本 L1 根据 workflow 配置讲解，不代表最近一次 GitHub Actions 通过。发布或审查时还需要查看实际 run log。
