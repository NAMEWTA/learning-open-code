测试质量：先选证据层，再跑命令

15 分钟短课：只学一个判断技能，看到一个 AionU 改动后，如何判断它应该由 Vitest、Playwright E2E 还是 CI/构建烟测来证明。

L1：module-test-quality 主线：证据分层 目标：选对验证入口

L0 回链：先看 AionU 项目地图； 完整清单看 本模块 reference。 交叉证据：Cron 测试、 Team Mode 测试、 Assistants / Skills 测试。

1. 模块职责：它不是功能模块，而是证据系统

在 L0 架构图里，它横跨配置、测试资产、CI 和构建烟测，是工程质量层，不属于某一条运行时业务功能链路。 AionU 的测试质量模块由配置、测试资产、helper 和 CI workflow 共同组成。 open-ai-desktop/AionU/vitest.config.ts 决定哪些文件进入 node 或 jsdom 环境； open-ai-desktop/AionU/playwright.config.ts 决定 Electron E2E 如何串行运行、保存 report 和结果； open-ai-desktop/AionU/tests/ 承载真正的断言。 读这个模块时，不要先问“覆盖率多少”，先问“这个行为应该在哪一层被证明”。

open-ai-desktop/AionU/ ├── vitest.config.ts ← Vitest node、jsdom、coverage ├── playwright.config.ts ← Playwright Electron E2E ├── tests/ ← unit、integration、e2e、fixtures ├── .github/ │ └── workflows/ ← PR checks、E2E artifacts、build smoke └── .claude/ └── skills/ ← testing 质量规则

2. 三层证据怎么选

第一层是 Vitest node：纯函数、adapter mapper、process service、配置迁移、文件路径、错误归一都优先放这里。 第二层是 Vitest jsdom：React 组件、hook、localStorage、ResizeObserver、滚动、Arco DOM 状态使用 *.dom.test.tsx 或 *.dom.test.ts。 第三层是 Playwright Electron E2E：只有用户路径、真实窗口、backend HTTP、native dialog、sidebar 刷新、团队页或会话页联动才值得上 E2E。

命令也按这个顺序收窄：先跑单个文件，例如 bunx vitest run tests/unit/cron/useCronJobs.dom.test.ts； 改用户链路时再跑指定 E2E，例如 bunx playwright test --config playwright.config.ts tests/e2e/specs/cron-crud.e2e.ts。 全量 bun run test 和 bun run test:e2e 是提交前或高风险改动的收口，不是每次读码的起点。

3. 读测试先看调用面

单元测试通常 mock 边界，比如 Cron hook mock ipcBridge.cron 和 renderer emitter，证明 loading、错误、启停、时区修复是否正确。 E2E 则尽量走真实 UI：Team helper 明确通过侧边栏创建团队，因为直接 POST 会绕过 TeamCreateModal.onCreated → refreshTeams()。 对 Skills Hub，E2E 会 mock 原生文件选择，然后通过 bridge 再读后端状态，证明导入不仅显示成功，也真的写入了技能库。

这就是本模块的核心判断：Vitest 证明局部行为，Playwright 证明用户链路，CI 证明跨平台和打包形态。当前覆盖率更多是趋势信号：项目规范写了 80% 目标，但配置里的 threshold 仍为 0，Codecov 也是 informational，不能把它误读成硬阻断。

4. 检索练习

暂停回忆：如果你修改了 Team 创建 modal 中的助手选项过滤逻辑，最小验证应该从哪两层开始？

参考答案与反馈

先跑 jsdom / unit 层，验证选项过滤、禁用态或选择工具函数；再跑 Team 创建相关 E2E，验证真实侧边栏按钮、modal、选择助手、创建后导航和 sidebar 刷新。只跑 bridge API 不够，因为它绕过了 TeamCreateModal 的 UI 刷新链路。

推荐主来源：继续阅读 测试选择速查， 然后打开 open-ai-desktop/AionU/tests/e2e/README.md 对照 E2E 的启动约束。下一步可以向 agent 追问： “请按某个具体改动帮我选择最小测试集”。
