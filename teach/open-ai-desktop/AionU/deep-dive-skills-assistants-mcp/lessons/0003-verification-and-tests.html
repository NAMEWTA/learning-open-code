Skills / Assistants / MCP：验证与测试证据

学习目标：能区分 smoke 测试、UI 行为测试和数据库快照测试分别证明了什么。

L4：验证与局限 预计 13 分钟 主来源：ext-skills/ext-mcp
1. 指定 E2E 的证据强度 2. 更强的补充测试 3. 性能、可靠性与局限 4. 改进练习

本课主源码文件不超过 3 个：tests/e2e/specs/ext-skills.e2e.ts、tests/e2e/specs/ext-mcp.e2e.ts、tests/e2e/specs/assistant-settings-conversation-defaults.e2e.ts。完整测试矩阵见 参考笔记测试表。

1. 指定 E2E 的证据强度

ext-skills.e2e.ts 和 ext-mcp.e2e.ts 是扩展能力的 smoke 测试。前者进入 agent settings，断言页面文本长度大于 50；后者进入 capabilities/tools，断言页面包含 MCP 相关词、页面文本长度大于 50，并且至少有一个 Arco switch。它们的价值是发现“扩展加载后页面崩溃、路由不可用、基础控件消失”。

它们的局限也明显：没有断言具体 extension skill 名称，没有验证 Assistant 是否绑定该 Skill，没有验证 MCP 连接测试返回 tools，也没有追到会话创建 snapshot。因此这两份测试不能单独证明 Skills / Assistants / MCP 映射完全正确。

2. 更强的补充测试

assistant-settings-conversation-defaults.e2e.ts 是更强证据。它先通过 HTTP 取 API /api/skills 和 API /api/mcp/servers，创建自定义 Assistant，固定选择 Skill/MCP，再从 SQLite 的 conversation_assistant_snapshots 读取 resolved_skill_ids 与 resolved_mcp_ids。另一个用例把默认值切到 auto，在 Guid 页选择 Skill/MCP，验证 snapshot 和 assistant_preferences 都记录了本次选择。

这类测试能覆盖“设置页选择 -> 创建请求 -> 后端快照/偏好”的闭环，比只看 UI 文案强。相邻的 assistant-settings-skills.e2e.ts 则覆盖默认 Skills/MCP 单一控件、hub 跳转、内置 Assistant 只读摘要和固定 Skill 选择持久化，可作为 UI 层补充。

3. 性能、可靠性与局限

性能上，Assistant 编辑器用 Promise.all 并行加载详情、Skills 和 MCP catalog；MCP 批量导入后测试连接时有 concurrency 默认 4，避免一次性打满。MCP catalog 用名称归一化去重，builtin 与 user 采用不同 key，减少重复展示。

可靠性上，MCP 测试失败只写 UI 状态并给错误提示，不会把它误当作会话实时状态；OAuth 只对 http/sse/streamable_http transport 生效。已知局限包括：Assistants PRD README 为空；extension-contributed MCP 在 Tools 页只读展示；指定 ext smoke 测试断言较弱；Assistant 默认值变更不会更新旧会话，这是有意的快照语义但容易误解。

4. 改进练习

小任务：如果只能补一个测试来提高 confidence，你会补 UI 层还是 snapshot 层？覆盖哪条断言？

参考反馈

优先补 snapshot 或 bridge 层。一个高价值用例是加载 examples extension 后，通过 extensions.get-skills 和 extensions.get-mcp-servers 取到具体名称，再创建 Assistant 并发起 Guid 会话，断言 create payload 的 conversation_overrides.skill_ids、mcp_ids 以及后端 snapshot 都包含目标项。这样比“页面长度大于 50”更能证明映射链路。
