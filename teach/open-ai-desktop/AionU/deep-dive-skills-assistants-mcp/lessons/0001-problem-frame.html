1. 为什么容易混淆

AionU 里 Assistant、Skill、MCP 都会出现在“能力”语境里：Assistant 可以有默认 Skills 和默认 MCP；extension 可以同时贡献 assistant、skill、mcpServers；会话创建后，发送框又会显示 loaded skills 和 loaded MCP。于是维护时很容易把“谁定义能力”“谁选择能力”“谁真正运行能力”混成一件事。

混淆的根源是时间点不同。Assistant 设置页保存的是角色定义和默认选择策略；Skills Hub 保存的是可选技能目录；Tools/MCP 设置页保存的是工具服务器目录和最近一次手动测试结果；Guid 创建会话时才把这些选择解析成一次会话快照。一个运行中的会话不会因为你后来改了 Assistant 默认值就自动改写历史快照。

2. 三个边界

概念它负责什么它不负责什么
Assistant角色身份、主 agent、规则、推荐 prompts、默认模型/权限/Skills/MCP。不拥有 Skill 文件内容，也不直接启动 MCP server。
Skill可注入的提示或操作说明，来源可为 builtin、custom、cron、extension。不等于外部工具进程，也不代表 Assistant 本身。
MCP工具服务器配置、transport、enabled 状态、工具列表和测试状态。不定义 persona，也不替代 Skill 的提示语义。

packages/desktop/src/common/types/agent/assistantTypes.ts 把边界写得很清楚：AssistantDetail 里同时有 defaults、capabilities 和 preferences。其中 defaults.skills 与 defaults.mcps 是“默认选择模式和值”，不是 Skill 或 MCP 的定义本体。

3. AionU 的设计取舍

AionU 选择把能力拆成目录、默认值、会话快照三层。目录层便于全局管理和扩展贡献；默认值层让 Assistant 像一个可复用角色；快照层让历史会话可复现。这个取舍牺牲了一点即时性：你必须区分“新会话默认值”和“当前会话已经加载的能力”。

扩展示例进一步说明了边界。examples/hello-world-extension/aion-extension.json 的 contributes 同时声明 assistants、agents、skills、mcpServers。这不是让四者合并成一个对象，而是让扩展一次性向四个注册表贡献条目。

判断法：如果问题是“角色默认带哪些能力”，查 Assistant；如果问题是“这个技能是否存在、能否导入”，查 Skills Hub 和 API /api/skills；如果问题是“工具服务器能否连接、有哪些 tools”，查 Tools/MCP 和 API /api/mcp；如果问题是“这次会话实际加载了什么”，查 conversation extra 或 assistant snapshot。

4. 检索练习

暂停回忆：用户说“我刚在 Assistant 里选了一个 MCP，但老会话里看不到”，最先应该解释哪条边界？

参考反馈

应先解释“Assistant 默认值只影响新会话”。AionU 在创建会话时把默认 MCP 解析成会话快照；旧会话已经持有自己的 loaded MCP 信息，不会被后续 Assistant 设置自动改写。排查时再看 Guid 创建请求和 conversation extra，而不是只盯 Tools/MCP 页。
