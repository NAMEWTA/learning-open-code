1. Assistant 编辑器状态

useAssistantEditor 是 Assistant 设置页的状态中枢。编辑已有 Assistant 时，它并行加载三类资源：ipcBridge.assistants.get 取角色详情，ipcBridge.fs.listAvailableSkills 取 Skill 目录，ensureBackendMcpCatalog() 取 MCP 目录。随后把 detail.defaults.skills.mode、detail.defaults.mcps.mode、detail.capabilities.default_skill_ids、detail.defaults.mcps.value 填进本地 state。

这里的设计取舍是把“选择器显示状态”和“后端持久化 shape”分开。UI 可以显示“Remember last used automatically”，但保存时仍写成 { mode: 'auto' }；固定选择才写 { mode: 'fixed', value: [...] }。内置 Assistant 更收敛：源码只允许更新主 agent、默认模型、权限、思考级别，不允许直接改默认 Skills/MCP。

2. Skill 导入与绑定

Skill 绑定有两步：先保证 Skill 存在于全局目录，再把名字放进 Assistant 默认能力。pendingSkills 是“用户已选但还没真正导入”的过渡态；保存前会逐个调用 ipcBridge.fs.importSkills.invoke，刷新 Skill 列表后再组装 defaults.skills。

if (pendingSkills.length > 0) {
  const skillsToImport = pendingSkills.filter(
    (pending) => !availableSkills.some((available) => available.name === pending.name)
  );

  for (const pendingSkill of skillsToImport) {
    await ipcBridge.fs.importSkills.invoke({ skill_path: pendingSkill.path });
  }
}

const defaults = {
  skills: { mode: defaultSkillsMode, value: selectedSkills },
  mcps: { mode: defaultMcpMode, value: selectedMcpIds },
};

自动注入 builtin skill 还有一层反向列表。AssistantEditorSections 会把自动注入 skill 合入显示值；用户取消某个自动项时，不是把它从 builtin 目录删除，而是写入 disabled_builtin_skills。这能保留全局 builtin 行为，同时允许单个 Assistant opt out。

3. MCP 到 adapter/backend

Tools/MCP 设置页由 McpManagement 组合多个 hook：useMcpServers 读取后端用户 MCP、本地 builtin MCP 和 extension MCP；useMcpServerCRUD 调用 create/import/update/delete/toggle；useMcpConnection 调用连接测试并把返回的 tools 写进本地 UI 状态；useMcpOAuth 只对 URL transport 检查和触发登录。

ipcBridge.ts 把 API 面切得很直：Assistant 是 API /api/assistants，Skill 是 API /api/skills，MCP 是 API /api/mcp，extension 贡献则从 API /api/extensions 读取。这个分层让设置页职责清晰：Assistant 保存默认能力，Tools/MCP 保存工具服务器目录，Guid 创建会话时再把 MCP id 和 session MCP server 快照发给 API /api/conversations。

Assistant 编辑器 -> /api/assistants + /api/skills
Tools/MCP 设置 -> /api/mcp/servers + /api/mcp/test-connection
Extension examples -> /api/extensions/skills + /api/extensions/mcp-servers
Guid 创建会话 -> /api/conversations，携带 assistant overrides 和 MCP snapshot

4. 判断练习

判断题：“MCP 连接测试成功”是否等于“当前会话一定正在使用这个 MCP”？

参考反馈

不等于。McpServerHeader 的提示也说明手动检查只验证配置可用性，不代表当前会话实时状态。当前会话是否使用某个 MCP，要看创建会话时的 selected MCP ids、session MCP servers，以及 conversation extra 中的 loaded MCP snapshot。
