1. 模块职责与位置

职责落点说明
统一 renderer 调用面common/index.ts + adapter/ipcBridge.ts把 renderer 可见能力统一暴露为 ipcBridge.xxx.invoke/on。
HTTP/WS 适配adapter/httpBridge.ts把 REST 与 WebSocket 包装成与 bridge.buildProvider/buildEmitter 同形的对象。
main 侧统一中继adapter/main.ts接收 preload 送来的统一 adapter 事件、广播到所有窗口，并可转发到 WebSocket 客户端。
映射修正*Mapper.ts修正前后端字段名、默认语义、枚举和值域差异。
共享 API clientcommon/api/*提供旋转 API client 与协议转换辅助，供公共逻辑复用。

5. 关键依赖

依赖类型为什么关键
@office-ai/platform bridge桥接抽象旧 renderer 调用面依赖它的 provider / emitter 形状，HTTP/WS 适配层也在模拟这套形状。
electron运行时框架ipcMain、BrowserWindow、原生对话框、通知、窗口控制都依赖它。
浏览器 fetch / WebSocketWeb 运行时本地 backend API 与实时事件的核心传输层。
common/api/ClientFactory.ts共享 API client说明 common 层不只管 renderer-main 桥，也承载多 provider 协议统一。
tests/e2e/helpers/httpBridge.ts测试适配从测试侧验证 renderer 已迁到 HTTP 调用，而不是宿主侧伪造请求。

6. 调用 / 阅读示例

推荐阅读顺序 A：理解统一桥

1. 先读 preload/main.ts，确认 renderer 进入点只有 electronAPI.emit/on。

2. 再读 common/adapter/main.ts，确认统一 channel office-ai-bridge-adapter 怎样把事件送进 main。

3. 然后读 common/adapter/ipcBridge.ts，观察每个域是 bridge.buildProvider 还是 httpPost/wsEmitter。

推荐阅读顺序 B：挑一条能力链

1. IPC 样例：windowControls.minimize → process/bridge/windowControlsBridge.ts → BrowserWindow.minimize()。

2. HTTP/WS 样例：conversation.sendMessage → POST /api/conversations/:id/messages → conversation.responseStream / turn.completed。

3. 混合样例：webui.start 仍走 IPC，而 webui.changePassword 已走 backend HTTP。

renderer 调用
  -> common/index.ts 暴露的 ipcBridge
  -> ipcBridge.ts 判断域与传输
     -> IPC: preload/main.ts -> adapter/main.ts -> process/bridge/*
     -> HTTP: httpBridge.ts -> /api/*
     -> WS: wsEmitter/wsMappedEmitter -> /ws
  -> mapper 把 wire contract 修成前端可用类型
