Electron 主入口：先看它如何决定应用活成哪一种形态

学习目标：15 分钟内看懂 AionU 的 Electron 主入口如何把同一份桌面壳分流为版本输出、桌面窗口、WebUI 和重置密码四种运行形态。

L1 模块总览 预计 15 分钟 源码 3 个

1. 模块职责与架构位置

packages/desktop/src/index.ts 不是“什么都做一点”的杂糅入口，它更像一个总调度器：先完成 Chromium、Sentry、单实例锁、PATH 修复这些启动前置条件，再等待 app.whenReady() 进入真正的生命周期；真正的业务细节则下沉到 process/startup 和 process/utils。因此读这个模块时，重点不是记住所有 import，而是抓住三类决策：什么时候允许 app 继续初始化，什么时候必须先把 aioncore backend 拉起来，什么时候不创建窗口而改走无头模式。

在整体架构中的位置：它站在 Electron main process 的最外层，上游接收命令行参数、协议唤醒和系统事件，下游调用 backend manager、BrowserWindow、托盘、WebHost 与退出清理器。若你还没复习全局地图，先回到 L0 项目总览课 或 L0 架构参考页 再看这里，会更容易把主入口放回整体系统。

2. 启动主线：先处理 --version，再初始化 backend 相关链路

真正的主线从 handleAppReady() 开始，但第一个分流不是 backend，而是 CLI 版本输出。源码先判断 isVersionMode；如果命中 --version 或 -v，就直接打印版本并 app.exit(0) 返回，整个函数不会进入 initializeProcess()。只有不是版本模式时，才会继续初始化主进程基础设施，再启动 backend。

if (isVersionMode) {
  console.log(app.getVersion());
  app.exit(0);
  return;
}

await initializeProcess();

const backendStartup = await startBackendOrExit({
  startBackend: async () => backendManager.start(...),
  onStarted: (backendPort) => exposeBackendPort(backendPort),
  exitOnFailure: isWebUIMode || isResetPasswordMode,
});

后半段的顺序仍然很关键：initializeProcess() 先把存储、bridge、i18n 等 main 基础设施准备好；之后才调用 startBackendOrExit()。注释已经说明，初始化阶段可能暂时打开旧版 SQLite 目录做迁移，如果 backend 提前摸到同一份数据文件，就会和主进程发生竞争。也就是说，主入口先处理“纯 CLI 输出”这种零状态路径，再保证“数据面”可安全接管，最后才让 backend 进入健康检查。

startBackendOrExit() 本身很薄：成功时回传端口并执行 onStarted，失败时记录、捕获并按配置决定是否退出。它把“启动策略”留在入口，把“成功 / 失败 / 取消”的统一语义提炼进 backendStartup.ts，因此这个 helper 既可复用，也容易被单元测试锁死。

3. 模式分流：同一入口，四种落点

这四种落点并不在同一个阶段发生。--version 是最早的短路分支：app.whenReady() 之后立刻退出，不触发任何 process 初始化或 backend 启动。其余三种则共享同一条主线：先 initializeProcess()，再 startBackendOrExit()；然后 --resetpass 走 CLI 重置密码，不创建窗口；--webui 调用 startWebHost()，复用前面已经启动的 backend 端口，避免再起第二个进程碰撞 SQLite；默认桌面模式才会读“关闭到托盘”设置、创建主窗口、异步恢复 WebUI 偏好，并在窗口 ready 后触发延迟迁移。

if (isResetPasswordMode) {
  await resetPasswordCLI(username);
} else if (isWebUIMode) {
  await startWebHost({ backend: { kind: 'useExistingBackend', port } });
} else {
  createWindow({ showOnReady });
}

这里最值得记住的判断是：AionU 把“桌面壳”和“backend 运行时”解耦了。窗口只是默认表现层，不是启动链路的中心；真正先被保证可用的是 backend 和数据目录，然后才决定要不要把 UI 露出来。

4. 退出收尾与检索练习

退出阶段也延续同样思路：index.ts 只安装 installQuitCleanup()，把 before-quit 接到一个统一收尾器；具体实现位于 quitCleanup.ts，其中会设置 quitting 标记、销毁托盘、停止 cron resume 监听、停止 backend，并在 10 秒超时后强制放行退出。这样主入口仍然只表达“何时收尾”，而不在文件末尾堆一长串清理细节。

installQuitCleanup({
  onBeforeQuit: (handler) => app.on('before-quit', (event) => handler(event)),
  stopBackend: () => backendManager.stop(),
  destroyPetWindow: async () => { ... },
});

暂停回忆：请按顺序复述主入口的前四步：app.whenReady() 之后，哪个分支会在 initializeProcess() 之前直接退出？剩下的模式为什么必须等 backend 启动完成后再继续？

参考答案与反馈

正确顺序是：app.whenReady() 进入 handleAppReady() 后，先检查 isVersionMode；若命中就打印版本并 app.exit(0)。只有非版本模式，才会执行 initializeProcess()，然后通过 startBackendOrExit() 启动或确认 backend。原因有两层：一是初始化阶段要先接管数据目录、日志目录和配置恢复；二是 WebUI 与 resetpass 都依赖同一份 backend 运行时，不能各自再起一套进程去碰同一份 SQLite 数据。
