ACP 发送失败边界

同一个 send endpoint，在 ACP 平台层如何把 HTTP 非 2xx 转成可见错误。

课程 0003 边界：BackendHttpError 参考图谱

1. ACP 复用同一个发送接口

ipcBridge.acpConversation.sendMessage 最终复用 conversation.sendMessage，所以失败入口仍是 httpBridge 的 HTTP 非 2xx 处理。

if (!response.ok) {
  const message = await response.text();
  throw new BackendHttpError(message, response.status);
}

2. 平台 catch 决定用户看到什么

AcpSendBox 捕获错误后会调用 markSendFailed()，再用 buildSendFailureError() 分类。普通失败会追加 center tips；认证类错误会走 error stream。

sequenceDiagram
  participant ACP as AcpSendBox
  participant Adapter as httpBridge
  participant Backend as backend
  participant Runtime as runtimeView
  participant UI as MessageList
  ACP->>Adapter: sendMessage.invoke()
  Adapter->>Backend: POST /messages
  Backend-->>Adapter: 502 + error text
  Adapter-->>ACP: BackendHttpError
  ACP->>Runtime: markSendFailed()
  ACP->>UI: 添加 center error tips
  UI-->>ACP: 用户看到后端原始错误

3. E2E 证明范围

send-error-surfacing.e2e.ts 不证明所有 ACP 异常类型，只证明“发送接口返回后端原始错误时，UI 没有吞掉这段文本”。这对排查 agent 初始化失败特别关键。

4. 检索练习

如果测试只看到泛化错误文案，先查 buildSendFailureError.ts 的分类，再查 AcpSendBox.tsx 是否把原始 message 放进 tips。
