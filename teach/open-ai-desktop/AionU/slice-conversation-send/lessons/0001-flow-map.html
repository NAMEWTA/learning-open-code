入口点

用户看到的是 conversation 页面，但真正的提交入口在公共 SendBox。页面层先由 ChatConversation 按 conversation.type 分流：AionRS 会话进入 AionrsChat，ACP 会话进入 AcpChat。两者都渲染 MessageList 和平台 sendbox；本课主讲 AionrsSendBox。

SendBox 的 sendMessageHandler 做四件事：上传中直接返回；空输入且无 DOM 片段时返回；拼接引用和 DOM 片段；立即清空输入框后调用传入的 onSend(finalMessage)。这一层只负责“把草稿变成待发送文本”，不直接知道 AionRS 或 ACP。

主路径

AionrsSendBox.onSendHandler 收集上传文件和 @ 选择的文件，清空选择状态，然后看 runtime gate。若当前忙或已有排队命令，消息进入 useConversationCommandQueue；否则执行 executeCommand。

SendBox.submit
  -> AionrsSendBox.onSendHandler
  -> shouldEnqueueConversationCommand?
  -> executeCommand
  -> runtimeView.markSendStarted()
  -> ipcBridge.conversation.sendMessage.invoke()
  -> POST /api/conversations/:id/messages
  -> runtimeView.markSendAccepted(turn_id, runtime, msg_id)
  -> responseStream.on(message)
  -> useAionrsMessage.mergeLiveMessage()
  -> MessageList 渲染文本、工具、错误
  -> turnCompleted.on(event)
  -> runtime gate 回到可发送状态

关键判断：`ipcBridge` 这个名字保留了旧桥接形状，但会话发送不是传统 Electron IPC。`ipcBridge.ts` 把发送映射为 HTTP POST；`httpBridge.ts` 用 `fetch` 发请求，用 WebSocket 接收 `message.stream` 和 `turn.completed`。

状态与边界

状态更新分两条线：发送请求返回前，conversationRuntimeViewStore 用本地 pending 标记把 UI 变成 processing；请求被接受后记录 turn_id 和 msg_id；后端广播 turn.completed 后释放 gate。流式内容则由 useAionrsMessage 维护 waitingResponse、streamRunning、hasActiveTools，并把 chunk 合并进消息列表。

边界路径：平台 sendbox 传了 allowSendWhileLoading，所以忙时草稿不会被公共 SendBox 直接挡掉，而是由平台层检查 isBusy 和 hasPendingCommands 后排队。另一个失败边界在 ACP：如果 POST /messages 返回 502，httpBridge 抛 BackendHttpError，AcpSendBox 调用 markSendFailed 并添加 error tips；E2E 断言用户能看到后端原始错误。

检索练习

暂停回忆：不看上面的流程，写出“输入框提交后，哪一层决定立即发送还是排队？”

参考答案：不是公共 SendBox。它只调用 onSend；AionRS/ACP 平台 sendbox 用 runtime gate 计算 isBusy，再调用 shouldEnqueueConversationCommand 决定排队或执行。

主要来源：AionrsSendBox.tsx 的 onSendHandler/executeCommand 与 ipcBridge.ts 的 conversation.sendMessage。下一步若不清楚流事件如何变成具体气泡，继续追问 useAionrsMessage 和 MessageList。
