Team Mode：把多 Agent 团队装进同一个会话工作台

学习目标：15 分钟内看懂 Team Mode 的边界，以及一条 Team 消息如何从团队页进入现有 ACP / aionrs 会话运行时。

L1 模块总览 预计 15 分钟 主来源：TeamPage.tsx / TeamChatView.tsx / ipcBridge.ts

L0 回链：先看 AionU 项目地图。交叉参考：renderer-core 说明路由与 shell，common-adapter 说明桥接层，conversation-runtime 说明会话工作台。

1. 模块边界：Team Mode 不是新的聊天系统

Team Mode 位于 open-ai-desktop/AionU/packages/desktop/src/renderer/pages/team/。它负责团队页的 UI 编排：读团队记录、展示 leader 与 teammate tab、维护 active slot、显示运行状态、把权限模式写回 team record，并在删除成员或团队前清理关联 Cron。真正的 Agent 执行、消息持久化和 Team run 编排不在这个 renderer 目录中，而是通过 open-ai-desktop/AionU/packages/desktop/src/common/adapter/ipcBridge.ts 的 team 域进入 backend REST / WebSocket。

packages/desktop/src/renderer/pages/team/ ├── index.tsx ← /team/:id 入口，读取 team 记录 ├── TeamPage.tsx ← 团队工作台总装 ├── hooks/ ← tab、权限、session、run 状态 ├── components/ ← 创建 modal、tab、chat slot、发送 runtime └── utils/ ← workspace 视图与 Cron 清理

2. 打开团队页：先拿 team，再组装团队工作台

入口从 renderer-core 的路由表开始：open-ai-desktop/AionU/packages/desktop/src/renderer/components/layout/Router.tsx 在 TEAM_MODE_ENABLED 打开时把 /team/:id 指到 TeamIndex。TeamIndex 只做一件事：用 ipcBridge.team.get.invoke({ id }) 取团队记录，然后把 TTeam 交给 TeamPage。

TeamPage 再包两层上下文：TeamTabsProvider 负责 active slot、tab 排序、重命名和删除；TeamPermissionProvider 给会话发送框提供 team-only 能力，例如 warmupSession() 和 setSessionMode。这说明 Team Mode 的页面状态在 renderer，本地持久状态通过 common adapter 回到 backend。

3. 发送主线：TeamChatView 把“团队发送”注入现有 Chat

关键点在 open-ai-desktop/AionU/packages/desktop/src/renderer/pages/team/components/TeamChatView.tsx：它没有重写 ACP 或 aionrs 的聊天组件，而是按 conversation type 复用 AcpChat / AionrsChat，只额外传入 teamSendMessage 与 teamRuntime。leader 的输入走 ipcBridge.team.sendMessage；成员 tab 的输入走 ipcBridge.team.sendMessageToAgent。两者返回 ITeamRunAck 后，useTeamRunView 立即把 ack 映射成可显示的 run 状态。

const teamSendMessage = async ({ input, files }) => {
  if (isLeader) {
    const ack = await ipcBridge.team.sendMessage.invoke({ team_id, input, files });
    onTeamRunAck?.(ack);
    return;
  }
  const ack = await ipcBridge.team.sendMessageToAgent.invoke({
    team_id,
    slot_id,
    input,
    files,
  });
  onTeamRunAck?.(ack);
};

这样做的收益很直接：Team Mode 只接管“这条消息发给团队还是发给某个成员”，消息列表、附件、命令队列、模型选择和 stop 按钮仍尽量复用 conversation-runtime。完整接口和事件清单见 Team Mode reference。

4. 检索练习

暂停回忆：如果一个成员 tab 正在运行，为什么 Team Mode 不应该只依赖普通 conversation runtime 的 processing 状态？

参考答案与反馈

因为 Team run 是团队级状态，可能包含 leader、多个 teammate、pending wake、starting child turn、paused slot 等信息。useTeamRunView 需要消费 team.run* 和 team.childTurn* 事件，把每个 slot 的工作状态映射到发送框 gate；普通单会话 runtime 只知道当前 conversation 是否忙，不足以表达团队内多个 slot 的并发状态。
