1. 三条输入流

Team Mode 的难点不是“有没有事件”，而是同一段 UI 忙碌状态有三条输入：发送接口返回的 ITeamRunAck、WebSocket 推来的 team.run* 与 team.childTurn*、以及页面主动拉取的 getRunState 快照。它们都落到 open-ai-desktop/AionU/packages/desktop/src/renderer/pages/team/hooks/useTeamRunView.ts，再被 TeamPage 传给每个 slot。

ack 的价值是“快”：用户按下发送后，TeamChatView 立即拿到 REST 返回，调用 onTeamRunAck?.(ack)。如果只等 WebSocket，发送框可能短暂仍可点，成员 tab 也看不出 run 已被接受。WebSocket 的价值是“准”：它携带后端持续更新的 child 数量、pending wake 和 slot work。reconcile 的价值是“补”：重连、成员结构变化或 session 变化后，前端不假设本地 reducer 还完整，而是重新拉一次 run state。

2. 状态边界

这里有三个层次。团队级 run 表示一次任务整体是否 accepted/running/failed；child turn 表示某个成员会话中正在处理的 turn；slot work 表示某个 slot 是否有 pending wake、starting child、active turn、paused 或 suppressed work。不要把 activeRun 当成“所有输入框都忙”。真正控制输入框的是 slot 级状态。

export type ITeamRunAck = {
  team_run_id: string;
  team_id: string;
  target_slot_id: string;
  target_role: TeamRunTargetRole;
  accepted_slot_id: string;
  accepted_role: TeamRunTargetRole;
  status: TeamRunStatus;
  message_id?: string;
};

这段类型来自 open-ai-desktop/AionU/packages/desktop/src/common/types/team/teamTypes.ts。注意它没有 child turn 明细，只告诉你这次发送被哪个 slot 接受。因此 ack 只能生成一个保守的 slot work：先把 accepted_slot_id 标成有 pending wake；如果没有 accepted slot，才 fallback 到 target_slot_id。后续再由 WebSocket 或 reconcile 校准。

3. 判断法

维护时先问三个问题：第一，现象是不是刚发送后立刻出现？如果是，先查 ack 是否进入 applyAck()。第二，现象是不是 run 中途或 child turn 结束后出现？先看 runUpdated、childTurnCompleted 是否更新了 slot work。第三，现象是不是刷新、重连、增删成员后出现？先看结构事件有没有触发 reconcile。

这个判断法能避免两个常见误判：把 Team run 当成单会话 turn，或者把某个成员忙误扩散成整个 Team 忙。下一课会进入 ackToRunEvent() 和 reducer，专门看 ack 与 WebSocket 怎么合并。

4. 检索练习

暂停回忆：如果 leader 发送后，leader 输入框立即变忙，但 WebSocket 还没到，这个忙态最可能来自哪条输入流？

答案：来自 REST ack。TeamChatView 收到 ITeamRunAck 后调用 onTeamRunAck，useTeamRunView 再把 ack 转成带 pending wake 的 run event。WebSocket 后续负责校准，而不是第一时间唯一来源。
