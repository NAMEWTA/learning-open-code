学习目标

本课只看 renderer 收口：preload 注入的状态如何影响 React 根渲染，Router 如何决定默认首屏，以及 app-launch E2E 实际证明了什么。主要来源是 open-ai-desktop/AionU/packages/desktop/src/preload/main.ts:54-64、open-ai-desktop/AionU/packages/desktop/src/renderer/main.tsx:269-405、open-ai-desktop/AionU/packages/desktop/src/renderer/components/layout/Router.tsx:41-105 和 open-ai-desktop/AionU/tests/e2e/specs/app-launch.e2e.ts:10-27。

preload 把 main 状态带进 renderer

BrowserWindow 使用 preload 后，renderer 不是自己猜 backend port。preload 同步询问 main：backend port、初始语言、backend 是否失败、失败详情，然后通过 contextBridge 暴露到 window。

const backendPort = ipcRenderer.sendSync('get-backend-port') as number;
const initialLanguage = ipcRenderer.sendSync('get-initial-language') as string | null;
const backendStartupFailed = ipcRenderer.sendSync('get-backend-startup-failed') as boolean;
const backendStartupFailure = ipcRenderer.sendSync('get-backend-startup-failure') as unknown;
contextBridge.exposeInMainWorld('__backendPort', backendPort > 0 ? backendPort : 0);
contextBridge.exposeInMainWorld('__backendStartupFailure', backendStartupFailure ?? null);

这解释了为什么 `configService.initialize()` 能在 renderer 早期请求 `/api/settings/client`：Electron renderer 会优先使用 `window.__backendPort` 拼出 `http://127.0.0.1:{port}`。

React root 和 Router 首屏

renderer/main.tsx 先启动 Sentry、runtime patches、browser adapter、configService、i18n 和 provider 栈。正常路径里，Main 先等待 AuthProvider ready，再等待 bootstrapRendererConfig()；未 ready 时返回 null，避免 Router 提前渲染。

const Main = () => {
  const { ready } = useAuth();
  const [configReady, setConfigReady] = useState(false);

  useEffect(() => {
    if (!ready) return;
    void bootstrapRendererConfig().finally(() => setConfigReady(true));
  }, [ready]);

  if (!ready || !configReady) return null;
  return <Router layout={...} />;
};

桌面运行时的 `AuthProvider` 会因为 `window.electronAPI` 直接进入 authenticated/ready；WebUI 才会请求用户接口。这是桌面首屏能绕过登录页的关键。

Router 的默认落点

Router 使用 HashRouter。`/login` 在已认证时跳 `/guid`；受保护布局在 checking 时显示 loader，在未认证时跳 `/login`；index route 则跳 `/guid`。因此默认桌面首屏通常落在 `/guid`，再由懒加载页面显示内容。

<Route
  path='/login'
  element={status === 'authenticated' ? <Navigate to='/guid' replace /> : withRouteFallback(LoginPage)}
/>
<Route element={<ProtectedLayout layout={layout} />}>
  <Route index element={<Navigate to='/guid' replace />} />
  <Route path='/guid' element={withRouteFallback(Guid)} />
</Route>

边界、测试与练习

异常路径：如果 backend 启动失败，main 会写入 backendStartupFailureInfo，preload 暴露 window.__backendStartupFailure，renderer 根部改渲染 BackendStartupFailureDialog，而不是正常 provider + Router。

`app-launch.e2e.ts` 证明窗口有标题、body 可见且启动期没有关键 console error。它没有证明 `/guid` 的具体业务内容，也没有证明 backend 内部功能可用；它是启动烟雾测试，不是业务验收。

练习：如果 E2E 的 body 可见但出现 backend 失败弹窗，这条测试一定失败吗？

参考答案：不一定。前两个断言仍可能通过；第三个取决于错误收集器是否把相关 console 输出归为 critical。要证明正常 `/guid` 首屏，需要新增更具体的路由或文本断言。
