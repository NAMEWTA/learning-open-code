学习目标

本课只训练一个能力：看到 AionU 桌面启动问题时，先把源码定位到正确阶段。主要来源是 open-ai-desktop/AionU/packages/desktop/src/index.ts:943-950、open-ai-desktop/AionU/packages/desktop/src/index.ts:603-850 和完整参考图谱。

AionU 的首屏不是 renderer 单独决定的。Electron main 先做启动分支和 backend 前置；BrowserWindow 再加载 renderer；preload 把 backend 状态注入 window；React 最后根据失败状态、认证状态和 Router 默认规则进入首屏。

全链路地图

Electron 入口 ↓ 顶层前置：Chromium / Sentry / 单实例锁 / PATH / 模式标志 app.whenReady() ↓ handleAppReady() ↓ initializeProcess() ↓ process storage、bridge、i18n 前置 startBackendOrExit() ↓ backend port 或 backend failure 桌面模式分支 ↓ createWindow() ↓ preload 同步注入 backend port / failure renderer main.tsx ↓ React provider、认证 ready、renderer config ready Router.tsx ↓ 首屏：通常是 /guid；backend 失败时是启动失败弹窗

这张图里有 8 个阶段，所以本主题拆成 3 节短课：本课负责地图；第二课读 main/process 到窗口；第三课读 renderer、Router、失败边界和 E2E 证据。

  end

三条读码规则

先分模式。--version、--resetpass、--webui 会改写是否创建窗口；本切片聚焦默认桌面模式，但必须知道其他分支会提前退出或无窗口运行。
先 backend，后窗口。handleAppReady() 在创建桌面窗口前调用 initializeProcess() 和 startBackendOrExit()，避免 renderer 过早访问未准备好的配置和 backend port。
首屏由 Router 收口。renderer 的 Main 只在认证与配置 ready 后交给 Router；Router 再决定 /login、/guid 或兜底跳转。

交叉链接：如果你不确定 main 入口职责，先看 module-main-entry；如果不确定 process 前置，转到 module-process-infra；如果不确定 Router，转到 module-renderer-core。
