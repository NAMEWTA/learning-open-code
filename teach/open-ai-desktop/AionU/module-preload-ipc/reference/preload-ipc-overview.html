模块职责与架构位置

本模块负责把 Electron renderer 放进受控沙箱。preload 负责暴露最小可用 API；common/adapter/main.ts 负责把统一桥接事件接到 main；process/bridge/* 负责按能力域注册 provider；pet 多窗口 preload 负责各自极窄的专用通道。它处在 renderer 与 main 之间，是桌面端安全边界的一部分，而不是业务逻辑承载层。

主要 IPC channel 清单

Channel / 事件 方向 来源文件 用途
office-ai-bridge-adapter renderer ↔ main preload/main.ts / common/adapter/main.ts 统一业务桥接；renderer 用 emit 发请求，main 广播事件回所有窗口。
get-backend-port renderer ← main（sync） index.ts 把 backend 端口同步注入到 window.__backendPort。
get-initial-language renderer ← main（sync） index.ts 提供首屏语言，避免 renderer 首次加载时丢失初始值。
get-backend-startup-failed / get-backend-startup-failure renderer ← main（sync） index.ts 让安装完整性对话框等首屏逻辑可直接知道 backend 启动失败详情。
feedback:collect-logs renderer → main feedbackBridge.ts 收集近三天日志并转成可序列化字节数组。
feedback:capture-screenshot renderer → main feedbackBridge.ts 抓当前窗口截图；无窗口或失败时返回 null。
feedback:renderer-log renderer → main feedbackBridge.ts 把反馈上报阶段的诊断日志打到 main 控制台。
backend:recover-corrupted-database renderer → main index.ts 用户确认后走恢复流程并尝试重启 backend。
tray:navigate-to-guid 等 5 个托盘事件 main → renderer preload/main.ts 转成 window.dispatchEvent(new CustomEvent(...))，让 renderer 用 DOM 事件消费。
pet:state-changed / pet:eye-move / pet:resize main → pet renderer petPreload.ts 桌宠展示窗口的状态与外观同步。
pet:drag-start / pet:drag-end / pet:click / pet:context-menu pet renderer → main petHitPreload.ts 桌宠命中层交互事件。
pet:set-ignore-mouse-events / pet:hit-reset 双向 petHitPreload.ts 命中层鼠标穿透切换与重置。
pet:confirm-add / update / remove / theme main → pet confirm renderer petConfirmPreload.ts 确认浮窗的数据流与主题同步。
pet:confirm-respond / pet:confirm-drag-start / pet:confirm-drag-end pet confirm renderer → main petConfirmPreload.ts 确认响应和拖拽操作回传。

调用 / 阅读示例

推荐沿“窗口最小化”读一次完整链路：

  1. renderer 调用封装后的 ipcBridge.windowControls.minimize.invoke()。
  2. 主窗口 preload 通过 electronAPI.emit 把事件送进 office-ai-bridge-adapter。
  3. common/adapter/main.ts 中的 ipcMain.handle 解析 { name, data }，交给 bridge emitter。
  4. windowControlsBridge.ts 的 provider 找到聚焦窗口并执行 window.minimize()。
  5. 若窗口最大化状态变化，main 再用 maximizedChanged.emit 广播回 renderer。

这个例子说明：renderer 能表达意图，但真正有 OS 级副作用的对象始终留在 main。新增 IPC 能力时，最该审查的就是 preload 是否暴露过宽、provider 是否验证上下文，以及失败语义是否可预测。
