Preload 与 IPC：renderer 只能碰到被点名开放的能力

学习目标：15 分钟内看清 AionU 如何用 preload 把 renderer 限制在“受控桥接”里，而不是直接给它 Electron/Node 权限。

L1 模块总览 预计 15 分钟 主来源：preload/main.ts

L0 回链：先补整体地图可看 00-overview / 0001-project-map，再回到本课聚焦 preload 这条安全边界。

1. 模块职责：把“能做什么”改写成白名单

AionU 的 renderer 不是直接拿到 ipcRenderer 或 Node API，而是先经过 preload。主窗口入口 packages/desktop/src/preload/main.ts 用 contextBridge.exposeInMainWorld 只公开收口后的 electronAPI，再额外注入 backend 端口、初始语言和启动失败状态。这样 renderer 能做的是“调用白名单方法”和“读取启动快照”，而不是自由触达 main。

contextBridge.exposeInMainWorld('electronAPI', {
  emit: (name, data) => ipcRenderer.invoke(ADAPTER_BRIDGE_EVENT_KEY, JSON.stringify({ name, data })),
  on: (callback) => { /* 订阅统一桥接事件 */ },
  getPathForFile: (file) => webUtils.getPathForFile(file),
  collectFeedbackLogs: () => ipcRenderer.invoke('feedback:collect-logs'),
  captureFeedbackScreenshot: () => ipcRenderer.invoke('feedback:capture-screenshot'),
});

这里的边界很明确：跨域业务通信不开放裸 ipcRenderer，只允许经过 emit/on 进入统一桥；少数 Electron 专属能力，例如拖拽文件绝对路径、反馈截图、数据库恢复，则被列成显式独立方法，方便审计特权来源。

2. 在整体架构里的位置：preload 只负责转译，不负责业务

packages/desktop/src/ ├── preload/ ← renderer 可见能力的最窄入口 │ ├── main.ts ← 主窗口 bridge 与启动快照 │ ├── petPreload.ts ← pet 展示窗口只收事件 │ ├── petHitPreload.ts ← pet 命中层只发拖拽/点击 │ └── petConfirmPreload.ts ← pet 确认窗收确认事件并回传响应 ├── common/ │ └── adapter/ │ └── main.ts ← main 侧统一桥接与窗口广播 └── process/ └── bridge/ ← 各能力域的 provider 注册

preload 不实现窗口控制、主题切换或 WebUI 启动，它只把 renderer 请求安全地转译给 main。真正的统一接线点在 packages/desktop/src/common/adapter/main.ts：它接收 renderer 传来的事件名，把请求交给 main 侧 bridge emitter，再把结果广播回窗口。

ipcMain.handle(ADAPTER_BRIDGE_EVENT_KEY, (_event, info) => {
  const { name, data } = JSON.parse(info);
  return Promise.resolve(emitter.emit(name, data));
});

所以这条链可以记成三段：renderer 只能调用 preload 白名单；preload 只把命名好的事件交给 adapter；adapter 再分发给 main 已注册的 provider。权限口子不在 React 组件里，而在 preload 暴露面。

3. 对外接口与内部分层：主窗口是通用桥，pet 窗口是专用桥

主窗口 preload 对外接口分三类：通用业务桥 electronAPI.emit/on；少数原生特权，如 getPathForFile、反馈截图、数据库恢复；启动只读状态，如 __backendPort。而 pet 三个 preload 明显更窄：展示窗只订阅状态，命中层只发拖拽和点击，确认窗只处理确认事件与 respond 回传。AionU 没复用一个“大而全”的 preload，而是按窗口职责切成更小的表面。

这节课只保留理解边界所需的摘要；按窗口对象展开的完整暴露清单、只读全局和 channel 名，请直接看 reference 的接口总表。

这正是安全边界的关键启发：窗口越专用，暴露面越该变窄。

4. 阅读示例与检索练习

一个好用的阅读切口是“反馈截图”链路：renderer 在反馈上下文里调用 window.electronAPI.captureFeedbackScreenshot()，preload 把它收口成 ipcRenderer.invoke('feedback:capture-screenshot')，main 再定位当前窗口并抓图。测试文件 tests/unit/feedback/feedbackBridge.test.ts 进一步证明：没有所属窗口、窗口已销毁、截图为空时都返回 null，说明这条桥不仅有限权，还把失败语义固定了。

const result = await handler({ sender: {} });
expect(result).toBeNull();
expect(currentWindow.webContents.capturePage).not.toHaveBeenCalled();

暂停回忆：如果某个 renderer 页面想“直接”最小化窗口，它真正能接触到的最近一层代码应该是哪一层？为什么不是 BrowserWindow？

参考答案与反馈

最近一层是 preload 暴露的 electronAPI.emit 或经其封装后的 ipcBridge.windowControls.minimize 调用，而不是 BrowserWindow。因为窗口对象只存在 main 进程，renderer 只能通过 preload 白名单跨边界发请求；真正调用 window.minimize() 的代码在 main 侧 provider 里。
