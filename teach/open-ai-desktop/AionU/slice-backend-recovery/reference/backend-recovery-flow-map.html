4. 失败分类与 UI 去向

reason识别条件renderer 去向用户动作测试证据
`backend_package_architecture_mismatch``details.stage === 'startup_architecture_check'`。专用下载 Modal。下载匹配架构版本。`backendStartupFailure.test.ts` 架构不匹配用例。
`backend_incomplete_installation`packaged app 在 `resolve_binary` 阶段缺 `bundled-aioncore/`、runtime 目录、`managed-resources/` 或 backend binary。`InstallationIntegrityModalHost`，kind 为 `incomplete_installation`。发送诊断或下载最新版。unit 覆盖缺目录和缺二进制；E2E 覆盖 debug 注入弹窗与诊断上报。
`backend_incompatible_runtime`stderr/stdout 中出现 `GLIBC_x.y not found`。无 footer 的 runtime Modal。阅读所需 GLIBC 版本；当前 UI 不给重建按钮。unit 覆盖版本提取和排序。
`backend_local_data_repair_failed``BOOTSTRAP_SERVICE_INIT_FAILED` + `services.init` + agent_metadata 数据库查询 invalid UTF-8 签名。`InstallationIntegrityModalHost`，kind 为 `local_data_repair`。发送诊断；不直接重启。unit 覆盖完整签名与模糊文本不误判。
`backend_recoverable_database_corruption``BOOTSTRAP_DATA_INIT_FAILED` + `database.recoverable_corruption`。`InstallationIntegrityModalHost`，kind 为 `recoverable_database_corruption`。发送诊断或确认重建数据库。unit 覆盖分类、恢复可用性、restart 失败边界和 preload IPC。
`backend_data_migration_failed``BOOTSTRAP_DATA_INIT_FAILED` + `database.migration` 或 `database.schema_repair`。`InstallationIntegrityModalHost`，kind 为 `data_migration`。发送诊断；不显示重建按钮。unit 覆盖 migration 与 schema repair。
`backend_startup_failed`无法命中以上分类时兜底，保留 boundary code/stage。`InstallationIntegrityModalHost` 默认 incomplete style 文案。发送诊断；没有恢复按钮。unit 覆盖通用失败和无关服务初始化失败。

5. 数据库恢复边界

允许恢复

`recoverCorruptedDatabaseAfterUserConfirmation()` 第一行读取当前 failure，只有 `backend_recoverable_database_corruption` 继续执行。成功路径是 `stopBackend()`、`startBackendWithRecovery()`、`markReady(port, 'backendManager.recoverCorruptedDatabase')`、`reloadMainWindow()`。

拒绝恢复

没有 active failure，或 reason 是 `backend_data_migration_failed` 等其他类型时，函数抛出 `backend_corrupted_database_recovery_not_available`，并且不会 stop 或 restart backend。

重启失败

如果 `startBackendWithRecovery()` 抛错，main 会再次 `markBackendStartupFailed()` 并采集失败；单元测试确认此时不会 `markReady()` 或 `reloadMainWindow()`。

6. 证据边界

证据证明了什么没有证明什么
`tests/unit/bootstrap/backendStartupFailure.test.ts`分类器和 modal action 分流的输入输出稳定。不启动真实 Electron，也不证明真实资源目录一定缺失。
`tests/unit/bootstrap/recoverCorruptedDatabase.test.ts`恢复动作的守卫、成功顺序和 restart 失败边界。不证明 aioncore 内部数据库重建细节。
`tests/unit/bootstrap/recoverCorruptedDatabasePreload.test.ts`preload 暴露的恢复方法确实 invoke main channel。不证明 renderer 按钮点击后的完整 UI 反馈。
`tests/e2e/specs/installation-integrity.e2e.ts`debug 注入 `backend_incomplete_installation` 时，窗口中出现安装完整性弹窗、诊断按钮、下载按钮，点击诊断后记录上报标记。不覆盖 GLIBC、架构不匹配、数据库重建和所有真实二进制启动失败。
普通 app-launch 烟雾测试最多证明窗口和 renderer 能加载。不能证明 backend recovery 链路，也不能把可见 body 等同于 backend 启动成功。
