Renderer 核心：provider、路由与 shell 怎样接上

学习目标：15 分钟内看懂 AionU renderer 启动后，怎样从根入口依次拼出 provider 栈、受保护路由和统一 UI shell。

L1 模块总览 预计 15 分钟 主来源：main.tsx / Router.tsx / Layout.tsx

1. 从根入口先拿到“能渲染”的最低条件

packages/desktop/src/renderer/main.tsx 先做三件基础工作：尽早挂上 Sentry 与运行时补丁、提前初始化 configService，再把 i18n、样式和 provider 栈装进根组件。真正值得记住的是 Main 里的双重闸门：useAuth() 给出认证就绪信号，bootstrapRendererConfig() 等待配置初始化结束；两者没完成前直接返回 null，因此路由树不会过早读取空配置。

const Main = () => {
  const { ready } = useAuth();
  const [configReady, setConfigReady] = useState(false);

  useEffect(() => {
    if (!ready) return;
    void bootstrapRendererConfig().finally(() => setConfigReady(true));
  }, [ready]);

  if (!ready || !configReady) return null;
  return <Router layout={<ConversationHistoryProvider><Layout sider={<Sider />} /></ConversationHistoryProvider>} />;
};

这意味着 renderer 的第一个职责不是“展示页面”，而是先把全局运行环境变成稳定态。完整 provider 清单和职责见 参考文档。

2. Router 负责把认证状态翻译成页面入口

Router.tsx 用 HashRouter 包住整个 panel，并把页面组件全部做成懒加载。这里的关键不是路由数量，而是保护方式：ProtectedLayout 先看认证状态，checking 时给 AppLoader，未认证就跳 /login，认证后才把外层传入的 Layout 克隆出来承载子路由。于是 renderer 核心把“登录门槛”放在壳层前，而不是散落到每个页面组件内部。

<Route element={<ProtectedLayout layout={layout} />}>
  <Route index element={<Navigate to="/guid" replace />} />
  <Route path="/guid" element={withRouteFallback(Guid)} />
  <Route path="/conversation/:id" element={withRouteFallback(Conversation)} />
  <Route path="/settings/agent" element={withRouteFallback(AgentSettings)} />
</Route>

读这层时，把页面当作“挂点”即可：/guid 是默认工作台，/conversation/:id 和 /team/:id 是会话工作区，设置页是一串独立入口。完整路由表和历史兼容重定向请跳到 reference。

3. Layout 把系统级交互收口成统一壳层

Layout.tsx 不是普通页面容器，而是 renderer 的 UI shell。它在最外层建立 LayoutContext 与 NavigationHistoryProvider，然后接入深链、通知点击、浏览器通知、目录选择器、会话快捷键、托盘事件和全局导航暴露。换句话说，Router 决定“去哪页”，Layout 决定“这些页共享什么交互框架”。

useDeepLink();
useNotificationClick();
useBrowserNotification();
useConversationShortcuts({ navigate });

return (
  <LayoutContext.Provider value={{ isMobile, siderCollapsed: collapsed, setSiderCollapsed: setCollapsed }}>
    <NavigationHistoryProvider>
      <Titlebar workspaceAvailable={workspaceAvailable} />
      ...
      <Outlet />
    </NavigationHistoryProvider>
  </LayoutContext.Provider>
);

一个很实用的阅读顺序是：先看 hook 名字，再追它们各自监听的事件源。比如 useDeepLink 监听 main process 发来的深链 payload，useBrowserNotification 监听会话流消息并在后台标签页弹通知，二者都不直接渲染页面，却决定了用户如何被带回正确路由。

4. 检索练习

暂停回忆：如果用户启动应用后直接点击系统通知回到某个会话，这条链路更可能先经过 Router.tsx 还是 Layout.tsx？为什么？

参考答案与反馈

先经过 Layout.tsx。原因是通知点击由 useNotificationClick() 订阅并调用 navigate('/conversation/:id')；Router 随后只负责把这个路径匹配到对应页面。如果你先想到 Router，说明你抓到了“目标页面”，但还没分清“谁发起导航”和“谁解释导航”。

推荐主来源：先回读 main.tsx，再用官方文档对照 HashRouter 的职责。接着追问 agent：请按“事件源 → hook → navigate/状态 → 路由页面”带我读一次 renderer 的真实调用链。
