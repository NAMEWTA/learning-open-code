HTTP helper 怎样伪装成 bridge provider

L4 深度剖析 预计 15 分钟 主来源：httpBridge.ts / ipcBridge.ts

上一课：问题框架。下一课：错误、事件与 E2E。速查：provider-like invoke。

1. 兼容点是对象形状

@office-ai/platform 的旧 bridge provider 给 renderer 一个稳定对象：provider() 用于注册处理器，invoke() 用于发起调用。AionU 的 httpBridge.ts 没有要求调用方知道 fetch、method、URL、JSON 解包；它直接生产同形对象。这里的 provider 是空函数，因为 HTTP 路由处理器已经在 aioncore，不再由 renderer/main 侧注册。真正工作的部分是 invoke：解析路径、映射 body、调用 httpRequest，最后返回泛型结果。

这个设计的微妙点是：它兼容的是“调用协议”，不是“执行位置”。旧 IPC provider 的执行位置通常在 Electron main；HTTP provider 的执行位置在 backend；但 renderer 看到的 TypeScript 形状保持一致。这样一来，页面组件不需要知道某个能力是否完成迁移，类型签名也能继续表达参数和返回值。

type ProviderLike<Data, Params> = {
  provider: (handler: (params: Params) => Promise<Data>) => void;
  invoke: Params extends undefined ? () => Promise<Data> : (params: Params) => Promise<Data>;
};
export function httpPost<Data, Params = undefined>(
  path: string | ((params: Params) => string),
  mapBody?: (params: Params) => unknown
): ProviderLike<Data, Params> {
  return {
    provider: () => {},
    invoke: (async (params?: Params) => {
      const resolvedPath = typeof path === 'function' ? path(params!) : path;
      const body = mapBody ? mapBody(params!) : params;
      return httpRequest<Data>('POST', resolvedPath, body);
    }) as ProviderLike<Data, Params>['invoke'],
  };
}

2. URL 和 body 映射保留旧参数语义

renderer 仍可以传旧的前端参数名，例如发送消息时传 input、loading_id、inject_skills。adapter 负责把它变成 backend REST body：input 改为 content，路径从 conversation_id 拼到 URL。这个位置很关键：如果把映射散落到页面组件，后续每个 UI 调用点都会被 backend DTO 污染；如果放进 backend，则旧 renderer 语义无法显式保留。adapter 正好是 contract 翻译层。

读这种映射时要同时看“路径”和“body”。路径里的字段通常是资源定位，比如 conversation_id、team_id、job_id；body 里的字段才是命令参数。把二者分清，能避免把同一个 id 同时塞进 URL 和 body，或遗漏 backend 需要的 snake_case 字段。

sendMessage: httpPost<ISendMessageResult, ISendMessageParams>(
  (p) => `/api/conversations/${p.conversation_id}/messages`,
  (p) => ({
    content: p.input,
    files: p.files,
    loading_id: p.loading_id,
    inject_skills: p.inject_skills,
  })

3. response mapper 是 contract 保险丝

httpRequest 会统一解开后端的 { success, data } 包装；但解包不等于完成兼容。前端常依赖 camelCase、旧枚举或组合字段，后端 wire contract 可能是 snake_case 或另一套默认语义。withResponseMap 把“调用传输”和“返回值修正”串起来，例如 conversation 创建后再走 fromApiConversation，team 列表走 fromBackendTeamList，文件快照把 relative_path 修成 relativePath。维护者看到新 REST 能力返回结构和旧前端类型不一致时，应优先补 mapper，而不是让 UI 兼容两种形状。

mapper 还有一个测试价值：它把兼容规则集中在 adapter 附近，E2E helper 或 feature test 可以针对最终前端形状断言。否则每个测试都要知道 backend 的 wire DTO，测试会从“用户行为验证”滑向“后端字段实现细节验证”。

边界提醒：common/api/* 不是这里的 REST route adapter。它处理多 provider client、协议转换和 key 轮转；common/adapter/* 才是 renderer 调用面到 IPC/REST/WS 的 contract 层。

4. 读代码时的三步法

追一个 adapter provider 时，先看 ipcBridge.ts 的导出：它是 bridge.buildProvider、httpPost，还是手写 provider-like 对象。再看路径函数和 mapBody，确认旧参数如何变成 backend DTO。最后看是否包了 withResponseMap 或领域 mapper；若没有 mapper，说明后端返回已经满足旧前端语义，或这里存在潜在兼容风险。

做维护评审时，尤其要警惕“为了省代码删除 mapper”的改动。只要 renderer 仍以旧类型为 contract，mapper 就是兼容边界；删除它等于把迁移成本推给所有调用方和测试。

判断题：看到 provider: () => {} 就说明这个能力还没实现。

参考答案与反馈

错误。对 HTTP provider 来说，空 provider 是兼容旧 bridge 形状；真实处理器在 backend REST route。只有 stubProvider 这类显式返回默认值并打印 warning 的能力，才表示 backend 尚未实现。
