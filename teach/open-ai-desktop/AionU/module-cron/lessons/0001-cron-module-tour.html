Cron 模块：把“计划执行”接到会话工作台

学习目标：15 分钟内看懂定时任务列表页如何通过统一 hook 和 common adapter 调用 `/api/cron/*`，并把执行结果回收到会话历史。

L1 模块总览 预计 15 分钟 主来源：ScheduledTasksPage / useCronJobs / ipcBridge

L0 回链：先看 AionU 项目地图。交叉主题：桥接层看 module-common-adapter，路由壳层看 module-renderer-core，执行后的会话回收看 module-conversation-runtime。

1. 先给模块一句话定位

Cron 模块不是调度器本体，而是 AionU renderer 中的“定时任务管理面”。它负责让用户查看任务、切换启停、进入详情、手动创建，或跳到首页用 AI 对话创建任务；真正的任务存储、执行和事件流由 backend 通过 /api/cron/* 提供。

读码时先抓三层即可：open-ai-desktop/AionU/packages/desktop/src/renderer/pages/cron/ScheduledTasksPage/index.tsx 是列表入口，open-ai-desktop/AionU/packages/desktop/src/renderer/pages/cron/useCronJobs.ts 是状态和订阅层，open-ai-desktop/AionU/packages/desktop/src/common/adapter/ipcBridge.ts 是 HTTP/WS 调用面。

2. 一条主线：列表页怎样拿到任务

用户点击侧边栏的“定时任务”后，renderer-core 的路由命中 /scheduled，懒加载 ScheduledTasksPage。页面本身不直接写 fetch，而是调用 useAllCronJobs()：它先请求全部任务，再订阅 created、updated、removed 三类事件。这样列表页既能首次展示，也能在后台任务变化时自动更新。

const { jobs, loading, pauseJob, resumeJob } = useAllCronJobs();

const handleToggleEnabled = async (job) => {
  if (job.enabled) await pauseJob(job.id);
  else await resumeJob(job.id);
};

这里的关键判断是：页面只表达用户意图，hook 负责状态一致性，adapter 负责传输。完整 hook 返回值和事件优先级放在 reference。

3. adapter 把意图落成 `/api/cron/*`

useAllCronJobs() 最终调用的是 common adapter 的 ipcBridge.cron。这个名字沿用了旧桥接风格，但 Cron 域已经主要走 HTTP 和 WebSocket：列表、创建、更新、删除、立即执行是 REST；任务创建、更新、删除、执行完成通知是 WS 事件。

export const cron = {
  listJobs: httpGet<ICronJob[], void>('/api/cron/jobs'),
  updateJob: httpPut<ICronJob, { job_id: string; updates: ICronJobUpdateParams }>(
    (p) => `/api/cron/jobs/${p.job_id}`
  ),
  onJobUpdated: wsEmitter<ICronJob>('cron.job-updated'),
};

所以新增 Cron 能力时，不应先找 Electron main process，而应先问“backend 是否已经拥有这个状态”。如果答案是肯定的，接口通常应该落在 common adapter 的 HTTP/WS 域，再由页面 hook 消费。
