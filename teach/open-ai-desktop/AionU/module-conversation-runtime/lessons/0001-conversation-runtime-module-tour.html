Conversation Runtime：统一会话壳层怎样接住两种平台

学习目标：15 分钟内看懂 AionU 会话页怎样从路由入口进入统一壳层，再把消息、workspace、preview 与 ACP / aionrs 平台实现组织成同一套交互。

L1 模块总览 预计 15 分钟 主来源：conversation/index.tsx / ChatConversation.tsx / ChatLayout/index.tsx

1. 页面入口先把“当前会话”变成稳定对象

packages/desktop/src/renderer/pages/conversation/index.tsx 不是聊天本体，而是路由落点适配层。它从 URL 里的 :id 读取会话，监听 ipcBridge.conversation.listChanged 以便会话更新后重取数据，并在会话切换时调用 closePreview()，避免上一个会话的预览标签残留到新页面。如果目标会话已经被删除，它会 toast 提示后导航回首页。换句话说，conversation runtime 的第一步不是渲染消息，而是把“我要读哪个会话”这个前提先校准好。

useEffect(() => {
  if (!id) return;
  if (previousConversationIdRef.current !== id) {
    closePreview();
  }
  previousConversationIdRef.current = id;
}, [id, closePreview]);

const { data, isLoading } = useSWR(id ? `conversation/${id}` : null, () => {
  return getConversationOrNull(id!);
});

这层和 renderer-core 短课 的关系是：前者负责“路由到了会话页”，后者从这里开始负责“会话页内部怎么运行”。

2. ChatConversation 先做统一装配，再按平台分叉

packages/desktop/src/renderer/pages/conversation/components/ChatConversation.tsx 是本模块真正的总装点。它先统一处理标题栏、workspace 开关、定时任务按钮、预设助手信息与模型选择器，然后根据 conversation.type 把内容分发给 AcpChat、AionrsChat 或只读兼容页。重点不是 if/else 数量，而是“壳层复用、平台差异下沉”：Header 和三栏布局保持一致，平台差异只落在消息流和发送框实现中。

下面是总装层节选，省略了部分 props 和非关键分支。

if (conversation && conversation.type === 'aionrs') {
  return <AionrsConversationPanel conversation={conversation} sliderTitle={sliderTitle} />;
}

return (
  <ChatLayout conversation_id={conversation?.id} workspacePath={conversation?.extra?.workspace} ...>
    {conversationNode}
  </ChatLayout>
);

因此读源码时要先抓两层边界：ChatConversation 决定“当前会话属于哪种运行时”，而真正的平台行为放进 platforms/acp/ 与 platforms/aionrs/。完整入口清单和差异表见 reference。

3. ChatLayout 把消息、preview、workspace 固定在同一张桌面上

packages/desktop/src/renderer/pages/conversation/components/ChatLayout/index.tsx 是 conversation runtime 的 UI 壳层。它一边管理标题重命名、workspace 折叠、响应式宽度和拖拽分隔；一边通过 usePreviewContext() 决定 preview 是否打开。源码里保持稳定的是聊天区 children 所在的统一 DOM 结构，PreviewPanel 则按 isPreviewOpen 条件挂载，所以打开/关闭预览时消息区不会卸载重建。平台组件再往里面插入 MessageList 和各自的 SendBox，于是“消息流”“文件工作区”“文档预览”三块能力能在一份布局里长期并存。

packages/desktop/src/renderer/pages/conversation/ ├── index.tsx ← 路由入口，装载具体会话 ├── components/ │ ├── ChatConversation.tsx ← 会话总装，按类型分发平台 │ └── ChatLayout/ │ └── index.tsx ← 标题栏、消息区、preview、workspace 壳层 ├── platforms/ │ ├── acp/ ← ACP 消息流与发送框 │ ├── aionrs/ ← aionrs 消息流与发送框 │ └── useConversationCommandQueue.ts ← 统一排队与执行闸门 ├── Preview/ │ └── context/ │ └── PreviewContext.tsx ← 预览标签页与 sendbox 注入 └── Workspace/ └── index.tsx ← 文件树、改动列表与预览打开点

如果你想继续深读，推荐顺序是：先从 ChatLayout 看三栏骨架，再读 packages/desktop/src/renderer/pages/conversation/Preview/context/PreviewContext.tsx 和 packages/desktop/src/renderer/pages/conversation/Workspace/index.tsx，最后回到平台目录看发送与流式状态怎样消费这些公共能力。module-common-adapter 可视为候选回链，因为消息结构与 IPC/HTTP 桥接能力最终都要经过 common 层定义。

4. 检索练习

暂停回忆：为什么 AionU 没把 ACP 和 aionrs 完全拆成两套页面，而是先保留同一个 ChatLayout 壳层？请至少说出两个共享能力。

参考答案与反馈

因为两种平台共享的是会话工作台，而不是底层协议。最关键的共享能力至少有两类：一是消息区外层布局，例如标题栏、workspace 折叠、preview 面板；二是运行时协作能力，例如 useConversationRuntimeView 驱动的发送闸门、命令队列、文件预览注入。如果你只答“UI 统一”，说明方向对了，但还要继续把共享能力说具体。
