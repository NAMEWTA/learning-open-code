AionU 项目地图：先抓住三条主线

学习目标：15 分钟内建立 AionU 的全局读码坐标，知道入口、包边界和后续深挖方向。

L0 项目总览 预计 12 分钟 主来源：README + Electron 入口

1. 一句话定位

AionU 上游项目名是 AionUi。它是一款免费开源的 Cowork 桌面应用：用户在一个现代聊天界面里调用内置 Agent、外部 CLI Agent、多 Agent Team、Office 助手、Cron 自动化和远程 WebUI。对读源码的人来说，最重要的不是先记功能清单，而是抓住三条主线：桌面壳负责本机体验，aioncore backend 负责运行时能力，WebHost/WebCLI 让同一套 UI 脱离 Electron 运行。

后续读码时可以把 README 的功能词映射到源码层：会话、权限、技能、Team、Cron、远程访问是用户看到的功能；Electron main、preload、renderer、HTTP/WS adapter、backend lifecycle 是开发者需要追踪的结构。

2. 启动链路

先读三个入口即可建立骨架：open-ai-desktop/AionU/package.json 告诉你 workspace、脚本和依赖；open-ai-desktop/AionU/packages/desktop/src/index.ts 是 Electron 总入口；open-ai-desktop/AionU/packages/desktop/src/preload/main.ts 是 renderer 能触碰主进程能力的安全边界。

import './process/utils/configureChromium';
import { initializeProcess } from './process';
import { startBackendOrExit } from './process/startup/backendStartup';
import { BackendLifecycleManager, startWebHost } from '@aionui/web-host';

这几行展示了启动顺序的核心：先配置 Chromium，再初始化主进程存储与 bridge，然后启动 aioncore backend，最后根据参数进入桌面窗口、无窗口 WebUI 或 resetpass 分支。preload 只暴露受控的 electronAPI、backend port、启动失败信息和托盘事件，renderer 不直接调用 Node.js 或 Electron main API。

3. 顶层目录地图

AionU/ ├── packages/ ← workspace 包集合 │ ├── desktop/ ← Electron 三进程桌面应用 │ ├── web-host/ ← 后端启动与静态站点服务 │ ├── web-cli/ ← 独立 WebUI 命令行入口 │ └── shared-scripts/ ← 打包与资源校验脚本 ├── docs/ ← 用户、贡献者、PRD 文档 ├── resources/ ← 图标、演示图、打包资源 ├── tests/ ← Vitest 与 Playwright 用例 └── scripts/ ← 构建、WebUI、诊断脚本

读码顺序建议是：先看 open-ai-desktop/AionU/packages/desktop/src/index.ts 的分支，再看 open-ai-desktop/AionU/packages/desktop/src/renderer/main.tsx 的 provider 栈，最后用 open-ai-desktop/AionU/packages/desktop/src/common/adapter/ipcBridge.ts 对照 REST/WS API。完整技术栈、豁免清单和后续课程候选在 L0 参考文档。

4. 检索练习

暂停回忆：如果你要追踪“WebUI 模式如何复用桌面 backend”，应该先看哪个入口，为什么？

参考答案与反馈

先看 open-ai-desktop/AionU/packages/desktop/src/index.ts 的 isWebUIMode 分支，再跳到 @aionui/web-host 的 startWebHost。前者决定 Electron 进程是否创建窗口、是否复用已启动的 backend port；后者负责组合 backend handle 与静态服务。若你回答 renderer 入口，说明你抓到 UI，但还没抓到运行形态的分叉点。
