1. 项目定位

一句话描述：AionU / AionUi 是一个免费开源的跨平台 Cowork 桌面应用，把内置 Agent、外部 CLI Agent、多 Agent Team、Office 生成、Cron 自动化和远程 WebUI 统一到一个可观察、可授权的工作界面中。

用户侧入口来自 open-ai-desktop/AionU/readme.md：内置 Agent、零配置、任意 API Key、多 Agent、远程访问、跨平台、24/7 自动化。源码侧入口则集中在 Electron 三进程、aioncore backend 生命周期、WebHost 静态服务、REST/WS adapter 和 renderer Provider 栈。

3. 总体架构

graph TD User[用户操作] --> Renderer[React Renderer] Renderer --> Providers[Auth Theme Preview Feedback Providers] Renderer --> Adapter[common adapter] Adapter --> RestWs[REST / WebSocket to aioncore] Renderer --> Preload[preload contextBridge] Preload --> NativeIPC[Electron native IPC] NativeIPC --> Main[Electron main] Main --> ProcessInit[process initStorage / bridge / i18n] Main --> Backend[BackendLifecycleManager] Backend --> AionCore[aioncore backend] Main --> Window[BrowserWindow / tray / deep link] Main --> WebHost[startWebHost] WebHost --> Static[static server] WebCLI[aionui-web CLI] --> WebHost Static --> Browser[Browser WebUI]

关键判断：AionU 的 renderer 并不直接拥有核心 Agent 能力。它通过 common adapter 把业务请求路由到 aioncore 的 REST/WS；只有窗口、托盘、对话框、自动更新、缩放、CDP、deep link 等 Electron 原生能力保留在 IPC。

6. 架构决策

三进程边界严格化

Main 使用 Node.js 和 Electron main API，renderer 使用 DOM/React，preload 是唯一安全桥。这个决策降低了 Electron 权限泄漏风险。

业务桥接后端化

ipcBridge.ts 明确把会话、Team、Cron、助手等业务转向 REST/WS，Electron IPC 只保留原生桌面能力。

桌面与 WebUI 复用运行时

@aionui/web-host 让 Electron WebUI 模式复用已启动 backend，也让 aionui-web 独立启动自己的 backend。

启动失败显式可恢复

backend 启动失败会分类、上报并传给 renderer，UI 通过安装完整性弹窗或恢复数据库流程给用户反馈。

贡献规范服务架构边界

AGENTS 和 file-structure guide 把目录规模、命名、UI 组件、i18n、测试覆盖纳入硬约束，减少多 Agent 协作时的结构漂移。

patches/*.patch依赖补丁不是主业务路径。若构建或运行时依赖补丁行为，作为 L4 兼容性深挖材料。

10. 文档空白与风险

缺失架构文档：open-ai-desktop/AionU/docs/README.md 与 open-ai-desktop/AionU/AGENTS.md 都指向 docs/architecture/overview.md，但当前工作区没有该文件或 docs/architecture/ 目录。本参考文档因此以入口源码、贡献文档和 README 重建架构。

后端边界：AionU 启动并调用 aioncore，但本轮未读取 AionCore 仓库。所有 backend 内部行为都应在后续课程中以 AionCore 源码或 HTTP/WS contract 进一步验证。
