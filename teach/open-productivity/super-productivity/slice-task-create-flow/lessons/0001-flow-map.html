新增任务与短语法解析全链路

新增任务不是“输入框直接写 task”。它先把输入解析成临时状态，再由 `TaskService` 组装任务实体，最后通过带持久化元数据的 action 进入 meta-reducer 和 op-log。

3. 字段与持久化边界

提交时，src/app/features/tasks/add-task-bar/add-task-bar.component.ts 把临时状态折叠成 `taskData`：`date/time` 变为 `dueDay` 或 `dueWithTime`，`deadlineDate/deadlineTime` 变为 `deadlineDay` 或 `deadlineWithTime`，`estimate` 变为 `timeEstimate`，`spent` 变为 `timeSpentOnDay`，note 写入 `notes`。如果存在 timed due，组件随后调用 `scheduleTask` 补上 `remindAt` 并维护 Today 顺序；如果是重复任务，则交给 src/app/features/task-repeat-cfg/task-repeat-cfg.service.ts 创建 repeat cfg。

op-log 捕获的是 action，不是输入框状态。`TaskSharedActions.addTask` 是 Create op；`TaskSharedActions.applyShortSyntax`、`scheduleTaskWithTime` 是 Update op；新 tag 和 repeat cfg 各自生成自己的 persistent action。参考清单见 新增任务链路速查。

4. 暂停 60 秒判断

场景：用户在 Today 视图输入 `Write report +Client #urgent @tomorrow !friday 45m` 并回车。请先不看答案，判断它是否还属于 Today，Project/Tag 从哪里来，op-log 至少会捕获哪类 action。

参考答案：如果 `@tomorrow` 被解析为日期而非具体时间，最终 `dueDay` 是明天，因此不属于 Today，会进入 planner 的明天列表。Project 来自 `+Client` 匹配到的项目；Tag 来自已存在的 `#urgent`，若不存在会先确认创建 tag。op-log 至少捕获 `TaskSharedActions.addTask`；如果新建 tag、补提醒、创建 repeat cfg 或由 store effect 应用短语法，还会有对应的 persistent action。
