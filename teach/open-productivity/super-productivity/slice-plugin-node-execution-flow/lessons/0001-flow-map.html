2. 四层边界别混读

层存什么谁能写典型源码
renderer token 缓存 当前会话 grant token PluginBridgeService 在 main 返回 token 后写入 src/app/plugins/plugin-bridge.service.ts
main session grant token + webContentsId 只有 native Allow 或 persisted consent 命中后 mint electron/plugin-node-executor.ts
持久 consent uploaded 插件 ask-once 记录 只有 main Allow 后写入；renderer 只能 clear electron/plugin-node-consent-store.ts
脚本执行 vm 沙箱或 spawn 子进程 main 在校验 grant 后才执行 electron/plugin-node-executor.ts

`PluginBridgeService` 在构造时调用 `consumePluginNodeExecutionApi()`，必须早于任何 `new Function` 插件代码——否则插件可能抢走 one-shot handoff。 built-in 插件每会话弹 verified dialog；uploaded 插件走 unverified 文案，Allow 后可跨会话免重复弹窗，直到 disable、卸载或 re-upload 触发 `clearNodeExecutionConsent()`。

3. 关键入口速查

用户启用 node 插件
  -> plugin-management.component.ts
  -> PluginService.checkNodeExecutionPermission()
  -> _ensureNodeExecutionGrant()

启动加载 enabled 插件
  -> PluginService._fireOnReady()
  -> _ensureNodeExecutionGrant() + pingNodeBridge()
  -> PluginRunner.triggerReady()

插件调用 executeNodeScript
  -> PluginBridgeService._executeNodeScript()
  -> preload PLUGIN_EXEC_NODE_SCRIPT
  -> main executeScript()

4. 60 秒判断题

场景：Electron 桌面版，uploaded 插件已在上次会话 Allow 过 nodeExecution；用户这次冷启动后插件 tile 显示 “permission denied”，但设置页仍显示 enabled。你会先查哪两条链路？

参考答案与反馈

第一，查 `PluginService._fireOnReady()` 是否在本会话重新拿到 session grant：持久 consent 只跳过对话框，每次仍要 mint 新 token 并 ping bridge。 第二，查 main 的 grant 是否仍绑定当前 webContents——导航或窗口重建会清 session grant，但不应误删 persisted consent。 若 consent 存在却反复失败，再看 IPC/bridge unavailable 与 `_nodeExecutionDeniedThisSession` 是否记录了本会话拒绝。 细节索引见参考文档“失败症状表”。主要来源：src/app/plugins/plugin.service.ts、electron/plugin-node-executor.ts。 若某个边界仍不清楚，直接向 agent 追问下一跳源码。
