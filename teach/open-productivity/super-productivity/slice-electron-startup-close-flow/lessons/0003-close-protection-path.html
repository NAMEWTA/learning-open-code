关闭前保护：等待项、同步和 finish-day

关闭链路的关键不是“用户点关闭就退出”，而是 main process 先拦截关闭，通知 renderer 处理已注册等待项。同步保护和 finish-day 提醒都通过同一组 before-close IPC 释放或保留窗口。

上一课：启动主路径 相关 L1：op-log 同步 相关模块：Electron 宿主

关闭保护状态机

renderer registers wait ids -> SyncEffects: SYNC_BEFORE_CLOSE_ID -> FinishDayBeforeCloseEffects: finish-day wait id user close / app menu quit / native Cmd+Q -> electron/start-app.ts before-quit may prevent default -> closeWinAndQuit() -> electron/main-window.ts close handler -> minimize-to-tray: hide when allowed and not explicit quit -> ids.length > 0: send NOTIFY_ON_CLOSE(ids) -> ids.length === 0: setIsQuiting(true), close again renderer handles NOTIFY_ON_CLOSE -> sync.effects.ts runs sync and always releases its id -> finish-day effect: quit releases, cancel/finish-day keeps window -> BEFORE_CLOSE_DONE(id) -> main removes id; when empty, closes window for real

等待项由谁负责

等待项 注册位置 释放规则
同步等待项 src/app/imex/sync/sync.effects.ts 同步启用且 ready 后注册；收到关闭通知后停止当前任务、等待一个 macrotask、执行 sync；成功或失败都会 setDone。
finish-day 等待项 src/app/features/finish-day-before-close/finish-day-before-close.effects.ts 功能开关启用后就注册；关闭通知到来后才检查当天已完成任务。没有已完成任务会立即释放；用户选择 quit 才释放，选择 finish-day 会导航到 daily summary 并保留窗口。
main 等待项数组 electron/main-window.ts 只存 ID 和发送 IPC，不理解业务含义；业务是否完成由 renderer 回报。

最小源码锚点

mainWin.webContents.send(IPC.NOTIFY_ON_CLOSE, ids);

this._execBeforeCloseService.schedule(SYNC_BEFORE_CLOSE_ID);
this._execBeforeCloseService.setDone(SYNC_BEFORE_CLOSE_ID);

if (choice === 'finish-day') {
  this._router.navigateByUrl('/active/daily-summary');
}
同步失败后仍释放等待项是刻意设计：不能让用户因为一次 sync 错误永远无法退出。finish-day 的 finish-day 选择则相反，它是用户主动留在应用内继续收尾。
托盘 indicator 菜单里的 Quit 是单独边界：它当前直接调用 quitApp()，先设置 isQuiting=true 再 app.quit()，因此不会进入 NOTIFY_ON_CLOSE 等待项流程。排查托盘退出时不要套用 app menu quit 的保护路径。

60 秒判断题

点击窗口关闭按钮后应用只是隐藏到托盘，没有发出 NOTIFY_ON_CLOSE。这是 bug 吗？

参考答案：不一定。若启用了 minimize-to-tray，且这次不是显式 quit intent，electron/main-window.ts 的 close handler 会先 hide() 并阻止关闭。只有真正 quit 或不走托盘隐藏时，才进入 before-close 等待项流程。
