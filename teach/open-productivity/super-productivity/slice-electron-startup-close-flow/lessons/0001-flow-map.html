Electron 启动到关闭保护链路地图

本课只训练一个判断：看到桌面端启动、窗口创建、关闭前同步或 finish-day 问题时，先把它放回正确的进程边界。Electron main process 创建窗口和 IPC；Angular renderer 完成业务初始化，并通过 preload 暴露的 window.ea 回报就绪或释放关闭等待项。

主要来源：链路参考 后续短课：启动主路径 / 关闭保护 相关 L1：Electron 宿主 建议耗时：12-15 分钟 目标：能画出主路径并判断关闭是否应继续

路径图：不是同步调用栈

OS / 桌面入口 ↓ electron/main.ts ├─ 非 macOS：requestSingleInstanceLock() └─ startApp() ↓ electron/start-app.ts ├─ 注册协议、IPC、本地能力、app 事件 ├─ app ready → createMainWin() └─ before-quit / window-all-closed 兜底退出 ↓ electron/main-window.ts ├─ new BrowserWindow({ preload, contextIsolation, nodeIntegration: false }) ├─ loadURL(dev server 或 dist index.html) ├─ ready-to-show → show() └─ close → 可能 hide、通知 renderer、或真正 quit ↓ Angular renderer ├─ bootstrapApplication(...) ├─ StartupService → APP_READY └─ SyncEffects / FinishDayBeforeCloseEffects → BEFORE_CLOSE_DONE

第一层入口很短：electron/main.ts 在非 macOS 上先拿单实例锁，拿不到就退出，拿到才进入 startApp()。 

if (!IS_MAC) {
  const isLockObtained = app.requestSingleInstanceLock();
  if (!isLockObtained) {
    process.exit(0);
  } else {
    startApp();
  }
} else {
  startApp();
}
关键边界：startApp() 不会直接调用 Angular 的 bootstrapApplication()。它创建 BrowserWindow 并加载 URL；Angular 是窗口里加载的 renderer 应用，完成初始化后再用 IPC 发回 APP_READY。

三个关键判断

问题 先看哪里 判断结果
窗口为什么还没出现？ electron/start-app.ts app ready 之后才调用 createMainWin()；GPU、userData、托盘、IPC 都在这之前或同一阶段注册。
页面出现是否等于业务可用？ electron/main-window.ts ready-to-show 只说明窗口可显示；APP_READY 才是 Angular 初始化后给 main 的业务就绪信号。
点击关闭会不会退出？ electron/main-window.ts 如果 minimize-to-tray 生效，会先隐藏；如果有 before-close 等待项，会发 NOTIFY_ON_CLOSE；等待项清空后才继续关闭。

关闭前保护的正确读法

main process 不知道“同步完成了吗”或“用户要不要 finish-day”的业务答案。它只维护一个等待项数组：renderer 注册 ID，关闭时 main 发通知，renderer 完成自己的业务后回报 BEFORE_CLOSE_DONE。这让同步保护和 finish-day 提醒可以并列存在。 

ipcMain.on(IPC.REGISTER_BEFORE_CLOSE, (ev, { id }) => {
  ids.push(id);
});

ipcMain.on(IPC.BEFORE_CLOSE_DONE, (ev, { id }) => {
  ids = ids.filter((idIn) => idIn !== id);
  if (ids.length === 0) {
    destroyTaskWidget();
    mainWin.close();
  }
});

mainWin.on('close', (event) => {
  if (!getIsQuiting()) {
    event.preventDefault();
    ids.length > 0 ? mainWin.webContents.send(IPC.NOTIFY_ON_CLOSE, ids) : _quitApp();
  }
});

读这段时不要把“关闭事件”理解为“立刻销毁窗口”。第一次 close 常常只是一个拦截点；真正销毁窗口发生在等待项清空、setIsQuiting(true) 之后。

60 秒判断题

场景：用户按 Cmd+Q，同步已启用，今天有已完成任务，finish-day 提醒也启用。renderer 收到关闭通知后，用户在弹窗里选择 finish-day。窗口应该立刻退出吗？

先在脑中排顺序：before-quit、closeWinAndQuit、NOTIFY_ON_CLOSE、同步、finish-day 选择、BEFORE_CLOSE_DONE。

参考答案：不应立刻退出。原生 quit 会先进入 main 的 before-quit，在尚未完成关闭保护时阻止默认退出并转到窗口 close 流程。同步等待项成功或失败后都会调用 BEFORE_CLOSE_DONE；但 finish-day 选择会导航到 /active/daily-summary，不调用 setDone，所以该等待项仍未释放，窗口保持打开。

复习时先打开参考页的“排障表”。相关模块可回看 Angular 应用装配 与 op-log 同步架构。如果你要继续下钻，可以让 agent 带你读同步等待项、IPC handler wrapper 或 BrowserWindow 安全配置。
