Electron 主路径：从进程入口到 APP_READY

本课把启动主路径拆细：Electron 先建立进程、窗口和 IPC 管道；Angular renderer 再自己 bootstrap；业务 ready 只能由 renderer 通过 APP_READY 回报给 main process。

上一课：链路地图 下一课：关闭前保护 相关 L1：Electron 宿主

主路径分层

electron/main.ts -> requestSingleInstanceLock() -> startApp() electron/start-app.ts -> initializeProtocolHandling() -> initIpcInterfaces() -> app ready -> createMainWin() electron/main-window.ts -> new BrowserWindow({ preload, contextIsolation: true, nodeIntegration: false }) -> loadURL(dev server 或 dist index.html) -> ready-to-show: 只代表窗口可显示 src/main.ts -> bootstrapApplication(AppComponent, ...) src/app/core/startup/startup.service.ts -> window.ea.informAboutAppReady() -> electron IPC.APP_READY -> gpu-startup-guard mark launch successful

三个不要混淆的 ready

信号 来源 说明
app ready electron/start-app.ts Electron app 初始化完成，可以创建窗口和初始化部分本地能力。
ready-to-show electron/main-window.ts 窗口内容可显示，不保证 Angular 数据、store、sync 或插件已初始化。
APP_READY src/app/core/startup/startup.service.ts renderer 完成业务启动后主动通知 main；GPU startup guard 也依赖这个信号，而不是 ready-to-show。

最小源码锚点

appIN.on('ready', () => createMainWin());

mainWin.once('ready-to-show', () => {
  mainWin.show();
});

// renderer 业务启动完成后
window.ea.informAboutAppReady();
排查启动白屏时，先看 main 是否建窗，再看 renderer 是否加载，最后才看 APP_READY 是否发回。一个窗口能显示，不代表业务状态已经可用。

60 秒判断题

窗口已经 ready-to-show 并显示了，但下次启动仍被 GPU startup guard 认为上次启动失败。最可能缺的是哪个信号？

参考答案：缺 APP_READY。源码注释明确 GPU guard 的成功标记应在 Angular boot 后的 IPC.APP_READY handler 中发生，不能用 ready-to-show 替代。
