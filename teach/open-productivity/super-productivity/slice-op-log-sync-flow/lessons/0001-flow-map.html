从 NgRx action 到远端 operation 的同步路径

这条链路的核心判断是：本地 action 先变成可回放 operation，写进 SUP_OPS 后才允许上传；远端 operation 下载后必须校验、判冲突、apply，再推进游标。

1. 路径图

src/app/root-store/meta/task-shared.actions.ts
  ↓ meta.isPersistent / entityType / opType
src/app/op-log/capture/operation-capture.meta-reducer.ts
  ↓ pending counter；remote replay 会被 meta.isRemote 过滤
src/app/op-log/capture/operation-log.effects.ts
  ↓ uuidv7 + vectorClock + validateOperationPayload()
src/app/op-log/persistence/operation-log-store.service.ts
  ↓ appendWithVectorClockUpdate() 同事务写 ops + vector_clock
src/app/op-log/sync/operation-write-flush.service.ts
  ↓ 等 pending=0，再拿 operation-log 锁
src/app/imex/sync/sync-wrapper.service.ts
  ↓ downloadRemoteOps() 先于 uploadPendingOps()
src/app/op-log/sync/operation-log-upload.service.ts
src/app/op-log/sync/operation-log-download.service.ts
  ↓ provider: superSyncOps 或 fileSnapshotOps
src/app/op-log/sync/remote-ops-processing.service.ts
  ↓ schema migration / SYNC_IMPORT filter / LWW / state validation
src/app/op-log/apply/operation-converter.util.ts
  ↓ meta.isRemote，避免二次捕获

先记住一个不变量：src/app/op-log/capture/operation-log.effects.ts 写本地 operation，src/app/op-log/sync/remote-ops-processing.service.ts 应用远端 operation。两者不能互相混淆。

2. 三个关键判断

你看到的现象优先判断证据入口
用户刚点完操作就触发同步，远端少了一条变更。上传前是否完成 flush：pending counter 是否归零，operation-log 锁是否重新获取成功。src/app/op-log/sync/operation-write-flush.service.ts
远端 replay 后本地又生成同一条 operation。转换出来的 action 是否带 meta.isRemote，capture 层是否过滤 remote action。src/app/op-log/apply/operation-converter.util.ts
同步结束却不应该显示已同步。session validation latch、pending ops、piggyback ops、permanent rejection 是否仍然存在。src/app/imex/sync/sync-wrapper.service.ts

3. provider 分叉

SuperSync 是真正的 operation API：packages/sync-providers/src/super-sync/super-sync.ts 通过 POST /api/sync/ops 上传，通过 GET /api/sync/ops 下载，并要求 E2EE；payload 加密发生在 src/app/op-log/sync/operation-encryption.service.ts。文件型 provider 则由 src/app/op-log/sync-providers/file-based/file-based-sync-adapter.service.ts 包成 fileSnapshotOps：远端同步文件里有完整 state、recentOps、vectorClock 和 syncVersion，整个文件可压缩/加密。

4. 暂停 60 秒判断

场景：Client A 下载到 Client B 的远端 operation，同时用户在 A 上新建任务。为了不让这个新建任务拿到“没包含远端 operation”的旧 vector clock，系统应该把这个本地 action 立刻写入 op-log，还是先缓冲到远端 apply 后再写？为什么？

参考答案：应该先缓冲。src/app/op-log/capture/operation-capture.meta-reducer.ts 在远端 apply 窗口会把本地 persistent action 放入 deferred buffer；远端 apply 完成并合并远端 vector clock 后，src/app/op-log/sync/process-deferred-actions-flush.util.ts 再驱动 src/app/op-log/capture/operation-log.effects.ts 写这些本地 action。这样新 operation 的 clock 会支配刚应用的远端 operation，避免下一轮被判为 CONCURRENT。
