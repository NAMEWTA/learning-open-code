从“开始追踪”到时间记录的主路径

这条链路的核心判断是：当前追踪任务是本地 UI 状态，累计时间才是可同步事实。用户点击开始后，应用先只设置当前任务；每个 tick 才把 wall-clock delta 写进任务耗时和 work context；停止、5 分钟间隔、后台恢复或 idle 修正会把累计时间 flush 成 persistent action。

注意两条并行事实：任务时间来自 task.timeSpentOnDay[date] 的毫秒累加；work context 时间来自 timeTracking.project/tag[contextId][date] 的 s/e/b/bt 摘要。后者的 s 和 e 在 src/app/features/time-tracking/store/time-tracking.reducer.ts 中按分钟取整，服务于 worklog 起止时间展示。

2. 四个关键判断

发生了什么 立即改状态吗 立即进 op-log 吗 判断依据
点击任务行 play/pause 改 currentTaskId 或清空它 不会 src/app/features/tasks/store/task.actions.ts 的 current task action 没有 persistent meta。
每秒 tick 命中当前任务 会，累加 task.timeSpentOnDay 和 task.timeSpent 不会马上进；先进入 batch accumulator src/app/features/tasks/task.service.ts 先 dispatch Add time spent，再累积等待 flush。
5 分钟到、停止追踪、切任务或后台恢复 本地时间已经在 tick 时更新 会，dispatch [TimeTracking] Sync time spent 和可能的 [TimeTracking] Sync sessions src/app/features/tasks/task.service.ts 的 flush 把累计毫秒和 context 摘要转换为 persistent action。
idle 初始化先扣回误计时间 会，从当前任务扣掉 idleTime，再清空 current task 会，先 dispatch persistent [Task] Remove time spent，随后 current task 变化触发 batch flush src/app/features/idle/store/idle.effects.ts 先调用 TaskService.removeTimeSpent()，再调用 setCurrentId(null)。
远端同步回放 [TimeTracking] Sync time spent 会，只在 remote action 上累加任务时间 不会二次写本地 op-log src/app/features/tasks/store/task.reducer.ts 过滤本地 sync action；src/app/op-log/capture/operation-capture.meta-reducer.ts 过滤 remote action。

3. idle、focus、metric 怎么插进来

Idle 不是另一个计时器，而是修正已经发生的计时。src/app/features/idle/store/idle.effects.ts 检测到空闲后，如果有 current task，会先按 idleTime 从任务中扣回，这一步是带 persistent meta 的 [Task] Remove time spent operation；然后才清空 current task 并打开 dialog，清空动作会让 batch accumulator flush 其中已累计的 duration。最终净效果由扣回 operation 和后续 [TimeTracking] Sync time spent operation 共同形成。用户确认后，空闲时间要么通过任务路径重新记到某个 task，要么作为 break 写到 work context。这样做能避免“离开电脑的时间既算任务又算休息”。

Focus mode 与 tracking 是双向同步。src/app/features/focus-mode/store/focus-mode.effects.ts 会在追踪开始时恢复或自动启动 session，在追踪停止时 pause session；session pause/resume/start/break 也可能反向清空或恢复 current task。完成 focus session 后，src/app/features/metric/metric.service.ts 会记录 focus session duration，src/app/features/metric/store/metric.actions.ts 让它作为 METRIC operation 进入 op-log。

4. 暂停 60 秒判断

场景：用户追踪任务 A 30 秒后离开电脑 4 分钟；idle dialog 弹出，用户选择“把这 4 分钟记为 break”，随后回到任务 A。请暂停 60 秒，判断下面三件事：

  1. 离开前那 30 秒最后会不会进 op-log？
  2. 离开电脑的 4 分钟会不会留在任务 A 的 timeSpent 里？
  3. focus session 如果当时在运行，会发生什么方向的联动？
参考答案：1. 会。tick 已经更新本地任务时间，idle 初始化先生成 persistent [Task] Remove time spent 扣回误计 idleTime；随后清空 current task 触发 batch flush，生成 [TimeTracking] Sync time spent。2. 不应该留在任务 A：idle 初始化会从当前任务扣回 idleTime，用户选 break 后通过 work context 的 break 路径归属。3. src/app/features/focus-mode/store/focus-mode.effects.ts 监听 openIdleDialog 并 pause focus session；如果用户把 idle 归为 break，idle 结果路径会完成或展示 focus overlay。
