状态流

setCurrentTask / unsetCurrentTask
  → tasks.currentTaskId                 本地 UI 状态，不持久化

tick(duration, date)
  → [TimeTracking] Add time spent        本地 action，非 persistent
  → task.timeSpentOnDay[date] += duration
  → task.timeSpent = sum(timeSpentOnDay)
  → subtask delta 可增量更新 parent task
  → timeTracking.project/tag context     s/e 按分钟更新
  → BatchedTimeSyncAccumulator           暂存 TASK duration
  → _unsyncedContexts                    暂存 PROJECT/TAG/TODAY_TAG context

idle 初始化
  → [Task] Remove time spent              persistent TASK update，扣回 idleTime
  → setCurrentTask(null)                  本地 current task 改变，触发 batch flush

flush: 5 分钟 / currentTaskId 改变 / 显式后台恢复
  → [TimeTracking] Sync time spent       persistent TASK update
  → [TimeTracking] Sync sessions         persistent TIME_TRACKING update
  → op-log capture
  → SUP_OPS.ops + SUP_OPS.vector_clock
本地 `syncTimeSpent` reducer 是 no-op，因为本地任务时间已经由 Add time spent 写过；远端回放时带 `meta.isRemote`，才会在 reducer 中累加 duration。

Idle 与 Focus 交互

场景 主文件 行为 容易误判的点
检测到 idle src/app/features/idle/store/idle.effects.ts 满足阈值后 dispatch idle trigger；如果配置要求 current task 且当前无任务，则 reset idle。 idleTime 来自系统空闲 API，但 dialog 打开后显示值由内部 poll 维护。
idle 初始化 src/app/features/idle/store/idle.effects.ts 有 current task 时先 dispatch persistent [Task] Remove time spent 扣回 idleTime，再清空 current task，并打开 idle dialog。 这是避免重复计时的核心，不是 dialog 结束后才扣；清空 current task 还会触发 batch flush，flush 的是 accumulator 中的累计 duration，净效果由这笔扣回 operation 和后续同步 operation 共同形成。
idle 结果记任务 src/app/features/idle/store/idle.effects.ts 已有任务用 `TaskService.addTimeSpentAndSync`；新任务创建时直接带 `timeSpent` 和 `timeSpentOnDay`。 已有任务路径会同步；新任务路径依赖任务创建 action 的持久化。
idle 结果记 break src/app/features/idle/store/idle.effects.ts 通过 work context break time 归属，focus session 可能 complete 并显示 overlay。 break 时间不是任务 `timeSpent`。
tracking start → focus src/app/features/focus-mode/store/focus-mode.effects.ts 追踪开始时，work session 可 unpause；break 可被 skip；也可按配置静默 auto-start focus session。 这一层会跳过 remote ops，避免同步回放触发 UI 联动。
tracking stop → focus src/app/features/focus-mode/store/focus-mode.effects.ts current task 从有到无时，如果 focus/break timer 正在运行，则 pause focus session 并保存 paused task。 使用 pairwise 保留前一个 task id。
focus/break → tracking src/app/features/focus-mode/store/focus-mode.effects.ts session pause 可 unset current task；resume/start/break complete/skip 可恢复 paused task。 break 是否暂停 tracking 取决于 focus config。
focus 完成 → metric src/app/features/focus-mode/store/focus-mode.effects.ts
src/app/features/metric/metric.service.ts		 complete focus session 后记录 duration，低于 1000ms 的 session 被忽略。 metric 记录的是 focus session duration，不是任务 `timeSpent`。

Persistence / op-log 速查

Action 或数据 entityType payload 特点 capture 说明
[TimeTracking] Sync time spent TASK { taskId, date, duration } src/app/op-log/capture/operation-capture.service.ts 明确按 action type 提取 entityChanges。
[Task] Remove time spent TASK { id, date, duration } src/app/features/tasks/store/task.actions.ts 带 persistent meta；idle 初始化扣回误计时间时会作为普通 TASK Update operation 被 capture。
[TimeTracking] Sync sessions TIME_TRACKING { contextType, contextId, date, data } 用于项目/标签维度 `s/e/b/bt` 同步。
[TimeTracking] Update Work Context Data TIME_TRACKING { ctx, date, updates } 用于手动 worklog 编辑等更新。
[Metric] Log Focus Session METRIC { day, duration } src/app/op-log/validation/validate-operation-payload.ts 对 metric focus session 有专门 payload 形状。
currentTaskId 无 本地 store 字段 不持久化，不跨设备同步。 
persistent action
  → src/app/op-log/capture/operation-capture.meta-reducer.ts
      pending counter + remote/deferred guard
  → src/app/op-log/capture/operation-log.effects.ts
      Operation payload + vector clock + validation
  → src/app/op-log/persistence/operation-log-store.service.ts
      SUP_OPS.ops add + SUP_OPS.vector_clock put in one transaction
  → src/app/op-log/sync/operation-write-flush.service.ts
      sync 前等待 pending counter 归零并重新取得写锁

故障排查

症状 优先检查 判断问题 后续深挖
任务时间本地增加，但另一设备没有增加 src/app/features/tasks/task.service.ts
src/app/op-log/capture/operation-log.effects.ts		 tick 写本地成功，但 batch flush 或 persistent capture 失败。 检查 `syncTimeSpent` 是否 dispatch、op-log pending counter 是否卡住。
同步刚触发时少上传一笔最新时间 src/app/op-log/sync/operation-write-flush.service.ts
src/app/op-log/sync/operation-log-sync.service.ts		 上传前必须等待 operation 写入完成。 看 `flushPendingWrites()` 是否 timeout，或写锁是否被长期占用。
idle 后任务时间重复计算 src/app/features/idle/store/idle.effects.ts idle 初始化应先扣回 current task 的 idleTime，再由 dialog 结果重新归属。 检查 `removeTimeSpent`、`setCurrentId(null)`、`addTimeSpentAndSync` 顺序。
focus break 开始后任务仍在计时 src/app/features/focus-mode/store/focus-mode.effects.ts 是否配置了 break 期间暂停 tracking；effect 是否 dispatch `unsetCurrentTask`。 重点查 Pomodoro `autoStartBreakOnSessionComplete` 与 Flowtime `autoStartFlowtimeBreakOnSessionEnd`。
focus session 完成但 metrics 没有记录 src/app/features/focus-mode/store/focus-mode.effects.ts
src/app/features/metric/metric.service.ts		 duration 是否大于 1000ms，`completeFocusSession` 是否真正触发。 检查 metric reducer 的 `focusSessions` 追加和 persistent action capture。
父任务耗时和子任务耗时不一致 src/app/features/tasks/store/task.reducer.util.ts 子任务增量更新父任务时日期 delta 是否正确。 对比 `timeSpentOnDay` 旧值、新值、父任务当前值。
移动端后台恢复后时间跳变 src/app/features/android/store/android-foreground-tracking.effects.ts
src/app/features/ios/store/ios-background-tracking.effects.ts		 wake-up tick、reset tracking anchor、flush accumulated time 的顺序是否正确。 原生 foreground service 的权威计时与 Web 层 tick 是否重复入账。
