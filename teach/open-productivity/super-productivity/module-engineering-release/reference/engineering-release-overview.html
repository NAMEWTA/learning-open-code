测试、构建、发布与多平台工程化参考

模块职责

职责一句话：把同一套 Angular renderer 和 workspace 包，按 Web/PWA、Electron 桌面、Capacitor Android/iOS、内置插件和商店发布的不同要求，组织成可测试、可打包、可发布的产物链。

它承接 L0 总览里的“测试/构建/发布”层；Electron L1、Mobile/PWA L1 与 Plugin L1 分别负责桌面、移动/PWA 和插件运行时边界。当前主题只说明它们在工程化链路中的构建接点。

模块骨架

super-productivity/ ├── package.json ← npm scripts、workspaces、Electron main、发布 provider ├── angular.json ← Angular build/test/lint target 与输出目录 ├── electron-builder.yaml ← 桌面包、asar 文件边界、平台目标、afterPack ├── capacitor.config.ts ← Capacitor appId、webDir、Android/iOS 插件边界 ├── e2e/ ← Playwright 配置、POM、fixtures、截图/视频脚本 ├── tools/ ← env、version、release notes、packaging hooks ├── scripts/ ← wait-for-app/WebDAV/SuperSync 与迁移辅助脚本 ├── packages/ ← sync 包、plugin-api、vite-plugin、plugin-dev 构建 ├── eslint-local-rules/ ← NgRx/effects 架构 lint rule 与 RuleTester spec ├── build/ ← electron-builder 平台资源、entitlements、PKGBUILD ├── fastlane/ ← App Store lane 与 Android metadata ├── snap/ ← Snap install hook └── .github/ ├── actions/ ← 复用的 Electron/E2E 环境 setup └── workflows/ ← PR、smoke、release、商店发布分流

入口清单

入口职责读码判断
package.json所有工程化链路的第一跳：开发、lint、unit、E2E、package tests、frontend build、Electron build、Capacitor sync、release notes、workspace/plugin build。先按 script 名分类，不要直接把 build 理解成全平台 release。
angular.jsonsp2 应用构建入口为 src/main.ts，assets 包含 src/assets、src/manifest.json、src/static；Electron 产物输出到 .tmp/angular-dist，Web/Capacitor 输出到 dist/browser。区分 production、productionWeb、stage、stageWeb 的输出和 service worker。
electron-builder.yaml定义 appId、artifactName、files、排除项、.tmp/app-builds、Windows/Linux/macOS 目标、Snap 目标、Flatpak 配置段和 afterPack；当前 Linux targets 是 AppImage、deb、snap、rpm，不包含 Flatpak。桌面包缺文件、asar 过大、平台格式或签名问题先看这里。
capacitor.config.ts移动端 appId/name、webDir 指向 dist/browser、Android includePlugins、iOS WebView 行为。移动端构建先有 Web 产物，再 npx cap sync，最后进 Gradle/Xcode。
packages/build-packages.js构建 sync/plugin 相关包，并把内置插件复制到 src/assets/bundled-plugins。插件资产缺失、shared 包 dist 缺失、plugin build 顺序问题先看这里。
e2e/playwright.config.ts定义 Playwright 测试目录、global setup、workers、reporter、baseURL、webServer、artifacts。E2E 是否启动 4242 服务、是否使用 Docker URL、失败产物在哪里，都在这里。
eslint.config.js
eslint-local-rules/		TS/HTML lint、package boundary 限制、effects 架构规则、no-console 和本地 RuleTester spec。这些是架构回归守卫，不只是格式检查。
tools/tools/load-env.js 写运行时 env 生成文件，tools/git-version.js 写版本生成文件，tools/release-notes.js 写 GitHub/Play Store notes，tools/afterPack.js 修补 Linux 包。凡是“构建前写文件”或“发布前生成元信息”，优先查工具脚本。
build/Electron 平台资源、MAS yaml、entitlements、AppX 图标、Linux PKGBUILD 模板、release notes。这是打包资源/元数据目录，不是 Angular 构建输出。
fastlane/iOS/macOS App Store Connect lane、Android metadata、changelog 文本。商店上传和“What's New”文本在这里，不等同于二进制构建。
snap/Snap install hook 提示连接 idle/file sync 所需接口。Snap 运行时权限问题不只看 electron-builder 的 snap 段。

Workflow触发实际做什么判断信号
.github/workflows/ci.ymlPR、手动、release publishedlockfile registry、dependency review、lint、unit、Electron node tests、frontend E2E build、E2E、Lighthouse。PR 主守门，不发布产物。
.github/workflows/plugin-tests.ymlplugin-api、vite-plugin、plugin-dev 相关 PR检测变更插件，构建 shared packages，逐插件安装依赖并运行测试。插件包边界专用守门。

.github/workflows/build.ymlmaster/release/test 分支与 v* tag创建 draft release；Linux/macOS/Windows 桌面包；tag 时上传到 GitHub Release，Windows 还走 SignPath。桌面 release 主线。
.github/workflows/build-android.ymlmaster/release/test 分支、v* tag、手动Node/Java/Android SDK、dev versionCode、dist:android:prod、上传 APK、tag 时 GitHub Release 和 Play internal。Android 二进制与 Play Store 文案交汇点。
.github/workflows/auto-publish-google-play-on-release.ymlrelease published、手动把 build-android 上传到 internal track 的 release promote 到 Play production。Android production 发布的最后一跳，不重新构建 APK。
.github/workflows/build-ios.ymlv* tag、手动Xcode、证书/profile、版本号、Web build、Capacitor iOS sync、archive/export IPA、Fastlane 上传。iOS App Store 主线。
.github/workflows/build-update-web-app-on-release.ymlrelease published、手动lint/test、buildFrontend:prodWeb、把 dist/browser 通过 SSH 部署到 Web 服务器。Web/PWA 发布，不涉及 Electron 包。
.github/workflows/build-publish-to-snap-on-release.ymlrelease published、手动从 GitHub Release 下载 .snap，用 Canonical snapcraft container 上传 edge/stable。发布已有 snap asset，不重新构建。
.github/workflows/build-publish-to-aur-on-release.ymlrelease published、手动用 build/linux/PKGBUILD_template 生成 AUR PKGBUILD 并重试发布。AUR 元包更新。

.github/workflows/build-create-windows-store-on-release.ymlv* tag/test 分支、手动加载 Store 专用 electron-builder yaml，构建 appx artifact，并收集 Partner Center 需要的 release notes。Microsoft Store artifact，上传仍偏手动。

阅读示例 A：追 npm run dist

  1. 从 package.json 找到 dist：先跑 buildAllElectron:prod，再调用 electron-builder。
  2. buildAllElectron:prod 会执行 env、preCheck、buildFrontend:prod:es6、electron:build。
  3. preCheck 合并 lint、unit、i18n test、E2E；这比 build 的 noTests 分支更重。
  4. buildFrontend:prod:es6 通过 angular.json 输出 .tmp/angular-dist；electron:build 生成主进程 JS 与 preload bundle。
  5. electron-builder.yaml 把 Electron 编译产物与 Angular renderer 产物打进 .tmp/app-builds，并在 Linux 下执行 tools/afterPack.js。

阅读示例 B：追 npm run dist:android:prod

  1. package.json 显示它先跑 buildFrontend:prodWeb，输出 dist/browser。
  2. capacitor.config.ts 的 webDir 说明 Capacitor 同步的正是 dist/browser 里的 Web 产物。
  3. sync:android 调 npx cap sync android，随后 assemble:android:prod 进入 Gradle release assemble。
  4. .github/workflows/build-android.yml 负责 CI 侧 Java/SDK/keystore、dev versionCode、APK artifact、GitHub Release 上传与 Play internal track；.github/workflows/auto-publish-google-play-on-release.yml 再把 internal release promote 到 production。
  5. tools/release-notes.js 和 fastlane/metadata/android 处理 Play Store “what's new”，不是 APK 构建本身。
