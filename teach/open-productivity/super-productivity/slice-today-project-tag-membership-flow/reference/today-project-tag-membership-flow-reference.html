Today、Project 与 Tag 归属判定速查

遇到“任务为什么出现在这里”时，按本页顺序定位：先看 action 写侧，再看字段和排序列表，最后看 WorkContext 与 MenuTree。

一页结论

Today

成员真相是 task.dueWithTime 或 task.dueDay；TODAY_TAG.taskIds 只保排序。

Project

任务字段有 projectId；Project 视图列表读 project.taskIds 和 project.backlogTaskIds。

普通 Tag

成员真相是 task.tagIds；tag.taskIds 只参与排序和拖拽位置。

MenuTree

只组织 Project/Tag 导航树，水合实体并排序；它不决定任何任务是否属于某个视图。

判定算法

给定 task 与当前 state：
1. Today:
   if task.dueWithTime 存在:
     使用 isTodayWithOffset(dueWithTime, todayStr, offset)
   else:
     判断 task.dueDay === todayStr
   使用 TODAY_TAG.taskIds 仅恢复顺序，过滤陈旧 id。

2. 普通 Tag:
   判断 task.tagIds.includes(tagId)
   使用 tag.taskIds 排序；缺失但属于该 tag 的任务追加到末尾。

3. Project:
   写侧保持 task.projectId 与 project.taskIds/backlogTaskIds 同步。
   active Project context 直接返回 project.taskIds/backlogTaskIds。

4. MenuTree:
   Project/Tag 实体先被业务 selector 过滤，再由 MenuTree 水合排序。
   导航树不修改 Today、Project 或 Tag 的任务成员。

写侧动作到字段/列表

动作写入侧影响不要误解为
TaskSharedActions.addTasktask-shared-crud.reducer.ts新增 task；按 projectId 写 Project 列表；按普通 tagIds 写 Tag 排序；若 dueDay === todayStr 写 Today 排序。不是把 TODAY 塞进 task.tagIds。
TaskSharedActions.updateTasktask-shared-crud.reducer.tstagIds 变化会更新普通 Tag 排序；dueDay 变化会更新 planner days 与 Today 排序。不是单纯 entity patch；它有跨 store side effects。
TaskSharedActions.scheduleTaskWithTimetask-shared-scheduling.reducer.ts写 dueWithTime、清 dueDay、可写 remindAt；若时间落在今天则更新 Today 排序。不是 date-only Today 计划。
TaskSharedActions.planTasksForTodaytask-shared-scheduling.reducer.ts写 dueDay = today，清提醒，可按需清 dueWithTime，并把任务加入 TODAY_TAG.taskIds。不是普通 tag 添加。
TaskSharedActions.moveToOtherProjectproject-shared.reducer.ts从旧 Project 的主列表/backlog 移除；加入目标 Project 主列表；更新任务和子任务 projectId。不是只改任务字段。
deleteTag / deleteTagstag-shared.reducer.ts从所有任务移除被删 tag，清理孤儿任务、repeat cfg、time tracking 与 issue provider 默认 tag。不是只删除 Tag entity。

常见混淆点

不要把四个概念混成一个：TODAY_TAG.taskIds 是 Today 排序；task.tagIds 是普通标签成员；project.taskIds 是 Project 主列表；dueDay/dueWithTime 是日期计划字段。

  • Today 虚拟标签不是普通 Tag；`TODAY` 出现在 task.tagIds 属于 legacy/错误数据，需要清理。
  • 普通 Tag 成员可以在 tag.taskIds 缺失时自愈，因为 selector 会从 task.tagIds 重算。
  • Project 列表不是纯 selector 推导；写侧会维护 taskIds 和 backlogTaskIds。
  • MenuTree 只影响导航结构。一个 Project 隐藏出菜单，并不代表它的任务失去 Project 归属。
  • resolveDisplayTagIds 是展示 fallback，不能拿来判断同步、op-log 或真实成员关系。

排查顺序

  1. 先看 action：是 addTask、updateTask、scheduleTaskWithTime、planTasksForToday、moveToOtherProject 还是 tag/project delete。
  2. 再看写侧 meta-reducer：确认字段和排序列表有没有同时维护。
  3. 再看实体状态：任务字段、Project 列表、Tag 列表是否一致。
  4. 最后看 selector：active WorkContext 是 Project 还是 Tag；若是 Tag，再区分 Today 和普通 Tag。
  5. 若只是侧边栏没有入口，转去 MenuTree、hidden/archive 过滤和 tree order，而不是查任务成员。
