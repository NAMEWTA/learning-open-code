主流程分解

1. 自动导入到 backlog

  1. `poll-to-backlog` 等待初始同步完成，按 `defaultProjectId`、`isAutoAddToBacklog`、`pollingMode`、agenda view、poll interval 筛 provider。
  2. `IssueService.checkAndImportNewIssuesToBacklogForProject` 用 `TaskService.getAllIssueIdsForProviderEverywhere` 收集当前任务和归档任务里的 issue id。
  3. provider 实现 `getNewIssuesToAddToBacklog`，例如 Jira 调 `findAutoImportIssues$`，GitLab 调 `getProjectIssues$`，OpenProject 拉最近 100 个 work package，Plainspace 拉 `getMyTasks$`。
  4. 候选 issue 经过二次过滤后进入 `addTaskFromIssue`，由 provider `getAddTaskData` 产出 `IssueTask` 字段。
  5. `TaskService.add` 派发 `TaskSharedActions.addTask`；有 `dueWithTime` 时走 `addAndSchedule`；有关联父 issue 时可能走 `addSubTaskTo`。

2. 远端 issue 更新已有任务

  1. `poll-issue-updates` 选出已有 issue task：普通 provider 默认限当前上下文，`always` provider 全局，ICAL 与 `plugin:*` provider 走 calendar issue task 全局选择器；迁移后的 `GITHUB/CLICKUP/GITEA/LINEAR/TRELLO/AZURE_DEVOPS` key 不是 `plugin:*`，仍按普通或 `always` 规则。
  2. `IssueService.refreshIssueTasks` 按 issue provider key 分组，调用 provider 的 `getFreshDataForIssueTasks`。
  3. 基础实现用 `issueLastUpdated` 判断远端是否更新，并保护用户本地 schedule：默认删除 `dueDay/dueWithTime`，Calendar/Plainspace 等可覆盖。
  4. 有更新时调用 `TaskService.update`，落到 `TaskSharedActions.updateTask`；同时通过 `ISSUE_REFRESH_MAP` 通知正在显示的 issue 详情。
  5. 支持 `getRemovedRemoteTasks` 的 provider 可安全删除无本地投入的孤儿任务，Plainspace 有“全量消失时跳过”的防误删门。

3. 本地任务写回 provider

  1. `IssueTwoWaySyncEffects` 监听 update、short syntax、schedule、planner、tag、delete、addTask 等本地 action。
  2. 只有 `isDone`、`title`、`notes`、`dueWithTime`、`dueDay`、`timeEstimate`、`tagIds` 等同步字段会进入 push 计算。
  3. 含 `issueLastSyncedValues` 或 `issueWasUpdated` 的 poll/bookkeeping 更新会被跳过，避免 pull 后立刻 push 回去。
  4. `computePushDecisions` 先检查方向：`off/pullOnly` 跳过；无 baseline 跳过；fresh provider value 与 baseline 不同则 provider wins；否则 push。
  5. push 成功后只推进实际写过字段的 baseline；若远端还有未拉取变化，则保持 `issueLastUpdated` stale，让下一轮 poll 拉回来。

动作判断表

动作或函数 分类 会创建/更新任务吗 op-log 边界
`IssueProviderActions.addIssueProvider/updateIssueProvider/sortIssueProvidersFirst` provider 配置 不会 显式配置变更带 provider 持久化 meta
`IssueProviderActions.upsertIssueProvider/upsertIssueProviders/updateIssueProviders` 导入/同步 provider 状态 不会 源码注释标明通常不带持久化 meta
`getNewIssuesToAddToBacklog` provider 拉取 不会 无任务 op；只返回候选 issue 数据
`IssueService.addTaskFromIssue` 导入编排 会，可能创建主任务、定时任务或子任务 通过 `TaskService` 进入任务 action
`TaskSharedActions.addTask/updateTask/deleteTask/deleteTasks` 任务状态 会 带 `PersistentActionMeta`，是任务 op-log 的主要边界
`addSubTask` 任务状态 会创建子任务 `src/app/features/tasks/store/task.actions.ts` 中带 `OpType.Create`
`DeletedTaskIssueSidecarService` / `DeletedTagTitlesSidecarService` 本地临时 sidecar 不会 刻意不进 action payload，不持久化，不同步
`adapter.pushChanges/createIssue/deleteIssue` 远端 provider 写入 不直接写任务；成功后可能更新任务同步基线 由监听本地任务 action 的 effect 触发

现象 先查 判断方法
新 issue 没进 backlog src/app/features/issue/store/poll-to-backlog.effects.ts 检查 provider 是否 enabled、`defaultProjectId` 是否匹配、`isAutoAddToBacklog` 是否为真、poll interval 是否大于 0、是否被 agenda view 排除。
重复导入 src/app/features/issue/issue.service.ts
src/app/features/tasks/task.service.ts		 检查 `issueId + issueType + issueProviderId` 是否一致；Calendar 还要看 issue id candidates；Plainspace/Calendar 使用确定性 task id 防跨设备重复。
远端更新没有反映到任务 src/app/features/issue/store/poll-issue-updates.effects.ts
src/app/features/issue/base/base-issue-provider.service.ts		 确认任务被选入轮询范围，provider 是否实现 fresh data，`issueLastUpdated` 是否阻止了更新。
本地修改没写回 provider src/app/features/issue/two-way-sync/issue-two-way-sync.effects.ts
src/app/features/issue/two-way-sync/compute-push-decisions.ts		 看字段是否在 syncable set，adapter 是否存在，方向是否允许 push，是否缺 baseline，fresh provider value 是否已经变化。
