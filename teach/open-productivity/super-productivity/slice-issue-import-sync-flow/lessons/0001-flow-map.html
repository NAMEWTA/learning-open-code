从 provider 候选 issue 到任务与写回

学完这一课，你要能回答一个问题：某段代码是在拉取 provider 状态、创建/更新本地任务，还是把本地任务变化写回 provider？ 完整入口清单见 参考文档。 相关 L1：Issue Provider 与日历集成、 任务领域、 op-log 同步。

路径图

  1. 触发导入轮询。 初始同步完成后，切到项目会筛出启用、绑定当前项目、允许自动加入 backlog、非 agenda view、且有轮询间隔的 provider；`always` 模式走常驻轮询。 src/app/features/issue/store/poll-to-backlog.effects.ts
  2. 只委托，不写任务。 effect 自身 `dispatch: false`，它只调用 `IssueService.checkAndImportNewIssuesToBacklogForProject`。失败时记录 log、弹 snack，并返回 `EMPTY` 保持轮询流继续。
  3. 去重后向 provider 要候选 issue。 `IssueService` 先收集同一 provider 在当前任务和归档任务里的 issue id，再调用 provider 的 `getNewIssuesToAddToBacklog`，最后再次过滤已存在 id。 src/app/features/issue/issue.service.ts
  4. 候选 issue 映射成任务字段。 provider 的 `getAddTaskData` 返回 `IssueTask`：例如 title、issuePoints、dueDay、issueLastUpdated、issueLastSyncedValues。这里还不是 NgRx action。
  5. 创建或更新本地任务。 `addTaskFromIssue` 合并 provider 字段、默认项目/标签/备注、backlog 标记，然后调用 `TaskService.add`、`addAndSchedule` 或 `addSubTaskTo`。这一步才落到任务 action 和 op-log 边界。
  6. 本地变化另走写回链路。 后续用户改标题、完成状态、日期或标签时，双向同步 effect 监听任务 action，计算 push 决策，再经 adapter 写回 provider。 src/app/features/issue/two-way-sync/issue-two-way-sync.effects.ts

三个判断点

看到的动作 任务是否变化 边界判断
provider `getNewIssuesToAddToBacklog` 否 只是从远端取候选 issue；不创建任务，不写任务 op-log。
`IssueService.addTaskFromIssue` 是，可能创建主任务、定时任务或子任务 它把 provider 字段和本地默认值合并后交给 `TaskService`。
`IssueTwoWaySyncEffects._pushChanges$` 可能只更新 sync baseline 先看 baseline 与方向配置；provider 已变化或无 baseline 时不会 push。
`IssueProviderActions.upsertIssueProvider` 否 这是 provider 配置状态，不是任务创建；导入/同步用途不带持久化 meta。
不要混成一步： backlog 轮询负责“发现并导入缺失 issue”；已有 issue task 的远端更新由 `poll-issue-updates` 调 `refreshIssueTasks`；本地任务变化写回 provider 由 `issue-two-way-sync` 监听任务 action。

60 秒判断题

场景：Jira provider 的 backlog poll 返回 `ABC-7`，但当前任务或归档任务中已经有同一 `issueProviderId + issueType + issueId`。 代码应该怎么做？

  1. 创建一个新的 backlog 任务，避免漏掉远端 issue。
  2. 不创建新任务；根据位置提示已存在、恢复归档或移动到当前上下文。
  3. 直接把 Jira issue 写回远端，更新远端状态。
参考答案与反馈

选 2。`addTaskFromIssue` 会先调用重复检测；归档任务会恢复，当前项目 backlog 中的任务只提示已存在，其他列表中的任务可能被移动到当前上下文。provider 拉取不会自动写回远端。
