L2 垂直切片 · 预计 15 分钟内 · 目标：能判断远端 ops 下载后进入哪条 apply 或冲突路径

远端 operation 下载后的分流图

这节只练一个判断：下载到的 ops 不是马上写 NgRx。它们先经过下载过滤、full-state gate、schema migration、clean-slate 过滤和 vector clock 判定，最后才进入 direct apply、LWW 或 destructive full-state 分支。

父级 L1：Operation Log 与同步模块。相关 L2：本地操作到同步的路径图。源码锚点见 参考文档。

1. 先看总分流

入口锚点：src/app/imex/sync/sync-wrapper.service.ts 在 super sync 下载阶段调用 OperationLogSyncService.downloadRemoteOps()，并在用户取消 conflict dialog 时提前返回。

// sync-wrapper.service.ts — download phase entry
const isNeverSyncedAtSyncStart = !(await this._opLogSyncService.hasSyncedOps());

const downloadResult = await this._opLogSyncService.downloadRemoteOps(
  syncCapableProvider,
  {
    forceFromSeq0: isProviderSwitch || undefined,
    isNeverSynced: isNeverSyncedAtSyncStart,
  },
);
this._consecutiveSuperSyncAuthFailures = 0;

if (downloadResult.kind === 'cancelled') {
  return 'HANDLED_ERROR';
}
SyncWrapperService.sync()
  ↓ capture isNeverSynced, then download phase
OperationLogSyncService.downloadRemoteOps()
  ↓ delegates I/O
OperationLogDownloadService.downloadRemoteOps()
  ↓ pagination, dedupe applied ids, decrypt → newOps
OperationLogSyncService (orchestrator branches)
  ├─ fileSnapshotOps + snapshotState → hydrate snapshot，不创建 SYNC_IMPORT
  ├─ incoming SYNC_IMPORT/BACKUP_IMPORT → conflict gate 先问是否丢本地 pending
  └─ 普通 newOps → RemoteOpsProcessingService.processRemoteOps()
       ↓ schema migration
       ↓ SyncImportFilterService clean-slate 过滤
       ├─ SYNC_IMPORT/BACKUP_IMPORT 在 batch 内 → 跳过普通冲突检测，直接 full-state apply
       ├─ skipConflictDetection=true → force download 后直接 apply
       ├─ CONCURRENT → ConflictResolutionService.autoResolveConflictsLWW()
       └─ 无冲突 → applyNonConflictingOps()
不要把四类路径合并：普通 remote ops 走 vector clock；incoming full-state 先过 gate；`SYNC_IMPORT` / `BACKUP_IMPORT` 在 `processRemoteOps()` 内直接 full-state apply；`REPAIR` 属于 full-state 过滤和 store 清理分类，但不是这个 direct full-state 分支的触发条件。

3. apply handler 的入口

普通无冲突 ops 会进入 src/app/op-log/sync/remote-ops-processing.service.ts 的 applyNonConflictingOps()。它调用 packages/sync-core/src/remote-apply.ts：先 appendBatchSkipDuplicates(..., 'remote', { pendingApply: true })，再让 src/app/op-log/apply/operation-applier.service.ts 批量 replay。

replay 不是逐个 dispatch 原始 action，而是发 bulkApplyOperations()。随后 src/app/op-log/apply/bulk-hydration.meta-reducer.ts 逐 op 调用 src/app/op-log/apply/operation-converter.util.ts，转换出的 action 带 meta.isRemote，防止远端 replay 又被 capture 成本地 op。LWW 生成的 [TASK] LWW Update 最终由 src/app/root-store/meta/task-shared-meta-reducers/lww-update.meta-reducer.ts 替换或重建实体。

4. 60 秒自测

场景：Client A 下载到 Client B 的 `SYNC_IMPORT`，同时 A 有一个未上传的用户新建任务。应该先把 `SYNC_IMPORT` 交给 `processRemoteOps()` 直接应用，还是先跑 conflict gate？如果用户选择使用远端数据，哪些本地 op-log 写入会发生？

参考答案：先跑 SyncImportConflictGateService.checkIncomingFullStateConflict()，因为 incoming full-state 会丢弃 meaningful pending。若用户选择使用远端数据，forceDownloadRemoteState() 会先保存可撤销备份，清理本地 full-state ops，标记 unsynced ops 为 rejected，重置 lastServerSeq 后从 0 下载；随后用 processRemoteOps(..., { skipConflictDetection: true }) 应用远端 ops。普通 remote ops 不走这个 destructive 分支。
