  → TimeBlockSyncEffects.upsertOnTaskChange$
      ├─ 检查 enabled plugin + isAutoTimeBlock + definition.timeBlock
      ├─ per-task debounce COALESCE_MS + groupBy 合并突发 action
      ├─ per-task 队列串行；跨任务 MAX_PARALLEL_TIME_BLOCK_HTTP = 3

  → TimeBlockSyncEffects delete 效应 → timeBlock.deleteEvent

  → backfillOnAutoTimeBlockEnabled$：syncRangeWeeks 内已有 dueWithTime 任务批量 upsert
字段映射到 time block说明

常见判断清单

现象先看判断规则
Schedule 没有某会议块 calendarEvents$ 过滤链 可能 regex 过滤、已导入为任务、被 skip/hidden，或 provider 未启用。
第二台设备重复导入同一事件 generate-calendar-task-id.ts 检查是否在 sync window 内提前 auto-import；确定性 id 应收敛为同一 task。
排程后 Google 日历无块 time-block-sync.effects.ts 确认 isAutoTimeBlock、插件 timeBlock 定义、snack TIME_BLOCK_ERROR；看队列是否被 delete 覆盖。
改标题后日历标题旧 upsertOnTaskChange$ updateTask 相关字段会 upsertIfScheduled；等待 1s 合并窗口结束。
拖会议块无效 canMoveEvent iCal 与 reference calendar 不可移动；需插件 updateIssue。
开启 auto time block 后旧任务没块 backfillOnAutoTimeBlockEnabled$ 仅 backfill syncRangeWeeks 内未来 scheduled task；每 provider 只执行一次。

与相邻链路的边界

  • vs issue 双向同步：Jira/GitLab 等字段写回走 IssueTwoWaySyncEffects；calendar time block 是独立 HTTP 副作用，dispatch: false，不写 task op-log。
  • vs Planner/Schedule 投影：calendar 事件进入视图后仍是外部数据；只有导入才产生 Task；time block 则从已有 Task.dueWithTime 写出。
  • vs 插件 agenda 读：同一插件可能同时提供 agenda 读（useAgendaView）与 time block 写（isAutoTimeBlock）；二者 API 与缓存路径不同。
