Operation Log、本地持久化与同步架构

一句话：这个模块把 NgRx action 记录成可回放的 operation，先写入本地 SUP_OPS，再通过统一 provider 口同步到文件型远端或 SuperSync 服务端。

1. 它接在 L0 地图的哪一层

L0 项目地图把 NgRx 视为运行时状态，把 SUP_OPS 和 operation log 视为可回放持久化来源。本课只看边界：src/app 负责 Angular/NgRx 接线，@sp/sync-core 放框架无关算法，@sp/sync-providers 放 provider 协议，SuperSync server 是远端 append-only log。

NgRx action
  ↓ capture
Operation
  ↓ local durable write
SUP_OPS: ops + state_cache + vector_clock
  ↓ orchestration
OperationSyncCapable provider
  ├─ fileSnapshotOps: 例如 Dropbox / WebDAV / Nextcloud / OneDrive / LocalFile
  └─ superSyncOps: Fastify / Prisma / PostgreSQL

2. 先认四个入口

src/app/root-store/meta/task-shared.actions.ts 给 action 加 meta.isPersistent、entityType、opType。src/app/op-log/capture/operation-log.effects.ts 监听这些本地 action，创建 operation、校验 payload、写入 store，并触发即时上传。src/app/imex/sync/sync-wrapper.service.ts 是用户触发或周期触发的一轮 sync 入口：它取得 OperationSyncCapable provider 后，先调用 downloadRemoteOps()，再调用 uploadPendingOps()。src/app/op-log/sync/operation-log-sync.service.ts 则负责 op-log 层的下载/上传方法，以及 fresh-client gate、piggyback、force upload/download 等保护逻辑。

完整入口清单见 模块参考：那里包括 OperationCaptureService、persistence adapters、provider factory、sync-core public exports 与 SuperSync API。

3. 阅读示例：更新任务如何进日志

updateTask: (...) => ({
  ...taskProps,
  meta: {
    isPersistent: true,
    entityType: 'TASK',
    entityId: taskProps.task.id,
    opType: OpType.Update,
  },
})

这个 meta 是捕获开关。meta-reducer 只登记非远端 persistent action；effect 再顺序处理，避免 remote replay 再次写入日志。

const op: Operation = {
  id: uuidv7(),
  actionType: action.type as ActionType,
  opType,
  entityType: action.meta.entityType,
  entityId,
  payload: multiEntityPayload,
  vectorClock: newClock,
};
await this.opLogStore.appendWithVectorClockUpdate(op, 'local');
this.immediateUploadService.trigger();

写入点是 appendWithVectorClockUpdate()：operation 和 vector clock 在同一事务中更新。这样 UI 的乐观状态和磁盘日志即使在离线时也有可回放来源。

await this.writeFlushService.flushPendingWrites();
const result = await this.uploadService.uploadPendingOps(syncProvider, ...);

同步前的 flush 是关键保护：如果用户刚点完任务就立刻同步，编排器必须等 action 对应的 operation 落盘后再上传，否则远端会少一条本地变更。

4. 检索练习

暂停 60 秒，凭记忆回答：为什么 replay 远端 operation 时不会再次生成新的本地 operation？

参考答案：远端 operation 会先转换回带 meta.isRemote: true 的 persistent action；capture meta-reducer 和 persist effect 都会过滤远端 action，所以 reducer 能应用状态变化，但 op-log 不会二次追加同一意图。
