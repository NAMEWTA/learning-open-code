Operation Log 与同步模块参考

模块职责

一句话：把本地用户意图从 NgRx action 捕获为不可变 operation，以 SUP_OPS 作为可回放持久化来源，并通过统一 provider 边界同步到文件型远端或 SuperSync 服务端。

它位于 L0 总览 中的“NgRx Store + op-log + packages”层：运行时状态仍在 NgRx，长期一致性来源转到 operation log。

内部分层结构

NgRx action capture
  ├─ PersistentActionMeta 标记用户意图
  ├─ operationCaptureMetaReducer 登记本地 action
  └─ OperationLogEffects 顺序写入

operation conversion / apply
  ├─ action → Operation：uuidv7、opType、entityType、payload、vectorClock
  ├─ RemoteOpsProcessingService：迁移、SYNC_IMPORT 过滤、冲突检测、校验
  ├─ OperationApplierService：批量 replay 到 NgRx store
  └─ convertOpToAction()：Operation → action，带 meta.isRemote 防止二次记录

IndexedDB / SQLite persistence
  ├─ SUP_OPS.ops：append-only operation log
  ├─ state_cache：快照与 compaction 基准
  ├─ vector_clock：本地因果状态
  └─ OpLogDbAdapter：后端无关事务口

sync orchestration
  ├─ SyncWrapperService：一轮 sync 入口；获取 provider，先 downloadRemoteOps() 再 uploadPendingOps()
  ├─ OperationLogSyncService：op-log 层 download/upload、fresh-client gate、piggyback、force upload/download
  ├─ OperationLogUploadService / DownloadService：provider I/O
  └─ RemoteOpsProcessingService：应用远端 ops、冲突与校验

provider / server boundary
  ├─ fileSnapshotOps：例如 Dropbox / WebDAV / Nextcloud / OneDrive / LocalFile 的 sync-data.json
  └─ superSyncOps：Fastify API + PostgreSQL operations 表

远端 apply 入口

下载到远端 operation，或上传时 piggyback 收到别的客户端 operation 后，阅读入口从 provider I/O 转到 apply 管线：

  • src/app/op-log/sync/remote-ops-processing.service.ts：RemoteOpsProcessingService.processRemoteOps() 先做 schema migration、SYNC_IMPORT 过滤、冲突检测，再决定直接应用或进入冲突解决。
  • src/app/op-log/apply/operation-applier.service.ts：OperationApplierService.applyOperations() 使用 replayOperationBatch() 和 bulkApplyOperations() 批量 dispatch 到 NgRx store。
  • src/app/op-log/apply/operation-converter.util.ts：convertOpToAction() 把 operation 转回 action，并设置 meta.isRemote: true，避免 replay 又被 capture 成新 operation。

阅读路径：从 action 到同步队列

  1. 在 src/app/root-store/meta/task-shared.actions.ts 找一个 action，例如 TaskSharedActions.updateTask，确认它带 isPersistent: true、entityType: 'TASK'、opType: OpType.Update。
  2. 到 src/app/op-log/capture/operation-capture.meta-reducer.ts 看本地 action 如何调用 OperationCaptureService.incrementPending()；注意 remote action 和 replay 窗口会被过滤或延迟。
  3. 到 src/app/op-log/capture/operation-log.effects.ts 看 writeOperation() 如何组装 Operation，然后调用 appendWithVectorClockUpdate(op, 'local')。
  4. 到 src/app/op-log/persistence/operation-log-store.service.ts 看 appendWithVectorClockUpdate() 如何在同一事务中写 ops 和 vector_clock。
  5. 到 src/app/imex/sync/sync-wrapper.service.ts 看全局 sync 周期如何先下载再上传，并通过 WrappedProviderService 获取 OperationSyncCapable provider。
  6. 到 src/app/op-log/sync/operation-log-sync.service.ts 看上传前的 flushPendingWrites()，确认刚捕获的 operation 已经落盘。
  7. 如果要追远端 operation 如何落到本地状态，按 src/app/op-log/sync/remote-ops-processing.service.ts、src/app/op-log/apply/operation-applier.service.ts、src/app/op-log/apply/operation-converter.util.ts 的顺序读。
  8. 最后根据 provider 类型分叉：文件型 provider 读 src/app/op-log/sync-providers/file-based/file-based-sync-adapter.service.ts；SuperSync 读 src/app/op-log/sync-providers/super-sync/super-sync.ts 和 packages/super-sync-server/src/sync/sync.routes.ts。
