交付路径对比

场景分享链路Widget 勾选链路
应用前台存活push onShareWithAttachment$push onWidgetDoneDrainRequest$ + pull getWidgetDoneQueue
冷启动 / 进程被杀pull getPendingShareData；onResume 再 pull 一次仅 onResume$ 触发 drain（无 JS push）
Activity 配置重建savedInstanceState != null 时不重跑 handleIntent，防重复建任务队列仍在 SharedPreferences，resume 时 drain

Widget 快照推送触发器

  1. pushOnStateChange$：selectAndroidWidgetData 变更后 debounce 500ms；跳过 remote op hydration 窗口。
  2. pushOnSyncWindowEnd$：sync window 从 true→false 时补推，避免远程变更被守卫吞掉。
  3. pushOnPause$：进入后台前立即推送最新快照，无 debounce。
  4. WidgetDataService.pushCurrent：JSON 与上次成功推送相同时跳过；失败不更新 _lastPushedJson 以便重试。

drain 过滤逻辑

getTaskDoneChangesToApply(queueJson, taskEntities) 解析 JSON object 后仅保留：

  • isDone 为 boolean；
  • 任务实体仍存在；
  • 当前 task.isDone !== targetIsDone（已是目标状态则跳过，避免冗余 op）。

drain 前等待 isAllDataLoadedInitially$，防止 store 未 hydrated 时误判任务缺失。

与提醒队列对比

提醒 Done/Snooze 队列（slice-android-native-reminder-flow）与 Widget/分享队列共享同一心智模型：native commit() 持久化、Angular 冷启动 pull、温启动 onResume$ 补捞。差异在于分享额外有 warm-start push（onShareWithAttachment$），Widget 勾选额外有 contentless drain 广播（onWidgetDoneDrainRequest$），而提醒 action 多在 ReminderActionReceiver 后台完成调度。
