Android 分享 intent、Home Widget 与队列 drain

Android 端不走 Capacitor 插件接分享，而是由 CapacitorMainActivity 解析 ACTION_SEND，写入 ShareIntentQueue，再经 SUPAndroid 桥推入 onShareWithAttachment$；桌面 Widget 则反向由 Angular 推送今日任务快照，用户勾选后写入 WidgetDoneQueue，由 drainWidgetDoneQueue$ 拉取并调用 setDone。

2. 两条链路的共同模式

链路native 写入触发 AngularAngular 消费
分享 intentShareIntentQueue.setPending（始终持久化）push onShareWithAttachment$ 或 pull getPendingShareData；triggerGetShareData 置 isFrontendReadyhandleShare$ 创建任务 + 附件；checkPendingShareOnResume$ 补捞
Widget 勾选WidgetDoneQueue.setTarget（last-wins map）onResume$ 或 onWidgetDoneDrainRequest$（无内容信号）drainWidgetDoneQueue$ → getWidgetDoneQueue → getTaskDoneChangesToApply
Widget 列表KeyValStore widget_data 快照NgRx 变更 / pause / sync window 结束WidgetDataService.pushCurrent → updateWidget

设计要点：native 侧只持久化队列，不把任务数据拼进 JS 字符串；Angular 始终是唯一的状态写入方（NgRx setDone）。Widget 渲染时用 WidgetDoneQueue.peek() 叠加 pending 勾选，用户能立刻看到反馈，不必等 WebView 启动。

3. 三个判断点

判断一：分享后没生成任务，先查 handleIntent 是否因空白 EXTRA_TEXT 被跳过；再查冷启动时 android-interface.ts 是否成功 getPendingShareData，以及 handleShare$ 是否因空 path 被 guard 掉。

判断二：Widget 勾选了但任务未变，查 drainWidgetDoneQueue$ 是否等到 isAllDataLoadedInitially$；再查 getTaskDoneChangesToApply 是否因任务已删除或已处于目标 isDone 而跳过（这是正常去重，不是 bug）。

判断三：同步后 Widget 列表过期，查 pushOnStateChange$ 的 hydration 守卫——远程 op 应用期间会暂停推送；sync window 关闭后靠 pushOnSyncWindowEnd$ 补推一次。

4. 60 秒自测

场景：用户在桌面 Widget 勾选任务 A 为完成，此时应用进程完全被杀；随后从浏览器分享一篇长文到 Super Productivity，再手动打开应用。打开后应发生哪两件事、各自走哪条队列？若只完成了分享建任务、任务 A 仍为未完成，应优先查哪两个 drain 触发点？

参考答案：应发生（1）分享数据经 ShareIntentQueue 被 getPendingShareData / flushPendingShareIntent 消费，handleShare$ 创建带附件的任务；（2）Widget 勾选经 WidgetDoneQueue 在 onResume$ 触发 drainWidgetDoneQueue$，getWidgetDoneQueue 后 setDone。若分享成功但 A 未变，优先查冷启动是否只走了分享 pull 而未触发 Widget drain——确认 onResume$ 是否发射、isAllDataLoadedInitially$ 是否阻塞；其次查 getTaskDoneChangesToApply 是否判定 A 已 done 或已删除。
