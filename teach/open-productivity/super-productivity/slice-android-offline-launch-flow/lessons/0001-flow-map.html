L2 垂直切片 · 预计 15 分钟内 · 目标：能判断冷启动时走 legacy 在线 WebView 还是 Capacitor 离线宿主，并跟到 Angular bootstrap

Android 离线启动、legacy 迁移与 Activity 切换

Manifest 把 FullscreenActivity 注册为 LAUNCHER，但 LaunchDecider 会在 onCreate 首段把新装/离线用户切到 CapacitorMainActivity，由 Capacitor 加载打包进 APK 的离线 bundle；legacy 升级用户（存在 databases/SupKeyValStore）仍留在在线 WebView，通过 window.SUPAndroid 与 Angular 通信。完整索引见 参考文档；父模块见 移动端宿主导览。

2. 三层边界

层关键组件负责什么不要误判
Activity 路由LaunchDecider + Manifest LAUNCHER新装 → MODE_OFFLINE；有 SupKeyValStore 的升级 → MODE_ONLINE；结果写入 SharedPreferences分享/OAuth intent 直达 CapacitorMainActivity，不经过 FullscreenActivity。
WebView 宿主CapacitorMainActivity vs FullscreenActivity离线加载 APK 内 www；legacy 加载 BuildConfig.ONLINE_SERVICE_HOST 在线 URL两代都注入 SUPAndroid，但仅 Capacitor 侧 Capacitor.isNativePlatform() 为 true。
Angular 桥接android-interface.ts + StartupOverlayService冷启动拉取分享/提醒/Widget 队列；overlay 文案交给 Web 添加任务条android.effects.ts 的 onResume$ 只覆盖温启动，冷启动队列必须在 android-interface 模块初始化时 drain。

3. 三个判断点

判断一：升级后仍打开在线页，查 LaunchDecider.hasLegacyData() 是否发现 databases/SupKeyValStore，或 SharedPreferences 中 launch_mode 是否已被写成 1。强制离线可在构建配置设 LAUNCH_MODE=2（见 android/README_OFFLINE.md）。

判断二：启动闪一下又重开，查 WebViewRecovery.scheduleRelaunch：瞬态 WebView 初始化失败会延迟 2.5s 经 FullscreenActivity 自动重开一次；预算耗尽才进 WebViewBlockActivity 终端阻断屏。

判断三：分享/Widget 冷启动丢事件，查 CapacitorMainActivity.handleIntent 是否因 savedInstanceState != null 跳过（配置变更重建），以及 flushPendingShareIntent 是否在 Angular ready 后才推送 onShareWithAttachment$。

4. 60 秒自测

场景：用户从 Play 升级安装（非全新安装），本地仍有旧版 SupKeyValStore 数据库，但构建时 LAUNCH_MODE=2。冷启动后应用加载的是离线 Capacitor bundle 还是在线 WebView？若 WebView 初始化失败触发自动重开，第二次启动走哪条 Activity 路径？

参考答案：BuildConfig.LAUNCH_MODE=2 时 getLaunchMode() 直接返回 MODE_OFFLINE，忽略 legacy 数据与 SharedPreferences，因此加载离线 Capacitor bundle。自动重开由 WebViewRecovery.relaunchNow 显式启动 FullscreenActivity，再次经 LaunchDecider 路由——在 LAUNCH_MODE=2 下仍会切到 CapacitorMainActivity，不会退回 legacy 在线页。
