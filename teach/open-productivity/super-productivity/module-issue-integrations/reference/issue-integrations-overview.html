Issue Provider、日历集成与外部工作项模块参考

模块职责

本模块负责把外部工作系统中的 issue、VTODO、calendar event 和插件工作项统一接入 super-productivity 的任务域。它的边界不是“每个 provider 的 API 细节”，而是五个稳定问题：配置在哪里、如何搜索/导入、如何刷新已有任务、哪些字段能写回远端、日历事件和 time block 是否走独立链路。

架构位置：它位于 src/app/features/issue/、src/app/features/calendar-integration/、src/app/plugins/issue-provider/ 和少量 Electron IPC 之间，最终依赖 TaskService、ProjectService、WorkContextService 把外部工作项落到本地任务。

链路判断表

问题先读入口核心判断
为什么 provider 出现在设置里？src/app/features/issue/issue.const.ts、provider *-cfg-form.const.ts、插件 registry内置 provider 来自 ISSUE_PROVIDER_TYPES 和 form map；插件 provider 来自 runtime 注册。
新远端 issue 为什么进 backlog？src/app/features/issue/store/poll-to-backlog.effects.ts需要启用、默认项目、自动 add to backlog、有效 poll interval；agenda view 插件不走这条。
已有 issue task 为什么标题/状态更新？src/app/features/issue/store/poll-issue-updates.effects.ts轮询取当前上下文、always 全局或 Calendar/插件全局任务，再调用 provider 的 fresh data 方法。
本地任务改动为什么写回远端？src/app/features/issue/two-way-sync/issue-two-way-sync.effects.ts监听本地 task actions，字段必须在 adapter mapping 中且方向允许 push。
为什么没有写回？src/app/features/issue/two-way-sync/compute-push-decisions.ts常见原因是方向不允许、没有 baseline，或远端字段已变化导致 provider wins。
日历事件为什么显示但不变成任务？src/app/features/calendar-integration/calendar-integration.service.ts可能被正则过滤、已 linked、已 skip、已 hidden、reference calendar，或不在自动导入条件内。
本地 scheduled task 为什么出现在 Google Calendar 类日历？src/app/features/calendar-integration/time-block/time-block-sync.effects.ts只有插件 provider 支持 timeBlock 且配置开启 isAutoTimeBlock 才写回。
Jira 在浏览器/Electron 请求路径为何不同？src/app/features/issue/providers/jira/jira-api.service.ts、electron/jira.tsElectron 通过 preload IPC 到主进程 fetch；浏览器通常依赖扩展或 fallback。

issue-to-task 映射规则

字段来源落地位置说明
provider getAddTaskData()Task 部分字段返回 title、isDone、issueLastUpdated、点数、附件数、due 字段、notes、baseline 等 provider 专属映射。
IssueService.addTaskFromIssue()任务创建参数补 issueType、issueProviderId、issueId、默认项目、默认标签、默认 note、backlog 或 Today 行为。
Calendar eventdueDay 或 dueWithTimeall-day event 映射到 dueDay，有具体时间的事件映射到 dueWithTime。
CalDAV VTODOdueDay、dueWithTime、deadline 字段start/due 分别映射；etag/hash 用作更新时间标记。
PlainspaceisDone、title、dueWithTime、baseline远端 scheduledAt 进本地 dueWithTime，本地改动可推回远端。
插件 provider基础 issue 字段 + fieldMappingsadapter 从插件定义提取 sync values，可按 direction 拉取 tag、due、title、done 等字段。

阅读示例：追一次 backlog 自动导入

  1. 在 src/app/features/issue/store/poll-to-backlog.effects.ts 找 pollNewIssuesToBacklog$ 和 pollNewIssuesToBacklogAlways$，确认触发条件。
  2. 进入 IssueService.checkAndImportNewIssuesToBacklogForProject()，看它如何收集已有 issue id、调用 provider 的 getNewIssuesToAddToBacklog() 并过滤重复。
  3. 继续看 IssueService.addTaskFromIssue()，确认默认项目、默认标签、自动导入不继承当前 tag、Calendar/Plainspace 确定性 id、dueDay 与 dueWithTime 的处理。
  4. 如果 provider 是内置，跳到对应 *-common-interfaces.service.ts；如果是迁移 key 或 plugin:*，跳到 PluginIssueProviderAdapterService。
  5. 最后回到任务领域 reference，确认创建出的 Task 如何进入 Project、Tag、Today 或 backlog 视图。

维护边界

改动类型应放位置不要误放
新增内置 provider 配置字段provider model、默认配置、cfg form、IssueProvider 类型。不要只改 UI form，store 类型和默认值也要跟上。
远端字段映射为任务字段provider getAddTaskData() 或插件 fieldMappings。不要在 polling effect 中硬编码 provider 字段。
拉新 issue 进 backlogprovider getNewIssuesToAddToBacklog() + 通用 PollToBacklogEffects。不要在 provider-specific effects 里另起重复轮询。
本地字段写回远端two-way-sync adapter 和 computePushDecisions()。不要放进 getFreshDataForIssueTask()，那是拉远端变化。
日历事件显示/隐藏/自动导入calendar-integration 服务和 effects。不要把 iCal 事件当普通 issue API 逐个 getById()。
Jira 桌面请求能力Electron IPC 与 jira-api.service.ts executor 分支。不要让主进程理解业务映射；它只代发请求和设置 header。
