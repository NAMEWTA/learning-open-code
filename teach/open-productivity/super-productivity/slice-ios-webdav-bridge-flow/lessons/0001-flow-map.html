iOS WebDAV 插件、no-cache 与同步 provider 全链路

iOS 上 WebDAV 同步不走 CapacitorHttp，而是由 CustomViewController 注册的 WebDavHttp 插件经 URLSession 发请求；JS 侧 WebDavHttpAdapter 在 native 路径额外注入 no-cache 头，并与 Swift 侧禁用 URLCache 形成双层防线，避免 content-hash 冲突检测读到陈旧 sync-data.json。

2. 三层 no-cache 防线

层级位置作用
请求头WebDavHttpAdapter.NO_CACHE_HEADERSnative 路径附加 Cache-Control: no-cache, no-store，提示上游代理勿返回陈旧副本。
会话策略WebDavHttpPlugin.swift lazy URLSessionrequestCachePolicy = .reloadIgnoringLocalCacheData，urlCache = nil，禁用 iOS 默认共享缓存。
冲突检测依赖WebdavApi.upload 上传前 GET用 content-hash 比对 expectedRev；若 GET 命中缓存，会误判远端未变并静默覆盖（#7144）。

Web 端用 fetch({ cache: 'no-store' }) 达到同等效果；不在浏览器请求中加 Cache-Control，是为避免 CORS 预检失败。

3. 三个判断点

判断一：插件方法未定义或请求直接失败，先查 CustomViewController 是否在 capacitorDidLoad 调用了 registerPluginInstance(WebDavHttpPlugin())，以及 Capacitor 是否加载了同名 WebDavHttp 插件。

判断二：双端数据不一致却未弹冲突，查上传前 GET 是否返回了与本地 expectedRev 相同的 hash——结合日志中 cacheHeaders（age、x-cache）判断是客户端还是代理缓存；对照 Swift 侧 urlCache = nil 是否生效。

判断三：恢复前台后远端变更未拉取，ios-interface.ts 的 onResume$ 只负责补时与 flushPendingWrites，真正 WebDAV 拉取仍由 SyncTriggerService → SyncWrapperService.downloadRemoteOps 触发；勿把 resume Subject 当成 HTTP 桥。

4. 60 秒自测

场景：iOS 客户端 A 与 B 共用 WebDAV；B 修改了远端 sync-data.json 后，A 在后台挂起数小时再回前台触发同步，A 却用旧内容覆盖了 B 的更新且未报冲突。最可能失效的是哪一层 no-cache 防线？应优先查看哪两个日志字段？

参考答案：最可能是上传前 GET 读到了陈旧 body，导致 WebdavApi.upload 的 content-hash 比对认为 currentHash === expectedRev 而放行 PUT。优先查 WebDavHttpAdapter.response() 日志里的 cacheHeaders（若出现 age>0 或 x-cache=HIT 则怀疑代理缓存），以及 Swift 插件是否使用带 urlCache = nil 的专用 URLSession 而非默认配置。
