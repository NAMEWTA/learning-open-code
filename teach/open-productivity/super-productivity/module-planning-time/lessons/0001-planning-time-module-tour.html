L1 模块总览 · 预计 15 分钟 · 目标：能判断时间相关 feature 的边界和第一阅读入口

计划、排程、追踪、反馈这四件事别混读

一句话：Planner 和 Boards 帮用户决定“哪些任务在哪个计划面板里”，Schedule 把这些任务映射成时间轴，time-tracking 记录 work context 的真实工时，focus/idle/reminder/metric/history 再围绕追踪结果提供提醒、修正和回顾。

1. 它在 L0 地图中的位置

L0 项目地图已经把 Super Productivity 看成 Angular UI、NgRx store、op-log 和平台桥接的组合。本主题位于 task/work-context 之上：它不定义任务领域模型，而是消费 task、tag/project work context、calendar、Electron idle 线索，把“计划时间”和“实际时间”对齐。

src/app/root-store/feature-stores.module.ts 是第一张索引表：这里注册了 focusMode、idle、metric、boards、timeTracking、planner、reminders。注意 Schedule、History、Worklog 没有自己的 feature store，主要是视图与聚合服务。

2. 边界速记

Planning surface
  Planner: day -> taskIds；计划到某天
  Schedule: task/planner/calendar -> ScheduleDay[]；渲染时间轴
  Boards: panel cfg + taskIds；按条件组织任务

Tracking and feedback
  timeTracking: project/tag/date -> start/end/break
  focus-mode: 本地 timer、session、break；完成后写 metric
  idle/reminder: 平台或 worker 信号；修正或提前触达

Review
  worklog/history: 从 task archive + timeTracking 聚合工时
  metric: 每日评价、focusSessions、反思与评分

完整入口清单、依赖表和边界判断见 模块参考。这节课只建立阅读顺序：先看谁拥有状态，再看谁只是把多个状态映射成页面。

3. 阅读示例：从计划到日程

src/app/features/planner/store/planner.actions.ts 里的持久 action 说明 Planner 自己只保存“某天有哪些任务”。它会进 op-log，但不等于完整任务状态。

'Plan Task for Day': ({ task, day }) => ({
  task,
  day,
  meta: {
    isPersistent: true,
    entityType: 'PLANNER',
    entityId: task.id,
    opType: OpType.Update,
  },
})

接着读 src/app/features/schedule/schedule.service.ts。ScheduleService 把 timeline tasks、repeat cfg、planner day map、calendar events、timeline config 和 currentTaskId 合成 ScheduleDay[]，所以它更像投影层，不是新的业务事实来源。

return this.buildScheduleDays({
  daysToShow: daysToShow(),
  timelineTasks,
  taskRepeatCfgs,
  calendarEvents,
  plannerDayMap,
  timelineCfg,
  currentTaskId,
});

4. 检索练习

暂停 60 秒，凭记忆判断：如果用户把任务拖进 Boards 的“已排程”列，最终应该修改哪个领域事实？

参考答案：Boards 可以更新 panel 的手动顺序，但“任务被排程”本身仍应落到 task 的 dueDay、dueWithTime 或提醒字段；源码中 BoardPanel 会打开调度弹窗或 dispatch TaskSharedActions，而不是让 Boards 成为 schedule 状态源。
