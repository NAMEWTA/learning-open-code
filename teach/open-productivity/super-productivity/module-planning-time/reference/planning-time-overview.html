计划排程、时间追踪与专注反馈模块参考

模块职责

一句话：这个模块群把 task/work-context 的领域事实转换成计划视图、日程视图、真实工时、专注反馈、空闲修正和历史指标。它位于 L0 总览 的 Angular feature/store 层，并依赖 op-log 同步边界持久化部分 action。

子模块职责边界第一入口
Planner维护 day -> taskIds 的计划天列表，合并 overdue、deadline、repeat projection、calendar event 和容量估算。不拥有 task 内容；只保存计划归属和顺序。src/app/features/planner/store/planner.reducer.ts
src/app/features/planner/store/planner.selectors.ts
Schedule把任务、planner day map、重复任务、calendar events、timeline config 映射为周/月/日程事件。没有独立 store；是视图投影和拖拽交互层。src/app/features/schedule/schedule.service.ts
src/app/features/schedule/map-schedule-data/map-to-schedule-days.ts
Boards维护 board/panel 配置、过滤条件、手动 taskIds 顺序，并在拖拽时触发任务标签、项目、排程或 backlog 变化。看板配置属于 Boards；任务事实仍走 task/project actions。src/app/features/boards/store/boards.reducer.ts
src/app/features/boards/board-panel/board-panel.component.ts
time-tracking按 project/tag/date 记录工作开始、结束、休息次数和休息时长；合并当前与归档状态。不渲染回顾页；只提供工时状态来源。src/app/features/time-tracking/store/time-tracking.reducer.ts
src/app/features/time-tracking/time-tracking.service.ts
focus-mode维护本地 timer、session、break、overlay 状态；与当前任务追踪互相同步，完成后写 focus session metric。timer 多为本地体验状态；完成记录落到 Metric。src/app/features/focus-mode/store/focus-mode.reducer.ts
src/app/features/focus-mode/store/focus-mode.effects.ts
idle消费 Electron 或扩展 idle 时间，打开空闲对话框，回滚已记时间并让用户分配到任务、休息或计数器。平台检测在 Electron；前端负责业务修正。src/app/features/idle/store/idle.effects.ts
electron/start-app.ts
reminder将 task 的 schedule/deadline reminder 写入 worker，触发提醒弹窗或倒计时 banner。旧 reminders store 仍加载；新提醒事实主要在 task 字段。src/app/features/reminder/reminder.service.ts
src/app/features/reminder/store/reminder-countdown.effects.ts
metric保存每日评价、focusSessions、反思，结合 Worklog 计算 productivity/sustainability。不是原始工时来源；工时优先来自 Worklog。src/app/features/metric/store/metric.reducer.ts
src/app/features/metric/metric.service.ts
history/worklog按 active work context 读取 task archive、当前 task state 与 timeTracking，生成历史页、导出和每日工时。Worklog 是聚合视图，不是新的同步事实源。src/app/features/worklog/worklog.service.ts
src/app/features/history/history.component.ts

整体架构位置

Task / WorkContext domain
  ↓ consumed by
Planning surfaces
  ├─ Planner: planned day and order
  ├─ Schedule: derived timeline events
  └─ Boards: configurable task panels
  ↓ current task / schedule / user intent
Tracking and correction
  ├─ timeTracking: project/tag daily sessions
  ├─ focus-mode: local session and break timer
  ├─ idle: platform idle correction
  └─ reminder: task reminder worker and countdown
  ↓ aggregation
Review
  ├─ worklog/history: archive + task + timeTracking
  └─ metric: daily rating, focus sessions, reflections

阅读示例：计划到日程再到回顾

  1. 从 src/app/app.routes.ts 找到 /planner、/schedule、/boards 的页面入口；从 src/app/routes/context.routes.ts 找到 project/tag 下的 history 与 metrics。
  2. 到 src/app/root-store/feature-stores.module.ts 核对哪些 feature 有 store：Planner、Boards、timeTracking、focusMode、idle、metric、reminders 都注册；Schedule、History、Worklog 不是独立 feature state。
  3. 读 src/app/features/planner/store/planner.actions.ts 与 src/app/features/planner/store/planner.reducer.ts，确认 Planner 保存 day taskIds，并通过 entityType: 'PLANNER' 持久化。
  4. 读 src/app/features/schedule/schedule.service.ts，观察 buildScheduleDays() 需要 task timeline、repeat cfg、calendar、planner day map 和 timeline config；这说明 Schedule 是聚合投影。
  5. 读 src/app/features/time-tracking/store/time-tracking.actions.ts，区分 syncTimeTracking 的 work context 会话数据与 syncTimeSpent 的 task 累计耗时。
  6. 读 src/app/features/focus-mode/store/focus-mode.effects.ts 的 logFocusSession$：focus 完成后调用 MetricService.logFocusSession()，由 Metric store 记录到当天。
  7. 读 src/app/features/idle/store/idle.effects.ts：idle 触发时先撤销当前任务已记 idle time，再让用户把空闲时间重新分配。
  8. 读 src/app/features/worklog/worklog.service.ts 与 src/app/features/metric/metric.service.ts：回顾页从 archive、当前 task、timeTracking、metric 组合出历史和评分。
