移动端后台恢复链路地图

本课只练一个判断：看到“从后台回来后时间、数据或同步不对”，先沿哪条主链路追。完整源码索引见 参考文档。

一眼路径图

主路径不是“原生层直接写业务数据”，而是原生层发出生命周期/服务读数，Angular Web 层完成补时、op-log 落盘和同步补偿。

Android Activity onPause/onResume -> window.SUPAndroid.onPause$/onResume$ -> AndroidForegroundTrackingEffects -> TaskService.flushAccumulatedTimeSpent() -> OperationWriteFlushService.flushPendingWrites() -> SUP_OPS IndexedDB -> SyncTriggerService / SyncWrapperService iOS Capacitor appStateChange -> iosInterface.onResume$ -> IosBackgroundTrackingEffects -> TaskService.flushAccumulatedTimeSpent() -> OperationWriteFlushService.flushPendingWrites() PWA / Web visibilitychange -> visible: HydrationStateService.openSyncWindow() -> hidden: SyncTriggerService emits I_VISIBILITY_HIDDEN -> other triggers: Android resume / user activity / online / interval / initial sync -> SyncEffects.triggerSync$ -> downloadRemoteOps() then uploadPendingOps()

三个锚点文件先记住： src/main.ts:472-526 负责后台前 drain， src/app/features/android/store/android-foreground-tracking.effects.ts:327-397 负责 Android 恢复补时， src/app/imex/sync/sync-trigger.service.ts:63-79 负责 visible 恢复时打开 sync window；真正触发 sync 的入口还包括 hidden、Android resume、用户活动、online、interval 和 initial sync。

三个判断点

现象 先查哪里 判断
后台回来少记时间 android-foreground-tracking.effects.ts 或 ios-background-tracking.effects.ts 是否先补 wall-clock gap，再 flush 累积时间。
关闭/杀进程后本地改动丢失 src/main.ts 与 operation-write-flush.service.ts 是否进入 `BackgroundTask.beforeExit`，并等 pending counter 归零和写锁释放。
恢复后同步状态长期不对 sync-trigger.service.ts 与 sync-wrapper.service.ts 是否先 download remote ops，再 upload pending ops；失败时是否保持可重试状态。
PWA cache 不写业务数据。ngsw-config.json 只定义资源缓存和少量 API freshness；真正的数据一致性仍由 op-log、hydration 和 sync 负责。

60 秒判断题

用户在 Android 上开始计时，切后台 20 分钟后回来，任务时间补上了，但另一台设备没有看到这 20 分钟。你先查哪一段？

A. service worker cache 配置；B. Android 恢复补时后的 op-log flush/upload；C. iOS AppDelegate 生命周期。

答案：B。 Android 恢复补时只说明本地 NgRx/TaskService 已被修正；跨设备可见还要确认 syncTimeSpent 是否被捕获成 op、flushPendingWrites() 是否完成，以及后续 uploadPendingOps() 是否把 unsynced ops 推到 provider。

下一步
