移动端后台恢复、持久化刷新与同步补偿参考

本页用于排查 Android/iOS/PWA 从后台恢复后的链路边界。短课入口： 0001-flow-map.html。

总链路

Native / Browser signal -> Web lifecycle bridge -> time compensation -> pending operation write flush -> SUP_OPS hydration / state cache -> sync trigger window -> download remote ops -> upload pending local ops -> deferred local actions / validation / status
层 职责 不要误判
原生层 Activity/App 生命周期、前台 service 读数、Android SharedPreferences 凭证桥。 原生层不直接把业务模型同步到远端。
Angular Web 层 补时、dispatch 持久化 action、flush pending writes、触发 sync。 Web 层必须等数据初始化完成后才启动自动同步。
op-log 层 把 persistent action 写入 SUP_OPS，用 vector clock 支撑上传、下载和冲突判断。 localStorage/legacy pf 不是当前主业务日志。
PWA service worker 缓存 app shell、关键 assets 和少量外部 API。 service worker 不负责 op-log 落盘，也不负责 provider sync。

前后台入口

Android

android/app/src/main/java/com/superproductivity/superproductivity/CapacitorMainActivity.kt:178-188 将 JavaScriptInterface 注入为 window.SUPAndroid。 同文件 android/app/src/main/java/com/superproductivity/superproductivity/CapacitorMainActivity.kt:474-484 在 Activity onPause()/onResume() 中调用 onPause$/onResume$。

Web 侧 src/app/features/android/android-interface.ts:165-192 把这些入口包装成 ReplaySubject、Subject 和 isInBackground$。Android 用 ReplaySubject(1) 的原因是 WebView 可能已被杀，冷启动恢复时仍要交付最近一次 resume。

iOS

src/app/features/ios/ios-interface.ts:17-22 使用 Capacitor appStateChange 在 isActive 时发出 iosInterface.onResume$。iOS pause 的持久化不在这个 effect 做，而在 src/main.ts:505-526 的 BackgroundTask.beforeExit 预算里完成。

PWA / Browser

src/app/imex/sync/sync-trigger.service.ts:74-79 在 visibilitychange 变为 visible 时立即打开 sync window； src/app/imex/sync/sync-trigger.service.ts:154-162 在 hidden 时发出最后一次 sync 触发。PWA 没有 Android/iOS 原生 service 读数，只依靠浏览器事件和 Web 层 flush/sync。

持久化与数据初始化

启动时 src/main.ts:266-275 通过 APP_INITIALIZER 实例化 DataInitService。随后 src/app/core/data-init/data-init.service.ts:37-53 调用 OperationLogHydratorService.hydrateStore()，完成 profile 初始化和 op-log hydration。

src/app/core/data-init/data-init-state.service.ts:5-14 暴露一次性的 isAllDataLoadedInitially$。许多同步和上传入口都等这个信号，避免启动阶段 config 未加载就上传到默认 provider。

op-log flush 与 sync 补偿

persistent action 的捕获由 src/app/op-log/capture/operation-capture.meta-reducer.ts:214-284 同步递增 pending counter，实际写入由 src/app/op-log/capture/operation-log.effects.ts:101-145 的 effect 顺序处理。写入时 src/app/op-log/capture/operation-log.effects.ts:306-325 以单个 transaction 追加 op、更新 vector clock，并触发 immediate upload。

同步主循环在 src/app/imex/sync/sync-wrapper.service.ts:465-631： 先 downloadRemoteOps()，再 uploadPendingOps()。上传前 src/app/op-log/sync/operation-log-sync.service.ts:169-182 再次调用 flushPendingWrites()，防止“action 已 dispatch 但 op 尚未写入”导致漏传。

补偿机制 位置 解决的问题
sync window src/app/op-log/apply/hydration-state.service.ts:70-89 恢复触发到真正 sync 之间，压住 selector effects，避免 stale op。
deferred actions src/app/op-log/capture/operation-log.effects.ts:604-668 用户在 remote apply 期间操作时，等 sync 后用新 vector clock 补写。
remote apply validation src/app/op-log/sync/remote-ops-processing.service.ts:79-90 下载/冲突/应用后做 Checkpoint D，失败不应标记为 IN_SYNC。
LWW re-upload src/app/imex/sync/sync-wrapper.service.ts:552-610 冲突解决产生 local-win ops 后，立即尝试再上传，失败则下轮 sync 重试。
immediate upload src/app/op-log/sync/immediate-upload.service.ts:17-47 SuperSync 下 op 落盘后近实时上传；静默失败由正常 sync 接管。

Service Worker 缓存边界

src/main.ts:178-186 通过 Angular ServiceWorkerModule.register('ngsw-worker.js') 注册 service worker，但条件是非 Electron、非 native、生产或 stage。 同文件 src/main.ts:415-438 也显式在 Electron/native 下反注册已有 service worker。

ngsw-config.json 把 app shell、关键 assets 和外部图片/API 分组缓存。这个配置可能影响“用户看到的是哪版 Web bundle”，但不参与 SUP_OPS 写入、flushPendingWrites() 或 provider sync。 初始同步前会经过 src/app/core/initial-pwa-update-check.service.ts:21-46 的 update check；它只决定是否提示刷新，随后才进入 SyncEffects.triggerSync$。
