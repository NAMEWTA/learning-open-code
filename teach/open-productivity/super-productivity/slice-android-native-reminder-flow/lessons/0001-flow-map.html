Android 原生提醒：调度、触发与 action 回写

Android 提醒不走 Capacitor LocalNotifications 定时，而是 NgRx 选出带 `remindAt` 的任务后，经 `SUPAndroid` 桥注册 `AlarmManager`；到点后 `ReminderAlarmReceiver` 弹通知，Snooze/Done 由 `ReminderActionReceiver` 在后台处理，Angular 下次启动或 `onResume` 时再消费队列写回状态。

2. 三层边界

层关键组件负责什么不要误判
Angular 调度`MobileNotificationEffects` + `generateNotificationId`根据 NgRx 任务列表增删原生闹钟；同一 `task.id` 始终映射同一 `notificationId`它不是弹通知的地方，只负责注册/取消 Alarm。
原生闹钟`ReminderNotificationHelper` + `ReminderAlarmStore` + `BootReceiver``AlarmManager` 精确唤醒；参数持久化；重启/更新后重注册应用进程被杀不影响已注册闹钟；更新后靠 `BootReceiver` 恢复。
通知 action`ReminderActionReceiver` + 三个 Queue + `ReminderModule`Snooze 纯原生重调度；Done 清闹钟并入队；Angular hydrated 后写回 NgRxSnooze 不经过 Capacitor；冷启动靠 pull 队列，不是 push JS 回调。

3. 三个判断点

判断一：提醒到点没响，先查调度是否成功——mobile-notification.effects.ts 是否因 `disableReminders`、权限或 `remindAt` 已过期而跳过；再查 ReminderNotificationHelper.scheduleReminder 是否写入 `ReminderAlarmStore`。

判断二：另一台设备已完成任务但本机仍弹窗，看 ReminderAlarmReceiver.isTaskStale：它会用 SuperSync 快速拉取 `taskIdsToCancel` 与重排时间；任何网络/凭证错误都 fail-open 继续弹通知。

判断三：通知上点 Done 任务没变，查 action 是否只入了 `ReminderDoneQueue`——需等 reminder.module.ts 在数据加载后消费 `onReminderDone$`；温启动还要靠 android.effects.ts 的 `checkReminderQueuesOnResume$` 再 drain 一次。

4. 60 秒自测

场景：用户在通知上点「Snooze 10m」，应用此时完全被杀。10 分钟后通知再次弹出，但任务 `remindAt` 在 Angular 里仍是旧值，直到用户手动打开应用才更新。这条链路里，哪两步是设计如此、哪一步若缺失才算 bug？

参考答案：设计如此的是（1）`ReminderActionReceiver` 在后台直接 `AlarmManager` 重调度，不依赖 WebView；（2）`ReminderSnoozeQueue` 只持久化 `{taskId, newRemindAt}`，Angular 冷启动或 `onResume` 时才由 `ReminderModule._handleSnoozeAction` 写回 NgRx。若 10 分钟后根本没再弹通知，才是 bug，应查 `ReminderNotificationHelper.scheduleReminder` 或 `ReminderAlarmStore`；若打开应用后 `remindAt` 仍不更新，才查 `onReminderSnooze$` 订阅与 hydration 守卫。
