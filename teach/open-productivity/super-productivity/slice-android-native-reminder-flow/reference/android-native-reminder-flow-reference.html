阶段速查

阶段入口输出失败时表现
调度注册 `scheduleNotifications$` → `CapacitorReminderService.scheduleReminder` → `JavaScriptInterface.scheduleNativeReminder` `AlarmManager` PendingIntent + `ReminderAlarmStore` 条目 无通知；日志见 `CapacitorReminderService` / `ReminderNotifHelper`
闹钟触发 `ReminderAlarmReceiver.onReceive(ACTION_SHOW_REMINDER)` `ReminderNotificationHelper.showNotification` 或 stale 抑制 过期任务仍可能弹窗（fail-open）；stale 时主动 `cancelReminder`
通知 action `ReminderActionReceiver`（Snooze 10m / 1h / Done） 重调度闹钟 + SharedPreferences 队列；Done 还 `ReminderAlarmStore.remove` 通知消失但 NgRx 未变 → 队列未被 Angular drain
Angular 回写 冷启动 `android-interface.ts` + `onResume` `android.effects.ts` + `reminder.module.ts` `dismissReminderOnly` / `setDone` / snooze 更新 `remindAt` hydration 前事件被 `_handleAfterDataLoaded` 推迟
重启恢复 `BootReceiver` on `BOOT_COMPLETED` / `MY_PACKAGE_REPLACED` 遍历 `ReminderAlarmStore.getAll` 重注册未来闹钟 Play 自动更新后提醒静默失效直到用户打开应用
跨设备同步 `SyncReminderWorker` 每 15 分钟（需 SuperSync 凭证） 取消远端已完成任务闹钟；调度远端新增/改期提醒 仅影响后台增量；触发瞬间仍靠 `isTaskStale` 快速校验

Intent / Action 常量

常量定义位置用途
`ACTION_SHOW_REMINDER``ReminderAlarmReceiver`AlarmManager PendingIntent action；携带 notificationId、reminderId、relatedId、title、reminderType、triggerAtMs
`ACTION_SNOOZE` / `ACTION_SNOOZE_1H``ReminderActionReceiver`通知 action；分别 +10 分钟 / +1 小时重调度
`ACTION_DONE``ReminderActionReceiver`取消通知与闹钟；`ReminderDoneQueue.addTaskId`
`REMINDER_TASK_ID``CapacitorMainActivity` intent extra用户点击通知正文打开应用；冷启动写入 `ReminderTapQueue`

通知 ID 与 PendingIntent 规则

generateNotificationId(taskId) 对字符串做 32 位哈希取正模，保证同一任务重复调度覆盖同一闹钟而非叠两条通知。

ReminderNotificationHelper 中：内容点击 PendingIntent 用 notificationId 作 requestCode；Done/Snooze action 分别用 notificationId xor 0x10000000、0x20000000、0x30000000，避免 action 之间 requestCode 冲突。

闹钟注册使用 PendingIntent.getBroadcast(..., notificationId, ...)，与取消路径共用同一 requestCode，保证 `cancelReminder` 能精确撤销。

AlarmManager 策略

  • 默认：setExactAndAllowWhileIdle(RTC_WAKEUP, triggerAtMs, pendingIntent)，Doze 下仍尽量准时。
  • Android 12+ 无 SCHEDULE_EXACT_ALARM 时降级为 setAndAllowWhileIdle（非精确）。
  • 每次成功注册后 ReminderAlarmStore.save；取消时 remove 并 NotificationManagerCompat.cancel。
  • useAlarmStyle：非 DUE_DATE 且全局配置开启时使用 alarm 渠道与 CATEGORY_ALARM；同时 isOngoing=true 使通知持久。

队列与桥接消费顺序

  1. WebView 初始化时 android-interface.ts 同步 pull：getReminderTapQueue、getReminderDoneQueue、getReminderSnoozeQueue。
  2. 事件推入 onReminderTap$ / onReminderDone$ / onReminderSnooze$（ReplaySubject，保留冷启动事件）。
  3. reminder.module.ts 通过 _handleAfterDataLoaded 等待 store hydrated 再 dispatch。
  4. 温启动：android.effects.ts 的 checkReminderQueuesOnResume$ 在 onResume$ 再次 get-and-clear 队列。

与 iOS 路径对比

同一 CapacitorReminderService.scheduleReminder 在 iOS 走 LocalNotifications.schedule，action 由 Capacitor 回调 action$ 进入 reminder.module.ts。Android 的 Snooze/Done 在 ReminderActionReceiver 原生完成，只有状态回写才进入 Angular——这是两端架构的根本差异。
