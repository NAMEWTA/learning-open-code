插件生命周期全链路参考

这份文档用于复查 Super Productivity 插件从 manifest/社区列表到加载、初始化、Bridge/API、OAuth、Node executor 和 issue provider 接入的边界。短课入口见 生命周期地图、 启动初始化、 Bridge 与 issue provider。

2. 加载阶段速查

阶段发生什么关键判断
discover built-in `BUNDLED_PLUGIN_PATHS` 加上 discover-only 插件路径，逐个读取 `manifest.json` 和可选图标。 issue provider 迁移插件可能被自动启用，但前提是用户未已有显式 metadata。
discover uploaded 从 `PluginCacheService.getAllPlugins()` 读取上传缓存，解析 manifest，检查 uploaded 插件不能复用内置 id。 缓存 id 必须等于 manifest id，否则忽略。
load enabled `_loadEnabledPlugins()` 只对 `state.isEnabled` 为真的插件调用 `activatePlugin()`。 manifest 已发现不代表插件代码已运行。
activate 设置 loading，调用 `_loadPluginLazy()` 加载资源、i18n、base config，再执行插件。 非手动启动的 nodeExecution 插件只做平台可用性检查，grant 在 `onReady` 前申请。
ready `_fireOnReady()` 对 nodeExecution 插件先拿 grant、ping bridge，再调用 `PluginRunner.triggerReady()`。 `onReady` 是插件做运行时初始化的安全位置；不要把顶层 `plugin.js` 执行等同于 ready。

3. Plugin API 与 Bridge

host-side plugin.js

src/app/plugins/plugin-runner.ts 创建 `PluginAPI`，并把生命周期注册函数交给 runner 内部 Map。`PluginAPI.registerHook()`、`registerIssueProvider()`、`startOAuthFlow()` 等方法再转发到 Bridge。

packages/plugin-api/src/types.ts 是插件作者看到的 manifest、hooks、Node script、OAuth 等类型契约。

iframe index.html

src/app/plugins/ui/plugin-index/plugin-index.component.ts 从 `PluginService.getPluginIndexHtml()` 取缓存的 HTML，构造 `PluginIframeConfig`，再设置 iframe `srcdoc`。

src/app/plugins/util/plugin-iframe.util.ts 注入 API script。iframe 内的 `PluginAPI` 通过 `postMessage` 调宿主，宿主校验 `bridgeToken` 与 `bridgeGeneration`。

边界允许什么拒绝或限制什么
iframe API allowlist 任务、项目、标签、snack、dialog、persist、部分 hook、work context、Node script 等有限方法。 菜单、header button、shortcut、config handler 等注册只支持 `plugin.js`，iframe 中返回 unsupported。
Bridge bound methods 所有调用都绑定当前 pluginId，持久化 key 会落到插件命名空间。 未在允许 action 列表里的 NgRx action 会被拒绝。
iframe sandbox 静态属性允许 scripts、same-origin、forms、popups、modals。 同源是为了 packaged desktop 渲染，但安全注释明确承认这是权衡；危险 Node 能力仍由 main consent 守门。

4. OAuth 边界

层关键源码职责
插件定义 packages/plugin-api/src/issue-provider-types.ts
packages/plugin-dev/google-calendar-provider/src/plugin.ts		 `configFields` 可放 `oauthButton`，其 `oauthConfig` 声明 authUrl、tokenUrl、clientId、scopes、平台 client id。
Bridge src/app/plugins/oauth/plugin-oauth-bridge.service.ts 校验 OAuth 配置，解析平台有效配置，调用 `prepareRedirectUri()`，打开浏览器，等待 code，换 token，持久化到本地 OAuth token store。
OAuth service src/app/plugins/oauth/plugin-oauth.service.ts 生成 PKCE verifier/challenge/state，构造授权 URL，校验 HTTPS tokenUrl/authUrl，刷新 token，校验 redirect state。
Electron main electron/plugin-oauth.ts 启动 `127.0.0.1` loopback server；只允许 https 授权 URL；系统浏览器返回 code/error/state 后发 IPC 给 renderer。

OAuth token 是本地存储，不走 op-log 同步。web 环境若插件没有 `webClientId`，Bridge 会清理 token 并报告不可用。

5. Node executor 边界

检查点源码失败时现象
manifest 权限 packages/plugin-api/src/types.ts
packages/plugin-dev/sync-md/src/manifest.json		 没有 `nodeExecution` 时不应调用 `executeNodeScript()`；web 环境直接不可用。
renderer grant src/app/plugins/plugin.service.ts
src/app/plugins/plugin-bridge.service.ts		 `_ensureNodeExecutionGrant()` 拿不到 grant 时，`onReady` 不会继续；用户拒绝会变成可重新启用的 disabled 状态。
main process 授权 electron/plugin-node-executor.ts main 校验 safe plugin id、webContents、grant token；uploaded 插件的同意可持久化，disable/uninstall/re-upload 会清理。
脚本执行 electron/plugin-node-executor.ts 简单脚本走 `vm.runInContext()`；复杂脚本 spawn 当前 Electron 二进制并设置 `ELECTRON_RUN_AS_NODE=1`；超时和脚本错误返回失败结果。

6. Issue provider 注册与调用

步骤源码边界判断
插件声明 provider 类型 packages/plugin-dev/github-issue-provider/src/manifest.json
packages/plugin-dev/google-calendar-provider/src/manifest.json		 `type: "issueProvider"` 让管理/设置层能识别插件用途，但实际 provider 定义仍要由插件代码注册。
插件代码注册定义 packages/plugin-dev/github-issue-provider/src/plugin.ts
packages/plugin-api/src/issue-provider-types.ts		 `registerIssueProvider()` 必须提供 `getHeaders`、`searchIssues`、`getById`、`getIssueLink`、`issueDisplay`、`configFields`。
Bridge 校验并登记 src/app/plugins/plugin-bridge.service.ts
src/app/plugins/issue-provider/plugin-issue-provider-registry.service.ts		 不能注册到内置 key；registry key 默认为 `plugin:<pluginId>`，迁移插件可用 manifest 中的 `issueProviderKey`。
调用 provider src/app/plugins/issue-provider/plugin-issue-provider-adapter.service.ts
src/app/plugins/issue-provider/plugin-http.service.ts		 adapter 从 store 取 issue provider 配置，再用 registry 找 definition，创建 HTTP helper 调 `searchIssues()`、`getById()` 等。
可选 sync adapter src/app/plugins/issue-provider/plugin-sync-adapter.service.ts Bridge 立即注册 adapter 的条件是 `createIssue`、`deleteIssue` 或 `fieldMappings` 且 `updateIssue`。后续 `IssueSyncAdapterResolverService.getAdapter()` 的懒创建条件更宽：`createIssue`、`deleteIssue`、`updateIssue` 任一存在即可。

`PluginHttpService` 限制 HTTP method、URL scheme 和私网访问。`allowPrivateNetwork` 只允许由内置插件 manifest 开启，云 metadata endpoint 始终被阻断。

7. 故障排查表

症状优先检查常见根因
管理页看到社区插件，但列表里没有已安装插件卡片 是否上传 zip 或是否为内置 bundled 插件 社区 JSON 只是推荐列表，不会自动缓存插件。
插件显示 not-loaded `pluginMetadata.isEnabled` 与 `PluginState.status` manifest 发现成功，但用户未启用或启用状态未持久化。
iframe 空白或 API 调用超时 是否有非空 `index.html`、bridge token/generation 是否匹配、方法是否在 allowlist iframe-only 插件缺资源、旧 iframe 消息、调用了 host-only 注册 API。
OAuth 按钮失败 authUrl/tokenUrl 是否 HTTPS、redirectUri 是否匹配平台、state 是否一致 web 无 `webClientId`、系统浏览器打不开、loopback 端口无效或被占用。
Node script 返回 permission denied Electron 运行时、manifest 权限、grant token、用户是否拒绝 web 环境、用户拒绝、本次会话 token 被 revoke、main 校验 webContents 不匹配。
issue provider 搜索为空 provider 是否注册、store 中 issueProviderKey 是否命中 registry、getHeaders 是否能返回凭据 插件代码未执行、注册 key 冲突、OAuth token 缺失、HTTP helper 阻止私网或 metadata 请求。
搜索可以，双向同步不可用 是否有 `createIssue/deleteIssue/updateIssue`，以及字段 push 是否有 `fieldMappings` 和 `updateIssue` 只读 provider 不会有写回 adapter；updateIssue-only provider 可由 resolver 懒创建 adapter，但普通任务字段 push 仍取决于 field mappings。
