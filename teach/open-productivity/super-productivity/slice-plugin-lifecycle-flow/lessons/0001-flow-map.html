插件生命周期路径图

本课只训练一个判断技能：看到一个插件入口时，先判断它处于“展示、发现、加载、Bridge 调用、桌面授权、provider 调用”哪一段，再去对应源码。

主要源码： src/main.ts， src/app/plugins/plugin-initializer.ts， src/app/core/startup/startup.service.ts， src/app/plugins/plugin.service.ts， src/app/plugins/plugin-bridge.service.ts。 细节索引见 插件生命周期参考文档。

1. 先看整条路径

社区列表只用于管理页展示，不代表已安装。
manifest 发现内置路径和 uploaded cache 生成 PluginState。
启用判断只有 isEnabled 的插件才进入 activatePlugin。
加载执行读取资源，new Function 执行 plugin.js。
Bridge/APIPluginAPI 绑定到宿主服务、iframe、OAuth。
边界能力OAuth、nodeExecution、issue provider 另有守门点。

入口不是单一文件。src/main.ts 注册的 PLUGIN_INITIALIZER_PROVIDER 是 provider token 接线；真实初始化由 src/app/core/startup/startup.service.ts 的 _initPlugins() 等初始数据和同步完成后直接调用 PluginService.initializePlugins()。社区插件列表来自 JSON 和管理页，只是“可见目录”，不是运行时自动发现机制。 

src/main.ts provider token
  -> PLUGIN_INITIALIZER_PROVIDER
     只注册 'PLUGIN_INITIALIZER'

StartupService deferred init
  -> wait data loaded + current sync done/disabled
  -> PluginService.initializePlugins()
  -> discover manifest
  -> activate enabled plugin
  -> bridge / OAuth / Node / issue provider

2. 三条边界别混在一起

web iframe 边界处理插件 UI：宿主把 `index.html` 包进 `srcdoc`，再通过 postMessage、bridge token 和允许方法列表转发调用。Electron OAuth 边界处理授权登录：renderer 构造 PKCE 流程，main process 开 loopback server 和系统浏览器。Node executor 边界处理本机执行：插件必须声明 `nodeExecution`，宿主在 `onReady` 前申请 main process grant 并 ping bridge。

issue provider 也要拆开看：插件初始化时可以调用 registerIssueProvider() 登记 provider；Bridge 立即注册 sync adapter 的条件是 create/delete 或 fieldMappings+updateIssue。后续双向同步的 resolver 还可能为 updateIssue-only provider 懒创建 adapter。provider 已注册不等于已经开始双向同步。

3. 关键判断表

你看到的现象先判断下一步源码
插件出现在 Community Plugins 只是列表展示，尚未安装 管理页 JSON 读取；不是 `initializePlugins()`
manifest 被发现但插件没跑 `isEnabled`、平台、Node 授权是否允许加载 `PluginService.activatePlugin()`
iframe UI 能打开但 API 调用失败 方法是否在 iframe allowlist，bridge token/generation 是否匹配 参考文档的 iframe bridge 章节
issue provider 能搜索但不能推送 先分清 Bridge 是否立即注册 adapter、resolver 是否能懒创建 adapter、字段 push 是否有 mappings PluginBridgeService._registerIssueProvider() 与 IssueSyncAdapterResolverService.getAdapter()

4. 60 秒判断题

一个第三方插件显示在 Community Plugins；用户下载并上传后，manifest 声明了 `iFrame: true` 和 `nodeExecution`，但启动时没有任何 issue provider。你会先得出哪三个结论？

参考答案与反馈

第一，社区列表只说明它被展示，不说明已安装或已启用。第二，上传后发现 manifest 也不等于代码执行；必须启用，并在 Electron 中拿到 nodeExecution grant，`onReady` 才会跑。第三，没有 issue provider 很可能是插件没有调用 `registerIssueProvider()`，或注册失败；这与 Node 授权是不同边界。下一步读 reference 中“加载阶段”和“issue provider 注册/调用”两节。

复习建议：先读 启动初始化时序，再读 Bridge 与 issue provider 调用时序。相邻主题可回看 插件系统 L1 与 Issue 导入与同步 L2。
