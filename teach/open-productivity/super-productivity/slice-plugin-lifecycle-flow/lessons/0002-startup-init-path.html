启动初始化：provider token 不是实际调用栈

本课只拆一个常见误判：PLUGIN_INITIALIZER_PROVIDER 在 Angular provider 列表里出现，不代表插件会在 bootstrap 阶段立即初始化。真实运行路径由 StartupService._initPlugins() 在数据加载和当前同步结束后直接调用 PluginService.initializePlugins()。

相关课程：生命周期地图 / Bridge 与 issue provider。 相关 L1：插件系统、 Angular 应用装配。

调用时序

src/main.ts
  -> bootstrapApplication(AppComponent, { providers: [PLUGIN_INITIALIZER_PROVIDER, ...] })
     注意：这里注册的是 'PLUGIN_INITIALIZER' provider，不是运行时实际调用点

src/app/core/startup/startup.service.ts
  -> deferred init setTimeout(...)
  -> _initPlugins()
     -> wait isAllDataLoadedInitially$
     -> wait afterCurrentSyncDoneOrSyncDisabled$
     -> PluginService.initializePlugins()

src/app/plugins/plugin.service.ts
  -> discover built-in + uploaded manifests
  -> load enabled plugins
  -> activatePlugin()
  -> _fireOnReady()

真实源码锚点

// src/app/plugins/plugin-initializer.ts
export const PLUGIN_INITIALIZER_PROVIDER = {
  provide: 'PLUGIN_INITIALIZER',
  useFactory: () => initializePlugins,
  multi: true,
};

// src/app/core/startup/startup.service.ts
await Promise.all([
  this._dataInitStateService.isAllDataLoadedInitially$.pipe(take(1)).toPromise(),
  this._syncWrapperService.afterCurrentSyncDoneOrSyncDisabled$.pipe(take(1)).toPromise(),
]);
await this._pluginService.initializePlugins();

三个判断点

你看到的现象先看哪里判断
Angular provider 里有 PLUGIN_INITIALIZER_PROVIDER src/main.ts
src/app/plugins/plugin-initializer.ts		 这是 provider token 接线；本课程只把它当成可 grep 的历史/扩展入口。
插件要等数据和同步后才出现 src/app/core/startup/startup.service.ts 真实初始化在 _initPlugins()；它等待初始数据和当前 sync 完成或禁用。
manifest 已发现但代码未运行 src/app/plugins/plugin.service.ts 只有 enabled 插件才进入 activatePlugin()；nodeExecution 插件还要拿 grant。

异常路径

_initPlugins() 捕获初始化错误并记录日志，不能让插件初始化失败阻断整个应用启动。单个插件加载失败时，状态会落在插件服务自己的 loading/error 分支，而不是回滚 Angular bootstrap。

60 秒判断题

如果插件没有在冷启动第一时间出现，你会先查 PLUGIN_INITIALIZER_PROVIDER 是否被 Angular 执行，还是查 StartupService._initPlugins() 的等待条件？

参考答案与反馈

先查 _initPlugins()。源码里 provider 只是把 initializePlugins 暴露为 token；真实执行路径等待数据 hydration 和当前 sync 完成后才调用 PluginService.initializePlugins()。
