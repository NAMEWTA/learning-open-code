插件系统、插件 API 与运行时加载

一句话：插件系统让外部代码通过受控 API 扩展 super-productivity 的任务、UI、hooks、OAuth、Node 执行和 issue provider 能力。它连接 L0 全局地图、Angular L1、Electron L1 和 Issue L1，但边界各不相同。

1. 先分清两条初始化线

src/main.ts 里有 PLUGIN_INITIALIZER_PROVIDER，但它只是把 'PLUGIN_INITIALIZER' token 放进 Angular DI provider 表；当前插件运行时并不是靠 Angular APP_INITIALIZER 执行。真正链路在 src/app/core/startup/startup.service.ts 的 _initPlugins：根组件触发 StartupService.init() 后，延迟任务等待初始数据加载和当前同步结束，再调用 PluginService.initializePlugins()。

Angular provider token
  src/main.ts → PLUGIN_INITIALIZER_PROVIDER
  作用：DI 接线点

runtime plugin 初始化
  AppComponent → StartupService.init()
  → deferred init → _initPlugins()
  → wait data init + sync done
  → PluginService.initializePlugins()

2. 发现、加载、执行不是一回事

PluginService.initializePlugins() 先发现内置插件和上传插件，只读取 manifest、icon、缓存元数据和 enabled 状态；随后只激活 enabled 插件。激活时才由 PluginLoaderService 拉取插件 manifest、plugin.js、可选 index.html、icon 与 i18n，再交给 PluginRunner 用 new Function 执行代码。插件拿到的不是 Angular service，而是 PluginAPI；它再通过 PluginBridgeService 访问任务、项目、标签、UI、同步、OAuth、secret 和白名单 NgRx action。

3. 桌面能力要回主进程

OAuth 和 Node 执行最容易误判。OAuth 的 token 管理由 renderer 服务协调，但 Electron 桌面上 loopback server、系统浏览器打开和回调转发在 electron/plugin-oauth.ts。Node 执行更严格：preload 只暴露一次性 consumePluginNodeExecutionApi()，renderer 的 PluginBridgeService 先取得主进程 grant token，真正脚本执行在 electron/plugin-node-executor.ts，并绑定 webContents、native consent dialog、session grant 和可选上传插件持久 consent。

plugin code
  → PluginAPI.startOAuthFlow() / executeNodeScript()
  → PluginBridgeService / PluginOAuthBridgeService
  → window.ea controlled bridge
  → Electron main process
  → loopback OAuth server or Node executor

4. Issue provider 是适配层

插件扩展外部工作项时，不会绕过 Issue L1 的通用链路。插件调用 registerIssueProvider() 后，PluginIssueProviderRegistryService 生成 plugin:<id> 或迁移用 key；PluginIssueProviderAdapterService 把 definition 适配成 IssueServiceInterface，让搜索、导入、刷新继续走 issue feature。桥接阶段只在 definition 提供 createIssue、deleteIssue 或 fieldMappings + updateIssue 时立即注册 two-way sync adapter；Issue L1 的 IssueSyncAdapterResolverService 还会为只有 updateIssue 的 provider 按需懒创建 adapter，因此排查时要分清“是否有 adapter”和“是否有可同步字段”。

暂停 60 秒判断：一个 Google Calendar 插件无法完成授权，另一个 GitHub issue 插件能搜索但不会写回本地任务改动。你先查哪两条链路？

参考答案：授权失败先查 PluginOAuthBridgeService → PluginOAuthService → Electron plugin-oauth.ts 或平台弹窗路径；不要先查 issue polling。能搜索但不写回，先看插件 definition 是否声明可映射的 fieldMappings，再看是否有 updateIssue 支持字段推送；如果只是 calendar/event 类直接写回，IssueSyncAdapterResolverService 可能会为 updateIssue-only provider 懒创建 adapter。完整入口清单见 插件系统参考。主要来源：src/app/core/startup/startup.service.ts、src/app/plugins/plugin.service.ts、src/app/plugins/plugin-bridge.service.ts。如果某个边界仍不清楚，直接向 agent 追问下一跳源码。
