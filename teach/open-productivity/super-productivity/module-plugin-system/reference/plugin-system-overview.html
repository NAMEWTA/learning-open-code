插件系统、插件 API 与运行时加载参考

职责一句话：在 Angular renderer 中发现、启用、加载和卸载插件，用公开 @super-productivity/plugin-api 合同隔离插件作者视角，并在需要桌面系统能力时通过受控 bridge 进入 Electron 主进程。

架构位置：本模块接在 L0 总览的外部系统扩展层；Angular L1 负责应用装配，Electron L1 负责主进程和 IPC 边界，Issue L1 负责外部工作项通用链路。

职责边界

层级负责什么不负责什么
Angular provider 接线PLUGIN_INITIALIZER_PROVIDER 把 token 放入 DI provider 表。不等于实际插件初始化；不要把它当作当前 APP_INITIALIZER 执行链。
Startup runtime等待数据和同步完成后调用 PluginService.initializePlugins()。不读取 manifest 细节，也不执行插件代码。
PluginService发现 manifest、维护状态、激活 enabled 插件、清理插件运行时。不直接实现每个插件 API 的业务动作；那些在 bridge 或业务 service 中。
PluginLoader/Runner加载资源并执行插件入口脚本，管理 ready/unload callback。不判断 issue provider 业务语义，也不拥有 Electron 系统权限。
PluginAPI/Bridge公开 API、绑定 pluginId、命名空间化持久化、限制 action dispatch。不让插件直接访问 Angular DI 容器或任意 NgRx action。
Electron 主进程OAuth loopback、系统浏览器、Node grant、VM/child process 执行。不承载插件业务状态；renderer 通过 window.ea 受控进入。
Issue provider adapter把插件 definition 翻译成 issue search/import/refresh/sync 能力。不替代 Issue L1 的轮询、导入、two-way-sync effects。

阅读示例

  1. 插件冷启动没出现：先看 src/app/core/startup/startup.service.ts 的 _initPlugins 是否等到数据加载和同步完成，再看 PluginService.initializePlugins() 是否发现 manifest，最后看 enabled 状态是否让 activatePlugin() 执行。
  2. 插件按钮存在但点击没效果：从 PluginRunner 的 menu/sidePanel 注册看起，再进 PluginBridgeService 的注册数组和 UI 组件读取 signal 的位置。
  3. OAuth 失败：先看 PluginOAuthBridgeService.startOAuthFlow() 选择了 Electron、native 还是 web 分支；Electron 下继续查 plugin-oauth.ts 的 loopback port、HTTPS auth URL 和 PLUGIN_OAUTH_CB。
  4. Node 执行失败：先确认 manifest 有 nodeExecution permission、运行在 Electron、renderer 持有 grant token；再查主进程 PLUGIN_REQUEST_NODE_EXECUTION_GRANT 和 PLUGIN_EXEC_NODE_SCRIPT。
  5. 插件 provider 搜索正常但不写回：先分辨是字段同步还是直接事件写回。任务字段写回需要 definition 同时提供可用的 fieldMappings 和 updateIssue；而只有 updateIssue 的 provider 仍可能由 IssueSyncAdapterResolverService 懒创建 adapter，用于 calendar move 这类直接推送。
