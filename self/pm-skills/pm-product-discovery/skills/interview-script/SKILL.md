## 客户访谈脚本

创建结构化的访谈脚本，挖掘真实洞察而非仅是用户意见。遵循《The Mom Test》原则——询问他们的生活，而非你的创意。

### 领域背景

客户访谈是持续发现**第一阶段（探索）**的来源之一。其他来源：利益相关方访谈、使用分析、数据分析、问卷、市场趋势、SEO/SEM 分析。PM 需要直接接触用户、利益相关方、工程师和设计师——「没有中间代理」。**产品三人组**（PM + 设计师 + 工程师——Teresa Torres）应共同参与发现，而非仅 PM 一人。

### 背景

你正在为 **$ARGUMENTS** 的研究准备客户访谈脚本。

如果用户提供了文件（用户画像、假设列表、产品简报或以往的访谈笔记），请先阅读。

### 操作指引

1. **明确研究目标**：
   - 团队需要回答什么具体问题？
   - 这项研究将为哪些决策提供信息？
   - 哪些假设需要验证？

2. **创建访谈脚本**，包含以下部分：

   ### 开场（2-3 分钟）
   - 介绍自己和目的（学习而非推销）
   - 设定期望：「没有对错答案。我们是来向你的经验学习的。」
   - 请求录音许可（如适用）
   - 确认可用时间

   ### 热身：背景与情境（5 分钟）
   - 「告诉我你的角色是什么，典型的一天/一周是怎样的。」
   - 「你做[与产品领域相关的活动]多久了？」
   - 目标：建立融洽关系并理解他们的背景

   ### 核心探索：待完成工作（15-20 分钟）

   **当前状况与行为**（过去时，具体实例）：
   - 「带我回顾一下你上一次[做我们正在探索的事]。发生了什么？」
   - 「你使用了什么工具或方法？」
   - 「花了多长时间？还有谁参与了？」

   **痛点与挫败感**（观察而非引导）：
   - 「这件事最困难的部分是什么？」
   - 「如果你能挥一下魔杖，什么会改变？」
   - 「你尝试过什么来解决这个问题？结果如何？」

   **期望成果**（用他们的话，而非你的）：
   - 「对你来说，在这个方面'好'是什么样子？」
   - 「你怎么知道这件事运作得顺利？」

   **支付意愿/优先级**（实际投入）：
   - 「你目前在这上面花了多少时间/金钱？」
   - 「你是否找过更好的解决方案？找到了什么？」
   - 「你愿意放弃什么来解决这个问题？」

   ### 探究技巧
   当遇到有趣的话题时使用：
   - **「多跟我说说这个」** — 打开任何话题
   - **「为什么？」**（温和地问，2-3 次） — 触及根本原因
   - **「能给我一个具体的例子吗？」** — 从意见转向事实
   - **「后来发生了什么？」** — 跟随故事走向
   - **「这让你感觉如何？」** — 捕捉情感强度

   ### The Mom Test 规则
   - 询问**他们的生活**，而非你的创意
   - 询问**过去**，而非未来（「你会用 X 吗？」是无用的问题）
   - **少说多听** — 目标是 80/20 分配
   - 访谈过程中**绝不推销**
   - 寻找**强烈的情感** — 它们意味着真实的痛苦或愉悦
   - **赞美只是噪音** — 「听起来很酷！」对你不提供任何信息

   ### 总结（3-5 分钟）
   - 「还有什么我没问到但你觉得重要的事情吗？」
   - 「关于这个话题，我还应该和谁聊？」
   - 感谢他们的时间
   - 分享后续步骤（如有）

3. **定制脚本**：使问题适配具体产品领域、用户画像和研究目标。根据可用访谈时间添加或删除部分。

4. **包含笔记模板**：
   ```
   参与者：[姓名 / ID]
   日期：[日期]
   关键工作：[他们试图达成什么]
   当前解决方案：[他们今天使用什么]
   最大痛点：[他们的 №1 挫败感]
   期望成果：[成功的样子]
   支付意愿：[他们投入了多少 / 愿意投入多少]
   意外发现：[一些意想不到的事情]
   后续跟进：[下一步]
   ```

保存为 markdown。同时包含脚本和笔记模板。
