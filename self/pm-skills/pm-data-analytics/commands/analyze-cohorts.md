# /analyze-cohorts -- 同期群分析

按同期群分析用户留存和参与度模式。上传数据或描述需求，获取留存曲线、功能采用趋势和可操作的洞察。

## 调用方式

```
/analyze-cohorts [上传用户活动数据的 CSV 文件]
/analyze-cohorts 1-6 月注册用户的月度留存率，按获客渠道分组
/analyze-cohorts 帮我对引导流程重新设计进行同期群分析
```

## 工作流

### 第 1 步：接收数据或定义分析需求

两种路径：
- **有数据**：用户上传包含用户级别数据的 CSV/电子表格（user_id、signup_date、activity_date、event_type 等）
- **无数据**：用户描述分析需求 → 生成 SQL 查询和分析框架

### 第 2 步：定义同期群

询问：
- 什么定义了一个同期群？（注册周/月、获客渠道、套餐层级、首次使用的功能）
- 什么是留存事件？（登录、核心操作、任何活动、购买）
- 什么时间粒度？（每日、每周、每月）
- 什么时间范围？

### 第 3 步：分析

应用 **cohort-analysis** skill：

**如提供了数据：**
- 使用 Python（pandas）处理数据以创建同期群表
- 计算每个同期群每个周期的留存率
- 生成留存曲线
- 识别模式：改善/恶化的同期群、季节性效应、异常情况
- 比较各同期群的功能采用情况

**如描述分析需求：**
- 设计同期群分析框架
- 生成提取数据的 SQL 查询
- 创建分析模板电子表格
- 定义指标和可视化方法

### 第 4 步：生成报告

留存表、关键发现、同期群对比、留存基准、建议、后续查询。

### 第 5 步：提供后续建议

- "需要我**按另一个维度进一步细分**吗？"
- "需要我基于这些留存阈值**设置指标告警**吗？"
- "需要我**设计实验**以改善最弱同期群的留存吗？"

## 注意事项

- 同期群分析的好坏取决于留存事件的定义——推动使用有意义的操作，而非仅是「登录了」
- 早期同期群通常因创始用户偏差而看起来不同——比较时需注意
- 季节性效应可能伪装成趋势——标注同期群差异是否可能由日历驱动
