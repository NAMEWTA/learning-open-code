# 交付文档集：使 AI 构建的代码可审查的文档

## 目的

AI 代理能快速编写代码，但不会留下*意图*的持久记录——系统应该做什么、谁被允许做什么、密钥存放在哪里、哪些规则真正得到了验证。没有这些记录，没有人（也没有审计代理）能判断代码是否可以安全交付。本 skill 定义了恢复可审查性所需的小型文档集。

这些文档存放在 `/documentation/` 目录下，面向两类读者：人类审核者和下一个 AI 编码代理。它们是每次后续审计的**预期状态**（intended-state）部分——安全或性能审查的好坏，取决于它能否将代码与一份意图进行比对。

## 文档集的组织方式

该文档集**不是**一个固定列表——它由一个小型**核心**加上**条件性**文档组成，仅在对应能力存在时才添加。

- **核心文档**——每个可审查的应用都有这些面，因此始终产生产出。
- **条件性文档**——仅当应用实际具有该能力时才包含。如果没有，在 `architecture.md` 中写一行（「无定时任务——无需 `cron.md`。」）而非凭空创建一份空文档。可审查性来自一份诚实的映射，「我们不做 X」也是映射的一部分。
- 大多数文档由 `/document-app` 从代码逆向生成。唯一的例外是 `tests.md`，它由 `/derive-tests` *从其他文档推导*而来——它是验证映射，而非对某个子系统的描述。

对当前状态要极其坦诚但不必偏执。目标是一份准确的映射，而非一张干净的体检报告。每份文档简短、以表格和要点为主，跳过泛泛的理论。

## 核心文档

每个条目：文件名 · 一句话目的 · 必须涵盖的内容 · 审核者如何使用它。

1. **`architecture.md`**——系统是什么以及如何组织在一起。
   - 必须涵盖：产品概述 + 关键假设；技术栈；auth/session/claims 端到端流转方式；信任边界（如服务角色 vs 客户端）；简短的**已知风险/假设**列表（每项附代码中出现的位置，而非通用检查清单）；所有已产出文档的「相关文档」索引。
   - 审核者用途：根文档——其他所有文档都通过此处交叉引用。

2. **`flows.md`**——权限和副作用被实际执行的旅程。
   - 必须涵盖：每条承载业务关键功能的流程为 角色 + 前置条件 + 成功结果；跨 UI → 服务器 → 数据 → 作业 → 服务商 → 代理的分步序列；**每个受保护步骤的鉴权检查**（哪个 claim/role/scope，针对哪个资源，以及预期的*拒绝*情况）；**信任边界穿越**（浏览器→服务器、服务器→服务商、作业→应用、代理→工具、webhook→应用）；每个步骤造成的状态变更和副作用（写操作、邮件入队、作业触发、外呼调用）。
   - 审核者用途：静态 `permissions.md` 矩阵无法展示的运行时视图——授权在*哪里*、以*什么顺序*被强制执行，以及在哪里可能被跳过。
   - **反 PRD 规则：**不涉及权限、数据完整性、外部副作用、资金、隐私或运维安全的流程不属于此处。这是一份安全/运维地图，而非功能规格说明。

3. **`permissions.md`**——谁被允许做什么。
   - 必须涵盖：角色/claims；scope 的来源（token vs DB）；资源 × 操作 × 角色矩阵；哪些表使用行级安全（RLS），哪些依赖代码强制检查。
   - 审核者用途：访问控制审计将代码与之比对的基线。`flows.md` 展示其动态运行；本文档是静态参考。

4. **`variables.md`**——配置和密钥，按风险映射。
   - 必须涵盖：名称 · 使用者 · 作用域（server/client）· 来源 · 轮换 · 风险 的表格；明确确认无密钥打包到客户端；上线前检查清单。
   - 审核者用途：密钥/PII 泄露面和应急响应期间的轮换计划。

5. **`tests.md`**——验证映射：哪些已文档化的规则被实际检查、哪些仅为建议、哪些完全没有检查。
   - 必须涵盖，分为三个清晰分隔的部分以避免错误地显示为全绿：
     - **现有覆盖**——仓库中*当前*存在的测试，每条关联到其验证的规则（映射反映现实，而非愿望清单）。
     - **建议测试**——尚未编写的推荐用例，按**测试类型**（自动化单元/集成 · 受保护实时 · 人工审查）标注。
     - **缺口**——已文档化但无任何验证的规则，按跨越缺口所暴露的影响排序。
   - 每行包含：用例 → 规则 → 预期行为（包括拒绝/负面情况）→ 证据来源（文档 + 代码）→ 状态（现有 / 建议 / 无）。同时标注哪些检查是 CI 必需的，阻止合入 `main`。
   - 审核者用途：「文档化 == 已实现」的可操作形式——展示其他文档声称的每条规则当前是否被某个测试覆盖、仅被建议、或未经验证。
   - 由 `/derive-tests`（而非 `/document-app`）生成，因为它是从其他文档和现有测试套件*推导*而来，而非从某个子系统读取。

## 条件性文档（仅在对应能力存在时纳入）

6. **`emails.md`**——系统发送的所有通知。*仅当应用发送事务性或自动化邮件时才纳入。*
   - 必须涵盖：队列 → 处理器 → 服务商路径；模板及其接受的变量；重试/退避行为；发送失败时从哪里排查。
   - 审核者用途：发现未经验证的模板输入和 PII 暴露边界。

7. **`cron.md`**——所有定时工作及如何安全运维。*仅当存在定时或后台作业时才纳入。*
   - 必须涵盖：作业清单表格（作业 → 调度 → 函数 → 密钥 → 限制 → 重试）；每个作业如何保持幂等性；内部调用如何进行身份验证；从哪里查看最近运行记录。
   - 审核者用途：发现可伪造的触发器和不受限的后台作业。

8. **`seo.md`**——单页应用如何处理 SEO 和社交预览。*仅当存在公开/可索引或面向爬虫的路由时才纳入。*
   - 必须涵盖：预览方案（静态 meta / 预渲染 / 边缘 HTML）；路由 → 需要 SEO → 仅公开数据 的表格；动态元数据如何被清理；爬虫 vs 人类路由分流。
   - 审核者用途：发现仅公开数据的违规行为和爬虫路由上的元数据注入。

9. **`automation.md`**——嵌入式代理和其他自动化路径。*仅当应用内嵌 AI 代理、LLM 工作流、工具调用、webhook 或外部自动化时才纳入。*
   - 必须涵盖，每个自动化/代理：触发器 + 所有者 + 是否自动运行或仅在批准后运行；可读取的输入以及**可调用的确切工具/API**（工具面本身就是一道硬护栏）；**导向**（prompt）所在位置 vs **非 prompt 硬护栏**；返回应用的**输出契约**（schema、验证、失败处理）；**应用拥有的副作用 vs 代理建议的操作**；以及控件——审批门禁、审计/时间线日志、速率限制、重试、熔断开关。
   - 审核者用途：使隐藏的自动化路径可见，并划清代理*建议*什么与应用*强制执行*什么之间的界限——这是现代 AI 构建应用中风险最高的层面。

## 注意事项

- 每份产出的文档在 `architecture.md` 的「相关文档」部分添加对自身的引用，使文档集保持可被发现。
- 跳过任何不适用的条件性文档，用一行说明而非凭空创建内容。
- 将示例和成品模板排除在这些文档之外——它们描述的是*本*系统，而非通用方法。
- 代理操作上下文文件（`CLAUDE.md` / `AGENTS.md`）是*不同*的产物——是从这些文档推导出的指令，而非系统文档。它由 `/ship-check` 在交接步骤中产出，而非此处。
- `tests.md` 由 `/derive-tests` 产出；其余由 `/document-app` 产出。
- 不要包含「更新日期」行；文件的 git 历史是权威来源。
