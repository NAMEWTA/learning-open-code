# /derive-tests -- 将意图转化为测试

文档说明了系统*应该*做什么。审计发现代码*没有*做什么。测试是阻止该缺口在下一次 AI 编辑后重新打开的手段。

## 前置条件：文档化的意图

测试从文档推导而来，所以文档先行。如果 `/documentation/*.md` 缺失或单薄，先运行 `/document-app`。

## 工作流

### 1. 读取意图和已有测试

读取适用的系统文档，应用 **shipping-artifacts** 和 **intended-vs-implemented** skills。然后盘点**现有测试套件**——测试文件、实际断言了什么、CI 中运行了什么。

### 2. 提取值得测试的规则

提取关键、确定性的规则——那些违反后会跨越信任、数据、资金、租户或隐私边界的规则。

### 3. 构建覆盖地图

每行一个用例：规则 → 预期行为（含否定情况）→ 证据来源（文档 + 代码）→ 测试类型 → 状态（现有/建议/无）。

测试类型：**单元测试**、**集成测试（确定性）**、**受保护实时测试**、**手动测试**。

### 4. 提出测试建议

对每条规则写出测试用例：名称、准备/操作/断言意图以及必须拒绝的否定情况。

### 5. 推荐 CI 门禁

推荐匹配仓库技术栈和现有工具的 CI 设置：每次 PR 运行确定性本地测试集、受保护实时测试保持可选、通过 required status check + 分支保护门禁合入 main。

### 6. 报告覆盖和缺口

在 `tests.md` 中分为三个清晰分隔的部分：现有覆盖、建议测试、缺口。

## 注意事项

- 这是「文档化 == 已实现」的验证部分：审计发现今天的缺口，这些测试阻止它明天重新出现。
- 不要为了制造覆盖而捏造规则。如果文档保持沉默，缺口在文档中。
