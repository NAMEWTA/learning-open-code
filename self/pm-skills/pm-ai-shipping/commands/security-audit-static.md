# /security-audit-static -- 审计你已有的代码

针对 AI 构建代码的聚焦、自包含的安全审计。保持一个小型、持久的引擎——映射边界，对照实现检查意图，在报告前自我反驳——并拒绝输出任何无法用引用证据支撑的内容。

## 审计（小型引擎，强约束）

### 1. 将入口点映射到信任边界和漏洞接收点

优先考虑召回——完整读取范围内的每个文件，然后 grep handler、route、RPC 和共享辅助函数名称。

### 2. 检查四条高价值路径

授权、数据访问、会话/身份和输入→输出编码。比较同级 handler——如果一个执行了检查而另一个省略了，省略就是一个发现。

### 3. 交叉引用预期 vs 实现

对照 `/documentation/*.md` 应用 **intended-vs-implemented** skill。文档记载但代码中未强制执行的规则本身就是一个发现。

### 4. 自我反驳每个候选发现

对每个发现，尝试证伪它。默认**保留**，除非找到引用证据（文件 + 行号）证明存在真正的清理器/编码器/验证器/授权检查在接收点阻止了漏洞利用。

明确攻击者和受害者。**永远不要对 SSRF/出站网络接收点、共享计费、数据暴露、跨租户/跨主体流程或服务端执行/渲染应用攻击者等于受害者的反驳。**

### 5. 仅报告能幸存下来的

## 输出

按文件分组、按严重程度排序，标准格式：
```
<file>:
  N. [严重程度] [类别] <位置>
     风险等级 | 攻击场景 | 影响 | 解决方案
```

## 注意事项

- 不要报告没有具体影响的通用加固建议。
- 本命令仅覆盖安全。过度获取、索引和缓存使用 `/performance-audit-static`。
- 端到端文档+审计+交付包使用 `/ship-check`。
