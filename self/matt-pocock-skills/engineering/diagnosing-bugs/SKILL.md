# 诊断 Bug

一种针对疑难 bug 的规范。仅在明确有理由时才跳过阶段。

在探索代码库时，阅读 `CONTEXT.md`（如果存在）以获取相关模块的清晰心智模型，并检查你接触区域中的 ADR。

## 阶段 1 — 构建反馈循环

**这就是技能本身。** 其他一切都是机械性的。如果你有一个针对此 bug 的**紧凑**通过/失败信号——一个在*这个* bug 上变红的信号——你就会找到原因；二分法、假设测试和插桩都只是消费它。如果你没有一个，盯再多的代码也救不了你。

在此投入不成比例的精力。**要果断。要创新。拒绝放弃。**

### 构建方法——大致按此顺序尝试

1. **失败测试**——在触及 bug 的任何接缝处——单元、集成、端到端。
2. **Curl / HTTP 脚本**——针对运行中的开发服务器。
3. **CLI 调用**——使用夹具输入，将 stdout 与已知正确的快照进行 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，在 DOM/控制台/网络上断言。
5. **重放捕获的追踪。** 将真实的网络请求/负载/事件日志保存到磁盘；在隔离中通过代码路径重放它。
6. **抛弃型测试架。** 启动系统的最小子集（一个服务、mock 的依赖），通过单个函数调用执行 bug 代码路径。
7. **属性/模糊循环。** 如果 bug 是\"有时输出错误\"，运行 1000 次随机输入并寻找失败模式。
8. **二分测试架。** 如果 bug 出现在两个已知状态之间（提交、数据集、版本），自动化\"在状态 X 启动、检查、重复\"，以便你可以 `git bisect run` 它。
9. **差分循环。** 通过旧版本 vs 新版本（或两种配置）运行相同输入并 diff 输出。
10. **HITL bash 脚本。** 最后手段。如果必须由人类点击，用 `scripts/hitl-loop.template.sh` 驱动*他们*，使循环仍然结构化。捕获的输出反馈给你。

构建正确的反馈循环，bug 就修复了 90%。

### 收紧循环

将循环视为产品。一旦你有了*一个*循环，**收紧**它：

- 我能让它更快吗？（缓存设置、跳过无关初始化、缩小测试范围。）
- 我能让信号更锐利吗？（断言具体症状，而非\"没崩溃\"。）
- 我能让它更确定吗？（固定时间、种子 RNG、隔离文件系统、冻结网络。）

一个 30 秒的不稳定循环几乎不比没有循环好；一个 2 秒的确定性循环是紧凑的——一种调试超能力。

### 非确定性 bug

目标不是干净的复现，而是**更高的复现率**。循环触发 100 次，并行化，增加压力，缩小时间窗口，注入 sleep。一个 50% 不稳定的 bug 是可调试的；1% 不行——持续提高比率直到可调试。

### 当你确实无法构建循环时

停下来并明确说明。列出你尝试了什么。向用户请求：(a) 访问能复现它的任何环境，(b) 一个捕获的产物（HAR 文件、日志转储、核心转储、带时间戳的屏幕录制），或 (c) 添加临时生产插桩的权限。在没有循环的情况下，**不要**继续假设。

### 完成标准——一个能变红的紧凑循环

阶段 1 在循环**紧凑**且**能变红**时完成：你能说出**一条命令**——一个脚本路径、一个测试调用、一个 curl——你**至少已经运行过一次**（粘贴调用及其输出），并且：

- [ ] **能变红**——它驱动实际的 bug 代码路径并断言**用户的确切症状**，因此它能在这个 bug 上变红，修复后变绿。不是\"运行无错误\"——它必须能够*捕获这个具体的 bug*。
- [ ] **确定性**——每次运行相同结果（不稳定的 bug：按上述固定、高复现率）。
- [ ] **快速**——秒级，而非分钟级。
- [ ] **代理可运行**——你可以无人值守运行；人类参与仅通过 `scripts/hitl-loop.template.sh`。

如果你发现自己在读代码构建理论之前这条命令还不存在，**停——直接跳到假设正是此技能要防止的失败。** 没有能变红的命令，就没有阶段 2。

## 阶段 2 — 复现 + 最小化

运行循环。观察它变红——bug 出现。

确认：

- [ ] 循环产生**用户**描述的失败模式——不是恰好在附近的另一个不同失败。错误的 bug = 错误的修复。
- [ ] 失败在多次运行中可复现（或者，对于非确定性 bug，以足够高的比率可复现以进行调试）。
- [ ] 你已捕获确切的症状（错误消息、错误输出、缓慢计时），以便后续阶段能验证修复确实解决了它。

### 最小化

一旦变红，将复现缩小到**仍能变红的最小场景**。**一次一个**地削减输入、调用者、配置、数据和步骤，每次削减后重新运行循环——只保留对失败关键的要素。

为什么值得：最小复现在阶段 3 中缩小了假设空间（剩下更少的可怀疑移动部件），并在阶段 5 中成为干净的回归测试。

当**每个剩余元素都是关键的**时完成——移除其中任何一个都会使循环变绿。

在复现**且**最小化之前不要继续。

## 阶段 3 — 假设

在测试任何假设之前生成 **3-5 个排序的假设**。单一假设生成会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它做出的预测。

> 格式：\"如果 <X> 是原因，那么 <改变 Y> 将使 bug 消失 / <改变 Z> 将使其恶化。\"

如果你不能陈述预测，这个假设就是个感觉——丢弃或锐化它。

**在测试之前向用户展示排序列表。** 他们通常有能立即重新排序的领域知识（\"我们刚部署了对 #3 的更改\"），或者知道他们已经排除的假设。廉价的检查点，巨大的时间节省。不要在此阻塞——如果用户不在线，按你的排序继续。

## 阶段 4 — 插桩

每个探测必须映射到阶段 3 的一个具体预测。**一次改变一个变量。**

工具偏好：

1. **调试器 / REPL 检查**——如果环境支持。一个断点胜过十条日志。
2. **针对性日志**——在区分假设的边界处。
3. 绝不\"记录一切然后 grep\"。

**用唯一前缀标记每条调试日志**，例如 `[DEBUG-a4f2]`。最后的清理变成一条 grep。未标记的日志保留；标记的日志移除。

**性能分支。** 对于性能回归，日志通常是错误的。相反：建立基线测量（计时测试架、`performance.now()`、性能分析器、查询计划），然后二分。先测量，后修复。

## 阶段 5 — 修复 + 回归测试

在修复**之前**编写回归测试——但仅当存在**正确的接缝**时。

正确的接缝是指测试在调用点处运行**真实的 bug 模式**的地方。如果唯一可用的接缝太浅（bug 需要多个调用者的单调用者测试、无法复制触发 bug 的链条的单元测试），那里的回归测试会给出虚假信心。

**如果不存在正确的接缝，这本身就是发现。** 记录下来。代码库架构正在阻止 bug 被锁定。将此标记给下一阶段。

如果存在正确的接缝：

1. 将最小化的复现转化为该接缝处的失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. 针对原始（未最小化的）场景重新运行阶段 1 反馈循环。

## 阶段 6 — 清理 + 事后分析

声明完成前必须完成：

- [ ] 原始复现不再复现（重新运行阶段 1 循环）
- [ ] 回归测试通过（或接缝缺失已记录）
- [ ] 所有 `[DEBUG-...]` 插桩已移除（`grep` 前缀）
- [ ] 抛弃型原型已删除（或移动到明确标记的调试位置）
- [ ] 被证明正确的假设在提交/PR 消息中陈述——以便下一个调试者学习

**然后问：什么能预防这个 bug？** 如果答案涉及架构变更（没有好的测试接缝、纠缠的调用者、隐藏的耦合），将具体情况交接给 `/improve-codebase-architecture` 技能。在修复**之后**提出建议，而非之前——你现在比开始时拥有更多信息。
