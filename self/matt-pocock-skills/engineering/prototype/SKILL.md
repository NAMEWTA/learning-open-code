# 原型

一个原型是**回答一个问题的抛弃型代码**。问题决定形态。

## 选择分支

识别正在回答哪个问题——从用户的提示词、周围代码中，或在用户在线时询问：

- **"这个逻辑/状态模型感觉对吗？"** → [LOGIC.md](LOGIC.md)。构建一个小型交互式终端应用，将状态机推过难以在纸面上推理的场景。
- **"这应该是什么样子？"** → [UI.md](UI.md)。在一个路由上生成几种截然不同的 UI 变体，可通过 URL 搜索参数和浮动底栏切换。

两个分支产生非常不同的产物——搞错会浪费整个原型。如果问题确实模糊且用户不在线，默认选择更匹配周围代码的分支（后端模块 → 逻辑；页面或组件 → UI），并在原型顶部声明假设。

## 两者通用的规则

1. **从第一天起就是抛弃型的，并明确标记。** 将原型代码放在靠近它实际使用位置的地方（在它正在为之制作原型的模块或页面旁边），以便上下文显而易见——但命名要让偶然的读者能看出它是原型，而非生产代码。对于抛弃型 UI 路由，遵循项目已有的路由约定；不要发明新的顶层结构。
2. **一条命令运行。** 无论项目现有的任务运行器支持什么——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能够不假思索地启动它。
3. **默认无持久化。** 状态存在于内存中。持久化是原型正在*检查*的东西，而不是它应该依赖的东西。如果问题明确涉及数据库，使用一个临时数据库或本地文件，带有清晰的"PROTOTYPE——请清除我"的名称。
4. **跳过润色。** 没有测试，没有超出使原型*可运行*所需之外的错误处理，没有抽象。重点是快速学到东西然后删除它。
5. **展示状态。** 每次操作后（逻辑）或每次变体切换时（UI），打印或渲染完整的相关状态，以便用户看到什么发生了变化。
6. **完成后删除或吸收。** 当原型回答了它的问题时，要么删除它，要么将验证过的决策合并到真实代码中——不要让它烂在仓库里。

## 完成后

从原型中唯一值得保留的是*答案*。将其捕获到某个持久的地方（提交消息、ADR、issue 或原型旁边的 `NOTES.md`）以及它所回答的问题。如果用户在线，这个捕获是一次快速对话；如果不在，留下占位符以便他们（或你，在下一次）可以在删除原型之前填写结论。
