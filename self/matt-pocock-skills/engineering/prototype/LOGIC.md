# 逻辑原型

一个小型交互式终端应用，让用户手动驱动状态模型。当问题涉及**业务逻辑、状态转换或数据形态**时使用——这类问题在纸面上看起来合理，但只有当你用真实案例驱动它时才会发现问题。

## 何时适合使用此方式

- "我不确定这个状态机是否处理了 X 然后 Y 的边界情况。"
- "这个数据模型是否真的能让我表示……的情况？"
- "我想在写代码之前感受一下 API 应该长什么样。"
- 任何用户想要**按按钮并观察状态变化**的场景。

如果问题是"这应该长什么样"——走错分支了。请使用 [UI.md](UI.md)。

## 流程

### 1. 明确问题

在写代码之前，写下你要原型化的状态模型和要回答的问题。一段话，放在原型的 README 中或文件顶部的注释里。回答错误问题的逻辑原型纯属浪费——把问题明确写出来，以便后续检查，无论用户是在现场查看还是离线回看。

### 2. 选择语言

使用宿主项目所用的语言。如果项目没有明显的运行时（例如文档仓库），则询问用户。

匹配项目现有的工具链约定——不要仅为原型添加新的包管理器或运行时。

### 3. 将逻辑隔离在可移植模块中

将实际逻辑——即回答问题的那部分——放在一个小型、纯净的接口后面，这个接口可以被提取出来，稍后直接放入真实代码库中。TUI 外壳是一次性的；逻辑模块不应该是。

正确的形态取决于问题：

- **纯 reducer**——`(state, action) => state`。适用于动作为离散事件、状态为单一值的情况。
- **状态机**——显式状态和转换。适用于"哪些操作在当前状态下是合法的"本身就是问题的一部分。
- **一组纯函数**操作普通数据类型。适用于没有隐式当前状态——只有转换的情况。
- **具有清晰方法接口的类或模块**，适用于逻辑确实拥有持续内部状态的情况。

选择最适合所问问题的形态，*而非*最容易接入 TUI 的形态。保持纯净：无 I/O、无终端代码、无用于控制流的 `console.log`。TUI 导入并调用它；没有反向流动。

这正是原型在其生命周期之后仍有价值的原因。当问题得到回答后，经过验证的 reducer / 状态机 / 函数集可以被提取到真实模块中——TUI 外壳则被删除。

### 4. 构建最小的 TUI 来暴露状态

将其构建为**轻量级 TUI**——每个 tick，清屏（`console.clear()` / `print("\033[2J\033[H")` / 等效操作）并重新渲染整个帧。用户应始终看到一个稳定的视图，而不是不断增长的滚动历史。

每帧有两个部分，按以下顺序排列：

1. **当前状态**，美化和便于 diff 的格式（每字段一行，或格式化的 JSON）。使用**粗体**表示字段名或节标题，使用**暗淡**表示不太重要的上下文（时间戳、ID、派生值）。原生 ANSI 转义码即可——`\x1b[1m` 粗体，`\x1b[2m` 暗淡，`\x1b[0m` 重置。无需引入样式库，除非项目中已有。
2. **键盘快捷键**，列在底部：`[a] add user  [d] delete user  [t] tick clock  [q] quit`。加粗按键，淡化描述，或反之——怎么清晰怎么来。

行为：

1. **初始化状态**——单个内存对象/结构体。启动时渲染第一帧。
2. **一次读取一个按键（或一行）**，分发给修改状态的处理器。
3. **每次操作后重新渲染**完整帧——不要追加，要替换。
4. **循环直到退出。**

整个帧应能在一屏内显示。

### 5. 使其可通过一条命令运行

向项目现有的任务运行器（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）添加一个脚本。用户应运行 `pnpm run <prototype-name>` 或等效命令——永远不需要记住路径。

如果宿主项目没有任务运行器，只需将命令放在原型 README 的顶部。

### 6. 交付

给用户运行命令。他们会自己驱动它；有趣的时刻是他们说"等等，这不应该可能发生"或"嗯，我以为 X 会不一样"——这些是*想法*中的 bug，这正是整个练习的目的。如果他们想添加新操作，就添加。原型会演进。

### 7. 记录答案

当原型完成使命后，问题的答案是唯一值得保留的东西。如果用户在附近，问他们学到了什么。如果不在，在原型旁边留一个 `NOTES.md`，以便在原型被删除之前填写答案（或者由你来填写，如果你观察了整个会话）。

## 反模式

- **不要添加测试。**需要测试的原型不再是原型。
- **不要连接到真实数据库。**使用内存存储，除非问题专门涉及持久化。
- **不要泛化。**不要"如果我们以后想支持 X 怎么办"。原型回答一个问题。
- **不要把逻辑和 TUI 混在一起。**如果 reducer / 状态机引用了 `console.log`、提示符或终端转义码，它就不再是可移植的。保持 TUI 为纯模块之上的薄外壳。
- **不要把 TUI 外壳发布到生产环境。**外壳是为从终端手动驱动而优化的。它背后的逻辑模块才是值得保留的部分。
