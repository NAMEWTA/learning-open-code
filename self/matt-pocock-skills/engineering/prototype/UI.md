# UI 原型

在单一路由上生成**多个截然不同的 UI 变体**，可通过浮动底部栏切换。用户在浏览器中切换变体，挑选一个（或从每个变体中借鉴一些元素），然后丢弃其余的。

如果问题涉及逻辑/状态而非外观——走错分支了。请使用 [LOGIC.md](LOGIC.md)。

## 何时适合使用此方式

- "这个页面应该长什么样？"
- "我想在确定之前看看这个仪表盘的几个选项。"
- "试试设置页面的不同布局。"
- 任何用户原本需要花一整天在脑海中三个模糊模型之间做选择的场景。

## 两种子形态——强烈推荐子形态 A

当 UI 原型**与应用的其余部分碰撞**时更容易评判——真实的页头、真实的侧边栏、真实的数据、真实的密度。一个孤立的 disposable 路由是真空的：每个变体在隔离状态下看起来都挺好。只要有合理的现有页面可以承载变体，就默认使用子形态 A。只有在原型确实没有邻近宿主编排的情况下才使用子形态 B。

### 子形态 A——对现有页面的调整（推荐）

路由已存在。变体在**同一路由**上渲染，通过 `?variant=` URL 搜索参数控制。现有的数据获取、参数和认证保持不变——只替换渲染部分。这是默认方式；除非有特定理由，否则选择此方式。

如果原型的对象还没有页面但*自然地属于某个页面内部*（仪表盘的新区域、设置页面上的新卡片、现有流程中的新步骤）——这仍然是子形态 A。将变体挂载在宿主页面内部。

### 子形态 B——新页面（最后手段）

仅当原型化的对象确实没有可嵌入的现有页面时使用——例如一个全新的顶级界面，或一个无法合理嵌入任何地方的流程。

创建一个**一次性路由**，遵循项目已有的路由约定——不要发明新的顶级结构。命名时使其明显是原型（例如在路径或文件名中包含 `prototype` 一词）。同样使用 `?variant=` 模式。

在确定使用子形态 B 之前，做一次合理性检查：真的没有可以嵌入的现有页面吗？空路由会隐藏设计问题，而有内容的页面则会暴露这些问题。

在两种子形态中，浮动底部栏完全相同。

## 流程

### 1. 明确问题并选定 N

默认为 **3 个变体**。超过 5 个就不再是截然不同的，而是噪音——以此为上限。

在原型所在位置或文件顶部注释中写下一行计划：

> "Three variants of the settings page, switchable via `?variant=`, on the existing `/settings` route."

无论用户是否在场反馈，这都有效。

### 2. 生成截然不同的变体

草拟每个变体。每个变体需满足：

- 页面的目的及其可访问的数据。
- 项目的组件库/样式系统（TailwindCSS、shadcn、MUI、纯 CSS 等）。
- 清晰的导出组件名称，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须**结构上不同**——不同的布局、不同的信息层次、不同的主要交互方式，而不仅仅是不同的颜色。三个微调的卡片网格不是 UI 原型，而是壁纸。如果两份草稿太相似，用明确的"不要使用卡片网格"指导重做其中一份。

### 3. 将它们连接起来

在路由上创建一个单一的切换器组件：

```tsx
// pseudo-code — adapt to the project's framework
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形态 A（现有页面）：将所有现有数据获取保持在切换器之上；每个变体只更改渲染的子树。

对于子形态 B（新页面）：`/prototype/<name>` 下的一次性路由挂载相同的切换器。

### 4. 构建浮动切换器

屏幕底部居中的小型固定位置栏，包含三部分：

- **左箭头**——循环到上一个变体（循环回绕）。
- **变体标签**——显示当前变体键，如果变体导出了名称，也显示名称。例如 `B — Sidebar layout`。
- **右箭头**——向前循环（循环回绕）。

行为：

- 点击箭头会更新 URL 搜索参数（使用框架的路由器——Next 用 `router.replace`，React Router 用 `navigate` 等），使变体可分享且刷新稳定。
- 键盘：`←` 和 `→` 箭头键也可循环。当 `<input>`、`<textarea>` 或 `[contenteditable]` 获得焦点时，不拦截箭头键。
- 在视觉上与页面区分开来（例如高对比度药丸形状、微妙阴影），使其明显不是被评估设计的一部分。
- 在生产构建中隐藏——通过 `process.env.NODE_ENV !== 'production'` 或等效检查进行门控，以防原型的 stray merge 将切换栏发布给用户。

将切换器放在一个共享组件中，以便两种子形态都可以复用。将其放置在项目中共享 UI 的所在位置。

### 5. 交付

提供 URL（以及 `?variant=` 键）。用户会在方便时翻阅。有趣的反馈通常是**"我想要 B 的页头加上 C 的侧边栏"**——这才是他们真正想要的设计。

### 6. 记录答案并清理

一旦某个变体胜出，写下是哪一个以及为什么（commit message、ADR、issue，或者如果离线运行且用户尚未回复，则在原型旁边放一个 `NOTES.md`）。然后：

- **子形态 A**——删除失败的变体和切换器；将胜出者合并到现有页面中。
- **子形态 B**——将获胜变体提升为真实路由，删除一次性路由和切换器。

不要让变体组件或切换器遗留。它们会迅速腐化并让下一个阅读者困惑。

## 反模式

- **仅颜色或文案不同的变体。**那是微调，不是原型。真正的变体在结构上存在分歧。
- **变体之间共享过多代码。**共享 `<Header>` 是可以的；共享 `<Layout>` 则违背了目的。每个变体应该可以自由地抛弃布局。
- **将变体连接到真实的数据变更。**只读原型完全可以。如果变体需要变更数据，将其指向一个桩——问题是"这应该长什么样"，而不是"后端是否正常工作"。
- **将原型直接提升到生产环境。**变体代码是在原型约束下编写的（无测试、最小错误处理）。合并时需正确重写。
