# 问 Matt

你不会记住每个技能，所以来问。

一个**流程**是一条穿越技能的路径。大多数路径沿着一条**主流程**运行，两条**入口匝道**汇入其中。其余都是独立的。

## 主流程：想法 → 交付

大多数工作行进的路线。你有一个想法并想要构建它。

1. **`/grill-with-docs`**——通过访谈打磨想法。当你**有一个代码库**时从这里开始：它是有状态的，将学到的内容保留在 `CONTEXT.md` 和 ADR 中。（没有代码库？使用 `/grill-me`——参见独立技能。）
2. **分支——你能在对话中解决所有问题吗？** 如果一个问题需要可运行的答案（状态、业务逻辑、你必须看到的 UI），通过原型绕行，由 **`/handoff`** 双向桥接（参见跨会话）：
   - **`/handoff`** 输出，然后在那个文件上打开一个新会话，
   - **`/prototype`** 用抛弃型代码回答问题，
   - **`/handoff`** 传回你学到的内容，并从原始想法线程引用它。
3. **分支——这是一个多会话构建吗？**
   - **是** → **`/to-prd`**（将线程转为 PRD）→ **`/to-issues`**（将 PRD 拆分为可独立领取的 issue）。因为 issue 相互独立，**在每个之间清除上下文**：每个 issue 启动一个新会话，通过传入 PRD 和要处理的单个 issue 来启动 **`/implement`**。
   - **否** → **`/implement`** 直接在这里，在同一个上下文窗口中。

### 上下文卫生

将步骤 1-3 保持在**一个不间断的上下文窗口**中——在 `/to-issues` 之前不要压缩或清除——以便质询、PRD 和 issue 都建立在相同的思考基础上。然后每个 `/implement` 从 issue 工作，以全新上下文开始。

这的限制是**[智能区域](https://www.aihero.dev/ai-coding-dictionary/smart-zone)**：模型仍然能清晰推理的窗口（在 state-of-the-art 模型上约 120k token）。如果一个会话在 `/to-issues` 之前接近它，不要在降级状态下推进——`/handoff` 并在一个新线程中继续。

## 入口匝道

一种起始情况，产生工作然后汇入主流程。

- **Bug 和请求堆积** → **`/triage`**。它将 issue 移过分类角色并产生代理就绪的 issue，后续由 **`/implement`** 拾取。

  分类仅适用于**不是你创建的** issue——bug 报告、传入的功能请求、任何原始到达的内容。`/to-issues` 产生的 issue 已经是代理就绪的，所以**不要分类它们**。

## 代码库健康

不是功能工作——维护。

- **`/improve-codebase-architecture`**——每当你有空闲时间时运行，以保持代码库适合代理操作。它浮现深化机会；选择一个会*产生一个想法*，你可以将其带入主流程的 `/grill-with-docs`。

## 跨会话

- **`/handoff`**——当一个线程已满或你需要分支出去（例如进入 `/prototype` 会话）时，这将对话压缩为一个 markdown 文件。你不会在原地继续——你**打开一个新会话并引用该文件**来携带上下文。它是上下文窗口之间的双向桥梁。当你想要一个**新会话**但需要**保留当前对话**时使用它。
- **`/compact`**（内置）——留在**同一对话**中，让较早的回合被摘要。在**阶段之间的有意断点**使用它，当你不介意丢失逐字历史时。不要在阶段中间压缩——代理可能迷失方向。`/handoff` 分叉；`/compact` 继续。

## 独立技能

完全脱离主流程。

- **`/grill-me`**——与 `/grill-with-docs` 相同的无情访谈，但适用于你**没有代码库**时。无状态：它不在本地保存任何内容，不构建 `CONTEXT.md`。用它打磨任何不在仓库中的计划或设计。
- **`/teach`**——通过多次会话学习一个概念，使用当前目录作为有状态的工作区。
- **`/writing-great-skills`**——编写和编辑高质量技能的参考。
