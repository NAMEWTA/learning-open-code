# ADR 格式

ADR 存放在 `docs/adr/` 中，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

按需延迟创建 `docs/adr/` 目录——仅在需要第一个 ADR 时创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：背景是什么，我们做了什么决定，以及为什么。}
```

就是这样。一个 ADR 可以只是一个段落。其价值在于记录*已经*做出了决定以及*为什么*——而不是填满各个章节。

## 可选章节

仅在这些章节能带来真正价值时才包含。大多数 ADR 不需要它们。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——当决策被重新审视时有用
- **Considered Options**——仅当被拒绝的替代方案值得记住时
- **Consequences**——仅当需要指出不明显的下游影响时

## 编号

扫描 `docs/adr/` 中现有的最大编号，然后加一。

## 何时提供 ADR

以下三个条件必须全部满足：

1. **难以逆转**——以后改变主意的成本是有意义的
2. **没有上下文会令人惊讶**——未来的读者看到代码会想"他们到底为什么这样做？"
3. **真正权衡的结果**——存在真正的替代方案，你出于特定原因选择了其中一个

如果一个决定容易逆转，跳过它——你会直接逆转它。如果它不令人惊讶，没人会好奇为什么。如果没有真正的替代方案，除了"我们做了显而易见的事"之外没什么可记录的。

### 什么算

- **架构形态。** "我们使用 monorepo。""写模型是事件溯源的，读模型投射到 Postgres。"
- **上下文之间的集成模式。** "Ordering 和 Billing 通过领域事件通信，而不是同步 HTTP。"
- **带有锁定效应的技术选择。** 数据库、消息总线、认证提供商、部署目标。不是每个库——只是那些需要花一个季度才能换掉的技术。
- **边界和范围决策。** "客户数据由 Customer 上下文拥有；其他上下文仅通过 ID 引用。"明确的"不"和"是"一样有价值。
- **有意识地偏离显而易见的路径。** "我们使用手动 SQL 而不是 ORM，因为 X。"任何合理读者会假设相反的情况。这些可以阻止下一位工程师"修复"一个有意的决定。
- **代码中不可见的约束。** "由于合规要求，我们不能使用 AWS。""由于合作伙伴 API 合同，响应时间必须低于 200ms。"
- **拒绝不明显的替代方案。** 如果你考虑了 GraphQL 并出于微妙原因选择了 REST，记录下来——否则六个月后有人会再次提议 GraphQL。
