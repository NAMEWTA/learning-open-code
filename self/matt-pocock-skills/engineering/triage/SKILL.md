# 分类

将项目 issue 追踪器上的 issue 移过一个由分类角色组成的小状态机。

如果此仓库将外部 Pull Request 视为请求面（参见 issue 追踪器配置），分类也覆盖它们：**PR 是附带代码的 issue**——相同角色、相同状态、相同状态机，下面标记了一些针对 PR 的差异。将裸 `#42` 解析为 issue 或 PR，按追踪器配置。

分类期间发布到 issue 追踪器的每条评论或 issue **必须**以此免责声明开头：

```
> *此内容由 AI 在分类期间生成。*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md)——如何撰写耐用的代理简报
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)——`.out-of-scope/` 知识库如何工作

## 角色

两个**类别**角色：

- `bug`——某物损坏
- `enhancement`——新功能或改进

五个**状态**角色：

- `needs-triage`——维护者需要评估
- `needs-info`——等待报告者提供更多信息
- `ready-for-agent`——完全明确，可供 AFK 代理使用
- `ready-for-human`——需要人工实现
- `wontfix`——不采取行动

对于 PR，相同状态对照附带的代码解读：`ready-for-agent` 表示附带了简报，代理应进行 diff 的下一步；`ready-for-human` 表示可供人工合并。

每个已分类的 issue 应恰好携带一个类别角色和一个状态角色。如果状态角色冲突，标记它并在做任何其他事情之前询问维护者。

这些是规范的角色名——issue 追踪器中使用的实际标签字符串可能不同。映射应已提供给你——如果没有，运行 `/setup-matt-pocock-skills`。

状态转换：未标记的 issue 通常首先进入 `needs-triage`；从那里移动到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。`needs-info` 在报告者回复后返回 `needs-triage`。维护者可随时覆盖——标记看起来不寻常的转换并在继续前询问。

## 调用方式

维护者调用 `/triage` 并用自然语言描述他们想要的。解读请求并执行。示例：

- "显示任何需要我注意的事项"
- "我们看看 #42"（issue 或 PR）
- "将 #42 移到 ready-for-agent"
- "有哪些可供代理领取？"

## 显示需要注意的事项

查询 issue 追踪器并按三个桶呈现，最早的在先：

1. **未标记**——从未被分类。
2. **`needs-triage`**——评估进行中。
3. **`needs-info` 且自上次分类笔记以来有报告者活动**——需要重新评估。

当 PR 在范围内时，在这些桶中包含外部 PR 并在每行标记 `[PR]` 或 `[issue]`。发现仅浮现*外部* PR（追踪器配置定义谁算外部）——协作者进行中的 PR 不是分类工作。此过滤仅用于发现；明确命名的 PR 始终被分类，无论作者是谁。

显示计数和每个项的摘要。让维护者选择。

## 分类特定 issue 或 PR

1. **收集上下文。** 阅读完整的 issue 或 PR（正文、评论、标签、作者、日期；对于 PR，还有 diff）。解析任何先前的分类笔记，以免重新询问已解决的问题。使用项目的领域词汇表探索代码库，尊重该区域的 ADR。对代码库运行两项检查：(a) **冗余**——搜索所请求行为的现有实现，按领域概念（而不仅仅是请求的措辞），并报告你查找的位置。如果找到，它是已实现的 `wontfix`（步骤 5）。(b) **先前拒绝**——阅读 `.out-of-scope/*.md` 并浮现任何类似此请求的内容。

2. **推荐。** 告诉维护者你的类别和状态推荐及理由，加上与请求相关的简要代码库摘要——包括是否已实现。等待指示。

3. **验证声明。** 在任何质询之前，检查声明是否成立。对于 bug，从报告者的步骤复现它。对于 PR，确认 diff 做了它声称的事情——检出、运行相关测试或命令。报告发生了什么：已确认（附代码路径）、失败或细节不足（强烈的 `needs-info` 信号）。确认的验证产生更强大的代理简报。

4. **质询（如需要）。** 如果请求需要充实，一起运行 `/grilling` 和 `/domain-modeling` 技能——一次一个问题地将其质询成形，在决策落地时锐化领域术语并内联更新 `CONTEXT.md`/ADR。

5. **应用结果：**
   - `ready-for-agent`——发布代理简报评论（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human`——与代理简报相同结构，但注明为何无法委派（判断调用、外部访问、设计决策、手动测试）。
   - `needs-info`——发布分类笔记（下方模板）。
   - `wontfix`——关闭，评论取决于*原因*：
     - **已实现**——变更是代码库中已存在的。指出它在哪里；不要写入 `.out-of-scope/`（该知识库用于*被拒绝*的请求，而非已构建的请求）。
     - **被拒绝（bug）**——礼貌解释，然后关闭。
     - **被拒绝（enhancement）**——写入 `.out-of-scope/`，从评论链接到它，然后关闭（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage`——应用角色。如有部分进展，可选评论。

## 快速状态覆盖

如果维护者说"将 #42 移到 ready-for-agent"，信任他们并直接应用角色。确认你将要做什么（角色变更、评论、关闭），然后执行。跳过质询。如果移动到 `ready-for-agent` 而没有质询会话，询问他们是否要撰写代理简报。

## Needs-info 模板

```markdown
## 分类笔记

**目前我们已经确定：**

- 要点 1
- 要点 2

**我们仍需要你提供（@报告者）：**

- 问题 1
- 问题 2
```

将质询期间解决的所有内容捕获在"目前我们已经确定"下，以免工作丢失。问题必须具体且可操作，而非"请提供更多信息"。

## 恢复先前的会话

如果 issue 或 PR 上存在先前的分类笔记，阅读它们，检查报告者是否已回答任何未决问题，并在继续前呈现更新后的画面。不要重新询问已解决的问题。
