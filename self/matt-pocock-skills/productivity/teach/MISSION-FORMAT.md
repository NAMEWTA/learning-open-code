# MISSION.md 格式

`MISSION.md` 位于工作区根目录。它记录了用户学习这个主题的_原因_。每个教学决策——接下来教什么、展示哪些资源、设计哪些练习——都应追溯到这份文档。

## 模板

```md
# 学习目标：{主题}

## 为什么
{1-3 句话。用户正在追求的具体的现实世界目标。当他们掌握这项技能后，他们的生活或工作会发生什么变化？避免使用"理解 X"这类抽象表述——要追问背后的实际结果。}

## 成功的样子
- {用户将能够做到的一件具体的、可观察的事}
- {另一件具体的事}
- {……}

## 约束条件
- {时间、预算、已有承诺、学习偏好，任何限制方法的条件}

## 不在范围内
- {用户明确表示目前不想追求的相邻主题——保护最近发展区}
```

## 规则

- **每个工作区一个目标。** 如果用户想学两个不相关的东西，那就是两个工作区。
- **具体优于抽象。** "十月前跑完半程马拉松"优于"变得更健康"。"为我的团队交付一个 Rust CLI"优于"学习 Rust"。
- **追问模糊性。** 如果用户不能清晰表达原因，在写任何东西之前先和他们交流。一个糟糕的目标比没有目标更糟。
- **当现实变化时修订。** 目标会变。当用户的目标转移时，更新此文件——不要让一个过时的目标继续指引未来的课程。
- **保持简短。** 如果 `MISSION.md` 超过一屏，它就不再是指南针，而变成了计划书。
