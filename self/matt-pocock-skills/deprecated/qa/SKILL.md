# QA 会话

运行一个交互式 QA 会话。用户描述他们遇到的问题。你进行澄清、在后台探索代码库以获取上下文，并提交耐用、以用户为中心、使用项目领域语言的 GitHub issue。

## 对于用户提出的每个问题

### 1. 倾听并适度澄清

让用户用自己的话描述问题。最多问 **2-3 个简短的澄清问题**，聚焦于：

- 他们期望什么 vs 实际发生了什么
- 复现步骤（如果不明显的话）
- 是持续发生还是间歇性发生

不要过度访谈。如果描述已经足够清晰可以提交，就继续下一步。

### 2. 在后台探索代码库

在与用户交谈的同时，在后台启动一个 Agent（subagent_type=Explore）来理解相关区域。目的不是找到修复方案——而是：

- 学习该区域使用的领域语言（检查 UBIQUITOUS_LANGUAGE.md）
- 理解该功能应该做什么
- 识别面向用户的行为边界

这些上下文帮助你写出更好的 issue——但 issue 本身不应引用具体文件、行号或内部实现细节。

### 3. 评估范围：单个 issue 还是拆分？

在提交之前，判断这是一个**单个 issue** 还是需要**拆分为多个 issue**。

需要拆分的情况：

- 修复涉及多个独立领域（例如"表单验证有问题 AND 成功消息缺失 AND 重定向坏了"）
- 存在明显可分离的关注点，不同的人可以并行处理
- 用户描述了具有多个不同失败模式或症状的问题

保持为单个 issue 的情况：

- 是一个地方的一个行为出了问题
- 所有症状都由同一个根本行为引起

### 4. 提交 GitHub issue

使用 `gh issue create` 创建 issue。不要先让用户审核——直接提交并分享 URL。

Issue 必须是**耐用的**——即使经过大规模重构，仍应有意义。从用户视角编写。

#### 单个 issue

使用此模板：

```
## 发生了什么

[用通俗语言描述用户实际经历的行为]

## 我期望的结果

[描述期望的行为]

## 复现步骤

1. [开发者可以遵循的具体、编号步骤]
2. [使用代码库中的领域术语，而非内部模块名]
3. [包含相关的输入、标志或配置]

## 补充上下文

[来自用户或代码库探索的任何额外观察，有助于框定问题——例如"这仅在使用 Docker 层时发生，文件系统层不会"——使用领域语言但不引用具体文件]
```

#### 拆分（多个 issue）

按依赖顺序创建 issue（阻塞项优先），以便引用真实的 issue 编号。

每个子 issue 使用此模板：

```
## 父 issue

#<父 issue 编号>（如果你创建了跟踪 issue）或"在 QA 会话中报告"

## 出了什么问题

[描述这个具体的行为问题——仅此切片，而非整个报告]

## 我期望的结果

[此切片的期望行为]

## 复现步骤

1. [针对此 issue 的步骤]

## 被阻塞于

- #<issue 编号>（如果此 issue 在另一个被解决之前无法修复）

或"无——可以立即开始"（如果没有阻塞项）。

## 补充上下文

[与此切片相关的任何额外观察]
```

拆分时：

- **优先创建多个细粒度 issue 而非少量粗粒度 issue**——每个 issue 应可独立修复和验证
- **诚实地标注阻塞关系**——如果 issue B 确实在 issue A 修复之前无法测试，就明确说明。如果相互独立，都标注为"无——可以立即开始"
- **按依赖顺序创建 issue**，以便在"被阻塞于"中引用真实的 issue 编号
- **最大化并行度**——目标是多人（或多代理）可以同时领取不同的 issue

#### 所有 issue 正文的规则

- **不包含文件路径或行号**——这些会很快过时
- **使用项目的领域语言**（如果存在 UBIQUITOUS_LANGUAGE.md，请参考）
- **描述行为，而非代码**——"同步服务无法应用补丁"而不是"applyPatch() 在第 42 行抛出异常"
- **复现步骤是必填项**——如果无法确定，向用户询问
- **保持简洁**——开发者应该能在 30 秒内读完 issue

提交后，打印所有 issue URL（附带阻塞关系摘要）并询问："下一个问题，还是结束了？"

### 5. 继续会话

持续进行直到用户表示完成。每个 issue 相互独立——不要批量处理。
