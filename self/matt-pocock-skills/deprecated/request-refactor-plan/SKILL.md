---

当用户想要创建重构请求时，将调用此技能。你应按以下步骤进行。如果认为某些步骤不必要，可以跳过。

1. 向用户询问他们想要解决的问题的详细、长篇描述，以及任何潜在的解决方案思路。

2. 探索仓库以验证他们的断言并理解代码库的当前状态。

3. 询问他们是否考虑过其他选项，并向他们展示其他选项。

4. 就实现方案访谈用户。要极其详细和彻底。

5. 敲定实现的确切范围。明确你计划修改什么和不修改什么。

6. 查看代码库中该区域的测试覆盖情况。如果测试覆盖不足，询问用户的测试计划。

7. 将实现拆分为小粒度提交计划。牢记 Martin Fowler 的建议："让每个重构步骤尽可能小，这样你总能看到程序在正常工作。"

8. 创建包含重构计划的 GitHub issue。issue 描述使用以下模板：

<refactor-plan-template>

## 问题陈述

开发者面临的问题，从开发者视角描述。

## 解决方案

问题的解决方案，从开发者视角描述。

## 提交

一份长的、详细的实现计划。用通俗语言编写计划，将实现拆分为尽可能小的提交。每个提交应使代码库保持在可工作状态。

## 决策文档

所做实现决策的列表。可包括：

- 将要构建/修改的模块
- 这些模块将被修改的接口
- 来自开发者的技术澄清
- 架构决策
- Schema 变更
- API 约定
- 具体交互方式

不要包含具体的文件路径或代码片段。它们可能很快就会过时。

## 测试决策

所做测试决策的列表。包括：

- 什么是好测试的描述（只测试外部行为，不测试实现细节）
- 哪些模块将被测试
- 测试的参考先例（即代码库中类似类型的测试）

## 不在范围内

描述本次重构不在范围内的事项。

## 补充说明（可选）

关于重构的任何补充说明。

</refactor-plan-template>
