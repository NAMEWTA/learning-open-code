当模糊想法需要一个以上的代理会话才能转化为计划时，调用此技能。它在一个 markdown 文件中创建一个有状态的决策地图，并驱动用户通过一系列工单来解决未决问题——这可能需要原型构建、调研或讨论。

## 决策地图

决策地图是一个紧凑的 Markdown 文件，每个规划工作一个文件，与项目一起进行 git 跟踪。它是权威产出物——**整个地图在每个会话中都作为上下文加载**，因此必须保持紧凑。

工单期间创建的资产应从地图链接，而不是在其中重复。

### 结构

编号条目（"工单"），每个工单按其编号作为独立章节：

```markdown
## #1: 关系型数据库还是非关系型数据库？

阻塞于：#<工单编号>、#<工单编号>
类型：调研 | 原型 | 讨论

### 问题

<此处为问题>

### 答案

<此处为答案>
```

每个工单的大小必须适配一个 100K token 的代理会话。

## 工单类型

工单有三种类型：

- **调研**：阅读文档、第三方 API 或知识库等本地资源。创建一个 markdown 摘要作为资产。当需要当前工作目录之外的知识时使用。
- **原型**：编写 UI 或逻辑代码来测试假设，或探索设计空间。使用 /prototype 技能。创建一个原型作为资产。当"应该是什么样子"或"应该如何行为"是关键问题时使用。
- **讨论**：与代理进行对话。使用 /grilling 和 /domain-modelling 技能。默认选项。

## 战争迷雾

地图*故意*在前沿之外是不完整的。你的工作是调研前沿，并逐一解决工单以推进前沿。一片一片地驱散战争迷雾。

在某个时刻，战争迷雾应已被推进到足够远，以至于通往终点的路径已经清晰。此时不再需要更多工单，决策地图可以视为"已完成"。

## 调用方式

有两种方式调用此技能：**引导启动**和**恢复**。

### 引导启动

用户带着一个模糊想法调用。

1. 运行 /grilling 和 /domain-modelling 会话以暴露未决决策。
2. 编写一个新的决策地图——大部分是迷雾，前沿已识别，可轻易决策的条目已内联解决。
3. 停止。地图构建是一个会话的工作；不要同时解决工单。

### 恢复

用户带着现有地图的路径和一个工单编号调用。

1. 将**整个地图**作为上下文加载。
2. 运行一个会话来解决工单，根据需要调用技能。如有疑问，使用 `/grilling` 和 `/domain-modelling`。
3. 将会话解决的内容记录在工单正文中。
4. 添加新发现的工单（带有正确的 `blocked_by` 边）。
5. 停止。

如果所做决策使地图的其他部分失效，更新或删除这些节点。

## 并行

用户可以选择并行运行工单，因此请预料其他代理会对地图进行更改。

## 跳过决策地图

很多时候，初始质询不会产生战争迷雾。没有未解决的工单。除了实现，无事可做。

在这些情况下，你应向用户提供跳过决策地图的选项——因为只有在需要进行多会话决策时才需要决策地图。

如果他们跳过，你应建议直接实现或使用 `/to-prd` 来安排多会话实现。
