对 `HEAD` 和用户提供的固定点之间的 diff 进行双维度审查：

- **标准**——代码是否符合此仓库文档化的编码标准？
- **Spec**——代码是否忠实地实现了原始 issue / PRD / spec？

两个维度作为**并行子代理**运行，以免相互污染上下文，然后此技能汇总它们的发现。

Issue 追踪器应已提供给你——如果 `docs/agents/issue-tracker.md` 缺失，运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 确定固定点

用户所说的任何内容即为固定点——一个 commit SHA、分支名、tag、`main`、`HEAD~5` 等。如果他们没有指定，请询问。

一次性捕获 diff 命令：`git diff <fixed-point>...HEAD`（三点，以便与 merge-base 进行比较）。同时通过 `git log <fixed-point>..HEAD --oneline` 记录提交列表。

在继续之前，确认固定点可解析（`git rev-parse <fixed-point>`）且 diff 非空。错误的引用或空 diff 应在此处失败——而不是在两个并行子代理内部。

### 2. 识别 spec 来源

按以下顺序查找原始 spec：

1. 提交消息中的 issue 引用（`#123`、`Closes #45`、GitLab `!67` 等）——通过 `docs/agents/issue-tracker.md` 中的工作流获取。
2. 用户作为参数传递的路径。
3. `docs/`、`specs/` 或 `.scratch/` 下与分支名或功能匹配的 PRD/spec 文件。
4. 如果什么都没找到，询问用户 spec 在哪里。如果他们说没有，**Spec** 子代理将跳过并报告"无可用的 spec"。

### 3. 识别标准来源

仓库中任何记录代码应如何编写的文档，如 `CODING_STANDARDS.md` 或 `CONTRIBUTING.md`。

### 4. 并行启动两个子代理

发送一条包含两个 `Agent` 工具调用的消息。对两者都使用 `general-purpose` 子代理。

**标准子代理提示词**——包含：

- 完整的 diff 命令和提交列表。
- 你在步骤 3 中找到的标准来源文件列表。
- 任务简述："报告——按相关文件/代码块——diff 中违反文档化标准的每一处。引用标准（文件 + 规则）。区分硬性违规和判断性差异。跳过工具已强制执行的内容。不超过 400 字。"

**Spec 子代理提示词**——包含：

- diff 命令和提交列表。
- spec 的路径或获取的内容。
- 任务简述："报告：(a) spec 要求但缺失或不完整的需求；(b) diff 中存在但 spec 未要求的行为（范围蔓延）；(c) 看起来已实现但实现方式似乎有误的需求。每个发现引用 spec 原文。不超过 400 字。"

如果 spec 缺失，跳过 Spec 子代理并在最终报告中注明。

### 5. 汇总

在 `## 标准` 和 `## Spec` 标题下呈现两份报告，原文或稍作清理。不要合并或重新排序发现——两个维度是有意分离的（参见*为什么是两个维度*）。

以一行摘要结束：每个维度的总发现数，以及每个维度内最严重的问题（如有）。不要跨维度选出一个优胜者——这正是分离所要防止的重新排序。

## 为什么是两个维度

一个变更可以通过一个维度而未能通过另一个：

- 遵循所有标准但实现了错误功能的代码 → **标准通过，Spec 失败。**
- 完全按照 issue 要求但违反项目约定的代码 → **Spec 通过，标准失败。**

分别报告可以防止一个维度掩盖另一个。
